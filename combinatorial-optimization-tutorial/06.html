开篇：两个程序员的故事

大家好，我是你们的讲师。今天我们要讲两种截然不同的算法思想——贪心和动态规划。让我先用一个故事来引入。

几年前，我和两个朋友（小张和小李）参加一个算法比赛。有一道题是这样的：给定一组任务，每个任务有开始时间和结束时间，你最多能参加多少个互不重叠的任务？

小张说："这还不简单？每次选结束时间最早的任务，然后跳过和它冲突的，继续选下一个最早结束的。"他花了3分钟就写完了代码，提交，通过。

小李说："等等，万一最优方案不是这样选的呢？我用动态规划，保证找到最优。"他花了一个小时写了一个 DP，优化了空间复杂度，各种剪枝……提交，也通过了。

比赛结束，小张拿了第七名，小李拿了第九名——小张因为速度快，多做了两道题。

课后我们分析那道题，发现小张的贪心策略（最早结束时间优先）确实是正确的——它保证找到最多任务数。小李的动态规划虽然同样正确，但犯了"杀鸡用牛刀"的毛病。

这个故事的教训是两面的：一方面，贪心算法如果正确，就是最优雅、最高效的方案；另一方面，不是所有问题都能贪心，动态规划虽然慢一些，但适用范围更广。理解两者的差异和适用条件，是本章的核心目标。

贪心算法

什么是贪心

贪心算法（Greedy Algorithm）的核心思想是：在每一步选择"看起来最好"的选项，并期望这个局部最优选择最终导致全局最优解。它不像穷举那样考虑所有可能性，也不像动态规划那样先求解子问题再组合——它只看眼前，一步到位。

贪心算法的典型结构：

贪心算法框架：
1. 将问题分解为一系列步骤
2. 每一步根据某个"贪心准则"（Greedy Criterion）选择当前最优
3. 做出选择后，缩小问题规模
4. 直到所有步骤完成

经典案例：活动选择

活动选择问题（Activity Selection）：给定 n 个活动，每个活动有开始时间 sᵢ 和结束时间 fᵢ，选择尽可能多的互不重叠的活动。

贪心准则：每次选结束时间最早的活动。

活动列表（按结束时间排序）：
活动 A: [1, 4]   ✓ 选
活动 B: [3, 5]   ✗（与A重叠）
活动 C: [0, 6]   ✗（与A重叠）
活动 D: [5, 7]   ✓ 选
活动 E: [8, 9]   ✓ 选
活动 F: [5, 9]   ✗（与D重叠）

最优解：{A, D, E}，共3个活动

代码：
sort activities by finish time
last_finish = 0
selected = []
for each activity (s, f) in sorted order:
    if s >= last_finish:
        selected.append(activity)
        last_finish = f

经典案例：Huffman 编码

Huffman 编码是最优前缀码的贪心构造算法：每次合并频率最小的两个节点，构造一棵二叉树。它被用于 JPEG、ZIP 等几乎所有压缩算法中。

贪心准则：每次选择频率最小的两个字符合并。

经典案例：Kruskal 最小生成树

我们在第2章已经学过：Kruskal 算法每次选"权重最小且不会形成环"的边。这就是一个贪心算法，而且——请注意——它是正确的。它保证了全局最优。

贪心案例 贪心准则 是否最优 时间复杂度
活动选择 最早结束时间优先 是 O(n log n)
Huffman 编码 最小频率合并 是 O(n log n)
Kruskal MST 最小权边且无环 是 O(E log E)
0/1背包 性价比最高优先 否 O(n log n)
找零钱 最大面额优先 视面额系统而定 O(n)

注意：0/1背包的性价比贪心并不保证最优——这就是贪心的"陷阱"。能正确的时候非常高效，但出错了你也察觉不到。

一个让我印象深刻的教训：我的一位同事曾经想用贪心法解决一个"最小化加权完成时间"的调度问题。他选择了"最短处理时间优先"（SPT）作为贪心准则，觉得非常合理。运行结果看起来也不错——直到他有一天发现，最优解比他"最优的贪心解"还要好23%。问题出在哪里？这个调度问题中，任务的权重不同，不能只看处理时间。正确的策略是"按处理时间/权重的比值排序"（Smith's Rule）。这个案例说明：贪心算法的正确性不能靠"感觉"来判定，必须要有严格的证明。

贪心正确性的条件

贪心选择性质

一个问题能用贪心算法求解的前提是满足贪心选择性质（Greedy Choice Property）：可以通过一系列局部最优选择来构造全局最优解。也就是说，你不需要预先求解子问题——每次选当前最优，然后把剩余问题继续贪心即可。

最优子结构

另一个条件是最优子结构（Optimal Substructure）：问题的最优解包含其子问题的最优解。这个条件也是动态规划的基础。

拟阵结构简介

拟阵（Matroid）是一个抽象的数学结构，它统一了一大类"贪心可解"的问题。一个拟阵 M = (S, I) 由有限集合 S 和 I ⊆ 2^S（独立子集族）组成，满足三条公理：

  1. 空集独立性：∅ ∈ I。
  2. 遗传性：如果 A ∈ I 且 B ⊆ A，那么 B ∈ I。
  3. 交换性：如果 A, B ∈ I 且 |A| < |B|，那么存在 x ∈ B \ A 使得 A ∪ {x} ∈ I。

拟阵上的贪心算法一定最优：在拟阵上，选择权重最大的独立集时，贪心算法（每次选最大权重的元素加入，如果保持独立性就保留）保证找到最优解。

很多经典问题可以建模成拟阵上的贪心：

  • 图拟阵（Graphic Matroid）：S = 边集，I = 所有无环子集（MST 正好是拟阵上的贪心）。
  • 划分拟阵（Partition Matroid）：集合被划分为若干组，每组只能选有限个元素。
  • 向量拟阵（Vector Matroid）：一组向量中的线性无关子集。

拟阵理论告诉我们：为什么 MST 可以用贪心而 TSP 不行？因为 MST 对应图拟阵（贪心最优），而 TSP 的解结构——哈密顿环——不构成拟阵。这就是数学给出的清晰分界线。

一句话总结：如果一个问题可以建模成拟阵，贪心算法就是精确算法。如果不能，贪心最多只是一个启发式（近似算法）。这个结论是组合优化中最优雅也最实用的结果之一。我们将在第19章深入讨论拟阵理论。

动态规划核心思想

从斐波那契数列说起

经典的斐波那契数列递归计算 F(n) = F(n-1) + F(n-2) 会重复计算很多次——F(3) 被计算了2次，F(2) 被计算了3次，浪费了大量时间。如果我们把每个子问题的结果存下来（记忆化），需要的时候直接查表——计算量就从 O(2ⁿ) 降到了 O(n)。

这个"存下来"的思想就是动态规划的核心。动态规划（Dynamic Programming, DP）通过将问题分解为重叠的子问题，存储子问题结果避免重复计算，从而高效求解。

DP 的两个关键条件

  • 最优子结构（Optimal Substructure）：问题的最优解可以从子问题的最优解构造出来。
  • 重叠子问题（Overlapping Subproblems）：子问题被重复求解。

遇到一个问题时，思考流程是：

DP 设计步骤：
1. 定义状态：dp[i][j] 表示什么？（"什么"决定了"怎么算"）
2. 定义状态转移方程：dp[i][j] = f(dp[i-1][j], dp[i][j-1], ...)
3. 定义初始条件/边界：dp[0][*] = ?
4. 确定计算顺序：自顶向下（递归+记忆化）还是自底向上（递推）
5. 确定答案在哪里：dp[n][m] 还是 max(dp[*])？

DP 经典案例

0/1 背包问题

有 n 个物品，第 i 个物品重量 wᵢ，价值 vᵢ，背包容量 W，每个物品最多选一次。求总价值最大。

定义状态：dp[i][c] = 考虑前 i 个物品，容量为 c 时的最大价值
转移方程：
    dp[i][c] = max(
        dp[i-1][c],              // 不选第 i 个物品
        dp[i-1][c-wᵢ] + vᵢ      // 选第 i 个物品（前提 c ≥ wᵢ）
    )
初始条件：dp[0][*] = 0
答案：dp[n][W]
时间复杂度：O(nW)

空间优化（滚动数组）：
    由于 dp[i][*] 只依赖 dp[i-1][*]，可以只用一维数组：
    for c from W down to wᵢ:  // 注意倒序！
        dp[c] = max(dp[c], dp[c-wᵢ] + vᵢ)
关于时间复杂度：0/1背包的 O(nW) 看起来是多项式，但它实际上是伪多项式时间（Pseudo-Polynomial Time）——因为 W 是输入数值而不是输入长度（W 的数值可以用 log₂W 个比特表示，所以输入长度是 log W，而 O(nW) 不是 n 的多项式，而是 n 和 2^logW 的多项式）。这解释了为什么背包问题是 NP 难的——如果 W 很大（例如 10⁹），O(nW) 是完全不可行的。但 W 较小时，这个 DP 非常实用。

最长公共子序列（LCS）

给定两个字符串 X 和 Y，求它们的最长公共子序列。子序列不要求连续，但顺序必须一致。

定义状态：dp[i][j] = X[1..i] 和 Y[1..j] 的 LCS 长度
转移方程：
    if X[i] == Y[j]:
        dp[i][j] = dp[i-1][j-1] + 1
    else:
        dp[i][j] = max(dp[i-1][j], dp[i][j-1])

答案：dp[m][n]
时间复杂度：O(mn)

编辑距离（Edit Distance）

将字符串 A 转换成 B 所需的最少操作次数（插入、删除、替换字符）。常用于拼写检查和基因序列比对。

定义状态：dp[i][j] = A[1..i] 转换成 B[1..j] 的最小编辑距离
转移方程：
    if A[i] == B[j]:
        dp[i][j] = dp[i-1][j-1]
    else:
        dp[i][j] = 1 + min(
            dp[i-1][j],     // 删除 A[i]
            dp[i][j-1],     // 插入 B[j]
            dp[i-1][j-1]    // 替换 A[i] → B[j]
        )

答案：dp[m][n]
时间复杂度：O(mn)

矩阵链乘法

给定一系列矩阵 A₁, A₂, ..., Aₙ，确定最优的乘法顺序（加括号方式），使总乘法次数最小。

定义状态：dp[i][j] = 计算 A₁×⋯×Aⱼ 的最小乘法次数
（其中 Aₖ 的维度为 pₖ₋₁ × pₖ）

转移方程：
    dp[i][j] = min {
        dp[i][k] + dp[k+1][j] + pₖ₋₁·pₖ·pⱼ
    } for k = i to j-1

初始条件：dp[i][i] = 0（单个矩阵不需要运算）
答案：dp[1][n]
时间复杂度：O(n³)

这个问题的特点是：子问题不是前缀（第1到第i个），而是区间（第i到第j个）。
这就是区间DP（Interval DP）的典型框架。
DP 案例 状态定义 时间复杂度 空间优化
0/1 背包 dp[i][c] O(nW) 滚动数组 → O(W)
LCS dp[i][j] O(mn) 滚动数组 → O(min(m,n))
编辑距离 dp[i][j] O(mn) 滚动数组 → O(min(m,n))
矩阵链乘 dp[i][j]（区间） O(n³) 难以优化，O(n²)

DP 进阶技巧

记忆化搜索 vs 递推

记忆化搜索（Memoization）是自顶向下的——从原问题出发，递归求解子问题，把结果存下来。递推（Tabulation）是自底向上的——从最基础的状态开始，逐步填表构建答案。

两者的选择建议：

  • 如果所有状态都需要计算，递推通常更快（没有递归函数调用开销）。
  • 如果状态空间稀疏（只有一部分状态会被用到），记忆化搜索更优（只计算需要的）。
  • 如果状态转移较复杂，记忆化搜索更容易写（直接对应递归关系）。

状态压缩

状态压缩（State Compression）是指用位运算（Bitmask）来表示集合。例如，TSP 的 DP 解法 dp[mask][v] 中，mask 是一个 n 位的二进制数，表示哪些城市已经访问过了。这种"集合 DP"在 n ≤ 20 时非常实用。

// TSP DP（状态压缩）：
// dp[mask][v] = 从起点出发，访问了 mask 中的城市，最后停在 v 的最小距离
// mask 的二进制第 i 位 = 1 表示访问了城市 i

初始化：dp[1<<0][0] = 0（起点为城市0）
for mask = 1 to (1<<n)-1:
    for u = 0 to n-1:
        if mask & (1<<u):  // u 在 mask 中
            for v = 0 to n-1:
                if not (mask & (1<<v)):  // v 不在 mask 中
                    new_mask = mask | (1<<v)
                    dp[new_mask][v] = min(dp[new_mask][v],
                                          dp[mask][u] + dist[u][v])

答案：min over v of dp[(1<<n)-1][v] + dist[v][0]
时间复杂度：O(n²·2ⁿ)

滚动数组优化空间

当 DP 状态 dp[i][j] 只依赖 dp[i-1][*] 时（如 0/1 背包、LCS），可以用两个一维数组甚至一个一维数组来替代二维数组，将空间复杂度从 O(nm) 降到 O(m)。这是最常见的 DP 空间优化技巧。

DP 设计的心法：很多人在学 DP 时陷入一个误区——他们试图直接写出最优的 DP 方程。正确的方法不是这样。你应该从最直观的（可能是 3维 或更高维）状态定义开始，验证正确性，然后逐步优化维度和空间。先让代码"对"，再让它"快"。我记得自己刚开始学 DP 时，花了整整两周死磕 0/1 背包的滚动数组优化——实际上，先写出二维 DP，理解透了，再优化成一维，这个循序渐进的过程反而更快。

贪心 vs DP：一张总结表

对比维度 贪心算法 动态规划
决策方式 只做一次性选择，不回头 探索所有子问题，综合决策
正确性条件 贪心选择性质 + 最优子结构 最优子结构 + 重叠子问题
适用范围 窄（只对特定结构的问题正确） 宽（几乎所有最优子结构问题）
时间复杂度 通常 O(n log n) 或更低 通常 O(n²) 或更高
典型应用 MST, Huffman, 活动选择 背包, LCS, 编辑距离, TSP(小规模)
证明难度 需要证明局部最优→全局最优 需要证明状态转移正确性
