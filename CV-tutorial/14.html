14.1 评估不是"跑个准确率"那么简单

在我刚入行的时候，我评估一个模型的方式非常简单直接：跑一下测试集，看Top-1准确率有多高。如果A模型的准确率是78.5%，B模型是78.2%，那么A模型"更好"——结论就是这么简单。

后来在一家公司做技术选型评审时，我差点因为这种简单粗暴的评估方式犯下大错。当时我们团队需要为一个移动端App选择一个实时分类模型。我对比了ResNet-50和EfficientNet-B3——ResNet-50的准确率略高0.3个百分点。我毫不犹豫地推荐了ResNet-50。但最终在实测中，ResNet-50在手机上的推理延迟是350ms（严重卡顿），而EfficientNet-B3只要110ms（流畅体验）。用户根本不关心0.3%的准确率差异——他们在乎的是相机打开后能不能实时识别。

这次经历让我深刻理解了：任何脱离实际部署约束的模型评估都是不负责任的。模型评估从来不是单一指标的比较——它是在精度、速度、内存、功耗等多个维度上的综合权衡。

14.2 分类模型的核心评价指标

在CV分类任务中，最常用的评估指标包括以下几个维度：

指标类别指标名称含义单位/范围
精度Top-1 Accuracy模型预测的最高概率类别是否正确百分比（越高越好）
精度Top-5 Accuracy模型预测的前5个类别中是否包含正确类别百分比（越高越好）
模型大小参数量（Parameters）模型中可学习参数的总数M（百万）
计算量FLOPs / MACs一次前向传播所需的浮点运算次数G（十亿）
速度延迟（Latency）单个样本的推理时间（含预处理和后处理）毫秒（越低越好）
吞吐量Throughput单位时间内处理的样本数samples/s（越高越好）
存储模型文件大小保存模型权重所需的磁盘空间MB（越低越好）

注意：Top-1和Top-5已经是20年ImageNet竞赛的标准指标。但随着数据集难度降低（模型越来越强），Top-5区分度已经不够了——几乎所有现代模型在ImageNet上都达到95%以上的Top-5。因此，近年来的研究更依赖Top-1和校准误差（ECE，Expected Calibration Error）来评估模型。

14.3 经典主干网络的性能演进

让我们从第8章到第12章讨论过的所有模型出发，做一个完整的性能对比。下面这张表整理了各个模型在ImageNet上的指标，所有数据均来自原始论文或官方实现，在统一的224×224输入分辨率下比较：

年份模型参数量FLOPsTop-1 Acc核心贡献
2012AlexNet61M0.7G56.5%深度学习破冰
2014VGG-16138M15.3G71.6%规则化深度设计
2015ResNet-5025.6M4.1G76.2%残差学习
2015ResNet-15260.3M11.3G77.8%超深网络可行
2020EfficientNet-B766M37.0G84.4%NAS + 复合缩放
2020ViT-B/16 (JFT)86M16.0G84.2%Transformer视觉化
2021Swin-B88M15.4G83.5%层次化Transformer
2022ConvNeXt-B89M15.4G83.5%CNN现代化改造

从这张表可以清楚看到几个趋势：

  • 精度在持续提升：AlexNet的56.5%到ConvNeXt的83.5%，十年间提升了27个百分点——这在CV领域是惊人的进步速度。
  • 参数量不是精度的唯一决定因素：VGG-16有138M参数（最多）但精度（71.6%）远不如只有89M参数的ConvNeXt-B（83.5%）。"更高效"的架构设计比堆参数更重要。
  • FLOPs和参数量不一定正相关：ResNet-152（60M参数, 11.3G FLOPs）和VGG-16（138M参数, 15.3G FLOPs）——VGG的参数更多但FLOPs相仿，因为VGG的计算集中在全连接层（参数多但计算效率高），而ResNet的计算集中在卷积层。
📈 重要提醒：不要直接跨论文比较数字！上面的表格已经尽量统一了条件，但不同论文中报告的准确率可能来自不同的训练策略、数据增强、推理配置（如是否使用多crop测试）。例如，ViT-B在ImageNet上从头训练（不使用JFT预训练）的Top-1是77.9%，远低于表中的84.2%（JFT预训练版本）。公平的比较需要控制所有变量除了目标模型本身——这也是为什么像timm（PyTorch Image Models）这样的工具库如此重要：它在统一的训练框架下提供了大量模型的公平对比。

14.4 不同场景的模型选型

不同的部署场景对模型的要求截然不同。下面是几个典型场景的选型指南：

场景核心约束推荐模型选型理由
移动端App（实时推理）延迟 < 50ms，模型 < 10MBMobileNetV3, EfficientNet-Lite, ShuffleNetV2深度可分离卷积大幅降低计算量，模型小、延迟低
云端API（高精度优先）精度最高，延迟次要ConvNeXt-L, Swin-L, ViT-L (JFT pre-trained)大模型 + 大数据预训练，追求极致精度

自动驾驶（安全关键）高鲁棒性，低延迟ResNet-50/101, ConvNeXt-T成熟稳定，推理优化完善，对恶劣光照鲁棒性好
科研/学术基准需要与SOTA公平对比根据领域选择：分类用ConvNeXt / ViT / Swin；分割用Swin / ConvNeXt；检测用ResNet-FPN遵循所在子领域的标准backbone选择，确保对比公平

这张表只是一个起点。在实际选型中，你还需要考虑：模型在你的特定数据集上的表现（ImageNet精度高不代表在你的数据集上一定好——领域迁移的影响不可忽视）、推理框架的支持程度（ONNX / TensorRT / CoreML是否针对你的模型做过优化）、以及团队维护成本。

14.5 理论FLOPs vs 实际延迟

一个常见的误区是拿FLOPs来预测实际推理速度。FLOPs衡量的是理论计算量，但实际延迟受到多种因素的复杂影响：

  • 内存带宽：深度可分离卷积的FLOPs很低（因为参数量小），但它的计算密度（compute intensity，计算量/访存量）也很低，实际中受限于内存带宽而不是算力。这就是为什么MobileNet在CPU上很快，在GPU上却不那么快——GPU的算力优势被内存带宽瓶颈抵消了。
  • 算子效率：3×3卷积已经被cuDNN等底层库优化到了极致（接近硬件的理论峰值），而7×7 depthwise conv、自注意力等操作还没有同样级别的优化。所以即便两个操作的FLOPs相同，实际延迟可能差几倍。
  • 并行度：自注意力机制在batch size较大时可以利用GPU的高并行度，batch size=1时（在线推理场景）则效率低下。
# 一个生动的例子：理论FLOPs vs 实际延迟
# 假设我们对比两个模型，在相同的GPU上测试：
#
#                       FLOPs      理论占比    实测延迟(ms)    实际占比
# Model A (ConvNeXt-T):  4.5G      1.0×           12.3           1.0×
# Model B (Swin-T):      4.5G      1.0×           18.7           1.5×
#
# FLOPs完全相同的两个模型，实际延迟差了50%！
# 原因：Swin的窗口注意力操作不如ConvNeXt的深度可分离卷积
# 在cuDNN中优化得好。
#
# 更极端的例子（CPU推理）：
# Model C (MobileNetV3): 0.2G      —              15.2           —
# Model D (ResNet-18):   1.8G      —              22.0           —
# 计算量差9倍，但延迟只差45%！
# 原因：MobileNet的深度可分离卷积在CPU上内存带宽受限。

因此，千万不要只看FLOPs就判断模型速度。正确的做法是：在目标硬件上直接测延迟。对于GPU推理，用torch.cuda.synchronize()计时；对于CPU推理，用time.perf_counter()计时。多次运行取中位数，同时报告batch size的影响。

14.6 模型Scaling定律

近年来，一个重要的观察是：模型性能（准确率）和计算资源（模型大小、数据量、训练计算量）之间存在可预测的幂律关系——这就是Scaling Law（缩放定律）。

具体来说，对于一个给定的架构族（如ResNet或ViT系列），模型的测试Loss通常与训练计算量呈幂律关系：Loss ∝ C^(-α)，其中C是训练计算量（FLOPs × 数据量），α是某个正的常数。这意味着：如果你想将Loss降低一半，你需要将计算量增加约2^(1/α)倍——通常这个系数在10到100之间。

Scaling Law的一个重要结论是：增加模型大小、数据量和计算量三者需要同步。如果只增加模型大小而不增加数据量，模型的收益会快速饱和（过拟合）；如果只增加数据量而不增加模型大小，模型的收益同样会饱和（欠拟合）。最优策略是三者按比例同步增长。

💡 实用建议：如果你的资源有限（比如只有4张GPU），不要试图复现一个ViT-Huge/JFT-300M级别的工作。Scaling Law告诉我们，"小模型+足够数据"的性价比远高于"大模型+不足数据"。对于大部分实际项目，选择一个合适的Tiny或Small版本模型（如ConvNeXt-T、Swin-T、ResNet-50），用比你当前数据量大10倍的数据增强，得到的收益通常大于换一个更大的基础模型。

14.7 选型决策指南

总结一下，在实际工程中选择主干网络时，建议遵循以下决策流程：

  1. 明确约束：你的部署平台是什么（CPU/GPU/移动端/边缘设备）？延迟要求是多少？模型大小限制是多少？
  2. 确定精度目标：你的任务需要什么级别的精度？如果是"够用就行"（比如内部工具），选择一个轻量模型即可。如果要在基准上超越SOTA，需要大模型+大数据+长训练。
  3. 选择架构族：如果你需要快速迭代和稳定的支持，选择成熟架构（ResNet、ConvNeXt）。如果你在做前沿研究，选择当前SOTA架构。
  4. 做小规模对比实验：在缩小版的数据集上跑2-3个候选模型的快速对比，不要一开始就全量训练。
  5. 在目标硬件上测实际延迟：在不同batch size下测试实际延迟，结合你的QPS（每秒查询数）需求做决策。
  6. 考虑维护成本：选择团队中大多数人熟悉的架构。在一个四人小团队里，使用大家都会的ResNet往往比使用最新的Swin Transformer更明智——因为你不能保证四年后这个项目还有人在维护。
⚠️ 踩坑故事：在一次竞标项目中，我们的算法团队被甲方要求使用"最新最好的模型"。我们花了大量精力把几个检测任务的backbone从ResNet换成了Swin Transformer。结果一个月后，甲方要求把模型部署到他们的边缘设备（Jetson Nano）上——Swim-T在Jetson上的推理速度只有不到2 FPS，完全不可用。我们不得不连夜把所有backbone换回ResNet-18（15 FPS），才勉强交差。这个故事的教训是：选型时不要追求"最好"的模型，而要追求"最合适"的模型——"最好"是对你的约束条件而言的，不是对基准排行榜而言的。
