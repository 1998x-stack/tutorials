第23章：性能诊断工具链

大家好，我是你们的讲师。想象这样一个场景：你在凌晨两点被 on-call 电话叫醒，用户反馈线上 Redis 的 P99 延迟从平时的 1ms 飙到了 500ms。你登录到服务器，redis-cli 敲了一个 PING，返回了 "PONG"——所以 Redis 还活着。然后呢？面对一个复杂的生产系统，从哪个指标开始查？用什么工具定位根因？

很多运维同学遇到这类问题的第一反应是"重启大法"，或者盲目地增大 maxmemory、调整 hz 参数。但实际上，Redis 自带了一整套从应用层到系统层的诊断工具链——从 INFO 命令输出的数十个指标，到 latency monitor 的毛刺采样，再到 slowlog 的命令级别耗时统计。这一章，我们就把这些工具的源码原理和实战用法讲透。一旦你理解了每个指标"从哪里来、怎么算的"，诊断问题就不再是拍脑袋，而是一条逻辑清晰的推理链。

诊断工具 适用场景 数据来源 定位精度
INFO 命令 全局健康检查、性能基线 server.c 运行时计数器 秒级宏观
LATENCY 系列 延迟毛刺、周期性抖动 latency.c 采样钩子 微秒级
SLOWLOG 命令级慢查询 slowlog.c 命令耗时统计 命令粒度
redis-benchmark 压测、容量规划 客户端工具（独立进程） QPS/延迟分布
perf + FlameGraph CPU 热点、内核态阻塞 Linux perf_events 函数级
MEMORY 系列 内存异常、碎片分析 zmalloc.c 统计 + object.c 键级

INFO 命令全解

INFO 命令是 Redis 诊断的"第一站"。它的实现代码在 server.c 的 infoCommand() 函数中——本质上就是一个大型的 sprintf，把 server 结构体中的各种计数器和配置项格式化输出。我们可以通过 INFO ALL 获取全部信息，也可通过 INFO server、INFO memory 等按 section 查询。

我们来逐一拆解每个 section 的关键指标和对应的源码出处：

Server Section

记录了 Redis 进程的基本信息，包括 uptime_in_seconds（进程存活时长）、config_file（配置文件路径）、tcp_port、server_id 等。其中最有诊断价值的是 uptime_in_seconds——如果这个值很短，说明 Redis 刚发生过重启，那最近的性能抖动很可能是重启后的预热（page cache 冷、client 重连 storm）导致的。

Clients Section

connected_clients 是当前连接数，client_longest_output_list 是输出缓冲区最大的 client 的 pending 数据量。这个字段尤其值得关注——如果它持续增长，说明某个 consumer 消费能力跟不上，输出缓冲区在无限膨胀，最终会触发 client-output-buffer-limit 断开连接。对应的源码逻辑在 networking.c 的 clientsCronHandleTimeout() 和 writeToClient() 中。

Memory Section

# INFO memory 关键指标
used_memory:             # zmalloc_used_memory() 实时统计
used_memory_rss:         # RSS，进程物理内存占用
mem_fragmentation_ratio: # RSS / used_memory，碎片率
used_memory_peak:        # 历史峰值
total_system_memory:     # 系统总内存
maxmemory:               # 配置的最大内存限制

mem_fragmentation_ratio 是最容易被误解的指标。很多同学看到比值大于 1.5 就紧张，但事实上 Redis 的 zmalloc 统计的是"实际申请的内存"，而 RSS 包含了内存碎片、jemalloc 的 arena 预分配、以及共享库的代码段。jemalloc 倾向于预分配大块内存并切分使用，所以 1.1~1.3 是正常范围，超过 1.5 才需要关注。如果小于 1.0，说明操作系统发生了 swap——这比碎片严重一百倍。

小贴士：INFO 的每个 section 都对应 server.c 中一个 genRedisInfoString() 的子函数。如果你想添加自定义指标，比如某个业务 key 的命中率，可以直接在对应的 gen 函数中追加一行 info = sdscatprintf(info, "my_metric:%d\r\n", val);

Stats Section

这是查询性能问题的黄金入口。重点关注：total_commands_processed（QPS 推算）、instantaneous_ops_per_sec（瞬时 OPS）、total_connections_received（连接速率，突增可能预示断线重连风暴）、keyspace_hits / keyspace_misses（缓存命中率）、expired_keys（过期淘汰速率）、evicted_keys（内存淘汰速率，如果 > 0 说明 maxmemory 太小）。

Replication / CPU / Cluster / Keyspace

Replication section 中的 master_repl_offset 和 slave_lag 是判断复制延迟的依据。CPU section 的 used_cpu_sys 和 used_cpu_user 可用来判断是否有异常的系统调用开销。Keyspace section 展示每个 db 的 key 数量和过期 key 数量——如果某个 db 的 key 数量突然暴涨，那可能是业务侧的缓存 key 未设过期时间，会导致 expired_keys 统计归零，内存只增不减。

Latency Monitor

Latency monitor 是 Redis 中一个非常精巧的"延迟显微镜"，它的源码在 latency.c 中，核心思想是：在关键路径上插入采样钩子，记录事件循环中超过阈值的延迟毛刺。启用方式很简单：

# 设置延迟采样阈值（微秒），超过此值的操作会被记录
CONFIG SET latency-monitor-threshold 100

# 查看延迟诊断报告
LATENCY DOCTOR

# 查看延迟事件的原始采样数据
LATENCY HISTORY command

# 查看延迟事件的时空分布图（ASCII 图表）
LATENCY GRAPH

我们来看它的源码实现。在 ae.c 的事件循环中，antirez 插入了关键的延迟测量点：

/* ae.c 中的延迟采样 —— 位于 aeProcessEvents() 中 */
int aeProcessEvents(aeEventLoop *eventLoop, int flags) {
    int processed = 0, numevents;

    if (eventLoop->maxfd != -1 ||
        ((flags & AE_TIME_EVENTS) && !(flags & AE_DONT_WAIT))) {
        int j;
        struct timeval tv, *tvp;
        long long latency;

        /* 记录进入 epoll_wait 前的时间戳 */
        latencyStartMonitor(latency);

        /* epoll_wait 等待事件就绪 */
        numevents = aeApiPoll(eventLoop, tvp);

        /* 记录 epoll_wait 返回后的时间戳，计算差值 */
        latencyEndMonitor(latency);
        latencyAddSampleIfNeeded("eventloop", latency);

        /* 如果 epoll_wait 的延迟超过阈值，记录调用栈 */
        if (latency > server.latency_monitor_threshold) {
            latencyMonitorEvent("eventloop", latency, NULL);
        }
    }
    // ... 处理文件事件、时间事件
}

这里的 latencyStartMonitor 和 latencyEndMonitor 本质上就是读取 clock_gettime(CLOCK_MONOTONIC) 或 gettimeofday() 来计算微秒级耗时。latencyAddSampleIfNeeded 会检查耗时是否超过 latency-monitor-threshold，只有超过的样本才被记录到 server.latency_events 字典中。

每个延迟事件类型都被组织成一个链表，记录了最近 160 个采样的时间戳和延迟值，以及触发时的事件循环上下文。LATENCY GRAPH 用 ASCII 字符画出这些样本的分布，可以直观地看出延迟毛刺的周期性——如果是固定时间间隔出现，那大概率是 cron 任务（过期键清理、AOF 刷盘、rehash）导致的；如果是突发的无规律毛刺，则可能是系统层面的抖动（swap、内核调度、THP 的 page 合并）。

诊断技巧：当你发现 "eventloop" 延迟高时，不要马上断定是 Redis 的问题。在 aeApiPoll() 返回前，程序是阻塞在内核的 epoll_wait 上的——如果这段时间很长，说明没有事件到达，而不是事件处理慢。真正的问题往往出在 command 延迟事件上，它表示一个命令从解析完成到执行结束的耗时。

Slowlog

Slowlog 是最简单也最常用的命令级诊断工具。它的源码在 slowlog.c 中，核心数据结构是 slowlogEntry：

/* slowlog.c —— slowlogEntry 结构体 */
typedef struct slowlogEntry {
    robj **argv;            /* 命令参数数组 */
    int argc;               /* 参数个数 */
    long long id;           /* 自增的唯一 ID */
    long long duration;     /* 命令执行耗时（微秒） */
    time_t time;            /* 命令进入时间（unix 秒戳） */
    long long curenterr;   /* 当前 errno */
    sds repr;               /* 命令的字符串表示 */
} slowlogEntry;

每条慢查询记录包含命令参数、执行时长（微秒）、时间戳和命令的字符串表示。Redis 使用一个固定长度的链表来存储这些记录，长度由 slowlog-max-len 控制——超过长度时淘汰最旧的记录。慢查询的阈值由 slowlog-log-slower-than 控制，单位微秒，默认 10000（10ms）。

关键代码在命令执行完成的路径上：

/* server.c —— call() 函数中，命令执行后进行慢查询判定 */
void call(client *c, int flags) {
    long long dirty;
    ustime_t start, duration;
    int client_old_flags = c->flags;

    /* 记录命令开始执行的时间戳 */
    start = server.ustime;
    c->cmd->proc(c);              /* 执行命令 */
    duration = ustime() - start;  /* 计算耗时 */

    /* 如果耗时超过阈值，记录 slowlog */
    if (duration >= server.slowlog_log_slower_than) {
        slowlogPushEntryIfNeeded(c, c->argv, c->argc, duration);
    }
    // ...
}

这个代码揭示了一个容易被忽视的事实：slowlog 记录的是从命令处理函数开始到结束的耗时，不包含网络 IO 和协议解析的时间。所以如果 slowlog 里显示某个 SET 命令耗时 50ms，那一定不是 SET 本身慢（SET 是 O(1) 的），而是另有原因。我们来看看一个真实的踩坑案例：

踩坑实录：曾经有一个团队反馈线上 Redis 的 slowlog 中频繁出现 SET 命令，时长在 30~100ms 之间。SET 明明是 O(1) 操作，怎么会慢到 100ms？团队排查了硬盘 IO、网络延迟、系统负载，都找不出原因。最后我让他们用 LATENCY HISTORY command 和 LATENCY HISTORY eventloop 对比时间段，发现每当有 BGSAVE 执行时，SET 的慢查询就会集中出现。打开源码一查，真相是：BGSAVE 的 fork() 之后，Linux 的 COW（Copy-On-Write）机制导致 Redis 主进程写入任何内存页时都产生页面复制。在写密集型场景下，COW 开销被成倍放大——SET 命令本身不慢，但 SET 触发的内存写入触发了 COW 页面复制。解决方案是：关闭 THP、调整 repl-backlog-size、将 BGSAVE 安排在低峰期。所以，slowlog 告诉你"什么命令慢"，但不告诉你"为什么慢"——你需要结合其他工具做交叉验证。

从资源消耗的角度，我们可以把慢查询分为两类：

  • 计算密集型的慢命令：KEYS *、SORT、ZRANGEBYSCORE 等 O(N) 或 O(log N + N) 的命令，慢的原因是 CPU 计算量过大。解决思路是控制集合大小、使用 SCAN 替代 KEYS、或者用 SORT 的 LIMIT 参数限制返回数量。
  • IO 密集型的慢命令：SWAPDB、FLUSHALL、BGSAVE、AOF rewrite 等命令，慢不是因为 CPU 计算，而是因为 IO 操作（fsync、write、fork）阻塞了事件循环。解决思路是：FLUSHALL ASYNC、控制 AOF rewrite 的频率、replica-serve-stale-data yes。

redis-benchmark

redis-benchmark 是 Redis 自带的压测工具，位于 src/redis-benchmark.c。它不是一个运行时的诊断工具，而是在部署前做容量规划、在变更后做性能回归验证的利器。基本用法：

# 基本压测：100个客户端并发，50000个请求
redis-benchmark -h 127.0.0.1 -p 6379 -n 50000 -c 100

# 测试特定命令的吞吐
redis-benchmark -t SET,GET,LPUSH -n 100000

# 以管道方式压测（模拟批量请求）
redis-benchmark -n 100000 -P 10

# 集群模式压测
redis-benchmark --cluster -h 127.0.0.1 -p 7000

# 测试某个 key 的内存占用
redis-cli MEMORY USAGE mykey
redis-cli MEMORY STATS

输出示例（不同硬件差异较大，以下仅做参考）：

====== SET ======
  100000 requests completed in 1.24 seconds
  100 parallel clients
  3 bytes payload
  keep alive: 1

99.88% <= 0.5 milliseconds
100.00% <= 1 milliseconds
80580.29 requests per second

====== GET ======
  100000 requests completed in 1.22 seconds
  100 parallel clients
  3 bytes payload
  keep alive: 1

99.92% <= 0.5 milliseconds
100.00% <= 1 milliseconds
82015.36 requests per second

值得关注的几个参数组合：

  • payload size 的影响：使用 -d 1024 测试 1KB 的 value 和默认 3 字节相比，QPS 可能下降 20-30%，因为网络带宽和内存分配成了瓶颈。
  • Pipeline 深度：-P 10 开启管道可以大幅提升吞吐（5~10 倍），但这不代表生产环境的真实 QPS——生产环境的请求是随机到达的，没法无限制 pipeline。
  • 集群压测：--cluster 模式下会先通过 CLUSTER SLOTS 获取槽位分布，然后对每个节点分别建连压测。

perf 与火焰图

在诊断 CPU 相关的性能问题时，INFO 和 slowlog 只能告诉我们"命令慢"，而 perf 配合火焰图能告诉我们"哪个函数吃了最多的 CPU"。这是源码级问题定位的大杀器。

基本流程如下：

# 第一步：采集 Redis 进程的 CPU 样本
# -a: 采集所有 CPU
# -p: 指定 pid
# -g: 记录调用栈（关键！）
perf record -a -g -p `pgrep redis-server` -- sleep 30

# 第二步：生成 perf 报告
perf report -g --stdio

# 第三步：生成火焰图
# 需要从 GitHub 下载 FlameGraph 工具集
git clone https://github.com/brendangregg/FlameGraph.git

# 生成 folded stack 格式
perf script > out.perf
./FlameGraph/stackcollapse-perf.pl out.perf > out.folded
./FlameGraph/flamegraph.pl out.folded > redis-cpu.svg

用浏览器打开生成的 redis-cpu.svg 文件，你会看到从底向上的调用栈火焰图。每个方块的宽度代表该函数占用的 CPU 样本比例。重点关注那些"又宽又靠上"的函数——它们就是 hot function。

结合我们学过的源码，常见的 CPU 热点包括：

  • dictFind / dictAdd：如果占比较高，说明有大量的 key 操作，同时注意观察 dictRehash 是否也占用了显著比例——如果是，说明 rehash 正在进行中（可考虑增大 hash-max-ziplist-entries 或合理规划 key 数量）。
  • rdbSave / rdbWriteRaw：BGSAVE 导致的 CPU 飙升，应该错开持久化窗口。
  • aeProcessEvents / aeApiPoll：占比高说明事件循环空转少，CPU 利用率高——这是好事，如果占比低反而说明大部分时间在 epoll_wait 空闲等待。
  • expireCycle / activeExpireCycle：过期键集中清理占 CPU，说明键的 TTL 集中相同，可能导致延迟抖动。
  • zmalloc / jemalloc 相关函数：如果内存分配函数占总 CPU 超过 10%，说明数据操作的密集程度很高，或者分配器的参数需要调整。
实战排查案例：某游戏后端 Redis CPU 持续 90% 以上，slowlog 没有明显慢查询，perf 火焰图显示 expireCycle 占用 47% 的 CPU。翻看源码 expire.c 的 activeExpireCycle() 后发现，Redis 每次只扫描 20 个过期键，但如果有大量过期键集中在同一秒过期（业务方给 100 万个 key 设置了完全相同的 TTL），Redis 就需要不断轮询，导致 CPU 飚高。最终将 TTL 进行随机化处理（在原始 TTL 上加上一个随机偏移），问题立即解决。

诊断思维框架

工具再多，没有系统化的诊断思维也等于零。这里分享一个我总结的"问题定位三步法"：

第一步：现象 → 指标

把用户反馈的"感觉比较慢"转化为可测量的指标。具体来说：用 INFO 采集基线数据，用 slowlog 看是否有个别命令极慢，用 latency monitor 看是否有周期性毛刺。这一步的目标是把"慢"精确到某个维度：是 CPU 不够？是 IO 卡顿？还是内存有问题？

第二步：指标 → 模块

确定了问题维度后，对照源码中的对应模块进行定位。比如 CPU 高的问题，用 perf 看热点函数是哪个模块的；IO 卡顿的问题，用 strace 看文件系统和网络系统调用的耗时分布；内存异常的问题，用 MEMORY STATS 分析数据结构层面的内存分布。

第三步：模块 → 根因

定位到具体模块后，深入源码阅读对应的逻辑分支。比如发现是 freeMemoryIfNeeded() 占用了大量 CPU，就要阅读 evict.c 中内存淘汰算法的执行路径——是 LRU 采样的开销，还是逐出大量 key 导致的写操作连锁反应？

在问题分类上，我们需要区分两类常见的性能故障：

  • 延迟类问题：表现为 P99 剧增、命令响应慢。根因往往是单次操作的耗时过长（COW、THP、大 key 操作、fork 阻塞）。定位工具：latency monitor → slowlog → perf。
  • 吞吐类问题：表现为 QPS 下降、CPU 打满。根因往往是某些操作过于频繁（过期键密集清理、rehash 抖动、断线重连风暴）。定位工具：INFO stats → MEMORY STATS → redis-benchmark 对照。

最后，画一张诊断决策流程图给自己贴在墙上：
