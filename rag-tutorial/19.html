第十九章：多模态RAG

大家好，我是你们的讲师。在之前的章节中，我们一直默认一个问题：RAG的知识库由纯文本构成。但在2026年的今天，这个假设已经严重过时了。真实世界的企业知识库中充斥着图片、图表、表格、流程图、产品照片、音频记录和视频内容。一份2025年的行业报告显示，企业文档中超过60%的"知识"是以非文本形式呈现的——产品手册中的爆炸图、财报中的趋势图、会议中的白板照片、培训视频的截图。

想象一下这个场景：你正在为一个医疗诊断辅助系统构建RAG。一位医生上传了一张X光片，询问"这个病人的肺部阴影可能是什么问题？" 传统的文本RAG完全无法处理——它既不能理解图像内容，也不能根据图像特征检索相关的病历和文献。这正是多模态RAG要解决的挑战。我们将从CLIP模型出发，讨论如何构建一个能够同时理解文本和图像的检索系统，并探讨表格、图表和音频等更多模态的处理策略。

9.1 CLIP模型与图像-文本联合检索

OpenAI的CLIP（Contrastive Language-Image Pre-training）是多模态RAG的基石模型之一。CLIP的核心思想非常优雅：将图像和文本映射到同一个语义空间中，使得相关的图像-文本对在向量空间中距离更近。通过这种方式，我们就可以用文本查询来检索相关的图像——这正是多模态RAG所需要的核心能力。

以下是使用CLIP模型构建图像-文本联合检索的示例：

import torch
from PIL import Image
import clip
import numpy as np
from pathlib import Path

# 加载CLIP模型（ViT-B/32是速度和性能的良好平衡）
device = "cuda" if torch.cuda.is_available() else "cpu"
model, preprocess = clip.load("ViT-B/32", device=device)

class MultimodalIndex:
    """
    多模态索引：支持通过文本查询检索图像。
    """
    def __init__(self):
        self.image_embeddings = []
        self.image_paths = []

    def index_images(self, image_dir: str):
        """
        批量索引指定目录下的所有图像。
        """
        image_dir = Path(image_dir)
        for img_path in image_dir.glob("*.png"):
            image = preprocess(Image.open(img_path)).unsqueeze(0).to(device)
            with torch.no_grad():
                embedding = model.encode_image(image)
                embedding = embedding / embedding.norm(dim=-1, keepdim=True)
            self.image_embeddings.append(embedding.cpu().numpy())
            self.image_paths.append(str(img_path))

    def search(self, query: str, top_k: int = 5):
        """
        使用文本查询检索最相关的图像。
        """
        # 编码文本查询
        text = clip.tokenize([query]).to(device)
        with torch.no_grad():
            text_embedding = model.encode_text(text)
            text_embedding = text_embedding / text_embedding.norm(
                dim=-1, keepdim=True
            )

        # 计算与所有图像嵌入的余弦相似度
        similarities = []
        for img_emb in self.image_embeddings:
            sim = (text_embedding.cpu().numpy() @ img_emb.T).item()
            similarities.append(sim)

        # 返回top-k结果
        top_indices = np.argsort(similarities)[::-1][:top_k]
        results = []
        for idx in top_indices:
            results.append({
                "path": self.image_paths[idx],
                "score": similarities[idx]
            })
        return results

# 索引一些产品照片
index = MultimodalIndex()
index.index_images("./product_images/")

# 搜索包含特定内容的图像
results = index.search("红色外观的电子产品")
for r in results:
    print(f"匹配图像: {r['path']} (相似度: {r['score']:.3f})")
    # 可以将匹配的图像路径传递给VLM进行进一步的视觉问答

9.2 表格检索策略对比

表格是多模态RAG中最常见也最棘手的数据类型之一。表格中的信息既有结构化的部分（行、列、数值关系），又有非结构化的部分（表头描述、单元格文本）。处理表格有多种策略，各有优劣：

策略 方法描述 检索质量 实现复杂度 典型延迟
纯文本转换 将表格转为Markdown/CSV文本，用文本嵌入检索 中 低 低
结构化描述 用LLM生成表格的自然语言摘要后索引 高 中 中高
SQL + 向量混合 表名/描述用向量检索，数据用SQL精确查询 极高 高 中
Table Embedding 使用专门的表格嵌入模型（如TAPAS） 高 高 中
多模态VLM直出 将表格截图后直接用VLM理解 中高 低 高

在工业实践中，我们推荐混合策略：对于简单的键值表，用结构化描述就足够了；对于需要精确计算的大型结构化表格，SQL+向量混合是最可靠的方式；对于PDF中的格式复杂的表格，纯文本转换配合精确的解析器是最稳妥的选择。

9.3 多模态生成——VLM的集成

检索到相关的图像后，下一步是让模型能够"看懂"这些图像并基于图像内容生成回答。这需要视觉语言模型（VLM）的参与。在2026年，主流的选择包括GPT-4V、Claude 3.5 Vision、和开源的LLaVA-NeXT或Qwen-VL系列。

多模态RAG的典型生成流程是：

  1. 用户提交一个可能包含图像查询的多模态问题（例如一张照片+文字描述）。
  2. 检索阶段：使用CLIP同时索引文本和图像，找到最相关的文档和图像。
  3. 融合阶段：将检索到的文本段落和图像路径一起组装成多模态上下文。
  4. 生成阶段：将多模态上下文和原始问题一起交给VLM，生成最终回答。

关键挑战在于"对齐"——图片中的信息如何与文本段落中的信息相互补充而不是矛盾。例如，一张产品照片显示"红色外观"，但对应的产品说明书说"有红色和蓝色两种版本"。系统需要理解这两者并不矛盾，而是互补关系。

9.4 多模态RAG的工程挑战

多模态RAG在工程实现上比纯文本RAG复杂得多，主要体现在以下几个方面：

存储成本： 图像嵌入通常比文本嵌入维度更高（CLIP ViT-L的嵌入维度是768，但一张图像压缩后仍有数KB），多模态索引的存储成本是纯文本的10-50倍。

延迟管理： VLM的推理延迟远超纯文本LLM。GPT-4V处理一张图像的额外延迟通常在1-3秒。建议对多模态查询和非多模态查询使用不同的路由策略。

一致性评估： 如何评估多模态RAG的输出质量？忠实度指标需要扩展到图像层面——需要检查生成的描述与原始图像内容是否一致，这在技术上远比文本对比困难。

检索精度： CLIP在处理细粒度视觉特征时（如区分两种相似的汽车型号）表现不佳。对于需要精确视觉识别的场景，建议结合专门的分类模型或OCR工具进行预处理。

2026年的行业现状：多模态RAG已是标配而非选项。 截至2026年中，几乎所有主流RAG框架（LlamaIndex、LangChain、Haystack）都已内置了多模态支持。在用户调查中，超过70%的企业RAG项目涉及至少一种非文本模态。如果你的RAG系统至今仍只支持纯文本，你的竞争力可能已经落后了。不过，不要追求"一次支持所有模态"——先从最常见的表格和图片入手，逐步扩展到图表和流程图，最后再考虑音频和视频。从文本RAG到多模态RAG是一次架构升级，而非简单的插件配置，需要投入足够的工程资源。

9.5 小结与下章预告

多模态RAG将RAG的能力从文本世界扩展到了视觉世界。我们学习了CLIP模型的图像-文本联合检索方法，讨论了表格处理的多种策略，并了解了VLM在多模态生成中的作用。尽管多模态RAG在存储、延迟和评估方面带来了新的挑战，但它已经从一个实验性功能变成了企业RAG系统的标配能力。如果你想了解更前沿的方向——如何将RAG与知识图谱结合起来，实现超越向量检索的推理能力——请继续阅读下一章：GraphRAG与知识图谱增强。
