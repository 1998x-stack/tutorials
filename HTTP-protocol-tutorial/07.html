第七章：MIME类型与内容协商

欢迎来到第七章。上一章我们全面了解了HTTP头部字段，这一章我们聚焦一个非常具体的主题：MIME类型和内容协商。这两个概念是Content-Type系列头部的核心，也是浏览器和服务器之间沟通"用什么格式传输数据"的桥梁。

你可能有过这样的经历：明明访问的是同一个URL，但浏览器和curl返回的数据格式不一样。或者上传图片时报错"文件格式不支持"——这些都跟MIME类型有关。今天我们就来把这个知识点彻底讲透。

7.1 什么是MIME类型

MIME的全称是 Multipurpose Internet Mail Extensions（多用途互联网邮件扩展）。没错，它最早是为电子邮件设计的——用来在邮件中传输非ASCII字符、图片、音视频等附件。后来HTTP沿用了这个机制，用来标识传输内容的格式。

MIME类型的结构是：type/subtype，也就是"大类/小类"的格式。例如 text/html 表示"文本大类下的HTML子类"。

  • type：大类，如 text、image、audio、video、application、multipart、message 等。
  • subtype：小类，具体格式，如 html、png、json、octet-stream 等。
  • 还有可选的参数，如 charset=utf-8 或 boundary=----xxx。

所有标准化的MIME类型由 IANA（互联网号码分配机构）维护和注册。目前IANA注册了上千种MIME类型，从 application/vnd.openxmlformats-officedocument.spreadsheetml.sheet（Excel文件）到 text/vcard（电子名片），非常丰富。

核心认知：MIME类型是HTTP理解"这个数据是什么"的关键。服务器通过 Content-Type 头告诉浏览器响应体的格式，浏览器据此决定如何渲染或处理。如果你传了一个错误的MIME类型，浏览器就会用错误的方式处理数据——最常见的后果就是"乱码"或"下载而不是显示"。

7.3 Content-Type头

Content-Type 是Entity Header（实体头），出现在请求和响应中，告诉对方发送的数据是什么格式。

它的完整语法是：

Content-Type: type/subtype; parameter=value

三个常见用法：

1. charset参数——指定字符编码。比如 Content-Type: text/html; charset=utf-8。这个参数对于text类型尤其重要，告诉浏览器用什么编码解读文本内容。如果没有charset，浏览器就会用默认编码（通常是系统的编码），就容易出现乱码。

2. boundary参数——用于multipart类型。比如：

Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW

boundary是一个分隔符字符串，把不同部分的数据隔开。浏览器在发送含文件上传的表单时自动生成。很多人在写爬虫或自动提交表单时被这个搞到崩溃——boundary必须和请求体里的完全一致。

3. MIME嗅探风险——当Content-Type缺失或与实际内容不匹配时，浏览器会尝试"嗅探"内容类型。比如你返回了一个 text/plain 但实际内容是HTML，浏览器仍然可能将其渲染为页面。这种行为有安全风险——攻击者可能上传一个看似图片的文件，但实际包含JavaScript，绕过安全检查后导致XSS攻击。

踩坑故事：有个同事做文件上传功能，只校验了文件的扩展名，没验证Content-Type。结果用户上传了一个 .txt 文件，但实际上内容是HTML + JavaScript。服务器把Content-Type设置为 text/plain，但浏览器嗅探后发现里面有HTML标签，强行当作HTML渲染了——XSS漏洞就这样产生了。解决方案是返回 X-Content-Type-Options: nosniff 头，禁止浏览器嗅探，同时服务器端严格校验文件内容。

7.4 内容协商机制

内容协商（Content Negotiation）是HTTP的一个优雅设计：同一个URL，可以根据客户端的偏好返回不同格式的内容。

举个例子：访问 https://api.example.com/user/123，有的客户端想要JSON格式，有的想要XML格式。服务端支持返回两种格式，那它怎么知道该给哪个？这就靠内容协商。

内容协商有两种主要方式：

1. 服务端驱动（Server-driven）——客户端通过请求头表达偏好，服务端根据这些偏好返回最合适的格式。

  • Accept：告诉服务器客户端期望的MIME类型
  • Accept-Language：期望的语言（zh-CN、en-US等）
  • Accept-Encoding：期望的内容编码（gzip、br等）
  • Accept-Charset：期望的字符编码（已被大多数浏览器弃用）

2. 主动协商（Agent-driven）——服务器返回300 Multiple Choices 或 406 Not Acceptable，让客户端自己选择。

在实践中，绝大多数内容协商是服务端驱动的。浏览器自动发送Accept系列头，服务器读取后决定返回什么格式。

7.5 Accept系列头实战

Accept头最精妙的设计是质量值（Quality Values，q=）：

Accept: text/html, application/xhtml+xml, application/xml;q=0.9, */*;q=0.8

这条头的意思是：我最想要 text/html（默认q=1.0），其次接受 application/xhtml+xml（也默认q=1.0），然后是 application/xml（q=0.9，90%的意愿），最后什么都行（*/*，q=0.8）。

通配符：

  • */*：匹配所有MIME类型
  • text/*：匹配所有text子类型
  • application/*：匹配所有application子类型

Accept-Language实战：

Accept-Language: zh-CN,zh;q=0.9,en;q=0.5

首选简体中文（zh-CN），其次任何中文（zh），最后是英文。browser默认发送的语言偏好基于你的操作系统和浏览器语言设置。

Accept-Encoding实战：

Accept-Encoding: gzip, deflate, br

告诉服务器客户端支持gzip、deflate、brotli三种压缩算法。服务器通常选择gzip或brotli（后者压缩率更高，现代浏览器都支持）。

接口调试技巧：测试一个API是否支持内容协商，可以用curl指定不同的Accept头：

curl -H "Accept: application/json" https://api.example.com/user/1
curl -H "Accept: text/html" https://api.example.com/user/1

看看返回格式是否不同。很多框架（如Spring、Express、Django REST Framework）都原生支持内容协商，配置一下就能让一个接口同时服务前端和API调用。

7.6 字符编码——中文乱码的根源

字符编码跟MIME类型关系密切，尤其是文本类型的响应。这里我想重点聊聊中文开发者最常遇到的问题——乱码。

现在UTF-8已经是绝对的主流。根据W3Techs统计，超过98%的网站使用UTF-8编码。但如果你遇到乱码，排查路径通常是：

  1. 第一步：检查Content-Type头——响应头里有没有 charset=utf-8？不声明charset的话，浏览器只能用猜测的。
  2. 第二步：检查HTML/CSS中的meta声明——HTML中 <meta charset="utf-8"> 有没有写？CSS文件开头的 @charset "utf-8" 有没有？
  3. 第三步：检查文件实际编码——文件真的以UTF-8保存的吗？有时候编辑器把它存成了GBK或Latin-1。
  4. 第四步：检查数据库连接编码——如果数据从数据库读取，连接字符串里有没有加 ?useUnicode=true&characterEncoding=utf-8？

踩坑故事：好几年前做一个博客系统，中文标题在本地开发环境好好的，上了服务器就变成"????????"。查了一下午，发现是数据库连接配置里没指定UTF-8编码。MySQL默认用Latin-1存数据，中文词变成问号存入，读出来自然也是问号。教训是：任何一个环节的编码不一致，就会导致乱码——文件编码、HTTP响应头、数据库连接、数据库表字符集，这四个地方要全部统一为UTF-8。
