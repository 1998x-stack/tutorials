第14章 5G RRC与信令流程

大家好，我是你们的讲师。前面几章我们走过了物理层的信道和信号、MAC层的调度、RLC层的分段重传——这些都是协议栈的"执行层"。今天我们来聊协议栈中真正负责"管理"的层——RRC（Radio Resource Control，无线资源控制层）。

RRC层是空口协议栈的控制中枢。用户是否连接到小区、用什么配置、什么时候切换、什么时候进入省电模式——所有这些"决策"都由RRC层做出。如果把整个空口协议栈比作一家公司，物理层是工人，MAC层是车间主任，RLC层是质检员，那么RRC就是CEO——它不直接干活，但每一个重大决策都由它拍板。

在5G NR中，RRC层相比4G LTE做了重大革新：引入了RRC_INACTIVE状态（兼顾省电和快速恢复），简化了状态转换，增强了配置的灵活性（增量配置、小区组配置等）。理解RRC层和信令流程，是从事5G网络优化和问题排查的必备技能。

14.1 RRC三种状态与状态转换

5G NR定义了三种RRC状态，UE在不同状态下有不同的行为和能力：

  • RRC_IDLE：UE在小区选择/重选状态，没有与任何gNB建立RRC连接。监听寻呼（周期性的DRX），可以接收广播消息（SIB1等）。没有专用的无线资源，省电效果最好。
  • RRC_CONNECTED：UE与gNB建立了RRC连接，有完整的UE上下文。可以收发数据，基站知道UE在小区级别的精确位置。UE监听PDCCH获取调度，功耗较高。
  • RRC_INACTIVE：5G NR新增的中间状态。UE保持CM-CONNECTED（核心网连接），保留UE上下文在gNB和UE两端，但暂停了RRC连接。UE不监听PDCCH，更省电，同时可以快速恢复（Resume）到CONNECTED状态。

状态转换的关系如下：

  • RRC_IDLE ↔ RRC_CONNECTED：通过RRC Setup（建立）或RRC Release（释放）
  • RRC_CONNECTED ↔ RRC_INACTIVE：通过RRC Release with SuspendConfig（挂起）或RRC Resume（恢复）
  • RRC_INACTIVE ↔ RRC_IDLE：通过RRC Release（释放到IDLE）
  • RRC_INACTIVE → RRC_CONNECTED：通过RRC Resume完成快速恢复

为什么要有INACTIVE？手机大部分时间是不传数据的——你刷微信，90%的时间在看，只有10%的时间在收发。如果每次不传数据都回到IDLE，下次发消息要重新注册，时延几百ms；如果一直保持CONNECTED，电池半天就耗尽。INACTIVE状态就是"半连接"——留住了核心网连接，暂停了空口连接，既省电又快速恢复。实测从INACTIVE到CONNECTED的恢复时延约5-15ms，远低于IDLE到CONNECTED的50-200ms。

14.2 小区搜索与同步

UE开机后，从一片空白到成功连接网络，第一步是小区搜索。这个过程是物理层和RRC层协作完成的：

  1. SSB检测：UE在预设的同步栅格（Sync Raster）上扫描SSB。检测到PSS/SSS后，获取物理小区ID（PCI），完成时频同步。
  2. MIB读取：UE解码PBCH获取MIB。MIB共56比特，包含：系统帧号（SFN的10个最高有效位）、SSB的子载波间隔、SSB的位置（对于Sub-3GHz频段）、CORESET 0的配置（用于接收SIB1）。
  3. SIB1获取：根据MIB中CORESET 0的配置监听PDCCH，接收PDSCH上的SIB1（System Information Block 1）。SIB1包含：公用陆地移动网（PLMN）列表、TAC（跟踪区码）、Cell Identity、随机接入参数、其他SIB的调度信息。

完成以上三步后，UE知道了小区的所有基础配置，具备了发起随机接入的前提条件。

14.3 随机接入：CBRA与CFRA

随机接入是UE从"听众"转变为"发言人"的第一步。5G NR支持两种随机接入方式：

特性 CBRA（基于竞争的随机接入） CFRA（非竞争随机接入）
触发场景 初始接入、RRC重建、RRC_INACTIVE恢复 切换（Handover）、波束失败恢复、上行同步
步骤 Msg1-Msg4（四步随机接入） MsgA-MsgB（两步随机接入，即3GPP Rel-16引入）
前导码 从随机接入前导码池中随机选择 基站提前分配的专用前导码
冲突风险 高（多UE选择同一前导码导致冲突） 无（专用前导码唯一）
平均时延 约10-20ms（含冲突解决） 约3-5ms

CBRA的四步流程：Msg1（UE发送Preamble）→ Msg2（gNB回复Random Access Response，包含TA和UL Grant）→ Msg3（UE发送RRC Setup Request）→ Msg4（gNB回复RRC Setup，包含UE Contention Resolution ID）。CFRA的两步流程：MsgA（UE发送Preamble + 数据）→ MsgB（gNB回复成功确认或回退指示）。

性能差异：两步随机接入（CFRA）将传统四步合并为两步，时延降低约50%。3GPP Rel-16引入的CFRA增强版本对于URLLC业务至关重要——在一些工业控制的实测中，两步随机接入的端到端时延可以达到5ms以内。但CFRA需要基站提前分配专用前导码资源，不适合初始接入场景。

14.4 注册流程全链路

当UE完成随机接入并建立了RRC连接后，下一步就是向核心网注册。这是UE从"连上基站"到"接入网络"的关键一步：

  1. RRC Setup Complete：UE向gNB发送RRC Setup Complete消息，其中携带NAS（非接入层）消息——Registration Request。gNB选择AMF后，通过N2接口（NGAP协议）将Registration Request转发给AMF。
  2. 身份验证：AMF触发UE身份验证流程。AMF向AUSF请求鉴权向量，AUSF从UDM获取鉴权数据。按照5G-AKA或EAP-AKA'流程，AMF向UE发送鉴权请求（包含RAND和AUTN），UE验证网络合法性后回复鉴权响应（RES*）。AMF比较RES*与预期值后确认鉴权通过。
  3. NAS安全激活：鉴权成功后，AMF与UE协商NAS安全参数（加密算法、完整性保护算法），激活NAS安全上下文。之后所有NAS消息都受加密和完整性保护。
  4. UE上下文建立：AMF向gNB发送Initial Context Setup Request，包含UE的AMF-UE-NGAP-ID、UE聚合最大比特率、安全能力等。gNB回复Initial Context Setup Response，确认UE上下文建立成功。
  5. Registration Accept：AMF向UE发送Registration Accept，包含5G-GUTI（临时标识）、TAI列表、网络切片信息等。UE回复Registration Complete。

至此，UE在核心网完成注册，具备了建立PDU会话的能力。

注册失败排查思路：如果UE注册失败，可以从以下维度逐步排查。第一，空口侧：UE是否成功接收到RRC Setup、UE的Registration Request是否成功传输。第二，AMF侧：AMF是否收到Registration Request、UE的SUCI/SUPI是否能被UDM识别。第三，鉴权侧：AUSF是否返回了正确的鉴权向量、UE是否通过鉴权。第四，上下文侧：gNB是否成功建立UE上下文。实际工程中，90%的注册失败都能在前两步定位到原因。

14.5 PDU会话建立

注册只是"登录网络"，真正要上网还得建立PDU（Protocol Data Unit）会话——相当于UE和UPF之间的数据传输通道。PDU会话建立的流程如下：

  1. UE请求：UE向AMF发送NAS消息PDU Session Establishment Request，包含DNN（数据网络名称，如"internet"）、S-NSSAI（网络切片标识）、PDU会话类型（IPv4/IPv6/以太网等）。
  2. SMF选择：AMF根据DNN和S-NSSAI选择合适的SMF（会话管理功能），通过N11接口转发UE请求。
  3. UPF选择：SMF根据DNN、UE位置、网络策略选择合适的UPF实例。如果需要边缘计算，SMF会选择靠近RAN的UPF（本地分流UPF）。
  4. N4会话建立：SMF通过N4接口向UPF下发转发规则——PFCP规则（Packet Forwarding Control Protocol），包括PDR（包检测规则）、FAR（转发动作规则）、QER（QoS执行规则）和URR（用量上报规则）。
  5. DRB建立：SMF通过AMF向gNB发送PDU Session Resource Setup Request，gNB为UE建立数据无线承载（DRB），将N3隧道（gNB-UPF）的GTP-U端点信息回复给AMF。
  6. PDU Session Accept：SMF向UE发送PDU Session Establishment Accept，包含分配给UE的IP地址、QoS规则等。UE收到后即可通过该PDU会话发送/接收数据。

整个过程涉及UE、gNB、AMF、SMF、UPF五个网元的协作，是5GC SBA架构"服务化调用"的典型体现。

14.6 踩坑实录：RRC_INACTIVE状态转换Bug导致寻呼丢失

2024年底，某主流芯片厂商的5G终端芯片在现网测试中发现了一个令人头疼的问题：手机放在桌上不动的时候，经常会有十几分钟收不到微信消息，但一旦点亮屏幕就"轰炸式"地涌进来。用户感知极为糟糕。

工程师让终端芯片进行了数十次跟踪日志分析。比对后发现，出问题的模式非常一致：终端进入RRC_INACTIVE状态后，会周期性唤醒监听寻呼（Paging），但唤醒后有时能收到寻呼，有时收不到。而且收不到寻呼的时间段与终端移动到新gNB的时机高度相关。

深入排查后，根因终于浮出水面：终端芯片的RRC层在处理RRC Release with SuspendConfig（挂起到INACTIVE）后，未能正确更新存储的"寻呼周期"参数。当终端在INACTIVE状态下发生小区重选（移到了相邻小区的覆盖范围），终端应该使用新小区SIB1中广播的默认寻呼周期，但芯片的bug导致它仍然使用旧小区的寻呼周期。新旧小区的寻呼周期不一致，导致终端在错误的时间点唤醒监听寻呼——自然什么都听不到。

教训：RRC_INACTIVE虽然是一个优秀的省电方案，但它引入了一个新的复杂度：UE在INACTIVE状态下可以移动（靠小区重选，不需要切换），但必须正确获取新小区的广播信息。任何参数的"缓存错位"都可能导致严重的寻呼失败。这个bug暴露后，芯片厂商在后续版本中增加了INACTIVE状态下小区重选后的参数校验逻辑——每次唤醒监听从SIB1中重新读取寻呼周期，不再使用缓存值。这个案例告诉我们：协议栈中最容易出bug的地方，往往是"状态转换的边缘"。

// 3GPP TS 38.304: RRC_INACTIVE寻呼周期计算公式
// 寻呼帧(PF): SFN mod T = (T div N) * (UE_ID mod N)
// 寻呼时机(PO): i_s = floor(UE_ID / N) mod Ns
// T: 寻呼周期（从SIB1获取），N: 每周期寻呼帧数
// UE_ID = 5G-S-TMSI mod 1024

// 常见故障：UE小区重选后仍使用旧小区T值
// 旧T=128ms vs 新T=256ms → PO计算全面错位
// 修复：每次唤醒监听前从SIB1重新读取T值

至此，我们走完了5G空口协议栈的全部核心层：从物理层的信道和信号，到MAC层的调度和HARQ，到RLC层的分段和重传，再到RRC层的状态管理和信令流程。这八章（第7-14章）构成了5G协议栈的完整体系。从下一章开始，我们将从"软件协议"走向"硬件工程"——基站的实际硬件架构、部署规范、开通调测和网络优化。第15章，我们来看一台5G基站的内部到底长什么样。
