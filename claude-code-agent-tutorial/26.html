第26章 前沿趋势：Long-Running Agents

大家好，我是你们的讲师。在前面 25 章里，我们搭建的 Agent 处理的任务大多是以"分钟"或"小时"为单位衡量的——跑一个测试、重构一个函数、写一个 CRUD 接口。这些任务从开始到结束，Agent 一直在上下文中保持活跃，你盯着终端，看着它一步步完成。但请你想一个问题：如果任务需要几天甚至几周呢？

2025 年底，我在一个开源项目中尝试用 Agent 做一次完整的代码库迁移——从 Python 2 遗留代码迁移到 Python 3+，涉及 200 多个文件。开始时一切顺利，Agent 热情高涨。但到了第二天，我发现它开始"忘记"之前做出的决策——同一个模块反复重构了三次，每个重构都和前一个矛盾。它开始编造不存在的 API，声称"已经修复"其实根本没触碰过的代码。我意识到，我遇到了当时几乎所有 Agent 都有的天花板：它们是为短跑而生的，不是为马拉松而设计的。

这一章，我们将探讨 AI Coding Agent 最前沿的课题——Long-Running Agents。这不是一个"锦上添花"的话题，而是 Agent 从"玩具"走向"生产力工具"必须跨越的门槛。

26.1 短任务 Agent 的天花板

当我们说一个 Agent 是"短任务优化"的，指的是它的设计假设是：一次会话（session）内完成任务，上下文连续递增，模型始终"在线"。这个假设在大部分场景下成立，但是一旦任务时间拉长，四个问题就会依次浮出水面。

上下文腐烂（Context Rot）：随着对话进行，早期的决策、文件内容、讨论结果被越来越多的新信息挤压到上下文窗口的边缘，最终被截断或注意力稀释。模型开始"遗忘"自己在第 50 轮做出的关键设计决策，在第 300 轮时做出与之矛盾的选择。

状态漂移（State Drift）：Agent 维护的内部状态（文件修改记录、已完成任务列表、当前目标）和实际代码库状态之间的差距逐渐扩大。Agent 认为"已经改完了 X 文件"，但实际上 X 文件因为某个步骤的回滚已经变回了原样。

决策疲劳（Decision Fatigue）：模型在长时间上下文中表现出决策质量下降的趋势——早期的决策通常是深思熟虑的，而越往后，模型越倾向于选择"最快"而不是"最好"的方案。这类似于人类的决策疲劳，但机制不同：它不是"累了"，而是早期示例在注意力中的权重被不断稀释。

错误累积（Error Accumulation）：在短任务中，一个错误的工具调用或一个不准确的推理可以被立即发现和纠正。但在长任务中，微小的错误会在后续步骤中被放大和复合——一个错误的重构决策可能导致后续所有在这之上的工作全部报废。

踩坑实录：上面提到的 Python 2 迁移项目，我最终不得不让 Agent 在每次 Git commit 后"硬重置"上下文——清空对话历史，只保留最新的项目状态描述和 PROGRESS.md。这虽然解决了上下文腐烂问题，但代价是 Agent 每次重新加载时都要花 10-15 轮"重新理解"项目结构。我意识到：我们需要更优雅的解决方案，而不是粗暴地"重启"。

26.2 Anthropic cwc-long-running-agents 项目解读

2025 年 11 月，Anthropic 开源了 cwc-long-running-agents 项目，这是目前公开资料中针对 Long-Running Agent 设计最系统化的实践。随后在 2026 年 3 月，他们又发布了后续论文《Harness Design for Long-Running Application Development》，将实践经验提炼为可复用的架构模式。

这个项目的核心问题很简单却极其困难：如何构建一个能在多个会话（sessions）之间保持连贯性的 Agent？注意，这不是单个会话内的优化——跨会话意味着 Agent 的上下文每次都是全新的，它必须通过"外部工件"来维持对任务的理解和进度追踪。

cwc-long-running-agents 的解决方案可以归纳为三个核心设计模式：Default-FAIL 契约（防止 Agent 幻觉式地宣称完成）、Fresh-Context Evaluator（独立的评估机制）、以及 Kill Switch / Steering（人的控制机制）。我们逐一深入。

26.3 Default-FAIL 契约：反向门禁机制

这是 cwc-long-running-agents 中最有反直觉色彩的设计。在传统 Agent 中，成功标准的默认状态是"未验证"，Agent 可以通过工具调用来验证它们。但 Default-FAIL 反转了这个逻辑：每一个成功标准（criterion）在初始时都被标记为 FAIL。Agent 不能通过简单地说"我完成了"来将一个标准标记为 PASS——它必须经过严格的门禁流程。

这个门禁的核心是 PreToolUse Hook：在每次工具调用之前，系统检查 Agent 即将写入的结果文件所对应的证据文件是否在最近 N 轮内被读取过。如果没有——说明 Agent 没有基于最新的证据来确认结果——写入操作被直接拒绝。

来看实现代码：

# Default-FAIL 契约核心实现
# 每个成功标准默认 FAIL，必须读取证据后才能标记 PASS

import os
import json
from typing import Dict, List, Optional

class DefaultFailContract:
    """
    Default-FAIL 契约：所有成功标准初始为 False。
    Agent 必须读取证据文件后才能将标准标记为 True。
    """

    def __init__(self, criteria_path: str):
        self.criteria = self._load_criteria(criteria_path)
        self.last_read_turn: Dict[str, int] = {}
        self.current_turn = 0
        self.READ_WINDOW = 5  # 证据文件需在最近 5 轮内读取

    def _load_criteria(self, path: str) -> Dict:
        with open(path) as f:
            return json.load(f)

    def pretool_use_hook(self, tool_name: str,
                         args: dict) -> bool:
        """
        PreToolUse Hook：
        在工具执行前调用。返回 False 则拒绝执行。
        """
        self.current_turn += 1

        # 记录 Read 操作
        if tool_name == "Read":
            fpath = args.get("file_path", "")
            self.last_read_turn[fpath] = self.current_turn
            return True

        # 对 Write/Edit 目标进行证据检查
        if tool_name in ("Write", "Edit"):
            target = args.get("file_path", "")
            for cid, criterion in self.criteria.items():
                if target not in criterion.get("result_files", []):
                    continue
                # 检查所有证据文件是否在窗口期内被读取
                for ev_file in criterion.get("evidence_files", []):
                    last_read = self.last_read_turn.get(ev_file, 0)
                    if self.current_turn - last_read > self.READ_WINDOW:
                        print(
                            f"[Default-FAIL] Blocked: {target} "
                            f"- evidence '{ev_file}' stale "
                            f"(last read {self.current_turn - last_read} turns ago)"
                        )
                        return False  # 拒绝写入
        return True

    def evaluate_all(self) -> Dict[str, str]:
        """评估所有标准的状态"""
        status = {}
        for cid, criterion in self.criteria.items():
            result_files = criterion.get("result_files", [])
            if not result_files:
                status[cid] = "NO_RESULT"
                continue
            rf = result_files[0]
            if not os.path.exists(rf):
                status[cid] = "FAIL"  # 默认就是 FAIL
                continue
            with open(rf) as f:
                content = f.read().strip()
            status[cid] = "PASS" if content == "PASS" else "FAIL"
        return status

    def reset(self):
        """重置契约状态（新会话时调用）"""
        self.last_read_turn.clear()
        self.current_turn = 0
        # 注意：criteria 本身保留，结果文件也保留

这个设计巧妙之处在于：Agent 无法绕过证据链。它不能在一个全新的会话中直接写"PASS"——因为 PreToolUse Hook 会检查它是否真正阅读过测试报告、日志文件、diff 输出。这个机制在不同会话间同样生效，因为 last_read_turn 每次重置为 0。

深度洞察：Default-FAIL 的设计哲学是信任但验证的反面——它是不信任，要求证明。这对 LLM 的应用至关重要：因为模型天生有"讨好用户"的倾向（sycophancy bias），给它一个"证明你完成了"的任务，它更倾向于"声称完成"而不是"坦诚未完成"。Default-FAIL 从架构层面堵死了这条捷径。

26.4 Fresh-Context Evaluator：独立审判官

Default-FAIL 解决了"虚假完成"问题，但还有一个更隐蔽的问题：Agent 无法客观评估自己的工作质量。想象一下让一个考生既答题又给自己打分——即使他诚实，他的评估也会受到答题过程中各种偏见的影响。

Fresh-Context Evaluator 的解决方案是：创建一个独立的评估 Agent，它拥有完全干净的上下文窗口——从未看到过构建过程中的任何对话。它唯一的工具是 Read——没有 Write、没有 Edit、没有 Bash。它进入场景，阅读项目文件、阅读测试结果、阅读 Default-FAIL 的评估文件，然后输出 PASS 或 NEEDS_WORK，附带具体的改进建议。

为什么必须是"干净上下文"？因为在构建过程中积累的几百轮对话中，模型对自己的工作建立了"心理模型"——它会倾向于认为"我花了这么多功夫，应该是对的"。干净的上下文消除了这种自我辩护偏差。评估者只看到结果（artifacts），看不到过程（process），从而做出更客观的判断。

在 cwc-long-running-agents 中，Evaluator 的输出会被写入一个标准化的 JSON 文件，然后 Builder Agent 在下一轮 session 开始时读取它——如果 Evaluator 给出的 PASS 数量达到阈值，Builder 可以宣布任务完成；否则 Builder 根据 NEEDS_WORK 的反馈继续工作。

26.5 Kill Switch 与 Steering 机制

在长任务中，人的角色发生了根本变化。在短任务中，人是"操作者"——Agent 做一步，你看一步，有问题立即打断。但在长任务中，人无法全程盯着终端。你可能在 Agent 工作时去开会、吃饭、睡觉。因此，你需要远程控制的能力——不是实时打断，而是异步干预。

Kill Switch（紧急刹车）：当一个文件 AGENT_STOP 被创建时（由外部监控脚本或人工操作），Kill Switch 机制会在下一次 PreToolUse Hook 中检测到它，并立即终止所有工具调用。这不是优雅关闭——它是紧急停止。设计原则是：Kill Switch 必须独立于 Agent 运行环境。即使 Agent 的进程挂起、上下文损坏、Python 解释器崩溃，Kill Switch 仍然可以工作。

# kill_switch.sh
# 被外部 cron 或手动触发，创建停止信号文件
touch /path/to/project/AGENT_STOP
# Agent 端的 Kill Switch 检测（在 PreToolUse 和 PostToolUse 中检查）
def check_kill_switch(project_dir: str) -> bool:
    """返回 True 表示需要停止"""
    return os.path.exists(os.path.join(project_dir, "AGENT_STOP"))

Steering（方向盘）：比 Kill Switch 温和但同样强大的机制。当文件 STEER.md 的内容发生变化时，Steering 机制会在下一次工具调用前"展示"给 Agent 一次。注意是一次——不是每次。这样可以避免 Agent 因为反复看到的转向指令而分心。典型用法：STEER.md 中写"暂停对 API 层的修改，先完成数据库迁移"，然后 Agent 在下一个工具调用前读到这条指令，调整自己的方向。

关键设计原则：Kill Switch 和 Steering 都是"文件感知"的，而不是"信号感知"的。为什么选择文件？因为文件是持久化的——即使整个机器重启，信号文件依然存在。相比之下，Unix 信号（SIGTERM、SIGINT）在 Agent 重启后会丢失。对于跨会话的任务，持久化控制是必须的。

26.6 Agent-Maintained Handoff：跨会话状态传递

当一个 Agent 任务需要跨越多个会话（比如因为上下文预算耗尽、机器重启、或者单纯到下班时间了），如何确保下一个会话能无缝继续？cwc-long-running-agents 的方案是 Agent-Maintained Handoff：状态传递完全由外部文件承载，而不是 Agent 的内部状态。

核心交付物是三个文件：

  • PROGRESS.md：由 Agent 维护的状态文件，包含"已完成事项"、"进行中事项"、"下一步计划"、"已作出的关键决策"、"已知问题"。新会话开始时，Agent 首先读取 PROGRESS.md（内容会被压缩到系统提示中）。
  • Git commits：每个里程碑（milestone）由 Git commit 锁定。Git 是整个系统的"真相来源"（source of truth）。Agent 不依赖自己的记忆来判断某个模块是否已修改——它直接看 git diff。
  • commit-on-stop.sh：当 Kill Switch 触发或 Agent 检测到即将停止时，自动提交所有未提交的变更。确保即使异常中断，也不会丢失工作。
# commit-on-stop.sh
# 在 Agent 停止时自动保存未提交的工作
#!/bin/bash
PROJECT_DIR="$1"
cd "$PROJECT_DIR" || exit 1

if ! git diff --quiet || ! git diff --cached --quiet; then
    git add -A
    git commit -m "chore: auto-save before stop [agent]"
    echo "[commit-on-stop] Uncommitted changes saved."
else
    echo "[commit-on-stop] No uncommitted changes."
fi

这种设计背后的哲学非常清晰：Agent 的内存是易失的、不可靠的；而文件系统是持久化的、可审计的。不要把关键状态放在模型权重里——把它写在磁盘上。

26.7 Meta-Harness 的下一步

Long-Running Agents 的故事还没有结束。2026 年 3 月，Lee 等人在 arXiv（2603.28052）上发表了 Meta-Harness 的概念：一个能够自我优化的 Harness。这可能是 Agent 架构最令人兴奋的方向。

Meta-Harness 的工作流程是一个闭环优化：

  • Propose：Harness 生成一组候选配置——不同的上下文压缩策略、不同的完成检测模式、不同的工具超时设置。
  • Score：在验证集上运行所有候选配置，收集准确率和 token 消耗数据。
  • Pareto-keep：保留 Pareto 前沿上的配置——那些在不显著降低准确率的前提下减少 token 消耗的方案。
  • 循环往复，Harness 在不断的使用中自我进化。

论文报告的数据令人震惊：+7.7 准确率点数提升，同时减少约 4 倍的上下文 token 消耗——而这一切都没有更换模型！这意味着当前我们使用的 Agent 远未发挥模型的全部潜力，Harness 中还有大量"低垂的果实"等待采摘。

维度 短任务 Agent 长任务 Agent 权衡
会话模型 单会话，上下文持续累积 多会话，每次上下文干净启动 单会话简单但扩展性差；多会话增加恢复成本
状态管理 Agent 内部记忆 外部文件 + Git 内部快但不可靠；外部慢但可审计
完成检测 模型显式声明 Default-FAIL + Fresh Evaluator 声明式效率高但易幻觉；验证式可靠但开销大
错误恢复 同会话回滚 跨 session Git checkout 回滚快但深度有限；checkout 彻底但丢失未提交工作
人的角色 实时监督 异步控制（Kill Switch / Steering） 实时监督更安全但消耗人力；异步控制解放人力但增加延迟
Harness 优化 手动调参 Meta-Harness 自动优化 手动直观但效率低；自动高效但可能过拟合

展望 2027 年，Long-Running Agents 的演进方向是清晰的：从工具到同事。今天的 Agent 需要你告诉它每一步做什么；明天的 Agent 应该能接受一个季度目标，然后每周向你汇报一次进展。Default-FAIL 契约、Fresh-Context Evaluator、Kill Switch/Steering、Meta-Harness——这些不只是学术概念，它们是让 AI 从"实习生"变成"高级工程师"的必经之路。

核心金句：Long-Running Agents 不是"让 Agent 跑得更久"，而是重新思考 Agent 和人之间的关系——从工具到同事。
