第21章 移动与水下传感网

大家好，我是你们的讲师。想象一下：你在地面上使用蓝牙耳机——信号延迟、偶尔断连、音质打折扣——让你觉得"无线通信也就这样了"。但如果你把这副蓝牙耳机丢进游泳池，你会发现它连10厘米外的手机都连不上。再把场景拉远：你在南海中间部署了一组水下地震监测节点，需要将数据传到30公里外的岸基站。无线电波在水里传播距离不到1米就没了——你得用声波。而且这组节点可能被洋流冲走、被鱼群碰歪、或者被路过的AUV（自主水下航行器）撞到。

欢迎来到移动与水下传感网的世界。在这里，电磁波被海水吸收殆尽，声波变成了唯一的通信载体，网络拓扑的变动速度远超陆地WSN，延时容忍成为网络协议设计的第一原则。这是一个"一切都被重新定义"的领域——通信介质、协议栈、定位方法、能量获取方式，全部跟陆地WSN完全不同。

两个核心背景： 我国70%以上的国土面积是水域（含海洋），水下传感网在海洋资源勘探、海底管线监测、海啸预警和水下安防中扮演着不可替代的角色。另一面，移动的收集节点（无人机/无人车/无人艇）正在彻底改变数据采集模式——从"固定的传感器等你去读"变成了"移动的收集器来找你"。

移动性给WSN带来的挑战

前面所有章节讲的WSN技术，大部分假设节点是静止的。一旦引入移动性，问题变得复杂得多：

  • 拓扑剧烈变化——固定网络中，邻居关系是相对稳定的。移动场景下，节点的邻居列表可能在几分钟内完全刷新。链路中断和重建是常态而非异常。
  • 路由失效——基于静态拓扑建立的路由表在移动环境中迅速过期。传统的AODV和DSR按需路由开销巨大，因为每一个数据包都需要重新发现路由。
  • 通信窗口短暂——移动收集器经过一个静态节点附近的通信窗口可能只有几秒到几十秒。需要在极短时间内完成邻居发现→关联→数据交换→断开的全流程。
  • 定位困难——移动节点自身位置不断变化，而水下又无法使用GPS（电磁波被水体完全吸收），需要依赖声学定位系统。

移动Sink数据收集策略

移动Sink（也叫mobile collector或data mule）是目前应用最多的移动WSN模式。一个装载了收发器的小车/无人机/无人船以规划的路径在传感器区域移动，经过节点附近时自动收集数据。这种方法特别适合监测区域广阔但通信基础设施缺乏的场景（如农田、森林、矿区）。

三种主流策略：

  • 直接通信：移动Sink进入每个节点的通信范围后直接接收数据。实现简单，但Sink需要精确移动到每个节点的通信范围内，路径规划耗时。
  • Rendezvous点（会合点）：网络中选取部分节点作为"会合点"，其他节点通过多跳将数据聚合到会合点。移动Sink只需访问会合点，不必遍历每一个节点。这会合点的选择是一个针对旅行商问题的变种——在节点能量和延迟约束下，最小化Sink的路径长度。
  • 路径规划：利用机器学习和优化算法（如遗传算法、强化学习）动态规划Sink的最优移动路径。对于大规模动态网络，离线规划结合在线调整是主流方案。

DTN延迟容忍网络

当网络连接是间歇性的、端到端路径不存在时，传统的TCP/IP协议栈就失效了。DTN（Delay/Disruption Tolerant Networking，延迟容忍网络）是专门为此设计的"网络中的网络"。

DTN的核心概念是Bundle协议（RFC 5050/9172）。它工作在TCP/IP之上（或其他传输协议之上），在应用层之下增加了一个Bundle层。数据以"Bundle"为单位传输，每个Bundle包含用户数据和元信息（源/目标EID、存活时间、优先级等）。

DTN的关键机制：

  • 存储转发（Store-Carry-Forward）——节点收到数据后，如果下一跳不可达，将数据持久化存储（写入闪存/磁盘），等链路恢复后再转发。这与TCP的"缓冲+重传"不同——TCP的缓冲是短暂的、面向端到端的，而DTN的存储是长期的、面向逐跳的。
  • 保管传输（Custody Transfer）——发送方将Bundle的"保管责任"移交给接收方后即可释放缓存。如果接收方在存活期内无法转发，需要通知发送方重新发送。这提供了端到端的可靠性而不需要端到端的连接。
  • 喷泉码（Fountain Code）——一种无速率纠错码。发送方不断生成编码符号，接收方只要收够足够数量的符号（不管哪些）就能解码出原始数据。在链路质量极不稳定的水下环境中，喷泉码远比传统ARQ重传高效。
有意思的工程细节： DTN的存储转发机制在NASA的深空网络（IPN）项目中大规模使用。火星轨道器和火星车之间每天只有几个通信窗口（火星自转导致），每次窗口可能只有10-15分钟。数据先通过DTN在地面站→火星轨道器→火星车之间逐跳转发——跟WSN的多跳概念一模一样，只是跳距变成了几亿公里。

水下声学信道——一切重新开始

水下无线传感器网络（UWSN）是整个WSN领域中最"硬核"的分支之一。通信介质从电磁波变成了声波（声纳），这带来了根本性的变化：

参数维度 陆地WSN（电磁波） 水下WSN（声波）
传播速度 3×10⁸ m/s（光速） 约1500 m/s（声速，取决于温盐深）
传播延迟 可忽略（~3.3μs/km） 巨大（~0.67s/km）——1km距离往返延迟1.3秒
可用带宽 数MHz~GHz 极窄：10~50kHz（取决于距离）
数据速率 Mbps~Gbps kbps~几十kbps
衰减特性 与距离平方成反比 与距离和频率同时相关（Thorp模型）
多径效应 中等（室内严重） 极为严重（水面/海底/海面多次反射）
多普勒效应 可忽略（低速场景） 严重（声速慢，相对运动导致频移大）
能量消耗 发送≈接收（无线电） 发送能耗远大于接收（声学功放）
节点能量补给 可更换电池/太阳能 极困难（水压、防腐、洋流冲击）
定位方式 GPS + 射频测距 声学基线定位（LBL/SBL/USBL）
部署维护成本 低 极高（需潜水员/ROV/AUV）

水下声学信道的衰减可以用Thorp传播模型近似描述——衰减系数随频率呈指数级增长，这是水下系统设计最基本的物理约束：

// 水下声学信道Thorp衰减模型
// 参考文献: Brekhovskikh & Lysanov, "Fundamentals of Ocean Acoustics"

// Thorp经验公式：衰减系数 α(f) (dB/km)
// f 的单位为 kHz
function thorpAttenuation(f) {
    var fSq = f * f;
    var fLog = Math.log10(f);

    if (f < 0.4) {
        return 0.11 * fSq / (1 + fSq) + 44 * fSq / (4100 + fSq)
               + 2.75e-4 * fSq + 0.003;
    } else {
        return 0.11 * fSq / (1 + fSq) + 44 * fSq / (4100 + fSq)
               + 2.75e-4 * fSq + 0.003 + 0.0033 * fLog;
    }
}

// 总传播损耗 TL = 扩展损耗 + 吸收损耗
// TL(d, f) = n * 10 * log10(d) + α(f) * d / 1000
// n=2 球面扩展（深水）, n=1.5 实际扩展（浅水）, n=1 柱面扩展（浅水波导）

function totalPathLoss(d, f, n) {
    // d: 距离(米)
    // f: 频率(kHz)
    // n: 扩展因子
    var spreadingLoss = n * 10 * Math.log10(d);
    var absorptionLoss = thorpAttenuation(f) * d / 1000;
    return spreadingLoss + absorptionLoss;  // dB
}

// 示例：水深约为500m时
// 距离1km, 频率10kHz, 球面扩展 (n=2)
var loss1 = totalPathLoss(1000, 10, 2);   // ≈ 30 + 3.5 ≈ 33.5 dB

// 距离10km, 频率10kHz
var loss2 = totalPathLoss(10000, 10, 2);  // ≈ 40 + 35 ≈ 75 dB —— 吸收损耗随距离线性增长

// 距离1km, 频率50kHz —— 频率翻5倍，衰减翻几倍
var loss3 = totalPathLoss(1000, 50, 2);   // ≈ 30 + 55 ≈ 85 dB —— 高频被严重吸收

// 结论：水下通信必须在距离和频率之间做痛苦的选择
// 长距离（>10km）只能用极低频率（<5kHz），数据速率极低

这个模型揭示了水下通信的根本困境：你想传得远，就必须用低频（<10kHz），但低频意味着极低的数据速率（几百bps）。你想要高数据速率，就必须用高频，但高频信号在几百米后就被海水吸收殆尽。这就是为什么水下声学modem的通信速率远低于大家的直觉——一个10km通信距离的水声modem，典型速率仅为500-5000bps。

水下MAC协议的特殊性

陆地无线网络中大放异彩的CSMA/CA协议，在水下声学网络中完全行不通。原因很简单：CSMA依靠"先听后说"来避免碰撞——侦听信道空闲后才发送。但声学信号在水中的传播延迟高达0.67秒/公里。当A节点开始发送时，远端的B节点还没有"听到"A的信号，B以为信道空闲就也开始发送——碰撞。这是所谓的暴露节点问题在水下的极端放大版。

因此水下MAC协议的主流方案转向了基于CDMA（码分多址）的方案或预分配TDMA。CDMA的好处是多个节点可以在同一频率和时间发送，只要使用的扩频码正交，接收端可以解调出各自的信号。这对于长传播延迟环境来说是"天作之合"——你不需要等待信道空闲，直接发就行，反正码分正交能扛住碰撞。

AUV组网与水下定位

AUV（Autonomous Underwater Vehicle，自主水下航行器）是水下传感网的重要组成部分。它既可作为移动Sink收集固定节点的数据，也可搭载传感器进行大范围海洋环境调查。AUV之间的组网（UWAAN——Underwater Acoustic Ad-hoc Network）是当前的研究热点：一群AUV以自组织方式形成声学通信网络，协同完成海底地形测绘、目标搜索和水下管线巡检等任务。

水下声学定位是UWSN的基础设施，主要使用基线定位方法：

  • LBL（长基线）——在海床部署3-4个已知位置的声学应答器（间距数百到数千米）。待定位节点测量到各应答器的往返时间（TWR），通过三边定位计算自身位置。精度最高（可达厘米级），但部署和维护成本极高。
  • SBL（短基线）——在船底安装多个水听器（间距数米），通过测量目标声信号到达各水听器的时延差来定位。精度一般但灵活。
  • USBL（超短基线）——在同一个换能器阵列内测量信号的相位差来估计方位角和俯仰角。设备紧凑（一个筒内集成），适合船载或AUV搭载，但精度受声线弯曲影响。
水下部署的残酷现实： 节点一旦部署到海底，你基本就失去了对它的一切控制。电池无法更换、固件无法升级、防水壳的O型密封圈几年后必然老化。水下WSN的节点设计必须遵循"set and forget"原则——部署前做最充分的测试，部署后接受"少即是多"的通信方案。

一个真实的"踩坑"故事

2021年，我参与了一个海洋环境监测项目：使用AUV携带CTD传感器（测量电导率/温度/深度）对某海湾进行水质调查。AUV按预设路径航行，每隔30分钟上浮一次通过卫星回传数据。方案听起来万无一失——直到实际海试那天。

AUV入水后，前两个小时数据一切正常。第三个小时开始，水面控制台显示AUV的通信状态出现了间歇性中断——控制台发送的声学modem握手请求有时能得到应答，有时得不到。一开始以为是鱼群干扰或船只噪声。检查了AUV的声学换能器（安装在AUV尾部），没有发现物理损坏。

随着时间推移，中断越来越频繁。到第五个小时，AUV完全没有回应了。我们开始担心最坏的情况——AUV可能已经丢失。团队紧急调度了一艘橡皮艇到AUV最后已知位置。声纳搜索了三个小时后，终于在一个意想不到的位置发现了AUV——离预定路径偏移了3公里！

事后分析发现，问题出在AUV的航向控制算法和声学modem的握手交互时序上。AUV使用的是一种简化版的握手协议：控制台发送WAKEUP信号→AUV回复READY→控制台发送数据请求。由于水下声学传播延迟大，控制台发送WAKEUP后等待REPLY的超时设置为5秒。通常情况下足够。但当AUV转弯时（两侧推进器差速），转弯产生的湍流噪声使AUV端的声学modem信噪比下降——WAKEUP信号虽然能被AUV收到，但AUV回复的READY信号在湍流噪声下无法被控制台成功解调。控制台收不到READY，判断AUV不响应，没有再尝试联系。而AUV端认为自己已经回复了READY，正在等待控制台的数据请求——双方进入了"死锁"状态。

此时的AUV仍按照预定航线自主航行，只是与控制台失联。直到三个小时后有人想到检查卫星定位——发现AUV已经漂出了搜索范围。最后的教训是：水声通信的握手协议必须有"乱序恢复"机制——每一方都要能处理对方"假死"的情况，通过超时自动回退到初始状态重新握手。这个场景跟TCP的SYN丢失问题非常类似，但在水下，由于传播延迟极大，超时要设置得更长，状态机也要设计得更鲁棒。

设计原则： 水下声学通信协议的鲁棒性设计，遵循一个朴素的准则——"先假设所有消息都会丢失，再优化那些收到的"。容忍消息丢失的协议（如ALOHA变种、基于CDMA的并发传输），在水下往往比"先握手再通信"的精巧协议表现更好。

好了，从移动的Sink到深海的声学Modem，我们走完了移动与水下传感网的旅程。下一章我们将回到"地面"，但视角完全不同——多媒体传感网。摄像头、麦克风、毫米波雷达——当WSN节点从"采集几字节的温度数据"到"处理几MB的视频流"，整个系统设计理念又将被颠覆。
