大家好，我是你们的讲师。从第1章到第25章，我们一直在Linux 0.11这个"古老"但精炼的操作系统内核中探索。但Linux的故事远远没有停在0.11。事实上，从1991年12月0.11发布到1994年3月1.0正式版发布，这短短不到两年半的时间里，Linux内核经历了爆炸式的增长——代码量从约14,000行膨胀到超过176,000行，外部贡献者从Linus一个人发展到遍布全球的上百名开发者。每一次版本升级都不仅仅是在修bug，而是引入了影响至今的设计决策。

引言

如果说0.11是Linux内核的"胚胎期"，那么从0.12到1.0的这一段演化史就是它的"儿童期"——所有的核心能力都在这个阶段被建立起来。理解这段历史，能让你看清楚：今天Linux内核中那些你认为理所当然的功能，从来都不是一蹴而就的。它们是在真实需求的推动下，一步步被设计、被实现、被重构出来的。

本章将沿着0.12 → 0.95 → 0.96 → 0.98 → 0.99 → 1.0的版本主线，逐个分析每一个版本带来哪些影响深远的设计。你会发现，Linux的演化不是线性的渐进——而是一波接一波的"架构跳跃"。

0.12——虚拟内存与交换

1992年1月，距离0.11发布仅一个月，Linus就发布了0.12。这个版本的发布速度令人瞠目，但它确实带来了影响最深远的机制之一——交换（swapping）。

在0.11中，所有进程的代码和数据在创建时就被完整加载到物理内存中。如果你同时运行6个程序，而物理内存只够装5个——对不起，系统会直接报错。这在只有4MB内存的386 PC上是一个严重的限制。0.12引入了一个交换守护进程（swapd），它会在物理内存不足时将不活跃的页面换出到磁盘上的交换分区，并在需要时再换入。页面换出策略采用了第二次机会算法——一种简化的LRU近似实现。虽然从现代角度看这个算法非常简陋，但它奠定了Linux虚拟内存系统的基础。

除了换页，0.12还添加了几个重要的功能：符号链接让Linux文件系统首次支持了"快捷方式"；伪终端（PTY）为远程登录（telnet）提供了基础设施；系统调用从0.11的约40个增加到70个。但最重要的也许是——从0.12开始，Linux正式采用GNU GPL v2作为许可证。此前Linus使用的是自己写的一份宽松许可，但他意识到GPL能更好地保护开源软件不被私有化。这个决定，怎么说都不为过。

许可证的历史转折：Linus在0.12的发布公告中写道："我把许可证变成了GNU GPL，因为我觉得这样更好。另外，我听说RMS在MIT经常谈论GPL的事情——虽然我不完全认同他的一些观点，但GPL确实是一个好的许可证。" 这个务实的选择守护了Linux三十年来的自由。

0.95——虚拟文件系统（VFS）

0.95发布于1992年3月，距离0.12仅两个月。它的最大成就是虚拟文件系统（Virtual File System, VFS）框架的诞生。这个框架的引入，是Linux发展史中最关键的一次架构决策。

在0.11中，"文件系统"就是MINIX文件系统——没有选择，没有抽象，所有文件操作直接调用MINIX FS的实现函数。随着社区希望Linux能读取MS-DOS的FAT格式磁盘，Linus意识到不能为每一种文件系统都硬编码一套操作路径。他需要的是一个抽象层：VFS。

VFS的核心是一组函数指针结构体。每一种文件系统只要实现了这些回调函数，就能无缝挂载到Linux的目录树中。

/* Linux 0.95 VFS 核心数据结构（简化） */
struct file_operations {
    int (*read)(struct inode *, struct file *, char *, int);
    int (*write)(struct inode *, struct file *, char *, int);
    int (*open)(struct inode *, struct file *);
    void (*release)(struct inode *, struct file *);
};

struct inode_operations {
    int (*create)(struct inode *, const char *, int, int);
    int (*lookup)(struct inode *, const char *, int, struct inode **);
    int (*mkdir)(struct inode *, const char *, int);
    int (*rmdir)(struct inode *, const char *, int);
};

struct super_operations {
    int (*read_super)(struct super_block *, void *, int);
    int (*write_super)(struct super_block *);
    int (*statfs)(struct super_block *, struct statfs *, int);
};

这个设计的精妙之处在于：VFS本身就是"管理文件系统的文件系统"。它不关心底层是MINIX还是FAT——它只关心file_operations、inode_operations、super_operations这三个"合同"。任何遵守合同的文件系统都可以被挂载。这个框架至今没有本质变化——你去读Linux 6.x的include/linux/fs.h，仍然能看到struct file_operations，只不过函数指针多了几十个而已。

0.96——ext文件系统与SCSI

1992年5月发布的0.96引入了两个重要的子系统。extfs（extended filesystem）是ext2的前身，由Rémy Card开发。相比MINIX FS，extfs支持更大的分区（最大2GB vs 64MB）和更长的文件名（255字符 vs 14字符），并首次引入了时间戳。SCSI子系统的加入则让Linux可以连接SCSI硬盘、扫描仪和CD-ROM——这是Linux进入企业市场的第一步。

到0.96为止，VFS已经支持了三种文件系统类型：

文件系统最大分区最大文件文件名长度特点
MINIX FS64 MB64 MB14 / 30 字符设计简单，代码量少，是0.11的默认FS
extfs2 GB2 GB255 字符支持时间戳，性能更好，ext2的前身
FAT (MS-DOS)2 GB2 GB8.3 格式兼容DOS双启动，由社区贡献

同时，0.96还加入了UNIX域套接字（UNIX Domain Socket）支持，让同一台机器上的进程可以通过socket通信——这是后来Linux进程间通信（IPC）体系中的重要一环。

0.98——TCP/IP网络栈

网络支持是Linux走向"联网操作系统"的关键里程碑。0.98（1992年9月）包含了由Ross Biro贡献的初始TCP/IP网络栈。这个网络栈的架构清晰分层：

/* Linux 0.98 网络栈分层结构 */
+------------------------------+
| BSD Socket 层 (用户态API)     |  ← socket(), bind(), listen(), accept()
+------------------------------+
| INET 层 (协议族实现)          |  ← TCP, UDP, IP, ICMP 协议处理
+------------------------------+
| 链路层 (设备驱动)             |  ← NE2000, 3c501 等网卡驱动
+------------------------------+
| 物理层                       |  ← 以太网、串行线
+------------------------------+

/* 在0.98上已经可以运行 telnet 和 ftp */

虽然0.98的网络栈还很原始——只支持以太网，没有路由表概念，只有最基本的TCP拥塞控制——但它证明了Linux可以成为一个网络操作系统。Linus在发布公告中自豪地写道："使用标准化的telnet/ftp客户端可以正常工作了。下一步是让Linux可以作为NFS客户端。" 事实是，在网络能力加持后，Linux的社区参与度开始指数级增长。

踩坑回忆：网络栈的"黑暗时代"：Ross Biro贡献的初始网络栈虽然可用，但存在严重的竞态条件（race condition）。在高负载下，TCP连接会随机断开，甚至触发内核panic。Linus在0.99的开发周期中几乎重写了整个网络层。Alan Cox后来接手了网络栈的维护工作，在1.0发布前将其稳定下来。这个故事告诉我们——网络栈的并发安全，从一开始就是内核开发中最棘手的问题之一。

0.99——4GB线性空间

0.99（1992年12月）是一个被低估的版本，因为它引入了整个Linux进程模型中最重要的设计——每个进程独立的4GB线性地址空间。

在0.11中，所有进程共享同一套页目录表。这意味着一个进程能看到其他进程的内存——虽然通过段保护机制能防止越权访问，但从地址空间布局上看，所有进程都挤在同一块"画布"上画图。这个设计严重限制了进程隔离性和内存管理功能。

0.99的变革是：每个进程都拥有自己的页目录（page directory），独立管理自己的4GB线性地址空间。当你切换进程时，内核会同时切换CR3寄存器中的页目录基址，使CPU立即映射到新进程的地址空间。这个设计——即每个进程独立页表——是现代Linux进程模型的基石。0.99还首次引入了mmap系统调用，让文件可以映射到进程地址空间。

从0.99开始，Linux的进程模型基本定型：task_struct管理进程元数据，mm_struct管理地址空间，页目录/页表管理虚实映射。这个三分架构一直延续到今天。

1.0——正式版发布

经过漫长的开发（0.99有大量alpha/beta子版本，从0.99a一直迭代到0.99z），Linux 1.0正式版于1994年3月14日发布。这一天的到来比Linus预期晚了整整一年——他原本计划1993年初就发布1.0，但网络栈的稳定性和设备驱动的适配花了远超预期的时间。

1.0版总共包含约176,250行代码。让我们用一张表来直观感受0.11到1.0之间发生了什么：

有趣的是，1.0发布时的README文件仍然保留着Linus的一贯幽默风格。他在/usr/src/linux/README中写道："Linux是一份免费的操作系统——它不像那些卖几百美元的东西，但也不必期望它什么都能做。" 这个谦虚的描述和今天Linux统治世界的现实形成了强烈的反差。

从1.0到今天：1.0发布至今已经超过30年。今天的Linux 6.x内核代码超过3,000万行，支持的硬件架构超过30种，文件系统超过50种。但如果你翻开0.12的swap代码和0.95的VFS代码，你会发现——核心思想没有变。变得是规模、性能和功能边界，不变的是内核架构最核心的那些设计决策。

这就是0.11到1.0的演化史——一段从"业余爱好"到"专业操作系统"的旅程。但Linux的脚步从未停歇。从1.0到今天的6.x，内核规模从17万行膨胀到3000万行。下一章，我们将对比0.11与现代内核，看看哪些变了、哪些没变——你会发现，有些东西真的穿越了30年的时光。
