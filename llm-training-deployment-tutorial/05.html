第5章 分布式训练基础

大家好，我是你们的讲师。前面几章我们搭建了一个完整的"单卡认知"——理解了数据怎么清洗、文本怎么tokenize、Transformer结构怎么工作。但在现实中，没有任何一个大模型是在单张GPU上训练的。即使是最小的1.5B模型，在FP16下也需要约3GB显存来存放参数，但训练时的激活值、梯度、优化器状态会把显存需求放大5-10倍。7B模型的话，单卡H100（80GB）也放不下。这就是为什么我们需要分布式训练。

分布式训练的本质很简单：把一个人搬不动的大石头，切成小块几个人一起搬。但"切"的方式不同，就有不同的并行策略。这一章我们从最基础的开始，先理解单卡训练的瓶颈，再到数据并行和通信原语，最后讲到FSDP——这是通往大规模训练的"第一级台阶"。

5.1 为什么单卡训不了大模型？

我们来算一笔账。假设你想训练一个7B参数的模型，使用FP16（2 bytes per parameter），来看看每一块的显存需求：

模型参数：7B × 2 bytes = 14GB。勉强放得下。

梯度：和参数一样大，也是14GB。

优化器状态：AdamW需要存储两个动量项（一阶动量和二阶动量），每个也是FP32精度（4 bytes）。所以优化器状态 = 7B × 4 × 2 = 56GB。

激活值（Activations）：这是最大的一块。每个transformer层在forward过程中要保留中间激活值用于backward计算梯度。对于7B模型，batch_size=1、seq_len=4096时，单层的激活值可能需要数十MB到上百MB，32层加起来大约20-40GB。

算下来，光参数+梯度+优化器状态就已经84GB了，超过了一张H100的80GB显存。这还没算激活值！所以即使batch_size=1，7B模型在单卡H100上也训练不了。

模型规模 参数(FP16) 梯度(FP16) 优化器(FP32) 激活值(典型) 总计 单卡H100能否训练
1.5B 3GB 3GB 12GB 4-8GB 22-26GB 可以（batch_size较小）
7B 14GB 14GB 56GB 20-40GB 104-124GB 超显存
13B 26GB 26GB 104GB 40-80GB 196-236GB 不可能
70B 140GB 140GB 560GB 200-400GB 1-1.2TB 需要大规模分布式

另外还有算力瓶颈和时间瓶颈：即使单卡能放下7B模型，以H100每秒约1000TFLOPS的算力，训练2万亿token也需要几百天。分布式训练的核心目标就是把这几百天压缩到几天——通过让多张GPU并行工作。

5.2 数据并行(DP/DDP)

数据并行（Data Parallelism）是最简单、最直观的并行策略。核心思想：每张GPU都存一份完整的模型副本，但分配到不同的数据批次。正向传播时各算各的，反向传播后通过通信同步梯度，确保所有卡上的模型参数保持一致。

DP (Data Parallel)是PyTorch最早的实现。它的工作方式是：master GPU（rank 0）收集所有卡的梯度并求和，再广播回去。有一个很明显的性能问题——所有通信流量都经过rank 0，导致它成为瓶颈，GPU利用率的扩展性很差。

DDP (Distributed Data Parallel)是改进版。它使用了Ring AllReduce算法：所有GPU排成一个环，梯度在环上依次传递并累加，最后每个GPU都得到完整的聚合梯度。Ring AllReduce的通信量只有2*(N-1)/N * 模型大小的数据交换（随着N增大趋近于2倍模型大小），而且带宽利用率极高。DDP是目前绝大多数中小规模分布式训练的默认选择。

# 使用PyTorch DDP的标准代码框架
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

def train():
    dist.init_process_group(backend="nccl")
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    model = MyModel(7e9).to(rank)
    ddp_model = DDP(model, device_ids=[rank])

    dataloader = get_data_loader(batch_size=2, world_size=world_size)
    optimizer = torch.optim.AdamW(ddp_model.parameters(), lr=3e-4)

    for batch in dataloader:
        # DDP自动处理梯度同步
        loss = ddp_model(batch)
        loss.backward()
        optimizer.step()
        optimizer.zero_grad()

但数据并行有个很大的局限性：每张卡都要存一份完整的模型、梯度和优化器状态。对于7B模型，即使你用DDP跑在64张卡上，每张卡仍然需要~100GB以上的显存——DDP并没有解决显存问题，只是解决了计算速度问题（通过增加并行度来增大global batch size）。这正是FSDP和ZeRO要解决的痛点。

5.3 通信原语深入

分布式训练的本质是通信。不理解通信原语，就难以理解分布式训练的性能瓶颈。下面是四个最核心的原语。

Broadcast：将一份数据从root节点发送到所有其他节点。例如，初始化时将模型参数从rank 0广播到所有节点。通信量 = 数据大小。时间复杂度O(N)。

AllReduce：在所有节点上执行归约操作（通常是求和），并将结果返回到所有节点。Ring AllReduce的实现将AllReduce分解为ReduceScatter + AllGather两步，通信量 = 2 × (N-1)/N × 数据大小。这是DDP中梯度同步的核心原语。

ReduceScatter：将所有节点的数据按块求和，每个节点持有求和后的一块数据。通信量 = (N-1)/N × 数据大小。Ring AllReduce的第一步。

AllGather：每个节点将自己持有的一块数据广播给所有其他节点。通信量 = (N-1)/N × 数据大小。Ring AllReduce的第二步。

原语 描述 通信量 主要用途
Broadcast 一个节点→所有节点 N × 数据大小 初始化参数广播
AllReduce 所有节点求和→所有节点 2(N-1)/N × 数据大小 梯度同步（DDP）
ReduceScatter 所有节点求和→分布到各节点 (N-1)/N × 数据大小 ZeRO梯度分片
AllGather 各节点的数据块→所有节点 (N-1)/N × 数据大小 ZeRO参数收集

理解通信量很重要，因为它直接决定了分布式训练的通信开销占比。一个经验公式：如果通信时间大于计算时间，说明你的计算/通信比太低，需要增大batch size或者改用更高效的并行策略。

5.4 通信与计算重叠

一个常见的误解是：分布式训练中，通信时间是"额外"的，是白花的时间。但好的工程实现可以让通信和计算重叠（overlap），把通信"藏"在计算的背后。

具体来说，在DDP的反向传播过程中，模型各层的梯度是按从后往前的顺序计算的。前面的层（靠近输入的层）还在计算梯度时，后面的层（靠近输出的层）的梯度已经计算完毕了。因此，我们可以在后面的层计算一完成就立即启动该层梯度的allreduce，而不必等到所有层的梯度都计算完毕。这就是梯度异步通信，也称为bucket communication。

PyTorch DDP默认会按参数大小"分桶"（bucket），每个桶的梯度计算完后立即触发allreduce。默认的bucket大小是25MB，但实际项目中建议调整这个值：如果计算密度低（如小模型），减小bucket大小可以让通信更早开始；如果计算密度高（如大模型），增大bucket可以减少通信次数。

实战调优：你可以通过设置环境变量TORCH_DISTRIBUTED_DETAIL=DEBUG来观察每个桶的通信时间。如果发现某个桶的通信时间长于下一桶的计算时间，说明overlap不够好。优化手段包括：①增大batch size提高计算密度；②调整bucket大小（torch.distributed.set_default_ddp_bucket_size()）；③开启gradient_as_bucket_view=True减少内存拷贝。

5.5 FSDP零冗余优化

前面说过，DDP最大的问题是没有减少单卡显存占用——每张卡都持有完整的模型状态。ZeRO（Zero Redundancy Optimizer）系列论文（Microsoft, 2020）提出了一种打破这个限制的思路：既然N张卡存了N份相同的数据，为什么不各自只存一份？

ZeRO-1：将优化器状态分片到各卡，每卡只存1/N的优化器状态。这节省了AdamW两个动量项的大部分内存（56GB → 56GB/N）。当N=8时，优化器状态从56GB降到7GB，效果显著。

ZeRO-2：在ZeRO-1的基础上，将梯度也分片。每卡只存1/N的梯度。梯度内存从14GB降到14GB/N。

ZeRO-3：在ZeRO-2的基础上，将模型参数也分片。每卡只存1/N的参数。参数内存从14GB降到14GB/N。但注意：ZeRO-3在每次forward和backward之前需要allgather来收集完整的参数，之后要reduce_scatter来更新梯度。这增加了通信量，所以ZeRO-3的通信开销比ZeRO-2高，但在模型大到放不下单卡时是必须的选择。

FSDP（Fully Sharded Data Parallel）是PyTorch在ZeRO思想上的官方实现。它封装了ZeRO-1/2/3，通过sharding_strategy参数来控制分片程度。FSDP的最大优势是：你可以在不修改模型代码的情况下，通过包裹模型来实现ZeRO（类似DDP的用法）。

# FSDP使用示例（ZeRO-3模式）
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import ShardingStrategy

# ZeRO-3: 参数/梯度/优化器状态全部分片
model = FSDP(
    MyModel(7e9),
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # ZeRO-3
    device_id=rank,
)

# ZeRO-2: 只分片梯度和优化器状态
model = FSDP(
    MyModel(7e9),
    sharding_strategy=ShardingStrategy.SHARD_GRAD_OP,  # ZeRO-2
    device_id=rank,
)

# 训练代码与DDP完全一致
for batch in dataloader:
    loss = model(batch)
    loss.backward()    # 自动执行allgather（前向）+ reduce_scatter（反向）
    optimizer.step()

踩坑故事：一个团队在迁移从DDP到FSDP时遇到了一个非常隐蔽的性能问题。他们的H100集群在DDP下GPU利用率达到85%，但切换到FSDP（ZeRO-3）后利用率跌到了30%。排查了三天，发现问题的根源是FSDP的参数预取（prefetch）没开。FSDP在forward过程中，每个transformer层需要在计算前allgather参数。如果不开预取，每层都要等待参数收集完成才能开始计算，造成了大量的GPU空闲。启用forward_prefetch=True和limit_all_gathers=True之后，利用率回到了78%。还有一个坑：当使用FSDP + gradient checkpointing时，checkpoint会重新触发allgather，导致重复通信。解决方案是启用reshard_after_forward=False，在checkpoint重计算阶段不重新分片参数。

选择建议：对于7B以下模型，DDP通常已经足够（配合activation checkpointing），ZeRO-2作为可选项。对于7B-13B模型，FSDP ZeRO-2是最佳性价比——显存节省显著且通信开销可控。对于13B以上模型，FSDP ZeRO-3是唯一可行的方案，但需要仔细调优预取策略和通信配置以达到可接受的效率。
