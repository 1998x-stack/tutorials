第3章 Verilog HDL语法精要（上）

大家好，我是你们的讲师。从本章开始，我们要亲手写代码了。Verilog 是一种硬件描述语言（HDL）——请记住这个词里的"描述"二字。你不是在"编程"（programming），而是在"描述电路"（describing hardware）。这个思维转变决定了你能不能写出可综合、可落地的代码。

Verilog 的语法看起来和 C 语言有点像——有 if-else、case、有赋值语句。但每一行 Verilog 都会在芯片上变成实实在在的电路。所以本章和下一章我们不求"语法全面"，而是追求"语法→电路"的映射关系清晰。这一章先讲模块结构、wire/reg 本质、assign 连续赋值和运算符体系。

3.1 模块是 Verilog 的基本单元

一个 Verilog 模块对应芯片上的一个硬件模块。模块之间通过端口（port）连接——这些端口在物理上就是芯片引脚之间的连线。

3.1.1 模块定义与端口方向

module counter #(
    parameter WIDTH = 8  // 参数化设计
) (
    input  wire       clk,      // 输入
    input  wire       rst_n,    // 输入
    output reg [WIDTH-1:0] cnt  // 输出
);
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n)
            cnt <= 0;
        else
            cnt <= cnt + 1;
    end
endmodule

端口方向有三种：

  • input——输入端口，模块内部只能读，不能写。在综合后对应一个输入引脚。
  • output——输出端口，模块内部可以赋值。在综合后对应一个输出引脚。
  • inout——双向端口，用于三态总线（如 I2C 的 SDA）。FPGA 内部需要三态缓冲器 IOBUF 原语实现。

关于端口声明方式：Verilog-2001 引入了 ANSI 风格的声明方式（如上例），比 Verilog-1995 的非 ANSI 风格（端口列表只写名字，类型在 module 内部声明）更推荐。现代工具链都支持 ANSI 风格。

3.1.2 参数化设计

#(parameter ...) 让模块具有可配置性。实例化时可以重写参数：

// 实例化一个 16 位宽度的计数器
counter #(.WIDTH(16)) u_counter (
    .clk  (sys_clk),
    .rst_n(sys_rst_n),
    .cnt  (counter_val)
);

参数化是 IP 复用的基石。好的参数化设计可以通过修改参数来适配不同规格，而不用复制粘贴代码。

3.2 wire 与 reg 的本质区别

这是 Verilog 初学者最大的困惑来源。我听到过最经典的错误答案是："wire 是组合逻辑，reg 是时序逻辑"——完全错误。正确的理解是：

  • wire——物理连线。代表两个端口之间的导线。wire 不能保存值，必须由连续赋值 assign 或模块端口连续驱动。
  • reg——一个"保持值的变量"。它不一定综合成触发器！在 always @(*)（组合逻辑块）中赋值的 reg，综合结果还是组合逻辑（LUT），只是仿真时需要在过程块中保持值。
核心记忆规则：只要在 always 块中被赋值的信号，就必须声明为 reg类型；在 assign 语句中被赋值的信号，必须声明为 wire。这个规则纯粹是 Verilog 语法层面的要求，和综合结果是组合还是时序没有必然关系。SystemVerilog 中引入了 logic 类型来消除这个困惑，但如果你还在用 Verilog-2001，仍需遵守这个规则。

3.3 assign 连续赋值

assign 描述组合逻辑——等号右边任意输入变化，左边立即更新（经过传播延迟后）。综合器会把 assign 映射到 LUT 组合逻辑。

3.3.1 基础用法

// 一个半加器
module half_adder (
    input  wire a, b,
    output wire sum, carry
);
    assign sum   = a ^ b;    // 异或——求和位
    assign carry = a & b;    // 与——进位位
endmodule

3.3.2 条件赋值（条件运算符）

条件运算符 ? : 是 Verilog 中实现 MUX 的简洁方式：

// 2:1 多路选择器
module mux2to1 (
    input  wire [3:0] a, b,
    input  wire       sel,
    output wire [3:0] y
);
    assign y = sel ? b : a;
endmodule

条件运算符可以嵌套，但要注意：嵌套层数越深，综合后的 LUT 级联深度越大，路径延迟越长。一般不超过 3-4 层。

关于除法和取模：组合逻辑的除法和取模会消耗大量 LUT——一个 32-bit 的无符号除法大约需要 1000+ 个 LUT，且路径延迟极大。避免在组合逻辑中用 / 和 %。如果需要除法，考虑用右移（除以 2 的幂）或迭代减法实现。Vivado HLS 和 Vitis HLS 对于非 2 的幂的除法会生成流水线除法器，但资源消耗仍然可观。

3.5 实战：用 assign 实现 ALU

下面这个 ALU 展示了 assign 和条件运算符的组合用法：

// 一个简单的 ALU，支持 4 种操作
module alu_assign (
    input  wire [7:0] a, b,
    input  wire [1:0] op,    // 00:加 01:减 10:与 11:或
    output wire [7:0] result,
    output wire       carry
);
    wire [8:0] sum = a + b;               // 9-bit 加法(含进位)
    wire [8:0] diff = a - b;              // 9-bit 减法(含借位)

    assign {carry, result} = (op == 2'b00) ? sum :
                             (op == 2'b01) ? diff :
                             (op == 2'b10) ? {1'b0, a & b} :
                                             {1'b0, a | b};
endmodule

这个电路会综合出：一个 8-bit 加法器（含进位链 CARRY4）、一个 8-bit 减法器、两个 8-bit 与/或门、以及一个 4:1 的 MUX（约 4 个 LUT）。不管是加法和减法，还是与和或运算，所有操作都并行计算，MUX 选择结果。这就是硬件设计的思维——用面积换延迟。

3.6 一个真实的踩坑故事：把 reg 当成"寄存器"

2016 年我被朋友公司请去"救火"——他们的 FPGA 项目，数据路径上的信号总是"延迟一拍"。我看了一眼代码，发现一个典型的初学误区：

// 问题代码——把 reg 全当成了触发器
module data_path (
    input  wire [7:0] a, b, c,
    output wire [7:0] y
);
    reg [7:0] sum, temp;

    always @(*) begin
        sum = a + b;          // sum 是组合逻辑结果
        temp = sum + c;       // temp 也是组合逻辑
    end

    assign y = temp;
endmodule

这位同事说："dout 总比预期晚一个时钟周期，我想着加一级寄存器打拍，所以把 sum 和 temp 都声明成 reg……"他以为 reg 会自动变成触发器。实际上，always @(*) 中的 reg 仍然是组合逻辑，sum 和 temp 只是 LUT 的输出。他观察到"延迟一拍"是别的原因——可能是前一模块的流水线效应。

真正的寄存器（触发器）需要 always @(posedge clk)。把 reg 和触发器等同起来，是 Verilog 学习中最常见的概念混淆之一。换成 SystemVerilog 的 logic 类型可以避免这个问题，但在工程代码中，建议保持 wire/reg 的显式声明，让代码阅读者能一眼看出这个信号的驱动方式——wire 表示被 assign 驱动或被例化端口驱动，reg 表示在 always 块中被赋值。
