第3章 推荐系统的评估体系

大家好，我是你们的讲师。前两章我们聊了推荐系统是什么、它是怎么一步步发展到今天的。这一章我们聊一个看似"不那么性感"但实际上极其关键的话题——评估体系。

我见过太多团队：花了大半年训练了一个"效果爆好"的模型，离线AUC涨了3个点，全组欢天喜地地上线，然后一看业务数据——CTR掉了。为什么会这样？因为离线指标和在线业务之间，隔着一整个评估体系的鸿沟。这一章，我们就来系统性地分析：推荐系统到底该怎么评估。

1. 离线评估指标详解

离线评估是用历史数据模拟模型的预测能力。它的优点是快速、低成本、可复现；缺点是"历史不等于未来"。我们先来看最常用的几个指标。

1.1 AUC：最常用的排序能力指标

AUC（Area Under the ROC Curve）是推荐系统中最重要的离线指标之一。它的直观含义是：随机选一个正样本和一个负样本，模型把正样本排在负样本前面的概率。

数学上，AUC = 0.5 表示模型没比随机好，AUC = 1.0 表示模型能完美区分正负样本。工业级推荐系统的AUC通常在0.65到0.85之间——超过0.9反而要警惕了，很可能是数据泄露了。

但是AUC有一个重要缺陷：它对全量样本一视同仁。如果某个用户只有1个正样本和100个负样本，AUC会把他对整体指标的贡献拉低——但也许这个用户只是不太活跃，并不意味着模型对他"失效"了。这就引出了下一个指标。

1.2 GAUC：按用户分组的AUC

GAUC（Group-wise AUC）是阿里在2017年发表的DIN论文中重点使用的指标。思路很简单：先对每个用户单独计算AUC，然后按用户的样本数量或活跃度做加权平均。

为什么需要GAUC？因为推荐系统的本质是"对每个用户做个性化排序"，而不是"全局排序"。AUC衡量的是"把所有用户的样本混在一起排序"的能力，但一个用户的正样本被排到另一个用户负样本前面的情况，在个性化推荐中完全没有意义。GAUC更能反映模型在"每个用户自己的列表上排得怎么样"。

1.3 NDCG@K：关注排序顶部的质量

NDCG（Normalized Discounted Cumulative Gain）是一个信息检索领域的重要指标。它衡量的是：模型排序中前K个结果，"好的"内容是不是排在了"差的"前面。NDCG的核心价值在于引入了位置衰减——排在第1位比排在第10位重要得多。

公式看起来有点复杂，但直觉很简单：我们给每个位置一个"权重"，越靠前权重越高，然后看模型有没有把好东西放在权重高的位置。

在精排阶段，NDCG@K比AUC更贴近实际体验。两个模型可能有相同的AUC，但NDCG@10更高的那个，用户看到的前几名结果质量更好。

1.4 召回阶段的指标：Recall@K 和 Precision@K

在召回阶段，我们用不同的指标。召回的目标是从百万级候选池中"别漏掉用户可能喜欢的"，而不是"精准排序"。

  • Recall@K：用户真正喜欢的物品中，有多少出现在了前K个召回结果中。召回的目标是"不要漏"。
  • Precision@K：前K个召回结果中，有多少是用户真正喜欢的。虽然召回阶段不太要求精准，但也不能给下游灌太多噪声。
  • HR@K（Hit Rate@K）：在K个召回结果中，至少有一个命中用户喜欢的物品的比例。这是推荐场景中最常用的召回指标。

需要注意的是，召回指标的"真实"计算需要对整个候选池做穷举，这在工业规模下不可行——你不可能把100万个物品全部送给模型算一遍。实践中通常用"采样评估"或"全量离线评估"（只对小规模实验这么做）。

2. 在线指标体系

离线指标只是"开胃菜"，真正决定推荐系统价值的，是在线业务指标。每个指标衡量用户行为的不同侧面：

指标 衡量内容 典型计算方式 优化目标
CTR（Click-Through Rate） 用户看到推荐后点击的比例 点击次数 / 曝光次数 提升点击率，短期互动
CVR（Conversion Rate） 点击后完成目标行为（购买/注册）的比例 转化次数 / 点击次数 提升最终的商业转化
停留时长（Dwell Time） 用户在推荐内容上的停留时间 会话时长或单篇内容阅读时长 提升用户参与深度
留存率（Retention） 用户次日/7日/30日是否回来 次日登录用户数 / 当日登录用户数 长期用户粘性
多样性（Diversity） 推荐内容涵盖的不同类别/主题的广度 推荐列表中不同类别的数量或熵值 避免信息茧房
新颖性（Novelty） 推荐内容是否超出用户已知的范围 用户未交互过的物品比例 促进"发现"体验
覆盖率（Coverage） 推荐系统覆盖了多少物品池中的内容 推荐过的物品数 / 总物品数 长尾物品分发机会

惨痛教训：某短视频平台的一个真实案例——团队优化了CTR，连续两周A/B实验CTR都有显著提升（+5%），团队准备全量上线。但在最后的数据复核中，发现有经验的用户"次日留存"下降了2个百分点。原因很典型：为了优化CTR，模型倾向于推荐用户已经知道会点击的"舒适区"内容（同质化的搞笑视频），但这种内容看多了用户会腻，长期来看反而导致流失。这个事情告诉我们：短期指标（CTR）和长期指标（留存）需要同时监控，任何只看单一指标就匆忙上线的做法都是危险的。

3. A/B实验设计与统计学

上线一个新模型之前，必须通过A/B测试验证。但A/B测试远比"分流看指标"复杂。

3.1 统计显著性的基本概念

A/B测试的核心逻辑是：我们把用户随机分为两组，实验组看到新模型的结果，对照组看到旧模型的结果，然后比较两组的核心指标。

  • 原假设（Null Hypothesis, H₀）：新模型和旧模型效果没有差异
  • 备择假设（Alternative Hypothesis, H₁）：新模型效果优于旧模型
  • p-value：在原假设成立的前提下，观察到当前结果（或更极端结果）的概率。如果p-value < 0.05，我们通常说"结果显著"
  • 置信区间（Confidence Interval）：给出指标差异的一个范围。比如"CTR提升 [0.3%, 1.5%] 的概率是95%"，这比单纯的"显著/不显著"提供更多信息
  • 最小可检测效应（Minimum Detectable Effect, MDE）：在给定的样本量下，能够检测到的最小真实效应。如果你的实验天数不够，可能一个真实存在的提升也被判定为"不显著"

3.2 样本量计算

一个常见的错误是：样本量不够就下结论。样本量取决于三个因素：

  • 指标本身的方差（CTR越波动，需要的样本量越大）
  • 你想要检测的最小效应（想检测0.1%的提升，比想检测1%的提升需要更多样本）
  • 显著性水平（通常是95%置信度 + 80%统计功效）

举例来说，如果你的产品日活是100万，CTR基线是5%，你想检测0.2个百分点的提升，按照标准公式大约需要200-300万样本——这意味着你需要跑实验3天以上才能得出可靠结论。

A/B测试的常见陷阱：

  • 网络效应干扰：社交类产品中，实验组用户的行为会通过社交关系影响对照组用户（比如实验组用户发了更多帖子，对照组用户刷到的内容也变多了），导致两组不独立
  • 新奇效应（Novelty Effect）：新模型上线的头几天用户因为"新鲜感"指标变好，但几周后回落
  • 多比较问题：同时看了20个指标的A/B实验，总有一个"显著"的——需要做多重比较校正
  • 早期暂停：看到"显著"就提前终止实验，这会导致最终结论不可靠

4. 离线/在线一致性困境

这是推荐系统领域最让人头疼的问题之一——"离线AUC涨了，在线CTR没涨，甚至跌了"。为什么？

4.1 覆盖偏差

离线评估用的数据是"过去模型推荐并展示给用户的内容"。但新模型上线后，它会推荐不同的内容——这些新内容在训练数据中没有被充分探索过。换句话说，离线评估只覆盖了"旧模型探索过的空间"的准确性，没法保证"新模型发现的新空间"的效果。

4.2 探索盲区

离线数据中，未被曝光的物品没有用户反馈。所以离线模型只能学会"对这些已经曝光过的物品打分"，但不知道"那些从未曝光过的物品用户是否喜欢"。这就是为什么召回阶段的新物品冷启动总是离线评估的盲区。

4.3 离线回放（Offline Replay）方法论

为了解决上述问题，学术界提出了一套更严谨的离线评估方法——离线回放（也叫Counterfactual Evaluation或Interleaved Evaluation）。基本思路是：在旧模型生成的日志数据上，通过逆概率加权（Inverse Propensity Scoring, IPS）等技术来"模拟"新模型的效果。

简单来说，就是给日志中每个样本分配一个"重要性权重"——如果新模型有很高的概率给某个物品高分，而这个物品在日志中恰好出现了，那么它的反馈就更可信。离线回放比简单的AUC+Recall评估更可靠，但实现复杂度也高得多，在工业界还未普及。

import numpy as np
from sklearn.metrics import roc_auc_score

def compute_auc(y_true, y_pred):
    """计算AUC —— 随机选一个正样本排在负样本前面的概率"""
    return roc_auc_score(y_true, y_pred)

def compute_ndcg(y_true, y_pred, k=10):
    """
    计算NDCG@K
    y_true: 真实相关度（0/1或连续值）
    y_pred: 模型预测分数
    k: 只看前k个
    """
    # 按预测得分降序排列
    order = np.argsort(y_pred)[::-1]
    y_true_sorted = y_true[order][:k]

    # DCG: Discounted Cumulative Gain
    dcg = np.sum((2 ** y_true_sorted - 1) / np.log2(np.arange(2, k + 2)))

    # IDCG: Ideal DCG（按真实相关度降序排列时的DCG）
    ideal_order = np.argsort(y_true)[::-1]
    y_true_ideal = y_true[ideal_order][:k]
    idcg = np.sum((2 ** y_true_ideal - 1) / np.log2(np.arange(2, k + 2)))

    return dcg / idcg if idcg > 0 else 0.0

# 示例
y_true = np.array([1, 0, 1, 0, 0, 1, 0, 0, 0, 0])
y_pred = np.array([0.9, 0.1, 0.8, 0.3, 0.2, 0.7, 0.4, 0.05, 0.15, 0.01])
print(f"AUC: {compute_auc(y_true, y_pred):.4f}")
print(f"NDCG@5: {compute_ndcg(y_true, y_pred, k=5):.4f}")
print(f"NDCG@10: {compute_ndcg(y_true, y_pred, k=10):.4f}")

警告：不要盲目追求离线指标！

我见过一个团队花了一个季度优化离线AUC，从0.72涨到了0.80，全组感觉"稳了"。上线一跑A/B，CTR反而跌了。排查发现：模型学会了利用数据中的"未来信息"——用户第5天买了的东西，在第3天的历史行为中正好出现过类似商品，这本身是合理的；但问题是，训练数据中第3天的特征包含了第5天的统计信息（由于数据拼接时的错误），导致了时间泄露。离线AUC看起来很漂亮，但那个"好效果"是不可复现的幻觉。永远记住：离线评估告诉你的不是"模型好不好"，而是"在历史上这个模型会有多好"——这两者有本质区别。

5. 构建评估体系的黄金法则

总结一下，一个完整的推荐系统评估体系应该包含四层：

  • 第一层：离线指标（AUC, GAUC, NDCG, Recall@K）——快速迭代，发现问题
  • 第二层：离线回放（Counterfactual Evaluation）——更接近在线效果的离线评估
  • 第三层：A/B测试（统计显著性检验）——短期核心指标验证
  • 第四层：长期线上监控（留存、多样性、覆盖率）——长期效果和副作用监控

每一层都有它的价值，但也都有它的盲区。优秀推荐团队的决策总是"综合四层信息"做出的，而不是只看单一指标。

好了，这一章的内容就到这里。我们从AUC聊到了NDCG，从A/B测试的统计学原理聊到了离线/在线一致性困境。评估体系是整个推荐系统的"指挥棒"——指标设对了，团队的优化方向就对了。现在，我们已经知道"推荐系统是什么"、"它怎么来的"、"怎么评估它好坏"，下一章我们就要进入具体的算法了——第4章：协同过滤，推荐系统最经典的基石。你会发现，这个1990年代发明的算法到今天依然充满生命力。
