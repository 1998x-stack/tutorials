从手工到自动化

大家好。过去五年间，LLM 数据工程经历了一场静悄悄的革命。2019-2022 年，数据团队的日常工作可以用一个词概括——"手搓"。数据工程师们手动编写 Python 脚本，一条一条地添加过滤规则：去掉行数太少的文档，去掉标点符号占比过高的文档，去掉重复的 n-gram 片段……每个规则都有一个阈值，每个阈值都需要人工反复调优。

我认识一位在头部大厂负责数据清洗的工程师，他告诉我一个真实的经历：2022 年他们团队花了两周时间调优一个困惑度阈值，从 80 调到 100，又从 100 调到 90，最后发现最佳值还是 80。但六个月后换了新的训练目标，最佳阈值又变了。这种"人工网格搜索"式的调参模式，在数据量从 TB 级增长到 PB 级的今天，已经越来越不可持续。

进入 2025-2026 年，一个新的范式正在形成：用 AI 来构建 AI 的数据流水线。自动化数据准备系统不再是"能不能做"的问题，而是"不做就会掉队"的问题。Meta、Google、Anthropic 等前沿实验室都已将自动化数据管线列为基础设施建设的核心方向。

2026 年初，一个名为 DataEvolver 的系统引起了业界广泛关注。这是第一个被公开报道的"自进化数据准备系统"——它像 AutoML 自动搜索最佳模型架构一样，自动搜索最佳的数据清洗流水线。下面我们就深入分析这个系统的核心思想。

DataEvolver 的核心思想

DataEvolver 的设计哲学非常简洁：将数据准备过程拆解为最小的原子操作，然后让系统自动探索这些操作的最佳组合方式。它采用了三层抽象架构。

第一层：操作符层（Operator Layer）。 操作符是数据变换的原子单位。每个操作符做一件事：过滤（filter）、清洗（clean）、增强（augment）、改写（rewrite）。例如，一个"MinHash 去重"操作符接收文档集，输出去重后的文档集；一个"困惑度评分"操作符为每个文档附加一个 perplexity 分数。每个操作符都有可配置的超参数——阈值、窗口大小、签名数量等。

第二层：流水线层（Pipeline Layer）。 多个操作符编排成一个有向无环图（DAG），构成完整的数据准备流水线。操作符的执行顺序直接影响最终数据质量——比如先过滤再做困惑度评分与先评分再过滤，结果可能截然不同。

第三层：自进化层。这是最关键的创新。 系统自动评估当前流水线的效果——通过下游任务性能、数据多样性指标、保留率等多个维度——然后调整操作符的选择、顺序和超参数，进入下一轮迭代。这就像一个超参数优化算法，但优化的是整个数据流水线而非单个模型。

以下是 DataEvolver 概念性配置的一个示例：

# DataEvolver 概念性配置（YAML 风格）
pipeline:
  operators:
    - name: language_filter
      params: { target_langs: ["zh"], threshold: 0.8 }
    - name: heuristic_filter
      params: { min_chars: 200, max_line_ratio: 0.3 }
    - name: exact_dedup
      params: { hash: "sha1" }
    - name: minhash_dedup
      params: { num_perm: 128, threshold: 0.85 }
    - name: ppl_scorer
      params: { model: "kenlm-zh", threshold: 80 }
  evolution:
    search_strategy: "bayesian_optimization"
    eval_metric: "downstream_accuracy"
    max_iterations: 50
    diversity_constraint: 0.2
维度传统手动方式自动化系统（DataEvolver）
规则调优人工网格搜索，每次数天贝叶斯优化自动搜索，每次数小时
操作符编排固定顺序，凭经验设计自动探索最优执行顺序
迭代周期周级别小时级别
可复现性依赖工程师记录系统自动记录每次迭代的完整配置
人力投入全职数据工程师团队自动化系统 + 少量监督
最佳方案发现依赖个人经验系统化搜索，减少遗漏
关键洞察： DataEvolver 的"自进化"能力并不意味着完全取代人类。它更像一个智能助手——自动探索工程师不会想到的操作符组合，加速了"设计→评估→优化"这个循环，但最终的数据质量验证和业务对齐仍然需要人类的判断。

Agent-Driven 数据工程

DataEvolver 展示了自动化数据管线的可能性，但 2025-2026 年的另一个前沿方向更进一步：用 LLM agent 来管理数据管线。如果说 DataEvolver 是"自动搜索"，那 Agent-Driven 数据工程就是"理解意图再行动"。

想象这样一个场景：你对系统说"帮我把这批数据清洗一下，目标是减少重复数据并提升中文内容的质量"。一个 LLM agent 理解你的意图后，自动生成一个数据准备计划——调用 URL 过滤操作符先过滤垃圾域名，接 MinHash 去重，再用中文分类器评分，最后通过困惑度过滤。整个过程不需要你了解每个操作符的具体参数。

目前，这类系统的典型架构是三层设计：

  • Agent Layer（智能体层）： 接收用户的自然语言指令，解析意图，分解为子任务，生成执行计划。这层通常由一个经过工具使用微调的 LLM 驱动。
  • Data Operator Layer（数据操作层）： 注册了所有可用的数据操作符及其输入输出接口。Agent 通过函数调用（function calling）来编排这些操作符。
  • Data Serving Layer（数据服务层）： 负责数据存储、版本管理、缓存加速。确保 Agent 生成的流水线可以高效执行。
一个提醒： Agent-driven 数据工程不是要取代数据工程师。恰恰相反，它的目的是把工程师从"每天调阈值"的重复劳动中解放出来，让他们可以专注于更高级的工作——设计新的数据操作符、分析数据质量的根本问题、制定数据策略。AI 做"怎么做"，人类思考"做什么"和"为什么做"。

自动化的关键挑战

自动化听起来很美好，但实践中面临的挑战同样严峻。我在和几位一线团队的交流中总结了三个最核心的问题。

挑战一：评估标准不统一。 自动化系统需要明确的优化目标，但"好的训练数据"本身就是一个难以精确定义的概念。是用下游任务准确率来评估？还是用数据多样性指标？或者是两者的加权组合？不同的评估标准会导向完全不同的数据流水线。更麻烦的是，下游任务的性能评估本身就需要大量计算资源——跑一次评测可能需要数百 GPU 小时。

挑战二：自动化 vs 可解释性的权衡。 一个 AutoML 风格的搜索过程可能找到一组表现很好的操作符组合，但很难解释"为什么这个组合比另一个好"。这在需要合规审计的场景中尤为棘手——监管机构可能要求你解释"为什么移除了这批文档"。Dolma 项目倡导的透明可复现精神，在自动化系统中需要付出额外的工程努力才能实现。

风险警示： 纯指标驱动的自动优化存在一个严重的隐患——局部最优陷阱。系统可能发现"把困惑度阈值设到 50 可以提升下游任务 2%"然后开始积极淘汰所有困惑度超过 50 的文档。但这样做会系统性地丢失高难度、高信息密度的文档（比如学术论文、技术文档的复杂段落），导致数据多样性大幅下降。2026 年的解决方案是在自动化优化中加入多样性约束和人工审核节点——在每轮优化后，抽样检查被过滤掉的数据，确保没有"误杀"有价值的内容。

挑战三：计算成本。 自动化搜索需要反复评估不同流水线的效果，这本身就是计算密集型的。一次完整的 DataEvolver 搜索（50 轮迭代）可能需要数万 GPU 小时。目前业界的主流做法是将自动化搜索限定在核心环节——比如过滤阈值和去重参数——而不是对整个流水线做全局搜索，以此控制成本在可接受范围内。

未来展望

展望未来，自动化数据准备系统有几个确定性的发展方向。

第一，从 batch 处理走向 streaming 处理。 当前的大多数数据准备流水线仍然是 batch 模式——下载一批数据，清洗，存储。但未来的 LLM 训练需要持续的数据供给，自动化系统需要支持增量式、流式的数据处理。当一个新来源的数据不断涌入时，系统可以自动评估其质量并动态调整处理策略。

第二，从"建一次用多次"到"持续优化"。 传统的数据项目是一次性的——建好一个数据集，反复用于训练。但 2026 年的前沿实践表明，最佳的训练效果来自数据的持续迭代更新。自动化系统可以监测模型训练的实时指标（如 loss 曲线、梯度变化），反向指导数据清洗策略的调整。

第三，自动化与在线数据选择的融合。 自动化数据准备解决的是"数据进入训练集之前"的问题，而在线数据选择解决的是"训练过程中如何动态采样"的问题。这两个方向的融合是 2026-2027 年的重要趋势——一个完整的自动化数据系统应该既能离线优化清洗流水线，也能在线调整训练时的数据采样分布。这正是我们下一章要深入探讨的主题。
