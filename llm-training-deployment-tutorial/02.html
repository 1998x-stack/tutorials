第2章 预训练数据工程

大家好，我是你们的讲师。上一章我们聊了大模型训练的全景图，知道了预训练是整条链路的基石。但预训练的第一步是什么？不是写模型代码，不是配分布式集群——是搞数据。如果你去问任何一个大模型团队的负责人"训练中最耗时、最容易被低估的环节是什么"，十有八九的答案都是数据。

我认识一个做医疗大模型的团队，光数据清洗就花了六个月。他们从医院拿到的原始病历数据，有GBK编码的、有UTF-8编码混着Big5的，有同一份文档被OCR识别了三遍产生三个不同版本的，还有大量包含患者姓名、身份证号的PII数据需要脱敏。等数据终于洗干净了，训出来的模型在医疗问答上表现还不如直接用GPT-4。为什么？因为数据质量比数据量更重要。混入10%的低质量数据，效果可能下降30%。所以这一章，我们来拆解预训练数据工程的每一个环节。

2.1 数据从哪里来？

训练一个大模型需要多少数据？当前主流模型大约要用5-15万亿token。这么多数据不可能靠人工编写，必须从各种公开和私有数据源大规模采集。

Common Crawl是最大的公开数据源，每月抓取数十亿网页，原始数据量在PB级别。它基本上是所有大模型的"主食"，占比通常在60-80%。但Common Crawl的原始数据质量参差不齐——有大量机器生成的垃圾页面、重复内容、乱码文本。因此需要经过多轮清洗才能使用。

书籍和学术论文提供了高质量的长文本数据。Google的BooksCorpus、arXiv论文、PubMed文献等都是经典来源。书籍的优势在于语言规范、逻辑连贯、知识密度高。但版权问题需要谨慎处理，现在很多团队选择采购授权数据而非爬取。

代码数据在代码模型和通用模型中越来越重要。GitHub上公开的代码仓库、Stack Overflow问答等都是优质来源。代码数据不仅用于培养代码生成能力，还能增强模型的逻辑推理能力和结构化理解能力。DeepSeek-V2使用了超过1万亿token的代码数据进行训练，效果显著。

Wikipedia虽然总量不大（约40-60亿token），但质量极高，几乎是所有模型的"必选菜"。它起到了知识锚点的作用，帮助模型建立对世界事实的基准理解。

多语言数据的采集是一个被严重低估的挑战。中文互联网数据量巨大但噪声也极高——自媒体文章、营销号内容充斥着各种夸张表述和虚假信息。高质量的日文、韩文、阿拉伯文数据则更难获取。很多模型在非英语语言上的表现不佳，根因不在于模型架构不行，而在于训练数据中该语言的占比和质量的不足。

数据源 典型占比 质量等级 主要挑战 典型处理方式
Common Crawl 60-80% 低→高（取决于过滤） 噪声多、重复高 多轮过滤+去重
书籍/论文 5-15% 高 版权、格式不一致 格式统一+版权过滤
代码 5-20% 中→高 语法校验、许可证合规 编译验证+license过滤
Wikipedia 1-3% 极高 覆盖范围有限 基础清洗即可
社交媒体 0-5% 低 噪声极大、事实性差 多数团队弃用

2.2 数据清洗管线

从互联网上扒下来的原始数据，直接喂给模型就是灾难。你需要一条完整的数据清洗管线。下面是我在实践中总结的典型流程，按处理顺序排列。

第一步：语言检测。你得知道一段文本是中文、英文还是乱码。常用的工具有fastText的语言检测模型或pycld3。这里有个容易被忽略的细节：不要把"置信度不够高"的文本直接丢掉——很多数学公式、代码片段会被语言检测器误判为"无语言"而丢弃。建议对低置信度文本用更宽松的阈值或做二次分类。

第二步：PII过滤。PII（Personally Identifiable Information）包括身份证号、手机号、邮箱、银行卡号等。这不是可选项——根据《个人信息保护法》，使用包含PII的数据训练模型可能构成违法行为。常用的方法是正则匹配+NER模型辅助检测。但注意不要过度过滤：把"请联系邮箱example@test.com"这种测试占位符也删掉没问题，但把"192.168.1.1"当PII过滤掉就尴尬了。

第三步：HTML标签清洗。Common Crawl的原始数据保留了大量HTML标签。需要把tag去掉、把转义字符还原。但这个环节有个常见坑：很多页面的正文内容嵌在复杂的DOM结构中，简单remove tag会把正文也弄丢。因此最好使用boilerplate removal工具（如trafilatura、justext）来提取正文，而不是粗暴地strip tag。

第四步：质量分类器。这一步的核心是训练一个二分类模型来区分"高质量文本"和"低质量文本"。通常的做法是：以Wikipedia、Gutenberg书籍等作为正样本，以机器生成的垃圾内容作为负样本，训练一个fastText或BERT分类器。LLaMA团队在论文中透露，他们的质量分类器过滤掉了Common Crawl中大约80%的页面——也就是说，你抓回来的100页里只有20页能用。

第五步：困惑度（Perplexity）过滤。用一个已有的语言模型计算文本的perplexity。困惑度太高的文本通常意味着它包含大量语法错误、乱码或者非自然语言内容。困惑度太低则可能意味着文本过于简单（如儿童读物）或存在大量重复。一般设定一个双端阈值，比如保留PPL在15-100之间的文本。

# 典型的数据清洗管线伪代码
def clean_pipeline(raw_docs):
    # 1. 语言过滤
    docs = [d for d in raw_docs if detect_lang(d.text) == "zh"]

    # 2. PII脱敏
    docs = [mask_pii(d) for d in docs]

    # 3. 正文提取 + HTML清洗
    docs = [extract_main_content(d) for d in docs]

    # 4. 质量分类
    quality_filter = load_classifier("quality_model.bin")
    docs = [d for d in docs if quality_filter(d.text) > 0.5]

    # 5. 困惑度过滤
    ppl_model = load_lm("small_lm.pt")
    docs = [d for d in docs if 15 < ppl_model.ppl(d.text) < 100]

    # 6. 去重（下节详解）
    docs = deduplicate(docs)

    return docs

2.3 去重技术深入

数据去重可能是数据工程里最容易被低估的技术环节。直觉上你会觉得"重复数据不是坏事，顶多多学几遍"，但实验结果恰恰相反——重复数据会导致模型在重复模式上过拟合，降低泛化能力，甚至损害模型的知识广度。

精确去重（Exact Dedup）是最简单的方法：计算每个文档的hash值（如SHA-256），删除hash完全相同的文档。但精确去重只能解决完全相同的副本，解决不了"几乎相同"的文档——比如同一篇新闻在不同网站上的转载，内容95%相同但有细微差异。这时候就需要模糊去重。

MinHash LSH是当前最主流的模糊去重方法。它的核心思想是：将文档转换成一组fingerprint（签名），如果两个文档的签名集合有足够多的重叠，就认为它们是近似重复的。具体来说，先把文档分词后得到n-gram集合，然后用多个随机hash函数对这个集合进行最小hash操作，生成一个签名向量。最后用LSH（Locality Sensitive Hashing）将这些签名分桶，桶内两两比较相似度。

这里有个工程实践层面的关键参数：n-gram的大小和阈值的选择。n=5对于自然语言文档效果较好，但对于代码数据，n=10或更高才能避免误判（两个代码文件有相同的import语句不代表它们是重复的）。相似度阈值设在0.7-0.8是常见选择，但要根据你的数据领域做调优。

去重对下游性能的实际影响非常显著。Raffel et al. (2020) 在C4数据集的实验表明，不做去重时，模型在多个NLU任务上的表现显著低于做过去重的版本。更极端的案例来自GPT-3论文：不做去重的模型在训练过程中出现了明显的"重复记忆"现象——它会一字不差地背诵训练集中的段落。而在代码领域，重复代码数据还会导致模型容易生成"样板代码"而非真正的解决方案。

踩坑故事：我一个朋友所在的团队曾经在一个中文大模型项目上，用从网上收集的30TB"清洗后"数据直接开始训练。训练了两周后，他们发现验证集loss不再下降，而且模型生成的文本经常出现大段重复。排查了一个星期才发现问题：他们的去重管线里MinHash的n-gram设成了3（三字符），导致几乎所有中文文本都被判定为相似——因为中文中"的""了""是"这些高频字让所有文档的签名都趋于一致。结果就是把大部分唯一的内容也误删了，造成了严重的数据多样性损失。后来把n-gram改成6，重新跑了一遍去重，效果恢复正常。

2.4 数据配比策略

当你的数据源和清洗管线都搭建好了，下一个关键问题是：不同领域的比例应该是多少？这就是数据配比（Data Mixing）要解决的问题。

直觉上你会觉得"按自然界的自然分布来就好"——互联网上有多少英语文本、多少代码、多少数学内容，就按这个比例采样。但实际经验告诉我们，最优配比和自然分布存在系统性的偏差。代码和数学数据即使在自然分布中占比不高，也需要大幅提高它们的采样权重，因为这两种数据类型对模型的推理能力增益极大。

LLaMA系列的数据配比提供了一个很好的参考：英语文本约占67%、代码约占15%、多语言文本（含中文、日文、法文等）约占10%、学术论文和书籍约占8%。而DeepSeek系列则在代码和数学上给了更高的权重，因为他们发现高质量的代码+数学训练数据可以直接提升模型在所有任务上的表现——这就是所谓的"推理能力迁移"。DeepSeek-V2的训练数据中，代码+数学占比超过了30%。

DoReMi方法是2023年提出的一种自动数据配比优化方法。其核心思想是：先用一个小模型在不同的数据配比下训练，然后使用一个参考模型（如固定权重的预训练模型）来评估每个领域的训练难度，自动调整配比让各领域"均衡受益"。简单说，如果模型在数学数据上进步快、在新闻数据上进步慢，就多给数学数据配比。实验结果表明，使用DoReMi优化的配比方案在同等数据量下，模型的整体性能提升了2-7个百分点。

实践技巧：如果你没有资源做DoReMi这种需要额外训练的配比优化，一个简易但有效的方法是"领域加权上采样"——对推理密集型领域（数学、代码、科学）给予2-3倍于自然分布的采样权重，对容易导致overfitting的低质量领域（社交媒体、营销内容）降低权重甚至完全排除。在开始大规模训练前，用一个小模型（如1B参数量）跑几个不同的配比方案，每个方案训练10B token左右，然后在下游任务上评估，选最优配比。

2.5 数据质量评估

最后一个问题：你怎么知道你的数据是好的？等模型训完再评估就太晚了。我们需要在数据阶段就对质量有定量的评估。

度量指标1：Perplexity分布。用一个训练好的reference LM（如已有的开源模型）计算数据集中每个文档的perplexity，观察整个数据集的PPL分布。理想的分布应该是单峰且峰值在15-30之间（对于英文数据）。如果出现双峰分布，说明数据可能包含两种质量差异巨大的子集，需要进一步排查。

度量指标2：信息熵。计算每个文档的n-gram熵。熵值过低意味着文本高度重复（如论坛上的"+1""顶""支持"等无意义回复），过高则意味着随机性过强（乱码）。对于中文数据，字符级bigram熵在6-8 bits是健康的范围。

度量指标3：下游任务代理评估。这是最可靠的评估方法。用你的数据的一小部分（如50B token）训练一个小模型（如1.5B参数），然后在多个下游benchmark上评测。具体做法是：准备2-3个不同配比或不同处理方式的数据版本，分别训练小模型，对比它们在MMLU、HellaSwag、ARC等任务上的表现。如果某个数据版本在所有任务上表现都差，就说明数据有问题。这种方法成本可控（训练1.5B模型只需要几十张GPU·小时），但信息量极大。

# 数据质量评估流程
def evaluate_data_quality(dataset_path, ref_model="gpt2"):
    # 1) 计算PPL分布
    ppls = []
    for batch in data_loader(dataset_path):
        loss = ref_model.compute_loss(batch)
        ppls.append(torch.exp(loss))
    print(f"PPL分布: mean={mean(ppls):.1f}, p90={percentile(ppls, 90):.1f}")

    # 2) 信息熵检查
    entropies = compute_ngram_entropy(dataset_path, n=5)
    low_entropy_ratio = sum(e < 4.0 for e in entropies) / len(entropies)
    print(f"低熵文档比例（可能重复）: {low_entropy_ratio:.1%}")

    # 3) 小模型训练验证（最优方案）
    trainer = Trainer(model=SmallLM(1.5e9), data=dataset_path)
    trainer.train(steps=10000)
    scores = evaluate_on_benchmarks(trainer.model)
    return scores

还有一个容易被忽视的维度：数据的新鲜度。如果你的模型要在2026年的使用场景中表现良好，训练数据中2025-2026年的内容应该占一定比例。常识、事实性知识和流行文化都在快速变化。想象一下，如果训练数据只截止到2022年，用户问"DeepSeek-R1是什么"时，模型可能完全不知道。一般建议保留5-10%的训练预算给最新数据。

核心要点回顾：数据工程不是一次性的工作，而是一个持续迭代的过程。好的数据团队会不断地做"数据版本实验"——改进去重参数、调整配比权重、清洗管线增加新步骤，每次改动后都用小模型验证效果。这听起来繁琐，但比起直接花几百万美元跑一次大规模训练才发现问题，成本低了不止一个量级。
