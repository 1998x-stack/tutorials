第26章 V3.1/V3.2演进

想象一下：你手里有两位顶级专家。一位是推理大师，解数学题、写代码样样精通，但每次回答问题都要先花几分钟"思考人生"——即使只是问现在几点。另一位是快枪手，日常对话对答如流，但遇到复杂的逻辑推理问题就容易"翻车"。在过去，你得分别请这两位专家，按需切换。

DeepSeek 在 2025 年做了一件大胆的事：他们把这两位专家"合并"成了同一个人。这就是 V3.1 混合推理架构的核心思路——让同一个模型既能快速应答日常问题，又能深度思考复杂推理，用户按需选择。而在随后的 V3.2 中，他们更进一步，用动态稀疏注意力重新定义了长上下文推理的效率边界。本章我们就来深入拆解这 2025 年的两次重要迭代。

26.1 从 V3 到 V3.1

故事要从 2025 年 3 月说起。DeepSeek 发布了 V3-0324 版本，这实际上是 V3 的增强版——它将 R1 的知识蒸馏能力融入了 V3 的底座之中，使得非推理模式下的数学和编程能力有了显著提升。这个版本在开发者社区获得了非常好的口碑，许多人认为它是当时"性价比最高的开源模型"。

但 DeepSeek 团队很清楚，这只是权宜之计。R1 的推理能力来自于 GRPO 强化学习训练出的思维链（Chain-of-Thought），而 V3 的快速响应能力来自于传统的 SFT 训练。两者从根本上就是不同的推理范式，简单地将 R1 的知识蒸馏到 V3 中，并不能让 V3 真正学会"深度思考"。

关键洞察：知识蒸馏只能迁移"答案"，无法迁移"思考过程"。真正的混合推理需要一个模型同时掌握两种推理模式，并在运行时根据任务特性自适应选择。

于是，2025 年 8 月 21 日，DeepSeek 发布了 V3.1——一个真正实现了混合推理架构的统一模型。这一版本标志着 DeepSeek 从"独立推理模型"时代进入了"融合推理"时代。

V3.1 的技术报告中有几组数据非常亮眼：

评测基准 R1-0528 V3-0324 V3.1
SWE-bench Verified 44.6% 45.4% 66.0%
Terminal-Bench 5.7 — 31.3
Aider 编程评测 — — 71.6%
MMLU — — 88.5%
BrowseComp 8.9 — 30.0

SWE-bench 从 45.4% 跃升到 66.0%，Terminal-Bench 从 5.7 暴涨到 31.3——这些不是渐进式的改进，而是代际式的飞跃。正如 DeepSeek 团队所说："V3.1 不是 V3 的增强版，它是一个新物种。"

26.2 混合推理架构

混合推理架构（Hybrid Reasoning Architecture）是 V3.1 最核心的贡献。在 V3.1 之前，DeepSeek 维护着两条独立的模型线：deepseek-chat（快速响应）和 deepseek-reasoner（深度推理）。用户需要根据任务性质手动选择模型。这不仅增加了用户的决策成本，也意味着公司需要维护两套模型权重和推理服务。

V3.1 将两条线合并为一条。合并后的模型支持两种模式：

  • Think Mode（推理模式）——模型内部生成完整的 <think>...</think> 思维链，适合数学证明、复杂编程、逻辑推理等任务。在 API 中对应 deepseek-reasoner。
  • Non-Think Mode（非推理模式）——模型直接生成回答，跳过思维链，适合日常对话、信息检索、内容摘要等任务。在 API 中对应 deepseek-chat。

在 Web 端和 App 端，DeepSeek 提供了一个"深度思考"（DeepThink）开关。用户关闭它时，模型以非推理模式快速响应；打开它时，模型切换到推理模式。这种设计比 GPT-5 的自动路由策略更加透明——用户拥有控制权，而不是让模型自行判断是否需要思考。

// 非推理模式 API 调用 (Non-Think)
curl https://api.deepseek.com/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "model": "deepseek-chat",
    "messages": [{"role": "user", "content": "什么是光的波粒二象性？"}]
  }'

// 推理模式 API 调用 (Think)
curl https://api.deepseek.com/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "model": "deepseek-reasoner",
    "messages": [{"role": "user", "content": "证明质数有无穷多个"}]
  }'

在幕后，V3.1 是通过一种称为"混合训练"（Hybrid Training）的方法实现的。训练数据同时包含推理样本（带 <think> 标签）和非推理样本（不带标签），模型学习在两种模式之间灵活切换。更关键的是，推理模式下的思维链被大幅压缩——相比 R1，V3.1 的推理输出 token 数减少了 20%-50%，大幅降低了推理延迟和成本。

有趣的花絮：V3.1 还在模型中内置了搜索能力。通过新增的 <|search_begin|> 和 <|search_end|> 特殊 token，V3.1 可以在推理过程中主动发起网络搜索，获取实时信息后继续推理。这类似于 Gemini 2.5 的"深度搜索"能力，但 DeepSeek 在一个开源模型中实现了它。

26.3 DSA 动态稀疏注意力

如果说 V3.1 的重心是"推理融合"，那么 V3.2 的重心就是"效率革命"。2025 年 12 月 1 日发布的 V3.2，带来自注意力机制问世以来最大胆的改进之一——DSA（DeepSeek Sparse Attention，动态稀疏注意力）。

要理解 DSA 为何重要，我们需要先回顾一个根本性的矛盾：标准的多头注意力（MHA）的计算复杂度是 O(L²)，其中 L 是序列长度。这意味着上下文长度每翻一倍，注意力计算的耗时就要翻四倍。在 128K 上下文下，注意力计算占据了整个推理延迟的 60% 以上。

DSA 的思路很简单但深刻：大多数 token 之间的注意力分数其实低得可怜。只有少数 token 才对当前输出有实质影响。既然如此，为什么要计算所有 token 之间的注意力？

DSA 采用了一个两阶段的稀疏化方案：

  • 第一阶段：闪电索引器（Lightning Indexer）——一个轻量级的打分模块（仅 64 个 head，FP8 精度），为每个查询 token 计算历史所有 token 与它的相关性分数。
  • 第二阶段：Top-K 选择——对于每个查询 token，只保留分数最高的 2,048 个 token（无论上下文有多长），然后仅在这些选中的 token 上执行完整的注意力计算。

这个设计将复杂度从 O(L²) 降到了 O(L·k)，其中 k=2,048 是一个恒定值。当上下文长度为 128K 时，DSA 的计算量只有标准注意力的 1.6%。

上下文长度 标准注意力计算量 DSA 计算量 节省比例
8K 64M 16M 75%
32K 1,024M 66M 93.6%
128K 16,384M 262M 98.4%

但稀疏化必然带来信息损失。为了将损失降到最低，DeepSeek 采用了两阶段训练策略：第一阶段冻结主模型，只训练索引器（1,000 步，21 亿 token）；第二阶段解冻所有参数联合训练（15,000 步，9,437 亿 token），且索引器和主模型的梯度互不干扰。

最终结果令人惊喜：稀疏版本的 V3.2 在几乎所有评测上与密集版本的 V3.1-Terminus 表现持平甚至略优：

  • MMLU-Pro：85.0 vs 85.0（完全相同）
  • GPQA-Diamond：79.9 vs 80.7（相差不到 1%）
  • AIME 2025：89.3 vs 88.4（稀疏版本反而更高）
  • Codeforces Rating：2,121 vs 2,046（稀疏版本高出 75 分）
直觉：为什么稀疏注意力有时反而更好？一个可能的解释是——稀疏化强制模型关注真正重要的信息，起到了某种"注意力正则化"的效果，类似于 Dropout 在防止过拟合方面的作用。

26.4 Agent 能力强化

V3.2 的另一大主题是 Agent 能力的全面强化。DeepSeek 团队建立了一个大规模的 Agent 任务合成管线——覆盖 1,800 多个虚拟环境、85,000 条 Prompt，涵盖了从简单的 API 调用到复杂的多步骤任务规划。

具体来说，V3.2 在以下几个 Agent 维度上做了专项优化：

  • Function Calling——严格模式函数调用正式上线，支持嵌套调用和并行工具调用。V3.1 的 Beta API 在 V3.2 中正式 GA。
  • Code Agent——模型不仅能生成代码，还能在沙箱环境中执行代码、查看错误输出、自动修复 bug，形成"生成-执行-调试"闭环。
  • Search Agent——利用内置的搜索 token，模型可以自主发起搜索请求、阅读搜索结果、综合信息回答问题。

一个典型的 Code Agent 工作流程长这样：

用户: "写一个 Python 脚本，从某个 API 获取股票数据并绘制 K 线图"

Agent 思考:
<think>
1. 需要选择一个免费的股票 API（比如 yfinance）
2. 用 matplotlib 绘制 K 线图
3. 需要处理日期格式和图表的标注
我先试试写一个基础版本
</think>

Agent 执行:
> 生成代码 → 执行 → 报错: yfinance 未安装
> 自动安装依赖 → 重新执行 → 成功
> 检查输出图像 → 发现 X 轴日期格式混乱
> 修改代码 → 重新执行 → 图像正常

Agent 回答:
"脚本已写好并验证通过。使用 yfinance 获取数据，
用 mplfinance 绘制了最近 30 个交易日的 K 线图。"

这种"思考-执行-验证"的循环，让 V3.2 在 SWE-bench 上从 V3.1 的 66.0% 进一步提升到了接近 72%，在 Terminal-Bench 上也继续保持领先。

局限性提醒：Agent 能力越强，对安全对齐的要求就越高。V3.2 在 Agent 模式下引入了额外的安全约束层，防止模型在自主执行代码时做出危险操作。在部署 V3.2 Agent 应用时，一定要记得在沙箱环境中运行。

26.5 V3.2-Speciale

在 V3.2 发布的同时，DeepSeek 还推出了一个"特别版"——V3.2-Speciale。这个版本不是为了日常使用而设计的，它的目标非常纯粹：在顶级竞赛中拿到金牌。

Speciale 采用了极端的推理增强策略——它允许模型生成超长的思维链，在数学和编程问题上花更多时间"深思熟虑"。这种"慢思考"模式在竞赛场景中极其有效：

竞赛 V3.2-Speciale 成绩 对标人类
IMO 2025（国际数学奥林匹克） 金牌（35/42 分） 金牌分数线
CMO 2025（中国数学奥林匹克） 金牌 —
IOI 2025（国际信息学奥林匹克） 金牌（492/600 分） 约第 10 名
ICPC 2025 世界总决赛 金牌 约第 2 名

在 AIME 2025 上，Speciale 达到了 96.0% 的准确率，超越了 GPT-5-High 的 94.6% 和 Gemini-3.0-Pro 的 95.0%。在 HMMT（哈佛-MIT 数学锦标赛）上更是达到了惊人的 99.2%。

Speciale 的成功来自于两个关键技术的结合：一是 V3.2 本身的 DSA 架构提供了足够长的"思考空间"；二是从 DeepSeek-Math-V2 继承的定理证明能力，让模型学会了真正的数学推理——不是匹配模式，而是构造证明。

有意思的细节：Speciale 目前仅通过 API 提供，不支持工具调用模式。DeepSeek 的工程师解释说，Speciale 的"极致推理"模式会生成极其冗长的思维链，如果同时启用工具调用，输出长度可能会突破数万 token 的限制。他们将其定位为"研究原型"而非生产级模型。

26.6 SWE-bench 提升之路

让我们用 SWE-bench 这条曲线来串联 V3 → V3.1 → V3.2 的 Agent 能力演化。SWE-bench 是一个评测模型解决真实 GitHub Issue 的基准：模型需要阅读代码仓库、定位 bug、生成修复代码。这是衡量模型真正编程能力的关键指标。

  • V3（2024.12）：SWE-bench 约 42%。模型能够理解代码，但修复的准确率一般。
  • R1（2025.01）：SWE-bench 约 44.6%。推理能力增强了代码分析，但改善幅度有限。
  • V3-0324：SWE-bench 约 45.4%。R1 知识蒸馏带来小幅提升。
  • V3.1（2025.08）：SWE-bench 66.0%。混合推理架构让模型能够在编码任务中灵活切换深度思考模式，准确率跃升。
  • V3.2（2025.12）：SWE-bench ~72%。Agent 管线 + DSA 稀疏注意力进一步提升了模型对长代码上下文的处理能力。

从 42% 到 72%，一年时间提升了 30 个百分点。这背后不是简单的模型参数规模增加，而是架构和训练方法的系统性进化——混合推理解决了"思考深度"问题，DSA 解决了"长代码理解"问题，Agent 管线解决了"自主修复"问题。

感悟：V3.1 和 V3.2 向我们展示了"渐进式创新"的力量。在 V3 这样一个已经非常优秀的基座上，DeepSeek 通过架构融合和效率优化，仅用 12 个月就将模型能力推到了一个全新的高度。对于正在做 AI 应用开发的你来说，这是一个重要的启示——别总想着从头造轮子，找到正确的优化方向，日拱一卒，效果往往比大刀阔斧的更可持续。

V3.1 和 V3.2 为 DeepSeek 在 2025 年画上了一个漂亮的句号。但团队并没有止步于此——在 2026 年 4 月，一个更加雄心勃勃的版本横空出世。下一章，我们将走进 DeepSeek-V4 的世界，看看 1.6 万亿参数、百万 token 上下文、以及华为昇腾 950 适配是如何实现的。
