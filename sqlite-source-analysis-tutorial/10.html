第10章 表达式编译

大家好，我是你们的讲师。上一章我们学习了 Lemon 解析器如何把 token 序列归约为语法结构。这一章我们要沿着编译链路继续前进，看看其中最重要也最复杂的部分——表达式编译。

在 SQL 中，表达式无处不在：WHERE age > 18 中的 age > 18 是一个布尔表达式，SELECT name || '!' 中的 name || '!' 是字符串连接表达式，ORDER BY score + bonus 中的 score + bonus 是算术表达式。所有这些表达式都需要从解析器构建的抽象语法树翻译成 VDBE 字节码。

这项工作主要在 src/expr.c 中完成，约 65 KB 代码，是 SQLite 源码中最大的文件之一。核心函数 sqlite3ExprCode() 负责将 Expr 树递归地编译为 opcode 序列。这一章我们就来深入理解这个递归编译过程。

Expr 树的构建

当 Lemon 解析器执行一条文法规则的 reduce action 时，它会调用 sqlite3ExprAlloc() 或 sqlite3Expr() 来创建 Expr 节点。这些节点组织成一棵二叉树，即表达式树（Expr tree）。

Expr 结构体定义在 src/sqliteInt.h 中，其核心字段如下：

struct Expr {
    u8 op;                  /* 操作类型，如 TK_PLUS, TK_EQ, TK_COLUMN */
    char affinity;          /* 类型亲和性 */
    u32 flags;              /* 表达式属性标志位 */
    Expr *pLeft, *pRight;  /* 左右子节点（二叉树结构） */
    ExprList *x.pList;     /* 表达式列表（如 IN 的右侧列表） */
    Token token;            /* 原始 token 文本 */
    Token span;            /* 表达式对应的原始 SQL 文本范围 */
    int iTable;            /* TK_COLUMN: 对应的表编号 */
    int iColumn;           /* TK_COLUMN: 对应的列编号 */
    Select *pSelect;       /* TK_SELECT, TK_EXISTS: 子查询 */
    AggInfo *pAggInfo;     /* 聚合函数信息 */
    union {
        FuncDef *pFunc;    /* TK_FUNCTION: 函数定义 */
        Expr *pAgg;        /* 聚合函数包装 */
        int iValue;        /* TK_INTEGER: 整数值 */
        double rValue;     /* TK_FLOAT: 浮点值 */
    };
};

例如，表达式 age > 18 对应的 Expr 树结构是：

  • 根节点：op = TK_GT（大于比较），pLeft = column expr，pRight = integer expr
  • 左子节点：op = TK_COLUMN，iTable = 0（第一个表），iColumn = 2（第三列，假设 age 是第三列）
  • 右子节点：op = TK_INTEGER，iValue = 18

而对于 a + b * c，解析器会根据优先级生成如下树：

  • 根节点：TK_PLUS，左为 TK_COLUMN(a)，右为 TK_STAR（乘法的优先级高于加法，所以乘法在树的更低层）
  • 右子节点 TK_STAR：左为 TK_COLUMN(b)，右为 TK_COLUMN(c)
💡 Expr 树本身不包含类型信息——在解析阶段，SQLite 还不知道列的类型。类型信息是在稍后的"类型推导"阶段通过 affinity 机制决定的。这是 SQLite "动态类型"设计的直接体现。

sqlite3ExprCode() 递归代码生成

sqlite3ExprCode() 是表达式编译的核心入口。它的签名为：

void sqlite3ExprCode(Parse *pParse, Expr *pExpr, int target){
    /* pParse: 编译上下文
     * pExpr:  要编译的表达式树
     * target: 结果存放的目标寄存器编号
     */
}

这个函数的工作方式是递归遍历 Expr 树，根据每个节点的 op 类型生成对应的 VDBE 指令。代码生成的核心逻辑是一个巨大的 switch 语句：

switch( pExpr->op ){
    case TK_PLUS:
    case TK_MINUS:
    case TK_STAR:
    case TK_REM:
    case TK_BITAND:
    case TK_BITOR:
    case TK_SLASH: {
        /* 算术运算：先递归编译左右子表达式，再生成算术 opcode */
        sqlite3ExprCode(pParse, pExpr->pLeft, target);
        sqlite3ExprCode(pParse, pExpr->pRight, regRight);
        sqlite3VdbeAddOp3(v, arithOp, target, regRight, target);
        break;
    }
    case TK_INTEGER: {
        /* 整数常量：生成 OP_Integer 指令将值加载到寄存器 */
        sqlite3VdbeAddOp2(v, OP_Integer, pExpr->iValue, target);
        break;
    }
    case TK_COLUMN: {
        /* 列引用：生成 OP_Column 指令从游标中读取列值 */
        sqlite3VdbeAddOp3(v, OP_Column, pExpr->iTable, pExpr->iColumn, target);
        break;
    }
    case TK_STRING: {
        /* 字符串常量：加载字符串到寄存器 */
        sqlite3VdbeAddOp4(v, OP_String8, 0, target, 0, pExpr->token.z, pExpr->token.n);
        break;
    }
    /* ... 数百种 case ... */
}

这个 switch 覆盖了所有 TK_* 类型，从最简单的字面量到最复杂的子查询表达式。每次递归调用都会产生几条到几十条不等的 VDBE 指令。

值得注意的是，sqlite3ExprCode() 并不是唯一的表达式编译入口。SQLite 还有一系列针对特定场景的辅助函数：

函数用途
sqlite3ExprCode()通用表达式编译，结果存到指定寄存器
sqlite3ExprCodeAndCache()编译并缓存结果，用于需要多次读取的表达式
sqlite3ExprIfTrue()编译为跳转指令——如果表达式为真则跳转
sqlite3ExprIfFalse()编译为跳转指令——如果表达式为假则跳转
sqlite3ExprCodeTemp()编译到临时寄存器，返回寄存器编号

sqlite3ExprIfTrue() 和 sqlite3ExprIfFalse() 是实现 WHERE 子句短路求值的关键——它们不是把表达式的结果算出来放到寄存器里，而是直接生成条件跳转指令。这样 WHERE a=1 AND b=2 就可以在 a=1 为假时直接跳过 b=2 的计算。

不同类型表达式的编译策略

SQLite 对不同类别的表达式采用不同的编译策略：

常量折叠： 像 1 + 2 这样的纯常量表达式，在编译时就被求值优化掉了。这个优化在 src/expr.c 的 sqlite3ExprCode() 之前就已经完成了——在 sqlite3ExprAnalyze() 阶段，常量子表达式会被替换为计算结果。所以 SELECT 1 + 2 编译后只有一个 OP_Integer 3，没有加法指令。

列引用： TK_COLUMN 节点编译为 OP_Column 指令。注意这里列名已经被解析器转换为表编号和列编号了。列编号是用的 logical column index（逻辑列序号），而不是 storage column index（存储列序号），这两者在某些情况下（比如使用了 WITHOUT ROWID 表）可能不同。

函数调用： 内置函数（如 length()、substr()）和用户自定义函数都走同一个路径：

case TK_FUNCTION: {
    /* 为每个函数参数递归生成代码 */
    sqlite3ExprCodeExprList(pParse, pExpr->x.pList, 0, SQLITE_ECEL_DUP);
    /* 生成 OP_Function 指令 */
    sqlite3VdbeAddOp4(v, OP_Function, 0, target, nFarg,
                      (void*)pDef, P4_FUNCDEF);
    break;
}

子查询： TK_SELECT、TK_EXISTS、TK_IN 等涉及到子查询的表达式，编译策略完全不同。它们会在当前 VDBE 程序中嵌入一个子程序，通过 OP_Once 或 OP_OpenRead 来执行子查询。子查询的结果被放入一个临时表中，然后外层表达式通过 OP_Found、OP_NotFound、OP_Column 等指令来使用这些结果。

设计亮点： SQLite 没有独立的"表达式优化器"。表达式的优化（如常量折叠、短路求值、子查询扁平化）分散在代码生成的过程中，由 expr.c 中的各个函数各自完成。这种设计虽然不如独立优化器那么系统化，但胜在简单直接——每一步代码生成都可以根据上下文做出最优的本地决策。

类型推导与 Affinity

SQLite 是动态类型的数据库——列没有固定的类型约束，任何行都可以存入任意类型的值。但 SQLite 又有一个叫做 Affinity（类型亲和性） 的机制，用于指导类型转换。

sqlite3ExprAffinity() 函数负责确定一个表达式的类型亲和性。它的规则是：

  • 列引用 — 返回该列声明的 affinity（BLOB、TEXT、NUMERIC、INTEGER、REAL 之一）
  • 字面量 — 返回 SQLITE_AFF_NONE
  • 二元运算 — 默认 SQLITE_AFF_NUMERIC，因为算术运算期望数值操作数
  • 函数返回值 — 由函数定义指定（通过 sqlite3CreateFunc() 注册）
  • CASE / CAST — CAST 显式指定 affinity，CASE 的结果类型与第一个 THEN 相同

在代码生成时，SQLite 会在适当的位置插入 OP_ApplyAffinity 或 OP_Cast 指令来实现类型转换。例如：

/* 对于 WHERE int_col = '123'，自动插入类型转换 OP_Cast */
sqlite3VdbeAddOp2(v, OP_Cast, regRight, SQLITE_AFF_INTEGER);

这就是为什么 SELECT * FROM t WHERE id = '42' 能正确匹配 id=42 的行——尽管 '42' 是字符串字面量，但 affinty 机制告诉 SQLite 应该把它当作整数来比较。

⚠️ 踩坑实录：Affinity 机制有时会带来令人惊讶的行为。比如 SELECT '5' > 10，按照字符串比较 '5' > '10' 为 true（因为 '5' > '1' 的字典序），但 SQLite 由于 affinity 机制，会将字符串 '5' 转换为数字 5，结果为 false。这个"隐式类型转换"曾经让很多从 MySQL 迁移过来的开发者感到困惑——MySQL 的行为不同。SQLite 的行为实际上是符合 SQL 标准的。

IN、CASE、CAST 的编译实现

几个特殊表达式的编译模式值得专门讲解。

IN 表达式： expr IN (v1, v2, ..., vN) 的编译有三种策略：

  • 小列表（N ≤ 10）： 直接生成 N 个 OP_Eq 比较指令，用 OR 连接起来
  • 大列表（N > 10）： 将列表值构建成一个内存表（OP_MakeRecord），然后用 OP_Found 进行二分查找
  • 子查询： expr IN (SELECT ...) 编译为打开子查询的游标，然后对每一行做 OP_NotFound 检查

CASE 表达式： CASE x WHEN v1 THEN r1 ELSE rD 的编译生成以下结构的 VDBE 代码：

    OP_Eq          x, v1 → 不相等跳转到 L1
    OP_ResultRow   r1 → 返回结果
    OP_Goto        L2 → 跳过其他分支
L1: OP_Eq          x, v2 → 不相等跳转到 L3
    OP_ResultRow   r2
    OP_Goto        L2
L3: OP_ResultRow   rD → 默认值
L2: ...

CAST 表达式： CAST(expr AS type) 编译为 OP_Cast 指令，它会将寄存器中的值转换为目标类型。具体的转换规则非常复杂——比如从浮点数转换为整数时，SQLite 会截断小数部分，而非四舍五入；将字符串转换为 NUMERIC 时，会尝试解析前缀数字。

实战：追踪一条 WHERE 表达式的编译过程

让我们追踪 SELECT * FROM t WHERE age > 18 AND city = 'Beijing' 中的 WHERE 子句编译过程来结束本章。

首先，解析器为条件表达式 age > 18 AND city = 'Beijing' 构建 Expr 树：

  • TK_AND（根节点）
  •  ├── TK_GT：age > 18
  •  │ ├── TK_COLUMN（age 列，iTable=0, iColumn=1）
  •  │ └── TK_INTEGER（18）
  •  └── TK_EQ：city = 'Beijing'
  •    ├── TK_COLUMN（city 列，iTable=0, iColumn=2）
  •    └── TK_STRING（"Beijing"）

接下来，sqlite3ExprIfFalse() 被调用来编译这个条件（因为 WHERE 子句的语义是：条件为假时跳过该行，为真时保留）：

/* 这是生成的 VDBE 指令序列（经过简化和标注） */
addr1: OP_Column     0 1 reg1    ← 读取 age 到 reg1
addr2: OP_Integer    18 reg2     ← 加载常量 18 到 reg2
addr3: OP_Ge         reg2 reg1 addr7  ← 如果 reg1 < reg2（即 age ≤ 18），跳转到跳过该行
addr4: OP_Column     0 2 reg3    ← 读取 city 到 reg3
addr5: OP_String8    "Beijing" reg4  ← 加载字符串到 reg4
addr6: OP_Eq         reg4 reg3 addr9  ← 如果不相等，继续（未匹配），跳转到 skip 标签
addr7: OP_Goto       0 0 addr8   ← 条件不满足，跳到下一行
addr8: OP_Next       0 0 addr1   ← 读取下一行
addr9: OP_ResultRow  ...         ← 条件满足，输出该行

注意这里 AND 的短路求值效果：如果 age > 18 为假（即 age ≤ 18），直接跳转到下一行（addr3 的跳转），根本不会去检查 city 条件。这正是 sqlite3ExprIfFalse() 的巧妙之处——它为 AND 编译出 if false then jump 的级联结构。

从 SQL 到 opcode： 你看，一条简单的 WHERE 子句背后涉及到 Expr 树的构建、类型亲和性的推导、递归代码生成、短路求值的实现——每一个环节都是精细的工程权衡。表达式编译是 SQLite 前端中最复杂、最密集的逻辑，理解它你就理解了 SQL 查询的"前半生"。
