开篇：当清洗速度成为瓶颈，是时候请 GPU 出场了

大家好，我是你们的讲师。上一章我们用 Datatrove 搭建了一套完整的预处理管道，在 CPU 集群上跑得不错。但如果你处理的数据规模达到上百 TB 甚至 PB 级别，CPU 的处理速度就会变成你的主要瓶颈。尤其是 MinHash 去重这一步——生成数百万个文档的 256 个 n-gram 签名，在 CPU 上需要计算海量的哈希值，耗时惊人。

我有一个亲身经历：在 2024 年底参与一个千亿参数模型的预训练项目时，我们的数据团队用 256 核的 CPU 集群跑 MinHash 去重，整整跑了 7 天还没跑完。项目排期本来就很紧，数据预处理占了这么多时间，训练那边一直在催。后来我们尝试了 NVIDIA 的 NeMo Curator，在 8 张 H100 GPU 上跑同样的任务，不到 12 小时就完成了——速度快了 14 倍，成本反而更低。这一章我们就来深度拆解，GPU 加速清洗到底是怎么做到的，以及如何把它集成到你的数据流水线中。

24.1 NeMo Curator 概述

NeMo Curator 是 NVIDIA 推出的 GPU 加速数据清洗框架，是 NeMo 生态（Megatron-LM、NeMo Framework）的重要组成。与 Datatrove 不同，NeMo Curator 的定位非常明确：在 GPU 集群上以极致速度完成数据清洗。它利用 cuDF（GPU DataFrame）和 RAPIDS 生态来将传统上在 CPU 上运行的 Pandas/NumPy 操作移植到 GPU 上。

特性 Datatrove NeMo Curator
运行平台 CPU（可扩展至 Slurm） GPU（推荐 NVIDIA H100/A100）
去重速度（1TB） 约 2 小时 约 8 分钟（8×H100）
MinHash（1TB） 约 300 GB/hr 约 4 TB/hr（8×H100）
PII 检测 基于规则 + CPU 推理 GPU 加速 NER 模型 + 规则
质量分类 CPU 推理 GPU 推理（批量，大幅降低延迟）
集成训练框架 通用（无绑定） 原生集成 Megatron-LM / NeMo

但 GPU 加速并不是"插上显卡就能变快"这么简单。要让 GPU 高效工作，数据必须经过精心组织的批处理，最大程度利用 GPU 的并行计算能力。NeMo Curator 在后台做了大量工程优化，包括自动批处理大小调整、混合精度计算和异步 I/O 重叠。

24.2 GPU 加速去重的性能飞跃

去重是数据清洗中最消耗计算资源的步骤之一。我们分别看看精确去重和模糊去重在 GPU 加速下的表现。

精确去重

精确去重的核心是计算每个文档的哈希指纹（如 SHA256），然后通过哈希表查找重复项。这个过程在 CPU 上受限于单核哈希计算速度。NeMo Curator 使用 cuDF 的 GPU 哈希模块，将文档指纹计算并行化到数千个 CUDA 核心上执行：

import nemo_curator as nc
from nemo_curator.datasets import DocumentDataset

# 读取数据
dataset = DocumentDataset.read_json(
    "/data/raw_dataset/",
    add_filename=True,
)

# GPU 加速的精确去重
dedup = nc.ExactDuplicates(
    id_field="doc_id",
    text_field="text",
    hash_method="sha256",
)

# 执行去重——自动利用所有可用 GPU
dedup_result = dedup.duplicates(dataset)

# 清洗：移除重复文档
cleaned = dedup.remove_duplicates(dataset, dedup_result)
cleaned.to_json("/data/deduped_dataset/", write_to_filename=True)

print(f"去重前文档数: {len(dataset.df):,}")
print(f"去重后文档数: {len(cleaned.df):,}")
print(f"重复率: {(1 - len(cleaned.df) / len(dataset.df)):.2%}")

在 8 张 H100 GPU 上，NeMo Curator 的精确去重吞吐量可以达到 18 TB/hr，而同等规模的 CPU 集群（512 核）只能达到约 2 TB/hr。这个差距的原因很简单：SHA256 哈希计算是 SIMD 友好型的操作，GPU 的数千个 CUDA 核心天然适合做这种密集型并行计算。

模糊去重（MinHash）

模糊去重在精确去重的基础上增加了 n-gram 切分和 LSH 分桶，计算量更大。NeMo Curator 的 GPU 加速 MinHash 实现同样使用了 RAPIDS cuDF：

from nemo_curator import MinHash
from nemo_curator.modules import LSH

# GPU 加速的 MinHash
minhash = MinHash(
    seed=42,
    num_hashes=256,
    num_buckets=16,
    use_gpu=True,  # 启动 GPU 加速
)

# 计算 MinHash 签名
signatures = minhash.fit_transform(dataset)

# LSH 分桶查找近似重复
lsh = LSH(
    num_hashes=256,
    num_buckets=16,
    buckets_per_shuffle=1,
)
duplicate_pairs = lsh.fit_transform(signatures)

# 基于发现的重复对进行去重
cleaned = minhash.remove_duplicates(dataset, duplicate_pairs)
cleaned.to_json("/data/deduped_dataset_minhash/")

GPU 加速的 MinHash 可以达到 4 TB/hr 的吞吐量，相比 CPU 版本的 300 GB/hr 提升了约 13 倍。这意味着一个原本需要跑一周的去重任务，现在可以在一个晚上做完。

性能对比的启示：GPU 加速不是简单地"把 CPU 代码搬到 GPU 上运行"。NeMo Curator 的加速依赖于 RAPIDS 生态的深度优化——包括 GPU 内存管理、批次大小自动调优、以及异步 I/O 与计算的重叠。如果你只是用 PyTorch 把 MinHash 的哈希计算移植到 GPU 上，远达不到这个性能水平。

24.3 GPU 加速的 PII 检测与多语言评分

除了去重，NeMo Curator 在另外两个环节也提供了显著的 GPU 加速：PII（个人身份信息）检测和多语言文档质量评分。

PII 检测与脱敏

随着全球数据合规法规（如 GDPR、中国《个人信息保护法》）的收紧，PII 检测已成为数据清洗的标配环节。NeMo Curator 内置了 GPU 加速的 PII 检测模块：

from nemo_curator.modules.pii import PIIAnnotator

# GPU 加速的 PII 检测
pii = PIIAnnotator(
    text_field="text",
    entities=["PERSON", "EMAIL", "PHONE", "ADDRESS", "ID_NUMBER"],
    use_gpu=True,
    batch_size=1024,
)

# 标注 PII 位置
annotated = pii.annotate(dataset)

# 执行脱敏——用 [REDACTED] 替换 PII
from nemo_curator.modules.pii import PIICleaner
cleaner = PIICleaner(
    text_field="text",
    replacement="[REDACTED]",
)
cleaned = cleaner.clean(annotated)

PII 检测的 GPU 加速主要来自两个方面：一是基于 GPU 的 NER（命名实体识别）模型推理，利用 TensorRT 进行推理加速，单卡吞吐量可达每秒 5,000+ 文档；二是正则表达式匹配利用 GPU 的并行字符串处理能力，相比 CPU 版本的线性扫描有数十倍的提升。

多语言文档评分

质量评分是另一个计算密集环节——需要对每篇文档运行一个分类器来预测其质量分数。NeMo Curator 支持加载预训练的 GPU 质量分类器，批量处理文档：

from nemo_curator.classifiers import QualityClassifier

# 加载 GPU 加速的质量分类器
classifier = QualityClassifier(
    model_path="/models/quality_classifier.onnx",
    use_gpu=True,
    batch_size=4096,   # GPU 批量推理
)

# 对所有文档进行质量评分
scores = classifier.infer(dataset)

# 过滤掉低质量文档
filtered = scores[scores["quality_score"] > 0.6]
filtered.to_json("/data/scored_dataset/")

在多语言场景下，你可以为每种语言训练一个专用的质量分类器，然后在一轮 GPU 推理中完成所有文档的评分。NeMo Curator 会自动根据文档的语言标签选择合适的分类器，用户只需提供一个模型映射表。

24.4 成本分析：GPU 清洗到底值不值？

很多同学看到"用 GPU 做数据清洗"的第一反应是：这太奢侈了吧？GPU 多贵啊，用 CPU 慢慢洗不是更省钱吗？我们来算一笔账。

方案 硬件配置 处理 1T token 的时间 每小时成本 总成本
CPU 集群 512 vCPU（64 节点 × 8 核） 约 9 小时 ~$85/hr（按需） ~$765
GPU - 按需 8 × H100 SXM（1 DGX） 约 3 小时 ~$240/hr ~$720
GPU - 竞价 8 × H100 SXM（1 DGX） 约 3 小时 ~$140/hr ~$420
GPU - 预留 8 × H100 SXM（1 DGX） 约 3 小时 ~$65/hr ~$195

按 1 万亿 token 的处理规模计算，GPU 方案的总成本不仅不比 CPU 贵，在竞价和预留实例的场景下，甚至便宜了 45%-75%。原因很简单：GPU 虽然单价贵，但它跑得快，总的工作时间短，而且在去重和质量评分这类计算密集环节，GPU 的时间优势还会被进一步放大。如果算上时间成本——比如项目延期带来的机会成本——GPU 清洗方案的 ROI 就更显著了。

省钱技巧：如果你的预算有限，可以采用混合策略——用 CPU 集群做文本提取和格式转换（I/O 密集型，GPU 优势不大），仅将去重和质量评分这类计算密集型环节迁移到 GPU 上。这样可以用最少的 GPU 资源换取最大的性能提升。

24.5 深度集成：NeMo Curator + Megatron-LM

NeMo Curator 的另一个核心优势是它与 Megatron-LM 训练框架的原生集成。清洗好的数据可以直接输出为 Megatron-LM 要求的二进制格式，省去了格式转换的中间步骤：

from nemo_curator import MegatronDatasetWriter

# 将清洗后的数据导出为 Megatron-LM 训练格式
writer = MegatronDatasetWriter(
    output_dir="/data/megatron_ready/",
    tokenizer_path="/models/tokenizer.model",
    seq_len=4096,
    write_in_binary=True,        # 二进制格式，加速加载
    process_count=16,            # 并行的 tokenize worker 数
)

writer.write(dataset)

# 输出文件结构：
# /data/megatron_ready/
#   ├── text_document.bin    # 二进制的 tokenized 数据
#   ├── text_document.idx    # 索引文件
#   └── dataset_info.json    # 元数据

这个集成意味着你可以直接从原始文档出发，经过 NeMo Curator 的全流程清洗和 tokenization，得到 Megatron-LM 直接可读的训练数据格式。不再需要额外编写格式转换脚本，也不需要在训练时重新进行 tokenize——后者对于大规模预训练来说是一个不可忽视的时间开销。

踩坑实录：在一次大规模清洗中，我们使用竞价实例运行 NeMo Curator，结果在运行到第 15 个小时时被云厂商回收了实例——所有未保存的中间结果全部丢失。教训是：对于超过 6 小时的 GPU 任务，切忌使用竞价实例而不做 checkpoint。NeMo Curator 支持设置 checkpoint_dir 参数，务必开启并设置合理的 checkpoint 间隔（建议每 1 小时或每处理 100GB 数据 checkpoint 一次）。

本章小结

NeMo Curator 证明了 GPU 加速在数据清洗领域的巨大潜力。它将精确去重的吞吐量从 2 TB/hr 提升到 18 TB/hr，将 MinHash 从 300 GB/hr 提升到 4 TB/hr，同时还有 PII 检测和多语言质量评分的 GPU 加速能力。更重要的是，从成本角度看，GPU 清洗方案在竞价或预留实例下甚至比纯 CPU 方案更便宜——前提是你的工作负载是计算密集型的。

不过，无论我们用 Datatrove 还是 NeMo Curator 清洗数据，都面临一个更根本的问题：我们怎么知道洗出来的数据是"好"的？质量过滤器的阈值设置是否合理？数据配比是否最优？这些问题的答案不能靠猜，需要通过系统性的实验来验证。下一章，我们就来学习如何用数据质量评估与消融实验来回答这些关键问题。
