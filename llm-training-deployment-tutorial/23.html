第23章 模型量化技术深度

大家好，我是你们的讲师。今天开始我们进入课程第六部分——部署与工程实践。第一个重头戏是模型量化。量化这件事，说简单很简单——把FP32转成INT4就完事了，对吧？但真正做过生产部署的人都知道，这里面的坑多到让你怀疑人生。

讲个真实经历。去年有个做AI客服的朋友找到我，他们把一个70B的模型部署到生产环境，只用了4张A100，跑FP16推理。刚开始只有几百个用户时一切正常。等到用户量涨到日均十万次对话，每张卡的显存被KV Cache撑爆了，OOM了十几次。运维团队连夜紧急扩容到8张卡，成本直接翻倍。他们找到我的时候，我说：你最大的问题不是卡不够，而是模型太"胖"了。一个70B的FP16模型光权重就占了140GB，再加上KV Cache、中间激活，4张A100（每张80GB）实际上连模型都放不下，更别说做推理了。后来我们用INT4量化把权重压到35GB，配合GQA和KV Cache优化，4张A100跑得稳稳的，延迟还降了一半。这就是量化的价值——让模型瘦身、让部署变轻、让成本降低。

这一章我们深入量化技术的核心：从数学原理到主流算法（GPTQ、AWQ、GGUF），再到最新的FP8推理。学完这一章，你能回答"我的模型该用什么量化方式？定多少精度？"这个关键问题。

23.1 量化的数学基础

量化的本质是用更少的比特数来表示模型的权重和激活值。大模型训练和推理默认使用FP32（32位浮点）或FP16（16位浮点），而量化把它们压缩到INT8（8位整数）、INT4（4位整数）甚至更低。压缩比越大，精度损失越大，但速度越快、显存占用越少。这个取舍就是量化技术的全部。

对称量化 vs 非对称量化。这是最基础的分类。对称量化把数值范围映射到以0为中心的对称区间，比如FP32的[-a, a]映射到INT8的[-127, 127]，公式是 x_q = round(x / scale)，其中scale = max(|x|) / 127。非对称量化则引入零点偏移（zero point），映射到[0, 255]，公式是 x_q = round(x / scale) + zero_point。对称量化的优势在于计算时少一个zero_point的操作，更高效；非对称量化则能更好地处理非对称分布的数值（比如ReLU后的激活值都是正数），精度更高。

逐通道 vs 逐张量。量化粒度决定了精度和计算复杂度。逐张量（per-tensor）量化为整个权重矩阵用一个scale，计算简单但精度低——因为不同通道的数值范围差异很大，一个大的离群值会拉高整个矩阵的scale，导致其他通道的精度损失严重。逐通道（per-channel）量化给每个输出通道单独算scale，精度好很多，但每个通道需要一个scale参数，存储和计算开销稍高。在GPTQ和AWQ中，还有更细粒度的逐组（per-group）量化，比如每128个权重一组，这是目前精度和效率的最佳平衡点。

量化误差的来源。量化的误差主要来自两个方面。一是截断误差：当原始数值超出量化范围时会被截断（clip），那些大值的尾部信息就丢失了。二是舍入误差：从FP32映射到INT时，连续的浮点数被离散化成整数值，这个过程必然有精度损失。舍入误差在高精度量化（INT8）中影响较小，但在INT4中可能非常显著，特别是当权重分布不均匀时。

量化类型 位宽 压缩比(vs FP16) 常见方法
FP32→FP16 16 2x 混合精度训练
FP16→INT8 8 4x GPTQ、AWQ、LLM.int8()
FP16→INT4 4 8x GPTQ-4bit、AWQ-4bit、GGUF Q4
FP16→INT2/NF2 2 16x BitNet、GGUF Q2_K

理解了这些基础概念，我们来看三种主流量化算法。它们各自有不同的设计哲学和应用场景。

23.2 GPTQ算法

GPTQ（Post-Training Quantization for Generative Pre-trained Transformers）是目前业界应用最广的量化算法之一。它的核心思想是逐层量化、基于Hessian矩阵做误差补偿。听起来很吓人，但概念其实很优雅。

从OBQ说起。GPTQ的前身是OBQ（Optimal Brain Quantizer），灵感来自最优脑外科（Optimal Brain Surgery）——一种经典剪枝算法。OBQ的思路是：每次量化一个权重后，调整所有未量化的权重来补偿这个量化造成的误差。数学上，它用Hessian矩阵来衡量每个权重的重要性：某个权重的Hessian值越大，说明它对最终输出的影响越大，量化它的时候就要更谨慎。每次量化后，OBQ会用一阶泰勒展开更新其他权重，最小化输出误差。

GPTQ的优化。OBQ的问题是太慢了——每量化一个权重就要更新一次，对于175B的GPT-3来说根本不可行。GPTQ做了三个关键优化。第一，固定顺序：不再逐权重选择最优顺序，而是按列（即按输入神经元）依次量化，避免反复排序的开销。第二，延迟批处理：把Hessian矩阵的Cholesky分解批量处理，而不是逐列计算。第三，分组量化：每128个权重一组共享scale和zero_point，降低了存储开销。

Hessian矩阵近似。GPTQ在量化前只需要一个calibration数据集（通常几百个样本），前向传播一次收集各层的Hessian信息。它不用精确计算每个权重的Hessian，而是通过Fisher信息矩阵近似——实际上就是梯度的平方期望。这个近似的优点是计算量小，而且不容易过拟合到特定的calibration数据。

# GPTQ量化核心流程（伪代码）
def gptq_quantize(layer, calibration_data, group_size=128):
    # 1. 收集Hessian矩阵（Fisher近似）
    hessian = compute_fisher_information(layer, calibration_data)

    # 2. 对权重矩阵逐列量化
    for col_index in range(layer.weight.shape[1]):
        weight_col = layer.weight[:, col_index]
        # 找到当前列所属的group
        group_idx = col_index // group_size

        # 3. 计算该group的scale和zero_point
        scale = (weight_col.max() - weight_col.min()) / (2**bit_width - 1)
        zero_point = round(-weight_col.min() / scale)

        # 4. 量化权重
        q_weight = round(weight_col / scale) + zero_point
        q_weight = clamp(q_weight, 0, 2**bit_width - 1)

        # 5. 量化误差 + Hessian补偿
        error = weight_col - dequantize(q_weight, scale, zero_point)
        # 用Hessian的逆来更新后续列，补偿当前列的量化误差
        if col_index < layer.weight.shape[1] - 1:
            layer.weight[:, col_index+1:] -= error * hessian_update

    return q_weight, scale, zero_point

精度与效率。GPTQ在INT4量化下通常能将模型精度损失控制在1%以内（perplexity增加不到0.5）。对于8B～70B规模的模型，在单张A100上完成量化只需要1-2小时。一个重要的经验是：calibration数据集的选择对结果影响很大。我见过有人用C4数据集做calibration，量化后的模型在代码任务上掉精度严重，换用代码相关的calibration数据后就好了。

实战建议：GPTQ的group size参数需要根据模型大小调整。对于7B以下的小模型，推荐group_size=128；对于30B以上的大模型，group_size=32虽然存储开销更大，但精度明显更好。另外，desc_act（是否按激活值排序列）这个参数在GPTQ中争议很大——开启desc_act能提升精度，但会破坏矩阵乘法的连续性，降低实际推理速度。简单说：离线分析用desc_act，线上部署关掉它。

23.3 AWQ算法

AWQ（Activation-Aware Weight Quantization）是GPTQ之后最受关注的量化方法，由MIT韩松团队提出。AWQ的核心洞察是：不是所有权重都同等重要。一小部分权重（大约1%）对模型输出的影响远大于其他权重，这些关键权重通常对应那些激活值较大的通道。

Salient Channel保护。AWQ发现了一个有趣的规律——权重中那些对应高激活值的通道（salient channels），它们的数值范围也往往特别大。传统的量化方法会把整层的scale拉大来包容这些离群值，导致其他通道精度下降。AWQ的做法是：不强行量化这些通道，而是保留它们为FP16，或者用一个大的scale单独处理。这就是所谓的salient channel保护。

Scaling Factor搜索。AWQ引入了一个巧妙的技术——在量化前对权重乘以一个per-channel的scaling factor（通过对激活值的统计来确定），然后在推理时把scaling factor融合到后续的LayerNorm或线性层中，不增加额外计算开销。这个scaling factor是通过小规模搜索找到的。具体来说，AWQ在几个scaling候选值（2的幂次，比如1, 2, 4, 8…）中做网格搜索，选出一个让最终输出误差最小的值。

AWQ vs GPTQ。两者都是post-training量化，不需要重新训练。主要的区别在于：GPTQ通过Hessian补偿来做误差修正，精度高但计算量大；AWQ通过激活值分析和scaling factor保护关键通道，速度更快（因为不需要反复的Hessian分解），且在低比特（INT4以下）场景下精度更好。在7B模型上，AWQ的量化速度比GPTQ快2-3倍，而精度几乎持平。

特性 GPTQ AWQ
核心机制 Hessian矩阵误差补偿 激活感知的salient channel保护
量化速度 较慢（需要Hessian分解） 较快（网格搜索scaling factor）
INT4精度 优秀 优秀（低比特场景略好）
calibration数据需求 ～128样本 ～128样本
推理速度 受desc_act影响 稳定（scaling factor融合到LN）
生态支持 AutoGPTQ、vLLM、TGI AutoAWQ、vLLM、TGI、MLC-LLM

AWQ特别适合对推理速度有极致要求的场景。我见过一个边缘部署的案例：在Jetson Orin上跑7B模型，用FP16根本跑不动（显存不够），用GPTQ INT4延迟在200ms左右，换成AWQ INT4后延迟降到120ms，因为AWQ避免了desc_act导致的矩阵乘法不连续问题。

23.4 GGUF格式与实践

如果你在开源社区下载过模型，一定见过GGUF（GPT-Generated Unified Format）这个格式。GGUF是llama.cpp生态的标准模型格式，它的设计目标很简单：一个文件搞定一切。你下载一个llama-2-7b.Q4_K_M.gguf，直接就能用llama.cpp跑推理，不需要额外的tokenizer.json、config.json等文件。

量化格式演进。GGUF的前身是GGML（2023年初），经历了多次迭代。GGML的早期版本只支持简单的Q4_0（4位对称量化），精度很差。后来发展出Q4_1（4位非对称量化），精度稍好但存储更大。接着出现了K-quant（Q2_K到Q6_K），这是GGUF真正成熟的关键。

K-quant策略。K-quant的核心思想是重要性加权的混合精度量化。模型的每一层对最终输出的"重要性"不同，K-quant用Hessian轨迹（trace of Hessian）来衡量每层的敏感度。敏感度高的层用高位宽量化（比如Q6_K：6位），敏感度低的层用低位宽量化（比如Q2_K：2位）。最终得到一个名为Q4_K_M的混合精度方案：大部分层用4位，部分关键层用5-6位，不重要的层用2-3位。这种策略在4.5位的平均位宽下能达到接近5位纯量化的精度。

量化级别 平均位宽 模型大小(7B) 精度损失 推荐场景
Q2_K 2.56 bit 2.7 GB 明显（10-15% perplexity上升） 极低资源场景、实验
Q3_K_M 3.35 bit 3.3 GB 中等（5-8%） 低资源部署
Q4_K_M 4.35 bit 4.1 GB 轻微（1-3%） 主流推荐、日常使用
Q5_K_M 5.15 bit 4.8 GB 几乎无损 精度敏感场景
Q6_K 6.5 bit 5.6 GB 极小（<0.5%） 接近FP16精度
Q8_0 8.0 bit 6.7 GB 基本无损 精度最高需求

对比建议。GGUF生态最强大的地方在于对个人设备和CPU推理的支持。GPTQ和AWQ依赖GPU的INT4/INT8 Tensor Core加速，在GPU上没有对手。但如果你要在MacBook上跑模型（Apple Silicon的Metal加速），或者纯CPU推理，GGUF是唯一的选择。GGUF的另一个优势是零依赖——一个二进制文件就能跑，不需要Python环境，不需要CUDA。

踩坑案例：有一次我在MacBook Pro M2上测试GGUF推理，下了Q4_K_M版本，发现内存占用高达6GB（M2只有16GB统一内存），推理到一半系统开始swap，速度从20 tok/s暴跌到2 tok/s。排查了半天，发现问题是llama.cpp的context size设置默认4096，但系统实际申请了KQ_cache+KV_cache=两倍。把context size降到2048，并且开启flash attention（llama.cpp的-fa flag），内存占用降到了3.5GB，速度回到15 tok/s。教训：GGUF的精度很好，但也要关注context size对显存/内存的影响，不是文件大小能装下就完了。

23.5 FP8推理

FP8是2024-2025年NVIDIA在Hopper架构（H100/H200/B100）上主推的新数据类型。与INT4/INT8不同，FP8保留了浮点数的动态范围优势——INT8在数值范围上有限制，大值容易被截断，而FP8可以在更大范围内保持相对精度。

NVIDIA Transformer Engine的FP8推理。Transformer Engine（TE）是NVIDIA在NeMo和TensorRT-LLM中集成的FP8推理方案。FP8有两种格式：E4M3（4位指数+3位尾数，范围大、精度低）和E5M2（5位指数+2位尾数，范围更大、精度更低）。TE的做法很聪明：权重用E4M3，激活用E5M2。理由很简单：权重的数值分布相对稳定且集中在[-1, 1]之间，E4M3足够用；激活值可能非常大（尤其是在attention中），需要E5M2的大动态范围来避免截断。

FP8 vs INT4/INT8。FP8的精度高于INT8——在LLaMA-70B上做benchmark，FP8的perplexity与FP16几乎没有区别，而INT8通常会有0.1-0.3的perplexity上升。在推理速度上，H100的FP8 Tensor Core能提供两倍于FP16的吞吐量。但FP8也有它的短板：对硬件有强依赖。只有H100/B100及以上型号支持FP8 Tensor Core，H100以下的卡跑不了。INT4虽然精度略低，但几乎兼容所有带Tensor Core的GPU（Turing架构之后）。

MoE模型的FP8量化挑战。MoE模型（如Mixtral 8x7B、DeepSeek-V3）给FP8量化带来了两个额外挑战。第一，expert利用率不均：某些expert被频繁路由到，某些很少被用到，导致权重分布差异大，需要一个全局的scaling策略，而不是每个expert各自为政。第二，gate router的精度敏感：MoE的router虽然只占模型参数的0.01%左右，但它对精度的要求极高——router决定了token去哪个expert，一个错误的量化可能导致token被路由到错误的expert，引发级联效应。实践表明，router层必须保留FP16，或者至少用INT8，用INT4会导致router准确率下降5%以上。

# FP8量化配置（TensorRT-LLM风格）
tensorrt_llm_quant_config = {
    "quant_algo": "FP8",
    "kv_cache_quant_algo": "FP8",      # KV Cache也用FP8量化
    "weight_data_type": "FP8_E4M3",     # 权重用E4M3
    "activation_data_type": "FP8_E5M2", # 激活用E5M2
    "exclude_modules": ["lm_head"],     # lm_head保留FP16
    "per_channel_scaling": True,
    # MoE专用配置
    "moe_config": {
        "router_data_type": "FP16",     # router必须保留高精度
        "shared_expert_dtype": "FP8",   # shared expert可以用FP8
        "routed_expert_dtype": "FP8",
        "expert_scaling": "global"      # 全局scaling而非per-expert
    }
}

核心结论：截至2026年，量化技术的选型已经比较成熟。如果你在部署场景中有H100以上显卡，首选FP8——精度接近无损，速度比INT4略慢但比FP16快一倍。如果你受限于显存容量（需要7B模型跑在单卡上同时处理大batch），INT4是你的不二选择。如果你在个人设备上跑（MacBook、CPU Server），GGUF是唯一的可行方案。从趋势上看，QAT（Quantization-Aware Training）正在成为主流——与其训练完再量化损失精度，不如在训练阶段就让模型适应低精度计算。不过那是进阶话题了，我们在第28章的实战项目中会涉及。
