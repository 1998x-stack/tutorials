第七章：向量数据库入门

大家好，走到这一章，我们的RAG知识体系已经搭建了半壁江山。我们有了文档块（chunk），有了嵌在向量空间中的embedding，现在需要一个地方来存放这些向量、并对它们进行高效的相似性搜索。这就是向量数据库的舞台。

从技术演进的角度看，向量数据库是RAG爆发的"助产士"。如果没有向量数据库的高效ANN（Approximate Nearest Neighbor，近似最近邻）搜索能力，语义检索的时间成本会高到无法落地。试想一下：对于100万个chunk，如果每次检索都要计算100万次向量点积——哪怕每次计算只要0.01毫秒，总延迟也要10秒。这对于在线服务是不可接受的。向量数据库把这个问题从O(n)降低到了O(log n)甚至更低。

一、为什么需要向量数据库

你可能问：直接用FAISS不行吗？为什么还要搞一个专门的数据库？答案是：生产级RAG需要的远不止是"能搜索向量"。

  • 数据持久化与CRUD：FAISS是一个内存索引库，不支持直接在磁盘上增删改。向量数据库提供完整的CRUD（Create, Read, Update, Delete）操作。
  • 元数据过滤：你需要按时间、权限、来源等metadata字段过滤，仅靠向量距离不够。
  • 分布式与高可用：当数据量超过单机内存时，需要分片和复制。
  • 混合检索：同时支持向量检索和全文检索（如BM25）并进行融合。
  • 监控与运维：查询延迟监控、索引构建进度、数据备份恢复等运维能力。

二、ANN算法原理：HNSW / IVF / PQ

向量数据库的核心是ANN算法。了解这些算法的基本思想，对你做性能调优和选型大有帮助。

算法 全称 核心思想 优点 缺点
HNSW Hierarchical Navigable Small World 分层图结构，从粗到精逐层导航 召回率极高，搜索速度快 构建速度慢，内存占用大
IVF Inverted File Index 聚类分桶，只在最近的几个桶中搜索 内存友好，构建快 召回率受聚类质量影响
PQ Product Quantization 向量分段压缩，显著减小内存占用 内存极度节省 精度损失，搜索需解码
IVF+PQ 组合方案 IVF分桶 + PQ压缩向量 兼顾速度和内存 精度比HNSW低
DiskANN Disk-based ANN 基于SSD的图索引，突破内存限制 支持超大数据集 I/O开销，延迟略高

在大多数RAG应用中，HNSW是默认首选。它提供了最好的召回率-速度平衡，Milvus、Qdrant、Chroma等都默认使用HNSW或其变体。

三、主流向量数据库对比

选择向量数据库是RAG系统架构中的关键决策。以下是对主流选项的对比：

数据库 部署模式 核心索引 混合检索 分布式 生态 适合场景
Chroma 嵌入式/客户端 HNSW 有限 否 Python生态极好 快速原型、学习、小规模
Milvus 分布式服务 HNSW/IVF/PQ/DiskANN 强 是 成熟、GPU加速、云原生 大规模生产、企业级
Qdrant 服务/嵌入式 HNSW 强 是 Rust编写，性能优秀 中等规模、高吞吐
Pinecone 全托管云服务 专有 强 是 Serverless、零运维 不想运维、快速上线
FAISS 嵌入式库（内存） HNSW/IVF/PQ 不支持 否 Meta出品，极致性能 纯向量搜索、离线批处理
Elasticsearch 分布式服务 HNSW 原生强 是 全文搜索王者扩展向量 已有ES、需要全文+向量
pgvector PostgreSQL扩展 IVFFlat/HNSW 强（SQL） 继承PG SQL生态、事务支持 已有PG、需要事务一致性
选型建议：（1）学RAG的起点选Chroma——极简API，3行代码跑通；（2）小规模生产选Qdrant——单机性能强悍，运维简单；（3）大规模企业选Milvus——分布式架构成熟、GPU加速、生态完备；（4）已用ES选ES 8.x的vector——减少技术栈复杂度；（5）不想运维选Pinecone——但注意数据隐私和数据出口费用。

四、使用Chroma进行基本CRUD操作

Chroma是目前入门门槛最低的向量数据库。让我们通过完整的CRUD操作把它用熟：

import chromadb
from chromadb.utils import embedding_functions

# ——— 创建客户端与集合 ———
client = chromadb.PersistentClient(path="./chroma_rag_db")
collection_name = "my_rag_knowledge"

# 如果集合已存在则删除（演示用）
existing = client.list_collections()
if any(c.name == collection_name for c in existing):
    client.delete_collection(collection_name)

# 使用BGE模型作为embedding函数
embedding_func = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="BAAI/bge-small-zh-v1.5"
)

collection = client.create_collection(
    name=collection_name,
    embedding_function=embedding_func,
    metadata={"description": "RAG教程知识库"}
)

# ——— CREATE: 添加文档 ———
collection.add(
    documents=[
        "RAG是检索增强生成的缩写，是2023年最火的AI应用范式。",
        "向量数据库通过ANN算法实现高效的近似最近邻搜索。",
        "BGE-M3是BAAI发布的多语言Embedding模型。",
    ],
    metadatas=[
        {"source": "chapter01", "page": 3, "author": "讲师"},
        {"source": "chapter07", "page": 1, "author": "讲师"},
        {"source": "chapter06", "page": 5, "author": "讲师"},
    ],
    ids=["doc001", "doc002", "doc003"]
)

# ——— RETRIEVE: 搜索 ———
results = collection.query(
    query_texts=["什么是RAG?"],
    n_results=2,
    where={"source": {"$ne": "chapter01"}},  # metadata过滤
)
print("检索结果:", results["documents"])

# ——— UPDATE: 更新文档 ———
collection.update(
    documents=["RAG是检索增强生成的缩写，用于改善LLM的知识局限性。"],
    ids=["doc001"],
    metadatas=[{"source": "chapter01", "version": 2}]
)

# ——— DELETE: 删除文档 ———
collection.delete(ids=["doc003"])

# 验证最终数量
print(f"集合中现有文档: {collection.count()} 个")

五、性能调优：efSearch与Recall的权衡

在使用HNSW索引时，一个关键的调优参数是efSearch——搜索时的动态候选集大小。它直接控制着"搜索的深度"：

  • efSearch越小（如64）：搜索速度极快，但可能错过最优结果。
  • efSearch越大（如512）：召回率接近99%，但延迟增加2~5倍。
  • 推荐起始值：300~400，然后根据实际延迟目标调低。

另一个关联参数是索引构建时的efConstruction和M（每个节点的最大连接数）。M越大，索引质量越高但构建越慢。一般M取16~32，efConstruction取100~500。

# 以Qdrant为例展示索引参数配置
from qdrant_client import QdrantClient
from qdrant_client.models import VectorParams, HnswConfigDiff

client = QdrantClient(":memory:")

client.create_collection(
    collection_name="optimized_rag",
    vectors_config=VectorParams(
        size=768,
        distance="Cosine",
        hnsw_config=HnswConfigDiff(
            m=32,              # 每个节点最大连接数
            ef_construct=200,  # 索引构建时的搜索范围
        ),
    ),
)

# 搜索时配置ef
search_result = client.search(
    collection_name="optimized_rag",
    query_vector=[...],   # 你的query embedding
    limit=10,
    search_params={"ef": 300},  # 搜索时的候选集大小
)

六、MetaData Filtered Search

在实际RAG应用中，很少有不带过滤条件的纯向量搜索。元数据过滤允许你在搜索前先缩窄范围，既提升搜索速度又提升搜索结果的相关性。

常见的过滤场景：

  • 时间范围过滤：仅搜索2025年之后的文档。
  • 部门过滤：仅搜索用户所属部门的文档。
  • 文档类型过滤：仅搜索PDF或仅搜索网页。
  • 权限级别过滤：仅搜索用户有权访问的内容。
踩坑实录：有个团队在Milvus上部署RAG时遇到了严重的性能问题——每天的批量索引构建耗时10小时以上，远超预期。排查后发现，他们在元数据过滤中使用了一个低基数（low-cardinality）字段作为主过滤条件，但该字段没有建立倒排索引，导致每次搜索都触发全量扫描。教训是：元数据过滤字段也需要建立索引，特别是在高吞吐的在线场景中。Milvus的标量字段支持倒排索引和布隆过滤器，务必在建表时一并配置好。
最佳实践：在生产环境中，不要把向量数据库当作万能存储。推荐采用"双库架构"——向量数据库负责向量+关键metadata+文本chunk，关系数据库负责用户管理、权限树、操作日志等结构化数据。各司其职，系统才会健壮。
