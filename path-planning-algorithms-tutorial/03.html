大家好！上一章我们学会了用 C-Space 把"有形状的机器人"抽象成"一个点"。但抽象成点之后呢？我们怎么让这个点在自由空间中"走"起来？这就需要把连续的空间离散化——用图（Graph）这种数据结构来建模。

你可能觉得图论是数学课上的东西，和机器人路径规划八竿子打不着。但事实上，路径规划中一半以上的经典算法都是"图搜索算法"——Dijkstra、A*、D* Lite、JPS 全部是。如果你不了解图是什么、图怎么表示、怎么遍历，你就不可能真正理解这些算法。反过来说，一旦你掌握了图论基础，A* 和 Dijkstra 对你来说就只是"在图上做搜索"——一个极其清晰的概念。

讲个故事。几年前我带一个实习生做项目，让他实现 A* 算法。他花了一天写出了代码，但跑起来总是报索引越界。我帮他一看——他用邻接矩阵存了一个 1000×1000 的网格图，也就是一个 100 万乘 100 万的矩阵。100 万个节点 × 100 万个节点 = 10¹² 个元素，每个元素就算只占 1 字节，那也是 1 TB——他的 16 GB 内存当然扛不住。这就是不了解图的表示方法在实践中的代价。

图是什么

最简洁地说：图（Graph） 由节点（Node / Vertex）和边（Edge）组成，记作 G = (V, E)。节点代表"状态"或"位置"，边代表"两个节点之间有关系"。

在路径规划的语境下：

  • 节点 = C-Space 中的离散化位置（比如栅格地图中的一个格子）
  • 边 = 从一个节点到相邻节点的可行运动（比如向上走一步、向左走一步）
  • 边的权重（Weight） = 从一个节点到另一个节点的代价（比如距离、时间、能耗）

图的两种基本类型是：

  • 无向图（Undirected Graph）——边没有方向，(u, v) 和 (v, u) 是同一回事。对应到路径规划中，就是"如果你能从 A 走到 B，那么你也能从 B 走回 A"（比如双向通行的道路）。
  • 有向图（Directed Graph / Digraph）——边有方向，(u → v) 不代表 (v → u)。对应到路径规划中，就是单行道、或者"可以下楼梯但不能上楼梯"的场景。

真实的路径规划场景可以很自然地用图来表达：

  • GPS 导航图——路口是节点，道路是边，道路长度或通行时间是权重。这是一个典型的有向加权图（因为有单行道，且不同方向可能有不同通行时间）。
  • 室内栅格地图——每个格子是一个节点，相邻格子之间有边，权重通常是 1（水平/垂直）或 √2（对角）。这是一个无向无权图（如果忽略地形差异）。
  • 游戏寻路——虚拟地图的导航网格（NavMesh）中，凸多边形的中心是节点，相邻多边形连通处有边。

图的表示方法

在计算机中，图主要有两种表示方法：邻接矩阵（Adjacency Matrix）和邻接表（Adjacency List）。初学者经常在这上面踩坑，选错了表示方法会导致内存爆炸或性能瓶颈。

邻接矩阵是一个 |V| × |V| 的二维数组，matrix[i][j] 表示从节点 i 到节点 j 是否有边（或边的权重）。

邻接表则是一个长度为 |V| 的数组，每个元素是一个链表（或列表），存储了该节点所有的邻居节点和对应的边权重。

下面这个对比表格清楚地展示了两种方法的差异：

操作 邻接矩阵 邻接表
空间复杂度 O(|V|²) O(|V| + |E|)
查询 (u, v) 是否有边 O(1) O(degree(v))
遍历节点 u 的所有邻居 O(|V|) O(degree(u))
添加一条边 O(1) O(1)
删除一条边 O(1) O(degree(v))
适合的图类型 稠密图（|E| ≈ |V|²） 稀疏图（|E| << |V|²）
性能陷阱：邻接矩阵的空间复杂度是 O(|V|²)。在路径规划中，一张 1000×1000 的栅格地图就有 10⁶ 个节点。如果用邻接矩阵，你需要 10¹² 个元素——就算每个元素只存 1 字节的权重，也需要 1 TB 内存。而用邻接表，每个节点平均有 4-8 个邻居，只需要约 8×10⁶ 个元素，约 64 MB。栅格地图天然就是稀疏图——绝大部分节点之间没有直接相连的边，所以永远用邻接表！

在路径规划实践中，我们几乎总是使用邻接表。根本原因在于：路径规划中的图（尤其是栅格地图）是高度稀疏的——每个节点只与它的直接邻居相连（4 个或 8 个），而不是与其他所有节点相连。用邻接矩阵来存这种稀疏图是对内存的巨大浪费。

图的遍历

有了图的表示，下一个问题就是：怎么在图上"走"？这就是图的遍历（Graph Traversal）。两种最基本的遍历方法是广度优先搜索（BFS）和深度优先搜索（DFS）。

广度优先搜索（BFS）

BFS 的核心思想是：从起点出发，先探索所有距离为 1 的节点，再探索所有距离为 2 的节点，以此类推。想象你把一块石头扔进池塘，水波一圈一圈向外扩散——BFS 就是这样的"波纹扩散"。

BFS 的一个重要性质是：在无权图中，BFS 找到的第一个到达终点的路径，就是最短路径（边数最少）。这是因为 BFS 按离起点的距离逐层探索，第一次遇到终点时一定是最短的距离。

下面是 BFS 的 Python 实现。这段代码也是后面 Dijkstra 和 A* 的"骨架"——理解了它，你就理解了搜索算法的一半。

from collections import deque

def bfs(grid, start, goal):
    """在栅格地图上执行 BFS 搜索。
    grid: 2D 数组，0=可通过，1=障碍物
    start, goal: (row, col) 元组
    """
    rows, cols = len(grid), len(grid[0])
    visited = [[False] * cols for _ in range(rows)]
    # 用于回溯路径的父节点记录
    parent = {}
    # 四个方向：上、下、左、右
    directions = [(-1, 0), (1, 0), (0, -1), (0, 1)]

    queue = deque([start])
    visited[start[0]][start[1]] = True

    while queue:
        r, c = queue.popleft()

        if (r, c) == goal:
            # 回溯路径
            path = []
            node = (r, c)
            while node in parent:
                path.append(node)
                node = parent[node]
            path.append(start)
            return path[::-1]

        for dr, dc in directions:
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols:
                if not visited[nr][nc] and grid[nr][nc] == 0:
                    visited[nr][nc] = True
                    parent[(nr, nc)] = (r, c)
                    queue.append((nr, nc))

    return None  # 未找到路径

深度优先搜索（DFS）

DFS 的策略则相反：从起点出发，沿着一条路径一直走到头，走不通了再回头（回溯）。DFS 在路径规划中很少直接用于找路（因为它找到的路径通常不是最短的），但在拓扑排序、环检测、连通性分析等任务中很有用。

以下是 DFS 的 Python 实现（递归版本）：

def dfs(grid, start, goal):
    """在栅格地图上执行 DFS 搜索。"""
    rows, cols = len(grid), len(grid[0])
    visited = [[False] * cols for _ in range(rows)]
    parent = {}
    directions = [(-1, 0), (1, 0), (0, -1), (0, 1)]

    def dfs_recursive(r, c):
        if (r, c) == goal:
            return True
        visited[r][c] = True
        for dr, dc in directions:
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols:
                if not visited[nr][nc] and grid[nr][nc] == 0:
                    parent[(nr, nc)] = (r, c)
                    if dfs_recursive(nr, nc):
                        return True
        return False

    if dfs_recursive(start[0], start[1]):
        path = []
        node = goal
        while node in parent:
            path.append(node)
            node = parent[node]
        path.append(start)
        return path[::-1]
    return None
选 BFS 还是 DFS？在路径规划中，如果你只需要找到一条路径（不要求最短），且图非常大（深度远大于宽度），DFS 可能更快找到路径——但这只是少数情况。绝大多数情况下，路径规划的首选是 BFS（或后面要讲的 A*）。不过 DFS 的思路在 RRT 这类采样算法中有重要的变体应用，我们在第 11 章会讲到。

图在路径规划中的应用

现在我们知道了图的定义、表示和遍历。但在路径规划中，"图"是怎么来的呢？主要有以下几种方式：

栅格到图的转换

这是最简单也最常见的方式。一张 2D 栅格地图中，每个格子（cell）是图的一个节点。如果两个格子相邻且都可通过，它们之间就有一条边。在 4-连通（4-connected）模型中，每个格子最多有 4 个邻居（上下左右）；在 8-连通（8-connected）模型中，则还有四个对角线方向。

这种转换隐含了一个重要的假设：离散化精度。栅格越细，地图分辨率越高，路径质量越好，但图的节点数越多，搜索越慢。这是一个经典的内存-质量权衡。第 4 章我们会深入讨论这个问题。

可视图（Visibility Graph）

对于多边形障碍物的环境，可视图是一种优雅的构造方法：以起点、终点和所有多边形的顶点为节点，如果两个节点之间的连线不穿过任何障碍物，则添加一条边。在可视图上运行 Dijkstra 或 A* 得到的就是最短路径（但路径可能紧贴障碍物边缘，不够安全）。

可视图的优点是找到的路径是最优的（在几何意义上），缺点是构造可视图本身的计算量不小——你需要对每对节点做线段-多边形相交检测，复杂度为 O(|V|²)。

导航网格（Navigation Mesh / NavMesh）

NavMesh 是游戏 AI 中的主流方法。它将可行走区域分割成一系列凸多边形，以每个凸多边形的中心（或边中点）为节点构造图。相比栅格图，NavMesh 的节点数少得多，搜索更快，且路径更加自然。NavMesh 的缺点是实现复杂，且不适用于高维 C-Space。

在后面的课程中，我们主要使用栅格图作为图的来源——因为它最直观、最容易在代码中生成、且与我们将要实现的算法配合最好。所以，当你听到"把地图离散化为栅格"时，请记住：这本质上就是用每个格子作为节点，构造一张大型稀疏图。而你接下来要学的所有搜索算法，都在跑在这张图上。

思维框架：路径规划 = 图构造 + 图搜索。把环境转化为图（本章内容），然后在图上执行搜索算法（第 5-14 章内容）。这两个步骤是相互独立的——你可以在栅格图上做 A*，也可以在可视图上做 A*，A* 算法本身不需要知道图的来源。这种"解耦"的设计，让你可以自由组合不同的建图方式和搜索策略。

本章我们建立了图论的基础知识——图的结构、两种表示方法及其权衡、BFS 和 DFS 遍历。最重要的是，我们理解了路径规划中的"图"从何而来。下一章我们将具体研究路径规划中最常用的图类型——栅格地图与代价地图。我们将看到，在实际的机器人系统中，地图不是一张简单的二维数组——它有分层结构、有概率模型、有各种精妙的工程技巧。
