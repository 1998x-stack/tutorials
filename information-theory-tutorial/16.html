第16章 循环码与CRC

大家好，我是你们的讲师。你有没有想过一个问题：当你从网上下载一个文件——无论是一张照片、一个压缩包、还是一个软件安装包——系统怎么能确保数据没有被损坏？

你可能会说"那还不简单，校验一下就行"。但具体怎么校验？一个几GB的安装包，如果从头到位都传错了几个比特，你怎么知道？更关键的是，这个验证过程必须快到不影响用户体验——没有人愿意下载完一个文件后再花10分钟等它校验。

答案很可能就是CRC（Cyclic Redundancy Check，循环冗余校验）。它几乎出现在每一个数字通信协议中：以太网、WiFi、USB、蓝牙、ZIP压缩包、PNG图片……甚至你每天启动电脑时，BIOS也用CRC来验证固件是否完整。

而CRC背后的数学，正是循环码（cyclic code）。如果说上一章的线性分组码是"用向量"，循环码就是"用多项式"——更优雅，更适合硬件实现。

循环码的代数表示

循环码是线性分组码的一个子类，增加了一个关键约束：任何码字的循环移位仍然是码字。也就是如果(c0, c1, ..., cn-1)是码字，那么(cn-1, c0, c1, ..., cn-2)也是码字。

这个看似简单的性质，让循环码有了极其丰富的代数结构。我们可以用多项式来表示码字：

c(x) = c0 + c1x + c2x2 + ... + cn-1xn-1

每个系数ci ∈ {0, 1}，多项式在GF(2)上运算（加法就是异或）。循环移位对应着多项式乘以x后模(xn - 1)：

c'(x) = x · c(x) mod (xn - 1)

一个(n, k)循环码由一个生成多项式g(x)唯一确定，它的次数是n - k。每个码字多项式c(x)都可以写成：

c(x) = m(x) · g(x)

其中m(x)是k - 1次的信息多项式。也就是说，编码就是把消息多项式乘以生成多项式。g(x)必须是xn - 1的一个因式。

举个具体例子：考虑(7, 4)循环码（注意不同于(7, 4)汉明码，但两者同构）。生成多项式可以选择g(x) = 1 + x + x3。消息(1011)对应多项式m(x) = 1 + x2 + x3，编码后：c(x) = m(x)g(x) = (1 + x2 + x3)(1 + x + x3) = 1 + x + x3 + x5 + x6，对应码字(1101011)。

校验也同样优雅：c(x)能被g(x)整除，所以如果接收到的r(x)除以g(x)有余数，就说明出错了。

CRC的原理与实现

CRC的全称是Cyclic Redundancy Check——它本质上就是一个循环码用于检错的场景。注意：CRC只检错，不纠错。

CRC的流程非常简洁：

  1. 将数据看作一个二进制多项式d(x)
  2. 在数据末尾追加r个0（相当于d(x) · xr），其中r是CRC校验位的长度
  3. 用生成多项式g(x)（次数为r）去除这个多项式
  4. 将余数（r位）附加到原始数据后面发送
  5. 接收方做同样的除法，如果余数为0，认为数据完好；否则检测到错误

CRC之所以如此流行，核心原因是：它可以用硬件极其高效地实现。关键组件是一个LFSR（线性反馈移位寄存器，Linear Feedback Shift Register）——一组移位寄存器和异或门。

让我用Python从头实现一个CRC-32来展示内部原理：

def crc32_polynomial_division(data: bytes) -> int:
    """
    用多项式长除法实现CRC-32。
    生成多项式: x^32 + x^26 + x^23 + x^22 + x^16 +
                x^12 + x^11 + x^10 + x^8 + x^7 +
                x^5 + x^4 + x^2 + x + 1
    二进制: 0x04C11DB7
    """
    # 生成多项式（去掉最高位x^32，隐含在算法中）
    POLY = 0x04C11DB7
    # 初始值
    crc = 0xFFFFFFFF

    for byte in data:
        # 将当前字节与CRC的高8位异或
        crc ^= (byte << 24)
        for _ in range(8):
            if crc & 0x80000000:  # 检查最高位
                crc = (crc << 1) ^ POLY
            else:
                crc <<= 1
            crc &= 0xFFFFFFFF  # 保持32位

    return crc ^ 0xFFFFFFFF  # 最后异或

# 测试
test_data = b"Hello, Information Theory!"
crc_result = crc32_polynomial_division(test_data)
print(f"数据: {test_data}")
print(f"CRC-32: 0x{crc_result:08X}")

# 验证 - 与内置CRC-32对比
import binascii
expected = binascii.crc32(test_data) & 0xFFFFFFFF
print(f"内置CRC-32: 0x{expected:08X}")
print(f"匹配: {crc_result == expected}")

CRC的LFSR实现只需要几个寄存器加异或门，在硬件上跑一个时钟周期处理1比特。现代实现会使用"查表法"一次处理8比特（1字节），速度提升8倍，几乎不消耗额外硬件资源。

常见CRC标准

不同的应用场景对CRC的检错能力和码长需求不同，因此存在多种CRC标准。

CRC标准 生成多项式 校验位数 典型应用
CRC-8 0x07 (x8 + x2 + x + 1) 8 1-Wire、SMBus
CRC-16-CCITT 0x1021 (x16 + x12 + x5 + 1) 16 XMODEM、Bluetooth、PPP
CRC-16-IBM 0x8005 (x16 + x15 + x2 + 1) 16 USB、Modbus
CRC-32 0x04C11DB7 32 Ethernet、ZIP、PNG、Gzip
CRC-64-ECMA 0x42F0E1EBA9EA3693 64 文件一致性校验

其中CRC-32可能是最熟悉的一个——你每次用Git提交代码时，Git都会用CRC-32（实际上是更快的变体）来校验对象存储的完整性。Ethernet帧尾部也有一个32位的CRC-32字段，确保数据帧在物理介质上传输时没有被噪声干扰。

CRC的检错能力

CRC的检错能力可以用生成多项式的数学性质来精确描述。对于一个r位的CRC：

  • 所有长度为≤r的突发错误都能被检测。因为任何长度≤r的突发错误对应的多项式次数＜r，不能被g(x)（次数为r）整除。
  • 如果g(x)包含因子(x + 1)，可以检测所有奇数个比特错误。
  • 长度为r + 1的突发错误，未被检测到的概率为2-(r-1)。
  • 更长的突发错误，未被检测到的概率≈ 2-r。

这意味着CRC-32的未检测错误概率大约是2-32 ≈ 2.3 × 10-10。如果每秒传输1Gbps的数据，平均每传输约18TB的数据才可能出现一次未检测到的错误——对于绝大多数应用来说，这完全足够。

重要提示：CRC只能检错，不能纠错！它的设计目标是以极低的额外开销发现传输错误——对于CRC-32来说，每4GB数据才增加32比特开销（约十亿分之一的冗余率），性价比极高。但如果你需要纠错能力，那就需要更强大的码结构，比如BCH码或Reed-Solomon码。

一个真实的踩坑故事

这里分享一个我在嵌入式开发中遇到的教训。某次项目中，我们使用一个自制的通信协议在两块板子之间传数据，CRC-16校验。一切正常——直到某天现场反馈数据偶尔出错。我们检查了好多天，百思不得其解，因为CRC-16会漏检的情况实在是太罕见了。

最后发现：问题不在CRC的检错能力，而在于CRC的计算范围。我们写固件时，一个同事不小心把CRC计算的起始地址算错了2个字节——CRC覆盖了数据的一部分，却没覆盖另一部分。结果CRC本身是对的（因为它覆盖的那部分确实没错），但实际数据错了。CRC没有错，是CRC计算错了。

这个教训告诉我们：CRC本身是可靠的，但实现CRC的人可能不可靠。在调试这类问题时，永远先确认CRC计算的范围是否正确，这是最容易被忽视的环节。

小结与预告

循环码用多项式的代数结构为纠错编码带来了全新的视角。CRC作为循环码最成功的应用之一，以其极低的成本和极高的检错能力，成为数字通信的事实标准。从你浏览网页到下载文件，CRC始终在默默守护数据的完整性。

但循环码和上一章的汉明码都是分组码——每次处理固定长度的消息块。而通信往往是连续的比特流，每比特都不能等。下一章我们要讲的卷积码就是为"流式"纠错而生的——配合Viterbi算法，它成为了深空通信、无线通信等领域的王者。
