开篇：当互联网不再够用，我们开始自己造数据

大家好，我是你们的讲师。今天我们要讨论一个有点"拧巴"的话题——合成数据。你可能会有疑问：前几章我们一直在讲如何清洗、过滤、去重，目的就是把"人造"的数据搞干净。现在怎么反过来要教大家用模型生成数据？这不是自己给自己挖坑吗？

这个问题问得很好。让我讲一个真实的故事：2025 年初，我参与了一个大模型预训练项目，目标是要训练一个在表格理解和结构化推理方面特别强的模型。我们翻遍了 Common Crawl、各类开源数据集，发现了一个扎心的事实——互联网上天然的优质表格数据非常稀缺。大部分表格要么是广告页面里的产品比价表、要么是排版混乱的 HTML 表格、要么干脆是图片而不是文字。我们面临一个选择：要么接受现有数据中那可怜的一丁点表格样本（大概不到 0.1%），让模型在这个能力上天生残缺；要么——自己生成。我们选择了后者。这个过程让我深刻理解了合成数据的价值、风险，以及那根不能触碰的红线。

19.1 FinePhrase 与结构化改写

2026 年，HuggingFace 团队发表了一个意味深长的研究——FinePhrase。他们发现一个反直觉的结论：用 LLM 对互联网文本进行结构化改写，得到的合成数据效果远超原始网络文本。而且更关键的是：生成器的参数规模超过 1B 之后，收益就不再增长了。

FinePhrase 的核心思路是把杂乱的网络文本"翻译"成几种高信息密度的格式：

改写格式 原始输入 合成输出 对模型能力的提升
表格 (Table) 一段描述销售额的段落 Markdown 表格 结构化推理 +12%
教程 (Tutorial) API 文档片段 步骤式教程 指令遵循 +15%
FAQ 论坛问答帖子 格式化问答对 知识召回 +9%
数学推导 (Math) 结论性陈述 带步骤的推导过程 数学推理 +18%

下面是一个具体的 FinePhrase 改写示例——把一段描述性文本改写成表格：

【原始输入】
GPT-4o的推理速度在标准设置下约为每秒45个token，
使用Flash Attention后提升到约82个token/s，
如果启用KV-cache则达到约120个token/s。
在成本和效率方面，标准设置每百万token约0.15美元，
启用优化后可以降低到约0.08美元。

【FinePhrase改写输出 —— 表格格式】
| 配置模式 | 推理速度 (token/s) | 成本 ($/M token) |
|----------|-------------------|-----------------|
| 标准设置 | 45                | 0.15            |
| +Flash Attention | 82          | 0.10            |
| +KV-cache | 120                | 0.08            |

FinePhrase 的另一个关键发现是生成器规模 1B 封顶。实验表明，使用 1B 左右的模型做改写，效果和 70B 模型几乎无差异，但成本只有后者的几十分之一。这对于预算有限的团队来说是个巨大的利好消息——合成数据不需要烧大模型的钱。

关键洞察：FinePhrase 的实验表明，合成数据的质量更多地取决于改写方案的设计（prompt engineering 和 target format 的设计），而不是生成模型的大小。1B 模型 + 精心设计的 prompt 可以打败 70B 模型 + 粗糙的 prompt。

19.2 知识图谱引导的合成：Sutra-10B 模式

如果 FinePhrase 解决的是"如何把现有文本变好"，那 Sutra-10B 的思路就是"从零开始生成本来不存在的内容"。Sutra-10B 的团队设计了一套基于知识图谱（Knowledge Graph, KG）的合成数据方法，专门针对教育类数据集。

他们的做法是：先构建一个学科的知识图谱，节点是概念，边是"前置依赖"关系（比如"要理解链式法则，必须先掌握导数定义"）。然后让 LLM 沿着图中的概念路径逐一生成教学材料——确保覆盖完整、不遗漏关键知识点、且遵循合理的教学顺序。

这个方法的威力体现在两个维度：

  • 概念覆盖率：传统的网络爬虫有严重的"马太效应"——热门概念被无数文章覆盖，冷门概念几乎找不到资料。KG 引导可以保证每个概念至少被生成一次。
  • 前置依赖准确：模型不会跳过一个需要前置知识的概念——KG 显式约束了生成顺序，避免了"还没讲矩阵乘法就教特征分解"这种错误。

在实践中，KG 引导合成的典型 pipeline 是这样的：

# Sutra-10B 风格的 KG 引导合成流程

1. 构建领域知识图谱
   - 从教科书、维基百科提取概念节点
   - 标注 prerequisite 关系边

2. 拓扑排序得到教学路径
   - 确保每个概念的所有前置依赖已被覆盖

3. 沿路径生成教学内容
   for concept in teaching_path:
       prompt = f"""
       请根据以下概念，生成一段教学文本：
       概念：{concept.name}
       前置知识：{concept.prerequisites}
       目标受众：大学生初级水平
       需要包含：定义、例子、常见误区
       """
       generated_text = llm.generate(prompt)

4. 质量验证
   - 概念覆盖检查：KG 中每个节点至少命中一次
   - 推理链验证：确保推导步骤完整
   - 领域专家抽样评审
实现建议：如果资源有限，不需要从零建 KG。可以直接使用 Wikipedia 的分类体系（Category Graph）或 DBpedia 的结构化数据作为知识图谱骨架。很多教育类 KG（如 Khan Academy 的概念图谱）也有开源版本可用。

19.3 LLM 改写技术：信号放大

合成数据不仅仅是"无中生有"，更重要的应用场景是信号放大——把数据中已经存在但不够突出的有用信号放大，让模型更容易学到。实践中主要有三种有效的改写策略：

策略一：体裁-受众变换 (Genre-Audience Transformation)

把一篇面向专家的学术论文段落，改写成面向本科生的讲解；或者把一段技术文档改写成面向产品经理的概要。这种方式不增加信息量，但显著改变了信息的可学习性——模型在预训练阶段就能接触到更多样化的表达方式。

策略二：忠实致密化 (Faithful Densification)

这是最让我惊喜的一种策略。它的做法是：把一段信息密度低的文本（比如社交媒体帖子、论坛闲聊）中的有用信息"挤出来"，重新组织成高密度的形式。举个例子：一个 Reddit 帖子里可能藏着非常有价值的工程经验，但被几百字的抱怨和调侃包围着。忠实致密化就是把这几十个字的核心知识提取出来，写成一段干净的教程。关键是"忠实"——不能添加原文没有的信息。

策略三：去包装 (De-wrapping)

你可能见过这种网页：一篇高质量的技术文章，被广告、推荐模块、评论区、导航栏层层包裹。模型在训练时会学到大量的"包装噪声"。去包装就是用 LLM 把这些干扰去掉，只保留正文核心内容。听起来简单，但实践效果出奇地好——在 FineWeb-Edu 的实验里，去包装后的文本在质量评分上平均提升了 30% 以上。

实践经验：如果你只能选择一种改写策略，优先做"去包装"。这是性价比最高的——只需要识别内容边界然后裁剪，几乎不引入幻觉风险。

19.4 Model Collapse 与合成-天然混合策略

这是合成数据领域绕不开的房间里的大象。2024 年 Nature 上那篇关于 Model Collapse 的研究让整个行业警醒：当模型反复用合成数据训练，分布会逐渐偏离真实人类分布，最终导致不可逆的质量崩塌。

但这是不是意味着不能用合成数据？当然不是。关键在于比例和混合方式。

截至 2026 年初，业界已经形成了一些共识性的最佳实践：

混合比例 合成 : 天然 实验效果 Model Collapse 风险
轻度增强 1 : 9 小幅提升，安全 极低
推荐比例 1 : 3 显著提升，经过验证 可控
平衡 1 : 1 收益见顶 中等
高风险 3 : 1 短期提升，长期崩塌 高
纯合成 1 : 0 不可逆坍缩 必然发生
红线警告：切勿在没有任何天然数据的情况下完全依赖合成数据。比例不超过 1/3（即合成:天然 = 1:2）是目前经过大规模验证的安全边界。如果合成数据比例超过 50%，请确保每次训练迭代都混入新的天然数据来"刷新"分布。

19.5 人工验证循环与多样性评估

合成数据最被诟病的一点是多样性不足——同一个 LLM 生成的文本在句式、词汇、论证方式上往往存在模式化的痕迹。为此，建立一套闭环的验证机制至关重要。

在我的团队实践中，我们采用了"三层次验证"：

  • 第一层：自动化质量检查。每条合成数据都需要通过一系列硬性指标：长度下限、语言识别置信度、与原文的语义相似度（防止过度偏离）、n-gram 多样性分数。
  • 第二层：多样性度量。计算合成数据集的 self-BLEU、distinct-n (dist-1/dist-2)、压缩比等指标。如果 self-BLEU 过高（比如 > 0.7），说明生成内容同质化严重，需要调整生成参数（提高 temperature，增加 prompt 多样性）。
  • 第三层：领域专家抽样审核。每 1000 条中随机抽 10 条，由该领域的专家人工评估。专家需要标注：事实正确性（1-5）、教学清晰度（1-5）、是否存在幻觉（是/否）。

下面是一个多样性检查的简单脚本思路：

# 合成数据多样性检查
from collections import Counter

def diversity_metrics(texts):
    all_tokens = []
    for text in texts:
        tokens = text.split()
        all_tokens.extend(tokens)

    total = len(all_tokens)
    unique = len(set(all_tokens))

    # Type-Token Ratio
    ttr = unique / total if total > 0 else 0

    # Self-BLEU (简化为bigram precision)
    bigrams = [tuple(all_tokens[i:i+2])
               for i in range(len(all_tokens)-1)]
    unique_bigrams = len(set(bigrams))
    bigram_diversity = unique_bigrams / len(bigrams) if bigrams else 0

    return {
        "type_token_ratio": round(ttr, 4),
        "bigram_diversity": round(bigram_diversity, 4),
        "total_tokens": total
    }

# 如果 TTR < 0.3 或 bigram_diversity < 0.1
# 说明合成数据的多样性严重不足，需要调整生成策略

19.6 实战：FineWeb-1T-Sample 改写 Pipeline

让我用一个实际的案例来串联本章的知识点。假设我们手头有 HuggingFace 的 FineWeb-1T-Sample（约 100B token，经过质量筛选的 Common Crawl 数据），现在想通过合成数据进一步增强它的质量。

Pipeline 设计如下：

  • 第一步（分类）：用一个轻量分类器（如 DistilBERT）将 100B token 按内容类型分为：教程类、文档类、问答类、叙事类、表格类等。
  • 第二步（改写分配）：对教程类 -> 用 FinePhrase 的"步骤式教程"改写；对问答类 -> 用 FAQ 改写；对表格类 -> 用 Table 改写。
  • 第三步（控制比例）：每个类型最多改写原数据的 20%，总合成数据量控制在总数据量的 25%（远低于 1/3 的安全线）。
  • 第四步（混合）：将改写后的数据与原始数据按章节打乱混合，确保训练时看到的是天然与合成交替的序列。
  • 第五步（验证）：三层次验证 + 小规模消融实验（如用 1B 模型在 C4 基准上对比混合前后的效果）。
实践建议：合成数据的最佳使用方式是"润色"而非"替代"。我们的目标是补充天然数据中不足的信号（如表格、推导步骤、结构化指令），而不是用合成数据替代天然数据的多样性和真实性。

本章小结

合成数据已经从"被迫之举"演变为预训练数据工程中的标准组件。FinePhrase 告诉我们，1B 模型 + 精心设计的改写 prompt 足以产出高质量的增强数据；Sutra-10B 展示了知识图谱引导下"无中生有"的可能性；而 Model Collapse 的警示则提醒我们——合成是工具，不是解药。在合成与天然之间找到正确的平衡，把 1/3 的安全比例刻在心里，才是可持续的数据增强之道。
