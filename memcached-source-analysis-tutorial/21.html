第21章 Meta Protocol：新一代协议

大家好，我是你们的讲师。如果你跟着课程一路看下来，应该注意到了 ASCII 协议和二进制协议各自的问题：ASCII 带宽浪费、解析慢；二进制协议扩展性差、opcode 爆炸。那么有没有一种协议既能保持 ASCII 的灵活性，又能达到二进制的性能？答案是 Meta Protocol——Memcached 1.6 中引入的新一代协议。

Meta Protocol 的核心理念可以用一句话概括：用模式标志（M）取代固定的命令变体。它不再为"静默 SET"单独定义一个 opcode，而是用一个 quiet 标志来修饰 SET 命令；不再为"GET + 更新 TTL"搞一个新命令，而是用一个 T 标志告诉 GET "帮我更新过期时间"。这种"命令 + 标志"的模型不仅大幅简化了协议实现，还让 Memcached 的能力边界有了质的飞跃——比如 stale-while-revalidate 模式，这是旧协议完全无法做到的。

Meta Protocol 的设计动机

Meta Protocol 的设计者 Dormando 在社区讨论中多次提到一个观点："我们不想再增加新的 opcode 了"。二进制协议的 opcode 表从最初的十几个扩展到了 30 多个，每个新功能都意味着一个新的 opcode，甚至需要一对同步/静默变体。这不仅让客户端库的实现变得复杂，也让服务端的 dispatch 逻辑越来越臃肿。

Meta Protocol 的解决方案是：定义一组通用的、面向操作的命令（mg 对应 GET、ms 对应 SET、md 对应 DELETE 等），然后通过模式标志（mode flags）来微调每个命令的行为。这就像是函数式编程中的"高阶函数"——不是增加更多的基础函数，而是让现有函数通过参数获得更强的表达能力。

设计哲学：Meta Protocol 的目标是成为一个"超集协议"——它要取代 ASCII 协议和二进制协议，成为 Memcached 的唯一协议。目前虽然三个协议并存，但所有新特性（stale-while-revalidate、win flag、客户端竞态保护）都只在 Meta Protocol 中提供。

Meta Protocol 命令格式

Meta Protocol 的命令采用紧凑的单行格式，所有的元信息都编码在命令行的标志字段中：

# meta get (mg)
  mg <key> [flags]\r\n
  ex: mg mykey v k T120\r\n

# meta set (ms)
  ms <key> [flags]\r\n
  <data>\r\n
  ex: ms mykey v k2 T3600\r\nhello\r\n

# meta delete (md)
  md <key> [flags]\r\n

# meta arithmetic (ma)
  ma <key> [flags]\r\n

# meta debug (me)
  me <key>\r\n

# 常用标志说明:
  v   返回值（value）
  k   返回 key
  h   返回 key 的 hash
  T   设置新的过期时间（后面跟秒数）
  N   设置 CAS token
  F   设置 flags
  O   设置 opaque
  W   设置 win flag（竞态条件防护）
  s   静默模式（不返回响应）
  q   快速失败（不触发 LRU 维护）

每个标志都是单个字符，可以自由组合。例如 mg mykey v k 表示"读取 mykey，返回 value 和 key"。这种紧凑的编码方式比二进制协议的固定 header 更灵活，因为在 Meta Protocol 中，标志的数量是可以扩展的——只需在服务端解析新的标志字符即可。

模式标志（M）详解

模式标志是 Meta Protocol 的核心创新。它允许客户端在每个命令中精细控制服务端的行为，而不需要改变协议版本或升级客户端：

标志 适用命令 含义
v mg, ma 返回 value 数据
k mg, ma 返回 key（用于确认操作的是哪个 key）
h mg 返回 key 的哈希值
T mg, ms, ma 更新/设置过期时间（类似 TOUCH）
N ms, ma 设置操作必须匹配的 CAS token（类似 CAS 语义）
F ms 设置 flags 值
O mg, ms, md, ma 设置 opaque（响应中原样返回）
W mg 启用 win flag（客户端竞态条件保护）
s mg, ms, md, ma 静默模式（不返回响应，等价于 noreply）
q mg 快速模式——如果 key 不存在立即返回，不触发 LRU 维护

这种设计的优势在组合使用中体现得淋漓尽致。比如 mg mykey v T3600 表示"读取 mykey，返回 value，同时将它的过期时间重置为 3600 秒"——这在二进制协议中需要 GAT 命令才能做到，而在 Meta Protocol 中只是 GET 加一个 T 标志而已。

stale-while-revalidate 模式

这是 Meta Protocol 最令人兴奋的特性，也是旧协议完全无法实现的功能。

stale-while-revalidate（简称 SWR）解决的是一个经典问题：对于一个即将过期但计算代价很高的缓存项（比如一个需要 500ms 数据库查询+渲染的页面），你是选择"在过期时让第一个请求等待 500ms 重新计算"，还是"在即将过期时就提前刷新"？SWR 模式给出了第三种答案：允许客户端读取过期数据，同时异步触发后台刷新。

在 Meta Protocol 中，SWR 通过 stale 标志位和 TTL 实现：

# 客户端设置 SWR: 正常 TTL=60s，stale TTL=300s
ms mykey v k T60 W300\r\n
hello\r\n

# 客户端读取，允许返回过期数据：
# 如果数据在 60s 内，正常返回
# 如果数据在 60-300s 之间，返回数据 + stale 标记
# 如果数据超过 300s，返回 miss
mg mykey v k\r\n

# 服务端响应（stale 情况）:
VA mykey 0 5
hello

# 注意：stale 情况下的响应不带 HD（hot data）标记
# 客户端看到 stale 标记后，可以异步发起一次刷新

在源码层面，SWR 的实现涉及 item_get() 函数的扩展——当发现 item 已经过期但仍在 stale TTL 窗口内时，不会返回 NOT_FOUND，而是返回 item 数据并标记为 stale：

// item.c 中的伪逻辑
item *item_get(const char *key, const size_t nkey, conn *c,
               const bool return_stale) {
    item *it = assoc_find(key, nkey);
    if (it == NULL)
        return NULL;

    // 检查是否过期
    if (it->exptime != 0 && it->exptime < current_time) {
        if (return_stale && it->exptime + it->stale_window > current_time) {
            // 在 stale 窗口内——返回数据，标记为 stale
            c->stale = true;
            return it;
        }
        // 真正过期，删除
        do_item_unlink(it);
        return NULL;
    }
    return it;
}
SWR 的最佳实践：我曾在 CDN 场景中使用 SWR 来缓存图片的缩略图元数据。常规 TTL 设为 1 小时，stale TTL 设为 24 小时。当缩略图生成服务宕机时，CDN 仍然可以提供"略微过时"的缩略图，而不是直接返回 404。这大幅提高了系统的可用性。唯一需要注意的是：stale 窗口不要太长，否则数据会越来越陈旧，最终影响用户体验。

客户端竞态条件保护：win flag + CAS token

在分布式缓存场景中，一个经典的问题是"惊群效应"（thundering herd）——当某个热门 key 过期时，大量并发请求会同时尝试重新计算并写入这个 key。即使使用了 CAS，也只能让一个写成功、其他的失败重试——但重试仍然意味着重复计算。

Meta Protocol 提出的解决方案是 win flag（胜利标志）。它的工作流程如下：

  1. 客户端 A 使用 mg mykey v W 读取 key，带上 win flag
  2. 如果 key 过期（miss），服务端返回 EN（empty）并附带一个 win token
  3. 多个并发读取的客户端都会收到 win token，但只有第一个使用这个 token 调用 ms 写入的才能成功
  4. 其他客户端看到 win 失败后，不需要重新计算，直接读取新写入的数据即可
# 客户端 A（获取 win token）
> mg hotkey v W\r\n
< EN 0 0\r\n
< WA 12345678\r\n

# 客户端 A 使用 token 写入（成功）
> ms hotkey v T3600 N12345678\r\n
< HD 1\r\n

# 客户端 B 也尝试写入（失败——token 已被使用）
> ms hotkey v T3600 N12345678\r\n
< NS 1\r\n

这个机制比 CAS 更高级：CAS 是在"已经存在的情况下避免覆盖"，而 win flag 是从"不存在的状态下获取写权限"。这本质上是一种分布式租约（distributed lease）机制，只是 Memcached 用一种非常轻量的方式实现了它。

踩坑：Meta 协议中 TTL 语义变化导致缓存击穿。有一次我们从旧版客户端迁移到支持 Meta Protocol 的新客户端，发现缓存命中率骤降，数据库压力暴增。排查后发现原因：新客户端在 SET 操作时使用 T 标志设置了 TTL，但理解错了单位——旧协议中 exptime 的单位是秒（超过 30 天视为 Unix 时间戳），而 Meta Protocol 的 T 标志的 TTL 值始终是秒数偏移量，不再支持"超过 30 天视为绝对时间"的隐式语义。我们的代码里传了一个未来的时间戳作为 TTL，结果被解释为相对于当前时间的秒数偏移，导致几乎立即过期。这个问题的教训是：升级协议时一定要重新检查 TTL 的语义，不能理所当然地认为"和以前一样"。

与旧协议的对比：功能矩阵表

为了让你直观地感受 Meta Protocol 的能力，我们来看这张功能对比表：

功能特性 ASCII 协议 二进制协议 Meta Protocol
基本 GET/SET/DELETE ✓ ✓ ✓
CAS 乐观锁 ✓ (gets/cas) ✓ (opaque CAS) ✓ (N 标志)
静默操作（noreply） ✓ ✓ (Q 变体) ✓ (s 标志)
GAT（GET + 更新 TTL） ✗ ✓ ✓ (T 标志)
stale-while-revalidate ✗ ✗ ✓
win flag 竞态保护 ✗ ✗ ✓
返回 key 和 hash ✗ ✓ (GETK) ✓ (k/h 标志)
SASL 认证 ✗ ✓ ✗（暂不支持）
流水线（pipelining） 有限 ✓ ✓
扩展性 差（需要解析新关键词） 中（需要新 opcode） 好（增加新标志即可）

从表中可以清楚地看到，Meta Protocol 在功能覆盖上是 ASCII 和二进制协议的超集——除了 SASL 认证（因为 Meta Protocol 的设计原则之一就是轻量，不做认证层，推荐在网络层解决安全问题）。

迁移建议：新项目直接用 Meta Protocol

如果你正在启动一个新的 Memcached 项目，我的建议非常明确：直接用 Meta Protocol。理由有三：

  • 功能最全：SWR、win flag、灵活的标志组合——这些是旧协议无法提供的
  • 未来兼容：Memcached 社区明确表示新特性只在 Meta Protocol 中提供。选用 Meta Protocol 意味着你有最快的特性更新速度
  • 性能相当：虽然 Meta Protocol 是文本格式（不是二进制的），但得益于紧凑的标志编码和最小化的响应格式，它的实际性能接近于二进制协议。特别是对于 value 较大的场景，带宽差异可以忽略不计

不过，对于有历史负担的旧项目，迁移到 Meta Protocol 需要一些额外的工作量——你需要更新客户端库，测试 TTL 语义的变化，并确保所有的工具链（如监控、调试脚本）也支持新协议。不必急于一次性迁移，三个协议可以在同一个 Memcached 服务上共存。

章节小结：Meta Protocol 是 Memcached 协议演进的集大成者。它用"命令 + 模式标志"的模型取代了固定的 opcode 表，赋予了每个操作前所未有的灵活性。stale-while-revalidate 和 win flag 这两个特性，更是解决了分布式缓存中长期存在的"缓存击穿"和"惊群效应"问题。理解 Meta Protocol 的设计思路，对你设计自己的网络协议也有很大的启发价值。下一章我们将走向 Memcached 内存管理的另一个方向——当内存不够用的时候，能不能用 SSD 来兜底？这就是 External Storage（extstore）的故事。
