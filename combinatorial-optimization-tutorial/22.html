开篇：一个让我损失了半年奖金的在线决策

大家好，我是你们的讲师。2017年我为一个电商平台做"实时出价系统"（Real-Time Bidding）的优化顾问。系统需要在几百毫秒内决定：对当前的这个广告展示机会，要不要出价？出多少钱？

我们设计了一个相当复杂的离线模型——用过去30天的数据训练了一个竞价策略，在离线测试中表现完美。上线那天，正好是"双十一"预热期，流量激增。我们的模型在第一个小时就把预算花掉了40%——因为它按照"历史最优"出价，没有意识到今天的流量是平时的10倍，竞价的激烈程度也完全不同。等到我们发现这个问题时，一半的预算已经花在了低质量的展示上，而后半天的大促高峰却没钱出价了。

那个月的营销效果惨不忍睹，我的咨询费被扣了一半。

这个教训深刻而痛苦：在离线环境中完美，不意味着在在线环境中能活下来。这就是本课程中"在线与随机优化"这一章要解决的问题——当你必须在信息不完全的情况下实时做决策，而且决策不可撤销时，你该怎么办？

核心区别：标准的组合优化（第1-21章）解决的是"在已知所有数据后，找到最优解"的问题。在线优化解决的是"数据逐段到来，边到边决策，且不可回头"的问题。这就像高考（标准优化）和辩论赛（在线优化）的区别——高考你可以把所有题看完再作答，辩论赛你必须在听到对方发言的瞬间就回应。

好，让我们正式进入在线与随机优化的世界。

在线算法

模型设定与竞争比

在线算法（Online Algorithm）的框架如下：输入不是一次性给出的，而是按时间顺序逐步到达。在每个时间步，算法必须基于当前已知信息做出决策，且决策不可撤销。算法不知道未来的输入。

如何评估一个在线算法的好坏？不能用"绝对最优"来衡量——因为不知道未来，在线算法自然比不过能预知未来的最优离线算法。于是我们引入竞争比（Competitive Ratio）：

对于最小化问题，在线算法ALG的竞争比为 c 如果：
对任意输入序列，ALG(I) ≤ c · OPT(I)

其中 OPT(I) 是"离线最优"——即一开始就知道全部输入时的最优值。

如果 c 是一个常数（不随输入规模变化），则称算法具有常数竞争比。

注意竞争比和近似比的区别：近似比是和"已知全部输入的最优解"比（但受计算能力限制），竞争比也是和"已知全部输入的最优解"比（但受信息限制）。两者看似相同，但难度源头不同——近似难的源是计算，在线难的源是信息。

对抗性输入与最坏情况分析

竞争比分析假设"最坏情况"的输入序列。这引出了对抗性输入（Adversarial Input）的概念——你的在线算法的对手是一个"邪恶"的上帝，他可以根据你算法的行为选择最不利的输入序列。

对抗性分析非常残酷，但它也促使我们设计出真正鲁棒的在线算法。毕竟，在广告竞价中，"对抗性输入"并不是理论幻想——你的竞争对手的策略就是对抗性的。

经典在线问题

滑雪板租赁问题——2竞争比

这是在线算法中最经典的入门问题，几乎每一本教材都会提到它。

场景：你要去滑雪，一种选择是每次花1元租滑雪板，另一种选择是花B元买一副自己的滑雪板。你不知道自己会滑多少次（这个赛季会不会受伤？会不会出差？会不会突然失去兴趣？）。你的决策：每次去滑雪时，是租还是买？

问题设定：
- 租一次：1元
- 买一副：B元（买了以后无限次使用）
- 你不知道总滑行次数 T（对抗性对手会选择最坏的T）

一个自然策略（"失望边界"策略）：
- 先租 B-1 次
- 如果第 B 次还去滑，就买（因为再租就超过买的价格了）

竞争比分析：
- 如果 T ≤ B-1，你花了 T 元而最优花了 T 元 → 竞争比 = 1
- 如果 T ≥ B，你花了 (B-1) + B = 2B-1 元而最优花了 B 元 → 竞争比 ≈ 2
- 最大竞争比 = 2 - 1/B，在 B→∞ 时趋近于 2

这就是"2竞争比"的滑雪板租赁问题。而且可以证明：没有确定性在线算法能比2做得更好。

生活中的竞争比：下次你犹豫"是买断一个软件还是按月订阅"时，问自己一个问题——你打算用多久？如果你不确定，滑雪板租赁问题的结论是：先订阅到总价接近买断价，然后买断。这个策略的竞争比是2，意味着在最坏情况下你的花费不会超过最优策略的2倍。不是完美，但还不错。

在线分页——LRU与最优竞争比

在线分页（Paging）是操作系统中内存管理的核心问题。给定容量为k的缓存和不断到达的页面请求，当缓存满了而新页不在缓存中时，必须换出一个页。目标是最小化缺页次数。

经典算法：

  • LRU（Least Recently Used）：换出最久未使用的页——竞争比 = k
  • FIFO（First In First Out）：换出最早进入缓存的页——竞争比 = k
  • LIFO（Last In First Out）：换出最新进入缓存的页——没有常数竞争比（可以被对抗性输入打败）

令人惊讶的是：k竞争比对任何确定性在线分页算法都是最优的。不存在竞争比小于k的确定性在线分页算法。

但如果引入随机化，可以做得更好——随机化版本的标记算法（Marking Algorithm）可以达到 O(log k) 的竞争比。这显示了随机化在在线算法中的强大力量。

在线问题 确定性最优竞争比 随机化最优竞争比 经典算法
滑雪板租赁 2 e/(e-1) ≈ 1.58 失望边界策略
在线分页 k O(log k) LRU / 标记算法
在线二分图匹配 ∞（无界） 1 - 1/e ≈ 0.632 Karp-Vazirani-Vazirani
在线TSP 2 — 回头策略
关键观察：随机化在线算法的竞争比常常和确定性在线算法完全不同。对滑雪板租赁，随机化只能把2降到1.58——不算巨大的改进。但对在线匹配，确定性算法根本做不到常数竞争比，而随机化能做到0.632——这是一个质的飞跃。这告诉我们：在在线算法设计中，有时候"运气"不仅仅是运气，而是突破信息限制的必要手段。

在线二分图匹配——AdWords问题

这是在线算法在互联网广告领域最重要的问题。

场景：广告主提交了关键字列表和每日预算。用户每次搜索时，搜索引擎必须在几百毫秒内决定展示哪个广告——且每个广告主有预算上限，花完就不再出价。这个决策是在线的：你只知道当前搜索词，不知道今天后续还有哪些搜索词。

这个问题由Karp、Vazirani和Vazirani在1990年提出并解决。他们提出了一个简洁而优雅的随机化算法：

在线二分图匹配（KVV算法, 1990）

设定：二分图 G = (L, R, E)，左侧顶点 L 预先已知
      右侧顶点 R 逐一到达（一个搜索词就是一个右侧顶点）
      每个右侧顶点有边连接到左侧顶点
      每个左侧顶点最多能被匹配一次

算法（随机化）：
    对每个右侧顶点 r ∈ R（按到达顺序）：
        如果 r 没有邻接的可用左侧顶点，跳过
        否则，以均匀随机的方式选一个邻接的左侧顶点进行匹配

竞争比：这个简单算法达到 1 - 1/e ≈ 0.632 的竞争比
        并且这是最优的——没有随机化算法能做得更好

这个结果是令人惊讶的：如此简单的随机算法，在线性匹配这个看似困难的问题上，居然达到了最优的63.2%竞争比。这个结果也是2000年代AdWords竞价系统（Google Ads的前身）的理论基石之一。

随机优化

从在线算法自然延伸到随机优化。区别在于：在线算法的分析假设对抗性输入，随机优化则假设输入遵循某种概率分布。

多臂老虎机：探索-利用权衡

你走进一家陌生的赌场，面前有K台老虎机（Multi-Armed Bandits）。每台老虎机的赢钱概率不同，但未知。你手上有N个硬币。你的目标是在N次投币后，总赢钱数最大。

这就是探索-利用（Exploration-Exploitation）困境：你应该多尝试不同的老虎机来收集数据（探索），还是集中在目前表现最好的老虎机上（利用）？如果探索太多，你浪费了在"烂老虎机"上的投币机会；如果利用太多，你可能会错过一台更好的老虎机。

让我们形式化这个问题：

多臂老虎机（K-Armed Bandit）

设定：
- K台老虎机，第 i 台的赢钱期望为 μᵢ（未知）
- 每轮 t = 1, ..., T，选择一台老虎机 aₜ
- 获得随机收益 rₜ ~ distribution(μ_{aₜ})
- 目标：最大化总收益 Σ rₜ

后悔（Regret）：Regret(T) = T · maxᵢ μᵢ - 𝔼[Σ rₜ]
即"最优策略"和"我们的策略"之间的期望差距

UCB算法（上置信界）

Auer, Cesa-Bianchi和Fischer在2002年提出了UCB（Upper Confidence Bound）算法，是多臂老虎机问题中最经典的方法。

核心思想：乐观地面对不确定性。对每台老虎机，维护一个"表现的上置信界"——我们对它的收益越不确定，这个置信界就越高。每次选置信界最高的老虎机。

UCB1算法

初始化：每台老虎机至少玩一次
对每轮 t = 1, ..., T:
    对每台老虎机 i，计算：
        Qᵢ(t) = 平均收益（利用项）
        Uᵢ(t) = √(2·ln t / nᵢ)  （探索项，nᵢ是玩i的次数）
    选择 aₜ = argmaxᵢ [ Qᵢ(t) + Uᵢ(t) ]
    观察到收益 rₜ，更新 Qᵢ 和 nᵢ

核心公式：Qᵢ(t) + √(2·ln t / nᵢ)
          ↑ 利用      ↑ 探索（上置信界）

UCB的关键在于：随着时间t增长，置信界变紧凑（ln t 增长慢），探索自然减少；对玩得少的机器（nᵢ小），置信界宽（1/√nᵢ大），更容易被选中探索。这个机制实现了"自动衰减"的探索，无需手动调参。

一个真实的教训：我的一个朋友在一家外卖公司做菜品推荐。他们用UCB来探索新菜品——给用户推荐没试过的餐厅。结果上线一周，转化率大幅下降。排查发现：他们用的是非平稳（Non-Stationary）的收益——菜品的"受欢迎度"在午餐和晚餐时段完全不同。标准的UCB假设收益分布恒定，但外卖市场的偏好随时段、天气、节假日变化。他们后来改用Discounting UCB（给旧数据打折扣）来解决这个问题。这个教训：任何时候用多臂老虎机算法，先问自己——收益分布是平稳的吗？

Thompson Sampling

相比UCB的"乐观面对不确定性"，Thompson Sampling采取了一种贝叶斯方法：维护每台老虎机收益率的后验分布，每次从这些分布中抽样，选择抽样值最大的老虎机。

Thompson Sampling

初始化：每台老虎机的参数 αᵢ = 1, βᵢ = 1（Beta分布先验）
对每轮 t = 1, ..., T:
    对每台老虎机 i，从 Beta(αᵢ, βᵢ) 中采样 θᵢ
    选择 aₜ = argmaxᵢ θᵢ
    观察到收益 rₜ ∈ {0, 1}（得/失）
    更新 α_{aₜ} = α_{aₜ} + rₜ, β_{aₜ} = β_{aₜ} + (1 - rₜ)

Thompson Sampling虽然计算上比UCB复杂一点（需要采样），但它在实践中往往表现更好，特别是在收益分布非平稳或奖励稀疏的场景下。Google和Microsoft的在线广告系统都在使用Thompson Sampling的变体。

ε-greedy

最简单的策略：以概率 1-ε 选择当前平均收益最高的老虎机（利用），以概率 ε 随机选择一台老虎机（探索）。虽然简单，但通过适当衰减ε（例如 εₜ = 1/√t），可以达到对数后悔界。

随机对偶

在线问题的对偶分析框架

在线算法设计中最优雅的工具之一，是将线性规划对偶性引入在线环境。核心思想是：设计在线算法时，同时维护原问题的解和对偶问题的解，利用对偶的弱对偶性来证明竞争比。

在线Primal-Dual框架（针对最小化问题）

1. 将原问题写成一个线性规划（LP）
2. 写出它的对偶规划（Dual LP）
3. 设计在线算法，使得在每个时间步：
    - 更新原变量（做出在线决策）
    - 更新对偶变量（为未来的决策预留"空间"）
4. 保证：在每个时间步，原变量的增量 ≤ c · 对偶变量的增量
5. 由弱对偶性：原值 ≤ c · 对偶值 ≤ c · OPT
   因此竞争比 = c

这个框架的美妙之处在于：你不需要知道未来的输入，只需在当前步确保原增量和偶增量之间的比率。只要比率被控制住了，最终竞争比就自动保证了。

应用实例

在线Primal-Dual方法已经成功应用于多个经典问题：

  • 在线集合覆盖：O(log n)竞争比
  • 在线分页：k竞争比（与LRU一致的竞争比）
  • AdWords的在线分配：由Devanur和Hayes在2009年用P-D框架给出了(1-1/e)竞争比的新证明
  • 在线Steiner树：O(log n)竞争比
一个深刻的观点：在线Primal-Dual框架揭示了"实时定价"和"在线优化"之间深刻的联系。在很多场景下，对偶变量就是"价格"——当需求高时价格上涨（对偶变量增大），从而抑制过度消费，保护有限资源。这正是一个好的在线算法做的事：通过隐式地维护一个"影子价格"系统，保证在任何输入序列下都不会太离谱。这也许不是巧合——市场机制本质上就是一个大规模的在线优化系统。

现实应用

在线与随机优化不是纯理论——它们支撑着现代互联网经济的基础设施。

在线广告分配

Google Ads、Facebook Ads、程序化广告交易平台的背后都是在线分配算法。每天有数十亿次广告展示机会，每次机会必须在几十毫秒内做出决策。UCB、Thompson Sampling和在线P-D框架是这些系统的核心技术。一个千分点的效率提升，就意味着数十亿美元的收入增量。

外卖骑手实时调度

美团、饿了么、Uber Eats的平台每天要调度数百万骑手。订单是"在线到达"的——你只知道当前有哪些单，不知道5分钟后会有多少订单。同时骑手位置实时变化。这里的核心挑战是：接单（预留骑手）还是等待（可能有更"顺路"的单）？这个问题被建模为在线二分图匹配的变体，KVV的(1-1/e)匹配算法是理论基石。

云计算资源弹性分配

Amazon AWS的Auto Scaling、Google Cloud的自动扩缩容，本质上是在线资源分配问题。你不知道下一秒的流量会飙升还是暴跌——但必须现在决定要不要加虚拟机。分配多了浪费钱，分配少了丢请求。这里在线算法和随机优化（结合流量预测）一起工作，达到成本和服务质量之间的平衡。
