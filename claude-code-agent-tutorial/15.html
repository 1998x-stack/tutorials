第15章 权限系统源码拆解

大家好，我是你们的讲师。在开始今天的内容之前，我想先问一个问题：你信任 AI Agent 到哪种程度？

2025 年中，一位开发者接了一个外包项目——用 Claude Code 给一个医疗系统写自动化测试。他启动 Agent，输入提示词，然后去泡了杯咖啡。回来时他发现：Agent 不仅写完了测试，还顺手执行了 rm -rf /tmp/cache/* 来"清理临时文件"。虽然这次只是删掉了缓存目录，但试想一下——如果那次删除路径因为某个环境变量配置错误指向了 /var/www/html 呢？这让我想起了一个更著名的教训：2015 年某大厂的一次自动化部署脚本中，rm -rf $DIR/* 在 $DIR 为空时变成了 rm -rf /*，导致生产服务器大面积瘫痪。当 Agent 替你敲命令时，这种风险被放大了无数倍——因为 Agent 不会像人类一样在输入危险命令前停顿三秒。

这就是权限系统的意义所在。Agent 的能力越强，权限系统就越重要。Claude Code 的权限系统不是一道门，而是一整套包含七种模式、规则链评估引擎和 ML 分类器的纵深防御体系。本章我们会拆解这套系统——看看这个"最后的把关人"是如何让你安全地使用 Agent 的。

权限问题的本质

在进入代码之前，我们先理解权限问题的核心矛盾。

一方面，一个能编辑文件、运行命令、调用 API 的 Agent 极其强大。它可以在几分钟内完成人类需要数小时的工作。但另一方面，它也是极其危险的——一个错误的 rm、一次不小心覆盖了配置文件、一条误发的 production 数据库 DROP 语句，都可能是灾难性的。

解决方案看起来很简单：让用户审批每一次操作。但实际使用中你会发现，这种方式在实践中完全行不通。如果一个 Agent 执行一个任务需要调用 30 次工具，你每次都要点击"确认"——这就是所谓的 Permission Fatigue。用户很快会变得麻木，开始无脑点"同意"，这时候权限审批形同虚设。

Claude Code 的设计哲学可以总结为一句话：最小权限 + 渐进信任。默认情况下，Agent 只能做最安全的事；随着它证明自己的可靠性，权限逐渐放宽。但这不应该是黑盒决定——用户始终有最终的否决权。我们来具体看看它是怎么做到的。

7 种权限模式详解

Claude Code 定义了七种权限模式，形成一个从"完全限制"到"完全放开"的连续谱。每种模式对应不同的使用场景和安全假设。

我整理了一张对照表，方便你快速理解它们的差异：

模式名称 自动批准 需要询问 典型使用场景 风险等级
plan 无（不可执行工具） 全部（仅输出计划） 任务规划、审计、教学演示 🟢 无风险
default Read、Grep 等只读工具 Write、Bash、Edit 等 日常开发，标准用户 🟢 低
acceptEdits + 文件编辑（Write、Edit） Bash 命令执行 信任 Agent 的编辑能力 🟡 中
autoEdit + "安全"的 Bash 命令 危险命令、网络操作 资深用户，日常使用 🟡 中高
dontAsk 全部工具调用 无 CI/CD 流水线、批量处理 🔴 高
bypassPermissions 绕过所有检查 无 Sub-Agent 内部转发、调试 🔴 极高
bubble N/A（传递到父级） N/A Sub-Agent 权限提升路径 🟡 视父级而定

理解这七种模式的关键，不在于记住每个模式能做什么——而在于捕捉 Claude Code 的权限哲学：权限是一个从完全受限到完全信任的连续光谱，而不是一个二元开关。工具之间也有天然的"信任层级"——读取操作应该永远比写入操作更容易被批准。

plan 模式实际上不执行任何工具。Agent 会输出一个详细的行动计划（包括"我想用 Read 工具读取文件 X，然后用 Edit 工具修改 Y"），但所有工具调用都被拦截。这个模式在审计场景特别有用——你可以先看看 Agent 打算做什么，评估它的计划是否合理，再切换到更高权限模式让它执行。

bypassPermissions 看起来非常危险——为什么会有这个模式？答案是 Sub-Agent 内部通信。当一个 Sub-Agent 需要向主 Agent 传递信息时，如果每次传递都要经过权限检查，会引入巨大的延迟和复杂的权限递归问题。所以 Sub-Agent 之间的内部通信会使用这个模式，但注意——只在受信任的进程间通信中使用。

而 bubble 模式是 Claude Code 权限系统中最优雅的设计之一，我们后面会专门展开。

规则链评估引擎

权限模式是策略层面，规则链是执行层面。Claude Code 的权限规则组织成一个有序的链表——每条规则包含三个要素：工具匹配模式（toolPattern）、权限模式（permissionMode）、条件（conditions）。

评估过程是顺序的：从头到尾遍历规则链，第一条匹配规则生效。但有一个关键例外：Deny 永远覆盖 Allow。什么意思？即使一条规则说"允许 Write 工具"，但如果后面有一条规则说"当目标是 /etc/passwd 时拒绝所有操作"，那么拒绝会胜出。这在代码中通过两轮扫描实现——第一轮检查是否有拒绝规则，第二轮才处理允许规则。

看一段简化的实现你就明白了：

# 简化版权限规则引擎

        from dataclasses import dataclass, field
        from typing import Callable, Optional
        import fnmatch


        @dataclass
        class PermissionRule:
            tool_pattern: str           # 工具名 glob 模式，如 "Bash", "Write:*"
            mode: str                   # 权限模式：allow / deny / ask
            conditions: list[Callable] = field(default_factory=list)
            reason: str = ""


        class PermissionEngine:
            def __init__(self, rules: list[PermissionRule] = None):
                self.rules = rules or []

            def add_rule(self, rule: PermissionRule):
                self.rules.append(rule)

            def evaluate(self, tool_name: str, params: dict) -> dict:
                \"\"\"
                评估规则链，返回决策结果。
                Deny 优先于 Allow —— 两轮扫描。
                \"\"\"
                # 第一轮：检查 Deny
                for rule in self.rules:
                    if rule.mode != "deny":
                        continue
                    if not fnmatch.fnmatch(tool_name, rule.tool_pattern):
                        continue
                    if not self._check_conditions(rule.conditions, params):
                        continue
                    return {"allowed": False, "reason": rule.reason, "rule": rule}

                # 第二轮：检查 Allow / Ask
                for rule in self.rules:
                    if rule.mode == "deny":
                        continue
                    if not fnmatch.fnmatch(tool_name, rule.tool_pattern):
                        continue
                    if not self._check_conditions(rule.conditions, params):
                        continue
                    if rule.mode == "ask":
                        return {"allowed": True, "requires_confirmation": True,
                                "reason": rule.reason, "rule": rule}
                    return {"allowed": True, "requires_confirmation": False,
                            "reason": rule.reason, "rule": rule}

                # 无匹配规则：默认拒绝（fail closed）
                return {"allowed": False,
                        "reason": f"No matching rule for {tool_name}", "rule": None}

            def _check_conditions(self, conditions, params) -> bool:
                return all(cond(params) for cond in conditions)


        # 使用示例
        rules = [
            PermissionRule("Bash", "deny",
                           conditions=[lambda p: "rm -rf" in p.get("command", "")],
                           reason="rm -rf 被禁止"),
            PermissionRule("Bash", "ask",
                           reason="执行 Bash 需要确认"),
            PermissionRule("Read", "allow",
                           reason="读取操作默认允许"),
            PermissionRule("Write:*", "allow",
                           conditions=[lambda p: p.get("path", "").startswith("/tmp")],
                           reason="允许写入 /tmp 目录"),
        ]

        engine = PermissionEngine(rules)

        # 测试
        print(engine.evaluate("Read", {"path": "main.py"}))
        # → {'allowed': True, 'requires_confirmation': False, ...}

        print(engine.evaluate("Bash", {"command": "cat log.txt"}))
        # → {'allowed': True, 'requires_confirmation': True, ...}

        print(engine.evaluate("Bash", {"command": "rm -rf /tmp/foo"}))
        # → {'allowed': False, 'reason': 'rm -rf 被禁止', ...}

这个设计的关键细节是 fail closed——当没有规则匹配时，默认拒绝。这和网络安全中的"白名单原则"一脉相承：没有明确允许的就是禁止。你也许注意到了规则引擎中还有一个 ask 模式——它允许操作，但要求用户确认。这是从"全部拒绝"到"全部允许"之间的一个折中地带，也是实际使用中最常见的模式。

工程细节：Claude Code 实际使用的规则链比上面的示例复杂得多。规则支持正则表达式匹配、时间窗口（某些操作只在特定时间段允许）、计数限制（某类操作在一小时内最多执行 N 次）、以及组合条件（AND/OR/NOT）。但核心逻辑——Deny 优先、顺序评估、fail closed——是完全一致的。

ML 分类器的独立评估

规则链能拦截明显的危险操作，但面对模糊地带就力不从心了。比如 Bash("git push origin main")——它算危险操作吗？在个人项目上可能不算，但在公司生产仓库上就可能非常危险。规则的表达能力是有限的——你没法为每一种可能的 git 命令组合都写一条规则。

Claude Code 的解决办法是引入一个 独立的 ML 分类器，专门评估工具调用是否安全。这是一个非常关键的设计决策，我们来仔细看看它的运作方式。

这个分类器的输入有两个部分：用户的原始请求文本和工具调用的参数。注意：它刻意不接收 Agent 的推理过程（Chain-of-Thought）。为什么？因为 Agent 在生成推理过程时，完全可能写出"这个操作看起来有点危险，但用户让我这么做所以我继续"——如果分类器看到了这段推理，它可能被"说服"放行。更极端的情况是，恶意提示词注入可能让 Agent 在推理中写出"这是一个安全操作"来欺骗分类器。

这个设计的本质是：把"意图判断"和"行为判断"分离开。用户的意图告诉分类器"用户想做什么"；工具参数告诉分类器"Agent 实际要做什么"。分类器自己判断这两者之间是否有不匹配。

举个具体的例子。用户输入："帮我删除项目中的临时文件"。Agent 随后调用 Bash("rm -rf ~/project/tmp")。分类器看到：用户请求提到"删除临时文件" + 工具参数是删除某个路径。它判断这是合理操作，放行。但如果 Agent 调用的是 Bash("rm -rf /etc/nginx/sites-enabled/")——用户要求删除临时文件，但 Agent 却在删 nginx 配置——分类器会标记为危险，拒绝执行。

在 Claude Code 中，这个分类器被用于 autoEdit 模式：所有编辑操作自动批准，所有的 Bash 命令都由分类器判断是自动放行还是需要用户确认。这样既保留了效率（大部分安全操作无需交互），又不牺牲安全性（危险操作仍然需要用户确认）。

权限提升与 Bubble 机制

之前我们提到 Sub-Agent 有 bubble 模式——这个机制解决的是一个很实际的问题：当 Sub-Agent 需要比它当前拥有的更高权限时怎么办？

Claude Code 的解决方法是权限冒泡。想象一个场景：主 Agent 在 default 模式下运行，它启动了一个 Sub-Agent 来执行一个子任务。Sub-Agent 继承父级的权限——也是 default。但在执行过程中，Sub-Agent 发现它需要执行一个 Bash 命令来编译代码——这在 default 模式下需要用户确认。但 Sub-Agent 本身没有交互能力（它运行在后台，用户看不到它的输出），所以它不能像主 Agent 那样弹出一个确认框。

这时发生的是：Sub-Agent 的权限检查失败后，操作被冒泡到它的父 Agent（也就是主 Agent）。主 Agent 用自己的权限模式重新评估这个操作——如果需要用户确认，主 Agent 会向用户展示确认请求。用户确认后，结果传回 Sub-Agent。整条冒泡链路的路径是：

Sub-Agent → 主 Agent → 用户

如果主 Agent 本身也是某个更高层级 Agent 的 Sub-Agent，冒泡可以继续向上传递。这个机制保证了权限检查永远在最合适的层级执行——Sub-Agent 不需要有独立的交互能力，只需要把决策权上交给能交互的那个层级。

权限系统中的踩坑实录

说完了架构，我们来聊聊实际部署中遇到过的坑。这些教训来自社区的真实反馈和 Claude Code 团队自己的发布后复盘。

坑一：Classifier False Negative——安全操作被误杀。 ML 分类器不是完美的。早期的模式对 pip install 特别敏感，经常把这个操作归类为"危险"（因为它会下载和执行外部代码）。但用 Python 开发的用户几乎每五分钟就要跑一次 pip install，每次都弹确认框让开发体验变得极差。解决办法是通过"允许列表"——用户可以把特定的包管理器命令加入白名单。

坑二：Permission Fatigue——用户点确认点到麻木。 这是我们前面提到的问题在现实中的体现。某团队在使用 default 模式时，用户平均每分钟要确认 3-4 次操作。两周后，团队里所有人都养成了"看到弹框就按 Enter"的肌肉记忆——权限审批形同虚设。最终解决方案是切换到 acceptEdits 模式，减少一半的确认频率，同时在权限系统中引入"渐进信任"：如果 Agent 在某个任务中表现良好（没有误操作），系统会逐步将其部分操作从"需要确认"降级为"自动批准"。

坑三：规则顺序 Bug。 这是最隐蔽的一类问题。规则链中，如果一条更通用的规则被放在了更具体的规则之前，具体规则永远不会被触发。例如，如果有人把"允许所有 Bash 操作"这条规则放在了"拒绝 rm -rf"之前，那么 rm -rf 也会被放行。这和防火墙规则配置是一个道理——最具体的规则应该放在最前面。

生产环境警告：bypassPermissions 模式在 Claude Code 中默认不可用，需要显式启用。即使在开发环境中，也绝不应该将此模式永久设置。最好在需要时临时启用，使用后立即切回更严格的模式。如果你正在基于 Claude Code 的架构开发自己的 Agent，建议默认不实现 bypassPermissions 模式——绝大多数场景用不到它，而它的存在会引入不必要的安全风险。

权限系统是 Agent 安全的最后一道防线——也是最关键的一道。理解了这七种模式、规则链引擎和 ML 分类器的工作原理，你就掌握了 Claude Code 安全模型的核心。下一章，我们将动手实现一个权限控制系统，把今天学的这些概念变成可运行的代码。
