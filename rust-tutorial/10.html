开篇：那个用了unwrap的生产事故

我还清楚地记得那个周三下午。我们部署了一个新版本的API网关，上线15分钟后，PagerDuty开始疯狂报警——整个服务的错误率从0.1%飙升到了35%。追溯日志发现，罪魁祸首就是一行代码：.unwrap()。

一个第三方API偶尔返回空的响应体。我们内部代码在解析响应时潇洒地写了个response.body.parse::<Config>().unwrap()——结果空响应触发了panic!，worker线程崩溃，连接池迅速耗尽，连锁反应几乎瘫痪了整个服务。

从那天起，我的团队立了一个规矩：除了测试代码和原型验证，任何unwrap都必须经过代码评审并给出明确的理由。

这就是Rust错误处理的精髓——它不让你"忘记"处理错误，而是用类型系统逼你做选择。这一章我们就来系统学习Rust的错误处理机制。

panic! 与不可恢复错误

当程序遇到无法继续执行的状况时，可以调用panic!宏。它会打印错误消息、展开栈（unwinding）、清理资源，然后退出。

fn divide(a: i32, b: i32) -> i32 {
    if b == 0 {
        panic!("除数不能为0！传入的值: {b}");
    }
    a / b
}

fn main() {
    println!("{}", divide(10, 0));
}

运行上述代码会输出：

thread 'main' panicked at src/main.rs:3:9:
除数不能为0！传入的值: 0
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

什么时候应该panic？

在Rust中，panic!意味着"无法恢复的错误"。适合使用panic!的场景：

  • 代码逻辑错误：索引越界、除以0、非法状态——这些属于"编程bug"，应该修复代码而非处理错误
  • 原型阶段：快速验证想法，用unwrap或expect临时处理
  • 测试代码：测试中unwrap失败就是测试失败，合理
  • 配置错误：程序启动时配置加载失败，没必要继续运行
栈展开 vs 中止：默认情况下Rust使用栈展开（unwinding），会逐层释放栈上的资源然后退出。如果你的目标平台对二进制大小敏感（比如嵌入式），可以在Cargo.toml中设置panic = "abort"来直接终止进程，但这会跳过Drop代码的执行。

Result<T, E>：可恢复错误

大部分错误是可恢复的。文件不存在？可以创建。网络超时？可以重试。解析失败？跳过这一行继续。Rust使用Result<T, E>枚举来表示可能失败的操作：

enum Result<T, E> {
    Ok(T),
    Err(E),
}

Result的基本使用

use std::fs::File;

fn main() {
    let result = File::open("config.txt");

    let file = match result {
        Ok(f) => f,
        Err(e) => {
            eprintln!("打开文件失败: {e}");
            return;
        }
    };
}

便捷方法

方法 Ok时的行为 Err时的行为
unwrap() 返回内部值 panic!
expect(msg) 返回内部值 panic!并附带自定义消息
unwrap_or(default) 返回内部值 返回默认值
unwrap_or_else(fn) 返回内部值 调用闭包生成默认值
is_ok() true false
is_err() false true
ok() Some(T) None
err() None Some(E)
unwrap的诱惑：unwrap()打字快、省事——这是它最大的诱惑，也是它最大的危险。一个简单的原则：生产代码中，如果你觉得"这里绝对不会出错"但用了unwrap，请在旁边注释说明理由。更好的做法是用expect("理由")。

?运算符：错误传播的优雅写法

写match处理每个Result虽然安全，但代码会变得非常冗长。Rust提供了?运算符来简化错误传播：

use std::fs::File;
use std::io::{self, Read};

fn read_username() -> Result<String, io::Error> {
    let mut file = File::open("username.txt")?;  // 出错时提前返回 Err
    let mut username = String::new();
    file.read_to_string(&mut username)?;
    Ok(username.trim().to_string())
}

?运算符的作用：如果Result是Ok，提取其中的值给左值；如果是Err，将错误转换后返回给调用者。

错误自动转换依赖From<E> trait。上面例子中File::open返回io::Error，而函数返回值也是io::Error，所以直接匹配。但不同类型呢？

use std::num::ParseIntError;

fn parse_number(s: &str) -> Result<i32, ParseIntError> {
    s.parse()?  // 自动将 ParseIntError 向上传播
}
?在main中的使用：从Rust 2021 Edition开始，main函数可以返回Result：fn main() -> Result<(), Box<dyn std::error::Error>>。这样你就能在main中使用?了。程序出错时，会以非零退出码结束并打印错误信息。

自定义错误类型

随着项目变大，你会发现需要自己的错误类型。Rust中自定义错误的最佳实践是定义一个枚举，为每种错误原因创建一个变体：

use std::fmt;

#[derive(Debug)]
pub enum MyError {
    NotFound(String),
    PermissionDenied,
    NetworkTimeout { service: String, retries: u32 },
    IoError(std::io::Error),  // 包装其他错误
}

impl fmt::Display for MyError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            MyError::NotFound(id) => write!(f, "资源未找到: {id}"),
            MyError::PermissionDenied => write!(f, "权限不足"),
            MyError::NetworkTimeout { service, retries } => {
                write!(f, "服务 {service} 超时（已重试{retries}次）")
            }
            MyError::IoError(e) => write!(f, "IO错误: {e}"),
        }
    }
}

impl std::error::Error for MyError {}  // 可选方法覆盖

手动实现Display和Error很繁琐——在实际项目中，大家几乎都用thiserror crate来简化：

// Cargo.toml: thiserror = "2.0"
use thiserror::Error;

#[derive(Error, Debug)]
pub enum MyError {
    #[error("资源未找到: {0}")]
    NotFound(String),

    #[error("权限不足")]
    PermissionDenied,

    #[error("服务 {service} 超时（已重试{retries}次）")]
    NetworkTimeout { service: String, retries: u32 },

    #[error(transparent)]
    IoError(#[from] std::io::Error),
}

#[from]属性会自动生成From<io::Error> for MyError的实现，这样在?运算符中就能自动转换。

anyhow与eyre：应用层错误处理

在应用代码（而非库代码）中，你往往不关心具体的错误类型，只关心"出错了，告诉用户或者记录日志"。这时候anyhow crate就派上用场了：

// Cargo.toml: anyhow = "1.0"
use anyhow::{Context, Result};

fn read_config() -> Result<Config> {
    let content = std::fs::read_to_string("config.toml")
        .with_context(|| "无法读取配置文件")?;
    let config: Config = toml::from_str(&content)
        .with_context(|| "配置文件格式不正确")?;
    Ok(config)
}

fn main() -> Result<()> {
    let config = read_config()?;
    println!("{config:?}");
    Ok(())
}

anyhow::Result<T>是Result<T, anyhow::Error>的别名，它像一个"兜底"的错误类型——可以自动转换任何实现了std::error::Error的错误类型。而Context trait的with_context方法让你可以为错误添加上下文信息。

库 vs 应用：一个重要的设计原则——库代码应该定义自己的细粒度错误类型（配合thiserror），让调用者知道具体出了什么问题；应用代码使用anyhow，只关心错误的传播和展示。这类似于Go语言中"库返回具体类型，main/main-like代码使用error"的哲学。

eyre是anyhow的替代者，提供了更灵活的错误报告定制能力，但在简单场景下两者功能几乎等价。对于大多数项目，直接从anyhow入手即可。

本章小结

Rust的错误处理体系可以用一句话概括：用类型把错误显式化，不让任何失败无声无息地溜走。panic!处理不可恢复的错误，Result处理可恢复的错误，?简化传播路径，thiserror和anyhow让生产代码更加优雅。
