大家好，我是你们的讲师。上一章我们学习了信号机制——一种进程间的异步通信方式。但无论是进程创建、信号处理、还是文件读写，所有内核操作都依赖于一个更基础的资源——内存。

内存管理是操作系统的"地基"。没有高效、安全的内存管理，任何其他的内核机制都无法可靠运行。从本章开始，我们将用两章的篇幅，深入Linux 0.11的内存管理源码。

这一章我们聚焦在物理内存上：物理内存如何布局？如何探测？如何分配和回收？下一章我们再聚焦分页机制：虚拟地址如何转换为物理地址？缺页异常和写时复制是如何实现的？

引言——内存管理的核心问题

内存管理需要回答以下几个核心问题：

  1. 物理内存有多大？——启动时探测可用内存
  2. 物理内存如何分区？——内核用多少？用户用多少？缓冲区占多少？
  3. 物理页如何分配和回收？——谁来管理空闲页？分配策略是什么？
  4. 虚拟地址如何映射到物理地址？——分页机制是如何工作的？

前三个问题在本章回答，第四个问题留到下一章。

历史背景：1991年，一台典型的PC有4MB内存（Linus的386 DX/33配了4MB，花了他3,500美元）。Linux 0.11的设计就是针对4MB-16MB内存范围的。对比今天动辄16GB甚至64GB的服务器，4MB显得如此渺小——但令人惊讶的是，0.11的内存管理代码（mm/目录）只有不到1,000行，却已经实现了完整的分页、保护、共享、写时复制等现代内存管理特性。

物理内存布局

Linux 0.11对物理内存进行了清晰的分区。系统启动后，物理内存被划分为以下几个区域：

Linux 0.11 物理内存布局（假设总内存4MB）：

起始地址        结束地址        大小        用途
──────────────────────────────────────────────────
0x000000    0x008FFF    约36KB      内核代码和数据段
0x009000    0x00AFFF    8KB         内核栈和临时数据
0x00B000    0x02FFFF    约148KB     head.s/GDT/IDT/页目录/页表
──────────────────────────────────────────────────
0x030000    0x03FFFF    64KB        磁盘缓冲区（1KB缓冲块 * 64块）
──────────────────────────────────────────────────
0x040000    0x04FFFF    64KB        虚拟盘（可选）
──────────────────────────────────────────────────
0x050000    0x3FFFFF    约3.7MB     主内存区——可供用户进程使用
──────────────────────────────────────────────────

更直观的内存布局图：

低地址
  │
  ▼
┌─────────────────┐ 0x000000
│  内核代码+数据  │ 约36KB (bootsect + setup + head + kernel text/data)
├─────────────────┤ 0x009000
│  内核栈等临时   │ 8KB
├─────────────────┤ 0x00B000
│  页目录+页表    │ 占用连续内存 (0.11预分配所有页表空间)
├─────────────────┤ 0x030000 (~192KB)
│  缓冲区         │ 64KB (64个1KB缓冲块，用于块设备I/O)
├─────────────────┤ 0x040000 (~256KB)
│  虚拟盘(可选)   │ 64KB (编译时可选的RAM disk)
├─────────────────┤ 0x050000 (~320KB)
│                │
│  主内存区      │ 约3.7MB
│  (用户进程)    │ 这里是get_free_page()分配内存的来源
│                │
├─────────────────┤ 0x3FFFFF (~4MB)
高地址

这个布局中有几个需要注意的设计决策：

  • 内核占据了最低的物理地址：这是x86传统——实模式下的中断向量表在0x00000处，保护模式下的GDT/IDT也被安排在这里
  • 缓冲区被放在内核和主内存之间：缓冲区的用途是缓存块设备（如软盘、硬盘）的数据，它实际上是文件系统性能的关键
  • 主内存区从0x050000开始：所有用户进程的代码、数据、堆栈都从主内存区分配

这个分区的物理起点和终点在mm/memory.c和kernel/sched.c中定义：

/* 内存边界定义 */
#define LOW_MEM           0x040000    /* 主内存起始地址：256KB */
#define PAGING_MEMORY     (0x1000000) /* 最多支持16MB物理内存 */
#define HIGH_MEMORY       (end_mem)   /* 实际物理内存上限 */

/* 主内存区分页的起始位置 */
unsigned long HIGH_MEMORY = 0;       /* 实际物理内存上限 */
unsigned long LOW_MEM = 0x040000;    /* 主内存起点256KB */

注意LOW_MEM被设为0x040000（256KB），这意味着256KB以下的物理内存（内核代码、页表、缓冲区等）不能被当作"主内存"进行页框分配。

物理内存探测

物理内存有多大？在启动时，这是一个必须回答的问题。在实模式下，setup.S通过BIOS中断INT 0x15来探测内存大小：

! boot/setup.S — 通过BIOS探测扩展内存大小

! INT 0x15, AH=0x88 — 获取扩展内存大小（从1MB开始）
! 返回值：ax = 扩展内存大小（以KB为单位）

memory_probe:
    movb $0x88, %ah     ! 功能号：读取扩展内存大小
    int  $0x15          ! 调用BIOS中断
    movw %ax, %ds:2     ! 将结果存储在0x90002位置

    ! 如果ax=0，说明BIOS不支持此功能，使用默认值
    ! 默认值在Makefile中配置（如4MB、8MB等）

这个探测到的内存大小被存储在物理地址0x90002处——这个地址是setup.S约定好的"参数传递区"，后续的head.S和main.c会从这里读取参数。

在main.c中，内核读取这个值并计算可用内存的上限：

/* init/main.c — 读取内存大小 */

void main(void)
{
    ...
    /* 从0x90002读取扩展内存大小（KB） */
    memory_end = (*(unsigned short *)0x90002) * 1024 + 1024*1024;
    /* 注意：BIOS只报告1MB以上的扩展内存 */
    /* 所以总内存 = 1MB + 扩展内存 */

    /* 限制最大16MB（0.11的分页机制最多支持16MB） */
    if (memory_end > 16*1024*1024)
        memory_end = 16*1024*1024;

    /* 主内存区从1MB开始到实际内存结束 */
    main_memory_start = memory_start; /* 通常是0x040000或更大 */
    main_memory_end = memory_end;

    /* 初始化内存管理 */
    mem_init(main_memory_start, main_memory_end);
    ...
}
为什么最多16MB？Linux 0.11的分页机制使用了两级页表（10位页目录+10位页表），一个页目录管理4MB（1024个页表项 * 4KB），而0.11只使用了4个页目录项（前4个），所以只能管理16MB。如果你看现代x86-64系统，使用的是4级页表，可以管理高达256TB的虚拟地址空间——这就是差距！

mem_map数组

物理内存以页为单位管理（每页4KB）。Linux 0.11用mem_map[]数组来跟踪每个物理页的使用状态：

/* mm/memory.c — 物理页状态数组 */

#define PAGE_SIZE 4096        /* 每页4KB */

/* mem_map数组：每个字节代表一个物理页的使用计数 */
/* 数组下标 = 物理页号（物理地址/PAGE_SIZE） */
/* 值 = 0 表示空闲 */
/* 值 > 0 表示被引用（引用计数） */
/* 值 = 100 表示被内核保留（不可分配） */
unsigned char mem_map [PAGING_PAGES] = {0,};

#define PAGING_PAGES  (16*1024*1024 / PAGE_SIZE)  /* 16MB共4096页 */

内存初始化函数mem_init()扫描物理内存，将不可用的页标记为USED：

/* mm/memory.c — 内存初始化 */

#define USED 100        /* 保留页标记 */

void mem_init(long start_mem, long end_mem)
{
    int i;

    /* 1. 将所有物理页标记为空闲 */
    for (i = 0; i < PAGING_PAGES; i++)
        mem_map[i] = 0;

    /* 2. 对主内存区之外的页（内核占用的页）标记为USED */
    /*    start_mem = 主内存区起始地址（如0x050000） */
    /*    end_mem   = 实际物理内存上限 */

    /* 主内存区之前的物理页——都是内核/缓冲区占用的，不能分配 */
    i = start_mem / PAGE_SIZE;          /* 计算主内存区起始页号 */
    while (i-- > 0)
        mem_map[i] = USED;              /* 内核占用页标记为100 */

    /* 主内存区之后的物理页——超出实际内存范围的，标记为USED */
    /* 如果实际内存只有4MB，超出4MB的部分应该标记为USED */
    i = end_mem / PAGE_SIZE;            /* 计算实际内存末端页号 */
    while (i < PAGING_PAGES)
        mem_map[i++] = USED;            /* 超出范围的页标记为USED */

    /* 初始化完成后：
     * mem_map[0..kernel_end_page]    = USED  (内核占用)
     * mem_map[main_mem_start_page..end_page] = 0     (可用)
     * mem_map[end_page..4095]         = USED  (超出范围)
     */
}

你可以把mem_map想象成一个宿舍的管理员登记表：每个房间（物理页）都有状态——空闲（0）、有人住（引用计数>0）、或者已废弃不能住（USED=100）。每次分配物理页，管理员就找一个空闲房间，把状态改为1（有人住）；每次释放，引用计数减1，减到0就恢复空闲。

get_free_page() / free_page()

物理页的分配和释放是内存管理最基本、最频繁的操作。Linux 0.11用get_free_page()和free_page()来管理：

/* mm/memory.c — 分配一页物理内存 */

unsigned long get_free_page(void)
{
    unsigned long page;

    /* 1. 遍历mem_map数组，找第一个空闲页 */
    for (page = 0; page < PAGING_PAGES; page++) {
        if (mem_map[page] == 0) {      /* 找到一个空闲页 */
            mem_map[page]++;            /* 引用计数设为1 */
            break;
        }
    }

    /* 如果没找到空闲页，返回0 */
    if (page >= PAGING_PAGES)
        return 0;

    /* 2. 将物理页中的内容清零 */
    /*    这是一个关键的安全措施——防止新进程读到旧数据！ */
    /*    物理地址 = 页号 * 4KB */
    page = page * PAGE_SIZE;           /* 转为物理地址 */

    /* 用rep stosl清零（一次4字节，共1024次） */
    clear_page(page);

    return page;                        /* 返回物理地址 */
}

/* mm/memory.c — 释放一页物理内存 */

void free_page(unsigned long addr)
{
    unsigned long page;

    /* 1. 检查地址合法性 */
    if (addr < LOW_MEM)             /* 不能释放内核占用的内存 */
        return;
    if (addr >= HIGH_MEMORY)        /* 不能释放超出实际范围的内存 */
        return;

    /* 2. 计算页号 */
    page = addr / PAGE_SIZE;

    /* 3. 减少引用计数 */
    if (mem_map[page]-- <= 0) {     /* 如果已经是0，说明有Bug */
        mem_map[page] = 0;          /* 恢复为0 */
        panic("trying to free free page");  /* 内核Panic */
    }

    /* 注意：当mem_map[page]减到0时，这一页才真正成为"空闲" */
    /* 之后get_free_page()会找到它并重新分配 */
}

这两个函数构成了物理内存管理的基础。注意get_free_page()中清零操作的重要性——如果没有清零，新进程可能看到之前使用该页面的进程残留的数据，这是一个严重的安全漏洞。

函数作用复杂度备注
get_free_page()分配1页空闲物理内存O(n)扫描mem_map返回物理地址；清零后返回
free_page(addr)释放1页物理内存O(1)递减引用计数；减到0才真正释放
free_page_tables()释放进程的整个页目录/页表O(n)进程exit时调用，递归释放所有页面
put_page(addr)将物理页映射到进程的线性地址O(n)缺页处理时调用
注意碎片问题：get_free_page()从mem_map的头部开始遍历，这实际上是一种"地址优先"的分配策略。长时间运行后，低地址的物理页可能被频繁分配和释放，导致"外部碎片"——虽然物理页是4KB对齐的，不存在内部碎片，但频繁的分配和释放会导致TLB效率下降。现代内核使用更复杂的链表或树结构来管理空闲页，以降低碎片化和提高分配效率。

线性地址空间布局

除了物理内存管理，Linux 0.11还建立了线性地址空间（Linear Address Space）的布局。每个进程看到的是连续的线性地址，通过页表映射到物理地址。

Linux 0.11 线性地址空间布局（每个进程）：

┌─────────────────┐ 0x00000000 (0)
│  代码段 (Code)   │ 由 exec 加载，通常从 0x00000000 开始
│                  │ 长度 = a_text (由a.out头指定)
├─────────────────┤
│  数据段 (Data)   │ 紧跟代码段之后
│                  │ 长度 = a_data + a_bss
├─────────────────┤
│  堆 (Heap)       │ 通过 brk() 系统调用调整
│                  │ 从数据段结束到栈底
├─────────────────┤
│                  │
│    未映射区域    │
│                  │
├─────────────────┤
│  栈 (Stack)      │ 从 0x4000000 (64MB - 1页) 向下增长
│  初始ESP         │ 参数和环境变量在此
└─────────────────┘ 0x4000000 (64MB)

进程0（CPU空闲进程）的特殊布局：
│  共享内核页表    │ 前640KB映射到物理地址0-640KB
│  对应phys 0-640KB│ 这样可以访问内核代码和数据

Linux 0.11给每个普通进程分配了64MB的线性地址空间。虽然物理内存只有4MB，但线性地址空间允许进程认为自己有连续的大地址空间。只有进程实际访问的页面才会被映射到物理内存，这就是"虚拟内存"的精髓。

进程0（即"idle进程"）比较特殊——它的线性地址空间前640KB直接映射到物理地址0-640KB，这样它可以直接访问内核代码和缓冲区。严格来说，进程0不是一个普通进程，它是内核初始化完成后用于"空闲等待"的实体。

/* init/main.c — 初始化进程0 */

void main(void)
{
    /* ... */

    /* 设置每个进程的线性地址空间范围 */
    /* 0.11使用分页，每个进程最多64MB线性地址空间 */
    /* 但内核在最高端共享一个"内核页面目录" */

    /* 设置进程0的LDT（代码段和数据段描述符） */
    /* 代码段基址 = 0x00000000，段限长 = 64MB */
    /* 数据段基址 = 0x00000000，段限长 = 64MB */

    /* 设置进程0的TSS和LDT */
    set_tss_desc(0, &init_task.tss);
    set_ldt_desc(0, &init_task.ldt);

    /* ... 然后切换到进程0执行 ... */
    __asm__("movl $0x17, %eax\n\t"  /* 加载用户数据段选择子 */
            "mov %ax, %fs\n\t"      /* fs指向用户数据段 */
            /* 通过iret切换到进程0 */
            "pushl $0x17\n\t"       /* 用户栈段 */
            "pushl $0\n\t"          /* 用户栈指针 */
            "pushfl\n\t"            /* EFLAGS */
            "pushl $0x0f\n\t"       /* 用户代码段 */
            "pushl $1f\n\t"         /* 返回地址 */
            "iret\n"
            "1:\t"
            "movl $0x17, %eax\n\t"
            "mov %ax, %ds\n\t"
            "mov %ax, %es\n\t"
            /* 现在进程0的用户态开始执行 */
            /* 它会调用fork()创建进程1（init进程） */
            );

这告诉我们一个有趣的事实：在Linux 0.11中，内核本身并不是一个"进程"，而是所有进程共享的"特权层代码"。进程0（idle进程）是最底层的空闲进程，进程1（init进程）是所有用户进程的祖先。

关键认知：在Linux中，内核态代码是"附着"在进程上的——当一个进程通过系统调用进入内核态，它并没有切换到"内核进程"，而是当前进程的特权级从ring 3提升到了ring 0。这就是为什么内核态栈是每个进程内核栈（在task_struct中），而不是全局共享的。这个设计让Linux的进程模型既统一又高效。

理解了物理内存的管理后，下一个关键问题是：线性地址如何映射到物理地址？缺页异常如何处理？写时复制如何实现？这些问题的答案都藏在分页机制中。下一章，我们将进入内存管理（下），深入学习关于分页、缺页异常和写时复制的全部细节。
