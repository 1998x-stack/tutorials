第25章 性能调优与生产踩坑

大家好，我是你们的讲师。前24章我们一直在"读代码"，这一章我们要做一件更有意思的事——把代码推到生产环境里跑起来，然后看它怎么翻车，再把它救回来。

如果你以为把一个编译好的 Memcached 扔到服务器上就能跑出高性能，那就太天真了。参数怎么配、内存怎么分、连接数怎么管——每一步都可能让你从"高性能缓存"跌入"高延迟灾难"。这一章我会带你走一遍 Memcached 生产环境调优的全流程，从参数配置到监控指标，从热 key 治理到内核参数调优，最后送上几个我用"血泪"换来的踩坑故事。

核心参数调优实战

Memcached 的参数不多，但每一个都对性能有显著影响。我们最常用的四个参数是 -m、-c、-t 和 -f：

参数 默认值 推荐范围 影响
-m 64 MB 物理内存的 60-80% 可用内存越大，缓存命中率越高
-c 1024 取决于 QPS，通常 10000-50000 限制最大并发连接数
-t 4 CPU 核数 - 2（或等于核数） Worker 线程数，直接影响吞吐
-f 1.25 1.10 - 1.50 Slab class 的 growth factor，决定内存利用率
-n 48 48 - 128 最小分配单元，影响小对象的内存效率
-R 20 20 - 200 每次事件循环最多处理的请求数
-o modern 关闭 建议开启 启用现代特性：meta protocol、extstore 等

下面是一组 benchmark 数据，展示不同参数组合下的性能差异：

# 测试环境：8 vCPU / 32GB RAM / SSD, Ubuntu 22.04
# 测试工具：mcperf
# 场景：100 字节 key + 200 字节 value，90% GET + 10% SET

# 默认配置（4 线程）
mcperf --servers=127.0.0.1:11211 --num-conns=10 --num-calls=1000000
  -> QPS: 1,020,000  p99: 2.5ms

# 调优配置（8 线程 + 增大连接数 + modern 模式）
memcached -m 8192 -c 32768 -t 8 -f 1.15 -R 100 -o modern
mcperf --servers=127.0.0.1:11211 --num-conns=10 --num-calls=1000000
  -> QPS: 1,840,000  p99: 1.1ms

# 极限配置（16 线程 + 不绑定 CPU 时的反面案例）
memcached -m 8192 -c 65536 -t 16
mcperf --servers=127.0.0.1:11211 --num-conns=10 --num-calls=1000000
  -> QPS: 1,670,000  p99: 3.8ms  ← 线程数过多导致锁争用反而变慢

从上表可以看出几个关键结论：

  • 线程数不是越多越好。在 8 vCPU 的环境下，-t 8 比 -t 4 提升了 80% 的 QPS，但 -t 16 反而下降了（线程切换和锁争用的开销超过了并行收益）。
  • -R 参数常被忽略但很重要。增大 -R 可以减少事件循环的切换开销，但太大会导致单次循环中处理请求过多，增加尾延迟。
  • -o modern 应该默认开启。它并不只是"新功能开关"，某些 modern 模式下的优化项（如 no_lru_crawler）直接减少了后台线程对主线程的干扰。
调优原则：不要一次性改所有参数。每次只改一个参数，跑 benchmark，记录结果，再改下一个。用数据驱动而不是凭感觉调优。

热 key 问题的根因与解决方案

热 key（hot key）是缓存系统中最常见也最难缠的问题之一。当一个或少量 key 被极高频率地访问时（比如微博热搜第一名、双十一的秒杀商品），这个 key 所在的 Memcached 节点会成为性能瓶颈——所有客户端请求都涌向同一台机器。

热 key 的根因有三个层面：

  • 业务层面——用户访问天然服从 Zipf 分布，头部 key 的访问频率可能比尾部高出几个数量级。
  • 路由层面——一致性哈希虽然能均匀分布 key，但它对"单 key 过热"无能为力（key 只有一个，只能存在于一个节点上）。
  • 节点层面——单节点 CPU 成为瓶颈，worker 线程满载，p99 延迟飙升。

解决方案通常采用分层缓存的策略：

# 热 key 治理的典型架构

# 第一层：本地缓存（客户端进程内）
#   - 在应用进程中增加一个 LRU cache（如 Guava Cache、Caffeine）
#   - TTL 设置很短（1-5 秒），只用于吸收单 key 的热点峰值
#   - 示例伪代码：
def get_with_local_cache(key):
    val = local_cache.get(key)       # 查本地缓存
    if val is None:
        val = memcached.get(key)     # 查分布式缓存
        if val is not None:
            local_cache.put(key, val, ttl=2)  # 本地缓存 2 秒
    return val

# 第二层：Memcached 集群（分布式缓存）
#   - 正常 rehash 分片
#   - 对于热 key，可以考虑在应用层做 key 扩散
#     例如将 hot_key 拆为 hot_key_0 ~ hot_key_9，随机读写
#     目的是让热 key 的请求分散到不同节点

# 第三层：数据库（最后防线）
#   - 当两层缓存都 miss 时回源数据库
#   - 设置合理的回源限流，防止数据库被打崩
经验数据：在我们的一次生产压测中，增加本地缓存后，Memcached 集群的 QPS 从 500 万降到了 80 万（热 key 请求被本地缓存吸收了），而业务方看到的延迟从 8ms 降到了 0.5ms。这是一个典型的"多花钱少办事"的反直觉案例——增加了一层缓存，总延迟反而大幅下降。

连接数过多导致的内存分配失败

每个连接到 Memcached 的客户端都会消耗大约 16KB 的读缓冲区（在 conn_new() 中通过 malloc 分配）。这个缓冲区在连接建立时分配，在连接关闭时释放。

看起来 16KB 不大对吧？但如果你有 10 万个并发连接，仅读缓冲就需要 1.6GB 的内存。加上写缓冲（不定大小，取决于响应数据量）、连接对象本身（struct conn 约 1KB）、以及其他辅助结构，单个连接的完整开销可能在 20-30KB 左右。10 万连接就是 2-3GB 内存被"连接"吃掉了，而不是被"缓存数据"吃掉了。

更严重的问题发生在内存不足时：如果 Memcached 无法为新的连接分配读缓冲，conn_new() 会返回 NULL，连接被拒绝——但此时客户端看到的是"连接超时"或"连接被拒绝"，而运维这边看到的 stat 是 listen_disabled_num 在增长。

惨案回忆：我经历过的一次线上事故——某个微服务因为 bug 不断地创建和销毁到 Memcached 的连接（连接池设置错误，每次请求都新建连接），导致 Memcached 上的连接数在 30 分钟内从 2000 飙升到 8 万。结果是 Memcached 的内存被连接缓冲消耗殆尽，大量缓存数据被 evict，缓存命中率从 95% 暴跌到 20%，数据库被击穿，最终引发级联故障。事后分析发现：如果当时设置了 -c 20000（最大连接数限制），Memcached 会在达到上限后拒绝新连接，而不是让连接缓冲无限制地吞噬缓存内存。

解决方案：

  • 在客户端使用连接池（connection pool），复用连接。
  • 在 Memcached 侧设置合理的 -c 参数，限制最大连接数。
  • 监控 curr_connections 和 listen_disabled_num，设置告警阈值。

Chunk Size 选择不当的惨案

这是 slab allocator 的特色问题。回顾第 8 章的内容，Memcached 的 slab allocator 将内存划分为不同大小的 chunk（slab class 1 的 chunk 最小，class N 的 chunk 最大），每个 key-value pair 被放入第一个能容纳它的 chunk class 中。

如果 -f（growth factor）设置不当，会导致大量的内存浪费。让我们看一个真实案例：

# 某公司的 Memcached 内存使用分析
# slab class 分布（-f 1.25 默认值）

Slab Class   Chunk Size     Used Chunks     Wasted %
1             96 B            2,000          < 1%
2            120 B           15,000          < 1%
3            152 B          120,000           2%
4            192 B          850,000           5%
5            240 B        1,200,000          12%   ← 问题开始
6            304 B        1,500,000          18%   ← 实际数据平均 260B
7            384 B          400,000          25%   ← 浪费最严重
8            480 B          100,000          30%
...
Total                       ~4M items    浪费 ~50%

分析发现，这个业务场景中 90% 的 value 大小在 200-280 字节之间。数据被分配到 class 5（chunk 240B，浪费 0-40B）、class 6（chunk 304B，浪费 24-104B）和 class 7（chunk 384B，浪费 104-184B）。结果就是大量内存被 chunk 内部的碎片浪费掉了。

解决办法是将 -f 调小：

# 调整 growth factor 为 1.10
memcached -m 8192 -f 1.10 -o modern

# 调整后的 slab class 分布：
# Class 5:  chunk 216B  → 200-280B 的数据被更精细地分类
# Class 6:  chunk 238B
# Class 7:  chunk 262B  ← 最适合 260B 数据的 class
# Class 8:  chunk 288B
# 内存浪费从 ~50% 降到了 ~15%
经验法则：在确定 -f 之前，先用实际数据做一次"数据大小分布分析"。在业务中打点记录 value 大小的分布，然后选择能覆盖主要峰值的 growth factor。如果 value 大小集中在少数几个区间，甚至可以考虑自定义 chunk size。

STAT 监控指标解读

在生产环境运维 Memcached，你需要读懂它的统计信息。以下是最关键的几个指标：

指标 含义 警告阈值 说明
evictions 因内存不足被淘汰的 item 数 持续增长 > 0 意味着内存不够用了，需要扩容或减少数据量
get_hits / get_misses GET 命中/未命中次数 命中率 < 85% 命中率过低说明缓存失效或内存不足
cmd_set SET 命令总数 相对变化 > 50% SET 突增通常意味着缓存穿透或数据未复用
curr_connections 当前连接数 接近 -c 上限的 80% 连接数过高需要排查客户端连接池配置
listen_disabled_num 因连接数满被拒绝的连接数 > 0 需要立即扩容 -c 或排查连接泄漏
reclaimed LRU 尾部的过期 item 被复用的次数 持续增长 说明有大量过期 item，可能 TTL 设置过短
slab_reassign_rescues Slab reassign 过程中 rescue 的 item 数 持续 > 1000/s 频繁的 slab 迁移会影响性能

获取这些指标的命令是 stats，也可以通过 stats slabs、stats items 查看更细粒度的信息。在生产环境中，建议每 10-30 秒采集一次 stats，送入 Prometheus + Grafana 进行可视化监控。

踩坑集锦

Nagle 算法延迟

Nagle 算法是 TCP 协议层的一个优化：它会将小的数据包合并后再发送，以减少网络中的小包数量。但这个优化对 Memcached 是"毒药"——因为 Memcached 的请求/响应通常很小（几十到几百字节），Nagle 算法会等待凑够一个 MSS 再发送，人为引入了 40ms 的延迟（经典 Nagle 延迟）。

解决方法是在 Memcached 的 socket 上设置 TCP_NODELAY 标志。这个设置在源码中的位置：

// memcached.c, 在 conn_new() 中
// 新连接建立后立即设置 TCP_NODELAY
int flag = 1;
setsockopt(sfd, IPPROTO_TCP, TCP_NODELAY, &flag, sizeof(flag));

// 从 1.4.x 版本开始默认开启，但建议在客户端侧也设置
// 如果使用 mcperf 等工具，加上 --tcp-nodelay 参数

如果你发现你的 Memcached 延迟不稳定、每隔一段时间出现 40ms 的毛刺，首先要检查的就是 Nagle 算法是否被意外开启了。

TIME_WAIT 堆积

如果客户端频繁创建和关闭到 Memcached 的短连接，服务端会积累大量 TIME_WAIT 状态的 socket。每个 TIME_WAIT socket 占用一定的内核内存，而且过多的 TIME_WAIT 会导致端口耗尽（客户端侧）或文件描述符耗尽（服务端侧）。

解决方案：

  • 客户端侧：使用连接池，避免频繁创建和关闭连接。
  • 服务端侧：可以调整内核参数来加速 TIME_WAIT 回收：
    net.ipv4.tcp_tw_reuse = 1（在客户端使用）
    net.ipv4.tcp_fin_timeout = 15（减小 TIME_WAIT 超时）
注意：新版本 Linux 内核中 tcp_tw_recycle 已被移除（因 NAT 场景的兼容性问题），不要试图用这个参数。使用 tcp_tw_reuse 即可满足大部分需求。

NUMA 绑定问题

在多路服务器（如双路 Intel Xeon）上，内存访问有 NUMA（Non-Uniform Memory Access）效应——CPU 访问本地内存比访问远程内存快得多。如果 Memcached 进程被调度在一个 NUMA 节点上，但分配的内存来自另一个 NUMA 节点，会导致显著的性能下降（实测 10-20%）。

解决方案是使用 numactl 将 Memcached 绑定到固定的 NUMA 节点：

# 查看 NUMA 拓扑
numactl --hardware

# 将 Memcached 绑定到 node 0
numactl --cpunodebind=0 --membind=0 memcached -m 8192 -t 4

# 或者使用 --interleave 让内存在所有节点间交织
numactl --interleave=all memcached -m 8192 -t 8

到底用 --cpunodebind 还是 --interleave？经验是：如果单台服务器只跑一个 Memcached 实例，用 --interleave=all 简单省事；如果一台服务器跑多个 Memcached 实例，建议每个实例绑定不同的 NUMA 节点。

性能测试工具

最后，介绍一下最常用的两个 Memcached 性能测试工具。

mcperf

mcperf 是 Memcached 官方仓库中附带的性能测试工具，位于 vendor/mcperf/ 目录。它的特点是简单、轻量、专注于吞吐和延迟测量：

# 安装 mcperf
cd memcached/vendor/mcperf
make

# 基本用法
mcperf --servers=127.0.0.1:11211 \
       --num-conns=10 \
       --num-calls=100000 \
       --timeout=5

# 指定读写比例
mcperf --servers=127.0.0.1:11211 \
       --ratio=9:1 \          # 90% GET + 10% SET
       --num-conns=20 \
       --num-calls=500000

# 指定 value 大小（模拟真实负载）
mcperf --servers=127.0.0.1:11211 \
       --valuesize=1024 \     # 1KB value
       --keysize=32

# 输出示例
# Read: 500000 gets, 4321.2 ops/s (avg 2.3 ms)
# Write: 50000 sets, 431.1 ops/s (avg 23.1 ms)

memaslap

memaslap 是 libmemcached 生态中的压测工具，功能更全面，支持多线程、多连接、持续时间设置等高级特性：

# 安装 memaslap
apt-get install libmemcached-tools

# 基础压测：100 万次操作
memaslap --servers=127.0.0.1:11211 \
         --seconds=60 \
         --threads=4 \
         --conns=20

# 模拟真实负载：不均匀分布
memaslap --servers=127.0.0.1:11211 \
         --seconds=120 \
         --concurrency=500 \
         --tps=10000 \
         --execute-number=1000000 \
         --window=5

# 输出示例
# get_misses: 0.00%, get_hits: 100.00%
# min_ latency: 0.1ms, max_latency: 12.3ms, avg_latency: 0.8ms
# ops/s: 52100, net: 45MB/s
压测要义：不要只测"最佳情况"的吞吐。真正有意义的压测是"在接近生产负载下的延迟分布"。建议用 --ratio 参数模拟真实的读写比例，用 --valuesize 模拟真实的数据大小分布，用 --tps 固定目标 QPS 来观察延迟的稳定性。
