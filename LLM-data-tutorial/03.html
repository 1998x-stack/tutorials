HTML 中的噪声问题

大家好，上章我们学会了如何从 Common Crawl 下载 WARC 文件并解析出 HTML。但如果你觉得接下来就是把 HTML 标签删掉就去喂模型，那你就要踩大坑了。

一个典型的现代 HTML 页面长什么样？顶部是导航栏，左边是侧边栏，右边是广告位，底部是页脚，中间十几行是你想读的文章，然后还有"相关阅读"推荐和评论区。真正有价值的正文内容，在页面中可能只占 10-20%。如果把整个页面当作训练数据，你的模型不仅学不到有用的知识，还会学到"点击这里"、"立即购买"、"猜你喜欢"这种垃圾信号。

有人会说："那我直接用 <p> 标签提取不就行了？"没那么简单。导航栏里的链接也是 <p>，广告文案也可能是 <p>。在 DOM 树的结构中，正文、导航、广告在标签层面是完全没有区别的。你需要基于文本密度、DOM 结构、上下文特征来判断——什么才是真正的正文。

一个经典的翻车案例： 某个早期的开源 LLM 训练项目，直接对 Common Crawl 做了标签剥离（strip tags）就喂给模型。结果模型在生成文本时频繁生成"ADVERTISEMENT"和"Click here"这样的片段——因为广告文本已经被模型学习为"自然语言"的一部分。这就是把 HTML 当纯文本来读的直接后果。

Trafilatura：正文提取的事实标准

经过多年的探索和迭代，行业内逐渐收敛到了一个解决方案——Trafilatura。这个 Python 库由 Adrien Barbaresi 开发，2021年以来已经成为数据工程流水线中正文提取的事实标准。它被 RefinedWeb、FineWeb、Dolma 等几乎所有主流数据集项目采用。

Trafilatura 的核心算法结合了两种方法：文本密度分析和DOM 树分析。文本密度分析扫描页面中的文本块密度分布——正文区域通常文本密集且连续，而导航栏虽然链接多但实际文字少。DOM 树分析则利用 HTML 标签的结构特征——正文通常位于 <article>、<main> 等语义标签内，而广告往往包裹在 <div class="ad"> 中。

使用 trafilatura 非常简单：

import trafilatura

# 直接从一个 URL 提取
downloaded = trafilatura.fetch_url("https://example.com/article")
text = trafilatura.extract(downloaded)

# 或者从 HTML 字符串提取
with open("page.html", "r", encoding="utf-8") as f:
    html_content = f.read()
text = trafilatura.extract(html_content, output_format="txt")

# 也可以输出 Markdown 格式保留结构
markdown = trafilatura.extract(html_content, output_format="markdown")

# 更精细的控制：指定语言偏好
text = trafilatura.extract(
    html_content,
    include_links=False,      # 是否保留超链接
    include_images=False,     # 是否保留图片描述
    favor_precision=True,     # 偏向精度 vs 召回
)

为什么 RefinedWeb 会选择 trafilatura？2023年 TII 团队在构建 RefinedWeb 数据集时，对多个正文提取工具做了系统的基准测试。结果如下：

提取方法语言精确率召回率速度（文档/秒）特点
TrafilaturaPython94.2%91.5%~500文本密度 + DOM 混合，多语言支持好
ReadabilityPython88.3%85.1%~800Mozilla 项目，基于类 Readability 算法
BoilerpipeJava85.7%80.2%~1200速度快但精度一般，已有多年未维护
jusTextPython90.1%82.3%~600基于块分类，对短文本表现好

Trafilatura 在精确率和召回率之间取得了最好的平衡。更重要的是，它对多语言内容的支持远超其他方案——这对需要处理多语种数据的 LLM 项目至关重要。

替代提取方案

Trafilatura 是主流选择，但并不是唯一选择。根据你的场景，以下方案也值得考虑：

resilient（Rust 实现）： 2024年由社区开发的 Rust 版本的正文提取库，性能比 trafilatura 提升了约 5-10 倍。如果你的数据量达到 PB 级别，用 Rust 可以显著降低计算成本。但它的生态还没有 trafilatura 成熟，文档和社区支持相对不足。

newspaper4k： 专门为新闻文章设计的提取工具，在处理新闻类页面时精确率甚至略高于 trafilatura。它的局限也很明显：只能处理新闻类结构，对于博客、论坛、技术文档等非新闻页面效果不佳。如果你的数据源主要是新闻网站（如某些专有数据集），这是一个不错的选择。

LLM-based 提取： 这是 2025 年的新趋势。直接让一个小的语言模型（如 Llama-3.2-3B）来识别页面的正文区域。这种方法的优点是极大的灵活性——可以处理各种奇特的页面布局。缺点是速度太慢（每秒只能处理几个页面）和成本高昂。目前主要用于对精度要求极高的场景，如构建评测集黄金数据。

我们的建议： 对于绝大多数 LLM 数据工程项目，先使用 trafilatura 做第一轮提取。如果发现某个特定类型的站点 trafilatura 效果不好，再针对性地用其他方案做补充。不要一开始就追求"最好的提取方案"——工程上 90% 的精度配合大规模处理，远比 95% 的精度但只能处理小规模数据更有效。

提取质量验证

你可能会问：我怎么知道 trafilatura 提取出来的内容行不行？答案是：你没法自动验证，必须人工抽检。

RefinedWeb 团队的做法值得借鉴。他们在清洗过程中设置了质量门禁：从每个处理批次中随机采样 1000 条 URL，由人工逐条检查提取结果，判断正文是否完整、是否有遗漏、是否混入了噪声。具体评分标准包括：

  • 完整性： 正文内容是否从头到尾完整保留，有无被截断
  • 纯净度： 是否混入了导航栏、广告、评论区等非正文内容
  • 结构性： 段落分割是否合理，标题层级是否保留
  • 异常检测： 是否提取到了空白页面、验证码页面、确认页面

常见的提取失败场景包括：

  • 确认页面（CAPTCHA 或年龄验证）： 很多网站会拦截爬虫，返回一个只有"请完成人机验证"的页面。Trafilatura 会忠实地提取出这行字。
  • 404 / 软 404： 页面不存在时返回的提示文本，如"对不起，您访问的页面不存在"。
  • Cookie 通知弹窗： 某些页面把 Cookie 通知做得非常大，占据了几乎整个视口，导致正文被挤到看不见的地方。
  • 视频页面： YouTube、B站等视频页面的"正文"其实只是描述文字，信息密度极低。
踩坑分享： 我见过一个团队在提取中文新闻数据时，发现模型学到了一句话："欢迎发表评论，评论内容不超过200字。"这句话在几乎所有新闻页面底部的评论区出现，被 trafilatura 当成了正文的一部分。半年后他们才发现训练语料中有数百万条数据都带着这句"瓶盖"。这就是正文提取的一个缩影——每一个被误包含的噪声片段，最终都会在模型行为中留下痕迹。

正文提取对下游的影响

正文提取看起来只是流水线中不起眼的一步，但它对最终模型质量的影响比你想象的大得多。

提取方法MMLUHellaSwagARC-Challenge备注
粗暴标签剥离（strip_tags）38.2%52.1%31.5%包含了大量导航和广告
Readability41.5%55.3%34.8%部分广告文本被保留
Trafilatura43.1%57.8%36.2%最干净的提取结果
Trafilatura + 人工抽检修复43.6%58.2%36.9%额外的人力投入带来边际收益

（以上数据基于 RefinedWeb 团队在 1B 参数模型上的内部测试）

有趣的是，不同提取方法在 MMLU 上的差距大约有 5 个点。对于 1B 参数的模型来说，这 5 个点的差距几乎完全来自数据质量的差异。如果换算到 70B 模型，虽然绝对值会变化，但相对差距通常会被放大。

正文提取作为整个流水线的第一步过滤，它的质量直接决定了后续所有步骤的基准线。如果在这里就已经引入了大量噪声和错误，后续的规则过滤、质量分类器再怎么努力，也很难完全弥补。这就是为什么业内常说："Garbage in = Garbage out"，而正文提取就是你阻止"garbage in"的第一道防线。

提取出干净的正文之后，接下来的挑战是：这是一个多语言的互联网数据，你需要的语言在所有页面中可能只占一小部分。下一章我们来聊聊如何用语言识别技术从海量多语言数据中精准定位目标语言的内容。
