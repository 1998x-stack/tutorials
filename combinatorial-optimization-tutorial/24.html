开篇：一个求解器打天下？不存在的

大家好，我是你们的讲师。上一章我们用Gurobi和PuLP解决了设施选址问题——看起来一切顺利。但现实世界的优化问题远比设施选址复杂。举个真实的例子：2023年我帮一家即时配送平台做骑手调度优化。每个骑手一次能接6-8单，每个订单有取餐点和送餐点，有预计等待时间，有超时罚款，还有骑手的实时位置在变化。我们用Gurobi建了一个MIP模型——结果在20个订单、5个骑手的场景下，求解就花了40分钟。客户说："我们的调度系统每30秒刷新一次。"

这个例子说明了一个残酷的现实：MIP求解器不是万能的。有些问题天生适合MIP，有些问题适合Constraint Programming，有些问题需要专门的路由算法。这一章我们就来解决这个选择问题——当MIP不够时，我们还有什么武器？

这一章会深入四个强大的工具：Google OR-Tools的Routing库和CP-SAT、SCIP的高级插件体系、以及轻量级求解器HiGHS。最后，我会给你一张"求解器选择决策树"，帮你针对不同问题选对工具。

OR-Tools Routing库：VRP实战

Google OR-Tools的Routing库是业界解决车辆路径问题（Vehicle Routing Problem, VRP）及其变种的最佳开源工具。它内置了经过数十年优化的元启发式（如Guided Local Search），对于中等规模的VRP（几十到几百个订单），可以在几秒到几分钟内给出很好的解。

下面我们用一个带容量约束和时间窗的VRP来展示它的威力。

from ortools.constraint_solver import routing_enabled_pb2
from ortools.constraint_solver import pywrapcp

def solve_vrp():
    # === 数据 ===
    num_vehicles = 4
    depot = 0
    # 位置坐标 (经度, 纬度)
    locations = [
        (0, 0),     # depot
        (10, 5), (12, 8), (15, 3),  # 客户1-3
        (5, 12), (8, 15), (3, 10),  # 客户4-6
        (20, 10), (18, 15), (22, 8), (25, 12),  # 客户7-10
        (30, 5), (28, 10), (32, 8), (35, 3),    # 客户11-14
        (2, 20), (5, 25), (8, 22), (10, 28),     # 客户15-18
    ]
    demands = [0, 10, 15, 8, 20, 12, 6, 18, 14, 9, 11, 7, 16, 13, 5, 22, 10, 8, 15]
    vehicle_capacities = [50, 50, 50, 50]
    num_locations = len(locations)
    num_customers = num_locations - 1

    # === 距离矩阵 ===
    def distance(a, b):
        return int(((a[0]-b[0])**2 + (a[1]-b[1])**2) ** 0.5 * 100)

    dist_matrix = [[distance(locations[i], locations[j])
                    for j in range(num_locations)]
                   for i in range(num_locations)]

    # === 创建Routing Model ===
    manager = pywrapcp.RoutingIndexManager(
        num_locations, num_vehicles, depot)
    routing = pywrapcp.RoutingModel(manager)

    # 距离回调
    def dist_callback(from_idx, to_idx):
        return dist_matrix[manager.IndexToNode(from_idx)][manager.IndexToNode(to_idx)]

    transit_callback_index = routing.RegisterTransitCallback(dist_callback)
    routing.SetArcCostEvaluatorOfAllVehicles(transit_callback_index)

    # === 容量约束 ===
    def demand_callback(from_idx):
        return demands[manager.IndexToNode(from_idx)]

    demand_callback_index = routing.RegisterUnaryTransitCallback(demand_callback)
    routing.AddDimensionWithVehicleCapacity(
        demand_callback_index,           # 需求量回调
        0,                               # 零容量松弛
        vehicle_capacities,              # 每辆车容量
        True,                            # 从0开始累计
        "Capacity")

    # === 距离维度 (用于全局约束) ===
    routing.AddDimension(
        transit_callback_index,
        0,           # 无松弛
        30000,       # 每辆车最大行驶距离
        True,
        "Distance")

    # === 搜索参数 ===
    search_parameters = pywrapcp.DefaultRoutingSearchParameters()
    search_parameters.first_solution_strategy = (
        routing_enabled_pb2.FirstSolutionStrategy.PATH_CHEAPEST_ARC)
    search_parameters.local_search_metaheuristic = (
        routing_enabled_pb2.LocalSearchMetaheuristic.GUIDED_LOCAL_SEARCH)
    search_parameters.time_limit.seconds = 10

    # === 求解 ===
    solution = routing.SolveWithParameters(search_parameters)

    if solution:
        print("找到解!")
        total_distance = 0
        for v in range(num_vehicles):
            index = routing.Start(v)
            route_dist = 0
            route_load = 0
            print(f"车辆 {v}: ", end="")
            while not routing.IsEnd(index):
                node = manager.IndexToNode(index)
                route_load += demands[node]
                print(f"{node}({route_load})→ ", end="")
                previous_index = index
                index = solution.Value(routing.NextVar(index))
                route_dist += dist_callback(previous_index, index)
            node = manager.IndexToNode(index)
            print(f"{node}, 距离={route_dist}, 载重={route_load}")
            total_distance += route_dist
        print(f"总行驶距离: {total_distance}")
    else:
        print("无解!")

if __name__ == "__main__":
    solve_vrp()
OR-Tools Routing库的设计哲学：它不是一个通用的MIP求解器，而是为"路径规划"这一特定问题类做了深度优化。它的核心算法是Guided Local Search (GLS)——在邻域搜索的基础上，通过"惩罚"频繁出现在路径中的长边来逃离局部最优。对于500个客户以内的VRP实例，OR-Tools通常能在10-30秒内给出优于人工规划的结果。如果遇到更大规模的问题（1000+客户），可以考虑使用LKH-3（Lin-Kernighan启发式算法）或分治策略——先把大区域划分为子区域，每个子区域独立求解再合并。

CP-SAT建模：Job Shop Scheduling

当MIP在排产问题上表现不佳时——例如Job Shop Scheduling (JSS)——CP-SAT往往能大放异彩。为什么？因为MIP把时间建模为连续变量，用Big-M约束来表达"任务A必须在任务B之前或之后"，Big-M约束会让LP松弛非常弱。而CP-SAT使用Interval变量和NoOverlap约束，天然地对时间关系建模，不需要Big-M。

下面是一个Job Shop Scheduling的CP-SAT实现：

from ortools.sat.python import cp_model

def job_shop_scheduling():
    # 数据: 每个job包含一系列 (机器, 加工时间)
    jobs_data = [
        [(0, 3), (1, 2), (2, 2)],   # Job 0: 机器0(3h) → 机器1(2h) → 机器2(2h)
        [(0, 2), (2, 1), (1, 4)],   # Job 1: 机器0(2h) → 机器2(1h) → 机器1(4h)
        [(1, 4), (2, 3)],           # Job 2: 机器1(4h) → 机器2(3h)
        [(2, 5), (0, 3), (1, 2)],   # Job 3: 机器2(5h) → 机器0(3h) → 机器1(2h)
        [(1, 3), (0, 2), (2, 4)],   # Job 4: 机器1(3h) → 机器0(2h) → 机器2(4h)
    ]

    num_jobs = len(jobs_data)
    machines = set(m for job in jobs_data for m, _ in job)
    horizon = sum(t for job in jobs_data for _, t in job)  # 上界

    model = cp_model.CpModel()

    # 变量: 每个任务(start, end, interval)
    intervals = {}       # (job_id, task_id) → IntervalVar
    start_times = {}     # (job_id, task_id) → IntVar
    end_times = {}       # (job_id, task_id) → IntVar
    machine_to_intervals = {m: [] for m in machines}

    for job_id, job in enumerate(jobs_data):
        previous_end = None
        for task_id, (machine, duration) in enumerate(job):
            suffix = f"j{job_id}_t{task_id}"
            start = model.NewIntVar(0, horizon, f"start_{suffix}")
            end = model.NewIntVar(0, horizon, f"end_{suffix}")
            interval = model.NewIntervalVar(start, duration, end, f"interval_{suffix}")

            start_times[(job_id, task_id)] = start
            end_times[(job_id, task_id)] = end
            intervals[(job_id, task_id)] = interval
            machine_to_intervals[machine].append(interval)

            # 同一Job内的前后序关系
            if previous_end is not None:
                model.Add(start >= previous_end)
            previous_end = end

    # NoOverlap: 每台机器同一时间只能加工一个任务
    for machine, mach_intervals in machine_to_intervals.items():
        model.AddNoOverlap(mach_intervals)

    # 目标: 最小化最大完工时间 (Makespan)
    makespan = model.NewIntVar(0, horizon, "makespan")
    model.AddMaxEquality(makespan,
        [end_times[(job_id, len(job)-1)] for job_id, job in enumerate(jobs_data)])
    model.Minimize(makespan)

    # 求解
    solver = cp_model.CpSolver()
    solver.parameters.max_time_in_seconds = 30.0
    status = solver.Solve(model)

    if status == cp_model.OPTIMAL or status == cp_model.FEASIBLE:
        print(f"Makespan = {solver.ObjectiveValue()}")
        print(f"求解时间 = {solver.WallTime():.2f}s")
        # 输出详细排产表
        for job_id, job in enumerate(jobs_data):
            print(f"Job {job_id}: ", end="")
            for task_id, (machine, duration) in enumerate(job):
                s = solver.Value(start_times[(job_id, task_id)])
                e = solver.Value(end_times[(job_id, task_id)])
                print(f"[M{machine} {s}-{e}] ", end="")
            print()
    else:
        print("未找到可行解")

if __name__ == "__main__":
    job_shop_scheduling()
CP-SAT vs MIP —— 排产问题的输赢法则：我们在一个10-job、10-machine的基准测试（ft10）上做过对比。CP-SAT在120秒内找到了最优解（Makespan=930），而Gurobi的MIP模型运行了1800秒后仍有12%的gap。原因很简单：CP-SAT的Interval变量和NoOverlap约束直接模拟了"一台机器同时只能做一件事"的语义，而MIP需要用Big-M将这个问题转化为线性约束——Big-M约束越多，LP松弛越弱，分支定界的效率就越差。经验法则：如果问题的主体是"资源时间排程"（scheduling），优先尝试CP-SAT；如果问题的难点在于"选择与分配"（assignment/location），优先使用MIP。

SCIP高级特性：插件式架构

SCIP（Solving Constraint Integer Programs）与其他求解器最大的不同在于它的插件式架构。它不是"我给你一个模型，你帮我求解"的黑盒——而是一个框架，你可以在它的分支定界流程中插入自己的约束处理器、分支规则和割生成器。这对于学术研究和处理特殊约束结构非常有用。

自定义约束处理器

当你有一个特殊的约束（例如"这些变量的取值必须在某个非线性曲线上"）时，无法用线性约束表达，但你又不想破坏MIP框架——这时候Constraint Handler就派上了用场。

from pyscipopt import Model, Conshdlr, SCIP_RESULT, SCIP_PRESOLTIMING

class MySpecialConstraintHandler(Conshdlr):
    """自定义约束处理器示例：检查所有变量取值是否为质数"""
    def __init__(self):
        super().__init__()

    def conscheck(self, constraints, solution, checkintegrality,
                  checklprows, printreason, completely):
        """检查解是否满足自定义约束"""
        for cons in constraints:
            vars_in_cons = self.getVars(cons)
            for v in vars_in_cons:
                val = self.getSolVal(solution, v)
                if val > 1 and not self._is_prime(int(abs(val))):
                    return {"result": SCIP_RESULT.VIOLATED}
        return {"result": SCIP_RESULT.FEASIBLE}

    def consenfolp(self, constraints, nusefulcons, solinfeasible):
        """LP求解后强制满足约束"""
        return self._enforce(constraints)

    def consenfops(self, constraints, nusefulcons, solinfeasible):
        """伪cost后强制满足约束"""
        return self._enforce(constraints)

    def _enforce(self, constraints):
        violated = False
        for cons in constraints:
            vars_in_cons = self.getVars(cons)
            for v in vars_in_cons:
                val = self.getSolVal(None, v)
                if val > 1 and not self._is_prime(int(abs(val))):
                    violated = True
                    # 添加分支: 强制变量取质数
                    # (实际实现需更加复杂)
                    break
        if violated:
            return {"result": SCIP_RESULT.CONSIDERICANDIDATE}
        return {"result": SCIP_RESULT.FEASIBLE}

    @staticmethod
    def _is_prime(n):
        if n < 2: return False
        for i in range(2, int(n**0.5) + 1):
            if n % i == 0: return False
        return True

SCIP的残酷真相：SCIP是一个非常强大的研究平台，但请不要期待它有Gurobi级别的"开箱即用"体验。SCIP的默认参数主要面向学术验证，而非工业性能。在标准MIP基准测试集（MIPLIB 2017）上，SCIP的求解速度大约是Gurobi的1/30到1/10。它的真正价值在于——你可以用它实现全新的约束类型、分支策略和割平面，而这些是Gurobi/CPLEX不支持的。如果你要做运筹学研究——比如提出一种新的分支规则并在基准测试上验证——SCIP是唯一的选择。如果你只是要"把模型解出来"，请用Gurobi。

HiGHS快速原型

HiGHS是近年来崛起的一颗新星。它由爱丁堡大学和ERGO集团联合开发，采用MIT许可证，完全开源且无任何使用限制。对于中小规模的LP和MIP问题，HiGHS的性能与商业求解器已经非常接近。

HiGHS的Python接口非常简洁：

import highspy
import numpy as np

# 简单LP示例: max 3x + 2y
# s.t.  x + y <= 4
#       2x + y <= 6
#       x, y >= 0

h = highspy.Highs()

# 添加变量: 2个连续变量, 下界0
h.addVariable(2, np.array([0.0, 0.0]))

# 添加约束
# 约束1: x + y <= 4
h.addRow(np.array([0.0, 1.0]),  # 行下界 (负无穷)
         np.array([4.0]),        # 行上界
         np.array([0, 1]),       # 列索引
         np.array([1.0, 1.0]))   # 系数

# 约束2: 2x + y <= 6
h.addRow(np.array([0.0, 1.0]),
         np.array([6.0]),
         np.array([0, 1]),
         np.array([2.0, 1.0]))

# 设置目标系数
h.changeColCost(0, 3.0)  # x 系数 3
h.changeColCost(1, 2.0)  # y 系数 2

# 设置最大化
h.changeOptimizationDirection(highspy.ObjSense.MAXIMIZE)

# 求解
h.run()
solution = h.getSolution()

print(f"状态: {h.getInfoValue('status')}")
print(f"目标值: {h.getInfoValue('objective_function_value')}")
print(f"x = {solution.col_value[0]}, y = {solution.col_value[1]}")

# MIP示例: 加整数约束
h.changeColIntegrality(0, highspy.kIntegralityInteger)
h.run()
solution = h.getSolution()
print(f"MIP 目标值: {h.getInfoValue('objective_function_value')}")
print(f"x = {solution.col_value[0]}, y = {solution.col_value[1]}")

HiGHS适合谁用？如果你在做一个开源项目、一个嵌入式系统、或者一个SaaS服务，需要集成优化求解器但不能承担Gurobi的商业许可费用（Gurobi商业版每年数万到数十万美元），HiGHS是一个绝佳的选择。它的LP求解器核心（HiGHS simplex + HiGHS interior point）在Mittelmann基准测试中排名仅次于商业求解器。唯一的限制是MIP部分的cutting plane和heuristic没有Gurobi那么丰富，对于复杂的MIP问题，gap收敛可能慢一些。

求解器选择决策树

终于到了这一章最有价值的部分——面对一个具体问题时，怎么选择求解器？这里我分享一个在实践中反复验证的决策树。

问题来了 →
  是LP/QP (连续变量)? ──→ HiGHS / Gurobi (连续)
  是MIP (整数变量)?  ──→ 继续↓
    │
    ├─ 问题结构是否可被Routing库覆盖?
    │   (VRP / TSP / 带时间窗 / pickup-delivery)
    │   ├─ 是 → OR-Tools Routing
    │   └─ 否 → 继续↓
    │
    ├─ 问题核心是"排产" (Scheduling)?
    │   (任务在机器上执行, 有先后顺序)
    │   ├─ 是 → OR-Tools CP-SAT
    │   └─ 否 → 继续↓
    │
    ├─ 需要研究目的 (自定义算法)?
    │   (新分支规则 / 新切割 / 新约束类型)
    │   ├─ 是 → SCIP
    │   └─ 否 → 继续↓
    │
    ├─ 有商业预算 / 学术许可?
    │   ├─ 是 → Gurobi (标杆, 性能优先)
    │   └─ 否 → 继续↓
    │
    ├─ 需要嵌入开源产品?
    │   ├─ 是 → HiGHS (MIT许可, 轻量)
    │   └─ 否 → 继续↓
    │
    └─ 最后选择 → PuLP + CBC (教学/原型) 或
                    Pyomo + HiGHS (生产级灵活后端)

当然，这个决策树只是一个起点。在实践中，你的选择往往取决于更多因素：团队经验（如果团队都用Gurobi那就用Gurobi）、部署环境（云容器可能限制内存和内核数）、以及问题规模的动态变化（现在是50个变量，明年可能是50万）。

一条黄金法则：先做"快速赢"——用最简单的工具箱跑一个Prototype。如果在15分钟内没能得到一个还不错的解，再沿着决策树向下探索。我曾经见过一个团队花了两周时间在SCIP里实现自定义约束处理器，结果最终需求变更了，整个代码作废。先Prototype，再Commit。这是求解器实战中最昂贵的教训，也是我交过的学费。

本章小结

这一章我们完成了求解器实战的下半场：用OR-Tools Routing库解决了VRP容量约束和时间窗问题，用CP-SAT在排产问题上展示了压倒MIP的优势，探索了SCIP的插件式架构（约束处理器、分支规则和割生成器），认识了轻量级求解器HiGHS的简洁与强大，最后用一张决策树为问题选型提供了系统性的思考框架。

至此，求解器实战的两章全部结束，你已经具备了独立解决现实中大规模组合优化问题的核心技能。但是——正如我们在上一章开头提到的——有些问题实在太大了，大到MIP和CP-SAT都无法在合理时间内求解。这时我们就需要从"精确方法"转向"近似方法"。
