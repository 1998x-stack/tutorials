18.1 Jupyter Notebook里跑通的模型 ≠ 生产系统

有一个著名的数据：在业界，只有约13%的数据科学项目真正进入了生产环境。剩下的87%死在了Jupyter Notebook里。为什么？因为"训练一个效果不错的模型"和"构建一个可靠、可维护的ML系统"之间有一个巨大的鸿沟。

我亲身经历过这个痛。一个客户流失预测模型，在Notebook上AUC 0.88，领导说"上线"。我花了一周把代码打包成脚本放到服务器上跑定时任务。两周后模型效果断崖式下跌——原来一个上游数据源的格式变了，我的代码既没有验证也没报警，默默地把错误数据喂给了模型，然后模型默默产出了错误预测。从那以后我才真正理解：ML工程化不只是"部署"这一步，而是一个贯穿始终的系统工程思维。

18.2 Pipeline：让数据流自动化

sklearn的Pipeline把多个步骤串联成一个整体。这不仅仅是代码组织方式的问题——它直接关系到模型在生产环境中是否可靠。

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.compose import ColumnTransformer
from sklearn.ensemble import RandomForestClassifier

# ColumnTransformer：对不同列做不同预处理
preprocessor = ColumnTransformer([
    ('num', StandardScaler(), ['age', 'income', 'tenure']),
    ('cat', OneHotEncoder(handle_unknown='ignore'), ['city', 'plan']),
])

# 完整Pipeline
pipeline = Pipeline([
    ('preprocessor', preprocessor),
    ('feature_selector', SelectFromModel(RandomForestClassifier())),
    ('classifier', RandomForestClassifier(n_estimators=200))
])

pipeline.fit(X_train, y_train)

# 保存整个pipeline（不只是模型！）
import joblib
joblib.dump(pipeline, 'full_pipeline.pkl')

# 生产环境使用：输入原始数据，返回预测
predictions = joblib.load('full_pipeline.pkl').predict(raw_data)
💡 Pipeline的核心价值：把预处理逻辑和模型打包在一起保存。这样你在训练时怎么处理数据的，推理时就用完全一样的方式处理。消除了"训练代码里做了标准化，部署代码忘了做"这类人为错误。在生产环境中，保存 Pipeline 而不是保存模型——这是血的教训换来的经验。

18.3 MLflow：实验管理与模型注册

当你同时跑十几个实验，参数各不同，一周后回来看"best_model_v3_final_2.pkl"时，你已经完全忘了它对应什么参数、什么数据、什么结果。MLflow就是为这个痛点而生的。

MLflow组件功能核心概念
Tracking记录实验参数、指标、模型Run、Experiment
Models标准化模型格式，跨框架部署Flavor、Signature
Registry模型版本管理、阶段切换Stage、Version
import mlflow
import mlflow.sklearn

mlflow.set_experiment("customer_churn_prediction")

with mlflow.start_run():
    # 记录参数
    mlflow.log_params({'n_estimators': 200, 'max_depth': 10})

    # 训练模型
    model = RandomForestClassifier(n_estimators=200, max_depth=10)
    model.fit(X_train, y_train)

    # 记录指标
    mlflow.log_metrics({'accuracy': 0.87, 'f1': 0.82, 'auc': 0.91})

    # 保存模型（自动记录环境依赖）
    mlflow.sklearn.log_model(model, "random_forest_model")

# 之后查看：mlflow ui → http://localhost:5000

18.4 特征存储：告别"离线在线不一致"

ML系统中最隐蔽的bug之一是"训练-服务偏差"（Training-Serving Skew）：训练时用的特征计算逻辑，和线上推理时不一样。比如训练时"用户过去30天点击次数"是离线批计算的值，但线上推理时用的是另一个实时服务计算的值——两者可能因为时间窗口定义、数据延迟等原因产生差异。

特征存储（Feature Store）解决的就是这个问题：所有特征的定义、计算、存储、服务都在一个地方。业界常用的有Feast（开源）、Tecton（商业）。虽然对于小团队可能过重，但了解这个概念非常重要——即使不用专门的Feature Store，你也要确保训练和推理的特征计算代码是同一套。

18.5 MLOps成熟度模型

等级特征适合
L0: 手动Notebook训练，手动部署，无监控个人项目、探索期
L1: 自动化Pipeline数据→训练→部署全自动化，有监控初创公司、小团队
L2: CI/CD/CT持续训练、自动回滚、A/B实验中大型公司、核心业务
🔑 实战建议：不要一上来就追求L2。从L0到L1是性价比最高的投资——花一两周把训练流程Pipeline化、加上基本的异常监控，就能避免80%的生产事故。MLflow + sklearn Pipeline + 基础监控脚本，这就已经是一个不错的生产系统了。等技术债务积累到一定程度再考虑引入Kubeflow这类重型MLOps平台。
