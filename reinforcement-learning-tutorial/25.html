第二十五章：调试与可视化

同学们好，欢迎来到第二十五章。走到这一步，你们已经掌握了从DQN到PPO再到SAC的一大串算法，也学会了用SB3快速训练智能体。但有一个残酷的现实我必须告诉你们——RL代码写出来只是第一步，让它真正收敛才是噩梦的开始。

如果监督学习是"写代码2小时，调参2天"，那强化学习就是"写代码2小时，调参2周，debug 2个月"。这不是夸张——我见过太多同学在RL项目上卡住，就是因为不会看训练日志、不会分析奖励曲线、不知道从哪下手排查问题。这一章，咱们就把"RL调试"这件事系统性地讲清楚。

踩坑故事：我曾经花了一整个星期卡在一个Bug上——DQN在CartPole上就是学不会。奖励曲线从头到尾都是平的，不是震荡，是毫无波澜的平线。我检查了网络结构、学习率、epsilon衰减、replay buffer……全部看起来没问题。最后我差点放弃，随手打印了一下状态数据，发现Gymnasium返回的状态是(4,)维度的float64数组，而我定义的第一层网络输入是4，但PyTorch默认的Linear层没有报错——它只是默默地把float64当作一个奇怪的数据类型处理了，导致梯度压根没更新！一个dtype的问题，浪费了我整整七天。从那以后我学乖了：任何看起来"不该有问题"的地方，都是最值得怀疑的地方。

25.1 TensorBoard集成

SB3最方便的一点就是默认集成了TensorBoard日志。你只需要在训练时指定tensorboard_log参数：

from stable_baselines3 import PPO

model = PPO("MlpPolicy", "CartPole-v1",
            tensorboard_log="./logs/",
            verbose=1)
model.learn(total_timesteps=100000)

训练结束后，在终端里启动：

tensorboard --logdir ./logs/

然后在浏览器打开http://localhost:6006，你会看到一系列曲线。对于RL来说，下面这些指标是你必须能读懂的：

  • train/episode_reward_mean —— 最核心的指标，每轮的平均奖励。正常的曲线应该是逐渐上升然后趋于平稳。如果一直是平的，说明没有学到任何东西。
  • train/value_loss —— 值函数的损失。应该逐渐下降然后稳定在较小的值。如果剧烈震荡或爆炸，说明值函数不稳定。
  • train/policy_loss —— 策略损失。对于PPO等算法，策略损失应该是平稳波动的，不会无限增大。
  • train/entropy_loss —— 策略的熵损失。正常的熵应该是逐渐下降（策略越来越确定）但不会降到零。如果熵迅速降到接近零，说明策略过早收敛到了一个次优解。
  • train/approx_kl —— PPO的KL散度近似值。如果这个值一直很大（远超过target_kl），说明策略更新步长太大，可能导致崩溃。
  • train/clip_fraction —— PPO中被裁剪的样本比例。通常应该在0.1-0.3之间。太高说明策略更新太激进，太低说明更新太少。

怎么看"坏信号"：reward曲线如果长时间低于随机策略的水平，说明策略根本没学习；value_loss如果突然冲到天际，检查学习率和奖励尺度；entropy如果一步到位降到零，说明你的策略"学会了"但学会的是错的——这是RL里最难debug的场景之一。

25.2 Weights & Biases (WandB)

TensorBoard很好，但当你开始跑几十上百个实验时，你就会发现它不够用了——你需要实验追踪、超参数对比、远程看结果、和团队分享。这时候就该上 Weights & Biases（简称WandB）了。

接入方式非常简单。先注册一个WandB账号（免费），然后：

import wandb
from wandb.integration.sb3 import WandbCallback

wandb.init(
    project="my-rl-project",
    config={
        "env": "CartPole-v1",
        "algo": "PPO",
        "lr": 3e-4,
        "total_timesteps": 100000,
    }
)

model = PPO("MlpPolicy", "CartPole-v1",
            learning_rate=3e-4,
            tensorboard_log="./logs/",
            verbose=1)

model.learn(
    total_timesteps=100000,
    callback=WandbCallback(
        model_save_freq=10000,
        model_save_path="./models/",
        verbose=2
    )
)

这样你每次训练的所有指标都会自动同步到WandB的云端。你可以：

  • 在网页上实时看曲线，比TensorBoard方便得多
  • 用一个链接把实验结果分享给导师或同事
  • 对比不同超参数配置下的训练曲线
  • 自动生成实验报告，记录下每轮实验的配置和结果

长期做RL研究的人都会告诉你：好记性不如烂记录本，而WandB就是你最完美的实验记录本。我每跑一个实验之前都会检查——这次实验的hypothesis是什么？我在验证什么？不改记录不清的实验，等于没有做过。

25.3 奖励曲线诊断

在RL调试中，奖励曲线就是你的生命体征监测仪。学会读奖励曲线，你就能在95%的情况下快速定位问题。下面这几种是常见的曲线模式：

  • 平坦直线（不学习）：奖励一直在一个低水平晃动，没有任何上升趋势。原因可能是：探索不够（epsilon衰减太快）、奖励信号太稀疏、网络完全没有在学习（检查梯度）。
  • 先升后崩（灾难性遗忘）：一开始奖励上升得很好，突然某一步崩盘，再也回不来。典型的策略崩溃——更新步长太大，策略一下"翻过了山头"掉进了深渊。调低学习率或者增大clip范围。
  • 逐步上升后趋平（正常收敛）：看到这个曲线应该高兴——你的训练一切正常。奖励逐渐上升到某个水平然后稳定。这时候可以考虑是否继续训练。
  • 剧烈震荡（策略不稳定）：曲线像过山车一样上下乱跳。常见原因：batch size太小、replay buffer太小、奖励设计不合理（奖励尺度太大或太小）。

诊断铁律：永远先看奖励曲线，再看其他指标。奖励曲线告诉你"有没有问题"，其他指标告诉你"哪里有问题"。不要跳过第一步直接进入第二步。很多同学一上来就看loss曲线，结果loss降得很漂亮，奖励却毫无变化——这是因为网络只是在"拟合自己"而没有真正提升策略。

25.4 五大常见Bug

下面这个表是我多年踩坑和同学们的血泪史总结的。我敢打赌，只要你做RL超过一个月，这五个Bug你至少会遇到三个：

# Bug 症状 解决方案
1 评估时epsilon未关闭 评估结果远差于训练奖励，曲线忽高忽低 评估时设置 epsilon=0 或 deterministic=True
2 奖励归一化出错 奖励始终为零，或数值爆炸到无法收敛 检查 RunningMeanStd 状态，使用 VecNormalize 时注意不要调用两次
3 梯度爆炸 loss突然跳到无穷大，奖励断崖式下跌 检查梯度范数（torch.nn.utils.clip_grad_norm_），适当降低学习率
4 Replay Buffer OOM 训练到一半内存溢出，或者训练越来越慢 检查 buffer_size，不要设置超过物理内存承受范围（Atari建议1e5-1e6）
5 Atari预处理错误 奖励一直徘徊在随机水平，完全学不会 检查观测是否已转为灰度（84x84）、是否帧堆叠（4帧）、是否已归一化到[0,1]

第1个Bug我见过太多次了——训练时用epsilon-greedy探索效果好，但评估时也带着epsilon随机乱试，结果评估分数还不如随机策略。这个问题一旦知道就异常好修，但不知道的时候会让你怀疑人生。

第2个和第3个Bug经常一起出现。奖励归一化做好之后，梯度爆炸的概率也会大幅下降。我把这称为"RL的蝴蝶效应"——一个小疏忽在训练中期放大成灾难。

第4个Bug是典型的"没想到资源会不够"。Atari的一帧是84x84x4，如果用float32存，1e6帧大概需要1.1GB。如果你设了buffer_size=5e6还要跑并行环境，内存分分钟爆炸。

第5个Bug是Atari新手最容易忽略的——原始Atari帧是210x160的RGB图像，直接扔给网络根本学不动。忘记做灰度化和帧堆叠，或者做了但顺序不对，都会导致完全学不会。

25.5 性能分析与优化

当你的算法终于开始收敛了，下一个问题就是：跑得太慢了。RL训练动辄几千万步，每一步都涉及到环境交互、数据存取样、神经网络前向和反向传播。学会做性能分析非常重要。

环境交互是最大瓶颈。如果是Python模拟环境（比如Gymnasium自带的经典控制问题），单步执行很快。但如果你用Atari或者自定义的物理仿真（比如MuJoCo、PyBullet），env.step可能就是最慢的一环。解决方法是使用向量化环境（VecEnv），同时跑多个环境实例。SB3的SubprocVecEnv或DummyVecEnv可以轻松实现。

GPU利用率：如果你的GPU利用率低于50%，说明batch size太小了，GPU在大部分时间等CPU送数据。适当增大batch size或者n_steps。对于PPO，一般batch size在64-4096之间。如果你用的是DQN，考虑用更高效的采样方式。

性能分析工具：用 py-spy 可以快速定位Python代码中的性能热点。用 torch.profiler 则可以看到PyTorch算子的执行时间。一个简单的做法是：

# 安装
pip install py-spy

# 在训练运行时，另一个终端执行
sudo py-spy record -o profile.svg --pid <训练进程PID>

# 用浏览器打开profile.svg就能看到火焰图

火焰图里横轴越长表示耗时越多的函数。如果你的env.step占了一半以上的时间，就该考虑向量化环境；如果模型forward占了大部分时间，检查网络是否太大。没有profile就做优化，等于盲人摸象。

优化优先级：先profile定位瓶颈，再针对性优化。优化顺序建议：环境交互（向量化）> 数据采集效率 > 网络结构 > 超参数调整。不要上来就改网络结构——大概率你的瓶颈不在那里。

好了，掌握了这些调试技巧和可视化方法，你已经从一个"写代码的RL选手"升级成了"会诊断的RL医生"。从下一章开始，我们将离开RL的"经典"世界，进入RL的前沿领域——那些正在改变这个领域的最新研究方向。强化学习还没有走到尽头，恰恰相反，真正激动人心的东西才刚刚开始。
