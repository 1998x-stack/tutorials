第20章 Context Engineering 深度分析

大家好，我是你们的讲师。上一章我们在 Agent 中集成了 MCP 支持，让它能和外部工具对话了。但有一个问题很多人会忽略——当你把 System Prompt、工具定义、对话历史、MCP 工具描述全部塞进上下文后，你的 API 账单会像坐了火箭一样飞涨。我在第一个月用 Claude Code 的时候就踩过这个坑：每天跑几小时的 Agent 任务，月底收到 API 账单时差点从椅子上摔下来。

这引出了一个关键问题：在 Agent 系统中，Context 不仅仅是你和模型说话的内容，它本身就是系统的一个核心资源。如何管理这个资源——如何最大化缓存命中率、如何最小化 token 浪费、如何在长对话中保持一致性——这已经形成了一门新学科：Context Engineering。2025-2026 年，这可能是 AI Agent 领域最被低估的技能。

Context Engineering 是什么

Context Engineering 不是 Prompt Engineering。Prompt Engineering 关心的是"对模型说什么能让它表现更好"，而 Context Engineering 关心的是"如何组织整个对话结构以最大化系统效率"。它位于 Prompt Engineering 和 Systems Design 的交叉点上。

具体来说，Context Engineering 要解决以下几个问题：

  • 缓存效率：如何安排消息顺序使得尽可能多的内容被缓存命中？
  • 断点放置：在上下文的哪些位置放置 cache_control 断点能最大化效用？
  • 预热策略：在正式对话前发送什么消息可以"预热"缓存，降低后续请求的实际成本？
  • 对话压缩：当上下文过长时，如何压缩历史而不丢失关键信息？
  • Token 预算管理：在有限的上下文窗口中，如何分配 System Prompt、工具定义、对话历史、工具结果的 token 配额？

这些问题的背后是一条根本原则：上下文是有限的，而且很贵。Anthropic API 每百万输入 token 收费 $3（标准）/$0.30（缓存读），输出 token 收费 $15。在一个典型的 Agent session 中，你可能发送了 50 万输入 token——其中 92% 可能被缓存命中，但剩下的 8% 加上输出部分，运行一天可能就消耗 $50-100。如果缓存命中率从 92% 降到 80%，成本直接翻倍。

Prefix 复用率 92% 的秘密

Claude Code 能达到约 92% 的缓存读取率，这来自于一个精心的设计。先看看一个典型的 Agent 请求上下文中包含哪些内容：

  1. System Prompt — 约 4-6K tokens，基本不变
  2. Tool Definitions — 约 8-12K tokens，随注册的工具数量变化，但通常在一个 session 内稳定
  3. Assistant 消息 — 模型生成的思考过程，逐轮增长
  4. User 消息 — 用户的指令，每轮至少一条
  5. Tool Results — 工具执行结果，可能是最长的一部分（文件内容、命令输出等）

92% 的缓存命中率意味着什么呢？在一个 100K token 的请求中，92K 是缓存读取（不产生实际计算成本，只收 10% 的缓存服务费），只有 8K 是新的非缓存内容（全额收费）。

做个对比：如果你没有做任何缓存优化，每次请求 100K token 全价收费，成本是缓存优化方案的 5-8 倍。一个月的账单从 $3000 降到 $500，这就是 Context Engineering 的力量。

Claude Code 实现高缓存命中率的关键策略：

  • 稳顶原则：System Prompt 永远放在最前面，而且内容在 session 内不变化。这确保了每次请求的前 4-6K tokens 始终命中缓存。
  • 工具定义缓存块：工具定义紧随 System Prompt，作为一个整体缓存块。只要工具注册表不变化，这个块也全部命中缓存。
  • 增量更新：只有最新的几轮对话消息和工具结果是未缓存的新内容。历史对话缓存随着时间自然滑动。

预热调用的设计巧思

Claude Code 有一个很少被注意到的设计：在用户发送第一条消息之前，它会先自动发送一个预热调用（warmup call）。这个调用使用 Claude Haiku——最便宜也最快的模型——目的不是为了完成任务，而是为了：

  • 初始化缓存：让 System Prompt 和工具定义在后续的真实请求中命中缓存。
  • 检测 API 配额：检查 API Key 是否有效、是否有足够的配额执行任务。
  • 探测网络连通性：确认 API 端点可达、网络延迟在正常范围内。
  • 检测对话主题（可选）：分析用户的工作目录，推断可能的任务类型。

为什么用 Haiku 而不是 Sonnet 做预热？原因很简单：性价比。预热调用不需要复杂的推理能力——它只需要加载上下文、触发缓存。Haiku 的成本是 Sonnet 的 1/5，速度却差不多。而且 Haiku 的缓存 TTL 虽然比 Sonnet 短（约 5 分钟 vs 10 分钟），但预热的目的只是为下一个请求"暖场"，5 分钟完全够用。

这个预热设计告诉我们一个重要的原则：缓存不是天然生效的，需要主动"种"进去。预热调用是在缓存系统中"播种"，让后续请求能收获。

Cache 断点自动放置算法

Anthropic API 最多支持在每个请求中放置 4 个 cache_control breakpoints。Claude Code 需要决定这 4 个断点放在哪里能最大化缓存效用。这是一个典型的资源分配问题。

断点放置策略：

  • System Prompt 末尾（必须）：这是第一个也是最稳定的断点。后续内容变化时，之前的内容仍然完全缓存。
  • Tool Definitions 末尾（高度推荐）：工具定义通常在 session 内不变，是第二个断点的理想位置。
  • 每 N 轮用户消息后（策略性）：取决于配置的轮数间隔，通常在 3-5 轮后放一个断点。这样历史对话的一部分可以被缓存。
  • 压缩前（可选）：在上下文压缩操作前放置断点，确保压缩后的摘要能被缓存。

这里有一个关键的约束：Anthropic API 要求 cache_control breakpoints 严格按照从 true 到 false 的顺序放置。也就是说，一旦某个位置的 cache_control 设为 false（非缓存），后面的内容就不能再出现 true。这其实很合理——API 需要确定一个清晰的"缓存前缀"范围。

以下是一个简单的断点优化器实现：

from dataclasses import dataclass
from typing import List, Optional


@dataclass
class CacheBreakpoint:
    """A candidate position for a cache_control breakpoint."""
    location: str  # human-readable label
    index: int     # position in the messages list
    estimated_tokens: int
    stability: float  # 0-1, how likely this content is to change


class CacheBreakpointOptimizer:
    """Places cache_control breakpoints optimally subject to API limits."""

    MAX_BREAKPOINTS = 4

    def __init__(self, messages: List[dict]):
        self.messages = messages
        self._candidates: List[CacheBreakpoint] = []

    def scan(self) -> List[CacheBreakpoint]:
        """Identify all candidate breakpoint positions."""
        self._candidates = []
        for i, msg in enumerate(self.messages):
            role = msg.get("role", "")
            tokens = msg.get("estimated_tokens", 0)
            if role == "system" and i == 0:
                self._candidates.append(
                    CacheBreakpoint("system_prompt", i, tokens, 1.0)
                )
            elif role == "tool" and tokens > 2000:
                self._candidates.append(
                    CacheBreakpoint(f"tool_result_{i}", i, tokens, 0.8)
                )
            elif role == "user" and i > 0:
                self._candidates.append(
                    CacheBreakpoint(f"user_message_{i}", i, tokens, 0.6)
                )
        return self._candidates

    def optimize(self) -> List[int]:
        """Select up to MAX_BREAKPOINTS breakpoints maximizing stability."""
        candidates = self.scan()
        # Prefer stable, large blocks first; then fill remaining slots
        candidates.sort(key=lambda x: (x.stability, x.estimated_tokens),
                        reverse=True)
        selected = candidates[:self.MAX_BREAKPOINTS]
        selected.sort(key=lambda x: x.index)
        return [bp.index for bp in selected]

    def apply(self) -> List[dict]:
        """Return messages with cache_control breakpoints inserted."""
        indices = self.optimize()
        result = []
        for i, msg in enumerate(self.messages):
            result.append(msg)
            if i in indices:
                result[-1]["cache_control"] = {"type": "ephemeral"}
        return result

实际 Claude Code 的放置算法更复杂——它会考虑消息的预估 token 大小、历史稳定性、再加上一些启发式规则（比如 system prompt 和 tool definitions 是最高优先级的断点位置）。但核心思想是相同的：用有限的断点槽位覆盖尽可能多的稳定内容。

缓存失效陷阱：如果工具定义在 session 中发生变化（比如新的 MCP Server 被添加），之前放在 tools definitions 末尾的缓存断点就完全失效了。整个前缀（system prompt + old tool definitions）需要重新计算。这就是为什么在生产环境中，Claude Code 会尽量保持工具注册表的稳定性，只在必要时才更新。对于重度用户来说，建议在添加或移除 MCP Server 后主动触发一次"缓存预热"，让新内容进入缓存。

HuggingFace 社区分析文章解读

2025 年底，HuggingFace 社区出现了几篇对 Claude Code 上下文模式进行深度逆向分析的文章，作者通过分析 Claude Code 的通信日志，还原了它的消息结构设计。几个核心发现：

  • 消息结构分析：每个请求包含约 8-15 条消息，其中 System Prompt 始终在第 0 位。工具结果占据了总 token 的 40-60%，是内容的大头。
  • Token 分布分析：令人惊讶的是，工具结果消耗的 token 远多于模型输出。在一个典型的 50 轮对话中，工具结果占用了约 70% 的 token。这意味着，优化缓存命中率的关键不在于让模型"少说话"，而在于减少重复性的工具结果。
  • 缓存命中率衰减：随着对话轮数增加，缓存命中率会缓慢下降（从 95% 逐渐降到 85% 左右），因为历史对话越长，新的非缓存内容越多。Claude Code 通过定期压缩历史来对抗这个衰减。

其中一个最有价值的发现是：大多数 token 花在了工具结果上，而不是模型输出上。这告诉我们，如果你想省 token，重点不是让模型少思考，而是优化工具返回的数据量——比如在 read_file 时只读取相关片段而不是整文件，在 bash 命令中限制输出长度。

不同模型的 Cache 行为差异

不同 Claude 模型的缓存行为有显著差异，了解这些差异有助于设计合理的上下文策略：

模型 缓存 TTL 每轮缓存衰减 最佳断点数 性价比
Claude Haiku ~5 分钟 较高（小模型缓存浅） 2-3 便宜但缓存窗口短
Claude Sonnet ~10 分钟 低（最稳定） 3-4 最佳性价比，推荐用于 Agent
Claude Opus ~10 分钟 低（缓存效果好） 4 最贵但绝对节省最大

Sonnet 的缓存行为是最可预测的。它不像 Haiku 那样容易因为模型负载波动而刷新缓存，也不像 Opus 那样成本高昂。对于大多数 Agent 应用场景，Sonnet 是缓存效率和推理质量的 sweet spot。

有趣的一点是：虽然 Opus 的每 token 成本更高，但它的缓存命中率也是最高的（因为用户更倾向于在复杂任务中使用 Opus，而复杂任务往往有更长的稳定前缀）。这意味着 Opus 用户的绝对节省数额最大——可能每月 $500-1000。

各消息类型的缓存命中率统计：

消息类型 平均 Token 占比 缓存命中率 优化建议
System Prompt 5% ~99% 保持稳定，不要动态修改
Tool Definitions 8-12% ~95% 批量更新，减少碎片化变更
Assistant Messages 15-20% ~80% 定期压缩历史对话
User Messages 5-10% ~60% 精简提示，去除非必要内容
Tool Results 55-70% ~85% 限制输出长度，使用增量读取
Context Engineering 可能是 2025-2026 年 AI Agent 领域最被低估的技能——它不性感，不涉及炫酷的模型或复杂的算法，但它直接决定你的 API 账单是 100 美元还是 1000 美元。一个优秀的 Context Engineer 和一个普通开发者之间的差距，可能比模型本身的代际差距还要大。花时间优化你的上下文结构，是你能为自己的 Agent 系统做的最有性价比的投资。
