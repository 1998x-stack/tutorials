第27章 软件无线电 (SDR)

同学们，我先问一个有点"暴露年龄"的问题：你们中有多少人见过真正的"对讲机"？不是微信语音，不是手机通话，而是那种按下PTT键、对着话筒喊"Over"的无线电设备？

传统无线电设备是完全由硬件定义的——每一个频段、每一种调制方式、每一种滤波特性，都由电路板上固定的电阻、电容、电感、混频器和滤波器决定。你要换一种通信制式？对不起，重新设计一块PCB。这个局面在1990年代被一位叫Joe Mitola的工程师彻底颠覆，他提出了一个今天看来理所当然但当时极为激进的想法：为什么不用软件来定义无线电的功能？

这就是软件无线电 (Software Defined Radio, SDR)的起点。它把尽可能多的信号处理功能从硬件搬到了软件中——ADC尽可能靠近天线，之后的滤波、调制/解调、编解码等所有操作都在通用处理器或FPGA上用代码完成。2026年的今天，一块不到200元人民币的RTL-SDR加密狗就能接收10kHz到1.7GHz的几乎所有民用无线电信号，而一套完整的GNU Radio+USRP B210开发平台则可以让你在一台笔记本电脑上搭建出4G基站。

27.1 SDR的核心理念

SDR的核心可以浓缩成一句话：尽可能将信号链路数字化，在软件中完成传统硬件的功能。传统超外差接收机和SDR接收机的结构对比一目了然：

环节 传统硬件无线电 软件无线电 (SDR)
天线接收 天线 → 带通滤波器 天线 → 带通滤波器
频率变换 多级混频器 + 本振 (LO) + 中频滤波器 单级混频器(可选)，或直接射频采样
模数转换 在中频 (IF) 或基带进行ADC 尽可能靠近天线——射频ADC或零中频
解调 专用解调芯片 (如FM、AM解调器) 软件实现：AM/FM/PSK/QAM全部可切换
滤波 晶体滤波器、LC滤波器 数字FIR/IIR滤波器（系数可编程）
解码 专用解码芯片 纯软件解码器（可OTA升级）
升级 换硬件 更新软件/FPGA比特流

SDR的"理想"架构是直接将天线连接到ADC——没有混频器、没有本振、没有中频。实际中受限于ADC的采样率和动态范围，这个理想一直在逼近但还未完全实现。目前主流的SDR方案是零中频 (Zero-IF)架构：射频信号经过LNA放大后直接正交下变频到基带I/Q两路，再分别进入ADC采样。

做SDR开发的第一步是选硬件。从入门到专业级，SDR硬件平台的跨度非常大：

27.3 GNU Radio入门

GNU Radio是SDR世界中最主流的开源开发框架。它提供了一个"流图 (Flow Graph)"编程模型——你把各种信号处理模块（信号源、滤波器、解调器、分析器、信宿）像乐高积木一样拖拽连接，然后系统会自动调度数据流在模块间传递。

一个最简单的GNU Radio流图长这样：

  [信号源] ──▶ [处理模块1] ──▶ [处理模块2] ──▶ [信宿]
  (Source)       (Block)         (Block)        (Sink)

数据在流图中以采样点的"流"的形式传递，每个模块按照自己设定的"产出率"消费上游的数据、生产下游的数据。GNU Radio运行时会自动处理各个模块之间的速率匹配——如果一个模块每秒产生1000个采样点，另一个模块每秒消耗2000个，GNU Radio会在中间插入自动插值/抽取逻辑。

实战：用GNU Radio + RTL-SDR接收FM广播

我们来做第一个真正意义上的SDR实验：接收本地的FM广播电台。完整流图如下：

┌─────────────────────────────────────────────────────────────┐
│                    FM广播接收流图 (GNU Radio)                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────┐    ┌──────────────┐    ┌──────────────────┐   │
│  │ RTL-SDR  │───▶│  Rational    │───▶│  WBFM Receive   │   │
│  │ Source    │    │  Resampler   │    │  (宽带FM解调器)   │   │
│  │  (硬件)   │    │  (重采样)    │    │                  │   │
│  └──────────┘    └──────────────┘    └────────┬─────────┘   │
│          2.4 Msps       → 480 ksps           │              │
│                                                │             │
│                ┌───────────────────────────────┘             │
│                ▼                                             │
│  ┌──────────────────────┐    ┌────────────────────────────┐  │
│  │  Rational Resampler  │───▶│  Audio Sink (声卡输出)     │  │
│  │  (480→48 kHz)        │    │  (听FM广播!)              │  │
│  └──────────────────────┘    └────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────┐                               │
│  │  QT GUI Frequency Sink   │  ← 频谱可视化（调试用）       │
│  └──────────────────────────┘                               │
└─────────────────────────────────────────────────────────────┘

各模块参数的关键设置：

  • RTL-SDR Source: Ch0 = 本地FM电台的频率（例如北京地区中央人民广播电台FM 106.1 MHz），Sample Rate = 2.4 Msps，Ch0 Gain = 30-40 dB（增益过高会导致ADC削波失真）。
  • 第一级重采样 (2.4 Msps → 480 ksps): FM广播信道的带宽通常为200kHz，480 ksps的I/Q数据已足够。降低采样率可减少后续计算负载。
  • WBFM Receive: 宽带FM解调器。关键参数：Quadrature Rate = 480 kHz，Audio Decimation = 1，Deviation = 75 kHz（标准FM偏差）。
  • 第二级重采样 (480 kHz → 48 kHz): 将解调后的音频采样率降到声卡支持的48 kHz。
  • Audio Sink: 声卡输出。注意Sample Rate = 48 kHz要与上游匹配。

你可能觉得流图看起来有点复杂，但它反映了一个非常重要的SDR理念：每一级处理都有明确的速率和功能，模块之间通过采样率变换连接。这和我们之前学过多速率信号处理（第12-13章）是直接对应的——抽取、插值、CIC滤波器这些概念都在GNU Radio的模块里得到了实际应用。

🎯 第一次听到的瞬间：我至今记得第一次把GNU Radio流图跑通、从声卡里听到FM广播的那个瞬间——一种奇妙的"物理世界数字化"的感觉涌上来。你能在频谱图上看到电台信号随着音乐旋律波动的包络，能手动调整增益看噪声如何淹没信号——这种掌控感是普通FM收音机完全无法提供的。强烈推荐每个DSP学习者都买一个RTL-SDR玩一玩。

27.4 从GNU Radio到FPGA：在Zynq上部署实时OFDM收发机

GNU Radio很好，但它的处理是在PC的CPU上运行的。对于低延迟、高吞吐率的通信系统（如实时OFDM收发、4G/5G原型），CPU上的软件处理速率跟不上。这时候就要把计算密集型的DSP处理搬到FPGA上。

Zynq + AD9361是工业界最流行的SDR硬件组合之一：

  • Zynq (Xilinx)：在单芯片上集成了ARM Cortex-A处理器（PS）和FPGA可编程逻辑（PL）。
  • AD9361 (ADI)：高性能射频前端，覆盖70 MHz - 6 GHz，支持最大56 MHz瞬时带宽，内置12位ADC/DAC、自动增益控制（AGC）、正交校正等。

一个典型的Zynq SDR系统中，DSP任务的分配是这样的：

           ┌─────────────┐
           │  天线 + AD9361 │  ← 射频前端：LNA、混频器、ADC/DAC
           └──────┬──────┘
                  │ I/Q数据流 (12bit, 并行)
                  ▼
┌──────────────────────────────────────────────┐
│           Zynq FPGA  (PL)                     │
│                                               │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐  │
│  │ DDC/DUC │─▶│  FIR滤波  │─▶│ FFT/IFFT   │  │
│  │  (CIC)  │  │  (整形)   │  │ (OFDM核心) │  │
│  └─────────┘  └──────────┘  └──────┬─────┘  │
│                                     │        │
│  ┌─────────┐  ┌──────────┐  ┌──────┴─────┐  │
│  │ 维特比   │  │ QAM映射  │◀─│ 同步/信道   │  │
│  │ 译码器   │  │ /解映射  │  │ 估计(Schmidl│  │
│  └────┬────┘  └──────────┘  │ -Cox)      │  │
│       │                     └────────────┘  │
└───────┼──────────────────────────────────────┘
        │ AXI4-Stream (高速)
        ▼
┌──────────────────────────────────────────────┐
│     ARM Cortex-A (PS)                         │
│                                               │
│  上层控制软件                                 │
│  - MAC层调度                                  │
│  - 参数配置 (AGCI/OFFSET校正)                  │
│  - 网络协议栈 (IP/UDP)                          │
│  - 状态监控                                   │
└──────────────────────────────────────────────┘

注意到一个关键的设计原则：吞吐率敏感的处理放PL，控制敏感的处理放PS。FFT、FIR滤波、CIC抽取这些数据流密集的操作在FPGA上完成，MAC调度、网络协议和用户界面在ARM上运行。PL和PS之间通过AXI4-Stream协议交换I/Q数据，典型吞吐量可达数Gbps。

⚠️ 重要提醒：SDR的法律与道德边界

SDR给了你监听和发射几乎所有频率信号的能力——但这不意味着你可以随意使用。每个国家都有严格的频谱管制法规：

  • 中国：《中华人民共和国无线电管理条例》规定，任何无线电发射设备必须取得型号核准，个人不得擅自设置使用无线电电台。非法发射可处2-20万元罚款，情节严重可追究刑事责任。
  • 美国：FCC Part 15 / Part 97规定，SDR设备只能用于接收（如RTL-SDR），发射需有业余无线电执照（如HackRF在Ham频段使用）。
  • 欧盟：CE认证要求类似的限制。

安全准则：在合法频段接收信号（FM广播、航空、气象卫星等公开频段）通常没问题。发射前务必确认你拥有相应频段的授权（如业余无线电操作证）。不要干扰应急通信、航空管制、移动运营商基站等受保护频率——这既是法律问题，也是安全问题。

27.5 踩坑：RTL-SDR的晶振频率偏差——不解调前频率校准就看不到星座图

讲一个我亲身经历的糗事，也是SDR新手最容易踩的坑。

几年前我第一次拿到RTL-SDR，兴冲冲地想做一个简易的ADS-B接收机（解码飞机广播的航班位置信息）。按照网上的教程配置好dump1090，天线放在窗台，打开频谱分析一看——哇，一大片信号！频率扫描在1090 MHz附近确实有个尖峰。但是我运行解码程序，一个飞机都解不出来。

折腾了一整天，我甚至怀疑天线坏了、RTL-SDR坏了、电脑USB口供电不足……直到我偶然在网上看到一句话："RTL-SDR的晶振精度约为±50ppm，不解调前校准频率偏移，你永远看不到正确的星座图。"

50ppm是什么意思？在1090 MHz下，50ppm的晶振误差意味着实际接收频率可能偏移 1090e6 × 50e-6 = 54.5 kHz！ADS-B信号的每个信道带宽只有1 MHz，54.5 kHz的频偏足够让你的解调器完全误判——你会看到星座图上符号疯狂旋转，解码器报CRC错误，啥都解不出来。

校准前（无频率校正）                        校准后（PPB校正 = +42ppm）
┌──────────────────────────────┐       ┌──────────────────────────────┐
│                              │       │      ●                       │
│    ●           ●             │       │   ●     ●                    │
│       ●   ●   ●   ●         │       │  ●       ●                   │
│    ●   ●   ●   ●   ●   ●    │       │   ●     ●                    │
│       ●   ●   ●   ●         │  →    │      ●                       │
│    ●   ●   ●   ●   ●   ●    │       │                              │
│       ●   ●   ●   ●         │       │  (QPSK星座图，散点清晰)      │
│                              │       │                              │
│  (散点旋转模糊，无法解码)    │       │                              │
└──────────────────────────────┘       └──────────────────────────────┘

解决方案：RTL-SDR软件（如rtl_test、Gqrx）都提供了频率偏移校准功能。你用已知准确的信号（例如已知频率的FM广播电台，或GPS驯服后的标准频率源）测量出当前设备的ppm偏差，然后在SDR软件的配置里填入这个偏移值。

🔧 校准步骤：
  1. 下载安装 rtl-sdr 驱动和 rtl_test 工具。
  2. 在终端运行 rtl_test -p 1000000（每秒一次测量，持续60秒）。
  3. 程序会输出当前晶振的ppm误差值，例如 Relative error: +42 ppm。
  4. 在GNU Radio的RTL-SDR Source模块中，将 Frequency Correction (ppm) 参数设为 42。
  5. 重新运行流图——你会发现解码成功率从0%飙升到95%以上。

这件事给我的教训是：在SDR的世界里，"不准"是常态而不是异常。晶振漂移、温度变化、连接器损耗、天线阻抗不匹配——每一个环节都可能让你的信号处理结果完全跑偏。SDR的精髓不是"有一个完美的硬件"，而是"用软件来校正硬件的不完美"——频率偏移校准、DC偏移去除、I/Q不平衡补偿，这些全都是数字域的自适应信号处理问题，恰好是我们从第15章就开始学习的内容。

27.6 本章小结

这一章我们从SDR的核心思想出发，沿着"为什么做 -> 用什么做 -> 怎么做 -> 如何部署 -> 有何边界"的线索，完整地走了一遍软件无线电的工程生态：

  • SDR的理想架构是"ADC尽量靠近天线"，现实中的主流是零中频架构。
  • 硬件选择从入门级RTL-SDR到专业级USRP X410跨度巨大，按需选择。
  • GNU Radio用流图模型实现了可编程的无线电处理链路，FM广播接收是天然的入门实验。
  • Zynq+AD9361展示了"FPGA实时DSP + ARM高层控制"的工业级SDR架构。
  • 法律边界不容忽视：接收自由但发射受限。
  • 踩坑告诉我们频率校准是SDR的第一步，50ppm的晶振误差足以让一切解码失败。

SDR可以说是DSP所有核心知识的终极综合练兵场——采样定理、混频、数字下变频、FIR/CIC滤波、FFT、解调、同步、信道估计——这些你在前25章学到的每一个概念，在SDR里都被用到了极致。
