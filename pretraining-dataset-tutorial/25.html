开篇：你的数据集真的比别人的好吗？

大家好，我是你们的讲师。前两章我们分别用 Datatrove 和 NeMo Curator 搭建了数据清洗流水线，洗出了干净的数据。但有一个问题一直悬而未决：怎么证明你洗出来的数据是"更好"的？

也许你调整了质量过滤器的阈值，从 FineWeb 的 18 条规则中选择了不同的参数组合；也许你在多个数据源之间调整了配比，增加了代码数据的比例，减少了 Common Crawl 的占比。这些改动真的有效吗？还是只是"自我感觉良好"？

我见过太多团队在这个环节犯的错误了。有的团队花了两周时间精心调优了一套质量过滤规则，自我感觉非常满意，结果训练出来的模型和基线版本相比，benchmark 分数纹丝不动——他们调整的规则根本没有触及数据质量的要害。更惨的是一些团队，在未经验证的情况下大幅增加了某个数据源的占比，导致模型在关键能力（如推理）上出现了倒退。这一章我们就来系统地学习如何用代理模型训练、消融实验和下游评测来回答一个核心问题：我的数据方案到底有没有变好？

25.1 代理模型评估：小成本，大价值

在完整的预训练（动辄数月、耗资数百万美元）之前验证数据质量，最常用的方法就是代理模型。你不需要训练 70B 的巨型模型，只需要在候选数据上训练一个 1B-3B 参数的小模型，然后测量它的困惑度（Perplexity）和下游 benchmark 分数。

为什么代理模型有效？因为数据质量和模型大小之间的交互是层次性的：如果数据有根本性问题（如大量重复、噪声过多、质量不达标），小模型上会表现得比大模型更明显。小模型的容量有限，无法"克服"数据中的缺陷——数据好坏直接反映在 loss 曲线上。

import torch
from transformers import AutoTokenizer, AutoModelForCausalLM, Trainer, TrainingArguments
from datasets import load_dataset

def train_proxy_model(
    dataset_path: str,
    model_name: str = "HuggingFaceTB/SmolLM2-360M",
    output_dir: str = "./proxy_model",
    max_steps: int = 5000,
):
    """训练一个代理模型来评估候选数据集的质量"""

    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModelForCausalLM.from_pretrained(model_name)

    dataset = load_dataset("json", data_dir=dataset_path, split="train")

    def tokenize_fn(examples):
        return tokenizer(
            examples["text"], truncation=True, max_length=2048, padding=False
        )

    tokenized = dataset.map(tokenize_fn, batched=True, remove_columns=["text"])

    args = TrainingArguments(
        output_dir=output_dir,
        per_device_train_batch_size=8,
        gradient_accumulation_steps=4,
        max_steps=max_steps,
        logging_steps=100,
        save_steps=1000,
        learning_rate=3e-4,
        warmup_steps=500,
        bf16=True,
        ddp_find_unused_parameters=False,
    )

    trainer = Trainer(
        model=model,
        args=args,
        train_dataset=tokenized,
    )

    # 训练并记录 loss 曲线
    trainer.train()

    # 返回最终验证 loss 作为数据质量的代理指标
    eval_results = trainer.evaluate()
    return eval_results["eval_loss"]


# 对比两个候选数据集
loss_v1 = train_proxy_model("/data/dataset_v1/")
loss_v2 = train_proxy_model("/data/dataset_v2/")

print(f"数据集 V1 最终 loss: {loss_v1:.4f}")
print(f"数据集 V2 最终 loss: {loss_v2:.4f}")

if loss_v2 < loss_v1:
    print("数据集 V2 质量更优，推荐用于正式训练！")
else:
    print("数据集 V1 质量更优。")
代理模型的陷阱：代理模型的 loss 和最终大模型的性能之间存在相关性，但不是 100% 正相关。小模型对某些数据特征（如反复重复、模板化文本）更敏感，这可能导致你错误地拒绝一些对大模型有益的数据（如多样化的长尾内容）。建议同时跟踪多个不同大小的代理模型（360M、1B、3B），取它们的共识结论。

25.2 数据组成消融实验

消融实验（Ablation Study）是数据配比调整中最强大的工具。它的核心思路是：在控制其他因素不变的前提下，系统地移除或改变单一数据源，观察模型性能的变化，从而推断该数据源的贡献。

一个经典的数据消融实验设计如下：

实验组 数据组成 回答的问题
基线组 完整数据（100%） 预期的最佳性能
去网页组 完整数据 - Common Crawl 网页数据贡献了多少能力？
去代码组 完整数据 - GitHub 代码 代码数据对推理能力的影响？
去书籍组 完整数据 - 书籍语料 长文本数据是否必要？
去重增强组 完整数据 + 更严格的去重 更严格的去重是否能提升质量？
质量过滤增强组 完整数据 + 更严格的质量过滤 更激进的过滤是否会丢失有用信息？

每组实验训练一个代理模型，然后在相同的 benchmark 集上评测。一般来说，每个实验做 2-3 个随机种子取平均，以避免单次训练的噪声干扰。

def run_ablation_experiment(
    data_configs: list[dict],
    proxy_model: str = "HuggingFaceTB/SmolLM2-360M",
    benchmarks: list[str] = ["mmlu", "gsm8k", "humaneval"],
):
    """对多个数据配置方案进行消融实验"""

    results = {}
    for config in data_configs:
        config_name = config["name"]
        dataset_path = config["path"]

        print(f"====== 运行消融实验: {config_name} ======")

        # 训练代理模型
        loss = train_proxy_model(
            dataset_path=dataset_path,
            model_name=proxy_model,
            output_dir=f"./models/{config_name}",
            max_steps=5000,
        )

        # 在 benchmark 上评测
        benchmark_scores = evaluate_on_benchmarks(
            model_path=f"./models/{config_name}",
            benchmarks=benchmarks,
        )

        results[config_name] = {
            "loss": loss,
            **benchmark_scores,
        }

    return results

# 执行消融实验
configs = [
    {"name": "baseline", "path": "/data/baseline/"},
    {"name": "no_web", "path": "/data/ablation/no_web/"},
    {"name": "no_code", "path": "/data/ablation/no_code/"},
    {"name": "strict_dedup", "path": "/data/ablation/strict_dedup/"},
    {"name": "strict_quality", "path": "/data/ablation/strict_quality/"},
]

results = run_ablation_experiment(configs)

# 分析结果
for name, scores in results.items():
    print(f"{name}: loss={scores['loss']:.4f}, "
          f"MMLU={scores['mmlu']:.2f}, "
          f"GSM8K={scores['gsm8k']:.2f}, "
          f"HumanEval={scores['humaneval']:.2f}")

25.3 质量指标与下游性能的相关性

我们在第 13 章介绍了各种基于模型的质量评分方法（Perplexity 评分、FastText 分类器、BERTScore 等）。但有一个重要问题：这些质量分数和下游模型性能之间到底有多大相关性？

基于多组消融实验的数据，我们可以建立质量分数与下游性能之间的相关性分析：

质量指标 与 MMLU 的相关系数 与 GSM8K 的相关系数 与 HumanEval 的相关系数 计算开销
Perplexity（GPT-2 打分） 0.52 0.48 0.31 低
FastText 质量分类器 0.61 0.55 0.43 低
FineWeb 启发式规则得分 0.58 0.53 0.39 极低
LLM-as-a-Judge 评分 0.73 0.68 0.59 高
合成指标（多指标加权） 0.78 0.72 0.64 中

从表中我们可以看出几个重要规律：

  • 单一指标远远不够——Perplexity 与数学推理的相关性只有 0.48，这意味着单靠 Perplexity 筛选数据，可能对推理能力的提升帮助不大。
  • 合成指标效果最好——将多种质量分数（启发式规则 + 分类器 + 困惑度）加权组合，可以得到与下游性能最相关（~0.78）的单一质量分数。
  • LLM-as-a-Judge 相关度最高但代价太大——如果用大模型逐篇评估每篇文档的质量，处理 TB 级数据的成本是不可接受的。通常只在构建"黄金验证集"时使用，用于标定其他轻量级指标。
实用建议：在实际项目中，我们采用"两阶段"策略——先用 FineWeb 启发式规则 + FastText 分类器做第一轮快速过滤（过滤掉最差的 30%-50%），然后对通过第一轮的数据，用合成指标进行第二轮排序和筛选。这样既控制了计算成本，又实现了较高的质量保障。最终选出的数据会在代理模型上验证后再投入正式生产。

25.4 训练过程中的 Perplexity 监控

数据质量不仅要在训练前评估，更要在训练过程中持续监控。Perplexity 曲线是最直观的质量信号：

  • Loss 不下降或下降极慢 → 数据中可能混入了大量非自然语言噪声（如未清洗干净的 HTML 片段、乱码文本）
  • Loss 突然升高 → 可能是数据混入了不匹配分布的内容（如领域偏差或语言漂移）
  • Loss 下降到某个点后震荡不降 → 可能是数据多样性不足，模型已经"学完"了数据中的模式
  • 验证 Loss 在某个点后开始上升 → 明确的过拟合信号，数据可能重复率过高或严重不足

一个有效的监控实践是设置"验证集全景（validation corpus）"——从干净的验证集（如 FineWeb 的验证分割、经过人工标注的高质量样本）中固定一个子集，在训练过程中定期（每 100-1000 步）计算验证 Perplexity。这个验证集的数据不应该和训练集有任何重叠，以确保 Perplexity 的测量是无偏的。

25.5 下游 Benchmark 与人工评估

Perplexity 下降并不总是意味着模型在变"好"。有时 Perplexity 下降只是因为模型过拟合了训练数据的统计模式，而下游任务能力反而停滞或下降。因此，定期在下游 benchmark 上评测是必不可少的。

对于中文大模型，推荐的标准评测套件包括：

  • C-Eval / CMMLU — 中文综合知识评测
  • GSM8K / MATH — 数学推理能力
  • HumanEval / MBPP — 代码生成能力
  • MMLU — 英文综合知识评测
  • HellaSwag / WinoGrande — 常识推理

但 benchmark 评测有一个众所周知的局限——数据污染。如果训练数据中混入了上述 benchmark 的测试样本，评测分数会虚高。因此，除了标准 benchmark，强烈建议补充人工评估作为质量验证的最终防线：

# 人工评估样本抽取与评分框架

def prepare_human_eval_samples(
    model_outputs: list[dict],
    num_samples: int = 200,
    sampling_strategy: str = "stratified",
):
    """准备人工评估样本，按能力维度分层抽样"""
    samples = []
    dimensions = ["general_knowledge", "reasoning", "code", "math", "creative"]
    for dim in dimensions:
        dim_samples = [s for s in model_outputs if s["dimension"] == dim]
        # 按难度分层
        easy = [s for s in dim_samples if s["difficulty"] == "easy"]
        medium = [s for s in dim_samples if s["difficulty"] == "medium"]
        hard = [s for s in dim_samples if s["difficulty"] == "hard"]
        # 每层均匀抽取
        per_level = num_samples // (len(dimensions) * 3)
        samples.extend(np.random.choice(easy, per_level))
        samples.extend(np.random.choice(medium, per_level))
        samples.extend(np.random.choice(hard, per_level))
    return samples

在我参与的一个项目中，我们发现某个数据配比方案在 MMLU 和 GSM8K 上都取得了最好成绩，团队几乎决定采用这个方案。但人工评估发现，模型在用这个数据方案训练后，生成的文本在事实性和一致性上明显不如另一个配比方案——虽然那套方案的 benchmark 分数低了 2-3 个百分点。最终我们选择了后者，因为基准测试的虚高分数不值得牺牲实际用户体验。

血的教训：永远不要只看 benchmark 分数做决策。2024 年有一篇被广泛讨论的论文发现，通过简单的数据重复，可以将一些 benchmark 的分数提升 5-10 个百分点，但模型在实际使用中的表现反而下降了。Benchmark 是信号，不是目标。数据质量的最终衡量标准，是模型在实际应用场景中的表现。

25.6 持续改进循环：识别失败模式，修复数据，重新训练

数据质量优化不是一次性的工作，而是一个持续改进的循环：

  1. 训练基线模型，在更多测评维度上建立全面的性能画像
  2. 分析失败案例——在哪些任务上表现差？生成结果有什么系统性缺陷？
  3. 回溯数据根因——这些失败是否是数据问题导致的？缺少某个领域的数据？某种模式的数据质量太差？
  4. 针对性改进数据——补充缺失的数据源，改进对应领域的过滤规则，调整配比权重
  5. 重新训练并验证——用新数据训练，确认修复措施是否有效

举个例子：我们在一次评测中发现，模型在回答与"法律"相关的中文问题时表现很差——给出的信息往往过时或不准确。回溯分析发现，训练数据中的法律相关内容主要来自新闻网站和论坛讨论，而不是专业的法律文档。于是我们做了三件事：一是补充了一批高质量的法律条文和司法解释数据；二是过滤掉了法律领域中明显不可靠的论坛帖子（通过域名黑名单 + 关键词过滤）；三是提高了法律数据在训练时的采样权重。经过两轮迭代后，模型在法律问答上的准确率提升了近 20 个百分点。

关键原则：数据质量改进和软件工程中的 bug 修复是一样的——要先定位根因，再修复，最后验证。不要盲目地"多加数据"或"收紧过滤"，而要基于失败模式做精准改进。一个针对性修复常常比十次盲目调整效果更好。

本章小结

数据质量不是靠感觉来衡量的，而是要靠系统性的实验来验证。代理模型让你用最小的成本评估候选数据集的质量；消融实验帮你理解每个数据源对模型能力的实际贡献；Benchmark 和人工评估构成了质量验证的"客观 + 主观"双重防线；持续改进循环则确保你的数据质量一直在迭代优化，而不是一次性的"洗好就完"。
