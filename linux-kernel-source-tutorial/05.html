大家好，我是你们的讲师。上一章我们跟着bootsect.S走完了引导的前半程——从BIOS拿到控制权、搬移自身、加载了setup和system模块。现在CPU已经跳转到了0x90200处的setup代码。这个位置，正处于16位实模式世界的"黄昏"——setup.S的任务只有一个：把CPU从实模式切换到保护模式。

这一切换，是x86体系结构中最精彩的部分。它需要精确操作各种控制寄存器、系统表、以及处理那些让人头疼的兼容性问题。但本质上，它只做三件事：准备GDT，设置CR0的PE位，远跳转刷新CS。让我们一步步来看。

引言——从实模式到保护模式，32位世界的入口

实模式是x86的"安全模式"——CPU启动后默认处于这个模式，以便兼容1978年的8086。在实模式下，CPU像一个快一点的8086：16位寄存器、1MB地址空间、没有内存保护、没有多任务支持。而保护模式才是x86真正的实力所在——32位寄存器、4GB地址空间、基于段的保护机制、分页、多任务。

Linux 0.11启动过程中，setup.S和head.S共同完成了从实模式到保护模式的切换，然后转身设置了分页机制。整个过程就发生在几百行汇编代码之中——它们就像是系统的"产房"，所有的核心机制都在这里从无到有地诞生。

实模式 -> 保护模式 -> 分页 -> C内核

  bootsect.S       setup.S         head.S          main.c
  (16-bit real)    (16-bit real)   (32-bit prot)   (C code)
       |               |               |              |
       v               v               v              v
  ──────────     ──────────      ──────────      ──────────
  加载内核       关中断, 获取     设置GDT/IDT,    初始化各
  到内存         内存信息,        开启A20门,      子系统,
                 ─────            设置分页         启动shell
                   ↓
                 切换保护模式

好，让我们进入代码。

setup.S源码分析

setup.S的第一件事，是借用BIOS中断获取系统的硬件信息。这些信息会被存放在内存中的固定位置（0x90000 ~ 0x901FF区域，也就是原来bootsect驻留的位置），供后续的main.c读取。

! setup.S —— 获取硬件信息的BIOS调用序列

! 1. 获取扩展内存信息 (通过INT 0x15, AX=0xE801)
!    得到连续内存的大小(以KB为单位)
!    结果存放在 0x90002 + 0x90004
    mov ax,#0xE801
    int 0x15
    mov [2],ax      ! 0x90002: 1MB以上的连续内存大小(KB)
    mov [4],bx      ! 0x90004: 扩展内存大小(KB)

! 2. 获取光标位置 (通过INT 0x10, AH=0x03)
!    存放在 0x90000 (0x90000=列, 0x90001=行)
    mov ax,#0x0300
    int 0x10
    mov [0],dx

! 3. 获取显示模式 (通过INT 0x10, AH=0x0F)
!    存放在 0x90006
    mov ah,#0x0F
    int 0x10
    mov [6],bx      ! 0x90006 = 显示页, 0x90007 = 显示模式

! 4. 获取第一个硬盘参数 (通过INT 0x13, AH=0x08)
!    存放在 0x90080 ~ 0x9008F
    mov ax,#0x0800
    int 0x13
    ...  ! 填充硬盘参数表

! 5. 获取键盘重复率等

这些BIOS调用不需要完全记住——你只需要知道它们的存在。后续在main.c中，会通过setup.h中的宏来读取这些参数。它们共同构成了内核的"硬件探测"阶段——在一个每次启动可能硬件配置都不同的PC上，这是必要的第一步。

获取完硬件信息后，setup.S做了一件关键的事：

! 关中断 —— 进入保护模式之前必须关中断
    cli     ! Clear Interrupt Flag —— 禁止所有可屏蔽中断

! 将system模块从 0x10000 ~ 0x8FFFF 搬移到 0x00000
! 因为保护模式和分页启用后，中断向量表需要重新设置
! 把system放在0x00000开始的位置，后面head.S的代码才能正确执行
    mov ax,#0x0000
    cld             ! 方向标志清零(字符串操作正向递增)
do_move:
    mov es,ax       ! ES = AX (目标段地址)
    add ax,#0x1000  ! 每次搬移64KB (0x1000 * 16 = 64KB)
    cmp ax,#0x9000  ! 是否搬完了? (0x0000 ~ 0x9000 共9个段)
    jz  end_move    ! 如果到了0x9000，搬完了
    mov ds,ax       ! DS = AX (源段地址)
    sub di,di       ! ES:DI = 0x0000:0x0000
    sub si,si       ! DS:SI = 源地址
    mov cx,#0x8000  ! 搬移0x8000个字 = 64KB
    rep movw        ! 重复执行movw
    jmp do_move
end_move:
    ! 此时system模块已在0x00000位置

这个搬移非常关键！bootsect当初把system模块加载到0x10000，现在setup要把它搬到0x00000。为什么？因为分页机制启用后，内核期望自己位于物理地址0x00000（通过页表映射到虚拟地址0xC0000000或0x00000000，取决于具体实现）。而且，因为马上要重新设置IDT（中断描述符表），原来占据0x00000的中断向量表可以被覆盖了。

搬移为什么不直接使用memcpy：因为此时还没有C运行时环境——没有栈、没有全局变量、没有任何C库函数。你只能使用x86汇编的rep movw这种最基本的块传输指令。这也解释了为什么内核的引导代码必须用汇编来写——C语言的运行时环境还没有准备好。

进入保护模式

system模块搬移完成之后，setup.S开始准备进入保护模式。这是整个启动流程中技术含量最高的部分。

在切换之前，我们来总结一下setup.S前半段所获取的硬件信息，这些信息会在main.c中被读取：

BIOS调用获取的信息存放地址main.c中的读取方式
INT 0x15, AX=0xE801扩展内存大小 (KB)0x90002, 0x90004EXT_MEM_K 宏
INT 0x10, AH=0x03光标位置 (行列号)0x90000SCREEN_INFO 结构体
INT 0x10, AH=0x0F显示模式 (页号/模式)0x90006ORIG_VIDEO_MODE 宏
INT 0x13, AH=0x08硬盘参数 (柱面/磁头/扇区)0x90080 ~ 0x9008Fhd_info 结构体

这些信息是整个系统的"硬件清单"——内核需要知道有多少内存、显示器的分辨率、硬盘的几何参数，才能正确驱动这些设备。

! 1. 加载临时GDT (全局描述符表)
!    这里的GDT非常简单，只包含必要的描述符
    lgdt    gdt_48      ! 加载GDT基址和限长

! 2. 加载临时IDT (中断描述符表) —— 空的，因为还没设置中断
    lidt    idt_48      ! 加载IDT基址和限长

! 3. 设置CR0寄存器的PE(Protection Enable)位
    mov ax,#0x0001
    lmsw    ax          ! Load Machine Status Word —— 设置CR0的PE位

! 4. 远跳转刷新CS —— 这是进入保护模式的"临门一脚"
    jmpi    0,8         ! 跳到 段选择子=8(即GDT的第2项)的代码段
                        ! 偏移量为0
                        ! 此时CPU以32位模式执行后续指令

! 关键数据结构: GDT (在setup.S末尾定义)
gdt:
    .word   0,0,0,0     ! 第0个描述符: 空描述符(必须为全0)
                        ! 第1个描述符: 代码段
    .word   0x07FF      ! 段限长 = 8MB (2047个4KB粒度段)
    .word   0x0000      ! 基址 = 0x00000000
    .word   0x9A00      ! 存在位=1, DPL=0, 代码段, 非一致, 可读
    .word   0x00C0      ! G=1(4KB粒度), D=1(32位), 段限长高4位=0
                        ! 第2个描述符: 数据段
    .word   0x07FF      ! 段限长 = 8MB
    .word   0x0000      ! 基址 = 0
    .word   0x9200      ! 存在位=1, DPL=0, 数据段, 可写
    .word   0x00C0

gdt_48:
    .word   0x07FF      ! GDT限长 = 2KB (最多256个描述符)
    .word   0x90200+gdt ! GDT基址 = 0x90200 + gdt的偏移
    .word   0x0000      ! 因为实模式段地址是0x9020(段基址*16=0x90200)

jmpi 0,8 这一行是进入保护模式的关键。段选择子8（二进制1000）的含义：

  • 位0（TI）= 0：使用GDT
  • 位1-2（RPL）= 0：请求特权级Ring 0
  • 位3-15（Index）= 1：GDT的第1项（代码段描述符）

这个jmpi执行后，CS寄存器的可见部分被刷新为GDT第1项描述符定义的段信息：基址0x00000000、限长8MB、32位代码段、Ring 0特权级。CPU从此进入了32位保护模式。

为什么远跳转能刷新CS？ 这是x86体系结构的一个关键细节。CR0的PE位置1后，CPU并没有立即进入保护模式——它还在用实模式的流水线执行代码。必须有一个远跳转（或远调用/远返回）来强制CPU刷新CS寄存器的"隐藏部分"（描述符缓存），重新从GDT加载段信息。这就是为什么jmpi是进入保护模式的"临门一脚"。

head.S——内核的头部

在setup.S末尾的jmpi 0,8之后，CPU跳到物理地址0x00000开始执行——这里存放的是system模块的第一部分，即boot/head.S。head.S是在32位保护模式下执行的第一个代码段，使用32位指令集和32位寻址。

! head.S —— 内核的"头部"，在32位保护模式下运行
! 所有寄存器都是32位的了 (EAX, EBX, ECX, ...)

startup_32:
    ! 1. 重新设置GDT（更加完善）
    lgdt    gdt_descr   ! 加载在head.S中定义的新GDT
    lidt    idt_descr   ! 加载IDT（此时IDT为空，没有中断处理程序）

    ! 2. 重新设置段寄存器
    mov ax,#0x10        ! 0x10 = 16 = GDT第2项(数据段选择子)
    mov ds,ax
    mov es,ax
    mov fs,ax
    mov gs,ax
    lss esp,stack_start ! 设置栈指针(在main.c中定义)

    ! 3. 检测A20门是否开启
    !    如果没有开启，CPU将陷入死循环——这是故意的
    xorl %eax,%eax      ! EAX = 0
1:  incl %eax           ! EAX = 1
    movl %eax,0x100000  ! 尝试写入1MB以上的地址
    movl 0x100000,%edx  ! 读回来
    cmpl %eax,%edx      ! 相等吗?
    je 1b               ! 相等 -> A20没开启(地址回绕) -> 死循环

    ! 4. 检测数学协处理器
    movl %cr0,%eax
    andl $0x80000011,%eax ! 保留ET, MP, EM, TS, PG位
    orl  $2,%eax          ! 设置EM位? (实际代码相反——启用协处理器)
    movl %eax,%cr0
    jmp  1f
1:  fdisn               ! 重置协处理器

    ! 5. 设置分页 (见下一节)
    ...

A20门检测这段代码很有设计感。如果A20门没有开启，写入0x100000（1MB以上的地址）的数据会被回绕到0x00000处。检测方法就是写一个不同的值到0x100000，然后读回来——如果读到的是之前0x00000位置的值（而不是刚写进去的值），说明A20门没有开启。如果这种情况发生，head.S直接进入死循环——与其在诡异的地址回绕状态下运行然后崩溃，不如在启动阶段就停下来告诉你"有问题"。

分页机制初始化

head.S最关键的初始化工作是分页机制。Linux 0.11使用两级页表：页目录(Page Directory)和页表(Page Table)。

! head.S —— 初始化分页

! 页目录和页表的位置(在head.S中定义)
! pg_dir = 0x0000 (页目录在物理地址0, 紧跟在head.S代码后面)
! pg0    = 0x2000 (第一个页表在物理地址0x2000)
! pg1    = 0x3000 (第二个页表在物理地址0x3000)

    ! 创建页目录pg_dir
    ! 页目录共1024项，每项4字节，占用4KB (0x0000 ~ 0x0FFF)
    movl $pg0+7,pg_dir      ! 页目录[0] = 第一个页表地址(0x2000) + 7
                            ! 位0(P)=1(存在), 位1(RW)=1(可读写)
                            ! 位2(US)=1(用户态可访问)
    movl $pg1+7,pg_dir+4    ! 页目录[1] = 第二个页表地址(0x3000) + 7
    ! 其余页目录项都置0(无效)
    movl $pg_dir+8,%ecx     ! ECX = 页目录项数 - 2
    movl $0,pg_dir+8(%ecx)  ! 清零
    ...

    ! 创建页表pg0 (映射 0x00000000 ~ 0x003FFFFF, 即低端4MB)
    ! 每个页表有1024项，每项4字节，映射4MB地址空间
    movl $pg0,%edi          ! EDI = 第一个页表地址
    movl $0x0007,%eax       ! EAX = 0x0007 (物理页地址0 + P+US+RW位)
1:  stosl                   ! 写入页表项并递增EDI
    addl $0x1000,%eax       ! 下一个物理页 (每页4KB)
    cmp eax,$0x400007       ! 是否到了4MB边界?
    jb 1b                   ! 没到就继续

    ! 同样的方式创建pg1 (映射 0x00400000 ~ 0x007FFFFF, 即第二个4MB)
    ! 内核实际只需要8MB映射空间

    ! 设置CR3(页目录基址寄存器)
    movl $pg_dir,%eax
    movl %eax,%cr3          ! CR3 = 页目录物理地址

    ! 设置CR0的PG位(分页启用)
    movl %cr0,%eax
    orl  $0x80000000,%eax   ! 设置PG位(位31)
    movl %eax,%cr0           ! 此处分页正式启用!

    ! 分页启用后，CPU立即开始使用地址转换

分页启用后，虚拟地址0x00000000 ~ 0x007FFFFF被映射到物理地址0x00000000 ~ 0x007FFFFF。内核在0x00000地址运行，而虚拟地址和物理地址此时是一一对应的（恒等映射）。这样做的好处是：分页刚刚开启时，不会因为地址翻译改变而立刻导致指令崩溃。

分页映射关系示意图:

  虚拟地址空间                   物理地址空间
  =============                  =============
  0x00000000 ───┐              0x00000000 ─── Head.S代码
  0x00001000 ───┤              0x00001000 ─── 更多代码
  ...           ├── 页目录[0]    ...           ├── 内核代码 + 数据
  0x00200000 ───┤  ────────→   0x00200000 ───┤   (system模块)
  ...           │  pg0页表     ...           │
  0x003FF000 ───┘              0x003FF000 ───┘
                               (4MB边界)
  0x00400000 ───┐              0x00400000 ─── 第二个4MB
  ...           ├── 页目录[1]    ...           │
  0x007FF000 ───┘  ────────→   0x007FF000 ───┘
                 pg1页表

跳转到main.c

分页和所有保护模式基础设施搭建完成后，head.S执行最后一步——跳转到C语言编写的main()函数。但在跳转之前，它需要做一件事：压栈参数。

! head.S —— 准备调用main.c

    ! 将setup.S收集的硬件参数压栈
    ! main(int argc, char *argv, char *envp) 的C调用约定
    pushl $0            ! envp = NULL
    pushl $0            ! argv = NULL
    pushl $0            ! argc = 0

    ! 实际上0.11的main()函数不通过栈接收参数
    ! 它通过全局变量直接读取setup收集的信息
    ! 所以这个压栈更像是"占位"

    ! 设置main函数的返回地址(虽然它永远不会返回)
    pushl $L6           ! 压入返回地址

    ! 跳转到main函数的入口
    movl main,%eax      ! main是main.c中main()函数的地址
    jmp *%eax           ! 跳转到main

L6:
    jmp L6              ! 如果main返回了, 死循环(这不应该发生)

注意这里使用了一种精巧的跳转方式：通过movl main,%eax; jmp *%eax而不是简单的call main。这是因为main是一个在C语言中定义的符号，其地址在链接时确定。head.S中通过声明main为外部符号来引用它。这种跳转方式保证了无论链接器把main放在什么地址，都能正确跳转到。

从这一刻开始，CPU执行从汇编代码移交给C语言——init/main.c中的main()函数。这是一个历史性的时刻：从此以后，所有的初始化工作都在易读、易维护的C语言中进行。

关键踩坑——A20门的历史包袱

A20门（A20 Gate）是x86历史上最著名的"兼容性包袱"之一。它背后的故事很好地诠释了"硬件兼容性如何影响软件设计"。

故事要从8086说起。8086有20根地址线（A0~A19），最大寻址1MB。当程序访问超过1MB的地址时（比如0x100000），地址会"回绕"（wrap around）到0x00000。这是8086的正常行为，很多早期程序确实依赖这个回绕特性。

到了80286，地址线增加到24根（A0~A23），可以访问16MB内存。但问题来了：一些古老的8086程序在访问0x100000时，本意是访问0x00000——它们依赖地址回绕。如果在286上不进行回绕，这些程序就会崩溃。IBM的解决方案是在键盘控制器上增加一个控制信号——A20 Gate——专门控制第20根地址线的启用和禁用。

当A20门关闭时，CPU物理地址的第20位（A20）被强制置为0。这样，即使程序访问0x100000（二进制1 0000 0000 0000 0000 0000），A20位被强制清0后，实际访问的是0x000000（二进制0 0000 0000 0000 0000 0000）。这就是"地址回绕"的硬件实现。

A20门的作用:
  A20=0: 第20根地址线强制为0
  虚拟地址 0x100001 → 物理地址 0x000001  (回绕了!)
  虚拟地址 0x200000 → 物理地址 0x000000  (回绕了!)

  A20=1: 所有地址线正常工作
  虚拟地址 0x100001 → 物理地址 0x100001  (正常)

开启A20门的方法(在setup.S中):
  inb     $0x92, %al    ! 读取端口0x92(键盘控制器端口)
  orb     $0x02, %al    ! 设置第1位(A20门控制位)
  outb    %al, $0x92    ! 写回——A20门开启
A20门未开启的症状：这是内核开发中最让人困惑的Bug之一。你的代码看起来完全正确，内核编译通过，但启动后出现各种诡异的行为——变量莫名其妙被修改、函数调用返回错误地址、内存内容被破坏。这些都是因为A20未开启导致高地址访问被回绕到了低地址，覆盖了关键数据。如果你在写内核时遇到这种"鬼打墙"式的问题，第一个应该检查的就是A20门是否正常开启。

head.S中的A20检测代码我已经在前面展示过了——它通过写一个值到0x100000然后读回来验证。这是一个简单但100%可靠的检测方法。Linux 0.11的A20代码虽然简单，但后来在Linux 2.x中，A20的开启逻辑变得非常复杂——因为有些BIOS已经开启了一些PC的A20门，有些没有，有些BIOS使用的方式不同。内核必须在各种可能的初始状态下正确识别并确保A20开启。

好了，汇编代码的漫长旅程结束了。从bootsect.S的512字节引导扇区开始，经过setup.S的保护模式切换，再到head.S的分页初始化——现在内核已经完全运行在32位保护模式下，分页机制正常运作，栈指针设置完毕，可以调用C函数了。
