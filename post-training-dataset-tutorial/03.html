第3章 数据格式与标准规范

大家好，我是你们的讲师。我有一个真实的故事想分享给你。

2024 年初，某公司训练团队集齐了堪称豪华的数据集——FLAN、Alpaca、ShareGPT、OpenAssistant，近 200 万条精心筛选的 SFT 数据。模型架构是当时最强的开源基座，训练算力也是顶配。所有人都期待着一个能与 GPT-4 媲美的模型诞生。

然后训练开始了。Loss 曲线像过山车一样忽上忽下。训练了三天，模型连"1+1 等于几"都答不对。团队第一反应是训练代码有 bug，排查了整整一周，从优化器到数据加载器，从学习率到分布式通信——全都没问题。

最后发现的问题很简单：四个数据集的字段名不一样。有的用 instruction，有的用 prompt，有的用 query。数据加载器读入后统一按 instruction 字段处理，结果三个数据集的 instruction 字段全部为空——模型实际上只在一个数据集上训练。故事虽然扎心，但它告诉我们一个铁律：数据格式不是小事，它是训练管线中最容易被忽视但影响最致命的环节。

3.1 SFT 标准格式详解

经典三字段格式：instruction/input/output

这是 SFT 领域最经典、应用最广泛的格式。它起源于 Stanford Alpaca 项目，后续被大量数据集沿用：

{
  "instruction": "翻译以下句子为英文",
  "input": "今天天气真好",
  "output": "The weather is great today."
}

三个字段的含义非常直观：instruction 描述任务指令，input 是任务的输入内容（可以为空），output 是期望的模型输出。这种格式特别适合单轮指令任务，在训练时通常会将 instruction 和 input 拼接作为模型的输入，output 作为预测目标。

多轮对话格式：messages 数组

为了支持多轮对话场景，业界逐渐转向更灵活的 messages 格式。每条消息都有 role 和 content 两个核心字段：

{
  "messages": [
    {"role": "system", "content": "你是一个乐于助人的AI助手。"},
    {"role": "user", "content": "请用 Python 写一个排序算法"},
    {"role": "assistant", "content": "以下是用 Python 实现的快速排序："},
    {"role": "user", "content": "能改成降序吗？"},
    {"role": "assistant", "content": "当然，只需要修改比较条件即可：..."}
  ]
}

messages 格式的优势在于能完整保留对话的上下文结构。训练时，模型需要根据之前的对话历史预测 assistant 的回复。系统提示词（system message）用于设定模型的行为准则，在多轮对话中尤为重要。

ShareGPT 格式 vs OpenAI Chat 格式

在 messages 格式的基础上，还有两种主流变体：

维度 ShareGPT 格式 OpenAI Chat 格式
典型字段 conversations / id / from / value messages / role / content
角色标识 "from": "human" / "gpt" "role": "user" / "assistant" / "system"
数据集来源 ShareGPT、LMSYS-Chat-1M 等对话采集 OpenAI API 数据、合成数据集
训练框架适配 需要自定义转换 大多数框架直接支持（如 Axolotl、LLaMA-Factory）
典型示例 [{"from": "human", "value": "你好"}, {"from": "gpt", "value": "你好！"}] [{"role": "user", "content": "你好"}, {"role": "assistant", "content": "你好！"}]
建议：如果你的项目从零开始，推荐直接采用 OpenAI Chat 格式（messages + role + content）。它是最广泛支持的格式，而且从 ShareGPT 格式转换到 OpenAI 格式比较简单，反之则要困难得多。

3.2 偏好数据格式详解

偏好数据的核心格式是 prompt/chosen/rejected 三字段：

{
  "prompt": "写一篇关于 AI 的 500 字文章",
  "chosen": "人工智能（AI）是当今科技领域最热门的话题之一...（500 字优质文章）",
  "rejected": "AI 很好很强大，是未来的趋势。（50 字敷衍回答）"
}

在多轮对话场景中，prompt 也可以是一个 messages 数组，表示完整的对话历史。

对于更精细的偏好标注，一些数据集采用了多维度扩展格式：

{
  "prompt": "如何快速赚到 100 万？",
  "chosen": {"content": "...", "rating": 5, "helpful": true, "harmless": true},
  "rejected": {"content": "...", "rating": 1, "helpful": false, "harmless": false},
  "dimensions": {
    "helpfulness": {"chosen": 5, "rejected": 1},
    "honesty": {"chosen": 4, "rejected": 2},
    "harmlessness": {"chosen": 5, "rejected": 3}
  }
}

这种多维标注格式让模型可以学习到更细粒度的偏好信号——它不仅能学会"哪个回答更好"，还能学会"好在哪个方面"。但要注意，多维标注显著增加了标注成本，通常只在核心数据集上使用。

3.3 格式转换实战

在实际项目中，你几乎总是需要将不同来源的数据统一为同一种格式。下面是一个使用 HuggingFace datasets 库进行格式转换的实战示例：

from datasets import load_dataset, Dataset

# 加载 ShareGPT 格式的数据
dataset = load_dataset("json", data_files="sharegpt_data.json")["train"]

def convert_sharegpt_to_messages(example):
    """将 ShareGPT 格式转换为 OpenAI Chat 格式"""
    messages = []
    role_map = {"human": "user", "gpt": "assistant"}
    for turn in example["conversations"]:
        messages.append({
            "role": role_map.get(turn["from"], turn["from"]),
            "content": turn["value"]
        })
    return {"messages": messages}

# 批量转换
converted = dataset.map(convert_sharegpt_to_messages)

# 验证转换结果
print(converted[0]["messages"][:2])
# [{"role": "user", "content": "你好"}, {"role": "assistant", "content": "你好！请问有什么可以帮助你的？"}]

# 保存为标准格式
converted.to_json("converted_data.json", orient="records")

这段代码虽然简单，但在实际项目中每天都要运行——数据团队不断从不同渠道采集新数据，统一的格式转换管线是确保数据质量的第一道关卡。

3.4 常见格式陷阱

格式问题最常见也最隐蔽。这里列出五个高发陷阱，建议你在数据预处理管线中逐一检查：

陷阱 现象 后果 检查方法
字段名不匹配 部分数据用 query 而不是 instruction 训练时该字段为空，数据被跳过 遍历所有样本检查字段名集合
多余空格 JSON 键或值前后有不可见空格 字段匹配失败，数据加载报错 字符串 .strip() 检查长度变化
缺少 role 标签 messages 数组中部分消息缺少 role 训练框架解析异常，进程崩溃 检查每条消息是否包含 role 字段
对话角色轮换错误 连续两条 user 消息没有 assistant 回复 模型学到错误的对话模式 验证 messages 数组 role 交替顺序
编码不一致 部分数据是 GBK 编码，部分是 UTF-8 中文字符乱码，训练 loss 异常 统一转换为 UTF-8 并检查非法字符
踩坑提示：角色轮换错误是最容易被忽视的问题。一个典型的场景是——从真实对话采集的数据中，用户说了两句话模型才回复。如果你不加处理地直接用，模型可能会学到"用户说两遍我才回答"的模式。正确的做法是：将连续同角色的消息拼接为一条消息，确保 user/assistant 严格交替。

数据格式看似是一个"低级"问题，但它直接影响训练能否正常进行。一个系统的数据预处理管线应该包含格式统一、字段校验、编码检查和异常样本过滤四个环节。在第 23 章数据质量管线搭建中，我们会详细讨论如何构建自动化格式检查系统。
