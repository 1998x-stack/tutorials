第22章：Sentinel 哨兵与 Cluster 集群

大家好，我是你们的讲师。前面两章我们学会了"让数据多一份副本"——主从复制。但有了副本之后，还缺一个关键环节：当主库挂了，谁来把某个从库提升为主库？如果是半夜 3 点，难道要运维人员从床上爬起来手动操作吗？

这就是 Sentinel（哨兵）要解决的问题——自动故障转移。而如果你想把数据分散到多台机器上，突破单机内存上限，就需要 Cluster（集群）——自动数据分片。

Sentinel 和 Cluster 是 Redis 高可用的两套方案，它们解决的是不同层面的问题，也可以组合使用（Sentinel 管理 Cluster 中的每个 shard）。本章我们深入两者的源码实现，理解它们的核心机制和设计哲学。

Sentinel 架构

Sentinel 是一个独立进程，不是 Redis server 的一部分。它的代码在 sentinel.c 中，编译后生成 redis-sentinel 可执行文件。一个典型的 Sentinel 部署通常有 3 或 5 个节点（奇数个，用于投票达成共识），它们互相通信，共同监视一组 Redis 主从实例。

Sentinel 的核心数据结构是 sentinelState，它持有所有被监控的实例信息：

/* sentinel.c - sentinelState 全局状态 */
struct sentinelState {
    char myid[CONFIG_RUN_ID_SIZE+1];      /* 当前哨兵自己的 ID */
    uint64_t current_epoch;               /* 当前纪元（每次故障转移递增） */
    dict *masters;                        /* 所有被监控的主库（名称->sentinelRedisInstance） */
    int tilt;                             /* Tilt 模式标志 */
    int running_scripts;                  /* 正在运行的脚本数 */
    mstime_t tilt_start_time;             /* Tilt 模式开始时间 */
    mstime_t previous_time;               /* 上次执行时间 */
    list *scripts_queue;                  /* 待执行脚本队列 */
    char *announce_ip;                    /* 宣告的 IP */
    int announce_port;                    /* 宣告的端口 */
    unsigned long sim failure_flags;      /* 模拟故障（测试用） */
    int deny_scripts_reconfig;            /* 拒绝脚本重配置 */
} sentinel;

Sentinel 的核心工作是监控（Monitor）、通知（Notification）和自动故障转移（Automatic Failover）。这三个职责分别在 sentinelTimer() 定时任务中完成，该函数每 100ms 执行一次：

/* sentinel.c - 哨兵主循环 */
void sentinelTimer(void) {
    /* 1. 检查是否进入 Tilt 模式（哨兵系统时间异常时启用安全模式） */
    sentinelCheckTiltCondition();

    /* 2. 对所有被监控的实例执行状态检查 */
    sentinelHandleDictOfRedisInstances(sentinel.masters);

    /* 3. 运行待执行的脚本 */
    sentinelRunPendingScripts();

    /* 4. 清理已完成的脚本 */
    sentinelCollectTerminatedScripts();

    /* 5. 杀死超时的脚本 */
    sentinelKillTimedoutScripts();
}

这其中最关键的是 sentinelHandleDictOfRedisInstances()，它遍历所有 master，然后递归处理每个 master 下的所有 replica 和关联 sentinel：

/* sentinel.c - 状态处理核心 */
void sentinelHandleDictOfRedisInstances(dict *instances) {
    dictIterator *di;
    dictEntry *de;
    sentinelRedisInstance *instance;

    di = dictGetIterator(instances);
    while ((de = dictNext(di)) != NULL) {
        instance = dictGetVal(de);

        /* 1. 发送 PING 并接收 PONG 进行活性检测 */
        sentinelSendPing(instance);

        /* 2. 更新实例可用性状态（SDOWN/ODOWN 判定） */
        sentinelCheckSubjectivelyDown(instance);
        sentinelCheckObjectivelyDown(instance);

        /* 3. 如果实例是 master，触发故障转移流程 */
        if (instance->flags & SRI_MASTER)
            sentinelFailoverStateMachine(instance);

        /* 4. 更新实例的 sentinel 信息（交换哨兵间的信息） */
        sentinelProcessHelloMessage(instance);
    }
    dictReleaseIterator(di);
}

哨兵判定机制

Sentinel 的故障判定分为两个层次：主观下线（SDOWN）和客观下线（ODOWN）。

SDOWN（Subjectively Down）

每个 Sentinel 独立向被监控的实例发送 PING。如果在 down-after-milliseconds 时间内没有收到有效回复（PONG 或加载中），这个 Sentinel 就将该实例标记为 SDOWN。这是一个本地判断，其他 Sentinel 可能有不同看法。

ODOWN（Objectively Down）

当 Sentinel 将一个 master 标记为 SDOWN 后，它会向其他 Sentinel 询问"你们怎么看？"（通过 sentinel is-master-down-by-addr 命令）。如果收到了足够多的确认（达到 quorum 数量），这个 Sentinel 就将 master 标记为 ODOWN，并触发故障转移。

/* sentinel.c - ODOWN 判定 */
void sentinelCheckObjectivelyDown(sentinelRedisInstance *master) {
    /* 统计投了"已下线"票的哨兵数量 */
    dictIterator *di;
    dictEntry *de;
    unsigned int quorum = master->quorum;
    unsigned int votes = 0;  /* 自己先投一票 */

    di = dictGetIterator(master->sentinels);
    while ((de = dictNext(di)) != NULL) {
        sentinelRedisInstance *si = dictGetVal(de);
        if (si->flags & SRI_MASTER_DOWN) votes++;
    }
    dictReleaseIterator(di);

    /* 投票数 >= quorum，判定为客观下线 */
    if (votes >= quorum) {
        master->flags |= SRI_O_DOWN;
    } else {
        master->flags &= ~SRI_O_DOWN;
    }
}

这个设计借鉴了分布式共识的思想：单个哨兵的判断可能因为网络分区而误判，但多个哨兵交叉验证后，误判的概率大大降低。

故障转移流程

ODOWN 确认后，Sentinel 进入故障转移流程。这个流程由 sentinelFailoverStateMachine() 驱动，这是一个典型的状态机：

  1. SENTINEL_FAILOVER_STATE_WAIT_START: 等待故障转移开始计时器到期。
  2. 选举领导者：Sentinel 之间通过 Raft 简化版算法选举一个 Leader 来执行故障转移。每个 Sentinel 向其他节点发送竞选请求，收到多数票（> N/2）的成为 Leader。
  3. SENTINEL_FAILOVER_STATE_SELECT_SLAVE: 从健康的从库中选出新主库。选择规则优先级：replica-priority 最小 → 复制偏移量最大（数据最新） → runid 字典序最小。
  4. SENTINEL_FAILOVER_STATE_SEND_SLAVEOF_NOONE: 向选中的从库发送 SLAVEOF NO ONE，让它变成新主库。
  5. SENTINEL_FAILOVER_STATE_WAIT_PROMOTION: 等待新主库完成角色切换。
  6. SENTINEL_FAILOVER_STATE_RECONF_SLAVES: 将其余从库指向新主库（SLAVEOF new_master_ip port）。
/* sentinel.c - 选择新主库（简化） */
sentinelRedisInstance *sentinelSelectSlave(sentinelRedisInstance *master) {
    sentinelRedisInstance **slaves = /* 获取所有从库列表 */;
    int n = /* 从库数量 */;

    /* 过滤掉不符合条件的从库 */
    /* 条件：S_DOWN / O_DOWN 不选，断线超过 5 秒不选，
     * 优先级 0 不选，与主库断连超过 10 * down-after-milliseconds 不选 */
    /* ... */

    /* 按优先级排序：priority 越小越靠前 */
    qsort(slaves, n, sizeof(sentinelRedisInstance*), compareSlavesForPromotion);

    return slaves[0]; /* 返回最合适的从库 */
}

int compareSlavesForPromotion(const void *a, const void *b) {
    sentinelRedisInstance **sa = (sentinelRedisInstance **)a;
    sentinelRedisInstance **sb = (sentinelRedisInstance **)b;

    /* 1. 比较 replica-priority（越小越优先） */
    if ((*sa)->slave_pri < (*sb)->slave_pri) return -1;
    if ((*sa)->slave_pri > (*sb)->slave_pri) return 1;

    /* 2. 比较复制偏移量（越大越优先，数据越新） */
    if ((*sa)->slave_master_offset > (*sb)->slave_master_offset) return -1;
    if ((*sa)->slave_master_offset < (*sb)->slave_master_offset) return 1;

    /* 3. 比较 runid（字典序，越小越优先） */
    return strcasecmp((*sa)->runid, (*sb)->runid);
}
部署建议：Sentinel 一般部署 3 个或 5 个节点（奇数），quorum 设为 2（3 节点）或 3（5 节点）。不要把 Sentinel 和 Redis 部署在同一台机器上——如果机器宕了，Sentinel 和 Redis 一起挂，高可用就无从谈起。另外，Sentinel 本身没有密码认证的隔离能力，务必在网络层面做好隔离。

Cluster 哈希槽

如果说 Sentinel 解决的是高可用问题，那 Cluster 解决的就是水平扩展问题。它将数据分散在多个节点上，理论上可以扩展到上千个节点。

Redis Cluster 使用哈希槽（Hash Slot）来进行数据分片。整个键空间被划分为 16384 个 slot（0~16383），每个 key 属于其中一个 slot。slot 到节点的映射关系由整个集群维护。

/* cluster.h - 集群状态结构 */
typedef struct clusterState {
    clusterNode *myself;              /* 当前节点 */
    uint64_t currentEpoch;            /* 当前纪元 */
    int state;                        /* 集群状态：REDIS_CLUSTER_OK / FAIL */
    int size;                         /* 至少包含一个 slot 的节点数 */
    dict *nodes;                      /* 所有节点（名称 -> clusterNode） */
    Node *slots[REDIS_CLUSTER_SLOTS]; /* slot -> 节点映射（核心！） */
    zskiplist *slots_to_keys;         /* slot -> key 的跳表索引 */
    /* ... */
} clusterState;

key 到 slot 的映射计算方式非常简单：

/* cluster.c - 计算 key 的 slot */
unsigned int keyHashSlot(char *key, int keylen) {
    int s, e;
    /* 检查是否有 hash tag：{...} */
    for (s = 0; s < keylen; s++)
        if (key[s] == '{') break;

    /* 没有 hash tag，对整个 key 计算 CRC16 */
    if (s == keylen) return crc16(key, keylen) & 0x3FFF;

    /* 有 hash tag，只对大括号内的内容计算 CRC16 */
    for (e = s+1; e < keylen; e++)
        if (key[e] == '}') break;
    if (e == keylen || e == s+1)
        return crc16(key, keylen) & 0x3FFF;

    /* 只对 { } 内的部分计算 CRC16 */
    return crc16(key+s+1, e-s-1) & 0x3FFF;
}

注意 hash tag（{...}）的作用：当你在 user:{1001}:name 和 user:{1001}:age 这样的 key 中使用相同的 hash tag 时，它们会被分配到同一个 slot，从而支持 multi-key 操作。这就是 Cluster 模式下实现「关联数据放在一起」的关键机制。

客户端向集群的任意节点发送请求时，节点会计算 key 所属的 slot，然后在 clusterState.slots[] 中查找该 slot 所属的节点。如果 slot 属于当前节点，直接处理；否则返回 MOVED 重定向，告诉客户端正确的节点地址。

Cluster 数据分片与迁移

当集群需要扩容、缩容或者重新平衡时，就需要在节点之间迁移 slot。迁移过程以 slot 为单位，核心命令是 MIGRATE。

迁移过程中，slot 可能处于"正在迁移"的中间状态。这时需要处理ASK 重定向机制：

  • MOVED 重定向：slot 已经确认属于另一个节点。客户端收到 MOVED 后，应该更新自己的路由缓存，直接向新节点发送请求。
  • ASK 重定向：slot 正在迁移中，key 可能还在旧节点上也可能已经迁走了。客户端收到 ASK 后，先向新节点发送一个 ASKING 命令（临时标记），然后再发请求。ASK 是"一次性"的，不会更新客户端的路由缓存。
/* cluster.c - MOVED 重定向 */
void clusterRedirectClient(client *c, clusterNode *n, int hashslot, int ask) {
    if (ask) {
        /* slot 迁移中，使用 ASK 重定向 */
        addReplyErrorFormat(c, "-ASK %d %s:%d",
            hashslot, n->ip, n->port);
    } else {
        /* slot 已经确认属于另一节点，使用 MOVED 重定向 */
        addReplyErrorFormat(c, "-MOVED %d %s:%d",
            hashslot, n->ip, n->port);
    }
}
踩坑实录：某团队将 Redis Cluster 用于电商购物车场景。购物车数据包含多个 key，比如 cart:1001（购物车内容）、cart:1001:price（价格快照）。他们天真地直接使用这些 key 名，没有加 hash tag。结果多个 cart 相关的 key 被随机分配到不同的 slot 上，当业务需要在一个原子操作中读取所有购物车数据时，就触发了 CROSSSLOT 错误——因为 Cluster 不支持跨 slot 的 multi-key 操作。解决方法是把 key 改为 cart:{1001} 和 cart:{1001}:price，利用 hash tag 强制它们进入同一个 slot。教训是：使用 Cluster 前，必须提前规划好所有 key 的 hash tag 设计，尤其是需要在同一事务中操作多个 key 的场景。

Gossip 协议与集群通信

Cluster 节点之间通过 Gossip 协议（流言协议）来交换状态信息。每个节点维护一个集群内所有节点的列表，并定期与其他节点交换信息。这种去中心化的通信方式避免了单点瓶颈。

Gossip 消息通过 clusterMsg 结构体定义，消息类型主要有：

  • PING：定期发送，携带本节点的状态信息以及随机选取的其他 2~3 个节点的信息。
  • PONG：对 PING 的回复，也包含节点状态。
  • MEET：邀请一个新节点加入集群。
  • FAIL：广播某个节点已下线。
  • PUBLISH：跨节点传播 Pub/Sub 消息。

通信由 clusterCron() 驱动，这是一个每秒执行 10 次的周期性函数（基于 server.hz）：

/* cluster.c - clusterCron 核心逻辑（简化） */
void clusterCron(void) {
    /* 遍历所有已知节点 */
    di = dictGetSafeIterator(server.cluster->nodes);
    while ((de = dictNext(di)) != NULL) {
        clusterNode *node = dictGetVal(de);

        /* 如果节点已经断线过久，标记为 PFAIL */
        if (node->flags & (CLUSTER_NODE_PFAIL|CLUSTER_NODE_FAIL))
            continue;
        if (node_timout && (now - node->ping_sent) > node_timeout) {
            node->flags |= CLUSTER_NODE_PFAIL;
            clusterSendFail(node->name); /* 广播 FAIL 消息 */
        }

        /* 定期发送 PING */
        if (node->link && /* 有连接 */
            (now - node->link->ctime > node_timeout / 2) && /* 已建立连接超过半超时 */
            (node->ping_sent == 0)) /* 没有正在等待的 PING */
        {
            clusterSendPing(node->link, CLUSTERMSG_TYPE_PING);
        }

        /* 每个周期随机选择几个节点进行 Gossip 交换 */
        /* 这是集群状态传播的核心机制 */
    }
    dictReleaseIterator(di);

    /* 检查是否需要进行 slot 迁移 */
    /* 如果当前节点有 IMPORTING 状态的 slot，尝试从源节点拉取数据 */
}

Gossip 的"随机传播"特性使得集群状态的变化最终传递到所有节点，但存在一定的延迟。这也是为什么 Cluster 设计中允许少量时间内的"不一致视图"——只要最终所有节点都能达成一致。

Sentinel vs Cluster vs 代理模式

下面这张表对比了 Redis 三种分布式方案的核心差异：

对比维度 Sentinel Cluster 代理模式（Codis/Twemproxy）
架构角色 主-从 + 监控进程 去中心化 P2P 代理层 + 后端 Redis
数据分片 不支持（全量副本） 16384 hash slot 一致性哈希 / 预分片
故障转移 Sentinel 自动切换 集群内部自动完成 依赖外部组件（zk/etcd）
多 key 操作 完全支持 仅同 slot 支持 同实例支持
客户端复杂度 低（直连主从） 高（需处理 MOVED/ASK） 低（直连代理）
可扩展性 差（单机内存上限） 好（最大约 1000 节点） 好（加代理和后端即可）
适用规模 GB 级别 TB 级别 TB 级别
运维复杂度 低 中 高（多组件依赖）
典型部署 1主2从 + 3哨兵 6~9 节点起步 需独立代理集群

选择哪套方案主要看你的规模和团队能力：

  • 数据量 < 50GB： Sentinel 方案最省心，单主 + 双从 + 三哨兵，足够应对大多数场景。
  • 数据量 50GB~1TB： Cluster 方案更合适，天然支持水平扩展，但需要注意 hash tag 设计和客户端兼容性。
  • 已有基础设施依赖代理模式：如果你的团队已经在用 Codis 或 Twemproxy，迁移到 Cluster 的收益需要评估——代理模式虽然多了一跳，但迁移成本最低。
