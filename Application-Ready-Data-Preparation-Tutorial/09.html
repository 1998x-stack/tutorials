第9章 上下文感知清洗：LLMClean方法

大家好，我是你们的讲师。上一章我们学会了用LLM发现数据中的错误——但发现问题只是第一步。真正的难点是：找到错误之后，怎么修？如果你直接对LLM说"把这个错误修掉"，它可能会给出一个语法正确但语义离谱的"修复"。这就是数据清洗中最棘手的挑战——缺乏上下文感知。这一章，我们来解剖LLMClean，一个将领域本体知识与LLM推理能力融合的上下文感知清洗框架。

为什么规则不够用

传统数据清洗依赖固定规则：IF-THEN-ELSE、查询表、正则替换。这些方法在小规模、规则明确的场景下表现良好，一旦进入复杂领域就捉襟见肘。原因有三。

第一，规则的边界效应。规则永远只覆盖"边界内"的情况，数据中的边缘情况总是超出规则的覆盖范围。第二，规则的静态性。数据分布会漂移，业务规则会变化，但写在代码里的规则不会自动适应。第三，规则的碎片化。一个数据库可能涉及数十张表、数百个字段，每个字段的语义都需要独立定义规则，这种碎片化导致维护成本呈指数增长。

思考题：假设你有一个"患者登记表"，其中"地址"字段包含了邮政编码。当你发现某条记录的邮编和城市不匹配时，正确的修复应该是"修正邮编"还是"修正城市"？如果没有上下文信息——比如登记时哪个字段出错概率更高——你根本没法做出正确的修复选择。这就是"上下文感知"的核心价值。

本体功能依赖原理

LLMClean的核心创新在于提出了一种新的数据依赖形式：本体功能依赖（Ontology Functional Dependency, OFD）。要理解OFD，我们先对比一下它和传统FD的区别。

维度传统FD本体功能依赖(OFD)
依赖表示A → B（列相等约束）A onto→ B（语义关联约束）
知识来源数据表本身领域本体 + 数据表
推理能力精确匹配语义相似度 + 推理
覆盖范围列间等值关系列间复杂语义关系
容错性零容忍可处理近似依赖
典型例子ZipCode → City（精确映射）Symptom onto→ Diagnosis（语义关联）

OFD的精髓在于：它不再要求列与列之间的精确函数依赖，而是将领域本体作为中间层，让LLM根据领域知识推断列与列之间"应该存在"的约束。例如，在医疗数据中，"症状"和"诊断"两个字段不存在严格的一对一映射，但领域本体告诉我们某些症状组合与特定诊断强相关——这就构成了一条OFD。

LLMClean系统架构

LLMClean的工作流程分为三个主要阶段，形成一个完整的清洗流水线。

第一阶段：本体分析。系统接收数据表及其模式描述，LLM首先对每个字段进行语义分析，理解字段的现实含义。同时，系统从外部知识源（如医疗分类代码、地理编码数据库）注入领域本体信息，构建字段与本体概念的映射关系。

第二阶段：OFD生成。基于本体分析的结果，LLM自动发现并生成字段间的本体功能依赖。这一阶段不需要人工标注，完全由提示工程引导LLM完成。

第三阶段：规则驱动的清洗。将生成的OFD转换为可执行的清洗规则，对数据表中的错误进行自动修复。规则可以是SQL可执行的CHECK约束，也可以是Python函数。

架构关键点：LLMClean并不是每次清洗都调用LLM。OFD的生成是一次性成本，生成之后的清洗阶段不需要LLM推理——这在实际应用中大幅降低了成本和延迟。LLM只在"规则生成"和"异常处理"两个环节出现。

LLM生成OFD的流程

我们来看看LLM是如何生成本体功能依赖的。这个过程需要精心设计提示词，引导LLM从表结构描述中提取领域语义关系。

任务：为一个物联网传感器数据表生成本体功能依赖(OFD)

表结构：
- sensor_id: 传感器唯一标识（字符串）
- sensor_type: 传感器类型（温度/湿度/气压/风速）
- reading_value: 读数（浮点数）
- unit: 单位（字符串）
- timestamp: 时间戳
- location: 安装位置

领域知识：
- 温度传感器读数范围：-40°C 到 85°C
- 湿度传感器读数范围：0%RH 到 100%RH
- 气压传感器读数范围：300hPa 到 1100hPa

请生成所有可能的OFD，格式如下：
OFD1: sensor_type →onto→ reading_value_range
说明：传感器类型通过领域本体约束了读数的合理范围

OFD2: sensor_type →onto→ unit
说明：每种传感器类型对应特定的度量单位
...
输出更多发现：

LLM输出的OFD会经过一个验证和精炼步骤。系统会对每条OFD进行抽样测试，用数据中的已知正确记录验证OFD的正确性。如果发现冲突，提示LLM调整或拆分OFD。这个过程迭代2-3轮后，OFD集就基本稳定了。

实战案例：医疗数据中的邮政编码陷阱

这是一个从LLMClean论文中改编的真实案例。一家医院的数据集包含数万名患者的"城市"和"邮政编码"字段。传统FD方法在这个数据集上表现不佳，因为城市与邮编之间存在大量的例外情况——比如某些邮政编码覆盖多个城市，或者同一城市在不同时期有不同邮编。

LLMClean的做法是：首先将"邮政编码"字段映射到地理编码本体，然后让LLM自动理解城市与邮编之间的语义关系。LLM发现了43条OFD规则，其中一条类似于"邮政编码开头的两位数字与省份/直辖市之间存在强关联"。传统FD只能处理精确匹配，而OFD发现了一个更灵活的模式：

-- 传统FD无法处理的"近似匹配"情况
-- 城市 = "北京市", 邮编 = "100001" (东城区) → 正确
-- 城市 = "北京市", 邮编 = "200001" (黄浦区) → 明显错误
-- 城市 = "北京市", 邮编 = "102200" (昌平区) → 正确
--
-- 传统FD要求：城市相同 → 邮编相同
-- 但在北京，不同区的邮编不同，这条FD不成立
--
-- OFD版本：城市 "北京市" →onto→ 邮编以 "10" 或 "102" 开头
-- 这条OFD允许了合理的多值映射，同时捕获了异物编码错误

最终，LLMClean在这个数据集上检出了传统方法无法发现的3,200条错误记录，修复后数据的可用性提升了42%。更重要的是，整个清洗规则的构建只用了不到2小时，而传统方法至少需要2名数据工程师工作一周。

好，LLMClean展示了"理解上下文再清洗"的威力。但你可能在想：LLMClean需要一定的领域知识输入，难道每个新领域都要"重复造轮子"吗？下一章我们会介绍一种更有野心的方案——ZeroED，一种不需要任何标注数据、不需要领域预设的零样本错误检测方法。
