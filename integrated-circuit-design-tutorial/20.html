第20章 片上网络(NoC)

大家好，我是你们的讲师。上一章我们讨论了SoC架构设计——处理器选型、总线协议、存储层次。当我说到"总线互连"时，用了一个比喻叫"高速公路"。但如果你的SoC有20个主设备（CPU、GPU、NPU、DMA、各个加速器）和30个从设备（各级存储、外设），一条高速公路是远远不够的——你会需要一整座"城市交通网络"。这就是片上网络（Network-on-Chip，NoC）登场的理由。

NoC不是"可选方案"——对于超过一定规模的SoC，它几乎是唯一的出路。如果你今天在设计一颗16核以上的服务器CPU、或者一颗整合了数十个IP模块的AI SoC，你几乎肯定在使用NoC。这一章，我们就来深入理解NoC的原理、设计和工程实践。

20.1 总线与NoC演进

为什么总线不香了？让我们看看传统共享总线架构在扩展时遇到的三个核心问题：

第一个问题：仲裁冲突。在共享总线上，同一时刻只能有一个主设备发起传输。当有N个主设备时，每个设备的平均总线带宽只能分到1/N。在智能手机SoC中，GPU和CPU和NPU和Display Controller同时疯狂请求数据——一个简单的仲裁器无法公平高效地分配带宽。

第二个问题：物理布局。当芯片上模块数量增多时，一条贯穿芯片的总线必须跨越越来越长的物理距离。长走线导致巨大的RC延迟，限制了总线频率。AXI在典型工艺下很难跑到2GHz以上，而一个独立的NoC路由器可以工作在更高的时钟域中，只需要在边界做异步桥接。

第三个问题：功耗。长总线需要驱动很大的电容——每个寄存器、每个多路选择器都挂在总线上。在一个大型SoC中，总线互联网络消耗的功耗可以占到整个芯片动态功耗的15-25%。NoC通过分段化、数据打包和分布式路由，可以显著减少每比特数据的传输功耗。

交叉开关（crossbar）比共享总线好一些，但当端口数超过8-10个时，面积和功耗开始急剧上升——因为交叉开关的面积复杂度是O(N²)。NoC正是解决这个问题的方案：把数据打包成"包"（packet），通过网络路由器的分布式转发来实现任意节点间的通信，而不是用一根或一堆导线连接所有节点。

20.2 拓扑结构

NoC的拓扑——即路由器和链路如何连接——是第一个设计决策。常见的拓扑有：

  • Mesh（网格）：最经典的NoC拓扑。节点排列成二维网格，每个路由器连接四个邻居和一个本地节点。Mesh的优势是路由简单、可扩展性好。缺点是直径较大——角落到对角的延迟高。
  • Torus（环形网格）：在Mesh的基础上增加了环形环绕链路。延迟更低（因为直径减半），但对布线有更高的要求——环绕走线需要在芯片边缘弯曲。
  • Ring（环形）：所有节点连成一个环。结构最简单，但扩展性差——每增加一个节点，最坏情况延迟就增加一跳。Ring适合节点数较少的SoC。
  • Tree（树状）：按层次连接。延迟可以做到log(N)量级，但根节点容易成为瓶颈。Fat Tree使用了加宽高层链路的策略来缓解这个问题。
  • CMesh（Concentrated Mesh）：在Mesh的基础上，每个路由器服务多个本地节点。减少了路由器数量，但增加了本地路由器的复杂度。CMesh是Intel某些多核CPU采用的结构。
  • Clos网络：来自电话交换网络的拓扑，通过多级交换实现非阻塞（non-blocking）互连。在高端网络芯片和数据中心交换机中很常见。

拓扑选择的两个关键指标是对分带宽（Bisection Bandwidth）和直径（Diameter）。对分带宽——把网络切成两半，穿过切面的总带宽——决定了网络理论上能支持的最大吞吐量。直径——网络中两个节点之间的最大最短路径长度——决定了最坏情况下的通信延迟。

20.3 路由与流控

有了拓扑，下一个问题就是：数据包在网络中怎么走？这就是路由算法和流量控制的工作。

路由算法

确定性路由（Deterministic Routing）：最常见的是XY路由——先沿X方向走到目标列，再沿Y方向走到目标行。XY路由简单、无死锁、易于硬件实现。缺点是负载不均匀——网络中心区域的链路压力很大。

自适应路由（Adaptive Routing）：路由器根据当前网络的拥塞状况选择路径。如果东向链路拥塞了，选择走北向绕路。自适应路由能更好地平衡负载，但对硬件复杂度要求更高。自适应路由必须特别注意死锁避免——绕路有可能引入循环依赖。

流量控制

最常用的NoC交换方式是虫洞交换（Wormhole Switching）：一个数据包被拆分成若干微片（flit），路由器的缓冲区只需要能存储一个或几个flit的大小。虫洞交换的延迟很低——只要头部flit获得了一条路径，后续flit就像"虫洞"一样穿过去——但缺点是链路会被阻塞（head-of-line blocking）。

虚拟通道（Virtual Channel，VC）是解决这个问题的关键技术。每条物理通道被划分为多个独立的虚拟通道，每个VC有自己独立的缓冲区。当一个VC的数据包被阻塞时，其他VC可以继续使用该物理通道。VC极大地提高了NoC的利用率和公平性。

基于信用的流控（Credit-based Flow Control）：接收端通告自己还有多少空闲缓冲区空间（credit），发送端在没有credit时不能发送新的flit。这是最常用的无损流控机制。

// XY路由算法（伪代码）
// 输入: 当前节点坐标 (cur_x, cur_y)
//       目标节点坐标 (dst_x, dst_y)
//       可用输出端口状态 (port_status)
// 输出: 选择的路由端口

function xy_routing(cur_x, cur_y, dst_x, dst_y, port_status):
    // 计算偏移
    offset_x = dst_x - cur_x
    offset_y = dst_y - cur_y

    // 先X维度路由
    if offset_x != 0:
        if offset_x > 0:
            if port_status.EAST == AVAILABLE:
                return EAST
        else:
            if port_status.WEST == AVAILABLE:
                return WEST
        // 首选方向不可用（可能死锁避免）
        // 对于确定性路由：不应该发生，除非链路故障
        return BLOCKED

    // X维度已对齐，再Y维度路由
    if offset_y != 0:
        if offset_y > 0:
            if port_status.SOUTH == AVAILABLE:
                return SOUTH
        else:
            if port_status.NORTH == AVAILABLE:
                return NORTH
        return BLOCKED

    // 已经到达目标
    return LOCAL

// 自适应路由（最小路径变体）
function adaptive_routing(cur_x, cur_y, dst_x, dst_y, congestion_table):
    // 列出所有使目标更近的候选端口
    candidates = []
    if cur_x < dst_x: candidates.push(EAST)
    if cur_x > dst_x: candidates.push(WEST)
    if cur_y < dst_y: candidates.push(SOUTH)
    if cur_y > dst_y: candidates.push(NORTH)

    // 按拥塞程度排序（最小拥塞优先）
    sort candidates by congestion_table[port]

    return candidates[0]  // 选最不拥塞的端口

20.4 服务质量与死锁

QoS类别

在一个NoC中，不同类型的流量有不同的QoS需求：

  • Best-effort（BE）：尽力而为，没有带宽或延迟保证。适用于批量数据传输、背景任务。网络空闲时跑得快，拥塞时等一等。
  • Bandwidth Reservation（BR）：保留带宽。适用于视频流、显示刷新等持续高带宽流量。通过VC reservation或TDMA时隙分配实现。
  • Latency Guarantee（LG）：最严格的服务级别。适用于实时控制信号、中断传递——延迟必须可控。通常通过专用VC和优先级仲裁实现。

死锁、活锁与饥饿

死锁（deadlock）是NoC设计中最令人头疼的问题。当两个或更多数据包互相等待对方释放资源时，就形成了死锁。经典的死锁场景是：路由器A北向的VC被B占用，B西向的VC被C占用，C南向的VC被D占用，D东向的VC被A占用——形成了一个循环等待。

死锁避免的常见方法包括：

  • 转向限制（Turn Restriction）：在Mesh网络中，禁止某些特定的转向组合。比如"禁止北转东"—即如果一个包先向北走，就不能再转向东。著名的up/down路由和Odd-Even路由都是基于这个思想。
  • Dateline（虚拟层）：在Torus网络中划一条"日期变更线"，包每跨过dateline一次就切换一个虚拟层。不同虚拟层之间不能形成循环依赖。
  • VC专用的死锁避免：为每个VC分配严格的路由规则，确保不会形成循环依赖。

活锁（livelock）：包在网络中永远无法到达目的地，一直在绕圈。自适应路由如果没有"最短路径保证"，有可能出现活锁。解决方案是给包一个"最大跳数"计数器，超时后直接丢弃或采用确定性路由。

饥饿（starvation）：某个数据包因为优先级太低，某些节点或VC的数据包始终无法获得服务。在优先级仲裁中，必须做"aging"（老化）处理——长时间未获得服务的包自动提高优先级。

真实的踩坑故事——NoC死锁，流片后才找到：我有一个朋友参与了一颗28nm的AI推理芯片设计。这颗芯片用了4×4的Mesh NoC连接16个NPU核。设计时团队对死锁做了充分的分析——XY路由理论上无死锁，仿真中也从来没有出现过死锁。芯片流片回来，单核测试通过，双核测试通过，但是当16个核跑一个特定的并行计算模式时——所有核同时向对角线方向的对等核发送数据——NoC死锁了。原因是他们的路由器实现中做了一些"优化"：为了降低延迟，在路由计算和数据转发之间插入了一条旁路路径。这条旁路在仿真中从来没有被触发的流量模式下形成了一条意想不到的循环依赖。最后定位到问题后，硬件已经无法改变，只能靠软件打补丁——修改数据分发模式，避免那个特定的通信模式。教训是什么？NoC的死锁分析必须在所有流量模式下进行，不能只依赖路由算法本身的无死锁证明。最好在RTL验证中就引入"最恶意流量模式"（worst-case traffic pattern）的形式化检查。

20.5 NoC设计方法学

商业NoC IP

在工业界，大多数团队不会从零开始设计NoC——而是使用成熟的商业NoC IP：

  • Arteris FlexNoC：最著名的商业NoC IP之一。FlexNoC支持AMBA AXI/ACE/CHI协议的透明桥接、物理感知布局（physical-aware布局优化）、多时钟域支持、以及复杂的QoS配置。FlexNoC被NVIDIA、Mobileye等公司广泛使用。
  • NetSpeed（已被Intel收购）：NetSpeed的NoC IP曾以可配置性和可视化设计工具著称。被Intel收购后，其技术被用于Intel的内部SoC设计。

开源NoC

OpenSoC Fabric是一个开源NoC生成框架，支持多种拓扑、路由算法和流量控制配置。虽然在商业级SoC中直接使用还有距离，但对于学术研究和原型验证是极好的工具。

NoC性能分析

BookSim是斯坦福大学开发的开源NoC模拟器，被学术界和工业界广泛用于NoC架构的早期评估。BookSim支持各种拓扑、路由算法、VC配置和流量模式，可以在架构探索阶段快速评估不同NoC配置的延迟和吞吐量。

面向Chiplet的NoC

在Chiplet（小芯片）架构中，NoC的概念需要从片内扩展到片间。不同die之间通过UCIe、BoW（Bridge of Wires）等die-to-die接口连接。NoC-to-NoC桥接成为了一个新的设计问题——需要在物理层差异、协议差异和时钟域差异之间做转换。

NoC是现代大型SoC的"神经系统"——它把几十甚至上百个IP核连接成一个有机的整体。理解了NoC，你就理解了大型芯片设计的"交通规划"。但无论NoC的通道有多宽、路由器有多快，数据最终要到"存储"这个目的地。下一章我们来讨论SoC中另一个核心子系统——存储器系统设计，从SRAM到DRAM，从Flash到新型存储，我们会完整地过一遍片上存储架构的设计要点。我们下一章见。
