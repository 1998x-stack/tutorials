第24章：生产环境踩坑指南

大家好，我是你们的讲师。如果你在网上搜索"Redis 踩坑"，你会发现成千上万篇技术文章，每一篇背后都是一个工程师在凌晨三点盯着屏幕掉头发的故事。我自己从业十几年，亲眼见过因为一个 bigkey 导致整个电商站首页卡死 30 秒的、因为 THP 没关导致从库复制延迟达到数小时的、因为 maxmemory 设了一个很整的数字导致淘汰策略反复触发 OOM 的。

这些坑有一个共同的特点：文档里都写了，但你读的时候不会意识到"原来这真的会发生"。这一章，我们不谈理论，专门讲生产环境中最常见、破坏力最大的几类故障，每一个都配上真实踩坑案例和源码层面的解释。等你看完这一章，以后再遇到类似问题，至少不会在同一个地方再摔一次。

Bigkey 问题

Bigkey 的发现

Bigkey 是 Redis 生产环境中的头号杀手。所谓 bigkey，就是某个 key 的 value 特别大——比如一个 Hash 中有上百万个 field，或者一个 List 长达几十万条，或者一个 String 有几十 MB。Bigkey 的危害有三个层面：

  • 操作阻塞：O(N) 命令（如 HGETALL、SMEMBERS、LRANGE）需要遍历整个数据结构的全部元素，期间事件循环被阻塞。
  • 网络拥堵：读取 bigkey 时，大数据量从 Redis 传输到客户端，占用网络带宽，拖慢其他请求。
  • IO 放大：删除 bigkey 时，释放大量内存块会导致 free() 操作耗时过长，阻塞事件循环。

如何发现 bigkey？Redis 提供了 --bigkeys 扫描参数：

# 扫描整个实例的 bigkey（不会阻塞，使用 scan 渐进式扫描）
redis-cli --bigkeys

# 输出示例
# Scanning the entire keyspace to find biggest keys as well as
# average sizes per key type.  You can use -i 0.1 to sleep 0.1 sec
# per 100 SCAN commands (not usually needed).
[00.00%] Biggest string found so far '"session:abc123"' with 1024 bytes
[00.05%] Biggest hash  found so far '"user:profile:99999"' with 20345 fields
-------- summary -------
Sampled 100000 keys in the keyspace!
Total key length in bytes is avg 22.34 (max 64)

Biggest string found '"session:abc123"' has 1024 bytes
Biggest   hash found '"user:profile:99999"' has 20345 fields
Biggest    set found '"blocked_ips"' has 312044 members
Biggest   list found '"notification:queue"' has 401223 items
Biggest   zset found '"leaderboard:global"' has 500000 members

注意 --bigkeys 只统计每个类型最大的 key，不会列出所有大 key。如果想全面扫描，可以用自定义的 Lua 脚本配合 MEMORY USAGE 命令逐 key 检查。

Bigkey 的删除

在 Redis 4.0 之前，删除一个包含数百万字段的 bigkey，DEL 命令内部会遍历释放所有内存节点，这个过程可能耗时数秒甚至数分钟，完全阻塞事件循环。从 4.0 开始，UNLINK 命令被引入：

/* db.c —— unlinkCommand 异步释放逻辑 */
void unlinkCommand(client *c) {
    /* 将 key 的释放操作放入后台任务队列 */
    if (delGenericCommand(c, 1, 1) == C_OK) {
        /* 如果是 bigkey，后台线程会异步释放内存 */
    }
}

/* lazyfree.c —— 后台释放 bigkey */
size_t lazyfreeFreeHashFromBioTable(robj *o) {
    dict *d = o->ptr;
    size_t freed = dictSize(d);
    /* 后台 BIO 线程执行实际释放，不影响主线程 */
    if (dictSize(d) > LAZYFREE_THRESHOLD) {
        bioCreateBackgroundJob(BIO_LAZY_FREE, o, NULL, NULL);
        return freed;
    }
    /* 小字典直接释放 */
    dictRelease(d);
    return freed;
}

关键点：UNLINK 在 O(1) 时间内从键空间中删除 key，然后将底层数据结构的释放操作交给后台 BIO 线程。释放阈值是 LAZYFREE_THRESHOLD（在源码中定义为 64），超过此值的字典才会走异步释放路径。小于 64 的 key 仍然同步释放，因为队列化开销反而大于直接释放的开销。

踩坑实录：某社交平台的 Redis 实例每 30 秒出现一次 5~8 秒的卡顿，监控显示 CPU 不飙升、IO 不繁忙，但命令就是卡住。排查了两天，最后发现是一个缓存 session 的 bigkey——一个 Hash 中有 300 万个 field，每天凌晨的 Session 清理脚本用 DEL 删除它，导致每次删除都卡死 5 秒以上，而它恰好每 30 分钟被清理一次。更坑的是，这个 key 并非业务有意设计的"大 key"，而是由于一个代码 bug 导致同一个用户的 session 被不断追加新字段，日积月累膨胀到 300 万字段。后续修复：改用 UNLINK、增加 key 的 TTL、限制 Hash 的 max field 数量。

还有一点：bigkey 的回复缓冲区膨胀问题。当多个客户端同时读取同一个 bigkey 时，Redis 需要为每个 client 准备完整的输出数据，client-output-buffer-limit 可能被触发导致客户端被批量断开。更严重的是，copy-on-write 机制下 bigkey 的复制操作会放大 RSS 内存——这也是 lazyfree 工具函数中优先使用 UNLINK 的原因之一。

Hotkey 问题

Hotkey 是比 bigkey 更隐蔽的问题。一个 key 可能在几毫秒内就处理完毕，但如果它每秒被访问 100 万次呢？一个非常热的 key 会让它所在的 Redis 节点的 CPU 先被打满，而集群中其他节点却非常空闲。

Hotkey 的发现

# redis-cli 自带的热 key 检测（Redis 6.0+）
redis-cli --hotkeys

# 或者通过 MONITOR + 分析（生产环境慎用 MONITOR，它本身就消耗 CPU）
redis-cli MONITOR | head -100000 | awk '{print $4}' | sort | uniq -c | sort -nr | head -10

Hotkey 的挑战在于：

  • 单节点瓶颈：即使集群有 12 个分片，一个 hotkey 只能落在一个节点上，该节点 CPU 打满而其他节点空闲。
  • Cluster 迁移困难：Redis Cluster 不支持按 key 粒度迁移——如果你想把 hotkey 搬到其他节点，你需要迁移整个 slot，而这个 slot 下可能还有其他正常 key。

Hotkey 的解决方案

对于 hotkey，业界成熟的方案主要是"打散"和"本地缓存"：

  • 客户端本地缓存：引入 caffeine 或类似缓存库，在应用层面缓存热点 key，秒级过期。缺点是存在数据一致性问题（虽然 Redis 本身也不保证强一致性）。
  • Key 打散：将一个 hotkey 分解为多个子 key（如 hotkey:0 ~ hotkey:99），客户端读写时随机选一个。缺点是业务代码侵入性强。
  • 读写分离：将读操作分摊到 replica 节点，但前提是业务可以接受最终一致性。
踩坑实录：某电商大促期间，Redis Cluster 的一个节点 CPU 飙到 99%，其他节点不到 20%。排查发现是一个"秒杀商品详情"的 key 被频繁读取——所有用户的抢购请求都命中同一个 key。当时临时方案是在 Nginx 层加了一层 proxy cache，缓存 1 秒，瞬间降低了 90% 的请求。长期修复是将秒杀商品详情拆分为"静态模板 + 动态库存"两个 key，动态库存使用本地缓存 + WebSocket 推送。记住：Redis 不能解决所有性能问题，hotkey 在极端场景下需要有架构方案。

Fork 引发的血案

Redis 的 RDB 快照和 AOF rewrite 都依赖 fork() 系统调用来创建子进程。fork 本身是"快速"的（毫秒级），但如果 Redis 占用了数十 GB 的内存，fork 的耗时可能达到秒级——而这期间，父进程是完全挂起等待的。

THP 放大 COW

警告：THP 是 Redis 性能的头号隐形杀手。

透明大页（Transparent Huge Pages，THP）是 Linux 内核的一个特性，它自动将 4KB 小页合并为 2MB 大页来减少 TLB miss。听起来很美好，但在 Redis 的场景下它带来了灾难性的副作用：

  • 正常情况下，fork 后的 COW 是以 4KB 页为粒度的——子进程只读走了一页，父进程修改该页时只需复制 4KB。
  • 开启 THP 后，COW 的粒度变为 2MB。也就是说，父进程修改任意一个字节，内核要复制整整 2MB 的数据！
  • 如果 Redis 实例占用 20GB 内存，写密集型场景下 RSS 可能膨胀到 60GB+，直到把系统内存耗尽触发 OOM Killer。

查看和关闭 THP 的方法：

# 检查 THP 状态
cat /sys/kernel/mm/transparent_hugepage/enabled
# 输出 [always] madvise never   —— 表示已开启（always）
# 输出 always madvise [never]   —— 表示已关闭

# 临时关闭
echo never > /sys/kernel/mm/transparent_hugepage/enabled

# 永久关闭
# 在 /etc/rc.local 中添加：
if test -f /sys/kernel/mm/transparent_hugepage/enabled; then
   echo never > /sys/kernel/mm/transparent_hugepage/enabled
fi

vm.overcommit_memory

fork() 需要为子进程复制父进程的内存映射表——即使使用了 COW，内核也需要预判父子进程可能会占用多少内存。如果系统配置了 vm.overcommit_memory=0（默认），内核在 fork 时会保守地检查系统是否有足够的"潜力"容纳两个大进程，如果估算内存不足，fork 直接返回 ENOMEM。这是一个非常诡异的问题：明明你的 Redis 只用了 4GB 内存，系统剩下 32GB，但 fork 却报 ENOBUFS！

# 推荐的设置（永久写入 /etc/sysctl.conf）
echo "vm.overcommit_memory = 1" >> /etc/sysctl.conf
echo "vm.overcommit_ratio = 100" >> /etc/sysctl.conf
sysctl -p

设置 vm.overcommit_memory=1 表示"总是允许 overcommit"，内核不会在 fork 时做悲观估算。这是 Redis 官方文档中白纸黑字写明的建议，但我在生产环境中发现至少有一半的服务器没有配置它。

Fork 耗时监控

Redis 会记录每次 fork 的耗时，可以通过 INFO 命令查看：

# 查看 fork 耗时
INFO latest_fork_usec
# latest_fork_usec:3079   —— 最近一次 fork 耗时 3079 微秒（约 3ms）

# 在日志中查看 fork 信息
# 3631:M 28 Jun 2024 10:23:01.411 * Background saving started
# 3631:M 28 Jun 2024 10:23:01.415 * DB saved on disk
# 3631:M 28 Jun 2024 10:23:01.416 * RDB: 0 MB of memory used by copy-on-write
踩坑实录：某支付公司的 Redis 实例每日 10:00 准时出现 2~3 秒的卡顿。排查了一周，发现 10:00 是系统 crontab 中 AOF rewrite 的定时任务——fork 了一个 28GB 的 Redis 进程，耗时 2.7 秒。而 2.7 秒的挂起期间，所有客户端请求堆积在 socket buffer 中，一旦 Redis 恢复，burst 式的请求处理又导致了瞬间的 CPU 飚高和延迟毛刺。解决方案：将 AOF rewrite 安排在凌晨低峰期、减小 auto-aof-rewrite-percentage 避免一次性 rewrite 大文件、开启 THP 后 COW 内存膨胀到 60GB。最终在第 3 项找到了根因——关闭 THP 后，fork 耗时从 2.7 秒降到 300 毫秒，COW 内存从 60GB 降到 4GB。

关于 fork 的一个核心建议：如果你的 Redis 实例内存超过 20GB，务必考虑 sharding——不管是集群还是 proxy 方案，将大实例拆分成多个小实例，每个实例内存控制在 10GB 以内。这不仅仅是 fork 的问题，也关系到故障恢复时间（RDB 加载速度）、内存碎片率等多个维度。

CPU 飙升排查

CPU 飙升是 Redis 最常见且最容易"猜错根因"的问题。我们把常见的 CPU 飙升原因按发生频率排列：

症状特征 可能根因 确认手段 解决方案
CPU 飙升 + slowlog 有慢命令 计算密集型命令（KEYS/SORT/ZRANGE） 查看 slowlog 耗时分布 SCAN 替代 KEYS、增加 LIMIT、控制集合大小
CPU 飙升 + 大量过期键 activeExpireCycle 集中清理 INFO stats 看 expired_keys 速率 TTL 随机化、提高 hz 分散清理频率
CPU 飙升 + 大量写操作 AOF rewrite 子进程竞争 CPU 检查是否有子进程运行（INFO persistence） 限流 AOF rewrite 频率、降低 rewrite 触发阈值
CPU 飙升 + 连接数暴增 断线重连风暴（reconnection storm） CLIENT LIST 看连接 IP 分布 增加 tcp-backlog、调整 client timeout
CPU 平稳但响应慢 fork 阻塞（非 CPU 问题） 检查 latest_fork_usec 分片降低实例内存、关闭 THP

关于"断线重连风暴"再多说几句：当网络抖动或 Redis 主从切换时，大量客户端同时断线又同时重连，建连过程（accept + TLS 握手 + 认证）本身是 CPU 密集的。如果客户端库使用连接池且池大小设得很大，这个风暴会自激式放大——每个客户端重连后立即发出大量请求，Redis 处理请求又导致延迟升高，某些客户端触发超时再次断连，恶性循环。

小贴士：处理断线重连风暴的一个有效手段是客户端侧实现"指数退避"重连策略——第一次重连等待 1 秒，第二次 2 秒，第三次 4 秒……最大不超过 30 秒。这能避免所有客户端在同一时刻同时发起重连。

内存问题排查

内存碎片

前面说过，mem_fragmentation_ratio = used_memory_rss / used_memory。这个值在 1.0~1.3 间属于正常，1.5 以上需要关注。碎片率高的原因主要有几种：

  • jemalloc arena 预分配：jemalloc 会预分配大块内存（类似 malloc 的 arenas），这部分未使用的内存计入 RSS 但不计入 used_memory，导致碎片率虚高。
  • 键值频繁变更：不断 SET/DEL 导致内存申请释放的碎片化。jemalloc 的分级 bucket 基本能应付，但如果 key 的大小分布特别不均匀（比如既有 10 字节也有 1MB），就会出现跨 bucket 的浪费。
  • 大量 key 过期：过期键的批量释放导致内存空洞无法立即回收给操作系统。这时候 MEMORY PURGE 命令可以尝试归还内存给 OS（但并非总是有效）。

内存泄漏

严格来说，Redis 的 C 代码经过十余年打磨，普通的内存泄漏已经很少见了。但有一种"软泄漏"非常普遍——client output buffer 无限增长。当某个消费者客户端消费能力不足，Redis 的回复数据在输出缓冲区中堆积。如果不设置 client-output-buffer-limit，这个缓冲区可能无限制增长：

# 设置普通客户端输出缓冲区限制
client-output-buffer-limit normal 0 0 0

# 设置 pub/sub 客户端限制（默认 32MB，8MB，60 秒）
client-output-buffer-limit pubsub 32mb 8mb 60

关于 pub/sub 的缓冲区尤其要警惕——发布者的消息速率超过订阅者的消费能力时，缓冲区会像吹气球一样膨胀。我记得有一个案例，一个监控系统订阅了一个高吞吐的 Redis channel，但某个订阅者因 bug 挂起后没有及时关闭连接，结果这个连接的输出缓冲区涨到了 3GB，最终直接把 Redis 的 RSS 撑爆。

OOM 问题

一个常见的配置陷阱是：maxmemory 设置得"刚刚好"——比如内存总共 32GB，留 2GB 给系统，设了 30GB。看上去很合理，但问题在于 used_memory 统计的是"申请的内存"，而 RSS 是"物理占用"。由于内存碎片和 jemalloc 的预分配，RSS 通常比 used_memory 高 10~30%。当 used_memory 还差得远时，RSS 可能已经超过 30GB，触发 OOM Killer。建议 maxmemory 设为物理内存的 60~70%，留出足够的余量给碎片和系统。

配置陷阱

最后总结几个 Linux 内核参数的配置陷阱。这些参数在普通应用下无关紧要，但对 Redis 可能是"致命"的：

参数 默认值 推荐值 影响的 Redis 功能
transparent_hugepage/enabled always never fork COW 放大、RSS 膨胀
vm.overcommit_memory 0 1 fork 是否成功
net.core.somaxconn 128 1024+ 高并发建连时的 TCP backlog 队列
tcp-backlog（redis.conf） 511 1024+ Redis 层面的 backlog，需匹配 somaxconn
vm.swappiness 60 1 Redis 被 swap 到磁盘导致灾难性延迟
kernel.sched_migration_cost_ns 500000 0 减少 NUMA 间的 task 迁移延迟

关于 net.core.somaxconn 和 tcp-backlog 的坑：默认的 somaxconn=128 意味着 TCP 层的 accept 队列最多只能排 128 个连接。在高并发场景下，如果 Redis 的 tcp-backlog 设到了 1024，但内核参数 somaxconn 只有 128，实际有效值仍然是 128。多余的连接请求会被内核直接丢弃，客户端表现为"连接超时"。这其实是一个很隐蔽的坑——你看到的是"连接被拒"，但根因是内核参数和 Redis 参数不匹配。
