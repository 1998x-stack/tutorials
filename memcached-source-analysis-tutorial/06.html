第6章 日志与统计系统

大家好，我是你们的讲师。想象一个场景：你的线上 Memcached 集群突然延迟飙升，QPS 骤降——你第一步会做什么？我猜你会先 telnet localhost 11211 然后敲一句 stats。没错，统计数据就是运维人员的"生命线"。这一章我们就来看看 Memcached 的日志与统计系统——它们分别是 logger.c 和 stats.c 两个模块。

日志和统计虽然看似都是"输出信息"，但定位完全不同：日志关心的是"发生了什么异常"，统计关心的是"系统整体有多健康"。前者是对离散事件的记录，后者是对聚合指标的度量。Memcached 对两者都有精巧的实现——特别是它的异步日志系统设计值得你仔细琢磨。

logger.c：环形缓冲 + 异步写

在早期的 Memcached 版本中，日志输出是同步的——直接 fprintf(stderr, ...)。这在低负载时没有问题，但在高并发场景下，fprintf 的系统调用开销会显著拖慢 worker 线程。想象一下：你的 worker 线程每秒处理 10 万次请求，其中 1% 需要写日志——那就是每秒 1000 次 write() 系统调用，这绝对不是一个小数字。

1.5 版本引入的 logger.c 用一个非常优雅的方案解决了这个问题：环形缓冲区 + 后台线程异步写。

核心数据结构

struct logger {
    pthread_t thread_id;       // 后台写日志的线程
    char *filename;            // 日志文件路径
    int fd;                    // 日志文件 fd
    struct logentry *buf;      // 环形缓冲区
    int size;                  // 缓冲区大小（默认 50,000 条）
    int read_pos;              // 消费者（写线程）读位置
    int write_pos;             // 生产者（worker 线程）写位置
    volatile bool enabled;     // 是否启用
};

关键设计思想：生产者无锁。Worker 线程调用 logger_log() 时，只需要原子地将日志条目追加到环形缓冲区的 write_pos 位置。没有锁竞争，没有系统调用，只是一个内存写操作——这在高并发路径上是至关重要的。

工作流程

Worker 线程（生产者）         后台线程（消费者）
      │                            │
      │  logger_log()               │
      │  └─► buf[write_pos] = msg   │
      │  └─► write_pos++            │
      │                            │  定期唤醒
      │                            │  └─► while (read_pos != write_pos)
      │                            │       └─► write(fd, buf[read_pos])
      │                            │       └─► read_pos++
      │                            │
      │  (无锁，无系统调用)           │  (批量 write，减少 syscall)

生产者（worker 线程）和消费者（日志线程）之间通过 read_pos 和 write_pos 两个指针来协调。当 write_pos 追上了 read_pos（缓冲区满了），新的日志条目会覆盖最旧的条目——这就是"环形"的含义。虽然会丢失日志，但这比阻塞 worker 线程要好得多。生产环境通常将缓冲区设得足够大（默认 5 万条），在正常负载下不会丢日志。

日志级别

Memcached 的日志系统有 4 个级别，通过 verbose 配置控制：

级别 对应参数 输出内容 线上推荐
LOG_DEBUG -vvv 几乎所有内部事件，包括每次连接、每次命令 不开启
LOG_INFO -vv 连接建立/关闭、get/set 操作摘要 调试时临时开启
LOG_WARNING -v 异常情况：连接超时、内存吃紧、淘汰事件 建议开启
LOG_ERROR 无条件 系统级错误：内存分配失败、端口绑定失败 始终输出

在源码中，日志宏的定义长这样：

#define LOG_DEBUG   0
#define LOG_INFO    1
#define LOG_WARNING 2
#define LOG_ERROR   3

// 使用示例
logger_log(LOG_WARNING, "Item %s expired from LRU", key);
logger_log(LOG_INFO, "Client connected from %s", inet_ntoa(addr));

注意 LOG_ERROR 是无条件输出的——即使你没有指定任何 -v 参数，系统级错误也会被记录。这确保关键错误永远不会被遗漏。

stats 系统：三大统计体系

Memcached 的统计系统由三个主要模块构成，分别对应三个不同的关注维度：

stats_state：全局运行状态

stats_state 是一个全局结构体（定义在 memcached.h 中），包含系统级的计数器：

struct stats_state {
    uint64_t curr_items;        // 当前存储的 Item 数量
    uint64_t total_items;       // 历史累计存储的 Item 数量
    uint64_t curr_connections;  // 当前连接数
    uint64_t total_connections; // 历史累计连接数
    uint64_t connection_structures; // 分配过的连接结构体数量
    uint64_t cmd_get;           // get 命令累计次数
    uint64_t cmd_set;           // set 命令累计次数
    uint64_t get_hits;          // get 命中次数
    uint64_t get_misses;        // get 未命中次数
    uint64_t evictions;         // 淘汰次数
    uint64_t bytes_read;        // 累计读取字节数
    uint64_t bytes_written;     // 累计写入字节数
    // ...
};

这些计数器全部是 uint64_t 类型，理论上可以累计到 1.8e19——在可预见的未来不会溢出。几乎所有计数器都在 memcached.c 中通过 STATS_LOCK() 宏保护下原子更新。

stats_slabs：Slab 级别的统计

当我们执行 stats slabs 命令时，返回的是每个 slab class 的独立统计信息：

stats slabs
STAT 1:chunk_size 96
STAT 1:chunks_per_page 10922
STAT 1:total_pages 1
STAT 1:total_chunks 10922
STAT 1:used_chunks 5
STAT 1:free_chunks 5
STAT 1:free_chunks_end 10912
STAT 1:mem_requested 400
STAT 1:get_hits 1200
STAT 1:cmd_set 5
STAT 1:delete_hits 0
STAT 1:evictions 0
STAT 2:chunk_size 120
...

这些统计信息来自 slabclass_t 结构体中的 stats 字段。它对于调优非常关键——比如你可以通过 free_chunks_end 看某个 class 是否有大量未使用的 chunk 在 page 末尾（内存浪费），或者通过 evictions 看哪个 class 在频繁淘汰。

stats_items：Item 级别的统计

stats items 返回的是每个 slab class 中 Item 的汇总信息，重点关注 LRU 相关的指标：

stats items
STAT items:1:number 5
STAT items:1:age 3600
STAT items:1:evicted 0
STAT items:1:evicted_nonzero 0
STAT items:1:evicted_time 0
STAT items:1:reclaimed 10
STAT items:1:outofmemory 0
STAT items:1:tailrepairs 0

其中 age 表示该 class 中最旧的 Item 在 LRU 队列中等待了多久（秒）——如果 age 很大但 evictions 很小，说明你的内存配置可能过多了；如果 age 很小且 evictions 很大，说明内存不够用了。

stats 命令的执行流程

当客户端通过 telnet 发送 stats\r\n 时，完整的处理链路如下：

1. libevent 检测到 socket 可读
2. drive_machine() 读取数据到 conn->rbuf
3. try_read_command() 识别出 "stats" 命令
4. dispatch_command() 匹配到 STAT 处理分支
5. 调用 stats_handler() 收集统计数据
6. 将统计结果格式化输出到 conn->wbuf
7. libevent 触发可写事件，发送数据

具体到 stats_handler() 的实现（位于 stats.c）：

void stats_handler(conn *c, const char *args) {
    char *sub command = args;

    if (strcmp(subcommand, "reset") == 0) {
        STATS_LOCK();
        memset(&stats_state, 0, sizeof(stats_state));
        STATS_UNLOCK();
        out_string(c, "RESET");
    } else if (strcmp(subcommand, "slabs") == 0) {
        append_stats_slabs(c);
    } else if (strcmp(subcommand, "items") == 0) {
        append_stats_items(c);
    } else if (strcmp(subcommand, "conns") == 0) {
        append_stats_conns(c);
    } else {
        // 默认：输出所有全局统计
        append_stats_state(c);
    }
}

注意子命令的分发方式：stats reset 重置所有计数器，stats slabs 输出 slab 级统计，stats items 输出 Item 级统计，stats conns 输出每个连接的详细信息，不加子命令则输出全局状态。

stats conns 的实现

stats conns（注意是复数 conns）是运维排查连接问题时非常有用的命令。它遍历所有活跃的连接结构体，输出每个连接的 fd、状态、读写缓冲区大小等信息：

stats conns
STAT 5:addr 127.0.0.1:54321
STAT 5:state conn_read
STAT 5:rbuf 4096
STAT 5:rsize 4096
STAT 5:wsize 0
STAT 5:total_time 120
STAT 6:addr 127.0.0.1:54322
STAT 6:state conn_write
...

它的实现原理是在 append_stats_conns() 中遍历全局 conns 数组（所有连接结构体的指针数组），对每个非空连接输出其状态。遍历过程需要持有 conn_lock，因此在连接数很多（数万）时，stats conns 本身会有一定的性能开销。

踩坑实录：我曾经在一个生产集群上执行 stats conns，结果命令卡住了将近 10 秒才返回。原因是集群有 3 万个连接，而 stats conns 在遍历过程中需要持有 conn_lock——这导致了所有新连接无法被 accept。从那以后，我养成了一个习惯：不在高峰期执行 stats conns，尤其在连接数超过 5000 的实例上。

如何添加自定义 stats 指标

如果你在修改 Memcached 源码做定制开发，很可能需要添加自己的统计指标。步骤很简单：

// 步骤1: 在 stats_state 结构体中添加新字段（memcached.h）
struct stats_state {
    // ... 现有字段
    uint64_t my_custom_counter;  // 新增
};

// 步骤2: 在 stat_defs 数组中添加定义（stats.c）
#define STAT_MYCUSTOM 25
static const char *stat_defs[] = {
    // ... 现有定义
    "my_custom_counter",   // 索引 25
};

// 步骤3: 在需要的地方更新计数器
STATS_LOCK();
stats_state.my_custom_counter++;
STATS_UNLOCK();

// 步骤4: stats 命令自动就会包含这个新指标

更灵活的方式是利用 EXTENSION_STAT 机制——你可以注册一个回调函数，在 stats 输出时被自动调用。这样甚至不需要修改核心代码，非常适合通过动态加载模块来扩展。

动手练习：尝试给 Memcached 添加一个"每秒 get 请求数"的统计指标。提示：需要在 stats_state 中添加一个时间戳字段和一个计数器的组合，每次 cmd_get 更新时检查时间戳是否变化，如果变化了就把上一秒的计数值存入一个单独的字段。stats 命令会把这个新字段一并返回。尝试修改代码、重新编译、用 telnet 验证你的新统计指标。

日志和统计系统是 Memcached 中"看不见的幕后工作者"——它们不直接参与核心的缓存逻辑，但没有它们，运维和调优将无从下手。读完这一章，你应该已经理解了环形缓冲区在日志系统中的应用、三大统计体系的各自定位，以及 stats 命令从接收到返回的完整链路。
