第6章 特征工程（上）

大家好，我是你们的讲师。前五章我们都在讲模型——协同过滤、矩阵分解、SVD、ALS。模型当然重要，但如果你问一个在大厂做了五年推荐系统的工程师"什么对推荐效果影响最大"，他很可能会告诉你——是特征工程，不是模型。

我见证过一个真实的case：一个团队花了一个月把模型从DeepFM换成DCNv2，离线AUC涨了0.2个点；而另一个团队只用了一周时间，加了一组"过去7天用户所属城市的热门品类点击率"特征，AUC直接涨了1.5个点。特征工程的"性价比"就是这么悬殊。

这一章和下一章我们分成上下两部分来讲特征工程。这一章（上）聚焦离线特征——用户画像、物品属性、统计特征、特征交叉和特征选择；下一章（下）聚焦实时特征、Embedding特征和Feature Store工程实践。

1. 推荐系统的四类特征

推荐系统的特征可以从来源分为四大类。理解这四类特征，是设计特征体系的第一步。

特征类别 具体例子 特点 数据来源
用户特征（User） 年龄、性别、城市、注册时长、会员等级、历史购买品类分布 相对稳定，变化缓慢 用户注册信息、行为日志聚合
物品特征（Item） 商品类目、价格区间、品牌、标签、上架时间、卖家信息 相对稳定，部分经常变化（如价格） 商品信息表、内容管理系统
上下文特征（Context） 当前时间（星期几、是否节假日）、用户设备类型（iOS/Android）、网络类型（WiFi/4G）、地理位置 每次请求不同，强时效性 在线请求参数、设备信息
交互特征（Interaction） 用户对该物品的过去点击次数、曝光次数、购买次数、平均停留时长 动态变化，信号最强但容易"带偏" 用户行为日志

在精排模型中，四类特征都会用到。但在召回阶段，特征的选择策略不同——召回模型通常只使用用户特征和物品特征的一部分（比如双塔模型中，用户侧用用户特征+上下文特征，物品侧用物品特征）。

一个值得注意的趋势是：随着深度学习模型的表达能力越来越强，模型对"人工精细特征"的依赖在降低，但远没有到"不需要特征工程"的程度。好的特征仍然比好的模型更容易带来业务收益——特别是在模型的训练数据不够大、或者业务场景比较特殊的场景下。

2. 统计特征设计

统计特征（Statistical Features）是推荐特征工程中最基础也最有效的一类特征。它们通过对用户历史行为做聚合统计，捕捉用户对商品的偏好强度。

2.1 "3个周期 × 5种行为"框架

工业界有一个非常实用的特征设计框架：对每个行为类型，统计3个时间窗口下的聚合值。

三个时间周期：

  • 短期（Short-term）：过去3天或1周——反映用户最近的即时兴趣
  • 中期（Mid-term）：过去7天或14天——反映用户近期的稳定兴趣
  • 长期（Long-term）：过去30天或90天——反映用户的长期偏好

五种常见行为（从弱到强）：

  • 曝光（Impression）：物品被展示给用户的次数
  • 点击（Click）：用户点击了物品（弱正反馈）
  • 点赞/收藏（Like/Favorite）：明确的兴趣表达（中等正反馈）
  • 评论/分享（Comment/Share）：社交参与行为（强正反馈）
  • 购买/转化（Purchase/Conversion）：最强烈的正反馈

3个周期 × 5种行为 = 15个基础统计量。如果再加上"按品类聚合"（用户在过去3天点击了多少"运动鞋"类目下的商品），特征数量很容易扩展到几十甚至上百个。

2.2 比率特征

除了计数类特征，比率特征往往更有信息量：

  • CTR：点击次数 / 曝光次数（用户对物品的真实喜好程度，排除曝光偏差）
  • CVR：转化次数 / 点击次数（从点击到购买的转化效率）
  • 点赞率：点赞次数 / 点击次数（满意度指标）
  • 跳出率：停留 < 3秒的点击次数 / 总点击次数（负反馈信号）

需要注意：比率特征在样本量不足时波动很大。比如一个商品只曝光了2次被点击了1次，CTR = 50%，但显然这个数字不可靠。通常的做法是加上"平滑"处理（如贝叶斯平滑）：

# 平滑CTR = (点击次数 + α) / (曝光次数 + α + β)
# 其中 α 和 β 是先验参数（通常取全局平均CTR的对应值）
def smoothed_ctr(clicks, impressions, global_ctr=0.05, prior_strength=100):
    alpha = global_ctr * prior_strength
    beta = prior_strength - alpha
    return (clicks + alpha) / (impressions + alpha + beta)

真实故事：「加特征」的力量

2019年，某电商平台的内容推荐团队做了一次非常"朴素"但效果惊人的实验。他们在原有的DeepFM模型之上，只增加了两组特征：第一组是"用户在过去3/7/30天内对各个一级类目的点击量"，第二组是"用户在过去3/7/30天内对各个一级类目的购买量"。就是这几行SQL聚合出来的特征，上线后CTR直接提升了接近一倍（翻倍！）。事后分析原因：原有模型主要依赖ID Embedding和上下文特征，缺乏对用户"类目偏好强度"的定量描述。加了类目统计特征后，模型终于能分辨"偶尔看看运动类目的用户"和"重度运动爱好者"之间的区别。这个案例完美说明了：在追求复杂模型之前，先把基础统计特征做扎实。

2.3 分布特征

除了总计数和比率，用户行为的"分布形态"也包含重要信息。比如：

  • 用户点击最多的Top-3类目：了解用户的核心兴趣领域
  • 用户最近交互物品的类目熵值：熵值高说明用户兴趣广泛，熵值低说明用户偏好专注
  • 用户购买商品的价格分布（均值、中位数、标准差）：了解用户的消费能力层级
  • 用户活跃时段分布：是"上班摸鱼型"还是"深夜刷机型"

3. 特征交叉

3.1 为什么需要特征交叉

单个特征的信息量是有限的。比如"用户年龄=25"和"商品类目=运动鞋"各自只能提供粗糙的信息，但两者结合在一起——"25岁的用户对运动鞋的兴趣"——才是真正有预测力的信号。

线性模型（如LR）无法自动学到这种特征组合模式。所以特征交叉（Feature Crossing / Feature Interaction）就成了必须的人工工作。

3.2 人工特征交叉的形式

  • AND交叉：用户性别=男 AND 商品类目=电子产品 → 一个新的二值特征。这是最常见的形式。
  • 笛卡尔积：用户年龄段 {青年,中年,老年} × 商品类目 {食品,服装,电子} → 9个新特征。维度随交叉特征数量指数级增长。
  • 数值特征的乘积/比值：商品价格 × 用户历史平均购买价格 → 一个连续特征，刻画"价格匹配度"。

3.3 组合爆炸问题

假设有1000个用户维度特征 × 1000个商品维度特征，做笛卡尔积会得到100万个交叉特征——这在计算和存储上都是不可接受的。这就是为什么学术界后来提出了FM等模型来自动化特征交叉（下一章会详述）。但在实践中，人工选择"高价值"的交叉仍然有效：

  • 优先选择粒度不太细的维度做交叉（比如用"一级类目"而不是"商品ID"）
  • 优先选择"业务直觉上有关联"的特征组合
  • 使用树模型（如GBDT）的特征重要性来选择值得交叉的特征对

4. 特征选择与重要性分析

特征不是越多越好。冗余特征会增加训练成本、带来过拟合风险、甚至引入噪声。特征选择的价值在于：用最少的特征达到最好的效果。

4.1 常用的特征选择方法

  • 信息价值（Information Value, IV）：评估一个特征对二分类目标的区分能力。IV > 0.3 表示特征很强，0.1-0.3 中等，< 0.1 较弱。
  • 特征重要性（Feature Importance）：训练一个树模型（LightGBM/XGBoost），直接输出每个特征的分裂增益总和。
  • 消融测试（Ablation Test）：从完整特征集中逐个删除特征，观察指标变化。这是最直接但也最耗时的方法。
  • 相关性分析：计算特征与目标之间的相关系数，或者特征两两之间的相关性（高相关的特征可以合并或删除其一）。

4.2 特征工程的成本意识

每一个特征都有"成本"：

  • 计算成本：这个特征的计算需要多少CPU/内存？会不会影响在线延迟？
  • 存储成本：这个特征需要多大空间？能否压缩或使用近似值？
  • 维护成本：这个特征的数据源是否稳定？上游表变更了怎么办？
  • 时效成本：这个特征是T+1计算还是实时计算？T+1意味着今天模型看不到最新的信息。

一个好的特征应该同时具备：强预测力、低计算成本、稳定的数据源。三者缺一，就需要重新评估是否值得上线。

import pandas as pd
import numpy as np

def compute_statistical_features(behavior_log, user_id_col, item_id_col, time_col, action_col):
    """
    从用户行为日志中计算常用统计特征
    behavior_log: DataFrame, 包含 userId, itemId, timestamp, action_type 等列
    """
    now = behavior_log[time_col].max()

    features = []

    for window in [3, 7, 30]:
        mask = (now - behavior_log[time_col]).dt.days <= window
        window_data = behavior_log[mask]

        # 按用户聚合
        user_stats = window_data.groupby(user_id_col).agg(
            **{
                f'{action_col}_count_{window}d': (action_col, 'count'),
                f'unique_items_{window}d': (item_id_col, 'nunique'),
            }
        ).reset_index()

        # 按用户+物品聚合（用户对这个物品的交互强度）
        user_item_stats = window_data.groupby([user_id_col, item_id_col]).agg(
            **{f'interaction_count_{window}d': (action_col, 'count')}
        ).reset_index()

        features.append(user_stats)
        features.append(user_item_stats)

    # 合并所有窗口的特征
    result = user_item_stats
    for f in features[1:]:
        result = result.merge(f, on=[user_id_col, item_id_col], how='left')

    return result.fillna(0)

# 计算比率特征
def compute_ratio_features(feature_df, numerator_col, denominator_col, new_col_name):
    """安全地计算比率特征，避免除零"""
    mask = feature_df[denominator_col] > 0
    feature_df[new_col_name] = 0.0
    feature_df.loc[mask, new_col_name] = (
        feature_df.loc[mask, numerator_col] / feature_df.loc[mask, denominator_col]
    )
    return feature_df

工业小贴士：特征日志的重要性

一个新手常犯的错误：上线新特征之后没有记录该特征的值分布。等模型出了问题，想排查"是不是这个新特征的问题"时，发现没有任何历史记录可以回溯。我的建议是：所有新上线的特征，在在线推理时都要落到日志里。这样当模型出现问题时，你可以对比"出问题的样本"和"正常样本"在特征分布上的差异，快速定位根因。不记日志的特征，等于没有上线。
