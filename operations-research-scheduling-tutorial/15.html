第15章 约束规划与CP-SAT

大家好，我是你们的讲师。在上一章我们讲到，元启发式算法虽然能在合理时间内给出不错的解，但它有个让人头疼的问题——它不告诉你这个解距离最优还有多远。说白了，你得到了一个"看着挺好"的方案，但不知道它是不是真得好。那么，有没有一种方法，既保留了强大的约束表达能力，又能给出最优性证明？答案是肯定的——这就是约束规划（Constraint Programming, CP），尤其是 Google OR-Tools 中的 CP-SAT 求解器。

我第一次接触 CP 是在多年前的一个项目中。当时团队要用 MILP 求解车间调度问题，模型写了上百行约束，Gurobi 跑了半小时才找到可行解，而且 gap 一直降不下去。团队里一位老工程师说："要不试试 CP？"我当时心想——一个"约束"怎么能和"数学规划"比？结果 CP-SAT 不到 10 秒就给出了一个更好的解，而且证明了最优。那天我对求解器的认知被彻底颠覆了。

CP vs MILP — 两种建模范式的根本区别

MILP（混合整数线性规划）和 CP（约束规划）的终极目标是一样的——求解优化问题。但它们的底层哲学完全不同，理解这一点是正确选型的前提。

MILP 的思路是松弛+分支：先把整数变量松驰为连续变量，用单纯形法求下界，然后通过分支定界逐步收紧。它的核心数学工具是线性不等式。MILP 擅长处理"线性结构明显"的问题——比如物流网络中的运输成本最小化、投资组合优化等。

CP 的思路是域传播+搜索：每个变量有一个取值域（domain），约束会不断剪掉域中不可能的值（这就是"传播"）。当所有约束都无法再剪域时，就进行分支搜索。CP 的核心武器是全局约束（Global Constraints）——比如 all_different、noOverlap，这些约束在传播时能利用问题结构大量剪枝，远比 MILP 拆成若干线性约束更高效。

用一句话总结：MILP 把问题压成线性不等式然后求松弛，CP 把约束当作剪枝器来缩小搜索空间。这导致了一个非常实用的结论：如果一个问题的约束是非线性的、高度结构的、或者逻辑性的（比如安排几个任务不能重叠），CP 通常更自然；如果一个问题的核心是线性目标加线性约束（比如最小化运输成本），MILP 可能更高效。

下面这张表可以帮助你快速判断：

特征 MILP CP
核心机制 线性松弛 + 分支定界 域传播 + 回溯搜索
变量类型 连续、整数、二进制 整数、区间、序列、集合
约束表达力 线性不等式 全局约束（all_different, noOverlap, cumulative）
找可行解速度 中等（依赖松弛质量） 快（全局约束强传播）
最优性证明 强（线性松弛提供紧下界） 弱（无自然松弛）
适用场景 线性目标、连续决策问题 调度、排班、组合优化

看到这里你会发现，对于调度问题这种"约束密集"的场景，CP 天然有优势——因为调度本质上就是给每个工序分配"在哪个时间区间执行"的决策，而"两个工序不能同时使用一台机器"就是天然的 noOverlap 约束。

我的经验之谈：如果你的调度问题涉及复杂的"辅助约束"——比如操作工只能操作特定机床、某些工序需要特殊工装夹具、或者物料有时间窗口限制——用 MILP 来建模会非常痛苦，因为每加一个这类约束就要加大量辅助变量和大 M 约束。而 CP 中这样的约束往往用一条 AddAllowedAssignments 之类的高层约束就解决了。这就是为什么在工业调度中，CP 写模型的速度往往是 MILP 的 3-5 倍。

CP-SAT 的核心概念

CP-SAT 是 Google OR-Tools 中的旗舰求解器。它底层的核心机制仍然是"约束传播 + 搜索"的 CP 范式，但做了三个革命性的改进：引入了SAT（布尔可满足性）技术作为传播引擎、加入了LNS（大邻域搜索）作为启发式探索、并且内置了对区间变量（IntervalVar）的一等公民支持。下面我们来逐一攻破。

区间变量（IntervalVar）—— 这是 CP-SAT 为调度领域量身打造的变量类型。一个 IntervalVar 代表一个"有开始时间和结束时间"的活动。它有三个基本属性：开始时间（start）、结束时间（end）、持续时间（duration）。在代码中你创建一个 IntervalVar 只需：

# 创建一个固定持续时间的区间变量（start 由求解器决定）
interval = model.NewIntervalVar(start, duration, end, 'name')

# 创建一个可选区间变量（可以不被选中执行）
optional = model.NewOptionalIntervalVar(start, duration, end, presence, 'name')

# 创建一个固定 start/end 的区间
fixed = model.NewFixedSizeIntervalVar(start, duration, 'name')

全局约束—— 这是 CP-SAT 的杀手锏。来看调度中最常用的三个：

  • noOverlap：给定一组区间变量，保证它们之间不重叠。这精确地描述了"一台机器一次只能处理一个工序"的场景。不需要像 MILP 那样做"排他约束"。
  • cumulative：给定一组区间变量和一个资源容量，保证在任何时间点上，正在执行的区间消耗的资源总量不超过容量。这描述的是"人力池"或"工具池"——比如车间只有 3 名高级技工，同一时间最多有 3 个需要高级技工的工序在加工。
  • circuit：在图上找到一个哈密顿回路。这描述的是"访问所有地点各一次"的 TSP 约束——用于带路径决策的调度问题。

可选区间和逻辑约束—— 现实调度中最棘手的部分就是"不确定做不做"的情况。比如某个工单有可能插单，或者某台设备可能停机维修。CP-SAT 的 OptionalIntervalVar 完美地对应了这种场景：presence 是一个布尔变量，当其值为 True 时该区间才被激活，否则该区间不参与任何约束传播。在这个机制之上，你可以轻松建模"如果执行任务 A 就不能执行任务 B"这样的逻辑约束：

# 两个可选任务不能同时被执行
model.AddBoolOr([presence_a.Not(), presence_b.Not()])

深入理解：CP-SAT 中的"SAT"部分指的是底层传播引擎大量借鉴了 SAT 求解技术——子句学习（Clause Learning）、冲突驱动回溯（Conflict-Driven Backtracking）、两级 Boolean 传播（Two-Watched Literal）等。这使得 CP-SAT 在遇到复杂约束组合时，表现远好于传统 CP 求解器。2026 年版本的 CP-SAT 还引入了自动对称性检测和动态重启策略，进一步缩小了对商业求解器的差距。

用 CP-SAT 建模调度问题

我们亲自动手，用 CP-SAT 建模一个完整的作业车间调度问题（JSP）。你将目睹 CP-SAT 的建模有多么简洁。

from ortools.sat.python import cp_model

def solve_jsp_cp_sat(jobs_data):
    """
    jobs_data: list of jobs, each job is a list of (machine_id, duration) tuples
    Example: 3 jobs × 3 machines
    jobs_data = [
        [(0, 3), (1, 2), (2, 2)],   # Job 0
        [(0, 2), (2, 1), (1, 4)],   # Job 1
        [(1, 4), (2, 3)],           # Job 2
    ]
    """
    model = cp_model.CpModel()
    machines_count = 3
    horizon = 0  # 时间上界

    # ---- 1. 创建区间变量 ----
    all_intervals = {}
    for job_id, job in enumerate(jobs_data):
        for task_id, (machine, duration) in enumerate(job):
            suffix = f'J{job_id}_T{task_id}'
            start = model.NewIntVar(0, 10000, f'start_{suffix}')
            end = model.NewIntVar(0, 10000, f'end_{suffix}')
            interval = model.NewIntervalVar(start, duration, end, f'interval_{suffix}')
            all_intervals[(job_id, task_id)] = (start, end, interval, machine)
            horizon += duration

    # ---- 2. 同一台机器上无重叠 ----
    machine_to_intervals = {m: [] for m in range(machines_count)}
    for (j, t), (_, _, interval, machine) in all_intervals.items():
        machine_to_intervals[machine].append(interval)

    for machine, intervals in machine_to_intervals.items():
        model.AddNoOverlap(intervals)

    # ---- 3. 工件内工序先后顺序 ----
    for job_id, job in enumerate(jobs_data):
        for task_id in range(len(job) - 1):
            _, end_prev, _, _ = all_intervals[(job_id, task_id)]
            start_next, _, _, _ = all_intervals[(job_id, task_id + 1)]
            model.Add(end_prev <= start_next)

    # ---- 4. 目标：最小化 makespan ----
    makespan = model.NewIntVar(0, horizon, 'makespan')
    all_ends = [end for _, (_, end, _, _) in all_intervals.items()]
    model.AddMaxEquality(makespan, all_ends)
    model.Minimize(makespan)

    # ---- 5. 求解 ----
    solver = cp_model.CpSolver()
    solver.parameters.max_time_in_seconds = 30.0
    status = solver.Solve(model)

    if status == cp_model.OPTIMAL or status == cp_model.FEASIBLE:
        print(f'Makespan: {solver.Value(makespan)}')
        for (j, t), (start, end, _, machine) in all_intervals.items():
            print(f'Job{j} Task{t} on M{machine}: '
                  f'{solver.Value(start)} -> {solver.Value(end)}')
    else:
        print('No solution found.')

对比第 12 章我们写过的 JSP 的 MILP 模型——你注意到差别了吗？MILP 中需要大量的大 M 约束来确保同一台机器上的工序不重叠，而 CP-SAT 只需要一行 model.AddNoOverlap(intervals)。这就是全局约束的力量。

不仅如此，CP-SAT 还可以轻松扩展以下实际场景：

  • 工具约束：在 existing 的机器约束之上，再加一层 cumulative 约束限定某些特殊工具同时使用的数量上限。
  • 人员排班：每个员工一次只能上一种班次——用 all_different 或 AddAllowedAssignments 约束每个时间段的人员分配。
  • 设置时间：工序之间的切换需要 setuptime——可以用 AddCircuit 或 model.Add + 辅助变量的方式建模工序之间的切换关系。

CP-SAT 求解器内部原理

既然你已经看到了 CP-SAT 的外在表现，我们简单聊聊它的"黑箱内部"在做什么。

CP-SAT 的求解流程大致是这样：

  1. 读取模型，将所有变量和约束转化为 SAT 层级的一阶逻辑子句和传播器。
  2. 运行域传播（Domain Propagation）：每个约束的传播器反复将不可能的值从变量的域中移除。这一步是"推理"而非"搜索"——不产生分支，只做逻辑推导。
  3. 当传播达到不动点时，进入LNS（大邻域搜索）阶段：随机选择一个"邻域"（通常是部分变量固定、部分变量释放），在这个缩小后的子问题上做进一步搜索。本质上是在"推理+小搜索"和"探索性大搜索"之间来回切换。
  4. 过程中持续记录找到的最佳解，并用证明最优性的下界来评估 gap。

2026 年的 CP-SAT 已经发展到了相当恐怖的程度。根据 OR-Tools v9.12 发布时的 benchmark 数据，在 MinePump、RCPSP、JSP 等经典调度问题上，CP-SAT 的表现已经超过了大多数开源的 MILP 求解器，并且在某些类别上紧追 Gurobi 和 CPLEX。尤其值得注意的是，CP-SAT 在 RCPSP（资源约束项目调度问题）上表现极为亮眼——它能够在 Gurobi 找不到可行解的问题上快速给出高质量解。

讲个真实的数据点：2025 年的 OR 社区 benchmark 中，有一个著名的 RCPSP 实例集 PSPlib（j120 系列）。在这个集合上，CP-SAT 在 60 秒内找到了 85% 实例的最优解，而 Gurobi 11 在同样时间内只有 52%。更关键的是，有 8 个实例 Gurobi 在 300 秒内连可行解都找不到——CP-SAT 却在 10 秒内给出了解。这不是说 MIP 求解器不行，而是说选对工具比工具本身更重要。CP-SAT 对调度的局部结构（区间、重叠、资源容量）有原生的支持，这就是它的核心优势。

好的，到这一步我们已经完整地理解了约束规划和 CP-SAT 的原理、用法和适用场景。CP-SAT 让你用更少的代码、更短的时间写出高质量的调度求解器。但问题来了——在一个生产级项目中，CP-SAT 不是唯一的求解器。Gurobi、CPLEX、COPT、SCIP……这么多求解器怎么选？它们各有什么强项？如何在一个项目中使用多个求解器？下一章，我们来一场主流求解器实战，带你看清全局。
