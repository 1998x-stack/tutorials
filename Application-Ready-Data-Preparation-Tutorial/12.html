第12章 函数合成驱动的清洗

大家好，我是你们的讲师。前面几章我们一直围绕着"LLM直接操作数据"这个思路展开：LLM逐行检测错误、LLM逐行修复缺失值。这种方式在逻辑上很自然，但在工程上并不高效——每一条数据都要调用一次LLM，成本和延迟都无法接受。有没有一种方法，让LLM不直接操作数据，而是生成操作数据的代码？一次生成，反复执行，既高效又可审计。这就是函数合成驱动的清洗思路。

从检测到修复的鸿沟

在实际数据清洗项目中，"找出错误"和"修复错误"之间隔着一道巨大的鸿沟。很多团队能做好前者，却做不好后者。原因很简单：检测是一个判断问题，修复是一个执行问题。

判断问题只需要回答"是/否有错"——这是一个分类任务。修复问题需要回答"正确的值应该是什么"——这是一个生成任务。对于LLM来说，生成任务的难度远高于分类任务，因为生成不仅要准确，还要保持整个数据集的一致性和可复现性。

另外还有一个更实际的工程约束：在绝大多数企业的数据流水线中，数据清洗步骤需要是可复现、可审计、可版本控制的。如果用LLM直接逐行修复数据，不可能做审计——今天的修复和明天的修复可能完全不同。而函数合成的方案完美解决了这个问题：生成的代码本身是可版本控制的，同样的代码在同样输入上永远给出同样的输出。

关键区别：LLM直接清洗 = 操作数据；函数合成 = 操作代码。后者将LLM从"数据操作者"降级为"代码生成者"，将执行权交还给确定性程序。数据工程师审核LLM生成的代码（一次审核），然后安心运行（反复执行）。

LLM合成数据清洗函数

函数合成的核心思路是：LLM根据数据样本和错误描述，生成一个可以自动修复数据的Python函数。这个函数可以被集成到数据流水线中，替代繁琐的手动清洗步骤。

我们来看一个具体例子。假设数据中发现了一个模式性问题："用户手机号字段混入了多种格式，有些是11位纯数字，有些是加了国家码的格式，有些含有分隔符。"

任务：生成一个数据清洗函数

问题描述：
"phone"字段的数据格式不统一，影响了后续的系统对接。
当前出现的格式包括：
- "13800138000" (标准11位)
- "+86 138 0013 8000" (带国家码和空格)
- "86-13800138000" (带国家码和连字符)
- "00138000" (仅8位，缺少区号)
- "138-0013-8000" (带连字符)

要求：
1. 将所有格式统一为11位纯数字格式，去掉国家码
2. 对于不足11位的号码，标注为无效并输出到日志
3. 输出一个Python函数 clean_phone(phone: str) -> str
4. 函数需要有类型注解和详细的docstring

LLM生成的代码通常具备良好的结构和清晰的逻辑。以下是一个典型的LLM输出片段：

import re

def clean_phone(phone: str) -> str:
    """
    统一手机号格式为11位纯数字。

    支持的输入格式：
    - 11位纯数字: 13800138000
    - 带国际码: +86 138 0013 8000, 86-13800138000
    - 带分隔符: 138-0013-8000

    参数:
        phone: 原始手机号字符串

    返回:
        标准化的11位手机号，无效时返回空字符串

    日志:
        记录所有被标记为无效的手机号
    """
    if not phone or not isinstance(phone, str):
        return ""

    # 移除非数字字符，但保留加号用于识别国家码
    cleaned = re.sub(r'[\s\-\(\)]', '', phone)

    # 处理带国家码的情况
    if cleaned.startswith('+86'):
        cleaned = cleaned[3:]
    elif cleaned.startswith('86') and len(cleaned) > 11:
        cleaned = cleaned[2:]

    # 只保留数字
    cleaned = re.sub(r'\D', '', cleaned)

    # 长度验证
    if len(cleaned) == 11:
        return cleaned
    else:
        print(f"[WARN] 无效手机号: {phone} → 清理后为 {cleaned}")
        return ""

验证规则的自动生成

除了修复函数，LLM还可以合成验证规则。一个常见的模式是"发现错误 → 总结规则 → 自动验证"的闭环。

具体流程如下：LLM先对一批手动标记的错误行进行分析，总结出错误模式，然后将这些模式编译成可执行的验证函数。这些验证函数随后被自动部署到数据入口，成为实时数据质量监控的一部分。

生产实践：在一个电商数据平台中，数据团队使用LLM每两周生成一批新的验证规则。规则部署后，通过监控"规则触发率"来评估效果。如果某条规则连续一周触发率为零，系统自动标记该规则为"可能已过时"并提示团队审查。这种"规则生命周期管理"将规则维护成本降低了约60%。

程序化数据修复模式

函数合成不仅可以处理单个字段的问题，还可以解决复杂的跨字段修复。我们先来对比一下不同修复模式的特性。

修复模式复杂度LLM合成难度可复现性典型例子
变换函数低低高日期格式统一、单位换算
标准化脚本中中高同义词映射、名称标准化
条件修复高中高跨字段联合约束修复
规则验证器中低高数据质量监控规则生成

以下是几种常见的程序化修复模式。

变换函数：将一个字段的格式统一转换为目标格式。最常见的应用包括日期格式标准化、地址标准化、单位换算。例如将所有"2025/01/01"和"01-01-2025"统一为ISO 8601格式。

标准化脚本：将同义词、缩写、别名统一为标准表达。例如"USA"、"United States"、"美利坚合众国"统一为"美国"。标准化的关键在于维护一个映射表，LLM可以帮助自动生成这个映射表。

条件修复：根据多个字段的联合条件进行修复。例如"如果status列的值为'已完成'但end_date列的值为空，则将end_date填充为当前日期并记录日志"。这种条件逻辑是LLM最擅长的代码生成类型。

不过，函数合成并非万能。当错误模式高度多样化、无法被几行代码概括时，函数合成的效果就会打折扣。这时需要结合之前的LLM逐行修复方法做分层处理。下一章，我们将进入一个与错误检测密切相关但有着本质区别的话题——异常检测。虽然听起来很像，但"异常"和"错误"其实是两种不同的概念。
