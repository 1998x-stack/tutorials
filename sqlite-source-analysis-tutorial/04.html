第4章 核心数据结构

大家好，我是你们的讲师。上一章我们建立了 SQLite 源码的"地形图"——知道了哪个文件在哪个位置、完成什么功能。但建筑工地上光有地点图是不够的，你还得知道用什么"建筑材料"。

在 C 语言的项目中，"建筑材料"就是 数据结构。SQLite 的源码之所以被公认为高质量 C 代码的教科书，很大程度上得益于它精心设计的数据结构体系。本章我们深入分析五个核心结构体——它们覆盖了从数据库连接、SQL 编译、模式管理、查询表示到执行状态的完整生命周期。理解了这五个结构体，你就能读懂 SQLite 源码中 80% 的内部逻辑。

sqlite3 — 万能的数据库连接

所有 SQLite 操作的起点都是一个 sqlite3 对象。当你调用 sqlite3_open() 时，SQLite 分配并初始化这个庞大的结构体。它定义在 src/sqliteInt.h 中，是整个源码中最重要的"上帝对象"。

struct sqlite3 {
    sqlite3_vfs *pVfs;            /* VFS 层实例 */
    int nDb;                      /* 数据库数量 */
    Db *aDb;                      /* 数据库数组，aDb[0] 是 main */
    int flags;                    /* 打开标志位 */
    int errCode;                  /* 最近一次错误码 */
    char *zErrMsg;                /* 错误消息字符串 */
    Lookaside lookaside;          /* Lookaside 内存分配器 */
    CollSeq *pDfltColl;           /* 默认排序规则 (BINARY) */
    int nChange;                  /* 最近修改影响的行数 */
    int nTotalChange;             /* 所有修改的影响行数总和 */
    int openFlags;                /* sqlite3_open_v2 的标志 */
    int nVDBe;                    /* 正在执行的 VDBE 数量 */
    u8 mDbFlags;                  /* 数据库标志位 */
    u8 dbOptFlags;                /* 优化标志位 */
    int nSqlExec;                 /* 当前 SQL 语句数 */
    Hash aModule;                 /* 注册的虚拟表模块 */
    int nExtension;               /* 加载的扩展数 */
    void **aExtension;            /* 扩展句柄数组 */
    int nMaxSorterMmap;           /* 排序器最大 mmap 大小 */
    sqlite3_mutex *mutex;         /* 连接级互斥锁 */
    BusyHandler busyHandler;      /* 忙处理器回调 */
    ...
};

这里有一个很重要的概念：aDb[] 是一个数组，每个元素代表一个附加的数据库。默认情况下只有 aDb[0]（main 数据库）和 aDb[1]（temp 临时数据库）。当你执行 ATTACH 'file.db' AS db2 时，SQLite 会在 aDb[] 中添加一个新元素。每个 Db 对象包含指向 B-Tree 的句柄 pBt 和数据库模式信息 pSchema。

💡 冷知识： sqlite3 结构体有超过 100 个字段。你没有看错——确实超过 100 个。但这并不是设计缺陷。因为 sqlite3 本质上是整个数据库引擎的"上下文"，它必须持有所有模块的引用：VFS、缓存、互斥锁、错误状态、内置函数、统计信息等等。有趣的是，由于字段数量的膨胀，D. Richard Hipp 在 3.20.0 版本中专门进行了一次"减肥"——把一些不常用的字段移到了扩展结构体中。

Parse — 编译期的总指挥

Parse 结构体是 SQL 编译阶段的"临时中央政府"。它只在 sqlite3_prepare_v2() 调用期间存在，编译完成后就被释放。它定义同样在 sqliteInt.h 中。

struct Parse {
    sqlite3 *db;                  /* 所属数据库连接 */
    int nErr;                     /* 错误数量 */
    char *zErrMsg;                /* 错误消息 */
    Token sLastToken;             /* 最近扫描的 token */
    Vdbe *pVdbe;                  /* 正在生成的 VDBE 程序 */
    int ckptMode;                 /* 检查点模式 */
    u8 eOrconf;                   /* ON CONFLICT 策略 */
    u8 disableTriggers;           /* 禁用触发器标志 */
    Table *pNewTable;             /* CREATE TABLE 构建中的新表 */
    Trigger *pNewTrigger;         /* CREATE TRIGGER 构建中的新触发器 */
    Select *pSelect;              /* 当前 SELECT 语句 */
    int nLabel;                   /* 标签数量 */
    int *aLabel;                  /* 标签地址数组 */
    ExprList *pConstExpr;         /* 常量表达式列表 */
    int nMem;                     /* 使用的内存单元数 */
    int nSet;                     /* 使用的集合数 */
    int nTab;                     /* 使用的表游标数 */
    int nCacheLine;               /* 缓存行大小 */
    int nLoop;                    /* 嵌套循环层数 */
    u8 explain;                   /* EXPLAIN 标志 */
    ...
};

Parse 的生命周期非常清晰：

  1. 创建 — sqlite3ParseObjectInit() 初始化 Parse 结构。
  2. SQL 分析 — 调用 sqlite3ParserParse()（Lemon 生成的解析器）进行语法分析，Parse 结构被传递给解析器的回调函数，逐步填充。
  3. 代码生成 — 在解析过程中，解析器回调函数调用 sqlite3Select()、sqlite3Insert() 等代码生成器，这些生成器将 VDBE 指令追加到 Parse->pVdbe 指向的程序中。
  4. 完成 — sqlite3ParseObjectFinish() 或 sqlite3ParseObjectReset() 释放资源。

这里有一个非常优雅的设计：当解析过程中遇到错误时，Parse 会继续工作但标记 nErr > 0。后续的代码生成器函数通过检查 Parse->nErr 来判断是否需要提前终止。这种"优雅降级"（graceful degradation）的方式，让 SQLite 可以在一次 prepare 调用中报告多个语法错误，而不是遇到第一个错误就崩溃退出。

Table 与 Index — 数据的骨架

数据库的"表"和"索引"在内存中用 Table 和 Index 结构体表示。它们描述了数据库的模式（schema），是所有 DML 操作的基础。

struct Table {
    char *zName;                  /* 表名 */
    int nCol;                     /* 列数 */
    Column *aCol;                 /* 列信息数组 */
    Index *pIndex;                /* 索引链表 */
    Select *pSelect;              /* 视图的 SELECT 定义 */
    FKey *pFKey;                  /* 外键链表 */
    int tnum;                     /* B-Tree 根页号 */
    u32 nRowEst;                  /* 估计行数 */
    u8 keyConf;                   /* 主键冲突策略 */
    u8 tabFlags;                  /* 表标志位 */
    ExprList *pCheck;             /* CHECK 约束表达式列表 */
    int nModuleArg;               /* 虚拟表参数数 */
    char **azModuleArg;           /* 虚拟表参数字符串 */
    int nCol;                     /* 列数 */
    u16 iPKey;                    /* 主键列索引 */
    ...
};

struct Index {
    char *zName;                  /* 索引名 */
    int nKeyCol;                  /* 索引列数 */
    int *aiColumn;                /* 索引列在表中的序号 */
    u8 *aSortOrder;               /* 排序方向 (ASC/DESC) */
    int *aiRowLogEst;             /* 对数估计 */
    Index *pNext;                 /* 属于同一表的下一个索引 */
    Table *pTable;                /* 所属的表 */
    int tnum;                     /* 该索引的 B-Tree 根页号 */
    u8 bNoQuery;                  /* 索引是否覆盖查询 */
    u8 bUnordered;                /* 无序索引标志 */
    ...
};

关键要理解 Table.tnum 和 Index.tnum 的含义。在 SQLite 中，每个表和一个索引都对应一个独立的 B-Tree，由其根页号（root page number）标识。tnum 就是这个根页号。表的主键 B-Tree 的 key 是主键值、data 是整行数据；索引 B-Tree 的 key 是索引列值、data 是对应行的主键值。

Column 结构体描述了单列的信息。SQLite 的类型系统用 affinity（类型亲和性）而不是严格的静态类型。每列有一个 affinity 字段（TEXT/NUMERIC/INTEGER/REAL/BLOB），它决定了 SQLite 如何在该列的数值上进行类型转换。

一个有趣的细节： SQLite 的 Table 结构也用于表示"视图"（view）和"虚拟表"（virtual table）。当 pSelect != NULL 时，这个 Table 实际上是一个视图；当 pIndex 指向的是一个特殊的虚拟表索引 时，说明这是一个虚拟表。相同的结构体在不同的上下文中有不同的含义——这是 C 语言中"联合体 + 类型标记"设计模式的典型应用。

Select 与 Expr — 查询的灵魂

如果说 Table 和 Index 代表了"数据长什么样"，那么 Select 和 Expr 就代表了"你想怎么查数据"。它们是 SQL 查询在内存中的"活"的表现形式。

Select 结构体包含了 SQL 查询的各个子句：

struct Select {
    ExprList *pEList;      /* SELECT 的列列表 */
    u8 op;                 /* 操作类型: TK_UNION/TK_INTERSECT/TK_EXCEPT */
    int nSelectRow;        /* 估计行数（对数尺度） */
    u32 selFlags;          /* SF_* 标志 */
    int iLimit, iOffset;   /* LIMIT 和 OFFSET 的寄存器号 */
    Select *pNext;         /* UNION 链中的下一个 SELECT */
    Select *pPrior;        /* 另一个链方向 */
    SrcList *pSrc;         /* FROM 子句（表源） */
    Expr *pWhere;          /* WHERE 子句表达式 */
    ExprList *pGroupBy;    /* GROUP BY 子句 */
    Expr *pHaving;         /* HAVING 子句 */
    ExprList *pOrderBy;    /* ORDER BY 子句 */
    Select *pOrderBy;      /* LIMIT/OFFSET */
    ...
};

而 Expr 结构体是"表达式树"的节点：

struct Expr {
    u8 op;                 /* 操作符类型: TK_* 常量之一 */
    u8 affinity;           /* 类型亲和性 */
    u8 op2;                /* 辅助操作符类型 */
    Expr *pLeft;           /* 左子树 */
    Expr *pRight;          /* 右子树 */
    ExprList *x.pList;     /* 参数列表（函数调用时） */
    int iTable;            /* 表游标号（引用列时） */
    int iColumn;           /* 列号（引用列时） */
    Token span;            /* 原始 SQL 文本范围 */
    char *zToken;          /* token 字符串（标识符/字面量） */
    union {
        int iValue;        /* 整数字面量 */
        double rValue;     /* 浮点数字面量 */
        FuncDef *pFunc;    /* 函数定义指针 */
        ...
    } u;
    ...
};

Expr 的 op 字段可以是超过 100 种不同的值——从基本的 TK_EQ（等号）、TK_GT（大于）、TK_PLUS（加法），到复杂的 TK_IN（IN 子查询）、TK_FUNCTION（函数调用）、TK_CASE（CASE WHEN）。这一百多种操作符构成了 SQL 语义的全部基础。

辅助结构体如 ExprList（有序表达式列表，用于 SELECT 列、ORDER BY、GROUP BY）、SrcList（表源列表，用于 FROM 子句中的表/子查询）、IdList（标识符列表），都是围绕 Expr 和 Select 的"配角"。

💡 如何快速上手： 如果你想验证对 Expr 树的理解，可以在调试版本中调用 sqlite3ShowExpr() 和 sqlite3ShowSelect() 函数——它们会将 Expr 树和 Select 结构以可读的文本形式打印出来。在 gdb 中，你可以这样用：call sqlite3ShowExpr(myExprPtr)。这是调试 SQL 编译逻辑的利器。

Vdbe 与 Op — 执行态的表示

SQL 被编译成 VDBE 字节码后，就变成了 Vdbe 和 Op 的天下。Vdbe 代表"一个正在/等待执行的数据库程序"，Op 是程序中的单条指令。

struct Vdbe {
    sqlite3 *db;              /* 数据库连接 */
    Op *aOp;                  /* 指令数组 */
    Mem *aMem;                /* 内存单元数组（寄存器） */
    int nMem;                 /* 内存单元数量 */
    VdbeCursor **apCsr;       /* 游标数组 */
    int nCursor;              /* 游标数量 */
    int pc;                   /* 程序计数器（当前执行到的指令下标） */
    int rc;                   /* 最近操作返回码 */
    u8 errorAction;           /* 错误恢复动作 */
    u8 minWriteFileFormat;    /* 最小兼容文件格式 */
    bft expired;              /* 是否已过期 */
    int nOp;                  /* 指令数量 */
    int nOpAlloc;             /* aOp 数组已分配大小 */
    ...
};

struct Op {
    u8 opcode;                /* 指令操作码 */
    u8 p5;                    /* 第5操作数（位标志） */
    int p1;                   /* 操作数1 */
    int p2;                   /* 操作数2 */
    char *p3;                 /* 操作数3（字符串指针） */
    int p4type;               /* p4 的类型 */
    union {                   /* 操作数4（多种可能类型） */
        int iValue;           /*   整数值 */
        char *zValue;         /*   字符串 */
        FuncDef *pFunc;       /*   函数指针 */
        ...
    } p4;
    int cnt;                  /* 执行次数统计 */
    char *zComment;           /* EXPLAIN 注释（调试用） */
};

VDBE 是一个典型的"寄存器式虚拟机"——aMem[] 是它的寄存器文件，每个 Mem 单元可以存储一个 SQL 值（整数、浮点、字符串、NULL 或 BLOB）。opcode 操作这些寄存器，完成数据加载、计算、比较和存储。

Op 的 p1 到 p5 五个操作数提供了灵活的参数化能力。不同的 opcode 对操作数有不同的解释：

Opcodep1 含义p2 含义p3/p4 含义说明
OP_OpenRead游标号根页号表名(p4)打开表/索引游标
OP_Column游标号列号—读取某行的某列
OP_ResultRow起始寄存器列数—返回一行结果给用户
OP_Next游标号跳转地址—前进到下一行
OP_Ge右操作数寄存器跳转地址—大于等于跳转
OP_Function参数数结果寄存器函数指针(p4)调用内置函数
OP_Transaction写事务标志——开始事务

Vdbe 的 pc（程序计数器）指向 aOp[] 中当前正在执行的指令。在 sqlite3_step() 的每次调用中，VDBE 从 pc 开始执行指令，直到遇到 OP_Halt（语句结束）或 OP_Yield（暂停并让出控制权）。每次调用只执行一次循环，所以 sqlite3_step() 可以逐行返回数据。

⚠️ 一个真实踩坑故事： 几年前我在做一个嵌入式项目时，遇到了一个诡异的 bug——一条 SELECT 查询在某些设备上返回了错误的结果。排查了三天后，发现原因是设备的内存非常有限，SQLITE_MAX_VDBE_OP 被我们的编译脚本设成了 10000。我们要查询的表有 JOIN + GROUP BY + ORDER BY，生成的 VDBE 程序超过了 10000 条指令，结果超出的指令被静默截断了！从此我养成了一个习惯：在项目中用 EXPLAIN your_query 查看生成的指令数量，并确保编译时的限制足够大。

好了，第 4 章我们解剖了 SQLite 源码中最核心的五个数据结构。从数据库连接 sqlite3 到编译上下文 Parse，从表模式 Table、查询表示 Select/Expr 到执行态 Vdbe/Op——你已经掌握了 SQLite 源码中"谁是谁、谁管谁"的基本框架。后续每一章都会基于这些结构体展开。
