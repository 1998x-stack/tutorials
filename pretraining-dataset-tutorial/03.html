开篇：中文大模型的"隐形天花板"

大家好，我是你们的讲师。上一章我们兴致勃勃地逛了全球数据集超市，但你可能已经注意到一个令人不安的事实——货架上的"优质中文语料"寥寥无几。

这不是你的错觉。截至 2026 年，中文高质量公开预训练语料的稀缺性，已经成了制约中文大模型发展的核心瓶颈。我可以举一个具体的数字来说明问题的严重性：英文高质量公开数据集的总量（Dolma、RedPajama、The Pile 等合计）已经超过 5T token，而中文高质量公开语料加起来可能还不到 500B token。差了整整一个数量级。

为什么会这样？是中文互联网的数据不够多吗？恰恰相反——中文互联网的数据量极为庞大，但问题是：这些数据要么被锁在各大平台的围墙花园里，要么质量参差不齐到无法直接使用，要么面临复杂的版权和合规问题。

本章我们就来逐一排查中文预训练语料的"家底"，看看我们有哪些武器可用，又面临哪些绕不开的坎。

3.1 SkyPile-150B：开源中文商业语料的破冰者

2023 年 11 月，昆仑万维发布了 SkyPile-150B，这是迄今为止最大规模的开源中文商业预训练语料库。它的出现，在一定程度上缓解了中文开源数据的燃眉之急。

SkyPile 的几个关键数字：

指标 数据
总 token 数 ~150B tokens（约 600GiB 原始文本）
数据来源 中文互联网网页（百度百科、知乎、豆瓣等），2010-2023 年
清洗流程 质量过滤 + 去重 + 安全过滤（共计 10 道处理步骤）
许可证 MIT 许可证，可自由用于商业和非商业用途
发布方 昆仑万维（Kunlun Tech）

SkyPile 的数据主要来自中文互联网的公开网页数据，经过了一套系统的清洗流程。它的开源许可证选择（MIT）非常友好——意味着无论是学术界、创业公司还是大型企业，都可以免费使用而不需要担心商业限制。这在开源中文语料中实属罕见。

对中文开发者的建议：SkyPile-150B 是目前训练中文大模型最稳妥的起步数据源。如果你在 2026 年启动一个中文预训练项目，请优先将 SkyPile 纳入数据组合。它的 150B token 虽然不算巨大，但胜在质量有保障、许可清晰。

一个标志性的事实足以说明 SkyPile 的价值：在它发布之前，中文开源语料的世界是零散的、碎片化的。每个团队都要从零开始爬取和清洗数据，浪费了大量重复劳动。SkyPile 第一次让中文预训练有了一个"标准基线"——就像 C4 对英文世界的意义。

3.2 WuDaoCorpora 2.0：学术灯塔下的遗憾

WuDaoCorpora 2.0 由北京智源人工智能研究院（BAAI）发布，是中文预训练数据领域的另一座里程碑。WuDaoCorpora 2.0 的规模达到 3TB 原始文本，涵盖新闻、百科、论坛、博客等多个领域。

WuDaoCorpora 的优势在于：

  • 规模宏大：3TB 原始文本，换算成 token 约 500B-800B（取决于编码方式）
  • 来源多样：从权威新闻媒体到垂直社区，覆盖了中文互联网的主要内容形态
  • 学术影响力强：智源发布的悟道系列模型均以此为基础，学术论文引用量很高

但令人遗憾的地方也很明显——许可证限制。WuDaoCorpora 采用的是非商业许可（仅限学术研究使用）。这意味着如果你在为商业产品训练模型，WuDaoCorpora 就不能作为训练数据。它的存在意义更多体现在：

  • 学术研究的标杆：发表论文时可以用 WuDaoCorpora 训练模型作为基线
  • 数据清洗流程的参考：智源公开了部分清洗规则，可以学习和借鉴
  • 非商业项目的选择：如果你的项目是非商业性的学术研究，WuDaoCorpora 是一个好选择
许可证红线：我见过不止一个团队在训练商业模型时"不小心"混入了 WuDaoCorpora 的数据，后来被发现后不得不重新训练。这是一个非常昂贵的教训——在天价的训练成本面前，事先做一遍完整的许可证审计比事后补救要划算得多。永远不要假设"大家都在用所以没问题"。

3.3 CLUECorpus2020：轻量级但干净

CLUECorpus2020 由清华大学和阿里巴巴等机构联合发布，是一个经过精细清洗的中文语料库。它的规模不大——约 100GiB（约 35B tokens）——但在中文 NLP 社区中占据着特殊的地位。

为什么它的影响力这么大？

CLUECorpus 关键数据：
  ┌──────────────┬────────────┬────────────┐
  │ 数据源       │ 原始大小   │ 清洗后大小 │
  ├──────────────┼────────────┼────────────┤
  │ 中文新闻     │  80 GiB    │  40 GiB    │
  │ 中文百科     │  50 GiB    │  30 GiB    │
  │ 中文论坛     │  60 GiB    │  20 GiB    │
  │ 中文博客     │  30 GiB    │  10 GiB    │
  ├──────────────┼────────────┼────────────┤
  │ 总计         │ 220 GiB    │ 100 GiB    │
  └──────────────┴────────────┴────────────┘

CLUECorpus 的清洗策略非常激进——宁缺毋滥。它去掉了超过一半的原始文本，留下来的都是经过多层过滤的高质量内容。这使 CLUECorpus 成为中文 NLP 领域最广泛使用的 baseline 数据集之一。很多在 CLUECorpus 上发表的论文结果，至今仍是重要的参考基准。

经验之谈：CLUECorpus 给了我们一个重要的启示：数据的"干净度"比"量大"重要得多。如果你的数据分析团队只有 2-3 个人，与其试图搭建一个能处理上百 TB 原始数据的庞杂流水线，不如先从 CLUE 这样的高信噪比数据集开始，把训练流程跑通，再逐步扩展。

3.4 中文 Wikipedia：小而精的知识宝库

中文 Wikipedia 在预训练数据中扮演着特殊的角色——它提供了高质量、结构化、经过人类审核的知识文本。虽然它的体量不大（约 1.5 GiB，不到 500M token），但知识密度极高。

在预训练中，Wikipedia 的核心价值在于它的锚定作用（anchoring effect）。简单来说：

  • Wikipedia 文本提供了"教科书级别"的表达方式——用词精准、逻辑清晰、事实正确
  • 加入 Wikipedia 数据可以帮助模型建立正确的语言先验
  • 即使在预训练中的占比只有 3%-5%，Wikipedia 数据对 QA 和事实验证能力的提升非常显著

对于中文 Wikipedia，有一个特别推荐的数据版本：olm/zh-wikipedia，这是由 OLMo 项目团队清洗过的版本。相比原始 Wikipedia 导出文件，olm 版本做了以下几项改进：

  • 移除了信息框、列表、表格等非自然语言的模板化内容
  • 清理了引用标记（如 [1]、[citation needed] 等）
  • 过滤了短页面（少于 50 个字符的页面被剔除）
  • 拆分了大页面（按节拆分，避免超长序列影响训练效率）

一个小技巧：在构建多语言 Wikipedia 数据集时，不要只取中文。英文 Wikipedia 的规模是中文的 20 倍以上，其中包含大量中文 Wikipedia 没有收录的条目。即使训练的是中文模型，适量混入英文 Wikipedia 也能显著提升模型在科技、医学等领域的知识覆盖。

3.5 中文书籍与代码语料

书籍语料是预训练中另一类高质量数据源。在中文领域，可用的开源书籍语料主要有以下几种：

数据集 规模 来源 许可证 特点
BookCorpus (中文版) ~2 GiB 爬虫收集的各类中文书籍 非商业（部分子集） 覆盖小说、散文、科普等
Gutenberg 中文作品 ~200 MiB Project Gutenberg 公版书 公共领域 ✅ 古典文学作品为主
mC4 中文书籍部分 ~10 GiB Common Crawl 检测到的书籍页面 ODC-BY 质量参差不齐
自定义爬取网文 无统一标准 各大网文平台 ⚠️ 版权风险极高 量大但法律风险最大

在代码数据方面，The Stack 包含了一个较大的中文子集。The Stack 是一个多语言代码数据集，其中与中文相关的内容主要来自：

  • 中文 GitHub 项目：README 文件中的中文文档、中文注释、中文 Issue 讨论
  • 中文技术博客：被 GitHub Pages 托管的个人技术博客
  • 中文文档字符串：开源项目中的中文文档字符串和 API 说明

虽然 The Stack 不是传统意义上的"自然语言语料"，但其中包含的中文技术文档对提升模型的中文代码理解和编程能力非常有价值。

一条实战原则：构建中文预训练数据集时，不要太执着于"纯中文"数据。一个 70% 英文 + 30% 中文的混合数据集，在很多场景下比 100% 中文的数据更有效。原因很简单：当前最高质量的数据清洗工具、最丰富的训练数据都是英文的，完全抛弃英文数据就等于主动放弃了大部分好数据。模型在英文数据上学到的推理能力和语言知识会迁移到中文上。

3.6 中文数据的核心痛点

写完以上所有内容，我必须坦诚地跟大家说：前面的每个数据集都有各自的短板，中文预训练语料面临着几个系统性的痛点。

痛点一：开源高质量中文数据极度稀缺

英文社区有 RedPajama、Dolma、The Pile、RefinedWeb 等一批高质量、大容量、商业友好的数据集。中文社区与之对标的只有 SkyPile-150B（150B tokens）。市场规模差距在 5-10 倍。这不是偶然的——英文 NLP 的研究和工程投入是中文的几十倍，高质量数据的发布和更新自然也更频繁。

痛点二：数据质量良莠不齐

中文互联网的生成内容质量方差极大。百度百科的内容质量相对稳定，但知乎、豆瓣等社区的内容质量高度依赖个体创作者。更麻烦的是，中文互联网充斥着 SEO 堆积文章、AI 伪造的内容农场，以及大量格式混乱的扫描版 PDF 转文字文本。这些数据混入训练集后，会让模型学会"不知所云的废话生成"能力。

痛点三：版权环境复杂

中文内容的版权保护力度和法律实践与欧美有显著差异。一方面，互联网平台（微信公众号、知乎、小红书等）的大量内容受版权保护但未被明确标注；另一方面，一些所谓的"开源中文数据集"可能在数据爬取时就存在版权瑕疵。这使得中文预训练数据的合规审查比英文场景更加困难。

痛点四：缺乏组织层面的基准

英文社区有强大的组织力——不管是 EleutherAI 发布的 The Pile，AI2 发布的 Dolma，还是 Together Computer 发起的 RedPajama，背后都有一个明确的研究机构或企业联盟在推动。中文社区目前缺乏这样的组织协作。SkyPile 是单一企业行为，WuDaoCorpora 背后有智源研究院但使用了受限许可。没有一个类似"中文 Dolma"或"中文 RedPajama"的联合项目来系统性地解决中文预训练数据问题。

踩坑故事：我曾经参与过一个中文模型的训练项目。团队从 GitHub 上搜集了多个"中文开源数据集"，加起来号称有 500B token。开训后前期的 loss 下降看起来很漂亮。但训练到 60% 时，我们发现了一个可怕的事实——这些所谓的"不同数据集"之间有超过 40% 的文档重叠。某个中文新闻数据集里的 70% 内容，在另一个"中文网页数据集"中以不同的 URL 形式完整出现。更糟的是，还有约 5% 的内容是机翻的英文 Wikipedia。由于没有进行跨数据源去重，模型的训练效率大打折扣，被迫回炉重造。这就是为什么不同数据源之间的交叉去重如此重要——我们会在第 10-12 章详细讲解去重技术。

3.7 我们能做什么？

面对上述困境，作为数据工程师我们可以从以下几个方向着手：

  • 策略一：善用已有的中文开源数据。SkyPile + CLUECorpus + 中文 Wikipedia 的组合，虽然不算大但至少能撑起一个不错的小型模型（1B-3B 参数）。
  • 策略二：从 Common Crawl 中提取中文子集。mC4 的中文部分是目前最大规模的中文网页数据集，虽然质量较低但经过清洗后可以用。我们有专门的章节讨论 CC 处理流程（第 5 章和第 23 章）。
  • 策略三：引入高质量英文数据做补充。不需要纠结于"纯中文模型"的概念——70% 英文 + 30% 中文的混合策略在实践中被广泛验证有效。
  • 策略四：自建领域语料。如果你特别关注某个垂直领域（医疗、金融、法律），可以考虑自建或购买该领域的专业中文数据。
核心观点：中文预训练数据的稀缺是一个结构性难题，短期内很难有根本性的改变。但这也意味着，谁能在中文数据工程上做出突破，谁就拥有了中文大模型竞争中最核心的壁垒。不是靠堆 GPU，而是靠对中文数据的深刻理解和系统化的数据工程能力。

本章小结

本章我们系统梳理了中文预训练语料的现状。SkyPile-150B 是目前最实用的商业友好中文数据集，CLUECorpus 和 Wikipedia 提供了高信噪比的补充，而 WuDaoCorpora 虽然规模大但受限于许可证。同时我们也看到了中文数据的四大痛点——稀缺性、质量方差、版权争议和缺乏组织协作。好消息是，这些痛点恰恰是数据工程的价值所在。

在下一章中，我们将把目光投向垂直领域与增强语料——代码、数学、学术论文、指令数据等专门场景。你会发现，在某些细分领域，中文并不缺乏优质数据，关键在于怎么找、怎么用。我是你们的讲师，我们第 4 章见。
