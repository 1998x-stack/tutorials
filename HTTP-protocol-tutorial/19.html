第十九章：代理与网关

各位同学好。从这一章开始，我们进入HTTP协议架构层面的话题。前面十几章我们聊的都是一个客户端和一个服务器之间的通信细节，但在真实世界里，客户端和服务器之间很少是直连的——中间会经过很多"中间人"。

这些中间人有不同的名字：代理（Proxy）、网关（Gateway）、反向代理、正向代理、透明代理、CDN节点……名字很多，但本质上它们做的是同一件事——接收一个HTTP请求，转发给另一个服务器，再把响应返回给客户端。

区别在于：谁来配置它？用它来做什么？客户端知不知道它的存在？

19.1 正向代理

正向代理（Forward Proxy）是离客户端最近的代理。客户端主动配置代理地址，把自己所有的HTTP请求都发给代理，由代理代为发送出去。

正向代理的典型特征就是：客户端知道代理的存在，并且主动使用它。你可以在系统设置里配，也可以在浏览器插件里配，甚至可以在代码里配 HTTP_PROXY 环境变量。

正向代理的几个核心用途：

  • 突破访问限制：某些地区的网络限制某个网站，你通过一个不在限制范围内的代理服务器来访问。这是最早的VPN形态——其实不是真正的VPN，就是HTTP/HTTPS代理。
  • 匿名化：代理服务器转发请求时，服务器看到的是代理服务器IP，而不是你的真实IP。但注意——X-Forwarded-For 头可能会泄露你的真实IP，后面我们会详细说。
  • 缓存加速：公司内网的代理可以缓存常用资源（比如npm包、Docker镜像、操作系统更新包），多个用户请求同一个资源时直接从缓存返回。
  • 内容过滤：公司或学校通过代理过滤特定类别的网站。代理查看请求URL，如果属于黑名单类别就直接阻断。

正向代理在HTTP层面是怎么工作的呢？看一个例子：

// 客户端发送给代理的请求
GET http://www.example.com/index.html HTTP/1.1
Host: www.example.com
Proxy-Authorization: Basic base64-credentials
Connection: Keep-Alive

注意一个细节：正向代理的请求行里用的是完整URL（http://www.example.com/index.html），而不是普通的相对路径（/index.html）。这是因为代理需要知道要连接哪个目标服务器。这也是HTTP协议中唯一出现完整URL的地方。

实战提示：很多HTTP客户端库（包括curl、Python requests、Node.js的fetch）都支持通过环境变量 HTTP_PROXY 和 HTTPS_PROXY 来配置代理。但注意HTTPS请求走正向代理时，客户端会先通过HTTP CONNECT方法建立一个隧道，代理只负责透传加密数据，无法查看HTTPS内容——这就是HTTPS的"端到端加密"特性。

19.2 反向代理

反向代理（Reverse Proxy）是离服务器最近的代理。客户端并不知道它的存在——客户端以为自己直接连的是目标服务器，实际上请求先到了反向代理，由反向代理转发给后端的真实服务器。

这和正向代理最大的区别是：正向代理是客户端配置的，反向代理是服务端部署的。

Nginx是最常见的反向代理软件。看看一个典型的Nginx反向代理配置：

upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}

server {
    listen 80;
    server_name www.example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

反向代理的核心价值：

  • 负载均衡：把请求分散到多台后端服务器，提升整体处理能力。
  • SSL终止（SSL Termination）：HTTPS证书和加解密工作由反向代理负责，后端服务器用HTTP通信即可，降低后端复杂度。
  • 缓存加速：对不常变化的响应（如静态资源）在反向代理层做缓存，减轻后端压力。
  • 路由与聚合：不同的URL路径转发给不同的后端服务，实现微服务架构的统一入口。
  • 安全防护：隐藏后端服务器真实IP，过滤恶意请求，限流。

关于网关（Gateway）这个概念——在很多语境下网关就是反向代理，但更严格的区分是：网关通常做了协议转换。比如HTTP网关可以把HTTP请求转成RPC协议、或者转成消息队列。反向代理则是对同一种协议（HTTP到HTTP）做转发。

核心理解：反向代理是微服务架构的标配。不管你用的是Nginx、HAProxy、Traefik还是Envoy，本质上都是在做同一件事——用统一的入口收束所有流量，做一层控制。我见过不少初创团队起步时只有一台服务器，不需要反向代理；但一旦扩展到两台服务器以上，就一定要在前面架一个反向代理。

19.3 透明代理

透明代理（Transparent Proxy）也叫"拦截代理"。它的特点是：客户端不需要做任何配置，甚至完全不知道它的存在。

透明代理的工作原理：在网络层（通常是路由器或交换机级别）对HTTP流量做拦截，不修改TCP/IP连接参数，但把HTTP请求重定向到代理服务器处理。

常见的透明代理使用场景：

  • 公司/学校网络管控：在网关路由器上对HTTP流量做透明拦截，不用每台电脑配置代理就能实现内容过滤。
  • CDN边缘节点：用户在浏览器里输入 www.example.com，DNS解析返回了CDN边缘节点的IP。用户直接连接这个IP，CDN节点扮演了反向代理的角色——对用户来说是透明的。
  • 运营商缓存：某些ISP会对常见资源（比如视频、系统更新包）做透明缓存，为的是节省跨网带宽。

踩坑故事：有一次我排查一个诡异的问题——用户反馈说访问我们网站时，看到的CSS样式"不太对"，但刷新一下又恢复正常了。查了半天发现，问题出在运营商的透明代理上。运营商会缓存静态资源来省钱，但缓存策略很粗暴——不管资源的新鲜度，直接返回旧版本。最后我们在所有静态资源URL里加了版本号参数（如 style.css?v=20240301），问题才解决。这也告诉我们：透明代理虽然是透明的，但它的副作用一点也不透明。

19.4 X-Forwarded-For链

当一个请求经过多层代理时，后端服务器想知道客户端的真实IP——不能直接用TCP连接的IP，因为那是最后一层代理的IP。

解决方案就是X-Forwarded-For这个请求头。每一层代理在处理请求时，都会把上一层的客户端IP追加到这个头里。

假设链路是：客户端(1.1.1.1) → 代理A(2.2.2.2) → 代理B(3.3.3.3) → 后端服务器

// 客户端发到代理A（无X-Forwarded-For）
GET /api HTTP/1.1
Host: example.com

// 代理A发到代理B（追加客户端IP）
GET /api HTTP/1.1
Host: example.com
X-Forwarded-For: 1.1.1.1

// 代理B发到后端服务器（再追加代理A的IP）
GET /api HTTP/1.1
Host: example.com
X-Forwarded-For: 1.1.1.1, 2.2.2.2

后端从X-Forwarded-For头里取最左边的IP，就是客户端的真实IP。

但这里有个信任问题：如果客户端自己伪造了X-Forwarded-For头呢？

// 客户端伪造X-Forwarded-For
GET /api HTTP/1.1
Host: example.com
X-Forwarded-For: 8.8.8.8   // ← 客户端自己加的

// 代理A追加的
X-Forwarded-For: 8.8.8.8, 1.1.1.1

所以只有你能信任的第一跳代理（即直接与你通信的代理）追加的IP才是可靠的。实际做法是：后端只信任X-Forwarded-For中最后一个被可信代理追加的IP地址。

在Nginx中，可以通过 real_ip_header 和 set_real_ip_from 来精确控制：

# 只信任来自内网代理的 X-Forwarded-For
set_real_ip_from 10.0.0.0/8;
real_ip_header X-Forwarded-For;
real_ip_recursive on;

除了X-Forwarded-For，还有几个相关的标准头：

  • X-Real-IP：Nginx常用，只包含一个IP（上一跳地址），不保存链。
  • X-Forwarded-Proto：指示原始请求的协议（http或https），对于判断是否走HTTPS很关键。
  • X-Forwarded-Host：指示原始请求的Host头。
  • Forwarded（RFC 7239）：标准化版本，把上述信息合并到一个头里，但实际使用不多。

安全提醒：如果你用 X-Forwarded-For 来做IP白名单、频率限制、地理定位等安全决策，一定要确保只信任可信代理传来的值。如果直接取最左边的IP，攻击者伪造一个内网IP的 X-Forwarded-For 就能绕过你的安全策略。这种攻击方式叫"IP欺骗"——虽老但依然有效。

19.5 CDN原理

CDN（Content Delivery Network，内容分发网络）是反向代理+缓存+全球分布式网络的集合体。它把一个网站的静态资源（以及日益增多的动态资源）部署到全球几百个甚至上千个节点上。

CDN的核心流程：

  1. DNS调度：用户请求 www.example.com，DNS解析返回离用户最近的CDN节点的IP地址，而不是源站IP。
  2. 边缘缓存：CDN节点收到请求后，检查本地是否有缓存。有缓存且未过期，直接返回。
  3. 回源（Origin Pull）：如果边缘节点没有缓存或缓存过期，节点会向源站服务器发起请求，获取资源后缓存在本地，再返回给用户。
  4. 缓存刷新（Purge）：源站内容更新后，需要通过API或控制台通知CDN节点让缓存失效。

CDN的DNS调度策略（如何选择最合适的边缘节点）：

策略原理优点缺点
地理就近根据请求IP的地理位置选择最近节点实现简单不考虑网络延迟和负载
延迟探测客户端实际测速后选择延迟最低的节点更准确额外探测开销
负载均衡考虑各节点的当前负载情况避免节点过载需要实时监控
Anycast多个节点共享同一IP，BGP路由自动选择故障转移快对BGP依赖大

CDN缓存的有效性取决于缓存控制策略：

  • 源站通过 Cache-Control: max-age=3600 告诉CDN缓存1小时。
  • 通过 Cache-Control: s-maxage=86400 可以给CDN设置与浏览器不同的缓存时间。
  • 通过设置 ETag 或 Last-Modified，CDN可以发送条件请求回源验证资源是否更新。

重要提示：很多人把CDN简单理解为"静态资源加速器"，但实际上现代CDN能做更多——边缘计算（在CDN节点上跑自定义代码）、动态加速（优化动态API请求的路由）、DDoS防护、WAF（Web应用防火墙）。在选择CDN服务商时，不要只看节点数量，还要看它对HTTP/2、HTTP/3的支持、缓存刷新速度、以及有没有你需要的额外功能。
