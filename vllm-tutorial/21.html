第21章 推测解码

自回归生成有个根深蒂固的限制——每一步只能生成一个 token。70B 模型每步 ~10ms，生成 1000 tokens 要 10 秒。推测解码（Speculative Decoding）用一个巧妙的"投机取巧"打破了这个限制。

21.1 核心思想：小模型猜，大模型验证

传统自回归解码 (Target Model, 70B):
Step 1: 算 T1 → 10ms
Step 2: 算 T2 → 10ms
...
Step 1000: 算 T1000 → 10ms
总时间: 10,000ms

推测解码 (Draft 0.5B + Target 70B):
Draft Model 快速生成 5 个候选: [T1', T2', T3', T4', T5'] → 5×0.5ms = 2.5ms
Target Model 一次验证全部 5 个:                      → 10ms
  T1'=T1 ✓ → T2'=T2 ✓ → T3'=T3 ✓ → T4'≠T4 ✗
  接受前 3 个，拒绝 T4', 从 T4 开始下一轮

实际: 12.5ms 生成 3 个 token → 4.2ms/token
原来: 10ms 生成 1 个 token → 10ms/token
加速比: 2.4×

为什么能加速：大模型一次 forward 的计算量几乎和输入 token 数无关（KV cache 已缓存）。验证 5 个 token 和验证 1 个 token 的计算量几乎一样。所以"一次验证多个"只花一次 forward 的时间。

21.2 vLLM 支持的三种方法

方法Draft 来源需要训练？加速比（典型）
n-gram已生成文本做 n-gram 匹配❌ 不需要1.3-1.8×
EAGLE轻量级预测头（基于 feature 层）✅ 需要训练2.0-3.5×
Medusa多个预测头并行猜 future tokens✅ 需要训练2.0-3.0×

21.3 n-gram 推测解码：开箱即用

# 最简单的推测解码，不需要额外模型
vllm serve meta-llama/Llama-3-70B \
    --speculative-config '{
        "method": "ngram",
        "num_speculative_tokens": 5,
        "ngram_prompt_lookup_max": 4
    }'

# 原理：如果 prompt 或已生成文本中出现过相同 n-gram，
# 则用后续 token 作为 draft。
# 适合: 代码生成（重复模式多）、模板填充

21.4 EAGLE：当前最强方法

# EAGLE 需要一个训练好的 draft model
vllm serve meta-llama/Llama-3-70B \
    --speculative-config '{
        "method": "eagle",
        "draft_model": "/path/to/eagle-llama-3-70b-draft",
        "num_speculative_tokens": 5
    }'

# EAGLE 的优势: 不仅是简单的"小模型猜"，
# 而是用大模型的中间层特征 + 轻量 head 预测，
# draft 质量远高于独立的 draft model

2026 年的 P-EAGLE（Parallel EAGLE）更进一步——不再是线性生成候选序列，而是并行生成多个可能分支，动态评估置信度，选择最可能的分支。在 8192 token 场景中加速高达 6.7×。

21.5 什么时候推测解码没用

推测解码不是免费的午餐：

  • temperature=0 效果最好：greedy 解码时 draft 模型更容易猜对 target 的输出
  • temperature 高时接受率低：random sampling 让 draft 和 target 更容易产生分歧，加速比比 1.1-1.3× 甚至无提升
  • 计算密集场景效果差：如果 prefill 是瓶颈而非 decode，推测解码帮不上忙
  • draft model 加载占用额外显存：虽然 draft model 通常很小（0.5B-1B），但额外的显存消耗需要考虑

实践建议：先试 n-gram——零成本，几分钟就能看出效果。如果 n-gram 加速比 >1.5× 且你需要更高加速，再考虑训练 EAGLE draft model。n-gram 在代码生成和结构化输出场景中表现最好。
