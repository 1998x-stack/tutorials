第5章：链表、压缩列表与 listpack

大家好，我是你们的讲师。上一章我们深入研究了 SDS——Redis 的字符串基石。字符串虽然强大，但它们终究是"原子"——只能存储一个值。当我们需要存储一个元素的列表、一个键的一系列字段时，就需要容器型的数据结构了。

Redis 的容器数据结构走过了一条非常有意思的演进路线。早期版本中，List 类型使用双向链表（adlist）存储，Hash 和 Sorted Set 使用压缩列表（ziplist）做紧凑存储。但随着时间推移，ziplist 的"连锁更新"问题逐渐暴露，antirez 在 Redis 5.0 中引入了一个新的替代方案——listpack。再到后来，quicklist 把链表和压缩列表结合起来，形成了一种混合数据结构。这一章我们要读的 adlist.c、ziplist.c 和 listpack.c，加起来超过 3000 行，但核心思想都非常清晰：用不同的内存布局和访问模式，在不同场景下做最优的选择。

adlist 双向链表

adlist（A generic doubly linked list）是 Redis 中最"标准"的链表实现。它的结构体定义简洁明了：

/* adlist.h — 链表的三个核心结构 */
typedef struct listNode {
    struct listNode *prev;
    struct listNode *next;
    void *value;
} listNode;

typedef struct listIter {
    listNode *next;
    int direction;
} listIter;

typedef struct list {
    listNode *head;
    listNode *tail;
    void *(*dup)(void *ptr);      /* 复制节点值 */
    void (*free)(void *ptr);      /* 释放节点值 */
    int (*match)(void *ptr, void *key); /* 匹配比较 */
    unsigned long len;
} list;

这个设计有几个值得注意的地方：

  • 多态通过函数指针实现：dup、free、match 是三个函数指针，它们让 list 可以存储任意类型的值。创建 list 时你可以传入自定义的 dup/free/match 函数，如果传 NULL 则走默认行为（浅复制、不释放、指针比较）。
  • 无环双向：区别于某些教材中的"循环链表"，Redis 的链表是普通的双向链表（head->prev = NULL, tail->next = NULL）。这简化了边界情况的处理。
  • 独立迭代器：listIter 是单独的结构体，支持从头部或尾部开始迭代（由 direction 字段控制，取值 AL_START_HEAD 或 AL_START_TAIL）。迭代过程中修改链表是安全的。

adlist 的 API 非常直观：listAddNodeHead、listAddNodeTail、listInsertNode、listDelNode、listSearchKey、listIndex 等。这些操作的时间复杂度如预期：头部/尾部插入删除 O(1)，按索引查找 O(N)。

但 adlist 有一个明显的缺陷：每个节点都有两个指针的开销（16 字节在 64 位系统上），而且节点在内存中分散分配，遍历时 cache miss 严重。对于少量元素的小列表来说，这个开销可以接受；但当列表中有成千上万个元素时，内存和 CPU 缓存的效率就很成问题。

ziplist 压缩列表

为了解决 adlist 的内存开销问题，Redis 设计了 ziplist——一个连续内存上的紧凑列表结构。它的核心理念很简单：把所有元素按顺序排列在一块连续内存中，用变长编码来存储每个元素的长度。

ziplist 内存布局

/* ziplist.c 中的文档描述了 ziplist 的布局 */
<zlbytes> <zltail> <zllen> <entry> <entry> ... <entry> <zlend>

/* 每个 entry 的编码格式 */
<prevlen> <encoding> <entry-data>
  • zlbytes（4 字节）：整个 ziplist 占用的字节数。
  • zltail（4 字节）：最后一个 entry 的偏移量（方便从尾部操作）。
  • zllen（2 字节）：entry 的数量。如果超过 65535，需要遍历才能得到真实数量。
  • zlend（1 字节）：结束标志，固定值 0xFF。
  • prevlen：前一个 entry 的长度，用于从后向前遍历。如果前一个 entry 长度 < 254 字节，用 1 字节存储；否则用 5 字节（第一个字节是 0xFE 标记，后 4 字节是真实长度）。
  • encoding：当前 entry 的编码类型和长度。可以是字符串编码（指定长度）或整数编码（int4/int8/int16/int24/int32/int64）。

ziplist 的最大优势是内存利用率极高——没有额外的指针开销，元素连续存放，对 CPU cache 非常友好。它的读取在大多数情况下也是高效的（通过 zltail 直接定位到尾节点，通过 prevlen 反向遍历）。

连锁更新问题

ziplist 有一个著名的问题——连锁更新（cascading update）。当一个 entry 被插入或删除时，它后面的所有 entry 的 prevlen 字段可能需要调整大小（从 1 字节变成 5 字节，或者反过来），进而引发后续 entry 的连锁调整。

具体来说，假设当前 ziplist 中所有 entry 的长度都在 250-253 字节之间，它们的 prevlen 都是 1 字节。现在我们在头部插入了一个 254 字节的 entry，那么它后面的第一个 entry 的 prevlen 就必须从 1 字节扩展到 5 字节。这一扩展导致第一个 entry 的总长度增加了 4 字节（从 250 变成 254），于是触发第二个 entry 的 prevlen 扩展，依此类推——就像多米诺骨牌一样传播下去。最坏的情况下，每次插入都触发一次 O(N^2) 级别的内存重分配。

实战观察：虽然连锁更新听起来很可怕，但在实际生产中它造成的性能问题非常罕见。原因有三：1) 触发条件苛刻，需要所有 entry 大小都落在 250-254 字节这个极其狭窄的区间内；2) ziplist 通常只用于存储少量元素（hash-max-ziplist-entries 默认 512，list-max-ziplist-size 默认 -2 即 8KB），元素数量有限；3) 每次调整只是 memmove 操作，对现代 CPU 来说并不算慢。但不可否认，这个设计缺陷确实是 ziplist 的"阿喀琉斯之踵"，也是 antirez 在 Redis 5.0 中引入 listpack 的直接原因。

listpack 替代方案

listpack 是 antirez 在 Redis 5.0 中引入的新型紧凑列表结构，它的设计目标非常明确：彻底解决 ziplist 的连锁更新问题。

listpack 和 ziplist 的核心区别在于：listpack 的每个 entry 不再依赖前一个 entry 的长度信息。每个 entry 仅包含自身信息的编码，不包含 prevlen 字段。

/* listpack entry 的通用编码格式 */
<encoding-type> <element-data> <element-tot-len>

每个 entry 在末尾包含了自身总长度（element-tot-len），这使得从后向前遍历时，可以"跳过"当前 entry 来找到前一个 entry 的起始位置。因为每个 entry 的长度信息是自包含的，修改一个 entry 不会影响其他 entry 的编码，连锁更新就此消失。

listpack 在 Redis 7.x 中已经逐渐取代 ziplist，成为 Stream、Hash（当元素较小时）等类型的底层编码。未来版本的 Redis 计划彻底移除 ziplist，全面切换到 listpack。

quicklist 混合结构

quicklist 是 Redis 3.2 引入的，用于替代 adlist + ziplist 作为 List 类型的底层实现。它的设计思路是典型的"分治"——将一个大列表切分成多个 ziplist（或 listpack）片段，每个片段作为一个 quicklistNode，节点之间用双向指针连接。

/* quicklist.h 中的核心结构 */
typedef struct quicklist {
    quicklistNode *head;
    quicklistNode *tail;
    unsigned long count;        /* 所有节点中的元素总数 */
    unsigned long len;          /* quicklistNode 数量 */
    int fill : QL_FILL_BITS;   /* 每个 ziplist 的最大大小控制 */
    unsigned int compress : QL_COMP_BITS; /* 压缩深度 */
    /* ... */
} quicklist;

fill 参数控制每个 ziplist 的大小。正值表示每个 ziplist 最多包含多少个 entry，负值（如 -1 到 -5）表示按 4KB 到 64KB 的字节数限制。生产环境中默认的 list-max-ziplist-size -2 意味着每个 ziplist 最大 8KB。

compress 参数控制压缩深度。由于列表的操作通常集中在两端（LPUSH/LPOP/RPUSH/RPOP），quicklist 可以将中间的节点用 LZF 算法压缩存储，节省内存。默认 compress 为 0（不压缩）。

踩坑实录：有一家公司的 Redis 集群每隔一段时间就会 CPU 飙升到 100%，持续几十秒后恢复正常。排查后发现，业务方用 LRANGE mylist 0 -1 读取一个包含几十万个元素的列表，而这个 List 的 quicklist 中每个 ziplist 被设置得过大（fill 参数被调到了 10000）。单个 ziplist 中的元素太多，导致 lrange 在解压和遍历时消耗了大量 CPU。更糟的是，这个列表的中间节点还开启了 LZF 压缩，lrange 遍历时需要逐一解压。最终解决方案是调小 list-max-ziplist-size，让每个 ziplist 包含更少的元素，分摊解压开销。教训是：quicklist 的 fill 参数不是一个可以随意调大的值——在大列表场景下，"拆细"往往比"合并"更有利于读写性能。
