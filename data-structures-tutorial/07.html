前言：找东西的终极方案

大家好，我是你们的讲师。想象你有一个巨大的图书馆，需要频繁地查找一本书。如果书是按编号排好序的，你可以用二分查找——O(log n)，不错了。但如果图书馆的每一本书都有一个"唯一的位置编号"，你拿着编号直接走到那个书架上拿书——这就是 O(1)。

哈希表（hash table）提供的就是这种魔力。它通过一个哈希函数把 key 直接映射到存储位置，让你能在常数时间内完成查找、插入和删除。它是"空间换时间"这个思想最完美的体现——我们多花一些内存来存储位置映射关系，换来的是快得多的查找速度。

在 Python 中，哈希表无处不在——dict 和 set 的底层就是哈希表。你每天都在用，但你可能不知道它背后发生了什么。

哈希表的核心思想

哈希表的核心逻辑可以用一个公式概括：

存储位置 = hash(key) % 数组长度

当你执行 d["name"] = "小明" 时，Python 内部做的事情是：

  1. 计算 key "name" 的哈希值
  2. 取模得到数组中的索引位置
  3. 把 ("name", "小明") 这个键值对存到那个位置

当你执行 d["name"] 时：

  1. 计算 key "name" 的哈希值
  2. 取模得到数组中的索引位置
  3. 直接取出该位置的值

这就是 O(1) 的秘密——不需要遍历，不需要比较，直接通过计算得到位置。当然，现实没有这么完美——不同的 key 可能计算出相同的位置（哈希冲突），这时候就需要处理冲突了。

哈希函数设计

一个好的哈希函数应该满足以下标准：

  • 确定性：相同的输入必须产生相同的输出
  • 均匀性：不同的输入应该均匀分布到整个输出空间
  • 高效性：计算要快，不能成为性能瓶颈

Python 中的 hash() 函数：

print(hash("hello"))   # 某个整数（每次运行可能不同，因为有随机种子）
print(hash(42))        # 42（小整数哈希为自身）
print(hash((1, 2)))    # 元组可哈希
# print(hash([1, 2]))  # TypeError: unhashable type: 'list'
# 可变类型不可哈希——因为它们的值变了哈希值也要变

这里有一个重要的设计选择：为什么 list 不能做 dict 的 key？因为 list 是可变的。如果你把 list 作为 key 存进去，然后修改了 list 的内容，它的哈希值就变了——你就再也找不到这个 key 了。所以 Python 要求 dict 的 key 必须是"可哈希的"（hashable），即不可变类型。

冲突解决：拉链法 vs 开放寻址法

哈希冲突是不可避免的——不同 key 算到了同一个位置。解决冲突有两种主流策略：

拉链法 (Chaining)

每个数组位置不存一个元素，而是存一个链表（或更高效的结构）。冲突的元素都放在同一个链表中：

# 拉链法的逻辑示意
hash_table = [
    0: [] -> (key1, val1) -> (key2, val2)  # key1 和 key2 冲突了
    1: [] -> (key3, val3)
    2: []  # 空
]
# 查找时先定位到链表头，再遍历链表找 key

开放寻址法 (Open Addressing)

如果位置被占了，就按一定规则找下一个空位：

# 线性探测的逻辑示意
# 插入 key3 时，位置 hash(key3) % size = 1 已经被占了
# 就检查位置 2、3、4... 直到找到空位
# 查找时也按同样的顺序找，直到找到 key 或遇到空位

Python 的 dict 使用的是开放寻址法（具体来说是带有随机探测的二次探测）。为什么 Python 选择开放寻址？一个主要原因是：更好的缓存局部性。开放寻址的元素都在同一个大数组里，CPU 缓存友好。而拉链法的链表节点可能散落在内存各处，缓存命中率低。

两种方法的优劣对比：

特性 拉链法 开放寻址法
实现复杂度 简单 较复杂
内存使用 链表指针有额外开销 连续数组，利用率高
缓存友好性 低（节点分散） 高（连续内存）
删除操作 直接删除链表节点 需用"墓碑"标记，不能真删除
负载因子容忍度 可超过 1.0 通常 < 0.7，否则性能急剧下降

哈希表的经典应用

两数之和 (LeetCode 1)

这是哈希表最经典的入门题——暴力法 O(n²)，哈希表优化到 O(n)：

# 暴力法 O(n²)
def two_sum_brute(nums, target):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] + nums[j] == target:
                return [i, j]
    return []

# 哈希表法 O(n)
def two_sum_hash(nums, target):
    seen = {}  # {num: index}
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    return []

注意两种写法的对比：暴力法 3 行代码，哈希表法 7 行——多写了 4 行，但把 O(n²) 变成了 O(n)。在数据量大的时候，这 4 行代码的价值不可估量。

字母异位词分组 (LeetCode 49)

from collections import defaultdict

def group_anagrams(strs):
    groups = defaultdict(list)
    for s in strs:
        # 用排序后的字符串作为 key
        key = "".join(sorted(s))
        groups[key].append(s)
    return list(groups.values())

# 输入: ["eat", "tea", "tan", "ate", "nat", "bat"]
# 输出: [["eat","tea","ate"], ["tan","nat"], ["bat"]]

这道题展示了哈希表的一种常见用法：把需要归类的元素映射到同一个 key 上。异位词排序后是一样的，所以它们共享同一个 key。

最长连续序列 (LeetCode 128)

def longest_consecutive(nums):
    num_set = set(nums)
    longest = 0

    for num in num_set:
        # 如果 num-1 不在 set 中，说明 num 是连续序列的开头
        if num - 1 not in num_set:
            current_num = num
            current_len = 1
            while current_num + 1 in num_set:
                current_num += 1
                current_len += 1
            longest = max(longest, current_len)

    return longest

# 输入 [100, 4, 200, 1, 3, 2]
# 输出 4（连续序列 [1, 2, 3, 4]）

这道题的巧妙之处在于：通过检查 num - 1 是否在 set 中，确保只有序列的起始元素才会进入内层循环。每个元素最多被访问两次，所以整体是 O(n)。

🔑 核心思维：当你写出 O(n²) 的嵌套循环时，停下来问自己一个问题——"我能用哈希表把这个内层循环优化掉吗？" 哈希表就是用来把"查找"从 O(n) 变成 O(1) 的。看到嵌套循环中内层是在"找东西"，立刻想到用哈希表。

LRU 缓存 (LeetCode 146)

LRU（Least Recently Used，最近最少使用）缓存是哈希表在实际工程中的重要应用。它要求：

  • get(key) 在 O(1) 时间内返回对应的 value
  • put(key, value) 在 O(1) 时间内插入/更新
  • 缓存满时淘汰最久未使用的 key

只用一个哈希表做不到"记录使用顺序"；只用一个链表做不到 O(1) 查找。所以经典的做法是哈希表 + 双向链表：

class DLinkedNode:
    def __init__(self, key=0, val=0):
        self.key = key
        self.val = val
        self.prev = None
        self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.cache = {}   # key → node
        self.capacity = capacity
        # 虚拟头尾节点，简化边界处理
        self.head = DLinkedNode()
        self.tail = DLinkedNode()
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node):
        """从链表中移除节点"""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node):
        """将节点添加到头部（最近使用）"""
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)
        self._add_to_head(node)
        return node.val

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            node = self.cache[key]
            node.val = value
            self._remove(node)
            self._add_to_head(node)
        else:
            if len(self.cache) >= self.capacity:
                # 淘汰尾部节点（最久未使用）
                tail_node = self.tail.prev
                self._remove(tail_node)
                del self.cache[tail_node.key]
            new_node = DLinkedNode(key, value)
            self.cache[key] = new_node
            self._add_to_head(new_node)

这个设计为什么巧妙？因为双向链表能在 O(1) 时间内删除任意节点（因为有 prev 指针），而哈希表能在 O(1) 时间内找到任意节点。两者结合，get 和 put 都是 O(1)。这是组合式思维的典范——两个简单的数据结构组合在一起，解决了一个单独无法解决的问题。

💥 踩坑实录：defaultdict 没 import 和遍历顺序的坑
先说第一个坑：面试中写 from collections import defaultdict 是很多人的肌肉记忆，但有一次我一个学生太紧张，直接在代码里用了 defaultdict(list) 但完全忘记 import。面试官一看代码跑不了，印象分直接归零。这个小问题反映了：面试前熟悉所有常用模块的导入路径，它是基本功的一部分。

第二个坑：dict 的遍历顺序。在 Python 3.7+ 中，dict 会保持插入顺序，但在 3.6 及以下版本中不会。我见过一个 bug：代码在开发环境（Python 3.9）一切正常，部署到老环境（Python 3.5）后，每次运行结果顺序都不一样——因为代码依赖了 dict 的顺序。教训是：永远不要依赖 dict 的遍历顺序来做逻辑判断，除非你明确使用了 OrderedDict 或确认 Python 版本。更安全的做法是明确你的意图——需要顺序就用 OrderedDict 或 list。

本章小结

哈希表是"空间换时间"思想的极致——它用额外的内存换来了近乎 O(1) 的查找速度。从最简单的两数之和，到经典的 LRU 缓存设计，哈希表在算法面试和工程实践中的出场率极高。
