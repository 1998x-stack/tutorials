开篇：2026 年，每个后端都需要"AI 素养"

五年前，后端工程师还可以说"AI 是算法团队的事，我只管 CRUD"。但到 2026 年，这句话已经不成立了。GPT-4o、Claude 4、Gemini 2.5 这些模型的能力已经深度渗透到后端开发的每一个环节——从代码生成到 API 设计，从数据清洗到运维诊断，AI 不再是锦上添花的玩具，而是后端工程师的日常标配工具。

但这里有一个巨大的误区：很多人以为"AI 后端"就是调一下 OpenAI 的 SDK，把用户输入转发给大模型再返回结果。这确实是最简单的方式，但也恰恰是最粗糙的方式。真实的生产级 AI 集成要处理的挑战远不止这些：token 成本控制、模型输出的不确定性、流式响应的网络问题、隐私合规、prompt 注入防护、以及如何在预算有限的情况下为用户提供高质量的回答。

我参与过一个 AI 客服系统的重构。早期版本直接调用了 Claude API，把用户的问题原封不动传给模型。结果上线第一个月 API 账单就突破了五位数——而更糟糕的是，当用户问"你们公司之前是不是出过安全事故"时，模型慷慨地"编造"了一段从未发生过的公关危机。这次踩坑让我们意识到：AI 集成不是"接个 API 就完事"，而是一整套系统工程。从此我们老老实实地补上了知识库检索、敏感词过滤、回答限域和成本监控——而这些，正是本章要讲的内容。

LLM API 集成实战

无论你选用哪种大模型，调用的模式都大同小异。我们以 OpenAI 和 Anthropic 的 API 为例，看看生产级集成的核心要点。

基础调用与流式响应

普通的后端 API 调用通常是"请求→等待→响应"的模式，但 LLM 的推理时间从几百毫秒到几十秒不等。如果不做流式（streaming），用户会在界面看到白屏等待，体验极差。因此生产环境必须使用流式响应——将模型生成的 token 逐块推送给客户端。

// Node.js: 流式调用 Claude API
import Anthropic from '@anthropic-ai/sdk';

const anthropic = new Anthropic({
  apiKey: process.env.ANTHROPIC_API_KEY,
});

// 设置 SSE 响应头
res.setHeader('Content-Type', 'text/event-stream');
res.setHeader('Cache-Control', 'no-cache');
res.setHeader('Connection', 'keep-alive');

const stream = anthropic.messages.stream({
  model: 'claude-sonnet-4-20260514',
  max_tokens: 4096,
  messages: [{ role: 'user', content: prompt }],
}).on('text', (text) => {
  // 逐块推送给前端
  res.write(`data: ${JSON.stringify({ type: 'text', content: text })}\n\n`);
}).on('error', (err) => {
  res.write(`data: ${JSON.stringify({ type: 'error', content: err.message })}\n\n`);
  res.end();
});

req.on('close', () => {
  stream.abort(); // 客户端断开时停止生成，节省 token
});

错误重试与退避策略

LLM API 不像普通数据库查询那样稳定。你可能会遇到：限流（429）、服务过载（503）、上下文超长（400）、以及模型自身的"抽风"——返回空内容或格式错误。一个健壮的集成必须包含重试逻辑。

async function callLLMWithRetry(messages, options = {}) {
  const maxRetries = options.maxRetries ?? 3;
  const baseDelay = options.baseDelay ?? 1000; // 1s

  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      const response = await anthropic.messages.create({
        model: 'claude-sonnet-4-20260514',
        max_tokens: 4096,
        messages,
      });
      return response;
    } catch (err) {
      // 4xx 错误不去重试（除 429 限流外）
      if (err.status >= 400 && err.status < 500 && err.status !== 429) {
        throw err;
      }

      if (attempt === maxRetries) throw err;

      const delay = baseDelay * Math.pow(2, attempt - 1); // 指数退避
      console.warn(`LLM call failed (attempt ${attempt}), retrying in ${delay}ms:`, err.message);
      await new Promise(r => setTimeout(r, delay));
    }
  }
}
实践建议：不要盲目提高重试次数。对于用户交互场景，2-3 次重试足够。更严重的问题应该通过告警通知值班工程师，而不是靠无限重试去"撞大运"。

RAG 架构详解

RAG（Retrieval-Augmented Generation）是目前让 LLM 回答"私有知识"问题的主流方案。它的核心思想是：不把知识塞进模型参数，而是从外部知识库中检索相关片段，拼入 prompt 让模型参考回答。

为什么不用 Fine-tuning？因为 Fine-tuning 成本高、周期长、更新慢——你可以 Fine-tune 一个模型来了解公司 2023 年的产品文档，但要让它知道今早上线的功能，唯一的办法就是 RAG。

一个典型的 RAG 流水线分为四个阶段：

阶段 操作 关键技术 注意事项
1. 文档切分 将原始文档（PDF/HTML/Markdown）切割为语义完整的 Chunk RecursiveCharacterTextSplitter, 语义分割, 重叠窗口 Chunk 太小丢失上下文，太大增加检索噪声。通常 512-1024 tokens
2. 向量化 将每个 Chunk 通过 Embedding 模型转为向量 text-embedding-3-large, 多模态 Embedding 选择适合你语言的 Embedding 模型，中文场景推荐 bge-large-zh
3. 检索 用户提问时，将问题向量化后在向量数据库中做相似度搜索 余弦相似度, HNSW 索引, 混合检索（向量 + 关键词） topK 值需调优，太小答案不全，太大引入噪声
4. 生成 将检索到的 Chunks 拼入 prompt，让 LLM 基于上下文回答 Prompt 模板, 引用标注, 限域指令 告诉模型"不知道就说不知道"，避免幻觉 
# Python: LangChain 实现的简易 RAG
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.chains import RetrievalQA

# 切分文档
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=200,
)
chunks = text_splitter.split_documents(raw_docs)

# 构建向量库
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=OpenAIEmbeddings(model="text-embedding-3-small"),
)

# 创建 RAG 问答链
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o"),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
    return_source_documents=True,
)

# 提问
result = qa_chain.invoke({"query": "我们的数据库连接池配置有什么推荐?答案请引用原文"})
print(result["result"])
关键经验：很多人以为 RAG 的瓶颈在 LLM 的质量，但实际上检索质量才是 RAG 的天花板。如果检索回来的 Chunk 和问题不相关，再强的模型也答不好。投入精力优化 Embedding 模型的选择、Chunk 切分策略和检索的 reranking，回报远超换一个更大参数的 LLM。

向量数据库选型

2026 年，向量数据库已经从"新概念"变成了后端技术栈中的标准组件。但选择仍然很多——主流的选项包括 Pinecone、Milvus、Weaviate 和 pgvector。它们各有取舍，核心的考量维度如下：

维度 Pinecone Milvus Weaviate pgvector
部署方式 全托管 SaaS 自托管 / Zilliz 云 自托管 / 云 PostgreSQL 插件
运维成本 最低，零运维 较高，需运维集群 中等 极低，随 PG 一起管
索引类型 专有（高优化） IVF_FLAT, HNSW, DiskANN HNSW IVFFlat, HNSW
混合搜索 支持（向量 + 元数据过滤） 支持 原生支持（向量 + 关键词） 需结合 PG 全文搜索
可扩展性 自动伸缩 分布式原生，支持百亿级 支持集群 受限于单节点 PG
适用场景 快速原型、中小规模 大规模生产、高 QPS AI-Native 应用、语义搜索 已有 PG 生态、小规模 RAG
成本 按量付费，大规模较贵 自托管控制成本 中等 免费（PG 插件）

我的建议是：如果团队没有专门的 Infra 人员，先上 pgvector。PostgreSQL 16+ 原生支持 HNSW 索引，配合 PG 本身的事务能力和成熟的备份恢复机制，对大多数后端团队的 RAG 场景完全够用。当向量规模突破千万级别且 QPS 要求很高时，再考虑引入专门的向量数据库。

踩坑记录：我们团队早期选了 Milvus，但运维团队完全不熟悉它的集群架构。线上出了几次索引构建失败导致检索返回空结果的问题，排障花了一周。后来切回 pgvector，虽然查询延迟从 5ms 升到了 20ms，但系统的稳定性和可维护性大幅提升。记住：最好的技术不一定是最高性能的，而是你的团队最能驾驭的。

AI 代码审查：用 LLM 做静态分析之外的 Code Review

传统代码审查依赖人工逐行阅读，效率低且覆盖有限。静态分析工具（ESLint、SonarQube）能发现语法问题、潜在 bug 和安全漏洞，但它们无法理解业务逻辑，也无法评判代码的可读性和设计质量。

LLM 的引入正在改变这件事。2026 年，基于大模型的 AI Code Review 已经成为成熟实践——它能理解代码的上下文，给出像资深工程师一样的 review 意见。

AI Code Review 的典型能力

# 一个典型的 AI Code Review Prompt
你是一位资深后端工程师，正在 review 以下 Go 代码变更。

评审维度：
1. 正确性：是否存在并发问题、边界条件遗漏？
2. 性能：是否存在不必要的内存分配、锁竞争、I/O 瓶颈？
3. 可维护性：函数长度是否合理？命名是否清晰？是否有重复逻辑？
4. 安全：是否存在 SQL 注入、XSS、权限缺陷？
5. 测试：变更是否覆盖了关键路径的测试？

代码 diff:
```diff
+ func GetUserOrders(ctx context.Context, userID string) ([]Order, error) {
+     rows, err := db.Query("SELECT * FROM orders WHERE user_id = " + userID)
+     if err != nil {
+         return nil, err
+     }
+     defer rows.Close()
+     ...
+ }
```

请逐条评审，按严重程度分类：Critical / Warning / Suggestion

LLM 很可能会指出：SQL 拼接导致注入风险（Critical）、缺少分页（Warning）、defer 应该在 err 检查之后（Suggestion）。这些发现对新人来说尤其珍贵——相当于每次提交代码都有一位资深工程师在帮你把关。

落地建议：不要用 AI 完全替代人工 Code Review。最佳实践是"AI 做第一轮自动化检查，人工负责第二轮深度评审"。AI 擅长发现"低级错误"和"模式违规"，而架构合理性、产品意图的一致性仍需人来判断。GitHub CodeQL + LLM 插件的组合是目前比较成熟的做法。

Prompt 缓存与成本优化

在生产环境中大规模使用 LLM，花出去的每一分钱都在燃烧。一个中等规模的 AI 客服系统，月均 token 消耗可能轻松超过 1 亿——按 Claude Sonnet 的定价，那就是每个月几千美元。成本优化不是锦上添花，而是让 AI 功能持续运行的生存问题。

Anthropic Prompt Caching

Claude API 支持 Prompt Caching 功能：当多个请求共享相同的 prompt 前缀时（比如系统指令 + 知识库内容），只有新增的 token 需要重新计算，缓存的部分享受 90% 的价格折扣。

// 利用 Prompt Caching 大幅降低固定前缀的成本
const response = await anthropic.messages.create({
  model: 'claude-sonnet-4-20260514',
  max_tokens: 1024,
  system: [
    {
      type: 'text',
      text: LONG_SYSTEM_PROMPT,  // 数千 token 的系统指令
      cache_control: { type: 'ephemeral' }  // 标记为可缓存
    }
  ],
  messages: [
    {
      role: 'user',
      content: [
        {
          type: 'text',
          text: userQuery,
        }
      ]
    }
  ]
});

// 缓存命中的判断：从 response 中读取 usage 统计
console.log(response.usage);
// 输出示例:
// {
//   input_tokens: 15000,      // 总输入
//   cache_creation_input_tokens: 14000,  // 本次创建了缓存
//   cache_read_input_tokens: 0           // 首次无缓存读取
// }
// 第二次调用时 cache_read_input_tokens 会变成 14000

Token 压缩策略

除了利用 API 层的缓存机制，应用层也能大幅优化 token 消耗：

  • 精简系统指令（System Prompt）：不要塞入无关范例。每条多余的 token 都是成本。定期审计 system prompt，去掉不再需要的指令。
  • 检索压缩（RAG Chunk 压缩）：检索到的文档片段先做摘要再输入给 LLM。可以用一个轻量模型对 topK 的结果做压缩，减少传递给最终模型的 token 量。
  • 多轮对话摘要：长对话的历史记录会线性增加 token 消耗。每 N 轮对话后，将历史对话压缩为一句话摘要，替换完整的对话历史。
  • 输出长度限制：设置合理的 max_tokens，不要让模型"自由发挥"。很多场景下 200-300 token 的回答就足够了。
// 对话历史压缩示例
async function compressHistory(messages) {
  if (countTokens(messages) < TOKEN_LIMIT) return messages;

  // 将历史对话浓缩为一句话摘要
  const summary = await anthropic.messages.create({
    model: 'claude-haiku-4-20260514',  // 用便宜的模型做压缩
    max_tokens: 128,
    messages: [{
      role: 'user',
      content: `将以下对话压缩为50字以内的摘要，只保留关键信息:\n${formatMessages(messages.slice(0, -2))}`
    }]
  });

  return [
    { role: 'system', content: `历史对话摘要: ${summary.content[0].text}` },
    messages[messages.length - 1]  // 只保留最新一条消息
  ];
}

一个真实的优化案例：我们团队在 AI 客服系统上通过"Prompt Caching + 历史摘要 + system prompt 裁剪"三管齐下，将月均 API 成本从 $4,200 降到了 $1,100，缩减了 74%。在 AI 领域，省钱和维护质量不是矛盾——优化成本的每一步几乎都在同时优化系统的响应速度和稳定性。
