第12章 可测试性设计(DFT)

大家好，我是你们的讲师。在前面的章节中，我们花了大量精力来"确保芯片设计是正确的"——功能验证确保逻辑没错，STA确保时序没问题。但是，当芯片从晶圆厂回来后，你怎么知道它制造出来的芯片是好的？一颗芯片上有数十亿个晶体管和数百亿个连接——任何一个晶体管失效、任何一条连线短路或断路，都可能导致芯片无法工作。

制造缺陷是不可避免的——即使是台积电最成熟的工艺，良率也达不到100%。在28nm工艺上，缺陷密度大约为0.1-0.2 defects/cm²——这意味着在100mm²的芯片上，平均有10-20个缺陷。但这些缺陷不一定导致芯片失效——它们可能落在"不敏感区域"（比如一根互连线上的多余电阻，如果不在关键路径上可能不影响功能）。你的任务是：在芯片出厂之前，找出所有导致功能失效的缺陷。这就是DFT（Design for Testability）的工作。

12.1 DFT核心概念

DFT的核心是两个概念：可控性（Controllability）和可观性（Observability）。

  • 可控性：从芯片的引脚（I/O pads）把一个内部节点的状态设置为特定值的难易程度。一个芯片内部深处的触发器，可能要通过几十级逻辑才能被外部引脚控制。
  • 可观性：从芯片引脚观测一个内部节点状态的难易程度。同理，要把内部节点的值传播到输出引脚可能需要经过大量逻辑。

一个好的DFT设计，就是要让每一个内部节点的可控性和可观性都尽可能高——理想情况下，让芯片制造商能通过ATE（Automatic Test Equipment）测试每一个逻辑门的好坏。

故障模型（Fault Model）是测试的基础。常见的故障模型包括：

故障模型 描述 物理原因 测试覆盖率目标 测试频率
Stuck-at（固定故障） 节点固定为0或1 短路到VDD/VSS、栅氧击穿 >98% 慢速（低速测试）
Transition（跳变故障） 信号无法在时钟周期内完成跳变 工艺偏差导致门延迟过大 >90% at-speed工作频率
Path Delay（路径延迟故障） 特定路径延迟超过时钟周期 互连线电阻电容过大、晶体管驱动不足 >85% at-speed工作频率
Bridging（桥接故障） 两条互连线短路 金属桥接（metal bridging） >95% 慢速或at-speed
为什么要区分这些故障模型？因为stuck-at测试只能发现"节点卡死在0或1"的缺陷。但芯片在高速运行时还会出现另一种情况——门延迟变大了，导致setup violation。这种"慢门"在慢速测试中可能表现正常（因为时钟周期足够长），只有在工作频率下才会暴露。这就是为什么先进工艺节点必须做at-speed测试（用芯片自己的工作频率测试）的原因。

12.2 扫描链设计

扫描链（Scan Chain）是DFT最核心的技术。它的基本思想很简单：把芯片中的所有触发器连接成一条或多条移位寄存器链。在测试模式下，你可以通过扫描链的输入引脚把一个测试向量（test pattern）"移入"所有触发器，然后在正常工作模式下运行一个时钟周期，再把结果从扫描链的输"移出"——从而检查每一个触发器的输出是否正确。

扫描触发器（Scan Flip-Flop）是在普通触发器的基础上增加了一个多路选择器（MUX），选择"功能输入（D）"或"扫描输入（SI）"中的一个写入触发器。当scan_enable信号为高时，所有触发器被连接成一条链，数据从scan_in逐级移入；当scan_enable为低时，触发器工作在正常功能模式。

扫描链的插入通常由EDA工具自动完成（Synopsys DFT Compiler、Tessent）。设计者需要做的是：

  • 在RTL中预留测试端口（test_se、test_si、test_so）
  • 确保扫描链的 stitching 不会造成时序问题
  • 为扫描链提供独立的时钟和复位

扫描模式分为两种操作阶段：

  • Shift（移位模式）：scan_enable为高，每个时钟周期向链中移入一位新数据，同时移出一位旧数据。TC（test clock）通常跑得较慢（10-50MHz），因为移位模式下功耗很高（所有触发器同时翻转）。
  • Capture（捕获模式）：scan_enable为低，运行一个（或几个）功能时钟周期。DUT在正常工作模式下执行一次运算，结果被捕获到触发器中。然后回到shift模式把结果移出。

12.3 ATPG与故障仿真

有了扫描链，ATE（自动测试设备）还需要知道"要移入什么数据"和"应该移出什么数据"。这就是ATPG（Automatic Test Pattern Generation）的工作。

ATPG工具（如Tessent TestKompress、Synopsys TetraMAX）根据网表和故障模型自动生成测试向量。核心算法包括：

  • D-Algorithm：最早的ATPG算法（1966年），通过在故障点创建D（好电路为1、故障电路为0）或D̅（好电路为0、故障电路为1）的差异，并将这个差异传播到输出端。
  • PODEM（Path-Oriented Decision Making）：比D-Algorithm更高效，通过回溯方式找到可以激活和传播故障的输入组合。
  • 压缩模式（Compression）：现代ATPG支持test compression，通过片上解压/压缩逻辑，用少量ATE通道驱动大量内部扫描链，大幅减少测试时间和测试数据量。

生成测试向量后，需要运行故障仿真（Fault Simulation）来评估这些向量能检测到多少故障。故障仿真的结果是故障覆盖率（Fault Coverage）——可检测故障数占总故障数的百分比。行业标准要求stuck-at故障覆盖率大于98%，transition故障覆盖率大于90%。

12.4 存储器BIST

对于SoC中的大量存储器（SRAM、ROM、Register File），使用扫描链测试是不现实的——存储器的内部节点太多，扫描方式需要的测试时间和数据量都太大。解决方案是MBIST（Memory Built-In Self-Test）：在芯片内部集成一个小型的测试控制器，让它自动对存储器运行测试算法。

MBIST架构包括：

  • BIST Controller：生成地址、数据、控制信号，运行测试算法（如March C-）。
  • Collar（存储器项圈）：包裹在存储器周围的多路选择逻辑，在正常模式和测试模式间切换。
  • Comparator：比较读出数据和期望数据，输出pass/fail信号。

最常用的存储器测试算法是March算法——它按特定顺序对存储器的每一个地址执行读写操作，可以检测出地址译码器故障、单元耦合故障、数据保持故障等。March C-算法只有10步，但能检测到几乎所有常见的存储器故障类型。

如果测试发现存储器有故障，BISR（Built-In Self-Repair）可以用片上备用的行/列替换故障行/列——就像SSD中的坏块管理一样。这使得芯片在存在少量存储器缺陷的情况下仍然可以作为良品出货，显著提升良率。

12.5 边界扫描与压缩

边界扫描（Boundary Scan）基于IEEE 1149.1标准——也就是大家熟知的JTAG。它最初是为了解决PCB板上芯片之间连线的测试问题——当芯片被焊接到PCB上后，你怎么知道引脚之间的焊点是不是好的？

JTAG在芯片的每个I/O引脚旁都加了一个"边界扫描单元"（BSC），这些BSC被串联成一条链，通过4个或5个专用引脚（TDI、TDO、TCK、TMS、可选的TRST）来控制。

JTAG的核心——TAP（Test Access Port）控制器——是一台16状态的有限状态机：

// TAP控制器状态机 (IEEE 1149.1)
// 状态编码（简化表示）
// 从 Test-Logic-Reset 开始:
//   Test-Logic-Reset
//       → Run-Test-Idle (当TMS=0)
//       → Select-DR-Scan (当TMS=1)
//           → Capture-DR
//           → Shift-DR (移入/移出数据)
//           → Exit1-DR → Update-DR
//       → Select-IR-Scan (当TMS=1 from Test-Logic-Reset)
//           → Capture-IR
//           → Shift-IR (移入指令)
//           → Exit1-IR → Update-IR

// 简化状态转移:
// TMS=1 时: 在当前状态选择下一状态
// TMS=0 时: 在当前状态保持或进入工作状态

always @(posedge TCK) begin
    case (current_state)
        TEST_LOGIC_RESET: next_state = TMS ? TEST_LOGIC_RESET : RUN_TEST_IDLE;
        RUN_TEST_IDLE:    next_state = TMS ? SELECT_DR_SCAN : RUN_TEST_IDLE;
        SELECT_DR_SCAN:   next_state = TMS ? SELECT_IR_SCAN : CAPTURE_DR;
        CAPTURE_DR:       next_state = TMS ? EXIT1_DR : SHIFT_DR;
        SHIFT_DR:         next_state = TMS ? EXIT1_DR : SHIFT_DR;
        EXIT1_DR:         next_state = TMS ? UPDATE_DR : PAUSE_DR;
        PAUSE_DR:         next_state = TMS ? EXIT2_DR : PAUSE_DR;
        EXIT2_DR:         next_state = TMS ? UPDATE_DR : SHIFT_DR;
        UPDATE_DR:        next_state = TMS ? SELECT_DR_SCAN : RUN_TEST_IDLE;
        SELECT_IR_SCAN:   next_state = TMS ? TEST_LOGIC_RESET : CAPTURE_IR;
        CAPTURE_IR:       next_state = TMS ? EXIT1_IR : SHIFT_IR;
        SHIFT_IR:         next_state = TMS ? EXIT1_IR : SHIFT_IR;
        EXIT1_IR:         next_state = TMS ? UPDATE_IR : PAUSE_IR;
        PAUSE_IR:         next_state = TMS ? EXIT2_IR : PAUSE_IR;
        EXIT2_IR:         next_state = TMS ? UPDATE_IR : SHIFT_IR;
        UPDATE_IR:        next_state = TMS ? SELECT_DR_SCAN : RUN_TEST_IDLE;
        default:          next_state = TEST_LOGIC_RESET;
    endcase
end

BSDL（Boundary Scan Description Language）文件描述了芯片的边界扫描实现细节——包括BSC单元的位置、指令寄存器长度、支持的指令等。它是ATE和板级测试工具用来与JTAG接口交互的"说明书"。

现代芯片的测试数据量非常大——一颗500M门的芯片可能需要数百MB的测试向量。为了减少测试时间（成本），工业界广泛使用测试压缩（Test Compression）技术。典型方案包括Synopsys的EDT（Embedded Deterministic Test）和Mentor的TestKompress。压缩率通常在10x-100x之间——意味着原本需要1000个ATE cycle的测试，现在只需要10-50个cycle。

LBIST（Logic Built-In Self-Test）是另一种方法：在芯片内部集成PRPG（伪随机向量发生器）和MISR（多输入特征寄存器），让芯片在系统启动时自我测试。不需要ATE，不依赖外部向量。缺点是故障覆盖率不如确定性ATPG高（通常80-90%），但胜在可以"随时测"——比如汽车芯片在每次启动时都运行LBIST来确保安全。
一个关于良率的真实故事：某公司设计了一颗28nm的IoT SoC芯片，在流片后遇到了一个令人困惑的问题——第一批晶圆的良率只有65%，远低于预期（目标85%）。经过一个月的排查，最终定位到问题：芯片内部有一个大容量的SRAM（用来做数据缓冲），这个SRAM的测试覆盖率在MBIST中达到了100%。但在实际测试中发现，SRAM周边的地址译码器和数据多路选择器没有被任何测试覆盖——它们不在扫描链中，MBIST也不测试它们。补上这部分DFT覆盖后，下一批晶圆的良率提升到了83%。这个案例告诉我们：DFT不是"做了就行"，而是"做对了才行"——每一条漏掉的路径都可能对应着真实的良率损失。
