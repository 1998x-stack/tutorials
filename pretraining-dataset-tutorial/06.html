开篇：2026 年，合规不是选择题

大家好，我是你们的讲师。前几章我们一直在谈"怎么找到好数据"和"怎么清理数据"。但作为一名数据工程师，有一件事比数据质量更重要——数据合规。

让我给你讲一个真实的案例。2024 年，某知名 AI 公司发布了他们的新一代开源大模型，在各类 benchmark 上表现优异，社区反响热烈。然而仅仅两周后，一位版权律师发现该模型的训练数据中包含了大量受版权保护的小说作品。随之而来的是集体诉讼、模型被迫下架、数十亿美元的市值蒸发。这个模型的训练数据质量确实很高——高到甚至包含了完整的《冰与火之歌》系列小说。但正是这份"过于高质量"的数据，让整个项目万劫不复。

到 2026 年，数据合规已经不是"锦上添花"的事情。欧盟 AI 法案（EU AI Act）已于 2026 年 8 月全面生效，其中明确规定：训练生成式 AI 模型的开发者必须"记录训练数据的来源、对训练数据进行版权分析、并确保符合 GDPR 要求"。违反规定的处罚是：全球年营收的 7% 或 3,500 万欧元，取较高者。这不是一个可以忽视的数字。

本章我们就来系统性地梳理数据合规的"红线"在哪里，以及如何在数据工程实践中守住这些红线。

6.1 数据集许可证全景：哪些能用？哪些不能用？

我们先来做一个"交通灯"式的分类，把常见数据集按许可证友好程度分成绿、黄、红三类：

等级 许可证 代表数据集 商业使用
🟢 绿灯 Apache 2.0 / MIT / CC BY RedPajama, SlimPajama, Falcon RefinedWeb, SkyPile 完全允许 ✅
🟡 黄灯 ODC-BY / CC BY-SA / ImpACT C4, mC4, Dolma, The Stack (Dedup) 原则上允许，需注意"署名"要求 ⚠️
🔴 红灯 非商业许可 / 有版权争议的子集 WuDaoCorpora（研究许可）, The Pile 的 Bibliotik 子集 严格禁止 ❌

但这只是"第一层"检查——数据集整体许可证只是表面。真正复杂的是数据集内部的子集许可证。以 The Pile 为例，虽然它整体采用 MIT 许可证发布，但其中多个子集（Bibliotik 书籍、ArXiv 论文等）有各自独立的版权主体和许可条款。

数据集许可证检查清单（以 The Pile 为例）：

  ├── The Pile (整体: MIT)
  │   ├── PubMed Central     → CC BY / CC0 ✅
  │   ├── ArXiv              → 各论文单独授权 ⚠️
  │   ├── GitHub             → 每个仓库 LICENSE 决定 ⚠️
  │   ├── FreeLaw            → 美国政府作品，公共领域 ✅
  │   ├── Wikipedia          → CC BY-SA ⚠️
  │   ├── Project Gutenberg  → 公共领域 ✅
  │   ├── Bibliotik          → 非商业使用 ❌ 高风险
  │   ├── StackExchange      → CC BY-SA ⚠️
  │   └── ...
最危险的心态："既然数据是公开在 GitHub/HuggingFace 上的，那应该就是能用的吧？"——这是大错特错的。公开发布 ≠ 允许用于模型训练。部分子集的许可证在不同的司法管辖区可能有不同的解释。作为数据工程师，你的责任不是假设"能用"，而是逐一验证"确实能用"。每一个未经核查的数据源，都是埋在你模型里的定时炸弹。

6.2 EU AI Act 对数据工程的具体影响

2026 年 8 月全面生效的欧盟 AI 法案，对数据工程提出了明确的要求。这不是泛泛的"数据质量指南"，而是具有法律约束力的合规要求：

  • 训练数据来源记录（Article 28(2)）：开发者必须记录训练数据的所有来源，包括数据集的名称、版本、获取时间和获取方式。这不只是"记个笔记"——你需要建立可追溯的数据来源管理体系。
  • 版权合规声明（Article 53(1)(c)）：开发者需对训练数据是否包含受版权保护的文本进行分析，并"实施尊重版权人权利的政策"。这实际上要求你在数据工程流水线中嵌入版权分析模块。
  • GDPR 合规：如果训练数据中包含欧盟居民的个人信息（PII），需要符合 GDPR 的要求。这通常意味着需要做 PII 检测和脱敏处理。
  • 透明度报告：需要公开发布训练数据的摘要、来源和清洗政策的文档。这也是为什么越来越多的开源模型现在都会附带"数据卡"（Data Card）。

对于中国开发者的实际影响：即使你的公司和用户都在中国，只要你的模型

  • 在欧盟境内提供服务，或者
  • 被欧盟的用户使用，或者
  • 训练数据中包含来自欧盟来源的数据

都可能受到 EU AI Act 的管辖。这不是一个"远在天边"的欧洲法规，而是直接影响你数据工程的现实约束。

建议从现在开始：无论你的模型今天是否面向欧盟用户，建议按照 EU AI Act 的标准来建设数据流水线。法规环境只会越来越严，不会越来越松。现在投入资源建立合规的数据工程体系，是在为未来三年省下十倍的时间和金钱。具体来说：养成记录每个数据源来源和许可证的习惯，在清洗管线中加入 PII 检测步骤，定期做数据卡审计。

6.3 PII 去除：保护用户隐私

PII（Personally Identifiable Information，个人身份信息）去除是数据合规中最具技术含量的环节之一。互联网公开数据中不可避免地包含：

  • 电子邮件地址、电话号码、身份证号
  • 家庭住址、IP 地址
  • 社交媒体用户名、个人主页 URL
  • 银行账号、信用卡号
  • 医疗记录、保险信息

如果这些信息进入了训练数据，模型可能会在生成时泄露这些隐私信息——这是多起真实诉讼的起因。PII 去除的主流工具有：

工具 类型 支持语言 特点
NeMo PiiModifier 基于正则 + NLP 多语言 NVIDIA NeMo Curator 内置，支持规则和模型两种模式
Presidio 基于正则 + ML 多语言 微软开源，可扩展的分析器 + 匿名化器架构
Faker 合成替换 英文为主 用于生成替换 PII 的假数据（如替换真实邮件为 user@example.com）
自建规则引擎 正则匹配 与规则设计相关 针对特定语言的 PII 模式定制

PII 去除有两种常见策略：

  • 检测 + 掩码（Masking）：将检测到的 PII 替换为占位符，如将邮箱替换为 [EMAIL]。优点是保持了文本的可读性，缺点是无法完全消除信息泄露风险。
  • 检测 + 移除（Removal）：直接删除包含 PII 的句子或文档。优点是最安全，缺点是可能删除大量正常内容（如新闻文章中引用的某人邮箱）。
NeMo PiiModifier 的 PII 检测示例：

  输入文本:
  "如需咨询请联系 support@example.com 或致电 138-0000-0000。"

  检测输出:
  "如需咨询请联系 [EMAIL] 或致电 [PHONE_NUMBER]。"

  支持的 PII 类型:
  ├── EMAIL          → [EMAIL]
  ├── PHONE_NUMBER   → [PHONE_NUMBER]
  ├── IP_ADDRESS     → [IP_ADDRESS]
  ├── CREDIT_CARD    → [CREDIT_CARD]
  ├── PERSON_NAME    → [PERSON]         (需 NLP 模型支持)
  ├── LOCATION       → [LOCATION]       (需 NLP 模型支持)
  └── ...

6.4 数据去污染：基准测试集的保护

数据合规不仅关乎法律，也关乎学术诚信。基准测试集污染（benchmark contamination）——即训练数据中"不小心"包含了测试集样本——已经成了大模型评测领域最严重的系统性问题之一。

为什么会发生污染？原因很简单：Common Crawl 爬取了包括 GitHub 在内的整个互联网。而 GitHub 上有大量的 benchmark 数据集、评测脚本和参考答案。当一个爬虫把 HumanEval 的 Python 题目和答案都爬下来并混入训练数据后，模型在 HumanEval 上的"高分"就不再是能力的体现，而只是"背答案"的结果。

一个著名的污染案例：2023 年，某热门开源模型声称在 GSM8K 数学推理上达到了接近 GPT-4 的水平。社区仔细检查后发现，该模型的训练数据中混入了大量 GSM8K 的训练集和测试集题目——它不是学会了推理，而是学会了 "当用户问到类似 xxx 的问题时，输出 yyy" 的模式匹配。更讽刺的是，当测试者将 GSM8K 题目中的数字做了微调（比如把"3 个苹果"改成"5 个苹果"）后，模型的准确率骤降 40%。这就是典型的 benchmark contamination。

基准去污染的常用技术是基于 n-gram Bloom filter 的方法：

基准去污染的可视化流程：

  步骤 1: 构建测试集 n-gram 索引
  ┌─────────────────────────────┐
  │ GSM8K 测试集题目           │
  │ "小明有 5 个苹果，小红有 3" │ → 提取所有 13-gram
  │ → Bloom Filter 索引        │
  └─────────────────────────────┘

  步骤 2: 扫描训练数据
  ┌─────────────────────────────┐
  │ 训练文档                   │
  │ "...小明有 5 个苹果..."    │ → 计算该文档的 n-gram
  │ → 与 Bloom Filter 比对     │
  └─────────────────────────────┘

  步骤 3: 判定污染
  如果训练文档中包含 ≥ 某个阈值的测试集 n-gram:
      → 标记为"疑似污染文档"
      → 选择: 移除 / 保留但降低采样权重

更先进的方案会使用更多技术组合：n-gram 匹配 + 嵌入相似度 + 语义哈希。我们在第 17 章会专门讨论基准去污染的深入技术。

6.5 许可证审计工作流

说了这么多，我们来总结一个可执行的许可证审计工作流：

  1. 数据源清单：列出所有拟用于训练的数据源（数据集名称、版本、来源 URL、下载时间）
  2. 数据集卡片审查：查看每个数据集在 HuggingFace 上的数据集卡片（Dataset Card），重点关注许可证字段和数据处理说明
  3. 子集逐项检查：对于包含多子集的数据集（如 The Pile），逐一检查每个子集的许可证
  4. 代码仓库许可证检查：对于代码类数据集（如 The Stack），检查源仓库的 LICENSE 文件
  5. 风险分级：按商业友好程度分级（绿灯 / 黄灯 / 红灯）
  6. 过滤受限来源：在数据处理管线中加入过滤器，排除不合规的数据源
  7. 文档溯源：记录每个数据源的处理方式、过滤结果和审计结论
当心"许可证漂移"：数据集的许可证是可能变化的。2024 年曾发生过一个事件：某个之前以 MIT 许可发布的数据集，在新版本中改为了非商业许可。如果你使用新版本的数据而没有注意许可证变更，就会面临合规风险。建议建立"许可证快照"机制——在下载使用数据集时，将当时的许可证文件或 URL 作为元数据记录下来，而不是"信任"未来版本可能保持同一许可证。

6.6 踩坑故事：那个用了"免费"数据集的公司

2023 年，一家 AI 创业公司为了快速训练出原型模型，在一个流行的开源"数据集集合"中下载了一个综合性的训练语料。公司从 HuggingFace 上下载了这个集合，翻了一下整体许可证（MIT），觉得"没问题"，就直接开训了。

九个月后，模型训练完成并上线了付费 API。但在上线后的第三个月，公司收到了一封法律函——某个大型出版商声称他们的版权书籍被包含在模型的训练数据中，并要求巨额赔偿。

经过内部审计，公司发现：他们使用的那个"MIT 许可的综合数据集"中，有一个子集是某爬虫项目从盗版网站上抓取的书籍数据。这个子集虽然在这个综合数据集的仓库里，但没有明确的许可证标注。公司的数据团队没有做子集级别的审计，就直接把整个数据集丢进了训练管线。

这个故事的结局是：公司花了 18 个月和数千万人民币来应对法律诉讼，最终和解并换掉了整个训练数据。教训是什么？——在数据合规这个问题上，省下的时间最终会以十倍的成本还回去。

安全第一原则：当你对某个数据源的版权状态不确定时，默认的决策应该是"不用"，而不是"先用了再说"。在数据来源上多花一周的审计时间，可以避免未来一年的法律纠纷。这句话值得写入每一个数据团队的 SOP。

本章小结

本章我们讨论了预训练数据工程中最严肃的话题——合规与安全。从数据集许可证的红绿灯分类，到 EU AI Act 的硬性要求，从 PII 去除的技术方案到基准去污染的方法论，再到可复制的许可证审计工作流。核心就一句话：数据合规不是法务部门的事，是数据工程的一部分。一个训练数据集的技术质量再高，如果合规有问题，它就是一颗定时炸弹。
