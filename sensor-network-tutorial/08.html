第8章 MAC层协议

大家好，我是你们的讲师。想象这样一个场景：在一个拥挤的会议室里，十几个人同时想发言，如果没有人协调，结果就是所有人都在说、没有人能听清——这就是无线通信中经典的"多路访问冲突"问题。在无线传感器网络中，成百上千个节点共享同一段无线频谱，如果大家想发就发，数据帧会在空中相互碰撞，最终全部损毁。谁来协调？怎么协调？这就是MAC层要回答的核心问题。

上一章我们搭建好了WSN的开发环境，本章将深入协议栈中最关键的一层——MAC（Media Access Control，介质访问控制）层。MAC层直接决定了信道利用效率、节点能耗和数据传输延迟，是WSN整体性能的"节拍器"。一个设计糟糕的MAC协议，哪怕路由协议再好，整个网络也会在碰撞和重传中迅速耗尽能量。

本章核心问题： 如何在共享无线信道上为成百上千个传感器节点提供高效、公平、低功耗的信道访问？竞争型和调度型两种思路各自适用于什么场景？学完本章，你将掌握WSN MAC层的完整知识图谱。

竞争型MAC：谁抢到谁用

竞争型MAC的基本哲学是"先听后说，冲突后退"。节点在发送前先监听信道是否空闲，如果空闲就发送；如果检测到冲突，就等待一个随机时间再重试。这套思路的鼻祖是Ethernet的CSMA/CD，但在无线环境中，我们没法一边发一边听（无线电在同一频率上不能同时收发），所以有了针对无线优化的CSMA/CA（Collision Avoidance，碰撞避免）。

CSMA/CA的工作原理

CSMA/CA的核心机制是"碰撞避免"而不是"碰撞检测"。它的工作流程如下：

// CSMA/CA 退避算法伪代码
function csma_ca_transmit(packet):
    // 步骤1：物理载波监听
    if channel_is_busy():
        // 信道忙，进入退避
        backoff_time = random(CW_min, CW_max) * slot_time
        wait(backoff_time)
        // 指数退避：连续碰撞时扩大竞争窗口
        CW = min(CW * 2, CW_max)

    // 步骤2：虚拟载波监听（RTS/CTS）
    if use_rts_cts:
        send_RTS(NAV_duration)
        wait_for_CTS(timeout)
        if no_CTS_received:
            // CTS超时，认为发生了碰撞
            CW = min(CW * 2, CW_max)
            goto step1

    // 步骤3：发送数据
    send(packet)
    wait_for_ACK(timeout)
    if ACK_received:
        // 发送成功，重置竞争窗口
        CW = CW_min
        return SUCCESS
    else:
        // ACK超时，重传
        if retry_count < MAX_RETRY:
            retry_count++
            CW = min(CW * 2, CW_max)
            goto step1
        else:
            return FAILURE

这个伪代码包含了CSMA/CA最核心的三个机制：物理载波监听（真的去听信道是否空闲）、虚拟载波监听（通过RTS/CTS交换预留信道时间）、指数退避（碰撞越多，等待窗口越大）。

关键参数： 在IEEE 802.15.4中，最小退避指数macMinBE=3（即CW_min=2^3=8个退避周期），最大退避指数macMaxBE=5（即CW_max=2^5=32）。每次退避单元时间为20个symbol，约0.32ms（2.4GHz频段）。这意味着一个节点最多等待32×0.32ms≈10ms后才能重试。

隐藏终端与暴露终端

在无线网络中，有两个经典问题一直是MAC协议设计者的噩梦：隐藏终端和暴露终端。

隐藏终端：节点A向B发送数据，节点C也在B的通信范围内但不在A的范围内。C监听不到A的发送，认为信道空闲，于是也向B发送——两个数据包在B处碰撞。这是无线网络中数据包丢失的首要原因。

暴露终端：节点B向A发送，节点C想向D发送。C监听到B的发送（因为C在B的范围内），认为信道忙于是等待。但实际上C的发送不会干扰A的接收（D也不在B的范围内），导致了不必要的延迟。

RTS/CTS握手机制就是为了解决隐藏终端问题而设计的。发送方先发一个RTS（Request to Send）短帧，接收方回复CTS（Clear to Send），RTS和CTS中都包含了预期的数据传输时长（NAV，Network Allocation Vector）。听到CTS的节点（包括隐藏终端）会在这段时间内保持沉默。

GTS机制不是万能的： 在密集部署的WSN中，RTS/CTS本身也可能发生碰撞。而且RTS/CTS的开销在小数据包场景下非常显著——传输一个30字节的传感器读数，却要额外花时间交换38字节的RTS/CTS握手帧，效率很低。因此在很多WSN MAC协议中（如IEEE 802.15.4），RTS/CTS是可选的，只在包较长或碰撞严重的场景下启用。

调度型MAC：轮流发言

与竞争型"谁抢到谁用"的思路相反，调度型MAC通过预先分配时隙来彻底避免碰撞。最经典的调度型MAC就是TDMA（时分多址）。

TDMA将时间划分为若干帧（Frame），每帧又划分为若干时隙（Slot），每个节点在属于自己的时隙内发送数据。关键的结构参数包括：每帧的时隙数（决定网络最大容量）、时隙长度（取决于最大数据包长度）、以及帧周期（决定最大延迟）。

调度型MAC的优点是显而易见的：

  • 无碰撞：每个节点在独占时隙中发送，完美避免隐藏终端问题
  • 能量高效：节点只在分配到的时隙唤醒，其他时间深度睡眠——占空比可以做到极低
  • 延迟有界：最大延迟 = 帧周期，适合实时应用

但它也有显著的缺点：

  • 时钟同步要求高：所有节点必须精确同步，这本身就需要额外的协议开销
  • 时隙分配不灵活：流量变化时，已分配的时隙可能浪费或不足
  • 扩展性问题：新节点加入需要重新分配时隙，在网络拓扑频繁变化时维护成本很高
设计启示： 竞争型MAC在低负载下延迟低、实现简单，但在高负载下碰撞严重、性能急剧恶化。调度型MAC在高负载下稳定高效，但在低负载下时隙浪费、灵活性差。聪明的MAC协议设计者往往会做混合设计——在低流量时用竞争模式，在高流量时切换为调度模式。

低功耗MAC协议

WSN的MAC协议设计有一个核心约束：节点大部分时间必须处于睡眠状态。监听信道本身就会消耗大量能量——事实上，无线电在接收（监听）状态下的功耗与发送状态相当。这意味着"一直开着收音机"的MAC协议在WSN中是不可接受的。

B-MAC——低功耗监听（LPL）

B-MAC（Berkeley MAC）是加州大学伯克利分校为TinyOS开发的经典低功耗MAC协议。它的核心思想是低功耗监听（Low Power Listening, LPL）：节点周期性地短暂唤醒，检测信道上是否有前导码，如果没有立即回去睡觉。发送方在发送数据前先发送一段长度略长于节点睡眠周期的前导码，确保接收方在唤醒时能检测到。

你可以这样理解：这就像你每隔5分钟从门缝里往外看一眼（采样监听），如果有邮递员在等，你就开门收信。邮递员则会在送信前大声喊5分钟（发送前导码），确保你在某次"看一眼"时能听见。

它的优点是极端简单、不需要同步，但缺点是前导码开销很大——不管数据包多小，发送方都要送一个几乎等于睡眠周期的前导码。在密集网络中，前导码造成的能量浪费和信道占用非常可观。

X-MAC——前导码选通

X-MAC对B-MAC做了关键改进：不发送连续的长前导码，而是发送短前导码序列，每个短前导码后面紧跟一个短暂的等待窗口，允许接收方在检测到前导码后立即发送ACK提前唤醒发送方。这样如果目标节点很快响应，发送方可以立即停止发送前导码，大大减少了前导码开销和信道占用。

实验数据显示，X-MAC相比B-MAC可以降低约50-60%的前导码能量消耗，且在多跳网络中端到端延迟也更低。

L-MAC——调度型低功耗

L-MAC（Lightweight MAC）走的是调度路线。它为每个节点分配一个固定的时隙，节点只在属于自己的时隙内活动。通过时隙复用的方式，L-MAC可以在保证无碰撞通信的同时实现极低的占空比（通常<1%）。但它依赖网络建立阶段的时隙协调，拓扑变化时需要时隙重分配。

IEEE 802.15.4 MAC超帧结构

IEEE 802.15.4标准（2003年发布，后续多次修订）定义了LR-WPAN（低速率无线个人区域网）的MAC层，是ZigBee、Thread、6LoWPAN等上层协议的基础。它的MAC层支持两种工作模式：信标使能模式和非信标使能模式。

在信标使能模式下，PAN协调器周期性地发送信标帧（Beacon），信标帧之间的时间区域称为超帧（Superframe）。超帧结构如下：

超帧部件 描述 功能
Beacon 信标帧（约10-25字节） 超帧起始标志，包含超帧结构描述和同步信息
CAP（竞争访问期） 使用CSMA/CA访问信道 普通数据的发送、新建入网请求
CFP（免竞争期） 包含7个GTS时隙 低延迟应用使用GTS直接发送，无需竞争
Inactive Period 无通信活动 节点进入睡眠状态，节省能量

CAP期间所有节点通过时隙化CSMA/CA竞争信道；CFP期间协调器将GTS（Guaranteed Time Slot，保证时隙）分配给特定节点，节点在自己的GTS内无需竞争直接发送。超帧中还包含一个非活动期，所有节点可以睡眠以节省能量。

超帧由两个参数控制：Beacon Order（BO）和Superframe Order（SO）。BO控制信标间隔（BI = 2^BO × 15.36ms），SO控制超帧活动期长度（SD = 2^SO × 15.36ms）。非活动期长度 = BI - SD。这些参数的设计对网络功耗和延迟有直接的影响。

MAC协议综合对比

协议 类型 功耗 延迟 吞吐量 同步需求 适用场景
CSMA/CA 竞争型 中 低（低负载） 中 无 一般WSN、低密度部署
TDMA 调度型 极低 有界（可控） 高 严格 工业控制、实时应用

前沿：TinyML增强MAC

2025-2026年，随着TinyML（微型机器学习）技术的成熟，一个全新的方向正在兴起——用微型ML模型优化信道接入。传统的CSMA/CA使用固定的退避算法，不知道网络流量的"模式"，碰撞了才被动退避。而TinyML增强的MAC则让每个节点在本地运行一个微型的流量预测模型（通常是一个几KB大小的轻量级神经网络），根据历史流量模式预测未来几毫秒的信道空闲概率，智能地选择发送时机。

加州大学洛杉矶分校（UCLA）2025年的实验表明，在100个节点的测试网络中，TinyML增强MAC相比传统CSMA/CA可以降低约32%的碰撞率和40%的重传能耗。更令人兴奋的是，模型推理仅消耗约0.8μJ的能量——这甚至不到发送一个字节所需能量的十分之一。

当然，TinyML增强MAC也有自己的挑战：模型需要在节点本地训练（联邦学习）以适应不断变化的网络环境，而传感器节点的RAM通常只有几十KB，能放下模型参数的空间非常有限。目前主流的方案是使用二元神经网络（BNN），将权重压缩到1bit，使得一个预测模型可以小到仅几百字节。

一个真实的"踩坑"故事

2022年，我在协助某工厂部署WSN设备监测系统时遇到了一个经典的CSMA碰撞问题。该工厂的冲压车间有约150台设备，每台设备上安装了一个振动传感器节点，要求每100ms上报一次三轴振动数据。部署完成后，经过一周的试运行，发现数据包丢失率平均高达40%以上，在高峰生产时段甚至超过60%。

最初的怀疑是信号干扰——车间里有大量电机和变频器，电磁环境确实复杂。但频谱分析仪测试后发现2.4GHz频段底噪正常。进一步抓包分析发现：所有150个节点使用默认的CSMA/CA参数（macMinBE=3, macMaxBE=5），在同时触发上报（设备同时启动时）引发大规模碰撞。

解决方法是两方面的：第一，我们将节点的上报时间做了随机化偏移——每个节点在启动后等待一个随机时间（0-100ms）再开始上报，避免所有节点同时竞争；第二，我们调整了退避参数，将macMinBE从3提高到5，让退避窗口更大。修改后一周的数据显示，丢包率从40%降至约3%，效果显著。

这个案例的教训是：WSN的参数配置绝不能照搬默认值。网络规模、流量模式、数据包大小——这些因素都会显著影响最优的MAC参数设置。部署前做好网络规划与仿真，部署后进行实测调优，才是正确的工程方法。
