第8章 GQA分组查询注意力

上一章我们聊了缩放定律——知道了"给大模型更多数据"比"把模型做大"更划算。但模型一旦大起来，推理阶段就有一个绕不过去的资源黑洞：KV Cache。你可能已经体验过这种情况：一个70B模型跑推理，batch size稍微开大一点，显存就直接爆了。问题出在哪？大部分显存都被注意力机制的KV Cache吃掉了。

这一章的主角——GQA（Grouped-Query Attention，分组查询注意力）——就是直接针对这个痛点设计的。它是从标准多头注意力（MHA）到多查询注意力（MQA）的演进过程中的一个"黄金中点"。理解了GQA，你就理解了LLaMA 2和LLaMA 3在推理效率上大幅领先同规模竞品的一个核心原因。

为什么GQA值得花一整章来讲？ 因为它不是那种"锦上添花"的优化——对于70B以上的大模型，GQA能直接让推理吞吐量提升30%-50%，同时几乎不损失模型质量。在LLaMA全线采用GQA之后，Mistral、Qwen、DeepSeek等主流开源模型也纷纷跟进。可以说，GQA已经成为现代大模型的标配组件。

8.1 MHA回顾：每个头都要自己的KV

在复习GQA之前，让我们先快速回顾一下标准多头注意力（MHA）中KV Cache是怎么产生的。如果你对这部分很熟悉，可以跳过直接看8.2节，但我建议你还是扫一眼——因为理解GQA的关键在于搞清楚"KV Cache到底怎么吃掉显存的"。

标准Transformer的自回归生成过程是这样工作的：每生成一个新token，模型需要计算当前token与之前所有token之间的注意力分数。但每次重新计算之前的注意力权重是非常浪费的——之前的Key和Value向量其实在上一轮已经算好了。所以实践中我们把这些Key和Value缓存下来，下一轮直接复用。这个缓存就是KV Cache。

在MHA中，每个注意力头都有自己的Query、Key和Value投影矩阵。假设我们有H个注意力头，隐藏层维度为D，每个头的维度为d = D/H。对于序列长度为L的生成过程，KV Cache的大小是：

KV Cache 内存 = 2 × L × H × d × sizeof(dtype)
                 = 2 × L × D × sizeof(dtype)

# 以 LLaMA 2 70B 为例：
# D = 8192, H = 64, 那么 d = 128
# 序列长度 L = 4096, dtype = float16 (2字节)
# 单条序列的 KV Cache = 2 × 4096 × 8192 × 2 = 134 MB
# 如果 batch size = 64，那就需要 134 × 64 = 8.5 GB！

# 这仅仅是一层的 KV Cache。70B模型有80层，
# 总 KV Cache = 8.5 GB × 80 = 685 GB！

# 是的，你没看错：70B模型用64的batch size推理，
# 光 KV Cache 就需要近700GB显存。
# 相比之下，模型权重本身只占约140GB（FP16）。

这个计算揭示了MHA的一个严重问题：随着序列长度和batch size的增长，KV Cache的显存消耗呈线性增长，很快超过模型权重本身。在长上下文场景（比如128K tokens）下，KV Cache更是天文数字。

MHA的另一个问题是"计算冗余"。直观上想：每个注意力头学到的Key和Value模式真的需要完全独立吗？在很多情况下，不同的注意力头可能会关注相似的语义模式——比如多个头都会学习到"主语-动词"的依赖关系，只是细节侧重不同。让每个头都各自维护一套KV投影，其实做了很多重复工作。

直觉类比： 想象一个大型图书馆有多个阅览室。在MHA中，每个阅览室（注意力头）都有自己独立的档案室（KV投影），每个档案室存一套完整的图书副本。这样当然查询效率高，但占地太大了。GQA的做法是：相邻的几个阅览室共享同一个档案室——找书的时候，不同的阅览室可以借阅同一套资料，但各自做不同的笔记（Query单独投影）。

8.2 MQA：一个极端的方案

在GQA出现之前，一个更激进的方案已经被提出了——MQA（Multi-Query Attention）。MQA的思路非常暴力：所有注意力头共享同一个Key投影和同一个Value投影，每个头只保留自己的Query投影。也就是说，无论有多少个头，KV只有一份。

MQA的效果如何？从显存角度看，它把KV Cache的大小降到了MHA的1/H——对于64头的模型，直接减少了98.4%的KV Cache。这是一个极其诱人的数字。但代价是模型质量的下滑：实验表明，MQA在多个benchmark上都会有可感知的质量损失，尤其是在需要精细语义理解的推理任务上。

为什么MQA会损失质量？原因不难理解：所有的查询头共享同一套Key-Value表示，意味着不同注意力头"看到"的信息是一样的，它们只能在Query侧做差异化的聚焦。这大大限制了注意力头的多样性表达能力。就好比一个团队做调研——所有人都去查同一份资料（共享KV），虽然效率高了，但观点和见解的同质化也严重了。

一点历史冷知识： MQA最早是2019年由Google在《Fast Transformer Decoding: One Write-Head Is All You Need》论文中提出的。这篇论文的影响力在当时很有限，因为2019年大家还在用小模型，KV Cache不是瓶颈。但到了LLaMA 2时代，随着模型规模膨胀到70B、上下文长度扩展到4K以上，MQA的这个"古老"思想突然变得极具价值。为什么MQA没有直接流行起来？原因很简单——质量损失太大。工业界需要的是一个"既要又要"的方案，而这正是GQA的舞台。

8.3 GQA：两全其美的中间方案

GQA由Google在2023年的论文《GQA: Training Generalized Multi-Query Transformer Models from Multi-Head Checkpoints》中提出，核心思想可以用一句话概括：把查询头分成G组，每组共享一对KV头。

当G=1时，GQA退化为MQA（所有头共享一组KV）。当G=H时，GQA等价于MHA（每个头有自己的KV）。所以GQA实际上是MHA和MQA的一个可调节的中间方案。通过调整组数G，我们可以在"推理效率"和"模型质量"之间找到最佳平衡点。

那么对于LLaMA系列来说，G取多少最合适呢？我们来看看各代模型的选择：

模型 隐藏维度 D 注意力头数 H KV头数 组数 G 每组头数
LLaMA 1 7B 4096 32 32 (MHA) 32 1
LLaMA 2 7B 4096 32 32 (MHA) 32 1
LLaMA 2 13B 5120 40 40 (MHA) 40 1
LLaMA 2 70B 8192 64 8 (GQA) 8 8
LLaMA 3 8B 4096 32 8 (GQA) 8 4
LLaMA 3 70B 8192 64 8 (GQA) 8 8
LLaMA 3 405B 16384 128 8 (GQA) 8 16

从这张表里可以读出几个重要信息：

  • LLaMA 1 全部采用MHA。 因为LLaMA 1主要面向学术研究，推理效率不是首要目标。
  • LLaMA 2 只在70B上使用了GQA。 7B和13B仍然是MHA——因为小模型的KV Cache压力没那么大，不需要引入GQA的额外复杂度。
  • LLaMA 3 全系列使用GQA，且统一为8个KV头。 从8B到405B，KV头数都是8。这个设计选择很有意思——即使是8B这样的小模型，Meta也换上了GQA，说明GQA带来的质量损失在LLaMA 3的规模和数据量下已经可以忽略不计了。
一个关键观察： KV头数固定为8，而查询头数从32（8B）增长到128（405B），也就意味着每组包含的查询头数从4个增长到16个。这说明GQA的"共享密度"在更大模型中反而更高——大模型有更多的参数冗余，对KV共享的容忍度更大。这也符合我们的直觉：模型越大，各注意力头的冗余度越高，共享KV的代价越小。

8.4 GQA的数学实现

下面让我们来看GQA的数学形式。假设我们有H个查询头，G个KV组（即G个KV头），每组的查询头数量为H/G。对于第g组：

# GQA 前向传播 (PyTorch伪代码)

import torch
import torch.nn.functional as F

def gqa_forward(query, key, value, num_heads, num_kv_heads):
    """
    Args:
        query: (batch_size, seq_len, hidden_dim)
        key:   (batch_size, seq_len, hidden_dim)
        value: (batch_size, seq_len, hidden_dim)
        num_heads: H (总查询头数)
        num_kv_heads: G (KV头数，= 组数)
    """
    B, L, D = query.shape
    head_dim = D // num_heads

    # 1. 投影到Q, K, V
    # Q: (B, L, H * head_dim)
    # K: (B, L, G * head_dim)
    # V: (B, L, G * head_dim)
    Q = q_proj(query).view(B, L, num_heads, head_dim)
    K = k_proj(key).view(B, L, num_kv_heads, head_dim)
    V = v_proj(value).view(B, L, num_kv_heads, head_dim)

    # 2. 关键步骤：将KV扩展到与Q相同的头数
    # repeats = num_heads // num_kv_heads = H/G
    # 每组内的所有查询头共享同一个KV
    K = K.repeat_interleave(num_heads // num_kv_heads, dim=2)
    V = V.repeat_interleave(num_heads // num_kv_heads, dim=2)
    # 现在 K, V: (B, L, H, head_dim)

    # 3. 转置为 (B, H, L, head_dim) 用于注意力计算
    Q = Q.transpose(1, 2)
    K = K.transpose(1, 2)
    V = V.transpose(1, 2)

    # 4. 标准缩放点积注意力
    scale = head_dim ** -0.5
    attn = (Q @ K.transpose(-2, -1)) * scale
    attn = F.softmax(attn, dim=-1)
    out = attn @ V  # (B, H, L, head_dim)

    # 5. 融合所有头
    out = out.transpose(1, 2).contiguous().view(B, L, D)
    return out

上面的代码中，最核心的是第2步的repeat_interleave操作。它的作用是把KV头"复制"到和查询头一样的数量。比如num_heads=64，num_kv_heads=8，那么每个KV头会被复制7份（repeats=8），最终得到64个KV头——与64个查询头一一对应。

实现细节： 在实际工程中，我们不会真的用repeat_interleave把KV显式扩展成H份——这样会浪费显存。更好的做法是在注意力分数计算时做"广播"（broadcasting）：Q的形状是(B, H, L, d)，K的形状是(B, G, L, d)，在计算Q×K^T时自动将K广播到H个查询头上。但在PyTorch中，更常见的实现是使用repeat_kv函数（如HuggingFace的实现），因为它更简洁且对编译器更友好。

8.5 GQA的实际收益：KV Cache与推理吞吐

GQA带来的好处是实实在在的，而且在大模型上效果更明显。我们用一个实际的对比来展示：

指标 MHA（LLaMA 2 70B） GQA（LLaMA 3 70B） 节省比例
KV头数 64 8 87.5%
每层KV Cache（单序列4K长度） 134 MB 16.8 MB 87.5%
全部80层KV Cache（单序列） 10.5 GB 1.3 GB 87.5%
batch size 64时总KV Cache 672 GB 84 GB 87.5%
MMLU基准分数 68.9 ~70.0 不降反升

KV Cache从672GB降到84GB是什么概念？这意味着同样用8张A100 80G（总共640GB显存），MHA模式下KV Cache就占满了全部显存，根本放不下模型权重。而GQA模式下，84GB的KV Cache加上140GB的模型权重，总共只需要224GB，还有大量富余空间来加大batch size或延长上下文。

真实踩坑： 我自己在做LLaMA 2 70B的推理优化时，曾经遇到过这样一个问题。我们使用了HuggingFace默认的实现，但发现即使我们只用batch size=1，显存占用也高达160GB（模型权重140GB + KV Cache 20GB）。后来排查发现，HF的某些版本实现中，即使我们传入的模型没有GQA（LLaMA 2 70B是有GQA的），它也默认在MHA模式下运行。解决方案是检查config.num_key_value_heads是否正确设置为8。如果你的实现中这个参数等于num_attention_heads（64），那就相当于退回到了MHA。这种配置错误非常隐蔽——模型推理不会报错，只是显存占用比预期高8倍，推理速度慢3-4倍。检查num_key_value_headers配置应该成为LLaMA推理优化清单的第一条。

8.6 GQA在训练和推理中的差异

理解GQA对训练和推理的不同影响，对于工程优化至关重要：

训练阶段： GQA对训练速度的影响不大。因为训练是并行处理整个序列的，不需要KV Cache。GQA在训练中主要影响的是模型参数数量——KV投影矩阵的参数变少了（从H×d减少到G×d），但这在大模型的总参数量中占比很小（通常不到0.5%），所以训练效率几乎没有变化。

推理阶段： GQA对推理的影响可以说是"脱胎换骨"的。Prefill阶段（并行处理prompt中的tokens）因为KV投影更少，计算量略微减少。Decode阶段（逐token生成）的改进则非常巨大——KV Cache内存占用降低到MHA的G/H，内存带宽需求也同步降低。在LLaMA 3 70B上，GQA使得decode吞吐量比同等规模的MHA模型提高了约2.5倍。

这种"训练几乎无差别，推理天差地别"的特性，使得GQA成为一个性价比极高的优化。你花在GQA上的设计成本是一次性的，而推理效率的提升是永久性的。

8.7 LLaMA的GQA配置演变

从LLaMA 1到LLaMA 3，GQA的配置变化反映了Meta对"质量-效率平衡"的持续探索：

  • LLaMA 1（2023.02）： 全部MHA。LLaMA 1的定位是研究基准模型，只关注质量，不考虑部署效率。
  • LLaMA 2（2023.07）： 70B模型引入GQA（8 KV heads），7B和13B保持MHA。这是一个务实的折中——小模型KV Cache压力小，不值得引入GQA；大模型必须用GQA才能实际部署。
  • LLaMA 3（2024.04）： 全线GQA，统一8 KV heads。原因有二：一是训练数据量达到15.6T，模型在充分训练后对GQA的质量损失容忍度更高；二是128K上下文长度下，即使8B小模型的KV Cache也大到不容忽视。
更深一层： LLaMA 3为什么选择8个KV头而不是4个或16个？从论文中的消融实验来看，KV头数从64（MHA）降到8，质量损失几乎可以忽略；但再降到4或2时，质量就会出现明显下降。而在部署侧，8个KV头已经把KV Cache减少了87.5%，再减少到4个虽然能再省一半，但边际收益已经不大。所以8是一个工程上的"甜点"——在质量和效率之间找到了最优平衡点。

这一趋势也深刻影响了开源社区。Mistral 7B使用了8个KV头，Qwen 2系列也全面采用了GQA，DeepSeek V2甚至更进一步提出了Multi-head Latent Attention（MLA）——一种比GQA更激进的KV压缩方案。可以说，GQA已经成为大模型注意力机制的事实标准。

说到KV压缩，其实还有更前沿的方向值得关注。比如我们刚才提到的MLA，它通过低秩分解把KV Cache压缩到更小的维度。还有正在探索中的"无KV Cache注意力"——通过线性注意力机制完全消除KV Cache。但这些都还是研究阶段，而GQA是经过大规模验证、可以在生产环境中放心使用的成熟技术。

好了，这一章我们深入剖析了GQA的原理、实现和收益。从MHA到MQA再到GQA，这条演进路径告诉我们：在大模型设计中，"恰到好处的共享"往往比"完全的独立"和"完全的共享"都更好。下一章，我们要聊的是另一个决定大模型质量的关键因素——预训练数据工程。LLaMA 1只用1T tokens就训练出了超越GPT-3 175B的模型，数据上的功夫功不可没。让我们一起看看Meta到底是怎么选数据的。
