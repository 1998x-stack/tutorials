第25章 模型部署与服务化

大家好，我是你们的讲师。前两章我们分别从代码视角和可视化视角学习了如何构建扩散模型推理管线。但一个残酷的现实是：能跑通和能上线是两回事。在 Jupyter Notebook 里生成一张猫图需要 3 秒钟，大家觉得"好快"；但如果在线上服务里一个请求需要 3 秒响应时间，加上排队和并发——算下来单卡 A100 可能每秒只能服务 2-3 个请求，吞吐量完全不够看。

这一章，我们将走完整条部署链路：从模型导出（PyTorch → ONNX → TensorRT）、推理优化（量化 / Torch Compile / 动态批处理）、到 API 服务工程设计（FastAPI + 请求排队 + 异步任务）。最后，我会分享一个真实踩坑案例——生产环境中的显存泄漏排查过程，这可能是你在任何官方文档中都学不到的经验。

25.1 模型导出：PyTorch → ONNX → TensorRT

在生产环境中，我们几乎从不直接使用 PyTorch 的 eager mode 跑推理——太慢了。标准的部署链路是先将模型编译成优化后的中间表示，再用 TensorRT 在 GPU 上生成极致优化的推理引擎。

整体流程分为三步：

# 第一步：PyTorch → ONNX 导出
import torch
from diffusers import UNet2DConditionModel

unet = UNet2DConditionModel.from_pretrained(
    "runwayml/stable-diffusion-v1-5", subfolder="unet",
    torch_dtype=torch.float16
).to("cuda")

# 构造模拟输入（ONNX export 需要 tracing）
dummy_latent = torch.randn(1, 4, 64, 64, dtype=torch.float16, device="cuda")
dummy_t = torch.tensor([999], dtype=torch.long, device="cuda")
dummy_encoder_hidden = torch.randn(1, 77, 768, dtype=torch.float16, device="cuda")

torch.onnx.export(
    unet,
    (dummy_latent, dummy_t, dummy_encoder_hidden),
    "unet.onnx",
    input_names=["latent", "timestep", "encoder_hidden_states"],
    output_names=["noise_pred"],
    dynamic_axes={"latent": {0: "batch"}, "encoder_hidden_states": {0: "batch"}},
    opset_version=17,
)

# 第二步：ONNX → TensorRT 转换（使用 trtexec）
# 命令行: trtexec --onnx=unet.onnx --saveEngine=unet.trt \
#   --minShapes=latent:1x4x64x64,timestep:1,encoder_hidden_states:1x77x768 \
#   --optShapes=latent:4x4x64x64,timestep:4,encoder_hidden_states:4x77x768 \
#   --maxShapes=latent:8x4x64x64,timestep:8,encoder_hidden_states:8x77x768 \
#   --fp16 --workspace=8192

上面的 --minShapes / --optShapes / --maxShapes 定义了 TensorRT 优化引擎支持的动态 batch 范围。这一步至关重要——如果你只指定了 batch=1 的 shape，后续的请求就无法使用动态批处理（dynamic batching）。建议 optShapes 设置为你的平均负载 batch size。

💡 经验之谈：在 TensorRT 转换中最容易踩的坑是 ops 不支持。Stable Diffusion 的 U-Net 中有一些 PyTorch 特有的操作（如 torch.split 的某些变体），ONNX 导出时可能报错。解决方案：使用 torch.onnx.export 的 custom_opsets 参数注册自定义操作，或者升级 TensorRT 到支持 ONNX opset 18+ 的版本（TensorRT 10.0+）。如果实在无法导出，可以退而使用 Torch-TensorRT（TorchScript + TensorRT 混合编译）作为替代方案。

25.2 推理优化技术

导出 TensorRT 引擎只是第一步。在整个推理管线中，还有多个可优化的环节：

优化技术效果（vs baseline）实现复杂度适用环节
FP16 推理2x 速度，-50% 显存低（一行代码）所有模型
INT8 量化（PTQ）3-4x 速度，-75% 显存中（需校准数据集）U-Net / VAE
Torch Compile1.2-1.5x 速度低（torch.compile(unet)）U-Net
动态批处理吞吐量 2-5x中（需请求排队逻辑）U-Net 推理
Scheduler 步数减少速度与步数成反比低（换 Scheduler）去噪循环
VAE 分块解码（tiling）大幅降低显存峰值低（参数开关）大图生成
连续批处理（vLLM）吞吐量 3-10x高（需框架支持）LLM + 扩散混合场景

其中 Torch Compile 是 2024 年 PyTorch 2.x 引入的重磅特性。它在第一次推理时编译计算图，后续调用跳过 Python 解释器直接执行编译后的内核。在 SDXL 的 U-Net 上，实测可以节省 20-30% 的推理时间——零侵入，一行代码：

import torch
from diffusers import StableDiffusionXLPipeline

pipe = StableDiffusionXLPipeline.from_pretrained(
    "stabilityai/stable-diffusion-xl-base-1.0",
    torch_dtype=torch.float16
).to("cuda")

# 仅添加这一行
pipe.unet = torch.compile(pipe.unet, mode="reduce-overhead", fullgraph=True)

# 第一次调用会编译（约 30-60 秒预热）
# 后续调用享受优化后的速度
image = pipe("a cat", num_inference_steps=20).images[0]
⚠️ 踩坑实录：我们在上线 sdpa（Scaled Dot-Product Attention）替代 xformers 时遇到了一个隐蔽的问题。xformers 的 memory-efficient attention 在某些 A100 驱动版本下会触发 CUDA illegal memory access 错误。当时排查了三天，逐一二分法回滚 commit，最后发现问题不是代码逻辑问题，而是NVIDIA 驱动版本 545.23.08 与 xformers 0.0.24 的一个兼容性 bug。解决方案是升级驱动到 550.54 并切换到 PyTorch 原生 sdpa。从此我学到了一课：在排查生产级推理问题时，永远不要忽略驱动 / CUDA 版本这个变量。

25.3 vLLM 在扩散模型中的应用

说到推理优化框架，就不得不提 vLLM。vLLM 最初是为 LLM 设计的推理引擎，其核心创新——PagedAttention 和 连续批处理（Continuous Batching）——在 2025-2026 年被逐步引入扩散模型推理场景。

连续批处理的精髓是：不等待完整的 batch 满员再处理，而是动态地在推理过程中将新到达的请求"插入"到当前的 batch 中。在扩散模型场景下，这意味着：当模型正在生成第 15 步时，一个新请求到达，它不需要等到当前请求完全结束才能开始处理——系统会在下一个调度周期将其合并进 batch。

对于扩散模型，vLLM 的适用性体现在两个层面：

  • 作为文本 encoder 推理引擎：在 RAG + 文生图的混合服务中，vLLM 可以高效处理大量的 prompt 编码请求，利用 PagedAttention 减少 KV Cache 的内存碎片。
  • 作为扩散 U-Net / DiT 的批处理层：社区正在开发将扩散模型的 step 调度与 vLLM 的连续批处理结合的新方案——让不同请求的 denoising step 共享同一个 GPU kernel launch。
# 使用 vLLM 加速 CLIP text encoder 推理
from vllm import LLM, SamplingParams

# 将 CLIP 模型通过 vLLM 加载
llm = LLM(
    model="openai/clip-vit-large-patch14",
    task="embed",      # embedding 模式
    max_num_seqs=256,  # 同时处理最多 256 个 prompt
    gpu_memory_utilization=0.3,  # 只为 text encoder 分配 30% 显存
)

prompts = ["a cat", "a dog", "a beautiful landscape"] * 10
outputs = llm.encode(prompts)  # 连续批处理，吞吐量远超逐条循环
clip_embeddings = [o.outputs.embedding for o in outputs]
🌟 核心洞察：在混合负载服务中（比如一个同时支持文本生成和图像生成的 API），使用 vLLM 管理所有的 text encoder + LLM 推理，使用 TensorRT 管理 U-Net / DiT 推理，两者通过异步队列连接——这是 2026 年最主流的生产架构。不要把鸡蛋放在一个篮子里：不同模型架构需要不同的优化引擎。

25.4 API 服务设计

一个生产级的 AIGC API 服务，不只是简单地在 FastAPI 里调用 pipe()。你需要考虑：请求排队、超时管理、异步任务、回调通知、成本核算……

下面是一个基于 FastAPI 的生产级架构设计：

from fastapi import FastAPI, BackgroundTasks
from pydantic import BaseModel
import asyncio
from typing import Optional

app = FastAPI(title="AIGC Generation Service")

# 请求模型
class GenRequest(BaseModel):
    prompt: str
    negative_prompt: Optional[str] = ""
    width: int = 1024
    height: int = 1024
    num_inference_steps: int = 20
    guidance_scale: float = 7.0
    seed: Optional[int] = None
    callback_url: Optional[str] = None  # 异步回调

class GenResponse(BaseModel):
    task_id: str
    status: str  # "queued" | "processing" | "completed" | "failed"
    estimated_seconds: int

# 全局任务队列（生产环境建议用 Redis Queue + Celery）
task_queue = asyncio.Queue(maxsize=128)

@app.post("/v1/generate", response_model=GenResponse)
async def generate(req: GenRequest, background_tasks: BackgroundTasks):
    task_id = f"task_{uuid.uuid4().hex[:8]}"

    # 检查队列长度，防止雪崩
    if task_queue.qsize() >= 100:
        raise HTTPException(status_code=503, detail="Server busy, try later")

    await task_queue.put((task_id, req))

    # 后台出队并执行推理
    background_tasks.add_task(process_queue, task_queue)
    return GenResponse(
        task_id=task_id,
        status="queued",
        estimated_seconds=task_queue.qsize() * 3
    )

async def process_queue(q: asyncio.Queue):
    """串行出队并推理（或动态批处理）"""
    task_id, req = await q.get()
    try:
        # 动态批处理：收集队列中所有等待的请求
        batch = [(task_id, req)]
        while not q.empty() and len(batch) < 4:  # 最大 batch 4
            batch.append(q.get_nowait())

        # 并行或批量执行
        results = await asyncio.gather(*[
            run_inference(tid, r) for tid, r in batch
        ])

        # 回调或存入结果缓存
        for tid, img in results:
            await save_result(tid, img)
            # 如果有 callback_url，POST 回结果

    except Exception as e:
        logger.error(f"Task {task_id} failed: {e}")
    finally:
        q.task_done()

上述设计中有几个关键点：

  • 请求排队：使用 asyncio.Queue 做内存队列，防止突发请求压垮 GPU。生产环境应替换为 Redis Queue 或 RabbitMQ。
  • 动态批处理：在 dequeue 时，一次性收集队列中等待的所有请求，组合成一个 batch 推理，大幅提高 GPU 利用率。
  • 异步回调：对于生成时间较长的请求（如 1024×1024 视频），使用 callback_url 机制异步通知，避免 HTTP 长连接超时。
  • 超载保护：当队列超过阈值时返回 503，让客户端自行重试——而不是让服务在 OOM 中崩溃。

25.6 真实踩坑：显存泄漏排查与模型热加载机制设计

这是 2024 年我负责的一个文生图 API 服务上线后遇到的真实问题。服务上线第三天，运维报警：单卡 A100 显存从 80 GB 稳步增长，每处理 1000 个请求后显存增加约 2 GB，最终在第 5000 个请求后触发 OOM，服务崩溃。

典型的显存泄漏（Memory Leak）。排查过程如下：

  • 第一轮排查：检查是否忘记 .detach() 或 .cpu() 释放中间张量。在推理代码中全量检查，没有发现显式泄漏。
  • 第二轮排查：使用 torch.cuda.memory_summary() 在每个请求执行前后打印显存快照。对比发现每次请求后 20 MB 左右的"增量"，累计在第 200 次请求后总量稳定——不是线性泄漏。
  • 第三轮排查（最终发现）：使用 torch.cuda.memory_snapshot() 分析内存分配记录，发现泄漏源是 Diffusers 的 VAE 解码过程中创建的 CUDA kernel 缓存未被释放。每次推理时，vae.decode() 会触发 cuDNN 的 autotune 过程，缓存大量自动调优的卷积 kernel 配置，这些配置不被 torch 的 garbage collector 回收。
⚠️ 解决方案与最佳实践：最终我们采用了三个措施彻底解决了泄漏问题：

① 设置 torch.backends.cudnn.benchmark = False，禁用 cuDNN autotune，避免不断积累 kernel 缓存。
② 引入 模型热加载机制——每处理 2000 个请求后，自动执行一次 pipe = pipe.to("cpu"); torch.cuda.empty_cache(); pipe = pipe.to("cuda")，彻底重置显存状态。这个过程约 2 秒，对服务可用性影响极小。
③ 添加 显存监控告警：在每次推理前检查 torch.cuda.memory_allocated() / torch.cuda.max_memory_allocated() 的比值，超过 85% 时触发预处理清理。

这些措施上线后，服务连续运行了 3 个月没有出现 OOM。有趣的是，这个问题在 Diffusers 0.28 版本中被官方部分修复——所以保持依赖更新也很重要。
