第13章 5G MAC与RLC层

大家好，我是你们的讲师。前面几章我们深入了物理层，了解了时频资源怎么分配、信道怎么映射、参考信号怎么用。但物理层本身没有"智能"——它只是一个听话的工人，告诉它传什么就传什么，不传就闲着。真正决定"什么时候传、传多少、给谁传"的，是MAC（Medium Access Control）层。

MAC层是协议栈中的"调度官"，它在每个传输时间间隔（TTI）做出关键的资源分配决策。做对了，全网吞吐量最大化；做错了，资源浪费、用户不满意。MAC层之上是RLC（Radio Link Control）层，负责数据的"物流管理"——把数据拆成合适的大小（分段）、组装回来（重组）、检查有没有丢包（重传）。

在5G时代，MAC层和RLC层的设计相比4G做了多处重要增强：更灵活的调度颗粒度、更细分的逻辑信道优先级、以及支持更复杂的URLLC和eMBB共存调度。理解MAC和RLC，你就理解了5G协议栈的"调度智慧"。

13.1 MAC层职责

5G NR MAC层（3GPP TS 38.321）的主要职责包括：

  • 调度（Scheduling）：在每个时隙决定哪些UE可以发送/接收数据、使用哪些RB、采用什么MCS。
  • 优先级处理（Priority Handling）：在不同逻辑信道和不同UE之间分配资源，确保高优先级业务（如URLLC）获得及时调度。
  • HARQ（混合自动重传请求）：管理HARQ进程，处理重传合并，提高传输可靠性。
  • 随机接入控制（Random Access Control）：管理UE的随机接入流程，包括前导码分配、冲突解决。
  • 逻辑信道复用（Logical Channel Multiplexing）：多个逻辑信道的数据在MAC层复用成传输块（TB），经过物理层传输。

关键指标：MAC调度器的性能直接影响整个小区和用户的体验。评价调度器好坏的三个核心指标是：吞吐量（总数据量）、公平性（各用户是否得到"应得"的资源）、时延（数据包从入队到出队的时间）。三者相互制约——一味追求吞吐量会牺牲公平性，一味追求公平性会增加时延。优秀的调度器在这三者之间找到最佳平衡点。

13.2 调度器类型

调度算法是MAC层的灵魂。以下是三种经典的调度算法：

调度算法 原理 吞吐量 公平性 复杂度 适用场景
轮询（Round Robin） 在每个调度周期，轮流向每个UE分配等量资源 低 最高（绝对公平） 最低 教学演示、均匀业务场景
最大C/I（Max SINR） 在每个调度周期把资源全部分配给信道条件最好的UE 最高 最低（边缘用户饥饿） 最低 信道质量差异小或只关心总吞吐量
比例公平（Proportional Fair, PF） 综合考虑瞬时信道质量和历史平均速率，优先级 = C/I / Ravg 高 高（兼顾边缘用户） 中 5G商用网络默认选择

比例公平调度器是目前5G商用网络中应用最广泛的算法。它的核心公式是：优先级 = 当前可达速率 / 历史平均速率。信道好的用户得到更多机会，但信道差的用户一旦信道恢复，因为它的历史平均速率低，优先级会迅速提升，保证不会被"饿死"。这种"成绩好的多学，成绩差的补课"的机制，兼顾了效率和公平。

在5G NR中，PF调度器还需要考虑更多因素：URLLC业务的时延要求、不同QoS流的GBR/Non-GBR特性、以及MU-MIMO配对用户之间的干扰协调。这些扩展使5G的调度器比4G复杂得多。

13.3 HARQ机制

HARQ（Hybrid Automatic Repeat Request）是物理层和MAC层之间的协作机制，处理数据包传输失败时的重传。5G NR采用了异步HARQ，每个HARQ进程独立管理重传。

HARQ的两个关键技术：

  • 增量冗余（Incremental Redundancy, IR）：每次重传发送不同的冗余版本（RV0/RV1/RV2/RV3），而不是简单重复原始数据。接收端将多次传输的软比特信息合并，获得编码增益。第一次传RV0，如果解码失败，重传RV2，合并后解码成功率大幅提升。
  • 软合并（Soft Combining）：接收端不解码成功的HARQ传输时，不丢弃软比特信息，而是缓存起来与下次重传的数据合并解码。Chase Combining（每次传相同数据，合并提高信噪比）和IR是两种主要的软合并方式。

下面是HARQ进程的状态机示意：

// HARQ进程状态机（简化表示）
typedef enum {
    HARQ_IDLE,          // 空闲：等待新数据传输
    HARQ_PENDING_ACK,   // 等待ACK/NACK反馈
    HARQ_RETX_PENDING,  // 等待重传调度
    HARQ_MAX_RETX       // 达到最大重传次数
} harq_state_t;

typedef struct {
    harq_state_t state;
    int tb_size;            // 传输块大小
    int retx_count;         // 当前重传次数
    int max_retx;           // 最大重传次数（通常4次）
    soft_buffer_t *buffer;  // 软合并缓冲区
} harq_process_t;

void harq_process_step(harq_process_t *proc, bool ack_received) {
    switch (proc->state) {
        case HARQ_IDLE:
            // 新数据传输触发
            proc->retx_count = 0;
            proc->state = HARQ_PENDING_ACK;
            break;

        case HARQ_PENDING_ACK:
            if (ack_received) {
                // 传输成功，释放HARQ进程
                clear_soft_buffer(proc->buffer);
                proc->state = HARQ_IDLE;
            } else {
                // NACK，触发重传
                proc->retx_count++;
                if (proc->retx_count >= proc->max_retx) {
                    proc->state = HARQ_MAX_RETX;
                    // 通知RLC层数据可能丢失
                } else {
                    proc->state = HARQ_RETX_PENDING;
                }
            }
            break;

        case HARQ_RETX_PENDING:
            // 等待基站调度重传资源
            // 一旦调度到重传资源，发送RV1/RV2/RV3
            proc->state = HARQ_PENDING_ACK;
            break;

        case HARQ_MAX_RETX:
            // 达到最大重传次数，丢弃该TB
            clear_soft_buffer(proc->buffer);
            proc->state = HARQ_IDLE;
            break;
    }
}

13.4 逻辑信道与传输信道映射

5G NR定义了多种逻辑信道，根据承载的业务类型不同，它们映射到不同的传输信道和物理信道：

逻辑信道 类型 承载内容 映射到传输信道
BCCH 广播控制 系统消息（MIB/SIB1/其他SIB） BCH（MIB）/ DL-SCH（SIB）
PCCH 寻呼控制 寻呼消息 PCH
CCCH 公共控制 初始接入时的RRC信令（RRC Setup Request等） UL/DL-SCH
DCCH 专用控制 UE专属的RRC信令（RRC Reconfiguration等） UL/DL-SCH
DTCH 专用业务 用户面数据（IP包、应用层数据） UL/DL-SCH

每个逻辑信道都有一个优先级值（1-16，数字越小优先级越高），以及相关的QoS参数（如优先级比特率PBR、令牌桶大小BSD）。MAC层的逻辑信道优先级（LCP）过程根据这些参数在每个调度周期决定先传哪个逻辑信道的数据。

13.5 RLC层三种模式

RLC层（3GPP TS 38.322）位于MAC层之上、PDCP层之下，主要负责分段/重组、重传检测和按序递交。RLC有三种工作模式，分别适用于不同场景：

模式 功能 确认机制 适用场景
TM（透传模式） 不添加RLC头，不分段不重组，直接透传 无确认 BCCH（系统消息）、CCCH（初始RRC信令）——这些消息无需分段，且对时延敏感
UM（非确认模式） 支持分段/重组，但不提供重传 无确认，丢失即丢弃 VoNR语音（实时业务，容忍少量丢包但不能等重传）、广播/组播
AM（确认模式） 支持分段/重组、重传、按序递交 ACK/NACK + 状态报告 普通数据业务（eMBB）、DCCH信令——必须确保数据完整和有序

AM模式是RLC最复杂的模式。它通过状态报告（Status PDU）反馈接收端的接收情况，发送端根据状态报告中的ACK/NACK决定哪些数据需要重传。RLC AM还维护一个重排序窗口，确保数据包按序递交到PDCP层。

选型建议：在5G网络规划中，每种DRB（数据无线承载）的RLC模式选择是需要仔细考虑的。VoNR使用UM模式（不重传，避免语音抖动），文件下载使用AM模式（确保完整性），直播流可以使用UM模式（允许偶尔丢帧但不能延迟）。选错了模式，要么应用体验差，要么资源浪费——没有"万能配置"。

13.6 RLC分段与重组

RLC的一个关键功能是把上层（PDCP）交付的SDU（服务数据单元）分割成适合MAC层传输的PDU。MAC层调度器在每个TTI通知RLC："这次可以传X个字节"，RLC根据这个配额把SDU切成大小合适的RLC PDU。

分段的逻辑是：如果当前SDU剩余大小正好等于MAC指示的大小，就完整发送；如果SDU太大，就切一部分并标记分段信息。分段信息记录在RLC PDU头中，包含序列号（SN）和分段偏移（SO）。接收端RLC根据这些信息将分段重组为完整的SDU。

RLC AM模式还有一个重要的功能：当接收端发现某个RLC PDU丢失时，会发送状态报告请求重传。发送端根据状态报告重传丢失的PDU，并可以调整分段边界——这就是"动态分段"能力。5G NR的RLC相比4G LTE支持更灵活的重分段（Re-segmentation），允许重传时使用不同的分段大小来适配新的MAC传输机会。

13.7 踩坑实录：RLC AM重排序定时器设置不当导致吞吐量骤降

2024年，某设备商在5G NR MAC/RLC联调测试中发现了一个反常现象：在弱信号区域（RSRP约-110 dBm），下行吞吐量从预期的200 Mbps骤降至只有30 Mbps，但信道质量（SINR）并不算太差。

排查过程充满了迷惑。从物理层看，PDSCH的BLER约15%，处于正常水平。HARQ重传率也在合理范围。但RLC层的统计显示：RLC层的重传率特别高，大量数据包在RLC层被重传，而且出现了明显的"窗口卡死"现象。

最终定位到根因：RLC AM的重排序定时器（t-Reassembly）被配置为35ms。在弱信号条件下，偶尔的HARQ重传会导致部分RLC PDU延迟到达，但并未真正丢失。然而，当某些PDU的延迟超过35ms时，接收端RLC判断为"丢包"，触发状态报告请求发送端重传。发送端重传的PDU到达后，接收端发现之前"丢失"的PDU其实也到了——于是出现了大量重复PDU，接收端不得不丢弃重复数据，导致有效吞吐量大幅下降。

解决方案：将t-Reassembly从35ms调整为100ms。这个调整给了HARQ重传更充足的时间，避免了误判"丢包"导致的重复传输。调整后，RLC重传率下降了90%，下行吞吐量恢复到180 Mbps。但需要注意的是，t-Reassembly也不能设得太大——否则真正丢失的PDU需要等待太久才能触发重传，导致用户面时延增加。

教训：RLC层的参数配置是"牵一发而动全身"的典型。t-Reassembly、t-StatusProhibit（状态报告禁止定时器）、t-PollRetransmit（轮询重传定时器）三个参数共同决定了RLC AM的行为，它们之间的互锁关系非常复杂。在实际优化中，建议从3GPP默认值开始，根据实际信道条件（时延分布、丢包率）逐步调整，不要一次性改太大。一个参数改错，全网的吞吐量都可能受影响。

MAC层负责调度、RLC层负责分段和重传，但整个协议栈的"大脑"在哪里？是RRC层。RRC层管理UE的状态、配置、移动性——它是空口信令的指挥中心。第14章我们将详细拆解RRC层和完整的5G信令流程。
