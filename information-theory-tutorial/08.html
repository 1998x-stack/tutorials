大家好，我是你们的讲师。

上一章我们用渐进均分性（AEP）证明了一个深刻的结果：一个独立同分布信源产生的n个输出符号，只需要大约2nH个码字就能描述。你可以理解为——一个英语文本文件，理论上可以压缩到原大小的H倍（以2为底）。

但这里有一个让人又爱又恨的微妙之处：AEP告诉我们"存在"这样的好编码。它没有告诉我们怎么造！就像一个建筑师告诉你"可以建一座100层的摩天大楼"，但是不给你设计图纸。

这就是信源编码理论要解决的问题：从"存在性"走向"构造性"。我们如何设计一个实用的编码方案，逼近那个理论上的极限——熵？

我自己第一次学到这个的时候，感觉就像——你知道了你自己能考100分（在理想条件下），但是你连试卷都还没看到。信源编码定理就是那张试卷的答案结构说明书。

编码的基本概念

在深入定理之前，我们需要统一一下语言。信息论中，编码这件事有非常精确的定义：

  • 信源符号集（Source alphabet）：信源输出的可能符号，记作 X = {x₁, x₂, ..., xm}
  • 码字（Codeword）：每个信源符号xi被映射到一个二进制串（或更一般地，某个码字母表上的串）
  • 编码（Code）：从信源符号到码字的映射函数，记作 C(x)
  • 码本（Codebook）：所有码字的集合
  • 码长（Code length）：码字C(xi)的长度，记作 li

但这只是定义。真正重要的区分在于编码的性质：

编码类型 定义 举例
唯一可译码
(Uniquely Decodable)		 任何编码序列只有一种解码方式 {0, 01} 不是唯一可译的——"01"可以解码为"0"+"1"或直接"01"
即时码
(Instantaneous / Prefix Code)		 没有任何码字是另一个码字的前缀 {0, 10, 110, 111} 是前缀码；{0, 01, 10} 不是
定长码
(Fixed-Length Code)		 所有码字长度相等 ASCII码：每个字符8位
变长码
(Variable-Length Code)		 码字长度可以不同 Huffman码、Shannon码

前缀码（Prefix Code）有一个非常优美的性质：它自同步。你不用"回头看"就能确定一个码字的边界在哪。这在实际通信中极其重要——想象一下你在实时视频通话时，解码器必须即时处理数据流，不可能等收到全部数据再解析。

接下来要介绍信息论中一个优雅的小定理——Kraft不等式。它就像一把数学"验钞机"，可以告诉你一组码长分配是否可能实现为前缀码。

Kraft不等式：对于任意二进制前缀码，码长 l₁, l₂, ..., l_m 满足
            m
           ∑    2^{-l_i} ≤ 1
           i=1

反过来，如果一组正整数 l_i 满足这个不等式，
就存在一个二进制前缀码以这些长度为码长。

这个不等式的直觉其实很简单：如果把所有可能的二进制码字想象成一棵完整的二叉树，根节点是空串，两个孩子分别是"0"和"1"，递归下去。选择一个码字相当于"砍掉"它下面所有子节点。2^{-l} 就是这个码字占用的"树空间比例"——短码字占用更多空间。所有码字占用的空间加起来不能超过1（整棵树）。

Kraft不等式的直觉：它是理解"什么样的码长分配是可能的"的钥匙——它告诉你：短码字"消费"了更多的Kraft预算。这就像你分蛋糕：每个人要的块（2^{-l_i}）不能超过整块蛋糕（1）。要块小蛋糕的人（长码字）可以有很多，要块大蛋糕的人（短码字）必须很少。

Shannon第一定理的正式陈述

准备好，我们要迎接信息论的第一个"大定理"了。Shannon信源编码定理（也叫Shannon第一定理）的正式陈述如下：

设信源X的熵为H(X)。对于任意无失真唯一可译码，其期望码长L = ∑ p(xi) li 满足：L ≥ H(X)。而且存在一个前缀码，使得 L < H(X) + 1。

这句话的含义极其深远：

  • 下界（Converse）：任何唯一可译码的平均码长都不可能低于信源的熵。这是通信的"热力学第二定律"——你不可能用少于H bits/符号的平均长度无损地表示信源。
  • 可达性（Achievability）：存在一种编码方案，平均码长不超过熵加1 bit。而且我们可以让这个"1 bit"随着编码块长的增加而任意减小（每符号）。

1951年，也就是香农发表信息论三年后，他在一篇后续论文中给出了一个具体的构造——Shannon码：

Shannon编码算法：
对于每个符号 x_i，码长 l_i = ⌈-log p(x_i)⌉

这个编码满足Kraft不等式吗？
检验：∑ 2^{-l_i} = ∑ 2^{-⌈-log p(x_i)⌉}
                 ≤ ∑ 2^{-(-log p(x_i))}   [因为 ⌈a⌉ ≥ a]
                 = ∑ p(x_i) = 1 ✓

所以它是前缀码，且平均码长 L ≤ H(X) + 1。

最优码长的下界

为什么熵是这个不可突破的下界？理解这个证明的直觉，比背诵证明本身更重要。

核心工具是Kraft不等式和相对熵（KL散度）的非负性。

对于任意唯一可译码，Kraft不等式告诉我们 ∑ 2^{-l_i} ≤ 1。定义辅助分布 q(x) = 2^{-l_i} / K，其中 K = ∑ 2^{-l_i} ≤ 1。那么：

L - H(X) = ∑ p(x_i) l_i - ∑ p(x_i) [-log p(x_i)]
         = ∑ p(x_i) [l_i + log p(x_i)]
         = ∑ p(x_i) [log p(x_i) - log(2^{-l_i})]
         = ∑ p(x_i) [log p(x_i) - log(q(x_i) K)]
         = ∑ p(x_i) [log p(x_i) - log q(x_i)] - log K
         = D(p || q) - log K
         ≥ 0 - 0 = 0   [因为 D ≥ 0 且 K ≤ 1 → -log K ≥ 0]

所以 L ≥ H(X)，等号成立当且仅当 D(p||q) = 0 且 K = 1，即 l_i = -log p(xi)——这就是"理想码长"条件。

完美码长的条件是：每个符号的码长恰好等于它的自信息量。但 -log p(xi) 通常不是整数，所以我们只能取整逼近——这也是Huffman编码要优化的地方。

编码效率与冗余度

在工程实践中，我们关心的是"你离理论极限有多近"。两个关键指标：

  • 编码效率 η = H(X) / L。η 越接近1越好。
  • 冗余度 R = 1 - η = (L - H) / L。

来看一个具体的例子。假设信源有4个符号：

import math

# 信源概率分布
probs = [0.5, 0.25, 0.125, 0.125]
H = -sum(p * math.log2(p) for p in probs)
print(f"熵 H = {H:.3f} bits/符号")

# Shannon编码
shannon_lengths = [math.ceil(-math.log2(p)) for p in probs]
L_shannon = sum(p * l for p, l in zip(probs, shannon_lengths))
print(f"Shannon编码平均码长 L = {L_shannon:.3f}")
print(f"效率 η = {H/L_shannon:.3f}")
print(f"冗余度 = {1 - H/L_shannon:.3f}")

# 理想情况（非整数）
ideal_lengths = [-math.log2(p) for p in probs]
L_ideal = sum(p * l for p, l in zip(probs, ideal_lengths))
print(f"\n理想（非整数）码长平均 = {L_ideal:.3f}")

# 输出：
# 熵 H = 1.750 bits/符号
# Shannon编码平均码长 L = 1.875
# 效率 η = 0.933
# 冗余度 = 0.067

这里效率93.3%，已经很不错了。但如果符号概率不是2的负幂次，效率可能会更低。这时Huffman编码可以做得更好。

编码方案 平均码长 L L - H 的差距 特点
定长编码 ⌈log₂m⌉ 可能很大（尤其当概率不均匀时） 简单，但浪费
Shannon编码 ∑ p(x)⌈-log p(x)⌉ < 1 bit 保证在 H+1 之内
Huffman编码 最优前缀码 ≤ Shannon码 理论上最优的前缀码
算术编码 接近 H 可任意小（长消息） 不需要整数码长

我当年在用Shannon码做一个文本压缩实验时，踩过一个坑。我天真地以为把概率排好序然后分配码长就完事了——结果发现解码时总是错位。原因是我没有验证Kraft不等式，分配的码字不是前缀码。Kraft不等式不仅是一个理论工具，在实际编码实现中，它就是你的"安全检查"。

核心信息：Shannon第一定理告诉我们，无损压缩的终极下界就是信源的熵。无论你用多聪明的算法，平均每个符号需要的比特数不能少于H(X)。Huffman、算术、LZ——所有压缩算法都在做同一件事：尽可能逼近这个极限。

那么，既然理论上限是H(X)，实际中我们怎么构造一个实用且最优的编码呢？Shannon码已经能保证在H+1之内，但这还不够好——我们需要的是"最优"的。下一章我们要讲的Huffman编码，就给出了一种令人拍案叫绝的贪心构造方法。

我至今还记得第一次学Huffman算法时的感受：面前是一个看似需要动态规划才能解决的全局优化问题，结果Huffman用一个简单的"自底向上合并"就搞定了——而且证明它是全局最优的。这就是天才的工作。

但在此之前，让我们先通过一个代码示例来直观感受一下Kraft不等式在实际中的验证作用：

def check_kraft(code_lengths):
    """
    验证一组码长是否满足Kraft不等式。
    如果满足，返回 (True, sum)；否则返回 (False, sum)。
    """
    total = sum(2 ** (-l) for l in code_lengths)
    return total <= 1, total

# 测试一些例子
examples = [
    ("例1", [1, 2, 3, 3]),        # 经典前缀码 0, 10, 110, 111
    ("例2", [1, 2, 2, 3]),        # 可能出问题
    ("例3", [2, 2, 2, 2]),        # 四个等长码
    ("例4", [1, 1, 2, 3]),        # 大概率违规
]

for name, lengths in examples:
    ok, s = check_kraft(lengths)
    print(f"{name}: 码长 {lengths} → ∑2^-l = {s:.3f} → {'✓ 可行' if ok else '✗ 不可行'}")

# 输出：
# 例1: 码长 [1, 2, 3, 3] → ∑2^-l = 1.000 → ✓ 可行
# 例2: 码长 [1, 2, 2, 3] → ∑2^-l = 1.125 → ✗ 不可行
# 例3: 码长 [2, 2, 2, 2] → ∑2^-l = 1.000 → ✓ 可行
# 例4: 码长 [1, 1, 2, 3] → ∑2^-l = 1.375 → ✗ 不可行

注意例2：[1, 2, 2, 3]看起来挺合理，但它不满足Kraft不等式！为什么？因为码长1意味着码字是"0"或"1"，占了一半的树空间。再加两个码长2的（各占1/4）和一个码长3的（占1/8），加起来1.125 > 1，已经没有足够的树空间了。

所以下次你设计变长编码的时候，先算算Kraft不等式——它能帮你避免走弯路。
