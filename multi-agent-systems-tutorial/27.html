第27章 Agent互联网与开放生态

欢迎来到全书的倒数第二章！上一章我们讨论了Agent在物理世界的协作——具身集体智能。如果说"具身智能"回答的是Agent和人、物理世界的关系，那这一章我们要回答一个更加宏大、也更激动人心的问题：Agent和Agent之间如何在一个开放的、跨组织的生态中互联互通？

想象一下你正在构建一个企业级的智能采购系统。你需要一个Agent去询价，另一个Agent去比价，第三个Agent去核验供应商资质——这些Agent可能来自不同的公司，运行在不同的平台上，使用不同的模型。它们要如何像人类之间交换名片、电子邮件、合同一样顺畅地沟通？答案就是——Agent协议栈。

核心洞察： 2024年的Agent是"孤岛"——每个Agent都是自己玩自己的。2025—2026年的Agent开始"联网"，就像1990年代互联网把孤立的计算机连接起来一样。Agent互联网（Internet of Agents, IoA）正在成为现实。它的市场规模有多大？2026年中国Agent市场449亿元，预计2029年达到3320亿元，年复合增长率超过94%。

27.1 协议全景：五层Agent协议栈

正如互联网有TCP/IP协议栈的分层架构（物理层→网络层→传输层→应用层），Agent互联网也在形成自己的分层协议体系。到2026年中，业界基本共识是五层Agent协议栈：

层级 协议 定位 类比互联网 状态
工具层 MCP (Model Context Protocol) Agent与工具/数据源的标准化接口 USB驱动 W3C标准提案，2024年底开源
协作层 A2A (Agent-to-Agent) Agent之间的通信、任务协商与协作 HTTP协议 Google开源，2025年发布
发现层 ANP (Agent Name Protocol) Agent注册、发现、身份验证 DNS/域名系统 社区草案，2025年底
传输层 AGTP (Agent Transport Protocol) Agent之间的结构化消息传输 TCP/IP IETF草案（2026年3月）
实时层 MOQT (Media over QUIC Transport) 音视频/实时数据流传输 WebRTC IETF草案（2026年3月）

这个五层协议栈不是由某一家公司设计的顶层架构，而是过去两年行业实践中"涌现"出来的结构。每层解决一个特定的问题，层与层之间通过标准接口解耦。这种分层方式的好处是——底层协议的演进不会影响上层应用，反之亦然。

// Agent互联网五层协议栈交互示意
//
// +----------------------------------------------------+
// |                   应用层                            |
// |  (采购Agent、客服Agent、分析Agent...任意业务场景)    |
// +---------------------------+------------------------+
//                           |
// +---------------------------v------------------------+
// |  发现层 ANP              |  Agent注册/发现/身份      |
// |  协作层 A2A              |  任务/协商/能力声明       |
// |  工具层 MCP              |  工具调用/数据访问        |
// +---------------------------+------------------------+
//                           |
// +---------------------------v------------------------+
// |  传输层 AGTP             |  可靠消息/流控制/路由     |
// |  实时层 MOQT             |  音视频/实时数据传输      |
// +---------------------------+------------------------+
//                           |
//                    [物理网络层]
//                  (TCP/IP, QUIC, etc.)

// Agent A（采购Agent） → Agent B（询价Agent）
// A2A消息体:
{
  "protocol": "A2A/1.0",
  "transport": "AGTP/0.1",
  "message_type": "task_request",
  "sender": "agent://acme-corp/purchasing-agent-01",
  "target": "agent://supplier-inc/quote-agent-03",
  "task": {
    "type": "price_inquiry",
    "params": {
      "sku": "SPR-2026-00421",
      "quantity": 5000,
      "delivery_deadline": "2026-08-15"
    },
    "deadline": "2026-07-02T12:00:00Z"
  },
  "auth": {
    "scheme": "agent_jwt",
    "issuer": "did:agent:acme-corp"
  }
}
类比指南： 如果你熟悉互联网协议栈，Agent协议栈的映射就非常直观。ANP ≈ DNS（找人），A2A ≈ HTTP（聊天），MCP ≈ USB驱动（用工具），AGTP ≈ TCP（可靠传输），MOQT ≈ WebRTC（实时流）。Agent互联网本质上就是互联网的"AI Agent版"。

27.2 Linux Foundation Agentic AI Foundation

协议能否成为真正的行业标准，取决于是否有足够强的治理机构来推动和保持中立性。2025年底成立的Linux Foundation Agentic AI Foundation（LF AI & Agent）就是为了解决这个问题而生的。

治理结构

LF Agentic AI Foundation采用Linux Foundation成熟的三层治理模型：

  • 理事会（Governing Board）： 由白金会员代表组成，负责战略方向、预算审批和项目管理
  • 技术监督委员会（TSC）： 由技术负责人组成，负责技术架构决策、标准制定和项目孵化
  • 工作组（Working Groups）： 包括MCP工作组、A2A工作组、安全工作组、互操作性工作组等，负责具体的标准推进

白金会员阵容

一个基金会的影响力第一个信号就是它的会员阵容。LF Agentic AI Foundation的七家白金会员几乎涵盖了全球云计算和AI基础设施的全部顶级玩家：

公司 核心贡献 关注方向
Google A2A协议、A2UI规范、Agent SDK Agent间通信标准
AWS Amazon Bedrock Agent、Multi-Agent Collaboration 云原生Agent基础设施
Microsoft AutoGen、Copilot Agent框架 企业级Agent编排
Cisco Agent网络安全框架 Agent通信安全和网络
Salesforce Agentforce、Einstein Trust Layer CRM领域的Agent应用
SAP Joule Agent、Business AI 企业资源管理的Agent集成
ServiceNow Now Assist、ITSM Agent IT服务管理的Agent自动化

这份名单的深意在于：这不是AI公司的联盟，而是"所有需要AI的公司"的联盟。Google和AWS提供Agent运行的云基础设施，Cisco提供安全网络，Salesforce、SAP、ServiceNow则代表Agent实际落地的企业场景。这种跨界联盟让协议标准不只是停留在技术文档，而是直接嵌入到企业的业务流程中。

生态观察： LF Agentic AI Foundation的出现是一个重要的分水岭。在此之前的Agent协议大多是"一家公司说了算"（比如MCP最初是Anthropic提出的），但到了2026年，这些协议必须走向中立化。Anthropic把MCP捐给W3C，Google把A2A捐给Linux Foundation——无一例外。因为没有一个企业愿意把自己的Agent生态建立在竞争对手的专有协议上。开源 + 中立治理，才是Agent互联网的根基。

27.3 IETF标准化进展

如果说LF Agentic AI Foundation是Agent协议的经济和政治背书，那么IETF（互联网工程任务组）的标准化则是技术层面的"盖棺定论"。IETF是互联网核心标准（HTTP、TCP/IP、TLS）的发源地，一个协议成为IETF RFC意味着它有了全球通用的技术权威。

2026年，两个关键的Agent协议正在IETF推进：

27.3.1 AGTP（Agent Transport Protocol）

正式文档：draft-hood-agtp-composition-00，由Chase Hood等人于2026年3月提交。

AGTP解决了Agent之间可靠消息传递的问题。它定义了一种轻量级的传输层协议，专门为Agent之间的消息交换优化：

  • 消息类型： 请求、响应、流、事件——覆盖Agent通信的全部模式
  • 路由： 支持点对点、发布-订阅、广播等模式
  • 流控制： 防止一个Agent发送消息过快压倒另一个Agent
  • 可组合性： 一个AGTP会话可以嵌套另一个AGTP会话（例如Agent A通过B转发请求给C）

27.3.2 MOQT（Media over QUIC Transport for AI Agents）

正式文档：draft-nandakumar-ai-agent-moq-transport-00，由Suhas Nandakumar等人于2026年3月提交。

MOQT解决的是Agent之间实时多媒体传输的需求。考虑一个具身Agent（如机器人）需要将摄像头视频流实时发送给另一个Agent进行分析——传统的HTTP请求-响应模式或A2A的JSON消息都无法满足这种高吞吐、低延迟的传输需求。MOQT基于Google的QUIC协议（已广泛应用于HTTP/3），专门为Agent之间的实时数据流设计了传输机制。

// AGTP消息格式示例
// 摘自 draft-hood-agtp-composition-00 (简化)

message AgentMessage {
    // 消息头
    string message_id = 1;         // UUID，全局唯一
    string session_id = 2;         // 会话标识
    AgentURI sender = 3;           // 发送Agent的URI
    AgentURI recipient = 4;        // 接收Agent的URI
    MessageType type = 5;          // REQUEST | RESPONSE | STREAM | EVENT

    // 路由信息（支持多跳转发）
    repeated AgentURI path = 6;    // 消息经过的Agent路径

    // 流控制
    uint32 sequence_number = 7;    // 流中消息序号
    optional uint32 window_size = 8; // 接收窗口

    // 有效载荷
    bytes payload = 9;             // 实际消息内容（通常是A2A消息）
    ContentType content_type = 10; // JSON | Protobuf | CBOR

    // QOS
    Priority priority = 11;        // HIGH | MEDIUM | LOW
    uint32 max_ttl_ms = 12;        // 最大存活时间（毫秒）
}

// MOQT帧结构（用于实时视频/音频传输）
message MediaFrame {
    string stream_id = 1;          // 流标识
    uint64 timestamp_us = 2;       // 时间戳（微秒）
    Codec codec = 3;               // H264 | VP9 | Opus
    bytes frame_data = 4;          // 编码帧数据
    bool key_frame = 5;            // 是否为关键帧
}
标准化节奏： IETF的标准化周期通常需要2-3年。AGTP和MOQT目前还是draft状态（00版本），预计2027—2028年进入RFC阶段。但协议的价值不只在最终标准——IETF draft本身就具有技术指导意义，早期采用者可以根据draft实现协议，获得先发优势。

27.4 跨组织Agent协作

协议的终极目标是实现跨组织的Agent互联。当Agent不再局限于一个企业内部的封闭环境，而是能够跨越组织边界协同工作时，真正的"Agent互联网"就诞生了。

企业间Agent互操作的三大挑战

信任问题： 你如何确保来自另一个企业的Agent不会窃取数据或执行恶意操作？解决方案是Agent身份与凭证——每个Agent在ANP中注册时绑定一个去中心化身份（DID），所有通信都经过Agent JWT签名认证。LF基金会正在制定Agent安全认证规范，定义了"Agent ID卡"的标准格式。

隐私保护计算： 跨组织协作中数据不能裸传。联邦式MAS的思路是——Agent不在组织之间共享原始数据，而是共享模型更新或加密计算结果。例如：两家银行的Agent要协作识别跨行洗钱，但它们不能交换客户交易明细。方案是使用安全多方计算（Secure MPC），每家的Agent在加密数据上运行洗钱检测模型，只输出"可疑/正常"标签。

语义互操作性： 不同组织的Agent可能使用不同的本体和术语。你说"客户"我说"用户"，你说"订单状态"我说"工单阶段"。解决方案是Agent能力声明——每个Agent公开自己的能力时附带语义标注（使用JSON-LD + Schema.org扩展），消费方Agent可以自动映射语义。

// Agent能力声明（通过ANP发布）
{
  "@context": "https://agent-schema.org/v1",
  "agent_id": "agent://acme-corp/logistics-agent-02",
  "name": "ACME物流调度Agent",
  "capabilities": [
    {
      "action": "query_shipping_status",
      "description": "查询物流订单的实时状态",
      "input_schema": {
        "type": "object",
        "properties": {
          "order_id": {"type": "string", "description": "物流订单号"},
          "tracking_number": {"type": "string", "description": "运单号（可选）"}
        },
        "required": ["order_id"]
      },
      "output_schema": {
        "type": "object",
        "properties": {
          "status": {"type": "string", "enum": ["pending", "in_transit", "delivered", "exception"]},
          "estimated_delivery": {"type": "string", "format": "date-time"},
          "current_location": {"type": "string"}
        }
      },
      "privacy_level": "business_public",
      "pricing": {"type": "free"}
    }
  ],
  "auth_required": true,
  "trust_score": 4.8
}

27.5 智能体经济：Agent-to-Agent支付与市场化

当Agent可以在组织之间自由协作，一个自然而然的延伸就是——Agent经济。Agent帮别的Agent干活，应该获得报酬。这不是科幻，而是在2026年已经发生的商业现实。

Agent-to-Agent支付

核心模式是微支付（Micropayments）。一个Agent调用另一个Agent的API服务，每次调用可能只花几分钱。传统的支付系统（信用卡、银行转账）无法处理这种频率和额度——手续费比支付金额还高。解决方案是Agent原生支付渠道，如基于加密稳定币的流支付（Streaming Payments）：Agent A每处理一个请求就实时向Agent B支付0.002 USDC。

API市场化

Agent的能力本质上就是API。当每个Agent都能通过ANP注册自己的能力并报价，就形成了一个Agent能力市场。开发者不用再手动注册API密钥和配置工具——Agent自动在市场中发现合适的服务、协商价格、完成对接。到2026年中，已经出现了多个Agent能力交易平台，如Agora、Agent-to-Agent Marketplace等。

身份和声誉系统

在一个开放的Agent经济中，你如何知道该信任哪个Agent？答案是去中心化的身份和声誉系统：

  • 身份： 每个Agent拥有一个去中心化标识符（DID），由ANP负责维护
  • 声誉： Agent完成任务后，协作方可以给它评分。评分记录在链上（或分布式账本），不可篡改
  • 质押： Agent可以提供质押金作为"诚信押金"，一旦违约则扣除质押
风险警示： Agent经济也带来了新的安全风险。2025年已经出现过"Sybil攻击"——一个恶意方创建大量虚假Agent好评来提升声誉。此外，Agent之间的自动支付也带来了"Agent欺诈"的可能（恶意Agent假装完成任务后逃逸）。这些问题正在通过声誉机制 + 争议仲裁Agent来解决——当一次交易出现争议时，一个中立的"仲裁Agent"会根据双方提供的证据做出裁决。这是一个Agent管理Agent的元问题，在安全领域被广泛讨论。

这些数字提醒我们——Agent互联网不只是一个技术话题，更是一个经济话题。未来的企业竞争力将不只看"有多少员工"，还要看"有多少Agent合作伙伴"。能优先接入Agent互联网的企业，将在效率和创新上获得代际优势。
