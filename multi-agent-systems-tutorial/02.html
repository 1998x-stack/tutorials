第2章 智能体的认知架构

想象一下，你让一个智能体去执行这样一个任务："帮我查一下上季度的销售数据，分析同比和环比趋势，然后写一份简短的报告。"这个任务对人类助手来说可能只需要三步：查询、分析、撰写。但对一个AI Agent来说，背后是一整套认知架构在运转——感知输入、内部推理、决定行动、观察结果、调整策略。这个循环如何在代码层面实现？它和人类的"思考—行动"循环有什么异同？

在上一章中，我们讨论了单体Agent的局限性——一个Agent试图做所有事情容易在多个环节出错。但要想理解为什么多个Agent协作能突破瓶颈，我们首先得明白一个Agent的内部到底是怎么工作的。这就好比你要组建一支足球队，你首先得知道一个球员怎么踢球，才能理解十一个球员怎么配合。

这一章，我们将深入智能体的"大脑"——它的认知架构。我们会从经典的感知-规划-行动循环入手，然后逐步展开LLM时代Agent的核心组件：ReAct推理模式、工具调用机制、记忆系统设计。最后，我们会诚实地审视单个Agent的能力边界——这些边界恰恰是推动我们将目光转向多智能体系统的根本原因。

2.1 感知-规划-行动循环

在AI和机器人学领域，Agent的基本认知模型可以归纳为一个经典的三阶段闭环：感知（Perception）→ 规划（Planning）→ 行动（Action）。这个模型最早来自控制论和机器人学，但在LLM时代得到了全新的诠释。

感知阶段： Agent接收并理解当前环境信息。在机器人场景中，这可能是摄像头图像或激光雷达数据。在LLM Agent场景中，这包括了用户输入的文本、系统Prompt、工具返回的结果、以及对话历史中的上下文。关键区别在于：LLM Agent的"感知"不是原始信号处理，而是对语义信息的理解——这意味着它可以"读取"文字中隐含的意图、情感和上下文。

规划阶段： 基于感知到的信息，Agent制定达成目标的行动策略。传统方法包括分层任务网络（HTN）、行为树、状态机等。LLM时代的规划则更加灵活——模型通过Chain-of-Thought或多步推理来规划下一步行动。推理不是预设的流程图，而是模型实时生成的思考链。

行动阶段： Agent执行选定的行动并观察结果。这可以是一个API调用、一个数据库查询、一段代码执行、或者一段文本回复。更重要的是，行动的结果会重新进入感知阶段，构成下一轮循环的输入。

阶段 传统Agent（机器人学） LLM Agent（当前） 关键差异
感知 传感器信号处理 语义理解 + 多模态输入 LLM Agent能处理抽象概念和隐含意图
规划 预设策略 / 搜索树 实时推理生成（CoT / ReAct） LLM Agent的规划是动态生成的，而非静态预设
行动 电机控制 / 简单操作 API调用 / 代码执行 / 文本回复 LLM Agent的操作空间更抽象、更广阔
反馈 物理传感器读数 工具返回 / 用户反馈 / 环境状态变更 LLM Agent的反馈通常以结构化或半结构化文本呈现

你可以把感知-规划-行动循环想象成一个"不断旋转的飞轮"。Agent每完成一次循环，都获取了新的信息，从而能做出更好的决策。这正是Agent区别于传统程序的核心特征：它不是一次执行就结束的静态逻辑，而是持续与环境交互的动态过程。

核心洞察： 很多人误以为Agent只是一个"能调用函数的LLM"。但真正的Agent是"具备持久化循环能力的LLM"。一次完整的感知→规划→行动→再感知的循环，才是Agent的最小单位。没有循环，就没有Agent——只有API调用。

2.2 ReAct模式：推理与行动交织

在认知循环的"规划"阶段，有几种主要的推理策略。如果用一个谱系来描述，一端是纯推理（只思考不行动），另一端是纯行动（只行动不思考）。而ReAct模式恰好位于二者中间——推理与行动交织进行。

纯推理（Chain-of-Thought, CoT）： 模型一步一步地推理，最终给出答案。适合数学题、逻辑题等不需要外部信息的场景。但CoT有个致命问题：它只能依赖模型内部的知识——如果模型不知道某个事实，推理再长也没用。

纯行动（Act-Only）： 模型直接调用工具或执行操作，没有显式的推理步骤。适合简单、确定的操作，比如"查询北京的天气"。但对于需要多步推理的复杂任务，盲目行动会导致灾难性错误。

ReAct（推理+行动）： Thought → Action → Observation 循环。模型先生成一个"Thought"（当前推理），然后决定一个"Action"（调用什么工具），得到"Observation"（工具返回的结果），再根据Observation生成下一个Thought——如此循环直到任务完成。

ReAct模式是2022年由Shunyu Yao等人提出的，它的核心洞察是：推理帮助模型在行动前想清楚，行动帮助推理获取外部信息。二者互相促进，而不是彼此替代。

ReAct循环的伪代码

def react_loop(task, max_steps=10):
    """ReAct循环：Thought → Action → Observation"""
    context = [
        {"role": "system", "content": SYSTEM_PROMPT_WITH_REACT},
        {"role": "user", "content": task}
    ]

    for step in range(max_steps):
        # Step 1: 模型生成推理 + 行动决策
        response = llm.chat(context)
        thought = extract_thought(response)
        action = extract_action(response)

        if action["type"] == "final_answer":
            return response  # 任务完成

        # Step 2: 执行行动，获取观察
        observation = execute_tool(action)
        context.append({"role": "assistant", "content": response})
        context.append({"role": "tool", "content": observation})

    return "Max steps reached."

这个循环最精妙的地方在于：它不需要预设任何流程图或状态机。Agent的每一步推理和行动都是动态生成的，模型根据当前上下文自主决定下一步做什么。这赋予了Agent极强的灵活性来应对不可预见的场景。

工程经验： 在实际项目中，我建议设置最大循环次数（max_steps）来防止无限循环。一个常见的错误是Agent在不断调用工具获取信息，但从未真正给出答案——这本质上是"无限推理但不敢决策"的回避行为。设置max_steps=10或15通常是合理的，超时后强制Agent给出阶段性结论。

2.3 工具调用机制

让Agent拥有行动能力的前提是工具调用（Function Calling）。简单来说，工具调用就是让LLM能够"请求"执行一个外部函数，并将函数的返回值纳入后续推理。这在2023年之前还是研究性话题，到2025年已经成为了LLM API的标准能力。

Function Calling的工作原理

工具调用的核心机制分为三步：

第一步：定义工具Schema。 每个工具都用JSON Schema描述其功能、参数名称、类型、必填项和描述。描述的质量直接影响Agent选择工具的准确率——这是我反复强调的一个经验。

tools = [
    {
        "type": "function",
        "function": {
            "name": "search_knowledge_base",
            "description": "搜索知识库，获取与查询最相关的文档片段",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {
                        "type": "string",
                        "description": "搜索关键词，越长越精确"
                    },
                    "top_k": {
                        "type": "integer",
                        "description": "返回结果数量",
                        "default": 3
                    }
                },
                "required": ["query"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "calculate",
            "description": "执行数学计算并返回结果",
            "parameters": {
                "type": "object",
                "properties": {
                    "expression": {
                        "type": "string",
                        "description": "数学表达式，如 '2 + 3 * 4'"
                    }
                },
                "required": ["expression"]
            }
        }
    }
]

第二步：模型推理并选择工具。 当用户提问时，LLM看到的是用户的自然语言描述和工具Schema列表。模型不是直接执行代码，而是生成一个JSON格式的工具调用请求——比如分析用户问题后选择"search_knowledge_base"并填入参数query="上季度销售数据"。

第三步：运行时执行工具并返回结果。 应用程序（而非LLM本身）负责执行工具函数，将结果注入到对话上下文中，然后LLM根据结果继续推理。

工具选择的挑战

工具调用的准确率并非100%。记住第1章中提到的关键数据：当你给一个Agent绑定超过20个工具时，工具选择的准确率会显著下降。以下是实际工程中总结的规律：

工具数量 选择准率（近似） 推荐策略
1-5个 ~98% 直接暴露给单Agent
6-15个 ~92% 分组归类，或使用路由器Agent
16-30个 ~80% 强烈建议拆分为多Agent，共享工具
30个以上 <70% 必须分拆，工具组+专业Agent架构

这个表格说明了一个重要事实：工具调用准确率不是线性的，而是随着工具数量增加而加速下降。背后原因是LLM在多分类问题上的Softmax概率分布——候选类别越多，正确类别的概率越被稀释。这就是为什么我们不能无限给一个Agent绑定工具。

踩坑实录： 一次线上事故让我记忆犹新。我们给一个Agent绑定了25个工具，其中有一个"delete_user_account"的敏感操作。Agent在处理一个完全不相关的"查询用户信息"请求时，由于tools描述之间的语义干扰，错误地调用了delete_user_account。幸好我们在中间层加了二次确认机制（每个"写操作"需要单独确认），否则后果不堪设想。教训：工具的description字段要写清楚使用条件，敏感操作要加防护层。这个经验后来直接推动了我们将该系统改造为多Agent架构——每个Agent只管理5-8个工具，准确率立刻回升到96%以上。

2.4 记忆系统设计

如果Agent的认知架构是"大脑"，那么记忆系统就是"海马体"——它决定了Agent能否在长时间跨度内保持一致的行动能力和上下文连续性。在LLM Agent中，记忆系统通常分为三个层次：

2.4.1 工作记忆

工作记忆就是LLM的上下文窗口（Context Window）——当前对话中所有可见的信息。包括系统Prompt、用户输入、历史对话、工具调用记录及其返回。工作记忆的特点是容量有限但访问极快。即便有了百万token级的上下文窗口，实际工作中你会发现：当上下文达到一定长度后，模型对"早期内容"的注意力会自然衰减——这就是所谓的"lost in the middle"问题。

2.4.2 短期记忆

短期记忆是对工作记忆的压缩和摘要。当对话超过一定轮次或上下文超过某个阈值时，我们需要对历史进行压缩——将非核心内容摘要成一个简短的"记忆块"，替换掉原始的完整历史。常见的实现方式包括：

  • 滚动窗口： 保留最近N轮对话，丢弃最早的内容。简单但粗暴。
  • 摘要压缩： 定期用LLM将历史摘要成一段话，替换原始内容。更优雅，但增加了Token开销。
  • 混合策略： 保留最近N轮完整对话 + 更早历史的摘要。兼顾短期细节和长期上下文。

2.4.3 长期记忆

长期记忆是Agent能够"记住"跨会话信息的核心机制。它的实现基础是向量数据库 + RAG（检索增强生成）。Agent将重要的信息向量化存储，当需要时执行语义检索，找到最相关的记忆片段加入上下文。

class LongTermMemory:
    """基于向量数据库的长期记忆系统"""

    def __init__(self, embed_model, vector_db):
        self.embed_model = embed_model
        self.vector_db = vector_db

    def store(self, content, metadata=None):
        """存储一条记忆"""
        embedding = self.embed_model.embed(content)
        self.vector_db.insert(
            vector=embedding,
            payload={"content": content, "metadata": metadata, "timestamp": now()}
        )

    def recall(self, query, top_k=5):
        """检索与查询最相关的记忆"""
        query_embedding = self.embed_model.embed(query)
        results = self.vector_db.search(query_embedding, top_k=top_k)
        return [r.payload for r in results]

    def forget(self, before_timestamp):
        """遗忘（清理）早期记忆"""
        self.vector_db.delete(filter={"timestamp": {"$lt": before_timestamp}})

这个设计看起来简单，但工程落地时有很多坑：记忆何时存储？是否所有对话都值得记忆？记忆检索的准确率如何保证？遗忘机制怎么设计才不让Agent"失忆"也不让它"过载"？我们会在后续的第18章（生产级MAS）中详细讨论这些记忆工程的细节。

设计原则： 记忆系统不是"把Agent看到的一切都存起来"。好的记忆系统应该像人脑一样——选择性记忆。只存储与目标相关的、异常的事件、以及重要的决策依据。绝大多数日常对话不值得被长期记住。一个过于庞大的记忆库会稀释检索质量，导致Agent"知道太多反而不知道什么重要"。

2.5 单一智能体的能力边界

我们花了整整一章来解剖单个Agent的内部结构——它的感知-规划-行动循环、ReAct推理模式、工具调用机制、记忆系统。你现在应该对Agent的内部运作有了清晰的理解。但是，理解一个Agent的能力，也意味着理解它的局限性。这些局限正是我们转向多智能体系统的根本原因。

上下文窗口的天花板

即便上下文窗口从128K扩展到1M甚至更多，单个Agent的"有效注意力范围"依然是有限的。研究表明，Transformer的注意力模式存在中心偏置——模型倾向于关注上下文的中后部内容，而开头内容容易被遗忘。这意味着无论窗口有多大，Agent在处理超长任务时都会出现"前文遗忘"。

单一推理视角的局限

一个Agent只有一条推理线。当一个问题可以从多个角度分析（比如市场需求+技术可行性+时间约束三重考量），一个Agent只能先后思考，无法同时从多个视角并行评估。这就像一个人做决策和一支团队做决策的区别。

工具数量与准确率的权衡

我们前面已经量化了这个问题：工具越多，选择越不准确。而一个实际的Agent系统往往需要访问数十甚至上百个工具——单一Agent根本无法优雅地管理这样的复杂度。

盲区与幻觉困境

单一Agent的知识全部来自它的训练数据和上下文。如果遇到训练数据之外的信息（私有数据库、实时数据），Agent要么不知道，要么更糟糕——幻觉。它可能会"编造"一个看似合理但实际上不存在的答案。多Agent系统可以通过Agent间的交叉验证和信息互补来大幅降低幻觉率。

关键转折： 单一Agent的能力边界不是"缺陷"，它是系统设计的自然约束。正如人类社会中，没有一个人能精通所有领域，专业分工才是高效的。下一章，我们将正式进入多智能体系统的世界——看看当多个Agent组成团队时，那些在单Agent中看似不可逾越的边界是如何被打破的。你会发现，从单体到多体的转变，不仅仅是数量的增加，更是系统性质的质变。

让我们用一张总结表来回顾本章的认知架构组件，然后踏出从单体走向多体的关键一步：

组件 核心功能 主要挑战
感知-规划-行动循环 持续交互的认知闭环 循环次数限制，早期信息衰减
ReAct模式 推理与行动交织，动态决策 最大步数控制，避免无限循环
工具调用 外部能力接入接口 工具过多时选择准确率下降
工作记忆 当前上下文管理 "Lost in the middle"注意力衰减
短期记忆 历史压缩与摘要 摘要丢失细节，成本控制
长期记忆 跨会话知识持久化 记忆选择策略，检索质量

下一章，我们将从"一个Agent的内部结构"转向"一群Agent的集体智能"。你会惊讶地发现，当多个Agent按照一定的规则组织起来时，它们能展现出单体Agent不具备的涌现能力。下一站：多智能体系统概述。
