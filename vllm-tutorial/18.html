第18章 安全与访问控制

vLLM 默认不做任何鉴权——任何人能访问 8000 端口就能调用你的昂贵 GPU。生产环境中，安全是必须加的最后一公里。

18.1 安全架构概览

安全分层架构:
┌─────────────────────────────────┐
│  外部用户/应用                    │
└───────────────┬─────────────────┘
                │ HTTPS (TLS)
                ▼
┌─────────────────────────────────┐
│  反向代理 (Nginx/Envoy)          │
│  - API 鉴权 (API Key / JWT)     │
│  - 速率限制 (Rate Limiting)      │
│  - IP 白名单/黑名单              │
│  - Prompt Injection 检测         │
└───────────────┬─────────────────┘
                │
                ▼
┌─────────────────────────────────┐
│  vLLM 实例 (内网，不暴露公网)      │
└─────────────────────────────────┘

18.2 API 鉴权

vLLM 支持通过 --api-key 参数启用简单的 token 鉴权：

# vLLM 自带 API Key 鉴权
vllm serve model --api-key "sk-your-secret-key"

# 客户端请求
curl http://localhost:8000/v1/chat/completions \
    -H "Authorization: Bearer sk-your-secret-key" \
    -d '{"model": "...", "messages": [...]}'

更复杂的鉴权（JWT、OAuth2、多租户）建议在反向代理层实现：

# Nginx + JWT 鉴权
location /v1/ {
    auth_jwt "vLLM API";
    auth_jwt_key_file /etc/nginx/jwt-public-key.pem;
    proxy_pass http://vllm_backend;
}

18.3 速率限制（Rate Limiting）

维度限制目的实现方式
每个 API Key 的 QPS防止单个用户打满 GPUNginx limit_req_zone / Envoy rate limit
每个 IP 的并发连接数防止爬虫/恶意调用Nginx limit_conn
请求 token 总量防止超长 prompt 霸占显存--max-model-len 硬限制
每日/每月 token 配额成本控制API Gateway 层计数
# Nginx 速率限制示例
limit_req_zone $binary_remote_addr zone=vllm:10m rate=30r/s;

location /v1/ {
    limit_req zone=vllm burst=10 nodelay;
    proxy_pass http://vllm_backend;
}

18.4 Prompt Injection 防护

LLM 服务的特色安全问题——用户可能通过精心构造的 prompt 绕过限制或获取不该访问的信息：

# 基础防护措施:

# 1. 限制输入长度（防超长 prompt 攻击）
# 应用层: 拒绝超过 8000 token 的 prompt
if len(tokenizer.encode(user_prompt)) > 8000:
    raise HTTPException(400, "Prompt too long")

# 2. System Prompt 隔离
# 用特殊分隔符隔离 user prompt 和 system prompt
# 防止用户通过 prompt 覆盖 system 指令

# 3. 输出过滤
# 对生成的文本进行关键词/正则检查
# 防止泄露敏感信息（如 SSN、信用卡号）

重要：vLLM 本身不做 Prompt Injection 防护。这是一个应用层问题，需要在 API Gateway 或业务代码中处理。不要直接把 vLLM 暴露给不可信用户而无任何过滤。

18.5 网络安全最佳实践

  • vLLM 实例只监听内网 IP：--host 127.0.0.1（单机）或 --host 10.0.0.x（内网），永远不要 --host 0.0.0.0 到公网
  • TLS 终结在反向代理：Nginx/Envoy 处理 HTTPS，vLLM 和代理之间用 HTTP
  • 最小权限原则：vLLM 容器不要以 root 运行，用 USER 1000
  • 网络隔离：vLLM Pod 放在 K8s 的 private subnet，只允许 API Gateway 访问

快速检查清单：上生产前确认——(1) 有没有鉴权层？(2) 公网能直接访问 vLLM 吗？(3) 有没有速率限制？(4) 对用户输入长度有限制吗？四项都是 Yes 才算安全基线。
