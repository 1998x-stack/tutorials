大家好，我是你们的讲师。上一章我们聊了printk和Bochs调试器——这些方法虽然有效，但效率不太高：加printk需要重新编译、重新启动，看输出就像在黑暗中摸索。Bochs的调试器虽然能设断点、看内存，但它的交互方式比较原始，没有源代码级别的单步跟踪。

这一章，我们要升级到一套更强大的武器——GDB + QEMU远程调试组合。你可以像调试普通用户态程序一样，在源代码级别单步跟踪Linux内核的执行。想象一下：你在schedule()函数里设一个断点，当QEMU中的Linux运行到这个函数时，GDB就会停下来，让你观察变量、单步执行、甚至修改内存——然后继续执行。就像你在IDE里调试一个普通的C程序，只不过这个"程序"是整个操作系统。

套用一句夸张但真实的话："用GDB单步跟踪内核，就像在Linus Torvalds的大脑里散步。"

引言——GDB远程调试的原理

GDB远程调试的基本原理很简单：

  • QEMU运行目标系统（在这里是Linux 0.11），它模拟了一个完整的x86机器
  • GDB运行在宿主机上，通过TCP连接到QEMU
  • QEMU暴露一个GDB服务器端口（默认是1234），等待GDB连接
  • GDB发送"读/写内存"、"暂停/继续执行"、"设置断点"等命令，QEMU执行后返回结果

关键点是：内核本身不需要任何修改。QEMU在模拟硬件层面提供了调试接口，内核感知不到自己被调试。这意味着0.11不需要像现代内核那样编译时打开kgdb选项——任何能在QEMU上跑的操作系统都可以用GDB远程调试。

调试的是"机器"而不是"操作系统"：QEMU的GDB服务器工作在硬件模拟层面。你设的断点是硬件断点（利用x86的调试寄存器DR0-DR7），而不是软件断点（int3指令）。所以即使内核完全没有"调试支持"，硬断点也一定有效。

QEMU+GDB调试环境搭建

要在0.11上使用GDB远程调试，你需要做两件事：编译带调试符号的内核，以及用正确的参数启动QEMU。

1. 编译带调试符号的内核

你需要在编译内核时加上-g标志，让GCC生成调试信息。在0.11的顶层Makefile中修改：

# 在 Makefile 中添加调试符号支持
CFLAGS = -Wall -O0 -g -fstrength-reduce -fomit-frame-pointer \
         -fcombine-regs -mstring-insns \
         -nostdinc -Iinclude

# -O0:  关闭优化，让变量值和代码行号一一对应
# -g:   生成完整的DWARF调试信息

注意这里把-O改成了-O0和-g。在调试模式下，我们宁愿内核运行慢一点，也要保证调试信息的准确性。优化后的代码（-O2或-O3）会做内联、重排指令、消除临时变量，导致你在GDB中看到的"当前行"与实际执行的指令对不上。

很多0.11实验环境都提供了一个make debug的快捷目标，它会自动使用调试选项重新编译内核：

$ make debug    # 使用-O0 -g重新编译
$ ls -l system.img   # 确认调试符号包含在内核中

2. 启动QEMU

QEMU的-s和-S参数是调试的关键：

$ qemu-system-i386 -m 16 -boot a -fda Image \
    -hda hdc-0.11.img \
    -s -S

# -s:   shorthand for "-gdb tcp::1234"，在端口1234开放GDB服务器
# -S:   在启动时暂停CPU，等待GDB连接

加了-S后，QEMU窗口不会立即启动内核——它会停在第一条指令之前（即BIOS的第一条指令），等待你从GDB发出"继续执行"的命令。

3. 连接GDB

在另一个终端中启动GDB：

$ gdb system.img   # 加载内核的调试符号
(gdb) target remote :1234   # 连接到QEMU的GDB服务器
Remote debugging using :1234
0x0000fff0 in ?? ()
(gdb)

GDB连接成功后，你会看到CPU当前停在0x0000fff0——这是x86 CPU复位后的第一条指令地址（BIOS入口）。从这里开始，你就可以设置断点、单步跟踪整个启动流程了。

权限问题：如果你在Linux上用源码编译的GDB，连接本地端口一般没有问题。如果遇到"Connection refused"，检查QEMU是否真的在运行、端口1234是否被占用，或者试试用qemu-system-i386 -gdb tcp::1234 -S替代-s。

关键断点设置——战略要点

连接上GDB后，你可以设置多个"战略要点"来观察内核启动和执行的关键阶段。以下是我推荐的几个断点位置：

断点位置对应源文件观察目标
0x7C00boot/bootsect.sBIOS将引导扇区加载到0x7C00后的第一条指令
start_kernelinit/main.cC语言入口，内核初始化的起点
schedulekernel/sched.c进程调度函数的入口
system_callkernel/system_call.s所有系统调用的统一入口
do_forkkernel/fork.c进程创建的完整流程
sys_writefs/read_write.c文件写入系统调用的实现
do_signalkernel/signal.c信号处理的入口
timer_interruptkernel/system_call.s时钟中断处理函数

在GDB中设置这些断点：

(gdb) b *0x7C00          # 在物理地址设断点（硬件断点）
(gdb) b start_kernel     # 使用符号名设断点
(gdb) b schedule
(gdb) b system_call
(gdb) b do_fork
(gdb) b sys_write

设好断点后，发出c（continue）命令让QEMU开始执行：

(gdb) c
Continuing.

Breakpoint 1, 0x7c00 in bootsect ()
(gdb) info registers
eax            0xaa55     0xaa55      ← 引导扇区的魔数
ecx            0x0        0x0
edx            0x80       0x80       ← 启动盘号（0x80=第一块硬盘）
ebx            0x0        0x0
esp            0x6ef4     0x6ef4
eip            0x7c00     0x7c00
cs             0xf000     0xf000     ← 实模式下的段值
ss             0x0        0x0
eflags         0x2        [ ]

看到eax = 0xaa55了吗？这是引导扇区的魔数，BIOS用它来验证引导扇区的有效性。在0x7C00这个断点上，你可以观察实模式下的寄存器状态、以及引导扇区刚刚被加载时的"原始面貌"。

跟踪系统调用——以write()为例

系统调用是用户态进入内核态的唯一途径，也是理解内核工作原理的最佳观测点。让我们以write()系统调用为例，跟踪一次完整的系统调用流程。

首先，让我们在关键点设好断点：

(gdb) b system_call        # 系统调用统一入口
(gdb) b sys_write          # write系统调用的C函数
(gdb) b tty_write          # tty设备写入函数

然后继续执行，直到某个用户程序调用write()（比如shell要输出一个提示符）：

(gdb) c
Continuing.

Breakpoint 2, system_call ()
(gdb) info registers eax ecx edx ebx
eax            0x4        0x4          ← 系统调用号（4 = __NR_write）
ebx            0x1        0x1          ← 第一个参数（fd = 1 = stdout）
ecx            0x...                  ← 第二个参数（缓冲区地址）
edx            0x...                  ← 第三个参数（缓冲区长度）

注意eax = 4，这符合我们在第9章学到的事实：Linux 0.11中__NR_write = 4。接下来，单步进入sys_write：

(gdb) si                    # 单步执行一条指令
0x0000xxxx in sys_write ()
(gdb) si                    # 继续单步...
0x0000xxxx in sys_write ()
(gdb) l                     # 列出当前源代码
(gdb) p fd                  # 打印fd变量的值
$1 = 1                      # 文件描述符1 = stdout
(gdb) p count               # 打印要写入的字节数
$2 = 23

继续单步，你会看到sys_write调用了tty_write。在tty_write中，内核把要输出的字符放入写队列，然后调用con_write将字符写入VGA显存（0xB8000）：

(gdb) b con_write          # 在console写入函数设断点
(gdb) c
Continuing.

Breakpoint 5, con_write ()
(gdb) x /20xw 0xB8000      # 查看VGA显存的内容
0xb8000:    0x7207    0x7200    0x7200    0x7200
# 前几个字符的ASCII值已经写入显存...

执行c继续后，你会在QEMU窗口中看到shell提示符出现了。整个过程你亲眼目睹了：一个字符从用户程序的write()开始，经过系统调用入口 → sys_write → tty_write → con_write → VGA显存——完成了从用户态到内核态、再到硬件的完整旅程。

哲学一刻：刚才你看到的这短短几行代码执行路径，正是操作系统"一切皆文件"和"用户态/内核态隔离"这两个核心设计思想的具象体现。抽象概念在单步跟踪中变得无比真实。

观察进程切换——追踪调度器

进程切换是操作系统最核心的机制，也是最"神奇"的部分——毕竟一条指令执行完，CPU还在运行，但当前"进程"已经变了。用GDB可以把这个"魔法"拆解成一个个可以观察的步骤。

在进程调度相关的函数上设断点：

(gdb) b schedule           # 调度函数入口
(gdb) b switch_to          # 实际切换的任务切换宏
(gdb) b timer_interrupt    # 时钟中断（schedule通常由时钟中断触发）

继续执行，当内核初始化完成后，schedule()会被调用：

(gdb) c
Continuing.

Breakpoint 3, schedule () at kernel/sched.c:127
(gdb) p current->pid       # 当前正在运行的进程
$3 = 1                     # 进程1（init进程）
(gdb) p current->counter   # 剩余时间片
$4 = 3                     # 还剩3个tick
(gdb) n                    # next line...
(gdb) p next               # 调度器选中的下一个进程
$5 = 0                     # task[0] = idle进程

然后执行到switch_to：

(gdb) c
Continuing.

Breakpoint 4, switch_to ()
(gdb) info registers       # 观察切换前的寄存器状态
(gdb) n                    # 执行switch_to（注意看current！）
(gdb) p current->pid       # 切换后current变了！
$6 = 0                     # 现在是PID=0的idle进程

这里有一个GDB的小窍门：你可以定义一个hook-stop，让GDB在每次停下来时自动打印关键信息：

(gdb) define hook-stop
> print "--- Stop ---"
> print current->pid
> print current->state
> print current->counter
> info registers eip esp
> end

(gdb) c
--- Stop ---
$7 = 0
$8 = 0
$9 = 0
eip            0x1234
esp            0x....

有了hook-stop，你在单步跟踪时就不用手动每次敲命令了——所有关键信息自动打印出来。这就是"GDB脚本化调试"的雏形。

GDB脚本自动化：把define hook-stop和常用断点放在一个.gdbinit文件中，每次启动GDB时自动加载。还可以用gdb -x script.gdb来执行更复杂的自动化调试脚本，比如"在system_call处暂停10次后，打印一次寄存器，然后继续"——这种批量调试对复现竞争条件的Bug特别有用。

内存检查技巧

GDB的x（examine）命令是检查内存最强大的工具。它接受三个参数：数量(N)、格式(u)、大小(f)，语法为x /Nuf 地址。

以下是我最常用的几个检查技巧：

# 查看栈内容（栈顶在esp指向的位置）
(gdb) x /10xw $esp         # 显示从esp开始的10个字（word，4字节）

# 反汇编当前指令附近的代码
(gdb) x /10i $eip          # 从eip开始反汇编10条指令

# 查看task_struct结构体的特定字段
(gdb) p *(struct task_struct *)current
(gdb) p current->pid
(gdb) p current->sigaction

# 查看页目录和页表
(gdb) p swapper_pg_dir     # 0.11中页目录的基地址

# 查看系统调用表
(gdb) p sys_call_table     # 打印系统调用表

# 以字符串形式打印内存
(gdb) x /s 0xB8000         # 查看VGA显存中的字符

# 查看某个特定task
(gdb) p *task[1]           # 打印task[1]（PID=1的进程控制块）

其中p *(struct task_struct *)current会输出task_struct的所有字段——让你直观地看到进程的完整控制块布局。这比任何文档都有说服力。

GDB命令含义调试场景
x/10xw 0x...以16进制显示10个字查看栈、堆中的原始数据
x/10i 0x...反汇编10条指令检查编译后代码是否正确
x/s 0x...以字符串形式显示查看内核字符串缓冲区
info registers显示所有寄存器断点命中时的CPU状态快照
bt显示调用栈回溯查看函数调用链
p 结构体指针打印结构体所有字段检查task_struct、inode等内核对象
list / l显示当前源代码查看断点附近的代码上下文

踩坑故事——GDB与内核的地址空间不一致

GDB远程调试虽然强大，但不是万能的。有一个陷阱让我浪费了整整两天时间。

事情是这样的：我在内核的某个函数里发现了一个Bug，需要在一个特定的物理地址上设置断点。当时我直接输入：

(gdb) b *0x10000          # 在物理地址0x10000设断点

然后继续执行。内核启动了，引导扇区执行了，setup.s执行了，head.s执行了——但断点从未被触发。我检查了反汇编，确认那个地址是有代码的。为什么断点没命中？

排查了半天，终于明白了原因：GDB使用的是线性地址（线性地址），而我在0x10000设置的断点是物理地址。当时内核已经开启了分页机制，CPU访问0xC0000000+offset的线性地址时，通过页表映射到物理地址0x00000000+offset。GDB发送给QEMU的断点请求使用的是线性地址，而物理地址0x10000对应的线性地址是0xC0010000！

正确的做法是：

# 错误的示范
(gdb) b *0x10000          # 物理地址，GDB当作线性地址处理，断点永远不会命中

# 正确的示范
(gdb) b *0xC0010000       # 线性地址，因为内核映射了0xC0000000 + 物理偏移
# 或者直接使用函数名
(gdb) b start_kernel      # 使用符号名，GDB会自动解析为正确的线性地址

这个问题在Bochs调试器中不会出现——因为Bochs的pb命令明确指定了物理地址。但GDB的b命令始终使用"当前地址空间"。在分页开启前，你的b *0x7C00能工作，因为此时物理地址==线性地址。分页开启后，二者不再相同。

地址空间规则：在GDB中设置符号断点（如b schedule）永远是最安全的——GDB会根据符号表自动解析为正确的地址。如果需要手动指定地址，记住0.11内核在分页开启后运行在高地址（0xC0000000+）的线性空间。不确定地址的时候，先用info proc mappings或info files查看内存布局。

这个故事也告诉我们一个重要的道理：理解内核地址空间布局是调试的前提。如果你不清楚当前处于实模式还是保护模式、分页是否开启、内核映射到了什么地址——你会被GDB的输出搞得晕头转向。
