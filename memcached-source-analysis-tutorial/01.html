第1章 为什么要读Memcached源码

大家好，我是你们的讲师。在开始第一堂课之前，我想问大家一个问题：你每天都在用缓存，但你真的了解缓存底层在做什么吗？

如果你用过 Memcached，你大概知道它是一个"内存里放 key-value 对的东西"——get 一个 key，set 一个 value，简单得像一个巨型 HashMap。但当你深入源码之后才会发现，这台看似简单的"内存机器"背后，藏着 slab allocator 对内存碎片的精巧博弈、多线程模型下无锁设计的极致追求、以及一套在 20 年生产环境中打磨出来的容错哲学。本课程就将带你一行一行地读懂这些设计决策。

缓存系统在现代架构中的角色

今天的互联网应用，从社交网络到电商平台，从新闻门户到实时推荐系统，几乎没有一个大型系统能离开缓存。为什么？因为数据库太慢了。

现代 SSD 的随机读延迟大约在 50-100 微秒级别，而内存访问延迟是纳秒级别——两者相差三个数量级。当你的应用需要每秒处理数百万次查询时，让每一条请求都穿透到数据库是不现实的。缓存层的作用就是用廉价的 DRAM 来吸收热点流量，保护后端的数据库和微服务。

在一个典型的 web 架构中，缓存通常出现在两个位置：本地缓存（进程内，如 LRU Cache）和分布式缓存（独立服务，如 Memcached / Redis）。本地缓存快但容量小、数据不一致；分布式缓存慢一些但容量大、可共享。Memcached 属于后者，但它通过极简的设计做到了接近本地缓存的性能——这也是它源码中最值得学习的地方。

核心观点：缓存不是银弹，但它是现代分布式系统的基石。理解 Memcached 的源码，本质上是理解"如何用最少的代码和最高的效率管理内存"这门手艺。

Memcached vs Redis 选型思考

说到分布式缓存，一定会遇到这个问题：Memcached 还是 Redis？这两个项目经常被放在一起比较，但它们的定位其实有本质区别。

维度 Memcached Redis
定位 纯 KV 缓存 数据结构服务器 + 缓存
数据结构 仅 String / Byte 数组 String, List, Set, ZSet, Hash, Stream 等
持久化 不支持（纯内存） RDB / AOF 持久化
多线程 Master-Worker 多线程 主版本 6+ 后引入多线程 I/O
内存管理 Slab Allocator（预分配，无碎片） jemalloc / tcmalloc + 自己封装
淘汰策略 LRU（三级队列） 6 种策略，含 LFU、LRU、TTL 等
代码量 ~2 万行 C ~20 万行 C
典型场景 数据库查询缓存、Session 存储 排行榜、消息队列、分布式锁

从这张对比表可以清晰看到：Memcached 选择了一条极简的道路——它不做持久化、不支持复杂数据结构、不提供事务。但正是这种"少即是多"的设计哲学，使得它的核心代码极度精简，几乎每一行都有明确的用途。当你打开它的源码目录，你不会像看 Redis 那样面对几十万行代码感到无从下手——你会看到一份可以一口气读完的工业级 C 代码范本。

选型建议：如果你只需要"缓存数据库查询结果"或"存放 Session"，Memcached 在内存效率上往往优于 Redis——因为它没有解析复杂数据结构的开销，且 slab allocator 不会产生内存碎片。但如果你需要 List、Set 等数据结构的运算能力，或者需要持久化保证，Redis 是更合适的选择。

为什么 Memcached 源码值得读

我常说一句话："读 Redis 源码是读一个产品，读 Memcached 源码是读一本教科书。"

Memcached 的代码量大约在 2 万行左右（核心逻辑仅 1 万行出头），这意味着一个有 C 语言基础的开发者，如果每天读 2 小时，一周之内就能通读核心代码。相比之下，Redis 的代码量超过 20 万行，Netty 的代码量也超过 10 万行——它们当然值得读，但学习曲线陡峭得多。

具体来说，Memcached 源码有以下几个独特的学习价值：

  • 工业级 C 代码的审美范本——代码风格一致、命名规范、注释到位。Brad Fitzpatrick 和 Dormando 等核心维护者写出了教科书级别的 C 代码，看这样的代码本身就是一种享受。
  • 极致的性能追求——每一个分支、每一处内存分配都经过精心设计。你会看到为了减少一次系统调用而做的努力，也会看到为了 cache line 对齐而在结构体布局上做的文章。
  • 完整的内存管理系统——Slab Allocator、LRU 淘汰、渐进式 rehash，这三个主题覆盖了内存管理面试中 80% 的高频考点。而 Memcached 的实现是所有开源软件中理解门槛最低的。
  • 经典的多线程网络模型——Master-Worker 架构、libevent 事件驱动、连接状态机，这套模型在今天仍然是最主流的 C/C++ 网络编程范式。
  • 协议设计实战——从 ASCII 文本协议到二进制协议再到 Meta Protocol，你能看到协议演进的完整脉络。
我的亲身经历：2018 年我在一家广告技术公司排查线上 Memcached 延迟毛刺问题。凌晨 2 点，p99 延迟从 1ms 跳到了 50ms，而 QPS 并没有明显变化。在花了两天分析火焰图、翻源码之后，发现问题出在 slab rebalance 触发了全局锁的竞争。那是我第一次切身感受到：不看源码，你永远不知道你的缓存中间件在背后做了什么。从那以后，我养成了一个习惯——生产环境用的每一个中间件，至少要把核心模块的源码通读一遍。

Memcached 的历史与版本

了解一个项目的历史，有助于理解它的设计决策。Memcached 的诞生故事在业界广为流传：

  • 2003 年：Brad Fitzpatrick 在 LiveJournal 工作期间，为了减轻数据库压力，用 Perl 写了第一版 Memcached。当时的 LiveJournal 是最大的博客平台之一，面对急剧增长的用户量，MySQL 已经不堪重负。
  • 2004 年：Anand Rajagopalan 和 Eric Berg 用 C 重写了 Memcached，性能得到大幅提升。这也奠定了 Memcached 作为高性能缓存系统的基石。
  • 2009 年起：项目维护移交给社区，Dormando（现任主要维护者）开始主导开发，引入了二进制协议、多线程支持等关键特性。
  • 至今：Memcached 仍然在 Facebook（全球最大的 Memcached 部署之一，管理着数万个实例）、Wikipedia（页面缓存）、Twitter（时间线缓存）等顶级互联网公司大规模使用。

当前最新版本是 1.6.41（2025 年发布），采用 BSD 许可证。20 多年来，Memcached 经历了从单线程到多线程、从纯文本协议到二进制+元协议、从纯内存到支持外部存储的演进，但其核心设计理念始终没有改变：简单、快速、可靠。

一个冷知识："Memcached" 名字来源于 "Memory Cache Daemon" 的合并缩写。有趣的是，"memcached" 的正确发音是 /mɛmˈkæʃt/（mem-casht），不是 "mem-cache-dee"。

下一章，我们会一起搭建 Memcached 的开发环境、获取源码，然后打开第一行代码。在动手之前，不妨先思考一个问题：你最想在 Memcached 源码中寻找什么答案？带着这个目标开始阅读，你的收获会大得多。
