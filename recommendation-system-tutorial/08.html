第8章 从LR到FM：特征交叉的进化

上一章我们深入探讨了特征工程的种种技法——从用户画像到物品画像，从统计特征到时序特征。但你可能已经注意到了一个问题：这些特征都是我们手工构造的。我们手动把"用户年龄=25"和"物品类别=电子产品"拼在一起，形成一个交叉特征。这种手工活，能不能让模型自己学会？

这就是本章要讲的核心问题——从逻辑回归到因子分解机，特征交叉从"人肉工程"走向了"模型自动化"的进化之路。

1. 逻辑回归：推荐系统的第一把锤子

如果你穿越回到2012年，去一家互联网公司的推荐团队面试，面试官问"你们用的是什么模型"，十有八九的答案是——LR，逻辑回归（Logistic Regression）。

在深度学习横扫一切之前，LR是推荐系统和广告CTR预估领域的绝对王者。它的统治力来自三个字：简单、快、可解释。

逻辑回归的数学形式非常简洁：

p(y=1|x) = σ(w^T x + b)
         = 1 / (1 + exp(-(w^T x + b)))

其中 σ 是 sigmoid 函数，它将任意实数映射到 (0,1) 区间，输出正好可以解释为点击概率。训练目标是最小化交叉熵损失（Cross-Entropy Loss）：

L = -y log(p) - (1-y) log(1-p)

在工业场景中，输入特征 x 经过 one-hot 编码或特征哈希（Feature Hashing）后，维度可以轻松达到千万甚至亿级。好在LR的优化算法非常成熟——FTRL（Follow The Regularized Leader）等在线学习算法可以高效处理这种稀疏大规模特征。

但LR的最大问题也恰恰在于此：它只能做线性组合，无法自动学习特征之间的交互关系。

什么意思呢？假设我们有两条特征："用户是大学生"和"当前时间是暑假"。单独看每一个特征，它们对"下载考研App"这个行为可能只有微弱影响。但两个特征同时出现——大学生+暑假——用户下载考研App的概率可能飙升。LR学不到这种交叉效应，除非你手工会把"大学生_AND_暑假"这个组合特征提前构造好并喂给它。

于是推荐系统的第一代工程师们，过上了一种悲壮的生活：白天疯狂做特征工程，晚上加班分析特征重要性。交叉特征、高阶组合、业务规则的枚举——所有模型学不会的东西，都靠人工来补。

真实故事：三年LR，一朝FM

2015年，我了解到一家中型电商公司的推荐团队，当时他们的CTR模型是LR，特征维度超过2亿，特征工程代码超过3万行。团队每周都要花大量时间做特征上线和AB测试，交叉特征的数量级增长几乎不可控。后来他们切换到FM，两周就复现了LR + 几万条手工特征的AUC，还把特征工程的工作量减少了大约70%。团队负责人在分享时感叹："以前我们以为推荐系统的核心竞争力是特征工程，后来才发现，我们只是在为模型的缺陷打工。"

2. FM：矩阵分解与LR的优雅结合

2010年，Steffen Rendle 提出了因子分解机（Factorization Machines, FM），这篇论文后来成为推荐系统领域引用最高的论文之一。FM的核心洞见极其优雅：与其为每一对特征组合学习一个独立的权重 w_ij，不如为每一个特征学习一个隐向量 v_i，然后用两个隐向量的点积来表示交叉权重。

数学上，FM模型的二阶部分是这样的：

ŷ(x) = w_0 + Σ w_i x_i + Σ Σ (v_i^T v_j) x_i x_j
          i=1..n    i=1..n j=i+1..n

注意最后一项：w_ij 被替换成了 v_i^T v_j。这里的 v_i 是特征 i 的 k 维隐向量（embedding）。

这个替换带来了一个巨大的优势：泛化能力。在LR中，如果特征对 (i,j) 在训练数据中从未同时出现过，那 w_ij 就是0，模型完全无法预测这个交叉。但在FM中，即使 (i,j) 没有共现过，v_i 可以从特征 i 与其它特征的交互中学习，v_j 可以从特征 j 与其它特征的交互中学习，模型仍然可以推断 (i,j) 的交叉效应。

FM还有一个工程上的神来之笔：它的计算复杂度可以优化到 O(kn)，而不是直观的 O(kn²)。关键公式变换：

Σ Σ (v_i^T v_j) x_i x_j = ½ [ (Σ v_i x_i)^2 - Σ (v_i x_i)^2 ]
 i<j                     i        i

这个变换把两两交叉的求和化简成了"先求和再平方"与"先平方再求和"的差，极大降低了计算量。在工程实现中，这意味着FM的前向推断几乎和LR一样快。

工业小贴士：FM的隐向量维度 k 是非常关键的超参数。在工业推荐场景中，k 通常在 8 到 64 之间。k 越大，模型的表达能力越强，但也越容易过拟合。经验法则是：特征总量越大、每个特征的频次越低，k 就应该越小。

FM有多成功？在KDD Cup的多个竞赛中，FM及其变体持续霸榜。尤其是2012年的KDD Cup——腾讯广告CTR预估大赛——FM几乎成了所有参赛队伍的标配组件。Rendle后来在论文中回忆："提交FM论文时，审稿人问得最多的问题是——这真的比SVM好吗？"后来的结果证明，不是比SVM好，而是好得多。

3. FFM：引入Field感知

FM虽然强大，但它有一个隐含的假设：所有特征交叉使用相同的隐向量。换句话说，特征"性别=男"在与特征"物品类别=电子产品"交叉时使用的 v，与它和"时间段=晚上"交叉时使用的是同一个向量。

这个假设合理吗？不太合理。性别特征在与不同特征交叉时，应该有不同的行为——与物品类别的交叉和与时间段的交叉，本质上是不同的交互模式。

Field-aware FM（FFM） 正是为了解决这个问题而提出的。FFM为每个特征学习多个隐向量——每个field对应一个。当特征 i 与特征 j 交叉时，使用专门为 field(j) 准备的向量 v_{i,f(j)}。

数学上，FFM的二阶交叉是：

Σ Σ (v_{i,f(j)}^T v_{j,f(i)}) x_i x_j
 i=1..n j=i+1..n

这里 f(j) 表示特征 j 所属的 field。假如有 m 个 field，每个特征就有 m-1 个隐向量（不与自己field中的特征交叉？实际上是的）。这显著增加了参数量。

FFM的代价也很明显：参数从 FM 的 O(nk) 增加到了 O(nmk)。对于一个有50个field、千万级特征的系统，FFM的参数量可能是FM的几十倍。这意味着：

  • 训练更慢：参数更新的计算量更大
  • 更容易过拟合：参数多了，对数据量的需求更大
  • 部署更难：模型体积大，推理内存需求高

踩坑提醒：FFM不是银弹。我曾经见过一个团队在数据量只有100万样本时强行上线FFM，结果AUC比FM还低了1.5个点——过拟合导致泛化能力下降。FFM更适合大厂级别的数据量（千万到亿级样本）。如果你的数据量在百万级，先试试FM，大概率就够了。

下面的表格总结了LR、FM、FFM三者的关键差异：

维度 LR FM FFM
参数数量 O(n) O(nk) O(nmk)
特征交叉 无（需手工构造） 二阶自动交叉 二阶 field-aware 交叉
计算复杂度 O(n) O(kn) O(kn² · m)
泛化能力 弱（未见过的交叉为0） 强（隐向量可泛化） 更强（field级别细分）
适合数据规模 任意规模 百万~千万级 千万~亿级
可解释性 ★★★★★ ★★★★ ★★★
典型应用时期 ~2014年 2015~2017年 2016~2018年

什么时候LR仍然是正确的选择？当你面对的是风控、合规等对可解释性有严格要求的场景时，LR依然是首选。比如，用户被误判为欺诈，产品团队需要明确回答"为什么"——LR可以给出每个特征的权重贡献，FM只能给出隐向量，解释起来困难得多。另外，如果你的数据非常少（几千到几万样本），LR的简单反而是优势。

4. 从数学公式到工业代码

理论说完了，我们来动手实现一个FM的前向计算。下面是用NumPy实现的FM核心逻辑——没有深度学习框架，只有纯粹的矩阵运算，这对于理解FM的本质非常有帮助。

import numpy as np

class FM:
    def __init__(self, n_features, k=8):
        """n_features: 特征总数（含one-hot展开）
           k: 隐向量维度
        """
        self.w_0 = 0.0  # 全局偏置
        self.w = np.zeros(n_features)  # 一阶权重
        self.V = np.random.normal(0, 0.01, (n_features, k))  # 隐向量矩阵

    def forward(self, x):
        """x: 稀疏特征向量 (n_features,) ——大多数元素为0
           返回: 预测得分 p(y=1|x)
        """
        # 线性部分
        linear = self.w_0 + np.dot(self.w, x)

        # 二阶交叉部分 (O(kn) trick)
        # 计算 Σ v_i * x_i
        sum_v = np.dot(self.V.T, x)       # shape: (k,)
        # 计算 Σ (v_i * x_i)^2
        sum_squared_v = np.dot(self.V.T ** 2, x)  # shape: (k,)

        # 公式: ½ * sum_k ( (sum_v_k)^2 - sum_squared_v_k )
        interaction = 0.5 * np.sum(sum_v ** 2 - sum_squared_v)

        # sigmoid 转换
        z = linear + interaction
        return 1.0 / (1.0 + np.exp(-z))

这个实现的精华在于 interaction 的计算——用 O(kn) 的时间完成了所有二阶交叉特征的计算。如果你在面试中被问到"FM的优化技巧"，这就是标准答案。

训练的梯度推导稍微复杂一些，但核心思想是：对 w_0、w_i 和 V_i 分别求偏导，然后用SGD更新。注意由于输入x是高度稀疏的，每次只需要更新非零特征对应的参数。

说到FM的工程实现，还有一个不得不提的细节：特征哈希（Feature Hashing）。在工业场景中，特征的数量可能达到数亿，直接用one-hot编码存储权重矩阵是不可能的。FM实现通常配合特征哈希使用：对原始特征ID做哈希映射到固定长度的桶（比如2^24=1600万），然后只维护这个尺寸的embedding矩阵。哈希冲突导致的精度损失通常可以忽略。

好了，从LR到FM再到FFM，我们看到了推荐模型如何一步步从"人肉做特征交叉"进化到"模型自动学习交叉"。LR是奠基者，FM是突破者，FFM是在特定方向上的优化。但故事还远没有结束——FM的二阶交叉到顶了吗？当模型开始引入深度学习，又会发生什么？下一章，我们将进入Wide&Deep和DeepFM的世界，见证推荐模型从"浅层模型"到"深度模型"的关键跨越。
