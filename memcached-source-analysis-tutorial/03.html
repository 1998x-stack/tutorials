第3章 源码全景：目录结构与模块划分

大家好，我是你们的讲师。上一章我们把环境搭好了，源码也编译通过了。现在你打开 memcached/ 目录，可能会看到一堆 .c 和 .h 文件——别慌，这一章就是来给你一张"源码地图"的。

如果把整个 Memcached 源码比作一座城市，那 memcached.c 就是市中心的主干道，slabs.c 是工业区，items.c 是物流仓库，assoc.c 是交通枢纽，proto_text.c 和 proto_bin.c 是城市的大门。读代码之前先知道每条街通向哪里，阅读效率会成倍提升。好，我们直接看目录。

顶层目录结构

Memcached 的源码结构非常扁平——几乎所有源码文件都位于根目录下，没有复杂的子目录嵌套。这是它代码量小、易于阅读的一个体现。下面我把每个文件和目录的功能用一句话说清楚：

文件/目录 一句话职责 优先级
memcached.c 主流程：main 函数、事件循环、连接管理、驱动一切的核心 必读
memcached.h 全局头文件：结构体定义、宏、全局变量声明 必读
slabs.c / slabs.h Slab Allocator：内存预分配、分桶、按需增长 必读
items.c / items.h Item 生命周期：分配、链接、删除、LRU 链表操作 必读
assoc.c / assoc.h 哈希表：key 查找、插入、渐进式 rehash 必读
proto_text.c ASCII 文本协议解析（get/set/delete 等命令） 必读
proto_bin.c 二进制协议解析（请求/响应编解码） 推荐
thread.c / thread.h 多线程模型：master-worker、连接分发 必读
logger.c / logger.h 日志系统：环形缓冲区、异步写日志 推荐
crawler.c LRU 爬虫：后台扫描陈旧 Item、主动淘汰 选读
extstore.c / extstore.h 外部存储：将冷数据换出到 SSD/NVMe 选读
proxy.c / proxy.h 内置 Proxy：Lua 路由、请求分发 选读
authfile.c SASL 认证文件管理 选读
bipbuffer.c Bip Buffer：一种高效环形缓冲区实现 选读
hash.c 哈希函数封装（默认 CRC-32，可替换） 选读
jenkins_hash.c Bob Jenkins 哈希算法实现 选读
stats.c / stats.h 统计收集与格式化输出 推荐
util.c / util.h 通用工具函数：字符串处理、安全函数 选读
t/ 测试套件（Perl 编写） 选读
doc/ 协议文档、变更日志 参考
阅读策略：这张表里标"必读"的文件是这个课程的核心覆盖范围，总共大约 1.2 万行代码。如果你每周读 3 个文件，一个月就能全部过完。千万别想着一次性通读所有文件——那会让人很快放弃。跟着本课程的章节顺序走，每个模块都会在最合适的时机出现。

核心模块调用关系

理解模块间的调用关系是读懂源码的关键。下面我用文字+箭头的方式描述核心模块之间的依赖关系：

                       ┌───────────┐
                       │   main()  │  (memcached.c)
                       └─────┬─────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
       ┌───────────┐  ┌───────────┐  ┌───────────┐
       │  thread   │  │  assoc    │  │   slabs   │
       │ (多线程)   │◄─┤ (哈希表)   │◄─┤ (内存管理) │
       └─────┬─────┘  └───────────┘  └─────┬─────┘
             │                              │
             │                    ┌─────────┼─────────┐
             ▼                    ▼         ▼         ▼
       ┌───────────┐       ┌─────────┐ ┌────────┐ ┌────────┐
       │ libevent  │       │  items  │ │crawler │ │extstore│
       │ (事件驱动)  │       │ (数据)  │ │(淘汰)  │ │(外部存储)│
       └───────────┘       └────┬────┘ └────────┘ └────────┘
                                │
                                ▼
                       ┌──────────────────┐
                       │  proto_text/bin   │
                       │  (协议解析)        │
                       └──────────────────┘

从这张图可以看到：slabs 和 assoc 是 Memcached 的两个基石，一个管"放哪儿"（内存分配），一个管"在哪儿"（哈希索引）。items 是连接这两者的桥梁——它从 slabs 拿内存，在 assoc 注册索引，自身维护 LRU 链表。所有的协议处理（proto_text / proto_bin）最终都转化为对 items + assoc 的操作。

代码量分布

用 cloc（Count Lines of Code）工具统计一下各文件的代码量，可以更直观地感受项目规模：

# 安装 cloc
sudo apt install cloc

# 统计源码目录
cloc memcached.c memcached.h slabs.c items.c assoc.c \
     proto_text.c proto_bin.c thread.c logger.c stats.c
文件 代码行数（约） 占比
memcached.c ~6,000 30%
proto_text.c ~2,200 11%
proto_bin.c ~1,800 9%
slabs.c ~1,500 7.5%
items.c ~1,200 6%
memcached.h ~1,000 5%
thread.c ~800 4%
assoc.c ~500 2.5%
logger.c ~450 2.2%
stats.c ~400 2%
其他 ~4,000 20%
总计 ~20,000 100%

看到这个数据你就明白了：Memcached 整个项目大约 2 万行代码，其中核心的 1 万行集中在 5-6 个文件中。这意味着每个文件的代码量控制得非常好——assoc.c 只有 500 行，一个下午就能读完。相比之下，Redis 的 server.c 就有 1.5 万行，Netty 的 NioEventLoop.java 也有 2000+ 行。Memcached 的这种"小而美"的特质正是它适合做源码教材的原因。

技巧：用 cloc --by-file memcached/ 可以一次性统计所有文件的代码行数。建议你在自己的源码目录跑一下，看看统计结果是否与上表一致——不同版本的代码量会略有差异。

辅助模块速览

除了核心模块，几个辅助模块也值得一提：

crawler.c — LRU 爬虫

后台线程定期扫描 LRU 链表，主动淘汰过期 Item。这是 Memcached 实现"惰性删除 + 主动淘汰"双重策略的关键模块。在 1.5 版本之后引入了 LRU 爬虫（LRU crawler），能够更精确地控制内存回收。

bipbuffer.c — Bip Buffer

Bip Buffer（Bipartite Buffer）是一种高效的环形缓冲区实现，用于 extstore 模块中管理磁盘 I/O 的读写缓冲区。它的特点是只使用两个指针就实现了无锁的读写，设计非常精巧，只有不到 200 行代码。

util.c — 安全工具函数

提供了一些 C 标准库的安全替代函数（如 safe_strtol、safe_strtoul），以及 URL 编码/解码、整数溢出检查等工具性函数。代码不长但很值得一看，可以学习怎么写鲁棒的 C 代码。

本课程的阅读路径

基于这张模块地图，我建议的阅读路线如下：

第一部分（第4-6章）：启动与配置
    memcached.c → memcached.h → logger.c → stats.c

第二部分（第7-14章）：内存管理核心
    slabs.c → items.c → assoc.c → crawler.c

第三部分（第15-18章）：网络与并发
    thread.c → memcached.c（事件循环部分）

第四部分（第19-23章）：协议与扩展
    proto_text.c → proto_bin.c → extstore.c → proxy.c

第五部分（第24-28章）：工程实践
    t/ → util.c → 综合对比

每一部分的内部文件之间有强烈的依赖关系——比如你不先读 slabs.c 就没法理解 items.c 中的内存分配逻辑。所以我强烈建议按照这个顺序来读，跳着读会让你频繁遇到"这个函数是干啥的"的困惑。

踩坑实录：我早期读源码有一个坏习惯——喜欢从最"有意思"的模块开始读。有次我先读了 extstore.c，结果发现里面调了一堆 bipbuffer 的函数，我又回头去读 bipbuffer.c，然后发现 extstore 还依赖 slabs 的存储分配，又去读 slabs.c……最后形成了"读源码三周，在五个文件之间反复横跳"的局面。不要重蹈我的覆辙，按顺序来才是最高效的。

现在你已经有了完整的源码地图——知道了每个文件的位置、职责和依赖关系。从下一章开始，我们会从 memcached.c 的 main() 函数出发，沿着 Memcached 的启动路径一行一行地走一遍。在这个过程中，你会亲眼看到 settings 如何初始化、slabs 如何分配第一块内存、worker 线程如何启动。我们第 4 章见。
