第18章 生产级多智能体系统

到目前为止，我们已经讲了协议（MCP、A2A）、安全（攻防、免疫系统），可以说多智能体系统的"基础设施层"已经覆盖得差不多了。但是，如果你现在去搭建一个生产级MAS，你会发现还缺一个关键拼图：工程化。

让我讲一段真实的经历。2025年初，我们团队把一个多Agent客服系统部署到生产环境。原型阶段一切完美——几个Agent协同工作，回答准确、流程顺畅。上线第一天，流量从测试的每分钟10个请求飙升到每分钟1000个——然后一切就崩了。不是Agent的逻辑有问题，而是：不知道哪个Agent在处理哪个请求（没有分布式追踪），一个Agent卡住了导致整个链路超时（没有熔断），几天后收到云账单才发现LLM调用费用是预算的8倍（没有成本监控）。

这些问题的本质是：原型系统中无关紧要的事情，在生产系统中成为致命问题。本章将逐一讨论这些生产级MAS的工程化挑战。

18.1 多智能体可观测性

可观测性（Observability）是生产级系统的第一块基石。在多Agent系统中，可观测性的挑战比单体Agent大得多——因为你不仅要观测每个Agent的状态，还要观测Agent之间的交互链路。

18.1.1 分布式追踪

在多Agent系统中，一个用户请求可能会触发多个Agent的协作。比如"写一份季度报告"这个请求，可能先触发调研Agent收集数据，然后触发分析Agent处理数据，最后触发文案Agent撰写报告。如果中间某个环节出了问题，你如何定位是哪个Agent、哪个步骤出了问题？

答案是分布式追踪。借鉴微服务架构中的OpenTelemetry标准，我们可以在MAS中为每个请求创建一个Trace，每个Agent处理对应一个Span。

Trace: "生成Q2季度报告"
├── Span: orchestrator-agent (耗时2.3s)
│   ├── Span: research-agent (耗时45.2s) ← 耗时异常！
│   │   ├── Span: tool-call: database-query (耗时3.1s)
│   │   ├── Span: tool-call: web-search (耗时12.4s)
│   │   └── Span: tool-call: competitor-api (耗时28.7s) ← 瓶颈
│   ├── Span: analysis-agent (耗时8.1s)
│   └── Span: writer-agent (耗时15.3s)
└── Span: evaluation (耗时1.2s)

从这个追踪树中可以一目了然地看到：整个请求耗时超过70秒，瓶颈在research-agent的competitor-api调用（28.7秒）。你可以进一步查看这个API调用返回了什么数据、是否超时重试了——所有信息都在Span的metadata中。

实现分布式追踪的关键是在Agent间传递上下文（Context）。当编排Agent调用调研Agent时，需要把Trace ID传递过去；调研Agent调用工具时，再把Span ID传递过去。A2A协议的metadata字段天然支持这种上下文传递——你可以把traceId和spanId放在metadata里。

18.1.2 日志、指标与仪表板

除了分布式追踪，生产级MAS还需要这三个支柱：

支柱 作用 MAS特有数据
日志（Logs） 记录每个Agent的详细决策过程 Agent推理链、工具调用参数、记忆访问记录
指标（Metrics） 聚合统计系统运行状态 Agent响应时间、成功率、工具调用频率、LLM Token消耗
仪表板（Dashboard） 可视化展示系统全局状态 Agent健康度热力图、协作拓扑图、SLA达标率

一个值得推荐的做法是：每个Agent在启动时注册自己的"信息端点"，编排层通过A2A协议的tasks.get接口（或专门的status端点）定期拉取Agent的状态信息，聚合后展示在仪表板上。这样你不仅可以知道"Agent是否活着"，还能知道"Agent当前在处理什么任务、已处理了多少、平均耗时多少"。

开源推荐： 不要从头造可观测性轮子。直接基于OpenTelemetry生态来构建MAS可观测性——用OTel SDK在每个Agent中埋点，数据导出到Jaeger（分布式追踪）、Prometheus（指标）和Grafana（仪表板）。LangGraph和CrewAI在2026年都已经内置了OTel集成，开箱即用。

18.2 熔断与降级模式

在网络不稳定的分布式系统中，熔断（Circuit Breaker）是一种经典的容错模式。在MAS中，由于Agent依赖LLM API和外部工具，不稳定性是一个固有属性——LLM可能超时、API可能限流、工具可能返回异常数据。熔断模式在MAS中同样适用，而且需要针对Agent的特点进行定制。

18.2.1 单Agent故障隔离

当一个Agent出现故障（连续超时、返回异常结果），不应该让它拖垮整个系统。熔断器会监控Agent的调用成功率，当错误率超过阈值时，熔断器"断开"——停止向该Agent发送请求，快速返回一个降级响应。

// 熔断器状态转换
正常 (CLOSED)
  └── 错误率 > 50% 且 最近请求 >= 10
      └── 断开 (OPEN)
          └── 等待 30 秒
              └── 半开 (HALF_OPEN)
                  ├── 测试请求成功 → CLOSED
                  └── 测试请求失败 → OPEN（重置等待计时器）

18.2.2 超时熔断

Agent调用LLM的延迟是不可预测的，特别是当模型需要多步推理（Inference-Time Scaling）时。一个复杂的推理任务可能需要几秒到几十秒。因此，为每类Agent调用设置合理的超时时间非常关键。

实践中的做法是：动态超时。根据Agent的历史P99延迟动态调整超时阈值，而不是用固定值。一个Agent在空闲时段可能只需要2秒，但在高峰期可能需要10秒——用固定超时要么导致误熔断，要么容忍了过长的等待。

18.2.3 功能降级

当某个关键Agent不可用时，系统不应该简单地报错。功能降级（Graceful Degradation）让系统在"不完美"的状态下继续运行：

  • 回退策略： 复杂Agent不可用→回退到简单Agent。例如分析Agent挂了→回退到直接查询缓存数据的简单模式。
  • 简化推理： 长推理链不可用→使用单步推理。例如多Agent协作挂了→退化为单Agent直接回答。
  • 缓存回退： Agent调用失败→返回最近的缓存结果（附带"结果可能不是最新的"提示）。

降级策略应该作为Agent系统设计的一部分，而不是事后补救。在设计每个Agent时，就应该明确回答一个问题：如果这个Agent挂了，系统应该怎么应对？

设计原则： 一个好的降级策略应该做到"用户无感或者小幅感知"。用户不需要知道系统内部有一个Agent宕机了——她只需要知道"系统还在工作，只是响应慢了一点"或"某些高级功能暂时不可用"。把Agent故障变成用户体验的可接受降级，而不是500错误。

18.3 Token经济学

如果说可观测性和熔断是"可靠性"问题，那么Token消耗就是"成本"问题。在生产级MAS中，Token消耗可能是一个惊人的数字——而且不同框架的消耗差异可以大到令人咋舌。

18.3.1 框架成本对比

2025年底，有团队对不同MAS框架做了基准测试，结果非常有意思。以下是在相同任务集（1000个任务/天）下的LLM调用成本估算（基于GPT-4o级别的定价）：

框架/架构 1000任务/天 10,000任务/天 年化成本（1000/天） 特点
LangGraph $12-18 $120-180 $4,380-6,570 图状态机，精确控制推理步数
CrewAI $18-25 $180-250 $6,570-9,125 角色化，多Agent轮次对话
AutoGen $25-40 $250-400 $9,125-14,600 Agent自由对话，收敛成本高

这些数据说明了一个关键事实：框架选择直接影响运营成本。LangGraph之所以最省，是因为它的图状态机结构对Agent的对话轮次有精确控制——Agent按预定义的图路径执行，不会有多余的"闲聊"或"自我纠正"。而AutoGen允许Agent之间自由对话，这种灵活性有时候会导致"话痨"——两个Agent就同一个问题讨论了5轮才达成一致，而这5轮对话的成本全部计入了你的账单。

18.3.2 成本优化策略

既然框架选择决定了成本基线，具体的成本优化还可以从以下几个方向入手：

模型分层： 不是所有Agent都需要最强的模型。调研Agent可以用Claude Haiku级别的小模型做初步筛选，只有需要深度分析的任务才调用Claude Opus或GPT-4o级别的大模型。

缓存策略： Promise缓存是一个被低估的成本优化手段。很多Agent任务有重复性——比如多个用户问同一个问题、Agent在多个推理步中反复查询同一个数据。对这些结果做语义级别的缓存（不是精确匹配，而是语义相似度匹配）可以显著减少Token消耗。

Prompt压缩： 现代LLM提供了Prompt Caching功能（相同的前缀token只计费一次），充分利用这个功能可以降低30-50%的输入Token成本。关键是把"不变的部分"（系统提示、Agent角色描述）和"可变的部分"（用户输入、中间结果）清晰分离。

一个实用的成本公式： 在预算有限的情况下，用这个简化公式估算MAS的月度成本：
月成本 ≈ 日任务量 × 每任务推理步数 × 每步Token数 × Token单价 × 30
假设一个任务平均5步推理，每步5000 Token，Token单价$3/1M：
1000/天 × 5 × 5000 × $3/1M × 30 = $2,250/月
这个数字可以帮助你在架构选型时做粗略的可行性判断。

18.4 难度感知路由

一个"MVP级"的MAS可能是"所有请求都发给所有Agent"。但生产级MAS必须做难度感知路由（Difficulty-Aware Routing）——根据任务的复杂程度，将任务分配给不同规模的处理链。

18.4.1 路由策略设计

路由的核心思想是：简单问题不要大炮打蚊子。一个典型的难度感知路由策略分为三层：

用户请求
    │
    ▼
┌─────────────────┐
│ 分类器 Agent    │  ← 判断任务复杂度（小模型，极低成本）
│ (Claude Haiku)  │
└────────┬────────┘
         │
    ┌────┴────┬──────────┐
    ▼         ▼          ▼
┌────────┐ ┌──────┐ ┌────────┐
│ 简单任务│ │中等任务│ │ 复杂任务│
│ 单Agent │ │2-3   │ │ 4+     │
│ 直接回答│ │Agent │ │Agent   │
└────────┘ │协作  │ │协作    │
           └──────┘ └────────┘
              │          │
              └──────────┘
                    │
                    ▼
           ┌────────────────┐
           │   输出路由     │
           │  (质量检查)    │
           └────────────────┘

这个路由策略的成本节省非常显著：假设80%的用户请求是简单问题（"今晚的股价是多少"），15%是中等复杂（"分析这只股票近一个月的走势"），只有5%是复杂问题（"给我一个投资组合建议并解释理由"）。用难度感知路由后：

  • 80%的任务走单Agent路径，成本极低（~$0.002/任务）
  • 15%走中等路径（~$0.02/任务）
  • 5%走复杂路径（~$0.10/任务）
  • 加权平均成本 ≈ $0.009/任务——比"所有任务都用多Agent"降低约60-70%

18.4.2 动态难度评估

难度评估本身也不能一成不变。同一个问题在不同时间点可能属于不同难度等级——比如"分析这个季度报表"在月末是简单问题（月初已经处理过类似的），在月初可能就是复杂问题（新的数据刚更新）。

一个进阶的做法是基于Session的难度调整：如果同一个用户连续提了多个简单问题，系统可以逐渐降低对其请求的难度评估（因为已经了解上下文了）。反之，如果用户在简单问题上反复纠缠（说明问题实际上不简单），系统可以提高难度评估。

踩坑实录： 我们在实现难度感知路由时犯过一个经典错误——分类器Agent对"看起来简单但实际复杂"的问题分类失误。一个用户问"帮我看看这个PDF有什么问题"，分类器将其归类为简单任务（单Agent），结果用户上传了一个200页的合规审计报告，单Agent完全处理不了。解决方案是加入"置信度阈值"机制：当分类器对难度判断的置信度低于某个阈值时，自动升级到更高级别的处理链。同时，在Agent处理过程中如果发现任务比预期更复杂，允许"请求升级"——就像在客服系统中升级给更高级的客服一样。

18.5 持续部署与A/B测试

最后，一个生产级MAS需要像任何现代软件系统一样，支持持续部署（CD）和A/B测试。但在Agent系统中，CD和A/B测试有一些特殊之处。

18.5.1 Agent的灰度发布

当你更新一个Agent的Prompt、调整其工具集、或者更换底层的LLM模型时，你不能直接把所有生产流量都切到新版本。你需要灰度发布：

  • 金丝雀发布： 让1%的流量走新版本Agent，99%走旧版本。监控新版本的关键指标（成功率、响应时间、用户满意度）
  • 逐步放量： 如果金丝雀稳定，逐步扩大到5%→20%→50%→100%
  • 自动回滚： 如果新版本的关键指标劣化超过阈值，自动切回旧版本

Agent系统的灰度发布有一个独特的难点：Agent行为的不确定性。传统软件中，新版本要么正确要么错误，边界清晰。但在Agent系统中，新版本可能"大多数时候正确，偶尔犯糊涂"——这种不确定性使得灰度发布的评估指标设计更加复杂。

18.5.2 Agent的A/B测试

除了灰度发布，A/B测试在Agent系统中也有独特的应用场景：

测试维度 对照组（A） 实验组（B） 评估指标
Prompt策略 详细角色描述 简洁角色描述 任务完成率、Token消耗
模型选择 Claude Sonnet GPT-4o 推理质量、延迟、成本
协作拓扑 串行3个Agent 并行2个+仲裁Agent 处理时间、结果一致性
记忆策略 全部对话存记忆 只存关键信息 检索准确度、Token消耗

A/B测试在MAS中比传统系统中的挑战更大——因为Agent的行为受到随机性（模型采样温度）的影响。同一个Agent用同样的Prompt、同样的输入，两次推理的结果可能不一样。这就意味着你需要更大的样本量才能得到统计显著的结果。经验值是：Agent A/B测试需要至少10倍于传统A/B测试的样本量。

18.5.3 在线评估Pipeline

A/B测试离不开评估。一个生产级MAS应该建立在线评估Pipeline：

持续评估 Pipeline ───────────┐
                              │
生产流量 ──→ 分流 ──→ A/B Agent ──→ 结果采集 ──→ 指标计算
                              │                          │
                              │                     ┌─────┴─────┐
                              │                     │ 达标？    │
                              │                     │ 通过/回滚 │
                              │                     └───────────┘
                              │
                        人工抽检样本 ──→ 质量评分
                                    (每周随机抽检100条)

其中人工抽检是Agent系统评估中不可替代的一环。自动化指标（成功率、响应时间）只能反映"Agent是否正常运行"，不能反映"Agent的回答质量"。建议每周随机抽检100-200条Agent输出，由人工打分评级。这些评分数据回馈到评估系统中，帮助校准自动评估模型。

总结一下： 生产级MAS的工程化挑战归结为三个核心问题：(1) 出了故障你能不能快速发现并定位（可观测性 + 熔断）；(2) 你的系统成本是否可持续（Token经济学 + 难度感知路由）；(3) 你能否安全地迭代和优化（CD + A/B测试）。这三个问题每一个都不容易，但它们决定了你的MAS能活多久。从原型到生产，最大的门槛不是AI能力，而是工程化能力。

本章可能是本教程中最"不AI"的一章——我们几乎没有讨论LLM、没有讨论推理能力、没有讨论Agent智能。但正是这些"不AI"的工程化细节，决定了AI系统在生产环境中的成败。我们讨论的分布式追踪、熔断降级、成本控制和A/B测试，本质上都是在回答同一个问题：如何让一个由多个AI Agent组成的系统，在生产环境中可靠、高效、可维护地运行？
