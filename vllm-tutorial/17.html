第17章 可观测性

生产环境有一条铁律：没有监控的服务就是定时炸弹。你不知道什么时候 OOM、不知道哪个用户把服务打挂了、不知道为什么 P99 延迟突然飙到 30 秒。这一章建立 vLLM 的可观测性体系。

17.1 三层监控体系

第一层 — 硬件指标:
├── GPU 利用率、显存用量、温度
├── CPU 使用率、内存使用率
└── 网络 I/O、磁盘 I/O

第二层 — vLLM 推理指标:
├── TTFT (Time to First Token)
├── TPOT (Time Per Output Token)
├── 吞吐量 (tokens/s, requests/s)
├── KV Cache 使用率、空闲 block 数
├── 前缀缓存命中率
└── 等待队列长度

第三层 — 业务指标:
├── 请求成功率、错误率
├── 用户维度 QPS
├── 平均/最大生成长度
└── Token 消耗量

17.2 关键指标详解

指标含义正常范围告警阈值
TTFT (P50)50% 请求的首 token 延迟<100ms-
TTFT (P99)99% 请求的首 token 延迟<500ms>1s
TPOT (mean)平均每 token 生成时间<15ms>30ms
GPU UtilizationGPU SM 计算利用率85-95%<70%（资源浪费）
KV Cache UsageKV Cache 已用比例70-90%>98%（接近 OOM）
Free Blocks空闲 Block 数量视 block 总数<100（快不够了）
Queue Depth等待队列长度<10>50（GPU 跟不上）
Prefix Cache Hit Rate前缀缓存命中率取决于场景RAG 场景 <50%（检查路由）

17.3 Prometheus 指标采集

vLLM 自动暴露 Prometheus 格式的指标：

# 查看原始指标
curl http://localhost:8000/metrics

# 关键 Prometheus 指标名称:
vllm:time_to_first_token_seconds       # TTFT
vllm:time_per_output_token_seconds     # TPOT
vllm:request_success_total             # 成功请求数
vllm:gpu_cache_usage_perc              # KV Cache 使用率
vllm:num_requests_waiting              # 等待队列
vllm:prefix_cache_hit_rate             # 缓存命中率
vllm:request_prompt_tokens             # 输入 token 数
vllm:request_generation_tokens         # 生成 token 数

17.4 Grafana 告警规则

# PrometheusRule 示例
groups:
  - name: vllm-alerts
    rules:
      - alert: HighTTFT
        expr: histogram_quantile(0.99, vllm:time_to_first_token_seconds) > 1.0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "P99 TTFT is above 1 second"

      - alert: LowKVCache
        expr: vllm:gpu_cache_usage_perc > 0.95
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "KV Cache usage above 95%, risk of OOM"

      - alert: HighQueueDepth
        expr: vllm:num_requests_waiting > 50
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Request queue is building up"

Oncall 经验：我见过最常触发的告警是 KV Cache 接近耗尽（>95%）。原因通常是某个用户发了超长的 prompt（"把所有文档都塞进来"）占据了大量 block。解法：设置 --max-model-len 硬限制单个请求长度，或者用 rate limiting 限制调用频率。

17.5 分布式链路追踪

对于复杂的推理链路（API Gateway → Router → vLLM），可以加 OpenTelemetry：

# vLLM 不内置 OTEL，但可以通过 sidecar 注入
# 或者用 vLLM 的 X-Request-ID header 做日志关联

# API 请求带 request_id
curl -H "X-Request-ID: abc-123" http://vllm:8000/v1/chat/completions ...

# vLLM 日志中会打印 X-Request-ID，方便 trace
