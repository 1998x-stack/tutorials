第20章 软核处理器集成

在 FPGA 里跑一个 CPU？听起来像是"套娃"——既然你有 ARM 或 x86 主机，为什么还要在 FPGA 逻辑里再实现一个处理器？答案很简单：灵活性。软核处理器让你在 FPGA 内部运行 C 代码来做控制流决策（if-else、循环、状态切换），同时让 RTL 硬件做数据路径加速（滤波、FFT、编解码）。两者通过 AXI 总线连接，各取所长。

本章我们学习三种主流的软核处理器——Xilinx MicroBlaze、Intel Nios II/V、以及开源 RISC-V 软核——并重点实战 MicroBlaze 的集成与软件开发。

20.1 什么是软核处理器

软核处理器是用 FPGA 的逻辑资源（LUT + Flip-Flop + BRAM）实现的 CPU。与之相对的是"硬核"处理器——比如 Zynq 系列中集成的 ARM Cortex-A9，它是芯片上固定的硅片区域，不消耗 FPGA 逻辑资源。下表对比了当前主流的软核处理器方案：

软核名称ISA资源消耗典型频率DMIPS/MHz特性适用厂商
MicroBlazeMicroBlaze v11/v12~900 LUTs（最小配置）100~200 MHz1.15AXI 原生支持、FPU 可选、MMU 可选AMD/Xilinx
Nios II/fNios II~1400 LUTs100~200 MHz1.03硬件乘除、指令缓存、分支预测Intel/Altera
Nios VRV32I~2000 LUTs~100 MHz~1.0基于 RISC-V，Intel 2021 年发布Intel/Altera
VexRiscvRV32I[M][C]~1200 LUTs（精简配置）100~150 MHz~1.2SpinalHDL 描述，高度可配置任意 FPGA
PicoRV32RV32I[M][C]~750 LUTs50~100 MHz~0.6最精简，C 语言 Tiny 实现任意 FPGA
SERVRV32I~200 LUTs~20 MHz~0.05位串行 RISC-V，最小门数任意 FPGA
一个有趣的对比：PicoRV32 只用 ~750 LUTs 就能跑 RV32I 指令集，但性能只有 ~0.6 DMIPS/MHz。MicroBlaze 消耗 ~900 LUTs 但达到 1.15 DMIPS/MHz——同样的资源，几乎翻倍的性能。这充分说明微架构设计（流水线深度、前推策略、分支预测）对性能的影响远大于指令集本身。

20.2 MicroBlaze 系统搭建：Vivado Block Design 实战

在 Vivado 中搭建 MicroBlaze 系统的典型步骤如下。我们以 Arty A7-100T（Artix-7） 开发板为例：

  1. 创建 Vivado 工程：选择目标器件（xc7a100tcsg324-1），添加 Verilog 顶层文件。
  2. 打开 IP Integrator（Block Design）：点击 "Create Block Design"。
  3. 添加 MicroBlaze：在 Diagram 面板中添加 MicroBlaze IP 核。Vivado 2023 以后的版本会弹出 "Run Block Automation" 向导。
  4. 运行 Block Automation：向导会自动添加：DDR 控制器（MIG）、AXI Interconnect、JTAG 调试模块（MDM）、时钟模块（Clocking Wizard）、复位模块（Proc Sys Reset）。
  5. 添加外设：手动添加 AXI GPIO、AXI UART Lite、AXI Timer。通过 "Run Connection Automation" 自动连接 AXI 总线。
  6. 验证设计：运行 "Validate Design"（F6），检查地址映射是否重叠。
  7. 生成 Output Products：Generate Bitstream，导出到 Vitis。

这个过程看起来大多是 UI 操作，但理解背后的原理很重要：Block Automation 本质上是在帮你实例化一系列 RTL 模块并通过 AXI 总线互联——这和第 19 章我们手写的 AXI4-Lite Slave 被挂到了同一个总线上。

20.3 软核 + RTL 协处理架构

软核处理器的真正价值不在于它自己能算多快——它当然跑不过硬核 ARM 或者独立 MCU——而在于它和 RTL 逻辑的紧密耦合。典型的架构模式是：

  • 控制平面（C 代码）：运行在 MicroBlaze 上，负责任务调度、参数配置、异常处理、人机交互。用 if-else 和 switch-case 表达业务逻辑。
  • 数据平面（RTL）：用 Verilog 实现 FIR 滤波器、FFT、编码器、协议解析器。通过 AXI4-Stream 与 MicroBlaze 或 DDR 直连，实现零拷贝数据传输。
  • 桥梁：AXI4-Lite 用于控制寄存器配置，AXI4-Stream 用于批量数据搬运，AXI Interconnect 负责地址路由。

这种"软核做控制，RTL 做加速"的模式在软件无线电（SDR）、机器视觉前端、高速数据采集等领域非常常见。

20.4 软件开发流程：Vitis 裸机编程

硬件比特流生成之后，进入软件开发阶段。使用 Vitis（Xilinx 的统一软件 IDE）或老旧的 SDK 工具：

  1. 创建 Platform 工程：导入 Vivado 导出的 .xsa（Xilinx Shell Archive）文件。
  2. 创建 Application 工程：选择模板（Hello World、Peripheral Tests 等）。
  3. 配置 BSP（Board Support Package）：设置 stdin/stdout 的 UART 设备、堆栈大小、中断优先级。
  4. 编写 C 代码：使用 Xilinx 提供的驱动库 API——XGpio_Initialize()、XUartLite_Send()、XTmrCtr_Start()。
  5. 调试：通过 JTAG 下载程序，设置断点，查看寄存器，串口终端观察输出。

下面是一个完整的裸机 C 程序示例，它通过 GPIO 读取按键状态，通过我们上一章手写的 AXI4-Lite PWM 外设控制 LED 亮度：

// MicroBlaze 裸机 C 程序：按键控制 LED PWM 亮度
// === 硬件系统假设 ===
// GPIO_0: 4-bit 按键输入 (DEVICE_ID = XPAR_GPIO_0_DEVICE_ID)
// GPIO_1: 4-bit LED 输出 (DEVICE_ID = XPAR_GPIO_1_DEVICE_ID)
// CUSTOM_PWM: 上一章手写的 AXI4-Lite 外设，0x44A00000 基地址
//   - reg_ctrl [0x00]: 占空比
//   - reg_status[0x04]: PWM 周期

#include <xparameters.h>
#include <xgpio.h>
#include <xil_printf.h>
#include <sleep.h>

#define PWM_BASEADDR    XPAR_CUSTOM_PWM_0_S00_AXI_BASEADDR
#define PWM_DUTY_REG    0x00
#define PWM_PERIOD_REG  0x04
#define STEP            50

XGpio gpio_led;
XGpio gpio_btn;

void pwm_set_duty(u32 duty) {
    *(volatile u32 *)(PWM_BASEADDR + PWM_DUTY_REG) = duty;
}

void pwm_set_period(u32 period) {
    *(volatile u32 *)(PWM_BASEADDR + PWM_PERIOD_REG) = period;
}

int main() {
    u32 duty = 500;
    u32 period = 1000;
    int ret;

    // Initialize GPIOs
    ret = XGpio_Initialize(&gpio_led, XPAR_GPIO_0_DEVICE_ID);
    if (ret != XST_SUCCESS) {
        xil_printf("GPIO LED init failed: %d\r\n", ret);
        return -1;
    }
    ret = XGpio_Initialize(&gpio_btn, XPAR_GPIO_1_DEVICE_ID);
    if (ret != XST_SUCCESS) {
        xil_printf("GPIO BTN init failed: %d\r\n", ret);
        return -1;
    }

    // Set data direction: LED = output (0), BTN = input (1)
    XGpio_SetDataDirection(&gpio_led, 1, 0x0);
    XGpio_SetDataDirection(&gpio_btn, 1, 0xF);

    // Initialize PWM
    pwm_set_period(period);
    pwm_set_duty(duty);

    xil_printf("\r\n=== MicroBlaze PWM Control Demo ===\r\n");
    xil_printf("Press BTN0 to increase brightness, BTN1 to decrease.\r\n");

    while (1) {
        u32 btn_val = XGpio_DiscreteRead(&gpio_btn, 1);
        if (btn_val & 0x01) {
            if (duty + STEP <= period) duty += STEP;
            else duty = period;
        }
        if (btn_val & 0x02) {
            if (duty >= STEP) duty -= STEP;
            else duty = 0;
        }
        pwm_set_duty(duty);
        // Reflect duty cycle on LED GPIO as visual feedback
        XGpio_DiscreteWrite(&gpio_led, 1, duty >> 6);
        sleep(1);  // debounce delay
    }

    return 0;
}
代码要点：注意我们直接通过 volatile u32 * 指针访问了 AXI4-Lite 外设的寄存器——这是最直接的方式，不需要额外的驱动框架。但实际工程中更推荐用 Xil_Out32() / Xil_In32() 宏（定义在 xil_io.h），它们在函数调用层之上加入了内存屏障保护，对某些架构（如 MicroBlaze 开启了写缓冲时）是必要的。

20.5 软核 vs 硬核：性能、功耗与灵活性的权衡

在 Xilinx 生态中，你有三种方案可以选择：（1）纯 FPGA + MicroBlaze（软核）；（2）Zynq-7000 / Zynq UltraScale+ 中的硬核 ARM Cortex-A 系列；（3）Versal 中的 AI Engine。理解它们的权衡至关重要：

  • 性能：Zynq 的 ARM Cortex-A9 在 667 MHz 下可达 ~2.5 DMIPS/MHz，而 MicroBlaze 在 150 MHz 下是 ~1.15 DMIPS/MHz。硬核的绝对性能高 5~10 倍。
  • 资源：MicroBlaze 消耗 ~900~3000 LUTs（取决于配置），硬核不消耗逻辑资源。
  • 灵活性：软核可以自定义指令扩展（MicroBlaze 允许用户添加自定义协处理器接口 —— FSL 或者 AXI-Stream）。硬核固定不变。
  • 功耗：软核的功耗与资源利用率成正比。如果只用少量外设，MicroBlaze 的功耗可以低至几十 mW。
踩坑回顾——速度等级的秘密：我在一个项目中使用 MicroBlaze 跑 100 MHz，在 Artix-7 速度等级 -2（最常用的中等速度等级）上时序完美通过。后来因为芯片缺货，改用了同一型号的 速度等级 -1（较慢等级），同样的比特流——连布局布线都没有改——时序报告显示 MicroBlaze 路径违例，最大频率只能跑到 88 MHz。最终不得不降频到 80 MHz 重新综合。教训：软核处理器通常是设计中逻辑路径最深的模块之一（尤其是取指→译码→写回的关键路径），在切换芯片速度等级时必须重新做时序分析，不能赌"应该差不多"。
