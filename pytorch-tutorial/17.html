第17章 性能优化基础

大家好，我是你们的讲师。这一章我们来聊聊每个深度学习工程师都关心的话题——性能优化。

你可能会想：无非就是训练时间长一点嘛，让 GPU 多跑一会儿不就完了？但实际情况是，在工业级项目中，训练速度直接决定了你的实验迭代上限。如果别人一天能跑 20 组实验，你一天只能跑 5 组，那别人的模型收敛速度、调优效果就是你的 4 倍。算力是有成本的——不管是自建集群还是用云 GPU，每一分每一秒都是在烧钱。

好消息是，从 PyTorch 2.0 开始，许多加速手段变得越来越简单。这一章我们会从最"暴力"的 torch.compile() 开始，然后逐步深入到 cuDNN 调优、GPU 内存管理、DataLoader 瓶颈排查，以及 fuse optimizer 等高级技巧。

17.1 torch.compile()：一行代码的魔法

如果要评选 PyTorch 2.x 时代最重要的特性，torch.compile() 绝对排第一。它把 Python 层面的 PyTorch 代码在运行时编译成高效的 GPU 内核，带来 30%~80% 甚至更高的加速——而你的改动仅仅是一行代码。

import torch

model = MyModel().cuda()

# 一行魔法
model = torch.compile(model)

# 后面的代码完全不用改
for batch in dataloader:
    pred = model(batch)
    loss = criterion(pred, labels)
    loss.backward()
    optimizer.step()

torch.compile() 的工作原理可以理解为三步走：

  • Graph Capture（图捕获）： 运行一次模型，Python 层面的操作被捕获成计算图
  • Graph Lowering（图降级）： 将高级操作分解成更小的原子操作（算子融合、消除冗余计算等）
  • Triton Kernel Generation（内核生成）： 自动为融合后的操作生成高性能的 Triton 内核

最终的效果就是：Python 解释器的开销被降到了最低，访存模式被优化了，GPU 利用率大幅提升。

17.2 mode= 参数的选择

torch.compile() 提供了三个 mode 选项，在不同场景下有不同的权衡：

mode 典型加速比 编译时间 显存消耗 推荐场景
"default" 1.3x ~ 1.6x 短（秒级） 低 日常开发、快速迭代
"reduce-overhead" 1.5x ~ 2.0x 中（秒到分钟） 中 生产训练、追求更极致速度
"max-autotune" 1.8x ~ 2.5x 长（分钟到小时） 高 部署前的最终性能调优

在大部分情况下，"default" 模式就足够好了。编译时间短、加速效果明显，不会打断你的开发节奏。如果你想要更激进一些，可以用 "reduce-overhead"——它会在内核启动开销上做更多优化。"max-autotune" 则适合在你准备发布模型之前，专门花一段时间来做最终的调优。

# 不同 mode 的使用方式
model_default = torch.compile(model, mode="default")
model_balanced = torch.compile(model, mode="reduce-overhead")
model_fastest  = torch.compile(model, mode="max-autotone")

17.3 dynamic=True：处理变长输入

torch.compile() 默认假定输入张量的形状是固定的——这对于静态图优化非常有利。但如果你处理的是 NLP 模型或者时间序列数据，输入长度可能会随着每个 batch 变化。这时候需要设置 dynamic=True：

model = torch.compile(model, dynamic=True)

开启后，编译器会生成可以适应不同输入形状的内核。代价是加速效果会略有降低（因为编译器不能做假设某些维度是常量来做极致的优化），但比完全不使用 torch.compile 还是要快得多。

什么时候用 dynamic=True？当你的模型有变长输入（如文本、序列数据、不同尺寸的图像），且你会使用 torch.nn.utils.rnn.pad_sequence 或类似的 padding 机制时。如果输入形状固定（如图像分类的 224x224），保持 dynamic=False（默认）以获得最佳性能。

17.4 cuDNN Benchmark：自动调优的加速与代价

另一个简单的加速方式：torch.backends.cudnn.benchmark = True。这个选项会在第一次运行卷积操作时，对当前输入尺寸做基准测试，选择最有效的卷积算法。对于固定输入尺寸的模型（如大多数图像分类模型），它可以带来 10%~30% 的加速。

torch.backends.cudnn.benchmark = True

但是需要注意：benchmark 模式会导致训练结果不可复现。因为每次选出来的卷积算法可能是不同的，即使同样的输入、同样的随机种子，得到的浮点运算顺序也可能不同，从而导致些许差异。所以它的典型使用场景是：

  • 正式训练时开启 benchmark 加速
  • 调试或需要精确复现时关闭 benchmark
# 一个常见的实践模式
if args.deterministic:
    torch.backends.cudnn.benchmark = False
    torch.use_deterministic_algorithms(True)
else:
    torch.backends.cudnn.benchmark = True

17.5 GPU 内存管理

显存是训练中的稀缺资源。PyTorch 用到了 CUDA 的缓存分配器——即使你用 del 删除了一个 Tensor，它占用的显存也不会立即释放给系统，而是被缓存在 CUDA 分配器中，以便后续的 Tensor 快速复用。这是为了提高性能，因为频繁向驱动申请/释放显存是很慢的。

几个有用的工具：

# 1. 清空缓存（释放未使用的缓存显存）
torch.cuda.empty_cache()
# 注意：这只释放"空闲的"缓存，不会影响正在使用的 Tensor

# 2. 查看显存使用情况
print(torch.cuda.memory_summary())
# 输出每层的分配细节，定位显存大户

# 3. 更细粒度的分析
print(f"Allocated: {torch.cuda.memory_allocated() / 1024**3:.2f} GB")
print(f"Cached: {torch.cuda.memory_reserved() / 1024**3:.2f} GB")

# 4. 当检测到 OOM 时自动清缓存（不推荐常规使用）
# torch.cuda.empty_cache()  # 在 OOM 异常捕获中调用

对于更深入的显存分析，可以用 torch.cuda.memory_snapshot() 来生成完整的分配历史记录，然后用 PyTorch 官方提供的 memory_viz 工具可视化。这个工具在调试"显存去哪了"的问题上非常好用。

显存优化三原则：
  1. 能不要就不要：检查代码中是否有不必要的中间变量。例如在自定义 loss 函数时，能原地操作就原地操作。
  2. 能小就小：使用 AMP、梯度 checkpointing 等技术减少显存占用。
  3. 能复用就复用：尽量复用已经在显存中的 Tensor，避免频繁地 .to(device) 和从显存拷贝到内存。

17.6 DataLoader 瓶颈排查

很多时候你的 GPU 利用率不高，不是因为模型计算不够快，而是因为 GPU 在等 CPU 喂数据。这就是 I/O 瓶颈。

真实故事：2022 年有一个团队在生产环境训练一个视频理解模型，发现 GPU 利用率只有 20% 左右。他们以为是模型太小、算力过剩，把 batch size 翻了一倍，但 GPU 利用率反而降到了 15%。后来用 nvidia-smi 一看，GPU 的 PCIe 读写一直接近于 0——说明 GPU 大部分时间在做空转等待。再检查 DataLoader，发现 num_workers=0（默认值）。把 num_workers 改为 4 之后，GPU 利用率一下子跳到了 85% 以上。这个团队花了几个月的时间在做所谓的"模型优化"来提升训练速度，结果加了一个参数四行代码就解决了。

DataLoader 的配置非常关键，特别是以下三个参数：

DataLoader(
    dataset,
    batch_size=64,
    shuffle=True,
    num_workers=4,       # 数据加载的子进程数
    pin_memory=True,     # 将数据固定在页锁定内存，加速 GPU 传输
    prefetch_factor=2,   # 每个 worker 预取的 batch 数量
    persistent_workers=True,  # worker 在 epoch 间不重启，减少开销
)

参数选择指南：

参数 推荐值 原理
num_workers CPU 核心数的一半到四分之三 更多的 worker 能并行加载数据，但过多会导致 CPU 争抢
pin_memory True 数据在页锁定内存中，.to("cuda") 时不需要先拷贝到临时内存
prefetch_factor 2 或 3 预取更多 batch 能进一步隐藏数据加载延迟
persistent_workers True 避免每个 epoch 结束后重建 worker 子进程的开销
一个常见的 DataLoader 误用：如果你在 __getitem__ 中做了大量的数据增强（如随机裁剪、颜色抖动等），这些操作会在 worker 进程中执行。如果 num_workers 太少，这些计算密集型操作会成为瓶颈。一个常用的优化是将一部分数据增强操作（如图片解码）放到 worker 里，另一部分（如 ToTensor、Normalize）放到 DataLoader 的 collate_fn 之前或者 GPU 上做。

17.7 Fused Optimizer：更高效的参数更新

从 PyTorch 2.12 开始，官方的 Adam（以及 SGD）已经支持 fused 实现。这个选项将多个独立的 CUDA kernel 调用合并成一个，减少了 kernel launch 开销，在更新大量小参数时效果尤为明显。

optimizer = torch.optim.Adam(
    model.parameters(),
    lr=1e-3,
    fused=True,          # 启用 fused 实现
)

optimizer = torch.optim.SGD(
    model.parameters(),
    lr=0.01,
    momentum=0.9,
    fused=True,
)

对比效果（在 A100 上训练 LLaMA 7B，参数约 7B 个）：

优化器 原始 Fused 提升
Adam 100 it/s 115 it/s +15%
SGD 120 it/s 132 it/s +10%

Fused 优化器的加速在大模型上尤其明显。如果你的 GPU 计算能力在 7.0 及以上（V100、A100、H100、RTX 30/40/50 系列），强烈建议开启 fused=True。

检查 fused 是否可用：torch.optim.Adam 在初始化时会自动检查 GPU 是否支持 fused 实现。如果不支持，只传 fused=True 也不会报错——PyTorch 会静默回退到普通实现。放心使用。
