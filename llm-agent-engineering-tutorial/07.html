第7章 多 Agent 协作模式

大家好，我是你们的讲师。走到这里，我们已经学了不少内容——ReAct 循环、Prompt 工程、工具调用、记忆系统。你的单 Agent 已经相当能干了。但让我们问一个关键问题：一个超级 Agent 能解决所有问题吗？

现实世界的系统很少由单一个体完成所有工作。一个电商平台，有客服团队、仓储团队、支付团队、运营团队——每个团队各司其职，通过协作完成整体业务。同样的道理也适用于 Agent 系统：把复杂任务拆解给多个专业 Agent，往往比用一个"万能 Agent"效果更好。

这一章我们会深入多 Agent 协作的三种核心模式、Agent 间通信协议的设计、同步与异步协作的选择，最后用一个完整的电商客服系统案例把这些知识串起来。

核心洞察： 多 Agent 系统的设计哲学和"微服务架构"有惊人的相似之处——每个 Agent 是一个独立部署的"服务"（具有自己的角色、知识库、工具集），通过明确定义的"接口"（通信协议）与其他 Agent 协作。当你把 Agent 看作"微服务"时，很多架构决策就变得清晰了。

7.1 为什么需要多 Agent

在深入技术细节之前，先回答一个根本问题：一个人 Agent 能搞定的事，为什么要用多个？

主要有四个原因：

  • 角色专业化（Role Specialization）： 一个 Agent 很难精通所有领域。让一个 Agent 同时做客服、数据分析、代码编写——它会在各个角色之间"精神分裂"。而每个 Agent 专注一个角色，System Prompt 可以极致优化，工具集也更有针对性。
  • 知识隔离（Knowledge Isolation）： 不同场景需要不同的知识库。客服 Agent 需要产品目录和 FAQ，研发 Agent 需要技术文档和技术栈信息。把它们放在一个 Agent 里，知识库会变得庞大且混乱。
  • 风险隔离（Risk Isolation）： 如果一个 Agent 崩溃或出错，不会影响其他 Agent。试想一个"万能 Agent"在帮用户查订单时被注入攻击——如果它同时拥有退款权限，后果很严重。而如果退款操作由独立的"财务 Agent"处理并需要额外验证，风险就被隔离了。
  • 可扩展性（Scalability）： 可以根据负载独立扩缩容。大促期间客服 Agent 需要 100 个实例，而数据分析 Agent 只需要 5 个——单 Agent 系统无法做到这种粒度控制。
设计原则： 单 Agent 能解决的事绝不用多 Agent。多 Agent 增加的不是能力，而是复杂性——通信开销、协调成本、调试难度都会大幅上升。设计的黄金法则是：先从单 Agent 开始，当它确实处理不了时，再考虑拆分。

7.2 三种编排拓扑

多 Agent 系统的架构可以从"谁在管理谁"这个角度分成三种基本拓扑。在很多实际系统中，这三种拓扑是混合使用的。

拓扑类型 结构 控制方式 适合场景
Supervisor（集中式） 一个主 Agent 管理所有子 Agent 主控/被控 层次清晰、任务可分解的流程
Peer-to-Peer（分散式） Agent 之间平等对话 协商/投票 需要多方辩论、达成共识的场景
Hierarchical（分层式） 多层管理结构 逐级委派 企业级复杂系统

Supervisor 模式

Supervisor 模式是最常见也最容易实现的多 Agent 拓扑。一个"主管 Agent"接收用户请求，分析需求，然后将任务分解并分派给不同的"专业 Agent"，最后汇总结果。

# Supervisor 模式核心逻辑
class SupervisorAgent:
    """主管 Agent——任务分解与结果汇总"""

    def __init__(self):
        self.specialists = {
            "order": OrderAgent(),       # 订单 Agent
            "shipping": ShippingAgent(), # 物流 Agent
            "payment": PaymentAgent(),   # 支付 Agent
            "complaint": ComplaintAgent() # 投诉 Agent
        }

    async def handle_request(self, user_message: str) -> str:
        """处理用户请求——分发到专业 Agent"""
        # 1. 分析意图，决定需要哪些专业 Agent
        intent = await self._classify_intent(user_message)

        # 2. 选择合适的 Agent 处理
        if intent == "order_query":
            result = await self.specialists["order"].handle(user_message)
        elif intent == "shipping_query":
            result = await self.specialists["shipping"].handle(user_message)
        elif intent == "complex":
            # 3. 复杂任务——需要多个 Agent 协作
            order_result = await self.specialists["order"].handle(user_message)
            shipping_result = await self.specialists["shipping"].handle(user_message)
            result = await self._merge_results(
                user_message, order_result, shipping_result
            )

        return result

    async def _classify_intent(self, message: str) -> str:
        """LLM 意图分类"""
        response = await llm_client.complete(
            f"将用户问题分类为 [order_query, shipping_query, "
            f"payment_query, complaint, complex]: {message}"
        )
        return response.text.strip()

    async def _merge_results(self, original_msg: str,
                              *results: str) -> str:
        """使用 LLM 合并多个 Agent 的结果"""
        context = f"用户问题: {original_msg}\n各专业回复:\n"
        for i, r in enumerate(results):
            context += f"Agent {i+1}: {r}\n"
        response = await llm_client.complete(
            f"{context}\n请将以上信息整合为一个完整的回复："
        )
        return response.text

Peer-to-Peer 模式

在 P2P 模式中，没有"主管"——所有 Agent 平等地对话和协商。这种模式最适合需要辩论和共识的场景，比如投资分析中的多空辩论。

# Peer-to-Peer 辩论模式
class DebateAgent:
    """参与辩论的 Agent"""

    def __init__(self, role: str, stance: str):
        self.role = role       # 如"金融分析师"
        self.stance = stance   # "看多"或"看空"

    async def argue(self, topic: str, opponent_points: list) -> str:
        """根据对方的观点进行辩论"""
        prompt = f"""你是{self.role}，立场是{self.stance}。
        讨论主题：{topic}
        对方的论点：{opponent_points}
        请从你的立场出发进行反驳并提出支持论据："""
        return await llm_client.complete(prompt)

# 使用多个辩论 Agent
async def investment_debate(topic: str):
    bull = DebateAgent("宏观分析师", "看多")
    bear = DebateAgent("风控分析师", "看空")

    rounds = 3
    bull_points = []
    bear_points = []

    for round in range(rounds):
        bull_arg = await bull.argue(topic, bear_points)
        bull_points.append(bull_arg)

        bear_arg = await bear.argue(topic, bull_points)
        bear_points.append(bear_arg)

    # 由第三个 Agent 作为裁判，综合双方观点
    return await judge_debate(topic, bull_points, bear_points)
踩坑实录： 2025 年我们在一个投资分析项目中尝试了 P2P 辩论模式——让三个 Agent（看多、看空、中立的宏观分析师）对某支股票进行辩论。理论上这是个好主意，但实践中发现：三个 Agent 很容易陷入"循环论证"——看多的不断找利好消息，看空的不断找利空消息，各说各话，根本无法收敛。最终我们加了一个"裁判 Agent"，在每一轮辩论后给出"达成共识的点"和"分歧点"，引导辩论聚焦在有实质分歧的问题上，效果才明显改善。

Hierarchical 模式

分层模式是 Supervisor 的扩展——不是一层管理，而是多级管理。比如，一个"业务线主管"管理"客服组长"和"运营组长"，而"客服组长"又管理"售前 Agent"和"售后 Agent"。

分层模式适合大型企业级系统，但实现复杂度也最高。一般建议只在确实需要时才使用——如果两层的 Supervisor 模式能解决问题，就不需要三层的分层模式。

7.3 Agent 间通信协议设计

多 Agent 系统能否正常工作，很大程度上取决于Agent 之间怎么通信。一个设计良好的通信协议，比任何内部优化都重要。

通信消息格式

建议采用结构化的消息格式，而不是让 Agent 之间自由对话：

# 标准化的 Agent 间消息格式
from dataclasses import dataclass
from typing import Optional, List
from enum import Enum
from datetime import datetime

class MessageType(Enum):
    REQUEST = "request"       # 请求
    RESPONSE = "response"     # 响应
    QUERY = "query"           # 查询信息
    NOTIFY = "notify"         # 通知
    ERROR = "error"           # 错误

class Priority(Enum):
    LOW = 1
    NORMAL = 2
    HIGH = 3
    CRITICAL = 4

@dataclass
class AgentMessage:
    """Agent 间通信的标准消息格式"""
    msg_id: str                    # 消息唯一 ID
    msg_type: MessageType          # 消息类型
    sender: str                    # 发送者 Agent ID
    recipient: str                 # 接收者 Agent ID
    conversation_id: str           # 会话 ID（跟踪上下文）
    content: str                   # 消息内容
    priority: Priority = Priority.NORMAL
    parent_msg_id: Optional[str] = None  # 父消息 ID（用于追踪链路）
    timestamp: str = None          # 时间戳

    def __post_init__(self):
        if self.timestamp is None:
            self.timestamp = datetime.now().isoformat()

通信模式

模式 描述 适用场景
请求-响应 一个 Agent 发送请求，等待回复 问答、查询类任务
发布-订阅 一个 Agent 发布事件，订阅者接收 状态变更通知、事件驱动
广播 一个 Agent 向所有人发送消息 紧急通知、全局状态更新
协商 多个 Agent 来回讨论达成一致 辩论、共识、投票

7.4 同步 vs 异步协作

多 Agent 协作的另一个关键设计决策是：Agent 之间的调用是同步等待还是异步通知？

同步协作

A Agent 调用 B Agent，然后等待 B 返回结果后再继续。这种模式逻辑简单、容易调试，但整体响应时间等于所有步骤的时间之和。

适合场景： 任务步骤之间有严格的依赖关系（必须先查订单信息，才能处理退款），且每一步都很快完成。

异步协作

A Agent 向 B Agent 发送消息后不等待，立刻继续做其他事。B 完成后通过回调或消息队列通知 A。这种模式吞吐量高、资源利用率好，但编程复杂度高。

适合场景： 需要处理大量并发请求，或某个 Agent 的处理时间很长（如生成复杂报告）。

import asyncio
from asyncio import Queue

class AsyncAgentBus:
    """基于消息队列的异步 Agent 通信总线"""

    def __init__(self):
        self.queues: dict[str, Queue] = {}
        self.callbacks: dict[str, callable] = {}

    def register_agent(self, agent_id: str):
        self.queues[agent_id] = Queue()

    async def send(self, message: AgentMessage):
        """异步发送消息到目标 Agent"""
        if message.recipient in self.queues:
            await self.queues[message.recipient].put(message)

    async def receive(self, agent_id: str) -> AgentMessage:
        """阻塞等待接收消息"""
        return await self.queues[agent_id].get()

    async def request(self, message: AgentMessage,
                      timeout: float = 30.0) -> AgentMessage:
        """同步方式的请求-响应（底层用异步实现）"""
        response_queue = Queue()
        self.callbacks[message.msg_id] = response_queue
        await self.send(message)

        try:
            response = await asyncio.wait_for(
                response_queue.get(), timeout=timeout
            )
            return response
        except asyncio.TimeoutError:
            raise TimeoutError(f"Agent {message.recipient} 响应超时")
实践建议： 大多数场景从同步协作开始。只有当你明确需要以下能力时才引入异步：(1) 高并发（每秒 100+ 请求），(2) 长耗时任务（超过 30 秒），(3) 需要任务排队和优先级控制。过早引入异步会让系统复杂性急剧上升，调试难度翻倍。

7.5 任务分配与结果聚合

两个在多 Agent 系统中反复出现的工程问题：任务怎么分？结果怎么合？

任务分配策略

  • 基于规则的分配： 根据关键词或简单分类路由到指定 Agent。实现简单，但不灵活。
  • 基于 LLM 的意图路由： 用 LLM 分析用户意图，动态决定分配给哪个 Agent（或哪些 Agent）。灵活，但增加了一次 LLM 调用的开销。
  • 竞标模式（Auction）： 任务广播给所有 Agent，每个 Agent 评估自己是否适合处理并报价（信心分数），由调度器选择最优的。适合大型系统。

结果聚合模式

class ResultAggregator:
    """多 Agent 结果聚合器"""

    @staticmethod
    async def merge_parallel(results: List[dict]) -> str:
        """合并多个并行 Agent 的返回结果"""
        merged = []
        for r in results:
            if r["status"] == "success":
                merged.append(f"[{r['agent_id']}]\n{r['content']}")
            else:
                merged.append(f"[{r['agent_id']}] (处理失败: {r['error']})")

        # 用 LLM 将多个结果整合为连贯的回复
        return await llm_client.complete(
            f"请将以下来自不同专家的回复整合为一份完整、连贯的回答：\n"
            + "\n\n".join(merged)
        )

    @staticmethod
    async def merge_sequential(results: List[dict]) -> str:
        """串联执行——后一个 Agent 的输出是前一个的输入"""
        # 结果本身已经是连贯的，直接返回最后一个 Agent 的输出
        return results[-1]["content"] if results else ""

7.6 实战：电商客服多 Agent 系统

让我们把本章所有的知识点串联成一个完整的电商客服多 Agent 系统。这个系统会处理典型的电商用户请求，包含多个专业化 Agent。

# 电商客服多 Agent 系统

class ECommerceSystem:
    """电商客服多 Agent 系统"""

    def __init__(self):
        # 专业 Agent 团队
        self.agents = {
            "router": RouterAgent(),        # 路由 Agent（Supervisor）
            "order": OrderAgent(),          # 订单查询 Agent
            "shipping": ShippingAgent(),    # 物流查询 Agent
            "return_refund": ReturnAgent(), # 退换货 Agent
            "recommend": RecommendAgent(),  # 商品推荐 Agent
            "complaint": ComplaintAgent(),  # 投诉处理 Agent
        }
        self.message_bus = AsyncAgentBus()
        for aid in self.agents:
            self.message_bus.register_agent(aid)

    async def handle(self, user_message: str) -> str:
        """处理用户请求"""
        # 1. 路由 Agent 分析意图
        route_result = await self.agents["router"].route(user_message)

        if route_result["type"] == "single":
            # 2. 简单查询——分配给一个专业 Agent
            agent = self.agents[route_result["target"]]
            return await agent.handle(user_message)

        elif route_result["type"] == "multi":
            # 3. 复杂请求——并行调用多个 Agent
            tasks = [
                self.agents[target].handle(user_message)
                for target in route_result["targets"]
            ]
            results = await asyncio.gather(*tasks)

            # 4. 路由 Agent 汇总结果
            return await self.agents["router"].summarize(
                user_message, dict(zip(route_result["targets"], results))
            )

        elif route_result["type"] == "escalate":
            # 5. 无法处理——转人工
            return "很抱歉，我无法处理您的请求，已转接人工客服。"


class RouterAgent:
    """路由 Agent——负责意图识别和任务分派"""

    async def route(self, message: str) -> dict:
        """分析用户意图，返回路由决策"""
        prompt = f"""分析用户问题，返回 JSON：
        {{
            "type": "single"|"multi"|"escalate",
            "targets": ["order"] | ["order", "shipping"] | [],
            "reason": "简短原因"
        }}
        可用 Agent: order(订单), shipping(物流),
                   return_refund(退换货), recommend(推荐),
                   complaint(投诉)
        用户: {message}"""

        response = await llm_client.complete(prompt)
        return json.loads(response.text)

    async def summarize(self, original_msg: str,
                         agent_results: dict) -> str:
        """汇总多个 Agent 的结果"""
        parts = [f"用户问题: {original_msg}"]
        for agent_name, result in agent_results.items():
            parts.append(f"[{agent_name}]: {result}")
        response = await llm_client.complete(
            "\n".join(parts) + "\n请整合为对用户的统一回复："
        )
        return response.text

class OrderAgent:
    """订单查询 Agent"""

    async def handle(self, message: str) -> str:
        # 调用订单系统 API 查询
        return "您的订单 #ORD-20260601 已于 2026-06-01 发货。"


class ShippingAgent:
    """物流查询 Agent"""

    async def handle(self, message: str) -> str:
        # 调用物流 API 查询
        return "您的包裹当前在【上海分拣中心】，预计明天送达。"


# 启动系统
async def main():
    system = ECommerceSystem()

    # 测试请求
    test_queries = [
        "我的订单什么时候到？",          # 单 Agent
        "帮我查一下订单 ORD-123 的物流状态", # 多 Agent
        "我要投诉快递员服务态度差",       # 投诉
    ]

    for query in test_queries:
        print(f"\n用户: {query}")
        response = await system.handle(query)
        print(f"系统: {response}")
架构要点： 这个电商客服系统的设计遵循了几个关键原则：(1) 路由 Agent 作为唯一的入口点，统一做意图分类；(2) 专业 Agent 各司其职，只负责自己领域内的任务；(3) 复杂任务通过并行调用提升效率；(4) 结果由路由 Agent 统一汇总，确保回复风格一致。你可以在第 26 章的项目实战中看到这个系统的完整实现。
