从"猜数字"游戏说起

我想起小时候玩过一个游戏——一个人在1到100之间想一个数字，另一个人来猜，每次猜完对方会告诉你"大了"还是"小了"。你最快的策略是什么？当然是每次都猜中间的那个数字。第一次猜50，对方说"小了"，下一次就猜75……用这种二分法，最多7次就能猜中（因为 2⁷ = 128 > 100）。

二叉搜索树（Binary Search Tree, BST）就是将这种"二分查找"的思想应用到树形结构的产物。它让"查找"从遍历所有节点（O(n)）变成了沿着一条路径直达目标（O(log n)）。听起来很棒，对吧？但先别急着欢呼——它有一个"致命缺陷"，我们会在本章末尾揭晓。

先来学习这颗"聪明"的树是如何工作的。

BST的定义与性质

二叉搜索树（BST）是一棵二叉树，并且满足以下性质：

  • 左子树上所有节点的值都小于根节点的值。
  • 右子树上所有节点的值都大于根节点的值。
  • 左右子树也都是 BST（递归定义）。

注意——是"所有"左子树节点都小于根，不仅是左孩子。这意味着：

# 这是一棵合法的BST：
#       5
#      / \
#     3   8
#    / \
#   2   4
# 因为 2 < 4 < 5 < 8，所有左子树节点(2,3,4)都小于5

# 这不是一棵合法的BST：
#       5
#      / \
#     3   8
#    / \
#   2   6   ← 6在根5的左子树中，但6 > 5，违反了BST性质！

最重要的推论：对BST进行中序遍历，得到的是一个严格升序的序列。这一点怎么强调都不过分——它是BST所有操作的核心，也是面试中考察BST最常见的切入点。

# 对上面的合法BST进行中序遍历
# 左→根→右: [2, 3, 4, 5, 8]
# 确实是一个升序序列

def is_sorted_by_inorder(root):
    """验证BST性质：中序遍历是否升序"""
    result = []
    def inorder(node):
        if not node:
            return
        inorder(node.left)
        result.append(node.val)
        inorder(node.right)
    inorder(root)
    return all(result[i] < result[i+1] for i in range(len(result)-1))

BST的查找

BST的查找就是"猜数字"游戏的树形版本。从根开始，比较目标值和当前节点的值：

  • 如果相等 → 找到了，返回节点。
  • 如果目标值 < 当前值 → 去左子树找。
  • 如果目标值 > 当前值 → 去右子树找。
  • 如果走到空节点 → 不存在，返回None。
def search_bst(root, target):
    """在BST中查找目标值"""
    if not root:
        return None
    if root.val == target:
        return root
    elif target < root.val:
        return search_bst(root.left, target)
    else:
        return search_bst(root.right, target)

# 迭代版本（更节省栈空间）
def search_bst_iterative(root, target):
    curr = root
    while curr and curr.val != target:
        if target < curr.val:
            curr = curr.left
        else:
            curr = curr.right
    return curr

查找的时间复杂度：

  • 最好情况：树是平衡的，O(log n) — 每次排除一半的节点。
  • 最坏情况：树退化成链表，O(n) — 每次只能排除一个节点。
  • 平均情况：随机插入的BST，O(log n) 的期望值。

BST的插入与删除

插入

插入的思路和查找类似——先找到"应该放哪里"，然后把新节点挂上去。

def insert_bst(root, val):
    """在BST中插入一个新值，返回新的根节点"""
    if not root:
        return TreeNode(val)
    if val < root.val:
        root.left = insert_bst(root.left, val)
    elif val > root.val:
        root.right = insert_bst(root.right, val)
    # 如果相等，通常不插入（BST通常不包含重复值）
    return root

删除（重头戏）

删除是BST最复杂的操作。根据被删除节点的子节点数量，有三种情况：

情况 描述 处理方法
情况1：叶子节点 没有子节点，比如删除上例中的 2 或 4 直接删除父节点指向它的指针即可
情况2：只有一个子节点 比如删除上例中的 8（假设8有右孩子9） 用子节点替换被删除节点（让父节点直接指向子节点）
情况3：有两个子节点 比如删除上例中的 3 找到右子树的最小节点（后继）或左子树的最大节点（前驱），替换值，然后递归删除那个替换节点 
def delete_bst(root, val):
    """在BST中删除一个值，返回新的根节点"""
    if not root:
        return None

    if val < root.val:
        root.left = delete_bst(root.left, val)
    elif val > root.val:
        root.right = delete_bst(root.right, val)
    else:
        # 找到了要删除的节点
        # 情况1和2：叶子或只有一个子节点
        if not root.left:
            return root.right
        if not root.right:
            return root.left

        # 情况3：有两个子节点
        # 找右子树的最小节点（后继）
        successor = root.right
        while successor.left:
            successor = successor.left
        # 用后继节点的值替换当前节点
        root.val = successor.val
        # 递归删除右子树中的后继节点
        root.right = delete_bst(root.right, successor.val)

    return root

删除常见于LeetCode第450题「删除二叉搜索树中的节点」。面试官特别喜欢考察你对"双子节点删除"的理解——尤其是用后继节点替换的思路。

验证BST

我们来看一道经典面试题——LeetCode 98「验证二叉搜索树」。题目很简单：给定一棵二叉树，判断它是否是合法的BST。

很多人的第一反应是："对树做中序遍历，检查结果是否升序。"这个思路是对的，但面试官可能会追问："你能不用额外的数组空间吗？"

这里介绍一种更优雅的做法——区间递归法：

def is_valid_bst(root):
    """验证BST，区间递归法"""
    def validate(node, low, high):
        if not node:
            return True
        # 当前节点的值必须在 (low, high) 区间内
        if node.val <= low or node.val >= high:
            return False
        # 左子树的值必须在 (low, node.val) 之间
        # 右子树的值必须在 (node.val, high) 之间
        return (validate(node.left, low, node.val) and
                validate(node.right, node.val, high))

    return validate(root, float('-inf'), float('inf'))

为什么不用 is_sorted_by_inorder 中的数组法？因为区间法的时间复杂度同样是 O(n)，但空间复杂度是 O(h)（h是树的高度），比数组法的 O(n) 空间更优。而且区间法更直接地利用了BST的定义本身——它没有依赖"中序遍历有序"这个推论，而是直接验证BST的递归约束。

面试锦囊：当面试官问"验证BST"这道题时，先给"中序遍历法"（最直观），然后主动提出"但我还可以用区间递归法，空间复杂度更优"。这展示了你从"直觉"到"优化"的思维过程，比只给出一个完美答案更有说服力。

BST的局限

现在到了本章最重要的部分——BST的"致命缺陷"。

考虑这种情况：你按升序依次插入 1, 2, 3, 4, 5 到一棵空的BST中。插入过程如下：

插入1:
  1

插入2:
  1
   \
    2

插入3:
  1
   \
    2
     \
      3

插入4:
  1
   \
    2
     \
      3
       \
        4

插入5:
  1
   \
    2
     \
      3
       \
        4
         \
          5

天哪！这哪里是树，这分明是一条链表！

血的教训：我曾经在一个生产系统中用BST来存储用户ID到会话信息的映射。数据量小的时候一切正常，查找飞快。但随着用户量增长，某些时间段的插入模式导致BST严重"偏科"——树的高度从 10 左右暴涨到了 5000 多。查询从"毫秒级"退化成了"需要遍历半棵树"——有时候一次查询要遍历几千个节点，系统响应时间从 10ms 飙到了 2 秒。那次故障让我深刻理解了：BST的理想性能建立在"数据随机分布"的假设上，而这个假设在现实中往往不成立。

这个场景并不是偶然的——很多现实数据是有序的（按时间戳插入、按自增ID插入），而这些数据恰恰会导致BST退化成链表。

BST的操作复杂度总结：

操作 平均复杂度 最坏复杂度
查找 (Search) O(log n) O(n)
插入 (Insert) O(log n) O(n)
删除 (Delete) O(log n) O(n)
中序遍历 (In-order) O(n) O(n)

看到没有——最坏情况下所有操作都退化为 O(n)。这就是"说好的二分查找呢？"

解决办法也很明确：让树保持平衡。在插入和删除时，通过各种旋转操作，自动维护树的高度在 O(log n) 量级。这就是下一章要讲的平衡二叉树——AVL树和红黑树。

面试金句："BST是平衡树的基础，但直接使用BST是有风险的——当输入有序时它会退化为链表。所以实际工程中我们使用自平衡的BST变体，比如红黑树和AVL树。红黑树用在需要频繁插入删除的场景（比如Java的TreeMap、C++ STL的map），AVL树用在查询多于修改的场景（比如数据库索引的候选结构）。"
