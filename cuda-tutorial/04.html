第一个CUDA内核

大家好，我是你们的讲师。前两章我们做了两件事：搭好了CUDA开发环境，了解了GPU的内部构造。现在，是时候亲手写一段在GPU上跑的代码了。

在编程世界里，每个新领域的第一个程序都有一个传统——"Hello, World!"。在CUDA的世界里，这个位置属于向量加法（vector addition）。它简单到不能再简单：两个等长数组对应元素相加。但它包含了CUDA编程的所有核心要素：内核函数定义、内存管理、数据搬运、内核启动、结果回传。可以说，你搞懂了向量加法，就搞懂了CUDA编程80%的基本流程。

__global__ 与内核启动语法 <<<>>>

先看一个最精简的CUDA内核和它的启动代码：

// 一个极简的CUDA内核——向量加法
__global__ void vecAdd(float *a, float *b, float *c) {
    // 计算当前线程负责的元素的全局索引
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    c[idx] = a[idx] + b[idx];
}

int main() {
    // ... 内存分配、数据准备 ...

    // 启动内核：1个grid，256个block，每个block256个线程
    vecAdd<<<256, 256>>>(d_a, d_b, d_c);

    // ... 结果回传、清理 ...
}

这里有三个关键元素需要理解：

  • __global__：函数限定符，告诉编译器这个函数是一个可以在GPU上执行、从CPU端调用的内核。它返回类型必须是 void。
  • <<<gridDim, blockDim>>>：内核启动的三尖括号语法。第一个参数是网格维度（多少个线程块），第二个参数是块维度（每个块多少个线程）。<<<256, 256>>> 意味着总共 256 × 256 = 65536 个线程。
  • threadIdx / blockIdx / blockDim：CUDA内置变量，用于让每个线程知道自己的"身份"。我们将在第5章深入讨论它们。
三尖括号扩展： <<<grid, block, sharedMemPerBlock, stream>>> 实际上有四个参数。后两个是可选的——sharedMemPerBlock 指定动态共享内存大小（第8章），stream 指定执行流（第12章）。在深入学习之前，只关心前两个就够了。

Host与Device代码分离

CUDA程序中有两个截然不同的"世界"：Host（CPU及其内存）和 Device（GPU及其显存）。函数和变量可以用限定符表明它们属于哪个世界：

限定符 在哪执行 从哪调用 典型用途
__global__ Device Host ↔ Device (CC 3.5+) 内核函数——入口点
__device__ Device Device only GPU上的辅助函数，只能被内核或其他device函数调用
__host__ Host Host only CPU上的普通函数（这是默认值）
__host__ __device__ 两者 两者 可以同时在CPU和GPU上编译执行的函数

由于Host和Device有各自独立的内存空间，我们必须用CUDA API管理数据搬运。三个最核心的API是：

  • cudaMalloc()：在Device上分配显存
  • cudaMemcpy()：在Host和Device之间拷贝数据，需要指定方向
  • cudaFree()：释放Device上的显存

下面是一个完整的向量加法程序：

// vecAdd.cu —— 完整的向量加法示例
#include <stdio.h>
#include <cuda_runtime.h>

// GPU内核：向量加法
__global__ void vecAdd(float *a, float *b, float *c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {          // 边界检查
        c[idx] = a[idx] + b[idx];
    }
}

int main() {
    int n = 1 << 20;        // 1048576 个元素
    size_t bytes = n * sizeof(float);

    // 1. 分配Host内存并填充数据
    float *h_a = (float*)malloc(bytes);
    float *h_b = (float*)malloc(bytes);
    float *h_c = (float*)malloc(bytes);
    for (int i = 0; i < n; i++) {
        h_a[i] = i * 1.0f;
        h_b[i] = (n - i) * 1.0f;
    }

    // 2. 分配Device内存
    float *d_a, *d_b, *d_c;
    cudaMalloc(&d_a, bytes);
    cudaMalloc(&d_b, bytes);
    cudaMalloc(&d_c, bytes);

    // 3. 从Host拷贝到Device
    cudaMemcpy(d_a, h_a, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(d_b, h_b, bytes, cudaMemcpyHostToDevice);

    // 4. 启动内核
    int blockSize = 256;
    int gridSize = (n + blockSize - 1) / blockSize;  // 向上取整
    vecAdd<<<gridSize, blockSize>>>(d_a, d_b, d_c, n);

    // 5. 结果拷回Host
    cudaMemcpy(h_c, d_c, bytes, cudaMemcpyDeviceToHost);

    // 6. 验证结果
    for (int i = 0; i < 10; i++) {
        printf("h_c[%d] = %f (期望 %f)\n", i, h_c[i], h_a[i] + h_b[i]);
    }

    // 7. 清理
    cudaFree(d_a);
    cudaFree(d_b);
    cudaFree(d_c);
    free(h_a);
    free(h_b);
    free(h_c);

    return 0;
}
注意观察流程： 这是一个固定的七步模式——分配Host内存 → 分配Device内存 → Host到Device拷贝 → 启动内核 → Device到Host拷贝 → 验证 → 清理。未来几乎所有CUDA程序都遵循这个流程，不同的只是kernel内部的逻辑。

CUDA错误检查

我刚学CUDA时犯过的最严重的错误是：从来不检查错误。CUDA API函数大部分返回 cudaError_t 类型，但很多人（包括曾经的我）会直接忽略返回值——毕竟CPU编程里malloc不检查返回值也挺常见的不是吗？

但GPU编程不一样。一个错误可能发生在调用之后的任意时刻（因为kernel是异步启动的），如果不显式检查，你很可能在错误发生很久之后才从"奇怪的结果"中发现问题。最好的习惯是：每次CUDA调用都检查错误。下面是一个通用的错误检查宏：

// cuda_error.h —— 错误检查宏
#include <stdio.h>
#include <cuda_runtime.h>

#define CUDA_CHECK(call) do {                                \
    cudaError_t err = call;                                  \
    if (err != cudaSuccess) {                                \
        fprintf(stderr, "CUDA Error at %s:%d — %s\n",       \
                __FILE__, __LINE__,                          \
                cudaGetErrorString(err));                    \
        exit(err);                                           \
    }                                                        \
} while(0)

// 使用示例
CUDA_CHECK(cudaMalloc(&d_a, bytes));
CUDA_CHECK(cudaMemcpy(d_a, h_a, bytes, cudaMemcpyHostToDevice));
vecAdd<<<gridSize, blockSize>>>(d_a, d_b, d_c, n);

// 异步错误需要手动检查
CUDA_CHECK(cudaDeviceSynchronize());
CUDA_CHECK(cudaGetLastError());  // 捕获启动参数错误
记住：kernel启动本身不会报错！ 如果内核启动参数非法（比如blockSize超过设备限制），<<<>>> 不会返回错误——错误会被"记录"下来，直到你调用 cudaGetLastError() 或下一次CUDA API调用时才会暴露。所以 cudaGetLastError() 加上 cudaDeviceSynchronize() 是最好的"查漏"组合。

编译与运行流程

你写好的 .cu 文件需要经过nvcc编译才能运行。nvcc的工作流程分为两步：

  1. 前端（离线编译）：nvcc将 .cu 文件的GPU代码编译成PTX（Parallel Thread Execution，一种虚拟指令集架构），同时将CPU代码部分传递给宿主C++编译器（gcc/clang/MSVC）
  2. 后端（运行时编译）：当程序运行时，NVIDIA驱动将PTX"即时编译"（JIT）成目标GPU的机器码（cubin）。这允许同一个PTX在不同的GPU架构上运行

常用的编译命令和选项：

# 基本编译
nvcc -o vecAdd vecAdd.cu

# 指定计算能力（推荐）
nvcc -arch=sm_80 -o vecAdd vecAdd.cu

# 查看寄存器使用量和shared memory使用量
nvcc -arch=sm_80 --ptxas-options=-v -o vecAdd vecAdd.cu

# 生成PTX中间代码（不生成最终二进制）
nvcc -arch=sm_80 -ptx vecAdd.cu

# 同时为多个架构生成代码
nvcc -gencode arch=compute_80,code=sm_80 \
      -gencode arch=compute_90,code=sm_90 \
      -o vecAdd vecAdd.cu

# 优化级别（-O2是默认）
nvcc -O3 -arch=sm_80 -o vecAdd vecAdd.cu

# 保留GPU代码在PTX中（向后兼容）
nvcc -gencode arch=compute_80,code=sm_80 \
      -gencode arch=compute_80,code=compute_80 \
      -o vecAdd vecAdd.cu

--ptxas-options=-v 这个选项特别有用——它会在编译时打印出每个kernel的寄存器使用量、共享内存使用量、以及是否有寄存器溢出（spill）。这个信息对性能优化至关重要，我们会在后续章节反复用到它。

第一个踩坑故事

我的第一个CUDA程序是一个向量加法。当时我写的时候漏掉了第三个参数 cudaMemcpyHostToDevice ——实际上我写的是 cudaMemcpy(d_a, h_a, bytes, cudaMemcpyDeviceToHost)，就是把方向搞反了。更糟的是，我没有检查返回值。结果呢？程序正常运行，没有崩溃，没有任何错误信息——但输出结果全部是0。

我花了整整一个下午去检查kernel逻辑、检查索引计算、检查线程数量——就是没想到去检查 cudaMemcpy 的方向。最后是一个同事路过看了一眼代码，说了一句："你这数据拷贝方向是不是反了？" 改完，结果正确。

这个教训我至今记忆犹新：CUDA程序99%的错误都出在Host-Device数据管理上，而不是kernel逻辑本身。从那以后我形成了两个习惯：一是永远用CUDA_CHECK宏包裹每个API调用；二是在写任何CUDA程序时，先让程序跑通"最小正确性验证"（比如打印前几个元素的值），再做任何优化。

永远不要假设数据已经在Device上了。 这是一个经典的思维陷阱：你刚做了一次 cudaMemcpy 把数据拷到了Device，但在下次kernel启动前，Host端的代码又修改了Host上的数据——然后你想当然地以为Device上的数据还是最新的。不是的。Host和Device是两个独立的内存空间，修改一端不会自动同步到另一端。如果记不清当前数据在哪一端，用 cudaMemcpy 显式同步一次。

好，你已经亲手写出了第一个CUDA程序并成功运行了它。下一章，我们将深入线程层次结构——grid、block、thread到底如何组织，索引怎么算，以及那些让初学者头疼的边界条件和block size选择策略。搞清楚这些，你以后写kernel就不会再"迷路"了。
