为什么你需要一套方法论？

大家好，我是你们的讲师。今天我们要聊一个所有刷题人最关心的话题：LeetCode刷题方法论。

先讲一个真实的故事。我的一个学弟小张，去年秋天开始准备大厂面试。他给自己定了一个目标：刷够500道题，然后稳稳拿offer。接下来的三个月里，他每天早上6点起床刷题，晚上下班继续刷到凌晨。LeetCode上的题目按照通过率从高到低一路刷下去，每一道题他都把AC代码保存下来，准备面试前再复习一遍。

三个月后，他刷了528道题，信心满满地去面某大厂。第一轮面试，面试官出了一道"合并K个升序链表"。小张觉得这题他做过——但他只记得自己用的是"两两合并"的方法，至于复杂度是多少、有没有更好的解法、为什么用优先队列更好，完全没有印象。面试官追问："你用优先队列的时间复杂度是多少？如果改成分治合并呢？"小张当场卡住。

结果可想而知。挂了。

而他的室友小王，只刷了220道题，却拿到了同一家公司的offer。区别在哪？小王用的是专题刷法：每一个专题先花半小时理解核心数据结构，然后刷10-15道经典题，做完之后做笔记总结模式。他刷的题目数只有小张的一半，但每道题都吃透了。

核心观点：刷题的数量≠面试成功率。真正重要的是方法论——如何选题、如何做题、如何复盘、如何举一反三。一个体系化的刷题方法，能让你用200道题的时间达到别人500道题的效果。

按专题刷 vs 按频率刷

这是初入刷题大门时最先面临的抉择：按专题刷还是按频率刷？我的答案是——分阶段。

第一阶段：按专题刷（建立模式识别）

初学者的核心任务是建立"数据结构-问题模式"的映射关系。当你看到一道题，能在3秒钟内判断出"这道题属于哪个专题、该用哪种数据结构"，你就成功了。按专题刷是实现这个目标最高效的方式。

推荐的专题顺序如下（由浅入深，环环相扣）：

阶段 专题 推荐题数 核心目标
1 数组 & 字符串 15-20 掌握双指针、滑动窗口、前缀和套路
2 链表 10-15 递归/迭代反转、快慢指针、dummy node技巧
3 栈 & 队列 10-15 单调栈、单调队列的应用场景
4 哈希表 10-15 空间换时间思想的落地
5 二叉树 15-20 递归遍历框架、DFS/BFS模板
6 递归 & 回溯 10-15 回溯三要素：选择、约束、结束条件
7 动态规划 15-20 状态定义 + 状态转移方程 = DP核心
8 贪心 & 二分 10-15 学会判断"什么时候能用贪心"
9 图 10-15 邻接表建图、BFS/DFS、拓扑排序
10 进阶结构 10-15 Trie、并查集、堆的高级应用

这个顺序不是随便排的。从数组到链表再到树，数据结构的复杂度递增；从递归到回溯再到DP，算法的抽象程度递增。一路走下来，你会发现后面的专题经常用到前面专题的知识——这是一种螺旋式上升的学习路径。

第二阶段：按频率刷（冲刺企业题库）

当专题刷完一轮（约120-150题），你已经有了基本的知识体系。接下来，进入面试冲刺阶段。这时候的工具不是LeetCode的"Top Interview Questions"，而是CodeTop（codetop.cc）。

CodeTop 聚合了各大厂近一年的面试真题，按出现频率排序。在这个阶段，你的策略应该是：

  • 目标公司优先：想面字节？搜"字节跳动"，按频率刷前30题。
  • 近3个月优先：看最近被问到的频率，放弃一年前的老题。
  • 限时实战模拟：每道题给自己15-20分钟，模拟面试环境。
一个重要的洞察：很多人在冲刺阶段犯的错误是"我已经刷过这道题了，直接跳过"。但你想一想——面试官问的就是这些高频题。如果你觉得自己"做过"就不需要再练了，那面试时很可能就只有"做过"的印象，写不出完整的bug-free代码。高频题值得反复练，直到变成肌肉记忆。

如何正确看题解？

这是刷题中最容易被忽视的技能。很多人做题的流程是这样的：打开题目→想了2分钟没思路→打开题解→"哦~原来是这样"→复制粘贴→AC→下一道。这样刷一年也没有效果。

正确的看题解方法，我称之为"20分钟原则"：

  1. 先自己思考20分钟。20分钟内不要看任何题解或提示。这20分钟里，你可以：在白板上画图、尝试不同的数据结构、写伪代码。即使最终没有解出来，这20分钟的训练价值远大于直接看题解。
  2. 20分钟后，如果还是没有思路，再去看题解。但看题解不是看代码，而是看三样东西：
看什么 为什么重要 具体要问自己的问题
解题思路 理解了思路，才能应对变形题 这道题的解法用了什么数据结构？为什么选它？核心的洞察是什么？
复杂度分析 面试必考，不能只会写不会分析 时间复杂度是多少？空间复杂度是多少？能优化吗？
易错点 避免面试中栽同样的坑 边界条件是什么？（空输入？单元素？全是重复值？）

看完这三样之后，自己把代码写一遍。不要抄，合上题解自己写。写完之后再和题解对照，看看哪里有差别、谁的写法更好。

绝对不要做的事情：直接抄代码。这不仅学不到东西，还会产生一种"我懂了"的错觉。面试官最怕的就是这种"背答案型"候选人——原题能答，稍微一变就倒。还有一个更隐蔽的坏处：抄代码会让你形成代码洁癖依赖，觉得别人的写法永远是更好的，丧失了自己解构问题的能力。

错题本管理

这一点很少有人认真做，但恰恰是拉开差距的关键。用 Notion、Excel、甚至是一个物理笔记本都可以——关键是有结构地记录。

这是我推荐的错题本模板：

字段 填写内容 示例
题号 LeetCode编号+题名 206. 反转链表
题目链接 方便复习直接打开 leetcode.cn/problems/reverse-linked-list
错误原因 为什么没做出来？ 递归思路不熟，base case没想清楚
关键insight 这道题最核心的套路 递归反转：head.next.next = head 是关键
复杂度 最优解的时间/空间复杂度 O(n), O(1) 迭代；O(n), O(n) 递归
同类题 可以举一反三的相关题目 92(反转区间)、25(K个一组反转)
下次复习 按间隔复习计划排期 2026-04-05

复习间隔怎么定？我用的是间隔复习法：

  • 第1次复习：1天后（检验短期记忆）
  • 第2次复习：3天后（检验是否真的理解了）
  • 第3次复习：1周后（检验是否已形成长期记忆）
  • 第4次复习：1个月后（最终巩固）

每次复习时，不要看之前的代码，完全重新做一遍。做完之后再和之前的解法对比。如果下一次仍然做不对，说明你没有真正理解——需要重新看题解、重新做笔记。

口述复杂度分析

面试中，写完代码之后几乎100%会被问到："这道题的时间复杂度是多少？" 如果你支支吾吾或者答错了，面试官会立刻对你产生"这个人的基本功不扎实"的印象。

口述复杂度分析要养成习惯——每做完一道题，不要急着提交，先对自己（或对着镜子）说一遍：

"""口述复杂度分析模板"""

def two_sum(nums, target):
    """
    时间复杂度: O(n) —— 遍历数组一次，哈希表查找O(1)
    空间复杂度: O(n) —— 最坏情况下存储n个元素到哈希表
    为什么不是O(n²)? 因为哈希表的查找是平均O(1)的，
    所以整体就是O(n)。
    能优化空间吗? 如果数组是有序的，可以用双指针做到O(1)空间。
    """
    seen = {}
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    return [-1, -1]

重点在于说清楚"为什么是这个复杂度"和"能不能优化"。面试官要考察的不是你是否记得复杂度公式，而是你是否真的理解算法的执行过程。

三句面试必说：
1. "这道题的时间复杂度是O(…)，因为……"
2. "空间复杂度是O(…)，主要的额外空间消耗是……"
3. "在……条件下，可以优化到O(…)"
把这三点练成习惯，面试的复杂度问题你就不会扣分。

从一道题学到一类题

为了让你更直观地理解"举一反三"的力量，我们以LeetCode上最经典的链表题——206. 反转链表——为例子，看看如何从一道题拓展到一整个题型家族。

反转链表 — 递归解法

def reverse_list_recursive(head):
    """反转整个链表 — 递归版"""
    if not head or not head.next:
        return head
    new_head = reverse_list_recursive(head.next)
    head.next.next = head
    head.next = None
    return new_head

递归解法的关键在于理解head.next.next = head这一行——它做了什么？假设当前节点是head，它的next是已经反转好的子链表的头节点。head.next现在指向的是反转后的最后一个节点，所以我们把那个节点的next指向head，完成反转。base case：空链表或单节点链表直接返回。

反转链表 — 迭代解法

def reverse_list_iterative(head):
    """反转整个链表 — 迭代版，三指针"""
    prev = None
    curr = head
    while curr:
        nxt = curr.next   # 保存下一个节点
        curr.next = prev  # 反转指针
        prev = curr       # 移动prev
        curr = nxt        # 移动curr
    return prev

迭代解法的核心是"三指针"技巧：prev、curr、nxt。每次循环中，curr指向当前要处理的节点，prev指向已经反转好的部分的头节点，nxt指向原始链表的下一个节点。画图理解会非常清晰。

从基础到变形

现在，如果你理解了反转链表的两种写法，下面这些"变形题"你就都能触类旁通了：

LeetCode 题目 与反转链表的关系 额外技巧
206 反转链表 基础题 递归/迭代两种写法都要会
92 反转链表 II 反转区间 [left, right] 找到left的前一个节点，记录边界
25 K 个一组翻转链表 按组反转，整组的区间反转 先判断是否有K个节点，再反转区间
24 两两交换链表中的节点 K=2的特例 递归解法很简洁
143 重排链表 反转后半部分+合并 先找中点，再反转后半，再合并
234 回文链表 反转后半部分+比较 快慢指针找中点

你可以清楚地看到：从206到92到25到24，难度递增，但核心解题方法都是"反转链表"的变体。如果你把206吃透了，意味着你同时掌握了6道题的核心思路。这就是举一反三的力量。

按难度分层的刷题目标

最后一个实用建议：你的刷题目标要按难度分层。不同难度的题目有不同的训练目的。

难度 目标题数 训练目的 建议用时 典型代表
简单 (Easy) 30道 建立信心、熟悉基础API和输入输出 5-10分钟/题 两数之和、反转链表
中等 (Medium) 100道 核心训练目标，覆盖大多数面试题 15-25分钟/题 三数之和、LRU缓存
困难 (Hard) 20道 挑战思路极限，为Sp Offer做准备 30-45分钟/题 合并K个升序链表、接雨水

注意比例：简单 : 中等 : 困难 ≈ 3 : 10 : 2。中等题是绝对主体，这也是面试中核心考察的难度。不要把大量时间花在困难题上——面试中遇到Hard的概率远低于Medium，而且大部分公司的面试难度也就是Medium+。

一个真实的踩坑故事：
我认识的一个朋友，刷了300多道题，自信满满地去面字节跳动。结果面试中遇到一道"字符串解码"（LeetCode 394），他之前刷过这道题的题解，记得大概思路是用栈。但他没有自己手写过——当时看题解觉得"懂了"就跳过了。面试时他写了10分钟，发现栈里存什么类型的数据都想不清楚了，最后交了一个有bug的版本。

事后他复盘说："我以为刷300道就稳了，结果面试考了一道原题的变形，因为我只记住了答案没理解思路。"

记住："看懂"和"能写出来"之间有一条鸿沟。只有亲手写了、调试了、跑通了，这道题才真正是你的。
每道题做完后，问自己三个问题：
1. 为什么选这个数据结构？—— 哈希表？栈？还是队列？有没有其他选择？
2. 有没有更优解？—— 时间复杂度还能降低吗？空间复杂度呢？
3. 边界条件是什么？—— 空输入？只有一个元素？全相同元素？负数？
把这三个问题变成肌肉记忆，你会发现自己的解题质量提升一个档次。

本章小结

在这一章中，我们从方法论的角度重新审视了"刷题"这件事。你学到了：

  • 为什么方法论比题量更重要——500道题可能不如200道题吃透效果好。
  • 按专题刷 vs 按频率刷——分阶段执行：先建立知识体系，再冲刺目标企业。
  • 20分钟原则——先自己思考再去看题解，看题解看思路不看代码。
  • 错题本管理——结构化记录+间隔复习，让每道题都留下痕迹。
  • 口述复杂度——养成每道题分析时间/空间复杂度的习惯。
  • 举一反三——以反转链表为例展示了如何从一道题拓展到一类题。

有了方法论，接下来就要真刀真枪地实战了。下一章，我为你精讲了15道大厂面试中出现频率最高、覆盖面最广的经典题目——从两数之和到LRU缓存，每一道题都会从数据结构选择、复杂度推导到口述模板，全方位讲解。
