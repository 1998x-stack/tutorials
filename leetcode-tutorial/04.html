双指针的本质

大家好，我是你们的讲师。今天要讲的是一个看起来"人畜无害"、但实际上变化多端的技巧——双指针（Two Pointers）。它是 LeetCode 面试中出现频率最高的技巧之一，据统计有超过 100 道题可以直接用双指针解决。

双指针的核心思想非常朴素：用两个指针的协调移动来替代多层循环。通常一个简单的 for 循环解决不了的问题，加一个指针就解决了。根据指针的移动方向和速度，双指针可以分成三种模式：

  • 相向指针（左右夹逼）——两个指针从两端向中间移动，典型问题：两数之和 II、三数之和、盛水容器
  • 同向指针——两个指针从同一端出发，一个快一个慢，典型问题：删除数组重复项、移动零
  • 快慢指针——速度不同，典型问题：环形链表检测
核心思维：双指针之所以高效，是因为它利用了数据本身的"序"（有序性）或"位置关系"来剪枝。当问题涉及"查找数组中满足某种条件的元素对"时，先用"排序 + 双指针"这个组合试试。

相向指针（左右夹逼）

相向指针是最直观的双指针模式。通常的做法是：先排序，然后用两个指针从左右两端向中间遍历。

两数之和 II — 输入有序数组（LeetCode 167）

这道题是 Two Sum 的有序版本。数组已排好序，我们可以用相向指针：left 指向开头，right 指向末尾，根据和与 target 的大小关系决定移动哪个指针。

def two_sum(numbers: list[int], target: int) -> list[int]:
    left, right = 0, len(numbers) - 1
    while left < right:
        current = numbers[left] + numbers[right]
        if current == target:
            return [left + 1, right + 1]
        elif current < target:
            left += 1
        else:
            right -= 1
    return []

时间复杂度 O(n)，空间复杂度 O(1)。相比第一章的 HashMap 解法，相向指针不需要额外空间——前提是数组已经有序。这也说明了不同的前置条件引导出不同的最优解。

三数之和（LeetCode 15）

从两数之和到三数之和，思路是把"三个数的和"转化为"固定一个数 + 两数之和"。

def three_sum(nums: list[int]) -> list[list[int]]:
    nums.sort()
    n = len(nums)
    res = []
    for i in range(n - 2):
        # 跳过重复数字
        if i > 0 and nums[i] == nums[i - 1]:
            continue
        left, right = i + 1, n - 1
        while left < right:
            total = nums[i] + nums[left] + nums[right]
            if total == 0:
                res.append([nums[i], nums[left], nums[right]])
                left += 1
                right -= 1
                # 跳过重复
                while left < right and nums[left] == nums[left - 1]:
                    left += 1
                while left < right and nums[right] == nums[right + 1]:
                    right -= 1
            elif total < 0:
                left += 1
            else:
                right -= 1
    return res

去掉重复结果是这道题最大的坑。需要在两个层面去重：外层 i 跳过重复，内层 left/right 跳过重复。有一道非常类似的题——最接近的三数之和（LeetCode 16），就用同样的框架，只需把判断条件改成记录最小差值即可。

盛最多水的容器（LeetCode 11）

这道题是一个"视觉化"的相向指针：两根高度的线围成一个容器，容器的盛水量由较矮的那根高度 × 宽度决定。移动指针时，我们总是移动较矮的那一侧——因为移动较高的一侧不可能增加容量（高度受限于较矮的线）。

def max_area(height: list[int]) -> int:
    left, right = 0, len(height) - 1
    ans = 0
    while left < right:
        area = min(height[left], height[right]) * (right - left)
        ans = max(ans, area)
        if height[left] < height[right]:
            left += 1
        else:
            right -= 1
    return ans
直观解释：为什么只移动较矮的那一侧？因为面积取决于短板。移动较高的一侧，瓶颈仍然是较矮的那根，而且宽度还在缩小——面积不可能变大。移动较矮的一侧才有机会遇到更高的线。这个"短板决定论"也适用于很多优化问题。

同向指针

同向指针的两个指针从同一起点出发，一快一慢，快指针负责"探索"，慢指针负责"标记"。这种模式最常用于数组原地去重和元素分类。

删除有序数组中的重复项（LeetCode 26）

给定有序数组，原地删除重复元素，使每个元素只出现一次。快指针遍历数组，慢指针标记不重复元素应该放置的位置。

def remove_duplicates(nums: list[int]) -> int:
    if not nums:
        return 0
    slow = 0
    for fast in range(1, len(nums)):
        if nums[fast] != nums[slow]:
            slow += 1
            nums[slow] = nums[fast]
    return slow + 1

核心就是不重复时 slow 前移并赋值。最终 slow + 1 就是不重复元素的个数。这个模板同样适用于"删除重复项 II"（LeetCode 80，允许两个重复）。

移动零（LeetCode 283）

把数组中所有 0 移动到末尾，同时保持非零元素的相对顺序。同样用同向指针：快指针遍历，遇到非零就和慢指针交换，慢指针指向当前已经处理好的序列的末尾。

def move_zeroes(nums: list[int]) -> None:
    slow = 0
    for fast in range(len(nums)):
        if nums[fast] != 0:
            nums[slow], nums[fast] = nums[fast], nums[slow]
            slow += 1

注意题目要求原地修改，不返回新数组。这段代码简洁到只有 5 行——但它清楚地展示了同向指针的精髓：slow 指针左边是已经处理好的部分，fast 指针负责扫描未知区域。

快慢指针

快慢指针在上一章的链表中已经讲过了。这里再补充一道非链表场景的经典题——用快慢指针的思维来解决数组问题。

寻找重复数（LeetCode 287）

给定一个包含 n+1 个整数的数组，数字都在 [1, n] 范围内，所以至少有一个重复数。要求不修改数组且只用 O(1) 额外空间。

这道题非常巧妙——把数组下标和值之间的映射看作一个链表。因为数字都在 1 到 n 之间，我们可以把 nums[i] 当作指向下标 nums[i] 的指针。有重复数意味着存在环，而环的入口就是重复数。

def find_duplicate(nums: list[int]) -> int:
    slow = nums[0]
    fast = nums[0]
    # 第一次相遇
    while True:
        slow = nums[slow]
        fast = nums[nums[fast]]
        if slow == fast:
            break
    # 找环入口
    slow = nums[0]
    while slow != fast:
        slow = nums[slow]
        fast = nums[fast]
    return slow

这个解法的时间复杂度 O(n)，空间复杂度 O(1)。它和 LeetCode 142（环形链表入口）本质上是同一个算法，只是把链表中的 slow = slow.next 换成了 slow = nums[slow]。这再次证明了算法模式的抽象能力——识别出不同问题的底层共同结构，是刷题进阶的关键。

注意陷阱：287 的限制非常严格——不能修改原数组（排除了排序和原地哈希），只能用 O(1) 额外空间（排除了 HashSet）。这迫使你想到"链表环检测"这个看似不相关的模型。如果你在面试中遇到限制很多的题目，试着想想能否把数组索引映射成指针。

踩坑与总结

双指针的坑大多集中在边界条件和指针移动时机上。

我印象最深的一次是写三数之和。当时我的去重逻辑写成了：

if total == 0:
    res.append(...)
    left += 1
    right -= 1
    # 错误：在移动前就去重
    while left < right and nums[left] == nums[left + 1]:
        left += 1

注意最后这行——nums[left] == nums[left + 1] 是在检查当前位置和下一个位置是否相同，这会跳过有效解。正确的做法是先移动指针（left += 1），然后检查当前位置和前一个位置是否相同：nums[left] == nums[left - 1]。

另一个经典场景是盛水容器，有人会把移动逻辑写成：

if height[left] <= height[right]:
    left += 1
else:
    right -= 1

注意我用了 <= 而不是 <。实际上这没问题，因为当高度相等时无论移动哪一侧，宽度都减少 1，面积都不会超过当前值。但如果你在移动时忘记更新 area——那么恭喜你，你会在死循环里待很久。

双指针类型 适用场景 经典题目 典型复杂度
相向指针 有序数组中的元素对查找 167, 15, 11, 16, 18 O(n) ~ O(n²)
同向指针 原地去重、元素分类 26, 283, 80, 27 O(n)
快慢指针 环检测、找中点 141, 142, 287, 876 O(n)
思维模型：看到题目要求"原地修改"或"O(1)额外空间"时，第一时间想到同向指针。看到"有序数组 + 目标值"时，第一时间想到相向指针。看到"找重复/找环"时，第一时间想到快慢指针。

双指针是一个用"空间"（多一个指针）换"时间"（少一层循环）的技巧。它的应用范围极广——下一章的二分查找本质上就是一种特殊的相向指针（left 和 right 在缩小搜索范围），而再下一章的滑动窗口其实就是同向指针的进阶版。我们下一章见！
