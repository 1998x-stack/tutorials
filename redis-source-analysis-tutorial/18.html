第18章：AOF 与混合持久化

大家好，我是你们的讲师。上一章我们深入了 RDB 持久化的实现——全量快照、fork 写时复制、紧凑的二进制格式。RDB 的加载速度确实快，但它有一个天生的弱点：两次快照之间的数据全部暴露在丢失风险中。对于缓存场景也许可以接受，但如果 Redis 存储的是订单状态、用户 Session 或计数数据，"丢几分钟数据"是完全不能容忍的。

这一章我们来学习 AOF（Append Only File）持久化——Redis 的增量日志方案。它会把每一条修改数据的命令以 RESP 协议的形式追加到文件末尾，重启时逐条重放这些命令来恢复数据。由于 AOF 的写入粒度可以控制在"每条命令"甚至"每秒批量刷盘"，数据丢失窗口可以缩小到 1 秒甚至更少。我们还会探讨混合持久化——RDB 的全量快照和 AOF 的增量日志如何珠联璧合。

AOF 写入流程

AOF 的写入流程可以概括为三个步骤：命令传播 → 缓冲 → 刷盘。

feedAppendOnlyFile — 命令传播

当一条写命令（SET、LPUSH、SADD 等）执行完成后，call() 函数会调用 propagate() 将命令传播给 AOF 和从节点。传播到 AOF 的入口就是 feedAppendOnlyFile：

/* feedAppendOnlyFile — 将命令追加到 AOF 缓冲区 */
void feedAppendOnlyFile(struct redisCommand *cmd, int dictid, robj **argv, int argc) {
    sds buf = sdsempty();

    /* Step 1: 如果操作的数据库编号与上次不同，追加 SELECT 命令 */
    if (dictid != server.aof_selected_db) {
        char seldb[64];
        snprintf(seldb, sizeof(seldb), "*2\r\n$6\r\nSELECT\r\n$%d\r\n%d\r\n",
                 numDigits(dictid), dictid);
        buf = sdscat(buf, seldb);
        server.aof_selected_db = dictid;
    }

    /* Step 2: 将命令参数编码为 RESP 协议格式 */
    buf = catAppendOnlyGenericCommand(buf, argc, argv);

    /* Step 3: 将 RESP 命令追加到 AOF 缓冲区 */
    if (server.aof_state == AOF_ON)
        server.aof_buf = sdscatlen(server.aof_buf, buf, sdslen(buf));

    /* Step 4: 如果是后台 I/O 线程写 AOF，通知线程 */
    if (server.aof_use_threads && server.aof_state == AOF_ON)
        aofThreadWake(&server.aof_client);

    sdsfree(buf);
}

这里有两个关键细节。第一，catAppendOnlyGenericCommand 负责将 argv 参数数组编码为 RESP 协议的 Bulk String 数组——这就是 AOF 文件里存储的格式。第二，aof_buf 是一个 SDS 字符串，所有待写入的命令都先堆积在这里，等待 flushAppendOnlyFile 统一刷盘。

flushAppendOnlyFile — 刷盘

flushAppendOnlyFile 是 AOF 写入磁盘的关键函数，它在 beforeSleep 中被调用，每秒最多执行一次（取决于 fsync 策略）：

/* flushAppendOnlyFile — 将 AOF 缓冲区写入磁盘 */
void flushAppendOnlyFile(int force) {
    ssize_t nwritten;
    int sync_in_progress = 0;

    /* Step 1: 如果 AOF 缓冲区为空，直接返回 */
    if (sdslen(server.aof_buf) == 0) {
        /* 如果距离上次 fsync 超过了 rewrite 阈值，触发 AOF 重写 */
        if (server.aof_fsync == AOF_FSYNC_EVERYSEC)
            sync_in_progress = bioPendingJobsOfType(BIO_AOF_FSYNC) != 0;
        goto update_interval;
    }

    /* Step 2: 将缓冲区写入 AOF 文件 */
    nwritten = write(server.aof_fd, server.aof_buf, sdslen(server.aof_buf));
    if (nwritten != (ssize_t)sdslen(server.aof_buf)) {
        /* 写入失败处理 */
        if (nwritten == -1) {
            serverLog(LL_WARNING, "Error writing to AOF file: %s",
                      strerror(errno));
        }
        goto write_err;
    }

    /* Step 3: 根据 fsync 策略决定是否立即调用 fsync */
    if (server.aof_fsync == AOF_FSYNC_ALWAYS) {
        /* always: 每条写入都 fsync */
        redis_fsync(server.aof_fd);
    } else if (server.aof_fsync == AOF_FSYNC_EVERYSEC) {
        /* everysec: 交给后台线程每秒 fsync 一次 */
        if (!sync_in_progress)
            aof_background_fsync(server.aof_fd);
    }
    /* no: 什么也不做，由 OS 决定刷盘时机 */

    /* Step 4: 缩短 AOF 缓冲区 */
    server.aof_buf = sdsempty();
}

fsync 三种策略

fsync 策略是 AOF 性能与安全性的核心权衡点。Redis 提供了三种选择：

策略 配置值 写入时机 fsync 时机 数据安全性 写入吞吐 典型延迟
每次写入都 fsync always 每条命令 每条命令后立即 fsync 最高（最多丢 1 条命令） 低（约 20% 吞吐） ~2ms/次
每秒 fsync everysec 每秒批量写 后台线程每秒一次 中等（最多丢 1 秒数据） 高（接近无 AOF） ~1ms（磁盘空闲时）
由 OS 决定 no 无固定间隔 由内核脏页回写策略决定 低（可能丢 30s+ 数据） 最高 取决于 OS 策略

生产环境推荐 everysec——它在绝大多数场景下做到了"几乎不输 RDB 的吞吐 + 最多丢 1 秒数据"的理想平衡。always 适合金融级场景，但你需要接受吞吐的大幅下降。no 基本不建议使用，除非你明确知道自己不需要 AOF 的数据安全保障。

踩坑实录：我经历过一次由于 everysec 策略引发的严重事故。当时线上某台 Redis 机器的磁盘 IO 出现了周期性抖动（来自同机的其他服务大量写日志），导致 AOF 的 fsync 延迟从 1ms 飙到了 2s+。Redis 的 everysec 策略中有个"保护机制"：如果后台 fsync 线程在 2 秒内没有完成上次的 fsync，主线程的下一次 flushAppendOnlyFile 会被阻塞等待——因为如果放任不管，缓冲区会无限膨胀。结果就是主线程被阻塞了 2 秒，所有客户端请求都排起了长队，延迟毛刺到达了秒级。根因不是 Redis 本身的问题，而是磁盘 IO 被共享机器的其他进程争抢了。解决方案是对 Redis 的磁盘做 IO 隔离（使用 cgroup 或独立磁盘），或者降级到 no 策略临时应急，忍受数据丢失风险。

AOF 重写机制

AOF 文件是追加写入的，随着运行时间的增长，文件会越来越大。考虑一个极端场景：你对同一个 key 做了 100 万次 INCR，AOF 里就记录了 100 万条 INCR key 命令。重启后 Redis 要逐条重放这 100 万条命令，耗时极长。更优雅的做法是：把这 100 万条 INCR 合并成一条 SET key 1000000。

AOF 重写（BGREWRITEAOF）就是做这个"压缩"工作的。它的核心思想是：基于当前内存中的数据状态，生成最小命令集。

/* rewriteAppendOnlyFile — AOF 重写的核心逻辑 */
int rewriteAppendOnlyFile(char *filename) {
    rio aof;
    FILE *fp;
    char tmpfile[256];

    /* 写入临时文件 */
    snprintf(tmpfile, sizeof(tmpfile), "temp-rewriteaof-%d.%d",
             (int)getpid(), (int)time(NULL));

    /* 打开临时文件 */
    fp = fopen(tmpfile, "w");
    if (!fp) {
        serverLog(LL_WARNING, "无法打开临时 AOF 文件: %s", strerror(errno));
        return C_ERR;
    }

    /* Step 1: 如果启用了混合持久化，先写入 RDB 格式头 */
    if (server.aof_use_rdb_preamble) {
        if (rdbSaveRio(&aof, &rsi, RDB_SAVE_AOF_PREAMBLE) == C_ERR) {
            goto werr;
        }
    } else {
        /* Step 2: 遍历所有数据库，生成最小 AOF 命令 */
        for (int j = 0; j < server.dbnum; j++) {
            redisDb *db = server.db + j;
            dict *d = db->dict;

            if (dictSize(d) == 0) continue;

            /* 写入 SELECT 命令 */
            snprintf(seldb, sizeof(seldb),
                     "*2\r\n$6\r\nSELECT\r\n$%d\r\n%d\r\n",
                     numDigits(j), j);
            if (rioWrite(&aof, seldb, strlen(seldb)) == 0) goto werr;

            /* 遍历当前数据库的所有键，为每个键生成重建命令 */
            di = dictGetSafeIterator(d);
            while ((de = dictNext(di))) {
                robj *o = dictGetVal(de);
                /* 根据对象类型调用对应的重写函数 */
                rewriteAppendOnlyFileRio(&aof, o, keystr, expire);
            }
            dictReleaseIterator(di);
        }
    }

    /* Step 3: 用临时文件原子替换正式 AOF 文件 */
    if (rename(tmpfile, filename) == -1) {
        serverLog(LL_WARNING, "重命名 AOF 临时文件失败: %s", strerror(errno));
        unlink(tmpfile);
        return C_ERR;
    }

    return C_OK;
}

AOF 重写利用了和 BGSAVE 同样的 fork 机制：主进程 fork 出子进程，子进程基于 fork 时的内存快照生成新的 AOF 文件。重写期间主进程继续处理请求，新的写命令同时写入旧的 AOF 文件和重写缓冲区。重写完成后，重写缓冲区的增量命令被追加到新 AOF 文件中，然后原子替换旧文件。这样就实现了"零停机 AOF 压缩"。

优化洞察：不同类型的键在重写时使用了不同的压缩策略。例如，List 使用 RPUSH 批量重建（一次性 RPUSH key elem1 elem2 ... elemL），Hash 使用 HMSET 批量重建。这样生成的 AOF 命令数远远少于原始操作数，重放速度也快得多。

混合持久化

Redis 4.0 引入了一个非常实用的功能——混合持久化（aof-use-rdb-preamble）。它的想法很直白：AOF 重写时，不再生成纯 RESP 文本，而是先生成一个 RDB 快照，然后在后面追加重写期间的增量 AOF 命令。

这样生成的 AOF 文件结构如下：

+-------------------+------------------------+
| RDB 格式全量数据    | AOF 格式增量命令        |
| (rdbSaveRio 生成)  | (重写期间的增量)        |
+-------------------+------------------------+
        ^                          ^
        |                          |
  启动时快速加载              重放增量命令
  (几秒完成)                 (秒级完成)

混合持久化的好处非常明显：

  • 加载速度接近 RDB：大部分数据通过 RDB 的二进制格式快速加载，不需要逐条重放命令。
  • 数据安全性接近 AOF：增量部分仍然是 RESP 协议，最多丢失 1 秒数据（取决于 fsync 策略）。
  • 文件大小适中：RDB 部分本身就是压缩的，比纯 AOF 小得多。

启用方式很简单，在 redis.conf 中设置：

# 开启混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes

目前混合持久化已经是 Redis 官方推荐的默认配置，大多数云厂商的托管 Redis 也默认开启了这个选项。

AOF 文件修复

任何持久化方案都绕不开一个现实问题：文件损坏了怎么办？意外的机器断电、磁盘坏道、或者写 AOF 的过程中进程崩溃，都可能导致 AOF 文件尾部不完整。

redis-check-aof 工具

Redis 提供了一个专门的修复工具 redis-check-aof，它可以扫描 AOF 文件并尝试修复：

# 检查 AOF 文件完整性
redis-check-aof appendonly.aof

# 修复 AOF 文件（会截断尾部不完整的命令）
redis-check-aof --fix appendonly.aof

修复的原理是：AOF 文件中每条命令都是以 *（数组长度标识）开头，并且每条 Bulk String 都有明确的前缀长度。工具从文件头开始逐条解析，当遇到解析失败时，就从失败位置开始向前回溯到上一个完整的 \r\n 边界，然后截断文件。这种"截断尾部"的方式虽然会丢失最后几条不完整的命令，但至少能保证 Redis 可以正常重启。

Multi-Part AOF（8.x 新特性）

Redis 8.x 引入了 Multi-Part AOF，它将单个 AOF 文件拆分成多个部分：base.aof（基础全量数据）、incr.aof（增量数据）和 manifest.aof（元数据清单）。这样做的好处是：即使增量文件损坏，基础文件仍然完好；修复时可以精确到损坏的部分，不需要整个重做。

小贴士：线上环境建议定期（比如每天凌晨低谷期）运行 redis-check-aof 对 AOF 文件做预防性检查。同时，AOF 文件和 RDB 文件应该存储在不同磁盘甚至不同机器上——否则磁盘挂了，两种持久化方案同时失效。多做一份异地冷备永远不嫌多。

总结

至此，持久化的两大支柱我们都讲完了。RDB 是全量快照，快但数据有丢失窗口；AOF 是增量日志，数据安全但文件大、加载慢。混合持久化取两者之长：以 RDB 格式做全量基础，以 AOF 格式做增量补充。

从源码层面，我们追踪了一条写命令从 feedAppendOnlyFile 追加到 aof_buf 缓冲区，到 flushAppendOnlyFile 写盘并通过 fsync 保证落盘的完整链路。AOF 重写通过 fork 子进程将百万条 INCR 合并为一条 SET，解决了文件无限膨胀的问题。最后我们看了 redis-check-aof 工具和 Multi-Part AOF 的新特性。

持久化保证了数据在磁盘上的安全，但内存终究是有限的。即使你把 32GB 的内存全用上，如果业务数据增长到了 40GB，Redis 该怎么办？拒绝写入？按照某种策略淘汰旧数据？下一章我们进入《键过期与内存淘汰》，看看 Redis 如何在有限内存中优雅地管理数据的生命周期。
