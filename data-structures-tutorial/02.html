前言：复杂度是什么？

大家好，我是你们的讲师。在上一章中，我们聊了为什么要学习数据结构——因为好的数据结构能让程序运行得更快、更省内存。但问题来了：怎么衡量"更快"和"更省"？你不能光靠感觉说"这个算法好像挺快的"——在面试里这么说，面试官会直接让你走人。

复杂度分析就是那把你手里的尺子。它的核心思想非常简单：当输入规模翻倍时，你的工作量翻了多少倍？如果输入翻倍，工作量也翻倍——那是 O(n)；如果输入翻倍，工作量翻了四倍——那是 O(n²)；如果输入从 100 变成 100,000，你的工作量几乎不变——那是 O(1)。就这一个问题，抵得上一整章教科书。

为什么要关心复杂度？

让我们先建立直觉。假设你写了一个程序，处理 1,000 条数据要 1 毫秒。数据量翻到 1,000,000 条呢？

  • O(1)：还是 1 毫秒 — 稳如泰山
  • O(log n)：大概 2 毫秒 — 几乎不变
  • O(n)：变成了 1,000 毫秒（1 秒）— 还能接受
  • O(n log n)：大约 2,000 毫秒（2 秒）— 开始有点慢了
  • O(n²)：1,000,000 毫秒 ≈ 16.7 分钟 — 已经不能忍了
  • O(2ⁿ)：宇宙毁灭都跑不完 — 别想了

看到了吗？O(n²) 和 O(n) 在数据量小的时候可能差别不大，但一旦数据量上来，那就是毫秒和分钟的区别。这就是为什么大厂面试必考复杂度分析——他们处理的是亿万级别的数据，O(n²) 的代码上线就是事故。

在我带过的学生中，最常见的错误就是"写完了再说"——暴力递归一把梭，跑个小样例发现通过了，就以为大功告成了。等到线上数据量一上来，程序直接超时，还不知道问题出在哪。所以，写代码之前，先想复杂度，这应该成为你的肌肉记忆。

大O表示法

大 O 表示法描述的是算法运行时间随输入规模增长的增长率。它忽略常数和低阶项，只关心"主导项"。下面我们逐一来看常见的复杂度级别，每个都配上实际的代码。

O(1) — 常数时间

不管数据有多大，执行时间固定。数组的随机访问就是典型的 O(1)：

arr = [10, 20, 30, 40, 50]
x = arr[3]   # 不管数组多大，这一步永远是一个内存地址计算

O(log n) — 对数时间

每操作一次，问题规模减半。二分查找是典型代表：

def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1
# 每次循环，搜索范围减半 → O(log n)

O(n) — 线性时间

遍历所有元素一次：

def find_max(arr):
    max_val = arr[0]
    for x in arr:        # 遍历一次
        if x > max_val:
            max_val = x
    return max_val

O(n log n) — 线性对数时间

高效排序算法的标准复杂度，如归并排序：

def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])    # 递归
    right = merge_sort(arr[mid:])   # 递归
    return merge(left, right)       # 合并 O(n)
# 递归深度 O(log n)，每层合并 O(n) → O(n log n)

O(n²) — 平方时间

双重嵌套循环——看到这个就要警惕了：

def has_duplicate(arr):
    for i in range(len(arr)):
        for j in range(i + 1, len(arr)):  # 嵌套循环
            if arr[i] == arr[j]:
                return True
    return False

O(2ⁿ) — 指数时间

不加优化的递归斐波那契，每一个调用分裂成两个：

def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)
# n=50 时，你的电脑可能就要冒烟了

空间复杂度

时间不是唯一的资源，内存也是。空间复杂度衡量的是算法所需的额外内存量。

原地算法 (In-place)

只使用 O(1) 额外空间的算法称为原地算法。比如数组反转：

def reverse_arr(arr):
    left, right = 0, len(arr) - 1
    while left < right:
        arr[left], arr[right] = arr[right], arr[left]  # swap
        left += 1
        right -= 1
    # 只用了一个临时变量 swap，O(1) 额外空间

递归栈空间

递归函数会占用调用栈空间。比如递归求阶乘：

def factorial(n):
    if n == 0:
        return 1
    return n * factorial(n - 1)
# 递归深度为 n，因此空间复杂度 O(n)

这里有个常见的坑：很多人只关注时间复杂度的优化，忽略了递归深度带来的栈溢出问题。我在面试中见过不止一个候选人，写出了漂亮的递归解法，但一问空间复杂度就卡住了。记住：递归不是免费的，每层递归都要吃栈空间。

最好 / 最坏 / 平均情况

同一个算法，在不同输入下的表现可能天差地别。以快速排序为例：

  • 最好情况：每次分区都选到中位数，递归深度 log n → O(n log n)
  • 最坏情况：每次分区都选到最大/最小值，比如输入已经有序且每次都选第一个元素做 pivot → O(n²)
  • 平均情况：随机输入下的期望复杂度 → O(n log n)

这就是为什么生产环境中的快排通常会在选择 pivot 时用"三数取中"或随机化策略——避免最坏情况的发生。在面试中，当你给出一个算法时，最好主动说出"最坏情况下的复杂度是 X，平均情况是 Y，因为……"，这会让面试官对你刮目相看。

均摊分析 (Amortized Analysis)

某些操作偶尔很贵，但大多数时候很便宜。均摊分析把"贵"的那次成本分摊到所有操作上。最经典的例子是动态数组的扩容。

Python 的 list 底层是一个动态数组。当数组满了需要扩容时，会分配一个更大的新数组（通常是当前容量的 1.125 倍），然后把所有旧元素拷贝过去——这单次是 O(n) 的操作。但每次扩容后，又能插入很多次 O(1) 的操作，直到下一次扩容。把扩容的 O(n) 分摊到所有插入操作上，均摊下来每次插入仍然是 O(1)。

类似的思想还会在后续的哈希表 rehash 中遇到。均摊分析的核心洞察是：不要因为偶尔的一次昂贵操作就否定整个数据结构的效率。

常见操作复杂度一览

数据结构 操作 平均复杂度 最坏复杂度
数组（随机访问） arr[i] O(1) O(1)
数组（查找未排序） 线性查找 O(n) O(n)
链表（查找） 遍历查找 O(n) O(n)
链表（头插） insert at head O(1) O(1)
二分查找 sorted array search O(log n) O(log n)
哈希表（查找） dict[key] O(1) O(n)
二叉搜索树 search O(log n) O(n)
排序（比较排序） 最坏情况 O(n log n) O(n log n)
💡 小技巧：不要死记硬背复杂度表。理解"为什么"比记住"是什么"重要一万倍。看到双重循环想到 O(n²)，看到递归分治想到 O(log n)，看到遍历一次想到 O(n)——真正理解了原理，你自然就能推导出来，不需要背。

如何用 Python 测量时间复杂度？

理论归理论，有时候你也需要实际测量来验证自己的分析。Python 的 timeit 模块可以帮你做这件事：

import timeit
import random

# 生成不同规模的数据
for n in [100, 1000, 10000, 100000]:
    arr = [random.randint(0, 1000) for _ in range(n)]
    # 测量线性查找的时间
    t = timeit.timeit(
        lambda: arr.index(500),
        number=100
    )
    print(f"n={n}: {t:.4f}s")

# 输出示例:
# n=100: 0.0002s
# n=1000: 0.0018s
# n=10000: 0.0190s
# n=100000: 0.1950s
# 可以看到 n 增长 10 倍，时间也增长约 10 倍 → 符合 O(n)

实际测量可以帮助你发现一些意想不到的问题——比如缓存未命中、内存分配开销等，这些都是纯理论分析看不到的。但记住，不能只靠测量：一是数据量不够大时差异不明显，二是测量环境有噪声。理论分析 + 实际验证，两者互为补充。

本章小结

复杂度分析是数据结构的"世界观"，它教我们用增长的视角看问题。今天你掌握的核心能力是：看到一个算法或一段代码，能判断出它的时间和空间复杂度。这个能力会贯穿整个课程，也会贯穿你所有的面试和技术工作。
