开篇：一个模型的母语决定了它的灵魂

大家好，我是你们的讲师。在开始本章之前，我想分享一个让我印象深刻的项目经历。2024 年底，我参与了一个面向东南亚市场的多语言大模型项目。我们的模型需要支持包括泰语、越南语、印尼语、高棉语（柬埔寨）、老挝语在内的 8 种语言。CEO 信心满满地说："我们用 GPT-4 的 Tokenizer 就好啦，都是 Unicode，没问题的。"结果模型训练出来后，泰语的生成质量惨不忍睹，越南语勉强及格，印尼语倒是不错。原因很简单：不是所有的语言在互联网上都是"平等的"。印尼语在 Common Crawl 中的占比大约是泰语的 4 倍、高棉语的 50 倍。模型在训练中"见"过多少某种语言的文本，直接决定了它在这种语言上的表现。这就是多语言预训练最残酷的现实——数据的不平等，决定了能力的不平等。

20.1 mC4：多语言数据集的标杆与局限

说到多语言预训练数据，第一个绕不开的就是 Google 的 mC4 (multilingual C4)。它是 C4 数据集的多语言版本，从 Common Crawl 中提取了覆盖 100+ 语言的文本，总规模达到数千亿 token。听起来很了不起，对吧？但实际使用过的团队都知道一个尴尬的事实——质量随语言的"冷门程度"急剧下降。

语言等级 mC4 中的占比 平均文档质量评分 典型问题
英语 ~45% 0.82 少量机器生成文本
欧洲主流语言（德、法、西） ~1-5% 0.71 翻译内容较多
亚洲主流（中、日、韩） ~0.5-3% 0.58 编码错误、混合语言
东南亚语（泰、越、印尼） ~0.05-0.5% 0.42 噪声大、长度短
非洲低资源语言（斯瓦希里、约鲁巴） < 0.01% 0.25 大量重复短文本、语言误标

mC4 的问题根源在于它用的是完全统一的管道处理所有语言——同样的语言检测模型、同样的过滤规则、同样的去重策略。但不同语言的特点截然不同：泰语没有空格分词，日语有汉字/假名混合，阿拉伯语是右到左书写。一刀切的 pipe 必然在低资源语言上表现糟糕。

核心教训：多语言数据清洗不能"一锅煮"。每种语言都需要定制化的处理策略——从分词、语言检测阈值、质量过滤规则到去重算法。

20.2 FineWeb-2：新一代多语言数据管道

HuggingFace 在 2025 年底发布的 FineWeb-2 在很大程度上修正了 mC4 的问题。FineWeb-2 的核心创新在于引入了语言自适应的处理管道：

  • 语言特定质量阈值：英语的 perplexity 阈值设为 15，而斯瓦希里语放宽到 25——因为在低资源语言上，高质量文本本身就有更高的 perplexity。
  • 语言特定去重窗口：CJK 语言的 n-gram 去重窗口大小与拉丁语系不同，因为汉字的信息密度远高于英文字母。
  • 保留混合语言文档：mC4 会粗暴地丢弃语言混杂的文档（如中英混排），但 FineWeb-2 意识到这类文档在多语言模型中反而有训练价值。

FineWeb-2 还做了一项非常有价值的工作：按语言发布了质量评分和过滤建议。如果你要处理某一种语言的数据，可以直接查阅 FineWeb-2 文档中的推荐配置：

# FineWeb-2 风格的语言自适应配置 (Python 伪代码)

LANGUAGE_CONFIG = {
    "en": {
        "perplexity_threshold": 15.0,
        "min_doc_length": 50,        # tokens
        "keep_code_switched": False,
        "n_gram_window": 3,          # Latin
    },
    "zh": {
        "perplexity_threshold": 18.0,
        "min_doc_length": 30,        # 汉字，信息密度高
        "keep_code_switched": True,  # 保留中英混排
        "n_gram_window": 2,          # 汉字2-gram即可
    },
    "th": {
        "perplexity_threshold": 22.0,
        "min_doc_length": 40,
        "keep_code_switched": True,
        "remove_non_thai_chars": True,  # 泰文特殊清洗
        "n_gram_window": 3,
    },
    "sw": {  # 斯瓦希里语
        "perplexity_threshold": 25.0,
        "min_doc_length": 30,
        "keep_code_switched": True,
        "n_gram_window": 3,
    }
}

def process_multilingual(docs, lang):
    cfg = LANGUAGE_CONFIG.get(lang, LANGUAGE_CONFIG["en"])
    return (
        docs
        .filter(lambda d: d.perplexity < cfg["perplexity_threshold"])
        .filter(lambda d: len(d.tokens) > cfg["min_doc_length"])
        .deduplicate(n_gram=cfg["n_gram_window"])
    )
推荐做法：如果你的项目涉及多语言数据，可以 FineWeb-2 作为 baseline，再用自己的数据做增量处理。FineWeb-2 的处理脚本和配置都是开源的，大大降低了多语言数据工程门槛。

20.3 跨语言迁移：高资源语言如何帮助低资源

一个令人鼓舞的现象是：在多语言模型中，高资源语言的知识可以迁移到低资源语言。这不是魔法，而是多语言模型共享 embedding 空间和 Transformer 层的自然结果。BLOOM 和 XLM-R 的实验都表明，在英语上训练学到的推理能力，一定程度可以"泛化"到训练数据中很少出现的语言上。

但迁移是有条件的，主要有三个因素决定了迁移效果：

  • 语言相似性：越南语和中文虽然都属于汉藏/南亚语系范围，但词汇层面的共享很少；而西班牙语和葡萄牙语之间就有大量同源词，迁移效果更好。
  • 共享 tokenization 的效果：Byte-level BPE（如 GPT-4 使用的 tokenizer）比 word-level tokenizer 对低资源语言更友好，因为它可以组合字符级表示。
  • 低资源语言的最低数据量：即使有迁移，每条低资源语言也需要一个最低量的训练数据——我在下一节会展开讲。

20.4 低资源语言的构建策略

当你要处理一种在 Common Crawl 中几乎不存在的语言时，mC4 那种"坐等数据上门"的思路行不通了。你需要主动出击。以下是业界验证有效的三种策略：

策略一：数据增强 (Data Augmentation)

对于极低资源语言，最直接的办法是用该语言中有限的种子数据，通过回译 (Back-Translation) 或噪声注入来生成更多变体。回译的核心操作是：把种子文本翻译成英语（或其他高资源语言），再翻译回原语言。两次翻译过程中引入的自然变异，相当于"免费"的数据增强。

策略二：机器翻译回填 (Machine Translation Backfill)

如果你有丰富的英语语料，可以把它们翻译成目标语言来填充。但这里有一个分界线——翻译数据不能作为预训练数据的主力（机器翻译的分布与天然文本差异太大），但在低资源语言严重不足时，"有总比没有好"。BLOOM 的实验表明，翻译数据最多可以占到该语言数据的 30%，超出这个比例就会显示出"翻译腔"对下游任务的不利影响。

策略三：文字归一化 (Script Normalization)

很多低资源语言有多个书写系统（如哈萨克语可以用西里尔字母也可以用拉丁字母），或者存在大量 unicode 变体（如中文的简体/繁体、日语的旧字/新字）。文字归一化就是把所有变体映射到一个统一的标准写法，避免模型把同一个词的不同写法当成不同的 token。

# 文字归一化示例
import unicodedata

def normalize_script(text, target_lang="zh"):
    if target_lang == "zh":
        # 繁体转简体（使用 opencc 库）
        # text = OpenCC.convert(text, "t2s")
        pass
    elif target_lang == "kk":  # 哈萨克语
        # 西里尔转拉丁
        CYRILLIC_TO_LATIN = {
            "а": "a", "ә": "á", "б": "b", "в": "v",
            "г": "g", "ғ": "ǵ", "д": "d", "е": "e",
            # ... 完整映射表
        }
        text = "".join(CYRILLIC_TO_LATIN.get(c, c) for c in text)

    # Unicode NFKC 归一化
    text = unicodedata.normalize("NFKC", text)
    return text
踩坑提醒：文字归一化需要非常小心——某些"看起来一样"的字符在特定语言中可能是不同的音素。比如越南语有大量带声调符号的拉丁字母，NFKC 归一化可能会破坏声调信息。始终在目标语言母语者的指导下做归一化映射。

20.5 语言识别的难点

多语言数据处理中一个看似简单实则棘手的问题是：怎么精确判断一段文本属于哪种语言？

你以为用 fastText 的语言分类模型就好了？现实要复杂得多。以下是我在实际项目中遇到的三个典型问题：

  • 语码混合 (Code-Switching)：一个句子中同时出现中文和英文是常态，比如"这个 PR 的 CI pipeline 挂了，需要 rebase 一下"，fastText 会输出中文概率 0.6，英语概率 0.4，然后被阈值过滤掉。但这类文本恰恰是技术领域最有价值的数据。
  • 方言变异：阿拉伯语的"方言"（埃及阿拉伯语、黎凡特阿拉伯语）与现代标准阿拉伯语差异巨大，fastText 很容易把它们误标为"阿拉伯语"，但实际上它们不应该混在一起处理。
  • CJK 字符重叠：中文、日文、韩文共享大量汉字字符（繁体中文和日文汉字几乎一样），加上韩文中偶尔出现的汉字，语言识别器经常搞混。
经验法则：如果你在做多语言项目，不要依赖单一的语言分类器。建议使用一个集成方法：fastText + 字符级 n-gram 特征 + 维基百科 anchor text 统计。当多个分类器不一致时，做人工抽样判断。

20.6 最低 Token 预算与 Chinchilla 的多语言适配

最后一个也是最重要的问题：每种语言需要多少训练数据？

回到第 1 章讲过的 Chinchilla 最优比——"20 倍参数量"。在多语言场景下，这个公式需要修正。一个保守但有效的经验法则是：

语言类型 最低数据量建议 备注
高资源语言（英、中） ≥ 500B tokens Chinchilla 推荐的基线
中资源语言（法、西、阿） ≥ 50B tokens 可通过跨语言迁移补足
低资源语言（泰、越） ≥ 10B tokens 低于此值效果急剧下降
极低资源语言（斯瓦希里、高棉） ≥ 1B tokens 需要配合迁移与增强
濒危/无标注数据 ≥ 100M tokens 主要依赖跨语言迁移

这些数字不是凭空而来的。BLOOM 和 Llama 3 的多语言实验中有一个一致的发现：当一种语言的数据低于 10B token 时，模型在这种语言上的 perplexity 会非线性地急剧上升。换句话说，10B 是一个"临界点"——低于这个量，模型几乎无法学会该语言的基本语法和表达模式。

实用建议：如果你的目标语言在初始评估中无法达到 10B token，优先考虑：① 用回译和翻译数据混合扩展到 10B；② 在第二阶段继续用该语言数据做 continual pretraining。两条路结合，效果比单用任何一条都好。

本章小结

多语言预训练不是一个"加上几十种语言的数据就行"的简单问题。从 mC4 到 FineWeb-2，我们看到行业正从"统一管道"走向语言自适应的精细处理。每种语言都需要定制化的清洗策略、恰当的质量阈值和足够的 token 预算。低资源语言更是需要综合运用迁移学习、数据增强和文字归一化等手段来弥补天然的不足。

在下一章中，我们将从"空间的维度"转向"时间的维度"——不再是不同语言之间的配比，而是同一个训练过程中数据出现的顺序如何影响模型学习效果。课程学习 (Curriculum Learning) 和动态数据调度，是一个经常被低估却可以显著提升训练效率的杠杆。
