第9章 多轮对话SFT数据集

上一章我们认识了三个英文 SFT 标杆数据集：FLAN Collection 侧重"做任务"，OpenHermes 2.5 侧重"高质量回答"，Alpaca 侧重"快速验证"。但不管是哪个，它们都有一个共同的特点——以单轮对话为主。一条指令、一个回答，结束了。

然而在真实的使用场景中，用户几乎不会只问一个问题就结束对话。他们会追问、会纠正、会换话题、会让模型回忆之前说过的话。为了让模型能够应对这种真实的多轮交互，我们需要专门的多轮对话 SFT 数据。这一章，我们将重点介绍两个最经典的多轮对话数据集——UltraChat 和 ShareGPT，并深入探讨构建多轮对话数据的特殊挑战。

多轮对话数据：让模型学会「聊天」而不仅是「答题」

单轮对话的局限性在于：它把模型的认知锁定在"一一对应"的问答模式中。模型学到的只是"输入→输出"的映射，而不是"对话上下文→合适的回应"的动态决策能力。多轮对话数据则强迫模型学会几项额外的关键能力：

  • 上下文追踪——理解多轮对话中累积的信息，并能引用之前轮次的内容
  • 话题衔接——在话题之间自然切换，而不是机械地回答问题
  • 指代消解——理解"它"、"那个"、"刚才说的"等代词在对话上下文中的具体指代
  • 角色维持——在多轮交互中保持一致的助人格调和行为模式

你可以这样理解：单轮 SFT 数据集教模型做一个"答题机器"，多轮对话数据集教模型做一个"聊天伙伴"。一个成熟的大模型产品，两者都需要。

UltraChat（清华高质量生成式对话）

UltraChat 是清华大学自然语言处理实验室在 2023 年发布的开源多轮对话数据集。它的特点是：完全由 GPT-4 生成，规模大、质量高、无隐私风险。

核心数据指标：

  • 20 万轮多轮对话，总计约 140 万条消息
  • 每条对话平均 7 轮（即 human-assistant 交替 7 次）
  • 完全由 GPT-4 生成，不包含任何真实用户对话，因此不存在用户隐私泄露的问题
  • 覆盖 30+ 个对话场景，包括日常闲聊、知识问答、写作辅助、编程帮助、创意头脑风暴等

UltraChat 的生成策略非常巧妙。它采用的是"双角色对话生成"方法：用一个 GPT-4 实例扮演"用户"，另一个实例扮演"助手"，让两者进行多轮对话。这种方式不仅生成了高质量的助手回复，也生成了自然的用户提问，解决了多轮对话数据中"用户提问不够真实自然"的常见问题。

UltraChat 的贡献：在 UltraChat 出现之前，开源社区的多轮对话数据要么来自真实用户（有隐私风险），要么数量太少无法支撑有效训练。UltraChat 证明了一条可行的路径——用顶级模型生成数据，既能保证质量又能规避隐私问题。这直接影响了后续一大批合成对话数据集的设计思路。

UltraChat 的数据格式采用 JSONL，每条数据是一个完整的对话历史：

// UltraChat 数据格式示例（简化版）
{
  "id": "uc_001234",
  "system_prompt": "你是一个乐于助人的AI助手。",
  "conversation": [
    {
      "user": "我最近在读《三体》，你觉得里面的黑暗森林法则有道理吗？",
      "assistant": "黑暗森林法则是刘慈欣在《三体》中提出的一个宇宙社会学假说..."
    },
    {
      "user": "那如果把这个法则应用到当今的国际关系中呢？",
      "assistant": "这是一个很有意思的思想实验。虽然国际关系与宇宙文明有本质不同..."
    },
    {
      "user": "明白了。那除了三体，你还有什么科幻小说推荐吗？",
      "assistant": "当然。如果你喜欢硬科幻，《海伯利安》系列和《沙丘》都是经典之作..."
    }
  ]
}
实际使用注意：UltraChat 的数据是纯英文的，虽然有中文翻译版（UltraChat-zh，我们第 11 章会讲），但原版主要用于英文模型训练。另外，虽然 GPT-4 生成的整体质量很高，但部分长对话的后半段会出现"话题漂移"——模型生成的内容逐渐偏离最初的对话主题。在实际使用中建议做一轮质量抽检，筛选掉明显跑题的对话。

ShareGPT（真实用户对话标杆）

如果说 UltraChat 是"实验室环境下的完美对话"，那 ShareGPT 就是"真实世界的野性对话"。ShareGPT 是一个由用户自发分享的 ChatGPT 对话记录的收集项目。用户通过 ShareGPT 浏览器插件，自愿将自己与 ChatGPT 的对话公开分享出来。

ShareGPT 最重要的一次应用是训练 Vicuna——UC Berkeley 在 2023 年发布的一个基于 LLaMA 微调的对话模型。Vicuna 团队收集了大约 70K 条 ShareGPT 对话记录（约 12 万轮对话），进行了格式清洗后用于训练，效果令人惊艳——仅用 7B 参数的 Vicuna-13B（注意命名，实际是 13B 模型）就在人工评估中达到了 GPT-3.5 90% 以上的水平。

ShareGPT 的核心优缺点：

维度 UltraChat ShareGPT（Vicuna 方案）
数据来源 GPT-4 仿真生成 真实用户与 ChatGPT 的对话记录
对话自然度 较高（但仍有"生成感"） 极高（真实用户行为）
对话多样性 30+ 预设场景 无限的真实场景分布
隐私风险 无（完全合成） 较高（含真实用户输入）
数据质量 高质量，GPT-4 级别 参差不齐，需大量清洗
格式统一性 统一格式 需大量格式归一化
使用许可 Apache 2.0 灰色地带（用户分享，非官方）

ShareGPT 数据的一个独特价值在于：它包含了大量真实用户的使用模式——用户如何追问、如何纠正模型的错误、如何在不满意时要求模型重写、以及用户真正关心的话题分布。这些信号是任何合成数据都无法完全复现的。

但它的风险也同样明显。真实用户对话中可能包含个人隐私信息（姓名、地址、邮箱等）、敏感内容或不当请求。使用 ShareGPT 数据进行训练，需要做严格的隐私过滤和内容安全审查。

隐私合规提醒：如果你的项目涉及商用场景，使用 ShareGPT 数据需要格外谨慎。这些对话记录是由用户自愿分享的，但并不意味着它们可以无限制地用于模型训练。在实际工程中，建议优先使用 UltraChat 这类无隐私风险的合成数据作为主力，将 ShareGPT 作为少量补充来增加对话的自然度。

多轮对话数据的特殊挑战

前面介绍了两个优秀的数据集，但它们只是多轮对话数据的"冰山一角"。在实际工程中，构建多轮对话 SFT 数据面临的挑战远比单轮数据复杂。以下是几个最典型的问题：

上下文连贯性保持

多轮对话中，模型需要理解整段对话的上下文，而不仅仅是最后一轮提问。一个常见的问题是：训练数据中的对话轮次太短（比如只有 2-3 轮），导致模型在推理时遇到 10 轮以上的长对话就无法维持上下文。建议在构建数据时，确保有相当比例的长对话（8 轮以上）样本。

角色一致性

在多轮对话中，模型需要始终保持一致的"人格"。如果训练数据中混入了角色不一致的样本（比如同一轮对话中模型一会儿自称"我"，一会儿用"助手"称呼自己），模型就会学混乱。

话题转换的自然度

真实对话中话题转换是自然的——"我们刚才讨论了量子物理，现在换个话题，你觉得 ChatGPT 会取代程序员吗？"但很多生成式对话数据的话题转换非常生硬，像是一个个不相关问题的堆砌。好的多轮对话数据应该包含自然的话题过渡。

多轮格式处理

不同数据集的多轮对话格式千差万别，需要统一为训练框架能处理的格式：

# 多轮对话格式转换示例
# 从 UltraChat 格式 → ShareGPT 格式

def ultrachat_to_sharegpt(ultrachat_item):
    """将 UltraChat 格式转为 ShareGPT 格式"""
    messages = []
    for turn in ultrachat_item["conversation"]:
        messages.append({
            "from": "human",
            "value": turn["user"]
        })
        messages.append({
            "from": "gpt",
            "value": turn["assistant"]
        })
    return {"conversations": messages}

# 从 ShareGPT 格式 → messages 格式（当前主流训练框架）
def sharegpt_to_messages(sharegpt_item):
    """将 ShareGPT 格式转为 HuggingFace messages 格式"""
    role_map = {"human": "user", "gpt": "assistant"}
    return [
        {"role": role_map[turn["from"]], "content": turn["value"]}
        for turn in sharegpt_item["conversations"]
    ]

一个踩坑故事：多轮对话中角色标记错误

数据格式问题听起来是小事，但如果犯错了，后果可能非常严重。这里分享一个我亲眼见过的问题。

2024 年初，一个团队正在用多轮对话数据微调一个 13B 模型。团队从几个数据源收集了约 15 万条多轮对话，其中包含 UltraChat 的子集、ShareGPT 的清洗版和一些自行生成的对话。他们在预处理时写了一个格式转换脚本，将不同格式统一为 ShareGPT 格式。

看起来一切顺利。Loss 曲线完美下降，训练指标一切正常。然后团队开始做人工评估——给模型一些多轮对话让它参与。结果出现了非常诡异的现象：

模型在对话中频繁混淆"用户"和"助手"的角色。有时候模型会替用户回答："你觉得呢？——哦对，我觉得这个方案不错。"——模型先替用户说了"你觉得呢？"，然后又以助手身份回答。有时候模型会自我重复："好的我来解释这个问题……也就是说……那你觉得呢？好的我来解释……"——陷入了角色错乱的死循环。

团队排查了很久才发现问题出在格式转换脚本中。在某个数据源的转换逻辑里，有一行代码将用户和助手的角色标记搞反了：

# 错误代码（搞反了角色）
def convert_role(original_role):
    mapping = {"human": "gpt", "assistant": "human"}  # 反了！
    return mapping.get(original_role, original_role)

# 正确代码
def convert_role(original_role):
    mapping = {"human": "human", "assistant": "gpt"}  # correct
    return mapping.get(original_role, original_role)

这个 bug 导致大约 10% 的训练数据中用户和助手角色互换。模型在 10% 的时间里学到的是"用户应该回答，助手应该提问"。虽然 90% 的数据是正确的，但 10% 的错误信号足以让模型在角色认知上产生混乱——尤其是多轮对话中角色是一个高频出现的模式特征，模型只要学错一点，就会在推理时反复出错。

教训总结：多轮对话数据的格式处理，特别是角色映射，是绝对不能出错的环节。建议的做法是：（1）格式转换后做一个抽样可视化——随机打印 20-30 条转换后的对话，人工确认角色顺序；（2）写一个验证函数，检查每条对话的格式是否满足"human→gpt→human→gpt..."的交替模式；（3）在训练前做一个小的 ablation test，用 1000 条干净数据训练一个 mini 模型，确认角色行为正常后再全量训练。

多轮对话数据是让模型学会"聊天"的关键。UltraChat 给了我们高质量的合成对话模板，ShareGPT 提供了真实对话的多样性和自然度。理解了它们各自的优缺点和那些"坑"，你就能在构建自己的多轮对话数据时少走很多弯路。
