第24章 训练自己的TTS模型

大家好，我是你们的讲师。上一章我们聊了开源TTS生态的全景——项目那么多，工具那么全，但说到底，只有真正动手训练过自己的模型，才算是真正入了TTS的门。纸上得来终觉浅，绝知此事要躬行，这一章我们就来走一遍完整的TTS训练流程。

我会以VITS模型在LJSpeech数据集上的训练为例，按"数据准备 → 配置解析 → 训练监控 → 踩坑诊断 → 推理导出"的顺序，手把手带你走完整个流程。这个流程不限于某一个框架——无论你用的是Coqui TTS还是ESPnet-TTS，核心思路都是通用的。

数据准备 — 从原始录音到可训练的格式

TTS训练的第一步，永远不是写模型，而是整理数据。一个高质量的TTS数据集是成功训练的基石，它的重要性甚至超过模型架构的选择。让我们以最经典的单说话人英文数据集LJSpeech为例来说明：

  • 录音规格：16kHz采样率、16bit量化、单声道（mono）、PCM WAV格式。LJSpeech共计约24小时的录音，由一位女性播音员朗读7本非虚构类书籍的段落。
  • 标注格式：每条录音对应一个文本文件，包含录音文件名和转录文本。典型的格式是CSV或JSON lines：LJ001-0001|He hoped there would be stew for dinner.
  • 数据划分：通常按 95% : 2.5% : 2.5% 划分训练集、验证集、测试集。注意要确保同一个来源的句子不会被拆分到不同集合——否则你的模型可能在"记忆"而不是"学习"。

如果你的数据是自己录制的，有几个关键要点：录音环境要安静（底噪低于-50dBFS），说话人要保持稳定的音色和语速，录音格式统一为16kHz/16bit/mono。不要在一个项目里混用手机录音和专业麦克风录音——不同设备带来的频响差异会让模型非常困惑。

实用工具：SoX（Sound eXchange）是处理音频文件的瑞士军刀。一句 sox input.wav -r 16000 -b 16 -c 1 output.wav 就能完成重采样、位深转换和声道合并。建议在数据预处理管道中统一使用SoX，避免不同库之间的音频编解码差异。

配置文件解析 — 读懂VITS的config.json

在启动训练之前，你需要面对一个绕不开的文件——配置文件。以VITS的config.json为例，我们来逐段读懂它的每个参数在做什么：

{
  "train": {
    "log_interval": 200,       // 每200步打印一次loss
    "eval_interval": 1000,     // 每1000步做一次验证集评估
    "save_interval": 10000,    // 每10000步保存一次checkpoint
    "epochs": 10000,           // 最大训练轮数（实际看loss收敛情况提前停止）
    "batch_size": 16,          // 批次大小，受GPU显存限制
    "learning_rate": 2e-4,     // 初始学习率，Adam优化器的标准起点
    "betas": [0.5, 0.9],       // Adam beta参数，GAN训练中beta1通常设得较低（0.5）
    "weight_decay": 0.0        // 权重衰减，VITS训练中通常不设L2正则化
  },
  "data": {
    "sampling_rate": 22050,    // 采样率（LJSpeech原始是22050Hz，非16kHz）
    "filter_length": 1024,     // STFT窗长，决定了频谱的时间分辨率
    "hop_length": 256,         // STFT帧移，256点对应约11.6ms（22050Hz下）
    "win_length": 1024,        // STFT窗长，通常等于filter_length
    "n_mel_channels": 80,      // 梅尔滤波器组数量，80是TTS领域的标准值
    "mel_fmin": 0.0,           // 梅尔滤波器最低频率
    "mel_fmax": null           // 梅尔滤波器最高频率（null表示使用Nyquist频率）
  },
  "model": {
    "inter_channels": 192,     // flow层的中间通道数
    "hidden_channels": 192,    // Transformer编码器的隐藏维度
    "filter_channels": 768,    // 前馈网络的中间层维度
    "n_heads": 2,              // 多头注意力的头数
    "n_layers": 6,             // Transformer编码器层数
    "kernel_size": 3,          // 卷积核大小
    "p_dropout": 0.1           // Dropout率
  }
}

几个值得特别注意的参数：sampling_rate必须与你的数据集一致，如果数据是16kHz但配置了22050Hz，模型输出的语音音调会完全错乱。batch_size直接影响模型训练的稳定性——VITS作为GAN架构，batch太小判别器学不稳定。如果显存不够（比如只有8GB），可以考虑梯度累积（gradient accumulation）来模拟更大的batch。mel_fmin和mel_fmax在中文场景下建议分别设为0和8000（16kHz采样率）——因为中文的声母（如zh/ch/sh）包含较多高频成分，保留高频信息有助于清晰度。

训练监控：看什么？怎么看？

训练启动后，你的终端开始疯狂滚动日志。新手最容易犯的错误是盯着某一个数字反复焦虑。实际上，你需要关注的指标分成三个层次：

第一层：Loss曲线。VITS的训练有三个主要的loss分量：Generator Loss（生成器损失，反映生成语音的质量）、Discriminator Loss（判别器损失，反映判别器的判别能力）、Feature Matching Loss（特征匹配损失，反映生成语音在判别器中间特征层上与真实语音的接近程度）。其中Generator Loss应该是波动下降的，Discriminator Loss应该在0.5-1.2之间波动——如果Discriminator Loss降到了0附近，说明判别器"太强了"，生成器已经无法骗过它；如果一直维持在1.5以上，说明判别器"太弱了"，无法有效指导生成器学习。

第二层：对齐图（Alignment）。VITS的Monotonic Alignment Search（MAS）模块会输出每一对文本-音频的对齐矩阵。用TensorBoard查看对齐图时，你应该看到一条从左上到右下的清晰的对角线——这说明文本中的每个音素都正确地对应到了音频中的相应位置。如果对齐图是模糊的、断裂的、或者出现横向条纹，说明对齐失败了，需要检查文本前端是否正确、音素集是否完整。

第三层：注意力热图。Transformer编码器的注意力权重也值得定期检查。正常情况下，注意力分布应该集中在对角线上，体现文本序列的自回归特性。如果注意力分布完全发散或者集中在个别位置上，可能意味着模型陷入了"注意力崩溃"——这是Transformer训练TTS时的常见问题，通常可以通过调整学习率或增加warmup步数来解决。

常见踩坑与诊断

我在训练TTS模型的过程中踩过的坑，比我在教程里写的代码还多。下面这个表格总结了几类最常见的问题和解决方案：

现象 可能原因 诊断方法 解决方案
对齐图模糊或断裂 文本前端出错，音素集不匹配 单独测试文本前端，检查音素序列 修复text-to-phoneme映射，统一音素集
输出全是噪声/嘶嘶声 声码器输入范围不对，或数据采样率不匹配 检查mel频谱的值域范围（正常应约在-11到+2之间） 确认采样率一致性，检查mel_fmin/mel_fmax设置
训练loss不下降 学习率过高或过低，batch太小 检查学习率曲线，实验不同batch size 从2e-4开始调lr，确保batch ≥ 8
生成语音"电音感" 频谱中有异常spike，或位深不匹配 可视化mel频谱，检查是否有竖条纹 检查音频位深是否为16bit，重新预处理
Discriminator Loss趋近于0 生成器太弱，判别器过拟合 检查Generator Loss是否异常高 降低判别器学习率，或增加生成器容量
合成语音中间有重复音节 注意力机制出现"打滑"（skipping/looping） 查看注意力热图，检查是否存在对角线跳变 增加位置编码，或调整attention温度参数

这里分享一个真实的"踩坑"经历。我第一次用VITS训练中文TTS时，花了整整三天时间在A100上训练了一个模型——满怀期待地运行推理脚本，结果出来的全是白噪声，一丁点语音都听不出来。我当时的第一反应是模型没有训练好，于是把batch size调大、学习率调小，又训练了三天，还是噪声。最后仔细检查才发现：数据是22050Hz采样率，但配置文件中写的是16000Hz。因为采样率不匹配，模型学到的频谱分布和推理时的声码器输入完全对不上——就像你用英语的发音规则去读一段中文文本，出来的东西当然谁也听不懂。这个教训让我养成了一个习惯：每次开始训练之前，必须跑一遍数据预处理管道来验证每一个中间步骤的输出。

血的教训：不要在训练过程中频繁中断和修改配置！每次修改后模型都需要重新收敛。更好的做法是：先跑一个微型版本（取100条数据），确认整个pipeline正确无误后，再用全量数据跑正式训练。微型训练只需要10-20分钟，可以在你吃午饭的时候跑完——等吃完了确认没问题再启动全量训练，这比训练一天后发现错了再重来高效得多。

从训练到推理

恭喜！经过几天（可能是一周）的等待，训练终于结束了。现在面对一堆checkpoint文件，你可能会想："选loss最低的那个肯定没错吧？"——千万不要这么做。

TTS训练中存在一个普遍现象：验证集loss在训练后期会持续下降，但合成听感却可能越来越差。这是因为GAN架构和声码器的loss函数并不能完美反映人耳听觉感知。我见过很多模型在训练到某个点之后，loss还在降但合成语音已经开始出现"金属声"或"电音感"——这是因为模型在过度拟合训练数据的高频细节，而这些细节在听觉上是有害的。

一个实用的策略是：间隔保存多个checkpoint（比如每5000步保存一个），然后全部跑一次batch的合成，用耳朵听一遍，选出听感最好的那个。通常最佳checkpoint出现在训练中期，而不是末期。更系统的方法是使用我们下一章要讲的MOS评测——从数值和听感两个维度综合选择。

选出最佳checkpoint后，你就可以导出模型了。典型的导出方式有两种：

  • PyTorch原生导出：保存state_dict，推理时加载到模型结构中。适合快速验证和调试。
  • ONNX/TensorRT导出：针对生产环境优化，可以大幅提升推理速度。VITS的ONNX导出需要处理MAS推理时的对齐搜索——大部分框架已经提供了现成的导出脚本。

好了，从数据到模型，我们完整走了一遍TTS的训练之旅。这个过程可能会让你觉得"坑比路多"——但相信我，当你第一次听到自己训练出来的模型说出一句完整、自然的话时，那种成就感是无可替代的。下一章，我们来解决一个更核心的问题：模型训好了，怎么证明它真的"好"？
