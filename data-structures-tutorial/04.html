前言：指针的艺术

大家好，我是你们的讲师。如果说数组是"老实人"——规规矩矩在内存里排好队，那么链表就是"叛逆者"——它不愿意被束缚在连续的内存里。每个元素都带有一个指向下一个元素的指针，你追我赶，串成一条链。

链表的精髓在于：你不需要搬家，只需要重新接线。在数组中插入一个元素，你得把后面的所有元素都往后挪；在链表中插入一个元素，你只需要改两个指针——这就是 O(1) 和 O(n) 的差距。

但同时，链表也有它的软肋：你要找第 5 个元素？对不起，没有捷径，你只能从头开始数。而且，每个多出来的指针也意味着额外的内存开销。这些特性决定了链表在特定场景下无可替代，但在其他场景下又远不如数组。

链表的内存模型

链表中的每个节点（node）包含两部分：数据域和指针域。指针域存储的是下一个节点的内存地址，而不是物理上的"下一个位置"。这意味着链表中的节点在物理内存中可以散落在任何地方——它们通过指针"连接"在一起。

# 链表节点看起来像是这样的（在内存中）：
# [data | next] → [data | next] → [data | next] → None
#  ^                            ^
#  |____ 它们不需要在内存中挨着

和数组相比，链表有两个重要的后果：

  • 无法随机访问——你要找第 i 个节点，必须从头遍历，O(n)
  • 不浪费连续空间——数组即使有空位也要提前分配连续内存，链表可以零碎地使用内存

单链表的实现

我们先来手写一个最简单的单链表。

class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

class LinkedList:
    def __init__(self):
        self.head = None

    def insert_at_head(self, val):
        """头插法：O(1)"""
        new_node = ListNode(val)
        new_node.next = self.head
        self.head = new_node

    def insert_at_tail(self, val):
        """尾插法：O(n) — 因为需要遍历到末尾"""
        if not self.head:
            self.head = ListNode(val)
            return
        cur = self.head
        while cur.next:
            cur = cur.next
        cur.next = ListNode(val)

    def delete(self, val):
        """删除第一个值为 val 的节点：O(n)"""
        if not self.head:
            return
        if self.head.val == val:
            self.head = self.head.next
            return
        cur = self.head
        while cur.next and cur.next.val != val:
            cur = cur.next
        if cur.next:
            cur.next = cur.next.next

    def search(self, val):
        """查找：O(n)"""
        cur = self.head
        while cur:
            if cur.val == val:
                return cur
            cur = cur.next
        return None

注意一个细节：insert_at_tail 是 O(n)——因为每次都要遍历到末尾。如果你频繁需要在尾部插入，可以考虑维护一个 tail 指针，这样尾部插入也能变成 O(1)。

哨兵节点技巧 (Dummy Node)

哨兵节点是链表题中最重要的技巧之一，它的作用是消除对头节点的特殊处理。来看一个例子——LeetCode 206. 反转链表：

# 不使用哨兵节点
def reverse_list(head):
    prev = None
    cur = head
    while cur:
        next_temp = cur.next
        cur.next = prev
        prev = cur
        cur = next_temp
    return prev   # 新的头节点

这个解法已经算简洁了。但有些题目中，你可能需要在头部之前插入节点，或者删除节点时头节点需要特殊处理。这时候 dummy node 就派上用场了：

# 使用哨兵节点的模式
dummy = ListNode(0)
dummy.next = head
# ... 在 dummy 后面做各种操作 ...
return dummy.next   # 不管怎么操作，dummy.next 就是新链表的头

什么时候该用哨兵节点？当你需要修改链表头，或者需要删除节点且被删除的可能是头节点时。比如 LeetCode 82（删除排序链表中的重复元素 II），用 dummy node 可以把头节点的处理统一到循环逻辑中，避免写两套代码。

快慢指针

快慢指针是链表第二大技巧。想象两个人跑步——一个跑得快（每次走两步），一个跑得慢（每次走一步）。它们之间的相对速度差可以帮我们解决很多问题。

判环 (LeetCode 141)

def has_cycle(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow == fast:
            return True
    return False
# 如果链表有环，快慢指针最终会在环中相遇

找环入口 (LeetCode 142)

快慢指针相遇后，把其中一个指针移回头节点，然后两个指针每次都走一步——它们再次相遇的位置就是环的入口。这个推导很有意思（可以画个图），本质是数学证明：快指针走的路程是慢指针的两倍。我们在工作中记住结论就好，但面试中建议把推导过程也说出来。

找链表中点

def middle_node(head):
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
    return slow  # slow 就是中点
# 这个技巧在"对链表进行归并排序"中很关键
🔑 核心规律：链表题中约 80% 的问题可以用快慢指针 + 哨兵节点解决。判环、找中点、找倒数第 k 个节点——这些都是快慢指针的经典应用。如果你面试中遇到链表题，第一反应应该是"这题能不能用快慢指针？能不能用 dummy node？"

链表类型对比

特性 单链表 双向链表 循环链表
每个节点存储 data + next prev + data + next 同单/双向
内存占用 较小 较大（多一个指针） 同对应类型
删除给定节点 O(n) 需找前驱 O(1) 有 prev 指针 同对应类型
遍历终点判断 next == None next == None 回到头节点
典型应用 哈希表拉链 LRU 缓存 约瑟夫问题、轮转调度

双向链表与循环链表

双向链表

双向链表中的每个节点不仅有 next 指针，还有一个 prev 指向前一个节点：

class DoublyListNode:
    def __init__(self, val=0, prev=None, next=None):
        self.val = val
        self.prev = prev
        self.next = next

双向链表的优势在于删除效率更高——如果你有一个节点的引用，单链表无法 O(1) 删除它（因为拿不到前驱节点），但双向链表可以（通过 prev 指针拿到前驱）。

循环链表

循环链表是另一种变体——尾节点的 next 指向头节点，形成一个环。这种结构在以下场景中很实用：

  • 约瑟夫问题：N 个人围成一圈，每隔 M 个人杀掉一个——循环链表可以很好地模拟这个过程
  • 操作系统的任务调度：时间片轮转调度算法通常会使用循环链表
  • 音乐播放器的播放列表：循环播放模式本质上就是一个循环链表

链表的反转——迭代与递归

反转链表是面试中的"Hello World"，但很多人在面试中写到一半会卡住。我们来仔细看两种写法：

# 迭代法（三个指针）
def reverse_iterative(head):
    prev = None
    cur = head
    while cur:
        next_node = cur.next   # 先保存下一个节点
        cur.next = prev        # 指向前一个
        prev = cur             # 前移
        cur = next_node        # 前移
    return prev

# 递归法（更优雅，但更难理解）
def reverse_recursive(head):
    if not head or not head.next:
        return head
    new_head = reverse_recursive(head.next)
    head.next.next = head   # 让下一个节点指向自己
    head.next = None        # 自己的 next 置空
    return new_head

递归写法的关键理解：reverse_recursive(head.next) 返回的是"以 head.next 为头节点的子链表反转后的新头"。这个思路需要多画几张图来消化——不要着急，多画几次就理解了。

💥 踩坑实录：被遗忘的 prev 指针
我自己的一个血泪教训：有一次在实现双向链表的删除操作时，只更新了 cur.prev.next = cur.next，忘了更新 cur.next.prev = cur.prev。结果链表后半部分的 prev 指针全部指向了一个已删除的节点。这个 bug 让我排查了整整两个小时——因为程序没有立刻崩溃，而是在后续的某个操作中才表现出奇怪的行为。教训是：链表的指针操作必须成对出现——你改了几个"正向"指针，就要改几个"反向"指针。写完后逐行检查每一个指针更新，不要想当然。

本章小结

链表的难点不在于代码量（实际上链表题的代码都很短），而在于指针操作的顺序和边界条件。多画图、多练习是掌握链表的唯一路径。记住两个核心技巧：哨兵节点简化头节点处理，快慢指针解决一大类查找问题。
