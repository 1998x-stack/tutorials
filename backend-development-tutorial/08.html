开篇：一个好的API设计能省掉100小时的沟通

我曾经加入过一个团队，他们的 API 风格……怎么说呢，是"自由发挥型"。有人用 GET /getUserInfo，有人用 POST /user_detail，还有人设计了 PUT /delete_product。前端同学每次接入新接口都要花半天来理解这个接口到底做什么、传什么参数、返回什么格式。跨团队协作时更是灾难——每次联调都是"你们这个接口为什么返回这个结构？"这种问题反复拉扯。

这不是技术问题，这是设计问题。一个设计良好的 API，就像一本清晰的使用说明书——看一眼就懂，不用翻文档。而一个设计糟糕的 API，会消耗团队大量的沟通成本，而且是每天都在消耗。在后端开发中，API 是对外暴露的"门面"——你代码写得再漂亮，API 设计得一团糟，别人对你的评价就是"这个后端不行"。

一个好的 API 设计，核心就三个词：一致、直观、自文档。这章我们来拆解 RESTful API 设计中的关键实践。

💡 核心原则：REST 不是一种协议，而是一种架构风格。它利用 HTTP 本身的能力来表达语义——不是把 HTTP 当隧道用（比如所有请求都用 POST），而是让每个请求的方法、URI、状态码都有明确的语义。

资源建模：名词复数、层级关系、避免动词

REST 的核心概念是"资源"（Resource）。资源是名词，不是动词。所以端点应该叫 /users，而不是 /getUsers。这个看起来是小事，但它影响了整个 API 的一致性。

资源建模的几个关键规则：

  • 使用名词复数：/users 而不是 /user，/orders 而不是 /order。复数一致性更好，因为集合操作天然是复数的语义。
  • 层级关系用路径表达：/users/{userId}/orders/{orderId} 表示"某个用户的某个订单"，清晰表达了资源之间的归属关系。
  • 避免动词：不要用 /getUsers、/createOrder。操作语义由 HTTP 方法表达——GET 是查询，POST 是创建，PUT 是全量更新，PATCH 是部分更新，DELETE 是删除。

下面是一个标准的资源建模对照：

HTTP 方法 端点 语义 幂等
GET /users 获取用户列表 是
GET /users/{id} 获取单个用户 是
POST /users 创建新用户 否
PUT /users/{id} 全量替换用户信息 是
PATCH /users/{id} 部分更新用户字段 否
DELETE /users/{id} 删除用户 是
🔧 实战建议：对于"操作"类接口（比如"发送邮件""启动任务"），如果语义上确实不适合用 CRUD 的操作，可以考虑用 POST /emails/send 或 POST /tasks/{id}/start 这种动词+名词的复合形式。REST 不是宗教，不需要死守规则，但一致性比"正确"更重要——团队内部约定好就行。

HTTP 状态码的正确用法

HTTP 状态码不仅仅是"200 表示成功，500 表示服务器挂了"。每个状态码都对应一种具体的语义场景，正确的状态码能让调用方不用看响应体就知道发生了什么。很多团队常年只用 200 和 500，把错误信息塞在响应体的 code 字段里——这其实是在浪费 HTTP 协议本身提供的能力。

下面是后端开发中最常用的状态码及其正确用法：

状态码 含义 典型场景
200 OK 请求成功 GET 查询成功、PUT 更新成功
201 Created 资源创建成功 POST 创建新资源后返回，应包含 Location 头部指向新资源 URI
204 No Content 请求成功但无返回体 DELETE 删除成功、某些 PATCH 操作
400 Bad Request 请求参数不合法 缺少必填字段、参数格式错误、JSON 解析失败
401 Unauthorized 未认证（身份不明） 未提供 token 或 token 过期
403 Forbidden 已认证但无权限 普通用户尝试访问管理员接口
404 Not Found 资源不存在 查询不存在的用户 ID
409 Conflict 资源冲突 创建重复的用户名、乐观锁版本冲突
422 Unprocessable Entity 语义错误 邮箱格式不合法、密码不满足复杂度要求
429 Too Many Requests 请求频率超限 超出 API 速率限制，应包含 Retry-After 头部
500 Internal Server Error 服务器内部错误 未捕获的异常（不应返回详细堆栈信息给客户端）
⚠️ 踩坑故事：一个合作团队把所有错误都返回 200 + {"code": 400, "message": "..."} 的格式。我们的监控系统只检测 HTTP 5xx 状态码来触发告警，结果他们的 API 实际上一直在返回"系统繁忙"的错误，但监控面板上全是绿色——因为 HTTP 状态码是 200。等我们发现问题时，已经影响了整整两周的用户体验。这就是"不用状态码表达语义"的代价。

分页策略：Offset-based vs Cursor-based

几乎所有的列表接口都需要分页。不分页的接口在数据量小的时候没问题，一旦数据量涨到万级以上，一次返回所有数据就是灾难——不仅慢，而且占带宽。分页有两大主流方式：基于偏移量的分页（Offset-based）和基于游标的分页（Cursor-based）。

Offset-based 是传统做法，用 ?page=2&size=20 来控制。优点是实现简单、前端可以实现"跳转到第 N 页"的功能。缺点是当数据库有新增或删除时，不同页面之间可能出现数据重复或遗漏——称为"幻读"。

Cursor-based 用 ?cursor=eyJsYXN0X2lkIjogMTAwMH0=&limit=20 来控制。游标指向一个具体的数据位置，下一页从该位置往后取。优点是数据一致性更好（不受中间插入的数据影响），性能更稳定（不需要 OFFSET 跳过大量行）。缺点是无法跳转到任意页。

两者的实现对比：

-- Offset-based 分页（SQL）
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20 OFFSET 40;
-- 当 OFFSET 很大时（比如第 10000 页），性能急剧下降
-- 因为数据库需要扫描并丢弃前面的行

-- Cursor-based 分页（SQL）
SELECT * FROM orders
WHERE created_at < '2026-06-15T10:00:00Z'
ORDER BY created_at DESC
LIMIT 20;
-- 游标指向最后一条记录的 created_at
-- 性能稳定，因为 WHERE 条件可以利用索引

// Cursor-based 分页的典型响应
{
  "data": [
    { "id": 1234, "title": "订单A", "created_at": "2026-06-15T10:00:00Z" },
    { "id": 1235, "title": "订单B", "created_at": "2026-06-15T09:55:00Z" }
  ],
  "pagination": {
    "next_cursor": "eyJsYXN0X2NyZWF0ZWRfYXQiOiAiMjAyNi0wNi0xNVQwOTo1NTowMFoifQ==",
    "has_more": true
  }
}
🔑 选择建议：实时动态数据（朋友圈、Feed 流、消息列表）用 Cursor-based；稳定数据或者需要跳页的场景（后台管理列表、历史订单查询）用 Offset-based。很多现代 API 实际上两种都支持，由客户端选择。

API 版本管理

API 一旦上线，就会有人依赖它。当你需要修改接口行为时，直接改原接口会破坏现有客户端的兼容性。这时候就需要版本管理。主流的版本管理方式有三种：

URL 路径版本：/v1/users vs /v2/users。最直观、最容易实现、也最常用。缺点是一旦版本号进入 URL，它就成了 URI 的一部分，语义上不太"RESTful"（资源 URI 应该代表资源本身，而不是资源的不同版本）。但在实践中，这是最广泛采用的方式。

Header 版本：通过自定义请求头如 Accept: application/vnd.myapp.v1+json。更"RESTful"，但客户端调用时需要额外设置 Header，调试不方便，浏览器直接访问也更麻烦。

Query 参数版本：/users?version=1。最简单，但版本号容易丢失（比如转发请求时 Query 参数可能被丢弃），也不够标准。

我个人推荐：内部 API 用 URL 路径版本，简单直接；对外公共 API 可以考虑 Header 版本，更符合 REST 规范。不管用哪种方式，关键是不要长期维护多个版本——制定好版本生命周期策略，比如每个版本支持 12 个月，过期前 3 个月发出弃用通知。

📌 版本管理最佳实践：不要在还没有破坏性变更的时候就引入版本号。很多团队在一开始就用 /v1/，但几年后 API 接口从未有过不兼容变更，这个前缀就是冗余的。版本号的正确时机是——当你必须做一个无法向后兼容的变更时。

OpenAPI 3.1 文档自动生成

API 文档最大的问题是——它经常过期。代码改了，文档没更新，这是常态。解决这个问题的核心思路是"文档从代码中生成"。OpenAPI 规范（前身为 Swagger）是这个领域的标准，OpenAPI 3.1 版本已经与 JSON Schema 完全兼容，表达能力更强。

使用 FastAPI（Python）可以自动生成 OpenAPI 3.1 文档，Go 生态中也有 swaggo 等工具。下面是 FastAPI 的示例：

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import Optional

app = FastAPI(title="订单服务", version="2.0.0")

class Order(BaseModel):
    id: int
    user_id: int
    product: str
    amount: float
    status: str  # pending | paid | shipped | cancelled

class CreateOrderRequest(BaseModel):
    user_id: int
    product: str
    amount: float

@app.get("/v1/orders", summary="获取订单列表")
async def list_orders(page: int = 1, size: int = 20):
    """返回分页的订单列表，默认每页20条。"""
    orders = await fetch_orders(page=page, size=size)
    return {"data": orders, "page": page, "size": size}

@app.post("/v1/orders", status_code=201, summary="创建订单")
async def create_order(req: CreateOrderRequest):
    """创建一笔新订单。"""
    order_id = await insert_order(req.dict())
    return {"id": order_id}

@app.get("/v1/orders/{order_id}", summary="获取订单详情")
async def get_order(order_id: int):
    order = await find_order(order_id)
    if order is None:
        raise HTTPException(status_code=404, detail="订单不存在")
    return order

这段代码会自动生成交互式 API 文档（访问 /docs 即可看到 Swagger UI），包括每个端点的请求参数、响应格式、状态码、甚至示例值。前端同学可以直接在浏览器里调试接口，不需要你再手动写文档了。

在这个基础上，你还可以接入 openapi-typescript 这样的工具，从 OpenAPI 规范自动生成前端 TypeScript 类型定义——这样前后端的类型就完全同步了，接口变了，TypeScript 编译就会报错。这才是真正的"自文档 API"。
