第10章 静态时序分析(STA)

大家好，我是你们的讲师。上一章我们学习了逻辑综合——RTL代码被转换成了门级网表。但综合出来的电路能跑到多快？会不会在某些条件下失效？这些问题单靠综合工具是无法回答的。这时就需要静态时序分析（Static Timing Analysis, STA）登场了。

在芯片设计的所有环节中，STA可能最让初学者感到"头大"。它的数学看起来很简单——无非是加法减法、max/min运算——但它的复杂性在于：你需要处理成千上万条路径、几十个工艺角、以及大量相互交织的约束条件。我见过很多设计师，RTL写得很好，但一看到PrimeTime的timing report就头皮发麻。这一章的目的，就是帮大家彻底搞懂STA——从基本概念到实战中常见的"坑"，一次性讲清楚。

10.1 时序弧与路径

STA的基本单位是时序弧（Timing Arc）。时序弧分为两类：

  • 单元时序弧（Cell Arc）：标准单元内部从输入引脚到输出引脚的延迟。一个NAND2单元有两条上升弧（A→Z, B→Z）和两条下降弧。
  • 连线时序弧（Net Arc）：单元输出引脚到下一级单元输入引脚之间的互连延迟。

一条完整的时序路径（Timing Path）由一串时序弧组成，从起点（输入端口或触发器的CK引脚）到终点（输出端口或触发器的D引脚）。同一条路径可能有不同的"sense"（上升沿触发、下降沿触发、上升沿输出等），STA工具需要分别分析。

在所有路径中，延迟最大的那条被称为关键路径（Critical Path）——它决定了芯片的最高工作频率。设计师的大部分时序优化工作就是围绕关键路径展开的。

业界最常用的STA工具是Synopsys PrimeTime（PT）。它的基本使用流程是：读入门级网表、读入SDC约束、读入寄生参数（SPEF）、设置工艺角、运行分析。PT是目前签核（signoff）级别的标准工具——在流片之前，所有时序报告必须由PT产生并确认。

10.2 Setup/Hold分析

数字电路时序的核心是setup和hold两个检查。让我们从数学上彻底搞懂它们。

Setup检查——确保数据在时钟沿到来之前已经稳定：

Tclk ≥ Tcq + Tcomb + Tsetup + Tskew

其中：Tclk是时钟周期，Tcq是触发器的时钟到输出延迟，Tcomb是组合逻辑延迟，Tsetup是触发器的建立时间要求，Tskew是时钟偏斜（从起点到终点触发器的时钟延迟差）。

如果这个不等式不满足，数据就没能在时钟沿之前准备好——setup violation。

Hold检查——确保数据在时钟沿之后保持足够长的时间不变：

Thold ≤ Tcq_min + Tcomb_min - Tskew

这里用的是"最小值"——因为在最快条件下（FF角、低温、高压），路径延迟最短，数据可能变化得太快，把上一拍的数据"冲掉"。hold violation通常发生在路径中组合逻辑太少或没有组合逻辑时——比如两个触发器直接串联（a direct connection）。

一个典型的timing report看起来是这样的：

*******************************************************************************
  Path Type:          Setup (max at worst corner)
  Startpoint:         core/u_fft/u_stage1/data_reg_reg_7_ (rising edge-triggered flip-flop)
  Endpoint:           core/u_fft/u_stage2/out_reg_reg_3_ (rising edge-triggered flip-flop)
  Required Clock:     core_clk (period = 2.000 ns)
*******************************************************************************
  Clock Path (launch clock):
    core_clk (rise) @ 0.000 ns
    + net delay      0.086 ns  (core_clk -> u_fft/u_stage1/data_reg_reg_7_/CK)
    = arrival at CK  0.086 ns

  Data Path:
    CK->Q delay      0.145 ns  (u_fft/u_stage1/data_reg_reg_7_/CK -> Q)
    + net delay      0.032 ns  (Q -> n432)
    + cell delay     0.211 ns  (n432 -> n567, AOI22_2)
    + cell delay     0.175 ns  (n567 -> n601, NAND2_4)
    + net delay      0.041 ns  (n601 -> D)
    = data arrival   0.604 ns

  Clock Path (capture clock):
    core_clk (rise) @ 2.000 ns (period 2.0ns, capture edge)
    + net delay      0.112 ns  (core_clk -> u_fft/u_stage2/out_reg_reg_3_/CK)
    = arrival at CK  2.112 ns
    - setup time     0.032 ns  (required setup for this flip-flop)
    = required time  2.080 ns

  Slack:
    required time    2.080 ns
    - data arrival   0.604 ns
    = slack          1.476 ns  (MET)
*******************************************************************************

这个报告中slack为+1.476ns，远超0——说明这条路径非常"富裕"，甚至可以考虑降电压或缩小管子尺寸来省功耗。

10.3 时钟不确定性

时钟不是完美的。在实际芯片中，时钟信号存在各种不确定性——STA中统称为Clock Uncertainty。

  • Jitter（时钟抖动）：时钟沿在时间上的随机变化。包括周期抖动（cycle-to-cycle jitter）和长期抖动（long-term jitter）。Jitter由PLL的噪声、电源噪声、热噪声引起。典型值：对于20MHz的晶振，jitter约为几十ps；对于GHz级的PLL输出，jitter约为几个ps到几十ps。
  • Skew（时钟偏斜）：同一时钟信号到达不同触发器的延迟差异。由时钟树的非对称性、工艺偏差、温度梯度引起。Skew可以是正skew（终点触发器比起点晚收到时钟——对setup有利，对hold不利）或负skew（起点触发器比终点晚收到时钟——对hold有利，对setup不利）。

STA中通过set_clock_uncertainty命令设置这些裕量：

set_clock_uncertainty -setup 0.15 [get_clocks core_clk]
set_clock_uncertainty -hold  0.05 [get_clocks core_clk]

setup uncertainty通常取jitter + 一些margin（约0.1-0.2ns），hold uncertainty取jitter的一半左右（因为jitter对hold影响较小）加上clock skew的不确定性。

10.4 OCV与AOCV

在先进工艺节点（28nm及以下），芯片不同区域的工艺参数会有差异——同一片晶圆上，左边的晶体管和右边的晶体管速度可能不同。这就是On-Chip Variation（OCV）。

传统的OCV处理方法很简单：给所有路径加一个统一的derate因子。例如在setup分析中，数据路径乘以1.1（变慢10%），时钟路径乘以0.9（变快10%），人为制造一个"最差情况"来分析。

# OCV derating in PrimeTime
set_timing_derate -early 0.9  [all_clocks]   # 时钟路径偏快
set_timing_derate -late  1.1  [get_cells *]  # 数据路径偏慢

但到了16nm/7nm/5nm节点，这种"一刀切"的derate过于悲观了——它会导致大量伪违例（false violaton），迫使设计师花费大量时间做不必要的修复。于是出现了更精细的方法：

  • AOCV（Advanced OCV）：考虑路径深度（stage-based）和位置信息（location-based）。路径越深，随机变化越趋于平均，derate可以相应减小。
  • POCV/SOCV（Parametric/Statistical OCV）：把每个单元的延迟视作一个统计分布（高斯分布），用统计方法计算路径延迟的概率分布。PrimeTime的POCV分析会输出一个"sigma"值——表示这条路径在多大概率下会违例。

不同工艺节点的典型derate值参考：

工艺节点 OCV方法 典型setup derate (late path) 典型hold derate (early path)
180nm - 65nm Flat OCV 1.02 - 1.05 0.95 - 0.98
40nm - 28nm Flat OCV 1.08 - 1.12 0.88 - 0.92
16nm - 12nm AOCV 1.05 - 1.08 (depth-dependent) 0.92 - 0.95 (depth-dependent)
7nm - 5nm POCV/SOCV 统计分布, sigma-based 统计分布, sigma-based
3nm POCV/SOCV 统计分布, 3-sigma signoff 统计分布, 3-sigma signoff
一个真实的硅片失效故事：我参与的一个16nm芯片项目，在STA阶段使用了1.1/0.9的flat OCV derate。所有setup和hold都通过了，芯片流片后功能测试也通过了。但在量产测试中，大约2%的芯片在低温（-20°C）条件下出现了随机的hold violation——表现为寄存器偶尔捕获到错误数据。最终定位到的原因是：floorplan中一条长时钟走线跨越了芯片上温度和工艺偏差最大的区域，导致clock skew远大于STA模型中的估计。修复方案是增加hold buffer，但芯片已经流片了——只能通过筛选（binning）把不合格的芯片降级到低温不工作的低端市场。这次教训之后，我们的STA流程中加入了location-based OCV分析和更激进的hold margin。

10.5 时序ECO与收敛

当STA发现时序违例后，需要进行时序ECO（Engineering Change Order）——即在门级网表上做最小范围的修改来修复时序，而不改变RTL代码。这通常发生在芯片设计流程的后期，此时RTL已经冻结，只允许在网表层面做改动。

常见的ECO修复手段：

  • Cell Swap（单元替换）：将标准单元替换为驱动能力更强（更大尺寸）或阈值电压更低（LVT/SVT）的版本。这是最快、成本最低的修复方式。
  • Buffer Insertion（插入缓冲器）：在长互连线中插入buffer来减少延迟（注意：增加buffer可能会增加延迟——在RC-dominated路径中，插入buffer把长线分成两段可以减少整体延迟）。
  • Size-up / Size-down：增大cell尺寸改善setup，减小cell尺寸改善hold。
  • Useful Skew：有意识地调整时钟到达时间，使关键路径上的终点触发器稍晚收到时钟（对setup有利）。但需要非常谨慎——调整一个触发器的skew会影响所有以它为起点的路径。
  • Hold Fixing：在短路径中插入delay cells或buffer chain来增加延迟，满足hold要求。
一个实用建议：在项目初期，就应该预留一定的"timing margin"——比如你的芯片目标频率是1GHz，那综合时应该用1.1GHz（周期从1.0ns压到0.9ns）来做，给物理设计阶段留出10%的退化空间。物理设计（布局布线）通常会比综合阶段的时序退化10-30%。如果你综合时已经"擦边"（slack刚刚好），到了物理设计阶段几乎一定会出现违例。

好了，静态时序分析的知识就先讲到这里。STA是一个"实践出真知"的领域——光看公式和理论是不够的，一定要多读timing report、多跑PrimeTime。下一章我们将进入另一个同样重要但方向不同的领域——功能验证方法学。如果说STA是回答"芯片跑得多快"，那功能验证就是回答"芯片的功能对不对"。敬请期待。
