第10章 训练稳定性与优化器

上一章我们讲了数据工程——花了大量篇幅说明"数据质量决定了模型能力的上限"。但数据再好，如果训练的优化器设置不当、学习率调度不合理、或者训练过程中出现了loss spike导致梯度爆炸，那模型可能根本无法收敛到好的结果。

训练一个LLaMA这样的大模型，就如同驾驶一艘巨型油轮穿越风暴海域。优化器是舵——它决定方向调整的幅度和时机。学习率调度是航线图——它告诉你什么时候该加速、什么时候该减速。而Gradient clipping是安全护栏——当风暴（loss spike）来袭时，它能防止船体被巨浪打翻。

这一章我们就来详细解析LLaMA系列在训练稳定性上的所有关键技术。

10.1 AdamW优化器配置

LLaMA系列使用AdamW作为默认优化器。AdamW是Adam优化器的改进版，核心区别是将weight decay与自适应学习率解耦。这一点非常重要——在Adam中，weight decay被混入了自适应学习率的计算中，导致实际施加的weight decay效果在不同参数上不一致。AdamW通过把weight decay单独拿出来、在参数更新后直接施加L2正则，解决了这个问题。

LLaMA的AdamW配置如下：

超参数 LLaMA 1 LLaMA 2 LLaMA 3 典型取值范围
β₁（一阶动量衰减） 0.9 0.9 0.9 0.85-0.95
β₂（二阶动量衰减） 0.95 0.95 0.95 0.90-0.999
ε（数值稳定项） 1e-5 1e-5 1e-5 1e-6 ~ 1e-8
Weight Decay 0.1 0.1 0.1 0.01-0.1
初始学习率 1.5e-4 ~ 3e-4 1.5e-4 ~ 3e-4 1.0e-4 ~ 3e-4 视模型规模而定

这里有几个值得注意的细节：

β₂ = 0.95 而非默认的 0.999。 标准的Adam优化器β₂默认值是0.999，这意味着二阶动量衰减得非常慢——对历史梯度的平方项有一个很长的记忆。但在大模型训练中，梯度分布变化较快，设置较小的β₂（0.95）可以让优化器更快速地适应当前梯度变化，对loss spike的响应更灵敏。这个改动据传来自训练GPT-3的经验。

ε = 1e-5 而非默认的 1e-8。 ε的作用是防止除以零，但更大的ε值实际上起到了"梯度预裁剪"的效果——它让自适应学习率对极小梯度值不那么敏感。在实践中，1e-5的ε值对于数值稳定性有一定的帮助，尤其是在使用bfloat16混合精度训练时。

10.2 学习率调度策略

LLaMA使用Cosine Decay with Warmup作为学习率调度策略。这个策略分为两个阶段：

第一阶段：Warmup（预热）。 训练开始时，学习率从0线性增长到峰值。这个过程通常需要2000-5000步。为什么需要warmup？因为在训练刚开始时，模型参数是随机初始化的，梯度信号噪声很大。如果一开始就用大人学习率，优化器很容易"一步踩空"——梯度更新方向偏离正确方向太远，导致无法恢复。Warmup给了优化器一个"热身"阶段，让参数逐渐适应数据的统计特性。

第二阶段：Cosine Decay（余弦衰减）。 学习率达到峰值后，按余弦曲线逐渐衰减到接近0。余弦衰减相比于线性衰减的优势在于：它在中间阶段保持较高学习率的时间更长，而在最后阶段平滑地降低到接近0。这有利于模型在训练后期精细地优化参数。

# LLaMA风格的学习率调度 (PyTorch伪代码)

import math
import torch.optim as optim

def get_llama_scheduler(optimizer, warmup_steps, total_steps, peak_lr, min_lr=0):
    def lr_lambda(current_step):
        # 1. Warmup阶段: 从0线性增长到peak_lr
        if current_step < warmup_steps:
            return float(current_step) / float(max(1, warmup_steps))

        # 2. Cosine Decay阶段: 从1下降到min_lr/peak_lr
        progress = float(current_step - warmup_steps) / float(
            max(1, total_steps - warmup_steps)
        )
        cosine_decay = 0.5 * (1.0 + math.cos(math.pi * progress))

        # 从peak_lr衰减到min_lr
        # 实际lr = min_lr + (peak_lr - min_lr) * cosine_decay
        return cosine_decay  # 返回系数，实际lr由外部乘peak_lr决定

    return torch.optim.lr_scheduler.LambdaLR(optimizer, lr_lambda)

# 使用示例
optimizer = optim.AdamW(model.parameters(), lr=3e-4, betas=(0.9, 0.95), eps=1e-5)
scheduler = get_llama_scheduler(
    optimizer,
    warmup_steps=2000,
    total_steps=500000,
    peak_lr=3e-4,
    min_lr=3e-5  # 通常衰减到峰值学习率的10%
)
实践技巧： 学习率调度的"最终值"不是0，通常是峰值学习率的10%左右。原因是：如果学习率降到0，模型在训练的最后阶段几乎不会更新，可能错过一些参数空间的优秀区域。保留一个小学习率（如3e-5 = 3e-4 × 10%），让模型在最后阶段仍然有细微的调整能力。

10.3 Weight Decay策略

LLaMA的weight decay为0.1——这个数值比其他常见的设置（0.01）要大一倍。更大的weight decay意味着更强的正则化效果，有助于防止过拟合。但在实践中，并不是所有参数都施加相同的weight decay。

LLaMA团队采用了分层的weight decay策略：

  • Attention层的weight矩阵： 施加完整的weight decay（0.1）
  • FFN层的weight矩阵： 施加完整的weight decay（0.1）
  • LayerNorm（RMSNorm）参数： 不施加weight decay。因为归一化层的参数通常只有scale（γ），施加正则化反而可能影响归一化效果
  • Embedding层： 通常施加较小的weight decay或完全不施加。词嵌入矩阵的特殊性在于它是"查表"操作，正则化它的参数反而可能弱化词汇区分度
  • 偏置项（bias）： 不施加weight decay。偏置项通常是标量，正则化它们没有意义
一个常见的误解： 很多人以为weight decay就是简单地"对所有参数施加0.1的L2正则"。但实际上，正确的做法是对参数分组——把需要weight decay的权重和不需weight decay的参数（norm、bias、embedding）分开。在PyTorch中，你需要这样配置：

optimizer = AdamW([{'params': decay_params, 'weight_decay': 0.1},
                      {'params': no_decay_params, 'weight_decay': 0.0}],
                      lr=3e-4, betas=(0.9, 0.95))

不分组的话，embedding和norm参数会被施加不恰当的weight decay，导致训练不稳定。

10.4 Gradient Clipping与Loss Spike处理

在大模型训练中，loss spike（损失值突然飙升）是训练稳定性的头号威胁。一个典型的loss spike场景：训练平稳进行了几万步，loss平缓下降，突然在某一步loss从3.0跳到了50.0以上，然后模型彻底发散——这就是"灾难性发散"。

LLaMA使用Gradient Clipping作为第一道防线。具体来说，将全局梯度的L2范数裁剪到max_grad_norm=1.0。也就是说，如果梯度的范数超过1.0，就按比例缩放回1.0。这个操作保留了梯度的方向，但限制了更新的幅度。

# Gradient Clipping 实现 (PyTorch)
import torch.nn.utils as clip

# 在每步backward()之后、optimizer.step()之前调用
clip.clip_grad_norm_(
    model.parameters(),
    max_norm=1.0,      # LLaMA标准设置
    norm_type=2.0      # L2范数
)

# optimizer.step() 继续执行

但Gradient Clipping并不是万能的。当loss spike特别严重时（比如loss从3跳到了100以上），即使裁剪了梯度，模型的参数也可能已经被严重破坏。这时候就需要更激进的手段——从checkpoint回滚并跳过异常batch。

LLaMA的训练脚本会定期保存checkpoint（通常是每1000-2000步）。当检测到loss spike时，训练框架会自动回滚到最近的checkpoint，跳过导致spike的那个batch的数据，然后用更小的学习率重新训练。这个操作是自动化的——LLaMA训练系统中有一个专门的"loss monitor"进程在后台监控loss的变化。

踩坑实录： 在一次大规模的分布式训练中，我们遇到了一个非常隐蔽的loss spike问题。模型在训练了5万步后突然在所有数据并行节点上同时出现loss spike，而且回滚后仍然在同一个训练步数附近再次spike。排查了整整一周后才发现问题根本不在优化器或模型——而是数据pipeline中的shuffle buffer设置不当。我们的数据预处理有一个bug：在某些数据shard的边界，shuffle buffer会被清空，导致模型连续看到了大量相同主题的数据（全是某一类特定文档），引发了数据分布的局部不平衡。修复方式是确保shuffle buffer在shard边界处部分保留，而不是完全清空。很多loss spike的根因不在模型或优化器，而在数据处理。

10.5 混合精度训练与bfloat16

LLaMA的训练全程使用bfloat16（BF16）混合精度，这是与许多其他模型使用float16（FP16）的一个重要区别。

bfloat16是Google Brain团队开发的一种16位浮点格式，它的指数位有8位（和float32相同），尾数位只有7位。而标准的float16（FP16）指数位5位，尾数位10位。这意味着：

  • BF16的优势： 动态范围与float32相同（因为指数位相同），可以表示非常大和非常小的数值。这使得BF16在训练中很少出现"上溢出"或"下溢出"的问题。
  • BF16的劣势： 精度较低（7位尾数 vs FP16的10位尾数），但实验表明这对于深度学习模型的训练来说通常不是问题。

为什么LLaMA选择BF16而不是FP16？核心原因是BF16的数值稳定性更好。FP16在训练大模型时容易出现梯度下溢出——当梯度值太小（小于约6e-8）时，FP16直接变为0，导致参数更新完全失效。为了解决这个问题，FP16混合精度训练需要额外的"loss scaling"技术，手动放大loss后再反传梯度，实现了相当于在FP16中表示小数值的能力。而BF16不需要这些花哨技巧——因为它可以直接表示到2^-126这么小的数值。

一个你可能不知道的细节： LLaMA在训练中使用了master weights技术。即使前向和反向传播使用BF16，优化器（AdamW）维护的参数状态（一阶动量、二阶动量）以及最终Parameter的更新都是在float32精度下完成的。具体做法是：模型参数同时保留BF16副本（用于前向/反向）和FP32副本（用于优化器更新），每步更新时，FP32参数更新后复制到BF16参数中。这种做法虽然增加了一倍的参数内存开销，但能显著提升训练的数值稳定性和最终模型质量。对于LLaMA 3 405B来说，master weights意味着额外的405B × 4字节 = 1.6GB的FP32参数存储——相比模型总内存消耗，这个代价是值得的。

10.6 Batch Size动态调整

LLaMA在训练过程中使用了一种batch size warmup技术——从较小的batch size开始，逐步增加到目标值。这个做法有以下考量：

  • 训练初期使用小batch： 模型参数刚开始收敛时，梯度噪声大。小batch size提供了更多的参数更新次数，帮助模型更快地逃离初始化时的"坏区域"。
  • 训练后期使用大batch： 模型接近收敛时，梯度一致性增强，大batch size可以更精确地估计梯度方向，同时提高训练吞吐。

LLaMA 1的具体实现是：在LLaMA 1 65B的训练中，batch size从初始的512 sequences（约0.5M tokens）逐步增加到最终的4M tokens（约4K sequences）。这个warmup过程持续了约5000步。

Batch size的动态调整在工程实现上需要注意——改变batch size意味着需要调整学习率（因为batch size和learning rate之间存在联动关系）。LLaMA的做法是保持token-level learning rate不变——即不管batch size怎么变，模型每处理一个token所对应的学习率更新步长是恒定的。这通过"scaling rule"实现：lr_effective = lr_base × sqrt(batch_size / batch_size_base)。

10.7 LLaMA 3大规模训练中的稳定性挑战

LLaMA 3的训练规模（15.6T tokens、405B参数、128K上下文）给稳定性带来了全新的挑战。Meta在论文中披露了几个特别的稳定性问题：

阶段性Loss Spike。 LLaMA 3的405B模型在训练过程中经历了多次阶段性的loss spike，这些spike呈现周期性特征——大约每训练一定的数据量就会发生一次。经过分析，Meta团队发现这些周期性spike与数据pipeline中特定类型数据的密集出现有关。他们的解决方案是进一步优化了数据shuffling策略，确保不同类型的数据更均匀地分布在整个训练过程中。

MoE负载均衡。 LLaMA 4虽然还没有正式发布，但LLaMA 3的论文中提到了他们为MoE架构所做的实验。在这些实验中，他们发现MoE的专家负载不均衡是一个主要的稳定性挑战——某些专家（expert）会被分配到远超平均水平的tokens，导致这些专家的参数更新过于激进，而其他专家的利用率严重不足。他们采用了辅助损失（auxiliary loss）来惩罚负载不均衡，但这个辅助损失的权重需要仔细调节——太大则影响主任务loss，太小则无法有效均衡负载。

# MoE负载均衡辅助损失的简化实现
def load_balancing_loss(gate_logits, num_experts, top_k=2):
    """
    Args:
        gate_logits: (batch_size, seq_len, num_experts)
        num_experts: 专家总数
        top_k: 每个token选择的专家数
    """
    # 1. 计算每个专家的路由概率
    routing_probs = F.softmax(gate_logits, dim=-1)
    # avg_probs: (num_experts,) 每个专家被选中的平均概率
    avg_probs = routing_probs.mean(dim=(0, 1))

    # 2. 计算每个专家实际处理的token比例
    _, topk_indices = torch.topk(routing_probs, k=top_k, dim=-1)
    # 计算各专家的频率
    expert_freq = torch.zeros(num_experts, device=gate_logits.device)
    for i in range(num_experts):
        expert_freq[i] = (topk_indices == i).sum().float()
    expert_freq /= expert_freq.sum()

    # 3. 负载均衡损失 = num_experts * sum(freq * prob)
    # 当分布完全均匀时，该损失最小
    balance_loss = num_experts * (expert_freq * avg_probs).sum()
    return balance_loss  # 乘以系数alpha后加入总loss
前沿动态： LLaMA 3的训练稳定技术已经成为开源社区的标准参考。比如，现在很多开源LLM训练框架（如Megatron-LM、DeepSpeed）都直接内置了LLaMA风格的训练配置模板。如果你正在搭建自己的训练pipeline，直接复用这些模板会比自己从零调参要靠谱得多。当然，理解每个参数的含义仍然是必要的——否则你无法在训练出现问题时进行有效的调试。

训练稳定性不是一个可以"一次调好、永久使用"的配置。它需要在训练过程中持续监控、分析和调整。Meta的LLaMA团队在论文中提到，他们的训练团队专门有一个"稳定值班"小组，24小时监控训练状态，一有异常立即干预。虽然对于我们个人开发者来说，不太可能组建这样的团队，但自动化监控和告警系统是必须的——至少要做到"检测到loss spike后自动保存checkpoint并发送告警"这种程度。
