第21章 模型与流水线并行

2018年，Google发布了BERT-Large，3.4亿参数——一块16GB的GPU勉强装得下。2020年，GPT-3横空出世，1750亿参数——需要350块A100 80GB才能把参数全部载入显存。而到了2024-2025年，万亿参数级别的MoE模型已经不再罕见。一个冷峻的事实摆在面前：你的模型太大了，单卡放不下了。

这时候数据并行也无能为力——因为数据并行的前提是每张卡上有一份完整的模型副本。如果模型本身装不进一张卡，数据并行就成了无米之炊。解决方案就是模型并行（Model Parallelism）：把模型本身切开，分布到多张GPU上，每张卡只负责模型的一部分。

这一章我们来拆解两种核心的模型并行策略——张量并行和流水线并行，以及它们如何组合成现代大模型训练的"3D并行"方案。

21.1 张量并行（Tensor Parallelism）

张量并行的思路是：把矩阵运算本身切分到多个设备上。一个线性层 Y = XW，如果W太大放不进单卡，就把W按行或按列切开。

列并行（Column Parallelism）

把权重矩阵W沿列方向切分：

W = [W₁, W₂]（将W切成两块）

前向计算：
  GPU 1: Z₁ = X · W₁
  GPU 2: Z₂ = X · W₂
  然后 AllReduce: Z = Z₁ ⊕ Z₂（拼接）

反向传播：
  GPU 1: ∂L/∂W₁ = Xᵀ · ∂L/∂Z₁
  GPU 2: ∂L/∂W₂ = Xᵀ · ∂L/∂Z₂
  每个GPU独立更新自己那部分W

行并行（Row Parallelism）

把权重矩阵W沿行方向切分：

W = [W₁; W₂]（将W按行切成上下两块）

前向计算：
  GPU 1: Z₁ = X₁ · W₁（其中X₁是X的前一半列）
  GPU 2: Z₂ = X₂ · W₂（其中X₂是X的后一半列）
  然后 AllReduce: Z = Z₁ + Z₂（求和）

Megatron-LM（NVIDIA的模型并行框架）的核心洞察是：将列并行和行并行组合起来，在每个Transformer层内实现完整的张量并行。

数学上，一个Transformer层的FFN（前馈网络）部分：

标准形式：Y = GeLU(X · W₁) · W₂

张量并行形式：
  步骤1（列并行）：Z = GeLU(AllReduce(X · [W₁₁, W₁₂]))  ← 按列切W₁
  步骤2（行并行）：Y = AllReduce(Z · [W₂₁; W₂₂])         ← 按行切W₂

每个GPU持有W₁的一列和W₂的一行，通信只发生在两个AllReduce点
关键优化：Megatron-LM在每个Transformer层只做两次AllReduce（一次在列并行输出，一次在行并行输出），而不是每步都通信。这使得张量并行在节点内（通过NVLink）的效率非常高。实际训练中，8卡A100节点内使用张量并行几乎不产生额外开销。

21.2 流水线并行（Pipeline Parallelism）

如果说张量并行是"横着切"——把一层切开，那么流水线并行就是"竖着切"——按层在模型深度方向切分。GPU 0负责1-10层，GPU 1负责11-20层，依此类推。

流水线并行的朴素实现有一个严重问题：同一时刻只有一个GPU在工作。GPU 0算完第1-10层才能传给GPU 1，这期间的GPU 1完全空闲。这个空闲比例就是流水线气泡（Pipeline Bubble）。

GPipe：F-then-B调度

GPipe（Google, 2019）提出了micro-batch的概念来缓解气泡问题：

  • 把一个大的mini-batch拆分成M个micro-batch
  • 每个micro-batch独立做前向+反向
  • 所有micro-batch的反向完成后，统一更新参数
GPipe F-then-B调度（4个micro-batch，4个stage）：

时间 →
Stage 0: [F1][F2][F3][F4]          [B4][B3][B2][B1]
Stage 1:     [F1][F2][F3][F4]      [B4][B3][B2][B1]
Stage 2:         [F1][F2][F3][F4]  [B4][B3][B2][B1]
Stage 3:             [F1][F2][F3]  [F4][B4][B3][B2][B1]

F = 前向，B = 反向
气泡区域：stage 3在F4之前只有3个F的空闲
气泡比例 ≈ (P-1) / (M)  （P=流水线stage数，M=micro-batch数）

GPipe的气泡比例为(P-1)/M。当M远大于P时，气泡可以忽略不计。但M增大意味着需要更多显存来保存中间激活值——这是一个典型的"以空间换时间"的权衡。

21.3 1F1B调度：进一步减少显存

GPipe的F-then-B调度把所有micro-batch的前向计算完成后才做反向，这意味着需要保存M个micro-batch的中间激活值。对于大模型，这常常导致显存爆炸。

1F1B（One-Forward-One-Backward）调度改变了策略：前向和反向交替执行，一旦某个micro-batch的前向完成就立即开始它的反向，释放激活内存。

1F1B调度（4个micro-batch，4个stage）：

时间 →
Stage 0: [F1][F2][F3][F4][B4][B3][B2][B1]
Stage 1:     [F1][F2][F3][B3][F4][B4][B1][B2]
Stage 2:         [F1][F2][B2][F3][B3][F4][B4][B1]
Stage 3:             [F1][B1][F2][B2][F3][B3][F4][B4]

关键差异：
  - GPipe需要同时保存M个micro-batch的激活值 → 内存峰值 = O(M)
  - 1F1B最多同时保存的激活值数量 = P（流水线深度）
  - 节省：约50%的激活内存（当M ≈ 2P时）
1F1B的显存优势：在GPT-3规模的训练中，使用1F1B调度可以将激活显存从GPipe的约240GB降低到约120GB。这使得在同样数量的GPU上可以训练更大的模型，或者使用更大的micro-batch size来提高吞吐量。

21.4 三种并行的混合：3D并行

现代大模型训练中最强大的武器是3D并行——同时使用数据并行、张量并行和流水线并行。

以GPT-3的训练配置为例（NVIDIA Megatron-LM + DeepSpeed）：

GPT-3 175B训练配置：
  总GPU数：1024块A100 80GB

  维度1：张量并行（TP=8）
    - 8块GPU通过NVLink组成一个"计算组"
    - 组内做张量并行，通信通过高速NVLink

  维度2：流水线并行（PP=16）
    - 每8块GPU（一个TP组）作为一个流水线stage
    - 共16个stage，组间通过InfiniBand通信

  维度3：数据并行（DP=8）
    - 16个流水线stage × 8个TP组 = 128组
    - 每组内做数据并行（DP=8），相当于batch size放大8倍

  总GPU = TP × PP × DP = 8 × 16 × 8 = 1024

通信拓扑

并行维度 通信模式 通信组大小 硬件 通信频率
张量并行（TP） AllReduce 8 NVLink（600GB/s） 每层2次
流水线并行（PP） P2P Send/Recv 相邻stage InfiniBand（400-800Gbps） 每micro-batch 2次
数据并行（DP） AllReduce 8 InfiniBand（400-800Gbps） 每轮一次
3D并行的配置原则：通信最密集的维度（TP）放在最快的硬件上（NVLink），通信量中等的维度（DP）放在InfiniBand上，通信量最少但延迟敏感的维度（PP）也放在InfiniBand上。这种"通信密度×带宽"的匹配关系是3D并行性能优化的核心。

21.5 并行策略选择指南

面对不同的模型大小、GPU数量和网络带宽，如何选择最优的并行策略？以下是一个实用的决策参考：

模型参数 GPU数量 GPU间互联 推荐策略 预期加速
< 1B 1-8 任意 单卡/数据并行 线性
1B-10B 8-64 NVLink DP + TP=2或4 近线性
10B-100B 64-256 InfiniBand TP=8 + PP + DP 70-85%线性
100B-1T 256-4096 InfiniBand 3D并行（TP+PP+DP） 50-70%线性
> 1T 4096+ 定制网络 3D并行 + ZeRO/FSDP 30-50%线性

关键原则提炼：

  • 优先用数据并行——它最简单，且总是能用（只要模型能放进单卡）
  • 模型太大时加张量并行——节点内NVLink带宽充裕，TP几乎无额外开销
  • 跨节点时加流水线并行——减少跨节点的通信量，但注意气泡问题
  • 最后用ZeRO/FSDP——当上述三种都调到最优后，再用显存优化技术压榨更多潜力

21.6 踩坑：流水线气泡的隐形杀手

真实案例：20-30%的GPU利用率被气泡"吃"掉了

某团队的训练日志显示GPU利用率只有68%，而他们使用了3D并行训练一个130B参数的模型。所有GPU都在跑——没有空闲——但利用率就是上不去。

调试过程：

  • 检查GPU计算核心利用率（sm efficiency）——确实只有68%
  • 检查内存带宽利用率——更低，只有55%
  • 检查通信时间——AllReduce只占15%的时间

问题最终定位在流水线气泡上。他们的配置是TP=8, PP=32, DP=4，总共1024块GPU。但PP=32意味着流水线有32个stage，micro-batch数量M=64。根据气泡公式：

气泡比例 = (PP - 1) / M = 31/64 ≈ 48%

但实际测量只有20-30%，因为GPipe的气泡公式是上界，实际中计算-通信重叠会掩藏一部分气泡。
然而即使30%的浪费，也意味着每天损失的算力价值超过10万元人民币。

解决方案：

  • 减少PP深度：从PP=32降到PP=16，同时将TP从8提升到16（节点内double TP）
  • 增加micro-batch数量：从M=64增加到M=128，但需要确认显存装得下
  • 使用1F1B调度替代F-then-B：节省出的显存正好支撑更大的M
  • 最终效果：GPU利用率从68%提升到87%

这个案例再次验证了分布式系统的一个永恒真理："利用率低"不一定是因为GPU在等数据，而是因为并行策略选择不当导致了结构性空闲。这个教训在下一章的分布式训练实战中还会反复出现。

总结与展望

这一章我们拆解了现代大模型训练的"三驾马车"——张量并行把一层切开、流水线并行把层切开、数据并行把数据切开。三种并行叠加的3D并行方案，是GPT-3乃至更大规模模型训练的基石。但无论是哪种并行策略，最终都要落脚到具体的工程实现上。下一章，我们将抛开理论图纸，拿起PyTorch写真正的分布式训练代码——从DDP到FSDP，从ZeRO到混合精度，把这一章和上一章的知识全部实战化。
