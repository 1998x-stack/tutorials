第26章 Memcached vs Redis/Valkey源码对比

大家好，我是你们的讲师。走到第26章，我们已经深入剖析了 Memcached 源码的每一个核心模块。如果你从第1章一路跟下来，现在应该对 slab allocator、LRU 三级队列、渐进式 rehash、libevent 事件模型这些概念如数家珍了。但一个很自然的问题会浮现在脑海中：同样做缓存，Redis 的源码是怎么处理这些问题的？它们的差异到底在哪？

这一章，我们来做一个系统的源码级对比。这不仅是为了满足好奇心——当你同时理解两种设计哲学后，你就能在选型时做出更精准的判断，也能从对比中获得更深层的架构洞察。我们还会讨论 Valkey——Redis 在 AGPL 许可证变更后 fork 出来的 BSD 协议分支——看看开源格局的变化如何影响技术选择。

整体架构：单线程 vs 多线程

Memcached 和 Redis 最根本的架构差异，在于第17章我们详细介绍过的 Master-Worker 多线程模型：Memcached 的 main thread 负责 accept 连接，worker threads 各自运行 event loop 处理请求。而 Redis 经典模型是单线程处理所有命令——一个 epoll 循环接收连接，同一个线程执行全部命令。

为什么两种截然不同的设计都能成功？

Redis 的设计基于一个关键洞察：纯 KV 操作在内存中的延迟瓶颈几乎永远是网络 I/O，而不是 CPU。一个简单的 GET/SET 操作只需要微秒级的 CPU 时间，单线程足以打满万兆网卡。单线程的好处显而易见：没有锁、没有竞争、没有上下文切换、实现简单到令人发指。Redis 的作者 antirez 曾多次强调，单线程是 Redis 的"超级力量"。

Memcached 的情况不同。它的 item size 可以大到 1MB，当你反复存取几百 KB 的二进制数据时，memcpy 的 CPU 开销就不可忽视了。更重要的是，Memcached 的 slab allocator 需要做内存管理——slab rebalance、LRU 维护、item 的引用计数操作——这些都会阻塞事件循环。因此，Memcached 从 1.4 版本开始就引入了多线程 worker 来分摊开销。

关键洞察：两种架构都是针对各自的 workload 优化的。Memcached 面向大 value、高并发、内存敏感的场景；Redis 面向小 value、丰富数据结构、持久化场景。不是谁比谁先进，而是谁更适配谁的业务。

内存管理：malloc vs slab allocator

这是两者源码层面差异最大的地方。

Redis 使用 jemalloc（默认）或 tcmalloc 作为通用内存分配器。Redis 的 zmalloc 模块只是对 malloc 的一层薄封装，加上 OOM 处理和内存统计。当你执行 SET foo bar，Redis 内部调用 zmalloc(sizeof(robj) + 3) 分配一个 3 字节的 sds 字符串——完事了，剩下的全交给 jemalloc 管理。

Memcached 不走这条路。第7章到第9章我们花了整整三章来讲解 slab allocator，它预先将内存划分为 slab class，每个 class 固定 chunk size，彻底消灭内存碎片。但代价是内部碎片——当你存一个 100 字节的 value 在 chunk size 为 256 字节的 slab 中时，156 字节被浪费了。

/* Redis 的内存分配 —— zmalloc.c */
void *zmalloc(size_t size) {
    void *ptr = malloc(size + PREFIX_SIZE);
    if (!ptr) zmalloc_oom_handler(size);
    *((size_t*)ptr) = size;  // 记录分配大小
    used_memory += size;     // 更新统计
    return (char*)ptr + PREFIX_SIZE;
}

/* Memcached 的 slab 分配 —— slab.c */
static void *slab_alloc_chunk(slabclass_t *p, const size_t size) {
    int idx;
    // 从空闲链表中取
    if (p->slots) {
        item *it = p->slots;
        p->slots = it->sl_next;
        if (it->sl_next) it->sl_next->sl_prev = 0;
        p->sl_curr--;
        return it;
    }
    // 没有空闲的，从 slab 中分配新的
    if (p->end_page_ptr) {
        void *ptr = p->end_page_ptr;
        p->end_page_ptr += p->size;
        if (--p->end_page_free == 0) p->end_page_ptr = 0;
        return ptr;
    }
    // 新分配一个 slab page
    return do_slab_newslab(p);
}

看到差异了吗？Redis 的分配逻辑只有 5 行有效代码，Memcached 则维护了一套复杂的预分配和回收体系。前者的简洁来自对通用分配器的信任，后者的复杂来自对确定性和可控性的追求。

数据结构：redisObject vs item

Redis 的数据类型五花八门——String、List、Set、ZSet、Hash、Stream、BitMap、HyperLogLog。怎么在源码层面统一管理？答案就是 redisObject：

// Redis 的 redisObject（server.h）
typedef struct redisObject {
    unsigned type:4;       // 类型: OBJ_STRING, OBJ_LIST, ...
    unsigned encoding:4;   // 编码: OBJ_ENCODING_RAW, INT, HT, SKIPLIST...
    unsigned lru:24;       // LRU 时间戳
    int refcount;          // 引用计数
    void *ptr;             // 指向实际数据
} robj;

这是在面向对象语言中常见的多态模式——type 决定操作分派，encoding 决定内存布局，ptr 指向实际的数据结构。比如一个 ZSet 可能用 skiplist 编码，也可能用 ziplist 编码——对使用者透明。

Memcached 的 item 则简单到粗暴：

// Memcached 的 item（memcached.h）
typedef struct _stritem {
    struct _stritem *next;    // hash 链表 next
    struct _stritem *prev;    // hash 链表 prev
    struct _stritem *h_next;  // hash 链
    rel_time_t      time;     // 访问时间
    rel_time_t      exptime;  // 过期时间
    int             nbytes;   // value 字节数
    unsigned short  refcount; // 引用计数
    uint8_t         nsuffix;  // suffix 长度
    uint8_t         it_flags; // ITEM_LINKED, ITEM_SLABBED...
    uint8_t         slabs_clsid; // 所属 slab class
    uint8_t         nkey;     // key 长度
    /* 然后拼上 key + suffix + value 的连续内存 */
} item;

设计哲学一目了然：Redis 追求语义丰富，Memcached 追求结构精简。Memcached 的 item 紧贴 slab chunk 分配策略，将所有元数据压缩到最小的空间里——如果你数一下字段的字节数，会发现它精心对齐到了 32 字节以内。

渐进式 rehash：主循环 vs 后台线程

两种实现都使用渐进式 rehash 来避免哈希表扩容时的大停顿，但调度方式完全不同。

Redis 的 rehash 在 主事件循环中分摊。每次处理完一批命令后，Redis 调用 databasesCron()，在其中迁移一小批 bucket。核心代码在 dict.c 的 dictRehash() 函数中：

int dictRehash(dict *d, int n) {
    int empty_visits = n * 10;  // 最多跳过的空桶数
    while (n-- && d->ht[0].used) {
        dictEntry *de, *nextde;
        while (d->ht[0].size >= (unsigned long)d->ht[0].sizemask
               && d->ht[0].table[d->rehashidx] == NULL) {
            d->rehashidx++;
            if (--empty_visits == 0) return 1;
        }
        de = d->ht[0].table[d->rehashidx];
        while (de) {
            uint64_t h = dictHashKey(d, de->key) & d->ht[1].sizemask;
            nextde = de->next;
            de->next = d->ht[1].table[h];
            d->ht[1].table[h] = de;
            d->ht[0].used--;
            d->ht[1].used++;
            de = nextde;
        }
        d->ht[0].table[d->rehashidx] = NULL;
        d->rehashidx++;
    }
    return d->ht[0].used != 0;
}

Memcached 的做法不同。第14章我们详细分析过，它使用 一个专门的维护线程 来做 rehash。这个线程在 assoc_maintenance_thread 中循环，每次迁移 1 个 bucket 后主动让出 CPU，由 pthread_cond_wait 控制迁移节奏。

设计哲学差异：Redis 选择了"借用主循环的空闲时间"，好处是不需要额外线程、没有同步开销；坏处是如果主循环一直繁忙，rehash 可能滞后。Memcached 选择了"专用后台线程"，好处是 rehash 进度不受请求负载影响；坏处是需要处理线程间同步。两种方法各有优劣，但都经过了十几年的生产验证。

协议设计：RESP vs ASCII/Binary/Meta

Redis 使用 RESP（REdis Serialization Protocol），一种人类可读的文本协议，以 \r\n 分隔。例如 *3\r\n$3\r\nSET\r\n$3\r\nfoo\r\n$3\r\nbar\r\n。RESP 在 2.0 版本定型，后来演化为 RESP3（Redis 6+ 引入），支持更丰富的数据类型返回。

Memcached 则有三套协议：

  • ASCII 协议（第19章）：get foo\r\n 这种最简形式，调试极其方便，但功能受限——不支持批量操作、不支持认证、返回格式解析复杂。
  • 二进制协议（第20章）：带固定 header 的二进制格式，性能更好，支持 SASL 认证，是生产环境推荐协议。
  • Meta Protocol（第21章）：新一代元协议，用一组 meta 命令统一了 GET/SET/DELETE/TOUCH 等操作，支持条件更新和自动过期。

从代码量上也能看出协议复杂度的差异：proto_text.c 约 3000 行，proto_bin.c 约 2500 行，proto_meta.c 约 2000 行；而 Redis 的 networking.c 处理 RESP 协议的部分超过 6000 行。

I/O 模型：epoll 单线程 vs 每线程 event_base

Redis 经典模型（6.0 之前）是 单进程单线程：一个 epoll 实例监听所有 fd，在同一个线程中读取、解码、执行、返回。Redis 6.0 引入了多线程 I/O——多个线程负责 read() 和 write()，但命令执行仍然是单线程。这本质上是一个"生产者-消费者"模型：IO threads 将解码后的命令塞入队列，主线程取出并执行。

Memcached 是 每个 worker thread 拥有独立的 event_base。main thread accept 连接后，将连接 fd 分配给某个 worker thread，后者在自己的 event loop 中注册读写事件。这意味着每个 worker 完全独立——没有共享的 epoll 实例，没有全局的事件调度。

过期策略：active expiry vs lazy only

这是一个很重要但容易被忽视的差异。

Redis 有过期键的主动清理机制：activeExpireCycle() 在每秒 10 次的 serverCron() 中，随机采样设置了 TTL 的 keys，删除其中过期的。此外在每次访问 key 时也会做 lazy free（被动删除）。两套机制配合，保证过期 key 不会长时间驻留内存。

Memcached 只有 lazy 删除。在 do_item_get() 中，当检查到 item 已过期时才会删除它。这意味着如果一个 key 永远不会被访问，即使它已经过期，也会一直占用 slab 内存直到被 LRU 淘汰覆盖。Memcached 1.6 引入了 lru_maintainer 线程 来主动扫描 LRU 队列，但本质上它做的是"淘汰"而非"过期清理"——过期是淘汰的一个因素，但不是独立的清理机制。

生产踩坑：我们曾在 Memcached 中设置了大量 1 小时 TTL 的缓存，但某些 key 在过期后仍然占用了大量内存。原因是这些 key 被 LRU 判定为"热点"（频繁访问），lru_maintainer 线程不会优先淘汰它们。最终我们不得不引入主动过期标记机制——在应用层定期写入"墓碑 key"来触发淘汰。如果您的场景对过期时效应有强要求，需要在应用层或者选择 Redis 来解决。

Valkey fork：开源格局的变化

2024 年 3 月，Redis 宣布其主分支从 BSD 许可证变更为 AGPLv3 并引入 RSALv2（Redis Source Available License） 双授权模式。这意味着云厂商不能再免费将 Redis 作为托管服务提供而不开源自己的管理工具。这一决定在开源社区引起了巨大震动。

Linux 基金会迅速反应，fork 了 Redis 7.2.4（最后一个 BSD 版本）成立了 Valkey 项目。Valkey 采用 BSD-3-Clause 许可证，吸引了 AWS、Google Cloud、Oracle、Ericsson 等公司的工程师加入核心维护团队。截至目前，Valkey 在 GitHub 上的 star 数已超过 20k，社区活跃度远超当时的预期。

对开发者来说这意味着什么？

  • 如果你是新项目选型，Valkey 和 Redis API 完全兼容，可以无缝切换
  • Valkey 的社区开发更开放（所有讨论在 public channel），合并请求的审查更透明
  • Redis Ltd. 仍然在积极开发 Redis Stack（包括 Search、JSON、TimeSeries 等模块），这是 Valkey 尚未覆盖的领域
  • Memcached 的 BSD 许可证从未改变过——20 年如一日。这是 Dormando 始终坚持的原则：缓存基础设施不应该有许可证风险

真实案例：Wikipedia 的缓存架构

Wikipedia 是全球最大的百科全书网站，也是 Memcached 最经典的部署案例之一。它的缓存架构非常清晰：

  • Memcached 用于页面渲染缓存——从数据库查询到的页面内容、解析后的模板、用户权限检查结果。纯 KV，value 大（HTML 页面），命中率极高。Wikipedia 最早在 2004 年就引入了 Memcached，至今仍在大量使用。
  • Redis 用于计数器和队列——编辑计数、API 限流、job queue。这些场景需要自增操作和 List 结构，Memcached 不擅长。

这个分工非常有代表性：Memcached 做"缓存"，Redis 做"数据结构和协调"。不是替代关系，而是互补关系。

好，这一章我们从源码层面系统对比了 Memcached 和 Redis/Valkey 的设计差异。理解这些差异让你在做技术选型时多了一个维度——不是看功能列表谁更丰富，而是看设计哲学谁更匹配你的场景。下一章，我们来聊聊 Memcached 的未来方向以及它背后的社区生态。
