引子：一个null引发的几十亿美元损失

你可能听说过Tony Hoare——他是null引用的发明者。在2009年的一次演讲中，他称null为"我的十亿美元错误"（Billion Dollar Mistake）。null的发明本来是为了让编译器能够标识"没有值"的情况，但实际上它成了无数崩溃和安全漏洞的根源：NullPointerException、Segmentation Fault、TypeError: Cannot read property of null……

Rust从根本上解决了这个问题：它没有null。取而代之的是Option<T>枚举——一个值要么是Some(T)（有值），要么是None（没有值）。不是靠运行时检查，而是靠编译器在编译期强制你处理None的情况。这个"强制"通过match的穷尽性检查来实现。

这一章，我们将深入到Rust的枚举系统——它远不止是"几个常量的集合"，而是Rust类型安全的核心武器。

枚举定义

用enum关键字定义一个枚举：

enum IpAddrKind {
    V4,
    V6,
}

let four = IpAddrKind::V4;
let six = IpAddrKind::V6;

这是最基础的枚举形式——类似C语言中的enum。但Rust的枚举远不止于此。

关联数据

这才是Rust枚举的真正亮点——每个变体可以关联不同类型的数据：

enum IpAddr {
    V4(String),
    V6(String),
}

let home = IpAddr::V4(String::from("127.0.0.1"));
let loopback = IpAddr::V6(String::from("::1"));

更进一步，每个变体还可以关联不同的类型和不同数量的数据：

enum Message {
    Quit,                            // 没有关联数据
    Move { x: i32, y: i32 },        // 匿名结构体
    Write(String),                   // 单个String
    ChangeColor(i32, i32, i32),     // 三个i32
}

这等价于定义了四个不同的结构体，但用枚举把它们统一成了一个类型。这意味着你可以写一个函数接受Message，然后处理它所有的变体——而不需要为每个变体单独写一个函数。

枚举上的impl方法

枚举和结构体一样，可以有impl方法：

impl Message {
    fn call(&self) {
        match self {
            Message::Quit => println!("退出消息"),
            Message::Write(text) => println!("写入: {text}"),
            _ => println!("其他消息类型"),
        }
    }
}

let msg = Message::Write(String::from("hello"));
msg.call();  // 写入: hello

Option枚举

这是Rust标准库中最重要、使用最广泛的枚举——没有之一：

enum Option<T> {
    Some(T),
    None,
}

Option<T>被预置到Rust的prelude中，所以你不需要手动引入——直接使用Some和None即可：

let some_number = Some(5);          // Option<i32>
let some_char = Some('a');          // Option<char>
let absent_number: Option<i32> = None;  // 需要显式标注类型

// Option<T> 和 T 是不同类型，不能混合运算
let x: i32 = 5;
let y: Option<i32> = Some(10);

// let sum = x + y;  // 编译错误！不能把i32和Option<i32>相加
let sum = x + y.unwrap_or(0);  // 正确：用unwrap_or提供默认值
为什么Option比null好？
  • 在Java/C++中，任何引用类型都能是null——编译器不会帮你检查
  • 在Rust中，如果你要表达"可能没有值"，必须使用Option<T>
  • T和Option<T>是不同类型，编译器禁止你把可能为None
  • 这迫使你在代码中显式处理了"空值"的情况——在编译阶段，而不是运行时

Option<T>有一整套方法来处理"有值/无值"的情况：

方法 作用 示例
unwrap()有值则返回，None则panicSome(5).unwrap() == 5
unwrap_or(default)有值则返回，None则用默认值None.unwrap_or(0) == 0
is_some()是否有值Some(5).is_some() == true
is_none()是否无值None.is_none() == true
map(f)有值时应用函数Some(5).map(|x| x * 2) 得到 Some(10)
and_then(f)链式操作（flat_map）常用于"可能失败的链式操作"
unwrap() 的危险：初学Rust时，你可能会觉得.unwrap()很方便——它让你快速获取Option或Result中的值。但在生产代码中，.unwrap()意味着"要么给我值，要么让程序崩溃"。仅在你100%确定不会None的情况下才使用它，比如"42".parse::<i32>().unwrap()。在其他场景下，用match或unwrap_or等方式更安全。许多clippy lint都会警告不必要的unwrap()调用。

match穷尽性

match是Rust中处理枚举的主要工具。它的一个核心特性是穷尽性检查（Exhaustiveness Check）：你必须覆盖枚举的所有变体，否则编译器会报错。

enum Coin {
    Penny,
    Nickel,
    Dime,
    Quarter,
}

fn value_in_cents(coin: Coin) -> u8 {
    match coin {
        Coin::Penny => 1,
        Coin::Nickel => 5,
        Coin::Dime => 10,
        Coin::Quarter => 25,
        // 如果少写一个变体，编译器会报错！
    }
}

这个特性在实践中非常重要。想象一下：你的团队给Coin增加了一个新变体HalfDollar（50美分），编译器会找到所有使用了match coin的地方，强制你处理这个新变体——你不可能遗漏。在C++中用switch的话，漏掉一个分支只是warning，很容易被忽略。

通配符 _ 和 other

当你不关心某些变体时，可以用通配符：

let num = 42;
match num {
    1 => println!("一"),
    2 => println!("二"),
    _ => println!("其他数字"),  // 匹配所有未列出的值
}

// 或者用 other 绑定（把匹配到的值捕获到一个变量中）
match num {
    1 => println!("一"),
    other => println!("其他数字: {other}"),
}
match穷尽性是安全网，通配符是逃生舱。在编写库代码时，尽量使用穷尽匹配（列出所有变体），这样当你增加新变体时，编译器会帮你找到所有需要修改的地方。在应用代码中，合理使用_通配符来简化代码。关键是有意识地选择，而不是随手写下_。

if let 与 while let

当你只关心一个特定的模式时，使用match会显得有些冗长：

let config_value = Some(3);

// 用match写——需要写一个通配符分支
match config_value {
    Some(max) => println!("最大连接数: {max}"),
    _ => (),  // 空语句，为了满足穷尽性
}

// 用if let——简洁多了
if let Some(max) = config_value {
    println!("最大连接数: {max}");
}

if let的语法是：if let 模式 = 表达式 { 分支 }。它只在模式匹配成功时执行代码块，否则跳过。

if let还可以搭配else：

if let Some(max) = config_value {
    println!("最大连接数: {max}");
} else {
    println!("未配置最大连接数，使用默认值");
}

while let

与if let类似，while let在模式一直匹配时持续循环：

let mut stack = Vec::new();
stack.push(1);
stack.push(2);
stack.push(3);

while let Some(top) = stack.pop() {
    println!("{top}");
}
// 输出:
// 3
// 2
// 1

while let非常适用于处理"可变的、可能为空的"数据源——不断从集合中取值，直到取完为止。

if let vs match 的选择：
  • 需要处理2个及以上变体 → 用match
  • 只关心1个特定变体，其他情况忽略 → 用if let
  • 只关心1个特定变体，但其他情况要做点事 → if let ... else

没有绝对的"更好的选择"——关键是代码的可读性。当你发现match里有大量的_ => ()，说明应该用if let了。

常用匹配模式

Rust的模式匹配能力远不止匹配枚举变体。以下是你最常用到的匹配模式：

模式类型 示例 说明
字面量1 => ...匹配具体值
范围1..=5 => ...匹配1到5（包含）
守卫x if x % 2 == 0 => ...匹配满足额外条件的值
解构Point { x, y } => ...解构结构体或枚举
绑定Some(val) => ...匹配并捕获内部值
通配符_ => ...匹配所有剩余情况
多重模式1 | 2 => ...匹配多个值（用|）
 
// 范围模式
let score = 85;
match score {
    90..=100 => println!("优秀"),
    80..=89  => println!("良好"),
    70..=79  => println!("中等"),
    60..=69  => println!("及格"),
    0..=59   => println!("不及格"),
    _        => println!("无效分数"),
}

// 守卫模式（guard）
let num = Some(4);
match num {
    Some(x) if x % 2 == 0 => println!("偶数: {x}"),
    Some(x) => println!("奇数: {x}"),
    None => println!("没有值"),
}

// 解构结构体
struct Point { x: i32, y: i32 }
let p = Point { x: 10, y: -5 };
match p {
    Point { x, y: 0 } => println!("在x轴上, x={x}"),
    Point { x: 0, y } => println!("在y轴上, y={y}"),
    Point { x, y } => println!("其他位置: ({x}, {y})"),
}

// 多重模式
let c = 'a';
match c {
    'a' | 'e' | 'i' | 'o' | 'u' => println!("元音字母"),
    'q' | 'x' | 'z' => println!("少见字母"),
    _ => println!("其他字母"),
}

守卫模式在实际开发中尤其常见——它让你能写出比单纯模式匹配更精细的逻辑。比如在处理HTTP请求时，你可以根据状态码的范围做不同的处理：

fn handle_status(code: u16) -> &'static str {
    match code {
        100..=199 => "信息响应",
        200..=299 => "成功",
        300..=399 => "重定向",
        400..=499 => "客户端错误",
        500..=599 => "服务器错误",
        _ => "未知状态码",
    }
}

本章小结

这一章我们领略了Rust中枚举与模式匹配的强大力量：用enum定义带有不同类型关联数据的变体；用Option<T>安全地替代了null；用match的穷尽性检查确保所有情况都被处理；用if let和while let在只关心一个模式时简化代码；以及各种匹配模式的灵活运用——范围、守卫、解构、多重模式等。

枚举和模式匹配的组合是Rust安全编程的基石。配合第6章的结构体，你已经有了构建复杂数据类型的基本工具。下一章，我们将跳出单个文件，学习Rust的项目管理与模块系统——如何把代码组织成模块，如何管理依赖，以及Rust独特的路径系统。
