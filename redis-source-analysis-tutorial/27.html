第27章：Module 开发实战

大家好，我是你们的讲师。上一章我们讲了 Lua 脚本——它在 Redis 内部嵌入了一个动态语言解释器，让你可以在 Redis 进程中运行自定义逻辑。但 Lua 脚本有它的天花板：性能受限于解释执行、无法操作网络层、不能管理持久化状态。如果你需要的是更底层、更高效、与 Redis 核心数据结构深度集成的能力，那就需要 Module 系统了。

Module 是 Redis 4.0 引入的一个里程碑式特性。它提供了一套 C API，让开发者可以编写动态链接库（.so 文件），在运行时加载到 Redis 进程中，注册新命令、创建新数据类型、甚至操作网络 IO。可以说，Module 系统把 Redis 从一个"固件"变成了一个"操作系统"——你可以在上面"安装"功能模块。Redis Stack 中的 RediSearch、RedisJSON、RedisTimeSeries 全都是基于 Module API 构建的。

本章我们从 module.c（约 7000 行，是 Redis 中最大的源文件之一）中提取核心脉络，理解 Module API 的设计哲学，并通过一个完整的实战例子带你走通 Module 开发的全流程。

Module API 体系

Module 系统的入口函数是 RedisModule_Init——这是一个宏，在每个 Module 的 OnLoad 函数中必须首先调用。它完成了 Module 与 Redis 核心的握手：版本检查、API 函数绑定、标志位设置。

/* 一个最小 Module 的骨架 */
#include "redismodule.h"

int RedisModule_OnLoad(RedisModuleCtx *ctx, RedisModuleString **argv, int argc) {
    /* 1. 初始化：版本校验 + API 函数绑定 */
    if (RedisModule_Init(ctx, "mymodule", 1, REDISMODULE_APIVER_1)
        == REDISMODULE_ERR) {
        return REDISMODULE_ERR;
    }

    /* 2. 注册自定义命令 */
    if (RedisModule_CreateCommand(ctx, "mycommand.set",
                                  MyCommand_Set, "write", 1, 1, 1) ==
        REDISMODULE_ERR) {
        return REDISMODULE_ERR;
    }

    return REDISMODULE_OK;
}

RedisModule_CreateCommand 的参数列表揭示了 Module 命令的注册细节：

/* module.c — RedisModule_CreateCommand 签名 */
int RM_CreateCommand(RedisModuleCtx *ctx, const char *name,
                     RedisModuleCmdFunc cmdfunc, const char *strategy,
                     int firstkey, int lastkey, int keystep);
  • strategy：标志位字符串，如 "write"（写命令）、"readonly"（只读）、"denyoom"（内存不足时拒绝）等。这会影响命令在集群模式和只读从库中的行为。
  • firstkey / lastkey / keystep：告诉 Redis 命令参数中的 key 的位置，用于集群模式下的 slot 计算。比如 firstkey=1, lastkey=1, keystep=1 表示第一个参数之后的第一个参数是 key。

除了命令注册，Module API 还提供了几大类功能：

API 类别 核心函数 用途
初始化与元信息 RedisModule_Init, RedisModule_SetModuleAttribs Module 注册、版本声明
命令注册 RedisModule_CreateCommand 将 C 函数注册为 Redis 命令
键空间操作 RedisModule_OpenKey, RedisModule_KeyType, RedisModule_ModuleTypeSetValue 读写 Redis 键值对
字符串处理 RedisModule_CreateString, RedisModule_StringToLongLong, RedisModule_StringPtrLen 操作 RedisModuleString 对象
内存管理 RedisModule_Alloc, RedisModule_Free, RedisModule_Realloc, RedisModule_AutoMemory Module 专用的内存分配与自动释放
数据类型 RedisModule_CreateDataType, RedisModule_ModuleTypeGetValue 注册自定义数据结构，集成 RDB/AOF/复制
回复构建 RedisModule_ReplyWithLongLong, RedisModule_ReplyWithString, RedisModule_ReplyWithArray 向客户端返回结果
阻塞客户端 RedisModule_BlockClient, RedisModule_UnblockClient 在后台线程中处理，完成后恢复客户端
通知与钩子 RedisModule_SubscribeToKeyspaceEvents, RedisModule_RegisterAuthCallback 订阅键空间事件、钩入认证流程

自定义命令实现

定义一个 Module 命令其实就是实现一个 int (*RedisModuleCmdFunc)(RedisModuleCtx *ctx, RedisModuleString **argv, int argc) 函数。来看一个完整的例子：

/* 实现 MYSET.GET — 获取自定义数据类型的值 */
int MySet_GetValue(RedisModuleCtx *ctx, RedisModuleString **argv, int argc) {
    /* 检查参数数量 */
    if (argc != 2) {
        RedisModule_WrongArity(ctx);
        return REDISMODULE_ERR;
    }

    /* 打开 key（第一个参数之后是 key 名） */
    RedisModuleKey *key = RedisModule_OpenKey(ctx, argv[1],
                                              REDISMODULE_READ);

    /* 检查 key 的类型 */
    int keytype = RedisModule_KeyType(key);
    if (keytype == REDISMODULE_KEYTYPE_EMPTY) {
        RedisModule_ReplyWithNull(ctx);
        RedisModule_CloseKey(key);
        return REDISMODULE_OK;
    }

    /* 读取 Module 自定义类型的值 */
    MySet *ms = RedisModule_ModuleTypeGetValue(key);
    if (ms == NULL) {
        RedisModule_ReplyWithNull(ctx);
    } else {
        RedisModule_ReplyWithLongLong(ctx, ms->count);
    }

    RedisModule_CloseKey(key);
    return REDISMODULE_OK;
}

这里有一个非常实用的机制叫 自动内存管理（AutoMemory）。当你在 Module 函数开头调用 RedisModule_AutoMemory(ctx) 后，Module 上下文中创建的 RedisModuleString、RedisModuleKey 等对象会在函数返回时自动释放。这极大地简化了 Module 代码中的内存管理——你不需要像普通 C 代码那样小心翼翼地在每个 return 之前释放资源。

小贴士：自动内存管理虽然方便，但要警惕它的陷阱。如果你在 Module 函数中创建了一个 RedisModuleString 并将其存入全局变量，开启 AutoMemory 后函数返回时会自动释放这个字符串——导致全局指针悬空。对于需要跨越函数调用存活的对象，应该使用 RedisModule_RetainString 增加引用计数，或者禁用 AutoMemory 手动管理。

自定义数据类型

Module 最强大的能力之一就是创建自定义数据类型。这种数据类型与 Redis 原生的 String、Hash、List 等类型享受完全相同的待遇：可以存到 RDB 文件中、可以写入 AOF 日志、可以通过主从复制同步到从库。要实现这一点，你需要提供一组类型方法回调：

/* 注册自定义数据类型的回调结构体 */
static RedisModuleTypeMethods mytype_methods = {
    .version = REDISMODULE_TYPE_METHOD_VERSION,
    .rdb_load = MySet_RDBLoad,     /* 从 RDB 文件反序列化 */
    .rdb_save = MySet_RDBSave,     /* 序列化到 RDB 文件 */
    .aof_rewrite = MySet_AOFRewrite, /* AOF 重写回调 */
    .free = MySet_Free,            /* 释放自定义类型的内存 */
    .digest = MySet_Digest         /* DEBUG DIGEST 命令的哈希值 */
};

/* 在 OnLoad 中注册自定义类型 */
int RedisModule_OnLoad(...) {
    /* 注册类型，返回类型 ID */
    RedisModuleTypeID *type_id = RedisModule_CreateDataType(
        ctx, "myset", 0, &mytype_methods);
    /* 将 type_id 保存到全局变量供后续使用 */
    ...
}

这些回调会在 Redis 的持久化、复制、调试过程中被调用。其中最核心的是 rdb_save 和 rdb_load——它们决定了你的自定义数据类型如何序列化和反序列化：

/* RDB 保存回调：将 MySet 结构写入 rdb 流 */
void MySet_RDBSave(RedisModuleIO *io, void *ptr) {
    MySet *ms = (MySet *)ptr;
    RedisModule_SaveUnsigned(io, ms->count);     /* 保存元素数量 */
    for (int i = 0; i < ms->count; i++) {
        RedisModule_SaveString(io, ms->elements[i]); /* 保存每个元素 */
    }
}

/* RDB 加载回调：从 rdb 流恢复 MySet 结构 */
void *MySet_RDBLoad(RedisModuleIO *io, int encver) {
    MySet *ms = RedisModule_Alloc(sizeof(MySet));
    ms->count = RedisModule_LoadUnsigned(io);   /* 读取元素数量 */
    ms->elements = RedisModule_Alloc(sizeof(void*) * ms->count);
    for (int i = 0; i < ms->count; i++) {
        ms->elements[i] = RedisModule_LoadString(io); /* 读取每个元素 */
    }
    return ms;
}

Module 阻塞命令

某些操作需要很长时间（比如外部 API 调用、磁盘 IO、复杂的计算），在 Redis 的单线程事件循环中执行会导致整个实例阻塞。Module API 提供了 阻塞客户端机制来解决这个问题：

/* 模块阻塞命令：异步处理 */
int MyBlockCmd(RedisModuleCtx *ctx, RedisModuleString **argv, int argc) {
    /* 创建一个后台任务 */
    struct my_task *task = RedisModule_Alloc(sizeof(*task));
    task->key = argv[1];
    RedisModule_RetainString(ctx, argv[1]);

    /* 阻塞当前客户端 */
    RedisModuleBlockedClient *bc = RedisModule_BlockClient(
        ctx,                          /* 当前上下文 */
        MyBlockCmd_Reply,             /* 解除阻塞后的回复函数 */
        MyBlockCmd_Timeout,           /* 超时回调 */
        MyBlockCmd_FreeData,          /* 释放数据的回调 */
        0                             /* 超时时间（0 表示不超时） */
    );
    task->bc = bc;

    /* 启动后台线程处理（使用 pthread） */
    pthread_t tid;
    pthread_create(&tid, NULL, MyBlockCmd_ThreadProc, task);
    pthread_detach(tid);  /* 分离线程，自动回收资源 */

    return REDISMODULE_OK;
}

/* 后台线程处理函数 */
void *MyBlockCmd_ThreadProc(void *arg) {
    struct my_task *task = arg;
    /* 执行耗时操作... */
    /* 解除阻塞客户端（传递结果数据） */
    RedisModule_UnblockClient(task->bc, task);
    return NULL;
}

当后台线程调用 RedisModule_UnblockClient 时，Redis 会在自己的事件循环中异步调用 MyBlockCmd_Reply 函数来构造回复并发送给客户端。整个过程不会阻塞 Redis 的主事件循环，完美解决了耗时操作的问题。

Redis Stack 模块生态

如果说 Redis 本体内核是"操作系统"，那么 Redis Stack 就是预装了一系列常用"应用程序"的发行版。每个 Stack 模块都是基于 Module API 构建的工业级实现：

模块 提供能力 核心数据结构 典型命令
RediSearch 全文搜索 + 向量检索 + 聚合 倒排索引、HNSW 图、Trie 字典树 FT.SEARCH, FT.AGGREGATE, FT.CREATE
RedisJSON JSON 文档存储与路径查询 JSON 解析树 JSON.SET, JSON.GET, JSON.ARRAPPEND
RedisTimeSeries 时序数据存储与聚合 压缩时间序列块 TS.CREATE, TS.RANGE, TS.MRANGE
RedisBloom 概率数据结构 Bloom Filter、Cuckoo Filter、Count-Min Sketch BF.ADD, BF.EXISTS, CMS.INCRBY
RedisGraph (已退役) 图数据库（7.0 后不再维护） 属性图矩阵 GRAPH.QUERY
踩坑实录：我见过一个令人印象深刻的 Module 事故——某团队写了一个自定义 Module 用于实时统计，在 free 回调中没有正确释放所有子对象，导致每次自定义类型被删除时都泄漏一部分内存。更可怕的是，这个 Module 被用于高频的"创建-删除"场景，内存泄漏速度很快。最终在线上运行两周后，Redis 进程的 RSS 从 2GB 暴涨到了 18GB，触发了 OOM killer。排查时用 INFO MEMORY 发现 used_memory 才 3GB，但 rss 已经到了 18GB——这正是典型的内存碎片 + 泄漏信号。修复方案：在 free 回调中使用 RedisModule_Free 释放每个子对象，并用 valgrind 验证所有路径。Module 代码必须像内核代码一样谨慎，因为你跑在 Redis 的地址空间里，任何泄漏都是 Redis 的泄漏。

实战：编写一个简单的计数器模块

让我们动手写一个完整的 Module。这个模块实现基于 HLL（HyperLogLog）思想的去重计数器——记录某个 key 的不重复元素数量（简化版，仅用固定大小的 bitmap 近似计数）：

/* dcard.c — 去重计数器 Module 完整实现 */
#include "../src/redismodule.h"
#include <string.h>
#include <stdio.h>

/* 计数器的数据结构：一个 64KB 的 bitmap + 计数 */
typedef struct {
    unsigned char bitmap[65536];  /* 64KB bitmap */
    long long count;              /* 近似计数 */
} DCard;

/* 自定义类型的回调 */
void DCard_Free(void *ptr) {
    /* 不需要额外释放，因为整个结构体在栈上连续分配 */
}

void DCard_RDBSave(RedisModuleIO *io, void *ptr) {
    DCard *dc = (DCard *)ptr;
    RedisModule_SaveUnsigned(io, dc->count);
    RedisModule_SaveStringBuffer(io, (char *)dc->bitmap, 65536);
}

void *DCard_RDBLoad(RedisModuleIO *io, int encver) {
    DCard *dc = RedisModule_Alloc(sizeof(DCard));
    dc->count = RedisModule_LoadUnsigned(io);
    size_t len;
    char *data = RedisModule_LoadStringBuffer(io, &len);
    memcpy(dc->bitmap, data, 65536);
    RedisModule_Free(data);
    return dc;
}

/* DCARD.ADD key element — 添加元素 */
int DCard_Add(RedisModuleCtx *ctx, RedisModuleString **argv, int argc) {
    if (argc != 3) return RedisModule_WrongArity(ctx);

    RedisModuleKey *key = RedisModule_OpenKey(ctx, argv[1],
        REDISMODULE_READ | REDISMODULE_WRITE);

    DCard *dc;
    if (RedisModule_KeyType(key) == REDISMODULE_KEYTYPE_EMPTY) {
        dc = RedisModule_Alloc(sizeof(DCard));
        memset(dc->bitmap, 0, 65536);
        dc->count = 0;
        RedisModule_ModuleTypeSetValue(key, DCardType, dc);
    } else {
        dc = RedisModule_ModuleTypeGetValue(key);
    }

    /* 使用元素的哈希值映射到 bitmap 位置 */
    size_t len;
    const char *elem = RedisModule_StringPtrLen(argv[2], &len);
    unsigned long h = 0;
    for (size_t i = 0; i < len; i++) {
        h = h * 31 + (unsigned char)elem[i];
    }
    int idx = h % 65536;
    int byte_pos = idx / 8;
    int bit_pos = idx % 8;

    if (!(dc->bitmap[byte_pos] & (1 << bit_pos))) {
        dc->bitmap[byte_pos] |= (1 << bit_pos);
        dc->count++;
    }

    RedisModule_ReplyWithLongLong(ctx, dc->count);
    RedisModule_CloseKey(key);
    return REDISMODULE_OK;
}

/* DCARD.COUNT key — 获取计数器值 */
int DCard_Count(RedisModuleCtx *ctx, RedisModuleString **argv, int argc) {
    if (argc != 2) return RedisModule_WrongArity(ctx);

    RedisModuleKey *key = RedisModule_OpenKey(ctx, argv[1], REDISMODULE_READ);
    if (RedisModule_KeyType(key) == REDISMODULE_KEYTYPE_EMPTY) {
        RedisModule_ReplyWithLongLong(ctx, 0);
        RedisModule_CloseKey(key);
        return REDISMODULE_OK;
    }

    DCard *dc = RedisModule_ModuleTypeGetValue(key);
    RedisModule_ReplyWithLongLong(ctx, dc->count);
    RedisModule_CloseKey(key);
    return REDISMODULE_OK;
}

/* Module 入口 */
static RedisModuleType *DCardType;

int RedisModule_OnLoad(RedisModuleCtx *ctx,
                       RedisModuleString **argv, int argc) {
    if (RedisModule_Init(ctx, "dcard", 1, REDISMODULE_APIVER_1)
        == REDISMODULE_ERR)
        return REDISMODULE_ERR;

    RedisModuleTypeMethods methods = {
        .version = REDISMODULE_TYPE_METHOD_VERSION,
        .rdb_load = DCard_RDBLoad,
        .rdb_save = DCard_RDBSave,
        .free = DCard_Free,
    };

    DCardType = RedisModule_CreateDataType(ctx, "dcard", 0, &methods);
    if (DCardType == NULL) return REDISMODULE_ERR;

    if (RedisModule_CreateCommand(ctx, "dcard.add",
                                  DCard_Add, "write deny-oom", 1, 1, 1)
        == REDISMODULE_ERR) return REDISMODULE_ERR;

    if (RedisModule_CreateCommand(ctx, "dcard.count",
                                  DCard_Count, "readonly", 1, 1, 1)
        == REDISMODULE_ERR) return REDISMODULE_ERR;

    return REDISMODULE_OK;
}

编译和加载这个 Module：

# 编译（在 Redis 源码根目录）
gcc -shared -fPIC -I. -o dcard.so dcard.c

# 加载到 Redis
redis-cli MODULE LOAD /path/to/dcard.so

# 测试
redis-cli DCARD.ADD myset hello
redis-cli DCARD.ADD myset world
redis-cli DCARD.COUNT myset
小贴士：Module 开发最友好的调试方式是 MODULE LOAD 和 MODULE UNLOAD 的组合。修改代码后不需要重启 Redis：先 MODULE UNLOAD mymodule，重新编译，再 MODULE LOAD /path/module.so。但要注意，UNLOAD 要求 Module 没有正在使用的自定义类型数据，否则会失败。

从 Lua 脚本到 Module Extension，我们看到 Redis 提供了一条从"在 Redis 内运行脚本"到"用 C 重写 Redis 的一部分"的完整能力阶梯。最后一章，我们不仅要展望 Redis 8.x 的前沿特性，还要把全课程学到的所有知识浓缩为一个动手项目——手写一个 Mini-Redis，让你真正从源码阅读者转变为源码创造者。
