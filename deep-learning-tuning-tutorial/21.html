推荐系统的调参——工业级AI的真正战场

大家好，我是你们的讲师。今天我们来聊推荐系统的调参。如果你觉得调CV或NLP模型已经够复杂了，那推荐系统会给你打开一扇新的大门——在这里，数据泄露、负采样偏差、多任务梯度冲突、特征工程......每一个问题都是工业级的坑。

推荐系统的特殊性在于：它不是在一个标准数据集上优雅地拟合，而是在海量、稀疏、动态变化的数据上做预测。正样本（用户点击/转化）极度稀缺，而负样本的选择方式直接决定了模型学到的决策边界。特征工程不是"锦上添花"而是"生死攸关"。多任务学习（同时预测点击率、转化率、停留时长等）带来了梯度冲突的数学难题。

先讲一个教训。某电商团队在做搜索排序模型时，离线AUC提升了0.7个点，团队挺兴奋。后来线上灰度没动静，甚至部分品类略降。排查后发现验证集是随机切的——同一个用户的连续行为同时出现在训练和验证里，模型学到了用户短期偏好泄漏。重新按时间切分以后，离线提升直接没了一半。这种故事在推荐领域太常见了，以至于wiki里专门强调："推荐系统必须按时间切分训练/验证集，避免同用户连续行为同时出现在两份数据中造成信息泄漏。"

负采样为什么有效

推荐系统的负采样问题，本质上是一个Noise Contrastive Estimation（NCE）问题。NCE的核心思想是：你不需要对所有item算分数来做一个精确的softmax归一化——你只需要学会区分"正样本"和"噪声分布中采样的负样本"。这就把O(|候选集|)的计算量降到了O(负样本数)。

为什么这不是一种"偷懒"而是一种有效的方法？因为在一个拥有百万甚至亿级item的候选集中，绝大多数item和当前用户是完全不相关的。模型不需要精确计算"用户对每个item的偏好概率"，只需要学会区分"用户喜欢的"和"绝大多数用户不喜欢的"。

但关键在于负样本怎么选。直觉上，"曝光未点击"不就是用户不感兴趣的直接证据吗？错。在推荐系统中，曝光未点击样本的分布已经经过前序模型的筛选——只有被推荐系统认为"可能感兴趣"的物料才会被曝光。用户不点击可能是因为没注意到（position bias）、已经看过（novelty）、或者当时不方便（context bias），而不是真正不感兴趣。wiki中的经验总结非常明确："召回负采样负样本效果肯定高于曝光未点击负样本"，因为全局随机负采样更接近召回的真实serving分布。

不同阶段需要不同的负采样策略：

  • 召回阶段：候选集是全量物料（百万到亿级），用全局随机负采样。每个正样本搭配10~1000个随机负样本。可以在随机负采样基础上加入少量曝光未点击样本作为困难负样本，比例约10%~30%。
  • 粗排阶段：候选是召回结果（千到万级），用精排结果的Top-K做负样本——这些样本经过了召回+粗排两关筛选但被精排判定为低分，与粗排面临的真正判别场景高度一致。
  • 精排阶段：候选是粗排结果（百到千级），用曝光未点击样本——它们与正样本的分布最接近。如果还不够，可以引入精排分数接近但未点击的hard negative。
关键认知："召回需要全局分布，精排需要局部判别。"负采样策略应该随着漏斗向下游移动而从"宽"变为"严"。全链路使用同一套负采样策略是大忌。

特征分箱的信息论解释

在推荐系统中，连续特征的处理是一个常被低估但极其重要的环节。wiki里有句直白的话："神经网络模型对于连续数值的处理是不如XGBoost的"。

原因在于：神经网络的Dense层对连续值的处理本质上是线性变换+非线性激活，在拟合阶梯状的分段函数时效率远低于树模型。而特征分箱（Feature Binning）本质上是在做非线性特征变换——将连续特征离散化为K个区间，每个区间获得一个独立的Embedding向量表示。

原始连续值 → Dense(1, hidden)            # 一个特征只有一个权重向量
分箱后ID   → Embedding(bucket_num, dim)   # 每个区间都有独立向量表示

分箱的威力在于：原始连续值只有一维的标量权重，而分箱后的ID能做多维Embedding。假设你分10箱、每箱Embedding维度是8，那模型对价格特征的表达能力从1维扩展到了80维。这就是为什么分箱后效果往往优于直接输入原始值——不是为了"离散化"本身，而是为了利用Embedding的丰富表达能力。

最常见的分箱方法是等频分箱：让每个箱包含等量的样本。等频分箱在推荐系统中是默认首选——每个箱样本均匀，模型训练稳定。与之对应的是等距分箱（间隔相等），在数据分布不均匀时可能出现空箱。

分箱数量是一个重要的超参数：太少会损失信息，太多则每个箱样本稀疏、Embedding难以学习。通常10~20箱是合理范围。数据量越大，箱数可以越多。还有一个关键技巧：将异常值单独成箱。比如某个特征99%的数据在[0, 100]，剩下1%在[100, 100000]，应该让[100, +inf)单独做一箱，避免极端值"压扁"其他正常分箱。

wiki中的建议很实用："在推荐系统中，神经网络模型对于连续数值的处理是不如xgb的，所以，最好对连续数值分箱，一般等频分箱足矣，但是，个人建议还是观察数据分布，把outlier单独一箱，如果还想完美一点可以用IV值看看分箱的优劣。"

多任务学习的梯度冲突

推荐系统几乎都是多目标的——同时预测CTR（点击率）、CVR（转化率）、停留时长、点赞概率等等。当这些任务共享底层网络时，一个根本性问题出现了：不同任务的梯度方向可能冲突。

想象一个场景：CTR任务希望模型学会"标题党"内容的特征（因为点击率高），但CVR任务希望模型忽略"标题党"（因为点击了也不转化）。两个任务对同一个特征的梯度方向可能是相反的——一个要增强这个特征的影响，另一个要削弱它。这就是梯度冲突。

PCGrad（Project Conflicting Gradients）是解决这个问题最实用的方法。核心思想是：当两个任务的梯度方向夹角大于90度时，将一个任务的梯度投影到另一个任务梯度的垂直方向上，消除冲突分量。这意味着参数更新时，不会因为A任务的需求而损害B任务已经学到的内容。

# PCGrad核心逻辑（伪代码）
for each pair of tasks (i, j):
    if cos(grad_i, grad_j) < 0:  # 梯度冲突
        grad_i = grad_i - proj(grad_i, grad_j)  # 移除冲突分量
        grad_j = grad_j - proj(grad_j, grad_i)
# 用修正后的梯度更新参数

Gradient Surgery的效果稳定、实现简单（几十行代码即可完成），且不需要引入额外超参数。wiki中来自一线从业者的经验是："推荐多目标任务可以考虑采用Gradient Surgery的方式，帕累托就算复杂度又高，效果也没有好很多，不推荐。"

实践中还有一个被反复强调的基础要点：各任务的Loss必须在同一量级。CTR的BCE Loss可能在0.3量级，CVR可能在0.05量级，如果直接相加，CVR任务基本"学不动"。解决方法是给每个Loss乘一个权重系数，使它们的加权值大致相等。或者使用Uncertainty Weight方法让模型自适应学习各任务权重。

多任务Loss平衡："多任务情况下，各loss想法限制在一个量级上，或者最终限制在一个量级上，初期可以着重一个任务的loss。"——初期先让点击率任务收敛，再逐步引入CVR等深层目标。

推荐系统调参独特性

推荐系统的调参有很多独有的经验法则，和CV/NLP截然不同。以下是几个核心要点：

Batch Size：推荐系统的batch size在32到512之间有效，超过512基本无正向收益，低于32训练太慢。wiki记录的经验是："batch size对于推荐来说32-64-128-512测试效果再高一般也不会正向了。"

学习率：从0.01开始，以两倍递减搜索（0.01 → 0.005 → 0.0025 → 0.001）。数据量极大时一个epoch足矣，更多epoch会导致过拟合——"对于数据量巨大的推荐系统的模型来说一个epoch足矣，再多就会过拟合。"

优化器选择："对于稀疏特征多的模型采用adagrad，稠密特征多的模型用adam。"稀疏特征（如用户ID、商品ID）的频次差异极大，Adagrad的自适应学习率能更好处理这种频次差异。稠密特征多的场景则Adam表现更好。

Embedding L2 Normalization：不要迷信"专家说embedding要做L2 norm"。wiki中有个血泪教训："做召回的同学，不要迷信专家们说的embedding做l2 norm，笔者就踩过这个坑，直接对embedding l2结果效果贼垃圾，查了半天，直接去掉l2，效果正常了。"

以下是推荐系统特有超参数的速查表：

超参数 推荐范围 调参技巧
Batch Size 32 ~ 512 超过512一般无正向收益；小于32训练太慢
初始学习率 0.001 ~ 0.01 从0.01开始，两倍递减搜索
负采样比例 1:10 ~ 1:1000 从1:1开始，逐步拉大；召回用全局随机，精排用曝光未点击
分箱数量 10 ~ 20 等频分箱默认首选；异常值单独成箱
多任务Loss权重 各Loss在同一量级 初期可侧重主任务，后期逐步平衡
Embedding维度 8 ~ 64 特征频次越高维度越大；不盲目加L2 Norm
训练Epoch 1 ~ 3 数据量大时1 epoch足矣；更多epoch易过拟合

推荐系统的调参还有很多值得深入的话题——热启动策略、两阶段训练（Adam预热+SGD收尾）、样本泄漏排查等等。但今天提到的负采样、特征分箱、多任务梯度冲突，已经是推荐调参最核心的三个支柱了。
