第23章 CLI shell 源码

大家好，我是你们的讲师。前二十多章我们一直在剖析 SQLite 核心库的源码——从词法分析到 VDBE 虚拟机，从 B-Tree 存储引擎到查询规划器。但有一个重要的"门面"我们一直没聊——那就是每个 SQLite 开发者每天都会用的 sqlite3 命令行工具。

你也许觉得它就是个"输入 SQL，输出结果"的简单 REPL。但如果你打开 src/shell.c.in 看一下——哇，这份源码超过 4 万行。为什么一个命令行 shell 需要 4 万行代码？因为 SQLite 的 shell 绝不是一个简单的 REPL——它还是 调试工具、诊断工具、格式化工具和扩展框架。今天我们就走进它的源码世界。

shell.c.in 的架构

严格来说，CLI shell 的源码并不是一个直接的 .c 文件，而是一个 .c.in 模板文件。这意味着它在编译前要经过预处理——SQLite 的构建系统通过 tool/mkshellc.tcl 脚本，把 shell.c.in 中的一些宏占位符替换为实际的代码片段，生成最终的 shell.c。

为什么需要这种模板机制？主要是为了处理 条件编译。SQLite 支持大量的编译时选项，shell 的行为需要根据核心库的编译配置自动调整。例如，如果核心库编译时去掉了 FTS5 支持（SQLITE_OMIT_FTS5），shell 中就不能出现 .fts5 相关命令。模板机制通过 @HAVE_FTS5@ 之类的占位符来解决这个问题。

shell.c.in 的整体结构大致分为以下几个大段：

代码段行数（约）职责
头文件引入 & 宏定义~500 行平台适配、HAVE_* 宏、数据结构定义
linenoise 行编辑库~2000 行嵌入式行编辑库，实现历史记录和行编辑
ShellState 结构体~300 行全局 shell 状态，保存所有配置和回调
点命令处理表~400 行所有 "." 命令的分发表
输出格式化函数~3000 行QRF 格式化引擎、mode 实现
点命令实现函数~8000 行每个点命令的具体逻辑
主循环 process_input~500 行读取-解析-执行的主循环
main 函数 & 参数解析~600 行命令行参数处理，初始化流程

这种"单文件 + 模板"的设计，与 SQLite 核心库的 amalgamation 哲学一脉相承——部署越简单越好。

主循环 process_input

shell 的核心是 process_input() 函数，它实现了经典的"读取-解析-执行"循环。我们来看看它的简化流程：

static int process_input(ShellState *p, const char *zSql){
    // 1. 跳过前导空白和注释
    while( sqlite3_isspace(*zSql) ) zSql++;

    // 2. 检查是否是点命令
    if( zSql[0]=='.' ){
        // 分离命令名和参数
        zCmd = &zSql[1];
        // 查找点命令表
        for(i=0; i<ArraySize(aDotCommands); i++){
            if( strcmp(zCmd, aDotCommands[i].zName)==0 ){
                // 执行点命令回调
                rc = aDotCommands[i].xCallback(p, nArg, azArg);
                break;
            }
        }
    } else {
        // 3. 普通 SQL——调用 sqlite3_prepare + sqlite3_step
        rc = sqlite3_prepare_v2(p->db, zSql, -1, &pStmt, &zTail);
        while( rc==SQLITE_OK ){
            rc = sqlite3_step(pStmt);
            if( rc==SQLITE_ROW ){
                // 4. 输出行数据（通过当前 mode 的回调）
                display_output(p, pStmt);
            }
        }
        sqlite3_finalize(pStmt);
        zSql = zTail; // 继续处理同一行的下一条 SQL
    }
}

注意到这个函数的关键设计：它同时处理点命令和 SQL。判断依据仅仅是第一个字符是不是 .。如果是点命令，就去查找 aDotCommands 数组；如果是普通字符，就当作 SQL 传给 SQLite 核心库执行。

还有一个重要的设计是 多语句支持——用户可以在同一行输入多条 SQL 语句，用分号隔开。核心库的 sqlite3_prepare_v2() 的第四个参数 zTail 会指向第一条语句之后的内容，循环继续处理下一条。

💡 有趣的是，process_input() 并不是一次读完整个输入——它按需读行。如果当前语句不完整（例如省略了分号就回车），shell 会输出 ...> 提示符等待继续输入。这个"连续行读取"的逻辑通过 continue_input() 辅助函数实现，它会检查括号是否匹配。

点命令解析

点命令是 SQLite shell 最强大的特性之一。超过 100 个点命令被定义在一个名为 aDotCommands[] 的数组中，这个数组是点命令的分发表：

typedef struct Command {
    const char *zName;      // 命令名
    const char *zHelp;      // 帮助文本
    int (*xCallback)(ShellState*, int, char**); // 回调函数
} Command;

static Command aDotCommands[] = {
    { "backup",     "BACKUP TABLE",        backup_callback       },
    { "bail",       "ON|OFF",              bail_callback         },
    { "database",   "LIST DATABASES",      database_callback     },
    { "dump",       "DUMP DATABASE",       dump_callback         },
    { "echo",       "ON|OFF",              echo_callback         },
    { "exit",       "EXIT SQLITE",         exit_callback         },
    { "headers",    "ON|OFF",              headers_callback      },
    { "help",       "SHOW HELP",           help_callback         },
    { "import",     "IMPORT DATA",         import_callback       },
    { "indexes",    "SHOW INDEXES",        indexes_callback      },
    { "mode",       "MODE SET",            mode_callback         },
    { "schema",     "SHOW SCHEMA",         schema_callback       },
    { "tables",     "SHOW TABLES",         tables_callback       },
    { "timer",      "ON|OFF",              timer_callback        },
    // ... 还有 80+ 个
};

这种设计非常直白——一个名字 + 一个回调函数。跟 VDBE 的 opcode 表是同样的思路。我最常用的几个点命令的实现值得我们快速看一下：

  • .tables — 本质上执行 SELECT name FROM sqlite_schema WHERE type='table' ORDER BY name，然后用默认格式输出结果。
  • .schema — 查询 sqlite_schema.sql 字段，即每个对象的 CREATE 语句。
  • .dump — 最复杂的点命令之一。它遍历所有表，为每张表生成完整的 INSERT 语句序列，并且智能处理自引用依赖（通过 PRAGMA foreign_keys 暂时禁用外键检查）。
  • .import — 读取 CSV 或其它分隔符文件，使用核心库的 sqlite3_import() 函数实现批量导入。
值得注意的工程决策：点命令不是通过 SQL 解析器解析的——它们是 shell 自己解析的。这意味着点命令不需要分号结尾，也不遵循 SQL 语法规则。但这也意味着点命令 不能 通过 SQL 连接驱动（如 libpq 风格的 API）发送，只能在交互式 shell 中使用。这个设计是有意为之——点命令是面向"人"的，不是面向"程序"的。

3.53.0 QRF 格式化库

在 2026 年发布的 SQLite 3.53.0 中，shell 引入了一个重要的新组件——QRF（Query Result Formatter）格式化库。它的核心目标是统一和美化查询结果的输出格式。

以大家最熟悉的 .mode box 为例，它输出的表格自带 Unicode 框线字符：

┌──────┬──────────┐
│  id  │  name    │
├──────┼──────────┤
│  1   │  Alice   │
│  2   │  Bob     │
│  3   │  Charlie │
└──────┴──────────┘

这个效果是靠 QRF 库实现的。它的核心数据结构是一个名为 QRFormatter 的结构体：

typedef struct QRFormatter {
    int nCol;           // 列数
    int *anWidth;       // 每列的最大显示宽度
    char **azTitle;     // 列标题
    char ***azData;     // 行数据（nRow x nCol）
    int nRow;           // 当前行数
    int nRowAlloc;      // 分配的行数容量
    void (*xRender)(struct QRFormatter*);  // 渲染回调
} QRFormatter;

QRF 的工作流程分为三个阶段：

  1. 收集阶段 — qrf_add_row() 将每一行的数据加入 azData 数组，同时更新 anWidth（每列的最大显示宽度）。
  2. 计算阶段 — 根据每列宽度、分隔符宽度、边框字符宽度，计算出整个表格的总宽度。
  3. 渲染阶段 — 调用 xRender 回调，根据当前的 mode（box、column、csv、json 等）生成最终的输出字符串。

.mode box 使用的是 Unicode 制表符（U+2500 ~ U+257F），这些字符在 shell.c.in 中以 UTF-8 编码直接硬编码。QRF 还支持 .mode box2（圆角框线）、.mode table（ASCII 框线）等多种风格。

💡 QRF 库的实现体现了 SQLite 团队的一贯作风——先把基础功能做对，再考虑性能优化。3.53.0 刚引入 QRF 时，所有行数据都先缓存到内存中再统一渲染。对于千万行级别的查询结果，这显然有内存压力。因此 3.54.0 中引入了"流式渲染"优化——当结果超过一定行数时，QRF 切换到逐行输出模式，不再缓存全部数据。这种"渐进式优化"的思路值得我们学习。

输出格式控制

shell 的输出格式控制是一个经典的"策略模式"案例。核心是 mode_callback() 函数，它处理 .mode 点命令：

static int mode_callback(ShellState *p, int nArg, char **azArg){
    const char *zMode = azArg[0];
    if( strcmp(zMode, "csv")==0 ){
        p->mode = MODE_Csv;
        p->colSep = ",";
        p->rowSep = "\r\n";
    } else if( strcmp(zMode, "column")==0 ){
        p->mode = MODE_Column;
        p->colSep = "  ";
        p->rowSep = "\n";
    } else if( strcmp(zMode, "box")==0 ){
        p->mode = MODE_Box;
        p->cMode = CMODE_Unicode;
        // 启用 QRF 格式化
        p->xOutputCallback = qrf_output_callback;
    } else if( strcmp(zMode, "json")==0 ){
        p->mode = MODE_Json;
        p->xOutputCallback = json_output_callback;
    }
    // ... 还有 markdown、html、insert、quote、tabs 等
}

每个 mode 对应一个 ShellState.mode 枚举值和一个 xOutputCallback 函数指针。当 VDBE 返回一行数据时，display_output() 函数会调用当前的回调来格式化输出。这种"回调函数指针"的设计让添加新的输出格式变得非常容易——加一个枚举值、写一个回调函数、在 mode_callback 中注册即可。

.separator 和 .headers 的实现就更简单了——它们只是修改 ShellState 结构体中的几个字段，输出回调函数在渲染时读取这些字段决定行为。

历史与编辑

SQLite shell 的行编辑功能是靠自己实现的一个迷你库——linenoise。没错，SQLite 没有依赖 GNU Readline，而是直接内置了一个精简的行编辑库。这是 SQLite 的一贯风格——"能不依赖外部库就不依赖"。

linenoise 是 Salvatore Sanfilippo（Redis 的作者）开发的一个轻量级行编辑库，全部代码只有几百行。SQLite 将其嵌入到 shell.c.in 中，并做了少量修改以适配自己的构建系统。

// linenoise 的核心结构
typedef struct linenoiseHistoryEntry {
    char *line;          // 历史记录行内容
    char *timestamp;     // 时间戳（可选）
} linenoiseHistoryEntry;

static linenoiseHistoryEntry *history;  // 历史记录数组
static int history_len;                 // 当前记录数
static int history_max_len = 100;       // 最大记录数（默认100）

linenoise 的工作原理非常有趣：

  • 它使用 tcgetattr()/tcsetattr() 将终端设置为"原始模式"（raw mode），这样每个按键都不会经过终端的行缓冲，而是直接传给程序。
  • 收到按键后，linenoise 判断是普通字符（直接插入）还是控制序列（如方向键、Backspace、Ctrl+C 等），然后更新内部的 linenoiseState 缓冲区。
  • 方向键上/下触发历史记录遍历，左/右触发光标移动。
  • 输入完成后，新的行被追加到 history 数组，同时写入 .sqlite_history 文件。

在 SQLite 3.52.0 之后，shell 还增加了对 .history 点命令的支持——它可以显示、搜索和管理历史记录。它的实现就是直接读取 .sqlite_history 文件并将其内容格式化为表格。

⚠️ 如果你在 shell 中按 Ctrl+C，你会注意到 SQLite 不会直接退出，而是中断当前查询。这个行为是在 signal_handler() 中实现的——它设置一个标志位 p->interrupt，核心库在 sqlite3_step() 的每次循环中检查这个标志，实现优雅中断。这比粗暴地终止进程要安全得多——正在进行的事务可以正确回滚。

实战：自定义 .mycommand 扩展点命令

理解源码最好的方式就是动手改一改。我们来实战练习——在 shell 中添加一个自定义点命令 .mycommand。

首先，在 aDotCommands[] 数组中添加一行：

// 在 aDotCommands 数组中添加：
{ "mycommand", "MY CUSTOM COMMAND", mycommand_callback },

然后，实现回调函数。这里演示一个"列出当前数据库所有表和行数统计"的命令：

static int mycommand_callback(ShellState *p, int nArg, char **azArg){
    sqlite3_stmt *pStmt;
    const char *zSql = "SELECT name AS '表名', "
                       "   (SELECT COUNT(*) FROM \"main\".\"??\") AS '行数' "
                       "FROM sqlite_schema WHERE type='table'";
    // 注意：这里为了可读性简化了 SQL，实际需要动态构建
    int rc = sqlite3_prepare_v2(p->db,
        "SELECT name FROM sqlite_schema WHERE type='table'", -1, &pStmt, 0);
    if( rc!=SQLITE_OK ) return SQLITE_ERROR;
    // 输出设置为 box 模式风格
    p->mode = MODE_Box;
    p->cMode = CMODE_Unicode;
    // 逐表查询行数并输出
    while( sqlite3_step(pStmt)==SQLITE_ROW ){
        const char *zTab = (const char*)sqlite3_column_text(pStmt, 0);
        // 动态构建 COUNT(*) 查询
        char zCount[256];
        sqlite3_snprintf(256, zCount,
            "SELECT COUNT(*) FROM \"%w\"", zTab);
        // ... 执行并输出
    }
    sqlite3_finalize(pStmt);
    return 0;
}

编译时，你需要先运行 tool/mkshellc.tcl 生成 shell.c，然后使用修改后的 shell.c.in。或者更直接——直接修改生成的 shell.c 然后编译：

gcc -o sqlite3-custom shell.c sqlite3.c -lpthread -ldl -lm

这就是 SQLite shell 可扩展性的体现——所有核心数据结构和回调接口都是公开的，开发人员可以在不修改 SQLite 核心库的情况下，定制完全属于自己的命令行工具。

💡 SQLite 官方其实并不鼓励用户直接修改 shell.c.in，因为合并时会增加冲突。更好的做法是在 shell.c.in 的末尾追加自定义命令——模板脚本在合并时会把 #include "shell_extra.c" 这类模式也一并处理。你只需要创建自己的 shell_extra.c，然后在模板中加一行 include 即可。
