第15章：网络 IO 架构

大家好，我是你们的讲师。前面两章我们花了大量精力讲 ae 事件循环——理解了 Redis 如何利用 epoll/kqueue 实现单线程的高并发 IO 处理。但有一个问题我相信很多同学心里一直悬着：Redis 的"单线程"究竟是什么意思？6.0 引入的 IO 多线程时到底改了些什么？这一章我们就来彻底拆解 Redis 的网络 IO 架构，从最底层的 socket 创建，到中间 connection 抽象层，再到 IO 多线程的调度机制，最后到输出缓冲区的管理。读完这一章，你就能回答面试官那句灵魂拷问："Redis 的单线程模型能支撑 10 万 QPS，瓶颈在哪里？"

Redis 网络模型演进史

Redis 的网络模型不是一成不变的，它经历了四次重要的架构变迁。理解这个演进过程，比直接看当前的代码更有价值，因为每一步变迁都对应着一个真实的生产瓶颈。

1.0–5.0：纯单线程时代

从 2009 年 Redis 诞生到 2020 年的 5.x 版本，Redis 一直是一个严格的单线程模型——一个主线程既负责 读 socket → 解析命令 → 执行命令 → 写 socket 的全链路。这个模型的优点非常明显：没有锁竞争、没有上下文切换、代码简单到令人发指。但它有一个不可忽视的瓶颈：当一条命令产生的网络数据量很大（比如 MGET 返回 10MB 数据），或者 PUBLISH 需要向 1000 个订阅者同时写数据时，写 socket 操作本身会阻塞事件循环，导致其他客户端的请求迟迟得不到处理。

6.0：IO 多线程引入

Redis 6.0 引入了一个关键优化：将网络 IO 操作（读请求和写响应）从主线程中剥离，交给多个 IO 线程并行处理。注意，这里剥离的仅仅是网络 IO，命令的执行仍然在主线程中完成。因为 Redis 的作者们发现，网络 IO（特别是大规模写）才是单线程场景下的主要瓶颈，而命令执行本身（通常是 O(1) 或 O(log N) 操作）在绝大多数场景下并不需要并行。

7.x/8.x：IO 线程持续增强

7.x 版本优化了 IO 线程的负载均衡策略，8.4 更是引入了 per-client IO 线程分配，进一步减少了线程间的同步开销。此外，8.x 在 IO 线程中解锁了对更多命令类型的支持，使得 IO 线程的利用率更高。

Redis 版本 网络模型 IO 线程数 命令执行 关键改进
1.0–5.0 纯单线程 0 主线程 无锁、简单、10万+ QPS
6.0 IO 多线程（读+写异步） 可配置（默认 3） 主线程 网络 IO 并行化，大 Key 读写不阻塞
7.x IO 多线程增强 可配置 主线程 更好负载均衡、减少线程间锁竞争
8.x per-client IO 线程分配 可配置 主线程 细粒度分配、减少同步开销
核心洞察：Redis 即便在 6.0 引入了 IO 多线程，命令执行仍然是单线程的。这意味着 KEYS * 或者 FLUSHALL 这样的 O(N) 命令仍然会阻塞所有其他请求。IO 多线程解决的是"网络 IO 开销"而不是"命令执行开销"。

连接建立

Redis 作为一个 TCP 服务器，所有的基础设施都构建在 POSIX socket API 之上。antirez 将这些底层操作封装在 anet.c 中，提供了简洁的错误处理和重试逻辑。

anet.c — TCP Socket 封装

anet.c 是 Redis 最底层的网络抽象层，它做的事情说起来很简单：创建 socket、绑定端口、监听、接受连接。但它在每个细节上都做了工业级的处理：

  • 地址复用：调用 setsockopt 设置 SO_REUSEADDR，避免服务器重启时因 TIME_WAIT 状态而无法绑定端口。
  • TCP_NODELAY：关闭 Nagle 算法，避免因小数据包聚合引入的毫秒级延迟。这对 Redis 这种以微秒计时的服务至关重要。
  • SO_KEEPALIVE：开启 TCP 保活探测，检测死连接。默认间隔 2 小时，可通过 tcp-keepalive 配置调整。
  • SO_REUSEPORT（可选）：在 Linux 3.9+ 内核上支持多进程监听同一端口，配合 SO_REUSEPORT 实现内核级的负载均衡。
/* anetTcpServer — 创建并绑定 TCP 监听 socket */
int anetTcpServer(char *err, int port, char *bindaddr, int backlog) {
    int s;
    struct sockaddr_in sa;

    /* 创建 IPv4 TCP socket */
    if ((s = socket(AF_INET, SOCK_STREAM, 0)) == -1) {
        anetSetError(err, "socket: %s\n", strerror(errno));
        return ANET_ERR;
    }

    /* 设置 SO_REUSEADDR — 允许地址重用 */
    anetSetReuseAddr(err, s);

    /* 设置 TCP_NODELAY — 禁用 Nagle 算法 */
    anetSetTcpNoDelay(err, s);

    /* 设置 SO_KEEPALIVE — 保活探测 */
    anetSetKeepAlive(err, s, ANET_KEEPALIVE_INTERVAL);

    /* 绑定并监听 */
    memset(&sa, 0, sizeof(sa));
    sa.sin_family = AF_INET;
    sa.sin_port = htons(port);
    sa.sin_addr.s_addr = bindaddr ? inet_addr(bindaddr) : htonl(INADDR_ANY);

    if (bind(s, (struct sockaddr*)&sa, sizeof(sa)) == -1) {
        anetSetError(err, "bind: %s\n", strerror(errno));
        close(s);
        return ANET_ERR;
    }

    if (listen(s, backlog) == -1) {
        anetSetError(err, "listen: %s\n", strerror(errno));
        close(s);
        return ANET_ERR;
    }
    return s;
}

当 ae 事件循环检测到监听 socket 可读时，会调用 acceptTcpHandler 接受新连接。

/* acceptTcpHandler — 接受 TCP 连接并注册到事件循环 */
void acceptTcpHandler(aeEventLoop *el, int fd, void *privdata, int mask) {
    int cport, cfd, max = MAX_ACCEPTS_PER_CALL;
    char cip[NET_IP_STR_LEN];

    while (max--) {
        cfd = anetTcpAccept(server.neterr, fd, cip, sizeof(cip), &cport);
        if (cfd == ANET_ERR) {
            if (errno != EAGAIN)
                serverLog(LL_WARNING,
                    "Accepting client connection: %s", server.neterr);
            return;
        }
        /* 为每个新连接创建 client 结构体 */
        acceptCommonHandler(cfd, 0, cip);
    }
}

注意 MAX_ACCEPTS_PER_CALL 这个限制——这是 antirez 的一个经典设计模式：每次事件循环最多接受 N 个新连接，防止瞬间大量连接涌入导致事件循环长时间停留在 accept 上，延误了已有连接的读写。

connection 抽象层

在 Redis 6.0 之前，网络连接的操作直接使用 POSIX socket API。但随着 TLS 支持的需求越来越强烈，antirez 意识到需要一层抽象来屏蔽底层传输协议的差异。于是 connection 抽象层在 6.0 被引入。

ConnectionType — 虚函数表设计

Redis 实现了一个轻量级的虚函数表模式——ConnectionType 结构体定义了一组函数指针，不同的传输类型（普通 TCP socket、TLS socket）各自提供一个实现表：

/* ConnectionType — Redis 连接类型虚函数表 */
typedef struct ConnectionType {
    void (*init)(void);                           /* 全局初始化 */
    void (*cleanup)(void);                        /* 全局清理 */
    int (*connect)(struct connection *conn,       /* 发起连接 */
                   const char *addr, int port,
                   const char *src_addr,
                   const char **err);
    int (*write)(struct connection *conn,          /* 写入数据 */
                 const void *data, size_t data_len);
    int (*read)(struct connection *conn,           /* 读取数据 */
                void *buf, size_t buf_len);
    int (*accept)(struct connection *conn,         /* 接受连接 */
                  ConnectionCallbackFunc accept_handler);
    int (*close)(struct connection *conn);         /* 关闭连接 */
    int (*set_write_handler)(struct connection *conn,  /* 注册写事件 */
                             ConnectionCallbackFunc handler, int barrier);
    int (*set_read_handler)(struct connection *conn,   /* 注册读事件 */
                            ConnectionCallbackFunc handler);
    const char *(*get_last_error)(struct connection *conn);
    int (*blocking_connect)(struct connection *conn,   /* 阻塞连接 */
                            const char *addr, int port,
                            long long timeout);
    int (*sync_write)(struct connection *conn,         /* 同步写 */
                      char *ptr, ssize_t size,
                      long long timeout);
    int (*sync_read)(struct connection *conn,          /* 同步读 */
                     char *ptr, ssize_t size,
                     long long timeout);
    int (*sync_readline)(struct connection *conn,      /* 同步读行 */
                         char *ptr, ssize_t size,
                         long long timeout);
} ConnectionType;

CT_Socket 和 CT_TLS 是两个内置的实现。默认情况下连接使用 CT_Socket，当 tls-port 开启时，TLS 连接使用 CT_TLS。上层代码通过 connRead、connWrite 等统一接口操作，完全不需要关心底层是普通 socket 还是 TLS：

/* 统一接口 — 不管底层是 TCP 还是 TLS，都是这么读写 */
static inline int connRead(connection *conn, void *buf, size_t buf_len) {
    return conn->type->read(conn, buf, buf_len);
}

static inline int connWrite(connection *conn, const void *data, size_t data_len) {
    return conn->type->write(conn, data, data_len);
}

static inline int connClose(connection *conn) {
    return conn->type->close(conn);
}
小贴士：这种虚函数表模式在 C 语言中非常常见——Linux VFS（文件系统）、Nginx 的事件模块、libuv 的句柄类型，都用的是同样的手法。如果你看懂了 Redis 的 ConnectionType，再去读其他 C 项目的"多态"实现会觉得很亲切。

IO 多线程机制

这是本章最核心的部分。Redis 6.0 引入了 IO 多线程，但它的实现方式可能和你想象的完全不同。

启动与初始化

initThreadedIO 在 Redis 启动时创建 IO 线程池。线程数量由 io-threads 配置指定（默认 3），但请注意，即使配置了 4 个 IO 线程，主线程也参与 IO 处理，所以实际的并行 IO 线程数是 3。

IO 线程的工作模式

IO 线程的处理模型非常精巧。它使用一个互斥锁 + 自旋锁的复合同步机制：

  • io_threads_pending — 原子计数器，标记有多少个待处理的客户端。IO 线程通过忙等待（自旋）检查这个计数器，一旦为 0 就退出。
  • io_threads_list — 每个 IO 线程维护一个客户端链表，主线程将待处理的客户端按 round-robin 分配到各个线程的链表中。
  • io_threads_mutex — 用于 IO 线程启动时的屏障同步，确保所有线程同时开始工作。
/* IO 线程主函数 — 每个 IO 线程都在这里循环执行 */
void *IOThreadMain(void *myid) {
    long id = (unsigned long)myid;
    char thdname[16];

    snprintf(thdname, sizeof(thdname), "io_thd_%ld", id);
    setThreadName(thdname);

    while (1) {
        /* 自旋等待：直到有客户端分配给本线程 */
        for (int i = 0; i < 1000000; i++) {
            if (io_threads_pending[id] != 0) break;
        }

        if (io_threads_pending[id] == 0) {
            /* 自旋超时，使用互斥锁睡眠等待 */
            pthread_mutex_lock(&io_threads_mutex[id]);
            pthread_mutex_unlock(&io_threads_mutex[id]);
            continue;
        }

        /* 处理本线程的客户端列表 */
        listIter li;
        listNode *ln;
        listRewind(io_threads_list[id], &li);
        while ((ln = listNext(&li)) != NULL) {
            client *c = listNodeValue(ln);

            if (io_threads_op == IO_THREADS_OP_WRITE) {
                writeToClient(c, 0);    /* 写客户端输出缓冲区 */
            } else if (io_threads_op == IO_THREADS_OP_READ) {
                readQueryFromClient(c->conn);  /* 读取客户端请求 */
            } else {
                serverPanic("io_threads_op 无效");
            }
        }

        /* 处理完毕，清空列表并置 pending 为 0 */
        listEmpty(io_threads_list[id]);
        io_threads_pending[id] = 0;
    }
}

IO 线程的调度时机

IO 线程不是在 every loop 都运行的。主线程在处理客户端 IO 时，会先尝试将工作分配给 IO 线程，如果 IO 线程不可用或者没有启用，就退化为传统的单线程模式。核心调度入口在 beforeSleep 和 afterSleep 函数中：

  • beforeSleep：在进入 ae 事件循环阻塞之前，检查是否有客户端需要写数据。如果有，启动 IO 线程进行并行写。
  • handleClientsWithPendingWritesUsingThreads：将待写客户端按 round-robin 分配给 IO 线程，然后主线程自己也处理一部分，最后等待所有 IO 线程完成。
踩坑实录：之前有团队在生产环境开启 IO 多线程后，发现 QPS 反而下降了。排查后发现是 io-threads-do-reads yes 开启了 IO 线程读，但机器只有 2 个 CPU core——线程上下文切换的成本超过了并行 IO 的收益。所以 IO 多线程不是越多越好，官方建议仅在 CPU core ≥ 4 的网络密集型场景下开启。

输出缓冲区管理

当客户端发送一条命令，Redis 执行完毕后，结果需要写回给客户端。这个"写回"的过程不是简单的 send()，Redis 使用了一套两级缓冲机制。

client.reply 链表

每个 client 结构体中有一个 reply 链表，所有待发送的响应数据都以 clientReplyBlock 的形式挂在这个链表上：

/* 输出缓冲区链表节点 */
typedef struct clientReplyBlock {
    int refcount;          /* 引用计数 */
    size_t size, pos;      /* 总大小、当前读取位置 */
    char buf[];            /* 柔性数组，实际数据 */
} clientReplyBlock;

当 addReply 被调用时，Redis 会分配 clientReplyBlock 并追加到 client->reply 链表末尾。然后在 beforeSleep 或事件循环的写事件触发时，调用 writeToClient 将链表中的数据逐步写回。

固定缓冲 + 动态缓冲

Redis 的输出缓冲分为两级：

  • 固定缓冲（client->buf）：一个 16KB 的栈上数组。小响应直接写在固定缓冲里，避免动态分配的开销。
  • 动态缓冲（client->reply）：当响应数据超过 16KB，或者固定缓冲已满，数据被写入 reply 链表（使用 zmalloc 分配）。

输出缓冲限制

为了防止某个大 Key 或慢客户端拖垮服务器内存，Redis 提供了 client-output-buffer-limit 配置，对普通客户端、发布订阅客户端、从库客户端分别设置：

# redis.conf 中的默认配置
client-output-buffer-limit normal 0 0 0           # 普通客户端无限制
client-output-buffer-limit replica 256mb 64mb 60  # 从库：硬限制256MB，软限制64MB/60s
client-output-buffer-limit pubsub 32mb 8mb 60     # 订阅：硬限制32MB，软限制8MB/60s

硬限制（hard limit）是一旦超过立即断开连接；软限制（soft limit）是超过后持续一段时间才断开。从库的输出缓冲区限制尤为重要，因为全量同步时的 RDB 传输可能瞬间撑爆缓冲区。

踩坑实录：在一次生产事故中，某团队将主从库之间的网络延迟从 1ms 提升到了 50ms（跨机房迁移），结果从库频繁与主库断开重连。排查发现，由于网络变慢，主库向从库同步数据的输出缓冲区迅速堆积，达到了默认的 soft limit（64MB/60s），从库被断开。断开后重新同步又产生新的缓冲区堆积，形成"断连→重连→再断连"的死循环。解决方案有两个：一是增大 client-output-buffer-limit replica 的值，二是优化网络延迟。这个案例告诉我们，输出缓冲区限制是保护机制，但设置不当本身就是灾难的源头。

总结

回顾一下这一章的内容：我们先从 Redis 网络模型的演进看到了设计决策背后的权衡——单线程的简洁与 IO 多线程的收益之间有一条非常清晰的边界。然后我们深入 anet.c 看了连接建立的底层细节，理解了 ConnectionType 虚函数表如何优雅地支持 TLS 扩展。接着我们解剖了 IO 多线程的调度机制——自旋+互斥的同步策略、round-robin 的负载分发。最后我们看了输出缓冲区的管理，两级缓冲策略和 client-output-buffer-limit 的详细语义。
