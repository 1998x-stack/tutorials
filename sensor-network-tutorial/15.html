第15章 WSN安全基础

大家好，我是你们的讲师。想象这样一个场景：你刚刚在办公楼的智能门锁系统中部署了一套基于WSN的无线门禁——每个门锁节点通过ZigBee网络与中央控制器通信，员工刷工牌开门，系统记录进出日志。一切都显得完美而高效。然而有一天，安保部门发现一个诡异的现象：每天凌晨3点，14楼走廊的门会莫名其妙地自动打开，然后关上。监控录像里空无一人。你查遍了中央控制器的日志，没有任何异常记录——没有非授权卡号的开门请求，没有异常的登录记录。

问题出在哪？答案是：攻击者根本不需要破解你的门禁卡或登录中央控制器。他只需要一台笔记本加一个ZigBee嗅探器，在你门锁上电入网的那一刻录制下完整的无线通信帧序列。然后，在凌晨3点，把录制的"开门指令"帧原封不动地重放一次——你的门锁就乖乖打开了。这就是WSN安全最典型的噩梦：无线媒介天生开放，攻击者不需要物理接触就能发动攻击。从本章开始，我们将用4章的篇幅深入探索传感网的安全防线。

本章核心问题： WSN的安全设计为什么不能照搬互联网的成熟方案？资源受限的节点上，我们如何在"够用"和"安全"之间找到平衡？学完本章，你将掌握WSN安全威胁模型的基础分析方法和核心密码学工具的使用原则。

WSN安全为什么特殊？

很多人一听说"安全"，第一反应就是："加个HTTPS/TLS不就行了？"在传统互联网上，这确实是标准答案。但到了WSN领域，事情远没那么简单。WSN面临四个独特的安全挑战，每一个都让传统方案失效。

资源受限——加密也是有代价的

一个典型的WSN节点，比如基于TI CC2530的ZigBee终端，只有8KB RAM和256KB Flash，主频32MHz。在这上面跑一次RSA-2048签名运算需要数秒甚至数分钟，耗电量相当于节点正常工作数小时的能耗。更致命的是，TLS握手需要多次往返通信和大量的证书解析操作——对于发送一个温度数据包只需几个毫秒的WSN来说，这个开销完全不可接受。因此，WSN中几乎不使用TLS，而是使用为资源受限环境专门设计的轻量级安全协议，如IEEE 802.15.4安全子层和DTLS（Datagram TLS的压缩版本）。

无线广播——信道天生开放

有线的网线需要物理接入才能监听。无线信号则不同——你的射频信号在空中传播，任何在通信范围内的设备都可以接收。这意味着攻击者不需要潜入你的机房、不需要插拔任何线缆，只需要在停车场停一辆车，架设一根全向天线，就能捕获你全部的网络流量。这就是所谓的"停车场攻击"（Parking Lot Attack）。

物理暴露——节点可能"被抓"

温湿度传感器挂在农田的桅杆上、结构振动传感器嵌在桥梁的混凝土中、烟雾探测器安装在天花板上——这些节点部署后通常无人值守，攻击者可以物理接触甚至直接拆解节点。一旦节点被物理捕获，存储在Flash中的密钥、固件代码全部暴露。攻击者可以提取密钥后伪装成合法节点加入网络。这在传统互联网中是罕见的——你的服务器放在机房里，有门禁、有监控、有保安。

无人值守——发现攻击时可能已经晚了

一个办公室网络发现异常流量，网管几分钟内就能定位处理。但一个部署在内蒙古草原上的环境监测WSN，节点出了问题需要派人驱车数百公里去现场。安全事件从发生到被发现，中间的时间窗口非常大。因此WSN安全不仅需要防御能力，还需要自愈能力——节点应当能在无人类干预的情况下检测异常并自我恢复。

安全威胁模型

在谈具体的防御手段之前，我们先建立一个清晰的威胁模型——不了解敌人长什么样，你建的城墙就是盲目的。

攻击面分析：从四层协议栈看安全缺口

按照WSN的五层协议模型，每一层都有独特的安全脆弱点：

协议层 典型攻击 攻击描述
物理层 干扰攻击（Jamming） 在传感网工作频段发射大功率噪声信号，使所有节点无法正常通信
链路层 碰撞攻击、耗尽攻击 精心构造的帧碰撞导致持续重传，耗尽节点能量
网络层 虫洞、Sybil、槽洞攻击 破坏路由逻辑，吸引或篡改网络层数据流
应用层 数据篡改、重放攻击 截获并篡改应用数据，或重放合法指令欺骗接收端

注意，这张表格只是"预告片"——下一章我们会深入每种攻击的机理和防御措施。本章的重点是我们如何在协议栈的基础层（即安全子层）建立防线。

攻击者分类

不是所有的攻击者都拥有相同的能力。理解攻击者的等级有助于你选择"够用"的安全方案，而不是过度设计：

  • Class I（被动监听者）：具备低成本嗅探设备，能捕获和分析空中无线信号。不能发送伪造帧。最常见，风险等级低。
  • Class II（主动干扰者）：在Class I基础上能发送自定义帧和干扰信号。能发起重放攻击、碰撞攻击。需要软件定义无线电（SDR）或同频射频模块。中等风险。
  • Class III（物理捕获者）：能物理获取节点，进行JTAG/SWD调试、Flash读取、密钥提取。需要专业硬件技能。高风险。
  • Class IV（国家级的对手）：具备解密能力、专用硬件、大规模部署能力。能够实施侧信道攻击（功耗分析、时序分析）。极高风险但通常不在民用WSN的防御范围内。
实践建议： 大多数民用WSN项目只需要防御Class I和Class II级别的攻击者。物理节点捕获后密钥必然泄露，因此设计上应假设"某个节点一定被攻破"，并确保单个节点被攻破不影响整个网络安全——这就是"零信任"思想在WSN中的体现。

CIA三元组在WSN中的含义

信息安全领域的CIA三元组——机密性（Confidentiality）、完整性（Integrity）和可用性（Availability）——在WSN中有各自独特的权重和实现方式。

机密性（Confidentiality）

确保无线信道中的数据不被非授权方解读。在WSN中，机密性通过链路层加密实现——每一帧的有效载荷在发送前加密，接收后解密。以IEEE 802.15.4为例，其安全子层支持AES-CCM*模式，对MAC帧体部分进行加密。但要注意：帧头（包括源地址、目的地址、帧控制域等）通常不加密，因为中继节点需要读取地址字段来转发。

重要提醒： 不加密帧头意味着"元数据"是暴露的。攻击者虽然不知道你在发送什么内容，但他知道"A地址的节点在向B地址的汇聚节点发送数据"——这可能本身就泄露了关键信息。例如，指挥控制中心的地址字段频繁出现在网络层头部，攻击者就能推断出哪个节点是指挥节点。这就是所谓的"流量分析攻击"。

完整性（Integrity）

确保数据在传输过程中未被篡改。WSN中使用消息认证码（Message Authentication Code，MAC）来保证完整性。发送方对数据计算MAC并附在帧尾，接收方用共享密钥验证MAC。任何数据位的修改都会导致MAC校验失败。注意这里的MAC不要和媒体访问控制（Medium Access Control）混淆——同一个缩写，含义天差地别。

可用性（Availability）

确保网络在攻击下仍能提供服务。对WSN来说，可用性往往是最容易被忽视却最难保证的目标。一个简单的持续干扰攻击（Continuous Jamming）就可以让整个区域内的全部节点"失声"。防御可用性威胁通常不是靠密码学，而是靠物理层抗干扰技术（跳频扩频）、拓扑冗余和自组织协议来实现。

WSN的CIA优先级排序： 大多数WSN应用中，可用性 > 完整性 > 机密性。原因很实际——一个无法上报数据的传感器网络，加密得再严也无用。这个排序与传统互联网（机密性优先）形成了鲜明对比。

对称加密：AES-CCM

WSN中最重要的加密算法是AES-CCM，它是IEEE 802.15.4标准指定的安全方案。CCM代表Counter with CBC-MAC，它同时提供了加密（CTR模式）和认证（CBC-MAC）功能，是一种"认证加密"（Authenticated Encryption, AEAD）模式。

你可能要问：为什么不用更常见的AES-GCM？原因有两个：

  • 硬件支持：802.15.4射频芯片（CC2530、CC2420等）内置了AES-CCM硬件加速器，但绝大多数不支持GCM的GHASH乘法运算。在软件中实现GHASH会消耗大量CPU和能量。
  • 短帧友好：WSN帧通常只有几十字节（802.15.4最大帧长127字节，有效载荷约80字节）。CCM在小包场景下认证开销更可控。GCM的GHASH需要额外的密钥派生步骤，对短帧来说效率不高。

AES-CCM的安全性依赖于一个关键原则：每个密文使用的Nonce必须唯一。在802.15.4中，Nonce通常由帧计数器、源地址和帧控制字段联合构成。如果Nonce重用，认证加密的安全性会完全崩溃。

// AES-CCM* 加密与认证（IEEE 802.15.4 安全子层）的伪代码流程
// 符号说明：
//   K     - 128位对称密钥（由密钥管理模块提供）
//   N     - Nonce = SourceAddress | FrameCounter | SecurityLevel
//   A     - 附加认证数据（帧头部分，不加密但需认证）
//   M     - 明文消息（MAC帧载荷）
//   C     - 密文
//   T     - 消息认证码（Message Authentication Code, 4/8/16字节）

function aes_ccm_encrypt_auth(K, N, A, M):
    // Step 1: 构建认证标记
    B0 = FormFlag0() | N | length(M)       // 标志字节0 + Nonce + 消息长度
    B = [B0] + PadTo16(A) + PadTo16(M)     // B0 + 填充后的A + 填充后的M

    // Step 2: 计算CBC-MAC
    X = 0^128                               // 初始向量全零
    for each 16-byte block Bi in B:
        X = AES_encrypt(K, X XOR Bi)        // CBC链式加密
    T = truncate(X, MAC_LEN)                // 取前MAC_LEN字节作为认证码

    // Step 3: CTR模式加密
    Ctr0 = FormFlag1() | N | 0x0000         // 计数器初始值
    S0   = AES_encrypt(K, Ctr0)             // 加密计数器0（用于掩码T）
    T_masked = T XOR truncate(S0, MAC_LEN)  // 对认证码做一次掩码

    // 对每个明文块
    C = []
    Ctr_i = Ctr0 + 1                        // 计数器自增
    for each 16-byte block Mj in M:
        Sj = AES_encrypt(K, Ctr_i + j)      // 生成密钥流
        Cj = Mj XOR truncate(Sj, len(Mj))   // 明文 XOR 密钥流 = 密文
        C = C || Cj

    // Step 4: 输出密文 + 掩码后的认证码
    return (C, T_masked)

function aes_ccm_decrypt_verify(K, N, A, C, T_received):
    // Step 1: 解密（CTR模式解密 = 加密，同操作）
    Ctr0 = FormFlag1() | N | 0x0000
    S0   = AES_encrypt(K, Ctr0)
    T_masked = T_received                   // 收到的掩码认证码

    M = []
    Ctr_i = Ctr0 + 1
    for each 16-byte block Cj in C:
        Sj = AES_encrypt(K, Ctr_i + j)
        Mj = Cj XOR truncate(Sj, len(Cj))  // 密文 XOR 密钥流 = 明文
        M = M || Mj

    // Step 2: 验证认证码
    T_computed = cbc_mac(K, N, A, M)        // 对解密后的数据进行CBC-MAC
    T_unmasked = T_masked XOR truncate(S0, MAC_LEN)

    if T_computed == T_unmasked:
        return (M, VALID)                   // 数据完整且来自可信源
    else:
        return (NULL, INVALID)              // 数据被篡改或来自非授权源
}

注意代码中的关键细节：认证在前，解密在后——这是安全设计的黄金法则。先解密再验证意味着你在校验之前就已经把未经认证的数据放入了内存，可能触发缓冲区溢出或其他攻击。所以在CCM设计下，接收端先验证MAC再决定是否使用解密后的数据。

轻量级公钥密码：ECC椭圆曲线

对称加密（AES-CCM）虽然高效，但它面临一个经典问题：密钥分发——两个从未通信过的节点如何安全地共享同一个密钥？这正是公钥密码的用武之地。

传统互联网使用RSA。但在WSN中，RSA-1024的模幂运算在8位微控制器上需要消耗大量时间和能量。幸运的是，椭圆曲线密码学（ECC, Elliptic Curve Cryptography）提供了更好的选择：在同等安全强度下，ECC的密钥长度远短于RSA。

安全等级 RSA密钥长度 ECC密钥长度 典型应用场景
80位（短期安全） 1024 bit 160 bit 非关键传感数据，快速密钥协商
112位（中级安全） 2048 bit 224 bit 工业WSN，门禁控制
128位（长期安全） 3072 bit 256 bit 医疗WSN，金融级安全
256位（极高安全） 15360 bit 521 bit 军事/政府级应用

以128位安全等级为例，ECC只需要256位密钥，而RSA需要3072位——差了12倍的存储空间。在RAM只有8KB的传感器节点上，这12倍意味着"可行"和"不可行"的区别。典型ECC实现（如TinyECC、MIRACL Light）在8位MCU上完成一次256位ECDH密钥协商约需1-2秒，消耗约5-10mJ能量——这是可以接受的。

ECC曲线选择建议： 在WSN中推荐使用Curve25519（X25519），它的设计对嵌入式实现极其友好——常数时间执行（抗侧信道攻击）、不需要检查是否为合法点、不需要随机数生成器。相比之下，NIST P-256曲线需要额外的点验证步骤，在受限平台上更容易出错。X25519是Libhydrogen和Monocypher等轻量级加密库的核心算法，强烈推荐。

密钥管理

加密算法选好了，但密钥从哪来？密钥管理是WSN安全中最棘手的问题之一。因为WSN节点数量大（成百上千个）、部署后无法干预、节点可能被捕获——密钥不能简单地"出厂时硬编码一个固定密钥"。以下是两种主流的WSN密钥管理方案。

密钥预分配：随机对密钥方案

这是经典的Eschenauer-Gligor方案。基本思想：从一个巨大的密钥池中，为每个节点随机分配一个密钥子集。两个需要通信的节点检查它们共享的密钥交集，如果存在共同密钥，就以此作为会话密钥。如果不存在，则通过密钥路径（多跳）间接建立。

简单来说，就像你参加一个行业交流会——每个人脖子上挂着一个写着"我会的技术"的牌子。如果你跟另一个人有相同的技术，你们就能直接开始深入交流。如果没有，就找共同认识的中间人介绍。

在实践中，这个方案的参数选择直接影响安全性：密钥池大小P、每个节点分配k个密钥、密钥路径长度。可以证明，当P足够大且k适中时，任意两个节点之间存在共享密钥的概率可以非常高（>99%），而单个节点被捕获泄露的信息量有限。

关键参数关系： 假设密钥池大小P=100000，每个节点分配k=250个密钥，则任意两个节点共享至少一个密钥的概率约为 1 - ((P-k)! / P!)^2 约等于 0.5。提高k到400，概率上升到约0.8。这就是"概率性密钥预分配"——你不需要保证每对节点都能直接通信，通过密钥路径间接建立即可。

动态密钥建立

预分配方案虽然简单，但安全性依赖于"大部分节点不被捕获"的假设。如果攻击者捕获了足够的节点（例如500个节点中的50个），就可能威胁到整个网络的密钥安全。动态密钥建立方案提供了更强的安全性：密钥不是在部署前固定的，而是在运行时通过安全协议协商生成。

一个完成度较高的方案是LEAP+（Localized Encryption and Authentication Protocol），它建立了四种不同类型的密钥，分别用于不同粒度的通信：

  • 个人密钥（Individual Key）：节点与汇聚节点共享的唯一密钥，用于安全单播通信。
  • 对密钥（Pairwise Key）：两个邻居节点之间共享的密钥，用于相邻节点间的安全通信。
  • 簇密钥（Cluster Key）：簇内所有节点共享的密钥，用于本地广播。
  • 全局密钥（Group Key）：全网共享的密钥，由汇聚节点维护和更新。

LEAP+的精妙之处在于，即使某个节点被攻破，攻击者也只获得该节点的个人密钥和部分对密钥——簇密钥和全局密钥可以通过定期更新来"绕过"被攻破节点。

认证与新鲜性

加密只解决了"别人看不懂"的问题。但你还面临两个问题：一是"我怎么知道消息是来自真正的A节点而不是冒充者？"（认证），二是"我怎么知道这条消息是刚刚发送的而不是攻击者昨天录下来的？"（新鲜性，Freshness）。

消息认证码（MAC）

如前面的AES-CCM流程所示，MAC在WSN中同时承担了认证和完整性验证的双重角色。发送方使用共享密钥K对数据计算MAC，接收方用同一个K重新计算并比较。如果一致，证明数据既未被篡改（完整性），又来自知道K的合法发送方（源认证）。

在IEEE 802.15.4中，MAC长度可以是4、8或16字节。MAC越长，安全强度越高，但通信开销也越大。对于大多数传感器数据（如温度读数），4字节MAC已经足够——攻击者伪造有效MAC的概率仅为2-32，在WSN的通信量下几乎不可能。

新鲜性保护：Nonce与时间戳

还记得本章开头的门锁故事吗？那就是重放攻击（Replay Attack）的经典案例。即使你加密了开门指令，攻击者不需要解密——他只需要完整地录制下加密后的帧序列，然后在合适的时候原样重放。接收端能正常解密（密文解密后正确的明文），认证也能通过（MAC也匹配），因为它本质上"就是"合法的帧——只不过是在错误的时间出现的。

防御重放攻击有三种常用方法：

  • 计数器（Frame Counter）：每个节点维护一个帧计数器，随每次发送递增。接收端只接受计数器值大于已记录最大值的帧。IEEE 802.15.4使用这种方式，缺点是节点重启后计数器重置，需要在Flash中持久化保存。
  • 时间戳（Timestamp）：发送方在帧中嵌入当前时间，接收端检查时间戳与本地时钟的差值是否在允许窗口内（例如±1秒）。缺点是需要全网时间同步（第11章讲过，时间同步本身就有开销）。
  • Nonce随机数：发送方在帧中嵌入一个随机数，接收端记录最近收到的所有Nonce，重复则拒绝。缺点是接收端需要维护Nonce列表，对大流量网络不友好。
真实案例 —— 固定密钥的智能门锁有多脆弱： 2019年，安全研究人员披露了某知名品牌ZigBee智能门锁的严重漏洞。该门锁使用出厂硬编码的128位密钥（所有门锁使用同一个密钥！），且没有帧计数器。攻击者用一只价值不到2000元的ZigBee嗅探器（如TI CC2531 USB Dongle + Wireshark）录制开门指令帧，之后可以无限次重放。更糟糕的真相是：该门锁甚至没有做任何形式的源身份认证——攻击者只要发送格式正确的帧就能开门，连重放都不需要。这个漏洞影响了当时市面上超过200万套门锁产品。这件事之后，ZigBee Alliance强制要求所有ZigBee 3.0认证设备必须实现Trust Center的安全引导（Security Bootstrapping）和动态密钥更新。教训很深刻：安全不是一个"加上就完了"的功能，而是需要贯穿产品全生命周期的系统工程。

对称加密 vs 非对称加密在WSN中的适用性

对比维度 对称加密（AES-CCM） 非对称加密（ECC）
计算开销 极低——硬件加速器下仅需数微秒/帧 较高——256位ECDH需1-2秒（8位MCU）
能耗 ~1-5μJ/帧（硬件加速） ~5-15mJ/次（密钥协商）
密钥管理 困难——需要安全的密钥分发机制 简单——公钥无需保密，私钥本地存储
扩展性 差——N个节点需管理O(N²)个对密钥 好——N个节点只需N个公私钥对
主要用途 数据加密传输（日常通信） 密钥协商和数字签名（少量关键操作）
抗节点捕获 差——密钥被提取后通信完全暴露 好——私钥被提取只影响单个节点

本章小结

WSN安全不是把互联网的TLS套过来就能解决问题的。资源受限、无线广播、物理暴露和无人值守四个特性决定了WSN安全必须"精打细算"。AES-CCM作为标准硬件加速的认证加密方案，是目前WSN链路层安全的基石；ECC提供了可接受的轻量级公钥密码方案，用于密钥协商；而密钥管理和新鲜性保护则是整个安全体系的"骨架"——没有它们，再好的加密算法也形同虚设。

不过，加密和认证只是安全的第一道防线。下一章我们将讨论另一个层面的安全问题——当攻击者不跟你"讲道理"的时候，他们通过干扰、冒充、路由欺骗等手段绕过你的加密防线。你将看到：即使AES-CCM坚不可摧，敌人照样有办法让你的传感网瘫痪。第16章，我们正式进入"攻击与防御"的硬核战场。
