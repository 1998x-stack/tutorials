第十七章：虚拟内存——地址翻译

你有没有想过这样一个问题：你电脑只有 16GB 物理内存，但你可以同时打开一个 20GB 的游戏、一个 4GB 的 IDE、外加十多个浏览器标签页——每个标签页还在不断开着新页面。这些程序占用的"虚拟空间"加起来远超物理内存，为什么系统还能正常运转？答案是虚拟内存（Virtual Memory）。

虚拟内存是计算机系统中最伟大的抽象之一。它不仅让物理内存看起来比实际大得多，还为每个进程提供了独立的地址空间，让进程之间互相隔离。理解虚拟内存的地址翻译机制，是理解系统性能和安全性的关键。本章我们从动机说起，再深入到页表、TLB 和缺页处理的细节。

17.1 虚拟寻址的动机

在没有虚拟内存的时代（或者在没有 MMU 的嵌入式系统上），程序直接访问物理地址。这种方式的问题显而易见——多个程序同时运行时，它们必须事先协商好内存分区，否则互相覆盖。虚拟寻址引入了一个间接层：CPU 发出的地址是虚拟地址，由内存管理单元（MMU）翻译成物理地址。

特性 物理寻址 虚拟寻址
地址空间 共享，所有程序看到同一个物理地址 隔离，每个进程有独立的虚拟地址空间
内存保护 无硬件保护，程序可以读写任意内存 页级权限位（读/写/执行），访问违规触发段错误
链接与加载 需要预先指定加载地址，位置相关 每个进程看到同样的布局（0x400000 开始），位置无关
内存超分配 总使用不能超过物理内存 进程可以分配比物理内存更大的空间（利用磁盘交换）
共享内存 困难，需要程序协调 不同进程的虚拟页映射到同一个物理页（如共享库）

换句话说，虚拟内存带来了四个核心好处：隔离（进程不会互相影响）、简化链接和加载（所有进程看到相同的地址布局）、共享（通过映射同一物理页实现）和交换（程序可以超出物理内存限制）。

想一想：为什么 C 语言的 NULL 指针解引用一定会崩溃？因为操作系统把虚拟地址 0 所在的页面标记为"未映射"——任何对该页的访问都会触发段错误。这是虚拟内存保护机制的体现。

17.2 页表与多级页表

虚拟地址到物理地址的翻译是通过页表（Page Table）完成的。页表是一个存储在物理内存中的数组，每个页表项（PTE）包含物理页号和一些标志位：

  • 有效位（valid bit）：该虚拟页是否已映射到物理页
  • 权限位（read/write/execute）：该页的访问权限
  • 物理页号（Physical Page Number, PPN）：物理内存的页码
  • 其他标志：dirty bit（是否被写）、accessed bit（是否被访问）等

最简单的情况是单级页表——每个虚拟页对应一个 PTE。在 64 位系统上，以 4KB 页大小计算，虚拟地址空间有 2^48 / 2^12 = 2^36 个页，每个 PTE 8 字节，单级页表需要 512GB——比整台服务器的内存还大，显然不现实。

这就是多级页表存在的理由。x86-64 使用四级页表，虚拟地址被分为以下部分：

一个 48 位的虚拟地址在 x86-64 上被拆解为：

63          48 47      39 38      30 29      21 20      12 11        0
+------------+----------+----------+----------+----------+-----------+
|  符号扩展   |  PML4   |  PDPT    |   PD     |    PT    | 页内偏移  |
|  (16 bits) | (9 bits) | (9 bits) | (9 bits) | (9 bits) | (12 bits) |
+------------+----------+----------+----------+----------+-----------+

翻译过程（四级页表遍历）：
1. CR3 寄存器 → PML4 页表基地址
2. PML4[索引]   → PDPT 页表基地址
3. PDPT[索引]   → PD 页表基地址
4. PD[索引]     → PT 页表基地址
5. PT[索引]     → 物理页号 (PPN)
6. PPN + 偏移量 → 最终物理地址

每个 9-bit 索引范围为 0-511，即每级页表有 512 个条目。关键的节省在于：如果某个 PML4 条目是"无效"的，那么它后面的整个子树都不需要分配。对于一个典型进程，大部分虚拟地址空间是未使用的，多级页表只在需要时分配中间节点——这使得页表占用的物理内存量从数百 GB 降到了数 MB。

核心要点：多级页表本质上是用"时间换空间"——每次地址翻译需要 4-5 次内存访问（每级查一次）。没有 TLB 的话，一次内存访问要慢 4-5 倍。这就是为什么 TLB 如此关键。

17.3 TLB——翻译后备缓冲器

TLB（Translation Lookaside Buffer）是 MMU 内部的一个小型硬件缓存，缓存了最近使用过的虚拟地址到物理地址的翻译结果。它的目标是让地址翻译的额外开销降到几乎为零。

每次 CPU 发出一个虚拟地址时，MMU 的流程大致是：

  1. 检查 TLB——如果命中（TLB hit），直接得到物理地址，无需查页表。
  2. TLB 未命中（TLB miss）——MMU 发起硬件页表遍历（hardware page table walk），从内存中读取 PTE，更新 TLB，再重试。

现代 CPU 的 TLB 大约有 64-256 个条目，分为指令 TLB（iTLB）和数据 TLB（dTLB），通常还有 L2 TLB（更大的二级 TLB，约 512-2048 个条目）。TLB 覆盖率取决于工作集大小和页大小。

TLB 刷新是上下文切换时不得不做的脏活——因为不同进程的虚拟地址到物理地址的映射不同，切换进程时必须使旧进程的 TLB 条目失效。在多核系统上，这需要通过处理器间中断（IPI）触发"TLB shootdown"——所有核心都收到中断，刷新自己的 TLB。频繁的上下文切换会带来大量的 TLB 刷新开销，这也是为什么线程（共享地址空间）比进程（独立地址空间）的切换成本低很多。

大页（Huge Pages）减少 TLB 压力：如果默认页大小是 4KB，一个 TLB 有 64 个条目，最多覆盖 256KB 的地址范围——这对于动辄几百 MB 的数据集来说杯水车薪。但 x86-64 支持 2MB 和 1GB 的大页。使用 2MB 大页时，同样 64 个 TLB 条目可以覆盖 128MB。对于数据库、大数据处理这类内存密集型应用，启用透明大页（Transparent Huge Pages, THP）或显式使用大页，可以显著降低 TLB miss 率，提升性能 10-30%。

17.4 缺页故障与Demand Paging

当程序访问一个 PTE 中有效位为 0 的虚拟页时，MMU 无法完成翻译——它触发一个缺页故障（page fault），将控制权交给内核的缺页处理函数。

缺页处理流程：

  1. CPU 触发缺页异常，保存触发地址（在 %cr2 寄存器中）
  2. 内核检查虚拟地址是否合法：如果是非法地址 → 发送 SIGSEGV
  3. 如果地址合法但页面不在内存中（被换出到磁盘了）：
  4. 内核找到一个空闲的物理页框（如果不够，先换出另一个页面）
  5. 从磁盘（swap 分区或文件）读入数据到该物理页
  6. 更新 PTE，设置有效位
  7. 返回用户态，重新执行触发缺页的那条指令——这次 TLB 命中

这种方式称为Demand Paging（请求分页）——内核只在程序真正访问某页时才加载它，而不是在启动时把所有代码和数据都加载进内存。这意味着程序启动更快，物理内存的利用率也更高。

用 perf 工具跟踪缺页次数：

$ # 跟踪程序的缺页次数
$ perf stat -e page-faults,minor-faults,major-faults ./my_program

   Performance counter stats for './my_program':
            1,234      page-faults
            1,200      minor-faults   # 缺页但页在内存中（如共享库懒加载）
               34      major-faults   # 需要从磁盘读页

$ # 也看缺页的代码位置
$ perf record -e page-faults ./my_program
$ perf report

缺页分两种：minor fault（页面已存在，只是还没关联到当前进程的页表——典型情况是共享库的懒加载）和 major fault（页面确实在磁盘上，需要 I/O 读取）。major fault 是性能杀手——一次磁盘 I/O 是几毫秒，如果程序频繁触发 major fault，性能会急剧下降。这正是 mmap 惰性分配的底层机制。

踩坑故事：我曾接手一个数据分析程序，处理 10GB 数据集需要 6 小时。直觉是算法问题，但 perf stat 显示 major fault 高达 200 万次。原因很简单：程序用 mmap 映射了整个数据集，然后 memset 预分配——但实际上，第一次写入会按 4KB 粒度触发缺页，200 万次 major fault 就是不断从磁盘按页读入数据。加上 madvise(MADV_SEQUENTIAL) 并增大 readahead 窗口后，时间降到了 40 分钟。理解缺页行为，就能读懂为什么你的程序会"卡"。

虚拟内存的地址翻译是计算机系统中最精妙的机制之一。它在硬件和软件的配合下，为每个进程提供了独立、安全、高效的内存抽象。下一章我们将深入学习虚拟内存的另一半——内存分配，看看当程序调用 malloc 时，内核是如何分配物理内存的。
