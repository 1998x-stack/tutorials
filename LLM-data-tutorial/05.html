规则的智慧：从 C4 到 FineWeb

大家好，经过前面几章的处理，我们已经拿到了"一种语言"的"正文文本"。但这距离成为 LLM 的训练语料，还差得远——这些文本中充斥着广告、模板页面、自动生成内容、代码片段、价格列表等不适合训练大模型的东西。

接下来的环节，是整个数据清洗流水线中最"土"但也最有效的手段——启发式规则过滤（Heuristic Filtering）。说是"土"，因为它的原理确实不复杂：就是定义一堆 if-else 条件，不符合的就丢弃。但就是这么"土"的方法，却能以极低的计算成本过滤掉 30-40% 的低质量数据。

启发式规则的进化史，可以看作一条从 C4 → CCNet → RefinedWeb → FineWeb 的演进路线。C4（2020年，Google）定义了第一套成体系的规则集，后续每个项目都在 C4 的基础上做增删改查。2024年的 FineWeb 是目前规则最完善的版本，HuggingFace 还开源了完整的规则实现，成为了行业的新标杆。

为什么规则过滤如此重要： 启发式规则是语料清洗的"第一道防线"。它的计算成本极低（通常是 O(n) 的统计量计算），远远低于后面会讲到的模型过滤。先用简单的规则过滤掉明显的垃圾，可以大幅减少后续复杂过滤步骤需要处理的数据量。在 300TB 的数据规模下，每提前一步过滤掉垃圾，都能节省大量的计算资源。

C4 经典规则集

C4（Colossal Clean Crawled Corpus）是 Google 在 2020 年为 T5 模型准备的数据集。它的规则集定义了现代数据清洗的基本框架：

规则条件过滤掉的比例设计意图
最少字符数文档 < 200 字符~8%过滤掉太短的、信息量不足的页面
最大字符数文档 > 10 万字符~1%过滤掉过长页面（通常是代码转储或自动生成）
必须包含标点符号没有 . ! ? 等结尾~5%过滤掉机器生成的杂乱文本
单词重复率top-3 单词占比 > 20%~3%过滤掉 keyword stuffing 和重复性垃圾
包含占位词含有 "lorem ipsum" 等~0.5%过滤掉模板占位文本
包含 JS/CSS含有 "javascript" 等关键词~6%过滤掉编程代码（非自然语言）
句子必须包含单词至少一句以单词开头~2%过滤掉纯标点和编号列表

这个规则集看似简单，但在2020年是非常前瞻性的工作。它第一次系统性地定义了"什么是一个好的训练文档"。每条规则过滤掉的速度都不大（0.5-8%），但组合起来可以去掉约 25-30% 的低质量文档，而且几乎不会误伤高质量内容。

从工程角度看，C4 规则集最大的优点是 可审计性。每条规则都有一个明确的数学定义，你可以随时检查某条规则过滤掉了什么、有没有误杀。相比之下，后面章节会讲到的基于模型的质量过滤，就像一个黑盒——你很难说清楚模型为什么觉得一篇文档质量低。

FineWeb 规则集扩展

2024年，HuggingFace 的 FineWeb 团队在 C4 规则的基础上做了大量的扩展和精化。这些改进反映了过去四年间行业对"什么是高质量数据"的认知升级：

{
  // FineWeb filter 配置（JSON 示例）
  "min_chars": 200,                    // 最少字符数
  "max_chars": 100000,                 // 最多字符数
  "min_punctuation_ratio": 0.1,        // 标点符号比例 > 10%
  "max_long_line_ratio": 0.3,          // 最长行比例 < 30%（长行通常是代码或表格）
  "max_short_line_ratio": 0.5,         // 最短行比例 < 50%
  "max_punctuation_ratio": 0.3,        // 标点比例 < 30%（太多标点 = 机器生成）
  "max_digit_ratio": 0.5,             // 数字比例 < 50%（数字太多 = 价格列表）
  "max_bullet_ratio": 0.9,            // bullet point 比例 < 90%
  "min_stop_word_density": 0.04,       // 停用词密度 > 4%（无停用词 = 非自然语言）
  "min_word_count": 50,               // 最少单词数
  "max_word_duplicate_ratio": 0.3,     // 单词重复率上限
  "max_top_ngram_ratio": 0.25,        // 高频 n-gram 占比上限
  "flag_placeholder_text": true,       // 标记占位符文本
  "flag_js_heavy": true               // 标记含大量 JS 关键词的文档
}

对比 C4，FineWeb 的创新点在于引入了更多的基于比例的规则。不是简单地看"是否包含标点"，而是看"标点占比是否在合理范围内"。这种细粒度的量化指标能更好地捕捉文本的统计特征——一篇正常文章的标点符号比例通常在 10-25% 之间，如果低于 5% 大概率是机器生成的噪音。

直接可用的工程资源： FineWeb 在 GitHub 上开源了完整的 filter 配置和规则实现（huggingface/datatrove），你可以直接拿过来用。它支持多进程并行处理，底层使用了 Rust 加速关键统计量的计算。这是目前最成熟、文档最完善的规则过滤开源实现，强烈建议直接复用而不是自己重写。

规则设计的工程实践

设计一套好的规则集，不仅仅是"拍脑袋想条件"。这里分享几条经过多个项目验证的工程实践：

第一，规则评估要在采样数据上做。 不要直接在大数据集上跑完整规则集。先采样几万个文档，逐一观察每条规则过滤掉了什么。一个很好的实践是：针对每条规则，打印出被它过滤掉的 20 个"边界案例"——也就是刚好达到或刚好没达到过滤阈值的文档。这能帮你判断阈值是否合理。

第二，规则的优先级有讲究。 先做范围最大的过滤（URL 去重、语言过滤等），再做粒度最细的过滤（重复行、高频率 n-gram）。这样做的原因是：范围大的过滤会改变数据分布，可能影响粒度细的过滤的结果。比如先做了语言过滤，再去判断标点符号比例——英语的标点符号比例自然比中文高，阈值设置就要调整。

第三，规则迭代的策略：先松后紧。 不要一上来就设计一套完美的规则集。先用宽松的规则跑一轮，观察保留数据的分布，再看哪些维度还有改进空间。RefinedWeb 团队的做法是：先跑 C4 基准规则集，然后分析被过滤掉的文档的类型，针对性地增加新规则。他们在第一轮过滤后发现有很多"重复性广告文案"被保留，所以增加了"最长行比例"规则来处理这类问题。

过度过滤的风险： 规则过滤最大的敌人不是"没过滤干净"，而是"误杀"。比如，"lorem ipsum"这条规则是用来过滤模板占位文本的——但医学文献中"lorem ipsum"实际上是拉丁语的一个真实短语（意为"痛苦本身"），有真实的医学文献里出现了这个短语。再比如，"最长行比例"规则会误杀包含大量代码示例的技术博客。每增加一条规则，都应该在采样数据上评估它的误伤率。一个经验法则是：每条规则的误杀率不应超过 0.5%。

规则过滤 vs 模型过滤

讲到这里，你可能会想：既然规则过滤这么"粗糙"，为什么不用更高级的模型过滤替代它？这个问题的答案其实很简单：规则过滤和模型过滤不是替代关系，而是串联关系。

规则过滤的优势：

  • 速度快： 每条规则只需要 O(n) 的统计量计算，不需要加载模型、不需要 GPU
  • 可解释： 你知道每条规则为什么把某条数据过滤掉，可以审计、可以调试
  • 零成本运维： 规则不会过时，不需要重新训练，一次写好永久有效

模型过滤的优势：

  • 高精度： 能捕捉到规则无法定义的"软性质量指标"，比如"这篇文章是否有教育价值"
  • 灵活性： 同一个模型可以在不同领域微调来适应不同的质量定义
  • 可以排序： 模型可以输出质量评分而不仅是二元判断，支持按质量做筛选

实际流水线中，两者的配合方式是：先用规则过滤掉 30-50% 明显是垃圾的数据，剩下的数据再送入模型做细粒度质量分类。这样既利用了规则过滤的效率，又借用了模型过滤的精度。如果反过来先做模型过滤——模型要处理海量垃圾数据，不仅浪费 GPU 算力，还会因为噪声太多而降低分类准确率。

OK，关于规则过滤就讲到这里。在 C4 和 FineWeb 的规则集之间，有一个特殊类型的规则值得单独拿出来讲——那就是 URL 和域名相关的过滤。这不是普通的"内容特征"，而是基于数据源的策略判断。下一章我们重点聊聊 URL 级别的清洗策略。
