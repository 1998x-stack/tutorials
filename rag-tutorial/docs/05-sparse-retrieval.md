# 稀疏检索与 BM25 算法

## 1. 稀疏检索基础

稀疏检索基于**词项匹配**，在倒排索引中找到包含查询词的文档。与稠密检索（语义匹配）互补。

### 核心组件
- **倒排索引**：词 → 文档列表的映射
- **词项权重**：衡量词在文档中的重要程度
- **打分函数**：综合计算文档与查询的相关性

## 2. 倒排索引

### 2.1 构建过程
```
文档1: "苹果手机很好用"
文档2: "华为手机也不错"
文档3: "苹果电脑性价比高"

倒排索引:
苹果 → [doc1, doc3]
手机 → [doc1, doc2]
很好用 → [doc1]
华为 → [doc2]
也不错 → [doc2]
电脑 → [doc3]
性价比高 → [doc3]
```

### 2.2 搜索过程
查询"苹果手机" → 分别查找"苹果"和"手机"的倒排列表 → 取交集 → [doc1]

## 3. BM25 算法详解

BM25 是目前最广泛使用的稀疏检索算法，Elasticsearch 的默认打分函数。

### 3.1 BM25 公式

```
BM25(D, Q) = Σ IDF(qi) × TF(qi, D) × (k1+1) / (TF(qi, D) + k1×(1-b+b×|D|/avgdl))

IDF(qi) = log((N - n(qi) + 0.5) / (n(qi) + 0.5) + 1)

其中:
- TF(qi, D)：词 qi 在文档 D 中的词频
- IDF(qi)：逆文档频率，衡量词的稀有程度
- |D|：文档长度
- avgdl：平均文档长度
- k1：词频饱和度参数（通常 1.2-2.0）
- b：文档长度归一化参数（通常 0.75）
```

### 3.2 关键参数调优

| 参数 | 作用 | 典型值 | 影响 |
|------|------|--------|------|
| k1 | 控制词频饱和度 | 1.2-2.0 | k1越大，词频影响越大 |
| b | 控制文档长度归一化 | 0.5-0.75 | b=0 关闭长度归一化，b=1 完全归一化 |

### 3.3 BM25 的优势
- 精确关键词匹配，对专有名词、代码、缩写效果极好
- 无模型训练，开箱即用
- 计算高效，毫秒级
- 可解释性强

### 3.4 BM25 的局限
- 无法理解语义相似（"汽车" vs "轿车"）
- 无法处理跨语言匹配
- 对自然语言描述的查询效果差
- 需要分词器支持

## 4. SPLADE：学习型稀疏检索

SPLADE 将稀疏表示与神经网络结合：
- 使用 BERT 等模型预测每个词的权重
- 输出高维稀疏向量（大部分为 0）
- 既有语义理解能力（神经网络），又保持稀疏性（倒排索引可以加速）

## 5. 中文分词注意事项

中文没有天然的词边界，分词质量直接影响检索效果：

```python
# 分词对比
import jieba

text = "检索增强生成技术详解"
# 通用分词
jieba.cut(text)  # ['检索', '增强', '生成', '技术', '详解']

# 自定义词典
jieba.add_word("检索增强生成")
jieba.cut(text)  # ['检索增强生成', '技术', '详解']
```

关键建议：
- 为领域术语建立自定义词典
- 使用搜索模式分词（`jieba.cut_for_search`）
- 考虑使用 BERT 类分词器（如 BERT Tokenizer）

## 6. 稀疏 vs 稠密检索

| 维度 | 稀疏检索（BM25） | 稠密检索（Embedding） |
|------|-----------------|---------------------|
| 匹配方式 | 精确词匹配 | 语义相似度 |
| 专有名词 | 优秀 | 一般 |
| 同义词 | 差 | 优秀 |
| 跨语言 | 不支持 | 支持 |
| 索引速度 | 快 | 慢（需推理） |
| 查询速度 | 快 | 中等（ANN） |
| 冷启动 | 零成本 | 需要模型 |
| 存储 | 小 | 大 |
