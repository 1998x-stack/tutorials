开篇：求解器——把数学变为现实的魔法棒

大家好，我是你们的讲师。在之前的22章里，我们学习了分支定界、分支切割、列生成、多面体理论等等——这些是组合优化的"内功心法"。但是，有一个问题一直萦绕在我心头：学完这些理论之后，怎么真正解决现实中的优化问题？

2022年我参与过一个真实的仓储物流优化项目。客户是一家区域性的冷链物流公司，每天有大约3000单的配送需要规划：15辆冷藏车，每辆车有不同容积和温区，每个订单有时间窗，有些货物不能混装（生鲜和冷冻不能放一起），还要考虑司机的工时限制。我们花了两周时间做数学建模，写了一个精巧的分支切割算法——结果在测试数据上跑了8个小时都没出最优解。客户说："我们可等不了8小时，每天中午前必须出方案。"

痛定思痛，我们换了个思路：用商业MIP求解器。一个同事花了半天时间把模型翻译成Gurobi代码，然后——你猜怎么着？Gurobi在17分钟内给出了最优解，gap小于0.5%。那一刻我真正理解了：求解器不是辅助工具，它是组合优化在工业化落地中的核心引擎。手写算法的时代，在绝大多数工业场景下，已经过去了。

所以这一章我们放下纸笔，打开电脑——从求解器实战开始，真正让我们的模型跑起来。

Gurobi/PuLP建模实战：设施选址问题

环境准备好了，现在就实战。我们选一个经典的设施选址问题（Facility Location Problem, FLP）作为例子。这个问题描述如下：

  • 有m个候选仓库位置，每个位置有建设成本f_i；
  • 有n个客户，每个客户的需求量为d_j；
  • 从仓库i到客户j的单位运输成本为c_{ij}；
  • 每个仓库有容量上限cap_i；
  • 决策：选择哪些仓库建设，以及每个仓库分别给哪些客户供货，使得总成本（建设+运输）最小。

这是一个经典的混合整数规划（MIP）问题——连续变量（运输量）和二元变量（是否建仓）交织在一起。

数学模型

先写出数学公式，再翻译成代码：

参数:
  I: 仓库集合, J: 客户集合
  f_i: 仓库i的建设成本
  c_{ij}: 从仓库i到客户j的单位运输成本
  d_j: 客户j的需求量
  cap_i: 仓库i的容量

变量:
  y_i ∈ {0,1}: 是否建设仓库i
  x_{ij} ≥ 0: 从仓库i运往客户j的货物量

目标:
  min  Σ_i f_i·y_i + Σ_i Σ_j c_{ij}·x_{ij}

约束:
  Σ_i x_{ij} = d_j            ∀j    (客户需求满足)
  Σ_j x_{ij} ≤ cap_i·y_i      ∀i    (容量约束 + 未建则不可发货)
  x_{ij} ≥ 0, y_i ∈ {0,1}

Gurobi版本

import gurobipy as gp
from gurobipy import GRB

# 数据
I = range(3)  # 3个候选仓库
J = range(10) # 10个客户
f = [100, 150, 120]
cap = [500, 600, 400]
d = [50, 60, 45, 80, 70, 55, 65, 40, 90, 75]
c = [[8, 12, 10, 9, 14, 11, 13, 15, 7, 10],
     [11, 9, 13, 8, 10, 12, 9, 14, 11, 8],
     [10, 11, 8, 13, 9, 10, 12, 8, 14, 9]]

# 建模
m = gp.Model("FacilityLocation")

# 变量
y = m.addVars(I, vtype=GRB.BINARY, name="open")        # 是否建仓
x = m.addVars(I, J, vtype=GRB.CONTINUOUS, name="ship") # 运输量

# 目标: 建设成本 + 运输成本
m.setObjective(
    gp.quicksum(f[i] * y[i] for i in I) +
    gp.quicksum(c[i][j] * x[i, j] for i in I for j in J),
    GRB.MINIMIZE
)

# 约束: 需求满足
demand_constrs = m.addConstrs(
    (gp.quicksum(x[i, j] for i in I) == d[j] for j in J),
    name="demand"
)

# 约束: 容量 & 建仓关联
capacity_constrs = m.addConstrs(
    (gp.quicksum(x[i, j] for j in J) <= cap[i] * y[i] for i in I),
    name="capacity"
)

# 求解
m.optimize()

# 输出结果
if m.Status == GRB.OPTIMAL:
    print(f"\n总成本: {m.ObjVal:.2f}")
    for i in I:
        if y[i].X > 0.5:
            print(f"仓库{i+1} 建设, 发货量: {sum(x[i,j].X for j in J):.0f}")
else:
    print("未找到最优解")

PuLP版本

同样的模型用PuLP实现。注意PuLP的语法更加"Pythonic"，但也相对啰嗦一些。

from pulp import LpProblem, LpMinimize, LpVariable, lpSum, LpBinary, LpContinuous, PULP_CBC_CMD

# 数据同上 (此处省略重复定义)
I, J, f, cap, d, c = range(3), range(10), [100, 150, 120], [500, 600, 400], \
                     [50, 60, 45, 80, 70, 55, 65, 40, 90, 75], \
                     [[8, 12, 10, 9, 14, 11, 13, 15, 7, 10],
                      [11, 9, 13, 8, 10, 12, 9, 14, 11, 8],
                      [10, 11, 8, 13, 9, 10, 12, 8, 14, 9]]

# 建模
prob = LpProblem("FacilityLocation", LpMinimize)

y = {i: LpVariable(f"open_{i}", cat=LpBinary) for i in I}
x = {(i, j): LpVariable(f"ship_{i}_{j}", lowBound=0, cat=LpContinuous)
     for i in I for j in J}

# 目标
prob += lpSum(f[i] * y[i] for i in I) + \
        lpSum(c[i][j] * x[i, j] for i in I for j in J)

# 需求约束
for j in J:
    prob += lpSum(x[i, j] for i in I) == d[j], f"demand_{j}"

# 容量约束
for i in I:
    prob += lpSum(x[i, j] for j in J) <= cap[i] * y[i], f"capacity_{i}"

# 求解 (使用CBC求解器)
prob.solve(PULP_CBC_CMD(msg=True))

print(f"总成本: {prob.objective.value():.2f}")
for i in I:
    if y[i].value() > 0.5:
        print(f"仓库{i+1} 建设")
观察：Gurobi和PuLP在语义上非常接近，但有几个细微差别值得注意：(1) Gurobi用addVars批量创建变量，PuLP用字典推导式手动创建；(2) Gurobi用gp.quicksum，PuLP用lpSum；(3) Gurobi的求和效率比Python原生的sum()高出一个数量级——当变量数超过百万时，sum()会产生巨大的中间对象，而quicksum/lpSum通过惰性求值避免了这个问题。在规模超过10万变量时，一定不要用Python的sum()。

求解器参数调优

安装好环境、写完模型——这只是第一步。80%的求解器实战在于参数调优。同一个模型，用不同的参数设置，求解时间可能差100倍。下面是我在实践中经常调整的核心参数。

参数 Gurobi名称 取值与含义 典型场景
搜索侧重 MIPFocus 0=平衡, 1=快速找可行, 2=最优性证明, 3=边界提升 需要快速可行解时设1；已找到好解想证最优时设2
割平面强度 Cuts 0=全关, 1=适度(default), 2=激进, 3=非常激进 难问题设为2或3；速度优先设0
启发式搜索 Heuristics 0.0~1.0，默认0.05，越大花越多时间找可行解 难度大的问题可提高至0.2-0.3
线程数 Threads 正整数，默认使用全部核心 共享服务器上设少些，让给他人
时间上限 TimeLimit 秒数，默认无限 生产环境一定要设！避免求解器无限运行
MIP Gap MIPGap 默认1e-4，可适当放松至0.01(1%) 工业应用1%-5%gap通常已足够
Debug小技巧：当你面对一个求解得很慢的模型时，我建议的调试顺序是：(1) 先检查模型是否正确——用m.write("model.lp")把模型导出为LP格式，用文本编辑器打开看看约束是否写对了；(2) 然后用默认参数求解10分钟，看log了解问题难点（是找不到可行解，还是bound收敛慢）；(3) 根据问题特点调参——找不到可行解，增加Heuristics和MIPFocus=1；bound收敛慢，加强Cuts。按这个顺序，80%的问题能在20分钟内解决。

Callback编程

Callback是求解器实战中最具"黑客精神"的部分。它让你在分支定界求解过程中注入自定义逻辑——这是实现用户割（User Cuts）、惰性约束（Lazy Constraints）和自定义分支的唯一途径。

Callback的类型大致可以分为三类：

  • Lazy Constraint Callback：在节点LP求解后，检查当前解是否违反了某些约束。若不满足，动态添加约束到模型中。适用于约束数量巨大（如TSP的subtour elimination）的情形。
  • Heuristic Callback：在分支定界的某个节点处，基于当前LP解构造出整数可行解并注入给求解器。通常能大幅加速找到好可行解。
  • Branch Callback：自定义分支规则，替代求解器默认的分支策略。

下面以TSP的子回环消除（Subtour Elimination）为例，展示LazyCallback的经典用法。

import gurobipy as gp
from gurobipy import GRB
import itertools
import math
import random

# 生成随机TSP实例
n = 20
cities = [(random.uniform(0, 100), random.uniform(0, 100)) for _ in range(n)]
dist = {(i,j): math.sqrt((cities[i][0]-cities[j][0])**2 +
                          (cities[i][1]-cities[j][1])**2)
        for i in range(n) for j in range(n) if i != j}

# 建模——先不加subtour约束
m = gp.Model("TSP_Lazy")
x = m.addVars(dist.keys(), vtype=GRB.BINARY, name="x")

# 出入度约束
m.addConstrs(gp.quicksum(x[i,j] for j in range(n) if j!=i) == 2 for i in range(n))

# 目标
m.setObjective(gp.quicksum(dist[i,j] * x[i,j] for i,j in dist), GRB.MINIMIZE)

# === 核心: Lazy Callback ===
def subtour_elimination(model, where):
    if where == GRB.Callback.MIPSOL:  # 找到整数可行解时触发
        # 提取当前解
        vals = model.cbGetSolution(x)
        selected = [(i,j) for i,j in dist if vals[i,j] > 0.5]

        # 构建图并找连通分量
        adj = {i: set() for i in range(n)}
        for i,j in selected:
            adj[i].add(j)
            adj[j].add(i)

        visited = set()
        subtours = []
        for start in range(n):
            if start not in visited:
                comp = []
                stack = [start]
                while stack:
                    v = stack.pop()
                    if v not in visited:
                        visited.add(v)
                        comp.append(v)
                        stack.extend(adj[v] - visited)
                if len(comp) < n:
                    subtours.append(comp)

        # 添加割约束
        for comp in subtours:
            edges_in = [(i,j) for i,j in dist
                       if i in comp and j in comp and i != j]
            m.cbLazy(gp.quicksum(x[i,j] for i,j in edges_in) <= len(comp) - 1)

m.Params.LazyConstraints = 1  # 重要: 开启惰性约束模式
m.optimize(subtour_elimination)

print(f"TSP最优行程长度: {m.ObjVal:.2f}")
一个让人抓狂的坑：我第一次写LazyConstraint的时候忘了设置m.Params.LazyConstraints = 1，结果Gurobi直接报错"Lazy constraints cannot be added ..."。这个参数必须在optimize()之前设置，否则solve阶段不允许动态添加约束。更隐蔽的问题出现在Debug的时候——如果你在回调函数中抛出了一个未被捕获的异常，Gurobi不会把异常传递到外层Python代码，而是直接终止求解并返回一个模棱两可的错误码。所以，在回调函数中用try/except包裹所有逻辑，并在except中显式打印错误信息。

关键洞察：很多新手遇到求解慢的第一反应是"加更多参数"或者"换求解器"。但实际上，90%的性能问题在于模型质量——你写的MIP formulation不够"紧"（tight），导致LP松弛的界很弱。在怀疑求解器之前，先用LP relaxation跑一次，看根节点的gap。如果根节点gap超过50%，调参数没用，必须回归到第14章的多面体理论和第16章的建模艺术，添加有效不等式来收紧模型。
