第27章 CAN总线软件架构：分层设计思想、驱动层、协议层、应用层分离

做CAN开发这么多年，我见过太多人一上来就写代码。

把报文收发、ID过滤、数据处理全塞在一个文件里。结果呢？项目后期改一个需求，牵一发而动全身。说白了，这就是没有分层设计带来的恶果。

今天咱们聊聊CAN总线的软件架构。我个人习惯把CAN软件分成三层：驱动层、协议层、应用层。这三层各司其职，互不干扰。你想想看，这样做的好处是什么？

维护方便，移植简单，团队协作也清晰。嗯，咱们一层一层拆开看。

27.1 为什么要分层？

先问个问题：你写的CAN代码，换一个MCU平台，要改多少行？

如果答案是「几乎全改」，那你的架构就有问题了。我在项目中遇到过这种情况——客户突然要求从STM32换到NXP平台。幸好当时做了分层设计，驱动层重写，协议层和应用层几乎没动。两周搞定，客户都惊讶。

分层设计的核心思想就一句话：各层只关心自己的事。

  • 驱动层：管硬件，收发报文
  • 协议层：管格式，解析打包
  • 应用层：管业务，处理数据

每一层都通过接口跟上下层通信。层与层之间不直接访问内部变量。这就是所谓的「高内聚、低耦合」。

核心原则：

  • 驱动层不知道报文里是什么数据
  • 协议层不知道硬件怎么收发
  • 应用层不知道报文怎么打包的

27.2 驱动层：硬件的「翻译官」

驱动层是离硬件最近的一层。它直接操作CAN控制器寄存器，负责最底层的收发工作。

我习惯把驱动层做得尽可能薄。只做三件事：

  1. 初始化CAN控制器：设置波特率、工作模式、过滤器
  2. 发送报文：把数据写入发送邮箱
  3. 接收报文：从接收FIFO读取数据

来看一个典型的驱动层接口：

/* 驱动层接口 - 硬件无关 */
typedef struct {
    uint32_t id;        /* 标准帧或扩展帧ID */
    uint8_t  data[8];   /* 数据场，最多8字节 */
    uint8_t  dlc;       /* 数据长度 */
    uint8_t  is_ext;    /* 1:扩展帧, 0:标准帧 */
    uint8_t  is_rtr;    /* 1:远程帧, 0:数据帧 */
} CanFrame_t;

/* 初始化CAN控制器 */
int CAN_Driver_Init(uint32_t baudrate);

/* 发送一帧报文 */
int CAN_Driver_Send(CanFrame_t *frame);

/* 接收一帧报文（非阻塞） */
int CAN_Driver_Receive(CanFrame_t *frame);

注意看，驱动层只处理CanFrame_t这种原始报文结构。它不关心ID是0x123还是0x456，也不关心data[0]代表什么物理量。

我的经验：驱动层一定要做超时处理。我曾经遇到过CAN总线被拉死的情况，驱动层没有超时机制，整个系统卡死在发送函数里。从那以后，我所有发送接口都加了超时参数。

27.3 协议层：数据的「翻译官」

协议层在驱动层之上。它把原始报文翻译成应用层能理解的数据。

说白了，协议层做两件事：

  • 打包：把应用层的数据（比如车速、转速）按照协议格式塞进报文
  • 解析：从收到的报文里提取出各个信号值

不同的项目用不同的协议。J1939、CANopen、或者自己定义的私有协议。协议层就是把这些协议逻辑封装起来。

举个例子，假设我们有一个车速信号，定义在ID=0x100的报文里，起始位bit 8，长度16位，分辨率0.01 km/h：

/* 协议层 - 解析车速信号 */
float Protocol_GetVehicleSpeed(CanFrame_t *frame)
{
    if (frame->id != 0x100) {
        return -1.0f;  /* 不是车速报文 */
    }
    
    /* 从bit 8开始取16位数据 */
    uint16_t raw = (frame->data[1] << 8) | frame->data[2];
    
    /* 应用分辨率 */
    float speed = raw * 0.01f;
    
    return speed;
}

你看，协议层只处理数据格式。它不关心这个车速值最终用来做什么——那是应用层的事。

注意：协议层不要直接调用驱动层函数。我曾经见过有人直接在协议层里写HAL_CAN_Transmit()，换平台时改到崩溃。正确的做法是：协议层通过回调函数或接口指针调用驱动层。

27.4 应用层：业务的「决策者」

应用层是最高层。它从协议层拿到解析好的数据，然后做业务逻辑处理。

比如：

  • 车速超过120 km/h时报警
  • 发动机水温过高时降功率
  • 收到远程诊断请求时回复数据

应用层不关心CAN报文怎么收发，也不关心数据怎么解析。它只关心：我拿到了什么数据？我要做什么决策？

/* 应用层 - 超速报警逻辑 */
void App_CheckSpeedLimit(void)
{
    float speed = Protocol_GetVehicleSpeed(&g_rxFrame);
    
    if (speed > 120.0f) {
        App_TriggerAlarm(ALARM_OVERSPEED);
    }
}

简单吧？应用层代码写起来最轻松，因为它不需要处理底层细节。

27.5 三层之间的调用关系

我习惯用一张图来描述这三层的关系。不过这里咱们用文字描述：

方向 调用链 说明
发送 应用层 → 协议层 → 驱动层 应用层给数据，协议层打包，驱动层发送
接收 驱动层 → 协议层 → 应用层 驱动层收报文，协议层解析，应用层处理

接收方向要注意：驱动层收到报文后，通常通过回调函数通知协议层。协议层解析完，再通过另一个回调通知应用层。这样每一层都是被动触发的，不会出现轮询浪费CPU。

避坑指南：我曾经在一个项目里，把协议层和应用层的回调函数写成了同步调用。结果协议层解析耗时太长，导致驱动层丢帧。后来改成异步队列，驱动层收完报文就扔进队列，协议层从队列里取数据解析。问题解决了。

27.6 实际项目中的分层建议

说了这么多理论，来点实际的。我总结了几条分层设计的原则：

  1. 驱动层不要包含任何业务逻辑。哪怕是一个简单的if判断，也要慎重。
  2. 协议层不要直接操作硬件。所有硬件操作都通过驱动层接口。
  3. 应用层不要直接调用驱动层。必须经过协议层。
  4. 每一层都要有独立的头文件。比如can_driver.h、can_protocol.h、can_app.h。
  5. 接口参数尽量简单。能用基本类型就别用结构体，减少耦合。

嗯，最后说一句。分层设计不是银弹。小项目、原型验证阶段，你可以适当简化。但一旦进入量产阶段，分层带来的维护优势是巨大的。

我见过太多「先跑起来再说」的项目，最后代码改到崩溃。你想想看，是现在多花两天做分层设计划算，还是以后花两个月重构划算？

答案不言而喻。
