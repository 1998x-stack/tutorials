第十四章：重定向与URL重写

各位同学好。这一章我们来聊HTTP中的重定向（Redirection）和URL重写（URL Rewriting）。这两个概念经常被混为一谈，但实际上它们是不同层面的机制：重定向是服务器告诉浏览器"去另一个地址"，由浏览器再次发起请求；而URL重写是在服务器内部改写请求路径，浏览器对此毫无感知。

先从一个URL重写的直观例子说起：当你访问某个博客文章时，URL可能是 /article/123，但服务器内部把它映射到了 /article.php?id=123。这个转换发生在服务器内部，浏览器地址栏里的URL没有变化。这就是URL重写。

再来看重定向的例子：当你访问 http://example.com 时，服务器返回 301 Moved Permanently 把浏览器指向 https://example.com，浏览器收到后重新发起HTTPS请求，地址栏也随之变成HTTPS。这就是重定向。

14.1 重定向的类型

HTTP定义了一系列3xx状态码用于重定向。不同状态码的含义差异很微妙，选错可能导致安全隐患。

状态码名称是否永久是否改变请求方法典型场景
301Moved Permanently永久可能（通常GET保持GET，POST可能变GET）域名迁移、HTTP→HTTPS
302Found临时可能（同301）临时页面、A/B测试
303See Other临时总是变为GET表单提交后重定向到结果页
304Not Modified-不变缓存验证（在缓存章节已经讲过）
307Temporary Redirect临时不改变（保证方法不变）临时维护、POST请求的临时跳转
308Permanent Redirect永久不改变（保证方法不变）POST/PUT请求的永久迁移

这里最容易被忽视的是301和302在请求方法上的行为。早期的浏览器在收到301/302时，不管原始请求是POST还是PUT，都会用GET去请求Location。这就产生了一个严重的问题——如果你用POST提交表单，服务器返回302让浏览器跳转到成功页，这个行为没问题。但如果你有个API用POST创建资源，服务器返回301表示资源已迁移，浏览器可能会用GET去请求新地址，而新地址的API如果只接受POST，请求就失败了。

为了解决这个问题，HTTP/1.1引入了307和308，明确保证不会改变请求方法。其中，307是临时重定向（对应302），308是永久重定向（对应301）。

// 301永久重定向（HTTP→HTTPS）
HTTP/1.1 301 Moved Permanently
Location: https://example.com/new-page

// 307临时重定向（POST保持POST）
HTTP/1.1 307 Temporary Redirect
Location: https://backup.example.com/api/data

重要选择指南：如果是GET请求的重定向，用301/302就可以，因为浏览器行为一致。如果是POST/PUT等非GET请求，要用307/308来保证请求方法不变。301/302对非GET请求的行为因浏览器而异，不可靠。我习惯记一个口诀："GET用301/302，POST用307/308"。

14.2 客户端重定向

除了服务器端通过3xx状态码重定向，客户端也可以自己完成重定向。常用的方式有两种：meta refresh 和 JavaScript redirect。

Meta Refresh：

<!-- 立即跳转 -->
<meta http-equiv="refresh" content="0; url=https://example.com/new-page">

<!-- 5秒后跳转 -->
<meta http-equiv="refresh" content="5; url=https://example.com/new-page">

这种方式在搜索引擎优化（SEO）中常见，但用户体验不好——等待几秒再跳转会让人困惑。content中的数字是秒数，设置为0就是立即跳转。

JavaScript Redirect：

// 推荐方式
window.location.replace('https://example.com/new-page');

// 另一种方式（会保留历史记录）
window.location.href = 'https://example.com/new-page';

window.location.replace() 会替换当前历史记录，用户按"返回"按钮不会回到这个页面；而直接赋值 window.location.href 会在历史记录中留下记录。

服务端 vs 客户端重定向如何选择？

  • 服务端重定向（3xx）：HTTP层面完成，最标准。搜索引擎会正确识别。用于域名变更、URL迁移、HTTP→HTTPS。
  • 客户端重定向：需要浏览器执行HTML或JavaScript。用于SPA应用中的路由跳转、登录后跳回、下载完成后跳转、或者定时跳转等场景。

还有一个重要差异：搜索引擎对301/302/307/308的处理是标准化的，会正确传递权重（PageRank）。但meta refresh和JavaScript重定向对搜索引擎不够友好，可能影响SEO。所以如果你的网站要迁移域名，一定要用服务器端301，而不是在旧页面放一条meta refresh。

实际建议：绝大多数场景下，用服务端重定向就够了。客户端重定向只用于两种情况：一是SPA内部的路由跳转（这是单页应用的自然设计）；二是需要用户交互后跳转（比如"下载完成，3秒后返回首页"）。别把meta refresh当作301的替代品。

14.3 URL重写

URL重写（URL Rewriting）完全不同——它在服务器内部修改请求路径，浏览器地址栏不发生变化。最典型的场景：把"漂亮的RESTful URL"映射到"实际的后端处理脚本"。

Nginx rewrite 规则：

# 将 /article/123 重写为 /article.php?id=123
location /article/ {
    rewrite ^/article/(\d+)$ /article.php?id=$1 last;
}

# HTTP 强制跳转 HTTPS
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}

Nginx rewrite的语法：rewrite 正则 替换目标 [flag]。常用的flag：

  • last —— 停止当前location处理，重新匹配新的location
  • break —— 停止当前location处理，不再匹配
  • redirect —— 临时重定向（302），浏览器地址栏变化
  • permanent —— 永久重定向（301），浏览器地址栏变化

Apache mod_rewrite 规则：

# 开启重写引擎
RewriteEngine On

# 将 /article/123 重写为 /article.php?id=123
RewriteRule ^article/(\d+)$ /article.php?id=$1 [L,QSA]

# 强制HTTPS
RewriteCond %{HTTPS} off
RewriteRule ^(.*)$ https://%{HTTP_HOST}/$1 [R=301,L]

Apache的 RewriteCond 比Nginx的正则匹配更灵活——它可以在RewriteRule之前加任意多个条件（Cond），实现复杂的路由逻辑。

SPA的index.html回退模式：这是URL重写最现代的应用场景。SPA（单页应用）只有一个HTML文件（通常是 index.html），但用户可能直接访问 /user/profile 或 /settings 等路由。服务器需要把所有非文件资源请求都指向 index.html：

# Nginx SPA回退
location / {
    try_files $uri $uri/ /index.html;
}

# Apache SPA回退
RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule ^ /index.html [L]

总结区别：URL重写是"偷梁换柱"——用户请求A，服务器悄悄处理B，用户不知道。重定向是"如实告知"——服务器说"我不在这，你去找那个地址"。两者用途不同但可以配合使用。它们还有个关键区别：URL重写不影响SEO，搜索引擎看到的URL就是用户访问的URL；重定向通常意味着搜索引擎最终会使用目标URL。

14.4 重定向循环

重定向循环（Redirect Loop）是每个Web开发者迟早都会遇到的问题。场景：页面A重定向到B，B重定向回A——浏览器陷入无限循环，最终在几十次跳转后显示"ERR_TOO_MANY_REDIRECTS"。

常见原因：

  • HTTP→HTTPS强制跳转，但CDN或负载均衡器在HTTPS终止后又用HTTP转发到后端，后端再强制跳回HTTPS
  • Nginx配置中加了 proxy_redirect 与后端重定向冲突
  • Cookie/Session认证判断出错：未登录 → 跳登录页 → 登录完成 → 跳回原页 → 又检测到未登录
  • 301永久重定向被浏览器缓存：这是最坑的——第一次访问时服务器返回了301，浏览器永久记住了这个重定向。后面即使服务器改回了正确的配置，浏览器仍然使用缓存的301跳转到旧地址，而旧地址又跳转到新地址……

调试方法：

  • 使用 curl -L 跟踪重定向链，可以清楚看到每一步的状态码和Location
  • 打开DevTools的Network面板，查看请求/响应序列
  • 如果怀疑是浏览器缓存了301，用无痕模式测试（无痕模式不缓存301）
  • 检查代理/CDN配置中是否有额外的重定向规则
# 用 curl 跟踪重定向，最多跟20次
curl -L --max-redirs 20 -v http://example.com 2>&1 | grep -E "^(> |< |Location)"

# 示例输出——看到这种循环就知道了
> GET / HTTP/1.1
< HTTP/1.1 301 Moved Permanently
< Location: http://www.example.com/
> GET / HTTP/1.1
< HTTP/1.1 301 Moved Permanently
< Location: http://example.com/
# ... 重复十几次后 curl 报错

踩坑故事：有次帮一个电商网站排查线上故障——用户访问首页报"重定向次数过多"。调查后发现，问题出在多个团队各司其职但缺乏沟通。运维在Nginx层加了一条将所有HTTP请求重定向到HTTPS的规则——return 301 https://$host$request_uri。开发团队在PHP代码里也检查了是否HTTPS，如果不是就返回301跳转到HTTPS。CDN那边也配置了HTTP到HTTPS的跳转。三层各自为政，形成了一个重定向链条：用户的HTTP请求到CDN → CDN重定向到HTTPS → 到Nginx又被重定向到HTTPS（但CDN内部用的还是HTTP）→ 到PHP又检测到不是HTTPS……实际上很多次重定向是在CDN和Nginx之间来回通过HTTP互跳的。最终我们把CDN到后端全部改成走HTTPS并在Nginx层彻底关闭了HTTPS强制逻辑，只保留CDN入口处的重定向，问题解决。这次教训让我明白了：重定向逻辑只能在离用户最近的一层实施一次，不应在多层重复配置。

如何防止重定向循环？

  • 重定向逻辑只在一个地方实施（通常是网关/CDN层或Web服务器层，不要两处都做）
  • 在应用代码中维护一个"已重定向次数"计数器，超过阈值强行终止
  • 使用 curl -L 在发布前检查所有重要URL的重定向链是否不超过1-2跳
  • 部署301变更时要特别小心——浏览器会永久缓存301，一旦配错后修复，用户的浏览器可能需要几周甚至更长时间才会更新
