开篇：疫情中的社交网络

大家好，我是你们的讲师。2020年疫情最严重的时候，我和几个同事参与了一个很有意思的项目——帮助一个大型社区设计线上兴趣配对系统。

这个社区有500多位居民，填写了自己的兴趣爱好——有人喜欢下围棋，有人爱烘焙，有人是摄影发烧友。系统的任务很简单：把居民两两配对（或者不配），使得每对共享至少一个共同兴趣，且让总的配对数量最大化。

我同事喊道："这不就是上一章讲的二分图匹配吗？左边是500个人，右边也是500个人——完美的二分图！"

我摇了摇头："不对。这不是二分图。因为左边的人和右边的人是同一批人。如果 A 和 B 配对了，那么 B 就不能再和 C 配对——这没问题。但关键在于图不是二分图：任何两个人之间都可能有一条边（如果他们有共同兴趣），包括 A 和 B 配对后，A 和 C 之间的边依然存在。这不是能划分成左右两部分的图。"

这个回答的潜台词是：二分图匹配解决不了"一张图上任意两个顶点之间都可能配对"的问题——这需要一般图匹配（General Graph Matching）。而一般图匹配比二分图匹配要难得多，原因在于一个东西：奇环（Odd Cycle）。

为什么奇环让问题变难？在二分图中，所有环都是偶环。搜索增广路时，你永远不会陷入"绕了一圈回到起点"的死循环。但在一般图中，奇环会让你在 DFS 搜索增广路时陷入无限循环——你进入了一个奇数长度的环，从左侧进入，绕了一圈从左侧出来，结果陷入了一个"左右不分"的窘境。匈牙利算法在一般图中直接失效。

直到1965年，Jack Edmonds 提出了花算法（Blossom Algorithm），才第一次给出了一般图最大匹配的多项式时间算法。这篇论文——《Paths, Trees, and Flowers》——被认为是组合优化领域最伟大的论文之一。

二分图方法的局限

我们先通过一个具体例子来理解二分图方法为什么在一般图中失败。

考虑一个简单的三角形——三个顶点三条边，这是个奇环（长度为3）。在匈牙利算法中，我们从空匹配开始，尝试匹配第一个顶点 u₁：

   u₁
  / \
 /   \
u₂───u₃

尝试匹配 u₁:
  DFS: u₁ → u₂ (未匹配) → 匹配!  M = {u₁-u₂}
尝试匹配 u₃:
  DFS: u₃ → u₂ (已匹配给 u₁)
        尝试从 u₁ 找交替路: u₁ → u₃ (未匹配) → 匹配!  M = {u₁-u₃, u₂}?

等等，u₂ 已经被匹配给 u₁ 了，但 u₁ 现在又要和 u₃ 配？
这显然不对——每个顶点只能出现在一条匹配边中。

这个问题出在哪？匈牙利算法假设：从已匹配的左部顶点出发总能找到一条不同的路径回到右部。但在奇环中，从匹配边的一端出发，会通过另一条边绕回来——破坏了一致的方向性。你无法在一个奇数长度的环中维持"左←→右"的交替模式。

一个帮助理解的类比：想象你在一个迷宫里走。二分图就像一条直线走廊，两边分别是男生和女生的房间——你交替进出，方向非常明确。奇环则像一个圆形的房间，你走了一圈后发现自己从同一侧回来了。这时你彻底迷失了方向——你不知道自己现在是在"左边"还是"右边"。这正是一般图匹配的困难所在。

解决这个问题的关键在于：不要试图在奇环内继续寻找增广路——把整个奇环收缩（contract）成一个"超级顶点"，在收缩后的图上继续搜索。这就是花算法的核心思想。

Edmonds花算法

Jack Edmonds 的花算法（Blossom Algorithm）是一般图最大匹配的"圣杯"。它的核心概念是花（Blossom）。

花和收缩

一个花是图中的一个奇环，满足：环上有 2k+1 个顶点，其中恰好 k 条边属于当前匹配 M，且这些匹配边在环上交替排列，环上有唯一一个"未匹配"的顶点（称为花茎，base）。

花的结构（M = {a-b, c-d, e-f}，用 === 表示匹配边）：

           b
          / \
         a   c
          \ /
           d
           |
           e === f

这里 a-b-c-d-e-f 形成了一个奇环（6条边，6个顶点？）
等等，示例需要更清晰：

      a ── b === c
      │         │
      │         │
      e === d ──┘

这里环 a-b-c-d-e-a 是奇环（5个顶点），
M = {b-c, d-e}，a 是未匹配的花茎。

当我们在 DFS 搜索增广路的过程中检测到一个花，我们将其收缩成一个虚拟顶点 b（blossom vertex）。收缩后的图规模变小了，但关键性质不变：原图有增广路 ⇔ 收缩后的图有增广路。

Edmonds 花算法框架：

    1. 从空匹配 M = ∅ 开始
    2. 对每个未匹配顶点 v，做 DFS 搜索增广路：
        a. 用交替路径遍历图，标记每个顶点的"奇偶性"
        b. 如果发现一个奇环（花），收缩它
        c. 在收缩后的图上继续搜索
        d. 如果找到增广路：
            i. 展开所有被收缩的花
            ii. 在每条花的内部重新分配匹配边
            iii. 翻转整条增广路上的匹配状态
    3. 返回 M

收缩和展开的操作并不简单。收缩时，花的所有顶点合并为一个虚拟顶点，花的外部边连接到这个虚拟顶点。展开时，需要根据增广路进入花的位置，在花内部重新构造匹配——这个过程需要精细的维护。

复杂度分析

标准实现的 Edmonds 花算法复杂度为 O(V³)。每个增广阶段进行一次 DFS，每次 DFS 可能发现多个花并进行收缩/展开操作。最坏情况下有 V 个增广阶段（匹配大小从 0 增长到 V/2），每个阶段 O(V²)。

优化版本（如带惰性更新的实现）可以达到 O(V·E·α(V))，其中 α 是反阿克曼函数——几乎 O(V·E)。但对于竞赛和实际应用，O(V³) 的实现已经足够处理大多数中小规模问题。

算法 时间复杂度 适用场景 实现难度
Edmonds 花算法（基础） O(V³) 通用，一般图 高
带惰性更新的 Blossom O(V·E·α(V)) 大规模一般图 极高
匈牙利算法 O(V·E) 仅限二分图 低
缩减为最大流 O(V·√E) 仅限二分图 中
一个诚实的建议：在实际项目中，不要自己实现花算法，除非你非常清楚自己在做什么。花算法的实现涉及大量数据结构细节，即使是经验丰富的程序员也容易出错。使用成熟的库（如 NetworkX 中 `max_weight_matching`、LEMON 图库、Boost Graph Library）比自己从头实现可靠得多。我曾经在一次竞赛中花了4个小时实现一个花算法，最终被一个展开步骤中的 off-by-one bug 折磨到崩溃——从那以后，我再也不在生产代码中手写一般图匹配了。

Tutte-Berge公式

匈牙利算法有增广路定理作为理论基础，一般图匹配也有一个深刻的理论基石——Tutte-Berge 公式。

设 G = (V, E) 是一个图，ν(G) 是最大匹配大小。Tutte-Berge 公式说：

ν(G) = ½ · min_{U ⊆ V} ( |V| - odd(G \ U) + |U| )

其中 odd(G \ U) 是从 G 中移除 U 后，得到的连通分量中
顶点数为奇数的分量个数。

这个公式的含义是：最大匹配的大小由"从图中移除一个顶点集 U 后剩下的奇分量"的某种度量决定。虽然形式有点绕，但它给出了最大匹配大小的完整刻画。

Tutte 定理是 Tutte-Berge 公式的特例：图 G 存在完美匹配 ⇔ 对任意 U ⊆ V，有 odd(G \ U) ≤ |U|。

注意这个条件和 Hall 定理的相似之处！Hall 条件要求 |N(S)| ≥ |S|（邻居数量 ≥ 左部大小），Tutte 条件要求 odd(G\U) ≤ |U|（奇分量数量 ≤ 删除顶点数）。两者在精神上是相通的——都是在说"删除一部分顶点后，剩下的结构要足够'平衡'才能有完美匹配"。

公式背后的直觉：为什么需要删除 U？假设图中有"孤立的"奇数个顶点——它们无法内部两两配对，必然需要一个外部顶点来"吸收"它们。U 就是这些"外部顶点"的候选集。如果删除 U 后剩下太多奇数分量（> |U|），每个奇数分量需要至少一个 U 中的顶点来配对——但 U 的顶点不够用了，于是完美匹配不可能存在。Tutte-Berge 公式告诉我们，最大匹配大小恰好是"全体顶点数减去这个最小值的一半"。

匹配多面体

现在我们进入一个更抽象、但也更强大的视角——把匹配问题看作多面体（Polytope）上的线性优化问题。

对于图 G = (V, E)，用决策变量 x_e ∈ {0,1} 表示边 e 是否被选入匹配。那么匹配 M 对应的解向量 x 满足：

约束 1：∑_{e ∋ v} x_e ≤ 1    ∀ v ∈ V
         （每个顶点最多关联一条匹配边）

约束 2：x_e ≥ 0              ∀ e ∈ E
         （非负性）

如果去掉整数约束 x_e ∈ {0,1}，代以 0 ≤ x_e ≤ 1，我们就得到了匹配问题的LP 松弛。

对于二分图，这个 LP 松弛的顶点（极点）自动是整数解。这是因为二分图的关联矩阵是全幺模矩阵（Total Unimodular，第12章会详述）。这意味着：求解二分图最大匹配可以直接用线性规划求解器，不需要整数规划。

但对于一般图，情况就不同了。简单的 LP 松弛会产生分数极点。例如三角形（3-环）：

三角形三条边各取 x_e = 0.5：
每个顶点的关联边和为 0.5+0.5 = 1（满足约束）
总匹配大小 = 1.5

但整数最大匹配只有 1（只能匹配一条边）！
LP 松弛给出了太宽松的界。

为了得到一般图的正确多面体，需要添加奇集约束（Odd Set Constraints）：

∑_{e ⊆ S} x_e ≤ ⌊|S|/2⌋    ∀ S ⊆ V，|S| 为奇数

这个约束说：奇数个顶点组成的集合 S 中，最多只能匹配
⌊|S|/2⌋ 条边——因为匹配边两两不相交嘛。

加上奇集约束后，LP 松弛的顶点就全部是整数了。这个多面体被称为匹配多面体（Matching Polytope）——Edmonds 在1965年的论文中首次完整刻画了它。

令人震惊的事实：奇集约束的数量是指数级的——2^V 个。但分离问题（判断一个解是否违反某个奇集约束）可以在多项式时间内求解（通过缩图上的最小割）。这正是多面体理论的威力所在——拥有指数数量的约束并不妨碍我们在多项式时间内求解，只要我们能在需要时高效找到被违反的约束。这是第14章的预告。

带权一般图匹配

当边上带有权值时，问题扩展为最大权匹配（Maximum Weight Matching）。这在现实世界中非常常见——配对不仅要有，还要好。

Edmonds-Johnson算法

1970年，Edmonds 和 Johnson 将花算法扩展到了带权情形。核心思想是引入对偶变量（Dual Variables）：

  • 每个节点 v 有一个对偶变量 y_v ≥ 0（节点权）
  • 每个奇集 S 有一个对偶变量 z_S ≥ 0（奇集权）
  • 对偶可行条件：对所有边 (u,v)，y_u + y_v + ∑_{S: u,v∈S, |S|奇数} z_S ≥ w(u,v)
  • 互补松弛条件：如果边 (u,v) 在匹配中，则对应的对偶不等式必须取等号

算法从空匹配和初始对偶变量开始，交替执行：

  1. 在对偶可行条件下，"尽可能"增大匹配（通过调整对偶变量使更多边满足相等条件）
  2. 当匹配不是最大时，调整对偶变量（降低某些 y_v，提高某些 z_S）
  3. 重复直到满足所有互补松弛条件

这个过程高度类似于 KM 算法（第10章）中通过调整顶点标号来加入新的相等子图边。但在这里，对偶变量不仅包括顶点，还包括奇集——这大大增加了复杂度。

核心洞察：带权一般图匹配（也称为"最大权匹配"）是组合优化中最"深"的多项式时间可解问题之一。它的对偶结构涉及顶点和奇集的双重变量，这让它比二分图带权匹配（只有顶点变量）复杂得多。但也正因如此，它在理论计算机科学中有着特殊的地位——它是最复杂的"多项式时间可解"问题之一，被用于检验新算法设计范式的极限。

实际上，有一个著名的故事：1982年，Nimrod Megiddo 提出了一种"通用方法"，可以将组合优化中的多项式时间算法转化为其带权版本。但他的方法依赖"将输入中的整数权值用二进制展开"，结果在实践中非常慢。直到今天，带权一般图匹配的实用实现仍然直接基于 Edmonds-Johnson 算法及其变体，而非 Megiddo 的通用方法。

工业应用

带权一般图匹配的应用包括：

  • 芯片布线（VLSI Routing）：在芯片布线阶段，需要将多个连接点配对（差分对布线），最小化总的线长和干扰。这些连接点之间没有天然的"左右"划分——它是一般图。
  • 图像特征匹配：计算机视觉中的特征点匹配（如 SIFT 特征在两张图片之间匹配），特征点之间没有二分结构。
  • 社交网络中的群落检测：在社区中发现"最紧密连接的对子"，用作群落划分的基础。

回到开篇的线上兴趣配对系统——我们最终用的是 NetworkX 库的 `max_weight_matching`，它底层实现的就是带权一般图匹配算法。对于500个结点，运行时间不到1秒。500个居民中的418个被成功配对，满意度达到92%。看到大家在线下活动（疫情解封后）第一次见面时的那种惊喜，我深刻体会到：好的算法不仅能优化数字，还能改善生活。

本章小结与下章预告

在这一章中，我们学习了：

  • 二分图方法的局限——奇环使匈牙利算法失效，Edmonds 用花收缩解决了这个问题；
  • Edmonds 花算法——检测奇环→收缩→继续搜索→展开，O(V³)；
  • Tutte-Berge 公式——最大匹配大小 = ½·min(|V| - odd(G\U) + |U|)，以及 Tutte 完美匹配定理；
  • 匹配多面体——奇集约束保证了 LP 松弛的积分性；
  • 带权一般图匹配——Edmonds-Johnson 算法用对偶变量处理权值，对偶变量包括节点权和奇集权。
