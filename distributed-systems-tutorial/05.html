引言：分布式系统的基础——网络

1980 年代，一位名叫 David Cheriton 的斯坦福教授在课堂上举了一台"永不故障"的服务器，问学生这台机器是不是完美的分布式系统。学生困惑不已——这就是一台单机啊？Cheriton 说："没错，但只要加上一根网线，它就不再完美了。"这个段子的深意是：一旦引入网络，一切都不确定了。

分布式系统中的所有节点都通过某种网络连接在一起。理解网络通信的延迟、可靠性和性能特征，是设计和调试分布式系统的基本功。本章将带你深入了解从物理层面到传输层的关键通信概念。

网络延迟的量级

在分布式系统中，延迟是决定性因素。不同级别的延迟差异巨大：

操作 延迟 相对比例（以 L1 为 1）
L1 cache 访问 ~0.5 ns 1x
分支预测错误 ~5 ns 10x
L2 cache 访问 ~7 ns 14x
主存访问 (RAM) ~100 ns 200x
SSD 随机读 ~150,000 ns (150 μs) 300,000x
同机房网络往返 ~500,000 ns (500 μs) 1,000,000x
跨机房（同城）往返 ~1,000,000 ns (1 ms) 2,000,000x
跨洲网络往返 ~150,000,000 ns (150 ms) 300,000,000x
工程师的直觉：当你的分布式系统跨洲部署时，一次网络往返的时间相当于执行了 3 亿条 CPU 指令。这就是为什么分布式算法优化往往关注"减少消息轮数"而非"减少计算量"——网络通信才是真正的瓶颈。

TCP vs UDP 深度对比

TCP 和 UDP 是传输层最核心的两个协议，它们的差异塑造了整个分布式应用的通信模式：

特性 TCP UDP
连接方式 面向连接（三次握手） 无连接
可靠性 可靠传输、自动重传（ARQ） 尽力而为，不保证送达
顺序保证 数据按序到达 可能乱序
拥塞控制 有（慢启动、拥塞避免、快速重传） 无（应用程序自行处理）
首部开销 20-60 字节 8 字节
传输模式 字节流（无消息边界） 数据报（有消息边界）
适用场景 HTTP, 数据库, RPC DNS, 流媒体, 实时游戏

三次握手与四次挥手

// TCP 三次握手建立连接
客户端                         服务器
  |                              |
  |------- SYN (seq=x) -------->|  第一次握手
  |<---- SYN+ACK (seq=y,ack=x+1)-|  第二次握手
  |------- ACK (ack=y+1) ------>|  第三次握手
  |                              |
  |======== 数据交换 ===========>|
  |                              |

// TCP 四次挥手断开连接
客户端                         服务器
  |                              |
  |------- FIN (seq=u) -------->|  第一次挥手
  |<------- ACK (ack=u+1) ------|  第二次挥手
  |                              |  （服务器可能还有数据要发）
  |<------- FIN (seq=v) --------|  第三次挥手
  |------- ACK (ack=v+1) ------>|  第四次挥手
  |                              |

为什么 TCP 不总是最佳选择

尽管 TCP 的可靠性非常吸引人，但在某些场景下它并非最优：

  • 队头阻塞 (Head-of-Line Blocking)：TCP 保证按序交付，但如果一个数据包丢失，后续所有已经到达的数据包都要在缓冲区中等待重传，导致延迟增加。即使应用只需要最新数据，也只能等待丢失的包重传完毕。
  • 连接建立开销：每次 TCP 连接需要 1 个 RTT 进行三次握手，加上 TLS 的话需要 2-3 个 RTT。对于微服务间的大量短连接，这个开销不可忽视。
  • 连接池管理：为减少连接开销，服务端需要维护连接池，但连接池的管理（保活、过期、重连）本身就是复杂度来源。

QUIC（基于 UDP 的下一代传输协议）正是为了解决这些问题而生。它内置了 0-RTT 连接建立、多路复用（避免队头阻塞）和更好的拥塞控制。HTTP/3 就是建立在 QUIC 之上的。

Nagle 算法与 Delay ACK——一个 40ms 的坑

这是一个非常真实的"踩坑"案例。某团队开发了一个实时交易系统，客户端发送小数据包（几十字节）到服务端。测试发现，每次请求的响应时间在局域网中竟然高达 40ms，远超预期。

排查后发现问题出在 Nagle 算法和 Delay ACK 的交互上：

  • Nagle 算法：为了减少小数据包数量，TCP 会在未收到前一小包 ACK 时，将后续小数据缓存起来，等到积累到 MSS 大小或收到 ACK 时才发送。
  • Delay ACK：接收方不立即发送 ACK，而是等待最多 200ms（Linux 默认 40ms），期望和数据一起捎带确认。

当 Nagle 算法在发送端等待 ACK，而 Delay ACK 在接收端等待数据时，就形成了典型的 "死锁"——两边都在等待对方。最后 Delay ACK 超时（40ms）发送了 ACK，"僵局"才被打破。

解决方案：对于交互式应用或延迟敏感的业务，应该禁用 Nagle 算法，即在 socket 上设置 TCP_NODELAY 选项。Redis 就是通过这种方式保证了微秒级的响应时间。对于批量数据传输，Nagle 算法仍然是有益的。

网络性能调优

下面是分布式系统服务端常见的网络参数调优经验值：

参数 推荐值 说明
TCP_NODELAY 开启 禁用 Nagle 算法，低延迟场景必选
SO_REUSEADDR 开启 允许快速重用 TIME_WAIT 状态的端口
SO_RCVBUF 4096 × 带宽（字节） 接收缓冲区，高吞吐场景适当增大
SO_SNDBUF 4096 × 带宽（字节） 发送缓冲区
net.ipv4.tcp_keepalive_time 600 (秒) TCP 保活间隔，默认 7200 太长
net.ipv4.tcp_tw_reuse 1 允许重用 TIME_WAIT 连接
最佳实践：不要盲目照搬参数值，需要根据实际业务场景进行基准测试。视频流服务的缓冲区和实时交易系统的缓冲区需求完全不同。始终以 perf 或 tcpdump 的实际测量结果作为调优依据。

网络通信是分布式系统的"血液系统"。理解了 TCP/UDP 的差异和网络延迟的本质，我们才能正确设计分布式通信协议。下一章将探讨一个所有分布式通信都必须面对的问题——序列化与编解码，即如何在节点之间高效地表达和解析数据。
