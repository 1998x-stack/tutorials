第九章：HTTP缓存

各位同学大家好。上一章讲了连接管理，一个完整的TCP连接可能花费几百毫秒才能建立。但如果能让资源在本地缓存起来，连请求都不用发，那速度就是质的飞跃。这就是缓存的意义——最好的网络请求，是没有请求。

我们来看一组数据：未命中缓存的请求从中国访问美国服务器大约需要300-500ms。而缓存命中的请求——直接从浏览器本地读，时间接近于0ms（实际上就是磁盘读取的几毫秒）。一个电商首页可能有200个请求，如果全部缓存命中，页面加载时间能从几秒降到几百毫秒，这就是缓存的价值。

9.1 为什么需要缓存

HTTP缓存解决三个核心问题：

  • 延迟——避免网络往返。让数据从"近处"获取，而不是从源服务器。
  • 带宽——减少数据传输量。一次304响应只有几十字节，远小于完整响应。
  • 服务器负载——降低源服务器的请求量。一个热门资源如果缓存率高，源服务器可能只需要处理1%的请求。

缓存的核心理念是新鲜度（Freshness）和验证（Validation）。新鲜度决定资源是否可以直接使用缓存副本；验证决定当资源可能过期时，如何向服务器确认它是否真的变了。

关键数据：Google的一项研究发现，浏览器缓存可以将页面加载时间降低60-80%。对于重复访问，这个效果更加明显——首次访问加载5秒的页面，第二次可能只要1秒。我在做性能优化时，第一步永远是检查缓存配置：改好缓存，不花一分钱硬件就拿到50%以上的性能提升，这是ROI最高的优化手段。

9.2 Cache-Control详解

Cache-Control 是HTTP/1.1引入的缓存控制主头，功能非常强大，可以说一个头搞定所有缓存策略。它的常见指令包括：

指令含义示例使用场景
max-age=N资源在N秒内是新鲜的max-age=31536000静态资源，缓存一年
public任何缓存（包括CDN）都可以存public, max-age=3600公开的非个性化内容
private仅浏览器可以缓存，中间代理不可private, max-age=600用户个人信息页面
no-cache每次使用前必须向服务器验证no-cacheHTML页面，需要最新版本
no-store完全不缓存（不存任何副本）no-store银行交易、支付接口
must-revalidate过期后必须验证，不可使用过期缓存must-revalidate, max-age=3600需要新鲜度的内容
immutable资源永远不会变（配合hash文件名）max-age=31536000, immutable带hash的静态资源

很多人会混淆 no-cache 和 no-store。这里有个重要的区别：no-cache 不是"不缓存"，而是"要验证才能用"——浏览器可以存下来，但每次使用前得问服务器"我的缓存还新鲜吗？"服务器说"没变"（304）就继续用。而 no-store 才是真的"绝对不能存"——敏感数据如银行余额、支付token，得用这个。

调试命令：想看一个资源的缓存策略？在终端里跑一个curl就能看到：

curl -I https://cdn.example.com/app.abc123.js

HTTP/1.1 200 OK
content-type: application/javascript
cache-control: public, max-age=31536000, immutable
etag: "abc123def456"

-I 参数只返回响应头，不下载body。看到 max-age=31536000 就知道这个资源被缓存一年，配合 immutable 告诉浏览器在这一年内永远不用验证。

9.3 ETag与If-None-Match

ETag（实体标签）是资源内容的"指纹"——一个字符串，资源变了它就变。服务器在响应中返回ETag头，浏览器在后续请求中用 If-None-Match 带上这个值，服务器比对后决定返回304还是200。

ETag的生成策略通常有三种：

  • 文件哈希——对文件内容做MD5、SHA1等哈希，内容变则哈希变。最准确，但计算哈希有CPU开销。
  • 版本号/时间戳——用文件的最后修改时间或版本号做ETag。简单，但在集群环境中不同服务器时间可能不一致。
  • 组合方法——哈希+版本号，兼顾准确性和性能。

ETag还有强弱之分：

  • ETag: "33a64df5"——强ETag，内容一变值就变。字节级别的比对。
  • ETag: W/"33a64df5"——弱ETag，语义等价即可视为相同。比如一份文档换个行、改个空格，强ETag会认为变了，弱ETag可能认为没变。前面加 W/ 前缀表示弱ETag。

ETag的验证流程：

第一次请求：
  GET /style.css
  响应：200 OK
    ETag: "abc123"
    Cache-Control: max-age=3600

第二次请求（1小时后，缓存过期）：
  GET /style.css
  If-None-Match: "abc123"
  响应：304 Not Modified
    ETag: "abc123"   ← 没变，告诉浏览器继续用缓存

9.4 Last-Modified与If-Modified-Since

这是比ETag更早的验证机制，基于时间戳。服务器在响应中返回 Last-Modified 头，浏览器后续请求用 If-Modified-Since 带上这个时间：

第一次请求：
  GET /data.json
  响应：200 OK
    Last-Modified: Tue, 28 Jun 2026 10:00:00 GMT
    Cache-Control: max-age=3600

第二次请求：
  GET /data.json
  If-Modified-Since: Tue, 28 Jun 2026 10:00:00 GMT
  响应：304 Not Modified  ← 资源在这个时间之后没改过

Last-Modified有一个难以避免的问题——时钟偏差（Clock Skew）。假如你的服务器是集群，节点A的时间比节点B快了10秒。用户从节点A获取资源得到Last-Modified时间T，然后验证请求落到节点B，节点B看这个时间说"没问题"，但实际资源已经变了——因为节点B的时间比节点A慢，它认为T是一个未来的时间。这就是为什么很多现代架构建议优先使用ETag，它对内容做哈希，不受时钟影响。

踩坑故事：有次做静态资源部署，我们更新了CSS文件但用户就是看不到新样式。排查发现问题出在Last-Modified上：CI/CD流水线部署文件时，因为用了文件复制命令，新文件的"最后修改时间"竟然比老文件的还早。浏览器拿着旧时间戳去验证，服务器说"你没变"，就一直用旧缓存。从那以后我们全面切换到了ETag+内容哈希方案，时间戳只作为补充机制。

9.5 缓存层级

HTTP缓存不是单层的，它是多层协作的系统：

层级位置特点缓存命中延迟
内存缓存浏览器内存中速度最快，容量小，页面关闭后释放~0ms
磁盘缓存浏览器磁盘上容量大，跨会话持久，速度略慢~1-5ms
Service Worker浏览器中（编程控制）可编程缓存策略，离线支持~0-5ms
CDN缓存边缘节点地理分布，减轻源站压力~10-50ms
反向代理缓存服务器前端（如Nginx、Varnish）保护应用服务器，聚合请求~1-10ms
源服务器应用服务器/数据库最终数据源头~100-1000ms

整个缓存流程像是一个漏斗：浏览器先从内存缓存找，没找到就查磁盘缓存，再没有就去CDN，最后才到源服务器。每一层命中都会跳过后续所有层，这就是逐级缓存的价值。

9.6 缓存策略最佳实践

不同类型的资源应该使用不同的缓存策略：

静态资源（JS、CSS、图片、字体）——缓存时间最长

Cache-Control: public, max-age=31536000, immutable

关键点：文件名或URL中带内容哈希。比如 app.a1b2c3.js，内容变了哈希就变，URL也跟着变，之前的缓存自动失效。这是最推荐的"cache-busting"方案。immutable 告诉浏览器一年内完全不需要重新请求。

HTML页面——每次都需要验证

Cache-Control: no-cache

HTML可能频繁更新，所以用 no-cache 让浏览器每次验证。配合 ETag 或 Last-Modified，内容没变就返回304（约50ms），变了就返回200（正常加载）。

API响应——视场景定

// 用户信息API（私有）
Cache-Control: private, max-age=300

// 公共数据API（可CDN缓存）
Cache-Control: public, s-maxage=600, max-age=60

// 实时数据API
Cache-Control: no-cache

注意 s-maxage 是专门给CDN/共享缓存用的指令，优先级高于 max-age。上面例子里，浏览器缓存60秒，但CDN可以缓存600秒。

最佳实践速查：当你不知道用什么缓存策略时，记住这三条：1）**带哈希的文件名** → 一年缓存 + immutable；2）**纯静态但没哈希** → max-age=3600 + 配合ETag验证；3）**动态内容** → no-cache + ETag验证。这样既保证性能又保证正确性。

踩坑故事：有次上线活动页面，运营急着更新文案，我们改了HTML内容就发布了。结果用户看到的是旧页面——因为我们给HTML也设了 max-age=86400（缓存一天）。CDN和浏览器都缓存了旧版本。运营打电话来骂，我们紧急给HTML去掉了缓存改成no-cache才恢复。从那以后，我们的HTML永远用 no-cache，API动态数据绝不设长缓存。
