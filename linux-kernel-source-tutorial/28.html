大家好，我是你们的讲师。欢迎来到第28章——我们这个课程的最后一章。前27章我们做了一件很难的事情：把Linux 0.11内核从引导扇区到设备驱动的每一段核心代码都读了一遍。但如果你只停留在"读代码"的层面，学习的闭环还没有真正完成。

引言

只有当你亲手写过内核代码之后，你才能真正理解内核。这不是什么玄学——这是一个工程事实。读代码是被动输入，写代码是主动输出。你在写代码的过程中踩过的每一个坑、调试的每一个bug、解决的每一个问题，都会让你对内核的理解上升一个台阶。

本章提供了两个项目方案：ramdisk驱动（主推，中等难度，涉及块设备驱动的完整流程）和简单proc文件系统（备选，主要涉及VFS接口实现）。建议你至少完成其中一项。如果时间充裕，两项都做。

项目一：实现ramdisk驱动

ramdisk是一个用内存模拟块设备的驱动。它的核心思想很简单：在一段预先分配的内存上模拟磁盘的块读写操作。当内核向ramdisk发出读块请求时，驱动从内存中拷贝数据到请求缓冲区；写请求则反向拷贝。

这个项目的精彩之处在于：它复用了我们在第19-20章学到的所有块设备驱动的知识——请求队列（request queue）、请求处理函数、设备号注册、blk_dev[]表。你将看到hd.c的代码模式如何被复用到一个新的驱动中。

/* rd.c — 在Linux 0.11上实现ramdisk驱动 */
/* 仿照 kernel/blk_drv/hd.c 的request处理模式 */

#include <linux/config.h>
#include <linux/sched.h>
#include <asm/system.h>
#include <asm/io.h>

/* ramdisk设备号 — 使用主设备号1（未使用的RAM盘） */
#define RAMDISK_MAJOR  1
#define RAMDISK_SIZE   (1024 * 1024)    /* 1 MB ramdisk */
#define BLOCK_SIZE     1024             /* 块大小 */

static char *rd_buffer = NULL;          /* 内存中的"磁盘" */

/* 请求处理函数 — 被do_hd_request()类似的模式调用 */
void do_rd_request(void) {
    int block;
    struct buffer_head *bh;
    char *disk_addr;

    while (1) {
        /* 从请求队列中取出一个请求 */
        if (!(CURRENT->bh)) break;

        bh = CURRENT->bh;
        block = CURRENT->block;
        disk_addr = rd_buffer + block * BLOCK_SIZE;

        if (CURRENT->cmd == WRITE) {
            /* 写：从buffer_head拷贝到ramdisk */
            memcpy(disk_addr, bh->b_data, BLOCK_SIZE);
        } else {
            /* 读：从ramdisk拷贝到buffer_head */
            memcpy(bh->b_data, disk_addr, BLOCK_SIZE);
        }

        /* 标记请求完成，唤醒等待的进程 */
        end_request(1);
    }
}

/* 初始化函数 — 在内核启动时调用 */
void rd_init(void) {
    int i;

    /* 1. 分配内存空间作为ramdisk的存储介质 */
    rd_buffer = (char *)kmalloc(RAMDISK_SIZE, GFP_KERNEL);
    if (!rd_buffer) {
        printk("rd_init: failed to allocate buffer\n");
        return;
    }

    /* 2. 清零所有"磁盘块" */
    memset(rd_buffer, 0, RAMDISK_SIZE);

    /* 3. 注册块设备 */
    blk_dev[RAMDISK_MAJOR].request_fn = do_rd_request;

    /* 4. 创建/dev/ram0设备节点 (mknod /dev/ram0 b 1 0) */
    printk("Ramdisk initialized: %d KB at 0x%x\n",
           RAMDISK_SIZE / 1024, (unsigned)rd_buffer);
}

实现步骤（请跟着做）：

  • 步骤1：在kernel/blk_drv/目录下创建rd.c文件
  • 步骤2：定义设备号为1（确认与现有设备不冲突）
  • 步骤3：实现do_rd_request()函数——处理读写请求
  • 步骤4：实现rd_init()——分配内存、将处理函数注册到blk_dev[]表
  • 步骤5：修改kernel/blk_drv/Makefile，将rd.c加入编译
  • 步骤6：在kernel/blk_drv/ll_rw_blk.c中引用外部函数
  • 步骤7：修改init/main.c，在init()中调用rd_init()
  • 步骤8：重新编译内核，用QEMU启动测试
  • 步骤9：在Linux 0.11中用mknod /dev/ram0 b 1 0创建设备节点

项目二：简单procfs（备选）

如果你对文件系统更感兴趣，可以考虑实现一个只读的/proc虚拟文件系统。procfs是一个"伪造"的文件系统——它不存储在磁盘上，而是动态生成内容。当用户cat /proc/meminfo时，内核实时收集内存信息并格式化为文本返回。

在0.11上实现procfs的核心思路：

/* 在Linux 0.11上实现一个简易procfs的核心步骤 */
/*
 * 关键难点：0.11只实现了MINIX文件系统，所有文件操作假设数据在磁盘上。
 * procfs没有磁盘数据，需要"虚拟化"inode的读取逻辑。
 */

/* 1. 注册proc文件系统类型 */
/* 在fs/super.c中添加proc_fs_type */
static struct file_system_type proc_fs_type = {
    NULL,           /* next */
    "proc",         /* name */
    FS_SINGLE,      /* flags */
    proc_read_super /* read_super */
};

/* 2. 实现proc_read_super — 创建根inode */
struct super_block *proc_read_super(struct super_block *s, void *data, int silent) {
    struct inode *root_inode;
    s->s_mounted = iget(s, PROC_ROOT_INO);
    root_inode = s->s_mounted;
    root_inode->i_mode = S_IFDIR | S_IRUGO | S_IXUGO;
    return s;
}

    return inode;

开发流程

从需求到测试的完整流程，不是"先写代码再debug"的蛮力模式，而是一个需要你反复循环的过程。下面这张checklist涵盖了0.11上开发内核模块的标准流程：

=== Linux 0.11 内核开发 Checklist ===

[ ] 需求分析
    [ ] 明确要做什么功能
    [ ] 确定与哪些现有模块交互
    [ ] 画出数据和函数调用流程图

[ ] 设计方案
    [ ] 选择将代码放在哪个目录
    [ ] 定义数据结构（有没有现成的可用？）
    [ ] 确定与内核其他部分的接口函数
    [ ] 考虑错误路径：内存不足？参数无效？

[ ] 编码
    [ ] 编写.c文件
    [ ] 编写.h文件（如果需要）
    [ ] 修改Makefile
    [ ] 在main.c或其他地方添加初始化调用

[ ] 编译
    $ make clean  # 清理旧的目标文件
    $ make        # 编译整个内核

[ ] 调试
    [ ] 编译错误：检查语法、函数声明、头文件路径
    [ ] 链接错误：检查外部符号是否声明、Makefile是否遗漏
    [ ] 运行时panic：检查空指针、未初始化变量

[ ] 测试
    [ ] 功能测试：功能是否如预期工作
    [ ] 边界测试：最坏情况下的行为（满盘、最大并发）
    [ ] 压力测试：重复操作1000次看是否稳定

[ ] QEMU验证
    $ make start  # 启动QEMU模拟器
    # 在模拟器中执行功能测试
高效开发的小技巧：在0.11上开发时，printk()是你最好的朋友——它等价于用户态的printf，但输出到内核日志。你可以把它当作"printf调试法"来用：在每个函数的入口和出口加printk，打印关键变量的值。当内核panic时，这些printk输出会停留在屏幕上，告诉你最后执行到哪里。这个方法虽然"原始"，但在0.11的简单环境下非常高效。

踩坑故事

内核开发远比用户态编程容易"翻车"。以下是初学者在写ramdisk驱动时最容易栽的三个跟头——我在教学和实践中都见过：

踩坑一：设备号冲突。0.11的设备号分配不像现代内核那样有统一的注册机制。如果你不小心使用了已经被占用的主设备号（比如2代表软盘），内核会尝试调用错误的请求处理函数，导致不可预测的行为。解决方法：分配设备号前，查看kernel/blk_drv/下所有驱动使用的设备号，确保你的设备号不与现有驱动冲突。

踩坑二：忘记初始化结构体中的指针字段为NULL。C语言中局部变量和kmalloc分配的内存不会自动清零。如果你定义了一个struct request但没有初始化所有字段，其中的函数指针可能包含随机值。当内核尝试调用这个"野指针"函数时——直接panic。0.11的内核没有现代内核的防御性检查（如BUG_ON()），一旦跳转到非法地址，你能看到的只有一串十六进制寄存器dump，没有任何错误信息。所以：每次定义结构体后，立即memset(&stru, 0, sizeof(stru))。

踩坑三：请求队列死循环。do_rd_request()中有一个while(1)循环。如果处理完一个请求后没有正确调用end_request(1)（标记请求完成），或者判断请求结束的条件写错了，循环会永远运行下去。更隐蔽的变体是：你处理了请求但没有从队列中移除它，下次同样的请求再次被处理——无限循环。参考hd.c的do_hd_request()，注意每一处end_request()的调用逻辑。

内核panic后怎么办？在QEMU中运行0.11时，如果遇到内核panic（屏幕显示"Kernel panic: ..."），QEMU会暂停CPU。此时你可以关闭QEMU窗口或按Ctrl+A然后按X退出。如果panic信息不足以定位问题，可以在关键路径上添加printk()输出，重新编译后再次运行。记住——在0.11的环境中，printk()是最可靠的调试手段。
