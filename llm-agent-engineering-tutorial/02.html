第2章 感知-思考-行动循环

大家好，我是你们的讲师。在上一章中我们了解了 Agent 的"全景图"，但有个问题可能一直萦绕在你脑海里：Agent 到底是怎么"思考"的？ 它不是像传统的 if-else 逻辑那样硬编码行为，也不是简单地让 LLM 生成一段文字。那么，Agent 是如何在每一步中决定"我现在要做什么"的？

答案就在本章的核心主题——感知-思考-行动循环（Perceive-Think-Act Loop），也就是业内常说的 ReAct 模式（Reasoning + Acting）。这个循环是所有 Agent 系统的心脏。无论你用 LangChain、AutoGen、CrewAI 还是自研框架，底层运行的都是一颗"ReAct 引擎"。理解了这个循环，你就掌握了 Agent 设计的精髓。

核心洞察： Agent 不是"更聪明的 LLM"，而是一种全新的计算范式。传统的"输入-处理-输出"管线被替换为"感知-推理-行动-观察"的闭环——Agent 在循环中不断感知环境变化、推理下一步行动、执行工具调用，然后观察结果并调整策略。每一次迭代，Agent 都离最终目标更近一步。

2.1 从 LLM 到 Agent 的架构飞跃

要理解 ReAct 循环，首先得搞清楚：普通的 LLM 调用和 Agent 循环到底有什么区别？

一个普通的 LLM 调用，就是"输入 Prompt → 模型生成 → 输出回复"的单次交易。你问"今天天气怎么样？"它回答"抱歉，我无法获取实时天气数据"——对话结束。没有后续行动，没有能力去获取信息。

Agent 循环则是这样的：用户提问 → LLM 分析问题 → 决定需要查询天气 API → 调用函数 → 拿到实时数据 → 组织成回答 → 输出给用户。注意这里的"循环"——LLM 不再是终点，而是"决策中枢"，它决定每一步要做什么，然后观察结果，再决定下一步。

下面这个表格清晰地展示了二者的差异：

维度 普通 LLM 调用 Agent 循环
调用次数 单次 多次迭代，直到任务完成
状态管理 无状态（除非显式传入历史） 有状态循环，每一步维护当前上下文
工具使用 无 动态选择并调用外部工具
决策粒度 一次生成完整回复 多步推理，每一步决定下一步
错误恢复 无（出错就重试整个请求） 可捕获错误并调整策略继续执行
计算消耗 可控（单次推理） 不确定性高（取决于迭代次数）

这个飞跃在工程上的意义是什么？用一个真实的例子来说明。2024 年，我们团队接手了一个项目：为客户构建一个自动化的竞品分析 Agent。需求是"给我最新的 10 篇关于竞品的新闻报道，总结关键信息"。如果只是简单的 LLM 调用，模型只能给出一个泛泛而谈的回复。但用 Agent 循环——调用搜索 API → 逐个阅读搜索结果 → 提取关键信息 → 交叉验证 → 生成结构化报告——结果的质量完全不在一个量级。这就是"感知-思考-行动"循环的力量。

小贴士： 一个常见的误解是"Agent 就是 LLM + 工具调用"。但事实上，工具调用只是"行动"环节的一部分。真正的 Agent 架构还需要"感知"（多模态输入、环境状态监控）和"推理"（规划、决策、反思）两个环节的完整设计。

2.2 ReAct 模式详解

ReAct 这个词是 Reasoning（推理）和 Acting（行动）的合成词，由 Shunyu Yao 等人在 2022 年的论文《ReAct: Synergizing Reasoning and Acting in Language Models》中提出。这篇论文的核心思想非常直观：推理和行动不是独立的两个阶段，而是相互增强的协同过程。

Thought / Action / Observation 三步循环

ReAct 循环的每一次迭代包含三个步骤：

  1. Thought（思考）： LLM 分析当前状态，解释"我在哪里、目标是什么、下一步应该做什么"。这一步是"推理"环节的核心，模型会回顾已经完成的工作，评估进度，然后规划下一步行动。
  2. Action（行动）： 根据思考结果，LLM 选择一个工具并给出调用参数。这一步可能是查询数据库、调用 API、搜索网络、执行代码等。关键特征是模型输出的是"工具调用指令"，而不是自然语言回复。
  3. Observation（观察）： 系统执行工具调用后将结果返回给 LLM。LLM "观察"这个结果，将其融入上下文，然后进入下一轮循环——再次 Thought → Action → Observation。

这个循环一直持续，直到 LLM 判断任务已经完成（或达到最大迭代次数限制）。

踩坑实录： 2024 年初我们在一个文档处理 Agent 中遇到了经典问题——Agent 陷入了无限循环。它不断 Thought "我需要读取这个文件的信息" → Action "调用 read_file" → Observation "文件内容太长无法直接处理" → Thought "我需要读取这个文件的信息"…… 解决方案是给循环加上"最大步数限制"（max_iterations）和"无进展检测"（stuck detection），如果连续 N 步没有实质性进展，就主动终止并上报。

ReAct 与 CoT 的对比

说到推理，你可能听说过 Chain-of-Thought（CoT）。CoT 让模型在回答前先"想一想"，生成中间推理步骤。ReAct 和 CoT 的理念有共通之处，但有一个本质区别：

维度 Chain-of-Thought ReAct
推理边界 纯文本推理，依赖模型内部知识 推理 + 外部信息获取，突破知识边界
可验证性 推理链不可验证（模型编造的"幻觉推理"） 每一轮 Observation 都是可验证的外部事实
工具交互 无 天然支持工具调用
错误恢复 一旦推理出错，整条链崩溃 可在下一步 Observation 中修正方向

这么说吧：CoT 是"闭着眼睛想"，ReAct 是"睁着眼睛做"。在实际应用中，两者并不冲突——你完全可以在 ReAct 的 Thought 步骤中嵌入 CoT 推理，让模型在"做"之前先深入"想"清楚。事实上，大多数现代 Agent 框架都是这么做的。

2.3 LLM 推理循环内部机制

你是不是好奇过：LLM 到底是怎么"决定"调用哪个工具的？这涉及到 LLM 推理循环的内部机制。让我们深入到底层看看。

System Prompt 中的"行为契约"

LLM 本身并不知道它是一个 Agent。它之所以表现出"思考 → 行动 → 观察"的行为模式，完全是因为 System Prompt 中定义了这样一个"行为契约"。典型的 Agent System Prompt 会包含以下关键指令：

你是一个智能助手，能够使用工具完成用户的任务。

每次回复时，请遵循以下格式：
1. 首先输出你的思考过程（Thought），分析当前状态和下一步行动计划
2. 然后输出你要调用的工具和参数（Action）
3. 等待工具执行结果（Observation），然后决定下一步

如果任务完成，请直接回复用户最终答案，不要输出工具调用。

你可以使用的工具：
- search_web(query): 搜索网络获取最新信息
- read_url(url): 读取指定URL的内容
- calculate(expression): 执行数学计算

这个 Prompt 的关键在于——它把"使用工具"变成了模型输出格式的一部分。模型不是在"想要调用工具"后才输出工具指令，而是在生成文本时，按照格式规范在适当位置输出工具调用标记。

工具调用的底层实现

从工程角度来看，不同模型提供商的工具调用实现方式略有不同：

  • OpenAI： 在 API 请求中通过 tools 参数传入工具定义，模型返回的响应中会包含 tool_calls 字段。服务端对工具调用进行了原生支持——模型在生成过程中会"暂停"文本输出，转而输出结构化的工具调用请求。
  • Anthropic（Claude）： 通过 tool_use 内容块实现，模型会在消息流中插入 tool_use 类型的 content block，包含工具名称和输入参数。
  • Google Gemini： 通过 functionCallingConfig 和 FunctionDeclaration 实现类似机制。

虽然接口各异，但底层逻辑是一致的：工具调用是模型输出的一种特殊形式——不是在自然语言文本中"提到"工具，而是以结构化数据的形式"请求"工具执行。

深入理解： 工具调用的本质是"约束解码"——LLM 的输出空间从"所有可能的 Token 序列"被约束为"符合工具调用格式的结构化数据"。这比让模型在自由文本中提到工具名称要可靠得多。这也是为什么 2026 年的主流模型都原生支持工具调用格式——它从根本上减少了"模型会说要用工具但格式不对"的问题。

2.4 工具编排的起源与发展

ReAct 循环解决了"模型如何决定使用工具"的问题，但还有一个同样重要的问题：多个工具的调用顺序和依赖关系如何管理？ 这就是工具编排（Tool Orchestration）的领域。

从线性链到图结构

工具编排的演化经历了三个阶段：

  • 第一阶段——线性链（2023）： 最早的 Agent 只是简单地将工具调用串成一条链——A → B → C。比如搜索 → 阅读 → 总结。这种模式简单直观，但无法处理分支和条件逻辑。
  • 第二阶段——条件路由（2024）： LangChain 引入的 Router 机制让 Agent 可以根据中间结果选择不同的下游工具。比如搜索的结果如果不足，就走"重写查询再搜索"的分支；如果充足，直接进入阅读阶段。
  • 第三阶段——图结构（2025-2026）： LangGraph 和类似的框架工具将编排升级为有向图（DAG），节点是工具调用或推理步骤，边是数据和状态流。一个节点可以有多个输入和多个输出，支持并行执行和复杂的状态转换。

到了 2026 年，图结构编排已经成为了行业标准。你定义一个状态图，Agent 就在这个图上"游走"——每个节点执行一个工具调用或推理步骤，每条边定义了状态转换条件。这种方式的表达能力远超过线性链。

实践经验： 我们团队在实践中发现，对大多数应用场景来说，不需要设计非常复杂的图结构。一个"星形"图——中心一个推理节点，周围环绕工具节点——就能覆盖 80% 的需求。只有在需要复杂条件分支和并行执行时，才引入更复杂的图拓扑。我建议你从简单开始，不要一开始就追求"图论级别的编排"。

2.5 完整 ReAct 循环代码示例

理论讲得够多了，让我们来看一个真正可运行的 ReAct 循环。下面这段代码不依赖任何 Agent 框架，只用 OpenAI SDK 实现了完整的 ReAct 循环：

import json
from openai import OpenAI

client = OpenAI()

# 定义工具
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的实时天气",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "城市名称"}
                },
                "required": ["city"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "search_web",
            "description": "搜索网络获取最新信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {"type": "string", "description": "搜索关键词"}
                },
                "required": ["query"]
            }
        }
    }
]

# 模拟工具执行
def execute_tool(name, args):
    if name == "get_weather":
        return json.dumps({"city": args["city"], "temp": "26°C", "condition": "晴"})
    elif name == "search_web":
        return json.dumps({"results": [f"关于'{args['query']}'的搜索结果"]})
    return json.dumps({"error": "未知工具"})

# ReAct 循环
def react_loop(user_message, max_steps=5):
    messages = [
        {"role": "system", "content": "你是一个智能助手。使用工具完成用户的任务，完成后直接回复。"},
        {"role": "user", "content": user_message}
    ]

    for step in range(max_steps):
        print(f"\n--- Step {step + 1} ---")
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=tools
        )

        msg = response.choices[0].message
        messages.append(msg)

        if msg.tool_calls:
            for tc in msg.tool_calls:
                func_name = tc.function.name
                func_args = json.loads(tc.function.arguments)
                print(f"Thought: 我决定调用 {func_name}({func_args})")
                result = execute_tool(func_name, func_args)
                print(f"Observation: {result}")
                messages.append({
                    "role": "tool",
                    "tool_call_id": tc.id,
                    "content": result
                })
        else:
            # 没有工具调用，任务完成
            print(f"Final Answer: {msg.content}")
            return msg.content

    return "达到最大步数限制"

# 运行
react_loop("北京今天的天气怎么样？顺便帮我搜一下最近的AI新闻。")

这段代码清晰地展示了 ReAct 循环的四个关键环节：

  1. 消息列表维护：每一步的推理过程和工具结果都追加到 messages 中，保持上下文连贯性。
  2. 工具调用检测：检查 LLM 返回的 tool_calls 字段，判断模型是否需要调用工具。
  3. 工具执行与结果注入：执行工具并将结果以 tool 角色消息注入回对话。
  4. 终止条件判断：当模型不再调用工具（直接输出文本回复）时，循环终止。

2.6 常见陷阱与优化技巧

在我过去两年帮助团队落地 Agent 系统的过程中，总结了一些 ReAct 循环中最容易出问题的"坑"和相应的优化策略。

陷阱一：工具结果爆炸

工具调用的结果可能非常庞大——查询数据库返回数千行记录、读取的网页有几万字。这些结果全部塞进消息列表，会让上下文窗口迅速膨胀。优化策略：

  • 对工具结果做摘要（Summarization），只保留关键信息
  • 设置工具结果最大长度（如 2000 tokens），超出则截断
  • 使用"滑动窗口"策略，只保留最近 N 轮的消息

陷阱二：模型"忘记"任务目标

在长时间的多步循环中，LLM 可能会"迷失"——忘记最开始的任务目标是什么。这在复杂任务中尤为常见。优化策略：

  • 在每一步的 System Prompt 开头重申原始任务目标
  • 使用"任务清单"（task checklist）——每完成一项就标记一项，让模型随时知道进度
  • 在每一步的 User Message 中注入"进度摘要"

陷阱三：循环无收敛

Agent 可能在几个工具之间反复切换，无法完成最终目标。这是早期 Agent 系统最令人头疼的问题。优化策略：

  • 设置硬性最大迭代次数（如上例的 max_steps）
  • 实现"停滞检测"——如果连续 N 步调用的工具和结果没有变化，主动退出
  • 在 System Prompt 中强调"如果尝试了合理次数仍无法完成，请如实告知用户并提供部分进展"
踩坑实录： 2025 年我们在一个自动化数据采集项目中使用 ReAct Agent，遇到了令人哭笑不得的问题——Agent 为了回答"2024 年全球 GDP 排名"这个问题，调用搜索工具 12 次，阅读了 8 个不同来源的网页，最后给出的答案还不如直接问 GPT-4 准确。原因是它读了太多互相矛盾的数据来源，反而把自己搞糊涂了。后来我们加了一条规则：对于常识性问题或模型本身具备的知识，优先使用模型内部知识，只在确需最新数据时才调用工具。这被称为"先想后查"策略。

优化技巧总结

优化方向 具体策略 效果
Token 控制 工具结果摘要、滑动窗口、max_tokens 限制 减少 50-70% Token 消耗
收敛保障 最大步数、停滞检测、早退机制（Early Exit） 减少 90% 的死循环问题
质量提升 先想后查、任务清单、定期反思 任务成功率提升 30-60%
性能优化 并行工具调用、缓存相同查询结果、流式输出 响应时间减少 40-80%
