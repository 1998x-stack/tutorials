第18章 查询规划器

大家好，我是你们的讲师。前面几章我们都在聊 SQLite 的"下半身"——怎么存数据、怎么保证事务、怎么加锁并发。从这一章开始，我们转向"上半身"——查询规划器 (Query Planner)，也就是 SQLite 中负责决定"怎么写 SQL 才能跑得快"的那个最聪明的组件。

查询规划器的任务听起来简单：给定一条 SQL 语句，找出执行它最高效的方式。但真正实现起来极其复杂——因为可能的执行计划数量随着 JOIN 的表数量呈阶乘级增长。对于 6 个表的 JOIN，可能的连接顺序就有 720 种（6!），再乘以每种顺序下每个表的索引选择方案，轻松上万。SQLite 的查询规划器要在毫秒级别从这上万个计划中挑出最优的一个——这就是我们这一章要揭秘的黑科技。

一个对比： PostgreSQL 和 MySQL 的查询规划器动辄几万行代码，使用了复杂的统计信息（直方图、相关性、MCV 列表）、并行化执行、自适应规划等高级技术。SQLite 的查询规划器（主要在 src/where.c 中）只有约 7000 行 C 代码，没有后台统计信息收集、没有并行执行。它用了一系列巧妙的启发式算法和穷举搜索的平衡策略，在"足够好"和"足够快"之间找到了自己的黄金分割点。这正是嵌入式系统的工程哲学——在资源约束下做到最好。

查询规划的三步

SQLite 的查询规划分为三个主要阶段：

阶段输入输出关键函数
语法树SQL 文本抽象语法树 (AST)sqlite3Parser() / Lemon
逻辑计划AST表达式分解、WHERE 子句拆分、等价类推导whereLoopInit()、whereSplit()
物理计划逻辑计划 + 索引信息执行计划 (VDBE 字节码)whereBestPath()、codeOneLoopStart()

第一步是解析器的工作（Lemon 生成的词法+语法分析），生成 AST。第二步是逻辑优化——把 WHERE 子句中的条件拆分成可以独立评估的片段，推导出等价类（例如 a=b AND b=c 意味着 a=c），确定哪些条件可以用索引来加速。第三步是真正"规划"的核心——枚举所有可能的访问路径（全表扫描、索引扫描、索引查找），计算每种方案的成本，选择最优的一个。

WHERE 子句分析

一切查询规划的核心是 where.c 中的 WhereLoop 结构体。它代表了访问一个表的一种可能方式：

/* WhereLoop 结构体 — 表示一种可能的表访问路径 */
/* 文件: src/where.c (简化版) */

struct WhereLoop {
    Bitmask prereq;              /* 这个 loop 依赖哪些前置表（用于 JOIN） */
    Bitmask maskSelf;            /* 这个 loop 对应的表 */
    WhereLoop **pNextLoop;       /* 链表中的下一个候选 loop */

    union {
        struct {
            int iIdxCur;         /* 索引游标号 */
            int iIndexCur;       /* 表游标号 */
            int nEq;             /* 等值条件的数量 */
            int nBtm;            /* 下界条件的数量 */
            int nTop;            /* 上界条件的数量 */
            int nSkip;           /* 可跳过的前缀列数 */
            Logest rRun;         /* 预估的运行成本 */
            Logest nOut;         /* 预估的输出行数 */
        } btree;                 /* B-Tree 索引访问路径 */
        struct {
            /* 全表扫描路径 */
            Logest rRun;         /* 运行成本 = 表页数 */
            Logest nOut;         /* 输出行数 = 表行数 */
        } fullScan;
    } u;

    Logest rSetup;               /* 建立这个 loop 的固定成本 */
    WhereLoop *pNextLoop;        /* 链表中的下一个项 */
    u8 wsFlags;                  /* 标志位 (索引类型、排序方式等) */
    u8 nLSlot;                   /* 可用插槽数 */
    u8 nLTerm;                   /* 实际使用的条件数 */
    WhereTerm *aLTerm[1];        /* 使用的 WHERE 条件数组 */
};

这个结构体是整个查询规划器的核心数据结构。对于每个表，SQLite 会构建多个 WhereLoop 候选者——一个代表全表扫描，若干个代表使用不同索引的扫描。然后规划器从这些候选者中挑出成本最低的那个。

WHERE 子句的分析过程大致如下：

  1. 拆分 — 用 whereSplit() 把 WHERE 子句在 AND 处拆分成单个条件
  2. 分类 — 对每个条件，判断它能不能用索引（sARGability，即 Search ARGument ABILITY）
  3. 绑定 — 将可用的条件绑定到对应的索引列上（whereLoopAddBtree()）
  4. 排序 — 判断索引能否满足 ORDER BY 子句，避免额外的排序操作
💡 sARGable 这个词是数据库领域的术语，意思是"可以作为搜索参数使用"。一个 WHERE 条件如果是 sARGable 的，意味着它可以直接被索引用于定位数据。例如 col = 42 是 sARGable 的（索引可以直接跳到 col=42 的位置），但 func(col) = 42 不是（因为索引存储的是 col 的原始值，不是 func(col) 的结果）。这是为什么在 SQL 中尽量不要对索引列使用函数的原因——函数会破坏索引的可搜索性。

Cost 模型

SQLite 的成本模型没有 PostgreSQL 那么复杂（不维护表的统计信息直方图），但它有一套有效的估算方法。成本以"处理一条记录的成本单位"来度量：

/* 成本估算的核心公式 */
/* 文件: src/where.c (whereLoopAddBtree() 中的计算逻辑) */

/* 1. 全表扫描的成本 = 表中的页数 */
pNew->u.btree.rRun = estTablePages(pTab);

/* 2. 索引查找的成本 = 索引树深度 + (等值匹配数 * 每行I/O成本) */
/* 索引树深度估算: 对于 N 行记录的 B-Tree, 深度 ≈ log_pagesize(N) */
int treeDepth = estBTreeDepth(pTab->nRow);
pNew->u.btree.rRun = treeDepth;

/* 3. 索引扫描 (范围查询) 的成本 */
/* 对于 col > 100 AND col < 200 这样的范围查询 */
/* 成本 = log(N) + 估计范围内的行数 * 0.1 (每个索引项成本) */
pNew->u.btree.rRun = treeDepth + (estimatedRangeRows * 0.1);

/* 4. 输出行数估算 (用于 JOIN 顺序决策) */
/* 如果使用了等值条件: nOut ≈ nRow / 索引区分度 */
/* 如果没有等值条件: nOut ≈ nRow * 0.1 (启发式假设 10% 的选择率) */
if (nEq > 0){
    /* 等值条件通常只匹配少数行 */
    pNew->nOut = estimateOutputRows(pTab, pIdx, nEq);
} else {
    /* 范围或不带条件: 保守估计 10% */
    pNew->nOut = pTab->nRowEstimate / 10;
}

这里的估算精度虽然不如大型数据库，但在绝大多数场景下已经足够。SQLite 团队做过一个有趣的统计：在数千个真实世界的查询中，SQLite 的成本模型选出的执行计划与"理论最优"计划相差超过 2 倍的情况不到 5%。这是因为 OLTP 场景下的查询通常比较简单（1-3 个表的 JOIN 加上简单的 WHERE 条件），即便估算有误差，也基本不影响最终计划的质量。

估算项方法精度
表行数 (nRowEstimate)从 B-Tree 根页估算，或在 ANALYZE 后直接读取 sqlite_stat1良好
索引区分度ANALYZE 收集的 sqlite_stat4 采样数据较好 (有 ANALYZE) / 一般 (无)
选择率 (范围查询)默认 10%，有直方图数据时更精确一般
B-Tree 深度基于页大小和行大小估算良好

值得注意的是，ANALYZE 命令对 SQLite 的查询规划质量有显著影响。运行 ANALYZE 后，SQLite 会创建 sqlite_stat1、sqlite_stat4 等内部表，存储索引的统计信息。这能让规划器做出更准确的选择率估算。如果你发现某些查询的执行计划不是最优的，第一步就是运行 ANALYZE，然后看 EXPLAIN QUERY PLAN 的输出有没有变化。

连接顺序优化

对于多表 JOIN，连接顺序的选择是最影响性能的因素。SQLite 使用了一种叫做 "N 个最优" 的启发式算法：

/* 连接顺序选择算法 (简化伪代码) */
/* 文件: src/where.c — whereBestPath() */

/* 算法: "N 个最优" (N Best) — 不是穷举所有排列！ */
/* 对 k 个表的 JOIN，最多保留 N 个候选路径 */

#define N_BEST 12  /* SQLite 最多保留 12 个候选路径 */

void whereBestPath(WhereInfo *pWInfo, int nTable, WherePath **ppBest){
    WherePath *apBest[N_BEST];  /* 当前最优的前 N 个路径 */
    int nBest;                   /* 当前候选路径数量 */
    int iTable;

    /* 逐层构建: 每次添加一个表 */
    for (iTable = 0; iTable < nTable; iTable++){
        /* 对当前每个候选路径，尝试添加下一个表 */
        for (each candidate path pPath in apBest){
            for (each unjoined table T){
                /* 构建新的路径: 在 pPath 后追加表 T */
                newPath = extendPath(pPath, T);

                /* 计算新路径的成本 */
                newPath->cost = pPath->cost + bestLoopForTable(T)->cost;

                /* 插入到最佳列表中，保留前 N_BEST 个 */
                addToBestList(newPath, apBest, &nBest);
            }
        }
    }

    /* 返回最佳路径 */
    *ppBest = apBest[0];
}

这个算法的精妙之处在于它是 O(N_BEST * nTable^2) 的，而不是穷举的 O(nTable!)。当 N_BEST=12、表数量为 10 时，穷举需要 360 万种可能，而 SQLite 的算法只需要评估约 1200 个路径。这种"近似最优"的取舍在嵌入式场景下完全合理。

在 3.53.0 版本中，SQLite 引入了 星型 JOIN (Star Join) 改进。星型 JOIN 是数据仓库中常见的查询模式：一个大的事实表 (fact table) 与多个小的维度表 (dimension table) 做 JOIN，事实表包含外键引用维度表。优化器现在能识别这种模式，优先扫描事实表，然后对每个维度表做索引查找，而不是反过来。这个优化在某些数据仓库查询中带来了 10 倍以上的性能提升。

EXISTS-to-JOIN 优化

SQLite 的查询规划器还包含一系列基于规则的优化。其中最实用的一个就是 EXISTS-to-JOIN 转换：

/* EXISTS 子查询 → JOIN 转换 */
/* 优化前 */
SELECT * FROM orders o
WHERE EXISTS (
    SELECT 1 FROM customers c
    WHERE c.id = o.customer_id AND c.status = 'active'
);

/* 优化后 (规划器内部转换) */
SELECT o.* FROM orders o
INNER JOIN customers c ON c.id = o.customer_id
WHERE c.status = 'active';

这个转换的意义在于：子查询（EXISTS）通常会被独立执行，即对 orders 表的每一行都执行一次子查询。而转换为 JOIN 后，规划器可以对 orders 和 customers 的连接顺序做全局优化。对于数据量大的表，这个转换通常能带来数量级的性能提升。

3.53.0 版本对这个优化做了显著增强。以前只有"简单的 EXISTS 子查询"（只包含一个表且没有聚合函数）能被转换。现在范围扩展到了：包含 JOIN 的子查询、带有子查询的 IN 子句、以及某些带有聚合函数的子查询（如果能证明在连接后等价）。

omit-noop-join 优化

这个优化在 SQLite 社区中不太为人所知，但它在实践中非常有用。考虑以下查询：

-- 查询所有有订单的客户
SELECT c.* FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE c.status = 'active';

在这个查询中，orders 表只被用来做 LEFT JOIN，但 SELECT 子句中并没有引用 orders 的任何列。而且由于 WHERE 子句只过滤 customers 表，orders 表的存在"不影响输出结果"——所有 customer 的行都保留（LEFT JOIN 的语义）。

SQLite 的 omit-noop-join 优化能够检测到这种"不影响输出的 JOIN"，并直接跳过这个 JOIN——不扫描 orders 表。优化器判断的逻辑是：

  • 如果 JOIN 表没有在 SELECT、WHERE、ORDER BY、GROUP BY 中被引用
  • 并且没有任何索引条件绑定到 JOIN 表上
  • 那么这个 JOIN 可以安全地省略

这个优化在 ORM 生成的查询中特别常见——ORM 经常在 SELECT 中不加选择地 JOIN 关联表，即使上层代码根本不需要那些表的数据。SQLite 默默地帮你优化掉了。

⚠️ 注意： omit-noop-join 优化不会移除有副作用的 JOIN。例如，如果 JOIN 表上有 INSTEAD OF 触发器，或者连接条件涉及用户自定义函数（UDF）且有副作用，优化器会保守地保留 JOIN。安全永远是第一位的。

实战：用 .wheretrace 观察查询规划器决策过程

理论讲完了，我们来点实战。SQLite 的 CLI 工具提供了一个非常强大的调试命令——.wheretrace，可以打印出查询规划器在决策过程中的所有中间信息。

-- 在 sqlite3 CLI 中启用 wheretrace
sqlite> .wheretrace on

sqlite> CREATE TABLE t1(a, b, c);
sqlite> CREATE INDEX t1_a ON t1(a);
sqlite> CREATE INDEX t1_bc ON t1(b, c);
sqlite> INSERT INTO t1 VALUES ...
sqlite> ANALYZE;

sqlite> EXPLAIN QUERY PLAN SELECT * FROM t1 WHERE a = 10 AND b > 5;
QUERY PLAN
|--SEARCH t1 USING INDEX t1_a (a=?)

-- 实际内部输出 (.wheretrace):
-- WhereLoop: 表=t1, 方法=全表扫描, 成本=100.00, 输出=100行
-- WhereLoop: 表=t1, 方法=索引t1_a, 等值条件=1, 成本=2.50, 输出=1行
-- WhereLoop: 表=t1, 方法=索引t1_bc, 等值条件=0, 范围条件=1, 成本=15.00, 输出=10行
-- 选择索引 t1_a: 成本最低 (2.50 < 15.00 < 100.00)

.wheretrace 的输出清晰地展示了三个候选方案的评估过程：

  • 全表扫描：成本 100（对应 100 页的读取）
  • 索引 t1_a 查找（等值匹配 a=10）：成本 2.5，预计只输出 1 行
  • 索引 t1_bc 范围扫描（b > 5）：成本 15，预计输出 10 行

最终规划器选择了 t1_a，因为它的成本最低。

除了 .wheretrace，还有几个有用的调试命令：

命令作用
.wheretrace on打印查询规划器的候选路径评估过程
.eqptrace on打印表达式的代码生成过程
.vdbtrace on打印 VDBE 运行时执行的每条指令
EXPLAIN QUERY PLAN ...展示最终的查询计划（推荐日常使用）
EXPLAIN ...展示 VDBE 字节码（调试深层次问题）
💡 在调试慢查询时，我的标准流程是：
1. 先 EXPLAIN QUERY PLAN 看总体计划
2. 如果计划不合理，检查是否缺少索引或统计信息过时（运行 ANALYZE）
3. 如果仍然不满意，启用 .wheretrace 查看规划器的候选路径和成本估算
4. 找到成本估算的偏差（例如规划器认为某个索引扫描成本 10，实际上要扫描 1000 行），然后考虑创建更好的索引或改写查询
