第九章：处理器体系结构——流水线

上一章我们设计了 Y86-64 的 SEQ 处理器——每条指令依次经过 Fetch、Decode、Execute、Memory、Write Back、PC Update 六个阶段。它功能正确，但有一个致命问题：在任何时刻，6 个阶段中只有 1 个在做有用的事，其他 5 个都在睡觉。这好比一条只有一个人工作的流水线——其他工位空着，一个人在整条线上来回跑。有没有办法让所有阶段同时工作？有，这就是流水线（Pipelining）。

流水线是处理器设计中最伟大的思想之一。它的本质就一句话：把一条指令的执行过程切分成若干阶段，让不同指令的不同阶段在时间上重叠执行。听起来简单，但实际实现时要面对数据冒险、控制冒险、异常处理等一系列棘手的问题。这一章我们不仅要学流水线的原理，还要理解这些冒险是如何被克服的。这不仅仅是体系结构的知识，它直接影响到你写程序时如何布局代码、如何分支、如何组织循环——因为当你知道了 CPU 内部在做什么，你写的每一行代码都会更有"针对性"。

9.1 流水线通用原理

理解流水线最好的方式，是类比生活中的场景——洗衣房。

想象你有一个洗衣房，需要完成 4 个人的洗衣任务，每个任务分三个步骤：洗衣 → 烘干 → 折叠。每步需要 30 分钟。如果没有流水线，你需要依次完成每个人的三个步骤：第一个人的洗衣→烘干→折叠（90 分钟），再开始第二个人的（又一个 90 分钟）……4 个人总共需要 360 分钟。

但如果用流水线方式：当第一个人的衣服进入烘干时，你立即开始洗第二个人的衣服；当第一个人的衣服折叠时，第二个人进入烘干，第三个人开始洗衣……这样，从第三个人开始，每 30 分钟就能完成一个人的任务。虽然第一个人的衣服要 90 分钟才能拿到（延迟不变），但完成 4 个人的衣服只需要 180 分钟——吞吐量翻了一倍。

处理器流水线的逻辑完全一样：

指标 非流水线（SEQ） 流水线（PIPE）
每指令周期数（CPI） 6（理想情况） 1（理想情况）
单条指令延迟 6 个周期 6 个周期（不变）
吞吐量（指令/周期） 1/6 ≈ 0.167 1（理想情况）
加速比（理想） 1× 6×

流水线通过插入流水线寄存器（Pipeline Registers）来分隔各个阶段。在 Y86-64 的流水线实现 PIPE 中，我们有 F、D、E、M、W 五组流水线寄存器——分别对应 Fetch→Decode、Decode→Execute、Execute→Memory、Memory→Write Back 的边界。每个周期结束后，当前阶段的结果被锁存到流水线寄存器中，下个周期开始时，下一阶段从这些寄存器读取数据。

关键认识：流水线并不减少单条指令的执行时间（延迟反而可能略增，因为加入了流水线寄存器的开销），它提高了吞吐量。这是处理器设计中最重要的认知之一——在绝大多数场景下，吞吐量比单次延迟更影响系统性能。这就是为什么现代 CPU 的流水线深度达到 14-20 级（如 Intel Core 系列），而不会因为"单条指令变慢"而放弃流水线。

9.2 数据冒险与转发

流水线面临的最大敌人是冒险（Hazard）——下一条指令依赖上一条指令的结果，而上一条指令的结果还没写回寄存器。来看一个具体的例子：

# Y86-64 流水线中的数据冒险
irmovq $100, %rbx       # I1: 把 100 加载到 %rbx
addq %rbx, %rcx         # I2: 需要 %rbx 的值！但 I1 还没写回！
rmmovq %rcx, 0(%rbx)    # I3: 又需要 %rbx 和 %rcx!

三种冒险类型：

  • RAW（Read After Write）：最普通的冒险，写后才读。上面的例子就是 RAW。
  • WAR（Write After Read）：读后才写。在经典 5 级流水线中通常不会发生（所有读操作在 Decode 阶段完成，写操作在 Write Back 阶段）。
  • WAW（Write After Write）：两次写入同一寄存器。在 Y86-64 中也不会发生（只有一个写端口）。

解决 RAW 冒险有两种方法：

方法一：插入气泡（Stalling / Bubble）

让流水线暂停（stall），直到需要的值被写回。最笨但也最简单的办法。在 I1 的 Write Back 之前，I2 的 Decode 阶段被阻塞。代价是每解决一次冒险就损失几个周期。

方法二：转发（Forwarding / Bypassing）

更聪明的办法——结果还没写回寄存器？没关系，直接从流水线寄存器中"偷"出来。在 SEQ 的流水线版本 PIPE 中，设置了多条转发路径：

  • e → E：从 Execute 阶段的输出转发到 Execute 阶段的输入（自己算出的结果马上给自己用）
  • M → e：从 Memory 阶段的输出转发到 Execute 阶段的 ALU 输入
  • W → m：从 Write Back 阶段的输出转发到 Memory 阶段的数据写回路径
  • W → E：从 Write Back 阶段转发到 Execute 阶段

转发的工作原理：想象一下，I2 的 Execute 阶段需要 %rbx 的值，而此时 I1 的结果刚刚在 Execute 阶段算出来（放在流水线寄存器 E 中）。转发逻辑发现："I2 需要的寄存器正是 I1 要写入的寄存器"，于是直接让 I2 的 ALU 输入来自 I1 的输出，而不是从寄存器文件读取。这样一来，I2 不需要等待任何周期就能拿到正确的值。在实际的 PIPE 设计中，绝大多数 RAW 冒险都能通过转发解决——只有 load-use 冒险（从内存读取的值紧接着被下一条指令使用）还需要插入一个气泡。

9.3 控制冒险与分支预测

比数据冒险更棘手的是控制冒险——当遇到分支指令（条件跳转）时，流水线不知道该取哪条指令。在 SEQ 中这不是问题：Execute 阶段已经算出了条件码，确定了是否跳转。但在流水线中，Fetch 阶段取指时，分支结果还没算出来！

简单的做法：让流水线暂停，等分支结果出来再继续。但这会损失好几个周期。对于频繁出现分支的程序来说，这个代价太大了。

更好的做法是分支预测（Branch Prediction）——猜测分支结果，然后投机执行。猜对了，运行如飞；猜错了，要冲刷（flush）流水线，恢复正确状态。现代 CPU 的分支预测器准确率可达 95% 以上。

Y86-64 的 PIPE 实现中，采用了最简单的分支预测策略：

  • 条件跳转：预测为"跳转"（预测跳转）——装配了 BTB（Branch Target Buffer）来缓存跳转目标地址
  • 无条件跳转和 call：必然跳转，无需预测
  • ret：最让流水线头疼的指令，因为返回地址在栈里，要到 Memory 阶段才能读出来。PIPE 采用"等待直到 ret 到达 Execute 阶段"的策略

更先进的预测器（真实 CPU 中的）：

  • 静态预测：向后跳转预测为"跳转"（适合循环），向前跳转预测为"不跳转"
  • 动态预测：基于历史记录（2-bit 饱和计数器、两级自适应预测器等）

Spectre 漏洞的体系结构根源：2018 年曝光的 Spectre 漏洞震惊了业界，它的根源正是分支预测和投机执行。攻击者先让分支预测器"训练"到总是走某条路径，然后诱导 CPU 在错误的路径上投机执行——访问本不该访问的内存。投机执行的结果虽然最终被回滚（因为预测错误），但 CPU 的缓存状态已经被改变。通过测量缓存访问时间，攻击者就能推断出被访问的数据内容。这个漏洞告诉我们：体系结构的每一个"聪明设计"，都可能在安全层面留下意想不到的后门。在之后的章节中我们还会详细讨论这个安全问题。

9.4 流水线中的异常处理

异常是流水线设计中最容易被忽视的难点。考虑这样一个场景：一条指令发生了除法错误，而此时它后面的 3 条指令已经进入了流水线的不同阶段。我们怎么处理？

基本原则是"最老指令优先"——如果多条指令同时产生异常，只处理最早进入流水线的那条。因为后面指令的执行都是在前面指令的"投机"下进行的，前面的指令错了，后面的指令也就没有意义了。

在 Y86-64 的 PIPE 中，异常在以下阶段检测：

  • Fetch 阶段：检测到非法指令码或指令地址越界 → 用这个错误填充流水线
  • Execute 阶段：halt 指令在这里引发异常
  • Memory 阶段：内存访问越界
  • Write Back 阶段：不会产生新异常，只传递前面阶段的异常状态

每个流水线寄存器都有一个额外的字段来携带异常状态。当异常发生时，流水线会：

  1. 禁止修改寄存器文件和内存（防止"脏数据"）
  2. 在异常指令之后的阶段插入气泡（冲刷后续指令）
  3. 在异常指令到达 Write Back 阶段时，将异常信息和 PC 保存下来，交给操作系统处理

一个有趣的细节：如果异常指令在执行时，前面的指令也改写了条件码——这会不会有问题？不会，因为在 SEQ/PIPE 设计中，条件码在 Execute 阶段被修改，而异常指令的"异常状态"也在流水线寄存器中传递。如果最终异常被触发，已经在 Execute 阶段被修改的条件码会被丢弃（因为处理器的状态不会更新到寄存器文件和内存，但条件码在某些实现中可能已被修改——这是 Y86-64 PIPE 设计中的一个实际挑战）。

设计启示：流水线异常处理告诉我们一个深刻的道理——错误必须在正确的上下文中被处理。在流水线中，一条指令的错误不能污染其他指令的执行结果，而异常信息必须跟着指令走完整个流水线，直到它成为"最老"的那条指令才能被处理。这种设计思路对于你写多阶段数据处理过程也有启发——永远不要在中间阶段就丢弃上下文信息，因为你不知道后续阶段是否还需要它来做决策。

这一章从流水线的洗衣房类比开始，我们深入了解了数据冒险的三种类型及其对策（转发和气泡）、控制冒险的分支预测策略（从简单的总是跳转到复杂的动态预测器），以及异常处理的最老指令优先原则。流水线是计算机体系结构从"功能正确"走向"高性能"的必经之路。理解了流水线，你就理解了你写的程序在 CPU 中实际运行的节拍。下一章，我们将从处理器设计转向一个更贴近日常的话题——如何优化你的程序性能。
