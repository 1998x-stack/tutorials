第19章 网络协议栈

大家好，我是你们的讲师。你有没有想过，当你在浏览器里输入 https://www.google.com 并按回车，到页面出现在屏幕上，这中间到底发生了什么？DNS查询、TCP三次握手、TLS协商、HTTP请求、路由转发、ARP解析……这一连串看似复杂的步骤，在你按下的那一瞬间就全部完成了。而承载这一切的，正是操作系统的网络协议栈（Network Stack）。

网络协议栈是操作系统里"存在感最强"的子系统之一。你几乎感觉不到它的存在——直到某个深夜，你发现服务器上 ss -tuln 显示几十万个 TIME_WAIT 连接，或者生产环境突然出现诡异的"Connection refused"。这时候你就知道，不懂网络协议栈是要吃大亏的。本章我们沿着数据包穿越内核的完整路径，把网络协议栈的各个关键环节一一拆解。

先分享一个真实的踩坑经历。有次线上告警：某台Nginx反向代理服务器的响应延迟从2ms飙到了2秒。分析后发现这台机器的 /proc/net/netstat 中 ListenOverflows 计数器在疯狂增长——Accept队列满了，新连接被内核直接丢弃。调整 net.core.somaxconn 和 tcp_abort_on_overflow 后立即恢复。如果不懂内核TCP连接建立的内部机制，面对这种问题你就是两眼一抹黑。

从socket()到网卡：数据包的完整旅程

一条数据从用户态应用出发，到最终从网线（或无线）上飞出去，要穿越七个主要关卡。我们从自上而下的视角来梳理：

第一关：应用层

用户程序调用 send(sockfd, buf, len, flags)。这个调用看起来人畜无害，实际上它从用户态陷入内核态。应用层不关心底层是TCP还是UDP——它已经把数据交给socket抽象了。

第二关：Socket层

内核的socket通用层（net/socket.c）根据socket的协议族和类型，找到对应的协议操作函数表。这里用到了面向对象的设计思想——struct proto_ops 中挂着 sendmsg、recvmsg 等操作指针。对于TCP socket，它会调用 tcp_sendmsg。

第三关：传输层

TCP协议栈在这里做分段（MSS拆分）、添加TCP头部（源端口、目的端口、seq序号、确认号、窗口大小、校验和）。然后把生成的sk_buff塞入发送队列。TCP拥塞控制算法（CUBIC / BBR）也会在这一层做出速率决策——要不要降速、要不要快重传。

第四关：网络层

ip_output 为每个数据包添加IP头部，然后查路由表（FIB）决定数据包从哪个网卡出去。如果需要NAT，Netfilter钩子也会在这里介入改写地址。如果数据包太大超过MTU，IP层还会做分片（不过现代网络通常靠PMTUD避免分片）。

第五关：链路层

dev_queue_xmit 将数据包交给链路层。这里要查ARP缓存来填充目标MAC地址。同时处理VLAN标签（802.1Q）、QoS优先级映射。然后数据包被放入网卡的发送环（Tx Ring）。

第六关：驱动层

网卡驱动程序从Tx Ring取走描述符，通过DMA将数据从主存传输到网卡内部的FIFO缓冲区。然后网卡自动将数字信号调制成模拟信号从网线发出去。这个过程对CPU几乎透明——一次DMA可以传输数千个数据包而CPU完全不用参与。

关键数据结构：内核网络协议栈的核心数据结构是 struct sk_buff（socket buffer）。它贯穿整个协议栈的每一层，头部预留了足够的空间（head 到 data 之间），每经过一层只需调整指针位置，避免了内存拷贝。这正是内核网络子系统的性能秘诀之一。

// 网络层: [IP header][TCP header][HTTP header][payload]

三次握手的内核实现

TCP三次握手是我们面试中被问烂的话题，但面试很少问你：这个握手在内核里究竟是怎么实现的？理解这一点，你才能解释为什么高并发场景下会出现连接失败、SYN Flood如何防御、以及各种TCP内核参数的真正含义。

两个队列：SYN队列与Accept队列

服务端在调用 listen() 后，内核为这个监听socket创建两个队列：

  • SYN队列（半连接队列）：收到SYN包后，连接进入 SYN_RECV 状态，挂在这个队列上。队列大小由 net.ipv4.tcp_max_syn_backlog 控制。
  • Accept队列（全连接队列）：三次握手完成后（收到ACK），连接进入 ESTABLISHED 状态，转移到此队列。应用调用 accept() 从此取走连接。队列最大长度由 listen(fd, backlog) 的backlog参数和 net.core.somaxconn 中较小的值决定。
队列状态参数控制满了会怎样
SYN队列（半连接）SYN_RECVtcp_max_syn_backlog丢SYN包（可启用syncookies）
Accept队列（全连接）ESTABLISHEDbacklog / somaxconn发送RST或自动丢弃ACK（看 tcp_abort_on_overflow）

踩坑实录：很多应用（如Redis、Nginx）内部有自己的事件循环，listen() 的backlog设得很大（比如512或者1024），但很多人忘了检查 net.core.somaxconn 的默认值（噫吁嚱，大多数Linux发行版默认是128！）。结果就是backlog写了1024，实际生效只有128。高并发下Accept队列很快爆满，新连接直接RST断开——上层应用看到的错误就是"Connection refused"。

SYN Cookie防御SYN Flood

SYN Flood攻击的思路极其简单：攻击者伪造源IP发送大量SYN包，服务端回复SYN+ACK后等待客户端应答——但攻击者永远不回ACK。于是SYN队列被占满，正常用户的连接请求被丢弃。

内核的防御武器是SYN Cookie。开启 net.ipv4.tcp_syncookies = 1 后（多数发行版默认开启），内核在SYN队列满时不真正分配连接资源，而是将连接信息（MSS等）编码成一个cookie作为初始seq number发回去。等客户端的ACK回来时，内核从ACK中的seq-1解码出cookie，验证合法性后直接创建连接。这样一来，即使SYN Flood攻击再猛烈，服务端也无需为每个伪造SYN维护状态。

# 查看当前连接队列状态
$ ss -tnlp | head -20

# 查看SYN队列溢出统计
$ nstat -az | grep -E "Syncookies|ListenOverflows|ListenDrops"

# 内核参数调优
# 增大Accept队列上限
$ sysctl -w net.core.somaxconn=1024

# 启用SYN Cookie（数据完整性有轻微代价）
$ sysctl -w net.ipv4.tcp_syncookies=1

# 加快TIME_WAIT回收（谨慎使用）
$ sysctl -w net.ipv4.tcp_tw_reuse=1

Netfilter五链四表

Netfilter是Linux内核内置的数据包过滤和修改框架。你在用户态用 iptables 或 nftables 配置的规则，最终都会编译为内核Netfilter模块的hook处理函数。

五个钩子点（五链）

数据包在内核协议栈的不同阶段会触发不同的Netfilter钩子：

  • NF_INET_PRE_ROUTING（PREROUTING）：数据包进入网络层后、路由决策之前。适合做DNAT（目的地址转换）。
  • NF_INET_LOCAL_IN（INPUT）：目的地址为本机的数据包，在路由决策之后触发。
  • NF_INET_FORWARD（FORWARD）：非本机目的的数据包（路由器场景），在路由决策之后触发。
  • NF_INET_LOCAL_OUT（OUTPUT）：本机发出的数据包，在其路由之前触发。
  • NF_INET_POST_ROUTING（POSTROUTING）：所有即将发出的数据包，在路由之后、交给链路层之前触发。适合做SNAT（源地址转换）。

四张规则表

用户可以通过iptables/nftables将规则注册到不同的表中，每个表关注不同维度的处理：

  • raw表：优先级最高，用于关闭连接跟踪（NOTRACK），如高速DNS服务器无需走conntrack。
  • mangle表：修改IP包的特定字段（TOS/DSCP/TTL），用于QoS标记等场景。
  • nat表：地址转换——DNAT（端口映射）和SNAT（上网代理）。
  • filter表：最常见的表，防火墙规则（ACCEPT/DROP/REJECT）都在这里。

关键认知：Netfilter的架构是一张"钩子×表"的二维矩阵。每个钩子点可以挂载多个表，表之间有优先级排序。理解这个矩阵结构，你就明白了为什么 iptables -t nat -A PREROUTING 能实现端口转发，而 -A FORWARD 则不能。也明白了为什么nftables最终取代iptables——前者统一了用户态语法，内核机制并无本质变化。

#
#          [网卡]
#            |
#          PREROUTING (raw → mangle → nat)
#            |
#         [路由决策]
#          /        \
#     目的为本机    目的为转发
#        |             |
#      INPUT        FORWARD
#   (mangle→filter)  (mangle→filter)
#        |             |
#     [本地进程]     POSTROUTING
#                      (mangle→nat)
#                        |
#                     [网卡发送]

XDP：网卡驱动层的可编程数据路径

传统的iptables/nftables处理数据包需要包经过整个协议栈——从网卡→驱动→链路层→网络层→Netfilter钩子。这意味着大量的CPU时间花在了协议栈处理上，对于DDoS防御、负载均衡这种需要超高吞吐的场景，太慢了。

XDP（eXpress Data Path）的思路截然不同：它抢在网卡驱动把数据包送入协议栈之前——实际上是网卡驱动刚刚从DMA缓冲区拿到sk_buff之后、还没有提交给协议栈之前——就让eBPF程序直接处理这个数据包。处理结果可以是 XDP_DROP（丢弃）、XDP_PASS（放行入协议栈）、XDP_TX（从同一网卡原路发回）、或者 XDP_REDIRECT（转发到另一个网卡或用户态）。

这意味着：对于DDoS防御，你可以在驱动层面直接丢弃攻击流量，CPU几乎零开销。Cloudflare的L4 DDoS防御就是基于XDP实现的——每台服务器可以线速处理数百万PPS的攻击数据包，CPU占用率仅个位数。

性能对比：用iptables DROP一条规则，数据包要经过完整协议栈处理，包长路径约2000条指令。而XDP_DROP在驱动层就拦截了，只需不到100条指令，性能提升10到20倍。XDP要求网卡驱动支持（主流网卡如Mellanox、Intel i40e/ice、Broadcom BNXT都已支持），且需要eBPF JIT编译器的支持。

// XDP程序的极简示例（用C语言编写）
#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>

SEC("xdp")
int drop_all_packets(struct xdp_md *ctx) {
    // 拒绝所有数据包——极简DDoS防御
    return XDP_DROP;
}

// 实际场景：只丢弃特定源IP的攻击包
SEC("xdp")
int block_bad_ip(struct xdp_md *ctx) {
    void *data = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;

    struct ethhdr *eth = data;
    if ((void *)(eth + 1) > data_end) return XDP_PASS;

    // 解析IP头并查找黑名单...
    // 若命中则 return XDP_DROP;
    // 否则 return XDP_PASS;
}

内核旁路与DPDK

XDP是在"不绕过内核"的前提下极致优化。但有些人说：我追求极致性能，内核本身就是一个瓶颈——中断处理、系统调用、上下文切换、锁竞争，所有这些内核开销我都不要。这就是内核旁路（Kernel Bypass）的出发点。

为什么需要绕过内核？

标准内核协议栈的痛点非常明确：每个数据包可能触发一次中断（NAPI虽然合并中断，但开销仍在）、每次收发包都要系统调用、锁竞争在多核场景下严重、协议栈处理路径太长。结果就是：10Gbps网卡很容易被内核协议栈吃满CPU，而真正处理业务逻辑的时间所剩无几。

DPDK：用户态网络栈

DPDK（Data Plane Development Kit）的核心思想是：用 UIO（Userspace I/O）或 VFIO 框架把网卡的控制权交给用户态。用户态程序直接轮询（Polling）网卡的接收和发送队列，没有任何中断和系统调用开销。数据包从网卡DMA到用户态预先分配的巨页（Hugepage）内存上，应用直接用DPDK的API解析数据包。

代价呢？应用层需要自己实现TCP/IP协议栈——DPDK本身只提供数据面转发能力，不提供完整的TCP协议栈。要么用第三方库如mTCP、F-Stack、Seastar，要么干脆不用TCP只做UDP转发。DPDK也意味着独占一个或多个CPU核心（轮询不断），这对通用场景不友好。

io_uring zcrx：折中方案

DPKD太极端了——它把内核完全跳过了。有没有一种折中方案？io_uring + ZC_RX（Zero Copy Receive）提供了一个有趣的第三条道路。io_uring的 IORING_OP_PROVIDE_BUFFERS 让应用预注册缓冲区，网卡DMA直接将数据包写入预注册的缓冲区——省去了一次内存拷贝。用户态通过io_uring的completion queue拿到数据包引用，不用经过系统调用。这本质上是"hybrid bypass"：数据仍然经过内核协议栈，但消除了大部分的数据拷贝和系统调用开销。

路线对比：传统内核协议栈 = 灵活通用 + 性能一般。XDP = 内核框架内加速，不改协议栈体系。DPDK = 彻底绕过内核，性能极致但代价大。io_uring zcrx = 在现有框架内"取巧"，适合不想搞DPDK那么重、但又不满意原生TCP性能的场景。技术选型没有银弹，根据吞吐量需求和开发成本做权衡。

网络协议栈是操作系统中最复杂、最优雅的子系统之一。从应用层的socket抽象到硬件层的DMA传输，每一层都体现了分层抽象的设计哲学。这一章我们走完了数据包的完整旅程，理解了三次握手的内核实现、Netfilter的钩子框架、XDP的可编程数据路径、以及DPDK和io_uring zcrx两种绕过内核的加速方案。

下一章，我们将走进操作系统可观测性（Observability）的核心——eBPF。如果说网络协议栈关注的是"数据怎么走"，那么eBPF关心的就是"系统在干什么"。有了eBPF，我们不需要修改内核代码，不需要重启机器，就能在内核的任意位置注入观测代码。这将是你掌握操作系统调试与性能分析的又一大利器。
