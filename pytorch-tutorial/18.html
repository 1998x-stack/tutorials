第18章 可复现性与实验管理

大家好，我是你们的讲师。今天的话题可能不如模型结构或者训练技巧那么"酷"，但它的重要性绝对不容低估——可复现性（Reproducibility）与实验管理。

在深度学习的世界里，一个实验结果如果不能被复现，那它的价值就接近于零。更现实的问题是：你三个月前跑的那个 baseline 的 accuracy 是多少？你当时用的超参数是什么？数据集是哪个版本？如果你答不上来，那你在那个 baseline 之上"改进"的所有实验结果都是不可信的——因为你没法确认改进是真的来自你的新方法，还是仅仅因为随机种子不同。

这一章我会从最基础的随机种子设置讲起，然后讨论 deterministic 模式、实验配置管理、模型版本化和 checkpoint 命名策略。我会尽量给出可以直接用在项目中的模板代码。

18.1 设置所有随机种子

很多人以为设置种子就是跑一行 torch.manual_seed(42)。但实际上，"全"种子意味着你要对代码中所有可能引入随机性的环节都设置种子。下面这个函数几乎是我所有 PyTorch 项目的标准开头：

import torch
import numpy as np
import random

def set_all_seeds(seed: int):
    """为所有随机数生成器设置种子"""
    torch.manual_seed(seed)                    # PyTorch CPU
    torch.cuda.manual_seed_all(seed)           # PyTorch GPU（所有设备）
    np.random.seed(seed)                       # NumPy
    random.seed(seed)                          # Python 标准库
    # 注意：如果你使用 Python 3.2+，hash 的随机化也需要固定
    os.environ["PYTHONHASHSEED"] = str(seed)

# 使用
set_all_seeds(42)
你可能漏掉的一个：PYTHONHASHSEED。Python 3.2+ 在进程启动时会随机化字符串的哈希种子以防范哈希碰撞攻击。如果你的代码中使用了 set()、dict() 等基于哈希的数据结构（比如数据集中的类别映射），这个随机化会导致每次运行的数据读取顺序不同。设置 PYTHONHASHSEED 能消除这个不确定性。

注意：PYTHONHASHSEED 必须在进程启动时设置才生效，不能像其他种子那样在代码中间设置。通常的做法是在调用脚本前设置：PYTHONHASHSEED=42 python train.py，或者在脚本的最高处（在 import 任何东西之前）设置。

18.2 torch.use_deterministic_algorithms()

设置种子只能保证"随机数序列"相同，但不能保证所有的浮点运算都是确定性的。比如某些 CUDA 卷积实现使用了原子加法（atomic add），这会导致每次 forward 的微小差异——即使输入完全相同。

要强制 PyTorch 使用确定性的算法，可以使用：

torch.use_deterministic_algorithms(True)
torch.backends.cudnn.deterministic = True

第一个调用告诉 PyTorch 框架层的算子优先选择确定性实现。第二个调用告诉 cuDNN 选择确定性卷积算法。这样，相同输入 + 相同种子 = 完全相同的输出。

代价是性能下降：

配置 可复现性 训练速度 推荐场景
默认 不可保证 100%（baseline） 正式训练、大模型
+ 设置种子 大部分随机性消除 ~95% 日常实验
+ deterministic 完全确定 ~60%~80% 调试、论文复现
一个重要警告：并不是所有的 PyTorch 操作都有确定性实现。某些操作（如 torch.nn.functional.interpolate 的某些模式、torch.scatter_reduce 等）在启用 use_deterministic_algorithms(True) 时会抛出 RuntimeError。如果你的代码中包含这些操作，你可能不得不接受不确定性——或者寻找替代实现。

18.3 cudnn.deterministic vs cudnn.benchmark

这两个选项是有冲突的，不能同时开启：

# 选项 A：追求可复现
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False

# 选项 B：追求速度（默认推荐）
torch.backends.cudnn.benchmark = True
torch.backends.cudnn.deterministic = False

其实这两者并不完全互斥——你可以只设置 benchmark=False（不使用自动调优，使用默认卷积算法）而不设置 deterministic=True。这样虽然每次运行的结果可能不完全一致（因为默认卷积算法也不一定是确定性的），但至少不会有 benchmark 引入的算法切换差异。

最佳实践是：在你的训练脚本中提供一个命令行开关，让使用者可以在"可复现"和"高性能"之间自由切换：

import argparse

parser = argparse.ArgumentParser()
parser.add_argument("--seed", type=int, default=42)
parser.add_argument("--deterministic", action="store_true")
args = parser.parse_args()

if args.deterministic:
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
    torch.use_deterministic_algorithms(True)
    set_all_seeds(args.seed)
else:
    torch.backends.cudnn.benchmark = True
    set_all_seeds(args.seed)

18.4 Config Dict 模式：替代 argparse

随着实验数量的增加，用 argparse 管理超参数会变得越来越痛苦。每个实验有一套不同的参数，你完全记不住哪个实验用了什么参数。推荐使用 Config Dict 模式——用一个 Python dict 保存所有配置，序列化后随模型一起保存。

# config.py
config = {
    "exp_name": "baseline_resnet50",
    "seed": 42,
    "model": "resnet50",
    "batch_size": 256,
    "learning_rate": 1e-3,
    "weight_decay": 1e-4,
    "num_epochs": 100,
    "optimizer": "adam",
    "scheduler": "cosine",
    "data_augmentation": True,
    "mixed_precision": True,
    "compile": True,
    "deterministic": False,
    "dataset": "imagenet",
    "data_dir": "/data/imagenet",
    "num_workers": 8,
    "pin_memory": True,
}

# train.py
def train(config):
    print("Config:", config)
    set_all_seeds(config["seed"])
    model = create_model(config["model"])
    dataloader = create_dataloader(config)
    optimizer = create_optimizer(model, config)
    # ... 训练代码 ...

    # 训练完成后，将配置和模型一起保存
    torch.save({
        "model_state_dict": model.state_dict(),
        "optimizer_state_dict": optimizer.state_dict(),
        "config": config,                    # 保存完整配置！
        "final_accuracy": best_acc,
    }, f"{config['exp_name']}_final.pth")
YAML 也可以：很多团队喜欢用 YAML 文件来管理配置。PyTorch 社区中常见的做法是 config.yaml + yaml.safe_load() 来读取。两种方式都可以，关键原则是：配置必须与实验结果一起保存——不能只保存模型权重而不保存配置，否则这个模型权重基本就是废的。

18.5 模型版本化与 Checkpoint 命名

我不建议用 model_final.pth、model_final_v2.pth 这种命名方式。当你有 20 个实验的时候，这些名字完全无法帮助你区分。一个实用的命名策略是包含以下信息：

  • 实验名称 — 如 resnet50_baseline
  • 日期和时间 — 如 20260630_1430
  • git commit hash — 代码版本，精确到 7 位 hash
  • 关键指标 — 如 acc_76.3
import subprocess
from datetime import datetime

def get_git_hash():
    return subprocess.check_output(
        ["git", "rev-parse", "--short", "HEAD"]
    ).decode().strip()

def save_checkpoint(model, optimizer, config, epoch, metrics, path):
    timestamp = datetime.now().strftime("%Y%m%d_%H%M")
    git_hash = get_git_hash()
    acc = metrics.get("accuracy", 0)

    filename = (
        f"{config['exp_name']}_"
        f"{timestamp}_"
        f"{git_hash}_"
        f"epoch{epoch:03d}_"
        f"acc{acc:.1f}.pth"
    )

    checkpoint = {
        "epoch": epoch,
        "model_state_dict": model.state_dict(),
        "optimizer_state_dict": optimizer.state_dict(),
        "config": config,
        "metrics": metrics,
        "git_hash": git_hash,
        "timestamp": timestamp,
    }
    torch.save(checkpoint, f"{path}/{filename}")
    return filename

# 生成的文件名类似：
# resnet50_baseline_20260630_1430_a1b2c3d_epoch090_acc76.3.pth

这样的文件名看一眼就能知道是什么时候跑的、哪个代码版本、精度如何。配合 config 保存，你打开 checkpoint 文件就能拿到实验的完整配置。

18.6 可复现性检查清单

为了方便查阅，我整理了一份综合检查清单。每次开始一个重要的实验前，可以逐项对照：

项目 操作 重要性
PyTorch 种子 torch.manual_seed(seed) 必须
CUDA 种子 torch.cuda.manual_seed_all(seed) 必须
NumPy 种子 np.random.seed(seed) 必须
Python 随机种子 random.seed(seed) 必须
Python hash 种子 启动前设置 PYTHONHASHSEED 推荐
确定性的 CUDA 卷积 cudnn.deterministic = True 按需（论文复现时开启）
确定性算法 use_deterministic_algorithms(True) 按需（会降低速度）
关闭 benchmark cudnn.benchmark = False 按需（开启会更快但不确定）
DataLoader shuffle 种子 DataLoader(..., generator=torch.Generator().manual_seed(seed)) 推荐（确保数据顺序固定）
模型初始化种子 在 set_all_seeds 之后初始化模型 必须
配置记录 config dict 随 checkpoint 一起保存 必须
代码版本 git commit hash 记录在 checkpoint 中 强烈推荐
关于 "随机种子改 3%" 的真实故事： 我有一位朋友在 NeurIPS 2023 投稿前做了一组对比实验。新方法比 baseline 高了 3% 的 accuracy，这是一个非常性感的结果。他很兴奋地写完了论文。但提交前的最后一天，他决定再跑一次确认——结果新方法的 accuracy 降了 2%，baseline 反而高了 1%。两次结果差了 3%。 原因是什么？他第一次跑实验的时候，机器上正好有另一个进程占用了大量 CPU，导致 DataLoader 的 worker 产生了不同的数据打乱顺序——因为 DataLoader 的 shuffle 默认使用的是全局随机状态，而 worker 子进程的随机状态是不可控的。他以为设了 torch.manual_seed 就万事大吉了，但其实 DataLoader 需要单独设置 generator。 好消息是，他检查了一下发现实验可以复现——只是需要用完全相同的 CPU 负载条件。这显然不叫"可复现"。 解决方法是给 DataLoader 传入一个专属的 Generator： 
g = torch.Generator()
g.manual_seed(config["seed"] + epoch)  # 每个 epoch 不同的 shuffle
dataloader = DataLoader(dataset, shuffle=True, generator=g)
这件事告诉我们：只设一个 seed 远远不够，每个随机环节都要单独管理。

好了，这一章的内容就到这里。我们学习了如何正确地设置所有随机种子、如何使用 deterministic 模式、如何用 config dict 管理实验配置、以及如何为 checkpoint 命名。可复现性虽然不"性感"，但它是所有可靠研究的基石。没有可复现性，你的实验结果就是"周末哄自己开心"的数字游戏。
