第11章 平行机与批量调度

大家好，我是你们的讲师。前几章我们讨论的场景基本是"一台机器"或"一串机器"。但现实世界中很多工厂会做这么一件事：买好几台同样的设备放在一起。比如一个 CNC 加工车间里有 5 台完全相同的数控机床，一个数据中心里有几百台一模一样的服务器。这些设备可以并行工作——这就来到了平行机调度（Parallel Machine Scheduling）。

平行机调度看起来比流水车间简单？恰恰相反。当你有多个并行资源时，你就多了一个以前不存在的决策维度：哪个工件分配给哪台机器？这个"分配"问题一旦出现，组合爆炸就随之而来。

平行机的三种类型

平行机并不都是"一个模子刻出来的"。根据机器之间的关系，可以分为三个层次：

  1. 同速平行机（Identical Parallel Machines, P）——所有机器完全相同。工件在任意一台机器上的加工时间一样。这是最简单的平行机类型，记为 P / Cmax。比如一排完全相同的 CNC 机床。
  2. 异速平行机（Uniform Parallel Machines, Q）——机器速度不同，但每台机器对所有工件的速度是相同的比例。例如一台新机器速度是旧机器的 2 倍。工件 i 在机器 j 上的加工时间 = pᵢ / sⱼ，其中 sⱼ 是机器 j 的速度因子。
  3. 无关平行机（Unrelated Parallel Machines, R）——工件在不同机器上的加工时间没有固定比例关系。机器 A 适合加工薄壁件，机器 B 适合加工大型件。记为 R / Cmax。这是最一般的情况，也是最难的情况。
类型 三参数符号 加工时间关系 难度 典型场景
同速平行机 P / Cmax pij = pi（与机器无关） 弱 NP-hard 相同的数控机床集群
异速平行机 Q / Cmax pij = pi / sj NP-hard 不同代际的 GPU 服务器
无关平行机 R / Cmax pij 任意给定 强 NP-hard 特种加工设备的柔性组合

LPT规则与列表调度 — P||Cmax的近似算法

对于 P / Cmax（同速平行机，最小化 makespan），有一个非常简洁且有效的规则：LPT（Longest Processing Time first）规则。

LPT 的步骤很简单：

  1. 将所有工件按加工时间从大到小排序。
  2. 依次取出工件，将它分配给当前总负载最小的那台机器。

这个规则的本质是"大的先安排，小的填空隙"。直觉上，较大的工件如果留到最后，可能会导致某台机器的完工时间被"撑"得很大，所以先把大工件分散到各台机器上。

LPT 有一个非常重要的理论结果——它的近似比为 4/3 - 1/(3m)。翻译成人话：LPT 给出的 makespan 最多比最优值大 4/3 - 1/(3m) 倍。当 m=2 时，最坏情况是 4/3 - 1/6 = 7/6 ≈ 1.167，即最多比最优值差 16.7%。当 m 很大时，趋近于 4/3 ≈ 1.333，即最坏情况下差 33.3%。

近似比的意义：你可能会问："知道了 LPT 最坏情况下差 33% 又怎样？" 这个问题其实非常重要。假设你给一家工厂部署了一个 LPT 调度系统，上线第一天工厂老板问："你这个算法比最优差多少？" 你不能说"通常挺好"，你需要一个有理论保证的答案。知道近似比是 4/3，你就可以说："在最坏情况下，我们的调度方案不超过最优方案的 33%。" 这个保证值虽然不完美，但它让决策者有了预期的底线。相比之下，很多没有理论保证的启发式，你连最坏会差多少都不知道。

LPT 手工示例

假设有 7 个工件、3 台同速平行机。加工时间分别为：5, 9, 3, 8, 6, 7, 4。

LPT调度手工计算

工件（按时间降序）: 9, 8, 7, 6, 5, 4, 3
机器: M1, M2, M3

Step 2: 分配工件8 → machines负载: M1:9,  M2:8,  M3:0   → 选M3（当前负载最小=0）
Step 3: 分配工件7 → machines负载: M1:9,  M2:8,  M3:7   → 选M3（当前负载最小=7）
Step 4: 分配工件6 → machines负载: M1:9,  M2:8,  M3:13  → 选M2（当前负载最小=8）
Step 5: 分配工件5 → machines负载: M1:9,  M2:14, M3:13  → 选M1（当前负载最小=9）
Step 6: 分配工件4 → machines负载: M1:14, M2:14, M3:13  → 选M3（当前负载最小=13）
Step 7: 分配工件3 → machines负载: M1:14, M2:14, M3:17  → 选M1（当前负载最小=14）

最终:
  M1: 9 + 5 + 3 = 17
  M2: 8 + 6 = 14
  M3: 7 + 4 = 11
Makespan = max(17, 14, 11) = 17

最优 makespan？（手工尝试）
  M1: 9 + 4 + 3 = 16
  M2: 8 + 7 = 15
  M3: 6 + 5 = 11
最优 = 16
LPT偏差 = (17-16)/16 = 6.25%  < 4/3-1/9 ≈ 1.296

列表调度（List Scheduling）是 LPT 的通用框架：维护一个"待安排工件的列表"，每次从列表中取出一个工件（按某种优先级规则），分配给当前负载最小的机器。LPT 是这个框架下的一种具体规则（按加工时间降序排列）。其他常见的列表调度规则有：SPT（最短加工时间优先）、ECT（最早完工时间优先）、WECT（加权最早完工时间优先）等。

一个小历史：LPT 规则的最坏情况证明由 Graham 在 1969 年给出。你可能记得 Graham 正是三参数表示法的提出者之一。实际上，Graham 对调度理论的贡献远不止符号表示——他在近似算法方面的奠基性工作至今仍被广泛引用。这就是大师：不仅定义了语言，还提供了深刻的结果。

批处理机调度 — 烧入炉、热处理等应用

平行机调度中还有一个重要的变体——批处理机（Batch Processing Machine）。在这种场景下，一台机器能够同时加工多个工件，所有同批处理的工件的开始和结束时间相同。典型应用包括：

  • 烧入炉（Burn-in Oven）——半导体制造中，芯片需要在特定温度下"烧入"若干小时。一个炉子里可以同时放几十个芯片，但整个批次的加工时间取决于批次中最长的那个芯片所需的烧入时间。
  • 热处理炉（Heat Treatment Furnace）——机械零件需要回火、淬火等热处理，一个炉子可以装多个零件，批次加工时间取决于零件的材料厚度要求。
  • 消毒灭菌设备（Sterilization）——医疗器械的批量消毒，一次可处理多件器械。

批处理机又分为两大类：

  • 串行批（Serial Batch）——批次内工件的加工时间可以不同，批次的加工时间 = 批次内所有工件加工时间之和。这在数学上等价于单机调度（只是批处理减少了 setup 次数）。
  • 并行批（Parallel Batch）——批次内所有工件同时加工，批次的加工时间 = 批次内工件的最大加工时间。半导体烧入炉就是典型的并行批。

半导体制造中的批处理调度尤其复杂。一个晶圆厂可能有数十台烧入炉，每个批次有容量上限（比如每炉最多 25 片晶圆），不同产品需要不同的烧入时间，而且同一批次中的晶圆必须来自同一工艺步骤。这种约束的组合使得调度优化极具挑战性。

实际工程案例——半导体批调度翻车记：某 8 英寸晶圆厂曾导入一个批调度系统，初始版本使用了一个简单的贪心规则："先凑满一批再开始。" 结果这个策略导致一个严重问题——关键任务的晶圆在炉子前等了好几个小时，就为了"凑满"25 片，严重拉长了光刻区的 cycle time。后来改为"最长等待时间优先 + 可提前开始不等待满批"的策略，关键设备的利用率没有下降，但 cycle time 缩短了 30%。教训：批调度的核心矛盾是"利用率 vs 等待时间"——纯粹追求批次满载率反而会拖慢整体节奏。
