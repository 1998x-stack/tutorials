第8章 LangChain 1.x 统一 API

如果你在 2024 年接触过 LangChain，大概率体验过那种"改个版本号就要重写全部代码"的酸爽。从 0.1 到 0.3，每次大版本更新几乎都是一次重构——今天用 LLMChain，明天告诉你 deprecated 了请用 RunnableSequence，后天又说你该用 LCEL（LangChain Expression Language）。社区的吐槽声浪之大，以至于 LangChain 团队在 2025 年痛下决心，推出了 1.0 大版本，核心目标就是两个字——统一。

到 2026 年中，LangChain 1.x（最新 v1.2.16）已经是一个完全不同的框架了。这一章我们来拆解 1.x 的架构重塑，重点学习 create_agent 统一 API、ChatModel 抽象、Runnable 协议、以及新一代 Tool 集成方案。学完这一章，你会发现以前那些"灵光一现"的 Agent 封装，原来都可以用标准接口来表达。

版本提示： 本章基于 LangChain v1.2.16，Python 3.11+。所有代码示例均使用新 API，不涉及任何 0.x 时代的遗留写法。如果你项目中还有旧代码，建议先升级再阅读本章。

8.1 LangChain 1.x 架构总览

1.x 版本最核心的架构变化可以概括为一句话：万物皆 Runnable。在 0.x 时代，LangChain 的组件体系非常混乱——模型有 Model 类、链有 Chain 类、工具有 Tool 类、Agent 有 AgentExecutor 类，每个体系都有自己的抽象基类和调用方式。而 1.x 将这些全部统一到了 Runnable 协议之下。

概念 0.x 时代 1.x 时代
模型 BaseLLM, BaseChatModel（各自独立） ChatModel (统一接口，继承 Runnable)
链 LLMChain, SimpleSequentialChain（基类混乱） RunnableSequence (通过 | 操作符构造)
提示模板 PromptTemplate（独立的 format 方法） ChatPromptTemplate (也是 Runnable)
Agent AgentExecutor + 多种 Agent 类型 create_agent() 统一工厂函数
工具 @tool 装饰器 + Tool 类 @tool 装饰器 (增强) + ToolSpec
中间件 无统一抽象 RunnableMiddleware 体系

这个统一的本质意义在于：无论是模型调用、提示处理、工具执行还是 Agent 决策，都可以用同样的 invoke() 方法来执行，都可以用 pipe() 或 | 操作符来组合。这极大降低了学习曲线——你只需要学会一套 API 语义，就能驾驭整个框架。

升级迁移成本： 根据 LangChain 官方 2026 年初的统计，从 0.3 迁移到 1.x 平均每个项目需要改动约 40% 的代码量。但迁移后的代码量平均减少了 30%，因为 1.x 的 API 更加简洁统一。

8.2 统一 create_agent API 详解

在 1.x 中，创建 Agent 的方式被极大地简化了。不再需要手动拼接 Prompt、绑定工具、初始化 AgentExecutor——一个 create_agent() 工厂函数搞定一切：

from langchain.agents import create_agent
from langchain.chat_models import ChatOpenAI
from langchain.tools import tool

@tool
def search_weather(city: str) -> str:
    """查询指定城市的天气"""
    return f"{city}今日天气：晴，25-32℃"

@tool
def calculate(expression: str) -> str:
    """计算数学表达式"""
    return str(eval(expression))

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[search_weather, calculate],
    system_prompt="你是一个气象助手，擅长查询天气和计算。"
)

result = agent.invoke({"input": "北京今天多少度？顺便算一下 128*37 等于多少"})
print(result["output"])

create_agent() 内部自动完成了以下工作：将 tools 转换为模型可识别的工具定义格式、构造包含工具描述的 System Prompt、设置 ReAct 循环的终止条件、配置最大迭代轮次和错误处理策略。你甚至可以通过参数覆盖默认行为：

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[search_weather, calculate],
    system_prompt="你是一个气象助手。",
    max_iterations=10,           # 最大推理轮次
    max_execution_time=60,       # 最大执行时间（秒）
    early_stopping_method="generate",  # 超时时生成最终回复
    return_intermediate_steps=True,    # 返回中间步骤用于调试
)
踩坑实录： 我们团队在 2025 年底第一次迁移到 1.x 时，最大的坑来自 max_iterations 的默认值。旧版本默认是 15，1.x 改为 5。结果所有 Agent 在复杂任务上频繁返回不完整结果，排查了半天才发现是这个参数变化。升级后建议根据任务复杂度显式设置这个值。

8.3 模型封装：ChatModel 与 LLM 抽象

1.x 对模型层的重构是整个统一架构的基石。核心变化是：彻底废弃了 BaseLLM，统一使用 ChatModel。无论是 OpenAI 的 GPT-4o、Anthropic 的 Claude Opus 4.7、Google 的 Gemini 2.5 Pro，还是开源的 Llama 4、DeepSeek-V4，都通过 ChatModel 这个统一接口来调用。

统一的模型接口

from langchain.chat_models import ChatOpenAI, ChatAnthropic, ChatGoogle

# 不同厂商的模型拥有完全相同的接口
gpt4o = ChatOpenAI(model="gpt-4o", temperature=0.7)
claude = ChatAnthropic(model="claude-opus-4-7", temperature=0.7)
gemini = ChatGoogle(model="gemini-2.5-pro", temperature=0.7)

# 统一调用方式
result = gpt4o.invoke("你好！")
result = claude.invoke("你好！")
result = gemini.invoke("你好！")

多模型 Fallback 与路由

1.x 内置了强大的模型路由能力——这在生产环境中极为实用：

from langchain.chat_models import ChatOpenAI, ChatAnthropic
from langchain.chat_models.routing import FallbackModel

# 主模型 + 备用模型链
model = FallbackModel(
    primary=ChatOpenAI(model="gpt-4o"),
    fallbacks=[
        ChatOpenAI(model="gpt-4o-mini"),   # API 限流时降级
        ChatAnthropic(model="claude-sonnet-4-7"),  # OpenAI 不可用时切换
    ]
)

result = model.invoke("Explain quantum computing")
模型封装特性 说明 适用场景
ChatModel 统一接口 所有厂商模型通过同一基类调用 多厂商切换、测试对比
FallbackModel 自动降级到备用模型 生产环境高可用
RateLimiter 请求频率控制与退避 避免 API 限流
ModelRouter 按任务类型路由到不同模型 成本优化（简单任务用便宜模型）
TokenUsageTracker 实时跟踪 Token 消耗 成本监控与预算控制
实践建议： 在我们的项目中，"主模型 + Mini 模型 + 友商模型"的三层 Fallback 配置是最常见的生产级设置。GTP-4o 处理复杂推理，GPT-4o-mini 在限流时顶上，Claude Sonnet 作为跨厂商的最后防线。这样单模型故障也不会影响整体服务。

8.4 Runnable 协议与链式调用

Runnable 协议是 LangChain 1.x 的灵魂。任何继承自 Runnable 的组件都支持三个核心方法：invoke()（同步调用）、ainvoke()（异步调用）、batch()（批量调用）。更重要的是，Runnable 之间可以通过 | 操作符组合成管道：

from langchain.chat_models import ChatOpenAI
from langchain.prompts import ChatPromptTemplate
from langchain.schema import StrOutputParser

# 每个组件都是 Runnable
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个{role}专家"),
    ("human", "{question}")
])
model = ChatOpenAI(model="gpt-4o")
parser = StrOutputParser()

# 管道组合：prompt -> model -> parser
chain = prompt | model | parser

# 统一 invoke 方式
result = chain.invoke({
    "role": "Python",
    "question": "什么是装饰器？"
})
print(result)

这个 | 操作符的背后是 RunnableSequence 类——它将前一个 Runnable 的输出自动传递给下一个 Runnable 的输入。这种设计让代码的声明性大幅提升，你可以像搭积木一样组合功能模块。

对于更复杂的场景，1.x 提供了多种组合原语：

from langchain.runnables import RunnableParallel, RunnableBranch, RunnablePassthrough

# 并行执行：同时调用多个模型
parallel = RunnableParallel({
    "gpt_result": gpt_chain,
    "claude_result": claude_chain,
})

# 条件分支
branch = RunnableBranch(
    (lambda x: len(x) > 100, long_text_chain),
    (lambda x: "紧急" in x, urgent_chain),
    default_chain,
)

# 传递原始输入 + 加工输出
pipe = {"input": RunnablePassthrough()} | processing_chain
设计哲学： Runnable 协议带来的最大好处是"可组合性"。以前你需要学习不同的 API（Chain.run(), AgentExecutor.invoke(), Model.predict()），现在所有组件都响应同一个 invoke 方法。这种统一性是 LangChain 1.x 成熟的重要标志。

8.5 Tool 集成与新中间件系统

1.x 对工具系统也做了全面升级。除了经典的 @tool 装饰器，还引入了 ToolSpec 机制——通过声明式配置来定义工具的元数据、参数校验和错误处理。

from langchain.tools import tool, ToolSpec

# 经典方式依然支持
@tool
def send_email(to: str, subject: str, body: str) -> str:
    """发送邮件"""
    print(f"发送邮件给 {to}: {subject}")
    return "发送成功"

# 新的 ToolSpec 声明式方式
email_tool = ToolSpec(
    name="send_email",
    description="发送电子邮件",
    args_schema={
        "to": {"type": "string", "description": "收件人地址"},
        "subject": {"type": "string", "description": "邮件主题"},
        "body": {"type": "string", "description": "邮件正文"},
    },
    execute_fn=send_email_impl,
    timeout=30,  # 超时控制
    retry=3,     # 失败重试次数
)

# 工具中间件：日志、限流、监控
from langchain.tools.middleware import ToolLogging, ToolRateLimit, ToolMetrics

analytics_tool = ToolSpec(
    name="query_analytics",
    description="查询分析数据",
    execute_fn=query_db,
    middleware=[
        ToolLogging(),           # 记录每次工具调用
        ToolRateLimit(calls=10, period=60),  # 每分钟最多10次
        ToolMetrics(),           # 采集延迟和成功率
    ]
)

工具中间件是 1.x 的另一个亮点。它允许你在工具调用链上插入横切关注点——日志记录、频率控制、调用审计、缓存，而无需修改工具本身的业务逻辑。我们会在第 9 章详细讲解中间件系统的完整用法。

踩坑实录： 在 1.x 中使用 @tool 装饰器时，函数签名中的类型注解非常重要。如果参数没有类型注解，LangChain 会将其视为字符串类型。我们有一次给 count_users 工具的参数没加 int 注解，结果模型传入 "五百" 这样的中文数字，工具直接报错。加上 min_age: int 注解后，模型会自动转换成数字，问题解决。

8.6 生产级 Agent 示例代码

最后，我们来看一个完整的生产级 Agent 示例——一个支持多工具、带记忆功能、有错误处理的信息检索助手：

from langchain.agents import create_agent
from langchain.chat_models import ChatOpenAI, ChatAnthropic
from langchain.chat_models.routing import FallbackModel
from langchain.tools import tool, ToolSpec
from langchain.memory import ConversationBufferMemory
from langchain.tools.middleware import ToolLogging, ToolRateLimit

# --- 工具定义 ---

@tool
def search_web(query: str) -> str:
    """搜索网络信息。输入搜索关键词，返回搜索结果摘要。"""
    # 实际项目中调用 SerpAPI / Bing Search 等
    return f"关于「{query}」的搜索结果：..."

@tool
def query_database(sql: str) -> list:
    """执行 SQL 查询。输入 SQL 语句，返回查询结果列表。"""
    return [{"id": 1, "name": "示例数据"}]

# --- Agent 配置 ---

agent = create_agent(
    model=FallbackModel(
        primary=ChatOpenAI(model="gpt-4o", temperature=0.3),
        fallbacks=[
            ChatOpenAI(model="gpt-4o-mini"),
            ChatAnthropic(model="claude-sonnet-4-7"),
        ]
    ),
    tools=[
        search_web,
        query_database,
        ToolSpec(
            name="send_notification",
            description="向用户发送通知消息",
            execute_fn=lambda user_id, msg: f"通知已发送至 {user_id}",
            timeout=10,
            middleware=[ToolLogging(), ToolRateLimit(calls=5, period=60)]
        )
    ],
    memory=ConversationBufferMemory(
        memory_key="chat_history",
        return_messages=True,
        max_token_limit=2000
    ),
    system_prompt=(
        "你是一个企业级信息检索助手。你可以搜索网络、查询数据库、"
        "发送通知。请优先使用数据库查询获取确认信息，网络搜索作为补充。"
        "每次给出答案时请注明信息来源。"
    ),
    max_iterations=8,
    return_intermediate_steps=True,
)

# --- 执行 ---

result = agent.invoke({
    "input": "查询昨天的新增用户数，如果超过 1000 则发通知给我"
})

print(f"最终回答：{result['output']}")
print(f"中间步骤：{result['intermediate_steps']}")
生产实践： 上面的代码中，我们使用了 FallbackModel 实现跨厂商高可用、ToolSpec 的 middleware 做限流和日志、ConversationBufferMemory 保留对话上下文。这套配置已经在我们的项目中稳定运行了 6 个月以上，月均处理超过 50 万次 Agent 调用。关键是——所有代码加起来不到 60 行。

本章小结

LangChain 1.x 的"统一 API"不仅是一次版本升级，更是一种设计哲学的成熟。通过万物皆 Runnable 的协议抽象、统一 create_agent 工厂函数、ChatModel 多厂商适配、以及新一代 ToolSpec 中间件系统，1.x 让 Agent 开发从"拼积木"升级到了"搭管道"——更简洁、更可组合、更容易测试和部署。

接下来，我们将深入 1.x 中最令人兴奋的新特性之一——中间件系统。从 PII 脱敏到 Human-in-the-loop，内容审核到 Model Fallback，你会发现如何用标准的中间件模式来优雅地解决生产环境中那些"脏活累活"。
