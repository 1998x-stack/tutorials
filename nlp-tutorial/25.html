第25章：模型部署

恭喜你走到这里！经过前面24章的洗礼，你已经掌握了从文本预处理到Transformer、从BERT微调到模型训练优化的全套技能。你的模型在开发集上跑出了令人满意的指标，一切看起来都很完美——然后你把它交给业务方，对方问了一句："这个接口长什么样？QPS能到多少？"

你愣住了。

这就是我们这章要解决的问题。你的模型在Jupyter Notebook里跑得再好，不能在生产环境稳定提供服务，那它就只是一个昂贵的学术玩具。套用一句业内老话："Your model is useless until it's serving predictions." 这个从"能跑"到"能用"之间的鸿沟，就是我所说的部署鸿沟（Deployment Gap）。

我在2018年经历过一次惨痛教训。当时花了两周微调了一个BERT-base做意图分类，离线F1高达0.94。高高兴兴上线，结果每次请求要2.3秒——用户早跑了。那之后我才意识到：离线指标和在线体验之间隔着整个工程优化。这一章我将把那些年踩过的坑、学到的招全都倒出来。

部署一个NLP模型，核心要解决三个矛盾：速度 vs 精度、成本 vs 吞吐、灵活性 vs 稳定性。我们从最基础的模型格式标准化开始，一路走到高吞吐的LLM服务方案。

25.1 ONNX：跨框架模型标准化

你负责训练用PyTorch，隔壁团队推理用TensorFlow，运维那边要求用Triton Server——三个团队各玩各的，模型文件格式五花八门，每次交接都鸡飞狗跳。这就是ONNX要解决的核心问题。

ONNX（Open Neural Network Exchange）是微软和Facebook在2017年联合推出的中间表示格式。你可以把它理解为模型界的"通用语言"——不管你的模型是用PyTorch、TensorFlow还是JAX训练的，都能导出为一个标准的ONNX文件，然后在任何支持ONNX的运行时上推理。

PyTorch模型导出为ONNX极其简单。你的forward方法里不能有控制流（if/for loop）依赖动态数据（比如根据输入长度走不同分支），否则导出会失败。下面是一个标准导出流程：

import torch
import torch.onnx

model = load_my_bert_model()
model.eval()

dummy_input = torch.randint(0, 30522, (1, 128))  # batch=1, seq_len=128
dummy_attention_mask = torch.ones(1, 128)

torch.onnx.export(
    model,
    (dummy_input, dummy_attention_mask),
    "bert_intent.onnx",
    input_names=["input_ids", "attention_mask"],
    output_names=["logits"],
    dynamic_axes={
        "input_ids": {0: "batch_size", 1: "seq_len"},
        "attention_mask": {0: "batch_size", 1: "seq_len"},
        "logits": {0: "batch_size"},
    },
    opset_version=17,
)

dynamic_axes参数非常关键。如果不设置，导出的模型只能处理(1, 128)这一种输入形状。设了dynamic_axes，就可以接受不同batch size和序列长度的输入——这在生产环境中几乎是必须的。

导出后用ONNX Runtime做推理，你不需要任何深度学习框架，只需要一个轻量的runtime库：

import onnxruntime as ort

session = ort.InferenceSession("bert_intent.onnx")
inputs = {
    "input_ids": np.array([[101, 2054, 2003, ...]], dtype=np.int64),
    "attention_mask": np.array([[1, 1, 1, ...]], dtype=np.int64),
}
outputs = session.run(["logits"], inputs)
predicted = outputs[0].argmax(axis=1)

ONNX Runtime的推理速度通常比原生PyTorch eager模式快30%~60%，主要得益于计算图优化、算子融合和内存复用。如果你的模型在CPU上部署，这个差距甚至会拉到2~3倍。

最佳实践：ONNX不是你部署流程的终点，而是起点。我的习惯是：先导出ONNX并用ONNX Runtime验证精度对齐（保证diff在1e-5以内），然后再用ONNX作为输入传给TensorRT或OpenVINO做进一步优化。ONNX的规范化计算图也给后续所有优化工具提供了一个干净的输入。

25.2 TensorRT：NVIDIA的推理加速引擎

如果你有NVIDIA GPU，不可能没听说过TensorRT。它是NVIDIA官方的推理优化引擎，核心武器是层融合、精度校准和内核自动调优。我做过一个对比：同样的BERT-base模型，PyTorch FP32推理延迟35ms，TensorRT FP16直接降到9ms，INT8量化后更是杀到4ms——8倍差距。

TensorRT的工作流程三步走：

  1. 解析（Parse）：读入ONNX模型，构建计算图。
  2. 优化（Build）：执行图优化（层融合、常量折叠、死代码消除），为目标GPU选择最优kernel。
  3. 序列化（Serialize）：把优化后的引擎保存为.plan文件，部署时反序列化即可使用。

INT8量化是TensorRT的王牌功能。它需要你提供一份校准数据集（calibration dataset），TensorRT会在校准过程中收集每层激活值的分布，然后选择最优的缩放因子把FP32映射到INT8。校准数据集不需要太大——几百到一千条就够了，但必须覆盖真实推理场景中的数据分布。如果你用ImageNet的数据给一个医疗NLP模型做校准，量化后的精度会崩得非常难看。

INT4量化警告：从2024年开始，TensorRT支持了INT4量化，在LLM推理上可以把显存占用再砍一半。但INT4的精度损失对中小模型来说通常不可接受。我建议这条分界线以7B参数量为界——7B以下模型老老实实用FP16或INT8，7B以上模型再考虑INT4。

使用TensorRT的Python API示例：

import tensorrt as trt

TRT_LOGGER = trt.Logger(trt.Logger.WARNING)
builder = trt.Builder(TRT_LOGGER)
network = builder.create_network(1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH))
parser = trt.OnnxParser(network, TRT_LOGGER)

with open("bert_intent.onnx", "rb") as f:
    parser.parse(f.read())

config = builder.create_builder_config()
config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 1 << 30)  # 1GB

# FP16 开启
config.set_flag(trt.BuilderFlag.FP16)
# INT8 开启（需要校准器）
# config.set_flag(trt.BuilderFlag.INT8)

plan = builder.build_serialized_network(network, config)
with open("bert_intent.plan", "wb") as f:
    f.write(plan)

runtime = trt.Runtime(TRT_LOGGER)
engine = runtime.deserialize_cuda_engine(plan)

值得注意的是，TensorRT对动态形状的支持比较挑剔。如果你需要在推理时动态改变batch size或序列长度，必须在build时显式声明优化范围（optimization profile），指定最小、最常见、最大三个维度的值。选错了"最常见"值，TensorRT的kernel选择会跑偏，延迟反而比不用TensorRT还高。

25.3 vLLM：大模型高吞吐推理引擎

如果你部署的是LLM（比如Llama 3、Qwen、ChatGLM），上面说的ONNX和TensorRT当然也能用——但你会发现两个问题：显存浪费严重、首token延迟高。传统推理框架的KV Cache是为每个请求独立分配的，而不同请求的序列长度差异巨大，导致大量显存碎片和浪费。

vLLM在2023年横空出世，它的核心创新是PagedAttention——一个借鉴了操作系统虚拟内存分页机制的KV Cache管理方案。具体来说：

  • 传统的KV Cache是"连续内存分配"：每个请求的KV Cache必须在一块连续的内存区域中。这导致内存碎片和预留浪费（按最大长度预留）。
  • PagedAttention把KV Cache切成固定大小的"页"（block），通过页表将连续的逻辑地址映射到不连续的物理内存块。这彻底解决了内存碎片问题。

效果有多夸张？我在A100 80GB上用vLLM部署Llama 3-8B，对比HuggingFace的Transformers原生推理，吞吐量提升8~12倍。同样的硬件，原来只能塞4个并发请求，vLLM能用PagedAttention加上连续批处理（continuous batching）轻松跑30~40个并发。

部署vLLM非常简单，一行命令起一个兼容OpenAI API的服务：

# 安装
pip install vllm

# 启动服务（暴露 /v1/chat/completions 接口）
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3.1-8B-Instruct \
    --tensor-parallel-size 2 \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.90 \
    --max-num-seqs 256

参数说明：--tensor-parallel-size 2表示用张量并行将模型切到两块GPU上；--gpu-memory-utilization 0.90表示预留10%的显存给模型权重以外的开销（如激活值）；--max-num-seqs 256是最大并发请求数，设得太高可能导致显存溢出（OOM）。

调用时和调用OpenAI的API一模一样：

from openai import OpenAI

client = OpenAI(base_url="http://localhost:8000/v1", api_key="sk-xxx")
response = client.chat.completions.create(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[{"role": "user", "content": "什么是PagedAttention？"}],
    temperature=0.7,
    max_tokens=512,
)

除了PagedAttention，vLLM还支持前缀缓存（prefix caching）——如果多个请求共享相同的系统提示词（system prompt），它们的公共前缀KV Cache可以被复用，在RAG场景下能节省30%~60%的首次推理时间。

25.4 搭建REST API：从模型到服务

有了优化后的模型，下一步是把它包装成一个HTTP服务。FastAPI是目前Python生态中做模型服务最主流的选择——原生异步、自动生成OpenAPI文档、性能接近Go写的服务。

下面是一个完整的FastAPI推理端点，包含批处理和异步处理：

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import onnxruntime as ort
import numpy as np
from typing import List
import asyncio
from concurrent.futures import ThreadPoolExecutor

app = FastAPI(title="NLP Intent Classifier")

# 模型加载（全局单例）
session = ort.InferenceSession("bert_intent.onnx")
executor = ThreadPoolExecutor(max_workers=4)

class InferenceRequest(BaseModel):
    input_ids: List[List[int]]
    attention_mask: List[List[int]]

class InferenceResponse(BaseModel):
    predictions: List[int]
    probabilities: List[float]

def _run_inference(input_ids, attention_mask):
    """同步推理函数，由线程池执行"""
    outputs = session.run(
        ["logits"],
        {"input_ids": np.array(input_ids, dtype=np.int64),
         "attention_mask": np.array(attention_mask, dtype=np.int64)}
    )
    logits = outputs[0]
    preds = logits.argmax(axis=1).tolist()
    probs = np.max(softmax(logits, axis=1), axis=1).tolist()
    return preds, probs

@app.post("/predict", response_model=InferenceResponse)
async def predict(request: InferenceRequest):
    """批量预测接口"""
    if len(request.input_ids) == 0:
        raise HTTPException(status_code=400, detail="Empty batch")

    loop = asyncio.get_event_loop()
    preds, probs = await loop.run_in_executor(
        executor, _run_inference,
        request.input_ids, request.attention_mask
    )
    return InferenceResponse(predictions=preds, probabilities=probs)

@app.get("/health")
async def health():
    return {"status": "ok"}

if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8000)

几个设计要点：

  • 线程池隔离：ONNX Runtime的推理是CPU密集操作，不能直接放在async event loop里跑（会阻塞其他请求）。用run_in_executor把它丢到线程池中。
  • 模型全局单例：模型加载是一次性操作，放在模块级别，每次请求复用session对象。切忌每个请求都创建一个新session——内存会爆炸。
  • 客户端侧批处理：上面的接口允许客户端一次传入多条数据。相比每条请求单独调用，批处理能充分利用GPU的并行计算能力，吞吐量通常提升3~5倍。
  • 健康检查端点：/health是Kubernetes和负载均衡器做存活探针（liveness probe）的标准配置。没有它，运维团队没法自动恢复你的服务。

⚠️ 踩坑故事：一个请求超时引发的惨案：2022年我部署一个BERT命名实体识别（NER）服务，压测时P99延迟在200ms以内，很满意地上线了。上线第二天，一个用户传了一篇5000字的文章进来——我们的分词+编码逻辑没做长度截断，结果单个请求处理时间飙到47秒！下游的超时设置是30秒，请求一直在堆积，最终所有worker线程都被卡住，服务雪崩宕机。

事后复盘三件事：1）所有输入必须在API入口处做严格的最大长度限制和截断；2）加上超时保护（FastAPI的timeout中间件）；3）长文本应该走异步离线处理，而不是在线推理。这次事故让我明白一个道理：生产环境的敌人从来不是正常的流量，而是你没想到的边缘情况。

25.5 监控与可观测性

模型上线不是终点，而是起点。你永远不知道模型在你睡觉的时候会出什么幺蛾子——数据分布漂移了、GPU OOM了、延迟突然飙升了。没有监控，你就是在瞎跑。

我把模型服务的监控分为三个层次：

延迟与吞吐量监控

这是最基础的层面。用Prometheus + Grafana收集以下指标：

  • P50 / P95 / P99 延迟：平均延迟会说谎。P99能告诉你最慢的1%用户有多痛苦。如果P99 > 500ms就开始报警。
  • QPS（每秒查询数）：当前的流量水位。和基线对比，异常突增或突降都要关注。
  • GPU利用率：低于30%说明你的批处理策略没用好，高于95%说明该扩容了。
  • 显存使用量：随着时间推移缓慢增长可能意味着显存泄漏。

数据漂移检测

这是NLP部署独有的问题，也是最容易被忽略的。模型在上线时的数据分布和上线后的真实数据分布几乎一定会发生偏移——今天的用户输入和三个月前的用户输入在词汇、句式、主题上都有差异。随着时间推移，模型精度会悄然下降。

常用检测方法：

  • PSI（Population Stability Index）：比较预测概率分布的变化。PSI > 0.25说明发生了显著漂移。
  • Embedding-level drift：对输入文本的embedding做聚类，跟踪聚类中心的偏移量。
  • 无标签漂移检测：用模型输出的置信度分布作为代理指标——如果模型越来越"不确定"（置信度整体偏低），很可能数据已经变了。

反馈闭环

最理想的情况是有用户反馈标注——比如"这个回答有用"或"这个分类错了"。把这些反馈收集起来，定期（每天或每周）对模型做增量微调，形成"监控→报警→重训→部署"的闭环。这一步能做到的团队不超过20%，但做到了就是降维打击。

部署方案对比

方案 延迟（单请求） 吞吐量 硬件成本 适用场景 上手难度
PyTorch原生 / HF Transformers 高（30~50ms） 低 中 原型验证、小流量 低
ONNX Runtime 中（15~25ms） 中 低（CPU可用） 通用生产部署、跨平台 低
TensorRT FP16 低（5~10ms） 高 高（需要NVIDIA GPU） 高吞吐在线服务 中
TensorRT INT8 极低（3~5ms） 很高 高（需要NVIDIA GPU） 延迟敏感、成本敏感场景 中高
vLLM 中（首token延迟受batch影响） 极高（LLM场景） 高（需要大显存GPU） LLM推理、对话系统 低
Triton Inference Server 低（支持动态批处理） 很高 中高 多模型混部、企业级部署 高

选择建议：如果你团队只有CPU资源，ONNX Runtime是你的最佳选择。如果有NVIDIA GPU且部署的是BERT级别的模型，TensorRT FP16是性价比之王。如果部署的是7B以上的LLM，无脑选vLLM——它在这一领域几乎是没有对手的。

核心理念：从训练到部署是一条完整的水管

很多团队把训练和部署分成两拨人做，中间隔着一道厚厚的墙。训练团队写出来的模型不考虑部署约束（模型太大、算子不兼容、不支持动态形状），部署团队拿到模型后怨声载道地做各种workaround。正确的做法是：训练时就想到部署。把ONNX导出纳入训练流程的最后一个step，把INT8校准纳入训练后的标准操作，把延迟预算写进模型选型的技术指标。我在团队里推了一个原则：一个模型只有在"某人能用FastAPI调通它的推理接口"时才算训练完成——否则它就是一个半成品。
