项目概述：打造你的刷题大脑

同学们好，欢迎来到最后一章——也是整个课程中最特别的一章。前面的 27 章我们学习了各种算法、数据结构、面试技巧和竞赛策略。但你有没有想过一个问题：你用来记录和追踪这些知识本身的工具，是不是也应该被好好设计一下？

这一章我们不做算法题了，我们要做一个个人刷题笔记与跟踪系统——一个帮助你管理刷题进度、记录解题思路、制定复习计划、分析薄弱环节的全栈 Web 应用。这个项目不仅能让你把前 27 章学到的技术融会贯通，更重要的是——它是一个你可以放在 GitHub 上、写进简历里的真实作品。

项目定位：这不是一个"玩具项目"。一个设计良好的刷题追踪系统，可以成为你面试时展示系统设计能力、全栈开发能力、数据库设计能力和产品思维的活生生的案例。很多面试官看到这种项目都会感兴趣——"哎，这个你自己做的？说说你怎么设计的？"

核心功能设计

我们先从需求出发。一个刷题平台需要解决什么问题？

功能一：题目管理

这是最基础的功能——把 LeetCode 上的题目信息管理起来。每道题包含：标题、编号、难度（Easy/Medium/Hard）、标签（数组、DP、图……）、所属公司、LeetCode 链接。用户可以按任意维度组合筛选——比如"找所有 Medium 难度的 DP 题，且是 Google 考过的"。

功能二：刷题记录

每次你做完一道题，记录：做题时间、所用时长、解法描述、是否参考了题解、代码链接（GitHub Gist 或本地仓库）。这些数据日积月累后，可以回答很多有价值的问题："我最近一周刷了多少题？""DP 题的正确率是多少？""我平均每道题花多久？"

功能三：错题本

正如我们在第 27 章讨论的，错题本的核心不是"记下来"，而是"定期复习"和"错误分类"。系统要支持：标记错误类型（思路错误 / 边界遗漏 / 代码 bug / 超时）、添加反思笔记、设置复习提醒。

功能四：复习计划

系统根据遗忘曲线自动生成复习计划。比如你 6 月 15 日做了某道题，系统会自动安排 6 月 16 日、6 月 18 日、6 月 22 日、7 月 6 日……的复习提醒。复习时系统给你看题目标题和你的笔记，你回忆解法，然后点击"记得"或"忘了"来更新下一次复习时间。

一个差异化功能：在刷题记录中加入"思路类型"字段——比如"逆向思维"、"增量构造"、"二分答案"、"状态压缩"。当你积累了 50 条记录后，你可以按思路类型来筛选——看看自己哪些思维模式用得最多，哪些完全没用过。这能帮你发现自己思维上的盲区。

数据模型设计

我们用最简洁的数据库设计来支撑上述功能。下面是一个关系型数据库（PostgreSQL / MySQL）的表设计。

核心表结构

表名 字段 说明
problems id, leetcode_id, title, difficulty, url, created_at 题目基本信息
tags id, name (UNIQUE) 标签如 "Array", "DP"
problem_tags problem_id, tag_id 多对多关联
companies id, name (UNIQUE) 公司如 "Google", "Meta"
problem_companies problem_id, company_id 多对多关联
submissions id, problem_id, user_id, solved_at, duration_min, solution_summary, had_hint, code_url, error_type, created_at 每道题的提交记录
review_schedule id, problem_id, user_id, scheduled_date, status (pending/completed/skipped), reviewed_at 复习计划表 
-- 建表 SQL 示例（核心表）
CREATE TABLE problems (
    id SERIAL PRIMARY KEY,
    leetcode_id INTEGER NOT NULL UNIQUE,
    title VARCHAR(255) NOT NULL,
    difficulty VARCHAR(10) CHECK (difficulty IN ('Easy', 'Medium', 'Hard')),
    url VARCHAR(500),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE tags (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL
);

CREATE TABLE problem_tags (
    problem_id INTEGER REFERENCES problems(id) ON DELETE CASCADE,
    tag_id INTEGER REFERENCES tags(id) ON DELETE CASCADE,
    PRIMARY KEY (problem_id, tag_id)
);

CREATE TABLE submissions (
    id SERIAL PRIMARY KEY,
    problem_id INTEGER REFERENCES problems(id) ON DELETE CASCADE,
    user_id INTEGER NOT NULL,
    solved_at TIMESTAMP NOT NULL,
    duration_min INTEGER,
    solution_summary TEXT,
    had_hint BOOLEAN DEFAULT FALSE,
    code_url VARCHAR(500),
    error_type VARCHAR(50),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE review_schedule (
    id SERIAL PRIMARY KEY,
    problem_id INTEGER REFERENCES problems(id) ON DELETE CASCADE,
    user_id INTEGER NOT NULL,
    scheduled_date DATE NOT NULL,
    status VARCHAR(20) DEFAULT 'pending',
    reviewed_at TIMESTAMP,
    UNIQUE (problem_id, user_id, scheduled_date)
);
设计考量：这里把 user_id 直接作为外键引用，但实际项目中你会有一个独立的 users 表。error_type 字段是错题本的关键——一定要结构化数据，不要用自由文本填，否则后续统计非常困难。建议使用枚举值：思路错误、边界遗漏、代码bug、超时、无错误。

前端与后端架构

RESTful API 设计

后端采用经典的三层架构：Controller → Service → Repository。下面列出核心 API 端点：

方法 路径 功能
GET /api/problems 获取题目列表（支持 ?tag=&difficulty=&company= 查询参数）
POST /api/problems 添加新题目
GET /api/problems/:id 获取题目详情
POST /api/submissions 记录一次提交
GET /api/submissions?problem_id= 获取某题的历史提交
GET /api/stats/weakness 薄弱环节分析（按标签统计正确率）
GET /api/reviews/today 获取今日待复习题目
POST /api/reviews/:id/complete 完成一次复习（更新 schedule）

用户认证

对于个人项目，最简单的做法是使用 JWT（JSON Web Token）。用户注册/登录后，服务端返回一个 token，前端把 token 存在 localStorage 或 httpOnly cookie 中，每次请求带上。不需要 OAuth、不需要第三方登录，简单够用就行。

# JWT 认证流程（伪代码）
# 注册
@app.route('/api/register', methods=['POST'])
def register():
    data = request.get_json()
    username = data['username']
    password = hash_password(data['password'])  # 用 bcrypt 或 argon2
    db.execute('INSERT INTO users (username, password_hash) VALUES (?, ?)',
               (username, password))
    return jsonify({'message': 'ok'}), 201

# 登录
@app.route('/api/login', methods=['POST'])
def login():
    data = request.get_json()
    user = db.execute('SELECT * FROM users WHERE username = ?',
                      (data['username'],)).fetchone()
    if not user or not verify_password(data['password'], user['password_hash']):
        return jsonify({'error': 'invalid credentials'}), 401
    token = jwt.encode({'user_id': user['id'], 'exp': time() + 86400},
                       SECRET_KEY, algorithm='HS256')
    return jsonify({'token': token})

统计面板

一个好的统计面板能让你直观地看到自己的进步。建议包含以下图表：

  • 每日刷题数折线图——看趋势，警惕断档
  • 难度分布饼图——Easy / Medium / Hard 的比例是否合理
  • 标签正确率柱状图——哪些标签是你的弱项（正确率低于 60%）
  • 复习日历——按月显示复习完成情况（类似 GitHub 贡献图）
  • 累计刷题曲线——累积量与时间的对比，直观感受"复利效应"
技术选型建议：前端可以用 React + Chart.js（轻量级图表库）或 TailwindCSS 快速搭建 UI；后端可以用 Python Flask/FastAPI 或 Node.js Express；数据库用 PostgreSQL。如果你想快速出原型，甚至可以用 Python 的 Dash 或 Streamlit——但如果你是为了给面试官看，选主流技术栈（React + FastAPI + PostgreSQL）更稳妥。

扩展思考

基础版本做完了，如果你还有时间和精力，这里有三个有价值的扩展方向。

集成 LeetCode API

LeetCode 官方没有公开 API，但社区中有一些反向工程方案。你可以通过 GraphQL 接口（https://leetcode.com/graphql）来拉取题目信息、提交状态、甚至比赛结果。这样你的系统就可以自动同步你 LeetCode 上的提交记录，无需手动录入。

扩展创意：写一个浏览器插件，在你每次 LeetCode 提交通过后，自动弹窗询问"要不要记录到我的刷题系统？"——这会让使用体验变得极其流畅。面试官看到这个功能大概率会眼前一亮。

团队协作功能

刷题不一定是孤独的。你可以添加：

  • 刷题小组——几个人组成小组，互相可见对方本周刷题量
  • 题解分享——每个人可以发布自己的解题思路，其他人可以点赞评论
  • 周赛排名——记录每个人的周赛成绩，小组内比拼

这个功能特别适合准备校招的同学——几个同学组队刷题，互相监督、互相讲解，效果远好于一个人闷头刷。

自动化复习提醒

可以集成 Telegram Bot、邮件、或钉钉/飞书 webhook。每天早上 8 点自动发一条消息："你今天有 5 道题待复习，预计用时 45 分钟。" 这种自动化的提醒比任何"我决定每天复习"的誓言都有效——因为它不需要你动用意志力去记。

# 复习提醒生成的伪代码（每天凌晨运行）
def generate_daily_reviews(user_id):
    today = date.today()
    # 找出所有需要今天复习的题目
    schedules = db.execute('''
        SELECT p.id, p.title, rs.scheduled_date
        FROM review_schedule rs
        JOIN problems p ON p.id = rs.problem_id
        WHERE rs.user_id = ? AND rs.scheduled_date = ? AND rs.status = 'pending'
    ''', (user_id, today)).fetchall()

    if not schedules:
        return  # 今天就休息吧！

    # 发送通知
    send_notification(user_id, f"今日待复习 {len(schedules)} 题，加油！")

    for s in schedules:
        send_notification(user_id, f"复习: {s['title']} (LeetCode #{s['id']})")
技术选型：定时任务可以用 Celery（Python）或 cron 作业实现。通知系统可以用 Telegram Bot API，免费且简单——十几行代码就能搞定。

课程总结

好了，到这里我们整个 LeetCode 教程的 28 章就全部结束了。让我带你们回顾一下这一路走来的旅程。

第 1-7 章：我们从最基础的数组、链表、栈、队列开始，掌握了双指针、二分查找、滑动窗口和前缀和——这些是刷题的"基本功中的基本功"。

第 8-14 章：我们深入了树和递归的世界，体验了回溯的穷举之美，领略了贪心的精妙，然后一头扎进了动态规划的海洋——从基础 DP 到背包问题、区间 DP、状态机——DP 可能是整个课程中最难但也最值得花时间的部分。

第 15-17 章：我们进入了图论的领地——从图的基础遍历到拓扑排序、并查集、最短路径和最小生成树。图论题在面试中出现的频率可能不如 DP，但一旦出现，往往是区分候选人的关键。

第 18-22 章：我们学习了许多"进阶"数据结构——堆、单调栈、单调队列、Trie、字符串匹配算法、线段树和树状数组。这些工具在你遇到"那个特定类型的问题"时，会发挥出惊人的威力。

第 23-25 章：我们转向了更实际的话题——设计类题目、多线程与并发、面试实战策略。算法只是面试的一部分，沟通能力、系统设计能力、以及对工程实践的思考同样重要。

第 26-28 章：最后三章我们聊了竞赛思维、刷题策略——以及你在读的这一章，一个综合性的刷题平台项目。

阶段 章节 核心收获
基础数据结构 1-7 数组、链表、栈、队列、双指针、二分、滑动窗口、前缀和
核心算法 8-14 树、回溯、贪心、动态规划全系列
图论与进阶 15-22 图论、高级数据结构、字符串匹配
实战与项目 23-28 设计、并发、面试、竞赛、刷题平台

后续学习路线

课程结束了，但你的学习远没有结束。以下是我推荐你继续前进的方向：

  • 持续参加周赛——每周一场 90 分钟的高强度训练，是最好的保持状态的方式
  • 系统设计——如果你已经有了一定的算法基础，系统设计是下一步需要攻克的领域
  • 开源贡献——在真实的代码库中解决问题，和刷算法题是完全不同的体验
  • 教别人——写技术博客、做分享、或者像我一样写教程。教是最好的学
最后一句真心话：刷 LeetCode 不是目的，它只是手段。真正的目的是培养你把复杂问题拆解成简单步骤的思维能力。这种能力不会因为你不刷题就消失，它会伴随你的整个编程生涯。所以不要焦虑，不要和别人比题量。按照自己的节奏，每天进步一点点。总有一天，你回头看会惊讶于自己走过了多么远的路。
