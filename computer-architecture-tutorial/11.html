第11章 分支预测技术

同学们好！上一章我们提到控制冲突是流水线处理器的核心挑战之一， 而解决控制冲突最有效的手段就是分支预测（Branch Prediction）。 这一章我们来深入探讨这个在计算机体系结构中 既古老又常新的话题——从最简单的静态预测， 到现代处理器中准确率超过98%的高级动态预测器。

分支的代价

在深入技术细节之前，先让我们量化一下分支错误的代价。 典型的程序中大约每5条指令就有一条是分支指令（约20%）。 流水线越深，一次分支预测错误的代价就越高：

  • 5级流水线：分支在EX阶段解析 → 2周期错误惩罚
  • 14级流水线（Core架构）：13周期错误惩罚
  • 31级流水线（Pentium 4 NetBurst）：约30周期错误惩罚

分支预测错误的平均代价可以用以下公式计算：

分支惩罚 = (1 - 预测准确率) × 误预测惩罚周期

假设一个14级流水线、预测准确率95%、分支占比20%： 每条指令平均等待 = 20% × 5% × 13 ≈ 0.13个周期。 这看起来很小，但在追求每一分性能的高性能处理器中， 这个损失足以让架构师们夜不能寐。 如果我们将准确率从95%提升到97%， 平均等待降至20% × 3% × 13 ≈ 0.078个周期—— 减少了40%的损失！ 这就是为什么哪怕提升1%的分支预测准确率， 对处理器整体性能都有显著影响。

静态分支预测

静态分支预测是最简单的实现方式—— 不依赖任何运行时信息， 纯粹根据分支指令本身或编译器提供的提示来猜测方向。

  • Always Not-Taken（总是猜不跳转）： 流水线默认顺序执行。如果分支实际跳转，冲刷已取入的指令。 实现最简单，但对向后分支（循环）准确率极低， 总体约40%。
  • Always Taken（总是猜跳转）： 猜目标地址取指。对循环准确率高， 但对前向分支（if语句中的else分支）效果差。 总体约60%。
  • BTFN（Backward Taken, Forward Not-Taken）： 综合策略——向后分支猜跳转（适合循环）， 向前分支猜不跳转（适合条件语句）。 总体准确率约65-80%。
  • Profile-Guided（编译器辅助）： 编译器通过profile信息在指令中插入静态预测位。 IA-64架构的ld.s（投机加载）和分支提示指令 就是这一思路的体现。

静态预测的优点是完全不消耗硬件资源 （不查表、不更新状态）， 但缺点也很明显：对于数据相关的分支 （如 if (x > y)）， 静态预测完全无能为力—— 因为编译器在编译时根本无法知道x和y的值。

动态分支预测：1-bit与2-bit饱和计数器

动态预测器维护一个分支历史表 （Branch History Table, BHT）， 以分支指令地址的低位为索引， 记录每个分支的历史行为。

1-bit预测器： 每个表项只有一个位，记录上一次该分支是否跳转。 下次预测沿用上一次的结果。 问题在于：内层循环的最后一次迭代会错误预测 （循环退出时预测为跳转，实际不跳转）， 下一次循环开始又错误预测一次。 两次误预测让准确率在高频循环中下降明显。 例如一个10次迭代的循环，每次进入和退出都错两次， 共20次误预测，而总预测次数是100次（10次×10个外层循环）， 准确率仅80%。

2-bit饱和计数器（Saturating Counter）： 每个表项有2位，表示四种状态： Strongly Taken（11）、Weakly Taken（10）、 Weakly Not-Taken（01）、Strongly Not-Taken（00）。 状态转移规则是：分支实际跳转则计数器+1 （最高到11），不跳转则计数器-1（最低到00）。 预测规则是：高位为1（10, 11）猜跳转， 高位为0（00, 01）猜不跳转。

2-bit预测器的优势在于： 需要连续两次预测错误才会改变预测方向。 这消除了1-bit预测器在循环退出/进入时的"双误预测"问题—— 循环执行N次，只有最后一次退出（猜跳转→实际不跳转） 和第一次进入（猜不跳转→实际跳转）各一次误预测， 一共2次，而不是2N次。 对于10次迭代的循环，100次预测中只有2次错误， 准确率高达98%——当然，这是对特定模式而言。

下面是2-bit预测器的C语言模拟：

#include <stdio.h>
#include <stdint.h>

// 2-bit饱和计数器分支预测器模拟
#define BHT_SIZE 4096  // 4K表项

typedef struct {
    uint8_t counter[2];  // [1]: strong/weak, [0]: taken/not-taken
} saturating_2bit;

uint8_t predict(saturating_2bit *entry) {
    // 高位为1 → 预测跳转(taken)
    return (entry->counter[1]) ? 1 : 0;
}

void update(saturating_2bit *entry, uint8_t actual_taken) {
    if (actual_taken) {
        if (entry->counter[0] != 1 || entry->counter[1] != 1)
            // 增加计数器 (上限: 11)
            if (++entry->counter[0] == 0) entry->counter[1]++;
    } else {
        if (entry->counter[0] != 0 || entry->counter[1] != 0)
            // 减少计数器 (下限: 00)
            if (entry->counter[0]-- == 0) entry->counter[1]--;
    }
}

int main() {
    saturating_2bit bht[BHT_SIZE] = {0};
    uint64_t branch_pc;
    uint8_t actual;
    int correct = 0, total = 0;

    // 模拟: 循环10次的内层循环，外层重复100次
    for (int outer = 0; outer < 100; outer++) {
        for (int inner = 0; inner < 10; inner++) {
            branch_pc = (uint64_t)&inner;  // 用地址低位索引BHT
            uint32_t index = branch_pc & (BHT_SIZE - 1);
            actual = (inner < 9) ? 1 : 0;  // 前9次跳转,最后一次不跳转

            if (predict(&bht[index]) == actual) correct++;
            total++;
            update(&bht[index], actual);
        }
    }
    printf("2-bit预测器准确率: %.2f%%\n", 100.0 * correct / total);
    printf("总预测次数: %d, 正确: %d, 错误: %d\n",
           total, correct, total - correct);
    return 0;
}

1995年的Pentium MMX处理器首次使用了 2-bit饱和计数器预测器，配备4K表项的BHT。 在那之前，Pentium使用的是1-bit预测器。 切换到2-bit后，分支预测准确率从约80%提升到了约90%， 这在当时是一个巨大的进步。 虽然现在看来90%的准确率平平无奇， 但在1995年这已经是一个重要的突破。

全局与局部历史预测器

2-bit饱和计数器虽然好， 但对模式化的分支行为（如交替跳转、不跳转的模式）表现不佳。 假设一个分支的模式是"T, NT, T, NT, T, NT"—— 2-bit预测器每次都会误预测！ 因为每次实际结果都与上次相反， 计数器始终在两个"weak"状态之间摇摆。

为了解决这个问题， 计算机架构师们提出了更高级的预测方案：

  • 局部历史预测器（Local History Predictor）： 每个分支维护一个单独的n位历史移位寄存器 （PHT, Pattern History Table）。 用该分支最近N次的结果来索引一个2-bit饱和计数器表。 这样"T, NT, T, NT"这样的模式可以被学习。
  • 全局历史预测器（Global History Predictor）： 所有分支共享一个全局历史寄存器（GHR）， 记录最近N次分支的结果。用GHR的值来索引BHT。 不同分支的上下文可以通过GHR共享信息， 例如一个if语句内部的分支结果往往与外部if的结果相关。
  • Gshare预测器： 将GHR与分支PC进行XOR混洗后作为索引。 XOR操作将全局历史和局部分支地址结合， 减少了不同分支间的别名冲突（aliasing）。 Gshare的准确率比纯全局预测器高5-10%。
  • Tournament/Hybrid预测器： 同时运行多个预测器 （如一个局部预测器和一个全局预测器）， 然后用一个"选择器"来动态决定当前分支更相信哪一个。 Alpha 21264（1998年）率先使用了这种架构—— 它的选择器本身也是一个2-bit饱和计数器！
  • TAGE（TAgged GEometric）预测器： 现代处理器的标准配置。 使用多个不同长度的全局历史表 （几何级增长：2, 4, 8, 16, 32, 64, 128位...）， 每个表项带有一个标签（tag）来减少别名。 启用最长的可用历史进行预测。 TAGE的准确率可以超过98%。
现代分支预测器的准确率可以超过98%—— 但剩下的2%仍然会造成大量的流水线冲刷。 在一个20级流水线的处理器中， 2%的误预测率意味着每50条指令就要冲刷一次流水线， 损失约20×2% = 0.4 CPI。 这也是为什么分支预测至今仍是活跃的研究领域 ——最后那2%的难度远超前面98%。

分支目标缓冲（BTB）

预测方向只是问题的一半——我们还需要知道 跳到哪里去。 分支目标缓冲（Branch Target Buffer, BTB） 就是解决这个问题的结构。 BTB以分支指令的PC为索引，缓存对应的目标地址。 当预测器判断某个分支会跳转时， 直接从BTB读出目标地址，下一个周期从目标地址取指。

对于函数调用和返回，还有一个专用的结构： 返回地址栈（Return Address Stack, RAS）。 RAS是一个硬件维护的栈结构： 遇到函数调用指令（jal/jalr）时将返回地址压栈， 遇到返回指令（ret）时从栈顶弹出预测地址。 RAS的准确率极高，通常在99.9%以上， 因为函数调用的嵌套在绝大部分情况下是严格匹配的。 唯一的例外是setjmp/longjmp这样的非常规返回—— 它们会破坏RAS的栈结构，导致误预测。

2004年我参与一个RISC-V核的项目时， 最初只实现了2-bit预测器。 在CoreMark基准测试上性能看起来不错， 但切换到数据库类负载（TPC-H）时，性能暴跌了25%。 分析发现数据库的查询处理中有大量"不规律"的分支模式—— 外键比较、空值检查等，2-bit预测器完全应付不来。 升级到Gshare预测器后，性能回升了18%。 这个经历让我深刻认识到： 分支预测器的设计需要充分理解目标工作负载的模式， 没有一个"放之四海而皆准"的最优方案。

分支预测技术让处理器能够"猜测"未来指令的流向， 大幅减少了控制冲突带来的流水线停顿。 但仅仅处理好分支还不够—— 当一个指令执行过程中发生异常（如除零、页错误）， 处理器如何在不丢失程序状态的前提下妥善处理？ 下一章我们进入异常与中断处理的世界。
