第15章 Vivado工程实战全流程

前14章我们一直在写 Verilog 代码和做仿真——通过前几章的学习，你应该已经掌握了基础语法、组合/时序逻辑、状态机、通信接口甚至跨时钟域处理。但写代码只是 FPGA 开发的一半，另一半是把代码变成比特流、下载到芯片上让它跑起来。从这一章开始，我们将进入"工程实战"阶段——用真实的工具链完成完整的开发流程。

本章以 AMD Xilinx 的 Vivado 为主线，带你走一遍从新建工程到生成比特流的完整流程——包括综合、实现、约束编写和片上调试。Vivado 是目前 FPGA 行业使用最广泛的开发工具，掌握它是 FPGA 工程师的必备技能。

15.1 Vivado 工具概述

Vivado 是 Xilinx（2022 年被 AMD 收购后正式更名为 AMD Adaptive Computing）于 2012 年推出的新一代设计套件，取代了老旧的 ISE Design Suite。ISE 的时代从 1990 年代持续到 2013 年左右，它支持的最后一款芯片是 Spartan-6 和 Virtex-6。从 7 系列（Artix-7、Kintex-7、Virtex-7）开始，所有 Xilinx FPGA 都必须使用 Vivado。

Vivado 经历了多个大版本的演进：

版本系列发布时间关键变化
Vivado 2012.x - 2013.x2012-2013初始版本，支持 7 系列，替代 ISE
Vivado 2014.x - 2015.x2014-2015引入 IP Integrator（Block Design）、HLS 集成
Vivado 2016.x - 2018.x2016-2018Vivado ML（Machine Learning）版本推出，综合/布局布线算法大幅优化
Vivado 2019.x - 2020.x2019-2020Vitis 统一软件平台发布，Vivado 回归专注硬件开发
Vivado 2021.x - 2022.x2021-2022支持 Versal ACAP 系列，改善编译时间
Vivado 2023.x - 2024.x2023-2024XRT 深度集成，AI Engine 开发支持
Vivado 2025.x2025Linux 免费版（WebPACK）仍可用；支持 Ubuntu 22.04/24.04
Vivado 2026.12026-05取消 Linux 免费版授权！仅 Windows 保留 WebPACK；UI 重构

Vivado 提供两个授权层级：WebPACK（免费，支持 Artix-7、Kintex-7 等主流中低端器件）和 Design Edition（付费，支持全系列器件，包括 Virtex、Versal 高端系列）。对于本教程的学习路线，WebPACK 完全够用，因为我们的目标平台是 Artix-7（Basys3 开发板）。

一个 Vivado 工程涉及多种文件类型，理解它们的用途是避免混乱的第一步：

扩展名文件类型说明
.xpr工程文件Vivado 工程定义，XML 格式，包含源文件引用、器件型号、约束等配置
.xdc约束文件Xilinx Design Constraints，引脚分配 + 时序约束 + 调试约束
.bdBlock DesignIP Integrator 的图形化设计文件，描述 IP 核之间的连线
.wcfg波形配置仿真波形显示配置，保存了信号分组、颜色、进制等显示设置
.bit比特流文件生成的 FPGA 配置文件，通过下载器烧录到芯片 SRAM 中
.mcsSPI Flash 文件用于烧录到外部 SPI Flash（掉电不丢失），FPGA 上电自动加载
.ltx调试核文件ILA/VIO 等调试 IP 核的探针连接信息，下载比特流时需一并加载

15.2 工程创建到综合

先走一遍标准的工程创建流程。假设我们的目标是 Basys3 开发板（搭载 Xilinx Artix-7 XC7A35T），做一个 8 位 LED 流水灯 + 按键控制方向的小项目：

第一步：创建工程。在 Vivado 中点击 Quick Start → Create Project，选择工程名称和路径。下一步选择 RTL Project（不勾选 Do not specify sources at this time，这样后面可以立即添加源文件）。器件选择部分，可以按开发板搜索——Basys3 对应的器件是 xc7a35tcpg236-1。如果你使用的是自己画的板子，按实际型号搜索即可。

第二步：添加源文件。点击 Add Sources，选择我们写好的 Verilog 文件（或者在这个步骤中直接创建新的）。推荐选择 Copy sources into project——这样工程文件是独立副本，不会因为源文件移动而损坏工程。

第三步：添加约束文件。Vivado 会让你在创建向导中添加 XDC 文件。也可以之后在 Flow Navigator 的 Constraints → Add Sources 中添加。约束文件的内容我们 15.5 节会详细展开。

第四步：RTL Analysis。工程创建完成后，在 Flow Navigator 的 Synthesis → Run Synthesis 之前，建议先跑 RTL Analysis → Open Elaborated Design。这一步会把你的 Verilog 源码解析成 RTL 级电路结构，你可以查看 Schematic 视图看到综合前的逻辑结构——包括寄存器、组合逻辑、状态机等。这是检查"我写的代码到底描述了个什么电路"的最直观方式。

第五步：Synthesis。点击 Run Synthesis，Vivado 会将 RTL 代码映射到目标器件的 LUT、FF、DSP slice、BRAM 等资源上。综合完成后，打开 Synthesis → Open Synthesized Design，可以查看：资源利用率报告（LUT/FF/BRAM/DSP 用了多少）、时序预估报告（综合后工具会基于线负载模型估算延迟）、Schematic 视图（现在看到的是映射到 LUT 级别后的网表）。

快捷操作：Vivado 的 Tcl Console 非常强大。你完全可以用一行命令完成综合：synth_design -top top -part xc7a35tcpg236-1。掌握 Tcl 模式可以让你的工程完全可复现——这也是第 18 章会详细讨论的主题。

15.3 布局布线与比特流生成

综合完成后，就进入了 Implementation（实现）阶段。实现分为三个步骤：

opt_design（优化）：将综合后的网表进行逻辑优化——移除冗余逻辑、常量传播、扇出优化等。对于小型设计这一步通常很快，但对于百万门级设计，这一步能显著改善后续布局布线的质量。

place_design（布局）：将逻辑单元（LUT、FF、BRAM、DSP）分配到芯片上具体的物理位置上。布局质量直接决定了时序能否收敛——如果关键路径上的两个寄存器被工具放到了芯片的两端，布线延迟就会很大。Vivado 的布局器使用模拟退火算法，会在运行过程中多次迭代优化。

route_design（布线）：确定各逻辑单元之间的走线路径。Vivado 内部使用了大量不同的布线资源：本地连线（switch box）、通用连线、长线、全局时钟网络等。布线器需要一个合法且最优的解——它不仅要连通所有信号，还要满足时序约束中的建立时间和保持时间要求。

实现完成后，打开 Implementation → Open Implemented Design，可以查看 Device 视图——这是最酷的部分！你可以看到自己的逻辑在 FPGA 芯片上占据的物理位置：SLICE 被标为蓝色，BRAM 是矩形块，DSP48 是特殊的列结构。对于调试时序问题和拥塞问题非常有帮助。

最后一步是 Generate Bitstream。Vivado 会依次调用 write_bitstream 命令生成 .bit 文件，并自动调用 Open Hardware Manager 准备下载。连接好下载器（如 Digilent JTAG-HS3），选择对应的 FPGA 器件，点击 Program Device，你的设计就运行在真实的芯片上了。

15.4 ILA 片上逻辑分析仪

仿真再充分，真实硬件上的行为也可能与仿真不同——时钟抖动、电源噪声、温度漂移、信号毛刺，这些在仿真中都难以建模。这时就需要 ILA（Integrated Logic Analyzer）——Vivado 内部的片上逻辑分析仪。

ILA 的使用流程如下：

第一步：例化 ILA IP 核。在 IP Catalog 中搜索 "ILA"，双击配置。你需要设置：探针数量（Probe Count）、每个探针的位宽（Probe Width）、采样深度（Sample Depth，可选 1024/4096/16384 等）。采样深度决定了你能捕获多少个时钟周期的数据——深度越大，占用的 BRAM 资源越多。

第二步：在 RTL 中连接 ILA。在顶层模块中例化 ILA，把要观察的内部信号连接到探针输入上。例如：

// 在顶层模块中例化 ILA
ila_0 u_ila (
    .clk(clk_100m),
    .probe0(led_reg),       // 8-bit LED 寄存器
    .probe1(btn_debounced), // 按键信号
    .probe2(counter),       // 计数器
    .probe3(state)          // FSM 状态
);

第三步：设置触发条件。在 Hardware Manager 中打开 ILA 面板，可以在触发条件窗口中设置复杂的触发条件——比如 probe2 == 1000 && probe3 == 2'b01。也可以设置多个触发级（触发序列），类似于传统逻辑分析仪的触发链。

第四步：抓取波形并分析。点击运行触发（Run Trigger），等待条件满足后 ILA 会停止采集，波形窗口显示捕获的数据。你可以放大缩小、添加光标测量、导出为 VCD 文件。

调试技巧：ILA 的采样深度是有限的。一个常见策略是用更浅的深度（1024）先验证触发条件是否正确，确认后再加大深度（16384）捕获长序列。另外，不要把 ILA 留在最终比特流中——它会消耗 BRAM 和布线资源。调试完成后应移除 ILA 例化，重新生成生产版本的比特流。

15.5 约束文件（XDC）

XDC（Xilinx Design Constraints）是 Vivado 的统一约束格式，取代了 ISE 时代的 UCF。它基于 Tcl 语法，比 UCF 更灵活、更强大。约束主要分三类：

引脚约束：将顶层模块的 I/O 端口映射到芯片的物理引脚上，同时指定 I/O 标准。下面是一个完整的 Basys3 板 XDC 约束文件：

## ============================
## Basys3 XDC Constraint File
## 器件：xc7a35tcpg236-1
## 适用：8-bit LED + 按键输入
## ============================

## ---- 时钟 ----
set_property PACKAGE_PIN W5 [get_ports clk]
set_property IOSTANDARD LVCMOS33 [get_ports clk]
create_clock -period 10.000 -name sys_clk [get_ports clk]

## ---- 8 位 LED（共阳极接法） ----
set_property PACKAGE_PIN U16 [get_ports {led[0]}]
set_property PACKAGE_PIN E19 [get_ports {led[1]}]
set_property PACKAGE_PIN U19 [get_ports {led[2]}]
set_property PACKAGE_PIN V19 [get_ports {led[3]}]
set_property PACKAGE_PIN W18 [get_ports {led[4]}]
set_property PACKAGE_PIN U15 [get_ports {led[5]}]
set_property PACKAGE_PIN U14 [get_ports {led[6]}]
set_property PACKAGE_PIN V14 [get_ports {led[7]}]
set_property IOSTANDARD LVCMOS33 [get_ports {led[*]}]

## ---- 按键输入（低电平有效，内部上拉已启用） ----
set_property PACKAGE_PIN U18 [get_ports btn_rst]  ## 中央复位键
set_property PACKAGE_PIN V17 [get_ports btn_up]    ## 向上键（方向切换）
set_property IOSTANDARD LVCMOS33 [get_ports {btn_*}]

## ---- 未使用引脚配置 ----
set_property BITSTREAM.CONFIG.UNUSEDPIN Pulldown [current_design]
set_property CONFIG_MODE SPI [current_design]
踩坑实录——XDC 注释的陷阱：我刚接触 Vivado 时，从 Digilent 官网下载了 Basys3 的 master XDC 模板文件，打开一看，所有的 PACKAGE_PIN 行前面都有 ## 注释符号。我想当然地以为删掉 ## 就能用，但忽略了这份模板文件里每个引脚后面还写了多个备选 IOSTANDARD 配置，注释之间的格式相互依赖。结果我修修补补，觉得 LED 引脚的约束已经改好了，下载比特流后 LED 死活不亮。花了三个多小时一根一根查线，最后用 report_io 命令一看——IOSTANDARD 属性根本没设置，工具默认用了 LVTTL，但 Basys3 的 LED 需要 LVCMOS33 驱动。这个教训告诉我：永远不要信任模板文件的注释格式——一定要用 report_io -file io.rpt 确认约束生效。

时序约束：除了上面例子中的 create_clock，还要考虑输入延迟和输出延迟约束（set_input_delay / set_output_delay）——用于约束芯片引脚上的信号相对于时钟的有效窗口。对于初学者，至少要把所有时钟用 create_clock 定义清楚。对于板级接口（如外部 RAM、ADC），则需要更精确的 I/O 延迟约束。

调试约束：ILA 等调试 IP 的探针连接信息存储在 .ltx 文件中。在 Hardware Manager 中，需要同时加载 .bit 和 .ltx 文件才能看到信号名称。

15.6 2026 年的重要变化

2026 年 5 月，AMD 发布了 Vivado 2026.1，带来了一个让 Linux 用户社区震动的消息——Linux 上的免费 WebPACK 授权被取消。这意味着：如果你想在 Linux 上使用 Vivado，必须购买 Design Edition 授权（每年数千美元）。Windows 版不受影响，仍提供免费 WebPACK。

这一变化的影响是深远的。FPGA 行业的大量开发者和研究人员使用 Linux 作为主力开发环境——服务器集群、CI/CD、云端 FPGA 实例（如 AWS F1 实例）基本都是 Linux。AMD 的这一决策实质上把 Linux 上的 FPGA 开发推向了两条替代路线：

路线一：开源工具链——Yosys + nextpnr + openXC7。对于 7 系列器件（Artix-7、Kintex-7），openXC7 数据库已经提供了生产级别的支持，时序精度与 Vivado 的差距在 5% 以内。路线二：Windows 虚拟机——在 Linux 上运行 Windows 虚拟机安装 Vivado WebPACK，通过 USB 直通连接下载器。但这对于 CI/CD 场景基本不可行。

对于初学者和学习本教程的读者，我的建议是：如果你用 Windows，继续用 Vivado WebPACK 就好，不受任何影响。如果你用 Linux，不妨借此机会接触开源工具链——第 24 章我们会有专门的章节深入讲解。无论如何，Vivado 作为行业标准的地位短期内不会改变。
