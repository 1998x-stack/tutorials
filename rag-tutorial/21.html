第二十一章：Agentic RAG

大家好，我是你们的讲师。在之前的章节中，我们构建的RAG系统都遵循一个固定的流程：用户提问→检索文档→生成回答。这个流程虽然高效，但本质上是被动的——系统等在那里，收到一个问题就执行一次固定的操作。但如果你观察一个真正聪明的信息工作者怎么做事，你会发现他们不会"一条路走到黑"，而是：先查一点资料，发现信息不够，换一个角度再查，把不同来源的信息拼在一起，如果还不够就拆解成子问题分别查找，最终综合出一个答案。

让我分享一个让我印象深刻的教训。我们曾经为一个法律咨询团队构建RAG系统。律师们提的问题极其复杂——例如"根据过去五年类似案例的判决趋势，我们在这个合同纠纷中应该采取什么策略？" 传统的RAG流程只能回答"判决趋势是什么"或"类似案例有哪些"，但无法将这两个问题关联起来进行策略分析。我们花了大量时间优化检索和生成，效果始终不理想。直到我们转向了Agentic RAG——让系统自主规划：先检索判决趋势，再检索类似案例，然后综合分析，最后给出策略建议。回答了之后，律师追问"为什么是这个策略？"——系统又可以再检索相关的法条和判例来支撑自己的推理。这种"多步推理+工具调用+自我反思"的能力，正是Agentic RAG的核心价值。

11.1 从被动检索到主动编排

Agentic RAG的核心思想是：将RAG系统从一个"问答机器"升级为一个"信息工作代理"。这个代理拥有以下能力：

任务规划（Planning）： 面对一个复杂查询，Agent能够将其分解为多个子任务，并确定子任务的执行顺序。

工具使用（Tool Use）： Agent能够调用不同的工具——向量检索、图谱查询、网页搜索、计算器、代码执行——来完成不同的子任务。

自我反思（Reflection）： Agent在执行过程中能够评估自己的中间结果，判断是否需要补充信息或调整策略。

多步记忆（Memory）： Agent能够记住之前的检索结果和推理过程，在多轮交互中保持一致。

这种从"单次检索→生成"到"多步规划→工具调用→反思→迭代"的转变，是RAG架构的一次质的飞跃。

11.2 LangGraph：计划-执行-评估循环

LangGraph是LangChain团队推出的一个用于构建有状态代理工作流的框架。它通过定义节点（Nodes）和边（Edges）来构建一个有向图，Agent在图中按照条件执行路径。以下是一个使用LangGraph构建Agentic RAG的示例：

from typing import TypedDict, List, Literal
from langgraph.graph import StateGraph, END
from langchain_core.messages import HumanMessage, AIMessage

# 定义Agent的状态
class AgentState(TypedDict):
    messages: List  # 对话历史
    question: str   # 原始问题
    sub_queries: List[str]  # 分解后的子问题
    retrieved_docs: dict    # 子问题→检索结果
    current_step: str       # 当前执行步骤
    final_answer: str       # 最终答案

# 定义节点函数
def analyze_question(state: AgentState) -> AgentState:
    """第一步：分析问题，分解为子问题"""
    question = state["question"]
    # 使用LLM将问题分解为子问题
    sub_queries = decompose_question(question)
    state["sub_queries"] = sub_queries
    state["current_step"] = "retrieve"
    return state

def retrieve_information(state: AgentState) -> AgentState:
    """第二步：对每个子问题执行检索"""
    retrieved = {}
    for sq in state["sub_queries"]:
        # 选择最合适的检索策略
        if needs_vector_search(sq):
            docs = vector_search(sq, top_k=3)
        elif needs_web_search(sq):
            docs = web_search(sq)
        else:
            docs = kg_query(sq)  # 知识图谱查询
        retrieved[sq] = docs
    state["retrieved_docs"] = retrieved
    state["current_step"] = "evaluate"
    return state

def evaluate_results(state: AgentState) -> AgentState:
    """第三步：评估检索结果是否足够"""
    for sq, docs in state["retrieved_docs"].items():
        if not is_sufficient(docs):
            # 如果某个子问题的检索结果不足，
            # 重新分解或补充检索
            state["sub_queries"].append(
                generate_refined_query(sq, docs)
            )
            state["current_step"] = "retrieve"
            return state
    state["current_step"] = "generate"
    return state

def generate_answer(state: AgentState) -> AgentState:
    """第四步：综合所有信息生成最终答案"""
    context = assemble_context(state["retrieved_docs"])
    answer = llm_generate(state["question"], context)
    state["final_answer"] = answer
    state["current_step"] = "end"
    return state

# 构建图工作流
workflow = StateGraph(AgentState)

# 添加节点
workflow.add_node("analyze", analyze_question)
workflow.add_node("retrieve", retrieve_information)
workflow.add_node("evaluate", evaluate_results)
workflow.add_node("generate", generate_answer)

# 定义条件边
def router(state: AgentState) -> Literal["retrieve", "generate", "end"]:
    if state["current_step"] == "retrieve":
        return "retrieve"
    elif state["current_step"] == "generate":
        return "generate"
    return "end"

# 设置入口和边
workflow.set_entry_point("analyze")
workflow.add_conditional_edges("analyze", router)
workflow.add_conditional_edges("retrieve", router)
workflow.add_conditional_edges("evaluate", router)
workflow.add_edge("generate", END)

# 编译并运行
app = workflow.compile()
result = app.invoke({
    "question": "过去五年GPU价格趋势如何？",
    "messages": [],
    "sub_queries": [],
    "retrieved_docs": {},
    "current_step": "start",
    "final_answer": ""
})

print(f"最终答案: {result['final_answer']}")

11.3 ReAct模式与NVIDIA Blueprint

ReAct（Reasoning + Acting）是Agentic系统的核心设计模式。它让模型在"思考"和"行动"之间交替进行：先推理当前状态（Thought），决定需要做什么（Action），观察结果（Observation），然后继续推理。这个循环一直持续到问题被解决。

NVIDIA在2025年发布的Agentic RAG Blueprint v2.6将这一模式系统化，提出了一套完整的参考架构。该蓝图的核心组件包括：

  • Orchestrator（编排器）： 负责任务规划、状态管理和执行调度。
  • Tool Registry（工具注册表）： 集中管理所有可用的检索和计算工具，包括向量检索、知识图谱、SQL查询、网页搜索、代码解释器等。
  • Memory Store（记忆存储）： 保存Agent的中间推理结果，支持短期记忆（当前会话）和长期记忆（跨会话）。
  • Guardrails（安全护栏）： 在Agent执行的每一步进行安全检查，防止工具被滥用或信息泄露。

BluePrint v2.6的一个关键优化是"Token预算管理"——限制Agent在单个问题上的最大推理步数（默认6步）和总Token消耗。这有效防止了Agent在复杂问题上无限制地循环。

11.4 传统RAG vs Agentic RAG

对比维度 传统RAG Agentic RAG
流程控制 线性固定流程 动态编排，条件分支
检索次数 单次检索 多次检索（通常2-6次）
工具多样性 单一向量检索 多工具组合（向量/图谱/Web/计算）
推理能力 隐式推理（在生成中） 显式推理（思考链+反思）
任务分解 无 自动分解复杂查询
错误恢复 无（一次失败即结束） 支持自我修正和重试
端到端延迟 低（1-3秒） 高（5-30秒）
Token消耗 低 高（2-10倍）
适用场景 简单事实型问答 复杂分析、多步推理

11.5 何时使用（以及何时不要使用）Agentic模式

架构决策：为简单查询请使用传统RAG Agentic RAG虽然强大，但不是银弹。基于2025-2026年行业的实践数据，以下是决策指南：

应该使用Agentic RAG的场景：
- 需要多步推理的复杂查询（如"分析竞争对手去年的产品策略"）
- 需要从多个异构数据源综合信息（如"对比财报数据和市场分析报告"）
- 需要与外部工具交互（如"查询今天的股票价格并分析趋势"）
- 查询本身需要反复迭代和细化（如"帮我设计一个实验方案"）

不应该使用Agentic RAG的场景：
- 80%以上的用户查询是简单的知识问答（如"公司的年假政策是什么？"）
- 对延迟要求极高（P99 < 3秒）的在线系统
- Token成本敏感的应用（Agentic RAG的Token消耗是传统RAG的3-10倍）
- 需要高度确定性的场景（Agent的随机执行路径可能导致不可预测的结果）

最佳实践是使用一个"查询分类器"作为前置路由：简单问题走近路（传统RAG），复杂问题走远路（Agentic RAG）。这种混合架构通常能兼顾效率和能力。

11.6 工具设计模式

在Agentic RAG中，工具的设计质量直接影响Agent的执行效果。以下是一些经过验证的工具设计模式：

工具签名清晰： 每个工具的名称和描述应该让LLM一眼就能理解其用途和适用范围。模糊的描述会导致Agent选错工具。

幂等性： 工具应该设计为可重复调用而不产生副作用。如果Agent因为网络问题重试了一个工具，不应该造成数据重复或状态混乱。

错误消息结构： 工具失败时，返回的结构化错误消息应该包含足够的信息让Agent决定下一步行动（重试、换工具、还是跳过）。简单的报错会打断Agent的推理链。

带超时的执行： 每个工具调用必须设置超时时间。一个卡住的工具调用不应该阻塞整个Agent的执行。

11.7 小结与下章预告

Agentic RAG代表了RAG架构从"被动查询"到"主动代理"的范式转变。我们学习了LangGraph构建计划-执行-评估循环的方法、ReAct模式的工作机制、以及NVIDIA Agentic RAG Blueprint的参考架构。核心原则是：不是所有查询都需要Agent，也不是所有查询都能被传统RAG满足。使用智能路由来平衡能力和效率是成功的关键。完成了Agentic架构的探索之后，下一章我们将回到数据基础设施层面的一个重要话题——高级索引架构。你将学到如何构建层级索引、自校正索引和多索引共存的架构来支撑大规模RAG系统的性能需求。
