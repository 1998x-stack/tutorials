第22章 虚拟化与Overlay网络

大家好，欢迎来到第22章。过去四章我们都在聊物理层的事情——TSN让以太网准时，汽车以太网把以太网塞进车里，SPE把以太网送到一公里外的传感器。现在，让我们完全转换视角，从物理世界跳转到虚拟世界。

如果你在2010年之前管理过一个数据中心，你面对的场景是这样的：每个应用跑在一台独立的物理服务器上，服务器的网线插到一台物理交换机上，VLAN用来隔离不同的业务。网络是"硬"的——一根线、一个端口、一个VLAN，清清楚楚。但到了2010年代中期，虚拟化技术（VMware vSphere、KVM、Hyper-V）彻底改变了这个画面。一台物理服务器上可能跑着几十台虚拟机（VM），每台VM有自己的MAC地址和IP地址。更关键的是，这些VM可以在物理服务器之间自由迁移（vMotion），而迁移过程中VM的MAC地址和IP地址不能改变——否则正在进行的TCP连接就断了。

问题来了：如果VM从服务器A迁移到了服务器B，而服务器A和服务器B不在同一个二层网络（即它们之间隔着一个三层路由器），VM的MAC地址在路由器那边就"失效"了——路由器不会转发目标MAC为原始VM的帧，因为MAC不在本地广播域中。这个看似简单的"VM迁移后网络不通"的问题，催生了Overlay网络技术的大爆发。

22.1 VXLAN：将二层网络延伸到三层之上

VXLAN（Virtual eXtensible LAN）是解决上述问题最成功的Overlay技术。它的核心思想可以用一句话概括：把以太网帧封装在UDP数据包里，通过三层的IP网络传输。这样一来，物理网络基础设施可以是一个三层IP网络（路由器+IP转发），而虚拟网络呈现给VM的仍然是一个二层以太网段。

VXLAN由VMware和Cisco等公司在2014年联合提出，后来由IETF标准化为RFC 7348。它的几个关键概念如下：

VNI（VXLAN Network Identifier）：类似VLAN ID，但VLAN ID只有12位（4096个），而VNI有24位（约1600万个）。这意味着在多租户云环境中，每个租户都可以有自己独立的虚拟网络，完全不用担心ID冲突。

VTEP（VXLAN Tunnel Endpoint）：VXLAN隧道的端点，可以是物理交换机、虚拟交换机（如Open vSwitch）、或服务器上的软件VTEP。VTEP负责封装和解封装VXLAN报文。

一个VXLAN数据包的封装格式如下：

  +-------------------------------+
  | 外层MAC头 (14字节)            |  — 目的MAC: 下一跳或目标VTEP的MAC
  +-------------------------------+
  | 外层IP头 (20字节)             |  — 目的IP: 目标VTEP的IP
  +-------------------------------+
  | 外层UDP头 (8字节)             |  — 目的端口: 4789 (VXLAN)
  +-------------------------------+
  | VXLAN头 (8字节)               |  — VNI (24位) + 标志位
  |   + VNI: 24位虚拟网络标识符   |
  +-------------------------------+
  | 原始以太网帧 (内层)            |  — VM到VM的完整以太网帧
  |   + 内层MAC头                 |
  |   + 内层IP头 (VM的IP)         |
  |   + 载荷 (TCP/UDP等)          |
  +-------------------------------+

从VM的角度看，它完全不知道VXLAN的存在。VM-A发送一个标准的以太网帧到VM-B：源MAC是VM-A，目的MAC是VM-B，中间包含IP/TCP payload。这个帧先到达服务器上的虚拟交换机（vSwitch）。如果vSwitch发现目的MAC对应的VM在远端服务器上，它就把这个原始以太网帧交给VTEP处理。VTEP加上VXLAN头（包含VNI）、UDP头（端口4789）、IP头（源IP=本地VTEP，目的IP=远端VTEP）、以及新的MAC头。然后这个外层IP包通过物理网络路由到远端VTEP，远端VTEP解封装后把原始以太网帧交给VM-B。

整个过程对VM完全透明——VM以为自己还在同一个二层广播域里，实际上它的数据包已经在三层的IP网络上"跑了个来回"。

22.2 EVPN：BGP接管VXLAN的控制面

VXLAN解决了数据平面的封装问题，但它自身没有定义控制面——也就是VTEP如何知道"某个MAC地址对应哪个远端VTEP"。早期VXLAN依赖泛洪和学习（flood-and-learn）来发现MAC-to-VTEP的映射关系，类似传统以太网交换机的MAC学习方式。但这种方式在多播受限的网络中效果很差（因为需要组播来传播广播帧），而且在大规模数据中心中收敛速度慢。

EVPN（Ethernet VPN）正是为了解决这个痛点而生的。它使用BGP（边界网关协议）作为VXLAN的控制平面，通过MP-BGP（多协议BGP）来通告MAC地址、IP地址和VNI之间的映射关系。

EVPN的核心是MP-BGP EVPN NLRI（Network Layer Reachability Information）——这是一组新的BGP地址族，专门用于承载以太网相关的路由信息。EVPN定义了多种路由类型，其中最重要的是：

路由类型 名称 作用
Type 2 MAC/IP Advertisement Route 通告MAC地址和关联的IP地址，以及所属的VNI。这是最常用的路由类型。
Type 3 Inclusive Multicast Ethernet Tag Route (IMET) 通告VTEP的组播成员信息，用于BUM流量（广播、未知单播、组播）的传输。
Type 5 IP Prefix Route 通告IP前缀（不绑定MAC），用于L3VPN场景，让EVPN承载路由。

Type 2路由是EVPN中最核心的路由类型。当VM-A上线时，它连接的VTEP会学习到VM-A的MAC和IP地址，然后通过BGP向全网通告：

BGP EVPN Type-2 Route 示例:
+------------------------------+
| Route Distinguisher (RD)     |  — 区分不同租户/VRF
+------------------------------+
| Ethernet Segment Identifier  |  — 多归接入标识
+------------------------------+
| Ethernet Tag ID              |  — VLAN ID或VNI
+------------------------------+
| MAC Address (48 bits)        |  — 00:1A:2B:3C:4D:5E
+------------------------------+
| IP Address (32 or 128 bits)  |  — 192.168.1.100
+------------------------------+
| MPLS Label1 / VNI            |  — VXLAN VNI = 10001
+------------------------------+
| MPLS Label2 (可选)           |  — 用于服务映射
+------------------------------+

当其他VTEP收到这条Type 2路由后，它们就知道：要到达MAC 00:1A:2B:3C:4D:5E（IP 192.168.1.100），需要通过VNI 10001封装数据包，外层IP指向通告这条路由的VTEP。这样一来，MAC地址学习从"被动泛洪"变成了"主动通告"，效率大大提高。

Type 3（IMET）路由负责解决BUM流量的问题。每个VTEP通过Type 3路由通告自己所属的VNI和对应的组播地址（或Ingress Replication端点列表）。当VTEP需要发送广播帧时，它可以根据IMET路由将帧以组播或头端复制的方式发送给所有同VNI的远端VTEP。

VXLAN EVPN多租户数据中心转发流程示例： 假设有两个租户：Tenant-A（VNI 10001）和Tenant-B（VNI 10002），分别有三台VM分布在不同物理服务器上。当Tenant-A的VM1（IP 10.0.1.1，在Server-1上）首次向同租户的VM2（IP 10.0.1.2，在Server-2上）发送数据包时： 1. VM1发送ARP请求"谁有10.0.1.2的MAC？" 2. Server-1的VTEP收到ARP请求，查找本地没有VM2的MAC记录 3. VTEP通过IMET组播（或Ingress Replication）将ARP请求发给同VNI 10001的所有远端VTEP 4. Server-2的VTEP收到ARP请求，广播给本地VM2所在的端口 5. VM2回复ARP响应（包含自己的MAC地址） 6. Server-2的VTEP学习到VM2的MAC→端口映射，并通过BGP EVPN Type 2通告出去 7. Server-1的VTEP收到Type 2路由，学习到VM2的MAC→Server-2的VTEP IP 映射 8. 此后VM1→VM2的数据包直接通过VXLAN隧道封装发送，不再需要泛洪

22.3 NVGRE与STT：曾经的竞争者

VXLAN并不是Overlay网络领域唯一的方案。在2012-2014年期间，还有几个竞争者活跃过：

NVGRE（Network Virtualization using GRE）：由微软提出，使用GRE（Generic Routing Encapsulation）作为封装协议，而不是UDP。NVGRE也有24位的VNI（称为VSID，Virtual Subnet ID）。但NVGRE的GRE封装不能通过传统的负载均衡器（因为GRE不是UDP/TCP，无法通过五元组哈希），限制了其在某些网络环境中的部署。

STT（Stateless Transport Tunneling）：由Nicira（后被VMware收购）提出。STT使用TCP作为封装协议，但不进行实际的TCP三次握手——它只利用TCP头的"数据偏移"字段来携带内部帧的元数据，从而可以利用现有网卡的TCP分段卸载（TSO）硬件加速。STT的"伪TCP"设计比较取巧，但也让它无法通过传统的防火墙和NAT设备。

后来，IETF NVO3工作组又提出了Geneve（Generic Network Virtualization Encapsulation）作为通用的Overlay封装格式——它吸收了VXLAN、NVGRE、STT各自的设计经验，提供了一种更灵活的可扩展封装格式。但VXLAN凭借先发优势和广泛生态支持，一直是实际部署中最主流的方案。

下面这张表对比了四种主流的Overlay封装技术：

特性 VXLAN NVGRE STT Geneve
封装协议 UDP (4789) GRE (47) TCP (伪连接) UDP (6081)
VNI长度 24位 24位 64位 24位
最大租户数 ~16M ~16M 极大 ~16M
硬件卸载 广泛支持 有限 有限 逐步增加
标准状态 RFC 7348 RFC 7637 RFC (未标准化) RFC 8926
当前地位 主导 边缘化 边缘化 新项目采用

22.4 一个致命的MTU问题

讲完了VXLAN和EVPN这些"高大上"的技术，我想讲一个非常实际的问题——MTU。因为这个问题如果不处理好，整个VXLAN网络都可能"看起来通但实际上慢得离奇"。

标准以太网的MTU是1500字节。一个普通的IP数据包最大1500字节，直接封装进以太网帧发送。但当VXLAN介入时，整个情况变了：

原始以太网帧（1500字节MTU）→ VTEP添加VXLAN头（8字节）→ UDP头（8字节）→ 外层IP头（20字节）→ 外层MAC头（14字节）→ 总计增加了50字节的开销。

如果物理网络的MTU仍然是1500，一个装满1500字节的原始帧经过VXLAN封装后变成了1550字节——超过了物理MTU，IP层必须分片。分片增加了CPU负担，而且分片后的数据包如果有一个丢失，整个TCP段都要重传，性能急剧下降。

经典踩坑：2018年，某大型云服务商的租户报告说，他们的应用在某些服务器之间通信时出现"间歇性卡顿"。运维团队排查了DNS、路由、防火墙、负载均衡……什么都没发现。最后是一个老工程师说了一句："检查一下物理网络的MTU。"结果发现，云平台的underlay网络MTU配置的是1500（默认值），没有为VXLAN封装留出50字节的余量。当两个VM通过VXLAN隧道通信时，TCP MSS协商标出了1460字节的段大小（1500 - 20 IP - 20 TCP），但VXLAN封装后实际物理帧达到了1510字节。这导致一部分数据包在物理链路上被分片或丢弃，TCP发送方检测到超时后触发重传，应用层表现为"偶发的几秒卡顿"。解决方案很简单：将underlay网络的所有交换机端口MTU统一修改为1550（或更大），并启用IP-MTU发现（PMTUD）。修改完后，卡顿立即消失。最终根因非常简单：VXLAN封装增加了50字节开销，而underlay网络的MTU没有相应调整。

这个故事给我们的教训是：在部署任何Overlay网络之前，先把underlay网络的MTU配好。通常建议将物理网络MTU设为9000（巨帧，jumbo frame），这样即使加上多层封装（VXLAN + IPsec等）也绰绰有余。

22.5 Overlay网络的未来

Overlay网络技术经过十年的发展，已经成为现代数据中心网络的"标配"。VXLAN + EVPN的架构方案被几乎所有主流网络设备厂商支持，并且和SDN控制器深度集成。展望未来，有以下几个趋势值得关注：

eBPF加速Overlay：传统的VXLAN VTEP卸载依赖专用的硬件芯片（如ASIC）。但借助eBPF/XDP技术，可以在服务器的网卡驱动层直接完成VXLAN的封装和解封装，性能接近硬件卸载，但灵活性更强。

多云Overlay互连：随着多云和混合云架构的普及，Overlay网络需要跨越不同云平台和自建数据中心。Cisco SD-Access、VMware NSX、AWS Transit Gateway等方案都在朝着这个方向努力——让VXLAN/EVPN"统一"地管理跨云的网络。

安全Overlay：将加密（IPsec、WireGuard）集成到Overlay中，使虚拟机之间的通信即使在共享物理网络上也能保证机密性和完整性。Cisco的ACI和VMware的NSX都已经支持加密VXLAN隧道。
