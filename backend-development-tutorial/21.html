开篇：K8s 网络——让 Pod 之间互通的魔法

假设你第一次部署了一个包含三个服务的 K8s 应用：前端、后端 API、数据库。Service 创建好了，Pod 也跑起来了，但前端就是访问不了后端。你检查 Service 的 Endpoint——发现后端的 Pod IP 是 10.42.1.5，而前端 Pod 的 IP 是 10.42.3.12——它们在不同的 Node 上。你 ssh 到节点上 ping 10.42.1.5——不通。这是怎么回事？

K8s 的网络模型有一个根本要求：集群中所有 Pod 可以直接通信，不需要 NAT，无论它们在哪个 Node 上。但 Linux 原生网络并不支持跨主机的容器互通——你需要一套额外的网络插件来实现这个能力。这正是 CNI（Container Network Interface）要做的事情。

💡 核心认知：K8s 只定义了"网络模型"（每个 Pod 有独立 IP，所有 Pod 可以直接互通），而没有实现它。具体的网络实现由 CNI 插件负责——选对 CNI 插件，是 K8s 集群网络设计中最重要的决策之一。

CNI 插件对比：Calico vs Cilium (eBPF)

市场上最主流的两个 CNI 插件——Calico 和 Cilium——代表了两种不同的网络实现路线。

维度 Calico Cilium
底层技术 BGP（边界网关协议）+ iptables / nftables eBPF（扩展的伯克利包过滤器）
数据路径 内核网络栈（iptables/nftables 规则匹配） 绕过内核网络栈，直接在 eBPF 钩子中处理
吞吐性能 基线（作为参考基准） 高出约 30-50%（特别是小包场景）
延迟 基线 低约 2-5 微秒（对延迟敏感型应用显著）
NetworkPolicy 支持（基于 iptables） 支持（基于 eBPF Maps，更高效）
可观测性 基础 丰富（L3-L7 流量可视化，Hubble 监控栈）
Service Mesh 集成 需额外组件 内置 L7 策略 + 可替代 Sidecar 代理
内核要求 低（5.x+ 均可） 高（推荐 5.10+ 或 6.x+ 以发挥 eBPF 全部能力）
生产采用率 最高（成熟、稳定） 快速增长（当前约 30% 新集群选择）

选型建议：如果你的集群内核版本较旧、对运维复杂度敏感、或已经有一套成熟的 Calico 运维经验，Calico 仍然是稳妥的选择。如果你在规划新的集群，特别是追求高性能、需要深度可观测性、或计划引入 Service Mesh，Cilium 的 eBPF 架构将是更好的起点。

Ingress vs Gateway API：为什么 K8s 社区要推新一代网关标准

当你需要对外暴露 HTTP/HTTPS 服务时，最常见的做法是使用 Ingress——一个 DNS 名称 + 路径路由规则。但 Ingress 在过去的几年中暴露了一些设计上的局限性，这也是 K8s 社区推出新一代 Gateway API 的原因。

# Ingress（传统方式）
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
spec:
  rules:
  - host: api.myapp.com
    http:
      paths:
      - path: /v1
        pathType: Prefix
        backend:
          service:
            name: api-v1-service
            port:
              number: 8080
      - path: /v2
        pathType: Prefix
        backend:
          service:
            name: api-v2-service
            port:
              number: 8080

---
# Gateway API（新一代）
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: app-gateway
spec:
  gatewayClassName: nginx
  listeners:
  - name: https
    protocol: HTTPS
    port: 443
    tls:
      mode: Terminate
      certificateRefs:
      - name: myapp-tls
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-routes
spec:
  parentRefs:
  - name: app-gateway
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /v1
    backendRefs:
    - name: api-v1-service
      port: 8080
  - matches:
    - path:
        type: PathPrefix
        value: /v2
    backendRefs:
    - name: api-v2-service
      port: 8080

到 K8s v1.36（Haru）时，Gateway API 的大部分关键特性已达到 GA 状态，标志着它已经可以承担生产流量入口的重任。相比 Ingress，Gateway API 有三个关键改进。第一是角色分离——Ingress 是一个资源同时定义了"暴露哪些端口"和"流量怎么路由"，Gateway API 把这两个职责拆成了 Gateway（由基础设施团队管理）和 HTTPRoute（由应用团队管理），职责边界清晰。第二是跨 Namespace 支持——一个 Gateway 可以路由流量到不同 Namespace 的 Service，而 Ingress 默认只能路由到同 Namespace。第三是表达能力更强——请求改写、流量拆分、Header 匹配等高级路由能力是 Gateway API 的一等公民，不需要 Ingress Controller 的扩展注解。可以说，Gateway API 是 K8s 社区对 Ingress 六年实践的总结和升级，新项目建议直接采用 Gateway API。

Service Mesh 入门：Istio Ambient Mesh 的 Sidecar-less 架构

Service Mesh 曾是"让 K8s 网络变得更强大"的明星技术，但传统 Sidecar 模式（每个 Pod 旁边部署一个 Envoy 代理）带来了沉重的资源开销和运维复杂性——每个 Sidecar 占用 50-100MB 内存，一个 100 Pod 的集群就要额外消耗 5-10GB 内存。

Istio Ambient Mesh 是 Istio 在 2024 年引入的重大架构革新——它移除了 Sidecar，改为在每个 Node 上部署一个共享的 Layer 4 代理（ztunnel）和可选的 Layer 7 代理（waypoint proxy）。当流量经过时，ztunnel 进行 mTLS 加密和四层处理，需要七层能力（重试、超时、故障注入）的流量才被路由到 waypoint proxy。好处很明显——不需要修改 Pod 定义、不需要给每个 Pod 注入 Sidecar、内存开销从每个 Pod 50MB 降到每个 Node 100MB。

💡 什么时候需要 Service Mesh？如果你的服务间通信需要以下能力之一：mTLS 自动加密、金丝雀发布（流量百分比拆分）、请求级别的遥测、故障注入测试、重试/超时/熔断策略——并且你不想在每个服务代码里重复实现这些逻辑——那么 Service Mesh 值得引入。但如果你的服务数量少于 10 个，或者通信模式很简单，Mesh 的引入成本可能超过收益。

持久化存储：PV / PVC / StorageClass 三部曲

Pod 是"有生命周期的"——删除之后，容器内的数据就没了。对于无状态服务这没问题，但对于数据库、消息队列、文件存储系统，你需要持久化存储。K8s 通过三个层次的抽象来管理存储：PV（PersistentVolume，存储资源的供应端）、PVC（PersistentVolumeClaim，存储资源的需求端）、StorageClass（存储类，决定存储的类型和行为）。

# StorageClass：定义存储的"类型模板"
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iops: "3000"
  throughput: "125"
reclaimPolicy: Retain
allowVolumeExpansion: true

---
# PVC：声明需要的存储
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
  storageClassName: fast-ssd

---
# Pod 或 StatefulSet 中引用 PVC
apiVersion: v1
kind: Pod
metadata:
  name: postgres
spec:
  containers:
  - name: postgres
    image: postgres:17
    volumeMounts:
    - name: pgdata
      mountPath: /var/lib/postgresql/data
  volumes:
  - name: pgdata
    persistentVolumeClaim:
      claimName: postgres-data

这个三层模型的设计精妙之处在于解耦。StatefulSet 使用 volumeClaimTemplates 可以为每个副本自动生成独立的 PVC 和 PV，确保数据库集群的每个节点都有独立、持久的存储。StorageClass 的 allowVolumeExpansion: true 允许在 PVC 中修改 storage 字段后自动扩容——这是云原生存储的标配能力。

CSI 存储插件选型：云原生存储的分类与选择

CSI（Container Storage Interface）是 K8s 的存储插件标准，所有现代存储系统都通过 CSI 驱动与 K8s 集成。选择合适的存储系统，需要理解三类主要的云原生存储：

存储类型 代表方案 访问模式 性能 适用场景
块存储（Block） AWS EBS, GCP Persistent Disk, Azure Disk, Ceph RBD RWO（单节点读写） 高（低延迟、高 IOPS） 数据库（PostgreSQL、MySQL）、Kafka
文件存储（File） AWS EFS, NFS, Azure Files, GlusterFS RWX（多节点读写） 中（受网络延迟影响） 共享配置、日志聚合、静态文件
对象存储（Object） AWS S3, MinIO, Ceph RGW, GCS RWO / 通过 S3 API 访问 中（大带宽、高延迟） 备份归档、AI 训练数据、图片/视频
本地存储（Local） Local PV, OpenEBS, TopoLVM RWO 极高（本地 NVMe） 高性能数据库、时序数据库
⚠️ 一个真实的教训：我经历过一次由"存储配置错误"导致的严重故障。团队用 StatefulSet 部署了一个 Redis 集群，配了 PVC 使用 AWS EFS（文件存储）。开发环境一切正常，但上生产后 Redis 的延迟从 1ms 飙升到 50ms，TPS 暴跌。原因是 EFS 是一个网络文件系统，它的延迟和 IOPS 完全无法满足 Redis 这种内存数据库的写入需求。最终方案：改用 EBS 块存储 + 本地 SSD。一个教训：选择存储方案不能只看"能不能用"，更要看"适不适合"。数据库永远优先选块存储，不要为了"方便"（比如 RWX 多节点共享）牺牲性能。

网络让 Pod 之间能够通信，存储让数据在 Pod 生命周期之外存活——有了这两者，你才能在 K8s 上运行真正有状态的生产应用。但"把应用跑起来"只是第一步，更关键的问题是：如何保证它一直健康地运行？如何应对流量洪峰？如何控制谁有权限做什么？在下一章中，我们将深入 K8s 的生产运维——HPA、资源管理、健康检查、RBAC 和故障演练。
