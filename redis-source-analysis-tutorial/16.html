第16章：命令处理全流程

大家好，我是你们的讲师。上一章我们讲了网络 IO 架构——数据从网线进来，经过 epoll 的唤醒、IO 线程的分发，最终到达了 readQueryFromClient。但接下来呢？一段 TCP 字节流如何变成一条 Redis 命令？Redis 如何在百万分之一秒内找到正确的命令处理函数并执行？执行完后结果如何回到客户端？这一章我们沿着"命令从生到死"的全流程走一遍，从 RESP 协议解析，到命令查找与执行，再到 ACL 校验和响应返回。这是 Redis 源码中最重要的一条执行路径，理解了它，你就掌握了 Redis 后台的主干逻辑。

RESP 协议解析

Redis 自定义了一个文本协议叫 RESP（REdis Serialization Protocol）。它有两个版本：RESP2（传统行协议）和 RESP3（7.x 引入的类型化协议）。

RESP2 — 行协议

RESP2 的核心理念是"以行首字符标识类型"。一条命令就是一组 RESP 数组，每一行都以一个单字符前缀开头：

首字符 类型 示例 说明
+ Simple String +OK\r\n 简单字符串，常用于状态回复
- Error -ERR unknown command\r\n 错误消息
: Integer :1000\r\n 整数
$ Bulk String $5\r\nhello\r\n 二进制安全的字符串，以长度前缀
* Array *2\r\n$3\r\nGET\r\n$3\r\nkey\r\n 数组（用于命令和批量响应）

当你用 redis-cli 发送 SET name redis 时，实际传输的字节流是：

*3\r\n$3\r\nSET\r\n$4\r\nname\r\n$5\r\nredis\r\n

翻译成人话就是：一个数组包含 3 个元素，分别是 "SET"（3 字节）、"name"（4 字节）、"redis"（5 字节）。

RESP3 — 类型化协议

RESP3 在 RESP2 的基础上增加了更多数据类型，如 Map、Set、Null、Boolean、Double、Big Number 等。它的引入主要是为了支持更丰富的客户端交互场景——比如 HGETALL 的响应可以用 Map 类型原生态地表达键值对语义，而不是用数组对去套。

解析源码 — resp_parser.c

在 Redis 7.2 中，命令解析从 networking.c 中独立出来，放在了 resp_parser.c。核心入口是 readQueryFromClient：

/* readQueryFromClient — 从客户端连接中读取查询数据 */
void readQueryFromClient(connection *conn) {
    client *c = connGetPrivateData(conn);
    int nread;
    size_t qblen;

    /* 读取数据到 querybuf */
    nread = connRead(c->conn, c->querybuf + c->qb_pos,
                     c->querybuf_remaining);
    if (nread) {
        qblen = sdslen(c->querybuf) - c->qb_pos;
        c->qb_pos += nread;

        /* 尝试解析 querybuf 中的命令 */
        if (processInputBuffer(c) != C_OK) {
            /* 解析失败，继续等待更多数据 */
            return;
        }
    }
    /* 检查 querybuf 是否超过限制 */
    if (sdslen(c->querybuf) > server.client_max_querybuf_len) {
        sds ci = catClientInfoString(sdsempty(), c);
        serverLog(LL_WARNING, "Closing client that reached max query buffer length: %s", ci);
        sdsfree(ci);
        freeClient(c);
        return;
    }
}

querybuf 是一个 SDS 动态字符串，所有从 socket 读到的原始数据都先存在这里。然后 processInputBuffer 逐行解析，每解析出一条完整的命令就立刻执行——注意，它是一次性尽量多地消费 querybuf 中的数据，而不是"读一条命令就等下一次事件"。

优化细节：querybuf 用 SDS 实现意味着扩容是 O(1) 的。更重要的是，Redis 在 processInputBuffer 解析完成后会调用 sdsrange 来"裁剪"已消耗的字节，而不是重新分配内存。这种"读取→消费→裁剪"的模式避免了大量的内存拷贝。

命令查找

解析出命令名（比如 "GET"、"SET"）之后，Redis 需要在命令表中找到对应的处理函数。这个命令表在 server.c 的 populateCommandTable 中初始化：

命令表的构建

Redis 使用一个字典（server.commands）来存储命令名到 redisCommand 结构体的映射。每个命令对应一个 redisCommand 结构体，包含了命令的元信息和指向实现的函数指针：

/* redisCommand — 命令描述结构体 */
struct redisCommand {
    char *name;                          /* 命令名 */
    redisCommandProc *proc;              /* 实现函数指针 */
    int arity;                           /* 参数个数（负数表示至少多少个） */
    char *sflags;                        /* 字符串格式的标志 */
    uint64_t flags;                      /* 位掩码标志 */
    int firstkey, lastkey, keystep;      /* 键参数位置 */
    long long microseconds, calls;       /* 延迟统计 */
};

查找过程通过 lookupCommand 完成，命令名匹配是大小写不敏感的（这就是为什么 GET、get、Get 都有效）：

/* lookupCommand — 查找命令（大小写不敏感） */
struct redisCommand *lookupCommand(sds name) {
    return dictFetchValue(server.commands, name);
}

/* 命令表初始化时，所有命令名都被统一转换为小写 */
void populateCommandTable(void) {
    for (int j = 0; j < commandCount; j++) {
        struct redisCommand *c = commandTable + j;
        char *desc = commandTable[j].decl;
        /* ... 解析 desc 字段填充命令表 ... */
        c->name = sdsnew(desc);  /* 已经是小写 */
        dictAdd(server.commands, c->name, c);
    }
}

命令执行

找到命令后，Redis 进入执行阶段。这个阶段的入口是 processCommand——它是 Redis 命令执行的总阀门。

processCommand — 前置检查

在执行命令之前，processCommand 会做一系列检查——这就像登机前的安检，每道关卡都可能把命令拦截下来：

  • 命令是否存在：如果 lookupCommand 返回 NULL，返回 -ERR unknown command。
  • 参数个数：检查 c->argc 是否满足 cmd->arity（正数表示严格等于，负数表示至少多少个）。
  • 认证状态：如果开启了 requirepass 且客户端未认证，返回 NOAUTH。
  • ACL 权限：调用 ACLCheckCommandPermissions 校验用户是否有权执行该命令。
  • 集群重定向：在集群模式下，按键的 hash slot 定位目标节点，如果需要重定向，返回 MOVED 或 ASK。
  • 内存检查：如果已开启最大内存限制且当前内存超出，getMaxmemoryState 触发内存淘汰，淘汰失败则拒绝写入命令。
  • 子进程保护：如果正在执行 BGSAVE 或 BGREWRITEAOF，对于写命令可能会延迟执行（取决于配置）。
/* processCommand — 命令执行的前置检查与分发 */
int processCommand(client *c) {
    /* Step 1: 查找命令 */
    c->cmd = c->lastcmd = lookupCommand(c->argv[0]->ptr);
    if (!c->cmd) {
        addReplyErrorFormat(c, "unknown command '%s'",
                            (char*)c->argv[0]->ptr);
        return C_OK;
    }

    /* Step 2: ACL 权限校验 */
    int acl_retval = ACLCheckAllPerm(c, &acl_errpos);
    if (acl_retval != ACL_OK) {
        addReplyError(c, ...);
        return C_OK;
    }

    /* Step 3: 最大内存检查（写命令触发淘汰） */
    if (server.maxmemory && !(c->flags & CLIENT_MASTER)) {
        if (performEvictions() == EVICT_FAIL) {
            rejectCommand(c, shared.oomerr);
            return C_OK;
        }
    }

    /* Step 4: 如果开启了 multi/exec 事务，入队 */
    if (c->flags & CLIENT_MULTI &&
        c->cmd->proc != execCommand) {
        queueMultiCommand(c);
        addReply(c, shared.queued);
        return C_OK;
    }

    /* Step 5: 真正执行命令 */
    if (c->flags & CLIENT_DENY_BLOCKING &&
        (c->cmd->flags & CMD_BLOCKING)) {
        /* 阻塞命令在内部线程中不可用 */
        addReply(c, shared.busykeyerror);
    } else {
        call(c, CMD_CALL_FULL);  /* 执行 */
    }
    return C_OK;
}

call 函数 — 命令执行的核心

call(c, CMD_CALL_FULL) 是执行命令的最后一步。它做了以下几件事：

  • 调用 cmd->proc(c) ——多态调用命令的具体实现函数。
  • 统计执行时间，更新 cmd->microseconds 和 cmd->calls。
  • 如果开启了 slowlog 且执行时间超过阈值，记录慢日志。
  • 将命令传播到 AOF 缓冲区和从节点（如果开启了 AOF 或复制）。
小贴士：call() 是 Redis 命令执行统计的精确数据来源。你可以用 INFO COMMANDSTATS 查看每个命令的平均耗时和调用次数——这些数字就是在 call() 中被统计的。

ACL 权限校验

Redis 6.0 引入的 ACL（Access Control List）系统为 Redis 提供了细粒度的权限控制。ACL 校验的入口是 ACLCheckCommandPermissions：

  • 用户认证：每个 client 关联一个 user 结构体。默认用户是 default，可以通过 AUTH <username> <password> 切换用户。
  • 命令权限：ACL 规则可以指定用户能否执行某些命令类别（如 +@admin、-@dangerous）或特定命令（+SET、-FLUSHALL）。
  • 键权限：支持基于键的权限模式，例如 ~cache:* 表示用户只能访问以 cache: 为前缀的键。
  • ACL LOG：被拒绝的操作会记录在 ACL LOG 中，方便排查为什么某个命令被拒绝了。

响应返回到客户端

命令执行完毕后，结果通过 addReply 系列函数返回到客户端。Redis 提供了一组方便的高层 API：

  • addReply(c, shared.ok) —— 返回 +OK\r\n
  • addReplyBulk(c, obj) —— 返回 Bulk String
  • addReplyError(c, "msg") —— 返回 -ERR msg\r\n
  • addReplyLongLong(c, val) —— 返回 Integer
  • addReplyArrayLen(c, len) —— 设置 Array 长度前缀（流式构建数组响应）

这些响应数据先被写入 client->buf 或 client->reply 链表（详见上一章的输出缓冲区管理），然后在 beforeSleep 中批量刷入 socket。这个"攒一批再写"的设计显著减少了系统调用次数。

事务与流水线

Redis 的事务通过 MULTI / EXEC / WATCH 实现，虽然不像传统数据库那样支持回滚，但在某些场景下非常有用。事务的实现机制在 multi.c 中：

  • MULTI：设置 c->flags |= CLIENT_MULTI，之后的命令不会立即执行，而是 queueMultiCommand(c) 将命令入队到 c->mstate.commands 数组中。
  • EXEC：遍历 mstate.commands，对每条命令调用 call(c, CMD_CALL_FULL)，将结果存入 c->bpop 的多回复中，最后一次性返回给客户端。
  • WATCH：在 server.watched_keys 字典中记录被监视的键。如果在执行 EXEC 之前有任何被监视的键被修改（在 signalModifiedKey 中标记），EXEC 返回 nil，事务被丢弃。
  • 流水线（Pipeline）：客户端可以不等待回复连续发送多条命令。服务端读取 querybuf 时逐个解析执行，响应也都暂存在输出缓冲区中。流水线的性能提升主要来自减少网络 RTT，而非 Redis 处理速度的提升。
踩坑实录：几年前有团队在生产环境开启了 MONITOR 命令进行调试，结果发现 Redis 的吞吐下降了 50% 以上。为什么会这样？MONITOR 的实现原理是：在执行 call() 时，如果开启了 MONITOR，每条命令的执行信息都会被 replicationFeedMonitors 写入 MONITOR 客户端的输出缓冲区。换句话说，每条命令多了一个输出客户端——所有的写操作都被加倍了。更糟糕的是，如果 MONITOR 客户端消费速度跟不上，输出缓冲区会持续堆积，进而导致客户端被断开或内存膨胀。所以生产环境绝对不要长时间开启 MONITOR，确有调试需要可以用 redis-cli monitor | head -100 抓几秒就关掉。

总结

这一章我们完整追踪了一条 Redis 命令的生命周期：TCP 字节流 → querybuf 缓冲 → RESP 协议解析 → 命令表查找 → ACL 权限检查 → 前置校验 → call() 调用处理函数 → addReply 将结果写入输出缓冲区 → 最终通过 socket 发回客户端。这个过程在理想情况下只需要几十微秒，每一个环节都经过精心优化。

但是等等——SET name redis 执行完了，数据存在了内存中。断电怎么办？重启怎么办？内存数据丢失是所有数据库的天敌。下一章我们就来解决这个问题——深入解读 Redis 的 RDB 快照持久化机制。
