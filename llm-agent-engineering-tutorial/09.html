第9章 中间件系统

在生产环境中运行 Agent，最让人头疼的往往不是"模型不够聪明"，而是那些盘根错节的"脏活"：用户的输入里可能带着身份证号（PII），必须脱敏才能传给模型；Agent 要执行高危操作时需要人工审批；模型偶尔抽风给出错误答案，需要有兜底策略；最怕的是——Agent 被注入攻击，输出恶意内容。这些横切关注点如果散落在业务代码里，项目很快就会变成一团乱麻。

LangChain 1.x 的 中间件系统（Middleware）正是为了解决这些问题而生。它借鉴了 Web 框架（Express、FastAPI）中经典的中间件模式，让你可以在 Agent 的请求-响应管道中插入可插拔的处理层。这一章我们深入剖析中间件的设计理念和五个核心中间件的实战用法。

核心思想： 中间件是一种"AOP（面向切面编程）"风格的架构模式——它把"横切关注点"（安全、监控、审批、限流）从业务逻辑中抽离出来，统一放到管道中处理。这让业务代码更干净，也让安全策略可以集中管理和审计。

9.1 中间件架构设计理念

LangChain 1.x 的中间件系统基于 RunnableMiddleware 抽象类。每个中间件本质上是一个包装器，包裹住原始的 Runnable 组件，在 invoke 之前和之后执行自定义逻辑：

from langchain.runnables import RunnableMiddleware
from langchain.callbacks import CallbackManager

class LoggingMiddleware(RunnableMiddleware):
    """一个简单的日志中间件"""

    async def before_invoke(self, input_data: dict) -> dict:
        print(f"[LOG] 收到输入: {input_data}")
        return input_data  # 可以修改输入

    async def after_invoke(self, output_data: dict) -> dict:
        print(f"[LOG] 生成输出: {output_data}")
        return output_data  # 可以修改输出

    async def on_error(self, error: Exception) -> dict:
        print(f"[ERROR] 调用失败: {error}")
        raise  # 可以选择吞掉异常或重新抛出

中间件的执行顺序遵循"洋葱模型"——请求从外层向内层层穿过，响应从内层向外层层返回。这个模型和 Web 框架中的中间件如出一辙：

层 中间件 作用
第1层（最外层） PII 脱敏 在请求进入前清洗敏感数据
第2层 内容审核 检查输入是否包含恶意内容
第3层 Human-in-the-loop 高危操作需要人工确认
第4层 Model Fallback 主模型失败时自动降级
第5层（最内层） 实际 Agent 核心业务逻辑
设计要点： 中间件的顺序很重要。PII 脱敏应该在最外层，确保进入系统内部的所有数据都已经过清洗。Human-in-the-loop 应该在业务逻辑之前，避免 Agent 已经开始执行才被拦截，浪费 Token 和资源。

9.2 PII 脱敏中间件实现

PII（Personal Identifiable Information）脱敏是 Agent 系统中最基础也是最关键的安全措施。想象一下——客服 Agent 收到了用户的消息，里面包含"我的身份证号是 110101199001011234，帮我查一下订单"。如果不做脱敏，这个身份证号会直接传给 LLM，而 LLM 的日志、训练数据都有可能将此信息永久保留。

LangChain 1.x 内置了 PIIDetectionMiddleware，支持自动识别和替换敏感信息：

from langchain.middleware import PIIDetectionMiddleware
from langchain.chat_models import ChatOpenAI
from langchain.agents import create_agent

pii_middleware = PIIDetectionMiddleware(
    # 支持的 PII 类型
    entities=[
        "ID_CARD",      # 身份证号
        "PHONE",        # 手机号
        "EMAIL",        # 邮箱
        "BANK_CARD",    # 银行卡号
        "ADDRESS",      # 地址
        "IP_ADDRESS",   # IP 地址
    ],
    # 脱敏模式：mask（替换为 ***）或 hash（替换为哈希值）
    mode="mask",
    # 自定义脱敏符号
    mask_char="*",
    # 白名单：以下字段不脱敏
    allowlist_fields=["order_id", "product_name"],
)

# 将中间件应用到 Agent
agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[],
    system_prompt="你是客服助手。",
    middleware=[pii_middleware]
)

result = agent.invoke({"input": "我的身份证是110101199001011234，帮我查一下订单"})
# 模型实际看到的是："我的身份证是******************，帮我查一下订单"
踩坑实录： 我们早期部署 PII 脱敏时遇到过一个尴尬的问题——某用户的用户名是 "ILoveYou3000"，被正则识别成了信用卡号（因为包含数字序列），直接被脱敏成了 "ILove****"。后来我们在 allowlist 里加上了用户名相关的 context 规则，才解决这个误伤问题。PII 识别的精准度需要结合业务上下文不断优化。

9.3 Human-in-the-loop 审批流程

对于高风险操作（如执行写操作、发送消息、修改数据库），让 Agent 自主决策是不可接受的。Human-in-the-loop（HITL）模式要求在 Agent 执行某些操作之前，先暂停并等待人工确认。

from langchain.middleware import HumanApprovalMiddleware
from langchain.agents import create_agent

def approval_rule(tool_name: str, args: dict) -> str:
    """定义需要审批的工具和条件"""
    if tool_name == "send_email" and len(args.get("to", [])) > 10:
        return "批量发送邮件需要管理员审批"
    if tool_name == "delete_user":
        return "删除用户操作需要二级审批"
    if tool_name == "execute_sql" and "DROP" in args.get("sql", "").upper():
        return "DROP 操作需要 DBA 审批"
    return "auto_approve"  # 自动放行

hitl_middleware = HumanApprovalMiddleware(
    rule_fn=approval_rule,
    # 审批通知方式
    notification={
        "channel": "webhook",  # 或 slack、email、pagerduty
        "url": "https://hooks.example.com/approval",
        "timeout": 300,  # 等待审批超时（秒）
    },
    # 超时后的默认行为
    on_timeout="reject",  # 或 "approve"、"fallback"
)

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[send_email, delete_user, execute_sql],
    middleware=[hitl_middleware]
)

HITL 中间件在实际企业部署中尤为重要。根据我们团队的统计，在生产环境中，大约 5-8% 的 Agent 工具调用需要人工介入。虽然比例不高，但正是这 5% 的拦截避免了绝大多数潜在事故。

审批流程设计要点：
  • 分级审批： 低风险操作自动放行，中风险单人审批，高风险双人审批。
  • 超时策略： 审批超时后应该拒绝（reject）而非自动放行——安全优先于体验。
  • 审计追踪： 每一次审批决定（通过/拒绝/超时）都要记录，用于事后审计。
  • 审批上下文： 给审批人提供足够的上下文——Agent 的思考链、当前对话历史、操作的风险评估。

9.4 Model Fallback 策略与降级

模型不可用是生产环境中最常见的故障模式——API 限流、网络超时、模型服务中断、Token 配额耗尽。一个好的 Fallback 策略可以在这些故障发生时优雅降级，而不是直接返回错误给用户。

from langchain.middleware import FallbackMiddleware
from langchain.chat_models import ChatOpenAI, ChatAnthropic, ChatGoogle

fallback_middleware = FallbackMiddleware(
    # 按优先级排列的模型列表
    models=[
        ChatOpenAI(model="gpt-4o", timeout=30),
        ChatOpenAI(model="gpt-4o-mini", timeout=30),
        ChatAnthropic(model="claude-sonnet-4-7", timeout=30),
        ChatGoogle(model="gemini-2.5-flash", timeout=30),
    ],
    # 触发降级的条件
    fallback_triggers=[
        "RateLimitError",     # API 限流
        "TimeoutError",       # 超时
        "ServiceUnavailable", # 服务不可用
        "AuthenticationError",# 认证失败（如 API Key 过期）
    ],
    # 降级策略
    strategy="circuit_breaker",  # 熔断模式：连续失败 N 次后跳过
    circuit_breaker_threshold=3, # 连续失败 3 次触发熔断
    circuit_breaker_reset=60,    # 60 秒后重置熔断器
)

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),  # 主模型
    middleware=[fallback_middleware]
)
降级策略 说明 适用场景
线性降级 按列表顺序依次尝试，直到成功 简单的多模型备选
熔断模式 连续失败 N 次后跳过该模型，一段时间后恢复 避免不断请求已故障的服务
并发尝试 同时请求多个模型，取最快返回的结果 对延迟敏感的场景
质量降级 主模型失败后，切换到能力较弱但更稳定的模型 非核心功能可接受较低质量
踩坑实录： 我们曾遇到过一个特别隐蔽的 Fallback 问题——某个深夜 OpenAI 的 GPT-4o 服务降级，Fallback 自动切换到 Claude。但问题在于，我们的 Prompt 是为 GPT 优化的（包含大量 System Message 中的格式要求），Claude 理解方式不同，导致输出格式完全不对，下游解析全部崩溃。所以 Fallback 不仅仅是"换个模型"，还需要考虑 Prompt 适配。如今我们在每个 Fallback 链中都加入一层 Prompt 适配中间件。

9.5 内容审核与安全过滤

Agent 的输出安全是另一个不容忽视的问题。无论是用户通过 Prompt 注入诱导 Agent 输出不当内容，还是 Agent 自身推理产生有害结果，都需要有审核机制来拦截。

from langchain.middleware import ContentModerationMiddleware

moderation_middleware = ContentModerationMiddleware(
    # 审核范围：输入和输出都审核
    scope=["input", "output"],
    # 敏感类别
    categories=[
        "hate_speech",      # 仇恨言论
        "violence",         # 暴力内容
        "sexual_content",   # 色情内容
        "personal_info",    # 个人信息泄露
        "code_injection",   # 代码注入攻击
        "prompt_leakage",   # Prompt 泄露（尝试套取 System Prompt）
    ],
    # 审核动作
    on_violation_input="block",     # 输入违规：阻止请求
    on_violation_output="replace",  # 输出违规：替换为安全内容
    violation_message="输入包含不允许的内容，请重新表述。",
    # 自定义审核规则
    custom_rules=[
        r"ignore all previous instructions",  # Prompt 注入关键词
        r"你是一个(免费|无限制)的AI",         # 角色冒充尝试
    ],
    # 调用外部审核 API（如 OpenAI Moderation、Azure Content Safety）
    external_api="openai_moderation",
    api_threshold=0.8,  # 审核分数阈值
)

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[],
    middleware=[moderation_middleware]
)
安全实践： 内容审核应该是"双层防护"——第一层在输入时拦截恶意请求，第二层在输出时检查结果。输入审核可以防止 Prompt 注入，输出审核则可以发现 Agent 自身推理产出的问题。两层的检测标准可以不同：输入层更严格（宁可误杀不可放过），输出层则可以适当宽松（避免过度过滤正常内容）。

9.6 自定义中间件开发实战

现实场景中的需求往往超越框架内置中间件的能力。幸运的是，LangChain 1.x 让自定义中间件的开发变得非常简单。我们来看一个实际的例子——开发一个"成本控制中间件"，在每次 Agent 调用结束后累加 Token 消耗，当达到预算上限时自动熔断：

from langchain.runnables import RunnableMiddleware
from langchain.callbacks import CallbackManager
import time

class BudgetControlMiddleware(RunnableMiddleware):
    """
    预算控制中间件：
    - 跟踪每次调用的 Token 消耗
    - 当日预算超限时自动拒绝请求
    - 发送预算告警
    """

    def __init__(self, daily_budget: int, alert_threshold: float = 0.8):
        super().__init__()
        self.daily_budget = daily_budget    # 每日预算（Token 数）
        self.alert_threshold = alert_threshold
        self.current_usage = 0
        self.alert_sent = False

    async def before_invoke(self, input_data: dict) -> dict:
        if self.current_usage >= self.daily_budget:
            raise BudgetExceededError(
                f"当日预算已用尽（{self.current_usage}/{self.daily_budget}）"
            )
        return input_data

    async def after_invoke(self, output_data: dict) -> dict:
        # 从回调中提取 Token 消耗
        token_usage = output_data.get("token_usage", 0)
        self.current_usage += token_usage

        usage_ratio = self.current_usage / self.daily_budget
        if usage_ratio >= self.alert_threshold and not self.alert_sent:
            await self._send_alert(f"预算使用已达 {usage_ratio:.0%}")
            self.alert_sent = True

        return output_data

    async def _send_alert(self, message: str):
        """发送告警（可接入 Slack/PagerDuty 等）"""
        print(f"[BUDGET ALERT] {message}")

# 使用自定义中间件
budget_middleware = BudgetControlMiddleware(
    daily_budget=1_000_000,  # 每日 100 万 Token
    alert_threshold=0.8,
)

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[search_web, query_database],
    middleware=[
        budget_middleware,
        PIIDetectionMiddleware(),
        ContentModerationMiddleware(),
    ],
)
中间件开发最佳实践：
  • 保持无状态或显式管理状态： 中间件实例可能会被复用，状态管理要小心。像 BudgetControlMiddleware 中的 current_usage 最好使用 Redis 等外部存储。
  • 不要吞掉你不理解的异常： on_error 中除非你有明确的恢复策略，否则应该重新抛出异常。
  • 性能敏感： 中间件中的每个操作都会增加端到端延迟。把耗时操作（如调用外部审核 API）改为异步执行。
  • 可观测性： 每个中间件都应该记录自己的执行时间和决策结果，方便排查问题。

实际上，LangChain 1.x 还提供了一个 MiddlewareStack 类，用于更方便地管理多个中间件的组合和排序：

from langchain.middleware import MiddlewareStack

# 用一个 Stack 统一管理所有中间件
middleware_stack = MiddlewareStack(
    middlewares=[
        ("pii", PIIDetectionMiddleware(mode="mask")),
        ("moderation", ContentModerationMiddleware()),
        ("budget", BudgetControlMiddleware(daily_budget=1_000_000)),
        ("fallback", FallbackMiddleware(models=[...])),
        ("hitl", HumanApprovalMiddleware(rule_fn=approval_rule)),
    ],
    # 发生异常时回滚已执行的中间件
    rollback_on_error=True,
)

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[...],
    middleware=middleware_stack,
)

本章小结

中间件系统是 LangChain 1.x 中最具工程价值的特性之一。它用标准化的 AOP 模式解决了 Agent 生产化中一系列棘手的"横切关注点"——从 PII 脱敏保护用户隐私，到 Human-in-the-loop 确保关键操作可控，从 Model Fallback 保证服务可用性，到内容审核防止安全事件。更重要的是，这套机制让你可以像搭积木一样组合这些能力，并在需要时轻松编写自定义中间件。
