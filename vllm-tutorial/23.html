第23章 nano-vLLM 源码剖析

前面 22 章咱们都在讲"怎么用"vLLM。这一章换个视角——如果让你自己写一个最小版本的 vLLM，你会怎么写？nano-vLLM 就是这样一个项目：~1200 行 Python + Triton，把 vLLM 的核心机制讲得明明白白。

23.1 项目概览

23.2 LLMEngine：推理的总指挥

class LLMEngine:
    def __init__(self, model_path, **kwargs):
        self.scheduler = Scheduler()
        self.model_runner = ModelRunner(model_path)
        self.sequences = {}  # request_id → Sequence

    def step(self):
        # 1. 调度：决定这一轮哪些请求参与
        batch = self.scheduler.schedule(
            waiting_queue=self.waiting_queue,
            running=self.running_seqs
        )
        # 2. 执行：GPU 前向推理
        outputs = self.model_runner.forward(batch)
        # 3. 采样：从 logits 中选择 token
        for req_id, logits in outputs.items():
            next_token = sample(logits, self.sequences[req_id].sampling_params)
            self.sequences[req_id].append_token(next_token)
        return outputs

23.3 Scheduler：三步决策

class Scheduler:
    def schedule(self, waiting_queue, running):
        # 1. 移除完成的序列
        done = [r for r in running if r.is_finished()]
        for r in done:
            self.kv_cache.free_blocks(r.block_table)
            running.remove(r)

        # 2. 拉入等待序列（如果显存够）
        free = self.kv_cache.free_block_count()
        for r in waiting_queue:
            needed = r.required_blocks()
            if needed <= free:
                r.block_table = self.kv_cache.allocate(needed)
                running.append(r)
                free -= needed

        # 3. 组织 batch（prefill + decode 混合）
        return Batch(tokens=[...], block_tables=[...])

23.4 ModelRunner：GPU 执行与 KV Cache

class ModelRunner:
    def __init__(self, model_path):
        self.model = load_model(model_path)
        self.kv_cache = self._allocate_kv_cache()

    def forward(self, batch):
        # 1. 构造输入（含 position_ids 和 block_tables）
        inputs = self.prepare_inputs(batch)

        # 2. Attention kernel 通过 block_tables 找到正确的 K/V
        # （nano-vLLM 用 Triton 实现，官方 vLLM 用 C++/CUDA）

        # 3. CUDA Graph 回放（decode 阶段跳过 kernel launch 开销）
        with self.cuda_graph.replay():
            return self.model(**inputs)

    def _allocate_kv_cache(self):
        # Block 大小 = 16 tokens
        # 总 block 数 = 可用显存 / (每 block KV 数据量)
        num_blocks = self._calculate_max_blocks()
        return KVCache(num_blocks=num_blocks, block_size=16)

23.5 关键Triton Kernel

# nano-vLLM 的 KV Cache 写入 kernel（Triton）
@triton.jit
def store_kvcache_kernel(
    key_ptr, value_ptr,        # 当前 forward 的 K/V 输出
    key_cache, value_cache,   # 全局 KV cache 存储
    slot_mapping,              # 逻辑位置 → 物理位置
    ...
):
    token_idx = tl.program_id(0)
    slot = tl.load(slot_mapping + token_idx)  # 物理位置

    if slot >= 0:
        # 写入 KV cache 对应位置
        for head_idx in range(num_kv_heads):
            kv_offset = slot * head_dim + head_idx * block_size * head_dim
            tl.store(key_cache + kv_offset, tl.load(key_ptr + ...))
            tl.store(value_cache + kv_offset, tl.load(value_ptr + ...))

23.6 nano-vLLM vs 官方 vLLM

方面vLLMnano-vLLM
代码量10,000+ LOC~1,200 LOC
语言Python + C++ + CUDAPython + Triton
KV CachePagedAttention C++/CUDATriton kernel + slot mapping
TP完整 NCCL sharding基础 torch.distributed
安装需编译环境pip install 即可
适用生产服务学习、研究、原型
性能 (小模型)1360 tok/s1434 tok/s (+5%)

为什么 nano-vLLM 在小模型上竟然更快？不是因为它设计了更好的算法，而是因为它更轻量——没有 V1 的进程通信开销、没有复杂的调度逻辑、没有兼容性代码路径。在简单场景下，"少即是快"。但在复杂生产场景（多 GPU、大模型、高并发），官方 vLLM 的各种优化才开始发挥作用。

23.7 从 nano-vLLM 学到什么

读 nano-vLLM 源码，你能搞清楚三件事：

  1. KV Cache 到底怎么存的——Block Table 数据结构、分配/回收逻辑、Triton kernel 怎么通过 slot mapping 读写分散的物理 block
  2. 调度器到底怎么决策的——从等待队列取请求的条件是什么、显存不够时的退避策略
  3. 模型执行怎么衔接调度——ModelRunner 怎么把 Scheduler 的"纸上决策"变成 GPU 上的真实运算

强烈建议你 clone 下来读一遍——1200 行，一杯咖啡的时间。读完你会对整个教程的前 14 章有更深的理解。
