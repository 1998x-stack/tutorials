开篇：那个忘记锁版本的惨痛教训

2019年，我负责的一个Node.js项目在CI上突然全线飘红——测试全部失败。排查了一上午，发现原因是某个间接依赖发布了新的次版本（minor version），而这个版本引入了一个破坏性变更。我们的package.json写的是"^1.2.0"，自动拉到了1.3.0——然后整个项目就炸了。那天我们花了整整8小时才定位到问题，最后加了个package-lock.json的精确锁定才解决。

转到Rust后我惊讶地发现：Cargo从第一天起就默认生成了Cargo.lock文件，对应用项目会精确锁定每个依赖的版本。这不是一个"可选的最佳实践"——这是Cargo的默认行为。Rust社区对依赖管理的严肃态度，从这一设计就可见一斑。

上一章我们学习了测试，写好了代码还确保了它的正确性。接下来，是时候深入Cargo的更高级功能了——从依赖管理到Feature系统，从编译配置优化到发布自己的crate。这一章将让你从"会用Cargo"进阶到"善用Cargo"。

依赖管理 — 版本约束、git依赖、path依赖

语义化版本约束

Rust依赖版本遵循语义化版本（SemVer）：主版本.次版本.修订号。Cargo支持多种版本约束写法：

写法 含义 示例
"1.2.3" 精确版本 ^1.2.3 的简写 接受 1.2.3 到 <2.0.0
"^1.2.3" 兼容的版本更新 接受 1.2.3 到 <2.0.0
"~1.2.3" 只允许修订号变化 接受 1.2.3 到 <1.3.0
"=1.2.3" 精确锁定版本 只能用 1.2.3
">=1.2.0" 最低版本要求 接受任何 >=1.2.0 的版本
"1.*" 通配符 接受 1.0.0 到 <2.0.0

Cargo的默认行为是^——它认为次版本号的变化是兼容的。这符合SemVer的约定：次版本增加意味着添加了新功能但不破坏现有API。

Git依赖

当你的项目需要依赖一个尚未发布到crates.io的crate，或者想使用某个crate的最新未发布提交时，可以指定Git仓库地址：

[dependencies]
my_crate = { git = "https://github.com/user/my_crate.git", branch = "main" }
# 或者指定 tag / rev（commit hash）
my_crate = { git = "https://github.com/user/my_crate.git", tag = "v0.2.0" }
my_crate = { git = "https://github.com/user/my_crate.git", rev = "a1b2c3d4" }

Path依赖

在开发多个相互依赖的本地crate时（比如monorepo结构），Path依赖是最方便的：

[dependencies]
my_utils = { path = "../my_utils" }

Path依赖的crate不需要发布到crates.io——Cargo会直接从本地文件系统引用。这在工作空间（workspace）中尤其常见，我们稍后会提到。

SemVer实战原则：你的库如果发布了1.0.0之后，永远不要在次版本中做破坏性API变更。这是Rust社区的铁律。一个反例是tokio团队在1.x版本中通过feature flag来引入新功能，完全不需要用户改代码——这才是SemVer的典范实践。

Features — 条件编译、默认features、feature组合

Features是Cargo中一个极其强大但容易被初学者忽略的功能。它允许你在编译时通过条件编译来选择性地启用或禁用crate的某些功能。

定义一个Feature

在Cargo.toml中声明features：

[features]
default = ["std"]
std = []
serde_support = ["dep:serde"]
async = ["dep:tokio"]

然后在代码中通过cfg属性检测：

#[cfg(feature = "serde_support")]
impl Serialize for MyType {
    // ...
}

#[cfg(not(feature = "std"))]
pub fn allocate_in_embedded() { /* 无std环境下的分配器 */ }

默认Features与用户覆盖

用户在Cargo.toml中引入你的crate时，可以主动关闭默认features：

[dependencies]
my_crate = { version = "0.1", default-features = false }
# 只启用 serde_support，不启用默认的 std
my_crate = { version = "0.1", default-features = false, features = ["serde_support"] }

这个设计对no_std环境（嵌入式开发、WASM等）至关重要。没有这个机制，每个no_std crate都要维护一个独立的代码分支——有了features，同一个crate可以同时为std和no_std服务。

Feature组合的技巧

Feature名有一个隐含约定：如果feature触发某个依赖，可以用dep:前缀来避免歧义（Rust 2024 Edition已稳定支持）。同时要注意feature之间会产生组合爆炸——每增加一个feature，你要保证它与其他所有feature的组合都是正确的。这也是Rust社区强调"feature应该尽量正交"的原因。

最佳实践：大型crate（如tokio、clap）通常有几十个feature。作为用户，你可以用cargo tree -f "{p} {f}"来查看当前所有已启用的feature的树状结构。这个命令比看文档更直观也更能保证准确性。

Profile配置 — dev/release、优化级别、LTO

Cargo通过Profiles来管理不同场景下的编译选项。每个Profile是一组编译器标志的集合，你可以覆盖默认值。

Profile 默认用途 优化级别 调试信息
dev cargo build 默认 0（不优化） 完整调试符号
release cargo build --release 3（积极优化） 无（可单独开启）
test cargo test 0 完整调试符号
bench cargo bench 3 无

你可以在Cargo.toml中覆盖这些设置：

[profile.dev]
opt-level = 1  # 开发编译也做轻度优化

[profile.release]
opt-level = 3      # 默认就是3，可以不写
lto = true         # 链接时优化（Link-Time Optimization）
codegen-units = 1  # 单codegen单元，牺牲编译时间换取运行性能
strip = "symbols"  # 移除符号表，减小二进制体积
panic = "abort"    # panic时直接abort而非unwind，减小体积

这里有一些关键配置的解读：

  • LTO：链接时优化，让LLVM在链接整个程序时做跨模块的优化。能提升5-15%的性能，但显著增加链接时间。
  • codegen-units：Rust默认将crate分为16个并行编译单元来加速编译。但单元之间的边界会阻止某些优化。设为1时所有代码在同一个单元中编译，性能最优。
  • strip：Release版本如果不需要调试或符号信息（比如分发给用户的CLI工具），strip掉符号可以显著减小二进制体积。
  • panic = "abort"：去除panic时栈展开的代码，二进制体积能小10-30%。代价是panic时无法得到栈回溯。
一个真实的坑：我曾经把一个--release编译的服务部署到生产环境，结果它在某个并发压力下panic了——但因为我们用了panic = "abort"，没有栈回溯，只看到了Aborted (core dumped)。花了整整两天才复现并定位问题。所以线上环境建议保留调试符号（设为strip = "none"），或者在panic时至少能看到backtrace。

发布到crates.io — crate命名、文档、README、发布流程

写了一个好的crate，当然想分享给社区。发布到crates.io需要几个步骤。

发布前准备

首先确保你的Cargo.toml信息完整：

[package]
name = "my_awesome_crate"
version = "0.1.0"
edition = "2024"
description = "一段简短但能吸引人的描述"
license = "MIT OR Apache-2.0"
repository = "https://github.com/yourname/my_awesome_crate"
homepage = "https://github.com/yourname/my_awesome_crate"
documentation = "https://docs.rs/my_awesome_crate"
keywords = ["rust", "awesome", "example"]
categories = ["algorithms", "data-structures"]

然后登录crates.io并获取API token：

cargo login
# 在提示下输入你在 https://crates.io/me 上生成的token

检查并发布

在发布之前，运行以下命令确保一切就绪：

cargo publish --dry-run  # 模拟发布，检查是否有错误
cargo package --list      # 查看哪些文件会被打包进crate

正式发布：

cargo publish

crate一旦发布，是不能删除的——你只能发布新版本（yank可以标记某个版本"已废弃"，但不会移除包文件）。这个策略确保了任何人的构建不会因为依赖被删除而不可复现。

命名艺术：crates.io上的crate名称遵循"先到先得"原则。好的crate名应该短、好记、体现功能。比如serde（serialize/deserialize）、tokio（日语"时"的读音）。如果你的crate名和功能高度相关，用户会更愿意尝试——但要注意不要侵犯他人的商标权。

cargo子命令 — clippy、fmt、audit、deny、nextest

Cargo本身是"可扩展的"——任何名字为cargo-xxx的可执行文件放到PATH中，就可以通过cargo xxx来调用。以下是最常用的cargo子命令：

命令 安装方式 用途
cargo clippy rustup component add clippy 代码lint，比编译器更严格的检查
cargo fmt rustup component add rustfmt 自动格式化代码（官方风格）
cargo audit cargo install cargo-audit 检查依赖中已知的安全漏洞
cargo deny cargo install cargo-deny 依赖许可审查、安全审计、禁止特定crate
cargo nextest cargo install cargo-nextest 下一代测试运行器，比cargo test更快更智能
cargo expand cargo install cargo-expand 展开宏，查看宏展开后的代码
cargo tree 内置 显示依赖树
cargo outdated cargo install cargo-outdated 检查哪些依赖有更新的版本

工作空间（Workspace）

当一个项目包含多个相互关联的crate时，工作空间是标准的管理方式。在根目录的Cargo.toml中定义：

[workspace]
members = [
    "core",
    "cli",
    "web",
]

工作空间的好处：所有成员共享同一个Cargo.lock和target/目录，避免重复编译；cargo test可以一次性运行所有成员的测试；成员之间可以用path依赖相互引用。
