开篇：互联网的"数字化石"

大家好，我是你们的讲师。如果把大模型预训练比作炼油工业，那么 Common Crawl 就是我们这个时代的"石油矿藏"。

这个比喻非常贴切。原油不能直接加进汽车油箱，Common Crawl 的原始数据也不能直接喂给大模型。但正如现代工业离不开石油，2026 年几乎每一个有竞争力的预训练模型——不管它叫 GPT-5 还是 Llama 4——都以 Common Crawl 作为最主要的数据源。Common Crawl 通常贡献了训练数据中 60%-80% 的 token 量。

Common Crawl 是什么？简单来说，它是一个非营利组织，从 2011 年开始每个月爬取整个互联网的公开页面，并将这些爬取结果免费提供给全世界。截至 2026 年，每次月度爬取产生约 30-50 亿个页面，约 300TB 的原始数据。从 2011 年至今积累的数据总量已经超过了 50PB。

本章我们来深度解剖 Common Crawl——它的内部格式、数据存储方式、核心挑战，以及如何从这座"数据矿山"中提炼出高质量的预训练语料。

5.1 Common Crawl 的数据格式体系

Common Crawl 提供了三种数据格式，分别服务于不同的使用场景。理解这三者的区别，是使用 Common Crawl 的第一步。

格式 全称 内容 用途 典型压缩比
WARC Web ARChive 完整的 HTTP 响应（含请求头、响应头、HTML 正文） 需要原始页面信息的场景 ~5:1
WET Web Extract Text 从 HTML 中提取的纯文本（去除标签、脚本、样式） 大规模文本分析、预训练数据提取 ~10:1
WAT Web Archive Metadata 页面的元数据（URL、抓取时间、响应码、链接图等） 链接分析、图分析、元数据筛选 ~8:1

三种格式的关系是递进的：WARC 是"母版"，包含最完整的信息；WET 是从 WARC 提取的纯文本，是最常用的预训练数据源；WAT 只保留了元数据，适合做大规模的分析和过滤。

一个 WARC 记录的典型结构：
  WARC/1.0
  WARC-Type: response
  WARC-Date: 2024-03-15T12:34:56Z
  WARC-Target-URI: https://example.com/article
  Content-Length: 8642

  HTTP/1.1 200 OK
  Content-Type: text/html

  <html>
  <head><title>示例文章</title></head>
  <body>
    <article>
      <h1>文章正文</h1>
      <p>这是文章的正文内容...</p>
    </article>
  </body>
  </html>

  对应的 WET 版本（纯文本提取后）：
  WARC/1.0
  WARC-Type: conversion
  WARC-Target-URI: https://example.com/article

  示例文章
  文章正文
  这是文章的正文内容...
效率建议：对于大多数预训练场景，直接使用 WET 格式即可，不需要从 WARC 开始处理。但如果你需要提取页面特定的结构信息（比如只提取 <article> 标签内的内容），或者想自己实现比 Common Crawl 官方更好的文本提取器，就需要从 WARC 开始处理。我个人建议：先用 WET 快速获得基线数据，再逐步优化。

5.2 如何访问 Common Crawl

Common Crawl 的存储路径非常简单直观。所有数据都存放在 Amazon S3 上，可以通过 s3://commoncrawl/ 路径公开访问（无需 AWS 账号认证）。

数据按"爬取批次"组织，每个批次以 crawl-data/CC-MAIN-{日期} 的形式命名。例如：

常见爬取批次路径：
  s3://commoncrawl/crawl-data/CC-MAIN-2024-10/
  s3://commoncrawl/crawl-data/CC-MAIN-2024-18/
  s3://commoncrawl/crawl-data/CC-MAIN-2024-22/
  s3://commoncrawl/crawl-data/CC-MAIN-2024-33/
  s3://commoncrawl/crawl-data/CC-MAIN-2025-04/
  s3://commoncrawl/crawl-data/CC-MAIN-2025-13/

  每个批次内包含：
    ├── warc/       # WARC 文件（.warc.gz）
    ├── wet/        # WET 文件（.warc.wet.gz）
    ├── wat/        # WAT 文件（.warc.wat.gz）
    └── robots.txt  # 该爬取批次的 robots 规则

Common Crawl 大约每 1-2 个月发布一次新的爬取批次。每个批次的 WARC 文件数量在 50,000-80,000 个之间，每个文件压缩后约 1GB。

访问工具方面，推荐使用 s3cmd 或 aws s3 CLI。由于 Common Crawl 的 S3 bucket 是公开的，下载不需要认证。但对于大规模下载批量数据，建议使用 S3 直接复制（s3-dist-cp）的方式，或者通过 EC2 实例在 AWS 内部下载（免流量费）。

流量陷阱：千万不要从中国大陆直接 S3 下载 Common Crawl 的几百 TB 数据！网络流量费用和带宽限制会让你崩溃。正确的做法是：租用 AWS 东京或新加坡区域的 EC2 实例，在 AWS 内部网络环境下下载、处理数据，然后再将处理结果压缩后传输回国。或者更好的方案是——直接使用清洗后的公开数据集（如 RedPajama、Dolma），它们已经替你完成了大部分工作。

5.3 Common Crawl 的核心挑战

Common Crawl 既慷慨又"混乱"。它的慷慨在于：免费、开放、几乎无限的互联网数据。它的"混乱"体现在以下几个维度：

挑战一：海量多语言混杂

Common Crawl 每次爬取覆盖约 100+ 种语言。在一个批次的 30 亿页面中，英语约占 40%-45%，中文约占 5%-8%，其余是其他语言。如果不对数据进行语言过滤，模型会被各种语言的混合文本搞糊涂。语言识别是 CC 清洗的第一步——我们在第 9 章会深入讨论这个问题。

挑战二：HTML 噪音与模板内容

原始页面中包含大量与内容无关的元素：导航栏、广告位、侧边栏、页脚、Cookie 声明、评论区等。Common Crawl 自带的 WET 提取器只做了基础的标签剥离，保留了大量"噪音"文本。很多页面的"有效内容"只有总文本量的 20%-30%。

挑战三：垃圾内容与 SEO 污染

互联网上有大量为了搜索引擎排名而生成的"内容农场"页面。这些页面通常由 AI 或廉价写手批量生产，内容质量极低——语义逻辑混乱、关键词堆砌、重复套话。更糟的是，这些内容往往披着"正规文章"的外衣，单纯靠规则很难识别。

挑战四：跨批次重复

同一个网页可能会被多次爬取（因为网站内容没有变化）。两期连续的 CC 快照之间可能有 40%-60% 的页面重复。如果不做跨批次去重，模型会反复学习相同的页面，造成严重的过拟合。

踩坑故事：2023 年，我们团队第一次尝试从 Common Crawl 构建预训练数据集。我们天真地下载了最近 3 个月的 CC 快照（约 600TB 压缩数据），简单跑了一遍 C4 风格的清洗规则，以为就能得到还不错的训练数据。结果发现：清洗后的 token 里，约 15% 的内容看起来是"正常的文章开头然后突然变成了购物推荐列表"——这是 HTML 清洗不彻底导致的模板残留。另外，跨批次去重后发现 有 50%+ 的 token 是重复的——同一个知乎回答出现在了两期 CC 快照中。这次教训让我们深刻地意识到：Common Crawl 的清洗不是一个"跑一遍脚本就行"的事情，而是一个需要多层过滤的工程系统。

5.4 FineWeb：CC 清洗的终极案例

2024 年，HuggingFace 发布了 FineWeb 数据集和配套的清洗流水线，这可能是目前最完整的 Common Crawl 处理方案。FineWeb 将一个批次的 CC 原始数据（约 100TB 的原始 WET），清洗到了约 5-7TB 的高质量 token——缩减了 95% 以上。

FineWeb 的清洗管线如下：

FineWeb 处理管线（从原始 CC 到高质量数据）：

  原始 CC WET（100TB）
      │
      ▼
  1. 语言过滤（fastText）
      │ 保留英语、中文、法语等主要语言
      │ 丢弃低资源语言和混合语言页面
      │ ~60TB 通过
      ▼
  2. URL 过滤（Blocklist）
      │ 屏蔽已知的垃圾域名、成人内容、内容农场
      │ ~50TB 通过
      ▼
  3. 精确去重（文档级 SHA-1）
      │ 移除完全相同的文档
      │ ~35TB 通过
      ▼
  4. 模糊去重（MinHash + LSH）
      │ 移除 90%+ 相似度的近重复文档
      │ ~20TB 通过
      ▼
  5. 启发式过滤（行级 + 文档级规则）
      │ 标点密度、符号率、单词长度、重复 n-gram 等
      │ ~12TB 通过
      ▼
  6. 分类器评分（基于 FastText 或 BERT 的质量分类器）
      │ 对每个文档进行质量评分（0-1）
      │ 保留评分 top 60% 的文档
      │ ~7TB 通过
      ▼
  最终高质量输出（5-7TB）

注意看这个缩减曲线：从 100TB 到 7TB，缩减了 93%。这意味着每 15 个 CC 页面中只有 1 个最终进入了训练集。这不是因为 FineWeb 太"挑剔"——而是因为互联网上的大部分页面确实不适合作为训练数据。

用 FineWeb 的经验教训：FineWeb 给我们的最大启示不是它的具体清洗参数，而是它的模块化设计。每个清洗步骤都是独立的模块，可以单独启用、禁用或替换。你可以替换语言检测器（比如从 fastText 换成 CLD3），可以调整模糊去重的阈值，可以替换质量分类器的模型。不要试图找到一个"万能参数"——应该建立一个可组合的清洗系统。这是我们第 9-14 章要深入展开的内容。

5.5 为什么 95% 的 CC 数据被丢掉？

很多第一次接触 FineWeb 的朋友会问一个问题：丢掉 95% 的数据，难道不会丢失有价值的信息吗？

答案是：会，也不完全会。我们需要理解"丢失"的到底是什么：

  • 50%-60% 的原始数据是"非内容"：导航菜单、广告、模板化页脚、版权声明、Cookie 横幅等。这些"文本"虽然以文字形式存在，但本质上不是人类可读的"内容"，丢弃它们没有损失。
  • 20%-30% 是重复内容：同一篇文章的多次出现、同一网站的相似页面、多个网站的互相转载。丢弃重复内容不仅没有损失，还能提升训练效率。
  • 10%-15% 是低质量杂音：内容农场的 SEO 堆积、机翻文本、碎片化的错误格式页面。这些内容实际上是有害的——它们会降低模型生成质量。
  • 仅剩约 5%-10% 是真正有价值的内容：但它们分散在海量的噪音中，需要通过多层清洗才能"筛"出来。

换句话说，丢掉 95% 的 Common Crawl 数据，丢掉的几乎是"噪音"，而保留的是"信号"。这和我们从原油中提炼汽油是一个道理——原油里只有 20%-30% 的汽油组分，其余的是沥青、重油和化工原料。你不会因为"浪费了 70% 的原油"而心疼，因为那些部分本来就不适合做燃料。

核心认知升级：Common Crawl 的 95% 缩减率不是"效率低下"的标志，而是"质量过滤严格"的标志。如果你的清洗流水线只去掉了 50% 的原始数据，那很可能说明你过滤得不够严格——许多噪音仍然留在数据中。当你的模型在训练后出现莫名其妙的质量问题时，问题往往不在于"数据不够多"，而在于"坏数据太多了"。

5.6 踩坑：中文 CC 的特殊问题

如果你处理的是 Common Crawl 中的中文部分，还有一些"中国特色"的挑战要特别注意：

  • 编码问题：部分老旧的中文网站仍然使用 GB2312 或 GBK 编码，但 CC 的 WET 文件可能会将这些错误地标记为 UTF-8，导致中文内容变成乱码。
  • 繁简混杂：同一个页面可能同时包含繁体和简体中文。模型在训练时会在两种书写系统之间摇摆，影响中文生成的稳定性。
  • 内容农场的泛滥：中文 SEO 内容农场比英文世界更为严重。大量的"伪原创"文章通过同义词替换或语序调整生成，虽然看起来"不重复"，但信息熵极低。
  • 机器翻译污染：大量英文文章被机器翻译成中文并发布。这些翻译往往生硬、不自然，混入训练数据后会污染模型的中文表达能力。
中文 CC 的编码踩坑：我曾经处理过一批从 Common Crawl 中提取的中文文本。在统计 token 数量时发现结果出奇地少——后来检查才发现，大约 30% 的中文页面因为编码被错误识别为 UTF-8，导致内容变成了类似于"æµè¯"的乱码字符串。这些乱码被 tokenizer 切碎成了大量无意义的 token，白白浪费了宝贵的训练容量。在处理中文 CC 时，一定要增加编码检测步骤，用 chardet 或 cchardet 对提取的文本做编码嗅探，拒绝那些被错误解析的页面。

本章我们深入解剖了 Common Crawl——预训练数据的"石油矿藏"。我们了解了 WARC/WET/WAT 三种格式的差异和用途，学会了如何访问 CC 的数据，认识到了从原始 CC 到高质量训练数据的巨大鸿沟。FineWeb 案例向我们展示了 95% 缩减率的清洗流水线，而那些踩坑故事则提醒我们：Common Crawl 的价值不是"拿来就能用"，而是"经过精心提炼才能成为训练数据"。
