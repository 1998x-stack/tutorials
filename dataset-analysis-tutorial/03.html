第3章：数据集元信息与特征系统

上一章我们加载了IMDb和CIFAR-10数据集，并且通过 print(dataset) 瞥见了 features 这个神秘属性。当时我们看到text列的类型是 Value(dtype='string')，label列是 ClassLabel(names=['neg', 'pos'])。但你有没有想过——这些类型信息是怎么来的？如果加载的数据里某一列看起来是整数但实际上是字符串，datasets会怎么处理？

这些问题背后是datasets库的核心基础设施——特征（Features）系统。它不仅是数据类型的"元数据"，更决定了数据如何在Arrow后端中被编码、序列化、访存和在框架间转换。不理解Features，你就不可能真正掌控datasets库。这一章我们把它彻底讲透。

Features体系全景

Features 是datasets库中每一列数据的"类型声明"。它告诉库：这一列存储的是什么类型的数据？允许哪些值？有哪些约束条件？

你可以把Features理解为 数据库中的Schema——它定义了数据集中每一列的名称、类型和约束。但比数据库Schema更灵活的是，Features支持嵌套和组合，能够表达非常复杂的结构化数据。

下表是datasets库中最核心的Feature类型一览：

Feature类型 描述 典型用途 示例
Value 标量值类型 数值、字符串、布尔 Value("int32")
ClassLabel 分类标签 类别型目标变量 ClassLabel(names=["pos","neg"])
Sequence 序列/列表 变长序列、词ID列表 Sequence(Value("int32"))
Array2D 二维数组 灰度图矩阵 Array2D((32,32), "float32")
Array3D 三维数组 RGB图像张量 Array3D((32,32,3), "uint8")
Image 图像数据 JPEG/PNG图片 Image()
Audio 音频数据 WAV/MP3录音 Audio(sampling_rate=16000)
Translation 翻译对 平行语料 Translation(languages=["en","zh"])
TranslationVariableLanguages 多语言翻译 多语言翻译对 TranslationVariableLanguages()

每一种Feature都对应着一个Arrow数据类型。当你定义或查看一个数据集的features时，你实际上是在指定Arrow列的类型。让我们逐一深入。

核心原则：Features不仅是"元数据"，它还直接决定了数据在内存中的布局方式、序列化效率和框架兼容性。选择正确的Feature类型，和选择正确的数据库列类型一样重要。

Value类型详解

Value(dtype) 是最基础、最常用的Feature类型。它映射到Arrow中的标量类型——一个单元格只存一个值的那种。

支持的dtype

datasets库支持的dtype与Arrow的类型系统一一对应：

dtype Arrow类型 Python类型 说明
int8 Int8 int 8位有符号整数
int16 Int16 int 16位有符号整数
int32 Int32 int 32位有符号整数（最常见）
int64 Int64 int 64位有符号整数
float16 Float16 float 半精度浮点
float32 Float32 float 单精度浮点（最常见）
float64 Float64 float 双精度浮点
string LargeUtf8 str UTF-8字符串
bool Boolean bool 布尔值
binary Binary bytes 二进制数据
timestamp[ns] Timestamp datetime 时间戳（纳秒精度）

类型推断与检测

当你用 load_dataset() 加载数据时，datasets库会自动推断每一列的dtype。推断逻辑大致如下：

# 加载一个CSV文件时，datasets自动推断类型
dataset = load_dataset("csv", data_files="mydata.csv")
print(dataset["train"].features)
# {'age': Value(dtype='int64'), 'name': Value(dtype='string'), 'score': Value(dtype='float64')}

自动推断很方便，但也会踩坑。比如一列数据中大部分是数字，但某一行混入了一个"NULL"字符串，导致整个列被推断为string而非数字类型。或者一列邮政编码是纯数字（"100000"），但期望作为字符串处理，自动推断会把它当作整数，前导零被吃掉。

# 手动指定Features，避免自动推断的陷阱
from datasets import Features, Value

my_features = Features({
    "zip_code": Value("string"),   # 邮政编码需要保持前导零
    "age": Value("int32"),
    "income": Value("float64"),
})
dataset = load_dataset("csv", data_files="data.csv", features=my_features)
真实踩坑案例：某团队加载自定义CSV文件训练分类器，所有数值列都被自动推断为string类型。他们直接用这个数据训练模型，损失曲线看起来"异常平滑"。排查了三天才发现——模型实际上是在把字符串当作分类特征处理，完全没有学到数值关系。正确的做法是加载时指定features，或者用 .cast() 方法显式转换类型。

cast() 类型转换

如果已经加载完了才发现类型不对，可以用 cast() 或 cast_column() 修正：

# 将age列从int64转为float32
dataset = dataset.cast_column("age", Value("float32"))

# 批量转换多个列
new_features = dataset["train"].features.copy()
new_features["age"] = Value("float32")
new_features["score"] = Value("int32")
dataset["train"] = dataset["train"].cast(new_features)

ClassLabel与分类特征

ClassLabel 是分类任务的标配Feature类型。它存储一个类别名称列表（names），并在内部维护 整数ID → 类别名称 和 类别名称 → 整数ID 的双向映射。

定义与使用

from datasets import ClassLabel

# 通过名称列表定义
label = ClassLabel(names=["negative", "neutral", "positive"])
print(label.num_classes)  # 3
print(label.names)        # ['negative', 'neutral', 'positive']

# 映射操作
print(label.str2int("positive"))  # 2
print(label.int2str(0))            # negative

# 批量转换
ids = label.str2int(["negative", "positive", "neutral"])
print(ids)  # [0, 2, 1]

GLUE MRPC实战

GLUE基准中的MRPC（微软研究释义语料库）数据集是ClassLabel的典型应用场景——判断两个句子是否为同义关系：

from datasets import load_dataset

mrpc = load_dataset("glue", "mrpc")
print(mrpc["train"].features["label"])
# ClassLabel(names=['not_equivalent', 'equivalent'], id=None)

# 标签分布
train = mrpc["train"]
label_counts = {}
for label in train["label"]:
    name = train.features["label"].int2str(label)
    label_counts[name] = label_counts.get(name, 0) + 1

print(label_counts)
# {'not_equivalent': 2177, 'equivalent': 2751}
实用技巧：在数据探索阶段，用 int2str 把标签ID转为名称，可以让你的数据分析代码更具可读性。特别是在画混淆矩阵或做错误分析时，显示类别名称而不是数字会让结果一目了然。

不均衡标签的处理

ClassLabel 本身不提供任何处理不均衡的机制——它只是标签的元信息。但知道标签分布是后续采样策略（分层采样、过采样/欠采样）的前提。我们会在第7章的分层采样中深入这个话题。

复杂类型：Sequence, Array2D, Translation

真实世界的数据很少是简单的"一列一个标量"。一条样本可能包含变长的文本token序列、图像矩阵、多种语言的翻译对。这时候就需要复杂Feature类型出场了。

Sequence 变长序列

Sequence(feature, length=-1) 表示一列中的每个单元格是一个列表，列表中每个元素的类型由 feature 指定。length 为 -1 表示变长。

from datasets import Sequence, Value, Features

# 变长整数序列（如token IDs）
features = Features({
    "input_ids": Sequence(Value("int32")),
    "attention_mask": Sequence(Value("int32")),
    "label": Value("int32"),
})

# 定长序列（固定长度的坐标）
coords = Sequence(Value("float32"), length=3)  # 每个样本3个float

Sequence 在NLP数据集中无处不在——每一条文本经过tokenizer之后产生的input_ids就是一个变长整数序列。

Array2D / Array3D 图像矩阵

Array2D 和 Array3D 用于存储固定尺寸的多维数组，对应灰度和彩色图像：

from datasets import Array2D, Array3D

# 32×32灰度图
gray_features = Array2D((32, 32), "float32")

# 32×32×3 RGB图像
rgb_features = Array3D((32, 32, 3), "uint8")

# 在实际数据集中
cifar_features = Features({
    "img": Array3D((32, 32, 3), "uint8"),
    "label": ClassLabel(names=[
        "airplane", "automobile", "bird", "cat", "deer",
        "dog", "frog", "horse", "ship", "truck"
    ])
})

注意Array2D/3D和Image的区别：Image存储的是压缩后的图像字节（JPEG/PNG格式），访问时实时解码为PIL Image对象；Array2D/3D存储的是解码后的像素数值数组。前者体积小但访问有解码开销，后者体积大但访问快。选择哪种取决于你的使用场景——如果频繁读取图像像素（比如做数据增强），Array3D更合适；如果只是做元数据索引，Image更省空间。

Translation 双语数据

机器翻译数据集中，每条样本包含源语言和目标语言的一对句子：

from datasets import Translation, Features

# 英中翻译
trans_features = Features({
    "translation": Translation(languages=["en", "zh"]),
})

# 实际使用
sample = {"translation": {"en": "Hello world", "zh": "你好世界"}}

Translation 本质上是一个 dict 类型的列，键是语言代码，值是对应语言的文本。在多语言翻译的场景下，可以用 TranslationVariableLanguages ——它允许不同样本使用不同的语言对组合。

Audio 特征

音频数据集使用 Audio 特征：

from datasets import Audio, Features

# 定义音频特征，指定采样率
audio_features = Features({
    "audio": Audio(sampling_rate=16000),
    "transcription": Value("string"),
})

# 访问音频数据时自动解码为numpy数组
dataset = load_dataset("librispeech_asr", "clean", split="train")
sample = dataset[0]
print(sample["audio"])
# {'path': '/path/to/file.flac', 'array': array([...]), 'sampling_rate': 16000}

Audio 特征背后完成了一个重要工作——自动解码。无论源文件是WAV、MP3还是FLAC格式，加载后统一的输出是一个包含原始波形数组和采样率的字典。这消除了手动处理音频编解码的负担。

设计哲学：datasets库的复杂Feature类型遵循一个统一的原则——让数据"看起来"是Python原生的。你访问Sequence时得到list，访问Array2D时得到numpy数组，访问Audio时得到包含array的dict。背后复杂的序列化/反序列化工作由库自动完成。

自定义Features

在实际项目中，你的数据很少是教科书中那种规整的"文本+标签"格式。你可能有一个数据集，其中每条样本包含：一段文本、一个类别标签、一张图像和一个嵌套的元信息字典。这时就需要自定义Features了。

构建自定义Schema

from datasets import Features, Value, ClassLabel, Image, Sequence

my_features = Features({
    "id": Value("int64"),
    "text": Value("string"),
    "label": ClassLabel(names=["entailment", "neutral", "contradiction"]),
    "image": Image(),
    "bbox": Sequence(Value("float32"), length=4),    # [x1, y1, x2, y2]
    "metadata": {
        "source": Value("string"),
        "timestamp": Value("timestamp[ns]"),
        "annotator_id": Value("int32"),
    }
})

这个Features定义了一个非常接近真实场景的数据集schema：包含文本、图片、边界框和嵌套的元信息。有了它，你可以在 load_dataset 时就约束好数据的类型结构：

dataset = load_dataset(
    "json",
    data_files="my_multimodal_data.jsonl",
    features=my_features,
)

# 验证特征是否正确
print(dataset["train"].features)
# 所有列的类型都和我们定义的一致

# 访问嵌套字段
sample = dataset["train"][0]
print(sample["metadata"]["source"])  # 直接访问嵌套键
自定义Features的陷阱：如果你指定的Features与数据文件的实际内容不匹配，datasets可能会静默地进行类型转换或截断。例如，如果你把一列声明为 Value("int32") 但数据中包含了浮点数，会被截断为整数而不会报错。始终在加载后通过 print(dataset["train"].features) 和抽样检查来验证类型匹配。

嵌套的Features

Features支持任意深度的嵌套。这在处理JSON格式的复杂数据时非常有用：

nested_features = Features({
    "conversation": Sequence({
        "role": Value("string"),
        "content": Value("string"),
        "metadata": {
            "timestamp": Value("int64"),
            "model": Value("string"),
        }
    }),
    "summary": Value("string"),
})

# 访问嵌套的对话数据
# sample["conversation"][0]["role"] -> "user"
# sample["conversation"][0]["content"] -> "你好，请介绍一下你的功能"

这种嵌套结构在处理对话数据集（如ShareGPT格式）时非常常见。每条样本包含多轮对话，每轮对话又包含角色、内容和元信息。用嵌套Features可以完整地表达这种结构。

一个真实的"踩坑"故事

回到本章开头提到的问题——某团队用datasets加载自定义CSV训练分类器时遇到的神秘的"异常平滑"损失曲线。

当时团队收集了一批用户行为数据，总共50万条，包含用户年龄、收入、活跃天数等数值特征和最终的转化标签。团队用 load_dataset("csv", data_files="users.csv") 加载数据后直接开始训练LightGBM。第一次训练就发现损失曲线和预期不同——下降得太"完美"了，几乎是一条平滑的弧线。

团队leader怀疑数据有问题，让一位工程师检查数据特征。那位工程师打印了 dataset["train"].features，发现一个惊人的事实：所有列的类型都是string。

原因在于CSV文件中个别行存在格式异常——某一行的年龄字段是未知的"unknown"字符串，某一行的收入字段有逗号"45,000"。datasets的自动推断器碰到这种情况后，"安全起见"把所有列都推断成了string。

LightGBM在训练时把所有string列当作分类特征处理，为每一个唯一的字符串值学习一个单独的权重。模型确实"拟合"了数据——但它学到的是"哪个用户ID属于哪个类别"，而不是"年龄和收入如何影响转化"。这种"虚假拟合"的损失曲线看起来很美，但完全没有泛化能力。

教训：永远不要信任自动类型推断。始终在加载数据后检查 features，特别是从CSV或JSON文件加载时。如果发现类型不对，用 cast_column() 或自定义Features修正。在数据进入模型之前做一次"类型审计"，可以避免后续数天的无效调试。

修正方法很简单：

from datasets import Features, Value, ClassLabel

# 定义正确的类型
correct_features = Features({
    "age": Value("int32"),
    "income": Value("float64"),
    "active_days": Value("int32"),
    "label": ClassLabel(names=["not_converted", "converted"]),
})

# 重新加载
dataset = load_dataset("csv", data_files="users.csv", features=correct_features)

这次加载时，datasets会尝试把CSV中的字符串转换为声明的类型，遇到无法转换的值会抛出异常。团队就能在数据加载阶段发现问题，而不是在训练结束后。

本章小结

Features 系统是datasets库的基石。这一章我们详细拆解了它的每个组成部分：

  • Value：标量类型，支持从int8到timestamp的全谱系
  • ClassLabel：分类标签，提供int2str/str2int双向映射
  • Sequence、Array2D/3D、Translation、Audio：复杂类型的取舍和使用场景
  • 自定义Features：用嵌套结构表达现实世界的数据schema

掌握了Features，你就掌握了datasets库的"类型语言"。下一章我们将深入Features的物理实现——Apache Arrow，理解为什么列式存储能带来数量级的性能提升，以及如何在8GB内存的笔记本上处理百GB级别的数据集。
