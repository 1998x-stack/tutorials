第16章 Cache设计与优化

上一章我们讲完了存储层次和局部性原理——现在大家已经知道为什么我们需要缓存了。这一章，我们要深入缓存内部，看看Cache到底是怎么设计的，以及工程师们为了挤出一丁点儿命中率而付出的天才般的努力。坦白说，Cache设计是计算机体系结构中最精妙的领域之一——每一个比特的增减、每一条路径的选择，都充满了权衡。

Cache的三大映射方式

先问一个最基本的问题：当CPU发出一个内存地址，缓存怎么知道数据是否存在？这就是映射方式要解决的问题。核心是把内存地址翻译成缓存中的位置。

直接映射（Direct-Mapped）：每个内存块只能缓存在缓存中的一个位置。公式很简单：Cache Line Index = (Block Address) mod (Number of Cache Lines)。地址分成三个部分：Tag | Index | Offset。优点是硬件极简——只需要一个比较器就能完成查找。缺点是如果两个经常访问的地址映射到同一个位置，会产生冲突缺失，反复互踢——这叫"cache thrashing"。

全相联（Fully Associative）：任意内存块可以放在缓存的任意位置。查找时需要对比所有Cache Line的Tag——也就是说需要N个比较器并行工作。这在硬件上极其昂贵（面积和功耗），但冲突缺失为零（只有容量缺失和强制缺失）。实际上，全相联只用于TLB这样的小型结构。

组相联（Set-Associative）：最实用的折衷方案。缓存被分成若干组（Set），每个组包含N个Cache Line（N-way）。每个内存块可以映射到某一组，但组内有N个位置可选。地址现在是：Tag | Set Index | Offset。典型的现代处理器使用8-way到16-way组相联。

地址分解示例（4KB缓存，16字节块，4-way组相联）：
  Offset = log2(block_size) = log2(16) = 4 bits
  Number of sets = cache_size / (associativity × block_size)
                  = 4096 / (4 × 16) = 64 个组
  Set Index = log2(64) = 6 bits
  Tag = address_width - Index - Offset

  虚拟地址 (32-bit):
  [31:10] Tag (22 bits) | [9:4] Index (6 bits) | [3:0] Offset (4 bits)

替换策略

当组内所有位置都被占满，需要载入新的Cache Line时——必须踢掉一个。问题来了：踢谁？

LRU（Least Recently Used）：理论最优的近似，踢掉最久未访问的那一个。实现需要跟踪组内所有行的访问顺序。对于低路数（2-way、4-way）完美可行；对于高路数（16-way以上），硬件代价太高。全相联下几乎不可能实现。

伪LRU（Pseudo-LRU / Tree-based PLRU）：用一个二叉搜索树来近似LRU——每个内部节点有一个比特表示"上次走的是左子树还是右子树"。查找和更新都很快，只需要O(log N)的存储。实践中与真实LRU的性能差距不到1%。

随机替换（Random）：直接随机选一个踢掉。讽刺的是，对于大容量缓存，随机替换的性能几乎和LRU一样好——因为大缓存的冲突概率本身较低，偶尔的"坏"替换影响不大。而且硬件实现几乎零成本。

RRIP（Re-Reference Interval Prediction）：这是现代Intel处理器（自Sandy Bridge起）实际使用的策略。它给每个Cache Line一个"重新引用间隔预测值"，新载入的行有此值，被访问时重置——值小的表示"很快会再次被用"，值大的表示"可能很久不会再用"。替换时选值最大的。你可以理解成一种"动态预测的LRU"。

真实故事：Intel从Sandy Bridge微架构开始，将L3缓存的替换策略从传统的LRU改为了RRIP。为什么？因为某些工作负载（如数据库哈希连接）会产生大量"扫描式"的数据访问，每次扫描新的数据，就把LRU中的旧数据全部踢走——尽管这些旧数据马上就会被重新用到。RRIP通过识别引用模式（区分扫描流和重用流），大大缓解了这个问题。这个改动在缓存密集型负载上带来了5-15%的性能提升。你看，一个替换策略的改进，实际效果可能比增加缓存容量更显著。

写入策略

读操作很简单：命中就返回数据，不命中就从下一级取。写操作要复杂得多——因为CPU不仅要读，还要写；而且写操作可能发生在缓存里，也可能直通内存。这就需要用不同的写入策略来处理。

写直达（Write-Through）：写操作同时写入缓存和下一级存储。优点是：一致性模型简单（下级总是有最新数据）。缺点是：每次写操作都要访问下级存储，带宽消耗大，写操作延迟高。

写回（Write-Back）：写操作只写入缓存，该Cache Line被标记为"脏"（Dirty）。只有当该行被替换（evict）时，才把脏数据写回下一级。优点：大大减少写带宽，写操作速度快（只写入L1）。缺点：一致性模型复杂，替换时需要两步操作。

写入缺失时还有两个子策略：写分配（Write-Allocate）先把内存块载入缓存再写入，适用于写回缓存；不写分配（No-Write-Allocate）直接写入下一级，不载入缓存，适用于写直达缓存。

现代处理器的实践：L1/L2/L3都使用写回+写分配。为什么？因为写操作也有局部性。你的代码可能频繁更新一个变量——如果每次都穿透到内存，太浪费了。写回策略让这些更新在缓存中完成高速聚合。

Cache性能优化十大技术

好，现在进入最精妙的环节。工程师们为了减少缺失率、降低缺失惩罚，开发出了下面这些天才般的优化技术：

  1. 更大的块大小：利用空间局部性，一次取更多数据。但块太大会浪费带宽（取来的数据可能没用），同时增加缺失惩罚。
  2. 更高的相联度：减少冲突缺失。从直接映射到2-way就能消除大量冲突，再到4-way、8-way效果递减。代价是更慢的访问时间。
  3. Victim Cache：一个很小的全相联缓存（通常4-16行），用于"回收"被替换出去的Cache Line。如果刚被踢出的行又被用到（一种常见模式），可以在victim cache中找到，避免一次L2访问。
  4. 预取（Prefetching）：硬件在CPU实际需要之前，提前把数据从下一级拉上来。Intel的硬件步长预取器（hardware stride prefetcher）能识别固定步长的访问模式（如数组遍历），提前加载后面N次迭代需要的数据。
  5. 非阻塞缓存（Non-blocking Cache / Hit-Under-Miss）：在等待缺失数据的返回期间，允许后续的缓存命中继续服务。理想情况下可以完全隐藏缺失延迟。
  6. 多级包含策略：Inclusive（上级是下级的子集，L3包含所有L2的数据）、Exclusive（各层互不重叠，AMD早期使用）、NINE（Non-Inclusive Non-Exclusive，最常用，灵活但简单）。
  7. Critical Word First：块中的某个字首先被CPU请求。不必等整个块传输完，这个关键字优先返回给CPU，CPU可以立即继续执行。
  8. Early Restart：类似Critical Word First，但给整个块中请求的word优先服务。
  9. Merging Write Buffer：多个写操作合并成一次写操作。如果程序连续写了4个相邻字节，合并写缓冲区可以把它们合并为一个32-bit写操作。
  10. Way Prediction：在组相联缓存中，预测哪一路可能命中，只比较这一路的Tag，节省比较的延迟和功耗。

Cache一致性基础（Preview）

在单核处理器中，缓存设计相对直接。但一旦进入多核世界，麻烦来了：同一个内存地址的副本可能同时存在于多个核心的L1/L2中。如果其中一个核心修改了它的副本，其他核心中的副本就成了"过时数据"。这就是Cache一致性（Cache Coherence）问题。

MESI协议是最经典的解决方案。每个Cache Line有四个状态：

  • M (Modified)：仅此缓存有副本，且已被修改（脏）。
  • E (Exclusive)：仅此缓存有副本，未修改（干净）。
  • S (Shared)：多个缓存有副本，都干净。
  • I (Invalid)：此副本无效。

当核心A要写一个Shared的行时，需要先发送Invalidate消息给其他核心，等收到确认后才写入。这与我们之前讲的写回策略配合——通过"窥探总线"（Bus Snooping）来监听其他核心的请求。核心间通过缓存一致性协议来协调，保证每个核心看到的内存视图是一致的。

False Sharing（伪共享）是并发编程中最隐蔽的性能杀手。考虑两个线程，线程A修改变量x，线程B修改变量y。如果x和y恰好在同一个Cache Line中，每次A写x时，这个Cache Line在B的缓存中被标记为Invalid。B要写y时，必须先重新载入——然后又把A的缓存搞失效。两个线程像"打乒乓球"一样来回传递这个Cache Line，尽管它们访问的是不同的变量。伪共享可以导致30-60%甚至更高的性能损失。

// 伪共享演示 (C代码风格)
struct Data {
    int x;  // 线程A修改
    int y;  // 线程B修改
};

// 修复前：x和y在同一cache line中
struct Data d;

// 修复后：用padding分开
struct FixedData {
    int x;
    char padding[64];  // 64字节 => 一个cache line
    int y;
};

// 实际效果：修复前，两个线程同时写x和y时
// 每秒约 2000万次操作（互相拖累）
// 修复后，每秒约 8000万次操作（几乎独立）
// 性能差异：4x！

我曾经在生产环境中遇到过这个问题。一个多线程日志系统里，两个线程各自写不同的日志级别计数器（errorCount和warnCount），结果这两个计数器碰巧在同一个Cache Line里。系统吞吐量只有预期的40%。排查了很久才通过perf的cache-miss事件发现这个"交互陷阱"——两个计数器明明没有共享变量，却因为Cache Line的粒度互相拖累。修复起来就是加了一行padding，性能立刻提升了2.5倍。False sharing是彻头彻尾的"体系结构级别的Bug"——代码看起来完全正确，硬件行为却让你大吃一惊。
