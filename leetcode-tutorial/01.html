数组基础与哈希表本质

大家好，我是你们的讲师。今天我们开启 LeetCode 刷题之旅的第一章——数组与哈希表。为什么把这两个放在第一章？原因很简单：数组和哈希表是面试中出现频率最高的数据结构。据我统计，大约 30% 的 LeetCode 题目直接或间接用到了它们。无论你是刚刷完"Hello World"的新手，还是准备冲刺大厂 offer 的老手，这两个结构都是你的基本功。

先简单回顾一下：数组（Array）是一段连续的内存空间，每个元素通过下标访问。它的核心优势是 O(1) 的随机访问——只要你知道下标，就能瞬间拿到数据。但代价是插入和删除需要移动元素，最坏情况下是 O(n)。而哈希表（Hash Table / HashMap）则是通过哈希函数把键映射到存储位置，实现了平均 O(1) 的查找、插入和删除。

数组和哈希表之间有一个非常深刻的关系，我称之为"空间-时间权衡（Space-Time Tradeoff）"：数组节省空间但查询慢；哈希表消耗额外空间但查询快。刷 LeetCode 时，你几乎每天都在做这种取舍。我们来看一个最经典的例子。

核心提炼：数组提供索引，哈希表提供映射。当你需要"根据值找到下标"或者"判断某个元素是否存在"时，第一反应应该是哈希表。

两数之和 — 经典范式演变

打开 LeetCode 第一题——Two Sum（两数之和）。题目说：给定一个整数数组和一个目标值，找出数组中两个数之和等于目标值的下标。这道题是所有 LeetCode 玩家的"见面礼"，也是展示"暴力 → 优化"思维过程的绝佳素材。

暴力解法

最直接的想法：双重循环，枚举所有数对。这是典型的"新手写法"——能跑，但不好看。

def two_sum_brute(nums, target):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] + nums[j] == target:
                return [i, j]
    return []

时间复杂度 O(n²)，空间复杂度 O(1)。当 n = 10⁵ 时，这个循环要跑约 50 亿次——面试官看到这里基本就摇头了。

哈希表优化

核心思路：遍历数组时，把"当前元素的值 → 下标"存入 HashMap。对于每个元素 nums[i]，检查 target - nums[i] 是否已经在 HashMap 中——如果在，直接返回两个下标。这就是"用空间换时间"的典型应用。

def two_sum(nums, target):
    seen = {}  # 值 -> 下标
    for i, val in enumerate(nums):
        complement = target - val
        if complement in seen:
            return [seen[complement], i]
        seen[val] = i
    return []

时间复杂度降为 O(n)，空间复杂度升到 O(n)。一次遍历搞定，代码简洁且高效。

记忆口诀：遇到配对问题，先想哈希表。Two Sum 的变体非常多——Three Sum 实际上就是把第一层固定后，对剩余部分调用 Two Sum 逻辑。掌握了这道题的思路，你就拿到了"配对查找"类问题的钥匙。

我说一个真实的故事。有一次我面试一位候选人，他 Two Sum 写得飞快，HashMap 流畅地用了出来。但当我追问"为什么 HashMap 能在 O(1) 时间找到 complement"时，他愣住了。他知道怎么用，但不知道底层原理。所以请记住：哈希冲突、负载因子、扩容机制，这些概念面试官一定会问，不要只会调 API。

原地哈希技巧

接下来这道题会颠覆你对数组的认知——First Missing Positive（缺失的第一个正数，LeetCode 41）。题目要求：找出未排序数组中最小的缺失正整数，要求时间复杂度 O(n)、空间复杂度 O(1)。

如果可以用额外空间，直接上个 HashSet 就搞定了。但问题要求 O(1) 额外空间——那怎么办？答案出奇地巧妙：把数组本身当作哈希表来用。

核心思想

对于长度为 n 的数组，缺失的第一个正数一定在 [1, n+1] 这个范围内（想想为什么？如果 1 到 n 都在，答案就是 n+1）。所以我们可以遍历数组，把每个数字放到它"应该"在的位置上：数字 1 放下标 0，数字 2 放下标 1……以此类推。

def first_missing_positive(nums):
    n = len(nums)

    # 第一步：把每个数放到它应该在的位置
    for i in range(n):
        while 1 <= nums[i] <= n and nums[nums[i] - 1] != nums[i]:
            # 把 nums[i] 交换到下标 nums[i]-1
            correct_pos = nums[i] - 1
            nums[i], nums[correct_pos] = nums[correct_pos], nums[i]

    # 第二步：找到第一个位置不对的数字
    for i in range(n):
        if nums[i] != i + 1:
            return i + 1

    return n + 1

这个技巧叫"原地哈希"（In-place Hashing）。关键在于 while 循环——每次交换都能让至少一个元素归位，总交换次数不超过 n，所以整体仍是 O(n)。

注意：这里的 while 不能写成 if！因为交换后 nums[i] 变成了一个新的数字，它可能也不在正确的位置上，需要继续处理。我见过很多人在这里写 if 然后百思不得其解——相信我，我也犯过这个错。

什么时候用原地哈希？题目出现约束 "1 到 n"、"0 到 n-1"、"长度为 n 的数组包含范围在 1..n 内的数"——这些都是强烈的提示信号。类似题目还有 Find All Numbers Disappeared in an Array（LeetCode 448）和 Find the Duplicate Number（LeetCode 287）。

最长连续序列

来看一道考验"思维转换"的题目——Longest Consecutive Sequence（最长连续序列，LeetCode 128）。给定一个未排序数组，找出最长连续序列的长度（要求 O(n) 时间复杂度）。

比如输入 [100, 4, 200, 1, 3, 2]，最长连续序列是 [1, 2, 3, 4]，输出 4。

排序的解法非常简单——先排序再遍历，但时间是 O(n log n)，不满足要求。如何做到 O(n)？关键一步：用 HashSet 存储所有数字，然后只从序列的起点开始计数。

def longest_consecutive(nums):
    num_set = set(nums)
    longest = 0

    for num in num_set:
        # 只有当 num-1 不存在时，num 才是序列起点
        if num - 1 not in num_set:
            current = num
            length = 1

            while current + 1 in num_set:
                current += 1
                length += 1

            longest = max(longest, length)

    return longest

这个解法的精妙之处在于：只从序列起点开始统计。对于每个数，如果它的前驱（num-1）也在集合中，说明它不是序列起点，直接跳过。这样一来，每个元素最多被访问两次（一次在外部循环，一次在 while 扩展），总复杂度是 O(n)。

重要模式：当你需要判断"某个元素的邻居是否存在"时，HashSet 是最佳选择。这个模式在后续的图论章节中还会反复出现——Union-Find 和 DFS 都能解决连续序列问题，但 HashSet 方案是最简洁的。

这里我讲一个自己的踩坑经历。我第一次写这道题时，在 for num in nums（原数组）而不是 for num in num_set（去重后的集合）上循环。结果有很多重复元素，while 循环把同一个序列跑了七八遍，LeetCode 直接报 TLE（Time Limit Exceeded）。被教育了一顿之后我才反应过来——在 HashSet 上遍历，天然去重，是最优选择。

实战踩坑与技巧总结

学完三个经典题目，我们来盘点一下数组和哈希表相关的常见坑点和最佳实践。

Pitfall 1：哈希冲突导致最坏 O(n)

HashMap 的平均时间复杂度是 O(1)，但这是在有良好哈希函数的前提下的数据。如果所有键的哈希值都相同（比如有人恶意构造数据，或者你的 __hash__ 实现极差），查找会退化为 O(n) 的链表/红黑树遍历。Python 的字典在底层做了大量优化，面试时一般不需要深入实现细节，但你要知道这个概念。

Pitfall 2：遍历数组时修改元素

在 Python 中遍历列表并同时修改它——这是灾难的根源。要么使用索引遍历并原地修改，要么构建一个新的列表。在 LeetCode 上最常见的错误就是在 for num in nums 时删除了元素，导致下标错乱。

Pitfall 3：空间换时间并非免费

哈希表虽然快，但不是没有代价。当数据量极大（比如数组长度 10⁷）时，一个 O(n) 的 HashMap 可能占用数百 MB 内存，直接导致 MLE（Memory Limit Exceeded）。在嵌入式或资源受限场景中，要优先考虑原地算法。

问题类型 推荐技巧 时间复杂度 空间复杂度
配对查找（Two Sum） HashMap 缓存已遍历元素 O(n) O(n)
判断元素是否存在 HashSet / 布尔数组 O(n) O(n)
范围 1..n 内原地哈希 交换到正确下标 O(n) O(1)
连续序列 / 连通性 HashSet 起点扩展 O(n) O(n)
频率统计 Counter / 字典计数 O(n) O(k)
刷题建议：数组与哈希表是 LeetCode 的"基本功中的基本功"。我建议你在完成本章后，把 LeetCode 上标签为"Array"和"Hash Table"的前 20 道高频题刷一遍，每个题目写出暴力解和优化解两个版本，体会"空间-时间权衡"的具体感觉。
