第21章 工具调用与Function Call数据

大家好，我是你们的讲师。在前两章，我们专注于让模型变得"更聪明"——通过 CoT 数据学会逐步推理，通过 GRPO 强化学习学会自我纠错。但一个只会"思考"不会"行动"的模型，就像一个满腹经纶却足不出户的学者——知识渊博，但无法改变世界。

这一章要讨论的，正是模型的"行动力"——工具调用（Tool Use）和函数调用（Function Calling）。这不仅仅是模型能力的延伸，更是 AI 从"对话系统"进化到"Agent 系统"的关键一步。

让模型学会「使用工具」——从纯文本到行动能力的跨越

什么是工具调用数据？简单来说，就是教会模型在适当的时机调用外部工具（API、数据库、计算器、搜索引擎等）来完成任务。与传统 SFT 数据"instruction → response"的两段式结构不同，工具调用数据是一个多轮交互过程：

用户：帮我查一下北京今天的天气？
模型：我想我需要调用天气 API。[tool_call: get_weather(city="北京")]
系统：[tool_result: {"temperature": 28, "condition": "晴", "humidity": "45%"}]
模型：北京今天天气晴朗，气温 28°C，湿度 45%，非常适合户外活动。

看到区别了吗？模型不再是一次性给出最终回答，而是在"思考"之后决定"调用什么工具、传入什么参数"，然后根据工具的返回结果来组织最终的回答。这种"思考-行动-观察"的循环，正是 Agent 系统的核心模式。

训练模型学会工具调用，从数据角度看，需要解决三个核心问题：

  • 何时调用——模型需要知道哪些问题需要调用工具（比如实时数据查询），哪些问题可以直接回答（比如常识知识）。
  • 调用什么——在多个可用工具中，选择最合适的工具和正确的参数。
  • 如何解读结果——工具返回的数据可能是结构化的 JSON、错误码、或者空结果，模型需要正确地理解这些信息并给出用户友好的回答。

Glaive Function Calling v2 — 函数调用首选数据集

在函数调用数据领域，Glaive Function Calling v2 是目前最成熟、最常用的开源数据集之一。它由 Glaive AI 团队发布，包含约 10 万条高质量的工具调用训练样本。

Glaive v2 的核心优势在于它的场景覆盖完整性：

  • 单函数调用：最基本的场景——模型识别到需要调用一个函数，正确生成调用，并解读结果。例如："帮我算一下 2 的平方根" → 调用 math.sqrt(2)。
  • 多函数串行调用：需要依次调用多个函数的场景。例如："帮我查一下北京的天气，然后根据温度给我推荐穿搭" → 先调用 get_weather，再根据结果调用 outfit_recommendation。
  • 嵌套函数调用：一个函数的输出是另一个函数输入的一部分，模型需要理解函数之间的数据依赖关系。
  • 并行函数调用：多个独立的函数调用可以同时发出——这在 2025 年之后的模型 API 中已经得到原生支持（如 OpenAI 的 parallel_tool_calls 参数）。
  • 错误恢复：工具返回错误时的处理——重试、降级、或向用户报告错误。这是工具调用数据中最容易被忽视但也最重要的能力。
使用建议：Glaive v2 的数据格式仿照 OpenAI 的 tool calling message 规范，这意味着如果你在训练一个兼容 OpenAI API 格式的模型，可以直接用 Glaive v2 的数据进行微调。在 HuggingFace 上搜索 Glaive 的仓库，你会发现不仅有训练数据，还配套有评估基准和格式化脚本——这是函数调用数据生态中一个难得的"开箱即用"方案。

ToolBench — 大规模工具 Agent 数据

如果说 Glaive v2 是函数调用领域的"精品店"，那 ToolBench 就是"超级市场"。ToolBench 由清华大学的研究团队在 2024 年发布，是一个大规模的、基于真实 API 的工具调用数据集。

ToolBench 的规模令人印象深刻：

  • 16,000+ 个真实 API 工具——从 RapidAPI 收集的真实 RESTful API，涵盖天气、金融、社交媒体、电商、地图等各类领域
  • 数十万条工具对话——每条对话包含：用户意图 → 工具选择 → 参数填充 → API 调用 → 结果解读的完整轨迹
  • 工具规划 + 调用 + 结果解读全流程覆盖——不仅训练模型"调哪个 API"，还训练"什么时候调"和"怎么处理返回结果"

ToolBench 的生成方法值得一提：他们使用 ChatGPT 作为"教师模型"来生成工具调用的推理轨迹和标注数据。具体来说，对每个用户 query，教师模型会根据工具列表（tool list）和 API 文档决定调用哪个工具、填入什么参数，然后模拟 API 返回结果并给出最终回答。整个过程构成了一个"思考-调用-回答"的完整循环。

维度 Glaive Function Calling v2 ToolBench
规模 ~100K 条 数十万条 + 16K+ API 工具
API 来源 预设的函数定义（虚构/简化） RapidAPI 真实 RESTful API
场景覆盖 单/多/嵌套/并行 + 错误恢复 工具规划 + 调用 + 结果解读全流程
数据格式 OpenAI function calling 格式 自定义格式（含工具规划步骤）
适用场景 标准 API 兼容的函数调用训练 复杂工具链规划和多步 Agent 训练

函数调用数据格式详解

函数调用数据的核心是搞清楚消息格式。目前业界最通用的格式是 OpenAI 定义的 function calling / tool calling 消息规范。下面是一个完整的例子：

{
  "messages": [
    {
      "role": "system",
      "content": "你是一个有用的助手。你可以使用以下工具：\n\n"
                "工具：get_weather\n"
                "描述：获取指定城市的天气信息\n"
                "参数：\n"
                "  - city (string, 必填)：城市名称\n"
                "  - date (string, 可选)：日期，格式 YYYY-MM-DD"
    },
    {
      "role": "user",
      "content": "北京明天天气怎么样？"
    },
    {
      "role": "assistant",
      "content": null,
      "tool_calls": [
        {
          "type": "function",
          "function": {
            "name": "get_weather",
            "arguments": "{\"city\": \"北京\", \"date\": \"2026-07-01\"}"
          }
        }
      ]
    },
    {
      "role": "tool",
      "tool_call_id": "call_abc123",
      "content": "{\"temperature\": 32, \"condition\": \"晴\", "
                "\"humidity\": \"30%\", \"wind\": \"3级\"}"
    },
    {
      "role": "assistant",
      "content": "北京明天（7月1日）天气晴朗，气温 32°C，"
                "湿度 30%，吹 3 级风。天气比较炎热，建议做好防晒措施。"
    }
  ]
}

这个格式中有几个关键要点：

  • function 定义：在 system message 或单独的工具列表中声明可用的工具名称、描述、参数 schema。模型需要根据这些定义来决定调用哪个工具。
  • tool_call 消息：assistant 消息中 content 为 null，由 tool_calls 字段携带工具调用信息。模型需要学会在这种格式下输出"调用决定"。
  • tool_response 消息：role 为 "tool" 的消息，携带工具的执行结果。这个结果可能是结构化的 JSON，也可能是原始字符串。模型需要学会根据这个结果生成最终回答。

函数调用数据的质量要求

函数调用数据比普通 SFT 数据多了一层质量维度——工具交互的正确性。以下是最关键的质量检查点：

  • 参数正确性：调用的函数参数是否与 API schema 完全匹配？参数名、参数类型、参数值格式都必须精确无误。一个常见的错误是参数名拼写错误（比如把 "temperature_unit" 写成 "temp_unit"）。
  • 调用顺序合理性：在多步函数调用中，调用的先后顺序是否合理？比如"先查询用户信息，再根据用户信息推荐商品"——这个顺序是对的。但如果反过来——先推荐再查用户信息——那就是错的。
  • 错误恢复能力：工具返回错误时，模型是否尝试了合理的恢复策略？比如 API 返回 429（请求过于频繁）时，模型应该等待后重试，而不是直接向用户报错。数据中应该包含这类"含错误-有恢复"的样本。
一个踩坑故事：函数参数幻觉——模型编造不存在的 API 参数名

2025 年初，我在参与一个智能客服项目时遇到了一个让人哭笑不得的问题。我们为模型配备了一个查询订单状态的工具，API 定义中只有两个参数：order_id（字符串，必填）和 customer_id（字符串，可选）。

训练完成后，我们发现模型在与用户对话时，产生了一堆匪夷所思的调用记录。比如用户说"帮我查一下上周的订单"，模型调用了：query_order(order_id="xxx", customer_id="yyy", order_status="处理中")。看到了吗？order_status 这个参数根本不存在！模型"凭空"编造了一个参数名。

排查之后发现，问题出在我们的训练数据上。我们使用了多个来源的函数调用数据，其中一版数据定义了一个类似的查询函数，那个函数确实包含 order_status 参数。当不同来源的数据混合训练时，模型"学到了"一个不存在的参数，然后在实际推理中产生了错误的调用。

解决方案是我们在数据预处理管线中新增了一个"参数名白名单校验"步骤——对每条函数调用数据，检查所有被调用的参数名是否都在对应 API 的 schema 定义中。不匹配的数据要么修正要么丢弃。这个看似简单的步骤，也让我深刻认识到：函数调用数据的质量治理，必须从"数据格式正确"延伸到"语义内容正确"。

函数调用数据集纵览

除了 Glaive v2 和 ToolBench 之外，还有一些值得关注的函数调用数据集：

数据集 规模 特色
Glaive Function Calling v2 ~100K 格式规范，单/多/嵌套函数调用全覆盖，含错误恢复
ToolBench 数十万条 + 16K+ API 真实 API，工具规划-调用-结果解读全流程
BFCL (Berkeley Function Calling Leaderboard) 评测集 ~2K 业界标准评测基准，虽不是训练集但对标训练标杆

ToolAlpaca ~3.9K 基于自指令生成的工具调用数据，轻量级

工具调用数据是连接"语言模型"和"Agent 系统"的关键桥梁。通过这一章，你了解了函数调用数据的核心结构（function definition → tool_call → tool_response 三要素循环），掌握了 Glaive v2 和 ToolBench 这两个标杆数据集的特点和适用场景，也学会了函数调用数据的质量要求以及一个经典的"参数幻觉"踩坑教训。

但工具调用只是 Agent 能力的起点。当模型学会调用单个工具后，下一步是让它学会在多个步骤中综合运用工具、处理中间结果、并根据反馈动态调整行动策略——这就是真正意义上的 Agent 能力。下一章，我们将深入探讨 Agent 多步推理数据——SWE-bench、环境交互轨迹、以及 Agent 数据与普通 SFT 数据的根本区别。
