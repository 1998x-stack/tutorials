第二章：信息的表示与处理——整数

大家好，我是你们的讲师。在进入本章之前，我想先问你一个问题：假设你在 C 语言里写了一句 int x = 0x12345678;，然后你把 x 的低地址字节取出来——你猜它等于多少？

你可能觉得这个问题太简单。但事实是，你的 CPU 架构决定了答案：在 x86 机器上低地址字节是 0x78，在 ARM 大端模式下却是 0x12。同一条语句，同一段数据，不同的 CPU 看到的是完全不同的字节排列。这就是整数表示的第一个"坑"——字节序。

比起字节序更隐蔽的是补码的数学性质、无符号数与有符号数比较时的隐式转换、以及溢出可能造成的灾难性后果。1996 年，欧洲航天局的 Ariane 5 火箭在发射 40 秒后爆炸，直接损失 3.7 亿美元——根因就是一个 64 位浮点数到 16 位整数的溢出转换。虽然那是浮点数，但类似的整数溢出安全隐患至今仍潜伏在无数生产代码中。2014 年 OpenSSL 心脏滴血漏洞，其根因本质上也是一个整数溢出问题。

理解整数的表示，不是一个学术练习题——它是写出安全、正确、可移植代码的前提。

2.1 大端序与小端序

字节序（Endianness）这个词来自乔纳森·斯威夫特的小说《格列佛游记》。小说中，小人国里有两派政治势力争执不休——一派认为吃鸡蛋应该从大的一端（Big-Endian）敲开，另一派认为应该从小的一端（Little-Endian）敲开。计算机界的"大小端"之争，和小说里一样有趣且持久。

大端序（Big-Endian）把数据的最高有效字节存放在最低地址，小端序（Little-Endian）把最低有效字节存放在最低地址。看一个例子，假设我们在内存地址 0x100 处存放一个 4 字节的整数 0x12345678：

内存地址 0x100 0x101 0x102 0x103
大端序（Big-Endian） 0x12 0x34 0x56 0x78
小端序（Little-Endian） 0x78 0x56 0x34 0x12

大端序很符合人的直觉——你从左往右读过去，就是数字本身的样子。小端序则反着来，但对 CPU 而言，小端序在做加法时效率更高，因为加法从低位开始，CPU 可以直接从低地址读起，无需额外的字节偏移计算。

怎么检测当前机器的字节序？一段简单的 C 代码就够了：

#include <stdio.h>

int main()
{
    int x = 0x12345678;
    unsigned char *p = (unsigned char *)&x;

    for (int i = 0; i < sizeof(x); i++)
        printf("byte[%d] = 0x%02x\n", i, p[i]);

    if (p[0] == 0x78)
        printf("Little-Endian\n");
    else if (p[0] == 0x12)
        printf("Big-Endian\n");
    return 0;
}

在 x86-64 的 Linux 上运行，输出是：

byte[0] = 0x78
byte[1] = 0x56
byte[2] = 0x34
byte[3] = 0x12
Little-Endian

这里有个关键操作——我们用 unsigned char * 指针来重新解释 int 对象的内存。这是一个在 C 语言中完全合法的技巧（char 类型被语言标准特例允许访问任何对象的内存表示），它能让我们以字节为单位看到数据的底层布局。

实用建议：在网络传输中，为保证不同架构的机器能正确解析数据，网络字节序被定义为大端序。这就是为什么你总能看到 htons()、htonl() 这类函数——它们把本机字节序的整数转换为网络字节序（大端）。如果你的 x86 机器不调用这些函数就直接发送多字节整数，接收方如果是大端机器就会解析出反过来的数值。

2.2 补码编码的数学本质

大多数人知道负数在计算机中以补码（Two's Complement）表示，但很少有人认真思考过补码的数学含义。补码的核心思想可以用一个简单的公式表达。

对于一个 w 位的二进制数，如果把它解释为有符号整数：

B2T(x) = -x_{w-1} * 2^{w-1} + Σ (x_i * 2^i)   (i = 0 到 w-2)

其中 B2T 表示 Binary to Two's Complement。最高位（符号位）的权重是负的，其余位的权重都是正的。这意味着：

  • 当最高位是 0（即正数区域）时，补码和普通二进制一样
  • 当最高位是 1 时，数值等于 -2^{w-1} 加上剩余位的正数部分

举个例子，对于 4 位补码 1101：

B2T(1101) = -1*2^3 + 1*2^2 + 0*2^1 + 1*2^0
          = -8 + 4 + 0 + 1
          = -3

这就解释了补码的一个核心性质——不对称性。以 8 位整数为例：

类型 最小值 最大值
char (有符号) -128 127
unsigned char 0 255
int (有符号) -2,147,483,648 2,147,483,647
unsigned int 0 4,294,967,295

⚠️ 绝对不要对 INT_MIN 调用 abs()！

abs(INT_MIN) 在 C 标准中是 未定义行为（undefined behavior）。INT_MIN = -2147483648，而 INT_MAX = 2147483647，补码的不对称性导致 -INT_MIN 根本不能在 32 位有符号整数范围内表示出来。abs() 函数内部试图求反，结果会产生溢出，进而导致不可预期的结果——常见情况下返回的还是 INT_MIN，即 abs(INT_MIN) == INT_MIN。这是一个真实的源头 bug。

补码的另一个漂亮性质是：加法和减法不必区分有符号还是无符号。同样的二进制位，同样的加法器电路，无论你把它们解释为有符号还是无符号，加法操作在二进制层面一模一样。这个性质是 IEEE 标准刻意设计的，目的就是让 CPU 省掉一套额外的 UAL 电路。

2.3 无符号数与有符号数的转换

C 语言中，无符号数和有符号数之间的隐式转换可能是最常见、也最隐蔽的 bug 来源之一。它的规则可以总结为一句话：当表达式中同时出现 signed 和 unsigned 时，signed 会被隐式提升为 unsigned。

来看一个经典陷阱：

#include <stdio.h>

int main()
{
    int a = -1;
    unsigned int b = 0;

    if (a < b)
        printf("a < b (有符号比较)\n");
    else
        printf("a > b (无符号比较！)\n");

    return 0;
}

这段代码会输出什么？直觉告诉我们 -1 < 0 成立，但实际输出是：

a > b (无符号比较！)

原因在于：C 语言规定当有符号 int 和无符号 unsigned int 出现在同一个表达式中比较时，有符号 int 被隐式转换为无符号 unsigned int。-1 的二进制表示是全 1，被解释为无符号数就成了 0xFFFFFFFF，也就是 4294967295——当然大于 0。

让我们用一个更实际的例子来说明这个问题有多危险：

int sum_elements(int arr[], unsigned int count)
{
    int sum = 0;
    for (int i = 0; i < count - 1; i++)  // 危险！
        sum += arr[i];
    return sum;
}

如果 count = 0，那么 count - 1 就不是 -1，而是 UINT_MAX = 4294967295！循环会越界访问整个内存，直接导致段错误。正确的写法应该把 count 声明为 int，或者在调用前检查。

关键原则：在 C/C++ 中，尽量不要混用 signed 和 unsigned。如果不得不混用，务必让编译器开启 -Wsign-compare 警告（GCC/clang 默认已启用），并且把这类警告当作错误处理。GCC 的 -Werror=sign-compare 可以帮你强制消除这类隐患。

2.4 整数溢出与安全编码

整数溢出的本质很简单：一个固定位数的整数，能表示的范围是有限的。当运算结果超出这个范围时，就会发生溢出——高位被丢弃，结果回绕（wrap around）。但在安全领域，整数溢出从来都不是小事。

最常见的溢出场景：

  • 无符号加法回绕：UINT_MAX + 1 = 0
  • 有符号加法溢出：INT_MAX + 1 = INT_MIN（未定义行为！）
  • 乘法溢出：假设分配 n * sizeof(T) 字节内存，如果 n 太大导致乘法溢出，实际分配的内存远小于预期

怎么安全地检测加法溢出？以下是一个 C 语言的实现：

#include <limits.h>

/* 安全加法：检测有符号加法是否溢出 */
int safe_add(int a, int b, int *result)
{
    if (a > 0 && b > INT_MAX - a) return -1;  // 正溢出
    if (a < 0 && b < INT_MIN - a) return -1;  // 负溢出
    *result = a + b;
    return 0;
}

在安全编码中，分配大小检查是整数溢出的高发区。看一个经典的错误模式：

/* 错误版本：乘法可能溢出 */
void *xmalloc_bad(size_t n)
{
    return malloc(n * sizeof(int));  // if n is too large, overflow!
}

/* 正确版本：先检查再分配 */
void *xmalloc_good(size_t n)
{
    if (n > SIZE_MAX / sizeof(int))  // 防止乘法溢出
        return NULL;
    return malloc(n * sizeof(int));
}

真实案例：OpenSSL 心脏滴血漏洞（Heartbleed, CVE-2014-0160）

2014 年震惊全球的 Heartbleed 漏洞，本质上就是一个整数溢出问题。OpenSSL 的 TLS 心跳（Heartbeat）扩展中，客户端发送一个心跳请求，包含一段数据及其长度。服务器端在处理时，没有验证声明的长度与实际数据长度是否一致。攻击者可以声明长度为 65535 字节，但实际只发送几个字节——服务器端直接使用这个长度调用 memcpy()，把堆内存中超过请求内容的部分也发送回去，其中可能包含其他用户的私钥、密码等敏感信息。补丁方案只增加了一行检查：if (payload_length > actual_length) return error;。

现代安全编码实践：

  • C 语言中优先使用 size_t（无符号类型），避免有符号/无符号混用
  • GCC/Clang 的 -ftrapv 选项可以在有符号溢出时产生运行时 trap（但性能有损耗）
  • 现代编译器提供内置溢出检测函数：__builtin_add_overflow()、__builtin_mul_overflow() 等
  • 在 C++20 中，推荐使用 std::add_sat/std::clamp 等安全算术操作

到这里，我们已经深入理解了整数在计算机中的表示机制——从字节序到补码数学，从隐式转换陷阱到溢出检测。这些知识不只是考试用的理论，它们是写出可靠系统软件的基本功。我在工作中见到过太多因为忽略这些细节而导致的线上事故——很多 bug 的根源，说穿了就是一个符号用错了、一个类型转换没想到、一个溢出没检查。
