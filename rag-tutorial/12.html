第十二章：查询优化与重写技术

大家好，我是你们的讲师。前几章我们一直在完善检索系统本身——BM25、稠密检索、混合检索、重排序。但有个问题被我们忽略了：用户输入的质量。

真实的用户查询是什么样的？"这个怎么弄？"、"帮我看看"、"到底是什么原因"——模糊、碎片化、口语化。如果直接把这些原始查询送给检索系统，结果能好吗？显然不能。所以我们需要在检索之前，对用户的查询进行优化和重写。这就是本章的主题。

1. 常见查询问题

让我们看看真实世界中用户的查询有多"离谱"：

问题类型 原始查询示例 问题分析
查询过短 "价格"、"怎么用" 信息量不足，无法区分文档
口语化严重 "那个啥，就是手机上那个蓝色的图标怎么弄" 包含大量噪声词，关键词稀疏
多意图混杂 "公司的收入怎么样和员工人数" 包含两个独立的子问题
指代不清 "它支持吗？"（上一句提到了某产品） 代词没有上下文无法解析
拼写/语法错误 "机器学学"、"转却率" 分词出问题，BM25可能匹配不到

2. HyDE（假设性文档嵌入）

HyDE（Hypothetical Document Embeddings）是一个非常巧妙的方法。它的核心思路是：先用LLM根据用户的查询生成一个假设性的理想答案，然后用这个生成的答案去检索知识库。原理是：生成的"理想答案"在语义空间中应该和真实的知识库文档更接近，因此用它的向量去检索效果会更好。

流程如下：

  1. 用户输入查询 Q："深度学习和传统机器学习有什么区别？"
  2. 调用LLM生成一个假设性文档 H："深度学习是机器学习的一个子集，它使用多层神经网络……"
  3. 用嵌入模型将 H 编码为向量 V_h
  4. 用 V_h 在知识库中做向量检索
import openai
from sentence_transformers import SentenceTransformer

# 加载嵌入模型
model = SentenceTransformer('BAAI/bge-large-zh-v1.5')

def hyde_retrieve(query, k=5):
    # 步骤1: 使用LLM生成假设性文档
    prompt = f"""请根据以下问题，生成一段详细的回答，就像是你在回答这个问题一样。
    要求回答包含具体信息，语气专业。

    问题：{query}

    回答："""

    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.3,
        max_tokens=300,
    )
    hypothetical_doc = response.choices[0].message.content

    # 步骤2: 将假设性文档编码为向量
    # BGE：查询侧需要加指令前缀
    hyde_vector = model.encode(
        "为这个句子生成表示以用于检索相关文章：" + hypothetical_doc,
        normalize_embeddings=True
    )

    # 步骤3: 在向量数据库中检索（此处为伪代码示意）
    # results = vector_db.search(hyde_vector, top_k=k)
    # return results

    return hypothetical_doc  # 返回用于调试查看

# 示例
query = "Transformers模型为什么比RNN效果好？"
generated = hyde_retrieve(query)
print("HyDE生成的假设文档：")
print(generated[:200])
⚠️ 什么时候不应该用HyDE？
HyDE不是万能的，在某些场景下它甚至可能降低检索质量：
1. 知识库高度结构化时：如果你的知识库是产品规格表、法律条文等精确文本，LLM生成的"假设性文档"可能是自由格式的，和知识库的分布差异很大，反而降低检索效果。
2. 查询本身已经很精确时：如搜索"劳动法第47条"这种精确查询，增加HyDE步骤会引入噪声。
3. 延迟敏感场景：HyDE需要一次LLM调用（数百毫秒到数秒），对实时性要求高的系统不友好。
4. LLM可能产生幻觉：生成的假设文档可能包含错误信息，这些错误信息会影响检索结果。建议只在测试中充分验证后再上线。

3. Step-Back Prompting（回溯提示）

当用户的问题是高度具体的细节问题时，直接用原始查询检索可能无法找到相关文档。Step-Back Prompting的思路是：先让LLM将具体问题"退一步"，转化为更通用的概念性问题，然后用通用问题去检索。

例如：

  • 原始问题："BERT的[CLS] token的输出维度是多少？"
  • Step-Back问题："BERT模型的架构设计原理是什么？"

LLM在回答"维度是多少"之前，可能需要先了解BERT的架构。Step-Back Prompting先帮助LLM检索到架构相关文档，再结合具体问题生成答案。这种"先抽象后具体"的策略在多层知识库中效果显著。

4. 查询分解（Query Decomposition）

当用户的一个问题包含多个独立意图时（如"2024年公司的收入和利润是多少"），直接检索往往会顾此失彼。查询分解就是让LLM把复杂问题拆成多个简单子问题，分别检索后合并结果。

def decompose_query(complex_query):
    prompt = f"""请将以下复杂问题分解为多个独立的简单子问题。
每个子问题应该只询问一个方面的信息，并且可以独立检索。

复杂问题：{complex_query}

请以JSON数组格式输出，每个子问题是一个字符串：
"""

    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.1,
    )
    import json
    sub_queries = json.loads(response.choices[0].message.content)
    return sub_queries

# 示例
query = "Transformer模型中self-attention的计算复杂度是多少？为什么它比RNN更适合长序列？"
sub_qs = decompose_query(query)
for i, q in enumerate(sub_qs):
    print(f"子问题{i+1}: {q}")
# 输出:
# 子问题1: Transformer模型中self-attention的计算复杂度是多少？
# 子问题2: 为什么Transformer比RNN更适合处理长序列？

5. 查询路由（Query Routing）

不是所有查询都应该走同样的检索路径。查询路由的核心思想是：根据查询内容，动态选择最合适的检索策略。例如：

  • 事实性问题（"北京的常住人口是多少？"）→ 走结构化数据/表格检索
  • 概念性问题（"什么是机器学习？"）→ 走向量语义检索
  • 代码问题（"Python列表推导式的语法"）→ 走代码库BM25检索
  • 闲聊/打招呼（"你好"）→ 不走检索，直接回复

查询路由通常有两种实现方式：

  • 基于规则：用关键词匹配或正则表达式做路由。简单但覆盖不全。
  • 基于分类器：训练一个小模型（如BERT分类器）做查询分类。效果好但需要标注数据。

6. 方法对比总结

方法 核心思路 LLM调用次数 适用场景 效果提升
HyDE 生成假设文档再检索 1次 短查询、语义检索场景 MRR +5~15%
Step-Back Prompting 具体问题退化为抽象问题 1次 深层次技术问题 Recall +10~20%
查询分解 多意图拆分为单意图 1次（生成多个子问题） 复合型问题 覆盖率 +15~25%
查询路由 动态选择检索策略 0~1次 混合型知识库 端到端 +10~20%

踩坑故事："我昨天问的那个问题"

做一个客服问答机器人时，发现用户经常问"我昨天问的那个问题呢"、"之前说的那个方案具体是什么"。这些指代性问题如果不处理，检索系统完全找不到相关信息。

一开始的解决方案是简单地把历史对话拼接起来，但发现效果很差——拼接后的文本既长又混乱，检索质量反而下降。后来我们采用了多轮对话压缩技术：

每次对话结束后，用LLM生成一个会话摘要（Session Summary），存储在用户会话中。当下一次用户提问涉及指代时，将当前查询与会话摘要一起作为输入，让LLM先做指代消解（Coreference Resolution），生成一个独立的问题，再去检索。

这个方案让指代性查询的准确率从35%提升到了78%。

本章小结

查询优化是RAG系统中常被忽视但极其重要的环节。HyDE通过假设性文档增强短查询的检索质量；Step-Back Prompting将具体问题抽象化以匹配更广泛的文档；查询分解处理复合问题；查询路由为不同问题选择最优检索路径。这些技术的核心理念是：不要假设用户的查询是完美的——在检索之前，先花时间把查询优化好。

当检索完成、查询也已经优化，下一步就是把检索到的文档和用户的问题一起交给LLM来生成了。但如何组织这些上下文？如何设计提示模板？这就是下一章——上下文增强与提示工程的内容。第13章见！
