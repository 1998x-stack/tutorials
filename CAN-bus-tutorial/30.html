第30章：CAN总线项目实战：基于STM32的CAN总线数据采集系统、上位机显示、故障诊断

终于到了最后一章。

说实话，前面29章讲了很多理论、协议、硬件、软件。但真正让CAN总线发挥价值的，是把它做成一个完整的系统。今天我们就来做一个实战项目：基于STM32的CAN数据采集系统，配上位机显示，再加故障诊断功能。

这个项目我做过不下5个版本。最早是在一个工程机械项目上，客户要求实时监控发动机参数。那时候我熬夜调代码，踩了不少坑。今天我把经验都倒出来，你直接拿去用。

30.1 系统整体架构

先看整体框架。这个系统分三层：

  • 底层：STM32 + CAN收发器，负责采集传感器数据，通过CAN总线发送
  • 中间层：USB-CAN适配器，把CAN数据转成PC能识别的格式
  • 上层：上位机软件，显示数据、存储、故障诊断

说白了，就是传感器→STM32→CAN总线→USB-CAN→PC上位机。这个链路我建议你用现成的USB-CAN工具，比如周立功的或者PCAN。自己写USB驱动？嗯，除非你想多花两周时间。

硬件清单：

  • STM32F103C8T6 开发板（便宜够用）
  • TJA1050 CAN收发器（或者SN65HVD230）
  • USB-CAN适配器（推荐PCAN-USB）
  • 传感器：温度、压力、转速模拟信号
  • 12V电源（工业现场常用）

30.2 STM32端程序设计

STM32这边要做三件事：采集数据、打包成CAN帧、发送出去。我习惯用HAL库，因为移植方便。

先看CAN初始化。这里有个坑：波特率配置。我记得有一次项目，CAN通信死活不通，查了半天发现是时钟树配错了。STM32的CAN时钟来自APB1，最高36MHz。你算波特率的时候一定要确认这个时钟。

// CAN初始化配置
void CAN_Init(void)
{
    CAN_HandleTypeDef hcan;
    CAN_FilterTypeDef sFilterConfig;
    
    hcan.Instance = CAN1;
    hcan.Init.TimeTriggeredMode = DISABLE;
    hcan.Init.AutoBusOff = ENABLE;
    hcan.Init.AutoWakeUp = DISABLE;
    hcan.Init.AutoRetransmission = ENABLE;
    hcan.Init.ReceiveFifoLocked = DISABLE;
    hcan.Init.TransmitFifoPriority = DISABLE;
    hcan.Init.Mode = CAN_MODE_NORMAL;
    
    // 波特率500kbps，APB1时钟36MHz
    // 计算公式：Baud = APB1 / (Prescaler * (BS1 + BS2 + 1))
    hcan.Init.SyncJumpWidth = CAN_SJW_1TQ;
    hcan.Init.TimeSeg1 = CAN_BS1_8TQ;
    hcan.Init.TimeSeg2 = CAN_BS2_7TQ;
    hcan.Init.Prescaler = 4;  // 36M / (4 * (8+7+1)) = 500k
    
    if (HAL_CAN_Init(&hcan) != HAL_OK)
    {
        Error_Handler();
    }
    
    // 配置过滤器，接收所有ID
    sFilterConfig.FilterBank = 0;
    sFilterConfig.FilterMode = CAN_FILTERMODE_IDMASK;
    sFilterConfig.FilterScale = CAN_FILTERSCALE_32BIT;
    sFilterConfig.FilterIdHigh = 0x0000;
    sFilterConfig.FilterIdLow = 0x0000;
    sFilterConfig.FilterMaskIdHigh = 0x0000;
    sFilterConfig.FilterMaskIdLow = 0x0000;
    sFilterConfig.FilterFIFOAssignment = CAN_RX_FIFO0;
    sFilterConfig.FilterActivation = ENABLE;
    
    HAL_CAN_ConfigFilter(&hcan, &sFilterConfig);
    
    // 启动CAN
    HAL_CAN_Start(&hcan);
    HAL_CAN_ActivateNotification(&hcan, CAN_IT_RX_FIFO0_MSG_PENDING);
}

个人经验：我习惯把CAN过滤器设置成接收所有ID，然后在应用层做过滤。这样调试方便，后期改ID也不用动底层。当然，量产产品建议用硬件过滤，省CPU资源。

数据采集部分，我用ADC采集模拟量。比如温度传感器，0-5V对应0-100度。采集后打包成CAN帧发送。

// 数据采集和发送任务
void DataAcquisitionTask(void)
{
    CAN_TxHeaderTypeDef txHeader;
    uint8_t txData[8];
    uint32_t txMailbox;
    
    // 采集温度
    uint16_t adc_value = HAL_ADC_GetValue(&hadc1);
    float temperature = (adc_value / 4095.0) * 100.0;
    
    // 打包成CAN帧
    txHeader.StdId = 0x100;  // 温度数据ID
    txHeader.ExtId = 0;
    txHeader.IDE = CAN_ID_STD;
    txHeader.RTR = CAN_RTR_DATA;
    txHeader.DLC = 4;  // 4字节数据
    
    // 温度值放大100倍，保留两位小数
    uint16_t temp_int = (uint16_t)(temperature * 100);
    txData[0] = (temp_int >> 8) & 0xFF;
    txData[1] = temp_int & 0xFF;
    txData[2] = 0;  // 预留
    txData[3] = 0;
    
    // 发送
    HAL_CAN_AddTxMessage(&hcan, &txHeader, txData, &txMailbox);
}

这里要注意：CAN帧一次只能发8字节。如果你的数据多，就分多帧发。我一般一个传感器一帧，ID不同。比如0x100是温度，0x101是压力，0x102是转速。这样上位机解析也方便。

30.3 上位机显示设计

上位机我用C#写，因为开发快。核心功能就三个：接收CAN数据、解析显示、存储日志。

接收部分，我用PCANBasic库。这个库是PEAK公司提供的，免费。你装好PCAN-USB驱动后，直接调用API就行。

// C# 接收CAN数据
public void ReceiveCANData()
{
    TPCANMsg msg;
    TPCANStatus status;
    
    while (true)
    {
        status = PCANBasic.Read(PCANBasic.PCAN_USBBUS1, out msg, out _);
        
        if (status == PCANBasic.PCAN_ERROR_OK)
        {
            // 根据ID解析数据
            switch (msg.ID)
            {
                case 0x100:  // 温度
                    float temp = (msg.DATA[0] << 8 | msg.DATA[1]) / 100.0f;
                    UpdateTemperatureDisplay(temp);
                    break;
                    
                case 0x101:  // 压力
                    float pressure = (msg.DATA[0] << 8 | msg.DATA[1]) / 10.0f;
                    UpdatePressureDisplay(pressure);
                    break;
                    
                case 0x102:  // 转速
                    uint16_t rpm = (uint16_t)(msg.DATA[0] << 8 | msg.DATA[1]);
                    UpdateRPMDisplay(rpm);
                    break;
            }
            
            // 记录日志
            LogCANMessage(msg);
        }
        
        Thread.Sleep(10);  // 10ms轮询一次
    }
}

显示界面建议：

  • 仪表盘显示实时数据（温度表、压力表、转速表）
  • 波形图显示趋势（最近100个点）
  • 数据表格显示原始CAN帧（方便调试）
  • 状态栏显示总线负载率、错误帧计数

30.4 故障诊断功能

故障诊断是项目的灵魂。没有诊断，出了问题你只能干瞪眼。

我设计了三层诊断：

  1. 物理层诊断：检查CAN_H和CAN_L电压、终端电阻
  2. 数据链路层诊断：检查错误帧、总线负载率
  3. 应用层诊断：检查数据合理性、超时判断

物理层诊断，我建议你在上位机加一个“总线健康检查”按钮。点击后，STM32会返回当前总线的状态。

// 故障诊断命令处理
void DiagnosticHandler(uint8_t cmd)
{
    CAN_TxHeaderTypeDef txHeader;
    uint8_t txData[8];
    uint32_t txMailbox;
    
    txHeader.StdId = 0x7DF;  // 诊断请求ID
    txHeader.DLC = 8;
    
    switch (cmd)
    {
        case 0x01:  // 查询总线状态
            txData[0] = 0x01;
            txData[1] = (HAL_CAN_GetState(&hcan) == HAL_CAN_STATE_LISTENING) ? 0x01 : 0x00;
            txData[2] = error_counter;  // 错误计数
            txData[3] = bus_load;       // 总线负载率
            break;
            
        case 0x02:  // 查询节点状态
            txData[0] = 0x02;
            txData[1] = node_status;    // 0:正常 1:警告 2:被动 3:离线
            txData[2] = tx_error_count;
            txData[3] = rx_error_count;
            break;
            
        case 0x03:  // 复位节点
            txData[0] = 0x03;
            txData[1] = 0xAA;  // 复位确认码
            HAL_CAN_Reset(&hcan);
            break;
    }
    
    HAL_CAN_AddTxMessage(&hcan, &txHeader, txData, &txMailbox);
}

曾经踩过的坑：有一次现场总线频繁报错，我查了三天。最后发现是一个节点的CAN收发器坏了，导致总线电压异常。从那以后，我每个项目都加物理层诊断。你想想看，如果连总线电压都监测不到，你怎么定位问题？

上位机的诊断界面，我习惯用颜色标识：绿色表示正常，黄色表示警告，红色表示故障。这样现场工程师一眼就能看出问题。

诊断项目 正常范围 警告范围 故障范围
CAN_H电压 2.5V ± 0.5V 2.0V-2.0V 或 3.0V-3.5V <2.0V 或 >3.5V
CAN_L电压 2.5V ± 0.5V 2.0V-2.0V 或 3.0V-3.5V <2.0V 或 >3.5V
终端电阻 60Ω ± 10% 50-54Ω 或 66-70Ω <50Ω 或 >70Ω
总线负载率 <30% 30%-60% >60%
错误帧率 0 1-5帧/分钟 >5帧/分钟

30.5 数据存储与回放

数据存下来才有价值。我一般存两种格式：

  • CSV格式：方便Excel打开，做数据分析
  • 二进制格式：方便回放，模拟现场工况

回放功能特别有用。比如客户说“昨天下午3点设备出问题了”，你直接把那个时间段的CAN数据回放一遍，就能复现问题。我靠这个功能解决过好几个疑难杂症。

// 数据回放功能
public void ReplayData(string filename)
{
    byte[] rawData = File.ReadAllBytes(filename);
    int offset = 0;
    
    while (offset < rawData.Length)
    {
        // 解析时间戳
        uint timestamp = BitConverter.ToUInt32(rawData, offset);
        offset += 4;
        
        // 解析CAN帧
        TPCANMsg msg;
        msg.ID = BitConverter.ToUInt32(rawData, offset);
        offset += 4;
        msg.LEN = rawData[offset++];
        msg.MSGTYPE = rawData[offset++];
        
        for (int i = 0; i < msg.LEN; i++)
        {
            msg.DATA[i] = rawData[offset++];
        }
        
        // 模拟发送
        PCANBasic.Write(PCANBasic.PCAN_USBBUS1, ref msg);
        
        // 按原始时间间隔发送
        Thread.Sleep((int)(timestamp - lastTimestamp));
        lastTimestamp = timestamp;
    }
}

30.6 项目调试心得

最后分享几个实战经验：

  1. 先跑通通信，再做功能。我见过太多人一上来就写应用层代码，结果CAN都没通。先用CAN分析仪发个标准帧，确认STM32能收到，再往下做。
  2. 日志要详细。每个CAN帧的收发都要打日志，包括时间戳。出了问题，日志就是你的救命稻草。
  3. 加看门狗。工业现场环境恶劣，STM32可能死机。加个独立看门狗，保证系统能自动恢复。
  4. 预留升级接口。我习惯在CAN协议里留一个“固件升级”命令。这样现场设备不用拆下来，通过CAN就能升级程序。

我的习惯：每个项目做完，我都会写一份“调试记录”。把踩过的坑、解决的方法都记下来。下次遇到类似问题，直接翻记录。这个习惯帮我省了无数时间。

记住，CAN总线不难，难的是细心和耐心。每个字节、每个位、每个时序，都可能影响整个系统的稳定性。多动手、多测试、多总结，你也能成为CAN总线专家。
