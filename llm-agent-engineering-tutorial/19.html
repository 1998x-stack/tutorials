第19章 OpenManus 分层架构

2025 年初，一个名为 OpenManus 的开源项目在 GitHub 上引发了不小的轰动。它的出现背景很耐人寻味——当时 Manus（一个闭源的通用 Agent 产品）以一段令人惊艳的 Demo 刷爆了技术社区，展示了 Agent 自主完成复杂任务的能力。但 Manus 是闭源的，开发者们既好奇它的内部实现，又希望有一个可自部署的替代方案。OpenManus 正是在这种"求贤若渴"的氛围中诞生的。

我第一次试用 OpenManus 时的感受是：它不像一个"框架"，更像一个"Agent 操作系统"。它的分层架构设计——从 BaseAgent 到 Manus 的四层体系——让你能清晰地看到 Agent 能力是如何一层层构建起来的。这章我们就来解剖这个架构，看看它的设计哲学和工程实现。

一句话总结： OpenManus 的分层架构就像一个"Agent 开发的乐高系统"——每一层都在上一层的基础上增加新的能力，从最底层的通用接口到顶层的全功能通用 Agent，你可以根据需求选择任意一层作为起点。

19.1 OpenManus 的诞生背景与定位

要理解 OpenManus 的设计，首先要理解它要解决的问题。2024 年到 2025 年间，Agent 框架越来越多，但开发者的痛点也很明显：

  • 框架耦合过深： LangChain 和 AutoGen 都有自己的抽象层，一旦选择就很难切换底层模型或工具生态
  • 配置复杂： 很多框架为了灵活性引入了大量的配置项，新手往往不知道从哪下手
  • 部署成本高： 许多框架依赖云服务或特定基础设施，本地部署体验不佳
  • MCP 支持滞后： 2025 年 MCP 协议快速普及，但很多框架对 MCP 的支持仍是"事后补丁"

OpenManus 的定位非常明确：一个轻量级、可本地部署、MCP 原生支持的通用 Agent 平台。它的核心理念是"分层抽象、按需组合"——不像 LangChain 那样提供大一统的抽象，而是让你在多个抽象层级之间自由选择。

特性 OpenManus LangChain AutoGen CrewAI
架构风格 分层继承 组件链式 对话驱动 角色声明式
MCP 支持 原生内置 需额外包 部分支持 2026 年新增
本地部署 一键启动 需配置 较复杂 较简单
工具生态 20+ 内置工具 数百集成 自定义为主 通过集成
学习曲线 平缓 陡峭 中等 平缓

截至 2026 年中，OpenManus 在 GitHub 上已获得超过 20k Star，社区活跃度很高。它的"轻量 + MCP 原生"定位特别适合以下场景：需要快速搭建 Agent 原型、希望完全本地运行、或者需要深度定制 Agent 行为的开发者。

19.2 分层 Agent 体系详解

OpenManus 最核心的设计是它的四层 Agent 继承体系。让我们从底层到顶层逐一剖析。

第一层：BaseAgent

BaseAgent 是所有 Agent 的基类，定义了最基本的接口和生命周期。它通常只包含几个核心方法：

from abc import ABC, abstractmethod
from typing import Optional

class BaseAgent(ABC):
    """All agents inherit from this base class."""

    def __init__(self, name: str = "Agent"):
        self.name = name
        self.state = {}  # shared state dict

    @abstractmethod
    async def run(self, prompt: str) -> str:
        """Execute the agent with the given prompt."""
        pass

    async def step(self) -> bool:
        """Execute a single step in the agent loop.
        Returns False when the task is complete.
        """
        return False

    def reset(self):
        """Reset agent state for a new task."""
        self.state = {}

这一层只定义了"一个 Agent 应该长什么样"——有一个名字、一个状态容器、一个运行入口。没有任何 LLM 调用逻辑，也没有工具调用能力。它是纯粹的接口约定。

第二层：ReActAgent

ReActAgent 继承自 BaseAgent，实现了经典的 ReAct（Reasoning + Acting）循环。这是整个架构中最关键的一层——它引入了 LLM 推理和思考-行动循环：

class ReActAgent(BaseAgent):
    """Agent with ReAct (Reasoning + Acting) loop."""

    def __init__(self, name: str, llm_config: dict):
        super().__init__(name)
        self.llm = LLM(config=llm_config)
        self.memory = []  # conversation history

    async def think(self) -> bool:
        """LLM reasons about next action.
        Returns False if LLM decides to respond directly.
        """
        response = await self.llm.ask(self.memory)
        if response.tool_calls:
            self.current_tool_calls = response.tool_calls
            return True
        self.final_response = response.content
        return False

    async def act(self) -> str:
        """Execute the tool calls decided in think()."""
        results = []
        for tc in self.current_tool_calls:
            result = await self.execute_tool(tc.name, tc.args)
            results.append(result)
        return "\n".join(results)

    async def run(self, prompt: str) -> str:
        self.memory.append({"role": "user", "content": prompt})
        while True:
            needs_action = await self.think()
            if not needs_action:
                return self.final_response
            action_result = await self.act()
            self.memory.append({"role": "assistant", "content": action_result})
设计亮点： 注意 think() 和 act() 是分离的两个方法。这意味着你可以重写 think() 来改变决策逻辑（比如加上反思步骤），而不需要动执行逻辑。这种"单一职责"的设计是分层架构的核心优势。

第三层：ToolCallAgent

ToolCallAgent 在 ReActAgent 的基础上增加了完整的工具调用框架。它维护了一个工具注册表，管理工具的注册、发现和执行：

  • 工具注册： 通过 register_tool() 方法将工具函数注册到 Agent 中
  • 工具发现： Agent 自动将已注册的工具转换为 LLM 可理解的 tool schema（JSON Schema 格式）
  • 工具执行： 封装了工具调用的完整生命周期——参数校验、执行、错误处理、结果格式化
  • MCP 集成： 原生支持通过 MCP 协议发现和调用远程工具

第四层：Manus

Manus 是顶层 Agent，继承自 ToolCallAgent。它代表了 OpenManus 能够提供的"开箱即用"的完整 Agent 能力。Manus 层的主要贡献在于：

  • 预设工具集： 预注册了 20+ 个常用工具（浏览器、文件操作、代码执行、API 请求等）
  • 智能规划： 增强了 think() 方法，加入任务分解和子目标管理能力
  • 错误恢复： 当工具调用失败时，自动尝试替代方案或重新规划
  • 流式输出： 支持实时流式显示思考过程和执行结果
四层设计哲学： 每一层解决一个问题——BaseAgent 定义"是什么"，ReActAgent 解决"怎么思考"，ToolCallAgent 解决"怎么行动"，Manus 解决"怎么好用"。这种分层让开发者可以根据需求选择切入点：只需要 ReAct 循环？用 ReActAgent。需要工具调用？用 ToolCallAgent。想要直接用？用 Manus。

19.3 Think-Act 循环与工具调度

理解 OpenManus 的 Think-Act 循环是掌握它的关键。这个循环在 ReActAgent 层实现，但在每一层都有增强。让我们看看一个完整的任务执行流程：

# OpenManus Think-Act 循环（伪代码流程）
async def agent_loop(agent, user_prompt):
    agent.reset()
    agent.memory.add_user_message(user_prompt)

    max_iterations = 30  # 防止死循环

    for step in range(max_iterations):
        # ===== Think Phase =====
        agent.update_state({"step": step})

        # LLM 决定：是调用工具还是直接回复
        response = await agent.llm.chat(
            messages=agent.memory.messages,
            tools=agent.get_tool_schemas(),
            system_prompt=agent.system_prompt
        )

        if response.is_text_response():
            # LLM 认为任务已完成
            agent.memory.add_assistant_message(response.content)
            return response.content

        # ===== Act Phase =====
        for tool_call in response.tool_calls:
            try:
                # 参数校验
                validated_args = tool_call.validate(agent.tool_registry)

                # 执行工具
                result = await agent.execute_tool(
                    tool_call.name,
                    validated_args,
                    timeout=60.0
                )

                # 结果处理
                agent.memory.add_tool_result(
                    tool_call.id, result
                )

            except ToolNotFoundError:
                agent.memory.add_tool_error(
                    tool_call.id,
                    f"Tool '{tool_call.name}' not found"
                )
            except TimeoutError:
                agent.memory.add_tool_error(
                    tool_call.id, "Tool execution timed out"
                )
            except Exception as e:
                agent.memory.add_tool_error(
                    tool_call.id, f"Unexpected error: {str(e)}"
                )

    # 超过最大迭代次数
    return "Max iterations reached. Task may be incomplete."

这个循环有几个值得注意的设计点：

  • 迭代上限： 30 次迭代的硬限制防止 Agent 无限循环。这个数字不是拍脑门定的——根据社区统计，95% 的合法任务在 30 步内能完成
  • 分层异常处理： 工具不存在、超时、通用错误——每个异常都有独立的处理路径，LLM 能根据错误类型做出不同的恢复策略
  • 流式状态更新： 每一步的状态变化都会触发事件，UI 层可以实时展示 Agent 的"思考过程"

19.4 状态管理与异常处理

OpenManus 的状态管理采用了"共享状态 + 分层存储"的架构。每个 Agent 实例维护一个 state 字典，但这个字典在不同层级有不同的责任：

层级 状态内容 生命周期
BaseAgent.state name, task_id, start_time 整个 Agent 生命周期
ReActAgent.memory 对话历史、工具调用记录 单次 run() 调用
ToolCallAgent.registry 已注册工具列表、工具调用统计 Agent 创建到销毁
Manus.plan 任务分解计划、子任务进度 单次任务

异常处理方面，OpenManus 采用了"三层防线"策略：

  • 第一层——工具执行异常： 工具调用超时、参数校验失败等，由 ToolCallAgent 捕获并重试
  • 第二层——LLM 调用异常： API 超时、Token 耗尽、格式错误，由 ReActAgent 捕获，自动降级或重新请求
  • 第三层——全局异常： 上面两层都未能处理的异常，由 BaseAgent 捕获，记录错误并安全终止
生产环境提醒： 虽然 OpenManus 内置了异常处理，但在生产环境中建议加上自己的兜底逻辑。特别是对于长时间运行的任务，建议在外部再加一层超时控制的 watchdog，防止 Agent 因为某些边界情况卡死在循环中。

19.5 本地部署与模型配置

OpenManus 的一大优势就是本地部署极其简单。以下是标准的部署流程：

# 1. 安装
pip install openmanus

# 2. 最小配置 config.toml
[llm]
model = "claude-sonnet-4-20250601"  # 或 gpt-4o, deepseek-v3 等
api_key = "sk-..."                  # 从环境变量读取更安全
base_url = "https://api.anthropic.com"

[llm.parameters]
temperature = 0.7
max_tokens = 4096

# 3. 一行代码启动 Agent
openmanus run "帮我调研一下2026年最流行的前端框架"

OpenManus 支持多种模型后端。下面是 2026 年主流的配置组合：

模型 适用场景 推荐配置
Claude Sonnet 4 通用任务、代码生成 temperature=0.3, max_tokens=8192
GPT-4o 工具调用密集任务 temperature=0, max_tokens=4096
DeepSeek-V3 本地部署、成本敏感 temperature=0.7, max_tokens=4096
Qwen3-72B 中文场景、本地部署 temperature=0.5, max_tokens=4096
本地部署技巧： 如果你使用 Ollama 或 vLLM 运行本地模型，可以在 config.toml 中将 base_url 设置为 http://localhost:11434（Ollama）或 http://localhost:8000（vLLM）。OpenManus 会自动适配 OpenAI 兼容的 API 格式。

19.6 实战：用 OpenManus 搭建研究助手

理论说得再多，不如动手写一个实战项目。下面我们用 OpenManus 搭建一个自动化研究助手，它能接收一个研究课题，自动搜索资料、整理信息、生成报告。

from openmanus import Manus, ToolCollection
from openmanus.tools import WebSearch, WebFetch, FileWrite

# 1. 创建 Manus Agent
agent = Manus(
    name="ResearchAssistant",
    llm_config={
        "model": "claude-sonnet-4-20250601",
        "temperature": 0.3
    }
)

# 2. 注册研究所需的工具
tools = ToolCollection([
    WebSearch(max_results=5),         # 搜索最多 5 条结果
    WebFetch(max_chars=10000),        # 抓取页面内容（字符限制）
    FileWrite(output_dir="./reports") # 保存到 reports 目录
])

for tool in tools:
    agent.register_tool(tool)

# 3. 执行研究任务
result = await agent.run("""
请完成以下研究任务：

课题：2026年边缘计算在物联网中的应用趋势

研究要求：
1. 搜索最新的行业报告和技术文章
2. 总结边缘计算在 IoT 中的三大应用方向
3. 列出每个方向的主要技术方案和代表厂商
4. 将研究结果保存到 research_report.md 文件中

格式要求：
- 使用 Markdown 格式
- 包含小标题、数据表格和参考链接
""")

print(f"研究完成，结果已保存。最终总结：{result}")

运行这个 Agent，你会看到它经历了以下步骤：

  • 调用 WebSearch 搜索"2026 边缘计算 IoT 趋势"
  • 遍历搜索结果，调用 WebFetch 抓取相关文章内容
  • 基于抓取到的信息，进行分析和总结
  • 调用 FileWrite 将报告写入本地文件
  • 返回最终摘要给用户
效果对比： 在我们团队的测试中，同样的研究任务——人工完成大约需要 2-3 小时（搜索 + 阅读 + 整理），而配置得当的 OpenManus Agent 只需要 3-5 分钟。当然，质量上仍然需要人工审核和补充，但作为"第一稿生成器"，效率提升是数量级的。

值得注意的是，这个研究助手只是一个起点。你可以在它的基础上增加更多工具：数据库查询（用于知识库检索）、代码执行（用于数据分析）、邮件发送（用于自动分发报告）等。随着工具生态的扩展，Agent 的能力边界会不断拓宽。

本章小结

在这一章中，我们深入剖析了 OpenManus 的分层架构设计。从 BaseAgent 的接口约定，到 ReActAgent 的思考-行动循环，再到 ToolCallAgent 的工具管理能力和 Manus 的全功能整合——每一层都在解决特定的问题，让开发者可以按需选择切入点。同时，我们也介绍了本地部署的流程和一个完整的研究助手实战案例。

OpenManus 的分层设计给了我们一个重要启示：好的框架不是把所有能力堆在一起，而是把问题分解成清晰的层次，让每一层聚焦于一个职责。这也是为什么 OpenManus 在上手简单的同时，仍然保持了很强的灵活性。
