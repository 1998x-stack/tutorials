第十二章：跨域资源共享（CORS）

各位同学好。在上一章我们聊认证的时候，你的前端代码调用后端API获取Token——如果前端跑在 localhost:3000，后端在 api.example.com，你大概率见过这样的报错：

Access to fetch at 'http://api.example.com/token' from origin 'http://localhost:3000'
has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present.

这就是浏览器同源策略（Same-Origin Policy）在保护你——但它保护的同时也制造了麻烦。这一章我们就来彻底搞懂CORS是什么、怎么工作、以及常见的坑怎么填。

12.1 同源策略

首先得理解：CORS不是用来"打开"安全限制的，它是用来绕过浏览器安全限制的。这个安全限制叫做同源策略。

所谓"同源"（Same-Origin），必须同时满足三个条件：

  • 协议（Scheme）相同 —— http:// 和 https:// 不同源
  • 主机（Host）相同 —— 域名或IP必须完全一致
  • 端口（Port）相同 —— 默认端口（80/443）和显式端口（8080）被视为不同
URL AURL B是否同源原因
http://example.com/page1http://example.com/page2是协议、主机、端口均相同
http://example.comhttps://example.com否协议不同（http vs https）
http://example.comhttp://api.example.com否主机不同（子域名）
http://example.comhttp://example.com:8080否端口不同

同源策略到底禁止了什么？很多人以为它"禁止跨域请求"，这是不准确的。实际上，浏览器允许发出跨域请求，真正被禁止的是读取跨域响应。也就是说，请求发出去了，服务器也处理了，但浏览器把响应结果扣下了——你的JavaScript拿不到数据。

具体来说，同源策略限制的是：

  • DOM访问 —— iframe跨域时不能操作对方页面的DOM
  • AJAX请求（fetch / XMLHttpRequest） —— 可以发，但读不到响应
  • Cookie、localStorage、IndexedDB —— 只对同源页面开放

核心理解：同源策略是浏览器的策略，不是HTTP协议本身的限制。你用curl或者Postman发请求，不会有任何跨域问题——因为curl没有同源策略。这也是为什么后端服务之间通信完全不需要关心CORS。记住：CORS是浏览器帮你做的安全检查，不是服务器之间的安全机制。

12.2 CORS简介

CORS（Cross-Origin Resource Sharing）的机制很简单：服务器在响应头中告诉浏览器"我允许你这个来源访问我的资源"。核心就是Access-Control-Allow-Origin这个响应头。

HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://myapp.com
Content-Type: application/json

{"message": "你好，跨域世界"}

当Access-Control-Allow-Origin的值匹配请求的Origin时，浏览器就允许JavaScript读取响应内容。这个值也可以是*（通配符），表示允许任意来源——但后面我们会看到，通配符在某些场景下有约束。

CORS将请求分为两类：简单请求（Simple Request）和预检请求（Preflighted Request）。

简单请求必须同时满足以下条件：

  • 方法只能是 GET、HEAD、POST
  • 除了浏览器自动添加的Header（User-Agent、Host等），只能添加 Accept、Accept-Language、Content-Language、Content-Type 这几个Header
  • Content-Type 的值只能是 application/x-www-form-urlencoded、multipart/form-data、text/plain 之一
  • 没有使用 XMLHttpRequest 的 withCredentials

简单请求的流程：浏览器直接发出请求，服务器返回 Access-Control-Allow-Origin，浏览器检查是否允许。

常见误区：很多新手以为只要加上 Access-Control-Allow-Origin: * 就万事大吉了。实际上，如果请求带了Authorization头（比如Bearer Token），或者Content-Type: application/json，这个请求就不是"简单请求"了，会触发预检流程。

12.3 预检请求（Preflight）

当请求不满足简单请求的条件时，浏览器会先发送一个OPTIONS请求作为预检，询问服务器是否允许实际的请求。这就是你经常在DevTools里看到的"preflight"——很多后端开发者第一次看到OPTIONS请求时还以为是黑客攻击。

// 预检请求（Preflight）
OPTIONS /api/data HTTP/1.1
Host: api.example.com
Origin: http://myapp.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization

// 预检响应
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: http://myapp.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400

预检请求的关键头：

  • Access-Control-Request-Method —— 告诉服务器实际请求会用什么HTTP方法
  • Access-Control-Request-Headers —— 告诉服务器实际请求会带哪些自定义头

服务器返回的关键头：

  • Access-Control-Allow-Origin —— 允许的来源
  • Access-Control-Allow-Methods —— 允许的方法列表
  • Access-Control-Allow-Headers —— 允许的请求头列表
  • Access-Control-Max-Age —— 预检结果缓存时间（秒），在这段时间内不再发预检请求

预检请求带来的问题是：一次实际请求变成了两次。如果API的延迟本身就不低，预检请求会让延迟翻倍。优化方法：

  • 合理设置 Access-Control-Max-Age（比如86400秒=1天），减少重复预检
  • 如果确定场景安全，尽量使用简单请求（避免自定义Header、避免非简单Content-Type）

踩坑故事：有次我帮一个团队排查线上API响应慢的问题。他们做了前后端分离，前端在 app.example.com，后端在 api.example.com。前端用的是 fetch 发送 POST application/json 请求，每次请求都带 Authorization 头。这触发了预检——每次实际请求前都有一个OPTIONS请求。问题在于后端没有设置 Access-Control-Max-Age，导致每个API调用都发两次请求，延迟直接翻倍。更坑的是，后端对OPTIONS请求的处理逻辑有问题，每次还走了一遍完整的认证流程（查数据库、验Token），导致OPTIONS比实际请求还慢。最后我们把OPTIONS路由提前返回204，并设置了24小时的预检缓存，延迟从800ms降到了300ms。

12.4 携带凭证

跨域请求中经常需要携带Cookie或HTTP认证信息。默认情况下，跨域请求是不会携带这些凭证的——这是安全考虑。如果需要携带，需要两边都做出配置。

前端侧：

  • XMLHttpRequest 需要设置 withCredentials = true
  • fetch 需要设置 credentials: 'include'
// fetch 带凭证
fetch('http://api.example.com/user', {
  credentials: 'include'
});

// XHR 带凭证
let xhr = new XMLHttpRequest();
xhr.withCredentials = true;
xhr.open('GET', 'http://api.example.com/user');
xhr.send();

服务端侧：

必须返回以下响应头：

  • Access-Control-Allow-Credentials: true
  • Access-Control-Allow-Origin 不能是 *，必须指定具体的来源
  • 如果需要暴露自定义响应头给前端读取，需要设置 Access-Control-Expose-Headers
HTTP/1.1 200 OK
Access-Control-Allow-Origin: http://myapp.com
Access-Control-Allow-Credentials: true
Set-Cookie: session=abc123; SameSite=None; Secure

重要提醒：当 Access-Control-Allow-Credentials: true 时，Access-Control-Allow-Origin 必须是一个具体的源，不能用 *。这是浏览器的强制要求——因为"允许所有来源还能携带Cookie"会完全破坏凭证保护机制。如果你需要动态支持多个域名，需要在服务端代码中根据请求的 Origin 头动态构建允许列表。

12.5 CORS常见问题

最后，我们总结几个开发中最常见的CORS问题及其解决方案。

问题1：通配符Origin + Credentials

错误配置：

Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true

浏览器会拒绝这个响应，报错："The value of 'Access-Control-Allow-Origin' in the response must not be the wildcard '*' when the request's credentials mode is 'include'."。解决方案：服务端读取请求头的 Origin，判断是否在白名单中，然后返回具体的Origin。

问题2：预检缓存（Preflight Cache）

设置 Access-Control-Max-Age 可以减少OPTIONS请求的次数。如果不设置，浏览器默认的缓存时间各浏览器不同（Chrome约5秒，Firefox约24小时）。建议显式设为一个合理的较大值，比如 86400（1天）。需要注意的是，Access-Control-Max-Age 的单位是秒，最大建议不超过86400。

问题3：DevTools调试CORS错误

当你看到CORS报错时，记住几步排查流程：

  1. 打开DevTools的Network面板，找到被拦的请求
  2. 查看请求头中是否携带了 Origin（确认浏览器正确添加了来源头）
  3. 查看响应头中是否有 Access-Control-Allow-Origin（确认服务器返回了正确的CORS头）
  4. 如果看到OPTIONS请求（预检），检查预检响应是否返回了正确的 Access-Control-Allow-Methods 和 Access-Control-Allow-Headers
  5. 如果涉及Cookie，检查 Access-Control-Allow-Credentials 是否为 true，以及 Access-Control-Allow-Origin 是否不是通配符

问题4：多域名场景

很多人会想到前端通过 Origin 头来动态判断。但更好的方式是在网关层统一处理CORS。Nginx示例：

# 在反向代理配置中统一处理CORS
location /api/ {
    if ($http_origin ~* (https?://(example\.com|app\.example\.com))) {
        add_header Access-Control-Allow-Origin $http_origin;
    }
    add_header Access-Control-Allow-Methods "GET, POST, PUT, DELETE, OPTIONS";
    add_header Access-Control-Allow-Headers "Content-Type, Authorization";
    add_header Access-Control-Max-Age 86400;

    # 处理预检请求
    if ($request_method = OPTIONS) {
        return 204;
    }
}

补充提示：开发环境用CORS代理（比如webpack-dev-server的proxy配置、或者 http-proxy-middleware）也可以避免跨域问题。它的原理是在开发服务器上做一层代理转发——让前端请求发给同源的开发服务器，开发服务器再转发给后端API。这样浏览器看到的就是同源请求，不需要处理CORS。但这只是开发阶段的权宜之计，生产环境还是需要正确配置CORS。
