项目概述

大家好，欢迎来到本课程的最后一章。经过前面 27 章的学习，我们已经从 Common Crawl 的数据采集一路走到了在线动态配比的前沿技术。现在是时候把所有知识串联起来了——今天我们要做一个综合实战项目：从零构建一个完整的语料清洗流水线。

项目的目标非常具体：从 Common Crawl 的一个快照中提取出 10GB 高质量中文训练数据。我们会用到前 27 章中几乎每一个核心知识点——正文提取、语言识别、URL 过滤、启发式规则、两种去重算法、困惑度评分、质量分类器，以及最终的 Tokenization 和数据格式化。项目输出是清洗后的 JSON Lines 格式数据 + 一份完整的评估报告。

为了增加真实感，我参考了 2025 年一位工程师在知乎上分享的项目经历：他当时受命为公司构建一个中文语料清洗流水线，从零开始用了三周时间完成了这个项目。他的经验告诉我们——最难的不是单个环节的技术实现，而是把所有环节拼成一个可靠、可复现的流水线。你今天在本地做的这个项目，本质上和那些大厂数据团队的工作没有区别。

项目设计

整个项目分为 7 个步骤，每个步骤对应一到几章的知识：

  • Step 1：数据下载。 选择一个 Common Crawl 快照（如 CC-MAIN-2024-50），下载其中的 WET 文件（约 50GB 压缩体积）。WET 文件只包含纯文本内容，不包含 HTML 标记，适合作为数据处理的起点。使用 aws s3 cp 命令从 Amazon S3 加速下载。
  • Step 2：正文提取 + 中文识别。 对 WET 中的每条记录，使用 trafilatura 做进一步的正文精提取，然后用 fastText 语言分类模型识别每条文档的语言。保留中文（zh）且置信度高于 0.8 的文档。
  • Step 3：URL 过滤 + 启发式规则过滤。 基于 URL 黑名单（如广告、低俗内容站点）过滤垃圾域名然后应用启发式规则——最小字符数（≥200 字）、最大行数比例（行数/总字符数 < 0.3）、标点符号占比（5%-40%）等。
  • Step 4：精确去重 + MinHash 去重。 先做 SHA-1 精确去重，再做 MinHash 近邻去重（128 个哈希函数，Jaccard 相似度阈值 0.85）。
  • Step 5：困惑度评分 + 教育分类器评分。 用 KenLM 中文语言模型计算每条文档的困惑度，过滤 perplexity > 80 的文档。再用一个基于 BERT 的中文教育质量分类器给文档打分，保留评分 ≥ 3.0 的文档。
  • Step 6：Tokenization 和格式化。 使用你选用的 tokenizer（如 Qwen 或 Llama 的中文词表）对数据进行 tokenization，统计 token 数量，输出为 JSON Lines 格式。
  • Step 7：质量评估。 计算最终数据集的各项指标——重复率、中文占比、平均困惑度、教育评分分布。在小模型（1.5B）上训练 1000 步，验证 loss 持续下降。
# 完整流水线骨架代码
import json
from pathlib import Path
from datatrove.executor import LocalPipelineExecutor
from datatrove.pipeline.readers import WETReader
from datatrove.pipeline.filters import (
    LanguageFilter, URLFilter, HeuristicFilter,
    PerplexityFilter, FineWebEduFilter
)
from datatrove.pipeline.dedup import ExactDedup, MinHashDedup
from datatrove.pipeline.writers import JsonLinesWriter

def build_pipeline(input_path: str, output_path: str):
    # Step 1: 读取 WET 文件
    reader = WETReader(input_path)

    # Step 2-3: 过滤阶段
    pipeline = (
        reader
        .filter(LanguageFilter(target_langs=["zh"], threshold=0.8))
        .filter(URLFilter(blocklist="url_blacklist.txt"))
        .filter(HeuristicFilter(min_chars=200, max_line_ratio=0.3))
    )

    # Step 4: 去重阶段
    pipeline = (
        pipeline
        .dedup(ExactDedup())
        .dedup(MinHashDedup(num_permutations=128, threshold=0.85))
    )

    # Step 5: 质量过滤阶段
    pipeline = (
        pipeline
        .filter(PerplexityFilter(model="kenlm-zh", threshold=80))
        .filter(FineWebEduFilter(model="bert-zh-edu", threshold=3.0))
    )

    # Step 6: 输出
    pipeline = pipeline.write(JsonLinesWriter(output_path))

    return pipeline

if __name__ == "__main__":
    executor = LocalPipelineExecutor(
        pipeline=build_pipeline(
            input_path="s3://commoncrawl/cc-main-2024-50/wet/",
            output_path="./cleaned_data/"
        ),
        tasks=16  # 并行任务数
    )
    executor.run()

关键技术选择

在实现这个项目时，有几个技术决策会影响最终的效果和开发效率。

为什么选 DataTrove 作为流水线框架？ 经过第 23 章的学习，你应该已经熟悉了 DataTrove 的模块化设计。它天然支持 WET 读取、语言过滤、MinHash 去重等我们需要的几乎所有操作符。而且 DataTrove 的 LocalPipelineExecutor 可以在单机上运行排错，调试好后再无缝切换到分布式环境。

为什么选 WET 而不是 WARC？ WARC 文件包含完整的 HTTP 响应头、请求元数据和原始 HTML。WET 文件只包含纯文本内容。对于我们的项目来说，WET 文件体积小得多（约 WARC 的 1/5），不需要额外的 HTML 解析步骤，是快速验证流水线的理想起点。如果你想做更精细的正文提取（如第 3 章所述），可以换成 WARC 输入。

为什么选 fastText 做中文分类？ fastText 模型体积小、推理速度快，在语言识别任务上精度足够高。它的分类模型只有几百 MB，可以在 CPU 上高速运行。相比之下，如果使用基于 BERT 的语言识别模型，虽然精度略高，但推理速度慢 10-20 倍，不适合大规模数据处理。

硬件提示： 完成这个项目建议至少 16GB 内存和 8 个 CPU 核心。MinHash 去重是内存最密集的环节——128 个签名版本需要为每个文档存储签名，当处理数百万文档时，内存占用会迅速攀升。如果你的硬件有限（比如只有 8GB 内存），可以先使用 1/10 的数据做测试——把 WET 文件列表只取前 10%，跑通整个流程后再扩大规模。流程验证通过后，再用完整数据跑一次。

项目验收标准

一个好的项目需要有明确的验收标准。以下是这个实战项目的核心指标：

  • 数据量： 清洗后不少于 8GB（指 token 化后的数据量，不是原始文本量）。如果低于这个数字，说明过滤可能过于激进——需要回头检查各步骤的保留率。
  • 重复率： 最终数据集的精确 + 近邻重复率低于 5%。过高的重复率说明去重阶段的效果不达标。
  • 中文占比： 最终数据中中文文档占比 > 95%。这是项目的基本要求——清洗对象是中文数据。
  • 平均困惑度： 低于 60。高于 60 说明数据中仍有大量噪声，需要加强过滤。
  • 教育评分： 平均教育评分 > 3.0（基于 5 分制的教育分类器）。低于 3.0 说明数据质量不够高。
  • 模型验证： 用清洗后的数据在小模型（1.5B）上训练 1000 steps，训练 loss 呈现稳定的下降趋势。
终极检验标准： 数据工程的最终检验标准从来不是数据本身有多"干净"，而是模型用这些数据训练出来的效果。一个困惑度很低但高度同质化的数据集，和困惑度略高但信息丰富多样的数据集，后者往往能训练出更强的模型。所以永远不要为了追求指标而牺牲数据的内容价值——这是 FineWeb-Edu、Dolma 等一系列标杆项目给我们的核心启示。

常见问题与解决方案

根据我了解到的一线工程师的真实反馈，以下四个问题是这个项目中最高频的"卡点"：

问题 1：下载太慢。 Common Crawl 的 WET 文件存储在 AWS 的 S3 公开存储桶中。从国内环境直接下载可能很慢。解决方案：使用 AWS S3 加速端点（如 s3-accelerate）或者通过 AWS CLI 的 --region us-east-1 参数指定区域。如果还是太慢，可以考虑从 HuggingFace 的镜像站点下载。

问题 2：内存不足。 MinHash 去重阶段需要将所有文档的 MinHash 签名加载到内存中。100 万篇文档的 128 维签名约占用 1GB 内存，但 Python 对象开销可能使其膨胀到 4-5GB。解决方案：减少签名数量（从 128 降到 64），这虽然会略微降低去重精度，但大幅减少内存占用。或者在 DataTrove 中启用磁盘溢出（spill-to-disk）模式。

问题 3：中文识别不准。 fastText 在多语言混合的短文本上可能出现误判。一篇中英文混合的技术博客可能被判定为英文。解决方案：提高 fastText 的置信度阈值到 0.85 或 0.9，或者改用更精确的语言识别模型（如基于 BERT 的 multilingual 分类器）。

问题 4：过度过滤导致数据量不够。 各步骤加起来的累积保留率可能只有 5-8%，导致最终数据量远低于目标。解决方案：逐步骤检查保留率——每个步骤的保留率应该在我们的经验范围内。如果某一步骤的保留率异常低，对应的过滤参数可能过于严格，需要适当放宽。

部署警告： 这个项目最常见的失败模式不是"技术不会"，而是"本地跑通了，部署到集群挂了"。分布式环境和单机环境有本质差异——路径问题（S3 vs 本地路径）、权限问题（S3 访问凭证）、资源限制（每个节点的 CPU/内存配置不同）、依赖版本不一致。建议一开始就考虑分布式部署：即使你在单机上开发测试，也要按照分布式规范来写代码——使用相对路径、配置外部化、依赖版本锁定。

结语：LLM 数据工程的未来

到这里，我们 28 章的学习旅程就全部结束了。让我们回顾一下走过的路：我们从 Common Crawl 的数据采集出发，经历了正文提取、语言识别、规则过滤、URL 过滤、精确去重、MinHash 近邻去重、多粒度去重；然后深入了困惑度过滤、教育分类器、LLM 辅助标注、去污染；接着学习了多源配比、分布式架构、数据存储、可复现性、质量评估体系；之后探索了 Tokenization、词表设计、合成数据、领域适配；最后通过 DataTrove、Dolma、六大范式对比、自动化系统和在线数据选择，来到了这个综合实战项目。

这一路上的每一个环节，都是将"原始数据"转化为"训练语料"不可或缺的一环。让我再强调三条核心原则——它们不是抽象的理论，而是来自无数实践者用 GPU 小时换来的经验：

  • 第一条：Quality over Quantity。 从 FineWeb-Edu 到 DCLM，从 RefinedWeb 到 FLUX，2024-2026 年所有重要的实验都在反复验证同一个真理：用更少的、更高质量的数据训练，总能胜过多而杂的数据。但要注意，"高质量"的定义取决于你的目标——不要盲目追求一个单一的评分指标。
  • 第二条：透明可复现。 Dolma 项目给了整个社区一个标杆：数据工程不应该是黑箱。记录每次清洗操作的参数，公开数据配比的比例，分享失败的实验。在 AI 安全性越来越受到重视的今天，透明可复现的数据工程不仅是学术道德的要求，更是行业规范的基础。
  • 第三条：持续迭代。 不存在"完美的清洗方案"。每次训练任务的结束，都应该是下一次数据优化的开始。数据工程本质上是一个持续的工程优化过程——而不是一次性的数据准备任务。

最后，用一句在数据工程圈子里广为流传的话来为大家送别："LLM 的竞争，最终是数据的竞争。掌握了数据工程，就掌握了 AI 的源头。"
