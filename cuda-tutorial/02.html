CUDA编程环境搭建

大家好，我是你们的讲师。如果你在一台新机器上搭建过深度学习环境，你一定知道那种"装了半天最后跑不起来"的痛苦。CUDA开发环境比那还要敏感一点——因为你要同时跟驱动、Toolkit、编译器、链接器四个层面打交道。任何一个环节版本不匹配，都会导致"明明装好了，但就是跑不通"的尴尬局面。

这一章，我会带你从头到尾搭好一个干净、可工作的CUDA开发环境。不管你用Windows、Linux还是macOS，我都会给出对应的方案。你不只是在"安装软件"——你是在为自己的GPU计算之旅打好地基。地基稳了，后面几十章才能走得顺畅。

CUDA Toolkit安装

首先，需要搞清楚两个概念：NVIDIA驱动和CUDA Toolkit不是一回事。驱动让你的操作系统"认识"GPU，而Toolkit里面装着编译器（nvcc）、CUDA库、头文件和开发工具。好消息是：如果你装的是最新的NVIDIA驱动，它已经内置了runtime库，足够运行已经编译好的CUDA程序。但你要自己写代码、编译，就必须再装Toolkit。

安装完成后，用以下两个命令验证环境：

# 检查驱动版本和GPU信息
nvidia-smi

# 检查CUDA Toolkit版本
nvcc --version

这两个命令输出中各自有一个CUDA版本号——注意，它们可能不一样！nvidia-smi 显示的是驱动支持的最高CUDA版本，而 nvcc --version 才是你实际安装的Toolkit版本。只要Toolkit版本 不高于 驱动支持的最高版本，就能正常工作。

操作系统 安装方式 安装命令 / 步骤 注意事项
Ubuntu/Debian Network Installer (.run) 或 deb包 sudo apt install nvidia-cuda-toolkit 或从NVIDIA官网下载runfile apt源版本可能滞后，建议从官网下载指定版本
RHEL/CentOS/Fedora RPM包 配置NVIDIA CUDA仓库后 sudo dnf install cuda 需要先禁用Nouveau开源驱动
Windows (10/11) EXE网络安装器 从NVIDIA官网下载EXE，一路Next即可 安装后需要将 C:\Program Files\NVIDIA GPU Computing Toolkit\CUDA\vX.Y\bin 加入PATH
macOS 官方PKG 从NVIDIA Developer Zone下载对应版本 Apple自2016年后不再使用NVIDIA GPU。仅限有NVIDIA独显的老款Mac
推荐做法： 在Linux上，我推荐用runfile安装。它给你最大的控制权——你可以选择安装路径，可以管理多版本CUDA共存，卸载的时候也干净。deb包会把驱动和Toolkit绑定在一起，后续升级容易出问题。

开发工具链概览

CUDA的工具链和常规C/C++开发类似，但多了一个关键环节——nvcc编译器。nvcc是你的主要编译工具：它负责把 .cu 文件编译成GPU可执行的代码。它的工作流程大致是：首先将 .cu 文件拆分成GPU代码和CPU代码两部分，GPU代码编译成PTX（并行线程执行，一种中间指令集），CPU代码交给宿主的C++编译器（gcc/clang/MSVC）处理。最后再把两部分链接到一起。

除了nvcc，还有几个值得一提的组件：

  • NVRTC（Runtime Compilation）：一个运行时编译库，允许你在程序运行的时候编译CUDA代码。适合需要动态生成kernel的场景
  • CUDA Math Library：提供了一系列高度优化的数学函数，包括三角函数、指数函数、随机数生成等
  • cuBLAS / cuFFT / cuRAND / cuSPARSE：CUDA生态中的领域库，我们会在第18章详细介绍

下面是一个简单的Makefile示例，它足够用来管理本书大多数章节的编译：

# 简单的CUDA项目Makefile
NVCC = nvcc
NVCCFLAGS = -O2 -arch=sm_80  # 根据你的GPU计算能力调整
TARGET = hello_cuda
SOURCES = main.cu

all: $(TARGET)

$(TARGET): $(SOURCES)
	$(NVCC) $(NVCCFLAGS) -o $@ $^

clean:
	rm -f $(TARGET)

run: $(TARGET)
	./$(TARGET)

如果你更喜欢CMake（在大型项目中更推荐），下面是一个对应的 CMakeLists.txt：

cmake_minimum_required(VERSION 3.20)
project(hello_cuda LANGUAGES CUDA CXX)

# 启用CUDA架构自动检测
set(CMAKE_CUDA_ARCHITECTURES "80" CACHE STRING "GPU architecture")

# 添加可执行文件
add_executable(hello_cuda main.cu)

# 设置编译选项
target_compile_options(hello_cuda PRIVATE -O2 -Wall)

# 链接CUDA库
target_link_libraries(hello_cuda PRIVATE cudart)

注意CMake从3.18版本开始原生支持CUDA作为一门语言（LANGUAGES CUDA），不需要再手动调用 find_package(CUDA) 了。这是一个很大的改进——你可以像管理C++代码一样管理CUDA代码。

验证安装——设备查询程序

环境装没装好，跑一个完整的程序才知道。NVIDIA官方提供了一个叫 deviceQuery 的工具（在Toolkit的samples目录下），但其实我们完全可以自己写一个，既能验证环境，又能熟悉一下CUDA API：

// deviceQuery.cu —— 查询GPU设备属性
#include <stdio.h>
#include <cuda_runtime.h>

int main() {
    int deviceCount;
    cudaGetDeviceCount(&deviceCount);

    if (deviceCount == 0) {
        printf("没有找到CUDA可用设备！\n");
        return -1;
    }

    for (int dev = 0; dev < deviceCount; ++dev) {
        cudaDeviceProp prop;
        cudaGetDeviceProperties(&prop, dev);

        printf("设备 %d: %s\n", dev, prop.name);
        printf("  计算能力:       %d.%d\n", prop.major, prop.minor);
        printf("  多处理器数量:   %d\n", prop.multiProcessorCount);
        printf("  每个SM的CUDA核心数: %d\n", prop.maxThreadsPerMultiProcessor / 32 * 64);
        printf("  全局内存:       %.2f GB\n", prop.totalGlobalMem / 1024.0 / 1024 / 1024);
        printf("  共享内存/块:    %.2f KB\n", prop.sharedMemPerBlock / 1024.0);
        printf("  寄存器/块:      %d\n", prop.regsPerBlock);
        printf("  最大线程/块:    %d\n", prop.maxThreadsPerBlock);
        printf("  最大线程维度:   [%d, %d, %d]\n", prop.maxThreadsDim[0],
               prop.maxThreadsDim[1], prop.maxThreadsDim[2]);
        printf("  最大网格维度:   [%d, %d, %d]\n", prop.maxGridSize[0],
               prop.maxGridSize[1], prop.maxGridSize[2]);
        printf("  内存时钟频率:   %.0f MHz\n", prop.memoryClockRate / 1000.0);
        printf("  内存总线宽度:   %d bits\n", prop.memoryBusWidth);
        printf("  内存带宽:       %.2f GB/s\n",
               2.0 * prop.memoryClockRate * (prop.memoryBusWidth / 8) / 1e6);
        printf("\n");
    }
    return 0;
}

编译运行：

nvcc -o deviceQuery deviceQuery.cu
./deviceQuery

输出中最重要的几个属性，解释如下：

属性 含义 为什么重要
计算能力 (Compute Capability) 如8.0、8.9、9.0，表示GPU架构的代际 决定你的代码能用哪些硬件特性（比如Tensor Core需要7.0+）
多处理器数量 (SM Count) GPU内部有多少个流式多处理器 并行度的上限——SM越多，能同时运行的线程块越多
全局内存大小 显存总量 能处理多大的数据集——深度学习的batch size受它限制
内存带宽 显存读写速度 (GB/s) 对大多数计算任务来说，这是最重要的性能指标之一
最大线程/块 一个线程块最多能有多少线程 影响你的block size选择（通常是1024）

常见安装坑与排查

说一个真实的故事。2021年一个朋友买了块RTX 3090，兴冲冲装好驱动，跑 nvidia-smi 正常识别。然后装CUDA 11.4 Toolkit，一切顺利。写了一个简单的"Hello GPU" kernel，编译通过。运行——"No CUDA-capable device is detected"。他当时整个人都懵了：刚买的显卡，难道坏了？折腾了三个小时，最后发现问题出在：驱动和Toolkit版本不匹配。他通过Ubuntu的apt自动装的是470系列驱动（支持CUDA 11.4），但他自己手动装了CUDA 12.0的Toolkit。驱动不认识12.0生成的PTX指令，直接说"没有CUDA设备"——其实设备好好的，是驱动和编译器之间"语言不通"。

排查清单： 如果你遇到"no CUDA-capable device"或其他环境问题，按以下顺序检查：
1. 运行 nvidia-smi——如果看不到GPU，先搞定驱动
2. 运行 nvcc --version——确保Toolkit版本不高于nvidia-smi显示的CUDA版本
3. 检查 PATH 是否包含 /usr/local/cuda/bin
4. 检查 LD_LIBRARY_PATH 是否包含 /usr/local/cuda/lib64
5. 检查 /usr/local/cuda 软链接是否指向你想要的版本目录
多版本CUDA管理小技巧： 我经常在同一台机器上装多个CUDA版本（比如11.8和12.5），用于兼容不同的项目。方法是把 /usr/local/cuda 软链接指向目标版本：sudo ln -sf /usr/local/cuda-12.5 /usr/local/cuda。切换版本只需要重新链接并更新PATH即可。

另外，还有一个容易被忽视的问题：内核模块冲突（特别是Linux上）。如果你用的是开源的Nouveau驱动（很多Linux发行版默认启用），它和NVIDIA闭源驱动是冲突的。必须禁用Nouveau，NVIDIA驱动才能正常工作。具体做法是在 /etc/modprobe.d/blacklist-nouveau.conf 中添加 blacklist nouveau，然后重新生成initramfs并重启。
