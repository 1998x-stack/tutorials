第1章：Redis 源码全景图

大家好，我是你们的讲师。今天开始我们将一起走进 Redis 的源代码世界。很多同学用 Redis 用了好几年，对 SET/GET、缓存雪崩、主从复制这些概念如数家珍，但遇到线上 latency 突然飙升、内存莫名其妙暴涨、或者客户端连接卡死的时候，依然只能靠"重启大法"或者盲目抄别人的配置。原因很简单：你看到的只是黑盒外面的仪表盘，而从今天起，我们要打开这个黑盒。

Redis 的源码库大约只有 3 万行核心 C 代码，在一众工业级系统中属于极其"苗条"的存在。MySQL 的代码量是百万级，Nginx 是十几万行，而 Redis 的核心部分（src/ 目录去掉第三方依赖和测试）仅仅相当于一本中等厚度的技术书。代码量小并不意味着简单——它在简洁性和性能之间达到了惊人的平衡，是学习 C 工程实践不可多得的教材。

更重要的是，作者 Salvatore Sanfilippo（antirez）在长达十余年的开发历程中，始终秉持一条核心原则：对抗复杂度。他宁可重构一个子系统，也绝不让它变得难以理解。这份"代码洁癖"，正是我们阅读源码时最能受益的地方。

读源码的三个层次

我把读源码的收益分成三个层次，大家可以对照一下自己处在哪个阶段：

  • 第一层：会用——理解命令的时间复杂度 O(1) 还是 O(N)，知道为什么 KEYS * 不能在生产环境用，为什么 bigkey 会阻塞。这个层次的开发者能写出"不犯错"的代码。
  • 第二层：会调优——能通过分析源码判断内存碎片来源、理解 IO 多路复用的调度策略、看懂 RDB/AOF 的写盘时机。这个层次能定位 90% 的生产问题。
  • 第三层：会造轮子——从 Redis 的 zmalloc、ae 事件循环、skiplist、dict 等模块中提取通用设计模式，应用到自己的中间件或存储系统开发中。antirez 的编码风格和架构决策本身就是最好的范本。

源码版本选择与项目总览

本课程以 Redis 7.2.x（2023 年发布的 LTS 稳定版）为主线进行讲解。7.x 系列引入了大量现代化改进，包括多线程部分的成熟化、Function 替代 Lua scripts 的尝试、以及新的 listpack 编码方案。同时我们也会在最后一章前瞻 Redis 8.x 中的变化。建议大家从 github.com/redis/redis 拉取最新的 7.2 分支进行对照阅读。

拿到源码后，你首先看到的是这样的目录结构：

源码阅读路线图

面对 60 多个 C 文件，初学者最容易犯的错误就是"从第一行开始读"——打开 server.c 从 main() 一路看下去，结果在 ae 事件循环和网络初始化中迷失方向。我推荐三种路线，你可以根据自己的背景选择：

路线一：自底向上（推荐有 C 基础的同学）

从最底层的基础数据结构开始，逐步向上层业务逻辑推进。这条路的好处是每一步都有坚实的理解基础，不会出现"跳过底层直接看命令，结果看不懂编码转换"的尴尬。顺序是：内存管理 → 基础数据结构（SDS → 链表 → 字典 → 跳表 → 整数集合） → 对象系统 → 单命令实现 → 事务/Lua → 网络 IO 框架 → 持久化 → 复制与集群。

路线二：自顶向下（适合想快速解决实际问题的同学）

从 main() 启动流程切入，理解 Redis 是如何初始化、进入事件循环、接收网络请求的。然后按照"接到一条命令→解析→查找命令表→调用对应的处理函数→操作数据类型→返回结果"这条链路逐层深入。这种路线阅读效率高，但遇到底层数据结构的细节时需要回头翻阅。

路线三：混合路线（本课程采用）

我们这门课采用的路线是：先花几章把底层数据结构讲透（第3~7章），然后以 redisObject 为跳板延伸到命令实现（第8~12章），再用专门的篇幅详解网络 IO 和事件循环（第13~16章），最后是持久化、复制、集群等分布式主题。这种先主干后分支的路线，能让你在前半段建立清晰的代码全景印象，后半段逐步填充细节。

阅读阶段 涵盖章节 核心文件

Stage 2：对象与命令 第8~12章 object, t_string, t_hash, t_list, t_set, t_zset, stream, geo

编码风格与代码约定

antirez 的编码风格非常统一，掌握这些约定能让你在阅读时节省大量时间：

  • 缩进与格式：使用 4 空格缩进，花括号采用 K&R 风格（左花括号不换行）。函数名使用 snake_case，全局变量通过 server. 前缀访问（例如 server.port、server.db）。
  • 返回值约定：函数成功返回 C_OK（0），失败返回 C_ERR（-1）。这与 POSIX 传统一致。很多函数还通过指针参数返回结果，用返回值表示状态。
  • 内存管理：Redis 封装了 zmalloc() / zfree() / zrealloc() 系列函数，替换了标准 libc 的 malloc/free。这些函数在分配失败时会自动打印日志并退出，因此源码中几乎不需要检查返回值是否为 NULL——这在 C 工程中是一个非常实用的约定。
  • 日志打印：使用 serverLog(level, format, ...) 宏，level 可取 LL_WARNING、LL_NOTICE、LL_VERBOSE 等。antirez 会在关键的逻辑分支中加入 serverLog，这也是我们调试时的重要线索。
  • 错误处理模式：Redis 大量采用 goto cleanup 模式做资源释放。当一个函数中有多个失败返回点时，统一跳转到函数末尾的 cleanup 标签执行释放逻辑。这在 C 语言中是最优雅的错误处理方式之一。

来看一个典型的 Redis 函数片段，体会它的编码风格：

/* 从 sds 中删除 len 个字节 */
sds sdstrim(sds s, const char *cset) {
    char *start, *end, *sp, *ep;
    size_t len;

    sp = start = s;
    ep = end = s + sdslen(s) - 1;
    while (sp <= end && strchr(cset, *sp)) sp++;
    while (ep > start && strchr(cset, *ep)) ep--;
    len = (sp > ep) ? 0 : ((ep - sp) + 1);
    if (s != sp) memmove(s, sp, len);
    s[len] = '\0';
    sdssetlen(s, len);
    return s;
}
代码观察：这个函数不到 15 行，没有检查 NULL、没有复杂的错误分支、两个 while 循环对称处理首尾。这正是 antirez 所说的"代码应该是叙事的，而不是说服的"——读这段代码就像读一个清晰的算法描述。

搭建源码阅读环境

工欲善其事，必先利其器。在开始逐行阅读之前，我们先把开发环境配好。

编译 Redis（带调试符号）

默认的 make 会使用 -O2 优化，这会导致 GDB 单步跟踪时变量被优化掉或者跳转混乱。强烈建议在调试时关闭优化：

# 开发调试版本（-O0，包含 -g 调试符号）
make REDIS_CFLAGS="-g -O0" REDIS_LDFLAGS="-g -O0"

# 运行测试（验证环境正常）
make test

# 生产编译（默认 -O2）
make

GDB 调试配置

创建一个 .gdbinit 文件，加入常用宏定义，可以极大提升调试效率：

# Redis 专用 GDB 宏
set print pretty on
set pagination off

# 打印 redisObject
define pro
    print (robj *)$arg0
end

VSCode + clangd 代码跳转

对于代码浏览，我推荐使用 VSCode 配合 clangd 插件，配合 compile_commands.json 实现精确跳转：

# 安装 bear，用于生成 compile_commands.json
sudo apt install bear

# 重新编译并生成编译数据库
make clean
bear -- make REDIS_CFLAGS="-g -O0"

# 然后在 VSCode 中安装 clangd 插件并禁用默认的 C/C++ IntelliSense
# clangd 会自动读取 compile_commands.json
踩坑实录：曾经有一位同学反馈，线上 Redis 每过几秒就有一次毫秒级的延迟抖动，他用了各种 profiling 工具都查不出原因。后来我让他打开 Redis 的微秒级日志（LATENCY DOCTOR），发现是文件事件循环中的一个 usleep(100) 导致的——为了在空转时降低 CPU 占用，antirez 在 ae.c 中加了这个"善意"的 sleep。那位同学读完 ae.c 中事件循环的精确时序控制代码后，恍然大悟，调整了 hz 参数和 busy-reply 阈值彻底解决了问题。这就是读源码的力量：你知道它为什么这么设计，才知道怎么调。
