第11章 Scaling Laws与算力规划

大家好，我是你们的讲师。假设你有1000万美元的预算，要训练一个顶级大模型。你会训多大的模型？用多少数据？训多久？这些问题不是拍脑袋决定的——有Scaling Laws（缩放定律）来指导你。Scaling Laws可以说是大模型训练的"经济学"，它回答了一个核心问题：当模型大小、数据量、算力都增加时，模型性能会如何变化？

我见过最可惜的一个案例是：某团队拿到一批A100配额后，决定训一个70B的dense模型，数据用了1万亿token。训了三个月，模型确实work了。但如果他们当时看过Scaling Laws，会发现同样的算力预算下，改训一个40B模型但用2万亿token，性能会更好。这就是我们常说的"过度训练（Overtraining）"——模型太大了，数据跟不上。这一章我们就来弄明白：给定手里的资源，怎么最优地分配模型大小和数据量。

11.1 Kaplan vs Chinchilla: 两大Scaling Law

Scaling Laws的研究经历了两大里程碑式的论文。理解它们之间的差异，是理解整个训练规划学科的起点。

Kaplan Scaling Law（2020, OpenAI）。这是第一篇系统性研究大模型缩放行为的论文。Kaplan等人训练了从1M到1.5B参数的模型，总结出三条关键规律：（1）模型性能随参数量呈幂律增长——参数翻倍，loss下降约0.1；（2）模型性能随数据量也呈幂律增长，但增速较慢；（3）算力（计算量）是最终的约束，最佳分配是"更多参数、较少数据"。Kaplan的结论强烈建议：当算力预算增加时，应该把大部分算力投入到增大模型上，而不是增大数据量。

Chinchilla Scaling Law（2022, DeepMind）。DeepMind团队重新审视了Scaling Law，提出了计算最优（Compute-Optimal）训练的概念。他们用更大的模型（最长达16B参数）和更广泛的实验范围，得出了与Kaplan截然不同的结论：模型参数量和训练token数应该同等比例增长。具体来说，当算力预算翻倍时，模型参数量和数据量都应该增加约1.4倍（而不是Kaplan说的"大部分给模型"）。Chinchilla定律的启示是：大多数现有模型都是"过度训练"的——模型太大，数据太少。

维度 Kaplan Law (OpenAI 2020) Chinchilla Law (DeepMind 2022)
核心发现 参数比数据更重要 参数和数据同等重要
最优参数-数据比 N ∝ C^0.73, D ∝ C^0.27 N ∝ C^0.5, D ∝ C^0.5
典型结论 给更多算力，优先扩大模型 算力翻倍，模型和数据各翻1.4倍
对LLaMA的影响 — LLaMA 65B只训1.4T tokens（按Chinchilla应训4T+）
对DeepSeek的影响 — DeepSeek选择用更多token训练较小模型，验证Chinchilla

DeepMind训练的Chinchilla模型（70B参数，1.4万亿token）在多数benchmark上超过了GPT-3（175B参数，3000亿token），证明了"小模型+大数据"路线的有效性。这个发现也间接催生了LLaMA系列——LLaMA 7B在1万亿token上训练，性能超过了许多更大的模型。

关键洞察：Kaplan和Chinchilla的结论差异很大程度上来源于实验设置的不同。Kaplan训练小模型（最大1.5B）做外推预测，而Chinchilla用更大的模型（最大16B）验证了外推的准确性。业界目前更认可Chinchilla，因为它的实验规模更大、结论更贴近实战。但要注意：Scaling Laws是统计规律，不是物理定律——它们描述了"平均行为"，但数据质量、架构设计等不可量化的因素对实际性能有巨大影响。

11.2 计算最优训练

"计算最优"的意思是：给定固定的算力预算（以FLOPs计），选择模型大小N和训练token数D的组合，使得最终模型的loss最低。Chinchilla给出了一个简洁的公式：N_opt ≈ (C/(6×1.4e3))^0.5，其中C是总FLOPs预算。

但这只是理论。在现实中你还需要考虑几个约束：

显存约束。更大的模型需要更多的显存。如果你只有100张H100（每张80GB），就算公式说最优模型是70B，你也放不下——因为训练还需要优化器状态、梯度、激活值。所以实际上N的选择是"公式建议的上限"和"硬件能装下的上限"取最小值。

数据约束。你有多少数据？如果只有几百B token，却去训一个70B模型，数据会被反复重复使用很多个epoch，导致严重的overfitting。Chinchilla的研究表明：在最优训练中，数据不应该被重复使用（即只训一个epoch）。如果你的数据量达不到公式建议的D，那就得缩小模型。

推理成本约束。大模型不仅在训练时烧钱，推理时也烧钱。有时候你选择较小的模型+更多的数据，虽然训练loss可能稍高，但推理成本低得多，综合性价比更高。这就像买车——虽然SUV空间更大，但如果你大多数时候一个人开，小轿车可能更经济。

# 计算最优训练的参数估计（Python伪代码）

def estimate_optimal_params(flops_budget, model_params_mapping=None):
    """
    给定FLOPs预算，估算最优的模型参数量和token数
    基于Chinchilla公式：N_opt ∝ C^0.5, D_opt ∝ C^0.5
    """
    # 参考点：Chinchilla 70B, 1.4T tokens
    # FLOPs ≈ 6 * N * D = 6 * 70e9 * 1.4e12 = 5.88e23

    # 如果你有1e24 FLOPs（约170,000 H100小时）：
    flops = 1e24
    n_opt = 70e9 * (flops / 5.88e23) ** 0.5  # ≈ 91B
    d_opt = 1.4e12 * (flops / 5.88e23) ** 0.5  # ≈ 1.8T tokens
    return n_opt, d_opt

#    或者缩小模型到 N = D_actual * 6 * (最佳参数效率比)

# 2. 推理成本约束 → 1个70B部署成本 vs 2个35B
#    有些场景用多个小模型比单个大模型更划算

11.3 算力预算规划

从理论到实践，算力规划需要回答三个具体问题：需要多少FLOPs？需要多少GPU？需要多少钱？

FLOPs估算。一次标准训练的前向+反向大约需要 6×N×D FLOPs（N=参数量，D=token数）。对于dense模型，这是业界通用的估算公式。例如训练一个7B模型在2万亿token上：6 × 7e9 × 2e12 ≈ 8.4e22 FLOPs。H100的峰值理论算力是1979 TFLOPS（FP16），实际训练利用率通常在35-50%之间。所以实际需要的GPU小时数 = FLOPs / (GPU峰值 × 利用率)。用8.4e22 / (1979e12 × 0.4) ≈ 106,000 GPU小时。1张H100跑约12年，256张H100跑约17天。

Cost分解。训练成本 = GPU成本 + 网络成本 + 存储成本 + 电力成本。GPU成本是大头（约80%），而GPU成本中约40%是采购/租赁费，30%是电力，10%是散热。以H100按需租赁价格约$2-3/GPU-hour计算，上面那个训练约$250,000-300,000。

模型规模 Dense参数量 训练数据量 总FLOPs 256xH100训练时间 估算成本（租赁）
小 7B 2T tokens 8.4e22 ~17天 ~$300K
中 30B 3T tokens 5.4e23 ~110天 ~$1.8M
大 70B 4T tokens 1.68e24 ~340天 ~$5.6M
超大 405B 10T tokens 2.43e25 ~5年（2048张卡约9个月） ~$80M+

实用估算技巧：不要被这个月的表吓到。实际训练中，你可以用小规模外推法：先用1/1000的配置跑一个小实验（比如在一个节点上跑1000步），记录吞吐量（tokens/second/GPU），然后线性外推到全规模。这个方法比纯理论公式更准确，因为它考虑了你的具体集群配置（网络拓扑、存储I/O等的影响）。

11.4 MoE模型的Scaling

MoE架构的出现让Scaling Laws变得更加复杂——因为MoE有"总参数量"和"激活参数量"两个变量，而Scaling Law的公式是针对dense模型的"激活参数量"来定义的。

MoE的等效dense规模。假设一个MoE模型总参数量671B（DeepSeek-V3规模），激活参数量37B（top-2路由）。从计算量的角度看，它相当于一个37B的dense模型。但从知识容量的角度看，671B参数蕴含了更多的知识。所以MoE模型在Scaling Law中的表现是：在同等的激活参数量下，MoE的loss远低于dense模型。

DeepSeek的实践数据。DeepSeek-V3在训练了2.8万亿token后，性能超过了GPT-4（后者据传是1.8T参数的dense MoE）。DeepSeek的关键发现是：MoE模型的Scaling Law曲线比dense模型更陡——意味着每增加1单位算力，MoE模型的收益更大。这很好理解，因为MoE的计算效率更高（只激活部分参数），所以同样的算力下MoE可以"看"更多的数据，从而学得更好。

Expert数量与Scaling。那是不是专家越多越好？理论上，更多的专家意味着更大的知识容量，但同时路由器需要从更多专家中做选择——路由的难度增加了，而且每个专家分到的训练数据更少（因为token被分散到更多专家）。有研究表明专家的最优数量大约在64-128之间，超过之后收益递减。

# MoE vs Dense 的Scaling对比
# LLaMA 405B (dense): FLOPs per token ≈ 405e9 * 2 = 810e9 FLOPs
# DeepSeek-V3 (MoE):  FLOPs per token ≈ 37e9 * 2 = 74e9 FLOPs

# 同样的FLOPs预算下，MoE可以处理约10倍于dense的token量
# 这是MoE在"计算最优"意义上的巨大优势

# 参数: 37B | 训练数据: 2.8T tokens | 训练FLOPs: 6 * 37e9 * 2.8e12 ≈ 6.2e23
# → 不同的是，MoE的知识容量更大，结果更好

11.5 Scaling Law的局限

在结束这一章之前，我想泼一盆冷水——Scaling Laws有很大的局限性，在实战中不要盲信。

数据质量问题。Scaling Laws假设所有token的质量是等价的。但现实中，1000个随机网页文本和1000个经过精校的数学题目，对模型能力的提升差异巨大。DeepSeek和Qwen的实践表明，数据质量比数据量更重要——他们花大量精力做数据清洗、去重、配比，而不是单纯追求token数量。任何不纳入数据质量的Scaling Law都是不完整的。

推理能力不遵循Scaling Law。这是2025-2026年最火的讨论。传统的Scaling Law描述的是"next token prediction loss"和算力的关系。但推理能力（reasoning ability）——比如数学解题、代码编写——并不简单地随着loss下降而提升。DeepSeek-R1的RLVR训练后，loss几乎没有变化，但在推理benchmark上提升了30%+。这说明推理能力是一个和loss不完全相关的独立维度。

后训练阶段的Scaling。Scaling Laws主要研究预训练阶段。但后训练（SFT、RLHF）阶段的Scaling行为完全不同。有研究表明：后训练数据量的Scaling速度远慢于预训练——翻倍SFT数据带来的收益，远小于翻倍预训练数据。而且RLHF的奖励模型有独特的Scaling行为，和语言模型的Scaling不同。

架构变化颠覆Scaling Law。当Transformer架构本身发生变化（比如引入MoE、加入long context技术、使用新的激活函数），基于Transformer的Scaling Law就要重新校订。DeepSeek在发布MoE模型时就指出，他们需要为自己的MoE架构重新拟合Scaling Law参数。

踩坑案例：某团队根据Scaling Law外推，估算一个100B模型在5T token上训练后的loss。他们用1B、3B、10B的小模型做了一系列实验，拟合出Scaling曲线，然后外推到100B。结果真实训练完成后，loss比预测高了0.15。原因是他们的小模型实验用的数据质量很高（精心筛选），但在大规模训练时数据量不够，掺入了低质量数据。数据分布的差异，让整个Scaling Law外推失效了。教训是：Scaling Law的迁移条件是数据分布必须一致，否则外推结果不可靠。
