第26章 基础模型与扩散模型

从2024年到2026年，时序预测领域经历了一次范式级别的转变。想象一下：过去我们做时间序列预测，每接触一个新数据集，都要从头开始做 EDA、调参数、跑交叉验证、选模型。这个过程短则几天，长则数周。但如果你能直接输入一段历史数据，一个已经见过数十亿时间点的预训练模型就能给出预测呢？这就是 Foundation Model（基础模型）给时序预测带来的改变。

与此同时，另一个方向的进展同样令人兴奋——扩散模型（Diffusion Model）开始进入时序领域。它不再只预测一个点估计，而是能生成完整的预测分布，甚至能完成时间序列的填充（imputation）和生成（generation）。本章我们就来聊聊这两个前沿方向，看看它们已经做到了什么，还有哪些局限。

26.1 时序领域的基础模型浪潮

所谓 Foundation Model，是指在大规模、多样化数据上预训练的通用模型，能够 zero-shot 迁移到未见过的任务上。在 NLP 领域我们有 GPT、BERT，在 CV 领域有 ViT、DINO。而在时序领域，2024 年成了"基础模型元年"——Google、AWS、Nixtla、Salesforce 等团队相继发布了各自的大模型。

下面是目前最受关注的几个时序基础模型：

模型 发布方 参数量 预训练数据量 核心亮点
TimesFM Google Research 200M 1000亿时间点 Decoder-only Transformer，patch-based 输入
Chronos / Chronos-2 Amazon (AWS) 20M-710M 约84亿时间点 Tokenization + 语言模型架构，多尺度训练
TimeGPT Nixtla ~100M 约1000亿时间点 首个商业时序基础模型，API 即用
Moirai Salesforce 14M-311M 多领域数据集 统一步长和频率，支持任意频次输入
Lag-Llama ServiceNow & Oxford ~250M 约 72 个公开数据集 引入 lag 特征作为 token，轻量级
核心洞察：基础模型的共性是"大量预训练 + zero-shot 预测"。它们不是为了在某个特定数据集上刷新 SOTA 而设计的，而是希望做到"开箱即用"——你给它一段历史数据，它直接吐出预测，不需要任何训练。

26.2 Tokenization：时序数据如何喂给大模型

基础模型源于 NLP 和 CV，而时间序列和自然语言有一个根本性的差异——时序数据是连续的浮点数，而语言模型依赖离散的 token。这就引出了一个关键问题：如何把连续的时间序列转换为离散 token？

目前主流的策略有三种：

  • Patch embedding（TimesFM 方案）：将连续的序列切成等长的 patch，每个 patch 经过一个小小的 MLP 映射成一个 embedding vector。这类似 ViT 把图像切成 patch 的做法。
  • Binning / 分桶量化（Chronos 方案）：将连续值映射到离散的桶中，例如将数值归一化后在 [-20, 20] 区间内分 4096 个桶，每个值找到对应的桶编号作为 token。Chronos 正是用了这种方法。
  • Scaling + tokenization（混合方案）：先对序列做 scaling（标准化），再用 binning 或 patch 方式处理。

Chronos 的 binning 方法值得展开说。它先计算历史序列的均值和标准差，进行标准化，然后将标准化后的值落入 4096 个等宽 bin 中。这有什么好处？把连续值变成离散 token 后，Chronos 可以直接用标准的语言模型架构（基于 T5 的 encoder-decoder）来训练和推理，无缝复用 NLP 领域的大规模训练基础设施。

# Chronos 风格的 binning tokenization（简化示例）
import numpy as np

def chronos_tokenize(series, num_bins=4096, scale_range=(-20, 20)):
    # 1. 标准化
    mean, std = series.mean(), series.std()
    scaled = (series - mean) / (std + 1e-8)

    # 2. 截断到 range 内
    clipped = np.clip(scaled, *scale_range)

    # 3. 映射到离散 bin
    bin_width = (scale_range[1] - scale_range[0]) / num_bins
    tokens = np.floor((clipped - scale_range[0]) / bin_width).astype(int)
    tokens = np.clip(tokens, 0, num_bins - 1)

    return tokens, (mean, std)

# 使用
series = np.array([10.2, 10.5, 10.1, 9.8, 10.3])
tokens, stats = chronos_tokenize(series)
print("Tokens:", tokens)
# 输出某个 bin index
print("  Bins:", set(tokens))

26.3 Zero-shot 还是 Fine-tune？

基础模型最吸引人的卖点是 zero-shot 能力。但实际使用中，你需要做一个取舍：

  • Zero-shot：直接加载预训练权重，不做任何微调，在你的数据上推理。
  • Fine-tune：基于预训练权重，用你的领域数据做少量微调。

根据 2025-2026 年多个 benchmark 的结果，结论大致是这样的：如果你的数据模式比较"通用"（比如有清晰的趋势和季节性），zero-shot 的 TimesFM 或 Chronos 已经能超越传统的 SARIMA 甚至 LightGBM。但如果你的数据有很强的领域特性（比如电力负荷、金融高频交易），fine-tune 后的模型通常比纯 zero-shot 高出 10-20%。

实践建议：在实际项目中，我会用基础模型先跑一轮 zero-shot 预测，把它作为一个强大的 baseline。然后判断——如果 baseline 已经够好（比如 MASE < 0.8），就不必再花时间微调；如果不够，再考虑 fine-tune 或用传统模型。这比从头开始建模省了至少一半时间。

26.4 扩散模型：从 CV 到时序的跨界

扩散模型（Diffusion Model）你可能在图像生成领域听说过——Stable Diffusion、DALL-E 用的就是扩散模型。但你有没有想过，扩散模型也能用来做时间序列预测？

本质上，扩散模型通过学习从噪声到数据的逐步去噪过程，能够生成高质量的样本。在时序场景下，它的优势在于：

  • 概率预测：不是输出一个单点，而是生成完整的预测分布
  • 多模态输出：能捕捉未来路径的多种可能（这对金融和能源预测非常关键）
  • 缺失值填充：自然支持条件生成，可以填充任意位置的缺失值
模型 年份 核心思路 适用场景
CSDI 2021 基于 DDPM，在时间维度做条件扩散 缺失值填充、概率预测
TimeDiff 2023 引入未来信息的条件机制，预测性能强 单步和多步预测
TS-DiT 2024 DiT 架构（Diffusion Transformer），可扩展 长时间序列生成与预测
D3VAE 2023 扩散+VAE 混合，生成质量高 时序数据生成与增强

TS-DiT（Time Series Diffusion Transformer）是 2024 年的一个重要进展。它把 Transformer 作为扩散模型的主干（backbone），替代传统的 U-Net。实验表明，随着模型和数据的增大，TS-DiT 的性能持续提升——这暗示着时序扩散模型也遵循 scaling law。

26.5 2026 年的现状与思考

到了 2026 年，时序基础模型和扩散模型的发展已经更加成熟。一些共识正在形成：

  • 基础模型是优秀的 zero-shot baseline，但还不是万能药。在 M4、M5 竞赛风格的 benchmark 上，TimesFM 和 Chronos-2 已达到甚至超越了 tuned SARIMA；但在长周期、高噪声的工业数据上，传统模型仍然有竞争力。
  • 扩散模型在概率预测上有独特优势。对于需要预测区间的场景（风险管理、库存安全线），扩散模型的分布估计比分位数回归更加准确和灵活。
  • 融合正在进行。最新的工作已经开始尝试把基础模型的 embedding 和扩散模型的生成能力结合起来——用基础模型做特征提取，用扩散模型生成分布。
  • 推理成本仍然是瓶颈。扩散模型需要多步去噪，推理速度比单步模型慢一个数量级。在实际生产环境中，这仍然是个不可忽视的问题。

26.6 真实故事：TimesFM 与 1000 亿个时间点

讲一个实际发生的故事。2023 年底，Google Research 团队发表 TimesFM 时，最让人震撼的不是模型架构——其实它的设计相当简洁，就是 decoder-only Transformer + patch embedding。真正让人印象深刻的是它的预训练规模：1000 亿个时间点。

这 1000 亿个时间点来自哪里？Google 内部的各种时间序列数据：服务器负载、搜索流量、YouTube 观看量、Google Cloud 客户指标……几乎涵盖了你能想象到的所有类型的数据。团队花了大量的精力做数据清洗和标准化，确保不同来源序列的一致性——这个过程比模型设计本身还要耗时。

结果呢？TimesFM 在 zero-shot 设定下，在很多数据集上已经超过了专门为这些数据训练的模型。这告诉我们一个道理：在深度学习时代，数据的规模和质量往往比模型架构的精巧程度更重要。你不需要特别复杂的模型，你只需要足够多、足够好的数据。

当然，这也引出一个反思：中小团队没有 Google 这样的数据资源，该怎么办？答案可能是社区共建的基础模型（如 Chronos、Moirai），以及针对特定领域的轻量化微调。基础模型的价值在于它把"数据规模壁垒"从个体转移到了社区。

谨慎提醒：目前的基础模型有一个共同的弱点——可解释性差。当模型预测失误时，你很难知道是哪里出了问题。在需要合规性和审计的行业（如金融、医疗），这个问题的严重性不容低估。因此，在可预见的未来，基础模型会越来越多地与传统统计模型互补使用，而不是完全替代。

基础模型和扩散模型代表了时序预测的前沿方向，但目前在生产环境中，从模型到服务还有很长的路要走。下一章我们就来聊聊从 Jupyter Notebook 到生产 API 的全过程——MLOps 和模型部署的最佳实践。
