第十八章：HTTP安全头部

各位同学好。前面三章我们一直在聊HTTPS、TLS和证书——这些都属于"传输层安全"。但HTTP协议本身也提供了一些非常强大的安全机制：安全响应头。这些头部字段可以让浏览器帮你做很多安全检查，有时候只需一行配置就能防御一整类攻击。

我个人把HTTP安全头部称为"性价比最高的安全措施"——配置简单（几句话的事）、防御效果好（浏览器级别的保护），而且大部分头部对用户完全透明。很多安全扫描工具（比如Mozilla Observatory、Qualys SSL Labs）会根据这些头部给网站打分，所以配置好它们也是安全合规的一部分。

18.1 HSTS（Strict-Transport-Security）

HSTS的中文名是"HTTP严格传输安全"。它的作用是告诉浏览器：这个网站只能用HTTPS访问，如果你发现我用HTTP访问，请自动升级到HTTPS，不要给我机会加载不安全的版本。

为什么需要HSTS？你已经知道HTTPS很重要了，但有个问题：用户在浏览器地址栏输入 example.com 时，浏览器默认使用 http:// 协议。服务器要做一次301重定向到 https://example.com。但就在这个重定向发生之前——在第一次HTTP请求发出时——攻击者可以进行SSL剥离攻击（SSL Stripping）：把服务器返回的301重定向篡改掉，让用户停留在HTTP版本上。

HSTS的配置方式：

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

参数说明：

  • max-age：告诉浏览器"记住这个设置多少秒"。31536000秒=1年，是推荐值。
  • includeSubDomains：规则适用于当前域名及其所有子域名
  • preload：告诉浏览器预加载这个域名到HSTS预加载列表

HSTS Preload是一个特殊的机制。即使设置了HSTS，用户在第一次访问时仍然有被SSL剥离攻击的风险——因为浏览器还没有收到过HSTS头。HSTS Preload列表是一份硬编码在浏览器中的域名列表，包含在全球所有主流浏览器中。如果你把域名提交到 https://hstspreload.org 并通过审核，浏览器内置了就永远不需要第一次HTTP请求来"学习"HSTS规则了。

HSTS的"坑"：设置 includeSubDomains 时一定要小心——它会把规则应用到你域名的所有子域名上。如果你有一个子域名跑在HTTP上（比如一些老系统不支持HTTPS），配置了 includeSubDomains 后这个子域名就彻底不可用了。我就吃过这个亏：某次给主站点配HSTS时顺手加了 includeSubDomains，导致公司一个用了十年的HTTP-only子域名系统（供应商的系统，没法改）全部无法访问。最后花了三周迁移那个系统到HTTPS才恢复。所以如果你不确定所有子域名都支持HTTPS，不要加 includeSubDomains。

18.2 CSP（Content-Security-Policy）

CSP是HTTP安全头部中最强大、也最复杂的一个。它的全称是"内容安全策略"，主要作用是防御跨站脚本攻击（XSS）和数据注入攻击。

CSP通过白名单机制控制页面可以加载哪些资源：哪些域名可以执行脚本、哪些域名可以加载图片和样式、是否可以嵌入iframe等。用大白话说就是：告诉浏览器，只有我允许的这些来源的内容，你才执行。

一个典型的CSP配置：

Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://analytics.example.com 'nonce-abc123';
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  img-src 'self' https://images.example.com data:;
  font-src 'self' https://fonts.gstatic.com;
  frame-ancestors 'none';
  report-uri /csp-violation-report

常用指令：

指令作用示例值
default-src所有资源类型的默认白名单'self'
script-src允许加载JavaScript的来源'self' https://cdn.example.com
style-src允许加载CSS的来源'self' 'unsafe-inline'
img-src允许加载图片的来源'self' data: https://*.img.com
connect-src允许发起的fetch/XHR请求的目标'self' https://api.example.com
frame-src允许嵌入的iframe来源'self' https://player.vimeo.com
frame-ancestors允许把页面嵌入到哪些来源的iframe中'none' 或 'self'
report-uri / report-to违规报告的上报地址/csp-violations

Nonce和Hash：当你想执行内联脚本（<script>标签直接写代码）时，CSP默认会阻止它。你可以通过nonce（一次性随机数）或hash来允许特定的内联脚本。

# 服务端生成nonce，在CSP和script标签中同时使用
Content-Security-Policy: script-src 'nonce-r@nd0m'

# HTML中：
<script nonce="r@nd0m">
  // 这段内联脚本会被允许执行
  console.log('hello');
</script>

Report-Only模式：直接启用CSP可能会不小心把合法的资源给拦截了（比如某个第三方CDN的JS库）。CSP提供了"监控模式"：

Content-Security-Policy-Report-Only:
  default-src 'self';
  script-src 'self' https://cdn.example.com;
  report-uri /csp-violation-report

在Report-Only模式下，浏览器仍然会加载被阻止的资源（不会真的拦截），但会把违反策略的情况上报给 report-uri 指定的地址。开发者可以先收集一段时间的报告，确认策略不会误杀合法资源后，再切换到真正的 Content-Security-Policy 头。

CSP实施建议：不要试图一步到位配置完美的CSP——这对现有项目来说基本不可能。正确的做法是：先用 Content-Security-Policy-Report-Only 收集违规报告，逐步调整策略，等违规报告基本清零后，再切换到正式策略。另外，大多数前端框架（React、Vue、Angular）在开发模式下会使用内联脚本（eval、inline style），可能导致CSP违规，记得在构建时关闭这些特性。

18.3 X-Frame-Options

X-Frame-Options 是一个相对简单的头部，用于防御点击劫持（Clickjacking）攻击。点击劫持的原理是：攻击者把目标网站嵌入到一个透明的iframe中，用户在攻击者的页面上点击，实际上点到了目标网站上的按钮（比如"删除账号"或"点赞"）。

# 禁止任何iframe嵌入
X-Frame-Options: DENY

# 只允许同源页面嵌入
X-Frame-Options: SAMEORIGIN

两个可选值：

  • DENY：完全不允许被嵌入到任何iframe中——最严格的设置
  • SAMEORIGIN：只允许在同源页面中嵌入——如果你自己的页面需要iframe引用自己的内容

CSP的frame-ancestors与X-Frame-Options的关系：CSP的 frame-ancestors 指令可以实现同样的功能，而且更灵活——你可以指定具体的允许来源列表。

# CSP方式（更推荐，支持更多选项）
Content-Security-Policy: frame-ancestors 'self' https://trusted-partner.com;

如果你同时设置了 X-Frame-Options 和 CSP frame-ancestors，所有主流浏览器都会优先遵循CSP的设置。但为了兼容旧版浏览器（比如IE），建议两者都设置。

最佳实践：除非你确实需要把你的页面嵌入到其他网站的iframe中（比如做支付跳转后的回显页面），否则设置 X-Frame-Options: DENY 或 frame-ancestors 'none'。如果你确实需要iframe嵌入（比如提供widget服务），那就明确指定允许的来源，不要用通配符。

18.4 X-Content-Type-Options

这个头部虽然名字长，但功能很简单：它只有一个值 nosniff，作用是禁止浏览器进行MIME类型嗅探（MIME Sniffing）。

什么是MIME嗅探？有时服务器返回的资源没有正确设置 Content-Type 头，或者设置的值不准确。过去，浏览器会尝试"猜"这个文件是什么类型——这就是MIME嗅探。听起来很智能，但这是个安全隐患：

  • 攻击者上传一个 .txt 文件（Content-Type设为 text/plain），但文件内容实际上是HTML代码，包含恶意的JavaScript
  • 浏览器访问这个文件时，虽然 Content-Type 说这是文本文件，但浏览器"嗅探"后觉得像HTML，于是把它当HTML解析了——XSS攻击成功

设置 X-Content-Type-Options: nosniff 后，浏览器会严格遵守服务器返回的 Content-Type，不再自作聪明地猜类型。

X-Content-Type-Options: nosniff

这是最简单的安全头部之一——一行配置，零负面影响，强烈建议所有网站都加上。

18.5 Referrer-Policy

Referrer-Policy 控制的是HTTP Referer头（没错，拼写是referrer，但HTTP头的名字是referer——这是当初HTTP规范的一个拼写错误，将错就错保留至今了）在用户从你的网站跳转到其他网站时，你会不会把这个请求的来源信息告诉目标网站。这个信息泄露在隐私保护意识增强的今天越来越受关注。

你可能会想：Referer泄露出什么了？很多：当用户在银行管理后台访问了一个外部链接，Referer可能包含 https://bank.example.com/admin/accounts/12345——完整路径可能暴露了敏感信息。另外，如果在URL参数中包含了session token或密码重置token（虽然不推荐这么做），Referer也会把这些泄露给外部网站。

# 只发送同源请求的来源信息（跨域不发送）
Referrer-Policy: same-origin

# 不发送任何Referer信息
Referrer-Policy: no-referrer

# 只发送来源（简化版URL，不包含路径和查询参数）
Referrer-Policy: strict-origin-when-cross-origin

主要策略值：

从2020年开始，strict-origin-when-cross-origin 已经成为浏览器的默认行为（即使你没有设置这个头部）。但显式设置仍然是好习惯——你可以更精确地控制Referer策略，而且不会被浏览器未来可能的默认值变更影响。

18.6 Permissions-Policy

Permissions-Policy（之前叫 Feature-Policy）是一个比较新的安全头部，用于控制浏览器API的访问权限。它可以限制当前页面以及其子iframe能够使用哪些浏览器功能。

在Permissions-Policy出现之前，如果你在页面中嵌入了一个第三方iframe（比如一个广告或评论区），这个iframe的代码也可以使用浏览器的各种API——比如摄像头、麦克风、地理位置、通知推送等。Permissions-Policy让你可以禁用这些API在iframe中的使用。

# 限制所有页面和iframe不能使用摄像头和麦克风
Permissions-Policy: camera=(), microphone=()

# 只允许当前页面使用地理位置，iframe中不允许
Permissions-Policy: geolocation=(self)

# 允许当前页面和指定域名使用WebUSB
Permissions-Policy: usb=(self "https://trusted-device.example.com")

常见权限名称：

策略值说明典型场景
no-referrer永远不发送Referer头银行、医疗等隐私敏感网站
same-origin仅同源请求携带完整Referer，跨域不携带大多数应用（平衡方案）
strict-origin-when-cross-origin同源发送完整URL，跨域只发送源（无路径），HTTPS→HTTP不发送默认值（推荐！）
origin始终只发送源（无路径）内容聚合站点
unsafe-url始终发送完整的URL（包括路径和参数），即使跨域不推荐——完全不安全
权限说明建议值
camera摄像头访问() 禁止（除非你的网站需要拍照）
microphone麦克风访问() 禁止（除非需要录音）
geolocation地理位置信息(self) 仅允许自己用
notifications桌面通知(self)
paymentPayment Request API(self)
fullscreen全屏API(self)
accelerometer加速度传感器() 禁止（大多数网站不需要）
autoplay自动播放音频/视频(self)

快速配置指南：如果你不想逐一配置每个Permission，一个保守但安全的策略是：Permissions-Policy: camera=(), microphone=(), geolocation=()。这禁用了三个最敏感的隐私API。等你的网站确实需要某个API时，再改为 (self)。另外，Permissions-Policy的语法已经从旧的 Feature-Policy 头发生了变化，请使用新的 Permissions-Policy 头。

18.7 安全头部配置示例

最后，我们把这些安全头部汇总到一个完整的Nginx配置示例中，方便你参考配置：

server {
    listen 443 ssl http2;
    server_name example.com;

    # SSL证书配置
    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # 安全头部
    add_header Strict-Transport-Security    "max-age=31536000; includeSubDomains; preload" always;
    add_header X-Content-Type-Options       "nosniff" always;
    add_header X-Frame-Options              "DENY" always;
    add_header Referrer-Policy              "strict-origin-when-cross-origin" always;

    # 内容安全策略（请根据实际项目调整）
    add_header Content-Security-Policy      "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:;" always;

    # 权限策略
    add_header Permissions-Policy           "camera=(), microphone=(), geolocation=(), payment=(self)" always;

    # ... 其他配置
}

重要提醒：添加安全头部后，一定要用工具测试效果。推荐使用 Mozilla Observatory（https://observatory.mozilla.org）或 securityheaders.com 来扫描你的网站。这些工具会给出评分和具体的改进建议。一个好的目标是达到A级评分——这通常只需要正确配置上述6个头部中的4-5个。

好了，第十八章的内容就到这里。我们从HSTS、CSP、X-Frame-Options、X-Content-Type-Options、Referrer-Policy到Permissions-Policy，把这六个关键的安全头部都讲清楚了。这些头部配置简单但防御效果显著，是Web安全中投入产出比最高的措施之一。希望你在读完这一章后，马上检查自己网站的安全头部配置——你可能会发现，只需要几分钟的配置就能让网站的安全性提升一大截。
