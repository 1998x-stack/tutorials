贪心思想 — 局部最优 → 全局最优

大家好，我是你们的讲师。今天我们来聊聊贪心算法——一种非常"任性"的算法。说它任性，是因为贪心算法从不在"未来"做规划，它只在每一个当下做看起来最好的选择。然后神奇的是，有时候这些局部最优累积起来，恰好就是全局最优。

贪心和动态规划（DP）是面试中容易混淆的一对概念。它们的核心区别在于：贪心只看眼前，DP 回头看所有可能。贪心不会为"后悔"留余地——选了就是选了，不回头。而 DP 会记录所有可能的状态，最后选最好的那个。

用一个日常生活的比喻：你在一座山上，想找到下山的最陡路径。

  • 贪心：每走一步都选最陡的方向。大部分时候能下山，但可能走进一个局部最低的洼地出不来。
  • DP：先在地图上算出所有可能的路径，选出最陡的那条。结果最优，但需要完整的地图信息。
贪心使用前提：一个问题能用贪心解决，意味着该问题具有最优子结构（全局最优包含局部最优）和贪心选择性质（局部最优选择不会影响全局最优的实现）。这两个性质在数学上需要严格证明——但在 LeetCode 面试中，更多是靠经验和直觉来判断。

区间调度

区间调度是贪心算法最经典的应用场景之一。核心思路：每次选择结束时间最早的区间，这样能给后面的区间留出最大空间。

无重叠区间（LeetCode 435）

题目：给定一组区间，移除最少的区间使剩余区间互不重叠。等价于求"最多能保留多少个不重叠的区间"。

def erase_overlap_intervals(intervals):
    if not intervals:
        return 0

    intervals.sort(key=lambda x: x[1])  # 按结束时间排序
    end = intervals[0][1]
    count = 1  # 至少保留一个区间

    for i in range(1, len(intervals)):
        if intervals[i][0] >= end:
            count += 1
            end = intervals[i][1]

    return len(intervals) - count

贪心策略的证明：按结束时间排序后，第一个结束的区间一定属于某个最优解——因为如果最优解的第一个区间不是最早结束的，我们可以用最早结束的那个替换它，结果不会变差。这就是"贪心选择性质"的典型证明思路。

用最少箭引爆气球（LeetCode 452）

这道题是 435 的孪生兄弟——区别在于"重叠区间"的定义：在 435 中，"边界相接"不算重叠（>=），而在 452 中，边界相接也是能一起引爆的（>）。

def find_min_arrow_shots(points):
    if not points:
        return 0

    points.sort(key=lambda x: x[1])
    arrows = 1
    end = points[0][1]

    for i in range(1, len(points)):
        if points[i][0] > end:  # 注意：边界相等不算重叠
            arrows += 1
            end = points[i][1]

    return arrows

对比这两道题，你就能理解 LeetCode 面试的一个真相：很多题目只是改了改条件（等号 / 不等号），解法框架完全一样。所以刷题时不要死记代码，要理解为什么要按结束时间排序、为什么是这个判断条件。

区间调度心法：遇到区间重叠问题，第一反应是按结束时间排序。这是区间贪心的"万能起手式"。少数情况会按开始时间排序（如合并区间 56），但绝大多数区间贪心都以"最早结束"为贪心策略。

跳跃游戏系列

跳跃游戏是贪心算法中"思维方式最精妙"的一类题目。

跳跃游戏（LeetCode 55）

题目：给定一个非负整数数组，每个元素代表你在该位置可以跳跃的最大长度。判断你是否能到达最后一个位置。

最直接的思路是 DP：用 dp[i] 表示能否到达位置 i，转移时看前面有没有能跳到 i 的位置。但这不是最优解。贪心解法更巧妙：维护一个最远可达位置（max_reach），遍历过程中不断更新。

def can_jump(nums):
    max_reach = 0
    for i, step in enumerate(nums):
        if i > max_reach:
            return False      # 当前位置不可达
        max_reach = max(max_reach, i + step)
        if max_reach >= len(nums) - 1:
            return True
    return False

这段代码的时间复杂度是 O(n)，空间复杂度是 O(1)。它不需要知道"怎么跳到的"，只关心"最远能到哪"。贪心的力量就在这里——跳过所有冗余信息，只关注最关键的变量。

跳跃游戏 II（LeetCode 45）

升级版：确保能到达最后位置的情况下，求最少跳跃次数。

def jump(nums):
    n = len(nums)
    if n == 1:
        return 0

    jumps = 0
    cur_end = 0     # 当前这一跳能到达的最远位置
    farthest = 0    # 截至当前位置能到达的最远位置

    for i in range(n - 1):
        farthest = max(farthest, i + nums[i])
        if i == cur_end:
            jumps += 1
            cur_end = farthest
            if cur_end >= n - 1:
                break

    return jumps

这个算法的思路是：在"当前跳"的范围内遍历，同时记录下一跳能到的最远位置。当遍历到当前跳的边界时，跳跃次数加 1，边界更新为下一跳的最远位置。

这里有一个很微妙的点：为什么不是每次到最远位置了才跳？因为每一跳的决策是由"在上一跳范围内选择下一跳的起点"决定的。你不需要在每一跳的起点做"跳还是不跳"的决策——你只需要在这个范围内选一个能跳得最远的点作为下一跳的起点。

其他经典贪心

分发饼干（LeetCode 455）

最直观的贪心入门题：胃口小的孩子优先吃小饼干。对两个数组排序，然后用双指针匹配。

加油站（LeetCode 134）

这道题的贪心思路非常优雅：如果从站点 i 到站点 j 的累积油量不够了，那么 i 到 j 之间的任何站点都不能作为起点。

def can_complete_circuit(gas, cost):
    total, curr = 0, 0
    start = 0
    for i in range(len(gas)):
        diff = gas[i] - cost[i]
        total += diff
        curr += diff
        if curr < 0:
            start = i + 1
            curr = 0
    return start if total >= 0 else -1

这个解法的证明：因为整个数组的 total >= 0 是存在解的充要条件。当我们发现 curr < 0 时，说明 start 到 i 之间的任意站点都不能作为起点——因为无论从哪个站点出发，到 i 的时候油量都不可能大于从 start 出发的油量。

监控二叉树（LeetCode 968）

这道题是 Hard 题目，也是贪心在二叉树上的应用。每个节点有三种状态：无覆盖、有监控、已被覆盖。贪心策略是尽可能让父节点安装监控（这样做可以覆盖更多子节点）。用后序遍历+状态标记来解决。

贪心的陷阱：监控二叉树是一道"看似能用 DP 但贪心更优"的题目。很多人在做这道题时会试图用树形 DP，但代码会非常复杂。而贪心方案——后序遍历 + 三个状态标记——代码简洁且高效。但前提是你能证明"父节点装监控一定不差于子节点装监控"这个贪心策略的正确性。不能证明的话，千万不要在面试中贸然使用贪心，否则被追问证明会很尴尬。

踩坑与总结

坑 1：贪心证明的重要性

贪心算法最大的坑就是：你以为它是对的，但它不对。我曾经在一次模拟面试中遇到这样一道题：给定一组会议时间，选择尽可能多的会议参加。我秒答"按结束时间排序"，面试官点了点头，然后追问："那如果会议有优先级权重呢？"我沉默了——贪心的局部最优策略不再成立，因为"选结束时间最早"不等于"选权重和最大"。这个场景就需要 DP 了。

坑 2："贪心不对"的调试经历

分享一个我自己的经历。有一次刷 LeetCode 55 跳跃游戏，我第一反应是 DFS——遍历所有可能的跳跃路径。对于小规模测试用例，DFS 能过。但 n = 10⁴ 时直接 TLE。我花了半个小时优化剪枝，还是 TLE。最后看了题解——贪心 O(n) 就解决了。这次经历让我深刻意识到：当问题具有"可达性"这类性质时，贪心往往是最高效的工具。如果你发现自己的回溯或 DP 在超时边缘，不妨想想：这题能不能贪心？

坑 3：贪心 vs DP 的选择

在面试中，如果一个问题可以用贪心也可以用 DP，贪心显然是更好的答案——效率更高。但如果你不确定贪心是否正确，建议先给 DP 方案，然后说"这个问题可能可以贪心优化，但我需要再验证一下贪心策略的正确性"。这比给出了一个错误贪心方案要好得多。

问题类型 贪心策略 时间复杂度 是否需要证明
区间调度（435） 按结束时间排序 O(n log n) 较容易
引爆气球（452） 按结束时间排序 O(n log n) 较容易
跳跃游戏 I（55） 维护最远可达位置 O(n) 中等
跳跃游戏 II（45） 分层遍历 + 边界更新 O(n) 中等
加油站（134） 累计油量 + 起点复位 O(n) 较难
分发饼干（455） 小胃口吃小饼干 O(n log n) 容易
监控二叉树（968） 父节点装监控优先 O(n) 较难
本章心法：贪心算法是一个"看山是山，看水是水"的算法。当你真正理解了它的核心——局部最优累积成全局最优——你会发现它无处不在。但也要记住：贪心不是万能的。当你怀疑"这个贪心策略对吗"的时候，尝试构造反例：如果你能找到一个反例，那就不能用贪心；如果你找不到，试着证明它。即使在面试中，这个"尝试证明"的过程本身也会给面试官留下好印象。

贪心算法的"任性"有其限度——它只能在具有贪心选择性质的问题上施展拳脚。当局部最优不等于全局最优时，我们需要更强大的工具：动态规划。下一章我们将进入动态规划的世界，那是算法面试中最重要、也最需要练习的一章。准备好了吗？我们下一章见！
