第16章 机器人仿真

大家好，我是你们的讲师。在上一章我们学会了用ROS 2搭建机器人的软件骨架——节点通信、坐标变换、Launch编排。但有一个灵魂拷问：你的代码真的能驱动真实的机器人吗？你在Jupyter Notebook里计算出来的轨迹，装到真机上会不会直接把关节撞到硬限位？答案是你绝对不想在生产线上才知道答案。这就是仿真存在的意义。

2026年，机器人仿真已经不再是"锦上添花"的工具，而是整个研发流程的第一环。从宇树H2的行走策略在Isaac Sim中训练了相当于200年的虚拟时间后直接部署到真机，到Figure 02的抓取策略在仿真中完成了超过100万次尝试——Sim-to-Real已经成为行业标准。这一章，我将带你了解仿真引擎的选择、URDF/SDF建模、域随机化技术，并通过一个完整的MuJoCo + Python实战让你亲手写出第一个机器人仿真程序。

16.1 为什么仿真先行

让我用一个真实的故事来说明仿真有多重要。2021年，我参与了一个工业协作机器人的项目——目标是让机器人在产线上完成"取螺丝刀→插入螺丝孔"的动作。我们在真机上写了三周的代码，期间烧坏了两台驱动器（因为逆运动学在奇异点附近发了无穷大的速度指令），撞断了一把末端工具（因为默认的碰撞检测没有打开），还有一次差点打到旁边的工程师。如果没有仿真，这些事故中的任何一次都可能导致人员受伤或设备报废。

仿真的四大价值：第一，安全——在仿真中你可以放心地测试极限工况、故障模式和边界条件，任何"爆炸"都只是重来一次。第二，成本——一台工业机器人本体造价在$20K到$100K之间，而仿真引擎全是开源的。第三，并行——你可以在100台云端GPU上同时运行1000个仿真实例，一天内收集相当于真机1000天的训练数据。第四，可重复——仿真中的每一次运行都是完全确定性的，这让回归测试和bug复现变得极其简单。

2026年的Sim-to-Real范式已经进化到"Zero-Shot Transfer"——在仿真中训练好的策略直接部署到真机上，不需要任何微调。核心技术是域随机化（Domain Randomization），我们会在16.4节详细展开。

16.2 三大仿真引擎对比

目前业界主流的机器人仿真引擎有五款，但真正在2026年占据统治地位的是MuJoCo、Isaac Sim和Gazebo。它们各自有明确的定位和适用场景。

MuJoCo（Multi-Joint dynamics with Contact）：由DeepMind（现Google DeepMind）开源的物理引擎，以其极致的速度（可达到实时速度的10,000倍以上）和数值稳定性著称。MuJoCo使用Convex Optimization求解器处理接触和摩擦，几乎不会出现"穿透"或"爆炸"问题。它的定位非常明确——快速物理模拟和强化学习训练。如果你的目标是训练一个行走策略或抓取策略，MuJoCo是不二之选。

NVIDIA Isaac Sim：基于Omniverse平台的GPU加速仿真引擎。Isaac Sim的优势在于逼真的视觉渲染（RTX光线追踪）、丰富的传感器仿真（LiDAR、深度相机、IMU）和完美的RL训练流水线（直接集成Isaac Gym和Isaac ORBIT）。缺点是硬件要求高（至少一块RTX 4090），且配置比MuJoCo复杂得多。2026年NVIDIA发布的Isaac GR00T平台进一步将Isaac Sim定位为人形机器人的首选仿真环境。

Gazebo（与ROS 2集成）：Gazebo Classic（原Gazebo 11）已经退役，现在是Gazebo（原Ignition Gazebo）。Gazebo最大的优势是与ROS 2的无缝集成——你可以直接在仿真中启动ROS 2节点，在仿真中跑SLAM、导航、MoveIt就像在真机上一样。Gazebo的传感器插件生态非常成熟，可以模拟带噪声的激光雷达、失真相机和编码器。它更适合"全系统集成测试"而非"大规模策略训练"。

引擎 物理后端 渲染质量 仿真速度 RL支持 ROS 2集成 最佳场景
MuJoCo 凸优化求解器 基础OpenGL 极快（>10,000x） 原生（dm_env） 通过ros2_control桥接 策略训练、控制算法验证
Isaac Sim PhysX 5.x RTX光线追踪 快（GPU加速） Isaac Gym + ORBIT 官方扩展 视觉策略训练、人形机器人
Gazebo ODE / Bullet / DART OGRE 2.x 中（1x-10x） 第三方集成 原生（最佳） 全系统集成测试、传感器验证

选型建议：做强化学习训练选MuJoCo（快而稳）；做人形机器人和视觉策略选Isaac Sim（渲染真实）；做系统集成测试选Gazebo（ROS 2原生支持）。我在实际项目中往往是三者兼用——MuJoCo训练策略，Isaac Sim验证视觉，Gazebo做最终的系统级测试。

16.3 URDF与SDF建模

在第8章我们已经接触过URDF（Unified Robot Description Format），它是ROS生态中描述机器人运动学和动力学模型的标准XML格式。在仿真中，URDF/SDF模型的质量直接决定了仿真结果的可靠性。

URDF的局限与SDF的改进：URDF有一个人尽皆知的限制——它只能描述树形结构（Tree-Structured）的机器人，不支持闭链（Parallel Kinematic Mechanisms）。对于Stewart平台、Delta机器人这类并联机构，URDF会束手无策。此外，URDF不支持"场景"描述——你没法在URDF中定义桌子上的螺丝刀和旁边的障碍物。SDF（Simulation Description Format）解决了这些问题：它支持任意拓扑结构、支持场景组合（多个模型）、支持传感器插件定义。

传感器插件：在仿真中"模拟"传感器的关键是传感器插件。一个激光雷达插件会从仿真场景中采样点云并添加高斯噪声；一个IMU插件会在每个仿真步计算线加速度和角速度；一个相机插件会在指定位置渲染图像。这些插件让仿真数据尽可能接近真实传感器输出，是实现Sim-to-Real的核心环节。

<!-- SDF示例：一个带激光雷达的机器人模型 -->
<?xml version="1.0" ?>
<sdf version="1.9">
  <model name="my_robot">
    <link name="chassis">
      <inertial>
        <mass>15.0</mass>
        <inertia>
          <ixx>0.3</ixx> <iyy>0.3</iyy> <izz>0.2</izz>
          <ixy>0.0</ixy> <ixz>0.0</ixz> <iyz>0.0</iyz>
        </inertia>
      </inertial>
      <visual name="visual">
        <geometry><box><size>0.5 0.4 0.2</size></box></geometry>
      </visual>
      <collision name="collision">
        <geometry><box><size>0.5 0.4 0.2</size></box></geometry>
      </collision>
    </link>

    <joint name="chassis_to_lidar" type="fixed">
      <parent>chassis</parent>
      <child>lidar_frame</child>
      <pose>0 0 0.15 0 0 0</pose>
    </joint>

    <link name="lidar_frame">
      <visual>
        <geometry><cylinder><radius>0.04</radius><length>0.05</length></cylinder></geometry>
      </visual>
    </link>

    <plugin name="lidar" filename="libLidarPlugin.so">
      <topic>/scan</topic>
      <frame>lidar_frame</frame>
      <range>30.0</range>
      <resolution>0.5</resolution> <!-- 度 -->
      <noise>0.01</noise>
    </plugin>
  </model>
</sdf>

16.4 域随机化

Sim-to-Real的核心挑战是什么？是"仿真与现实之间的鸿沟"（Reality Gap）。在仿真中训练的策略直接部署到真机上，往往表现很差——因为仿真中的物理参数是精确的、光照是完美的、传感器是没有噪声的。域随机化（Domain Randomization, DR）是目前最有效的弥合手段。

域随机化的原理：在每次重置仿真环境时，随机化环境中的多个参数——包括但不限于机器人的质量、关节摩擦系数、地面摩擦系数、相机曝光时间、光照角度、纹理颜色、传感器噪声水平。这样一来，策略在训练中看到的是"无数种不同的环境"，而真机只是这无数种环境中的一种。策略被迫学会适应各种参数变化，从而自然地泛化到真机上。

三大随机化维度：

  • 视觉域随机化（Visual DR）：随机化纹理（随机颜色、随机图案）、光照（方向、强度、色温）、相机噪声（散粒噪声、镜头畸变）、背景（随机图片或纯色）。典型的范围：光照强度0.5-1.5倍、纹理RGB随机偏移±50。这对于视觉策略尤其重要——避免模型"死记硬背"训练场景。
  • 动力学域随机化（Dynamics DR）：随机化质量（±20%）、惯性张量（±20%）、关节库仑摩擦（0.1-0.9 Nm）、粘性摩擦（0.01-0.1 Nm·s/rad）、地面摩擦系数（0.2-1.0）。这使得策略对机械加工公差和磨损造成的参数漂移具有鲁棒性。
  • 观测域随机化（Observation DR）：在状态观测中加入高斯噪声或延时。比如IMU读数增加±0.05m/s²的噪声、关节编码器读数增加±0.01°的噪声。这防止策略对"完美状态"产生过拟合。

效果数据：根据2025年的一项系统综述，应用域随机化后策略从仿真迁移到真机的成功率从平均27%提升到了68%。在最优配置下（全维度随机化 + 自适应DR调度），甚至实现了92%的零样本迁移成功率。但有一个关键点——过度随机化会损害训练效果。例如，如果把关节摩擦随机化到0-10Nm的范围，策略将学到"不管怎么用力关节都不动"，在真机上反而表现很差。好的DR设计是一门"合理设置随机化范围"的艺术。

16.5 实战：MuJoCo + Python仿真

让我们动手做一个实际的仿真——在MuJoCo中创建一个简单的二轴机械臂，用Python控制它从一个位置运动到另一个位置。MuJoCo的Python API（mujoco和dm_control）非常简洁，三十行代码就能跑起来一个仿真。

# ============================================
# MuJoCo 二轴机械臂仿真 — 位置控制
# ============================================
import mujoco
import numpy as np

# 机械臂XML模型：两个旋转关节串联
XML = """
<mujoco model="simple_arm">
  <option timestep="0.002" />  <!-- 500Hz仿真步长 -->

  <worldbody>
    <light name="top" pos="0 0 2"/>
    <body name="base" pos="0 0 0">
      <geom type="box" size="0.1 0.1 0.05" rgba="0.5 0.5 0.5 1"/>
      <body name="link1" pos="0 0 0.05">
        <joint name="joint1" type="hinge" axis="0 0 1"/>
        <geom type="cylinder" size="0.03 0.2" rgba="0.2 0.6 0.8 1"/>
        <body name="link2" pos="0 0 0.2">
          <joint name="joint2" type="hinge" axis="0 1 0"/>
          <geom type="cylinder" size="0.025 0.18" rgba="0.8 0.3 0.2 1"/>
          <body name="end_effector" pos="0 0 0.18">
            <geom type="sphere" size="0.02" rgba="0.1 0.8 0.1 1"/>
          </body>
        </body>
      </body>
    </body>
  </worldbody>
</mujoco>
"""

# 创建模型和数据
model = mujoco.MjModel.from_xml_string(XML)
data = mujoco.MjData(model)

# 设置目标关节角度（弧度）
target_position = np.array([0.5, -0.3])

# 运行仿真
for step in range(5000):
    # 简单的P控制器：τ = Kp * (q_desired - q_current)
    Kp = 50.0
    error = target_position - data.qpos[:2]
    data.ctrl[:2] = Kp * error  # 力矩控制

    mujoco.mj_step(model, data)

    if step % 500 == 0:
        print(f"Step {step}: q = {data.qpos[:2]}, τ = {data.ctrl[:2]}")

print(f"最终关节位置: {data.qpos[:2]}")
print(f"目标位置: {target_position}")
print(f"末端执行器位置: {data.geom_xpos[-1]}")

这段代码首先定义了一个两自由度机械臂的MuJoCo XML模型——包含两个旋转关节（joint1绕Z轴、joint2绕Y轴）和一个绿色的末端小球。主循环中我们实现了一个最简单的P控制器，根据关节位置误差计算控制力矩。注意mujoco.mj_step()是MuJoCo的单步推进函数，每次调用推进一个timestep（这里设置为0.002秒，即500Hz）。

踩坑实录：我曾经花了整整一周的时间，试图将一个在MuJoCo中训练好的抓取策略直接部署到真机上。结果机器人一启动就开始剧烈振荡——我反复检查了代码逻辑、神经网络权重，都没有问题。最后用高速摄像机一帧一帧地对比仿真和真机：原来我在MuJoCo中设置的关节摩擦系数是0，而真机上的谐波减速器自带的摩擦力矩高达0.8Nm。这个摩擦力的差异就是"Reality Gap"的典型例子。后来我在训练中加入了摩擦力的域随机化（0.01到1.0Nm），策略在真机上立刻稳定了。教训：永远不要信任仿真中的默认物理参数，每个参数都应该与真机的实测值对标。
