第26章 SQLite WASM 与 JS 绑定

大家好，我是你们的讲师。今天我们要聊一个非常有意思的话题——SQLite 在浏览器里跑起来了。

2019 年，WebAssembly（WASM）生态逐渐成熟。SQLite 社区中出现了不少第三方项目尝试把 SQLite 编译成 WASM——比如 sql.js——但官方一直没有正式出手。直到 2022 年，SQLite 团队宣布正式支持 WASM 构建，整个生态才真正进入了"官方背书"的时代。到了 2026 年的 3.54.0 版本，SQLite 的 WASM/JS 绑定已经成为一个成熟的、功能完备的发布组件，与传统的 C API 一样经过了 TH3 测试的严格验证。

这一章，我们就从源码入手，深入理解 SQLite 的 WASM 架构：从 Emscripten 编译流程、官方 JS API 设计、VFS 在浏览器中的实现，到真实世界的使用场景。你将在这一章结束时获得在浏览器中完整运行 SQLite 的能力。

💡 为什么要在浏览器中运行 SQLite？想象一下这些场景：一个 PWA 应用需要在客户端存储和分析上万条数据，一个数据可视化工具需要在前端执行 SQL 查询进行过滤聚合，一个在线教育平台让学生在浏览器里直接练习 SQL 而不需要安装任何东西——所有这些场景，SQLite WASM 都是最佳答案。

WASM 编译流程

把 SQLite 编译成 WASM，本质上就是把 C 源代码用 Emscripten 编译器（一个将 LLVM 字节码翻译为 WASM 的工具链）重新编译一次。但 SQLite 的特殊之处在于它的 amalgamation 构建模式——这使得 WASM 编译也采用了"一个源文件 + 一个头文件"的极简路径。

官方 WASM 构建入口在 ext/wasm/GNUmakefile 中。核心编译命令大致如下：

# 简化后的 WASM 编译命令
emcc -Os -s WASM=1 \
  -s EXPORTED_RUNTIME_METHODS='["ccall","cwrap","FS","allocateUTF8"]' \
  -s EXPORTED_FUNCTIONS='["_sqlite3_open","_sqlite3_close","_sqlite3_prepare_v2", \
                          "_sqlite3_step","_sqlite3_finalize","_sqlite3_column_text", \
                          "_sqlite3_column_int","_sqlite3_errmsg","_sqlite3_exec"]' \
  -s MODULARIZE=1 -s EXPORT_ES6=1 \
  -I. sqlite3.c -o sqlite3.js

几个关键编译选项的含义：

  • -s WASM=1 — 启用 WASM 输出，而不是 asm.js 回退。
  • EXPORTED_FUNCTIONS — 显式导出 C 函数符号，让 JS 侧可以调用。注意 C 函数名需要加下划线前缀。
  • MODULARIZE=1 — 输出为模块化 JS 文件，避免污染全局命名空间。
  • EXPORT_ES6=1 — 输出 ES6 module 格式，方便在现代前端项目中使用 import 加载。

编译产物包括两个文件：sqlite3.js（胶水 JS，负责加载实例化和导出 API）和 sqlite3.wasm（真正的 WASM 二进制）。在整个编译过程中，SQLite 团队对 amagalmation 应用了一些条件编译宏——例如禁用平台相关的系统调用、启用内存文件系统支持等——来确保 WASM 环境下的兼容性。

官方 WASM 架构

理解官方 WASM 架构的关键在于一个核心设计决策：SQLite 的 WASM 绑定不是简单地把 C API 一对一映射到 JS，而是构建了一个更高层的、符合 JS 惯例的封装层。这个封装层的源码集中在 ext/wasm/ 目录下。

源文件职责JS API 示例
wasm_glue.cC 侧的胶水层，暴露关键内部函数给 JS底层内存操作、VFS 回调注册
sqlite3-api.js主要 JS API 封装sqlite3.open(), db.exec()
sqlite3-vfs.jsWASM VFS 实现OPFS VFS 注册、内存 VFS 切换
sqlite3-worker1.jsWorker 模式支持在 Web Worker 中运行 SQLite
sqlite3-opfs-async-proxy.jsOPFS 异步代理OPFS 文件操作的异步桥接

JS API 的设计遵循了几个核心原则：

  • Promise 友好 — 所有可能涉及异步操作的方法（如 OPFS 文件访问）都返回 Promise，与现代 async/await 编程模型无缝对接。
  • 资源自动管理 — JS 中不需要手动调用 sqlite3_finalize()。当 Statement 对象被垃圾回收时，底层的 prepared statement 会被自动释放。
  • 错误转异常 — SQLite 错误码在 JS 层被转换为可以 throw 的异常对象，包含 code 属性和人类可读的 message。
对比传统 C API 与 JS API： 在 C 中你写 sqlite3_prepare_v2(db, sql, -1, &stmt, NULL) 然后循环 sqlite3_step()；在 JS 中你写 const stmt = db.prepare(sql); while(stmt.step()) {...}。底层驱动逻辑完全一致，但上层接口经过了精心设计的"JavaScript 化"改造。

特别值得一提的是 JS API 中的 db.exec() 方法。它接受一个 SQL 字符串以及可选的回调函数，自动完成 prepare→step→finalize 的完整生命周期，并将结果以 JS 数组对象的形式返回。这是日常使用中最常用的 API：

import sqlite3Init from './sqlite3.js';

const sqlite3 = await sqlite3Init();
const db = new sqlite3.oo1.DB(':memory:');

// exec() 自动处理 prepare/step/finalize
const result = db.exec("SELECT id, name FROM users WHERE age > ?", {
  bind: [18],
  returnValue: 'resultRows',
  rowMode: 'object'
});
console.log(result); // [{id:1, name:"张三"}, {id:2, name:"李四"}]

VFS 在 WASM 中的实现

一个数据库引擎最关键的底层基础设施就是 VFS。在 WASM 环境中，SQLite 无法直接访问操作系统的文件系统——浏览器沙箱把这一切都挡住了。那么问题来了：SQLite 的数据库文件存在哪里？

答案是：存在浏览器的存储机制中。SQLite WASM 支持三种 VFS 后端：

VFS 类型存储位置持久性适用场景
memdb（默认）WASM 线性内存会话级（页面刷新即丢失）临时计算、数据转换
opfsOrigin Private File System页面级持久（可跨会话）PWA、客户端本地数据库
opfs-wl（3.53.0 新增）OPFS + Web Locks页面级持久 + 跨 Tab 安全多 Tab 共享数据库

memdb VFS 是最简单的实现。它基于 Emscripten 内置的内存文件系统（MEMFS），所有数据都存储在 WASM 的线性内存中。不需要任何持久化存储权限，开箱即用。缺点显而易见：页面刷新后数据就消失了。

opfs VFS 使用了 Chrome 83 引入的 Origin Private File System（OPFS）API。OPFS 为每个源（origin）提供了一个隔离的、高性能的文件系统沙箱。SQLite 的 OPFS VFS 实现（在 sqlite3-vfs.js 中）将 SQLite 的 xRead、xWrite 等 VFS 调用映射到 OPFS 的 FileSystemFileHandle 读写操作。

opfs-wl VFS 是 3.53.0 版本的重大创新。它在 OPFS 的基础上引入了 Web Locks API 来实现跨 Tab 的并发控制。简单来说，当多个浏览器 Tab 页打开同一个 SQLite 数据库文件时，opfs-wl 通过 Web Locks 来协调锁的获取，从而避免了多 Tab 并发写入导致的数据损坏。

⚠️ OPFS 的兼容性问题。 虽然 Chrome、Edge、Safari 和 Firefox 在 2026 年都已经支持 OPFS，但不同浏览器的实现细节存在差异。特别是 iOS Safari 上的 OPFS 性能远不如桌面版 Chrome。在生产环境中，建议加上 opfs 能力检测，在不支持时优雅降级到 memdb。

浏览器端 SQLite

在实际项目中，SQLite WASM 的应用场景非常广泛。我这里列出几个典型场景，并附上对应的架构决策：

  • 客户端离线缓存 — 在 PWA 中，使用 OPFS VFS 作为持久化存储。网络请求返回的数据先写入 SQLite，下次离线时从本地直接读取。相比 IndexedDB，SQLite 的优势在于支持复杂的查询（JOIN、GROUP BY、子查询），数据分析和聚合效率远高于游标遍历。
  • SQL 在线教学平台 — 完全在浏览器内存中创建临时数据库，学生执行 SQL 语句后立刻看到结果。不需要后端服务器，不需要搭建数据库环境，零成本分发。著名的 SQLZoo、SQLBolt 等平台的后继者已经开始使用 SQLite WASM 方案。
  • 数据 ETL 预处理 — 在浏览器中读取 CSV/JSON 文件，插入 SQLite 内存数据库进行清洗和转换，然后导出处理后的数据。所有计算在本地完成，不需要上传到服务器，既保护隐私又节省带宽。

这里有一个真实案例。2025 年，一个开源的数据可视化工具 VizHub 将其后端从 PostgreSQL 迁移到了 SQLite WASM。用户在浏览器中上传 CSV 文件后，所有数据查询、聚合、过滤全部在本地 SQLite 中完成，只有在需要保存项目时才会将 SQLite 数据库文件上传到云存储。这个架构调整使服务器的并发负载降低了 85%，用户交互延迟从平均 300ms 降到了 10ms 以内。

💡 浏览器 SQLite 的性能非常可观。在 Chrome 中，一个内存数据库可以轻松处理百万行级别的数据查询（简单的 SELECT 在毫秒级）。OPFS 场景稍有折扣，但日常的 CRUD 操作也完全在用户可以接受的范围。SQLite WASM 不是浏览器数据库的"玩具方案"——对绝大多数应用场景来说，它比 IndexedDB 更快、更强。

Node.js 中的 SQLite

既然 WASM 方案可以在浏览器中运行，那它同样可以在 Node.js 环境中运行。不过在 Node.js 世界，SQLite 的生态要更加丰富——这里有一个选择取舍的问题。

方案原理性能API 风格适用场景
better-sqlite3Node.js C++ 原生插件★★★★★（原生最优）同步 API生产服务、CLI 工具
官方 WASM（Node）WASM 在 V8 中运行★★★★☆（~95% 原生性能）Promise 异步跨平台打包、Electron 应用
sql.js第三方 WASM 封装★★★★☆同步 API浏览器兼容、遗留项目

better-sqlite3 是目前 Node.js 生态中最流行的 SQLite 绑定，它通过 Node.js 的原生插件机制直接链接到系统的 SQLite 库。性能方面，它比 WASM 方案快大约 5-10%（主要体现在函数调用开销的减少上）。它的 API 是同步的——这听起来违背了 Node.js 的"异步优先"原则，但作者认为对于本地数据库操作来说，同步 API 的简洁性远超微乎其微的异步收益。

官方 WASM 方案的优势则在于：不需要原生编译。你不需要系统上有 C++ 编译器，不需要 node-gyp，不会遇到平台兼容性问题。一个 npm install @sqlite.org/sqlite-wasm 就可以在任何平台上运行。对于 Electron 应用来说，这尤其重要——你再也不用为每个平台编译原生插件了。

结论： 如果你在写一个需要部署到生产环境的后端服务，better-sqlite3 仍然是最佳选择（性能、稳定性、生态成熟度）。如果你在写 Electron/跨平台 CLI 工具，或者你的部署流程极度厌恶原生编译，官方 WASM 方案是完全够用的替代品。

源码中的 WASM 相关

说了这么多 API 层面的东西，让我们回到源码，看看 ext/wasm/ 目录下的实际文件结构。这是 3.54.0 版本中 WASM 子系统的全貌：

ext/wasm/
├── GNUmakefile              ← WASM 构建主 Makefile
├── Makefile                 ← 兼容性 Makefile（调用 GNUmakefile）
├── wasm_glue.c              ← C 侧胶水层代码
├── wasm_glue.h              ← C 侧胶水头文件
├── sqlite3-api.js           ← 主 JS API 封装（~3500 行）
├── sqlite3-vfs.js           ← VFS 实现（内存/OPFS）
├── sqlite3-worker1.js       ← Worker 模式封装
├── sqlite3-opfs-async-proxy.js ← OPFS 异步代理
├── sqlite3-api-proto.js     ← API 原型和类型定义
├── api-config.js            ← 运行时配置
├── test/                    ← WASM 相关的测试
│   ├── test_main.js         ← 核心功能测试
│   ├── test_vfs.js          ← VFS 测试
│   ├── test_opfs.js         ← OPFS VFS 专项测试
│   └── …
└── demo/                    ← 示例应用
    ├── demo-1.html          ← 基础演示
    ├── demo-2.html          ← OPFS 持久化演示
    └── …

几个关键文件的内幕：

  • wasm_glue.c — 这是 C 侧对 JS 暴露的"桥接层"。它定义了一些在标准 amalgamation 中不暴露的内部函数，例如用于内存分配追踪的函数、VFS 注册的回调桩等。它还包含了自定义的 SQLite 编译时配置（比如启用 SQLITE_ENABLE_API_ARMOR 来增强参数校验）。
  • sqlite3-api.js — 这是 WASM 包的核心文件，约 3500 行。它定义了 sqlite3 命名空间下的所有公共 API，包括面向对象的封装（oo1 子命名空间）。阅读这个文件可以深入了解 JS API 的设计哲学。
  • sqlite3-vfs.js — VFS 实现的文件。如果你关心"数据库文件到底存在了哪里"，读这个文件就对了。它实现了 registerVFS() 函数，允许在运行时动态注册和切换 VFS。

实战: 在浏览器中运行 SQLite

理论说得再多，不如动手跑一次。让我们实战构建一个在浏览器中运行的 SQLite 示例。

首先从官方 npm 包安装：

npm install @sqlite.org/sqlite-wasm

然后在你的 HTML 页面中：

<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <title>SQLite WASM 示例</title>
</head>
<body>
  <h1>浏览器 SQLite 控制台</h1>
  <textarea id="sqlInput" rows="4" cols="60">
    SELECT 'Hello, SQLite WASM!' AS message;
  </textarea>
  <button id="runBtn">执行</button>
  <pre id="output"></pre>

  <script type="module">
    import sqlite3Init from '@sqlite.org/sqlite-wasm';

    const sqlite3 = await sqlite3Init();
    const db = new sqlite3.oo1.DB(':memory:');

    document.getElementById('runBtn').onclick = () => {
      const sql = document.getElementById('sqlInput').value;
      try {
        const result = db.exec(sql, { returnValue: 'resultRows' });
        document.getElementById('output').textContent =
          JSON.stringify(result, null, 2);
      } catch (e) {
        document.getElementById('output').textContent = `错误: ${e.message}`;
      }
    };
  </script>
</body>
</html>

这里面的核心流程：

  1. sqlite3Init() 异步加载并初始化 WASM 模块，返回 sqlite3 命名空间对象。
  2. new sqlite3.oo1.DB(...) 打开一个数据库连接。这里使用 :memory: 表示内存数据库。
  3. db.exec(sql, options) 执行 SQL 并返回结果。通过 returnValue: 'resultRows' 让结果以数组形式返回。

如果要使用持久化的 OPFS VFS，只需要把 :memory: 替换为你的数据库文件名，并在初始化时指定 VFS：

// 尝试使用 OPFS VFS，回退到内存数据库
let db;
try {
  const opfs = sqlite3.capi.sqlite3_vfs_find("opfs");
  if (opfs) {
    db = new sqlite3.oo1.DB("/persistent.db", { vfs: "opfs" });
    console.log("使用 OPFS 持久化存储");
  }
} catch {
  db = new sqlite3.oo1.DB(":memory:");
  console.log("OPFS 不可用，使用内存数据库");
}

就这样，SQLite 在你的浏览器里跑起来了。不需要服务器，不需要安装，不需要配置——一个 HTML 文件就够了。这就是 WASM 的魔力，也是 SQLite 团队在过去四年里持续投入 WASM 生态的意义所在。

一个小故事： 2024 年，一位前端开发者在一场技术大会上展示了他在浏览器中运行 SQLite 做实时数据分析的 demo。会后一位后端开发者找到他说："你这个东西方便是方便，但我还是觉得不踏实——浏览器里跑数据库，听起来就像在纸飞机上装引擎。"前端开发者笑着打开笔记本离线模式，跑了一遍完全一样的 demo，然后说："现在知道引擎装在哪里了吗？"这个故事很好地说明了一个事实：WASM 打破了"前端只能做 UI、后端才能做数据"的传统分工。SQLite WASM 把数据库引擎的能力带到了每一个浏览器 Tab 中。
