第15章 混合精度训练

大家好，我是你们的讲师。这一章我们来聊一个训练加速的"免费午餐"——混合精度训练（Mixed Precision Training）。

为什么说它是"免费午餐"？因为它基本不需要改模型结构、不需要调超参数，只需要在训练代码里加几行代码，就能获得 1.5x 到 3x 的加速，同时显存占用还能降低近一半。这种好事在深度学习的世界里可不多见。

混合精度训练的核心思想很简单：不是所有计算都需要 FP32 的精度。梯度、激活值、中间特征——这些张量中的大部分信息对精度并不敏感，用 FP16 或 BF16 存就够了。只有少数关键操作（比如损失计算、权重更新）才需要保持 FP32 的精度。我们把高精度和低精度混合起来用，这就是"混合精度"这个名字的由来。

15.1 FP16 与 BF16：两种低精度格式

在深入代码之前，我们需要先搞清楚两种主要的低精度浮点格式：FP16（半精度浮点数）和 BF16（Brain Floating Point）。

FP16 是 IEEE 754 标准中定义的 16 位浮点数格式，有 5 位指数位和 10 位尾数位。BF16 则是由 Google 的 Brain 团队提出的格式，它牺牲了尾数位，保留了和 FP32 一样的 8 位指数位。这带来的直接影响是：BF16 的动态范围与 FP32 相同，但精度更低；FP16 的精度更高，但动态范围窄得多。

下面这张表可以让你一目了然地看到它们的区别：

格式 指数位 尾数位 动态范围 精度 适用硬件
FP32 8 23 ~3.4e38 高 所有 GPU
FP16 5 10 ~65504 较高 V100, A100, H100, RTX 系列
BF16 8 7 ~3.4e38 较低 A100+, H100, RTX 4090+

注意看动态范围这一列：FP16 最大只能表示到 65504，而 BF16 和 FP32 一样可以表示到 3.4e38。这个差异在实际训练中非常关键。

FP16 的"精度陷阱"：FP16 的动态范围很窄，这意味着梯度值很容易下溢到零（underflow）。如果你的模型激活值或梯度的绝对值小于 2-24（约 6e-8），FP16 就直接把它截断成零了。这就是为什么 FP16 训练需要配合 loss scaling 的原因。

BF16 的出现解决了这个问题。它的指数位跟 FP32 一样多，所以几乎不会出现下溢。代价是尾数位只有 7 位，精度比 FP16 还低一些。但在深度学习训练中——尤其是大模型训练——动态范围比精度重要得多。模型权重通常分布在零附近一个较宽的范围内，尾数精度不够导致的误差在反向传播中会被 batch 平均掉，但梯度下溢是直接丢失信息，无法恢复。

所以一个很好的经验法则是：如果硬件支持 BF16（Ampere 架构及以上），优先用 BF16；如果只能用 FP16（V100 或更老的卡），必须配合 Gradient Scaler。

15.2 torch.amp.autocast：一行代码开启混合精度

PyTorch 从 1.6 版本开始就在 torch.amp 模块中提供了自动混合精度（Automatic Mixed Precision, AMP）的支持。所谓"自动"，是指 PyTorch 会自动判断每个操作应该用 FP16/BF16 还是 FP32 来执行。比如矩阵乘法（GEMM）和卷积这类计算密集型的操作会用低精度加速，而损失计算、softmax 等精度敏感的操作会自动回退到 FP32。

使用方法非常简单——只需要用 autocast 上下文管理器包裹前向传播：

import torch
from torch.amp import autocast

model = MyModel().cuda()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for batch in dataloader:
    inputs, labels = batch
    inputs, labels = inputs.cuda(), labels.cuda()

    optimizer.zero_grad()

    # autocast 上下文：自动选择精度
    with autocast(device_type="cuda"):
        outputs = model(inputs)
        loss = criterion(outputs, labels)

    loss.backward()
    optimizer.step()

就这么多。在 autocast 的上下文中，torch.nn.Linear 的矩阵乘法会以 FP16/BF16 执行，而 torch.nn.CrossEntropyLoss 的 log-softmax 计算会以 FP32 执行。你完全不需要手动控制哪个操作用什么精度。

device_type 参数：从 PyTorch 2.0 开始，autocast 需要显式传入 device_type 参数。如果使用 CUDA 设备，传 "cuda"；如果使用 Apple Silicon 的 MPS 后端，传 "mps"；如果使用 CPU，传 "cpu"。在 PyTorch 2.12 中，device_type 是必填参数。

15.3 GradientScaler：FP16 训练的守护神

如果你使用的是 FP16（比如在 V100 上训练），光有 autocast 是不够的。还记得我们前面说的 FP16 动态范围窄的问题吗？在训练过程中，梯度值的分布范围很大，很多梯度可能会小于 FP16 的最小可表示正数（约 6e-8），从而下溢到零。

解决方案是 Gradient Scaler：在反向传播之前，先把损失值乘以一个缩放因子（比如 216 = 65536），这样计算出的梯度也等比例放大了。等梯度传回到权重之前，再把它缩小回去。这样原本会下溢的小梯度，放大之后就能被 FP16 精确表示了。

这里有监控里听到的一个经典踩坑故事。2020 年有一位工程师在训练一个 BERT 模型时，发现 loss 完全不下降。他检查了模型结构、学习率、数据预处理，折腾了两天都没找出问题。最后他在反向传播后打印了一下梯度——发现所有层的梯度都是 tensor(0.)。模型训练了 48 小时，梯度从第一步开始就全部下溢到了零，模型从头到尾就没学过任何东西。原因就是他没有使用 GradientScaler。

正确的用法是 GradScaler + autocast 配合使用：

from torch.amp import autocast, GradScaler

model = MyModel().cuda()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
scaler = GradScaler()  # 默认 init_scale=2**16

for batch in dataloader:
    inputs, labels = batch
    inputs, labels = inputs.cuda(), labels.cuda()

    optimizer.zero_grad()

    with autocast(device_type="cuda"):
        outputs = model(inputs)
        loss = criterion(outputs, labels)

    # 1. scaler.scale(loss) 放大损失
    # 2. loss.backward() 计算放大后的梯度
    scaler.scale(loss).backward()

    # 3. scaler.step(optimizer) 内部先 unscale 梯度
    #    如果梯度未溢出，则执行 optimizer.step()
    #    如果梯度溢出，跳过这一步
    scaler.step(optimizer)

    # 4. 更新缩放因子
    scaler.update()

GradScaler 内部维护了一个动态的缩放因子。如果连续多次没有发生梯度溢出，他会逐渐增大缩放因子；如果检测到溢出，他会减小缩放因子。整个过程自动进行，你只需要按照上面的模式写代码就行。

GradientScaler 的工作流程总结：
  1. scaler.scale(loss) — 对损失乘以缩放因子（默认 216）
  2. loss.backward() — 计算放大后的梯度
  3. scaler.step(optimizer) — 将梯度 unscale 回来，检查是否溢出；如果未溢出则更新权重
  4. scaler.update() — 根据本轮是否发生溢出动态调整缩放因子

15.4 BF16：更简单的选择

如果你使用的是 Ampere 架构（A100）或更新的 GPU（H100、RTX 4090 等），那么你不需要 GradScaler。直接使用 BF16 格式配合 autocast 就足够了：

from torch.amp import autocast

model = MyModel().cuda()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for batch in dataloader:
    inputs, labels = batch
    inputs, labels = inputs.cuda(), labels.cuda()
    optimizer.zero_grad()

    with autocast(device_type="cuda", dtype=torch.bfloat16):
        outputs = model(inputs)
        loss = criterion(outputs, labels)

    loss.backward()
    optimizer.step()

注意到区别了吗？没有 GradScaler，没有 unscale，没有 update——代码跟纯 FP32 训练几乎一模一样，只是因为加了一个 autocast 就获得了加速。这是因为 BF16 的动态范围和 FP32 一致，梯度不需要缩放过。

当然，如果你需要在同一个脚本里同时支持 FP16（V100）和 BF16（A100），可以这样写：

dtype = torch.bfloat16 if torch.cuda.get_device_capability() >= (8, 0) else torch.float16

with autocast(device_type="cuda", dtype=dtype):
    outputs = model(inputs)
    loss = criterion(outputs, labels)

torch.cuda.get_device_capability() 返回 GPU 的计算能力（Compute Capability）。Ampere 架构（A100）的计算能力是 8.0，所以 >= (8, 0) 就能覆盖所有支持 BF16 的 GPU。

15.5 实际加速效果与显存收益

说了这么多原理，来看看实际效果。下面这组数据来自在一个 ResNet-50 + ImageNet 的实验中（batch size=256，A100 80GB）：

精度模式 吞吐量（img/sec） 显存占用 收敛所需 epoch 代码改动量
FP32 1200 28.5 GB 90 无
AMP (FP16 + Scaler) 2800 15.2 GB 90 +4 行
AMP (BF16) 2700 15.2 GB 90 +2 行

可以看到：吞吐量提升了 2.3 倍，显存占用降低了一半，而收敛所需的 epoch 数和 FP32 完全一致。对于大模型来说收益更明显。在 Llama 7B 的 Fine-tuning 场景中，AMP（BF16）是事实上的标配——没有人会在 FP32 下微调 7B 模型，因为显存放不下，速度也慢得无法接受。

15.6 注意事项与常见问题

混合精度训练虽然简单，但也不是完全没有坑。下面几个问题值得注意：

  • Batch size 和 learning rate 可能需要微调。低精度下梯度噪声变大，有些实验发现需要用稍大的 batch size 或者稍小的 learning rate 来补偿。
  • 某些操作不在 autocast 的白名单里。比如 torch.sqrt、torch.sum 等操作在 autocast 下会以 FP32 执行，这没有问题。但如果你的自定义操作中有一些数值不稳定的计算，可能需要手动 with autocast(device_type="cuda", enabled=False): 来关闭。
  • 分布式训练中每个 rank 都需要自己的 scaler。如果你用 DDP，每个进程创建自己的 GradScaler，不需要额外同步——scaler 的状态不跨进程传播。
  • 验证/推理时不要用 autocast。虽然用了也不会错，但推理阶段通常直接用 FP16 或 BF16 做推理，节省显存就够了，不需要 autocast 的开销。
一个容易被忽略的坑：如果在 FP16 训练中使用了 torch.mul 或 torch.add 等逐元素操作来计算 loss（而不是使用 PyTorch 内置的损失函数），这些操作可能会以 FP16 执行，导致 loss 精度不够，影响梯度缩放的效果。建议总是使用 torch.nn 模块提供的损失函数，它们已经被优化为在 autocast 下自动回退到 FP32。
