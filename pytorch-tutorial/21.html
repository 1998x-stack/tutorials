第21章 分布式训练入门

大家好，我是你们的讲师。让我们面对一个现实：现在的模型越来越大，单卡训练已经不够用了。BERT-large有3.4亿参数，Llama-70B有700亿参数，而你手头只有8张40GB的A100。怎么把模型和数据分配到多张卡上，让它们协同工作？这就是分布式训练要解决的问题。

很多人一听到"分布式"就觉得很复杂——要配置网络、管理进程通信、处理同步问题。好消息是，PyTorch的分布式工具链在过去几年里变得越来越易用。这一章我们从最基础的Why和What讲起，然后用一个实际的2-GPU例子，带你把DDP（Distributed Data Parallel）跑通。

21.1 为什么要分布式？

分布式训练的动机可以概括为三个词：内存、速度、规模。

  • 内存：一个大模型可能本身就需要几十GB显存放参数，加上梯度、优化器状态、激活值，单卡根本装不下。分布式可以把模型切分到多张卡的显存里。
  • 速度：如果单卡一个batch要1秒，理论上8卡可以把batch size放大到8倍（global batch size），或者在每个卡上放一个小batch并行计算，一次迭代的时间可以降到1/8（理想情况下）。
  • 规模：想在合理时间内完成训练，需要更大的计算资源。分布式是从单卡到大规模集群的必经之路。

21.2 三种并行策略

根据你要并行的是什么，分布式训练分为三种主要策略：

策略 如何切分 通信量 典型场景
Data Parallel
（数据并行）		 每卡一份完整模型，数据切分到各卡 每步all-reduce梯度（通信量与模型大小成正比） 模型能装进单卡，但需要更大吞吐的场景
Model Parallel
（模型并行）		 模型的不同层放在不同卡上，数据逐层流水 每层forward/backward传输激活和梯度 模型太大装不进单卡，但层与层可以自然切分
Pipeline Parallel
（流水线并行）		 模型按层分成多个stage，每个stage在一张卡上，micro-batch流水执行 stage之间传输激活和梯度（需处理bubble） 模型非常大且层数深，可以分成多个stage

这一章我们重点讲数据并行（DDP），因为它是最常用、最易上手的分布式训练方式。模型并行和流水线并行在下一章FSDP中会涉及。

21.3 分布式训练的核心概念

在写代码之前，先理解几个关键概念。所有分布式训练都围绕 torch.distributed 包展开：

  • WORLD：所有参与训练的进程（process）构成一个"世界"。WORLD_SIZE 就是进程总数。
  • RANK：每个进程在world中的唯一编号，从0到WORLD_SIZE-1。RANK=0通常是主进程（负责打印日志、保存checkpoint等）。
  • LOCAL_RANK：进程在本地机器上的编号。如果你有2台机器各4卡，LOCAL_RANK是0-3，而RANK是0-7。
  • MASTER_ADDR / MASTER_PORT：RANK=0进程的地址和端口，其他进程通过它建立通信。
  • BACKEND：通信后端。GPU训练一般用 nccl（NVIDIA Collective Communication Library），CPU用 gloo。

初始化的标准代码：

import torch.distributed as dist

def setup(rank, world_size):
    dist.init_process_group(
        backend="nccl",           # GPU训练用nccl
        init_method="env://",     # 从环境变量读取MASTER_ADDR等
        rank=rank,
        world_size=world_size
    )
    torch.cuda.set_device(rank)   # 每个进程绑定一张卡

def cleanup():
    dist.destroy_process_group()

21.4 DistributedSampler：正确的数据分配

在数据并行中，每个进程只处理全局batch的一部分。如果直接用普通的 DataLoader，每个进程都会加载完整的数据集——等于每个batch被重复处理了 WORLD_SIZE 次。

DistributedSampler 解决了这个问题：它确保每个进程拿到数据集中不相交的一个子集，并且会在每个epoch自动shuffle。

from torch.utils.data.distributed import DistributedSampler
from torch.utils.data import DataLoader

dataset = YourDataset(...)
sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,
    rank=rank,
    shuffle=True,
    seed=42
)
dataloader = DataLoader(dataset, batch_size=32, sampler=sampler)

for epoch in range(num_epochs):
    sampler.set_epoch(epoch)  # 每个epoch必须调用，保证不同epoch的shuffle不同
    for batch in dataloader:
        # 训练代码
        pass
小贴士：每个epoch一定要调用 sampler.set_epoch(epoch)。否则每个epoch的数据shuffle顺序都一样，模型可能会学到一些伪相关。这个问题在分布式训练中很容易被忽略。

21.5 DDP：Distributed Data Parallel

DDP（torch.nn.parallel.DistributedDataParallel）是PyTorch推荐的数据并行方案。它会把模型复制到每张卡上，然后在每次反向传播时自动执行gradient all-reduce——把所有卡上的梯度求和后再平均，确保每个卡的模型参数更新一致。

DDP的工作机制可以概括为：

  1. 每张卡用自己的数据子集做前向传播，计算loss。
  2. 每张卡独立做反向传播，计算梯度。
  3. DDP在反向传播结束后自动触发 all-reduce 操作：所有卡的梯度求和取平均。
  4. 每张卡用平均后的梯度更新自己的模型参数。
  5. 因为初始参数相同、梯度相同，所有卡的模型保持同步。
from torch.nn.parallel import DistributedDataParallel as DDP

model = YourModel().to(rank)
model = DDP(model, device_ids=[rank])

optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for batch in dataloader:
    inputs, labels = batch
    inputs, labels = inputs.to(rank), labels.to(rank)

    outputs = model(inputs)
    loss = criterion(outputs, labels)

    optimizer.zero_grad()
    loss.backward()    # DDP在这里自动执行梯度all-reduce
    optimizer.step()

注意：DDP只用包裹一次model。所有后续操作——定义optimizer、前向传播、反向传播——都和单卡代码完全一样！这也是DDP最大的优点：侵入性极小。

21.6 torchrun：启动分布式训练

以前启动分布式训练需要手动用 torch.distributed.launch 或者 mp.spawn，非常麻烦。现在推荐的方式是使用 torchrun：

# 在2张卡上运行训练脚本
torchrun --nproc_per_node=2 train.py

torchrun 会自动帮你设置以下环境变量：

  • LOCAL_RANK、RANK、WORLD_SIZE、MASTER_ADDR、MASTER_PORT
  • LOCAL_WORLD_SIZE（本机上的进程数）

因此你的 train.py 入口只需要从环境变量读取这些值：

import os
import torch
import torch.distributed as dist

def main():
    local_rank = int(os.environ["LOCAL_RANK"])
    rank = int(os.environ["RANK"])
    world_size = int(os.environ["WORLD_SIZE"])

    dist.init_process_group(backend="nccl")
    torch.cuda.set_device(local_rank)

    model = YourModel().cuda(local_rank)
    model = DDP(model, device_ids=[local_rank])

    # ... 其余训练代码

    dist.destroy_process_group()

if __name__ == "__main__":
    main()

用 torchrun 启动：

torchrun --nproc_per_node=2 train.py
完整2-GPU训练示例：
torchrun --nproc_per_node=2 --master_port=29500 train.py 启动后，train.py 中每个进程会自动拿到自己的RANK和LOCAL_RANK。每张卡上的batch_size=32，全局batch_size=64。loss curve和单卡基本一致，但训练速度接近2倍。

21.7 DDP 常见陷阱

DDP虽然看起来简单，但有几个容易踩的坑：

问题 现象 解决方案
broadcast_buffers=False BN的running_mean/var不同步，多卡推理结果不一致 设置 DDP(model, broadcast_buffers=True)
find_unused_parameters=True 缺失 模型中有参数没有参与loss计算（如预训练模型的特征提取器被冻结），DDP报错 设置 DDP(model, find_unused_parameters=True)
batch_size设置错误 每个进程设置了全局batch size而不是per-GPU batch size per-GPU batch_size = global_batch_size / world_size
使用 nn.DataParallel DataParallel是单进程多线程，不支持 torch.compile，性能差 不要用 DataParallel，用 DDP + torchrun
踩坑故事：有个朋友刚开始做分布式训练，听说PyTorch有 nn.DataParallel（注意，不是DistributedDataParallel），觉得名字挺像就用上了。DataParallel确实可以多卡训练，但它的实现是单进程多线程——一个进程内创建一个线程做前向，然后用一个线程做all-reduce。这导致三个问题：第一，GIL限制了多线程的计算并行；第二，不支持 torch.compile；第三，只能用在一台机器上。等模型从ResNet-50换到ViT-large之后，DataParallel直接OOM了。最后换成DDP + torchrun，不仅速度翻了3倍，而且代码还更简洁了。现在PyTorch官方已经明确说DataParallel是legacy，新项目不应该使用。
下章预告：DDP解决了"数据太大"的问题，但解决不了"模型太大"的问题。当你的模型大到单卡连参数都放不下时，就需要更激进的手段——FSDP（Fully Sharded Data Parallel）。下一章我们进入大模型训练的世界，看看怎么把Llama-70B这样的巨无霸在多卡上跑起来。
