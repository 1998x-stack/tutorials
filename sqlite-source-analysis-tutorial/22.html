第22章 表达式索引与自修复

大家好，我是你们的讲师。上一章我们探索了 FTS 全文搜索，这一章来聊一个较为"小众"但极其强大的特性——表达式索引 和 3.53.0 引入的 自修复索引（Self-healing Index）。

考虑一个常见场景：你的表中有 name 列，想做大写不敏感搜索。WHERE LOWER(name) = 'john' 能工作但无法利用普通索引。表达式索引 CREATE INDEX i ON users(LOWER(name)) 解决了这个问题——它把表达式的结果作为索引 key，让查询规划器能在 LOWER(name) = 'john' 上使用索引。

表达式索引 = 索引列是一个表达式，而不是原字段。 它的实现涉及索引创建、存储、查询匹配和自修复等多个子系统。3.53.0 的自修复机制在数据库领域开创了"索引自我诊断"的先河。

表达式索引的原理

表达式索引的创建入口是 sqlite3CreateIndex()（src/build.c）。当解析器遇到 CREATE INDEX ON t(expr) 时，它识别出索引列不是普通列名而是一个表达式：

/* build.c — 表达式索引的判断逻辑 */
void sqlite3CreateIndex(Parse *pParse, ...){
  Index *pIndex;
  for(i=0; i<pList->nExpr; i++){
    Expr *pExpr = pList->a[i].pExpr;
    if( pExpr->op==TK_COLUMN ){
      pIndex->aiColumn[i] = pExpr->iColumn; /* 普通列 */
    }else{
      pIndex->aiColumn[i] = -2;             /* 表达式！ */
      pIndex->aColExpr[i] = pExpr;          /* 记录表达式定义 */
    }
  }
  /* 禁止非确定性函数 */
  if( sqlite3ExprContainsNonDeterministicFunction(pExpr) ){
    sqlite3ErrorMsg(pParse, "expressions in index must be deterministic");
  }
}

aiColumn[i] = -2 是关键标记：SQLite 的普通索引用 aiColumn[i] 存储列号，而表达式索引用 -2 标记"请到 aColExpr 查找表达式定义"。

在查询阶段，规划器通过 sqlite3ExprCompare() 做语法树结构匹配——比较的是表达式树的拓扑结构，而非字符串。所以 LOWER(name) = 'john' 和索引 LOWER(name) 能匹配上，即使写法不同。

表达式索引的存储

表达式索引的数据存储在 B-Tree 中，与普通索引使用同一存储引擎。区别在于 key 的生成方式——普通索引直接读列值，表达式索引需要执行表达式计算。在 VDBE 层面，表达式索引的插入操作在执行 OP_MakeRecord 之前先调用 OP_Function 计算表达式值。

这个额外计算在大多数场景下可以接受，但如果表达式中包含昂贵的函数（如复杂 JSON 操作），写性能会显著下降。

索引类型Key 来源更新代价
普通索引直接读列值低
部分索引列值（有条件）低
表达式索引表达式计算结果中（需计算表达式）
自修复索引表达式 + 版本戳稍高（加 4 字节校验）

索引到期的原因

普通索引不会"过期"，但表达式索引可能不一致。虽然 SQLite 禁止非确定性函数（random()、current_timestamp 等），但以下灰色地带仍可能导致索引过期：

  • 用户自定义函数暗含外部依赖（如全局变量或配置表）
  • 排序规则（collation）变更影响索引顺序
  • ICU 扩展更新改变字符串比较行为
⚠️ 真实踩坑： 一个同事编写了自定义函数 calc_discount(price, region)，它根据外部配置表计算折扣，被标记为 SQLITE_DETERMINISTIC 用于表达式索引。某天配置表更新了折扣规则，但索引中存储的还是旧值——查询返回了过期的折扣信息。教训是：不要给依赖外部状态的函数贴上 "确定性" 标签。

3.53.0 自修复索引机制

SQLite 3.53.0 引入的自修复索引给每个索引条目附加一个 版本戳（4 字节）。每次重建索引时版本戳递增。查询时对比版本戳，发现不匹配时触发修复流程：

/* Index 结构的新增字段 */
struct Index {
  u32 iSelfHealVersion;   /* 当前索引版本号 */
  u8 bSelfHealing;        /* 是否启用自修复 */
};

/* 索引扫描时检测 */
static int indexEntryIsValid(Index *pIdx, const unsigned char *pKey,
                              int nKey, i64 rowid){
  if( pIdx->bSelfHealing==0 ) return 1;
  u32 iEntryVer;
  memcpy(&iEntryVer, pKey + nKey - sizeof(u32), sizeof(u32));
  return (iEntryVer == pIdx->iSelfHealVersion);
}

/* 自修复路径 — 重新计算表达式并修正 */
static int sqlite3IndexSelfHeal(Index *pIdx, i64 rowid, ...){
  /* 1. 重新计算表达式值 */
  /* 2. 与旧的索引条目对比 */
  /* 3. 不一致则修正并递增版本戳 */
  /* 4. 记录修复日志 */
}

这个机制在正常查询路径上只增加了一次 4 字节比较，开销几乎为零。修复流程只在检测到不一致时才触发——而这在实践中非常罕见。

自修复索引通过 CREATE INDEX ... WITH selfhealing 启用，或通过 REINDEX idx WITH selfhealing 对已有索引开启。

设计哲学： SQLite 没有采用"预防性"方案（锁定所有依赖），而是选择了"检测+修复"的轻量方案。这是一种"允许出错但自动纠错"的设计思路——与 ECC 内存、TCP 丢包重传一脉相承。

REINDEX EXPRESSIONS 语句

3.53.0 还提供了 REINDEX EXPRESSIONS 语句，用于手动触发所有表达式索引重建。实现逻辑：遍历所有表的索引，检查 aiColumn[i] == -2 标记，对每个表达式索引执行完整重建并递增版本戳。

-- 重建所有表达式索引
REINDEX EXPRESSIONS;

-- 只重建特定表上的
REINDEX EXPRESSIONS table_name;

-- 重建单个索引
REINDEX index_name;

实战：创建和使用表达式索引

场景一：大小写不敏感搜索

CREATE TABLE users(id INTEGER PRIMARY KEY, name TEXT, email TEXT);
INSERT INTO users VALUES (1, 'Alice', 'Alice@Example.com');

-- 无索引时全表扫描
EXPLAIN QUERY PLAN
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';
-- SCAN users

-- 创建表达式索引后使用索引查找
CREATE INDEX idx_email_lower ON users(LOWER(email));
EXPLAIN QUERY PLAN
SELECT * FROM users WHERE LOWER(email) = 'alice@example.com';
-- SEARCH users USING INDEX idx_email_lower

场景二：JSON 字段查询加速

CREATE TABLE products(id INTEGER PRIMARY KEY, data JSON);
CREATE INDEX idx_price ON products(JSON_EXTRACT(data, '$.price'));

-- 利用表达式索引加速范围查询
SELECT * FROM products
WHERE JSON_EXTRACT(data, '$.price') BETWEEN 500 AND 1000;
-- SEARCH products USING INDEX idx_price

场景三：函数结果排序

CREATE INDEX idx_body_len ON posts(LENGTH(body));

-- ORDER BY 表达式也使用索引！
SELECT id, LENGTH(body) FROM posts ORDER BY LENGTH(body) DESC LIMIT 10;
💡 权衡： 表达式索引不是银弹。每次 INSERT/UPDATE 都增加表达式计算开销。如果写入量大且表达式复杂，可考虑"预计算列"——增加一个计算列并在其上建普通索引，避免每次写入都做表达式求值。
