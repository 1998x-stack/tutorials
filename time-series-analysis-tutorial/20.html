第20章 树模型与时序预测

大家好。上一章我们学习了如何把时间序列转换成 ML 模型能吃的特征表格。有了这张表格，接下来该用什么模型呢？

如果去 Kaggle 的时序竞赛中逛一圈，你会发现几乎所有获奖方案都有一个共同的主角——树模型。XGBoost、LightGBM、CatBoost 这些梯度提升树早就取代了纯 ARIMA 和简单线性回归，成为工业界时序预测的事实标准。为什么？因为树模型天然适合时序预测的几个核心挑战。

20.1 为什么树模型适合时序数据

树模型在时序预测上有几个不可替代的优势：

非线性关系。真实世界的时序数据几乎从不是线性的。促销活动的边际效益递减、温度对电量的非线性影响（空调在极热和极冷时都开得更猛）、广告投入的饱和效应——树模型的决策边界天然能捕捉这些非线性关系，不需要你手动设计交互项或多项式变换。

特征交互自动捕捉。想象一下这个场景：某款冰激凌的销量 = 温度 × 是否周末 × 是否在学校附近。这种三层交互用线性模型需要手动构造，用树模型则会被自动捕捉——树的分裂过程本质就是在学特征之间的条件依赖关系。

对异常值相对鲁棒。时序数据中异常值几乎不可避免。树模型基于阈值做分裂，单点的异常值对整体树结构的影响有限——它不会被"拉偏"到异常点方向。相比之下，线性模型和神经网络对异常值的敏感度高得多。

缺失值友好。XGBoost 和 LightGBM 原生支持缺失值，它们会在训练时自动学习缺失值的最佳分裂方向。你不需要在预处理阶段做复杂的填充策略（尽管我们上一章讨论的特征工程仍然重要）。

20.2 XGBoost vs LightGBM vs CatBoost

三大主流梯度提升树框架各有特点，选择哪个取决于你的数据规模和场景：

特性XGBoostLightGBMCatBoost
训练速度（大数据集）中等快中等
类别特征处理需手动编码需手动编码原生支持
缺失值处理原生支持原生支持原生支持
默认参数效果较好一般（需调参）好
GPU 加速支持支持支持
时序预测常见程度极高高中等
小数据集表现好可能过拟合好

总体建议：默认用 LightGBM——速度快、内存低、效果好。如果数据量小于 1 万行，用 XGBoost 或 CatBoost。如果特征中有大量类别变量，重点考虑 CatBoost（它不需要做 one-hot 编码）。

20.3 实战：XGBoost 时序预测管道

下面是一个完整的 XGBoost 时序预测流程，结合了上一章的特征工程技术：

import pandas as pd
import numpy as np
import xgboost as xgb
from sklearn.metrics import mean_absolute_error

# 1. 加载数据
df = pd.read_csv('sales.csv', parse_dates=['date'])
df = df.sort_values('date')

# 2. 特征工程
def make_features(df):
    data = df.copy()
    # 滞后特征
    for lag in [1, 2, 3, 7, 14, 28]:
        data[f'lag_{lag}'] = data['sales'].shift(lag)
    # 滚动统计量
    data['rolling_mean_7'] = data['sales'].shift(1).rolling(7).mean()
    data['rolling_std_7'] = data['sales'].shift(1).rolling(7).std()
    # 日历特征
    data['dayofweek'] = data['date'].dt.dayofweek
    data['month'] = data['date'].dt.month
    data['is_weekend'] = data['dayofweek'].isin([5, 6]).astype(int)
    return data

df = make_features(df)
df = df.dropna()

# 3. 时序划分（不用随机划分！）
train = df[df['date'] < '2024-01-01']
test = df[df['date'] >= '2024-01-01']

features = [c for c in df.columns if c not in ['date', 'sales']]
X_train, y_train = train[features], train['sales']
X_test, y_test = test[features], test['sales']

# 4. 训练
model = xgb.XGBRegressor(
    n_estimators=500, learning_rate=0.05,
    max_depth=6, subsample=0.8,
    early_stopping_rounds=50
)
model.fit(X_train, y_train,
    eval_set=[(X_test, y_test)],
    verbose=False)

# 5. 预测与评估
y_pred = model.predict(X_test)
print(f'MAE: {mean_absolute_error(y_test, y_pred):.2f}')

这段代码看起来很简单，但请注意第 3 步中的关键行：train = df[df['date'] < '2024-01-01']。这里用的是时间划分，而不是随机抽样。很多初学者在这个地方犯错——用 train_test_split 默认的随机划分会导致模型"看到未来"，在验证集上表现异常好，上线后直接崩盘。

20.4 特征重要性：别被误导

树模型的 feature importance 是一个诱人的工具，但也是一个陷阱。XGBoost 默认提供三种重要性指标：

  • weight：特征被用于分裂的次数。容易被取值多的特征主导。
  • gain：特征在分裂时带来的平均信息增益。更可靠，但在多叉树中可能有偏。
  • cover：特征覆盖的样本数。受特征取值分布影响大。
踩坑警告：我见过一个团队用 XGBoost 预测股票收益率，特征重要性排名第一的是 lag_1（前一天的收益率），占到了总重要性的 60%。他们据此得出结论"股市有很强的动量效应"，兴高采烈地部署了策略。结果实盘一个月亏了 15%。问题出在哪里？Gain 指标偏爱在树根附近分裂的特征。lag_1 就是因为相关性高、最早被选中分裂，所以 gain 值虚高。真正有价值的其实是排名第二到第五的特征组合。永远不要只看 importance 排名就下结论——做 permutation importance 或 SHAP 值来交叉验证。

20.5 时序预测中的超参数调优

树模型的超参数调优和普通 ML 任务有两点显著不同：

验证策略必须是时序的。不能用随机 K-Fold，必须用时序交叉验证（下一章我会详细讲）。在调优时，即使你觉得麻烦，也请至少使用一个单独的验证集（按时间分割），不要依赖 early stopping 在随机划分的验证集上的表现。

关注过拟合时域。时序模型很容易在近期数据上过拟合。一个实用的技巧是用滑动窗口做 backtest：在历史的不同时间点重复回测，取平均性能。如果模型在某个时间段表现特别差但在总体表现好，说明它可能只是在"记住"最近的模式。

常用超参数调优方向：

参数调优方向时序场景建议
learning_rate小（0.01-0.1）时序数据噪声大，用小学习率 + 多轮迭代
max_depth3-8特征数少时浅（3-5），特征多时深（6-8）
subsample0.6-0.9时序数据样本间有相关性，subsample 不宜太小
min_child_weight5-30调高可减少对噪声的过拟合
lambda / alpha0.1-10正则化在时序中比表格数据更重要
关键洞察：时序预测中，特征工程对效果的影响远大于模型选择和超参数调优。我曾经做过一个对照实验：同样的 LightGBM 模型，一个用了精细的时序特征（滞后、滚动、日历、傅里叶项），一个只用了原始值和少量滞后。前者的 MAE 比后者低了 37%，而两者的超参数只是默认值。把时间花在特征工程上，回报远高于调参。

20.6 著名的 99.7% 准确率翻车故事

这个故事值得每一个做时序预测的人铭记于心。几年前，一位量化分析师在某个论坛上兴奋地展示自己的模型——用 XGBoost 预测某只股票的涨跌方向，达到了99.7% 的准确率。论坛瞬间炸锅，有人质疑、有人崇拜、有人想掏钱买策略。

但很快，有人指出了问题：他的代码里有这样一行：

from sklearn.model_selection import train_test_split
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

train_test_split 默认是随机划分的。而股票数据是高度自相关的时间序列——昨天的价格和今天高度相似。随机划分意味着测试集中大量样本的"邻居"在训练集中。换句话说，模型不是在"预测"未来，而是在"回忆"过去见过的类似模式。这不叫预测，这叫look-ahead bias。

这就是 99.7% 真相。当你看到时序预测中离谱的高分时，第一反应应该是：是不是哪里泄露了？大多数时候，答案都是肯定的。

20.7 下一站：自动化

树模型给了我们强大的预测能力，但也带来了新的问题：特征工程要手动做，超参数要手动调，季节性和节假日要手动编码。有没有更自动化的工具？

有的。下一章我们要学习 Prophet 和 Nixtla 生态，它们把时序预测的很多重复劳动自动化了。Prophet 的"点几下就能预测"理念和 Nixtla 的规模化预测工具，会让你的工作效率提升一个数量级。但理解和手动做特征工程的能力，仍然是你的核心竞争力——自动化工具出了问题，最终还是靠你对特征和数据的理解来排查。
