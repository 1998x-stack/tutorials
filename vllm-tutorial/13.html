第13章 V1架构下篇：性能与迁移

上一章看了 V1 的架构设计。这一章咱们用数据说话——V1 比 V0 到底快了多少？以及如果你是从 V0 过来的，有什么要注意的兼容性变化。

13.1 V0 vs V1 性能对比

指标V0V1提升
7B 模型吞吐量1,200 tok/s2,100 tok/s+75%
70B 模型首 token 延迟 (TTFT)320ms185ms-42%
GPU 利用率65%92%+41%
长上下文推理效率基准2.3×+130%
P99 延迟标准差 (高 QPS)基准-62%大幅降低

最让我意外的是 P99 延迟：标准差降低 62% 说明 V1 的调度一致性好得多——长请求不再严重干扰短请求。这对于生产环境的 SLA 保障意义重大。

13.2 兼容性变化（V0→V1）

功能 / 参数V0V1迁移建议
best_of支持（生成多个候选取最佳）🔴 已移除在应用层实现
每请求 logits 处理器支持🔴 已移除使用全局 logits 处理器
GPU↔CPU KV cache 置换支持（用于抢占）🔴 已移除不再需要（新调度器无需置换）
logprobs返回每个 token 的 log 概率🟡 语义有变化检查 logprobs 的字段名和格式
Block Size可配置固定 16无需调整

迁移注意：如果你从 V0 升级到 V1，先在一个测试环境跑一遍。重点关注 (1) logprobs 格式是否兼容你的下游代码 (2) 是否用了 best_of 参数 (3) 自定义 logits processor 是否在 V1 中正常工作。

13.5 选择 V0 还是 V1

选 V1（推荐）:
├── 新项目、新部署
├── 需要稳定性和长期支持
├── 需要多模态/推测解码等新特性
└── 对 CPU 开销敏感

暂时留在 V0（有限场景）:
├── 依赖 best_of 参数（V1 已删除）
├── 大量自定义 CUDA kernel 尚未迁移到 V1
└── 已经大量调优的 V0 生产环境（逐步迁移）

建议：如果你现在开始一个新项目，直接用 V1。V0 不会再新增功能。如果你在生产中用 V0，不需要紧急升级——V0 的 bug 仍在修复。但新功能不会再 backport 到 V0。
