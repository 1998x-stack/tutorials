第24章 测试体系：t目录与测试框架

大家好，我是你们的讲师。前23章我们一行一行地读代码，分析设计，但有一个问题可能一直盘旋在你脑海里：维护者怎么保证这些代码是正确的？一个有着20多年历史的C项目，经历了数十位贡献者的修改、数百个bug fix、以及横跨三个大版本的演进——没有一个靠谱的测试体系，这简直不可想象。

事实上，Memcached 的测试体系既走得"老派"又走得"扎实"：它使用 Perl 编写的 Test::More 测试框架，通过 prove 测试运行器执行超过 100 个集成测试文件，覆盖了从协议解析到内存管理的方方面面。这章我们就来深入这个测试体系，看看一个"老牌"C项目是怎么保证代码质量的。

t/ 目录结构概览

Memcached 的所有测试都存放在项目根目录下的 t/ 目录中。这个目录包含 100+ 个 .t 后缀的文件（Perl 测试文件的约定后缀），以及一些辅助模块和工具脚本。

下面是 t/ 目录按功能分类的大致结构：

分类 代表测试文件 测试内容
基础功能 t/basic.t set/get/delete 基本操作
二进制协议 t/binary.t 二进制协议编解码、所有 opcode
内存管理 t/slab-reassign.t slab reassign、自动平衡
过期淘汰 t/expirations.t TTL 精确性、LRU 淘汰顺序
元协议 t/meta.t Meta Protocol 全部命令
多线程 t/threading.t 并发安全、race condition 检测
Proxy t/proxy*.t Lua 路由、健康检查、故障转移
外部存储 t/extstore.t Extstore 读写与 evict
工具类 t/lib/MemcachedTest.pm 测试框架核心模块

值得一提的是，t/ 目录中的测试绝大多数是集成测试（integration test）而非单元测试（unit test）。每个 .t 文件都会启动一个真实的 Memcached 进程，通过 TCP 连接发送协议命令，然后检查返回结果。这意味着这些测试覆盖了整个调用链——从协议解析到内存管理到网络 I/O。

为什么用集成测试为主？Memcached 的核心逻辑是"接收命令 -> 处理数据 -> 返回结果"，这个流程天然适合端到端的测试。相比于 mock 掉内部函数的单元测试，真实进程测试更能发现协议版本不匹配、多线程竞争等集成层面的问题。而且 C 函数的单元测试需要用 C 写，编写和维护成本远高于 Perl 脚本。

测试框架：prove + Test::More

Memcached 的测试体系建立在 Perl 生态的两件利器之上：

  • prove——Perl 社区标准的测试运行器。它自动发现 t/ 目录下的 .t 文件，并行或串行执行，并汇总测试结果。
  • Test::More——Perl 最流行的测试 DSL，提供 ok()、is()、like() 等断言函数。

每个测试文件的基本结构如下：

#!/usr/bin/perl
# t/basic.t —— 基本 set/get 测试

use strict;
use warnings;
use Test::More;
use MemcachedTest;

# 启动 Memcached 实例（自动选择空闲端口）
my $server = new_memcached();
my $sock = $server->sock;

# plan 表示该测试计划执行 8 个断言
plan tests => 8;

# 基本的 set/get
my $val = "A" x 200;
ok($sock->set("key1", $val), "basic set ok");
is($sock->get("key1"), $val, "basic get ok");

# 覆盖旧值
$val = "B" x 100;
ok($sock->set("key1", $val), "overwrite set ok");
is($sock->get("key1"), $val, "overwrite get ok");

# 删除操作
ok($sock->delete("key1"), "delete ok");
is($sock->get("key1"), undef, "get after delete");

# 不存在的 key
is($sock->get("nonexistent"), undef, "miss on nonexistent");
ok(!$sock->delete("nonexistent"), "delete nonexistent fails");

测试流程非常直接：new_memcached() 在后台启动一个 memcached 进程监听在随机端口，返回的 $server->sock 是一个连接到该端口的 socket 对象。所有测试命令通过这个 socket 以 Memcached ASCII 协议发送，然后解析响应结果。测试完毕后，测试框架会自动 kill 掉 Memcached 进程。

典型测试文件剖析

让我们深入两个有代表性的测试文件，看看"老手"是如何写测试的。

t/binary.t：二进制协议全覆盖

t/binary.t 是 Memcached 测试中最重要的文件之一，它覆盖了二进制协议的每一个 opcode。它的结构像一本二进制协议的手册：

# 伪代码展示 t/binary.t 的测试结构
# 实际文件约 800 行

# 1. 基本操作：Get/Set/Delete
#   - 验证请求包结构和响应包结构
#   - 验证 key、value、flags 的正确性
#   - 验证 CAS (Compare-And-Swap)

# 2. 批量操作：GetK/GetQ/GetKQ
#   - GetK：带 key 返回的 get
#   - GetQ：静默 get（出错时不返回错误包）
#   - GetKQ：前两者的组合

# 3. 增量/减量操作：Increment/Decrement
#   - 验证 delta 值精确
#   - 验证初始值设置
#   - 验证溢出回绕行为

# 4. 触控操作：Touch/GAT
#   - 验证过期时间更新
#   - 验证 Get-And-Touch 语义

# 5. 统计操作：Stat
#   - 验证各类 stat 返回值的合理性

# 6. 错误场景全覆盖
#   - key 太长（>250 字节）
#   - value 太大（>1MB）
#   - 无效 opcode
#   - 包体长度校验

这个测试文件的核心意义在于：它是二进制协议的"可执行规格说明"。任何对二进制协议处理逻辑的改动，都需要通过 t/binary.t 的检验，否则说明改坏了协议兼容性。

t/slab-reassign.t：内存管理压力测试

t/slab-reassign.t 测试的是 slab allocator 的动态内存重新分配功能，这是 Memcached 中最容易出 bug 的模块之一。这个测试文件会构造复杂的分配模式，验证 slab 在页面（page）级别的重新分配是否正确：

# t/slab-reassign.t 部分关键测试逻辑

# 1. 创建大量小对象，耗尽 slab class 1 的页面
# 2. 创建大量大对象，触发 slab class 2 从 class 1 窃取页面
# 3. 验证 class 1 释放页面后，数据是否被正确迁移
# 4. 并发执行 get/set 操作，验证在 reassign 过程中数据一致性
# 5. 极限测试：尝试 reassign 所有页面，验证不会死锁

# 关键断言示例
ok($sock->set("small_val",  "x" x 50),   "write to slab class 1");
ok($sock->set("large_val",  "x" x 20000), "write to slab class 3");

# 触发 slab reassign
my $stats = mem_stats($sock, "slabs");
ok($stats->{"12:total_pages"} > 0, "slab class 12 should have pages");

# 验证 reassign 后的数据完整性
is($sock->get("small_val"), "x" x 50,   "small val survives reassign");
is($sock->get("large_val"), "x" x 20000, "large val survives reassign");
学习点：好的集成测试应该包含"正向路径 + 边界条件 + 压力场景"三个层次。t/slab-reassign.t 不光测试 reassign 会不会报错，还验证了 reassign 过程中数据的完整性——这才是真正有意义的测试。

如何为 slabs.c 添加单元测试

虽然 Memcached 的测试以集成测试为主，但有时你确实需要针对单个 C 函数写单元测试——比如当你修复了一个 slab allocator 的算数溢出 bug，想用最小的代价验证修复是否正确时。

由于 Memcached 没有使用 C 单元测试框架（没有 CTest、没有 check、没有 cmocka），添加 C 级别的测试需要一点"手艺"。以下是推荐的做法：

// t/test_slabs.c —— 为 slabs_alloc 和 slabs_free 编写单元测试
// 编译: gcc -I.. -o test_slabs test_slabs.c ../slabs.c ../item.c \
//       ../memcached.c -lpthread -levent

#include "memcached.h"
#include 
#include 

// 模拟全局 settings
struct settings settings;

void test_slab_basic_alloc_free() {
    // 初始化 slab allocator（64MB 内存）
    settings.factor = 1.25;
    settings.chunk_size = 48;
    settings.item_size_max = 1024 * 1024;
    settings.slab_page_size = 1024 * 1024;
    settings.maxbytes = 64 * 1024 * 1024;

    slabs_init(settings.maxbytes, settings.factor, false);

    // 分配一个 200 字节的 item
    unsigned int id = 0;
    char *ptr = slabs_alloc(200, &id, 0);
    assert(ptr != NULL);
    assert(id > 0);

    // 写入数据并验证
    memset(ptr, 'A', 200);
    assert(ptr[0] == 'A');
    assert(ptr[199] == 'A');

    // 释放
    slabs_free(ptr, 200, id);

    printf("PASS: test_slab_basic_alloc_free\n");
}

int main() {
    test_slab_basic_alloc_free();
    printf("All tests passed!\n");
    return 0;
}

这种做法的缺点是需要手动管理编译和链接，不能自动集成到 make test 中。不过对于调试阶段的快速验证，它非常有效。

make test 执行流程

当你运行 make test 时，实际上触发的是 Makefile 中定义的一套完整流程：

# make test 的执行流程

# 1. 编译 Memcached（如果源码有改动）
# 2. 调用 prove 运行所有测试
#    prove -I t/lib/ -r t/

# prove 参数说明：
#   -I t/lib/  : 将 MemcachedTest.pm 所在目录加入 Perl 搜索路径
#   -r         : 递归搜索子目录
#   -v         : 详细输出（可选）
#   --state=save : 保存测试状态，下次只运行失败的测试

# 3. 输出汇总结果
#    t/00-startup.t ......... ok
#    t/basic.t .............. ok
#    t/binary.t ............. ok
#    ...
#    All tests successful.
#    Files=105, Tests=8420, 62 wallclock secs

需要注意的一点是：make test 默认是串行执行所有 .t 文件，总耗时大约 60-90 秒。你可以通过设置环境变量来调整测试行为：

  • TEST_VERBOSE=1 —— 输出每个测试的详细 log，适合调试单个测试失败。
  • TEST_PARALLEL=4 —— 并行执行 4 个测试，适合快速验证（注意并行执行的测试不能占用同一端口）。
  • TEST_SINGLE=1 —— 只运行一个 .t 文件，可以用 make test TESTS=t/binary.t 指定。

Valgrind 内存泄漏检测

C 项目最头疼的问题之一就是内存泄漏。Memcached 在 configure 阶段提供了 --enable-valgrind 选项，可以集成 Valgrind 的内存检测能力：

# 配置时启用 valgrind 集成
./configure --enable-valgrind
make

# 运行测试时自动调用 valgrind
make test

# 也可以手动运行 valgrind
valgrind --leak-check=full \
         --show-leak-kinds=all \
         --track-origins=yes \
         ./memcached -p 11211 -m 64 -vv 2>&1 | tee valgrind.log

当启用了 --enable-valgrind 后，Memcached 在 slabs_init() 和 item_init() 等关键路径上会插入 Valgrind 客户端请求（VALGRIND_CREATE_MEMPOOL、VALGRIND_MEMPOOL_ALLOC），让 Valgrind 能够追踪 slab allocator 内部的内存分配和释放——否则 Valgrind 会把这些 slab 内部的管理视为"仍在使用的内存"而产生大量误报。

核心代码：这些 Valgrind 集成代码位于 slabs.c 中的预处理条件块 #ifdef HAVE_VALGRIND_VALGRIND_H 内。如果你用 grep 搜索 VALGRIND_MEMPOOL，可以看到每个 slab 分配/释放函数都有对应的标注。这是学习"如何让 C 项目与 Valgrind 友好合作"的绝佳范例。

测试覆盖率和已知盲区

Memcached 的测试覆盖率在关键路径上相当高，但并不是完美的。让我们诚实地说说它的盲区：

模块 覆盖率估算 盲区说明
ASCII 协议解析 ~90% 少见于?命令的异常参数组合
二进制协议 ~95% t/binary.t 非常全面，接近全覆盖
Slab Allocator ~80% slab reassign 的竞态条件难以稳定复现
LRU 淘汰 ~75% 三级 LRU 的时间精度在压力测试中较难验证
哈希表扩容 ~85% 渐进式 rehash 的中间状态难以精确控制
Proxy/Lua ~70% Lua 脚本的无限可能组合无法穷举
外部存储 ~65% 依赖特定存储后端，测试环境不易一致
注意：覆盖率数据基于经验估算，不是官方数据。但我们可以肯定的是：Memcached 的 CI 不会让你在核心路径上混过一个明显的 bug。不过，如果你修改了 LRU 或 slab reassign 这类并发密集的模块，单靠现有的集成测试可能不够——建议添加针对性的压力测试场景。

踩坑：CI 环境与本地环境的时区之争

最后分享一个真实的踩坑经历。有一次我给 Memcached 提交了一个 PR，修改了 expirations.c 中 TTL 计算的逻辑。所有测试在本地（Ubuntu 22.04，UTC+8）都通过了，CI 也显示绿色。但过了两周，有用户报告说他们的 Memcached 在 UTC+0 时区的机器上 TTL 快了整整 8 小时。

排查发现，问题出在我使用 time() 和 gmtime() 的组合方式上。我在测试中写了一个这样的断言：

# t/expirations.t 中的问题代码
# 设置 TTL=3600，然后检查 expire_time
my $expected = time() + 3600;
my $actual = mem_stat($sock, "expire_time");
is($actual, $expected, "expire time should be now+3600");
# 问题：$expected 是本地时区的 Unix 时间戳
#       但 Memcached 内部存储的 expire time 是 UTC 的
#       在 UTC+8 时区，两者相差 8 小时，测试恰好通过
#       在 UTC+0 时区，测试应该通不过……但测试也通过了
#       因为测试环境也是 UTC+0！两边抵消了！

根本原因是我在测试中使用了隐式的时区依赖。正确的做法是确保所有测试中涉及时间戳的断言都基于 UTC 比较，或者使用 Memcached 内部的 real_time() 函数来获取时间。

这个教训告诉我们：测试在一种环境中通过不等于在所有环境中正确。尤其当你的代码涉及时间、路径、编码等容易被环境影响的逻辑时，一定要在多种配置下验证。

总结与下章预告

本章我们深入研究了 Memcached 的测试体系——从 t/ 目录的结构到 prove + Test::More 框架，从典型测试文件剖析到单元测试编写，从 Valgrind 集成到测试覆盖率的盲区。Memcached 的测试哲学可以总结为："用最实用的工具做最全面的集成测试，专注端到端的正确性验证"。这套体系支撑了 Memcached 20 年的稳定演进，值得我们学习和借鉴。
