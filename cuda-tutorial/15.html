性能分析工具

大家好，我是你们的讲师。会用 profiler 才是高级 CUDA 程序员的标志——先测量，再优化。我见过太多热血沸腾的新手，花了两周时间把内核代码改得面目全非，自认为做了"天衣无缝的优化"，结果一跑 benchmark——不仅没变快，反而慢了。为什么？因为他们不知道瓶颈在哪里。

打个比方：你开车觉得动力不够，直觉告诉你要换发动机。但如果实际上问题是轮胎打滑呢？你换了再强的发动机也跑不快。GPU 性能优化也一样——你得先搞清楚到底是什么在限制性能，是计算太慢（compute-bound），还是数据搬运太慢（memory-bound），或者是调度开销太大（latency-bound）。没有 profiler，你就是在盲目地"换发动机"。

核心原则： 永远不要凭感觉做优化。Profiler 是你的眼睛，它告诉你 GPU 的每一毫秒都花在了哪里。本章介绍的工具——Nsight Systems、Nsight Compute、Roofline 模型——是你工具箱中最核心的三件套。学会它们，你的优化效率至少提升 10 倍。

Nsight Systems入门——从宏观鸟瞰

Nsight Systems（nsys）是你最先应该打开的 profiler。它的定位是宏观性能分析：它告诉你整个应用程序的时间花在了哪里，包括 CPU 端的 API 调用、内存拷贝、以及 GPU 内核的执行。你可以把它想象成整个应用的"交通摄像头"——它不关心内核内部的具体细节，但它能清晰地给出全局的时间线（timeline view）。

使用 Nsight Systems 非常简单。假设你有一个 CUDA 程序叫 my_app，只需要在命令行前加上 nsys profile：

# 基本用法：收集完整的时间线
nsys profile -o my_profile ./my_app

# 常用选项
nsys profile --stats=true -o my_profile ./my_app   # 输出统计摘要
nsys profile -t cuda,nvtx -o my_profile ./my_app   # 只追踪CUDA和NVTX事件
nsys profile -c cudaProfilerApi -o my_profile ./my_app  # 用cudaProfilerStart/Stop控制范围

运行完成后会生成一个 .nsys-rep 文件，用 Nsight Systems GUI（在 Windows 或 Linux 上都可以安装）打开它。你会看到：

  • CPU 端 timeline：显示你的主线程在做什么——调用 cudaMemcpy、启动内核、同步等待等
  • GPU 端 timeline：显示 GPU 端的实际执行情况——各个内核的持续时间和重叠情况、内存传输是否与计算重叠
  • Runtime API 和 Driver API 的调用频率和耗时：帮助你发现 API 调用本身是否成了瓶颈

看 timeline 的时候，重点关注三个信号：CPU 和 GPU 之间是否有长时间的"空白"（说明同步等待太多）、内存拷贝是否与内核执行重叠（理想情况下应该用 CUDA Stream 让它们重叠）、是否有不必要的同步点（如隐式的 cudaMemcpy 同步）。

实用技巧： 在代码中插入 NVTX 标记（nvtxRangePush() / nvtxRangePop()），可以让你在 timeline 上看到每个阶段的名称——比如"数据加载"、"前向传播"、"反向传播"。这对理解大型应用程序的性能至关重要。只需要包含 <nvtx3/nvToolsExt.h> 并链接 -lnvToolsExt 即可。

Nsight Compute入门——从微观深挖

如果说 Nsight Systems 是"交通摄像头"，那么 Nsight Compute（ncu）就是"引擎诊断仪"。它深入到每个 kernel 的内部，告诉你为什么这个 kernel 跑得不够快——包括占用率（occupancy）、内存带宽利用率（memory bandwidth utilization）、计算吞吐量（compute throughput）等关键指标。

基本用法也很直接：

# 对单个可执行程序进行kernel分析
ncu -o my_kernel_analysis ./my_app

# 常用flags
ncu --set full -o my_analysis ./my_app      # 收集所有metrics（最详细）
ncu --set basic -o my_analysis ./my_app     # 基本metrics（速度快）
ncu --kernel-name "my_kernel" -o my_analysis ./my_app  # 只分析特定kernel
ncu --launch-skip 3 --launch-count 2 ./my_app  # 跳过前3次启动，分析接下来2次
ncu --section MemoryWorkloadAnalysis ./my_app    # 只看内存相关指标
ncu --section SpeedOfLight ./my_app             # 看计算吞吐和带宽的"光速"对比

在 ncu 的 GUI 或 CLI 输出中，有几个核心指标要特别关注：

  • Occupancy（实际 / 理论）：如果实际 occupancy 远低于理论值，说明你的 kernel 被寄存器或共享内存限制了
  • Memory Throughput：接近设备理论带宽 → memory-bound；远低于 → 访问模式有问题
  • Compute Throughput：接近理论峰值 → compute-bound；远低于 → warp 发散或指令效率低
  • Sector/Wavefront 分析：帮助你发现全局内存的合并访问（coalesced access）情况
常见的"自我欺骗"： 很多人在初次使用 ncu 时，看到 Memory Throughput 达到了 60% 的理论带宽就觉得"已经够好了"。但请记住，对于许多 memory-bound kernel，优秀的目标是 80% 以上。60% 意味着你有一半的带宽潜力被浪费了——通常是因为非合并访问（unaligned / strided access）。

Roofline模型

Roofline 模型最早由 Berkeley 的 CS 系提出，是一种极其直观的可视化分析方法。它的核心思想很简单：一个 kernel 的性能上限由两个因素决定——计算能力和内存带宽，谁是瓶颈一目了然。

Roofline 图的 x 轴是操作强度（operational intensity，单位是 FLOPs/Byte），y 轴是性能（FLOPS）。图中有一条斜线（带宽限制区）和一条平线（计算限制区），形成一个"屋顶"形状。你的 kernel 落在哪个区域，就决定了它的优化策略：

kernel 在 Roofline 上的位置 含义 优化方向 典型场景
斜线区域（memory-bound） 性能受限于内存带宽 改进数据局部性、合并访问、使用共享内存 向量加法、Stencil、SpMV
平线区域（compute-bound） 性能受限于计算能力 减少指令数、使用 Tensor Cores、优化 ILP GEMM、卷积、FFT
天花板以下很远 有严重的实现缺陷 检查 occupancy、warp 发散、内存访问模式 任何未优化的 kernel

在 Nsight Compute 中，Roofline 分析是内置的。你只需要运行 ncu --set roofline ./my_app，GUI 中就会自动生成 Roofline 图。将你的 kernel 放到图上，一看便知它是被"算力天花板"还是"带宽天花板"挡住了。

Roofline 的最大的价值： 它帮你避免了"方向性错误"。假如你的 kernel 明显是 memory-bound，你去优化指令并行度（ILP）就是在错误的方向上使劲——你应该去优化数据访问模式。反之，如果 kernel 已经是 compute-bound，你再优化内存也没用，不如考虑用 Tensor Cores 或降低精度。Roofline 告诉你：不要跟错误的天花板打架。

Occupancy分析

Occupancy 的定义很简单：每个 SM 上活跃的 warp 数量 / SM 支持的最大 warp 数量。它的重要性在于：高 occupancy 能让 GPU 通过 warp 切换来隐藏延迟（latency hiding）。但这不代表 occupancy 越高越好——有时候低 occupancy 反而能跑得更快（比如每个线程需要大量寄存器时）。

影响 occupancy 的主要因素有三个，它们共同决定了每个 SM 能同时驻留多少个 warp：

  • 寄存器限制：每个线程使用的寄存器数量 × 每个 SM 想要驻留的线程数 必须 ≤ SM 的寄存器总量。A100 每个 SM 有 65536 个寄存器，如果你每个线程用 32 个寄存器，那么每个 SM 最多驻留 65536/32 = 2048 个线程（即 64 个 warp），occupancy 为 64/64 = 100%。但如果你用 64 个寄存器，就降到了 1024 个线程，occupancy 50%
  • 共享内存限制：每个 block 使用的共享内存 × 每个 SM 的 block 数 ≤ SM 共享内存总量。这是最常见的 occupancy 杀手——过多的共享内存会直接减少并发的 block 数量
  • Block 大小限制：每个 SM 最多 32 个 block（Maxwell 之后）、最多 64 个 warp。block 太小（如 64 线程/block）会导致每个 SM 能容纳的线程数受限

下面是一个计算 occupancy 的代码片段，你可以用 CUDA 自带的 occupancy API 来预测：

// CUDA Occupancy API —— 在运行时计算理论 occupancy
int min_grid_size, block_size;
// 为给定的内核函数和block_size计算最大occupancy
cudaOccupancyMaxPotentialBlockSize(
    &min_grid_size,      // 返回达到最大occupancy的最小grid size
    &block_size,         // 返回达到最大occupancy的block size
    my_kernel,           // 内核函数指针
    0,                   // 动态共享内存大小（字节）
    0                    // block大小建议（0 = 自动）
);

int max_active_blocks;
cudaOccupancyMaxActiveBlocksPerMultiprocessor(
    &max_active_blocks,  // 返回每个SM的活跃block数
    my_kernel,           // 内核函数指针
    block_size,          // block大小
    0                    // 动态共享内存大小（字节）
);

float occupancy = (max_active_blocks * block_size / 32.0f)
                  / _ConvertSMVer2Cores(dev_prop.major, dev_prop.minor) * 100.0f;

记住两件事：第一，理论 occupancy 不等于实际 occupancy——编译器优化、指令流水线等因素可能导致实际的 warp 调度不如预期；第二，目标 occupancy 因 kernel 而异——对于 memory-bound 的 kernel，高 occupancy 有助于隐藏延迟，一般是好事情；但对于 compute-bound 的 kernel，occupancy 的影响较小，瓶颈在计算单元本身。

一个实用的简单规则： 如果 kernel 是 memory-bound，尽量让 occupancy > 50%（让更多的 warp 可供切换来隐藏内存延迟）；如果 kernel 是 compute-bound，occupancy > 25% 通常就够了，因为计算单元本身不会因为 warp 不够而闲置。当然，这只是一个经验法则，具体情况要用 ncu 来验证。

NVIDIA Visual Profiler (nvvp) 与 nvprof

你可能在老旧文档或 StackOverflow 的老帖子里看到过 nvprof my_app 或 NVIDIA Visual Profiler（nvvp）。它们曾经是 CUDA 性能分析的标配工具，但现在已经被 Nsight Systems 和 Nsight Compute 完全取代。NVIDIA 在 CUDA 11.x 及后续版本中已经移除了 nvprof 和 nvvp，只提供 Nsight 系列工具。

不过，如果你仍然在使用较老的 CUDA 版本（如 CUDA 10.x），或者在一些 HPC 集群上遇到了只安装了旧版工具的情况，你可能还是会遇到它们。简单了解一下它们的定位就好：

  • nvprof：命令行 profiler，功能介于 nsys 和 ncu 之间——能做 timeline 分析也能看 kernel 的 metrics，但两者都不如 Nsight 工具深入。常用命令如 nvprof --metrics achieved_occupancy ./my_app
  • nvvp：GUI 版本的 profiler，提供可视化 timeline 和 kernel 分析。在 CUDA 10.x 及之前是主要的图形化分析工具

如果你在全新安装的 CUDA 12/13 上尝试运行 nvprof，你会发现它根本不存在了。遇到这种情况，请直接用 nsys 和 ncu。所有 nvprof 能做的，Nsight 工具做得更好。

迁移提醒： 如果你在网上搜索 CUDA 性能分析的相关问题，注意帖子的日期。2020 年之前的回答大概率在介绍 nvprof/nvvp，而 2022 年之后的应该都是 Nsight 工具。作为新学者，直接学 Nsight 系列即可，不必在老工具上浪费时间。

一个真实的教训

最后分享一个让我铭记至今的故事。几年前，我在优化一个流体模拟的 kernel。直觉告诉我，"这一定是 compute-bound 的——因为算法很复杂，计算量大"。于是我花了整整三周时间，做了各种指令级别的优化：减少除法、用 intrinsic、循环展开、甚至手写汇编级别的 PTX 调优。每次改动，性能提升都微乎其微——1%、2%、最多 5%。

直到第四周，我终于决定用 Nsight Compute 看一下 Roofline。结果让我目瞪口呆：这个 kernel 根本不在 compute-bound 区域，它完全被钉在 memory-bound 的天花板上。三个星期的指令级优化，相当于在带宽受限的马路上给汽车换更快的引擎——问题根本不在引擎上。如果我第一天就用 ncu 看一下 Roofline，5 分钟就能发现问题：全局内存访问模式是非合并的，每次读取都浪费了 75% 的带宽。改成合并访问后，不到两小时，性能提升了 4 倍。

这个教训让我从此变成了"profiler 优先"的信徒。每次开始优化一个新的 kernel，我的第一步永远是：ncu --set roofline ./my_app。5 分钟后，我就知道我应该往哪个方向努力了。

在下一章，我们将进入一个同样重要的领域——调试与错误处理。如果说 profiler 是你的"眼睛"，那调试工具就是你的"手术刀"。当你的内核跑出了错误的结果——或者干脆崩溃了——你需要知道如何精准地定位问题。准备好了吗？
