第5章 对称密码基础

大家好，我是你们的讲师。前面几章我们走过了从古典密码到香农信息论的完整理论路线：我们知道了替换和置换是最基本的操作，知道了混淆和扩散是设计原则，知道了完美的 OTP 虽然安全但不实用。现在，让我们进入现代密码学的第一个实践领域——对称密码。

对称密码这个名字来源于它的密钥使用方式：加密和解密使用同一个密钥。想象一下，你有一把锁和一把钥匙——用这把钥匙锁上箱子，也只能用同一把钥匙打开。这就是对称密码。它直观、高效、快速，是当今互联网上数据加密的主力军。无论你是在浏览 HTTPS 网站、发送 WhatsApp 消息还是使用网银，最终的批量数据加密都是通过对称密码完成的。

🎯 本章核心：对称密码分为两大类——分组密码（Block Cipher）和流密码（Stream Cipher）。我们将学习分组密码的两大经典结构（Feistel 网络和 SPN 结构），以及分组密码的多种工作模式。你还将看到为什么 ECB 模式被称为"密码学界的定时炸弹"。

5.1 分组密码（Block Cipher）与流密码（Stream Cipher）

对称密码根据处理数据的方式可以分为两大类：

分组密码将明文划分为固定大小的块（如 64 位或 128 位），对每个块用相同的密钥进行加密。你发送的明文长度不一定是块大小的整数倍——不足的部分需要进行填充（Padding）。

流密码则是将密钥扩展为任意长度的密钥流（keystream），然后将密钥流与明文逐位进行 XOR 操作。可以理解为：流密码试图在实用层面"模拟"一次性密码本——用伪随机生成器产生一个长的密钥流。

特性 分组密码 流密码
加密方式 将数据分成块，逐块加密 逐位或逐字节加密
典型代表 AES, DES, SM4 ChaCha20, RC4, Salsa20
适用场景 文件加密、网络数据包 实时通信、流媒体
硬件实现 易于硬件加速 软件实现更简单
错误传播 一个块错误影响本块及后续块 错误不传播（仅影响出错位）

值得一提的是，分组密码和流密码之间并不存在严格的界限。很多分组密码可以通过特定的工作模式（如 CTR 模式）被当作流密码使用——我们很快就会讲到。

5.2 Feistel 网络结构

在 1970 年代，IBM 的密码学家 Horst Feistel 提出了一个影响深远的密码结构——Feistel 网络。这种结构后来被 DES 采用，并成为设计分组密码的主流范式之一。

Feistel 网络的美妙之处在于它的简洁性和对称性：无论加密还是解密，都可以使用完全相同的硬件结构，只是密钥的使用顺序不同。下面是 Feistel 网络的一轮操作：

一轮 Feistel 加密：

1. 将输入块分成左右两半: L₀ 和 R₀
2. 计算: L₁ = R₀
3. 计算: R₁ = L₀ ⊕ F(R₀, K₁)
4. (L₁, R₁) 作为下一轮的输入

其中 F 是"轮函数"，K₁ 是第1轮子密钥

解密时只需反向操作，使用反向子密钥顺序：

1. R₀ = L₁
2. L₀ = R₁ ⊕ F(L₁, K₁)

这里有一个关键点：轮函数 F 不需要是可逆的。因为无论 F 是什么函数，上面的解密步骤都能正确还原——只需用 XOR 把 F 的结果去除即可。这意味着设计者可以大胆地使用任意复杂的非线性函数作为 F，而不必担心如何求逆。这是 Feistel 网络最聪明的设计决策。

💡 直观理解：Feistel 网络就像一个双层的组合保险柜。第一轮转动左轮，把它加密到右边；第二轮转动右轮，把它加密到左边。经过多轮交替，明文的统计结构被彻底打散。只要轮数足够多，即使轮函数本身不是很安全，整个结构仍然可以很安全。

Feistel 网络的关键参数包括：块大小（一般 64 或 128 位）、轮数（DES 用 16 轮，更多轮通常更安全但更慢）、子密钥生成算法（从主密钥派生出每轮的轮密钥）。

5.3 替代-置换网络（SPN）

与 Feistel 网络并列的另一大分组密码结构是替代-置换网络（Substitution-Permutation Network, SPN）。AES（Rijndael）就是基于 SPN 结构的。SPN 的结构比 Feistel 更直观——它在一轮中同时对整个数据块进行操作，而不是分成两半。

SPN 的每一轮包含三个步骤：

  • SubBytes（替换层）——使用 S-box（替换盒）对每个字节进行非线性替换，实现混淆
  • ShiftRows + MixColumns（置换层）——通过行移位和列混合将字节间的依赖关系扩散到整个块中，实现扩散
  • AddRoundKey（加轮密钥）——将当前状态与轮密钥进行 XOR，将密钥信息注入

这种"替换-置换-加密钥"的循环结构非常优雅地实现了香农的混淆和扩散原则：

一轮 SPN 的结构：

输入状态 (16 字节, 128 位)
    ↓
┌─────────────────┐
│  SubBytes        │  ← 非线性替换（混淆）
│  (S-box)         │
├─────────────────┤
│  ShiftRows       │
│  + MixColumns    │  ← 线性扩散（扩散）
├─────────────────┤
│  AddRoundKey     │  ← 注入密钥信息
└─────────────────┘
    ↓
输出状态 (进入下一轮)

SPN 的主要优势在于扩散速度更快：在 Feistel 网络中，一轮只改变一半的数据；而在 SPN 中，一轮就能影响整个数据块。但代价是，SPN 的解密过程与加密过程不同，需要单独设计解密电路。

5.4 分组密码的工作模式

分组密码只能加密固定长度的块（比如 AES 一次加密 128 位）。但实际中的数据可以是任意长度的。为了用分组密码加密任意长度的数据，我们需要定义工作模式（Mode of Operation）。

让我们来看几种最常见的工作模式：

ECB（电子密码本模式）——最简单也最危险。每个明文块被独立加密，相同的明文块会产生相同的密文块。

ECB 模式:
C₁ = Eₖ(P₁)
C₂ = Eₖ(P₂)
C₃ = Eₖ(P₃)
...

CBC（密码块链接模式）——引入了一个初始向量（IV），每个明文块在加密前先与前一个密文块进行 XOR。相同的明文块在不同位置会产生不同的密文块。

CBC 模式:
C₁ = Eₖ(P₁ ⊕ IV)
C₂ = Eₖ(P₂ ⊕ C₁)
C₃ = Eₖ(P₃ ⊕ C₂)
...

CTR（计数器模式）——将分组密码当作流密码使用。用一个递增的计数器配合密钥加密产生密钥流，再与明文进行 XOR。

CTR 模式:
C₁ = P₁ ⊕ Eₖ(CTR)
C₂ = P₂ ⊕ Eₖ(CTR+1)
C₃ = P₃ ⊕ Eₖ(CTR+2)
...

GCM（Galois/Counter 模式）——在 CTR 模式的基础上增加了认证标签（Authentication Tag），同时提供加密和完整性认证。这是目前最推荐的工作模式之一。

模式 并行加密 认证 推荐程度
ECB 是 否 ❌ 不要使用
CBC 否 否 ⚠️ 谨慎使用（需搭配 HMAC）
CTR 是 否 ⚠️ 需搭配 MAC
GCM 是 是 ✅ 强烈推荐
⚠️ 踩坑故事：ECB 模式与"Linux 企鹅" 这是密码学教学中最著名的演示之一。2004 年，一名安全研究员拍摄了一张被 ECB 模式加密后的 Tux 企鹅图片。原始图片有大面积的色块——深灰色背景、白色肚子、黄色脚掌。被 ECB 加密后，由于相同的像素块被加密成相同的密文块，加密后的图片仍然清晰地显示出一只企鹅的轮廓！这意味着加密后的图像看起来仍然是加密的（每个像素都是乱码），但整体轮廓没有隐藏。攻击者不需要解密，仅通过观察密文的模式就能判断出这是一只企鹅。这就是 ECB 模式的根本问题：它没有隐藏明文中的模式信息。在现代密码学中，ECB 模式几乎永远不是正确的选择——如果你在代码中看到有人使用 ECB，几乎可以确定是一个 bug。

5.5 填充与 Padding Oracle 攻击

分组密码要求明文长度是块大小的整数倍。当不是时，需要填充。最常见的填充方案是 PKCS#7：如果需要填充 n 个字节，就填充 n 个值为 n 的字节。例如，AES-128 的块大小是 16 字节，如果明文是 14 字节，则需要填充 2 个 0x02。

但填充本身可能引入安全漏洞。最著名的例子是 Padding Oracle Attack（填充预言攻击）。

攻击过程：在 CBC 模式下，如果服务器在解密时反馈"填充错误"或"填充正确"这两种不同的响应，攻击者就可以通过逐个修改 IV 或前一个密文块来逐步推断出明文。这种攻击不需要暴力破解密钥，只需要向服务器发送精心构造的密文并从错误消息中获取信息。

Padding Oracle Attack 的基本思路：

1. 修改密文块 Cᵢ 的最后一个字节
2. 发送修改后的密文给服务器解密
3. 如果返回"填充正确"，说明猜测的字节导致了合法的填充
4. 利用这个信息推导出明文的最后一个字节
5. 继续猜测下一个字节，直到整个块被还原

这个攻击在 2010 年代早期非常流行——当时很多 ASP.NET 和 Java 应用都使用 CBC 模式并对填充错误给出详细的错误提示。教训是：永远不要让攻击者区分"填充错误"和"解密失败"。这就是为什么 GCM 等认证加密模式（Authenticated Encryption, AEAD）逐渐成为标准——它们同时提供加密和认证，在解密前先验证完整性，不给 Padding Oracle 任何机会。

5.6 对称密码的安全使用原则

综合上面的内容，我们可以总结出对称密码的安全使用原则：

  • 永远不要使用 ECB 模式——没有任何理由在新系统中使用 ECB
  • 总是使用认证加密（AEAD）——优先选择 GCM、ChaCha20-Poly1305 等同时提供加密和认证的模式
  • IV/Nonce 不能重复——特别是 CTR 模式和 GCM 模式下，IV 重复会导致灾难性后果（密钥流相同）
  • 密钥必须足够随机——使用安全的随机数生成器生成密钥，不要用用户密码直接作为密钥（用 KDF 派生）
  • 实现中的侧信道——注意比较密钥时使用 constant-time 比较，防止 timing attack
