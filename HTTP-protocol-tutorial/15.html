第十五章：HTTPS概述

各位同学好。咱们这个教程前面十几章都在讲HTTP——它的报文、方法、状态码、缓存、Cookie、CORS等等。但大家心里可能一直有个疑问：前面讲的这些都是明文传输的，那真正的线上网站不是都用HTTPS吗？没错，今天这一章我们就来补齐这个至关重要的拼图——HTTPS。

我从一个真实的故事开始。2015年的时候，我帮一家创业公司排查一个诡异的Bug。用户反馈在咖啡馆连接公共WiFi后，访问他们的网站会出现"账号异常"的提示。开始我们以为是程序Bug，排查了整整两周。最后发现——咖啡馆的WiFi被黑客做了中间人攻击（MITM），所有HTTP流量都被篡改了。因为网站当时只用了HTTP，用户登录时密码明文经过黑客的代理服务器，直接被截获。这就是没有HTTPS的代价。

核心认知：HTTPS = HTTP + TLS（传输层安全协议）。它不是一套新的应用层协议，而是在HTTP和TCP之间加了一层安全层。这一层负责加密、身份验证和完整性校验。换句话说，HTTP管的是"怎么说话"，TLS管的是"说话内容不被偷听"。

15.1 HTTP明文传输的风险

很多人觉得"我的网站没什么敏感信息，用HTTP就行"。这个想法非常危险。HTTP明文传输面临三大风险，安全领域称之为"CIA三元组被破坏"——不过这里不是中央情报局，而是机密性、完整性、可用性。

窃听（Eavesdropping）：HTTP报文在网络上传输时，沿途任何一个路由节点、中间代理都能看到完整的报文内容。你在咖啡厅连公共WiFi，同一个网络里的人用Wireshark抓个包，就能看到你访问了什么网站、发了什么数据。你以为HTTP POST请求里密码字段经过了MD5加密？错了，MD5是哈希，而且很多网站当年真就是明文传密码的。

篡改（Tampering）：攻击者不仅能看，还能改。中间人可以把你的请求内容替换掉。比如你在下载一个软件的安装包，HTTP连接被中间人篡改后，你下载的可能是带木马的程序。这就是为什么包管理器（如apt、npm）都会有校验和——但如果你连校验和页面都是HTTP加载的，那就陷入了"谁来保护保护者"的死循环。

冒充（Impersonation）：你访问 www.your-bank.com，但你确定你连接的那台服务器真的是你的银行吗？没有HTTPS，DNS劫持、ARP欺骗等手段可以让你的流量被引向一个假冒的服务器。你输入账号密码，那边照单全收。

真实案例：2017年，一个名为"EvilGinx"的开源工具在GitHub上发布。这个工具专门用于HTTPS中间人攻击——它会在目标用户和真实网站之间建立一个代理，当用户访问登录页时，EvilGinx同时代理了HTTP和HTTPS流量，截获凭证后转发给真实网站。用户以为自己输了正确密码登录成功，其实凭证已经被窃取。这个案例说明：即使是HTTPS，如果用户不注意证书警告，也可能会受骗。但至少HTTPS把攻击门槛提高了非常多——没有HTTPS的话，连EvilGinx这种工具都不需要，随便一个抓包软件就行。

总结一下，HTTP在公共网络上传输就像在明信片上写情书——每个经过的邮递员都能看。HTTPS则是在信封里写情书，只有收件人能打开。

15.2 加密基础

要理解HTTPS怎么工作，先得搞懂两种加密方式：对称加密和非对称加密。这两个概念是TLS的基石。

对称加密：加密和解密用同一把密钥。就像你有一个保险箱，用一把钥匙锁上，同一把钥匙打开。优点是速度快（硬件加速后性能极佳），缺点是密钥怎么安全地传给对方？你如果在网上传密钥，密钥本身又不加密，那跟没加密一样。

常见的对称加密算法：

算法密钥长度状态说明
AES128 / 192 / 256 位推荐目前最广泛使用的对称加密，硬件加速支持好
ChaCha20256 位推荐流密码，在移动设备上性能优于AES
DES56 位已废弃密钥太短，暴力破解易如反掌
3DES112 位已废弃DES的改进版，但速度慢且有安全漏洞

非对称加密：加密和解密用不同的密钥——公钥和私钥。公钥可以公开给任何人，私钥必须保密。用公钥加密的数据只有对应的私钥能解密。这解决了密钥分发问题——我直接把公钥给你，你用公钥加密消息，只有我能用私钥解密。但缺点是速度比对称加密慢几个数量级（RSA解密比AES慢几百倍甚至上千倍）。

常见的非对称加密算法：

算法特点状态
RSA基于大整数因数分解，经典但密钥较长仍广泛使用，但逐渐被取代
ECDSA基于椭圆曲线，同安全等级下密钥更短推荐（TLS 1.3首选）
Ed25519基于Curve25519，速度快、安全性高现代推荐

关键理解：HTTPS不是"只用非对称加密"或"只用对称加密"，而是两者结合——这就是混合加密。非对称加密用于安全地交换对称密钥（称为"会话密钥"），后续通信全部使用对称加密。这样既解决了密钥分发问题（用非对称加密），又保证了传输效率（用对称加密）。用个形象的比喻：非对称加密是"保险箱"，对称加密是"快递箱"。你用保险箱把快递箱的钥匙安全地寄给对方，之后双方用快递箱批量运送货物。

15.3 TLS握手过程

TLS握手是整个HTTPS通信中最复杂、也最核心的部分。我们以最常用的TLS 1.2握手为例，一步步拆解。我先用文字描述流程，再看每一步具体发生了什么。

TCP三次握手之后：

客户端                              服务端
  |                                    |
  |-------- ClientHello ------------>  |  步骤1：客户端打招呼
  |                                    |  发送支持的TLS版本、密码套件列表、
  |                                    |  随机数（Client Random）
  |                                    |
  |<------- ServerHello ------------- |  步骤2：服务器回应
  |                                    |  选定TLS版本、密码套件、
  |                                    |  随机数（Server Random）
  |                                    |
  |<------ Certificate -------------  |  步骤3：服务器发送证书
  |                                    |  包含公钥、域名、CA签名
  |                                    |
  |<--- ServerHelloDone ------------   |  步骤4：服务器"我说完了"
  |                                    |
  |-------- ClientKeyExchange ------>  |  步骤5：客户端生成Pre-Master Secret
  |                                    |  用服务器公钥加密后发送
  |                                    |
  |-------- ChangeCipherSpec ------>   |  步骤6：告知后续加密通信
  |-------- Finished --------------->  |
  |                                    |
  |<----- ChangeCipherSpec ---------   |  步骤7：服务器确认
  |<----- Finished ------------------  |  双方开始加密通信
  |                                    |
  |=========== 加密通信 =============> |
  |                                    |

现在我们来详细拆解每个步骤：

步骤1：ClientHello —— 客户端发起握手，告诉服务器它支持的TLS版本（比如TLS 1.2、1.3）、密码套件列表（比如哪些加密算法、哈希算法）、以及一个客户端生成的随机数（Client Random）。这个随机数后面会被用来生成会话密钥。

步骤2：ServerHello —— 服务器从客户端支持的列表中选定一个TLS版本和密码套件，也生成一个服务器随机数（Server Random）发给客户端。

步骤3：服务器证书（Certificate） —— 服务器把它的数字证书发给客户端。证书里包括服务器的域名、公钥、CA（证书颁发机构）的签名等。这是身份验证的关键一步——客户端需要验证这个证书是不是可信的。

步骤4：ServerHelloDone —— 服务器的初始消息发完了，告诉客户端"你可以开始你的回合了"。

步骤5：ClientKeyExchange —— 客户端生成一个Pre-Master Secret（另一个随机数），用服务器的公钥加密后发送给服务器。只有拥有对应私钥的服务器才能解密。此时，客户端和服务器都有了三个随机数：Client Random、Server Random、Pre-Master Secret。三者结合计算出Master Secret，再用Master Secret派生出实际用于对称加密的会话密钥。

步骤6：ChangeCipherSpec + Finished —— 客户端告诉服务器"从现在开始我要加密通信了"，然后发送一个Finished消息（用协商好的会话密钥加密之前所有握手消息的摘要）。服务器解密并验证这个Finished消息，确保之前的握手没有被篡改。

步骤7：服务器的ChangeCipherSpec + Finished —— 服务器做同样的操作，告诉客户端要开始加密了，并发送加密的Finished消息。客户端验证成功后，握手完成。

重要概念：握手的本质是认证 + 密钥协商两个过程。认证靠证书（证明服务端的身份），密钥协商靠随机数+非对称加密（确保只有合法双方能算出相同的会话密钥）。握手完成后的所有HTTP通信都使用这个会话密钥进行对称加密，效率极高。

15.4 证书信任链

刚才握手过程中，服务器发了一个数字证书。客户端怎么知道这个证书是可信的？这个信任不是凭空来的，它基于一个叫做PKI（公钥基础设施）的信任体系。

我们来看证书信任链的结构：

根证书（Root CA）
   ┃ 自签名，内置在操作系统/浏览器中
   ┃
   ▼
中间证书（Intermediate CA）
   ┃ 由根证书签发
   ┃
   ▼
服务器证书（Server Certificate）
   ┃ 由中间证书签发
   ┃ 绑定了你的域名（example.com）
   ▼
你的浏览器

根证书（Root CA）：这是信任的起点。操作系统和浏览器内置了一批根证书——比如DigiCert、GlobalSign、Let's Encrypt的根证书。根证书是自签名的（自己签自己），所以它们本身就是信任的锚点。全球大约有100多个受信任的根证书。

中间证书（Intermediate CA）：根证书太重要了，如果根证书的私钥泄露，整个信任体系就崩塌了。所以实际操作中，根证书很少直接签发服务器证书，而是先签发一个或多个"中间证书"，再由中间证书签发服务器证书。这样即使中间证书的私钥泄露，可以吊销它，根证书不动——不影响整个体系。

服务器证书：这是你的网站实际使用的证书。里面包含了域名（Common Name或SAN）、公钥、颁发者信息、有效期等信息。

浏览器验证证书的过程：

  1. 检查服务器证书的颁发者——找到签发它的中间证书
  2. 检查中间证书的颁发者——找到根证书
  3. 确认根证书在操作系统的受信任根证书列表中
  4. 逐级验证签名：根证书签名中间证书、中间证书签名服务器证书
  5. 检查服务器证书的域名是否匹配你访问的域名
  6. 检查证书是否在有效期内
  7. 检查证书是否被吊销（CRL或OCSP）

如果以上任何一步失败，浏览器就会显示安全警告——比如Chrome的"您的连接不是私密连接"页面（NET::ERR_CERT_AUTHORITY_INVALID）。

开发小贴士：你在开发环境访问 https://localhost 时经常看到证书警告，这是因为 localhost 的证书是自己签发的、没有被OS信任。你可以在本地生成自签名证书并添加到信任库，或者用 mkcert 这样的工具一键生成本地受信的证书。建议开发阶段就用HTTPS，避免上线后才发现HTTPS相关的问题。

15.5 HTTPS的误解

关于HTTPS，行业内流传着不少误解。我挑几个最常见的来澄清。

误解一："HTTPS很慢" —— 这个说法在10年前还有一定道理，但今天基本不成立了。TLS握手确实比纯TCP握手多了几个往返（RTT），但现代技术已经大幅缩小了这个差距：

  • TLS 1.3把握手从2-RTT降到了1-RTT（甚至0-RTT）
  • 会话复用（Session Resumption）可以跳过完整握手
  • 现代硬件（AES-NI指令集）让加解密几乎零开销

实际上，很多CDN和云服务厂商的数据显示，启用HTTPS后的性能差异在5%以内，而且HTTP/2强制要求使用HTTPS——而HTTP/2的多路复用性能收益远超这点开销。所以"HTTPS慢"是一个典型的以偏概全的误解。

误解二："只有登录页面需要HTTPS" —— 这是非常危险的想法。即使是一个展示型网站，没有用户输入，HTTPS也是必要的：

  • 中间人可以篡改页面内容——插入恶意广告、钓鱼脚本
  • HTTP页面中的"小绿锁"不仅仅保护用户数据，也保护内容的完整性
  • 浏览器对HTTP页面的功能有限制——许多Web API（如地理位置、Service Worker）要求安全上下文（Secure Context，即HTTPS或localhost）
  • SEO影响——Google将HTTPS作为排名信号

误解三："HTTPS能防所有攻击" —— HTTPS保护的是传输层安全，它不保护：

  • 服务器端的安全漏洞（SQL注入、XSS、CSRF）
  • 客户端被植入木马/恶意软件
  • DNS劫持（如果你通过HTTP获取IP，HTTPS也救不了——这就是HSTS存在的意义）
  • 社会工程学攻击（钓鱼网站可以用HTTPS——有证书不代表网站是合法的）

HTTPS只是安全体系中的一环，不是万能药。一个常见的攻击手法：攻击者先注册一个钓鱼域名（比如 g00gle.com），然后申请免费的HTTPS证书（Let's Encrypt），用户看到小绿锁就放松了警惕。记住：HTTPS保证的是"你和真正的g00gle.com在加密通信"，而不是保证"g00gle.com是合法的Google"。

踩坑故事：几年前我接手过一个遗留系统，当时开发团队认为"内网系统不需要HTTPS"，所有内部API都是HTTP。结果有一天，IT部门在交换机上配置端口镜像做流量审计，无意中发现某个内网系统的API在明文传输管理员密码。更可怕的是，那台交换机和办公网是直连的——意味着任何连上公司WiFi的员工都能抓包看到管理密码。这就是典型的"内网安全"幻觉。实际上，内网攻击是最常见的攻击来源之一（insider threat）。我们后来花了一周把所有内部服务全切到HTTPS，包括数据库连接都加了TLS。

好了，第十五章的内容就到这里。我们从HTTP的风险讲起，到加密基础、TLS握手、证书信任链，最后澄清了HTTPS的几个常见误解。下一章我们要深入TLS内部，看看TLS 1.2和1.3到底有什么区别、密码套件怎么选择、以及如何优化TLS性能。
