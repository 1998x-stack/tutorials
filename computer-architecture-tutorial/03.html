第3章 数据表示与运算

同学们好！上一章我们学习了支撑计算机的砖石——逻辑门和触发器。但是，有了这些硬件后，计算机怎样"理解"数字呢？当你写下 int a = -42; 或 double pi = 3.14159; 时，这些数字在硬件层面是如何表示的？为什么有些运算结果看起来"不对"？

数字的语言就是计算机的语言。如何表示数字，决定了从计算精度到功耗的方方面面。今天我们就来深入探讨这个看似简单但充满陷阱的领域。

3.1 整数的二进制表示

计算机内部所有信息都以二进制表示。对于整数，历史上出现过四种表示方式：

无符号数（Unsigned）：直接二进制编码，n位表示范围0到2n-1。例如8位无符号数的范围是0到255。

原码（Sign-Magnitude）：最高位为符号位（0正1负），其余位表示数值的绝对值。问题是：+0和-0有两种表示，且加减运算需要单独处理符号位，硬件实现复杂。

反码（One's Complement）：负数是正数按位取反。同样有正负零的问题。反码的加减需要"循环进位"（end-around carry），非常麻烦。

补码（Two's Complement）：负数是正数取反加一。这是现代计算机选择的方案，原因很简单：

  • 0只有一种表示（全0）。
  • 减法可以统一为加法：A - B = A + (-B)，其中-B就是B的补码。
  • n位补码的范围：-2n-1 到 2n-1-1。对于32位，就是-2,147,483,648 到 2,147,483,647。

补码的神奇之处在于：对一个数的补码再求一次补码，就得到了原数。而且扩展位宽时，只需要进行符号扩展（sign extension）——将最高位复制到所有新增的高位。例如，-3的4位补码是1101，扩展到8位就是11111101。

一个有趣的练习：在32位补码中，最小的负数-2,147,483,648（0x80000000）的负数是多少？答案是它自己！因为正数2,147,483,648超出了32位补码的表示范围。这就是所谓的"不对称范围"，是补码的一个重要特性。

3.2 整数运算与ALU

有了补码表示，加法器就可以同时处理加法和减法（被减数取反加一送入加法器）。但还有一个重要问题需要处理：溢出（Overflow）。

溢出检测规则：

  • 正数+正数=负数 → 上溢（overflow）。
  • 负数+负数=正数 → 下溢（underflow）。
  • 正数+负数 → 不可能溢出。

硬件上，可以通过比较进位输入和进位输出来检测：如果最高位的进位输入不等于进位输出，则发生了溢出。

乘法：经典算法是"移位相加"（shift-and-add），类似于十进制乘法的手算过程。将乘数的每一位从低到高检查，如果是1，将被加数左移对应位数后累加。现代处理器中，乘法器通常使用Booth编码或华莱士树（Wallace Tree）来提高效率。

除法：比乘法更复杂，也慢得多。恢复余数法和非恢复余数法是两种经典算法。这也是为什么在很多应用中，除以一个常数会被编译器优化为乘以其倒数再加一些修正操作。

ALU（算术逻辑单元）的设计非常优雅：一个1位的ALU slice包含加法器、AND/OR/XOR门和一个多路选择器，通过opcode选择运算类型。将32个这样的slice级联，就得到了32位ALU。每个slice的carry-out接到下一个slice的carry-in。控制信号决定是加法还是减法（对B输入取反），以及选择哪种逻辑运算。

下面是典型ALU的操作码表：

操作码 功能 描述
0000A AND B按位与
0001A OR B按位或
0010A + B加法
0011A - B减法（B取反加一）
0100A XOR B按位异或
0101A << B左移
0110A >> B (逻辑)逻辑右移（高位补0）
0111A >> B (算术)算术右移（高位补符号位）
1000SLT若A<B则置1，否则置0

这个精简单片ALU的设计思想是处理器数据通路的基础，我们在第8章设计单周期处理器时会再次遇到它。

3.3 浮点数与IEEE 754

整数很好用，但现实中我们需要表示非整数（如3.14）和非常大的数（如光速3×108 m/s）或非常小的数（如普朗克常量6.626×10-34）。这就是浮点数（floating-point number）的用武之地。

浮点数的核心思想是科学记数法：把一个数表示为 ± significand × baseexponent。在二进制中：± significand × 2exponent。

IEEE 754 标准规定了两种常用精度：

精度 总位数 符号位 指数位 尾数位 指数偏移 数值范围
单精度（float）321823127±1.18×10-38 到 ±3.40×1038
双精度（double）64111521023±2.23×10-308 到 ±1.80×10308

IEEE 754的精妙之处在于它的设计细节：

  • 隐含前导1：除了特殊值外，尾数的整数部分总是1（因为是"规范化"表示），所以不存储这个1，省出1位精度。
  • 偏移指数：指数使用"偏移"（bias）编码，单精度偏移127，双精度偏移1023。这使得指数可以表示为无符号数，简化了比较。
  • 特殊值：指数全1、尾数全0表示无穷大（正负）；指数全1、尾数非0表示NaN（Not a Number）；指数全0、尾数全0表示0（正负）；指数全0、尾数非0表示非规范数（denormalized number），用于表示接近0的极小数值。

⚠️ 历史教训：Ariane 5火箭爆炸（1996年）

1996年6月4日，欧洲航天局Ariane 5火箭在首飞37秒后爆炸，直接损失约3.7亿美元。调查发现，罪魁祸首竟然是一个浮点数溢出！火箭的惯性导航系统沿用了Ariane 4的软件，其中将一个64位浮点数转换为16位有符号整数时，数值过大导致溢出——而异常处理代码恰好把它当作"诊断信息"丢弃了，没有进行任何错误恢复。主备两个惯性导航单元在同一时刻以相同方式失效，火箭失去了方向控制。这个故事告诉我们三个教训：（1）浮点转整数的溢出后果可能极其严重；（2）不要把关键系统的异常处理留给"诊断"代码；（3）复用软件时一定要验证输入范围。这类错误在桌面应用中可能只是程序崩溃，但在航天系统中就是数亿美元的损失。

3.4 浮点运算的精度陷阱

浮点数最反直觉的特性是：基本的算术定律不一定成立。具体来说，浮点加法不满足结合律：(a + b) + c 不一定等于 a + (b + c)。这是因为浮点数的精度有限，运算过程中会对中间结果进行舍入。

下面这段C代码展示了经典的"灾难性抵消"（catastrophic cancellation）现象：

#include <stdio.h>
#include <math.h>

int main() {
    // 演示浮点结合律失效
    float a = 1.0f;
    float b = 1.0e-8f;
    float c = -1.0e-8f;

    float result1 = (a + b) + c;  // 1.0 + 1e-8 ≈ 1.0（精度丢失）→ 再减去1e-8 = 1.0
    float result2 = a + (b + c);  // 1e-8 + (-1e-8) = 0 → 1.0 + 0 = 1.0

    printf("(a + b) + c = %.15f\n", result1);
    printf("a + (b + c) = %.15f\n", result2);

    // 演示灾难性抵消
    double x = 1.0000000000000001;
    double y = 1.0000000000000000;
    printf("x - y = %.16f\n", x - y);  // 期望1e-16，实际上消除掉了大部分有效位

    // Kahan求和算法
    double vals[4] = {1.0, 1e100, 1.0, -1e100};
    double naive_sum = vals[0] + vals[1] + vals[2] + vals[3];
    printf("Naive sum = %f\n", naive_sum);  // 大数吃掉小数

    // Kahan求和
    double sum = 0.0, err = 0.0;
    for (int i = 0; i < 4; i++) {
        double y = vals[i] - err;
        double t = sum + y;
        err = (t - sum) - y;
        sum = t;
    }
    printf("Kahan sum = %f\n", sum);  // 正确结果应该是2.0
    return 0;
}

灾难性抵消发生在两个接近的数值相减时。例如，在二次方程求根公式中，如果 b2 远大于 4ac，那么 -b + sqrt(b2-4ac) 就会导致两个接近的大数相减，有效位数急剧损失。

💡 实践经验：我曾经在生产环境中遇到过一个ML训练任务，每轮loss看起来在下降，但模型就是不收敛。调试了两天，最后发现是一个数值稳定性问题——softmax+交叉熵的实现中，对非常小的logit值取了exp后出现了underflow，导致梯度传播完全消失。改用log-sum-exp技巧后模型正常收敛。从此我学到一个教训：数值稳定性不是"锦上添花"，而是工程落地的必要条件。

Kahan求和算法通过维护一个"补偿项"来保留加法中丢失的低位信息。虽然不能完全消除误差，但在大量浮点数求和时能显著提高精度——从O(n)的误差累积降到O(1)。

3.5 超越整数与浮点

除了整数和浮点数，还有其他数据表示方式：

定点数（Fixed-Point）：约定小数点固定在某个位置，本质上是整数乘以一个缩放因子。在DSP和嵌入式系统中广泛使用，因为没有浮点运算单元（FPU）的芯片上，定点数的运算速度远快于软件模拟的浮点。例如，Q15格式将16位数约定为-1到1之间的小数，小数点在第15位之后。

BCD（Binary-Coded Decimal）：每个十进制数字用4位二进制表示。虽然浪费空间（4位只能表示0-9，浪费了6种组合），但适合金融计算（可以精确表示0.1这样的十进制小数，而浮点数不行）。早期的COBOL语言和银行系统广泛使用BCD。

任意精度整数（BigInt / Arbitrary Precision）：用多个字（word）拼接表示任意大小的整数。Python的int、Java的BigInteger、Rust的num::BigInt都基于这个思想。代价是运算速度远低于硬件整数。

理解了数据是如何表示的，我们终于可以进入计算机体系结构中最核心的抽象层——指令集架构（ISA）。ISA定义了处理器能够理解和执行的所有指令，是软件与硬件之间的分界线。下一章，我们将探讨RISC与CISC之间持续了四十年的争论，以及为什么RISC-V是当下最适合学习的ISA。
