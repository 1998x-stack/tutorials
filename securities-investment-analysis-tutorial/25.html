第25章 实盘交易系统

大家好，我是你们的讲师。走到这一章，说明你已经完成了一个完整的量化策略从数据获取到回测评估的全流程。下一步是什么？实盘。

但我想先给你泼一盆冷水：从回测到实盘，难度不是一个量级的。回测里一切都很完美——数据是历史的，成交是即时的，价格是收盘价，没有网络延迟，没有交易失败。实盘里呢？你的API可能断连，订单可能被拒，股票可能突然停牌，费率可能调整，分红除权可能让你的持仓市值凭空消失一大块。这一章我们会把所有这些"实盘才知道的事"过一遍。

25.1 从回测到实盘：十倍难度

在量化圈有一句被反复验证的话："任何策略在实盘中的表现，都不会超过回测。" 如果你的回测年化收益是30%，实盘中能拿到15%-20%就已经是非常好的结果了。为什么差距这么大？因为回测无法模拟以下几个方面：

  • 网络延迟：从你的策略发出信号到券商收到指令，中间有几十到几百毫秒的延迟。对于高频策略，这个延迟足以吃掉全部利润。
  • 交易失败：订单可能因为各种原因被拒——资金不足、股票停牌、涨停买不进、跌停卖不出、价格偏离过大等。
  • 涨跌停与停牌：回测里不会遇到"买不到"或"卖不掉"的情况，实盘里天天都有。
  • 数据延迟：你看到的"实时行情"实际上有1-3秒的延迟（Level-1行情），甚至更低频的延迟。
  • 费率变化：印花税、佣金、过户费都可能调整。2023年8月，中国证监会将印花税从0.1%降至0.05%，这是实打实的成本变化。
  • 分红除权：如果你忘记处理除权除息，持仓市值会在除权日自动下跌——不是市场下跌，而是因为公司分红导致股价做了调整。

踩坑故事：实盘第一天，账户跌了4%

2022年，我把一个跑了半年回测、夏普比率1.8的策略部署到了实盘。模拟盘跑了两个月一切正常，于是我信心满满地充了50万进去。结果实盘第一天收盘，账户显示亏损4%。

我当时心态就崩了——策略逻辑没问题，行情也没大跌，钱去哪了？

仔细查了半天，发现是除权除息。那天刚好是我持仓的一只股票的分红除权日。公司在当天进行了每10股派10元的分红，股价在开盘前自动下调了除权金额。回测中我用了前复权数据，价格是调整后的连续价格；但实盘账户里的股票是按真实价格计算的——除权日股价"跳空下跌"，持仓市值直接缩水了分红部分。

更惨的是：我没有在策略中把分红再投资。分红金额以现金形式回到了账户，但我的策略代码只盯着持仓市值看，没把现金部分算进去——所以显示的"账户盈亏"是"-4%"。后来我在回测代码里加了一个dividend_reinvest=True的选项，专门模拟分红再投资。这个参数让我对策略的真实收益有了更准确的认识。

教训：回测中的"收益"不等于实盘中的"钱"。两者之间隔着一整个"现实世界"的距离。

25.2 券商量化平台入门

个人量化投资者想做程序化交易，目前主要有两条路：一是使用券商量化平台提供的API，二是使用第三方量化平台（如聚宽、米筐）的仿真交易功能。

国内主流的券商量化平台有以下几种：

平台 开发商 部署方式 支持券商 编程语言 门槛
QMT 迅投 本地客户端 国金、国信、中金等30+ Python 中等（需开通券商QMT权限）
Ptrade 恒生电子 本地客户端 华泰、中信建投、招商等 Python 中等（资金门槛30-50万）
BigQuant BigQuant 云端 仿真交易为主 Python/可视化 低（免代码搭建策略）
聚宽 (JoinQuant) 聚宽 云端/本地 仿真+部分券商实盘 Python 低
米筐 (RiceQuant) 米筐科技 云端 仿真交易为主 Python 低

如果你认真的想把量化作为长期事业，推荐使用QMT或Ptrade——它们提供本地化部署的专用客户端，可以直接对接券商柜台进行实盘交易，延迟低、可控性强。聚宽和米筐更适合策略研究和仿真验证阶段。

QMT下单代码示例

# QMT (迅投) 下单接口示例
# 注意：需要在QMT客户端中运行，不是独立的Python脚本

from xtquant import xtdata
from xtquant import xttrader
from xtquant.xttype import StockAccount

# 连接交易接口
acc = StockAccount('您的资金账户', '券商ID')
trader = xttrader.XtQuantTrader('您的QMT客户端路径')
trader.start()
trader.connect()

# 获取实时行情
xtdata.subscribe_quote('000001.SZ', period='1d')
quote = xtdata.get_market_data(['close', 'open', 'high', 'low', 'volume'],
                                ['000001.SZ'], period='1d')

# 下单买入
order_id = trader.order_stock(
    acc,
    stock_code='000001.SZ',
    order_type=23,       # 23=买入，24=卖出
    order_volume=1000,   # 买入1000股
    price_type=5,        # 5=市价单，11=限价单
    price=0              # 市价单价格填0
)

print(f"委托单号: {order_id}")

# 查询成交
trade_list = trader.query_trades()
for t in trade_list:
    print(f"成交: {t.stock_code} {t.traded_amount}股 @ {t.traded_price}")

# 断开连接
trader.stop()

重要提示：上面的代码仅为示例，实际使用QMT需要先在合作的券商处开通量化交易权限，下载安装QMT客户端，然后登录资金账户。不同券商的接口版本和参数略有差异，以对应券商的开发文档为准。

25.3 程序化交易监管新规（2025）

2024年到2025年，中国证监会对程序化交易的监管框架进行了重大升级。简单总结几个关键点：

申报速率限制——高频交易策略受到重点监管。单个账户的报撤单速率被严格限制，如果超过阈值（比如每秒300笔报撤单），交易所会采取限制交易等措施。这意味着高频策略（尤其是T+0回转交易）的空间被大幅压缩。

报告制度——开展程序化交易的投资者需要向交易所报告策略的基本信息，包括交易策略类型、服务器位置、交易软件信息等。之前那种"偷偷跑程序化"的时代已经结束了。

流量费——高报撤单的账户需要支付额外的流量费用。这个费率虽然不是很高，但对于高频策略来说，也是一笔不可忽视的固定成本。

这些监管要求对个人量化投资者的核心影响是：合规是第一位的。 在部署实盘之前，务必先跟你的券商确认程序化交易的报备流程，不要"先斩后奏"——轻则被限制交易，重则被认定为违规交易行为。

25.4 交易成本模型

回测中，很多人只用"万分之一佣金"这样的静态参数来计算交易成本。实盘中交易成本远比想象的复杂：

印花税——A股卖出时收取，目前为成交金额的0.05%（2023年8月从0.1%下调）。单向收取（仅在卖出时征收）。

佣金——买卖双向收取，一般在0.01%-0.03%之间。最低收费5元（部分券商已取消最低5元限制）。如果你交易频繁，佣金是一笔很大的隐性支出。

滑点（Slippage）——你看到的价格和实际成交价格之间的差异。对于流动性好的股票（如贵州茅台），滑点可能在0.01%以内；对于流动性差的股票（如日成交额只有几百万的小盘股），滑点可能高达0.5%-1%。

冲击成本（Impact Cost）——大单交易对市场价格造成的影响。如果你想一次性买入某只股票100万股，你的买入行为本身就会推高股价，导致平均成交价高于你的预期。冲击成本与交易量的平方根成正比，是机构投资者最主要的交易成本之一。

实盘建议：在回测时至少留出0.1%-0.3%的单边摩擦成本（含佣金+滑点）。如果你的策略年化收益只有5%，摩擦成本就能吃掉你一半以上的利润。所以高频低收益的策略在实盘中几乎没有生存空间——赚的钱全贡献给券商和交易所了。

25.5 实盘监控与风控体系

实盘和回测最大的区别是什么？回测出问题了你可以重来，实盘出问题了亏的是真金白银。所以实盘必须有风控体系。

一个最小可行的实盘风控体系至少包括三个层面：

实时风控——在交易执行层做硬性限制。单日亏损超过一定比例（比如-3%）时自动停止所有交易；单只股票的最大持仓集中度不超过总资产的20%；设置"黑名单"机制，手动将问题股票加入禁止交易列表。

异常告警——策略运行异常时自动通知。推荐使用钉钉机器人或企业微信Webhook推送告警消息——比邮件快得多。监控指标包括：策略连续亏损N天、成交价格偏离信号价格超过阈值、API连接断开超过X分钟等。

盘后复盘——每天收盘后自动生成交易日志，记录每笔交易的执行情况（委托时间、成交时间、滑点大小、延迟等）。这些数据是持续优化策略的基础。

# 钉钉告警推送示例
import requests
import json

def send_dingtalk_alert(message, webhook_url):
    """通过钉钉机器人发送告警消息"""
    headers = {'Content-Type': 'application/json'}
    data = {
        "msgtype": "text",
        "text": {
            "content": f"[量化风控] {message}"
        }
    }
    resp = requests.post(webhook_url, headers=headers,
                         data=json.dumps(data))
    return resp.status_code == 200

最后一句话：实盘交易系统是一个"防呆"的系统——它在你的策略"犯傻"的时候保护你。再好的策略也有表现不佳的时期，但一个好的风控体系可以确保你活到策略重新表现良好的那一天。

小结与预习

这一章我们从回测走向了实盘，讨论了实盘和回测之间的巨大差距、券商量化平台的选择、2025年程序化交易监管新规、完整的交易成本模型，以及实盘风控体系的搭建方法。核心就一句话：回测是为策略寻找信心，实盘是考验人性的修炼。
