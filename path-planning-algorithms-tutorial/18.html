大家好，欢迎来到第18章。前两章我们讨论了如何让机器人在运动过程中实时避障——DWA在速度空间中搜索，APF在势场中"漂流"。这些方法有一个隐式的假设：机器人可以执行任何命令速度。但现实世界中的机器人不是这样的。一辆汽车不能原地掉头，一架固定翼无人机不能悬停，一个带拖车的叉车倒车时车头和拖车的方向会形成"夹鱼效应"。忽略这些运动学约束去规划路径，就像要求一个不会游泳的人跳过一条河——路径在地图上看是通的，但机器人根本走不出来。

这一章，我们专门来研究运动学约束（kinematic constraints）以及如何在路径规划中满足它们。我们将学习三种逐渐复杂的方法：Dubins曲线（只有前进）、Reeds-Shepp曲线（可前进可后退）、以及运动基元（motion primitives）。这些都是"运动规划"而不是"路径规划"的范畴——它们输出的不是几何路径，而是机器人实际可以执行的运动序列。

为什么运动学约束重要

想象一个停车场场景。自动驾驶汽车需要从车位A开到车位B。A*在停车场的栅格地图上规划出了一条由水平和垂直线段组成的路径，看起来完美无缺。但当你把这个路径发给车辆控制器时，车辆在第一段直行后需要做一个90度左转。然而，汽车的最小转弯半径是5米——停车场通道宽度只有3米。车辆根本无法在这么窄的空间内完成这个转弯。结果就是：车辆要么在原地反复前后挪动试图"挤"过去（如果控制器允许倒车），要么直接放弃说"路径不可执行"。

这个例子说明了一个关键问题：路径规划器的输出是一个几何路径（一系列位置点），而机器人实际需要的是一个运动序列（一系列控制指令）。几何路径到运动序列的转换必须考虑机器人的运动学约束，否则规划出来的路径就是"纸上谈兵"。

这个教训在很多实际项目中都得到了惨痛的验证。2017年，某高校的无人驾驶方程式赛车队在比赛中遇到了一个令人抓狂的问题：他们的路径规划算法在仿真中跑得完美无比，平均速度比对手高出15%。但在实际赛道上，车辆频频冲出赛道、切弯失败。团队花了整整一个暑假排查，最终发现：他们的A*规划器只用了车辆后轴中心点的坐标来建模，完全忽略了车辆的前进方向和转弯半径。规划出来的路径上存在大量曲率超过车辆极限的弯道。路径本身有"不存在"，但从运动学看"不可行"。修复方案是将在路径规划阶段就引入车辆运动学模型——而不是在规划完成后再去"适配"。这在方法论上是一次重要的觉醒：约束应该前置，而不是后补。

核心原则：运动学约束必须在规划阶段就纳入考虑。后处理平滑可以缓解但不能解决运动学不可行的问题。如果你的全局规划器完全不考虑机器人的转弯半径，那么无论后处理做得多么精细，最终路径中都可能存在无法执行的段落。

Dubins曲线

Dubins曲线由Lester Dubins于1957年提出，是运动规划领域最经典的成果之一。它回答了这样一个问题：给定一个具有最小转弯半径的车辆（类似汽车），它在只能前进的条件下，从一种位姿（位置+朝向）到另一种位姿的最短路径是什么？

Dubins的答案是：在任何情况下，最短路径都由三种基本运动单元构成——左转圆弧（L, left turn）、右转圆弧（R, right turn）、和直线（S, straight）。而且，最短路径最多只需要三个这样的单元。组合起来，共有6种可能的路径类型：

  • LSL — 左转 → 直线 → 左转
  • LSR — 左转 → 直线 → 右转
  • RSL — 右转 → 直线 → 左转
  • RSR — 右转 → 直线 → 右转
  • LRL — 左转 → 右转 → 左转（三圆弧，无直线段）
  • RLR — 右转 → 左转 → 右转（三圆弧，无直线段）

Dubins曲线的一个关键特性是：所有圆弧段的半径都等于车辆的最小转弯半径。这意味着整条路径的每一段都是车辆在极限能力范围内可以执行的——只要给出了这条路径，车辆就一定可以"一把方向"走完，不需要任何额外的微调或修正。

下面是一个Dubins曲线生成器的Python实现。给定起始位姿和目标位姿，它计算出最短Dubins路径并返回沿路径的采样点：

import numpy as np
import math

def dubins_path(start, goal, turning_radius):
    \"\"\"计算从start到goal的最短Dubins路径。

    Args:
        start: (x, y, theta) 起始位姿
        goal: (x, y, theta) 目标位姿
        turning_radius: 最小转弯半径

    Returns:
        path_points: (N, 2) 路径采样点坐标
        path_type: 字符串，如"LSL", "RSR"等
        total_length: 路径总长度
    \"\"\"
    # 将goal转换到start的局部坐标系
    dx = goal[0] - start[0]
    dy = goal[1] - start[1]
    dtheta = goal[2] - start[2]

    cos_t = math.cos(start[2])
    sin_t = math.sin(start[2])

    # 局部坐标中的目标位置
    x_local = dx * cos_t + dy * sin_t
    y_local = -dx * sin_t + dy * cos_t
    theta_local = dtheta

    # 归一化到转弯半径为单位长度
    x = x_local / turning_radius
    y = y_local / turning_radius
    th = theta_local

    # 尝试6种Dubins路径类型，选取最短的
    types = ["LSL", "LSR", "RSL", "RSR", "LRL", "RLR"]
    best_path = None
    best_length = float('inf')

    for path_type in types:
        segments = compute_dubins_segments(x, y, th, path_type)
        if segments is None:
            continue

        # 三条线段的长度
        seg1_len, seg2_len, seg3_len = segments
        total_len = seg1_len + seg2_len + seg3_len

        if total_len < best_length:
            best_length = total_len
            best_path = (path_type, segments)

    if best_path is None:
        return None, None, float('inf')

    # 将路径缩放回原始单位并采样
    path_type, (seg1, seg2, seg3) = best_path
    total_length = (seg1 + seg2 + seg3) * turning_radius
    # 此处省略了具体的采样代码（分段插值圆弧和直线）
    # 完整实现可参考GitHub: python-dubins

    return path_type, total_length


def compute_dubins_segments(x, y, th, path_type):
    \"\"\"计算给定类型的Dubins路径三段长度。\”""
    # 这是一个简化版，实际实现需要解三角方程
    # 不同路径类型的几何构造不同
    if path_type == "LSL":
        # LSL: 左转-直线-左转
        u = x - math.sin(th)
        v = y - 1 + math.cos(th)
        d = u * u + v * v
        if d < 4:  # 无解
            return None
        t = math.atan2(v, u) - math.atan2(2, math.sqrt(d - 4))
        p = math.sqrt(d - 4)
        q = th - t
        return (t % (2*math.pi), p, q % (2*math.pi))

    elif path_type == "RSR":
        # RSR: 右转-直线-右转
        u = x + math.sin(th)
        v = y - 1 - math.cos(th)
        d = u * u + v * v
        if d < 4:
            return None
        t = math.atan2(v, u) - math.atan2(2, math.sqrt(d - 4))
        p = math.sqrt(d - 4)
        q = th - t
        return (t % (2*math.pi), p, (-q) % (2*math.pi))

理解Dubins曲线的直观方法：拿一张白纸，画一个圆——这个圆的半径就是车辆的最小转弯半径。Dubins曲线的本质就是：你的车只能沿着这个圆的圆周或切线方向运动。想象你在纸上画两个这样的圆，一个对应起始位姿，一个对应目标位姿，然后找到连接这两个圆的最短路径——这就是Dubins曲线。当你这样在纸上画一画的时候，LSL和RSR的类型就会变得一目了然。

Reeds-Shepp曲线

Dubins曲线只允许前进。但现实中，司机经常需要倒车——比如在狭小的停车位里。Reeds-Shepp曲线（由Reeds和Shepp于1990年提出）扩展了Dubins曲线，允许车辆既可以前进也可以倒车。这使得Reeds-Shepp曲线能够处理更复杂的场景，比如平行泊车——你知道的，那种需要"前后前后"反复调整的泊车方式。

Reeds-Shepp曲线的"词汇表"比Dubins丰富得多。它不仅包含L（左转）、R（右转）、S（直线），还引入了每种运动的前进（+）和后退（-）方向。因此，一个Reeds-Shepp路径类型可能是"L+S+R-"（前进左转→后退直线→后退右转）这样的组合。加上符号后，Reeds-Shepp曲线总共有48种不同的路径类型（word types），分为9个家族。

比较维度 Dubins曲线 Reeds-Shepp曲线
允许倒车 否 是
路径类型数量 6种 48种
适用场景 单向运动（飞机、高速公路） 需要倒车的场景（泊车、狭窄空间）
路径长度 通常更长（因为没有倒车选项） 更短（尤其在狭小空间中）
计算复杂度 低（检查6种类型） 中等（检查48种类型）
实际应用 固定翼无人机航迹规划 自动驾驶泊车、仓库机器人

在应用中，Reeds-Shepp最经典的场景就是自动泊车。想象你的车需要停入一个路边的平行车位：起始位姿在车位前方的道路上，目标位姿是车位内的一个紧贴路沿的位置。Reeds-Shepp规划器会自动计算出类似"前进右转→倒车左转→前进右转"这样的三段式路径，完美模拟了人类驾驶员的泊车操作。这其实就是目前很多量产车"自动泊车"功能的核心算法之一。

运动基元

Dubins和Reeds-Shepp曲线是针对特定运动学模型（汽车模型，即"自行车模型"）的解析方法。它们优美、高效、有理论保证。但如果我们面对的不是一辆汽车，而是一个双轮差速机器人、一个带拖车的叉车、或一个四足机器人呢？每种机器人的运动学模型都不同，不可能为每种模型都推导出解析的最短路径公式。

这就需要一种更通用的方法——运动基元（motion primitives）。

运动基元的核心思想是：离线预计算 + 在线查表。具体来说：

  1. 根据机器人的运动学模型，预先计算出一系列短时长（比如1-2秒）、运动学可行的轨迹片段。这些片段覆盖不同的起始速度和终止速度、不同的转弯方向。
  2. 将这些基元存储在库中，每个基元包含轨迹的位置序列、速度剖面、执行时间等信息。
  3. 在线规划时，从机器人的当前状态出发，搜索由这些基元"拼接"而成的完整路径。这就是所谓的晶格规划（lattice planning）。

运动基元的优势在于：它把"运动学可行性"这道难题从在线规划阶段转移到了离线准备阶段。在线规划时，无论选择哪个基元、怎么拼接，结果都自动满足运动学约束——因为每个基元在生成时就已经保证了可行性。这就像你玩积木——每块积木都经过验证是合格的，你只需要把它们拼起来，不需要担心每块积木本身的质量。

运动基元的另一个优势是通用性。无论你的机器人是全向轮、阿克曼转向、还是四足机器人，你都可以用同样的方法论：建立运动学模型 → 离线生成基元 → 在线搜索拼接。改变机器人只需要更换基元库，规划器本身不需要大改。

下面是一个简单的运动基元生成和使用示例。我们为一个差速驱动机器人生成8个基本运动基元，覆盖不同的速度和角速度组合：

import numpy as np
import math

class MotionPrimitive:
    def __init__(self, v, w, duration, dt=0.1):
        \"\"\"运动基元：以固定(v, w)运动duration时长。\”""
        self.v = v
        self.w = w
        self.duration = duration
        self.trajectory = self._generate(dt)

    def _generate(self, dt):
        \"\"\"生成基元轨迹（局部坐标）。\”""
        points = []
        x, y, theta = 0.0, 0.0, 0.0
        steps = int(self.duration / dt)
        for _ in range(steps):
            x += self.v * math.cos(theta) * dt
            y += self.v * math.sin(theta) * dt
            theta += self.w * dt
            points.append([x, y, theta])
        return np.array(points)


def create_primitive_library():
    \"\"\"为差速驱动机器人创建运动基元库。\”""
    primitives = []
    durations = [1.0, 2.0]
    vs = [0.5, 1.0]           # 两种线速度
    ws = [-1.0, -0.5, 0.0, 0.5, 1.0]  # 五种角速度

    for dur in durations:
        for v in vs:
            for w in ws:
                prim = MotionPrimitive(v, w, dur)
                primitives.append(prim)

    return primitives


def lattice_search(start_state, goal_state, primitives, obstacles):
    \"\"\"晶格搜索：用基元拼接从起点到终点的路径。\”""
    # 这是一个简化框架，实际实现需要图搜索算法
    # 每个状态通过应用基元转移到新状态
    # 不断扩展直到到达目标状态

    visited = {tuple(start_state): 0}
    parent = {tuple(start_state): None}
    frontier = [(tuple(start_state), [start_state])]

    while frontier:
        state_key, path = frontier.pop(0)
        current_state = np.array(state_key)

        if np.linalg.norm(current_state[:2] - goal_state[:2]) < 0.3:
            return path  # 找到路径

        for prim in primitives:
            # 应用基元得到新状态
            dx, dy, dtheta = prim.trajectory[-1]
            theta = current_state[2]
            new_x = current_state[0] + dx * math.cos(theta) - dy * math.sin(theta)
            new_y = current_state[1] + dx * math.sin(theta) + dy * math.cos(theta)
            new_theta = (current_state[2] + dtheta) % (2 * math.pi)
            new_state = np.array([new_x, new_y, new_theta])

            # 检查碰撞
            if check_collision(new_state, obstacles):
                continue

            new_key = tuple(np.round(new_state, 2))
            if new_key not in visited:
                visited[new_key] = 0
                new_path = path + [new_state]
                frontier.append((new_key, new_path))

    return None  # 无解


def check_collision(state, obstacles):
    \"\"\"简化的碰撞检测。\”""
    for obs in obstacles:
        if np.linalg.norm(state[:2] - obs) < 0.3:
            return True
    return False

运动基元和晶格规划在自动驾驶领域有着广泛的应用。例如，Uber ATG（Advanced Technologies Group）在2018年公开的技术论文中，就详细描述了如何使用运动基元构建自动驾驶汽车的局部运动规划器。在他们的系统中，在线规划器在毫秒级的时间内从基元库中搜索拼接出几秒钟的运动轨迹，实现了高速公路上安全、舒适的自动驾驶。

这一章我们学习了三种处理运动学约束的方法。从Dubins的解析公式到Reeds-Shepp的扩展，再到运动基元的通用框架，核心都在回答同一个问题："在这个物理世界中，机器人到底能做什么动作？"运动学约束不是规划完成之后再去检查的东西——它应该从一开始就嵌入在规划方法的选择中。一个优秀的运动规划师深谙"约束前置"的理念：在规划之前，先搞清楚你的机器人能做什么、不能做什么。
