第六章：HTTP头部字段

各位同学，第五章我们聊了状态码，现在进入HTTP协议中信息最丰富的部分——头部字段（Headers）。如果把HTTP报文比作一封信，状态码是信封上的"收件状况"标签，那么Headers就是信的内容提要、寄件人信息、邮寄方式说明……可以说，Headers是HTTP协议的"元数据"层。

在Web开发里你几乎每天都在跟Headers打交道：浏览器自动添加、服务器返回的Set-Cookie、CDN的缓存指令、API的认证Token……不理解Headers，就等于不理解HTTP的通信语言。而且，Headers也是排查问题的利器——我调试网络请求时，首先展开的就是Request Headers和Response Headers面板。

6.1 头部字段分类

HTTP头部字段按作用范围可以分为四类：

分类适用对象说明示例
General Headers（通用头）请求和响应不特定于请求或响应，描述消息的通用信息Cache-Control, Connection, Date
Request Headers（请求头）仅请求客户端发起的额外信息，如客户端能力、身份认证User-Agent, Accept, Authorization
Response Headers（响应头）仅响应服务器返回的额外信息，如服务器身份、指令Server, Set-Cookie, WWW-Authenticate
Entity Headers（实体头）请求和响应描述报文实体（body）的属性和内容Content-Type, Content-Length, Content-Encoding

你打开Chrome DevTools，随便看一个网络请求，就能看到这些头部的实际样子。值得说明的是，Entity Headers的分类在实际中有时会模糊——很多Entity Headers在当今的规范中也被归入Representation Headers，但核心思想不变：都是描述"数据本身"的。

6.2 常用通用头

Cache-Control——控制缓存的"总开关"。可以出现在请求和响应中。常见值：no-cache（每次都向服务器验证）、no-store（完全不缓存）、max-age=3600（缓存1小时）、public（中间缓存可缓存）、private（仅浏览器可缓存）。内容很多，我们第9章会专门讲。

Connection——管理连接行为。最常用的是 Connection: keep-alive 表示启用持久连接（HTTP/1.1默认就是keep-alive，其实不用特别指定）。另一个冷门但重要的用法是 Connection: close 告诉对方请求处理完后关闭连接。

Date——报文生成的日期和时间。格式严格遵循RFC 1123，比如：Date: Tue, 29 Jun 2026 08:00:00 GMT。所有HTTP响应都必须包含Date头（除了某些特殊情况）。它是排查缓存和时区问题的重要线索。

Transfer-Encoding——传输编码。最常用的是 Transfer-Encoding: chunked——分块传输。服务器在不知道响应体总大小时，可以一块一块地发，每块前面带一个长度字段。好处是动态内容可以边生成边发送，不用等全部数据算完。注意它和Content-Encoding的区别：Content-Encoding是内容编码（如gzip压缩），Transfer-Encoding是传输编码。

容易混淆的概念：很多人分不清 Content-Encoding 和 Transfer-Encoding。前者是"内容本身被压缩了"，是端到端的，浏览器解压后拿到的就是原始数据。后者是"传输过程中用了分块"，是逐跳的，代理可以去掉它。简单记：Content-Encoding影响的是body里的数据，Transfer-Encoding影响的是怎么把数据包在HTTP报文中。

6.3 常用请求头

Host——目标主机名和端口。这是HTTP/1.1唯一强制要求的请求头，也是实现虚拟主机的关键。同一个IP上托管多个域名，服务器通过Host头来区分你要访问哪个网站。没有Host头，互联网时代的虚拟主机就没法实现。

User-Agent——标识客户端的身份信息。比如 Mozilla/5.0 (Macintosh; Intel Mac OS X 10_15_7) ... Chrome/120.0.0.0。虽然可以用来做浏览器兼容性适配，但现在已经不推荐依赖它了——UA字符串被各种伪造，已经失去了可信度。

Accept系列——告诉服务器客户端期望接收的格式：Accept（MIME类型）、Accept-Language（语言）、Accept-Encoding（压缩算法）。下一章会详细展开。

Authorization——携带身份凭证，用于HTTP认证。常见格式：Authorization: Bearer eyJhbGciOiJIUzI1NiIs...（JWT Token）或 Authorization: Basic base64(username:password)。记住：这应该只在HTTPS连接下传输，明文传输认证信息是严重的安全隐患。

Cookie——客户端保存的会话信息。格式是键值对：Cookie: session_id=abc123; theme=dark。服务器通过Set-Cookie响应头来设置Cookie，浏览器自动在后续请求中携带Cookie头。我们第10章会细讲。

Referer——来源页面的URL。告诉服务器用户是从哪个页面跳转过来的。注意拼写——规范里就是少了一个"r"（应该是Referrer的笔误，但将错就错了）。在做防盗链和来源统计时非常有用，但注意敏感URL可能会泄露信息。

6.4 常用响应头

Server——服务器的软件信息。比如 Server: nginx/1.24.0。安全最佳实践建议不要暴露详细的版本号，避免被针对性攻击。

Set-Cookie——服务器要求客户端存储Cookie。每个Set-Cookie头设置一个Cookie，包含名称、值、过期时间、路径、域名、安全标志等属性。因为一个响应可以设置多个Cookie，所以Set-Cookie是HTTP里少数允许重复出现的头。

Location——重定向的目标URL。配合3xx状态码使用。比如301或302响应中，告诉浏览器要跳转到哪里去。

WWW-Authenticate——服务器在401响应中告知客户端应采用什么认证方式。比如 WWW-Authenticate: Basic realm="Admin Area"。浏览器收到后会弹出登录对话框。

Access-Control-* 系列——CORS（跨域资源共享）的控制头。包括 Access-Control-Allow-Origin、Access-Control-Allow-Methods、Access-Control-Allow-Headers 等。这是现代前后端分离架构中必须打交道的响应头，我们第12章会专门展开。

6.5 自定义头部

有时候标准Headers不够用，你可能会想定义自己的头。过去人们用 X- 前缀来表示非标准头，比如 X-Requested-With: XMLHttpRequest（jQuery时代判断AJAX请求的经典做法）。

但RFC 6648（2012年）已经正式废弃了X-前缀惯例。原因很简单：非标准头一旦被广泛采用，最终会标准化（比如 X-Forwarded-For 后来被规范为 Forwarded），而带X-的名字显得不合适。此外，X-前缀也未被正式认可为标准行为。

现在的推荐做法是：直接使用有意义的名称，不加X-前缀。比如电商系统定义一个 X-Request-Id → 现在应该用 Request-Id（或者更准确的 X-Request-Id 也行，但它已非推荐格式）。尽量选用和应用相关的名字，加上自己的命名空间避免冲突。

踩坑故事：以前我维护过一个老系统，代码里到处都是自定义的 X-MyApp-* 头。后来引入了一个第三方安全网关，它不认识这些头，在代理过程中把它们全丢掉了。系统莫名其妙地失效，查了三天才找到原因——网关默认过滤非标准头。从那以后我养成了一个习惯：能用标准头解决的问题，绝不发明自定义头。自定义头的正确用途是传递内部元数据，比如追踪ID、内部路由信息等。

6.6 头部大小写与重复

关于HTTP头的大小写问题，很多初学者会困惑：Content-Type、content-type、CONTENT-TYPE 哪个对？答案是：头部的名字是大小写不敏感的。根据HTTP规范，字段名不区分大小写。Content-Type 和 content-type 是完全等价的。

但惯例是采用首字母大写的驼峰式写法：Content-Type、User-Agent、Cache-Control。在代码中解析头部时，你应该用大小写不敏感的方式读取。

关于重复头部：HTTP协议允许同一个头部出现多次。但合并规则有讲究——大多数头可以用逗号合并，也就是说 Accept: text/html 和 Accept: application/json 两个头等价于 Accept: text/html, application/json。

但有一个特例：Set-Cookie。Set-Cookie 不能合并，每个Set-Cookie头单独设置一个Cookie。这就是为什么标准说"可以使用多个Set-Cookie头"却不能写成 Set-Cookie: a=1, b=2 的原因。

实用建议：编写HTTP客户端代码时，永远用大小写不敏感的方式读取头部。很多语言的HTTP库都默认帮你做了这件事（比如Python的Requests库），但如果你在手动解析原始报文，别假设头部一定是你期待的大小写。
