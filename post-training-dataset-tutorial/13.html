第13章 SFT数据配比与混合策略

好，现在你手里有一堆数据集了。OpenHermes 2.5、UltraChat、GSM8K、Firefly、Medical Meadow……它们各有特色，看起来都不错。但问题来了：到底按什么比例混在一起呢？

如果你觉得这个问题"凭经验拍脑袋就行"，那结果可能会让你大吃一惊。同样的数据集，不同的配比，训练出来的模型能力差异可能高达 10-20 个百分点。我见过一个团队，仅仅把对话数据的比例从 20% 调整到 35%，MT-Bench 分数就提升了将近 0.3 分——这是一个巨大的差异。

这一章，我们不谈具体哪个数据集好，而是深入探讨SFT 数据配比的方法论——如何通过系统化的实验，为你的模型找到最优的"数据配方"。

数据配比是SFT的第一道工序

在谈论数据配比之前，先建立一个共识：数据配比不是可有可无的"最后调整"，而是 SFT 的第一道工序。它的质量直接决定了模型的"能力结构"——什么能力强、什么能力弱。

你可以把 SFT 数据想象成给模型吃的"营养餐"。通用数据是主食，提供基础的碳水化合物（语言理解和对话能力）；推理数据是蛋白质，支持"肌肉"生长（复杂问题解决能力）；多轮对话数据是膳食纤维，帮助"消化系统"运作（上下文理解和延续能力）；安全数据是维生素，确保免疫系统正常（内容安全合规）。

营养餐讲究均衡——光吃主食会营养不良，只吃蛋白质也会出问题。数据配比也是同样的道理。

数据配比的第一性原理：模型的能力分布 ≈ 训练数据的能力分布 × 每条数据的"学习难度"加权。前者是你的输入（你给了模型什么数据），后者是模型的"学习偏好"（哪些类型的任务更容易被模型吸收）。由于"学习难度"难以预先评估，数据配比的本质就是通过实验找到"能力加权后的最优分布"。

典型配比参考

当然，虽然我们强调"没有通用的最优配比"，但有很多团队已经通过实践验证了"不错的起点配比"。以下是一个经过多个项目验证的通用 SFT 数据配比：

数据类别 推荐比例 覆盖范围 典型数据集示例
通用指令（英文） 35-40% 问答、写作、翻译、分类、摘要等 OpenHermes 2.5, FLAN Collection
通用指令（中文） 15-20% 中文问答、写作、知识问答等 Firefly, BELLE
多轮对话 20-25% 单轮/多轮对话、角色扮演、闲聊 UltraChat, ShareGPT, UltraChat-zh
推理数据 10-15% 数学推理、逻辑推理、编程解题 GSM8K, MATH, OpenOrca
安全与合规 3-5% 拒绝不当请求、安全边界测试 自定义安全数据

这个配比的基本逻辑是：通用指令做骨架，多轮对话做血肉，推理数据做筋骨，安全数据做防护。其中英文通用指令的比例高于中文，是因为 80% 以上的高质量开源 SFT 数据是英文的；如果你的目标场景主要是中文用户，可以适当调高中文数据的比例到 30-35%，同时降低英文数据到 25-30%。

配比实验设计方法论

知道了起点，下一步就是找到针对你项目的"最优配比"。下面是一套被验证过的方法论：

步骤一：预实验（Pilot Experiment）

在几小时到一天内完成的小规模实验，目的是快速淘汰明显不好的配比。用 5,000-10,000 条训练数据（从各数据源按不同配比采样），在 7B 模型上训练几百步，看 loss 收敛情况和基础评测指标。这个阶段目标是"排出"而不是"选优"——找到几个看起来有潜力的配比候选。

步骤二：小规模消融（Ablation Study）

在预实验筛选出的 3-5 个候选配比上，用 30,000-50,000 条数据做完整的小规模训练实验。在这个阶段，你需要做控制变量——保持训练超参数完全一致，只改变数据配比。每个配比至少训练到 loss 基本收敛，然后用一套固定的评测集做对比。

# 配比消融实验的代码框架
import itertools

# 定义待测试的配比候选
candidates = [
    {"general": 0.50, "dialogue": 0.25, "reasoning": 0.20, "safety": 0.05},
    {"general": 0.60, "dialogue": 0.20, "reasoning": 0.15, "safety": 0.05},
    {"general": 0.40, "dialogue": 0.35, "reasoning": 0.20, "safety": 0.05},
    {"general": 0.55, "dialogue": 0.25, "reasoning": 0.15, "safety": 0.05},
]

results = []
for ratio in candidates:
    # 组装混合数据集
    dataset = build_mixed_dataset(
        data_sources=[
            ("openhermes", ratio["general"]),
            ("ultrachat", ratio["dialogue"]),
            ("gsm8k", ratio["reasoning"]),
            ("safety_data", ratio["safety"]),
        ],
        total_size=50000  # 小规模消融，5万条
    )
    # 训练模型
    model = train_sft(dataset, model_name="llama-3-7b",
                      num_epochs=3, lr=2e-5)
    # 评测
    scores = evaluate(model, benchmarks=["gsm8k", "mt_bench", "truthful_qa"])
    results.append({"ratio": ratio, "scores": scores})
    print(f"Ratio {ratio}: {scores}")

步骤三：全量训练与验证

从小规模消融中选出最优的 1-2 个配比，扩展到全量数据（数十万到数百万条）进行完整训练。在这个阶段，数据配比已经不是主要变量了，你需要关注的是训练稳定性、收敛速度和最终的绝对性能。

一个容易被忽略的细节：在全量训练前，验证一下从小规模到全量的"配比一致性"。有时候在小规模上的最优配比，扩展到全量后反而效果一般——原因是大规模训练中不同数据类型的"过拟合速度"不一样。一个实用的技巧是：在小规模消融时，额外做一个"按比例放大"的验证，确认配比具有规模不变性。

多任务平衡技术

配比实验更多是"宏观调配"，而多任务平衡技术解决的是"微观优化"——在固定的宏观配比下，通过更精细的训练策略让每个任务类型得到恰当的训练信号。

温度采样

当数据集中各任务的数据量差异很大时（比如你有 100 万条通用数据但只有 1 万条安全数据），直接按比例采样会导致安全数据几乎没机会被模型看到。温度采样通过给数据量小的任务更高的采样权重来解决这个问题。

def temperature_sampling(datasets, temperature=1.0):
    """
    温度采样：temperature > 1.0 时，小数据量的任务获得更高的采样权重
    temperature = 1.0 时退化为按比例采样
    """
    # 按任务统计数据量
    task_sizes = {task: len(ds) for task, ds in datasets.items()}
    # 计算温度调整后的权重
    total = sum(task_sizes.values())
    weights = {
        task: (size / total) ** (1.0 / temperature)
        for task, size in task_sizes.items()
    }
    # 归一化
    norm = sum(weights.values())
    return {task: w / norm for task, w in weights.items()}

# temperature=2.0 时，小任务的采样权重会得到提升
sample_weights = temperature_sampling(
    {"general": 1000000, "safety": 10000}, temperature=2.0
)
# 输出: {"general": ~0.91, "safety": ~0.09}
# 默认比例是 0.99 vs 0.01，小任务权重提升了 9 倍

按任务损失加权

一个更高级的方法是动态监测每个任务类型的 loss，给 loss 更高的任务更大的梯度权重。这背后的直觉是：如果模型在某个任务上还在"犯错"（loss 高），就多学一点。这种方法在训练框架层面的实现比较复杂，需要逐任务计算 loss。

动态调度

训练的不同阶段，数据的最优配比可能不同。早期阶段需要更多的通用数据来建立基础能力，后期阶段可以增加推理数据来强化深度思考。动态调度就是在训练过程中逐渐调整配比。比如前 20% 的训练步数使用 70% 通用 + 30% 推理，后 30% 的训练步数调整为 40% 通用 + 60% 推理。

过拟合的预警信号

配比不当最常见的后果就是过拟合。不是指模型在某个任务上过拟合，而是指模型对训练数据中的某些模式"太熟了"，失去了泛化能力。以下是几个关键的预警信号：

  • Loss下降但Eval指标不涨：这是最经典的过拟合信号。训练 loss 在持续下降，说明模型在记住训练数据，但评测指标（GSM8K、MT-Bench 等）停滞不前甚至下降，说明学到的东西没有泛化。
  • 固定模板输出：模型对不同的问题开始使用极其相似的句式结构。比如无论问什么，模型都以"关于这个问题，我们需要从多个角度来分析"开头。这说明模型对训练数据中的某种固定表达模式过拟合了。
  • 多样性崩塌：评估模型回答的"词汇多样性"和"句式多样性"，如果多样性指标在训练后期急剧下降，说明模型开始在同一个"模式"下输出内容，这是过拟合的早期信号。

如果出现这些信号，常见的应对办法包括：降低相同来源数据的重复量（重复采样导致的过拟合最常见）、增加数据多样性（引入新的数据源或任务类型）、减少训练轮数（不是所有模型都需要训练 3 个 epoch）。

一个真实案例：配比失调导致模型变成「安全回答机器」

最后分享一个来自行业内的知名案例。2024 年，某团队在训练一个面向企业客户的大模型时，为了确保模型的安全合规性，在 SFT 数据中加入了大量的"安全拒绝样本"——即让模型学会在遇到敏感话题时拒绝回答。

团队很认真，安全样本的质量很高，覆盖了几百种敏感话题的拒绝话术。但他们犯了一个数据配比上的错误：安全样本的比例被设置到了 15%——远远超过了推荐的安全数据份额（3-5%）。

训练完成后，模型确实非常安全——安全过头了。用户问"今天的天气如何？"——模型回答"对不起，我无法回答这个问题。"用户问"Python 怎么排序一个列表？"——模型回答"关于编程的问题，我建议你查阅官方文档。"用户问"你能介绍一下你们公司吗？"——模型回答"作为一个 AI 助手，我不代表任何公司立场。"

模型变成了一个"安全回答机器"，拒绝一切看起来可能"不安全"的请求。问题的根源很明确：15% 的安全数据在训练中产生了过强的"拒绝回答"梯度信号，模型学会了把"回答先拒绝"作为默认策略——因为训练中最常见的模式就是"用户提问→AI 拒绝"。

教训：安全数据是一把双刃剑。太少不够安全，太多会让模型瘫痪。经验表明，安全样本的比例最好控制在3-5%。而且，安全数据不能简单地从其他通用数据中"切"出一块来配入，而应该作为独立的类别单独控制比例。另外，安全拒绝样本的表述应该多样化——不能让模型学会"反正就是拒绝"，而是让它学会"判断什么该拒绝，什么不该拒绝"。

数据配比看起来是一个"调比例"的简单活，实际上是一项需要严谨实验设计和细心观察的系统工程。一个合适的数据配比，能让模型的各项能力得到均衡发展；一个糟糕的配比，可能会让模型的某些能力严重受损，甚至产生意想不到的行为问题。

有了好的配比策略之后，我们终于可以处理一个在整个后训练数据工程中不可或缺的环节——数据去噪与去重。不管数据配比多完美，如果数据中存在大量重复或噪声，一切都是白费功夫。第 14 章，我们将拿起工程工具，实战 SFT 数据的清洗、去重和质量过滤。
