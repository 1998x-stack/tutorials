CUDA流与并发

大家好，我是你们的讲师。到目前为止，我们的所有代码都遵循一个简单的模式：拷贝数据到GPU → 启动内核 → 拷贝结果回CPU → 下一轮。每一步都在等上一步完成。但你有没有想过——CPU拷贝数据的时候GPU能不能同时做计算？ 答案是：能，而且你应该这样做。实现这种并发的工具就是CUDA流（stream）。

CUDA流是GPU上操作的逻辑队列。同一个流中的操作按顺序执行，但不同流中的操作可以"乱序"——硬件会尽可能地调度它们并发执行。本章我们会从默认流的陷阱讲起，到显式流、计算与传输重叠、流同步与事件，最后到CUDA 11.2引入的Stream Ordered Memory Allocator。掌握流，你就学会了让GPU同时做多件事的能力。

一句话总结： CUDA流 = GPU的"多线程"。一个流 = 一个逻辑工作队列。多个流 = GPU可以同时处理多个任务队列，实现计算-传输重叠和并发内核执行。

默认流的隐藏陷阱

如果你从来没有显式创建过CUDA流，那你一直在用的就是默认流（default stream），也叫"NULL stream"。对于CUDA 7.0之前的代码，默认流有个臭名昭著的行为：它会和所有其他流做隐式同步。 这意味着：你在默认流里启动了一个内核，它会在开始之前等待所有其他流中的操作完成，并且在它完成之前，其他流也不能开始新的操作。

特性 默认流（CUDA 7.0 前默认行为） 显式流（非阻塞）
跨流同步 隐式同步——默认流操作会等待所有流，且所有流会等待默认流 不同流之间互不阻塞
并发能力 基本没有——默认流会串行化所有操作 完全支持并发
适用场景 简单的一次性计算 需要并发的任何场景
性能潜力 低——GPU通常处于闲置状态等待同步 高——GPU可以同时做多件事

从CUDA 7.0开始，NVIDIA引入了一个编译选项 --default-stream per-thread，让每个host线程有自己的独立默认流，不再有隐式同步。但从最佳实践的角度，我强烈建议你永远不要依赖默认流做性能敏感的任务，显式创建和管理流才是正确的做法。

历史遗留问题： 很多老旧代码（尤其是2015年之前写的）用的都是默认流，这些代码在当时的GPU上"好像没什么问题"，但在现代GPU上性能远没有发挥出来。如果你在维护一个老的CUDA代码库，把默认流换成显式流往往是第一个立竿见影的优化。

创建和使用显式流

创建和使用显式流非常直接。用 cudaStreamCreate() 创建，然后通过 <<<grid, block, smem, stream>>> 的第四个参数指定内核在哪条流上执行。

// 创建3个流，每个流做一组独立的数据处理
cudaStream_t streams[3];
for (int i = 0; i < 3; ++i)
    cudaStreamCreate(&streams[i]);

// 在每个流中分别启动内核（这三个内核可能并发执行）
for (int i = 0; i < 3; ++i) {
    my_kernel<<<grid, block, 0, streams[i]>>>(d_data[i], N);
}

// 等待所有流完成
for (int i = 0; i < 3; ++i)
    cudaStreamSynchronize(streams[i]);

// 清理
for (int i = 0; i < 3; ++i)
    cudaStreamDestroy(streams[i]);

在这个例子中，三个内核分别在不同的流上执行。如果GPU有足够的硬件资源（SM单元、显存带宽等），这三个内核会在GPU上同时运行。GPU的硬件调度器（GigaThread Engine）会自动在不同的stream之间切换，尽可能保持所有SM都处于忙碌状态。

实际效果： 如果你的kernel是小规模的（每个block只需要少量SM资源），并发执行多个kernel可以显著提高GPU利用率。但如果每个kernel已经接近占满整个GPU（比如大型矩阵乘法），额外并发反而可能导致资源竞争导致性能下降。所以流的并发能力需要结合具体kernel的占用率来评估。

计算与数据传输重叠

CUDA流最实用、最经典的使用场景就是计算与数据传输重叠。现代GPU有独立的DMA引擎（称为Copy Engine），可以独立于计算引擎执行 cudaMemcpyAsync 操作。这意味着：在数据从CPU拷贝到GPU的同时，GPU可以同时执行另一个流中的内核——只要它们操作的是不同的显存区域。

要实现这个重叠，你需要使用固定内存（pinned memory）和异步内存拷贝：

// 分配固定内存（pinned memory）—— 异步拷贝的前提
float *h_data;
cudaMallocHost(&h_data, total_size * sizeof(float));  // 主机端固定内存

float *d_data;
cudaMalloc(&d_data, total_size * sizeof(float));

// 将数据分成N个chunk，每个chunk用独立的流处理
const int N_CHUNKS = 4;
const int chunk_size = total_size / N_CHUNKS;

cudaStream_t streams[N_CHUNKS];
for (int i = 0; i < N_CHUNKS; ++i)
    cudaStreamCreate(&streams[i]);

for (int i = 0; i < N_CHUNKS; ++i) {
    int offset = i * chunk_size;
    // 异步拷贝：不会阻塞CPU或GPU上的其他操作
    cudaMemcpyAsync(&d_data[offset], &h_data[offset],
                    chunk_size * sizeof(float),
                    cudaMemcpyHostToDevice,
                    streams[i]);

    // 在同一个流中启动内核 —— 确保拷贝完成后才执行
    my_kernel<<<grid, block, 0, streams[i]>>>(&d_data[offset], chunk_size);

    // 异步拷回结果
    cudaMemcpyAsync(&h_data[offset], &d_data[offset],
                    chunk_size * sizeof(float),
                    cudaMemcpyDeviceToHost,
                    streams[i]);
}

// 等待所有流完成
for (int i = 0; i < N_CHUNKS; ++i)
    cudaStreamSynchronize(streams[i]);

在这个流水线模式中：

  • 流0在拷贝chunk 0时，流1的内核可以开始处理chunk 1——只要chunk 1的拷贝已完成
  • 理想情况下，数据传输的延迟完全被计算掩盖，总耗时 ≈ max(总计算时间, 总传输时间)，而不是 总传输 + 总计算
  • 使用的chunk数量需要适当——太少则重叠不充分，太多则增加调度开销
管线化深度： 实际操作中，大多数GPU有两个Copy Engine（一个H2D、一个D2H）和一个Compute Engine。这就是为什么要至少分3个chunk——让H2D、Compute、D2H同时运行。如果系统同时支持双向传输（基于PCIe或NVLink），则管线化效果更明显。

流同步与事件

流之间的同步主要通过CUDA事件（event）实现。事件可以插入到流中，并作为其他流等待的信号点：

cudaEvent_t event;
cudaEventCreate(&event);

// 流A：计算中间结果
kernel_A<<<grid, block, 0, stream_A>>>(d_input, d_intermediate, N);
// 在流A中记录事件
cudaEventRecord(event, stream_A);

// 流B：等待流A中的事件完成，然后使用流A的结果
cudaStreamWaitEvent(stream_B, event, 0);
kernel_B<<<grid, block, 0, stream_B>>>(d_intermediate, d_output, N);

cudaStreamSynchronize(stream) 阻塞host（CPU）直到指定流中的所有操作完成。cudaDeviceSynchronize() 阻塞host直到设备上所有流中的所有操作完成。一般在benchmarking时使用 cudaDeviceSynchronize() 来确保所有操作都已完成，但在实际应用中应尽量减少同步点——因为每次同步都意味着CPU在空转等待。

最佳实践： 尽量减少host-side的显式同步。使用 cudaStreamWaitEvent 在设备端完成流之间的依赖管理，而不是在CPU端用 cudaStreamSynchronize 逐个等待。设备端的同步效率远高于CPU端同步。

流优先级

从CUDA 5.0开始，你可以为流设置优先级。高优先级的流中的操作在硬件调度时会优先执行。创建带优先级的流使用 cudaStreamCreateWithPriority：

int lo, hi;
cudaDeviceGetStreamPriorityRange(&lo, &hi);
// lo = 最低优先级（数值最小），hi = 最高优先级（数值最大）

cudaStream_t high_prio_stream, low_prio_stream;
cudaStreamCreateWithPriority(&high_prio_stream, cudaStreamNonBlocking, hi);
cudaStreamCreateWithPriority(&low_prio_stream, cudaStreamNonBlocking, lo);

// 高优先级的kernel会优先调度
critical_kernel<<<grid, block, 0, high_prio_stream>>>(...);
background_kernel<<<grid, block, 0, low_prio_stream>>>(...);

优先级什么时候有用？当你有延迟敏感的内核（比如需要快速响应的交互式应用）和后台任务（比如批处理）混合执行时。高优先级流确保关键任务不会被后台任务阻塞。但如果GPU没有被饱和，优先级的影响微乎其微——所有内核都能获得足够的资源。

注意： 不是所有GPU都支持流优先级。从Kepler架构开始支持，但具体的优先级范围因设备而异。如果设备不支持，所有流都是同一优先级。在代码中应使用 cudaDeviceGetStreamPriorityRange 检查支持情况。

CUDA Stream Ordered Memory Allocator

CUDA 11.2引入了一项重要的改进——Stream Ordered Memory Allocator。传统的 cudaMalloc 和 cudaFree 是同步的——它们会阻塞所有流，直到分配/释放完成。这在并发场景中是一个不小的开销。新的API允许在流中异步分配和释放内存：

// 传统方式 —— 同步分配，阻塞所有流
float *d_ptr;
cudaMalloc(&d_ptr, size);  // 同步操作，会等待所有流排空

// Stream Ordered方式 —— 异步分配，不阻塞
float *d_ptr;
cudaMallocAsync(&d_ptr, size, stream);  // 在stream中排队分配

// 内核使用
my_kernel<<<grid, block, 0, stream>>>(d_ptr, N);

// 异步释放 —— 等内核完成后再释放
cudaFreeAsync(d_ptr, stream);  // 在stream中排队释放

这个API的主要优点是：

  • 减少同步开销：分配和释放不再需要隐式同步所有流
  • 内存重用：驱动可以更智能地在不同流之间重用已释放的内存块
  • 降低延迟：适用于需要频繁分配临时缓冲区的工作负载
真实"踩坑"故事： 几年前我为一个视频处理管线做CUDA加速。管线有4个独立的处理阶段：解码、预处理、推理、后处理。我的第一个版本把所有操作都放到默认流里——代码简单，一运行，性能很差。用Nsight Systems一看，GPU利用率不到30%。问题很清晰：解码阶段的DMA传输在跑的时候，内核调度器在空等；内核在跑的时候，DMA引擎在闲置。解决方案就是把4个阶段分别放到4个显式流中，并用事件来管理它们之间的依赖。改完之后，DMA引擎和计算引擎同时运转，GPU利用率从30%飙升到80%以上。端到端吞吐量从原来的每秒处理20帧提升到了64帧——3.2倍的提升，没有动一行计算代码，只是把流安排明白了。所以我想说的是：在你开始微调内核之前，先看看你的流的安排是不是合理的。 有时候最大的瓶颈不在内核里，而在内核之外。

CUDA流是让GPU从"单线程"变成"多线程"的关键工具。正确的流管理可以实现计算与传输的完美重叠，大大提高GPU利用率和端到端的吞吐量。默认流用起来方便，但它会杀死并发——对于任何性能敏感的应用，显式创建和管理流都是必须的。下一章我们聊一个和流密切相关的话题——事件与性能计时。你现在的kernel到底是快是慢？你怎么测量？下一章我们做一次"时间警察"。
