第20章 指令微调与Agent能力

大家好，我是你们的讲师。上一章我们聊了RLVR，让模型学会了"思考"。但思考的最终目的是什么——是行动。一个只会回答问题的模型就像一个只会纸上谈兵的谋士，真正的价值在于它能调用工具、执行任务、与环境交互。这就要用到我们今天要讲的主题：Agent能力训练。

2025年以来，业界对LLM的期待已经从"聊天机器人"升级为"Agent操作系统"。模型不再只是"你说一句我回一句"的对话工具，而是要能自主完成多步骤任务：查数据库、写代码、调用API、操作浏览器、甚至协调其他Agent协作。这些能力的根基都在指令微调阶段——你在SFT数据里植入了什么能力，模型出厂时就具备什么能力。这一章我们来深入拆解Agent训练的四大核心模块。

20.1 Agent能力的训练基础

一个合格的Agent模型需要具备三大核心能力：Instruction Following（指令遵循）、Tool Use（工具调用）和Planning（规划）。这三个能力虽然相互关联，但训练方法各有侧重。

Instruction Following是最基础的能力。模型需要准确理解用户的指令意图，并按照指令的约束条件执行。这里的挑战在于：很多用户指令是有"隐性约束"的。比如用户说"帮我查一下今天的天气后发邮件给张三"，隐形约束是"先去查天气API得到结果，然后用结果构造邮件正文，再调发送邮件API"——而不是直接生成一个假的天气数据。训练instruction following的关键是在数据中构建多约束、多步骤的指令样本。

Tool Use是Agent区别于普通对话模型的标志性能力。模型需要知道：有哪些工具可用、每个工具的输入输出格式是什么、什么时候该调用工具、调完工具之后怎么处理返回结果。这本质上是一个API orchestration的问题，模型需要在对话流中无缝地插入function call、处理返回值、再决定下一步动作。

Planning是最难的能力。任务是"整理一个关于AI芯片市场分析的PPT"，模型需要先分解子任务：搜索AI芯片行业报告、提取关键数据、确定PPT框架、生成每一页内容、最后格式化输出。每一步都可能需要调用不同的工具，前一步的结果是后一步的输入。规划能力的训练不能只靠SFT，因为规划的搜索空间太大——同一个任务有无数种分解方式，标注者很难给出"最优解"。实践中更多用RL（比如RLVR的变体）来让模型在探索中学会规划。

能力 训练方法 数据构建难度 典型指标
Instruction Following SFT (多约束指令) 中等 IFEval, FollowBench
Tool Use SFT + RL (function call数据) 高 BFCL, ToolBench
Planning RL (多步探索) 极高 AgentBench, WebArena

核心认知：Agent能力的训练和传统SFT有一个根本不同——传统SFT是"记忆"，Agent训练是"策略"。记忆可以靠数据量堆，策略必须靠RL的试错机制去习得。很多团队在Agent训练上砸了海量的SFT数据，结果模型在未见过的工具和场景上仍然表现很差，就是因为缺少RL阶段的策略探索。想做好Agent，RL不是可选项而是必选项。

20.2 Function Calling训练

Function Calling（也叫Tool Calling）是Agent训练中最具体、可操作性最强的部分。说白了就是让模型学会：知道什么时候该调用工具、调用哪个工具、怎么填入参数。

首先来看function schema的设计。你在SFT数据中怎么描述一个工具，模型就怎么使用它。业界常用的格式是JSON Schema风格的function definition，包括function name、description、parameters（类型、枚举值、是否必填）。Description的质量极其关键——模型通过description理解"这个工具是干什么的"，措辞要清晰、无歧义、包含边界条件。

{
  "type": "function",
  "function": {
    "name": "search_flights",
    "description": "搜索航班信息。出发地和目的地用IATA三字码（如PEK、SHA）。日期格式为YYYY-MM-DD。",
    "parameters": {
      "type": "object",
      "properties": {
        "origin": {
          "type": "string",
          "description": "出发地IATA三字码，必填"
        },
        "destination": {
          "type": "string",
          "description": "目的地IATA三字码，必填"
        },
        "date": {
          "type": "string",
          "description": "出发日期，格式YYYY-MM-DD，必填"
        },
        "passengers": {
          "type": "integer",
          "description": "乘客人数，默认1",
          "default": 1
        }
      },
      "required": ["origin", "destination", "date"]
    }
  }
}

多工具调用的训练是更复杂的场景。用户说"帮我找一下后天从北京到上海的航班，然后订最便宜那班"，模型需要先调search_flights获取结果，解析结果后调book_flight来完成预订。这要求对话数据中包含完整的tool call chain：user query → function call → function response → next function call (or) → final answer。每条tool call数据都需要标注出完整的调用轨迹。

在实践中，多工具调用的SFT数据主要有两个来源：一是人工标注轨迹，成本高但质量可控；二是环境回放（Trajectory Replay）——让一个teacher模型（比如GPT-4）使用工具完成任务，记录完整的调用轨迹作为训练数据。环境回放的产出量可以做到人工标注的100倍以上，但质量噪声也大——teacher模型也可能犯工具调用错误。推荐的做法是先让teacher模型跑一遍，然后用自动化验证器过滤掉失败的轨迹，只保留成功轨迹做训练。

踩坑案例：有一家做AI客服的公司在训练function calling时遇到了一个尴尬的问题——模型学会了"过度调用"。用户说"帮我查一下订单"，模型直接调了三个API：查订单、查物流、查售后政策，一次对话发了12个API请求。原来他们的训练数据里大部分是多步操作场景，模型学到了"调用越多工具说明我越能干"的pattern。解决办法是在数据中加入不需要调用工具即可回答的样本（"我的用户名是什么"——这个不需要调任何API），以及在损失函数中对不必要的function call施加轻微的penalty。

20.3 多轮对话训练

Agent场景天然是多轮对话。但多轮对话的SFT和单轮有很大不同——你不能简单地把多轮对话当成"多个单轮对话拼起来"。关键的技术挑战有三个。

Loss Masking是最微妙的技术点。在多轮对话中，模型只需要预测"assistant"角色的token，而对"user"和"system"角色的token应该做loss masking（就是这些token不参与loss计算）。这个看起来很简单，但实践中频繁出问题：masking实现错了导致模型在user query上也在做预测，训练loss下降很快但实际对话效果一塌糊涂——因为模型学会了"预测用户会说什么"而不是"回答用户的问题"。

上下文窗口管理是第二个挑战。Agent的多轮对话可能非常长——几十轮甚至上百轮工具调用和结果返回。训练时你不可能把整个历史都塞进上下文。常用的策略是truncation with recent focus：保留最近的K轮对话作为完整上下文，更早的历史做摘要压缩或者直接丢弃。在数据准备阶段，每一条训练样本都可以做一次"滑动窗口切片"——从对话历史中截取连续的若干轮作为上下文，最后一轮作为预测目标。

对话状态跟踪是第三个关键技术。在多轮工具调用中，模型需要记住前面已经完成了什么、还有什么没做。比如用户先说"帮我订一张明天去上海的机票"，模型查完了航班；然后用户说"还是订高铁吧"——模型需要理解"目的地和时间不变，交通工具变了"，而不是当做一个全新的query。训练数据中需要有大量的指代消解（coreference resolution）和状态更新（state update）样例。

# 多轮对话Loss Masking示意
# user_tokens 和 assistant_tokens 是已经拼接好的token序列
# labels中只有assistant部分的token需要预测

def create_loss_mask(conversation):
    """
    conversation: [{"role": "user", "content": "..."},
                   {"role": "assistant", "content": "..."},
                   {"role": "user", "content": "..."},
                   {"role": "assistant", "content": "..."}]
    """
    labels = []
    loss_mask = []
    for turn in conversation:
        tokens = tokenizer.encode(turn["content"])
        if turn["role"] == "assistant":
            labels.extend(tokens)       # 需要预测
            loss_mask.extend([1] * len(tokens))
        else:
            labels.extend([-100] * len(tokens))  # 忽略
            loss_mask.extend([0] * len(tokens))
    return labels, loss_mask

20.4 Chain-of-Thought训练

Chain-of-Thought（CoT）训练的核心目标是把模型从"直接输出答案"变成"先思考再输出"。这对Agent能力至关重要——Agent在调用工具之前，需要先思考"需要什么信息、用什么工具获取、参数怎么填"。CoT就是这个"内部思考"的显式化。

CoT数据的构建是训练的第一步。早期的方法是从已有数据中"提炼"——让一个强模型（如GPT-4或Claude）对已有的QA数据进行"思维链标注"：对于给定的问题和答案，写出中间推理步骤。这种方法的优点是成本低、可扩展，但缺点是强模型标注的CoT质量参差不齐，偶尔会有逻辑跳跃。

从答案到思考过程的转变是关键。传统SFT数据的形式是Q: 问什么？A: 答什么。CoT数据的形式变成Q: 问什么？A: 第一步：理解问题...第二步：搜索相关信息...第三步：综合分析...第四步：得出结论。最终答案：...。训练时模型输出被划分为两个部分：CoT部分（思考过程）和final answer部分（最终答案）。有些训练方案还会对CoT部分施以较低的loss权重，让模型更关注最终答案的准确性，同时保留灵活思考的空间。

实践中一个有效的技巧是Staged CoT Training：先训练"简短推理"（2-3步），再训练"详细推理"（5-10步），最后训练"Agent式推理"（包含工具调用决策的推理）。这种渐进式的方法比一次性上复杂CoT的收敛速度快很多。类比一下：你不会让一个刚开始学数学的小学生直接做高数题，同样的道理，模型的推理能力也需要逐步培养。

实践建议：CoT训练要特别注意"幻觉蔓延"问题。模型在推理过程中可能会产生错误的前提假设，然后基于错误假设继续推理，导致整个chain崩塌。一个实用的缓解方法是checkpoint supervision——在CoT的关键节点插入验证步骤，要求模型确认前一步的结论是否正确后再继续。在训练时，如果发现某个训练样本的CoT包含逻辑断裂或事实错误，宁可丢掉这条数据也不要用来训练，否则模型会学到"不严谨的推理也是可以的"。

20.5 Agent训练的未来

Agent训练远没到成熟阶段。2026年的前沿探索展现出几个值得关注的趋势。

环境交互训练是最大的方向。目前大多数Agent训练还是基于静态数据（标注好的trajectory），模型没有机会在真实环境中试错。未来的Agent训练将更多采用RL in the loop的方式——模型直接部署到沙箱环境中，执行真实任务（如操作浏览器、编写代码、查询数据库），从环境反馈中学习。这个思路和RLVR一脉相承——只要环境能提供verifiable reward（任务成功/失败），模型就能从试错中提升。

自我改进循环是第二个方向。模型执行完一个Agent任务后，对自己生成的trajectory做反思（self-reflection），找出改进点，然后用改进后的trajectory做下一次训练。这个"执行→反思→训练"的闭环可以让模型在没有新标注数据的情况下持续提升。目前这个方向的主要挑战是反思质量——模型可能反思得不够深，或者反思出的改进方案本身就是错的。

多Agent协作是最具想象力的方向。未来的应用场景中不会只有一个Agent，而是一群各司其职的Agent协同工作：一个负责规划、一个负责编码、一个负责测试、一个负责撰写文档。训练多Agent协作跟单Agent完全不同——每个Agent需要在SFT阶段就学会"理解其他Agent的输出"、"在合适的时候移交控制权"、"分工合作完成目标"。这实际上是把Agent训练从"个体智能"扩展到了"群体智能"。Google的DataGemma和微软的AutoGen都在尝试这个方向，但目前还没有通用的训练方法论。

方向 核心方法 成熟度 关键挑战
环境交互RL RLVR + 沙箱环境 中（探索期） 环境通用性、训练效率
自我改进循环 Self-play + Rejection Sampling 中（实践中已有应用） 反思质量、过拟合风险
多Agent协作 联合SFT + 通信协议训练 低（早期研究） 通信效率、Credit Assignment

Agent训练是一个快速演进的前沿领域。从instruction following到tool调用，从单轮到多轮，从静态SFT到环境RL，每一步都在把模型从一个被动的"问答机器"变成主动的"任务执行者"。下一章我们要进入一个完全不同的但同样重要的领域——第21章「多模态模型训练」，让模型不再只懂文字，而是能"看见"图像、理解视频。这是一个全新的维度扩展，咱们下章见。
