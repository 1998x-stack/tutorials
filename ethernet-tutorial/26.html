第26章 超以太网：UEC与AI网络

大家好。在前面的章节中，我们依次学习了传统以太网、数据中心以太网、SDN等技术。有的同学可能会问：以太网发展到今天，难道还有什么是值得"从头来过"的吗？答案是：有。而且这一次的重塑，恰恰是因为一个我们每个人都在间接使用的东西——AI大模型。

你可能已经听说过，训练一个GPT-4级别的大模型需要成千上万张GPU协同工作几千个小时。这些GPU之间需要频繁交换参数和梯度数据——不是几兆字节，而是几百GB甚至TB级别的数据。这要求网络的带宽极高、延迟极低、丢包率无限趋近于零。用一句话概括：AI训练对网络的要求，是传统以太网从未面对过的极限挑战。

26.1 AI时代的网络"不可能三角"

AI分布式训练的网络流量模式和传统数据中心有本质区别。传统的Web服务请求是"南北向"的——客户端发请求，服务器回响应，流量模式像潮水一样有涨有落。而AI训练流量主要是"东西向"的——GPU卡之间不停地进行All-Reduce操作，把每张卡计算出的梯度汇总并分发。这个过程中，任何一个包的丢失都会导致整个同步操作延迟，因为所有卡必须一起等待才能进入下一轮计算。

今天的AI训练集群面临三个核心要求——我称之为AI网络的"不可能三角"：

大带宽：一张GPU在训练时的计算吞吐量已经达到TB/s级别，网络必须提供足够的带宽才能让数据"喂饱"GPU。一个1000卡的集群，仅All-Reduce阶段就需要在几十毫秒内传输数百GB数据。

超低延迟：每一次All-Reduce的延迟直接拖慢整个训练周期。如果一次All-Reduce需要50ms，而模型每次迭代需要做10次All-Reduce，那就是500ms的额外开销——乘以数十万次迭代，训练时间将成倍增长。

零丢包：在TCP通信中一个丢包就可能引发几百毫秒的重传超时（RTO），而AI训练中使用的RDMA（Remote Direct Memory Access）对丢包更加敏感——任何丢包都会导致整个通信组阻塞，直到重传完成。对于采用Go-Back-N重传机制的RoCEv2（RDMA over Converged Ethernet），一个丢包意味着后面所有包都要重传。

这"三个角"同时满足，传统以太网显然做不到。别说同时满足，单是"零丢包"这一条就已经卡住了绝大多数传统方案。

真实案例：PFC风暴摧毁千卡训练：2023年，某头部AI公司在部署千卡A100集群时遇到了一个令人头疼的问题：使用RoCEv2进行GPU通信，集群在稳定运行几小时后突然出现大规模训练中断。排查发现，根源是PFC（优先级流控）死锁——一台交换机的缓存耗尽后发送PFC暂停帧，逐级反压最终导致整个网络的所有队列都处于暂停状态。更糟糕的是，PFC风暴触发后，某些交换机的CPU被中断风暴打满，连SSH都连不上去，只能硬重启。这家公司后来不得不重写整个流控策略，将PFC队列从8个减少到3个，并大规模调整缓冲区水位——即便如此，每天仍然会偶发一到两次训练中断。

这个案例暴露了RoCEv2在AI场景的核心问题。RoCEv2本质上是在标准以太网上强行"嫁接"RDMA能力，它依赖于PFC来实现无损传输。但PFC本身是一个粗糙的"逐跳反压"机制——一旦某台交换机缓存满了，它就向上游发暂停帧，上游缓存满了就继续向上游发，最终形成"树形反压"，整个网络的所有流量都被阻塞。这好比你在高速公路上遇到堵车，结果不是大家慢慢挪，而是整条公路的所有入口全部被交警强制关闭——一个出口堵了，整个路网瘫痪。

26.2 RoCEv2的三宗罪

除了PFC死锁，RoCEv2在AI场景还有另外两个致命问题。

第一宗罪：ECMP哈希冲突。RoCEv2使用标准的ECMP（Equal Cost Multi-Path）在多条链路之间分发流量。ECMP的典型实现是对五元组（源IP、目的IP、源端口、目的端口、协议）做哈希，然后根据哈希值选择路径。问题是，AI训练的All-Reduce通信中，大量的流具有相同的五元组——因为它们本质上是在同一对GPU之间通信。这导致所有流量都被哈希到同一条链路上，其他链路处于空闲状态。换句话说，你花大价钱布设了8条100G链路，实际只用了1条的带宽。

第二宗罪：Go-Back-N重传。当RoCEv2检测到丢包时，它不会只重传丢失的那个包，而是重传从这个包之后的所有包——这就是Go-Back-N的含义。假设发送了100个包，第50个丢了，接收方会丢弃第51-100号包，并要求发送方从第50号开始全部重发。这意味着一个丢包导致51个包的重传开销。在高负载场景下，连续的丢包会触发雪崩式的重传效率下降。

第三宗罪：PFC死锁。前面已经讲过了——PFC的设计是"头痛医头"式的：哪里缓存满了就暂停哪里。在AI这种"全对全"（All-to-All）通信模式下，几乎每台交换机的每个端口都在高速收发，PFC风暴几乎不可避免。

这三宗罪叠加的后果是：即便是精心调优的RoCEv2网络，在千卡以上的AI集群中，有效带宽利用率通常只有40%-60%。换句话说，你买了100G的网卡，实际只能用40G-60G。

26.3 Ultra Ethernet Consortium的诞生

面对AI训练的"不可能三角"，业界意识到：小修小补已经不够了。RoCEv2是2014年前后的技术，那时候没人想到GPU集群会发展到十万卡规模。我们需要一种从零设计的、专门为AI和高性能计算优化的以太网方案。

2023年，Linux Foundation联合多家科技巨头成立了Ultra Ethernet Consortium（UEC）——超以太网联盟。创始成员包括AMD、Arista、Broadcom、Cisco、Eviden（原Atos）、HPE、Intel、Meta、Microsoft等。到2025年6月，UEC正式发布了1.0规范，定义了一套完整的超以太网体系架构。

UEC的核心理念不是另起炉灶，而是在标准以太网的基础上，对数据链路层、传输层和软件接口进行深度优化，使以太网能够满足AI训练对网络的极致要求。你可以把UEC理解成一个"AI特化版"的以太网协议栈。

关键认知：UEC不是要替代以太网，而是要把以太网改造成一个更好的"AI网络底座"。它保留了以太网的所有优点——开放标准、生态系统成熟、成本优势——同时解决了RoCEv2在AI场景的固有缺陷。这也是为什么它叫"超以太网"而非"新以太网"——它超越了传统以太网的能力边界，但依然是以太网家族的一员。

26.4 UEC的核心创新

26.4.1 LLR：链路层重传

LLR（Link Layer Retransmission）是UEC最关键的创新之一。它的思路非常符合工程师的直觉：在哪里出问题，就在哪里修复。

传统RoCEv2的丢包重传是端到端的——GPU A发到GPU B，中间经过5台交换机，如果中间某台交换机发生了丢包，最终由GPU B发现序列号不连续后通知GPU A重传。这个过程涉及到端到端的往返延迟，而且重传范围是整个路径。

LLR则把这个逻辑下放到了每一跳。每一台交换机都具备检测丢包和重传的能力。如果Switch 2发现从Switch 1收到的包有缺失，它直接向Switch 1请求重传丢失的包——不需要等到最终目的地才发现问题。这带来了两个巨大的好处：

第一，修复延迟大幅降低。端到端重传需要跨越整个网络路径，而逐跳重传只需要在相邻两台交换机之间完成。在典型的AI集群中，相邻交换机的RTT（往返时间）在1微秒以内，而端到端RTT可能在10-50微秒。

第二，重传范围极小。LLR只重传真正丢失的那个包（具体来说是一个信元，我们后面会讲），而不是从丢失点之后的所有数据。这对于AI训练这种"大吞吐、对延迟敏感"的场景至关重要。

26.4.2 CBFC：基于信用的流控

如果说LLR解决了"丢了怎么办"的问题，那么CBFC（Credit-Based Flow Control）解决的是"怎么让它不丢"的问题。

PFC的问题是"被动反压"——缓存满了才通知上游暂停。而CBFC采用了一种完全不同的思路：主动授信。

在CBFC中，下游交换机定期向上游交换机通告自己可用的接收缓存大小（以信用额度表示）。上游在发送数据前，需要确保自己有足够的信用额度。如果信用额度用完，就必须等待，直到下游发送新的信用更新。这个过程是连续的、细粒度的，而不是像PFC那样"一刀切"式的暂停/恢复。

想象一下：PFC就像一个粗放的"水库管理"——水位超过警戒线了才开闸泄洪；CBFC则像是精细的"城市供水"——每家每户根据水表数据实时调整供水压力。粗放管理对付普通场景也许够用，但在AI训练这种对每个数据包的到达时间都极度敏感的场景中，精细控制是唯一的出路。

26.4.3 报文喷洒（Packet Spraying）

这是UEC解决ECMP哈希冲突问题的方案，也是最优雅的设计之一。

传统ECMP的哈希方式意味着同一个流的所有包都走同一条路径。报文喷洒（Packet Spraying）则完全打破了这个约束——每个数据包独立选择路径，而不是每个流选择一条路径。在UEC中，数据包被切分为固定大小的信元（Cells），每个信元通过独立的路径到达目的地，在接收端重新组装。

这个设计的效果非常惊人：

假设你的Spine-Leaf网络有8条等价路径，传统ECMP上可能只有2-3条路径能有效利用（因为哈希冲突），带宽利用率不到40%。而采用报文喷洒后，所有8条路径被均匀填满，带宽利用率接近100%。这对于AI训练中的All-Reduce流量尤其重要——大量数据的均匀分布可以最大限度地缩短完成时间。

传统ECMP（哈希冲突严重）：
        GPU0 ──── Switch ─┬─ Leaf1 ──── Spine1 ──── Leaf3 ──── GPU2  (流量集中)
                         ├─ Leaf1 ──── Spine2 ──── Leaf3 ──── GPU2
                         ├─ Leaf1 ──── Spine1 ──── Leaf4 ──── GPU3  (轻度使用)
                         └─ Leaf1 ──── Spine2 ──── Leaf4 ──── GPU3  (轻度使用)
        有效带宽利用率：约40%

        UEC报文喷洒（信元级负载均衡）：
        GPU0 ──── Switch ─┬─ Leaf1 ──── Spine1 ──── Leaf3 ──── GPU2  (Cell 1)
                         ├─ Leaf1 ──── Spine2 ──── Leaf3 ──── GPU2  (Cell 2)
                         ├─ Leaf1 ──── Spine1 ──── Leaf4 ──── GPU3  (Cell 3)
                         └─ Leaf1 ──── Spine2 ──── Leaf4 ──── GPU3  (Cell 4)
        有效带宽利用率：约98%

26.4.4 选择性重传

报文喷洒解决了ECMP哈希冲突的问题，但它带来了一个新的挑战：由于同一个流的不同Cell走不同的路径，它们到达目的地时可能是乱序的。接收方需要一种机制来检测哪些Cell丢失了、哪些Cell到了，并且只请求重传真正丢失的那些Cell。

这就是选择性重传（Selective Retransmission）的用武之地。UEC在接收端维护一个位图（bitmap），记录每个Cell的接收状态。超时后，接收方只发送一个包含丢失Cell序列号的NACK（Negative ACK），发送方只补发这些特定的Cell。这和TCP的Selective ACK（SACK）思路类似，但粒度更细（Cell级别而非包级别），延迟要求也更苛刻。

选择性重传 + LLR的组合，使得UEC在出现丢包时，只需要在相邻交换机之间重传几个Cell，延迟增加不超过几微秒。而传统RoCEv2在同样场景下可能触发Go-Back-N重传全部数据，延迟暴增几百毫秒。两者的差距在AI训练场景中是天壤之别。

26.5 UEC vs RoCEv2 vs InfiniBand

说到AI训练的互联技术，不能不提 InfiniBand（IB）。IB在HPC和AI领域一直是RoCEv2的强劲对手。UEC 1.0的出现，让这场对决更加激烈。我们来看看三者的关键指标对比：

对比维度 RoCEv2 InfiniBand NDR UEC 1.0
标准化组织 IBTA IBTA Linux Foundation / UEC
开放程度 开放（基于以太网） 非开放（NVIDIA专有生态） 开放（基于以太网）
丢包恢复机制 Go-Back-N + PFC 硬件级信用流控（几乎无丢包） LLR + 选择性重传 + CBFC
负载均衡 ECMP（流级别） 动态自适应路由 报文喷洒（信元级别）
最大带宽利用率 40-60% 80-95% 95-100%
端到端延迟 中等 极低 极低
生态成熟度 高 中（NVIDIA主导） 低（2025年刚发布）
硬件成本 低 极高 中等（以太网生态）
与标准以太网兼容 是（需特殊配置） 否（专用设备） 是（共享物理层）

从表中可以清晰地看到：RoCEv2的劣势主要在丢包恢复和负载均衡上；InfiniBand虽然性能出色，但封闭生态和高昂成本是其致命弱点；UEC则在保持以太网开放生态的同时，通过协议创新实现了媲美甚至超越InfiniBand的性能。

有意思的观察：InfiniBand原本是一种独立的技术标准，但在2020年NVIDIA收购Mellanox后，InfiniBand实际上变成了NVIDIA的"准专有"技术——虽然IBTA（InfiniBand Trade Association）仍然存在，但绝大多数InfiniBand设备只来自NVIDIA。相比之下，UEC从一开始就强调多厂商协作和开放标准，这让人想起当年DIX联盟将以太网开放化的历史。历史不会简单重复，但确实有惊人的相似。

26.6 中国GSE方案

在UEC之外，中国科技界也在积极探索面向AI的以太网增强方案。其中最值得关注的是GSE（Global Scheduling Ethernet）——全局调度以太网。GSE由中国计算机学会（CCF）和多家国内企业共同推进，核心思路是在UEC的基础上增加一个全局调度层。

GSE的架构可以这样理解：UEC解决了每一条链路的可靠性问题（通过LLR和CBFC）和链路的利用率问题（通过报文喷洒），但GSE认为还缺一个"大脑"——一个全局视角的调度器。这个调度器知道全局的拓扑和负载情况，可以做出最优的路径选择。具体来说，GSE引入了集中式调度器 + 分布式执行的混合架构：集中式调度器根据全网拓扑和流量矩阵计算最优路径，分布式执行引擎在交换机本地快速响应实际流量变化。

目前GSE还处于标准制定阶段，但已经引起了业界的广泛关注。它和UEC的关系更像是互补而非竞争——UEC解决下层链路问题，GSE解决上层调度问题。未来GSE可能作为UEC的上层扩展被集成到超以太网规范中。

26.7 超以太网的未来之路

UEC 1.0在2025年6月的发布标志着超以太网从"纸上谈兵"进入了"真实部署"阶段。但挑战依然存在：

硬件支持：UEC的LLR、CBFC、报文喷洒等功能都需要新一代交换芯片和网卡的支持。Broadcom、Marvell等厂商已经宣布了支持UEC的芯片计划，但真正的量产需要时间。

生态建设：从RDMA到libfabric到PyTorch的分布式通信后端——整个软件栈都需要适配UEC。社群正在推进libfabric和NCCL/UCC对UEC的支持。

与现有网络的共存：大多数数据中心已经部署了传统的RoCEv2网络，如何实现UEC与传统网络的平滑过渡和混合作业，是一个现实的工程问题。

尽管如此，方向已经明确：AI时代需要新的网络。而超以太网，正是以太网进化史上又一场自我革命。从1980年DIX标准到2025年UEC 1.0，以太网用四十五年的时间告诉我们：一种技术能够存活半个世纪，不是因为它的初始设计多么完美，而是因为它始终保持着开放和进化的能力。
