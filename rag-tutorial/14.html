第十四章：生成与引用溯源

大家好，我是你们的讲师。经过前面13章的学习，我们的RAG系统已经能做到：高效检索相关文档、对候选文档重排序、优化用户查询、精心组织上下文。现在来到了最后一步——生成最终的回答。

这一步看似简单：把检索到的文档和问题一起给LLM，让它生成答案不就完了？但现实远没有那么简单。生成阶段有两个核心挑战：一是如何控制LLM的输出行为（长度、风格、格式），二是更重要的——如何确保LLM的回答有据可查，而不是胡编乱造。这就是引用溯源（Citation）要解决的问题。

1. 生成参数控制

LLM的生成行为由一组参数控制。在RAG场景中，合理的参数设置可以显著提升回答质量：

参数 推荐值（RAG场景） 作用 调高/调低的影响
temperature 0.1 - 0.3 控制输出的随机性 高→更富创造性但也更容易幻觉
top_p 0.1 - 0.3 控制词汇采样的核大小 高→多样性增加，低→确定性增加
max_tokens 500 - 1500 最大输出长度 太长→容易跑题，太短→信息不足
frequency_penalty 0.0 - 0.3 减少重复内容 高→避免重复，低→允许重复
presence_penalty 0.0 - 0.2 鼓励谈论新话题 高→话题多样性，低→聚焦当前

在RAG场景中，我们追求的是忠实于检索到的文档，而不是创造力。因此建议使用偏低的temperature（0.1-0.3）和top_p（0.1-0.3），让LLM尽量选择最可能的token，减少"自由发挥"的空间。

2. 约束LLM：只使用上下文

这是RAG生成中最关键的一条指令。LLM有强大的内置知识，但如果它用自己的知识而非检索到的文档来回答，RAG就失去了意义。一个强约束的prompt模板：

你是一个严格基于检索文档的问答助手。你绝对不能使用自己的知识。

指令：
1. 仔细阅读下方提供的所有文档
2. 只根据这些文档中的信息回答问题
3. 如果文档中没有相关信息，回答："抱歉，提供的资料中未包含相关信息"
4. 不要补充任何不在文档中的细节
5. 不要使用你的训练数据中的知识

检索文档：
{% for doc in documents %}
[文档{{ loop.index }}] {{ doc.content }}
{% endfor %}

用户问题：{{ query }}

请严格按照以上要求回答：

这种"强硬"的约束方式在实际项目中证明非常有效。一个好的经验法则是：prompt中至少出现三次"只基于文档回答"的约束——一次在开头设置基调，一次在指令中间强化，一次在最后提醒。

3. 内联引用机制

引用溯源是RAG系统不可或缺的功能。它让用户知道回答的每个部分来自哪个文档，增加了回答的可信度和可验证性。以下是一个带内联引用的生成示例：

# 带引用的RAG生成实现

def generate_with_citations(query, documents, llm_client):
    # 构建prompt
    doc_sections = []
    for i, doc in enumerate(documents):
        doc_sections.append(f"[来源{i+1}] {doc['title']}\n{doc['content']}")

    prompt = f"""你是一个问答助手，回答时请标注信息来源。

检索到的文档：
{'---'.join(doc_sections)}

用户问题：{query}

回答格式要求：
- 每个事实后面用【来源X】标注来自哪个文档
- 如果多个文档包含相同信息，标注所有来源如【来源1】【来源3】
- 不要编造来源编号

回答："""

    response = llm_client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.1,
        max_tokens=1000,
    )
    return response.choices[0].message.content

# 示例输出：
# "Transformer模型的self-attention机制可以并行计算【来源1】，
# 这是它相比RNN的主要优势之一。同时，Transformer通过位置编码
# 来感知序列顺序信息【来源2】。"

documents = [
    {"title": "Transformer架构详解", "content": "Self-attention允许并行计算..."},
    {"title": "RNN vs Transformer", "content": "Transformer使用位置编码..."},
]
answer = generate_with_citations(
    "Transformer为什么比RNN快？",
    documents,
    llm_client
)
print(answer)

4. 置信度标记

有时候，LLM从文档中找到的信息只能部分回答用户的问题。这时，让LLM标注回答的置信度可以帮助用户判断信息的可靠性：

  • 高置信度 (Certain)：文档中有明确的、直接的答案。
  • 中等置信度 (Likely)：文档中有相关的、但并非直接回答的信息。
  • 低置信度 (Uncertain)：只能从文档中推断或完全没有相关信息。

置信度标记可以通过在prompt中要求LLM在回答前先做一个置信度评估来实现。例如：

回答前，请先评估：
- CERTAIN: 文档中有直接答案 → 直接回答
- LIKELY: 文档中有相关信息但无直接答案 → 基于信息回答并标注"推测"
- UNCERTAIN: 文档中无相关信息 → 回答"无法回答"

输出格式：[CERTAIN/LIKELY/UNCERTAIN] 你的回答

5. 结构化输出（JSON模式）

许多现代LLM支持JSON结构化输出模式。在RAG场景中，结构化输出可以让后续处理更加方便：

{
  "answer": "Transformer通过self-attention机制实现并行计算...",
  "citations": [
    {"source_id": 1, "relevance": "high", "excerpt": "Self-attention允许并行计算"},
    {"source_id": 2, "relevance": "medium", "excerpt": "Transformer使用位置编码"}
  ],
  "confidence": "high",
  "follow_up_questions": [
    "什么是位置编码？",
    "Self-attention的计算复杂度是多少？"
  ]
}

JSON输出的好处是：便于下游系统解析、可以进行自动化验证（检查引用编号是否有效）、便于前端渲染。

⚠️ 警惕编造（Fabrication）模式
即使你给了LLM检索到的文档，它仍然可能编造内容。以下是三种最常见的编造模式：
1. 来源编造：LLM使用了文档中没有的来源编号（如标注【来源5】，但只提供了3个文档）。应对方案：在prompt中明确文档总数，并在生成后做引用编号有效性校验。
2. 事实编造：LLM从文档中提取了部分事实，然后用自己的知识"填充"了缺失的部分。应对方案：使用低temperature，并在生成后做事实一致性检查。
3. 逻辑编造：LLM虽然没有编造具体事实，但自己"脑补"了事实之间的因果关系。应对方案：prompt中明确要求"只陈述文档中存在的事实，不要添加因果关系"。
记住：LLM的天性是"帮用户找到答案"而不是"验证信息是否存在"，所以它会倾向于"创造"一个看似合理的答案而不是说"不知道"。你是最后一道防线。

6. 常见的生成失败模式及应对

失败模式 表现 应对方案
幻觉（Hallucination） LLM生成了文档中不存在的信息 降低temperature + 引用约束 + 生成后验证
照搬原文 LLM直接复制文档内容，没有做整合 在prompt中要求"用自己的话重新组织"
信息遗漏 LLM只使用了部分文档，忽略了其他相关文档 prompt中明确"请综合所有提供的文档"
过度生成 回答远超过用户问的范围 设置max_tokens + prompt要求简洁
答非所问 生成的内容与问题不直接相关 增加重排序精度 + prompt强化对齐

踩坑故事：当LLM以为自己是专家

一个让我印象深刻的案例。做一个医疗咨询RAG系统，知识库里有经过严格审核的医学文献。一切测试正常。结果上线第一天，用户问"糖尿病能治好吗"，LLM回答："根据文献，糖尿病目前无法根治，但可以通过生活方式干预和药物治疗控制血糖【来源1】【来源2】。"

看起来很完美，对吧？但医学审核团队发现了一个问题——LLM在回答中补充了一句："建议患者每天运动30分钟以上，控制碳水化合物摄入。"这句话本身是正确的，但它不在任何检索到的文档中。LLM用了自己的医学知识做了"补充"。

严格来说，这不算是"幻觉"——因为信息是正确的。但在法律法规上，这属于未经审核的医疗建议，一旦出问题公司要承担法律责任。解决方案是：在prompt的约束部分增加"不得添加任何检索文档中不存在的信息，即使你认为该信息是正确的"。同时，增加一个生成后校验步骤，用NLI（自然语言推理）模型验证生成的每个事实是否都能在文档中找到支持。

7. 生成后处理：质量护栏

在将答案返回给用户之前，我们可以设置一些轻量级的质量检查：

  • 引用校验：检查所有引用的来源编号是否都在1-N范围内（N为提供的文档数）。
  • 长度检查：如果答案过长或过短，触发告警。
  • 关键词检测：检测是否包含危险或敏感词汇。
  • 语义一致性：用嵌入模型计算答案和文档集合的语义相似度，过低则说明答案可能偏离了文档。

本章小结

生成与引用溯源是RAG系统的最终输出环节。通过合理设置生成参数（低temperature、低top_p）、强约束LLM只使用检索文档、实现内联引用机制、标注置信度、使用结构化输出，我们可以显著提升RAG系统生成回答的准确性和可信度。特别是引用溯源——它让RAG系统的输出不再是"黑箱"，而是每个事实都有据可查。

这一章也是RAG核心技术模块的收尾。从第8章到第14章，我们完整走过了从检索到生成的整个流程：稀疏检索 → 稠密检索 → 混合检索 → 重排序 → 查询优化 → 上下文增强 → 生成与引用。恭喜你，你已经掌握了RAG系统的全部核心技术！
