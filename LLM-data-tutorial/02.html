互联网的图书馆：Common Crawl 是什么

大家好，这一章我们正式开干。上一章我们聊了数据工程的全景图，现在让我们把目光对准整个流水线的起点——互联网上最大的公开网页数据集之一：Common Crawl。

Common Crawl 是一个非营利组织，从2007年开始就在做一件简单而疯狂的事：每月爬一次互联网，然后把爬取结果免费公开。到2025年，Common Crawl 的历史存储总量已经超过 20PB，每个月的增量大约 300TB。这意味着什么？意味着你只需要几行代码，就能拿到这个月互联网上数十亿网页的原始快照。

为什么几乎所有的大语言模型训练都离不开 Common Crawl？原因很简单：它是目前唯一公开可用的、规模达到互联网量级的网页数据集。Google 有自己的搜索索引、Meta 有自己的爬虫——但这些都是不公开的。对于那些没有自家爬虫的团队，Common Crawl 就是唯一的选项。OpenAI 的 GPT-3、Meta 的 Llama 系列、HuggingFace 的 FineWeb、AI2 的 Dolma，无一不是站在 Common Crawl 的肩膀上构建的。

一个重要的事实： 根据 EleutherAI 在2024年的一项统计，公开可训练的 LLM 训练数据中，超过 80% 的 token 最终都来源于 Common Crawl。无论你的流水线多么复杂，起点都是这里。

WARC / WAT / WET 三件套

Common Crawl 的数据不是简单的一堆 HTML 文件。每个爬取周期会产生三种格式的文件，分别对应不同的使用场景：

格式全称内容典型大小用途
WARCWeb ARChive原始 HTTP 响应（header + body）~1GB 每文件（gzip 压缩）完整的数据归档，需要原始资源的场景
WATWeb Archive TransformationWARC 的元数据摘要~100MB 每文件快速检索响应时间、Content-Type、链接关系等
WETWeb Archive Text纯文本抽取（去掉了所有 HTML 标签）~200MB 每文件只需要文本内容的场景，省去解析 HTML 的成本

WARC 是最原始的格式，保留了服务器返回的完整 HTTP 响应，包括响应头、Cookie、HTML 正文。WAT 是 WARC 的元数据版本，只包含结构化信息（如响应时间、Content-Type、提取的链接等），适合做统计分析。WET 是最精简的版本，提前帮你做了一步 HTML 标签剥离，只留下纯文本。

大多数数据工程流水线会选择下载 WET 文件来减少处理量，但如果你需要精确判断页面结构、或者想自己做更精细的正文提取，你就得用 WARC。行业内，FineWeb 同时使用了 WARC 和 WET——他们先用 WET 做快速过滤，再回到 WARC 做精确的正文提取。

获取数据的方式

Common Crawl 的数据全部托管在 AWS S3 上，任何人都可以免费下载。唯一的成本是你的网络带宽。

每个爬取周期（crawl）都有一个唯一的标识符，格式像 CC-MAIN-2025-40。你可以通过索引文件来获取某个爬取周期下所有 WARC 文件的路径列表：

快速下载 WARC 文件列表：
wget https://data.commoncrawl.org/crawl-data/CC-MAIN-2025-40/warc.paths.gz
gunzip warc.paths.gz
解压后每行就是一个 WARC 文件的相对路径，前面拼接上 https://data.commoncrawl.org/ 就是完整的下载地址。

不同的项目选择不同的快照策略，这里就体现了数据广度和数据新鲜度的权衡：

  • CCNet（2020年，Meta）：只使用了 2019年2月 的一个快照，然后用去重和质量过滤来保证数据质量。当时计算资源有限，他们用 5000 CPU core 花了 8.5 小时处理一个快照。
  • FineWeb（2024年，HuggingFace）：使用了从2013年到2024年的 96 个快照，覆盖了超过十年的互联网数据。更多的快照意味着更大的覆盖范围和更丰富的信息量，但也意味着更高的存储成本和更复杂的去重挑战。
  • Llama-3（2024年，Meta）：据说使用了超过 200 个快照，加上内部的网页索引数据，才能在 2024 年训练出 15T token 的史诗级语料。
数据新鲜度 vs 数据规模： 更多快照意味着更大覆盖、更多样性，但也带来更高的存储成本和去重难度。如果你只需要特定时间范围内的数据，可以只选择相关月份的快照；如果你的目标是最大化数据量，就尽可能多选。FineWeb 的 96 个快照方案是目前公开的最佳实践参考。

WARC 解析实战

拿到 WARC 文件后，怎么把它变成可用的文本数据？Python 社区最常用的工具是 warcio 这个库，由互联网档案馆（Internet Archive）维护。下面是一个完整的解析示例：

import warcio
from warcio.archiveiterator import ArchiveIterator
import gzip

# 打开压缩的 WARC 文件
with gzip.open("CC-MAIN-2025-40-00000.warc.gz", "rb") as f:
    for record in ArchiveIterator(f):
        # 只处理响应记录（response），跳过请求记录（request）
        if record.rec_type == "response":
            # 提取 URL
            url = record.rec_headers.get_header("WARC-Target-URI")

            # 提取 HTTP 状态码
            status = record.http_headers.get_statuscode()
            if status != "200":
                continue  # 只处理正常响应

            # 提取 HTTP 响应头
            content_type = record.http_headers.get_header("Content-Type")

            # 提取 HTML 正文（可能需要解压）
            payload = record.content_stream().read()
            html_text = payload.decode("utf-8", errors="replace")

            # 这里后续会接入 trafilatura 做正文提取
            print(f"URL: {url}")
            print(f"Content-Type: {content_type}")
            print(f"HTML 长度: {len(html_text)} 字符")

在实际的大规模处理中，WARC 文件会被分割成大小均匀的分片。CCNet 团队的做法是将 WARC 文件按 ~5GB 的大小进行分片，然后在 Spark 集群上并行处理。他们使用了 5000 个 CPU core，平均 8.5 小时处理完一个爬取周期的全部数据。2025年有了 Ray 和更强大的分布式框架，处理效率已经大幅提升——同样的工作 Spark 需要 8.5 小时，用 Ray 优化后可以缩短到 2-3 小时。

WARC 处理的陷阱

WARC 看起来简单粗暴，实际用起来坑不少。我整理了几个最常见的踩坑点：

踩坑一：同一个 URL 在不同快照中的内容可能完全不同。 页面会更新、改版、甚至被完全替换成不同主题。如果你的流水线一次性处理多个快照，要格外注意跨快照的版本管理。一个 URL 在2023年可能是一篇技术文章，到了2025年变成了一个 404 页面——你需要在流水线中记录快照时间戳，而不是只记录 URL。

编码问题： 虽然 Common Crawl 在 WET 中统一用 UTF-8 编码，但 WARC 中保留了原始响应。部分站点使用 GBK、Shift-JIS、Latin-1 等非 UTF-8 编码，解析时如果不做编码检测就会乱码。一个常见的做法是用 charset 库或 ftfy 库来自动检测和修复编码。

JS 渲染缺失： Common Crawl 的爬虫不执行 JavaScript。这意味着大量依赖 JS 渲染的现代 Web 应用（React、Vue 等单页应用）在 WARC 中几乎不包含有意义的文本内容。2025年的互联网中，超过 40% 的网站使用了某种形式的客户端渲染——这些站点在 Common Crawl 中基本上就是空的。不同团队对此的态度不同：有些直接丢弃，认为这些内容不"accessible"；有些则尝试在数据清洗后通过其他方式补充这些内容。这个问题至今没有很好的公开解决方案。

还有一个很多人忽略的问题：WARC 文件中混杂了大量的 自动生成页面——搜索结果的 URL 参数变体、CMS 生成的标签聚合页面、社交媒体自动同步的重复内容。这些在 WARC 级别很难区分，需要在后续的去重步骤中解决。

总之，Common Crawl 是一个宝库，但需要精心挖掘。拿到原始的 WARC 文件只是第一步——下一步是怎么从这些铺天盖地的 HTML 中提取出真正有用的正文内容。这正是我们下一章要讲的主题。
