第8章 正向运动学

大家好，我是你们的讲师。今天我们要进入机器人学中最重要、也最"踏实"的一个话题——正向运动学（Forward Kinematics, FK）。如果你问一个干了二十年的机器人工程师"你最核心的技能是什么"，他大概率会回答："正向运动学，是根。"

为什么FK如此重要？因为机器人要控制自己的末端执行器到达某个位置，首先必须知道一个基本映射关系：给定一组关节角度，末端执行器在哪儿？这个从关节空间（Joint Space）到任务空间（Task Space）的映射，就是FK。不管是后面要学的逆向运动学、微分运动学、动力学还是轨迹规划，没有一个能离开FK这个地基。可以毫不夸张地说：FK的精度，决定了整个机器人算法栈的上限。

8.1 FK的定义与意义

我们先给FK一个精确但直观的定义。设机器人的关节角度为向量 q = [q₁, q₂, ..., qₙ]ᵀ，末端执行器的位姿（位置+姿态）为 T ∈ SE(3)。FK就是这样一个函数：

FK: Q → SE(3)
    q ↦ T(q)

这里的 SE(3) 是三维空间中所有刚体变换的集合，包含一个3×3旋转矩阵和一个3×1平移向量，通常用一个4×4齐次变换矩阵表示。把关节角度映射成末端位姿，这听起来简单——但背后的数学和技巧，能让一个博士生研究好几年。

FK的意义可以归纳为三点：

  • 底层基础：FK是所有运动学算法的前提。没有FK，逆向运动学连目标函数都定义不了；没有FK，雅可比矩阵无从谈起；没有FK，动力学方程的符号推导也缺乏基准。
  • 精度标尺：机器人的绝对定位精度最终取决于FK模型的精度。如果FK模型中连杆长度错了1mm，那末端定位就一定会偏1mm以上（通常在远端的累积误差更大）。
  • 仿真桥梁：在仿真中，FK是连接虚拟关节与虚拟末端位姿的唯一桥梁。Isaac Sim、MuJoCo等每一步物理仿真都需要调用FK。

核心认知：FK的输入是"机器人本身的状态"（关节角度），输出是"末端在空间中的位置"。它不涉及力、不涉及速度、不涉及控制——它就是一个纯粹的几何学映射。理解这一点，你就不会在后续学习IK和动力学时混淆。

8.2 DH参数法实战

标准的FK建模方法就是Denavit-Hartenberg（DH）参数法。它用四个参数来描述相邻两个连杆坐标系之间的变换：α（连杆扭转角）、a（连杆长度）、d（连杆偏距）、θ（关节角）。这四个参数可以唯一确定一个4×4齐次变换矩阵。

DH参数法的核心在于坐标系放置。规则说起来并不复杂：

  • Z轴沿关节轴线方向
  • X轴沿相邻Z轴的公垂线方向，从当前关节指向下一个关节
  • Y轴由右手定则确定

但在实际操作中，坐标系放置恰恰是最容易出错的地方。我见过一个团队花了两周时间调试机器人，始终发现末端位置差了3mm，最后发现是某两个相邻关节的Z轴方向搞反了——一个人用右手法则画了坐标系，另一个人用了左手法则。这种"符号错误"排查起来极其痛苦。

下面是一个典型六轴机器人（类比KUKA KR系列）的DH参数表：

关节 i αᵢ₋₁（连杆扭转角） aᵢ₋₁（连杆长度） dᵢ（连杆偏距） θᵢ（关节角）
1 0° 0 d₁ θ₁
2 -90° a₁ 0 θ₂
3 0° a₂ 0 θ₃
4 -90° a₃ d₄ θ₄
5 90° 0 0 θ₅
6 -90° 0 d₆ θ₆

注意上表中的参数并非标准值——实际的a₁、a₂、d₁、d₄、d₆取决于具体机器人型号。比如UR5的d₁=0.089m、a₂=-0.425m；而KUKA KR10的几何参数就完全不同。这也提醒我们：DH参数是针对具体机器人的，不存在"通用DH表"。当你面对一台新机器人的时候，动手标定DH参数是第一步。

实战技巧：在分配DH参数时，我强烈建议你在纸上画清楚每个关节的坐标系——先画出Z轴，再画X轴（公垂线方向），最后确定Y轴。很多经验丰富的工程师也是这么做的。永远不要只在脑子里想坐标系，画出来才能发现问题。

8.3 变换矩阵递推

有了DH参数，接下来就是通过齐次变换矩阵的链式乘法来递推末端位姿。每个相邻连杆的变换矩阵为：

T = Rotₓ(α) · Transₓ(a) · Trans_z(d) · Rot_z(θ)

    展开为4×4矩阵：
    | cosθ   -sinθ·cosα   sinθ·sinα   a·cosθ |
    | sinθ    cosθ·cosα  -cosθ·sinα   a·sinθ |
    |   0        sinα         cosα        d    |
    |   0          0            0          1   |

从基座到末端的完整变换就是：

T₀→ₙ = T₀→₁ · T₁→₂ · ... · Tₙ₋₁→ₙ

在Python中手动实现非常直观：

import numpy as np

def dh_transform(alpha, a, d, theta):
    """返回相邻连杆的齐次变换矩阵"""
    ct = np.cos(theta)
    st = np.sin(theta)
    ca = np.cos(alpha)
    sa = np.sin(alpha)
    return np.array([
        [ct,  -st*ca,  st*sa,  a*ct],
        [st,   ct*ca, -ct*sa,  a*st],
        [0,      sa,     ca,     d ],
        [0,      0,      0,     1 ]
    ])

# 假设已定义DH参数表
dh_params = [
    (0,       0,     d1, theta1),
    (-np.pi/2, a1,   0,  theta2),
    (0,        a2,   0,  theta3),
    (-np.pi/2, a3,   d4, theta4),
    (np.pi/2, 0,     0,  theta5),
    (-np.pi/2, 0,    d6, theta6),
]

T = np.eye(4)
for alpha, a, d, theta in dh_params:
    T = T @ dh_transform(alpha, a, d, theta)

# T 即为末端位姿
print("末端位置:", T[:3, 3])
print("末端旋转矩阵:\n", T[:3, :3])

这段代码虽然简单，却是无数机器人的运动学引擎核心。一个很有意思的事实是：乘法顺序是 T @ dh_transform（左乘），这是因为我们在构建从基座到末端的链，每一步都相对于世界坐标系。

避坑指南：一个非常常见的错误是变换顺序搞反了。如果你用的是Modified DH（有些教科书遵循Craig的约定），变换矩阵的形式和乘法顺序与Standard DH不同。你可以在ROS的 kdl_parser 或Pinocchio中验证。我个人的建议是：始终使用你所用工具库的标准约定，URDF用的是Modified DH的变体，而大多数教科书讲的是Standard DH。混用会导致完全错误的结果。

8.4 Pinocchio库实战

手动实现DH参数法对理解原理非常有帮助，但在现代机器人开发中，我们通常使用成熟的运动学库。Pinocchio是法国INRIA开发的开源机器人动力学库，它从URDF文件自动解析运动学链并提供FK计算功能。在机器人社区中，Pinocchio已经成为事实标准之一。

使用Pinocchio计算FK只需几行代码：

import pinocchio as pin
import numpy as np

# 加载URDF模型
model = pin.buildModelFromUrdf("robot.urdf")
data = model.createData()

# 输入关节角度
q = np.array([0.0, -0.5, 0.8, 0.0, 0.5, 0.0])

# 计算正向运动学
pin.forwardKinematics(model, data, q)
pin.updateFramePlacements(model, data)  # 关键步骤！

# 获取末端执行器位姿
ee_id = model.getFrameId("tool0")
ee_pose = data.oMf[ee_id]  # 参考系原点 → 末端帧的变换

print("末端位置:", ee_pose.translation)
print("末端旋转:", ee_pose.rotation)

注意这里有一个非常重要的函数调用——pin.updateFramePlacements(model, data)。这是很多新手都会忘记的一步。forwardKinematics 只计算关节变换，而 updateFramePlacements 根据URDF中定义的相对位姿来填充所有frame（包括末端执行器、传感器安装座等）的绝对位姿。

我曾经亲眼见过一个项目，工程师花了两天时间查Bug，发现末端位姿明明看起来在动但就是不对——最后发现就是忘了调用 updateFramePlacements。这一点在官方文档中写得不够显眼，但我们在实际使用中必须形成肌肉记忆：每次调用forwardKinematics之后，紧跟updateFramePlacements。

最佳实践：在调试FK时，始终将Pinocchio的输出与手动DH计算的输出对比。如果两者不一致，大概率是你的DH参数定义有误，或者Pinocchio的frame索引搞错了。利用 model.frames 打印所有frame名称，确保你选中了正确的末端frame。

8.5 URDF建模入门

URDF（Unified Robot Description Format）是ROS生态下最通用的机器人描述格式。一个URDF文件通过XML描述机器人的连杆（link）和关节（joint）之间的树状拓扑关系。每个link可以包含视觉模型（visual）、碰撞模型（collision）和惯性参数（inertial）。

一个典型的URDF片段如下：

<robot name="my_robot">
  <link name="base_link">
    <visual>
      <geometry><box size="0.2 0.2 0.1"/></geometry>
      <origin xyz="0 0 0" rpy="0 0 0"/>
    </visual>
  </link>

  <joint name="joint1" type="revolute">
    <parent link="base_link"/>
    <child link="link_1"/>
    <origin xyz="0 0 0.1" rpy="0 0 0"/>
    <axis xyz="0 0 1"/>
    <limit lower="-3.14" upper="3.14" effort="300" velocity="3.14"/>
  </joint>

  <link name="link_1">
    <visual>
      <geometry><cylinder length="0.3" radius="0.05"/></geometry>
      <origin xyz="0 0 0.15" rpy="0 0 0"/>
    </visual>
    <inertial>
      <mass value="2.0"/>
      <inertia ixx="0.01" ixy="0" ixz="0" iyy="0.01" iyz="0" izz="0.01"/>
    </inertial>
  </link>
</robot>

从CAD模型到URDF的工作流通常是：CAD软件（SolidWorks/Onshape）→ 插件导出 URDF → 手动修正 → 在RViz或Pinocchio中验证FK。Fusion 360和Onshape都提供了URDF导出插件，但导出的URDF往往需要手动调整轴方向和坐标系，这是最费时的步骤。

实用工具：你可以用 check_urdf 工具（ROS自带）检查URDF的语法正确性，然后用 urdf_to_graphviz 生成拓扑图，直观地检查链接关系。在RViz中显示模型是最终的验证手段——如果模型在RViz中看起来是对的，FK大概率也是对的。

总结一下本章的核心链路：机器人几何 → DH参数 → 变换矩阵链 → 末端位姿。我们学习了FK的数学本质、DH参数的分配规则、齐次变换的链式递推、Pinocchio库的实战用法，以及URDF建模的基础。理解FK是整个运动学体系的第一块基石，它不涉及任何"反向推理"，纯粹是正向计算——从这个意义上说，FK是机器人的"本体感知"。
