第九章：稠密检索与语义搜索

大家好，我是你们的讲师。上一章我们聊了BM25和稀疏检索，它擅长精确匹配但天生无法理解语义。想象一下，你用中文问"怎么治感冒"，BM25能找到包含"感冒"的文档，但如果文档里写的是"上呼吸道感染的治疗方案"，BM25就无能为力了。这时候就需要稠密检索（Dense Retrieval）出场了。

稠密检索的核心思想是：把文本映射到一个连续向量空间中，让语义相近的文本在空间中彼此靠近。这样一来，即使查询和文档没有共同的关键词，只要它们的语义相近，就能被检索到。这可以说是RAG系统从"关键词匹配"走向"语义理解"的关键一步。

1. 稠密检索的原理

稠密检索的流程分为两阶段：

  • 编码阶段（Indexing）：将知识库中的每个文档通过一个编码器（Encoder）转化为固定维度的稠密向量（例如768维），存入向量数据库。
  • 查询阶段（Querying）：将用户查询用同样的编码器编码为向量，然后在向量数据库中搜索最相似的文档向量。

这里的核心是编码器——它必须能把"语义"编码进向量。最初的做法是用一个通用的BERT模型来做编码，但效果并不理想，因为BERT并不是针对检索任务训练的。于是就有了DPR（Dense Passage Retriever）。

2. DPR架构

DPR是Facebook AI在2020年提出的经典架构，它的关键创新在于：

  • 使用两个独立的BERT编码器——一个编码查询（Query Encoder），一个编码文档（Document Encoder）。
  • 通过对比学习（Contrastive Learning）训练，使相关查询-文档对的向量距离更近，不相关的距离更远。
  • 使用批内负采样（In-batch Negatives）高效生成训练样本。

虽然现在有了更好的模型（如BGE、E5、GTE），但DPR的设计思路依然是现代稠密检索系统的基础。

3. 相似度度量方法

将查询和文档编码为向量后，我们需要计算它们的相似度。常用的相似度度量有三种：

度量方法 公式 取值范围 特点 推荐场景
余弦相似度 (Cosine) cos(A,B) = A·B / (|A|×|B|) [-1, 1] 只关注方向，不受向量长度影响 最常用，通用场景
点积 (Dot Product) A·B = Σ a_i × b_i (-∞, +∞) 受向量长度影响，长度本身携带信息 向量已归一化时等价于余弦
欧氏距离 (Euclidean) ||A-B|| = √Σ (a_i-b_i)² [0, +∞) 值越小越相似，对尺度敏感 聚类分析、KNN场景
💡 小贴士：大多数现代嵌入模型（如BGE、text-embedding-3-small）默认输出归一化向量（L2 norm = 1）。在这种情况下，余弦相似度等价于点积，所以使用点积计算效率更高。如果你的嵌入模型没有归一化输出，建议先做归一化再存库。

4. 实战：使用SentenceTransformers + BGE

我们来看一个完整的稠密检索示例，使用目前社区非常流行的BGE（BAAI General Embedding）模型：

from sentence_transformers import SentenceTransformer
import numpy as np

# 1. 加载BGE模型
# BGE是BAAI发布的系列嵌入模型，支持中文
model = SentenceTransformer('BAAI/bge-large-zh-v1.5')

# 2. 编码知识库文档
documents = [
    "Python是一种高级编程语言，由Guido van Rossum创建。",
    "向量数据库用于高效存储和检索嵌入向量。",
    "RAG通过检索外部知识来增强大语言模型的生成能力。",
]
doc_embeddings = model.encode(documents, normalize_embeddings=True)

# 3. 编码用户查询
query = "什么是RAG技术？"
query_embedding = model.encode(query, normalize_embeddings=True)

# 4. 计算余弦相似度（向量已归一化，点积=余弦）
similarities = np.dot(doc_embeddings, query_embedding)

# 5. 获取Top-K结果
top_k = 2
top_indices = np.argsort(similarities)[-top_k:][::-1]
for idx in top_indices:
    print(f"文档: {documents[idx]}, 相似度: {similarities[idx]:.4f}")

# BGE特别要求：查询前加指令前缀
query_with_instruction = "为这个句子生成表示以用于检索相关文章：" + query
enhanced_embedding = model.encode(query_with_instruction, normalize_embeddings=True)
# 使用enhanced_embedding代替query_embedding重新计算
⚠️ 语义漂移（Semantic Drift）警告
稠密检索面临一个隐蔽的问题——语义漂移。当你的知识库持续更新，新加入的文档可能改变整个向量空间的分布。具体表现为：今天编码的"A"和明天编码的"B"即使语义相同，它们的向量也可能因为模型参数更新或编码批次不同而产生偏差。更隐蔽的是，当知识库包含大量相似文档时，它们的向量会挤在一起，导致检索结果高度重复、多样性降低。建议做法是：固定一个模型版本进行编码，不要频繁切换模型；对向量做PCA降维后再聚类分析，检查是否存在向量坍塌（Collapse）现象。

5. BGE的特殊使用模式

BGE模型系列（BAAI General Embedding）是目前中文社区最广泛使用的嵌入模型之一。使用它有几个特殊注意事项：

  • 查询指令前缀：BGE要求对查询（Query）侧添加指令前缀，例如 "为这个句子生成表示以用于检索相关文章："，但对文档侧不需要。如果不加，检索精度会下降3-5%。
  • 维度选择：bge-large-zh-v1.5 输出1024维向量。如果需要降低维度，可以在编码后使用PCA或Matryoshka表示学习。
  • 量化影响：BGE向量适合做量化（如int8量化），精度损失通常在1%以内，但内存降低4倍。

6. ANN（近似最近邻）搜索

当知识库规模达到百万级别时，暴力计算所有向量相似度（精确最近邻搜索）的耗时是不可接受的。这时需要近似最近邻（Approximate Nearest Neighbor, ANN）搜索。

主流的ANN算法包括：

  • HNSW（Hierarchical Navigable Small World）：基于图结构，搜索精度高，速度极快，但构建索引较慢且内存占用大。最推荐用于小到中等规模（百万级）。
  • IVF（Inverted File Index）：基于聚类和倒排，构建快，内存友好，但搜索精度略低于HNSW。
  • IVF-PQ（Product Quantization）：在IVF基础上对向量做乘积量化，大幅降低内存。适合超大规模（千万级以上），但会有量化精度损失。

踩坑故事：当"新能源汽车"变成了"电动汽车"

去年做一个汽车知识库项目，用户的查询是"新能源汽车的电池寿命"，稠密检索返回的结果全是"电动汽车电池维护"。看起来没问题对吧？但产品经理过来说——我们的知识库里还有"氢燃料电池车"的内容，为什么没有被检索到？

排查后发现：训练数据中"新能源汽车"和"电动车"经常一起出现，所以模型把二者编码得很近；但"氢燃料电池车"在训练数据中出现较少，它的向量落在了空间的另一个区域。这就是数据偏差导致的语义检索盲区。

解决方案：第一，我们在训练数据中人工增加了"新能源汽车≈氢燃料电池车"的配对数据；第二，在生产中采用了混合检索策略（下一章会详细讲），用BM25兜底。"新能源汽车"这个词在BM25中可以精确匹配到任何含该词的文档。这件事让我深刻认识到：稠密检索不是万能的，它需要和稀疏检索配合使用。

7. 稠密检索的局限与应对

理解了稠密检索的优势，我们也不能忽视它的局限：

  • 领域偏移（Domain Shift）：通用嵌入模型在你的垂直领域可能效果很差。建议在领域数据上做Fine-tuning。
  • 长文本编码：大多数模型最大输入长度是512 tokens（约500-700个中文字）。超长文档需要先分块再编码。
  • 语义漂移：如上文所述，知识库动态更新时的向量不一致问题需要重视。
  • 偶发的不合理结果：稠密检索偶尔会给出表面看似相关、实则无关的结果，这也是为什么需要重排序（第11章）。

本章小结

稠密检索通过语义向量实现了超越关键词匹配的深层理解。DPR架构奠定了现代稠密检索的基础，BGE、E5等模型降低了使用门槛，ANN算法使百万级检索成为可能。但稠密检索也有其固有限制，需要和稀疏检索互补使用。
