第26章 项目实战：智能客服系统

恭喜你走到这里！经过前面 25 章的系统学习，你已经掌握了 Agent 技术的全部核心技能——从基础概念到框架生态，从测试策略到安全治理。从这一章开始，我们将用两个完整的实战项目把前面所学融会贯通。

本章要搭建的是一个智能客服系统。别误会，这不是那种"您要查询什么？请按 1"的自动语音菜单，也不是简单的 FAQ 机器人。我们要构建的是一个真正能处理复杂工单、具备多轮对话能力、能检索企业知识库、并在必要时无缝移交给人类客服的 Agent 系统。想象一下：一个客户发来消息"我上周买的笔记本电脑蓝屏了，想退货"，Agent 需要先识别这是"售后-退货"类工单，然后查询订单信息，检索知识库中的退货政策，判断是否符合条件，再引导客户完成退货流程——如果遇到特殊情况，自动转接人工客服。

项目价值： 智能客服是 Agent 技术商业化最成熟的落地场景之一。据 Gartner 2026 年 Q1 报告，全球已有超过 40% 的中大型企业部署了 Agent 驱动的客服系统，平均降低了 35% 的人工客服成本。完成这个项目后，你将具备直接在真实业务中落地 Agent 客服系统的能力。

26.1 项目需求与系统设计

在动手写代码之前，我们先明确项目的需求范围和系统架构。一个生产级的智能客服系统需要具备以下核心能力：

需求模块 功能描述 优先级
工单分类 自动识别用户意图，将问题归类到正确的工单类型（售后、物流、账户、产品咨询等） P0
多轮对话 维护对话上下文，支持追问、澄清、信息收集等复杂交互 P0
知识库 RAG 基于企业知识库（产品手册、政策文档、FAQ）进行检索增强生成 P0
工单操作 查询订单状态、发起退款、修改地址、预约服务等后端操作 P1
情感检测 识别用户情绪（愤怒、失望、紧急），触发不同的响应策略 P1
人工接管 在 Agent 无法处理或用户要求时，无缝转接人工客服 P0

基于这些需求，我们选择的技术栈如下：

  • 框架： LangGraph —— 智能客服的工作流天然适合用状态图来建模，每个对话阶段对应一个图节点。
  • LLM： Claude Sonnet 或 GPT-4o —— 需要较强的推理能力和工具调用稳定性。
  • 向量数据库： Qdrant（轻量级，支持自托管）或 Pinecone（托管服务）。
  • Ebedding 模型： text-embedding-3-small 或 voyage-2。
  • 后端： FastAPI + WebSocket（用于实时对话）。
  • 可观测性： LangSmith + OpenTelemetry。
设计决策： 在框架选型上，这个项目选择了 LangGraph 而非其他框架。原因有三：第一，客服场景的状态转移逻辑复杂（分类→信息收集→检索→回应→升级），LangGraph 的状态图模型能清晰表达；第二，Human-in-the-loop 是 LangGraph 的内建能力，无需额外实现；第三，LangGraph 的持久化层（checkpointer）天然支持多轮对话的会话管理。

整个系统的架构可以分为三层：

这个架构的核心思想是"管道 + 网关"——每个 Node 负责一个独立环节，Agent 编排层作为智能网关协调整个流程。每个 Node 的输出都写入 LangGraph 的 State，供后续 Node 使用。

26.3 工单分类 Agent 实现

工单分类是整个系统的"入口"。用户的第一条消息决定了后续的整个处理路径。我们使用 LLM 的 Function Calling 能力来实现分类，而不是传统的关键词匹配或分类模型——因为 LLM 能理解语义层面的细微差别：

from typing import Literal
from pydantic import BaseModel
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver

# ---- 定义工单类型 ----
class TicketCategory(BaseModel):
    """工单分类结果"""
    category: Literal["售后_退货", "售后_维修", "物流查询",
                      "账户问题", "产品咨询", "投诉建议", "其他"]
    urgency: Literal["低", "中", "高", "紧急"]
    summary: str  # 一句话摘要
    customer_name: str | None = None
    order_id: str | None = None

# ---- 分类工具定义 ----
TICKET_CLASSIFY_TOOL = {
    "type": "function",
    "function": {
        "name": "classify_ticket",
        "description": "对用户输入的客服工单进行分类",
        "parameters": {
            "type": "object",
            "properties": {
                "category": {
                    "type": "string",
                    "enum": ["售后_退货", "售后_维修", "物流查询",
                             "账户问题", "产品咨询", "投诉建议", "其他"],
                    "description": "工单分类"
                },
                "urgency": {
                    "type": "string",
                    "enum": ["低", "中", "高", "紧急"],
                    "description": "紧急程度"
                },
                "summary": {
                    "type": "string",
                    "description": "问题摘要（20字以内）"
                },
                "order_id": {
                    "type": "string",
                    "description": "如果用户提到了订单号，提取出来"
                }
            },
            "required": ["category", "urgency", "summary"]
        }
    }
}

# ---- 分类节点 ----
def classify_node(state: dict) -> dict:
    """意图分类节点：分析用户输入，输出分类结果"""
    messages = state["messages"]
    user_message = messages[-1]["content"]

    response = llm.invoke(
        messages=[
            {"role": "system", "content": "你是一个智能客服工单分类专家。"
                                          "请分析用户消息，调用 classify_ticket 进行分类。"},
            {"role": "user", "content": user_message}
        ],
        tools=[TICKET_CLASSIFY_TOOL],
        tool_choice={"type": "function", "function": {"name": "classify_ticket"}}
    )

    # 解析 Function Calling 结果
    tool_call = response.tool_calls[0]
    category = tool_call["args"]

    state["ticket"] = category
    state["current_phase"] = "classified"
    return state
实战经验： 工单分类的准确率直接影响后续所有流程的质量。在我们的测试中，直接使用 LLM 的 Function Calling 做分类，在 7 个类别上的准确率可达 96% 以上。关键技巧是：category 的枚举值要具体且互斥（"售后_退货"比"售后问题"更精确），同时给 LLM 一个明确的中文描述。

26.4 多轮对话状态管理

客服场景的核心挑战之一是多轮对话的状态管理。用户可能在对话中不断补充信息、修改诉求、提出新问题。我们需要一个健壮的状态管理机制来追踪对话的完整上下文。

LangGraph 的 checkpointer 机制为这个需求提供了优雅的解决方案：

from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import StateGraph, END

# ---- 全局状态定义 ----
class ChatState(TypedDict):
    messages: list          # 对话消息历史
    ticket: dict | None     # 工单信息
    current_phase: str      # 当前阶段
    collected_info: dict    # 已收集的信息字段
    knowledge_results: list # 知识库检索结果
    agent_notes: str        # Agent 内部推理笔记
    handoff_requested: bool # 是否请求人工接管

# ---- 信息收集节点 ----
def info_collection_node(state: dict) -> dict:
    """主动向用户收集缺失信息"""
    ticket = state["ticket"]
    collected = state.get("collected_info", {})

    # 根据工单类型，判断需要收集哪些字段
    required_fields = {
        "售后_退货": ["order_id", "product_name", "reason", "purchase_date"],
        "售后_维修": ["order_id", "product_name", "fault_description"],
        "物流查询": ["order_id", "tracking_number"],
        "账户问题": ["account_name", "issue_type"],
        "产品咨询": ["product_name", "question_type"],
        "投诉建议": ["description"],
        "其他": ["description"]
    }

    missing = [f for f in required_fields.get(ticket["category"], [])
               if f not in collected]

    if not missing:
        state["current_phase"] = "info_complete"
        return state

    # 构造询问 Prompt
    prompt = f"""当前工单类别：{ticket['category']}
已收集信息：{json.dumps(collected, ensure_ascii=False)}
仍缺失字段：{', '.join(missing)}

请用友好的语气向用户询问缺失信息。一次最多问2个问题。
如果用户主动提供的信息不在列表中，也要接受并记录下来。"""

    response = llm.invoke([
        {"role": "system", "content": "你是一个客服对话Agent，负责收集用户信息。"},
        *state["messages"],
        {"role": "user", "content": prompt}
    ])

    state["messages"].append({"role": "assistant", "content": response.content})
    return state
避坑指南： 多轮对话最容易出现的问题有两个：一是"重复提问"——Agent 问过的问题用户已经回答，但因为状态管理不到位又问了第二遍。二是"语境丢失"——用户在回答中途突然切换话题，Agent 无法灵活适应。解决方案是：在每次写入状态时做去重检查，同时给 LLM 足够的 context 让它理解当前对话的全貌。

26.5 知识库 RAG 检索增强

企业客服离不开知识库。无论是产品手册、退货政策还是常见问题，Agent 都需要从知识库中检索相关信息来回答用户。这里我们使用 RAG（检索增强生成）模式：

from qdrant_client import QdrantClient
from openai import OpenAI

# ---- 初始化向量数据库客户端 ----
qdrant = QdrantClient(host="localhost", port=6333)
embed_client = OpenAI()  # 使用 text-embedding-3-small

COLLECTION_NAME = "customer_service_kb"

def retrieve_knowledge(query: str, top_k: int = 3) -> list[dict]:
    """从知识库中检索与查询最相关的文档片段"""
    # 1. 生成查询向量
    query_embedding = embed_client.embeddings.create(
        model="text-embedding-3-small",
        input=query
    ).data[0].embedding

    # 2. 向量检索
    results = qdrant.search(
        collection_name=COLLECTION_NAME,
        query_vector=query_embedding,
        limit=top_k,
        score_threshold=0.65  # 低于此阈值的忽略
    )

    return [{"text": r.payload["text"],
             "source": r.payload.get("source", ""),
             "score": r.score} for r in results]

# ---- RAG 检索节点 ----
def rag_retrieval_node(state: dict) -> dict:
    """RAG 检索节点"""
    # 获取最新的用户问题
    last_user_msg = None
    for msg in reversed(state["messages"]):
        if msg["role"] == "user":
            last_user_msg = msg["content"]
            break

    if not last_user_msg:
        state["current_phase"] = "respond"
        return state

    # 检索知识库
    knowledge = retrieve_knowledge(last_user_msg)

    # 将检索结果格式化后注入 context
    if knowledge:
        ctx_parts = []
        for i, doc in enumerate(knowledge, 1):
            ctx_parts.append(f"[{i}] 来源：{doc['source']}\n内容：{doc['text']}")
        knowledge_context = "\n\n" + "=" * 40 + "\n".join(ctx_parts) + "\n" + "=" * 40
        state["knowledge_results"] = knowledge
    else:
        knowledge_context = "\n\n（未找到相关知识库内容）"
        state["knowledge_results"] = []

    state["rag_context"] = knowledge_context
    state["current_phase"] = "retrieved"
    return state
知识库构建建议： 企业知识库的质量直接决定了 RAG 的效果。几个关键实践：（1）文档分块大小控制在 500-800 tokens，既保留上下文又不引入噪声；（2）每个块保留元数据（来源、部门、版本号），方便溯源；（3）对政策类文档使用"分层索引"——先检索到文档，再精确定位到具体段落；（4）定期更新向量索引，确保新政策在 1 小时内生效。

26.6 人工接管流程设计

再智能的 Agent 也有无法处理的情况。当遇到以下场景时，需要无缝转接人工客服：

  • 用户在连续三次对话中明确要求"转人工"
  • 情感检测模型判断用户情绪为"愤怒"或"紧急"
  • Agent 连续两次回答置信度低于阈值
  • 用户的问题涉及敏感操作（如大额退款、账号删除）
  • Agent 在 3 轮对话内未能成功收集必要信息
HANDOFF_CONDITIONS = {
    "user_requested": "用户明确要求转人工",
    "high_urgency": "紧急程度为'紧急'",
    "angry_sentiment": "检测到愤怒情绪",
    "low_confidence": "Agent连续低置信度",
    "sensitive_operation": "涉及敏感操作",
    "info_collection_failed": "信息收集超限失败"
}

def handoff_check_node(state: dict) -> dict:
    """检查是否需要人工接管"""
    ticket = state.get("ticket", {})
    messages = state["messages"]
    agent_cycle_count = state.get("agent_cycle_count", 0)

    reasons = []

    # 条件1：用户明确要求转人工
    last_user_msg = None
    for msg in reversed(messages):
        if msg["role"] == "user":
            last_user_msg = msg["content"].lower()
            break
    if last_user_msg and any(kw in last_user_msg
                             for kw in ["转人工", "找人工", "人工客服", "活人"]):
        reasons.append("user_requested")

    # 条件2：紧急程度为紧急
    if ticket and ticket.get("urgency") == "紧急":
        reasons.append("high_urgency")

    # 条件3：情感检测
    sentiment = state.get("sentiment", "neutral")
    if sentiment == "angry":
        reasons.append("angry_sentiment")

    # 条件4：Agent 循环次数过多（对话陷入僵局）
    if agent_cycle_count >= 3:
        reasons.append("info_collection_failed")

    state["handoff_reasons"] = reasons
    state["handoff_requested"] = len(reasons) > 0

    if state["handoff_requested"]:
        state["current_phase"] = "handoff"
    else:
        state["current_phase"] = "respond"

    return state

def handoff_node(state: dict) -> dict:
    """执行人工接管：生成交接摘要，转接客服工作台"""
    ticket = state["ticket"]
    reasons = state["handoff_reasons"]
    messages = state["messages"]

    # 生成交接摘要
    handoff_summary = f"""【工单转接摘要】
━━━━━━━━━━━━━━━━━━━━━━━━
类别：{ticket['category']}
摘要：{ticket['summary']}
紧急程度：{ticket['urgency']}
转接原因：{', '.join(HANDOFF_CONDITIONS[r] for r in reasons)}
━━━━━━━━━━━━━━━━━━━━━━━━
对话记录：
"""
    for msg in messages[-10:]:  # 最近10条消息
        role = "用户" if msg["role"] == "user" else "客服Agent"
        handoff_summary += f"{role}: {msg['content'][:200]}\n"

    # 在实际系统中，这里会调用客服工作台 API 创建转接工单
    # handoff_ticket_id = crm_api.create_handoff_ticket(handoff_summary)
    state["handoff_summary"] = handoff_summary
    state["handoff_complete"] = True

    response = llm.invoke([
        {"role": "system", "content": "你需要告知用户将转接人工客服，语气要真诚且安抚。"
                                      "告知用户已为他生成了问题摘要，人工客服将很快接入。"},
        {"role": "user", "content": "请告知用户转接人工客服。"}
    ])

    state["messages"].append({"role": "assistant", "content": response.content})
    return state
关键提醒： 人工接管不是"承认失败"，而是整体用户体验的一部分。好的接管流程应该让用户感觉"Agent 尽力了，人工接手也很顺畅"。实践中，我们建议在每个对话页面展示"转接进度条"——让用户知道他的问题已经被摘要、正在分配给合适的客服、预计等待时间。这能显著降低用户的焦虑感。

26.7 完整状态图编排

现在，我们把所有节点组装成一个完整的 LangGraph 状态图：

from langgraph.graph import StateGraph, END

# ---- 定义条件路由函数 ----
def route_after_classify(state: dict) -> str:
    """分类完成后决定下一个节点"""
    ticket = state.get("ticket", {})
    category = ticket.get("category", "")
    urgency = ticket.get("urgency", "")

    # 紧急工单直接走人工
    if urgency == "紧急":
        return "handoff_check"

    # 需要收集信息的类别
    info_intensive = ["售后_退货", "售后_维修", "账户问题"]
    return "info_collection" if category in info_intensive else "rag_retrieval"

def route_after_handoff_check(state: dict) -> str:
    """根据是否转接人工决定路由"""
    return "handoff" if state.get("handoff_requested") else "respond"

# ---- 构建状态图 ----
builder = StateGraph(ChatState)

# 注册节点
builder.add_node("classify", classify_node)
builder.add_node("info_collection", info_collection_node)
builder.add_node("rag_retrieval", rag_retrieval_node)
builder.add_node("handoff_check", handoff_check_node)
builder.add_node("handoff", handoff_node)
builder.add_node("respond", respond_node)

# 设置入口
builder.set_entry_point("classify")

# 添加边
builder.add_conditional_edges("classify", route_after_classify)
builder.add_edge("info_collection", "rag_retrieval")
builder.add_edge("rag_retrieval", "handoff_check")
builder.add_conditional_edges("handoff_check", route_after_handoff_check)
builder.add_edge("handoff", END)
builder.add_edge("respond", END)

# 编译图
checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# ---- 部署到 FastAPI ----
from fastapi import FastAPI, WebSocket
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()
app.add_middleware(CORSMiddleware, allow_origins=["*"],
                   allow_methods=["*"], allow_headers=["*"])

@app.websocket("/chat/{session_id}")
async def chat_websocket(websocket: WebSocket, session_id: str):
    await websocket.accept()
    config = {"configurable": {"thread_id": session_id}}

    while True:
        user_input = await websocket.receive_text()
        if user_input.lower() == "/exit":
            break

        # 调用 LangGraph 图
        for event in graph.stream(
            {"messages": [{"role": "user", "content": user_input}]},
            config,
            stream_mode="updates"
        ):
            # 将每一步的 Agent 回复推送回客户端
            for node_name, update in event.items():
                if "messages" in update:
                    last_msg = update["messages"][-1]
                    if last_msg["role"] == "assistant":
                        await websocket.send_json({
                            "type": "agent_response",
                            "node": node_name,
                            "content": last_msg["content"]
                        })

        # 发送阶段结束信号
        await websocket.send_json({"type": "phase_complete"})

26.8 部署与监控

生产部署需要考虑的关键点：

维度 方案 说明
容器化 Docker Compose Agent 服务 + Qdrant + Redis (会话缓存) 三容器部署
会话持久化 PostgreSQL + LangGraph Checkpointer 支持服务重启后恢复对话上下文
可观测性 LangSmith + Prometheus 追踪每次 Agent 调用的完整 Trace，监控 Token 消耗和延迟
限流 Redis Rate Limiter 每个用户每分钟最多 30 次调用
缓存 Semantic Cache 对重复问题（如"退货政策"）命中语义缓存，跳过 LLM 调用
灰度发布 路由规则 新版本 Agent 先分配 5% 流量，观察指标正常后全量
关键指标： 衡量智能客服系统运行效果的核心指标有三个：（1）自动解决率——Agent 成功处理的工单占比，目标 70%+；（2）人工接管率——需要转人工的比例，目标低于 30%；（3）客户满意度（CSAT）——对话结束后用户打分，目标 4.5/5 以上。建议在 Grafana 上建立这三个指标的实时看板。
