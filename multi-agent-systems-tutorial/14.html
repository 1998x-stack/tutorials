第14章 MCP协议：工具层的USB标准

在开始本章之前，我想邀请你做一个小实验。打开你的代码编辑器，创建一个新文件，写一个函数——功能是从PostgreSQL数据库中查询用户信息。现在，让你的AI助手（比如Claude或Copilot）帮你完成这个函数。你一定经历过这样的场景：AI写了正确的SQL查询，但连接数据库的方式和你预想的不同——它假设了某种数据库驱动、某种连接字符串格式、某种错误处理方式。每次换一个项目，这些"约定"都要重新对齐。

这个小实验揭示了Agent工具使用中一个根深蒂固的问题：每个Agent和每个工具之间的交互都是"手工定制"的。你是通过Function Calling把工具描述塞进Prompt的，工具是硬编码在Agent代码里的，换一个工具就要改一次Prompt和代码。这不就像在USB标准出现之前，每个外设都需要专门的接口吗？

MCP（Model Context Protocol）就是来解决这个问题的。Anthropic在2024年底开源了MCP，目标明确：成为AI Agent工具层的"USB标准"。到了2026年，MCP已经从Anthropic的一个项目变成了W3C的标准提案，SDK月下载量超过9700万次，社区贡献了超过10,000个MCP服务器。这不是一个"值得关注的协议"——而是Agent生态的事实标准。

14.1 MCP协议架构：JSON-RPC 2.0之上

MCP构建在JSON-RPC 2.0协议之上。JSON-RPC是一个轻量级的远程调用协议，类似GraphQL但更简单——它定义了一种标准的JSON格式来发起远程过程调用。MCP在此基础上增加了Agent工具使用所需的语义层。

MCP的核心架构遵循Client-Server模型：

  • MCP Client（通常是Agent或Agent框架）：发现可用的工具、发起调用、处理结果。
  • MCP Server（封装了具体能力的服务）：注册工具、接收调用请求、执行操作、返回结果。

一个MCP Client可以同时连接多个MCP Server，每个Server可以提供多个工具。Agent不需要知道工具的具体实现——它只知道"有一个Server，它提供了这些工具，每个工具有这些参数和返回值"。这正是"USB标准"的精髓：即插即用，协议统一。

一个典型的MCP消息交换如下：

// 1. Client -> Server: 列出可用工具
        {
            "jsonrpc": "2.0",
            "id": 1,
            "method": "tools/list",
            "params": {}
        }

        // 2. Server -> Client: 返回工具列表
        {
            "jsonrpc": "2.0",
            "id": 1,
            "result": {
                "tools": [
                    {
                        "name": "search_papers",
                        "description": "在学术论文库中搜索相关论文",
                        "inputSchema": {
                            "type": "object",
                            "properties": {
                                "query": {"type": "string", "description": "搜索关键词"},
                                "limit": {"type": "integer", "description": "返回数量", "default": 10},
                                "year_from": {"type": "integer", "description": "起始年份"}
                            },
                            "required": ["query"]
                        }
                    },
                    {
                        "name": "get_paper_detail",
                        "description": "获取论文详细信息",
                        "inputSchema": {
                            "type": "object",
                            "properties": {
                                "paper_id": {"type": "string", "description": "论文ID"}
                            },
                            "required": ["paper_id"]
                        }
                    }
                ]
            }
        }

        // 3. Client -> Server: 调用工具
        {
            "jsonrpc": "2.0",
            "id": 2,
            "method": "tools/call",
            "params": {
                "name": "search_papers",
                "arguments": {
                    "query": "multi-agent reinforcement learning",
                    "limit": 5,
                    "year_from": 2025
                }
            }
        }

        // 4. Server -> Client: 返回结果
        {
            "jsonrpc": "2.0",
            "id": 2,
            "result": {
                "content": [
                    {
                        "type": "text",
                        "text": "找到 3 篇相关论文:\n1. ...\n2. ...\n3. ..."
                    }
                ],
                "isError": false
            }
        }
设计洞察： MCP选择JSON-RPC 2.0作为基础层是一个非常务实的决定。JSON-RPC极其简单——只有一个规范文档，被几乎所有语言支持。对比GraphQL（需要Schema定义、Resolver、类型系统），JSON-RPC的零仪式感（zero ceremony）特性让它非常适合Agent工具的快速集成场景。实际上，一个MCP Server可以在30分钟内写完，不用任何框架——只需要处理JSON输入输出。

14.2 三大原语：Tools、Resources、Prompts

MCP定义了三种核心原语，覆盖了Agent与外部世界交互的所有场景：

原语 用途 交互方式 类比
Tools Agent可以调用的"动作"——写文件、发邮件、查询数据库、执行代码 request → response（请求-响应） API调用 / Function Calling
Resources Agent可以"读取"的数据源——文件内容、数据库记录、API响应、日志 subscribe → notification（订阅-推送） GET请求 / 文件读取
Prompts 预定义的"模板"——Agent可以直接使用的Prompt模板，包含上下文注入 request → response（请求-响应） Prompt模板库

14.2.1 Tools：Agent的"手"

Tools是最常用的原语。一个MCP Tool的定义包含以下要素：

  • name：工具的唯一标识符
  • description：工具功能的自然语言描述（Agent通过这个描述来决定何时使用此工具）
  • inputSchema：参数的JSON Schema定义（Agent根据这个Schema来构造调用参数）

一个Tool的生命周期很简单：Agent调用tools/list发现可用工具，然后通过tools/call执行工具并获取结果。

14.2.2 Resources：Agent的"眼"

Resources与Tools不同：Tools是"Agent做某事"，Resources是"Agent获取信息"。Resources支持订阅（subscription）机制——Agent可以订阅一个Resource，在有变化时自动收到通知。这在大规模监控和实时数据场景中非常有用。

典型的Resources操作：

// 列出可用资源
        {"method": "resources/list", "params": {}}
        // 响应: 返回资源列表（如文件路径、数据表名、API端点）

        // 读取资源内容
        {"method": "resources/read", "params": {"uri": "file:///logs/app.log"}}
        // 响应: 返回资源内容（文件内容、数据记录、API响应体）

        // 订阅资源变化
        {"method": "resources/subscribe", "params": {"uri": "db://orders/new"}}
        // 之后当有新订单时，Server主动推送通知

14.2.3 Prompts：Agent的"经验"

Prompts是MCP中最有趣的原语。它允许Server预定义一组Prompt模板，Agent可以直接拉取这些模板来获取上下文。这些模板不是死的文本——它们可以动态注入数据。

例如，一个PostgreSQL MCP Server可以提供一个"分析慢查询"的Prompt模板：

// 获取预定义的Prompt模板
        {"method": "prompts/get", "params": {"name": "analyze_slow_queries"}}
        // 响应：
        {
            "messages": [
                {
                    "role": "system",
                    "content": {
                        "type": "text",
                        "text": "你是一个数据库优化专家。以下是当前数据库中的慢查询日志，"
                                "请分析每个慢查询的原因并给出优化建议。"
                    }
                },
                {
                    "role": "user",
                    "content": {
                        "type": "resource",
                        "resource": {"uri": "db://slow_queries/recent"}
                    }
                }
            ]
        }

这意味着Agent不需要自己去构造"分析慢查询"的系统提示词——MCP Server直接给了它一个包含实时数据的完整Prompt模板。Agent只需要"用"就行了。

三个原语的分工： Tools = 你能做什么（动作），Resources = 你能知道什么（信息），Prompts = 你怎么做更好（经验）。这三者组合起来，让一个MCP Server不只是"暴露API"，而是"暴露了一个完整的交互接口"——Agent可以按照标准的协议发现Server的能力、获取信息、执行操作、甚至获得使用建议。这种"自助式"的接口设计，是MCP区别于传统API网关的关键。

14.3 传输层：stdio vs Streamable HTTP

MCP支持两种传输方式，它们的适用场景完全不同：

特性 stdio（标准输入输出） Streamable HTTP
连接方式 通过子进程的标准输入输出通信 HTTP(S) RESTful API
启动方式 Client启动Server子进程 Server作为独立服务运行
生命周期 跟随Client的进程生命周期 独立运行，独立管理
延迟 极低（进程内通信） 中（网络延迟）
安全模型 本地执行，继承Client的权限 需要API Key/OAuth认证
适用场景 本地工具（文件系统、代码执行、数据库CLI） 远程服务（云API、第三方SaaS、分布式系统）
部署复杂度 低（无需额外服务） 中高（需要运维HTTP服务）

如何选择： stdio模式适合你需要在本地快速集成工具的场合——比如在开发环境中让Agent访问本地文件、执行本地命令、操作本地数据库。它的启动配置就是一个进程命令，Claude Desktop等客户端通过配置JSON就能直接拉起。而Streamable HTTP模式适合需要远程访问和共享的工具——比如一个团队公用的数据库MCP Server、一个SaaS工具的集成Server。

一个典型的MCP配置（Claude Desktop的mcp_servers配置）：

{
            "mcp_servers": {
                "local-filesystem": {
                    "command": "npx",
                    "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"]
                },
                "local-postgres": {
                    "command": "npx",
                    "args": ["-y", "@modelcontextprotocol/server-postgres",
                             "postgresql://localhost:5432/mydb"]
                },
                "remote-github": {
                    "url": "https://mcp.example.com/github-server",
                    "headers": {
                        "Authorization": "Bearer YOUR_TOKEN"
                    }
                }
            }
        }

14.4 MCP Server生态：10,000+ Server的宇宙

截至2026年中，MCP生态已经拥有超过10,000个公开发布的Server。这个数字还在以每月数百个的速度增长。以下是几个最具代表性的Server类别：

  • Filesystem (官方)：文件读写、目录导航、文件搜索——Agent最基本的"手"
  • PostgreSQL / SQLite / MySQL (官方)：数据库查询、Schema探索、数据操作
  • GitHub (官方)：Issue管理、PR操作、代码搜索、仓库管理
  • Brave Search / Google Search (官方)：网络搜索——Agent的"眼睛"看世界
  • Slack (社区)：消息发送、频道管理、历史搜索
  • Puppeteer / Playwright (社区)：浏览器自动化——Agent亲自"看网页"
  • Jira / Linear (社区)：项目管理——Agent管理自己的待办事项
  • Sentry / Datadog (社区)：监控数据查询——Agent自查Bug

你可以写出自己的MCP Server：

# 一个极简的MCP Server——用FastMCP框架（推荐方案）
        from mcp import FastMCP

        # 创建一个MCP Server
        server = FastMCP("my-special-server")

        # 注册一个Tool
        @server.tool()
        def analyze_sentiment(text: str) -> str:
            """分析文本的情感倾向"""
            # 这里集成你的情感分析模型
            result = sentiment_model.predict(text)
            return f"情感分: {result.score}, 标签: {result.label}"

        # 注册一个Resource
        @server.resource("config://app/settings")
        def get_settings() -> str:
            """获取应用配置"""
            with open("config.yaml", "r") as f:
                return f.read()

        # 启动（stdio模式）
        if __name__ == "__main__":
            server.run(transport="stdio")

        # 也可以启动为HTTP服务
        # server.run(transport="http", host="0.0.0.0", port=8000)

是的——就这么简单。用FastMCP框架，定义一个Tool就像写一个Python函数一样自然。@server.tool()装饰器自动将函数转换为MCP兼容的Tool定义，包括从类型注解自动生成JSON Schema。你的Server写好了，任何MCP兼容的Client（Claude Desktop、VS Code、LangGraph Agent、CrewAI Agent）都能立即发现和使用这些工具。

建设性建议： 对于开发团队，我建议把MCP Server纳入你的"基础设施"清单，就像数据库、消息队列、API网关一样。每个团队应该有一个内部的MCP Server仓库——把你的内部工具、API、数据库访问都封装成MCP Server。这有三大好处：① Agent接入成本降至零（任何MCP兼容的Agent框架都能即插即用）；② 工具接口标准化（不再需要为每个Agent框架适配不同的工具格式）；③ 工具可复用（你的MCP Server同时服务于Claude Desktop、VS Code插件、LangGraph Agent等多种客户端）。

14.5 OAuth 2.1认证与安全

MCP的Authorization规范基于OAuth 2.1（2025年发布的最新OAuth版本，简化了OAuth 2.0中的许多复杂选项）。这不是可选项——如果你使用Streamable HTTP模式部署MCP Server，认证是必须考虑的。

MCP OAuth的核心流程：

1. Client发现Server需要认证（返回401）
        2. Client获取Server的Authorization Server信息
        3. Client发起OAuth 2.1授权码流程
        4. 用户授权（如果未登录，先登录）
        5. Client获得Access Token（短期）+ Refresh Token（长期）
        6. Client在后续请求中携带Access Token
        7. Token过期时使用Refresh Token自动续期

        // 请求Header示例
        {
            "method": "tools/call",
            "params": {...},
            "headers": {
                "Authorization": "Bearer mcpat_xxxxxx"
            }
        }

安全注意事项：

  • 永远不要在stdio模式中使用网络令牌——stdio是进程内通信，不需要认证，加了反而增加风险（Token可能被同进程的其他组件窃取）。
  • MCP Server的最小权限原则：每个Server只应有完成其功能所需的最小权限。比如"文件读取Server"不应该有写入权限。
  • 推荐使用短期Token（15-60分钟）+ Refresh Token机制，减少Token泄露的风险窗口。
安全踩坑： 有一个广泛传播的"反面案例"值得分享。一家初创公司把他们内部MCP Server直接暴露在公网上，没有认证——想着"这个Server只有我们自己知道地址，没人会发现的"。结果两周后，他们的数据库被一个爬虫发现并清空了。这个爬虫不是恶意攻击者，而是另一个团队做的"MCP Server发现实验"——没错，有人写了MCP Server的搜索引擎。这个教训很简单：任何TCP端口上的服务都会被发现，永远不要依赖"隐藏地址"来保护你的MCP Server。MCP OAuth不是为了"锦上添花"——它是生产部署的基本要求。

14.6 MCP + A2A：分工明晰的协议栈

你可能注意到，MCP只解决了"Agent和工具之间怎么通信"，没有解决"Agent和Agent之间怎么通信"。这就是A2A（Agent-to-Agent）协议的舞台。

MCP和A2A不是竞争关系，而是互补关系——它们分别处于Agent通信协议栈的不同层次：

  • MCP（Model Context Protocol）： 工具层协议。定义了Agent如何发现和调用外部工具、访问数据源。类比：USB标准（设备连接协议）。
  • A2A（Agent-to-Agent Protocol）： 应用层协议。定义了Agent之间如何发现彼此、如何协商任务、如何传递结果。类比：HTTP协议（Web通信协议）。

在实际架构中，两个协议配合使用：

# 一个支持MCP和A2A的Agent
        from mcp import MCPClient
        from a2a import A2AClient

        class FullStackAgent:
            def __init__(self):
                # MCP Client：连接工具
                self.mcp = MCPClient()
                self.mcp.connect_server("local://filesystem")
                self.mcp.connect_server("https://mcp.example.com/postgres")

                # A2A Client：连接其他Agent
                self.a2a = A2AClient()
                self.a2a.discover_agents("https://agent-hub.example.com")

            async def handle_task(self, task):
                # 1. 通过A2A查询是否有其他Agent可以协助
                available_agents = await self.a2a.query_capabilities(task)

                if available_agents:
                    # 2. 通过A2A委托子任务给专业Agent
                    subtask_results = await self.a2a.delegate(
                        agents=available_agents,
                        task=task
                    )
                    return subtask_results

                # 3. 没有合适的Agent时，通过MCP调用工具自己完成
                tool_result = await self.mcp.call_tool("search_papers", {...})
                return tool_result

这套分工体系的意义是深远的：MCP确保每个Agent都有"手"可以操作工具，A2A确保每个Agent都有"嘴"可以和其他Agent沟通。两者之上，Agent才能形成真正的"社会"——既不自给自足地活在孤岛上，也不依赖人类为它们搭建每一个通信桥梁。

协议栈的全景视角： 如果站在2026年回看，Agent通信协议栈已经非常清晰了：A2UI（用户界面）在最上层，A2A（Agent间通信）在中间层，MCP（工具交互）在底层。每层都有自己的标准，每层只关心自己的职责。这个分层架构直接类比于互联网的TCP/IP协议栈——每一层都为上层提供抽象，下层的变化不影响上层的设计。没有这个分层，Agent生态就会像前互联网时代的计算机网络——各自为政，无法互通。有了这个分层，我们正在进入一个"Agent互联网"的时代——任何Agent都能发现和调用任何工具、与任何其他Agent协作。

14.7 本章小结

MCP是Agent工具层的"USB标准"。它的价值不只是一个技术协议，而是对整个Agent生态的基础设施化。有了MCP，Agent不再需要为每个工具定制集成代码——任何MCP兼容的Agent可以即插即用任何MCP兼容的工具。三个原语（Tools、Resources、Prompts）全面覆盖了Agent与外部世界的交互需求。

更重要的是，MCP只是"Agent互联网"协议栈中的一层。下面我们有了工具标准（MCP），上面我们需要Agent间通信标准（A2A）。下一章我们将深入A2A协议——它要解决的问题是：Agent如何找到其他Agent？如何与其他Agent协商任务分工？如何确保Agent间的通信是安全且可信任的？如果说MCP让Agent有了"手"，那A2A让Agent有了"朋友圈"。第15章见。
