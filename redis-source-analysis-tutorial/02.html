第2章：编译与调试环境搭建

大家好，我是你们的讲师。上一章我们站在高处俯瞰了 Redis 的全貌，就像拿到了整座城市的地图。但要真正走进这座城市的每一条街道、每一栋建筑，你需要两样东西：一双合脚的鞋（一个可以自由调试的编译环境），和一把能打开每扇门的钥匙（GDB 和代码跳转工具）。这一章我们就来解决这两件事。

很多人觉得"编译环境有什么好讲的，不就是 make && make install 吗？"如果你只是想把 Redis 跑起来，确实如此。但我们的目标不是跑起来，而是让每一行代码在 GDB 中都能单步跟踪，让每一个变量的值都一览无余，让 clangd 能瞬间带你跳转到宏定义和函数实现。这个"带调试符号的开发版本"和你线上跑的优化版本，中间的差别恰恰决定了你能在源码中走多远。

更重要的是，Redis 的 Makefile 本身就是一个极佳的学习材料。它展示了如何用 GNU make 管理一个跨平台的 C 项目，如何处理条件编译、依赖检查、第三方库集成等问题。读懂了 Makefile，你就能在任何意外编译失败时迅速找到问题根因——而不用对着满屏的 error 发呆。

为什么需要自己编译 Redis

从包管理器安装的 Redis 二进制文件是用 -O2 优化编译的，这意味着三个问题：

  • 没有调试符号：GDB 中你看不到行号信息，无法设置源码级断点，看到的只有一堆经过内联和重排后的汇编指令。
  • 变量被优化：局部变量可能被优化掉，寄存器复用导致 print var 返回 <optimized out>。
  • 内联函数展开：很多关键函数如 zmalloc() 被内联后，你在调用栈中根本看不到它——但内存分配的细节恰恰是我们需要跟踪的。

此外，自己编译还能自定义一些编译选项。比如你想强制使用 jemalloc 替代 glibc malloc，想开启内存追踪（REDIS_MALLOC_TRACE），或者想测试不同的内存分配器之间的性能差异——这些都要求你能控制编译流程。

源码获取与编译

首先，从 GitHub 获取 Redis 源码。建议使用 7.2.x 分支，这是本课程的主线版本：

git clone https://github.com/redis/redis.git
cd redis
git checkout 7.2   # 切换到 7.2 分支
git log --oneline -5  # 确认版本

接下来是最关键的编译步骤。默认的 make 会使用 -O2 优化，这对开发调试很不友好。我们需要用调试模式编译：

# 清掉上次的编译产物（如果有的话）
make distclean

# 调试编译：关闭优化，开启调试符号，使用 jemalloc
make REDIS_CFLAGS="-g -O0" REDIS_LDFLAGS="-g -O0" MALLOC=jemalloc

# 验证编译产物
ls -la src/redis-server src/redis-cli
file src/redis-server | grep "not stripped"  # 确认包含调试符号

看到 not stripped 的输出意味着调试符号保留完整，可以在 GDB 中愉快地玩耍了。

编译完成后，建议立即运行测试，验证环境一切正常：

# 运行核心测试（大约需要 3-5 分钟）
make test

# 如果只想跑某个具体模块的测试
./runtest --single unit/type/string
踩坑实录：曾有一位同事在排查 Redis 内存碎片问题时，反复查看 zmalloc.c 中的 zmalloc_used_memory() 逻辑，但 GDB 中始终看不到某些变量的值。折腾了半小时才发现，他用的 Redis 是 apt install 的预编译版本，根本没有调试符号。更糟的是，他试图用 apt-get install redis-server-dbg 安装 debug 包，但版本和源码不匹配，断点地址全部错位。最后不得不重新手动编译，才找到了内存统计逻辑中的一个计数器溢出 bug。教训很简单：看源码一定要用自己编译的调试版本。

Makefile 关键目标解读

Redis 根目录的 Makefile 是整个构建系统的入口。它虽然只有两百多行，但使用了很多 GNU make 的高级特性。我们来逐个解析最重要的 make 目标：

目标 执行命令 说明
make all 依次编译 deps/ 和 src/ 默认目标，生成所有二进制文件
make install cp src/redis-* /usr/local/bin/ 安装到系统路径
make test 调用 tests/test_helper.tcl 运行 Tcl 编写的集成测试套件
make clean rm -f src/*.o src/redis-* 清理 src/ 编译产物
make distclean clean + deps 清理 完全清理，包含第三方依赖
make valgrind 启用 VALGRIND 宏 编译内存检测版本
make noopt REDIS_CFLAGS="-O0 -g" 等价于无优化调试编译

值得留意的是 make noopt 这个目标。antirez 贴心地为开发者准备了这个快捷方式，它会自动设置 REDIS_CFLAGS="-O0 -g"，免去你每次手动输入的麻烦：

# 最简单的调试编译方式
make noopt

# 等价于
make REDIS_CFLAGS="-O0 -g" REDIS_LDFLAGS="-O0 -g"

Makefile 中的依赖管理也很有意思。Redis 使用 %.o: %.c 模式规则自动推导编译命令，通过 REDIS_CFLAGS 和 REDIS_LDFLAGS 两个变量将所有编译和链接选项暴露给外界。这种设计的好处是：你可以在不修改 Makefile 的情况下，通过环境变量覆盖任何编译参数。

编译选项对比

不同的编译选项组合对应不同的使用场景。下面这个表可以帮助你快速选择：

场景 CFLAGS 内存分配器 调试符号 优化级别
生产部署 （默认） jemalloc（默认） strip 后移除 -O2
源码调试 -g -O0 jemalloc 完整保留 无优化
内存检测 -g -O0 + VALGRIND glibc malloc 完整保留 无优化
性能分析 -g -O2 jemalloc 保留 frame pointer -O2
二分查找 bug -g -O0 -DDEBUG 任意 完整保留 无优化
小贴士：如果只用 make 不加任何参数，Redis 默认使用 MALLOC=jemalloc。如果你的系统没有安装 jemalloc 开发包，编译会失败。这时可以用 MALLOC=libc 回退到 glibc malloc 来编译。不过请注意，生产环境强烈推荐 jemalloc——它在减少内存碎片方面远优于 glibc malloc，我们下一章会详细讲。

GDB 调试 Redis

调试环境配好后，我们来上手试一下 GDB。Redis 是一个典型的事件驱动服务器程序，启动后进入 aeMain() 事件循环，处理网络请求。我们可以在关键入口点设置断点逐步跟踪。

启动 GDB 挂载 Redis 服务器

# 方式一：直接启动并立即断在 main()
gdb --args src/redis-server --port 6380
(gdb) break main
(gdb) run

# 方式二：附加到正在运行的进程
gdb -p $(pgrep redis-server)

# 方式三：分析 core dump 文件
gdb src/redis-server core.12345

断点设置技巧

在源码阅读过程中，以下几个断点位置最常用：

# 在命令处理入口设置断点
(gdb) break processCommand

# 在键值查找函数设置断点
(gdb) break lookupKey

# 在事件循环的每次迭代中断点
(gdb) break aeProcessEvents

# 条件断点：只对特定 key 停
(gdb) break dbAdd if strcmp((char *)key->ptr, "mykey") == 0

# 观察点：监控变量的写入
(gdb) watch server.ustat_avg_ttl

常用 GDB 命令速查

命令 用法示例 说明
break b processCommand 设置函数断点
next n 单步执行（跳过函数）
step s 单步执行（进入函数）
print p *((robj*)$rdi) 打印变量或表达式
display disp server.port 每步自动打印表达式
backtrace bt 查看调用栈
frame f 2 切换到指定栈帧
info locals i lo 查看所有局部变量
info threads i th 查看所有线程
finish fin 执行到函数返回

配置 .gdbinit

每次启动 GDB 手动敲这些命令太慢了。把常用宏和设置写入 ~/.gdbinit 文件中，GDB 启动时会自动执行：

set print pretty on
set pagination off

# 打印 Redis 对象
define pro
    print (robj *)$arg0
    printf "type=%d, encoding=%d, refcount=%d\n", ((robj*)$arg0)->type, ((robj*)$arg0)->encoding, ((robj*)$arg0)->refcount
end
document pro
    Print a Redis object with type/encoding/refcount
end

# 打印 SDS 字符串
define psds
    printf "len=%zu, alloc=%zu, flags=%d, buf=%s\n", ((sdshdr8*)((char*)$arg0 - sizeof(sdshdr8)))->len, ((sdshdr8*)((char*)$arg0 - sizeof(sdshdr8)))->alloc, ((sdshdr8*)((char*)$arg0 - sizeof(sdshdr8)))->flags, (char*)$arg0
end
document psds
    Print an SDS string header info
end

有了这些宏，调试体验会提升一个档次。比如在断点处直接输入 pro $rsi 就能看到 Redis 对象的类型和编码，不用再手动计算结构体偏移量。

VSCode + clangd 配置

GDB 适合运行时调试，但日常的代码浏览和静态分析，还是 IDE 更顺手。我推荐 VSCode + clangd 的搭配，因为它支持精确的符号跳转、类型提示、错误诊断——这些对阅读 C 代码至关重要。

生成 compile_commands.json

clangd 需要 compile_commands.json 才能知道每个源文件的编译参数。最简单的生成方式是用 bear 工具包裹 make 命令：

# 安装 bear（以 Ubuntu 为例）
sudo apt install bear

# 在 Redis 源码目录执行
make distclean
bear -- make REDIS_CFLAGS="-g -O0" MALLOC=jemalloc

# 验证生成结果
ls -la compile_commands.json
jq '. | length' compile_commands.json  # 应该包含 60+ 个编译单元
小贴士：bear 的工作原理是 LD_PRELOAD 拦截了 gcc 的 exec 调用，将每个编译单元的命令行参数记录到 JSON 文件中。如果你不想安装 bear，也可以使用 CMake 的 CMAKE_EXPORT_COMPILE_COMMANDS 选项，但 Redis 用的是原生的 Makefile，所以 bear 是最方便的选择。

VSCode 配置

在 VSCode 中需要做两件事：安装 clangd 插件，禁用默认的 C/C++ IntelliSense。在 .vscode/settings.json 中添加：

{
    "C_Cpp.intelliSenseEngine": "disabled",
    "clangd.arguments": [
        "--background-index",
        "--clang-tidy",
        "--header-insertion=never"
    ]
}

重启 VSCode 后，打开 src/server.c，试试 Ctrl+点击 lookupKey 函数——clangd 会立即带你跳转到 db.c 中该函数的定义。再试试把鼠标悬停在 robj 类型上，会弹出详细的结构体定义。这种体验在未配置 clangd 的项目中是完全不可想象的。

clangd 还有一个隐藏功能——代码补全时显示函数签名和参数类型。当你输入 zmalloc( 时，它会提示参数类型为 size_t size，并且返回类型是 void *。这对于记住那些参数较多的 API（比如 sdscatfmt、dictAddRaw）非常有帮助。

提示：如果你发现 clangd 跳转到的是头文件中的声明而不是 .c 文件中的定义，多半是因为 compile_commands.json 没有正确生成。请确保：1) 先执行 make distclean 再 bear -- make；2) bear 捕获到的编译单元数与 src/ 目录下的 .c 文件数基本一致。如果还是不行，可以手动在 VSCode 中执行 "clangd: Restart language server"。

环境搭好之后，我们正式进入源码阅读的第一站——内存管理。在下一章中，我们将深入 zmalloc.c 和 zmalloc.h，看看 Redis 如何管理内存、如何与 jemalloc 协作、以及如何精确监控每一点内存的使用。
