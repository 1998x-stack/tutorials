第26章 前沿趋势盘点

同学们好，欢迎来到第六部分。在过去的25章里，我们完整地走过了推荐系统的技术栈——从最经典的协同过滤，到深度学习的各类模型，再到LLM驱动的推荐范式。这些知识在未来三到五年内都不会过时。不过，推荐系统这个领域，变是唯一的不变。在最后三章里，我们先抬头看看前方——站在2025-2026年这个时间节点，哪些技术趋势正在重塑推荐系统的未来？

这一章我们不深入任何一篇论文的细节，而是做一次全景式的巡礼，帮你建立一个"未来地图"。等到你在工作中遇到具体问题时，这张地图会告诉你去哪个方向寻找答案。

1. 生成式推荐：从打分到生成

让我们先从最颠覆性的变化说起。

过去二十年的推荐系统，本质上都是一个打分问题（Scoring Problem）：给定用户和物品，模型输出一个分数，排序后取Top-N。这个范式如此根深蒂固，以至于很少有人质疑它——直到生成式AI的出现。

生成式推荐（Generative Recommendation）把推荐重新定义为一个条件生成问题（Conditional Generation Task）：给定用户的行为序列，模型直接"生成"下一个推荐的物品。不是给所有候选物品打分，而是直接生成目标物品的ID。这个思路来自于一个简单的观察——推荐和文本生成在本质上是一样的：都是从一个序列预测下一个元素。

2023年，Meta发表了论文"Actions Speak Louder than Words: Trillion-Parameter Sequential Transducers for Generative Recommendations"（简称GR）。这是生成式推荐的开山之作。GR的核心架构是：用户行为序列 → Transformer Encoder → Autoregressive Decoder → 生成物品的Semantic ID。这个模型有惊人的万亿参数规模，在Meta的社交推荐场景上取得了显著的效果提升。当然，万亿参数不是谁都能玩的——GR更像是一次"探路者"式的探索，告诉大家这条路是可行的。

紧接着，快手在2024年发表了OneRec，这是生成式推荐在短视频场景的首次工业级落地。OneRec的特别之处在于：它不是用生成式模型替代整个推荐pipeline，而是把它用作精排阶段的一个"精化器"——先用传统双塔和精排模型产生候选，再用生成式模型对这些候选做端到端的优化排序。OneRec的论文报告了线上A/B测试的显著提升，证明了生成式推荐在工业场景的可行性。

2025年，阿里巴巴的LUM（Large Unified Model）进一步推动了这个趋势。LUM的核心贡献是验证了推荐系统的Scaling Law——随着模型参数和数据量的增加，推荐效果持续提升，没有出现饱和。我们稍后会专门讨论这一点。

生成式推荐相比传统pipeline有什么优势？主要有三点：

  • 端到端优化：传统pipeline的召回、粗排、精排、重排四个阶段各自优化，全局未必最优。生成式推荐用一个模型完成整个推荐过程，实现全局最优化。
  • 天然多样性：生成式模型有内在的随机性（通过temperature控制），天然带来推荐结果的多样性，不需要额外的多样性后处理。
  • 零样本泛化：由于Semantic ID编码了物品的内容语义，生成式推荐对冷启动物品有天然的零样本泛化能力——不需要额外的冷启动策略。

核心洞察：生成式推荐最大的意义不是"效果提升百分之多少"——而是它从根本上颠覆了推荐系统的建模范式。从"这个用户喜欢什么？我搜一下"变成"这个用户喜欢什么？我创造一下"。这个转变的影响可能比我们目前看到的还要深远。就像ChatGPT重新定义了人机交互一样，生成式推荐可能重新定义用户与内容之间的发现方式。

2. Semantic ID：给每个物品一个语义身份证

生成式推荐面临的第一道坎就是：物品ID怎么表示？

传统推荐系统中，每个物品被分配一个唯一的整数ID（Item ID 0, 1, 2, ...）。这些ID是无意义的——ID 42和ID 43之间没有任何语义关系，尽管它们在编号上相邻。这对生成式模型来说是个大问题：如果"生成ID 42"和"生成ID 43"在语义上没有连续性，模型就无法利用生成式模型的序列预测优势。

解决方案是Semantic ID——给每个物品一个语义上的"身份证"，使得语义相似的物品拥有相似的ID。想象一下用一串编码来表示一部电影：第一层表示"动作片"，第二层表示"好莱坞动作片"，第三层表示"1990年代好莱坞动作片"……最后落在具体的电影上。这样，模型在生成ID时会沿着语义路径逐层决策，每一步的选择都有明确的语义含义。

常见的Semantic ID生成方法有两种：

  • 层次聚类（Hierarchical Clustering）：对物品的内容embedding做层次聚类，每个聚类分配一个编码路径。优点是简单直观，缺点是聚类结果不稳定——重新聚类会改变所有物品的ID。
  • RQ-VAE（Residual Quantized VAE）：用多级残差量化器，每一层量化embedding的残差。优点是编码更加紧凑高效，且天然支持多尺度语义。

下面是用层次聚类生成Semantic ID的伪代码：

# Semantic ID生成：层次聚类伪代码
def generate_semantic_ids(item_embeddings, depth=3, branching_factor=10):
    """
    参数:
        item_embeddings: dict {item_id: embedding_vector}
        depth: 语义ID的层级深度
        branching_factor: 每层聚类数
    返回:
        semantic_ids: dict {item_id: [level_1_id, level_2_id, ...]}
    """
    current_items = list(item_embeddings.keys())
    current_embeds = np.stack([item_embeddings[i] for i in current_items])
    semantic_ids = {item: [] for item in current_items}

    for level in range(depth):
        # 对当前层级的embedding做KMeans聚类
        kmeans = KMeans(n_clusters=branching_factor, random_state=42)
        labels = kmeans.fit_predict(current_embeds)

        # 记录当前层级的聚类标签
        for item, label in zip(current_items, labels):
            semantic_ids[item].append(str(label))

        # 为下一层准备：每个聚类内的物品继续细化
        # 计算残差 = 原始向量 - 聚类中心
        cluster_centers = kmeans.cluster_centers_
        residuals = current_embeds - cluster_centers[labels]
        current_embeds = residuals  # 下一层在残差上继续聚类

    # 将多层编码拼接成string ID
    return {item: "/".join(ids) for item, ids in semantic_ids.items()}

# 使用示例
# item_42 的 semantic_id = "动作/好莱坞/1990s/终结者2"
# item_43 的 semantic_id = "动作/好莱坞/1990s/虎胆龙威"

看到这里你可能会问：Semantic ID好是好，但有什么坑吗？

确实有几个实际问题：

  • ID稳定性：当新物品加入或内容分布变化时，层次聚类的结果可能改变，导致所有物品的Semantic ID发生变化。这意味着已训练的模型需要重新训练。
  • 层级冲突：有些物品天然不适合严格的分层分类。比如一部"带有浪漫元素的科幻片"——它属于科幻还是浪漫？硬性的层次结构无法表达这种模糊性。
  • Mahalanobis Collapse：在使用RQ-VAE时，如果各层的编码本学习不充分，高层的量化残差可能变得非常小，导致模型实际上只用了底层编码，失去多尺度语义。这是2025年学术界正在积极解决的问题。

3. 搜推一体化

如果你问普通用户"你在淘宝上找到商品，是通过搜索还是推荐？"——大多数用户可能回答不上来。这恰恰说明了问题：搜索和推荐的界限是产品人为制造的，用户并不区分这两者。

用户的行为是一个连续谱：明确知道自己想要的，去搜索；不太确定想要什么的，去浏览推荐；想扩展一下的，用推荐结果里的搜索意图再搜一下。产品却把这个连续谱硬生生切成了两个独立的功能模块，各自有不同的模型、不同的团队、不同的评估指标——这合理吗？

TikTok的产品形态已经证明了这一点。在TikTok中，搜索和推荐是融合的：你可以在推荐流中直接搜索，搜索结果页的排序又包含推荐逻辑。用户根本不需要在两个模式之间切换。这种统一的体验正在成为主流。

学术界也注意到了这个趋势。SIGIR 2026上发表的GEMS框架（Generalized Embedding Multi-Space）提出了一个优雅的解决方案：多子空间梯度分解。GEMS的核心思想是：一个统一的模型应该有一个共享子空间（学习通用的表示能力），加上搜索特定子空间和推荐特定子空间。训练时使用Null-space Projection来保护大模型中已有的通用知识不被特定任务破坏。

GEMS的数学很简单：embedding = shared_projection(x) + search_projection(x) + recommend_projection(x)。对于搜索请求，梯度只回传到shared + search子空间；对于推荐请求，梯度只回传到shared + recommend子空间。这样共享子空间学习通用的语义表示，搜索和推荐子空间则处理各自的目标。

为什么要用Null-space Projection？因为大模型（LLM作为backbone）中已经编码了大量通用知识。如果直接做full fine-tuning，模型的通用知识会被任务数据"冲刷"掉。Null-space Projection确保在优化推荐目标时，模型的梯度与LLM原有知识的null space对齐——只更新与推荐相关的方向，不破坏通用语义。

工业观察：Meta和Google从2024年起就在内部推动搜推一体化项目。据Meta公开的技术分享，他们将Instagram的搜索和推荐模型合并训练后，不仅节省了一半的训练资源，搜索的recall和推荐的engagement都有2-3%的提升。这证实了共享表示对双方都有益——"1+1>2"在搜推一体化中真实存在。

4. 多Agent协同

如果说搜推一体化是让一个系统做多件事，那么多Agent协同就是让多个专业系统合作完成一件事。这两个方向看似矛盾，实则相辅相成——统一的是知识表示，专业的是任务执行。

想象一个未来推荐系统的架构：用户说"我想找一个类似《流浪地球》但轻松一点的科幻片看"。传统的推荐系统无法处理这种多约束的自然语言请求。但一个Agent化的推荐系统可以做到：

  • 意图理解Agent：解析用户自然语言，提取关键约束（"类似《流浪地球》"、"轻松一点"、"科幻片"）
  • 召回Agent：根据约束条件，从向量库中召回候选（使用多向量检索，每个约束条件一个向量）
  • 排序Agent：对召回的候选做精细排序
  • 解释Agent：为每个推荐结果生成自然语言解释（"推荐这部《外太空的莫扎特》，因为它也涉及父子关系和太空题材，但风格更加轻松诙谐"）
  • 对话Agent：根据用户的后续反馈进行多轮交互（"太轻松了，再硬核一点" → 重新调整）

这个架构看起来复杂，但它的核心机制已经出现在2025年的研究中。ChainRec（Chain-of-Thought for Recommendation）提出了"工具库 + 学习规划器"的框架：系统维护一个推荐工具库（向量检索、协同过滤、热榜、知识图谱查询……），一个轻量级的planner（通常是小型LLM）根据用户请求和上下文动态选择合适的工具链。每次推荐不再是走固定的pipeline，而是模型自动规划出一条最优路径。

最终愿景是什么？是一个单一的AI服务，无缝处理搜索、推荐和对话——用户可以用自然语言表达需求，系统自主决定使用哪些工具、调用哪些模型、如何呈现结果。这个愿景正在从论文走向原型，从原型走向产品。

5. Scaling Law在推荐中的验证

大语言模型领域有著名的Scaling Law：模型越大、数据越多，效果越好。推荐领域是否也遵循类似的规律？这个问题在2025年之前没有明确的答案——因为推荐系统的效果受太多因素影响（特征工程、目标设计、样本选择），单纯增大模型不一定有用。

阿里巴巴LUM的工作第一次给出了正面的回答。LUM系统性地构建了不同规模的模型（从1亿参数到1000亿参数），在相同的训练数据和评估条件下测量效果。结果如下表所示：

模型 参数量 训练数据量 核心指标（GAUC） 对比基线提升
DeepFM baseline ~10M 100B samples 0.632 —
LUM-Small 100M 100B samples 0.651 +3.0%
LUM-Medium 1B 500B samples 0.668 +5.7%
LUM-Large 10B 1T samples 0.679 +7.4%
LUM-Huge 100B 5T samples 0.685 +8.4%

数据清晰地展示了Scaling趋势：模型每增大一个量级，GAUC提升约1-2%。没有观察到显著的饱和。当然，1000亿参数的模型训练成本是天文数字——LUM-Huge据说用了上千张GPU训练数周。但趋势本身具有重要的战略意义：推荐系统的竞争正在变成算力的竞争。

这对我们普通团队意味着什么？

  • 大公司会持续投入超大规模模型，Scaling Law是它们的护城河——你有多少GPU就决定了你能用多大的模型。
  • 小公司不需要恐慌。Scaling Law在推荐中成立的前提是"在相同的训练数据和评估条件下"——而大多数中小团队距离这个条件还有很远。先做好特征工程、样本设计和基础模型，再考虑Scaling。
  • 效率优化会成为热点。模型蒸馏、量化、稀疏化MoE——这些在LLM领域成熟的效率优化技术正在被应用到推荐领域，让中小团队也能"以小博大"。

值得深思：Scaling Law在推荐领域的验证，意味着推荐系统的竞争格局可能会重演LLM领域的故事——少数拥有海量算力和数据的公司会拉开巨大差距。但同时，推荐系统不像LLM那样依赖"通用知识"，它更多依赖"领域知识"和"业务理解"。这意味着在特定垂直领域（电商、招聘、教育……），深耕业务理解的中小团队仍然有自己的机会。

总结

本章我们盘点了推荐系统在2025-2026年的五大前沿趋势：生成式推荐重新定义推荐范式，Semantic ID为物品提供了语义化的标识，搜推一体化正在消除人为的产品边界，多Agent协同将推荐带入智能体时代，Scaling Law验证了算力投入的价值。这些趋势共同指向一个方向：推荐系统正在从一个"匹配引擎"进化为一个"智能理解引擎"——它不再只是简单地匹配用户和物品，而是要理解用户的深层需求、内容的内在语义，以及两者之间丰富的动态关系。

但如果你的团队只有5个人、DAU只有10万，这些前沿技术跟你有什么关系？所有前沿都是建立在扎实的基础之上的。下一章，我们就从云端回到地面，聊聊一个真实的推荐系统应该怎么搭建——从0到1，从小到大的推荐系统架构设计。
