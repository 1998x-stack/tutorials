设计类题目概览

大家好，欢迎来到设计类题目的专场。如果你刷 LeetCode 时只关注"算法正确性"，那这一章可能会让你大开眼界。设计类题目（Object-Oriented Design / Data Structure Design）考察的不是你知不知道 Dijkstra 或者 Floyd，而是你能不能从零搭建一个满足特定约束的系统。这类题在面试中非常高频——尤其是 Facebook（Meta）、Amazon、Google 等大厂的第二轮面试，往往就是一道纯设计题，限时 30-45 分钟，没有算法，只有 OOD。

设计题的核心能力是什么？我总结为三个词：抽象（把业务需求抽象为数据结构和接口）、权衡（读快还是写快、空间换时间）、鲁棒（边界情况不会崩溃）。我们这一章会从最经典的 LRU 缓存开始，一路走到 LFU、迭代器设计，彻底吃透设计类题目的套路。

核心思路：设计题没有"标准答案"，但有"最佳实践"。面试官看重的是你做决策的过程——为什么选这个数据结构、为什么用这个时间空间权衡。代码能跑只是及格，讲清楚为什么才是高分。

LRU缓存 — 哈希表+双向链表实现

LRU Cache（LeetCode 146）可能是 LeetCode 上最著名的设计题，没有之一。要求很简单：设计一个缓存类，支持 get(key) 和 put(key, value) 操作，容量固定，当缓存满时淘汰最久未使用的键值对。要求两个操作都是 O(1)。

题目本身很直观，难点在于 O(1) 的实现。缓存淘汰的核心逻辑是：每次访问（get 或 put）都把当前键提到"最近使用"的位置，淘汰时从"最久未使用"的位置移除。什么样的数据结构能做到"快速插入、快速删除、快速移动"？答案是：哈希表 + 双向链表。

  • 哈希表（key -> Node）：提供 O(1) 查找——给定 key，一秒定位到链表节点。
  • 双向链表：提供 O(1) 插入和删除——因为每个节点都记录前驱和后继，摘下任意节点、在头部插入都是常数时间。

下面这个实现是面试中公认的"黄金标准"：

class DLinkedNode:
    def __init__(self, key=0, value=0):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}  # key -> node
        # 伪头尾节点，避免边界判断
        self.head = DLinkedNode()
        self.tail = DLinkedNode()
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove_node(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node):
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    def _move_to_head(self, node):
        self._remove_node(node)
        self._add_to_head(node)

    def _pop_tail(self):
        node = self.tail.prev
        self._remove_node(node)
        return node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._move_to_head(node)
        return node.value

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            node = self.cache[key]
            node.value = value
            self._move_to_head(node)
        else:
            if len(self.cache) >= self.capacity:
                tail = self._pop_tail()
                del self.cache[tail.key]
            node = DLinkedNode(key, value)
            self.cache[key] = node
            self._add_to_head(node)
伪头尾节点的妙用：如果你不用伪头尾节点（dummy head/tail），那么 _add_to_head 时需要检查 head 是否为 None，_pop_tail 时需要检查 tail 是否为 None——这些 if-else 会有一堆。面试中 95% 的候选人都栽在边界判断上。添加两个哑节点，代码直接清爽一倍。

时间复杂度的分析：get 和 put 都只涉及哈希表的 O(1) 查找和链表的 O(1) 指针操作，整体 O(1)。空间复杂度 O(capacity)，因为最多存储 capacity 个节点。

这里我必须分享一个亲身经历。我第一次面 Google 时，LRU 这道题我写了 80 行，满屏的 if node.prev is None 和 if node.next is None——我自己都看晕了。面完以后，一个在 Google 工作的学长给我看了上面这个伪头尾版本，我整个人都不好了。从那以后，我每次遇到链表设计题，第一件事就是加两个哑节点。这个习惯帮我拿了好几个 offer。

LFU缓存 — 频率链表设计

LRU 的升级版来了——LFU Cache（LeetCode 460）。LFU 的全称是 Least Frequently Used——当缓存满时，淘汰访问频率最低的键；如果多个键频率相同，则淘汰最久未使用的那个（LRU 作为 tiebreaker）。

这道题是 LeetCode 上公认的 Hard 题之一。难点在于：除了 O(1) 的 get/put，你还要维护"频率"这个维度。先分析需求：

  • 每次 get 或 put 已存在的 key，频率 +1
  • 频率最低的 key 被淘汰（同频次中最久未用）
  • 一切操作 O(1)

核心思路：双哈希表 + 双向链表群。

  • key_to_node：key → node（节点的值包括 key, value, freq）
  • freq_to_list：freq → 双向链表（所有频率为 freq 的节点，最新的在头部）
  • min_freq：全局变量，记录当前最小频率，用于淘汰
class DLinkedNode:
    def __init__(self, key=0, value=0, freq=1):
        self.key = key
        self.value = value
        self.freq = freq
        self.prev = None
        self.next = None

class DLinkedList:
    def __init__(self):
        self.head = DLinkedNode()
        self.tail = DLinkedNode()
        self.head.next = self.tail
        self.tail.prev = self.head
        self._size = 0

    def add_first(self, node):
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node
        self._size += 1

    def remove(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev
        self._size -= 1

    def remove_last(self):
        if self._size == 0:
            return None
        node = self.tail.prev
        self.remove(node)
        return node

    @property
    def size(self):
        return self._size

class LFUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.key_to_node = {}       # key -> node
        self.freq_to_list = {}      # freq -> DLinkedList
        self.min_freq = 0

    def _update_freq(self, node):
        freq = node.freq
        # 从旧频率链表中移除
        self.freq_to_list[freq].remove(node)
        if self.freq_to_list[freq].size == 0:
            del self.freq_to_list[freq]
            if self.min_freq == freq:
                self.min_freq += 1
        # 插入新频率链表
        node.freq += 1
        self.freq_to_list.setdefault(node.freq, DLinkedList()).add_first(node)

    def get(self, key: int) -> int:
        if key not in self.key_to_node:
            return -1
        node = self.key_to_node[key]
        self._update_freq(node)
        return node.value

    def put(self, key: int, value: int) -> None:
        if self.capacity == 0:
            return
        if key in self.key_to_node:
            node = self.key_to_node[key]
            node.value = value
            self._update_freq(node)
        else:
            if len(self.key_to_node) >= self.capacity:
                old_list = self.freq_to_list[self.min_freq]
                removed = old_list.remove_last()
                del self.key_to_node[removed.key]
            node = DLinkedNode(key, value, 1)
            self.key_to_node[key] = node
            self.freq_to_list.setdefault(1, DLinkedList()).add_first(node)
            self.min_freq = 1
注意 min_freq 的更新逻辑：当一个节点频率增加后，如果旧频率的链表变空，且该频率等于 min_freq，min_freq 才递增。这个细节非常容易漏。另外，插入新节点时 min_freq 必然重置为 1——因为新节点的频率就是 1。

LFU 的面试频率低于 LRU，但一旦被问到，就是区分"写过多道题"和"真正理解设计"的分水岭。能流利写出 LFU 的候选人，基本上设计类题目就不用再考了。

O(1)数据结构设计

LeetCode 上有几道题专门考察"常数时间操作"的设计能力。咱们挑两个最经典的。

常数时间插入、删除和获取随机元素

Insert Delete GetRandom O(1)（LeetCode 380）：设计一个数据结构，支持 insert(val)、remove(val)、getRandom() 三个操作，全部 O(1)。

这道题的巧妙之处在于 getRandom() 要求 O(1) 随机获取——这指向了数组（支持 O(1) 按索引访问）。但数组的删除是 O(n) 的，怎么解决？关键技巧：把要删除的元素和最后一个元素交换，然后 pop 末尾。

import random

class RandomizedSet:
    def __init__(self):
        self.nums = []          # 存储元素
        self.val_to_idx = {}    # val -> index

    def insert(self, val: int) -> bool:
        if val in self.val_to_idx:
            return False
        self.val_to_idx[val] = len(self.nums)
        self.nums.append(val)
        return True

    def remove(self, val: int) -> bool:
        if val not in self.val_to_idx:
            return False
        idx = self.val_to_idx[val]
        last = self.nums[-1]

        # 把最后一个元素覆盖到要删除的位置
        self.nums[idx] = last
        self.val_to_idx[last] = idx

        # 删除最后一个元素
        self.nums.pop()
        del self.val_to_idx[val]
        return True

    def getRandom(self) -> int:
        return random.choice(self.nums)

这个"交换 + pop"模式非常重要。删除任意位置元素 O(1) 在几乎任何语言里都做不到，但利用"不关心顺序"这个前提，通过交换末尾元素来绕过限制。这是面试中极高频的思维模式。

黑名单中的随机数

Random Pick with Blacklist（LeetCode 710）：给定范围 [0, n-1] 和一个黑名单数组 blacklist，让你每次 pick() 时均匀随机返回一个不在黑名单中的整数。

最直接的办法是：把所有白名单数字存进数组，然后随机取值。但如果 n 很大（比如 10⁹），白名单数组会大到爆炸。解法思路如下：

import random

class Solution:
    def __init__(self, n: int, blacklist: List[int]):
        black_set = set(blacklist)
        self.usable = n - len(black_set)  # 白名单数量
        self.mapping = {}

        # 把 [usable, n-1] 范围内的黑名单数字映射到白名单数字
        black_in_range = [b for b in blacklist if b >= self.usable]
        white_ptr = self.usable

        for b in black_in_range:
            while white_ptr in black_set:
                white_ptr += 1
            self.mapping[b] = white_ptr
            white_ptr += 1

    def pick(self) -> int:
        x = random.randrange(self.usable)
        return self.mapping.get(x, x)
核心思想：把黑名单数字"映射"到白名单数字。范围 [0, usable-1] 中如果有黑名单元素，就映射到 [usable, n-1] 中不在黑名单的数字。这样 pick 只需要在 [0, usable-1] 中随机取数，然后通过映射（如果有的话）拿到真正的值。

迭代器设计

迭代器设计题考察的是状态机思维——你怎么把一个复杂的数据结构遍历过程，封装成一个"每次调用 next() 返回一个元素"的迭代器。

扁平化嵌套列表迭代器

Flatten Nested List Iterator（LeetCode 341）：给你一个嵌套的整数列表，例如 [[1, 1], 2, [1, 1]]，设计一个迭代器把它展平为 1, 1, 2, 1, 1。

典型的"懒展开"策略——不要在构造函数里全部展开，而是使用时逐步展开。

class NestedIterator:
    def __init__(self, nestedList: [NestedInteger]):
        self.stack = [[nestedList, 0]]  # [当前列表, 当前索引]

    def next(self) -> int:
        self._ensure_top_is_integer()
        lst, idx = self.stack[-1]
        self.stack[-1][1] += 1
        return lst[idx].getInteger()

    def hasNext(self) -> bool:
        self._ensure_top_is_integer()
        return len(self.stack) > 0

    def _ensure_top_is_integer(self):
        while self.stack:
            lst, idx = self.stack[-1]
            if idx == len(lst):
                self.stack.pop()
                if self.stack:
                    self.stack[-1][1] += 1
                continue
            if lst[idx].isInteger():
                break
            # 遇到列表，推入栈顶
            self.stack.append([lst[idx].getList(), 0])
        else:
            return
        # 这里需要再检查一次栈顶是否遍历完
        while self.stack and self.stack[-1][1] == len(self.stack[-1][0]):
            self.stack.pop()
            if self.stack:
                self.stack[-1][1] += 1

很多 LeetCode 题解直接用 Python 的 yield 来生成器实现——面试时这样做确实快，但面试官很可能会追问："能不能不用生成器？"上面这个栈式版本才是面试官更想看到的：它展示了迭代器暂停和恢复的核心原理。

顶端迭代器

Peeking Iterator（LeetCode 284）：给一个标准迭代器，增加 peek() 方法，返回下一个元素但不移动指针。

class PeekingIterator:
    def __init__(self, iterator):
        self.iterator = iterator
        self._next = None
        self._has_peeked = False

    def peek(self):
        if not self._has_peeked:
            self._next = self.iterator.next()
            self._has_peeked = True
        return self._next

    def next(self):
        if self._has_peeked:
            result = self._next
            self._has_peeked = False
            self._next = None
            return result
        return self.iterator.next()

    def hasNext(self):
        return self._has_peeked or self.iterator.hasNext()

这个设计模式叫"预取缓存"（Prefetch Cache）：提前读取下一个元素并缓存，peek 时直接返回缓存。装饰器模式的一个简单应用。

面试加分项：当你实现迭代器时，提到"惰性求值"（Lazy Evaluation）——只在需要时计算，不做提前展开——面试官会认为你有函数式编程的思想。这在处理无限序列或大型数据集时非常重要。

完整代码与测试框架

为了你面试时能一次性写出无 bug 的设计题代码，这里给出 LRU Cache 的完整 LeetCode 提交版（包含所有类和主逻辑）：

class LRUCache:
    class _Node:
        __slots__ = ('key', 'value', 'prev', 'next')
        def __init__(self, key=0, value=0):
            self.key = key
            self.value = value
            self.prev = None
            self.next = None

    def __init__(self, capacity: int):
        self.cap = capacity
        self.cache = {}
        self.head = self._Node()
        self.tail = self._Node()
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node):
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_front(self, node):
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)
        self._add_front(node)
        return node.value

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            node = self.cache[key]
            node.value = value
            self._remove(node)
            self._add_front(node)
        else:
            if len(self.cache) == self.cap:
                lru = self.tail.prev
                self._remove(lru)
                del self.cache[lru.key]
            node = self._Node(key, value)
            self.cache[key] = node
            self._add_front(node)
生产环境 vs 面试环境：Python 的标准库 collections.OrderedDict 可以直接实现 LRU——move_to_end() + popitem(last=False) 六行代码搞定。但在面试中我建议你手写双向链表，因为面试官要考察的就是"你是否真的理解链表操作"。用 OrderedDict 会被当作调包侠。

设计题的通用面试模板

经历了 LRU、LFU、O(1) 集合、迭代器这四类典型题后，我们可以总结出设计题的通用解题框架：

步骤 动作 示例（LRU）
1. 理解需求 列出所有 API、时间/空间约束 get O(1), put O(1), 容量有限
2. 选择核心数据结构 根据操作约束反推结构 O(1) 查找→哈希表；O(1) 增删→链表
3. 组合数据结构 确定数据间的映射关系 哈希表 key→链表节点
4. 定义辅助方法 提取常见的指针操作 _remove, _add_front, _move_to_front
5. 实现 API 用辅助方法拼装 get = 查找 + 移动；put = 添加/更新 + 淘汰
6. 边界测试 空、满、重复、不存在 get 不存在的 key、put 已满的 cache

我用这个模板反复训练，从 LRU 到 LFU 到跳表设计，每道题从读题到写完不超过 15 分钟。你不需要背代码，而是背这个思维框架。

踩坑与总结

设计类题目最大的坑就是"过度设计"和"设计不足"之间的平衡。以 LRU 为例，面试中最常见的三个错误：

  • 不用的伪头尾节点：结果就是满屏的 if-else 判断 None，代码丑且容易出 bug。
  • 忘记在 put 更新时移动节点：put 已存在的 key 也要更新它的位置——很多人只改了 value 忘了动指针。
  • capacity = 0 的边界：LeetCode 460 的 LFU 有 capacity = 0 的测试用例，你要提前返回，否则除零错误。

此外还有一个非常重要的实操建议：面试写设计题时，先画出数据结构关系图。在白板或纸上画一下哈希表指向哪些节点、链表怎么连——这能帮你理清所有指针关系，同时也让面试官看到你的结构化思维。我 90% 的设计题 bug 都在画图阶段被消灭了，剩下的 10% 靠测试用例。

练习题推荐：完成本章后，建议刷以下题目巩固设计能力——LeetCode 146（LRU Cache）、460（LFU Cache）、380（Insert Delete GetRandom）、341（Nested Iterator）、284（Peeking Iterator）、355（Design Twitter）、622（Design Circular Queue）、641（Design Circular Deque）。其中 355 是微推系统的简化版，非常有意思，做起来像在写一个小型后端。

设计能力是区分"刷题机器"和"真正工程师"的分水岭。掌握了 LRU 和 LFU，你就掌握了"空间换时间"和"哈希表+链表"这个黄金组合。下一章我们将进入一个完全不同的领域——多线程与并发。在并发场景下，不仅仅是正确性，还有线程安全的挑战。我们下一章见！
