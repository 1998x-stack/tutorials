第19章 网络切片技术

大家好，我是你们的讲师。思考一个问题：同一张物理网络，如何同时服务一辆自动驾驶汽车（要求端到端时延<5ms、可靠性99.999%）和成千上万个智能水表（每月只发几个字节、电池要用十年）？更离谱的是，这两个业务可能用的是同一个基站、同一根光纤、同一个核心网。

答案就是本章的主题——网络切片（Network Slicing）。如果把5G网络比作一条高速公路，网络切片就是在这条高速公路上画出"专用车道"：一条留给救护车（超低时延），一条留给公交车（大带宽），一条留给货运卡车（海量连接），各走各的道，互不干扰。5G网络切片是让"一张网干所有事"成为现实的关键技术，也是运营商从卖流量转向卖服务体验的核心手段。

19.1 网络切片的核心概念

网络切片的思想说起来并不复杂：在一套共享的物理基础设施上，构建多个逻辑上隔离的端到端网络，每个网络（切片）拥有自己独立的资源、功能和SLA保障。每个切片从RAN到传输网再到核心网，是一条完整的逻辑链路。

类比来说，就像在同一个物理服务器上通过虚拟机运行多个操作系统——硬件是共享的，但每个虚拟机有自己的CPU、内存和磁盘配额。网络切片就是通信世界的"虚拟化"。

直观理解：去海底捞吃火锅，你点的番茄锅、麻辣锅和菌汤锅用的是同一口锅的物理空间，但中间的隔板让各锅汤底互不串味。网络切片就是网络世界里的"鸳鸯锅"——共享物理资源，隔离逻辑体验。

一个完整的网络切片包含以下组成部分：

  • RAN切片：无线侧的PRB资源预留、差异化调度策略
  • 传输网切片：承载网的资源隔离，FlexE硬隔离或VPN软隔离
  • 核心网切片：独立的AMF/SMF/UPF实例或共享实例中的差异化处理
  • 端到端编排：跨域资源协同，保证SLA端到端可达

19.2 切片标识：S-NSSAI

在5G网络中，每个切片都有一个唯一的标识符，叫作S-NSSAI（Single Network Slice Selection Assistance Information）。它由两部分组成：

S-NSSAI = SST (Slice/Service Type) + SD (Slice Differentiator)

SST:  8 bits, 标准类型由3GPP定义
SD:   24 bits, 运营商自定义, 用于区分同一类型下的不同切片实例

3GPP TS 23.501定义了以下标准化的SST值：

SST值 切片类型 典型场景 SLA要求
1 eMBB（增强移动宽带） 高清视频、AR/VR、云游戏 下行速率>100 Mbps, 时延<10ms
2 URLLC（超可靠低时延） 自动驾驶、工业控制、远程手术 时延<1ms, 可靠性99.999%
3 MIoT（大规模物联网） 智能抄表、传感器网络、物流追踪 百万级连接/km², 低功耗
4 V2X（车联网） 车车通信、车路协同 时延<3ms, 高速移动支持

运营商可以在SST基础上用SD字段进一步细分。比如中国移动可以定义 SST=1, SD=0x000001 为"普通用户eMBB切片"、SST=1, SD=0x000002 为"政企客户eMBB切片"——这样同一个物理网络中可以运行几十甚至上百个不同的切片实例。

19.3 切片选择流程

当一个UE（用户设备）想接入某个网络切片时，需要经过一套完整的选择流程。这个过程涉及UE、AMF、NSSF等多个网元：

UE → (1) RRC Setup Complete: 携带Requested NSSAI
    → gNB → (2) N2: Initial UE Message
    → AMF → (3) 查询NSSF: 选择目标切片实例
    → NSSF → (4) 返回Allowed NSSAI + 目标AMF Set
    → AMF → (5) 通知gNB: PDU Session建立到目标切片
    → UE ← (6) 确认切片接入成功

核心网元NSSF（Network Slice Selection Function）是切片选择的"大脑"。它维护着S-NSSAI到具体网络实例的映射关系，可以基于UE的签约数据、位置、业务类型等因素决定将UE分配到哪个切片实例。

关键点：UE在注册时携带的Requested NSSAI是"我想用什么切片"，AMF结合签约数据之后返回的Allowed NSSAI是"你能用什么切片"。如果UE请求了一个没有签约的切片，AMF会直接拒绝——这是运营商做切片权限控制的核心手段。

19.4 RAN切片与传输网切片

RAN侧切片实现

在无线接入网侧，切片主要通过两种机制实现：

  • PRB资源预留：为高优先级切片（如URLLC）预留一定比例的物理资源块，确保即使网络拥塞，该切片也有最低资源保障。
  • QoS差异化调度：gNB根据切片类型采用不同的调度策略——URLLC切片优先调度（抢占式），eMBB切片采用公平调度，MIoT切片采用稀疏调度以节省UE功耗。

RAN侧切片的最小粒度是DRB（Data Radio Bearer），每个DRB关联一个5QI值，而5QI值又可以映射到特定的S-NSSAI。通过这种方式，gNB知道每个数据包属于哪个切片，从而执行对应的调度策略。

传输网切片实现

在承载网侧，有两种主流的切片隔离方案：

方案 隔离级别 技术实现 适用场景
FlexE（灵活以太网） 硬隔离（物理级） 在以太网MAC层基于时隙分配带宽，5G接口上将一个100GE端口切成多个N×5GE的弹性子端口 URLLC、电力差动保护、工业控制
VPN软隔离 软隔离（逻辑级） 通过SRv6或MPLS VPN对切片流量打上不同的SID标签，在IP/MPLS网络中逻辑隔离 eMBB、企业专线、普通物联网

实践误区：很多初学者以为"有了切片=完全隔离"，这是不准确的。FlexE硬隔离能做到类似于物理隔离的效果——一个切片出问题不会影响另一个。但VPN软隔离只是在IP转发层面做了逻辑区分，如果传输链路出现拥塞，高优先级切片仍然可能受影响。工程部署中，关键业务必须用硬隔离。

19.5 行业应用案例

案例一：电网差动保护切片

国家电网的差动保护业务要求两端变电站之间的通信时延<10ms，且双向时延差<2ms。传统方案需要铺设专用光纤，成本极高。通过5G URLLC切片，在共享无线网络上为差动保护分配独立的FlexE子接口和PRB资源，实测端到端时延稳定在7ms以内，可靠性99.999%。这是5G切片在垂直行业最经典的应用之一。

案例二：港口远程控制切片

某大型港口部署了5G专网，将码头岸桥的远程控制（URLLC切片）、视频监控（eMBB切片）和传感器数据采集（MIoT切片）分别承载在三个独立切片上。远程控制切片的时延<5ms，视频切片的带宽>100Mbps，环境监测切片的连接数>5000个/平方公里。三个切片共享同一套基站和核心网基础设施，但SLA体验完全独立。

19.6 踩坑实录：切片资源配置不足

讲一个真实的案例。2023年，某运营商为一家制造企业部署了5G行业专网，规划了三个切片：URLLC（产线控制）、eMBB（AGV高清回传）、MIoT（环境监测）。初始配置时，工程师给eMBB切片分配了最多的PRB资源（60%），URLLC只给了20%，MIoT给了20%。

上线第一周就出问题了：AGV同时回传高清视频时，eMBB切片的流量暴增，虽然URLLC切片有逻辑隔离，但在传输网侧使用的是VPN软隔离，当eMBB流量占满回传链路时，URLLC的控制指令出现了偶发的300ms抖动。产线机械臂因为控制指令延迟，连续触发了三次急停报警。

排查过程持续了两天，最终定位到根本原因：传输网的带宽资源没有按切片比例做硬隔离，导致eMBB切片挤占了URLLC的带宽。最终整改方案是将传输网升级为FlexE硬隔离，URLLC切片的FlexE子接口带宽独立保障，问题彻底解决。

教训：网络切片不是"配了就行"，必须RAN+传输+核心网三层都做资源隔离。客户端往往只关注无线侧的切片配置，忽略了传输网的带宽争夺。这是目前行业专网中最常见的配置"翻车"原因之一。

19.7 S-NSSAI配置示例

下面是运营商在核心网中配置S-NSSAI的一个示例（简化格式）：

// 5G核心网切片配置文件 (NSSF配置片段)
slice:
  - sst: 1          // eMBB
    sd: "0x000001"  // 默认公网eMBB
    description: "Default public eMBB slice"
    nssf:
      - supported-plmn: "460-00"  // 中国移动
        amf-set: "amf-set-01"
        resource:
          prb-reserved: 60%        // PRB预留比例
          flexe-bandwidth: 10Gbps

  - sst: 2          // URLLC
    sd: "0x000001"  // 工业控制切片
    description: "Industrial URLLC slice"
    nssf:
      - supported-plmn: "460-00"
        amf-set: "amf-set-02"
        resource:
          prb-reserved: 25%
          flexe-bandwidth: 2Gbps   // FlexE硬隔离保障

  - sst: 3          // MIoT
    sd: "0x000001"  // 物联网切片
    description: "Massive IoT slice"
    nssf:
      - supported-plmn: "460-00"
        amf-set: "amf-set-03"
        resource:
          prb-reserved: 15%
          flexe-bandwidth: 1Gbps

这个配置文件清晰地展示了每个切片的SST/SD标识、PRB资源预留比例和传输带宽保障。在生产环境中，NSSF还会配置更细粒度的策略，例如基于UE位置、签约数据、业务类型等条件的切片选择规则。

工程建议：做切片部署规划时，一定要先把业务模型搞清楚——峰值带宽、平均包大小、时延要求、并发连接数。每个切片至少预留30%的弹性余量，给业务增长留空间。我们见过太多"开局一切完美，三个月后切片被业务打爆"的案例。

网络切片让5G网络从"一条路走到黑"变成了"多条路随心选"，是实现行业数字化转型的基石能力。但切片只是5G高阶能力的"第一块积木"——切出来的流量需要就近处理，否则低时延就变成了空谈。下一章，我们将学习MEC边缘计算，看切片+边缘如何组合实现真正意义上的超低时延业务闭环。
