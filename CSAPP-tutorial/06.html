第六章：程序的机器级表示——过程与栈

大家好，我是你们的讲师。如果说上一章的控制流是"程序如何做决策"，那么这一章的核心问题就是"程序如何调用函数"。

函数调用是编程中最基本、最频繁的操作。但你真的知道调用背后发生了什么吗？当函数 A 调用函数 B 时：

  • B 的局部变量存在哪里？A 的局部变量怎么保留下来的？
  • B 执行完后怎么知道回到 A 的哪条指令继续执行？
  • 如果 B 又调用了 C（甚至递归调用自身），这个链条怎么保证不乱套？

所有这些问题的答案，都指向同一个底层机制——栈（Stack）。

我不久前帮一位同事调试一个诡异的 crash，现象是程序在某个函数返回后栈上的局部变量被莫名其妙改写了。我看了半天 C 源码没发现问题，最后调出汇编一看，函数开头压栈保存了 %rbx，但后续代码中某条路径在修改 %rbx 后忘记恢复就提前返回了——导致调用者的 %rbx 被破坏。这就是栈帧管理混乱带来的典型 bug。也是你们学习本章后能避免的错误。

6.1 栈帧结构详解

x86-64 的程序栈向下增长（栈从高地址向低地址生长）。每个函数调用都会在栈上分配一段连续的区域，叫做栈帧（Stack Frame）。典型的栈帧包含以下部分：

地址方向 栈帧内容 说明
高地址
↑
（调用者的栈帧）
↓
低地址		 调用者的栈帧（更高地址处） 调用者的局部变量和状态
返回地址（saved %rip） call 指令自动压栈，指向 call 的下一条指令
被调用者的栈帧 保存的 %rbp（如果函数用帧指针）
保存的 callee-saved 寄存器
局部变量（local variables）
参数构造区（argument build area，用于调用其他函数时传递 >6 个参数）
← %rsp（栈顶指针） 当前最低地址处，栈顶

在 x86-64 中，%rsp 始终指向栈顶（最低地址）。%rbp（基指针）在编译优化级别较低时用于定位栈帧中的变量（因为当前函数内 %rsp 可能在变化，但 %rbp 固定），在 -O1 及以上优化级别中，编译器往往只使用 %rsp 而省略 %rbp（这样就可以把 %rbp 当作通用寄存器来用）。

关于栈溢出：每个线程/进程的栈空间是有限的（Linux 默认 8 MB）。递归调用过深或分配过大的局部变量（如 char buf[1024*1024]）会导致栈溢出——segmentation fault。运行 ulimit -s 可以查看当前栈大小限制。

6.2 call 与 ret 的微操作

call 和 ret 是过程调用的核心指令。它们的微操作非常简单且对称：

callq Label  的等效操作：
    pushq %rip        # 将当前指令的下一条地址压入栈
    jmpq  Label       # 跳转到目标函数

retq  的等效操作：
    popq  %rip        # 从栈顶弹出返回地址，存入 %rip
                      # 这会自动跳转到 call 的下一条指令

这里有一个重要的细节：%rip 不能直接通过 pushq 和 popq 操作（x86 指令集中没有显式的 push/pop %rip 指令），所以上述是逻辑等效而非实际微操作。但理解了这个逻辑，你就理解了函数调用的本质——保存返回点 + 跳转，以及返回——恢复返回点 + 跳转。

为什么 ret 能改变控制流？因为 %rip 寄存器指向当前正在执行的指令地址。popq %rip 从内存中取了一个值写入 %rip，CPU 接下来就会从那个地址取指令。这就是 ret 能跳转回调用者的原因。

安全启示：正因为 ret 指令从栈中恢复 %rip，如果有人能通过缓冲区溢出改写栈上的返回地址，就能让程序跳转到任意位置——这就是缓冲区溢出攻击的基本原理。我们在下一章会详细展开。

6.3 x86-64 调用惯例

调用惯例（Calling Convention）是一套函数调用时参数如何传递、寄存器如何保存、返回值如何处理的约定。x86-64 的 System V ABI（Linux、macOS 等使用的标准）的规则如下：

参数传递寄存器：

参数序号 64 位寄存器 32 位子集 C 类型示例
第 1 个参数 %rdi %edi int, char*, 结构体指针等
第 2 个参数 %rsi %esi 同上
第 3 个参数 %rdx %edx 同上
第 4 个参数 %rcx %ecx 同上
第 5 个参数 %r8 %r8d 同上
第 6 个参数 %r9 %r9d 同上
第 7 个参数及以后 从右向左依次压入栈中（调用者清理）

返回值：

  • 整数/指针类型返回值 → %rax
  • 浮点类型返回值 → %xmm0

寄存器保存规则：

  • Caller-saved（调用者保存）：%rax, %rcx, %rdx, %rsi, %rdi, %r8-%r11。如果调用者在调用后还要用到这些寄存器的值，需要自己保存它们（通常压栈）。
  • Callee-saved（被调用者保存）：%rbx, %rbp, %r12-%r15。如果被调用者想使用这些寄存器，必须先 push 保存原始值，在返回前 pop 恢复。换句话说，被调用者保证返回后这些寄存器的值和调用前一致。

来看一个具体例子，演示参数传递：

// C 代码
long func(long a, long b, long c, long d,
          long e, long f, long g, long h)
{
    return a + b + c + d + e + f + g + h;
}

// 对应的汇编（简化）
func:
    addq    %rsi, %rdi     # a += b
    addq    %rdx, %rdi     # a += c
    addq    %rcx, %rdi     # a += d
    addq    %r8,  %rdi     # a += e
    addq    %r9,  %rdi     # a += f
    addq    8(%rsp), %rdi  # a += 第 7 个参数（g）（已在栈上）
    addq    16(%rsp), %rdi # a += 第 8 个参数（h）
    movq    %rdi, %rax     # 返回值 = a
    ret

⚠️ 实战中的常见错误：在 x86-64 上，如果函数有超过 6 个整数参数，多余的参数在栈上传递，且位于返回地址之上（即更高地址）。新手经常用 0(%rsp) 去取第 7 个参数——不对！因为当前函数的栈帧已经改变了 %rsp，第 7 个参数在 8(%rsp)（考虑返回地址占 8 字节）甚至更远的位置。编译器的调试符号会记录这些信息，但这仍然是手写汇编或阅读反汇编时容易困惑的地方。

6.4 递归函数的栈行为

递归是理解栈帧动态变化的绝佳场景。我们以递归阶乘为例，从汇编层面观察每次递归调用如何分配新的栈帧。

// C 源码
long factorial(long n)
{
    if (n <= 1)
        return 1;
    return n * factorial(n - 1);
}

// GCC -O1 汇编
factorial:
    cmpq    $1, %rdi       # 比较 n 和 1
    jle     .L_base        # 如果 n <= 1，返回 1
    pushq   %rbx           # 保存 %rbx（callee-saved）
    movq    %rdi, %rbx     # %rbx = n （保存起来，因为下面 %rdi 会被覆盖）
    leaq    -1(%rdi), %rdi # %rdi = n - 1 (参数准备)
    call    factorial      # 递归调用 factorial(n-1)
    imulq   %rbx, %rax     # %rax = n * factorial(n-1)
    popq    %rbx           # 恢复 %rbx
    ret
.L_base:
    movq    $1, %rax       # 返回 1
    ret

关键观察：

  • 每次 call factorial 都会自动将下条指令地址压栈
  • pushq %rbx 保存当前 %rbx（因为 factorial 要利用它，而它是 callee-saved 寄存器，必须保持原样返回）
  • 递归深度为 n 时，栈上同时存在 n 个 factorial 栈帧（每个包含返回地址和 %rbx），总共 16n 字节
  • n = 10 时只占 160 字节；n = 100000 时占 1.6 MB——很可能触发栈溢出！

我们用一个 ASCII 图展示递归调用 factorial(3) 的过程栈：

初始调用 factorial(3):

   地址方向    栈内容
   ---------   ------------------------------
   高地址      调用者的栈帧
               factorial(3) 的返回地址 (to caller)
               factorial(3) 保存的 %rbx = 3
               factorial(2) 的返回地址 (回到 factorial+offset)
               factorial(2) 保存的 %rbx = 2
               factorial(1) 的返回地址 (回到 factorial+offset)
               factorial(1) 保存的 %rbx (未使用)
   ← %rsp     当前栈顶 (factorial(1) 的栈帧底部)

   展开过程：
   factorial(3): 检查 n=3, push %rbx=3, call factorial(2)
       factorial(2): 检查 n=2, push %rbx=2, call factorial(1)
           factorial(1): n<=1, return 1
       factorial(2): rax=1, imulq %rbx=2 → rax=2, pop %rbx, ret
   factorial(3): rax=2, imulq %rbx=3 → rax=6, pop %rbx, ret

递归 vs 迭代：递归的代码通常更简洁优雅，但每次调用都有额外的栈帧开销（函数调用指令本身约 1-2 ns，但栈帧分配和寄存器保存/恢复的代价在 5-10 ns 级别）。更重要的是，递归深度受限于栈空间。在现代高性能软件开发中，递归深度超过几百的情况通常需要改为迭代（或使用尾递归优化——如果编译器支持且函数是真正的尾递归）。GCC 的 -O2 能够对简单的尾递归执行优化，变为循环。

本章我们深入探索了函数调用在机器层面的完整过程。从栈帧布局到 call/ret 的微操作，从参数传递的寄存器惯例到递归调用中栈帧的动态分配与释放。理解这些机制，你就能真正理解函数调用不是什么"魔法"，而是基于栈和跳转的确定性机械过程。
