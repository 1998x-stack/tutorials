第12章：流式处理与大数据策略

2024-2026 年间，AI 模型的参数量从千亿级跃升至万亿级，而训练数据量从 TB 级跃升至 PB 级。LAION-5B 有 58 亿图文对，FineWeb 有 15 万亿 token，RedPajama-V2 有 30 万亿 token——这些数字意味着什么？意味着没有任何一台机器的内存能装下它们。你必须学会在"不看全量数据"的情况下完成训练。

这就是流式处理的用武之地。streaming=True 是 datasets 库中最重要的一个参数——它可能也是整个 Hugging Face 生态中最被低估的参数。有了它，你可以在 8GB 内存的笔记本上处理 TB 级的数据集；没有它，你可能连一个 50GB 的数据集都加载不了。

核心认知：流式处理不是"优化"，它是大数据场景下的硬性需求。如果你的数据集超过机器内存的 1/4，就必须用流式。这不再是"选配"，而是"必配"。

流式模式原理

流式模式和非流式模式（常规模式）的本质区别可以用一句话概括：常规模式是"先下载再访问"，流式模式是"边下载边访问"。

from datasets import load_dataset

# 常规模式：数据全部下载到本地，加载到内存
dataset_full = load_dataset("c4", "en", split="train")
print(type(dataset_full))  # 
# 此时数据集已全部下载到 ~/.cache/huggingface/datasets/

# 流式模式：数据按需下载，不存本地
dataset_stream = load_dataset("c4", "en", split="train", streaming=True)
print(type(dataset_stream))  # 
# 此时没有任何数据被下载

# 流式访问：只下载需要的部分
for i, example in enumerate(dataset_stream):
    if i >= 5:
        break
    print(example["text"][:50])  # 只下载前 5 条

流式模式返回的不是 Dataset，而是 IterableDataset。两者最大的区别在于：Dataset 支持随机访问（你可以问 "第 1000 条是什么？"），但 IterableDataset 不支持——它像 Python 的 generator，只能顺序迭代。你不能对它做 dataset[1000]，只能 for example in dataset。

特性 Dataset（常规） IterableDataset（流式）
数据存储 全部下载到本地缓存 迭代时按需下载
内存使用 加载整个数据集 只加载当前 batch
随机访问 支持（通过索引） 不支持（只能顺序迭代）
可重复迭代 支持 支持（每次重新下载）
洗牌 (shuffle) 全局洗牌 缓冲区内洗牌
map/filter 全部执行后返回新 Dataset 惰性执行，按需处理
适用场景 小数据（< 内存），需要随机访问 大数据（> 内存），顺序训练
踩坑故事：某团队接到了一个"多模态大模型训练"任务，核心训练数据是 LAION-5B——58 亿图文对，磁盘占用约 12TB。团队的实习生负责数据准备，他二话不说写了 load_dataset("laion/laion5b")（没加 streaming=True）。几个小时后，服务器磁盘满了，训练脚本报错。实习生以为是磁盘不够大，申请了更大的云盘——然后又爆了。最后一位资深工程师过来看了一眼，加了个 streaming=True，问题瞬间解决。那 12TB 数据根本不需要全部下载下来再训练——数据可以从 S3 上直接流式读取，每个 batch 用完即弃。多加了一个参数，省了 12TB 的磁盘和几个小时的无用等待。

2026 年流式改进

datasets v5.0.0（2026 年初发布）对流式处理做了多项重大改进。如果你还在用 v4.x，现在就应该升级。

Xet 后端：100 倍加速

v5.0.0 引入了全新的 Xet 后端，这是 Hugging Face 自研的分布式数据存储引擎。相比传统的 Git LFS 后端，Xet 在流式场景下的数据读取速度提升了 100 倍。启用方式很简单：

# datasets >= 5.0.0 默认启用 Xet 后端
# 你也可以手动指定
from datasets import load_dataset

# 使用 Xet 后端流式加载
dataset = load_dataset(
    "bigcode/the-stack-v2",
    split="train",
    streaming=True,
    # Xet 自动启用，不需要额外参数
)

# Xet 的核心优势：
# 1. 分块压缩：只传输需要的块，不传输整个文件
# 2. 智能缓存：热点数据在本地缓存
# 3. 并行传输：多线程并行下载 shard
# 4. 断点续传：中断后只重传未完成的块

洗牌算法重写

v5.0.0 之前，流式数据集的洗牌一直是个难题——你不能像常规数据集那样做全局洗牌。旧版本的做法是在一个小缓冲区（默认 1000 条）内随机排列，洗牌效果非常受限。v5.0.0 引入了新的洗牌算法，通过 max_buffer_input_shards 参数来控制：

# v5.0.0 的增强洗牌
dataset = load_dataset("c4", "en", split="train", streaming=True)

# 旧版洗牌（缓冲区大小 = 1000 条）
dataset_shuffled = dataset.shuffle(seed=42)

# 新版洗牌：多 shard 缓冲
dataset_shuffled_enhanced = dataset.shuffle(
    seed=42,
    buffer_size=10000,             # 缓冲区大小（条数）
    max_buffer_input_shards=4,     # 同时缓冲的 shard 数量
)

# 原理：同时打开 4 个 shard，各读取一部分到缓冲区，
# 在缓冲区内随机排列输出。这比单 shard 的洗牌效果好得多。
小技巧：如果你训练的 epoch 数 > 1，流式模式每次迭代都会重新下载数据——这看起来很低效，但实际上因为数据量太大，重新下载比缓存到本地再读取常常更快（磁盘 I/O 可能成为瓶颈）。你可以通过 load_dataset(..., streaming=True).take(1000) 来快速测试数据加载流程。

流式数据的处理模式

流式数据上的 map 和 filter 行为与常规模式有显著差异，理解这些差异对避免 bug 至关重要：

from datasets import load_dataset

dataset = load_dataset("c4", "en", split="train", streaming=True)

# 流式 map：惰性执行
# 注意：流式 map 不支持 num_proc 参数！
def clean(example):
    example["text"] = example["text"].strip()
    return example

dataset_cleaned = dataset.map(clean)
# 此时没有实际执行任何计算！
# 只有当你迭代 dataset_cleaned 时，clean 函数才被调用

# 流式 filter：也是惰性执行
def is_long_enough(example):
    return len(example["text"]) > 100

dataset_filtered = dataset_cleaned.filter(is_long_enough)

# 实际迭代
count = 0
for example in dataset_filtered:
    print(example["text"][:50])
    count += 1
    if count >= 3:
        break

# 流式支持的操作有限
# ✅ 支持的：map, filter, take, skip, shuffle, interleave, flatten
# ❌ 不支持的：select, sort, train_test_split, class_encode_column
关键限制：流式模式下，map 不支持 num_proc 参数！流式数据从设计上就是单进程的迭代器，多进程的语义在这里没有意义（每个进程会重新拉取数据流）。如果你需要对流式数据做多进程加速，正确的做法是先 dataset.to_iterable_dataset() 看看是否支持多进程，或者改用常规模式做预处理后再流式读取。

interleave / skip / take

除了基础的 map 和 filter，IterableDataset 还提供了几个流式场景专用的操作：

# take: 只取前 N 条
c4_sample = load_dataset("c4", "en", split="train", streaming=True)
few_samples = c4_sample.take(10)  # 迭代器，只产生 10 条

# skip: 跳过前 N 条
after_100 = c4_sample.skip(100)   # 跳过前 100 条

# interleave: 交错多个数据集
dataset1 = load_dataset("c4", "en", split="train", streaming=True)
dataset2 = load_dataset("fineweb", "en", split="train", streaming=True)

# 从两个数据集中交替取数据
from datasets import interleave_datasets
mixed = interleave_datasets(
    [dataset1, dataset2],
    probabilities=[0.7, 0.3],  # 70% C4, 30% FineWeb
    seed=42,
)

内存优化策略

即使使用了 streaming=True，不加控制的内存使用也可能导致 OOM。以下是几个重要的内存控制策略：

from datasets import load_dataset

# 1. 控制 batch_size
dataset = load_dataset("c4", "en", split="train", streaming=True)

# 不要一次性取太多数据
# ❌ 错误：把所有数据转换为 list
# all_data = list(dataset)

# ✅ 正确：按 batch 迭代
batch = []
for i, example in enumerate(dataset):
    batch.append(example)
    if len(batch) == 32:  # 每 32 条处理一次
        # 处理 batch
        batch = []

# 2. 利用 prefetch 机制
# datasets 内部使用多线程 prefetch 来隐藏 I/O 延迟
# 你可以通过环境变量调整 prefetch 大小
# export HF_DATASETS_STREAMING_PREFETCH_SIZE=1000

# 3. 100GB 数据 → 10GB 内存的实战配置
def create_memory_efficient_pipeline():
    """创建一个在 10GB 内存上处理 100GB 数据的管线"""
    dataset = load_dataset(
        "c4", "en",
        split="train",
        streaming=True,
    )

    # 预处理尽可能早地压缩数据
    def compress(example):
        # 丢弃原始文本中的冗余空格
        example["text"] = " ".join(example["text"].split())
        return example

    dataset = dataset.map(compress)

    # 过滤掉空文本
    dataset = dataset.filter(lambda x: len(x["text"]) > 0)

    return dataset

# 使用 yield 返回迭代器，避免构建中间列表
def batch_iterator(dataset, batch_size=8):
    batch = []
    for example in dataset:
        batch.append(example)
        if len(batch) == batch_size:
            yield batch
            batch = []
    if batch:
        yield batch

流式训练完整示例

让我们把所有知识整合成一个完整的流式训练示例——从流式加载到训练循环：

from datasets import load_dataset
from transformers import AutoTokenizer

# 1. 流式加载
dataset = load_dataset(
    "c4", "en",
    split="train",
    streaming=True,
)

# 2. 流式预处理（惰性）
tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")

def preprocess(example):
    encoded = tokenizer(
        example["text"],
        truncation=True,
        max_length=512,
        padding=False,
    )
    return {
        "input_ids": encoded["input_ids"],
        "attention_mask": encoded["attention_mask"],
    }

dataset = dataset.map(preprocess, batched=True, batch_size=1000)

# 3. 洗牌
dataset = dataset.shuffle(seed=42, buffer_size=10000)

# 4. 训练循环（伪代码）
def train_on_stream(dataset, num_steps=1000, batch_size=16):
    optimizer = ...  # 你的优化器
    model = ...      # 你的模型

    batch = []
    for step, example in enumerate(dataset):
        if step >= num_steps * batch_size:
            break

        batch.append(example)
        if len(batch) == batch_size:
            # 将 batch 转换为张量
            inputs = {
                "input_ids": torch.tensor([b["input_ids"] for b in batch]),
                "attention_mask": torch.tensor([b["attention_mask"] for b in batch]),
            }
            # 前向 + 反向
            outputs = model(**inputs)
            loss = outputs.loss
            loss.backward()
            optimizer.step()
            optimizer.zero_grad()
            batch = []

            if step % (100 * batch_size) == 0:
                print(f"Step {step // batch_size}, Loss: {loss.item():.4f}")

# 性能基准：流式 vs 非流式
对比场景 非流式 流式 备注
数据集大小 5GB 5GB 两者都能处理
内存峰值 8.5 GB 1.2 GB 流式占优
首次迭代延迟 120 秒（下载+加载） 0.5 秒 流式秒开
步进速度（step/s） 12.3 11.8 差距极小
数据集大小 500GB 500GB 非流式不可用
结论：当数据集可以放入内存时，流式和常规模式在训练速度上几乎没有差别。但当数据集超出内存时，流式模式是唯一的选择。与其等数据全量下载完再训练，不如在数据流到达的第一时间就开始训练。这个思路转换会让你在处理大数据集时的效率提升一个数量级。

流式处理让你能处理"一台机器放不下"的数据集。但当我们进入分布式训练场景时，你还需要额外解决"多个 GPU 怎么分数据"的问题——每个 GPU 都要看到不同的数据，但都不能加载全部数据。下一章的数据分片和分布式训练正好解决这个问题。
