第10章 作业车间调度（下）

大家好，我是你们的讲师。上一章我们认识了 JSP 的基本模型和析取图工具。但知道问题是什么、会建模，和能解出答案——这中间还隔着一条巨大的鸿沟。就好比你知道要爬一座山，也看到了山在哪里，但真正难的是找到一条上山的路。

今天我们就来攻 JSP 这座山。三种主流的求解范式会依次登场：分枝定界法追求精确最优，Shifting Bottleneck 寻找问题的结构要害，禁忌搜索用智能的邻域搜索逼近最优解。

分枝定界法解JSP — B&B的四要素

分枝定界法（Branch and Bound, B&B）是求解 JSP 精确最优解的核心框架。它的思路不难理解：系统地枚举所有可能的调度，但通过"上界"和"下界"来剪掉那些不可能比当前最优更好的分支。B&B 的设计围绕四个要素展开：

1. 分支策略（Branching）

在 JSP 中，分支通常通过固定析取弧的方向来实现。从初始的"所有析取弧未定向"开始，每一步选择一对未定方向的析取弧，尝试两种可能的方向，生成两个子节点。常用的分支策略包括：

  • 基于关键路径的分支——选取关键路径上的操作，为它涉及的未决定析取弧做分支。这个策略最常用，因为它直接针对影响 makespan 的操作。
  • 基于GT算法的分支——在 GT 算法产生冲突集时，每种选择作为一个分支。

2. 定界方法（Bounding）

定界是 B&B 的灵魂。好的下界可以大幅减少搜索量。JSP 常用的下界包括：

  • 基于操作时间之和的下界：L₁ = max(每个工件的总加工时间之和)。
  • 基于机器负载的下界：L₂ = max(每台机器的总加工时间)。
  • 基于关键路径的下界：考虑工艺约束后，从源点到当前节点的最长路径长度。
  • 调整后的下界：L = max(L₁, L₂, 关键路径长度) —— 综合取最大。

3. 搜索策略（Search）

  • Best-First Search：总是展开下界最小的节点。优点是找到最优解后可以快速收束；缺点是内存需求大。
  • Depth-First Search：沿着一条路径走到底。优点是内存小；缺点是可能在一个糟糕的分支上浪费太多时间。
  • 有限回溯：结合前两者的优势，先 depth-first 找到一个可行解作为上界，再用 best-first 搜索其余分支。

4. 剪枝规则（Pruning）

三种剪枝条件：

  • 下界 ≥ 当前全局上界：剪枝。因为这一分支永远不可能比已有的可行解更好。
  • 不可行：如果分支导致有向环（违反工艺约束），剪枝。
  • 占优（Dominance）：如果某个节点被另一个节点支配（意味着无论如何扩展都不会更优），剪枝。
分枝定界法解JSP — 伪代码

算法 BranchAndBound(JSP问题实例)
  初始化:
    上界 UB ← 用启发式求一个可行解
    下界 LB ← 基于机器/工件负载计算
    根节点 ← 所有析取弧未定向
    优先队列 Q ← {根节点}

  while Q 非空:
    取出下界最小的节点 N

    if LB(N) ≥ UB:
      continue  // 剪枝

    if N 中所有析取弧已定向:
      makespan ← 计算关键路径长度
      if makespan < UB:
        UB ← makespan
        更新最优解
      continue

    选择一对析取弧 (u,v)
    创建左子节点: 方向 u→v
    创建右子节点: 方向 v→u

    for 每个子节点 C:
      检查是否形成有向环 → 剪枝
      计算 LB(C)
      if LB(C) < UB:
        将 C 加入 Q

  return 最优解

Shifting Bottleneck启发式 — 识别并优化瓶颈机器

如果 B&B 追求的是"最优"，那么 Shifting Bottleneck（SB） 追求的是"最高性价比"。SB 算法由 Adams、Balás 和 Zawack 在 1988 年提出，它的核心思想极其优雅：找到当前最拥挤的机器（瓶颈），优化它，然后锁定它，再找下一个瓶颈。

算法步骤如下：

  1. 初始化：所有机器都未调度（未排序）。
  2. 迭代：
    1. 对于每台未调度的机器，求解一个单机调度子问题（考虑已调度机器带来的释放时间和交货期约束），得到一个候选序列和对应的 makespan。
    2. 选择 makespan 最大（即瓶颈最严重）的那台机器，锁定它的加工顺序。
    3. 标记该机器为"已调度"。
  3. 重新优化：每次调度新机器后，对所有已调度的机器进行局部重优化——因为新加约束可能会使之前调度的顺序变差。
  4. 重复直到所有机器都被调度。

SB 算法的实际表现非常好。在 FT10 上，SB 能很快找到 965 左右的解（最优是 930），而计算时间只需要几秒。相比 B&B 可能需要几小时甚至更久，SB 提供了一个极其诱人的"性价比"平衡。

Shifting Bottleneck 算法框架

输入：JSP 实例
输出：近优调度方案

1. M0 ← ∅  // 已调度的机器集合
   M ← 所有机器的集合

2. while M0 ≠ M:
   // 步骤A: 确定每个未调度机器的瓶颈指标
   for each k ∈ (M \ M0):
       求解单机调度问题 SP_k:
         - 考虑已调度机器带来的时间窗约束
         - 目标: 最小化该单机问题的最大延迟 L_max
       记录最优值 L_k

   // 步骤B: 选择瓶颈机器
   k* ← argmax L_k  // L_k 最大的那个机器
   将 k* 标记为已调度，锁定其上操作顺序
   M0 ← M0 ∪ {k*}

   // 步骤C: 重优化
   for each k ∈ M0:
       在当前全局约束下重新求解 SP_k
       如果新的解更好，更新 k 的加工顺序

3. 根据所有已析取的机器顺序，计算完整的 makespan

4. return 调度方案

禁忌搜索解JSP — 邻域设计与禁忌表管理

禁忌搜索（Tabu Search, TS）是目前求解 JSP 最成功的元启发式算法之一。它的核心思想是：从一个初始解开始，在"邻域"中搜索更好的解，同时用一个"禁忌表"记录最近访问过的解，防止走回头路。

在 JSP 的禁忌搜索中，最关键的设计是邻域结构：

  • N1 邻域：交换关键路径上相邻的、在同一台机器上的两个操作的顺序。这是最基本的邻域。
  • N5 邻域：移动关键路径上的一个操作到同一台机器上的另一个位置（不限于相邻交换）。范围更广，搜索能力更强。
  • N7 邻域：同时调整多个关键操作，是 N5 的扩展。搜索能力强但计算成本高。

禁忌表管理方面：

  • 禁忌长度：通常设置为 7-20 之间。太短容易循环，太长会限制搜索。
  • 特赦准则（Aspiration）：如果某个被禁忌的移动能产生比当前全局最优更好的解，则解除禁忌。这是防止"好解被浪费"的保险机制。

一个真实的工程踩坑：有一次我将一个 JSP 问题部署到生产线时，发现禁忌搜索的解决方案虽然在计算上非常优秀，但有一个致命的缺陷——它频繁地让同一个工件在机器之间来回切换。比如工件 A 在机器 1 上加工了 10 分钟，然后去了机器 3，过了 5 分钟又回到机器 1。这在理论上完全可行（Job Shop 允许多次访问同一台机器），但在工厂里这意味着操作工需要反复移动工件。最后我们不得不在目标函数中加上一个工件移动距离惩罚项。这再次说明：数学上可行不等于工程上可行。

最后，我们来对比 JSP 的几种主流求解方法：

方法 最优性 计算时间 适用规模 实现难度
分枝定界法（B&B） 精确最优 指数级（小规模适用） n ≤ 15, m ≤ 10 高
Shifting Bottleneck（SB） 近似（通常偏差 3%-8%） 几分钟 n ≤ 100, m ≤ 20 中
禁忌搜索（TS） 近似（可逼近 1%-3%） 几分钟到几小时 n ≤ 200, m ≤ 30 中
遗传算法（GA） 近似（偏差 5%-15%） 几分钟到几小时 n ≤ 500, m ≤ 50 低

JSP到底有多难？——FT10的传奇故事：1963年，Fisher 和 Thompson 提出了 FT10 这个 10×10 的 JSP 实例。从那天起，研究者们不断地寻找它的最优解。1970年代，有人找到 makespan = 1007；1980年代，有人推进到 960；直到 1989 年，才由 Applegate 和 Cook 用 B&B 法证明最优解是 930。整整 25 年！而且 FT10 只是 10 个工件、10 台机器而已，放到今天的制造业中，连"中小规模"都算不上。每次想到这个故事，我都会感慨：如果你在公司的 JSP 项目中没有找到最优解——别自责，前辈们花了四分之一个世纪才解决一个 10×10 的问题。

这一章我们深入了 JSP 的求解核心。B&B 提供了理论上的最优保障，SB 展现了结构洞察的力量，TS 则展示了智能搜索的艺术。下一章我们将进入另一个重要的调度场景——平行机与批量调度。当你有不止一台机器可以并行工作时，问题又会出现新的维度。我们下一章见。
