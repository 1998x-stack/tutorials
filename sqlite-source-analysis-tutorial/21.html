第21章 FTS全文搜索

大家好，我是你们的讲师。上一章我们学习了虚拟表接口——SQLite 的"存储引擎插件系统"。这一章我们来探索虚拟表家族中最重要的成员——FTS（Full-Text Search）全文搜索模块。

想象一下：你有数千篇文章，想找到所有包含"SQLite 性能优化"的文章。用 LIKE '%SQLite%性能%优化%' 不仅慢，而且无法实现精确短语匹配、按相关性排序等高级功能。FTS 就是为此而生的——它把倒排索引引擎装进了 SQLite。

FTS 是什么？ SQLite 内置的全文搜索引擎，基于倒排索引实现，支持短语搜索、前缀搜索、布尔搜索和自定义排序。FTS3/4/5 三代变体代码位于 ext/fts3/ 和 ext/fts5/ 目录。

FTS3/4/5 的演进

版本引入时间核心特点代码位置
FTS33.6.0 (2009)基础倒排索引，MATCH 语法ext/fts3/fts3.c
FTS43.7.4 (2011)压缩存储 compress/uncompress、matchinfo()ext/fts3/fts3.c（同文件）
FTS53.9.0 (2015)全新架构、BM25 排名、列查询ext/fts5/fts5*.c

FTS3 和 FTS4 共享同一个 fts3.c 源码文件——FTS4 只是在 FTS3 基础上增加了额外功能。FTS5 则是完全重写的架构。

倒排索引数据结构

倒排索引和传统 B-Tree 的"正向索引"恰好相反：正向索引回答"这篇文档里有哪些词"，倒排索引回答"这个词出现在哪些文档中"。

/* FTS3/4 的倒排索引结构（逻辑上） */
/*
 * 每个 segment 包含 term → doclist 的映射。
 * doclist 格式：
 * +------------------+------------------+-------+
 * | docid (varint)   | position list    | docid | ...
 * +------------------+------------------+-------+
 *
 * position list 是变长编码的词位置数组，用于短语查询。
 */

/* segment 合并操作的核心结构 */
typedef struct Fts3SegReader Fts3SegReader;
struct Fts3SegReader {
  int iStartBlock;
  int iEndBlock;
  const char *zTerm;       /* 当前 term */
  int nTerm;
  const char *aDoclist;    /* 当前 term 的文档列表 */
  int nDoclist;
};

Segment 合并 是 FTS 索引维护的核心。插入新文档时，FTS 不会立即重建整个索引，而是创建小 segment。后台合并进程将小 segment 归并为大 segment——类似于 LSM-Tree 的 compaction。如果大量写入时发现 FTS 表越来越慢，通常是合并策略需要调优。

分词器实现

分词器将文本切分成词条。默认的 simple 分词器按空格和标点切分并转小写：

/* fts3_tokenizer1.c — simple 分词器的核心 */
static int simpleNext(sqlite3_tokenizer_cursor *pCursor,
  const char **ppToken, int *pnToken,
  int *piStartOffset, int *piEndOffset, int *piPosition){
  simple_tokenizer_cursor *c = (simple_tokenizer_cursor *)pCursor;
  while( c->iOffset < c->nInput ){
    int i = c->iOffset;
    unsigned char ch = c->zInput[i];
    /* 跳过分隔符 */
    if( ch==' ' || ch==',' || ch=='.' || ch<=0x20 ){
      c->iOffset++; continue;
    }
    int start = i;
    while( i<c->nInput && !(c->zInput[i]==' ' || c->zInput[i]==',') ) i++;
    *ppToken = &c->zInput[start];
    *pnToken = i - start;
    *piStartOffset = start; *piEndOffset = i;
    *piPosition = c->iToken++;
    c->iOffset = i;
    return SQLITE_OK;
  }
  return SQLITE_DONE;
}

SQLite 还提供了 porter（词干提取，搜索 "running" 匹配 "run"）和 unicode61（支持非拉丁文字）两种内置分词器。中文文本用 unicode61 会按单字切分，如需精确中文分词需集成自定义分词器。

💡 FTS 表创建时通过 tokenize= 选项指定分词器。例如 CREATE VIRTUAL TABLE t USING fts4(content, tokenize=porter unicode61) 组合使用词干提取和 Unicode 分词。

FTS 查询

-- 创建 FTS4/5 表
CREATE VIRTUAL TABLE docs USING fts4(title, content);

-- 简单查询
SELECT * FROM docs WHERE content MATCH 'SQLite';
SELECT * FROM docs WHERE content MATCH 'SQLite OR 性能';

-- 短语查询
SELECT * FROM docs WHERE content MATCH '"数据库引擎"';

-- 前缀/布尔查询
SELECT * FROM docs WHERE content MATCH 'SQL*';
SELECT * FROM docs WHERE docs MATCH 'SQLite NOT Oracle';

-- FTS4 辅助函数
SELECT highlight(docs, 0, '<b>', '</b>') FROM docs WHERE content MATCH 'SQLite';
SELECT offsets(docs) FROM docs WHERE content MATCH 'SQLite';
SELECT matchinfo(docs) FROM docs WHERE content MATCH 'SQLite';
⚠️ MATCH 与子查询： 不要写 SELECT * FROM t WHERE rowid IN (SELECT docid FROM t WHERE content MATCH 'x')。外层 rowid IN 会绕过 FTS 索引，导致回表逐行检查。直接用 SELECT * FROM t WHERE content MATCH 'x'，让 MATCH 驱动查询。

FTS4 压缩与 FTS5 改进

FTS4 的 compress/uncompress 选项在数据写入时压缩、读取时解压。使用 zlib 压缩后，索引磁盘占用可减少约 60-70%，对移动端和嵌入式设备意义重大。

FTS5（3.9.0+）是彻底重写的设计，核心改进：

  • 内置 BM25 排名 — SELECT *, rank FROM t WHERE t MATCH 'x' ORDER BY rank
  • 前缀索引 — 通过 prefix='2,3' 选项加速前缀查询
  • 列权重 — title^10, content^1 让标题匹配权重更高
  • 64 位 docid — 支持更大的表
功能FTS3/4FTS5
排名算法需 matchinfo 自己算内置 BM25
压缩存储compress/uncompress自带压缩
前缀索引不直接支持prefix 选项原生支持
列权重不支持title^10 语法
NGram 索引不支持tokenize=ngram

真实故事：FTS 在移动端搜索中的应用

2019 年我参与了一个手机笔记应用的开发。用户写了数千条笔记，需要全局搜索标题和正文、按相关性排序并高亮关键词。

第一版用 LIKE '%keyword%'，笔记超过 500 条后搜索响应飙升到 3-5 秒。第二版用 FTS4 加触发器同步主表和索引表，搜索响应降到 50 毫秒以下，同时支持了短语搜索和近似匹配。FTS 索引额外占用约原始文本 30% 的空间，在移动端完全可以接受。

但是遇到了一个坑：批量导入时如果禁用了触发器，FTS 索引会与主表不一致。最终修复是在每次应用启动时执行 INSERT INTO t(t) VALUES('rebuild') 重建索引，虽然需要 1-2 秒，但确保了数据一致性。

移动端黄金组合： FTS 的倒排索引天然适合"写少读多"的移动端场景。笔记、短信、通讯录这些写入频率低但搜索频率高的数据，使用 FTS 可获得媲美原生搜索引擎的体验，且无需维护独立的搜索引擎服务。
