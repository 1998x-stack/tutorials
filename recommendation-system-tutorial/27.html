第27章 推荐系统架构设计

上一章我们抬头看天，畅想了推荐系统的前沿趋势。这一章我们低头看路——如果你明天就要开始搭建一个推荐系统，从哪里开始？这可能是整门课程中最实用的一章。我们不谈花哨的模型，只谈怎么从零到一、从小到大，以及怎么让你的团队睡得着觉。

1. 从0到1的推荐系统

你加入了一家创业公司，产品刚刚上线，DAU不到1万，老板说"我们也要推荐系统"。你怎么做？

很多人的第一反应是："先搭一个四阶段漏斗——召回、粗排、精排、重排，双塔模型做召回，DeepFM做精排……" 这就是典型的"照着大厂抄作业"的错误。

大厂的四阶段漏斗不是设计出来的，是进化出来的。YouTube在2016年才引入深度学习推荐，快手在2017年后才有了完整的四阶段架构。在那之前，他们用的是简单得多但同样有效的方案。对初创产品来说，最重要的是先跑起来，再优化。

这里推荐一个四阶段的渐进式路线图：

阶段 时间窗口 推荐策略 核心关注点 团队规模
Phase 1 第1-4周 热度排序 + 简单规则（最新、最热、编辑推荐） 快速上线、验证产品价值 1-2人
Phase 2 第2-3月 协同过滤（ItemCF + UserCF），UserBase热榜 利用用户行为数据，产生个性化 2-3人
Phase 3 第4-6月 LR/FM + 简单特征工程 排序模型引入，特征驱动效果提升 3-5人
Phase 4 第7-12月 Deep模型 + 多路召回 + 多阶段 工程体系化，A/B实验平台 5-10人

踩坑故事：我曾经认识一位创业公司的技术负责人，他的团队花了整整6个月搭建一个"媲美美团的推荐系统"——多路召回、实时特征、复杂的模型ensemble。结果产品上线后DAU只有几千，推荐系统每天的请求量还不如他们模型训练日志里的一条记录多。更讽刺的是，他们长期忽略了最基础的"热门内容"推荐，直到用户调研才发现——新用户进入App后看到的是"空白推荐流"（因为模型没有他们的行为数据）。最终他们回退到90%的热度推荐 + 10%的个性化推荐，迭代3个月后才逐步提高个性化比例。这个故事告诉我们一个血的教训：在没有足够用户行为数据之前，花哨的模型不如一条好规则。

渐进式路线图的核心原则：先用简单的方案验证产品价值，积累用户和数据，然后逐步引入更复杂的模型。每一步的升级都有明确的数据支撑——不是"为了用DeepFM而用DeepFM"。

2. 容量规划与技术选型

假设你的产品已经跑起来了，DAU到了10万甚至100万，这时候你需要认真考虑容量规划。让一个工程师最痛苦的事情之一就是——上线前没做容量预估，结果推荐服务的延迟在高峰时段从50ms飙到了5秒。

容量预估的基本公式很简单：

# QPS（每秒查询数）预估伪代码
DAU = 10_0000           # 日活用户数
avg_sessions_per_user = 3.0   # 平均每人每天启动app次数
requests_per_session = 20     # 每次启动触发几次推荐请求
peak_factor = 4.0             # 高峰时段因子（通常是均值2-5倍）

daily_requests = DAU * avg_sessions_per_user * requests_per_session
# = 10万 * 3 * 20 = 600万次/天

avg_qps = daily_requests / (24 * 3600)
# = 600万 / 86400 ≈ 70 QPS

peak_qps = avg_qps * peak_factor
# ≈ 280 QPS  (峰值)

# 存储预估
item_count = 100_0000               # 100万物品
embedding_dim = 64
embedding_size_bytes = item_count * embedding_dim * 4  # float32
# = 100万 * 64 * 4 = 256 MB  (embedding向量总大小)

# 延迟预算
latency_budget = {
    "total": 300,        # 总延迟上限 300ms
    "recall": 50,        # 召回阶段 50ms
    "pre_rank": 20,      # 粗排阶段 20ms
    "rank": 100,         # 精排阶段 100ms
    "re_rank": 20,       # 重排阶段 20ms
    "network": 110,      # 网络传输和其他开销
}

有了这些基础数字，你就可以做出明智的选型决策：

  • 300 QPS的峰值负载，单台8核16G的服务器就够了（大部分请求无状态，可以水平扩展）
  • 256MB的embedding，完全可以用内存加载，不需要专门的向量数据库
  • 300ms的延迟预算，所有模块可以采用同步调用，不需要引入异步消息队列

当你的DAU到了1000万时，上面的数字就需要重新计算了——峰值QPS可能达到2-3万，embedding可能达到几个GB，这时候你就需要Faiss进行近似最近邻搜索、用Redis做特征缓存、考虑模型分片部署。

不同规模的公司适用的架构差异很大：

公司规模 DAU范围 推荐架构 典型团队
初创/小团队 <10万 单服务 + 离线模型 + Redis缓存 1-2人兼职
成长期 10万-100万 召回的分离 + 基础实验平台 3-5人
中型 100万-1000万 独立召回/排序服务 + Faiss + A/B平台 5-15人
大型 1000万+ 多阶段漏斗 + 实时学习 + 多目标优化 20-50+人

3. 技术选型决策树

另一个常见的问题是：到底应该自己造轮子还是用现成的？这个问题没有标准答案，但有一个决策树可以帮你理清思路：

什么时候自己造（Self-Build）？

  • 你的推荐需求非常特殊，现有框架无法满足（比如你想做生成式推荐，目前没有开源方案）
  • 你的团队小于10人——听起来反直觉，但小而精的团队沟通成本低，造轮子效率反而高
  • 你需要对系统的每个细节有完全的控制权

什么时候用开源框架？

  • 你的需求比较标准——召回+排序，CTR预估，标准的推荐pipeline
  • 你想在现有的成熟方案基础上做定制化开发
  • 推荐的候选框架：TorchEasyRec（阿里开源，工业级，支持大规模分布式训练）、FuxiCTR（学术界风格，模型丰富，适合研究和对比实验）、DeepCTR（入门友好，但不再活跃维护）

什么时候用云服务？

  • 团队规模极小（1-3人），没有专门的算法工程师
  • 需要快速上线（几天内而非几个月内）
  • 愿意用成本换速度（云服务通常按量付费，规模上来后比自建贵很多）
  • 推荐的方案：阿里云PAI（国内）、AWS Personalize（海外）、Google Recommendations AI

务实建议：很多团队的最佳路径是"混合模式"——用开源框架训练模型（TorchEasyRec），用云服务处理基础设施（云上的特征存储、模型部署），核心模型和业务逻辑自研。这样既保证了对推荐效果的把控，又不需要在基础设施上重复造轮子。

4. 推荐团队组织

技术选型解决了"用什么做"的问题，另一个同样重要的问题是"谁来做"。推荐系统的建设不只是算法工程师的工作，它需要多种角色的紧密协作。

一个比较典型的推荐团队分工如下：

  • 算法团队（Algorithm Team）：负责模型设计、离线评估、特征设计。他们研究论文、做实验、调参数。核心交付是模型和实验报告。
  • 工程团队（Engineering Team）：负责数据pipeline、模型serving、基础设施。他们确保推荐服务稳定、低延迟、可扩展。核心交付是线上服务和监控告警。
  • 数据团队（Data Team）：负责特征平台、数据质量、实验平台。他们管理海量的用户行为数据，确保特征的正确性和时效性。核心交付是高质量的训练数据和特征服务。
  • 产品团队（Product Team）：负责业务指标定义、业务规则、用户研究。他们定义什么叫"好推荐"——不是AUC 0.8就是好，而是用户留存提升了、GMV增长了、投诉变少了。核心交付是MRD和实验分析报告。

这四个团队的协作方式，决定了整个推荐系统的成败。我见过最有问题的协作模式是"扔过墙"（Throw it over the wall）：产品写了一份需求发给算法，算法做了一版模型发给工程，工程部署完就不管了。每个阶段的信息损耗至少30%，到最后线上的推荐结果和产品最初的设想已经大相径庭。

更好的协作模式是端到端Squad——围绕一个业务目标（比如"提升视频推荐流用户时长"），从产品、算法、工程、数据各抽一个人组成一个小组，对最终指标共同负责。这个小组成员从问题定义、实验设计到线上效果分析全程参与，信息不经过任何中间人的过滤。

组织智慧：推荐系统中最难的不是模型，而是对齐——对齐业务目标和技术方案，对齐算法效果和工程约束，对齐离线指标和线上效果。很多公司投入了大量资源做推荐，最终效果不佳，往往不是因为模型不够好，而是因为团队之间的"信息孤岛"导致了大量的重复工作和无效沟通。如果你是一个推荐团队的技术负责人，花在"对齐"上的时间应该不少于花在"研究"上的时间。

5. 中小公司的务实路线

这一节是专门写给中小公司和大厂里的非核心业务团队的。你们不需要、也不应该复制美团或快手的推荐架构。

这里有三条务实原则：

原则一：更好的特征 > 更复杂的模型

对于大多数中小规模的应用，花时间设计好的特征比折腾模型的收益大得多。一个好的特征（比如"用户在过去24小时内对该类内容的点击次数"）可能让你的AUC提升2-3%，而把DeepFM换成DCNv2可能只提升了0.1-0.3%。先把特征工程做到极致，模型用最简单的DeepFM就够了。

原则二：离线批量（小时级）> 实时流式

实时推荐听起来很酷，但实现成本很高——你需要实时特征pipeline、实时模型更新、低延迟推理。对大多数场景来说，小时级的离线批量更新已经足够好了。用户的行为模式在几十分钟内不会有剧烈变化，你不需要在几秒内反映用户的每一次点击。只在确实需要的地方引入实时——比如新闻推荐中用户对"已读内容"的过滤。

原则三：一个精的通道 > 十个糙的通道

很多团队在刚开始做推荐时，喜欢搞"多路召回"——协同过滤一路、双塔一路、热榜一路、标签一路、社交一路……结果每一路都做得不够好，最后的融合策略也很难设计。不如集中精力做好一个召回通道（比如双塔模型）+ 一个排序模型（比如DeepFM），等这个pipeline跑顺了、吃透了，再考虑增加新的通道。这个策略的另一个好处是你的系统复杂度始终可控，出了问题你能快速定位。

该买的不要自己建

有一些基础设施级别的能力，除非你的团队规模和技术实力非常强，否则建议直接购买云服务：

  • 特征存储：自建特征平台需要解决存储一致性、低延迟读取、特征回溯等技术难题。直接用云上的Feature Store服务（如阿里云Flink + Hologres方案）。
  • 向量数据库：除非你的embedding规模在亿级以上，否则不要自建向量检索服务。用Mlivus Cloud、Pinecone、Weaviate Cloud这些托管服务，每年花的钱远低于一个工程师的工资。
  • A/B实验平台：自建实验平台需要考虑分流的一致性、统计显著性计算、长期效果归因。可以使用第三方服务或者用开源的分流SDK（如PlanOut）快速接入。

深夜原则：一个经常被忽视的架构评价标准是——如果你的推荐服务凌晨3点挂了，你的团队中至少有一个人能在30分钟内把它修好。如果你的架构需要3个人在3小时内才能修好，那就是过度的架构。对于大多数中小团队来说，最佳架构是让你的团队在凌晨3点还能睡得着觉的架构。复杂度是一种负债，不是资产。

总结

本章我们从务实的角度讨论了推荐系统的架构设计。核心结论是：不要一开始就照搬大厂的架构，而是根据你的业务规模和发展阶段，选择最合适的方案。从简单的热度排序起步，逐步引入协同过滤和模型，最后才构建完整的工业级pipeline。容量规划决定了你的系统能扛多大压力，技术选型决定了你的开发效率，团队组织决定了你的沟通成本，而务实原则决定了你能走多远。
