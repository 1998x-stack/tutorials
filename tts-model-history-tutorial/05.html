第5章 HMM统计参数合成

大家好，我是你们的讲师。上一章我们讲了拼接合成，它的核心理念是"从真实录音片段中选出最好的来拼"。这个方法效果不错，但问题也很棘手：为了最好的效果，你需要一个巨大、精心设计和标注的录音库；你无法灵活地改变声音风格；而且偶尔的拼接毛刺总会在最不该出现的时候冒出来。

那么，有没有一种方法，既能享受数据驱动带来的自然度提升，又能拥有类似于共振峰合成的灵活性和可控性？答案是统计参数合成（Statistical Parametric Speech Synthesis, SPSS）。这一章的主角，就是开启了SPSS时代的隐马尔可夫模型（HMM）。

HMM如何给语音建模？

想象你有一句录音"你好"。如果用HMM的思路建模，你会这样思考：这句话可以看作一个由四个音素组成的序列——"n"、"i"、"h"、"ao"。每一个音素在发音时，声道形状会经历一系列的状态变化（比如舌位从准备到稳定再到离开）。HMM恰好是一个可以用来建模这种"序列生成过程"的统计工具。

在拼接合成中，你需要存储每一个音素的实际录音片段。而在HMM合成中，你只需要存储每个音素的统计模型——一个描述该音素声音特征的概率分布。合成时，从这些分布中"采样"出最可能的声学参数序列，然后用声码器合成波形。

这种"从存储到建模"的转变，深刻地改变了TTS的技术格局。

隐马尔可夫模型基础回顾

隐马尔可夫模型是一种对时序数据建模的概率模型，包含两个核心组件：

  • 隐藏状态（Hidden States）：一系列不直接可见的状态，代表发音过程中声道所处的不同"阶段"。在语音HMM中，一个音素通常建模为3-5个由左到右的状态（left-to-right topology），对应"发音开始→稳定维持→过渡到下一个音"。例如，元音/a/可能用5个状态来建模，分别对应舌位从/a/前的过渡位置、到达/a/的稳定位置、保持稳定、开始向下一音素过渡、转至下一个音的起始位置。
  • 发射概率（Emission Probability）：每个隐藏状态产生观测特征（如MFCC、F0、能量）的概率分布。通常用高斯混合模型（GMM，Gaussian Mixture Model）来表示。简单说，就是"当声道处于这个形状时，最可能产出什么样的声学参数"。

HMM最早在1970-1980年代被广泛应用于语音识别（ASR）领域。有趣的是，语音识别和语音合成在很多技术上是"镜像"关系——识别的目标是给定观测找到最可能的文本，而合成的目标是给定文本生成最可能的观测。所以当HMM在ASR中获得巨大成功后，TTS研究者很自然地想到：为什么不把HMM反过来用呢？这就是HMM统计参数合成的出发点。

HTS工具包：HMM-based TTS

2002年，日本名古屋工业大学（Nagoya Institute of Technology）的Tokuda教授团队发布了HTS（HMM-based Speech Synthesis System）工具包。HTS基于CMU的HTK（Hidden Markov Model Toolkit）语音识别工具包进行了扩展和改造，使之适合语音合成任务。HTS成为了整个SPSS时代最具有影响力的开源工具，没有之一。

HTS的核心创新包括以下几个方面：

  • 上下文相关HMM（Context-Dependent HMM）：语音的发音受上下文影响极大——同一个音素在句子开头和中间、重读和轻读、前后接不同音素时，声学表现截然不同。HTS引入了上下文属性的概念，为每个音素标注数十个甚至上百个上下文特征，包括：前一个音素的类别、后一个音素的类别、音节内的位置、词内的位置、词性、句法边界类型、与重读/重音的关系等等。如此一来，"a"在"cat"中和在"father"中会被建模为不同的上下文相关HMM，尽管它们属于同一个音素。
  • 决策树聚类（Decision Tree Clustering）：上下文相关的组合爆炸问题非常严重——即使只有50个音素和20个二元上下文类别，潜在的状态数也能达到数万甚至数百万。但训练数据是有限的，很多上下文组合在数据中根本没有出现。HTS的解决方案是用决策树对HMM状态进行聚类：从所有状态开始，不断用语言学问题（"前一个音素是元音吗？""当前音节是重读音节吗？"）对状态集进行分裂，直到满足停止准则。同一叶子节点的状态共享同一个发射概率分布。这样就实现了"数据驱动的泛化"——即使某个上下文组合在数据中从未出现，也可以通过决策树找到最相似的已观察状态来近似。

一个典型的HTS训练流程如下：

# HTS标准的训练命令序列（简化版）

        # 阶段1：准备数据和对齐标签
        # 用HTK的HVite工具做强制对齐
        HVite -l lab/ -a -m -H hmms/hmm15/hmmdefs \
              -o output.mlf -S train.scp dict.txt monophone.list

        # 阶段2：训练单音素HMM（无上下文）
        # 迭代训练，逐步增加高斯混合分量
        HCompV -S train.scp -m -H hmmdefs proto.txt
        HERest -S train.scp -I aligned.mlf -H hmmdefs monophone.list

        # 阶段3：上下文相关问题定义
        # 生成上下文相关标注和问题集
        perl scripts/make_context_label.pl lab/*.lab > full_labels/

        # 阶段4：决策树聚类
        # 使用CART决策树聚合并共享参数
        perl scripts/tree_clustering.pl \
            full_labels/ trees.txt questions.hed

        # 阶段5：参数生成（合成时）
        # 从HMM序列生成最可能声学参数序列
        # 使用MLPG算法产生平滑的参数轨迹
        HVite -S test_labels.scp -H hmms/hmmdefs \
              -w wdnet -o output.mlf vocoder.list

HMM参数合成的训练流程

完整的HTS训练流程可以概括为以下几个步骤：

  1. 数据准备：收集录音数据（通常1-3小时），进行文本标注和音素级的强制对齐，提取声学特征（MFCC或LSP + logF0 + 清/浊音标志 + 能量）。
  2. HMM训练：从均匀初始化开始，使用Baum-Welch算法（Expectation-Maximization，EM算法）迭代估计模型参数。这是一个无监督学习过程，HMM会自动学习每个音素的状态数和每个状态的发射分布。
  3. 决策树聚类：训练上下文相关HMM并进行决策树聚类，生成共享的高斯参数。这一步是可扩展性的关键——它使得模型可以在小数据上工作（通过共享参数避免过拟合），也可以在大数据上工作（通过更细的聚类捕捉更丰富的声学变化）。
  4. 参数生成：合成时，根据输入的文本生成对应的上下文相关HMM序列，然后使用MLPG（Maximum Likelihood Parameter Generation，最大似然参数生成）算法从HMM序列中生成平滑的声学参数轨迹。MLPG不仅要考虑每一帧的静态特征，还要考虑其动态特征（delta和delta-delta系数），从而保证生成的参数在时间上是平滑且自然的。
MLPG算法的一个直观理解：MLPG算法做的事情，类似于"根据模糊的草图描出清晰的线条"。HMM提供了每一帧声学特征的分布（就像美术师画的一排淡淡的参考点），MLPG则在考虑所有帧之间的连续性约束下，找到一条"最可能"且"最平滑"的路径穿过这些分布。没有MLPG，逐帧最大值拼接会产生锯齿状的、不自然的语音——这在HMM合成中是一个决定性的技术细节。

参数合成的优势

相比拼接合成，HMM统计参数合成带来了几个革命性的优势：

  • 小语料即可工作：HMM系统只需要1-2小时的录音数据就可以合成出可用的语音。相比之下，拼接合成至少需要5-10小时才有好的效果。这在数据资源有限的语种（如小语种、方言）上是一个巨大的优势。
  • 声音可变换：通过修改HMM参数——比如改变基频均值（F0平均）和声道缩放因子（vocal tract length normalization）——可以在同一个模型上实现声音的转换。你可以让同一个TTS系统从"男声"变成"女声"，或者把20岁的声音变成60岁的声音。这是拼接合成完全做不到的。
  • 占用空间极小：训练好的HMM模型通常只有几十MB，而拼接合成需要存储数小时的录音数据（通常数百MB到数GB）。这对于嵌入式设备和手机应用来说是一个巨大的优势。
  • 稳定性高：参数合成的方式不存在"拼接失败"的问题——每一次合成都是从一个固定的统计模型中"计算"出来的，不会出现拼接合成那种偶尔的"爆音"或"断片"。结果高度确定，适合对稳定性要求高的商用场景。

"沉闷感"从何而来？

尽管HMM参数合成有这么多的优势，但听过HMM合成语音的人几乎都会有一个共同的感受——声音"闷闷的"，不清脆，有一种挥之不去的浑浊感。这就是HMM合成的标志性问题，通常被称为"过平滑"（Over-smoothing）或"水声"（buzziness）。

这个问题的根本原因有三个层面：

  • 统计平均效应：HMM本质上是在对大量的语音数据进行"平均"建模。决策树聚类会把相似的上下文合并到一个叶子节点中，这些合并后的数据的平均值会丢失很多细节——特别是那些对自然度至关重要的"微结构"。就像你把100张人脸照片叠在一起平均，得到的不再是一张具体的脸，而是一个模糊的人脸轮廓。
  • 声码器失真：MLPG生成的声学参数是平滑的，再用声码器（如MLSA滤波器）将其合成波形，这一逆过程本身就会引入失真。传统的LPC或MLSA声码器在"用参数重建波形"时，会产生一种被称为"buzziness"的金属感——特别是在高频段。
  • 动态特征损失：虽然MLPG尝试通过动态系数来保持参数变化的自然性，但它仍然是基于高斯假设的"最可能"路径。实际上，人类语音的频谱变化是非高斯、非线性的，很多细微的变化规律被高斯平均模型"平滑"掉了。

一个真实的"踩坑"经历：2008年，日本某研究团队在开发中文TTS系统时，使用HTS训练了一个普通话模型。合成的语音在听感测试中虽然MOS只有3.2左右，但已经基本可用。然而当团队把合成语音交给合作方进行"电话客服"场景测试时，发现了一个奇怪的问题：合成的语音在通过电话线路（8kHz采样率）后，清晰度下降得比真人语音要严重得多。分析之后发现，原因在于HMM平滑后的声学参数在频域上的能量分布相对均匀，而电话信道对能量分布敏感，均匀分布的信号经过通道后更容易产生感知上的失真。这个案例说明，参数合成的"过平滑"问题不仅仅是一个声音质量问题，它在特定的传输通道下还会被进一步放大——这是拼接合成和后来的神经网络系统都不曾遇到过的特殊挑战。

HMM统计参数合成在2000年代末到2010年代中期统治了TTS学术界，HTS工具包也是那个时期国际语音合成评测（Blizzard Challenge）中最基本的基准系统。但研究者们逐渐认识到，HMM的两大瓶颈——有限的数据建模能力（高斯假设）和依赖手工特征的局限性——需要一种更强大的模型来突破。此时，一个在图像和语音识别领域已经掀起革命的工具进入了TTS研究者的视野——深度神经网络（Deep Neural Networks, DNNs）。下一章，我们来看DNN如何接过HMM的接力棒，把参数合成的效果推上一个新的台阶。
