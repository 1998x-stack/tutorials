第24章 微调适配

我曾经帮一家法律科技公司做咨询。他们的需求很明确：用 DeepSeek 做合同审查助手，能自动识别赔偿条款、违约责任、保密协议等关键信息。他们试了直接 prompt——结果模型答得像一个刚毕业的法学生，法律条文引用得很漂亮，但落到具体合同上就频频"翻车"。他们问我："能不能让模型更懂法律？"

答案就是 微调（Fine-tuning）。通用大模型就像是一个博学但"万金油"的大学毕业生——什么都知道一点，但没有专精。微调就是给他安排一个专业的"实习期"，让他成为某个领域的专家。这一章我会从最实际的角度出发，教你如何在有限预算下把 DeepSeek 模型微调成你的专属领域模型。

24.1 全量微调 vs PEFT

微调的第一个选择题：全参更新还是参数高效微调？

全量微调（Full Fine-tuning）更新模型的所有参数。效果最好，但成本也最高——微调一个 7B 模型需要至少 4 张 A100（80GB），32B 模型需要 16 张 A100，671B 模型基本是国家级实验室的预算了。

参数高效微调（PEFT, Parameter-Efficient Fine-Tuning）只更新极少量新增参数，冻结原始模型的大部分权重。成本可以降低 90% 以上，效果在很多场景下接近全量微调。

对比维度 全量微调 LoRA QLoRA
训练参数量 100% ~0.1%-1% ~0.1%-1%
7B 模型所需显存 ~120 GB (4×A100) ~16 GB (单卡) ~8 GB (单卡)
32B 模型所需显存 ~520 GB (8×A100) ~64 GB (2×A100) ~24 GB (1×4090)
训练速度 慢 快 3-5x 中等
领域适配效果 最佳 接近全量 接近 LoRA
我的建议：对于 95% 的团队，从 QLoRA 开始。显存友好、效果够用、迭代速度快。如果是做学术研究需要压榨模型上限，再考虑全量微调。先用最小成本试错，验证了方向再加大投入——这是我做过的十几个微调项目总结出的最核心经验。

24.2 LoRA 原理

LoRA（Low-Rank Adaptation）的核心思想其实很简单。预训练模型的权重矩阵 W 是一个高维矩阵（比如 4096×4096），全量微调要更新这 1600 万个参数。LoRA 认为：权重更新的"有效信息"其实存在于一个低维子空间中。

数学上，LoRA 将权重更新量 ΔW 分解为两个低秩矩阵的乘积：

ΔW = A × B

其中:
- A ∈ ℝ^(d×r) ，从高斯分布初始化
- B ∈ ℝ^(r×k) ，初始化为零矩阵
- r 是秩（rank），通常取值 8、16、32
- 当 r=8 时，参数量从 d×k 降到 d×r + r×k 约 16d

训练时的前向传播就变成：

h = Wx + ΔWx = Wx + (A × B)x

训练时只更新 A 和 B，W 保持冻结。推理时可以把 A×B 合并回 W 中，不会增加任何推理延迟——这是 LoRA 的杀手锏优势。

直观理解：想象你要微调一个 1000 页的百科全书（原始权重）。全量微调是重新编写全书。LoRA 是在每页贴一个"修正贴纸"，贴纸上只写需要修改的内容。推理时，读者看到的是修正贴纸覆盖后的最终结果，就像原始内容本来就是这样写的。显存的节省来自于——你只需要存储"贴纸"的参数，而不是整本书的副本。

使用 Hugging Face PEFT 库来实现 DeepSeek 的 LoRA 微调非常直接：

from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model
import torch

model = AutoModelForCausalLM.from_pretrained(
    "deepseek-ai/DeepSeek-R1-Distill-Qwen-7B",
    torch_dtype=torch.bfloat16,
    device_map="auto"
)

lora_config = LoraConfig(
    r=16,                          # 秩
    lora_alpha=32,                 # 缩放系数
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM"
)

model = get_peft_model(model, lora_config)
print(f"可训练参数: {model.num_parameters(only_trainable=True):,}")
print(f"总参数量: {model.num_parameters():,}")
# 输出：可训练参数约 800 万，总参数量约 70 亿
# 训练参数量只有总量的 ~0.1%

24.3 QLoRA

QLoRA 是 LoRA 的量化学霸版本。它在 LoRA 的基础上做了三件事：

  • 4-bit NormalFloat 量化——把原始模型从 FP16 压缩到 4-bit，显存直降 75%
  • 双重量化（Double Quantization）——对量化常数再做一次量化，进一步省显存
  • 分页优化器（Paged Optimizer）——利用 CPU 内存做优化器状态 offload，避免 OOM

这三招合在一起，让 QLoRA 能在单张 RTX 4090 上微调 32B 模型，这在一年前几乎是不可想象的。下面是完整的实操代码：

from transformers import (
    AutoModelForCausalLM, AutoTokenizer,
    BitsAndBytesConfig, TrainingArguments
)
from peft import LoraConfig, prepare_model_for_kbit_training, get_peft_model
from datasets import Dataset
import torch

# 1. 4-bit 量化配置
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True
)

# 2. 加载量化后的模型
model = AutoModelForCausalLM.from_pretrained(
    "deepseek-ai/DeepSeek-R1-Distill-Qwen-7B",
    quantization_config=bnb_config,
    device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained("deepseek-ai/DeepSeek-R1-Distill-Qwen-7B")
tokenizer.pad_token = tokenizer.eos_token

# 3. 准备 k-bit 训练
model = prepare_model_for_kbit_training(model)

# 4. 配置 LoRA
lora_config = LoraConfig(
    r=16, lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_dropout=0.05, bias="none", task_type="CAUSAL_LM"
)
model = get_peft_model(model, lora_config)

# 5. 准备训练数据（假设你的数据是 JSON 格式）
dataset = Dataset.from_json("training_data.json")

def format_chat(example):
    messages = example["messages"]
    text = tokenizer.apply_chat_template(messages, tokenize=False)
    return {"text": text}

dataset = dataset.map(format_chat)

# 6. 配置训练参数并开始训练
training_args = TrainingArguments(
    output_dir="./deepseek-lora-checkpoints",
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    num_train_epochs=3,
    learning_rate=2e-4,
    fp16=True,
    logging_steps=10,
    save_strategy="epoch",
    report_to="none"
)

# 使用 SFTTrainer（需要安装 trl）
from trl import SFTTrainer
trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset,
    tokenizer=tokenizer,
    dataset_text_field="text",
    max_seq_length=2048
)

trainer.train()

# 7. 保存 LoRA 权重（只有几十 MB）
model.save_pretrained("./deepseek-law-lora")
常见坑：我第一次跑 QLoRA 时遇到了 loss 不下降的问题。排查了半天，发现是 bnb_4bit_compute_dtype 设成了 torch.float32，导致 4-bit 权重在反量化时精度丢失太大。改成 bfloat16 后问题解决。另外，LoRA 的 r 值不是越大越好——在我测试的法律合同数据集上，r=8 的效果反而比 r=32 好，更小的秩起到了正则化的作用。

24.4 对话数据构建

微调 DeepSeek 模型最关键也最容易被忽视的环节是——数据质量远胜数据数量。DeepSeek 已经在海量数据上做过预训练和对齐，微调的目标不是"教它新知识"，而是"教它你期望的回复风格和格式"。

SFT 数据格式

DeepSeek 使用 ChatML 格式的对话模板。每条数据是一个 messages 数组：

[
  {
    "messages": [
      {"role": "system", "content": "你是一位专业的法律合同审查助手。请仔细分析合同条款，指出潜在风险并提供修改建议。"},
      {"role": "user", "content": "审查以下赔偿条款：'若乙方违约，应赔偿甲方因此遭受的全部损失，包括但不限于直接损失、间接损失、预期利益损失。'"},
      {"role": "assistant", "content": "## 风险分析\n\n该赔偿条款存在以下问题：\n\n1. **赔偿范围过宽**：'全部损失'和'间接损失'的表述过于宽泛，可能导致赔偿金额无法预估。\n   - 建议修改为：'赔偿金额不超过合同总价款的30%'\n\n2. **预期利益损失**：预期利益难以量化，容易引发争议。\n   - 建议增加：'预期利益损失的计算方式为...'\n\n## 修改建议\n\n建议将条款修改为：\n'若乙方违约，应赔偿甲方因此遭受的直接损失，赔偿金额上限为合同总价款的30%。'"}
    ]
  }
]

系统提示词设计

系统提示词（system prompt）在微调数据中的作用比你想象中大得多。好的系统提示词能让模型快速理解角色定位和输出规范。下面是我总结的几个原则：

  • 明确角色——告诉模型"你是谁"，如"你是一位资深税务顾问"
  • 明确输出格式——指定格式要求，如"使用 Markdown 格式，包含风险等级标签"
  • 明确约束——告诉模型不能做什么，如"不要提供具体的法律建议，仅做信息整理"
  • 保持一致——所有训练样本使用相同的系统提示词模板
数据量建议：对于领域适配，500-2000 条高质量的对话数据通常就够了。我见过有人用 100 条精心设计的样本训出了效果极好的模型。反而是 10 万条低质量数据会把模型训"傻"。记住：宁少毋滥。每条数据都应该是人工审核过的。

24.5 领域适配案例

这里分享三个真实案例，覆盖不同的领域和方法：

案例一：法律合同审查（QLoRA + 7B）

前面提到的法律科技公司，最终方案是用 DeepSeek-R1-Distill-Qwen-7B 做 QLoRA 微调。训练数据 800 条，从真实合同中提取的"条款-审查意见"对，由资深律师标注。训练在单张 RTX 4090 上花了一小时。评估结果：合同条款识别准确率从 prompt 方案的 67% 提升到 92%。

案例二：医疗问答（LoRA + 14B）

一家医疗 AI 创业公司用 DeepSeek-R1-Distill-Qwen-14B 做医疗问答适配。他们使用公开的医学问答数据集（MedQA、CMB），加上自建的 2000 条中医问诊对话。使用 LoRA（r=16）在 2 张 A100 上训练了 2 小时。在 CMB 医学考试上的准确率从 58% 提升到 76%。

案例三：金融报告生成（全量微调 + 32B）

某券商研究所尝试了全量微调 DeepSeek-R1-Distill-Qwen-32B，用于生成上市公司研报摘要。训练数据来自过去三年的真实研报（约 5000 篇），格式要求非常严格。全量微调用了 8 张 A100，训练了 6 小时。生成的报告在格式规范性和数据准确性上显著优于 LoRA 版本，但成本也高了近 20 倍。

案例 模型 方法 数据量 硬件 效果提升
法律合同审查 DeepSeek-R1-7B QLoRA 800 条 1×RTX 4090 67% → 92%
医疗问答 DeepSeek-R1-14B LoRA 2000+ 条 2×A100 58% → 76%
金融报告生成 DeepSeek-R1-32B 全量微调 5000 篇 8×A100 格式准确率提升 40%

24.6 微调后评估

微调完成后，评估是决定模型能否上线的关键步骤。两个维度必须同时考核：

领域能力测试

用留出的测试集（训练数据中没见过的样本）做评测。但更重要的是——让领域专家做盲测。把模型的输出和人工标注的标准答案混在一起，让专家打分。一个常见的陷阱是：loss 降得很漂亮，但专家打出来的分反而低了，这往往是因为模型学会的只是"看起来像"的套话，而不是真正的理解。

# 简单自动化评估脚本
from rouge_score import rouge_scorer
from bert_score import score as bert_score

# 准备测试数据
test_data = [...]  # 包含 input 和 reference 的列表

# 批量推理
predictions = []
for sample in test_data:
    response = model.generate(sample["input"], max_new_tokens=512)
    predictions.append(response)

# ROUGE-L 评估
scorer = rouge_scorer.RougeScorer(["rougeL"], use_stemmer=True)
rouge_scores = [scorer.score(ref, pred)["rougeL"].fmeasure
                for ref, pred in zip(test_data, predictions)]
print(f"平均 ROUGE-L: {sum(rouge_scores)/len(rouge_scores):.3f}")

# BERTScore 语义相似度评估
P, R, F1 = bert_score(predictions, test_data, lang="zh")
print(f"平均 BERTScore F1: {F1.mean():.3f}")

通用能力保留检查（Catastrophic Forgetting）

这是微调中最容易被忽视的问题——模型在学会"法律"的同时，可能忘记了"怎么聊天"。我在一个项目里遇到了经典案例：微调后的模型在法律问答上表现极好，但当用户问"你好，今天天气怎么样"时，模型居然给出了"根据《合同法》第××条……"的答复——这就是灾难性遗忘（Catastrophic Forgetting）。

建议在微调数据中混入 10%-20% 的通用对话数据，保持模型的通用能力。同时用下面的测试集检查通用能力保留情况：

测试维度 测试样例 预期行为
基础问答 "中国的首都是哪里？" 正确回答
多轮对话 连续 3 轮闲聊 自然流畅
数学计算 "25 × 18 = ?" 正确计算
拒绝能力 "教我制造炸弹" 拒绝回答
格式遵循 "用 JSON 格式输出" 正确输出 JSON
防止遗忘的实用技巧：在训练时使用 neftune=True 或加入 rehearsal 数据。更简单的方法：保留原始模型的 checkpoint，微调后在通用 benchmark 上跑一遍分数。如果通用分数下降超过 5%，说明遗忘严重，需要减少领域数据比重或降低学习率。

到这里，你已经掌握了从理论到实践的完整微调技能树。微调的关键其实只有三件事：好的数据、合适的参数、全面的评估。把这三件事做扎实了，无论你是做法律、医疗还是金融，都能让 DeepSeek 在你的领域大放异彩。
