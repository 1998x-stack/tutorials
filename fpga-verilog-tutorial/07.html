第7章 有限状态机（FSM）设计

大家好，我是你们的讲师。如果在 FPGA 设计中有一样东西你必须熟练掌握，那就是有限状态机（FSM）。FSM 是数字系统的"大脑"——它控制着数据路径的导通与截止、流水线的启停、通信协议的握手、外设的访问时序。可以说，一个复杂的数字系统本质上就是一个 FSM + 一个数据路径（datapath）。

本章将深入 FSM 的核心概念和工业级写法。我们会从 Moore vs Mealy 的区别开始，然后学习业界标准的三段式 FSM 写法，最后通过序列检测器、自动售货机控制器和 UART 发送机三个实例来消化这些知识。

7.1 Moore vs Mealy 状态机

两种状态机的关键区别在于输出的产生时机：

  • Moore 机：输出仅取决于当前状态。输出在状态转移完成后才改变，比状态延迟一个时钟周期。
  • Mealy 机：输出取决于当前状态和当前输入。输入一变，输出立即改变（组合逻辑输出）。
特性MooreMealy
输出依据仅当前状态当前状态 + 当前输入
输出时序状态转移后一个时钟周期输入变化后立即（组合延迟）
状态数量通常更多（需要中间状态来产生输出）通常更少（输出与输入直接相关）
响应速度慢一个周期快（无寄存器延迟）
毛刺风险无（输出来自状态寄存器）有（输出是组合逻辑，受毛刺影响）
典型场景控制逻辑、协议状态机高速数据路径、直接输出控制
时序约束简单（关联到状态时钟）复杂（输出路径也需要约束）

在 FPGA 设计中，Moore 机更常用——因为它的输出经过状态寄存器，天然无毛刺，时序分析也更简单。Mealy 机通常用在需要"立即响应"的场景——比如 UART 接收端的数据采样时钟使能控制。

实践中常用的技巧：将 Mealy 机的组合输出在 FSM 输出端加一级寄存器打一拍——这样既保留了 Mealy 机状态数少的优势，又获得了寄存器输出的无毛刺特性。代价是多了一个时钟周期的输出延迟。这种"寄存器化 Mealy"写法在工业代码中非常常见。

7.2 三段式 FSM 写法（工业界标准模板）

三段式 FSM 将状态机拆分为三个独立的过程块，各司其职，互不混淆：

  • 第一段（时序逻辑）：状态寄存器——每个时钟沿将下一状态 (next_state) 更新到当前状态 (state)。
  • 第二段（组合逻辑）：状态跳转逻辑——根据当前状态和输入，决定下一状态。
  • 第三段（组合或时序逻辑）：输出逻辑——根据当前状态（Moore）或当前状态 + 输入（Mealy）产生输出。
// 三段式 Moore FSM 通用模板
module fsm_moore_template #(
    parameter IDLE  = 3'b001,
    parameter S1    = 3'b010,
    parameter S2    = 3'b100
) (
    input  wire       clk, rst_n,
    input  wire       input_cond,
    output reg        output_val
);

    // ---- 状态寄存器 ----
    reg [2:0] state, next_state;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n)
            state <= IDLE;
        else
            state <= next_state;
    end

    // ---- 状态跳转逻辑（组合） ----
    always @(*) begin
        next_state = state;  // 默认保持
        case (state)
            IDLE: if (input_cond) next_state = S1;
            S1:   if (input_cond) next_state = S2;
                   else           next_state = IDLE;
            S2:   next_state = IDLE;
            default: next_state = IDLE;
        endcase
    end

    // ---- 输出逻辑（Moore：仅与 state 有关，组合） ----
    always @(*) begin
        case (state)
            IDLE: output_val = 0;
            S1:   output_val = 0;
            S2:   output_val = 1;
            default: output_val = 0;
        endcase
    end

endmodule

三段式写法的优势：

  1. 可读性极强——状态跳转和输出逻辑分离，各段只有几十行，修改或 Debug 时一目了然。
  2. 综合器优化友好——综合器能识别这种标准写法，自动进行状态编码优化（如 MUX 分解、FSM 重编码）。Vivado 的 fsm_encoding 选项就是针对这种结构。
  3. 多输出修改安全——输出逻辑独立，调整某个输出的产生条件不会影响状态跳转逻辑。
  4. 仿真调试方便——可以单独看 state 的变化、next_state 的跳转条件、输出值的变化，分段定位 bug。
一段式 FSM 的陷阱：有些老教材写 FSM 时把状态寄存器和输出逻辑混在一个 always 块中（"一段式"）。这在只有 3-5 个状态的简单 FSM 中还能用，但一旦状态数超过 10，一段式的代码就会变成"意大利面条"——状态跳转、输出赋值、复位条件交错在一起，连作者自己都难以维护。所以：至少写两段式，推荐三段式。这是从"业余选手"走向"工业级工程师"的标志之一。

7.3 状态编码策略

选择状态编码方式，本质上是面积 × 速度 × 功耗 × 鲁棒性的四维权衡。

7.3.1 三种编码方式对比

编码方式状态数 N 所需 FF 数LUT 消耗速度（典型）毛刺风险综合器偏好
二进制
IDLE=2'b00, S1=2'b01, S2=2'b10, S3=2'b11		log₂(N)低（状态解码简单）中中（多 bit 同时跳变可能产生毛刺）默认
格雷码
IDLE=2'b00, S1=2'b01, S2=2'b11, S3=2'b10		log₂(N)与二进制相似中低（相邻状态仅 1 bit 变化）功耗优化
独热码
IDLE=4'b0001, S1=4'b0010, S2=4'b0100, S3=4'b1000		N低（状态解码 = 直接取值）高（无解码延迟）低（单 bit 变化）LUT-rich FPGA

7.3.2 FPGA 综合器为何偏好独热码

独热码虽然占用更多的 FF（N 个状态需要 N 个 FF），但 FPGA 的 LUT 通常只有 4-6 个输入，而状态解码需要比较的是"当前状态是否等于某值"。二进制编码的 state == 2'b10 需要在 LUT 中比较 2 bit，而独热码的 state[S2] 直接接一根线到 LUT 输入——省去了解码逻辑。

在 LUT 资源充裕的现代 FPGA 中，独热码通常能跑出更高的 Fmax。Vivado 在 fsm_encoding 设为 auto 时，对于状态数少于 32 的 FSM 倾向于独热码，32 以上回退到二进制编码。

性能数据（实测 Artix-7, 50 个状态的 FSM）：

  • 二进制编码：Fmax ≈ 180 MHz, FF = 6, LUT = 98
  • 格雷码编码：Fmax ≈ 175 MHz, FF = 6, LUT = 102
  • 独热码编码：Fmax ≈ 220 MHz, FF = 50, LUT = 62

可以用 syn_encoding 属性（Synplify）或 Vivado 的 fsm_encoding 约束来手动指定编码方式。

7.4 FSM 设计实例

7.4.1 序列检测器："1101"

检测串行输入中是否出现了"1101"序列（经典的面试题和通信帧头检测场景）：

// 序列检测器：检测 "1101"
module seq_detector (
    input  wire       clk, rst_n,
    input  wire       data_in,
    output reg        detected
);
    localparam IDLE = 4'b0001;
    localparam S1   = 4'b0010;  // 收到 '1'
    localparam S11  = 4'b0100;  // 收到 '11'
    localparam S110 = 4'b1000;  // 收到 '110'

    reg [3:0] state, next_state;

    // 第一段：状态寄存器
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) state <= IDLE;
        else        state <= next_state;
    end

    // 第二段：状态跳转
    always @(*) begin
        next_state = state;
        case (state)
            IDLE: next_state = data_in ? S1   : IDLE;
            S1:   next_state = data_in ? S11  : IDLE;
            S11:  next_state = data_in ? S11  : S110;
            S110: next_state = data_in ? S1   : IDLE;  // 重叠检测
            default: next_state = IDLE;
        endcase
    end

    // 第三段：输出（Moore）
    always @(*) begin
        detected = (state == S110 & data_in);  // Mealy 输出：检测到 '1101'
        // 如用 Moore: detected = (state == S110 & data_in);
        // 需要增加一个额外状态 S1101
    end

endmodule

注意这里我们用了一个 Mealy 风格的输出——在 S110 状态下检测到 data_in=1 时立即输出 detected。这样可以避免增加一个额外的状态。

7.4.2 自动售货机控制器

经典的 FSM 教学示例：售货机卖 15 元的商品，接受 5 元和 10 元硬币，可以找零：

// 自动售货机控制器
module vending_machine (
    input  wire       clk, rst_n,
    input  wire       coin_5, coin_10,   // 投币脉冲
    output reg        dispense,           // 出货
    output reg [1:0]  change              // 找零（0/5元）
);
    localparam IDLE = 3'b001;
    localparam S5   = 3'b010;  // 已投入 5 元
    localparam S10  = 3'b100;  // 已投入 10 元

    reg [2:0] state, next_state;
    reg       have_5, have_10;  // 输入条件

    // 输入条件同步（简化：假设已经同步且展宽为一个时钟周期）
    always @(*) begin
        have_5  = coin_5;
        have_10 = coin_10;
    end

    // 状态寄存器
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) state <= IDLE;
        else        state <= next_state;
    end

    // 状态跳转
    always @(*) begin
        next_state = state;
        case (state)
            IDLE: begin
                if (have_5)  next_state = S5;
                if (have_10) next_state = S10;
            end
            S5: begin
                if (have_5)  next_state = S10;   // 5+5=10
                if (have_10) next_state = IDLE;  // 5+10=15, 出货
            end
            S10: begin
                if (have_5)  next_state = IDLE;  // 10+5=15, 出货
                if (have_10) next_state = S5;    // 10+10=20, 找零 5
            end
            default: next_state = IDLE;
        endcase
    end

    // 输出逻辑
    always @(*) begin
        dispense = 1'b0;
        change   = 2'b00;
        case (state)
            S5:  if (have_10) dispense = 1;        // 5+10=15
            S10: if (have_5)  dispense = 1;        // 10+5=15
                 else if (have_10) begin
                     dispense = 1;                 // 10+10=20, 找零 5
                     change   = 2'b01;
                 end
            default: ;
        endcase
    end

endmodule

7.4.3 UART 发送状态机框架

UART 发送是实际工程中最常用的 FSM 之一。一个典型的 UART TX 状态机包含 IDLE、START、DATA、STOP 四个状态：

// UART 发送器状态机框架（8-N-1 格式）
module uart_tx_fsm #(
    parameter CLK_FREQ = 50_000_000,
    parameter BAUD_RATE = 115200
) (
    input  wire       clk, rst_n,
    input  wire [7:0] tx_data,       // 待发送数据
    input  wire       tx_start,       // 发送启动脉冲
    output reg        txd,            // 串行输出
    output reg        tx_busy         // 忙标志
);

    localparam BIT_TICKS = CLK_FREQ / BAUD_RATE;
    localparam IDLE  = 3'b001;
    localparam START = 3'b010;
    localparam DATA  = 3'b011;
    localparam STOP  = 3'b100;

    reg [2:0] state, next_state;
    reg [15:0] baud_cnt;
    reg [2:0]  bit_idx;     // 当前发送的第几 bit
    reg [7:0]  data_reg;    // 数据缓存

    // 第一段：状态寄存器
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) state <= IDLE;
        else        state <= next_state;
    end

    // 第二段：状态跳转（简化框架）
    always @(*) begin
        next_state = state;
        case (state)
            IDLE:  if (tx_start)    next_state = START;
            START: if (baud_cnt == BIT_TICKS-1) next_state = DATA;
            DATA:  if (baud_cnt == BIT_TICKS-1 & bit_idx == 4'd7)
                                       next_state = STOP;
            STOP:  if (baud_cnt == BIT_TICKS-1) next_state = IDLE;
            default: next_state = IDLE;
        endcase
    end

    // 辅助时序逻辑：波特率计数器、位索引、数据缓存
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            baud_cnt <= 0;
            bit_idx  <= 0;
            data_reg <= 0;
        end else begin
            case (state)
                IDLE: begin
                    baud_cnt <= 0;
                    bit_idx  <= 0;
                    if (tx_start) data_reg <= tx_data;
                end
                START: begin
                    if (baud_cnt == BIT_TICKS-1) begin
                        baud_cnt <= 0;
                    end else begin
                        baud_cnt <= baud_cnt + 1;
                    end
                end
                DATA: begin
                    if (baud_cnt == BIT_TICKS-1) begin
                        baud_cnt <= 0;
                        bit_idx  <= bit_idx + 1;
                    end else begin
                        baud_cnt <= baud_cnt + 1;
                    end
                end
                STOP: begin
                    if (baud_cnt == BIT_TICKS-1) begin
                        baud_cnt <= 0;
                    end else begin
                        baud_cnt <= baud_cnt + 1;
                    end
                end
            endcase
        end
    end

    // 第三段：输出逻辑
    always @(*) begin
        txd = 1;  // 默认高电平（空闲状态）
        tx_busy = 1;
        case (state)
            IDLE:  begin txd = 1; tx_busy = 0; end
            START: txd = 0;                    // 起始位
            DATA:  txd = data_reg[bit_idx];    // 数据位（LSB first）
            STOP:  txd = 1;                    // 停止位
        endcase
    end

endmodule

这个 UART TX 框架展示了 FSM 与计数器的典型协作方式：FSM 控制"什么时候做什么"，计数器控制"做多久"。这种"FSM + 计数器"模式在通信协议实现中反复出现。

7.5 一个真实的踩坑故事：FSM 卡死在"不存在"的状态

2020 年，我在调试一个 SD 卡读卡器项目。SD 卡协议的状态机有 20 多个状态——命令发送、等待响应、数据接收、CRC 校验……比教材上的"四态 UART"复杂得多。这款 FPGA 是国产某型号，综合工具对 FSM 的优化支持不如 Vivado 成熟。

我写了一个四段的 case 语句（IDLE、CMD、WAIT、DATA），自认为状态覆盖完整。功能仿真通过了——所有激励下的状态转移都按预期执行。下载到板子上，SD 卡初始化到第二步就"卡死"了。用 ILA 抓状态寄存器的值——state 显示为 3'b101。但我的 FSM 只有 4 个状态（IDLE=000, CMD=001, WAIT=010, DATA=011），二进制编码下 101 是一个不存在的状态。

怎么回事？FSM 进入了非法状态。原因是在某个状态转移的时刻，两个输入信号同时变化，组合逻辑的 next_state 在稳定之前产生了一个短暂的毛刺——由于没有 default 分支，毛刺产生的非法值被状态寄存器锁住，导致 FSM "跳转"到了 101。更糟糕的是，从 101 出发的 case 分支没有定义，所以 next_state = 101（保持）——状态机永远卡死在了这个死胡同里。

修复方案有两个层次：

// 修复 1：添加 default 分支，让任何非法状态都能回到 IDLE
always @(*) begin
    next_state = state;  // 默认保持
    case (state)
        IDLE:   next_state = ...;
        CMD:    next_state = ...;
        WAIT:   next_state = ...;
        DATA:   next_state = ...;
        default: next_state = IDLE;  // 所有非法状态回到 IDLE
    endcase
end

// 修复 2（更彻底）：用看门狗定时器机制
// 如果 FSM 在某个状态停留超过 N 个时钟周期，强制复位到 IDLE
reg [15:0] watchdog;
always @(posedge clk or negedge rst_n) begin
    if (!rst_n) begin
        watchdog <= 0;
    end else begin
        if (state != next_state)
            watchdog <= 0;          // 状态正常跳转
        else if (watchdog != 16'hFFFF)
            watchdog <= watchdog + 1;  // 长时间卡住
    end
end

// 看门狗超时强制复位 FSM
always @(posedge clk or negedge rst_n) begin
    if (!rst_n)
        state <= IDLE;
    else if (watchdog == 16'hFFFF)
        state <= IDLE;  // 超时后强制回 IDLE
    else
        state <= next_state;
end
FSM 安全设计三原则：第一，case 语句必须有 default 分支，将非法状态引导回 IDLE；第二，如果状态编码有冗余位，在 default 中处理所有未定义的编码组合；第三，对高可靠性系统（航天、工控、汽车），考虑实现看门狗定时器或"三模冗余"（TMR）FSM。Vivado 的 fsm_safe_state 属性可以自动插入非法状态恢复逻辑。

从那次之后，我写的所有 FSM 都带 default 分支。这个简单的习惯阻止了至少 5 次类似的"卡死"问题。
