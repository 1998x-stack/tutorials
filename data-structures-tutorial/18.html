开篇：一个"聪明反被聪明误"的故事

有一个经典的面试题：如果你面前有 1 元、2 元、5 元三种面额的硬币，要凑出 11 元，最少需要几枚硬币？你几乎本能地会回答：先拿 5 元，再拿 5 元，再拿 1 元——共 3 枚。这个答案是对的。

现在把硬币换成 1 元、3 元、4 元，要凑出 6 元。用同样的思路：先拿 4 元，再拿 1 元，再拿 1 元——共 3 枚。但这个答案是错的。最优解是拿两个 3 元，只需要 2 枚。

这就是贪心（Greedy）的陷阱——它让你"每次选当前看起来最好的"，在 1/2/5 的硬币体系中行得通，但在 1/3/4 的体系中就翻车了。什么时候贪心有效？什么时候不管用？这就是本章要回答的核心问题。

贪心的本质——"每一步选当前最优，期望最终全局最优"

贪心算法（Greedy Algorithm）的思路极其简单：在做每一步选择时，都选择当前看起来最优的选项，希望通过一系列局部最优选择，最终得到全局最优解。

这和 DP（动态规划）有什么区别？我们用一个对比表格来说明：

维度 贪心（Greedy） 动态规划（DP）
决策方式 只看眼前，选当前最优 考虑所有可能的路径
子问题依赖 当前选择不影响后续选择空间 当前选择影响后续子问题的状态
正确性 需要严格证明（贪心选择性质） 只要转移方程正确就成立
效率 通常 O(n) 或 O(n log n)，极快 通常 O(n²) 或更高，较慢
典型问题 区间调度、Huffman 编码、Dijkstra 背包、编辑距离、LCS

简单来说：贪心是一锤子买卖，DP 是货比三家。贪心快，但只对特定问题有效；DP 慢但通用。

什么时候贪心有效？

一个贪心算法正确，必须满足两个性质：

  • 贪心选择性质（Greedy Choice Property）——全局最优解可以通过一系列局部最优选择来达到。也就是说，你"每次都选最大的那个"不会把未来的路堵死。
  • 最优子结构（Optimal Substructure）——一个问题的全局最优解包含了它的子问题的最优解。这和 DP 的最优子结构是同一个概念。

判断贪心是否有效，有时并不容易。但有一个实用的"探测法"——先假设贪心有效，写一个贪心解法和一个暴力/DP解法（小规模数据），对比两者的结果。如果所有测试用例都通过，贪心大概率是对的；如果发现反例，就知道不能用贪心。这种方法在面试中非常实用——你可以先提出贪心思路，再用一个例子验证。

找零钱问题的正反例

我们用找零钱这道题，讲清楚什么情况下贪心有效、什么情况下无效。

硬币体系 凑 11 元的贪心结果 实际最优解 贪心有效？
1, 2, 5 5+5+1 = 3 枚 5+5+1 = 3 枚 ✅ 有效
1, 3, 4 4+1+1 = 3 枚（凑 6 元） 3+3 = 2 枚 ❌ 无效

为什么 1/2/5 有效而 1/3/4 无效？因为 1/2/5 体系中，任意大面额都是小面额的整数倍（倍数性），而 1/3/4 不是——3 和 4 没有倍数关系，导致"拿 4 块"可能不是最优的。

贪心的"试金石"：问自己一个问题——"如果我选了当前最大的那个，后面会不会因为我少了一个更大的而被迫用很多小的来弥补？"如果答案是"不会"，贪心可能有效；如果答案是"会"，那贪心就翻车了。

区间调度

区间调度是贪心算法最经典的舞台。核心思想：给你一堆区间，选出尽可能多的互不重叠的区间。

活动选择问题——按结束时间排序

你有 n 个活动，每个活动有开始时间和结束时间 [start, end]。你同一时间只能参加一个活动，问最多能参加多少个？

def activity_selection(activities):
    """活动选择：贪心策略——按结束时间最早排序"""
    # activities: [(start1, end1), (start2, end2), ...]
    # 按结束时间升序排序
    sorted_acts = sorted(activities, key=lambda x: x[1])

    count = 0
    last_end = 0

    for start, end in sorted_acts:
        if start >= last_end:
            # 当前活动可以参加
            count += 1
            last_end = end

    return count

activities = [(1, 4), (3, 5), (0, 6), (5, 7), (3, 8), (5, 9),
              (6, 10), (8, 11), (8, 12), (2, 13), (12, 14)]
print(activity_selection(activities))  # 4

为什么按结束时间最早排序是对的？因为结束时间越早，留给后面的活动的时间就越多。这个证明非常直观——如果有两个活动，一个早点结束一个晚点结束，选早结束的那个肯定不吃亏。

无重叠区间（LeetCode 435）

问题：给定一个区间的集合，移除最少数量的区间，使剩下的区间互不重叠。

def erase_overlap_intervals(intervals):
    """无重叠区间：求最少移除多少个区间"""
    if not intervals:
        return 0

    # 按结束时间排序
    intervals.sort(key=lambda x: x[1])

    count = 1  # 保留的区间数，至少有一个
    last_end = intervals[0][1]

    for i in range(1, len(intervals)):
        start, end = intervals[i]
        if start >= last_end:
            # 不重叠，保留
            count += 1
            last_end = end
        # 重叠的话，移除当前区间（保留结束时间更早的）

    return len(intervals) - count

print(erase_overlap_intervals([[1,2],[2,3],[3,4],[1,3]]))  # 1（移除 [1,3]）

这道题是"活动选择"的逆向版——最大保留 = 活动选择的结果，最少移除 = 总数 - 最大保留。

区间贪心的三种排序策略：
  • 按结束时间排序——最常见的策略，适用于求"最多不重叠区间"
  • 按开始时间排序——适用于某些区间合并问题
  • 按区间长度排序——较少用，但在特定问题中有效
面试中先试按结束时间排序，如果不行再换其他策略。

Huffman 编码

Huffman 编码是贪心算法在数据压缩中的经典应用。它的目标是：给定一组字符及其出现频率，为每个字符设计一个二进制编码，使得编码后的总长度最短。

Huffman 编码的核心思想是：出现频率越高的字符，用越短的编码。这个思想本身就是贪心的——每次都把频率最低的两个字符合并。

import heapq

class HuffmanNode:
    def __init__(self, char=None, freq=0, left=None, right=None):
        self.char = char
        self.freq = freq
        self.left = left
        self.right = right

    def __lt__(self, other):
        return self.freq < other.freq


def build_huffman_tree(chars, freqs):
    """构建 Huffman 树：使用最小堆"""
    heap = [HuffmanNode(char=c, freq=f) for c, f in zip(chars, freqs)]
    heapq.heapify(heap)

    # 贪心策略：每次合并频率最小的两个节点
    while len(heap) > 1:
        left = heapq.heappop(heap)   # 最小的
        right = heapq.heappop(heap)  # 第二小的
        merged = HuffmanNode(freq=left.freq + right.freq,
                             left=left, right=right)
        heapq.heappush(heap, merged)

    return heap[0]


def generate_codes(node, code="", codes=None):
    """生成每个字符的 Huffman 编码"""
    if codes is None:
        codes = {}
    if node.char is not None:  # 叶节点
        codes[node.char] = code or "0"
    else:
        generate_codes(node.left, code + "0", codes)
        generate_codes(node.right, code + "1", codes)
    return codes


# 测试
chars = ['a', 'b', 'c', 'd', 'e', 'f']
freqs = [5, 9, 12, 13, 16, 45]
root = build_huffman_tree(chars, freqs)
codes = generate_codes(root)
for char, code in sorted(codes.items()):
    print(f"'{char}': {code}")
# 输出类似：
# 'a': 1100, 'b': 1101, 'c': 100, 'd': 101, 'e': 111, 'f': 0

Huffman 编码保证了两件事：

  • 前缀编码（Prefix Code）——没有一个编码是另一个编码的前缀，保证了解码的唯一性。
  • 最短加权路径——所有字符的编码长度 × 频率的和最小。这个和正是 Huffman 树的带权路径长度（WPL）。

我们可以计算一下压缩率：假设每个字符原来用 8 位 ASCII 编码，总长度为 (5+9+12+13+16+45)×8 = 800 位。用 Huffman 编码后，总长度为 5×4 + 9×4 + 12×3 + 13×3 + 16×3 + 45×1 = 224 位。压缩率 = 224/800 = 72%，省了超过三分之二的空间！

跳跃游戏

LeetCode 55：给定一个非负整数数组 nums，你最初位于第一个下标。数组中的每个元素代表你在该位置可以跳跃的最大长度。判断你是否能够到达最后一个下标。

贪心解法（最优）

def can_jump_greedy(nums):
    """跳跃游戏：贪心解法 O(n)"""
    max_reach = 0  # 当前能到达的最远位置

    for i in range(len(nums)):
        if i > max_reach:
            return False  # 到不了当前位置
        max_reach = max(max_reach, i + nums[i])
        if max_reach >= len(nums) - 1:
            return True  # 已经能到达终点，提前退出

    return True

print(can_jump_greedy([2, 3, 1, 1, 4]))  # True
print(can_jump_greedy([3, 2, 1, 0, 4]))  # False

贪心解法的时间复杂度是 O(n)，空间复杂度是 O(1)。核心思想非常巧妙：我们不需要决定具体"跳到哪里"，只需要维护一个"最远能到达的位置"。如果当前能到达的最远位置 >= 数组长度 - 1，就说明可以到达终点。

DP 解法（对比）

def can_jump_dp(nums):
    """跳跃游戏：DP 解法 O(n²) —— 慢但思路直观"""
    n = len(nums)
    # dp[i] = 是否能到达位置 i
    dp = [False] * n
    dp[0] = True

    for i in range(1, n):
        for j in range(i):
            if dp[j] and j + nums[j] >= i:
                dp[i] = True
                break

    return dp[n - 1]

# 对比：n=10000 时，贪心 O(n) 秒出，DP O(n²) 要跑好几秒
# 这就是为什么好的算法设计如此重要

DP 解法需要检查每个位置之前的所有位置，时间复杂度 O(n²)。而贪心解法只需要遍历一次，时间复杂度 O(n)。为什么贪心在这里有效？因为跳跃游戏的"贪心选择性质"成立——每次维护最大的可达范围，这个选择不会让后续的决策变差。如果你能跳到更远的位置，你当然也可以选择跳到一个更近的位置——所以"跳最远"永远不亏。

贪心 vs DP 的选择决策表：
场景 推荐算法 原因
每一步的选择不影响后续 贪心 可以直接选局部最优
当前选择影响后续状态 DP 需要记录所有可能的路径
需要枚举所有解 回溯 贪心和 DP 都不适用
问题有"无后效性" 贪心或 DP 看能否证明贪心选择性质
无后效性（No After-effect）是一个重要的概念：当前状态确定后，后续决策只关心当前状态的值，不关心"是怎么到达这个状态的"。如果一个问题有后效性，贪心和 DP 都不好用。
一个实用建议：如果你不确定某个问题能不能用贪心，先用 DP 写一个正确的解法（小数据验证），再用贪心写一个快的解法。如果两者在所有测试用例上结果一致，贪心大概率是对的。LeetCode 上很多题既能用 DP 也能用贪心，但贪心的速度通常快几个数量级。

贪心常见题型

总结一下最常见的贪心题型：

题型 贪心策略 经典题目
区间贪心 按结束时间排序，选最早结束的 活动选择、无重叠区间(435)、用最少箭射爆气球(452)
分配贪心 排序后，小的配小的，或大的配大的 分发饼干(455)、两个数组的最小乘积和
排序贪心 自定义排序规则后顺序处理 会议室II(253)、安排课程(630)
跳跃贪心 维护最远可达位置 跳跃游戏(55)、跳跃游戏II(45)
货币贪心 尽量先用大面额 柠檬水找零(860)、特定硬币体系下的找零

贪心总结

贪心算法表面上看起来"简单"——不就是每一步选最优吗？但贪心的难度不在实现，在证明。你写一个贪心算法可能只需要 10 行代码，但证明它为什么正确可能需要好几页纸。

贪心难在设计，不是难在实现。证明贪心的正确性比写代码更重要。如果你在面试中遇到一个贪心题，不要急着写代码——先和面试官说清楚你的贪心策略（"每次选结束时间最早的"），再简单解释为什么这样是对的。面试官更看重你的思考过程，而不是代码本身。

这里有一个贪心适用判断 checklist，可以帮助你快速评估一个题目能否用贪心：

# 检查项 说明
1 能否举出一个反例？ 试着构造一个输入，让贪心策略失败。如果找不到，贪心可能有效
2 局部最优是否妨碍全局最优？ 选择一个局部最优后，后续的选择空间是否会受到不可接受的限制？
3 交换论证能否成立？ 假设最优解中某个选择不符合贪心策略，能否通过"交换"来改进或保持最优解？
4 是否存在重叠子问题？ 如果子问题大量重叠且你不需要所有解，DP 可能比贪心更安全
5 数据规模是否允许 DP？ 如果 n ≤ 1000，DP 通常可以接受；如果 n ≥ 10⁵，只能用贪心
踩坑故事——"贪心秒杀？小心翻车"

一位朋友在面试中遇到一道题：给定一组人的身高和体重，求最多能叠多高的"人梯"（上面的人必须比下面的人矮且轻）。他脱口而出："先按身高排序，然后贪心取！"面试官没有反驳，让他写代码。他花 5 分钟写完了，面试官问："你确定这是对的吗？"他想了想，举了一个反例：(160,60) (170,50) (180,70)——按身高排序后 (160,60) 不能站在 (170,50) 上面因为体重不满足。

正确解法是先按身高排，身高相同时按体重排，然后对体重求 LIS（最长递增子序列）。这不是贪心，而是排序 + DP。他因为急于用贪心，反而暴露了对问题理解的不足。教训是：不要为了"贪心"而贪心——先用小例子验证你的策略，确认无误再下手。
