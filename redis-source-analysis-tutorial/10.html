第10章：List 命令与阻塞操作

大家好，我是你们的讲师。前面两章我们讨论了 String 和 Hash 的命令实现——它们有一个共同点：所有操作都是"立即完成"的，没有等待、没有阻塞。但现实世界中的很多场景要求我们"等待数据到来"：一个消息队列的消费者在队列为空时应该原地等待，而不是不停地轮询。Redis 的 List 类型通过 BLPOP / BRPOP 等阻塞命令提供了这种能力。

阻塞操作是 Redis 单线程模型中最精妙也最容易出问题的部分。本章我们深入 t_list.c，不仅要看 LPUSH/LPOP 的常规操作，更要搞清楚当一个客户端在 BLPOP 上等待时，Redis 内部到底发生了什么——客户端去哪了？谁唤醒它？超时怎么处理？

quicklist 深层结构

在第五章我们提到过 ziplist 和 listpack，而 quicklist 是 Redis 3.2 引入的 List 专用数据结构。它不是单一的链表或压缩列表，而是一个双向链表 + listpack 节点的混合体。

/* quicklist.h */
typedef struct quicklist {
    quicklistNode *head;      /* 头节点 */
    quicklistNode *tail;      /* 尾节点 */
    unsigned long count;      /* 所有 listpack 中元素总数 */
    unsigned long len;        /* quicklistNode 节点数 */
    int fill : QL_FILL_BITS;  /* 每个节点 listpack 的最大填充大小 */
    unsigned int compress : QL_COMP_BITS; /* 两端不压缩的节点数 */
    unsigned int bookmark_count : QL_BM_BITS;
    quicklistBookmark bookmarks[];
} quicklist;

typedef struct quicklistNode {
    struct quicklistNode *prev;
    struct quicklistNode *next;
    unsigned char *entry;     /* 指向 listpack 的指针 */
    size_t sz;                /* entry 的大小（字节） */
    unsigned int count : 16;  /* 该节点中的元素数量 */
    unsigned int encoding : 2;/* 1=RAW, 2=LZF */
    unsigned int container : 2;/* 1=NONE, 2=LISTPACK */
    unsigned int recompress : 1;/* 临时解压标记 */
    unsigned int attempted_compress : 1;
    unsigned int extra : 10;
} quicklistNode;

三层结构各有职责：quicklist 是顶层的管理结构，记录链表头和尾以及全局统计信息；quicklistNode 是中间层的链表节点，每个节点指向一个 listpack；最底层的 listpack 存储实际元素。

两个关键参数：

  • fill（对应配置 list-max-listpack-size）：控制每个 quicklistNode 中 listpack 的最大大小。可以使用正数表示元素个数上限，也可以使用负数表示字节数上限（-1 = 4KB，-2 = 8KB，默认 -2）。
  • compress（对应配置 list-compress-depth）：控制两端有多少个节点不压缩。例如 compress=1 表示首尾各 1 个节点不压缩，中间的节点使用 LZF 算法压缩存储。
LZF 无锁压缩：Redis 的 LZF 压缩算法不是流式的，也不涉及动态字典，因此不需要锁。它的核心思想是：找到重复的字节序列，用 (偏移量, 长度) 元组替换。quicklistNode 中的 encoding 字段标记了该节点是否经过 LZF 压缩。当要访问一个压缩节点的数据时，需要先解压再读取——这个开销可能会让 LINDEX 在 compress 深度大的 List 上变慢。

List 命令源码

pushGenericCommand —— LPUSH / RPUSH

/* t_list.c */
void pushGenericCommand(client *c, int where) {
    robj *lobj;
    int pushed = 0;

    /* 查找或创建键 */
    lobj = lookupKeyWrite(c->db, c->argv[1]);
    if (lobj == NULL) {
        lobj = createQuicklistObject();
        dbAdd(c->db, c->argv[1], lobj);
    } else {
        if (checkType(c, lobj, OBJ_LIST)) return;
    }

    /* 逐个将 argv[2..] 插入到列表头或尾 */
    for (int j = 2; j < c->argc; j++) {
        c->argv[j] = tryObjectEncoding(c->argv[j]);
        if (where == LIST_HEAD) {
            listTypePush(lobj, c->argv[j], QUICKLIST_HEAD);
        } else {
            listTypePush(lobj, c->argv[j], QUICKLIST_TAIL);
        }
        pushed++;
    }

    addReplyLongLong(c, (lobj->ptr ? listTypeLength(lobj) : 0));
}

核心在 listTypePush 中，它调用 quicklistPush，根据 where 参数决定调用 quicklistPushHead 或 quicklistPushTail。这两个函数都会检查头/尾节点是否还有空间，如果没有则新建一个 quicklistNode。

lpopCommand / rpopCommand

弹出操作的实现同样简单——从 quicklist 头部或尾部移除并返回一个元素。Redis 7.0 后支持了 LPOP count 和 RPOP count 的批量弹出语法：

/* t_list.c */
void lpopCommand(client *c) {
    long long count = -1; /* 默认弹出 1 个 */

    /* 解析可选的 count 参数 */
    if (c->argc == 3) {
        if (getLongLongFromObjectOrReply(c, c->argv[2], &count, NULL) != C_OK)
            return;
    }

    robj *o = lookupKeyWriteOrReply(c, c->argv[1], shared.null[c->resp]);
    if (o == NULL || checkType(c, o, OBJ_LIST)) return;

    if (count == -1) {
        /* 无 count 参数：弹出 1 个元素，返回 Bulk String */
        robj *value = listTypePop(o, LIST_HEAD);
        if (value == NULL) {
            addReplyNull(c);
            return;
        }
        addReplyBulk(c, value);
        listTypeTryConvertQuicklistOrSetMaxSize(o);
        if (listTypeLength(o) == 0) dbDelete(c->db, c->argv[1]);
    } else {
        /* 有 count 参数：弹出 N 个元素，返回 Array */
        popMultipleCommand(c, o, LIST_HEAD, count);
    }
}

llenCommand —— 为什么它是 O(1)

LLEN 的实现异常简单：addReplyLongLong(c, listTypeLength(o))。而 listTypeLength 返回的是 quicklist->count 字段，这个字段在每次 push/pop 时都会更新，因此 LLEN 不需要遍历整个链表。

lindexCommand / lrangeCommand

LINDEX key index 需要遍历 quicklist 找到指定位置的元素。如果 index 为正数，从头开始遍历节点并累加 count；为负数时从尾开始。由于 quicklist 的每个节点都记录了该节点中的元素数量，我们可以快速跳过整个节点，而不必逐个元素遍历。

LRANGE key start stop 也是类似，它创建一个迭代器 listTypeInitIterator，定位到 start 位置，然后逐个读取直到 stop。

命令 时间复杂度 底层实现
LPUSH / RPUSH O(1) 每个元素 quicklistPushHead / Tail
LPOP / RPOP O(1) quicklistPopHead / Tail
LLEN O(1) 返回 quicklist->count
LINDEX O(N) 遍历 quicklistNode 定位
LRANGE O(S+N) 迭代器遍历
LINSERT O(N) 遍历查找插入点
LREM O(N) 遍历并匹配删除
LSET O(N) 遍历到索引后修改
LTRIM O(N) 保留区间内元素

阻塞命令实现

这是 List 模块中最复杂也最有趣的部分。当客户端执行 BLPOP key timeout 而 key 不存在或列表为空时，Redis 不会给客户端返回 null，而是把客户端挂起，等待另一个客户端向该 key 执行 LPUSH 或 RPUSH。

blockingPopGenericCommand —— 阻塞弹出入口

/* t_list.c */
void blockingPopGenericCommand(client *c, int where) {
    robj *o;
    mstime_t timeout;
    int j;

    /* 1. 解析超时时间 */
    if (getLongLongFromObjectOrReply(c, c->argv[c->argc-1], &timeout, NULL) != C_OK)
        return;
    if (timeout <= 0) timeout = 0; /* 0 表示无限等待 */

    /* 2. 先尝试非阻塞弹出——从列表中依次检查每个 key */
    for (j = 1; j < c->argc-1; j++) {
        o = lookupKeyWrite(c->db, c->argv[j]);
        if (o != NULL) {
            if (o->type != OBJ_LIST ||
                listTypeLength(o) == 0) continue;

            robj *value = listTypePop(o, where);
            /* ... 回复并清理空 key ... */
            return;
        }
    }

    /* 3. 所有 key 都为空——进入阻塞状态 */
    if (timeout != 0) c->bpop.timeout = timeout;
    blockForKeys(c, where, c->argv + 1, c->argc - 2, NULL, NULL);
}

blockForKeys —— 注册阻塞

/* blocking.c */
void blockForKeys(client *c, int where, robj **keys, int numkeys,
                  robj **target, robj **target_sd) {
    dictEntry *de;
    list *l;
    int j;

    /* 1. 设置客户端阻塞状态 */
    c->bpop.keys = zmalloc(sizeof(robj*) * numkeys);
    c->bpop.count = numkeys;
    c->bpop.target = target ? target[0] : NULL;
    c->bpop.where = where;
    c->bpop.numreplicas = 0;
    c->bpop.reploffset = 0;
    blockClient(c, BLOCKED_LIST);

    /* 2. 在全局 blocking_keys 字典中注册 */
    for (j = 0; j < numkeys; j++) {
        de = dictFind(c->db->blocking_keys, keys[j]);
        if (de == NULL) {
            /* 该 key 的阻塞链表还不存在，创建 */
            l = listCreate();
            dictAdd(c->db->blocking_keys, keys[j], l);
        } else {
            l = dictGetVal(de);
        }
        /* 把当前客户端追加到 key 的等待列表中 */
        listAddNodeTail(l, c);
    }
}

这个函数做了两件事：第一，把客户端标记为 BLOCKED_LIST 状态，此后事件循环不会再从这个客户端读取命令；第二，在 db->blocking_keys 字典中，以被等待的 key 为键，把客户端加入等待列表。

解除阻塞的时机

阻塞的解除发生在两种情况：

  • 数据到达：当另一个客户端执行 LPUSH key value 时，pushGenericCommand 调用 signalListAsReady，在 ready_keys 链表中记录该 key。然后 beforeSleep() 在处理待办事件时，遍历 ready_keys，找到所有等待该 key 的客户端并唤醒它们。唤醒后，客户端立即执行非阻塞的 LPOP 获取数据。
  • 超时：每次事件循环迭代时，handleClientsBlockedOnKeys 会检查阻塞客户端的超时时间，超时的客户端被解除阻塞并返回 null。
关键设计：解除阻塞的客户端不会重新执行阻塞命令——因为 blockForKeys 执行之前已经做过了非阻塞尝试（步骤 2 of blockingPopGenericCommand）。当数据到达时触发的是对 key 的 signal，被唤醒的客户端直接 LPOP 就能拿到数据。这个"先尝试非阻塞、再阻塞"的两阶段设计，避免了竞争条件和重复执行的开销。

阻塞状态管理

Redis 的阻塞机制涉及三个核心数据结构：

  • blocking_keys（字典）：key → 等待该 key 的客户端列表。用于在有新数据写入时找到所有等待者。
  • ready_keys（链表）：当某个 key 被写入导致其上的阻塞可能解除时，该 key 被加入就绪链表。beforeSleep 在处理就绪链表时，只会在必要的时候才去真正唤醒客户端——这种延迟处理避免了在命令执行中途嵌套处理阻塞逻辑。
  • BLOCKED_LIST（客户端状态标志）：client->flags 中的 CLIENT_BLOCKED 位标记。处于阻塞状态的客户端不会被事件循环的 IO 处理，但连接依然保持。
踩坑实录：BLPOP 支持同时监听多个 key，例如 BLPOP queue1 queue2 queue3 0。这里有一个容易被忽略的行为：当多个 key 同时有数据到达时，解除阻塞的优先级不是由 key 的排列顺序决定的，而是由 ready_keys 链表中 key 的插入顺序决定的——哪个 key 先被触发（有 LPUSH 写入），哪个就先响应。曾经有业务方想用 BLPOP 多 key 来实现"优先级队列"，结果发现优先级取决于写入顺序而非 key 顺序，与预期完全不符。正确的做法是用单个 List 配合 ZSet 或独立优先级字段来实现优先级队列。

List 作为消息队列

Redis List 的 LPUSH + BRPOP 模式是最简单的消息队列实现。它的优点是零依赖——不需要 Kafka 或 RabbitMQ，只要一个 Redis 实例就能跑通生产者-消费者模型。但和专业消息队列相比，它有几个明显的差距：

  • 消息可靠性：BRPOP 返回消息后，如果消费者崩溃，消息就丢失了（没有类似 Kafka 的 offset 或 RabbitMQ 的 ack 机制）。
  • 消息确认：没有内置的消息确认和重投递机制。可以自己用 RPOPLPUSH 或 BRPOPLPUSH 把消息先备份到一个 "processing" 列表来模拟确认。
  • 持久化：List 的持久化依赖 RDB/AOF，但消息队列通常需要更高的持久化保证。

对于轻度使用场景（比如内部通知、任务分发延迟不敏感），List 消息队列完全够用。但如果需要严格的消息可靠性和复杂的路由，应该选用 Stream（第 12 章会详细讲）或专业的消息队列系统。

BRPOPLPUSH 的妙用：BRPOPLPUSH source dest timeout 原子性地从 source 弹出元素并 push 到 dest。这可以用于实现可靠队列：BRPOPLPUSH 从主队列取出消息放入处理队列，处理完成后从处理队列删除。如果消费者崩溃，重启后可以从处理队列中找到未完成的消息。

List 在链表结构支持下实现了灵活的双端操作和阻塞等待，但它的元素是无序的且不能重复。接下来我们要讨论的 Set 和 Sorted Set 则展示了哈希表与跳表在集合运算和排序场景中的精巧配合。
