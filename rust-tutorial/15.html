开篇：那个被"已测过"坑死的夜间发布

在上一家公司，我们有一个核心支付服务，用C++编写，代码量20万行有余。每次发布前，QA团队会跑一遍回归测试——大约800个用例，耗时40分钟。某个周五晚上，一个看似人畜无害的修改被合并了进去：在某个回调函数里加了一个if (ptr) delete ptr;。测试全绿，发布上线。

周六凌晨，线上开始爆SIGSEGV。core dump一看——那个delete ptr在某个路径下double free了。为什么测试没发现？因为已有的单元测试覆盖率不到30%，集成测试只覆盖了Happy Path，那个特定的错误路径恰好没有被任何测试覆盖到。

那件事之后我深刻认识到一个道理：没有测试保证的代码，本质上是一堆"看起来能用"的未爆弹。Rust的设计者显然明白这一点——他们把测试工具直接内置到了语言和工具链中，不需要第三方框架，不需要纠结"该用JUnit还是TestNG"。在Rust中写测试，就像写fn main一样自然。

测试函数 — #[test]、assert!/assert_eq!/assert_ne!

在Rust中，任何函数只要加上#[test]属性，就会被识别为测试函数。你不需要继承某个Test基类，不需要用特殊的命名约定——一个属性就搞定。

第一个测试

创建一个库项目来看看效果：

cargo new --lib my_math
cd my_math

编辑src/lib.rs：

pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_add() {
        assert_eq!(add(2, 2), 4);
    }

    #[test]
    fn test_add_negative() {
        assert_eq!(add(-1, 1), 0);
    }

    #[test]
    fn test_add_fail() {
        // 这个测试会失败，故意留着演示
        assert_eq!(add(2, 2), 5);
    }
}

断言宏三兄弟

Rust标准库提供了三个核心断言宏，它们构成了测试的基础设施：

宏 用途 失败时输出
assert!(condition) 检查布尔表达式是否为真 assertion failed: condition
assert_eq!(left, right) 检查两个值是否相等 assertion `left == right` failed: left: X, right: Y
assert_ne!(left, right) 检查两个值是否不相等 assertion `left != right` failed: left: X, right: Y

每个宏还支持自定义消息，用format!风格的参数：

#[test]
fn test_with_message() {
    let result = add(2, 2);
    assert!(
        result == 4,
        "2 + 2 应该等于 4，但实际得到了 {}",
        result
    );
}
小技巧：assert_eq!要求比较的值实现了PartialEq和Debug trait。前者用于比较，后者用于失败时的输出。如果你的自定义类型需要测试，记得派生这两个trait：#[derive(Debug, PartialEq)]。

cargo test — 运行测试、过滤、--nocapture

运行所有测试只需一条命令：

cargo test

Cargo会编译所有测试（包括被#[cfg(test)]标注的模块），然后逐个运行并报告结果。输出大概是这样的：

running 3 tests
test tests::test_add ... ok
test tests::test_add_negative ... ok
test tests::test_add_fail ... FAILED

failures:

---- tests::test_add_fail stdout ----
thread 'tests::test_add_fail' panicked at src/lib.rs:18:9:
assertion `left == right` failed: left: 4, right: 5
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

failures:
    tests::test_add_fail

test result: FAILED. 2 passed; 1 failed; 0 ignored; 0 measured; 0 filtered out

过滤测试名称

当测试越来越多时，你不想每次都跑全部测试。cargo test支持按名称过滤：

# 只运行名称中包含 "add" 的测试
cargo test add

# 只运行 test_add_negative
cargo test test_add_negative

# 运行单个模块下的所有测试
cargo test tests::

--nocapture 查看输出

默认情况下，测试中所有println!输出是不会显示在终端上的——Rust会捕获它们，只有在测试失败时才释放输出。如果你想在开发时看到实时输出（比如调试有些测试为什么挂起），可以用：

cargo test -- --nocapture

注意两个--之间的空格：第一个--分隔cargo参数和测试二进制参数，--nocapture是传递给测试二进制本身的。

其他实用标志

  • cargo test -- --test-threads=1 —— 单线程运行，排查数据依赖问题
  • cargo test -- --ignored —— 只运行被 #[ignore] 标注的测试
  • cargo test --include-ignored —— 运行全部测试包括被忽略的

测试组织 — 单元测试与集成测试

Rust将测试分为两大类：单元测试和集成测试。这个区分不是随意划定的——它们有各自的目的和组织方式。

单元测试：与源码同在

单元测试通常与待测代码放在同一个文件中，放在一个被#[cfg(test)]标注的mod tests模块里。这个属性告诉编译器：只有在运行cargo test时才编译这个模块，普通cargo build会跳过它，这样测试代码不会影响最终二进制的大小。

// src/lib.rs
pub fn divide(a: f64, b: f64) -> Result<f64, String> {
    if b == 0.0 {
        Err(String::from("除数不能为零"))
    } else {
        Ok(a / b)
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_divide_normal() {
        assert_eq!(divide(10.0, 2.0).unwrap(), 5.0);
    }

    #[test]
    fn test_divide_by_zero() {
        assert!(divide(10.0, 0.0).is_err());
    }

    #[test]
    #[should_panic(expected = "除数不能为零")]
    fn test_divide_panic() {
        divide(10.0, 0.0).unwrap();
    }
}

注意#[should_panic]属性——它断言测试函数应该panic。你可以用expected参数来匹配panic消息，这在测试错误路径时非常有用。

集成测试：tests/目录

集成测试放在项目根目录的tests/文件夹中。每个.rs文件就是一个独立的测试crate——它们只能访问你库的公开API，就像外部用户使用你的库一样。

// tests/integration_test.rs
use my_math;  // 引入库crate

#[test]
fn test_add_integration() {
    assert_eq!(my_math::add(100, 200), 300);
}

#[test]
fn test_multiple_scenarios() {
    let cases = vec![(1, 1, 2), (0, 0, 0), (-1, 1, 0), (100, -50, 50)];
    for (a, b, expected) in cases {
        assert_eq!(my_math::add(a, b), expected);
    }
}

集成测试有个小陷阱：tests/下的每个文件都被编译为独立的crate，所以如果你有辅助函数想在多个集成测试之间共享，不要直接放在tests/*.rs中——放进去的每个.rs文件都会被当作测试入口。解决方案是使用tests/common/mod.rs这种子模块路径，这样Rust就不会把common.rs当作测试文件。

测试组织一览
┌──────────────┐
│ src/lib.rs │ ← 单元测试放在 #[cfg(test)] 模块中
│ 生产代码 + │ 可访问私有函数
│ 单元测试 │
└──────────────┘
┌──────────────┐
│ tests/ │ ← 集成测试，每个文件独立crate
│ *.rs 文件 │ 只能访问公开API
└──────────────┘
┌──────────────┐
│ 文档注释 │ ← 文档测试（见下一节）
│ /// 中的代码 │ 可以访问公开API
└──────────────┘

文档测试 — /// doc comments with code blocks

这是Rust最让人惊喜的特性之一。你在文档注释（///）中写的代码块，会被自动当作测试来运行！这意味着你的文档永远不会过时——如果代码和文档不一致，cargo test会告诉你。

/// 将两个数相加。
///
/// # 示例
///
/// ```
/// use my_math::add;
///
/// assert_eq!(add(2, 3), 5);
/// assert_eq!(add(-1, 1), 0);
/// ```
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

当你运行cargo test时，Rust会提取每个///注释中的代码块，将它们编译为独立的二进制并运行。这意味着：

  • 你的示例代码一定是可以编译通过且能正确运行的
  • API重构时如果改了签名，文档示例会自动报错
  • 用户看到的文档中的示例一定是真实可靠的
最佳实践：为每个公共函数写上带有示例代码的文档注释。这不仅提高了文档质量，还能免费获得额外的测试覆盖。crates.io上的高星crate几乎都遵循这一惯例——你在阅读它们的源码时会发现这一点。

如果文档中的代码块不应该编译（比如你只是想展示一个概念性的代码片段），可以在代码块开始处加上ignore或no_run：

/// 这是一个概念示例，不会真的编译：
///
/// ```ignore
/// let x = some_hypothetical_function();
/// ```
///
/// 这段代码会编译但不会运行（比如会无限循环）：
///
/// ```no_run
/// loop {
///     println!("永远运行");
/// }
/// ```

基准测试 — #[bench]、cargo bench、criterion

写完正确的代码之后，你自然会关心：这段代码跑得快不快？Rust提供了内置的基准测试（benchmark）支持，但需要注意稳定性约定。

内置 #[bench]（仅限nightly）

标准库的#[bench]属性目前还是nightly-only的特性。如果你愿意使用nightly工具链，可以这样写：

#![feature(test)]

extern crate test;
use test::Bencher;

#[bench]
fn bench_add(b: &mut Bencher) {
    b.iter(|| {
        for i in 0..1000 {
            std::hint::black_box(add(i, i));
        }
    });
}

std::hint::black_box防止编译器优化掉你的基准测试代码——它会告诉编译器"这个值虽然看起来没用，但请假设它被外部使用了"。

使用criterion（stable推荐）

对绝大多数Rust开发者来说（包括使用stable工具链的人），社区选择是criterion crate——它是Rust生态中最流行的基准测试框架，提供了比内置#[bench]更丰富的统计分析和报告。

在Cargo.toml中添加：

[dev-dependencies]
criterion = { version = "0.5", features = ["html_reports"] }

[[bench]]
name = "my_bench"
harness = false

然后创建benches/my_bench.rs：

use criterion::{black_box, criterion_group, criterion_main, Criterion};

fn fibonacci(n: u64) -> u64 {
    match n {
        0 => 1,
        1 => 1,
        n => fibonacci(n - 1) + fibonacci(n - 2),
    }
}

fn criterion_benchmark(c: &mut Criterion) {
    c.bench_function("fib 20", |b| b.iter(|| fibonacci(black_box(20))));
}

criterion_group!(benches, criterion_benchmark);
criterion_main!(benches);

运行：

cargo bench

criterion会生成漂亮的HTML报告（如果你启用了html_reports feature），放在target/criterion/reports/目录下。它还支持对比不同版本的性能变化——如果你在重构，它能告诉你"这次改动让代码快了3%还是慢了5%"。

基准测试的坑：我曾经"优化"了一个函数，在benchmark上快了20%，欢天喜地合并了代码。第二天CI上的基准测试告诉我不但没有优化，反而慢了5%——原来我跑benchmark的时候笔记本电脑插着电源，而CI跑在节能模式上。永远记住：基准测试对运行环境极度敏感。在同样的硬件、同样的负载、多次运行下取平均值，结论才可信。

本章小结

测试是代码质量的基石，而Rust把测试工具——从单元测试到集成测试、从文档测试到基准测试——都内建到了工具链中。这一章我们学习了如何用#[test]编写测试函数，用cargo test组织运行它们，用文档注释中的代码块自动保证文档与实现一致，以及用criterion做专业的性能基准测试。
