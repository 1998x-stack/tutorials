开篇：一个让前缀和"崩溃"的需求

假设你是某个电商平台的工程师，负责一个实时大屏展示系统。产品经理提出了一个需求：

"我们要在大屏上展示当天每小时的销售额折线图，还要支持任意时间段的总销售额汇总——比如看上午10点到下午3点的总和。"

你微微一笑，心想这还不简单？你立刻写了一个数组来存储每小时的销售额，然后预计算一个前缀和（Prefix Sum）数组。这样，任何区间[i, j]的总和就是 prefix[j] - prefix[i-1]，O(1)时间搞定。

产品经理看了很满意。然后第二天，他又来了：

"那个……我们要上线一个功能，允许运营人员手动修正某一小时的销售数据（比如发现某小时的统计有误）。能不能加个编辑按钮？"

你突然意识到问题所在：一旦修改了原数组中的一个值，整个前缀和数组都要重新计算——O(n)的更新成本。如果运营一秒修改十次数据，每次修改都导致O(n)的重算，系统就扛不住了。

你需要一种数据结构，既能高效地查询任意区间的信息，又能高效地更新单个点的值。这就是线段树（Segment Tree）和树状数组（Fenwick Tree / Binary Indexed Tree）解决的问题。

区间查询问题

先来明确一下我们要解决的问题形式化定义：

给定一个长度为n的数组arr，我们需要支持两种操作：

  • 查询 query(l, r)：返回 arr[l] + arr[l+1] + ... + arr[r] 的总和（也可以是最大值、最小值等）。
  • 更新 update(idx, val)：将 arr[idx] 的值修改为 val。

不同的解决方案在不同操作上的复杂度对比如下：

方案 区间查询 单点更新 代码量
前缀和 O(1) ✅ O(n) ❌ 极短
普通数组（暴力更新） O(n) ❌ O(1) ✅ 极短
线段树 O(log n) ✅ O(log n) ✅ 较长（~50行）
树状数组 O(log n) ✅ O(log n) ✅ 较短（~15行）

前缀和查得快但更新慢，暴力数组更新快但查得慢。线段树和树状数组在两者之间取得了漂亮的平衡——查询和更新都是O(log n)。

线段树的结构

线段树的核心思想是：用二叉树来管理区间信息。树中的每个节点代表原数组的一个区间。根节点代表整个数组[0, n-1]，左孩子代表左半区间，右孩子代表右半区间，以此类推，直到叶子节点代表单个元素。

如果你学过第8章的树基础，会意识到这是一种完全二叉树（注意不一定是满二叉树）。如果原数组有n个元素，那么线段树大约需要4n的空间来存储。

class SegmentTree:
    def __init__(self, arr):
        n = len(arr)
        self.n = n
        self.tree = [0] * (4 * n)  # 4倍空间
        self._build(arr, 0, 0, n - 1)

    def _build(self, arr, node, start, end):
        """建树：递归地将区间信息存储到对应节点"""
        if start == end:
            # 叶子节点：直接存原数组的值
            self.tree[node] = arr[start]
            return
        mid = (start + end) // 2
        left_child = 2 * node + 1
        right_child = 2 * node + 2
        self._build(arr, left_child, start, mid)
        self._build(arr, right_child, mid + 1, end)
        # 父节点 = 左子 + 右子（求和线段树）
        self.tree[node] = self.tree[left_child] + self.tree[right_child]

    def _query(self, node, start, end, l, r):
        """区间查询 [l, r] 的总和"""
        if r < start or l > end:
            # 查询区间与当前节点区间无交集
            return 0
        if l <= start and end <= r:
            # 当前节点区间完全被查询区间覆盖
            return self.tree[node]
        mid = (start + end) // 2
        left = self._query(2 * node + 1, start, mid, l, r)
        right = self._query(2 * node + 2, mid + 1, end, l, r)
        return left + right

    def query(self, l, r):
        return self._query(0, 0, self.n - 1, l, r)

    def _update(self, node, start, end, idx, val):
        """单点更新：将 arr[idx] 更新为 val"""
        if start == end:
            self.tree[node] = val
            return
        mid = (start + end) // 2
        if idx <= mid:
            self._update(2 * node + 1, start, mid, idx, val)
        else:
            self._update(2 * node + 2, mid + 1, end, idx, val)
        self.tree[node] = self.tree[2 * node + 1] + self.tree[2 * node + 2]

    def update(self, idx, val):
        self._update(0, 0, self.n - 1, idx, val)

这段代码虽然看起来有几十行，但核心逻辑只有一句话：每个节点代表一个区间，其值是两个子节点的值的和（或最大值/最小值等）。查询时，如果当前节点区间完全在查询范围内就直接返回，否则递归左右子树。更新时，先递归到叶子节点改值，然后回溯时更新所有父节点。

理解线段树的技巧：先只理解"求和线段树"这一种。一旦你掌握了求和版本，最大/最小值的线段树只是把"+"换成"max"或"min"而已。区间更新涉及懒标记（Lazy Propagation）略有复杂，建议作为进阶内容。

线段树的操作详解

建树 — O(n)

从根节点开始，递归地构建整棵树。每个叶子节点存一个元素的值，父节点合并两个子节点的信息。总共有2n-1个节点（对于满二叉树的情况），所以建树是O(n)。

区间查询 — O(log n)

查询[2, 5]的总和时，递归从根节点开始。如果当前节点区间完全被[2, 5]覆盖，直接返回节点值；如果完全没有交集，返回0；如果部分重叠，递归左右子节点。树的高度是log n，每层最多访问常数个节点，总复杂度O(log n)。

单点更新 — O(log n)

更新arr[3]的值时，从根节点开始，判断3在左子树还是右子树的区间，一路递归到对应的叶子节点，更新叶子节点的值，然后回溯时更新所有路径上的父节点。因为树高为log n，所以更新也是O(log n)。

区间更新（懒标记）

如果要给整个区间[2, 5]的每个元素都加上一个值x，按照上面的方式逐个更新叶子节点需要O(n log n)的时间，太慢了。懒标记（Lazy Propagation）的优化思路是：在父节点就把更新"存起来"（打上标记），不立即下推到子节点。只有当后续查询需要访问子节点时，才把标记往下"推"。这样，区间更新的复杂度也是O(log n)。

懒标记是实现复杂度的分水岭——很多算法竞赛选手在这里卡住。我的建议是：先掌握不带懒标记的线段树（单点更新+区间查询），它已经能解决80%的问题了。

树状数组（Fenwick Tree）

如果你觉得线段树的代码量有点大，好消息是：对于「单点更新 + 区间求和」这个最常见的子问题，还有一种更加轻量级的解决方案——树状数组（Binary Indexed Tree, BIT）。

树状数组由Peter Fenwick在1994年提出，它的核心是一个叫作 lowbit 的魔法操作：

lowbit(x) = x & (-x)

lowbit(x)返回的是x的二进制表示中最低位的1所代表的值。比如x=6（二进制110），lowbit(6)=2（二进制10）。这个简单的位运算，是树状数组全部奥秘的起点。

树状数组的直觉是这样的：每个下标i并不只存储arr[i]的值，而是"负责"存储从 i - lowbit(i) + 1 到 i 这个范围内的所有元素的和。这样设计的好处是，无论是求前缀和还是更新操作，都只需要沿着下标跳跃log n次。

class FenwickTree:
    """树状数组：支持单点更新 + 前缀和查询"""

    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)  # 1-indexed，注意长度是 n+1

    def lowbit(self, x):
        return x & (-x)

    def update(self, idx, delta):
        """在 idx 位置加上 delta"""
        # idx 基于 1
        while idx <= self.n:
            self.bit[idx] += delta
            idx += self.lowbit(idx)

    def query(self, idx):
        """求前缀和 [1, idx]"""
        s = 0
        while idx > 0:
            s += self.bit[idx]
            idx -= self.lowbit(idx)
        return s

    def range_query(self, l, r):
        """求区间和 [l, r]"""
        return self.query(r) - self.query(l - 1)


# 使用示例
arr = [1, 3, 5, 7, 9, 11]
ft = FenwickTree(len(arr))
for i, v in enumerate(arr, start=1):  # 1-indexed
    ft.update(i, v)

print(ft.range_query(2, 5))  # 3 + 5 + 7 + 9 = 24
ft.update(3, 10)             # arr[3] += 10 → arr[3] = 15
print(ft.range_query(2, 5))  # 3 + 15 + 7 + 9 = 34

你有没有注意到代码有多短？15行就搞定了。而且它只需要O(n)的额外空间（比线段树的4n省得多）。这就是树状数组的魔力：以极小的代码量，实现了和线段树一样O(log n)的查询和更新。

树状数组能做的事线段树都能做，但线段树代码量是树状数组的3-5倍。先考虑树状数组。 但是要注意：树状数组只能处理"前缀可合并"的问题（求和、求异或、求最大值——但最大值比较特殊），而且不能直接支持区间更新。如果你的问题需要区间更新（不是单点更新）或是求区间最大值/最小值，线段树是更好的选择。

实战对比：区间和问题

假设我们有数组 arr = [1, 3, 5, 7, 9, 11, 13, 15]，用三种不同的方案来做查询和更新：

操作 前缀和 线段树 树状数组
初始化 O(n) O(n) O(n)
查询区间[2, 6] O(1) — prefix[6]-prefix[1] O(log n) O(log n)
更新arr[3]=100 O(n) — 需要重建前缀和 O(log n) O(log n)
代码行数 ~3行 ~50行 ~15行
额外空间 O(n) O(4n) O(n)
支持区间更新 否 是（加懒标记） 有限支持（差分技巧）
适用范围 静态数据，只查不改 通用区间统计 单点更新+区间求和
踩坑实录：有一年我在写一个实时股票行情监控系统，行情数据每秒更新几十次，前端需要实时刷新某几只股票在某个时间段内的平均成交价。第一版我用了前缀和，每次更新都要重建整个数组——服务器CPU直接飙到90%。第二版换上了树状数组，CPU降到了5%。教训是什么？前缀和虽好，但只能在"数据不变"的前提下用。只要数据会变，就得考虑支持动态更新的数据结构。

如何选择：线段树 vs 树状数组

很多初学者会纠结"到底学哪个？"我的建议是：两个都学，但优先掌握树状数组。理由如下：

  • 如果你只需要做单点更新 + 区间求和，树状数组是首选——代码短、速度快、空间小。
  • 如果你需要区间更新（比如给整个区间加一个数）——线段树加懒标记更方便。树状数组配合差分技巧也能做到，但理解起来更绕。
  • 如果你需要求区间最大值/最小值——树状数组不擅长这个（除非你能巧妙地转化为前缀问题），线段树更自然。
  • 如果你在准备算法竞赛——两个都要熟练掌握。比赛中树状数组用来快速解决简单题，线段树用来解决需要区间更新的难题。

本章小结

在这一章中，我们学习了：

  • 区间查询问题的本质——前缀和查得快但更新慢，数组更新快但查得慢，线段树和树状数组在两者之间取得了平衡。
  • 线段树——用二叉树管理区间信息，支持区间查询和单点更新均为O(log n)，区间更新需要懒标记。
  • 树状数组——利用lowbit(x)函数的魔法，以极短的代码实现单点更新+前缀和查询O(log n)。
  • 三种方案的对比——前缀和（只查不改）、线段树（通用区间统计）、树状数组（轻量级区间求和）。

下一章我们将进入另一个非常实用的数据结构——字典树（Trie）。如果说线段树解决的是"区间"问题，那么Trie解决的是"前缀"问题：如何快速判断一个字符串是否在字典中？如何找到所有以"app"开头的单词？如何在前端输入框里实现"即打即搜"的自动补全？Trie给出了优雅的答案。我们下章见！
