开篇：接口思维的转变

在做代码评审时，我经常看到类似这样的代码：

fn print_twice_int(x: i32) {
    println!("{x}"); println!("{x}");
}
fn print_twice_float(x: f64) {
    println!("{x}"); println!("{x}");
}
fn print_twice_str(x: &str) {
    println!("{x}"); println!("{x}");
}

三个函数，功能一模一样，只是类型不同。我每次都问同一个问题："为什么不用泛型 + Display约束？"回答往往是"哦，我还没习惯Rust的trait方式"。

Trait是Rust中表示"行为"的机制。它告诉编译器：某种类型具有某些能力。理解trait是从"会写Rust"到"会设计Rust代码"的关键一步。

定义与实现Trait

Trait定义了一组方法签名——任何实现了该trait的类型都必须提供这些方法的具体实现：

pub trait Summary {
    fn summarize(&self) -> String;
}

pub struct Article {
    pub title: String,
    pub content: String,
}

// 为 Article 实现 Summary trait
impl Summary for Article {
    fn summarize(&self) -> String {
        format!("{}...", &self.content[..20.min(self.content.len())])
    }
}

pub struct Tweet {
    pub username: String,
    pub text: String,
}

impl Summary for Tweet {
    fn summarize(&self) -> String {
        format!("@{}: {}", self.username, self.text)
    }
}

孤儿规则（Orphan Rule）

一个重要规则：只有当trait或类型至少有一个定义在当前crate中时，才能为类型实现trait。你不能为外部类型实现外部trait。例如，你不能在自己的crate中为Vec<T>实现Display——因为两者都定义在标准库中。这个规则确保了trait实现的唯一性，避免歧义。

Newtype模式绕过孤儿规则：如果你想为外部类型实现外部trait，可以使用newtype模式——把外部类型包装在自己定义的结构体中：struct MyVec<T>(Vec<T>)，然后为MyVec实现trait。这种包装在编译时零开销。

默认实现与覆盖

Trait可以为方法提供默认实现，实现者可以选择覆盖默认行为：

pub trait Summary {
    fn summarize(&self) -> String;

    // 默认实现，依赖于summarize方法
    fn summary_with_source(&self) -> String {
        format!("【摘要】{}", self.summarize())
    }
}

// 使用默认实现的 summary_with_source
impl Summary for Article {
    fn summarize(&self) -> String {
        format!("{}...", &self.content[..20.min(self.content.len())])
    }
    // summary_with_source 使用默认实现
}

// 覆盖默认实现
impl Summary for Tweet {
    fn summarize(&self) -> String {
        format!("@{}: {}", self.username, self.text)
    }

    fn summary_with_source(&self) -> String {
        format!("【推文】@{}: {}", self.username, self.text)
    }
}

默认实现可以调用同一trait中的其他方法（即使它们没有默认实现），这使得trait可以像一个"接口 + 抽象类"的混合体。

Trait作为参数：impl Trait vs 泛型

有两种方式编写接受trait作为参数的函数：

// 方式1：impl Trait 语法（语法糖，适用于单个参数）
fn notify(item: &impl Summary) {
    println!("重大新闻！{}", item.summarize());
}

// 方式2：泛型 + Trait Bound（更通用）
fn notify<T: Summary>(item: &T) {
    println!("重大新闻！{}", item.summarize());
}

两种方式在功能上是等价的——都是单态化的静态分发。不同之处在于：

  • impl Trait适用于简单的参数签名，但无法表达"两个参数是同一个类型"的约束
  • 泛型参数可以表达更复杂的约束，例如两个参数必须是同一个类型
// "两个参数必须是同一个类型"——只能用泛型
fn pair<T: Summary>(a: &T, b: &T) {
    println!("{} ※ {}", a.summarize(), b.summarize());
}

// "两个参数都实现了Summary，但可以是不同类型"——impl Trait可以
fn two_items(a: &impl Summary, b: &impl Summary) {
    println!("{} || {}", a.summarize(), b.summarize());
}

用where子句让多个约束更清晰：

fn complex<T, U>(t: &T, u: &U) -> String
where
    T: Summary + Display,
    U: Summary + Clone,
{
    format!("t: {}, u: {}", t.summarize(), u.summarize())
}

Trait对象与动态分发

前面的impl Trait和泛型绑定的方式都是静态分发（Static Dispatch）——编译器为每个具体类型生成独立的函数副本。另一种方式是动态分发（Dynamic Dispatch）——通过dyn Trait在运行时通过虚函数表调用方法。

// 静态分发：编译时确定具体类型
fn static_dispatch(item: &impl Summary) {
    println!("{}", item.summarize());
}

// 动态分发：运行时确定具体类型
fn dynamic_dispatch(item: &dyn Summary) {
    println!("{}", item.summarize());
}

// 使用
let article = Article { /* ... */ };
let tweet = Tweet { /* ... */ };

static_dispatch(&article);    // 单态化为 Article 版本
static_dispatch(&tweet);      // 单态化为 Tweet 版本

dynamic_dispatch(&article);   // 运行时通过 vtable 调用
dynamic_dispatch(&tweet);     // 同上

对象安全（Object Safety）

不是所有trait都能用作dyn Trait。只有对象安全的trait才能被动态分发。一个trait是对象安全的，当且仅当它满足：

  • 方法的返回类型不是Self（不能返回Self类型）
  • 方法没有泛型类型参数

比如Clone trait就不对象安全，因为clone()返回Self。你没法写Box<dyn Clone>。

dyn Trait的性能权衡：动态分发引入了一个间接函数调用（通过虚函数表），比静态分发略慢，但差异通常在纳秒级别，只有极度热路径才需要考虑。动态分发真正的代价在于阻止了内联优化——编译器无法内联通过虚函数表调用的函数。但它的好处是减少了代码膨胀：无论多少个具体类型，只有一份动态分发的代码。

常用derive宏

Rust中一些trait的实现非常机械——比如Debug（格式化输出）、Clone（克隆）、PartialEq（相等比较）。手动为每个类型实现它们很繁琐，Rust提供了#[derive]属性来自动生成实现：

宏 Trait 用途 要求成员实现对应的trait
#[derive(Debug)] std::fmt::Debug {:?}格式输出，调试用 所有字段实现Debug
#[derive(Clone)] Clone 显式复制对象（.clone()） 所有字段实现Clone
#[derive(Copy)] Copy 按位复制（赋值不转移所有权） 所有字段实现Copy
#[derive(PartialEq)] PartialEq ==和!=比较 所有字段实现PartialEq
#[derive(Eq)] Eq 等价关系（PartialEq的强化版） 类型已实现PartialEq
#[derive(Hash)] Hash 哈希计算，用于HashMap的键 所有字段实现Hash
#[derive(Default)] Default 默认值（T::default()） 所有字段都有默认值 
#[derive(Debug, Clone, PartialEq, Default)]
struct User {
    id: u64,
    name: String,
    email: String,
}

let u1 = User::default();
let u2 = u1.clone();          // Clone
println!("{u2:?}");           // Debug
println!("{}", u1 == u2);     // PartialEq
derive vs 手动实现：derive宏只在成员的trait实现符合你的语义时才能使用。比如PartialEq的derive会逐字段比较——对于User结构体这很合理。但如果User有last_login字段而你不想把它纳入相等比较，就应该手动实现PartialEq。

本章小结

Trait是Rust的"行为接口"，它定义了一组方法契约。我们从定义和实现trait开始，学习了默认实现如何减少重复代码，理解了impl Trait、泛型约束和动态分发dyn Trait的区别，最后了解了derive宏带来的便利。

理解了trait，你已经掌握了Rust设计哲学中"基于行为的抽象"的精髓。但还有一个概念与trait紧密交织——生命周期。在带有引用的函数签名和结构体中，你经常会同时使用trait约束和生命周期标注。第13章，我们专门来讲生命周期——Rust中最让人"烧脑"但又必须掌握的概念。
