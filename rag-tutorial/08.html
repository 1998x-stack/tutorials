第八章：稀疏检索与BM25算法

大家好，我是你们的讲师。今天我们来聊一个既古老又充满活力的主题——稀疏检索与BM25算法。很多人觉得在深度学习席卷一切的今天，BM25这种"传统方法"是不是已经过时了？我的回答是：恰恰相反。在RAG系统中，BM25至今仍是最高频使用的检索算法之一，甚至可以说，没有BM25的RAG系统，就像没有盐的菜——能吃，但总觉得少点什么。

为什么BM25如此重要？因为它在精准匹配和关键词检索方面有着不可替代的优势。当用户搜索一个特定产品型号、一个法律条文编号、或者一个专业术语时，稠密检索往往表现不佳，而BM25却能精确命中。这一章，我们就来深入理解稀疏检索的原理与实践。

1. 倒排索引：稀疏检索的基石

在理解BM25之前，我们先要知道它的底层数据结构——倒排索引（Inverted Index）。传统的"正排索引"是以文档为中心，记录每篇文档包含哪些词。而倒排索引反过来，以"词"为中心，记录每个词出现在哪些文档中。

举个例子，假设我们有三个文档："我喜欢苹果"、"苹果很好吃"、"我喜欢香蕉"。倒排索引就是：

  • 我 → 文档1, 文档3
  • 喜欢 → 文档1, 文档3
  • 苹果 → 文档1, 文档2
  • 好吃 → 文档2
  • 香蕉 → 文档3

这种结构让关键词匹配变得极快——无需扫描所有文档，直接查词就行了。这也是为什么搜索引擎能在毫秒级返回结果的核心原因。

2. TF-IDF：从词频到重要性

有了倒排索引，我们还需要给每个文档-词对打分。最基础的算法是 TF-IDF（Term Frequency - Inverse Document Frequency）：

  • TF（词频）：一个词在文档中出现的次数。出现越多，越重要。
  • IDF（逆文档频率）：包含该词的文档数越少，词越稀有，权重越高。

TF-IDF 评分 = TF × IDF。简单粗暴，但有效。不过TF-IDF有一个明显的问题——它没有被归一化，长文档天然得分更高，因为词出现次数更多。BM25正是在这个基础上做了大量改进。

3. BM25公式详解

BM25（Best Matching 25）是概率检索模型的代表作，它在TF-IDF的基础上引入了两个关键参数来调节词频和文档长度的影响。公式如下：

Score(Q, D) = Σ IDF(q_i) × [ TF(q_i, D) × (k1 + 1) ] / [ TF(q_i, D) + k1 × (1 - b + b × |D| / avgdl) ]

看起来有点复杂，我们来拆解一下：

  • TF(q_i, D)：词 q_i 在文档 D 中的出现频率。
  • |D|：文档 D 的长度。
  • avgdl：所有文档的平均长度。
  • k1：词频饱和度参数，控制词频对评分的边际递减效应。
  • b：文档长度归一化参数，控制文档长度对评分的影响程度。

4. k1 和 b 参数调优

这两个参数是BM25的灵魂：

  • k1（默认 1.2 - 2.0）：控制词频饱和速度。k1=0时只考虑IDF，词频完全不起作用；k1越大，词频的影响越大。对于长文档集合，推荐较大的k1（如1.6-2.0）；对于短文本（如标题、查询），较小的k1效果更好。
  • b（默认 0.75）：控制文档长度归一化强度。b=0表示完全不考虑文档长度；b=1表示完全归一化。如果文档长度差异很大，b应该接近1；否则可以调低。

在实际项目中，我强烈建议你在自己的数据集上用网格搜索（Grid Search）来调优这两个参数，而不是直接使用默认值。

💡 小贴士：中文分词与自定义词典
BM25在中文场景下的表现高度依赖分词质量。如果使用 jieba 分词，务必加载领域自定义词典。例如，在医疗领域，"心肌梗死"这个词如果被分成"心肌/梗死"虽然也能理解，但作为完整短语检索时，IDF权重会完全不同。一个常见的做法是：收集领域术语表，通过 jieba.load_userdict("med_terms.txt") 加载自定义词典，可以显著提升检索命中率。

5. Python实现BM25

我们来看一个简洁的BM25实现，理解其核心逻辑：

import math
from collections import Counter

class BM25:
    def __init__(self, corpus, k1=1.5, b=0.75):
        self.k1 = k1
        self.b = b
        self.corpus = corpus
        self.doc_len = [len(doc) for doc in corpus]
        self.avgdl = sum(self.doc_len) / len(corpus)
        self.idf = self._compute_idf()

    def _compute_idf(self):
        import jieba
        df = {}
        total = len(self.corpus)
        for doc in self.corpus:
            tokens = set(jieba.lcut(doc))
            for word in tokens:
                df[word] = df.get(word, 0) + 1
        return {word: math.log(1 + (total - freq + 0.5) / (freq + 0.5))
                for word, freq in df.items()}

    def score(self, query, doc_index):
        doc = self.corpus[doc_index]
        doc_len = self.doc_len[doc_index]
        import jieba
        query_tokens = jieba.lcut(query)
        doc_tokens = Counter(jieba.lcut(doc))
        score = 0.0
        for token in query_tokens:
            if token not in self.idf:
                continue
            tf = doc_tokens.get(token, 0)
            if tf == 0:
                continue
            idf = self.idf[token]
            numerator = tf * (self.k1 + 1)
            denominator = tf + self.k1 * (1 - self.b + self.b * doc_len / self.avgdl)
            score += idf * numerator / denominator
        return score

    def search(self, query, top_k=5):
        scores = [(i, self.score(query, i)) for i in range(len(self.corpus))]
        scores.sort(key=lambda x: x[1], reverse=True)
        return scores[:top_k]

# 使用示例
corpus = [
    "自然语言处理是人工智能的重要分支",
    "深度学习在自然语言处理中取得了巨大成功",
    "BM25是一种经典的检索算法",
    "向量数据库用于存储和检索嵌入向量",
]
bm25 = BM25(corpus, k1=1.5, b=0.75)
results = bm25.search("自然语言处理算法", top_k=3)
for idx, score in results:
    print(f"文档{idx}: {corpus[idx]} (score: {score:.4f})")

6. SPLADE：学习式稀疏检索

传统BM25的问题在于它完全基于统计，无法理解语义。比如搜索"汽车"，包含"车辆"的文档不会被匹配到。SPLADE（Sparse Lexical and Expansion Model）通过Transformer模型来"学习"如何构建稀疏向量，兼具稀疏检索的高效和语义理解的能力。

SPLADE的核心思想是：对于一个输入文本，利用MLM（Masked Language Model）的预测分布来"扩展"词汇表上的权重。也就是说，输入"汽车"，它不仅会给"汽车"高权重，还会给"车辆"、"轿车"、"机动车"等语义相关词分配非零权重。最终输出仍然是一个稀疏向量（绝大多数维度为0），但语义覆盖面远超BM25。

7. BM25 vs 稠密检索对比

为了帮你更好地做技术选型，我整理了一张对比表：

对比维度 BM25（稀疏检索） 稠密检索（Dense Retrieval）
匹配方式 精确关键词匹配 语义相似度匹配
处理同义词 ✗ 无法处理 ✓ 天然支持
处理罕见术语 ✓ 优秀（如产品型号、代码） ✗ 容易丢失信息

踩坑故事：被"北京"坑了的一周

让我分享一个真实的教训。之前做一个法律文书检索项目，用户搜索"北京市朝阳区人民法院"，BM25居然把"南京市鼓楼区人民法院"排在了很靠前的位置。排查后发现，问题出在我们的分词上——jieba把"北京市"分成了"北京/市"，而"南京市"也含有"市"。加上我们的 idf 计算中"市"这个词极其常见，IDF很低，但"北京"和"南京"在文档中出现的频率差不多，导致分数相近。

解决方案是两方面的：第一，使用更细粒度的 n-gram，将"北京市"作为一个整体短语保留；第二，引入词性权重，给地名、人名等专有名词更高的权重。这之后检索准确率提升了12%。

8. BM25在RAG中的最佳实践

根据我的生产经验，以下几点非常重要：

  • 不要只用默认参数。用你自己的数据集做交叉验证调优 k1 和 b。
  • 分词词典要定制。领域术语、专有名词一定要加入自定义词典。
  • 考虑停用词。去除高频无意义词（"的"、"了"、"是"等）可以显著降低倒排索引的大小。
  • 与稠密检索结合。BM25和稠密检索不是二选一，而是互补关系——这也是我们第10章要讲的内容。
  • 对代码/型号类查询更友好。如果你的用户经常搜索"iPhone 15 Pro Max 256GB"这类精确短语，BM25是首选。

本章小结

稀疏检索以倒排索引为基础，通过BM25等算法实现高效的关键词匹配。BM25通过k1和b两个参数优雅地解决了词频饱和和文档长度归一化问题，至今仍是RAG系统中的核心组件。SPLADE等学习式稀疏检索模型则进一步模糊了稀疏与稠密检索的边界。
