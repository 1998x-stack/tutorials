第16章 主流求解器实战

大家好，我是你们的讲师。假设你是一家制造企业的运筹算法工程师，刚接到一个新项目：排产系统升级，要为新工厂做个排程引擎。你的第一个问题一定是——用什么求解器？Gurobi 太贵、CP-SAT 刚学完还不熟悉、SCIP 开源的性能够用吗？选求解器这件事，做对了事半功倍，选错了项目烂尾。今天这一章，我们来一次求解器的"全栈巡礼"——2026 年的求解器生态是什么样的，以及在实际调度项目中怎么做选型决策。

2018 年我参与过一个汽车零部件工厂的排程项目。起初我们基于 Gurobi 开发，License 花了大几十万，结果跑了两个月发现模型规模太大（几百个订单、几十台设备、数千个变量），Gurobi 的 MIP gap 一直收不拢。后来我尝试用 CP-SAT 重新建模，三天完成、两天调优，上线后排程时间从半小时缩短到 3 分钟。但这个项目的教训让我明白——没有"最好的求解器"，只有"最适合当前问题的求解器"。

求解器全景图（2026）

2026 年的求解器市场百花齐放。商业求解器每年都有大版本更新，开源求解器的性能差距也在不断缩小。我们先看看头部玩家的格局。

商业求解器：

  • Gurobi 11—— 迄今最广泛使用的商业 MILP 求解器。2025 年底发布的 11.0 版本引入了自动分解检测，能自动识别模型中的块对角结构并进行 Dantzig-Wolfe 分解。调度问题上表现优秀，尤其在带线性成本的 MILP 模型上。但 License 费用高昂（学术免费，商业授权年费数万至数十万美元）。
  • CPLEX 22—— IBM 的老牌求解器，20 年间一直是行业标杆。近年更新速度放缓，但在大规模 LP 和网络流问题上仍然很强。支持 OPL 建模语言，适合与 IBM 生态整合的项目。
  • Hexaly 14.5（原名 LocalSolver）—— 基于局部搜索的商业求解器，在高度约束的组合优化问题（如车辆路径、排班）上表现非常突出。尤其擅长"好解快出"场景：1 秒内给出的解往往比 Gurobi 跑 30 秒给的还好。Hexaly 14.0 加入了对决策变量的自动邻域定义功能。
  • COPT 7.0—— 国产求解器的代表，由上海杉树科技开发。2025 年在 LP/MILP 多项 benchmark 指标上取得了全球第一（下文会详细讲）。性价比极高。

开源求解器：

  • OR-Tools（CP-SAT + HiGHS）—— OR-Tools 不仅包含 CP-SAT，还在 2023 年集成了 HiGHS 作为 MILP 后端。CP-SAT 在调度问题上已经达到了商业级水平，而 HiGHS 作为纯 MILP 求解器，性能大约相当于 Gurobi 70-80% 的水平。关键是完全免费。
  • SCIP 9.2—— 学术界的宠儿，Constraint Integer Programming 的典范。SCIP 是个"大杂烩"：它同时支持 MILP、CP 和 SAT 技术。由于不依赖商业线性库（它用自家的 SoPlex），纯数值性能略逊于 Gurobi，但其模型表达能力极强。学术界很多新算法研究都在 SCIP 上做原型验证。
  • GLPK—— GNU 项目的一部分，纯 LP/MILP 求解器。性能差，但胜在干净、稳定、无依赖。适合教学和简单场景，不适合工业生产。

一个容易被忽视的选项——HiGHS：2023 年 HiGHS 赢得 COIN-OR Cup 后快速普及。它的 pure LP 求解已经非常接近商业水平，MILP 方面也在快速追赶。如果你有 MILP 需求但预算有限，HiGHS 是 CP-SAT 之外的绝佳组合。OR-Tools 内部已经集成了 HiGHS，所以你可以通过 Solver("HiGHS") 直接调用——不需要额外安装。

OR-Tools 深度使用

OR-Tools 最大的优势是"一体化"——它把 CP-SAT、MIP Solver、Routing Solver 整合在一个框架里。更妙的是，你可以用同一套模型切换求解后端：

from ortools.math_opt.python import mathopt

# Model 定义——与求解器无关
model = mathopt.Model(name="job_shop_demo")

# 变量
x = {j: model.add_integer_var(0, 1000, f"x_{j}") for j in range(10)}
y = {j: model.add_binary_var(f"y_{j}") for j in range(10)}

# 约束
model.add_linear_constraint(sum(x[j] for j in range(10)) <= 500)

# 目标
model.minimize(sum(x[j] + 10 * y[j] for j in range(10)))

# ---- 求解后端切换 ----
# 方法1: 用 Gurobi（需要 license）
params = mathopt.SolveParameters()
result = mathopt.solve(model, mathopt.SolverType.GUROBI, params)
print(f"Gurobi objective: {result.objective_value()}")

# 方法2: 用 CP-SAT（免费，但不擅长线性问题）
result = mathopt.solve(model, mathopt.SolverType.CP_SAT, params)
print(f"CP-SAT objective: {result.objective_value()}")

# 方法3: 用 HiGHS（免费，纯 MILP）
result = mathopt.solve(model, mathopt.SolverType.HIGHS, params)
print(f"HiGHS objective: {result.objective_value()}")

这个 math_opt 模块是 OR-Tools 近几年的重要创新。它提供一个solver-agnostic 的建模接口，底层自动帮你做模型转换——用 Gurobi 时自动生成 LP 格式、用 CP-SAT 时自动转化全局约束、用 HiGHS 时自动做原始-对偶转换。一套代码，切换三个后端，这在工业调度项目中非常有用——你可以先用 CP-SAT 快速出效果，需要更紧的松弛下界再换 Gurobi，而代码几乎不需要改动。

OR-Tools 的 Routing Solver 也是一个强大的工具——它专门为 VRP（车辆路径问题）做了高度优化。如果你要解决配送调度或多车路径问题，Routing Solver 通常比通用 MILP 或 CP-SAT 更快、更可靠。

各求解器在调度问题上的 Benchmark（2026）

光说不练假把式。我们来看看 2026 年的公开 benchmark 数据，选取三种典型调度问题：CVRPTW（带时间窗的车辆路径）、RCPSP（资源约束项目调度）、JSP（作业车间调度）。

看了这张表，你应该能总结出几条核心规律：

  • 对于JSP 和 RCPSP（高度约束的调度问题），CP-SAT 是当之无愧的冠军——免费、高性能、故障率低。
  • 对于CVRPTW（带线性成本的路径优化），Gurobi 和 Hexaly 更优——因为目标函数中距离成本是线性的，MILP 的松弛比较紧。
  • 商业求解器的核心价值不在于性能，而在于可靠性和支持。Gurobi 的文档、API 设计、社区生态是任何开源求解器都难以匹敌的。

一个值得警惕的现象：我们曾经接手过一个客户的项目，他们的排程系统用 Gurobi 做求解器，模型写得很"漂亮"——全部是线性约束，连工具限制都用大 M 法制成了线性。但问题在于，当问题规模扩大到 50 个工件 × 10 台机器时，模型中的大 M 约束数量超过了 5000 条，线性松弛几乎完全退化。Gurobi 在 600 秒内找不到一个可行解。后来我们用 CP-SAT 重写，模型更小（不再需要大 M），15 秒内给出了解。求解器选型不是越贵越好，关键要看问题结构。

COPT — 国产求解器的崛起

聊一聊国产求解器的标杆——COPT（Cardinal Optimizer）。杉树科技开发的 COPT 在近几年的 Mittelmann 和 Hans Mittelmann 的 LP benchmark 排行榜上表现抢眼。在 2025 年底发布的 benchmark 结果中，COPT 在 LP 求解的 11 项指标上排名全球第一，包括 LP 求解速度、内存效率、大规模问题处理等。

为什么会这样？一个重要原因是 COPT 团队对稀疏矩阵运算和预处理做了深度优化。很多实际问题的约束矩阵是极度稀疏的（非零元占比不到 1%），COPT 在这方面通过"超节点"技术大幅提升了内存局部性。另一个原因是中国本土的商业支持——你可以直接和开发团队沟通，Bug 修复周期以天计而非以季度计。

COPT 与 Gurobi 的性价比对比：

对比项 Gurobi 11 COPT 7.0
LP 求解速度 参考基准 平均快 10-15%
MILP 求解速度 参考基准 约 90-95%
大规模问题（>100 万变量） 略优 优秀
商业授权年费 $30K - $100K ¥10W - ¥30W
技术支持 邮件 + 社区 中文 + 微信 + 驻场

如果你的项目有国产化要求、或者预算有限但需要接近商业级的表现，COPT 是非常好的选择。不过需要提醒的是，COPT 目前最强的领域在 LP/MILP，在组合优化和约束规划领域尚未发力，所以如果你的项目偏调度，CP-SAT 可能仍然是更好的主求解器。

求解器选型决策流程：我在实际项目中总结了一个简单有效的四步法。第一步，看问题类型——调度/排班类优先试 CP-SAT，线性成本优化优先试 Gurobi/COPT，路径优化优先试 Hexaly/Routing Solver。第二步，按 80% 置信度选一个求解器做原型，控制在 2 天内搭建完成。第三步，用真实数据跑 benchmark——别用公开数据集，用你自己的业务数据。第四步，如果 gap 不合格，换另一个求解器再试。记住，先跑起来再优化——不要花两周选求解器，要花两周做试验。

好了，这一章我们从全景图出发，逐一分析了 2026 年主流求解器的优劣势和适用场景。你手上有了一张清晰的选型地图，也知道如何用 OR-Tools 的 math_opt 模块做到"一套代码换后端"。但纸上得来终觉浅——下一章，我们把目光转向真实世界的工程案例：半导体晶圆厂调度、港口码头调度、手术室排程。看看在这些让人"头大"的工业场景中，我们在前 16 章学到的工具到底是怎么被用到极致的。
