开篇：认证≠授权——90%的安全事故源于混淆两者

我曾收到过一个安全漏洞报告。一个电商平台的普通用户，在浏览器控制台里改了请求中的 user_id 参数，就能查看到其他用户的订单详情。开发团队委屈地说："我们做了认证啊——用户必须登录才能访问订单接口。"问题就出在这里：他们确实验证了"你是谁"（认证），但没有验证"你是不是这个订单的主人"（授权）。

认证（Authentication）和授权（Authorization），这两个词看起来只差几个字母，但混淆它们的代价极其高昂。认证是确认身份——"你是你所说的那个人"；授权是确认权限——"你有权限做这个操作"。一个完整的系统必须同时做好这两件事。

记住这个公式：认证 + 授权 + 审计 = 安全的访问控制。

🔑 一句话区分：认证是"你是谁"，授权是"你能干什么"。安检时出示身份证是认证，头等舱乘客才能进入休息室是授权。

Session vs Token 认证

在 Web 应用中，认证信息需要在每次请求中传递。最主流的两种方式是 Session 和 Token。它们没有绝对的优劣，取决于你的应用场景。

维度 Session 认证 Token 认证（JWT）
存储位置 服务端内存/Redis，客户端只存 Session ID 客户端持有完整的 Token，服务端只需验证签名
扩展性 需要集中式的 Session 存储（多实例共享 Redis） 天然无状态，任何服务器都能验证 Token 签名
撤销能力 直接删除服务端 Session，立即生效 需要黑名单机制（或依赖短生命周期）
安全边界 Session ID 泄露后，攻击者只能临时借用会话 JWT 泄露后，攻击者在有效期内可完全冒充用户
适用场景 传统服务端渲染应用，需要强会话管理 分布式系统、移动端、微服务、单页应用
🧠 一个简单的判断标准：如果你的服务是单体部署或者只有少数几台机器，Session 完全够用。如果你有十几个微服务，或者你的 API 要给第三方使用，Token 认证更合适。在现代后端开发中，Token 认证（尤其是 JWT）已经是主流选择。

JWT 深入：结构、签名与轮换策略

JWT（JSON Web Token）是目前最流行的 Token 格式。一个 JWT 由三部分组成，用点号分隔：header.payload.signature。

Header 包含签名算法和 Token 类型；Payload 包含声明（claims），比如用户 ID、角色、过期时间；Signature 是对前两部分的签名，确保 Token 没有被篡改。

一个典型的 JWT 是这样的：

// Header（Base64 URL 编码）
{
  "alg": "RS256",    // 推荐使用非对称算法 RS256 或 ES256
  "typ": "JWT"
}

// Payload（Base64 URL 编码）
{
  "sub": "user_12345",       // 用户标识
  "role": "admin",           // 角色
  "iat": 1751600000,         // 签发时间
  "exp": 1751603600,         // 过期时间（1小时后）
  "jti": "uuid-xxxxx"        // Token 唯一 ID，用于撤销
}

// Signature = RS256(base64UrlEncode(header) + "." + base64UrlEncode(payload), privateKey)

// 短 Token + 长 Refresh Token 轮换策略示意
// 1. 登录成功后，返回 access_token（15分钟过期）和 refresh_token（7天过期）
// 2. 前端在 access_token 过期前，使用 refresh_token 获取新的 access_token
// 3. refresh_token 使用一次就作废，返回新的 refresh_token（轮换）

POST /auth/refresh
Authorization: Bearer <refresh_token>

Response:
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",  // 新的 refresh token，旧的作废
  "expires_in": 900
}
⚠️ 一个常见的 JWT 误区：很多人以为 JWT 是"加密的"，所以把密码、手机号等敏感信息直接放在 Payload 里。但 JWT 的 Signature 只保证"未被篡改"，不保证"不可见"——Payload 只是 Base64 编码，任何人解码就能看到内容。绝对不要把密码明文放在 JWT 里。

OAuth 2.0 授权码流程详解

OAuth 2.0 不是认证协议，而是一个授权协议。它的设计初衷是让用户授权第三方应用访问自己在某个服务上的数据，而不需要把密码交给第三方。比如"使用微信登录某个 App"，背后的流程就是 OAuth 2.0 的授权码模式。

授权码流程（Authorization Code Flow）是最完整的 OAuth 2.0 流程，也是 2026 年绝大多数场景下的推荐方案：

  1. 用户点击"第三方登录"，前端将用户重定向到授权服务器的登录页（带上 client_id、redirect_uri、response_type=code）。
  2. 用户输入密码确认授权，授权服务器验证身份后，通过重定向将授权码（Authorization Code）发送回 redirect_uri。授权码是一次性的，且有效期很短（通常几分钟）。
  3. 后端服务器拿着授权码，加上 client_secret，向授权服务器的 Token 端点换取 Access Token。这一步在服务器端完成，授权码和密钥都不会暴露给浏览器。
  4. 授权服务器返回 Access Token（也可能同时返回 Refresh Token）。之后，客户端就可以用这个 Token 访问受保护的资源。

2026 年的最佳实践已经普遍采用 PKCE（Proof Key for Code Exchange） 扩展，即使是移动端或单页应用（无法安全保存 client_secret）也能安全使用授权码流程。PKCE 的核心是客户端在第一步生成一个 code_verifier 和对应的 code_challenge，第三步再用 code_verifier 来证明"我就是发起流程的那个人"。这防止了授权码被截获后的重放攻击。

📌 2026 年推荐方案：单页应用/移动端 → 授权码 + PKCE + 短 Token + Refresh Token 轮换。这是当前安全界公认的最佳实践组合。有现成的开源库（如 Ory Hydra、Keycloak、Auth0）可以直接部署或使用，不要自己实现 OAuth 2.0 服务器。

RBAC 权限模型设计

授权的最经典模型是 RBAC（Role-Based Access Control，基于角色的访问控制）。它的核心思想是将权限授予角色，然后将角色授予用户——用户不直接拥有权限，而是通过角色间接获得。

一个典型 RBAC 模型包含以下要素：

概念 说明 示例
用户（User） 系统的使用者 alice@company.com
角色（Role） 权责的集合 admin, editor, viewer
权限（Permission） 对某个资源的具体操作 order:read, order:write, user:delete
资源（Resource） 被保护的对象 /v1/orders, /v1/users

实际操作中，权限常用"资源:操作"的命名规范，比如 order:read 表示读取订单的权限。角色可以继承，比如 admin 可以包含 editor 的所有权限再加上管理权限。这种继承关系可以显著降低权限管理的复杂度。

💡 简单 vs 灵活：如果你的系统只有两三种角色（普通用户、管理员），可以直接在代码里用 if role == "admin" 判断，不需要引入完整的 RBAC 框架。RBAC 的最大价值体现在角色种类多、权限粒度细、且需要动态管理的场景。

实战：实现一个安全的登录系统

把前面所有的知识串起来，一个现代后端应用的认证系统大致是这样的：

// 登录端点（Go + JWT 伪代码）
func Login(w http.ResponseWriter, r *http.Request) {
    var req LoginRequest
    json.NewDecoder(r.Body).Decode(&req)

    // 1. 验证用户名密码
    user, err := authenticateUser(req.Email, req.Password)
    if err != nil {
        http.Error(w, "邮箱或密码错误", http.StatusUnauthorized)
        return
    }

    // 2. 生成 Access Token（15分钟）
    accessToken, err := generateJWT(map[string]interface{}{
        "sub":  user.ID,
        "role": user.Role,
    }, 15*time.Minute)

    // 3. 生成 Refresh Token（7天）
    refreshToken := generateRefreshToken()

    // 4. 存储 Refresh Token 的哈希到数据库
    storeRefreshTokenHash(user.ID, hash(refreshToken), 7*24*time.Hour)

    // 5. 返回 Token 对
    json.NewEncoder(w).Encode(map[string]interface{}{
        "access_token":  accessToken,
        "refresh_token": refreshToken,
        "expires_in":    900,
    })
}

// Auth 中间件
func AuthMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        token := extractToken(r.Header.Get("Authorization"))
        // 验证签名、检查过期时间、检查是否被撤销
        claims, err := verifyJWT(token)
        if err != nil {
            http.Error(w, "未授权", http.StatusUnauthorized)
            return
        }
        // 将用户信息注入到请求上下文
        ctx := context.WithValue(r.Context(), "user", claims)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

这是一个简化版本，但涵盖了核心流程：密码校验、Token 生成、中间件校验、以及 Refresh Token 轮换的支持。在实际生产系统中，你还需要考虑密码的哈希存储（使用 bcrypt 或 argon2）、登录频率限制、异常登录检测等因素。

认证体系建好之后，你的 API 终于有了"门锁"。但用户数据放哪里呢？绝大多数业务系统的核心都是数据，而数据的归宿是数据库。下一章我们就进入关系型数据库的世界——从表设计到索引优化，从事务隔离到死锁排查，这些是后端工程师每天都要面对的事情。
