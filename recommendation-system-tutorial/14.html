第14章 召回模型（下）：多兴趣与图召回

上一章我们详细讨论了双塔模型和向量检索——用两个网络将用户和物品映射到同一个向量空间，通过近似最近邻搜索完成召回。双塔模型简洁、高效、可扩展，是当前工业界使用最广泛的召回范式。

但双塔模型有一个不容忽视的短板：一个用户只有一个embedding。

想想看——你周末可能想打羽毛球、看科幻电影、吃日料。这三个兴趣差异巨大，但双塔模型只能用一个向量来概括所有这些兴趣。结果就是这个向量变成了"平均脸"——对每类兴趣都"沾一点边但都不太精准"。本章要介绍的MIND（多兴趣召回）和PinSAGE（图召回）从两个不同的方向试图解决这个问题。

1. MIND：一个用户，多个兴趣

2019年，阿里巴巴提出了MIND（Multi-Interest Network with Dynamic Routing）。MIND的核心思想非常直观：既然一个用户有多个不同的兴趣，那就让模型输出多个兴趣向量，每个向量代表用户兴趣的一个"方面"。

MIND使用了一种源自Capsule Network（胶囊网络）的技术——动态路由（Dynamic Routing）。在MIND中，用户的历史行为（点击过的物品embedding）被看作"底层特征"，而兴趣向量被看作"高层特征"。动态路由算法迭代地将底层特征分配到不同的高层胶囊中——每个胶囊最终代表一个独立的兴趣。

具体过程是这样的：

  1. 用户有 n 个历史行为（看过的商品/视频），每个行为对应一个 d 维embedding
  2. 设定要提取 K 个兴趣向量（K是超参数，比如4或8）
  3. 通过动态路由算法，将 n 个行为embedding分配到 K 个兴趣胶囊中
  4. 每个胶囊输出一个向量 v_k——代表用户的第 k 种兴趣

训练时，每个样本中的正样本物品会与"最匹配"的那个兴趣向量计算内积损失。也就是说，模型会自动选择最相关的那个兴趣来"解释"用户的这次点击。不同的物品会被分配到不同的兴趣胶囊中——这鼓励了兴趣的多样化。

在线推理时，MIND为每个用户生成 K 个兴趣向量，对每个向量做一次向量检索——共 K 次检索，结果合并去重。

深入理解：MIND的动态路由可以类比于"软聚类"。传统聚类（如K-Means）是硬分配——每个行为只属于一个簇。动态路由是软分配——每个行为以不同的程度被分配到多个兴趣胶囊。这和人的真实兴趣结构更吻合：一个用户对"手机"的兴趣可能同时涉及"电子产品"和"通信设备"两个兴趣维度。MIND的K值是一个重要的权衡：K太小不够区分兴趣多样性，K太大会产生很多冗余的空胶囊。阿里巴巴在论文中使用K=4~8。

但MIND也有它的局限性：动态路由的计算量随行为序列长度线性增长，且训练不太稳定（需要多次迭代路由）。后续的一些工作尝试用Self-Attention或CNN来替代动态路由，在保持多兴趣表达能力的同时提升训练稳定性。

2. 图召回：DeepWalk到PinSAGE

"你关注的人也关注了它"——这是社交推荐中最古老也最有效的直觉之一。图召回（Graph-based Recall）正是将这种直觉形式化为数学模型。

在推荐系统中，用户和物品天然构成了一个二部图（Bipartite Graph）：用户节点和物品节点，边代表交互（点击、购买、收藏）。图召回的目标是在这个图上"游走"，找到与目标用户语义相似的其他节点。

DeepWalk（2014年）是第一个将图神经网络思想引入推荐的工作。它的做法非常朴素：在用户-物品二部图上做随机游走（Random Walk），生成的路径序列当作"句子"，然后通过Word2Vec学习节点embedding。DeepWalk本质上是在图的局部邻域中做"共现统计"——在随机游走中经常一起出现的节点，它们的embeding向量就应该相似。

Node2Vec（2016年）在DeepWalk的基础上引入了biased random walk——可以控制游走是更倾向于BFS（广度优先探索邻居结构）还是DFS（深度优先探索远距离节点）。这个改进让Node2Vec可以同时捕获图的结构同质性和角色等价性。

但DeepWalk和Node2Vec有一个共同的缺陷：它们是直推式（Transductive）的——只能为训练时见过的节点生成embedding，无法泛化到新节点。在工业推荐系统中，每天都有大量新用户和新物品加入，直推式模型需要频繁重训，代价高昂。

PinSAGE（2018年，Pinterest） 是工业级图召回的代表作。Pinterest的核心创新在于：

  • 归纳式（Inductive）：通过GraphSAGE卷积，可以在新节点出现时即时生成embedding，无需重训
  • 重要性池化（Importance Pooling）：用随机游走的访问频次作为节点的重要性权重，取代对邻居的简单平均
  • 大规模工程化：在Pinterest 20亿节点、200亿边的图上实现了端到端训练和在线serving

PinSAGE的训练使用了多任务学习——同时优化"用户点击"、"用户收藏"等多个目标，让embedding同时携带多种语义信号。

工业小贴士：图召回在实际落地时，最大的工程挑战是图的更新和维护。PinSAGE的做法是"每天离线全量更新一次图结构 + 增量更新高活跃节点的embedding"。对于大多数中小团队，我不建议自研图召回系统——用DeepWalk或Node2Vec在每周更新的图快照上训练，就足以获得80%以上的效果，而工程成本可能不到PinSAGE的十分之一。

3. 多路召回策略设计

不管你用了多强的模型，工业推荐系统永远不会只依赖一路召回。为什么？因为世界上不存在一个全能召回通道——每一路召回都有自己的盲区，多路并行才能互相覆盖盲区。

以下是典型的召回通道及其特点：

召回通道 基本原理 优势 劣势
ItemCF召回 用户喜欢的物品的相似物品 简单、可解释、实时更新 覆盖面窄、同质化严重
双塔向量召回 用户向量 × 物品向量内积 覆盖面广、泛化能力强 单向量表达力有限
多兴趣召回 K个兴趣向量 × 物品向量 兴趣维度丰富 训练复杂、K值难调
图召回 图结构上的传播与游走 捕获社交/共现关系 图构建和维护成本高
热门/流行召回 全局热门、类别热门 冷启动稳定、覆盖长尾 个性化弱
标签/内容召回 基于标签匹配或倒排索引 精准、可控制 依赖人工标签体系
实时行为召回 用户刚点击/收藏的相似物品 反应快、捕捉即时兴趣 噪声大、波动剧烈

多路召回的权重分配有两种主要策略：

  • 静态权重：每路召回固定取Top-N个结果（比如每路取200个），合并后去重送给下游。权重通过经验或离线实验确定。
  • 动态权重：每路召回的Top-N数量根据用户状态动态调整——活跃用户多一些向量召回和实时行为召回，冷启动用户多一些热门召回和标签召回。

召回天花板公式：假设有 m 路召回，每路的独立召回率为 R_i（即该路召回了全部"好"物品的比例），且各路召回相互独立，则总的召回率为：R_total = 1 - ∏(1 - R_i)。计算一下：如果有4路召回，每路的R_i=30%，则 R_total = 1 - (0.7)^4 ≈ 76%。再加一路30%的召回，R_total = 1 - (0.7)^5 ≈ 83%。每增加一路召回，边际收益递减得非常快。这告诉我们，与其无限增加召回通道，不如把精力放在提升每路召回本身的精度上。

4. 召回融合与去重

多路召回的结果合并是一个看似简单、实则暗藏玄机的问题。主要需要处理两件事：去重和融合。

去重（Deduplication）：不同路的召回结果当然会有重叠——热门物品可能同时出现在向量召回和热门召回中。常见的去重方式包括：

  • 基于集合（Set-based）：用一个集合记录已经见过的物品ID，后续重复的直接跳过
  • Bloom Filter：当物品量极大时，用Bloom Filter做概率性去重（允许小概率的误判——宁可漏掉一次重复，也要保证查询速度）

融合（Merge）：去重之后，需要决定不同路召回的物品在最终列表中的顺序。常见方案有：

  • 等权交叉：A路第1名、B路第1名、C路第1名、A路第2名、B路第2名……这种方式保证了多样性
  • 加权评分：每路召回给一个置信度分数，乘以原始相似度后作为最终排序依据
  • 级联式：先把所有结果混在一起，再用一个轻量级粗排模型重新排序

下面是一个多路召回合并的简化实现：

import hashlib
from typing import List, Dict

class RecallMerger:
    """简单的多路召回融合器"""

    def __init__(self, bloom_size: int = 1_000_000):
        self.seen = set()  # 实际项目用Bloom Filter

    def _item_key(self, item_id: int, channel: str) -> str:
        return f"{channel}:{item_id}"

    def merge_and_dedup(
        self,
        recall_results: Dict[str, List[Dict]],
        channel_weights: Dict[str, float] = None,
        top_k: int = 500
    ) -> List[Dict]:
        """
        recall_results: {"vector_recall": [{"id":1, "score":0.9}, ...],
                         "itemcf_recall": [{"id":2, "score":0.8}, ...], ...}
        channel_weights: {"vector_recall": 1.0, "itemcf_recall": 0.8, ...}
        top_k: 最终保留的候选数
        """
        channel_weights = channel_weights or {}
        merged = []

        for channel, items in recall_results.items():
            weight = channel_weights.get(channel, 1.0)
            for item in items:
                key = self._item_key(item["id"], channel)
                if key in self.seen:
                    continue  # 去重
                self.seen.add(key)
                item["score"] = item["score"] * weight  # 加权
                item["channel"] = channel
                merged.append(item)

        # 按加权分数排序
        merged.sort(key=lambda x: x["score"], reverse=True)
        return merged[:top_k]

# 使用示例
merger = RecallMerger()
results = {
    "vector_recall": [{"id": 101, "score": 0.92}, {"id": 102, "score": 0.85}],
    "itemcf_recall": [{"id": 101, "score": 0.78}, {"id": 103, "score": 0.72}],
    "hot_recall":    [{"id": 999, "score": 0.60}, {"id": 104, "score": 0.55}],
}
weights = {"vector_recall": 1.0, "itemcf_recall": 0.8, "hot_recall": 0.3}
final = merger.merge_and_dedup(results, weights, top_k=500)

这个实现虽然简单，但它抓住了多路召回融合的核心逻辑：去重 + 加权合并。在工程落地时，每一路召回的score可能不在同一个量级上——向量召回的内积可能在-1到1之间，ItemCF的Jaccard相似度在0到1之间，热门召回的热度分数可能是百万级的。因此，分数归一化（score normalization）是融合之前必不可少的一步。常见的做法是对每路召回的分数做Z-Score标准化或Min-Max缩放。

进阶建议：在实际系统中，召回融合不应该是"录死"的。建议把你的召回融合策略变成一个可配置的组件——通道列表、每路取N个、每路权重、归一化方式，都通过配置文件控制。这样，推荐算法的同学可以不经过工程发版就调整召回策略，快速做A/B实验。一个设计良好的召回融合框架，可以让算法团队的生产力提升一个数量级。

回顾一下，从第8章到第14章，我们走完了一条从特征交叉到排序模型再到召回模型的完整技术路线。你看到了推荐系统的模型如何从朴素LR进化到注意力机制的DIN，再从单一向量召回进化到多兴趣和图召回。但"召回 + 排序"只是推荐系统的主干，在实际工业系统中还有大量问题需要解决——精排模型如何更好地优化业务指标？当同时优化多个目标（CTR、CVR、时长）时怎么办？精排结果如何重排以保证多样性？下一章开始，我们将进入进阶专题，首先来看——排序模型的精进之路。
