链表的本质与分类

大家好，我是你们的讲师。今天我们来攻克 LeetCode 面试中的"守门员"——链表操作。为什么说它是守门员？因为链表题考的不是复杂的算法，而是最基础、最朴素的指针操作能力。很多候选人做 Medium 动态规划题行云流水，但一到链表反转就手忙脚乱、指针乱飞。面试官正是用这类题来判断你的代码基本功是否扎实。

先回顾链表的基本概念。单链表（Singly Linked List）的每个节点包含一个值和一个指向下一个节点的指针。它的查找是 O(n)，但插入和删除在已知前驱节点的情况下是 O(1)。双链表（Doubly Linked List）多了一个前驱指针，可以双向遍历，但维护成本更高。循环链表（Circular Linked List）的尾节点指向头节点，常用于约瑟夫环问题。

class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

这个定义在 LeetCode 上贯穿始终。所有链表操作题都在和 next 指针打交道——链表的全部精髓，就在于怎么正确地重新排列这些指针。

核心思维：链表操作 = 指针重排。每次操作前，先问自己三个问题：① 哪些指针会丢失？② 是否需要临时保存？③ 修改顺序是什么？

虚拟头节点技巧

在链表操作中，最令人头疼的往往不是逻辑本身，而是边界条件——当头节点需要被修改时怎么办？虚拟头节点（Dummy Node）就是为此而生的。它让我们统一对待所有节点，不需要为头节点写特殊逻辑。

反转链表（LeetCode 206）

反转链表是链表面试题中出现频率最高的题目，没有之一。面试官喜欢它的原因是：代码短（只需要几行），但指针操作的细节非常多。

def reverse_list(head):
    prev = None
    curr = head
    while curr:
        next_temp = curr.next  # 保存下一个节点
        curr.next = prev       # 反转指针
        prev = curr            # prev 前移
        curr = next_temp       # curr 前移
    return prev

关键点在于 next_temp = curr.next 这一步。如果不先保存下一个节点，curr.next = prev 执行后我们就永远丢失了对原链表后续部分的引用——这就是著名的"指针丢失"问题。迭代翻转的四个步骤可以记忆为："保存→反转→前移→重复"。

合并两个有序链表（LeetCode 21）

这道题完美展示了 dummy node 的威力。我们创建一个虚拟头节点作为新链表的起点，然后不断将两个链表中较小的节点接到后面。最后返回 dummy.next 即可。

def merge_two_lists(l1, l2):
            dummy = ListNode(0)
            curr = dummy
            while l1 and l2:
                if l1.val < l2.val:
                    curr.next = l1
                    l1 = l1.next
                else:
                    curr.next = l2
                    l2 = l2.next
                curr = curr.next
            # 接上剩余部分
            curr.next = l1 if l1 else l2
            return dummy.next

看到 dummy 了吗？如果没有它，我们就要单独处理第一个节点谁来接的问题——代码会多出一堆 if-else。这就是虚拟头节点哲学的体现：用一个无关紧要的节点，换来了代码的整洁和逻辑的统一。

经验之谈：只要涉及"构建新链表"或者"可能会修改头节点"，先用 dummy node。这应该成为你的肌肉记忆。

快慢指针与环检测

快慢指针（Floyd's Algorithm）是链表问题中最优雅的 pattern 之一。它的核心思想是：两个指针以不同速度遍历链表，利用速度差来检测特定条件。

判断链表是否有环（LeetCode 141）

用两个指针，slow 每次走一步，fast 每次走两步。如果链表有环，fast 最终会追上 slow（二者相遇）；如果无环，fast 先到达末尾。

def has_cycle(head):
            slow = fast = head
            while fast and fast.next:
                slow = slow.next
                fast = fast.next.next
                if slow == fast:
                    return True
            return False

为什么快慢指针一定能相遇？数学上可以证明：假设环长为 L，当 slow 进入环时，fast 已经在环内距离 slow 若干步。每次移动，二者的距离缩短 1 步（因为 fast 比 slow 多走 1 步），所以最多 L 步后必然相遇。

环的入口节点（LeetCode 142）

这道题是 141 的进阶版——不仅判断是否有环，还要找出环的入口。解法需要一点数学推导：快慢指针相遇后，把一个指针放回头节点，然后两个指针都每次走一步，再次相遇的位置就是环的入口。

def detect_cycle(head):
            slow = fast = head
            while fast and fast.next:
                slow = slow.next
                fast = fast.next.next
                if slow == fast:
                    # 找到环后，查找入口
                    slow = head
                    while slow != fast:
                        slow = slow.next
                        fast = fast.next
                    return slow
            return None

这个结论的推导过程很有趣（头节点到环入口的距离 = 相遇点到环入口的距离），建议大家去 LeetCode 142 的官方题解看完整证明。

链表的中间节点（LeetCode 876）

快指针走两步，慢指针走一步，快指针到末尾时慢指针正好在中点。这个技巧在"重排链表"（143）等题目中会用到。

def middle_node(head):
            slow = fast = head
            while fast and fast.next:
                slow = slow.next
                fast = fast.next.next
            return slow

复杂链表操作

K个一组反转链表（LeetCode 25）

这是反转链表的"终极升级版"——不是反转整个链表，而是每 K 个节点为一组进行反转。这道题是 Hard 难度，但把它拆解后你会发现：它就是"反转链表"+"递归/循环"的组合。

def reverse_k_group(head, k):
            # 先检查是否有 k 个节点
            count = 0
            curr = head
            while curr and count < k:
                curr = curr.next
                count += 1
            if count < k:
                return head

            # 反转前 k 个节点
            prev = None
            curr = head
            for _ in range(k):
                next_temp = curr.next
                curr.next = prev
                prev = curr
                curr = next_temp

            # 递归处理后续链表
            head.next = reverse_k_group(curr, k)
            return prev

这道题的思路是：先找到第 K 个节点，反转前 K 个，然后递归处理剩下的。注意 head.next = reverse_k_group(curr, k) 这行——原来第一个节点反转后变成了最后一个，它的 next 指向下一组的反转结果。

重排链表（LeetCode 143）

给定链表 L0 → L1 → … → Ln-1 → Ln，要求重排成 L0 → Ln → L1 → Ln-1 → …。这个题是三个技巧的组合：找中点 + 反转后半 + 交叉合并。

def reorder_list(head):
            if not head or not head.next:
                return
            # 1. 找中点
            slow = fast = head
            while fast and fast.next:
                slow = slow.next
                fast = fast.next.next
            # 2. 反转后半部分
            prev = None
            curr = slow.next
            slow.next = None  # 断开链表
            while curr:
                next_temp = curr.next
                curr.next = prev
                prev = curr
                curr = next_temp
            # 3. 交叉合并
            first, second = head, prev
            while second:
                tmp1, tmp2 = first.next, second.next
                first.next = second
                second.next = tmp1
                first, second = tmp1, tmp2
注意：反转后半段后一定要把 slow.next 置为 None，断开前半段和后半段的连接。如果不做这一步，合并时会出现环形引用，形成死循环。我当初第一次写这道题时就忘了这一步，debug 了半小时才发现是"链表没断开"——这个错误非常隐蔽！

踩坑与总结

链表题的坑大多集中在指针操作上。我来说一个自己的"血泪史"。

有一次在面试中写反转链表，我一上来就写 curr.next = prev，然后才想起来保存 next——不出意外，我丢失了链表的后续部分。面试官看着我抓耳挠腮的样子，善意地提醒了一句"要不先画个图？"。这件事之后我养成了一个习惯：任何涉及指针修改的代码，先在纸上画出修改前后的指针状态。哪怕只是简单的反转链表，画图也能帮你发现那些"以为对了但实际错了"的细节。

另一个常见问题：环形引用（Cycle）。一旦链表中出现环，多数遍历操作都会陷入死循环。如果你提交 LeetCode 时出现 Time Limit Exceeded，第一反应应该是检查是否产生了环。常用的调试方法是在遍历时加入一个计数器（比如最大循环 1000 次），或者在本地打印节点值。

问题类型 核心技巧 时间复杂度 空间复杂度
反转链表 迭代：prev/curr/next 三指针 O(n) O(1)
合并/构建链表 Dummy Node 简化头节点处理 O(n) O(1)
环检测 / 找中点 快慢指针（步长 1:2） O(n) O(1)
分组反转 / 重排 分解为子问题 + 指针重连 O(n) O(1) / O(n/k)
相交链表 / 公共节点 双指针走完各自链表后互换 O(n) O(1)
刷题建议：链表是所有数据结构的"指针基本功"。我强烈建议你熟练掌握反转链表(206)、环形链表(141/142)、合并有序链表(21)这三道题——它们是很多 Hard 题的基础构件。每次写完代码后，用空链表和单节点链表验证边界条件。

链表操作的核心就两件事：掌握指针重排的顺序，用好 dummy node 简化逻辑。下一章我们将进入栈与队列的世界——这两种结构虽然底层实现简单，但搭配单调栈等技巧后能解决非常多看似复杂的问题。我们下一章见！
