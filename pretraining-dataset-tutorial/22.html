开篇：两种"不讲人话"的数据

大家好，我是你们的讲师。在大模型的世界里，代码和数学是两种非常特殊的数据——它们不讲人话。自然语言追求的是流畅、冗余、容错；而代码追求精确、无歧义、可执行；数学追求严谨、符号化、可证明。这两种数据在预训练中的重要性已经被反复证明——增加代码和数学数据的配比，可以直接提升模型的推理能力，这是行业内几乎公开的秘密。

但正因为它们特殊，它们的清洗和处理方式也和普通文本完全不同。2014 年我还在读研时，一个同学兴冲冲地从 GitHub 上爬了几百万个 Python 文件来训练代码模型，结果发现训练出来的模型生成的代码里到处都是版权声明（"MIT License"的开头文本被当成了代码的一部分）。这就是我们这一章要解决的问题——如何为模型准备干净、安全、有效的代码和数学训练数据。

22.1 代码数据解析：从原始文件到结构化训练样本

原始代码文件和普通文本最大的区别在于：代码有语法结构。一段 Python 代码的"meaning"不仅来自 token 序列，更来自它的 AST（抽象语法树）、数据流和控制流。因此，代码数据的预处理比普通文本多了一层结构化的需求。

一个标准的代码数据 pipe 包含以下步骤：

处理步骤 操作 为什么重要 常用工具
语言检测 识别文件的语言类型（Python/JS/Java...） 不同语言的处理策略不同 linguist, pygments
AST 解析 将源码解析为抽象语法树 提取函数签名、类定义、docstring tree-sitter
注释分离 将单行注释、多行注释与代码分开 可分别用于不同的训练目的 tree-sitter, regex
Docstring 提取 提取函数/类的文档字符串 高价值自然语言-代码配对数据 AST 遍历
代码归一化 统一缩进、去除空行变异 减少 tokenization 噪音 black, prettier
编译/语法验证 尝试编译/解析文件 过滤语法错误的文件 各语言编译器

下面是一个使用 tree-sitter 提取 Python 函数签名的示例：

# 使用 tree-sitter 提取函数签名和文档字符串
from tree_sitter import Language, Parser

PY_LANGUAGE = Language("build/my-languages.so", "python")
parser = Parser()
parser.set_language(PY_LANGUAGE)

def extract_functions(source_code):
    tree = parser.parse(bytes(source_code, "utf8"))
    root = tree.root_node
    functions = []

    def walk(node):
        if node.type == "function_definition":
            name_node = node.child_by_field_name("name")
            docstring_node = None
            body = node.child_by_field_name("body")
            if body and body.children:
                first_stmt = body.children[0]
                if first_stmt.type == "expression_statement":
                    docstring_node = first_stmt

            functions.append({
                "name": source_code[name_node.start_byte:name_node.end_byte],
                "signature": source_code[node.start_byte:body.start_byte],
                "docstring": (source_code[docstring_node.start_byte:docstring_node.end_byte]
                              if docstring_node else ""),
                "body_length": body.end_byte - body.start_byte,
            })

        for child in node.children:
            walk(child)

    walk(root)
    return functions

# 输出: 提取的函数名、签名、文档字符串
实践推荐：tree-sitter 是代码解析的标配工具。它支持 40+ 语言的语法解析，解析速度极快（每秒数百万行），比正则表达式或 pygments 更适合大规模代码数据预处理。

22.2 去污染：从训练数据中移除 benchmark 解题代码

这是代码数据领域最敏感也最棘手的问题。你肯定听说过 HumanEval 和 MBPP——这是衡量模型代码生成能力的两个核心 benchmark。而 GitHub 上恰好有大量针对这些 benchmark 的解题代码仓库。如果这些代码不小心混入训练数据，模型在 benchmark 上的分数就会"虚高"——不是因为它学会了编程，而是因为它背过答案。

去污染的基本方法（在第 17 章已经详细讲过）在代码场景下有一些特殊的注意事项：

  • N-gram 精确匹配：对 benchmark 中的代码做 n-gram 分片（通常 n=13），如果训练数据中的代码片段与之完全匹配，则移除。但这种方法可能漏掉变量名被替换的变体。
  • AST 指纹匹配：比 n-gram 更强健的方法。将代码解析为 AST，然后比较 AST 的结构指纹（忽略变量名等标识符）。如果两个代码的 AST 结构相似度超过阈值，标记为污染。
  • 功能等价检测：最严格的检测——运行模型生成的代码和 benchmark 的参考实现，检查是否对所有测试用例产生相同的输出。但这成本极高，一般只在模型发布前做。
一个真实的教训：2024 年某知名开源模型发布时宣称在 HumanEval 上达到了 85% 的 pass@1——接近 GPT-4 的水平。社区很快就发现，这个分数来自于一个"无意的"污染：模型的训练数据中包含了 HumanEval 的解题代码。重新评估后，去掉污染影响的真实分数不到 60%。这件事之后，该团队的数据管道中加入了严格的去污染步骤，但声誉损失已经无法挽回了。

22.3 The Stack 与许可证过滤

说到代码数据，就不能不提 The Stack——这是当前最大的开源代码数据集之一，包含超过 6TB 的代码文件。但 The Stack 的一个核心挑战是许可证合规。

不同开源许可证对训练数据的使用规定完全不同：

许可证类型 典型许可证 是否可用于预训练 备注
宽松许可证 MIT, Apache 2.0, BSD 通常可以 明确允许复制和修改
弱 Copyleft LGPL, MPL 有争议 不同法域解释不同
强 Copyleft GPL, AGPL 高风险 分发时可能要求开源
无许可证 未声明 最危险 默认保留所有权利

The Stack v2 的一个重要改进就是在每个仓库级别做了许可证检测。但这里有一个工程上的 tricky 问题：怎么确定一个仓库的许可证？

表面上看，只要检查根目录有没有 LICENSE 文件就行。但实际中：一个仓库的不同文件可能使用不同的许可证，一个仓库的 submodule 可能来自另一个许可证的项目，有些仓库干脆就没有 LICENSE 文件（默认"保留所有权利"——这是最危险的情况）。

法律风险提示：2025 年已经出现了多起针对使用 GPL 代码训练模型的诉讼。虽然法律上对"用 GPL 代码训练模型是否构成衍生作品"还没有最终判例，但多家公司的法务团队已经建议在训练数据中完全排除 GPL/AGPL 代码，直到法律环境更加明确。不要为了多几个百分比的 benchmark 分数去冒法律风险。

一个实际的许可证过滤 pipeline 如下：

# 许可证过滤 pipeline 伪代码

REJECTED_LICENSES = {
    "gpl-2.0", "gpl-3.0", "agpl-3.0",
    "cc-by-nc-4.0", "cc-by-nc-sa-4.0",
    "unknown", "no-license"
}

def filter_by_license(repo_metadata):
    repo_license = detect_license(repo_metadata.get("license_file", ""))
    repo_license = repo_license.lower()

    # Step 1: 根目录许可证检测
    if repo_license in REJECTED_LICENSES:
        return "reject"

    # Step 2: 没有 LICENSE 文件 -> 默认保留所有权利
    if not repo_metadata.get("has_license_file"):
        return "reject"  # 保守策略

    # Step 3: 文件级别的 override
    # 检查是否有文件头注释包含 "GPL" 关键字
    # 如果有，即使根目录是 MIT，该文件也标记为高风险
    return "accept"

22.4 数学数据的特殊处理

如果说代码数据的挑战在于结构解析，那么数学数据的挑战则在于符号处理。数学文本大量使用 LaTeX 公式、特殊符号和复杂排版，预处理时稍有不慎就会丢失关键信息。

LaTeX 解析的三大难点

  • 公式与正文的混淆：LaTeX 中 `$...$` 是行内公式，`$$...$$` 是行间公式。但很多网页中的 `$` 符号不是公式边界符（例如价格 "$99"）。简单地用正则匹配会误伤大量非公式内容。解决方法是使用专门的 LaTeX 检测模型（如 LaTeXParsed）来区分真正的公式。
  • 公式归一化：同一个数学公式可以用无数种不同的 LaTeX 方式表达。例如分数 `\frac{1}{2}` 和 `\dfrac{1}{2}` 视觉效果不同但语义相同。归一化的做法是把所有 LaTeX 表达式解析为 MathML 或内部表示 (如 OptiMath)，再序列化为标准格式。
  • 公式去重：两个文档可能用不同的 LaTeX 写法表示完全相同的公式。传统的文本去重会对 `\frac{a}{b}` 和 `\dfrac{a}{b}` 视为不同的内容——但从语义上它们完全一样。正确的做法是把公式渲染成图片或转为规范形式后再做去重。

OpenWebMath 处理流程

OpenWebMath 是目前最成熟的大规模数学数据集构建方案。它的处理流程大致如下：

# OpenWebMath 风格的数学数据提取流程

def process_math_html(html_content):
    # 1. HTML 标签提取，保留数学结构
    math_regions = extract_math_regions(html_content)

    # 2. LaTeX 公式标准化
    normalized = []
    for region in math_regions:
        if is_inline_math(region):
            latex = extract_latex_inline(region)
        else:
            latex = extract_latex_display(region)

        # 使用 OptiMath 转换为标准形式
        standard_latex = optimatex_normalize(latex)
        normalized.append(standard_latex)

    # 3. 公式渲染验证
    for n in normalized:
        try:
            render_math_to_image(n)  # 尝试渲染
        except RenderingError:
            mark_as_corrupted(n)     # 无法渲染的公式标记为损坏

    # 4. 公式去重：转为规范形式后去重
    canonical_forms = [to_canonical(n) for n in normalized]
    deduplicated = deduplicate_by_canonical(canonical_forms)

    return " ".join(deduplicated)
关键参数：在数学数据的处理中，"公式纯度"是一个比文本质量更重要的指标。实践建议：每篇文档中公式占比不低于 20% 才保留。低于这个阈值的文档，其中的公式太稀疏，不值得投入处理资源。

22.5 Chain-of-Thought 数据的构建

代码和数学数据的终极应用，是用它们来构建 Chain-of-Thought (CoT) 训练数据。2022 年 Google 的 CoT 论文和 2024 年 OpenAI 的 o1 模型已经反复证明了：在推理任务上，让模型学会"逐步思考"比直接输出答案要有效得多。

构建 CoT 数据有两种主流方法：

  • 人工标注：让数学/编程专家写出一步步的解题过程。质量最高，但成本极其昂贵（一个数学 CoT 标注可能要 10 美元一个样本）。
  • 合成生成：用强模型（如 GPT-4o, Claude Opus 4.7）对已有的数学题生成逐步推理，再通过验证（答案是否正确、推导是否完整）来筛选。这是 2025-2026 年主流方法。

合成 CoT 数据的关键在于验证环节——如果生成的推理路径中有一步错了，它就是一个"有毒"的训练样本。在实践中，验证通常分为两步：

先验证最终答案的正确性（和标准答案对比），再验证推理步骤的完整性（用另一个模型检查每一步是否逻辑自洽）。只有两步都通过的样本才进入训练集。

数据-代码交叉的黄金地带：最有价值的 CoT 数据往往位于代码和数学的交集——算法推理。比如"用二分法在一个旋转有序数组中查找目标值"这样的问题，既有代码实现又有数学分析。这类数据的训练效果通常优于纯代码或纯数学数据。

22.6 算法竞赛与竞技编程数据

最后，我想特别提一下竞技编程 (Competitive Programming) 数据的价值。2025 年以来，Codeforces、AtCoder 等竞赛平台的解题代码成为预训练数据中的新星。原因是：这类数据天然包含了问题描述 → 算法设计 → 代码实现 → 测试验证的完整闭环，是 CoT 推理链条的完美体现。

处理竞技编程数据时，一个有效的做法是保留每道题的多版本解（不同人的不同解法），让模型接触到同一问题从不同角度切入的多样性。但要注意去除掉那些提交失败（Wrong Answer / Time Limit Exceeded）的代码版本——它们对模型是负样本。

下面是一个简单的竞技编程数据整理策略：

# 竞技编程数据筛选逻辑
def filter_cp_data(submissions):
    """去除非最优解和失败提交"""
    valid = []
    for sub in submissions:
        # 只保留通过全部测试的提交
        if sub.verdict != "AC":  # Accepted
            continue
        # 同一问题保留前3种不同的解法
        if sub.problem_id in seen:
            seen[sub.problem_id] += 1
            if seen[sub.problem_id] > 3:
                continue
        else:
            seen[sub.problem_id] = 1
        valid.append(sub)
    return valid
踩坑提醒：竞技编程数据非常容易过拟合！你想象一下，如果模型在训练中看到了一万道 Codeforces 题目的题解，那它在 Codeforces 的测试集上自然会表现异常好——但这不是真正的编程能力提升。一定要控制竞技编程数据在总数据中的比例，建议不超过代码数据的 10%。

本章小结

代码和数学数据是预训练数据中最"特殊"的两种类型——它们有各自的结构规则、清洗要求和质量标准。代码数据需要 AST 解析、去污染和许可证过滤的三重保障；数学数据需要 LaTeX 规范化、公式渲染验证和语义去重。更重要的是，它们的交叉地带——算法推理和竞技编程——是最能提升模型推理能力的数据类型。当你把这部分数据处理好，你训练的模型在推理任务上会有质的飞跃。

在下一章中，我们将从理论走向实战——Datatrove。这是一个由 HuggingFace 开源的端到端数据处理框架，我们会手把手带你从 Common Crawl 原始数据一路走到清洗干净的训练数据。你会发现，"知道怎么做"和"实际能做到"之间，差的就是一个 Datatrove。
