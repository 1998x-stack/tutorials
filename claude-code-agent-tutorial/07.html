第7章 System Prompt 工程

大家好，我是你们的讲师。上一章我们实现了四件套工具，现在 Agent 已经能干活了。但你有没有发现一个问题：我们 v1 Agent 的 System Prompt 只有一句话——"你是一个 AI Coding Agent。你有 bash/read/write/edit 四个工具……"——就这么一句话，Agent 的表现就已经相当不错了。那 Claude Code 的 System Prompt 到底写了多少？

答案是：上万 tokens。这不是因为他们喜欢写长篇大论，而是因为 System Prompt 承载了海量的关键信息：工具的详细行为规范、输出格式约定、安全规则、代码风格指南、项目特定的约定……每一段文字都在塑造 Agent 的行为方式。System Prompt 的质量直接决定了 Agent 的表现天花板——给定再好的工具集，如果 System Prompt 写得不好，Agent 就像一个不知道自己在哪、该做什么的实习生。

这一章我们拆解 Claude Code 的 System Prompt 工程——它的分层架构、每一层的作用、Prompt Cache 的优化策略、以及 SKILL.md 的懒加载机制。学完这一章，你将能够为自己的 Agent 设计出专业级别的 System Prompt。

System Prompt 的分层架构

Claude Code 的 System Prompt 不是一块"铁板"，而是由多个独立的部分按顺序拼接而成的。每一部分都有自己的来源、内容和缓存策略。整个拼接过程在启动时和运行时动态完成。

五层结构从底到顶依次是：

┌───────────────────────────────────────────────┐
│  Layer 5: Session Memory (运行时动态注入)       │
│  内容: 当前对话的摘要、正在进行的任务状态        │
│  缓存策略: 不缓存（每轮变化）                    │
├───────────────────────────────────────────────┤
│  Layer 4: Skills (按需加载)                     │
│  内容: SKILL.md 文件中定义的领域知识            │
│  缓存策略: 按需加载，加载后缓存                 │
├───────────────────────────────────────────────┤
│  Layer 3: Project Knowledge                     │
│  内容: CLAUDE.md 项目级指南                     │
│  缓存策略: Prompt Cache 断点                    │
├───────────────────────────────────────────────┤
│  Layer 2: CLI Context                           │
│  内容: --model 等参数、当前工作目录信息         │
│  缓存策略: Prompt Cache 断点                    │
├───────────────────────────────────────────────┤
│  Layer 1: Base System Prompt (硬编码)           │
│  内容: Claude Code 的"人设"、通用规则           │
│  缓存策略: Prompt Cache 断点（高命中率）        │
└───────────────────────────────────────────────┘

每一层的加载时机和缓存策略不同。Base Layer 几乎永远是相同的（除非 Claude Code 版本更新），所以它的缓存命中率极高。Project Layer 和 CLI Context 随项目变化，但同一个 session 内不变。Skills 按需加载。Session Memory 每轮都变，根本不适合缓存。

这样设计的根本原因是经济性——把静态内容放在 Prompt Cache 的前端，让每次 API 调用只需要传输很少的新 token，大幅降低成本。Anthropic 官方数据显示，这种分层设计达到了约 92% 的 prefix 缓存命中率。

逐层拆解与示例

下面我们一层层来看每一部分实际长什么样（注：以下内容是简化示例，Claude Code 的真实 Prompt 远为复杂且持续更新）：

Layer 1：Base System Prompt（硬编码人设）

这是 Claude Code 的"出厂设置"——定义 Agent 的基础身份、能力和行为准则。它写死在代码中，不在任何配置文件中暴露：

你是 Claude Code，Anthropic 官方出品的 AI 编程助手。
你的核心能力：

1. 工具使用：你可以调用 bash、read、write、edit、grep 等工具来探索和修改代码库。
   每次工具调用后，仔细观察结果，决定下一步操作。

2. 安全准则：
   - 在修改重要文件前，先读取确认内容
   - 执行危险命令（rm -rf、sudo 等）前征求用户同意
   - 不要安装未经验证的第三方包
   - 不要在不理解的情况下修改生产环境配置

3. 输出规范：
   - 中文回答使用口语化表达，保持专业但友好
   - 解释你的思考过程，让用户理解你的每一步
   - 遇到不确定的情况，诚实地承认，不要编造

4. 任务完成：
   - 不要过早宣称"已完成"，验证你的修改确实生效
   - 在确认任务完成后，给出清晰的总结

Layer 2：CLI Context（运行时上下文）

用户通过命令行参数和环境变量提供的上下文。不是通过 System Prompt 字符串拼接——而是通过 API 参数传递：

# 通过 CLI 参数隐式传递的上下文
# --model claude-sonnet-4-20250514
# --max-tokens 8192
# 当前工作目录: /home/user/projects/my-app
# 操作系统: Linux x86_64
# Git 仓库: my-app (main 分支, 最新 commit: a1b2c3d)

Layer 3：Project Knowledge（CLAUDE.md）

存放在项目根目录或 .claude/ 目录下的 CLAUDE.md 文件。这是项目团队为 Agent 写的"入职手册"——告诉 Agent 这个项目用什么语言、什么框架、代码风格是什么、目录结构如何。它会被完整注入到 System Prompt 中：

# CLAUDE.md (项目级知识)
## 项目概述
my-app 是一个基于 FastAPI 的 RESTful 后端服务。

## 技术栈
- Python 3.12, FastAPI, SQLAlchemy 2.0, PostgreSQL
- 前端: React 18 + TypeScript (在 frontend/ 目录)

## 代码规范
- Python: 使用 Black 格式化, 行长度 88
- TypeScript: 使用 Prettier, 单引号, 尾逗号
- 所有 API 端点必须有单元测试 (pytest)

## 目录结构
- backend/src/ — Python 源码
- frontend/src/ — TypeScript 前端
- tests/ — 测试文件 (镜像 backend/src 结构)
- docker/ — Docker Compose 配置

## 常见操作
- 启动后端: cd backend && uvicorn src.main:app --reload
- 运行测试: pytest tests/
- 数据库迁移: alembic upgrade head

Layer 4：Skills（领域知识按需注入）

SKILL.md 文件定义了 Agent 在特定场景下需要的专业知识。比如 Django 技能、Rust 编译器错误诊断技能。这部分不会在启动时加载，而是在检测到相关上下文时才注入。

Layer 5：Session Memory（对话动态摘要）

随着对话进行，Agent 需要对已经完成的工作做摘要，压缩进上下文。这部分不是传统意义上的"System Prompt"——它是通过上下文压缩机制在运行时动态生成的。我们会在第 10 章详细展开。

Prompt Cache 断点策略

理解完分层结构后，最关键的问题来了：Anthropic 的 Prompt Cache 是如何在这些层之间设置断点的？

Prompt Cache 的工作原理是：API 服务器会缓存 API 请求的开头部分（prefix）。如果下一个请求的开头与缓存中的内容匹配，服务器只需处理有变化的那部分，大幅降低成本并减少延迟。

Claude Code 利用了这个特性，在 System Prompt 的每一层边界设置了 cache_control 断点：

# Python 示例：分层组装 System Prompt 并设置缓存断点
import anthropic
from anthropic.types import TextBlockParam, CacheControlEphemeralParam

system_prompt = []

# Layer 1: Base System Prompt (缓存，几乎永久命中)
system_prompt.append(TextBlockParam(
    type="text",
    text=BASE_PROMPT,
    cache_control=CacheControlEphemeralParam(type="ephemeral")
))

# Layer 2: CLI Context (缓存，session 内稳定)
system_prompt.append(TextBlockParam(
    type="text",
    text=f"当前工作目录: {cwd}\nGit 信息: {git_info}",
    cache_control=CacheControlEphemeralParam(type="ephemeral")
))

# Layer 3: Project Knowledge (缓存，项目不变则稳定)
if claude_md_content:
    system_prompt.append(TextBlockParam(
        type="text",
        text=f"## 项目知识\n{claude_md_content}",
        cache_control=CacheControlEphemeralParam(type="ephemeral")
    ))

# Layer 4: Skills (缓存，加载后稳定)
for skill in active_skills:
    system_prompt.append(TextBlockParam(
        type="text",
        text=f"## {skill.name}\n{skill.content}",
        cache_control=CacheControlEphemeralParam(type="ephemeral")
    ))

# Layer 5: Session Memory (不缓存，每轮变化)
system_prompt.append(TextBlockParam(
    type="text",
    text=f"## 会话状态\n{session_summary}"
    # 没有 cache_control —— 每轮都变，缓存无意义
))

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    system=system_prompt,  # 传入数组而不是字符串
    messages=messages,
    tools=tools
)

断点的位置选择是一门学问。Anthropic 官方透露，Claude Code 在生产中的缓存命中率达到了约 92%。这意味着每次 API 调用中，只有约 8% 的 system prompt token 需要重新计算——大部分都命中了缓存。这个比例是精心选择断点位置的结果。

来看实际的成本影响。假设 System Prompt 总长度为 6000 tokens（这是一个合理的估计），不缓存的话每次请求都要传输和处理这 6000 tokens。缓存命中率 92% 意味着每次请求只有 6000 × 8% = 480 tokens 需要重新处理。以 Claude Sonnet 4 的定价计算，一个 20 轮的 session 可以节省大约 6000 × 20 × 0.92 ≈ 110K tokens 的处理量——这不是小数目。

Prompt Cache 使用技巧：在开发自己的 Agent 时，记住缓存断点的三个黄金法则：
1. 静态内容放前面：越少变化的内容越靠前，缓存命中概率越高
2. 断点不要太多：Anthropic 的 API 最多支持 4 个 cache_control 断点。超过这个数量后面的会失效
3. 变化的内容放最后：比如 Session Memory 放最后且不加 cache_control，确保它不会"污染"前面的缓存

SKILL.md 注入时机与懒加载

Skills 是 Claude Code 中一个非常巧妙的设计。Claude Code 内置了 20+ 个预定义技能，覆盖了最常见的开发场景（React、Django、Docker、Git 等）。但注意一个关键事实：Skills 不是在启动时全部加载的——而是懒加载的。

触发技能加载的三种方式：

  • 关键词匹配：用户的 Prompt 中包含技能相关的关键词（如 "docker" 触发 Docker 技能）
  • 工具调用关联：模型调用了与某个技能相关的工具
  • 用户显式引用：用户直接提到某个技能的名称

Slash Command 是技能的一个特殊形式——比如你现在正在使用的 /tutorial-publisher 就是一个通过 Slash Command 触发的技能。它的加载时机是在用户输入 / 后触发自动补全时。

为什么 Skills 必须懒加载？原因很简单：上下文预算。Claude Code 的 System Prompt 如果一次性加载所有 20+ 个 Skill 的内容，至少会增加 5000-10000 tokens。考虑到大多数 session 只用到其中 1-3 个 Skill，提前加载所有内容纯粹是在浪费宝贵的上下文空间。懒加载让 Agent 可以把更多 token 预算留给实际的代码分析。

Skills 的懒加载机制由 SkillTool 管理，它本身是一个特殊的工具——当条件触发时，SkillTool 会向 System Prompt 中注入对应的技能内容。注入后的内容会通过 Prompt Cache 缓存，后续调用不再重复加载。

Prompt 工程中的踩坑教训

Claude Code 的 System Prompt 工程并非一帆风顺。以下是一些真实发生过的问题：

坑 1：注入顺序导致的行为冲突

如果 CLAUDE.md 中定义了一条"项目不使用 Docker"，而 Skills 层加载了 Docker 技能并告诉 Agent"使用 Docker 容器运行应用"，Agent 就会产生矛盾行为。Claude Code 的解决办法是高的层覆盖低的层——CLAUDE.md 的优先级高于自动加载的 Skills——但如果有多个 Skills 互相冲突，行为就不可预测了。

坑 2：Cache 断点位置错误导致缓存命中率暴跌

早期版本的 Claude Code 在 Session Memory 之前也设置了 cache_control 断点。这意味着一旦 Session Memory 变化（每轮对话都会变），整个前缀缓存就失效了——缓存命中率直接掉到个位数。Anthropic 花了一个版本才修复这个问题：Session Memory 不加缓存断点，让它成为"缓存屏障"之后的动态部分。

坑 3：CLAUDE.md 内容过于冗长

有些项目在 CLAUDE.md 中写了大量的设计文档、架构决策记录（ADR）甚至周报。这些内容对 Agent 编码没有帮助，但白白占用了上下文预算。我的建议是：CLAUDE.md 应该只包含 Agent 完成任务真正需要知道的信息——技术栈、代码规范、目录结构、常用命令。其他的内容放 README 就好。

安全警告：SKILL.md 是 Prompt 注入攻击向量

如果第三方代码库包含恶意的 SKILL.md 或 CLAUDE.md 文件，这些文件的内容会被注入到 System Prompt 中——这是一个潜在的 Prompt 注入攻击面。恶意内容可以尝试让 Agent 执行危险命令、泄露敏感信息、或被 Social Engineering 引导做出不安全行为。

防御措施：在运行来自不可信来源的项目时，手动检查 .claude/ 目录下的配置文件；不在 CLAUDE.md 中放置敏感信息（API Key、密码等）；使用 Claude Code 的信任策略（trust policy）对特定项目限制权限。

层 来源 示例内容 缓存策略 典型 Token 数
