第15章 AutoGen 核心架构

从第 8 章到第 14 章，我们系统地学习了 LangChain / LangGraph 生态。从这一章开始，我们将把目光转向微软阵营的 Agent 框架——AutoGen。如果你在 2024-2025 年间关注过 AI Agent 领域，一定对 AutoGen 不陌生：它由微软研究院推出，凭借"多 Agent 对话"的核心理念迅速走红，在 GitHub 上获得了超过 3.5 万颗 Star。

我第一次接触 AutoGen 是在 2024 年初。当时我们团队正在为一个金融客户构建多 Agent 系统，尝试用 LangChain 实现 Agent 间通信时遇到了不少麻烦——消息路由、对话管理、角色分配都需要大量手写代码。切换到 AutoGen 后，这些机制已经内置好了：你只需要定义 Agent 的角色和工具，剩下的对话流程由框架自动管理。这种"开箱即用"的体验让我印象深刻。

重要背景： 截至 2026 年中，微软已将 AutoGen 置于维护模式，不再积极开发新功能。社区分支 AG2（原 AutoGen 2）接管了活跃开发，而微软推荐的新方向是 Microsoft Agent Framework (MAF)。本章和第 16、17 章仍以 AutoGen 原始架构进行教学，因为其设计思想对理解后续的 AG2 和 MAF 至关重要。第 18 章会详细讲述生态演进和迁移路径。

15.1 AutoGen 设计哲学与架构分层

AutoGen 的设计哲学可以概括为一句话：以对话为中心的多 Agent 协作。在 AutoGen 看来，Agent 之间的通信就是"对话"——每个 Agent 都是对话的参与者，它们通过发送和接收消息来协作完成任务。这种设计让多 Agent 系统的构建变得非常直观。

AutoGen 的架构分为三个层次：

层次 组件 职责
应用层 AssistantAgent、UserProxyAgent、GroupChat 面向开发者的编程接口，定义 Agent 行为和交互模式
核心层 Agent、ConversableAgent 基类，消息队列，事件循环 Agent 生命周期管理、消息路由、对话状态维护
基础设施层 LLM 客户端、代码执行器、Tool 注册表 与外部系统（LLM、Docker、文件系统）的集成适配

这种分层设计的好处是：你可以在应用层快速搭建 Agent 系统，而不需要关心底层的消息路由和 LLM 调用细节。如果需要深度定制，核心层和基础设施层也提供了充分的扩展点。

架构洞察： AutoGen 的核心创新不在于 LLM 调用能力——这方面 LangChain 做得更早。它的真正价值在于多 Agent 通信原语：receiving、sending、broadcasting 消息的机制全部内置，开发者只需要关注 Agent 的业务逻辑。

15.2 Agent 基类体系：Agent → ConversableAgent → AssistantAgent

AutoGen 的 Agent 体系是一个清晰的继承链：

Agent 抽象基类

Agent 是所有 Agent 的抽象基类，定义了最基本的契约。每个 Agent 都必须实现以下核心能力：

  • 发送消息（send）： 向指定 Agent 发送消息。
  • 接收消息（receive）： 处理来自其他 Agent 的消息。
  • 生成回复（generate_reply）： 根据接收到的消息生成回复（可能借助 LLM、工具执行或简单规则）。
  • 对话历史（chat_history）： 维护与该 Agent 相关的所有消息记录。

ConversableAgent

ConversableAgent 是 AutoGen 中最重要的中间层，它继承自 Agent，增加了"可对话"的能力。这个类实现了：

  • LLM 调用封装——自动将对话历史转换为 LLM 请求格式。
  • 工具注册与管理——维护一个工具注册表，供 LLM 在推理时选择调用。
  • 人工输入代理（Human Input Mode）——可以在关键步骤要求人类确认。
  • 自动回复钩子——支持自定义回复逻辑（如执行代码、查询数据库）。

AssistantAgent 与 UserProxyAgent

AssistantAgent 是 ConversableAgent 的最常用子类，代表一个由 LLM 驱动的 AI 助手。它负责接收任务、规划步骤、调用工具、生成回复。你几乎不需要重写这个类，只需要配置系统提示词和注册工具。

UserProxyAgent 则扮演"人类代理"的角色。它可以执行代码、发送人工输入、批准或拒绝 AssistantAgent 的行动请求。在自动模式下，它将代码执行结果自动回传给对话。

from autogen import AssistantAgent, UserProxyAgent

# 创建 Assistant Agent（由 LLM 驱动）
assistant = AssistantAgent(
    name="assistant",
    llm_config={
        "config_list": [{"model": "gpt-4o", "api_key": "..."}],
        "temperature": 0.1,
    },
    system_message="你是一位资深数据分析师。利用工具完成用户的数据分析任务。"
)

# 创建 User Proxy Agent（执行代码、提供人工输入）
user_proxy = UserProxyAgent(
    name="user_proxy",
    human_input_mode="TERMINATE",  # 仅在需要终止对话时请求人工输入
    max_consecutive_auto_reply=10,
    code_execution_config={
        "work_dir": "coding_workspace",
        "use_docker": True,
    }
)
版本注意： AutoGen 0.2.x 和 0.3.x 的 API 有较大差异。本文示例基于 AutoGen 0.2.x（最后一个稳定主线版本）。AG2 社区分支在 0.2.x 基础上增加了许多新特性，但核心 API 保持兼容。如果你在生产中使用，建议直接使用 AG2。

15.3 Tool 注册与函数绑定

AutoGen 的工具注册机制非常简洁：你定义好 Python 函数，然后将其"注册"到 Agent 上。Agent 在推理时会自动判断是否调用这些工具，并解析返回结果。

from typing import Annotated, Dict, List
import json

# 第1步：定义一个工具函数
def search_stock_price(
    ticker: Annotated[str, "股票代码，如 AAPL、GOOGL"]
) -> Dict:
    """查询实时股票价格"""
    # 模拟 API 调用
    price_data = {
        "AAPL": 198.50, "GOOGL": 175.20, "MSFT": 425.30
    }
    return {
        "ticker": ticker,
        "price": price_data.get(ticker, "未知"),
        "time": "2026-06-30 14:30:00"
    }

# 第2步：创建 Agent 并注册工具
assistant = AssistantAgent(
    name="financial_analyst",
    llm_config={"config_list": [...]}
)

# 注册工具（AutoGen 会自动生成 Function Calling Schema）
user_proxy.register_for_llm(
    name="search_stock_price",
    description="查询指定股票代码的当前价格",
    api_style="function"
)(search_stock_price)

# 第3步：启动对话
user_proxy.initiate_chat(
    assistant,
    message="帮我查一下苹果（AAPL）和微软（MSFT）的当前股价"
)

AutoGen 会自动完成以下工作：将函数签名转换为 LLM Function Calling Schema → 在 LLM 请求中包含工具定义 → 拦截 LLM 返回的工具调用请求 → 执行函数并返回结果 → 将结果注入对话上下文。整个过程对开发者完全透明。

实用技巧： AutoGen 的 register_for_llm 和 register_for_execution 是分开的。前者告诉 LLM 有哪些工具可用，后者定义实际执行的函数逻辑。这种分离让你可以在不改变 LLM 认知的情况下，替换工具的实现（比如从模拟 API 切换到真实 API）。

15.4 消息传递机制与对话管理

AutoGen 的消息传递机制是其最值得深入理解的部分。每个 Agent 内部维护着 chat_history 列表，记录了所有来往消息。消息的格式统一为 {"role": ..., "content": ..., "name": ...}，与 OpenAI 的 Message 格式高度一致。

消息类型

消息角色 说明 示例
user 用户或 UserProxyAgent 的输入 {"role": "user", "content": "分析这份数据"}
assistant AssistantAgent 的 LLM 回复 {"role": "assistant", "content": "我将会用Python分析..."}
tool 工具调用的返回结果 {"role": "tool", "content": "{"result": 42}"}
function LLM 发出的工具调用请求 {"role": "assistant", "function_call": {...}}

对话流程控制

initiate_chat 是启动一次对话的标准入口。你可以通过以下参数控制对话的行为：

# 启动并控制对话流程
user_proxy.initiate_chat(
    assistant,
    message="根据 Q2 财报数据分析公司表现",
    max_turns=5,                    # 最大对话轮次
    clear_history=False,            # 是否清除历史
    silent=False,                   # 是否静默模式（不打印消息）
    summary_method="reflection_with_llm",  # 对话总结方式
)

# 一次对话完成后，可以用 send 继续交流
user_proxy.send(
    recipient=assistant,
    message="刚才的分析不够详细，请补充毛利率趋势"
)
实践经验： max_turns 是一个非常重要的安全参数。如果没有设置，Agent 可能陷入无限循环——尤其是在涉及代码执行时：Agent 执行代码 → 得到结果 → 尝试修复 → 再次执行 → 又发现问题 → 继续循环。设置合理的最大轮次是防止 Token 浪费的第一道防线。

15.5 代码执行能力集成

AutoGen 一个显著特点是内置了代码执行能力。UserProxyAgent 可以配置为自动执行 AssistantAgent 生成的代码，并将结果反馈到对话中。这相当于给 LLM 安装了"双手"——它不仅可以说，还可以动手计算、绘图、分析数据。

三种执行模式

模式 配置方式 适用场景
本地执行 use_docker=False 开发调试阶段，执行后自动清理临时文件
Docker 执行 use_docker=True 生产环境隔离，防止恶意代码影响宿主机
禁用代码执行 code_execution_config=False Agent 仅对话，无需执行代码 
# 配置代码执行环境
user_proxy = UserProxyAgent(
    name="agent_coder",
    human_input_mode="NEVER",
    code_execution_config={
        "work_dir": "agent_workspace",    # 工作目录
        "use_docker": "python:3.11-slim", # Docker 镜像
        "timeout": 60,                     # 超时时间（秒）
        "last_n_messages": 3,              # 仅执行最近 N 条消息中的代码
    },
    max_consecutive_auto_reply=5,
)
安全提醒： 在生产环境中，务必使用 Docker 代码执行模式。2025 年曾爆出多起 Agent 代码注入攻击事件——攻击者通过构造恶意输入，诱使 Agent 生成并执行了删除文件或窃取数据的代码。Docker 沙箱是阻隔这类攻击的最基本防线。第 17 章会详细讨论安全沙箱的配置和最佳实践。

15.6 实战：双 Agent 协作系统

让我们将前面学到的知识综合起来，搭建一个"数据分析双 Agent 系统"。这个系统包含两个 Agent：一个负责规划和分析，另一个负责编码和执行。

import autogen
from typing import Annotated

# 工具：模拟数据库查询
def query_sales_data(
    quarter: Annotated[str, "季度，如 2026Q1, 2026Q2"],
    region: Annotated[str, "地区，如 华东、华南、华北"]
) -> str:
    """查询指定季度和地区的销售数据"""
    data = {
        ("2026Q1", "华东"): "销售额 1250 万，同比增长 15%",
        ("2026Q1", "华南"): "销售额 980 万，同比增长 8%",
        ("2026Q2", "华东"): "销售额 1420 万，同比增长 18%",
        ("2026Q2", "华南"): "销售额 1050 万，同比增长 12%",
    }
    return data.get((quarter, region), "无数据")

# 规划 Agent
planner = AssistantAgent(
    name="planner",
    system_message="你是一位数据分析规划师。分析用户需求，制定分析方案，"
                   "然后交给 coder 执行。最后基于执行结果给出商业洞察。",
    llm_config={"config_list": [{"model": "gpt-4o", "api_key": "..."}]},
)

# 编码 Agent（可以执行 Python 代码）
coder = UserProxyAgent(
    name="coder",
    human_input_mode="NEVER",
    code_execution_config={
        "work_dir": "analysis_output",
        "use_docker": True,
    },
    max_consecutive_auto_reply=5,
)

# 注册工具给 planner
planner.register_for_llm(
    name="query_sales_data",
    description="查询销售数据"
)(query_sales_data)

# 启动对话
coder.initiate_chat(
    planner,
    message="请分析 2026 年上半年华东和华南地区的销售趋势，"
            "并给出改善华南地区业绩的建议。"
)

这个双 Agent 系统的协作流程是：Coder 接收用户请求 → 将请求转发给 Planner → Planner 制定分析计划并调用查询工具获取数据 → Planner 给出分析方案 → Coder 执行 Python 分析代码（生成图表、计算指标等） → 结果回传给 Planner → Planner 基于结果撰写商业洞察报告 → Coder 将最终报告呈现给用户。

在实际运行中，你还可以给 Coder 加上 human_input_mode="TERMINATE"，让它在每个关键输出时请求你的确认——这样既保留了 Agent 的自主性，又保持了人类审批的管控能力。

拓展思考： 你可以在这个基础上叠加更多 Agent——比如增加一个"质量审查 Agent"专门检查输出格式，或增加一个"知识库 Agent"专门检索历史报告。AutoGen 的架构让这种扩展非常自然，只需要新增 Agent 并在对话中引入即可。

本章小结

本章我们深入剖析了 AutoGen 的核心架构：从"以对话为中心"的设计哲学，到 Agent 基类体系（Agent → ConversableAgent → AssistantAgent），从工具注册机制到消息传递模型，再到内置的代码执行能力。最后通过一个双 Agent 数据分析系统的实战，展示了如何将这些组件组合成一个完整的协作系统。

AutoGen 的真正价值在于它将多 Agent 通信的复杂性封装在了框架内部——你不需要手动实现消息路由、对话历史管理、工具调用调度等基础设施，只需要定义 Agent 的角色和工具，剩下的交给框架。这也是为什么它在 2024-2025 年间能成为最受欢迎的多 Agent 框架之一。

下一章，我们将更进一步，探索 AutoGen 最强大的特性之一——GroupChat 与协作模式。当群聊中超过两个 Agent 时，如何管理发言顺序？如何让 Agent 们通过辩论达成共识？这些就是第 16 章要解决的问题。
