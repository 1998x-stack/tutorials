第三章：信息的表示与处理——浮点数

大家好，我是你们的讲师。在上一章我们学习了整数的表示，有符号/无符号之间的转换陷阱，以及整数溢出的严重后果。但说实话，整数再反直觉，至少它和数学上的整数还"差不多"。浮点数才是真正的"异次元生物"。

让我讲一个真实的故事。1991 年海湾战争期间，美国爱国者导弹防御系统在沙特阿拉伯的达兰基地未能拦截一枚伊拉克的 Scud 导弹。导弹击中了兵营，造成 28 名美国士兵死亡。调查发现，爱国者系统的雷达跟踪软件中存在一个浮点精度问题：系统用 24 位定点数表示时间（以 0.1 秒为单位的计数器），乘以 1/10 得到实际秒数。但 1/10 在二进制中是一个无限循环小数，无法精确表示。系统每运行一小时，误差就积累约 0.343 秒。爱国者系统连续运行了 100 多个小时后，累计误差达到 0.34 秒——对于以高速飞行的 Scud 导弹，0.34 秒的误差足以让拦截系统失去目标。

这个故事告诉我们两件事：第一，用浮点数表示连续量（时间、距离、金额）必然存在精度误差；第二，误差是会积累的，在长时间运行的系统里，微小的误差可能酿成灾难。

本章我们就来深入理解浮点数的二进制表示——IEEE 754 标准。

3.1 IEEE 754 浮点格式详解

IEEE 754 是当今几乎所有 CPU 都采用的浮点数标准。它定义了三种核心组件：符号位（S）、阶码（Exponent）和尾数（Mantissa / Significand）。一个浮点数 V 的通用公式是：

V = (-1)^S × M × 2^E

其中 S 是符号位（0 正 1 负），M 是尾数（二进制小数），E 是阶码（整数，经过移码调整）。

单精度（float，32 位）和双精度（double，64 位）的字段分布如下：

精度类型 总位数 符号位 S 阶码 E（含偏置） 尾数 M 偏置值（Bias） 大致范围
float (单精度) 32 1 bit (bit 31) 8 bits (bit 30-23) 23 bits (bit 22-0) 127 ±1.2×10^-38 ~ ±3.4×10^38
double (双精度) 64 1 bit (bit 63) 11 bits (bit 62-52) 52 bits (bit 51-0) 1023 ±2.2×10^-308 ~ ±1.8×10^308

注意尾数位数的微妙之处：float 明明有 23 位尾数，但实际精度大约是 24 位有效二进制位——为什么？这就要谈到规格化与非规格化的区别了。

3.2 规格化、非规格化与特殊值

IEEE 754 根据阶码 E 的值把所有浮点数分为三类：规格化（Normalized）、非规格化（Denormalized / Subnormal）和特殊值（Special）。

规格化值（阶码不全 0 也不全 1）：

这是最常用的情形。阶码被解释为有符号整数，但经过偏置（Bias）处理：真实阶码 E = 存储的阶码值 - Bias。对于 float，Bias=127，所以存储的阶码值为 1~254 时，真实 E = -126~127。尾数 M 有一个隐含的 1——尾数字段存储的是小数部分（fraction），实际尾数 = 1 + fraction。这就是为什么 23 位尾数能提供约 24 位的精度。

非规格化值（阶码全 0）：

阶码全 0 时，真实 E = 1 - Bias（而不是 0 - Bias），且隐含的 1 变成了隐含的 0。即实际尾数 = 0 + fraction。非规格数的用途是表示非常接近 0 的数，在 0 附近提供"逐渐下溢"的特性，而不是从最小规格数直接跳转到 0——这避免了两个问题：一是"突然下溢"造成的精度断层，二是提供了有符号的 +0 和 -0。

特殊值（阶码全 1）：

  • 如果尾数全 0：表示 无穷大（Infinity），分 +∞ 和 -∞。比如 1.0 / 0.0 = +inf。
  • 如果尾数非 0：表示 NaN（Not a Number）。比如 sqrt(-1.0) = NaN。

一个可视化例子：以 8 位浮点格式（1 位符号 + 4 位阶码 + 3 位尾数）为例，从 0 到正无穷的值分布可以直观看到——靠近 0 的区域有更多的可表示数值（非规格化的贡献），越往大数值走，间隔越稀疏。这种"不均匀分布"是浮点数最反直觉的特性。金融系统中用整数（单位：分）而非浮点数来存储金额，正是因为这个原因。

3.3 浮点舍入策略

因为浮点数只能表示一部分实数，当一个实数落在两个可表示浮点数的正中间时，就需要舍入。IEEE 754 定义了 4 种舍入模式：

模式 行为 举例（最近的两个可表示值为 1.0 和 1.1，实际值为 1.05）
向偶数舍入（Round to Nearest, Ties to Even） 舍入到最近的值；恰好在正中间时，取偶数位 1.05 → 1.0（0 是偶数）
向零舍入（Round toward Zero） 直接截断小数部分 1.05 → 1.0；-1.05 → -1.0
向下舍入（Round Down, Floor） 向负无穷方向取整 1.05 → 1.0；-1.05 → -2.0
向上舍入（Round Up, Ceiling） 向正无穷方向取整 1.05 → 2.0；-1.05 → -1.0

向偶数舍入是默认模式，也是大多数情况下使用的模式。它的设计原理和"银行家舍入（Banker's Rounding）"一样——避免单纯的四舍五入带来的统计偏差。在四舍五入中，0.5 总是向上舍入，大量交易后会产生向正方向的系统性偏移。而向偶数舍入在正中间时"五五开"，长期统计下来偏差更小。

实践经验：在数据库和财务计算中，如果你需要可控的舍入行为，务必显式指定舍入模式，不要依赖语言默认行为。例如，SQL 的 ROUND() 函数和 C 语言的 rint() 家族函数都提供了选择不同舍入模式的能力。

3.4 为什么 0.1 + 0.2 ≠ 0.3

这是计算机科学中最著名的"令人困惑的等式"。在任何主流编程语言中运行 0.1 + 0.2 == 0.3，结果都是 false。让我们看看为什么。

根本原因很简单：二进制无法精确表示某些十进制小数。

就像十进制无法精确表示 1/3 = 0.33333... 一样，二进制也无法精确表示 1/10 和 1/5。在二进制中：

0.1（十进制）= 1/10 = 0.00011001100110011...（二进制，无限循环！）
0.2（十进制）= 1/5  = 0.00110011001100110...（二进制，无限循环！）

计算机用有限的 23 位（float）或 52 位（double）尾数截断这个无限循环小数，于是产生了一个微小的截断误差。两个带有截断误差的值相加，误差叠加，最终与 "0.3" 的近似值不相等。

我们来用 C 语言验证：

#include <stdio.h>

int main()
{
    float a = 0.1;
    float b = 0.2;
    float c = a + b;

    printf("0.1 + 0.2 = %.15f\n", c);
    printf("0.3       = %.15f\n", 0.3);
    printf("Equal? %d\n", c == 0.3);

    // 用 double 也一样
    double d = 0.1 + 0.2;
    printf("\ndouble: 0.1 + 0.2 = %.17f\n", d);
    printf("double: 0.3       = %.17f\n", 0.3);
    printf("Equal? %d\n", d == 0.3);

    return 0;
}

输出：

0.1 + 0.2 = 0.300000011920929
0.3       = 0.300000000000000
Equal? 0

double: 0.1 + 0.2 = 0.30000000000000004
double: 0.3       = 0.30000000000000000
Equal? 0

float 版本的误差大约在 1.19e-7 量级（相对误差约 4e-7），double 版本的误差大约在 5.55e-17 量级。double 的误差小得多，但依然不为 0。

正确比较浮点数的方法：永远不要直接用 == 比较浮点数。而是检查两个数的差是否在某个可接受的 epsilon 范围内：

#include <math.h>

int approx_equal(double a, double b, double eps)
{
    return fabs(a - b) < eps;
}

// 使用示例：
double eps = 1e-9;  // 根据应用场景调整
if (approx_equal(0.1 + 0.2, 0.3, eps))
    printf("可以认为是相等的\n");

注意：epsilon 不能选得太小（可能永远返回 false），也不能选得太大（误判错误值）。对于大多数科学计算，1e-9 或相对 epsilon 是合理的选择。

⚠️ 不要用浮点数存储货币！

对于金融和会计软件，浮点数的精度误差是不可接受的。想象一下，一个银行系统每天处理百万笔交易，每笔因浮点舍入损失 1 分钱——一年就是几十万。正确的做法是：

  • 用整数存储"分"或者"厘"（如用 long long 存储金额的最小单位）
  • 或者使用十进制浮点库（如 IEEE 754-2008 的 decimal64/decimal128, Python 的 decimal.Decimal）
  • 数据库中使用 DECIMAL 类型而非 FLOAT/DOUBLE

这一章的篇幅比整数那章短一些，但浮点数带来的"坑"数量绝对不少。理解 IEEE 754 的标准机制——规格化、非规格化的划分、舍入策略、有限精度的根本原因——能帮助你在遇到浮点数相关 bug 时快速定位根因，而不是对着 0.30000000000000004 发呆。
