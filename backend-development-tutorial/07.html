开篇：设计模式不是教条，是沟通语言

我在职业生涯早期犯过一个每个后端工程师都会犯的错误——过度设计。那时我刚读完《设计模式：可复用面向对象软件的基础》这本书，感觉自己拥有了"屠龙之术"。

于是在一个简单的用户注册功能里，我"塞"进了工厂模式（创建不同类型的用户）、策略模式（不同的验证规则）、观察者模式（注册成功发通知）、模板方法模式（注册流程的骨架）。代码抽象出了 8 个接口和 17 个类——全部都是为了实现用户注册这一个功能。同事 review 代码时的表情我至今记忆犹新："你写的这是用户注册还是航天飞机发射？"

后来我逐渐理解了：设计模式的价值不在于"炫技"，而在于提供了共享的"词汇"。当你说"这里用策略模式来处理不同的支付方式"，其他工程师马上能理解你的设计意图——不需要读完整段代码才知道你要做什么。但前提是，这个模式真的解决了问题，而不是为了用模式而用模式。

这一章我们不讲 23 种经典 GoF 模式的具体实现（那是另一门课的内容），而是聚焦后端开发中最核心的设计原则和架构模式——这些才是真正决定代码质量的东西。

SOLID 原则在后端的实践

SOLID 是面向对象设计的五个基本原则。虽然它诞生于 OOP 语境，但其核心思想是跨越编程语言的——即使在 Go 这种不支持继承的语言中，SOLID 原则同样适用。

原则 含义 后端实践示例 违反的后果
SRP
单一职责		 一个类/模块只负责一件事 UserService 只做用户相关的业务逻辑，不负责发送邮件、生成PDF 一个模块承担过多职责，任何一方的改动都可能影响其他功能
OCP
开闭原则		 对扩展开放，对修改关闭 支付方式通过策略模式扩展，新增支付渠道不需要修改已有代码 每新增一个功能都需要修改核心类，导致核心类越来越臃肿
LSP
里氏替换		 子类/实现类应该能替换父类/接口 任何实现了 PaymentGateway 接口的类都应该能无缝替换使用 调用方需要判断具体实现类型，破坏了多态的意义
ISP
接口隔离		 接口应该小而专，不该让实现类依赖不需要的方法 把"大而全"的 UserAPI 拆分成 UserReader、UserWriter、UserAdmin 实现类被迫实现空方法或抛异常，接口变得不可靠
DIP
依赖倒置		 依赖抽象，不依赖具体实现 UserService 依赖 UserRepository 接口，而非具体的 PostgresUserRepository 高层模块直接依赖低层模块，无法替换实现、难以测试
🎯 实践指南：SOLID 不是必须遵守的宗教信条，它是判断设计质量的"嗅觉工具"。当你闻到代码有"坏味道"——一个类改了又改、接口臃肿不堪、改了一个地方炸了三个地方——翻出 SOLID 原则对照一下，通常能找到病根。对于后端开发，SRP 和 DIP 是性价比最高的两个原则——前者让你的代码好理解，后者让你的代码可测试。

贫血模型 vs 充血模型

这是 Martin Fowler 在《企业应用架构模式》中提出的一对经典概念，也是后端开发中争论最多的话题之一。

贫血模型（Anemic Domain Model）：对象只包含数据（字段 + getter/setter），不包含业务逻辑。业务逻辑全部放在 Service 层。这是大多数后端项目的默认选择——Spring Boot、Go 的典型项目中随处可见。

充血模型（Rich Domain Model）：对象既包含数据也包含业务逻辑。典型的 DDD（领域驱动设计）实践。Service 层只负责协调和编排，领域对象自己处理业务规则。

// 贫血模型（最常见的写法）
// User 只是一个"数据袋子"
type User struct {
    ID       int64
    Name     string
    Email    string
    Status   int
    CreatedAt time.Time
}

// 所有业务逻辑在 Service 层
type UserService struct {
    repo UserRepository
}

func (s *UserService) ActivateUser(id int64) error {
    user, err := s.repo.FindByID(id)
    if err != nil {
        return err
    }
    if user.Status == 2 { // 已激活
        return ErrAlreadyActivated
    }
    user.Status = 1
    return s.repo.Update(user)
}
// 充血模型（DDD 风格）
// User 自己管理业务逻辑和不变约束
type User struct {
    ID        UserID
    name      Name
    email     Email
    status    UserStatus
}

// 业务逻辑在领域实体中
func (u *User) Activate() error {
    if u.status == UserStatusActive {
        return ErrUserAlreadyActive
    }
    if !u.email.IsVerified() {
        return ErrEmailNotVerified
    }
    u.status = UserStatusActive
    return nil
}

// Service 层只做协调
type UserApplicationService struct {
    repo   UserRepository
    events EventPublisher
}

func (s *UserApplicationService) ActivateUser(id UserID) error {
    user, err := s.repo.FindByID(id)
    if err != nil {
        return err
    }
    if err := user.Activate(); err != nil { // 调用领域逻辑
        return err
    }
    if err := s.repo.Save(user); err != nil {
        return err
    }
    s.events.Publish(UserActivated{UserID: id})
    return nil
}

两种模型各有适用场景：贫血模型简单直观，适合 CRUD 为主的应用；充血模型在业务逻辑复杂、规则众多的系统中更能体现价值，但学习曲线也更陡。我的建议是：不要一开始就上充血模型——先在 Service 层组织业务逻辑，当 Service 层的代码膨胀到难以管理时，再考虑是否将部分内聚的业务逻辑下沉到领域对象中。

💡 切身体会：我接手过一个使用充血模型的电商系统。领域对象非常丰富——Order、Product、Customer、Payment——但问题在于，业务逻辑在实体类、Service、Domain Service 之间散落得到处都是。跟踪一个"下单"的完整流程需要在 7-8 个类之间跳转。这个教训说明：架构模式本身不会让代码变好，纪律和一致性才会。选择一种模式后，全团队要遵守统一的组织方式，否则混合模式比没有模式更糟糕。

分层架构与依赖倒置

Controller → Service → Repository 的分层模式是后端开发最经典的架构模板。它之所以经久不衰，是因为它解决了核心问题：关注点分离。

  • Controller（处理层）：处理 HTTP 请求/响应、参数解析、请求校验、路由匹配。不包含业务逻辑。
  • Service（业务逻辑层）：编排业务流程、实现业务规则、事务管理。不关心数据从哪里来、响应格式是什么。
  • Repository（数据访问层）：封装数据持久化逻辑。不关心业务规则。

依赖倒置原则（DIP）在这里的应用是：Service 层依赖 Repository 接口，而不是具体的数据库实现。这带来的直接好处是——单元测试时可以用 mock 的 Repository 替代真实的数据库，测试速度从秒级降到毫秒级。

// 定义接口（在 Service 层定义，而非 Repository 层）
type UserRepository interface {
    FindByID(ctx context.Context, id int64) (*User, error)
    Save(ctx context.Context, user *User) error
}

// Service 依赖接口，而不是具体实现
type UserService struct {
    repo UserRepository  // 依赖抽象
}

// 测试时可以轻松替换为 mock
type mockUserRepo struct{}

func (m *mockUserRepo) FindByID(ctx context.Context, id int64) (*User, error) {
    return &User{ID: id, Name: "test"}, nil
}

这个模式看似简单，但实践中常见的问题是"层与层之间职责不清"——Controller 里直接调用 Repository、Service 里写 HTTP 请求代码、Repository 里塞业务逻辑。维护好依赖方向和职责边界，比用什么框架都重要。

DDD 战术设计概要

领域驱动设计（DDD）分为"战略设计"和"战术设计"两部分。战略设计关注限界上下文和通用语言，适合在微服务拆分和架构规划阶段使用。战术设计则提供了一组具体的建筑块（Building Blocks），让你将领域模型映射到代码中。这里我们聚焦战术设计的核心概念：

  • Entity（实体）：有唯一标识、可变的领域对象。两个相同属性的实体会因为 ID 不同而被视为不同。例如：User（用户）、Order（订单）。
  • Value Object（值对象）：没有唯一标识、由属性值定义相等性的对象。不可变的。例如：Address（地址）、Money（金额）、Email（邮箱）。在 Go 中常用 struct 实现，在 Java 中常用 record。
  • Aggregate（聚合）：一组相关对象的集合，通过聚合根（Aggregate Root）控制一致性边界。例如：Order（聚合根）+ OrderItem + ShippingAddress。外部只能通过聚合根来操作聚合内部的对象。
  • Repository（仓储）：为聚合提供"集合风格"的访问接口。每个聚合一个 Repository。注意它和 DAO 的区别——Repository 工作在整个聚合的粒度上，DAO 通常是单表的。
  • Domain Event（领域事件）：领域中有意义的事件。例如：OrderPlaced（订单已下单）、PaymentReceived（已收款）。通常用于跨聚合通信和事件溯源。
⚠️ 慎用 DDD：DDD 适合"核心子域"——也就是你的业务真正产生价值的地方。对于支撑子域（如邮件通知、日志记录）和通用子域（如认证、权限），用简单的 CRUD 或现成的 SaaS 服务即可。我见过太多团队把整个项目用 DDD"武装到牙齿"，结果一个简单的用户管理模块也要建 5-6 个类，开发效率严重下降。把 DDD 用在刀刃上，而不是削铅笔用牛刀。

Clean Architecture：用例驱动的架构设计

Clean Architecture 是 Robert C. Martin（Uncle Bob）提出的架构思想，它的核心是依赖规则（Dependency Rule）——源代码的依赖关系只能从外层指向内层，内层对外层一无所知。

典型的 Clean Architecture 分层（从内到外）：

  1. Entity（企业级业务规则）：最内层，包含企业级的核心业务对象和规则。不依赖任何框架或外部库。
  2. Use Case（应用级业务规则）：包含应用特定的业务流程。编排 Entity 完成某个用例。这是整个架构的"主角"。
  3. Interface Adapters（接口适配器）：将 Use Case 的输出适配为外部系统需要的格式。Controller、Presenter、Repository 的实现都在这一层。
  4. Frameworks & Drivers（框架和驱动）：最外层，数据库、Web 框架、消息队列——所有具体的技术实现都在这里。

Clean Architecture 不是万能药，它适合业务逻辑复杂且预计会长期演进的系统。对于"写完上线再也没人碰"的内部工具或者 MVP 原型，用这么重的架构是过度工程。但如果你正在构建一个核心业务系统（比如支付平台、电商核心交易链路），Clean Architecture 在长期维护中展现的价值非常明显——因为依赖关系被严格管理，技术栈更换（比如从 PostgreSQL 换到 CockroachDB、从 REST 换到 gRPC）时，核心业务代码可以完全不受影响。
