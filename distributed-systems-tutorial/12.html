第12章 网络协议栈深度

前面几章我们一直在聊分布式系统的通信——RPC、消息队列、服务网格。但不管你用哪种通信方式，最终都依赖底层的网络协议。选 HTTP/1.1 还是 HTTP/2？要不要上 QUIC？WebSocket 能承载多少并发连接？TCP 拥塞控制是 BBR 还是 CUBIC？这些选择不是"随便挑一个"的问题——它们直接决定了你的系统的延迟天花板、吞吐上限和崩溃概率。

这一章我们不谈应用框架，只聊传输层和应用层的核心协议。理解它们的工作原理，你才能真正理解为什么有些系统"卡"在一个你意想不到的地方。

6.1 HTTP/2 vs HTTP/1.1

HTTP/1.1 的问题在微服务时代被放大了。最核心的痛点是 队头阻塞（Head-of-Line Blocking）——在同一个 TCP 连接上，请求必须串行，前一个没处理完后面的就得等着。浏览器想了一个歪招：开多个连接（通常 6 个），但治标不治本。

HTTP/2 做了几个关键改进：

特性HTTP/1.1HTTP/2收益
多路复用一个连接一次一个请求一个连接同时多个 Stream减少连接数，消除队头请求阻塞
Header 压缩明文，无压缩HPACK 压缩（静态表+动态表+Huffman）Header 从 500-800 字节降到 50-100 字节
Server Push不支持服务端主动推送资源减少 RTT，提前加载 CSS/JS
二进制分帧文本协议，按行解析二进制帧，流式解析解析效率高，更易扩展
优先级不支持Stream 优先级和依赖树关键资源优先传输

但是注意：HTTP/2 没有解决 TCP 层面的队头阻塞。虽然 HTTP/2 可以在一个 TCP 连接上多路复用多个 Stream，但 TCP 是可靠按序传输协议——如果一个 TCP 包丢了，所有 Stream 都要等待这个包重传。这就是所谓的"TCP 层 HoL Blocking"，也是 QUIC 要解决的核心问题。

实测数据：Google 的测试表明，在 1% 丢包率的网络环境下，HTTP/2 的吞吐量比 HTTP/1.1 还差——因为 HTTP/2 所有流共享一个 TCP 连接，丢包影响所有流。HTTP/1.1 用 6 个连接，丢包只影响 1/6 的请求。所以 HTTP/2 在低丢包率、高带宽的网络下表现最好，在弱网环境反而可能不如 HTTP/1.1。

6.2 QUIC 与 HTTP/3

QUIC（Quick UDP Internet Connections）是 Google 为了解决 TCP 的"老问题"而从零设计的传输协议。它在 UDP 之上实现了可靠传输、加密、多路复用——本质上是一个"在用户空间实现的 TCP+TLS+HTTP/2"。HTTP/3 就是基于 QUIC 的 HTTP 协议。

QUIC 的核心改进：

  • 0-RTT 握手：之前连接过的客户端可以缓存服务器参数，第二次连接时直接发送数据，省掉 1-2 个 RTT。对于 HTTP 这种短连接场景，延迟降低非常显著。
  • 连接迁移：TCP 连接靠 IP:Port 标识，切换网络（WiFi→4G）必须重新建连。QUIC 连接靠 Connection ID 标识，IP 变了连接还在——APP 从 WiFi 切到 4G 时视频不中断。
  • 独立流：QUIC 在 UDP 之上实现了多流（类似 HTTP/2 的 Stream），但每个流独立丢包重传——一个流丢包不影响其他流。这才是真正解决了队头阻塞。
  • 前向纠错（FEC）：发送冗余数据包，允许部分丢包无需重传即可恢复。但这功能在正式 RFC 中移除了——实测显示 CPU 开销超过了收益。
TCP + TLS 1.3 握手（2 RTT）：
Client                          Server
  │                                │
  ├─── SYN ─────────────────────►  │
  │◄─── SYN-ACK ─────────────────  │
  ├─── ACK ─────────────────────►  │  ← TCP 握手（1 RTT）
  │                                │
  ├─── ClientHello ─────────────►  │
  │◄─── ServerHello + Finished ──  │
  ├─── Finished ────────────────►  │  ← TLS 握手（1 RTT）
  │                                │
  ├─── HTTP 请求 ───────────────►  │  ← 第 3 个 RTT 才开始发数据
  │                                │

QUIC 0-RTT（之前连接过）：
Client                          Server
  │                                │
  ├─── 0-RTT 数据 + 握手信息 ───►  │  ← 直接发数据！
  │                                │
  │◄─── 响应 + 握手完成 ──────────  │
  │                                │

Google 的真实数据：YouTube 迁移到 QUIC 后，视频重新缓冲率降低了 20% 以上。Google 搜索的页加载延迟减少了 2-8%（取决于网络条件）。目前全球互联网流量中 QUIC 已经占到了 20% 以上——主要由 Google、Facebook、Cloudflare 等服务驱动。

6.3 WebSocket 与长连接

HTTP 是"你问我答"的请求-响应模型——服务端不能主动给客户端发消息。WebSocket 在 HTTP 之上建立了一个双向通信通道，连接建立后客户端和服务端可以随时互相发消息。

WebSocket 适合推送场景（实时行情、聊天、协作编辑），但连接数管理是个大问题。一个 TCP 连接消耗大约 10-50KB 内存（取决于内核参数和用户态缓存）。100 万个 WebSocket 连接意味着 10-50GB 内存——这还不算业务逻辑。从 C10K（1 万并发连接）到 C10M（1000 万），你面对的不是线性扩展的问题，而是架构的质变。

WebSocket 的心跳保活也是一个常被忽视的细节：

  • TCP 的 keepalive 默认 2 小时才发一次探测，对 WebSocket 来说太慢了。
  • 应用层心跳太频繁（如每 5 秒）会浪费带宽和 CPU。
  • 心跳间隔通常取 30-60 秒，配合指数退避重连。
  • 大规模部署时，要避免"心跳风暴"——所有客户端同时断连同时重连，服务端瞬间被连接请求打满。

6.4 TCP 拥塞控制演进

TCP 拥塞控制算法从 1988 年的 Tahoe 演进到今天，每一代都对应着不同时代的网络特征。

算法年份核心机制公平性高吞吐低延迟
Tahoe1988慢启动 + 拥塞避免 + 快速重传好差中
Reno1990增加快速恢复，丢包即减半窗口好差中
BIC/CUBIC2005/2008三次函数探测带宽，丢包不减速太多较好高差（Bufferbloat）
BBR2016基于带宽和 RTT 建模，不依赖丢包信号中高好

CUBIC 是目前 Linux 的默认算法。它的窗口增长是三次函数曲线——窗口距离上次丢包点越远，增长越快，这样能快速利用空闲带宽。CUBIC 的问题是它依赖丢包作为拥塞信号，但是在高带宽网络中，为了制造一个丢包信号，缓冲区可能要积压很多数据——这就是"Bufferbloat"问题，导致延迟飙升。

BBR（Bottleneck Bandwidth and Round-trip propagation time） 是 Google 的革新之作。它不再依赖丢包信号，而是通过实时测量瓶颈带宽和最小 RTT 来推算最优发送速率。BBR 在 YouTube 和 Google Cloud 上验证的效果很显著——同样网络条件下延迟降低了数十倍，吞吐提升了 2-25 倍。

踩坑故事：BBR 的"不公平"问题

某 CDN 厂商在边缘节点启用了 BBR 后发现，BBR 流和 CUBIC 流竞争时，BBR 会抢占 CUBIC 的带宽。原因：BBR 的 pacing rate 基于实测带宽，它不会像 CUBIC 那样在丢包时大幅降速。一个 BBR 连接可能占掉 70-80% 的共享瓶颈带宽，而 CUBIC 只分到 20-30%。在混合部署环境中，要么全上 BBR，要么全上 CUBIC，混用会导致严重的公平性问题。

6.5 协议选择决策树

不同场景适合不同的协议组合。下面是一个快速参考：

场景推荐协议原因
服务间内部 RPCgRPC (HTTP/2)多路复用、双向流、Protobuf 高效序列化
Web 实时推送（行情/通知）WebSocket双向实时通信，浏览器原生支持
文件传输（大文件）TCP + 应用层分块TCP 可靠传输、丢包重传保证完整性
弱网环境（移动端/卫星）QUIC / HTTP/30-RTT 握手、连接迁移、独立流不受丢包影响
IoT / 低功耗设备MQTT over TCP头部极小、协议轻量、支持 QoS 等级
视频直播/实时音视频WebRTC (基于 QUIC/SCTP)低延迟、自适应码率、支持 NAT 穿透
高丢包率环境KCP / QUICUDP 基础上的可靠协议，比 TCP 在丢包下表现好

协议选择看起来是"技术选型"，其实是"对网络的认知"——你到底理解不理解你的系统在什么网络条件下运行。数据中心内部 0.01% 的丢包率和移动网络 5% 的丢包率，选择的协议栈完全不同。下一章我们将从一个全新的角度看待通信——当多个节点需要就"发生了什么"达成一致时，组通信与广播协议是如何工作的。
