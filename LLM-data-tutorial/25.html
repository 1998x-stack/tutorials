一部六年的进化史

如果说前面的课程是"如何洗数据"的方法论，那么这一章我们要回答一个更宏观的问题："数据清洗本身是怎么进化的？"从 2020 年到 2026 年，六篇具有里程碑意义的论文（或数据集），定义了六个关键进化节点。这六篇论文串联起来，就是一部 LLM 数据工程的进化史。

我曾经在一次技术分享中画了这样一张进化图：C4（2020）→ CCNet（2020）→ The Pile（2020）→ RefinedWeb（2023）→ FineWeb（2024）/ Dolma（2023）→ FLUX（2026）。台下的一位听众问：为什么 2020 年突然冒出三篇？我当时的回答是：因为那年 GPT-3 刚发布，所有人都意识到了数据的重要性，但没有人知道怎么做最好，于是大家分头探索了三条完全不同的路径。

这个探索过程充满了有趣的戏剧性。保守派做规则过滤，激进派做困惑度过滤，理想主义者做多源混合。到了 2023 年，RefinedWeb 突然跳出来说"你们都不对，纯网页+精细清洗就够了"。而到了 2024 年，FineWeb 又说"RefinedWeb 的方向对了，但还可以加 LLM 来做得更好"。2026 年的 FLUX 则又往前推了一步——它发现质量-数量的权衡是可以被突破的。每一步都是对前人的继承和超越，这本身就是数据工程最迷人的地方。

C4 vs CCNet：规则 vs 困惑度

2020 年，Google 发布了 T5 模型及其配套的 C4 数据集（Colossal Clean Crawled Corpus）。C4 的清洗策略非常直接——用一系列启发式规则来过滤 Common Crawl 中的低质量网页。规则包括：只保留结尾标点为终符号的句子、移除包含"lorem ipsum"等占位文本的页面、移除包含 JavaScript 警告的页面、丢弃少于 3 个句子的文档等等。这些规则加起来大约 20 多条，简单粗暴但有效。

几乎在同一时间，Meta AI 的 CCNet 论文采用了完全不同的路线。CCNet 的核心洞察是：高质量文本通常具有较低的语言模型困惑度。他们训练了一个基于 n-gram 的语言模型，用困惑度对每个文档进行评分，然后只保留困惑度低于某个阈值的文档。这种方法不需要手动设计规则，而是让语言模型自动"识别"哪些文本看起来像高质量内容。

维度C4CCNet
核心方法规则过滤（~20 条启发式规则）困惑度过滤（语言模型评分）
计算成本极低（规则匹配即可）中等（需运行语言模型推理）
可解释性强（每条规则有明确含义）弱（困惑度高低难直观解释）
泛化能力差（规则需人工调整适配新语言）好（训练语言模型即可适配新领域）
数据保留率~15%~25%
下游效果C4 成为后续多年基准被 Dolma 等继承
最大贡献定义了"规则过滤"这个范式定义了"模型辅助过滤"这个范式

回过头来看，C4 和 CCNet 的共同贡献在于：它们分别定义了数据清洗的两大基础范式。C4 证明了规则过滤的可行性——它的方法虽然简单，但可复现、可审计、可解释。CCNet 则开创了"用模型来评估数据质量"的思路，这个思路后来演化成了困惑度过滤、质量分类器、LLM 辅助评分等一系列更先进的方法。

The Pile vs RefinedWeb：多源 vs 纯网页

2020 年的第三篇里程碑是 EleutherAI 的 The Pile。作为一个开源社区项目，The Pile 提出了一种截然不同的哲学：好的训练数据应该是多源的。他们从 22 个来源收集数据——包括 PubMed、arXiv、GitHub、Stack Exchange、YouTube 字幕、书籍等——并将这些来源按照一定比例混合。The Pile 的核心理念是：不同来源提供不同维度的知识，混合后可以产生"1+1>2"的效果。

这个理念在 2023 年被 RefinedWeb 彻底挑战了。TII（Technology Innovation Institute）的团队做了一个激进实验：他们只用 Common Crawl 网页数据，但做了极其精细的清洗——经过 8 轮严格的过滤和去重——然后用清洗后的纯网页数据训练了一个模型。结果令人震惊：这个模型在几乎所有基准测试上都达到了与 The Pile 相当甚至更好的性能。

颠覆性发现： RefinedWeb 证明了一个此前没人敢相信的结论——精细清洗的纯网页数据可以达到甚至超越多源混合的效果。在此之前，几乎所有研究都默认"多源混合是必须的"。RefinedWeb 的发现意味着，与其花大量精力收集和整理多个来源的数据，不如把精力集中在做好一个来源（网页数据）的精细清洗上。这个发现彻底改变了行业的方向——FineWeb、DCLM、FLUX 后来都在"纯网页 + 精细清洗"这条路上走得更远。

需要说明的是，"多源 vs 纯网页"并不是一个非黑即白的选择。后来的实践证明，在网页数据的基础上加入适量的代码、论文和书籍数据，仍然是有益的。Dolma 的 6 源混合策略在 RefinedWeb 之后仍然取得了优秀的效果。关键在于：你不应该把多源混合当作"银弹"——如果网页数据本身清洗得不够精细，混入再多来源也无济于事。

FineWeb vs Dolma：LLM 分级 vs 透明研究

到了 2023-2024 年，数据清洗领域迎来了两座新的里程碑：AI2 的 Dolma 和 HuggingFace 的 FineWeb。它们代表了数据工程这个领域逐渐成熟后的两个不同方向。

FineWeb 的核心创新在于引入了LLM 辅助质量分级。HuggingFace 团队用 Llama-3-70B 对 FineWeb 中的每个文档进行教育质量评分（从 0 到 5 分），然后只保留评分最高的 8.7% 的文档，形成了 FineWeb-Edu 数据集。这个 1.3T token 的"瘦身版"数据集在 MMLU 等基准上的表现，反而超过了完整的 15T 版本。这彻底颠覆了"数据越多越好"的传统认知。

Dolma 的创新不在于方法论上的突破——它的清洗方法基本上是 CCNet 和 C4 的综合——而在于工程实践上的突破。Dolma 是第一个完全透明的 3T+ token 数据集。所有代码、配置、中间统计信息全部公开。这种透明度使得 Dolma 成为了数据工程研究的"实验平台"——研究人员可以在 Dolma 的基础上修改清洗策略，然后精确对比效果差异。

发布方HuggingFaceAI2（Allen Institute for AI）

FineWeb 和 Dolma 实际上是互补的。FineWeb 提供了质量方法论——告诉我们"用 LLM 来评判数据质量是有效的"；Dolma 提供了透明研究的基础设施——让研究人员可以复现、验证和改进清洗方法。如果你在做研究，你可以用 Dolma 的基础设施来复现 FineWeb 的方法；如果你在做工程，你可以用 FineWeb 的方法结合 DataTrove 来快速搭建流水线。

FLUX (2026)：超越质量-数量权衡

2026 年，数据清洗领域的最新进展来自 FLUX 项目。FLUX 的核心发现是："质量-数量"的权衡是可以被突破的。在此之前，社区普遍认为：你必须在"保留更多数据（但质量较低）"和"保留较少数据（但质量更高）"之间做选择。FLUX 通过一种新的过滤策略，同时实现了更高的保留率和更高的数据质量。

FLUX 的方法借鉴了 FineWeb-Edu 的 LLM 分级思路，但做了几个关键改进。首先，他们使用了一个更高效的评分模型（经过蒸馏的小型 LLM），使得对 15T token 进行评分成为一项可负担的任务。其次，他们不再使用单一的评分阈值，而是采用了一种"分档采样"的策略——不同质量档次的文档按照不同的概率保留，而不是简单地"超过阈值就留，否则就丢"。最后，他们在评分之外还引入了一个"多样性"指标，确保保留的数据集在主题覆盖上不会过于偏窄。

FLUX 的成果非常亮眼：在 3B 参数模型上使用 60B tokens 训练，MMLU 达到了 32.14%，超过了同等计算预算下的 DCLM 和 FineWeb-Edu 基线。更重要的是，FLUX 的数据保留率约为 FineWeb-Edu 的 2 倍（约 17% vs 8.7%），但在下游任务上的表现反而更好。这意味着 FLUX 成功打破了"质量-数量"的传统权衡关系。

从数据中可以清晰看到两个趋势。第一个是清洗方法的持续进步：从 C4 到 FLUX，六年间 MMLU 从 22.10 提升到 32.14，进步幅度高达 45%。第二个趋势更耐人寻味：后期范式（FineWeb-Edu、FLUX）的保留率更低，但效果反而更好——这再次验证了"质量 > 数量"的核心命题。而 FLUX 的突破在于，它证明了这个命题并不是一条"此消彼长"的曲线——高质量和高保留率可以兼得。

选择指南：我应该使用哪种范式？

现在你可能会问：六种范式，我该选哪一个？答案取决于你的资源、目标和场景。下面是一份实用的选择指南：

资源有限，预算紧张。 如果你只有有限的 GPU 资源，目标是快速得到一个可用的数据集，那么仿照 FineWeb-Edu 的做法是最优选择——做激进的规则过滤，然后用一个轻量级的质量分类器做排序和筛选。不需要复杂的配置，DataTrove 加上内置的 FineWeb 配置就能完成。代价是保留率较低（约 8-10%），但质量有保障。

追求模型的通用能力。 如果你在训练一个通用型 LLM，目标是让它"什么都会一点"，那么参考 Dolma 的 6 源混合策略会更好。Dolma 的比例经过了大量消融实验验证，是一个经过实战检验的起点。在此基础上，你可以根据你的测试集反馈微调各源的比例。

做垂直领域的专用模型。 假设你在训练一个医学 LLM、法律 LLM 或代码 LLM，那么 RefinedWeb 风格的处理方式——从相关领域的网页数据出发，做精细清洗——然后加入适量的领域文档（论文、书籍）做重采样，通常是效果最好的方案。Dolma 的研究表明，高质量源经过 2-3 倍上采样本后效果显著提升。

追求极致性能，不计成本。 如果你在做最前沿的 LLM 研究，预算充足，目标是冲击 SOTA，那么 FLUX 风格是最优选择。用 LLM（即使是蒸馏后的小模型）对每一篇文档做质量评分，配合分档采样策略，在数据集规模和质量之间找到最优平衡点。

核心建议： 没有绝对最好的范式——取决于你的资源、目标和场景。但所有范式中有一条共同的原则：理解你的数据。不管你用哪种范式，第一步永远是拿出一部分数据做详细的 EDA（探索性数据分析），了解数据的分布、噪声类型和质量特征。没有一个范式能替代你对数据的理解。

最后，我想强调一点：这些范式不是彼此替代的关系，而是层层叠加的关系。一个现代的清洗流水线通常包含了所有范式的元素——C4 的规则、CCNet 的困惑度、The Pile 的多源混合、RefinedWeb 的精细清洗、FineWeb 的 LLM 评分、Dolma 的透明度工程、FLUX 的分档采样。真正的专家不是"使用"某一种范式，而是根据具体的场景和数据特征，灵活组合不同范式的元素。

清洗范式的演进远未结束。下一章我们将进入一个更前沿的话题——自动化数据准备系统。当数据清洗的流程变得越来越成熟，一个自然而然的问题是：能不能让系统自动决定清洗策略，而不是依靠人工反复试错？这正是第 26 章要探讨的内容。
