第二十六章：HTTP/2

同学们好。这一章我们要聊的是HTTP协议家族中的一个重要成员——HTTP/2。如果你在2015年之前做Web开发，你应该体验过那个"页面加载慢、资源多、连接不够用"的年代。HTTP/2的出现，彻底改变了Web性能的面貌。

HTTP/2的核心理念不是"推翻重来"，而是"在现有基础上优化"。它保持了HTTP/1.1的语义——同样的请求方法、状态码、头部字段——只是改变了数据的传输方式。这意味着后端代码几乎不需要修改，就能享受到性能提升。下面我们逐一深入。

26.1 HTTP/1.1的局限性

要理解HTTP/2为什么好，首先得知道HTTP/1.1哪里不好。我在第一章提到过，HTTP/1.1虽然引入了持久连接和管线化，但仍然有几个硬伤：

队头阻塞（Head-of-Line Blocking）：这是最严重的问题。在HTTP/1.1的持久连接中，请求是串行处理的——同一连接上，前一个请求的响应没到达之前，后面的请求都得等着。虽然理论上管线化可以解决，但由于实现复杂性，浏览器在实际使用中几乎都不开启管线化。结果就是，一个慢请求会阻塞后面所有的请求。

头部冗余：每次请求都带着大量重复的头部字段。比如 User-Agent、Accept、Cookie、Host 等字段，每个请求都一摸一样地发一遍。一个典型的HTTP请求，头部可能占500-800字节，十个请求就是5-8KB的冗余数据。

连接数量限制：浏览器对同一域名最多建立6-8个TCP连接（以前是2个）。为了突破这个限制，Web开发者想出了各种"奇技淫巧"：域名分片（Domain Sharding）——把资源分布到多个子域名下，每个域名都能开6个连接，变相增加并发。Image Sprites——把很多小图标合成一张大图，减少请求数。Concatenation——把JS/CSS文件合并成一个。这些做法增加了前端构建的复杂度。

问题HTTP/1.1表现HTTP/2改进
队头阻塞一个请求阻塞，后面的全部等待多路复用，一个流阻塞不影响其他流
头部冗余每次请求重复发送相同头部HPACK压缩，头部压缩率可达80-90%
连接限制浏览器限制6-8个连接/域名单一连接，无限并发流
协议开销明文文本协议，解析慢二进制帧，解析高效
服务器推送无法主动推送资源Server Push可以提前推送

小知识：我曾经维护过一个电商网站，首页有200多个HTTP请求（图片、CSS、JS、字体）。为了优化加载速度，团队用了域名分片，把静态资源分散到cdn1.example.com到cdn4.example.com四个域名下。这样每个域名开6个连接，总共24个并发。部署HTTP/2之后，所有这些分片、Sprite、合并都可以取消了，一个连接就够了。这个案例很典型地说明：HTTP/1.1的优化技巧，到了HTTP/2时代反而成了反模式。

26.2 二进制帧层

HTTP/2最核心的变化，就是把HTTP/1.1的文本协议改成了二进制协议。别小看这个变化——它不仅仅是编码方式的改变，而是整个数据组织方式的革命性重构。

HTTP/2引入了一个二进制帧层（Binary Framing Layer），位于应用层和传输层之间。在这个层中，HTTP的请求和响应被拆分成一个个帧（Frame），不同帧属于不同的流（Stream）。

帧（Frame）是HTTP/2的最小通信单位，包含帧头（长度、类型、标志、流标识符）和负载数据。常见的帧类型有：

  • DATA —— 传输请求体或响应体数据
  • HEADERS —— 传输头部数据
  • PRIORITY —— 设置流的优先级
  • RST_STREAM —— 终止某个流（不像HTTP/1.1那样只能断整个连接）
  • SETTINGS —— 连接参数协商（双方在连接建立时交换）
  • PUSH_PROMISE —— 服务器推送的预告
  • GOAWAY —— 优雅关闭连接

流（Stream）是虚拟的请求-响应通道。每个流都有一个唯一的整数ID。多个流可以同时在同一个TCP连接上交错传输——这就是多路复用。在HTTP/1.1中，你要下载a.css和b.js，要么排队等，要么开两个连接。在HTTP/2中，你把a.css的帧和b.js的帧交错着发到同一个连接上就行了。

// HTTP/2帧结构示意（非精确字节对齐）
+-----------------------------------------------+
| Length (24 bits)     | Type (8 bits) | Flags   |
+-----------------------------------------------+
| Stream Identifier (32 bits, 最高位保留)          |
+-----------------------------------------------+
| Frame Payload (长度由Length字段决定)              |
+-----------------------------------------------+

多路复用的细节特别优雅：帧的流标识符让接收方能够把交错的帧重新组装成完整的请求或响应。这意味着服务器可以同时处理多个请求，哪个先准备好就先发哪个的帧，不需要按请求顺序响应。浏览器收到后根据流ID重新排序组装。

理解关键：多路复用消灭了HTTP/1.1的队头阻塞，但请注意——这只是HTTP层的队头阻塞。TCP层仍然有队头阻塞（丢失的数据包会阻塞后续所有数据流的重组），这个问题要到HTTP/3才会解决。这就像高速公路上的小车道——HTTP/2让车可以并行从多个入口上高速，但如果高速路上发生了车祸（TCP丢包），所有车道的车都得堵着。

26.3 HPACK头部压缩

HPACK（RFC 7541）是HTTP/2的头部压缩算法。它的设计目标是解决HTTP/1.1中头部重复传输的效率问题。前面说过，一个典型HTTP请求头部有500-800字节，其中大部分字段（Host、User-Agent、Accept、Cookie等）每次请求都一摸一样。

HPACK的核心机制是静态表和动态表：

静态表：HTTP/2预先定义了61个常见头部字段及其常用值的静态映射。比如 :method: GET 对应索引2，:scheme: https 对应索引7。发送方只需要发送一个字节的索引号，接收方查表就能还原出完整的头部名称和值，而不是每次发送一长串文本。

动态表：连接建立后，发送方和接收方各自维护一个动态表，记录这个连接中出现过的头部字段。第一次发送完整头部时，双方把这个头部存入动态表并分配索引。后续相同的头部可以直接用索引替代。动态表的大小可以通过 SETTINGS_HEADER_TABLE_SIZE 协商，默认是4096字节。

此外，HPACK还使用了Huffman编码对头部值进行压缩。特别是一些长字符串（比如Cookie、User-Agent），Huffman编码可以减少30-40%的体积。

压缩方式原理压缩效果
静态表索引预定义61个常见头部的索引常见头部可压缩到1-2字节
动态表索引记录连接中出现过的头部重复头部可压缩到1字节
Huffman编码统计编码，高频字符用短编码减少30-40%体积

安全警告：HPACK的动态表大小是有限制的，默认只有4KB。我曾经遇到过一个问题——后端返回的Cookie头部特别大（几千字节），导致动态表被快速填满，然后新增头部只能使用Huffman编码发送，效率反而下降。后来排查发现Cookie里塞了一个完整的Base64编码的用户JSON，把这个修掉之后动态表表现就正常了。这个教训是：不要往Cookie里放大量数据，头部越大，HPACK动态表的效率越低。

26.4 服务器推送（Server Push）

服务器推送是HTTP/2的一个特色功能。顾名思义，服务器可以在客户端没有明确请求的情况下，主动发送资源给它。推送的资源由客户端的 PUSH_PROMISE 帧预告，然后服务器发送这些资源的数据帧。

典型场景：浏览器请求一个HTML页面，服务器知道这个页面依赖a.css和b.js。与其等浏览器解析HTML后发现需要a.css再发请求，服务器不如在发送HTML响应的同时，主动把a.css和b.js推送给客户端。

// 服务器推送的时序示意
> GET /index.html
< HEADERS:index.html
< PUSH_PROMISE: /style.css   (流2)
< PUSH_PROMISE: /app.js       (流3)
< DATA: index.html content
< HEADERS+DATA: /style.css    (流2)
< HEADERS+DATA: /app.js       (流3)

PUSH_PROMISE 帧必须在响应的DATA帧之前发送，这是一个重要的顺序约束——因为浏览器需要在接收数据之前就知道有哪些资源会被推送。如果发现某个推送的资源已经在缓存中了，浏览器可以发送 RST_STREAM 帧取消这个推送。

应用问题：服务器推送虽然听起来很美好，但在实际使用中遇到了几个问题：

  • 缓存冲突：服务器不知道浏览器是否已经缓存了某个资源，可能推送了浏览器已经有的内容，浪费带宽。
  • 过度推送：如果推了非关键路径上的资源，会抢占关键资源的带宽。
  • 浏览器取消机制不完善：虽然浏览器可以 RST_STREAM 取消推送，但服务器可能已经在推了，浪费已传输的带宽。

为了解决这些问题，Cache Digests（RFC草案）被提出——浏览器可以向服务器报告自己缓存了哪些资源，服务器据此决定推什么不推什么。不过这个机制还没有广泛部署。

实际建议：在目前的实践中，我不建议大部分网站使用Server Push。原因有三：一是前面说的缓存问题很难解决；二是浏览器已经非常擅长预测资源需求（比如preload hint）；三是很多主流浏览器和CDN（Chrome、Safari、Fastly、Akamai）已经逐步降低了对Server Push的支持优先级。更推荐的做法是用Link: rel=preload 配合 103 Early Hints 状态码，这是更轻量且兼容性更好的方案。Server Push更像是一个"理论上很美"但"实践中陷阱多"的功能。

26.5 流优先级与流量控制

在HTTP/2的多路复用中，多个流共享一个TCP连接。问题来了：如果同时有10个请求的流在传输，浏览器应该优先分配带宽给哪个流？这就是流优先级（Stream Priority）要解决的问题。

HTTP/2的流优先级系统采用依赖树（Dependency Tree）模型。每个流可以：

  • 指定它依赖哪个流（父流）
  • 在兄弟之间分配权重（1-256的整数）

一个典型的优先级结构是这样的：

// 流优先级依赖树示例
     (root)
      /    \
    流1    流2
   (html)  (数据API, 权重16)
     |
    流3
  (CSS/JS, 权重12)
     |
    流4
  (图片, 权重4)

在这个结构中，HTML（流1）的优先级最高，因为它需要先被解析才能发现子资源。CSS/JS（流3）是HTML的子节点，浏览器在解析过程中发现了它们。图片（流4）优先级最低——可以先展示文本内容，图片懒加载。

权重决定的是同级别流之间的带宽分配比例。比如两个图片流，权重分别是12和4，那么它们的带宽分配比例就是3:1。

流量控制（Flow Control）是另一个重要机制。在HTTP/2中，每个流和整个连接都有独立的流量控制窗口。接收方通过 WINDOW_UPDATE 帧告诉发送方"我还想接收多少字节"。这防止了某个流消耗过多连接资源。

// 流量控制窗口示意
// 初始窗口大小默认65535字节
INITIAL_WINDOW_SIZE = 65535

// 接收方处理了数据后，发送WINDOW_UPDATE增加窗口
WINDOW_UPDATE: stream_id=3, increment=16384

要点总结：HTTP/2的流量控制是端到端的，不是HOP-BY-HOP的。这意味着中间代理不能修改流量控制窗口。窗口大小可以通过 SETTINGS 帧的 SETTINGS_INITIAL_WINDOW_SIZE 参数调整。如果窗口设得太小，会降低吞吐量；设得太大，某个慢消费者可能占用过多内存。一般建议保持默认值，除非你有非常明确的需求。

踩坑故事：有次我们部署HTTP/2之后，发现某些用户的上传速度极慢。排查了好久，最终发现问题出在Nginx的 http2_max_concurrent_streams 和 http2_max_requests 配置上。当时我们设置了一个较低的并发流限制（10个），在HTTP/1.1时代足够了。但迁移到HTTP/2后，因为一个连接承载了所有请求，10个并发流很快被打满，剩余请求不得不排队等待。更糟的是，我们的API设计中有一个长轮询请求一直占用一个流，导致其他9个流被业务请求迅速占满，后续请求全部阻塞。最终我们把并发流限制提高到128，并把长轮询迁移到了WebSocket上。总结：迁移到HTTP/2后，一定要评估并发流的配置，不要用HTTP/1.1的"连接数"思维来配置HTTP/2的"流数"。
