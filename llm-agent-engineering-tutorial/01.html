第1章 Agent 技术概述

大家好，我是你们的讲师。今天我们来聊聊一个在 2025、2026 年彻底改变了 AI 工程格局的概念 —— LLM Agent。如果你在过去两年里关注过 AI 领域，一定听过这样的说法："2023 年是 LLM 元年，2024 年是 RAG 元年，2025-2026 年是 Agent 元年。"这个判断并非夸张——从 OpenAI 发布 GPT-4 开始，大家就意识到大语言模型的能力边界不止于聊天；而真正让 LLM 从"对话工具"进化为"数字员工"的关键，就是 Agent 架构。

很多人问我：有了 GPT-4、Claude 这样的强大模型，为什么还需要 Agent？打个比方，大模型像是一个知识渊博但不会动手的教授，而 Agent 就是给这位教授配备了双手、眼睛和行动力——它能调用 API、操作数据库、阅读文档、执行代码，甚至协同多个"同事"一起完成任务。这不仅是能力的量的提升，更是质的飞跃。

核心洞察： LLM 解决的是"理解与生成"的问题，而 Agent 解决的是"感知-决策-行动"的闭环问题。两者结合，才是真正的智能系统。

1.1 什么是 AI Agent

学术界对 Agent 的定义可以追溯到 1995 年 Russell & Norvig 的《Artificial Intelligence: A Modern Approach》，他们将其描述为"能够通过传感器感知环境、通过执行器影响环境的实体"。但放到 2026 年的 LLM 语境下，这个定义有了更具体的含义：

  • 感知（Perception）： 接收多模态信息——用户输入的文本、上传的图片、系统日志、数据库查询结果、API 返回值等。
  • 推理（Reasoning）： 由 LLM 作为"大脑"，对感知到的信息进行分析、规划、决策。这是 Agent 区别于传统自动化脚本的核心所在。
  • 行动（Action）： 调用外部工具执行具体操作——发出 HTTP 请求、执行 SQL 查询、写入文件、发送邮件、操作浏览器等。

这个"感知-推理-行动"循环，通常简称为 ReAct 模式（Reasoning + Acting），是几乎所有现代 Agent 框架的理论基础。我们会在第 2 章深入剖析这个循环的每一个环节。

小贴士： 区分"AI Agent"和"AI 应用"的关键标志是：Agent 具备自主决策能力，能够在没有人类逐一指令的情况下，根据目标自行规划执行步骤。如果每一步都由人类写出精确指令，那只是传统自动化，不是 Agent。

1.2 发展历程全景（2023-2026）

回顾 Agent 的发展史，短短四年间经历了从概念萌芽到工程成熟的全过程。让我带你快速过一遍这段激动人心的历程。

2023 年：AutoGPT 引爆概念

2023 年初，GPT-4 发布后不久，一个名为 AutoGPT 的开源项目在 GitHub 上一夜爆红，短短几周斩获数万 Star。它展示了一个令人震惊的能力：给定一个目标（比如"帮我研究一下量子计算的现状并写一份报告"），AutoGPT 会自动分解任务、搜索网络、阅读结果、迭代优化，最终输出成品。这是公众第一次直观感受到"AI 自主完成任务"的震撼。

同期，BabyAGI 和 AgentGPT 也相继出现。这些早期项目虽然概念超前，但工程上非常粗糙——Token 消耗惊人、经常陷入死循环、缺乏错误恢复机制。我清楚记得 2023 年中尝试部署 AutoGPT 时的情景：启动脚本后它兴高采烈地去查资料，结果十分钟后因为 API 超时而卡住，然后开始疯狂重试，最后账单上多了 20 美元——什么结果都没出来。

踩坑实录： 2023 年的 AutoGPT 很容易陷入"自言自语"的死循环——它不断生成思考链却无法落地执行，导致 Token 疯狂消耗却没有任何产出。这是早期 Agent 缺乏合理终止条件和错误边界设计的典型教训。

2024 年：框架爆发

2024 年是 Agent 框架的"春秋战国时代"。上半年，LangChain 凭借其庞大的工具生态成为最流行的 Agent 开发框架，但也因 API 频繁变更被开发者戏称为"改链狂魔"。年中，微软发布了 AutoGen，引入了多 Agent 对话范式——让多个 Agent 角色通过对话协作完成任务。下半年，CrewAI 以"角色化 Agent"的概念异军突起，让开发者可以为每个 Agent 定义角色、目标和职责，像组建团队一样搭建 Agent 系统。

这一年，Agent 从"技术 demo"走向了"可用产品"。GitHub Copilot 的 Agent 模式、各类 AI 编程助手都开始实际落地。

2025 年：工程化成熟

进入 2025 年，行业焦点从"能不能做"转向了"怎么做稳、怎么做快"。几个关键趋势出现：

  • 状态图驱动： LangGraph 将 Agent 定义为状态机，用图结构管理复杂工作流，解决了传统链式架构难以处理分支和循环的问题。
  • 可观测性： Agent 的调试和监控工具日趋成熟，LangSmith、Arize 等平台提供了完整的 Trace 和分析能力。
  • 沙箱安全： 代码执行 Agent 的安全问题被高度重视，Docker 沙箱、e2b.dev 等方案成为标配。

2026 年：协议标准化

2026 年，MCP（Model Context Protocol） 和 A2A（Agent-to-Agent） 协议成为行业焦点。MCP 由 Anthropic 提出，为 LLM 提供标准化的工具接入协议——类似于"AI 世界的 USB 接口"。A2A 协议则由 Google 主导，定义了 Agent 之间的通信标准。这两个协议的落地意味着 Agent 生态从"各自为战"走向"互联互通"，其重要性不亚于 HTTP 对 Web 的意义。

趋势观察： 2026 年上半年，主流云厂商（AWS、Azure、GCP）均已内置 MCP Server 支持。部署一个 Agent 的复杂度已经从 2024 年的"需要专职团队"降低到"一个配置文件"。

1.3 Agent 核心能力矩阵

经过四年的演化，行业对 Agent 需要具备哪些核心能力已经形成了共识。我们可以用"四维能力矩阵"来概括：

能力维度 说明 典型实现 成熟度
工具调用 (Tool Use) Agent 调用外部 API、函数、数据库等的能力 Function Calling、MCP Protocol、OpenAI tool_use 成熟
记忆 (Memory) 短期对话上下文 + 长期知识存储 RAG、向量数据库、Mem0、memgpt 成熟
规划 (Planning) 将复杂目标分解为可执行的子任务序列 ReAct、Plan-and-Execute、Tree-of-Thoughts 发展中
反思 (Reflection) 对自身输出进行评估、纠错、迭代优化 Self-Critique、Reflexion、Eval-based loops 早期

这四个能力维度不是孤立的，而是层层递进的关系。工具调用是基础，记忆提供了上下文连续性，规划让 Agent 具备解决复杂问题的能力，而反思则是 Agent 实现自我改进的关键。一个成熟的 Agent 系统需要在这四个维度上都有良好的设计。

实践经验： 在我们团队过去一年落地 Agent 项目的经验中，最容易忽视但最重要的能力是"反思"。没有反思机制的 Agent 就像没有代码 review 的开发者——同样的错误会反复出现。加上一个简单的自评环节，通常能让任务成功率提升 30-50%。

下面这段伪代码展示了一个 Agent 工具调用的典型流程——LLM 分析用户需求、决定调用哪个工具、解析返回结果并生成最终回复：

# Agent 工具调用伪代码示例
def agent_loop(user_query):
    messages = [{"role": "user", "content": user_query}]

    while not task_complete:
        # 1. LLM 推理：理解当前状态，决定下一步行动
        response = llm.invoke(messages, tools=available_tools)

        if response.has_tool_call:
            # 2. 执行工具调用
            tool_name = response.tool_call.name
            tool_args = response.tool_call.arguments
            tool_result = execute_tool(tool_name, tool_args)

            # 3. 将工具结果追加到对话中
            messages.append(response.message)
            messages.append({
                "role": "tool",
                "content": tool_result
            })
        else:
            # 4. LLM 认为任务已完成，输出最终回复
            return response.content

1.4 典型应用场景

Agent 技术的落地速度远超许多人预期。到 2026 年中，以下场景已经成为 Agent 应用的"主战场"：

智能客服

这是 Agent 应用最成熟的场景。不同于传统的对话机器人和 RAG 问答系统，Agent 驱动的客服不仅能回答问题，还能执行操作——查询订单、发起退款、修改订阅、预约服务。大型电商平台已在 2025 年底实现了超过 70% 的客服工单由 Agent 自主处理。

代码生成与辅助编程

从 GitHub Copilot 到 Cursor、Windsurf，Agent 化编程助手已经成为开发者标配。2026 年的编程 Agent 不仅能补全代码，还能自主诊断 Bug、编写测试、创建 PR、管理整个 Feature 的开发流程。

研究助手

Agent 可以自主进行文献检索、数据提取、交叉验证、报告撰写。一个配置得当的研究 Agent 可以在几分钟内完成人类研究员需要数天的文献综述工作。我们会在第 27 章搭建一个完整的研究助手项目。

自动化运维

在 DevOps 领域，Agent 正在接管告警响应、故障排查、自动修复等任务。结合 MCP 协议，运维 Agent 可以无缝对接 Prometheus、Grafana、PagerDuty、Kubernetes 等基础设施。

金融分析

Agent 系统可以同时接入实时行情数据、财报数据库、新闻流，进行多维度分析并生成投资报告。由于金融场景对正确性要求极高，这个场景通常采用"多 Agent 辩论"模式——多个 Agent 扮演不同角色（多头、空头、风控），通过辩论达成共识。

注意： Agent 能力越强，潜在的破坏力也越大。2025 年曾发生过一起知名事件：某公司的运维 Agent 在排查性能问题时，误判了一个关键服务并执行了重启操作，导致线上服务中断 47 分钟。这提醒我们：Agent 自主行动的边界和审批机制是生产部署中绝对不能省略的环节。

1.5 框架生态对比

截至 2026 年中，Agent 框架生态已经形成了清晰的格局。下面这张对比表可以帮助你快速了解各框架的定位和差异：

框架 定位 核心特点 适用场景 GitHub Stars
LangChain / LangGraph 通用 Agent 框架 图结构状态机、生态最丰富、LangSmith 可观测性 复杂工作流、企业级应用 100k+
CrewAI 角色化多 Agent 角色定义清晰、团队协作模式、上手简单 内容创作、研究分析 25k+
AutoGen / AG2 多 Agent 对话 Agent 对话驱动、GroupChat 模式、代码执行沙箱 多 Agent 协作、辩论式推理 35k+
OpenManus 通用 Agent 平台 分层架构（Data/Reasoning/Tool）、MCP 原生支持 通用自动化、工具编排 20k+
OpenAI SDK 厂商原生 SDK 原生 Assistants API、内置 Code Interpreter、File Search OpenAI 生态、快速原型 N/A

选择哪个框架取决于你的具体需求：如果追求生态丰富度和企业级特性，LangChain/LangGraph 是首选；如果需要快速搭建多 Agent 协作系统，CrewAI 和 AutoGen 各有千秋；如果偏好轻量和 MCP 原生支持，OpenManus 值得关注。我们会在第 8 章到第 22 章逐一深入讲解每个框架的实战用法。

选型建议： 不要一开始就纠结框架选择。这个教程的设计思路是"先通识，再专精"。前 7 章我们会把 Agent 的核心概念和通用设计模式讲透，之后再用 15 章的篇幅逐一深入主流框架。等你学完前 7 章，自然会对选型有自己的判断。
