第9章 VLAN与802.1Q

大家好，欢迎回到以太网通信课程。上一章我们讲了生成树协议如何通过阻塞冗余链路来消除环路——但STP有一个隐含的前提：整个网络是一个广播域。什么意思？就是说，任何一台主机发出的ARP广播、DHCP发现、NetBIOS名称解析等广播帧，会传遍网络中所有的交换机、所有的端口、所有的设备。在一个小型的办公网络里，这没什么问题。但在一个拥有几百台甚至几千台设备的园区网络中，广播流量会变成一个灾难。

这一章我们要学的东西——VLAN（Virtual Local Area Network，虚拟局域网）——就是为了解决"广播域太大"这个问题而诞生的。VLAN可以说是现代交换网络中最基础也最重要的技术之一。你理解了VLAN，才算真正理解了交换机的灵魂。

9.1 一个校园网被ARP广播瘫痪的故事

真实案例：201X年，某大学校园网在开学季突然大面积瘫痪。网络中心排查发现，核心交换机的CPU使用率持续在99%，端口的广播报文数量达到了每秒数万帧。进一步分析发现，罪魁祸首是一台学生宿舍里的个人电脑——这台机器中了ARP蠕虫病毒，以每秒几百个的速度向网络发送伪造的ARP请求。由于整个校园网在同一个广播域中（没有划分VLAN），这个ARP广播帧被交换机转发到了校园网的每一个角落——从教学楼到图书馆、从食堂到行政楼。最终，全网的终端设备都在处理这些无用的广播帧，正常的数据通信完全无法进行。网络中心的工程师最后不得不物理断开宿舍楼的接入交换机，才让校园网恢复了正常。事后，他们在核心交换机上划分了VLAN，不同楼宇之间通过三层路由通信。从此以后，即使某个VLAN内有广播风暴，其他VLAN也能正常工作。

这个故事说明了什么？广播域需要被适当隔离。在没有VLAN的时代，隔离广播域的唯一办法是使用路由器——路由器不会转发广播帧，但路由器的端口数量有限，价格昂贵。VLAN的出现改变了这一切：它允许你在同一台交换机上创建多个相互隔离的逻辑网络。

9.2 802.1Q标签：给以太网帧加上"身份证"

VLAN的核心机制是在以太网帧中插入一个标签（Tag），用来标识这个帧属于哪个VLAN。这个标签由IEEE 802.1Q标准定义。我们来看看这个标签长什么样。

802.1Q标签被插入到以太网帧的源MAC地址之后、EtherType/Length字段之前。它的总长度为4字节（32比特），结构如下：

+--------+----+----+------------------+
|  TPID   | PCP|DEI |       VID        |
| 2字节   | 3b | 1b |     12比特      |
+--------+----+----+------------------+
| 0x8100  | 优先级|   | VLAN标识符(0-4095)|
+--------+----+----+------------------+

让我们逐一拆解这四个字段：

  • TPID（Tag Protocol Identifier，标签协议标识符）：2字节，固定为0x8100。接收方交换机看到这个值就知道后面跟着一个802.1Q标签。0x8100在正常的EtherType空间里本应表示某种上层协议，但802.1Q占用了这个值来标记VLAN帧——在最早的实现中，这确实和某些协议产生了冲突，后来IEEE通过调整帧格式解决了这个问题。
  • PCP（Priority Code Point，优先级编码点）：3比特，取值范围0~7。用于表示帧的优先级，用于QoS。数值越大，优先级越高。7是最高优先级（网络控制流量），0是最低（尽力而为）。我们在第11章讲QoS时会详细讨论这个字段。
  • DEI（Drop Eligible Indicator，丢包合格指示）：1比特，之前称为CFI（Canonical Format Indicator）。当DEI=1时，表示这个帧在网络拥塞时可以优先被丢弃。这是802.1Q-2011修订版中新增的字段。
  • VID（VLAN Identifier，VLAN标识符）：12比特，取值范围0~4095。其中0x000表示"未标记"或"优先级标签"（只带PCP没有VLAN），0xFFF保留不用。所以实际可用的VLAN数量是4094个（1~4094）。

插入4字节标签后，以太网帧的最小长度（64字节）和最大长度（1518字节）都会相应增加——带标签的帧最小68字节，最大1522字节。早期的交换机如果不支持802.1Q，可能会因为这些"超长"帧而丢弃它们。这也是老设备兼容性问题的一个重要来源。

实用技巧：在Wireshark中抓包时，如果一个以太网帧带有VLAN标签，你会看到Ethernet II头部后面多了一个"802.1Q Virtual LAN"协议层。展开它就能看到VID、PCP和DEI三个字段。这是快速验证VLAN配置正确与否的最直接方法。如果你发现某个帧的VLAN标签不符合预期，那就说明中间的交换机可能做了错误的标签操作。

9.3 Access端口、Trunk端口与Hybrid端口

交换机端口根据处理VLAN标签的方式，可以分为三种类型：Access端口、Trunk端口和Hybrid端口。理解这三种端口的区别是VLAN配置的基础。

Access端口（接入端口）：通常用于连接终端设备（PC、服务器、打印机等）。Access端口属于且只属于一个VLAN（称为PVID，即端口VLAN ID）。当数据帧从Access端口进入交换机时，交换机会给它打上PVID对应的VLAN标签。当数据帧从Access端口发往终端设备时，交换机会把VLAN标签剥离掉——因为终端设备通常不理解802.1Q标签。这个过程叫做"去标签发送"（untagged）。所以，对于连接在Access端口上的主机来说，VLAN是"透明"的——它根本不知道自己是属于哪个VLAN。

Trunk端口（干道端口）：用于连接交换机之间，或交换机和路由器之间。Trunk端口允许多个VLAN的帧通过，并且帧在传输时保留VLAN标签。两台交换机之间通过Trunk链路连接时，它们可以在同一条物理链路上传输属于不同VLAN的帧，互不干扰。

Hybrid端口（混合端口）：这是某些厂商（特别是华为、H3C）特有的端口类型。Hybrid端口允许同时发送带标签和去标签的帧——你可以配置哪些VLAN的帧在出端口时去标签，哪些保留标签。它比Access和Trunk更灵活，但配置也更复杂。

特性 Access端口 Trunk端口 Hybrid端口
典型用途 连接终端设备 交换机互连、路由器子接口 需要灵活标签控制的场景

9.4 Native VLAN与VLAN 1的安全问题

Native VLAN是Trunk端口上的一个重要概念。在802.1Q中，Trunk端口会为一个特定的VLAN（默认是VLAN 1）发送不带标签的帧——这个VLAN就是Native VLAN。为什么要有Native VLAN？主要是为了向后兼容：一些不支持802.1Q的旧设备可以从Trunk链路接收Native VLAN的帧而不被丢弃。

但Native VLAN也带来了严重的安全隐患。最著名的攻击方式叫做VLAN跳跃攻击（VLAN Hopping）——攻击者发送一个"双重标签"的帧（外层标签VID=Native VLAN，内层标签VID=目标VLAN），当帧通过Trunk端口时，交换机会剥离外层标签（因为Native VLAN的帧在出Trunk端口时是不带标签的），然后内层的VLAN标签就暴露出来了。如果内层标签指向一个敏感VLAN（比如管理VLAN），攻击者就可以跨VLAN访问本不应能到达的资源。

这就是为什么安全最佳实践强烈建议：不要使用默认的VLAN 1作为Native VLAN，最好将Native VLAN修改为一个未使用的VLAN ID（比如VLAN 999），并且显式地将Native VLAN从Trunk端口的允许VLAN列表中排除。此外，所有管理接口应当放在一个专门的管理VLAN中，不要放在VLAN 1中。

安全建议：在生产网络中，VLAN 1 应该被视为一个"不信任"的VLAN。不要将任何用户设备放在VLAN 1中，不要通过VLAN 1来管理交换机（即管理IP地址不要配在VLAN 1接口上）。很多网络入侵的初始突破口就是攻击者进入了VLAN 1，然后利用VLAN跳跃攻击进入了管理网络。记住：VLAN 1不是你的朋友，它是攻击者的好朋友。

9.5 QinQ：运营商的"VLAN套娃"

4094个VLAN听起来很多，但对于运营商级别的网络来说，这是远远不够的。想象一下，一家运营商可能需要为成千上万个企业客户提供以太网专线服务，每个客户都需要自己的VLAN隔离——4094个VLAN几天就用完了。

解决这个问题的是QinQ（也称为802.1ad，或"Provider Bridging"）。QinQ的核心思想就是"VLAN套娃"——在标准的802.1Q标签外面再套一层新的802.1Q标签。外层标签称为服务提供商标签（Service Tag, S-Tag），用于标识运营商的网络；内层标签称为客户标签（Customer Tag, C-Tag），由客户自己管理。这样一来，可用的VLAN数量就变成了4094 × 4094 = 约1677万个——完全够用。

QinQ的帧格式是这样的：

+--------+--------+--------+--------+----------+
| 目的MAC | 源MAC | S-Tag | C-Tag | EtherType |
|  6字节  | 6字节 | 4字节 | 4字节 |   2字节   |
+--------+--------+--------+--------+----------+
                  |       |
              TPID=0x88a8  TPID=0x8100
             (S-Tag)       (C-Tag)

注意外层标签的TPID从0x8100变成了0x88a8——这是802.1ad专门分配给S-Tag的EtherType值。交换机的Trunk端口如果配置为服务提供商端口，会给来自客户端的帧自动添加S-Tag。在运营商网络内部，所有帧都带着双层标签传输。到了目的端，运营商剥离S-Tag，将原始的C-Tag帧交付给客户。

QinQ的优点很明显：扩展了VLAN数量、对客户透明、可以复用现有的802.1Q硬件。但它也有缺点：帧变得更长（最多可能达到1522+4=1526字节），而且丧失了C-Tag中的PCP优先级信息（因为运营商只能看到S-Tag的PCP）。

9.6 语音VLAN与LLDP-MED自动配置

在企业网络中，IP电话是一个特殊的终端设备。电话通常连接在用户的PC和交换机之间（PC串联在电话后面），电话需要自己的VLAN（语音VLAN），而PC使用另一个VLAN（数据VLAN）。问题来了：电话怎么知道它应该使用哪个VLAN？

这就用到了LLDP-MED（Link Layer Discovery Protocol - Media Endpoint Discovery）。LLDP-MED是LLDP的一个扩展，专门用于IP电话等媒体终端设备的自动配置。当交换机连接到IP电话时，交换机会通过LLDP-MED告诉电话："你的语音VLAN是VLAN 10，你的DSCP值应该设为46（EF），你的网络策略是XXX。"

配置语音VLAN时，交换机会做以下事情：

  • 在Trunk端口上允许语音VLAN通过
  • 对发送给电话的帧打上语音VLAN标签
  • 对来自电话的帧保留语音VLAN标签
  • 对来自PC的帧（通常连接在电话的Pass-through端口上）去标签，分配数据VLAN

很多运维事故都是因为语音VLAN配置不当导致的。比如，没有配置LLDP-MED导致电话获取不到正确的VLAN信息，或者语音VLAN的QoS优先级没有配置导致通话质量差。在大型企业中，IP电话的排障常常是网络工程师最头疼的问题之一——因为问题可能出在VLAN配置、LLDP协商、DHCP选项、或QoS策略中的任何一个环节。

9.7 PVID配置错误：一次真实的"惨案"

真实案例：某金融公司在进行一次交换机替换时，工程师将新交换机的端口配置为Trunk模式并连接到核心交换机，但忘记修改端口的PVID（默认PVID=1）。而核心交换机的Trunk端口Native VLAN配置为VLAN 99。结果，新交换机发往核心交换机的BPDU、CDP/LLDP等二层控制帧都打上了VLAN 1的标签（因为PVID=1），而核心交换机期望这些帧是"去标签"的（因为Native VLAN=99）。于是，核心交换机把这些帧当成了VLAN 1的普通数据帧丢弃了（因为Trunk端口没有允许VLAN 1通过）。最终，新交换机无法成功建立STP邻居，STP认为出现了环路而阻塞了所有端口，导致整个机柜的网络中断近两个小时。

这个故事的教训是：修改端口模式后，一定要检查PVID/Native VLAN的一致性。对Trunk端口来说，PVID应该等于对端的Native VLAN。如果Trunk端口的PVID和Native VLAN不匹配，就会导致控制帧被错误标记或丢弃。很多"奇怪"的二层连通性问题，根因都在这个细节上。

9.8 总结与预告

这一章我们深入探讨了VLAN——现代交换网络最重要的技术之一。我们从广播域隔离的需求出发，分析了802.1Q标签的四个字段，对比了Access、Trunk、Hybrid三种端口类型的行为差异，讨论了Native VLAN的安全风险，了解了运营级网络中的QinQ技术，还看了一个PVID配置错误的真实案例。你现在应该能够理解：VLAN的本质是在同一个物理网络上创建多个逻辑隔离的二层网络，每个VLAN是一个独立的广播域。

但是，当我们把多条物理链路连接在一起构成Trunk时，一个新的问题出现了：如果一条Trunk链路不够用怎么办？你可以在两台交换机之间接两条甚至四条物理链路，但STP会让其中一半处于阻塞状态——这不是浪费吗？下一章我们就来解决这个问题。链路聚合与LACP——把多条物理链路捆绑成一条逻辑链路，既增加带宽又提供冗余，还不被STP阻塞。这才是真正的高可用网络设计。
