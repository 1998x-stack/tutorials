第11章 以太网流控与QoS

大家好。前面的章节我们解决了网络的连通性问题——交换机怎么转发、环路怎么避免、带宽怎么聚合。这些都非常重要，但它们只回答了"能不能通"的问题。一旦网络"通了"，新的问题马上浮现出来：当网络出现拥塞时，谁的流量会被优先处理？谁的数据会被丢弃？

在一个真实的网络中，流量是有"贵贱"之分的。比如，语音通话的IP包如果延迟超过150毫秒，人耳就能明显感觉到通话质量下降；而一个后台数据备份任务晚几分钟完成，完全没人关心。然而，如果不加任何干预，交换机对待所有帧都是一视同仁的——先进先出，塞不下就丢。这种"公平"反而导致了关键流量得不到保障。

这一章我们讲流控（Flow Control）和QoS（服务质量，Quality of Service）——它们回答了"当网络不够用时，怎么分配"这个问题。这两个技术是现代以太网从"尽力而为"走向"可保障"的关键。

11.1 802.3x PAUSE帧：简单粗暴的"别发了"

最早的以太网流控机制是802.3x PAUSE帧。它的工作方式非常直接：当接收端的接收缓存快要溢出时，向发送端发送一个PAUSE帧，告诉对方"暂停发送X微秒"。发送端收到PAUSE帧后，会在指定的时间内停止发送任何数据。

PAUSE帧是一种MAC控制帧，使用EtherType 0x8808，目的MAC地址是01-80-C2-00-00-01（这是一个组播MAC，但是不会被交换机转发到其他端口——它是"点到点"的流控机制）。PAUSE帧的主体包含一个2字节的暂停时间（Pause Time）字段，单位是512比特时间（即512个比特的传输时间）。

PAUSE帧的工作流程如下：

发送端                        接收端
  |                              |
  |======= 正常数据帧 ==========>|
  |                              |  // 接收缓存快满了
  |<====== PAUSE (time=100) =====|
  |                              |  // 发送端暂停传输
  |    (等待100个512比特时间)     |
  |                              |
  |======= 正常数据帧 ==========>|
  |                              |
  |<====== PAUSE (time=0) =======|  // 缓存恢复了，通知继续发送
  |                              |

PAUSE帧看起来很方便，但它有一个致命的缺点：它是"全端口"暂停。当接收端口因某个拥塞流而触发PAUSE时，发送端口会被迫暂停所有出方向的流量——包括那些去往其他不拥塞的目的地的帧。一个接收端的拥塞会"传染"给发送端的所有流量，这种现象被称为Head-of-Line Blocking（队头阻塞）。想象一下，因为一个慢速接收者，所有快速接收者的数据都被堵在了发送端——这显然不合理。

因此，PAUSE帧在大多数场景下是不建议启用的。最好的实践通常是关闭PAUSE帧，让TCP的端到端拥塞控制来管理流量。但在一些特殊场景（比如无损网络）中，PAUSE帧的前身提供了重要的设计思路——这就是PFC。

11.2 802.1Qbb PFC：按优先级暂停

为了解决PAUSE帧的"一刀切"问题，IEEE 802.1Qbb工作组在2010年定义了优先级流控（Priority-based Flow Control, PFC）。PFC是无损以太网（Lossless Ethernet）的核心技术，也是RoCEv2（RDMA over Converged Ethernet）能运行在以太网上的前提条件。

PFC的原理非常巧妙：它将以太网的流量划分为8个优先级（对应802.1Q标签中的PCP字段，优先级0~7），并为每个优先级独立维护一个发送队列和接收缓存。当某个优先级的接收缓存接近溢出时，交换机只发送针对这个特定优先级的PAUSE帧，其他7个优先级的流量完全不受影响。

PFC帧的格式与802.3x PAUSE帧相似，但暂停时间字段换成了一个优先级使能向量（Priority Enable Vector）加上8个优先级暂停时间（每个优先级2字节）：

+---------------------+--------+--------------------------------+
| 字段                | 字节数 | 说明                           |
+---------------------+--------+--------------------------------+
| Destination MAC     | 6      | 01-80-C2-00-00-01              |
| Source MAC          | 6      | 发送端MAC地址                  |
| EtherType           | 2      | 0x8808 (MAC Control)           |
| MAC Control Opcode  | 2      | 0x0101 (PFC)                   |
| Priority Enable     | 2      | 8比特使能位 + 8比特保留        |
| Priority 0 Pause    | 2      | 优先级0的暂停时间               |
| Priority 1 Pause    | 2      | 优先级1的暂停时间               |
| ...                 | ...    | ...                            |
| Priority 7 Pause    | 2      | 优先级7的暂停时间               |
+---------------------+--------+--------------------------------+

PFC的应用场景非常明确：RoCEv2（RDMA Over Converged Ethernet，即融合以太网上的RDMA技术）需要无损传输。RDMA是一种"内核旁路"的数据传输技术，数据直接从应用缓冲区到网卡，不需要经过操作系统内核。如果丢包了，RDMA的恢复机制非常昂贵——它需要重传整个数据包甚至整个连接，性能会急剧下降。因此，RoCEv2的流量（通常映射到优先级3或4）需要PFC来保证"零丢包"。

真实案例：某公司部署了一套基于RoCEv2的分布式存储集群，链路层使用25GbE。部署后发现存储性能极不稳定——时延抖动严重，IOPS波动高达50%。排查了整整一周，最终发现交换机上没有启用PFC。当网络出现瞬时拥塞时，RoCEv2的帧被普通数据帧一起"公平"地丢弃，RDMA的重传机制导致性能雪崩。启用PFC并将RoCEv2流量映射到优先级3后，存储性能恢复了稳定，IOPS波动降到了5%以内。这个案例说明：PFC不是锦上添花，而是RoCEv2的刚性需求。

11.3 数据中心TCP Incast问题

在数据中心中，有一种非常特殊的拥塞场景叫TCP Incast。它发生在"多对一"的模式下：多个同时向同一个接收端发送数据的场景。比如，在分布式存储系统中，一个客户端请求读取一个文件块，这个文件块被分布存储在多个存储节点上（通常是8~16个节点）。所有存储节点几乎同时向客户端发送数据，导致客户端交换机的入端口缓存在瞬间被淹没。

TCP Incast的典型特征是什么？吞吐量骤降。由于交换机缓存溢出导致丢包，所有并发的TCP连接同时进入超时重传（Retransmission Timeout, RTO）状态。TCP的RTO最小通常是200ms（在很多操作系统中），这意味着在200ms内，所有并发连接都处于"静默"状态，吞吐量几乎降为零。200ms后，所有连接同时恢复发送，然后再次导致缓存溢出——如此反复，形成一个吞吐量的"锯齿"波形。

解决TCP Incast有几种常用的方法：

  • RED/WRED（随机早期检测）：在缓存溢出之前，随机丢弃一些帧，让部分TCP连接提前降低发送速率，避免所有连接同时超时。
  • 调整TCP的RTO最小值：将RTOmin从200ms降低到1ms或更低——但很多操作系统不允许这么低的RTO。
  • 启用ECN（显式拥塞通知）：在缓存接近溢出时，交换机标记帧的ECN位，TCP发送端根据ECN标记降低发送速率——不需要丢包。
  • 使用PFC：优先级流控可以在缓存溢出前暂停发送端，避免丢包。

11.4 队列调度：SP、WRR与DRR

QoS的核心是队列调度——当多个出站队列都有数据等待发送时，交换机按照什么顺序从各个队列中取出帧进行发送。最常见的三种调度算法是：严格优先级（Strict Priority, SP）、加权轮询（Weighted Round Robin, WRR）和亏空轮询（Deficit Round Robin, DRR）。

严格优先级（SP）：优先级最高的队列有数据就优先发送，只有当高优先级队列为空时才发送低优先级队列。SP的好处是低延迟——高优先级流量永远不需要等待。但缺点也很明显：如果高优先级流量一直不停，低优先级流量可能会被"饿死"（Starvation），永远轮不到发送。所以SP通常与流量监管（Traffic Policing）配合使用——先限制高优先级流量不超过某个带宽，确保低优先级流量有"喘息"的机会。

加权轮询（WRR）：每个队列被分配一个权重（Weight），交换机按权重比例轮流从各个队列中取帧发送。比如队列1权重=3，队列2权重=1，队列3权重=1，那么每轮调度中，队列1发送3个帧，队列2发送1个，队列3发送1个。WRR保证了所有队列都能得到服务，不会出现"饿死"的情况。但WRR的一个问题是：它只关心帧的数量，不关心帧的大小。如果一个队列的帧特别大（比如巨型帧jumbo frame），即使数量比例公平，实际带宽分配也会失衡。

亏空轮询（DRR）：DRR是WRR的改进版本，它解决了帧大小不均的问题。每个队列维护一个赤字计数器（Deficit Counter），每个轮次给队列增加一个配额（Quantum）。如果当前帧的大小小于或等于赤字计数器的值，就发送这个帧，并扣减相应的赤字；如果帧太大，就保留赤字到下一轮。DRR保证了基于字节的带宽公平分配，而不仅仅是基于帧的数量。DRR的缺点是实现比WRR复杂一些，计算开销稍大。

在实际产品中，很多交换机使用的是这些算法的混合模式——比如"SP + WRR"混合调度：高优先级队列使用SP，低优先级队列使用WRR。这样既能保证关键流量的低延迟，又能防止低优先级流量被饿死。

11.5 DSCP与802.1p优先级映射

QoS通常涉及两个层面的优先级标记：

  • 二层：802.1p（即802.1Q标签中的PCP字段，3比特，0~7）
  • 三层：DSCP（Differentiated Services Code Point）（IP头中的ToS字段的高6比特，0~63）

网络管理员需要建立从DSCP到802.1p的映射关系。通常，一组合理的映射是：

流量类型 DSCP值 DSCP别名 802.1p PCP 典型应用
网络控制 48 (0x30) CS6 7 路由协议(BGP/OSPF)、STP BPDU
语音 46 (0x2E) EF 6 IP电话RTP流
视频会议 34 (0x22) AF41 5 Zoom/Teams视频
关键数据 26 (0x1A) AF31 4 RoCEv2、存储流量
批量数据 18 (0x12) AF21 3 大文件传输、备份
运营管理 10 (0x0A) AF11 2 SSH、SNMP、Syslog
尽力而为 0 (0x00) BE 1 普通上网流量
背景流量 8 (0x08) CS1 0 P2P下载、软件更新

这张表给出了一个常用的DiffServ标记方案。需要注意的是，并不是所有网络都使用相同的映射——不同的组织有不同的QoS策略。但一个基本原则是：语音和视频流量不能走尽力而为的队列，否则抖动和延迟会严重影响用户体验。

一个重要的提醒：QoS是一个"端到端"的系统。如果只有你的接入交换机正确标记了DSCP和PCP，但核心交换机或路由器没有相应的队列配置，所有的标记都白费了。更糟糕的是，如果上游设备不支持QoS或者配置错误，它可能直接丢弃带有高优先级标记的帧。因此，QoS的设计必须贯穿整个网络路径——从终端到核心再到出口路由器，每一跳的QoS策略需要保持一致。

11.6 PFC优先级与RoCEv2的映射实践

在数据中心中，PFC的优先级分配是一个精心设计的决策。下面是典型的PFC优先级到流量类型的映射：

PFC优先级 802.1p PCP 典型流量类型 是否需要无损
7 7 网络控制 (BPDU, 路由协议) 否
6 6 IP电话语音 (RTP) 否
5 5 视频会议 否
4 4 RoCEv2 存储/计算流量 是
3 3 iSCSI / FCoE 存储流量 是
2 2 管理流量 (SSH, SNMP) 否
1 1 尽力而为数据 否
0 0 背景流量 (备份, 批量传输) 否

在这个映射中，仅有两个优先级（3和4）启用了PFC，其余6个优先级使用传统的丢包式拥塞控制。这种"部分无损"的设计是经过慎重考虑的——因为PFC也有代价：它会导致拥塞扩散（Congestion Spread）。如果PFC链路上某个优先级出现拥塞，PAUSE帧会一路向上游传播，可能导致整个网络中的该优先级流量都受到影响。因此，只对真正需要无损的流量启用PFC，是一个最佳实践。

11.7 总结与预告

这一章我们探讨了以太网的流控和QoS机制。从802.3x PAUSE帧的局限性说起，我们深入了解了PFC如何通过"按优先级暂停"实现无损以太网，以及它在RoCEv2场景中的关键作用。我们还讨论了TCP Incast这个数据中心特有的拥塞场景和ECN缓解方案。最后，我们学习了SP、WRR、DRR三种队列调度算法和DSCP-to-802.1p的优先级映射。QoS不是一个"配置一下就能好"的功能——它需要从端到端的视角来规划和调优。

到目前为止，我们讨论的所有流量都是"单播"的——从一个源发往一个目的。但现实中还有一种重要的通信模式：一对多的组播（Multicast）。无论是IPTV直播、金融行情推送还是视频会议中的共享桌面，组播都在背后默默工作。下一章，我们将学习以太网如何处理组播流量——以及为什么IGMP Snooping是交换机上不可或缺的功能。
