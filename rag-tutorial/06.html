第六章：向量嵌入技术

大家好。前面几章我们把文档切成了块，现在要做最关键的一步——把这些文本块转化为计算机能"理解语义"的数字表示。这个转化的产物就是Embedding（嵌入向量），它是整个语义搜索的基石。

想象一下：你无法直接问一台计算机"找一段和这个内容相近的文字"，因为计算机只懂数字。Embedding就是一座桥梁，把"RAG的全称是检索增强生成"这段话，映射成一组浮点数（例如[-0.023, 0.451, -0.178, ...]），让计算机可以计算两段文本之间的"语义距离"。这个距离越近，语义越相似。

选对了Embedding模型，你的RAG系统就成功了一半。选错了——哪怕后面的检索和生成再精妙，也无法弥补语义表示层面的缺陷。

一、Embedding的直觉：语义空间

我们可以把高维的embedding向量投射到2D/3D空间中想象："猫"和"狗"的向量距离很近（它们都是宠物/哺乳动物），但"猫"和"总统"的距离很远。更重要的是，Embedding可以捕捉类比关系——经典的例子是"国王 - 男人 + 女人 ≈ 女王"。

在RAG的语境中，我们需要embedding模型满足：一个问题的embedding，应该和包含答案的文档片段embedding在语义空间里尽可能接近。比如"RAG是什么？"这个查询的向量，应该靠近"RAG是检索增强生成的缩写"这段文本的向量。

三、稀疏Embedding vs 稠密Embedding

很多人以为embedding就是"一段文本变成一个向量"，但实际上有两种风格迥异的embedding：

  • 稠密Embedding（Dense Embedding）：如BGE、OpenAI ADA。每个元素都是非零的浮点数，维度在256~3072之间。优势是捕捉语义相似性，"苹果"和"iPhone"可以很近。劣势是对精确关键词匹配不敏感。
  • 稀疏Embedding（Sparse Embedding）：如SPLADE、BGE-M3的sparse输出。大多数维度为0，只有"重要词"对应的维度非零。优势是精确匹配和术语检索。典型应用是BM25的向量化版本。

我们会在第8~10章详细讨论两种检索方式的融合策略。这里先记住：稠密embedding管"语义"，稀疏embedding管"精确匹配"。

四、ColBERT的多向量表示

传统的embedding模型将一段文本压缩为一个向量（单向量表示），这有信息损失——"苹果很好吃"和"苹果公司发布了新产品"这两句话的向量可能会被平均成一个模棱两可的"苹果"。

ColBERT（Contextualized Late Interaction over BERT）采用了不同的思路：每个token都有一个独立的向量。检索时，计算查询和文档的所有token对之间的相似度，然后用MaxSim（取每个查询token与最相似文档token的相似度）聚合。这种方法显著提升了细粒度匹配的质量，尤其适合命名实体和精确信息检索。

记忆口诀：单向量Embedding是"把整段话画成一幅画"；ColBERT是"把每个字都画成一幅画，然后比较时看每个字分别在对方那里找到最像的"。代价是存储和计算复杂度增加了（需要存储N个向量而不是1个），但检索质量在精细化场景下有显著提升。

五、使用BGE模型生成Embedding

下面是使用BGE模型生成embedding的完整代码示例：

from sentence_transformers import SentenceTransformer
import numpy as np

# 加载BGE模型
# 注意：BGE模型要求特殊的前缀格式
model = SentenceTransformer("BAAI/bge-m3")

# BGE的重要特性：查询和文档需要不同的前缀
# 这大幅提升了检索精度
queries = ["RAG系统的核心组件有哪些？"]
documents = [
    "RAG系统由索引、检索、增强和生成四个核心组件组成。",
    "向量数据库Milvus支持GPU加速的ANN搜索。",
    "大语言模型的幻觉问题可以通过RAG技术缓解。"
]

# 正确使用query_prefix和passage_prefix
query_embeddings = model.encode(
    queries,
    normalize_embeddings=True,
    prompt_name="query"  # BGE会自动添加 "Represent this sentence for searching..."
)

doc_embeddings = model.encode(
    documents,
    normalize_embeddings=True,
    prompt_name="passage"  # 文档不需要special prefix处理
)

# 计算余弦相似度
similarities = np.dot(query_embeddings, doc_embeddings.T)
print("相似度矩阵:")
for i, q in enumerate(queries):
    for j, d in enumerate(documents):
        print(f"  Q: {q[:20]}... ↔ D{j}: {similarities[i][j]:.4f}")

# 输出最大相似度索引
best_idx = similarities[0].argmax()
print(f"\n最匹配文档: {documents[best_idx]}")
print(f"相似度分数: {similarities[0][best_idx]:.4f}")

六、领域微调：用Triplet Loss让Embedding更懂你

通用embedding模型在通用场景下表现不错，但在专业领域（医学、法律、金融）可能不够精准。一个有效的手段是用你自己的数据微调embedding模型。

最常用的微调方式是Triplet Loss。你需要构造三元组：(Anchor, Positive, Negative)。Anchor是查询，Positive是相关的文档，Negative是不相关的文档。训练目标是让Anchor和Positive的距离小于Anchor和Negative的距离，且margin足够大。

# Triplet Loss 微调框架示例
from sentence_transformers import SentenceTransformer, losses, InputExample
from torch.utils.data import DataLoader

model = SentenceTransformer("BAAI/bge-small-zh-v1.5")

# 构造你的领域数据三元组
train_examples = [
    InputExample(texts=["什么是RAG", "检索增强生成", "今天天气很好"]),
    InputExample(texts=["向量数据库有哪些", "Milvus和Chroma", "如何做番茄炒蛋"]),
    # ... 更多你的领域数据
]

train_dataloader = DataLoader(train_examples, batch_size=16, shuffle=True)
train_loss = losses.TripletLoss(model=model)

model.fit(
    train_objectives=[(train_dataloader, train_loss)],
    epochs=10,
    warmup_steps=100,
    output_path="./fine_tuned_bge",
)

微调后的模型在特定领域的检索Recall通常可以提升5~15个百分点。而且由于BGE是开源模型，你可以完全掌控微调过程。

微调最佳实践：（1）至少准备500个三元组才能看到明显效果；（2）使用Hard Negative Mining——选择那些"看起来相关但其实不相关"的困难样本作为Negative；（3）先冻结底层再解冻——前几轮只训练顶层适配器，后期再解冻全部层进行精调；（4）微调后务必在固定验证集上评估，防止过拟合。
