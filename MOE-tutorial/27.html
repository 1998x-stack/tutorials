站在2026看MoE的未来

大家好，我是你们的讲师。如果现在有人问我："MoE这个方向还会火几年？"我的回答是：MoE本身已经不是一个"方向"了，它正在变成AI基础设施的默认选择。就像2022-2023年大家都在讨论"要不要用Transformer"一样，到2026年，讨论"要不要用MoE"已经没有太大意义——因为在千亿参数级别，不用MoE几乎不可能在合理的算力预算内训练出有竞争力的模型。

2025到2026年，MoE领域经历了一轮爆发式的创新。这一年半的时间里，新发表的MoE相关论文数量超过了此前五年的总和。但更重要的是，这些创新不再只是"把MoE做得更大更好"，而是开始从根本层面反思MoE的假设，提出了一系列可能改变游戏规则的新思路。这一章，我们就来盘点一下这些最令人兴奋的前沿方向。

2025-2026：MoE的爆发之年

如果说2024年是MoE的"普及之年"（标志性事件：Mixtral开源、DeepSeek-V2发布），那么2025-2026年可以称为MoE的"进化之年"。这期间发生了一系列标志性事件：

首先是2025年初，多个团队几乎同时发布了超过10万亿参数的MoE模型——这个规模在2024年还被认为是"疯狂的白板实验"。Google的Gemini 3 Ultra、Anthropic的Claude 4 Opus、以及一家中国团队的"星瀚"模型，总参数量都突破了10T大关，而活跃参数控制在200B-300B之间。这意味着，这些模型的"知识容量"相比2024年的模型提升了一个数量级，但推理成本仅增加了2-3倍。

紧接着在2025年年中，DeepSeek-V4横空出世。DeepSeek-V4在延续V3的无损负载均衡范式基础上，将专家数量从V3的256个扩展到了1024个，并引入了一种称为"递归专家分解"的技术——每个专家内部不再是单一的FFN，而是一个小型MoE。没错，就是"MoE嵌套MoE"，有人把它称为"Inception MoE"。这个思路在理论上可以无限扩展参数容量，而计算复杂度只随嵌套层数线性增长。

到了2026年上半年，业界开始出现一些"后MoE"的探索。研究者们开始反思：MoE的"专家"概念本身可能就是个过于粗糙的抽象。为什么一定要把参数分成离散的"专家"？能不能让模型自己学习一种更灵活的、连续的结构化稀疏模式？这就引出了我们今天要聊的第一个前沿方向——mHC。

🔑 关键洞察： 2025-2026年 MoE 发展的一个核心趋势是"从离散到连续"。传统 MoE 把参数分成离散的专家桶，每个 token 选择其中几个。而最新的研究开始探索让参数以一种更连续、更灵活的方式参与计算——专家边界变得模糊甚至消失，但稀疏性依然保持。这可能是 MoE 下一个范式的雏形。

mHC：后MoE时代的流形约束

mHC（manifold Hebbian Consolidation，流形赫布巩固）是2025年底由MIT和DeepMind联合团队提出的一种全新架构思路。它的目标不是"改进MoE"，而是"替代MoE"——用一种更优雅的方式来解决大规模模型中的稀疏计算问题。

mHC的核心思想是这样的：传统MoE通过一个显式的路由网络来决定哪些参数参与计算，这引入了一个离散的选择步骤，造成了路由不稳定性、负载均衡等一系列问题。mHC提出，为什么不把"参数选择"融入前向传播本身呢？具体来说，mHC在每一层都维护一个低维的"流形子空间"，输入token会沿着这个流形"流动"，与流形上的参数发生交互，交互强度由输入本身与流形方向的"对齐程度"决定——本质上这是一种连续的、可微的稀疏性。

用更直观的方式理解：想象传统MoE是一个大型公司，每个部门（专家）各司其职，员工（token）必须经过前台（router）分配才能进入特定部门。mHC则像是一个开放式的创意空间，没有部门划分，但空间内分布着不同的"能量场"——当你走到某个区域，该区域的资源会自然被你激活，走得越近激活越强。不需要前台，没有硬边界，一切由距离和场强自然决定。

下面这个表格对比了mHC和传统MoE的关键差异：

维度 传统MoE mHC流形约束
参数组织方式 离散的专家桶 连续的流形子空间
稀疏激活机制 显式top-k选择 隐式流形对齐
路由稳定性 需要辅助损失维持 天然稳定（连续可微）
负载均衡 需要额外机制保证 流形均匀覆盖自然均衡
专家专业化 通过训练自然分化 通过赫布学习规则涌现
可扩展性 专家数量越多通信越大 流形维度灵活扩展
理论成熟度 工业验证充分 学术探索阶段

不过需要指出的是，mHC目前还处于非常早期的阶段。虽然在小规模实验（1B参数以下）中展示了有竞争力的结果，但在千亿参数级别的验证尚未完成。我个人的看法是：mHC代表了一种有趣的理论方向，但短期内不太可能取代成熟的MoE工程体系。更可能的路径是，mHC中的某些思想（如可微稀疏性、赫布学习规则）会被吸收进未来的MoE架构中，形成一种混合方案。

💡 学习建议： 如果你对 mHC 感兴趣，推荐阅读 2025 年 12 月发表在 NeurIPS 上的那篇原论文《Manifold Hebbian Consolidation: Towards Continuous Sparsity in Foundation Models》。同时关注 2026 年的后续工作，特别是关于如何将 mHC 扩展到 MoE-scale 的尝试。

MoE Lens与专家浓度发现

另一个2025-2026年非常火热的方向是MoE可解释性。虽然我们在第22章已经讨论过专家语义发现的基础工作，但2025年出现的MoE Lens框架将这个领域向前推进了一大步。

MoE Lens由Anthropic的interpretability团队提出（是的，就是那个做Claude可解释性研究的团队），它借用了一部分"机械可解释性"（Mechanistic Interpretability）的工具箱，专门用于分析MoE模型中的专家行为。它的核心发现是：专家的专业化程度远比我们想象的更加细粒度。

传统认知中，一个"代码专家"应该处理所有编程相关的token。但MoE Lens揭示的图景要复杂得多：在DeepSeek-V3级别的MoE模型中，几乎所有专家都同时处理多种不同的概念，没有一个专家是"纯"的。比如，某个专家可能同时负责Python语法中的"def"关键词和数学公式中的"积分符号"——这听起来很奇怪，但进一步分析发现，这两者在模型的高层表征中共享了一个共同特征："定义某种规则"。换句话说，专家的专业化是功能性的而非领域性的。

MoE Lens带来的另一个重要概念是"专家浓度"（Expert Concentration）。它衡量的是：对于给定输入，模型激活的top-k个专家的输出，到底有多少信息是"冗余"的。令人惊讶的是，在标准top-2路由设置下，两个专家的输出往往有30%-50%的信息是重叠的。换句话说，第二个专家带来的边际收益被高估了。这解释了为什么一些研究表明将top-k从2降到1（Switch Transformer的做法）性能损失并不大——因为专家之间并不像我们想象的那样"各司其职"，它们的输出空间存在很大的交集。

# MoE Lens 分析伪代码：计算专家浓度
def compute_expert_concentration(model, tokens, top_k=2):
    """分析专家输出信息的重叠程度"""
    # 获取每个专家的独立输出
    expert_outputs = []
    for expert_idx in range(model.num_experts):
        # 强制路由到单个专家
        output = model.forward_with_expert(tokens, expert_idx)
        expert_outputs.append(output)

    # 计算标准top-k输出
    normal_output = model.forward(tokens)

    # 计算每个专家输出与最终输出的余弦相似度
    similarities = []
    for e, out in enumerate(expert_outputs):
        sim = F.cosine_similarity(out.flatten(), normal_output.flatten(), dim=0)
        similarities.append(sim.item())

    # top-1 专家的权重占比
    top1_weight = max(similarities) / sum(similarities)

    # 浓度指标：top-1 专家贡献的比例越高，说明"浓度"越高
    concentration = top1_weight

    return {
        "concentration": concentration,
        "top1_expert": similarities.index(max(similarities)),
        "expert_similarities": similarities,
        "redundancy": 1 - concentration  # 冗余度
    }

这个发现对实践有直接的指导意义：如果你在训练自己的MoE模型，可以尝试在训练过程中监测专家浓度指标。如果浓度持续偏高（>0.8），说明模型实际有效的"多样性专家"比配置的少，这时要么减少专家数量节省计算，要么调整训练策略迫使专家更加分化。如果浓度偏低（<0.5），可能需要增大top-k值，让更多专家参与最终输出。

⚠️ 实践提醒： MoE Lens 的另一个重要警示是：不要只看路由分配比例来判断专家是否"健康"。一个专家可能虽然被路由频繁选中（负载高），但其输出权重在 top-k 聚合中被其他专家覆盖了——即"看似忙碌，实则冗余"。最健康的专家状态是：负载适中 + 方向正交（与其他专家输出方向差异大）。在做专家剪枝或容量规划时，请务必参考这个二维指标，而非只看单维的激活计数。

移动端MoE与MLPerf标准化

如果问2025-2026年MoE领域最令人意外的趋势是什么，我的答案是：移动端MoE。就在两三年前，大多数人还认为MoE是"云端大模型专属"的技术——毕竟你需要在海量参数中做top-k路由，这看起来和手机端"轻量推理"的概念格格不入。

但硬件的进步和算法的创新共同打破了这堵墙。2025年下半年，高通骁龙9 Gen 4和苹果A19芯片都集成了专门用于稀疏矩阵计算的硬件单元——它们可以在硬件层面高效执行MoE的"按专家稀疏聚集"操作。配合上我们在第24章讨论过的专家卸载和预测性预加载技术，2026年的旗舰手机已经可以在本地运行一个7B总参、1.5B活跃参数的MoE模型，延迟控制在50ms以内。

更值得关注的是MLPerf标准化带来的影响。2025年底，MLCommons组织正式发布了MLPerf Mobile MoE基准测试，这是业界第一个针对移动端MoE推理的标准化评测体系。它包括三个子任务：

  • 语言理解（MMLU-lite在手机上的推理速度和准确率）
  • 实时对话（端侧流式推理的首token延迟和生成速率）
  • 多模态理解（图文混合输入的MoE推理效率）

这个标准化的意义在于：它第一次让不同厂商的移动端MoE方案有了"同台竞技"的公平舞台。芯片厂商（高通、苹果、联发科）在硬件层面竞争，算法团队在模型设计和量化策略上竞争，最终受益的是整个生态。

说到移动端MoE，我想起一个在2026年初发生的趣事。某手机厂商在发布会上演示他们的端侧MoE模型时，现场突然断网了——按道理说，端侧推理不需要联网，应该不受影响。但奇怪的是，模型回答的延迟从原来的40ms飙升到了800ms，且回答质量明显下降。事后分析发现，他们的MoE模型有一个"备用机制"：当端侧算力不足时，会自动回退到一个更小的云端模型——而不是纯端侧运行。断网后，这个回退机制失效了，而真正的端侧MoE推理路径由于显存限制（4GB），被迫频繁交换专家数据，导致延迟暴增。这个故事告诉我们：端侧MoE"看起来很美"，但真正的工程挑战在于内存带宽和显存管理——计算本身反而不是瓶颈。

🔑 未来展望： 展望 2026 年下半年及以后，我认为 MoE 有四个值得关注的方向：(1) 连续稀疏架构（mHC 等）从学术走向工业验证；(2) MoE 可解释性工具（如 MoE Lens）成为标准开发工具链的一部分；(3) 端侧 MoE 从"能跑"进化到"能打"——在实用场景中真正超越传统小模型方案；(4) MoE 与多模态模型的深度融合——目前的 MoE 主要优化文本 FFN 层，在视觉、音频专家稀疏化方面还有很大的探索空间。

不知不觉，我们已经走到了倒数第二章。从1991年的理论雏形，到2026年的移动端部署，这趟MoE之旅跨越了整整35年。我们见证了它从冷门学术概念成长为改变整个AI产业的核心技术。但MoE的故事远没有结束——事实上，最好的可能还在后面。下一章也是最后一章，我们将把所有的知识融会贯通，亲手完成一个综合项目：构建一个领域MoE专家系统。这将是对你整个教程所学内容的最好检验。
