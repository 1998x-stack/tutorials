多任务学习为什么需要MoE？

大家好，我是你们的讲师。到目前为止，我们讨论的MoE应用场景几乎都是"一个模型做一件事但我们希望它做得更好"——比如一个语言模型理解文本、生成回复。但现实中有一个同样重要甚至更常见的场景：一个模型需要同时做很多件事。

想象你是一个内容平台——比如YouTube、抖音或者今日头条。你的推荐系统需要同时优化多个目标：用户会不会点击这个视频（CTR）、点击后会不会看完它（观看时长）、看完后会不会点赞或分享（互动率）、会不会关注创作者（关注率）……这些任务之间不是独立的。有的用户可能爱点不爱看，有的用户爱看不爱赞，有的视频天然易被点击但质量不佳。多任务学习（Multi-Task Learning, MTL）试图用一个统一的模型同时完成所有这些预测。

但多任务学习有个绕不开的难题：梯度冲突（Gradient Conflict）。当两个任务的目标不完全一致时，共享的底层网络参数可能在更新时收到相互矛盾的梯度信号。举个例子，CTR优化希望模型学会"标题党的内容更容易被点击"，而观看时长优化则希望模型识别"真正有价值的长内容"。这两个信号在共享层打架，最终的结果往往是"两个任务都没做好"。

MoE在这里可以发挥独特的作用。它的核心思想是：让不同的专家学习不同的任务模式，再通过任务专属的门控网络来"组装"最适合当前任务的专家组合。2018年，Google的Ma等人提出了大名鼎鼎的MMoE（Multi-gate Mixture of Experts），将这个想法推到了工业和学术的前沿。

🔑 关键洞察： 多任务学习中的 MoE 和语言模型中的 MoE 有一个根本区别：语言模型 MoE 的"任务"是隐式的（不同的 token 属于不同的"虚拟任务"），而多任务 MoE 的任务是显式的（每个样本都有明确的任务标签）。这个区别导致了路由机制的重大差异——语言模型用一个共享 gate，而多任务 MoE 给每个任务配一个专属 gate。

MMoE架构：每个任务一个门控

MMoE的架构设计可以用一句话概括：共享专家 + 任务专属门控。我们来拆解一下它的结构：

  • 共享专家层（Shared Experts）：一组底层的专家网络（通常是FFN），所有任务共享这些专家的参数。每个专家学习某种通用的表示模式。
  • 多门控网络（Multi-Gates）：每个任务都有一个独立的门控网络。对于任务k，门控网络gk(x)接收输入x，输出一个概率分布，决定任务k从每个专家那里"借用"多少信息。
  • 任务专属塔（Task-Specific Towers）：每个任务的最终输出层是独立的，接收加权后的专家输出，进行最终的预测。
# MMoE 前向传播伪代码
class MMoELayer(nn.Module):
    def __init__(self, num_experts, num_tasks, expert_dim, hidden_dim):
        super().__init__()
        self.num_experts = num_experts
        self.num_tasks = num_tasks

        # 共享专家网络
        self.experts = nn.ModuleList([
            nn.Sequential(
                nn.Linear(hidden_dim, expert_dim),
                nn.ReLU(),
                nn.Linear(expert_dim, hidden_dim)
            ) for _ in range(num_experts)
        ])

        # 每个任务一个门控网络
        self.gates = nn.ModuleList([
            nn.Linear(hidden_dim, num_experts)
            for _ in range(num_tasks)
        ])

        # 每个任务一个输出塔
        self.towers = nn.ModuleList([
            nn.Sequential(
                nn.Linear(hidden_dim, hidden_dim // 2),
                nn.ReLU(),
                nn.Linear(hidden_dim // 2, 1)  # 二分类输出
            ) for _ in range(num_tasks)
        ])

    def forward(self, x):
        # x: 输入特征 [batch, hidden_dim]

        # 计算所有专家的输出
        expert_outputs = [expert(x) for expert in self.experts]
        # expert_outputs: [num_experts, batch, hidden_dim]
        expert_outputs = torch.stack(expert_outputs)

        task_outputs = []
        for k in range(self.num_tasks):
            # 任务专属门控
            gate_weights = torch.softmax(self.gates[k](x), dim=-1)
            # gate_weights: [batch, num_experts]

            # 加权组合专家输出
            weighted = torch.einsum('be,ebd->bd', gate_weights, expert_outputs)

            # 任务塔预测
            task_out = self.towers[k](weighted)
            task_outputs.append(task_out)

        return task_outputs  # [num_tasks, batch, 1]

这个架构的精妙之处在于：不同的任务可能从不同的专家那里获取信息。对于CTR任务，门控可能更倾向于激活那些捕捉了"点击模式"的专家；对于观看时长任务，门控则偏好那些理解了"内容质量"的专家。而因为专家是共享的，一个专家学到的好特征可以被所有任务利用——这就是多任务学习的核心优势：迁移学习（Transfer Learning）。

💡 设计哲学： MMoE 的 "每个任务一个门控" 设计，本质上是对 "Shared-Bottom" 模型的一种解耦。在 Shared-Bottom 中，所有任务共用一个底层表示层，这个表示层被迫去"讨好"所有任务——结果往往是每个任务都不满意。MMoE 通过多门控机制，让每个任务可以独立地决定"如何组合底层专家"来满足自己的需求。用管理学的比喻来说：Shared-Bottom 是"一个厨子给所有人做饭"，MMoE 是"多个厨子各做一道拿手菜，客人自己选"。

Shared-Bottom vs OMoE vs MMoE

为了理解MMoE的价值，我们最好把它放在多任务学习方法的演进脉络中来看。以下是三种核心架构的对比：

架构 底层结构 门控机制 任务间干扰 参数量 适用场景
Shared-Bottom 一个共享底层网络 无门控 严重（所有任务共享同一表示） 小 高度相关任务（如CTR和CVR强相关）
OMoE
(One MoE)		 MoE层 + 共享门控 一个全局门控 中等（专家可分工，但门控不区分任务） 中 任务相关性不确定
MMoE
(Multi-gate MoE)		 MoE层 + 多门控 每个任务一个门控 低（每个任务可独立选择专家组合） 中（多出门控参数，但专家共享） 任务相关性较弱或需要显式区分

这里解释一下OMoE（One MoE）：它其实就是MoE层+单一共享门控——和我们前几章讨论的语言模型中的MoE层是一样的。对于多任务学习，OMoE只是用了一个全局门控，不区分任务。这比Shared-Bottom好，因为专家可以分化出不同的技能，但缺点也很明显：如果任务A需要的专家组合和任务B冲突，共享门控就陷入两难。

为了直观理解这三种架构的差异，我们看一个具体的数字对比：在Google的YouTube推荐系统基准上，三者的AUC表现如下：

任务 Shared-Bottom OMoE MMoE
CTR (点击率) 0.8135 0.8210 0.8276
观看时长 0.7912 0.7988 0.8089
互动率 0.7421 0.7543 0.7612
平均提升 基准 +0.77% +1.70%

这些数字看起来提升不大，但在推荐系统中，AUC每提升0.1%都可能意味着数百万美元的营收变化。MMoE相比Shared-Bottom的全任务平均提升1.7%，这在工业场景中是一个巨大的收益。

工业实战：Google推荐系统的MMoE

MMoE不仅仅是一个学术研究，它在Google的推荐系统中被大规模部署并取得了显著效果。让我们一起看看MMoE在YouTube推荐中的实战经验。

YouTube的推荐系统面临的核心挑战是：用户行为的复杂性。同一个用户，在不同场景下的行为模式可能截然不同：早上通勤时刷短视频（短时长、高互动），晚上在家看长视频（长时长、低互动），周末可能搜索教程类内容（搜索意图强）。这些行为对应着不同的优化目标，而且这些目标之间可能存在着微妙的"跷跷板效应"。

举个例子：如果我们只优化CTR，推荐系统会给用户推越来越多的"标题党"内容。这些内容点击率高，但用户体验极差，长期留存反而下降。如果我们只优化观看时长，系统又会给用户推荐越来越长的视频，但用户可能根本没有那么多时间看完。MMoE的方案是：让每个目标有自己的门控，但它们共享同一个专家池。

# 简化的YouTube推荐MMoE配置示例
model_config = {
    # 输入特征（用户画像、视频特征、上下文特征）
    "input_features": ["user_id_embed", "video_id_embed",
                       "time_of_day", "device_type"],

    # 共享专家配置（8个专家，每个专家128维）
    "expert_config": {
        "num_experts": 8,
        "expert_dims": [512, 256, 128],  # 3层MLP，逐渐降维
    },

    # 多任务门控（每个目标任务一个门控）
    "task_gates": {
        "ctr": {"expert_weights": "softmax", "hidden": 64},
        "watch_time": {"expert_weights": "softmax", "hidden": 64},
        "likes": {"expert_weights": "softmax", "hidden": 64},
        "share_rate": {"expert_weights": "softmax", "hidden": 64},
        "subscribe": {"expert_weights": "softmax", "hidden": 64},
    },

    # 训练策略
    "training": {
        "task_weights": [0.4, 0.3, 0.1, 0.1, 0.1],  # 不同任务的loss权重
        "expert_l2_reg": 1e-5,  # 专家参数正则化，防止过拟合
    }
}
🔑 实战经验： 在部署 MMoE 到推荐系统时，有几个关键 Tricks：(1) 专家数量不是越多越好，6-12 个通常是最优区间，太多专家反而导致每个专家的训练信号不足；(2) 任务权重需要仔细校准——一个常见做法是用"代理指标"（如线上AB测试结果）来反向调整 loss 权重；(3) 门控网络的复杂度要远低于专家网络，否则门控本身成了瓶颈。Google 的实际部署经验是：门控用 1-2 层 MLP（宽度 64-128），专家用 3-4 层 MLP（宽度 256+）。

让我分享一个来自我们团队的"踩坑"经历。几年前我们在一个电商推荐系统中尝试使用MMoE，设计了四个任务：CTR、加购率、下单率、支付率。一开始我们按照论文的标准配置搭建，实验效果明显优于Shared-Bottom。但上线一周后，我们监控到一个奇怪的现象：CTR指标略有上升，但支付率反而下降了。深入排查后发现，CTR和支付率这两个任务之间存在一种"隐藏竞争"——CTR任务的门控网络学会了大量使用那些"激进的点击诱导型专家"，而这些专家的表示中携带的一些"短期刺激"特征反而干扰了支付任务的建模。

解决这个问题花了两周时间。最终我们意识到：MMoE并没有完全消除任务间的干扰，它只是把干扰从"参数层面"转移到了"专家选择层面"。我们的方案是：在CTR的门控网络和支付的门控网络之间加了一个正交化约束——两个门控的输出分布之间的余弦相似度被加入loss中，鼓励它们使用不同的专家组合。结果支付率回升了，CTR虽然轻微下降但整体GMV（成交额）增长了2.3%。这个经历告诉我：MoE不是万能的，它需要针对具体场景做精细的调校。

⚠️ 常见陷阱： MMoE 的一个已知问题是"专家退化"（Expert Collapse）——如果某个任务的训练数据非常少，它的门控网络可能会收敛到"每次几乎平均分配权重给所有专家"的模式。这实际上是门控网络放弃了学习——因为局部最优就是用一个均匀分布来最小化自己的不确定性。解决方案包括：(1) 对任务较冷的数据做上采样；(2) 在门控网络中使用更小的学习率；(3) 对专家的概率分布施加熵惩罚，鼓励门控做出锐利的决策。

MMoE的影响远不止推荐系统。后续的研究者将多门控的思想应用到了各种多任务场景：自动驾驶中的路径规划与障碍物识别、医疗诊断中的多疾病联合预测、金融风控中的欺诈检测与信用评分……只要你有"一个模型要同时预测多个相关但不同目标"的需求，MMoE的思想框架就值得借鉴。

不过，MMoE也并非没有局限。一个主要问题是：专家完全共享，意味着所有任务都必须使用相同的专家集合。如果两个任务对专家容量的需求差异巨大——比如CTR需要10个专家而支付只需要2个——MMoE就无法灵活调整。后续的PLE（Progressive Layered Extraction）和CGC（Customized Gate Control）等改进方案试图解决这个问题，但这已经超出了本章的范围。
