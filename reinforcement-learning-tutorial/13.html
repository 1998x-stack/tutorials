第十三章：PPO与TRPO

同学们好，欢迎来到第十三章。上一章我们学习了Actor-Critic方法，大家应该已经体会到了"演员-评委"这个框架的优雅之处。但不知道你们有没有注意到一个问题：策略梯度方法的更新步长非常难调。步长太小，学得比乌龟还慢；步长太大，策略直接崩掉。这一章要讲的PPO和TRPO，就是专门解决这个问题的两大杀器。

先抛一个结论：PPO是当今RL领域应用最广泛的算法，没有之一。从Atari游戏到MuJoCo机器人，从Dota 2到ChatGPT的RLHF训练，PPO都是默认首选。它凭什么这么火？读完这章你就知道了。

13.1 策略崩溃问题

咱们先回忆一下策略梯度方法的更新公式：

∇J(θ) = E[ ∇log πθ(a|s) · A(s,a) ]

看起来很简单对吧？但这里藏着一个致命的陷阱：每次更新，我们都在改变策略πθ本身。而在监督学习中，你用的是固定的数据分布。一旦更新出问题，大不了重新训练。但RL不一样——策略变了，你下一轮采样的数据分布也跟着变。如果一次更新把策略推到了一个坏区域，智能体就会采集到大量低质量的轨迹，然后越陷越深，再也爬不出来。

这就是所谓的策略崩溃（Policy Collapse）。我亲眼见过一个同学训了三天三夜的策略梯度模型，最后一步更新把所有参数炸飞，累积回报从+500直接掉到-2000。他当时的表情我到现在都记得。

策略崩溃的原因很直接：策略梯度的步长没有一个天然的"刹车"机制。在参数空间中，相差一个很小步长的两个策略，在动作分布上可能天差地别。一次大步更新，足够让一个训练良好的策略完全失忆。

核心洞察：策略梯度的问题不在于"梯度方向不对"，而在于"走多远没谱"。步长需要根据策略变化的剧烈程度来动态调整——这就是TRPO和PPO的出发点了。

13.2 信任域方法 TRPO

2015年，John Schulman提出了TRPO（Trust Region Policy Optimization），给了策略更新一个"安全带"。

TRPO的核心思想是：每次更新时，强制新旧策略之间的KL散度不超过一个阈值δ。KL散度衡量的是两个概率分布之间的差异——KL越小，新旧策略的行为就越接近。

形式化地说，TRPO要解决这样一个约束优化问题：

maxθ   E[ πθ(a|s) / πold(a|s) · A(s,a) ]
s.t.   KL( πold(·|s) || πθ(·|s) ) ≤ δ

这个约束条件是什么意思呢？直白点说：你可以更新策略，但不能"变得太离谱"。KL散度这个小阀门，控制了每次更新的剧烈程度。

但TRPO的实现非常复杂。要解这个带约束的优化问题，需要用共轭梯度法（Conjugate Gradient）来近似计算Fisher信息矩阵的逆，还要做线搜索（Line Search）来确保每次更新都满足约束。这涉及到二阶导数、Fisher-vector乘积、Hessian矩阵近似……代码量动辄上千行，调试起来更是噩梦。

加深理解：KL散度"约束"和"惩罚"是两种不同的思路。TRPO用的是约束（硬性上限δ），而后面要讲的PPO-Adaptive KL可以理解成自适应惩罚。约束的好处是每步都安全，代价是实现复杂。惩罚的好处是简单，但需要精细调节惩罚系数。

TRPO在理论上非常漂亮——它保证了策略更新单调不减（至少是近似单调）。但实际落地时，绝大多数团队很快就放弃了它的完整实现，转而使用PPO。原因很简单：PPO更简单，效果一样好。

13.3 PPO-Clip

还是John Schulman，2017年，提出了PPO（Proximal Policy Optimization）。如果说TRPO是用"安全带"绑着策略，那PPO就是用"限速器"——不需要复杂的二阶优化，只需要一个巧妙的截断（Clipping）技巧。

PPO的损失函数长这样：

LCLIP(θ) = E[ min( rt(θ) · At,   clip(rt(θ), 1-ε, 1+ε) · At ) ]

其中 rt(θ) = πθ(at|st) / πold(at|st) 是新旧策略在动作上的概率比。

这个公式看起来是不是有点烧脑？咱们用直观的方式拆解一下：

  • 情况一：At > 0（好动作） —— 我们希望增加这个动作的概率。但rt被clip在1+ε以内，意味着你最多把概率提升到原来的1+ε倍。以ε=0.2为例，概率最多增加20%。再高不要了！
  • 情况二：At < 0（坏动作） —— 我们希望降低这个动作的概率。同样，rt被clip在1-ε以上，概率最多降低到原来的1-ε倍。降低幅度也被限死了。

min操作的作用是：取真实目标和clip目标中更小的那个。当At>0时，如果rt超过了1+ε，clip版本会小于真实版本，min选择clip版，梯度为0——停止增加；当At<0时，如果rt低于1-ε，min选择真实版（更小），但clip版本已经卡住了终点——实际上梯度也消失了。

你看，PPO不需要解任何约束优化问题，只需要在损失函数里加一个min和clip，就实现了和TRPO类似的效果。这就是"简单却有效"的典范。

踩坑实录：我第一次实现PPO的时候犯过一个极其愚蠢的错误——忘了重新计算log_probs。PPO每次更新都要计算rt，也就是新旧策略的概率比。我用的是同一个模型在更新前后的参数来算。但我在代码里直接用之前保存的log_probs当分母，没有传入新参数重新计算。结果rt永远等于1，clip永远不会触发，PPO直接退化成普通的策略梯度。整整两天查不出原因，直到我打印出rt的值才发现全是1。大家记住：一定要用当前策略重新计算动作的概率，再和旧概率做比值！

13.4 PPO实现细节

PPO虽然核心思想简单，但一个"工业级"的实现里包含了很多工程细节。我们来梳理一下完整的PPO算法流程：

# PPO-Clip 算法伪代码（单次迭代）

def ppo_iteration(policy, value_net, optimizer, trajectories):
    # 1. 从轨迹计算优势函数（使用GAE）
    advantages = compute_gae(trajectories.rewards,
                             trajectories.values,
                             gamma=0.99, lam=0.95)
    returns = advantages + trajectories.values  # 目标回报
    advantages = (advantages - advantages.mean()) / (advantages.std() + 1e-8)  # 标准化

    # 2. 保存旧策略的动作概率（用于计算r_t）
    old_log_probs = policy.log_prob(trajectories.states,
                                     trajectories.actions).detach()

    # 3. 多轮优化（通常3-10个epoch）
    for epoch in range(K_epochs):
        # 为每个epoch随机打乱数据
        indices = shuffle(range(len(trajectories)))

        for batch in minibatches(indices, batch_size=64):
            # 计算当前策略的log_prob
            curr_log_probs = policy.log_prob(batch.states, batch.actions)
            # 概率比 r_t
            ratios = torch.exp(curr_log_probs - old_log_probs[batch.indices])

            # PPO-Clip损失
            surr1 = ratios * advantages[batch.indices]
            surr2 = clip(ratios, 1 - epsilon, 1 + epsilon) * advantages[batch.indices]
            policy_loss = -min(surr1, surr2).mean()

            # 值函数损失（带clip）
            values_pred = value_net(batch.states)
            values_clipped = old_values + clip(values_pred - old_values,
                                               -epsilon, epsilon)
            value_loss = max((values_pred - returns[batch.indices])**2,
                             (values_clipped - returns[batch.indices])**2).mean()

            # 熵奖励（鼓励探索）
            entropy = policy.entropy(batch.states).mean()
            total_loss = policy_loss + 0.5 * value_loss - 0.01 * entropy

            # 反向传播
            optimizer.zero_grad()
            total_loss.backward()
            optimizer.step()

        # 可选：KL早停
        kl_div = kl_divergence(old_policy, policy, batch.states)
        if kl_div > target_kl * 1.5:
            print(f"KL散度超限 ({kl_div:.4f})，提前停止")
            break

    return policy, value_net

关键的实现细节：

  • GAE优势估计（第2行）：用λ参数在偏差和方差之间做权衡。λ=0相当于TD误差，λ=1相当于蒙特卡洛回报。实践中λ=0.95效果最好。
  • 多轮优化（第11行）：PPO一个重要创新是"重复使用同一批数据"——这在策略梯度中是反直觉的，但因为clip限制了更新幅度，所以是安全的。
  • 优势函数标准化（第5行）：把所有优势归一化成均值为0、方差为1。这能极大提升训练的稳定性，避免某些样本的极端优势主导梯度。
  • 值函数Clip（第28-31行）：和策略一样，值函数的更新也被clip约束住。防止价值估计剧烈抖动。
  • 熵奖励（第33行）：给策略加一个熵最大化的小偏置，鼓励探索。系数通常很小（0.01或更小），太大反而影响收敛。
  • KL早停（第37-40行）：可选但非常实用。如果某次更新后KL散度超过了预设阈值的1.5倍，就提前结束这一轮，防止策略崩坏。

调参心得：ε=0.2是PPO论文给出的默认值，在大多数任务上都工作良好。如果你发现训练不稳定，试试降低到0.1或0.15。如果发现学得太慢，可以稍微提升到0.25。同理，K_epochs（每批数据的重复使用次数）默认3-10，不要贪多——同一批数据重复用太多反而会过拟合。

13.5 PPO为什么称霸

聊了这么多原理和细节，我们来总结一下PPO为什么能成为RL领域的"万金油"：

  1. 实现简单：只需要几十行PyTorch代码就能跑通。对比TRPO需要共轭梯度和Fisher矩阵，PPO的门槛低太多了。
  2. 训练稳定：clip机制限制了单步更新的幅度，几乎不会出现策略崩溃。即使超参不太对，PPO也能给出可接受的结果。
  3. 通用性强：同样的超参设置，在Atari、MuJoCo、2D导航等不同任务上都能工作。不需要为每个任务单独调参。
  4. 样本效率尚可：通过多轮优化重复利用数据，比单步策略梯度方法的效率高得多。
  5. 生态完整：Stable-Baselines3、Ray Rllib、OpenAI Baselines等主流库的一等公民。社区资源丰富。

我们来做一个全面对比：

对比维度 PPO TRPO A3C
更新方式 Clip + 多轮SGD KL约束 + 共轭梯度 单步策略梯度
实现难度 低（~50行核心代码） 高（需要二阶导数） 低（~30行）
训练稳定性 高 高 低
样本效率 中高 中 低
计算开销 中 高 低
适用离散动作 是 是 是
适用连续动作 是 是 需改造
大规模分布式 容易 困难 容易
超参敏感度 低 低 高

说个内幕：OpenAI在训练ChatGPT的RLHF阶段，用的就是PPO。把RLHF拆开来看：策略（Policy）就是ChatGPT模型本身，奖励模型（Reward Model）是Critic，而PPO就是这个"演员-评委"系统的稳定器。没有PPO的clip机制，RLHF训练过程中语言模型很容易"说疯话"——一次更新就学会了重复无意义的胡言乱语因为那样能获得奇怪的奖励信号。PPO保证了每次更新都不会偏离原始语言模型太远。

当然，PPO也不是万能的。它的"近端"约束虽然带来了稳定，但也限制了单步更新的信息利用率。在某些对样本效率要求极高的场景下，PPO并不总是最优选择。另外，PPO的多个超参（ε、K_epochs、GAE的λ、熵系数等）虽然容错空间大，但要想达到极致性能，还是需要花时间调参。

从PPO和TRPO的故事中，我们可以学到RL算法设计的一个核心原则：限制更新幅度比追求最优步长更实用。TRPO选择了严格的约束（KL边界），PPO选择了软性的clip——后者在工程上胜出了。这提醒我们，在RL这个领域，一个"足够好且简单"的方案，往往比"精确但复杂"的方案走得更远。
