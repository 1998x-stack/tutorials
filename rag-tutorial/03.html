第三章：RAG核心架构

大家好，欢迎回到课堂。上一章我们打下了LLM的基础，现在可以正式进入RAG的核心架构了。如果你还记得第一章的四阶段总览，这一章就是把每个阶段放大来看，搞清楚它们之间如何协同、数据如何在管道中流动。

我常说，RAG系统就像一条有四个工位的智能生产线：索引工位（Index）负责把原材料加工成半成品，检索工位（Retrieve）负责按订单查找合适的零件，增强工位（Augment）负责把零件组装成待加工件，最后生成工位（Generate）负责精加工成成品。每一个工位都有独特的技术挑战，也有独立的优化空间。

一、四阶段架构深度解析

我们先用一张表把每个阶段的输入、输出和核心任务看清楚：

阶段 输入 输出 核心技术 离线/在线
Index 原始文档（PDF/HTML/DB/API） 向量索引 + 文本索引 文档解析、文本分块、Embedding 离线（可定时更新）
Retrieve 用户查询 Top-k 候选文档片段 向量搜索、BM25、混合检索 在线（毫秒级响应）
Augment Top-k 候选片段 + 原始查询 增强后的Prompt 重排序、查询重写、上下文压缩 在线（百毫秒级）
Generate 增强后的Prompt 带引用的最终答案 LLM推理、引用标注、格式控制 在线（数百毫秒~数秒）

值得注意的是，Index阶段是离线完成的——这意味着我们可以在构建索引时花费大量计算资源，不必担心延迟。而Retrieve→Augment→Generate是串联的在线路径，每个环节都贡献到总延迟中，因此对性能有严格要求。

二、离线索引与在线推理的数据流

为了让大家有更直观的理解，我们用一个文本形式的拓扑图来展示RAG系统的完整数据流：

┌─────────────────────────────────────────────────────────────┐
│                    离线索引管道 (Offline)                      │
│                                                              │
│  原始文档 ──→ 文档解析器 ──→ 文本分块 ──→ Embedding模型      │
│  (PDF/HTML)    (Unstructured)  (256-512 tokens)   │          │
│                                                     ↓        │
│                       ┌──────────────────┐                  │
│                       │  向量数据库        │                  │
│                       │  + 全文搜索引擎     │                  │
│                       └────────┬─────────┘                  │
└────────────────────────────────┼────────────────────────────┘
                                 │
┌────────────────────────────────┼────────────────────────────┐
│                    在线推理管道 (Online)                       │
│                                                              │
│  用户查询 ──→ 查询编码 ──→ 混合检索 ──→ 重排序 ──→ 上下文增强│
│               (相同的LLM)     (向量+BM25)   (Cross-encoder)  │
│                                                              │
│  增强后的Prompt ──→ LLM生成 ──→ 带引用的回答 ──→ 返回用户     │
│                                                              │
└─────────────────────────────────────────────────────────────┘

这个拓扑揭示了一个重要设计原则：离线索引和在线推理使用相同的Embedding模型。如果离线用BGE-M3索引文档，但在线用OpenAI ADA查询编码，两个向量处于不同的语义空间，检索质量会大打折扣。这个看似细节的问题，是我见过最多的"低级错误"之一。

设计黄金法则：RAG系统的离线索引和在线检索必须使用完全相同的编码器（Encoder）——同一模型、同一参数、同一tokenizer。任何不一致都会导致向量空间偏移，检索质量断崖式下降。这条规则唯一的例外是Cross-encoder重排序，因为它不需要向量空间一致性。

三、第一个完整RAG Demo

让我们跳出纯理论，写一个真正可运行的Python RAG程序。这次我们使用FAISS作为向量数据库，HuggingFace的BGE模型做Embedding，加上一个轻量级的重排序步骤。

from sentence_transformers import SentenceTransformer
import faiss
import numpy as np

# ============ Phase 1: Index ============
# 模拟的知识库文档
documents = [
    "RAG是检索增强生成的缩写，由Lewis等人于2020年提出。",
    "向量数据库Milvus支持GPU加速的相似性搜索。",
    "BGE-M3是BAAI发布的通用多语言Embedding模型。",
    "HNSW算法是向量搜索中最常用的索引结构。",
    "Chroma是一个轻量级的开源向量数据库。",
    "重排序通常使用Cross-encoder模型来精排检索结果。"
]

# 加载Embedding模型（同一模型用于索引和查询）
model = SentenceTransformer("BAAI/bge-small-zh-v1.5")

# 生成文档嵌入
doc_embeddings = model.encode(documents, normalize_embeddings=True)
dimension = doc_embeddings.shape[1]

# 构建FAISS索引
index = faiss.IndexFlatIP(dimension)  # Inner Product ≈ Cosine Similarity
index.add(doc_embeddings.astype(np.float32))

# ============ Phase 2: Retrieve ============
query = "有哪些开源的向量数据库？"
query_embedding = model.encode([query], normalize_embeddings=True)

# 检索Top-3
D, I = index.search(query_embedding.astype(np.float32), k=3)
retrieved = [documents[i] for i in I[0]]
print("检索结果:", retrieved)

# ============ Phase 3: Augment ============
# 这里简化：直接将检索结果拼接
context = "\n".join(f"- {doc}" for doc in retrieved)
augmented_prompt = f"""基于以下资料回答问题：

{context}

问题：{query}
回答："""

# ============ Phase 4: Generate ============
# 实际场景中调用 LLM API，这里用模拟展示
print("\n增强后的Prompt:")
print(augmented_prompt)
print("\n[模拟LLM回答]: 开源的向量数据库包括Milvus、Chroma、FAISS等。")

四、一个真实的"坑"：检索对了但生成错了

我第一次搭建RAG系统时犯过一个经典的错误。当时我们的知识库中包含大量产品规格文档，用户问"最大支持多少并发连接"，系统检索到了相关文档片段，但答案却引用了错误版本的信息。为什么？因为检索到了相关文档，但文档中包含多个版本的对比信息，LLM错误地关注了旧版本的数据。

这个经历的教训是：检索"相关"不等于检索"可用"。检索到的文档可能包含噪音、矛盾信息、或者缺乏必要的上下文。因此Advanced RAG中引入了重排序、上下文过滤和压缩等技术——这些都是为了确保送入LLM的信息是"精确可用"的，而不仅仅是"相关"的。

分层调试法：当RAG系统输出错误答案时，不要直接怀疑LLM。先问三个问题：（1）检索到的文档真的包含正确答案吗？（检查Retrieve阶段输出）（2）如果包含，Augment阶段是否正确地从文档中提取了关键信息？（3）如果前两个都正确，再怀疑LLM的生成质量。这种分层排查法能帮你快速定位99%的RAG问题。
