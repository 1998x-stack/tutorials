第4章：SDS 动态字符串

大家好，我是你们的讲师。上一章我们弄清楚了 Redis 的内存分配机制——zmalloc 加上了统计头部来追踪每次分配，底层由 jemalloc 提供高效无锁分配。现在我们要问一个问题：Redis 中什么数据分配最频繁？答案是字符串。

键是字符串，大部分值也是字符串（或者包含字符串的复合结构）。在 Redis 的命令处理通路中，字符串的拼接、切分、复制操作每秒发生数十万次。如果字符串操作的效率不够高，整个系统的瓶颈就在这里。而 C 语言的标准字符串——以 '\0' 结尾的 char 数组——在现代高性能系统中已经被证明是一个糟糕的设计。Redis 的作者 antirez 于是自己动手造了一个轮子，叫 SDS（Simple Dynamic String）。

SDS 是 Redis 数据结构的起点，也是你阅读 Redis 源码中第一个会反复遇到的类型。从第 1 章的 SET key value，到第 6 章的字典 key，SDS 几乎无处不在。彻底理解 SDS，是你走进 Redis 源码的必修课。

为什么不用 C 字符串

C 语言自带的字符串有三大硬伤：

  • 缓冲区溢出（Buffer Overflow）：执行 strcat(dest, src) 时，C 标准库假设 dest 有足够的空间——如果假设不成立，就悄悄覆盖了 dest 后面相邻的内存。这种 bug 极难定位，因为错误不会立即暴露，只会在某个不相关的时刻以 segfault 的形式出现。
  • 获取长度是 O(n)：strlen(s) 需要遍历整个字符串直到遇到 '\0'。Redis 中大量需要获取 key 的长度——想象一下每执行一条命令都要 O(n) 扫描一次 key，QPS 上十万后这个开销就不可忽视了。
  • 二进制不安全：C 字符串用 '\0' 标记结束，意味着字符串内容中不能包含 '\0'。但 Redis 要存储的数据多种多样——图片、序列化对象、加密结果——这些都可能包含空字节。

SDS 通过一个简单的设计同时解决了这三个问题：用额外的字段记录长度和容量，而不是依赖终止符。那么它的结构是什么样的呢？

SDS 数据结构

Redis 7.2 中定义了五种 SDS 头部类型，用 flags 字段区分：

类型 flags 值 len 类型 alloc 类型 容量范围 头部总大小
sdshdr5 0 （未使用 flags 低 3 位） （未使用） ≤ 31 字节 1 字节
sdshdr8 1 uint8_t uint8_t ≤ 255 字节 3 字节
sdshdr16 2 uint16_t uint16_t ≤ 65535 字节 5 字节
sdshdr32 3 uint32_t uint32_t ≤ 4GB 9 字节
sdshdr64 4 uint64_t uint64_t ≥ 4GB 17 字节

sdshdr8 是日常使用最多的类型，让我们看看它的结构体定义：

/* sds.h 中的 sdshdr8 定义 */
struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;      /* 字符串当前长度（不含 \0） */
    uint8_t alloc;    /* 实际分配容量（不含头部和 \0） */
    unsigned char flags; /* 低 3 位表示类型，高 5 位保留 */
    char buf[];       /* 真正的字符串数据 */
};

这里有几个重要的设计细节：

  • __packed__ 属性：告诉编译器不要做内存对齐，结构体按字段密集排列。sdshdr8 恰好占用 3 字节（len + alloc + flags），后面紧跟着 buf[] 数组。这个细节对内存节省至关重要——如果不对齐，光填充字节就可能浪费数十字节。乘以百万级的 key 就是 GB 级别的浪费。
  • buf[] 是柔性数组成员：它不占用头部空间，实际大小在运行时决定，并通过 malloc(len + 头部大小 + 1) 一次性分配出来。
  • sdshdr5 的特殊性：它用 flags 的低 3 位（0-5）存储长度，没有单独的 len 和 alloc 字段。这意味着它不支持扩容——一旦创建就不可变。它主要用于存储那些永远不会改变的短字符串（比如某些配置值）。

SDS 核心 API

SDS 的 API 设计非常用心，每一个函数都体现出"用最小的开销做最多的事"的哲学。

字符串创建

sdsnewlen 是所有 sdsnew 系函数的核心。它的逻辑清晰明了：

/* sds.c 中的 sdsnewlen 核心实现 */
sds sdsnewlen(const void *init, size_t initlen) {
    void *sh;
    sds s;
    char type = sdsReqType(initlen);   /* 根据长度选择合适的头部类型 */
    if (type == SDS_TYPE_5 && initlen == 0) type = SDS_TYPE_8;
    int hdrlen = sdsHdrSize(type);     /* 计算对应头部的字节数 */
    unsigned char *fp;                  /* 指向 flags 字段的指针 */

    sh = zmalloc(hdrlen + initlen + 1); /* 一次分配：头部 + 数据 + \0 */
    if (sh == NULL) return NULL;
    s = (char*)sh + hdrlen;            /* s 指向数据区起始位置 */
    fp = ((unsigned char*)s) - 1;      /* flags 在 s 的前一个字节 */
    /* ... 根据 type 设置 len, alloc, flags ... */
    if (initlen && init)
        memcpy(s, init, initlen);      /* 复制数据 */
    s[initlen] = '\0';                 /* 末尾补 \0（兼容 C 函数） */
    return s;
}

注意 sdsnewlen 末尾仍然加了一个 '\0'——即使 SDS 本身不依赖 '\0' 判断结束，这么做可以保证 SDS 字符串能直接作为参数传给标准 C 函数（比如 printf、strcmp），无需任何额外转换。这是向后兼容性的巧妙处理。

扩容策略

sdscat、sdscpy 等修改字符串内容的函数，最终都会调用 sdsMakeRoomFor 来确保有足够空间。它的扩容策略非常经典：

/* sdsMakeRoomFor 扩容逻辑伪代码 */
if (newlen < SDS_MAX_PREALLOC)    /* SDS_MAX_PREALLOC = 1024*1024 (1MB) */
    newlen *= 2;                    /* < 1MB 时翻倍增长 */
else
    newlen += SDS_MAX_PREALLOC;     /* >= 1MB 时线性增长 1MB */

/* 新容量可能会导致头部升级，比如 sdshdr8 → sdshdr16 */
type = sdsReqType(newlen);
if (type != sds[-1]) {             /* 需要更换头部类型 */
    /* 重新分配内存，搬移数据，更新头部 */
}

这个策略在很多动态数组实现中都能看到：小规模时指数增长（减少 realloc 次数），大规模时线性增长（避免过度浪费）。对于 Redis 常见的短字符串（几十到几百字节），翻倍策略意味着大多数字符串在生命周期中只需要 1-2 次扩容。

惰性空间释放

当执行 sdstrim 或 sdsclear 时，SDS 并不会因为内容减少而缩小容量。它只是修改 len 字段，保留已有的分配空间：

/* 截断时只修改 len，不释放底层空间 */
void sdsclear(sds s) {
    sdssetlen(s, 0);   /* 只改长度，不改 alloc */
    s[0] = '\0';
}

/* sdstrim 也是类似 */
sds sdstrim(sds s, const char *cset) {
    /* ... 从两端删除匹配字符 ... */
    s[len] = '\0';
    sdssetlen(s, len);  /* 同样不改 alloc */
    return s;
}

这个设计源自"未来可能还要再增长"的预期——避免立即释放又立即分配的乒乓行为。当然，如果你确定不再需要那么多空间，可以手动调用 sdsRemoveFreeSpace 触发真正的 realloc。

SDS 性能优化：embstr 与 raw 编码

不是所有的 SDS 字符串在内存中都是独立分配的。Redis 7.x 中，当字符串长度 ≤ 44 字节时，采用的是 embstr（embedded string） 编码，即 redisObject 和 SDS 头部 + 数据一次 malloc 分配在连续内存中。

44 这个数字是怎么来的？合理的内存分配对齐是关键。一个 robj 占用 16 字节（type/encoding/lru/refcount/ptr 等字段），sdshdr8 头部 3 字节，加上 1 字节的 '\0'，留给数据的就是 64 - 16 - 3 - 1 = 44 字节。为什么是 64？因为 jemalloc 的 size class 中有一个 64 字节的档位，一次性分配刚好对齐到这个档位，不会产生任何内存浪费。

/* object.c 中创建字符串对象 */
robj *createStringObject(const char *ptr, size_t len) {
    if (len <= OBJ_ENCODING_EMBSTR_SIZE_LIMIT)  /* 阈值 = 44 */
        return createEmbeddedStringObject(ptr, len);
    else
        return createRawStringObject(ptr, len);   /* SDS 单独分配 */
}

embstr 编码的优势：

  • 减少内存碎片：一次分配，一个连续内存块，vs 两次分配可能产生的两个不连续内存块。
  • 改善缓存局部性：访问对象时，robj 和字符串数据在相邻的内存地址，CPU cache line 一次就能加载完。
  • 减少一次指针解引用：直接通过 robj.ptr + sizeof(robj) 就能得到 SDS 头部，无需额外的间接寻址。
踩坑实录：有一次，一个业务团队在 Redis 中存储了一批经过压缩的图片数据（二进制格式），大小约为 200-500 字节。他们在读取数据后，使用 strlen() 来判断数据长度，而不是用 sdslen()。结果某些压缩后的数据中包含了 '\0' 字节，strlen() 在第一个 '\0' 处就返回了，导致后续的解压缩和图片渲染全部失败。这个问题花了整整两天才定位到——因为绝大多数压缩数据都不包含 '\0'，在测试环境中从未复现。根因就是 C 程序员的本能习惯：看到 char* 就用 strlen()。在 Redis 中，有了 SDS 就一定不要用 C 标准库的字符串函数来操作它。老老实实用 sdslen()，它在 Redis 源码中无处不在，因为它是 O(1) 且二进制安全的。
小贴士：在 GDB 中调试 SDS 时，可以直接用我们第 2 章配置的 psds 宏：psds <sdspointer>，它会打印出 hidden header 中的 len、alloc 和字符串内容。如果你发现 len 和 strlen() 的结果不一致，别慌——大概率是因为字符串中包含空字节，这恰恰是 SDS 的二进制安全特性在起作用。
