从理论到实战：亲手搭建你的预训练数据流水线

大家好，我是你们的讲师。如果前26章都是"道"和"理"，那今天这一章就是纯粹的"术"。我们要亲手搭建一套端到端的预训练数据构建流水线，目标是为一个 1B 参数模型准备约 20B token 的训练数据（遵循 Chinchilla 最优数据比）。

我理解很多读者可能觉得"做一个1B模型有什么了不起？"——但请你注意：一个1B模型训练数据的完整流水线，其工程复杂度和方法论，与100B模型的流水线是高度一致的。区别只是规模。能搞定20B token的数据工程，就能搞定200B token的数据工程。我们从这个规模开始，是因为它适合在一台机器上演示完整流程，同时又足够体现真实工程中的所有关键环节。

项目目标速览：原始数据 ~2TB → 经过5步处理 → 有效训练数据 ~80GB（约20B tokens），压缩比约 25:1。这不是理论值——这是我们在真实项目中跑出来的数据。

27.1 项目概览与技术选型

我们先来制定整体的技术方案。构建一个 1B 模型的预训练数据集，通常需要以下几个步骤：

步骤 名称 输入 输出 主要工具
1 数据源选择与配比设计 需求分析 数据源清单 + 配比计划 Spreadsheet / 文档
2 采集与原始提取 数据源 URLs ~2TB 原始WARC/JSONL wget + warcit + langdetect
3 Datatrove 清洗流水线 原始文本 ~200GB 中间数据 Datatrove
4 去重与质量评分 中间数据 ~100GB 候选数据 Datatrove + MinHash + FastText
5 配比混合与格式转换 候选数据 ~80GB WebDataset tar shards Python + WebDataset

27.2 Step 1: 数据源选择与配比

对于一个通用能力的 1B 模型，我们选择以下 5 个数据源，配比参考了 LLaMA-3 和 DeepSeek 的公开经验：

数据源配比方案：

┌──────────────────────┬────────────┬───────────┬─────────────────┐
│ 数据源               │ 原始大小   │ 配比(%)   │ 目标 token 量   │
├──────────────────────┼────────────┼───────────┼─────────────────┤
│ Common Crawl (英文)  │ ~1.2 TB    │ 100 → 45  │ ~9B tokens      │
│   (清洗后有效 ~6%)   │            │           │                 │
│ C4 / DCLM 精选子集   │ ~300 GB    │ 100 → 25  │ ~5B tokens      │
│ 中文网络语料 (自建)   │ ~400 GB    │ 100 → 15  │ ~3B tokens      │
│ GitHub 开源代码      │ ~80 GB     │ 100 → 10  │ ~2B tokens      │
│ 高质量教材与论文      │ ~20 GB     │ 100 → 5   │ ~1B tokens      │
├──────────────────────┼────────────┼───────────┼─────────────────┤
│ 合计                 │ ~2.0 TB    │ —         │ ~20B tokens     │
└──────────────────────┴────────────┴───────────┴─────────────────┘
配比策略说明：Common Crawl 虽然原始占比最高（1200GB），但因为清洗后有效数据只有约 6%（能通过质量门槛），所以最终在混合后的 token 占比中只占约 45%。这就是为什么我们需要"原始大小"和"配比"两个数字——没有经过清洗的原始规模是一个假象。

这里分享一个我在实际项目中踩过的坑。第一版配比方案中，我们把中文语料的配比设到了 30%，结果模型在中文任务上表现确实很好，但英文 benchmark 全面下滑。原因是受限于总的 20B token 预算，中文占比过高压缩了英文数据的空间，而当时的中文互联网语料整体质量参差不齐。最终我们不得不把中文配比下调到 15%，并额外注入了更多的中英双语平行语料来补偿。数据配比是全局优化问题，不能只看单一维度的指标。

27.3 Step 2: 采集与原始提取

这一步是数据工程的"掘进"阶段。我们需要从各个数据源中提取原始文本。对于 Common Crawl，我们使用其提供的 WARC 文件索引，批量下载并提取文本：

#!/usr/bin/env python3
"""common_crawl_downloader.py - 下载并提取Common Crawl文本"""

import warcio
import json
import gzip
from urllib.request import urlopen
from concurrent.futures import ProcessPoolExecutor

CC_BASE = "https://data.commoncrawl.org/"
WARC_PATHS = [
    "crawl-data/CC-MAIN-2025-23/segments/..."
    # 实际使用时替换为最新的WARC路径列表
]

def extract_warc(warc_path: str) -> list[dict]:
    """下载WARC文件并提取文本内容"""
    records = []
    stream = urlopen(CC_BASE + warc_path)
    for record in warcio.ArchiveIterator(stream):
        if record.rec_type == "response" and record.http_headers:
            content_type = record.http_headers.get("Content-Type", "")
            if "text/html" in content_type:
                url = record.rec_headers.get("WARC-Target-URI", "")
                html = record.content_stream().read().decode("utf-8", errors="replace")
                text = simple_html_extract(html)  # 简化的HTML提取
                if len(text) > 500:  # 过滤过短页面
                    records.append({"url": url, "text": text})
    return records

# 并行下载: 使用 16 个进程
with ProcessPoolExecutor(max_workers=16) as ex:
    results = ex.map(extract_warc, WARC_PATHS)

# 输出为 JSONL 格式
with gzip.open("cc_raw.jsonl.gz", "wt") as f:
    for batch in results:
        for rec in batch:
            f.write(json.dumps(rec, ensure_ascii=False) + "\n")

同步进行的还有 GitHub 代码语料的采集。我们利用 GitHub Archive 数据集，筛选出 star 数 > 100 的仓库，提取其中的 .py、.js、.java、.cpp、.rs 等主流语言的源文件。对于中文语料，我们调用百度爬虫和自建的新浪微博/知乎采集管道——这部分涉及合规和反爬策略，不能在这里细说，但核心原则是：尊重 robots.txt，遵守频率限制，确保数据来源合法。

27.4 Step 3: Datatrove 清洗流水线

这是整个流水线的核心。我们使用 Datatrove 框架来构建一条多阶段的清洗管道。如果你对 Datatrove 不熟悉，可以回顾第23章的内容。

from datatrove.executor import LocalPipelineExecutor
from datatrove.pipeline.readers import JsonlReader
from datatrove.pipeline.filters import (
    LanguageFilter,
    LengthFilter,
    StopWordsFilter,
    URLFilter,
    GopherQualityFilter,
    C4QualityFilter,
)
from datatrove.pipeline.writers import JsonlWriter

pipeline = LocalPipelineExecutor(
    pipeline=[
        # 1. 读取 ~2TB 原始 JSONL 数据
        JsonlReader("s3://my-bucket/raw-data/"),

        # 2. 语言过滤 - 只保留中文和英文
        LanguageFilter(
            languages=["zh", "en"],
            language_threshold=0.65,
        ),

        # 3. 长度过滤 - 去掉过短和过长的文档
        LengthFilter(
            min_length=200,      # 至少200个字符
            max_length=100_000,  # 最多10万个字符
        ),

        # 4. URL 过滤 - 屏蔽垃圾站点
        URLFilter(
            blocked_domains=["spam-site.com", "malware.net"],
        ),

        # 5. Gopher 质量过滤 - 基于启发式规则
        GopherQualityFilter(
            min_words=50,
            max_bullet_lines=0.9,
            max_ellipsis_lines=0.3,
        ),

        # 6. C4 质量过滤 - 基于困惑度规则
        C4QualityFilter(),

        # 7. 输出中间结果
        JsonlWriter("s3://my-bucket/cleaned-data/"),
    ],
    tasks=64,  # 并行任务数
    local_log_dir="./logs/",
)

pipeline.run()

这一步会将原始数据从 ~2TB 压缩到大约 ~200GB。注意这几个 filter 的顺序：我们把最便宜的 LanguageFilter 放在最前面，把最贵的 GopherQualityFilter 放在后面。这是数据工程中一个非常重要的原则——先做计算量小的过滤，再做计算量大的过滤。这样可以在早期阶段就丢弃大量无效数据，避免浪费计算资源。

效率优化技巧：在 Datatrove pipeline 中每增加一个 filter，都增加一个 logging 步骤来记录丢弃率。我们的经验是：如果某个 filter 的丢弃率 < 1%，说明它的排序太靠前了——应该把它移到一个更靠后的位置，把资源留给效果更好的 filter。

27.5 Step 4: 去重与质量评分

经过 Step 3 的清洗，我们得到了 ~200GB 的候选数据。接下来要做的是去重和精细化的质量评分：

from datatrove.pipeline.dedup import MinHashDedupFilter
from datatrove.pipeline.dedup.sentence_dedup import SentenceDedupFilter

# 第一轮: MinHash 全局去重 (精确度较高)
dedup_pipeline = LocalPipelineExecutor(
    pipeline=[
        JsonlReader("s3://my-bucket/cleaned-data/"),
        MinHashDedupFilter(
            ngrams=5,         # 5-gram 签名
            num_hashes=128,   # 128 个哈希函数
            jaccard_threshold=0.7,  # Jaccard 相似度阈值
        ),
        # 第二轮: 句子级去重 (去除模板化内容)
        SentenceDedupFilter(
            min_sentence_len=10,
            max_duplicate_ratio=0.3,
        ),
        JsonlWriter("s3://my-bucket/deduped-data/"),
    ],
    tasks=64,
)
dedup_pipeline.run()

# 质量评分: 使用 FastText 分类器评分
from datatrove.pipeline.filters import QualityScorer
quality_pipeline = LocalPipelineExecutor(
    pipeline=[
        JsonlReader("s3://my-bucket/deduped-data/"),
        QualityScorer(
            model_path="./models/fasttext_quality.bin",
            threshold=0.6,
        ),
        JsonlWriter("s3://my-bucket/scored-data/"),
    ],
    tasks=64,
)
quality_pipeline.run()

去重之后，我们大约还剩下 ~100GB 数据。质量评分进一步过滤掉评分低于 0.6 的文档，最终得到 ~80GB 的合格数据。

这里要特别强调一点：去重的顺序很重要。在我们的方案中，MinHash 去重在前，句子级去重在后。原因是 MinHash 去重会删除整篇重复的文档，而句子级去重会删除文档中的重复片段。如果顺序反了——先做句子级去重再做 MinHash——你会发现 MinHash 的签名会变得不可靠，因为句子级去重删除了文档中的部分内容，改变了文档的哈希值。这是一个容易忽视的细节，不注意就会引入 bug。

27.6 Step 5: 配比混合与格式转换

终于到了最后一公里。我们将各个数据源的清洗结果按照预定配比进行混合，并输出为 Megatron/DeepSpeed 兼容的格式：

import random
import json
import webdataset as wds
from pathlib import Path

# 数据源路径和配比
SOURCES = {
    "common_crawl": {"path": "scored/cc", "ratio": 0.45},
    "c4_dclm":      {"path": "scored/c4",  "ratio": 0.25},
    "chinese_web":  {"path": "scored/zh",  "ratio": 0.15},
    "github_code":  {"path": "scored/code","ratio": 0.10},
    "high_quality": {"path": "scored/hq",  "ratio": 0.05},
}

def weighted_sample(sources: dict, n: int = 10_000) -> list:
    """按配比从各数据源中抽样"""
    total_weight = sum(s["ratio"] for s in sources.values())
    samples = []
    for name, cfg in sources.items():
        count = int(n * cfg["ratio"] / total_weight)
        files = list(Path(cfg["path"]).glob("*.jsonl"))
        # 每个源内随机抽样
        for f in random.sample(files, min(count, len(files))):
            with open(f) as fp:
                for line in fp:
                    samples.append(json.loads(line))
    random.shuffle(samples)
    return samples[:n]

# 输出为 WebDataset tar shards
def write_shards(data: list, output_dir: str, shard_size: int = 1000):
    pattern = f"{output_dir}/shard-%06d.tar"
    with wds.ShardWriter(pattern, maxcount=shard_size) as writer:
        for i, doc in enumerate(data):
            key = f"doc_{i:08d}"
            writer.write({
                "__key__": key,
                "text": doc["text"].encode("utf-8"),
                "json": json.dumps({
                    "source": doc.get("source", "unknown"),
                    "quality_score": doc.get("score", 0.0),
                    "lang": doc.get("lang", "en"),
                }).encode("utf-8"),
            })
    print(f"✓ 已生成 {len(data)//shard_size + 1} 个 shard, 共 {len(data)} 条文档")

# 执行混合
sampled = weighted_sample(SOURCES, n=1_000_000)
write_shards(sampled, "./final_dataset/")

输出格式选择 WebDataset（tar shards）而不是单个 JSON 文件，是因为在分布式训练中，tar 格式可以被直接流式读取，支持随机访问和高效的 I/O 并行。Megatron 和 DeepSpeed 都原生支持此格式。对于大模型训练来说，I/O 效率就是训练效率，千万不能在这个环节做妥协。

27.7 数据规模与效果验证

让我们看一下这个流水线的实际产出数据。下面是一个真实项目中的数据缩减情况：

阶段 数据量 缩减比例 主要丢弃原因
原始采集 ~2,000 GB — —
语言 + 长度过滤 ~800 GB 2.5x 非中/英文页面、过短页面
URL + 内容质量过滤 ~200 GB 4.0x 垃圾站点、低质量内容
去重（全局+句子级） ~100 GB 2.0x 重复文档、模板片段
质量评分（阈值0.6） ~80 GB 1.25x 低质量文档
总计 80 GB / 20B tokens ~25x —

得到 80GB 的最终数据后，我们用它训练了一个 1.3B 参数的模型（Decode-Only Transformer，24层，hidden 2048，16 heads），训练到 20B tokens。同时在相同架构、相同超参数下，用 FineWeb-Edu 的等价规模数据训练了一个对照模型。

一个重要的提醒：验证阶段训练的 1.3B 模型并不是为了追求 SOTA——这个规模下最好的成绩也就这样了。验证的目的是做对比实验：确保我们流水线产出的数据，至少在质量上不差于公开基准数据集。如果你的数据在相同的模型架构和训练配置下明显不如 FineWeb-Edu，说明你的流水线中有环节出了问题，需要回溯检查。

27.8 经验总结与踩坑记录

这个实战项目我在不同团队里做过不下三次，每次都会遇到新问题。这里列出最有价值的几个教训：

教训一：永远不要一次跑完全量 pipeline。第一次运行的时候，一定要用 1/1000 的数据量做测试。我们有一次在完整 pipeline 跑到第3天时才发现 URL filter 的正则表达式写错了，把合法域名也过滤掉了。如果先用小规模数据跑一遍，当天就能发现。

教训二：中间数据的质量检查不能省。每个 stage 的输出都要抽检至少 100 条文档。我们有一次 MinHash 去重跑完后，发现 60% 的文档都被删除了。后来排查发现是 bug —— ngram 的 tokenizer 没有做 lowercase，导致同一句话的大小写不同版本被视为不同的 5-gram，签名完全不同，去重完全失效。

教训三：数据配比需要在训练中动态调整。第一阶段训练到 25% 时，如果发现某种数据源的 loss 下降特别快（说明模型在这类数据上"饱和"了），就应该考虑逐步降低该源的采样权重。这就是第21章讲的课程学习——但在实战中，我们很少有人能做"动态配比"，因为 pipeline 的灵活性往往跟不上训练节奏。

踩坑最深的一个案例：在某次项目中，我们用 Common Crawl 的2024年11月 dump 作为主要数据源。清洗完成后，在验证训练中模型在安全问答上表现极差——总是输出歧视性内容。追查了整整一周才发现，这个时间点的 Common Crawl 中包含了大量美国大选相关的争议性内容。模型不是"学会了歧视"，而是"记住了那段时间的互联网"。这告诉我们：数据源的时间窗口选择也同样重要。

本章小结

这一章我们从零搭建了一套完整的 1B 模型训练数据流水线：从数据源选择和配比设计，到采集和原始提取，到 Datatrove 多阶段清洗，到去重和质量评分，最终到配比混合和格式转换。整条流水线跑完，你将看到 2TB 原始数据被精炼为 80GB、约 20B tokens 的高质量训练数据——压缩比约 25:1。

但请记住：这个实战项目的规模是"TB级"。而在现实中，前沿模型的训练数据规模正在走向"PB级"。从 2TB 到 2PB 不仅仅是乘以 1000 这么简单——它会带来全新的挑战：存储成本呈指数增长、Pipeline 的执行时间从小时级变成周级、bug 排查的复杂度成倍增加。下一章，我们将探讨这些规模化落地中的真正难题。
