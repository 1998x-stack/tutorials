第4章 以太网帧格式深度解析

大家好，我是你们的讲师。上一章我们看到了数据在穿过OSI协议栈时是如何被层层封装的——从HTTP请求到TCP段，再到IP包，最终被塞进以太网帧中通过网线发出去。但那个"最终被塞进去"的以太网帧，里面到底长什么样？每一个字节在干什么？

如果你曾经用Wireshark抓过包，你一定会看到类似这样的信息："Frame 1: 74 bytes on wire (592 bits), 74 bytes captured..."。Wireshark会帮你把帧的各个字段解析得很清楚。但Wireshark不会告诉你的是：为什么前导码是7个字节的0xAA？为什么最小帧长必须是64字节？为什么VLAN标签插在源地址后面而不是别的位置？这一章，我们来回答这些"为什么"。

4.1 前导码与帧起始定界符

一个以太网帧在传输时，第一个出现在线缆上的不是目的地址，而是前导码（Preamble）：7个字节的0xAA，也就是二进制10101010的重复模式。这个模式的作用是让接收端的PHY芯片与发送端的时钟同步。为什么需要同步？因为以太网的物理层没有独立的时钟信号线——时钟信息是"隐式"地嵌入在数据信号中的。接收端需要先"听"一段固定的波形，才能锁定发送端的数据速率和相位。

前导码之后是1个字节的帧起始定界符（SFD, Start Frame Delimiter）：0xAB，二进制10101011。注意看它的最后两位：不是"10"而是"11"。这个细微的变化告诉接收端："前面7个字节是给你的热身时间，现在真正的帧数据要开始了，做好准备！"

帧结构概览（DIX Ethernet II）：

+------+------+----------+----------+------+------------+------+
|前导码 | SFD  | 目的地址 | 源地址   | 类型  | 有效载荷   | FCS  |
| 7字节 | 1字节 |  6字节   | 6字节   | 2字节 | 46-1500字节| 4字节|
| 0xAA | 0xAB |          |          |      |            |      |
+------+------+----------+----------+------+------------+------+

注意：前导码和SFD是物理层添加的，不被MAC层视为"帧"的一部分。
Wireshark在显示帧时通常不显示前导码和SFD。

4.2 目的地址与源地址

接下来是目的MAC地址（6字节）和源MAC地址（6字节）。每个MAC地址都是全球唯一的48位标识符，由IEEE负责分配管理。关于MAC地址的详细结构（OUI、单播/组播/广播的区别），我们会在第5章专门展开。这里只需要记住两个关键信息：目的地址可以是单播（指向特定设备）、组播（指向一组设备）或广播（全F，指向所有设备）；源地址永远是单播地址（因为数据帧必定是从某个特定设备发出的）。

还有一个容易被忽略的细节：地址字段在传输时，最先发送的是目的地址字段的最高位字节中的最低有效位（LSB first）。换句话说，如果你在Wireshark中看到的目的MAC地址是"01:80:C2:00:00:00"，那么在线缆上最先传输的比特是这个地址的第一个字节的LSB——也就是"1"（因为0x01的二进制是00000001）。这个细节在调试一些底层硬件问题时非常重要，比如STP（生成树协议）的组播地址识别。

4.3 类型/长度字段

这是以太网帧中最"精妙"的一个字段，也是一个历史和妥协的产物。

在DIX Ethernet II帧格式中，这个2字节的字段被称为EtherType（以太类型），用于标识上层协议。比如0x0800代表IP协议，0x0806代表ARP，0x8100代表VLAN标签帧（802.1Q），0x86DD代表IPv6。接收端MAC层根据这个字段知道应该把有效载荷交给哪个上层协议处理。

在IEEE 802.3帧格式中，同一个2字节字段被解释为长度（Length），表示有效载荷的字节数。IEEE 802.3帧后面紧跟着LLC头部（3字节，包含DSAP/SSAP/Control），由LLC来标识上层协议。

那么问题来了：接收端怎么知道这个字段是EtherType还是Length？IEEE在802.3x-1997修订中给出了一个优雅的解决方案：如果这个字段的值大于或等于0x0600（1536），就解释为EtherType；否则解释为Length。因为IEEE 802.3帧的有效载荷长度最大为1500字节，加上LLC头部的3字节也才1503，完全不会跟EtherType的取值区间（从0x0600开始）重叠。这个巧妙的"双重解释"机制使得同一个物理介质上可以同时传输DIX和802.3两种帧格式。

4.4 有效载荷：为什么是46-1500字节？

有效载荷（Payload）的最小值是46字节，最大值是1500字节。这两个边界各有其工程设计上的理由：

最小46字节：这是由CSMA/CD协议决定的。在10 Mbps的半双工以太网中，最远两端之间的往返传播延迟（称为"碰撞窗口"或"最小时隙"）是512位时（51.2 µs）。为了让发送端能够在发送完成之前检测到碰撞，帧的总长度（不包括前导码和SFD）必须至少为512位 = 64字节。帧头（目的地址+源地址+类型/长度）+ FCS = 6+6+2+4 = 18字节。因此有效载荷最少为64-18 = 46字节。如果上层数据不足46字节，MAC层会自动添加填充字节（Padding），使帧长达到最小要求。

最大1500字节：这个值被称为标准MTU（Maximum Transmission Unit）。1500是一个历史性的"折衷"：更大的MTU意味着更高的一次性传输效率（因为帧头开销占比更低），但同时也意味着更长的线缆占用时间（其他设备必须等待更久），以及更高的CRC校验错误成本（一个比特错误会污染整个帧）。1500这个数字是DIX联盟在1980年设计的，它被证明是一个在效率和可靠性之间非常平衡的选择。即使在今天的高速网络中，1500仍然是绝大多数以太网链路的默认MTU。

关于巨型帧（Jumbo Frame）：在数据中心环境中，MTU经常被调整为9000字节，这种超长帧被称为"巨型帧"。巨型帧的好处很明显：减少帧头开销比例、降低CPU中断频率（每帧中断一次，包数减少了）。我在一家云服务商工作时，发现他们的存储后端网络启用了9000字节的巨型帧，吞吐量比1500字节MTU提升了约15%。但巨型帧也有风险：如果路径上某一跳不支持巨型帧，就会导致IP分片或丢包。我之前在MTU不匹配的案例中已经踩过这个坑了（第3章提到过）。

4.5 FCS与CRC32校验

帧尾的4字节帧校验序列（FCS, Frame Check Sequence），是通过CRC32算法对整个MAC帧（目的地址到有效载荷）计算得到的32位校验值。CRC32的计算过程可以这样理解：发送端把整个帧视为一个巨大的二进制数，除以一个固定的32位多项式（IEEE标准中使用的多项式是0x04C11DB7），得到的余数就是FCS。接收端用同样的多项式对收到的帧（包括FCS）进行计算，如果余数为0，说明帧在传输过程中没有错误。

CRC32可以检测出所有单比特错误、双比特错误、奇数个比特错误、以及所有长度不超过32位的突发错误。对于长度超过32位的突发错误，CRC32的漏检率约为1/2^32——也就是大约四十亿分之一的概率。这个可靠性对于大多数应用已经足够高。

需要特别注意的是：如果接收端CRC校验失败，帧会被直接丢弃（没有"重传请求"）。以太网链路层不提供可靠传输——那是上层（如TCP协议）的职责。所以你会看到这样的情况：如果一个以太网帧在传输过程中被损坏，接收端默默地把它扔掉，发送端对此一无所知，直到上层协议发现数据缺失才触发重传。

4.6 VLAN标签帧（802.1Q）

标准以太网帧在源地址和类型/长度字段之间留有一个"空隙"。1998年，IEEE 802.1Q标准利用了这4个字节的空间，定义了一种扩展帧格式——VLAN标签帧。这4个字节被分为：

  • TPID（Tag Protocol Identifier，2字节）：固定为0x8100，用于标识这是一个VLAN标签帧。交换机和网卡看到这个值就知道后面跟着的不是EtherType而是VLAN信息。
  • TCI（Tag Control Information，2字节）：包含12位的VLAN ID（最多4096个VLAN）、3位的PCP（Priority Code Point，用于802.1p QoS优先级）和1位的DEI（Drop Eligible Indicator，丢包指示）。

添加VLAN标签后，帧的总长度增加了4字节，因此FCS需要重新计算。为了保证帧间间距（Interframe Gap）不变，实际的传输时间会略微延长。不过对于今天吉比特及以上的链路来说，这几纳秒的差异几乎可以忽略不计。

4.7 一个踩坑故事：巨型帧的数据中心陷阱

前面提到了巨型帧（Jumbo Frame）的好处。但巨型帧在实际部署中存在着一个隐蔽的"陷阱"，我2018年在客户现场就遇到过一次。

那是一家金融科技公司，他们在数据中心内部启用了9000字节的巨型帧，用于加速存储网络（iSCSI）和大数据批处理作业。配置完成后测试，一切正常——存储吞吐量提升了约20%，所有人很高兴。但两周后，运维团队发现了一个奇怪的问题：每天晚上10点到12点的数据备份窗口，有几台数据库服务器的网络性能会突然"掉底"，iSCSI连接频繁超时，备份任务常常失败。

排查过程一波三折。我们检查了交换机的端口统计，发现这几个端口的CRC错误计数器在高峰期快速增长——每小时增加几千个错误。CRC错误意味着物理层或链路层存在信号质量问题。但检查了链路预算（光模块的光功率、光纤的弯曲半径、连接器清洁度），都没有发现问题。更奇怪的是，同样的服务器和交换机端口，白天跑9000字节的巨型帧完全正常，偏偏晚上10点以后出问题。

最终，一位老工程师在物理检查中发现了一条被踩踏过的光纤跳线——它被机柜门夹住，形成了一个角度很小的弯折。白天温度高时，光纤的弯曲损耗不明显；晚上温度降低，光纤的折射率发生变化，那个弯折处的损耗突然增大，导致信号质量下降到恰好触发大量CRC错误。而巨型帧比标准帧更容易受到信号质量下降的影响——因为一个比特错误就会导致整个9000字节的帧被丢弃，相比之下，如果使用1500字节的MTU，同样的信号质量只会破坏较小的帧。更关键的是，iSCSI的上层是TCP，TCP在遇到连续丢包时会触发拥塞控制算法（进入"慢启动"或"拥塞避免"阶段），发送窗口急剧缩小，吞吐量断崖式下跌。

教训：巨型帧放大了物理层问题的"可见度"。在标准MTU环境中，光纤连接器的轻度污染可能只会导致零星的重传，几乎不会被注意到；但在巨型帧环境中，同样的污染可能造成明显的性能下降。如果你决定在你的网络中启用巨型帧，必须先确保物理层的光纤链路质量——包括使用OTDR测试光纤损耗、使用光纤显微镜检查连接器端面、确认链路预算有充足的余量（至少3 dB以上）。

好了，从7字节的前导码到4字节的FCS，我们已经把以太网帧的每一个字节都走了一遍。但还有一个至关重要的部分我们没有深入展开——那6字节的MAC地址。为什么MAC地址是48位的？单播和组播怎么区分？交换机是怎么通过MAC地址"学会"转发的？下一章（第5章：MAC地址与寻址机制）我们来找这些答案。
