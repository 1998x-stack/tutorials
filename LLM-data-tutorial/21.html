合成数据的崛起

前面 20 章我们一直在讨论一个核心问题：如何从互联网的汪洋大海中筛选、清洗、组织真实文本。这很自然——毕竟 LLM 训练数据的基石就是真实世界中产生的文本。但有人问了一个更激进的问题：如果真实数据不够用，我们能不能让模型自己"创造"训练数据？

这个想法在 2023 年听起来还像天方夜谭。但到了 2026 年，合成数据（synthetic data）已经从一个边缘概念成长为 LLM 训练数据生态中的重要组成部分。根据 2026 年初的行业调研，主流 LLM 训练数据中合成数据的占比已从 2024 年的约 2% 增长到了约 15%。这个数字在代码和数学领域更高——DeepSeek-Coder-V3 的训练数据中，合成数据占比超过了 40%。

为什么需要合成数据？最直接的原因是：真实数据覆盖不了所有场景。想要模型学会解微积分证明题？互联网上高质量的数学推理数据非常稀少。想让模型学会像顶级工程师一样编写单元测试？GitHub 上的测试代码数量远远不够。想让模型掌握某个垂直领域的专业知识？公开语料中这类内容的密度极低。合成数据的核心价值就在于——它可以根据需求定向生成训练数据，填补真实数据中的空白。

2024 年 Phi-4 的发布让整个行业对合成数据刮目相看。微软团队仅用 14B 参数的 Phi-4 就在多个基准上超越了 200B+ 的 GPT-4o。他们在技术报告中揭示了一个关键秘诀：高质量合成数据。Phi-4 的训练使用了多轮自生成（self-generated）数据，包括教科书级别的数学推理、编程竞赛题详解、多步逻辑推理链。这些数据来自一个精心设计的种子课程和一个迭代式的生成-过滤-训练循环。

Self-Play 与模型生成

合成数据生成最激动人心的模式是 Self-Play——让模型自己生成数据，用这些数据训练自己或下一代的模型，形成"生成 → 过滤 → 训练 → 生成更好数据"的正向循环。

这个范式的早期代表是 2023 年的 Textbooks Are All You Need 论文。Meta 团队提出：与其从互联网上爬取数据，不如用 LLM 生成教科书级别的训练数据。他们用 Llama-3 生成了数千万条包含详细解题步骤的数学问题、解释性文章和代码教程。生成过程高度依赖 prompt 工程——不是简单地让模型"生成数据"，而是设计复杂的 prompt 模板，引导模型产出特定格式和特定难度的内容。

# LLM 合成数据生成 prompt 模板示例

MATH_REASONING_PROMPT = """你是一位数学教师，正在为你的学生编写一道{难度}级别的{主题}题目。
请严格按照以下格式输出：

## 题目
[一个{难度}的{主题}问题]

## 解题步骤
[分步骤详细解释，每一步都要有数学推导]

## 最终答案
[精确的最终答案]

## 常见错误分析
[学生可能犯的2-3个典型错误，以及为什么]

要求：
- 题目必须可解且答案唯一
- 解题步骤不能跳跃
- 难度级别：{难度}
- 主题：{主题}
"""

# 批量生成示例
def generate_synthetic_math_data(llm, topics, difficulties):
    all_data = []
    for topic in topics:
        for diff in difficulties:
            prompt = MATH_REASONING_PROMPT.format(
                topic=topic, difficulty=diff
            )
            response = llm.generate(prompt, temperature=0.7)
            parsed = parse_response(response)
            all_data.append(parsed)
    return all_data

生成合成数据只是第一步，真正的挑战在于质量控制。与真实数据不同，合成数据没有天然的"真实性"保证——模型可能在编造看起来合理的错误答案。因此，合成数据 pipeline 中必须包含严格的验证和过滤环节。典型的质量控制手段包括：

生成策略工作方式适用场景质量控制机制
Self-Instruction从种子样本出发，让模型生成变体指令微调数据语义相似度去重 + 人工抽样审核
Back-Translation中→英→中的翻译循环翻译质量、语义一致性BLEU/COMET 评分过滤
Chain-of-Thought 生成要求模型展示完整推理链数学、逻辑推理反向验证答案一致性
Evol-Instruct逐步提升指令复杂度复杂指令遵循难度分级 + LLM-as-Judge 评分
Code Synthesis生成编程题目和测试用例代码生成训练编译执行验证 + 单元测试
Role-Playing让模型扮演特定角色对话客服、教学对话规则验证 + 毒性过滤
合成数据的"质"比"量"更重要： 一个常见的误区是认为合成数据越多越好。实际上，大量低质量的合成数据可能稀释模型从真实数据中学到的知识。2025 年的一篇系统研究发现，在预训练阶段加入超过 20% 的低质量合成数据会使模型在 MMLU 上的得分下降 3-5%。关键指标不是数量，而是多样性（coverage）和准确性（correctness）。1000 条高质量、多样化的合成样本，效果远好于 10 万条模板化的重复生成。

数据增强技术

除了让模型直接生成新数据，另一类方法是对现有真实数据进行增强（augmentation），创造出更多变体。这些技术的好处是保持了数据的基本事实正确性，同时增加了多样性。

回译（Back Translation） 是最经典的数据增强方法之一。以中英文翻译为例：将英文句子翻译成中文，再将中文翻译回英文。如果两次翻译的结果一致，说明原句的语义清晰；如果结果不一致，原句可能有多义性，这个样本本身就值得关注。回译不仅用于翻译任务，还被广泛应用于分类、问答、摘要等场景——任何需要语义不变变换的任务都可以使用。

改写增强 则更进一步：用 LLM 对同一文本进行多种风格的改写，比如将学术论文摘要改写成科普风格，将技术文档改写成教程风格。这种方法在 2025-2026 年被大量用于训练模型的"风格适应"能力——让模型在同一知识下具备多种表达方式。

结构化数据生成 是 2025 年后兴起的另一个方向。它的思路是：利用非结构化的真实文本（如维基百科文章），自动生成与之对应的结构化训练数据——问题-答案对、推理链、知识图谱三元组等。这种方法的价值在于，它将知识从原始的"阅读格式"转化为模型更易学习的"练习格式"。

有趣的现象： 2025 年的一项对照实验发现，用 500 条高质量的合成 QA 对进行微调的 7B 模型，在领域问答任务上的表现超过了用 5000 条未筛选真实领域数据进行微调的版本。这个反直觉的结果说明了"数据密度"的概念——合成数据可以精确地命中知识的薄弱环节，而真实数据则是等概率地从已有分布中采样。当你时间有限时，精心设计的合成数据可能是比盲目采集真实数据更高效的策略。

2026 年的合成数据实践

到了 2026 年，合成数据生成已经从实验室技术演变为体系化的工程实践。一个标志性的进展是 DataEvolver——一个能够自动构建数据准备流水线的智能系统。DataEvolver 不需要人工编写数据清洗规则，它通过分析目标数据集的结构和质量，自动设计合成数据生成的 prompt 模板和过滤规则。

这种"从人工写规则到 AI 自动构建数据增强 pipeline"的转变，代表了数据工程领域的一个范式飞跃。在一个典型的使用场景中，工程师只需要提供 50-100 条高质量种子数据，DataEvolver 就能自动扩展出一个包含数千条多样化合成样本的数据集，并自动评估每条样本对下游任务的预期贡献。

但合成数据并非没有风险。最受关注的问题是 模型坍缩（Model Collapse）——当用模型生成的数据训练同一模型（或同族模型）时，模型的输出质量会随着代际迭代而逐代下降。这背后的机制是：模型生成的数据存在偏差（bias），这些偏差会随着再训练被放大，形成恶性循环。可以想象，如果互联网上充斥着 LLM 生成的内容（这正在成为现实），未来的训练数据将不可避免地包含越来越多的合成文本——模型坍缩可能成为一个系统性问题。

模型坍缩的真实案例： 2025 年的一项追踪实验发现，仅经过 5 轮自我训练循环（生成 → 过滤 → 训练 → 再生成），模型的文本多样性下降了约 40%，罕见词汇的使用频率减少了 60%，而生成的错误信息（hallucination）率从 2% 上升到了 12%。更令人担忧的是，混合真实数据和合成数据只能延缓、不能完全阻止这个趋势。当前的共识是：合成数据在训练数据中的占比不应超过 15-20%，且必须配合严格的质量过滤和多样化保持策略。

合成数据为 LLM 训练开辟了新的可能性，但它并不是"免费的午餐"。领域特定的合成数据挑战——如何为代码、医学、法律等专业领域生成高质量合成数据——各有一套独特的方法论。在真实数据和合成数据的组合拳下，不同领域需要完全不同的数据策略。下一章，我们将深入探讨各个专业领域的数据适配策略。
