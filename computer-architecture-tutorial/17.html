第17章 虚拟内存与TLB

上一章我们看到了缓存如何通过局部性原理把慢速内存"伪装"成快速存储。现在我们要继续深入——不仅仅是性能问题，还有一个更根本的需求：如何让多个程序安全、高效地共享内存？答案就是虚拟内存（Virtual Memory）。如果说Cache是对程序员透明的硬件优化，那虚拟内存就是操作系统和硬件一起为我们构建的美丽幻觉——每个程序都觉得自己拥有整个地址空间。这一章，我们来揭开这个幻觉背后的硬件机制。

虚拟内存的动机与原理

虚拟内存不是因为它"很酷"才被发明的——它解决三个根本问题：

第一：隔离（Isolation）。想象一下，如果没有虚拟内存，进程A错误的指针操作可能直接覆盖进程B的数据——造成灾难性后果。虚拟内存为每个进程提供独立的地址空间，进程A的地址0x1000和进程B的地址0x1000指向不同的物理内存位置。一个进程的崩溃不会影响到另一个。

第二：大内存的错觉。虚拟内存让进程可以使用比物理RAM更大的内存空间——操作系统把不常用的部分换出到磁盘（swap），需要时再换入。进程甚至不知道自己的某些数据在磁盘上待过。

第三：简化的编程模型。程序员不需要手动管理内存覆盖（overlay）——那种手动把代码段载入、移出的古老做法。每个程序看到一个连续的、从0开始的虚拟地址空间，而底层的物理布局由操作系统和硬件管理。

核心思想非常清晰：将程序看到的地址（虚拟地址）和物理存储位置（物理地址）分离。这个分离，是操作系统安全性和多任务能力的基础。

页表与地址转换

虚拟内存以页（Page）为单位进行管理。物理内存被划分为相同大小的帧（Frame）。现代系统通常使用4KB作为标准页面大小——这个选择是基于平衡考虑的：页面太小则页表太大，页面太大则内部碎片严重。

地址转换的基本结构：

虚拟地址分解（以4KB页，48位虚拟地址为例）：

| 63:48 | 47:39 | 38:30 | 29:21 | 20:12 | 11:0 |
| 符号扩展 | PML4 | PDP  |  PD   |  PT   | Offset |
|         | 索引  | 索引  | 索引  | 索引  |        |

页大小 = 4KB = 2^12，所以 Offset = 12 bits
每级页表有 512 个条目（2^9），所以每级索引 9 bits
4级页表：9+9+9+9+12 = 48 bits 虚拟地址

物理地址分解：

|   51:12   | 11:0 |
|   PPN     | Offset|
| 物理页号  | 页内偏移|

为什么是多级页表？以x86-64为例，如果使用单级页表，则需要 2^48 / 2^12 = 2^36 个页表条目。每个条目8字节，总大小 2^39 = 512GB——显然不可行。多级页表的精髓在于：大多数进程只会使用地址空间中很小一部分（通常是底部一部分、顶部一部分），那些未使用的区域在页表中可以标记为空指针，不需要分配物理页来存放页表。

来看一个完整的地址转换的例子。假设有一个虚拟地址0x7F3A8B1C，4KB页面大小：

虚拟地址: 0x7F3A8B1C
二进制:    0111 1111 0011 1010 1000 1011 0001 1100

4KB页 => Offset = 12 bits
Offset:   0001 1100 (0x01C)  → 页内偏移

剩余20bits作为VPN（虚拟页号）:
VPN:      0111 1111 0011 1010 1000 (0x7F3A8)

4KB页, 4级页表 (x86-64 PML4):
  PML4索引: bits 47:39
  PDP索引:  bits 38:30
  PD索引:   bits 29:21
  PT索引:   bits 20:12

完成后得到物理页号（PPN），与offset拼接即得到物理地址。

RISC-V的Sv39方案使用三级页表（39位虚拟地址，也是4KB页），结构更简洁，特别适合嵌入式到服务器场景的灵活配置。

TLB：地址转换的Cache

好，现在问题是：每次内存访问都要经历4次页表查询（4级页表），每次查询又是一次内存访问。这意味着一次虚拟地址的内存访问，在幕后可能需要4-5次物理内存访问！如果不做任何优化，性能将暴跌5倍。

解决方案就是TLB（Translation Lookaside Buffer）——专门用于缓存虚拟地址到物理地址映射的硬件结构。本质上，TLB就是页表的"缓存"。它利用了地址转换的局部性：如果一个虚拟页被访问过，那么同一页中的其他地址很可能也被访问。TLB通常分成L1 TLB（指令和数据分开）和L2 TLB（统一）。

关键参数——TLB Reach（覆盖范围）：

TLB Reach = TLB 条目数 × 页面大小

L1 D-TLB (64 entries, 4KB pages):
  Reach = 64 × 4KB = 256KB

L2 TLB (2048 entries, 4KB pages):
  Reach = 2048 × 4KB = 8MB

64个条目×4KB = 只有256KB的覆盖范围！一个典型工作集动辄数MB乃至GB，L1 TLB的覆盖范围完全不够。这就是大页面（Huge Pages）的价值所在——如果使用2MB页面，64个条目×2MB = 128MB覆盖范围，一个数量级的提升。很多数据库和虚拟机管理程序强制启用透明大页（Transparent Huge Pages，THP）就是为了减少TLB缺失。在Linux上设置 echo always > /sys/kernel/mm/transparent_hugepage/enabled 后，TLB缺失率可以大幅降低。

处理器 L1 I-TLB L1 D-TLB L2 TLB 页面支持
Intel Core (Golden Cove) 64 entries, 8-way 64 entries (4K), 16 (2M/4M) 2048 entries, 8-way 4K, 2M, 1G
AMD Zen4 64 entries (4K), 32 (2M) 64 entries (4K), 32 (2M) 2048 entries, 8-way 4K, 2M, 1G
Apple M2 (Firestorm) 192 entries, 6-way 128 entries, 8-way ~2048 entries 4K, 16K, 64K, 2M
ARM Cortex-X3 64 entries (4K), 32 (2M) 48 entries (4K) ~1024 entries 4K, 16K, 64K, 2M, 1G

TLB缺失处理

当TLB缺失时，处理器必须从页表中获取正确的映射。有两种处理方式：

硬件页表遍历（Hardware Page Table Walker）。x86和ARM使用这种方法。处理器内部有一个专用状态机，当TLB缺失时自动遍历页表（自动发起内存读取，解析各级页表）。架构上，CR3寄存器（x86）保存了页表基地址。在RISC-V中，对应的寄存器是 satp CSR。

软件TLB缺失处理。MIPS和一些轻量级RISC-V实现使用这种方法。TLB缺失时触发一个异常（TLB Miss Exception），操作系统通过软件处理页表遍历。优点是灵活性高（OS可以自定义页表结构），缺点是异常处理开销大，这对于性能敏感场景不可接受。

在x86上，完整的页表遍历路径：CR3 → PML4 → PDP → PD → PT → PPN + Offset。每级页表条目不仅包含下一级的物理地址，还包含权限位（R/W、User/Supervisor）、存在位（Present）、访问位（Accessed）、脏位（Dirty）。操作系统利用Accessed位来实现页面置换（NRU算法），利用Dirty位来确定页面是否需要换出到磁盘。

安全视角：熔断与幽灵

2018年1月，安全研究团队披露了两个惊天动地的CPU漏洞——Meltdown（熔断）和Spectre（幽灵）。这俩漏洞不是软件bug，而是体系结构设计中的根本性弱点。

Meltdown利用了乱序执行的一个副作用：当一条指令尝试访问内核地址空间（比如读 kernel_data[0]），虽然硬件最终会因为权限检查失败而丢弃结果——但在乱序执行引擎中，在权限检查完成之前，后续指令可能已经使用这个"非法"数据做了计算，比如把它作为数组索引去访问内存，进而影响了Cache状态。攻击者通过测量Cache的访问时间来推断被丢弃数据的内容。修复方法是KPTI（Kernel Page Table Isolation）——把内核页表和用户页表分开，用户态下不映射内核空间，消除侧信道。

Spectre更难以修复——它利用了分支预测。攻击者先"训练"分支预测器让其倾向于某个方向，然后精心构造输入使得预测错误，在错误的预测路径上执行的操作会把数据泄露到Cache中。Spectre的变种包括：Bounds Check Bypass（绕过数组边界检查）、Branch Target Injection（污染分支目标地址）。

熔断漏洞告诉我们：体系结构中的性能优化（推测执行）可能在无意中打开安全漏洞——这是一种全新类型的硬件威胁。修复的代价极其高昂：Meltdown+KPTI在某些I/O密集负载上导致5-30%的性能下降。每一代新处理器都在硬件层面加强安全性——但实际上没有一种软件补丁能100%消除Spectre类攻击，因为它利用的是CPU工作方式的根本特性。

这背后更深远的教训是：传统上，体系结构假设"软件是可信的"，所有优化都围绕这一假设。但现实世界不是这样的——软件中有各种各样的攻击面。从那以后，安全成为微架构设计的首要约束之一，与性能、功耗同等重要。Intel的Golden Cove微架构、AMD的Zen 3/4都大幅加强了安全边界（更精细的分支预测隔离、更严格的推测窗口控制）。

// 演示TLB友好的内存访问 vs TLB不友好的访问

#include <stdio.h>
#include <stdlib.h>
#include <sys/time.h>

#define PAGE_SIZE 4096
#define NUM_PAGES 1024
#define NUM_ACCESSES 1000000

// TLB友好：顺序访问，每一页内的地址连续
void tlb_friendly_access(char *mem) {
    for (int i = 0; i < NUM_ACCESSES; i++) {
        volatile char c = mem[i % (NUM_PAGES * PAGE_SIZE)];
    }
}

// TLB不友好：跨页跳跃访问，每访问一个不同页
void tlb_unfriendly_access(char *mem) {
    for (int i = 0; i < NUM_ACCESSES; i++) {
        int page = (i * 53) % NUM_PAGES;  // 质数步长，遍历所有页
        volatile char c = mem[page * PAGE_SIZE];
    }
}

// 实际测试发现，tlb_friendly_access 比 tlb_unfriendly_access
// 快 3-5 倍，尽管它们访问的总数据量相同。
// 差别就在于: 友好版本每次访问同一页（TLB命中），
// 不友好版本每次访问新页（TLB频繁缺失）

所以，下次你在优化应用性能时，记得关注两件事：大页面是否已启用，以及数据访问模式是否对TLB友好。一个简单的调整（比如启用透明大页）可能带来20-30%的性能提升，完全不需要改代码。
