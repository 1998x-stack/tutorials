第3章 开发环境搭建

大家好，我是你们的讲师。前两章我们从概念上理解了 Agent 是什么、它的核心运行机制是怎样的。但说一千道一万，Agent 开发终究是一项工程实践——你得有一个称手的开发环境才能真正上手。

我见过太多人兴致勃勃地开始学 Agent 开发，结果第一天就卡在环境配置上：Python 版本冲突、API Key 泄露到 GitHub、Docker 镜像拉不下来、日志不会看、出了错不知道从哪里查起…… 这些问题看似琐碎，但每一个都能浪费你好几个小时甚至一整天。所以这一章，我们一次性把环境搭好，后面的学习才能一路顺畅。

本章目标： 到本章结束时，你将拥有一个完整可用的 Agent 开发环境——Python 3.12+ 虚拟环境、安全的 API Key 管理、三家主流模型提供商的 SDK 配置、Docker 开发容器、以及完善的调试工具链。这个环境将伴随你完成本教程后续所有的实战练习。

3.1 Python 环境与包管理

Agent 开发的主力语言毫无疑问是 Python。截至 2026 年中，Python 3.12 是推荐的生产版本，3.13 也已发布但生态兼容性仍在完善中。我推荐使用 Python 3.12 作为开发环境。

使用 uv：现代 Python 包管理器

如果你是 2024 年之前开始用 Python 的，大概率习惯用 pip + venv 或 Poetry。但在 2026 年，uv（Astral 团队出品）已经成为了 Python 包管理的事实标准——它用 Rust 编写，速度比 pip 快 10-100 倍，而且完全兼容 pip 的依赖格式。

# 安装 uv（macOS/Linux）
curl -LsSf https://astral.sh/uv/install.sh | sh

# 创建新项目
uv init llm-agent-project
cd llm-agent-project

# 创建虚拟环境
uv venv --python 3.12

# 激活虚拟环境
source .venv/bin/activate  # macOS/Linux
# .venv\Scripts\activate   # Windows

# 安装依赖
uv add openai anthropic langchain-core langchain-openai python-dotenv

uv 的优势不仅仅是速度。它自动管理 Python 版本（不需要你手动安装多个 Python）、自动生成 uv.lock 文件锁定所有依赖版本、支持 uv sync 一键同步环境。在团队协作中，别人只需要 uv sync 就能获得和你完全一致的依赖环境。

小贴士： 如果你偏好成熟的 Poetry，完全没问题。但是 uv 的锁定速度快得多——在大型项目中 uv 的依赖解析只需要几秒钟，而 Poetry 可能需要几分钟。个人推荐切换到 uv。

Python 依赖清单

以下是 Agent 开发最常用的 Python 包清单，按类别组织：

类别 包名 用途

3.2 LLM API Key 管理与安全存储

这是 Agent 开发中最基础但也最容易出安全问题的一环。"不小心把 API Key 提交到了 GitHub"——2025 年的一项统计显示，每天有超过 1000 个新的 API Key 因代码泄露而被扫描抓取。一个泄露的 Key 可能在几分钟内被滥用，造成数千美元的损失。

切身体会： 2024 年我在做一个开源项目时，一个协作者不小心把 .env 文件提交到了公开仓库。10 分钟内，我们的 OpenAI API Key 就被抓取并用于生成色情内容，账单上瞬间多了 800 美元。从那以后，我在所有项目中都强制使用 pre-commit hook 来检查 Key 泄露。

安全实践三步走

第一步——使用 .env 文件存储 Key：

# .env 文件（永远不要提交到 Git！）
OPENAI_API_KEY=sk-proj-xxxxxxxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxx
DEEPSEEK_API_KEY=sk-xxxxxxxxxxxx
LANGCHAIN_API_KEY=lsv2_xxxxxxxxxxxx
LANGCHAIN_TRACING_V2=true

# 可选配置
OPENAI_BASE_URL=https://api.openai.com/v1
DEFAULT_MODEL=gpt-4o
MAX_TOKENS=4096

第二步——使用 .gitignore 保护 .env：

# .gitignore
.env
.env.*
*.key
__pycache__/
.venv/

第三步——使用 python-dotenv 加载配置：

from dotenv import load_dotenv
import os

load_dotenv()  # 加载 .env 文件

# 获取配置
openai_key = os.getenv("OPENAI_API_KEY")
if not openai_key:
    raise ValueError("OPENAI_API_KEY 未设置，请检查 .env 文件")

# 使用配置
from openai import OpenAI
client = OpenAI(api_key=openai_key)

更安全的替代方案

.env 文件方案在开发阶段足够了，但在生产环境或团队协作中，你可能会考虑更安全的方案：

  • 环境变量注入： 通过 CI/CD 平台（GitHub Actions、GitLab CI）或容器编排（Kubernetes Secrets）注入环境变量
  • 密钥管理服务： 使用 AWS Secrets Manager、HashiCorp Vault、Azure Key Vault 等专业服务
  • 1Password CLI： 使用 op run 命令在本地临时注入凭证，Key 不会以明文形式保存在磁盘上
推荐做法： 对于个人开发，.env + .gitignore 已经足够。对于团队协作，建议使用 1Password CLI 或 Doppler，它们是安全性和便利性的最佳平衡点。

3.3 主流模型提供商接入配置

Agent 开发中你需要连接多个模型提供商。以下是最常用的三家配置方式。

OpenAI

import openai
from openai import OpenAI

client = OpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    # 如果需要代理或兼容接口
    # base_url=os.getenv("OPENAI_BASE_URL")
)

# 简单调用
response = client.chat.completions.create(
    model="gpt-4o",  # 2026年主力模型
    messages=[{"role": "user", "content": "Hello"}]
)

Anthropic Claude

import anthropic

client = anthropic.Anthropic(
    api_key=os.getenv("ANTHROPIC_API_KEY")
)

response = client.messages.create(
    model="claude-sonnet-4-20260514",  # 2026年Claude Sonnet 4
    max_tokens=4096,
    messages=[{"role": "user", "content": "Hello"}]
)

DeepSeek

from openai import OpenAI

# DeepSeek 使用兼容 OpenAI 的 API 格式
client = OpenAI(
    api_key=os.getenv("DEEPSEEK_API_KEY"),
    base_url="https://api.deepseek.com/v1"
)

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": "Hello"}]
)
选型建议： 2026 年的主流选择是 GPT-4o 和 Claude Sonnet 4 作为"主力模型"，DeepSeek 作为"高性价比备选"。在 Agent 场景中，建议使用 Claude 或 GPT-4o 作为推理中枢（Reasoning Engine），因为它们在高复杂度的 ReAct 循环中表现更稳定。简单任务（如数据提取）可以使用 DeepSeek 降低成本。

3.4 框架 SDK 安装

本教程后续章节会深入讲解多个框架，但开发环境中可以一次性安装好它们：

# LangChain 生态（推荐）
uv add langchain-core langchain-openai langchain-community langgraph

# AutoGen / AG2
uv add pyautogen

# CrewAI
uv add crewai

# OpenAI Assistants API（如需）
uv add openai  # 已安装

# 可选：工具开发
uv add mcp   # MCP 协议 SDK
uv add httpx # HTTP 客户端

安装完成后，用下面这个快速验证脚本来确认环境没问题：

import importlib

packages = [
    "openai", "anthropic", "langchain_core",
    "langgraph", "autogen", "crewai"
]

print("=== 环境验证 ===")
for pkg in packages:
    try:
        mod = importlib.import_module(pkg)
        ver = getattr(mod, "__version__", "unknown")
        print(f"  ✅ {pkg} == {ver}")
    except ImportError:
        print(f"  ❌ {pkg} 未安装")
print("=== 完成 ===")

3.5 Docker 开发环境与 DevContainer

在实际项目中，你几乎肯定需要在 Docker 环境中运行 Agent——不管是本地开发、测试还是生产部署。Docker 环境能保证"在我机器上能跑"变成"在任何机器上都能跑"。

最小 Dockerfile

FROM python:3.12-slim

WORKDIR /app

# 安装 uv
COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /bin/

# 复制依赖文件
COPY pyproject.toml uv.lock ./

# 安装依赖（利用 Docker 缓存层）
RUN uv sync --frozen --no-dev

# 复制源码
COPY . .

CMD ["uv", "run", "main.py"]

DevContainer 配置（VS Code）

如果你用 VS Code，DevContainer 是一个非常棒的方案——它把开发环境也容器化，团队成员共享完全一致的环境配置。

# .devcontainer/devcontainer.json
{
    "name": "LLM Agent Dev",
    "build": {
        "dockerfile": "Dockerfile",
        "context": ".."
    },
    "customizations": {
        "vscode": {
            "extensions": [
                "ms-python.python",
                "ms-python.vscode-pylance",
                "charliermarsh.ruff"
            ]
        }
    },
    "postCreateCommand": "uv sync",
    "forwardPorts": [8000, 9090],
    "mounts": [
        "source=${env:HOME}${env:USERPROFILE}/.config/,target=/root/.config/,type=bind"
    ]
}
为什么用 DevContainer？ 你不需要在宿主机上安装任何 Python 版本或包——容器里已经全部配置好了。只需 VS Code + Docker，打开项目就会自动提示"Reopen in Container"。对于 Agent 项目来说，容器环境尤其重要，因为 Agent 经常需要执行代码、调用系统工具，容器提供了天然的安全隔离。

3.6 调试工具与日志配置

Agent 系统的调试比传统应用要复杂得多——不是"请求-响应"的线性模式，而是多步推理、工具调用交织在一起的循环过程。如果没有好的调试工具和日志系统，出错时你根本不知道 Agent 在哪里出了问题。

结构化日志

import logging
import json
from datetime import datetime

class AgentLogger:
    """Agent 专用的结构化日志记录器"""

    def __init__(self, name="agent"):
        self.logger = logging.getLogger(name)
        self.logger.setLevel(logging.DEBUG)
        handler = logging.StreamHandler()
        handler.setFormatter(logging.Formatter(
            '%(asctime)s | %(levelname)-8s | %(message)s'
        ))
        self.logger.addHandler(handler)

    def log_step(self, step_num, thought, action=None, observation=None):
        """记录 ReAct 循环的每一步"""
        entry = {
            "step": step_num,
            "thought": thought[:200],
            "action": action,
            "observation": str(observation)[:200] if observation else None
        }
        self.logger.info(f"Step {step_num}: {json.dumps(entry, ensure_ascii=False)}")

    def log_error(self, step_num, error):
        self.logger.error(f"Step {step_num} Error: {error}")

    def log_metric(self, name, value):
        self.logger.info(f"METRIC {name}={value}")


# 使用
logger = AgentLogger()
logger.log_step(1, "用户想查天气，我需要调用天气API",
                action={"tool": "get_weather", "args": {"city": "北京"}},
                observation={"temp": "26°C"})

LangSmith 集成

对于生产级的 Agent 调试，LangSmith 是 2026 年最成熟的选择。它提供了完整的 Trace 追踪——每一次 LLM 调用、每一步工具执行、每一个 Token 消费都被记录下来，并可以在 Web UI 中可视化查看。

import os
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_API_KEY"] = "lsv2_..."
os.environ["LANGCHAIN_PROJECT"] = "llm-agent-tutorial"

# 然后所有的 LangChain 调用都会被自动追踪
踩坑实录： 刚开始使用 LangSmith 的时候，我犯了一个典型错误——所有调试日志都打印到控制台，结果在 ReAct 循环中每步输出几百行，根本看不清。后来我养成了"三步日志法"：每一步只记录 Thought 的摘要（前 100 字符）、Action 的名称和关键参数、Observation 的长度和摘要。这样即使 Agent 执行了 20 步，你也能一眼看出它在做什么、卡在了哪里。
