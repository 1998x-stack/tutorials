开篇：Git 不只是"Save button"

我见过太多刚入行的后端工程师把 Git 当成"高级版的保存按钮"——每天下班前 git add . && git commit -m "update"，偶尔再 git push 一下就觉得自己会用版本控制了。直到有一次，有人在 main 分支上直接 git push --force，把同事一天的代码全部覆盖掉了。那天的"复盘会"气氛极其凝重。

另一次更惨痛的经历。一个同事在合并 PR 时选择了"Rebase and merge"，但 rebase 过程中遇到了冲突，他没有仔细阅读冲突标记就直接强行 push 了 resolved 版本。结果是 rebase 把一些本不应该出现的旧代码带入了主线，导致一个微服务的 RPC 接口签名被静默修改。这个 bug 在测试环境没有被发现，上线后直接导致调用方服务全面报错，影响了线上流量超过 30 分钟。

这些故事说明了一个道理：Git 不只是一个工具，它是一套协作契约。不理解 Git 的底层原理和团队约定的工作流，你不仅效率低，还可能给团队带来灾难性的后果。这一章我们不只讲命令，更重要的是讲清楚"为什么"。

Git 核心概念：objects、refs、DAG

Git 本质上是一个内容寻址的文件系统，加上一个在此基础上构建的版本控制用户界面。理解这个定位，你就理解了 Git 的所有设计选择。

Git 的核心数据结构是 objects，存储在 .git/objects/ 目录下。共有四种对象类型：

  • Blob：文件内容。文件名不是 blob 的一部分——文件名存储在 tree 对象中。这就是 Git 可以高效处理文件移动和重命名的原因。
  • Tree：目录结构，包含文件名、权限和指向 blob 或其他 tree 的指针。
  • Commit：一次提交的快照，包含指向根 tree 的指针、父 commit 的 SHA、作者/提交者信息、提交消息和时间戳。
  • Tag：一个固定的引用，通常指向一个 commit（附注标签还包含标签信息和 GPG 签名）。

这些对象通过 SHA-1 哈希值进行寻址，commit 之间通过 parent 指针形成有向无环图（DAG）。分支（branch）只是一个指向某个 commit 的可移动指针——这就是 Git 创建分支如此之快的根本原因，它实际上只是在 .git/refs/heads/ 下写了一个 40 字节的文件。

💡 认知跃迁：一旦你理解了"分支只是一个指针"这件事，Git 的很多"神奇"行为就变得理所当然了。git merge 创建一个新的 merge commit，有两个 parent。git rebase 是重新"粘贴"commit 到一个新的 base 上。git reset 只是把这个分支指针移到一个不同的 commit。这不是魔法，是指针操作。

另一个常被忽视的概念是 staging area（索引/暂存区）。它的存在让你可以分批次地组织提交——"我要把这个文件的这部分改动和那个文件的改动放在同一个 commit 里，其他改动留到下一个 commit"。很多新手不理解 staging area 的意义，直接 git add . && git commit 走天下。但当你需要精确控制提交粒度时（比如修复一个安全漏洞你要确保相关改动在一个 commit 中，不相关的改动不要混入），staging area 就是你的最佳武器。

分支策略对比：GitFlow vs GitHub Flow vs Trunk-Based

没有银弹。不同团队规模、发布节奏、风险容忍度对应不同的分支策略。下面是 2026 年三种主流分支策略的对比：

维度 GitFlow GitHub Flow Trunk-Based Development
核心思想 多分支并行，严格的分支角色 功能分支 + PR，简洁 所有开发者直接在主干（或短生命期分支）上工作
主要分支 main, develop, feature, release, hotfix main + feature branches main（或 trunk）
发布流程 从 develop 创建 release 分支，稳定后合并到 main PR 合并到 main 即发布 小批量提交到 trunk，使用 feature flag 控制发布
适合场景 有固定发布周期的产品 持续部署的 Web 应用 成熟的 CI/CD 团队，追求极致交付速度
分支生命期 数天到数周 数小时到数天 数小时以内
复杂度 高（需要频繁合并） 低 低（但纪律要求极高）

我的建议：对于大多数后端团队，从 GitHub Flow 开始就够了。GitFlow 的复杂度在大团队中容易失控，而 Trunk-Based 对 CI 自动化和团队纪律要求极高。只有当你确实遇到 GitFlow 要解决的问题（如同时维护多个版本）时，才值得引入它的复杂度。

合并与变基：merge vs rebase vs squash

这是 Git 社区最著名的"宗教战争"。其实它们各有用途，不存在"永远用 merge"或"永远用 rebase"的绝对正确。

# 场景：从 main 分支拉出 feature 分支开发了 3 天，
# 期间 main 分支前进了 5 个 commit

# 方案一：merge（创建 merge commit）
$ git checkout feature
$ git merge main
# 结果：feature 分支上多了一个 merge commit
# 优点：真实记录分支合并历史
# 缺点：commit 图变得复杂

# 方案二：rebase（重新"垫"在 main 的最新 commit 上）
$ git checkout feature
$ git rebase main
# 结果：feature 分支的 commit 被重新应用到 main 的顶端
# 优点：线性历史，干净整洁
# 缺点：改变了 commit 的 SHA，不要对公共分支做 rebase

# 方案三：squash（压缩成一个 commit）
$ git checkout main
$ git merge --squash feature
$ git commit -m "feat: add payment module"
# 结果：feature 的所有改动被压缩成一个 commit
# 优点：保持 main 历史极简
# 缺点：丢失了中间的开发细节
🤝 团队约定建议：一个实用的规则是——合并到公共分支用 merge，保持自己的功能分支整洁用 rebase。具体来说：在自己的 feature 分支上可以自由 rebase（甚至 rebase -i 交互式重写历史），但在合入 main 分支时使用 merge commit 或 squash merge。不要在已经推送到共享仓库的分支上做 rebase——这是唯一绝对的 Git 禁忌。

提交规范：Conventional Commits

统一提交信息格式的价值远超"好看"。规范的提交信息可以自动生成 CHANGELOG、驱动语义化版本、辅助 CI 流程决策（"这次提交包含了 breaking change，需要发大版本"）。

Conventional Commits 规范的格式如下：

<type>[optional scope]: <description>

[optional body]

[optional footer(s)]

# 好的提交示例
feat(auth): implement OAuth 2.0 authorization code flow

- Add authorization endpoint with code challenge support
- Add token exchange endpoint with PKCE verification
- Add refresh token rotation mechanism

Closes #234

fix(api): correct user query pagination off-by-one error

The offset calculation was incorrectly adding 1 to the page
number, causing the first item of each page to be skipped.
Fixed by using (page-1)*limit as the offset.

fixes #567

docs(readme): update deployment instructions for K8s v1.28

chore(deps): bump golang from 1.22 to 1.23

BREAKING CHANGE: user service endpoint returns 202 instead of 200

使用 git log --oneline 看一下规范的提交历史——每行都是一个清晰的意图表达。"fix: correct user query pagination" 比 "update" 好了 100 倍。在 2026 年，semantic-release、release-please 等工具已经能基于 Conventional Commits 自动管理版本发布，配置一个 CI 工作流就可以实现"合并 PR 即发布新版本"的自动化。

Code Review 最佳实践

Code Review 不是"找茬"，它是知识共享和集体代码所有权的实践。一个健康的 Code Review 文化，比任何静态分析工具都更能提升代码质量。

以下是我在多年实践中总结的 Code Review 检查清单：

  • PR 粒度要小：一个 PR 应该只做一件事。500 行以内的 diff 最容易审查。超过 1000 行的 PR，reviewer 会本能地"滑动浏览"而忽略关键细节。
  • PR 描述要清晰：说明背景、改动内容、测试方式。好的 PR 描述让 reviewer 不用猜测"为什么要这么改"。
  • 关注"设计"而非"风格"：代码风格问题交给 linter/formatter。Review 应该关注算法正确性、边界情况处理、并发安全、错误处理、性能影响、可测试性。
  • 建设性反馈：不要说"这段代码写得不好"，而要说"这里考虑用 sync.Map 替代显式加锁，可以减少锁竞争"。给出具体的改进建议，而不是笼统的批评。
  • 及时 Review：PR 搁置超过 24 小时会严重影响开发效率。建立团队约定——"Review 响应时间不超过 4 小时"。
  • 从 reviewer 的角度思考：提交 PR 之前，自己先读一遍 diff。如果你自己都看不懂自己写的代码，reviewer 肯定也看不懂。
🌟 高阶建议：Code Review 的最高境界，是团队中每个人都能理解代码库的每一部分——不是因为每个人读过了每一行代码，而是因为每次 Review 都是一次知识传递。一个新同事接手一个模块后，通过 Review 他过去不熟悉的改动，他能逐渐建立起对整个系统的理解。这就是"集体代码所有权"的真实含义。
