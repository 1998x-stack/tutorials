开篇：做一锅"恰到好处"的训练数据

大家好，我是你们的讲师。到目前为止，我们已经走访了全球的数据集超市（第 2 章），处理了中文数据的独家难题（第 3 章），搜集了垂直领域的高价值语料（第 4 章），见识了 Common Crawl 这座数据油田（第 5 章），并划好了数据合规的红线（第 6 章）。现在，我们面临一个"幸福的烦恼"：数据源太多了，该怎么把它们混在一起？

想象一下你在做饭。食材都准备好了：上等的雪花牛肉（高质量网页数据）、高原有机蔬菜（Wikipedia/书籍）、秘制酱料（代码和数学数据）、以及一些特殊的香料（指令数据）。问题是：每种食材放多少，才能做出一道完美的菜？

这就是数据配比（data mixing / data composition）要回答的问题。它不是简单的"把所有数据堆在一起随机采样"，而是精心设计不同数据源的采样比例，以最大化模型在目标任务上的表现。这个问题的答案，远比看起来要复杂得多。

7.1 70-15-10-5：经验法则的由来

在数据配比这个领域，流传最广的"经验法则"是所谓的 70-15-10-5 配比：

数据类别 建议占比 代表数据源 作用
网页数据 70% Common Crawl 清洗版、C4、RefinedWeb 提供广泛的常识和语言基础
书籍 + Wikipedia 15% BookCorpus、Project Gutenberg、Wikipedia 提供高质量、长篇、结构化的叙事文本
代码 + 学术论文 10% The Stack、ArXiv、OpenWebMath 提升推理、逻辑和结构化思维能力
其他（对话、指令等） 5% FLAN、UltraChat、StackExchange 提升对话和指令遵循能力

这个比例不是凭空出现的——它大致反映了 LLaMA 系列和 GPT 系列模型论文中披露的数据配比。LLaMA 论文中提到，他们的训练数据中 Common Crawl 占 67%，C4 占 15%，Wikipedia 占 4.5%，GitHub 占 4.5%，Books 占 4.5%，ArXiv 占 2.5%，StackExchange 占 2.0%。把这些数字按照类别合并，你会发现非常接近 70-15-10-5 这个比例。

但这里有一个重要的提醒：这些比例是 token 级别的比例，不是文件数量或字节数级别的比例。 因为不同类型的文本经过 tokenizer 后的压缩比不同——数学文本包含大量数字和符号，token 数通常比等长自然语言文本要少；中文文本的 token 数也比英文要多（因为中文字符被切得更碎）。所以按 token 配比和按字节配比是不同的，这是我们在实际配比时必须明确的。

实操建议：如果你的团队刚起步，直接使用 70-15-10-5 作为初始配比是一个安全的起点。不必在起步阶段就花大量时间做配比优化——先用标准配比跑通实验，再用消融实验来验证和调整。很多团队浪费了几个月试图"优化"配比，结果发现效果还不如直接用 LLaMA 的配方。

7.2 静态混合 vs 课程学习

将多个数据源混合在一起，有两种基本的策略：

静态混合（Static Mixing）：在整个训练过程中，保持各数据源的采样比例不变。这是最常用的方法，实现简单、效果稳定。

课程学习（Curriculum Learning）：随着训练的推进，动态调整不同数据源的比例。典型策略是"先易后难"——早期多使用质量高但话题窄的数据（Wikipedia、书籍），后期逐渐过渡到更广泛但噪声更多的数据（网页文本）。

那么哪种更好呢？多项研究（包括 Llama 团队的经验分享和 DeepSeek 的技术报告）表明一个出乎意料的结论：精心设计的静态混合方案，通常不低于甚至优于复杂的课程学习方案。

静态混合 vs 课程学习的效果对比（基于 1.3B 参数模型）：
  ┌─────────────────┬──────────────┬──────────────┬──────────┐
  │ Benchmark       │ 静态 50-30-20│ 简单课程学习 │ 复杂课程  │
  ├─────────────────┼──────────────┼──────────────┼──────────┤
  │ LAMBADA         │    3.12      │    3.15      │   3.14   │
  │ HellaSwag       │    53.8%     │    53.5%     │   53.6%  │
  │ ARC-Challenge   │    28.5%     │    28.3%     │   28.7%  │
  │ PIQA            │    71.8%     │    71.4%     │   71.6%  │
  └─────────────────┴──────────────┴──────────────┴──────────┘

这并不意味着课程学习没有价值，而是说它的边际收益远低于大多数人的预期。在数据配比上取得 90% 的效果靠静态混合，剩下的 10% 靠课程学习。如果你在一个 10 亿美元的预训练项目中，那 10% 的课程学习收益当然值得追求；但如果你在一个 1B 参数的原型实验中，静态混合就足够了。

一个反直觉的事实：在 DeepSeek-V2 的技术报告中，他们对比了多种数据配比策略，发现了一个有趣的现象——"均匀混合"（即仅仅按数据源大小比例采样）的效果，居然比很多复杂的"优化配比"方案更好。这暗示着：在数据配比这件事上，简单方法往往被低估了。如果你发现一个复杂的配比方案只在某个特定 benchmark 上提升了 0.5%，但对其他 benchmark 都没有影响——那大概率是随机噪声，不是真正的改进。

7.3 经典模型的数据配比揭秘

让我们来看看几个代表性大模型的数据配比策略。由于越来越多的公司开始披露训练数据细节，我们可以做出一个相对完整的对比：

数据源 LLaMA 2 (70B) Mistral (7B) DeepSeek-V2 OLMo (7B)
网页文本 ~67% ~70% ~65% ~55%
书籍 ~4.5% ~5% ~8% ~10%
Wikipedia ~4.5% ~3% ~3% ~5%
代码 ~4.5% ~10% ~12% ~15%
学术论文 ~2.5% ~2% ~5% ~5%
对话/指令 ~2% ~5% ~5% ~5%
其他 ~15% (C4) ~5% ~2% ~5%
总 token 数 2T ~8T (多轮) 8.1T 3T

观察这张表，你会发现几个明显的趋势：

  • 代码配比在持续上升：从 LLaMA 2 的 4.5% 到了 OLMo 的 15% 和 DeepSeek-V2 的 12%。这说明行业越来越认可代码数据对推理能力的核心贡献。
  • 书籍数据的占比也在上升：OLMo 和 DeepSeek-V2 都显著增加了书籍语料的比例，从 LLaMA 2 的 4.5% 翻倍到了 8%-10%。
  • 对话/指令数据的占比虽然小，但已经成为标准配置：几乎每个模型都会在预训练的最后阶段混入 1%-5% 的指令格式数据。
  • 没有一个固定的"最优配比"：每个模型根据其目标能力侧重点不同，配比有明显差异。Mistral 更注重代码能力（10%），DeepSeek-V2 注重综合能力（论文、书籍、代码都占比偏高）。
对中文模型的数据配比建议：如果你在训练一个中英双语模型，可以考虑：网页数据 60%（其中英文 40% + 中文 20%），书籍 15%，代码 12%，Wikipedia 5%（中英混合），学术论文 5%，对话/指令 3%。这个配比已经在多个开源中文模型上被验证过效果不错。

7.4 如何优化数据配比：代理模型方法

前面我们说的都是"参考别人的经验值"。但如果你的场景特殊（比如你特别关注医疗问答能力），或者你想验证自己的配比假设，就需要一种系统的优化方法。

目前业界最常用的配比优化方法是 代理模型方法（Proxy Model Approach）：

  1. 设计 N 组不同的数据配比方案
  2. 用每组配比训练一个小型代理模型（1B-3B 参数），只训练到一阶段（如 100B token）
  3. 在多个下游任务上评测这 N 个模型
  4. 根据评测结果选择最优配比
  5. 将最优配比放大到全尺寸模型（7B/70B 参数）的训练中

代理模型方法的有效性基于一个关键假设：数据配比对小型模型和大型模型的影响是"单调的"——即对 1B 模型有利的配比变化，对 70B 模型也是有利的。多项研究验证了这一假设大致成立（虽然不绝对）。

代理模型调优流程示例：

  配比方案候选集:
  ┌─────────────┬──────┬──────┬──────┬──────┐
  │ 方案        │ 网页 │ 书籍 │ 代码 │ 数学 │
  ├─────────────┼──────┼──────┼──────┼──────┤
  │ Baseline   │ 70%  │ 15%  │ 10%  │  5%  │
  │ Code+      │ 60%  │ 15%  │ 20%  │  5%  │
  │ Math+      │ 65%  │ 15%  │ 10%  │ 10%  │
  │ Book+      │ 60%  │ 25%  │ 10%  │  5%  │
  └─────────────┴──────┴──────┴──────┴──────┘

  训练 4 个 1.3B 代理模型 → 在 8 个 benchmark 上评测
  → 选择在目标任务上综合得分最高的方案
  → 将配比应用于 13B 或 70B 模型
经验教训：不要只靠"直觉"来选择配比。我在一个项目中见过团队凭"感觉"把代码配比从 8% 提高到了 25%，理由仅仅是"代码很重要"。结果训练出来的模型确实代码能力提升了，但通用知识问答能力下降了 10%。浪费了数百万元的算力。任何配比调整都应该先通过代理模型验证——这是数据工程中最有价值的投资回报比之一。

7.5 Token 分布 vs 字节分布：一个容易被忽略的细节

你可能会觉得："配比不就是控制每种数据源占总 token 数的百分比吗？"——答案是：不完全对。

问题出在 tokenizer 对不同语言和领域数据的编码效率上。我们来看一个具体的例子：

不同数据类型的 token 化效率：
  ┌──────────────┬───────────────┬─────────────┬────────────┐
  │ 数据类型     │ 原始字符数    │ token 数    │ 压缩比     │
  ├──────────────┼───────────────┼─────────────┼────────────┤
  │ 英文新闻     │ 1,000 chars   │ ~250 tokens │ 4:1        │
  │ 中文新闻     │ 1,000 chars   │ ~600 tokens │ 1.7:1      │
  │ Python 代码  │ 1,000 chars   │ ~220 tokens │ 4.5:1      │
  │ LaTeX 论文   │ 1,000 chars   │ ~180 tokens │ 5.6:1      │
  │ 中文代码注释 │ 1,000 chars   │ ~500 tokens │ 2:1        │
  └──────────────┴───────────────┴─────────────┴────────────┘

看到问题了吗？同样的 1,000 个字符，英文新闻被压缩为 250 个 token，而中文新闻被压缩为 600 个 token——中文的 token 效率只有英文的 40%。这意味着，如果你按"token 占比"来配比，中文数据在"字节层面"的实际占比会被系统性地低估。

这带来的实际后果是：假设你想让模型在预训练中看到 30% 的中文 token 和 70% 的英文 token，如果按照 token 比例采样，中文数据在字节/字符层面的实际输入量将非常庞大——因为每个中文 token 编码的信息量远少于英文 token。这会导致模型吸收的中文"内容"远超你的预期，可能挤压其他语言的训练效果。

踩坑故事：我们在一个中英双语模型的训练中就踩过这个坑。我们的数据配比目标设为"30% 中文 token + 70% 英文 token"。训练了 10 万步后我们检查训练状态，发现 loss 下降曲线中英文的部分明显滞后于同等规模的单语模型。后来才意识到问题所在：中文数据按 token 占比 30% 采样时，按字节计算的实际输入占比接近 50%。模型花了太多容量"消化"中文文本，而中文文本的信息密度又低于同字节数的英文。最终我们改为按"字节占比"来配比——中文 20% 字节 + 英文 80% 字节——效果明显改善。所以请记住：按 token 配比还是按字节配比，选定了就要明确记录。

7.6 从 Chinchilla 视角看数据配比

回到第 1 章我们讨论的 Chinchilla 最优数据比。Chinchilla 告诉我们：对于一个 N 参数的模型，最优训练 token 数大约是 20N。那么问题来了——这 20N 个 token 应该朝着什么样的配比去分配？

这里有一个有趣的权衡：当你有"太少的 token"时和"太多的 token"时，配比策略应该不同：

  • 数据不足时（你只有 10N token，低于 Chinchilla 最优的 20N）：你需要专注于最高质量的数据。建议提高 Wikipedia/书籍/代码等高质量数据的占比（增加到 30%-40%），减少网页数据的占比。用"精炼"来弥补"量不足"的短板。
  • 数据充裕时（你有 40N+ token，远超 Chinchilla 最优）：你有"浪费数据"的空间。此时可以降低高质量数据的占比，更多地使用网页数据来覆盖长尾知识。高质量数据的作用从"提供知识"变成了"提供语言基准"。
  • 数据处于最优附近（你有 15N-25N token）：标准配比（70-15-10-5）最适用。

这个视角告诉我们：数据配比不是一个独立于模型规模和数据总量的变量，而是与它们密切耦合的。同样的配比，在不同的数据总量下，对模型能力的影响可能是完全不同的。

一句话总结配比的"哲学"：
数据配比的本质不是"寻找黄金比例"，而是在给定数据总量和模型规模的约束下，最大化每个 token 对模型能力的边际贡献。高质量数据适合"少而精"地使用，网页数据适合"多而泛"地使用。好的配比就是让这两者在你的资源约束下达到最优平衡。
