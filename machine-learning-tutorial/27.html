27.1 模型不部署，等于不存在

你花了两周调出了一个AUC 0.92的模型，存在了joblib文件里。然后呢？如果这个模型不能给业务系统提供实时推理服务，它的价值就是零。模型部署是ML价值实现的"最后一公里"，也是技术栈最多、坑最多的一环。

我最早做模型部署的时候，天真地以为就是"Flask包一层API"，结果上线第一天就翻车：并发一上来服务就崩了，因为我的模型是单线程的，而且没有做任何批处理优化。后来逐步引入了异步处理、模型量化、请求队列、健康检查——这些都是血的教训。

27.2 FastAPI：ML模型服务的标配

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import joblib
import numpy as np

app = FastAPI(title="ML Model Service")

# 启动时加载模型
model = None
@app.on_event("startup")
async def load_model():
    global model
    model = joblib.load("model.pkl")

# 定义输入schema
class PredictRequest(BaseModel):
    features: list[float]

class PredictResponse(BaseModel):
    prediction: float
    probability: float | None = None

@app.post("/predict", response_model=PredictResponse)
async def predict(request: PredictRequest):
    if model is None:
        raise HTTPException(status_code=503, detail="模型未加载")
    X = np.array(request.features).reshape(1, -1)
    pred = model.predict(X)[0]
    prob = model.predict_proba(X)[0].max() if hasattr(model, 'predict_proba') else None
    return PredictResponse(prediction=float(pred), probability=prob)

@app.get("/health")
async def health():
    return {"status": "ok", "model_loaded": model is not None}

# 运行: uvicorn app:app --host 0.0.0.0 --port 8000 --workers 4
💡 FastAPI为什么是ML部署的首选？(1) 异步支持——高并发时不会阻塞。(2) 自动生成OpenAPI文档——访问/docs就能看到交互式API文档。(3) Pydantic数据验证——自动校验输入格式，错误请求在到达模型前被拦截。(4) 启动事件——模型在服务启动时加载一次到内存，避免每个请求都重新加载。

27.3 模型优化：让推理更快更省

技术原理效果
量化 (Quantization)把FP32权重压缩到INT8/INT4模型缩小4-8倍，推理快2-4倍，精度损失<1%
剪枝 (Pruning)删除接近0的权重模型缩小30-90%，需要重训练恢复精度
ONNX统一模型格式，跨框架部署推理加速30-200%（取决于模型和硬件）
TensorRTNVIDIA推理优化引擎GPU推理加速3-10倍
vLLMLLM专用高吞吐推理相比HuggingFace推理快10-30倍
# ONNX导出示例：sklearn/PyTorch → ONNX → ONNX Runtime推理
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
import onnxruntime as ort

# sklearn模型转ONNX
initial_type = [('float_input', FloatTensorType([None, X_train.shape[1]]))]
onnx_model = convert_sklearn(model, initial_types=initial_type)

# 用ONNX Runtime推理
session = ort.InferenceSession(onnx_model.SerializeToString())
pred = session.run(None, {'float_input': X_test.astype(np.float32)})[0]

27.4 Docker：一次构建，到处运行

# Dockerfile 示例
FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY model.pkl .
COPY app.py .

EXPOSE 8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "2"]
⚠️ Docker部署常见问题：(1) 模型文件太大（>1GB）导致镜像构建和推送极慢——应使用模型下载脚本在容器启动时拉取模型，而不是COPY进镜像。(2) 内存不足——单个模型推理需要2GB显存，两个worker同时推理需要4GB，确保容器内存限制高于 peak 内存使用。(3) 忘记设置health check——K8s/Docker Compose依赖health check来自动重启不健康的服务。

27.5 模型监控：部署不是终点

模型上线后，你需要持续监控以下指标：

  • 数据漂移（Data Drift）：输入特征的分布是否偏离了训练集？比如训练时用户平均年龄35岁，现在变42岁了——模型可能在新分布下失效
  • 概念漂移（Concept Drift）：同样的输入，目标变量的含义变了。比如"用户流失"的定义从"30天不活跃"改成了"14天不活跃"
  • 模型性能：预测延迟、吞吐量、错误率、业务指标（如预测准确率和转化率的变化）

基础的监控可以用Prometheus收集指标 + Grafana可视化，或直接用简单的日志 + 定期脚本做统计。复杂的漂移检测可以用Evidently AI或NannyML等专用工具。

🔑 部署哲学：好的部署不是"一劳永逸"，而是"可以随时安全地更新"。你的部署架构应该支持：蓝绿部署（新旧版本同时运行，流量切换）、金丝雀发布（先给5%流量验证新模型）、自动回滚（指标异常时自动切回旧版本）。模型不是静态的——它会衰减、需要重训练、需要被更好的模型替换。部署系统要为此而设计。
