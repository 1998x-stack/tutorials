第一章：RAG概述与演进

大家好，我是你们的讲师。今天我们要开启一段非常有意思的旅程——RAG（Retrieval-Augmented Generation，检索增强生成）。在开始之前，我想问大家一个问题：你有没有遇到过这样的场景——问ChatGPT一个关于最近新闻的问题，它却一本正经地编造了一个答案？或者问它你公司内部的知识库问题，它表示"毫不知情"？这些令人沮丧的时刻，恰恰催生了过去两年AI工程领域最火热的技术范式之一。

RAG不是一个花哨的学术概念，它是实实在在解决大模型落地难题的一套工程方法论。2023年被很多人称为"RAG元年"，从那时到现在，几乎每一个生产级大模型应用都在某种程度上使用了RAG。为什么？因为大模型本身有三大绕不过去的痛点，而RAG恰好是这三大痛点的手术刀。

一、大模型的三大痛点

痛点一：幻觉（Hallucination）

2023年初，一位律师在法庭文件中引用了ChatGPT生成的六个案例——结果全是编的。法官差点对他进行制裁。这件事在当时轰动一时，被称为"AI幻觉诉讼第一案"。大模型本质上是一个next token prediction机器，它并不知道"事实"是什么，它只知道"在给定上下文中哪个词最可能出现在这里"。当它缺乏相关知识时，它不会说"我不知道"，而是会自信地胡编乱造。这种"流畅的胡说"远比明显的错误更危险。

痛点二：知识截断（Knowledge Cutoff）

即使是最新的GPT-4o或Claude 3.5 Sonnet，也有一个训练数据的截止日期。你问它"2026年世界杯决赛结果"，如果它的训练数据只到2025年，它要么拒绝回答，要么猜测。在快速变化的业务场景中——比如电商平台的实时库存查询、金融市场的即时行情、或者企业内部每天更新的知识库——模型的知识滞后是一个致命问题。

痛点三：黑箱不可控

大模型是端到端训练的神经网络，你无法控制它"使用了哪部分知识"来生成答案。当你问一个法律问题，你希望它引用具体的法条和判例来源，而不是给你一段"看起来很像那么回事"的总结。没有引用溯源能力的AI，在企业合规场景下基本不可用。

这三个痛点，每一个都直指大模型落地的要害。而RAG的解决方案非常优雅：不给模型喂更多的训练数据，而是给它配一本可以随时查阅的参考书。

二、RAG的诞生背景：为什么2023年是转折点

其实"检索+生成"的思路并不新。早在2020年，Lewis等人就发表了经典的《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》论文。但为什么直到2023年RAG才真正爆发？原因有三：

  • LLM的成熟：2022年底ChatGPT发布后，大家发现生成能力已经足够好，短板在知识层面。RAG正好补这块短板。
  • 向量数据库的崛起：Pinecone、Milvus、Chroma等向量数据库在2023年迅速成熟，让语义搜索从学术研究变成了工程师可用的工具。
  • 嵌入模型的平民化：OpenAI的text-embedding-ada-002和后来的BGE系列让高质量embedding唾手可得。

我清楚记得2023年夏天，我在一家电商公司做客服系统改造。纯靠prompt engineering的LLM客服被用户投诉"答非所问"，退货政策常常讲错。我们花了三天时间，用最朴素的RAG——文档碎片化存Chroma，检索后用GPT-3.5-turbo来生成——准确率从62%飙到了91%。这是RAG的魔力：它不要求模型变得更强，只要求它学会查资料。

三、Naive RAG → Advanced RAG → Agentic RAG 三阶段演进

RAG的发展经历了三个清晰的阶段，每个阶段都解决了前一个阶段的痛点：

阶段 时期 核心特征 局限性
Naive RAG 2023年上半年 文档分块 → Embedding → 向量检索 → LLM生成 检索质量差，对检索结果无过滤，无法处理多跳问题
Advanced RAG 2023下半年~2024年 增加重排序、查询重写、HyDE、混合检索、滑动窗口 流程固定，缺乏自主推理能力
Agentic RAG 2025年至今 多轮规划、工具调用、自我反思、动态决定是否检索 复杂度高，延迟增加，需要更可靠的Agent框架

从Naive到Advanced，我们学会了如何让检索更精准、更全面。而从Advanced到Agentic，我们学会了让系统自己决定"什么时候查、查什么、怎么用"。这三种模式不是替代关系——在大多数生产场景中，80%的查询用Naive或Advanced RAG就足够了，只有需要多步推理的复杂查询才需要Agentic的介入。

演进速记：Naive RAG = 查字典；Advanced RAG = 带目录、索引和交叉引用的百科全书；Agentic RAG = 一个会主动说"等等，这个问题需要先查A再查B"的研究助理。三者的根本区别在于系统是否有自主规划和迭代能力。

四、RAG vs 微调 vs 提示工程

在做RAG之前，很多人会犹豫：我到底是该微调模型，还是用RAG，还是靠prompt engineering？让我们用一个表格把它们说清楚：

维度 RAG 微调（Fine-tuning） 提示工程（Prompt Engineering）
知识更新 实时，替换文档即可 需要重新训练/微调 需要修改prompt模板
外部知识 原生支持，天然适合 需要将知识注入训练数据 只能通过context window提供
引用溯源 天然支持，可返回原文片段 不支持，知识内化于参数 需通过prompt要求模型提供
推理成本 检索+生成，token消耗较大 推理时无额外检索开销 token消耗取决于prompt长度
开发周期 数天~数周 数周~数月 数小时~数天
适合场景 知识问答、文档分析、客服 风格迁移、特定格式输出 简单任务、快速原型
可维护性 高（知识库独立管理） 低（修改知识需重训） 中（prompt易碎）

现实中的最佳实践往往是三者结合。用RAG提供知识，用prompt engineering控制行为，用微调优化输出风格——但这节课我们聚焦RAG，先把它做扎实。

五、RAG系统四阶段总览架构

一个标准RAG系统的工作流程可以划分为四个阶段，这也是我们整个课程的核心线索：

  1. Index（索引）：文档加载 → 解析 → 分块 → Embedding → 存入向量数据库。这是离线阶段，负责建立知识库。
  2. Retrieve（检索）：用户查询 → 查询编码 → 向量检索（+稀疏检索） → 返回候选文档。这是在线阶段的第一步。
  3. Augment（增强）：对检索结果进行重排序、过滤、去重、压缩，并与原始查询合并成最终prompt。
  4. Generate（生成）：将增强后的上下文送入LLM，生成带引用的最终答案。

下面是最最简化的RAG代码——不要被它的简单骗了，它包含了RAG的全部灵魂：

from openai import OpenAI
import chromadb

# 1. Index — 把文档片段存进向量库
client = chromadb.Client()
collection = client.create_collection("my_knowledge")
collection.add(
    documents=["RAG是检索增强生成的缩写",
               "向量数据库主要用于存储和检索嵌入向量"],
    ids=["doc1", "doc2"]
)

# 2. Retrieve — 检索相关片段
query = "什么是RAG?"
results = collection.query(query_texts=[query], n_results=1)

# 3. Augment — 构造增强提示
context = results["documents"][0][0]
prompt = f"基于以下内容回答问题：\n{context}\n问题：{query}"

# 4. Generate — 让LLM生成回答
llm = OpenAI()
response = llm.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": prompt}]
)
print(response.choices[0].message.content)

这就是RAG的全部秘密——不复杂，但每一个环节都有大量可以深挖的细节。从下一章开始，我们会逐一深入每个环节。

学习建议：如果你对RAG完全陌生，建议动手运行这个最小demo，感受一下"检索→增强→生成"的完整流程。哪怕是空跑一次，也比只看不写有效十倍。在你的本地环境配好OpenAI key和Chroma后，这段代码应该能在5分钟内跑起来。
