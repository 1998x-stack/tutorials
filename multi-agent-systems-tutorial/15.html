第15章 A2A协议：智能体互联网

还记得上一章我们讨论的MCP吗？MCP解决了Agent与工具之间的"最后一公里"——让每个Agent都能通过统一协议调用数据库、文件系统、API等各种工具。但这里留下了一个更棘手的问题：Agent与Agent之间怎么通信？

假设你有一个专门做市场调研的Agent，还有一个专门写营销文案的Agent，它们来自不同团队、不同公司、跑在不同的框架上。你希望调研Agent完成后自动通知文案Agent开工，并在过程中协商调整方向——这该怎么搞？

这就是A2A（Agent-to-Agent）协议要解决的问题。如果说MCP是Agent世界的USB标准（让Agent插上工具），那么A2A就是Agent世界的HTTP协议（让Agent互相访问）。没有A2A，每个多Agent系统就是一个封闭的孤岛；有了A2A，我们才能真正谈论"Agent互联网"。

15.1 A2A的诞生与标准化之路

A2A协议由Google在2025年4月发起，最初的目标非常朴素：让不同组织开发的Agent能够互相发现、通信和协作。Google在发布v0.9草案后就迅速将其捐给了Linux Foundation，走开放治理路线——这是非常明智的一步。想一想，如果每个云厂商都推一套自己的Agent互联协议，那就和互联网早期的"七国八制"一样混乱。开放标准才是Agent互联网的基础设施。

到2025年底，A2A社区已经吸引了超过200家企业和组织的参与，包括Microsoft、Salesforce、SAP、LangChain、CrewAI等关键玩家。经过近一年的迭代，A2A v1.0在2026年第二季度正式发布，成为业界首个成熟的Agent互联标准。

时间节点 事件 意义
2025年4月 Google发布A2A v0.9草案 首份Agent间通信规范草案
2025年6月 捐给Linux Foundation 走向开放治理，避免厂商锁定
2025年Q3 200+企业参与社区 涵盖云厂商、AI框架、企业软件
2025年Q4 兼容性测试轮次 多厂商互操作性验证通过
2026年Q2 A2A v1.0发布 首个成熟版本，标记GA

作为一个参与了几个月A2A社区讨论的开发者，我想说一个切身感受：A2A的设计哲学非常务实。它没有试图发明一个全新的通信范式，而是站在了现有成熟技术的基础上——JSON-RPC 2.0作为消息格式，gRPC和HTTP/2作为传输层绑定。这意味着任何有Web开发经验的团队都可以在几小时内实现一个A2A端点。

设计哲学： A2A的核心原则是"不强求统一运行时"。你的Agent可以跑在Python上，我的Agent可以跑在Node.js上，只要双方都实现了A2A协议端点，就能无缝协作。协议不关心里面的"大脑"是什么——是LLM、符号推理还是混合架构，都无所谓。

15.2 Agent Card：智能体的"名片"

在互联网世界中，两个系统要通信，首先要知道对方在哪、提供什么能力。A2A引入了Agent Card机制——每个Agent通过/.well-known/agent-card.json路径暴露自己的"名片"。

这个机制借鉴了互联网的/.well-known/标准目录（Apple的App Site Association、Let's Encrypt的ACME挑战都用它）。当你看到一个Agent的URL是https://agents.example.com/research-agent，你只要访问https://agents.example.com/research-agent/.well-known/agent-card.json就能获取它的完整能力描述。

一个典型的Agent Card长这样：

{
  "protocolVersion": "1.0",
  "agentName": "market-research-agent-v2",
  "agentDescription": "专注于市场调研与竞争分析的智能体",
  "capabilities": {
    "skills": [
      {
        "id": "competitor-analysis",
        "name": "竞品分析",
        "description": "对指定行业的Top 10竞品进行深度分析",
        "inputModes": ["text"],
        "outputModes": ["text", "json"],
        "maxDuration": "PT30M"
      },
      {
        "id": "trend-report",
        "name": "趋势报告",
        "description": "生成行业趋势月度报告",
        "inputModes": ["text", "file"],
        "outputModes": ["text", "json", "file"],
        "maxDuration": "PT1H"
      }
    ]
  },
  "endpoints": [
    {
      "url": "https://agents.example.com/research-agent/a2a",
      "protocol": "http",
      "auth": {
        "scheme": "bearer-token",
        "credentials": "required"
      }
    }
  ],
  "authSchemes": ["bearer-token", "oauth2"],
  "defaultInputModes": ["text"],
  "defaultOutputModes": ["text"]
}

这里有几个关键设计值得注意。首先是技能（Skill）的粒度——一个Agent对外暴露的不是一个笼统的"I can do research"，而是具体的、可调用的技能单元。每个技能都有明确的ID、描述、输入输出格式和最大执行时长。这就让调用方可以精准匹配需求。

其次是时长声明——maxDuration字段使用ISO 8601持续时间格式（PT30M表示30分钟）。这让调用方可以判断是同步等待还是走异步流程。如果技能声明最长30秒，调用方可以放心用同步请求；如果声明1小时，调用方应该使用异步模式。

最后是多端点支持——一个Agent可以暴露多个端点（比如HTTP和gRPC各自一个端点），调用方根据自身能力选择。这种灵活性在跨组织的异构环境中非常关键。

一个有趣的类比： Agent Card就像是智能体的"简历"。它告诉潜在的合作者：我是谁、我能做什么、我的工作方式是什么、你要怎么联系我。在多Agent经济中，一份好的Agent Card就和一份好的API文档一样重要——它决定了你能否被发现和合作。

15.3 任务生命周期状态机

A2A最核心的抽象是Task。一个Task代表一次完整的Agent间协作请求——从提交到完成（或失败/取消）的完整生命周期。A2A v1.0定义了一个清晰的状态机：

                  ┌─────────┐
                  │SUBMITTED│
                  └────┬────┘
                       │
                  ┌────▼────┐
             ┌────│ WORKING │────┐
             │    └─────────┘    │
             │                   │
      ┌──────▼──────┐    ┌──────▼──────┐
      │INPUT_REQUIRED│    │AUTH_REQUIRED│
      └──────┬──────┘    └──────┬──────┘
             │                   │
             │    ┌─────────┐    │
             └───▶│ WORKING │◄───┘
                  └────┬────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
    ┌─────▼────┐ ┌────▼────┐ ┌────▼─────┐
    │COMPLETED │ │ FAILED  │ │ CANCELED │
    └──────────┘ └─────────┘ └──────────┘
                      │
                 ┌────▼─────┐
                 │ REJECTED │
                 └──────────┘

这个状态机设计的精妙之处在于两个暂停状态：INPUT_REQUIRED和AUTH_REQUIRED。它们让A2A能够处理真实世界中那些"需要停下来等人"的场景。

举个例子。假设调研Agent在执行"竞品分析"任务时发现需要访问一个受保护的内部数据库，它就会将任务状态切换到AUTH_REQUIRED，然后等待调用方提供授权凭证。调用方获取到凭证后，再次调用tasks.send并附加认证信息，任务就会回到WORKING状态继续执行。整个过程不需要取消和重建任务——这就是暂停状态的价值。

同样地，INPUT_REQUIRED对应的是Agent在执行过程中发现缺少关键信息，需要向调用方追问的场景。这在实际应用中非常常见——Agent不可能一开始就预见到所有需要的输入。

状态 含义 可否恢复 典型触发场景
SUBMITTED 任务已提交，等待调度 - 收到tasks.send请求
WORKING 任务正在执行 自动 调度器分配资源开始处理
INPUT_REQUIRED 缺少输入，等待补充 是 Agent推理时发现必要信息缺失
AUTH_REQUIRED 需要授权凭证 是 Agent需要访问受保护资源
COMPLETED 任务成功完成 否 Agent产出最终结果
FAILED 任务执行失败 否 不可恢复的错误发生
CANCELED 被调用方取消 否 用户或调用方发出取消指令
REJECTED 被Agent拒绝 否 技能不可用、资源耗尽或策略拒绝
踩坑实录： 我们在实际集成A2A时遇到过一个典型问题——调用方没有正确处理AUTH_REQUIRED状态。我们的Agent尝试访问Salesforce数据时返回了AUTH_REQUIRED，但调用方的客户端直接把所有非COMPLETED状态统一当作"失败"处理，导致大量任务被误标记为FAILED。修复很简单：在客户端状态处理逻辑中增加对暂停状态的识别和恢复机制。但这件事说明了一个真相——A2A协议本身再完善，客户端实现不好也是白搭。

15.4 长时运行任务与异步协作

A2A的核心假设之一是：Agent间协作往往是长时间运行的。一个市场调研任务可能需要30分钟，一个趋势分析可能需要1小时，有些复杂任务甚至需要跨天执行。这意味着A2A必须支持异步协作模式，而不是简单的请求-响应。

A2A定义了两种任务执行模式：

同步模式（Short-lived Tasks）： 调用方发送tasks.send请求并保持连接，等待最终结果。这种方式适用于已知可以在短时间内完成的任务（比如几秒到几分钟）。底层使用HTTP/2的Server-Sent Events（SSE）或gRPC的流式响应，让调用方可以实时接收进度更新。

异步模式（Long-lived Tasks）： 调用方发送tasks.send请求后立即断开连接，通过两种机制获取结果：
- 轮询（Polling）： 调用方定期发送tasks.get查询任务状态
- Webhook回调（Push）： 任务完成时，Agent主动向调用方注册的回调URL推送结果

异步模式下的消息流大致如下：

调用方Agent                    服务方Agent
    │                               │
    │── tasks.send({skill, input}) ─▶│
    │   返回 taskId, status=SUBMITTED│
    │◀───────────────────────────────│
    │                               │
    │          [异步执行中...]       │
    │                               │
    │── tasks.get({taskId}) ────────▶│
    │   返回 status=WORKING, 进度30% │
    │◀───────────────────────────────│
    │                               │
    │          [更多执行...]         │
    │                               │
    │── tasks.get({taskId}) ────────▶│
    │   返回 status=COMPLETED, 结果  │
    │◀───────────────────────────────│

Webhook模式更进一步，Agent在任务提交时会包含一个callbackUrl参数：

{
  "id": "task-20260630-001",
  "skill": "competitor-analysis",
  "input": {"industry": "电动汽车", "region": "中国"},
  "callbackUrl": "https://caller-agent.example.com/callback/task-20260630-001",
  "metadata": {
    "priority": "high",
    "sourceAgent": "pipeline-orchestrator-v3"
  }
}

任务完成或状态变化时，服务方Agent会向callbackUrl发送一个POST请求，包含完整的结果或状态更新。这让编排Agent不需要持续轮询，被动接收通知即可——在大规模生产环境中，这种模式可以显著降低网络和计算开销。

最佳实践： 建议所有生产级A2A集成都同时支持轮询和Webhook两种模式。轮询作为回退方案（当Webhook不可靠时使用），Webhook作为主要通知机制。此外，建议为回调URL添加超时重试机制——服务方Agent在回调失败时应该重试3次，间隔逐渐增大。

15.5 MCP + A2A：分层协作的完整画面

现在我们有了两个协议，它们的关系可以用一个简单的分层模型来理解：

┌─────────────────────────────────────────────────────┐
│                    Agent 世界                        │
│                                                      │
│  ┌─────────────┐         ┌─────────────┐           │
│  │   Agent A   │◄──A2A──▶│   Agent B   │           │
│  │  (分析师)   │         │  (文案师)   │           │
│  └──────┬──────┘         └──────┬──────┘           │
│         │                       │                   │
│         │ MCP                   │ MCP               │
│         ▼                       ▼                   │
│  ┌─────────────┐         ┌─────────────┐           │
│  │ MCP Server  │         │ MCP Server  │           │
│  │ (数据库)    │         │ (API网关)   │           │
│  └─────────────┘         └─────────────┘           │
│                                                      │
│         ▲ 工具层 (MCP: Agent ↔ Tool)                │
│         │ 协作层 (A2A: Agent ↔ Agent)               │
└─────────────────────────────────────────────────────┘

实际系统中的工作流通常同时涉及两个协议。拿一个典型的"竞品分析报告生成"流程来说：

  • Step 1：编排Agent通过A2A向调研Agent提交"分析电动车行业"的任务
  • Step 2：调研Agent通过MCP查询内部数据库获取销售数据，通过A2A向第三方数据Agent购买行业报告片段
  • Step 3：调研Agent完成分析后通过A2A向调用方推送结果
  • Step 4：编排Agent收到结果后，通过A2A向文案Agent提交"撰写报告"的任务
  • Step 5：文案Agent通过MCP调用文档生成工具、图片生成工具来制作最终报告

这个过程中，MCP负责所有"Agent触达工具"的通信，A2A负责所有"Agent触达Agent"的通信。两者互补而不重叠——就像TCP/IP协议栈中，IP层负责路由寻址，TCP层负责可靠传输，各司其职。

关键洞察： MCP + A2A 的组合构成了 Agent 互联网的"南北向"和"东西向"通信基础设施。MCP 是南向接口（Agent向下连接工具），A2A 是东西向接口（Agent之间横向连接）。两个协议合在一起，才构成了完整的Agent网络通信栈。

15.6 A2A安全模型

跨组织的Agent通信当然离不开安全。A2A v1.0定义了多层次的认证和授权机制，让调用双方都能确认对方身份并且控制访问权限。

A2A在传输层支持以下几种认证方式：

认证方式 安全级别 适用场景 实现复杂度
API Key / Bearer Token 基础 内部Agent、信任网络 低
OAuth 2.0 中 跨组织Agent协作 中
OIDC (OpenID Connect) 高 身份联邦、企业级 中高
mTLS (双向TLS) 最高 金融、医疗等高安全场景 高

认证方式在Agent Card的authSchemes字段中声明，调用方可以根据自身能力选择双方都支持的方式。如果调用方尝试的认证方式不被服务方支持，服务方返回AUTH_REQUIRED状态并附上支持的方案列表。

此外，A2A还支持作用域（Scope）级别的授权控制。即使调用方通过了认证，也只能访问它被授权的Skill。例如，一个Agent Card暴露了三个技能，但某个调用方只被授予了其中两个的访问权限——Agent在收到第三个技能的调用请求时会直接返回REJECTED。

安全提醒： A2A的安全模型只覆盖了"传输层"的通信安全，不涉及Agent内部的安全问题。即使两个Agent通过mTLS完美认证对方，如果其中一个Agent被提示注入攻陷，它仍然可能向另一个Agent发送恶意请求。所以A2A安全必须和智能体免疫系统（第17章）结合使用——传输安全不是业务安全的全部。

15.7 A2A的局限与未来

A2A v1.0是一个优秀的开端，但它不是万能的。坦白说，当前版本有几个值得关注的局限：

第一个局限是协商能力有限。当前的A2A允许Agent间发起任务，但不支持双向协商。如果服务方Agent说"我不做这个，但可以做类似的X"，调用方目前无法优雅地处理这种"反提案"。这需要引入类似合同网协议（Contract Net Protocol）中的招标-投标-授标机制，这在A2A v1.0中还没有。

第二个局限是缺乏市场发现机制。Agent Card只解决了"名片"问题——你知道一个Agent的URL后可以查它有什么技能。但你如何发现有哪些Agent可用？A2A没有定义Agent注册中心或目录服务。这个缺口正在由社区项目（如Agent Registry、Agent Marketplace）填补，但还没有形成统一标准。

第三个局限是信用与声誉系统。当几十万个Agent在互联网上互相调用时，你如何知道哪个Agent是靠谱的？A2A目前没有内置的声誉评分或信用机制。一些研究团队正在探索基于区块链的Agent声誉系统，但这离生产还有距离。

社区动态： A2A社区已经成立了多个SIG（Special Interest Group）来推动v1.1中这些能力的落地。其中Agent Discovery SIG和Agent Reputation SIG最值得关注。如果你对Agent互联网有热情，加入社区贡献是最好的学习方式——标准还在快速演进中。

好了，本章我们深入探讨了A2A协议的完整图景——从Agent Card的发现机制、任务生命周期的状态机、异步协作模式，到MCP+A2A的分层架构和安全模型。这些内容可能有些密集，但请记住一个核心信息：A2A让Agent不再是一座孤岛。它是构建Agent互联网的关键基础设施。

有了通信协议（MCP）和协作协议（A2A），我们的基础设施层就完备了。但接下来的问题更棘手：当Agent互联网真正形成，安全威胁也随之而来。开放意味着暴露，互联意味着攻击面扩大。下一章，我们将站到攻击者的视角，审视多智能体系统面临的安全威胁——你会发现，AI时代的安全攻防完全是一场全新的游戏。下一站：多智能体安全攻防。
