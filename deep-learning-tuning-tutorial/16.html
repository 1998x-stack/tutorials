NaN——让每个深度学习工程师心跳骤停的四个字母

你一定经历过这个场景：深夜，你盯着终端窗口，loss正常下降，你正准备去睡觉——然后屏幕上突然跳出四个字母：nan。它可能出现在loss上、出现在验证集评估时、甚至出现在某个中间层的输出里。你的心跳瞬间加速，大脑开始疯狂回溯："我改了啥？刚才那个commit是不是有问题？这个实验已经跑了6个小时了……"

NaN（Not a Number）是深度学习训练中最高优先级的阻断信号。不解决NaN，后续所有调参、评估、实验对比都毫无意义。但NaN的诊断有一个关键难点：看起来都是"loss变成nan"，但不同的出现模式对应完全不同的根因——用渐变型的方法（降学习率）处理突发型（坏样本），或者反过来，都会浪费大量时间。

我把NaN的出现模式归纳为三种，每一种都有自己的"指纹"。

NaN出现的三种模式

模式一：突然出现（突发型NaN）。Loss在某一step之前一切正常，然后突然从某个值直接跳变为nan。这是最常见的模式，也是最好诊断的。根因几乎总是"坏样本 + 数值不稳定操作"的组合：某条样本包含无效值（NaN/inf）、或者交叉熵中碰到了log(0)、除以零、平方根负数。其特征是NaN出现前loss没有任何发散趋势——这个step是0.23，下一个step就是nan。

模式二：逐渐发散到NaN（渐变型NaN）。Loss先正常下降，然后开始缓慢或加速上升，最终在若干step后溢出为NaN。这是梯度爆炸的典型信号——梯度被连乘效应放大到无法控制的值（尤其在RNN、Transformer、深度ResNet中常见），参数更新过于剧烈，导致数值溢出。特征是NaN出现之前有清晰的预告——你可以看到loss曲线从平稳到慢慢抬头，然后斜率越来越大，像飞机失速一样无可挽回。

模式三：间歇性出现。Loss时不时变成NaN然后又自己恢复。这是最诡异也最容易被"忽视"的模式。它通常发生在FP16混合精度训练的边界——数值刚好踩在FP16表示范围（6.5e-5到65504）的边缘，导致部分参数溢出或下溢。这种NaN很难复现，因为每次触发的batch可能不同。如果发现间歇性NaN，优先检查混合精度配置和loss scaling策略。

重要提醒：NaN的三种模式指向完全不同的根因。不要看到NaN就直接上梯度裁剪——突发型NaN来自坏样本，裁剪无法让坏样本变好。先诊断模式，再选择方案。

系统性排查方法

面对NaN，不要慌乱，按照三步走系统排查：

第一步：用torch.isnan()逐层检查输入数据。在第一个batch加载后立即插入检测代码。很多NaN的根源非常"低级"——数据文件损坏、预处理除零、padding后mask未正确处理。先确认你的输入数据中没有NaN：

# 将这段代码插入data loading之后、模型前向之前
        def check_nan_in_data(batch):
            for key, tensor in batch.items():
                if torch.is_tensor(tensor) and tensor.is_floating_point():
                    has_nan = torch.isnan(tensor).any()
                    has_inf = torch.isinf(tensor).any()
                    if has_nan or has_inf:
                        print(f"[数据异常] {key} 包含 NaN={has_nan} Inf={has_inf}")
                        return False
            return True

第二步：计算loss前检查logits范围。交叉熵中的log(0)是最常见的NaN来源。如果你的logits在softmax后出现了0或1的极端值，log就会产生无穷大或NaN。在计算loss之前，检查logits的范围是否合理：

# 在 loss 计算前插入
        def check_logits(logits, label="logits"):
            if torch.isnan(logits).any():
                print(f"[NaN检测] {label} 包含 NaN")
            if torch.isinf(logits).any():
                print(f"[Inf检测] {label} 包含 Inf")
            if logits.numel() > 0:
                print(f"{label} 范围: [{logits.min().item():.4f}, {logits.max().item():.4f}]")

第三步：用torch.autograd.detect_anomaly()精确定位异常操作。如果前两步没有发现问题，用这个工具包裹前向和反向传播：

with torch.autograd.detect_anomaly():
            loss = model(batch)
            loss.backward()

PyTorch会在检测到NaN梯度的第一个操作处抛出异常和堆栈信息，精确告诉你"是哪个算子、哪一行代码"产生了NaN。这比你在代码里满天插入print语句高效一百倍。

最佳实践：detect_anomaly会增加2-3倍的计算开销和额外显存，千万不要默认开启。仅在排查阶段短时使用，定位问题后立刻关闭。另外，detect_anomaly和梯度裁剪不兼容——裁剪后的梯度不再反映真实回传路径，可能无法捕捉原始问题，排查时先关掉裁剪。

梯度裁剪的救急与局限

梯度裁剪（Gradient Clipping）是NaN出现后最常被"救急"使用的工具。它的原理很简单：在反向传播后、参数更新前，如果梯度的范数超过预设阈值，就将其缩放到阈值以内：

# 按范数裁剪（推荐，保留方向信息）
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

        # 按值裁剪（每个梯度分量独立截断，仅用于极端情况）
        torch.nn.utils.clip_grad_value_(model.parameters(), clip_value=0.5)

max_norm=1.0是一个保守的默认值。来自DeepSeek和LLaMA等主流训练实践的推荐——大多数时候1.0能有效拦截异常梯度而对正常更新影响很小。对于PPO强化学习，常用更激进的0.2；对于不同规模的模型，先用1.0试，观察裁剪触发比例。如果超过30%的step触发了裁剪，说明阈值太低或梯度分布本身就有问题。

但这里有一句非常重要的提醒：梯度裁剪可以"治标"让你继续训练，但可能掩盖真实的代码bug。

知乎上有位答主说得很直接："如果实在无法解决，梯度CLIP防止NaN也行，但这样可能治标不治本。"更具体的：突发型NaN来自坏样本——裁剪无法让坏样本变好，你只是屏蔽了它的影响。渐变型NaN来自过高的学习率或不良初始化——裁剪降低了有效学习率，让你以为问题修复了，但实际上你只是把学习率打了折，收敛速度大幅下降却浑然不觉。

更隐性的问题是：对于Adam优化器，裁剪不仅缩放当前step的梯度，还会影响动量估计（一阶矩和二阶矩）。频繁的裁剪让Adam的动量估计值偏离真实分布，长期使用可能导致optimizer状态退化。这就像一个一直在吃止痛药的人——疼痛消失了，但病灶还在恶化。

混合精度的数值陷阱

混合精度训练（AMP）是加速训练的利器，但它引入了一类特有的NaN问题。FP16的表示范围是6.5e-5到65504——这意味着小于6.5e-5的值会被直接"下溢"为0，大于65504的值会被"上溢"为inf。在反向传播中，梯度很容易踩到这两个边界。

FP16的小问题在于指数范围窄。它的5位指数位决定了可表示的最小正数是6.1e-5——梯度中的小值很容易消失。对比之下，BF16的指数范围和FP32完全相同（8位指数位），只是精度更低（7位尾数 vs 23位尾数），意味着它能表示的最小正数跟FP32一样（约1.2e-38），但小数精度差。这个特性让BF16在训练中远优于FP16——它几乎不会产生梯度下溢问题。

如果你的硬件不支持BF16（A100及之后的GPU支持，V100及之前的仅支持FP16），你需要使用Loss Scaling来缓解FP16的梯度下溢问题：

scaler = torch.cuda.amp.GradScaler()

        for batch in dataloader:
            with torch.cuda.amp.autocast():
                loss = model(batch)

            scaler.scale(loss).backward()
            scaler.unscale_(optimizer)
            torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
            scaler.step(optimizer)
            scaler.update()

注意顺序：先unscale_再裁剪再step。如果在梯度被缩放的状态下做裁剪，阈值需要做调整，容易出错。GradScaler会动态调整缩放因子——如果连续若干step没有出现梯度溢出，它会把scale调大；如果出现梯度溢出，它会把scale调小并跳过该step。这也是为什么在使用AMP时，间歇性NaN几乎可以忽略一部分——它们出现在scaler调整scale的阶段。

NaN排查速查表

NaN成因 检测方法 修复方案
输入数据包含NaN/Inf torch.isnan()检查每个batch的输入 修复数据管线，过滤坏样本，检查预处理
交叉熵中的log(0) 检查softmax输出是否有精确0值 在log中加epsilon（1e-8），或用稳定的CrossEntropyLoss实现
梯度爆炸（LR过大） 监控梯度范数是否持续增长 降低学习率，添加warmup，启用梯度裁剪
FP16梯度下溢 检查AMP autocast范围内的操作 使用GradScaler，或切换到BF16（硬件支持时）
除零或sqrt负数 detect_anomaly定位精确操作 在脆弱操作前加数值保护（加epsilon, clamp）
FP16前向溢出 检查模型输出是否含NaN 检查权重初始化、降低LR、检查激活函数输出范围

NaN诊断工具函数

下面是一个完整的NaN诊断工具函数，集合了上述诊断方法的精华。在模型训练脚本中插桩使用，遇到NaN时可以自动打印所有关键信息：

import torch

        def diagnose_nan(model, batch, optimizer, logger=print):
            """全链路NaN诊断工具"""
            # 1. 检查输入数据
            for name, tensor in batch.items():
                if torch.is_tensor(tensor):
                    if torch.isnan(tensor).any():
                        logger(f"[致命] 输入 {name} 含 NaN")
                    if torch.isinf(tensor).any():
                        logger(f"[致命] 输入 {name} 含 Inf")

            # 2. 带异常检测的前向+反向
            with torch.autograd.detect_anomaly():
                output = model(batch["input"] if "input" in batch else batch[0])

                if torch.isnan(output).any():
                    logger(f"[致命] 模型输出含 NaN，范围: [{output.min():.4f}, {output.max():.4f}]")

                loss = output.mean()
                loss.backward()

            # 3. 检查梯度范数
            total_norm = 0.0
            for p in model.parameters():
                if p.grad is not None:
                    total_norm += p.grad.norm().item() ** 2
            total_norm = total_norm ** 0.5
            logger(f"[诊断] 梯度范数: {total_norm:.4f}")

            if total_norm > 100:
                logger(f"[警告] 梯度范数 > 100，疑似梯度爆炸")
            elif total_norm < 1e-6:
                logger(f"[警告] 梯度范数 < 1e-6，疑似梯度消失")

            # 4. 检查参数是否含NaN
            for name, param in model.named_parameters():
                if torch.isnan(param).any():
                    logger(f"[致命] 参数 {name} 含 NaN")

            return not any(torch.isnan(p).any() for p in model.parameters())

将此函数封装在你的训练脚本中，在训练早期（头几个epoch）周期性调用，可以在NaN产生时第一时间获取完整的诊断信息。比手动翻阅日志大海捞针要高效得多。
