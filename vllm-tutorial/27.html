第27章 实战项目：搭建 RAG 推理服务

知识学了不用等于白学。这一章咱们从头搭建一个完整的 RAG 推理服务——从 vLLM 部署到向量检索到前缀缓存优化，把全书的知识点串起来。

27.1 项目目标

搭建一个智能客服 Q&A 系统：

用户问："退货流程是什么？"
    ↓
1. 向量检索: 从知识库找到相关文档
2. 构造 prompt: System Prompt + 检索到的文档 + 用户问题
3. vLLM 推理: 基于文档内容回答问题
4. 返回答案 + 引用来源

技术栈:
├── vLLM (推理引擎)
├── sentence-transformers (向量化)
├── ChromaDB (向量数据库)
├── FastAPI (API 服务)
└── Prometheus + Grafana (监控)

27.2 架构设计

                    ┌──────────────────┐
                    │  用户 (浏览器/App) │
                    └────────┬─────────┘
                             │
                    ┌────────▼─────────┐
                    │   FastAPI        │
                    │   (API Gateway)  │
                    └────────┬─────────┘
               ┌─────────────┼─────────────┐
               ▼             ▼             ▼
        ┌───────────┐ ┌───────────┐ ┌──────────┐
        │ Embedding  │ │ ChromaDB  │ │  vLLM    │
        │ Service    │ │ (向量库)   │ │ (推理)   │
        └───────────┘ └───────────┘ └──────────┘

关键设计:
1. System Prompt 固定 → Prefix Cache 高命中率
2. 检索到的文档拼在 System Prompt 后面 → 动态部分很短
3. 用相同的 System Prompt 模板 → 所有请求共享前缀

27.3 核心代码

# rag_service.py
from vllm import LLM, SamplingParams
from sentence_transformers import SentenceTransformer
import chromadb

class RAGService:
    def __init__(self):
        # vLLM 推理引擎（启用 prefix caching）
        self.llm = LLM(
            model="Qwen/Qwen2.5-7B-Instruct",
            gpu_memory_utilization=0.95,
            enable_prefix_caching=True,   # ← 关键！
            kv_cache_dtype="fp8"
        )
        self.embedder = SentenceTransformer("BAAI/bge-small-zh")
        self.vector_db = chromadb.PersistentClient("./kb_store")

        self.system_prompt = """你是智能客服助手。请基于以下参考文档回答用户问题。
如果文档中没有相关信息，请如实说不知道。

参考文档:
{context}

请用中文回答，语气友好专业。"""

    def retrieve(self, query, top_k=3):
        query_vec = self.embedder.encode(query)
        results = self.vector_db.query(query_vec, n_results=top_k)
        return "\n\n".join(results["documents"][0])

    def ask(self, question):
        # 1. 检索
        context = self.retrieve(question)

        # 2. 构造 prompt
        prompt = self.system_prompt.format(context=context)
        full_prompt = f"{prompt}\n\n用户问题: {question}\n回答:"

        # 3. vLLM 推理（prefix cache 自动命中 system_prompt 部分）
        sampling_params = SamplingParams(
            temperature=0.3, max_tokens=512, stop=["用户问题:", "\n\n\n"]
        )
        outputs = self.llm.generate([full_prompt], sampling_params)

        return {
            "answer": outputs[0].outputs[0].text,
            "sources": context
        }

27.4 性能优化路径

步骤优化预期效果
1 (基线)基础 RAG + vLLM 默认配置~800 tok/s 吞吐
2启用 Prefix Caching~1400 tok/s (+75%)
3FP8 KV Cache~1900 tok/s (+36%)
4优化 System Prompt 长度~2200 tok/s (+16%)
5加 n-gram 推测解码~2800 tok/s (+27%)

这个项目的核心洞察：RAG 场景是 Prefix Caching 的最佳应用——system prompt + 检索文档在每个请求中占了 80%+ 的 prompt tokens。开启 caching 后，vLLM 只算用户问题那 20%。这就是为什么从 800 → 1400 tok/s 的跃升如此显著。

27.5 拓展方向

  • 多轮对话：加入对话历史，用 LMCache 缓存历史 KV
  • 混合路由：简单问题直接答，复杂问题走 RAG → 用语义路由分发
  • 流式输出：FastAPI + SSE，stream=True
  • 多模型部署：小模型处理简单 FAQ，大模型处理复杂推理
