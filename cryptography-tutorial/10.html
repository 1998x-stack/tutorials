第10章 椭圆曲线密码学

同学们，上一章学完RSA后，你可能会有一个疑惑：既然RSA能用，为什么还需要椭圆曲线密码学（Elliptic Curve Cryptography, ECC）？答案是两个字：效率。

回想一下RSA-129的故事——一个425位的RSA模数在1994年被1600台计算机花了8个月破解。而今天的RSA推荐长度是3072位。这意味着什么？意味着RSA的密钥越来越长，运算越来越慢。随着移动设备、物联网（IoT）和嵌入式系统的普及，我们需要一种能在资源受限设备上运行的公钥密码。ECC就是在这样的需求下登场的。

🔑 核心数据：256位的ECC密钥提供的安全性等同于3072位的RSA密钥。也就是说，ECC用不到RSA十分之一的密钥长度，达到了相同的安全等级。不仅更小，而且更快——在移动设备上，ECC的密钥生成速度比RSA快数十倍。

10.1 椭圆曲线是什么？

不要被名字吓到。虽然椭圆曲线在数学中有复杂的背景，但在密码学的语境下，你只需要理解一个概念：椭圆曲线是一个可以在上面"做加法"的点的集合。

密码学中使用的是定义在有限域上的椭圆曲线，其方程形式为：

y² = x³ + ax + b  (mod p)

其中 p 是一个大素数，a 和 b 是系数，满足 4a³ + 27b² ≠ 0

曲线上的点 (x, y) 加上一个特殊的"无穷远点"构成一个群。
在这个群上，我们可以定义"点加法"运算。

点加法的几何意义非常直观：在曲线上取两个不同的点P和Q，画一条直线穿过它们，直线与曲线交于第三个点R'，然后取R'关于x轴的对称点R，这就是P+Q的结果。如果P和Q是同一个点，那就画切线——这叫"点加倍"。

椭圆曲线密码学的安全性依赖于椭圆曲线离散对数问题（ECDLP）：给定曲线上两个点G和Q = k×G（即G自加k次），求k。对于精心选择的曲线参数，这个问题没有比完全穷举更有效的解法。

10.2 点乘运算

ECC的核心运算叫作标量乘法：Q = k × G，其中k是一个大整数（私钥），G是曲线上的一个基点（公开参数），Q是结果点（公钥）。

这个运算乍看起来需要做k次加法——如果k是个256位数，那得加到宇宙毁灭。好在我们可以用"加倍-累加"法（类似快速幂）：

要计算 k × G，将 k 表示为二进制：
k = 13 = 1101₂

算法：
result = O（无穷远点）
遍历 k 的二进制位从高位到低位：
    result = 2 × result  （点加倍）
    如果当前位是1：
        result = result + G  （点加法）

计算过程：
1. result = O，位1 → result = O + G = G
2. 加倍 → result = 2G，位1 → result = 2G + G = 3G
3. 加倍 → result = 6G，位0 → 跳过加法
4. 加倍 → result = 12G，位1 → result = 12G + G = 13G ✓

这样只需要约log₂(k)次加倍和约log₂(k)/2次加法——对于256位的k，只需要几百次运算，瞬间完成。

10.3 ECDH密钥交换

ECC版本的Diffie-Hellman密钥交换称为ECDH。它的原理与DH一样，但使用椭圆曲线运算：

公开参数：椭圆曲线E，基点G（阶为一个大素数n）

Alice生成私钥 dA，计算公钥 QA = dA × G
Bob生成私钥 dB，计算公钥 QB = dB × G

Alice计算共享密钥：S = dA × QB = dA × (dB × G)
Bob计算共享密钥：S = dB × QA = dB × (dA × G)

双方得到同样的S！攻击者只知道QA和QB，但无法计算出dA或dB。

ECDH被广泛应用于TLS 1.3中，是当前互联网主流密钥交换方案。值得注意的是，在实际部署中ECDH通常使用临时密钥（即每次连接生成新的密钥对），以实现前向安全性（Forward Secrecy）——即使服务器的长期私钥泄露，过去的会话也不能被解密。

10.4 ECDSA签名

ECDSA是ECC版本的DSA签名算法。签名过程需要一对密钥和一个随机数k（这个k非常关键！）：

签名（使用私钥 d）：
1. 随机选择 k ∈ [1, n-1]（绝对不能重复！）
2. 计算点 R = k × G，取 R 的x坐标作为 r
3. 计算 s = k⁻¹ × (H(m) + d × r) mod n
4. 输出签名 (r, s)

验证（使用公钥 Q）：
1. 计算 u1 = H(m) × s⁻¹ mod n
2. 计算 u2 = r × s⁻¹ mod n
3. 计算点 P = u1 × G + u2 × Q
4. 如果 P 的x坐标等于 r，接受签名

看到那个"随机选择k"了吗？这是ECDSA安全中最致命的一环——如果k值重复或可预测，私钥将直接暴露。

⚠️ 踩坑故事——索尼PlayStation 3的悲惨教训：2010年，黑客团队fail0verflow宣布成功获取了索尼PS3的ECDSA私钥，这意味着他们可以签署任何未授权的游戏和软件在PS3上运行。漏洞出在哪里？索尼在ECDSA签名中使用了固定的k值（k = 4）。这意味着签名的(r, s)中的r值完全相同——每款游戏的签名都有相同的r。黑客只需要拿到两个不同的签名，就可以通过简单的数学计算恢复出私钥。这个漏洞让索尼的整个数字版权管理系统形同虚设。索尼的工程师显然知道ECDSA需要一个随机k——但他们选择了"省事"的固定值，结果付出了沉重的代价。

10.5 Curve25519与EdDSA

2006年，著名密码学家Daniel J. Bernstein设计了一个新的椭圆曲线——Curve25519（也称为X25519）。它解决了很多困扰NIST曲线的问题：

特性 NIST P-256 Curve25519
设计哲学 商业闭源，NIST标准 完全开源，公开评审
曲线参数 由SHA-1种子生成（有争议） Nothing-up-my-sleeve设计
实现难度 高，容易引入侧信道漏洞 低，常数时间实现简单
性能 快 更快
密钥格式 DER/PEM，带ASN.1元数据 32字节原始格式

与Curve25519配套的签名算法是EdDSA（Edwards-curve Digital Signature Algorithm），特别是它的实例化Ed25519。与ECDSA不同，EdDSA是确定性的——它不依赖随机数生成器，而是从私钥和消息中派生出一个伪随机的nonce。这彻底避免了索尼PS3那种k值重复的灾难。

💡 为什么EdDSA更安全？在ECDSA中，如果你没有好的随机数生成器（比如在嵌入式设备上），k值的质量就无法保证。而EdDSA完全消除了对随机数的依赖——它用哈希函数从私钥和消息中派生出k，确保每次签名都有唯一的、不可预测的k值。这是一个"默认安全"的设计哲学，避免了最常见的ECDSA实现错误。

10.6 踩坑故事：NSA后门争议

椭圆曲线密码学历史上最持续的争议，是围绕NIST P-256（即secp256r1）曲线的参数来源问题。

NIST P-256的曲线参数（包括系数a、b、基点G等）是通过一个"随机"种子生成的——据说这个种子来自SHA-1输出的某个值。但问题是：没有人能验证这个种子是否真的随机。如果NSA知道这个种子对应某种特殊属性，使得它们能更快地计算离散对数，那么NIST P-256就可能存在一个只有NSA知道的"后门"。

2007年，密码学家Dan Bernstein和Tanja Lange等人公开质疑了NIST曲线的参数来源。尽管没有证据表明NIST曲线真的有后门，但这种"信任但无法验证"的状态让很多人感到不安。相比之下，Bernstein设计的Curve25519选择了完全透明的参数生成方式：曲线的系数是通过"Nothing-up-my-sleeve"原则选择的——使用数字π的小数部分或简单的自然常数作为参数来源，任何人都可以复现。

🔍 技术细节：所谓Nothing-up-my-sleeve（NUMS）是指参数的选择方式公开透明，没有隐藏信息。例如，Curve25519的模数p = 2^255 - 19——这个数本身就是特殊的（Mersenne数的变形），没有人为操纵的空间。NIST曲线的参数则是通过SHA-1对一个"不明来源"的种子哈希得到的。虽然理论上这也没问题，但密码学的核心原则之一是"不要信任，要验证"（Don't trust, verify）。

2013年斯诺登曝光的文档显示，NSA确实在推动使用含有潜在后门的密码标准（如Dual_EC_DRBG随机数生成器事件）。虽然这与NIST曲线没有直接关系，但给NIST曲线的信任度带来了进一步的负面影响。如今，Curve25519/Ed25519已经成为许多现代系统（包括SSH、TLS 1.3、Signal、WhatsApp）的首选曲线。

理解完ECC的核心原理，下一章我们将进入数字签名的世界。签名是公钥密码学中与加密同等重要的应用——它解决了"谁发的消息"和"消息被改了吗"这两个根本问题。更重要的是，ECDSA的故事（尤其是索尼的悲剧）告诉我们：签名算法的实现细节比算法本身更容易出问题。
