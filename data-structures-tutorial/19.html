开篇："朋友圈"里的连通性难题

假设你正在开发一个社交平台——就叫它"FriendLink"吧。上线第一周用户注册量突破了10万，产品经理跑来找你提了一个需求：

"我们要在用户主页展示'你们之间有多少个共同好友'，还要判断两个用户是否属于同一个社交圈——比如A和B是好友，B和C是好友，那么A和C应该被视为同一个圈子里的人。"

你第一反应是：这不就是图论里的连通性问题吗？用DFS或BFS遍历一下用户的好友图不就完了？确实可以。但当用户量从10万涨到1000万的时候，每次查询都做一次DFS遍历——最坏情况下要遍历整个图——这个代价太大了。你需要一种能够动态地维护连通关系、并且能在近似常数时间内回答"两个节点是否连通"的数据结构。

这就是并查集（Union-Find Disjoint Set）登场的时刻。

并查集的由来

并查集，英文叫 Disjoint Set Union（DSU）或 Union-Find，是一种用来管理不相交集合的数据结构。它解决的核心问题是动态连通性——在数据不断变化（添加新的连接关系）的过程中，高效地判断两个元素是否属于同一个集合。

回到"FriendLink"的例子：每一个用户初始时都是独立的——自己就是一个社交圈。当两个用户加好友时，我们把他们所在的社交圈"合并"起来。当需要判断两个用户是否在同一个社交圈时，我们只需要"查找"他们各自属于哪个圈子，然后比较一下圈子编号是否相同。这就是并查集最朴素的直觉：合并（Union）+ 查找（Find）。

并查集在计算机科学中有着广泛的应用：社交网络的好友推荐、图像处理中的连通区域标记、Kruskal最小生成树算法、游戏中的地图连通性判断——凡是需要管理"等价关系"的场景，都能看到并查集的身影。

核心直觉：并查集本质上是一种"找老大"的结构。每个集合都有一个"老大"（根节点），你问一个人属于哪个帮派，他就帮你找到老大。合并两个帮派，就让一个老大认另一个老大做老大。就这么简单。

并查集的基本操作

并查集的核心操作只有两个：查找（Find）和合并（Union）。

在最基础的实现中，我们用数组来存储每个元素的"父节点"：

# 最基础的并查集实现
class UnionFind:
    def __init__(self, n):
        # 初始时，每个元素都是自己的父节点（自成一派）
        self.parent = list(range(n))

    def find(self, x):
        # 查找x所属集合的根节点
        while self.parent[x] != x:   # 不断向上找父节点
            x = self.parent[x]
        return x

    def union(self, x, y):
        # 合并x和y所属的两个集合
        root_x = self.find(x)
        root_y = self.find(y)
        if root_x != root_y:
            self.parent[root_x] = root_y  # 将一个根挂到另一个根下

初始时，每个元素都是自己的"老大"（parent[i] = i）。find操作沿着父节点指针一路往上走，直到找到根节点（根节点的父节点是自己）。union操作就是先找到两个元素的根，如果根不同，就把一个根挂到另一个根下面。

这个实现有一个严重的问题：当合并深度很大时，find操作可能退化为O(n)！想象一下：每次都把root_x挂到root_y下面，时间长了就形成了一条长长的链——查找最下面的元素需要遍历整条链。

路径压缩

如何解决上面说的"长链"问题？答案非常巧妙：在find的过程中，顺手把路径上的所有节点都直接指向根节点。

def find(self, x):
    if self.parent[x] != x:
        # 递归地找到根，并将当前节点直接指向根
        self.parent[x] = self.find(self.parent[x])
    return self.parent[x]

这个优化叫做路径压缩（Path Compression）。它的效果非常惊人：经过一次find之后，从x到根路径上的所有节点，下次再find就是O(1)了。多次操作之后，整个树的形状会变得非常扁平，几乎所有的节点都直接挂在根节点下面。

我还记得第一次在Kruskal算法里用上路径压缩时的震撼：之前跑一个100万个节点的最小生成树需要好几秒，加上路径压缩后瞬间出结果——"O(n)到O(α(n))的飞跃"就是这么直观。

小技巧：路径压缩有两种写法——递归和迭代。递归写法简洁易懂，但如果路径非常长（比如深度>1000），Python的递归深度限制可能导致RecursionError。生产环境建议用迭代写法：while parent[x] != x: parent[x] = parent[parent[x]]; x = parent[x]。这叫"两步一跳"（halving），效果和完整路径压缩几乎一样。

按秩合并

路径压缩优化的是find操作本身，但还有一个问题：谁合并谁？我们之前的union是随意地把一个根挂到另一个根下面，这可能导致树的高度增加。有没有更好的策略？

答案是按秩合并（Union by Rank）。这里的"秩"通常指树的高度（或者节点数量）。合并时，总是把矮的树挂到高的树下面，这样可以避免树的高度增加。

class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [1] * n   # 记录每个根节点的树高（近似）

    def find(self, x):
        # 路径压缩
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]

    def union(self, x, y):
        root_x = self.find(x)
        root_y = self.find(y)
        if root_x == root_y:
            return

        # 按秩合并：将矮树挂到高树下
        if self.rank[root_x] < self.rank[root_y]:
            self.parent[root_x] = root_y
        elif self.rank[root_x] > self.rank[root_y]:
            self.parent[root_y] = root_x
        else:
            # 两个树一样高，合并后高度+1
            self.parent[root_y] = root_x
            self.rank[root_x] += 1

当路径压缩和按秩合并同时使用时，并查集的单次操作复杂度是O(α(n))，其中α(n)是反阿克曼函数（Inverse Ackermann Function）。阿克曼函数是一个增长极其迅速的数学函数，它的反函数增长极其缓慢——对于宇宙中所有原子数量级别的输入，α(n)都不会超过5。所以你可以近似地认为并查集的操作是常数时间的。

并查集是解决动态连通性问题的最优数据结构，两个优化加在一起复杂度接近常数。 这是算法设计"从坏到好"的完美案例：朴素实现O(n) → 路径压缩近似O(log n) → 两者结合O(α(n))。

经典题目

题目一：省份数量（LeetCode 547）

有n个城市，其中一些彼此相连，另一些没有相连。给你一个n×n的矩阵isConnected，其中isConnected[i][j] = 1表示第i个城市和第j个城市直接相连。返回省份数量（直接或间接相连的城市算同一个省份）。

这个问题的本质就是求连通分量个数。用DFS可以解，但并查集更加直观：

def find_circle_num(isConnected):
    n = len(isConnected)
    uf = UnionFind(n)

    # 遍历矩阵的上三角部分
    for i in range(n):
        for j in range(i + 1, n):
            if isConnected[i][j] == 1:
                uf.union(i, j)

    # 统计不同的根节点数量
    roots = set()
    for i in range(n):
        roots.add(uf.find(i))
    return len(roots)

题目二：岛屿数量（LeetCode 200）

给你一个由'1'（陆地）和'0'（水）组成的二维网格，请计算网格中岛屿的数量。岛屿总是被水包围，并且每座岛屿只能由水平方向和/或竖直方向上相邻的陆地连接形成。

这道题的经典解法是DFS/BFS，但并查集同样可解——把每个为'1'的格子初始化为一个集合，然后向右和向下检查相邻格子，如果是'1'就合并。

def num_islands(grid):
    if not grid:
        return 0
    rows, cols = len(grid), len(grid[0])
    uf = UnionFind(rows * cols)

    def index(r, c):
        return r * cols + c

    for r in range(rows):
        for c in range(cols):
            if grid[r][c] == '1':
                # 检查右边和下边
                if r + 1 < rows and grid[r+1][c] == '1':
                    uf.union(index(r, c), index(r+1, c))
                if c + 1 < cols and grid[r][c+1] == '1':
                    uf.union(index(r, c), index(r, c+1))

    # 统计所有为'1'且是根节点的个数
    roots = set()
    for r in range(rows):
        for c in range(cols):
            if grid[r][c] == '1':
                roots.add(uf.find(index(r, c)))
    return len(roots)

题目三：冗余连接（LeetCode 684）

树可以看成是无向无环图。现在往一棵树里多加了一条边，导致出现了环。请你找出这条多余的边。

这道题是并查集的"杀手锏"——它完美地利用了并查集检测环的能力：在遍历边的过程中，如果发现两个节点已经连通（属于同一个集合），说明当前这条边就是多余的。

def find_redundant_connection(edges):
    n = len(edges)
    uf = UnionFind(n + 1)  # 节点编号从1开始

    for u, v in edges:
        if uf.find(u) == uf.find(v):
            # 已经连通，说明这条边是多余的
            return [u, v]
        uf.union(u, v)
    return []
踩坑实录：之前用DFS处理冗余连接问题时，每次加一条边都要重新DFS整个图来检测环，时间复杂度O(n²)。线上数据量一大就超时。换成并查集后，每次操作O(α(n))，检测200万条边不到0.1秒。从O(n²)到O(n·α(n))，这就是数据结构的价值。

并查集 vs DFS/BFS

维度 并查集 DFS / BFS
核心能力 动态连通性（随时加边，随时查询） 静态连通性（给定图，一次遍历）
单次查询复杂度 O(α(n)) ≈ O(1) O(n + m)（每次都要重新遍历）
空间复杂度 O(n) O(n + m)（需要建图）
实现难度 极低（20行代码） 中等（需要递归/栈）
适用场景 动态加边 + 频繁查询 静态图的一次性遍历
能否检测环 能（union时发现已在同一集合） 能（遇到已访问节点）

并查集的工程实践

在真实工程中，并查集往往不是单独使用的，而是作为某个更大算法的一部分：

  • Kruskal最小生成树：按边权重排序，用并查集判断是否会成环，是经典"贪心+并查集"的组合。
  • 图像连通区域标记：图像处理中的Connected Component Labeling（CCL），两步法中的第一步用并查集合并等价标签。
  • 社交网络分析：动态维护用户之间的连通关系，判断"你可能认识的人"。
  • 在线编译器的类型推导：某些类型系统的实现用并查集管理"类型是否等价"的关系。
小心并查集的"离线"陷阱：并查集不支持"断开"操作（无法删除已建立的连接）。如果你的场景需要频繁断开连接，并查集就不适用——你需要考虑动态图（Dynamic Graph）或Link-Cut Tree等更高级的数据结构。

本章小结

在这一章中，我们学习了：

  • 并查集是什么——一种管理动态连通关系的数据结构，核心操作是find和union。
  • 路径压缩——find时将路径上的节点直接指向根，将复杂度从O(n)降到近似O(log n)。
  • 按秩合并——union时将矮树挂到高树下，两者结合达到O(α(n))的惊人效率。
  • 经典题目——省份数量（连通分量计数）、岛屿数量（二维连通性）、冗余连接（环检测）。
  • 并查集 vs DFS/BFS——并查集的核心优势在于"动态"场景：随时加边、随时查询，每次操作都是常数时间。

并查集是一种"小但美"的数据结构。它的思想简单到可以用一句话概括——"找老大，合并帮派"——但两个精妙的优化却让它达到了理论最优的复杂度。这种"用简单的思路加上巧妙优化达到极致性能"的设计哲学，是整个数据结构和算法学习的精髓。

下一章，我们将进入另一个强大的高级数据结构——线段树与树状数组。如果说并查集解决的是"连通性"问题，那么线段树解决的是"区间查询"问题：如何快速求出一个数组中任意区间的和、最大值、最小值，同时还能支持对数组的实时更新。这是一个在算法竞赛和工程实践中都极其常用的工具，敬请期待！
