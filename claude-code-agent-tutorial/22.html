第22章 Harness 设计模式对比

大家好，我是你们的讲师。经过前面 21 章的学习，我们几乎把 Agent 的每一个组件都自己实现了一遍——从最基础的 Agent Loop 到复杂的 MCP 集成，从上下文管理到 Skills 系统。现在是时候把目光从"组件"转向"模式"了。因为决定一个 Agent 系统能否在真实世界中稳定运行的，不是它有多少功能，而是它的完成模式（Finish Pattern）设计。

我第一次意识到完成模式的重要性，是在一个深夜的自动化测试中。我写了一个 Agent 来自动修复代码 lint 错误——听起来很简单对吧？Agent 运行了一个小时，修复了 30 个文件，然后"说"它完成了。但我检查日志时发现：它在第 15 个文件之后就再也没有实际调用过任何工具，剩下的 45 分钟它一直在用自然语言"分析"代码，然后"总结"已经完成的工作。它以为自己在工作，其实只是在写总结报告。从那以后，我就再也不敢用简单的"模型说完成就算完成"的策略了。

Harness 的三次进化

Harness 的完成模式设计经历了三次明显的进化，每一次都是对前一代局限性的回应：

Gen 1：Implicit Finish（Claude Code 风格，2024-2025）

Implicit Finish 是 Claude Code 最早采用的模式。它的核心假设是：模型知道它什么时候做完了。Harness 不主动判断完成状态，而是让模型通过"不调用工具 + 给出总结性回复"来隐式地表示完成。这种模式的洞察在于：减少 Harness 对模型思考过程的干预，让模型自由发挥。

但 Implicit Finish 有一个先天缺陷：它高度依赖模型的行为一致性。Claude 模型被训练成"直接行动，少废话"，所以当它不调用工具并且开始总结时，大概率是真做完了。但其他模型——特别是那些被训练成"先叙述再行动"的模型——会在还没做任何事之前就开始"叙述计划"，然后 Harness 误判为完成了。

Gen 2：Adaptive Finish（2025，模式感知）

Adaptive Finish 引入了模式检测。Harness 不再完全依赖模型的自我声明，而是分析模型的输出模式——它是在"叙述计划"还是"总结结果"？它说"Let me"之后真的行动了吗？Adaptive Finish 的核心能力是"轻推一下"（nudge）：当检测到模型陷入叙述模式而不行动时，附加一条"请继续执行操作"的消息。

Gen 3：Meta-Harness（2026，自优化）

Meta-Harness 是目前最前沿的模式，由 Lee et al. 在 2026 年的论文中提出。它的核心想法很激进：让 AI 自己优化自己的 Harness 参数。通过一个 propose -> score -> Pareto-keep 的循环，Meta-Harness 可以在不修改模型权重的情况下，仅靠优化 Harness 参数就获得 7.7 个准确率点的提升。

Implicit Finish vs Adaptive Finish vs Explicit Finish

Arize AI 在 2026 年 5 月发布了一份重要的对比实验数据，测试这三种完成模式在标准 coding benchmark 上的表现：

完成模式 模型 平均 Token 消耗 假完成数 任务正确率
Implicit Finish Sonnet 4 22.0K 0 33.3%
Adaptive Finish Sonnet 4 23.4K 2 (caught) 44.4%
Explicit Finish Sonnet 4 24.7K 0 44.4%
Implicit Finish GPT-4o 28.1K 5 22.2%
Adaptive Finish GPT-4o 26.5K 1 (caught) 38.9%
Explicit Finish GPT-4o 25.8K 0 33.3%
Implicit Finish Gemma 4 30.2K 8 16.7%
Adaptive Finish Gemma 4 27.1K 3 (caught) 33.3%
Explicit Finish Gemma 4 26.3K 0 27.8%

从数据中可以读出几个关键信息：

  • Implicit Finish 对 Claude 工作良好，但对其他模型惨不忍睹。Sonnet 4 上 0 假完成，但 Gemma 4 上出现了 8 次假完成。
  • Adaptive Finish 是唯一在所有模型上都提升正确率的模式。它用极小的 token 开销（1-2K）换来了 10-15 个百分点的正确率提升。
  • Explicit Finish 最安全但也最死板。零假完成，但任务正确率受限于模型自身的判断能力。

为什么 Implicit Finish 是"Claude 形状的假设"

Implicit Finish 之所以对 Claude 有效、对其他模型无效，根源在于模型的训练方式不同。

Anthropic 在训练 Claude 时，有一个明确的目标：直接行动，少做叙述。Claude 被训练成——当它知道该做什么的时候，它会直接调用工具，而不是长篇大论地描述"我打算做什么"。这种"行动导向"的训练使得 Claude 的"不调用工具 + 总结"行为模式高度可信。

但 GPT 系列模型的历史训练数据中，有大量"叙述然后行动"的模式——模型会说"I'll search for X now"，然后再调用搜索工具。当模型说出"I'll search"但还没实际搜索时，它并不是"完成了"——它只是在"预告"。这个"预告"阶段的长度从几行到几百行不等，但 Implicit Finish 的 Harness 无法区分"预告"和"总结"。

值得注意的趋势是：这个差距正在缩小。2026 年的 GPT-5 和 Claude 4.5 在行为模式上越来越趋同——AI 供应商都意识到"直接行动"比"叙述行动"更高效。但差距缩小的速度很慢，因为训练数据的分布不是一天能改变的。

设计的启示非常明确：Harness 必须模型感知（model-aware）。你的 Harness 在为单一模型优化时表现很好，不代表它在多模型场景下同样有效。如果你计划支持多个模型，一定要做跨模型的完成模式测试。

Adaptive Harness 的实现原理

Adaptive Harness 的核心是一个"轻推检测器"（Nudge Detector），它分析模型的输出文本，判断模型是否需要"推一把"：

import re
from typing import Optional


class NudgeDetector:
    """Detects narrate-then-act patterns and generates nudges."""

    # Future-tense markers that often precede false finishes
    NARRATION_PATTERNS = [
        r"\bI'?ll\b", r"\bLet me\b", r"\bI will\b",
        r"\bI['']?m going to\b", r"\b接下来\b", r"\b让我\b",
    ]

    # Patterns that indicate genuine completion (summary, not plan)
    COMPLETION_PATTERNS = [
        r"\b(?:完成|done|finished|complete)\w*\s*[.。!！]?\s*$",
        r"\b(?:fixed|resolved|updated|created)\s+\d+",
    ]

    def __init__(self):
        self._narration = [
            re.compile(p, re.IGNORECASE) for p in self.NARRATION_PATTERNS
        ]
        self._completion = [
            re.compile(p, re.IGNORECASE) for p in self.COMPLETION_PATTERNS
        ]

    def needs_nudge(self, assistant_text: str,
                    has_tool_call: bool,
                    recent_tool_calls: bool = True) -> bool:
        """Determine if the assistant needs a nudge to proceed."""
        # If already acting, no nudge needed
        if has_tool_call:
            return False

        # If recently active, be more lenient
        if not recent_tool_calls:
            return False

        # Check for narration patterns
        for pattern in self._narration:
            if pattern.search(assistant_text):
                return True

        return False

    def is_completion(self, assistant_text: str) -> bool:
        """Check if the assistant is summarizing completion."""
        for pattern in self._completion:
            if pattern.search(assistant_text):
                return True
        return False

    def generate_nudge(self, model: str = "default") -> str:
        """Generate a context-appropriate nudge."""
        if model.startswith("claude"):
            return "Please proceed with the action."
        else:
            return "Please proceed with executing the next step."


class AdaptiveHarness:
    """Agent harness with adaptive finish detection and nudging."""

    def __init__(self, model: str, max_nudges: int = 3):
        self.model = model
        self.detector = NudgeDetector()
        self.max_nudges = max_nudges
        self.nudge_count = 0

    async def step(self, messages: list) -> dict:
        """Execute one step of the agent loop with nudge support."""
        response = await self._call_model(messages)
        text = response.get("content", "")
        has_tool_call = bool(response.get("tool_calls"))

        # Check if this is a false finish needing a nudge
        if self.detector.needs_nudge(text, has_tool_call):
            if self.nudge_count < self.max_nudges:
                nudge = self.detector.generate_nudge(self.model)
                messages.append({"role": "user", "content": nudge})
                self.nudge_count += 1
                # Retry with nudge
                response = await self._call_model(messages)
                self.nudge_count = 0  # reset after successful nudge
                return response

        # Check for genuine completion
        if not has_tool_call and self.detector.is_completion(text):
            return {"status": "complete", "response": response}

        return response
跨模型测试建议：即使你只计划部署一个模型，也强烈建议用 2-3 个不同模型的输出模式来测试你的 Harness。为什么？因为模型会更新。你今天用的 Claude Sonnet 4 的行为模式，和一年后的 Claude Sonnet 5 可能完全不同。如果你只在单一模型上测试过 Harness，模型更新后你的 Agent 可能突然开始"假完成"或"拒绝停止"。在多模型上做过测试的 Harness 对模型行为的鲁棒性更强。

Meta-Harness：让 AI 优化自己的 Harness

2026 年，Lee et al. 在 arXiv 上发表了论文 "Self-Optimizing Agent Harnesses"（arXiv 2603.28052），提出了 Meta-Harness 的概念。这篇论文的核心发现是：Harness 参数的优化空间远大于大多数人意识到的。

传统的 Harness 调优靠手动——开发者根据自己的经验设置 max_nudges、cache breakpoint 位置、compression threshold 等参数。Meta-Harness 将这个过程自动化了：

  1. Propose：从一个基础 Harness 配置开始，随机扰动生成一组候选配置。
  2. Score：在验证集上运行每个候选配置，收集准确率、token 消耗、延迟等指标。
  3. Pareto-keep：保留 Pareto 最优的配置（即没有其他配置在准确率和效率上都优于它）。
  4. Repeat：在保留下来的配置附近继续采样，直到收敛。

实验结果令人印象深刻：在保持模型不变的情况下，仅通过优化 Harness 参数，Meta-Harness 获得了 +7.7 准确率点的提升，同时上下文 token 消耗减少了约 4 倍。论文作者开源了一个 75 行的参考实现（名为 harness-forge），展示了如何用不到 100 行代码实现 Meta-Harness 的核心逻辑。

Meta-Harness 告诉我们一个深刻的道理：不要轻易责怪模型表现不好——也许你的 Harness 才是瓶颈。与其等待下一代模型，不如花时间优化你的 Harness 参数。这可能能带来比模型升级更大的收益。

你应该选哪种模式

面对三种完成模式，如何选择？这里提供一个决策框架：

场景 推荐模式 理由
仅用 Claude 模型 Implicit Finish Claude 的行为一致性高，Implicit Finish 在 token 效率上最优
多模型切换 / 开源模型 Adaptive Finish 跨模型鲁棒性最好，nudge 机制弥补了模型行为差异
生产环境 + SLA 要求 Explicit Finish 零假完成，任务可预期，适合需要严格审计的场景
前沿研究 / 高正确率目标 Meta-Harness 自动优化，不需要手动调参，上限最高
成本极度敏感 Implicit Finish 最低的 token 消耗，但需要配合缓存优化

从我个人的经验和社区反馈来看，Adaptive Finish 是目前最推荐的默认选择。它在 token 效率（只比 Implicit 多花 1-2K tokens）、鲁棒性（跨模型表现稳定）和实现复杂度（核心逻辑约 60 行代码）之间取得了最好的平衡。Meta-Harness 虽然上限更高，但需要额外的验证基础设施来运行参数优化循环，对于大部分团队来说可能有些过度设计。

设计模式选定了，但上线之后真正的挑战才开始——Agent 出错了怎么排查？日志怎么组织才能快速定位问题？在下一章中，我们将讨论一个所有人都需要但很少人认真对待的话题：Agent 调试与可观测性。
