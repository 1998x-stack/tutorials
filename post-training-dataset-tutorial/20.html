第20章 GRPO/RLVR推理增强数据

大家好，欢迎回到课程。在上一章，我们学习了思维链（CoT）数据——通过让模型在 SFT 阶段学会输出逐步推理过程来提升推理能力。但只靠 SFT 还不够：模型虽然学会了"如何写出推理步骤"，但并没有学会"如何判断推理步骤的对错"，更没有学会"如何通过自我纠错来改进推理"。

这一章要解决的就是这个问题。我们要介绍的 GRPO 和 RLVR，是 2025-2026 年推理增强领域最大的技术突破——它们彻底改变了"后训练数据"的定义方式。

2025-2026 最大变革：人类标注不再必需——可验证奖励的崛起

在进入技术细节之前，让我们先退一步，看清这个范式转变的全貌。

传统 RLHF（人类反馈强化学习）的流程是：

1. 模型生成多个回答
2. 人类标注员排序这些回答（哪个更好、哪个更差）
3. 用排序数据训练一个 Reward Model（奖励模型）
4. 用 PPO 算法，让模型根据 Reward Model 的分数优化自身策略

这个流程有两个致命问题：第一，标注成本极高——让人类标注员评估数学推理的对错，不仅慢，而且不可靠（标注员之间的一致性经常只有 70-80%）；第二，Reward Model 本身会"作弊"——Reward Model 在训练过程中会逐渐学会捕捉一些与任务无关的特征（比如回答长度、语气），而不是真正的推理质量。

2024-2025 年，DeepSeek 和多家研究机构同时发现了一条全新的路径：在许多推理任务中，我们根本不需要人类标注，也不需要 Reward Model——我们可以通过自动化的"可验证奖励"（Verifiable Rewards）来驱动强化学习。

什么是可验证奖励？简单说就是：如果能用确定的规则来判断一个回答是否正确，那就不需要人类来判断。对于数学题，规则就是"最终答案是否与标准答案一致"；对于编程题，规则就是"代码是否通过了单元测试"；对于逻辑推理题，规则就是"推导是否服从给定的公理系统"。这些规则是完全自动化的、确定性的、零成本的——只要定义好了验证逻辑，就可以无限量地生成训练信号。

为什么这是一个革命性的转变？在 RLHF 时代，后训练数据的瓶颈永远是"人类标注"。标注员的招聘、培训、质量控制、成本控制——这些构成了后训练数据流水线中最昂贵、最耗时的环节。可验证奖励的出现意味着：对于数学、代码、逻辑推理等存在"客观正确性"的领域，后训练数据的生产可以完全自动化——模型生成回答 → 验证器自动打分 → 强化学习优化。人类只需要定义验证规则，不再需要逐条标注。这使得推理增强数据的规模化成为可能。

GRPO（Group Relative Policy Optimization）——DeepSeek-R1 的核心算法

GRPO（Group Relative Policy Optimization，组相对策略优化）是 DeepSeek-R1 的核心训练算法，也是 2025 年推动推理模型能力飞跃的关键技术。

GRPO 的设计思想非常优雅。传统的 PPO 需要一个独立的 Critic 模型（或 Value Model）来估计每个状态的价值——这相当于需要额外训练一个"评分员"来告诉策略模型"你这一步做得怎么样"。而 GRPO 的洞察是：既然我们在同一批样本中已经采样了多个回答，为什么不直接用组内回答的相对表现来替代 Critic 模型呢？

具体来说，GRPO 的工作流程如下：

对于每一个训练问题 q：

1. 从当前策略 π_θ 中采样 G 个回答 {o₁, o₂, ..., o_G}
   （DeepSeek-R1 论文中 G 的取值范围为 8-64）

2. 对每个回答 o_i，通过可验证奖励函数 R 计算得分 r_i
   （例如：数学题答对得 1 分，答错得 0 分）

3. 计算组内得分的均值和标准差：
   μ = (1/G) Σ r_i
   σ = sqrt((1/G) Σ (r_i - μ)²)

4. 对每个回答计算归一化优势值（Advantage）：
   A_i = (r_i - μ) / σ
   ← 这就是"Group Relative"的含义

5. 将归一化优势值 A_i 作为训练信号，优化策略 π_θ
   ← 回答比组内平均好的样本被正向激励
   ← 回答比组内平均差的样本被负向抑制

6. 加入 KL 散度约束，防止策略更新过猛

这个流程的精妙之处在于：它完全不依赖人类标注，也不依赖额外训练的 Reward Model——只用组内相对表现作为训练信号。归一化之后的优势值让模型学会了"在同一个问题的不同回答中，哪些是相对更好的"。DeepSeek-R1 正是通过这种方法，在数学推理、代码生成等领域达到了与 OpenAI o1 相当的水平。

维度 PPO（传统方法） GRPO（DeepSeek-R1）
需要 Critic Model 是——额外训练一个价值网络 否——用组内归一化替代
需要 Reward Model 是——需要人类偏好数据训练 否——用可验证奖励即可
训练复杂度 高——4 个模型（策略/Critic/Reward/参考） 低——2 个模型（策略/参考）
回答采样数 单采样（1 个回答 per query） 组采样（8-64 个回答 per query）
奖励来源 人类标注 → Reward Model 打分 自动化验证器直接评分
适用场景 通用对齐（写作、创意、对话） 推理任务（数学、代码、逻辑）
数据角度的解读：GRPO 对数据的依赖方式和传统 SFT 完全不同。在 SFT 中，数据是一次性准备好的——"收集 → 清洗 → 训练"三步走。而在 GRPO 中，数据是"动态生成的"——模型在训练过程中不断采样新回答，验证器实时打分，组内相对优势实时计算。这意味着数据管线的设计思路也要随之调整：你不再需要准备几十万条标注好的 instruction-response pair，而是需要准备（1）高质量的 prompt 库（用于采样）和（2）可靠的可验证奖励函数（用于打分）。在第 28 章的综合实战中，我们会详细讨论如何搭建这样一条"动态数据管线"。

RLVR 数据设计原则

RLVR（Rule-Based Verifiable Rewards，基于规则的可验证奖励） 是 GRPO 的"奖励信号供应端"。它的工作就是定义：什么样的回答是好的，什么样的回答是坏的。不同的推理任务需要不同的验证策略：

  • 数学任务：答案字符串匹配 / 符号验证
    最简单的验证器检查最终答案是否与标准答案完全一致。更可靠的方案是用符号计算引擎（如 SymPy、Wolfram Alpha）验证两个数学表达式是否等价——因为"x=3"和"x=12/4"在字符串层面不同，但数学上等价。
  • 编程任务：单元测试执行
    验证器将模型生成的代码放入预定义的测试用例中执行，检查是否通过所有测试。这是最可靠的验证方式——代码要么通过测试，要么不通过，没有模糊地带。
  • 逻辑推理：规则验证器
    对于逻辑谜题、约束满足问题等，验证器检查推导过程是否违反了给定的约束条件。例如在数独问题中，验证器检查每一行、每一列、每一个宫格是否都满足数字不重复的条件。

下面是 RLVR 验证器的一个典型实现伪代码：

class RLVRVerifier:
    """基于规则的答案验证器"""

    def verify_answer(self, problem: dict, response: str) -> float:
        task_type = problem["type"]

        if task_type == "math":
            # 从 response 中提取最终答案
            extracted = self.extract_final_answer(response)
            # 符号等价验证（比字符串匹配更鲁棒）
            return 1.0 if self.symbolic_equal(extracted, problem["answer"]) else 0.0

        elif task_type == "code":
            # 提取代码块
            code = self.extract_code(response)
            # 运行单元测试
            passed = self.run_unit_tests(code, problem["test_cases"])
            return passed / len(problem["test_cases"])

        elif task_type == "logic":
            # 检查所有约束条件
            violations = self.check_constraints(response, problem["constraints"])
            return 1.0 if violations == 0 else 0.0

        else:
            raise ValueError(f"Unknown task type: {task_type}")

DAPO 数据优化——动态采样策略

DAPO（Dynamic Adaptive Policy Optimization for RLVR）是 GRPO 的一个重要改进版本，由字节跳动和清华大学的研究人员联合提出。DAPO 的核心洞察是：GRPO 中的"均匀组采样"并非最优策略——不同的问题，组内回答的质量分布差异很大，需要动态调整采样策略。

DAPO 引入了几个关键的数据优化策略：

  • 动态采样过滤无区分度样本：如果某个问题下，所有 32 个回答全部答对（得分全部为 1），或者全部答错（得分全部为 0），那么这个问题的训练信号是毫无区分度的——所有回答的归一化优势值都是 0，训练梯度为零。DAPO 会动态地丢弃这类问题，或者推迟到策略更新到一定阶段后再重新采样。
  • Clip-Higher：传统 GRPO 对优势值的裁剪（clipping）范围是对称的，但 DAPO 发现对高优势值采用更宽松的裁剪范围有助于维持探索动力。
  • Token-Level Loss：将损失从回答级别分解到 token 级别，让模型在推理链的不同部分获得更细粒度的训练信号。
  • Overlong Reward Shaping：当模型生成的回答超过预设长度限制时，对超出部分的奖励进行惩罚——鼓励模型生成"高效"的推理链，而不是通过无意义的啰嗦来掩盖推理的薄弱。
一个有趣的发现：DAPO 团队发现，GRPO 训练中的"无区分度样本"比例随着时间的推移而增大。在训练初期，模型能力还不够强，每个问题下的 32 个回答往往既有对的也有错的，组内方差很大，训练信号充足。但随着训练的推进，模型能力提升，越来越多的简单问题被"彻底征服"——所有回答都对了，组内方差趋近于零。如果不对这些样本做过滤，模型就会在不提供任何训练信号的问题上浪费计算资源。DAPO 的动态采样策略正是针对这一现象设计的——根据模型当前能力动态调整采样问题的难度分布。

RLVR 数据质量门控

RLVR 虽然不依赖人类标注，但它的质量仍然需要精细的控制。下面几个"质量门控"是实践中积累出的关键经验：

  • 过滤所有回答全对或全错的 query：如前所述，这类 query 无法提供有效的组内训练信号。保留它们只会浪费计算资源。
  • 验证器校准：确保验证器本身没有系统性偏差。例如，如果验证器在检查数学答案时使用了错误的容差阈值（eps=1e-3 vs eps=1e-6），会导致一些"近似正确"的答案被错误地判定为错误，给模型提供反向的训练信号。
  • 移除过度简单样本：如果模型在某个类别的问题上已经达到了 95% 以上的准确率，这类问题的训练价值就非常有限了。应该从采样池中移除，替换为更具挑战性的问题。
门控类型 检测条件 处理方式
无区分度 query G 个回答全对或全错 跳过本轮采样，推后重新采样
验证器漂移 在验证集上的 F1 分数低于阈值 重新校准验证规则或更新验证器
难度不足 模型在同类问题上的准确率 > 95% 从采样池中移除，替换为更难题
奖励溢位 组内奖励方差趋近于 0 检查是否所有回答都达到了满分阈值

冷启动 SFT + RL 的两阶段数据 Pipeline

把第 19 章和本章的知识点串联起来，就构成了 DeepSeek-R1 论文中著名的完整训练配方：

┌─────────────────────────────────────────┐
│        阶段一：冷启动 SFT                │
│                                         │
│  数据：CoT 推理数据（OpenThoughts3,     │
│        DeepSeekMath-SFT 等）            │
│  方法：标准交叉熵 SFT 训练              │
│  目标：让模型学会输出连贯的推理链       │
│  输出：具有基础推理能力的 SFT 模型       │
├─────────────────────────────────────────┤
│        阶段二：RL 增强（GRPO）           │
│                                         │
│  数据：推理问题 + 可验证奖励函数        │
│        （数学答案匹配、代码单元测试等） │
│  方法：GRPO 组采样强化学习              │
│  目标：通过试错和自我纠错优化推理质量   │
│  输出：具有强推理能力的最终模型         │
│         （DeepSeek-R1 级别）             │
└─────────────────────────────────────────┘

这个两阶段流程有一个容易被忽视的关键细节：阶段二的 GRPO 训练质量高度依赖于阶段一 SFT 模型的质量。如果 SFT 模型的推理链质量不够好（比如逻辑跳跃严重、中间步骤过少），GRPO 阶段就需要更多的采样和迭代来弥补这个缺陷。反之，如果 SFT 阶段的 CoT 数据质量已经很高，GRPO 阶段的收敛速度会大幅加快。在 DeepSeek-R1 的实验中，一个高质量的 SFT 冷启动可以让 GRPO 阶段的采样数量减少 40% 以上。

提醒：GRPO 和 RLVR 虽然强大，但它们并不是"银弹"。这类方法目前最擅长的领域是"有客观正确答案"的任务——数学、代码、逻辑推理。对于"主观判断"类任务（比如写作风格、创意生成、情感对话），可验证奖励难以定义，RLHF 仍然是更合适的选择。所以一个完整的后训练数据流水线通常需要同时包含 CoT/GRPO 推理数据和 RLHF 偏好数据——两者服务于不同的能力维度。
