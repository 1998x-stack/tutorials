第14章 文本分类实战

大家好，欢迎来到实战章节。前面我们花了整整四章讲词向量、卷积网络到Transformer这些"内功心法"，今天到了"打木桩"的时候了——文本分类。

文本分类在NLP界的地位，就像手写数字识别在CV界的地位：最经典、最基础、最容易上手。你几乎找不到一个NLP工程师没做过文本分类的。情感分析、垃圾邮件过滤、新闻归类、意图识别……这些都是文本分类的典型应用。可以说，如果你只会一个NLP任务，那就应该是文本分类。

今天我们会从数据准备一直走到模型训练和评估，用PyTorch手写一个TextCNN，搭一套完整的文本分类pipeline。这条链路里的每一环——TextCNN怎么用卷积捕捉n-gram特征、类别不平衡怎么办、准确率为什么有时候骗人——都是你以后做任何NLP项目都会反复遇到的核心问题。

前置知识：建议你先回顾第4章（文本表示）了解词嵌入，第6章看看卷积在时序数据上的基本用法。如果你对PyTorch的nn.Conv2d和nn.Embedding还不熟，花十分钟去PyTorch官网看一下文档，不亏。

14.1 TextCNN —— 用卷积做文本分类

TextCNN是Yoon Kim在2014年EMNLP上提出的模型，思想非常朴素：把文本看成一张一维的"图像"，用不同大小的卷积核去捕捉不同长度的n-gram特征。

你可能会觉得奇怪——卷积不是用来做图像的吗？用在文字上能行？我们先想一个问题：在图像里，一个3x3的卷积核可以捕捉局部像素的纹理模式；在文本里，如果用一个高度为2、宽度等于词向量维度的卷积核，它就相当于在滑动窗口内捕捉"两个连续词的组合特征"——也就是bigram。同理，高度为3的卷积核捕捉trigram，高度为4的卷积核捕捉4-gram。

TextCNN的核心结构就三步：

  1. 词嵌入层：把每个token映射成d维向量，得到 [batch_size, seq_len, embed_dim] 的张量。一般用预训练词向量（如GloVe）或随机初始化然后参与训练。
  2. 多尺度卷积层：用多组不同大小的卷积核（通常window sizes取[2, 3, 4]或[3, 4, 5]）分别做一维卷积（严格说是二维卷积但height维度滑动，width固定为embed_dim）。每个卷积核的输出经过ReLU激活，然后做一个max-over-time pooling——取每个feature map的最大值作为这个卷积核提取到的"最强特征"。
  3. 全连接分类层：把所有卷积核pooling后的结果拼接成一个向量，过dropout，再过一个线性层映射到类别数，最后用softmax得到概率分布。

下面是TextCNN的PyTorch实现：

import torch
import torch.nn as nn
import torch.nn.functional as F


class TextCNN(nn.Module):
    def __init__(self, vocab_size, embed_dim, num_classes,
                 kernel_sizes=(2, 3, 4), num_filters=100, dropout=0.5):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.convs = nn.ModuleList([
            nn.Conv2d(1, num_filters, (k, embed_dim), padding=(k - 1, 0))
            for k in kernel_sizes
        ])
        self.dropout = nn.Dropout(dropout)
        self.fc = nn.Linear(len(kernel_sizes) * num_filters, num_classes)

    def forward(self, x):
        # x: [batch_size, seq_len]
        x = self.embedding(x)               # [batch, seq_len, embed_dim]
        x = x.unsqueeze(1)                  # [batch, 1, seq_len, embed_dim], 添加channel维度

        conv_results = []
        for conv in self.convs:
            out = conv(x)                   # [batch, num_filters, out_len, 1]
            out = F.relu(out.squeeze(3))    # [batch, num_filters, out_len]
            out = F.max_pool1d(out, out.size(2)).squeeze(2)  # [batch, num_filters]
            conv_results.append(out)

        x = torch.cat(conv_results, dim=1)  # [batch, len(kernels)*num_filters]
        x = self.dropout(x)
        return self.fc(x)                   # [batch, num_classes]

代码里的几个细节值得注意：

  • padding=(k-1, 0) 的目的是让卷积后的序列长度和输入一致（至少不会短太多），这样max_pooling之前每个feature map的长度可控。
  • 为每个kernel size单独建一个nn.Conv2d，而不是共享参数，因为我们要的不同粒度的特征。
  • padding_idx=0 表示词表中索引0对应的token（通常预留作标志）在嵌入时全零，梯度更新时也被忽略。

你可能会问：TextCNN都2014年的东西了，现在还用它？答案是——很多工业级文本分类系统依然在用。对于短文本分类（比如几行字的评论、标题、查询），TextCNN的效果跟BERT-base在一个量级，但推理速度是BERT的10-20倍，参数量不到1/50。这就是经典模型的魅力：它没有在SOTA榜上出现，但实实在在地在生产环境里活着。

14.2 构建数据集：标注、清洗与划分

在开始训练之前，让我们面对一个残酷的现实：你永远拿不到一份完美的数据。现实中80%的NLP时间都花在数据处理上，模型训练反而是最轻松的一环。

14.2.1 数据标注

标注质量直接决定模型的上限。这里有几条血泪经验：

  • 标注指南比标注本身更重要：三页纸的标注规范至少要写明"边界案例怎么处理"。比如情感分类中，"这个产品还行"是积极还是中性？没有指南的话，三个标注员能打出三个标签。
  • 做一致性检查：抽10%的样本让两个标注员独立标注，计算Cohen's Kappa系数。Kappa应大于0.7，低于0.6说明标注指南需要修订。
  • 避免标签泄露：标注时只看文本本身，不要看用户画像、时间戳等额外信息，否则模型会学到"捷径"——比如学会根据用户ID判断情感（因为老用户的评价通常更正面）。

14.2.2 文本清洗

清洗策略和任务紧密相关。新闻分类可能只需要去除HTML标签和特殊字符；社交媒体文本则需要处理表情符号、URL、@提及、不规范标点等。一般流程是：

def clean_text(text, mode='general'):
    import re

    # 1. 统一小写（中文不需要，但英文需要）
    text = text.lower()

    if mode == 'social':
        # 社交媒体：保留情感信号但标准化
        text = re.sub(r'@\w+', '@USER', text)    # 替换用户提及
        text = re.sub(r'https?://\S+', 'URL', text)  # 替换链接
        text = re.sub(r'#(\w+)', r'\1', text)     # 去除#号但保留文本
    elif mode == 'news':
        # 新闻：清理更激进
        text = re.sub(r'<[^>]+>', '', text)         # 去HTML标签
        text = re.sub(r'[^\w\s]', '', text)         # 去标点

    # 2. 空白规范化
    text = re.sub(r'\s+', ' ', text).strip()
    return text

14.2.3 数据划分

经典的三分法：训练集（60-70%）、验证集（15-20%）、测试集（15-20%）。但要注意两个陷阱：

  • 分层采样（Stratified Split）：直接用sklearn.model_selection.train_test_split时一定要设stratify=y参数，否则小类别的样本可能在某个分片里完全消失。
  • 时间泄漏：如果数据带有时间戳（比如按天收集的新闻），不要随机切分。应该按时间顺序划分——用前80%的时间段做训练，后20%做测试。否则你等于让模型"预知未来"。

14.3 类别不平衡：沉默的杀手

现实中的分类数据几乎永远是不平衡的。垃圾邮件只占全部邮件的5-10%；差评在商品评价里可能只占20%；诈骗交易的占比甚至低于1%。类别不平衡不是异常，而是常态。

💥 踩坑故事：我2020年做一个金融风控项目，要识别违规交易。原始数据中正常交易占99.82%，违规占0.18%。用原始数据直接训练了一个LightGBM，准确率99.80%，看起来"好极了"。上线第一周，模型把全部交易都判为正常——因为全判正常的准确率是99.82%，比模型自己还高0.02个百分点。这个模型做了整整一周的"稻草人"，被业务方追着骂了一周才定位到问题。

这个故事说明一个残酷的道理：在不平衡数据集上，准确率是骗人的指标。下面介绍三种主流应对策略。

14.3.1 过采样（Oversampling）

复制少数类样本，使各类样本数基本平衡。最简单的做法是从少数类中有放回地抽样到和多数类一样多。更聪明一点的是SMOTE（Synthetic Minority Oversampling Technique），它在少数类样本之间的连线上插值生成新样本。不过SMOTE在文本领域的效果没那么好——在词空间中插值可能产生无意义的"句子"。

14.3.2 加权损失（Weighted Loss）

不在数据层面动手脚，而是在损失函数里给少数类更高的权重。PyTorch里直接设nn.CrossEntropyLoss(weight=class_weights)，其中class_weights通常取各类样本数倒数的归一化值：

from sklearn.utils.class_weight import compute_class_weight

class_weights = compute_class_weight(
    'balanced',
    classes=np.unique(train_labels),
    y=train_labels
)
class_weights = torch.tensor(class_weights, dtype=torch.float).to(device)

criterion = nn.CrossEntropyLoss(weight=class_weights)

加权损失的优点是零改动成本——改一行loss就行；缺点是对于极度不平衡（比如正负比1:1000）的情况，少数类的梯度信号会被放大到不稳定。

14.3.3 Focal Loss

Focal Loss是RetinaNet（2017）为解决目标检测中的类别不平衡而提出的，后来被广泛引用到文本分类中。它的核心思想是：让模型更关注那些"难分"的样本，而不是已经分对的"容易"样本。

class FocalLoss(nn.Module):
    def __init__(self, alpha=0.25, gamma=2.0, reduction='mean'):
        super().__init__()
        self.alpha = alpha
        self.gamma = gamma
        self.reduction = reduction

    def forward(self, logits, targets):
        ce_loss = F.cross_entropy(logits, targets, reduction='none')
        pt = torch.exp(-ce_loss)
        focal_loss = self.alpha * (1 - pt) ** self.gamma * ce_loss

        if self.reduction == 'mean':
            return focal_loss.mean()
        elif self.reduction == 'sum':
            return focal_loss.sum()
        return focal_loss

alpha控制正负样本的权重（类似于加权损失中的class_weight），gamma控制"关注难例"的程度——gamma越大，模型越聚焦于低置信度的样本。实践中alpha=0.25、gamma=2.0是一个不错的起点。

三种方法的对比：

方法适用场景优点缺点
过采样轻度不平衡（1:10内）实现简单，不修改模型结构容易过拟合；SMOTE在文本上效果有限
加权损失中度不平衡（1:10 ~ 1:100）只需改一行loss，零额外开销极度不平衡时梯度不稳定
Focal Loss重度不平衡（1:100以上）动态关注难例，鲁棒性最强多了一个gamma要调参

实战建议：先尝试加权损失——因为它改动最小。如果验证集上的少数类Recall还不够，再加Focal Loss或过采样。不要一上来就上全套，越复杂的策略越难调试。

14.4 评估指标：准确率何时说谎

上文提到了，类别不平衡时准确率会骗人。那么我们用什么指标？

14.4.1 混淆矩阵（Confusion Matrix）

混淆矩阵是一切分类指标的基础。四类结果：TP（真正例）、FP（假正例）、FN（假负例）、TN（真负例）。画出来就是一个2x2（或多类别下的NxN）矩阵。重要的是不要只看数字，要横纵向归一化看——行归一化看Recall（每类被正确召回的比例），列归一化看Precision（预测为某类的样本中真正属于该类比例）。

14.4.2 Precision、Recall、F1

  • Precision（精确率） = TP / (TP + FP)。关注的是"模型说它是正例，它到底是不是"。
  • Recall（召回率） = TP / (TP + FN)。关注的是"所有真正的正例，模型找出来了多少"。
  • F1 Score = 2 * P * R / (P + R)。Precision和Recall的调和平均，单指标场景下的首选。

在多分类场景下，常用的聚合方式是macro-F1（每个类算F1再取平均，给少数类同等权重）和weighted-F1（按各类样本数加权平均）。如果你的任务是类别不平衡且关注少数类，看macro-F1；如果更关注整体表现，看weighted-F1。

下面是同一个模型在不同数据分布下指标的表现：

数据场景准确率Macro-F1说明
类别平衡（1:1）85.0%84.8%两者接近，准确率可信
轻度不平衡（5:1）87.2%76.3%准确率已经"虚高"了约10个点
重度不平衡（100:1）98.9%32.1%准确率几乎废了，模型在"摆烂"

看到没有？在重度不平衡场景下，准确率98.9%和废物模型Macro-F1才32.1%同时出现——所以永远不要只看准确率。

14.5 端到端训练流程

把上面所有内容串起来，一个完整的训练循环大概长这样：

def train_one_epoch(model, dataloader, optimizer, criterion, device):
    model.train()
    total_loss = 0
    correct = 0
    total = 0

    for batch in dataloader:
        text, labels = batch['text'].to(device), batch['label'].to(device)

        optimizer.zero_grad()
        logits = model(text)
        loss = criterion(logits, labels)
        loss.backward()

        # 梯度裁剪——防梯度爆炸，对文本分类不是必须但有益
        torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

        optimizer.step()

        total_loss += loss.item()
        preds = logits.argmax(dim=1)
        correct += (preds == labels).sum().item()
        total += labels.size(0)

    return total_loss / len(dataloader), correct / total


def evaluate(model, dataloader, criterion, device):
    model.eval()
    total_loss = 0
    all_preds, all_labels = [], []

    with torch.no_grad():
        for batch in dataloader:
            text, labels = batch['text'].to(device), batch['label'].to(device)
            logits = model(text)
            loss = criterion(logits, labels)

            total_loss += loss.item()
            all_preds.extend(logits.argmax(dim=1).cpu().numpy())
            all_labels.extend(labels.cpu().numpy())

    accuracy = (np.array(all_preds) == np.array(all_labels)).mean()
    f1 = f1_score(all_labels, all_preds, average='macro')

    return total_loss / len(dataloader), accuracy, f1, all_preds, all_labels


# ---------- 主训练循环 ----------
num_epochs = 20
best_f1 = 0.0

for epoch in range(1, num_epochs + 1):
    train_loss, train_acc = train_one_epoch(
        model, train_loader, optimizer, criterion, device
    )
    val_loss, val_acc, val_f1, _, _ = evaluate(
        model, val_loader, criterion, device
    )

    print(f"Epoch {epoch:2d} | Train Loss: {train_loss:.4f} (Acc: {train_acc:.4f}) "
          f"| Val Loss: {val_loss:.4f} (Acc: {val_acc:.4f}, Macro-F1: {val_f1:.4f})")

    # 用macro-F1做早停和模型保存
    if val_f1 > best_f1:
        best_f1 = val_f1
        torch.save(model.state_dict(), 'best_textcnn.pt')
        print(f"  -> Best model saved (F1: {best_f1:.4f})")

注意几个工程细节：

  • 用macro-F1而不是准确率做早停的监控指标——因为上面我们已经证明了准确率会骗人。
  • 梯度裁剪（gradient clipping）在文本分类里不是必须的，但加上几乎不增加成本，能防止偶发性梯度爆炸导致loss飞升。
  • 每轮都记录验证集指标而不是只看训练集——这是防止过拟合的第一道防线。

完整Pipeline总结

把今天的内容整合一下，一个项目级的文本分类完整流程是：

  1. 数据准备：标注（指南 -> 标注 -> 一致性检查） -> 清洗（按任务选择清洗策略） -> 分层划分训练/验证/测试集
  2. 数据分析：统计类别分布 -> 判断是否不平衡 -> 根据不平衡程度选择策略（加权损失 / Focal Loss / 过采样）
  3. 模型构建：设计TextCNN或选用更强的模型 -> 建立预处理pipeline（分词、构建词表、padding成batch）
  4. 训练与调优：设定监控指标（推荐macro-F1）-> 训练并早停 -> 在测试集上做最终评估
  5. 结果分析：看混淆矩阵 -> 分析错误case -> 决定是否需要回退到数据阶段做标注修正

这个流程几乎适用于所有NLP分类任务。无论你是做情感分析还是新闻分类还是意图识别，骨架都一样，变的只是具体的数据和模型尺寸。如果你能把这条链路跑通，你就已经具备了"做一个有实用价值的NLP系统"的基本能力。
