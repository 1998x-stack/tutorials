梯度下降：深度学习唯一的优化语言

大家好，我是你们的讲师。上一章我们建立了一套科学的调参方法论——先诊断再下手，控制变量法，记录一切。今天我们进入第一块具体的技术知识：梯度下降。无论你用什么框架（PyTorch、TensorFlow、JAX），用什么优化器（Adam、SGD、Lion），底层跑的本质上都是同一个核心算法：梯度下降。

我在面试算法工程师时经常问一个问题："梯度为什么指向最陡上升方向？"能把这个讲清楚的人，对优化算法的理解深度和对超参数调优的直觉力，通常远高于只会调API的人。因为你对调参的理解深度，不会超过你对梯度下降的理解深度。

这一章我们不纠结于手推公式，而是建立几何直觉——让你理解梯度背后发生了什么，它的噪声从哪来，Adam为什么比SGD聪明，动量是怎么让参数"滚起来"的。

从方向导数理解梯度

先问一个最基础的问题：梯度为什么是"最陡下降方向"？

想象你在一个山谷中，被蒙上眼睛，任务是走到谷底。你每走一步，需要决定朝哪个方向迈出。没有视觉，唯一的信息是脚下的坡度——你能感觉到哪个方向是往下最陡的。你沿着最陡的方向迈一步，再感受，再迈一步，直到到达谷底。这就是梯度下降的物理直觉。

数学上，梯度指向的是函数在某一点增长最快的方向。那下降最快的方向呢？就是梯度的反方向。这是为什么？我们可以从泰勒展开来看。对一个可微函数 f(θ)，在某点 θ₀ 附近做一阶泰勒展开：

f(θ₀ + Δθ) ≈ f(θ₀) + ∇f(θ₀)·Δθ

我们想要选择一个方向 Δθ 使得 f 下降最快。在约束 ||Δθ|| = ε（步长固定）下，f(θ₀ + Δθ) - f(θ₀) ≈ ∇f(θ₀)·Δθ = ||∇f(θ₀)||·||Δθ||·cos(φ)，其中 φ 是梯度方向与 Δθ 的夹角。想让这个差值最小（即下降最多），只需要 cos(φ) = -1，也就是 Δθ 与梯度方向完全相反。这就是为什么负梯度方向是最陡下降方向。

这个推导看起来简单，但它揭示了一个关键事实：梯度下降是一阶方法——只用了梯度信息，也就是函数的一阶导数信息。它假设函数在局部是线性的，沿着最陡的方向走一步，然后再重新估计。这就是为什么当损失曲面极度弯曲时，梯度下降会遇到困难——线性的局部近似不再准确了。

调参启示：当你发现loss曲线震荡剧烈、难以收敛时，本质上是在说"泰勒一阶近似在这个区域内不够准确"。此时可以考虑：降低学习率让每一步走得更谨慎、或者使用自适应优化器（如Adam）为每个参数独立调整步长。

随机梯度下降的噪声特性

讲完了全批量梯度下降，我们来说说实际中几乎没人会用的"理论版"——全批量梯度下降（BGD）。BGD每次更新需要用全部训练数据计算一次梯度。假如你有100万张图片，每更新一次参数就要遍历全部100万张——太慢了。于是有了随机梯度下降（SGD）：每次随机选一个小批量（mini-batch），用这一小批样本的梯度来近似全量梯度。

这带来了一个关键特性——噪声。我们先用统计学视角理解一下：

  • 全批量梯度 = 真实梯度（期望中的梯度方向）
  • 小批量梯度 = 真实梯度 + 随机噪声
  • 噪声的方差 ∝ 1/B，其中 B 是batch size

换句话说，batch越大，梯度估计越准确，噪声越小；batch越小，梯度越"抖"，每次更新方向可能偏离真实梯度较远。

这个噪声是双刃剑——好的一面：噪声帮助参数逃离局部极小值（saddle point和shallow local minima），让你有机会找到更好的解。你可以理解为，在优化过程中加入随机性，类似于模拟退火中的"温度"参数。坏的一面：噪声太大导致训练不稳定，loss曲线像心电图一样上下乱窜，收敛速度变慢，甚至无法收敛。

从中心极限定理的角度看：小批量梯度可以看作是真实梯度加上一个近似高斯的噪声项。这个噪声项的方差由batch size决定。随着batch增大，梯度估计更加集中在真实梯度附近。这就是为什么大batch训练时loss曲线平滑，小batch时曲线粗糙。

实战经验：如果你发现训练loss曲线震荡剧烈，不要急着怀疑模型或数据——先检查batch size是不是太小了。对于大多数CV任务，batch size 32-128是一个合理的起步范围。如果显存允许，增大batch size是平滑梯度最直接的方法。但注意，我们后面会讲到，大batch也有自己的问题。

一阶方法与二阶方法的对比

前面说梯度下降是"一阶方法"——只用了梯度（一阶导数）。你可能会问：如果用上二阶导数（Hessian矩阵），会不会更好？答案是：理论上是，但现实中几乎不可行。

牛顿法就是最典型的二阶方法。它的更新方向不仅考虑梯度，还考虑了曲率信息：

牛顿法更新: θ ← θ - H⁻¹·∇f(θ)
        梯度下降更新: θ ← θ - lr·∇f(θ)

Hessian 矩阵 H 告诉我们损失曲面在每个方向上的曲率——在某些方向"山路陡峭"（曲率大），在其他方向"比较平缓"（曲率小）。牛顿法利用曲率信息，在每个方向上自适应地调整步长：在陡峭方向走小步，在平缓方向走大步。理想情况下，牛顿法一步就能达到二次型函数的极小值。

那为什么实践中没人用牛顿法？因为Hessian矩阵的维度是参数量×参数量。一个ResNet50有2500万个参数，它的Hessian矩阵就有2500万×2500万 ≈ 6.25×10¹³ 个元素——用float32存需要250TB内存。而且求逆矩阵的复杂度是 O(n³)，完全无法落地。

那么Adam是怎么"近似"曲率信息的呢？Adam维护了两个统计量：

  • 一阶矩 mₜ：梯度的指数移动平均——相当于动量，告诉参数应该朝哪个方向更新
  • 二阶矩 vₜ：梯度平方的指数移动平均——告诉参数在当前方向上的梯度大小，近似反映了曲率信息

Adam用 vₜ 来缩放每个参数的学习率：如果某个参数的梯度一直很大（损失曲面在这个方向陡峭），vₜ 就大，有效学习率就变小。如果梯度一直很小（曲面平缓），有效学习率就变大。这种"逐参数自适应学习率"的机制让Adam在一阶方法中获得了二阶方法的某些好处，而计算成本只比SGD高一点点。

动量法的物理直觉

还记得我们说的蒙眼下山的例子吗？如果你是沿着最陡方向走一步、停一步、再感觉、再走一步，那你走路的轨迹会非常"zigzag"（锯齿状）——在山谷中左右摇摆不断，整体前进缓慢。

动量法（Momentum）的直觉来自物理学：把参数想象成一个在山谷中滚动的球。球不只是受当前坡度（梯度）的影响，还保留了之前的运动速度（动量）。即使某个时刻脚下一空（梯度为0），球也会因为惯性继续向前滚动。

vₜ = α·vₜ₋₁ + lr·∇f(θₜ)
        θₜ₊₁ = θₜ - vₜ

其中 α 是动量系数，通常取0.9。它控制着"惯性"有多大。α=0 时退化为标准SGD。α=0.9 时，历史梯度的影响以指数速度衰减，最近10步左右的梯度方向主导了当前更新方向。

动量带来的两个关键优势：

  • 穿越平坦区域：当参数进入一个坡度很小的平坦区域时，梯度几乎为零。没有动量的话，参数会"卡住"几乎不动。但动量让球"记住"之前的运动方向，可以穿越平坦区域继续前进。这在优化景观中非常关键——很多优秀的解被平坦区域包围着。
  • 抑制震荡：在狭长的山谷中，梯度方向在窄的方向上不断正负交替（zigzag）。动量对正负交替的方向做了平均——正负抵消，振幅减小。而在前进方向上，梯度始终为正，动量累积。结果是：震荡被抑制，前进速度加快。

Nesterov动量（NAG）是标准动量的一个改进版。它的思路很巧妙：先沿着上次的方向"探一步"，在探到的位置计算梯度，再用这个梯度来修正最终方向。这相当于让球在滚到谷底前就开始"刹车"，不容易冲过头。

一个常见误解："PyTorch里设置 optimizer=SGD(momentum=0.9) 就自动加了动量。" 不完全对。torch.optim.SGD 的 momentum 参数默认是 0，也就是纯SGD。必须显式设置 momentum=0.9 来启用动量。很多人训练效果不好，可能只是因为没有开启动量。另外，PyTorch的SGD实际上内置了Nesterov动量选项——设置 nesterov=True 即可。

四大优化器对比

现在我们已经理解了梯度下降、随机梯度、动量、自适应的基本概念，下面是四种最核心优化器的对比表。这张表是后续几章的基础，建议保存参考：

优化器 核心机制 计算开销 收敛速度 泛化性能 调参难度 典型LR
GD（全批量） 全数据集计算梯度 极高（每步遍历全量） 慢 好（梯度准确） 低 0.1~1.0
SGD（随机） 小批量梯度估计 低 中等 较好（噪声正则化） 高 0.01~0.1
SGD+Momentum 梯度+历史动量累积 低（多存一组v） 快 最好 中等 0.01~0.1
Adam 自适应学习率+动量 中等（存m+v两组） 最快 较好（略逊Momentum） 低 3e-4~1e-3

留意一下Adam的泛化性能标注是"较好"而非"最好"。这不是偶然——Adam因为自适应学习率机制，倾向于收敛到尖锐极小值，测试集泛化性能往往不如SGD+Momentum。这就是我们下一章要展开的核心话题。

建立直觉：当你跑一个新模型时，先用Adam快速看到收敛趋势（判断代码有没有bug），然后再切到SGD+Momentum去追求最佳泛化效果。这个策略在大型团队中是标准做法，我们叫它"两阶段训练法"——下一章会详细讲解。

本章我们从梯度下降的数学直觉出发，理解了负梯度方向是最陡下降方向的泰勒展开解释、小批量梯度的噪声本质和方差公式、一阶方法vs二阶方法的取舍、动量法的物理模型。有了这些基础，我们再来看优化器时就不再是"调API"了——你知道每个旋钮背后的几何意义。
