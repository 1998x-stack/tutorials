第18章 工程目录规范与版本管理

先讲一个真实的故事。2020 年，我加入了一个 FPGA 开发团队，接手一个已经进行了 8 个月的雷达信号处理项目。打开工程目录，里面的结构是这样的——一个文件夹，名叫 final_project_v3_final_actually，里面堆了四百多个文件：top_v1.v、top_v2.v、top_v3_good.v、top_v4_final.v、top_v5_real_final.v……每个人用各自的命名风格，IP 核的版本随机，没有 .gitignore，整个 build 目录提交到了 Git 仓库（6GB）。原工程师离职后，没有人能完整地编译出一版可用的比特流。

这不是段子，这是 FPGA 行业每天都在发生的事情。硬件工程比软件工程更脆弱——一个错误的引脚约束可能导致改板、一个不兼容的 IP 版本可能导致整个工程报废。工程规范不是文档负担，是生存保障。

18.1 为什么需要工程规范

FPGA 工程规范和软件工程规范有一些相似之处，但也有自己的独特挑战：

  • 工具版本敏感：Vivado 2020.1 打开的工程，Vivado 2021.1 可能直接打不开（或者打开后 IP 核需要重新升级）。软件工程很少遇到"GCC 9 编译通过的代码到了 GCC 10 就编译不过"——但 FPGA 世界这是常态。
  • IP 核版本锁定：同一个 IP 核的不同版本生成的 RTL 可能不同，综合结果也不同。如果两个人用的 IP 版本不一致，他们的综合结果就是不可比的。
  • 大型二进制文件：.bit 文件（几 MB 到几十 MB）、.mcs 文件、IP 缓存目录（几百 MB 到几 GB）——这些不应该进入版本控制。
  • 硬件依赖：同一个 RTL 代码在不同的器件上可能需要不同的约束文件，同一块板子可能有不同的修订版本。

好的工程规范能解决以上所有问题——它的核心目标只有四个字：可复现性。在任何机器上、任何时候、任何人，都能重复生成完全一致的比特流。

18.2 推荐的目录结构

经过多个项目锤炼，我推荐以下的七目录布局：

my_fpga_project/
├── src/            # RTL 源文件（Verilog/VHDL/SystemVerilog）
│   ├── top.v
│   ├── uart_tx.v
│   ├── spi_master.v
│   └── ...         # 按模块组织，不要把所有 .v 文件平铺
│
├── sim/            # 仿真相关：Testbench + 仿真脚本
│   ├── tb_top.v
│   ├── tb_uart_tx.v
│   ├── modelsim_run.do
│   └── wave.do
│
├── constraints/    # 约束文件：XDC（Vivado）/ QSF+SDC（Quartus）
│   ├── top.xdc
│   ├── timing.xdc
│   └── debug.xdc   # 调试约束与正式约束分离
│
├── ip/             # IP 核配置文件（只存 .xci/.xcix，不存 IP 生成文件）
│   ├── clk_wiz_0.xci
│   ├── fifo_generator_0.xci
│   └── blk_mem_gen_0.xci
│
├── build/          # 构建输出（添加到 .gitignore，不进入 Git）
│   ├── vivado_project/
│   └── bitstreams/
│
├── doc/            # 文档：数据手册摘录、时序图、接口协议说明
│   ├── basys3_sch.pdf
│   └── uart_protocol.md
│
└── tb/             # 额外的验证文件：Cocotb 脚本、Verilator 测试
    └── test_cocotb.py

这里有两个关键原则：源文件和工具生成文件分离（src/ 和 build/ 分开）、源文件和约束文件分离（src/ 和 constraints/ 分开）。这样无论工具版本怎么变，你的 RTL 和约束始终是干净可用的。

18.3 Git 管理策略

FPGA 项目的 Git 管理和软件项目的 Git 管理有一个根本区别——FPGA 工具有大量的生成文件、缓存文件和二进制文件，它们绝对不应该进入版本控制。下面是推荐的 .gitignore 配置：

# ====================================
# FPGA Project .gitignore
# 适用：Vivado / Quartus 工程
# ====================================

## ---- Vivado 生成文件 ----

*.jou          # Vivado 日志文件
*.log          # 综合/实现日志
*.str          # 综合报告
*.rpt          # 各种报告文件
*.xdc.bak      # XDC 备份

## ---- Vivado IP 缓存（体积巨大！） ----
ip/ip_user_files/
.ip_cache/
.ip_meta/
.xil/

## ---- 构建输出目录 ----
build/
*.cache/
*.hw/
*.sim/
*.runs/

## ---- 比特流和烧录文件 ----
*.bit
*.bin
*.mcs
*.prm
*.ltx

## ---- Quartus 生成文件 ----
*.qpf          # Quartus 工程文件
*.qsf          # Quartus 设置文件（引脚分配变更频繁，慎入 Git）
*.smsg
*.summary
*.pin
*.sof
*.pof

踩坑实录——8GB 的 Git 仓库：我曾经加入一个团队，FPGA 项目的 Git 仓库体积达到了惊人的 8GB。原因是什么？每个人都在 git add .，每次编译后都会生成新的 .bit 文件（包含不同时间戳），新的 IP cache 目录，新的综合报告。项目进行了半年，每个成员都 git clone 过一次，但因为仓库太大，新人的 git clone 经常超时失败。最后我们用了一周时间，写了一个专门的 git-filter-repo 脚本来清理历史——因为 .gitignore 只能阻止未来的提交，已经提交到历史中的大文件需要用 git filter-repo 或 BFG Repo-Cleaner 才能移除。从那以后，我在每个项目中都要求：.gitignore 必须在 git init 之前就写好！

18.4 团队协作规范

除了 Git 策略，团队协作中的代码风格和流程规范同样重要：

Verilog 命名规范：建议团队统一采用以下规则——信号名小写蛇形（data_valid、fifo_empty）、参数/常量大写（DATA_WIDTH、CLK_PERIOD）、模块名小写加下划线（uart_tx、spi_master）、时钟信号前缀 clk_（clk_100m、clk_200m）、复位信号后缀 _n 表示低电平有效（rst_n）。always 块中对同一寄存器的赋值不应分散在不同的 always 块中。

代码注释：每个顶层模块文件头部应有模块说明（功能、端口列表、修订历史），每个 always 块应有简短注释说明其用途。对于状态机，状态编码（独热/格雷/二进制）应在注释中说明并附上状态转移图文字描述。

预提交检查：在 git commit 之前，应自动运行 Verilator lint 检查语法和常见设计错误。虽然 Verilator 不完全支持 SystemVerilog 的所有特性，但对于可综合的 Verilog 子集，它的 lint 检查比 Vivado 和 Quartus 更严格，能发现很多工具不会报错的潜在问题（例如未使用的信号、位宽不匹配、组合逻辑环路等）。

CI/CD 概念：对于大型团队，可以考虑搭建持续集成流水线——每次 push 后自动完成综合并输出资源利用报告。不过 FPGA 的综合和实现耗时很长（小型设计 5-10 分钟，大型设计数小时），CI/CD 的配置需要精心设计（比如只对特定的文件变更触发综合）。

18.5 可复现性：Tcl 脚本

实现可复现构建的最有效方法——完全抛弃 GUI 操作，用 Tcl 脚本管理整个 Vivado 流程。Tcl 脚本是纯文本，可以放进 Git 中做版本控制、代码审查和 diff。

下面是一个完整的 Vivado Tcl 构建脚本示例：

# ============================================
# build.tcl — Vivado Non-Project Mode 构建脚本
# 用法：vivado -mode batch -source build.tcl
# ============================================
set outputDir ./build

# 创建设计库
create_project -force my_fpga_project ./build
set_property part xc7a35tcpg236-1 [current_project]

# 添加 RTL 源文件
add_files -scan_for_includes ./src
add_files [glob ./src/*.v]
add_files [glob ./ip/*.xci]

# 添加约束文件
add_files -fileset constrs_1 [glob ./constraints/*.xdc]

# 综合
set_property strategy Flow_PerfOptimized_high [get_runs synth_1]
launch_runs synth_1 -jobs 8
wait_on_run synth_1
open_run synth_1

# 报告综合后资源
report_utilization -file $outputDir/post_synth_util.rpt

# 实现（布局布线）
launch_runs impl_1 -jobs 8
wait_on_run impl_1
open_run impl_1

# 报告实现后资源及时序
report_utilization -file $outputDir/post_impl_util.rpt
report_timing_summary -file $outputDir/post_impl_timing.rpt

# 生成比特流
launch_runs impl_1 -to_step write_bitstream
wait_on_run impl_1

puts "=== Build Complete ==="
puts "Bitstream: $outputDir/my_fpga_project.bit"

为了让 .gitignore 的配置更有针对性，下面这张表帮你决定哪些文件应该跟踪、哪些应该忽略：

文件/目录是否进 Git理由
src/*.v✅ 必须RTL 源码——工程的核心资产
constraints/*.xdc✅ 必须引脚和时序约束定义
ip/*.xci✅ 必须IP 核配置——锁定版本和参数
sim/tb_*.v✅ 必须Testbench 源码——验证的基石
build.tcl✅ 必须可复现构建脚本（替代 GUI 操作）
build/ 目录❌ 忽略工具生成文件，体积大且可重现
*.bit / *.bin / *.mcs❌ 忽略比特流文件，可重新生成
.ip_cache / .xil/❌ 忽略IP 缓存，动辄数 GB
*.jou / *.log / *.rpt❌ 忽略运行时日志，无版本价值
ip/ip_user_files/❌ 忽略IP 生成输出，可重新生成
work/ transcript❌ 忽略ModelSim 工作目录
为什么 Tcl 比 GUI 好？原因很简单——GUI 操作不具备可复现性。你在 Vivado GUI 中点了一个按钮、拖了一个 IP 端口，这些操作不会被记录到任何地方。换一个人、换一台机器，结果可能不同。而 Tcl 脚本可以进入 Git，可以通过代码审查，可以自动运行在 CI 服务器上。Tcl 的另一个优势是：你可以用 report_* 命令自动生成各种报告，而不用手动点菜单。同一个项目，同一条命令，永远产生相同的结果。

关于 IP 版本锁定：当你在 IP Catalog 中配置好一个 IP 核后，Vivado 会生成一个 .xci 文件（或 .xcix）。这个文件是 XML 格式的，记录了 IP 核的所有配置参数。把这个文件提交到 Git，就相当于锁定了 IP 核的版本。任何人用这个 .xci 文件重新生成 IP，都会得到完全相同的配置。注意不要提交 .xci 对应的生成文件（.dcp、网表等）——它们会占用大量空间且可以被重新生成。

同时，建议在工程的根目录放一个 README.md 或 VERSION 文件，记录使用的工具版本（如 "This project targets Vivado 2025.2 on Ubuntu 22.04"）。这样当工具升级时，你可以知道当前工程的"基线环境"是什么。
