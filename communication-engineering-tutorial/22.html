第22章 5G安全架构

大家好，我是你们的讲师。讲了这么多章的网络能力——切片、MEC、QoS——你可能觉得"5G好强大"。但任何通信系统都有一个无法回避的底线问题：安全。如果网络不安全，再高的速率、再低的时延都毫无意义——你敢让一台不安全的5G网络控制你的自动驾驶汽车吗？

5G的安全设计比4G全面得多。原因很简单：5G要服务的场景太"硬核"了——远程手术、工业控制、自动驾驶、金融交易，任何一个安全漏洞都可能造成物理世界的实质性伤害。3GPP从设计5G的第一天起就把安全作为核心需求（Security by Design），而不是事后打补丁。本章将带你全面了解5G安全架构的方方面面。

22.1 4G vs 5G安全特性对比

在深入具体技术之前，我们先从整体上对比一下4G和5G的安全能力差异：

安全特性 4G (LTE) 5G (NR) 意义
用户身份保护 IMSI明文传输（易被截获） SUCI加密传输（防IMSI Catcher） 防止伪基站窃取用户身份
认证机制 EPS-AKA 5G-AKA + EAP-TLS 增加企业级证书认证支持
空口加密算法 EEA1/EEA2 (128-bit) NEA1/NEA2/NEA3 (128-256 bit) 算法升级，支持256位密钥
空口完整性保护 用户面无完整性保护 用户面可选完整性保护 防止用户数据被篡改
漫游安全 GRX/IPX基本保护 SEPP + N32接口端到端保护 漫游信令防篡改
网络切片安全 无 切片隔离 + NSSAAF认证 防止切片间攻击
SBA安全 无 TLS + OAuth 2.0 核心网服务间安全通信
密钥层次 单层密钥体系 多层密钥分层派生 前向安全性更强

整体结论：4G安全体系设计于2000年代，当时的安全威胁模型相对简单。5G安全体系从2016年开始设计，充分考虑了IMSI Catcher、伪基站、信令风暴、切片攻击等新型威胁。安全不是5G的"附加功能"，而是5G架构的内在组成部分。

22.2 5G认证框架

5G支持两种认证机制，覆盖了不同的使用场景：

5G-AKA（认证与密钥协商）

这是5G的默认认证机制，继承自4G的EPS-AKA但做了增强。核心流程：

UE                  gNB/AMF              AUSF/UDM
 |                    |                    |
 |← (1) Authentication Request (RAND)     |
 |                    |                    |
 | (2) 用预置密钥K计算AUTH响应            |
 |                    |                    |
 |→ (3) Authentication Response (RES*)───→|
 |                    |                    |
 |                    |   (4) UDM验证RES* |
 |                    |←───Auth OK/Fail───|
 |                    |                    |
 | (5) 派生会话密钥: KAUSF → KSEAF → KAMF → KgNB

5G-AKA相比4G最大的改进是前向安全性——如果攻击者获取了当前会话密钥，也无法推算出之前的会话密钥。这是通过引入KAUSF（锚点密钥）和更复杂的密钥派生树实现的。

EAP-TLS（可扩展认证协议-传输层安全）

这是5G新增的认证方式，主要面向行业专网和企业客户。EAP-TLS使用数字证书（X.509）代替SIM卡进行认证，企业员工可以用公司颁发的证书接入5G专网，而不需要更换SIM卡。这对于将5G集成到现有企业IAM（身份和访问管理）体系中非常有价值。

工程视角：如果你在部署5G行业专网时遇到企业客户说"我不想给所有人换SIM卡"，EAP-TLS就是你的解决方案。企业只需要部署一套CA证书体系，员工的终端的证书可以直接注册。目前主流的5G模组（高通X75、联发科M80等）都支持EAP-TLS。

22.3 SUCI：订阅隐藏标识符

4G时代最大的安全痛点之一就是IMSI Catcher（国际移动用户识别码捕获器）。攻击者部署一个伪基站，强制UE以明文方式发送IMSI（国际移动用户识别码），从而获取用户身份、位置信息，甚至可以用于后续的定向攻击。

5G用SUCI（Subscription Concealed Identifier，订阅隐藏标识符）彻底解决了这个问题。SUCI是对SUPI（Subscription Permanent Identifier，即5G版的IMSI）的加密版本：

SUPI = IMSI (15位十进制数, 如 460012345678901)
   ↓  UE使用网络侧公钥加密
SUCI = SUPI类型 + 域名 + 路由指示 + 加密后的MSIN
   ↓  (传输过程中任何人都无法解密)
网络侧收到SUCI → UDM用私钥解密 → 恢复SUPI

在空口和网络接口上传输的始终是SUCI而不是SUPI。即使攻击者截获了SUCI，没有网络侧的私钥也无法还原出用户的真实身份。这彻底杜绝了IMSI Catcher攻击。

部署陷阱：SUCI虽然安全，但有一个实际工程问题——公钥分发。UE需要在注册之前就知道用哪把公钥加密SUPI。如果UE没有配置公钥（比如刚出厂或换了运营商），它只能退回到明文传输SUPI。3GPP定义的解决方案是通过USIM卡预置公钥或通过RRC信令下发公钥。但某些早期5G终端没有正确实现这个机制，导致在实际网络中仍然是明文传输SUPI——这是很多安全审计发现的高危问题。

22.4 空口安全：加密与完整性保护

5G空口安全包括两个层次：加密（Ciphering）和完整性保护（Integrity Protection）。

加密算法

5G定义了三种加密算法，均支持128位和256位密钥：

  • NEA1：基于SNOW 3G（祖冲之算法的3G演进版本），128位或256位密钥
  • NEA2：基于AES（高级加密标准），使用128位密钥（CTR模式）
  • NEA3：基于ZUC（祖冲之算法），中国自主设计的流密码算法，128位或256位密钥

完整性保护算法

  • NIA1：基于SNOW 3G
  • NIA2：基于AES-CMAC
  • NIA3：基于ZUC-MAC

5G相比4G的重要改进是：用户面数据（U-Plane）也支持完整性保护。在4G中，只有控制面（C-Plane）信令有完整性保护，用户面的数据包可以被篡改而不被发现（这是一个真实的安全漏洞——中间人可以篡改用户下载的文件）。5G中运营商可以选择性地对特定QoS Flow启用用户面完整性保护。

性能权衡：启用用户面完整性保护会带来额外的计算开销（每个数据包都需要计算MAC-I），对gNB和UE的CPU都有额外消耗。对于eMBB业务，完整性保护的开销大约是5-10%的吞吐量损失。因此工程实践是：URLLC和工业控制业务启用户面完整性保护，普通eMBB业务只启用加密。在基站配置中，这是通过QoS Profile中的integrityProtection参数控制的。

22.5 漫游安全：SEPP与N32接口

5G漫游安全相比4G有了质的飞跃。4G漫游时，不同运营商之间的信令通过IPX（互联协议交换）网络传输，信令内容没有端到端加密——IPX运营商可以"看到"信令内容，存在隐私泄露风险。

5G引入了SEPP（Security Edge Protection Proxy，安全边缘保护代理），部署在运营商网络边界。归属地和拜访地的SEPP之间通过N32接口通信，信令在N32接口上经过端到端加密和完整性保护，IPX网络只能看到加密后的数据包，无法解析信令内容。

拜访地PLMN                   IPX网络                归属地PLMN
  |                            |                        |
  |  N32-c (信令+上下文)      |                        |
  |←───────────────────────────|────────────────────────→|
  |  N32-f (加密后的HTTP/2)   |                        |
  |←───────────────────────────|────────────────────────→|
  |                            |                        |
  SEPP-A                    IPX网络                 SEPP-B
  (加密所有SBI信令)       (只能看到密文)         (解密回原始信令)

工程意义：SEPP的引入意味着运营商之间的漫游信令在传输过程中是面到面加密的——即使是IPX网络运营商也无法查看信令内容。这对于企业客户尤其重要：你的用户在漫游时，其位置信息、业务使用情况不会暴露给第三方网络。

22.6 网络切片安全

网络切片引入了新的安全挑战——如果切片之间的隔离做得不好，攻击者可能通过一个切片攻击另一个切片。5G通过以下机制保障切片安全：

  • 资源隔离：不同切片在RAN、传输网、核心网侧都有资源隔离措施（FlexE硬隔离、独立UPF实例、独立AMF实例等）
  • 切片认证：NSSAAF（Network Slice-Specific Authentication and Authorization Function）对特定切片做二次认证。即使UE已经通过了主认证（5G-AKA），接入特定切片时可能还需要额外的认证
  • 切片间的网络隔离：不同切片使用不同的VLAN/VXLAN ID、不同的IP地址段，防止二层/三层越权访问

22.7 SBA安全：TLS与OAuth 2.0

5G核心网采用了SBA（Service-Based Architecture），网元之间通过HTTP/2 API直接通信。这种架构带来了新的安全需求——如何保障服务间的调用安全？

5G SBA安全的核心是两个机制：

  • TLS（传输层安全）：核心网所有服务间API通信默认启用TLS 1.2/1.3，保障传输通道的加密和双向身份认证
  • OAuth 2.0：核心网引入NRF（网络仓库功能）作为授权服务器，所有服务调用都需要先向NRF申请访问令牌（Access Token），然后带着令牌调用目标服务API
// OAuth 2.0 令牌请求示例 (SBI接口)
// NF服务消费者 → NRF → 获取令牌 → 调用目标NF

POST /nrf-nfm/v1/oauth/token HTTP/1.1
Host: nrf.5gcn.local
Content-Type: application/json

{
  "grant_type": "client_credentials",
  "nf_instance_id": "nf-consumer-01",
  "nf_type": "SMF",
  "target_nf_type": "PCF",
  "scope": "npcf-policyauthorization",
  "client_credentials": {
    "iss": "smf-01.5gcn.local",
    "sub": "smf-01",
    "aud": "nrf.5gcn.local"
  }
}

// 响应: Access Token
HTTP/1.1 200 OK
{
  "access_token": "eyJhbGciOiJSUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 3600
}

安全最佳实践：生产环境中，NF之间的TLS证书必须使用CA签发的正式证书（而非自签名证书），且证书至少每12个月轮换一次。OAuth 2.0的Access Token有效期不应超过1小时，并应在NRF侧配置令牌撤销机制。这些细节在运营商的安全审计中经常被检查。

22.8 真实事件：伪基站与SUCI防护

2022年，某运营商在一次例行安全巡检中发现其部分5G覆盖区域存在异常的信令交互——某些基站在没有收到UE注册请求的情况下持续向核心网发送"注册完成"消息。进一步调查发现，这是某种升级版的伪基站攻击：攻击者部署了支持5G NSA（非独立组网）模式的伪基站，利用NSA模式下LTE锚点优先级高于NR的特性，诱骗UE回落到4G锚点，然后通过4G接口发起IMSI捕获。

这次事件暴露了一个残酷的现实：即便5G空口安全已经很强，但只要网络中有4G锚点的存在（NSA模式），攻击者可以利用"降级攻击"绕过5G的安全防护。

该运营商的应对措施包括三个方面：第一，加速SA（独立组网）部署，减少NSA锚点的依赖；第二，在核心网侧启用SUCI强制加密策略——拒绝所有以明文SUPI注册的UE；第三，在终端侧通过固件升级确保所有UE在5G网络中仅使用SUCI注册。经过这三个措施的组合，伪基站攻击在三个月内被完全遏制。

行业教训：这个案例说明，5G安全不是"开关一开就万事大吉"。安全是一个持续博弈的过程。运营商需要定期做安全审计和渗透测试，关注3GPP安全更新（SA3工作组），及时修补安全漏洞。尤其对于承载行业客户业务的切片和MEC平台，安全评估应该作为上线前的强制性检查项。

5G安全架构是一个庞大而精细的体系，从UE到核心网、从空口到传输网、从认证到授权，每一层都有对应的安全机制。安全不是5G的"枷锁"，而是5G承载千行百业的信用基础。我们从第19章一路走过切片、MEC、QoS、安全——5G高阶能力的四大支柱已经全部覆盖。从下一章开始，我们将进入第五部分工具与实战，首先学习MATLAB通信仿真——用代码把前面学的理论跑起来。
