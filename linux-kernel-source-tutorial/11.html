大家好，我是你们的讲师。上一章我们学习了进程调度——CPU如何在多个进程之间切换。但有一个问题我们还没回答：进程是怎么来的？又是怎么没的？

进程的生命周期由四个核心系统调用掌管：fork()、execve()、exit()、wait()。如果把进程比作一个"人"，那么fork就是"出生"（产生一个新实体），exec就是"整容"（更换外貌但继续是这个人），exit就是"死亡"（停止存在），wait就是"送葬"（清理遗物）。

在Linux 0.11中，这些机制的代码量加起来不到500行——但它们构成了所有多任务操作系统的基石。

下面这张表快速对比这四大系统调用的定位：

系统调用调用号角色返回值
fork()2创建新进程（"出生"）父进程返回子进程PID，子进程返回0
execve()11替换当前进程的代码和数据（"整容"）成功不返回，失败返回-1
exit()1终止当前进程（"死亡"）不返回
waitpid()7等待子进程退出并回收资源（"送葬"）返回子进程PID或出错返回-1

引言——fork、exec、exit、wait

在UNIX/Linux系统中，创建一个新进程的方式与其他操作系统（如Windows的CreateProcess）有本质不同。UNIX的做法是：先用fork()复制当前进程，然后用execve()将新进程替换为想要的程序。这种"分步执行"的设计，让UNIX的进程创建变得极其灵活——你可以在fork和exec之间修改新进程的环境，比如重定向文件描述符、设置用户ID等。

下面这张图展示了进程从创建到消失的完整过程：

进程生命周期全景图：

    ┌──────────┐
    │ 父进程   │
    │ PID=100  │
    └────┬─────┘
         │ fork()
         ▼
    ┌──────────┐
    │ 子进程   │  fork()返回后，父子进程几乎是"双胞胎"——
    │ PID=101  │  共享代码段，拥有独立的堆栈和数据段（写时复制）
    └────┬─────┘
         │ execve("/bin/ls", ...)
         ▼
    ┌──────────┐
    │ 新进程   │  execve()把当前进程替换为/bin/ls
    │ PID=101  │  PID不变，但代码/数据/堆栈全部替换
    └────┬─────┘
         │ exit(0)
         ▼
    ┌──────────┐
    │ ZOMBIE   │  进程已死，task_struct未释放
    │ PID=101  │  等待父进程调用wait()回收
    └────┬─────┘
         │ 父进程wait()
         ▼
    ┌──────────┐
    │ 已释放   │  task_struct被释放
    │ 再见     │  进程彻底消失
    └──────────┘
关键区别：fork创建的是调用者的"副本"，execve加载的是全新的程序。一个进程可以fork多次，但execve一次之后，它就"变成"了另一个程序。所以Shell的工作方式是：先fork出一个子进程，再在子进程中execve用户输入的命令。

fork()源码分析

fork()是系统调用号2（sys_fork），它的实现包括两个关键步骤：

/* kernel/fork.c — fork系统调用 */

int sys_fork(void)
{
    struct task_struct *p;
    int i;

    /* 1. 查找空闲的任务槽 */
    /* find_empty_process() 遍历task数组，返回第一个空槽的索引 */
    i = find_empty_process();
    if (i < 0)
        return -EAGAIN;  /* 进程数已达上限64 */

    /* 2. 分配并复制进程控制块 */
    p = (struct task_struct *) get_free_page();  /* 分配一页物理内存 */
    if (!p)
        return -EAGAIN;

    *p = *current;  /* 字节对字节复制task_struct！ */
    p->pid = get_pid();  /* 分配新PID */

    /* 3. 修改子进程的数据以区别于父进程 */
    p->counter = 0;         /* 子进程的初始counter */
    p->start_time = jiffies;
    p->tss.eax = 0;         /* fork()对子进程返回0 */

    /* 4. 复制页表（不复制物理页——写时复制） */
    copy_page_tables(p);    /* 复制页目录和页表 */

    /* 5. 复制文件相关数据 */
    p->pwd = current->pwd;  /* 共享当前目录 */
    if (p->pwd) p->pwd->i_count++;  /* 增加inode引用计数 */
    /* ... 复制文件描述符表 ... */

    /* 6. 设置TSS和LDT */
    set_tss_desc(gdt + (i << 1) + FIRST_TSS_ENTRY, &(p->tss));
    set_ldt_desc(gdt + (i << 1) + FIRST_LDT_ENTRY, &(p->ldt));

    /* 7. 设置子进程为就绪状态 */
    p->state = TASK_RUNNING;  /* 让子进程进入调度队列 */
    return p->pid;           /* 对父进程返回子进程PID */
}

这段代码有几个值得注意的设计：

  • get_free_page()：从主内存区分配一页（4KB）来存放新的task_struct。task_struct的大小约为1KB左右，用一整页有点浪费，但简化了页对齐的管理
  • *p = *current：通过结构体赋值复制整个task_struct，包括所有字段。这意味着子进程"继承"了父进程的大多数属性——打开的文件、信号处理方式、工作目录等
  • tss.eax = 0：这是fork()"一个调用两次返回"的关键！父进程fork()返回的是子进程的PID（在eax中），而子进程返回的是0。通过直接设置TSS中eax字段的值为0，当子进程被调度执行时，它看到的eax就是0
  • copy_page_tables()：只复制页表不复制物理页——这是写时复制的基础

写时复制（Copy-on-Write）

写时复制是操作系统历史上最重要的性能优化之一。如果不使用写时复制，每次fork()都要完整复制父进程的所有物理页——对于只占用几百KB内存的进程还好，但对于占用几百MB的进程，fork()的代价将大到不可接受。

写时复制的核心思想是：fork()时只把物理页标记为"只读"并共享，直到某一方真正写入时才复制物理页。

Linux 0.11已经实现了写时复制的基本机制：

/* kernel/fork.c — copy_page_tables：复制页表 */

int copy_page_tables(struct task_struct *p)
{
    int i;
    unsigned long *from_page_dir, *to_page_dir;

    /* 获取父进程和子进程的页目录基地址 */
    from_page_dir = (unsigned long *) ((current->tss.cr3 & 0xFFFFF000));
    to_page_dir = (unsigned long *) ((p->tss.cr3 & 0xFFFFF000));

    /* 遍历1024个页目录项 */
    for (i = 0; i < 1024; i++) {
        /* 如果页目录项有效，复制页表 */
        if (*from_page_dir & 1) {  /* Present位为1 */
            /* 分配新页表，复制旧页表内容 */
            unsigned long *pg_table;
            pg_table = (unsigned long *) get_free_page();

            /* 复制页表项 */
            memcpy(pg_table, (void *)(*from_page_dir & 0xFFFFF000), PAGE_SIZE);

            /* 关键！将新页表项的R/W位清0（写保护） */
            /* 这样父子进程共享物理页，但都只能读不能写 */
            /* 写入时触发缺页异常，do_wp_page()再复制 */
            unsigned long *pte;
            int j;
            for (j = 0; j < 1024; j++)
                pte[j] &= ~2;    /* 清除R/W位（bit 1），设为只读 */

            /* 设置子进程的页目录项 */
            to_page_dir[i] = ((unsigned long) pg_table) | 7;

            /* 父进程的页表项也要设为只读！ */
            for (j = 0; j < 1024; j++)
                from_page_dir[i][j] &= ~2;

            /* 刷新TLB使修改生效 */
            invalidate();
        }
    }
    return 0;
}

关键步骤提炼一下：

  1. fork()只复制页表，不复制物理页
  2. 父子进程的页表项中的R/W位都被设置为0（只读）
  3. 任何一方尝试写入共享页面时，CPU触发写保护异常（Page Fault，vector 14）
  4. 缺页异常处理函数do_wp_page()判断这是写时复制场景，分配新物理页，复制数据，更新页表
效率对照：假设一个进程占用了4MB物理内存（1000个页）。没有写时复制的fork需要：复制1000个物理页 = 4MB数据拷贝。有了写时复制的fork只需要：复制页表（一个页目录+4个页表 = 20KB） + 设置写保护。差距是约200倍！而且如果fork后立即exec，没有写时复制的系统还要再释放这些物理页——做了大量的无用功。

execve()加载程序

execve()是系统调用号11。它的作用：用一个新的可执行程序替换当前进程。PID不变，但代码段、数据段、堆栈全部替换。

Linux 0.11支持的可执行文件格式是a.out（Assembler OUTput），后来才过渡到ELF。a.out的头部结构很简单：

/* include/linux/a.out.h — a.out可执行文件头 */

struct exec {
    unsigned long a_info;   /* 包含魔数和标志位 */
    unsigned long a_text;   /* 代码段长度 */
    unsigned long a_data;   /* 数据段长度 */
    unsigned long a_bss;    /* BSS段长度（未初始化数据） */
    unsigned long a_syms;   /* 符号表长度 */
    unsigned long a_entry;  /* 程序入口地址 */
    unsigned long a_trsize; /* 代码段重定位表长度 */
    unsigned long a_drsize; /* 数据段重定位表长度 */
};

do_execve()的核心流程：

/* kernel/execve.c — do_execve 核心流程（简化） */

int do_execve(unsigned long *eip, long tmp, char *filename, char **argv, char **envp)
{
    struct exec ex;
    unsigned long page;

    /* 1. 打开可执行文件 */
    if (sys_open(filename, O_RDONLY) < 0)
        return -ENOENT;

    /* 2. 读取文件头 */
    read_exec_header(fd, &ex);

    /* 3. 检查魔数是否合法 */
    if (ex.a_info != 0x0064011B && ex.a_info != 0x00640103)
        return -ENOEXEC;

    /* 4. 释放旧进程的页表 */
    free_page_tables(current->tss.cr3);  /* 释放所有物理页！ */

    /* 5. 为新的可执行文件分配页表 */
    /*    将代码段加载到进程线性地址空间 */
    page = get_free_page();
    /* ... 从文件读取代码段和数据段 ... */

    /* 6. 设置新程序的栈顶 */
    current->brk = ex.a_text + ex.a_data + ex.a_bss;
    current->start_stack = setup_arg_pages(argv, envp);  /* 设置参数和环境变量栈 */

    /* 7. 修改TSS中的EIP为新程序的入口 */
    eip[0] = ex.a_entry;   /* 设置返回用户态后的第一条指令地址 */
    /* 修改CS:EIP，使得从内核态返回用户态时直接执行新程序 */

    /* 8. 关闭可执行文件并返回 */
    sys_close(fd);
    return 0;
}

execve()最有趣的一点是：如果它成功了，它永远不会返回——因为当前进程的代码段已经被完全替换了。如果execve()返回了一个错误，说明执行失败（文件不存在、格式不对、权限不够）。所以用户程序通常这样写：

/* 典型的fork+exec模式 */
pid_t pid = fork();
if (pid == 0) {
    /* 子进程 */
    execve("/bin/ls", argv, envp);
    /* 如果execve返回了，说明出错了 */
    exit(1);
} else {
    /* 父进程 */
    wait(pid);
}

exit()与wait()

exit()是进程的"死亡"系统调用。Linux 0.11中的do_exit()负责清理进程资源：

/* kernel/exit.c — 进程退出 */

int do_exit(long code)
{
    int i;

    /* 1. 释放进程占用的物理内存 */
    free_page_tables(current->tss.cr3);  /* 释放所有用户空间页 */

    /* 2. 关闭所有打开的文件 */
    for (i = 0; i < NR_OPEN; i++)
        if (current->filp[i])
            sys_close(i);

    /* 3. 释放当前目录和根目录的inode */
    iput(current->pwd);
    current->pwd = NULL;
    iput(current->root);
    current->root = NULL;

    /* 4. 释放子进程（将子进程托付给init，PID=1） */
    for (i = 0; i < NR_TASKS; i++)
        if (task[i] && task[i]->pgrp == current->pid) {
            task[i]->pgrp = task[1]->pgrp;   /* 收养 */
            /* ... 设置新的父进程为init ... */
        }

    /* 5. 向父进程发送SIGCHLD信号 */
    current->signal |= (1<<(SIGCHLD-1));
    /* 通知父进程子进程已退出 */

    /* 6. 状态设为ZOMBIE——保留task_struct但不运行 */
    current->state = TASK_ZOMBIE;
    current->exit_code = code;   /* 保存退出码 */

    /* 7. 触发调度，让出CPU */
    schedule();
    /* 注意：这里永远不会回来——进程已经ZOMBIE了 */
    return -1;
}

最核心的一步是第6步：状态设为TASK_ZOMBIE。这意味着进程"已经死了，但尸体还没被收走"——task_struct没有被释放，因为它里面还保留着退出码，等待父进程来取。

父进程通过wait()或waitpid()来"收尸"：

/* kernel/exit.c — waitpid */

int sys_waitpid(pid_t pid, unsigned long *stat_addr, int options)
{
    struct task_struct **p;
    int flag;

    while (1) {
        flag = 0;
        for (p = &LAST_TASK; p > &FIRST_TASK; --p) {
            if (!*p || *p == current)
                continue;   /* 跳过自己和空槽 */

            if (pid != -1 && (*p)->pid != pid && ...)
                continue;   /* 不匹配等待的PID */

            flag = 1;       /* 找到了子进程 */

            if ((*p)->state == TASK_ZOMBIE) {
                /* 找到僵尸子进程！收集退出码 */
                unsigned long tmp = (*p)->exit_code;
                /* 释放task_struct */
                free_page((unsigned long)*p);
                task[p - task] = NULL;
                return tmp;
            }
        }
        if (!flag)   /* 没有子进程 */
            return -ECHILD;
        /* 子进程还没退出，当前进程进入睡眠 */
        current->state = TASK_INTERRUPTIBLE;
        schedule();
        /* 醒来后再检查一遍 */
    }
}
注意：waitpid()中的current->state = TASK_INTERRUPTIBLE会让父进程进入可中断睡眠。如果父进程在等待期间收到了信号，它会被唤醒并返回-EINTR。这是UNIX系统调用的典型模式——"如果无法立即完成，就主动让出CPU"。

孤儿进程与init收养

如果父进程在子进程之前退出，会发生什么？子进程会成为孤儿进程（orphan process）。在Linux 0.11中，所有孤儿进程会被init进程（PID=1）自动收养：

回顾do_exit()中的这段代码：

/* 在父进程退出时，将子进程托付给init */
for (i = 0; i < NR_TASKS; i++)
    if (task[i] && task[i]->pgrp == current->pid) {
        task[i]->pgrp = task[1]->pgrp;   /* 收养为init的子进程 */
    }

init进程（PID=1）是系统启动时创建的第一个用户进程，它的代码中有一个永真循环——不断调用wait()来清理僵尸子进程。所以孤儿进程最终会被init收养并清理，不会形成"永久性孤儿"。

踩坑故事——僵尸进程泄漏

这是一个每个写C程序的人几乎都遇到过的经典问题。你会发现这个故事的影子出现在几乎所有UNIX/Linux教材的"常见错误"章节里。

某位开发者写了一个网络服务程序，主进程不断accept连接，每次接收到新连接就fork一个子进程去处理。看起来一切正常——服务上线后稳定运行了好几天。

但几天后，运维报告：服务进程数达到上限，无法再fork新的子进程。检查ps输出，发现系统里有上百个<defunct>进程——也就是僵尸进程。

问题出在哪里？

父进程fork子进程后，没有调用wait()或waitpid()来回收子进程的退出状态。子进程处理完请求exit()之后，状态变为TASK_ZOMBIE。父进程没有wait()，子进程的task_struct就永远得不到释放——它占用的进程槽位（task数组的64个位置之一）一直被占用着。

Linux 0.11的task数组只有64个槽位。当僵尸进程占满所有槽位后，fork()的find_empty_process()找不到空槽，返回-EAGAIN。系统再也无法创建新进程。

/* 错误的代码模式 */
while (1) {
    int fd = accept(listen_fd, ...);
    if (fork() == 0) {
        /* 子进程：处理请求 */
        handle_request(fd);
        exit(0);
    }
    /* 父进程：没有wait()！子进程变僵尸 */
}

/* 正确的代码模式 */
while (1) {
    int fd = accept(listen_fd, ...);
    if (fork() == 0) {
        handle_request(fd);
        exit(0);
    } else {
        wait(NULL);  /* 等待子进程结束——但这是同步阻塞的！ */
    }
}

/* 更好的方式：使用SIGCHLD信号异步回收 */
/* 设置SIGCHLD的处理函数为SIG_IGN或自定义处理 */
signal(SIGCHLD, SIG_IGN);  /* 系统自动回收，不产生僵尸 */

注意正确的代码模式也有问题——如果父进程同步wait()，就无法同时处理新连接了。所以实际工程中通常使用信号机制（SIGCHLD）或双fork技巧来异步回收子进程。

这个故事的教训是：每个fork()都要对应一个wait()，否则就会产生僵尸进程。在Linux 0.11的有限进程槽位下（64个），僵尸进程很容易就会把资源耗尽。这个教训也延续到了现代Linux——只是现代系统的进程上限更高（默认数万），但原理完全一样。

理解了一个进程的"生老病死"，下一章我们将学习进程之间的一种重要的异步通信方式——信号机制。信号是UNIX世界中最古老、最基础的进程间通信方式之一，它让进程能够"不经意间"被通知某些事件的发生。
