第20章 流水线与数据并行

上一章讲了张量并行——单节点内多 GPU 协作。但遇到 405B 这种单节点根本装不下的超大模型怎么办？或者想用 8 张 GPU 服务更多并发请求？这就要用到流水线并行（Pipeline Parallelism, PP）和数据并行（Data Parallelism, DP）。

20.1 流水线并行：按层切分

流水线并行不切分单个层，而是把模型按层分段，不同 GPU（或节点）负责不同的层：

Llama-3-405B, 32 层, PP=4 (4 个节点):

节点 1 (GPU 0-7): Layers 0-7    ← TP 在节点内处理这 8 层
节点 2 (GPU 0-7): Layers 8-15
节点 3 (GPU 0-7): Layers 16-23
节点 4 (GPU 0-7): Layers 24-31

请求流程:
Input → 节点1 (Layer 0-7) → 节点2 (Layer 8-15) → 节点3 (Layer 16-23) → 节点4 (Layer 24-31) → Output

通信只发生在层段边界（把中间激活传给下一个节点），所以对带宽要求比 TP 低得多——25GbE 以太网就够了。

20.2 PP + TP 混合：超大模型的标配

# 2 节点 × 8 GPU = 16 GPU，运行 Llama-3-405B
vllm serve meta-llama/Llama-3-405B \
    --tensor-parallel-size 8 \      # 节点内 8 GPU TP
    --pipeline-parallel-size 2 \    # 2 节点 PP
    --distributed-executor-backend ray  # 多节点需要 Ray

# 显存分配:
# 每节点 8 GPU TP 拆分 16 层 → 每 GPU ~50GB 权重
# 总 KV cache 分布在所有 GPU 上

20.3 数据并行：吞吐优先

数据并行不切分模型，而是在不同 GPU 上运行完整模型副本，各自处理不同的请求。适合"模型能放进单 GPU，但想要更高吞吐"的场景：

# 8 GPU: 4 个独立引擎，每个用 2 GPU (TP=2)
vllm serve meta-llama/Llama-3-70B \
    --data-parallel-size 4 \
    --tensor-parallel-size 2

# 架构:
├── Engine 1 (GPU 0,1): 完整模型副本 + 独立 KV Cache
├── Engine 2 (GPU 2,3): 完整模型副本 + 独立 KV Cache
├── Engine 3 (GPU 4,5): 完整模型副本 + 独立 KV Cache
└── Engine 4 (GPU 6,7): 完整模型副本 + 独立 KV Cache

# 请求被分发到不同 Engine，吞吐 ≈ 4× 单 Engine

20.4 三种并行策略对比

策略切分方式通信量适用局限
TP每层权重切分高（每层 2×AllReduce）单节点多 GPU跨节点性能差
PP按层切分低（层边界传激活）跨节点大模型有流水线气泡
DP完整模型副本零（不通信）高吞吐 8B-70B每 GPU 需完整显存

20.5 MoE 模型的特殊并行策略

MoE（Mixture of Experts）模型如 DeepSeek-V3、Mixtral 有额外的并行维度：

Expert Parallelism (EP):
MoE 层有 8/64 个 expert，不同 GPU 负责不同 expert
一个 token 只被路由到少数 expert（如 top-2）
→ 通信是 All-to-All（而非 All-Reduce）
→ 总通信量小得多

vLLM 原生支持 MoE 模型的 EP，配置同 TP 参数

20.6 选择策略的决策树

模型能放进 1 GPU 吗？
├── 是 → 不需要并行，或者用 DP 提升吞吐
└── 否 → 模型能放进 1 个节点（多 GPU）吗？
    ├── 是 → TP=N（N = 需要的 GPU 数）
    └── 否 → PP × TP 混合
            ├── PP = 节点数
            └── TP = 每节点 GPU 数

经验之谈：大部分团队接触不到 405B 级别——70B 模型用 2-4 GPU TP 就够了。DP 更适合常见的 8B-13B 模型 + 多 GPU 做高吞吐服务。先确认你的瓶颈是显存不够还是吞吐不够——前者用 TP，后者用 DP。

并行策略搞清楚了。下一章是"快进键"——推测解码，不增加 GPU 也能让生成提速 2-6 倍。
