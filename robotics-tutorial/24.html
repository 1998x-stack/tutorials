第24章 嵌入式与实时系统

大家好，我是你们的讲师。如果你之前一直在Linux桌面环境或者仿真中开发机器人算法，现在是时候聊聊"硬件上的那些事"了。你可能有过这样的经历：PID控制器在Gazebo中跑得完美无缺，驱动频率稳稳地维持在1kHz——但烧录到MCU上之后，同样的代码跑出来控制循环有时800Hz，有时1.2kHz，机器人也跟着"忽快忽慢"。

欢迎来到实时系统的世界。在这里，"快"不是最重要的，确定性地快才是。一个机器人控制系统，宁可稳定地在990Hz运行，也不要在大部分时间跑1200Hz但偶尔掉到200Hz。本章我们来探讨嵌入式平台的选型、FreeRTOS的任务调度、micro-ROS桥接MCU和ROS 2，以及工业通信总线的选择，帮你搭建一个真正"硬实时"的机器人底层系统。

嵌入式平台选型

机器人底层硬件的选择直接影响着你的系统能实现什么样的控制性能。下面我梳理了五种最常见的选择，各有各的战场，不存在"最好的平台"，只有"最适合的平台"。

平台 核心架构 算力 实时性 适用场景 典型价格
STM32 Cortex-M4/M7 低（~400 DMIPS） 硬实时 电机控制、关节伺服、底层I/O $5~$50
ESP32 Xtensa LX6/RISC-V 低（~600 DMIPS） 硬实时 IoT传感节点、WiFi/BLE通信 $3~$15
Raspberry Pi Cortex-A72/A76 中（~10 GFLOPS） 软实时（Linux） ROS 2上层、视觉处理、导航 $35~$100
NVIDIA Jetson Cortex-A78C + GPU 高（~100 TOPS AI） 软实时（Linux + PREEMPT_RT） AI推理、视觉SLAM、VLA模型 $200~$1,500
FPGA 可编程逻辑 并行硬件级 硬实时（ns级确定性） 超高带宽I/O、自定义通信协议 $50~$5,000

在典型的机器人架构中，这几种平台是协同工作的：STM32负责关节级别的电机伺服（1kHz硬实时控制环），ESP32负责无线传感节点的数据采集，Raspberry Pi运行ROS 2的上层逻辑（导航、规划），Jetson处理GPU密集的AI推理任务。各层之间通过工业通信总线连接，形成完整的计算层级。

选型建议：如果你做的是一台从零开始的机器人，我建议从STM32 + Raspberry Pi的组合入手。STM32处理电机控制和传感器驱动，RPi跑ROS 2。这对组合性价比极高，社区资源也非常丰富。不要一开始就上Jetson或FPGA——除非你确定自己真的需要GPU算力或ns级的确定性。

FreeRTOS基础

FreeRTOS是目前嵌入式领域最主流的实时操作系统（RTOS），在STM32、ESP32等MCU上广泛使用。它的核心价值在于提供了抢占式优先级调度（Preemptive Priority-based Scheduling），让你能够精确控制每个任务的执行时机和周期。

FreeRTOS的任务调度基于优先级：高优先级任务就绪时会立即抢占低优先级任务的CPU时间。这是实现"1kHz控制环"的关键——你把控制任务设为最高优先级，它就能保证每1ms被调度一次，不受其他任务（如串口通信、LED闪烁）的干扰。

下面是一个典型的FreeRTOS电机控制任务代码，展示了如何实现1kHz的硬实时控制环：

#include "FreeRTOS.h"
#include "task.h"
#include "queue.h"

// 电机控制任务：1kHz硬实时控制环
void motor_control_task(void *pvParameters)
{
    TickType_t xLastWakeTime;
    const TickType_t xFrequency = pdMS_TO_TICKS(1);  // 1ms = 1kHz

    // 初始化任务的时间基准
    xLastWakeTime = xTaskGetTickCount();

    while (1)
    {
        // 1. 读取编码器位置和速度
        int32_t position = read_encoder();
        int32_t velocity = read_velocity_estimate();

        // 2. 接收来自ROS 2的目标指令（通过队列）
        JointCommand cmd;
        if (xQueueReceive(motor_cmd_queue, &cmd, 0) == pdTRUE)
        {
            // 3. PID控制计算
            float error = cmd.target_position - position;
            float derivative = -velocity;  // 微分项来自速度反馈
            integral += error * dt;
            float output = Kp * error + Ki * integral + Kd * derivative;

            // 4. 限幅并输出到驱动器
            output = clamp(output, -CURRENT_LIMIT, CURRENT_LIMIT);
            set_motor_current(output);
        }

        // 5. 精确延迟到下一个1ms tick
        vTaskDelayUntil(&xLastWakeTime, xFrequency);
    }
}

这个代码的关键在于vTaskDelayUntil——它指定的是"下一次wake up的绝对时间"，而不是"再等多久"。两者的区别在于：如果一次执行因为某种原因耗时超过了1ms，vTaskDelayUntil会立即进入下一次循环而不会额外等待，从而防止了时序偏差的累积。

除了任务调度，FreeRTOS还提供了丰富的IPC（Inter-Process Communication）机制：队列（Queue）用于任务间的数据传递，二值信号量（Binary Semaphore）用于中断与任务间的同步，互斥量（Mutex）用于保护共享资源。在电机控制场景中，通常的模式是：中断服务程序（ISR）从编码器读取数据后释放一个信号量，控制任务等待信号量后执行PID运算——这样保证编码器读数是最新的。

micro-ROS

传统上，MCU上是没有ROS接口的——这意味着你不能直接在STM32上发布/订阅ROS 2的topic。你得在MCU和RPi之间通过串口或CAN传递自定义协议，再在RPi上写一个bridge node做协议转换。这不仅增加了开发工作量，还引入了一个额外的故障点。

micro-ROS就是为了解决这个问题而生的。它把ROS 2的发布-订阅机制直接带到了MCU上，让你的STM32蓝牙扩展板可以像一个标准的ROS 2节点一样发布传感器数据、接收控制指令。魔法背后是一个叫做DDS-XRCE（eXtremely Resource Constrained Environment）的协议——它是标准DDS的轻量化版本，专门为内存和算力受限的嵌入式设备设计。

micro-ROS的架构分为两部分：Client运行在MCU上（通常占用不到50KB的Flash），Agent运行在RPi或Jetson上（作为标准ROS 2节点）。Client和Agent之间通过UART、SPI、WiFi或以太网通信，Agent将Client的数据桥接到完整的ROS 2 DDS网络中。

下面是micro-ROS在STM32上的一个发布者示例，实现了将MCU侧编码器数据直接发布为ROS 2的JointState消息：

#include <micro_ros_platformio.h>
#include <rclc/rclc.h>
#include <sensor_msgs/msg/joint_state.h>

rcl_publisher_t joint_state_publisher;
sensor_msgs__msg__JointState joint_state_msg;

void setup() {
    // 初始化micro-ROS（通过UART连接Agent）
    set_microros_serial_transports(Serial);
    delay(2000);

    rclc_support_t support;
    rclc_allocator_t allocator = rclc_get_default_allocator();
    rclc_init_support(&support, 0, NULL, allocator);

    rcl_node_t node = rclc_get_zero_initialized_node();
    rclc_node_init_default(&node, "stm32_motor_node", "", &support);

    // 创建JointState发布者
    rclc_publisher_init_default(
        &joint_state_publisher,
        &node,
        ROSIDL_GET_MSG_TYPE_SUPPORT(sensor_msgs, msg, JointState),
        "joint_states");
}

void loop() {
    // 填充消息并发布
    joint_state_msg.header.stamp.sec = get_seconds();
    joint_state_msg.header.stamp.nanosec = get_nanoseconds();
    joint_state_msg.position.data = read_encoder_position();
    joint_state_msg.velocity.data = read_encoder_velocity();

    rcl_publish(&joint_state_publisher, &joint_state_msg, NULL);

    // 1kHz发布频率
    delay(1);
}

关键洞察：micro-ROS让MCU成为ROS 2的一等公民。你不再需要在底层和上层之间写"胶水代码"——整个系统从传感器到决策层都用统一的ROS 2接口通信。这不仅减少了代码量，更重要的是降低了延迟和出错概率。2026年，micro-ROS已经在STM32G4、ESP32、甚至某些RISC-V平台上稳定运行。

工业通信总线

在机器人的内部，各组件之间的通信延迟直接决定了你能达到的控制性能。你选择的通信总线会影响整个系统的实时性、可靠性和可扩展性。下面是几种主流工业通信总线的对比：

总线 最大速率 同步精度 拓扑 最佳场景 典型延迟
EtherCAT 100 Mbps < 1 μs 环形/线形 多轴同步、高速伺服 < 10 μs
CAN-FD 8 Mbps < 10 μs 总线形 汽车、工业关节模组 ~100 μs
Modbus TCP 100 Mbps ~1 ms 星形 PLC对接、传统工厂设备 ~1-5 ms
EtherNet/IP 100 Mbps ~100 μs 星形 离散制造、传感器网络 ~100 μs-1 ms

对于高性能机器人应用，EtherCAT是当之无愧的首选。它采用"数据处理在硬件层完成"（on-the-fly processing）的机制——每个从站设备在数据帧经过时取走和填入自己的数据，整个过程几乎没有软件开销。分布式时钟（Distributed Clocks）技术可以让上百个轴的同步误差控制在1微秒以内。这也是为什么几乎所有高端机器人（KUKA、ABB、FANUC）的内部总线都是EtherCAT。

CAN-FD（CAN with Flexible Data-Rate）是传统CAN的升级版，在保持CAN的可靠性和鲁棒性的同时，将数据段速率提升到了8 Mbps。CAN-FD在关节模组级别的通信中很常见——每个关节内部的MCU通过CAN-FD上报位置、速度和温度，接收力矩指令。虽然速率远不及EtherCAT，但CAN-FD的硬件成本极低、抗干扰能力强，非常适合分布式关节架构。

踩坑故事：我们在一个四足机器人项目里遇到过这样的问题——用CAN总线连接12个关节模组，控制频率设到了2kHz。在大部分工况下都正常，但当机器人做跳跃动作时，所有关节同时需要传输大量数据，CAN总线的仲裁机制导致高优先级报文（紧急停止）被低优先级数据阻塞了。排查了整整两周才意识到是总线负载率太高（超过了70%），最后不得不降回1kHz控制频率并启用CAN-FD的高速率模式才解决。教训是：设计通信架构时一定要计算峰值负载，而不是平均负载。

实时控制实战

现在我们把这些知识点串起来，搭建一个真正的1kHz硬实时控制系统。从传感器读数到控制指令输出，整个链路的每一步都需要精心设计以确保确定性。

最关键的设计原则是中断驱动（Interrupt-Driven Sensing）。不要用轮询方式读传感器——轮询的时序取决于代码路径，而代码路径的执行时间是不确定的。正确的方式是将传感器数据准备好后通过中断通知CPU。在FreeRTOS中，ISR里只做最少的操作（通常只是读取硬件寄存器和释放信号量），真正的数据处理放在任务中完成。

对于运行Linux的处理器（Raspberry Pi、Jetson），如果你需要硬实时性能，有两个选择。一是使用Linux PREEMPT_RT内核补丁——它将Linux从软实时提升到接近硬实时的水平，中断延迟可以控制在50微秒以内。二是使用裸金属（Bare-Metal）方案——完全不跑操作系统，直接在硬件上轮巡。裸金属的延迟最低、确定性最高，但开发效率也最低。

最后，无论你选择哪种方案，一定要做延迟测量和抖动分析。用GPIO引脚在控制循环的开始和结束处输出脉冲，用示波器测量脉冲宽度的变化，就能知道你的控制循环到底稳不稳。如果抖动超过了控制周期的10%，说明你的实时设计还有改进空间。

嵌入式和实时系统是机器人的"神经系统"——它不直接决定你的算法有多聪明，但它决定了你的算法能不能在正确的时间执行正确的动作。MCU和RTOS的选型、micro-ROS和通信总线的配置，这些都是在真机部署之前必须想清楚的事情。下一章，我们把这些硬件和软件能力全部整合到一起，深入探讨真机部署与集成——从Sim-to-Real的迁移检查清单，到系统集成测试，再到安全规范和CI/CD流水线，让你的机器人真正走向真实世界。
