Switch Transformer的设计哲学：少即是多

大家好，我是你们的讲师。前几章我们花了大量篇幅讨论稀疏门控MoE——从1991年Jacobs提出概念，到2017年Shazeer将其引入深度学习，再到Token-Choice vs Expert-Choice的路由博弈和负载均衡的种种技巧。信息量确实不小。但如果你问我："MoE真正在工业界起飞是从哪篇论文开始的？"我的回答会毫不犹豫：2021年的Switch Transformer。

Switch Transformer由Google的William Fedus、Barret Zoph和Noam Shazeer等人提出，发表在JMLR上。这篇论文的核心贡献可以用一句话概括：将Top-2路由简化为Top-1，然后在万亿参数规模上验证了可行性。你可能觉得"从Top-2降级到Top-1"是一种倒退——毕竟上一章我们刚讨论了Top-2比Top-1在表达能力上的优势。但Switch Transformer告诉我们：工程实践中的"足够好"往往比理论上的"最优"更有价值。少即是多，简单即是强大。

让我用一个亲身经历来说明"简化"的意义。2021年我刚读完这篇论文时，正带队做一个多语言翻译模型。我们照搬了Sparsely-Gated MoE的Top-2路由方案，每天晚上盯着训练日志里的aux loss曲线，看着那些微小的负载不均衡波动焦虑不已。直到有一天，我试着注释掉了第二个专家的加权求和，只保留Top-1——奇迹发生了：训练速度提升了40%，而BLEU值几乎没有下降！那一刻我才真正理解Switch Transformer的核心洞察：很多时候，多一个专家带来的收益被通信和负载均衡的成本完全抵消了。

🔑 关键洞察： Switch Transformer 的核心贡献不在于算法创新，而在于"做减法"的工程智慧。它将 Top-2 路由简化为 Top-1，大幅降低通信量和计算复杂度，同时通过扩大专家数量来弥补单个专家表达能力的损失。这个取舍在万亿参数规模下被证明是极其成功的。

Top-1路由：一个Token只选一个专家

Switch Transformer的路由策略是整篇论文中最引人注目的设计选择。在Sparsely-Gated MoE中，每个token路由到top-2个专家，然后对两个专家的输出加权求和。Switch Transformer将其简化为：每个token只路由到得分最高的一个专家。

这个看似简单的改动，带来了几个立竿见影的好处：

  • 计算量减半：每个token只需执行一次专家FFN前向，而不是两次。
  • 通信量减半：在分布式设置中，all-to-all通信的数据量也相应减少。
  • 负载均衡更简单：Top-2路由需要确保两个专家之间的负载都均衡，而Top-1只需要关注每个token的唯一分配，路由冲突概率大幅下降。
  • 路由器更易训练：只有一个专家被选中，梯度信号更清晰，路由器的训练更加稳定。

Switch Transformer路由的数学表达如下：

# Switch Transformer Top-1 路由前向
def switch_forward(x, experts, router):
    # x: [batch, seq_len, d_model]  输入张量
    # experts: [E] 个 FFN 专家
    # router: 线性层 + softmax

    batch_size, seq_len, d_model = x.shape
    x_flat = x.view(-1, d_model)             # [N, d_model], N = batch * seq_len

    # 1. 计算路由得分
    router_logits = router(x_flat)           # [N, E]
    router_probs = softmax(router_logits, dim=-1)

    # 2. Top-1 选择：只需取最大值的索引
    expert_weights, expert_indices = top_1(router_probs)  # 各 [N, 1]

    # 3. 专家容量限制：每个专家最多处理 capacity 个 token
    capacity = int(capacity_factor * N / E)

    # 4. 按专家分组，丢弃超出容量的 token
    dispatch_mask = expert_mask(expert_indices, E, capacity)
    # dispatch_mask[i, e, pos] = 1 表示 token i 被分配到专家 e 的第 pos 个位置

    # 5. 稀疏计算
    output_flat = zeros_like(x_flat)
    for e in range(E):
        # 取出分配到专家 e 的 token
        selected = dispatch_mask[:, e, :].nonzero()
        if selected.numel() == 0:
            continue
        token_indices = selected[:, 0]
        expert_input = x_flat[token_indices]
        expert_output = experts[e](expert_input)
        output_flat[token_indices] += expert_output * expert_weights[token_indices]

    return output_flat.view(batch_size, seq_len, d_model)

注意上面代码中的capacity_factor参数——它是Switch Transformer中至关重要的超参数。我们接下来详细分析它。

Capacity Factor实战调参

Capacity Factor（容量因子）是Switch Transformer引入的一个关键概念，直接影响模型的训练质量和效率。它的定义很简单：

capacity = capacity_factor × (total_tokens / num_experts)

其中total_tokens是一个batch中的总token数，num_experts是专家数量。capacity决定了每个专家最多能处理多少个token。如果路由分配超过capacity，超出的token就会被丢弃（dropped），不参与当前层的计算。

⚠️ 重要概念： Token Drop 是 Switch Transformer 中的一个关键机制。当某个专家被分配的 token 数量超过其容量时，超出部分的 token 会直接被丢弃——它们不会通过该 MoE 层，而是通过残差连接直接传递到下一层。这意味着模型可能会丢失信息！因此 capacity factor 不能设得太小，否则大量 token 被丢弃会损害模型质量；但也不能设得太大，否则每个专家处理过多 token，稀疏激活的优势就丧失了。
Capacity Factor 特点 Token Drop率 推荐场景
1.0 严格负载均衡，每个专家恰好处理平均token数 高（10-30%） 负载极度不均衡时慎用
1.25 略微冗余，多数场景下性价比最高 中（2-5%） 通用推荐值
1.5 充足余量，token drop极少 低（<1%） 质量敏感型任务
2.0 过大的容量，接近稠密计算 几乎为0 调试阶段或小规模试验

实战中有几个重要的调参经验。第一，capacity factor和辅助损失权重需要协同调整：如果你把aux_loss权重设得很大，路由会更均衡，capacity factor就可以适当降低。第二，训练早期使用较大的capacity factor：模型训练初期路由还在学习，分配非常不均匀，此时建议用1.5甚至2.0，等训练稳定后再逐步降低到1.25。第三，结合batch size一起调：batch越大，统计上路由越均匀，capacity factor可以越低。

提到调参，我必须分享一次惨痛的教训。曾经为了追求极致效率，我直接把capacity factor设为1.0，想着"正好每个专家处理等量的token，多完美"。结果训练了10万步，模型loss死活降不下去。一排查发现token drop率高达35%——三分之一的信息被丢掉了！这相当于每三层就有一层的信息完全丢失，模型怎么可能学得好。后来把capacity factor调到1.25，drop率降到3%，loss曲线立刻正常了。我在这上面浪费了整整一周的GPU时间，血泪教训：永远不要在capacity factor上贪心。

💡 实战建议： 对于大多数 MoE 训练任务，建议从 capacity_factor=1.5 开始，观察 token drop 率，然后逐步降低。如果在某次降低后出现 loss 震荡或 perplexity 升高，回退到上一个值。一个健康的训练过程应该保持 token drop 率在 1-3% 以下。

万亿参数模型训练实录

Switch Transformer最令人震撼的成就是：它成功训练了一个1.6万亿参数的模型（Switch-C）。这个规模在当时是空前的——即使放在今天，万亿参数模型也是极少数团队能够触及的领域。

让我们看看他们是怎么做到的。Switch Transformer是在Google的TPU v3集群上训练的，使用了2048个TPU核心。关键配置如下：

配置项 Switch-C (1.6T) Switch-XXL (395B) Switch-Base (83B)
总参数量 1.6T 395B 83B
活跃参数 ~6.2B ~3.7B ~1.7B
专家数量 2048 256 64
Top-1 路由 是 是 是
Capacity Factor 1.25 1.25 1.25
训练语料 C4 (800GB) C4 C4
硬件 2048 TPUv3 512 TPUv3 128 TPUv3

一个特别值得注意的细节是：Switch-C总共1.6T参数，但每个token只激活约6.2B参数——激活率不到0.4%！相比之下，稠密T5-XXL只有11B参数，但100%都被激活。Switch-C的总参数是T5-XXL的145倍，但计算量仅为其约56%。这就是稀疏MoE的魔力。

Switch Transformer在训练中还引入了一项重要改进：流水线并行（Pipeline Parallelism）。与Sparsely-Gated MoE的数据并行+模型并行不同，Switch Transformer将不同的Transformer层分配到不同的设备上，形成流水线。MoE层中的2048个专家分布在所有TPU核心上，每个核心负责一小部分专家。当token被路由到其他核心上的专家时，通过all-to-all通信传输数据。

# Switch Transformer 训练流水线示意
# 假设 4 个 TPU 核心，每个核心负责 512 个专家中的 128 个

TPU 0:  专家 [0-127]   +  Layer 1 Attention  +  Layer 1 MoE(部分)
TPU 1:  专家 [128-255] +  Layer 1 Attention  +  Layer 1 MoE(部分)
TPU 2:  专家 [256-383] +  Layer 2 Attention  +  Layer 2 MoE(部分)
TPU 3:  专家 [384-511] +  Layer 2 Attention  +  Layer 2 MoE(部分)

# 前向传播流程：
# Step 1: TPU 0/1 处理 Layer 1 Attention，TPU 2/3 空闲
# Step 2: Layer 1 MoE All-to-All 通信，TPU 2/3 开始 Layer 2 Attention
# Step 3: 流水线填满，所有核心全速运行

这个设计的精妙之处在于：MoE层的通信可以和Attention层的计算重叠，最大限度减少通信等待时间。Switch Transformer报告的训练效率达到了惊人的86%的模型并行利用率——在2048个TPU上，这个数字是非常优秀的。

最终的结果令人振奋：Switch-C在C4语料上的perplexity达到了9.56，远超同计算预算的T5-XXL（11.33）。在SuperGLUE等NLU benchmark上，Switch Transformer也取得了当时最优的成绩。更重要的是，这篇论文彻底打消了工业界对"MoE能否训到万亿参数"的疑虑——从此以后，万亿参数不再是一个幻想。

🔑 关键启示： Switch Transformer 告诉我们一个深刻的工程哲理：有时候，"放弃"才是更好的策略。放弃 Top-2 改用 Top-1，放弃了理论上的表达能力上限，换来了实际训练中的稳定和高效。在 AI 工程中，理论最优和实际最优往往是两回事。Switch Transformer 的选择为后续所有 MoE 工作树立了一个榜样：做务实的简化，而不是追求理论上的完美。

Switch Transformer为MoE的工业级部署铺平了道路，但它也在分布式训练方面留下了不少待解决的问题。其中最核心的挑战是：如何高效地在千卡集群上调度MoE的计算和通信？这一问题的答案正是我们下一章的主题——GShard。
