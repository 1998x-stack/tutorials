第二十三章：框架对比 — LangChain vs LlamaIndex

大家好，我是你们的讲师。今天我们要聊一个在RAG社区里热度最高、争论也最多的话题——LangChain和LlamaIndex，到底该怎么选？

你可能遇到过这样的情况：项目刚启动时，看了一圈教程，发现LangChain的生态最大、教程最多，于是果断选了它。埋首开发三个月后，数据索引逻辑越来越复杂，每次要加一个新的文档源都得写一大串chain代码，调试起来让人头疼。这时你听说了LlamaIndex，发现它处理数据管线的优雅程度令人叹服——但为时已晚，迁移成本太高了。

💡 真实教训：某知名AI创业公司在2024年初搭建企业知识库系统时，团队内部对框架选型争执不下。最终选择LangChain后，发现数据索引部分的代码占了整个项目60%以上的维护工作。六个月后他们不得不引入LlamaIndex专门做数据侧，两个框架并存，维护复杂度翻倍。一位工程师事后感叹："如果一开始就理解两个框架的定位差异，我们至少能省下三个月的时间。"

1. 核心理念对比

要理解两者差异，首先要明确它们各自的定位：

  • LangChain——一个通用的大型语言模型（LLM）应用框架。它的核心抽象是Chain和Agent，目标是让开发者能用LLM构建任意类型的应用，不仅仅是RAG。
  • LlamaIndex——一个以数据为中心的RAG框架。它的核心抽象是Index和Retriever，专注于如何高效地连接数据和大模型。

简单来说：LangChain想成为LLM世界的Spring框架，什么都能做；LlamaIndex则像是一个专门为RAG场景定制的数据库中间件，做得更精更专。

2. 功能对比一览

对比维度 LangChain LlamaIndex
核心抽象 Chain、Agent、Tool Index、Retriever、QueryEngine
数据摄入功能 基础，需配合第三方Loader 丰富，原生支持100+数据源
查询路由 通过RouterChain实现 通过RouterQueryEngine实现
Agent支持 极其丰富（OpenAI、ReAct、Plan-Execute等） 基础Agent能力
文档分块 基础分块策略 高级分块（SentenceSplitter、SemanticSplitterNodeParser）
检索策略 需手动组合Retriever 内置10+种检索策略和组合模式
生态系统 最大的LLM应用生态，集成最广 快速增长，专注于RAG领域
学习曲线 中等偏陡（概念太多） 中等（概念明确但术语特有）

3. 数据摄入：代码对决

我们先看一个实际的场景：从PDF和网页中构建RAG索引。这是两个框架最直观的差异所在。

LangChain实现：

from langchain_community.document_loaders import PyPDFLoader, WebBaseLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

# 加载PDF
pdf_loader = PyPDFLoader("report.pdf")
pdf_docs = pdf_loader.load()

# 加载网页
web_loader = WebBaseLoader("https://example.com/docs")
web_docs = web_loader.load()

# 合并后分块
all_docs = pdf_docs + web_docs
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512, chunk_overlap=50
)
chunks = splitter.split_documents(all_docs)

# 构建向量存储
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=OpenAIEmbeddings()
)

LlamaIndex实现：

from llama_index.core import SimpleDirectoryReader, VectorStoreIndex
from llama_index.core.node_parser import SentenceSplitter
from llama_index.vector_stores.chroma import ChromaVectorStore
from llama_index.embeddings.openai import OpenAIEmbedding
import chromadb

# 直接读取整个目录（自动识别PDF、HTML等）
documents = SimpleDirectoryReader(
    input_files=["report.pdf", "https://example.com/docs"]
).load_data()

# 内置高级分块器
parser = SentenceSplitter(chunk_size=512, chunk_overlap=50)
nodes = parser.get_nodes_from_documents(documents)

# 构建索引（一步到位）
chroma_client = chromadb.PersistentClient()
chroma_collection = chroma_client.create_collection("my_docs")
vector_store = ChromaVectorStore(chroma_collection=chroma_collection)

index = VectorStoreIndex.from_documents(
    documents,
    transformations=[parser],
    vector_store=vector_store,
    embed_model=OpenAIEmbedding()
)

可以看到，LlamaIndex的数据管线更加声明式——你告诉它"我要做什么"，它自动处理类型检测、元数据提取和节点关系维护。而LangChain更偏向命令式，你需要手动编排每一步。

4. Agent与编排能力

但在Agent编排方面，LangChain完胜。如果你的RAG系统需要复杂的多步推理、工具调用和状态管理，LangChain的Agent框架会让你事半功倍：

from langchain.agents import create_openai_tools_agent
from langchain.agents import AgentExecutor
from langchain.tools import Tool
from langchain_openai import ChatOpenAI

# 定义RAG工具
retrieval_tool = Tool(
    name="knowledge_base",
    func=lambda q: vectorstore.similarity_search(q, k=3),
    description="搜索企业内部知识库"
)

calculator_tool = Tool(
    name="calculator",
    func=lambda expr: eval(expr),
    description="执行数学计算"
)

# 创建Agent
agent = create_openai_tools_agent(
    llm=ChatOpenAI(model="gpt-4"),
    tools=[retrieval_tool, calculator_tool],
    prompt=...  # system prompt
)
agent_executor = AgentExecutor(agent=agent, tools=[retrieval_tool, calculator_tool])

# 执行：Agent自动决定先检索再计算
result = agent_executor.invoke({
    "input": "去年销售额是多少？比前年增长了多少百分比？"
})

LlamaIndex虽然也支持Agent（通过OpenAIAgent和ReActAgent），但其Agent生态远不如LangChain成熟。如果你需要复杂的Agent编排，LangChain是更好的选择。

5. 混合使用模式（推荐方案）

在实际生产环境中，越来越多团队采用"LlamaIndex做数据层 + LangChain做Agent层"的混合架构。下面是一个典型的实现：

# 数据层 — LlamaIndex
from llama_index.core import VectorStoreIndex, Settings
from llama_index.vector_stores.pinecone import PineconeVectorStore
import pinecone

pinecone.init(api_key="...")
pinecone_index = pinecone.Index("enterprise-kb")
vector_store = PineconeVectorStore(pinecone_index=pinecone_index)

index = VectorStoreIndex.from_vector_store(vector_store)

# 将LlamaIndex的检索器包装为LangChain工具
from llama_index.core.tools import QueryEngineTool

kb_tool = QueryEngineTool.from_defaults(
    query_engine=index.as_query_engine(similarity_top_k=5),
    name="enterprise_kb",
    description="企业知识库检索"
)

# Agent层 — LangChain
from langchain.agents import create_openai_tools_agent, AgentExecutor

agent = create_openai_tools_agent(
    llm=ChatOpenAI(model="gpt-4o"),
    tools=[kb_tool, ...],
    prompt=prompt
)
executor = AgentExecutor(agent=agent, tools=[kb_tool, ...])

这种模式在实践中被证明是最优解——数据层享受LlamaIndex强大的索引能力，Agent层利用LangChain成熟的编排生态。

6. 框架选型决策树

为了帮助你快速决策，这里有一个简洁的选型指南：

  • 纯RAG项目（知识库问答、文档检索）：首选LlamaIndex，数据管线极其丝滑
  • 复杂Agent系统（多工具、多步推理）：首选LangChain，Agent生态无出其右
  • 快速原型验证：LlamaIndex，代码量少一半
  • 已有LangChain投资：不必强行迁移，用LlamaIndex做数据层插件式接入
  • 团队经验：选择团队最熟悉的框架，稳定性比技术先进性更重要
💡 高级建议：当你的RAG系统超过1万条文档时，考虑不要直接使用框架内置的向量存储集成，而是通过框架+原生客户端的方式，这样可以在保留框架便利性的同时，获得数据库层面的最大灵活性。

7. "什么时候两个都不用？"

这是一个值得认真思考的问题。在以下场景中，你可能根本不需要LangChain或LlamaIndex：

  • 极简RAG：只有几十个文档，直接用OpenAI Embedding API + FAISS + 直接调用LLM API，代码不过100行
  • 高吞吐低延迟：金融交易、实时客服等场景，框架的抽象层会引入不必要的开销，最好直接用底层API
  • 已有基础设施：公司已有ES、Milvus等基础设施，直接调用SDK比套一层框架更可控
⚠️ 我见过最惨痛的教训：有团队为了"跟上技术潮流"，在只有5个PDF文件的内部工具中强上了LangChain + Chroma + GPT-4的完整架构。结果项目延期两个月，最终发现用25行Python脚本就能搞定——框架不是万能的，因地制宜才是关键。

好了，这一章我们深入对比了LangChain和LlamaIndex两个主流框架的定位、优劣势和选型策略。理解它们的差异，能让你在搭建RAG系统时做出更明智的技术决策。下一章，我们将深入探讨向量数据库的选型与性能优化——毕竟框架再好，底层的检索引擎性能才是RAG系统真正的基石。
