第十三章：上下文增强与提示工程

大家好，我是你们的讲师。走到这一步，我们的RAG系统已经完成了检索、重排序、查询优化等一系列操作，拿到了高质量的相关文档。现在的问题是：如何把这些文档有效地喂给LLM？

你可能觉得这很简单——把文档拼接起来加上问题不就行了？但事实远没有这么简单。Prompt的设计、上下文的组织、文档的排列顺序、指令的措辞——这些细节都会对最终生成的答案质量产生巨大影响。这就是上下文增强（Context Augmentation）与提示工程（Prompt Engineering）的用武之地。

1. 上下文窗口容量规划

LLM的上下文窗口是有限的。GPT-4有128K、Claude有200K，但这不意味着你可以把所有检索到的文档都塞进去。原因有二：

  • Lost-in-the-Middle效应：LLM对输入中间位置的内容关注度明显低于开头和结尾。
  • 信噪比：塞入大量低相关度的文档会稀释高相关文档的影响力。
  • 成本：输入token越多，推理成本和延迟越高。

我建议的做法是：根据任务复杂度，将上下文严格控制在 2000-6000 tokens 之间。这个范围足以包含3-5个高质量文档块，同时留出足够的空间让LLM生成详细回答。

2. Lost-in-the-Middle 问题

⚠️ Lost-in-the-Middle 效应——不要忽视文档顺序
斯坦福大学和UC Berkeley的研究人员发现了一个令人震惊的现象：当上下文中的相关文档放在开头或结尾时，LLM的答案准确率超过80%；但当相关文档放在中间位置时，准确率骤降到50%以下。这就是Lost-in-the-Middle效应。这意味着：仅仅把最相关的文档放到prompt的中间位置，就相当于浪费了一半的检索质量。解决方法是：将检索到的文档按照相关性降序排列，并在prompt设计时把最关键的文档放在开头（或结尾），而不是中间。

3. RAG提示模板设计

好的Prompt模板是RAG系统的"最后一公里"。让我们看几种经过实战检验的模板：

模板一：基础RAG模板

你是一个智能问答助手。请根据以下检索到的文档内容来回答用户的问题。

检索到的文档：
{% for doc in documents %}
[文档{{ loop.index }}]
{{ doc.content }}
{% endfor %}

用户问题：{{ query }}

回答要求：
1. 请只基于以上文档内容回答，不要使用你自己的知识。
2. 如果文档内容不足以回答问题，请明确回答"根据提供的资料，无法回答该问题"。
3. 回答要简洁、准确、有条理。

回答：

模板二：带引用的RAG模板

你是一个需要提供引用来源的问答助手。回答时请在每个事实后面标注来源。

检索到的文档：
{% for doc in documents %}
[来源{{ loop.index }}]
{{ doc.content }}
来源ID: {{ doc.id }}
{% endfor %}

问题：{{ query }}

请用以下格式回答：
- 陈述事实内容 [来源X]
- 下一个事实 [来源Y]
- ...

注意事项：
- 每个事实必须标注来源
- 不要编造来源
- 如果多个来源一致，可以合并引用

模板三：多轮对话RAG模板

你是智能客服助手。以下是对话历史和检索到的相关知识。

对话历史：
{% for msg in history %}
{{ msg.role }}: {{ msg.content }}
{% endfor %}

相关知识文档：
{% for doc in documents %}
[文档{{ loop.index }}] {{ doc.content }}
{% endfor %}

当前用户问题：{{ query }}

请结合对话历史和检索文档回答当前问题。
如果你需要澄清用户的问题，请先提出澄清问题。

4. 上下文排列策略对比

策略 做法 优点 缺点
相关性降序 最相关文档放最前 符合开头重要性，简单 忽略结尾位置的利用
相关性升序 最相关文档放最后 利用结尾重要性 用户可能过早看到不相关内容
"三明治"策略 次相关→最相关→次相关 同时利用开头和结尾 中间文档可能被忽略
交替排列 按不同检索源交替排列 避免同一来源的偏见 可能打乱逻辑连续性

根据我的实践经验，"三明治"策略（最相关文档放在开头，次相关放中间，再次相关放结尾）在大多数场景下表现最好。它最大化了"开头+结尾"的注意力优势。

5. 上下文压缩

当知识库文档很长时，直接全部塞入prompt会浪费token。上下文压缩（Context Compression）技术应运而生：

  • LLMLingua：微软开源的工具，使用一个小模型逐句评估每个句子与查询的相关性，丢弃低相关性的句子。可以将上下文压缩到原来的20-30%，同时保留大部分有用信息。
  • 选择性过滤：根据文档中的metadata（如标题、章节名）做粗粒度过滤。例如，只保留与查询主题相关的章节。
  • 摘要式压缩：用LLM对每个文档块生成简短摘要，用摘要代替原文。适合超长文档（如论文全文、法律合同）。
# LLMLingua 使用示例（伪代码示意）
from llmlingua import PromptCompressor

compressor = PromptCompressor(
    model_name="microsoft/llmlingua-2-xlm-roberta-large-meetingbank",
    device_map="cuda"
)

query = "什么是向量数据库的特性？"
documents = [
    "向量数据库是一种专门用于存储和查询向量嵌入的数据库系统...",
    "传统数据库基于精确匹配和范围查询，而向量数据库基于相似度...",
    "Milvus是一个开源的向量数据库，它支持多种索引类型...",
]

# 压缩上下文
compressed_prompt = compressor.compress_prompt(
    context="\n".join(documents),
    question=query,
    rate=0.5,  # 压缩率为50%
)

print(f"原始长度: {compressed_prompt['origin_tokens']} tokens")
print(f"压缩后长度: {compressed_prompt['compressed_prompt_tokens']} tokens")

6. Metadata注入

在上下文中注入文档的元数据（Metadata）可以显著提升LLM对信息的利用质量。常见的metadata包括：

  • 来源信息：文档标题、URL、作者、发布日期。
  • 层级信息：所属章节、目录路径。
  • 置信度信息：该文档的检索分数、重排序分数。
  • 时效性：文档的更新时间、数据采集时间。

例如，与其只传入"2024年公司营收增长20%"，不如传入"根据公司2024年年报第5页，2024年公司营收增长20%"。LLM可以根据这个来源信息判断文档的可信度和时效性。

踩坑故事：被"幻觉"的真相

做一个内部知识库问答系统时，发现LLM总是给出不准确的答案。检查了检索系统、重排序、所有环节都没有问题。最后发现——问题出在prompt里的一句话：

我们的prompt模板中写了一句"如果你知道答案，可以直接回答，不需要参考文档"。这个看似无害的指令给了LLM一个"出口"——它经常选择"直接回答"而非"基于文档回答"。结果是：对于LLM训练数据中覆盖到的信息，即使知识库中有更准确（或更新）的版本，LLM也倾向于用自己的知识回答。

教训是：RAG的prompt必须严格约束LLM只基于提供的文档内容回答。任何暗示"可以使用自己的知识"的措辞都应该去掉。把"如果你知道答案"改为"请严格基于以下文档回答"。

本章小结

上下文增强和提示工程是RAG系统的"最后一公里"，决定了检索质量最终能有多少转化为回答质量。我们需要关注：上下文窗口的合理规划、Lost-in-the-Middle效应的应对、prompt模板的精心设计、上下文压缩技术的应用，以及metadata的有效注入。一个好的prompt模板可以让检索效果提升20-30%——这个数字甚至超过了更换一个更好的嵌入模型带来的提升。

检索完成、上下文也组织好了，下一步就是把所有信息交给LLM生成最终回答了。但在生成阶段还有一个关键问题：如何确保LLM的回答有据可查、不会胡编乱造？这就是下一章——生成与引用溯源要讨论的内容。第14章见！
