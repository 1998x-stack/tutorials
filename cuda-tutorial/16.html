调试与错误处理

大家好，我是你们的讲师。GPU 调试比 CPU 调试困难得多——没有 printf（至少在早期）、没有完整的堆栈跟踪（stack trace）、经常只给你一个 " unspecified launch failure " 的错误信息，然后就没了。更糟的是，很多 bug 不会导致程序崩溃，而是悄悄地给出一个错误答案——你可能要跑完三天的模拟，最后才发现结果全是 NaN。

为什么 GPU 调试这么难？原因有几条。第一，CPU 和 GPU 是异步的——内核启动后主线程继续执行，当你发现错误时，出错的 kernel 可能已经是"过去式"了。第二，GPU 上有成千上万个线程在同时运行，即使在同一个 warp 里，谁在何时访问了哪块内存也很难追踪。第三，GPU 的硬件不是为调试设计的——它的执行模型非常"弱化"，每个线程的行为高度依赖调度器的随机决策。

好消息是： NVIDIA 在过去几年开发了一系列非常强大的调试工具——compute-sanitizer、cuda-gdb、以及 CUDA 13.x 中引入的硬件辅助调试功能。学会它们，GPU 调试就不再是"玄学"了。本章的目标是让你掌握一套完整的调试方法论，遇到任何 GPU bug 都能有条不紊地定位和修复。

compute-sanitizer内存检查

compute-sanitizer 是你在 GPU 调试中应该最先使用的工具。它是 NVIDIA 在 CUDA 11.0 中推出的内存检测工具（替代了老旧的 cuda-memcheck），无需重新编译程序，只需在你的程序前面加上 compute-sanitizer 即可。它能够检测出 GPU 代码中几乎所有常见的内存错误。

下面这张表列出了 compute-sanitizer 提供的主要检测工具和它们能发现的问题：

工具模式 命令 检测内容 典型误报率
Memory Check compute-sanitizer ./app（默认模式） 越界访问（out-of-bounds）、非法地址（illegal access）、使用未初始化内存 低
RaceCheck compute-sanitizer --tool racecheck ./app 共享内存和全局内存的数据竞争（data race）、读取未初始化共享内存 中等
InitCheck compute-sanitizer --tool initcheck ./app 检测是否读取了未初始化的设备内存（包括 malloc 和 mallocAsync 分配的区域） 低
Synccheck compute-sanitizer --tool synccheck ./app 检测错误的同步模式，如 __syncthreads() 使用不当（barrier divergence） 中等

最常用的是默认的 Memory Check 模式。你可能会惊讶于它的"发现能力"——很多你以为"运行正常"的程序，一跑 compute-sanitizer 就会暴露出来隐藏的越界或未初始化问题。它报告的错误信息格式也很清晰：错误类型、出错的 kernel 名称、block 和 thread 索引、以及发生错误的地址和指令。

使用建议： 让你的 CI/CD 流程中包含 compute-sanitizer 这一步。每次提交代码之前，至少跑一次 compute-sanitizer ./my_tests。这会把 bug 扼杀在摇篮里，而不是等到上线后才被用户发现。代价是程序会运行得慢 5-10 倍（因为每次访存都需要硬件辅助检查），但对于测试流程来说，这完全值得。

cuda-gdb基础

当 compute-sanitizer 告诉你"哪里错了"但你还想知道"为什么错"的时候，就该 cuda-gdb 出场了。cuda-gdb 是 GDB 的 CUDA 扩展版本，它能让你在 GPU 内核中设置断点、单步执行、检查变量值——就像调试 CPU 程序一样。

cuda-gdb 的启动方式和 GDB 类似：

# 启动cuda-gdb
cuda-gdb ./my_app

# 设置断点——可以在主机代码或GPU内核函数中设置
(gdb) break my_kernel
(gdb) break my_kernel:42              # 在my_kernel的第42行设置断点
(gdb) break my_kernel if threadIdx.x == 0 && blockIdx.x == 0  # 条件断点

# 运行到断点
(gdb) run

# 当命中断点后，检查CUDA线程状态
(gdb) info cuda threads               # 列出所有活跃的CUDA线程
(gdb) cuda thread (0,0,0)             # 切换到指定线程
(gdb) cuda block (1,0,0)              # 切换到指定block

# 检查变量和内存
(gdb) print threadIdx.x
(gdb) print my_variable
(gdb) x/10f &my_array[0]              # 查看内存中的10个float值

# 单步执行
(gdb) step                            # 单步进入
(gdb) next                            # 单步跳过
(gdb) continue                        # 继续执行

# 查看调用栈
(gdb) backtrace
(gdb) info registers                  # 查看GPU寄存器

cuda-gdb 最强大的功能是条件断点（conditional breakpoint）。由于 GPU 有成千上万个线程，你不可能在每个线程上都停——你会疯掉的。通常的调试策略是：在某个特定的线程上设置条件断点（比如 threadIdx.x == 0 && blockIdx.x == 0），然后观察该线程的行为是否符合预期。

注意： 编译内核时必须加上 -G 标志（设备端调试模式），cuda-gdb 才能正常工作。但这个标志会大幅降低内核执行速度（因为编译器会关闭很多优化），并且会增加内核的寄存器使用量。你的程序在 -G 模式下表现可能和 Release 模式不同——特别是涉及 race condition 的 bug。所以策略是：先用 -G 模式锁定问题的大致范围，然后去掉 -G 确认最终的修复是否生效。

常见Bug分类与排查

GPU 编程中遇到的 bug 虽然种类繁多，但绝大多数可以归为以下四类。掌握了每类的典型症状和排查工具，你就能很快地定位问题：

Bug 类型 典型症状 可能原因 首选排查工具 修复方向
越界访问 cudaErrorLaunchFailure、随机崩溃、结果部分正确 索引计算错误、grid/block 配置太大、没有做边界检查 compute-sanitizer（Memory Check） 检查索引公式、加上边界保护条件
数据竞争 结果不确定（每次运行结果不同）、某些运行是对的 多个线程同时读写同一共享/全局内存位置、缺少 __syncthreads() compute-sanitizer（RaceCheck） 添加同步屏障、用原子操作、重新设计数据布局
非法内存访问 "an illegal memory access was encountered"、程序崩溃 空指针被传入 kernel、内存释放后仍在访问（use-after-free） compute-sanitizer（Memory Check） 检查 cudaMalloc 返回值、确保 cudaMemcpy 方向正确
数值错误 NaN、Inf、精度偏差严重 未初始化变量、除以零、累加溢出、精度不足（FP16下常见） compute-sanitizer（InitCheck）+ cuda-gdb 条件断点 初始化所有变量、使用更安全的分母检查、改用更高精度
错误同步 死锁（程序卡住）、cudaErrorIllegalState __syncthreads() 放在条件分支中、不同步的 __threadfence() 使用 compute-sanitizer（Synccheck）+ cuda-gdb 确保 barrier 对所有线程可达、使用 cooperative groups
调试黄金法则： 遇到 GPU bug 时，永远先跑 compute-sanitizer。它是最快、最便宜的工具。它解决不了的问题才需要 cuda-gdb。很多人在 bug 出现的第一时间就扎进 cuda-gdb 的单步调试中，结果是浪费了大量时间在一个 compute-sanitizer 5 秒就能捕获的越界问题上。从简单工具开始，逐步升级。

防御性编程与错误处理宏

在 CUDA 编程中，防御性编程不是一种选择——而是必需品。GPU 是异步的，如果一个 kernel 启动失败，它不会立即报错——错误会被推迟到下一个同步点（cudaMemcpy、cudaDeviceSynchronize 等）。如果你不去检查返回值，你可能会得到一个看似"正常"的结果，但其实是错的。

以下是一套我每个 CUDA 项目中都会包含的错误处理宏。你应该把它放在一个公共的头文件中，比如 cuda_utils.h：

#include <stdio.h>
#include <cuda_runtime.h>

// 检查CUDA API调用的返回值
#define CUDA_CHECK(call) do { \
    cudaError_t err = call; \
    if (err != cudaSuccess) { \
        fprintf(stderr, "CUDA error at %s:%d — %s\n", \
                __FILE__, __LINE__, cudaGetErrorString(err)); \
        exit(EXIT_FAILURE); \
    } \
} while(0)

// 检查最后一次kernel启动是否成功
#define KERNEL_CHECK() do { \
    cudaError_t err = cudaGetLastError(); \
    if (err != cudaSuccess) { \
        fprintf(stderr, "Kernel launch error at %s:%d — %s\n", \
                __FILE__, __LINE__, cudaGetErrorString(err)); \
        exit(EXIT_FAILURE); \
    } \
} while(0)

// 检查cudaDeviceSynchronize —— 捕获异步执行中的错误
#define SYNC_CHECK() do { \
    cudaError_t err = cudaDeviceSynchronize(); \
    if (err != cudaSuccess) { \
        fprintf(stderr, "CUDA sync error at %s:%d — %s\n", \
                __FILE__, __LINE__, cudaGetErrorString(err)); \
        exit(EXIT_FAILURE); \
    } \
} while(0)

使用示例：

// 用CUDA_CHECK包装所有API调用
CUDA_CHECK(cudaMalloc(&d_a, n * sizeof(float)));
CUDA_CHECK(cudaMemcpy(d_a, h_a, n * sizeof(float), cudaMemcpyHostToDevice));

// 启动内核后立即检查启动是否成功
my_kernel<<<grid, block>>>(d_a, d_b, n);
KERNEL_CHECK();

// 在同步点检查异步错误
SYNC_CHECK();

这个简单的做法能让你在开发早期就抓住 90% 的常见错误。很多新手 CUDA 程序员会在辛苦 debug 几小时后才发现，原来问题只是一行 cudaMalloc 的返回值没检查——传了一个空指针进 kernel，跑出了 illegal memory access。

进阶建议： 在 Debug 构建中，可以在每个 kernel 启动后都调用 SYNC_CHECK()。但在 Release 构建中，去掉 SYNC_CHECK()（因为它会引入不必要的同步，拖慢性能）。用宏的条件编译（#ifdef DEBUG）来控制这个行为是最佳实践。

内核printf调试

最后说一个简单但极其实用的技巧：在 CUDA kernel 内部使用 printf()。是的，从 Fermi 架构（Compute Capability 2.0+）开始，CUDA 就支持在设备端直接调用 printf。它的行为和在 CPU 上类似——输出到标准输出流，但有一些重要的注意事项。

__global__ void debug_kernel(float *data, int n) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;

    // 只在特定线程打印——避免输出爆炸
    if (tid == 0) {
        printf("=== Kernel debug info ===\n");
        printf("gridDim: (%d,%d,%d), blockDim: (%d,%d,%d)\n",
               gridDim.x, gridDim.y, gridDim.z,
               blockDim.x, blockDim.y, blockDim.z);
    }

    // 只打印前几个线程的值
    if (tid < 4) {
        printf("block(%d) thread(%d): data[%d] = %f\n",
               blockIdx.x, threadIdx.x, tid, data[tid]);
    }
}

使用 printf 调试时，要注意以下几点：

  • 输出是有缓冲的：printf 的输出存储在 GPU 上的一个环形缓冲区中，程序结束后才会 flush 到终端。如果程序崩溃了（比如 illegal memory access），缓冲区里的内容可能会丢失
  • 不要无限制地打印：如果你在 tid == 0 的线程里打印了 printf，你会得到一个完全正常的输出——一条消息。但如果你在 tid < 100000 里全部打印，你的终端会被刷爆，而且 printf 本身的性能开销会彻底改变 kernel 的执行时间
  • 在 Release 构建中要移除：printf 在设备端有非平凡的开销，编译到 Release 版本之前应该注释掉或通过宏条件编译来控制

printf 适用于"这个 kernel 到底有没有跑？thread index 对不对？"这种快速验证场景。但它不能替代 compute-sanitizer 或 cuda-gdb——对比来说，printf 暴露的是"我看到的问题"，而 compute-sanitizer 暴露的是"你没想到的问题"。

一次三天调试的教训

我想分享一个让我非常难忘的调试经历。有一次我在做一个金融风险模拟的 CUDA 项目，跑了三天的蒙特卡罗模拟后，发现所有结果都变成了 NaN。我们试了各种方法：检查索引、检查输入数据、甚至重写了半个 kernel。三天过去了，我们毫无头绪。

最后是一个刚入职的同事提议："要不跑一下 compute-sanitizer？" 我们当时觉得——"怎么可能，程序明明没崩溃啊"。结果 compute-sanitizer 跑下来，几乎立刻就报告了问题：kernel 中有一段共享内存声明为 __shared__ float cache[256]，但我们只在其中 256 个线程中的 255 个里进行了初始化，有一个线程没有写入——于是那个位置保留着之前 kernel 留下的垃圾值，而且是 NaN。这个 NaN 在后续的并行规约中就像病毒一样扩散，最终感染了整个结果。

compute-sanitizer 明确指出："WARNING: 1 byte of uninitialized shared memory was detected"。解决方式就是一行 memset 的事情。我们花了三天，comput-sanitizer 花了 3 秒。从此以后，"先跑 compute-sanitizer"就成了我团队的铁律。
