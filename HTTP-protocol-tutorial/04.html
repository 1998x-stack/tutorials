第四章：HTTP请求方法

大家好，我是你们的讲师。上一章我们学会了看HTTP报文，其中请求报文的第一行就是请求方法。方法可以说是HTTP协议的"动词"——它告诉服务器你想对资源做什么。Web开发中90%的场景只用GET和POST，但HTTP协议其实定义了9种方法，各有各的用途和讲究。

4.1 方法概览

HTTP/1.1标准定义的方法一共有9个：

方法RFC含义是否有Body
GETRFC 9110获取资源否
POSTRFC 9110提交数据/创建资源是
PUTRFC 9110完整替换资源是
DELETERFC 9110删除资源通常无
PATCHRFC 5789部分更新资源是
HEADRFC 9110获取响应头部（不含body）否
OPTIONSRFC 9110查询资源支持的通信选项否
TRACERFC 9110回显请求，用于诊断否
CONNECTRFC 9110建立隧道（用于HTTPS代理）否

日常开发中，GET和POST占所有HTTP请求的95%以上。PUT、DELETE、PATCH主要在RESTful API中使用。而TRACE因为安全原因（容易引发跨站脚本攻击），大多数生产服务器都禁用了它。

冷知识：早年间有些防火墙和代理只允许GET和POST通过。如果你在做全栈API开发，发现PUT/DELETE请求莫名其妙被拦截，可以先检查一下中间件是否支持。我自己就遇到过阿里云WAF默认只放行GET/POST，PUT请求全给挡了的情况。

4.2 安全性与幂等性

这是面试中的高频考点，也是API设计时必须理解的两个概念：

  • 安全性（Safe）：该方法是否不会修改服务器上的资源。安全的方法应该像"只读"一样——调用一次和调用一百次，服务器上的数据没有任何变化。
  • 幂等性（Idempotent）：多次执行同一个请求，结果和一次执行相同。换句话说，重复请求不会产生副作用。

来看各个方法的安全性与幂等性：

方法是否安全是否幂等
GET是是
HEAD是是
OPTIONS是是
TRACE是是
PUT否是
DELETE否是
POST否否
PATCH否否
CONNECT否否

注意看，安全的方法全都是幂等的，但幂等的不一定是安全的（比如PUT和DELETE）。POST最特殊——既不安全也不幂等。这就是为什么浏览器刷新POST页面时会弹出"确认重新提交表单"的对话框——因为重复提交可能导致重复下单、重复扣款等严重后果。

设计原则：在设计API时，遵循"方法语义"比"实现方便"更重要。很多人图省事，把所有API都写成POST，这在技术上是可行的，但会让API失去语义清晰度。想象一下，如果你的用户要看文档，你给他一把刀——用户会觉得你很奇怪。同理，删除操作应该用DELETE，查询应该用GET。

4.3 GET详解

GET是最常用的HTTP方法，用于"获取"资源。它的核心特征是：

  • 只读：GET请求不应该改变服务器上的任何状态
  • 参数在URL中：通过query string传递，如 ?page=1&limit=20
  • 可缓存：浏览器和CDN可以缓存GET请求的响应，提升性能
  • 可收藏/可分享：URL可以保存为书签，可以复制发给别人
// 一个典型的GET请求
GET /products?category=electronics&sort=price_asc HTTP/1.1
Host: shop.example.com
Accept: application/json
If-None-Match: "abc123"

什么时候不该用GET？

  • 提交密码/敏感信息：GET的参数会出现在URL里，会被浏览器历史、服务器日志完整记录
  • 大量数据：上一章说过，URL长度有限制
  • 修改数据：GET语义是只读的，虽然技术上你可以用GET修改数据，但这会让所有语义约定失效。缓存代理可能直接返回缓存结果而不去服务器，导致修改"失败"但返回200

踩坑回顾：当年做一个CMS系统，编辑删除文章的功能用了GET请求（当时图省事直接用了 <a href="delete?id=123">）。结果被百度爬虫顺着链接把所有文章都删了……后来我养成了一个习惯：任何有副作用的操作，绝不使用GET。搜索引擎爬虫、预加载插件、甚至浏览器的预连接机制都可能触发GET请求。

4.4 POST详解

POST用于"提交"数据给服务器，通常用于创建资源或触发操作。它的核心特征：

  • 数据在body中：POST请求的数据放在消息体里，不会出现在URL中
  • 不幂等：多次提交可能产生多个资源（重复下单）
  • 不可缓存：浏览器默认不缓存POST响应
  • 没有大小限制：相对于GET的URL长度限制，POST的body理论上没有上限
// 一个典型的POST请求
POST /api/orders HTTP/1.1
Host: shop.example.com
Content-Type: application/json
Content-Length: 87
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

{"product_id": "prod_123", "quantity": 2, "shipping_address": "北京市海淀区..."}

POST最常见的场景是表单提交和RESTful API的资源创建。使用POST时一定要考虑CSRF（跨站请求伪造）问题——攻击者可以在另一个网站上构造一个表单，自动提交POST请求到你的服务器。解决方案包括CSRF Token、SameSite Cookie、以及校验Referer头。

POST vs PUT：POST创建资源时，由服务器决定资源的URL（如 /orders/12345）；PUT则要求客户端指定资源的完整URL（如 /users/123）。POST可以重复提交创建多个资源，PUT重复提交替换同一个资源。

4.5 PUT、DELETE、PATCH的区别

这三个方法在RESTful API中非常常见，它们的区别很微妙：

  • PUT：完整替换资源。你发送一个完整的资源表示，服务器用你发送的内容完全覆盖已有的资源。PUT是幂等的——不管调用多少次，结果都一样。
  • DELETE：删除资源。也是幂等的——第一次删除返回200/204，第二次删除返回404（因为资源已经不存在了），但服务器状态没有额外变化。
  • PATCH：部分更新资源。你只发送需要修改的字段，而不是整个资源。PATCH不保证幂等性——同一个PATCH请求执行两次，结果可能不同。
// PUT — 完整替换
PUT /api/users/123 HTTP/1.1
Content-Type: application/json

{"name": "张三", "email": "zhangsan@example.com", "age": 28}

// PATCH — 部分更新
PATCH /api/users/123 HTTP/1.1
Content-Type: application/json

{"age": 29}

// DELETE — 删除
DELETE /api/users/123 HTTP/1.1

我见过团队里有人争论"PATCH到底传什么格式"，JSON Patch（RFC 6902）定义了一套操作格式，但实践中更多人直接用JSON传要修改的字段。我的建议是：团队统一规范，文档写清楚就好。但如果你要设计公共API，建议遵循JSON Patch或JSON Merge Patch标准。

经验之谈：很多新手搞混PUT和PATCH，简单记住——PUT像"覆盖保存"，PATCH像"编辑部分字段"。想象一个Word文档：PUT是把整篇文档替换掉，PATCH是只修改某个段落。如果你的API中，客户端需要先GET完整数据、修改一两个字段、再PUT回来——那不如改成PATCH。

4.6 HEAD与OPTIONS的实用场景

HEAD和OPTIONS虽然不常用，但在某些场景下非常有用。

HEAD方法和GET完全一样，唯一的区别是服务器返回的响应不包含body。什么时候用？

  • 检查资源是否存在：发HEAD请求看返回200还是404，不用下载完整内容
  • 检查资源大小：通过 Content-Length 响应头判断文件大小，决定是否下载
  • 检查资源最后修改时间：通过 Last-Modified 头判断是否需要重新下载
  • 检查链接有效性：爬虫和监控系统经常用HEAD来验证URL是否有效
// HEAD请求示例
$ curl -I https://example.com/large-file.zip
HTTP/1.1 200 OK
Content-Length: 1073741824
Last-Modified: Mon, 15 Apr 2024 10:00:00 GMT
ETag: "abc123"
Content-Type: application/zip

// 通过HEAD我们就知道这是个1GB的文件，决定要不要下载

OPTIONS方法用来查询服务器支持哪些功能。最典型的应用就是CORS预检请求：

// 浏览器在跨域请求前自动发送OPTIONS预检
OPTIONS /api/data HTTP/1.1
Host: api.example.com
Origin: https://frontend.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Authorization, Content-Type

// 服务器回复
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://frontend.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Max-Age: 86400

跨域请求时，如果请求不是简单请求（如使用了自定义头部、PUT方法等），浏览器会先发一个OPTIONS预检请求，询问服务器是否允许这个跨域请求。服务器通过响应头告诉浏览器允许哪些来源、方法、头部。这整套机制就是CORS（跨域资源共享），我们后面会有专门章节深入讲。

踩坑回顾：有次后端改了一个GET接口为POST，但前端忘了更新axios配置中的method。结果OPTIONS预检过了，POST也发出去了，但后端路由没有POST处理，返回404。更坑的是，因为OPTIONS请求成功（204），浏览器控制台没报跨域错误，只显示404。查了整整一个下午才发现是method没改。教训：接口变更方法时，前后端要同步确认，且后端应该返回更有区分度的错误信息。
