第22章 FSDP与大模型训练

大家好，我是你们的讲师。上一章我们讲了DDP——把数据分到多卡上并行训练。但DDP有一个前提：完整的模型必须能装进单张显卡。如果你的模型是Llama-70B（700亿参数，以FP16计就要140GB显存），就算你用最新的H100（80GB），也装不下。这时候DDP就无能为力了——你连一份模型都放不下，怎么在每个卡上复制一份？

这就引出了本章的主角：FSDP（Fully Sharded Data Parallel）。FSDP的核心思想很简单：既然模型太大装不进单卡，那把模型也"切片"，每张卡只保存一部分参数，需要的时候再通过通信把其他卡上的参数取过来。听起来像分布式文件系统？没错，思路确实类似。

22.1 大模型训练的显存瓶颈

一个大型模型的显存消耗由四部分组成：

组件 说明 占用量（Llama-7B, FP16为例）
模型参数 所有weight和bias ~14GB（7B * 2 bytes）
梯度 反向传播计算的梯度，与参数大小相同 ~14GB
优化器状态 Adam需要存储momentum和variance（各4 bytes） ~56GB（7B * 8 bytes）
激活值 前向传播中的中间结果，用于反向传播 取决于batch size和序列长度，通常10-30GB

总合起来，Llama-7B在FP16下训练需要大概100GB左右的显存。一张A100（80GB）勉强够用batch_size=1，而Llama-70B则需要将近1TB——没有任何单卡能装下。

FSDP的思路是：把参数、梯度、优化器状态都分片到各卡上，每卡只保存 1 / WORLD_SIZE 的份额。在需要的时候，通过 all-gather 通信把完整的参数收集回来。

22.2 FSDP1 vs FSDP2

PyTorch的FSDP经历了两代API演变：

  • FSDP1（torch.distributed.fsdp.FullyShardedDataParallel）：类DDP的包装方式，用 FullyShardedDataParallel 包裹模型。参数分片由框架自动管理。优点是兼容性好，大部分现有模型可以无缝切换。缺点是wrap策略不够灵活，必须对整个模型或子模块做统一的sharding。
  • FSDP2（torch.distributed._composable.fsdp.fully_shard）：新一代API，基于composable API设计。可以对模型的不同部分使用不同的sharding策略。更灵活，性能更好。缺点是相对较新，部分旧版PyTorch不支持。
小贴士：如果你在用PyTorch 2.4+，建议直接上FSDP2（fully_shard）。新的composable API在性能和灵活性上都优于FSDP1，而且代码写起来更自然。如果你的项目还在用PyTorch 2.0-2.3，FSDP1也是完全可以用的选择。

22.3 FSDP2 实战：fully_shard

FSDP2的使用模式是：对模型的每一层（例如Transformer的每一个block）分别调用 fully_shard，最后对整个模型调用一次。这样每一层内部的参数会被分配到一个通信组内，层与层之间可以流水执行。

from torch.distributed.fsdp import fully_shard, MixedPrecisionPolicy
import torch.nn as nn

class TransformerBlock(nn.Module):
    def __init__(self, dim, n_heads):
        super().__init__()
        self.attention = nn.MultiheadAttention(dim, n_heads)
        self.mlp = nn.Sequential(
            nn.Linear(dim, dim * 4),
            nn.ReLU(),
            nn.Linear(dim * 4, dim),
        )
        self.layernorm1 = nn.LayerNorm(dim)
        self.layernorm2 = nn.LayerNorm(dim)

    def forward(self, x):
        x = x + self.attention(self.layernorm1(x), self.layernorm1(x), self.layernorm1(x))[0]
        x = x + self.mlp(self.layernorm2(x))
        return x

class BigModel(nn.Module):
    def __init__(self, num_blocks=32, dim=4096, n_heads=32):
        super().__init__()
        self.embed = nn.Embedding(32000, dim)
        self.blocks = nn.ModuleList([
            TransformerBlock(dim, n_heads) for _ in range(num_blocks)
        ])
        self.norm = nn.LayerNorm(dim)
        self.output = nn.Linear(dim, 32000)

    def forward(self, tokens):
        x = self.embed(tokens)
        for block in self.blocks:
            x = block(x)
        x = self.norm(x)
        return self.output(x)

# 创建模型
model = BigModel()

# FSDP2: 逐层shard
for block in model.blocks:
    # 每个transformer block单独shard
    fully_shard(block)

# 最后对整个模型shard
fully_shard(model)

# 设置混合精度策略
mp_policy = MixedPrecisionPolicy(
    param_dtype=torch.bfloat16,    # 参数用BF16存储
    reduce_dtype=torch.float32,    # 梯度all-reduce用FP32
)
# 在optimizer中应用混合精度
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)

22.4 混合精度策略

FSDP的 MixedPrecisionPolicy 有三个关键参数：

参数 作用 推荐值
param_dtype 参数在内存中存储的精度 torch.bfloat16（节省一半显存）
reduce_dtype all-reduce通信时梯度的精度 torch.float32（避免梯度精度损失）
output_dtype 前向传播计算中使用的精度 通常与 param_dtype 保持一致

使用BF16 + FP32的组合是最常见的。BF16节省了参数存储的显存，而FP32保证了梯度通信的精度。

22.5 分片检查点（Sharded Checkpointing）

FSDP训练中，每个进程只保存模型的一部分参数。如果每个进程单独保存自己的分片，恢复时也需要同样数量的GPU。这在使用不同数量GPU恢复训练时很麻烦。

PyTorch提供了 distributed state_dict 来管理分片检查点：

from torch.distributed.checkpoint import save, load
from torch.distributed.checkpoint.state_dict import (
    get_state_dict,
    set_state_dict,
)

# 保存
state_dict = {
    "model": model.state_dict(),
    "optimizer": optimizer.state_dict(),
}
save(state_dict, checkpoint_id="path/to/checkpoint")

# 加载
state_dict = {
    "model": model.state_dict(),
    "optimizer": optimizer.state_dict(),
}
load(state_dict, checkpoint_id="path/to/checkpoint")
model.load_state_dict(state_dict["model"])
optimizer.load_state_dict(state_dict["optimizer"])

这种方式的优点是可以从不同数量的GPU上恢复——比如用8卡训练，用4卡也可以加载检查点继续训练。

22.6 什么时候需要 TP 和 PP？

FSDP不是万能的。当模型继续增大到数百亿甚至上万亿参数时，光靠FSDP可能还不够。这时候需要引入其他并行策略：

  • Tensor Parallelism（TP）：把一个层的计算切分到多张卡上。例如，一个巨大的线性层 Y = XW，可以把 W 按列切分到多卡上，每卡只做部分计算，最后all-reduce结果。典型的应用是Megatron-LM的列并行和行并行。
  • Pipeline Parallelism（PP）：把模型按层分成多个阶段（stage），每个阶段放在不同的卡上。数据以micro-batch的形式在stage之间流水传递。

一般来说，在模型规模上的选择策略是：

  • 模型能装进单卡 → DDP即可
  • 模型稍大于单卡 → FSDP
  • 模型远大于单卡且单层就很大 → FSDP + TP
  • 模型超级大且有深层次结构 → FSDP + TP + PP（俗称3D并行）

22.7 主流方案对比

方案 参数分片 梯度分片 优化器状态分片 通信量 易用性
DDP 否（每卡完整副本） 否（每卡完整梯度） 否（每卡完整状态） 每个参数每个step一次all-reduce ★★★★★
FSDP2 是（每卡 1/N） 是（每卡 1/N） 是（每卡 1/N） 每个参数每个step两次all-gather + 一次reduce-scatter ★★★★
DeepSpeed ZeRO-3 是（每卡 1/N） 是（每卡 1/N） 是（每卡 1/N） 与FSDP类似，但实现细节不同 ★★★★
Megatron-LM 是（TP+PP切分） 是 是 较高（需要TP内部all-reduce + PP通信） ★★★

对于大部分用户，FSDP2是性价比最高的选择。DeepSpeed ZeRO-3在性能上可以和FSDP2媲美，但需要额外安装DeepSpeed库。Megatron-LM则更适用于超大规模模型训练（100B+），配置复杂度也更高。
