第24章 Python密码学编程

上一章我们用 OpenSSL 命令行操作了密钥、证书和加密——但命令行能做的事情终究有限。当你的应用需要自动生成密钥、在运行时动态加密数据、或者通过网络建立安全信道时，你需要用编程语言来调用密码学函数。而在 Python 生态中，你有一批成熟、活跃且经过安全审计的密码学库可供选择。

为什么专门要讲 Python？因为 Python 是安全工程师、DevOps 工程师和数据科学家最常用的脚本语言。用 Python 写密码学代码，你可以在半小时内实现一个加密聊天工具、一个文件加密器、或者一个密钥管理系统——如果换成 C 或者 Java，同样的功能可能需要数天。但便利也伴随着风险：Python 生态中有大量的密码学"玩具库"和恶意包，用错一个就可能让整个系统形同虚设。

24.1 两大主力库

Python 生态中有两个最可靠的密码学库，它们的定位和适用场景不同：

库名 定位 安装 适用场景
cryptography 高层 API + 底层 hazmat pip install cryptography Fernet 加密、X.509 证书处理、TLS 包装
PyCryptodome 底层密码原语集合 pip install pycryptodome AES/RSA/ECC 直接操作、自定义协议

简单来说：cryptography 提供了更安全的高层 API（比如 Fernet 让你几乎不可能用错），而 PyCryptodome 让你能精细控制每一个密码学操作。前者适合 90% 的应用场景，后者适合协议实现和底层开发。

24.2 cryptography 库：Fernet 与 hazmat

Fernet 是 cryptography 库中最推荐的对称加密方案。它使用 AES-128-CBC 加密 + HMAC-SHA256 认证，解决了"加密但没认证"这个经典陷阱。

from cryptography.fernet import Fernet

# 生成密钥（必须妥善保管！）
key = Fernet.generate_key()
print(key)  # b'...' (Base64 编码的 32 字节密钥)

cipher = Fernet(key)

# 加密
token = cipher.encrypt(b"这是绝密消息")
print(token)  # b'gAAAAAB...'

# 解密
plaintext = cipher.decrypt(token)
print(plaintext.decode())  # "这是绝密消息"

Fernet 的最大优势是"防呆设计"：你不需要选择加密模式、不需要自己拼接 IV、不需要操心 padding、不需要分开计算 HMAC——所有这些容易出错的事情，Fernet 都帮你自动处理好了。加密后的 token 包含了 IV、密文和 HMAC 签名，解密时会自动验证完整性。

对于更复杂的场景，cryptography 提供了 hazmat（hazardous materials，危险材料）层——这个名字本身就是警告：这些 API 需要你具备密码学知识才能安全使用：

from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.primitives import padding
import os

# AES-256-GCM —— 推荐使用的对称加密模式
key = os.urandom(32)
iv = os.urandom(12)  # GCM 推荐 12 字节 IV

cipher = Cipher(algorithms.AES(key), modes.GCM(iv))
encryptor = cipher.encryptor()

plaintext = b"这是一段需要加密的消息"
ciphertext = encryptor.update(plaintext) + encryptor.finalize()
tag = encryptor.tag  # GCM 认证标签

print(f"密文: {ciphertext.hex()}")
print(f"认证标签: {tag.hex()}")
💡 实践经验：除非你明确知道自己需要定制某些参数，否则始终优先使用 Fernet 而非 hazmat。hazmat 的名字不是开玩笑的——一个常见的错误是用了 ECB 模式、或者忘记验证 GCM 的 tag、或者硬编码了 IV。这些错误在代码审查中很难发现，但攻击者可以轻易利用。

24.3 PyCryptodome：精细控制

PyCryptodome 是 PyCrypto 的活跃分支，提供了更丰富的底层算法支持。下面的例子展示如何用 RSA 公钥加密一个对称密钥（和上一章 OpenSSL 的混合加密完全对应）：

from Crypto.PublicKey import RSA
from Crypto.Cipher import PKCS1_OAEP, AES
import os

# 生成 RSA 密钥对
key = RSA.generate(2048)
private_key = key.export_key()
public_key = key.publickey().export_key()

# 用公钥加密对称密钥
session_key = os.urandom(32)
rsa_cipher = PKCS1_OAEP.new(RSA.import_key(public_key))
enc_session_key = rsa_cipher.encrypt(session_key)

# 用对称密钥加密大文件
aes_cipher = AES.new(session_key, AES.MODE_GCM)
plaintext = b"这是要加密的长文本..."
ciphertext, tag = aes_cipher.encrypt_and_digest(plaintext)

print(f"加密的会话密钥: {enc_session_key.hex()}")
print(f"密文: {ciphertext.hex()}")
print(f"认证标签: {tag.hex()}")

ECC 操作在 PyCryptodome 中同样直接：

from Crypto.PublicKey import ECC
from Crypto.Signature import DSS
from Crypto.Hash import SHA256

# 生成 ECC 密钥对
ecc_key = ECC.generate(curve='P-256')

# 签名
message = b"这条消息需要签名"
h = SHA256.new(message)
signer = DSS.new(ecc_key, 'fips-186-3')
signature = signer.sign(h)

# 验签
verifier = DSS.new(ecc_key.public_key(), 'fips-186-3')
try:
    verifier.verify(h, signature)
    print("签名验证通过！")
except ValueError:
    print("签名验证失败！")

24.4 实现加密聊天协议

学了这么多 API，我们来做一个有实际意义的项目——一个简单的端到端加密聊天协议。客户端和服务器之间传输的消息经过 AES-GCM 加密，密钥通过 ECDH 密钥交换协商：

# 伪代码：加密聊天协议的核心逻辑
# 1. 握手阶段
# 客户端生成临时 ECC 密钥对，发送公钥给服务器
# 服务器也生成临时 ECC 密钥对，返回公钥
# 双方计算 ECDH 共享密钥，派生 AES 会话密钥

# 2. 加密通信阶段
# 每条消息用一个随机 nonce（12字节）
# cipher = AES.new(session_key, AES.MODE_GCM, nonce=nonce)
# ciphertext, tag = cipher.encrypt_and_digest(message)
# 发送: nonce || ciphertext || tag

# 3. 解密
# cipher = AES.new(session_key, AES.MODE_GCM, nonce=nonce)
# plaintext = cipher.decrypt_and_verify(ciphertext, tag)
🔑 设计要点：（1）每个会话生成临时的 ECDH 密钥对，确保前向安全性——即使服务器长期私钥泄露，历史会话也无法解密；（2）每条消息使用独立的 nonce，绝对不能用重复的 nonce 加密不同的消息——GCM 中 nonce 重用会彻底摧毁安全性；（3）decrypt_and_verify 是原子操作，绝不能先解密后验证——攻击者可能构造密文让你解密出恶意内容。

24.5 hashlib 与 HMAC

Python 标准库中的 hashlib 提供了哈希和 HMAC 功能。虽然它的 API 简单，但正确使用仍然需要一些注意：

import hashlib
import hmac

# 简单哈希（不要直接用于密码存储！）
data = b"hello world"
sha256_hash = hashlib.sha256(data).hexdigest()
print(f"SHA-256: {sha256_hash}")

# 安全的密码哈希（使用专门的 KDF）
# hashlib.pbkdf2_hmac 内置了加盐和迭代
salt = hashlib.sha256(b"random_salt").digest()
dk = hashlib.pbkdf2_hmac('sha256', b'my_password', salt, 100000)
print(f"PBKDF2 密钥派生: {dk.hex()}")

# HMAC: 带密钥的消息认证
key = b"secret_key"
message = b"important message"
mac = hmac.new(key, message, 'sha256').hexdigest()
print(f"HMAC: {mac}")
⚠️ 常见错误：很多开发者直接用 SHA-256 哈希密码。这是一个严重的安全错误——没有加盐的哈希可以被彩虹表快速破解；即使加了盐，单次 SHA-256 也太快，每秒可以计算数亿次。始终使用专门的密码哈希函数：PBKDF2、bcrypt、scrypt 或 Argon2。hashlib.pbkdf2_hmac 可以胜任，但推荐使用专门的库如 bcrypt 或 argon2-cffi。

24.6 密钥安全存储

密码学中有一条铁律：密钥泄露 = 一切归零。在 Python 应用中有几个常见的密钥管理实践：

方法 安全等级 适用场景 注意事项
环境变量 低 开发环境、单服务 进程列表会暴露环境变量；子进程继承
密钥文件（600 权限） 中 服务启动时加载 需要保护文件系统访问；备份时注意
密钥派生（KDF） 中 从密码派生密钥 需要高强度 KDF 参数；用户密码不能太弱
HSM / 密钥管理服务 高 生产环境、合规要求 需要额外基础设施；AWS KMS / Azure Key Vault

一个实用的折中方案是用 python-dotenv 从 .env 文件加载密钥（注意：.env 永远不要提交到 Git 仓库！）：

# ENCRYPTION_KEY=your-base64-encoded-key-here

import os
from dotenv import load_dotenv
from cryptography.fernet import Fernet

load_dotenv()
key = os.environ["ENCRYPTION_KEY"].encode()
cipher = Fernet(key)

24.7 踩坑故事：PyPI 仿冒包事件

2020 年，安全研究人员在 Python 包索引（PyPI）上发现了一系列恶意包，它们采用了"仿冒"（typo-squatting）策略——包名和流行的密码学库极其相似：cryptographhy（多了一个 h）、cryptography2、pycryptographhy 等。

🕵️ 事件详情：这些恶意包在被导入时，会执行恶意的 setup.py 代码，窃取环境变量中的密钥、SSH 凭据、AWS 凭证等敏感信息，然后发送到攻击者控制的服务器。由于 Python 包安装时允许任意代码执行（setup.py 中的构建脚本），开发者只需要一个手误——比如 pip install cryptographhy 而不是 cryptography——整个开发环境的凭据就会被窃取。

类似的攻击在 npm（npm crossenv 恶意包感染了数千个项目）、RubyGems 等生态中反复上演。核心教训是：安装包之前，仔细检查包名。更安全的做法是使用 pip install --require-hashes 模式，配合 requirements.txt 中的哈希锁定，确保你安装的包确实是预期的版本。此外，使用 pipenv 或 poetry 这类工具可以自动锁定依赖关系并校验哈希。

🔑 三要三不要：要从官方源安装，要使用虚拟环境隔离项目，要锁定依赖版本和执行哈希校验。不要用 sudo pip install，不要在 requirements.txt 中写裸版本号（不带哈希），不要让 CI/CD 系统的 API token 暴露在环境变量中传给不可控的 pip 包。

用 Python 操作密码学 API 的能力是你工具箱中的又一件利器。在下一章中，我们将把目光投向密码学在区块链中的应用——比特币地址是如何从公钥生成的、以太坊交易的签名机制、以及零知识证明如何帮助区块链扩容。你会发现，前 23 章学到的一切，在区块链的世界中被以一种精妙的方式组合在了一起。
