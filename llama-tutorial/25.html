第25章 推理部署方案

经过前面24章的学习，你已经理解了LLaMA的架构、训练方法和微调技巧。但"能跑"不等于"能上线"。一个模型在开发机上生成一个句子需要3秒，和在生产环境中处理100个并发请求每个只需要200毫秒，中间隔着一整套推理优化和部署技术栈。

这一章我们不讨论算法原理——那些在前面的章节都讲过了。这一章是纯粹的工程实战：面对vLLM、Ollama、llama.cpp、TensorRT-LLM、TGI这些推理框架，你应该怎么选？不同的部署场景（实时对话、批处理、边缘设备）应该用哪套方案？生产环境中遇到显存OOM和请求排队又该怎么处理？

25.1 推理框架全景对比

2025年的LLM推理框架生态已经相当成熟，每个框架都有自己擅长的场景。先来看一张全景对比表：

框架 核心加速技术 支持GPU 支持CPU 量化格式 最佳场景
vLLM PagedAttention, continuous batching 是 否 GPTQ, AWQ, FP8 高并发GPU推理
Ollama llama.cpp后端, 开箱即用 是 是 GGUF 个人开发、本地部署
llama.cpp GGUF量化, CPU优化, Metal/CUDA 是 是 GGUF 边缘设备、CPU推理
TensorRT-LLM TensorRT编译优化, 内核融合 是 (NVIDIA only) 否 FP8, INT4, INT8 NVIDIA GPU极致性能
TGI continuous batching, 令牌水印 是 否 GPTQ, AWQ HuggingFace生态集成

选择的决策依据主要看三个维度：你的GPU（型号、数量、显存）、你的QPS要求、你的延迟要求。下面我们来逐一深入每个框架。

25.2 vLLM：PagedAttention 与高并发推理

vLLM是目前最流行的开源推理框架，其核心创新是PagedAttention——将KV Cache分页管理，解决了传统推理中显存碎片化和利用率低的问题。

在标准实现中，LLM推理的KV Cache是预先分配固定大小的连续显存块。但问题是，一个序列的实际长度在生成完成之前是未知的，所以你必须按最大长度（如4096 tokens）来分配。这导致两个问题：第一，很多序列的实际长度远小于最大值，造成了显存浪费；第二，显存碎片化严重，多个请求的KV Cache大小不同，分配和释放后留下大量不可用的小碎片。

PagedAttention将KV Cache划分为固定大小的"页"（page），每页可以存储若干个token的KV值。不同序列的页可以散落在显存的任意位置，通过页表来索引。这就跟操作系统的虚拟内存管理是一个道理——解决了外部碎片问题，实现了接近100%的显存利用率。

# vLLM 部署 LLaMA 3-8B
# 安装: pip install vllm

# 启动API服务器（命令行）
# 单卡部署8B模型，支持OpenAI兼容API
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3.2-8B \
    --tensor-parallel-size 1 \       # 单卡部署
    --max-model-len 8192 \            # 最大上下文长度
    --gpu-memory-utilization 0.90 \   # GPU显存利用率
    --dtype bfloat16 \
    --port 8000

# 启动后，你就可以用OpenAI SDK来调用
# 相当于本地搭建了一个OpenAI兼容API服务器
# 客户端调用（OpenAI兼容API）
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="sk-no-key-required"     # vLLM不验证API key
)

# 流式输出
stream = client.chat.completions.create(
    model="meta-llama/Llama-3.2-8B",
    messages=[
        {"role": "user", "content": "解释一下PagedAttention的工作原理"}
    ],
    temperature=0.7,
    max_tokens=512,
    stream=True
)

for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)

vLLM的核心优势还包括continuous batching——在生成过程中动态调度批次。传统批处理必须等整个批次的所有序列都完成才能进行下一次调度，而continuous batching可以在序列粒度上做调度：某个序列生成到EOS后就立即腾出位置给新请求，不影响还在生成的序列。这使得vLLM在高并发场景下的吞吐量可以达到普通HuggingFace实现的10-20倍。

性能数据： 在单张A100-80G上部署LLaMA 3-8B，vLLM可以在8K上下文下达到约2500 tokens/s的生成吞吐量（batch size = 64），同时首token延迟控制在100ms以内。对比naive的HuggingFace实现，相同硬件下的吞吐量约150 tokens/s。vLLM的吞吐量优势随着并发数增大而更加明显。

25.3 Ollama：个人开发者的最佳选择

如果你只是想在自己的电脑上跑LLaMA做实验，或者搭建一个个人用的AI助手，Ollama是上手最快、体验最好的方案。它底层基于llama.cpp，但封装成了一套极简的命令行工具。

# 安装Ollama（Linux/macOS）
curl -fsSL https://ollama.com/install.sh | sh

# 拉取并运行LLaMA 3.2-8B
ollama pull llama3.2:8b    # 下载模型（第一次耗时较长）
ollama run llama3.2:8b     # 启动交互式对话

# 就这么简单——两条命令，一个可对话的LLaMA就在你的终端里了

Ollama的Modelfile机制让你可以快速自定义模型行为，而无需写任何代码：

# Modelfile 示例：创建一个自定义的医疗助手
FROM llama3.2:8b

# 设置系统提示词
SYSTEM """你是一个专业的医疗助手。
请基于可靠的医学知识回答用户问题。
如果不知道答案，请明确说'我不确定'而不是编造。"""

# 设置推理参数
PARAMETER temperature 0.3      # 低温度，更确定的回答
PARAMETER top_p 0.9
PARAMETER num_ctx 8192         # 上下文窗口大小

# 创建自定义模型
ollama create medical-assistant -f ./Modelfile
ollama run medical-assistant

Ollama也提供了OpenAI兼容的API接口，方便集成到你的应用中：

# 启动Ollama API服务（默认监听11434端口）
ollama serve

# 调用API
curl http://localhost:11434/v1/chat/completions \
  -d '{
    "model": "llama3.2:8b",
    "messages": [{"role": "user", "content": "你好，请介绍一下自己"}],
    "stream": true
  }'

Ollama的局限性在于不适合高并发生产环境。它是单机单服务的架构，缺乏vLLM那样的高级调度策略。对于个人使用或小团队内部的部署来说完全够用，但如果要支撑数十万的DAU，还是建议使用vLLM或TensorRT-LLM。

25.4 llama.cpp 与 GGUF 格式

llama.cpp是LLM推理领域的一个革命性项目。它的最大贡献是实现了纯CPU推理——这意味着你不需要昂贵的NVIDIA GPU就能运行LLaMA模型。

llama.cpp的核心是GGUF（GPT-Generated Unified Format）格式。GGUF是一种专门为推理优化的模型存储格式，它将模型权重、分词器配置、超参数等所有信息打包成一个文件。搭配llama.cpp的量化工具，你可以将模型量化为各种精度：

量化级别 每参数位数 8B模型大小 70B模型大小 质量损失 推荐场景
Q4_K_M 4.5 bit ~4.5 GB ~39 GB 轻微 兼顾质量和大小
Q5_K_M 5.5 bit ~5.3 GB ~46 GB 很小 高质量需求
Q8_0 8 bit ~8.5 GB ~73 GB 几乎无损 精度敏感任务
Q2_K 2.5 bit ~2.9 GB ~24 GB 明显 极限压缩

Q4_K_M 是社区最广泛使用的量化级别——它在模型质量和文件大小之间取得了最好的平衡。LLaMA 3-8B原始bfloat16权重约16GB，Q4_K_M量化为4.5GB，压缩比超过3.5倍，而MMLU分数只下降不到2%。

# llama.cpp 使用示例

# 1. 下载GGUF模型（或者自己量化）
# 从HuggingFace下载已经量化好的GGUF文件
wget https://huggingface.co/bartowski/Llama-3.2-8B-Instruct-GGUF/resolve/main/Llama-3.2-8B-Instruct-Q4_K_M.gguf

# 2. 编译llama.cpp
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
make -j

# 3. 运行推理（纯CPU）
./llama-cli \
    -m Llama-3.2-8B-Instruct-Q4_K_M.gguf \
    -p "解释一下什么是Transformer注意力机制" \
    -n 256 \
    -t 8 \            # 使用8个CPU线程
    --temp 0.7

# 4. GPU加速推理（需要CUDA支持）
# 编译时启用CUDA: make LLAMA_CUDA=1 -j
./llama-cli \
    -m Llama-3.2-8B-Instruct-Q4_K_M.gguf \
    -p "解释一下什么是Transformer注意力机制" \
    -n 256 \
    -ngl 35 \          # 将35层卸载到GPU
    --temp 0.7
使用建议： llama.cpp在Apple Silicon（M1/M2/M3/M4）上的表现尤其出色。因为Apple统一内存架构允许GPU直接访问系统内存，一个M2 Ultra（192GB统一内存）甚至可以运行LLaMA 3-70B的Q4量化版本，这在同等价位的NVIDIA硬件上是做不到的。对Mac用户来说，llama.cpp + GGUF是目前性价比最高的本地LLM方案。

25.5 推理优化技术

除了框架层面的优化，还有几种通用的推理优化技术值得了解：

Continuous Batching（连续批处理）：前面在vLLM部分已经介绍过。它的核心思想是"不让GPU闲着"。在传统的批处理中，一批请求必须全部处理完才能接收新请求。而continuous batching在序列级别做调度——每个序列生成一个token后就检查是否需要插入新序列。这样可以最大化GPU利用率，将吞吐量提升5-10倍。

Speculative Decoding（推测解码）：标准自回归生成每次只生成一个token，无法利用GPU的并行计算能力。推测解码的思路是用一个更小的草稿模型（draft model）一次生成K个候选token，然后用目标LLaMA模型并行验证这些token的正确性。如果草稿模型足够准确（通常覆盖率达到80-90%），则推理速度可以提升2-3倍。在LLaMA场景中，可以使用LLaMA 3.2-1B或3B作为草稿模型来辅助8B或70B的推理。

# vLLM 中启用 speculative decoding
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3.2-8B \
    --speculative-model meta-llama/Llama-3.2-1B \   # 草稿模型
    --num-speculative-tokens 5 \                     # 每次推测5个token
    --tensor-parallel-size 1

KV Cache 量化：长上下文推理时，KV Cache的显存消耗是最大的瓶颈。以8B模型、128K上下文为例，仅KV Cache就需要约32GB显存（bfloat16精度）。通过将KV Cache量化为FP8或INT8，可以将其减半（16GB或8GB），而质量损失极小。目前vLLM和TensorRT-LLM都支持KV Cache量化。

25.6 部署场景选择决策树

面对这么多框架和技术，选择困难症犯了怎么办？下面这个决策树可以帮助你快速定位最适合的部署方案：

你的部署场景是？
│
├─ 个人实验 / 本地开发
│   └─ Ollama（最简单）
│      可选: llama.cpp（需更多控制）
│
├─ 生产环境高并发（QPS > 100）
│   ├─ 有NVIDIA GPU → vLLM（通用最佳）
│   └─ 需要极致性能 → TensorRT-LLM（仅NVIDIA）
│
├─ 边缘设备 / 无GPU
│   └─ llama.cpp + GGUF Q4_K_M
│
├─ 中文场景 / 快速部署
│   └─ Ollama + GGUF（或vLLM + AWQ量化）
│
└─ HuggingFace深度集成
    └─ TGI（Text Generation Inference）

对于一个典型的SaaS产品来说，推荐的部署架构是：vLLM作为推理引擎 + OpenAI兼容API + 水平扩展。vLLM提供了最高的性能和最成熟的生态，OpenAI兼容API让你的应用代码与模型解耦，将来切换不同模型时客户端代码完全不需要修改。

25.7 API 设计最佳实践

在生产环境中部署LLaMA模型时，API的设计质量直接决定了系统的可用性和可维护性。以下是几条核心最佳实践：

OpenAI 格式兼容：无论你使用哪个推理框架，API接口都应该兼容OpenAI的格式。这样做的好处是：所有支持OpenAI的生态工具（如LangChain、LlamaIndex、各类Chat UI）都能无缝接入你的服务。vLLM、Ollama、TGI都原生支持这一格式。

流式输出（Streaming）：LLM推理的时间通常在1-10秒级别，如果不做流式输出，用户体验就是"发送消息 -> 等待 -> 一次性显示所有内容"。而流式输出让用户看到token逐个出现，感知延迟从"等待10秒"变成了"思考时间1秒+逐字显示9秒"，体验大幅提升。所有主流框架都支持SSE（Server-Sent Events）流式输出。

多轮对话管理：LLaMA模型的API本身是无状态的——每次调用都是独立的。多轮对话需要在客户端维护历史消息列表，并在每次请求中发送完整的历史。如果不加控制，历史越长占用的上下文窗口就越长，推理成本线性增长。典型策略是：保留system prompt + 最近N轮对话（通常N=10-20轮），超出部分被截断或摘要压缩。

# 多轮对话的上下文管理
MAX_HISTORY = 10  # 保留最近10轮对话

messages = [
    {"role": "system", "content": "你是一个有用的助手。"}
]

def add_message(role, content):
    messages.append({"role": role, "content": content})
    # 截断历史：保留system + 最近MAX_HISTORY*2条消息
    if len(messages) > MAX_HISTORY * 2 + 1:
        system = messages[0]
        messages[:] = [system] + messages[-(MAX_HISTORY * 2):]

# 每次请求时压缩token数
def estimate_tokens(messages):
    return sum(len(m["content"]) for m in messages) * 1.3  # 中文字符≈1.3 token

# 如果超过限制，移除最早的非system消息
MAX_TOKENS = 6000
while estimate_tokens(messages) > MAX_TOKENS and len(messages) > 1:
    messages.pop(1)  # 移除最早的用户/助手对话
踩坑故事： 我们的一个AI客服项目上线第一天，服务就挂了。排查发现是两个问题叠加导致的。第一，某个用户连续发了100条消息，历史消息列表膨胀到远超上下文限制，导致推理OOM。第二，vLLM的队列没有做限制，当所有worker都在处理长上下文请求时，新来的短请求也必须等待前面的长请求完成——这叫"饥饿"，是continuous batching的一个经典问题。

解决方案是三层限制：应用层限制每轮对话的消息数（最多20轮）；推理层设置 max_num_batched_tokens 限制单批次总token数；队列层设置超时机制，对于等待超过30秒的请求直接返回错误（让客户端重试）。从那以后，服务再也没因为这个原因挂过。

25.8 性能对比与选型建议

最后，我们用一个实际场景来对比各框架的性能差异。测试环境：单张A100-80G，LLaMA 3-8B，8K上下文，请求输入长度1024 tokens，输出长度256 tokens。

框架 首token延迟 吞吐量 (tokens/s) 最大并发 显存占用 部署难度
HuggingFace (naive) ~150ms ~150 低 ~17GB 低
vLLM ~80ms ~2500 高（64+） ~18GB 中
Ollama (GGUF Q4_K_M) ~120ms ~800 中 ~5GB 低
TensorRT-LLM (FP8) ~60ms ~3200 高（64+） ~12GB 高
TGI ~90ms ~2000 中 ~17GB 中

我的选型建议：

  • 个人开发 / 学习实验： Ollama。5分钟上手，支持所有主流模型，API兼容OpenAI格式，后续迁移到vLLM只需改base_url。
  • 中小规模生产（DAU < 10万）： vLLM。性能足够好，生态最成熟，社区支持最活跃。在单张A100-80G上用vLLM部署8B模型，可以支撑大约50-100的并发请求。
  • 大规模生产（DAU > 100万）： TensorRT-LLM + vLLM混合。TensorRT-LLM用于延迟敏感的场景（如实时对话），vLLM用于吞吐优先的场景（如离线批处理）。
  • 边缘部署 / 无GPU： llama.cpp + GGUF Q4_K_M。在Apple M系列芯片上体验极佳，在x86 CPU上也能达到可用的推理速度（约5-10 tokens/s）。
