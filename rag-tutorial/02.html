第二章：大语言模型基础

大家好，欢迎回到课程。第一章我们全景式地了解了RAG——它是什么、为什么需要它、它经历了怎样的演进。但要真正理解RAG，我们首先得理解它服务的对象——大语言模型（LLM）。如果LLM是RAG的"大脑"，那我们必须搞清楚这个大脑的运作方式，才能知道如何给它喂最好的"养料"。

这一章我们不讲太深的数学——你不会看到矩阵求导或者注意力公式的推导。我们的目标是建立关于LLM的直观理解，特别是那些和RAG设计直接相关的知识点。相信我，理解了这一章，你后面做检索增强时会多出很多"原来如此"的时刻。

一、Transformer与注意力机制：直觉层面的理解

所有现代大模型的核心都是2017年Google提出的Transformer架构。Transformer最革命性的组件是自注意力机制（Self-Attention）。用一句话概括它：对于序列中的每个词，模型会决定其他所有词对它的"重要程度"，然后基于这些重要程度来融合信息。

举个具体的例子。句子"那只没有牵绳的狗追着猫跑，它跑得很快"——这个"它"指的是狗还是猫？人类通过常识知道，狗追猫所以跑得快的是狗。Transformer的注意力机制做的就是类似的事情：在编码"它"这个词时，模型会给"狗"更高的注意力权重，因为"狗"和"它"在语法和语义上最相关。通过这种机制，Transformer可以建立长距离的依赖关系，这是之前RNN和LSTM难以做到的。

在RAG的语境下，注意力机制的意义在于：当我们将检索到的文档片段拼接到用户问题前面时，LLM通过注意力机制可以"聚焦"到相关段落上，忽略不相关的噪音。这就是为什么RAG的prompt设计如此重要——上下文的质量直接影响注意力分配的质量。

二、从GPT到Claude：大模型族谱

了解当前主流模型的谱系，有助于你选择RAG系统中的"生成引擎"。

模型家族 代表模型 架构特点 上下文窗口 RAG适配度
GPT系列 GPT-4o, GPT-4o-mini Dense Transformer, RLHF 128K tokens 极高，生态成熟，function calling稳定
Claude系列 Claude 3.5 Sonnet, Opus Constitutional AI, 长上下文 200K tokens 极高，长文档理解突出
Gemini系列 Gemini 1.5 Pro MoE, 原生多模态 1M tokens 极高，海量窗口适合大规模RAG
Llama系列 Llama 3.1 70B, 405B 开源Dense, GQA 128K tokens 高，可本地部署，适合隐私场景
Qwen系列 Qwen2.5 72B 开源，中英双语优化 128K tokens 高，中文场景表现优秀
DeepSeek系列 DeepSeek-V3, R1 MoE, 推理增强 128K tokens 高，推理能力突出，适合复杂RAG

选择"生成引擎"的核心考量：推理成本、上下文窗口大小、对中文的支持程度、以及是否支持structured output / function calling。对于大多数RAG应用，GPT-4o-mini或Claude 3.5 Haiku是最具性价比的选择——它们足够聪明，又足够便宜。

三、LLM为什么会幻觉？深入理解Token生成机制

要理解幻觉，先要理解LLM是怎么"说话"的。

LLM的本质是一个next token predictor。给定一段文本，它预测下一个最可能出现的token（一个token大约对应0.75个英文单词或0.5个汉字）。它逐token地生成回答，每次生成都基于之前生成的所有内容。

# Tokenization 演示 — 感受一下文本如何被切分成token
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")

text = "RAG combines retrieval with generation to improve LLM accuracy."
tokens = tokenizer.tokenize(text)
token_ids = tokenizer.encode(text)

print("原始文本:", text)
print("Token序列:", tokens)
print("Token数量:", len(tokens))
print("对应ID:", token_ids)

# 输出：
# 原始文本: RAG combines retrieval with generation to improve LLM accuracy.
# Token序列: ['rag', 'com', '##bines', 'retrieval', 'with', 'generation',
#             'to', 'improve', 'll', '##m', 'accuracy', '.']
# Token数量: 12

看到"combines"被拆成了"com"和"##bines"、"LLM"被拆成了"ll"和"##m"吗？这就是subword tokenization。这个机制有两个重要后果：

  • 概率性本质：模型在每个位置输出一个概率分布，然后采样——这意味着同一个问题每次可能得到不同答案。采样参数（temperature, top-p）控制"创造性"和"确定性"的平衡。
  • 无内部事实数据库：模型没有"去查事实表"这样的内部操作。所有"知识"都编码在数十亿参数中，表现为统计规律。当你问"爱因斯坦的生日是哪天"，模型不是在"回忆"，而是在计算"在'爱因斯坦的生日是'这个前缀下，最可能出现的token序列"。

这就解释了幻觉的根本原因：模型会优先输出统计上最合理的回答，而不是事实上正确的回答。当某个事实在训练数据中出现频率不高，或者模型混淆了相似事实时，就会产生幻觉。

实战踩坑：我有一个朋友在做医疗问答RAG系统时，发现模型偶尔会产生"看起来非常专业但完全是编造"的医学建议。追查发现，有的检索片段中提到了"阿司匹林用于心脏病预防"，而另一个片段中提到了"儿童退热用药建议"，模型在注意力机制中把两段信息错误关联，编造出了"阿司匹林可用于儿童退热"的危险建议。这个教训告诉我们：RAG中的检索质量直接影响生成安全，绝不能忽视检索结果中的噪音。

四、上下文窗口与Tokenization基础

每个LLM都有一个上下文窗口（Context Window）——它能"同时看到"的最大token数量。这个窗口既包括用户输入的prompt，也包括模型生成的回答。

上下文窗口对RAG设计有决定性影响：

  • 窗口大小决定了你能塞多少检索结果。如果窗口只有8K tokens，留给检索结果的空间可能只有5-6K tokens，大约3-5个详细的文档片段。
  • 中间信息的"迷失"问题：研究表明（Liu et al., 2023），LLM对长上下文中的信息利用是不均匀的——开头和结尾的信息更容易被注意到，中间部分常常被"忽略"。这意味着在构造RAG prompt时，最相关的检索结果应该放在最前面或最后面。
  • Token计数是RAG的成本核心。每个token都要花钱（推理成本），且随着检索内容增加线性增长。优化RAG系统很大程度上是在优化"用最少的token传递最多的信息"。
实战技巧：在处理长文档RAG时，不要一股脑地把所有检索结果塞进prompt。尝试以下策略：先检索top-k（例如20个片段），然后用一个轻量重排序模型挑选top-3到top-5，最后只把精选后的结果送入LLM。实践表明，5个高质量的片段通常比20个带噪音的片段效果更好，而且成本只有后者的四分之一。

本章小结

我们理解了Transformer注意力机制的核心直觉——模型如何决定"关注什么"；了解了主流LLM家族的谱系和各自特点；深入到了token生成层面，明白了幻觉产生的根本原因；并讨论了上下文窗口对RAG设计的关键影响。掌握这些基础，你就具备了设计RAG系统所需的核心认知框架。下一章，我们将正式进入RAG的核心架构——看看四个阶段是如何协同工作的。
