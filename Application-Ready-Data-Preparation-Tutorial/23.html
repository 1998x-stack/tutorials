为什么去重让模型更好

大家好，我是你们的讲师。有没有想过一个问题：你用 15T tokens 训练了一个模型，其中有多少是真正有效的信息？更尖锐地问——有多少是你已经在其他数据里见过的？

2022 年，一篇题为 "Deduplicating Training Data Makes Language Models Better" (Lee et al., 2022) 的论文引爆了这个方向。研究团队发现，在 C4 数据集中，仅仅做一次精确去重，就能让模型在多个基准上的困惑度降低 1-2 个点。更令人震惊的是，他们估算 Common Crawl 这样的超大语料中，重复率高达 30-50%。也就是说，你花几百万美元训练模型，小一半的算力都在学已经学过的东西。

理解去重的价值，关键在于理解"重复"的不同形态：有的重复是精确的——同一段文字出现在不同的 URL 中；有的是近似的——两段文档 90% 相似，只是段落顺序做了调整；有的是语义的——维基百科和百度百科对同一概念的描述完全不同文，但说的是一件事。对付不同形态的重复，需要完全不同的武器。

核心洞察： "重复数据不仅浪费算力，更致命的是它会扭曲模型的概率分布。如果某段话出现 100 次而另一段同等重要的内容只出现 1 次，模型会错误地认为前者的重要性是后者的 100 倍。去重的本质，是恢复数据的真实分布。"

精确去重与近似去重

去重策略可以按照精度从高到低排列，每种策略都有自己的适用场景和成本。我们先从最基础说起。

精确去重：Exact Match

精确去重是最朴素也是最有效的方法。它的核心思想是：如果两篇文档的文本完全相同（或哈希值相同），只保留一份。具体实现通常用 sha256 对文档内容做哈希，然后通过布隆过滤器或哈希表判断重复。

但精确去重的局限性也很明显——稍微改动一个标点符号、添加一个空格，两篇几乎一模一样的文章就会被认为是"不同"的。这就是近似去重登场的理由。

MinHash + LSH：近似去重的基石

MinHash (Broder, 1997) 是近似去重领域最经典的技术。它的思路很巧妙：把文档转化为一组特征（通常用 n-gram 或 shingle），然后用多个独立的哈希函数为每个文档生成一个"签名"（signature）。两篇文档的 Jaccard 相似度，就可以通过它们的 MinHash 签名相等的比例来近似估计。

为了提高效率，MinHash 通常和 LSH（Locality-Sensitive Hashing，局部敏感哈希） 配合使用。LSH 将签名矩阵分桶（banding），只有落在同一桶中的文档对才需要计算真正的相似度，避免了 O(n²) 的两两比较。

下面是精确去重和近似去重主要策略的对比：

下面是 MinHash + LSH 的一个简化 Python 实现示例：

import hashlib
import random

def minhash_signature(text, num_hashes=128, shingle_size=5):
    """为一段文本生成 MinHash 签名"""
    # 生成 shingle（n-gram）
    shingles = set()
    for i in range(len(text) - shingle_size + 1):
        shingle = text[i:i+shingle_size]
        shingle_hash = hashlib.md5(shingle.encode()).hexdigest()
        shingles.add(int(shingle_hash, 16))

    # 生成随机哈希函数（使用 universal hashing）
    seed = 42
    rng = random.Random(seed)
    a = [rng.randint(1, 2**31) for _ in range(num_hashes)]
    b = [rng.randint(0, 2**31) for _ in range(num_hashes)]
    p = 2**31 - 1

    signature = []
    for i in range(num_hashes):
        min_hash = min((a[i] * s + b[i]) % p for s in shingles)
        signature.append(min_hash)
    return signature

def lsh_bucket(signature, bands=16, rows_per_band=8):
    """将签名分桶，用于近邻搜索"""
    buckets = {}
    for b in range(bands):
        start = b * rows_per_band
        end = start + rows_per_band
        band_hash = hashlib.md5(
            str(signature[start:end]).encode()
        ).hexdigest()
        if band_hash not in buckets:
            buckets[band_hash] = []
        buckets[band_hash].append(b)
    return buckets

在实际大规模应用中，业界普遍使用 datasketch 或 spark-lsh 等成熟库来加速。例如，HuggingFace 的 Datatrove 和 AI2 的 Dolma 工具链都将 MinHash + LSH 作为标准配置。

语义去重

精确和近似去重只能处理文本层面的重复，但真正的挑战在于语义层面的重复——内容相同、表达不同。这时，我们需要语义去重。

SemDeDup：语义级去重

SemDeDup (Abbas et al., 2023) 利用预训练模型的嵌入表示来检测语义相似的文档。具体来说，它使用 Sentence-BERT 等模型将文档编码为稠密向量，然后通过 K-Means 聚类将文档分组，在同一簇内基于余弦相似度去除冗余。

SemDeDup 的一个重要发现是：不需要把所有的语义重复都去掉。论文实验表明，移除 50% 的语义重复即可获得大部分收益，继续增加去重力度反而会损害模型在多样性任务上的表现。他们建议将去重阈值设置在 0.7-0.8 的余弦相似度范围内。

FairDeDup 与 D4

FairDeDup (Yun et al., 2024) 在 SemDeDup 的基础上增加了群体公平性约束。它的核心洞察是：无差别的语义去重可能会系统性移除某些群体的代表性样本。例如，对某个少数群体的文本，由于表达方式较为相似，去重算法可能会将它们全部判定为"冗余"而删除，导致模型对该群体的理解能力下降。FairDeDup 通过在聚类去重过程中引入分组感知的正则化项来解决这个问题。

D4 (Diverse Dedup for Datasets) 则是另一个有趣的方向。它关注的不再是"两篇文档是否重复"，而是"移除这篇文档会降低多少数据集的总体多样性"。D4 利用行列式点过程 (DPP) 来量化每篇文档对数据集多样性的边际贡献，从而在去重时保留最有"差异性"的样本。

实践建议： 在大规模数据工程中，推荐采用"三级去重"流水线——先做精确去重（成本最低），再做 MinHash 近似去重，最后根据预算和需求决定是否做语义去重。每一级都用上一级的输出作为输入，这样可以大幅降低语义去重的计算量。

质量过滤策略

去重去掉的是重复，过滤去掉的是低质量。在预训练场景中，"低质量"的定义五花八门：垃圾广告、机器生成的无意义文本、格式混乱的文档、包含有害内容的页面……不同的过滤策略捕捉不同维度的质量问题。

困惑度过滤

困惑度 (Perplexity, PPL) 过滤是最经典的质量信号之一。它的想法很简单：用一个已经训练好的语言模型来评估每篇文档——如果模型对这篇文档"感到困惑"（PPL 很高），说明文档质量可能很低。

C4 数据集就是使用困惑度过滤的典型代表。他们用 KenLM 在 Wikipedia 文本上训练一个 5-gram 语言模型，然后过滤掉 PPL 过高的文档。但这有一个方法论上的隐患：语言模型会偏好自己训练数据相似类型的文本。用 Wikipedia 训练的 PPL 模型天然会偏好 encyclopedia 风格，而对高质量的论坛讨论、技术博客产生误判。

SuperFiltering：弱到强过滤

SuperFiltering (Li et al., 2024) 提出了一种巧妙的"弱到强"过滤范式。它的核心洞察是：训练一个弱小的分类器（比如使用 n-gram 特征的逻辑回归或小 BERT）来识别高质量数据，然后用这个弱分类器去过滤更大规模的数据。这个"弱"模型对训练数据的"偏见"反而成了优势——它的学习边界比大模型更保守，不会因为看了太多数据而对噪声不敏感。

具体实现上，SuperFiltering 的流程如下：人类专家标注几千条高质量/低质量的样本，训练一个小型质量分类器（比如 DistilBERT），然后用这个分类器去评分整个数据集。实验证明，这种方法可以用极少的标注成本（几千条）达到与百万级标注接近的过滤效果。

QuRating

QuRating (Wettig et al., 2024) 则提出了一个更有趣的角度：与其用单一质量分数，不如让模型对文档在多个维度上评分——知识密度、教育价值、叙事流畅度、事实准确性等。每个维度独立评分，然后根据下游任务的需求加权组合。这种多维度评分在需要精细控制数据配比时非常有用。

去重与过滤的平衡艺术

去重和过滤不是越激进越好。实际上，过度去重和过度过滤都会带来反效果。

FLUX (Foucart et al., 2025) 的研究发现了一个反直觉的结果：当前主流的数据去重和过滤策略，平均浪费了约 25% 的可用有效 token。也就是说，大量的有用信息被去重或过滤算法错误地丢弃了。FLUX 的方法是通过对每个文档评估"信息边际贡献"来决定是否保留，而不是依赖于硬阈值。

数据要"够脏"还是"够干净"？ 现实的答案是——两者都要。预训练阶段适度包含一些"次优"数据反而能提升模型的鲁棒性。关键的平衡技巧包括：

  • 域自适应过滤阈值：不同领域的数据质量分布差异很大，应该为每个域设置独立的质量阈值
  • 分层去重：对高质量语料（如书籍、论文）使用宽松去重，对低质量语料（如网页）使用严格去重
  • 渐进式过滤：先做宽过滤保留尽可能多的数据，随着训练进展逐步收紧
  • 多样化保留：在去重时考虑多样性指标，确保每个主题和风格都有足够的代表样本

实战案例：去重太狠，模型成了"复读机"

踩坑实录： 某团队在训练一个代码大模型时，对训练数据做了激进的文档级去重（阈值设为 0.6 的 MinHash 相似度）。结果模型在生成代码时表现出严重的"复读机"现象——当用户请求"写一个排序算法"时，模型只输出冒泡排序；当请求"写一个 web server"时，模型永远输出 Flask hello world。检查发现，去重阈值太严，导致所有"不同实现方式但核心逻辑相似"的代码都被去掉了。剩下的都是"最典型的"示例，模型的生成多样性彻底丧失。将阈值调整到 0.85 后问题解决。

这个案例提醒我们：去重的目标不是最小化重复率，而是最大化数据集的信息密度和多样性。一个好的去重策略应该像一位编辑——保留不同角度的报道，而不是只留下一篇"代表性"文章。

好了，去重和过滤为我们准备好了高质量的干净数据。但下一个问题随之而来：不同来源、不同质量的干净数据，应该按什么比例混合才能达到最佳训练效果？这正是我们下一章要深入探讨的话题——数据混合法则与域选择。
