刷了100题还是感觉不会？那是方法不对

同学们好，你可能经历过这样的挫败感：每天刷两三道题，刷了三四个月，LeetCode 上显示"已解 350 题"。但打开一场周赛，Q2 还是卡住；面试面了一道"Medium 原题"，你居然想不出最优解。于是你开始怀疑自己的智商——"我是不是不适合写代码？"

停。你不是笨。你的问题只有一个：光刷不总结，光做不复盘。这就像健身——你每天都去健身房，但每个动作只做一遍就换下一个，从不加重量，从不做组间休息，三个月后你发现肌肉一点没长。你说是健身没效果，还是你的方法有问题？

残酷的真相：刷题数和面试通过率的相关性远比你想象的低。我辅导过的学员中，有人刷了 800 题还在挂面试，有人只刷了 120 题就拿到了大厂 offer。差别在于后者会建立知识体系，而前者只是在堆砌题量。

五遍刷题法

我强烈推荐的刷题方法，来自 Google 前工程师、LeetCode 著名答主"耗子叔"的"五遍刷题法"的改进版本。核心思想：一道好题至少做五遍，每一遍的目标不同。

第一遍：暴力解与学习

目标：先不管效率，写一个能过的版本。如果不会，直接看题解——不是抄，而是理解了思路之后关掉题解自己写。

这一遍的意义在于：让你知道"这道题用了什么知识点"。比如你知道某个题用了"并查集 + 路径压缩"，你要顺便把并查集的原理复习一遍。

第二遍：优化

目标：写出最优解，并分析时空复杂度。如果最优解涉及你没用过的数据结构（比如线段树），花时间学明白。

这一遍要做的不仅仅是写代码——你还要 口头讲出来。找一个朋友，或者对着镜子，用中文把解题思路讲一遍。"这道题的核心是……我选择这个数据结构的理由是……边界条件是……"——你会发现，讲出来比写出来难得多，但也有效得多。

第三遍：隔天重做

目标：测试记忆。打开题目，不看之前的代码，在 15-20 分钟内 AC。如果做不到，说明你上次没真懂。重新学习。

这一遍是很多人跳过的一步。他们觉得"我看过了，我都懂了"。但你真的懂了吗？隔一天再写一遍，很多"我以为我懂了"的地方会原形毕露——奇奇怪怪的 corner case、边界条件的一格之差、排序方向选反……这些问题只有在独立重做时才会暴露。

第四遍：一题多解

目标：同一道题，用至少两种完全不同的方法解出来。比如 "Two Sum" 可以用暴力法、HashMap、排序+双指针，每种都要写一遍。

为什么？因为面试时你可能会被问到"能不能换个思路"或者"如果内存不够用怎么办"。只有你尝试过不同解法，才能从容应对。

第五遍：周回顾

目标：一周后回过头来，把本周做过的所有题再看一遍。不需要重写——但要能在 5 秒内说出每道题的核心套路和最优解的时间复杂度。

这一遍训练的是快速分类能力。当你看到一道新题，能脱口而出"这题不就是 DP + 记忆化搜索吗？"的时候，你就真的内化了。

高效实践：五遍刷题法不是让你每一道题都刷五遍。把题目分为三档——S 级（经典高频，如 Two Sum、LRU Cache）、A 级（中频但套路典型）、B 级（偏题怪题）。S 级五遍全做，A 级做到第三遍，B 级一遍带过即可。

遗忘曲线与复习策略

德国心理学家艾宾浩斯有一个著名的遗忘曲线：学完一个知识后，20 分钟后遗忘 42%，1 小时后遗忘 56%，1 天后遗忘 74%，1 周后遗忘 77%。应用到刷题场景，就是——你刷完一道题如果不复习，一周后你大概率只记得"我做过这道题"，但完全想不起解法细节。

怎么对抗遗忘？两个工具：间隔重复和错题本。

间隔重复

不是每天重复，而是按递增的时间间隔重复：

  • 第 1 次：学完当天重做
  • 第 2 次：1 天后
  • 第 3 次：3 天后
  • 第 4 次：7 天后
  • 第 5 次：14 天后

你可以用 Anki（间隔重复软件）或者 LeetCode 自带的"收藏"功能来管理复习计划。不要靠脑子记——你永远会高估自己的记忆力。

错题本体系

错题本不是"这道题我交了 WA 就记下来"。它应该包含以下信息：

  1. 题目编号和链接——方便回顾
  2. 错误原因——是思路错了？corner case 没处理？还是代码实现 bug？要把具体原因写清楚
  3. 正确解法概述——用自己的话写，不需要贴完整代码
  4. 学到了什么教训——这个教训可以泛化到其他题吗？
  5. 复习时间戳——上次复习是什么时候，下次计划什么时候复习
# 错题本示例格式（可以用 Markdown 或 Notion 管理）
#
# ## 错题记录
#
# ### 2026-06-15  LeetCode 875 (Koko Eating Bananas)  ⭐ Medium
# - 错误原因: 直接想求 k，没有意识到可以用二分答案
# - 正确思路: 对 k 进行二分搜索，用 can_finish(k) 验证
# - 教训: "求最小满足条件的值" -> 二分答案模式
# - 下次复习: 2026-06-22
#
# ### 2026-06-18  LeetCode 42 (Trapping Rain Water)  ⭐ Hard
# - 错误原因: 双指针移动时机搞反了
# - 正确思路: 维护左右最大高度，矮的一侧移动
# - 教训: 双指针题写之前先在纸上画指针移动流程
不要过度管理：错题本不是让你当展览馆的。不要花大量时间把笔记做得花里胡哨（画图、排版、截图）。核心是：记录下来，定期回看。如果记了笔记但再也没翻开过，那就是在骗自己。

一题多解：从暴力到最优的思维训练

一题多解是锻炼思维广度的最好方法。我们来看一个经典例子——Best Time to Buy and Sell Stock（LeetCode 121）：给定股票价格数组，最多交易一次，求最大利润。

解法一：暴力枚举（O(n²)）

def max_profit_bruteforce(prices):
    n = len(prices)
    max_profit = 0
    for i in range(n):
        for j in range(i + 1, n):
            max_profit = max(max_profit, prices[j] - prices[i])
    return max_profit

解法二：一次遍历（O(n)）

def max_profit(prices):
    min_price = float('inf')
    max_profit = 0
    for price in prices:
        if price < min_price:
            min_price = price
        elif price - min_price > max_profit:
            max_profit = price - min_price
    return max_profit

解法三：转化为最大子数组和

def max_profit_kadane(prices):
    # 将价格差看作收益数组，求最大连续子数组和
    max_so_far = 0
    max_ending_here = 0
    for i in range(1, len(prices)):
        diff = prices[i] - prices[i - 1]
        max_ending_here = max(0, max_ending_here + diff)
        max_so_far = max(max_so_far, max_ending_here)
    return max_so_far

你看，一道看起来平平无奇的题，可以用三种完全不同的思路来解。暴力解让你确认题目理解无误；一次遍历是面试官期待的最优解；Kadane 解法让你看到了这道题与"最大子数组和"之间深刻的联系。这就是一题多解的价值——它把孤立的知识点织成了网。

练习建议：每周选 2-3 道题，强制自己写出至少两种解法。在代码注释里标注每种解法的时间/空间复杂度。坚持下去，一个月后你会发现自己看问题的角度明显开阔了。

制定个人刷题计划

没有计划地刷题就像没有地图的旅行——你可能走了很远，但到不了想去的地方。来，我们做一个实际的刷题规划。

不同时间线的计划

三个月冲刺计划（每天 2-3 小时）：

  • 第 1 个月：基础数据结构（数组、链表、栈、队列、哈希表、树）— 完成 80 道
  • 第 2 个月：核心算法（二分、回溯、DP、贪心、图论）— 完成 80 道
  • 第 3 个月：专项突破 + 模拟面试 — 完成 40 道 + 每天 mock interview

六个月稳扎稳打计划（每天 1-1.5 小时）：

  • 前 2 个月：跟本课程节奏，学一章刷一章的题
  • 第 3-4 个月：按标签刷，每个标签 15-20 道
  • 第 5 个月：刷目标公司高频题 + 周赛
  • 第 6 个月：集中模拟面试 + 查漏补缺
一个实用工具：LeetCode 的"Company Tag"功能（付费版）可以按公司查看高频题。如果你有明确的目标公司，这是最高效的刷题策略——因为每个公司的高频题池其实很小（100-200 道），把这些题吃透，面试中的原题概率非常高。

踩坑与总结

最后来聊聊一个很多人困惑的问题："我刷了 500 题了，为什么还是不会？" 我帮你拆解一下，常见的原因有四个：

原因一：刷了但没思考。 刷题变成了肌肉记忆——打开题目，想起来"做过类似的"，把记忆中的代码默写出来，提交通过，下一题。在这个过程中，你没有做任何元认知（思考自己的思考过程）。你写了代码但没想清楚"为什么这个解法是对的"、"为什么用这个数据结构而不是那个"、"如果约束条件变了怎么调整"。

原因二：重复做已经会的题。 人天生倾向于做自己擅长的事情——因为舒服。你刷 500 题，其中 400 道是数组和字符串的 Easy，重复劳动。真正让你进步的是那些让你不舒服的题——Hard 的 DP、复杂的图论、生僻的数据结构。走出舒适区，才有真正的成长。

原因三：不写总结。 还是那个道理——"知道"不等于"掌握"。把一道题的解法用中文写出来、讲给别人听、隔天再做一遍，这三种方式中任何一样都比"看一遍题解然后 next"有效 10 倍。

原因四：忽视基本功。 有些人刷了 500 题，但 BST 的中序遍历都写不利索——递归的 base case 写错，iterative 写法背不下来。这是巨大的隐患。因为你不知道面试官会问到多基础的问题——我见过有人能在白板上写出红黑树的删除操作，但连二分查找都写错边界。把基础数据结构的基本操作练到肌肉记忆，这是刷题的下限。

我的亲身踩坑：我曾经刷了 300 多题，自我感觉良好，去面一家心仪的公司。面试官问了一道 Medium 难度的"求二叉树的最大深度"。我笑了——这不是送分题吗？于是我刷刷刷写了递归一行解法。面试官说："很好，写个迭代版本。" 我愣住了。我从来没写过迭代的树遍历，全都是递归。结果我花了 15 分钟磕磕绊绊写了又改，最后还是没写对。那次面试挂了。那个教训让我明白：基础题的多种实现方式，就是你代码能力的照妖镜。

刷题不是一个"做完就完了"的过程，它是一个知识内化的过程。从"看过"到"做过"到"能讲出来"到"能举一反三"，每一层都需要额外的努力。但好消息是：一旦你建立了体系、找到了方法，之后的每一道题都会成为你能力增长的阶梯，而不是时间的浪费。
