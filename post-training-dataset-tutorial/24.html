第24章 训练框架数据适配

大家好，我是你们的讲师。上一章我们搭建了一条完整的后训练数据质量管线，从去重到格式转换，把一堆"原材料"变成了"洁净的训练数据"。但一个新的问题出现了：这些干净的数据，不同训练框架认得吗？

真实情况是，每个训练框架对数据集都有自己的格式偏好。你用的SFT数据可能是{"instruction": "...", "output": "..."}格式，但TRL的SFTTrainer要求messages格式，而LLaMA-Factory又支持alpaca格式。本章就来解决这个"最后一公里"的问题——数据准备好了，怎么喂给训练框架。

数据准备好了，怎么喂给训练框架——三大框架的格式要求

目前后训练领域最主流的三套框架分别是：

  • TRL (Transformer Reinforcement Learning) —— HuggingFace官方出品的训练框架，集成了SFT、DPO、PPO、GRPO等主流后训练算法
  • Axolotl —— 社区最流行的全流程微调工具，以配置文件驱动、支持广泛而著称
  • LLaMA-Factory —— 中文社区最主流的微调框架，对中文数据和模型的支持尤其出色

理解这三个框架的数据适配方式，基本上就能覆盖后训练数据格式转换的90%场景。我们逐个来看。

TRL (Transformer Reinforcement Learning)——HuggingFace官方标准

TRL是HuggingFace团队维护的强化学习微调库，提供了SFTTrainer、DPOTrainer、GRPOTrainer等多个训练器。每个训练器对数据格式的要求不同：

SFTTrainer的数据格式

SFTTrainer支持三种格式，其中最推荐的是messages格式（即OpenAI兼容格式）：

[
  {
    "messages": [
      {"role": "system", "content": "你是一个数学助手。"},
      {"role": "user", "content": "计算 23 × 45"},
      {"role": "assistant", "content": "23 × 45 = 1035"}
    ]
  },
  {
    "messages": [
      {"role": "user", "content": "解释什么是梯度下降"},
      {"role": "assistant", "content": "梯度下降是一种优化算法..."}
    ]
  }
]

这是目前最标准化的格式，也被称为"ShareGPT格式"。它的特点是每个样本包含一个完整的对话轮次列表，每个轮次通过role字段标识角色。这个格式被主流tokenizer的chat_template原生支持——你只需要在加载tokenizer时指定apply_chat_template=True，框架就会自动拼接对话。

为什么messages格式是未来趋势？因为它天然支持多轮对话、system prompt、工具调用等多种场景。2025年以来，几乎所有主流训练框架都开始转向messages格式作为首选输入格式。如果你现在开始一个新的后训练项目，强烈建议直接使用messages格式存储数据。

DPOTrainer的数据格式

DPO训练需要偏好对比对数据。TRL的DPOTrainer要求一个包含chosen和rejected两个对话列表的格式：

[
  {
    "chosen": [
      {"role": "user", "content": "如何学好数学？"},
      {"role": "assistant", "content": "学好数学需要坚持练习和思考..."}
    ],
    "rejected": [
      {"role": "user", "content": "如何学好数学？"},
      {"role": "assistant", "content": "我不知道"}
    ]
  }
]

chosen列表是偏好回答的完整对话，rejected列表是拒绝回答的完整对话。注意两者的user轮次必须完全一致——这样模型才能正确学习"在同样的输入下，哪个输出更好"。

GRPOTrainer的数据格式

GRPO（Group Relative Policy Optimization）是2025年兴起的主流强化学习方法，TRL在0.14版本后正式支持。GRPO的数据格式相对简单——只需要一个prompt字段和一个可选的reward_func：

[
  {"prompt": "证明勾股定理"},
  {"prompt": "sorted(array)的时间复杂度是多少？"},
  {"prompt": "写一个Python装饰器用于计时"}
]

GRPOTrainer不提供标准答案——它通过分组采样、让模型自己生成多个输出，然后由reward function（或reward model）对每个输出打分，用组内相对奖励来更新策略。

Axolotl——社区最流行的全流程微调工具

Axolotl由OpenAccess AI Collective维护，是目前GitHub上最活跃的微调框架之一。设计哲学是"配置文件驱动"——绝大多数训练参数通过YAML配置文件设定，极大降低了使用门槛。

Axolotl支持的数据格式非常广泛，包括：

  • alpaca格式：{instruction, input, output}，最经典的SFT格式
  • sharegpt格式：{conversations: [{from, value}]}，支持多轮对话
  • openai格式：{messages: [{role, content}]}，TRL兼容格式
  • gemini格式：{contents: [{role, parts}]}，Google格式兼容
  • 偏好格式：{chosen, rejected} 或 {conversations_chosen, conversations_rejected}

Axolotl最强大的地方在于它的自动对话模板处理——你不需要在数据中手动插入特殊token（如<|im_start|>、<s>等）。只需要在YAML配置中指定tokenizer的chat_template，Axolotl会调用tokenizer.apply_chat_template()自动完成模板化拼接。这个特性在2025年后发展成为业界标准做法。

一个典型的Axolotl数据配置片段：

# axolotl_config.yaml
datasets:
  - path: my_sft_data.jsonl
    type: sharegpt
    conversation: llama3
    field:
      messages: conversations
      role: from
      content: value
  - path: my_dpo_data.jsonl
    type: hh_rlhf
    field_chosen: chosen
    field_rejected: rejected

这个配置告诉Axolotl：左侧数据是sharegpt格式，使用llama3的对话模板；右侧数据是HH-RLHF格式的偏好数据。

社区生态优势：Axolotl在2025-2026年间迅速发展了最丰富的社区适配器生态。无论你用的是LLaMA 3、Qwen 2.5、DeepSeek-V2、Mistral还是Gemma，Axolotl都能通过简单的配置切换支持。这使得它成为"数据工程师最爱的微调框架"——不需要改动代码，只需要换一个配置文件和数据集路径，就能适配不同的模型和数据格式。

LLaMA-Factory——中文社区最主流的微调框架

LLaMA-Factory由国内开发者开发维护，在中文社区拥有极高的渗透率。它对中文数据的支持深度远超其他框架，同时提供了Web UI界面（类似于AutoDL的"炼丹炉"），极大降低了使用门槛。

LLaMA-Factory支持五种数据格式：

格式名称 字段结构 适用场景
alpaca instruction, input, output 标准单轮SFT
sharegpt conversations: [{from, value}] 多轮对话SFT
openai messages: [{role, content}] 兼容格式（推荐）
rlhf chosen, rejected 或 better, worse DPO/偏好对齐
kto type, messages KTO（无偏好对训练）

LLaMA-Factory对中文的特殊支持体现在几个方面：一是内置了Qwen、Yi、ChatGLM、Baichuan、DeepSeek等中文模型的全部tokenizer配置；二是对中文计数单位、标点符号和编码有较好的容错处理；三是提供了大量中文数据集的预处理脚本。

一个LLaMA-Factory的数据集注册示例（在dataset_info.json中）：

{
  "my_chinese_sft": {
    "file_name": "chinese_sft.json",
    "formatting": "sharegpt",
    "columns": {
      "messages": "conversations"
    },
    "tags": {
      "role_tag": "from",
      "content_tag": "value",
      "user_tag": "human",
      "assistant_tag": "gpt"
    }
  },
  "my_preference_data": {
    "file_name": "pref_data.json",
    "formatting": "rlhf",
    "columns": {
      "chosen": "chosen",
      "rejected": "rejected"
    }
  }
}

注意这里的tags字段——它指定了对话模板中的角色标签映射。中文数据集中，用户角色常用"human"或"用户"，助手角色常用"gpt"或"助手"。LLaMA-Factory允许通过配置灵活映射，不需要修改原始数据。

格式转换适配实战——用datasets.map统一格式

了解了各框架的要求后，核心问题变成了：如何将我们已有的数据格式，转换为目标格式？答案就在datasets.map()。

下面这个函数可以将任意格式的数据集转换为三大框架通用的格式：

from datasets import Dataset
import json

def convert_to_messages_format(
    dataset: Dataset,
    source_format: str = "alpaca",
    add_system_prompt: str = None
) -> Dataset:
    """
    将各种常见数据集格式统一转换为messages格式，
    兼容TRL/Axolotl/LLaMA-Factory。

    支持的source_format:
      - "alpaca": {instruction, input?, output}
      - "sharegpt": {conversations: [{from, value}]}
      - "plain_instruction": {instruction, response}
    """

    def format_row(example):
        messages = []

        # 可选：添加system prompt
        if add_system_prompt:
            messages.append({
                "role": "system",
                "content": add_system_prompt
            })

        if source_format == "alpaca":
            # Alpaca格式：如果有input则拼接，否则只用instruction
            user_content = example["instruction"]
            if example.get("input"):
                user_content += "\n" + example["input"]

            messages.append({
                "role": "user",
                "content": user_content
            })
            messages.append({
                "role": "assistant",
                "content": example["output"]
            })

        elif source_format == "sharegpt":
            # ShareGPT格式：直接映射from → role
            role_map = {
                "human": "user",
                "gpt": "assistant",
                "system": "system",
                "user": "user",
                "assistant": "assistant"
            }
            for turn in example["conversations"]:
                role = role_map.get(turn["from"], "user")
                messages.append({
                    "role": role,
                    "content": turn["value"]
                })

        elif source_format == "plain_instruction":
            messages.append({
                "role": "user",
                "content": example["instruction"]
            })
            messages.append({
                "role": "assistant",
                "content": example["response"]
            })

        return {"messages": messages}

    return dataset.map(format_row)


def convert_to_preference_format(
    dataset: Dataset,
    chosen_field: str = "chosen",
    rejected_field: str = "rejected"
) -> Dataset:
    """将偏好数据转换为TRL/DPO兼容格式"""

    def format_pref(example):
        chosen_msgs = []
        rejected_msgs = []

        # chosen对话
        for turn in example[chosen_field]:
            chosen_msgs.append({
                "role": turn["from"].replace("human", "user").replace("gpt", "assistant"),
                "content": turn["value"]
            })

        # rejected对话
        for turn in example[rejected_field]:
            rejected_msgs.append({
                "role": turn["from"].replace("human", "user").replace("gpt", "assistant"),
                "content": turn["value"]
            })

        return {
            "chosen": chosen_msgs,
            "rejected": rejected_msgs
        }

    return dataset.map(format_pref)


# ─── 使用示例 ───
if __name__ == "__main__":
    # 加载Alpaca格式数据 → 转为messages格式
    alpaca_data = Dataset.from_json("data/alpaca_format.jsonl")
    messages_data = convert_to_messages_format(
        alpaca_data,
        source_format="alpaca",
        add_system_prompt="你是一个有用的AI助手。"
    )
    messages_data.to_json("data/trl_compatible.jsonl")

    # 加载偏好数据 → DPO格式
    pref_data = Dataset.from_json("data/preference_raw.jsonl")
    dpo_data = convert_to_preference_format(pref_data)
    dpo_data.to_json("data/dpo_compatible.jsonl")

这个转化函数的优势在于：你只需要写一次转换逻辑，输出的messages格式可以直接被三个框架识别（TRL原生支持、Axolotl通过openai格式支持、LLaMA-Factory通过openai格式支持）。如果数据团队统一使用messages格式作为"交换格式"，各框架之间的迁移成本几乎为零。

常见适配坑

即使掌握了格式转换方法，在实际适配过程中还是有一些容易掉进去的坑。

坑一：tokenizer chat_template不匹配。这是最常见的问题。不同的基座模型使用不同的对话模板——LLaMA 3使用<|begin_of_text|>...<|eot_id|>，Qwen 2使用<|im_start|>...<|im_end|>，DeepSeek使用<｜begin▁of▁sentence｜>...<｜end▁of▁sentence｜>。如果你用LLaMA 3的chat_template去格式化Qwen的数据，得到的训练序列会是错的！绝对不要假设一个tokenizer的chat_template适用于另一个模型。

坑二：特殊token处理。一些训练框架默认将<pad>、<unk>、<eos>等特殊token追加到对话末尾，但另一些框架不会。如果你在DPO训练中看到loss一直不下降，大概率是特殊token处理出了问题——rejected回答和chosen回答的序列长度差异恰好通过特殊token被模型"识破"了。

坑三：多轮对话拼接方式不同。不同的框架对多轮对话的拼接逻辑存在细微差异。有些框架在每一轮之间插入\n换行，有些用\n\n，有些什么都不加。这种差异在高精度微调中会带来可测量的效果差异。建议在训练前打印几个样本的tokenized结果，仔细检查拼接格式是否符合预期。

一个真实的排查案例：我在2025年初帮助一个团队排查DPO训练loss不稳定问题。他们用Axolotl训练的loss在0.5左右剧烈振荡，模型效果完全不可用。排查了两天，最后发现问题出在chat_template上——他们的tokenizer是从HuggingFace下载的，但下载时不小心用了一个社区用户修改过的配置，导致user和assistant的role token顺序被颠倒了。模型实际上在学习"assistant说话在前，user说话在后"这种完全错误的对话模式。修复chat_template后，loss立刻恢复正常。教训是：永远在训练前第一个epoch打印10个格式化后的样本，人工检查一遍。

三大框架对比表

最后，我们用一个综合对比表来总结三个框架的差异，帮助你根据实际需求做出选择：

维护方 HuggingFace官方 OpenAccess AI Collective hiyouga (个人/社区)

选择建议很直接：如果你需要GRPO或最新的强化学习算法，选TRL；如果你在多模型之间频繁切换实验，选Axolotl；如果你的重心在中文本地化应用、或者团队中非工程师成员也需要参与训练，选LLaMA-Factory。当然，这三个框架并不互斥——很多团队在不同的项目阶段会使用不同的框架。

趋势观察：2025-2026年间，这三个框架呈现出"格式收敛"的趋势。TRL从0.15版本开始支持直接读取Axolotl格式的配置文件；Axolotl新增了openai格式的原生支持；LLaMA-Factory则开始支持TRL的GRPOTrainer后端。预计到2027年，这些框架之间的数据格式差异将进一步缩小，最终统一到messages格式标准。
