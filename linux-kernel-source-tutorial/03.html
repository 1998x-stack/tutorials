大家好，我是你们的讲师。上一章我们建立了整门课程的地图——了解了Linux 0.11的目录结构、设计哲学和启动流程。现在到了"动手"的时候了：我们要在你的电脑上，让这个30多年前的老内核重新跑起来。

有人可能会问："为什么不能直接在物理机上跑？" 答案是——首先，Linux 0.11是为Intel 386/486设计的，它在现代的CPU上虽然可以运行（x86向后兼容），但很多硬件细节（比如IDE控制器、VGA显示模式）已经完全不一样了。其次，在模拟器中运行有一个巨大的优势——内置调试器。Bochs允许你在内核的任意位置设置断点、查看内存、单步执行——这种"x光透视"般的调试体验，是学习内核的利器。

好了，让我们开始搭建环境吧。

引言——工欲善其事必先利其器

要运行Linux 0.11，我们需要四样东西：一个模拟器（Bochs或QEMU）、一份编译好的内核镜像（Image文件）、一个磁盘镜像（包含根文件系统）、以及一份模拟器配置文件（bochsrc）。

好消息是：所有这些资源都可以在网上免费获取。最早的Linux 0.11完整运行环境，在oldlinux.org上可以下载到打包好的版本。你不需要自己去找古董硬件。

学习目标：请确保完成本章后，你能够（1）启动Bochs并看到Linux 0.11的shell，（2）在shell中执行几条基本的Linux命令，（3）知道如何关闭虚拟机。这个"最小可用环境"将陪伴你通过整门课程的所有实操环节。

Bochs模拟器安装

Bochs（发音为"box"）是一个开源的x86 PC模拟器，由Kevin Lawton在1994年创建。与QEMU和VirtualBox不同，Bochs模拟的是整个x86硬件——从CPU指令周期到南桥芯片，精确到每个外设的寄存器级别。

为什么选择Bochs而不是更快的QEMU？因为Bochs内置了一个非常强大的调试器。你可以用它查看GDT、LDT、页表、寄存器、内存——所有你在课堂上需要用到的底层调试功能，Bochs调试器都支持。QEMU虽然也可以通过GDB进行远程调试，但设置起来更复杂。

特性BochsQEMU
模拟精度指令级精确（每条指令都模拟）动态翻译（部分指令直接让CPU跑）
内置调试器有（强大，支持info gdt/tab等命令）无（需配合GDB远程调试）
速度慢（精确模拟的代价）快（动态翻译的性能优势）
调试功能查看GDT/IDT/页表/寄存器通过GDB查看，功能弱于Bochs
适用场景内核调试和学习日常快速测试

安装Bochs非常直接。在主流操作系统上：

# Ubuntu / Debian
sudo apt-get install bochs bochs-x bochsbios

# Fedora / RHEL
sudo dnf install bochs bochs-x

# macOS (使用Homebrew)
brew install bochs

# Windows
# 从 https://bochs.sourceforge.io 下载安装包
# 或通过 Chocolatey: choco install bochs

# 验证安装成功
bochs --version
# 应该输出类似: Bochs x86-64 emulator x.x.x

注意：一定要安装bochs-x（或bochs-x11）包，否则Bochs无法打开图形窗口显示虚拟机的屏幕。如果你是在远程SSH会话中操作，可以考虑安装bochs-term来使用终端模式。

版本提醒：Bochs 2.6.x和2.7.x是最常用的版本。Bochs 2.8在某些系统上可能需要额外的SDL库。建议使用系统包管理器提供的版本——如果版本太旧，可以考虑从源码编译。但不要在这上面花太多时间——任何能运行的Bochs版本都可以用来学习0.11。

准备内核镜像与磁盘

我们需要从社区维护的资源站点下载两份文件：Image（内核镜像）和hdc-0.11.img（磁盘镜像，包含根文件系统）。

如果你是从零开始搭建，可以去oldlinux.org或GitHub上搜索"linux-0.11-lab"项目——社区已经打包了可以直接运行的开发环境。这里也列出了最简洁的手动获取方式：

# 创建工作目录
mkdir ~/linux-0.11-lab && cd ~/linux-0.11-lab

# 获取内核源码（可选，但我们后面需要读代码）
# 可以从BrGIS或其他托管站点获取
wget http://www.oldlinux.org/Linux.old/bochs/linux-0.11-devel-040805.zip
unzip linux-0.11-devel-040805.zip

# 目录结构如下:
linux-0.11-lab/
  +-- boot/            # 引导启动代码
  +-- kernel/          # 内核核心
  +-- fs/              # 文件系统
  +-- mm/              # 内存管理
  +-- ...
  +-- Image            # 预编译的内核镜像文件
  +-- hdc-0.11.img     # 根文件系统磁盘镜像
  +-- bochsrc.bxrc     # Bochs配置文件
  +-- Makefile         # 编译用的Makefile

如果你没有找到现成的打包文件，下面是一个简化方案：在任何Linux发行版上，你可以从GitHub克隆社区维护的0.11项目。一些热心开发者甚至做了可以直接用docker启动的0.11环境。

配置bochsrc

bochsrc是Bochs的配置文件，告诉模拟器使用多少内存、从哪个磁盘启动、使用什么显示模式等。下面是一个针对Linux 0.11的完整配置，我会逐行解释：

# bochsrc - Linux 0.11 模拟器配置文件

# 设置内存大小(以MB为单位)
megs: 16

# BIOS和VGABIOS路径
romimage: file=/usr/share/bochs/BIOS-bochs-latest, address=0xf0000
vgaromimage: file=/usr/share/bochs/VGABIOS-lgpl-latest

# 启动盘——A盘对应软盘0，从Image启动
floppy_bootsig_check: disabled=0
floppya: 1_44=Image, status=inserted

# 第二块"磁盘"——就是根文件系统hdc-0.11.img
ata0: enabled=1, ioaddr1=0x1f0, ioaddr2=0x3f0, irq=14
ata0-master: type=disk, path="hdc-0.11.img", mode=flat

# 使用哪种显示库
display_library: x

# 日志输出
log: bochsout.log
panic: action=ask

# 键盘 —— 关键！映射AT键盘
keyboard: type=at, serial_delay=200, paste_delay=10000

# 时钟
clock: sync=realtime, time0=local

# CPU配置
cpu: count=1, ips=4000000

我们来解读几个关键选项：

megs: 16 —— 给虚拟机分配16MB内存。0.11内核设计时假设最多16MB内存，设置更大的值反而可能导致内核无法正确处理高地址。

floppya: 1_44=Image —— 将编译好的内核镜像作为A盘（软盘0），Bochs启动时会先尝试从软盘启动。

ata0-master: type=disk, path="hdc-0.11.img" —— 将hdc-0.11.img映射为第一块IDE硬盘的主盘。内核启动后会挂载它为根文件系统。

cpu: ips=4000000 —— 限制模拟速度为每秒400万条指令。不设置的话，Bochs会使劲全力模拟，导致虚拟机跑得飞快（甚至超出原始386的速度），导致串口输出太快来不及显示。

路径提示：Bochs的BIOS和VGABIOS文件路径因操作系统和安装方式而异。在Ubuntu上通常在/usr/share/bochs/或/usr/local/share/bochs/下。如果你不确定，可以用find /usr -name "BIOS-bochs-latest" 2>/dev/null来查找。

首次启动

配置完成后，在包含Image、hdc-0.11.img和bochsrc的目录中，运行以下命令：

bochs -f bochsrc

# Bochs会弹出对话框询问启动模式:
# 1. Start simulation (直接启动)
# 2. Begin simulation from ... (从指定地址启动)
# 3. Continue simulation from ... (继续)
# 6. Quit (退出)
#
# 输入 1 并回车，Bochs窗口将开始运行

# 你会在Bochs的屏幕上看到:
# Loading image ...  (bootsect阶段输出)
# 然后屏幕清空，开始内核初始化
# 最后显示:
#    "/ #" 或 "/ # _"
# 这就是Linux 0.11的shell提示符

如果你看到了/ #提示符，恭喜你——Linux 0.11在你的电脑上"复活"了。现在试试几个基本命令：

/ # ls
bin   dev   etc   home  root  tmp   usr   var
/ # cat /etc/issue
Welcome to Linux 0.11
/ # uname -a
Linux 0.11 #1 Thu Dec 5 12:34:56 1991
/ # cat /proc/version
Linux version 0.11 (root@...) (gcc version 1.40) #1 ...
/ # ls /bin
sh  login  mount  umount  sync  chmod  ...
/ # df
Filesystem  1k-blocks   Used Available Use% Mounted on
/dev/hd1         ?       ?         ?     ?    /

你可以在这个shell里做任何在普通Linux shell里能做的事——当然功能要少得多。没有管道|，没有重定向>，没有tab补全。但你可以ls、cat、cd、echo。这个shell就是/bin/sh——一个简化版的Bourne Shell。

历史瞬间：你当前看到的这个shell提示符，与1991年Linus Torvalds在赫尔辛基的宿舍里看到的完全一样。他当时通过拨号网络连接到大学的UNIX服务器，在本地运行自己的内核，在MINIX上编译代码——每次修改完内核都要重启电脑来测试。我们今天用Bochs模拟这一切，已经比他当年幸福太多了。

要关闭Bochs，在Bochs控制台按Ctrl+C然后输入quit，或者直接关闭Bochs窗口。注意不要直接在Bochs虚拟机里执行shutdown——0.11没有完善的关机机制。

QEMU+GDB备选方案

Bochs虽然调试功能强大，但速度确实慢。如果你只想快速测试内核修改后的效果，或者你更喜欢GDB的调试界面，可以考虑使用QEMU。

QEMU的用法更加简单——一条命令就能启动：

# 安装QEMU
sudo apt-get install qemu-system-x86

# 启动Linux 0.11
qemu-system-i386 -m 16 -fda Image -hda hdc-0.11.img -boot a

# 启用GDB调试模式 (监听TCP 1234端口)
qemu-system-i386 -m 16 -fda Image -hda hdc-0.11.img -boot a -s -S
#   -s  : 等价于 -gdb tcp::1234
#   -S  : 启动时暂停CPU，等待GDB连接

第二个命令启动后，QEMU会打开窗口但CPU处于暂停状态。你需要在另一个终端用GDB连接上去：

# 在另一个终端
gdb
(gdb) target remote localhost:1234
(gdb) file system  # 加载符号表
(gdb) b main       # 在main函数设置断点
(gdb) c            # 继续执行
(gdb) info registers  # 查看寄存器
(gdb) x/16x 0x1000   # 查看内存

QEMU+GDB的方案没有Bochs内置调试器那么强大——比如你不能直接"info gdt"或"info tab"来查看GDT或页表（这些都是Bochs调试器独有的命令）。但它的优势在于速度快、集成到VS Code等IDE中方便、以及你可以在熟悉的环境（GDB）中工作。

需求场景推荐方案
读代码、调试引导流程、查看底层状态Bochs (内置调试器)
快速测试、日常开发、集成IDEQEMU + GDB
初次上手、跟着本课程做实验Bochs (优先推荐)
我的建议：两个都装上。Bochs作为主要调试工具——每次在代码中遇到不理解的跳转、寄存器配置、页表映射时，都可以在Bochs调试器中停下来查看。QEMU作为快速测试工具——当你只是编译了内核想知道能不能启动时，QEMU的启动速度比Bochs快很多。

好了，开发环境搭建完成了。现在你的电脑上已经可以运行一个1991年的操作系统内核了。从下一章开始，我们将真正进入代码世界——从按下电源键的那一刻开始，追踪内核的第一条指令，一步一步看它如何从BIOS手中接过控制权，进入保护模式，最终拉起用户态的shell。
