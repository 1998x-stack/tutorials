第27章 项目实战：自动化研究助手

如果说上一章的智能客服是 Agent 技术在"服务"场景的标杆应用，那本章的自动化研究助手就是 Agent 在"知识工作"领域的杀手级用例。想象一下：你是一名分析师，老板让你研究"2026 年全球 AI Agent 框架市场的竞争格局"。传统做法是花两三天时间搜索资料、阅读报告、整理笔记、写 PPT。而现在，你只需要把这个问题输入给研究助手 Agent，它会在几分钟内输出一份结构完整、数据详实、图表精美的研究报告。

这不是未来愿景——这种 Agent 已经在 2026 年的许多咨询公司、投行和科技企业内部广泛使用。我们的目标是构建一个多 Agent 协作的研究系统，它由多个专门化的 Agent 组成，像一支真正的分析师团队一样工作：有人负责搜索资料，有人负责交叉验证，有人负责数据分析，有人负责报告撰写。

项目亮点： 这个项目将综合运用本教程的多项核心技术——多 Agent 协作（第 7 章）、LangGraph 状态图（第 10 章）、条件分支（第 11 章）、RAG（第 12 章）、MCP 协议（第 20 章），以及测试（第 23 章）和成本优化（第 24 章）的最佳实践。它是一个真正的"集大成者"项目。

27.1 项目设计：从问题到报告

我们先来定义研究助手的"输入-输出"规格：

维度 说明
输入 一个研究问题（如"2026 年小模型（SLM）在企业级应用中的渗透率"）
研究深度 快速（5 分钟，浅层概述）/ 标准（15 分钟，结构化分析）/ 深度（30 分钟，多源交叉验证）
输出格式 Markdown 报告 / HTML 报告 / PDF 报告 / 演示文稿大纲
报告结构 摘要 → 背景 → 方法论 → 核心发现 → 多维度分析 → 数据图表 → 结论与建议 → 参考文献

研究助手的工作流程分为五个阶段：

  1. 问题分析： 解析研究问题，识别关键术语和边界，生成搜索策略。
  2. 信息收集： 并行搜索多个数据源（Web 搜索、学术数据库、企业内部知识库）。
  3. 多角度分析： 从不同视角（技术、市场、竞争、风险）对收集到的信息进行分析。
  4. 报告生成： 整合分析结果，生成结构化报告。
  5. 质量审查： 自我检查报告质量，补充缺失信息，修正错误。
设计决策： 为什么不用单一 Agent 而用多 Agent？原因有三个。第一，专业化：搜索 Agent 擅长调用搜索 API，分析 Agent 擅长推理，写作 Agent 擅长文字组织——各司其职比"万金油"效果好得多。第二，并行性：多 Agent 可以并行工作，搜索和分析可以同时进行，大幅缩短总耗时。第三，可审计性：每个 Agent 的产出都有独立记录，方便追溯和修正。

27.3 搜索聚合 Agent 实现

# ---- 搜索迭代策略 ----
class SearchStrategy:
    """智能搜索策略：根据初步结果动态优化搜索词"""

    @staticmethod
    def generate_refined_keywords(initial_results: list,
                                  research_question: str) -> list[str]:
        """基于初步搜索结果，生成更精准的二次搜索关键词"""
        # 分析已获取结果中的高频词和信息缺口
        prompt = f"""研究问题：{research_question}

已搜索到的结果标题：
{chr(10).join(r['title'] for r in initial_results[:10])}

请分析：
1. 这些结果覆盖了哪些维度？存在什么信息缺口？
2. 生成3-5个更精准的二次搜索关键词来填补缺口
3. 关键词应具体、可操作，避免大而泛的词汇

输出 JSON 格式的关键词列表。"""

        response = llm.invoke(prompt)
        new_keywords = parse_keywords(response.content)
        return new_keywords
搜索优化技巧： 研究助手的搜索质量很大程度上取决于搜索迭代策略。不要只搜一轮——第一轮搜索获取概览，根据结果识别"信息缺口"（哪个维度信息不足？哪个来源权威但未被查询？），然后生成更精确的二次搜索词。三轮搜索迭代通常能覆盖 90% 以上的相关信息。

资料收集完成后，分析员需要从多个维度对信息进行深度分析。我们定义了一个分析矩阵——每个维度对应一个独立的分析任务：

        "prompt": "从落地应用角度分析：主要应用场景有哪些？典型案例？"

    },
    {

# ---- 并行执行多维度分析 ----
from concurrent.futures import ThreadPoolExecutor, as_completed

def run_parallel_analysis(search_results: list, dimensions: list) -> dict:
    """并行执行多维度分析"""
    analysis_results = {}

    with ThreadPoolExecutor(max_workers=len(dimensions)) as executor:
        future_map = {}

        for dim in dimensions:
            task = Task(
                description=f"基于搜索结果，进行{dim['name']}分析。\n\n"
                            f"分析要求：{dim['prompt']}\n\n"
                            f"搜索结果：{json.dumps(search_results, ensure_ascii=False)}",
                expected_output=f"{dim['name']}分析报告",
                agent=analyst
            )
            future = executor.submit(task.execute)
            future_map[future] = dim["name"]

        for future in as_completed(future_map):
            dim_name = future_map[future]
            analysis_results[dim_name] = future.result()

    return analysis_results


# ---- 交叉验证机制 ----
def cross_validate_findings(analysis_results: dict) -> dict:
    """交叉验证各维度的分析结果，识别矛盾和不一致"""
    prompt = """以下是各维度分析结果，请进行交叉验证：

{analysis_text}

请逐项检查：
1. 不同维度的分析是否存在矛盾或不一致之处？
2. 关键数据点是否有多个独立来源支撑？
3. 识别出2-3个可信度最高的核心发现
4. 识别出1-2个需要进一步核实的不确定结论

输出 JSON 格式。"""

    response = llm.invoke(prompt.format(
        analysis_text=json.dumps(analysis_results, ensure_ascii=False)
    ))
    return json.loads(response.content)
为什么交叉验证重要： 2025 年有一项研究发现，LLM 生成的研究报告中约有 15-20% 的事实性错误源于"单一信源偏差"——Agent 只从一个来源获取了信息，而这个来源本身可能不准确。交叉验证机制通过强制多源核对，能将错误率降低到 3% 以下。如果你是研究助手的用户，这点差异决定了你是"信任它"还是"不信任它"。

                "title": "主要厂商市场份额",
                "code": f"""
labels = {share_data['labels']}
sizes = {share_data['sizes']}
colors = ['#8b5cf6', '#3b82f6', '#10b981', '#f59e0b', '#ef4444']

plt.figure(figsize=(8, 8))
plt.pie(sizes, labels=labels, colors=colors, autopct='%1.1f%%',
        startangle=90, pctdistance=0.85)
plt.title('{share_data["title"]}')
plt.savefig('outputs/market_share.png', dpi=150, bbox_inches='tight')
print("图表已生成: market_share.png")
"""
            })

    return chart_specs


def render_html_report(markdown_report: str,
                       chart_specs: list[dict]) -> str:
    """将 Markdown 报告和图表渲染为 HTML 格式"""
    # 在实际系统中，这里会使用 markdown 转 HTML 的库
    # 并在相应位置插入图表图片
    html_parts = [
        "",
        "研究报告",
        ""
    ]

    # 转换 Markdown 内容到 HTML（简化版本）
    html_content = markdown_to_html_simple(markdown_report)
    html_parts.append(html_content)

    # 在数据概览部分插入图表
    for chart in chart_specs:
        img_tag = f""
        html_parts.append(img_tag)

    html_parts.append("")
    return "\n".join(html_parts)
增强建议： 在正式版本中，可以考虑使用 plotly 生成交互式图表（鼠标悬停显示数据、缩放、筛选），而不是静态图片。交互式图表在研究报告中的信息传达效率比静态图高约 40%。另外，如果数据量足够，还可以加入简单的预测建模——用线性回归或指数平滑预测未来趋势，这会让报告的分析深度上一个台阶。

# ---- 主入口函数 ----
def run_research(research_question: str,
                 depth: str = "standard",
                 output_format: str = "html"):
    """执行完整的研究流程"""
    print(f"🔍 开始研究：{research_question}")
    print(f"📊 研究深度：{depth}")

    # Step 1: 问题分析
    print("\n[Step 1/5] 分析研究问题...")
    research_plan = research_manager.execute_task(
        f"分析研究问题，制定研究计划：{research_question}"
    )

    # Step 2: 搜索收集
    print("[Step 2/5] 执行多源搜索...")
    search_results = searcher.execute_task(
        f"根据研究计划执行搜索：{research_plan}"
    )

    # Step 3: 多角度分析
    print("[Step 3/5] 执行多维度分析...")
    analysis_results = run_parallel_analysis(
        search_results, ANALYSIS_DIMENSIONS
    )
    validated = cross_validate_findings(analysis_results)

    # Step 4: 生成图表
    print("[Step 4/5] 生成可视化图表...")
    charts = generate_charts(analysis_results)
    for chart in charts:
        exec(chart["code"])  # 实际生产环境需严格沙箱化

    # Step 5: 生成报告
    print("[Step 5/5] 撰写研究报告...")
    manager_summary = {
        "question": research_question,
        "plan": research_plan,
        "validation": validated
    }
    report_md = generate_report(manager_summary, analysis_results, search_results)

    if output_format == "html":
        report = render_html_report(report_md, charts)
        output_path = f"outputs/report_{int(time.time())}.html"
    else:
        output_path = f"outputs/report_{int(time.time())}.md"

    with open(output_path, "w", encoding="utf-8") as f:
        f.write(report)

    print(f"\n✅ 研究完成！报告已保存至：{output_path}")
    return output_path
