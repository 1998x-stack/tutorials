SFT——数据质量比模型大小更重要

大家好，我是你们的讲师。上一章我们深入研究了LoRA微调的数学本质。但做微调的人都知道一个道理：LoRA决定了你能跑多快，而数据决定了你能跑多远。

SFT（Supervised Fine-Tuning）是大模型应用中最关键的一环。它是在预训练大模型基础上，使用高质量标注数据让模型学会遵循指令、适应特定输出格式和领域风格的过程。但很多人对SFT有一个根本性的误解——他们觉得SFT就是"把数据丢进去训练"，忽略了数据工程本身是一门科学。

今天这章，我们来系统梳理SFT数据工程的科学方法。你会学到：为什么调prompt比做SFT优先级更高？400条高质量数据为什么比4000条脏数据更有效？那些让无数人掉坑的Chat Template不一致问题到底怎么排查？以及为什么大模型SFT中后期loss与效果脱节严重。

Prompt先行原则

在做任何SFT之前，你必须先做一件事——调prompt验证基座模型的基本能力。这是整个SFT流程中最重要也是最容易被跳过的一步。

为什么？因为SFT不能凭空创造能力，它只能在已有能力的基础上进行"精调"。如果基座模型本身在面对你的任务时连基本正确的回答都给不出来，那SFT无论怎么调都是白费功夫。

Tang AI的金句："先调prompt，保证让模型可以按照你想输出的样式先输出。如果怎么调都不行建议换模型，说明能力不行。正常来说，通过prompt至少有20%以上的答案是正确的，才有SFT成功的可能性。"

这个"20%门槛"是我见过的SFT工程中最常被忽视的铁律。很多人一上来就花几周时间造数据、调LoRA参数，结果模型SFT后表现仍然很差。回头一看，基座模型本身连最简单的prompt都处理不好——问题的根源不在SFT，在模型选型。

具体做法：在你决定做SFT之前，先花1-2天时间手写prompt，在基座模型上测试。如果精心设计的prompt能让正确率达到20%以上，SFT才有希望把上限推到80-90%。如果怎么调prompt正确率都不到20%，那就果断换基座模型——别指望用SFT来"补短板"。补不了的。

反过来也一样——如果prompt不复杂就能有不错的效果，那也别把prompt构造得太复杂。最简单有效的方案就是最好的方案。

数据质量 vs 数量的边际收益

假设你已经确认基座模型够用了，下一步就是数据构造。这里有一个反直觉的结论：SFT不需要海量数据。

Tang AI的经验非常直接："对于多任务问题，每个垂域400条高质量数据足矣。" 这个数字听起来小得令人不安，但经过大量实践验证，400条/领域确实是SFT数据量的"甜点区"——低于这个数字可能覆盖不全关键模式，高于这个数字的边际收益迅速递减。

关键在于"高质量"三个字。宁可400条精标数据，不要4000条脏数据。高质量数据的标准包括：

  • 风格统一性：这是SFT数据中最关键但也最容易被忽视的要求。所有样本的输出格式必须严格一致。如果输出要求JSON格式，所有样本必须全部用```json包裹；如果SQL答案统一用select a, b, c，就永远不要出现select *。风格一致性直接决定了SFT后模型在实际使用中的输出稳定性。混入多种风格会让模型"学偏"——表现在训练loss正常，但模型输出格式混乱。
  • 训练分布覆盖测试分布：当发现测试集效果远不如训练集时，优先检查测试集中是否存在训练集未覆盖的数据模式，然后针对性地补充少量数据。Tang AI特别强调："先加5个、10个，再看训练效果"——不需要大量补充，针对性加几条就见效。
  • 强模型生成 + 多模型校验：使用比目标模型更强的模型（如GPT-4、Claude）来生成SFT训练数据，再用2-3个不同模型对生成的QA对做交叉验证。保留多模型一致认为高质量的样本。有人工校验资源的情况下加一道人工检验最保险。
数据构造流水线：设计种子prompt集合 → 强模型（GPT-4/Claude）生成候选回答 → 多模型（GPT-4 + Claude + Gemini）交叉验证打分 → 一致性筛选 → 人工抽检10-20% → 去重和格式统一 → 成品数据集。这条流水线在多个团队中被证明是最可靠的SFT数据生产方式。

SFT数据工程的隐形杀手

这可能是本章最重要的一个章节。SFT数据工程中有几个极其隐蔽的bug，一旦触发，轻则训练白费，重则模型完全不可用。而这些bug的共同特点是——训练loss看起来一切正常。

隐形杀手一：Chat Template不一致。训练时使用一种对话模板，推理时用另一种模板，模型输出会一塌糊涂——多余的特殊token、角色标签显示在回答中、回答被截断、甚至完全无法工作。而最危险的是：训练loss仍然会正常下降。

BugBuster喵的提醒非常到位："大模型微调还有一个常见坑：chat template不一致。训练用一种模板，推理用另一种模板，效果会非常难看，而且不容易第一时间发现。"

你的检查清单：加载tokenizer后确认chat_template字段非空；写一个端到端验证函数，确保同一个messages序列在训练和推理时的格式化结果完全一致；将template写入训练配置文件而非依赖默认值；不同推理框架（vLLM、TGI）对chat template的自动处理可能不同，必须逐项验证。

隐形杀手二：Loss只计算assistant部分。SFT训练中，loss应该只计算assistant回复部分的token。User问句和system prompt必须被mask掉。如果你在实现时不小心把user部分也计入了loss，模型会学到"预测用户的问法"——这完全不是你想要的。BugBuster喵说："assistant部分和user部分都参与loss，模型学得乱七八糟。"

隐形杀手三：训练和推理预处理不一致。数据清洗、截断策略、特殊token处理——训练和推理必须完全相同。Tokenizer不一致是最常见的问题：训练时用的tokenizer和推理时的不匹配，导致生成结果异常。特别是当你在Hugging Face上加载模型时，tokenizer_config.json中的chat_template字段可能在不同版本的transformers库中不同，必须显式检查。

自查清单（开始SFT前必做）：
1. 训练时的chat template格式化输出一段消息，保存为txt
2. 推理时用同一个messages列表，用含add_generation_prompt=True的方式格式化
3. 对比两个输出——前缀部分必须完全一致
4. 检查特殊token的id：tokenizer.eos_token_id、bos_token_id、pad_token_id在训练和推理中是否一致
5. 确认loss mask正确：system prompt和user部分mask=0，assistant部分mask=1

数据混合与课程学习

SFT数据除了质量，还有两个非常重要的"组织"维度。

单轮 vs 多轮数据比例。BugBuster喵明确指出："大模型SFT里，单轮和多轮数据比例就是超参数。"这个比例对模型最终的对话行为有直接影响：纯QA场景（如问答机器人、分类任务）单轮占比80-100%；对话助手场景（如客服、聊天）多轮占比30-50%。比例失衡会导致模型在某种对话模式下表现极差。这个比例不是拍脑袋定的——把它当成超参数进行实验搜索。

课程学习：由易到难排序。对数据集标注难易程度，训练时先喂简单样本、再逐步增加困难样本的比重。Tang AI的经验是："可以对数据集设置一个难易程度，让模型先学简单后学难的，有利于模型的加速收敛。"具体策略：前20%的training steps以简单样本为主，中间60%均匀混合，最后20%增加困难样本比例。这在数据量大、任务多样时效果尤其明显。

SFT不需要思维链和Few-shot。这是一个重要的决策原则。CoT和Few-shot是推理时激发模型能力的prompt技巧，不是训练格式。Tang AI说得很直白："思维链+fewshot肯定是可以提高原始模型能力的，但是SFT就不用加思维链和fewshot了，直接用样本堆死它。而且思维链+fewshot也超级耗时。所以还不如直接造大量高质量样本，直接让模型学习。暴力但是很有效。"

SFT vs RAG的选择：还有一个更根本的决策问题——你的问题到底该用SFT还是RAG？BugBuster喵给出了清晰的判断标准："别把微调当成万能开关。很多企业知识类问题，知识应该放在检索系统里，不该硬塞进模型参数里。微调用来调风格、格式、流程、领域表达，检索用来提供事实依据。这个判断做对，比调50组参数更重要。"

记住：SFT适合调整输出风格和格式规范，RAG适合注入知识。两者不是竞争关系，是互补关系。

SFT监控：不要迷信Loss

最后也是最重要的一条原则——不要迷信Loss。

Tang AI的原话非常直白："事实上loss，对于大模型训练仅能做一个基本的判断了。基本上到了中后期，loss小，效果不一定好，甚至还会出现loss先下降后上升的情况，即使是在训练集。"

这不是个别现象。大模型SFT训练中后期，loss与模型实际效果之间关联度很低。可能出现loss下降但效果变差（过拟合开始），或者loss先降后升（模型从"学习范式"转向"记忆数据"）。

正确做法：定期保存checkpoint，在每个checkpoint上跑业务指标（准确率、F1、人工评测等）来评估效果。不要等到全部训练结束再看结果——那是浪费算力。Loss仅作为训练是否正常的初步参考——如果loss完全不降，说明训练有问题；但只要loss在降，并不代表模型在变好。

训练轮数（Epoch）方面，SFT不同于预训练——数据量通常有限，2-3个epoch后模型就会从"学习范式"转向"记忆数据"。建议在验证集上监控指标变化，一旦指标不再提升或开始下降就停止。通常2-3个epoch是合理的范围，不要多跑。

下面是SFT常见陷阱的一览表，建议你在每次训练前对照检查：

陷阱 症状 排查方法 解决方案
Chat Template不一致 训练loss正常，但模型输出乱七八糟（多余特殊token、格式混乱） 对比训练和推理时同一messages的格式化结果 将template写入训练配置，写单元测试验证一致性
Loss包含user部分 模型开始"预测用户的问法"，回答质量差 检查loss mask实现，确认user部分被mask 只对assistant部分token计算loss
Tokenizer不一致 生成结果出现乱码或异常token 检查训练和推理的tokenizer配置是否相同 统一tokenizer，确认special token id一致
数据风格不统一 训练loss正常，但模型输出格式五花八门 随机抽样检查训练数据的格式一致性 统一所有样本的输出格式
训练分布不覆盖测试分布 测试集效果差而训练集好 分析测试集bad case，检查训练集是否有相似样本 针对性补充5-10条样本
单轮/多轮比例失衡 模型在某种对话模式下表现极差 检查数据集中单轮/多轮数据分布 根据应用场景调整比例作为超参数搜索
训练轮数过多导致过拟合 验证集指标下降但训练loss仍在降低 对比不同checkpoint的验证集指标 早停，通常2-3个epoch足够
训练和推理预处理不一致 生成结果与训练时观察到的行为不同 对比数据清洗、截断策略、特殊token处理 确保数据管线完全一致

SFT是连接预训练模型和实际应用的桥梁，而数据工程是这座桥梁的基石。掌握了LoRA微调的数学本质（第23章）和SFT数据工程的科学方法（本章），你已经具备了让大模型"学会做新事情"的完整能力。但"学会做"不等于"做得好"——后者要靠RLHF。下一章，我们将进入RLHF对齐训练的底层逻辑——从PPO到DPO再到GRPO，看RL如何将模型能力推到上限。我们第25章见。
