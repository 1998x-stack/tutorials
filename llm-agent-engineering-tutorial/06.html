第6章 记忆系统设计

大家好，我是你们的讲师。这一章我们要讨论一个非常"人味儿"的话题——记忆。

你有没有遇到过这种情况：你跟客服 Agent 说"我上周买了个手机"，它问你"什么手机？"；你说"我刚才说了，iPhone 15 Pro"，它说"抱歉，我没有之前对话的记录"。是不是很令人恼火？这正是 Agent "没有记忆"的典型表现。

对于人类来说，记忆是天生的——我们有短期记忆（记住刚刚说过的话）和长期记忆（记住昨天、上周、去年的事）。但对 Agent 来说，记忆是需要精心设计的系统能力。没有记忆的 Agent，永远只能活在"当下"，每个对话都是第一次见面。而有良好记忆设计的 Agent，可以记住用户的偏好、之前的对话内容、长期积累的知识——这才是真正"智能"的体现。

核心洞察： LLM 本身的上下文窗口（Context Window）不是真正的"记忆"——它只是当前请求的临时存储。真正的记忆系统需要解决三个问题：(1) 记忆什么（选择），(2) 存在哪里（存储），(3) 怎么检索（检索）。就像人类大脑不会记住所有细节一样，好的记忆系统也懂得"遗忘"的艺术。

6.1 记忆的分类：感知记忆/短期记忆/长期记忆

借鉴认知心理学的 Atkinson-Shiffrin 记忆模型，Agent 的记忆系统也可以分为三个层次：

记忆类型 人类类比 Agent 实现 容量 持久性
感知记忆 眼睛看到东西的瞬间印象 当前用户输入 + 系统状态 毫秒级别 不持久
短期记忆 说一句话时记住前几句话 对话历史（消息列表）+ 上下文窗口 几分钟到几小时 会话内
长期记忆 记住昨天和去年的事 向量数据库 / 知识图谱 / 文件系统 天到年 跨会话

感知记忆

感知记忆是 Agent 接收到的原始输入。在文本场景中，就是用户刚刚输入的那条消息。在多模态场景中，还包括图片、音频、文件等。感知记忆的特点是"未经过滤"——Agent 接收到什么就是什么。

短期记忆

短期记忆就是 Agent 的上下文窗口——当前会话中所有对话历史的累积。LLM 的一次 API 调用中，所有 messages 参数里的内容就是 Agent 的短期记忆。问题是：上下文窗口不是无限的（GPT-4o 是 128K，Claude Sonnet 4 是 200K），而对话会越来越长。

长期记忆

长期记忆是存储在 Agent 外部的、跨会话的知识。它可以包括：用户画像和偏好、历史对话的摘要、学习到的知识、已完成的任务记录。长期记忆通常使用向量数据库或结构化存储来实现。

设计原则： 并不是所有信息都值得记住。好的记忆系统设计遵循"分层存储"原则——最近和最重要的信息放在短期记忆（快速读取），历史信息压缩后放入长期记忆（按需检索）。记住该记住的，忘记该忘记的，这才是聪明的记忆系统。

6.2 上下文窗口管理与滑动窗口

上下文窗口管理是 Agent 开发中最常见的工程挑战之一。随着对话进行，消息列表越来越长，既消耗 Token 也增加模型处理时间。如果不加管理，一个忙碌的客服 Agent 可能半天就能把 128K 的上下文窗口塞满。

滑动窗口策略

滑动窗口（Sliding Window）是最简单也最有效的上下文管理策略——只保留最近 N 轮对话：

from collections import deque

class SlidingWindowMemory:
    """滑动窗口短期记忆"""

    def __init__(self, max_rounds: int = 20):
        self.messages = deque(maxlen=max_rounds * 2)  # 每轮 2 条（用户+助手）
        self.max_rounds = max_rounds
        self.system_prompt = None

    def set_system_prompt(self, prompt: str):
        """设置 System Prompt（不参与滑动）"""
        self.system_prompt = {"role": "system", "content": prompt}

    def add_message(self, role: str, content: str):
        """添加消息，自动淘汰最旧的"""
        self.messages.append({"role": role, "content": content})

    def get_context(self) -> list:
        """获取完整的上下文列表"""
        if self.system_prompt:
            return [self.system_prompt] + list(self.messages)
        return list(self.messages)

    def get_token_estimate(self) -> int:
        """粗略估计 Token 数量（中文字符数 * 2）"""
        total = 0
        for msg in self.messages:
            total += len(msg["content"]) * 2
        return total

进阶策略：Token 计数 + 智能裁剪

单纯按"轮数"滑动窗口有一个问题——不同轮次的长度差异很大。用户可能某一轮发了一段很长的代码，而其他轮次只是简单的问答。更好的策略是按 Token 数量来裁剪：

class TokenAwareMemory:
    """基于 Token 计数的智能记忆管理"""

    def __init__(self, max_tokens: int = 64000):
        self.messages = []
        self.max_tokens = max_tokens

    def add_message(self, role: str, content: str):
        self.messages.append({"role": role, "content": content})
        self._trim()

    def _trim(self):
        """从最旧的消息开始删除，直到总 Token 数低于限制"""
        while self._total_tokens() > self.max_tokens and len(self.messages) > 2:
            # 保留 System Prompt 和最新的用户消息
            removed = self.messages.pop(0)
            print(f"丢弃旧消息: {removed['role']} ({len(removed['content'])} chars)")

    def _total_tokens(self) -> int:
        return sum(len(m["content"]) for m in self.messages) * 2
踩坑实录： 2024 年我们构建一个对话式数据分析 Agent，用户会在对话中上传 CSV 文件并逐步分析。问题来了——每次分析的结果都留在消息列表中，5 轮对话后上下文就超过了 80K Token。更糟的是，用户每轮都要引用之前的分析结果。单纯用滑动窗口会导致"失忆"。最终我们采用"摘要 + 关键数据保留"策略：每轮结束后，LLM 自动生成一个简短的结构化摘要（包含关键数据和结论），只保留摘要和最近 2 轮的完整对话。效果很好——既控制了 Token，又没有丢失关键信息。

6.3 向量数据库选型

长期记忆的核心存储方案是向量数据库（Vector Database）。它将信息编码为向量嵌入（Embedding），存储在向量数据库中，然后通过语义相似度搜索来检索相关信息。

截至 2026 年中，以下三款向量数据库是 Agent 开发的主流选择：

数据库 部署方式 支持的索引 适合场景 成本
Chroma 本地嵌入式 / 客户端-服务端 HNSW, IVF 原型开发、单机部署、小到中等规模 免费
Pinecone 全托管云服务 自动优化 生产环境、高可用要求、大规模 按量付费
Qdrant 本地 / Docker / 云服务 HNSW, 过滤索引 自托管生产环境、需要过滤功能 免费（自托管）

Chroma 快速上手

Chroma 因为极简的 API 设计，是原型开发的首选：

import chromadb
from chromadb.utils import embedding_functions

# 初始化 Chroma
client = chromadb.PersistentClient(path="./agent_memory")
collection = client.get_or_create_collection(
    name="conversation_memory",
    embedding_function=embedding_functions.DefaultEmbeddingFunction()
)

# 存储记忆
def save_to_memory(user_id: str, content: str, metadata: dict = None):
    """将对话内容存入长期记忆"""
    collection.add(
        documents=[content],
        metadatas=[metadata or {"user_id": user_id, "timestamp": "2026-06-30"}],
        ids=[f"{user_id}_{hash(content)}"]
    )

# 检索记忆
def search_memory(user_id: str, query: str, top_k: int = 5):
    """根据语义搜索用户的历史记忆"""
    results = collection.query(
        query_texts=[query],
        n_results=top_k,
        where={"user_id": user_id}
    )
    return results['documents'][0]
选型建议： 如果你在搭建原型或个人项目，Chroma 是最快的选择——一条 pip install chromadb 就能开始用。如果部署到生产并需要高可用和 SLA，Pinecone 的零运维体验最好。如果你需要自托管且对数据安全有要求（很多金融、医疗客户的场景），Qdrant 的自托管方案最成熟。

6.4 摘要记忆与历史压缩

向量数据库不是长期记忆的唯一方式。还有一种非常实用且成本更低的方式——摘要记忆（Summary Memory）。

核心思想：每过一段时间（比如每 5 轮对话），让 LLM 对之前的对话进行摘要，用摘要替换原始对话历史。这样信息没有被丢失，但 Token 消耗大幅减少了。

class SummarizingMemory:
    """带摘要功能的记忆系统"""

    def __init__(self, llm_client, summary_interval: int = 5):
        self.llm = llm_client
        self.summary_interval = summary_interval
        self.messages = []
        self.summary = ""
        self.turn_count = 0

    def add_and_compress(self, role: str, content: str):
        """添加消息，必要时进行摘要压缩"""
        self.messages.append({"role": role, "content": content})
        self.turn_count += 1

        if self.turn_count % self.summary_interval == 0:
            self._summarize()

    def _summarize(self):
        """使用 LLM 生成对话摘要"""
        conversation = "\n".join(
            f"{m['role']}: {m['content'][:500]}"
            for m in self.messages[-self.summary_interval * 2:]
        )

        summary_prompt = f"""请对以下对话进行摘要，保留关键信息（用户偏好、已解决的问题、重要决定）。
        历史摘要（如有）：{self.summary}
        最新对话：{conversation}
        新摘要："""

        response = self.llm.complete(summary_prompt)
        self.summary = response.text

        # 保留最近的 2 轮完整对话 + 摘要
        self.messages = self.messages[-(self.summary_interval * 2):]

    def get_context(self):
        """获取包含摘要的上下文"""
        context = []
        if self.summary:
            context.append({
                "role": "system",
                "content": f"【历史对话摘要】{self.summary}"
            })
        context.extend(self.messages)
        return context
实践经验： 摘要记忆是我们团队在大多数生产项目中使用的方案。原因很简单——它不需要额外依赖（不需要向量数据库），成本低（一次摘要调用比向量存储便宜得多），效果也不错。当然，对于需要精确回忆特定细节的场景（比如"我三个月前说过的那句话"），向量数据库的语义搜索是无法替代的。最佳实践是两者结合：摘要记忆处理日常压缩，向量数据库用于深度检索。

6.5 混合记忆架构实战

在实际生产中，单一的记忆方案通常不够用。最有效的是混合记忆架构（Hybrid Memory Architecture），它结合了短期记忆、摘要记忆和向量长期记忆的优势。

三层混合架构

class HybridMemory:
    """混合记忆系统：短期 + 摘要 + 向量长期"""

    def __init__(self, llm_client):
        # 第一层：短期记忆（滑动窗口）
        self.short_term = SlidingWindowMemory(max_rounds=10)

        # 第二层：摘要记忆
        self.summary_memory = SummarizingMemory(llm_client, summary_interval=5)

        # 第三层：长期记忆（向量数据库）
        self.vector_client = chromadb.PersistentClient(path="./memory_store")
        self.long_term = self.vector_client.get_or_create_collection(
            name="long_term_memory"
        )

    def add_interaction(self, user_id: str, user_msg: str, agent_reply: str):
        """添加一次交互到所有记忆层"""

        # 短期记忆
        self.short_term.add_message("user", user_msg)
        self.short_term.add_message("assistant", agent_reply)

        # 摘要记忆
        self.summary_memory.add_and_compress("user", user_msg)
        self.summary_memory.add_and_compress("assistant", agent_reply)

        # 长期记忆（选择性存储——只存重要内容）
        if self._is_important(user_msg, agent_reply):
            self.long_term.add(
                documents=[f"用户: {user_msg}\n助手: {agent_reply}"],
                metadatas=[{"user_id": user_id, "timestamp": str(datetime.now())}],
                ids=[f"{user_id}_{uuid4()}"]
            )

    def get_relevant_context(self, user_id: str, query: str) -> str:
        """融合所有记忆层获取上下文"""
        context_parts = []

        # 1. 短期记忆
        short_ctx = self.short_term.get_context()
        context_parts.append("【最近对话】\n" + format_messages(short_ctx))

        # 2. 摘要记忆
        if self.summary_memory.summary:
            context_parts.append(
                "【会话摘要】\n" + self.summary_memory.summary
            )

        # 3. 长期记忆（语义检索）
        long_results = self.long_term.query(
            query_texts=[query],
            n_results=3,
            where={"user_id": user_id}
        )
        if long_results['documents'][0]:
            context_parts.append(
                "【历史相关信息】\n" + "\n---\n".join(long_results['documents'][0])
            )

        return "\n\n".join(context_parts)

    def _is_important(self, user_msg: str, agent_reply: str) -> bool:
        """判断内容是否重要到需要存入长期记忆"""
        keywords = ["偏好", "设置", "记住", "喜欢", "不喜欢",
                    "重要", "计划", "项目", "客户"]
        return any(kw in user_msg or kw in agent_reply for kw in keywords)

记忆检索优先级

在混合架构中，检索信息的优先级顺序是：

  1. 短期记忆（当前对话的上下文）—— 直接拼接，无需检索
  2. 摘要记忆（当前会话的历史压缩）—— 直接读取摘要
  3. 长期记忆（所有历史对话）—— 语义搜索获取最相关片段
性能优化： 在混合架构中，要注意"检索时机"——不是每一步都需要查向量数据库。只有当前短期记忆中找不到足够信息时，才触发长期记忆检索。这样既保障了响应速度（避免不必要的向量查询），也保证了信息完整性。

6.6 Mem0 等记忆层框架

虽然在 2024-2025 年出现了 Mem0、memgpt 等专门做 Agent 记忆的框架，但到 2026 年，记忆功能已经逐步内化到了主流 Agent 框架中。LangChain、CrewAI 和 AutoGen 都内置了成熟的记忆系统。

不过 Mem0 依然是值得关注的选择，因为它专注于记忆这一件事：

# Mem0 示例 — 专注记忆层的专业框架
from mem0 import Memory

# 初始化记忆系统
memory = Memory(
    config={
        "vector_store": {
            "provider": "qdrant",
            "config": {
                "host": "localhost",
                "port": 6333,
            }
        },
        "llm": {
            "provider": "openai",
            "config": {
                "model": "gpt-4o",
            }
        }
    }
)

# 添加记忆（自动提取关键信息）
memory.add(
    messages=[
        {"role": "user", "content": "我叫张三，喜欢科技类书籍"},
        {"role": "assistant", "content": "好的，张三，我记住了你喜欢科技类书籍"}
    ],
    user_id="user_123"
)

# 检索记忆
results = memory.search(
    query="用户的兴趣爱好是什么？",
    user_id="user_123"
)
# 返回: [{"text": "用户喜欢科技类书籍", ...}]
选型总结：
  • 如果你使用 LangChain 生态：langchain.memory 模块提供了 ConversationBufferMemory、SummaryMemory、VectorStoreRetrieverMemory 等开箱即用的实现
  • 如果你使用 CrewAI：内置的 短期记忆、长期记忆 和 实体记忆 三层架构
  • 如果你需要独立于框架的记忆层：Mem0 提供了最灵活的统一接口
  • 如果项目非常小：手写一个基于 Chroma 的简单记忆系统（就像本章前面示例那样）完全够用
