第11章 FastSpeech：并行合成革命

大家好，我是你们的讲师。上一章我们讲声码器从"分钟级"进化到了"毫秒级"。但声码器只是TTS流水线的一半——另一半是声学模型，而当时最好的声学模型Tacotron 2还是一个自回归架构。解码器逐帧生成梅尔频谱，每生成一帧都要等上一帧的输出。

2019年，来自微软和浙江大学的团队发表了FastSpeech，打出了"非自回归并行合成"这面大旗。它的口号很简单：不要逐帧生成，要一次性生成全部。

自回归的三大罪状

在讲FastSpeech之前，我们得先聊聊自回归TTS（也就是Tacotron 2）的三大问题。这些问题在Tacotron 2发布时就已经存在，只是当时大家沉浸在"MOS 4.53"的喜悦中，暂时忽略了它们。

罪状一：推理慢。这一点我们已经反复强调了。自回归解码的时间复杂度是O(T)的——T是梅尔帧数。对于包含20个字的中等长度句子，大约需要生成200帧梅尔频谱。每帧都要Attention→LSTM→投影→PostNet，累积起来就是几百步的顺序计算，无法并行。

罪状二：稳定性差。这是自回归最让人头疼的问题。自回归模型本质上是"用自己之前的预测来预测下一步"——一个错误会被不断放大。在Tacotron 2中，偶尔会出现注意力崩溃（attention alignment突然发散）、漏词、重复词等问题。更糟糕的是，这些问题无法在训练时被发现（因为训练使用Teacher Forcing，避免了误差累积），只有在推理时才暴露。学术界把这叫做"Exposure Bias"（曝光偏差）。

罪状三：不可控。自回归TTS没有显式的"时长"概念。你不能说"这句话读快一点"或者"在这个词上停一下"。所有韵律变化都隐含在模型中，用户无法干预。

关键洞察：自回归TTS的这三个问题有一个共同的根源——时长和文本的对齐是隐式学习的。在Tacotron中，注意力机制同时承担了两个任务："决定读到哪里了"和"决定怎么发音"。这导致注意力既要做位置对齐又要做内容选择，容易混淆。FastSpeech的核心思路就是把"对齐"显式化。

FastSpeech的核心创新

FastSpeech的架构设计围绕一个核心理念：把"音素到梅尔帧的时长映射"显式建模出来。它包含三个关键组件：

  • 前馈Transformer（Feed-Forward Transformer）：使用FFT Block（Feed-Forward Transformer block）替代了Tacotron的RNN+Attention，本质上是一个Transformer编码器的变体，可以并行处理整个输入序列。
  • 时长预测器（Duration Predictor）：一个小型卷积网络，输入是音素级别的编码表示，输出是每个音素对应的梅尔帧数（也就是时长）。它是一个回归模型，预测每个音素持续多少帧。
  • 长度调节器（Length Regulator）：根据时长预测器的输出，将音素级别的隐表示"展开"为帧级别的隐表示。这是FastSpeech最核心的模块。
# FastSpeech 长度调节器（Length Regulator）示意

import torch

def length_regulator(hidden_states, duration):
    """
    hidden_states: [B, T_phoneme, dim]  音素级别的隐表示
    duration:      [B, T_phoneme]        每个音素对应多少帧
    返回:          [B, T_mel, dim]       帧级别的隐表示
    """
    B, T_p, D = hidden_states.shape
    outputs = []
    for b in range(B):
        expanded = []
        for t in range(T_p):
            # 将第t个音素的隐表示复制 duration[b,t] 次
            repeat = int(duration[b, t].item())
            expanded.append(hidden_states[b, t].unsqueeze(0).repeat(repeat, 1))
        outputs.append(torch.cat(expanded, dim=0))
    return pad_sequence(outputs, batch_first=True)

# 示例：音素时长
# 输入音素:   ni3 (3帧)  hao3 (4帧)  shi4 (3帧)  jie4 (4帧)
# 展开结果:   [ni,ni,ni, hao,hao,hao,hao, shi,shi,shi, jie,jie,jie,jie]
# 共 3+4+3+4 = 14 帧，正好对应梅尔频谱的时间维度

有了长度调节器，FastSpeech可以：

  • 并行生成：前馈Transformer一次性处理所有音素，长度调节器一次性展开所有帧，没有顺序依赖。
  • 可控语速：直接把时长预测器的输出乘以一个系数——乘以0.8就是"说快20%"，乘以1.2就是"说慢20%"。
  • 稳定可靠：不再有注意力崩溃问题，因为根本没有注意力——对齐是显式通过时长来控制的。

知识蒸馏训练

FastSpeech有一个关键的训练细节：它需要一个预训练的Teacher模型来提供时长监督。

具体来说，FastSpeech的训练分为两步：

  1. 训练Teacher（Tacotron 2）：用标准的Tacotron 2训练一个自回归TTS模型。
  2. 蒸馏（Distillation）：用Teacher模型合成大量文本-音频对，从Teacher的注意力对齐中提取每个音素的时长（注意力权重峰值的位置差异就是音素间的帧数差异），然后用这些时长标签训练FastSpeech的时长预测器。

这个两步训练策略虽然在学术上有效，但在工程上并不优雅——你需要先训一个Teacher模型，再训一个Student模型。为了克服这个限制，FastSpeech 2直接在训练数据上标注时长（通过MFA等工具强制对齐），不再需要蒸馏。

蒸馏的哲学：知识蒸馏在FastSpeech中有一种微妙的"双重作用"。一方面，它让Student模型学到了对齐信息；另一方面，Teacher合成数据的"不完美"反而成了Student的"平缓目标"——因为Teacher的音质有限，Student不需要学习那些过于尖锐的频谱细节，这对非自回归模型的稳定性是有益的。FastSpeech 2放弃蒸馏后，虽然去掉了训练复杂度，但生成的频谱细节确实更难学——这可能是FastSpeech 2使用音高/能量预测器来补偿的原因之一。

FastSpeech 2：全面升级

2020年11月，微软团队发表了FastSpeech 2，针对FastSpeech的不足做了三项重要改进：

  • 告别知识蒸馏：直接使用MFA（Montreal Forced Aligner）从原始录音中提取音素时长，不再需要Teacher模型。
  • 显式的变音信息预测：引入音高（F0）预测器和能量（Energy）预测器。这些变音信息（prosodic features）作为辅助输出，帮助模型学习更丰富的韵律变化。
  • 移除Teacher-Student差距：FastSpeech 1因为蒸馏了Tacotron 2的输出，音质上限受Teacher限制。FastSpeech 2直接学习真实录音的频谱，理论上限更高。
# FastSpeech 2 的辅助预测器
# 在FFT Block之上增加三个辅助预测头

# 1. 音高预测器（Pitch Predictor）
pitch_pred = Conv1d(hidden_dim, 1)  # 输出连续F0值
pitch_target = extract_f0(waveform)  # 从真实音频中提取F0

# 2. 能量预测器（Energy Predictor）
energy_pred = Conv1d(hidden_dim, 1)  # 输出每帧能量的L2范数
energy_target = mel_spec.norm(dim=1) # 从梅尔频谱计算帧能量

# 3. 时长预测器（Duration Predictor）
dur_pred = Conv1d(hidden_dim, 1)  # 输出每个音素的时长
dur_target = mfa_alignment(text, waveform)  # MFA工具强制对齐

# 训练时，通过添加这些辅助损失项来指导学习
L_total = L_mel + λ_pitch * L_pitch + λ_energy * L_energy + λ_dur * L_dur

FastSpeech 2在MOS上追平了Tacotron 2（在多个数据集上达到了4.2-4.4），同时将梅尔频谱生成速度提升了约270倍。更重要的是——它第一次让TTS可以在普通CPU上达到实时合成。

历史性时刻：当时有工程师在Twitter上分享了一段视频——在一台2015年的Intel i5处理器上，FastSpeech生成了一段10秒的语音，耗时8秒。CPU上的RTF只有0.8，接近实时。这个Demo引发了大量转发，因为在此之前，TTS的CPU推理一直被当作一个次要问题——"反正我们有GPU"。但现实是，绝大多数线上部署环境没有GPU。FastSpeech让"纯CPU实时TTS"成为可能，这对TTS技术的普及起到了巨大的推动作用。

270倍加速的工程意义

FastSpeech的重要程度，怎么强调都不过分。在工程意义上，它把TTS从"实验室技术"变成了"可部署产品"：

维度 Tacotron 2（自回归） FastSpeech 1 FastSpeech 2
生成方式 逐帧自回归 并行一次生成 并行一次生成
Mel生成速度（V100） ~100帧/秒 ~27000帧/秒 ~27000帧/秒
MOS 4.53（+WaveNet） ~3.8（蒸馏损失） ~4.2-4.4
语速可调 否 是 是
训练复杂度 单阶段 两阶段（蒸馏） 单阶段（需MFA）
注意力崩溃 偶尔发生 不会 不会
CPU实时推理 不行 基本可以 可以

从表格可以清晰看到：FastSpeech系列在"速度"和"稳定性"两个维度上完胜Tacotron 2，在"质量"维度上全面追平（FastSpeech 2）。TTS技术从2016年的WaveNet开始，到2020年的FastSpeech 2，完成了"质量追平人类"和"速度追上实时"的双重目标。

踩坑实录：曾经在一个语音交互产品中尝试把FastSpeech 2部署到生产环境。测试时在标准数据集（LJSpeech英文女声）上效果很好。但上线后用户反馈中文合成"有点怪"——后来排查发现，MFA（蒙特利尔强制对齐工具）对中文的声调边界检测不够准确。中文每个音节不只有音素时长，还有声调信息，F0预测器在中文上需要额外调优。这个经历告诉我们：FastSpeech框架是通用的，但时长对齐工具在不同语言上的表现差异巨大。选择TTS方案时，一定考虑目标语言的工具链成熟度。

FastSpeech的"并行合成"思想深刻影响了后来几乎所有的TTS研究。从VITS到NaturalSpeech，从Glow-TTS到CosyVoice，虽然每个模型在架构选择上各不相同，但"非自回归、并行生成、显式时长建模"这三个核心原则被广泛继承。

不过，FastSpeech系列的思路是"保持声学模型和声码器分离"——声学模型生成频谱，声码器把频谱变成波形。但总有人在想：既然两者都是深度学习模型，为什么不把它们合并成一个呢？下一章我们讲解的VITS，就是回答这个问题的终极答案。
