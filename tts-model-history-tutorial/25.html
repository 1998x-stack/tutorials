第25章 评估与A/B测试实践

大家好，我是你们的讲师。上一章我们亲手训练了一个TTS模型——经过几天的等待，checkpoint终于保存下来了。用脚本合成了几条语音，听起来似乎还可以。但问题来了："还可以"到底是多好？比基线模型好吗？比竞品好吗？用户觉得好吗？

如果你不能回答这些问题，你就无法判断你的模型是"好"还是"看起来好但实际上不好"。在TTS领域，这种困惑尤其普遍——因为语音质量是一个高度主观的体验。我记得有一个团队花了三个月优化模型的MOS，从3.8提升到了4.1，结果上线后用户的投诉率反而上升了。后来才发现，虽然模型整体更自然了，但偶尔会出现一些"毛刺"，而这种不稳定的质量比稳定的低质量更让用户反感。这个故事告诉我们一个道理：评测不是终点，而是指导优化的罗盘。评测方法错了，优化方向也就歪了。

这一章，我们来搭建一套完整的TTS评测体系，从主观评测到自动化指标，从说话人相似度到A/B测试——全部都是可以直接拿去用的实战方案。

主观评测平台搭建

TTS评测的黄金标准始终是"人听了觉得怎么样"。虽然自动化指标越来越强，但没有任何一个数值能完全替代人耳的主观感受。搭建一个主观评测平台并不需要很复杂——实际上，一个简单的HTML页面就够了。

一个好的主观评测平台应该满足三个原则：

  • 随机化：每个听评者听到的音频顺序应该是随机打乱的，避免顺序效应（人们往往倾向于给前几条和最后几条音频更高的分数）。
  • 盲测：听评者不应该知道正在听的是哪个模型的声音。文件名应该用随机ID替代，比如"sample_A.wav"和"sample_B.wav"而不是"our_model.wav"和"baseline.wav"。
  • 评分界面UX：评分操作越简单越好。我见过最有效的设计是：页面上只显示一个播放按钮、一个5级李克特量表（1-5分），和一个"下一题"按钮。不要在同一页上放太多信息，不要要求听评者做复杂操作。
实操建议：不要自己招募听评者做一次性的MOS评测——这样得到的数据往往不稳定。更好的做法是利用众包平台（如Amazon Mechanical Turk或国内的问卷星）收集至少20名听评者的评分。在使用听评者数据之前，一定要做"筛选测试"：混入几条质量明显很差（比如噪声）的样本，把不认真听的人筛掉。根据经验，大约15-20%的众包听评者数据会被筛掉——这是正常的。

自动化MOS评估

人工评测虽然可靠，但代价高昂——组织一次20人参与的MOS评测，从准备到收集数据往往需要3-5天。在模型迭代频繁的开发阶段，你不可能每次调参都组织一次人工评测。这时，自动化的MOS预测模型就派上了用场。

目前最常用的两个工具是：UTMOS和SpeechMOS。它们都是基于深度学习的MOS预测模型，输入一段音频，输出预测的MOS分数。使用方法非常简单：

# 安装SpeechMOS
pip install speechmos

# Python脚本：批量预测MOS
from speechmos import SpeechMOS

predictor = SpeechMOS()
scores = {"sample_A.wav": None, "sample_B.wav": None}

for wav_path in scores.keys():
    score = predictor.predict(wav_path)
    scores[wav_path] = score
    print(f"{wav_path}: MOS = {score:.3f}")

# 输出:
# sample_A.wav: MOS = 3.847
# sample_B.wav: MOS = 4.123

自动化MOS的局限性也同样明显：

  • 领域偏移：UTMOS/SpeechMOS都是在特定数据集上训练的（如VCC2020、BVCC），如果你的合成语音风格与训练数据差异较大，预测分数可能不准确。
  • 无法捕捉"灾难性失败"：如果模型偶尔产生严重的爆破音或断句错误，人工评测会大幅扣分，但自动化MOS可能会"平均掉"这些偶发问题。
  • 对噪声敏感：自动化MOS对背景噪声比较敏感，有时会把高质量的带有轻微底噪的合成语音误判为低分。

所以我的建议是：自动化MOS用于内部筛选和快速迭代，人工MOS用于最终验收。两者是互补关系，不是替代关系。

说话人相似度脚本实战

如果你在做语音克隆或多说话人TTS，除了听感自然度，"音色像不像"是一个核心指标。衡量说话人相似度的标准方案是：提取参考音频和合成音频的说话人嵌入向量（speaker embedding），然后计算它们的余弦相似度。

当前最主流的说话人嵌入提取模型是ECAPA-TDNN，它已经在说话人验证（Speaker Verification）任务上超越了x-vector和ResNet-based的方法。如果你需要更轻量的方案，Resemblyzer（基于GE2E损失训练的LSTM）也是一个不错的选择。下面是一个完整的相似度计算脚本：

# 安装依赖
# pip install speechbrain torchaudio

import torch
import torchaudio
from speechbrain.inference.speaker import EncoderClassifier

# 加载ECAPA-TDNN预训练模型
classifier = EncoderClassifier.from_hparams(
    source="speechbrain/spkrec-ecapa-voxceleb",
    savedir="pretrained_models/spkrec-ecapa-voxceleb"
)

def compute_speaker_similarity(ref_path, syn_path, classifier):
    """计算参考音频与合成音频的说话人相似度"""
    # 加载并重采样到16kHz
    def load_audio(path):
        signal, sr = torchaudio.load(path)
        if sr != 16000:
            resampler = torchaudio.transforms.Resample(sr, 16000)
            signal = resampler(signal)
        return signal

    ref_wav = load_audio(ref_path)
    syn_wav = load_audio(syn_path)

    # 提取说话人嵌入（512维向量）
    ref_embedding = classifier.encode_batch(ref_wav).squeeze()
    syn_embedding = classifier.encode_batch(syn_wav).squeeze()

    # 计算余弦相似度
    similarity = torch.nn.functional.cosine_similarity(
        ref_embedding, syn_embedding, dim=0
    )

    return similarity.item()

# 示例用法
ref_audio = "reference_speaker.wav"
syn_audio = "synthesized_output.wav"
score = compute_speaker_similarity(ref_audio, syn_audio, classifier)

print(f"说话人相似度: {score:.4f}")
# 一般 score > 0.75 表示音色保持较好
# score > 0.85 表示非常相似

判断相似度分数的经验阈值是：0.75以上说明音色一致性可以接受，0.85以上说明非常相似。不过需要注意，这个阈值是和模型强相关的——换一个嵌入提取模型，阈值的含义就会不同。建议你用一个"正例集"（同一个人的不同录音）和"负例集"（不同人的录音）先校准一下阈值。

A/B测试设计

单个数值（无论是MOS还是相似度）只能告诉你"模型有多好"，但很多时候你更需要知道的其实是：模型A比模型B好吗？好多少？这时就需要A/B测试登场了。

一个规范的A/B测试应该这样设计：

  • 样本量估算：在TTS评测中，通常需要20-30条测试句子、20名以上听评者，才能得到统计上有效的结果。具体样本量取决于你期望检测到的效果量（effect size）——如果你只想检测0.2分以上的MOS差异，大约需要30人；如果你想检测0.1分的差异，可能需要100人以上。
  • 随机化分组：每个听评者随机分配到一组，或采用"配对设计"——每个听评者同时听到A和B的同一段文本，然后选择偏好。配对设计更高效，因为消除了文本本身的方差。
  • 统计分析：收集数据后，使用配对t检验（paired t-test）或Wilcoxon符号秩检验来评估A和B的差异是否统计显著。p值小于0.05通常被认为是显著差异。
我的真实经历：有一次我们团队在对比两个TTS模型时，A模型的MOS比B模型高了0.15分，团队一致决定上线A模型。结果上线后，用户反馈数据却显示B模型的满意度更高。问题出在哪里？我们后来发现，虽然A模型平均分更高，但它偶尔会产生一些"机械感极强的句子"——而这些句子的概率虽然只有5%，却恰好出现在用户最常使用的几个句子上。平均分掩盖了"质量不稳定"这个更致命的问题。从此以后，我们在做TTS评测时不仅看平均值，还必看"最差情况"——最低分、方差、以及异常值的分布。

结果可视化

分析完数据只是第一步——如果你不能把结果以直观的方式展示给团队成员（或者更重要的——非技术背景的老板），那你的分析就很难转化为决策。这里推荐几种最适合TTS评测结果的可视化方式：

  • 箱线图（Box Plot）：展示MOS评分的分布情况——中位数、四分位数、异常值。通过箱线图可以一眼看出哪个模型不仅分数高，而且稳定性好。对于TTS评测，稳定性往往比平均值更重要。
  • 小提琴图（Violin Plot）：在箱线图的基础上叠加了概率密度分布，可以更清楚地看到评分集中在哪个区域。如果某个模型的评分分布是"双峰"（即一部分人给了很高的分，另一部分人给了很低的分），这可能意味着这个模型在某些文本上表现极好，在另一些文本上表现极差。
  • 雷达图（Radar Chart）：如果要对比多个维度（自然度、清晰度、韵律、音色相似度、稳定性），雷达图可以展示每个模型的多维"轮廓"。用雷达图很容易发现一个模型是"全面均衡"还是"偏科严重"。

Python的matplotlib和seaborn库都可以轻松绘制上述图表。如果你需要快速出图，pandas的.plot()方法加上seaborn的样式配置，几行代码就能生成生产级别的可视化报告。

最后，让我们用一个总览表来对比不同评估方法的适用场景和成本：

评估方法 衡量什么 实施难度 可靠性 适用阶段
人工MOS评测 整体听感自然度 高（需要招募听评者） 高（黄金标准） 最终验收
自动化MOS (UTMOS) 整体听感（近似） 低（pip安装即可） 中（有领域偏移风险） 快速迭代
说话人相似度 音色一致性 中（需要预训练模型） 中高（可量化） 语音克隆场景
A/B偏好测试 模型间相对优劣 中（需要设计实验） 高（直接反映偏好） 模型选型、上线决策
可懂度测试 内容是否可听清 高（需要转录任务） 高（客观可测量） 特殊场景（导航、通知）
重要提醒：统计显著性不等于实际显著性。p < 0.05只意味着"这个差异不太可能是随机误差造成的"，但并不意味着这个差异在实际产品中有意义。MOS相差0.05分可能是统计显著的（如果样本量足够大），但听感上没有人能分辨出来。在汇报结果时，一定要同时报告"效应量"（effect size），而不是只给p值。
