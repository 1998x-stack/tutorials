# 文本分块策略详解

## 1. 为什么分块很重要

分块（Chunking）是 RAG 索引阶段的关键步骤。分块质量直接影响：
- **检索精度**：块太大→噪声多；块太小→语义不完整
- **生成质量**：上下文窗口有限，需要精确命中相关信息
- **成本控制**：块越大→送入 LLM 的 token 越多→成本越高

## 2. 六大分块策略

### 2.1 固定大小分块（Fixed-Size Chunking）
最朴素的方法：按固定 token 数切分。

```python
# 固定大小分块示例
def fixed_chunk(text, chunk_size=512, overlap=50):
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunks.append(text[start:end])
        start = end - overlap  # 重叠窗口
    return chunks
```

- **优点**：实现简单，计算成本低
- **缺点**：不考虑语义边界，可能在句子中间切断
- **典型配置**：512-1024 tokens，50-100 tokens 重叠

### 2.2 语义分块（Semantic Chunking）
使用 LLM 或规则识别语义边界，在段落/章节/主题变化处切分。

- **实现方式**：计算相邻句子的 Embedding 相似度，在相似度骤降处切分
- **Agentic Chunking**：使用小型 LLM 判断切分边界（RAGArena 实验：Recall@10 提升 40%）

### 2.3 层级分块（Hierarchical Chunking）
构建多级索引树：Document → Section → Paragraph → Sentence。

- **KohakuRAG 方法**：自底向上聚合 Embedding，父节点 = 聚合子节点向量
- **检索时**：先匹配粗粒度节点，再下钻到细粒度块
- **优势**：保留文档结构，支持高精度引用

### 2.4 命题分块（Proposition Chunking）
将文本拆解为原子性事实命题。

```
原文："小明于2023年加入谷歌，担任高级工程师。"
命题1：小明于2023年加入谷歌。
命题2：小明担任高级工程师。
```

- **优势**：最大化检索精度
- **缺点**：索引成本高，丢失上下文连贯性

### 2.5 上下文增强分块（Contextual Retrieval）
在每个 Chunk 前添加 LLM 生成的文档摘要（Anthropic 方法）：

```
[文档摘要：这是一篇关于2024年AI发展趋势的技术博客，重点讨论大模型的应用...]
[具体内容] 在2024年，OpenAI发布了GPT-4o，支持多模态输入...
```

- **效果**：检索失败率降低高达 49%

### 2.6 Late Chunking（末段分块）
先从整个文档生成 Embedding，再切分（Jina Embeddings v3 支持）。

- 每个 Chunk 保留整个文档的上下文感知
- 适合需要全局理解的场景

## 4. 分块实战建议

1. **不要死扣一个参数**：不同文档类型可能需要不同策略
2. **先语义后固定**：优先尝试语义分块，不够好再调参
3. **重叠是保险**：50-100 token 的重叠可以覆盖边界信息丢失
4. **保留元数据**：每个 Chunk 记住它来自哪个文档、哪个章节
5. **关注更新**：SCAR（2026 年 6 月）通过自适应上下文扩展实现了 92.8% 召回率，使用更少的 Chunk

## 5. SCAR：语义连续性感知检索

2026 年 6 月最新研究（SCAR）引入了**结构连续性惩罚项**，在检索时自适应扩展相邻 Chunk：
- 使用查询相关性加权决定扩展多少
- 结构连续性惩罚防止无关 Chunk 混入
- 效果：92.8% 召回率，Chunk 数量减少 22.9%
