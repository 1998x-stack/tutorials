开篇：一个错误的集合选择让API慢了100倍

在一次代码评审中，我看到同事写了一个"性能优化"：他把一个需要频繁按名字查找的配置表从HashMap改成了Vec，理由是"HashMap哈希计算有开销"。结果服务上线后，API响应时间从15ms飙升到1.5秒——他用了Vec的线性搜索（O(n)）替代了HashMap的哈希查找（平均O(1)），而配置表有20万条记录。

这个故事告诉我们：选对集合类型，比一点点微优化重要一万倍。Rust标准库提供了几种最常用的集合类型，每种都有自己的性能特征和使用场景。

这一章我们来逐个讲解：Vec、HashMap、HashSet、VecDeque，以及它们的迭代方式。

Vec<T>：动态数组

Vec<T>是Rust中最常用的集合类型——一个在堆上分配的可增长数组。它就像C++的std::vector，但更安全、更易用。

创建Vec

// 方式一：vec!宏
let mut v1 = vec![1, 2, 3];

// 方式二：Vec::new()
let mut v2: Vec<i32> = Vec::new();

// 方式三：预分配容量
let mut v3: Vec<i32> = Vec::with_capacity(100);

with_capacity预分配指定容量的内存，如果你知道最终大小，它能避免多次重新分配。

增删改查

let mut v = vec![1, 2, 3];

// 添加
v.push(4);                     // 尾部追加
v.extend([5, 6]);              // 追加多个

// 访问
let third = &v[2];             // 索引访问（可能panic）
let third = v.get(2);          // 安全访问：返回 Option<&T>

// 修改
v[1] = 20;                     // 索引修改

// 删除
let last = v.pop();            // 尾部弹出
let first = v.remove(0);       // 删除指定位置（O(n)）

// 插入
v.insert(0, 0);                // 指定位置插入（O(n)）
索引越界：使用v[i]语法时如果索引超出长度，程序会panic!。如果你不确定索引是否有效，优先使用v.get(i)，它返回Option让你安全处理越界情况。

性能要点

Vec的内存布局是连续的——所有元素在堆上排成一行。这意味着：

  • 索引访问：O(1)，CPU缓存友好
  • 尾部增删：分摊O(1)，大多数零开销
  • 中间插入/删除：O(n)，需要移动后续元素
  • 扩容：容量不够时加倍，重新分配并复制所有元素

HashMap<K, V>：键值对字典

当你需要按键名快速查找值的时候——HashMap是首选。它在内部使用哈希表实现，平均情况下增删查都是O(1)。

创建与基本操作

use std::collections::HashMap;

// 创建
let mut scores = HashMap::new();
scores.insert(String::from("蓝色"), 10);
scores.insert(String::from("红色"), 50);

// 查询
let score = scores.get("蓝色");           // Option<&i32>
let score = scores["蓝色"];               // 直接索引（可能panic）

// 更新
scores.insert("蓝色".to_string(), 25);    // 覆盖旧值

// 存在则插入
scores.entry("绿色".to_string()).or_insert(100);

// 删除
scores.remove("红色");

// 遍历
for (key, value) in &scores {
    println!("{key}: {value}");
}

所有权影响

这是HashMap最需要关注的点。当你把值插入HashMap后，该值的所有权会转移到HashMap中。

let name = String::from("Alice");
let mut map = HashMap::new();
map.insert(name, 30);   // name的所有权被转移

// println!("{name}");  // 编译错误！name已被移动

如果你只想插入引用，可以插入&str而不是String——但要保证引用在HashMap生命周期内有效。

entry API：entry是HashMap最强大的API之一。它返回一个Entry枚举，让你可以链式操作：map.entry(key).or_insert(default)只在键不存在时插入；map.entry(key).and_modify(|v| *v += 1).or_insert(0)可以在一次查找中完成"存在则修改、不存在则插入"。

HashSet：无重复集合

HashSet<T>是HashMap<T, ()>的简便封装——它只关心键是否存在，不关心值。最适合用来做去重和成员关系判断。

use std::collections::HashSet;

let mut set = HashSet::new();
set.insert("apple");
set.insert("banana");
set.insert("apple");  // 重复，被忽略

println!("{}", set.len());            // 2

// 成员检查
if set.contains("apple") {
    println!("有苹果");
}

// 删除
set.remove("banana");

集合运算

HashSet支持数学上的集合运算：

let a: HashSet<_> = [1, 2, 3].into_iter().collect();
let b: HashSet<_> = [3, 4, 5].into_iter().collect();

// 并集
let union: HashSet<_> = a.union(&b).cloned().collect();   // {1,2,3,4,5}

// 交集
let intersection: HashSet<_> = a.intersection(&b).cloned().collect(); // {3}

// 差集
let diff: HashSet<_> = a.difference(&b).cloned().collect();  // {1,2}

// 对称差
let sym_diff: HashSet<_> = a.symmetric_difference(&b).cloned().collect(); // {1,2,4,5}
HashSet与Vec的去重性能对比：对100万个随机整数去重，HashSet比Vec去重（先排序后去重）快5-10倍。但是HashSet的元素需要实现Hash和Eq trait——幸运的是所有基本类型和标准库类型都已经实现了。

VecDeque：双端队列

当你需要同时在两端添加或删除元素时，VecDeque<T>（读作"vector-deck"）是比Vec更好的选择。它在内部使用环形缓冲区（ring buffer），两端操作都是O(1)。

use std::collections::VecDeque;

let mut deque: VecDeque<i32> = VecDeque::new();

// 两端添加
deque.push_back(3);    // [3]
deque.push_back(4);    // [3, 4]
deque.push_front(2);   // [2, 3, 4]
deque.push_front(1);   // [1, 2, 3, 4]

// 两端弹出
let front = deque.pop_front();  // Some(1)
let back = deque.pop_back();    // Some(4)

// 查看两端
let front = deque.front();      // Some(&2)
let back = deque.back();        // Some(&3)
什么时候用VecDeque而不是Vec？当你的使用模式是"先进先出"（FIFO）时——比如任务队列、广度优先搜索的待访问列表。Vec在头部插入/删除是O(n)，而VecDeque在两端都是O(1)的。但如果你只需要尾部操作，Vec总是更快（更简单的内存布局）。

四种集合的选型指南

需求 推荐集合 特点 适用场景
有顺序、按索引访问 Vec<T> 连续内存、CPU缓存友好 列表、堆栈、存储
按键快速查找 HashMap<K,V> 平均O(1)查找、无序 配置表、缓存、索引
去重与成员检查 HashSet<T> 自动去重、集合运算 白名单、黑名单、标记
双端高效操作 VecDeque<T> 两端O(1)、环形缓冲区 队列、滑动窗口、BFS

当然，Rust标准库中还有LinkedList、BTreeMap、BinaryHeap等更多集合类型，但以上四种是日常开发中使用频率最高的。在你能清晰回答"为什么LinkedList不适合做缓存"之前，优先用这四种就够了。

遍历与迭代：不可变/可变/所有权

Rust的集合遍历有三种模式，理解了它们才真正理解所有权的运作方式：

let v = vec![1, 2, 3];

// 1. 不可变引用遍历（最常用）
for item in &v {
    // item: &i32，只读
    println!("{item}");
}

// 2. 可变引用遍历
let mut v = vec![1, 2, 3];
for item in &mut v {
    *item += 1;
}

// 3. 所有权遍历（消耗原集合）
for item in v {
    // item: i32，v不再可用
    println!("{item}");
}
// println!("{v:?}"); 编译错误！v已被移动

对HashMap的遍历也是同样的三种模式。重点记住：所有权遍历会消耗原集合，之后就不能再用它了。

更灵活的是使用迭代器方法（第14章会详细讲解）：

let v = vec![1, 2, 3];
let doubled: Vec<i32> = v.iter().map(|x| x * 2).collect();
遍历时修改集合：Rust的借用规则严格禁止在不可变借用存在时修改集合。这意味着你不能在for item in &vec的循环体中vec.push()——编译器会拒绝。这在其他语言中是常见的bug来源（迭代时修改导致未定义行为），Rust直接帮你杜绝了。

本章小结

这一章我们认识了Rust标准库最核心的四种集合类型：Vec作为动态数组是主力容器，HashMap解决键值对查找问题，HashSet主打去重与成员检查，VecDeque擅长双端操作。每种集合都有自己的适用场景和性能取舍，选对了事半功倍。

但真正的代码世界并不是永远风平浪静的。接下来我们要面对一个重要的主题——错误处理。如何优雅地应对异常情况？怎么保证你的程序在出问题时不会突然崩溃？Rust的错误处理哲学与其他语言完全不同，它会把错误纳入类型系统，让你不得不面对它们。第10章，让我们直面Rust的错误处理。
