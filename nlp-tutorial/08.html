第8章：序列到序列模型 Seq2Seq

想象一下这个场景：你打开谷歌翻译，输入一句中文"今天天气真不错"，回车，瞬间出来一串英文——"The weather is really nice today"。这个过程看似简单，背后却藏着一个革命性的思想：把一个序列（中文）映射到另一个序列（英文）。这就是 Seq2Seq（Sequence-to-Sequence）模型要解决的核心问题。没有它，机器翻译就还是老一套的"查词典 + 调语序"的笨办法。

Seq2Seq 模型最早由 Sutskever 等人于 2014 年提出（还记得第6章讲过的"序列模型"吗？），从此改变了 NLP 的格局。它不仅撑起了机器翻译的半边天，还是对话系统、文本摘要、语音识别等众多任务的基石。今天这一章，我们就来拆解这个"序列魔术师"是怎么工作的。

💡 读前提示：如果你对 RNN / LSTM 的记忆有点模糊了，建议先回看第6章和第7章。Seq2Seq 的核心组件就是编码器和解码器两个 RNN，理解它们对今天的内容至关重要。

8.1 核心思想：Encoder-Decoder 架构

Seq2Seq 的架构说白了就两句话：一个 RNN 负责"读"，一个 RNN 负责"写"。"读"的那个叫 Encoder（编码器），"写"的那个叫 Decoder（解码器）。合在一起，就是 Encoder-Decoder 架构。

具体怎么工作的？我们一步步来看：

  1. 编码阶段：Encoder 逐词读取源句子，每个时间步输入一个词（通常是一个词向量），同时更新自己的隐藏状态 h_t。读完最后一个词后，Encoder 的最终隐藏状态 h_T 就浓缩了整个输入句子的信息。
  2. 上下文向量：这个 h_T 被当作 "上下文向量"（context vector）传给 Decoder，作为 Decoder 的初始隐藏状态。
  3. 解码阶段：Decoder 从 <SOS>（句子起始符）开始，逐词生成输出。每个时间步，Decoder 根据当前的隐藏状态和上一个输出的词，预测下一个词的概率分布。
  4. 终止条件：当 Decoder 输出 <EOS>（句子结束符）时，生成过程停止。

用公式来说，编码过程就是：

h_t = f(W · [h_{t-1}, x_t] + b)，其中 f 是激活函数（通常是 tanh 或 LSTM 的门控机制）。解码过程则是：P(y_t | y_<t, C) = softmax(W_s · h'_t + b_s)，其中 C 就是编码器传来的上下文向量。

这里有一个很关键的点：输入序列和输出序列的长度可以不同！这就是为什么 Seq2Seq 如此强大——中文句子有6个字，翻译成英文可能有8个词，这在传统方法里非常棘手，但在 Seq2Seq 框架下自然地被解决了。

8.2 Teacher Forcing：让模型"抄作业"的艺术

训练 Seq2Seq 时有个大麻烦：Decoder 在时间步 t 要预测 y_t，但它是用自己上一步的预测 y_{t-1} 作为输入的。如果上一步预测错了怎么办？那错误就会像滚雪球一样越滚越大。这就叫 "exposure bias"——模型在训练时从没见过自己的错误。

聪明的解决方法是 Teacher Forcing。思路很简单：训练的时候，不让 Decoder 用自己的预测作为下一步输入，而是直接把真实的目标词喂给它。就像老师手把手教学生写作业——"上一部你该写 'weather'，别管你猜的是什么，我给你正确答案，你接着往下写"。

训练时（Teacher Forcing）： y_{t} 的输入 = ground truth y_{t-1}
推理时（不用）： y_{t} 的输入 = 模型自己预测的 &hat;y_{t-1}

这样做的好处显而易见：训练更稳定，收敛更快。但代价是训练和推理之间存在 "gap"——训练时模型被保护得太好，推理时一遇到错误就不知所措。实践中，大家常用 Scheduled Sampling（计划采样）来改进：随着训练进行，逐渐降低 Teacher Forcing 的比例，让模型慢慢学会"自己走路"。

8.3 踩坑实录：当 Seq2Seq 撞上长句子

讲个真事。我读研的时候做机器翻译实验，Seq2Seq 模型在评测集上 BLEU 值（翻译质量指标，下一章会讲）能跑到 32 分，看着还不错。但上线后用户反馈：翻译超过 30 个词的句子基本不能用。特别是那种法律文本、新闻长句——"本合同由甲乙双方在平等自愿的基础上，经友好协商，就……达成如下协议……"——翻到后半段，Decoder 已经开始胡言乱语了。

我调了整整一周的参：加大 LSTM 的隐藏层维度、加深层数、修改学习率，什么都试了。最长也就把可用长度推到 40 个词。为什么就是不行？

后来仔细看了研究的论文才恍然大悟——问题出在 编码器的"信息瓶颈"上。Encoder 把整句话压缩成一个固定长度的向量 C，这个向量的容量是有限的。句子越长，每个词能分到的"信息预算"就越少。就像用一个 U 盘拷贝一部电影没问题，但要拷贝一整季《权游》，容量就不够了。

⚠️ 踩坑总结：Seq2Seq + 固定长度上下文向量 = 长句子灾难。我在实验中试过将隐藏层从 256 维升到 1024 维，效果提升不到 2 个 BLEU 点。根本解法不是增大容量，而是改变架构——也就是下一章要讲的 Attention 机制。各位如果项目里碰到长文本生成问题，先做长度分布分析，对超过一定长度的句子考虑分句或改用带 Attention 的模型。

8.4 信息瓶颈问题（The Bottleneck Problem）

刚才的故事引出了一个核心问题：Encoder 的最终隐藏状态 h_T 是整个模型的瓶颈。不管输入句子是 10 个词还是 100 个词，压缩到同一个维度，信息损失是必然的。

我们来做一个简单的信息论计算：假设隐藏状态是 256 维浮点数，每个浮点数 32 位，那么上下文向量总共只有 8192 位的信息容量。而一个 50 个词的句子，假设每个词用 300 维词向量表示，输入信息总量是 50 × 300 × 32 = 480000 位。压缩比接近 60:1！

换句话说，Encoder 必须把 60 倍的信息丢掉才能塞进那个小向量里。这怎么可能不出问题呢？

学界对这个问题的探索催生了两个重要方向：

  • 增大上下文容量：比如用双向 RNN（BiRNN）捕获前后文信息，或者加大隐藏维度——但治标不治本。
  • 引入 Attention（注意力机制）：不再把信息压缩成一个向量，而是让 Decoder 在每个解码步骤都能"回头看"源句子的所有位置，按需提取信息。这是下一章的详细内容。

8.5 代码实战：用 PyTorch 实现一个简单的 Seq2Seq

光说不练假把式。下面我们用 PyTorch 实现一个基于 GRU 的 Seq2Seq 模型，用于一个玩具级的任务：反转字符串（比如 "hello" → "olleh"）。虽然简单，但能清晰展示 Encoder-Decoder 的完整流程。

import torch
import torch.nn as nn
import torch.optim as optim
import random

class Encoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.gru = nn.GRU(embed_dim, hidden_dim, batch_first=True)

    def forward(self, x):
        # x: [batch_size, seq_len]
        embedded = self.embedding(x)          # [b, seq_len, embed_dim]
        output, hidden = self.gru(embedded)    # hidden: [1, b, hidden_dim]
        return hidden                          # 返回最后一个时间步的隐藏状态


class Decoder(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_dim):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.gru = nn.GRU(embed_dim, hidden_dim, batch_first=True)
        self.fc = nn.Linear(hidden_dim, vocab_size)

    def forward(self, x, hidden):
        # x: [batch_size, 1] (上一个预测的词)
        embedded = self.embedding(x)           # [b, 1, embed_dim]
        output, hidden = self.gru(embedded, hidden)
        prediction = self.fc(output.squeeze(1))  # [b, vocab_size]
        return prediction, hidden


class Seq2Seq(nn.Module):
    def __init__(self, encoder, decoder, device):
        super().__init__()
        self.encoder = encoder
        self.decoder = decoder
        self.device = device

    def forward(self, src, trg, teacher_forcing_ratio=0.5):
        batch_size, trg_len = trg.shape
        trg_vocab_size = self.decoder.fc.out_features

        # 存储所有时间步的输出
        outputs = torch.zeros(batch_size, trg_len, trg_vocab_size).to(self.device)

        # 编码阶段
        hidden = self.encoder(src)

        # 解码器第一个输入是 <SOS>
        decoder_input = trg[:, 0].unsqueeze(1)  # [b, 1]

        for t in range(1, trg_len):
            output, hidden = self.decoder(decoder_input, hidden)
            outputs[:, t, :] = output

            # Teacher Forcing 决策
            use_teacher_forcing = random.random() < teacher_forcing_ratio
            top1 = output.argmax(1)  # 模型预测的词
            decoder_input = trg[:, t].unsqueeze(1) if use_teacher_forcing else top1.unsqueeze(1)

        return outputs

看到关键部分了吗？在第38-40行，我们用 teacher_forcing_ratio 来控制"抄作业"的比例。训练时设成 0.5~1.0，推理时必须设成 0.0（完全用自己的预测）。这正是 Teacher Forcing 在代码中的体现。

训练脚本的大体流程如下：

# 假设已经准备好 vocab、dataloader
encoder = Encoder(vocab_size=50, embed_dim=32, hidden_dim=64)
decoder = Decoder(vocab_size=50, embed_dim=32, hidden_dim=64)
model = Seq2Seq(encoder, decoder, device).to(device)

criterion = nn.CrossEntropyLoss(ignore_index=0)  # 忽略 padding
optimizer = optim.Adam(model.parameters(), lr=0.001)

for epoch in range(20):
    for src, trg in dataloader:
        src, trg = src.to(device), trg.to(device)
        optimizer.zero_grad()
        output = model(src, trg, teacher_forcing_ratio=0.5)
        # output: [b, trg_len, vocab_size]
        loss = criterion(output[:, 1:].reshape(-1, 50), trg[:, 1:].reshape(-1))
        loss.backward()
        optimizer.step()
    print(f"Epoch {epoch+1}, Loss: {loss.item():.4f}")
🔎 注意：上面计算 loss 时我们忽略了 trg[:, 0]（即 <SOS> 标记），从第1个时间步开始预测。同时用 ignore_index=0 跳过了 padding 位置。这些细节在实际工程中经常被忽略，但出错了很难排查。

8.6 Seq2Seq 的应用场景

Seq2Seq 的"序列到序列"能力让它在很多任务中大放异彩：

任务输入输出说明
机器翻译源语言句子目标语言句子最经典的应用，Google 翻译早期核心
文本摘要长文档简短摘要Extractive（抽取式）→ Abstractive（生成式）
对话系统用户输入系统回复Seq2Seq + 合适的训练数据即可搭建聊天机器人
语音识别音频特征序列文本序列输入变长，输出也变长
代码生成自然语言描述代码片段Text-to-Code 任务

8.7 本章小结

这一章我们学到了什么？

  • Encoder-Decoder 架构是 Seq2Seq 的骨架，Encoder 负责"理解"，Decoder 负责"生成"。
  • Teacher Forcing 是训练 Decoder 的关键技巧——用真实值替代模型自己的预测，让训练更稳定，但要注意训练-推理之间的 gap。
  • 信息瓶颈问题是 Seq2Seq 的固有缺陷：固定长度的上下文向量无法承载长句子的全部信息，模型在处理长序列时性能急剧下降。
  • PyTorch 实现展示了完整的编码-解码流程，Teacher Forcing 的可控参数让你能灵活地在"抄作业"和"独立行走"之间切换。

但是等等——故事还没完。前面说的"信息瓶颈"问题是不是让你觉得 Seq2Seq 有点不够完美？确实如此。2015 年 Bahdanau 等人提出了一种优雅的解决方案：注意力机制（Attention）。它让 Decoder 不再死盯着那一个上下文向量不放，而是可以随时"回头看"源句子的每一个词，按需提取信息。这一突破直接催生了 Transformer，进而演变成了你每天都在用的 ChatGPT。
