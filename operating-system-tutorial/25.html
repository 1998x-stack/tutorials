第25章 动手写一个小OS

大家好，我是你们的讲师。前面24章讲了OS的各种理论和机制——进程、内存、文件系统、中断、调度——你学了一堆概念，但可能总觉得隔了一层纱。什么是"GDT"？什么又是"保护模式"？操作系统到底是怎么从按下电源键一路启动到显示桌面的？这些问题的终极答案只有一个——自己动手写一个。

这一章我们不依赖任何现成的内核，从零开始用汇编和C语言写一个微型操作系统。它不大——只有几百行代码——但它会完整经历真实OS的启动流程：BIOS引导、进入保护模式、实现内存分配器、设置中断、最后从Ring 0切换到Ring 3执行用户进程。当你亲眼看到自己写的OS在QEMU中打印出"Hello World"的那一刻，前面学的所有理论都会串联成一条线。

说句掏心窝的话，我学操作系统最有收获的一段时间，不是上课也不是看书，而是大三那年在X86模拟器上写了一个迷你内核。从实模式到保护模式踩了无数坑——GDT写错一个字节整个系统不断重启、IDT的中断门设置成任务门导致键盘中断不响应、切换到用户态时栈没切干净导致#GP异常——每解决一个问题，对OS的理解就深一层。如果你也想真正"懂"操作系统，没有比动手写一个更好的方式了。

一、用QEMU从零启动

计算机按下电源键后，CPU首先运行BIOS（Basic Input/Output System）。BIOS执行硬件自检后，寻找可启动设备——如果找到了我们的虚拟磁盘，它会将磁盘的第一个扇区（512字节，称为引导扇区或Boot Sector）加载到内存地址 0x7C00，然后跳转过去执行。你的OS的第一个脚印就从这里开始。

1. 编写Bootloader

引导扇区用16位实模式汇编编写（x86刚启动时处于16位实模式）。我们用BIOS的 int 0x10 中断在屏幕上打印字符：

; boot.asm — 16位实模式引导程序
[org 0x7c00]        ; 告诉汇编器代码将被加载到 0x7C00
[bits 16]

start:
    ; 设置段寄存器
    xor ax, ax
    mov ds, ax
    mov es, ax
    mov ss, ax
    mov sp, 0x7c00

    ; 用 BIOS int 0x10 打印字符串
    mov si, msg
print_loop:
    lodsb               ; 加载 SI 指向的字符到 AL，SI++
    or al, al           ; 检查是否为 0（字符串结束符）
    jz hang
    mov ah, 0x0e        ; BIOS 的 TTY 输出功能
    int 0x10
    jmp print_loop

hang:
    hlt
    jmp hang

msg: db "Hello from my OS!", 0

; 填充到 510 字节，以 0x55AA 结尾（引导扇区标识）
times 510 - ($ - $$) db 0
dw 0xaa55

2. 编译并运行

# 编译汇编代码
$ nasm -f bin boot.asm -o boot.bin

# 创建虚拟磁盘镜像（1.44MB 软盘镜像）
$ dd if=/dev/zero of=disk.img bs=512 count=2880
$ dd if=boot.bin of=disk.img bs=512 count=1 conv=notrunc

# 用 QEMU 启动
$ qemu-system-x86_64 -drive format=raw,file=disk.img -m 256M

# 你应该能看到 QEMU 窗口中显示出 "Hello from my OS!"

踩坑提醒：引导扇区必须恰好512字节，最后两个字节必须是 0x55 0xAA（小端序，所以代码中写 dw 0xaa55）。如果最后两个字节不对，BIOS会认为这个磁盘不可启动。另一个常见坑是 ORG 0x7C00 和段寄存器必须匹配——很多新手在实模式下的地址计算上栽跟头。

二、进入保护模式

实模式有很多限制：只能访问1MB内存、没有内存保护、只能使用16位寄存器和寻址。现代OS必须运行在32位或64位的保护模式（Protected Mode）或长模式（Long Mode）下。从实模式进入保护模式的步骤是固定的三步曲：设置GDT → 开启A20地址线 → 设置CR0保护模式位。

1. 设置GDT（全局描述符表）

保护模式下，内存段不再由段寄存器直接决定，而是通过GDT中的段描述符来定义。每个段描述符包含了段的基地址、长度和访问权限。最少需要两个描述符：代码段和数据段。段描述符的8字节结构如下：

字节偏移7-4位3位2位1位0位含义
5 (flags+limit_high)G (粒度)D/B (默认操作大小)L (64位码)AVL (软件可用)段长度[19:16]
4 (access)P (段存在)DPL (特权级)S (系统/代码数据)E/C/D (类型扩展)W/R/A (读写/已访问)
2-3基地址[23:16] / 基地址[31:24]
0-1段长度[15:0] / 基地址[15:0]

看起来复杂，但实践中只需要套用固定模板——上面代码中的数值就是标准的平坦内存模型（Flat Memory Model）模板。

; GDT 定义（32位保护模式）
gdt_start:
    ; 第一个描述符必须是空描述符（GDT 规范要求）
    dd 0x0
    dd 0x0

; 代码段描述符
gdt_code:
    ; base=0x0, limit=0xfffff, 32位代码段, Ring 0
    dw 0xffff        ; limit low
    dw 0x0           ; base low
    db 0x0           ; base mid
    db 10011010b     ; access byte: present, Ring 0, code, executable, readable
    db 11001111b     ; flags+limit: 4K granularity, 32-bit
    db 0x0           ; base high

; 数据段描述符
gdt_data:
    ; base=0x0, limit=0xfffff, 32位数据段, Ring 0
    dw 0xffff        ; limit low
    dw 0x0           ; base low
    db 0x0           ; base mid
    db 10010010b     ; access byte: present, Ring 0, data, writable
    db 11001111b     ; flags+limit
    db 0x0           ; base high

gdt_end:

; GDT 描述符（给 lgdt 指令用）
gdt_descriptor:
    dw gdt_end - gdt_start - 1   ; GDT 大小 - 1
    dd gdt_start                 ; GDT 基地址

; 段选择子常量
CODE_SEG equ gdt_code - gdt_start
DATA_SEG equ gdt_data - gdt_start

2. 开启A20地址线

A20地址线是历史遗留问题——早期的80286 CPU有24条地址线，但为了兼容8086的20位地址线回绕特性，A20地址线默认被屏蔽。进入保护模式前必须开启它，否则无法访问1MB以上的内存。最常见的开启方式是通过键盘控制器（8042）的端口：

; 开启 A20 地址线（通过键盘控制器端口）
enable_a20:
    call wait_kbc        ; 等待键盘控制器空闲
    mov al, 0xd1         ; 发送写输出端口命令
    out 0x64, al
    call wait_kbc
    mov al, 0xdf         ; 设置 A20 使能位
    out 0x60, al
    call wait_kbc
    ret

3. 切换保护模式

; 加载 GDT
lgdt [gdt_descriptor]

; 设置 CR0 寄存器的保护模式位（PE 位）
mov eax, cr0
or eax, 0x1          ; 设置 PE (Protection Enable) 位
mov cr0, eax

; 远跳转刷新流水线，加载代码段选择子
jmp CODE_SEG:protected_mode

[bits 32]
protected_mode:
    ; 设置数据段寄存器
    mov ax, DATA_SEG
    mov ds, ax
    mov es, ax
    mov ss, ax
    mov esp, 0x90000

    ; 现在我们在32位保护模式下！
    ; 可以寻址4GB内存，有硬件内存保护

关键认知：保护模式的切换是不可逆的——一旦设置了CR0的PE位，你无法在不重启的情况下回到实模式。GDT中的每个bit都有精确的硬件语义：access byte的每个位决定段的访问权限，granularity位决定limit的单位是字节还是4KB。一个bit写错，CPU就会产生#GP异常。我当年调试GDT时，就是因为代码段描述符的 db 11001111b 写成了 11001110b（没设置32位位标志），CPU把32位代码当16位解码，结果每条指令都少读了两个字节，蹦出来的字符完全不可理喻。

三、实现物理内存分配器

进入保护模式后，我们需要知道系统有多少物理内存可用，然后实现一个简单的分配器来管理它。早期的内核通过BIOS的 int 0x15, E820h 功能调用获取物理内存布局。

1. 用E820获取内存布局

在进入保护模式之前（实模式下），调用BIOS中断获取系统内存映射：

; 实模式下获取 E820 内存映射
detect_memory:
    mov di, 0x8000           ; 存储内存映射信息的目标地址
    xor ebx, ebx             ; 第一次调用 ebx=0
    mov edx, 0x534D4150      ; 'SMAP' 签名
    mov eax, 0xe820
    mov ecx, 24              ; 每个条目24字节

.loop:
    int 0x15                 ; 调用 BIOS E820
    jc .error                ; CF=1 表示出错
    add di, 24               ; 移动指针到下一个条目
    inc dword [mmap_entries] ; 条目计数+1
    test ebx, ebx            ; ebx=0 表示枚举完成
    jnz .loop
    ret

.error:
    ; 处理错误：至少可以使用低端1MB内存
    mov dword [mmap_entries], 1
    ret

E820返回的每个条目包含：起始地址（8字节）、长度（8字节）、类型（4字节）。类型1表示可用内存，类型2表示保留区域，类型3表示ACPI回收内存，类型4表示ACPI NVS内存。我们只使用类型1的可用内存区域。

2. 位图分配器

最简单的物理页分配器是基于位图（bitmap）的。我们把物理内存分成4KB大小的页面（page），每个页面在位图中占1个bit：0表示空闲，1表示已分配。

// mem.c — 位图物理内存分配器
#define PAGE_SIZE 4096
#define MAX_PAGES  (1024 * 1024)  // 最多管理4GB内存

static uint8_t bitmap[MAX_PAGES / 8];
static uint32_t total_pages;
static uint32_t free_pages;

void mem_init(uint64_t mmap_addr, uint32_t entries) {
    // 从 E820 结果计算总页面数
    total_pages = 0;
    for (int i = 0; i < entries; i++) {
        mmap_entry_t *e = (mmap_entry_t *)(mmap_addr + i * 24);
        if (e->type == 1) {  // 可用内存
            total_pages += e->length / PAGE_SIZE;
        }
    }
    free_pages = total_pages;
    // 将所有位初始化为0（空闲）
    memset(bitmap, 0, sizeof(bitmap));
}

void *page_alloc(void) {
    // 扫描 bitmap，找到第一个空闲页
    for (uint32_t i = 0; i < total_pages; i++) {
        if (!(bitmap[i / 8] & (1 << (i % 8)))) {
            bitmap[i / 8] |= (1 << (i % 8));
            free_pages--;
            return (void *)(i * PAGE_SIZE);
        }
    }
    return NULL;  // 没有可用内存
}

void page_free(void *addr) {
    uint32_t page = (uint32_t)addr / PAGE_SIZE;
    bitmap[page / 8] &= ~(1 << (page % 8));
    free_pages++;
}

设计理念：位图分配器虽然简单，搜索效率是O(n)，但它的优点是实现直观且内存开销极低——管理4GB内存只需要128KB的位图（4GB/4KB/8）。更高级的分配器（如buddy system或slab分配器）会在位图的基础上优化碎片和分配速度。对于我们自己写的小OS，位图分配器够用就好。

四、设置中断

没有中断的操作系统就像没有耳朵的人——无法感知外部世界的任何事件。键盘敲击、定时器滴答、磁盘完成I/O——这些都需要中断来通知CPU。在保护模式下，中断由IDT（Interrupt Descriptor Table）管理。

1. 编写IDT

IDT的每个条目（门描述符）大小为8字节，可以配置为中断门、陷阱门或任务门。中断门会自动清除IF位（禁止嵌套中断），陷阱门不会。大多数设备中断使用中断门。

// idt.c — 中断描述符表设置
struct idt_entry {
    uint16_t base_low;     // 中断处理函数低16位
    uint16_t sel;          // 代码段选择子
    uint8_t  always0;      // 必须为0
    uint8_t  flags;        // 类型和属性
    uint16_t base_high;    // 中断处理函数高16位
} __attribute__((packed));

struct idt_ptr {
    uint16_t limit;
    uint32_t base;
} __attribute__((packed));

struct idt_entry idt[256];
struct idt_ptr idtp;

void idt_set_gate(uint8_t num, uint32_t handler, uint16_t sel, uint8_t flags) {
    idt[num].base_low = handler & 0xFFFF;
    idt[num].base_high = (handler >> 16) & 0xFFFF;
    idt[num].sel = sel;
    idt[num].always0 = 0;
    idt[num].flags = flags;  // 如 0x8E 表示 Ring 0 中断门
}

void idt_install(void) {
    idtp.limit = sizeof(idt) - 1;
    idtp.base = (uint32_t)&idt;
    memset(&idt, 0, sizeof(idt));

    // 设置每个中断的处理函数
    idt_set_gate(0, (uint32_t)isr0, 0x08, 0x8E);
    idt_set_gate(1, (uint32_t)isr1, 0x08, 0x8E);
    // ... 一共设置256个
    idt_set_gate(32, (uint32_t)irq0, 0x08, 0x8E);  // 定时器
    idt_set_gate(33, (uint32_t)irq1, 0x08, 0x8E);  // 键盘

    // 加载 IDT
    __asm__ __volatile__("lidt %0" : : "m"(idtp));

    // 启用中断
    __asm__ __volatile__("sti");
}

2. 键盘中断处理

键盘中断是IRQ1（中断向量33）。当按键按下时，键盘控制器向CPU发送中断，CPU读取键盘端口 0x60 获得扫描码：

// 键盘中断处理函数
void keyboard_handler(void) {
    uint8_t scancode = inb(0x60);  // 从键盘端口读取扫描码

    if (scancode & 0x80) {
        // 按键释放（bit7=1），可以忽略
    } else {
        // 按键按下，在这里可以将扫描码转换为 ASCII
        char c = scancode_to_ascii(scancode);
        terminal_write(&c, 1);
    }

    // 发送 EOI（End of Interrupt）到 PIC
    outb(0x20, 0x20);
}

常见错误：中断处理函数执行时间必须尽可能短，因为CPU在中断处理期间会阻塞其他中断。如果需要做复杂处理，应该只做最必要的操作（如把扫描码存入缓冲区），然后触发软中断让下半部分（bottom half）去处理。另一个容易忽略的点：PIC（可编程中断控制器）需要在中断处理结束时发送EOI，否则不会接收后续中断——这会导致键盘只响应一次就彻底"哑"了。

五、跳到用户态

最后也是最激动人心的一步——从Ring 0（内核态）切换到Ring 3（用户态）。这标志着你的小OS具备了真正的多权限级保护能力。用户态的代码不能直接执行特权指令、不能访问内核内存，所有硬件操作必须通过系统调用。

1. 准备用户态环境

切换到Ring 3需要使用 iret 指令（中断返回），因为Intel的CPU架构中，唯一从高特权级到低特权级的合法路径就是中断返回。我们通过构造一个"伪造"的中断返回帧来实现切换：

// 切换到用户态（Ring 3）
void switch_to_user(void) {
    // 设置用户态代码段选择子（Ring 3）
    // 假设 GDT 中索引3是用户态代码段，索引4是用户态数据段
    #define USER_CS 0x18 | 0x3   // 索引3 + Ring 3
    #define USER_DS 0x20 | 0x3   // 索引4 + Ring 3

    // 构造中断返回帧，模拟一个从 Ring 3 到 Ring 0 调用后返回
    struct {
        uint32_t ss;         // 用户态栈段
        uint32_t esp;        // 用户态栈指针
        uint32_t eflags;     // 标志寄存器
        uint32_t cs;         // 用户态代码段
        uint32_t eip;        // 用户态入口点
    } __attribute__((packed)) frame;

    frame.ss  = USER_DS;
    frame.esp = 0xBFFFF0;   // 用户栈地址
    frame.eflags = 0x202;   // IF=1 (启用中断), IOPL=0
    frame.cs  = USER_CS;
    frame.eip = (uint32_t)user_process;  // 用户进程入口

    // 用 iret 指令切换到 Ring 3
    __asm__ __volatile__(
        "mov %0, %%eax\n"
        "pushl (%%eax)\n"    // ss
        "pushl 4(%%eax)\n"   // esp
        "pushl 8(%%eax)\n"   // eflags
        "pushl 12(%%eax)\n"  // cs
        "pushl 16(%%eax)\n"  // eip
        "iret\n"
        :: "r"(&frame)
    );
    // 控制流不会回到这里
}

2. 系统调用实现

用户进程无法直接调用中断或访问硬件。我们实现一个简单的 sys_write 系统调用——用户进程通过 int 0x80 软中断请求内核服务：

// 用户进程代码（在 Ring 3 执行）
void user_process(void) {
    // 用 int 0x80 调用 sys_write 系统调用
    // eax = 系统调用号 (1 = sys_write)
    // ebx = 文件描述符 (1 = stdout)
    // ecx = 字符串地址
    // edx = 字符串长度
    __asm__ __volatile__(
        "mov $1, %%eax\n"     // sys_write
        "mov $1, %%ebx\n"     // fd = stdout
        "mov $msg, %%ecx\n"   // buf
        "mov $14, %%edx\n"    // count
        "int $0x80\n"
        "1: jmp 1b\n"         // 无限循环
        :::
    );
    // 消息数据
    __asm__ __volatile__(".section .rodata\n"
                         "msg: .ascii \"Hello from Ring 3!\\n\"\n"
                         ".text\n");
}

// 内核中的系统调用分发器
void syscall_handler(struct regs_t *r) {
    switch (r->eax) {
        case 1:  // sys_write
            // r->ebx = fd, r->ecx = buf, r->edx = count
            terminal_write((char *)r->ecx, r->edx);
            break;
        case 2:  // sys_exit
            // 终止当前进程
            switch_to_idle();
            break;
        default:
            break;
    }
}

核心洞见：从 Ring 0 到 Ring 3 的切换是操作系统安全模型的基石。用户进程无法直接执行 cli、hlt 等特权指令，无法访问内核地址空间，所有硬件操作必须经过系统调用——而系统调用的入口由内核完全控制。这个看似简单的权限分离，正是现代操作系统"用户程序崩溃不影响整个系统"的根本原因。

写完这五个步骤，你的微型OS就具备了真实OS最核心的骨架：从BIOS引导、到保护模式、到物理内存管理、到中断处理、最后到用户态进程。虽然它没有文件系统、没有网络栈、没有进程调度（目前只有一个进程），但它已经完成了一个操作系统最本质的使命——管理和保护硬件资源，为应用程序提供执行环境。

如果你想继续扩展它，接下来可以做的事情包括：实现一个简易的FAT文件系统读取器、添加多任务调度器让多个用户进程轮流运行、或者扩展系统调用接口。每一件都是对前面章节所学知识的实践落地。
