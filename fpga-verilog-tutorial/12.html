第12章 流水线设计技术

想象你正在设计一个 FIR 滤波器——128 阶乘累加，16 位输入，16 个并行乘法器。你在仿真中看到漂亮的滤波波形，但综合后发现 fmax 只有 85 MHz，而你的系统需要至少 150 MHz。这是 FPGA 设计中最常见的场景：组合逻辑路径太长。解决办法就是本章的主角——流水线（Pipeline）技术。

12.1 流水线的本质

流水线的核心思想简单到令人惊讶：在组合逻辑路径中插入寄存器，将一条长的组合路径分割成若干段较短的路径。为什么要这么做？因为同步数字电路的最高时钟频率受限于最长的组合逻辑延迟：

fmax = 1 / (Tclk-q + Tlogic + Tsetup)

其中 Tclk-q 是触发器的时钟到输出延迟（通常 0.2-0.5 ns），Tsetup 是触发器的建立时间（通常 0.1-0.3 ns），而 Tlogic 是组合逻辑的传播延迟——这是变量最大的项。一个 32 位加法器级联了十几级 LUT，Tlogic 可能达到 5-10 ns。通过在路径中插入寄存器，你将 Tlogic 从"从头到尾的总和"变成了"最长分段的值"——原来的 10 ns 路径被切成两段 5 ns 路径，fmax 从 100 MHz 提升到 200 MHz。

流水线的代价是延迟（Latency）：插入 N 级流水线后，第一个结果需要等待 N 个时钟周期才能输出。但在数据流处理（音频、视频、网络包）中，延迟增加几到几十个时钟周期无关紧要——你关心的是吞吐量（Throughput），即每秒钟能处理多少个输入数据。流水线让每级处理在时间上重叠，使吞吐量几乎与级数成正比增长。

流水线还有一个常被忽略的收益：降低了每个时钟周期内组合逻辑的峰值电流。翻转率被分散到多个时钟沿，减少了同时开关的晶体管数量，从而降低了电源噪声（di/dt）。

流水线 vs 并行：两条路都可以提高吞吐量。并行是复制 N 份处理单元同时工作（面积 ×N，延迟不变），流水线是把处理分解成 N 段顺序执行（面积略增，延迟 +N 周期）。实际工程中经常混合使用——比如在 AES 加密引擎中，轮内步骤用流水线，多路数据块用并行。

12.2 如何选择流水线深度

流水线不是越深越好。它的选取是一个收益递减问题：当组合路径已经被切得很短时（比如只剩一级 LUT + 布线），继续插入寄存器对 fmax 的提升趋近于零，因为瓶颈变成了 Tclk-q + Tsetup + 布线延迟——触发器的自身开销。

工程中确定流水线深度的常用方法是：

  • 综合后打开时序报告，找到最差路径（Worst Path）。
  • 查看这条路径的Logic Delay（逻辑延迟）和Net Delay（布线延迟）。
  • 如果 Logic Delay 占总延迟的 60% 以上，说明有充足的空间通过流水线来提升频率。
  • 在组合逻辑最密集的位置（多个 LUT 级联后、通过 MUX 后的汇聚点）插入寄存器。
  • 每次插入后重新综合，检查新的最差路径——通常 3-5 级流水线就能解决 80% 的时序问题。

以下是非流水线 vs 2 级 vs 4 级流水线的典型对比：

指标非流水线2 级流水线4 级流水线
fmax（典型值）125 MHz210 MHz380 MHz
吞吐量（M results/s）125210380
Latency（时钟周期）124
LUT 使用量780790810
FF 使用量64128248

可以看到，从非流水线到 4 级流水线，频率提升了 3 倍以上，而 LUT 几乎不变（组合逻辑没变），FF 增加了不到 200 个。在 Artix-7 上，一个 16×16 乘法器大约消耗 270 个 LUT + 300 个 FF——这些 FF 在 FPGA 中几乎"免费"（每个 LUT 后面都附带一个 FF）。

需要注意的反直觉点：有时候插入流水线寄存器后 fmax 反而下降了。原因通常是寄存器插入导致了布线拥塞——寄存器之间大量的数据路径迫使布局工具把逻辑放在更远的位置，增加了 Net Delay。这种情况下可以尝试减少流水线级数，或者使用 /* synthesis keep */ 约束让工具保持寄存器的物理位置。这属于进阶的时序收敛技巧。

12.3 流水线冒险（Hazard）

当你把流水线用在反馈路径或条件分支中时，就不能简单地"插入寄存器"了——你需要考虑三种冒险：

数据冒险（Data Hazard）：当流水线的后一级需要用到前一级的计算结果时，后一级要等待前一级完成。这在 FIR 滤波器中不会出现（数据是前向流动的），但在递归结构（如 IIR 滤波器）或处理器中会出现。解决方案是前推（Forwarding / Bypassing）——将前一级的组合输出直接连到后一级的输入，而不是等寄存器写回。硬件实现就是一个多路选择器（Forwarding MUX），根据源地址和目标地址是否匹配来选择数据来源。

控制冒险（Control Hazard）：当流水线遇到条件选择时（如状态机转移、MUX 选择下一组数据源），你不知道下一个时钟周期应该处理哪条路径。在通用 CPU 中，这需要分支预测（Branch Prediction）和流水线冲刷（Flush）——预测错误时清除已经进入流水线的无效指令。在 FPGA 数据路径中，我们通常用推测执行加结果选择来处理：同时计算两条路径，等条件就绪后再用 MUX 选择正确的结果。

结构冒险（Structural Hazard）：两个流水线级同时需要访问同一个硬件资源（如同一个 BRAM 端口、同一个 DSP slice）。解决方法是：使用双端口 BRAM、资源复制（Duplication），或在流水线中插入停顿（Stall）。在 FPGA 中，结构冒险比 ASIC 少——因为 FPGA 有丰富的专用硬件块（每个 Block RAM 都是真双端口），很少出现端口竞争。

12.4 实战案例：4 级流水线乘法器

让我们用 Verilog 实现一个 4 级流水线的 16×16 位乘法器。该设计将部分积生成、Wallace 树压缩、最终加法分配到 4 个流水级中，目标是在 Artix-7（速度等级 -1）上达到 380 MHz 以上：

// 4-stage pipelined 16×16 unsigned multiplier
// Stage 1: Input registers + partial product generation
// Stage 2-3: Partial product reduction (carry-save adder tree)
// Stage 4: Final carry-propagate adder + output registers
// Target: 380 MHz on Artix-7 (speed grade -1)
// Latency: 4 cycles, Throughput: 1 result/cycle

module multiplier_16x16_pipe (
    input  wire         clk,
    input  wire         rst_n,
    input  wire         valid_i,
    input  wire [15:0]  a_i,
    input  wire [15:0]  b_i,
    output reg  [31:0]  result_o,
    output reg          valid_o
);

    // ---- Stage 1 registers ----
    reg [15:0] a_r1, b_r1;
    reg        v_r1;
    reg [31:0] pp [0:15];   // 16 partial products

    integer i;
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            a_r1 <= 16'd0; b_r1 <= 16'd0; v_r1 <= 1'b0;
        end else begin
            a_r1 <= a_i; b_r1 <= b_i; v_r1 <= valid_i;
            for (i = 0; i < 16; i = i + 1)
                pp[i] <= (b_i[i]) ? {16'd0, a_i} << i : 32'd0;
        end
    end

    // ---- Stage 2 registers: first compression stage ----
    reg [31:0] sum_s2, carry_s2;
    reg        v_s2;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            sum_s2 <= 32'd0; carry_s2 <= 32'd0; v_s2 <= 1'b0;
        end else begin
            v_s2 <= v_r1;
            // Wallace tree: compress 16 partial products to sum + carry
            // Simplified — a real implementation would use 3:2 compressors
            sum_s2   <= pp[0] ^ pp[1] ^ pp[2] ^ pp[3]
                      ^ pp[4] ^ pp[5] ^ pp[6] ^ pp[7];
            carry_s2 <= ((pp[0] & pp[1]) | (pp[0] & pp[2]) | (pp[1] & pp[2]))
                      ^ ((pp[3] & pp[4]) | (pp[3] & pp[5]) | (pp[4] & pp[5]));
        end
    end

    // ---- Stage 3 registers: remaining compression ----
    reg [31:0] sum_s3, carry_s3;
    reg        v_s3;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            sum_s3 <= 32'd0; carry_s3 <= 32'd0; v_s3 <= 1'b0;
        end else begin
            v_s3 <= v_s2;
            // Continue Wallace compression (pp[8:15])
            sum_s3   <= sum_s2 ^ carry_s2
                      ^ pp[8] ^ pp[9] ^ pp[10]
                      ^ pp[11] ^ pp[12] ^ pp[13] ^ pp[14];
            carry_s3 <= (sum_s2 & carry_s2)
                      | (carry_s2 & pp[8])
                      | (pp[8] & pp[9]);
        end
    end

    // ---- Stage 4 registers: final carry-propagate adder ----
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            result_o <= 32'd0; valid_o <= 1'b0;
        end else begin
            valid_o  <= v_s3;
            result_o <= sum_s3 + carry_s3 + pp[15];
        end
    end

endmodule

重点观察：每一级的 valid 信号（v_r1 → v_s2 → v_s3 → valid_o）都与数据路径保持相同的流水线深度。这是避免"下游采到无效数据"的关键。

另一个案例——流水线 AES-128 数据路径：AES-128 的 10 轮加密可以全部展开为 10 级流水线。每轮包含 SubBytes（S-Box 查表，16 个 S-Box 并行）、ShiftRows（硬连线重排）、MixColumns（伽罗瓦域乘法）、AddRoundKey（异或）。每级流水线插入一个寄存器，配合提前生成的轮密钥（Key Schedule），整个引擎可以达到每周期加密一个 128 位块的吞吐量。在 Artix-7 上实现大约消耗 12,000 LUT + 8,000 FF，fmax 约 180 MHz，加密吞吐量 ~2.3 GB/s。

12.5 踩坑：忘了延迟 data-valid 信号

有一次我设计一个多级 FIR 滤波器，数据路径经过 5 级流水线寄存器。仿真波形完美——输入数据经过 5 个时钟周期后正确输出。下板测试，发现输出在运行过程中会间歇性地出现异常值。

我用 Vivado 的 ILA（Integrated Logic Analyzer）抓取内部信号，发现一个奇怪的模式：输出错误总是发生在输入数据源切换数据速率的时候。仔细追踪后发现——我的 data_path 经过了 5 级流水线寄存器，而 data_valid 信号只打了 2 拍就送到了下游模块。于是当下游模块看到 data_valid 拉高时，它去读取 result_o——但 result_o 上还是第 2 个周期的中间结果，真正的有效数据还在流水线的第三级里！下游模块把未完成的计算结果当作有效数据拿去处理了。

修复起来很简单：把 data_valid 也经过 5 级流水线寄存器，和主数据路径完全对齐。但我花了整整半天才发现这个问题——因为仿真中我是连续输入数据的（没有间隙），data_valid 一直为高，所以仿真没有暴露这个 Bug。只有当下游模块检测 data_valid 的跳变沿来判断新数据时，拼凑的延迟才会出问题。

教训：控制信号（valid、ready、last、使能）必须和主数据路径经过完全相同的流水线深度。最简单的做法是：把有效信号和主数据放在同一个 always 块中一起寄存——这样它们的延迟永远不会不同步。

理解了流水线，你就掌握了 FPGA 高性能设计的核心技巧。流水线让你的设计跑得更快，但很快你会发现：一个系统中往往有多个模块工作在不同的时钟频率。当数据从一个时钟域跨越到另一个时钟域时，流水线寄存器就不够用了——你需要专门的跨时钟域同步技术。这正是下一章的主题。
