引言：一致性不是一个开关

很多刚接触分布式系统的开发者习惯问："你们的数据库是强一致还是最终一致？"仿佛一致性是一个二元的开关——要么全有，要么全无。但现实远比这复杂。一致性不是"有或没有"，而是一个从极强到极弱的连续谱系，每种模型都有不同的约束、不同的性能特征和不同的适用场景。

理解这个谱系是分布式系统架构师的核心能力。它决定了你是选择 Spanner、DynamoDB 还是 Cassandra，决定了你的应用是否能正确处理并发写入，也决定了你在出现故障时会看到怎样的行为。

一致性谱系

从最强到最弱，我们来梳理一下分布式系统中常见的一致性模型：

模型 强度 核心约束 典型系统
线性一致性 (Linearizability) 最强 操作仿佛在某个时间点原子地发生，实时顺序可见 Spanner, ZooKeeper (Zab)
顺序一致性 (Sequential Consistency) 强 每个进程内部顺序保持，全局顺序一致但不要求实时 共享内存多处理器
因果一致性 (Causal Consistency) 中等 有因果关系的操作按顺序呈现，并发操作可乱序 DynamoDB (会话保证), CRDT
最终一致性 (Eventual Consistency) 弱 如果没有新写入，所有副本最终一致 Cassandra, DNS, CDN
关键区分：线性一致性和顺序一致性的核心区别在于实时性。线性一致性要求一旦操作 A 完成，所有后续操作（在实时时间上）都必须看到 A 的结果。顺序一致性只要求存在一个全局顺序，但不要求这个顺序反映实时时间。换句话说，线性一致性 = 顺序一致性 + 实时约束。

线性一致性 (Linearizability)

线性一致性是最强的一致性模型，也是最昂贵的。它的正式定义是：系统中存在一个全局的顺序，使得每个操作看起来是在其调用和返回之间的某个时刻原子地执行的，并且这个顺序与实时时钟一致。

让我们用一段伪代码来理解线性一致性的"执行门槛"：

// 系统初始值: x = 0

// 时间轴上的操作序列
// 客户端A:    写 x = 1 (开始 10ms, 返回 20ms)
// 客户端B:    读 x (开始 15ms, 返回 25ms)
// 客户端C:    读 x (开始 30ms, 返回 35ms)

// 线性一致性要求：
// - 如果写操作在 10-20ms 间生效，B (15-25ms) 可能读到 0 或 1
// - 但 C (30-35ms) 在写返回之后，必须读到 1
// - 不能出现 C 读到 0 的情况

这个例子揭示了线性一致性的核心：一旦写操作返回，所有后续的读操作都必须看到这个写入。这就是线性一致性被称为"最强模型"的原因——它对用户的承诺非常明确，但也因此对性能有显著影响。

单机等价性：线性一致性有时也被称为"原子一致性"（Atomic Consistency），因为它提供的语义等同于单机系统——仿佛只有一个线程按顺序执行所有操作。这是为什么线性一致性对开发者最友好，但对系统实现最难。

因果一致性 (Causal Consistency)

因果一致性在"足够强"和"足够快"之间找到了一个工程上的甜蜜点。它的核心思想是：只有存在因果关系的操作才需要排序，没有因果关系的并发操作可以以任意顺序呈现。

让我们看一个社交媒体场景的例子：

  1. Alice 发了一条推文："今天天气真好！"
  2. Bob 回复 Alice："同意！"（这条回复因果依赖于 Alice 的推文）
  3. Carol 在另一个城市发了一条独立的推文："午饭吃了拉面"

因果一致性要求：如果某个用户同时看到了 Bob 的回复和 Alice 的推文，那么 Alice 的推文必须在 Bob 的回复之前出现（因为有因果关系）。但是 Carol 的推文可以出现在 Alice 推文之前或之后（因为没有因果关系）。

实现因果一致性可以利用我们在第 2 章学到的向量时钟。每个操作携带向量时钟，接收端根据向量时钟判断依赖关系，确保因果依赖的操作按序执行。

最终一致性与 CRDT

什么是最终一致性

最终一致性是最弱也最常用的一致性模型。它保证：如果系统停止接收写入，经过足够长的时间后，所有副本最终会收敛到相同的值。但"最终"是一个不确定的时间——可能是几毫秒，也可能是几秒。

最终一致性本身不解决冲突问题，因此需要额外机制来合并并发写入。CRDT（Conflict-free Replicated Data Types，无冲突复制数据类型）就是为此而生的一种优雅解决方案。

CRDT 示例

CRDT 的核心思想是：设计一种数学上确保的数据结构，使得并发更新可以无冲突地合并。下面看几个基本的 CRDT：

// G-Counter (Grow-Only Counter) — 仅增计数器
// 每个副本维护自己的计数，合并时取各维度的最大值

struct GCounter {
    int[] values;  // 长度 = 副本数

    void increment(int replica_id) {
        values[replica_id]++;  // 仅在自己维度增加
    }

    int query() {
        return sum(values);    // 求和所有维度
    }

    void merge(GCounter other) {
        for i in 0..len(values):
            values[i] = max(values[i], other.values[i]);
    }
}

// PN-Counter (Positive-Negative Counter) — 支持增减的计数器
// 使用两个 G-Counter：一个计增加，一个计减少

struct PNCounter {
    GCounter positive;  // 增加次数
    GCounter negative;  // 减少次数

    void increment(int replica_id) {
        positive.increment(replica_id);
    }

    void decrement(int replica_id) {
        negative.increment(replica_id);
    }

    int query() {
        return positive.query() - negative.query();
    }
}

// OR-Set (Observed-Remove Set) — 支持增删的集合
// 为每个元素附加唯一的 tag，删除时只删除观察到的 tag
// 这解决了"先删后加"的传统集合合并问题
CRDT 的工程价值：在最终一致性的系统中，最头痛的问题就是冲突解决。常见的"最后写入者获胜（LWW）"策略会丢失数据，而 CRDT 通过数学保证无冲突合并，非常适合协作编辑（如 Google Docs 的 OT 算法变体）、分布式计数器和购物车等场景。实际上，Redis 7.0 引入的 CRDT 功能正是利用了这一思想实现多活架构。

实际系统的选择

不同系统的一致性模型选择直接影响它们的架构和适用场景：

系统 一致性模型 实现机制 适用场景
Google Spanner 外部一致性 (=线性一致性) TrueTime + Paxos 金融、全球级事务
Amazon DynamoDB 最终一致性（可调至强一致） 向量时钟 + 去中心化复制 电商、高可用业务
Apache Cassandra 最终一致性（可调一致性级别） Quorum 读写 + Hinted Handoff 时序、IoT、大规模写入
MongoDB 可调一致性（Primary 读强一致） Primary-Secondary + Oplog 内容管理、实时分析
ZooKeeper 线性一致性（写操作） Zab (原子广播) 分布式协调、配置管理
踩坑故事：曾经有一个团队在 Cassandra 上构建了一个社交点赞系统。他们使用了默认的最终一致性配置，结果发现用户有时能看到自己的点赞在几秒后"消失"。原因是他们用了 QUORUM 读但写时才用了 ONE，导致读到的数据不够新。最终他们调整为读写都用 QUORUM，并配合 Lightweight Transaction（CAS 操作）解决了"重复点赞"问题。这个故事告诉我们："可调一致性"意味着你必须明确配置——默认值不一定适合你的场景。

从一致性模型的谱系可以看出，没有一种模型适用于所有场景。在下一章中，我们将把视角从"数据如何保持一致"转向"消息如何在节点之间传递"，深入探讨分布式系统的通信基础——TCP、UDP 及其背后的网络原理。
