第12章 垂直领域SFT数据

前面几章我们讨论了各种各样的通用数据集。通用数据集的目标是让模型成为一个"知识渊博的万事通"。然而在真实的产品需求中，我们往往需要模型在某个特定领域表现出色——比如写代码、做数学题、回答医疗问题、分析财务报表。一个什么都懂但什么都不精的模型，很多时候不如一个"领域专家"受用户欢迎。

这就是垂直领域 SFT 数据的价值所在。这一章，我们将看看如何通过领域特定的数据，把通用模型训练成领域专家——从代码到数学、从医疗到金融，以及不同领域数据如何与通用数据配合使用。

通用能力靠通用数据，领域专家靠领域数据

在开始之前，先明确一个核心理念：通用数据和领域数据不是谁替代谁的关系，而是互补关系。通用数据提供了基础的"语言能力"和"对话能力"——让模型知道如何组织语言、如何遵循指令、如何保持角色。领域数据则教会模型"领域知识"和"领域特有的思维模式"。

那么，什么时候需要领域数据？当你发现模型在通用场景下表现不错，但在特定领域——比如用特定的编码规范写代码、理解专业的医学文献、分析复杂的财务报表——表现不佳时，就是加入领域数据的时候了。

一个重要的直觉：模型在预训练阶段已经"读过"了大量领域相关的文本（比如 GitHub 上的代码、PubMed 上的医学论文），但这些文本只是"存在"于模型的参数中，模型并没有学会如何以领域专家的方式使用这些知识。领域 SFT 数据的核心作用，就是激活模型已经具备的领域知识，并将其转化为可用的领域技能。

代码领域 — CodeAlpaca 与 CommitPackFT

代码领域是垂直 SFT 数据最成熟的方向之一。几乎所有的通用模型都有专门的代码微调版本——Code LLaMA、StarCoder、DeepSeek-Coder 等。这背后有几个非常成功的代码 SFT 数据集。

CodeAlpaca

CodeAlpaca 是 2023 年推出的轻量级代码指令数据集，包含约 2 万条代码指令-回答对。它用 Alpaca 的 Self-Instruct 方法，专门针对代码生成和理解场景做了定制：种子指令全部是编程相关的任务（"写一个 Python 函数实现二分查找"、"解释这段代码的作用"等），然后让 GPT-3.5 生成指令和回答。

CodeAlpaca 的优势在于轻量。2 万条数据可能在几十分钟内就跑完一轮微调实验，非常适合代码 SFT 的基线验证。但它的局限性也很明显——数据量小、覆盖编程语言有限（主要是 Python），而且生成的代码质量受限于 GPT-3.5 的能力。

CommitPackFT

相比之下，CommitPackFT 是代码 SFT 领域的"重武器"。它由 HuggingFace 的 BigCode 项目发布，从 GitHub 上数百万个代码仓库的 commit 数据中精心筛选和清洗而来。

  • 约 400 万条代码相关的指令数据
  • 数据来源是真实的 GitHub commit 信息——包括代码变更、commit message、代码审查评论等
  • 覆盖多种编程语言：Python、JavaScript、Java、C++、Rust 等
  • 许可证兼容性经过严格筛选（仅使用允许性许可证的代码）

CommitPackFT 的价值在于它的真实性。每条数据来自真实的代码变更场景，而不是模型生成的"教科书式"的代码。这种数据教会模型的代码能力更贴近实际工程需求——包括代码调试、bug 修复、代码重构等实用技能。

// CommitPackFT 数据格式示例
{
  "repo": "pandas-dev/pandas",
  "commit_hash": "a1b2c3d4",
  "language": "Python",
  "instruction": "请修复以下代码中的bug：DataFrame.groupby()后使用agg()时列名丢失的问题",
  "code_before": "df.groupby('A').agg({'B': 'sum', 'C': 'mean'})",
  "code_after": "df.groupby('A', as_index=False).agg({'B': 'sum', 'C': 'mean'})",
  "explanation": "需要设置as_index=False来保持列结构..."
}

数学推理 — GSM8K 与 MATH 的更多使用细节

我们在第 10 章已经详细介绍了 GSM8K 和 MATH 这两个数学推理数据集。这里补充一些在垂直领域使用时的实践细节。

当数学推理是你的核心场景时，建议在 SFT 阶段就将这两个数据集系统地纳入训练数据。具体来说：

  • GSM8K 全部 8500 题作为基础数学推理训练。建议将训练集和测试集分开，只使用训练集（约 7500 题）做 SFT。
  • MATH 的 Level 1-4 题目（约 10000 题）作为进阶推理训练。Level 5 的题目过于困难，在 SFT 阶段反而可能带来噪声，更适合保留到之后的偏好对齐或强化学习阶段使用。
  • 为了增强泛化性，建议对每道题做指令模板随机化——用多种方式表述同一个数学问题（"请解答" / "一步步思考，然后给出答案" / "把这题做出来，并解释每一步"），防止模型对特定指令表述过拟合。
数学数据的 CoT 格式规范化示例： 
# 推荐的多步推理格式
Q: 一个长方形长12厘米，宽是长的三分之二，求面积。

A:
第一步：计算宽 = 12 × 2/3 = 8 厘米
第二步：面积 = 长 × 宽 = 12 × 8 = 96 平方厘米
答案：96 平方厘米
坚持这种"分步标注"格式，模型会学会在推理时自动写出结构化的解题过程。

医疗领域 — Medical Meadow

医疗领域是垂直 SFT 数据最有价值但也最复杂的场景之一。医疗问答对专业性的要求极高——一个错误的回答可能带来严重的后果。这也使得医疗 SFT 数据的构建比其他领域更加谨慎。

Medical Meadow 是当前最全面的开源医疗 SFT 数据集之一：

  • 10 万+ 条医疗领域的指令-回答对
  • 覆盖三大类别：临床问诊（症状描述→诊断建议）、医学知识问答（解剖学、病理学、药理学等）、医学文献解读（论文摘要、临床试验结果分析）
  • 数据来源包括：医学教科书 QA、NLM 的 MEDLINE 引文数据、公开的医疗论坛 QA 等
  • 经过专业的医学知识验证流程——部分数据由医学专业人士做了质量审核

医疗领域 SFT 数据的一个特殊挑战是免责声明的处理。在医学问答中，模型必须学会在适当的时候声明"这是一个 AI 生成的建议，不能替代专业医生的诊断"。好的医疗 SFT 数据会在回答模板中包含这类声明位置，让模型学会"在说建议之前先说不确定性"。

领域 典型数据集 规模 核心特点
代码 CodeAlpaca 2 万条 轻量级、快速验证
代码 CommitPackFT 400 万条 真实 commit、多语言
数学 GSM8K 8,500 题 小学难度、带 CoT
数学 MATH 12,500 题 竞赛级、难度分层
医疗 Medical Meadow 10 万+ 条 问诊/知识/文献三合一

领域数据与通用数据的配比策略

有了领域数据之后，最核心的问题来了：领域数据和通用数据按什么比例混合？有三种主流策略：

策略一：全量替换

完全用领域数据替代通用数据。这种方式只适用于极窄的垂直场景——比如你只需要模型做代码补全，不需要它做任何通用对话。对于大多数产品来说，这是不可取的，因为模型会失去通用对话能力。

策略二：增量混合

在通用 SFT 数据的基础上，按一定比例混入领域数据。这是最常用的策略。典型的比例是 70-80% 通用数据 + 20-30% 领域数据。如果领域专业性要求极高，可以调整到 50/50，但不推荐领域数据超过通用数据。

策略三：两阶段训练

先做通用 SFT（全量通用数据），再做领域 SFT（只使用领域数据）。这种策略的优势是：第一阶段的通用训练建立了完整的语言和指令遵循能力，第二阶段的领域训练在不破坏通用能力的前提下强化领域专长。但风险是第二阶段可能出现灾难性遗忘。

# 两阶段训练示意
# Phase 1: 通用 SFT
通用数据 = load_dataset("teknium/OpenHermes-2.5", split="train")
train_with(通用数据)  # 先让模型学会通用对话

# Phase 2: 领域聚焦 SFT
领域数据 = load_dataset("bigcode/commitpackft", split="train")
# 混入 10% 的通用数据防止遗忘
混合数据 = 领域数据[:90000] + 通用数据[:10000]
train_with(混合数据)  # 领域强化，同时保持通用能力

踩坑故事：只用代码数据微调导致通用对话能力灾难性遗忘

讲一个我在 2024 年遇到的真实案例。一个创业团队正在做一个面向程序员的代码助手产品，他们决定"专注于代码，不需要模型闲聊"。于是他们的数据策略非常简单直接：SFT 阶段只使用代码数据——从 CodeAlpaca、CommitPackFT 和其他几个代码数据集收集了约 50 万条纯代码相关的指令，零通用对话数据。

训练完成后，他们在 HumanEval 和 MBPP 上测试，代码生成能力确实优秀——超过了当时同等规模的开源模型。团队非常满意，开始做用户内测。

然后问题出现了。内测用户开始反馈：这个模型"好用但奇怪"。它能写出漂亮的 Python 函数，但当你跟它说"你好"的时候，它会回复你一段代码。当你问它"今天天气怎么样"，它会开始写一个天气查询 API 的调用示例。当你想和它讨论一下产品的技术方案时，它能回答，但语气生硬，像是在读技术文档。更严重的是，它无法理解非技术性的上下文——如果你在对话中提到了"心情"、"爱好"、"推荐一部电影"这类话题，模型会直接回复"我不理解你的问题，请提供一个编程相关的问题"。

这个问题的本质是灾难性遗忘（Catastrophic Forgetting）。虽然预训练阶段模型学过了通用语言能力，但在 SFT 阶段只使用代码数据，模型被"微调"成了只对代码内容敏感的"偏科生"。它并非不知道如何聊天（预训练阶段学过），而是 SFT 阶段的训练信号全部指向代码模式，导致通用对话的神经网络权重被严重削弱。

教训：除非你的产品真的只需要做一件事（并且用户绝不会问别的问题），否则永远不要把领域数据作为 SFT 的唯一数据源。一个稳健的数据策略应该是：通用数据保证能力的广度，领域数据决定能力的深度。即使你的目标是做垂直领域的专家模型，也至少要保留 30% 的通用数据来维持基础对话能力。记住——用户永远不会按照你预设的"领域边界"来提问。

垂直领域的 SFT 数据让我们能把通用模型变成"专家"。但问题又来了——当你面前有几十个不同来源、不同领域的数据集时，如何决定谁多谁少？把 OpenHermes、UltraChat、GSM8K、Firefly 等数据按什么比例混合才能达到最佳效果？这不仅仅是"拍脑袋"的问题，而是一套严谨的实验设计方法。下一章，我们将进入SFT 数据配比与混合策略的实战环节，看看如何通过系统化的实验找到最适合你的数据配方。
