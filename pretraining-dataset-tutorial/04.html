开篇：通用语料之外，别有洞天

大家好，我是你们的讲师。前两章我们聊了通用基座数据集（以英文为主）和中文语料的现状。但如果你觉得预训练数据就是"把 Common Crawl 洗干净然后喂给模型"，那就大错特错了。

实际上，让模型真正"聪明"起来的，往往不是成千上万亿 token 的通用网页文本，而是那些经过精心挑选、质量极高的垂直领域数据。代码数据让模型学会了推理，数学数据让模型学会了严谨，学术论文让模型学会了深度思考。这些数据在总量中占比不大（通常在 5%-15%），但对模型能力的贡献却远超其规模比例。

我常说一句话：通用语料决定了模型的下限，垂直语料决定了模型的上限。本章我们就来盘点那些"小而美"的垂直领域数据集。

4.1 The Stack：3TB 的代码宇宙

The Stack 是 BigCode 项目（ServiceNow 和 HuggingFace 联合发起）发布的多语言代码数据集。它的规模达到了惊人的 3.1TB，涵盖超过 300 种编程语言，是现成的代码预训练数据中最大、最全面的一个。

The Stack 的构建流程本身就是一篇教科书：

处理步骤 说明 数据缩减比例
原始 GitHub 仓库抓取 从 GitHub Archive 获取公开仓库列表，克隆代码 ~100% 起点
许可证过滤 根据仓库 LICENSE 文件，只保留允许训练的许可证 约保留 70%
大文件过滤 移除超过 1MB 的文件（自动生成的文件、数据文件） 约保留 50%
精确去重 去除完全相同的文件（MD5 hash） 约保留 35%
近重复去重 The Stack Dedup 版本，使用 MinHash + LSH 约保留 20%

这里有一个非常重要的版本区分：The Stack（原始版）和 The Stack Dedup（去重版）。原始版包含大量重复的代码——同一个开源库的代码被数十个项目的 fork 重复收录。去重版通过 MinHash 模糊去重，将这个比例从 100% 缩减到了约 20%，但仍然保留了绝大部分代码能力。

The Stack 中各语言排名（Top 10，按文件数）：
  ┌──────────────┬──────────┬───────────┐
  │ 编程语言     │ 文件数    │ 总大小    │
  ├──────────────┼──────────┼───────────┤
  │ JavaScript   │ 7.2M     │ 472 GiB   │
  │ Python       │ 5.6M     │ 298 GiB   │
  │ Java         │ 4.1M     │ 210 GiB   │
  │ TypeScript   │ 3.8M     │ 195 GiB   │
  │ C/C++        │ 2.9M     │ 185 GiB   │
  │ Rust         │ 1.4M     │ 72 GiB    │
  │ Go           │ 1.2M     │ 65 GiB    │
  │ Ruby         │ 1.1M     │ 55 GiB    │
  │ PHP          │ 1.0M     │ 52 GiB    │
  │ Shell        │ 0.9M     │ 38 GiB    │
  └──────────────┴──────────┴───────────┘
一个容易被忽视的细节：The Stack 中的代码质量差异巨大。从顶尖的框架代码（如 PyTorch、React）到刷题练习的 LeetCode 答案，再到质量低劣的学生作业，都在同一个数据集里。如果不对代码质量做区分，模型会学到一些非常不好的编程习惯。建议对大 Star 数的仓库加权采样——一个 10K Star 的项目的代码文件的训练权重，应该是一个无人维护的"helloworld"项目的 100 倍。

4.2 CodeParrot：Python 专属的 100GB

如果说 The Stack 是一个"代码大集市"，那么 CodeParrot 就是一个"Python 精品店"。CodeParrot 由 HuggingFace 团队开发，专注于 Python 代码，总规模超过 100GB。

CodeParrot 最大的特色是它对注释和文档字符串的特别处理。Python 代码的一个显著优势是：它通常包含丰富的注释（docstring、inline comments），这些注释可以视为代码和自然语言之间的"桥梁"。模型在阅读 Python 代码时，可以通过注释理解代码的意图——这对提升模型的"代码理解"能力至关重要。

CodeParrot 的清洗策略包括：

  • 只保留扩展名为 .py 的文件
  • 过滤掉长度超过 1000 行的文件（避免自动生成的代码污染）
  • 过滤掉只包含异常处理框架而无实质代码的"空壳"文件
  • 符号率（alphanumeric ratio）过滤：移除符号占比过高的杂乱文件
推荐做法：在预训练时，可以同时使用 The Stack（覆盖多语言能力）和 CodeParrot（深入 Python 专项）。两者是互补关系，不是替代关系。The Stack 负责广度，CodeParrot 负责深度。

4.3 OpenWebMath 与 ProofPile：数学与证明

数学推理能力是大模型能力皇冠上的明珠。2024 年以来，提升模型的数学能力成为研究热点，而专门的数学数据集在其中发挥了关键作用。

OpenWebMath 是一个从网页中提取的数学内容数据集，包含约 150B token。它的构建思路很有意思：

  • 从 Common Crawl 中筛选包含数学公式的页面（检测 LaTeX 标记 $$…$$ 或 \[…\]）
  • 提取页面中的纯文本和 LaTeX 公式
  • 过滤掉"题目集"类页面（避免数学 benchmark 数据污染）
  • 按 Perplexity 对页面进行质量排序

ProofPile 则更进一步，收录了来自 ArXiv、数学教科书、以及形式化证明系统（如 Lean、Coq）的证明文本。proofpile 包含约 8GB 的文本，虽然体量不大但知识密度极高。

ProofPile 数据组成：
  ┌──────────────────┬────────┬────────────────────────────┐
  │ 子集             │ 大小   │ 说明                       │
  ├──────────────────┼────────┼────────────────────────────┤
  │ ArXiv 数学论文   │ 5.0GB  │ 包含完整证明的数学论文     │
  │ 数学教科书       │ 1.5GB  │ OpenStax 等开源教材       │
  │ Lean 证明语料    │ 0.8GB  │ 形式化证明代码             │
  │ Coq 证明语料     │ 0.4GB  │ 另一种形式化证明系统       │
  │ StackExchange 数学│ 0.3GB  │ Math StackExchange 问答   │
  └──────────────────┴────────┴────────────────────────────┘
我的经验：如果只允许我从所有垂直数据中挑一个"最重要的"，我会选择数学数据。原因很简单：数学（特别是形式化证明）是自然语言中逻辑密度最高的表达形式。一个学会了数学推理的模型，会在代码、逻辑推理、乃至一般的知识问答中全面受益。数学数据是性价比最高的"能力催化剂"。

4.4 arXiv：2M+ 学术论文的知识金矿

arXiv 数据集包含了截至 2026 年超过 200 万篇学术论文的 LaTeX 源码和 PDF 解析文本，覆盖物理学、数学、计算机科学、生物学、统计学等几乎所有自然科学领域。

arXiv 数据在预训练中的价值体现在三个层面：

  • 知识密度：一篇 arXiv 论文的信息量等于成百上千篇普通网页文章。用 10GB 的 arXiv 数据替代 10GB 的 Common Crawl 数据，相当于把知识的"密度"提高了 10-100 倍。
  • 结构化表达：论文有固定的格式（摘要→引言→方法→实验→结论），这种结构化的知识呈现方式有助于模型学会"有头有尾"的写作逻辑。
  • 交叉学科知识：arXiv 论文涵盖了人类知识的最前沿，包含大量在 Wikipedia 和教科书中尚未收录的最新研究成果。

使用 arXiv 数据需要注意的一个重要问题：LaTeX 源码 vs 纯文本。arXiv 提供两种格式：

  • LaTeX 源码（推荐）：包含完整的论文排版信息，包括数学公式、引用、图表说明。模型可以从 LaTeX 源码中学到严谨的学术写作风格。
  • PDF 解析文本（不推荐）：由 PDF 自动提取的文本，常出现公式乱码、章节错位、引文丢失等问题。除非没有办法解析 LaTeX，否则应优先使用 LaTeX 源码。

4.5 FLAN Collection：指令数据的母版

FLAN Collection 是 Google 发布的指令微调数据集集合，但它对预训练数据工程的意义远不止于微调。FLAN Collection 包含 1,800+ 个 NLP 任务，所有任务都被统一转换成了"指令-输入-输出"的格式。

为什么一个微调数据集会出现在预训练数据课程中？答案是：越来越多的研究证明，在预训练阶段混入少量的指令格式数据，可以显著提升模型的"上下文学习"（in-context learning）能力。具体来说，在预训练的后期阶段（最后 5%-10% 的步骤）混入 FLAN Collection 风格的指令数据，可以让模型：

  • 更快地学会遵循指令
  • 更好地理解任务描述格式
  • 在少样本（few-shot）场景下表现更好

这种在预训练最后阶段加入指令数据的策略，被称为 "词汇对齐"或"格式预热"（format warmup）。它的数据量需求很小——通常只需几十亿 token——但对最终模型的使用体验改善非常明显。

实操建议：在你的预训练计划中，预留最后 5% 的训练步骤，将数据配比切换为"90% 常规预训练数据 + 10% 指令格式数据（FLAN/UltraChat 等）"。这会带来一个意想不到的好处：微调阶段需要的 SFT 数据量可以减少 30%-50%，因为模型已经在预训练阶段学会了"理解指令"的能力。

4.6 UltraChat 200k：高质量多轮对话

UltraChat 200k 是由清华大学和 Zhipu AI 联合发布的对话数据集，包含 200K 轮高质量的多轮对话。不同于一般的对话数据集，UltraChat 的对话是通过"角色扮演"方式由模型自动生成的——一个模型扮演"用户"，另一个扮演"助手"，通过这种互动产生了大量多样化的对话。

UltraChat 的设计亮点：

维度 特点 意义
话题多样性 涵盖 20+ 大类的日常对话 覆盖真实用户的大部分对话场景
轮次深度 平均每段对话 5-8 轮 训练模型保持长程上下文关联
指令复杂性 包含多步指令、条件约束 提升模型的精细指令遵守能力
格式多样性 交替使用中文和英文 支持中英双语场景
一个值得思考的趋势：2025-2026 年，越来越多的预训练项目开始将"对话数据"纳入预训练阶段，而不仅仅是微调阶段。这背后的逻辑是：对话本身就是一种特殊的文本分布——它包含了问答、讨论、辩论、解释等多种语言使用模式。提前让模型在预训练阶段"习惯"这些模式，可以显著减少后续对齐训练的工作量。

4.7 医疗与法律等专业领域数据

除了上述通用型垂直语料，一些专业领域还有自己的高质量数据集：

  • PMC Open Access（PubMed Central）：由 NIH 提供的开放获取生物医学文献集合，包含约 300 万篇全文论文。许可证为 CC BY / CC0，对商业使用友好。
  • Pile of Law：由 The Pile 团队扩展的法律文本数据集，包含法庭判例、法规、法律评论等。总规模约 256GB，但部分子集有许可证限制。
  • CommonVoice：Mozilla 的众包语音数据集，如果训练多模态或语音模型，这是最常用的开源源之一。
  • GOAT（General Multimodal Dataset）：多模态预训练数据集的代表，包含图像-文本配对数据，适合训练视觉-语言模型。

需要特别说明的是：专业领域数据虽然质量高，但不宜过量使用。医学论文的语言风格与日常对话差异巨大——过多的专业语料会让模型"说话像写论文"。在实践中，专业领域数据的占比通常控制在总数据量的 2%-5%，这是一种"够用就好"的原则。

4.8 垂直数据的整体配比参考

最后，我们用一个表格来总结本章介绍的所有垂直数据，以及它们在典型预训练项目中的建议配比：

垂直领域 代表数据集 建议占比 对模型能力的贡献
代码 The Stack, CodeParrot 8%-12% 推理、逻辑、结构化思维
数学 OpenWebMath, ProofPile 3%-5% 严谨推理、数字理解
学术论文 arXiv, PMC OA 3%-5% 深度知识、结构化表达
指令/对话 FLAN, UltraChat 1%-3%（仅预训练后期） 指令理解、对话能力
医疗 PMC Open Access 1%-2%（如需专项优化） 医学知识
法律 Pile of Law 1%-2%（如需专项优化） 法律知识、严谨用词

请记住：这些数据不是"越多越好"。代码加太多了，模型会变得"话里带代码"——回答问题时莫名地输出 JSON 或 import 语句。学术论文加太多了，模型会变得"啰嗦而严谨"。垂直数据的艺术在于——找到那个"足量但不过量"的甜点。

最后一个实战建议：垂直数据不要一次性全部混入训练。我建议的做法是：先用通用数据训练 70% 的步数，然后逐步引入垂直数据。这样做的好处是让模型先建立足够的语言基础，再在"微调阶段"（预训练的后半段）吸收垂直领域的知识。这比一开始就混合所有数据的效果通常更好。

本章小结

本章我们探索了通用网页之外的另一片天地——垂直领域与增强语料。从 The Stack 的庞大代码宇宙到 OpenWebMath 的数学精萃，从 arXiv 的学术金矿到 FLAN 的指令母版，每种数据都有其独特的价值定位。核心认知是：垂直数据在总量中占比虽小，但它们是模型能力的"放大器"，决定了一个通用模型能否在具体任务上脱颖而出。

在下一章中，我们将回到所有预训练数据的"原点"——Common Crawl，这个互联网最大的公开网页存档。我们会拆解它的内部格式，分析它的优缺点，并探讨如何从数百 TB 的原始网页中提取有价值的训练数据。你会发现，Common Crawl 既是石油矿藏，也是一片险象环生的丛林。我是你们的讲师，我们第 5 章见。
