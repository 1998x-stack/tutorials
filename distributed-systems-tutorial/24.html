第24章 可观测性

有一个流传很广的运维谚语："You can't fix what you can't see." 在单体应用时代，观察系统很简单——看一台机器的CPU和内存，翻一个日志文件。但在微服务和分布式系统里，一次用户请求可能穿越十几个服务、跨越多台机器、涉及不同的数据库和队列。这个时候，"监控"这个词已经不够用了，我们需要的是可观测性（Observability）。

可观测性不是某个单一工具，而是三大支柱（Three Pillars）的协同作战：Metrics（指标）告诉你"系统是不是出问题了"；Tracing（追踪）告诉你"哪个环节慢了"；Logging（日志）告诉你"具体报了什么错"。这三者缺一不可，联合作战才能应对分布式系统的复杂性。

Metrics：量化一切

指标是可观测性的第一道防线。好的指标设计让你在收到告警时就已经猜到了问题的大致范围。

RED 与 USE 方法论

Google SRE团队总结了两套实用的指标框架：

框架指标含义示例
RED
（面向服务）		Rate请求速率每秒多少请求
Errors错误数4xx/5xx响应数量
Duration延迟分布P50/P90/P99延迟
USE
（面向资源）		Utilization资源利用率CPU 80%、内存 70%
Saturation资源饱和度CPU run queue长度、磁盘IO等待
Errors资源错误网卡丢包、磁盘坏道

简单来说：RED方法适合度量"服务"（你的API、数据库连接池），USE方法适合度量"资源"（CPU、内存、磁盘、网络）。按这个模板去梳理，你基本不会遗漏关键指标。

Prometheus：Pull模型的胜利

Prometheus是云原生时代的事实标准监控系统。它的核心设计是Pull模型：Prometheus Server定期从各Target（应用暴露的/metrics端点）拉取指标数据。这比传统的Push模型（如Zabbix）有几个显著优势：你不需要知道Target在哪（Service Discovery自动发现）、如果Prometheus挂了不会堵住应用、检活也天然成立——拉不到数据说明Target可能挂了。

PromQL是它的查询语言，一句简单的查询就能回答"过去5分钟P99延迟超过200ms的服务有哪些"：

histogram_quantile(0.99,
  rate(http_request_duration_seconds_bucket[5m])
) > 0.2

Tracing：串起千丝万缕

Metrics告诉你某个服务延迟高了，但你想知道"这个请求具体在哪个环节慢了"——这就需要分布式追踪。

W3C Trace Context标准

为了让不同语言、不同框架的追踪数据能互相关联，W3C制定了Trace Context标准（traceparent 和 tracestate 两个HTTP头）。一个典型的 traceparent 头长这样：

traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
              ^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ ^^^^^^^^^^^^^^^^^^^ ^^
            版本         trace-id (16字节)            span-id (8字节)  标志

当一个请求经过Service A，A生成一个Span并把这个Header传给Service B；B接收后提取trace-id，把自己创建的新span挂在同一个trace树上。整个请求路径最终形成一棵Trace→Span DAG，每个Span记录了开始时间、结束时间、调用方和被调用方的信息。

Jaeger架构

Jaeger是CNCF毕业的分布式追踪系统，架构主要包含：

  • Agent：DaemonSet方式部署在每个节点上，接收应用发出的Span数据，批量发送给Collector。
  • Collector：校验、索引Span数据，写入后端存储（Elasticsearch或Cassandra）。
  • Query + UI：提供搜索和可视化界面，按Service、Operation、Tags过滤，展示火焰图和Trace Timeline。

代码插桩目前最主流的方式是使用OpenTelemetry SDK。在Go里引入几行代码就能自动为你的HTTP handler创建Span：

import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/trace"
)

tracer := otel.Tracer("payment-service")
ctx, span := tracer.Start(r.Context(), "process-payment")
defer span.End()

// 在span中记录自定义属性
span.SetAttributes(
    attribute.String("payment.id", paymentID),
    attribute.Float64("amount", amount),
)

Logging：最后的真相

Metrics给了你宏观视野，Tracing给了你链路视图，但当你要确认"这条SQL到底为什么慢"时，最终还是得看日志。

结构化日志

别再写这样的日志了：2024-06-30 12:00:00 ERROR user 12345 payment failed。没有字段名、没有可解析结构、grep完还得人工脑补。改成JSON格式：

{"level":"error","ts":"2024-06-30T12:00:00Z",
 "trace_id":"0af7651916cd43dd8448eb211c80319c",
 "service":"payment","user_id":12345,
 "msg":"payment failed","error":"insufficient_balance",
 "amount":99.99}

看到了吗？trace_id直接嵌在日志里。这样你在Jaeger里看到一个慢Trace，可以直接搜trace_id找到对应的应用日志——"三大支柱联动"就是靠这种字段关联起来的。

ELK Stack

Elasticsearch + Logstash + Kibana（以及更轻量的Filebeat替代Logstash做日志采集）是目前最主流的日志方案。Filebeat采集日志文件，发送到Elasticsearch，Kibana提供搜索和Dashboard。加上Logstash可以做字段解析和过滤。

生产环境中注意两个问题：一是日志采样，高吞吐服务（每秒几万QPS）用不着每条都记——设置采样率比如1%，或者使用自适应采样（慢请求全采，正常请求采样）。二是磁盘保护，配置log rotation和保留策略，避免日志把磁盘打满。

三大支柱联动：排障实战

假设你收到了一个告警："订单服务的P99延迟从50ms飙升到了5秒"。你按照以下链条一路查下去：

  1. 告警→Metrics：打开Grafana Dashboard，看到订单服务的RED指标——Rate没变，Errors没涨，Duration的P99确实飙升了。
  2. Metrics→Tracing：进入Jaeger，按"order-service"和"p99 latency"过滤，挑一个慢Trace查看。Trace显示总耗时4.8s，其中4.5s花在一个对"inventory-service"的gRPC调用上。
  3. Tracing→Logging：在Kibana中搜索这个Trace的trace_id，找到inventory-service的日志。日志显示："connection pool exhausted, waited 4500ms for a connection"。
  4. 结论：inventory-service的数据库连接池太小，被突发流量打满了。修复方案：调整连接池上限并增加实例数。
关键思路：永远不要只依赖一个数据源。Metrics告诉你"是什么"，Tracing告诉你"在哪里"，Logging告诉你"为什么"——三个步骤缺一环都可能让你在原地打转。

告警设计

告警设计是一门平衡的艺术。告警太少你会错过故障，告警太多你会陷入"狼来了"的麻木。

告警分级

级别含义响应时间示例
P0核心功能完全不可用立即响应（<15min）支付链路全红、用户无法登录
P1核心功能严重受损30分钟P99延迟>5s、交易成功率<90%
P2非核心功能受影响4小时后台报表延迟、某个次要API报错

SLI / SLO / SLA

  • SLI（Service Level Indicator）：你实际测量的指标，比如"请求成功率99.9%"、"P99延迟200ms"。
  • SLO（Service Level Objective）：你的目标值，比如"未来30天内，99.9%的请求成功，P99延迟<300ms"。
  • SLA（Service Level Agreement）：对客户的承诺，达不到要赔钱的。SLA一般比SLO宽松一些。

Burn Rate（烧钱率）是一个很实用的告警策略：如果你的SLO是99.9%（一个月允许43.8分钟的不可用时间），当过去1小时的错误预算消耗速度达到"按这个速度5小时内就会烧光一个月预算"时立刻报警。这比简单的静态阈值灵敏得多。

关于告警疲劳：如果你的告警群每天轰炸几百条消息但没人处理，说明你需要重新审视告警规则。一个简单的标准——每条告警触发后应该有人去排查；如果总是"看一眼发现没事然后静默"，那条规则就该优化或删除。

踩坑：Prometheus Cardinality爆炸

Prometheus的性能瓶颈几乎永远出在同一个地方——标签基数（Cardinality）爆炸。

曾经有个同事在上线新功能时，在Prometheus指标里加了一个 user_id 标签，用来按用户维度统计请求数。上线后Prometheus OOM了——几百万用户就意味着几百万个时间序列，每个序列都要在内存里维护。Prometheus不是为这种高基数标签设计的。

类似的教训：不要把 request_id、email、ip_address 作为标签。这些应该放在日志或者Trace里。标签设计原则是：基数越低的标签越适合做Prometheus标签（如 status_code、method、endpoint），几十个值以内的最佳。

踩坑记录：另一个常见问题是Grafana Dashboard信息过载。每个团队都想把自己的指标放上去，最后Dashboard上几十个Panel，五颜六色，根本不知道看哪里。好的做法是建立"层级Dashboard"——第一层只有一个Panel：系统健康度总评（红/黄/绿）；第二层按服务拆开；第三层才是具体指标的详细视图。P0值班人员只需要看第一层。

这一章我们详细讲解了可观测性的三大支柱——Metrics、Tracing、Logging——以及它们如何协同工作帮你排障。告警设计决定了你在故障发生时多久能知道，SLI/SLO/SLA决定了你承诺了什么样的服务质量。有了这些工具和方法，你的分布式系统才算真正"可观测"。下一章我们将进入一个更刺激的话题——调试与压测：当系统表现诡异、Bug难以复现的时候，我们怎么用混沌工程、故障注入和形式化验证来驯服分布式系统的复杂性？
