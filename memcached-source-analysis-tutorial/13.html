第13章 哈希表：assoc模块详解

大家好，我是你们的讲师。过去几章我们花了大量精力在内存管理上——从 slab allocator 到 item 结构体再到 LRU 淘汰。但 Memcached 作为 KV 缓存，最核心的操作是：通过 key 找到 value。这个"查找"的功能就是由 assoc（association）模块实现的。

assoc 模块是 Memcached 代码中最精简、最优雅的模块之一——它的全部代码不到 400 行，却实现了一个功能完整的、支持渐进式扩容的哈希表。本章我们聚焦哈希表的基础操作，下一章再看扩容和 rehash。好，我们开始。

数据结构：item** primary_hashtable

Memcached 的哈希表本质上是一个桶数组 + 链地址法的经典实现。核心数据结构极其简洁：

// assoc.c 中的全局变量
static item **primary_hashtable = NULL;  // 桶数组
static unsigned int hashpower = HASHPOWER_DEFAULT;  // 2^hashpower 个桶
static unsigned int hash_items = 0;  // 当前哈希表中的 item 数量

// hashsize(n) = 1 << n
#define hashsize(n)       ((unsigned long int)1 << (n))
#define hashmask(n)       (hashsize(n) - 1)

primary_hashtable 是一个指向指针数组的指针，每个数组元素是一个 item * 指针，指向该 bucket 链表的第一个 item。默认 hashpower = 16，即初始有 2^16 = 65536 个桶。

当客户端 set 一个 key 时，流程是：

  1. 用 hash 函数将 key 映射为一个 32 位整数
  2. 取低 hashpower 位作为桶索引（hash & hashmask(hashpower)）
  3. 在对应的 bucket 链表中查找（链地址法）

桶数组本身是用 calloc 分配的连续内存，初始化为 NULL（空链表）。

链地址法：item → h_next 单向链表

每个桶内部是一个单向链表，通过 item 结构体的 h_next 字段串联：

// memcached.h 中的 item 结构体
typedef struct _stritem {
    struct _stritem *next;    // LRU 双向链表
    struct _stritem *prev;
    struct _stritem *h_next;  // 哈希表冲突链（单向）
    // ... 其他字段
} item;

注意，h_next 和 next / prev 是独立的指针——item 同时活在两个数据结构中：

  • 哈希表：通过 h_next 串成单向链表，用于 key 的快速查找
  • LRU 链表：通过 next / prev 串成双向链表，用于淘汰决策

这种"双重存在"的设计是 Memcached 中的一个重要思想：item 不"属于"任何一个数据结构，而是同时参与了多个数据结构。有点像数据库中的"多索引"概念——一条记录可以同时被 B+ 树索引和哈希索引引用。item 的 next/prev 是 LRU 索引，h_next 是哈希索引。

assoc_find()：逐行解析

查找操作是 assoc 模块最频繁的调用路径（每次 get / set / delete 都需要查找）。代码非常精简：

item *assoc_find(const char *key, const size_t nkey) {
    // 1. 计算哈希值
    uint32_t hv = hash(key, nkey);

    // 2. 找到桶索引
    unsigned int oldbucket = hv & hashmask(hashpower);

    // 3. 遍历链表（rehash 时的双表处理，下一章展开）
    if (expanding && oldbucket < expand_bucket) {
        // rehash 过程中，此 bucket 已迁移到新表，需查新表
        oldbucket = hv & hashmask(hashpower + 1);
    }

    // 4. 遍历链表
    item *it = primary_hashtable[oldbucket];
    while (it) {
        if (nkey == it->nkey && memcmp(key, ITEM_key(it), nkey) == 0) {
            return it;  // 找到了！
        }
        it = it->h_next;
    }

    return NULL;  // 没找到
}

查找的时间复杂度：

  • 最优：O(1)——key 刚好是桶中的第一个元素
  • 平均：O(1 + α)，其中 α = hash_items / hashsize(hashpower) 是负载因子
  • 最差：O(n)——所有 key 都落在同一个桶里（哈希碰撞攻击）

在默认配置下，hashpower=16 有 65536 个桶。当存了 10 万个 item 时，平均负载因子约为 1.5，每个桶的链表平均长度不到 2。这个查找效率是非常高的。

优化的比较策略：注意 assoc_find 先比较 nkey（长度），再调用 memcmp。这是一个常见的优化——key 长度不同肯定不是同一个 key，所以可以在 memcmp 之前快速排除。key 长度相同的概率远小于长度不同的概率，所以这个"先比长度"的策略在统计学上能节省大量 memcmp 调用。

assoc_insert()：头插法

插入操作同样简洁——使用头插法将新 item 插入到对应桶的链表头部：

int assoc_insert(item *it) {
    uint32_t hv = hash(ITEM_key(it), it->nkey);
    unsigned int oldbucket = hv & hashmask(hashpower);

    // rehash 期间，可能插入到新表
    if (expanding && oldbucket < expand_bucket) {
        oldbucket = hv & hashmask(hashpower + 1);
        it->h_next = primary_hashtable[oldbucket];
        primary_hashtable[oldbucket] = it;
    } else {
        // 正常插入
        it->h_next = primary_hashtable[oldbucket];
        primary_hashtable[oldbucket] = it;
    }

    hash_items++;
    return 1;
}

为什么用头插法而不是尾插法？因为 O(1)。尾插法需要遍历到链表尾部，时间复杂度 O(n)。头插法直接插到头部，永远 O(1)。

你可能会担心：头插法会导致"后插入的 item 先被找到"——但这对功能没有影响，因为 key 是唯一的，找到任何一个匹配的就是对的。而最近插入的 item 恰恰更有可能被立即访问（局部性原理），所以头插法的查找效率反而可能更好。

插入后的链表结构：

bucket[hv] ──→ item_C (新插入) ──→ item_B ──→ item_A ──→ NULL
                    h_next          h_next      h_next

assoc_delete()：二级指针遍历删除

删除操作需要处理链表节点的移除。Memcached 使用了一种"二级指针"的技巧来避免特殊处理头节点删除的情况：

void assoc_delete(const char *key, const size_t nkey) {
    uint32_t hv = hash(key, nkey);
    unsigned int oldbucket = hv & hashmask(hashpower);

    // rehash 期间可能涉及双表
    if (expanding && oldbucket < expand_bucket) {
        oldbucket = hv & hashmask(hashpower + 1);
    }

    // 二级指针遍历
    item **prev = &primary_hashtable[oldbucket];
    item *it = *prev;

    while (it) {
        if (nkey == it->nkey && memcmp(key, ITEM_key(it), nkey) == 0) {
            // 找到了！通过 prev 指针修改前一个节点的 h_next
            *prev = it->h_next;
            hash_items--;
            return;
        }
        prev = &it->h_next;
        it = it->h_next;
    }
}

二级指针的技巧：prev 指向"当前节点的前一个节点的 h_next 字段"（或者头指针本身）。当找到要删除的节点时，*prev = it->h_next 直接修改前一个节点的 h_next（或者桶的头指针），将当前节点从链表中摘除。这个写法避免了"如果删除的是头节点，需要单独处理"的分支逻辑——比经典的双指针 prev/curr 遍历更简洁。

Hash 函数：从 Jenkins 到 MurmurHash3

哈希函数的质量直接决定了哈希表的性能。Memcached 使用了一个可插拔的 hash 函数体系：

// hash.c 中的函数指针
static uint32_t (*hash_fn)(const void *key, size_t length) = NULL;

支持的 hash 函数：

Hash 函数 启用方式 质量 速度
Jenkins lookup3 默认（无需配置） 优秀 快（纯 C 实现）
MurmurHash3 -o modern 或 -o hash_algorithm=murmur3 更均匀 更快（利用现代 CPU 特性）
CRC32 -o hash_algorithm=crc32 中等 很慢（仅用于兼容旧版本）

在最新的 1.6.41 版本中，使用 -o modern 开关会启用 MurmurHash3，同时也会开启其他"现代化"特性（如二进制协议全文支持）。这是推荐的做法。

为什么 hash 函数的质量很重要？考虑一个极端的反例：假设 hash 函数把所有 key 都映射到第 0 个桶。那么 assoc_find() 的性能会退化为 O(n)——每次 get 都要遍历 10 万个 item 的链表。这就是所谓的 hash 碰撞攻击。Jenkins hash 和 MurmurHash3 都经过了严格的随机性测试，能够抵抗这种攻击。

对比 Redis：Redis 使用 MurmurHash2 和基于大小写的 dictGenHashFunction。两者都选用了高质量的非加密哈希函数——不在安全上有保障，但在分布均匀性和速度之间取得了很好的平衡。如果你对哈希函数感兴趣，Bob Jenkins 的 lookup3.c 注释本身就值得一读——它包含了哈希函数设计的很多精妙思路。

踩坑：Hash Collision 导致单 bucket 链表过长

哈希碰撞在某些情况下会导致严重的性能问题。让我分享一个实际 case：

有一次，我们的 Memcached 实例在 QPS 没有明显变化的情况下，CPU 使用率从 20% 飙升到了 80%。火焰图显示 assoc_find() 的占比从正常的 3% 飙升到了 35%。

用 stats buckets 查看各 bucket 的 item 分布，发现有一个 bucket 链表中竟然有 200 多个 item！而其他 bucket 平均只有 2-3 个。这意味着 hash 函数对这个特定模式的前缀产生了"聚集"。

经过排查，发现我们的 key 模式是 user:{user_id}:profile，user_id 是递增的数字。在默认的 Jenkins hash 下，这个模式的 key 在低 16 位上有明显的"聚集"——因为数字的二进制表示中，低 16 位的分布不够均匀。

解决方案有两个：

  1. 立即修复：通过 -o hash_algorithm=murmur3 切换到 MurmurHash3，这个函数对数字 key 模式有更好的均匀性
  2. 长期方案：增大初始 hashpower（通过 -o hashpower_init=18），让桶的数量从 65536 增加到 262144，降低碰撞概率
教训总结：默认的 Jenkins hash 在大多数情况下表现良好，但在特定的 key 模式下可能产生严重的碰撞。如果你的 key 是可预测的（比如递增 ID + 固定前缀），强烈建议使用 -o modern 启用 MurmurHash3。另外，通过 stats buckets 命令定期检查哈希分布也是一个好习惯——如果发现某个 bucket 的 item 数量远超平均值（比如超过平均值的 10 倍），说明 hash 函数可能不适合你的 key 模式。

另外，如果你的业务面临潜在的"哈希碰撞攻击"风险（比如用户提供的 key 没有经过任何限制），可以考虑在 Memcached 前面加一层网关，对 key 做二次哈希。
