第28章 综合项目

大家好，我是你们的讲师。欢迎来到本教程的最后一章。过去 27 章中，我们从概率生成的理论基础出发，走过了 VAE、GAN、扩散模型、自回归模型四大流派，深入了 Stable Diffusion 的架构细节、DiT 和 FLUX 的前沿进展，学习了质量调优、模型部署、自回归统一范式以及实时生成新范式。现在，是时候把所有知识串联成一个完整的端到端项目了。

我们项目的目标是：构建一个面向室内设计领域的文生图服务。用户输入"现代风格客厅，浅灰色沙发，落地窗，绿植点缀"，系统返回一张高质量、符合室内设计规范的效果图。这个项目覆盖了模型选型、数据构建、LoRA 微调、ControlNet 接入、API 服务搭建到部署监控的全链路。

28.1 模型选型分析

第一步是选择基座模型。2026 年市场上有三个主要候选：

对比维度SDXLSD3-MediumFLUX.1-dev
参数量2.6B2.0B (MM-DiT)12B (MM-DiT)
推理显存 (FP16)~7GB~6GB~24GB
LoRA 训练显存~12GB~10GB~48GB (需量化)
原生分辨率1024×10241024×10241024×1024~2048×2048
生态成熟度极高（ControlNet, LoRA 丰富）中等中等（工具链增长中）
生成质量 (base)良好优秀顶尖

基于室内设计场景的需求，我们选择 SDXL + LoRA 作为起步方案。原因有三：

  • SDXL 的生态最成熟——ControlNet、LoRA、T2I-Adapter 等工具链完备
  • 训练和推理的硬件门槛最低——一张 RTX 4090（24GB）就能完成 LoRA 训练和推理
  • 室内设计场景不需要 FLUX 那样的极致画质，SDXL 足够
🌟 选型决策树：如果硬件资源充足（A100 80GB），首选 FLUX.1-dev；如果需要可控性和成熟工具链，首选 SDXL；如果追求最新架构且显存在 24GB 以内，SD3-Medium 是平衡之选。室内设计这种"需要精细控制（户型图 → 效果图）"的场景，SDXL 的 ControlNet 生态是最强优势。

28.2 数据集构建

数据集是 LoRA 微调成败的核心。我们构建了一个约 5000 对的室内设计数据集：

  • 数据来源：从 Houzz、Pinterest、住小帮等平台爬取室内设计图片（约 20000 张）
  • 清洗：去除水印、低分辨率（<512px）、构图不佳（自动美学评分 < 0.6）的图片，剩余约 8000 张
  • 标注：使用 CogVLM2 多模态大模型自动生成描述，人工审核抽检 10%，修正常见错误（如风格误判、颜色描述错误）
  • 筛选：由专业室内设计师剔除"设计不合理"（如动线混乱、尺度失调）的样本，最终保留约 5000 对
💡 数据质量 > 数据数量：我们的实验表明，5000 对精心筛选的室内设计数据微调出来的 LoRA，效果远好于 20000 对"原始爬取 + 自动标注"的数据。核心原因在于：文生图模型对 prompt-image 对齐非常敏感——如果标注的描述与图像内容不匹配，模型学到的是错误的文本-图像对应关系，生成结果会"文不对题"。

28.3 LoRA 微调实操

使用 Diffusers 的 LoRA 训练脚本，核心训练配置如下：

# train_lora_interior.py — 核心配置
accelerate launch train_lora.py \
    --pretrained_model_name_or_path="stabilityai/stable-diffusion-xl-base-1.0" \
    --dataset_name="./data/interior_design_5000" \
    --caption_column="prompt" \
    --resolution=1024 \
    --train_batch_size=2 \
    --gradient_accumulation_steps=4 \
    --max_train_steps=2000 \
    --learning_rate=1e-4 \
    --lr_scheduler="cosine_with_restarts" \
    --lr_warmup_steps=200 \
    --rank=64 \
    --alpha=32 \
    --mixed_precision="fp16" \
    --validation_prompt="现代风格客厅, 浅灰色沙发, 落地窗" \
    --validation_epochs=50 \
    --output_dir="./lora-interior-sdxl"

几个关键超参数的调优经验：

  • rank=64 是室内设计场景的 sweet spot——rank 太低（如 8）无法学到足够的设计风格细节，rank 太高（如 128）容易过拟合
  • 学习率 1e-4 加上 cosine with restarts 调度器——比固定学习率快约 30% 收敛
  • 验证频率：每 50 步做一次验证，通常 600-1000 步之间达到最佳效果。超过 1500 步后 FID 不降反升（过拟合）

28.4 ControlNet 接入

室内设计的核心需求是结构可控——用户希望框定布局，然后让 AI "填充"设计细节。我们使用 ControlNet 的 depth 模型来实现：

# 使用 ControlNet 进行深度图引导的室内设计生成
import torch
from diffusers import StableDiffusionXLControlNetPipeline, ControlNetModel
from diffusers.utils import load_image

# 加载 ControlNet（depth 模型）
controlnet = ControlNetModel.from_pretrained(
    "diffusers/controlnet-depth-sdxl-1.0",
    torch_dtype=torch.float16
)

# 组合主模型 + ControlNet
pipe = StableDiffusionXLControlNetPipeline.from_pretrained(
    "stabilityai/stable-diffusion-xl-base-1.0",
    controlnet=controlnet,
    torch_dtype=torch.float16
).to("cuda")

# 加载 LoRA 权重
pipe.load_lora_weights("./lora-interior-sdxl")

# 推理：输入 prompt + 深度图
prompt = "现代风格客厅, 浅灰色沙发, 木地板, 落地窗"
depth_image = load_image("user_floorplan.png")  # 用户提供的户型深度图

image = pipe(
    prompt=prompt,
    image=depth_image,
    controlnet_conditioning_scale=0.8,
    num_inference_steps=25,
    guidance_scale=7.5
).images[0]

image.save("generated_interior.png")

ControlNet conditioning scale 的调优经验：0.6-0.8 之间是室内设计的最佳范围。太低（<0.5）则结构完全不受控，太高（>1.0）则模型过度拟合深度图，导致纹理细节丢失。

⚠️ 一个真实的踩坑故事：项目上线前一周，发现 ControlNet 生成的图像中频繁出现"悬浮家具"——沙发离地 20cm，餐桌浮在半空。排查了两天才发现原因：深度图的归一化方式不一致！我们的控制管线的深度图预处理脚本用了 [0, 1] 归一化，但 ControlNet 期望的是 [-1, 1] 归一化。修复这个仅一行代码的 bug 后，所有家具都稳稳落地了。ControlNet 输入格式的坑是所有可控生成项目中最容易被忽略的细节。

28.5 API 服务搭建

后端使用 FastAPI + Diffusers + async queue 实现：

# api.py — 文生图 API 服务核心
from fastapi import FastAPI, BackgroundTasks
from pydantic import BaseModel
import asyncio
from diffusers import StableDiffusionXLControlNetPipeline
import torch

app = FastAPI()
request_queue = asyncio.Queue(maxsize=32)

# 模型热加载（启动时预热，避免首次请求慢）
class ModelManager:
    def __init__(self):
        self.pipe = None

    async def warm_up(self):
        self.pipe = StableDiffusionXLControlNetPipeline.from_pretrained(
            "stabilityai/stable-diffusion-xl-base-1.0",
            torch_dtype=torch.float16
        ).to("cuda")
        self.pipe.load_lora_weights("./lora-interior-sdxl")
        # 预热：一次空推理
        _ = self.pipe("warmup", num_inference_steps=2)

model_mgr = ModelManager()

class GenerateRequest(BaseModel):
    prompt: str
    depth_image_url: str = None
    num_steps: int = 25
    guidance_scale: float = 7.5
    controlnet_scale: float = 0.8

class GenerateResponse(BaseModel):
    image_url: str
    inference_time_ms: float

@app.on_event("startup")
async def startup():
    await model_mgr.warm_up()

@app.post("/generate", response_model=GenerateResponse)
async def generate(req: GenerateRequest, background: BackgroundTasks):
    # 异步队列处理，防止并发 OOM
    future = asyncio.get_event_loop().create_future()
    await request_queue.put((req, future))
    result = await future
    return result

这个架构的关键设计点：

  • async queue 限制并发数（maxsize=32），防止多个生成请求同时加载模型导致 OOM
  • 模型热加载：应用启动时做一次空推理，确保首次用户请求不受模型加载延迟影响
  • 单卡单模型：不动态切换 LoRA，每个部署实例只加载一个 LoRA，通过水平扩展支持多风格

28.6 部署与监控

部署方案采用 Docker + NVIDIA 容器工具包：

组件选型说明
GPU 实例A10G (24GB) 或 RTX 4090单实例支撑 SDXL 推理
模型部署Docker + Triton Inference Server支持动态批处理 (dynamic batching)
API GatewayKong / Nginx请求路由、速率限制、认证
监控Prometheus + Grafana延迟、QPS、显存使用、生成质量
CI/CDGitHub Actions + ArgoCD模型更新、LoRA 热更新

性能基准测试结果（基于 RTX 4090）：

  • 单图生成（25 步，1024×1024）：约 1.8 秒
  • 包含 ControlNet 推理：约 2.2 秒
  • 单实例最大并发：2 个请求（超过则 OOM）
  • 日处理能力（24h）：约 78000 次生成（估算）

A/B 测试上线策略：

  • 第一阶段（1 周）：5% 流量，人工审核每张生成结果
  • 第二阶段（1 周）：30% 流量，自动质量监控（CLIP score + 美学评分阈值）
  • 第三阶段：全量上线，设置 10% 的金丝雀流量持续监控
💡 成本估算（以 A10G 按需实例为例）： 单实例月费约 $800-1200（云厂商差异较大），单张图生成成本约 $0.003-0.005。如果每天 10000 次生成，月 GPU 成本约 $900-1500，加上存储、带宽、API Gateway 等辅助成本，总月成本约 $1200-2000。相比调用 Midjourney API（$0.04/张），自建方案在 10000 张/天量级上成本优势约 8-10 倍。
