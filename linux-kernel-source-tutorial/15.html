大家好，我是你们的讲师。从本章开始，我们将进入一个全新的世界——I/O与文件系统。前14章我们完成了进程管理、内存管理两大核心模块，从今天起，我们将探索内核如何与磁盘打交道：从最低层的缓冲区管理，到完整的MINIX文件系统，再到管道和exec。这一大块是Linux 0.11中代码量最密集的部分，也是理解"一切皆文件"哲学的根本所在。

引言——磁盘I/O的瓶颈

在计算机体系结构中，CPU和内存可以在一纳秒到几十纳秒内完成数据读写，但磁盘呢？即使是今天最快的NVMe SSD，一次随机I/O的延迟也在几微秒级别——比内存慢上千倍。而在Linux 0.11诞生的1991年，主流硬盘还是IDE接口的旋转磁盘，一次寻道加旋转延迟需要十几毫秒，比当时的CPU时钟周期慢了上千万倍。

这个巨大的速度鸿沟，就是缓冲区存在的根本原因。试想一下：如果一个用户程序每次读取一个字节，内核都要启动一次磁盘I/O，那系统会慢到完全不可用。缓冲区的作用就是——让磁盘I/O的次数尽可能少，让每次I/O传输的数据块尽可能大。

核心思想：缓冲区是位于内存中的一块区域，用于缓存磁盘块的数据。当进程请求读某个磁盘块时，内核先检查缓冲区中是否有该块的副本——如果有（称为"命中"），就直接返回；如果没有，才发起真正的磁盘I/O。这种"先查缓存"的策略，是计算机系统中最重要的性能优化手段之一。

缓冲区设计目标

Linux 0.11的缓冲区管理模块（代码在 fs/buffer.c 中）设计目标非常明确：

第一，减少磁盘访问次数。这是最核心的目标。通过缓存最近使用过的磁盘块，让后续的读操作可以直接从内存获取数据。根据程序的时间局部性原理（一个被访问的地址在不久的将来很可能会被再次访问），这个策略能够覆盖绝大多数场景。

第二，提供统一的块I/O接口。文件系统、设备驱动等上层模块不需要关心磁盘的具体细节，只需要通过bread()和bwrite()这样的标准函数来读写块数据。缓冲区作为中间层，封装了所有的底层操作。

第三，实现预读和延迟写策略。预读（read-ahead）是指当程序读取某个磁盘块时，内核主动将相邻的几个块也一并读入缓冲区——因为程序很可能会继续读取后面的数据。延迟写（delayed write）是指写操作不立即同步到磁盘，而是先写入缓冲区，标记为"脏"，等后续合适的时机再统一刷入磁盘——这样可以将多次小写入合并为一次大写入。

第四，缓存淘汰策略——LRU。当缓冲区用满时，需要淘汰（丢弃）一个旧缓冲区来给新数据腾位置。Linux 0.11使用了最经典的LRU（Least Recently Used，最近最少使用）策略：最近被访问过的缓冲区放在链表头部，长时间未被访问的逐渐移到尾部；淘汰时从尾部取一个缓冲区释放掉。

LRU与时钟算法：现代Linux内核使用的并不是简单的LRU链表，而是改进的"时钟算法"（Clock Algorithm，也称为"第二次机会"算法）。它是对LRU的一种近似实现，开销更小，而且能避免一次性扫描大量页面导致的性能抖动。不过，LRU作为教学来说是最直观的——理解LRU，再理解时钟算法就是一层窗户纸的事。

buffer_head数据结构

Linux 0.11中管理每个缓冲区的核心数据结构是struct buffer_head，定义在 include/linux/fs.h 中。让我们逐字段解读这个关键结构体：

struct buffer_head {
    char *b_data;            /* 指向数据块的指针（内存中的4KB缓冲区） */
    unsigned long b_blocknr; /* 块号（逻辑块号，从0开始） */
    unsigned short b_dev;    /* 设备号（主设备号+次设备号，0表示空闲） */
    unsigned char b_uptodate;/* 更新标志（1=数据有效，0=无效） */
    unsigned char b_dirt;    /* 脏标志（1=数据已被修改，尚未写回磁盘） */
    unsigned char b_lock;    /* 锁标志（1=正在传输，禁止操作） */
    unsigned char b_req;     /* 请求标志（1=已发出读写请求） */
    struct task_struct *b_wait; /* 等待该缓冲区的进程队列头 */
    struct buffer_head *b_prev; /* 前驱指针（双向链表用） */
    struct buffer_head *b_next; /* 后继指针（双向链表用） */
};

这个结构体本身并不存储数据——数据是通过b_data指针指向的一块4KB内存区域来存储的。结构体中的标志位（b_uptodate、b_dirt、b_lock）共同决定了缓冲区的状态机转换。

b_dev和b_blocknr的组合构成了全局唯一的"块标识"——这就好比一本书的"书架号+书编号"：有了这两个值，内核可以唯一地定位到任何一个磁盘块对应的缓冲区。当进程需要读取(dev=0x0302, block=100)时，内核首先通过Hash表查找是否已经有对应的缓冲区。

字段大小作用典型值
b_data4字节（指针）指向实际数据缓冲区0x1FFFFC00
b_blocknr4字节磁盘块号0~65535
b_dev2字节设备号，0=空闲0x0302（/dev/hda2）
b_uptodate1字节数据是否有效0或1
b_dirt1字节数据是否被修改0或1
b_lock1字节是否正在I/O0或1
b_wait4字节（指针）等待队列NULL或task_struct*

注意这些标志位的使用规范：b_uptodate表示缓冲区中的"数据副本"是否是磁盘上对应块的"最新版本"——刚完成读操作时它被置为1；b_dirt表示数据已经被修改但尚未写回磁盘——这对应延迟写策略。两者可以同时为1，表示"缓冲区中有最新的数据，但磁盘上的副本已过时"。

缓冲区池初始化

缓冲区并非由内核动态分配——在Linux 0.11中，缓冲区池是在系统初始化时一锤子定下来的。让我们看看 buffer_init() 函数做了什么。

Linux 0.11的内存布局大致如下：

物理内存分布（以16MB系统为例）：

        0x000000 ┌────────────────┐
                │ BIOS + 中断向量 │  1KB
        0x09FFFF │       ...       │
                ├────────────────┤
        0x100000 │      内核      │  Linux 0.11内核代码(大约120KB)
                │ (kernel code)   │
        0x12XXXX ├────────────────┤
                │                │
                │  缓冲区池      │  buffer_init() 从内核结束处开始
                │ (buffer cache) │  到主内存区起始处为止
                │                │
                │    ......      │
                ├────────────────┤
                │  主内存区      │  给用户进程使用的物理页面
                │ (main memory)  │
        0xFFFFFF └────────────────┘

buffer_init()的代码逻辑如下：

// fs/buffer.c - 简化后的初始化逻辑

        void buffer_init(long buffer_end)
        {
            struct buffer_head *h = start;
            void *d;
            int nr;

            // start = 内核代码结束位置（对齐到4KB边界）
            // 从start到buffer_end之间的全部内存都用作缓冲区
            for (; h < (struct buffer_head *) buffer_end; h++) {
                // 每个buffer_head结构体占据约32字节
                // 紧随其后的就是数据区（4KB对齐）
                d = (void *) (BLOCK_SIZE + (char *) h);
                h->b_dev = 0;        /* 空闲 */
                h->b_dirt = 0;
                h->b_uptodate = 0;
                h->b_lock = 0;
                h->b_data = d;       /* 指向数据区 */
                h->b_prev = NULL;
                h->b_next = NULL;
                nr = h - start;
                // 插入空闲链表尾部（LRU位置）
                insert_into_free_list(h);
            }
            nr_buffers = nr;        /* 记录总缓冲区数量 */
        }

这段代码的关键点在于：每个buffer_head结构体和它的数据区是紧挨着存放的，结构体占用大约32字节，数据区占用4,096字节（一个磁盘块的大小）。这种方式简单高效，没有额外的内存分配开销。系统中总的缓冲区数量取决于内核代码结束位置到主内存起始位置之间的可用空间大小。

初始化的最终结果是：所有缓冲区的b_dev都被置为0（空闲），并且被串联到一个空闲链表（free list）中。空闲链表是一个双向链表，链表头部是最近使用过的缓冲区，尾部是最久未使用的缓冲区——这恰好就是LRU淘汰策略的数据基础。

getblk()/brelse()——缓冲区分配与释放

理解了数据结构和初始化之后，我们来分析缓冲区管理中最核心的两个函数：getblk()（分配缓冲区）和brelse()（释放缓冲区）。

getblk()——缓冲区分配

当文件系统需要读一个磁盘块时，首先要做的不是发起I/O，而是调用getblk()——"给我一个能代表这个磁盘块的缓冲区"。它的完整逻辑如下：

struct buffer_head * getblk(int dev, int block)
        {
            struct buffer_head * bh;

            // 第一步：在Hash表中查找
            // 如果找到了(dev,block)对应的缓冲区且数据有效(b_uptodate=1)
            // 直接返回，不需要任何I/O
            if (bh = get_hash_table(dev, block))
                return bh;

            // 第二步：Hash表未命中，从空闲链表中取一个
            bh = get_free_buffer();

            // 第三步：更新Hash表
            // 从旧的Hash桶中移除，插入到新的(dev,block)桶中
            remove_from_hash_queue(bh);
            bh->b_dev = dev;
            bh->b_blocknr = block;
            insert_into_hash_queue(bh);

            // 此时数据无效(b_uptodate=0)，调用者需要发起读盘
            return bh;
        }

get_hash_table()使用了Hash表来加速查找——将dev ^ block作为Hash键，这样无论缓冲区池中有多少个缓冲区，查找都是常数时间O(1)。如果不使用Hash表而在空闲链表上逐个遍历，查询效率将是O(N)。

注意：get_free_buffer()在从空闲链表取缓冲区时，遵循的是从尾部取（即最久未使用的缓冲区）。但如果尾部缓冲区是"脏"的（b_dirt=1），需要先将它写回磁盘，等待写完成，再取下一个。这个"等待脏缓冲区刷盘"的过程会导致调用进程睡眠——这是I/O密集型场景下进程阻塞的一个重要原因。

brelse()——缓冲区释放

当文件系统使用完缓冲区后，调用brelse()释放：

void brelse(struct buffer_head * buf)
        {
            if (!buf) return;          /* 安全保护：空指针直接返回 */

            // 等待缓冲区上的I/O完成
            wait_on_buffer(buf);

            // 如果有进程在等待该缓冲区，唤醒它们
            if (buf->b_wait)
                wake_up(&buf->b_wait);

            // 将该缓冲区放回空闲链表尾部（LRU位置）
            // 尾部意味着"刚刚被释放"，所以它是最新的LRU候选
            insert_into_free_list(buf);
        }

请注意：释放缓冲区 ≠ 丢弃数据。释放缓冲区只是把buffer_head放回空闲链表，但它的Hash表项仍然存在——也就是说，如果另一个进程立即请求同一个磁盘块，通过Hash查找仍然可以命中这个缓冲区，即使它在空闲链表中。只有在该缓冲区被重新分配给另一个不同的磁盘块后，原来的数据才会失效。

bread()/bwrite()——块读写

有了getblk()和brelse()这两个底层函数，块读写接口就变得非常简洁。

bread()——读块

struct buffer_head * bread(int dev, int block)
        {
            struct buffer_head * bh;

            // 第一步：获取缓冲区
            bh = getblk(dev, block);

            // 第二步：如果数据有效，直接返回（缓存命中）
            if (bh->b_uptodate)
                return bh;

            // 第三步：发起磁盘I/O读
            ll_rw_block(READ, bh);

            // 第四步：等待I/O完成
            wait_on_buffer(bh);

            // 第五步：如果I/O成功，返回；否则报错
            if (bh->b_uptodate)
                return bh;

            brelse(bh);  // I/O失败，释放缓冲区
            return NULL;
        }

这个函数的流程体现了缓冲区管理的核心价值：对上层模块来说，我只是调用了一个bread(dev, block)，然后拿到了一块有效数据。至于这个数据是从缓存中直接取出的，还是从磁盘上读取的，上层模块完全不需要关心。

bwrite()——写块

void bwrite(struct buffer_head * bh)
        {
            // 标记为脏——这就是"延迟写"
            // 数据没有被立即写到磁盘，只是标记
            bh->b_dirt = 1;
        }

你没看错——Linux 0.11的bwrite()就这么简单。它什么都不写，只是把缓冲区标记为"脏"。实际的磁盘写入发生在以下两种情况下：

  • 缓冲区被重新分配时：当get_free_buffer()选中一个脏缓冲区来分配给新的磁盘块时，它必须先将脏数据写回磁盘。
  • 系统调用sync()时：用户或进程可以主动调用sync()，内核会遍历所有缓冲区，将脏缓冲区全部写回磁盘。

这种"延迟写"策略极大地减少了磁盘I/O次数。试想一个场景：用户在一个文件中写入1,000次（比如每写入一个字符调用一次write()），如果每次写入都要刷盘，就需要1,000次磁盘I/O。但有了延迟写，前999次都只是在内存中修改缓冲区，只有第1,000次（或缓冲区被回收时）才真正写入磁盘——减少了1,000倍的I/O次数。

sync()的同步保证：延迟写在提升性能的同时也带来了风险——如果系统在脏数据写回磁盘之前崩溃，数据就会丢失。这就是为什么数据库等需要强一致性的应用会调用fsync()来强制刷盘。Linux 0.11的update进程（现代内核中的pdflush/kworker）会每隔30秒自动调用sync()，确保脏数据不会在内存中停留太久。

踩坑故事——缓冲区泄漏导致的系统卡死

让我讲一个真实发生的案例。当年我们团队在做一个嵌入式Linux项目，内核版本基于2.4移植到一块ARM开发板上。系统启动后运行一段时间，就会突然卡死——确切地说，是文件系统操作全部停摆，ls命令挂住，cat一个文件也卡住不动，但内核还在运行（可以通过网络ping通）。

排查了整整两天，最后用printk插桩定位到问题：缓冲区泄漏了。

问题的根源是一个块设备驱动中的bug。驱动在处理某些错误路径时，调用了bread()获取了一个缓冲区，发现数据有问题后就直接返回了——但忘记调用brelse()。这个"被遗忘"的缓冲区就一直挂在已分配链表上，永远不被释放。

每次触发这个错误路径，系统就泄漏一个缓冲区。内核的缓冲区池是有上限的（假设nr_buffers = 512），泄漏到第513个时，getblk()中的get_free_buffer()遍历整个空闲链表都找不到可用的缓冲区——因为所有缓冲区都被"永久占用"了。系统陷入死锁：所有需要读盘的操作都在等一个可用缓冲区，而没有任何代码会释放缓冲区。

这个bug的教训是深刻的：在操作系统开发中，资源管理必须成对出现。申请了资源就要记得释放，就像malloc的对称是free，open的对称是close。在Linux 0.11的源码中，你可以看到每一对getblk()/brelse()都是严格配对的——Linus写代码的时候在这方面极其严谨。
