第5章 SFT + DPO 对齐

大家好，欢迎回到 DeepSeek 系列教程。前几章我们聊了模型架构、词表设计和数据管线——这些都是构建一个强大基座模型的关键组件。但有一个关键问题：预训练模型不等于好用的聊天助手。

想象一下，你花了几百万美金训练了一个 67B 参数的基座模型，它知道所有知识，但你对它说"帮我写一封工作邮件"，它可能回复你一段洋洋洒洒的论文摘要——因为它学到的训练数据来自互联网，这些数据中什么都有，但偏偏没有"听用户指令并照做"的模式。这就是为什么我们需要对齐（Alignment）。

本章我们将深入 DeepSeek LLM 的对齐策略：从 SFT（Supervised Fine-Tuning）的数据构建和两阶段微调方法论，到 DPO（Direct Preference Optimization）的原理、实现、以及与经典 RLHF 的对比分析。

5.1 SFT 数据构建

SFT 是基座模型变成助手模型的第一步。它的本质是：让模型学会"理解指令-生成回复"这个行为模式。预训练阶段的模型输入是"世界上最高的山是珠穆朗玛峰"，输出是预测下一个 token。SFT 阶段的输入变成了"用户：世界上最高的山是什么？\n助手："，输出是"世界上最高的山是珠穆朗玛峰"。

SFT 数据的质量直接决定了模型能"学"到什么。DeepSeek 的 SFT 数据来自两个渠道：

人工标注数据：招募专业的标注团队，按照详细的标注规范编写指令 - 回复对。人工标注的质量最高——标注员会注意回复的逻辑性、准确性、安全性和表达的自然度。但人工标注的成本极高，一条高质量的数据可能需要 5-10 分钟来完成，成本在几元到几十元人民币不等。

合成数据（Synthetic Data）：用已有的强模型（如 GPT-4）生成指令和回复，再由人工审核和筛选。合成数据的优势是规模大、成本低——可以用 API 批量生成。但合成数据可能存在偏差（比如"GPT 味"太重），而且如果合成数据本身有事实错误，模型会学到错误的模式。

DeepSeek 团队采用了一个混合策略：核心能力（数学推理、代码、安全性）以人工标注为主，保证质量；通用场景（翻译、写作、问答）以合成为主，保证覆盖度和规模。

一个值得思考的问题：合成数据是否会形成"模型近亲繁殖"？如果所有模型都用 GPT-4 的半合成数据做 SFT，大家学到的风格和知识边界会趋同。DeepSeek 的观点是：合成数据适合做"广度"，让模型接触到各种类型的指令；但"深度"——特别是数学推理、逻辑分析——必须靠人工标注的高质量数据来支撑。这有点像吃饭：合成数据是主食（填饱肚子），人工数据是营养补剂（关键微量元素）。

5.2 两阶段微调策略

我们在第 2 章简要提到了 DeepSeek 的两阶段微调策略，现在我们来深入展开为什么两个阶段优于一次到位。

第一阶段：全科学习。这个阶段的目标是"能力覆盖"——让模型在所有类型的任务上都达到基本可用的水平。数据涵盖了数学、代码、问答、翻译、摘要、写作、分析等数十个类别。每个类别包含数百到数千条高质量样本。

第一阶段的训练超参数比较保守：较小的学习率（通常是预训练最终学习率的 1/10），较少的训练步数（通常 1-3 个 epoch）。原因很简单——SFT 阶段要防止过拟合。基座模型已经学到了丰富的知识，SFT 只需要教会它"如何输出"，而不是重新学习"输出什么"。

第二阶段：对话精调。在第一阶段的基础上，使用专门的多轮对话数据继续微调。这个阶段的数据有几个特点：

  • 多轮交互——样本包含 3-10 轮对话历史，训练模型理解对话上下文
  • 风格多样——从正式到口语化，从简洁到详细，让模型学会根据场景调整表达
  • 纠错场景——包含"用户指出错误→模型承认并改正"的样本，训练模型的纠错能力
  • 拒绝不当请求——包含模型礼貌拒绝有害请求的样本

让我们用一个具体的对比来说明两阶段 vs 单阶段的效果差异：

用户输入: "写一首关于秋天的短诗"

单阶段微调（数学+对话混合）:
"让我们从诗歌理论开始分析。秋天作为古典诗歌的重要意象，
通常包含以下元素：落叶、西风、霜天、归雁...
根据中国古典诗歌的格律要求，五言绝句应遵循...
因此，我给出如下分析：...（500字理论分析后附了一首短诗）"

两阶段微调:
"秋风起兮白云飞，草木黄落兮雁南归。
一片落叶知秋至，半盏清茶送晚凉。
——送你一首轻快的秋日小诗。"

可以看出，单阶段微调中模型保留了过多的"推理链"习惯——它把写诗当作一道数学题来"推导"。两阶段微调通过两次分离的优化，让模型学会了在不同的任务上切换不同的行为模式。数学题就展示推理过程，写诗就直接写诗。

技术原因分析：这个问题的根本原因在于 Transformer 的注意力机制是"位置敏感"的。在数学推理数据中，模型学会了"先分析再回答"的位置模式——即前几个 token 应该是分析，后面的才应该是答案。当这种模式被过度泛化到所有任务上，就会出现"写诗先写理论"的奇怪行为。两阶段微调实际上是在告诉模型："对不同的输入前缀（指令），使用不同的输出风格。"第一阶段建立基本的"指令→回复"映射，第二阶段通过对齐数据修正和细化这个映射。

5.3 对齐的必要性

在讨论 DPO 之前，我们需要先理解为什么 SFT 本身不够，为什么需要额外的对齐步骤。

SFT 是"模仿学习"——模型从标注数据中学习如何回复。但标注数据有两个固有限制：

  • 覆盖度有限——你不可能为所有可能的指令都标注一个"完美回复"
  • 缺乏偏好信息——SFT 数据只有"好"的回复，没有"哪些回复更好"的对比信息

这就导致：SFT 后的模型知道"怎么做"，但不知道"怎么做更好"。比如对于"如何学习编程"这个问题，一个勉强可用的回答和一个真正有用的回答之间的差异，SFT 无法充分捕获。

对齐（Alignment）的目的就是让模型的输出更加符合人类的价值观、偏好和期望。这包括三个维度：

  • 有用性（Helpfulness）——回答是否真正解决了用户的问题
  • 诚实性（Honesty）——模型是否清楚表达了自己的不确定性，不编造事实
  • 安全性（Safety）——模型是否拒绝协助有害、非法、不道德的行为

这三个目标有时是冲突的。一个"有用"的回答可能意味着需要提供详细信息，但在某些场景下这些信息可能被滥用。一个"安全"的回答可能过于保守，拒绝回答很多正常问题。对齐的挑战就在于如何在三者之间找到平衡。

5.4 DPO 原理与实现

DPO（Direct Preference Optimization）由斯坦福大学的研究者在 2023 年提出，DeepSeek 是最早将其大规模应用到生产级模型的团队之一。

DPO 的核心思想可以用一句话概括：用偏好数据直接优化策略模型，无需显式训练奖励模型。

让我们一步步拆解 DPO 的数学原理：

Step 1: 准备偏好数据
对每个 prompt x，收集两个回答 y_w (偏好) 和 y_l (不偏好)
数据集 D = {(x_i, y_w_i, y_l_i)}

Step 2: DPO 损失函数
L_DPO(π_θ) = -E_{(x,y_w,y_l)~D}[log σ(β * r_θ(x,y_w) - β * r_θ(x,y_l))]

其中隐式奖励函数:
r_θ(x,y) = β * log(π_θ(y|x) / π_ref(y|x))

π_θ: 当前策略模型（我们要优化的）
π_ref: 参考模型（SFT 完成后的模型，冻结不变）
β: 控制 KL 散度约束的强度

直觉解释:
- 我们希望 r_θ(x,y_w) > r_θ(x,y_l)
- 即 π_θ(y_w|x) / π_ref(y_w|x) > π_θ(y_l|x) / π_ref(y_l|x)
- 也就是: 偏好回答的概率相对上升，不偏好回答的相对下降
- β 控制"能偏离基座模型多远"

DPO 的训练流程也非常直接——它几乎就是标准的语言模型微调，只是损失函数变成了上面的形式：

DPO 训练伪代码:

for each batch {(x, y_w, y_l)} in D:
    # 当前模型的 log-probability
    π_θ_y_w = model(x, y_w).log_prob
    π_θ_y_l = model(x, y_l).log_prob

    # 参考模型的 log-probability（不计算梯度）
    with torch.no_grad():
        π_ref_y_w = ref_model(x, y_w).log_prob
        π_ref_y_l = ref_model(x, y_l).log_prob

    # 隐式奖励
    r_w = β * (π_θ_y_w - π_ref_y_w)
    r_l = β * (π_θ_y_l - π_ref_y_l)

    # DPO 损失
    loss = -log(sigmoid(r_w - r_l))

    loss.backward()
    optimizer.step()

对比一下 RLHF 的流程：训练奖励模型（需要额外的标注数据）→ 用 PPO 优化策略（需要 4 个模型共同训练）→ 复杂的 KL 散度约束和 reward scaling。DPO 的优雅之处在于，它将 RLHF 中的"先拟合奖励、再优化策略"的两步过程，简化成了"直接优化策略"的一步过程。

实现陷阱：DPO 看似简单，但在实际实现时有几个容易踩坑的地方。第一，参考模型 (π_ref) 在训练过程中绝对不能更新——这个模型必须严格冻结在 SFT 完成时的状态。如果忘记冻结或使用了错误的 checkpoint，KL 约束会失效，模型可能"跑偏"。第二，β 参数的选择非常关键——β 太大，模型几乎不变（和 SFT 一样）；β 太小，模型过度优化偏好数据，可能损害通用能力。DeepSeek 的实践中，β 在 0.1-0.5 之间是一个合理的范围。

5.5 DPO vs RLHF

DPO 和 RLHF 是两种不同的对齐方法论，各有优劣。下面是 DeepSeek 在实践中总结的对比：

维度 RLHF (PPO) DPO
训练流程 3 步：训练奖励模型 → PPO 训练策略 → 迭代 1 步：直接优化偏好数据
模型数量 策略模型 + 奖励模型 + 价值模型 + 参考模型 = 4 策略模型 + 参考模型 = 2
训练稳定性 差：奖励 hack、策略崩溃、KL 爆炸 好：接近标准微调的训练曲线
超参数复杂度 高：学习率、KL 系数、GAE λ、clip 范围... 低：主要就是 β
内存占用 ~4 倍模型权重（4 个模型推理或训练） ~2 倍模型权重（策略 + 参考）
数据效率 中等：奖励模型可以利用无标注 prompt 做探索 低-中：严格依赖标注好的偏好对
在线探索能力 有：PPO 可以在训练中采样新回复 无：只在固定偏好数据上优化

可以看到，DPO 在训练简洁性和稳定性上有明显优势，但并非完全优于 RLHF。RLHF 的奖励模型可以在训练过程中持续提供反馈信号，而 DPO 只能在静态数据集上优化，缺乏"探索"能力。

DeepSeek 的实际做法是：以 DPO 为主，必要时用 RLHF 补充。对于大多数场景，DPO 已经能带来足够的对齐效果，而且训练成本低、迭代速度快。对于特别敏感或需要细粒度控制的场景（如安全对齐），则使用 RLHF 的奖励模型来提供更精细的约束。

行业趋势：在 2024-2025 年间，DPO 及其变体（如 KTO、IPO、SimPO）在开源社区中迅速普及。主要原因很简单——DPO 让中小团队也能做对齐，而不需要搭建完整的 RLHF 管线。但 GPT-4、Claude 等顶级闭源模型据信仍然主要使用 RLHF 的变体，因为奖励模型提供了更大的"优化空间"。这不是"谁更好"的问题，而是"资源约束下的最优选择"。

5.6 对齐效果评估

对齐效果不能只看指标——因为对齐的核心目标是"更有用、更诚实、更安全"，而这些难以用单一的客观指标衡量。

DeepSeek 的对齐评估体系包含三个层次：

自动评测：使用标准评测集（如 MT-Bench、AlpacaEval）对模型进行自动化评估。这些评测集包含多轮对话、不同难度等级的指令，通过 GPT-4 打分或规则匹配来评分。优点是快速、可重复；缺点是 GPT-4 的评判本身可能有偏差。

人工评测：这是对齐评估的"金标准"。DeepSeek 团队会定期组织内部的盲测（A/B test），让标注员对比不同版本模型的输出质量。评估维度包括：

维度 评分标准 示例
准确性 回答是否正确，是否有事实错误 "中国首都是北京" vs "中国首都是南京"
完整性 是否全面回答了问题 解释算法时是否涵盖了关键步骤
安全性 是否拒绝有害请求 对"如何制作炸弹"是否拒绝
表达自然度 语言是否流畅、自然、适合场景 口语 vs 书面语的场景适配
一致性 多轮对话中是否保持观点一致 同一主题前后回答不矛盾

红队测试（Red Teaming）：这是安全对齐最重要的环节。专门的安全研究团队会尝试用各种攻击手段诱导模型生成有害内容——包括越狱（jailbreak）、角色扮演诱导、多轮渐进式诱导等。红队测试的结果直接反馈到对齐数据的优化中，形成"攻击→防御→更强攻击→更强防御"的迭代循环。

一个有趣的发现：DeepSeek 在论文中分享了这样一个观察——DPO 对齐后的模型在安全性评测上通常比 RLHF 模型更"坚定"。原因可能是 DPO 直接优化了"偏好 vs 不偏好"的边界，而 RLHF 的奖励模型可能会在边界附近产生模糊的奖励分数，导致模型在某些边缘情况下做出不确定的选择。当然，这也意味着 DPO 模型可能更"刻板"一些——但这在安全场景下不一定是坏事。

经过 SFT + DPO 的对齐管线，DeepSeek LLM 67B Chat 在 MT-Bench 上的得分从基座模型的 4.5 提升到了 7.2（满分 10），在安全性评测上的有害回复率从 12% 降低到了 2% 以下。这说明对齐不是锦上添花，而是从"能用"到"好用"的决定性一步。

回顾这一章，我们深入探讨了 DeepSeek 的对齐策略：SFT 数据构建和两阶段微调的核心思路、DPO 的数学原理和工程实现、DPO 与 RLHF 的全面对比分析。对齐是让大模型真正"有用"的关键环节，也是 DeepSeek 从学术研究走向产品化的重要一步。

但 DeepSeek LLM 67B 仍然是一个稠密模型——所有参数在每次推理时都被激活。想要用更少的计算量获得更大的模型容量，我们需要换一条技术路线。下一章，我们将进入 MoE（Mixture of Experts）架构 的世界，这是 DeepSeek 后续所有突破性工作的基础。敬请期待。
