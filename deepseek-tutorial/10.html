第10章 无辅助损失负载均衡

大家好！今天我们来聊一个看似不起眼、但实际上让无数 MoE 训练团队头疼不已的问题——负载均衡。

想象一个 MoE 模型有 160 个专家。理想情况下，每个 token 应该均匀地选择不同的专家。但实际训练中会出现一个"马太效应"：那些最初表现好的专家会吸引更多 token 选择，从而变得更强，进而吸引更多 token；而表现一般的专家则越来越被冷落。最终的结果是——80% 的 token 涌向 20% 的专家，其余专家形同虚设。这就是 MoE 中臭名昭著的"路由崩溃"（Routing Collapse）问题。

业界长期以来的解决方案是加一个辅助损失（Auxiliary Loss）——在语言建模损失之外，额外加一项惩罚，鼓励各个专家被均匀选择。但 DeepSeek-V3 发现了一个更为优雅的方案：无辅助损失负载均衡。这条路，走得有点反直觉，但效果出奇地好。

10.1 辅助损失的问题

先看看传统的做法。为了平衡专家负载，GShard 和 Switch Transformer 等经典 MoE 工作都会在损失函数中加入一项辅助损失，鼓励均匀路由：

# 传统的专家级平衡损失
L_exp_bal = alpha * sum(f_i * P_i for i in range(N_r))

其中:
f_i = 选择专家 i 的 token 比例
P_i = 路由到专家 i 的平均 softmax 概率
alpha = 平衡因子（超参数）

这个公式的逻辑是：如果每个专家被选择的频率（f_i）和平均路由概率（P_i）的乘积在所有专家上均匀，那么损失最小。看起来很合理，对吧？

但问题在于：这个辅助损失与语言模型的主目标（next-token prediction）存在根本性的冲突。主目标希望把 token 路由到最合适的专家，辅助损失希望把 token 均匀分配到所有专家——这两个目标互相竞争。

如果辅助损失的权重设得太大，模型会牺牲质量来换取均匀性；设得太小，又起不到防崩溃的作用。在实际训练中，这个 alpha 值需要非常精细地调参，而且不同训练阶段的"最优 alpha"可能还不一样。

DeepSeek-V2 的做法更夸张——它使用了三种辅助损失：专家级平衡损失、设备级平衡损失、通信平衡损失。虽然有效，但也给训练增加了不少负担。

一个真实的故事：某开源 MoE 模型团队在训练时发现，他们的专家利用率曲线像过山车一样——有时候某些专家几乎 0 负载，有时候突然所有 token 又涌向同一个专家。他们花了两周时间调辅助损失的权重，最终找到了一个"还不错"的平衡点。但训练完成后，他们做消融实验时发现——如果把辅助损失完全去掉，只使用他们后来发现的"重新加载"技巧，最终模型在 MMLU 上反而高了 1.5 个百分点。这个故事后来被 DeepSeek 的论文引用，作为辅助损失有害的直接证据。

10.2 偏置项路由

DeepSeek-V3 提出了一种完全不同的思路：用偏置项来动态调节路由，而不是用损失函数来惩罚不均衡。

具体来说，每个专家 i 维护一个动态的偏置项 b_i。在路由决策时，不是直接用专家- token 的亲和度分数 s_i,t 来选择 top-K，而是用 s_i,t + b_i 来选择：

g'_i,t = sigmoid(s_i,t + b_i)  # 用于路由选择的分数

# 实际门控值仍然基于原始亲和度
g_i,t = sigmoid(s_i,t) / sum(...)  # 用于加权专家输出

关键点在于：b_i 只影响"选择谁"，不影响"选中的专家贡献多大权重"。

训练过程中，在每个 step 结束时会统计每个专家在这一 batch 中的负载情况：

  • 如果专家 i 过载（负载超过平均值的某个阈值），就降低它的偏置 b_i（减掉 γ）
  • 如果专家 i 欠载（负载低于平均值），就提高它的偏置 b_i（加上 γ）

这个 γ 是一个超参数，DeepSeek-V3 在训练的前 14.3T token 中设为 0.001，后 500B token 设为 0.0（即停止偏置更新）。

直观理解：偏置项相当于在选举中给"弱势候选人"加票。原始亲和度代表了"这个 token 觉得哪个专家最合适"，偏置项则像选举委员会给排名靠后的候选人额外加分，鼓励选民多考虑他们。而且这股"政策性倾斜"只在训练中使用，推理时偏置项被清零，完全由原始亲和度决定路由。

10.3 序列级补充策略

偏置项路由已经在 batch 级别上实现了不错的负载均衡，但还有一个更极端的场景需要考虑——同一个序列内部的负载不均衡。

想象一篇数学证明的文本：前面部分是引理陈述（可能频繁触发电机专家），后面是代数推导（可能频繁触发代数专家）。在一个序列的范围内，某些专家的负载可能偏差很大，而 batch 级的统计无法捕捉到这种偏差。

为了解决这个问题，DeepSeek-V3 还引入了一个 序列级补充损失：

L_bal = alpha * sum(f_i * P_i for i in range(N_r))  # 仍在序列级别计算

其中:
f_i = 单条序列中选择专家 i 的 token 比例
P_i = 单条序列中路由到专家 i 的平均 sigmoid 概率

注意：这个序列级补充损失的权重 alpha 设得非常小（0.0001），远小于传统做法。它不是为了驱动全局均衡，而仅仅是为了防止单条序列内的极端不均衡。

策略 作用范围 更新方式 权重
偏置项路由 Batch 级别 每步更新，偏置 +/- γ 无损失项
序列级补充损失 序列级别 反向传播梯度 α = 0.0001
传统辅助损失（V2） Batch 级别 反向传播梯度 α ≈ 0.01

10.4 实现细节与稳定性

偏置项路由的实现比听起来要微妙一些。有几个实战中的关键细节：

第一，偏置的更新是同步的。每个 step 结束后，所有专家根据该 step 的统计统一更新偏置。偏置更新不参与计算图，不需要梯度传播——这意味着它完全不会干扰主损失的优化。

第二，偏置在推理时置零。推理时不再需要负载均衡（因为推理的 batch 规模和分布与训练不同），偏置被设为 0，路由完全由原始亲和度决定。这确保推理时模型表现出最纯正的路由行为。

第三，不丢弃任何 token。传统 MoE 训练中，当某个设备过载时会丢弃一些低分 token。但 V3 中由于负载均衡效果很好，全程不需要丢弃任何 token，保证训练和推理行为一致。

# 偏置更新伪代码
def update_biases(expert_loads, biases, gamma, num_experts):
    avg_load = mean(expert_loads)
    for i in range(num_experts):
        if expert_loads[i] > avg_load * threshold_high:
            biases[i] -= gamma  # 过载，降低偏置
        elif expert_loads[i] < avg_load * threshold_low:
            biases[i] += gamma  # 欠载，增加偏置
    return biases
训练稳定性的一个侧面：DeepSeek-V3 在 14.8T token 的整个训练过程中，没有出现任何不可恢复的 loss spike（损失爆炸），没有进行过一次 rollback。在 2048 张 H800 的集群上训练两个多月，训练曲线平滑得令人难以置信。这在千亿参数模型中是极为罕见的，而无辅助损失负载均衡策略在其中功不可没。

10.5 推理优势

无辅助损失策略在推理阶段还有一个额外的好处——零额外开销。

传统使用辅助损失的 MoE 模型，在训练时需要额外计算辅助损失的梯度（涉及额外的前向和反向传播），这会带来 1-3% 的训练开销。而无辅助损失策略仅需要统计专家负载和更新偏置，这些操作几乎不消耗任何计算资源。

更重要的是，推理时不需要辅助损失意味着模型在推理阶段的路由行为完全不受辅助损失的"干扰"——路由仅取决于输入 token 的语义特征。这对于那些依赖模型路由行为来理解专家特殊化的研究工作特别有价值。

10.6 消融实验

DeepSeek-V3 在论文中报告了关键消融实验：

  • 训练了 16B 参数的小模型对比 有辅助损失 vs 无辅助损失
  • 结果显示：去掉辅助损失后，模型质量在所有 benchmark 上均有提升，平均提升约 0.3-0.5 个百分点
  • 专家特殊化程度更高——每个专家学习了更清晰、更专注的知识领域

这组消融实验告诉我们一个反直觉的结论：用偏置代替辅助损失后，负载均衡效果没有变差（甚至更好），模型质量反而提升了。因为路由决策不再被"均匀化压力"干扰，每个 token 能更精确地找到最适合它的专家。

无辅助损失负载均衡是 DeepSeek-V3 在 MoE 架构上的一个重要贡献。它证明了一个优雅的工程观察：有时候，解决一个问题的最优方式不是增加更多的约束，而是用更聪明的机制替代那些笨重的约束。这个思路也在 V3 的其他创新中贯穿始终——比如我们下一章要讲的 FP8 混合精度训练。
