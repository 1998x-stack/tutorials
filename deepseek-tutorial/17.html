第17章 量化压缩

2025 年初，一位开源爱好者在 GitHub 上展示了这样一幕：他用一块 RTX 3090（24GB 显存），成功运行了 DeepSeek-V3——一个 671B 参数的模型。评论区炸了。有人质疑这是造假，有人惊呼魔法诞生，更多的人在问同一个问题："24GB 的卡怎么能装下 671B 的参数？"

答案就是 量化（Quantization）。如果原始 FP32 精度的 671B 模型需要约 2.5TB 显存，那么经过 4-bit 量化后，它只需要约 335GB。再加上 MoE 稀疏推理——每次只激活 37B 参数——配合 CPU + GPU 的 Offloading 策略，用 24GB 的消费级显卡跑起来就成了可能。

这一章，我们来系统地了解量化的原理、DeepSeek 的量化方案，以及社区中的各种量化工具。

17.1 量化基础

量化（Quantization）本质上就是 用更少的比特数来表示一个数值。在深度学习中，模型的参数和激活值通常以 FP32（32 位浮点数）或 FP16/BF16（16 位浮点数）存储。量化的目标是用 INT8（8 位整数）、INT4（4 位整数）甚至更低精度的格式来近似表示这些值。

为什么能这样做？因为神经网络对精度的要求远比想象中宽松。研究表明，模型的权重通常在 -1 到 1 之间分布，而且相邻权重之间存在很强的冗余性。去除一些精度信息不会显著影响模型的输出质量——只要不超出某个阈值。

量化的核心挑战在于 精度与效率的 trade-off。用更少的比特数意味着：

  • 优点：模型体积缩小（INT4 是 FP16 的 1/4），显存需求降低，推理速度提升（更少的数据从 HBM 传输到计算单元）
  • 缺点：数值精度下降，可能导致模型在某些任务上的性能衰减
精度格式 比特数 相对 FP32 体积 典型用途
FP32 32 1x (基准) 训练时的基准精度
FP16 / BF16 16 0.5x 训练、推理的标准格式
FP8 (E4M3 / E5M2) 8 0.25x 训练加速、高效推理
INT8 8 0.25x 通用推理优化
INT4 4 0.125x 极致压缩推理
注意：量化并非没有代价。随着比特数的减少，模型在某些任务上的表现会逐渐退化。关键是要找到"甜点"——在可接受的性能损失范围内最大化压缩比。对于不同的模型架构和任务，这个甜点各不相同。

17.2 FP8 推理实践

DeepSeek-V3 的一个重要设计决策是：模型原生支持 FP8 推理。这意味着不需要任何额外的量化后处理步骤——训练完成后，模型权重已经是 FP8 格式，可以直接用于推理。

在 V3 的训练中，DeepSeek 使用了 FP8 混合精度策略：前向传播以 FP8 计算，反向传播以 BF16 计算，优化器状态以 FP32 保存。训练完成后，模型权重导出为 FP8 格式。因此，用户在下载 V3 的权重时，获得的就是一个 FP8 精度的模型。

FP8 推理的具体优势：

  • 显存砍半：相比 FP16/BF16，FP8 将模型权重体积从 671B × 2 bytes ≈ 1.34 TB 降至 671B × 1 byte ≈ 671 GB
  • 带宽减半：每次读取参数的数据量减半，Decode 阶段的显存带宽瓶颈得到明显缓解
  • 计算加速：H800/H100 的 FP8 Tensor Core 峰值算力是 FP16 的两倍（1979 vs 989 TFLOPS），在 Prefill 阶段带来显著加速
# FP8 推理的显存需求示意
# 原始参数 (FP32): 671B × 4 bytes = 2.68 TB   ❌ 不可能
# 训练/推理 (BF16): 671B × 2 bytes = 1.34 TB   ❌ 太大
# FP8 模型权重:     671B × 1 byte  = 671 GB     ⚠️ 仍然大但合理
# MoE 激活参数:     37B  × 1 byte  = 37 GB      ✅ 单卡可行
关于 MoE 的显存计算：虽然激活参数只有 37B，但整个模型的所有专家参数都需要存储在（分布式）显存中，因为不同 token 可能路由到不同的专家。FP8 让你用 671 GB 存储所有参数，而不是 1.34 TB。对于多卡推理场景，这大幅降低了对 GPU 数量和网络带宽的需求。

17.3 FP4 QAT（量化感知训练）

FP8 已经是很大的进步，但 DeepSeek 并不满足于此。在 V4 的研究中，团队进一步探索了 FP4 量化——用 4 个比特（半字节）来表示一个模型参数。

FP4 的应用远比 FP8 困难。原因很简单：FP8 可以表示大约 240 个不同的值（加上符号位和指数位），而 FP4 只有 16 种可能的取值。用 16 个值来近似表达原始的 32 位浮点参数，信息损失是巨大的。

为了解决这个问题，DeepSeek 采用了 QAT（Quantization-Aware Training，量化感知训练）的方法。与 PTQ（Post-Training Quantization，训练后量化）只在训练结束后做量化不同，QAT 在训练过程中就模拟了量化效果：

  • 在前向传播中插入 fake quantization 操作，将权重和激活值先量化到 FP4 再反量化回 FP32 参与计算
  • 反向传播时，梯度通过 Straight-Through Estimator (STE) 避开量化函数的不可导点
  • 模型在训练过程中学会适应低精度表示，减少最终的精度损失
QAT vs PTQ 的类比：想象你平时穿皮鞋走路（FP32），现在要换一双木屐（FP4）。PTQ 相当于你到了会场门口突然换鞋，肯定会摔跤。QAT 则是在家里先穿着木屐练习走路，直到适应了再到会场上走——当然会慢一点，但不会摔跤。

DeepSeek 在 V4 中的 FP4 QAT 方案取得了令人瞩目的成果。根据他们的实验数据，FP4 量化后的模型在 MMLU 上的得分仅下降了 0.8 个百分点，而模型体积从 FP8 的 671GB 进一步下降到了 335GB。这意味着 4 张 H100 就可以完整加载整个 671B 模型（包括所有专家参数）进行推理。

17.4 社区量化方案

除了 DeepSeek 自身的 FP8 和 FP4 方案，开源社区也为 DeepSeek 模型（特别是 V2 和 V3）贡献了多种量化工具。以下是主流的几种方案对比：

方案 原理 支持精度 特点
AWQ 激活值感知的权重量化，保留 1% 的重要权重为高精度 INT4, INT3, INT2 质量损失极小，适合生产部署
GPTQ 基于 Hessian 矩阵的逐层量化，最小化量化误差 INT4, INT3, INT2 经典方案，生态成熟
GGUF llama.cpp 使用的量化格式，多种量化级别（Q4_K_M, Q5_K_M 等） INT8 ~ INT2 多种 CPU + GPU 混合推理，消费级友好
EXL2 ExLlamaV2 的量化格式，按层分配不同 bit 宽度 INT8 ~ INT2 多种 GPU 推理速度快

AWQ（Activation-aware Weight Quantization）

AWQ 的核心洞察是：并非所有权重通道对量化同样敏感。对于少数"重要"的权重通道（激活值较大的通道），如果将它们与其他通道一样量化到低精度，性能损失会很显著。AWQ 的做法是：将约 1% 的重要通道保留为 FP16 高精度，其余 99% 量化到 INT4。这种"不均匀"的量化策略在保持模型质量的同时实现了接近均匀 INT4 的压缩比。

GPTQ（Post-Training Quantization for GPTs）

GPTQ 是最早被广泛应用的 LLM 量化方法之一。它基于 Optimal Brain Quantization (OBQ) 框架，逐层进行量化：对于每一层的权重，GPTQ 逐个将权重从 FP16 量化到 INT4，并使用 Hessian 矩阵更新未量化的权重以补偿已量化权重的误差。这种方法在数学上很优雅，但量化过程需要一定的校准数据和计算时间。

GGUF

GGUF 是 llama.cpp 项目定义的模型格式，它定义了一系列标准化的量化级别（如 Q4_K_M, Q5_K_M 等）。GGUF 的量化通常使用 k-quant 方法，根据权重的分布动态调整量化分组大小。GGUF 最大的优势是支持 CPU 推理（通过 llama.cpp），使得没有高端 GPU 的用户也能运行大模型。

# GGUF 量化级别说明
# Q4_0:  4-bit, 对称量化, 32个权重一组     → 最快, 精度最低
# Q4_K_M: 4-bit, k-quant medium, 性能均衡   → 推荐选择
# Q5_K_M: 5-bit, k-quant medium             → 精度更好, 体积稍大
# Q6_K:  6-bit, k-quant full                → 精度接近 FP16
# Q8_0:  8-bit, 对称量化                    → 精度几乎无损

17.5 量化后性能评估

量化一定会带来精度损失，问题在于损失多少、在哪些任务上损失。以下是一个大致的量化精度损失对照：

量化精度 模型体积 (671B) MMLU 得分变化 HumanEval 得分变化 适用场景
FP8 (原生) ~671 GB 基本无损失 (~0%) 基本无损失 (~0%) 数据中心推理
INT4 (AWQ/GPTQ) ~335 GB 下降 1-2% 下降 2-4% 多卡部署
Q4_K_M (GGUF) ~370 GB 下降 2-3% 下降 3-5% 消费级 GPU + CPU 混合
Q3_K_M (GGUF) ~280 GB 下降 4-6% 下降 8-10% 极致压缩

需要注意的是，量化对不同类型任务的损伤是不均匀的：

  • 知识问答类（如 MMLU）受影响较小——统计记忆类任务对精度不敏感
  • 推理类（如数学、逻辑）受影响适中——数值精度影响计算链的准确性
  • 代码生成类（如 HumanEval）受影响较大——代码需要精确的语法和逻辑
一个常见的误解："我都量化到 INT4 了，模型应该和 FP16 一样好。"事实并非如此。量化是一个有损压缩过程，即使在最好的 QAT 方案下，INT4 模型的某些能力（特别是复杂推理）也会受到影响。在实际部署中，应该在目标任务的代表性样本上进行量化前后的对比评估，确认损失在可接受范围内。

17.6 GPU 选型指南

最后，我们来一张关于"跑 DeepSeek-V3 到底需要什么显卡"的实用指南。这里我们假设使用 MoE 稀疏推理（每次只激活 37B 参数），但完整模型（所有专家参数）需要分布在推理集群的某处。

场景 推荐 GPU 数量 推理速度
个人尝试 / 聊天 RTX 4090 24GB 1 张 ~1-2 tok/s (需 CPU Offload)
开发者调试 RTX 4090 24GB 2 张 (NVLink) ~5-8 tok/s (GGUF Q4_K_M)
小型部署 A100 80G / H100 2-4 张 ~20-40 tok/s
生产级部署 H100 80G HBM3 8 张 ~50-100+ tok/s (批量推理)
大型 API 服务 H200 141GB 32+ 张 数百 tok/s (含 EP + Continous Batching)
个人用户实操建议：如果你只有一块 RTX 3090/4090（24GB 显存），推荐使用 llama.cpp 加载 DeepSeek-V3 的 GGUF Q4_K_M 版本，结合 GPU 层 offloading（将部分 transformer 层放到 GPU 计算，其余在 CPU 上运行）。虽然速度不快（1-3 tok/s），但对于个人聊天和实验完全足够。你需要大约 32GB+ 的系统 RAM。

量化压缩让大模型的部署门槛大幅降低——从需要数百张数据中心 GPU，到一张消费级显卡就能运行。但有一个问题浮现了：量化和推理优化虽然让模型"跑得快、跑得轻"，但我们如何知道模型在不同任务上的真实表现呢？下一个话题——评测体系——正是为了回答这个问题。让我们进入第 18 章。
