第14章 生产化部署与监控

如果你问一位有经验的 AI 工程师："开发 Agent 最难的是什么？"答案大概率不是"写 Agent 的逻辑"，而是"让 Agent 在生产环境稳定运行"。开发环境的 Agent 跑得再好，上了生产也可能问题百出——模型降级导致 Token 消耗暴增、Prompt 改动导致输出格式变化、某个用户的特殊输入触发了 Agent 的死循环……这些问题在开发环境基本发现不了，只有在生产环境、在真实用户的压力下才会暴露。

这一章我们聚焦 LangSmith Fleet——LangChain 生态的生产化平台，涵盖全功能解析、Tracing 监控、Prompt 版本管理、Agent 权限控制，以及自托管部署方案。这些是一个 Agent 系统从"能跑"到"跑得稳"的必由之路。

核心观点： 部署 Agent 不同于部署传统 Web 服务。传统服务"输入→处理→输出"是确定性的，而 Agent 是非确定性的——同样的输入可能产生完全不同的行为路径。这意味着监控和可观测性不是"锦上添花"，而是"生死攸关"。

14.1 LangSmith Fleet 全功能解析

LangSmith 是 LangChain 生态的生产化平台，2025 年底发布的 Fleet 版本是其迄今为止最大的一次功能升级。Fleet 版本将 LangSmith 从一个"调试工具"升级为"Agent 全生命周期管理平台"——覆盖开发、测试、部署、监控、优化五大环节。

功能模块 说明 Fleet 版本增强
Tracing 追踪每次 Agent 执行的完整链路 支持 LangGraph 状态图可视化和回放
Prompt 管理 版本化、A/B 测试、发布管理 新增 Prompt Registry 和自动回滚
Dataset & Evaluation 构建测试数据集，自动化评估 新增生产数据自动回流标注
Monitoring 实时监控、告警、仪表盘 新增 Agent 专属指标（循环次数、工具调用频率等）
Deployment 一键部署 Agent 服务 新增多环境管理和灰度发布
Security API Key 管理、访问控制 新增 Agent 身份认证和操作审计
Fleet 的定位变化： 在 Fleet 之前，LangSmith 主要是"看看 Trace，找找 Bug"的工具。Fleet 之后，它真正成为一个可以承载 Agent 全生命周期管理的平台——从开发阶段的提示词调试，到上线后的性能监控，再到持续的 Prompt 优化迭代，都能在一个平台完成。

14.2 Tracing 与指标监控

Tracing（链路追踪）是 LangSmith 的核心能力。每次 Agent 执行——从用户输入到最终输出——LangSmith 都会记录每一个步骤的详细信息：调用了哪个模型、用了什么参数、调用了哪些工具、工具返回了什么、花了多长时间、花了多少 Token。

import os
from langchain.callbacks import LangSmithTracer

# 1. 开启 Tracing（只需设置环境变量）
os.environ["LANGSMITH_TRACING"] = "true"
os.environ["LANGSMITH_API_KEY"] = "ls-..."
os.environ["LANGSMITH_PROJECT"] = "my-agent-production"

# 或者通过代码配置
from langsmith import Client

client = Client(
    api_url="https://api.smith.langchain.com",
    api_key="ls-...",
    project_name="my-agent-production",
)

# 所有 LangChain/LangGraph 的调用会自动 Trace
agent = create_agent(model=..., tools=...)
result = agent.invoke({"input": "你好"})
# 这个调用会自动出现在 LangSmith 仪表盘中

# 2. 自定义 Trace 元数据
from langchain.callbacks import configure_logger

with LangSmithTracer(
    project="my-agent",
    tags=["production", "customer-service"],
    metadata={
        "user_id": "user-123",
        "session_id": "session-456",
        "environment": "production",
    }
) as tracer:
    result = agent.invoke({"input": "我的订单怎么还没到？"})

# 3. 为 LangGraph 添加细粒度 Tracing
from langgraph.checkpoint import MemorySaver
from langgraph.callbacks import LangGraphTracer

app = graph.compile(
    checkpointer=MemorySaver(),
    # LangGraph 的每个节点执行都会 Trace
    callbacks=[LangGraphTracer(project="langgraph-agent")],
)

# 4. 监控指标
# LangSmith Fleet 提供预置仪表盘：
# - 请求量 / 延迟 / 错误率（常规指标）
# - Agent 循环次数分布（监控死循环）
# - 工具调用分布（哪个工具最热门）
# - Token 消耗趋势（成本监控）
# - 模型 Fallback 触发率（监控模型可用性）
关键指标解读：
  • Agent 平均循环次数： 正常值 2-5 次。如果趋势持续上升，说明 Agent 在处理任务时频繁陷入"思考不够、需要更多工具调用"的状态。>10 次说明可能有循环问题。
  • 工具调用成功率： 低于 95% 说明工具有问题——要么工具本身不稳定，要么 Agent 传入了不正确的参数。
  • 模型 Fallback 率： 超过 5% 说明主模型服务不稳定，需要考虑切换提供商或增加配额。
  • Token 消耗/请求： 出现异常峰值往往意味着 Agent 在某个环节产生了过长的输出，需要检查 Prompt 或工具返回内容。

14.3 Prompt 版本管理与 A/B 测试

Prompt 是 Agent 系统中最脆弱也最关键的组成部分。一个微小的 Prompt 改动——加了一句话、改了一个措辞——就可能导致 Agent 的行为发生巨大变化。Prompt 版本管理就是要让这些改动可追踪、可测试、可回滚。

from langsmith import Client
from langchain.prompts import ChatPromptTemplate

client = Client()

# 1. 在 LangSmith 中创建和管理 Prompt
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个{role}专家，请用{tone}的语气回答问题。"),
    ("human", "{question}"),
])

# 发布到 LangSmith Prompt Registry
prompt_id = client.create_prompt(
    name="customer-service-agent",
    description="客服 Agent 主 Prompt",
    tags=["production"],
)
client.create_prompt_version(
    prompt_id=prompt_id,
    prompt=prompt,
    version_name="v1.0-initial",
)

# 2. 从 LangSmith 加载 Prompt（生产环境读取）
from langsmith.prompts import load_prompt

# 加载指定版本
prompt_v1 = load_prompt("customer-service-agent:v1.0-initial")
prompt_v2 = load_prompt("customer-service-agent:v1.1-add-empathy")

# 加载最新发布版本
prompt_latest = load_prompt("customer-service-agent:latest")

# 3. A/B 测试配置
from langsmith.evaluation import ABTest

ab_test = ABTest(
    # 控制组：使用旧 Prompt
    control_prompt="customer-service-agent:v1.0-initial",
    # 实验组：使用新 Prompt
    treatment_prompt="customer-service-agent:v1.1-add-empathy",
    # 流量分配
    traffic_split=0.5,  # 50% 用户走旧版本，50% 走新版本
    # 评估指标
    metrics=["user_satisfaction", "resolution_rate", "avg_token_cost"],
    # 运行时长
    duration_hours=48,
)

# 4. 自动回滚策略
# LangSmith Fleet 支持配置"自动回滚触发器"
# 当某个指标（如错误率）超过阈值时，自动回滚到上一个稳定版本
client.configure_auto_rollback(
    prompt_id=prompt_id,
    # 触发条件
    triggers=[
        {"metric": "error_rate", "operator": ">", "threshold": 0.05, "window_min": 10},
        {"metric": "avg_loops", "operator": ">", "threshold": 10, "window_min": 5},
    ],
)
踩坑实录： 我们的团队曾犯过一个经典的错误——在周五下午直接更新了生产环境的 Prompt，然后下班回家。新 Prompt 让客服 Agent 在处理退货请求时多了一个"确认用户情绪"的步骤，结果这个步骤在某些情况下触发了 Agent 的无限循环（确认情绪 → 用户不耐烦 → 再确认 → 用户更不耐烦）。周一回来发现这个 Prompt 版本跑掉了 80 万 Token，产生了 3000 多个未完成的工单。从那以后，我们严格执行"Prompt 变更必须走 A/B 测试，且至少观察 24 小时才能全量发布"的铁律。

14.4 Agent 身份与权限管理

当 Agent 能够调用 API、操作数据库、发送消息时，权限管理就变得至关重要。你不希望客服 Agent 能访问用户密码，也不希望数据分析 Agent 能删除生产数据。LangSmith Fleet 提供了完整的 Agent 身份与权限管理能力：

# 1. Agent 身份定义
from langsmith.agents import AgentIdentity

agent_identity = AgentIdentity(
    agent_id="customer-service-v2",
    display_name="智能客服 V2",
    # 绑定的 API Key（自动轮换）
    api_key="sk-...",
    # 允许的工具列表
    allowed_tools=[
        "search_order", "check_refund_status", "update_shipping",
    ],
    # 禁止的工具列表
    forbidden_tools=[
        "delete_user", "modify_order_price", "export_all_data",
    ],
    # 模型权限
    allowed_models=["gpt-4o", "gpt-4o-mini"],
    # 操作限制
    rate_limits={
        "search_order": {"max_calls": 100, "per_window": 60},
        "update_shipping": {"max_calls": 10, "per_window": 60},
    },
    # 操作需审批
    requires_approval=[
        "process_refund",
        "send_bulk_email",
    ],
)

# 2. 在 Agent 中使用身份
from langchain.agents import create_agent

agent = create_agent(
    model=ChatOpenAI(model="gpt-4o"),
    tools=[search_order, process_refund, ...],
    # 绑定身份
    identity=agent_identity,
)

# 3. 审计日志
# 所有工具调用都会记录到审计日志
# 包括：谁调用的、调用了什么工具、传入了什么参数、结果如何
audit_log = client.get_audit_logs(
    agent_id="customer-service-v2",
    start_time="2026-06-01",
    end_time="2026-06-30",
    actions=["tool_call", "approval_request"],
)

for entry in audit_log:
    print(f"[{entry.timestamp}] {entry.agent_id} 调用 {entry.tool_name}")
    print(f"  参数：{entry.arguments}")
    print(f"  结果：{entry.result[:100]}...")
权限维度 控制粒度 典型的"禁止"操作
工具级 按工具名称精确控制 DROP TABLE, DELETE USER, 批量导出
模型级 限制可用的模型列表 禁止使用低成本模型处理复杂任务
频率级 调用频率限制 每分钟最多搜索 100 次
审批级 高危操作需人工审批 退款、删除数据、发送通知
时间级 限制操作时间段 非工作时间只能查询不能修改
权限管理的最佳实践： 遵循"最小权限原则"——给 Agent 刚刚够用的权限，不多给。很多安全事件都是因为 Agent 权限过大导致的（比如一个本应只读的 Agent 被注入了"修改"指令）。另外，定期审计 Agent 的调用日志，检查是否有"越权行为"的蛛丝马迹。

14.5 自托管部署方案

虽然 LangSmith Cloud 是大多数团队的首选，但有些场景需要自托管部署——数据主权要求、内网环境、合规需求、或者单纯的成本控制。LangSmith Fleet 支持完整的自托管方案。

# 1. Docker Compose 部署 LangSmith Fleet
# docker-compose.yml
"""
version: '3.8'
services:
  langsmith:
    image: langchain/langsmith-fleet:latest
    ports:
      - "1984:1984"
    environment:
      - LANGCHAIN_ENV=production
      - POSTGRES_URI=postgresql://user:pass@postgres:5432/langsmith
      - REDIS_URI=redis://redis:6379
      - SECRET_KEY=your-secret-key
    depends_on:
      - postgres
      - redis
    volumes:
      - ./data:/data
    deploy:
      replicas: 2  # 高可用部署

  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: langsmith
      POSTGRES_USER: user
      POSTGRES_PASSWORD: securepass
    volumes:
      - pgdata:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine
    volumes:
      - redisdata:/data
"""

# 2. 配置 LangSmith SDK 连接自托管实例
import os

os.environ["LANGSMITH_ENDPOINT"] = "https://langsmith.internal.example.com"
os.environ["LANGSMITH_API_KEY"] = "your-internal-api-key"
os.environ["LANGSMITH_PROJECT"] = "self-hosted-agent"

# 3. Agent 服务部署
# 使用 FastAPI 包装 Agent 为 HTTP 服务
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class QueryRequest(BaseModel):
    input: str
    user_id: str
    session_id: str

class QueryResponse(BaseModel):
    output: str
    trace_url: str

@app.post("/agent/query", response_model=QueryResponse)
async def agent_query(request: QueryRequest):
    config = {"configurable": {"thread_id": request.session_id}}

    with LangSmithTracer(
        project="self-hosted-agent",
        metadata={"user_id": request.user_id},
    ) as tracer:
        result = agent.invoke(
            {"input": request.input},
            config=config,
        )

    trace_url = f"https://langsmith.internal.example.com/traces/{tracer.run_id}"
    return QueryResponse(output=result["output"], trace_url=trace_url)

# 启动：uvicorn main:app --host 0.0.0.0 --port 8000
自托管 vs 云方案对比：
  • 成本： 自托管月均 $500-2000（取决于规模），云方案按量付费。小团队（<100 万次 Trace/月）云方案更划算。
  • 维护： 自托管需要专职 DevOps 管理 PostgreSQL、Redis 集群、备份恢复。云方案零维护。
  • 合规： 金融、医疗等行业通常要求数据不出境，自托管是唯一选择。
  • 可用性： 自托管可以做到 99.9%（多实例 + 主从数据库），云方案承诺 99.95%。

14.6 生产级部署检查清单

最后，分享一份我们在经历了无数次生产事故后总结出来的"Agent 生产部署检查清单"。每次部署新 Agent 或重大更新前，逐项检查：

类别 检查项 通过标准
安全性 权限最小化 Agent 只拥有完成任务所需的最小权限
输入过滤 PII 脱敏、Prompt 注入检测、内容审核已配置
审计日志 所有工具调用和审批操作已记录
可靠性 Model Fallback 至少配置了 2 层 Fallback 模型
超时控制 设置了节点级别和图级别的超时
循环检测 配置了最大递归深度和循环检测
可观测性 Trace 完整性 所有 Agent 调用都接入 LangSmith Tracing
监控告警 错误率、延迟、循环次数、Token 消耗的关键告警已配置
Prompt 版本化 所有生产 Prompt 在 LangSmith Registry 中管理
性能 延迟预算 P95 延迟满足业务 SLA
成本预算 设置了每日 Token 预算和告警阈值
并发测试 通过预期峰值并发量的压力测试
流程 A/B 测试 Prompt 修改有 A/B 测试计划
灰度发布 新版本逐步放量，有回滚方案
应急响应 定义了"一键降级"到人工处理的流程
踩坑实录： 这份清单不是凭空想象的——每一条背后都有真实的事故教训。"P95 延迟满足业务 SLA"这一条，我们就有过血的教训：一个 Agent 在高峰期因为检索数据库的延迟增加，导致 P95 延迟从 3 秒飙到 30 秒，而我们的告警只看平均延迟（1.5 秒，看似正常）。结果用户大规模投诉"转圈圈"，我们才意识到 P95 的重要性。现在我们的监控面板上永远挂着 P50、P95、P99 三条线。
持续优化循环： Agent 的生产化不是一次性的工作，而是一个持续优化的循环：
  1. 从生产 Trace 中发现异常案例
  2. 将异常案例加入测试数据集
  3. 优化 Prompt 或 Agent 逻辑
  4. 在测试数据集上验证改进效果
  5. A/B 测试后灰度发布到生产
  6. 监控指标，回到步骤 1
这个"生产数据驱动优化"的循环，是 Agent 系统持续进化的引擎。
