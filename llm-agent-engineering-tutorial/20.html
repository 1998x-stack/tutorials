第20章 工具生态与 MCP 协议

2025 年 3 月，Anthropic 发布了 MCP（Model Context Protocol） 的草案。当时并没有引起太大的波澜——毕竟 AI 领域每天都有新协议、新标准冒出来。但到了 2026 年，MCP 已经成为 Agent 开发中不可或缺的基础设施。我在 2025 年底参加一个技术会议时，听到一位来自大厂的工程师感慨："MCP 对 Agent 生态的意义，不亚于 HTTP 对 Web 的意义。"当时觉得夸张，如今回头看，这个评价并不为过。

上一章我们学习了 OpenManus 的分层架构，而 MCP 正是 OpenManus 工具生态的基石。这一章，我们把视野放大，看看整个工具生态是如何运作的——从 OpenManus 的 20+ 内置工具，到 MCP 协议的设计细节，再到 Agent-to-Agent 通信协议。

核心观点： 工具生态正在经历从"紧耦合集成"到"即插即用标准化"的转变。MCP 协议就是这场转变的催化剂——它让工具提供者和 Agent 开发者可以各自独立工作，只需要遵守同一个"连接标准"。

20.1 OpenManus 工具生态概览

OpenManus 内置了 20+ 个开箱即用的工具，覆盖了 Agent 最常见的需求。这些工具按照功能可以分为以下几类：

工具类别 工具列表 用途
Web 工具 WebSearch, WebFetch, WebScrape, BrowserControl 信息检索、网页抓取、浏览器操作
文件工具 FileRead, FileWrite, FileEdit, FileSearch, DirList 本地文件读写、搜索、目录浏览
代码工具 PythonExecute, BashExecute, CodeReview, CodeFormat 代码执行、审查、格式化
API 工具 HttpRequest, GraphQLQuery, WebhookSend HTTP 请求、API 调用
数据工具 SQLQuery, CSVRead, JSONParse, DataViz 数据库查询、数据处理、可视化
通信工具 EmailSend, SlackNotify, WecomBot 消息推送、邮件发送

每个工具都遵循统一的接口规范：

from openmanus.tools import BaseTool
from pydantic import BaseModel, Field

class WebSearchInput(BaseModel):
    query: str = Field(description="Search query string")
    max_results: int = Field(default=5, ge=1, le=20)

class WebSearch(BaseTool):
    name: str = "web_search"
    description: str = "Search the web for current information"
    parameters: type = WebSearchInput

    async def execute(self, args: WebSearchInput) -> str:
        """Implement the actual search logic here."""
        results = await search_engine.search(
            args.query,
            num_results=args.max_results
        )
        return format_search_results(results)
工具设计原则： 每个 OpenManus 工具都是自描述的——它声明自己的 name、description 和参数模式。LLM 通过这些元数据来决定何时调用、传入什么参数。写工具时，"描述写得清晰"比"功能做得强大"更重要——一个描述模糊的强大工具，LLM 根本不会正确使用它。

20.2 MCP 协议深度解析

MCP（Model Context Protocol）本质上是一个"让 LLM 能够发现和使用工具"的标准协议。它定义了三个核心角色：

  • MCP Host： 发起连接的一方，通常是 LLM 应用或 Agent 框架（如 OpenManus）
  • MCP Server： 提供工具的一方，负责注册工具、处理调用请求、返回执行结果
  • MCP Client： Host 内部的组件，负责与 Server 建立连接、发现工具、转发调用

通信协议基于 JSON-RPC 2.0，通过标准输入输出（stdio）或 SSE（Server-Sent Events）进行传输。以下是 MCP 的核心流程：

// MCP 协议的核心流程
// 1. 客户端发送初始化请求
→ {"jsonrpc": "2.0", "id": 1, "method": "initialize",
   "params": {"protocolVersion": "2025-03-26",
              "capabilities": {"tools": {}}}}

// 2. 服务端响应，声明能力
← {"jsonrpc": "2.0", "id": 1, "result": {
     "protocolVersion": "2025-03-26",
     "capabilities": {
       "tools": {
         "listChanged": true  // 支持工具列表变更通知
       }
     }}}

// 3. 客户端发现工具列表
→ {"jsonrpc": "2.0", "id": 2, "method": "tools/list",
   "params": {}}

// 4. 服务端返回工具清单
← {"jsonrpc": "2.0", "id": 2, "result": {
     "tools": [{
       "name": "web_search",
       "description": "Search the web...",
       "inputSchema": {
         "type": "object",
         "properties": {
           "query": {"type": "string"},
           "max_results": {"type": "number", "default": 5}
         },
         "required": ["query"]
       }
     }]}}

// 5. 客户端调用工具
→ {"jsonrpc": "2.0", "id": 3, "method": "tools/call",
   "params": {
     "name": "web_search",
     "arguments": {"query": "MCP protocol 2026", "max_results": 10}
   }}

// 6. 服务端返回结果
← {"jsonrpc": "2.0", "id": 3, "result": {
     "content": [{
       "type": "text",
       "text": "Search results: ..."
     }]}}
MCP 的精妙之处： 它不仅仅是一个工具调用协议。通过 listChanged 通知机制，MCP Server 可以在运行时动态增减工具——Agent 不需要重启就能发现新工具。这让"工具热插拔"成为可能：比如在运维场景中，白天注册告警处理工具，晚上注册批量升级工具，无需停服。

20.3 自定义 MCP Server 开发

MCP 协议最强大的地方在于：任何人都可以编写 MCP Server，将自己的服务暴露为 LLM 可调用的工具。下面是一个自定义 MCP Server 的完整示例——假设我们要为公司的内部知识库写一个 MCP Server：

# knowledge_base_mcp_server.py
# 一个企业知识库的 MCP Server

from mcp.server import Server, NotificationOptions
from mcp.server.models import InitializationOptions
import mcp.server.stdio

async def main():
    """启动 MCP Server，提供知识库查询工具。"""
    server = Server("knowledge-base")

    # 注册工具：查询知识库
    @server.list_tools()
    async def list_tools():
        return [
            {
                "name": "kb_search",
                "description": "搜索企业内部知识库文档",
                "inputSchema": {
                    "type": "object",
                    "properties": {
                        "query": {
                            "type": "string",
                            "description": "搜索关键词"
                        },
                        "top_k": {
                            "type": "number",
                            "description": "返回结果数量",
                            "default": 3
                        },
                        "department": {
                            "type": "string",
                            "description": "按部门筛选",
                            "enum": ["engineering", "product", "design", "all"]
                        }
                    },
                    "required": ["query"]
                }
            },
            {
                "name": "kb_get_doc",
                "description": "获取指定文档的详细内容",
                "inputSchema": {
                    "type": "object",
                    "properties": {
                        "doc_id": {
                            "type": "string",
                            "description": "文档 ID"
                        }
                    },
                    "required": ["doc_id"]
                }
            }
        ]

    @server.call_tool()
    async def call_tool(name: str, arguments: dict) -> list[dict]:
        if name == "kb_search":
            results = query_vector_db(
                arguments["query"],
                top_k=arguments.get("top_k", 3),
                department=arguments.get("department", "all")
            )
            formatted = format_search_results(results)
            return [{"type": "text", "text": formatted}]

        elif name == "kb_get_doc":
            doc = get_document_by_id(arguments["doc_id"])
            return [{"type": "text", "text": doc["content"]}]

        raise ValueError(f"Unknown tool: {name}")

    async with mcp.server.stdio.stdio_server() as (read_stream, write_stream):
        await server.run(
            read_stream, write_stream,
            InitializationOptions(
                server_name="knowledge-base",
                server_version="1.0.0"
            )
        )

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())

然后，在 OpenManus 中连接这个 MCP Server 只需要一行配置：

# config.toml 中配置 MCP Server
[mcp_servers]
knowledge_base = { command = "python", args = ["knowledge_base_mcp_server.py"] }
# 也可以连接远程 MCP Server
weather = { url = "https://mcp.weather-api.com/sse" }
开发建议： 2026 年已经有丰富的 MCP Server 市场——你可以找到各种现成的 MCP Server：GitHub、Jira、Slack、Notion、Datadog 等。在写自定义 MCP Server 之前，先去 MCP Servers 仓库 看看有没有现成的。大部分情况下，你只需要配置而不是编码。

20.4 A2A（Agent-to-Agent）通信协议

如果说 MCP 解决了"Agent 如何与工具对话"的问题，那么 A2A（Agent-to-Agent） 协议解决的就是"Agent 如何与其他 Agent 对话"的问题。A2A 由 Google 在 2025 年 4 月提出，到 2026 年已经成为多 Agent 系统中的主流通信标准。

A2A 协议的核心概念：

概念 说明 类似物
Agent Card Agent 的"名片"，声明自己的能力、接口、认证方式 OpenAPI Spec
Task 一个可执行的工作单元，有状态、有结果 HTTP Request
Message Agent 之间交换的信息，可以是文本、结构化数据或流式内容 WebSocket Message
Artifact 任务产生的产物（文件、图片、数据表格等） HTTP Response Body

A2A 的典型工作流程如下：当 Agent A 需要 Agent B 的帮助时，A 首先获取 B 的 Agent Card（通过注册中心或直接请求），了解 B 的能力范围。然后 A 向 B 发送一个 Task 请求，B 开始执行任务并实时返回状态更新和中间结果，最终返回完成的 Artifact。

MCP vs A2A： 很多人会混淆这两个协议。简单来说，MCP 是"人-工具"接口（Agent 调用工具），A2A 是"人-人"接口（Agent 与 Agent 对话）。在实际系统中，两者经常配合使用——Agent A 通过 MCP 调用搜索引擎工具，同时通过 A2A 向 Agent B 请求数据分析服务。OpenManus 在 2026 年的版本中已经同时内置了对 MCP 和 A2A 的支持。

20.5 工具链动态组合

单个工具的能力是有限的，真正强大的是工具的组合使用。OpenManus 支持工具链的动态组合——Agent 在执行过程中可以根据当前上下文，自主决定工具调用的顺序和组合方式。

一个典型的工具链组合案例："帮我监控一下竞品的价格变化"

# Agent 自主组合工具链的决策过程（简化）

# Step 1: 搜索竞品信息
→ web_search(query="2026年智能手表竞品价格", max_results=10)

# Step 2: 抓取具体产品页面
→ web_fetch(url="https://example.com/product/abc")

# Step 3: 解析价格数据
→ python_execute(code="""
    from bs4 import BeautifulSoup
    # 解析抓取到的 HTML 提取价格
    ...
    """)

# Step 4: 对比历史价格
→ sql_query(query="""
    SELECT date, price FROM price_history
    WHERE product='abc' ORDER BY date
    """)

# Step 5: 发送通知
→ slack_notify(channel="#price-alert", message="...")

这种动态组合的能力来源于 OpenManus 的两个设计：第一，LLM 在每一步都能看到所有可用的工具和当前的中间结果；第二，工具的执行结果会回写到记忆中，作为后续决策的上下文。这就像是给了一个人全套工具箱和持续更新的工作台——他可以根据实际情况随时决定下一步用什么工具。

需要注意： 工具链越长，出错的概率越大。在实际项目中，建议对超过 5 步的工具链设置中间检查点——Agent 每完成几个步骤就总结一下当前进度，让人类或监控系统可以及时干预。我们内部有个经验法则：工具链每增加一步，成功率下降约 5-8%。

20.6 实战：构建自定义工具包

最后，让我们动手构建一个自定义工具包。假设我们是数据分析团队的一员，想为 Agent 添加一组"数据分析专用工具"：

from openmanus import Manus
from openmanus.tools import BaseTool
import pandas as pd
import matplotlib.pyplot as plt

# 工具 1: 加载 CSV 数据
class CSVLoader(BaseTool):
    name: str = "csv_loader"
    description: str = "加载 CSV 文件到 DataFrame，返回列名和前 5 行"

    async def execute(self, file_path: str) -> str:
        df = pd.read_csv(file_path)
        info = (
            f"行数: {len(df)}, 列数: {len(df.columns)}\n"
            f"列名: {list(df.columns)}\n"
            f"前 5 行:\n{df.head().to_string()}"
        )
        return info

# 工具 2: 数据统计摘要
class DataSummary(BaseTool):
    name: str = "data_summary"
    description: str = "生成数据集的统计摘要（均值、中位数、标准差等）"

    async def execute(self, data_description: str) -> str:
        # 实际项目中会维护全局 DataFrame 引用
        # 这里简化展示
        return "统计摘要生成中..."

# 工具 3: 数据可视化
class DataVisualizer(BaseTool):
    name: str = "data_viz"
    description: str = "根据数据和配置生成图表，保存为图片文件"

    async def execute(self, chart_type: str, x_col: str,
                      y_col: str, output: str = "chart.png") -> str:
        # 生成图表并保存
        fig, ax = plt.subplots(figsize=(10, 6))
        ax.set_title(f"{chart_type}: {x_col} vs {y_col}")
        plt.savefig(output, dpi=150, bbox_inches="tight")
        plt.close()
        return f"图表已保存到 {output}"

# 组装工具包并注册到 Agent
agent = Manus(name="DataAnalyst", llm_config={...})

agent.register_tool(CSVLoader())
agent.register_tool(DataSummary())
agent.register_tool(DataVisualizer())

# 运行数据分析任务
result = await agent.run("""
请帮我分析 sales_2026_q1.csv 文件：
1. 加载数据，了解整体结构
2. 计算各产品线的销售额统计
3. 生成一张月度销售趋势图，保存为 sales_trend.png
""")
工具包设计模式： 好的工具包应该遵循"高内聚、低耦合"原则——一组工具专注于一个领域（数据分析、DevOps、内容创作等），每个工具做一件小而明确的事。不要在一个工具里塞太多功能，LLM 需要简单清晰的"按钮"而不是瑞士军刀。

构建完工具包后，你可以把它打包为一个 Python 包，上传到 PyPI，或者通过 MCP Server 暴露给其他团队使用。在大型组织中，不同团队维护自己的 MCP Server，通过统一注册中心发现和共享——这就是"工具生态"的真正含义。

本章小结

这一章我们走遍了 OpenManus 的工具生态系统：从 20+ 内置工具的分类和使用，到 MCP 协议的深度解析，再到自定义 MCP Server 的开发、A2A 通信协议、工具链动态组合，最后通过一个数据分析工具包的实战案例串联了所有概念。工具是 Agent 的"双手"，而 MCP 协议就是"双手的接口标准"——掌握了这些，你就掌握了让 Agent 真正"动手做事"的能力。
