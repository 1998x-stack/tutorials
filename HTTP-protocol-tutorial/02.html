第二章：URL与资源定位

大家好，我是你们的讲师。上一章咱们聊了HTTP的宏观图景，今天开始进入细节——URL。你每天在浏览器地址栏里输入的那串东西，看似简单，但里面的门道真不少。我带团队面试前端和后台工程师的时候，经常发现好多人工作了几年，对URL的理解还停留在"就是网址"的地步。今天我们把这个坑填上。

2.1 URI、URL、URN的区别

很多同学搞不清URI、URL、URN这三者的关系。其实很简单：URI（统一资源标识符）是最大的概念，它有两个子类型——URL（统一资源定位符）和URN（统一资源名称）。用一句话记住：URI标识资源，URL告诉你"去哪找"，URN告诉你"它叫什么"。

举个例子：https://example.com/logo.png 是一个URL，因为它既标识了logo.png这个资源，又告诉了你它的位置（example.com服务器上）。而 urn:isbn:978-3-16-148410-0 是一个URN——它标识了一本书，但没告诉你上哪下载这本书。

概念英文作用例子
URIUniform Resource Identifier标识任意资源https://example.com/file、mailto:user@example.com
URLUniform Resource Locator定位资源（包含访问方式）https://example.com/path
URNUniform Resource Name命名资源（不包含位置）urn:ietf:rfc:2616

现实生活中，你在浏览器里用的全是URL。HTTP规范里说"URI"的时候，绝大多数情况就是指HTTP URL。我个人的建议是：日常开发说"URL"没问题，但读RFC文档碰到"URI"时，心里要知道它指的是广义的标识符。

面试高频题：URL和URI的区别是什么？标准回答：URI是标识资源的抽象概念（ID card），URL是定位资源的具体方式（home address）。所有URL都是URI，但不是所有URI都是URL。

2.2 URL的语法结构

一个完整的HTTP URL长这样：

  scheme://host:port/path?query#fragment
  https://api.example.com:443/users?id=123&page=1#top

咱们拆开来看每个部分：

组件是否必需说明例子
scheme是协议方案，HTTP通常是http或httpshttps
host是服务器域名或IP地址api.example.com
port否（有默认值）HTTP默认80，HTTPS默认443:443
path是资源路径，以/开头/users
query否查询参数，以?开头?id=123&page=1
fragment否片段标识，以#开头#top

这里有几个容易忽略的点：端口号如果用的是默认值（HTTP 80、HTTPS 443），浏览器地址栏通常会隐藏它，但实际请求中并不缺失。Fragment（#后面的部分）不会被发送到服务器，它只在浏览器端用于页面内定位。我刚入行时不知道这个，在fragment里放参数让后端处理，debug了一下午才发现真相……

踩坑回顾：有一次我在前端用 window.location.hash 存了API token，想着每次请求从hash里取。结果发现服务器根本收不到hash部分——fragment是由浏览器处理的，不会出现在HTTP请求中。这是一个典型的"新手坑"。

2.3 URL编码规则

URL里能用的字符其实很有限。RFC 3986规定，URL中只有字母（a-z, A-Z）、数字（0-9）以及 - _ . ~ 这些保留字符可以直接使用。其他字符——包括中文、空格、特殊符号——都需要百分号编码（Percent-encoding）。

编码规则很简单：把字符的UTF-8字节值前面加上 %。比如空格是 %20，中文"你"的UTF-8编码是 E4 BD A0，所以URL编码就是 %E4%BD%A0。

JavaScript中提供两个编码方法，很多人分不清：

  • encodeURI()：用于编码整个URL，不会编码URI中本来就允许的字符（如 ://?#）。适用于给一个完整的URL做编码。
  • encodeURIComponent()：用于编码URL参数的值，会编码几乎所有非字母数字字符（包括 / ? # & 等）。适用于给query string中的参数值做编码。
// 区别演示
const url = 'https://example.com/search?q=hello world&lang=zh-CN';

// encodeURI 保留 ://?#& 等结构字符
encodeURI(url);
// 结果: https://example.com/search?q=hello%20world&lang=zh-CN
// 注意: 中文场景下容易出问题，比如路径中的中文不会被编码

// encodeURIComponent 则会把几乎所有特殊字符都编码
encodeURIComponent('hello world');
// 结果: hello%20world

// 正确用法：分开编码
const base = 'https://example.com/search';
const params = 'q=' + encodeURIComponent('C# 教程') + '&page=' + encodeURIComponent('1');
const fullUrl = base + '?' + params;
// 结果: https://example.com/search?q=C%23%20%E6%95%99%E7%A8%8B&page=1

常见需要编码的字符有：空格（%20）、中文、#（%23）、%本身（%25）、&（%26）。

实用建议：在前端拼接URL参数时，永远不要手动拼接字符串。使用 URLSearchParams 或 new URL() 构造函数，它们会自动帮你处理编码。手动拼接 ?q=" + value 是很多线上bug的根源。

2.4 绝对URL与相对URL

绝对URL包含完整的scheme、host、path等信息，比如 https://example.com/images/logo.png。在任何地方都能直接使用。

相对URL则省略了部分信息，需要基于一个"基础URL"来解析。相对URL又分两种：

  • 协议相对URL：省略scheme，如 //cdn.example.com/lib.js，浏览器会自动使用当前页面的协议（http或https）。
  • 路径相对URL：省略scheme和host，如 /images/logo.png（以/开头，相对于根路径）或 ../css/style.css（相对于当前路径）。

路径解析时 .. 表示上级目录，. 表示当前目录。比如当前页面在 https://example.com/docs/tutorial/，那么 ../images/logo.png 解析为 https://example.com/docs/images/logo.png。

一个常见的坑是：在CSS文件里用相对路径引图片。假设你的CSS在 /css/style.css，里面写的 background: url("../images/bg.png")，这个路径是相对于CSS文件的位置，而不是相对于HTML页面！

2.5 URL长度限制

URL有没有长度限制？严格来说，HTTP规范没有规定URL的最大长度。但在实际应用中，限制无处不在：

环境限制长度说明
Internet Explorer2083字符IE对URL长度有严格限制，超过的部分会被截断
Chrome/Safari/Firefox约64KB～2MB现代浏览器限制较宽，但实践中不建议超过2000字符
Nginx默认8KB（可配置）超过 large_client_header_buffers 限制会返回414
Apache默认8KB超过返回414 Request-URI Too Large
CDN通常4-8KBCloudFront、CloudFlare等各有自己的限制

这就是为什么GET请求不适合传递大量数据。如果你需要提交一个很长的表单（比如带文本编辑器内容的表单），用POST而不是GET。URL长度限制在服务端也有——我遇到过Nginx配置没改，用户提交了一个带2000个参数的GET请求，结果Nginx直接返回414，连Tomcat都没到。

踩坑回顾：某次做导出功能，前端用GET请求传了一组筛选条件（500个ID），URL长度超过3万字符。测试环境好好的（服务端框架自动切到POST），上了生产Nginx直接返回414。后来改成POST + 请求体传参搞定。教训：永远不要用GET传递大量数据，哪怕它在浏览器里能打开。
