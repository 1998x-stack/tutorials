表格数据的特殊性

大家好。如果说有一种数据格式贯穿了我们整个课程——从数据清洗到标注到知识图谱——那就是表格。CSV文件、数据库表、Excel电子表格、Parquet文件——表格是最普适的结构化数据载体。但恰恰是这种最常见的格式，给LLM带来了最大的挑战。

为什么？因为LLM本质上是一个"序列模型"——它处理的是线性排列的token序列。文本是一维的，从左到右读就可以了。但表格是二维的，它的信息分布在行和列两个维度上，单元格之间的语义关系不仅来自相邻位置，还来自列标题的层级、行之间的父子关系、以及跨表格的引用关系。当你把一张表格"拍扁"成一维文本送给LLM时，大量的结构信息就已经丢失了。这就好比把一张地图变成了一串经纬度坐标列表——你虽然能读到每个点的位置，但永远看不到"路网"和"区域"的全貌。

核心矛盾： 表格是二维的结构化数据，而LLM是一维的序列模型。如何在两者之间架起桥梁，让LLM既能"看到"表格的全局结构，又能"理解"单元格之间的细粒度语义关系，是表格理解这个方向最核心的技术挑战。过去两年，这个领域涌现了大量创新——从更好的序列化策略，到专门的表格编码器，再到端到端的表格大模型。

TableGPT2：大表格模型

TableGPT2是2024年末由浙江大学和阿里云联合发布的模型，它代表了"专为表格而生的大模型"这个方向的最新成果。与通用LLM不同，TableGPT2在架构上做了三个关键设计，让它能够原生地理解和处理表格数据。

第一个设计是表格编码器。TableGPT2没有直接将表格序列化为文本，而是使用一个专门的Transformer编码器来处理表格的二维结构。每一行被编码为一个独立的向量序列，列标题和单元格值被分别编码以保留它们的角色差异。第二个设计是多模态融合层。表格编码器的输出与文本编码器的输出在融合层中交错对齐——这使得模型能够在"表格的行"和"文本的描述"之间建立交叉注意力，从而支持"用自然语言问表格问题"这类任务。第三个设计是结构感知的预训练目标。TableGPT2在预训练阶段使用了多种表格特有的训练任务，包括单元格值预测、行列关系推理、表结构重建等，让模型从底层就学会理解表格的二维本质。

在TableQA、表格分类、表格NLI等标准评测上，TableGPT2以7B参数的量级超越了当时130B的通用LLM，而且推理速度更快、结构数据任务的准确率更高。这有力证明了"专用表格模型"路线的有效性。

结构感知的表格表示

即使不使用TableGPT2这样的专用模型，在现有LLM框架下也可以通过巧妙的序列化策略来保留表格的结构信息。不同的序列化方案在信息保留度和token效率之间有不同的取舍，选择哪种方案取决于具体的下游任务。

序列化策略 表示方式 信息保留度 Token效率 适用任务
Markdown表格 用Markdown语法表示行列结构 中 中 通用问答、摘要
JSON行转置 每行转为一个JSON对象 中高 低（每个key重复） API集成、结构化查询
树格式 基于层级关系的嵌套表示 高（保留层次） 低 复杂嵌套表头、多级索引
行描述法 "列名1: 值1, 列名2: 值2, ..."逐行描述 高 中 实体匹配、模式对齐、数据操作
索引标记法 <row0><col0>值</col0></row0> 很高 最低 需要精确定位的任务

在实践中，行描述法通常是在通用性和效率之间最平衡的选择。它比Markdown表格更精确（显式标注了列名），比JSON更节省token（不需要重复的引号和括号），而且便于LLM逐行处理。对于需要理解复杂嵌套表头的场景，树格式虽然token开销大，但它是唯一能保留列间层级关系的方案。

# 表格序列化策略对比：行描述法 vs 树格式

def serialize_as_row_description(df):
    """行描述法：每行描述为一个句子"""
    lines = []
    cols = df.columns.tolist()
    for _, row in df.iterrows():
        pairs = [f"{col}: {row[col]}" for col in cols]
        lines.append(" | ".join(pairs))
    return "\n".join(lines)

def serialize_as_tree(complex_df):
    """树格式：保留嵌套表头"""
    import json

    # 假设 complex_df 的列是多层索引
    tree = {}
    for col_tuple in complex_df.columns:
        node = tree
        for level in col_tuple[:-1]:
            if level not in node:
                node[level] = {}
            node = node[level]
        node[col_tuple[-1]] = None

    return json.dumps(tree, ensure_ascii=False, indent=2)

# 示例
import pandas as pd

# 简单表格
df = pd.DataFrame({
    "姓名": ["张三", "李四"],
    "年龄": [28, 35],
    "部门": ["研发", "市场"]
})

print("=== 行描述法 ===")
print(serialize_as_row_description(df))

表格与文本的联合推理

在许多真实场景中，数据不仅以表格形式存在，还伴随着大量的文本描述——数据字典、业务文档、数据血缘注释。更常见的是，表格本身中就有文本列（例如"用户评价"列）。这种"表格+文本"的混合数据给LLM带来了一个独特的挑战：如何在理解表格结构的同时，理解表格中嵌入的自由文本语义。

Code Generation from Tables

一个非常实用的技术方向是"从表格生成代码"。给定一个表格和一个自然语言需求，让LLM生成操作该表格的代码（SQL、Pandas、或Spark）。这种方案的好处是：LLM不需要直接"理解"表格中的每一个值，它只需要理解表格的结构和用户的意图，然后让代码执行引擎去处理具体的数据。

# 表格联合推理：用自然语言查询表格并生成代码
import pandas as pd
from openai import OpenAI

client = OpenAI()

def table_question_to_code(df, question):
    """将自然语言问题转化为Pandas代码"""

    # 先提取表格概要信息
    table_info = f"表格有 {len(df.columns)} 列，{len(df)} 行。\n"
    for col in df.columns:
        table_info += f"  - 列 '{col}': 类型 {df[col].dtype}, "
        table_info += f"示例值 {df[col].dropna().sample(min(3, len(df))).tolist()}\n"

    prompt = f"""请根据以下表格信息和用户问题，生成Pandas代码。

表格信息:
{table_info}

用户问题: {question}

要求:
1. 代码操作的目标DataFrame名为 df
2. 只输出代码，不要解释
3. 代码必须可执行
4. 将结果赋值给 result 变量

代码:"""

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}]
    )

    code = response.choices[0].message.content
    # 执行生成的代码
    local_vars = {"df": df}
    exec(code, globals(), local_vars)
    return local_vars.get("result", None)

# 示例
sales_df = pd.DataFrame({
    "产品": ["A", "B", "A", "C"],
    "金额": [100, 200, 150, 300],
    "月份": ["1月", "1月", "2月", "2月"]
})

result = table_question_to_code(sales_df, "每个产品的总销售额是多少？")
print(result)
安全提示： 自动生成并执行代码是一个强大的能力，但也带来了安全隐患。在生产环境中，生成的代码应该在一个沙箱环境中执行（例如Docker容器或受限的子进程），并且应该添加超时保护和资源限制。此外，强烈建议在代码执行前让另一个LLM做一次代码审查，检查是否有破坏性操作（如删除文件、连接外部网络等）。

实战案例：一个嵌套表头让三个大模型全部翻车

讲一个让我印象非常深刻的踩坑案例。某头部互联网公司的数据分析团队需要从一份财报Excel报表中自动提取关键指标。这份报表的格式很典型：第一行是年份（2022、2023、2024），第二行是季度（Q1-Q4），第三行才是实际的指标名称（营收、利润、成本）。也就是说，同一个单元格的语义由三行表头共同定义——例如"2024年Q3营收"这个语义来自三行表头的组合。

团队尝试了三个主流大模型——GPT-4o、Claude-3.5、Gemini-1.5——都是直接用Markdown格式把表格发给模型，然后问"提取2024年Q3的营收数据"。结果三个模型全部翻车：GPT-4o提取成了"2024年全年的营收"（忽略了Q3这一列），Claude-3.5把利润当作营收读了出来（行索引偏移了一位），Gemini更离谱——它把三行表头的文字拼接成了一个巨长的列名，然后跟说"找不到2024年Q3"。三个模型在同一个嵌套表头问题上全军覆没。

教训总结： 嵌套表头是LLM表格理解的"阿克琉斯之踵"。通用LLM默认假设表格的第一行就是列名，当这种假设被打破时，模型的行为就变得不可预测。解决方案有两种：一是预处理步骤中将嵌套表头"拍平"为单层（通过拼接父级和子级表头），二是使用专门的表格理解模型（如TableGPT2）或结构感知的序列化方案（如我们前面提到的树格式）。在我们的项目中，一个简单的预处理——把"2024年,Q3,营收"拼成"2024年Q3_营收"——就解决了这个问题。有时候，问题不在于模型不够聪明，而在于我们用错了"接口"去喂数据。
