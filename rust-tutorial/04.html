引子：C语言老兵的"use-after-free"噩梦

2022年，Google的Chrome安全团队发布了一份报告：Chromium中70%以上的高危安全漏洞都是内存安全问题——use-after-free、缓冲区溢出、野指针。如果你关注过CVE安全公告，你会发现"memory corruption"这个关键词出现在几乎每一期公告里。

我曾在C++项目中碰到过一个极难排查的bug：一个服务运行几小时后随机崩溃，core dump显示某个对象的虚函数表指针被篡改了。原因是一条代码路径中的对象被提前释放了，而另一条路径还在使用它。这就是经典的use-after-free。最终定位花了三天，修复只花了三行。

Rust的所有权（Ownership）系统就是为了从根本上消灭这类问题而生的。它不是靠"小心一点"——而是靠编译器强制执行一套规则。如果你违反规则，连编译都过不了。

所有权的三条核心规则：
  1. Rust中的每个值都有一个所有者（Owner）变量。
  2. 同一时间只有一个所有者。
  3. 所有者离开作用域时，值被自动释放。

这三条规则听着简单，但它们才是Rust内存安全的基石。

什么是所有权

栈（Stack）vs 堆（Heap）

要理解所有权，首先要理解数据在内存中如何存放：

特性 栈（Stack） 堆（Heap）
分配方式后进先出（LIFO）自由分配，需寻找空闲空间
速度极快（push/pop）较慢（分配+释放）
大小固定大小，编译期已知动态大小，运行时决定
典型数据基本类型、固定数组String、Vec、Box
管理方式函数调用自动管理所有权系统管理

简单来说：栈上数据不需要所有权操心——函数进出栈帧时天然地分配和释放。真正需要所有权系统管理的是堆上数据——谁负责释放？何时释放？Rust用所有权给出了一个确定的答案。

作用域与释放

{                       // s 尚未声明
    let s = String::from("hello");
    // 这里可以使用 s
}                       // 作用域结束，s 被自动释放
                        // Rust在作用域结束处自动调用 drop()

在C++中，这个模式叫RAII（Resource Acquisition Is Initialization）。在Rust中，这是所有权的天然表现——变量离开作用域时，它的析构函数（drop）被自动调用。

Move语义

现在到了Rust所有权最独特的部分——Move。

变量赋值时的Move

let s1 = String::from("hello");
let s2 = s1;  // s1 的所有权被移动到 s2

// 此时 s1 已经"被移动"，不能再使用
// println!("{s1}");  // 编译错误！value borrowed here after move
println!("{s2}");  // 正确，s2 是新的所有者

在C++中，s2 = s1会做一次深拷贝——这意味着两倍的内存分配。在Rust中，这仅仅是移动了所有权指针，没有任何堆内存的复制。而编译器会保证旧的绑定不能再被使用——这就完全避免了"两个指针指向同一块内存然后double free"的经典C++问题。

内存视角：let s2 = s1时，s1的栈上数据（指针、长度、容量）被复制给s2，但堆上的字符串内容没有被复制。然后Rust编译器把s1标记为"已移动"，使其失效。所以堆内存只被s2拥有——当s2离开作用域时，它被安全地释放一次。

函数传参与返回值

所有权在函数边界上也遵循同样的规则：

fn take_ownership(s: String) {
    println!("{s}");
}  // s 在这里被drop释放

fn main() {
    let s = String::from("hello");
    take_ownership(s);
    // println!("{s}");  // 错误！所有权已经移入函数
}

同样，函数返回值也会转移所有权：

fn give_ownership() -> String {
    String::from("world")
}

fn take_and_give(s: String) -> String {
    s  // 返回所有权——其实就是个move
}

fn main() {
    let s1 = give_ownership();
    let s2 = take_and_give(s1);
    // 现在 s1 已不可用，s2 拥有字符串
}

你可能会觉得这样做很麻烦——传一个参数进去，还要传回来才能继续用。别急，这就是为什么我们需要引用。

Clone与Copy

Copy类型：栈上数据的特例

像整数、浮点数、布尔、字符这些纯栈上数据，它们的赋值行为不是move，而是Copy：

let x = 42;
let y = x;     // 这不是move，是Copy
println!("{x}");  // 没问题！x仍然可用
println!("{y}");  // y也有值42

// 为什么？因为i32只占4字节，全部在栈上
// "复制"就是逐位拷贝，几乎没有开销

实现了Copy trait的类型，在赋值或传参时不会发生所有权的转移——旧绑定仍然有效。常见Copy类型：所有整数/浮点数、bool、char、元素都是Copy类型的元组（如(i32, i32)）。

Clone：显式深拷贝

对于堆上数据（如String），如果你真的想复制一份数据而不是转移所有权，可以显式调用.clone()：

let s1 = String::from("hello");
let s2 = s1.clone();

println!("{s1}");  // 没问题，s1仍然持有所有权
println!("{s2}");  // s2是独立的副本

.clone()会复制堆上的数据，因此有运行时开销。但至少它是显式的——你每次看到.clone()，都知道这里有数据拷贝发生。这一点比C++的隐式深拷贝要透明得多。

不要随手.clone()！刚从其他语言转Rust时，一个常见的坏习惯是：为了通过编译器的借用检查，在不确定的地方随手加上.clone()。这虽然能让代码编译通过，但引入了不必要的性能开销。正确做法是：先思考数据的所有权关系，用引用（下一节马上讲）来避免不必要的拷贝。只有在确实需要一份独立副本时才使用.clone()。

引用与借用

引用（Reference）是Rust所有权的"逃生舱"——它允许你在不转移所有权的情况下访问数据。

不可变引用 &T

fn calc_len(s: &String) -> usize {
    s.len()
}  // s是引用，离开作用域不会释放它指向的数据

fn main() {
    let s = String::from("hello");
    let len = calc_len(&s);  // 传递引用：把s"借"给函数
    println!("'{s}' 的长度是 {len}");  // s仍然可用
}

这里的&s叫做"借用"——函数暂时借用了s的使用权，但没有拿走所有权。函数结束后，借用结束，s仍然是它的所有者。

可变引用 &mut T

如果函数需要修改借用的数据：

fn append_world(s: &mut String) {
    s.push_str(", world");
}

fn main() {
    let mut s = String::from("hello");
    append_world(&mut s);
    println!("{s}");  // "hello, world"
}

注意：要获得可变引用，原变量本身必须是mut的，且用&mut s传递。

悬垂引用（Dangling Reference）

悬垂引用是指引用指向的内存已经被释放了。Rust编译器在编译阶段就杜绝了这种情况：

fn dangle() -> &String {
    let s = String::from("hello");
    &s
}  // s被释放，但引用被返回了——悬垂引用！
// 编译器会直接报错：missing lifetime specifier
悬垂引用是"内存不安全"的头号公敌。C/C++中你可以在函数返回局部变量的指针/引用——编译器最多给个warning，但能编译通过。运行时那片内存可能已经被其他数据覆盖了，导致各种诡异的bug。Rust编译器直接判定这是非法操作——你的程序根本跑不起来。

借用规则

Rust的借用规则可以用一句话概括：

同一时间，要么只有一个可变引用，要么有多个不可变引用，但不能两者共存。

具体拆分为三条规则：

规则 含义 违反后果
规则1 任意时刻只能有一个可变引用（&mut T） 编译错误：cannot borrow as mutable more than once
规则2 任意时刻可以有多个不可变引用（&T） （这是允许的）
规则3 可变引用和不可变引用不能同时存在 编译错误：cannot borrow as mutable because it is also borrowed as immutable 
// 规则1：一个可变引用
let mut s = String::from("hello");
let r1 = &mut s;
let r2 = &mut s;  // 编译错误！不能有两个可变引用

// 规则2：多个不可变引用
let s = String::from("hello");
let r1 = &s;
let r2 = &s;      // 正确
println!("{r1}, {r2}");

// 规则3：可变与不可变不能共存
let mut s = String::from("hello");
let r1 = &s;
let r2 = &mut s;  // 编译错误！已有不可变引用时不能创建可变引用

引用的作用域

引用的作用域从它被创建开始，到最后一次使用结束（而不是到花括号结束）。这被称为NLL（Non-Lexical Lifetimes，非词法生命周期），是Rust 2018 Edition引入的改进：

let mut s = String::from("hello");

let r1 = &s;
let r2 = &s;
println!("{r1}, {r2}");  // 最后一次使用不可变引用

let r3 = &mut s;  // 现在可以！因为r1/r2已经不再被使用
println!("{r3}");

这个机制让代码更灵活——你不再需要为了满足借用检查器而手动加上花括号来缩小引用作用域。编译器足够聪明，知道引用什么时候"退休"。

记忆窍门：把引用想象成"图书馆的借阅证"。不可变引用是"阅览证"——很多人可以同时看同一本书（只读）。可变引用是"外借证"——一次只能有一个人把书借走（修改）。你不能在有人正在阅览时把书借走，也不能让两个人同时把同一本书借走。规则很朴素，对吧？

本章小结

这一章我们覆盖了Rust最核心的概念——所有权。三条规则、Move与Copy的区别、引用与借用的三种形式、以及借用检查的三条铁律。这可能是Rust学习曲线中最陡峭的一段，但也是Rust最值得学习的地方。理解所有权后，你再看C/C++中的内存泄漏、悬垂指针、数据竞争等问题，会发现它们本质上是"所有权规则未被遵守"的表现。

下一章，我们将把这些知识应用到实践中——学习Rust中最常用的两种数据类型：切片与字符串。你会看到所有权系统如何让字符串操作既安全又高效。
