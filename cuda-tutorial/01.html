GPU计算的前世今生

大家好，我是你们的讲师。在正式开始写任何一行CUDA代码之前，我想先花一整章的篇幅，带大家回到故事的开头。你可能会问："学一门技术直接上手不就好了，为什么要听历史课？" 我的回答是：GPU编程和CPU编程的思路有着本质区别——这种区别不是语法层面的，而是思维模型层面的。不理解GPU"从哪里来"，你就很难真正理解它"为什么长这样"，更不用说写出高性能的CUDA代码了。

这一章，我们会从1990年代的图形卡一路走到2026年的CUDA 13.3，看看这个当初只为了画三角形而生的硬件，是如何一步步变成今天驱动着全球AI基础设施的计算引擎的。理解这段历史，你将清楚地知道：为什么GPU有那么多的线程？为什么内存有那么多的层次？为什么我们需要学习一种全新的编程思维？

读史使人明智。 这一章不要求你记住任何API或语法——但读完它，你应该能回答这个问题："为什么GPU适合并行计算，而CPU更适合串行任务？" 带着这个理解，后面的每一章都在为这个答案增添细节。

从像素到计算——GPU的起源 (~1990s)

1990年代，如果你是个游戏玩家，那你一定听说过"3D加速卡"这个名字。那时候所谓的"显卡"其实是个非常单纯的设备——它的任务就是把CPU送过来的三角形数据变成屏幕上的像素点。这就是所谓的固定功能管线（fixed-function pipeline）：顶点进来，经过变换、光照、光栅化、纹理映射，像素出去。每个环节都是硬件写死的，程序员能做的只有调整参数。

转折点出现在2001年。NVIDIA发布了GeForce 3——这是世界上第一款支持可编程着色器（programmable shader）的消费级GPU。开发者突然可以自己写一段小程序，控制顶点和像素的处理方式了。虽然这些程序极短（通常只有几十条指令），用的也是类似汇编的古怪语言，但"可编程"这三个字打开了一扇全新的大门。

更有意思的是，一些脑子转得快的图形程序员发现了一个秘密：GPU本质上是一台巨大的并行浮点数计算器。你把数据打包成纹理（texture），把计算逻辑写成像素着色器（pixel shader），然后画一个覆盖全屏的四边形——GPU就会用成百上千个核心并行运算每个像素，而你得到的"渲染结果"其实就是计算结果。这听起来像是一个疯狂的 hack，但在2000年代初期，这恰恰是许多人做高性能计算的唯一途径。

你知道吗？ 早期GPU编程圈有一个著名的段子："你的算法能渲染吗？"意思是——如果你想把问题交给GPU处理，你得先把它包装成一个图形渲染问题。矩阵乘法？那是纹理采样。物理模拟？那是顶点变换。神经网络？那是像素着色器。这种方法虽然别扭，但它证明了GPU拥有巨大的计算潜力。

GPGPU时代——用图形API做科学计算 (~2000-2006)

"GPGPU"全称是General-Purpose computing on Graphics Processing Units。这个词在今天听起来平平无奇，但在2000年代初期，它代表着一种极其叛逆的计算哲学：用游戏显卡跑科学计算。

那个年代的研究者们面对的困境，今天的你很难想象。假设你是一位计算物理学家，想用GPU加速你的模拟——你不能像现在这样写 cudaMalloc() 然后 kernel<<<grid, block>>>()。你必须:

  • 把你的数组打包成OpenGL纹理（texture），注意纹理的尺寸必须是2的幂
  • 把你的计算逻辑写成GLSL像素着色器（pixel shader），用 texture2D() 来"读取"数据
  • 把计算结果写回到帧缓冲（framebuffer），然后从显存拷回CPU
  • 最要命的是——你没法在着色器之间直接传递任意数据，每个阶段的输出只能是颜色值（RGBA四个float）

下面是一个2005年左右风格的"GPGPU代码"的样子。是的，你没看错——这居然是在做科学计算：

// 2005年的"GPU计算"——用像素着色器做向量加法
// 你看到的每一行都在"渲染"，实际上是在算数

uniform sampler2D inputA, inputB;
uniform vec2 texelSize;

void main() {
    // "采样纹理" = 读取数组元素
    vec4 a = texture2D(inputA, gl_TexCoord[0].st);
    vec4 b = texture2D(inputB, gl_TexCoord[0].st);
    vec4 sum = a + b;      // 向量加法

    // "输出颜色" = 写入结果
    gl_FragColor = sum;
}

这其中的荒谬之处不言而喻：纹理采样器有数量限制，纹理尺寸有奇怪的对齐要求，中间结果只能以图像形式存储从而引入精度损失，出了bug你只能盯着屏幕上一堆奇怪的彩色条纹发呆——因为没有printf，没有断点，没有任何调试工具。当时有个著名的吐槽："写GPGPU程序就像在监狱里做数学——你能做，但每一步都无比憋屈。"

真实的"踩坑"故事： 我认识一位在2004年做流体模拟的研究员，为了调试一个GPU上的数值误差，他花了整整三周。最后发现原因是纹理插值（texture filtering）默认是开启的，导致GPU在读取他"数组"中的相邻元素时自动做了线性插值——他以为是数组读取，实际上硬件把它当成了图像采样。你们看，当计算被伪装成渲染时，连"读一个数"这种最基本的操作都可能被硬件"好心办坏事"。这种痛苦直接催生了一个想法：为什么不能让GPU提供一个真正的、不依赖图形API的计算接口？这个想法，就是CUDA的种子。

CUDA的诞生——2006年的范式革命

2006年秋天，NVIDIA做出了一个在当时看来极其冒险的决定。他们发布了G80架构（GeForce 8800 GTX），这是世界上第一款统一着色器架构（unified shader architecture）的GPU。什么意思呢？在G80之前，GPU内部有独立的顶点着色器单元和像素着色器单元，各自独立、数量固定。如果某个游戏需要大量顶点计算，像素单元就在闲着；反过来也如此。G80将这些全部统一成了通用的流处理器（stream processor），可以根据负载动态分配。

这个硬件变革背后，有一个关键人物不得不提：Ian Buck。Ian Buck在斯坦福读博期间就是GPGPU领域的先驱，他的博士论文题目就是"Stream Computing on Graphics Hardware"。毕业后他加入NVIDIA，和John Nickolls一起领导了一个秘密项目——目标是让GPU可以用C语言来编程，不再需要OpenGL或者DirectX。

2007年6月，CUDA 1.0正式发布。当时的发布内容其实很朴素：

  • C语言扩展：加了一小撮关键字（__global__、__device__、__shared__），你可以用熟悉的C语法写GPU程序
  • nvcc编译器：把CUDA代码编译成GPU能执行的PTX指令
  • CUDA API：管理设备内存、启动内核、同步——一切都不需要图形API了
  • 第一块CUDA显卡：GeForce 8800 GTX，768MB显存，128个流处理器

从此，程序员只需要写这样的代码，而不是上面那堆纹理采样的把戏：

// CUDA 1.0 —— 真正的GPU计算
__global__ void vecAdd(float *a, float *b, float *c, int n) {
    int i = threadIdx.x + blockIdx.x * blockDim.x;
    if (i < n) {
        c[i] = a[i] + b[i];
    }
}

int main() {
    float *d_a, *d_b, *d_c;
    cudaMalloc(&d_a, n * sizeof(float));
    cudaMalloc(&d_b, n * sizeof(float));
    cudaMalloc(&d_c, n * sizeof(float));

    cudaMemcpy(d_a, h_a, n * sizeof(float), cudaMemcpyHostToDevice);
    cudaMemcpy(d_b, h_b, n * sizeof(float), cudaMemcpyHostToDevice);

    vecAdd<<<128, 256>>>(d_a, d_b, d_c, n);

    cudaMemcpy(h_c, d_c, n * sizeof(float), cudaMemcpyDeviceToHost);
    return 0;
}
划时代的意义在于： 第一次，你不必把自己伪装成图形程序员来做科学计算了。CUDA的核心理念——"GPU是一种强大的协处理器，CPU负责控制，GPU负责大规模并行计算"——至今仍然是异构计算的基石。今天你看到的几乎所有AI加速库（PyTorch、TensorFlow、JAX）底层的GPU支持，都建立在这个理念之上。

CUDA生态的演进——从边缘到主流 (~2008-2020)

CUDA最初的几年其实非常艰难。2007-2009年，质疑声铺天盖地："谁需要用显卡做通用计算？这不过是个小众玩具。" Intel和AMD也分别推出了Larrabee和Stream SDK作为竞品。但NVIDIA坚持了每两年一代架构的节奏，每一代都带来重要的特性。下面这张表能帮你快速把握这段演化史：

架构 发布年份 代表显卡 关键创新 对CUDA开发者的意义
Fermi 2010 GTX 480 ECC显存、L1/L2缓存层次、真正的C++支持 可以写类、继承、虚函数——GPU编程不再像嵌入式开发
Kepler 2012 GTX 680 / K20 Dynamic Parallelism、Hyper-Q、Grid Management Unit GPU内核可以自己启动新内核——递归并行成为可能
Maxwell 2014 GTX 980 / M40 三级缓存优化、共享内存与L1分离、更低功耗 更高效的线程调度，为深度学习训练奠定基础
Pascal 2016 GTX 1080 / P100 Unified Memory、NVLink、16nm FinFET 写代码时可以不再关心显存在哪——cudaMallocManaged 自动迁移
Volta 2018 V100 Tensor Cores、独立线程调度、SM拆分 AI训练的革命——FP16矩阵运算吞吐提升8倍，Transformer开始起飞
Ampere 2020 RTX 3080 / A100 第三代Tensor Cores、MIG（多实例GPU）、稀疏化支持 一张GPU可以切分成多个实例、BF16支持、TF32精度

你可能注意到一个有趣的巧合：Volta架构和Transformer论文是同一年出现的。2017年"Attention Is All You Need"发表，2018年V100带着第一代Tensor Cores上市。这绝不仅仅是巧合——Volta的设计团队在架构规划时就深度调研了深度学习社区的需求。Tensor Cores专门为矩阵乘累加（GEMM）优化，而矩阵乘法恰恰是Transformer的核心运算。硬件和算法的协同进化，在GPU的历史上体现得淋漓尽致。

与此同时，软件生态也在快速膨胀。2012年AlexNet在GPU上训练的惊人效果点燃了深度学习的第一波热潮；2014年NVIDIA发布了cuDNN——一个专门优化深度学习算子的GPU库，让框架开发者不用自己写底层CUDA代码；2015年TensorFlow和2016年PyTorch相继选择CUDA作为唯一的GPU后端。到了2020年，已经没有任何一个严肃的AI项目能绕开CUDA了。

一个有趣的视角： 如果说Windows 95是"让每个人都能用PC"，那么CUDA就是"让每个人都能用GPU"。在CUDA出现之前，GPU编程是少数图形学博士的专属技能；在CUDA成熟之后，一个本科生花一个下午就能写出能在GPU上跑的并行程序。这种"民主化"的进程，才是CUDA对整个计算产业最深远的贡献。

现代CUDA全景 (~2020-2026)

进入2020年代，CUDA的发展速度不仅没有放缓，反而因为AI大模型的爆发而加速了。看看最近几年的关键节点：

  • Hopper (2022)：H100带来FP8 Transformer Engine、第四代Tensor Cores、DPX指令集、更强大的NVLink Switch系统——为大模型训练而生
  • Blackwell (2024-2025)：B200/B100采用双芯片设计，FP4支持、第二代Transformer Engine、第五代NVLink——专为推理优化
  • Rubin (即将到来)：据NVIDIA官方路线图，R100将采用全新架构，预计在2026-2027年面世

而在软件层面，截至2026年6月，CUDA的最新版本是CUDA 13.3（2026年5月发布）。这一版本带来了几个值得关注的新特性：

  • Tile Programming in C++：在C++层面引入了类似CuTe的tile抽象，让矩阵分块编程更加直观
  • CUDA Python 1.0：终于有了官方的Python编程接口，Python开发者可以在不写C代码的情况下利用CUDA的全部能力
  • CompileIQ 自动调优：编译时根据目标GPU型号自动选择最佳内核配置参数，告别手工tuning

从2007年CUDA 1.0发布时的128个流处理器、768MB显存，到今天H100的18432个CUDA核心、80GB HBM3显存——不到二十年，GPU的并行计算能力增长了超过两个数量级。而更令人震撼的是，这种增长还在持续：NVIDIA保持了"两年一代架构"的节奏在2026年依然没有放缓的迹象。

所以，为什么是现在学CUDA？ 因为在AI时代，理解GPU编程已经不是"加分项"而是"基本功"了。无论是训练大模型、部署推理服务，还是做高性能计算、图形渲染，CUDA都是那座绕不开的桥。今天全球排名前500的超算中，绝大多数都依赖GPU加速；每一个你用过的AI产品——从ChatGPT到Midjourney——底层都跑在CUDA代码上。学会CUDA，你就拿到了这个时代最强大的计算工具的钥匙。
