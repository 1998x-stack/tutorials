大家好，我是你们的讲师。上一章我们走完了汇编阶段的全部旅程——bootsect.S、setup.S、head.S共同完成了从实模式到保护模式再到分页的跨越，最终跳转到了C语言世界。现在，CPU在init/main.c的main()函数中执行，内核的"安营扎寨"工作正式开始了。

如果你从头读完了引导代码，你可能会有一个感觉：汇编阶段虽然复杂，但目标很明确——"做保护模式和分页，然后跳转"。而从main()函数开始，事情变得比汇编更"杂"——内存初始化、中断设置、调度器、终端、定时器、文件系统……主函数像一位指挥家，一个接一个地唤醒各个子系统的"乐手"。

这一章，我们沿着main()函数的调用序列，看看每个子系统初始化的核心逻辑。

引言——进入main.c，内核开始"安营扎寨"

在分析代码之前，我们要先理解一个关键问题：当main()开始执行时，硬件环境处于什么状态？

  • CPU模式：32位保护模式 + 分页已启用
  • 内存布局：system模块在0x00000 ~ 0x8FFFF，内核的代码和数据已经就位
  • 栈指针：指向一个合理的栈空间（在head.S中设置）
  • 中断：IDT已经设置但内容为空（所有中断处理函数还未注册），中断仍关闭
  • A20门：已开启
  • 页表：恒等映射（虚拟地址 == 物理地址），覆盖低端8MB

也就是说，硬件层已经准备好了，但操作系统层还是一片空白。main()需要为每个子系统分配资源、注册服务、建立数据结构，最后打开中断，进入用户态。这就像一个城市建设的总规划师——道路已经修好（硬件基础），现在要安装红绿灯、铺设水管、建立警察局和学校。

main.c初始化全景

让我们直接看main()函数的调用序列。我会在调用序列旁边标注每个函数的职责：

void main(void)
{
    // 以下所有初始化都在中断关闭(CLI)的状态下进行

    // ------- 核心子系统初始化 -------
    mem_init();          // 内存初始化: 标记内核占用, 初始化mem_map
    trap_init();         // 中断向量初始化: 填充IDT表项
    blk_dev_init();      // 块设备初始化: 清空请求队列(request[32])
    chr_dev_init();      // 字符设备初始化: (空操作)
    tty_init();          // 终端初始化: 串口+控制台+键盘
    time_init();         // 时间初始化: 设置8253 PIT定时器
    sched_init();        // 调度器初始化: 任务0的TSS+LDT

    // ------- 数据和I/O初始化 -------
    buffer_init();       // 缓冲区初始化: 建立缓冲头和数据区
    hd_init();           // 硬盘初始化: 注册中断处理
    floppy_init();       // 软盘初始化: 注册中断处理

    // ------- 开启中断, 进入用户态 -------
    sti();               // 开中断! (set interrupt flag)
    move_to_user_mode(); // 从内核态"模拟"进入用户态(任务0)

    // ------- 第一个真正的进程 -------
    init();              // fork出init进程(任务1)
}

// init() 内部:
void init(void)
{
    int pid = fork();    // 创建进程1(init)
    if (pid)             // 父进程(任务0)
        while(1) wait(NULL); // 等待子进程
    else                 // 子进程(init)
    {
        open("/dev/tty0", O_RDWR);  // 打开终端
        execve("/bin/sh", ...);     // 执行shell
    }
}

这个序列非常清晰地反映了一件事情：初始化顺序至关重要。例如，time_init()必须在sched_init()之前——因为调度器依赖jiffies（系统滴答）来切换进程，而jiffies的增量来自8253定时器中断，这个中断必须在trap_init()设置了IDT之后、调度器启动之前配好。

每个初始化函数的依赖关系和核心产出可以总结如下：

函数依赖核心产出
mem_init()无（最先调用）mem_map数组，标记已占用/空闲物理页
trap_init()IDT已加载（head.S完成）填充IDT的256个表项，注册异常处理函数
blk_dev_init()无清空块设备请求队列request[32]
chr_dev_init()无（空操作，仅占位）
tty_init()trap_init()（需要中断）串口/控制台/键盘初始化，注册中断处理
time_init()trap_init()（需要时钟中断）配置8253 PIT定时器，设置100Hz时钟中断
sched_init()time_init()（需要jiffies）任务0的TSS/LDT，调度器数据结构
buffer_init()mem_init()（需要内存）缓冲头和数据区（磁盘I/O缓存）
hd_init()buffer_init() + blk_dev_init()硬盘中断处理注册
floppy_init()同上软盘中断处理注册
编程启示：main()函数的代码质量非常高——它只有大约200行（包括init()），但结构极其清晰。每一个初始化函数的名字都精确描述了它的作用，调用顺序就是依赖关系的体现。这是"自我文档化代码"的典范——你不需要额外的文档，只看函数名和它们的调用顺序，就能理解整个初始化流程。

内存初始化mem_init()

mem_init()是main()调用的第一个函数。它要做的事情非常直白：告诉内核"哪些物理内存已经被我占了，剩下的你可以自由分配"。

// mm/memory.c 中的 mem_init()
// 定义在 mm/memory.c 中

void mem_init(long start_mem, long end_mem)
{
    int i;

    // HIGH_MEMORY = 实际物理内存大小(字节)
    // 在0.11中, HIGH_MEMORY通常为 16MB 或 4MB 或 2MB

    // 将mem_map数组全部置0
    // mem_map数组是内存页面的"位图" —— 0表示空闲
    for (i = 0; i < PAGING_PAGES; i++)
        mem_map[i] = 0;

    // PAGING_PAGES = HIGH_MEMORY / 4096
    // 对于16MB内存: 16 * 1024 * 1024 / 4096 = 4096个页

    // 标记内核占用的内存页
    // start_mem 到 end_mem 是内核代码+数据已占用的物理地址范围
    i = (start_mem - LOW_MEM) >> 12;   // 起始页号
    while (i < (end_mem - LOW_MEM) >> 12)
        mem_map[i++] = 255;             // 255 = 已占用(不可分配)
}

// low memory = 1MB (0x00000 ~ 0xFFFFF), 这部分不通过mem_map管理
// 实际可用的内存从 1MB 以上的地址开始

这个函数非常直观。mem_map是一个数组，每个元素代表一个4KB物理页面的状态：0表示空闲，255表示已占用或被保留。内核在引导阶段已经占用了从start_mem到end_mem的物理页面——这部分通过main()的参数传入（实际上是head.S提前计算好的）。

注意一个细节：LOW_MEM = 1MB。在Linux 0.11的设计中，物理地址0x00000 ~ 0xFFFFF（低端1MB）被保留给引导代码、BIOS数据、DMA缓冲区等特殊用途，不纳入页分配器管理。实际的可分配物理内存从1MB以上开始。

这个过程的结果就是：比如你的虚拟机有16MB内存，mem_map[256] ~ mem_map[4095]（对应1MB ~ 16MB的页面）被初始化为0（空闲），然后从start_mem到end_mem对应的页面被标记为255（已占用），剩下的页面就是内核可以分配给进程的自由内存。

各子系统初始化

接下来main()会依次调用多个初始化函数。每个函数只做非常具体的一两件事——这是良好模块化设计的体现。让我们逐个看：

trap_init() —— 设置IDT

// kernel/traps.c

void trap_init(void)
{
    int i;
    // 设置保留的中断向量号
    // set_trap_gate() 将中断/异常的处理函数地址填入IDT表项

    set_trap_gate(0, ÷_error);      // 除以零错误
    set_trap_gate(1, &debug);
    set_trap_gate(2, &nmi);
    set_trap_gate(3, &int3);         // 断点指令
    set_trap_gate(4, &overflow);
    set_trap_gate(5, &bounds);
    set_trap_gate(6, &invalid_op);
    set_trap_gate(7, &device_not_available);
    set_trap_gate(8, &double_fault);
    set_trap_gate(9, &coprocessor_segment_overrun);
    set_trap_gate(10, &invalid_TSS);
    set_trap_gate(11, &segment_not_present);
    set_trap_gate(12, &stack_segment);
    set_trap_gate(13, &general_protection);
    set_trap_gate(14, &page_fault);   // 缺页异常
    set_trap_gate(15, &reserved);
    set_trap_gate(16, &coprocessor_error);

    // 系统调用门 —— 允许用户态程序通过INT 0x80进入内核
    set_system_gate(0x80, &system_call);
}

注意set_trap_gate和set_system_gate的区别：前者设置的是陷阱门（Trap Gate），后者设置的是系统门（System Gate）。两者的关键区别在于特权级——陷阱门的DPL=0（仅内核态可触发），系统门的DPL=3（用户态也可以触发）。这就是为什么INT 0x80系统调用能从用户态调用——它的门描述符的DPL=3。

tty_init() —— 终端初始化

这个函数在kernel/chr_drv/tty_io.c中实现。它做的事情看起来简单，背后却非常复杂：初始化串口（RS232）、初始化控制台、初始化键盘。每个初始化都会注册对应的中断处理函数，设置缓冲区（读队列和写队列），以及建立终端参数结构。

time_init() —— 时间系统初始化

这个函数配置了8253可编程间隔定时器（PIT）。8253是IBM PC上的标准定时器芯片，它通过IRQ 0产生周期性时钟中断。time_init()设置8253的计数初值，使其以约100Hz（每10ms一次）的频率产生中断。

sched_init() —— 调度器初始化

// kernel/sched.c

void sched_init(void)
{
    int i;
    // 初始化任务0的TSS和LDT
    // 注意: 任务0就是当前正在执行的"内核进程"
    // 在Linux 0.11中, 任务0是"空闲任务"(idle)

    // 设置TSS0和LDT0
    set_tss(0);        // 将任务0的TSS填入GDT[(TSS0_INDEX)]
    set_ldt(0);        // 将任务0的LDT填入GDT[(LDT0_INDEX)]

    // 清除所有任务槽(slot), 设置任务0为运行态
    for (i = 0; i < NR_TASKS; i++)
        task[i] = NULL;         // 所有任务槽初始为空
    task[0] = &task0;           // 任务0指向init_task结构

    // 设置8253定时器的中断处理
    // 定时器中断(IRQ 0)对应INT 0x20
    set_intr_gate(0x20, &timer_interrupt);

    // 设置系统调用门
    set_system_gate(0x80, &system_call);
}

这里有个微妙之处：任务0（task 0）在sched_init()中初始化，但它此时并不是一个真正的用户态进程——它其实就是当前正在运行的内核。main()函数、init()函数、以及内核的所有初始化代码，实际上都是在"任务0"的上下文中执行的。当后续执行move_to_user_mode()时，任务0才被"伪装"成用户态进程。

move_to_user_mode()——内核态的"伪装"

这个函数是Linux 0.11中最精巧的小函数之一。它的目标是：在当前代码执行流中，模拟一次从用户态到内核态的中断返回——从而让CPU认为当前进程是从用户态进来的。

// include/asm/system.h

#define move_to_user_mode() \
    __asm__("movl %%esp,%%eax\n\t"     \
            "pushl $0x17\n\t"          \   // SS = 0x17 (用户数据段选择子)
            "pushl %%eax\n\t"          \   // ESP = 当前栈指针
            "pushfl\n\t"               \   // EFLAGS
            "pushl $0x0f\n\t"          \   // CS = 0x0F (用户代码段选择子)
            "pushl $1f\n\t"            \   // EIP = 下一条指令的地址
            "iret\n"                   \   // iret — 中断返回!
            "1:"
            : : )

// 这段代码的原理:
// iret指令从栈中弹出 EIP → CS → EFLAGS → ESP → SS
// 从而完成从Ring 0到Ring 3的模式切换
//
// 伪造的栈帧:
//   +----------------+
//   | SS = 0x17      |  ← 用户数据段(特权级3)
//   +----------------+
//   | ESP            |  ← 当前栈顶(内核栈)
//   +----------------+
//   | EFLAGS         |  ← 当前标志寄存器
//   +----------------+
//   | CS = 0x0F      |  ← 用户代码段(特权级3)
//   +----------------+
//   | EIP = 标号1:   |  ← 返回地址(连续执行)
//   +----------------+
//
// iret 执行时, CPU检测到返回目标的特权级(3)不同于当前(0),
// 于是将栈切换到用户栈, 并将特权级降低到Ring 3.
为什么需要"模拟"而不是"创建"？ 在Linux 0.11中，任务0实际上就是内核本身。它并不是通过fork()创建的——fork()需要一个已有的进程模板。Linus的做法是：直接让内核执行流"变成"一个用户态进程。这种方法简单直接——不需要为任务0单独创建进程控制块，也不需要经过fork的复杂流程。但它的代价是：任务0比较特殊，不能执行一般的用户程序（它没有自己的用户态代码段），所以任务0的唯一职责就是调用init()，然后等待子进程结束。

init()——第一个真正的进程

当move_to_user_mode()完成后，CPU实际上仍然在执行main()函数末尾的init()调用——只不过此时CPU已经处于用户态（Ring 3）了。init()中才是真正的"用户态进程"的开始：

// init/main.c

void init(void)
{
    int pid;

    // 第一次调用fork() —— 这是内核中的第一个系统调用
    // fork() 会创建一个新进程(任务1), 完全复制任务0的地址空间
    pid = fork();
    if (pid)    // pid != 0 —— 当前是父进程(任务0)
    {
        // 任务0的唯一工作: 等待子进程结束
        // wait()会让任务0进入睡眠状态
        while(1)
            wait(NULL);
    }
    else        // pid == 0 —— 当前是子进程(任务1)
    {
        // 任务1(init进程)的工作:
        // 1. 打开标准输入/输出/错误设备
        open("/dev/tty0", O_RDWR);  // fd = 0 (stdin)
        open("/dev/tty0", O_RDWR);  // fd = 1 (stdout)
        open("/dev/tty0", O_RDWR);  // fd = 2 (stderr)

        // 2. 执行shell
        // execve() 将当前进程的内存映像替换为/bin/sh
        execve("/bin/sh", argv, envp);

        // 如果execve返回, 说明失败了
        _exit(0);
    }
}

这就是你在Bochs屏幕上看到/ #提示符的全过程。fork()创建了一个子进程（init进程），init进程打开终端设备，然后执行/bin/sh。shell程序运行后，打印提示符，等待用户输入。

注意：fork()是第一次真正从用户态通过INT 0x80进入内核的系统调用。在此之前的move_to_user_mode()虽然让CPU进入了用户态，但并没有使用任何系统调用。所以，fork()是内核系统调用机制的"第一次实战检验"——如果syscall门或相关处理函数设置有误，fork()就会失败，系统将无法启动。

踩坑故事——初始化顺序的血泪教训

main()函数中初始化函数的顺序看似平淡无奇，但Linus在早期版本中在这个顺序上吃过苦头。

最经典的一个问题是：time_init()必须在sched_init()之前调用。为什么？因为sched_init()中设置了定时器中断的处理函数（set_intr_gate(0x20, &timer_interrupt)），而定时器中断处理函数每次被调用时会递增全局变量jiffies。如果time_init()没有提前配置好8253定时器芯片，定时器中断就不会产生，jiffies就不会增长——结果就是调度器的schedule()函数永远不会被调用，因为schedule()只在do_timer()中触发，而do_timer()又是定时器中断处理函数调用的。

在Linux 0.11的开发过程中，有一次Linus把time_init()放在了sched_init()之后。结果内核启动后，init进程fork出来之后，父进程（任务0）调用wait()进入睡眠，但schedule()从未被调用——因为jiffies从未增长——所以系统完全卡死。Linus花了好几个小时才找到这个Bug。

教训总结：内核初始化中的依赖关系往往不是显式的。time_init()和sched_init()之间并没有明显的函数调用关系——它们的依赖是通过硬件中断机制间接形成的。这就是为什么阅读内核代码时，不能只看函数调用图，还要理解硬件交互的隐式依赖。每个初始化函数做了什么事情、设置了什么硬件状态、影响了哪些全局变量——这些都需要综合理解。

类似的隐式依赖还有很多：trap_init()必须在任何可能触发中断的代码之前（否则中断来了找不到处理函数，系统直接崩溃）；buffer_init()必须在hd_init()之前（因为硬盘驱动需要缓冲区）；chr_dev_init()虽然是空函数（只是登记了设备号），但它必须出现在tty_init()之前，因为tty依赖字符设备框架的初始化。

这些依赖关系构成了内核初始化的"关键路径"。现代Linux内核的初始化顺序比0.11复杂了数百倍，但核心逻辑是一样的——只不过现代内核使用了一种更系统化的方式（如initcall机制）来声明和排序初始化函数，而不是像0.11这样在main()中硬编码调用顺序。

好了，main()函数的工作到此结束。从初始化子系统开始，到开启中断、进入用户态、执行shell——内核现在已经在全速运转了。在进入用户代码之前，还有最后一个重要的体系结构基础需要理解：x86保护模式下的各种机制——GDT、LDT、分页、特权级、任务切换。
