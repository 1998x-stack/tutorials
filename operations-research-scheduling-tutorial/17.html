第17章 实际工程案例分析

大家好，我是你们的讲师。如果你一直跟着这门课学到这里，恭喜你——你已经掌握了运筹学与调度理论的"十八般武艺"：线性规划、整数规划、动态规划、元启发式、CP-SAT、求解器选型……但我知道你心里可能还藏着一个问题——"这些东西在真实的工厂、港口、医院里到底是怎么用的？"

把教科书上的算法变成能赚钱的系统，中间隔着一条巨大的鸿沟。我见过太多"论文漂亮无比、落地一塌糊涂"的调度项目。问题不在于算法不行，而在于建模时忽略的细节，上线后会加倍报复你。今天，我们用三个真实行业的案例来打通"从理论到工程"的最后一公里。

半导体晶圆厂调度 — 最复杂的调度场景之一

先说结论：半导体晶圆制造（Semiconductor Wafer Fabrication）可能是人类已知的最复杂的离散制造调度问题。一个 300mm 晶圆厂，投资上百亿美元，一片晶圆要走 300-600 道工序，周期长达 12 周。而更"变态"的是它的 Re-entrant Flow（重入流）——晶圆在光刻、刻蚀、沉积等工序间反复循环，同一台设备在工艺的不同阶段会反复被同一个工件访问多次。这在传统 Flow Shop 和 Job Shop 中是不可能出现的。

重入流带来的直接后果是——传统调度方法几乎全部失效。如果你试图用 MILP 建模全部工序，变量数量会瞬间爆炸。工业界实际的做法是靠派工规则（Dispatching Rules）加少量局部优化来管理。比如经典的"最小剩余加工时间优先"（Least Work Remaining, LWKR）在晶圆厂中被广泛使用。

我亲身经历过一个项目——某 12 英寸晶圆厂的刻蚀区经常出现"机台等待"现象，即一批晶圆在刻蚀机前排队，但后面的光刻机在空转。现场工程师的传统做法是让刻蚀区尽量"塞满"来提升设备利用率，但这反而拖慢了整个线。我们做了一个轻量级的调度优化：只在刻蚀区的前两道工序做 CP-SAT 局部排程，后续放行给派工规则。改动不大，但整条产线的 cycle time 缩短了 12%。核心教训是：在复杂系统中，不要试图一次性解决全局优化——局部优化 + 启发式规则的混搭才是最务实的方案。

下面是一个简化版的半导体调度数学规划模型，只建模"光刻-刻蚀"两步重入的最短路径部分：

# 简化半导体调度模型 — 仅展示关键约束结构
# 假设 n 个批次，每个批次需要 visit 光刻机(L)和刻蚀机(E)各两次

# 变量：x_{i,op,s} = 工件 i 的工序 op 在设备 s 上的开始时间
# y_{i,i',s} = 工件 i 和 i' 在设备 s 上的顺序（0-1）

# 目标：最小化最大完工时间 (makespan)
Minimize: Cmax

# 约束 1: 工序先后顺序（同一个批次的工序不能乱序）
Subject To:
  x_{i,op2,s} >= x_{i,op1,s'} + duration_{i,op1}  for all i, op1 < op2

# 约束 2: 同一设备每次只能加工一个批次（排他约束，大 M 形式）
  x_{i,op,s} + duration_{i,op} <= x_{i',op',s} + M * (1 - y_{i,i',s})
  x_{i',op',s} + duration_{i',op'} <= x_{i,op,s} + M * y_{i,i',s}

# 约束 3: 重入流约束 — 同一个批次的第二次光刻必须在第一次刻蚀之后
  x_{i,L2,s_L2} >= x_{i,E1,s_E1} + duration_{i,E1}

# 约束 4: 下限
  Cmax >= x_{i,lastOp} + duration_{i,lastOp}  for all i

这个模型如果扩展到全部 600 道工序会膨胀到天文数字，所以实际中只会用于"瓶颈工序窗口"的短周期排程。

半导体调度的特殊挑战：除了重入流，晶圆厂还有批处理机（Furnace）——一台炉子一次能同时处理 100-200 片晶圆，且要求同批晶圆的工艺参数完全相同。这意味着调度中要决策"哪些晶圆要凑成一批一起加工"。这种 Batch Scheduling + Job Shop 的混合体在经典文献中几乎没有现成模型，必须自己摸索。

港口集装箱码头调度

港口调度是一个"三层联动"的系统工程。当我们说"调度一艘船"，实际上涉及三个层次：

  • 岸桥调度（Quay Crane Scheduling）—— 决定哪台岸桥负责哪个贝位的集装箱装卸。岸桥之间不能交叉（一台岸桥不能跑到另一台的作业区），且装卸顺序必须考虑船舶稳定性。
  • 场桥/龙门吊调度（Yard Crane Scheduling）—— 岸桥卸下的集装箱由集卡运到堆场，龙门吊负责在堆场中存取。龙门吊在集装箱堆叠中"翻箱"的次数直接影响效率。
  • 集卡调度（Internal Truck Scheduling）—— 集卡在码头前沿和堆场之间穿梭，何时去哪台岸桥、何时返回，决定了岸桥的"喂料"速度。

三层协同中存在典型的"牛鞭效应"放大——岸桥的作业节奏稍有波动，集卡排队长度就会剧烈震荡，传导到堆场可能造成大堵车。传统做法是每一层独立优化，但效果有限。近年来，基于 Actor-Critic 深度强化学习的方法被尝试用于三层协同优化：将岸桥的装卸速度、集卡的分配策略、龙门吊的调度策略用一个联合 RL 模型统一学习。

上海洋山港作为全球最大的自动化码头之一，在这方面的实践走在前列。洋山港四期采用 AGV（自动导引车）替代传统集卡，AGV 的路径规划和充电调度完全由调度系统自动完成。他们的核心做法是"分层规划 + 局部重调度"——全局用数学规划做宏观调度（时间段粒度为 15 分钟），局部用规则引擎做微观调整（秒级响应）。这套系统上线后，码头整体作业效率提升了约 20%，能源消耗降低了 15%。

港口的"意外之美"：洋山港项目中最令人意外的优化点不是算法本身，而是AGV 充电策略的优化。之前的做法是所有 AGV 在电量低于 30% 时自动返回充电站。但调度系统发现，这种"各自为政"的充电策略会导致高峰时 AGV 大量离场充电，造成运力不足。部署者用 CP-SAT 重新建模了充电调度——让 AGV 在"不耽误下一个任务"的前提下灵活选择充电时机。结果 AGV 的可用率从 82% 提升到了 94%，而电池寿命也因避免了"低电量急充"而延长了。这就是一个建模视角的胜利。

手术室排程调度

最后来看一个"最有人情味"的调度场景——手术室排程。这里绝对不是一个"最小化完工时间"那么简单的问题。

手术室排程的核心矛盾是择期手术 vs 急诊手术。择期手术可以提前一周甚至一个月预约，急诊手术随时可能到来。如果给择期手术排太满，急诊来了没资源，病人有生命危险；如果给急诊留太多空档，择期手术的利用率又太低。这是一个典型的两阶段随机调度问题。

华西医院（中国最大的医院之一）的调度团队在这个问题上做了非常扎实的工作。他们的方法分为两步：

  • 阶段一（周排程）：基于历史数据，用场景随机规划生成一周的手术室分配计划。每个手术室分配"择期手术块"和"急诊预留块"，预留块的大小由历史急诊到达分布决定。
  • 阶段二（日调整）：当天早晨，根据实际急诊到达预测，微调手术顺序。如果急诊多，可以临时压缩部分择期手术的时长预估（基于外科医生的历史统计）。

他们的验证数据显示，这套方法将手术室利用率从 65% 提升到了 82%，同时急诊等待时间下降了 35%。更关键的是，医生满意度提高了——以前医生经常因为急诊插单而等到晚上 8 点才做自己的择期手术，有了预留块后大多数择期手术能在下午 4 点前完成。

但即使如此，这位项目的负责人告诉我，手术时长的不确定性仍然是最大的挑战。一台手术的预估时长和实际时长可能差 2-3 倍——同一类型的胆囊切除术，在年轻患者身上可能 45 分钟就做完，但遇到肥胖或粘连严重的病人要 2 小时。用均值和方差来建模远远不够，需要更细粒度的基于患者特征的条件概率分布（比如身高、体重、既往手术史）。

理论模型到实际部署的 Gap——教科书上的模型很少有可以直接上线的。比如手术调度，你翻阅运筹学论文可能会看到带随机时长的两阶段随机规划、带 CVaR 约束的风险规避模型，模型很精美，但你想过没有——谁会去采集并维护"手术时长条件分布"所需的几千个数据点？谁来做特征工程的清洗？信息系统里有没有这些字段？部署后医生不按建议的时间表来，你又怎么办？这些"非技术问题"占了项目成功率的 80%。

从案例中提炼的通用工程原则

这三个案例看似完全不同——半导体、港口、医院——但在工程层面，它们的调度系统建设有惊人的相似之处。我总结为三个通用原则：

维度 半导体晶圆厂 港口集装箱码头 手术室排程
核心约束类型 重入流 + 批处理机 设备互锁 + 空间约束 资源随机性 + 优先级
主要不确定性 设备故障 + 返工率 船舶到港时间 + 天气 急诊到达 + 手术时长
调度频率 班次（8小时）+ 热批次 15 分钟滚动窗口 每日 + 实时插单
求解方法 局部 CP-SAT + 派工规则 分层 MILP + 强化学习 两阶段随机规划 + 规则
最关键的"非技术"环节 与 MES 系统的数据接口 码头操作系统的数据质量 外科医生的配合

三个通用原则如下：

  1. 不要试图一次解决全局问题。半导体晶圆厂只优化瓶颈区域、港口分层规划、手术室两阶段决策——这些都指向同一句话：分而治之，局部优化。
  2. 不确定性管理比确定性优化更重要。所有案例中最大的困难都不是"求解"而是"应对变化"。调度方案再漂亮，一遇到设备故障或急诊就在纸面上失效了。所以工业调度系统的核心能力是快速重调度而非"一步到位的优化"。
  3. 80% 的工作量在数据，20% 在算法。如果你做过真正的工业调度项目，一定深有体会——对接 MES、清洗数据、解决字段缺失和异常值、和现场工程师确认约束的"潜规则"……这些才是真正耗时的地方。写最后那个求解器的代码可能是整个项目中最轻松的一步。

我被问到最多的问题："你们做了这么多优化，最后给工厂省了多少钱？"这是个好问题。以晶圆厂为例，cycle time 缩短 12% 意味着什么？一个 12 英寸晶圆厂月产量 10 万片，每片晶圆售价约 $3000，缩短 12% 的 cycle time 相当于释放了约 1.2 万片/月的产能——折合每年超过 4 亿美元的产值增益。但这也是"最坏"的假设——如果你算出来的排程方案执行不了，或者工程师不信任直接手动改了，那一切都是空谈。工程实现能力，才是调度项目的真正天花板。

回顾一下，我们走进了三个真实的工业场景，看到了教科书上的算法如何被改造、组合甚至"打破"来适应现实。但故事还没结束——即使有了好的调度方案，如何把它变成一个稳定、可维护、能迭代的软件系统？下一章，我们进入调度系统的软件工程——从四层架构到 REST API，从测试策略到 LLM 时代的新范式。
