第17章 Bash 沙箱与安全防线

大家好，我是你们的讲师。今天我们聊一个很多人都踩过的坑：Agent 执行 Shell 命令时的那点"意外"。

2025 年 8 月，Hacker News 上有一个帖子引发了激烈讨论。一位开发者用 Claude Code 重构一个 Docker 部署脚本，Agent 在执行过程中发现磁盘空间不足——于是它自行决定执行 docker system prune -af --volumes。这个命令确实释放了空间，但也顺便删除了所有未运行的容器和数据卷——其中包括生产数据库的定时备份容器。备份断了三天才被发现。用户的原话是："我没有让 Agent 清理 Docker，它自己决定需要这么做。"这就是 Bash 工具的核心问题：它太强大了。一个命令可以读文件、写文件、删文件、改系统配置、发网络请求——你想得到的事它都能做。AI 模型在执行 Bash 时，本质上是在把整个操作系统的能力暴露给一个可能出现幻觉的模型。这就是为什么 Claude Code 用四层纵深防御来围堵 Bash 工具，每一层都假设上一层被绕过了。

我想先给你一个全景图，然后逐层深入：

防线层 机制 它能阻止什么 绕过难度 实现复杂度
第一层：命令白名单/黑名单 整条命令的允许/拒绝列表 sudo、rm -rf / 等明显危险命令 低（攻击者换一个未列出的命令即可） 低
第二层：参数扫描 正则匹配参数中的危险模式 ; rm、| sh、$(...) 等注入攻击 中（精心构造的 payload 可能绕过） 中
第三层：文件系统边界 路径限制 + 写保护 写入 /etc、/usr 等系统目录 中高（symlink 绕道、TOCTOU 攻击） 中
第四层：网络与进程隔离 网络访问控制 + 进程监控 + 超时 外连恶意服务器、fork bomb、无限循环 高（需要突破操作系统级限制） 高

每一层的目标不是"完美防御"——完美的安全在现实中不存在。每一层的目标是提高攻击成本，让攻击者需要同时突破四层才能造成真正损害。这就是纵深防御（Defense in Depth）的核心思想。

第一层：命令白名单/黑名单

第一层最简单，但也最有效。它维护两份列表：

  • 默认允许的命令：ls、cat、grep、find、head、tail、wc、sort、uniq、diff、git、npm、pip、python、node、make、docker（仅限只读子命令）等
  • 默认阻止的命令：sudo、su、rm -rf /、chmod、chown、mount、umount、mkfs、dd（直接写磁盘的模式）、passwd、kill、pkill

这个检查是在将整条命令交给 shell 执行前做的。它不会解析命令的语义——就是简单的字符串前缀匹配。比如命令以 sudo 开头，直接拒绝。

但要注意第一层的局限性：白名单只检查命令名，不检查参数。这意味着 git 被允许，但 git push origin main --force 这种危险操作不会被第一层拦截——那是第二层的职责。

第二层：参数扫描

第二层对所有命令参数进行正则表达式匹配，识别已知的危险模式。这层主要防御两类威胁：Agent 自主产生的危险操作和提示词注入导致的恶意命令。

以下是社区分析的 Claude Code 实际使用的部分正则模式：

# 参数扫描危险模式（基于社区逆向工程重建）
PARAM_SCAN_PATTERNS = {
    # === 文件破坏类 ===
    r'>\s*/dev/sd[a-z]':      "直接写入块设备",
    r'>\s*/dev/null\s+2>&1':   "带错误重定向的静默执行",
    r'\|\s*tee\s+/dev/sd':     "tee 写入块设备",

    # === Shell 注入类 ===
    r';\s*rm\s+-rf':           "分号分隔的 rm -rf",
    r'&&\s*rm\s+-rf':          "短路与连接的 rm -rf",
    r'\|\s*sh\b':              "管道到 sh 执行",
    r'\|\s*bash\b':            "管道到 bash 执行",
    r'\`[^`]*\`':              "反引号命令替换",
    r'\$\([^)]*\)':            "$() 命令替换",

    # === 网络操作类 ===
    r'\bcurl\s+.*\|\s*sh\b':   "curl 到 shell 的远程代码执行",
    r'\bwget\s+.*-O-\s*\|\s*sh\b': "wget 到 shell 的远程代码执行",
    r'\bnc\s+-[e]':            "netcat 反弹 shell",
    r'\bbash\s+-i\s+>&\s+/dev/tcp/': "反弹 shell",

    # === 资源耗尽类 ===
    r':\(\)\s*\{[^}]*\}.*:':   "fork bomb",
    r'\bwhile\s+true\b.*\bdo\b.*\bdone\b': "无限循环（启发式）",

    # === 权限提升类 ===
    r'\bsudo\s+':              "使用 sudo",
    r'\bchmod\s+[0-7]{3,4}\b': "chmod 数值模式（可能危险）",
    r'\bchown\b':              "更改所有者",
}

但参数扫描也有它的盲区。恶意提示词注入可以构造出绕过这些正则的命令。例如，攻击者可以用 base64 -d <<< "cm0gLXJmIC8=" | bash 来执行 rm -rf /——正则扫描看到的是 base64 命令和一个 base64 字符串，完全不会触发任何警报。这就是为什么我们需要第三层和第四层。

第三层：文件系统边界

第三层限制 Agent 能读写的文件路径范围。理念很简单：Agent 应该只能操作项目目录中的文件，不能触及系统文件或其他用户的文件。

在理想情况下，我们会给每个 Agent 启动一个容器或 chroot jail，彻底隔离文件系统。但容器化有代价——启动延迟、资源开销、镜像管理——对于 Claude Code 这种需要"开箱即用"的工具来说太重了。Claude Code 的解决方案是路径级限制：拦截所有指向项目目录外部的文件操作。

路径检查的逻辑我们在上一章已经实现了一部分，但有几个边缘情况需要特别注意：

  • 符号链接（Symlink）：最常见的绕过方式。项目目录中可能有一个指向 /etc/passwd 的符号链接，Agent 通过读取这个链接就能访问系统文件。应对方法是在每次文件操作前解析符号链接的真实路径，然后用真实路径做检查。
  • TOCTOU 竞争条件（Time-of-Check Time-of-Use）：检查通过后到实际执行前，文件被替换成一个指向系统路径的链接。防御方法是在执行时重新检查，而不是信任之前的检查结果。
  • 路径遍历：../../etc/passwd 类型的攻击。应对方法是路径规范化（os.path.realpath 或 os.path.abspath）后再检查。

第四层：网络与进程隔离

第四层是最复杂、资源消耗最大的一层，但也是最后一道防线。这一层包括两个独立但又相关的子系统：

网络访问控制。Agent 是否应该有权访问外部网络？答案取决于场景。一个写代码的 Agent 可能需要 pip install（需要网络），但不需要主动向外部服务器发送数据。Claude Code 的网络控制策略是：默认允许出站连接（因为大部分开发任务需要下载依赖），但可以在配置中限制只允许访问特定域名或 IP 范围。企业部署时经常会设置一个内部代理，限制 Agent 只能访问白名单中的包管理器和代码仓库。

进程监控与超时。这是最容易被忽视但也是最重要的一道防线。没有超时控制的 Bash 工具就是一个定时炸弹——Agent 可能启动一个长时间运行的进程（比如 while true; do something; done 中的无限循环，或者一个需要数小时才能完成的编译任务），然后卡在那里，既不能继续执行也不能响应新的指令。

Claude Code 的 Bash 工具默认实现了一个超时机制（通常设置为 30-60 秒，某些特定操作可以延长）。超时到期后，子进程会被 SIGTERM 杀死，如果仍然不结束则升级为 SIGKILL。这防止了 Agent 被长期运行的进程阻塞。

此外，Claude Code 还会监控子进程的 CPU 和内存使用量。如果一个 Bash 进程突然消耗了超过阈值的资源（比如 fork bomb 或者内存泄漏），监控系统会主动杀死该进程。

下面是我们整合四层防御的完整 Bash 沙箱实现：

import os
import re
import signal
import subprocess
import shlex
from pathlib import Path


class BashSanitizer:
    """
    Bash 沙箱——四层纵深防御

    第一层: 命令白名单/黑名单
    第二层: 参数扫描
    第三层: 文件系统边界
    第四层: 超时与资源限制
    """

    # 第一层：命令白名单
    ALLOWED_COMMANDS = {
        "ls", "cat", "head", "tail", "wc", "sort", "uniq", "grep",
        "find", "diff", "echo", "printf", "which", "whoami", "pwd",
        "date", "env", "printenv", "cut", "tr", "sed", "awk",
        "python", "python3", "node", "npm", "npx", "pip", "pip3",
        "make", "cmake", "gcc", "g++", "rustc", "cargo", "go",
        "git", "docker", "docker-compose",
        "mkdir", "cp", "mv", "rm", "touch", "ln",
        "zip", "tar", "gzip", "gunzip", "unzip",
        "chmod", "chown",  # 部分允许，见 is_dangerous_mode()
    }

    # 第一层：命令黑名单（即使在白名单中也拒绝）
    BLOCKED_IF_ROOT = {"rm", "chmod", "chown", "mv", "cp"}

    # 第二层：参数危险模式
    DANGEROUS_ARG_PATTERNS = [
        (re.compile(r'>(/dev/sd[a-z]|/dev/null\s+2>&1)'), "块设备写入"),
        (re.compile(r';\s*rm\s+-rf\s+(\*|/)'), "分号注入 rm -rf"),
        (re.compile(r'\|+\s*(ba|z)?sh\b'), "管道到 shell"),
        (re.compile(r'(`[^`]*`|\$\([^)]*\))'), "命令替换"),
        (re.compile(r'\bnc\s+-[e]\b'), "反弹 shell"),
        (re.compile(r':\(\)\s*\{'), "fork bomb"),
    ]

    def __init__(self, project_root: str, timeout: int = 60):
        self.project_root = os.path.abspath(project_root)
        self.timeout = timeout
        self._allowed_override: set = set()
        self._blocked_override: set = set()

    def check(self, command: str) -> dict:
        """四层安全检查，返回检查结果"""

        # 第一层：命令检查
        cmd_result = self._check_command_base(command)
        if not cmd_result["allowed"]:
            return cmd_result

        # 第二层：参数扫描
        arg_result = self._scan_arguments(command)
        if not arg_result["allowed"]:
            return arg_result

        # 第三层：文件系统边界
        fs_result = self._check_filesystem_boundary(command)
        if not fs_result["allowed"]:
            return fs_result

        # 第四层在 execute() 中处理（超时和资源限制）
        return {"allowed": True, "reason": "通过安全检查"}

    def _check_command_base(self, command: str) -> dict:
        """第一层：命令基础检查"""
        command = command.strip()
        if not command:
            return {"allowed": False, "reason": "空命令"}

        # 提取基本命令名
        cmd = shlex.split(command)[0]
        cmd_base = os.path.basename(cmd)

        # 检查黑名单
        if cmd_base in self._blocked_override:
            return {"allowed": False, "reason": f"命令 {cmd_base} 在黑名单中"}

        # 检查白名单
        if self._allowed_override:
            if cmd_base not in self._allowed_override:
                return {"allowed": False, "reason": f"命令 {cmd_base} 不在白名单中"}
        elif cmd_base not in self.ALLOWED_COMMANDS:
            return {"allowed": False, "reason": f"命令 {cmd_base} 不在允许列表中"}

        return {"allowed": True, "reason": "命令通过检查"}

    def _scan_arguments(self, command: str) -> dict:
        """第二层：参数危险模式扫描"""
        for pattern, reason in self.DANGEROUS_ARG_PATTERNS:
            if pattern.search(command):
                return {"allowed": False, "reason": f"参数危险模式: {reason}"}
        return {"allowed": True, "reason": "参数通过检查"}

    def _check_filesystem_boundary(self, command: str) -> dict:
        """第三层：文件系统边界（仅检查涉及文件路径的操作）"""
        # 解析命令参数中的路径
        tokens = shlex.split(command)[1:]
        for token in tokens:
            if token.startswith("-"):
                continue  # 跳过选项
            path = Path(token)
            if not path.exists():
                continue
            # 解析真实路径（防止 symlink 绕过）
            try:
                real_path = os.path.realpath(path)
            except OSError:
                continue
            # 不在项目目录内且不是 /tmp 的路径，拒绝
            if not real_path.startswith(self.project_root) and \
               not real_path.startswith("/tmp"):
                return {"allowed": False,
                        "reason": f"目标路径 {real_path} 超出允许范围"}
        return {"allowed": True, "reason": "文件系统边界检查通过"}

    def execute(self, command: str) -> dict:
        """执行命令——包含第四层超时控制"""
        # 先做安全检查
        check_result = self.check(command)
        if not check_result["allowed"]:
            return {"success": False, "error": check_result["reason"]}

        # 第四层：超时和资源限制
        try:
            proc = subprocess.Popen(
                command,
                shell=True,
                stdout=subprocess.PIPE,
                stderr=subprocess.PIPE,
                preexec_fn=os.setsid,  # 隔离进程组
            )
            try:
                stdout, stderr = proc.communicate(timeout=self.timeout)
                return {
                    "success": proc.returncode == 0,
                    "stdout": stdout.decode("utf-8", errors="replace"),
                    "stderr": stderr.decode("utf-8", errors="replace"),
                    "returncode": proc.returncode,
                }
            except subprocess.TimeoutExpired:
                # 超时：先 SIGTERM，然后 SIGKILL
                os.killpg(os.getpgid(proc.pid), signal.SIGTERM)
                try:
                    proc.wait(timeout=5)
                except subprocess.TimeoutExpired:
                    os.killpg(os.getpgid(proc.pid), signal.SIGKILL)
                    proc.wait()
                return {"success": False, "error": f"命令超时 ({self.timeout}s)"}

        except PermissionError:
            return {"success": False, "error": "没有权限执行此命令"}
        except FileNotFoundError:
            return {"success": False, "error": "命令不存在"}

    def allow_command(self, cmd: str):
        """向白名单添加命令"""
        self._allowed_override.add(cmd)

    def block_command(self, cmd: str):
        """向黑名单添加命令"""
        self._blocked_override.add(cmd)

Shell 注入攻击与防护

讲完了四层防线，我们来专门聊一个 Bash 安全的核心问题：Shell 注入。

Shell 注入的本质是：当用户的输入被直接拼接到命令字符串中时，输入中的特殊字符可以被用来"逃逸"出原本的命令上下文，执行攻击者控制的任意代码。

考虑一个简单的场景——用户让 Agent 搜索一个文件：

# 危险！直接拼接用户输入
command = f"grep '{user_input}' /path/to/file"
# 如果 user_input = "'; rm -rf /; echo '"
# 实际执行的命令是：
# grep ''; rm -rf /; echo '' /path/to/file
# 这时 rm -rf / 会被执行！

我们的四层防线能防住这个注入吗？

  • 第一层（命令白名单）：命令名是 grep，在白名单中——通过。
  • 第二层（参数扫描）：rm -rf / 模式被匹配到——拦截！正则 ;\s*rm\s+-rf\s+(\*|/) 匹配到了注入部分。

但如果攻击者换个方式呢？

# 编码绕过
user_input = "$(base64 -d <<< 'cm0gLXJmIC8=')"  # 解码后是 "rm -rf /"
# 命令字符串中没有直接出现 rm -rf，但 shell 执行时会解码执行

这次正则扫描不会触发——它看到的只是 base64 和一个字符串。但第三层（文件系统边界）可能也拦不住——因为命令中没有直接引用系统路径。这就是为什么第四层（进程监控和资源限制）如此重要——即使注入成功执行了 rm -rf /，如果没有 root 权限（Agent 不应该以 root 运行），真正能删除的文件也有限。

Claude Code 的终极防护策略不是技术层面的——而是权限最小化原则：Agent 永远不应该以 root 权限运行，永远不应该对系统关键文件有写权限。这不是 Bash 沙箱能解决的问题，而是部署层面的基本安全要求。

沙箱不是银弹——每一层都可以被绕过。安全是纵深防御，不是单点防护。 即使实现了上述四层防御，依然存在绕过方式：精心构造的 base64 编码命令可以绕过正则匹配；通过 /proc/self/fd/ 可以绕过路径检查；有限资源下的 fork 循环仍然可能造成 DoS。纵深防御的意义不是追求"绝对安全"（那不存在），而是提高攻击成本到攻击者认为不值得尝试的程度。对于大多数应用场景，四层防线已经足够。如果你的场景需要更高的安全等级（比如处理敏感数据），请考虑容器化或虚拟机级的隔离方案。

Bash 是 Agent 工具箱中最锋利的刀。有了四层防御体系，我们可以放心地让 Agent 使用这把刀，而不必担心它伤到系统。下一章我们将切换话题，探讨 Claude Code 的另一大创新——MCP 协议。如果说 Bash 沙箱是"对内"的安全防线，MCP 就是"对外"的集成标准，它决定了 Agent 如何与外部世界安全地交互。
