第16章 推理优化

2025 年初，DeepSeek 的 API 服务上线后不久，一个有趣的现象引起了业界的注意。用户发现，DeepSeek-V3 的 API 调用价格仅为 GPT-4 的几十分之一——输入每百万 token $0.27，输出每百万 token $1.10。这是什么概念？同样生成一篇 2000 字的文章，GPT-4 要花你大约 6 美分，而 DeepSeek 只要 0.2 美分。

价格差异的背后，是推理效率的鸿沟。一个 671B 参数的 MoE 模型，如果按照传统 dense 模型的推理方式部署，单次推理需要加载 671B 个参数到显存，生成每 token 需要做 671B × 2 次乘加运算——这需要庞大的 GPU 集群和天文数字的电费。但 DeepSeek 通过一系列推理优化技术，让这个庞然大物以极低的成本高效运行。

这一章，我们深入剖析 DeepSeek 推理优化体系的四个关键技术。

16.1 推理瓶颈分析

要理解推理优化，首先要搞清楚推理的两个关键阶段在硬件层面的瓶颈是什么。

Prefill 阶段：计算密集型

当用户输入一段 prompt 时，模型需要一次性处理所有的输入 token。这个阶段称为 Prefill（预填充）。由于可以并行计算所有 token 的注意力，Prefill 阶段的计算效率很高，主要受限于 GPU 的计算能力（FLOPs）。Prompt 越长，Prefill 的计算量越大，但 GPU 利用率也越高。

Decode 阶段：显存带宽密集型

在生成第一个 token 后，模型进入 Decode（解码）阶段——每次只生成一个 token，并将新 token 的 KV Cache 缓存起来用于后续步骤。这个阶段的主要瓶颈不是计算，而是 显存带宽。原因在于：Decode 的每次迭代都需要将完整的模型参数从 HBM 读取到计算单元，但只做一次 token 的前向传播。计算量很小，但读取全部参数的操作每步都要做一次。

对于 dense 模型来说，Decode 阶段的问题尤为突出——每次都需要读取全部参数（例如 405B 参数），但只产生极其有限的计算量。这导致 计算强度（Arithmetic Intensity）极低，GPU 的 Tensor Core 大部分时间在等待数据从 HBM 传输过来。

计算强度公式：Arithmetic Intensity = FLOPs / Bytes。在 Decode 阶段，FLOPs 很小，但读取参数产生的 Bytes 很大。DeepSeek-V3 作为 MoE 模型，每次推理只读取 37B 激活参数而非 671B 总参数，使得计算强度大幅提升。这也是 MoE 推理的核心优势。

16.2 MLA 的 KV Cache 节省

在前面的第 8 章中，我们详细拆解了 MLA（Multi-head Latent Attention）的数学原理。现在我们聚焦它在推理阶段带来的具体收益。

在传统的 MHA（Multi-Head Attention）中，推理时每个 token 都需要缓存完整的 Key 和 Value 矩阵，大小为 2 × num_heads × head_dim。对于 DeepSeek-V3 这样的模型，每层 attention 有 128 个 head，每个 head 的维度为 128，那么每个 token 的 KV Cache 大小就是 2 × 128 × 128 × 2 bytes（FP16）≈ 65,536 bytes ≈ 64 KB per layer。以 61 层 decoder layer 计算，每个 token 的 KV Cache 为 64 KB × 61 ≈ 3.9 MB。当上下文长度达到 128K 时，KV Cache 的总大小是 3.9 MB × 128,000 ≈ 500 GB——没有 GPU 能装得下。

MLA 通过 低秩压缩 从根本上解决了这个问题。MLA 不直接缓存原始的 K 和 V 矩阵，而是缓存一个经过压缩的潜向量（latent vector），其维度远小于原始的 head 维度之和。在推理时，只有这个低维的潜向量需要缓存，而原始的 K 和 V 可以在需要时即时解压缩出来。

指标 传统 MHA (128 heads) DeepSeek-V3 MLA 节省比例
每 token KV Cache 大小 ~3.9 MB ~0.3 MB ~92%
128K 上下文 KV Cache 总量 ~500 GB ~38 GB ~92%
所需 GPU 数量（单层推理） 7-8 张 A100 80G 1 张 A100 80G ~87%

这意味着，128K 上下文的推理任务在 MLA 加持下可以在单张 A100 80G 上完成。而在传统 MHA 下，同样的任务需要至少 7-8 张 GPU 才能装下完整的 KV Cache。MLA 不仅节省了显存，还消除了跨 GPU 传输 KV Cache 的通信开销。这就是为什么 DeepSeek 能把长上下文推理做得这么便宜。

16.3 MoE 稀疏推理

MoE 架构在推理时带来的最大好处是：虽然模型有 671B 参数，但每次推理只加载 37B 参数。

我们来算一笔账。在 FP8 精度下，37B 参数的尺寸为 37 × 10^9 × 1 byte ≈ 37 GB。加上 KV Cache（~38 GB 用于 128K 上下文）、激活值缓存和其他开销，总显存需求约为 80-90 GB。这意味着单张 H100 80G 或 A100 80G 刚好可以放下模型做推理——但仅限于单批次、短上下文的情况。

对于生产环境中的高并发推理，DeepSeek 结合了 FP8 量化（37B 参数进一步压缩到 37 GB）和 KV Cache 的 Page 管理（类似 vLLM 的 PagedAttention），实现多请求共享 GPU 显存。这使得单卡可以同时服务多个请求，大幅提升了吞吐量。

MoE vs Dense 的推理规模对比：同样部署在 8 张 H100 上的推理服务，dense 405B 模型由于每次需要激活全部参数，单卡的处理能力远低于 MoE 671B 模型（只激活 37B）。这就是为什么 DeepSeek 能以极低的 API 价格提供更强的模型服务。

16.4 动态专家并行 (EP)

当单卡承载不了推理负载时，我们需要将模型分布到多张 GPU 上。对于 MoE 模型，有一种天然的并行策略——专家并行（Expert Parallelism）。

专家并行的核心思想是：将不同的 expert 分配到不同的 GPU 上。例如，DeepSeek-V3 有 256 个路由专家 + 1 个共享专家，分布在 8 台 GPU 上，每台 GPU 承载约 32 个专家。推理时，每一层 MoE 通过 all-to-all 通信将 token 路由到对应的 expert 所在的 GPU，计算结果后再通过 all-to-all 返回。

DeepSeek 的创新在于引入了 动态专家并行。传统的专家并行是 static 分配——每个 GPU 上固定存放某些专家。但现实中的专家访问模式是不均衡的：某些"热门"专家可能被大量 token 路由到，而"冷门"专家可能几乎没有 token 经过。静态分配会导致热门专家所在的 GPU 成为瓶颈。

动态专家并行的做法是：

  • 在运行时监控每个专家的负载情况
  • 当检测到某些 GPU 的负载过高时，将部分"热门"专家的副本迁移到低负载的 GPU 上
  • 类似操作系统的"进程迁移"，但粒度是"专家"而非"进程"
# 动态专家并行伪代码
# 输入: token 集合 T, 专家集合 E 分布在 N 个 GPU 上
#
# 1. 每个 GPU 统计本地专家的路由计数
# 2. 通过 all-gather 收集全局路由分布
# 3. if 存在负载不均 (max_load / avg_load > 阈值):
#      识别热门专家 e_hot 和空闲 GPU g_idle
#      在 g_idle 上创建 e_hot 的副本
#      更新路由表，将部分 token 路由到副本
# 4. 执行 standard expert dispatch & combine
# 5. if 路由模式再次变化:
#      动态调整副本数量

这种动态调度的效果非常显著。根据 DeepSeek 的工程报告，动态专家并行将推理服务的 P99 延迟降低了约 40%，而吞吐量提升了约 30%。这意味着同样的 GPU 集群可以服务更多的用户请求。

16.5 Prefilling / Decode 分离

Prefill 阶段和 Decode 阶段对计算资源的需求截然不同。Prefill 是 compute-bound（计算密集型），需要大量的并行计算能力；Decode 是 memory-bandwidth-bound（带宽密集型），需要的是高速显存读写能力。把二者混在一起做，两个阶段的性能都会受到制约。

DeepSeek 采用了 Prefill-Decode 分离架构：

  • Prefill Worker 集群：由计算能力强的 GPU 组成（H100/H800），专注于快速处理用户输入的 prompt。一个长 prompt 在 Prefill Worker 上可以一次性算出所有 token 的注意力分布，生成完整的 KV Cache。
  • Decode Worker 集群：由显存带宽高的 GPU 组成，专注于快速逐 token 生成。Decode Worker 从 Prefill Worker 接收 KV Cache，然后逐步生成输出 token。

这两个集群之间通过高速网络连接，KV Cache 在 Prefill 完成后被传输到 Decode Worker。由于 MLA 压缩后的 KV Cache 尺寸很小（128K 约 38 GB），传输延迟可以在毫秒级别完成。

分离架构的实际效果：在传统的混合部署中，长 prompt 的 Prefill 可能会阻塞整个推理管线的 Decode 操作。用户会感觉模型"想很久才回答"。分离后，Prefill Worker 快速处理完 prompt，Decode Worker 全速生成，用户的体验变成了"几乎秒回"。

16.6 Batch 调度与吞吐优化

最后一个关键环节是 Batch 调度。当多个用户同时请求推理服务时，如何将不同用户的请求组织成 batch 以最大化吞吐，是一个关键的工程问题。

DeepSeek 采用了 Continuous Batching（连续批处理）技术。与传统的静态批处理（固定 batch size、等待所有请求完成后才返回）不同，Continuous Batching 的核心思想是：当一个请求生成完成后，立即从队列中拉取一个新请求加入当前的 batch，而非等待整个 batch 完成。这使得 GPU 始终处于满负荷状态。

具体实现上，DeepSeek 的推理框架做了以下优化：

  • 动态 KV Cache 管理：不同请求的 KV Cache 通过 page 管理（类似虚拟内存的分页），按需分配和回收。已完成的请求的 KV Cache 被立即释放给新请求。
  • Token 级调度：不是以"请求"为粒度而是以"token"为粒度进行调度。不同请求在 Decode 阶段的进度不同，Token 级调度允许先完成的请求立即离开 batch。
  • 优先级队列：对延迟敏感的交互式请求（如聊天）获得较高优先级，对吞吐敏感的批处理请求（如离线评估）置于较低优先级队列。
# Continuous Batching 流程
# 初始: batch = [req1, req2, req3]
#
# Step 1: 对 batch 中所有请求做 Decode，各生成 1 token
# Step 2: req2 生成了 <eos>，标记为完成
# Step 3: 从等待队列取出 req4，加入 batch
# Step 4: 新的 batch = [req1, req3, req4]
# Step 5: 重复 Step 1-4
#
# → 相比静态批处理，吞吐提升约 2-3 倍
实际效果：据 DeepSeek 官方披露，结合 MLA、MoE 稀疏推理、Expert Parallelism、Prefill-Decode 分离和 Continuous Batching 五大技术，DeepSeek-V3 的单节点推理吞吐量达到了同等规模 dense 模型的 8-10 倍。这也是为什么他们能把 API 价格定到 GPT-4 的几十分之一，并且仍然有合理的利润空间。

推理优化的目的是让模型跑得更快更便宜。但还有一个维度能让模型"跑得更轻"——那就是对模型本身进行压缩。下一章，我们来聊 量化压缩：如何用 FP8、FP4 甚至 INT4 的精度，让大模型在消费级显卡上流畅运行。
