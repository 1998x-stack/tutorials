开篇：数据库是大多数系统的瓶颈所在

我经历过一次让我记忆犹新的"数据库之夜"。一个在线教育平台，高峰期 10 万用户同时在线抢课。按理说抢课不是什么复杂操作——一个 SELECT 检查余量，一个 UPDATE 扣减名额。但就在那个晚上，数据库的 CPU 直接飙升到 100%，所有请求超时，系统彻底瘫痪。最后排查出来，罪魁祸首是一张不到 10 万行的表上的一个全表扫描——一个没有加索引的 WHERE 条件，在并发请求下把数据库打趴了。

这个故事的教训是：数据库知道怎么把数据存进去，但如果你不告诉它怎么快速找到数据，它就用最笨的办法——逐行扫描。在数据量小的时候这没问题，但一旦上了规模，一次全表扫描的代价是毁灭性的。

关系型数据库经过几十年的发展，已经把"正确性"这件事做得非常好了——ACID 事务、外键约束、范式设计。但"正确"不等于"快"。本章聚焦于一个核心问题：如何在保证数据正确性的前提下，把数据库的性能发挥到极致。

🗺️ 本章路线：范式设计（结构正确） → 索引优化（查询快） → 事务隔离（并发正确） → 锁与死锁（排查问题） → 实战案例（综合运用）。

范式与反范式：第三范式 vs 适度冗余

数据库范式是设计关系型表结构的基础理论。第三范式（3NF）是最常用的范式级别——它要求表中每一列都直接依赖于主键，不能有传递依赖。简单说就是"每个事实只存一次"。

但在实际的后端开发中，完全遵循 3NF 不一定是最好的选择。过度的范式化会导致大量的 JOIN 查询，而 JOIN 是关系型数据库中性能开销最大的操作之一。这就是"反范式化"的由来——有意引入适度冗余来换取查询性能。

维度 第三范式（3NF） 反范式化
数据冗余 几乎没有冗余 有控制的冗余
写入性能 好（只需更新一处） 较差（需要多处同步更新）
查询性能 较差（需要多个 JOIN） 好（一张表就能搞定）
数据一致性 天然一致 需要应用层或触发器维护
适用场景 写入频繁、一致性要求高 查询频繁、读多写少、性能敏感

一个经典的例子是订单表。在 3NF 设计中，orders 表只存 user_id，用户名称需要在查询时 JOIN users 表获取。但在实践中，很多系统会在 orders 表中冗余一个 user_name 字段——因为一旦订单创建完成，用户名几乎不会变，而每次查询订单时都要 JOIN 用户的代价是不值得的。

🔑 决策原则：先按 3NF 设计，然后用 EXPLAIN ANALYZE 找出性能瓶颈，再针对性地解范式化。不要一开始就反范式——你可能会为了一个不存在的性能问题引入复杂的一致性维护成本。

索引原理与优化

索引是数据库性能调优的第一手段。一个没有索引的表叫"堆表"（Heap），数据库只能顺序扫描。有索引的表才能走"随机访问"路径。

B-Tree 索引是关系型数据库中最常用的索引类型，适用于等值查询、范围查询、排序。GIN 索引适合"包含"查询（如数组、JSONB、全文搜索）。GiST 索引适合地理空间和模糊搜索。对于 PostgreSQL，选择正确的索引类型可以带来数量级的性能差异。

EXPLAIN ANALYZE 是 PostgreSQL 中最重要的性能分析工具。它能告诉你一条查询实际走了什么索引、扫描了多少行、用了多长时间。

-- 慢查询示例：没有索引的查询
EXPLAIN ANALYZE SELECT * FROM orders WHERE status = 'pending';
-- 输出：
-- Seq Scan on orders  (cost=0.00..4321.15 rows=152340 width=42)
--   Filter: (status = 'pending'::text)
--   Rows Removed by Filter: 847660
--   Planning Time: 0.052 ms
--   Execution Time: 312.4 ms
-- ↑ 全表扫描，扫描了 100 万行，花了 312ms

-- 创建索引后
CREATE INDEX idx_orders_status ON orders(status);

EXPLAIN ANALYZE SELECT * FROM orders WHERE status = 'pending';
-- 输出：
-- Bitmap Heap Scan on orders  (cost=12.34..1567.89 rows=152340 width=42)
--   Recheck Cond: (status = 'pending'::text)
--   -> Bitmap Index Scan on idx_orders_status  (cost=0.00..12.34 rows=152340 width=0)
--         Index Cond: (status = 'pending'::text)
--   Execution Time: 23.1 ms
-- ↑ 走索引扫描，只读索引然后回表取数据，花了 23ms，快了 13 倍

索引也不是越多越好。每个索引都会增加写入的开销——插入一行数据可能需要更新 5-6 个索引。一个常见的问题是过度建索引：有人把每个查询用到的字段都建上索引，结果写入性能大幅下降，而查询性能提升有限。

⚠️ 索引选择最佳实践（2026）：
• 高选择率的字段（唯一值占比高）才适合索引，比如 status 只有 3 个值，不算好索引；user_id 有很多不同值，是好索引。
• 组合索引 <= 3 个字段，字段顺序按选择性从高到低排列。
• 使用部分索引（Partial Index）如 WHERE status = 'pending'，只索引需要的数据。
• PostgreSQL 16+ 支持增量排序（Incremental Sort），对于组合索引的排序查询更友好。

事务隔离级别

事务隔离级别决定了并发操作之间的可见性。SQL 标准定义了四个隔离级别，从低到高分别是 READ UNCOMMITTED、READ COMMITTED、REPEATABLE READ、SERIALIZABLE。PostgreSQL 默认使用 READ COMMITTED，这也是绝大多数应用的首选。

隔离级别 脏读 不可重复读 幻读 适用场景
READ UNCOMMITTED 可能 可能 可能 几乎不用（PG 实际上将其当作 READ COMMITTED）
READ COMMITTED 不会 可能 可能 绝大多数应用，默认选择
REPEATABLE READ 不会 不会 可能 报表统计、需要快照一致性
SERIALIZABLE 不会 不会 不会 金融交易、严格一致性（性能代价大）

一个经典问题是"扣库存"场景下的并发控制。用 READ COMMITTED 级别时，两个并发的扣减请求可能都读到余量 > 0，都执行扣减，结果导致超卖。解决方案通常是用 SELECT ... FOR UPDATE（行级锁）或乐观锁（版本号机制）。

-- 安全的库存扣减方案：行级锁
BEGIN;
SELECT quantity FROM products WHERE id = 1001 FOR UPDATE;
-- 在应用层检查 quantity >= requested
UPDATE products SET quantity = quantity - 1 WHERE id = 1001 AND quantity >= 1;
COMMIT;

-- 乐观锁方案：版本号
UPDATE products
SET quantity = quantity - 1, version = version + 1
WHERE id = 1001 AND version = 5 AND quantity >= 1;
-- 如果影响行数 = 0，说明版本号不匹配或库存不足，需要重试

锁机制与死锁排查

数据库锁是保证并发事务正确性的基石，但也是死锁的源头。PostgreSQL 中的锁分为表级锁和行级锁。行级锁最常用——FOR UPDATE、FOR NO KEY UPDATE、FOR SHARE、FOR KEY SHARE。间隙锁（Gap Lock）在 REPEATABLE READ 和 SERIALIZABLE 级别中用于防止幻读，但也是死锁的常见原因。

当两个事务互相等待对方持有的锁时，就产生了死锁。PostgreSQL 会自动检测死锁并中止其中一个事务。但如果你不分析死锁日志，同样的死锁模式会反复出现。

-- 查看当前锁等待情况
SELECT blocked_locks.pid     AS blocked_pid,
       blocked_activity.usename  AS blocked_user,
       blocking_locks.pid     AS blocking_pid,
       blocking_activity.usename AS blocking_user,
       blocked_activity.query    AS blocked_statement,
       blocking_activity.query   AS current_statement_in_blocking_process
FROM  pg_catalog.pg_locks         blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity  ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks         blocking_locks
    ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.DATABASE IS NOT DISTINCT FROM blocked_locks.DATABASE
    AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
    AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
    AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
    AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
    AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
    AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
    AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
    AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.GRANTED;
💡 避免死锁的实用建议：确保所有事务按相同的顺序访问资源（比如总是先更新 order 再更新 payment）；保持事务简短，减少锁持有的时间；使用 NOWAIT 或 SKIP LOCKED 避免等待；在死锁发生时做好重试逻辑。

实战案例：一个慢 SQL 从 30 秒优化到 50ms

让我分享一个真实的优化案例。一个内容管理系统的"文章列表"接口，在数据量增长到 500 万行后，响应时间从最初的 200ms 暴涨到了 30 秒。查询大概是这样：

-- 原始慢查询
SELECT a.*, u.name AS author_name, c.name AS category_name
FROM articles a
JOIN users u ON a.author_id = u.id
JOIN categories c ON a.category_id = c.id
WHERE a.status = 'published'
  AND a.published_at BETWEEN '2026-01-01' AND '2026-06-30'
ORDER BY a.published_at DESC
LIMIT 20 OFFSET 100;

EXPLAIN ANALYZE 发现：虽然 published_at 上有索引，但排序走了磁盘文件排序（filesort），因为 500 万行中只有约 100 万行满足状态条件，LIMIT+OFFSET 又加剧了问题。

优化方案：

  • 建组合索引：CREATE INDEX idx_articles_pub_status ON articles(status, published_at DESC); —— 让索引直接满足排序和过滤条件。
  • 改 OFFSET 为游标：不再使用 page/offset，改为 WHERE published_at < last_seen 的游标分页。
  • 覆盖索引：只在索引里放 published_at、status、author_id、category_id，让查询完全通过索引完成（Index Only Scan），不需要回表。

优化结果：30 秒 → 50ms。600 倍的提升。没有加机器，没有改架构，只是让数据库更好地利用了索引。

🔧 记住这条原则：数据库优化的第一原则永远是"减少数据库需要处理的数据量"。不是把数据库调得更快，而是让它做更少的事。索引、分页、分区、缓存——所有优化手段的本质都是"少干活"。

关系型数据库是后端开发的基石，但它不是万能的。当你的系统需要处理海量写入、灵活的数据模型或毫秒级响应时，关系型数据库的局限性就会显现。这就是下一章要探讨的主题——NoSQL 与缓存策略。Redis 为什么这么快？MongoDB 在什么场景下比 PostgreSQL 更合适？缓存穿透、击穿、雪崩怎么解决？这些问题我们都将在下一章里一一拆解。
