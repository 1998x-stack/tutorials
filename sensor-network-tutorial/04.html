第4章 WSN网络拓扑

大家好，我是你们的讲师。第3章我们解决了"两个节点之间如何无线通信"的问题。但WSN很少只有两个节点——在典型场景中，可能有几十甚至上千个传感器节点散布在一个区域内。这些节点之间如何组织？谁跟谁说话？数据怎么汇聚到网关？这就要靠网络拓扑（Network Topology）来回答了。

选择一个正确的拓扑结构，其重要性堪比选择正确的传感器：选错了，轻则网络延迟和能耗不达标，重则整个系统在大规模部署后根本无法正常工作。我见过一个智能灯控项目，团队选了简单的星型拓扑，结果一栋楼里装了500个节点后网关频频过载——单网关的吞吐量根本不够。最终不得不把整个系统从星型改造为树型+分簇拓扑，相当于推到重来。这一章，我们就来深入探讨WSN的五种核心拓扑结构，帮你建立"拓扑优先"的设计思路。

学完本章你将能够： 根据项目的规模、覆盖范围、可靠性需求和能源约束，独立选择最优的拓扑结构，并理解为什么没有"万能拓扑"——每一种拓扑都是特定场景下的权衡产物。

五种拓扑结构详解

1. 星型拓扑（Star Topology）

这是最直观、最简单的拓扑结构：所有传感器节点（终端设备）直接与一个中心节点（网关/协调器）通信。终端节点之间不直接通​​信。优点非常突出：结构简单、延时低（单跳）、易于管理（集中式调度）。缺点也很致命：中心节点是单点故障，一旦网关失效，整个网络瘫痪；而且覆盖范围受限于中心节点与最远终端之间的单跳通信距离。

星型拓扑适合小范围、节点少、集中式管理的场景，如智能家居、小型办公室环境监测。蓝牙、BLE Mesh的"广播模式"本质上就是星型拓扑的一种变体。

2. 树型拓扑（Tree Topology）

树型结构引入了一个中间层——路由器节点（Router Node）。终端节点上行连接到路由器，路由器再连接到网关（根节点）。这样就形成了一个"根→分支→叶子"的树状结构。数据从叶子节点向根节点汇聚（汇聚流），命令从根节点向叶子节点分发（配置流）。

树型拓扑的优点是覆盖范围比星型大大扩展——通过多级路由器接力传输信号，可以覆盖大面积区域。但缺点是：中间路由器节点也是单点故障——某个路由器失效会导致它下面的整个子树失联；而且上层节点的通信负担更重（"靠近根的节点"需要转发大量子节点数据，电池消耗更快，这就是经典的能量空洞（Energy Hole）问题）。

3. Mesh拓扑（Mesh Topology）

Mesh是WSN中最灵活但也是最复杂的拓扑：任意两个在通信范围内的节点都可以直接通信，数据可以通过多条路径从源节点到达目标节点。这就是多跳通信（Multi-Hop Communication）和自愈（Self-Healing）的基础。

Mesh拓扑的突出优点：鲁棒性极高——一条路径中断，数据自动走备用路径；覆盖范围可通过节点接力无限扩展；节点间负载相对均衡（理论上）。但代价是网络复杂度和能耗大幅增加——每个Mesh节点需要持续维护路由表、周期性地交换路由信息，这些都消耗宝贵的能量。ZigBee的Mesh模式就是一个典型实例。

4. 簇状拓扑（Cluster Topology）

簇状拓扑是WSN中最具特色的拓扑组织形式。它将网络划分为多个簇（Cluster），每个簇有一个簇头（Cluster Head, CH）。簇内节点将数据发送给簇头，簇头进行数据融合后发送给网关或基站。簇头通常通过选举产生，并在运行过程中轮换以均衡能耗。

著名的LEACH（Low Energy Adaptive Clustering Hierarchy）协议就是簇状拓扑的典范——它是WSN中最早、最经典的分层路由协议。LEACH通过对簇头进行周期性轮换（随机选举），将能耗均匀分布到所有节点上，大大延长了网络寿命。LEACH为后续大量分簇/分层路由协议（如PEGASIS、TEEN、HEED）奠定了基础。

5. 混合拓扑（Hybrid Topology）

实际工程中，很少有系统只用一种纯粹的拓扑结构。最常见的是星型+Mesh混合：终端设备以星型方式连接到父节点（降低复杂度），而父节点之间以Mesh方式互联（保障可靠性）。ZigBee 3.0和Thread协议都支持这种架构。混合拓扑试图在"简单低耗"（星型）和"可靠灵活"（Mesh）之间找到最佳平衡点。

经典案例：ZigBee智能家居Mesh问题 2015年左右，很多智能家居厂商使用ZigBee的Mesh拓扑。ZigBee协调器负责管理整个网络，路由器节点（通常是插电设备如智能灯泡、智能开关）负责转发数据。问题是：路由器节点可能被用户关闭（关灯=断电），导致Mesh拓扑瞬间断裂。某知名厂商的智能灯控系统在用户"关灯睡觉"后，所有支持Mesh路由的灯泡同时断电，整个ZigBee网络瓦解——第二天早上用户发现"所有设备都离线了"。解决方案是将必须保持在线的重要路由器设备（如智能插座，用户一般不断电）与灯泡（可能被频繁断电）在路由策略中区分开。这个案例深刻地提醒我们：Mesh拓扑虽然鲁棒，但路由节点的电源可靠性是前提条件——如果路由器本身随时可能被关掉，"自愈"也只能在有限的范围内起作用。

自组织网络（Ad Hoc）

WSN的一个核心特征是自组织（Self-Organization）：节点部署前不需要预先规划拓扑结构和路由。打开电源后，节点会自动发现邻居、建立连接、形成网络。整个过程不需要人为干预。这是WSN与传统"规划-部署"网络（如蜂窝网络、Wi-Fi热点覆盖）的根本区别。

邻居发现（Neighbor Discovery）

这是自组网的第一步。节点上电后，首先在预定义的信道上广播Hello消息（或Beacon帧）。Hello消息中包含节点ID、剩余能量、链路质量等信息。附近的节点收到Hello消息后，将发送节点记录为"潜在邻居"并回复Ack。经过几个周期的Hello交换，每个节点都建立了一张邻居表（Neighbor Table）。

邻居表的质量直接影响上层路由协议的效果。一个常见问题是"邻居震荡"——节点在信号强度临界区域反复进入和离开邻居范围，导致路由表不停更新、频繁触发路由重建，浪费了大量能量。解决方法是使用链路质量评估（Link Quality Estimation, LQE）：不只看单次RSSI值，而是对一段时间内的链路质量进行加权统计（如ETX、PRR等指标），只有当链路质量稳定高于阈值时才将节点加入邻居表。

拓扑控制

既然自组织网络会自动形成拓扑，为什么还需要"拓扑控制（Topology Control）"？答案是：不控制的拓扑往往不是最优的。如果所有节点都以最大发射功率运行，虽然链路质量好、跳数少，但干扰严重、能耗巨大。拓扑控制的目标就是在保证网络连通的前提下，通过调整节点发射功率和活动状态来优化网络性能。

功率控制（Power Control）

节点根据与邻居节点的距离动态调整发射功率。近距离通信用低功率，远距离用高功率。这可以：减少对非目标节点的干扰（空间复用提升）、降低节点能耗。分布式功率控制算法如COMPOW（Common Power）和LMA/LMN（Local Mean Algorithm）是经典代表。

睡眠调度（Sleep Scheduling）

这是WSN拓扑控制最有效的技术手段——没有之一。基本原理是：让大部分节点在大部分时间处于睡眠状态（关闭无线电），只有少数节点保持唤醒以维持网络连通。通过占空比（Duty Cycle）参数来调节：占空比=活动时间/(活动时间+睡眠时间)。典型WSN节点占空比在0.1%-1%之间——一个节点每秒只醒来1-10ms，其余时间都在深度睡眠中。

睡眠调度的挑战在于：节点醒来时需要保证能与邻居通信（否则形成"聋子网络"）。这就需要同步或异步的唤醒调度：S-MAC采用同步调度（所有节点周期性地同时醒来交换消息）；B-MAC/LPL（Low Power Listening）采用异步调度（发送方先发一段长前导码，确保接收方在唤醒周期内检测到）。SMAC同步开销大但效率高，B-MAC/B-MAC实现简单但前导码额外消耗能量。

// 简单的邻接发现与链路质量评估伪代码
#define NEIGHBOR_TABLE_SIZE 20
#define LQE_WINDOW 10      // 评估窗口：最近10个Hello包

typedef struct {
    uint16_t node_id;
    int16_t  rssi[LQE_WINDOW];   // 最近LQE_WINDOW个包的RSSI
    uint8_t  hello_count;         // 收到的Hello包数
    uint32_t last_seen;           // 最后看到的时间戳
    uint8_t  is_router;           // 是否有路由转发能力
} NeighborEntry;

NeighborEntry neighbors[NEIGHBOR_TABLE_SIZE];

// 收到Hello消息后的处理函数
void on_hello_received(uint16_t src_id, int16_t rssi) {
    NeighborEntry *entry = find_or_create_neighbor(src_id);

    // 滑动窗口：将新RSSI存入窗口并移动指针
    for (int i = LQE_WINDOW - 1; i > 0; i--) {
        entry->rssi[i] = entry->rssi[i-1];
    }
    entry->rssi[0] = rssi;

    // 计算加权平均链路质量（越近的样本权重越大）
    float lqi = 0;
    float weights[LQE_WINDOW] = {0.5, 0.25, 0.125, 0.0625, 0.03125,
                                  0.0156, 0.0078, 0.0039, 0.0020, 0.0010};
    for (int i = 0; i < LQE_WINDOW; i++) {
        if (entry->rssi[i] != 0) {
            lqi += entry->rssi[i] * weights[i];
        }
    }

    // 只有链路质量超过阈值才保留该邻居
    if (lqi > RSSI_THRESHOLD) {
        entry->last_seen = get_tick();
        // 如果链路质量极好，可作为路由候选
        if (lqi > RSSI_ROUTER_THRESHOLD) {
            entry->is_router = 1;
        }
    } else {
        // 链路质量差，从邻居表中移除
        remove_neighbor(src_id);
    }
}

// 拓扑控制：根据邻居表动态调整发射功率
void adjust_tx_power() {
    int nearest_distance = find_nearest_neighbor_distance();
    int8_t required_power = estimate_min_power(nearest_distance);
    set_radio_tx_power(required_power + BACKOFF_DBM);
    // 加BACKOFF以保证链路余量，通常+3dB
}

多跳通信的能耗分析

很多人直觉上认为"多跳传输速度慢、效率低"，但WSN的视角不同。在WSN中，多跳通信的主要目的是节能，而不是提速。你可能会问：一个数据包经过n个节点转发，总耗能不是比单跳更大吗？答案是：不一定。

无线通信中的能耗模型大致是：发送能耗 ∝ 距离^α，其中α是路径损耗指数（自由空间α=2，实际环境α=2.7-4.0）。这意味着把通信距离翻倍，发送功耗要增加到2^α倍（α=3时就是8倍）。如果用两跳来替代单跳，每跳只需发送一半距离，每跳的功耗降至(1/2)^α = 1/8，这样两跳的总能耗大约是单跳的2×1/8=1/4。虽然实际不会这么极端（接收端的能耗也需要计入），但多跳传输在能量效率上的优势是根本性的。

最优跳数是多少？LBL（Lawrence Berkeley Lab）的研究表明：在典型的WSN部署中，最优跳数为根号下总节点数量级。但具体数字受很多因素影响——信道质量、节点密度、数据包大小、发送/接收能量的比值——需要通过能量模型仿真来精确确定。

工程建议： 在设计WSN拓扑时，不要追求"最大限度的Mesh连接"。一个常见的误区是把所有节点都设为路由节点（全对等Mesh），以为这样最可靠。实际上过密的Mesh连接会导致路由表爆炸、控制开销剧增，甚至出现路由环路。最佳的实践是：选择一部分节点（约20-30%）作为路由节点，其余作为终端节点。终端节点只跟父节点通信（类似星型），路由节点之间形成Mesh骨干。这就是ZigBee和Thread推荐的"星型+Mesh混合"模式。既保留了Mesh的鲁棒性，又控制了网络复杂度。

网络拓扑是WSN的"骨架"——它决定了系统的可靠性、延迟、能耗和可扩展性的上限。选择拓扑结构不能凭感觉，而应该根据应用场景的需求进行量化分析：有多少个节点？覆盖范围多大？电源可更换吗？对实时性要求多高？回答这些问题后，拓扑选择的答案就会自然浮出水面。

下一章我们将进入WSN的"软件视野"——传感网协议栈。拓扑决定了节点之间的"物理连接"，而协议栈则定义了节点之间如何"对话"。从物理层的波形到应用层的语义，五层协议栈各司其职。我们还会讨论一个在WSN领域中备受争议的话题：跨层优化——为什么传统分层网络理论在WSN中遭遇了挑战？这是非常有趣的内容，下一章见。
