开篇：关系型不是银弹——什么时候该选 NoSQL

我刚工作的时候，所有的数据都往 MySQL 里塞。用户信息存 MySQL，文章存 MySQL，日志存 MySQL，甚至缓存也用 MySQL 的一张 cache_kv 表。直到有一天，一个日志分析查询跑了 15 分钟，把整个数据库的 IO 打满了，影响了线上核心交易。当时的 DBA 无奈地说："别把 MySQL 当垃圾桶。"

那是我第一次意识到：没有一种数据库能解决所有问题。每种数据存储方案都是为特定场景优化的——关系型数据库擅长强一致和复杂查询，Redis 擅长极速读写，MongoDB 擅长灵活文档模型，Elasticsearch 擅长全文搜索。选对工具，比把一种工具用到极致更重要。

本章我们聚焦两个最常见的非关系型存储场景：缓存（Redis）和文档数据库（MongoDB）。它们覆盖了后端开发中 90% 以上的 NoSQL 需求。

🔑 核心思维：不要把数据库当作"存数据的地方"，而是当作"为特定查询模式设计的存储引擎"。关系型数据库的查询模式是"SQL + JOIN + 事务"，Redis 的查询模式是"key-value + 数据结构操作"，MongoDB 的查询模式是"文档 + 聚合管道"。不同的查询模式决定了不同的数据建模方式。

Redis 数据结构实战

很多人以为 Redis 只是"缓存"，存一些简单的 key-value。但 Redis 真正的强大之处在于它提供了丰富的数据结构，每种结构都对应一类经典问题的解法。

# String —— 最简单也最常用
SET user:1001:name "张三"
SET user:1001:login_count 0
INCR user:1001:login_count     # 异步增长计数器
EXPIRE user:1001:session 3600  # 设置过期时间

# Hash —— 适合存储对象
HSET article:2001 title "Redis实战" author "李四" views 0
HINCRBY article:2001 views 1
HGETALL article:2001

# List —— 消息队列 / 最新列表
LPUSH notifications:user:1001 "您有一条新消息"
RPOP notifications:user:1001
LLEN notifications:user:1001   # 队列长度

# Set —— 去重 / 集合运算
SADD tag:golang:articles 1001 1002 1003
SADD tag:redis:articles 1001 1004
SINTER tag:golang:articles tag:redis:articles  # 交集 => {1001}

# ZSet —— 排行榜 / 延迟队列
ZADD leaderboard:day:20260630 99.5 "user:1001"
ZADD leaderboard:day:20260630 88.3 "user:1002"
ZREVRANGE leaderboard:day:20260630 0 9 WITHSCORES  # Top 10

# Stream —— 消息队列（Redis 5.0+）
XADD mystream * sensor_id 123 temperature 25.5
XREADGROUP GROUP mygroup consumer1 COUNT 1 STREAMS mystream >
💡 数据结构的"单挑"能力：一个 ZSet 就能实现排行榜 + 分页 + 范围查询；一个 Stream 就能替代轻量级消息队列。很多时候，不需要引入 Kafka 或 RabbitMQ，一个 Redis Stream 就够用了。"简单就是最大的可靠"。

缓存模式：四大赛道

缓存和数据源之间的交互方式有几种经典模式。每种模式对应不同的读写比重和数据一致性要求。

模式 读流程 写流程 一致性 适用场景
Cache-Aside 先查缓存，未命中再查 DB，回填缓存 直接写 DB，删除缓存（或更新缓存） 最终一致（需要处理并发写） 读多写少，通用场景
Read-Through 缓存负责从 DB 加载数据（对应用透明） 直接写 DB 最终一致 需要缓存抽象层（如 RedisLabs）
Write-Through 同 Read-Through 先写缓存，缓存同步写 DB 强一致（但写延迟增加） 写操作需要立即可见
Write-Behind 同 Read-Through 先写缓存，异步批量写 DB 弱一致（有数据丢失风险） 写入吞吐量极大、可容忍延迟落盘

Cache-Aside（旁路缓存）是实践中用得最多的模式。它的核心逻辑是：读的时候先查缓存，没命中就查数据库然后写入缓存；写的时候直接写数据库，然后删除缓存（而不是更新缓存）。删缓存而不是更新缓存的原因是——避免并发写导致的缓存与数据库不一致。

🧠 "先删缓存还是先写 DB" 的经典问题：推荐顺序是"先写 DB，再删缓存"。如果先删缓存然后写 DB 失败，缓存就缺了数据。如果先写 DB 再删缓存，在删缓存之前的瞬间可能有读请求把旧数据读到缓存里（概率极低），可以通过过期时间来兜底。这个模式被称为 "Cache-Aside with Lazy Deletion"。

缓存三大坑：穿透、击穿、雪崩

使用缓存时，有三大经典问题需要特别注意。每一个都可能在流量高峰期让你的系统崩溃。

缓存穿透：查询一个不存在的数据。缓存里没有，数据库里也没有。每次请求都直接打到数据库上。如果同时有大量请求查询同一个不存在的 key，数据库会承受巨大压力。

解决方案：布隆过滤器（Bloom Filter）预先判断 key 是否存在；或者对空结果也做缓存（"空值缓存"），设置较短的过期时间。

缓存击穿：一个热点 key 在过期的一瞬间，大量并发请求同时涌入，全部穿透到数据库。

解决方案：互斥锁（Mutex），让第一个请求去加载数据，其他请求等待；或者"逻辑过期"策略——不设置过期时间，而是在 value 中存一个逻辑过期时间，发现过期后异步更新缓存。

缓存雪崩：大量 key 在同一时间过期，或者缓存节点整体宕机，导致海量请求直接打到数据库上。

解决方案：过期时间加随机偏移（比如基础时间 + 随机 0-300 秒），避免大量 key 同时过期；缓存集群高可用（Redis Sentinel 或 Cluster）；本地缓存兜底（多级缓存）。

// 缓存击穿的互斥锁方案（Go 伪代码）
func GetProduct(id string) (*Product, error) {
    key := "product:" + id
    // 1. 尝试从缓存读取
    if data, err := redis.Get(ctx, key); err == nil {
        return decodeProduct(data), nil
    }

    // 2. 缓存未命中，尝试获取锁
    lockKey := "lock:product:" + id
    acquired, err := redis.SetNX(ctx, lockKey, "1", 5*time.Second)
    if err != nil || !acquired {
        // 没拿到锁，等待后重读缓存
        time.Sleep(50 * time.Millisecond)
        return GetProduct(id) // 递归重试
    }
    defer redis.Del(ctx, lockKey)

    // 3. 拿到锁的线程查数据库
    product, err := db.FindProduct(id)
    if err != nil {
        return nil, err
    }

    // 4. 写入缓存
    redis.Set(ctx, key, encode(product), 1*time.Hour)
    return product, nil
}
⚠️ 踩坑故事：某次大促活动，运营在后台发布了一批优惠券，key 的过期时间统一设成了凌晨 0 点。结果 0 点一到，几百万个缓存 key 同时过期，数据库连接池瞬间被打爆，整个下单流程阻塞了 20 分钟。修复方案就是把过期时间改成"基础时间 + 随机偏移"，再也没出过同样的问题。

MongoDB 适用场景

MongoDB 是一个文档型 NoSQL 数据库，数据以 BSON（类似 JSON）格式存储。它和关系型数据库的核心区别是：文档模型允许你在同一个文档里保存关联数据，不需要 JOIN。

MongoDB 最适合的场景是：

  • 文档型数据：内容管理系统（文章、评论、标签都在一起）、商品目录（不同品类有不同的属性集合）
  • 灵活 Schema：数据字段经常变化，或者不同文档有不同的字段结构。MongoDB 不需要预定义表结构。
  • 高写入吞吐：MongoDB 的写入性能在不要求严格事务的情况下优于大多数关系型数据库。
  • 聚合管道：MongoDB 的聚合框架非常强大，支持复杂的文档处理流水线——过滤、分组、排序、投影、关联（$lookup）等。
// MongoDB 文档模型示例
{
  "_id": ObjectId("..."),
  "title": "MongoDB实战指南",
  "author": { "name": "王五", "email": "wang@example.com" },
  "tags": ["数据库", "NoSQL", "后端"],
  "metadata": {
    "word_count": 12000,
    "reading_time_min": 45
  },
  "comments": [
    { "user": "user1001", "content": "好文！", "created_at": ISODate("...") },
    { "user": "user1002", "content": "学习了", "created_at": ISODate("...") }
  ],
  "created_at": ISODate("2026-06-01T08:00:00Z")
}

// 聚合管道：按标签统计文章数
db.articles.aggregate([
  { $unwind: "$tags" },
  { $group: { _id: "$tags", count: { $sum: 1 } } },
  { $sort: { count: -1 } },
  { $limit: 10 }
])
📌 选型建议（2026）：大多数新项目应该从 PostgreSQL 开始。当你发现遇到了"关系型数据库的痛点"——比如频繁的 Schema 变更、深层嵌套的数据结构、极高的写入吞吐——再考虑加入 MongoDB。不要为了用 NoSQL 而用 NoSQL。

实战：用 Redis 实现分布式锁的注意事项

分布式锁是后端开发中最常见的 Redis 应用之一。但实现一个"正确"的分布式锁，比看起来要复杂得多。

// 安全的 Redis 分布式锁（Go）
// 使用 SET NX + EX 原子操作 + Lua 脚本解锁

func AcquireLock(redis *RedisClient, lockKey string, ttl time.Duration) (bool, string) {
    token := uuid.New().String()  // 唯一标识，防止误解锁
    ok, err := redis.SetNX(ctx, lockKey, token, ttl)
    return ok, token
}

func ReleaseLock(redis *RedisClient, lockKey string, token string) {
    // 用 Lua 脚本保证比较和删除的原子性
    script := `
        if redis.call("GET", KEYS[1]) == ARGV[1] then
            return redis.call("DEL", KEYS[1])
        else
            return 0
        end
    `
    redis.Eval(ctx, script, []string{lockKey}, token)
}

实现分布式锁的几个关键原则：

  • 原子性：加锁必须用 SET NX EX（或 SETNX+EXPIRE 的 Lua 包装），不能用两个命令分别调用——否则如果加锁后进程崩溃，过期时间就设不上了，导致死锁。
  • 唯一标识：解锁时只能用 Lua 脚本比较 token，确保"谁加锁谁解锁"。否则一个请求可能错误地释放另一个请求的锁。
  • 锁超时：设置合理的过期时间。如果任务执行时间可能超过锁的过期时间，需要实现"续期"机制（watchdog pattern）。
  • Redlock：对于要求极高可靠性的场景（金融交易），Redis 官方提出的 Redlock 算法要求在多个独立的 Redis 节点上获取锁，以规避单点故障。但学术界对此算法有争议，谨慎使用。

缓存和 NoSQL 解决了"数据读得快"和"数据模型灵活"的问题，但它们解决不了"异步通信"的问题。当你的服务需要和其他服务解耦地交换消息，或者你需要处理高吞吐的事件流时，就需要一个更专门的组件——消息队列。下一章我们深入 Kafka、RabbitMQ 和 Redis Streams，看看消息队列如何改变系统架构。
