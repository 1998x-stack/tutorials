第16章 Quartus Prime与ModelSim

上一章我们完整走了一遍 Vivado 的工程流程——从创建工程到生成比特流，再到 ILA 片上调试。但 FPGA 领域并不是 AMD Xilinx 一家独大。对于许多工程师来说，Altera（现为 Intel Altera，2025 年从 Intel 独立并获 Silver Lake 投资）的 Quartus Prime 同样是日常工作不可或缺的工具。

Intel 在 2025 年将 Altera 分拆为独立子公司，保留了 49% 的股份，Silver Lake 投资 $87.5 亿。这一重大变化也在 2026 年对 Quartus 的产品策略产生了影响——免费版 Quartus Prime Lite 继续可用，但 Pro 版本的价格策略有所调整。无论如何，学习 Quartus 依然非常值得，因为 Altera 的 FPGA 在工业、通信和军工领域有着庞大的存量市场。

16.1 Intel/Altera 工具链全景

Altera 的工具链比 Xilinx 的体系要更分散一些，理解各个组件的定位有助于你快速上手：

温馨提示：如果你是学生或自学者，从 Quartus Prime Lite 开始就足够了。它不仅免费，而且 Cyclone IV/V 开发板（如 Terasic DE10-Lite、DE0-CV）价格便宜（约 ¥300-600），非常适合入门。另外需要注意的是，Quartus 的安装包比 Vivado 小不少（~8GB vs ~30GB），对磁盘空间更友好。

16.2 Quartus 标准工程流程

Quartus 的工程流程与 Vivado 在概念上是对应的，但各个步骤的名称和顺序有所不同。以 Cyclone V 上的 LED 流水灯为例：

第一步：New Project Wizard。File → New Project Wizard。指定工程路径、顶层实体名称（Top-Level Entity，相当于 Vivado 的 top module）。在 Device Family 中选择 Cyclone V，具体器件按开发板确定（如 DE10-Lite 使用 5CEBA2F23C7）。

第二步：添加设计文件。与 Vivado 不同，Quartus 允许直接编辑文件或通过 Project → Add/Remove Files in Project 添加。Quartus 对 Verilog-2001 和 SystemVerilog 的支持较好，但某些 VHDL 的语法检查比 Vivado 更严格。

第三步：Pin Planner。Assignments → Pin Planner 打开引脚规划器。与 Vivado 的 XDC 编辑方式不同，Quartus 的引脚分配是图形化拖拽式的——你可以在器件封装图上直接点击并分配引脚。Altera 同时提供 QSF（Quartus Settings File） 文本格式的约束，等价于 XDC。

第四步：Analysis & Synthesis。Processing → Start → Start Analysis & Synthesis。这一步对应 Vivado 的 synth_design，负责语法检查、逻辑综合和资源映射。综合完成后，Compilation Report 中会显示资源利用率、fan-out 分析、综合警告等详细信息。

第五步：Fitter。Processing → Start → Start Fitter（或直接 Start Compilation 一步到位）。Fitter 对应 Vivado 的 place_design + route_design。Quartus 的 Fitter 的智能程度很高——它的增量编译（Incremental Compilation）功能比 Vivado 的 Incremental Implementation 成熟得更早。

第六步：Assembler & Programmer。Fitter 完成后，Assembler 自动生成 .sof（SRAM Object File，对应 .bit）和 .pof（Programmer Object File，对应 .mcs）。然后通过 Tools → Programmer 打开烧录界面，选择 USB-Blaster 下载器，点击 Start 将 .sof 烧录到 FPGA。

16.3 SignalTap II 逻辑分析仪

SignalTap II 是 Quartus 内置的片上调试工具，与 Xilinx 的 ILA 功能类似，但在配置和使用方式上有一些有趣的差异。

添加 SignalTap II 文件：File → New → SignalTap II Logic Analyzer File（.stp）。在 SignalTap II 窗口中，你需要配置：

  • 采样时钟（Clock）：选择采样时钟信号，通常是系统时钟。注意 SignalTap II 是上升沿采样的。
  • 探针信号（Nodes）：点击 List 按钮，在设计中搜索要观察的信号。Quartus 会自动列出所有可观察的信号，你可以把感兴趣的信号拖到 SignalTap II 的信号列表中。
  • 采样深度（Sample Depth）：可选 128/256/512/1K/2K/4K/8K/16K。深度越大，占用的 M9K/M20K Block RAM 越多。
  • 触发条件（Trigger Conditions）：支持两级触发和多个触发条件组合。每个探针可以设置触发值或范围。
  • 存储限定（Storage Qualifier）：可以选择只存储满足特定条件的样本——这对于减少 BRAM 占用或在有限的深度下捕获更长的有效数据很有用。

配置完成后，重新综合并下载包含 SignalTap II 的比特流。在 Programmer 中下载时，.sof 会自动包含调试逻辑。然后回到 SignalTap II 窗口，点击 Run Analysis 等待触发。

SignalTap II vs ILA 的小差异：在 ILA 中，你需要显式例化 IP 核并在 RTL 中连接探针。而 SignalTap II 是在综合后的网表上直接附加探针的——你不需要修改 RTL 源码。这意味着 SignalTap II 可以在不修改设计的情况下临时添加调试逻辑。但这也带来一个缺点：某些经过优化被综合工具合并或消除的内部信号可能无法被 SignalTap II 捕获。你需要设置 set_keep 综合属性来保留这些信号。

16.4 ModelSim 仿真集成

Quartus 自带的仿真器（Questa 的入门版）功能有限，大多数工程师选择在 Quartus 中调用 ModelSim 进行仿真。ModelSim 是 FPGA 行业最经典的仿真器——它比 Vivado Simulator 更快（尤其对于 VHDL）、波形显示更清晰、脚本控制（.do 文件）更成熟。

模型集成流程：

  • 在 Quartus 中设置 EDA 工具：Assignments → Settings → EDA Tool Settings → Simulation。指定工具为 ModelSim（或 Questa），语言为 Verilog，并设置输出网表格式。
  • 编译库文件：第一次使用时，需要编译 Altera 的器件库到 ModelSim 中。Quartus 提供了 Tools → Launch Simulation Library Compiler 来自动完成这一步。选择目标仿真器安装路径和器件家族，点击 Start Compilation。
  • 启动 ModelSim：在 Quartus 的 Processing → Start → Start Testbench Template，或者直接在 Tools → Run EDA Simulation → EDA RTL Simulation 中启动。
  • 使用 .do 脚本：ModelSim 的强大之处在于 .do 文件——一个 Tcl 命令序列文件。你可以写一个 run_sim.do：
# ModelSim .do 文件示例：自动运行仿真
vlib work
vmap work work

# 编译设计文件
vlog -work work +incdir+../rtl ../rtl/top.v
vlog -work work +incdir+../rtl ../rtl/uart_tx.v
vlog -work work ../tb/tb_top.v

# 加载仿真
vsim -voptargs=+acc work.tb_top

# 添加波形信号
add wave -hex /tb_top/u_uart_tx/*
add wave -hex /tb_top/clk
add wave -hex /tb_top/rst_n

# 运行仿真
run 100 us

# 退出
# quit -sim

在 ModelSim 命令行中执行 do run_sim.do 即可一键完成编译、加载、跑仿真和添加波形。

16.5 Vivado vs Quartus 对比

作为一名 FPGA 工程师，你大概率需要同时使用 Vivado 和 Quartus——因为你不知道下一个项目会用到哪家的芯片。下面这张对比表可以帮你在两个工具之间快速切换：

维度Vivado (AMD Xilinx)Quartus Prime (Intel Altera)
工程管理.xpr 文件（XML），自动刷新；Project Summary 面板清晰.qpf 文件，手动刷新更常用；界面稍显陈旧
IP CatalogIP Catalog 树状结构，搜索速度快；IP 版本独立锁定IP Catalog 与 Platform Designer 分离；IP 版本号与 Quartus 版本绑定
约束语法XDC（Tcl 基），set_property + create_clockQSF（属性键值对） + SDC（Tcl 基时序约束）
片上调试ILA（需 RTL 例化 IP 核）SignalTap II（网表级添加，无需改 RTL）
仿真器Vivado Simulator (xsim) 轻量够用；支持 SystemVerilog 覆盖率ModelSim/Questa OEM 更成熟；VHDL 支持更好
综合策略综合较快，默认策略偏向性能优化；支持 OOC 综合综合较慢但更彻底；增量编译能力更强
时序分析report_timing_summary 功能强大；图形化 Timing Report 交互性好Timing Analyzer 报告格式更传统；但 SDC 兼容性更好

学习曲线中等；GUI 现代化，用户友好中等偏陡；功能菜单分散，新手容易迷路

Quartus 的约束语法分为 QSF 和 SDC 两个层面。QSF 管理工程级属性（器件型号、引脚分配），SDC 管理时序约束。下面是一个典型的 SDC 约束示例：

## ====================================
## SDC Timing Constraints for Cyclone V
## ====================================

## 主时钟约束
create_clock -name sys_clk -period 10.000 [get_ports clk]

## PLL 输出时钟自动约束（需 derive_pll_clocks 扫描）
derive_pll_clocks
derive_clock_uncertainty

## 输入延迟（外部器件为 5ns 建立时间参考）
set_input_delay -clock sys_clk -max 5.000 [get_ports {data_in*}]
set_input_delay -clock sys_clk -min 2.000 [get_ports {data_in*}]

## 输出延迟（外部器件为 3ns 保持时间参考）
set_output_delay -clock sys_clk -max 4.000 [get_ports {data_out*}]
set_output_delay -clock sys_clk -min 1.500 [get_ports {data_out*}]

## 异步信号假路径约束
set_false_path -from [get_registers rst_meta*]
踩坑实录——PLL 静默重配置：我曾经把一个 Quartus 17 的工程迁移到 Quartus 23 上继续开发。打开工程，重新编译，一切看起来很顺利。直到我测试板子时发现——系统时钟频率变成了原来的两倍。排查了一整天，最后发现 Quartus 17 中配置的 PLL IP 在 Quartus 23 中打开时，工具"友好"地提示我 PLL 参数已在新版本中更新并自动迁移——但迁移后的倍频系数变了！Quartus 没有给出任何明显的警告，PLL 的输出频率从 50MHz 变成了 100MHz。从此以后，任何 IP 核的版本迁移，我都会在综合报告里手动验证 report_clocks 输出的频率值，绝不信任"自动迁移"的默认行为。

两家工具链各有长处。Vivado 的 UI 更现代化、调试工具更直观；Quartus 的编译质量更稳定、VHDL 支持和增量编译更强。优秀的 FPGA 工程师应该两套工具都会用——从实际项目需求出发选择最合适的平台。

下一章，我们将深入一个核心话题——无论是 Vivado 还是 Quartus，都离不开对 IP 核的集成和使用。PLL、FIFO、Block RAM、AXI GPIO——掌握这些常用 IP 核的配置和连接方式，是提升开发效率的关键。
