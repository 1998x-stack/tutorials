第19章 ASCII协议：proto_text详解

大家好，我是你们的讲师。这一章我们进入一个"看得见摸得着"的模块——ASCII 文本协议。如果你用 telnet 连过 Memcached，那你一定敲过这样的命令：set mykey 0 3600 5，然后回车输入 hello，再回车。这个看似简单的交互背后，是 proto_text.c 中约 2000 行代码在支撑着整个协议解析与分发的全过程。

从某种意义上说，ASCII 协议是 Memcached 最古老也最有魅力的接口。它不像二进制协议那样紧凑，但它的可读性极强——任何人都可以用 telnet/nc 直接与 Memcached 对话，不需要任何客户端库。正是这种"零依赖"的亲和力，让 Memcached 在早期迅速流行。不过，简单不等于简陋。ASCII 协议的状态机设计、tokenize 分发机制、以及 noreply 优化，都是值得仔细品读的经典代码。

ASCII 协议文本格式

我们先统一看一下 ASCII 协议的基本命令格式。每一行都遵循严格的文本结构，服务器按行读取并解析：

存储命令（SET/ADD/REPLACE）:
  <command> <key> <flags> <exptime> <bytes> [noreply]\r\n
  <data block>\r\n

  示例: set username 0 3600 5\r\nhello\r\n

读取命令（GET/GETS）:
  get <key>\r\n
  gets <key>\r\n

删除命令（DELETE）:
  delete <key> [noreply]\r\n

算术命令（INCR/DECR）:
  incr <key> <value> [noreply]\r\n
  decr <key> <value> [noreply]\r\n

注意每个字段的含义：flags是客户端传入的 32 位无符号整数（服务器不解析，直接存储原样返回）；exptime是过期时间，为 0 表示永不过期，小于 30 天视为相对秒数，否则视为绝对 Unix 时间戳；bytes是 data block 的字节数（不包含最后的 \r\n）。

一个容易忽略的细节：bytes后面必须紧跟着 [noreply]（如果有的话），中间不能有多余的空格。Memcached 使用 strtok_r 进行 tokenize，多余的空格会导致 token 数量不正确，从而被判定为命令格式错误。

process_command()：命令分发的入口

所有 ASCII 命令的解析起点是 process_command() 函数。它的工作流程非常清晰：

  1. Tokenize：将输入缓冲区的一行文本按空格拆分成 token 数组
  2. 查命令表：遍历命令映射表 command_table[]，通过 strncmp 匹配命令名称
  3. 执行：调用对应的处理函数

我们来看命令映射表的数据结构——它本质上是一个朴素的线性查找表：

static void process_command(conn *c, char *command) {
    token_t tokens[MAX_TOKENS];
    int ntokens;
    int comm;

    // Step 1: tokenize
    ntokens = tokenize_command(command, tokens, MAX_TOKENS);
    if (ntokens <= 0) {
        out_string(c, "CLIENT_ERROR bad command line format");
        return;
    }

    // Step 2: 查表匹配命令
    for (comm = 0; comm < NUM_COMMANDS; comm++) {
        if (!strncmp(tokens[COMMAND_TOKEN].value,
                     command_table[comm].name,
                     strlen(command_table[comm].name))) {
            break;
        }
    }
    if (comm == NUM_COMMANDS) {
        out_string(c, "ERROR");
        return;
    }

    // Step 3: 根据命令类型分流
    switch ((enum commands) comm) {
        case GET:
        case GETS:
            process_get_command(c, tokens, ntokens);
            break;
        case SET:
        case ADD:
        case REPLACE:
            process_update_command(c, tokens, ntokens, comm, false);
            break;
        case DELETE:
            process_delete_command(c, tokens, ntokens);
            break;
        // ... 其他命令
    }
}
值得留意的设计：命令匹配用的是朴素的线性查找——对于只有十几个命令的表来说，这比哈希表更简洁，而且 CPU cache 更友好。不要在任何地方都用"高级"算法，简单就是美。

GET 命令处理

GET 是最简单的命令，也是理解 Memcached 读写路径最好的起点。process_get_command() 的工作流程如下：

  1. 遍历命令行中所有的 key（支持多 key GET，一次 get key1 key2 key3）
  2. 对每个 key 调用 item_get()——这个函数会在哈希表中查找，同时更新 LRU
  3. 如果找到，调用 out_string() 或 add_iov() 发送 VALUE 行和数据
  4. 最后发送 END\r\n 标记结束

item_get() 最终调用 assoc_find() 在哈希表中定位 item。这里有个关键细节：GET 操作会通过 do_item_update() 将 item 移到 LRU 的 HOT 队列头部——表示这个 item 刚刚被访问过，不要急着淘汰它。当然，这一步是否执行取决于 settings.lru_maintainer_thread 配置。

性能细节：多 key GET 并不是在协议层面做了"批量操作"优化，它只是依次处理每个 key。真正的性能提升来自于减少了网络往返——客户端只需一次请求就能获取多个 key 的值。如果你需要原子性的批量操作，应该使用后面章节介绍的 Meta Protocol。

SET/ADD/REPLACE 命令处理

存储类命令的处理流程比 GET 复杂得多。process_update_command() 是核心函数，它的执行路径分为两个阶段：

第一阶段：命令解析与 item 预分配

在 process_update_command() 中，解析出 key、flags、exptime、bytes 等参数后，立即调用 item_alloc() 分配一个 item 结构体。注意此时 data block 还没有读上来——因为 TCP 是流式传输，value 数据可能还在路上。

第二阶段：分段读取 value 数据

Memcached 用了一个很有意思的技巧：当检测到命令需要读取 data block 时，它设置 c->noreply、c->item 和 c->ritem，然后把连接的状态切换为 conn_nread：

static void process_update_command(conn *c, token_t *tokens,
                                    const int ntokens, const int comm,
                                    bool meta) {
    // ... 参数解析 ...

    it = item_alloc(key, nkey, flags, realtime(exptime),
                    vlen, meta ? 2 : 0);
    // 切换到 nread 状态，等待 data block
    c->item = it;
    c->ritem = ITEM_data(it);
    c->rlbytes = it->nbytes;
    c->cmd = comm;
    c->substate = META_NREAD;  // 对于 ASCII: conn_nread
    conn_set_state(c, conn_nread);
}

当连接状态切换到 conn_nread 后，drive_machine() 会持续调用 read() 读取剩余字节，直到 c->rlbytes 归零。这时，数据已经完整地写入到了 item 的 data 区域中。然后调用 complete_nread() 完成最终的处理：存储 item、发送响应。

踩坑：大 value 的 SET 导致连接阻塞。有一次线上问题，业务方把一个 10MB 的图片 base64 编码后直接塞进 Memcached。结果这个 SET 请求将 worker 线程完全"焊死"在 conn_nread 状态上——因为这个线程在持续读取这个巨型 value 的过程中，无法处理其他连接。解决方案有两个：要么在应用层限制 value 大小（推荐 1MB 以内），要么配置 -I 参数（默认 1MB）显式拒绝超大 item。记住：Memcached 不是对象存储，它是缓存。

INCR/DECR 命令处理

INCR 和 DECR 是 ASCII 协议中唯一涉及"运算"的命令。它的实现非常巧妙：

  1. 调用 item_get() 找到对应的 item
  2. 将 item 的 data 部分解析为 64 位无符号整数（使用 strtoull）
  3. 执行加减操作（注意处理溢出：INCR 溢出则环绕，DECR 到 0 则不再减少）
  4. 将结果写回 item 的 data 区域（更新为 ASCII 字符串形式）
  5. 使用 do_item_update() 更新 LRU 位置
  6. 返回新的数值

这里有一个有趣的实现细节：Memcached 没有使用任何原子操作原语（如 __sync_fetch_and_add）来实现 INCR，因为它全程持有 item 的锁（通过 item_lock()）。这个锁保护的是哈希表中的那个 item 结构体本身——有了这把锁，对 item data 的读写自然就是串行化的。

数值解析的陷阱：Memcached 的 INCR/DECR 要求值必须是一个十进制的数字字符串，不能有前导空格或非数字字符。如果你传入 incr mykey 0x10，strtoull 不会解析十六进制——它只会返回 0。这在调试时常常让人困惑，以为 INCR 没有生效。

noreply 优化

这是 ASCII 协议中一个非常重要的优化特性。当客户端在命令末尾添加 noreply 关键字时，服务端处理完命令后不会发送任何响应。对于 SET、DELETE、INCR 等命令来说，这意味着节省了一次网络传输。

在源码中，noreply 的处理逻辑在 add_iov() 之前就判断了：

// 在 complete_nread 或 process_command 中:
if (c->noreply) {
    // 不调用 out_string()，直接返回
    conn_set_state(c, conn_new_cmd);
    return;
}
out_string(c, "STORED\r\n");  // 只有 noreply=false 时才发送

noreply 的应用场景非常明确：写操作不需要确认。比如日志处理场景中，每条日志写入 Memcached 后不需要关心是否写成功（丢了也没关系），这时用 noreply 可以减少约 30% 的网络开销。但注意：读操作不能用 noreply——你读数据就是想等结果，这是语义上矛盾的事情。

CAS 命令的协议实现

CAS（Compare-And-Swap）是 Memcached 实现乐观并发控制的核心机制。它由两个命令配合使用：

  • gets：和 get 类似，但会额外返回一个 CAS token（64 位无符号整数）
  • cas：替代 set，但需要携带 CAS token，只有 token 匹配时才执行更新

CAS token 本质上就是 item 的 cas_id 字段，每次 item 被修改时都会递增。源码中的实现路径非常直接：

// gets 命令发送 CAS token:
out_string(c, "VALUE %s %d %d %llu\r\n",
           ITEM_key(it), it->flags, it->nbytes - 2,
           (unsigned long long) ITEM_get_cas(it));

// cas 命令执行时比对:
if (ITEM_get_cas(it) == req_cas_id) {
    // 匹配——更新
    do_item_replace(it, new_it, c->hlen);
    out_string(c, "STORED\r\n");
} else {
    // 不匹配——说明被其他客户端改过了
    out_string(c, "EXISTS\r\n");
}

CAS 最常见的应用场景是"读取-修改-写回"模式。比如一个计数器的值需要先读取当前值，加 1，再写回。如果不加 CAS，两个并发客户端可能互相覆盖。有了 CAS，第二个写回的请求会因为 token 不匹配而被拒绝，客户端可以重新读取再尝试。

命令 功能 支持 noreply 响应格式
GET 读取 key 的值 否 VALUE <key> <flags> <bytes>\r\n<data>\r\nEND
GETS 读取 + 获取 CAS token 否 VALUE <key> <flags> <bytes> <cas>\r\n<data>\r\nEND
SET 存储（存在则覆盖） 是 STORED / NOT_STORED
ADD 存储（仅当 key 不存在） 是 STORED / NOT_STORED
REPLACE 存储（仅当 key 已存在） 是 STORED / NOT_STORED
DELETE 删除 key 是 DELETED / NOT_FOUND
INCR 原子自增 是 <new value> / NOT_FOUND
DECR 原子自减 是 <new value> / NOT_FOUND
CAS 条件更新（需 CAS token） 是 STORED / EXISTS / NOT_FOUND

process_get_command / process_update_command 等函数分析

我们快速过一遍 ASCII 协议部分的主要函数，方便你对照源码阅读：

process_get_command()

位于 proto_text.c 中，负责处理 get 和 gets 命令。它遍历所有的 key token，逐个调用 item_get()，然后通过 add_iov() 将结果写入连接输出缓冲区。注意它使用了 conn_set_state(c, conn_mwrite) 将连接状态切换到"多写"模式——这是为了处理多 key GET 的批量响应。

这个函数有个性能考量：如果 GET 的 key 列表很长（比如一次 GET 100 个 key），它会针对每个 key 调用 add_iov() 将响应追加到 msglist 中，最后统一通过 transmit() 发送。这样可以一次系统调用发送完整响应。

process_update_command()

处理 SET/ADD/REPLACE 等更新命令。核心逻辑包括：

  • 采用 tokenize 验证参数数量
  • 检查 key 长度（settings.max_nkey）和 value 大小（settings.max_item_size）
  • 创建临时 item（item_alloc），设定连接状态为 conn_nread
  • 等待数据读满后，在 complete_nread_ascii() 中完成最终的存储

process_delete_command()

处理 DELETE 命令，逻辑较简单：调用 item_delete()，如果 noreply 未设置则返回 DELETED 或 NOT_FOUND。

process_arithmetic_command()

处理 INCR/DECR 命令，核心是调用 do_add_delta() 函数进行数值运算。这个函数直接操作 item 的 data 区域，如果 data 不是合法的数字字符串，则返回 CLIENT_ERROR cannot increment or decrement non-numeric value。

踩坑续谈：大 value 阻塞的另一个视角。除了前面提到的 conn_nread 阻塞，大 value 还会引起另一个问题——输出缓冲区爆炸。当 GET 一个超大的 item 时，响应数据也会很大。如果客户端读取缓慢（TCP 窗口变小），服务端的输出缓冲区会持续堆积，最终触发 conn_closing 状态，服务端主动断开连接。这看起来就像"无故断连"。解决方案始终是那一条：控制 value 的大小，把 Memcached 当作纯缓存，不要当作对象存储。

从 ASCII 到二进制——协议的演进之路

ASCII 协议虽然简洁直观，但随着 Memcached 在生产环境中的大规模使用，它的局限性也逐渐暴露出来：

  • 解析效率低：每一行都要做 strtok、strncmp、atoi 等字符串操作，即使是简单的 GET 命令也需要遍历命令表
  • 带宽浪费："VALUE"、"STORED"、"END"这些文本关键字每次都在网络上传输，累积起来是相当可观的字节开销
  • 类型信息丢失：所有的 flags、exptime 都以 ASCII 字符串方式传输，解析后还得转成整数

正是这些瓶颈催生了二进制协议（Binary Protocol）的出现。二进制协议用固定长度的 header 代替了文本行解析，用 opcode 代替了字符串命令名，用 struct 代替了 tokenize。下一章我们会深入分析 binary 协议的细节。

不过我想强调一点：ASCII 协议并没有被废弃。即使在最新的 Memcached 1.6.x 中，ASCII 协议仍然被广泛支持和使用。很多 telnet 调试场景、简单的 get/set 操作、以及一些轻量级的客户端库仍然首选 ASCII 协议。它的生命力来自它的简单——你甚至可以用 echo "get mykey" | nc memcache-host 11211 来测试缓存内容。

章节小结：这一章我们深入了 proto_text.c 的核心代码。从 process_command() 的命令表分发，到 GET 的哈希查找+LRU 更新，再到 SET 的两阶段读取，再到 INCR 的原子运算和 CAS 的乐观锁——ASCII 协议虽然"简单"，但它覆盖了 Memcached 全部数据操作的基础路径。下一章我们将切换到二进制协议的世界，看看 24 字节的 Binary Header 是如何把性能再提升一个台阶的。
