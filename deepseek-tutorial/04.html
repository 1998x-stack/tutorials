第4章 数据管线

大家好，欢迎回到 DeepSeek 系列教程。上一章我们聊了 10 万 BPE 词表的设计——那是数据进入模型前的第一步。但问题是：这些数据从哪里来？如何从浩瀚的互联网中筛选出真正高质量的训练语料？中英文数据应该按什么比例混合？

如果说模型架构是大模型的"骨架"，训练算法是"肌肉"，那数据就是"血液"——它流经每一个参数，决定了模型的知识广度和质量上限。DeepSeek 在数据管线上的投入丝毫不亚于他们在架构创新上的精力。本章我们就来拆解 DeepSeek 的数据管线：从数据采集、清洗、去重，到质量过滤和配比实验。

4.1 数据来源

DeepSeek LLM 的预训练数据总量约为 2T tokens，这是一个精心构建的多源数据集。主要来源包括：

数据源 占比（估） 语言 特点
CommonCrawl ~60% 中/英/多语 海量但噪声大，需要深度清洗
书籍 ~10% 中/英 高质量长文本，知识密度高
学术论文 ~8% 英（为主） 专业知识，严谨推理
代码仓库 ~12% 编程语言 结构化数据，逻辑性强
维基百科 ~3% 中/英 高质量百科，知识覆盖广
新闻 / 社交媒体 ~7% 中/英 最新信息，口语化表达

这个配比反映了几个设计理念：

  • 互联网数据为主（CommonCrawl 占 60%）——只有互联网数据能提供足够的规模（T 级）和多样性
  • 代码数据占比很高（12%）——代码不仅是编程能力的来源，更关键的是它训练了模型的逻辑推理和结构化思维能力
  • 高质量长文本是知识的骨架——书籍和论文虽然占比不高，但提供了深度知识，是模型"深度"的来源
一个有趣的对比：LLaMA 2 的代码数据占比约为 5%，而 DeepSeek 的代码占比达到了 12%。这个差异可能部分解释了 DeepSeek 模型在 HumanEval 等代码评测上远远领先同规模 LLaMA 的原因。代码数据不仅教会模型写代码，更重要的是培养了模型的"精确推理"能力——毕竟代码不允许模糊，对就是对，错就是错。

4.2 清洗流程

原始互联网数据是极其肮脏的——HTML 标签、广告、垃圾信息、恶意内容、乱码、重复内容……DeepSeek 的清洗管线包含多个环节：

第一阶段：格式清洗。从 CommonCrawl 获得的 raw dump 首先是 WARC 格式的压缩数据。清洗管线首先提取纯文本内容，去除 HTML 标签、script、style 等无关元素。然后进行编码检测，丢弃非 UTF-8 编码的文档。这一步大约会丢弃 5-10% 的原始数据。

第二阶段：敏感信息过滤。包括 PII（个人身份信息，如身份证号、银行卡号）的正则匹配和过滤、色情和暴力内容的检测（基于关键词 + 分类器）、以及恶意 URL 的过滤。这一步不仅是合规要求，也直接影响模型的安全性。

第三阶段：低质量文档检测。这是最关键的环节。DeepSeek 使用了一套多维度的质量评分系统：

质量评分系统 = 加权组合多个指标:

1. 困惑度（Perplexity）得分
   - 用预训练的 n-gram 或小语言模型评估文档的"自然度"
   - 垃圾内容的困惑度通常很高

2. 语言模型概率
   - 用一个小型 LM 计算文档的平均 log-probability
   - 高质量文档的 LM 概率更高

3. 启发式规则
   - 标点符号比例（过大或过小都不好）
   - 停用词密度（英文）、字频分布（中文）
   - 平均句子长度（太短 = 碎片化，太长 = 可能乱码）
   - 重复 n-gram 比例（高重复 = 低质量）

4. 信息密度指标
   - 每 token 的熵
   - 命名实体密度

最终得分 = w1 × 困惑度 + w2 × LM概率 + w3 × 启发式得分
得分低于阈值 → 丢弃

整个清洗流程完成后，从 CommonCrawl 中筛选出的高质量文本通常只占原始数据的 10%-20%。也就是说，每 10TB 的原始网页数据，清洗后只剩下 1-2TB 可用于训练。

4.3 去重技术

互联网数据中有大量的重复内容——同一篇文章被多个网站转载、同一个新闻被多次报道、模板化生成的垃圾页面。如果不做去重，模型会在重复数据上"过拟合"，浪费训练 token 预算，还可能导致生成的输出模板化。

DeepSeek 的数据管线包含三个层次的去重：

精确去重（Exact Dedup）：最直接的方法。对每篇文档计算 URL 去重 + MD5 hash 对比。同一 URL 只保留最新版本，同一内容的 MD5 值一致则只保留一份。这个级别的去重通常能消除 5-10% 的重复。

模糊去重（Fuzzy Dedup via MinHash + LSH）：这是去重的主力。MinHash 是一种近似集合相似度计算方法，对一篇文档抽取 n-gram 特征集合，计算 MinHash 签名，再用 LSH（Locality Sensitive Hashing）高效地找到相似文档。

MinHash + LSH 去重流程:

文档 A: "深度学习是机器学习的一个分支领域..."
文档 B: "深度学习是机器学习的重要分支..."

1. 抽特征: 将每篇文档转化为 5-gram 集合
   A_fingerprint = {"深度学", "度学习", "机器学", ...}
   B_fingerprint = {"深度学", "度学习", "机器学", ...}

2. MinHash 签名: 用 k 个 hash 函数计算签名
   得到每篇文档的 k 维签名向量（k=128 或 256）

3. LSH 分桶: 将 k 维签名分成 b 个 band，每 band r 行
   签名相似度高 → 落入相同 bucket 的概率高

4. 去重判定: 落入相同 bucket 的文档对 → 计算 Jaccard 相似度
   相似度 > 阈值（通常 0.8-0.85）→ 保留一篇

模糊去重的效果非常显著——能消除约 15-25% 的近似重复数据。而且因为去掉了大量相似但不完全相同的"变体"，模型学到的知识更加纯净，不会因为重复而放大某些特定表述的权重。

跨数据集去重：除了在单个数据集内部去重，DeepSeek 还在不同数据源之间做去重。例如，某个网页内容可能既出现在 CommonCrawl 中，也出现在某个书籍合集中。如果没有跨数据集去重，同一内容会被模型"看到"多次，导致 token 预算的浪费和数据分布的偏差。

真实踩坑经历：有开源社区团队分享过这样的教训——他们在训练一个 7B 模型时没有做跨数据集去重，结果模型在评测中表现出"近乎完美"的准确率。后来发现，评测集的某些题目在训练数据中以不同的形式出现了 3 次以上，导致测试集泄漏。这提醒我们：去重不仅是效率问题，更是数据伦理问题。一个不严谨的数据管线可能让模型的评测分数虚高，掩盖真正的能力短板。

4.4 双语语料配比策略

DeepSeek LLM 是一个双语模型——目标是在中文和英文上都达到优秀的性能。但中英文数据的配比是一个微妙的问题。

直觉上，如果目标是在中文上表现好，那应该多放中文数据。但现实情况恰恰相反：高质量的中文互联网数据远少于英文。据 DeepSeek 的估计，同等质量门槛下，可获取的中文文本数据量只有英文的 1/5 到 1/3。如果强行按照 1:1 的比例混合，意味着中文数据很快就会被"榨干"，而英文数据还有很多未使用。

DeepSeek 的策略是：

  • 英文数据占主体（约 60-70%）——因为英文数据量大、质量高、多样性好
  • 中文数据保持一定比例（约 20-30%）——确保模型学到的中文知识足够丰富
  • 其他语言少量补充（约 5%）——主要用于代码和多语言泛化能力

这个配比的逻辑是：英文能力是中文能力的基础。高质量的英文语料中包含大量知识（科学、技术、推理模式），这些知识通过注意力机制可以迁移到中文理解中。实际上，实验中确实发现——即使中文数据比例不高，模型在中文任务上依然表现良好，得益于知识从英文向中文的跨语言迁移。

语言配比 C-Eval (中文) MMLU (英文) 观察
100% 英文 42.3 74.1 中文全靠迁移，能力有限
80% 英 + 20% 中 59.8 75.2 中文大幅提升，英文几乎不变
60% 英 + 40% 中 62.1 74.8 中文继续提升，英文略降
40% 英 + 60% 中 63.5 72.3 中文收益递减，英文明显下降

实验数据显示：20% 的中文数据就能让 C-Eval 提升超过 17 个百分点（从 42.3 到 59.8），而继续增加中文到 40% 时，中文收益开始递减，英文反而开始下降。因此 20-30% 的中文比例是一个合理的甜点区。

重要澄清：中文数据比例不是"越多越好"。有些团队盲目追求中文数据占比达到 50% 甚至 60%，结果模型的中文能力确实很好，但英文能力严重退化——更糟糕的是，由于英文是国际学术界和工程界的通用语言，英文能力退化意味着模型无法理解最新的论文、代码文档和 API。对于追求双语能力的模型来说，"中文足够好，英文足够强"是最优目标，而非"中文最好，英文凑合"。

4.5 质量过滤

数据质量的重要性怎么强调都不为过。DeepSeek 在论文中明确指出：数据质量比数据数量更重要。在有限的 token 预算下，用高质量的文档替换低质量的文档，能带来持续的模型性能提升。

DeepSeek 使用了混合式的质量过滤策略：

基于启发式规则的过滤：

  • 文档长度过滤（太短或太长都可能有问题）
  • 特殊字符比例（过多的乱码、无关符号）
  • 语言一致性检测（一段"中文"文本中英文比例过高可能是混合文本）
  • 广告 / 模板文本检测（"点击这里阅读全文"等模板化表述）

基于分类器的过滤：

DeepSeek 训练了一个小型分类器（基于 BERT 或类似架构），在人工标注的高质量和低质量文本对上微调。这个分类器可以对每个文档给出一个质量分数。相比于规则过滤，分类器能捕捉到更微妙的"质量信号"——比如文风的流畅度、推理的逻辑性、知识的深度等。

最终的过滤管线是两级的：先通过规则过滤快速筛掉明显不合格的文档（吞吐量大、成本低），再通过分类器对剩余文档进行精细评分（精度高、成本高）。这种"粗筛 + 精调"的组合能有效兼顾效率和精度。

4.6 数据配比实验

除了语言比例的实验，DeepSeek 还在更广泛的维度上探索了数据配比对模型性能的影响。这些实验为后来的 DeepSeek-V2 和 V3 提供了宝贵的数据策略指导。

实验一：数据源多样性。在其他条件不变的情况下，当数据来源从 3 个（CommonCrawl + 书籍 + 百科）扩展到 6 个（添加代码、论文、社交媒体）时，模型在 BBH 和 GSM8K 等推理任务上的分数提升了 5-8%。这说明数据源的多样性对推理能力的提升比单纯关注"总量"更重要。

实验二：重复数据的影响。当数据被重复训练 2-4 次时，模型性能反而有所提升——这可能是因为重复让模型有更多机会"记住"关键知识。但当重复超过 4 次后，性能开始饱和或下降。这说明"适度的数据重复"不一定是坏事，关键在于不要过度。

实验三：数据新鲜度。在固定预算下，纳入 2023 年的最新数据比多放一些 2020 年的数据对模型有帮助——与直觉一致，最新的数据包含了最新的知识和语言表达方式。但加入太多新鲜数据而压缩了基础知识的比例，性能也会下降。大约 15-20% 的新鲜数据比例是最优的。

总结一下数据配比的黄金法则：DeepSeek 的数据实验告诉我们，高质量数据的配比不是"单目标优化"——不能只看中文分数，不能只看总量，不能只追求新鲜度。它是在多个互斥目标（中文 vs 英文、质量 vs 数量、时效 vs 经典）之间寻找平衡。而找到这个平衡点的唯一方法，就是做大量系统的消融实验。这也是 DeepSeek 多步学习率调度器如此重要的原因——只有低成本的实验方法，才能支撑如此大规模的数据配比探索。

这一章的内容可能没有架构创新那么"性感"，但数据管线的重要性绝对不亚于模型设计。DeepSeek 在数据清洗、去重、配比上的系统方法论，是他们能训练出高质量模型的重要一环。用一句行业里的大实话来总结："垃圾进，垃圾出"——再好的模型架构，也救不了糟糕的数据质量。

经过预训练的数据准备和模型训练，我们得到的只是一个"基座模型"——它掌握了语言知识和推理能力，但还不会像助手一样与人类对话。下一步，我们需要对模型进行对齐微调，让它变成真正有用的聊天助手。下一章，我们就来深入探讨 DeepSeek 的 SFT + DPO 对齐策略。
