第6章 嵌入式系统基础

大家好，我是你们的讲师。前面五章我们主要在"纸上谈兵"——讨论概念、协议、拓扑和标准。从这一章开始，我们要开始真正触碰WSN的"肉体"：嵌入式系统。

WSN节点本质上是一个微型的嵌入式计算机：它有CPU、内存、外设接口，运行一套完整的软件栈。但和你的笔记本电脑不同，它的CPU只有MHz级别、RAM只有KB级别、存储只有几十到几百KB——而且只能用电池供电。这迫使WSN的嵌入式开发必须遵循一套完全不同的设计哲学：每一微安都精打细算，每一字节都弥足珍贵。

这一章我们从WSN节点的硬件架构讲起，然后深入MCU选型、实时操作系统（RTOS）和低功耗设计这三个核心课题。我会结合一个真实的故事开始：某团队在做冷链物流WSN节点时，选了一款功能强大的MCU（ARM Cortex-M4F，带浮点运算），信心满满地完成了开发。结果在功耗测试中发现——即使是睡眠模式，电流也有30μA，远超预算。而竞争对手用的是Cortex-M0芯片，同样是睡眠模式只有2μA，电池续航直接差了一个数量级。这个故事完美诠释了WSN嵌入式系统的核心原则：功能够用就好，功耗越低越好。

学完本章你将能够： 读懂WSN节点的硬件框图、理解MCU选型的关键参数、掌握RTOS在WSN中的角色，以及学会低功耗模式的配置方法——这些是动手做WSN节点开发的必备能力。

WSN节点硬件架构

一个典型的WSN节点由四个基本模块组成：

  • MCU模块（Microcontroller Unit）：节点的"大脑"。负责处理传感器数据、运行协议栈、控制无线电、管理电源。通常是一颗32位ARM Cortex-M系列MCU或RISC-V MCU，主频从几十MHz到几百MHz。
  • RF收发器（Radio Transceiver）：节点的"嘴巴和耳朵"。负责无线数据的发送和接收。在集成度较高的SoC方案中，RF收发器和MCU集成在同一个芯片中（如CC2538、nRF52840）。在分立的方案中，RF收发器是独立芯片（如CC1101、SX1278 LoRa芯片），通过SPI接口与MCU通信。
  • 传感器模块（Sensor Module）：节点的"五官"。可以是板上集成的传感器（如BME280温湿度气压传感器），也可以是外部扩展的传感器阵列（通过I²C/SPI/GPIO/ADC连接）。
  • 电源模块（Power Module）：节点的"心脏"。通常由电池供电（AA电池、锂电池、纽扣电池），通过DC-DC转换器（如TPS62740）将电池电压转换为稳定的工作电压（1.8V/3.0V/3.3V）。

在这四个模块中，MCU是能耗优化的主战场——一个精心配置的低功耗MCU（睡眠模式+唤醒感知）可以比一个"裸跑"的高性能MCU省电1000倍以上。这就是为什么WSN嵌入式工程师必须深入理解MCU的电源管理。

ARM Cortex-M系列在WSN中的应用

ARM Cortex-M系列处理器是WSN MCU市场的主流选择。不同系列的定位差异显著：

Core 架构 主频范围 DMIPS/MHz 典型功耗 WSN定位
Cortex-M0 冯诺依曼 8-48 MHz 0.84 10-80 μA/MHz 极低功耗终端节点
Cortex-M3 哈佛 24-120 MHz 1.25 30-150 μA/MHz 平衡型传感器节点
Cortex-M4 哈佛 24-200 MHz 1.25 (DSP) 50-200 μA/MHz 需信号处理的节点
Cortex-M33 哈佛 4-160 MHz 1.50 30-100 μA/MHz 安全要求高的节点

选型建议：如果你的节点只是定期采集温度/湿度/光照数据并发送（典型的WSN终端），Cortex-M0/M0+是最优选择——功耗最低、成本最低、代码尺寸最小。如果你的节点需要做数据融合、FFT或边缘AI推理（如振动分析、语音关键词检测），Cortex-M4的DSP指令集带来的性能提升值得额外的功耗代价。

主流WSN MCU对比

下面四款芯片是当前WSN项目中最为常见的MCU平台（或SoC，集成RF收发器）：

CC2538（TI）

ARM Cortex-M3 + IEEE 802.15.4 RF收发器集成SoC。支持ZigBee、6LoWPAN和Thread协议栈。32KB RAM + 512KB Flash。在2.4GHz频段提供250kbps数据率。缺点是功耗偏高（工作电流约24mA，睡眠功耗约1.3μA）。适合需要ZigBee协议栈的智能家居和楼宇自动化应用。

CC2650 / CC2652（TI）

继承CC2538的第二代多协议SoC。集成了ARM Cortex-M3（主处理器）+ Cortex-M0（射频控制协处理器）的双核架构。支持ZigBee、BLE、Thread和6LoWPAN多协议。独特优势是双核架构让射频任务从主MCU卸载到专用的射频协处理器上，简化了软件开发。CC2652的睡眠功耗低至0.7μA，工作电流约6mA（接收模式），支持ARM TrustZone安全扩展。

nRF52840（Nordic）

ARM Cortex-M4F（带浮点运算单元）+ BLE 5.3/NFC/Ant RF。256KB RAM + 1MB Flash。是目前功能最强大的WSN MCU之一。支持BLE Mesh、Thread和ZigBee。M4F内核使它在边缘AI推理（如TensorFlow Lite Micro）上有独特优势。缺点是功耗相对较高（工作电流约4.6mA接收），价格偏高。适合需要大量本地处理或复杂通信协议的高端WSN节点。

STM32WLE5（ST）

ARM Cortex-M4 + LoRa RF收发器（Semtech SX126X IP核）。是专为LoRaWAN终端设备设计的单芯片SoC。48KB RAM + 256KB Flash。集成了Sub-1GHz LoRa射频前端，支持470MHz/868MHz/915MHz频段。待机电流低至0.8μA（带RTC唤醒），接收电流约4.8mA。优势是单芯片实现LoRaWAN终端节点，BOM成本极低，开发工具链成熟（STM32Cube生态）。

实时操作系统概述

WSN的嵌入式软件可以在裸机（Bare Metal）或实时操作系统（RTOS）上运行。裸机开发的优点是占用资源最少（RAM/Flash消耗可忽略），缺点是难以处理并发任务——传感器采集、无线通信、电源管理、定时器调度如果全部在一个while(1)循环里处理，代码会迅速变得"意大利面条"化。当WSN节点需要同时处理3个以上任务时，强烈建议使用RTOS。

FreeRTOS

目前生态最成熟的开源RTOS，已被Amazon收购并深度集成了AWS IoT云连接能力。内核代码只有几千行，最小配置下RAM占用可控制在1KB以内。支持任务（Task）、队列（Queue）、信号量（Semaphore）和软件定时器（Software Timer）。Tickless模式是WSN的关键——在空闲时停止系统滴答定时器，让MCU进入深度睡眠，只有在任务需要响应中断或定时器到达时才唤醒。

TinyOS

学术界的传奇操作系统，由UC Berkeley开发，专为WSN设计。使用nesC语言（C的扩展，支持组件化编程）。TinyOS的事件驱动模型极其适合WSN节点的工作模式：绝大多数时间处于"等待事件"状态（睡眠），事件触发（传感器数据就绪、无线帧到达、定时器溢出）时快速处理并继续睡眠。TinyOS在2000年代是WSN研究的事实标准，虽然目前活跃度下降，但它引入了许多至今影响深远的设计思想。

Contiki / Contiki-NG

为资源受限物联网设备设计的开源操作系统。原生支持6LoWPAN、RPL和CoAP——这使得Contiki节点可以直接接入IPv6互联网。核心创新是Protothreads（轻量级线程）——在只使用几个字节RAM的情况下实现类似多线程的编程模型。Contiki-NG是Contiki的活跃继承者，支持CC2538、nRF52840等现代MCU，是WSN开发的一个很好的开源选择。

RIOT OS

相对年轻的WSN操作系统，设计目标是"类Linux的开发体验 + TinyOS的资源效率"。支持C/C++编程，完整的POSIX-like API，原生支持6LoWPAN和BLE。在活跃社区和代码质量方面表现良好。

如何选择RTOS？ 对于商业产品，建议首选FreeRTOS——生态成熟、文档丰富、AWS集成是加分项。对于学术研究或教学，Contiki-NG或RIOT OS的WSN原生特性（内置网络栈、传感器驱动架构）可以大大减少重复造轮子的工作。记住一个原则：不要为一个WSN产品去"发明"一个新的RTOS——世界上已经有足够多优秀的RTOS轮子了。

中断与定时器

在WSN嵌入式开发中，中断（Interrupt）和定时器（Timer）是两个最基本的机制。中断是MCU响应外部/内部事件的硬件机制——中断发生时，CPU暂停当前程序，保存现场，执行中断服务程序（ISR），然后恢复现场。定时器是MCU内部的硬件计数器，可以精确计时并在定时到达时产生中断。

在WSN中，中断和定时器的典型应用场景：

  • 定时器中断唤醒：节点在深度睡眠中，定时器（通常是一个独立的Run-While-Sleep定时器，如RTC或低功耗定时器）计数到达后产生中断，唤醒MCU进行传感器采集或无线发送。
  • 外部中断唤醒：传感器数据就绪（通过GPIO中断通知MCU）、无线帧到达（RF芯片通过IRQ引脚触发MCU中断）、按键或其他外部事件。
  • 看门狗定时器（Watchdog Timer, WDT）：WDT计数溢出时会复位MCU。这是防止节点"死机"的最后防线——如果MCU因某些原因（硬件故障、软件死锁）停止了正常运行，WDT可以自动复位系统。
// FreeRTOS任务创建与低功耗Tickless模式配置

#include "FreeRTOS.h"
#include "task.h"
#include "timers.h"

// 任务1：定时采集温度并发送（使用FreeRTOS软件定时器）
void sensor_task(void *pvParameters) {
    TickType_t last_wake_time = xTaskGetTickCount();
    const TickType_t interval = pdMS_TO_TICKS(60000); // 60秒间隔

    for (;;) {
        // 定时等待——比vTaskDelay更精确
        vTaskDelayUntil(&last_wake_time, interval);

        // 唤醒传感器（给传感器供电）
        gpio_set(SENSOR_POWER_PIN, 1);
        vTaskDelay(pdMS_TO_TICKS(10)); // 等待传感器稳定

        // 读取温度（I²C接口）
        uint16_t raw_temp;
        i2c_read(SHT30_ADDR, SHT30_TEMP_REG, &raw_temp, 2);
        float temperature = sht30_calibrate(raw_temp);

        // 关闭传感器电源
        gpio_set(SENSOR_POWER_PIN, 0);

        // 通过队列将数据发送到无线发送任务
        float *data = pvPortMalloc(sizeof(float));
        if (data) {
            *data = temperature;
            xQueueSend(wireless_queue, &data, 0);
        }
    }
}

// 任务2：无线数据发送
void wireless_task(void *pvParameters) {
    float *data;
    for (;;) {
        if (xQueueReceive(wireless_queue, &data, portMAX_DELAY)) {
            radio_power_up();           // 开启射频
            radio_send((uint8_t*)data, sizeof(float));
            radio_power_down();         // 关闭射频
            vPortFree(data);
        }
    }
}

// Tickless模式配置——实现极低功耗空闲
// 需要在FreeRTOSConfig.h中启用：
// #define configUSE_TICKLESS_IDLE         1
// 并实现以下两个回调函数：

// 进入低功耗前的准备
void vPreSleepProcessing(uint32_t ulExpectedIdleTime) {
    // 预计空闲时间(ticks) → 转换为实际睡眠时间
    uint32_t sleep_ms = ulExpectedIdleTime / configTICK_RATE_HZ * 1000;

    // 配置唤醒定时器（RTC或低功耗定时器）
    lptim_set_period(sleep_ms);
    lptim_start();

    // 进入深度睡眠模式
    // - 关闭所有非必要外设时钟
    // - 保持RAM内容（stop模式）
    // - 配置唤醒源（定时器中断、GPIO中断）
    MCU_EnterStopMode();
}

// 唤醒后的恢复
void vPostSleepProcessing(uint32_t ulExpectedIdleTime) {
    // 恢复系统时钟
    SystemClock_Config();

    // 重新初始化外设
    HAL_Init();

    // 停止低功耗定时器
    lptim_stop();
    // FreeRTOS滴答定时器会自动恢复
}

int main(void) {
    // 硬件初始化
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_I2C1_Init();
    MX_SPI1_Init();

    // 创建任务
    xTaskCreate(sensor_task, "Sensor", 256, NULL, 1, NULL);
    xTaskCreate(wireless_task, "Radio", 256, NULL, 2, NULL);

    // 启动调度器——FreeRTOS接管
    vTaskStartScheduler();

    // 永远不会到达这里
    while (1) {}
}

上面这段代码展示了WSN节点的典型任务结构：传感器采集任务 + 无线发送任务，通过队列通信。注意传感器供电控制的细节：只在采集时开启传感器电源，采集完立即关闭——即使传感器处于"空闲"状态，它也消耗电流。WSN节点优化的一个核心技巧就是：不让任何不需要工作的模块处于通电状态。

低功耗模式详解

WSN节点电池续航是所有设计的终极目标。大多数应用要求节点至少运行6个月到2年。以两节AA电池（约3000mAh）为例，如果节点平均电流消耗超过1mA（3000mAh/1mA ≈ 3000小时≈ 125天），连半年都撑不到。所以节点的平均电流必须控制在几十μA级别。这意味着99.9%以上的时间必须在睡眠中度过。

睡眠模式层级

ARM Cortex-M MCU一般支持3-4级睡眠模式，各级别的功耗和唤醒时间成反比：

模式 典型电流 唤醒时间 保留状态 适用场景
Sleep ~1-5 mA 几μs CPU时钟停，外设运行 短暂空闲（等待中断）
Deep Sleep ~10-100 μA 几μs到几十μs RAM保持，大部分外设时钟停 几秒到几分钟的空闲
Stop / Standby ~1-10 μA 几十到几百μs RAM保持，部分唤醒源有效 分钟级定时采集
Shutdown ~0.1-1 μA 几ms RAM丢失，仅有少数唤醒引脚有效 极低频率上报（小时级）
令人窒息的Bug：GPIO唤醒配置遗漏 这是我亲身经历的一个调试故事。某项目使用STM32WLE5做一个LoRaWAN温湿度传感器，目标续航是2年。在设计文档中，低功耗方案是"Shutdown模式 + RTC定时器唤醒"。但在第一轮原型测试中，节点进入Shutdown后——就再也醒不来了。万用表测电流，工作态6mA→Shutdown后0.9μA——睡眠确实成功了，但就是醒不过来。花了一整天逐行检查代码，发现RTC唤醒中断对应的GPIO引脚复用配置错了——RTC唤醒功能需要将特定的PC13引脚配置为RTC输出功能，代码里配成了普通GPIO输入。在Stop模式下GPIO的浮动输入可能还能被意外唤醒，但Shutdown模式只认"正确的唤醒源配置"。这个问题告诉我们：低功耗模式下所有的唤醒源配置必须在数据手册中逐字逐句核对，光靠"以前项目的经验"移植代码，一个脚注没看就可能让你浪费好几天。

唤醒源配置

能让MCU从深度睡眠中唤醒的来源通常包括：

  • RTC定时器唤醒（RTC Alarm）：最常用的唤醒方式，用于周期性的数据采集。功耗极低（RTC通常使用LSE晶振32.768kHz，功耗仅几μA）。
  • GPIO外部中断：传感器数据就绪引脚上升沿/下降沿触发。适用于事件驱动的采集（如门磁传感器、振动检测）。
  • 比较器唤醒：模拟信号超过阈值时触发。适用于阈值告警场景（如温度超过50℃时唤醒上报）。
  • 无线唤醒（Wake-on-Radio, WOR）：RF收发器在低功耗监听模式下检测到特定前导码后唤醒MCU。这是WSN异步MAC协议（如B-MAC、X-MAC）的硬件基础。

嵌入式基础是WSN从"纸上谈兵"到"真刀真枪"的桥梁。这一章我们拆解了WSN节点的硬件架构，对比了主流MCU方案，了解了RTOS的选择，深入探讨了低功耗设计——尤其是一个BP级的低功耗Bug排查故事。记住：好的WSN嵌入式设计，不是"功能最强的"方案，而是"能耗最优的"方案。
