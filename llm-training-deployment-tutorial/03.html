第3章 Tokenizer原理与实践

大家好，我是你们的讲师。上一章我们讲了数据清洗和配比，到了这一步，你的文本数据已经是干净的、高质量的、精心配比的——但模型还读不懂它。模型只认识数字、只处理固定长度的向量序列。把文本变成模型能吃进去的数字序列，这就是tokenizer的工作。

Tokenizer可能是整个训练链路中最不起眼但也最容易翻车的环节。为什么这么说？因为tokenizer的问题通常不会导致训练失败（loss还会正常下降），但会导致一个隐蔽得多的后果：模型在某些任务上永远达不到预期效果，而你根本想不到问题出在tokenizer上。我见过一个做中文代码生成模型的团队，训练数据和模型架构都没问题，但生成的代码永远缺括号——最后发现是因为英文括号在中文语境下被tokenizer分错了。所以这一章，我们深度拆解tokenizer的原理、实践和常见坑点。

3.1 为什么需要Tokenizer？

简单来说，tokenizer的作用就是将一段文本切分成一系列token（词元），再把每个token映射到一个整数ID。这个ID就是模型Embedding层的输入。

举个具体的例子。对于句子"I love large language models"，一个典型的BPE tokenizer可能会把它切分成["I", " love", " large", " language", " models"]，然后映射成[40, 1222, 539, 2541, 6735]。注意那个空格——BPE tokenizer通常会把空格保留在token内部，这对模型理解词边界很关键。

Tokenizer的选择直接影响模型的几个核心指标：

序列长度：同样的文本，不同的tokenizer会产生不同数量的token。例如，对于中文，"我喜欢大模型"可能被切分成["我", "喜欢", "大", "模型"]（4个token）。但如果tokenizer对中文字符的支持不好，可能每个字就是一个token——也就是5个token，会使有效上下文长度缩水20%。

推理速度：token数量直接决定了transformer的计算量（因为self-attention的复杂度是O(n²)）。一个效率低的tokenizer会让你的推理成本增加30-50%，而你甚至不知道问题出在哪里。

模型的知识表达：tokenizer的分词粒度决定了模型"看到"的信息单元。如果"reinforcement learning"被分成了一个token，模型可以直接学习到这个概念的整体语义。如果被分成了["rein", "forcement", "learning"]，模型就需要先从碎片中拼出概念，效率降低。

3.2 BPE算法深度解析

BPE（Byte Pair Encoding）是目前最主流的tokenization算法，被GPT系列、Llama系列等几乎所有主流模型采用。它的核心思想很巧妙：从字符级开始，逐步合并最频繁出现的相邻token对，直到达到目标词汇表大小。

我们来走一遍完整的BPE训练过程。假设我们有这样一个小小的训练语料：

# 初始语料（字符级拆分）
"low": 5, "lower": 2, "newest": 6, "widest": 3

# 第1步：初始词汇表（所有唯一字符）
l, o, w, e, r, n, s, t, i, d

# 第2步：统计所有相邻token对的频率
("l", "o"): 7, ("o", "w"): 7, ("w", "e"): 8, ("e", "r"): 2,
("e", "w"): 6, ("n", "e"): 6, ("s", "t"): 9, ...

# 第3步：合并最频繁的对 ("e", "s") — 但这里是 ("s","t") 出现9次
# 合并 ("s","t") -> "st"
新词汇表: l, o, w, e, r, n, s, t, i, d, st

# 第4步：重复。现在 ("e", "st") 出现9次，合并 -> "est"
新词汇表增加: est

# 第5步：继续合并，直到词汇表容量达到目标（通常是32k-128k）
# 最终可能学到的token: low, low, er, newest, widest, ...

这个迭代过程的关键洞察是：被合并的token对代表了语料中的常见子词模式。BPE本质上是在做一种基于频率统计的"词干提取"——"est"会被识别出来，因为它出现在"newest"和"widest"中；"low"会被保留为一个完整token，因为它高频出现。

实际训练中，BPE的词汇表大小一般在32,000到128,000之间。词汇表越大，每个token携带的信息量越多（压缩率越高），但Embedding层的参数也越多，推理时softmax的代价也越大。32K词汇表的embedding矩阵大约是(32768 × 4096) ≈ 1.34亿个参数——对于7B模型来说还不算大，但对于部署在手机上的小模型，这个开销就不可忽视了。

3.3 主流Tokenizer对比

不同模型家族使用不同风格的tokenizer，理解它们的差异对于选择合适的tokenizer（或者理解你使用的模型的tokenizer行为）至关重要。

特性 GPT系列 (tiktoken/BPE) Llama系列 (SentencePiece/BPE) BERT (WordPiece)
算法基础 BPE BPE (via SentencePiece) WordPiece
是否预分词 是（GPT-2分词器） 否（原始bytes输入） 是（Moses等工具）
空格处理 保留空格为token前缀 用_代替空格 丢弃空格
未知词处理 退化为byte级别 退化为byte级别 UNK token
典型词汇表大小 100K (gpt-4) 32K (Llama 2/3) 30K (BERT-base)
中文支持 好（单字+多字组合） 好（Byte-based） 差（容易分成单字）
多语言表现 良好 优秀（byte-level输入） 一般

其中需要特别关注的是SentencePiece的"不预分词"设计。传统的BPE（如GPT-2使用的）需要先用规则将文本切分成词（pre-tokenization），然后再对词内进行BPE。这意味着空格等分隔符被当作token边界，"hello"和"hello!"的分词结果可能在词级别完全不同。SentencePiece直接将原始文本（包括空格）作为输入，不做任何预分词，解决了多语言场景中空格边界不一致的问题——中文、日文等不使用空格的语言在预分词阶段往往表现很差，而SentencePiece就避免了这个问题。

选型建议：如果你要训练一个主要面向中文和英文的双语模型，推荐使用SentencePiece + BPE的组合（即Llama方案），词汇表大小设为32K-64K。如果主要是英文场景，GPT的tiktoken方案已经非常成熟。如果你需要支持大量低资源语言（如东南亚语言），千万要选byte-level的tokenizer（如SentencePiece的byte模式），避免产生大量UNK token导致信息丢失。

3.4 Tokenizer设计决策

选定了tokenizer类型之后，你还需要做几个关键设计决策。这些决策会直接影响模型的最终表现。

词汇表大小选择。词汇表太大（>256K）会导致embedding矩阵过大、推理时softmax计算开销剧增，而且很多token在训练中可能只出现几次，学习到的embedding质量很差。词汇表太小（<8K）则意味着每个token的信息量不足，序列太长。经验法则是：在训练数据的压缩率和模型容量之间取平衡。具体做法是：用不同大小的词汇表分别tokenize你的训练语料，观察每个token的平均压缩字节数（bytes per token），通常选取曲线"拐点"附近的词汇表大小。对于中英双语模型，64K左右是常见选择；对于纯英文，100K左右（即GPT-4的方案）更优。

特殊token设计。特殊token（special tokens）是不可忽视的细节。至少需要预留：<pad>（填充）、<bos>/<s>（序列开始）、<eos>/</s>（序列结束）、<unk>（未知词）。对于指令模型，还需要预留<|im_start|>和<|im_end|>来标记对话轮次。注意：一旦模型开始训练，词汇表就固定了，新加特殊token意味着要扩展embedding矩阵并重新训练或至少做embedding初始化，非常麻烦。所以训练前要把会用到的所有特殊token都预留好。

多语言tokenization策略。如果你要做多语言模型，一个常见策略是在训练tokenizer时按语言比例对数据进行上采样，确保低频语言也能产生足够的token。但不是所有语言的字符集都适合放在tokenizer中——比如阿拉伯语和希伯来语的从右到左书写方向、中文和日文的大字符集、韩语的组合式文字，每种都需要不同的处理策略。一个实用的建议是：确保每种目标语言在你的tokenizer训练数据中至少有1GB的纯文本，否则该语言在tokenizer中的覆盖率会很低。

3.5 Tokenizer常见问题

下面这几个问题是实际项目中反复出现的"tokenizer杀手"，每一个都是我或我身边的团队踩过的坑。

Tokenization Tax（分词税）。这是Andrej Karpathy提出的概念：不同词汇的token消耗差异非常大。比如"cat"在GPT-4的tokenizer中是1个token，"catastrophe"则需要3个token（"cat"+"astr"+"ophe"）。这意味着模型处理"catastrophe"需要3倍的计算量，但这个词并没有多3倍的语义信息。更极端的例子是密码学相关内容——一个base64编码的字符串可能需要几十个token来表达非常少的信息。这个问题在代码数据中尤其严重：变量名get_user_by_id可能被分成6-7个token，而g可能只占1个token。

中文分词的"坑"。中文没有空格分隔，对于基于英文设计的tokenizer来说是个大挑战。比如"南京市长江大桥"这个经典歧义句，英文式tokenizer可能会把它分成["南京", "市长", "江", "大桥"]而不是正确的["南京市", "长江大桥"]。更实际的问题：很多中文专有名词（如"强化学习""卷积神经网络"）如果被分成了多个碎片，模型需要花额外的训练步数才能把它们"拼回来"。好的tokenizer应该让这些高频中文术语作为完整token出现。

数字和代码的分词问题。数字是tokenizer的老大难。对于3.1415926535，一个基于BPE的tokenizer可能把它分成["3", ".14", "15", "92", "65", "35"]——完全没有保留数字的数值连续性。这直接导致大模型的数学能力受限：模型知道"3.14"是π的近似值，但当你问"3.14后面的数字是什么"时，它无法从token序列中推断出"15"紧随其后，因为表示层已经碎了。代码数据也有类似问题——缩进（空格和tab的混合）、换行符、Unicode标识符都可能导致奇怪的切分。

# 同一个数学表达式的不同tokenization结果对比
expression = "3.1415926535 * 2 = 6.283185307"

# GPT-4 tokenizer (tiktoken) 示例：
# ["3", ".141", "59", "265", "35", " *", " 2", " =", " 6", ".283", "185", "307"]
# 总共12个token，数字被碎片化

# 一个理想的"数学友好"tokenizer:
# ["3.1415926535", " *", " 2", " =", " 6.283185307"]
# 总共5个token，保留了数值完整性

踩坑故事：我之前参与的一个项目，团队花了几百万训练了一个代码生成模型，结果在生成Python代码时频繁出现IndentationError。排查了两周才发现是tokenizer的问题：我们的tokenizer训练数据里，缩进混用了空格和tab，导致tokenizer把"四个空格"和"一个tab"当成了不同的token来处理。虽然它们在视觉上都产生4个空格的缩进，但token ID不同，模型在学习缩进模式时出现了混淆。最离谱的是，我们后来发现tokenizer还把def和def （带空格）分成了不同的token，导致模型在预测函数定义时，有相当概率选错token。修复方法：在训练tokenizer之前，统一把代码数据中的tab转成4个空格。

关于Tokenizer的黄金法则：训练tokenizer前，一定要用你的目标数据做一次"分词预览"——随机抽取1000条文本，用训练好的tokenizer切分，仔细看有没有奇怪的切分情况。特别是中文专有名词、数学表达式、代码关键字和特殊符号。如果发现异常，回头调整tokenizer训练参数或数据预处理方式。这个步骤不到30分钟，但可以避免后续数月的痛苦调试。
