5.1 数据质量决定模型上限

有句老话叫"Garbage In, Garbage Out"。我在一家电商公司做用户流失预测的时候，花了两周调模型——XGBoost、LightGBM、甚至上了神经网络，AUC始终在0.62左右徘徊。后来有一天我突然决定认真做一遍数据审计，结果发现：20%的用户标签是错的（把"活跃"用户也标成了"流失"）。清洗之后，一个简单的逻辑回归就做到了0.78。这件事教会我：没人会因为模型不够复杂而失败，但很多人因为数据质量问题而失败。

5.2 缺失值处理：不是简单的dropna

真实世界的数据永远有缺失值。怎么处理？有三种思路：

策略做法适用场景风险
删除删除有缺失的行/列缺失比例 < 5%丢失信息，可能引入偏差
简单填充均值/中位数/众数填充数据MCAR（完全随机缺失）低估方差，扭曲分布
模型填充用KNN/回归/MICE预测缺失值缺失有规律可循过拟合，计算成本高
保留缺失信号添加is_missing指示列缺失本身包含信息增加特征维度
💡 一个被低估的技巧：缺失值本身可能是一个重要特征。比如在医疗数据中，"患者拒绝做某项检查"导致的数据缺失，本身可能反映了患者的经济状况或健康态度——这些都是预测的重要信号。所以在处理缺失值之前，先问自己："这个值为什么缺失？" 
import pandas as pd
import numpy as np
from sklearn.impute import SimpleImputer, KNNImputer

df = pd.read_csv('your_data.csv')

# 第一步：了解缺失情况
missing_ratio = df.isnull().sum() / len(df)
print(f"缺失超过30%的列:\n{missing_ratio[missing_ratio > 0.3]}")

# 第二步：对数值列，保留缺失信号 + 中位数填充
for col in df.select_dtypes(include=np.number).columns:
    if df[col].isnull().any():
        df[f'{col}_was_missing'] = df[col].isnull().astype(int)
df.fillna(df.median(numeric_only=True), inplace=True)

# 第三步：对类别列，用"未知"填充
for col in df.select_dtypes(include='object').columns:
    df[col] = df[col].fillna('未知')

5.3 异常值检测与处理

异常值（Outlier）会严重扭曲线性模型和基于距离的模型（如KNN、K-Means）。常用的检测方法：

  • IQR法：Q1 - 1.5×IQR 到 Q3 + 1.5×IQR 之外的值。适用于近似正态分布的特征
  • Z-score法：|z| > 3 视为异常。也假设正态分布
  • Isolation Forest：无需分布假设，专为异常检测设计，ML方法
⚠️ 谨慎处理异常值！不是所有"异常"都应该删除。在欺诈检测中，"异常"就是你要找的目标！在房价预测中，豪宅确实罕见但它是真实数据。永远先理解数据的业务含义，再决定怎么处理统计上的异常。

5.4 EDA：探索性数据分析的黄金流程

拿到数据后，不要直接建模。先花1-2小时认真做EDA。我的标准流程：

  1. 概览：df.info(), df.describe(), df.shape —— 了解数据规模、类型、内存占用
  2. 单变量分析：每个特征的分布（直方图、箱线图），看看有没有偏态、多峰
  3. 双变量分析：特征与目标的关系（散点图、分组箱线图、相关性热力图）
  4. 多变量交互：特征之间的相关性、VIF多重共线性检测
  5. 目标变量：分类问题看类别平衡，回归问题看分布形态

5.5 特征编码与标准化

模型只能吃数字，但你的数据可能有文本、类别、日期。编码是连接原始数据和模型的桥梁。

数据类型推荐编码方式适用模型
类别变量（无序，类别少）One-Hot Encoding线性模型、神经网络
类别变量（无序，类别多）Target Encoding, Frequency Encoding树模型、GBDT
有序类别Ordinal Encoding（Label Encoding）树模型
文本TF-IDF, Word EmbeddingNLP模型
数值（尺度差异大）StandardScaler, MinMaxScaler线性模型、SVM、NN
🔑 关键区别：树模型（决策树、随机森林、GBDT）不需要特征缩放！它们根据阈值划分，对特征尺度不敏感。但线性模型、SVM、KNN、神经网络必须缩放，否则大尺度特征会主导梯度更新。很多初学者把所有特征一股脑扔进StandardScaler再喂给XGBoost——虽然不会报错，但完全多此一举。 
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.compose import ColumnTransformer

# 区分数值列和类别列
num_cols = ['age', 'income', 'tenure']
cat_cols = ['gender', 'city', 'product_type']

preprocessor = ColumnTransformer([
    ('num', StandardScaler(), num_cols),
    ('cat', OneHotEncoder(handle_unknown='ignore', max_categories=10), cat_cols),
])

X_processed = preprocessor.fit_transform(X)

5.6 数据泄露：隐形的杀手

数据泄露（Data Leakage）是指训练数据中无意包含了关于目标的信息，这些信息在真实预测时是不可用的。这是ML项目中最隐蔽、最危险的错误——你的验证集效果完美，但上线后一塌糊涂。

常见泄露场景：

  • 在划分训练/测试集之前做了标准化（fit了整个数据集）
  • 用未来的数据预测过去（时序问题中特别常见）
  • 特征里包含了目标的衍生信息（比如用"是否有过退款"预测"用户是否流失"）
  • 在训练集和测试集中有重复的行或同一实体的多条记录
⚠️ 黄金法则：在任何数据变换（标准化、填充、编码）中，先在训练集上 fit()，再用同一个对象对测试集 transform()。永远不要对测试集单独fit。使用 sklearn.pipeline.Pipeline 可以很大程度上避免这个问题。
