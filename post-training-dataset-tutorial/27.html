第27章 数据策略设计与评估体系

大家好，我是你们的讲师。通过前26章的学习，我们已经走过了从基础概念到前沿趋势的完整知识旅程。你知道了所有主流的后训练数据集，掌握了数据管线的搭建方法，也了解了2025-2026年的最新范式演进。

但有一个问题始终没有被正面回答——假如明天你接到了任务："为我们的新模型设计后训练数据策略"，你应该从哪里开始？

这就是第27章要解决的核心问题。这一章不讲具体的数据集细节（前面已经讲透了），也不讨论某个孤立的技术点——而是带你走一遍从场景诊断 → 数据集组合 → 评估体系 → 迭代优化的完整策略设计流程。

没有银弹——不同场景需要不同的数据策略

在开始之前，我需要先卸掉你身上一个潜在的包袱：不要寻找万能的数据配方。

如果有人说"我们的数据策略在任何模型上都有效"，请保持怀疑。后训练数据策略天生就是场景相关的。一个通用对话助手和一个代码编程助手的后训练数据需求，就像健身教练和围棋教练的训练方案一样——都是"教练"但训练内容完全不一样。

为什么？因为不同的模型应用场景对能力维度的权重分配完全不同：

能力维度 通用对话 代码助手 推理专家 Agent 安全优先
指令遵循 ★★★★★ ★★★★ ★★★★ ★★★★★ ★★★★
多轮对话 ★★★★★ ★★★★ ★★★ ★★★★ ★★★
事实准确性 ★★★★ ★★★★★ ★★★★★ ★★★★ ★★★★★
推理深度 ★★★ ★★★★ ★★★★★ ★★★ ★★
工具调用 ★★ ★★★★ ★★ ★★★★★ ★★
安全/拒绝 ★★★★ ★★★ ★★ ★★★★ ★★★★★
创造力 ★★★★ ★★★ ★★ ★★ ★★

看到这张表，你就明白了：给一个"推理专家"模型灌入大量多轮闲聊数据，纯属浪费；给"安全优先"模型加入大量创造性写作数据，可能适得其反。数据策略的第一步，永远是明确你的模型要服务什么场景。

场景诊断框架

那么，如何系统地完成场景诊断呢？我推荐用下面的四个问题来引导：

  1. 用户是谁？ → 开发者？普通消费者？企业用户？学生？不同的用户群体对模型能力的期待完全不同。
  2. 核心使用场景是什么？ → 编程辅助？文档撰写？数学解题？客服对话？还是Agent自主操作？
  3. 关键成功指标（KSF）是什么？ → 回答准确率？用户留存率？任务完成率？还是安全违规率？
  4. 有什么约束条件？ → 基座模型能力上限如何？训练预算多少？数据合规要求多严格？

做完这四个问题的诊断，你应该对"我们的模型需要什么样的后训练数据"有了一个清晰的画像。接下来就可以进入具体的数据集组合设计了。

数据集组合方案

基于上面的场景诊断框架，我为你整理了四大典型场景的精确数据集组合与配比建议。这些方案融合了2025-2026年多个顶级团队公开的技术报告中的最佳实践。

场景1：通用对话模型

这是最经典的场景——像 ChatGPT 一样的通用对话助手，什么都得会一点，但不需要在某个领域极其深入。

【SFT 数据组合】       总量: 50K-100K
  OpenHermes-2.5:      30%  ← 高质量通用指令
  UltraChat-200K:      25%  ← 多轮对话能力
  ShareGPT:            20%  ← 真实用户对话分布
  LIMA:                 5%  ← 高质量种子数据
  SlimOrca:            10%  ← 多样化指令
  Dolly:               10%  ← 任务多样性

【偏好对齐数据】      总量: 20K-50K
  UltraFeedback:       50%  ← 多维度偏好信号
  Chatbot Arena:       30%  ← 真实用户偏好
  HH-RLHF:            20%  ← 帮助性和无害性

场景2：中文模型

面向中国市场的通用对话模型，重点是高质量中文指令和中文偏好对齐。

【SFT 数据组合】       总量: 80K-120K
  Firefly:             30%  ← 中文高质量指令
  BELLE:               20%  ← 中文指令多样性
  UltraChat-zh:        20%  ← 中文多轮对话
  OpenHermes-2.5:      15%  ← 英文能力保底
  MuSiQue-zh:          10%  ← 中文多跳推理
  COIG:                 5%  ← 中文对齐指令

【偏好对齐数据】      总量: 30K-50K
  BeaverTails:         40%  ← 中文安全偏好
  COIG-CQIA:           30%  ← 中文质量偏好
  UltraFeedback:       30%  ← 跨语言偏好信号
中文模型的一个常见陷阱：很多团队在做中文模型时，为了追求"国产化"全部使用中文数据进行后训练。这往往会导致模型的英文能力大幅退化——因为人话（自然语言）层面的知识很大部分是跨语言的。建议始终保留至少 15%-20% 的英文高质量数据，哪怕你的目标用户全是中文用户。

场景3：推理增强

需要深度推理能力的模型——数学、代码、逻辑、科学等场景。

【SFT 数据组合】       总量: 60K-100K
  OpenOrca:            25%  ← 高质量推理指令
  GSM8K + MATH:        20%  ← 数学推理基础
  DeepSeekMath:        15%  ← 数学推理扩展
  MetaMathQA:          10%  ← 数学推理增强
  CodeAlpaca:          15%  ← 代码推理
  NuminaMath-CoT:      15%  ← 多样化数学推理

【RLVR 数据】          总量: 可扩展（合成生成）
  Code验证数据:        40%  ← 单元测试通过/不通过
  Math验证数据:        40%  ← 标准答案比对
  Logic验证数据:       20%  ← 逻辑推导验证
推理增强的关键：可验证奖励信号（RLVR）比偏好对齐更适合推理场景。原因很简单——对于"1+1=?"，标准答案是确定的，你不需要人类偏好来告诉你哪个回答更好。所以推理增强的数据策略核心是：构建自动化验证器，而不是收集人类偏好。

场景4：安全优先

对安全性要求极高的场景——如儿童教育、医疗咨询、金融助手等。

【SFT 数据组合】       总量: 40K-80K
  TruthfulQA:          20%  ← 事实性安全
  安全SFT数据:         30%  ← 拒绝回答 + 安全引导
  OpenHermes-2.5:      20%  ← 通用能力保底
  HH-RLHF(SFT部分):    15%  ← 帮助性基础
  BeaverTails(SFT):    15%  ← 中文安全指令

【偏好对齐数据】      总量: 30K-60K
  HH-RLHF:             35%  ← 帮助性+无害性
  BeaverTails:         30%  ← 中文安全偏好
  Safety-Prompts:      20%  ← 红队测试偏好
  Anthropic-HH:        15%  ← 高质量安全偏好

评估体系设计

有了数据策略之后，你面临的下一个问题是——怎么知道这个策略是有效的？

一个完善的后训练数据评估体系应该分为三个层级：

第一层：自动化评估

这是评估体系的"第一道防线"，特点是速度快、成本低、可重复。主要包括：

  • AlpacaEval 2.0：自动评估指令遵循能力，使用 GPT-4 作为评估器，输出 LC win rate。优点是标准化程度高，缺点是仅覆盖通用对话场景。
  • MT-Bench：多轮对话评测，涵盖写作、角色扮演、推理、数学、代码等8个维度。每个维度评1-10分。
  • Arena-Hard：2025年推出的更具挑战性的自动评测基准，包含500个高难度prompt，在区分模型能力上比 AlpacaEval 更有区分度。
  • 领域专项评测：如 HumanEval（代码）、MATH/GSM8K（数学）、MMLU（综合知识）等。

第二层：人工评估

自动化评估覆盖不了所有维度——特别是创造力、语气、安全边界这些主观性强的方面。人工评估在这些维度上不可替代：

  • A/B 盲测：让评估者在不知道模型身份的情况下，对比两个模型的输出，选出更好的那个。Chatbot Arena 就是这种模式的代表。
  • 维度评分：对每个回答按多个维度（准确性、有用性、安全性、流畅性）分别打分。
  • 错误标注：让标注者指出模型回答中的具体错误类型（事实错误、逻辑矛盾、不安全内容等）。

第三层：红队测试

对于安全要求高的场景，需要专门的红队测试团队进行主动攻击：

  • 对抗性测试：设计prompt尝试绕过模型的安全防护。
  • 边界探索：测试模型在敏感话题、偏见、刻板印象等方面的表现。
  • 多语言安全：某些模型在英语上安全但中文上不安全，需要跨语言测试。
层级 速度 成本 覆盖维度 典型工具/方法
自动化评估 分钟级 低 指令遵循、知识、推理 AlpacaEval, MT-Bench, Arena-Hard
人工评估 天级 中 创造力、语气、安全性 A/B盲测、维度评分
红队测试 周级 高 安全边界、对抗鲁棒性 对抗性prompt、专业红队
评估体系建设的优先级建议：如果资源有限，优先建立自动化评估——选2-3个与你的场景最匹配的基准（例如通用对话选 AlpacaEval + MT-Bench，代码助手选 HumanEval + MBPP）。人工评估可以在模型能力基本达标后引入。红队测试建议在模型上线前1-2周集中进行。

迭代优化循环

评估不是终点，而是迭代优化的起点。一个完整的数据策略迭代循环包括以下步骤：

┌────────────────────────────────────────────┐
│           数据策略迭代循环                   │
├────────────────────────────────────────────┤
│                                            │
│  [评估发现问题]                              │
│    ↓ 如：AlpacaEval win rate 低于预期        │
│  [定位数据短板]                              │
│    ↓ 如：多轮对话维度得分低                   │
│  [定向补充数据]                              │
│    ↓ 如：增加 UltraChat 混合比到 30%         │
│  [重新训练]                                 │
│    ↓ 在同样条件下做一次小规模训练              │
│  [再次评估]                                 │
│    ↓ 看多轮对话维度是否改善，其他维度是否下降    │
│  [决定是否上线]                              │
│    ↓ 如果满意 → 推全量训练；不满意 → 继续迭代   │
│                                            │
└────────────────────────────────────────────┘

这个循环中有几个需要特别注意的地方：

  • 一次只改一个变量：迭代时不要同时调整多个数据来源的比例，否则你无法确定效果变化是由哪个变量引起的。
  • 关注遗忘效应：增加某类数据后，可能提升了目标能力但导致其他能力退化。每次迭代都要做全维度评估，不要只看目标维度的分数。
  • 用小规模实验加速：不需要每次迭代都做全量训练。用 10%-20% 的数据量做小模型（7B）的快速实验，收敛后再在目标模型上验证。
一个小技巧：建立"数据实验笔记本"——每次迭代都记录以下信息：实验编号、修改的数据字段（类型+比例）、训练配置（学习率+epoch数）、全维度评估结果、以及在不在的发现。积累10-20次迭代后，你就能从历史数据中发现规律，比如"每次增加推理数据超过15%都会导致创造力下降"，从而指导未来的策略决策。
