第9章 预训练数据工程

熟悉机器学习的朋友都知道一句老话：数据和特征决定了机器学习的上限，模型和算法只是逼近这个上限。在大模型时代，这句话的分量被放大了千百倍。LLaMA 1用1T tokens就训练出了超越GPT-3 175B的模型——这个反直觉的结果证明了一个道理：高质量的数据在量级上的"劣势"，可以通过精心的数据工程逆转。

这一章我们来拆解LLaMA预训练数据工程的完整体系。从数据来源到清洗流水线，从配比策略到质量评估，每一环都深刻影响最终模型的能力边界。相信我，读完之后，你再也不会觉得"大模型训练就是堆GPU"了。

本章的核心论点： 数据工程不是"洗数据"那么简单——它是大模型训练中投入产出比最高的环节。Meta在LLaMA 1论文中明确写道："一个精心筛选和配比的数据集，比简单的数据量堆砌能带来更多的模型能力提升。"这也是为什么LLaMA 1的13B模型能超越GPT-3 175B的底层原因之一。

9.1 LLaMA 1的数据来源全景

LLaMA 1的训练数据来自7个不同的来源，总计约1.4T tokens（去重后约1.0T）。每个来源都有其独特的价值和角色。来看这张完整的构成表：

数据来源 占比 token数量 采样次数 主要用途
CommonCrawl 67.0% ~940B 1.10 通用知识、语言建模
C4（Colossal Clean Crawled Corpus） 15.0% ~210B 1.06 高质量网页文本
Wikipedia 4.5% ~63B 2.45 事实知识、结构化信息
Books（Gutenberg + Books3） 4.5% ~63B 2.23 长文本建模、叙事能力
GitHub 4.5% ~63B 1.17 代码能力、逻辑推理
arXiv 2.5% ~35B 2.24 科学知识、数学推理
StackExchange 2.0% ~28B 2.10 问答能力、技术讨论

这张表最有意思的是"采样次数"这一列。它表示每个数据源中的内容在训练中被重复使用的平均次数。Wikipedia、Books和arXiv的采样次数约为2.2-2.5——也就是说这些高质量来源的数据被"多看了两遍多"。而CommonCrawl和C4的采样次数接近1——基本只看一遍就不重复了。

这种数据重采样策略是LLaMA数据工程的一个关键设计。高质量的数据（如Wikipedia、书籍）虽然量少但信息密度高，值得反复学习。低质量的数据（如CommonCrawl）量很大但噪声多，看一遍就够了。这个直觉后来被Chinchilla论文的形式化分析所证实：在固定算力预算下，对高质量数据进行上采样（upsampling）比使用更多低质量数据更有效。

数据配比的消融实验： LLaMA团队在论文中提到，他们对数据配比做了大量的消融实验。一个重要的发现是：如果只使用CommonCrawl训练，模型的知识面会严重偏向网页常见话题，对长尾知识（如罕见的历史事件、冷门的科学概念）的掌握很差。添加Wikipedia、Books和arXiv后，模型对事实性知识的准确率提升了约15%。而加入GitHub和StackExchange后，模型的代码生成和逻辑推理能力有明显的跃升。

9.2 数据清洗流水线

原始数据直接用于训练是不可行的。CommonCrawl的一个月dump就有数十TB的原始网页HTML，里面充斥着广告、导航栏、重复内容、乱码文本、PII（个人身份信息）等垃圾。LLaMA团队构建了一套多阶段的数据清洗流水线：

第一阶段：语言识别与过滤

使用fastText训练的语言分类器对每个文档进行语言判断，只保留以目标语言（主要是英语）为主的文档。这一阶段过滤掉了约30%的非目标语言网页。注意LLaMA 1主要面向英语，非英语数据被大量过滤——这也是LLaMA 1中文能力薄弱的主要原因之一。

第二阶段：质量过滤

这一步是数据工程最核心的部分。LLaMA团队训练了一个基于fastText的二分类器，用于评估文档的"质量"。训练正例来自Wikipedia、Books等高质量来源，负例来自被人工标注为低质量的网页。这个分类器会为每个文档打一个质量分数，低于阈值（论文中没有公开具体数值）的文档被丢弃。受训于Wikipedia的分类器会偏好那些语言规范、信息密集的文本——这正是LLaMA需要的。

# 使用fastText进行质量过滤的伪代码
import fasttext

# 1. 训练质量分类器
# 正例: Wikipedia + Books
# 负例: 低质量网页样本（广告、垃圾内容）
model = fasttext.train_supervised(
    input="quality_data.txt",
    lr=0.1,
    epoch=25,
    wordNgrams=2,

# 2. 对每个文档分类

    quality_score = scores[0] if labels[0] == "__label__high" else 1 - scores[0]

第三阶段：去重

网页数据中存在大量的重复内容——同一篇文章被多个网站转载，不同页面包含相同的段落。如果不做去重，模型会在重复数据上浪费大量的学习能力，甚至产生记忆偏差。LLaMA使用了两种去重策略：

  • 文档级去重（MinHash + LSH）： 对整个文档计算MinHash签名，通过LSH（局部敏感哈希）找到近似重复的文档对，只保留其中一份。这是粗粒度的去重，主要消除完全重复或高度相似的文档。
  • 行级去重（Substring Deduplication）： 在文档内部，删除重复的行或段落。例如一个网页中重复出现的导航栏、页脚等。这比文档级去重更精细，能捕捉到部分重复的内容。
踩坑实录： 我们团队在复现LLaMA风格的数据清洗时，遇到了一个典型问题——MinHash的阈值调参。最开始我们用了非常激进的去重参数（阈值设为0.5），结果发现大量有用的文档被当成重复删掉了。比如两篇关于"注意力机制"的技术博客，一篇从数学角度讲，一篇从工程角度讲，它们虽然有很多共同的术语（"Query"、"Key"、"Value"、"softmax"等），但内容完全不同。经过调试，我们把阈值设为0.7，保留了这类"主题相同但角度不同"的文档。这个经验告诉我们：去重阈值不能一刀切，需要在自己的数据上做标注实验来确定最优值。

第四阶段：PII移除

使用基于规则的方法（正则表达式）移除明显的个人身份信息：邮箱地址、IP地址、身份证号、电话号码等。这一步不仅是法律合规要求，也能防止模型在生成时"记忆"并泄露这些信息。LLaMA的实现相对保守——只移除明显匹配的模式，而不是尝试识别所有可能的PII。

一个值得深思的数据点： 从200TB的原始CommonCrawl数据到5TB的干净文本——97.5%的数据被丢弃。这意味着你喂给模型的每100个token中，只有2.5个token最终留在了训练集里。这个残酷的筛选比例说明了大模型训练中"数据质量优先"的哲学：宁可饿着，也不吃垃圾食品。

9.3 LLaMA 2的数据改进

LLaMA 2在数据工程上做了三个主要改进：

第一：数据量翻倍。 从LLaMA 1的1.0T tokens增加到2.0T tokens。增加的1T tokens主要来自更新的CommonCrawl dump和更多的代码数据。注意，这不仅仅是"多抓了一倍的网页"，背后对应的数据清洗工作量和计算成本也是翻倍的。

第二：更严格的去重。 LLaMA 2升级了去重流水线，加入了n-gram级别的精确去重（而非仅仅MinHash近似去重）。具体来说，他们使用后缀数组（suffix array）在整个数据集中找出重复的50-token子串并移除。这种精确去重的计算量很大——构建全局后缀数组需要对所有tokenized数据做一次全量扫描——但去重效果更好。实验表明，这种精确去重让模型在多个benchmark上的指标提升了1-2%。

第三：数据配比的优化。 LLaMA 2略微调整了数据配比，增加了高质量来源（Wikipedia、Books）的采样权重，减少了CommonCrawl的相对占比。这个调整基于LLaMA 1训练后的消融实验——团队发现高质量数据的边际收益递减速度远慢于低质量数据。

9.4 LLaMA 3的数据革命

LLaMA 3在数据工程上的投入可以用"惊人"来形容。从LLaMA 2的2T tokens到LLaMA 3的15.6T tokens，数据量增长了近8倍。但这不仅仅是数量的增长——数据工程方法论上也有了根本性的变化。

维度 LLaMA 1 LLaMA 2 LLaMA 3
总token数 ~1.0T ~2.0T ~15.6T
主要数据源数 7个 8个（增加更多代码） 10+个（增加多语言、合成数据）
数据筛选方法 fastText分类器 fastText + n-gram去重 多级质量过滤 + 自监督筛选
代码数据占比 ~4.5%（仅GitHub） ~8%（GitHub + 更多代码） ~25%（GitHub + Stack + 合成代码）
多语言数据 极少（以英语为主） 少量（英语为主） ~30%（50+语言）
合成数据 无 无 ~5%（代码解释、推理增强）

LLaMA 3数据工程的最大创新在于两个方面：

一是"自监督数据筛选"。 LLaMA团队发现，单纯依赖fastText分类器的质量分数并不足够——分类器本身也有噪声。于是他们提出了一个自举方法：先用现有模型（LLaMA 2）对候选文档计算perplexity，保留那些perplexity较低（即模型拟合较好）的文档。这个直觉是：如果一个大语言模型已经能很好预测一个文档的内容，说明这个文档的质量和风格符合它的训练分布。通过这种方式，LLaMA 3从海量网页中筛选出了更高质的子集。

二是大规模的合成数据。 LLaMA 3的代码数据中有一部分是"代码解释数据"——用LLaMA 2模型为现有代码片段生成逐行注释和解释。这就像一位老师傅一边写代码一边给你讲解为什么要这么写。这种数据对于培养模型的代码理解和推理能力特别有效。LLaMA 3的论文中报告，加入合成代码解释数据后，HumanEval（代码生成）的pass@1指标提升了约5%。

一个有趣的发现： LLaMA 3论文中披露了一个反直觉的结果——数据质量筛选并不是"越严格越好"。当质量筛选阈值过高时（即只保留最高质量的数据），模型的知识广度和多样性反而下降了。原因很简单：高度统一的高质量数据风格会导致模型的输出风格单一化，缺乏对不同类型文本的适应能力。最终LLaMA 3选择了中等偏严的筛选阈值，在质量和多样性之间取得了平衡。

9.5 数据质量 vs 数量的核心权衡

LLaMA系列的演进历程，本质上也是一部"数据质量与数量"的权衡史。让我们把它用一个框架来理解：

LLaMA 1的发现： 多不一定好。在固定算力下，用较少但更高质量的数据训练较小的模型，效果优于用大量低质数据训练更大的模型。这是数据质量 > 模型规模的实证。

LLaMA 3的发现： 在数据经过充分筛选和配比优化后，"量大管饱"仍然有效。15.6T tokens让405B模型获得了质的飞跃——但如果这15.6T tokens没有经过严格筛选，效果会大打折扣。质量和数量不是互斥的，而是层级关系——先有质量，再谈数量。

一个实用的建议： 如果你在有限预算下训练自己的大模型，下面是数据工程的优先级排序：
1. 清理去重（消除噪声和重复）——这是投入产出比最高的步骤
2. 质量分类（过滤低质内容）——需要一定的标注成本
3. 配比优化（调整数据源权重）——可以通过小规模消融实验完成
4. 合成数据增强（生成高质量补充数据）——大规模训练时才有显著收益

9.6 文档级去重 vs 行级去重

去重是数据工程中最容易被忽视但又影响极大的环节。我们可以把去重分为两个粒度来看：

文档级去重影响的是数据集的"广度"。如果同一个网页被抓取了多次，或同一篇文章被多个网站发布，文档级去重确保我们只保留一份。这防止了模型对特定文档的过度记忆——没有去重的模型在生成时可能会"背诵"训练集中的大段文字，这在伦理和法律上都是有风险的。

行级去重影响的是数据集的"精度"。网页中常见的复制粘贴内容（如版权声明、导航栏、评论区标准回复）会反复出现在不同文档中。行级去重能移除这些"模板噪声"，让模型更专注于文档的核心内容。LLaMA 2引入的50-token后缀数组去重就是行级去重的一种实现。

两种去重的效果差异如何？LLaMA 3团队报告：

  • 仅做文档级去重：模型在测试集上的困惑度降低约5%
  • 文档级+行级去重：困惑度额外降低3%
  • 行级去重在减少模型"背诵"现象上效果显著——模型从原始训练集中逐字生成文本的概率降低了约40%
踩坑实录： 在一次小规模实验中，我们尝试复现LLaMA 2的去重流程。用suffix array做50-token级别的精确去重时，我们遇到了一个意料之外的问题——内存爆炸。我们的数据集虽然只有50GB，但构建全局后缀数组时需要把所有tokenized数据加载到内存中，需要的内存是原始数据大小的5-10倍。在256GB内存的机器上跑了半天后OOM崩溃了。后来我们换成了分治策略：先把数据集分成100个chunk，在每个chunk内部分别做去重，然后再做chunk间的去重。这个方案虽然去重效果略差（无法跨chunk去重），但把内存需求降到了可接受的水平。去重的工程实现需要根据你的数据规模来决定策略，不能盲目照搬论文中的方法。

数据工程是一个永无止境的优化过程。Meta在LLaMA 3的论文中专门用了一整节来讲数据，而且写作团队特意强调："我们花在数据上的时间比花在模型架构上的时间多得多。"这句话值得我们每一个大模型从业者深思。
