第4章 Verilog HDL语法精要（下）

大家好，我是你们的讲师。上一章我们讲了模块、wire/reg、assign 和运算符。这些都是"静态"的语法要素——输入一变，输出跟着变。但数字电路的世界里还有"动态"的一面：时序逻辑。要描述时序行为，我们需要always 块和赋值方式的正确组合。

这一章的内容是 Verilog 学习的分水岭——掌握阻塞/非阻塞赋值的本质区别、理解 always 块的综合行为、避免意外产生锁存器，是区分"会写 Verilog"和"真正懂 Verilog"的标志。

4.1 always 块的两张面孔

always 块是 Verilog 中最强大的构造，但它有两种综合表现，必须区分清楚：

4.1.1 always @(*) —— 组合逻辑

敏感列表写 @(*)（Verilog-2001 自动敏感列表），表示"任意输入变化时执行块内语句一次"。综合结果：纯组合逻辑，映射到 LUT。

// 组合逻辑：always @(*)
always @(*) begin
    if (sel)
        y = b;
    else
        y = a;
end

综合成 2:1 MUX（一个 2-LUT）。与 assign y = sel ? b : a; 完全等价。

4.1.2 always @(posedge clk) —— 时序逻辑

敏感列表写时钟边沿，表示"在时钟沿触发"。综合结果：D 触发器，映射到 FF。

// 时序逻辑：always @(posedge clk)
always @(posedge clk) begin
    if (rst_n)
        q <= 1'b0;
    else
        q <= d;
end

综合成一个 D 触发器（含异步复位）。敏感列表中的 negedge rst_n 或 negedge rst 用于异步复位——如果用了异步复位，必须在敏感列表中列出。

致命错误——别写多时钟沿：有些初学者写出 always @(posedge clk or negedge clk)——这代表"时钟上升沿和下降沿都触发"。FPGA 中几乎没有 FF 支持双沿触发（除了少数特殊 I/O 寄存器），这种代码通常是不可综合的，或者综合出意想不到的电路。如果你确实需要双沿采样，使用 IDDR 或 ODDR 原语。

4.2 阻塞赋值 (=) vs 非阻塞赋值 (<=)

这是我们这一章最重要的知识点。甚至可以这样说：90% 的 Verilog 初学者 bug 都源于用错了赋值方式。

4.2.1 区别的本质

  • 阻塞赋值 =：语句按书写顺序执行。下一条语句执行时，上一条的赋值已经"生效"。可以理解为"立即生效"。
  • 非阻塞赋值 <=：所有语句同时"采样"等号右边的值，在过程块结束时一起更新左边的变量。可以理解为"并行生效"。

4.2.2 仿真时间步分析

理解阻塞/非阻塞的关键是仿真器的 delta 延迟机制：

特性阻塞赋值 =非阻塞赋值 <=
执行时机遇到即执行当前时间步末尾统一更新
右值采样在当前语句执行时在进入过程块时（所有右值先采好）
左值更新当前语句执行完立即更新当前时间步所有语句执行完后更新
顺序依赖性依赖书写顺序不依赖书写顺序
Verilog 中的用途组合逻辑（always @(*)）时序逻辑（always @(posedge clk)）
综合后电路LUT + 布线（组合）FF + LUT（时序）
硬件对应数据流通过 LUTFF 在时钟沿采样

4.2.3 同一个电路，不同的赋值方式

下面这个例子展示了用阻塞和非阻塞赋值实现同一个移位寄存器，结果完全不同：

// 错误：使用阻塞赋值实现移位寄存器
// 仿真结果：a = b = c = d（所有值相同，没有移位效果）
always @(posedge clk) begin
    a = data_in;    // a 立刻更新
    b = a;          // b = 新的 a（不是原来的 a）
    c = b;          // c = 新的 b
    d = c;          // d = 新的 c
end

// 正确：使用非阻塞赋值实现移位寄存器
// 仿真结果：a->b->c->d 逐拍移位
always @(posedge clk) begin
    a <= data_in;   // 采样 data_in
    b <= a;         // 采样旧的 a
    c <= b;         // 采样旧的 b
    d <= c;         // 采样旧的 c
end

阻塞赋值版本中，b = a 读到的是 a 被 data_in 更新后的新值——所有寄存器都在同一个时钟沿赋值为 data_in，变成了一个"直通"的寄存器，而不是移位寄存器。这就是仿真与综合结果不一致的经典案例——阻塞赋值版本在仿真中看起来是"所有位同时更新"，但综合工具仍可能综合出一个正确的移位寄存器（行为与书写顺序无关），导致仿真与硬件行为不一致。

黄金法则：时序逻辑（always @(posedge clk)）中永远用非阻塞赋值 <=。组合逻辑（assign 或 always @(*)）中永远用阻塞赋值 =。这个规则没有例外。所有工业级代码规范（如 OOC 编码规范、NASA 的 JPL 编码标准）都强制这一规则。

4.3 if-else 与 case 的综合行为

if-else 和 case 在仿真中的行为很容易理解——条件分支。但综合器会把它们映射成什么样的电路？如果不了解这一点，就可能在不知不觉中生成一堆意料之外的锁存器或优先级编码器。

4.3.1 if-else 综合：优先级编码器

if-else if-else 综合出优先级编码器——先判断的条件优先级高。在硬件中，这对应一个级联的 MUX 链：

// 综合成优先级编码器（sel0 优先级最高）
always @(*) begin
    if (sel0)      y = data0;
    else if (sel1) y = data1;
    else if (sel2) y = data2;
    else           y = data3;
end

这个电路的关键路径延迟与优先级级联的深度成正比。如果条件很多（超过 8-10 层），建议改用 case 生成并行 MUX，以降低延迟。

4.3.2 case 综合：并行 MUX

case 综合出并行 MUX——所有分支的优先级相同，综合器会尝试优化为平衡的选择树：

// 综合成并行 MUX
always @(*) begin
    case (sel)
        2'b00:   y = data0;
        2'b01:   y = data1;
        2'b10:   y = data2;
        2'b11:   y = data3;
        default: y = 'bx;   // 不关心，综合器会优化
    endcase
end

注意 default 子句——这是本章最重要的注意事项之一。

4.3.3 锁存器是怎么"意外"产生的

当 always @(*) 块中，某个分支没有给信号赋值时，综合器会推断该信号需要保持原值——这只能通过锁存器实现。这就是 latch 的根源。

触发方式：

  • if 缺失 else 分支
  • case 未覆盖所有分支且没有 default
  • always 块中某个条件分支没有为所有 output 赋值
// 这会产生 LATCH！
always @(*) begin
    if (en)
        q = d;
    // 没有 else——en=0 时 q 保持原值 -> latch！
end

正确的写法：要么补全 else，要么在 always 块开始处给 q 赋默认值：

// 不会产生 latch
always @(*) begin
    q = 1'b0;      // 默认值
    if (en)
        q = d;
end

4.3.4 casez/casex 的陷阱

casez 将高阻 z 视为"不关心"，casex 将 z 和 x 都视为"不关心"。它们用于描述某些位不必匹配的情形：

casez (sel)
    3'b1??: y = data0;  // 只要 bit2=1，其他位不关心
    3'b01?: y = data1;  // bit2=0, bit1=1, bit0 不关心
    3'b001: y = data2;
    default: y = 'bx;
endcase

陷阱在于：casez/casex 的仿真行为可能和综合结果不一致。仿真器中的 z/x 处理方式与综合器不同，容易造成仿真通过但实际不工作。一般建议：只有在编写 testbench 时才使用 casez，RTL 代码中使用普通的 case + default 即可。

case 编码风格建议：在 RTL 的 case 语句中，始终包含 default 分支。即使你确信所有分支都已覆盖（比如 2-bit 的 4 种可能），也写上 default: ...——这不仅可以防止意外产生的 latch，还有额外的安全作用：如果未来你扩宽了 sel 的位宽但忘了更新 case，default 分支会兜底。对于输出值，default 分支可以赋为 'bx（让综合器做最优化）或一个合法的默认值（让行为更可预测）。

4.4 parameter 与 localparam

4.4.1 parameter：可被外部重写的参数

parameter 可以在模块例化时通过 #() 重写：

module fifo #(
    parameter DATA_WIDTH = 8,
    parameter ADDR_DEPTH = 16
) (
    // 端口列表
);
    localparam ADDR_WIDTH = $clog2(ADDR_DEPTH);
    // ...
endmodule

4.4.2 localparam：模块内部常量

localparam 对外不可见，也不能被重写。用于定义模块内部的常量，比直接写数字更具可读性：

localparam IDLE    = 2'b00;
localparam START   = 2'b01;
localparam DATA    = 2'b10;
localparam STOP    = 2'b11;

4.4.3 generate 语句简介

generate 在综合时"展开"重复电路结构。常用于参数化位宽的处理：

// 参数化位宽的多路选择器
module mux_n #(parameter WIDTH = 8) (
    input  wire [WIDTH-1:0] a, b,
    input  wire             sel,
    output wire [WIDTH-1:0] y
);
    genvar i;
    generate
        for (i = 0; i < WIDTH; i = i + 1) begin : gen_mux
            assign y[i] = sel ? b[i] : a[i];
        end
    endgenerate
endmodule

generate for 和 generate if/else 允许你在综合时根据参数"生成"不同的电路。这在参数化设计中非常强大。

4.5 阻塞赋值对比表：综合与仿真的行为差异

场景代码仿真行为综合结果是否一致?
组合逻辑 with =always @(*) a = b; c = a;c 拿到 b 的值c 和 a 都是 b 的 LUT 输出一致 ✅
时序逻辑 with <=always @(posedge clk) a <= b; c <= a;c 拿到 a 的旧值两级 FF，移位寄存器一致 ✅
时序逻辑 with =always @(posedge clk) a = b; c = a;c 拿到 b 的值（直通）两级 FF，移位寄存器不一致 ❌
组合逻辑 with <=always @(*) a <= b;a 在 delta 延迟后更新组合逻辑 LUT不一致 ❌

这张表值得你贴在工位上。核心结论：对时钟沿赋值的用 <=，对其他情况赋值的用 =。

4.6 一个真实的踩坑故事：case 漏了 default，芯片多了两个 latch

2020 年，我在一个视频处理项目中，用 FPGA 做不同分辨率下的色度格式转换（4:4:4 ↔ 4:2:2 ↔ 4:2:0）。中间有一个"格式选择"模块，根据输入的模式信号选择对应的色度抽取滤波器系数。代码长这样：

// 有问题的代码风格——漏了 default
always @(*) begin
    case (fmt_sel)
        2'b00: coeff = coeff_444;
        2'b01: coeff = coeff_422;
        2'b10: coeff = coeff_420;
        // 没有 2'b11 的分支，也没有 default！
    endcase
end

我心想："fmt_sel 只有三位（实际只用到 00/01/10），不会出现 11。所以可以省掉 default。"这就是典型的"我以为"——在验证阶段，功能仿真通过了，所有情况都覆盖到了。但综合后网表的资源报告显示多了两个 latch。

问题在哪？综合器很诚实——它看到 case 没有覆盖 2'b11，而且没有 default，就推断 fmt_sel == 2'b11 时需要保持 coeff 的当前值。保持值就需要锁存器。虽然 2'b11 在正常操作中不会出现，但复位后 fmt_sel 的初始值可能是 X 或 11（取决于复位方式）——这时 coeff 输出进入不期望的锁存状态，导致后续的色度插值计算产生错误的像素值。

修复方式简单到令人羞愧：

// 修复：加 default
always @(*) begin
    case (fmt_sel)
        2'b00: coeff = coeff_444;
        2'b01: coeff = coeff_422;
        2'b10: coeff = coeff_420;
        default: coeff = coeff_444;  // 安全的默认值
    endcase
end

此后我在所有的 case 语句末尾写 default——即使只需要一行注释。三年来，从来没有因为这个原因再出过 latch 相关的问题。
