第十一章：HTTP认证

各位同学好。上一章我们讲了Cookie和Session——HTTP如何通过Cookie实现"记住你是谁"。但Cookie只是传递身份信息的一种载体。这一章我们从更底层的角度，看HTTP协议本身提供了哪些认证（Authentication）机制，以及在实际系统中我们如何选择。

11.1 认证 vs 授权

在深入具体的认证方案之前，有必要先厘清两个被大量混用的概念：认证（Authentication）和授权（Authorization）。

  • 认证——你是谁？验证用户身份的过程。登录时输入用户名密码就是认证。
  • 授权——你能做什么？确定已认证用户有哪些权限。比如管理员可以删除文章，普通用户只能看。

HTTP协议主要参与了认证这一层。它通过状态码401和响应头 WWW-Authenticate 告诉客户端"请提供身份凭证"，客户端通过 Authorization 请求头携带凭证。至于"这个用户是否有权限执行某操作"——那是应用层的责任，HTTP协议不管。

一句话总结：认证是"证明你是你"（出示身份证），授权是"你能进哪个门"（检查门票）。HTTP帮你解决第一个问题，第二个问题由你的业务代码解决。实际开发中经常有人在API里把401和403混用——记住：401是认证失败（没出示或出示了无效凭证），403是授权不足（认证成功了但没权限）。

11.2 Basic认证

Basic认证是最简单的HTTP认证方式。流程如下：

  1. 客户端请求受保护的资源
  2. 服务器返回 401 Unauthorized，带上 WWW-Authenticate: Basic realm="Protected Area"
  3. 浏览器弹出登录对话框，用户输入用户名和密码
  4. 浏览器将 用户名:密码 用Base64编码后，通过 Authorization: Basic base64(username:password) 发送
请求：
GET /admin HTTP/1.1
Host: example.com

响应（第一次，未认证）：
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Basic realm="管理后台"

重试（带凭证）：
GET /admin HTTP/1.1
Host: example.com
Authorization: Basic YWRtaW46cGFzc3dvcmQxMjM=
                    ↑ base64("admin:password123")

Basic认证的致命缺陷：Base64不是加密，是编码。任何人都可以轻松解码 YWRtaW46cGFzc3dvcmQxMjM= 得到用户名和密码。在HTTP明文传输下，中间人攻击可以轻松截获。所以：

  • 永远不要在HTTP上使用Basic认证
  • 即使使用HTTPS，Basic认证仍然有两个问题：密码每次请求都发送，增加了泄露风险；没有登出机制（浏览器会一直缓存credentials）
  • 现在Basic认证最常见的用途是内部服务间的简单鉴权、或做一次性调试

踩坑故事：刚工作那会儿维护一个内部监控系统，用了Basic认证做登录。因为是内部网络，大家觉得不用HTTPS也没事。结果有天安全扫描发现——有人在办公楼的公共WiFi上抓包，拿到了管理员密码。就因为从监控页面到服务器之间是HTTP明文传输，Base64的"密码"等于明文。那次教训让我养成了个习惯：不管内外网，涉及密码传输的一律用HTTPS。

11.3 Digest认证

Digest认证是Basic的改进版，它解决了"密码明文传输"的问题。核心思路是挑战-响应（Challenge-Response）：

  1. 服务器返回401时带上一个随机数（nonce）
  2. 客户端用MD5对 用户名:realm:密码 和 nonce:HTTP方法:URI 进行哈希运算
  3. 服务器用同样的方式计算哈希，比对结果

这样，密码永远不会在网络上直接传输，传输的只是哈希结果：

响应（挑战）：
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Digest realm="Test",
                  nonce="dcd98b7102dd2f0e8b11d0f600bfb0c093",
                  algorithm=MD5, qop=auth

请求（响应挑战）：
Authorization: Digest username="admin",
               realm="Test",
               nonce="dcd98b7102dd2f0e8b11d0f600bfb0c093",
               uri="/admin",
               response="6629fae49393a05397450978507c4ef1",
               qop=auth, nc=00000001, cnonce="0a4f113"

看起来Digest比Basic安全多了，为什么它没有流行起来呢？原因有几个：

  • MD5哈希本身已经被认为不够安全
  • 服务器需要存储用户的密码（或者密码的哈希），而更好的方案是存储密码的加盐哈希
  • 实现复杂度远高于Basic，但安全性提升有限
  • 无法防御中间人攻击（攻击者可以修改响应头中的nonce）
  • 不支持现代认证需求（如Token刷新、细粒度权限控制）

由于以上原因，Digest认证在实践中用得很少。如果你的系统需要比Basic更强的安全性，更推荐直接用HTTPS + Token认证方案，而不是升级到Digest。

11.4 Bearer Token

Bearer Token是当前最主流的API认证方式。它属于HTTP的Bearer认证方案（RFC 6750），格式为：

Authorization: Bearer <token>

最常见的Token类型是JWT（JSON Web Token）。它由三段Base64编码的字符串组成，用点号分隔：

// JWT结构：header.payload.signature
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJ1c2VyX2lkIjoxMjMsInJvbGUiOiJhZG1pbiIsImV4cCI6MTcwMDAwMDAwMH0.
sQ0XK9zOq5p5e6P8dVgJ7f3L2kZ1bN4mR8cH5tV6yBw
  • Header：声明签名算法（如HS256、RS256）和Token类型
  • Payload：存放用户ID、角色、过期时间等声明（claims）
  • Signature：对前两部分的签名，防止篡改

JWT的核心优势：

  • 无状态——服务器不需要存储Session，Token本身包含了用户信息
  • 跨域友好——适合微服务和前后端分离架构
  • 自包含——一次解密即可获取用户信息，不需要查数据库

OAuth2流程概述——Bearer Token在OAuth2中被广泛使用。典型的授权码流程：

  1. 用户点击"使用微信登录"
  2. 应用将用户重定向到微信的授权页面
  3. 用户同意授权
  4. 微信回调应用，附带一个授权码（Authorization Code）
  5. 应用服务器用授权码向微信服务器换取Access Token（Bearer Token）
  6. 后续请求携带Access Token访问资源

Token存储安全：Bearer Token应该存在哪里？这是个经典的安全问题。localStorage 最方便，但任何XSS漏洞都会导致Token泄露。HttpOnly Cookie可以防御XSS窃取，但容易受CSRF攻击（需要用SameSite或CSRF Token防御）。目前的主流建议是：对敏感度较高的应用，将Access Token存在内存变量中，Refresh Token存在HttpOnly Cookie中。没有完美的方案，需要根据你的安全等级和实现复杂度做权衡。

11.5 API Key认证

API Key认证是服务间通信中最简单的认证方式。它通常通过 X-API-Key 自定义头传递：

GET /v1/users HTTP/1.1
Host: api.example.com
X-API-Key: sk_live_abc123def456gh789

API Key通常由服务提供商生成，是固定不变的字符串。它的适用场景：

  • 公开API的开发者认证——如Stripe、OpenAI、GitHub的API
  • 内部服务间通信——简单、直接、轻量
  • 配合限流使用——按API Key统计请求次数，实施限流

API Key的注意事项：

  • API Key一般没有过期时间（不像JWT有exp字段），所以泄露后需要手动吊销
  • 不要在前端代码中硬编码API Key
  • 建议配合IP白名单使用，增加安全性
  • 定期轮换API Key

踩坑故事：有次我们开发移动端App，为了方便直接把API Key写死在客户端代码里。结果被竞争对手反编译APK拿到Key，然后恶意刷我们接口，导致服务器过载。更麻烦的是，因为Key没有绑定具体用户，我们没法区分正常请求和恶意请求，只能紧急发版更换Key。从那以后我们养成了几个习惯：移动端不走API Key，改用OAuth2 + Access Token；API Key总是绑定IP或特定权限范围；每个API Key都配独立的Rate Limit。

11.6 Session Cookie vs Token

这是现代Web架构中最常讨论的议题：Session Cookie方案和Token/JWT方案各有什么优缺点？我们在下面做一个全面的对比：

维度Session CookieToken / JWT
存储位置服务端（内存/Redis/数据库）客户端（Token自身携带信息）
扩展性需要共享Session存储（Sticky Session或集中式缓存）天然无状态，服务端无需共享
跨域支持困难（Cookie跨域需要特殊配置）天然支持（通过Authorization头携带）
移动端适用不友好（需要Cookie支持）完美（Header直接携带）
登出能力强（服务端清除Session即可）弱（Token过期前仍然有效）
Token吊销容易（删Session记录）困难（需维护黑名单或等过期）
信息安全好（敏感数据在服务端）一般（Payload可Base64解码，不要存敏感信息）
性能每次请求查Session存储无需查询，但验证签名有CPU开销
典型场景传统Web应用（服务端渲染）API服务、前后端分离、微服务

如何选择？我的建议是：

  • 传统Web应用（如管理后台、博客系统）→ Session Cookie方案，简单安全，框架原生支持
  • 前后端分离的SPA → Token/JWT方案，跨域友好，适合API调用
  • 微服务架构 → Token/JWT方案，每个服务无需共享Session存储
  • 移动端App → Token/JWT方案，天然适合不依赖Cookie的场景
  • 公开API → 优先推荐OAuth2 + Bearer Token，辅以API Key

实战建议：不要为了用JWT而用JWT。如果你在做一个传统的服务端渲染应用（PHP + Session、Java + JSP），用Session Cookie完全没问题。如果你的场景需要跨域、微服务、或者移动端，JWT更合适。"用JWT就不用查数据库"这个"性能优势"在绝大多数场景下并不是决定性因素——Session的查询时间（Redis，1-5ms）对用户体验影响微乎其微。
