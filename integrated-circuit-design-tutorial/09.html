第9章 逻辑综合

大家好，我是你们的讲师。上一章我们写了RTL代码，但写出的Verilog还只是文本文件——要让它变成真正的电路，还需要一个关键的步骤：逻辑综合（Logic Synthesis）。

如果把芯片设计比作建筑行业：RTL代码是建筑设计图纸，晶圆厂是施工队，那么逻辑综合就是"图纸到建材清单"的转换——你把一张画满了门窗、墙体、管线的图纸交给施工队，对方需要知道"要用多少块砖、多少根钢筋、多少袋水泥"。在芯片设计中，综合工具就是把RTL描述"翻译"成标准单元（standard cell）组成的门级网表。没有这一步，你的RTL代码就是一堆无法制造的文字。这一章，我们就来深入理解这个过程。

9.1 综合基本流程

逻辑综合的过程可以分为三个核心步骤：Translation（翻译）→ Optimization（优化）→ Mapping（映射）。这三个步骤由EDA工具（主要是Synopsys Design Compiler，或Cadence Genus）自动完成。

  • Translation（翻译）：综合工具读入RTL代码（Verilog/VHDL/SystemVerilog），将其解析为一个与工艺无关（technology-independent）的内部表示——通常称为GTECH网表。这个网表由基本的布尔逻辑门（AND、OR、NOT、MUX、DFF等）组成，不依赖任何具体的标准单元库。这一步相当于"把你的代码变成一张用通用元件搭起来的草图"。
  • Optimization（优化）：工具对GTECH网表进行逻辑优化——化简布尔表达式、合并冗余逻辑、消除共享子表达式、重定时等。优化的目标是满足时序约束的前提下，最小化面积和功耗。这一步是综合工具的核心竞争力所在——同一段RTL代码，不同工具的优化质量可能差异很大。
  • Mapping（映射）：将优化后的逻辑网表映射到目标工艺库（target library）提供的标准单元上。标准单元库包含了INV、NAND、NOR、AOI、OAI、DFF、MUX等基本单元的多种尺寸/驱动强度版本。工具会根据面积、延迟、功耗的权衡选择最合适的单元。

一个典型的Synopsys Design Compiler综合流程如下：

# ============================================
# Design Compiler 综合脚本示例
# ============================================
# 1. 设置库文件
set target_library      "tcbn28hpcplus.db"
set link_library        "* $target_library"
set synthetic_library   "dw_foundation.sldb"
set search_path         "./rtl ./scripts"

# 2. 读入RTL
analyze -format verilog {top.v datapath.v controller.v}
elaborate top

# 3. 施加约束
source -echo constraints.sdc
set_max_area 0
set_max_fanload 4 top/all_outputs()

# 4. 选择编译策略
compile_ultra -timing_high_effort_script

# 5. 分析结果
report_timing > reports/timing.rpt
report_area  > reports/area.rpt
report_power > reports/power.rpt

# 6. 保存网表
write -hierarchy -format verilog -output mapped/top.vg
write_sdc -output mapped/top.sdc

9.2 SDC约束编写

综合工具必须知道你的设计目标——工作频率、输入输出延时、时钟关系——才能进行有效的优化。这些信息通过SDC（Synopsys Design Constraints）文件提供。SDC是芯片设计中最重要、也最容易出错的文件之一。

以下是常用的SDC命令：

SDC命令 功能描述 示例
create_clock 定义时钟信号的周期和波形 create_clock -name clk -period 10 [get_ports clk]
create_generated_clock 定义由主时钟分频/倍频产生的派生时钟 create_generated_clock -divide_by 2 -source clk [get_pins div/out]
set_input_delay 定义输入端口相对于时钟的到达时间 set_input_delay -clock clk -max 2.5 [get_ports data_in]
set_output_delay 定义输出端口相对于时钟的所需建立时间 set_output_delay -clock clk -max 3.0 [get_ports data_out]
set_clock_uncertainty 为时钟抖动和裕度保留余量 set_clock_uncertainty -setup 0.2 [get_clocks clk]
set_false_path 标记不需要时序检查的路径 set_false_path -from [get_clocks rst_clk] -to [get_clocks core_clk]
set_multicycle_path 指定路径需要多个时钟周期完成 set_multicycle_path 2 -setup -from [get_pins slow_path/reg/CK]
set_max_fanout 限制最大扇出 set_max_fanout 8 [current_design]

一个完整的SDC文件示例：

# ============================================
# 芯片顶层约束文件 top.sdc
# 工艺: 28nm HPC+, 工作频率: 500MHz
# ============================================

# 主时钟定义: 500MHz -> period = 2.0ns
create_clock -name core_clk -period 2.0 [get_ports clk]
set_clock_uncertainty -setup 0.15 [get_clocks core_clk]
set_clock_uncertainty -hold  0.05 [get_clocks core_clk]
set_clock_transition -rise 0.08 [get_clocks core_clk]
set_clock_transition -fall 0.08 [get_clocks core_clk]

# 派生时钟: UART时钟(115200 baud)
create_generated_clock -name uart_clk \
    -divide_by 434 -source [get_ports clk] \
    [get_pins uart_divider/out]

# 输入延时: DDR接口
set_input_delay -clock core_clk -max 0.8 \
    [get_ports ddr_*_in]
set_input_delay -clock core_clk -min 0.3 \
    [get_ports ddr_*_in]

# 输出延时: AHB总线
set_output_delay -clock core_clk -max 0.6 \
    [get_ports haddr* hwdata*]
set_output_delay -clock core_clk -min 0.2 \
    [get_ports haddr* hwdata*]

# 异步路径: 标记为false path
set_false_path -from [get_clocks core_clk] \
               -to   [get_clocks uart_clk]
set_false_path -from [get_clocks uart_clk] \
               -to   [get_clocks core_clk]

# 慢速路径: 两个周期完成
set_multicycle_path 2 -setup \
    -from [get_pins slow_mult/pipe_reg*/CK]
一次真实的踩坑经历：在一个项目中，设计师用create_clock定义了一个200MHz时钟，但他忘记指定时钟名——工具自动命名为"clk"。之后他又用create_generated_clock定义了一个频率为100MHz的分频时钟，也忘了命名。结果工具把两个时钟都命名为"clk_1"和"clk_2"（自动递增）。两条异步路径的set_false_path因为时钟名写错，完全没有生效。综合完成后，STA报告在异步路径上有-200ps的setup violation。整个团队花了3天才定位到这个问题——不是RTL有问题，不是工艺有问题，只是因为时钟没写名字。从那以后，我团队规定：所有create_clock和create_generated_clock必须显式加-name参数。

9.3 时序路径分析

综合工具在做优化时，本质上是分析所有时序路径并确保每条路径都满足约束。时序路径分为四类：

  • in2reg：从输入端口到时序器件的D端。路径延迟由input_delay + 组合逻辑 + 寄存器setup时间组成。
  • reg2reg：从一个时序器件的CK端到另一个时序器件的D端。这是最常见的路径类型，也是约束最严格的。路径延迟 = Tcq + Tcomb + Tsetup。
  • reg2out：从时序器件的CK端到输出端口。路径延迟 = Tcq + Tcomb + output_delay。
  • in2out：从输入端口到输出端口（纯组合路径）。较少见，通常用set_max_delay约束。

对于reg2reg路径，时序检查的基本方程是：

setup检查：数据到达时间 ≤ 时钟到达时间 + 时钟周期 - setup时间
hold检查：数据到达时间 ≥ 时钟到达时间 + hold时间

综合工具会计算每条路径的Slack——即裕量。Slack为正表示满足约束，为负表示违例。两个关键指标：

  • WNS（Worst Negative Slack）：所有路径中最差的slack值。如果WNS为负，说明至少有一条路径不满足时序。
  • TNS（Total Negative Slack）：所有负slack路径的slack绝对值之和。TNS反映了违例的严重程度——如果WNS=-0.3ns但TNS也才-0.5ns，说明只有少数几条路径有问题；如果TNS很大（比如-500ns），说明整个设计都存在严重的时序问题。
工程师经验：在综合阶段看到WNS为负时不要慌——先看TNS。如果WNS=-0.3ns但TNS=-0.8ns（只有2-3条路径违例），大概率是代码中某几个特定module的问题，定位修复即可。如果TNS在-50ns以下，那你需要重新审视你的RTL架构和流水线划分了。

9.4 面积与功耗优化

综合工具不仅要做时序优化，还要同时优化面积和功耗——这三个目标往往是相互冲突的：

面积优化

  • 资源共享（Resource Sharing）：如果两个运算单元永远不会同时使用，综合工具可以共享一个硬件资源。例如两个不互斥的加法器可以合并为一个。
  • 操作符大小优化（Operator Sizing）：根据数据位宽选择合适的运算单元。如果你的加法器只需要16位，不要用32位的。
  • 逻辑门重构：用更少的门实现相同的逻辑功能。例如用AOI（AND-OR-INVERT）复合门替代多个离散门。

功耗优化

  • Clock Gating插入：综合工具可以自动在寄存器上插入clock gating逻辑——当寄存器的输入数据不变时，关闭时钟翻转，减少动态功耗。这是综合阶段最重要的低功耗技术（详细见第13章）。
  • 操作数隔离（Operand Isolation）：当一个数据通路的输出在不使用时，工具可以插入隔离逻辑，防止不必要的组合逻辑翻转。

在Design Compiler中，这些优化可以通过compile_ultra命令配合各种选项开启，比如-gate_clock、-retime、-spg等。

9.5 多角多模综合

一颗芯片需要在不同的工艺角（process corner）、电压、温度条件下都能正常工作。综合阶段需要同时考虑这些条件——这就是MCMM（Multi-Corner Multi-Mode）综合。

常见的工艺角包括：

  • SS（Slow-Slow）：NMOS慢、PMOS慢，对应最差的延迟情况。用于setup分析。
  • FF（Fast-Fast）：NMOS快、PMOS快，对应最好的延迟情况。用于hold分析。
  • TT（Typical-Typical）：典型工艺条件。
  • SF/FS：NMOS慢PMOS快或反之，用于特定工艺偏差分析。

电压和温度会进一步组合：低压低温（最慢）、高压高温（最快）、以及标称条件。一个典型的移动芯片可能有10-15个"corner+mode"组合需要同时满足约束。

不同的"工况"（mode）也各不相同：

  • 功能模式（Functional Mode）：芯片正常工作时。
  • 测试模式（Test Mode）：DFT扫描模式，时钟路径不同，需要额外的约束。
  • 低功耗模式（Low Power Mode）：部分电源关闭，需要特殊约束。
  • 调试模式（Debug Mode）：JTAG调试路径激活。

MCMM综合就是在一轮综合中同时满足所有corner+mode组合的约束——工具不再分别做多次综合分析，而是读取所有约束后在网表上一次性做多目标优化。这大幅减少了综合迭代次数。

setup角 vs hold角：初学者的一个常见困惑是，为什么setup和hold要分开分析。setup最差的角是SS（最慢的工艺、最低的电压、最高的温度）——数据路径最慢，setup最容易违例。hold最差的角是FF——数据路径最快，hold最容易违例。所以综合工具需要对SS角做setup优化，对FF角做hold修正。如果哪一天你看到有人在SS角分析hold问题，或者用FF角分析setup问题，那说明他对时序的基础理解还需要补补课了。
