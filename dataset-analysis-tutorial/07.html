第7章：Split切分与采样策略

前面几章我们加载的数据集大多已经有了现成的train/test/validation split。但现实项目里，这种情况更像是"例外"而非"常态"。你手头可能只有一堆原始CSV文件，或者从生产数据库导出的一张表，需要自己决定如何划分数据集。更麻烦的是——简单的随机切分在很多场景下是不够的，甚至可能是有害的。

想象一下：你要做一个医疗时间序列预测模型，按时间顺序收集了病人三年的就诊数据。如果你随机切分训练集和测试集，就可能把同一个病人的早期数据放在训练集、晚期数据放在测试集——然后你的模型测试分数看起来很漂亮，但实际上它已经"见过"这个病人的模式了。这就是数据泄露（Data Leakage）的一个典型案例。本章我们来系统学习Split切分和采样的各种策略。

Split切分原理

当你拿到一个没有预设split的数据集时，第一件事就是把它切分为训练集、验证集和测试集。datasets库提供了 train_test_split() 方法来完成这个任务。

train_test_split 基础

from datasets import load_dataset

# 加载一个没有预设split的数据集
dataset = load_dataset("csv", data_files="raw_data.csv", split="train")
print(len(dataset))  # 比如 10000

# 切分为训练集和测试集
split_dataset = dataset.train_test_split(test_size=0.2, seed=42)
print(split_dataset)
# DatasetDict({
#     train: Dataset({ num_rows: 8000 })
#     test: Dataset({ num_rows: 2000 })
# })

# 进一步从训练集中切分验证集
train_val = split_dataset["train"].train_test_split(test_size=0.125, seed=42)
# 0.125 * 8000 = 1000，即验证集占总样本的10%
final_dataset = DatasetDict({
    "train": train_val["train"],       # 7000
    "validation": train_val["test"],   # 1000
    "test": split_dataset["test"],     # 2000
})
小技巧：你不需要一次性完成完美切分。先用一个简单的80/20划分开始训练——随着项目推进，你可以随时调整split比例。重要的是保持seed一致，确保每次切分的分布是确定的。

参数详解

参数 类型 默认值 说明
test_size float / int 0.1 float时为比例（0-1），int时为样本数量
train_size float / int None 与test_size互斥，指定其中一个即可
seed int None 随机种子，保证可复现
shuffle bool True 是否在切分前shuffle数据
stratify_by_column str None 按该列进行分层采样

Seed的重要性

每次运行 train_test_split 时，如果不指定 seed，你会得到不同的随机切分。这会导致两个问题：

  • 不可复现：你和同事各自跑一次实验，结果不同，无法互相验证
  • 测试集污染：如果你在调参过程中反复重新切分数据，测试集的样本结构每次都在变，最终评测结果失去意义
# 正确的做法：固定seed
dataset.train_test_split(test_size=0.2, seed=42)

# 错误的做法：不设seed（每次结果不同）
dataset.train_test_split(test_size=0.2)  # 不要这样做
重要原则：一旦你确定了train/val/test的切分方案，就要把切分结果固定下来（比如保存为三个独立的Arrow文件）。后续所有实验、调参、评估都使用同一份切分。这就像一份学术论文——一旦你决定了实验方案，就不能在看了实验结果之后再回去调整训练集和测试集的划分。

分层采样

随机切分在大多数情况下工作良好，但有一个例外——类别不均衡。如果你的数据集中90%是类别A、10%是类别B，简单的随机切分可能导致某个split中完全没有类别B的样本。分层采样就是解决这个问题的。

什么情况下必须分层

  • 数据集的类别分布严重不均衡（如异常检测：99.9%正常 vs 0.1%异常）
  • 标签数量较少（少于1000条）
  • 你需要精准评估模型在每个类别上的表现

stratify_by_column 参数

from datasets import load_dataset

dataset = load_dataset("imdb", split="train")

# IMDB的标签是均衡的（50% pos, 50% neg）
# 但这里演示分层采样的用法
split = dataset.train_test_split(
    test_size=0.2,
    seed=42,
    stratify_by_column="label",  # 按标签列分层
)

# 验证分层结果
train_labels = split["train"]["label"]
test_labels = split["test"]["label"]

print(f"训练集: {sum(train_labels)/len(train_labels):.1%} positive")
print(f"测试集: {sum(test_labels)/len(test_labels):.1%} positive")
# 两者的positive比例应该几乎相同（≈50%）

在不均衡数据集上的效果

假设你有一个只有5%正样本的极度不均衡数据集：

# 生成模拟数据
import random
data = {
    "text": [f"sample_{i}" for i in range(10000)],
    "label": [1 if random.random() < 0.05 else 0 for _ in range(10000)],
}
dataset = Dataset.from_dict(data)

print(f"正样本比例: {sum(dataset['label']) / len(dataset):.2%}")
# 约5%

# 不设分层（随机切分）
split_random = dataset.train_test_split(test_size=0.2, seed=42)
test_pos_ratio = sum(split_random["test"]["label"]) / len(split_random["test"]["label"])
print(f"随机切分后的测试集正样本比例: {test_pos_ratio:.2%}")

# 设分层
split_strat = dataset.train_test_split(
    test_size=0.2, seed=42, stratify_by_column="label"
)
test_pos_ratio_strat = sum(split_strat["test"]["label"]) / len(split_strat["test"]["label"])
print(f"分层切分后的测试集正样本比例: {test_pos_ratio_strat:.2%}")

# 随机切分在运气不好时，测试集可能只有0.5-2%的正样本
# 分层切分确保测试集正样本比例稳定在5%左右
什么时候不需要分层？如果你的数据集足够大（>10万条）且标签分布没有严重倾斜（最少的类别>10%），随机切分已经足够。分层采样只在小样本或严重不均衡时是必要的。过度使用分层不会提高模型质量，只是徒增代码复杂度。

shard 分片

当你的训练需要分布式执行时——比如用4块GPU并行训练——你需要把数据集切分成多个分片（shard），每个GPU只负责训练自己的那个分片。这就是 shard() 方法的用武之地。

基本用法

dataset = load_dataset("imdb", split="train")

# 切分为4个分片
shard_0 = dataset.shard(num_shards=4, index=0)
shard_1 = dataset.shard(num_shards=4, index=1)
shard_2 = dataset.shard(num_shards=4, index=2)
shard_3 = dataset.shard(num_shards=4, index=3)

print(f"分片0的大小: {len(shard_0)}")  # 6250（25000/4）
print(f"分片1的大小: {len(shard_1)}")  # 6250

每个分片的大小几乎相等——如果数据集大小不能被分片数整除，前几个分片会多一个样本。

分布式训练中的shard

在PyTorch DDP（Distributed Data Parallel）训练中，shard的使用模式如下：

import torch
import torch.distributed as dist

# 初始化分布式环境
dist.init_process_group(backend="nccl")
rank = dist.get_rank()        # 当前GPU的编号（0, 1, 2, 3）
world_size = dist.get_world_size()  # GPU总数

dataset = load_dataset("imdb", split="train")

# 每个进程拿到自己的数据分片
my_shard = dataset.shard(
    num_shards=world_size,
    index=rank,
    contiguous=True,  # 是否连续切分（默认False为均匀间隔）
)

# 每个进程只在自己的分片上训练
dataloader = DataLoader(
    my_shard.with_format("torch"),
    batch_size=32,
    shuffle=True,
)
contiguous参数解析：
  • contiguous=True：样本按连续块分配（前2000条给GPU0，接下来2000条给GPU1...）
  • contiguous=False（默认）：样本按间隔分配（GPU0拿0,4,8,...，GPU1拿1,5,9,...）
对于大多数深度学习的SGD训练来说，contiguous=True 和 contiguous=False 的区别可以忽略——因为每个进程内部还会做shuffle。但如果你有特殊的样本排序需求（比如按序列长度排序以优化padding），可能需要选择特定的切分方式。

shard 与 seed 的配合

为了让多GPU训练的复现性更好，建议在shard之前先shuffle：

# 先在全局做一次确定性的shuffle
dataset = dataset.shuffle(seed=42)

# 然后再shard
my_shard = dataset.shard(num_shards=world_size, index=rank)

# 这样每次训练时，各个GPU拿到的数据分片是确定性的
# 确保实验完全可复现

select / take / take_every

除了切分和分片，日常工作中更频繁使用的其实是三个"轻量级"的数据选取方法——select、take 和 take_every。

select — 按索引精确选取

dataset = load_dataset("imdb", split="train")

# 选取特定索引的样本
subset = dataset.select([0, 5, 10, 100, 1000])
print(len(subset))  # 5

# 选取连续区间
subset = dataset.select(range(100))
print(len(subset))  # 100

# 高级：选取特定条件的索引
import numpy as np
positive_indices = np.where(np.array(dataset["label"]) == 1)[0]
positive_subset = dataset.select(positive_indices[:100])
print(len(positive_subset))  # 100 (全部是正面评论)

take — 取前N条

# 快速取前N条
head = dataset.take(5)
for sample in head:
    print(sample["text"][:50])

# take和select(range(N))等效，但更简洁
# 常用于数据预览和pipeline测试

take_every — 间隔采样

# 每隔k条取一条
every_100th = dataset.take_every(k=100)
print(len(every_100th))  # 250 (25000/100)

# 适用于快速原型验证
# 在全量数据上跑一次map可能要5分钟
# 在间隔采样数据上只需要几秒
small_proto = dataset.take_every(k=100)
result = small_proto.map(lambda x: {"len": len(x["text"])})
print(result["len"])  # 快速获得一个近似的长度分布
快速原型验证的技巧：在开发数据处理pipeline时，永远先用 .take_every(k=100) 或 .select(range(100)) 在子集上测试。确认逻辑正确后，再在全量数据上运行。这个简单的习惯可以让你的开发效率提升5-10倍——因为大部分的bug在小数据集上就能暴露，而你不必每次都等待全量数据的处理时间。

自定义Split策略

对于大多数场景，train_test_split 的分层模式已经足够。但有一些特殊的业务场景需要自定义切分逻辑。

时间序列的时序切分

对于时间序列数据，绝对不能使用随机切分——这会导致"未来数据泄露到过去"。正确的做法是按时间顺序切分：

# 假设数据集包含时间戳
dataset = load_dataset("csv", data_files="time_series.csv", split="train")

# 按时间戳排序
sorted_dataset = dataset.sort("timestamp")

# 前70%作为训练集，后20%作为验证集，最后10%作为测试集
n = len(sorted_dataset)
train = sorted_dataset.select(range(int(n * 0.7)))
validation = sorted_dataset.select(range(int(n * 0.7), int(n * 0.9)))
test = sorted_dataset.select(range(int(n * 0.9), n))

# 或者用split表达式
train = sorted_dataset.select(range(int(n * 0.7)))
val = sorted_dataset.select(range(int(n * 0.7), int(n * 0.9)))
test = sorted_dataset.select(range(int(n * 0.9), n))

用户级切分避免泄漏

在推荐系统、用户行为分析等场景中，最致命的错误是用户级别的数据泄露——同一个用户的部分行为在训练集中，另一部分在测试集中，导致模型作弊。

# 问题：随机切分会把同一个用户的行为分散到训练和测试集
from datasets import Dataset

data = {
    "user_id": ["U1", "U1", "U1", "U2", "U2", "U3", "U3", "U3", "U3"],
    "behavior": ["click", "view", "buy", "view", "click", "click", "buy", "view", "view"],
}
dataset = Dataset.from_dict(data)

# 用户级切分
# 第一步：获取唯一用户列表
unique_users = list(set(dataset["user_id"]))
print(f"唯一用户数: {len(unique_users)}")  # 3

# 第二步：随机切分用户列表（而不是切分样本）
from sklearn.model_selection import train_test_split
train_users, test_users = train_test_split(unique_users, test_size=0.3, random_state=42)

# 第三步：按用户ID过滤样本
train_dataset = dataset.filter(lambda x: x["user_id"] in set(train_users))
test_dataset = dataset.filter(lambda x: x["user_id"] in set(test_users))

print(f"训练集用户: {set(train_dataset['user_id'])}")
print(f"测试集用户: {set(test_dataset['user_id'])}")
# 确保没有任何重叠！
数据泄露是机器学习中最隐蔽、最危险的问题之一。常见的数据泄露来源包括：
  • 用户级泄露：同一用户的数据跨越训练/测试集
  • 时间泄露：测试数据的时间早于训练数据
  • 特征泄露：使用了未来信息构造的特征（如"该用户未来30天的购买金额"）
数据泄露的可怕之处在于：它会提升你所有实验的指标，让你误以为模型表现优秀。模型上线后才发现实际效果远低于预期——但那时你可能已经基于错误的指标做了关键决策。

Group KFold 概念

当你做交叉验证时，如果需要确保同一组的样本不会同时出现在训练和验证集中，就需要Group KFold。虽然datasets库没有直接实现Group KFold的方法，但可以借助scikit-learn并配合filter实现：

from sklearn.model_selection import GroupKFold
import numpy as np

# 假设每个样本有一个group_id（如患者ID、用户ID）
groups = np.array(dataset["user_id"])

# 5折分组交叉验证
gkf = GroupKFold(n_splits=5)
for fold, (train_idx, val_idx) in enumerate(gkf.split(dataset, groups=groups)):
    train = dataset.select(train_idx)
    val = dataset.select(val_idx)

    # 验证：训练集和验证集中的用户ID没有交集
    train_users = set(train["user_id"])
    val_users = set(val["user_id"])
    assert train_users.isdisjoint(val_users), "数据泄露！"
    print(f"Fold {fold}: train={len(train)}, val={len(val)}, 无泄露 ✓")

这种方法确保了每组（用户/患者/会话）的所有数据实例都完整地落在同一个折中，不会在训练集和验证集之间"分裂"。

一个真实的"踩坑"故事

讲一个在Kaggle比赛中真实发生过的事情。

某团队参加一个Kaggle竞赛，比赛的数据集是一个中等规模的二分类问题——预测用户是否会点击广告。团队的数据科学家小刘写了一个标准的训练pipeline，用 train_test_split(test_size=0.2) 切分数据。

第一次提交，公共排行榜上的AUC为0.76——还可以，但不惊艳。团队开始各种调参、特征工程、模型集成，折腾了一周，AUC涨到了0.78。然后有一天，小刘在重构代码时不小心把seed从42改成了123。重新训练后提交，AUC突然跳到了0.83。

团队一开始以为是"改进了特征"导致的提升。但回滚到旧代码后（但保留新seed），AUC还是0.83。继续回滚到旧seed（42），AUC回到了0.78。

问题很清楚了：不同的seed导致不同的数据切分，而某些切分方式恰好让"测试集更容易预测"。这不是模型变强了，而是"考试题目变简单了"。因为每次提交前都会重新切分数据，团队实际上是在用测试集调参——只不过是无意的。

更致命的是，最终的评测是在一个固定的、未公开的隐藏测试集上进行的。当团队用seed=123的模型在私有测试集上评估时，AUC只有0.72——比公共排行榜上的0.83低了整整11个点。

教训：永远固定seed，而且只固定一次。一旦你看了测试集的结果，就不要回去调整seed来"刷分"。如果你的实验结果对seed选择非常敏感（切换seed后指标波动超过1%），说明你的数据集太小或分布不稳定——这时候应该使用k-fold交叉验证来获取更稳健的评估。

小刘后来在团队中推行了一个规则：所有的数据切分代码中，seed都写在一个全局配置文件中，而且一旦项目启动就不再修改。后续的所有实验都基于同一份切分数据。这个简单的规则让他们后续的竞赛成绩稳定了很多。

# 正确的团队协作模式
# config.py
SEED = 42
TEST_SIZE = 0.2
VAL_SIZE = 0.1

# 所有成员共享同一个seed和切分参数
# 训练/验证/测试集对所有成员都是一致的

本章小结

这一章我们从切分到采样完整覆盖了Split的各种策略：

  • train_test_split：基础切分方法，seed的固定不可忽视
  • 分层采样：用stratify_by_column处理不均衡分布
  • shard分片：分布式训练场景下的数据划分
  • select / take / take_every：轻量级数据选取和快速原型验证
  • 自定义策略：时序切分、用户级切分、Group KFold避免数据泄露
