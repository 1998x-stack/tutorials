第24章 推理优化技术

大家好，我是你们的讲师。上一章我们讲了量化，把模型"瘦身"了。但瘦下来不等于跑得快——模型推理的瓶颈从来不只是权重的大小。在很多实际场景中，KV Cache能吃掉比权重更多的显存，attention计算的复杂度随序列长度平方级增长，GPU利用率可能只有个位数。这一章我们深入到推理优化的核心技术，让模型真正"跑起来"。

讲个有意思的踩坑经历。之前帮一个做AI写作的团队做推理优化，他们的模型是Qwen-72B，部署在8张H100上，单次请求的平均生成长度在2,000 tokens左右。团队发现一个奇怪的现象：如果50个用户同时请求，延迟翻了好几倍——从500ms涨到了4秒。他们以为是模型推理太慢，花重金升级到16张卡，结果延迟只降到3秒。我一看监控，问题出在KV Cache上：并发请求一多，所有请求的KV Cache挤在一起，GPU上的显存带宽成了瓶颈。每次生成一个token都要读取全部KV Cache，请求越多，每个请求分到的有效带宽越少。解决方法是改用PagedAttention + 更高效的KV Cache管理——不升级硬件，单靠优化把并发延迟打回了1.5秒。这就是推理优化的魅力：不花一分钱硬件，延迟减半。

24.1 KV Cache深入

KV Cache是自回归生成中最关键的优化。Transformer做推理时，每生成一个token都要计算当前token与之前所有token的attention。如果每步都重新算之前所有token的K和V，计算量会随着序列长度平方级增长。KV Cache的做法很简单：把之前每一步的K和V存下来，新的token只算自己的K和V，然后拼接到Cache里。这样每步的计算量从O(seq_len)降到O(1)。

KV Cache的内存占用。KV Cache是显存的"隐形杀手"。以LLaMA-70B为例（80层，GQA每组8个KV head，hidden_size=8192），每个token的KV Cache大小 = 2（K和V）× 80层 × 8个KV head × 8192 hidden / (8 head) × 2 bytes (FP16) = 2 × 80 × 8 × 1024 × 2 = 2.6 MB per token。听起来不多对吧？但如果有1,024个请求并发，每个请求生成2,048个token的回复，总KV Cache = 2.6 MB × 2048 × 1024 ≈ 5.5 TB。即使使用GQA后KV head数量大幅减少，KV Cache仍然是显存的最大占用者。实际上，在长序列、大batch推理中，KV Cache消耗的显存往往比模型权重还高。

PagedAttention（vLLM的虚拟内存管理）。PagedAttention是vLLM框架的核心创新，灵感来源于操作系统虚拟内存的分页机制。传统注意力机制中，KV Cache是按请求预先分配一块连续显存的——每个请求占据一个大块连续空间，这些空间之间有很多碎片，利用率很低。PagedAttention把KV Cache切分成固定大小的"页"（通常是16或32个token为一页），物理上不要求连续，通过页表来映射逻辑地址到物理地址。这个设计带来的好处非常显著：第一，消除了内部碎片——只需要分配实际用到的页。第二，允许内存共享——多个请求共享同一个prompt的KV Cache（这就是vLLM的Shared Prefix机制）。第三，请求级别的内存管理——一个请求结束后，它的页可以立即回收给其他请求。

Prefix Caching。在很多实际场景中，用户请求的prompt有大量重复前缀——比如系统提示词（system prompt）或者对话历史。如果每个请求都重新计算前缀部分的KV Cache，那是在浪费算力。Prefix Caching的机制是：把前缀的KV Cache缓存到CPU内存或磁盘上，新的请求来了先检查前缀是否匹配，命中就直接加载。在ChatGPT这样的对话场景中，多轮对话的历史消息几乎不变，prefix caching可以省掉70%以上的prefill计算。vLLM的--enable-prefix-caching参数就是干这个的。

KV Cache优化要点：对于生产部署，你至少需要考虑这三件事。第一，激活KV Cache量化——把KV Cache从FP16降到INT8甚至INT4，显存直接减半或减到四分之一，精度损失通常在0.5%以内。第二，Window Attention——不是所有历史token对当前token的影响都那么大，只保留最近N个token的KV Cache（比如4,096个），更早的丢弃。这种方法在长文档场景下特别有效。第三，Layer-wise KV Cache eviction——不同层的KV Cache重要性不同，底层可以多丢弃，顶层少丢弃。这三个技术组合使用，KV Cache的显存占用可以减少5-10倍。

24.2 投机解码(Speculative Decoding)

大模型推理最大的瓶颈是什么？是逐token串行生成。即使batch再大、GPU再强，你也必须在生成完第N个token之后才能生成第N+1个。这个"串行依赖"从根本上限制了推理的吞吐量。投机解码就是用来打破这个瓶颈的。

小模型打草稿+大模型验证。核心思路其实很朴素：用一个小模型（draft model）快速地一次性生成K个候选token，然后让大模型（target model）对这些token做一次并行验证——只做一次前向传播，检查这K个token哪些可以接受，哪些需要重新生成。因为小模型的计算量远小于大模型，而大模型的验证是并行进行的（一次前向传播可以同时处理多个token），整个过程比大模型逐token生成快得多。

Draft Model选择。draft model的选择直接影响投机解码的加速效果。理想情况下，draft model应该满足三个条件：第一，快——draft model的推理速度至少是target model的5-10倍，这通常意味着参数量在target model的1/10以下。第二，准——draft model和target model的生成分布要足够接近，否则接受率低，加速效果差。第三，够小——能塞进target model的剩余显存，不需要额外的GPU。在实践中，最常见的组合是：target model用70B，draft model用7B（或者更激进的2B）。如果draft model和target model来自同一个模型族（比如LLaMA-70B配LLaMA-7B），接受率通常在60-80%。如果来自不同族，接受率可能降到30%以下，加速效果就不明显了。

接受率优化。投机解码的核心指标是接受率（acceptance rate）。接受率越高，需要重跑的token越少，加速比越大。优化接受率的方法主要有三种。第一，Fine-tune draft model——用target model的输出分布来蒸馏draft model，让两个模型的分布更对齐。第二，动态调整草稿长度K——不是固定生成K个token，而是动态调整：如果前几步接受率高，增加K；如果接受率低，减小K。第三，树状投机解码（Tree-based Speculative Decoding）——不生成单一的token序列，而是生成一个包含多条候选路径的"tree"，大模型在这个tree上做并行验证。这种方法在target model输出分布不确定性高的时候特别有效。

# 投机解码核心逻辑（伪代码）
def speculative_decoding(target_model, draft_model, prompt, K=5):
    # 用target model对prompt做一次prefill
    target_kv_cache = target_model.prefill(prompt)

    generated = prompt[:]
    while not eos(generated):
        # Phase 1: Draft model快速生成K个候选token
        draft_kv_cache = draft_model.prefill(generated)
        draft_tokens = []
        for _ in range(K):
            logits = draft_model.decode(draft_kv_cache)
            next_token = sample(logits)
            draft_tokens.append(next_token)
            if next_token == EOS:
                break

        # Phase 2: Target model并行验证候选序列
        target_logits = target_model.decode_all(
            generated + draft_tokens,
            target_kv_cache
        )

        # Phase 3: 逐位置验证，接受匹配的token，拒绝不匹配的
        accepted = 0
        for i, draft_token in enumerate(draft_tokens):
            target_prob = softmax(target_logits[len(generated) + i - 1])
            if draft_token == sample(target_prob):
                accepted += 1
            else:
                # 从target的分布中重新采样
                generated.append(sample(target_prob))
                break

        if accepted == K:
            # 所有K个token都被接受
            generated.extend(draft_tokens)

    return generated

实战数据：在我的测试中，使用LLaMA-70B配LLaMA-7B作为draft model，在H100上投机解码（K=5）的端到端加速比约为1.8-2.5x。加速比随batch size增大而下降——在小batch下（batch=1）提升明显，大batch下由于GPU利用率已经很高，提升空间变小。另外，投机解码对短序列生成（<50 tokens）几乎没有加速——prefill开销占主导，草稿生成的时间折扣不够。最适合的是100-1,000 tokens的中等长度生成场景。

24.3 FlashAttention系列

FlashAttention是近年来对Transformer推理影响最深远的优化之一。它不是一个算法，而是一个IO-aware的精确attention计算方法，目标是用更少的高带宽内存（HBM）访问来完成attention计算。它的核心洞察是：attention的计算瓶颈不是FLOPs，而是内存带宽。

FlashAttention-1：Tiling和Recomputation。标准attention的做法是把Q、K、V全部读到HBM（高带宽显存）中，计算S=QK^T，然后把S写回HBM，再读出来做softmax，得到attention输出。这个过程涉及多次HBM读/写，而HBM的速度远低于SRAM（片上缓存）——H100的HBM带宽约3.35 TB/s，但SRAM带宽可达数倍于此。FlashAttention-1的核心是tiling（分块）：把Q、K、V切分成小块，每次只在SRAM中处理一个块。它通过一个叫"online softmax"的技巧来避免在计算完全部分之前等待全局统计量。这样，大矩阵的中间结果（S矩阵）不需要写回HBM，全部在片上完成。代价是：为了反向传播，需要在前向时重新计算部分结果（recomputation），而不是存储完整结果。这听起来像是"以计算换内存"，但由于HBM访问是真正的瓶颈，实际训练和推理速度都大幅提升。

FlashAttention-2：更聪明的分块和并行。FlashAttention-2在三个维度上做了优化。第一，减少非矩阵乘操作：重写了online softmax实现，把更多计算留给Tensor Core擅长的矩阵乘法。第二，更好的分块策略：在序列维度上做并行，而不是在batch/head维度上——这样充分利用了硬件对序列长度的并行能力。第三，降低wrap-level同步开销：在CUDA kernel层面做了大量微优化。FlAS的结果是：FlashAttention-2比FlashAttention-1快2-3倍，比标准的PyTorch attention实现快10倍以上。

FlashAttention-3：FP8和低精度加速。FlashAttention-3是2025年的最新版本，主要针对H100的FP8 Tensor Core做了深度优化。它引入了三个关键技术。第一，异步处理：把数据从HBM搬运到SRAM和SRAM上的计算重叠起来，隐藏memory latency。第二，FP8块内缩放：在tiling的基础上，对每个tile使用独立的FP8 scaling factor，避免了标准FP8量化中一个序列内数值范围差异大的问题。第三，低精度OFA（Online Fused Attention）：把attention计算链路的多个操作（缩放、mask、softmax）融合成一个FP8 kernel，进一步减少HBM访问。在H100上，FlashAttention-3比FlashAttention-2快1.5-2倍。

版本 核心改进 相对标准Attention加速 FP8支持 适用硬件
FA-1 (2022) Tiling + Online Softmax 2-3x 否 A100+
FA-2 (2023) 序列并行 + Kernel融合 8-10x 否 A100+
FA-3 (2025) 异步 + FP8 Tile Scaling 15-20x 是 H100+

FlashAttention对长序列推理的加速尤为显著。在256K token的序列上，标准attention需要512 GB的HBM（仅中间S矩阵），这远远超出了任何单卡的显存。而FlashAttention通过tiling把中间结果保留在SRAM中，理论上可以处理任意长度的序列——实际限制仅来自模型本身的计算量。LongLoRA、YaRN等长上下文工作都依赖FlashAttention的实现。

24.4 Continuous Batching

在传统的推理服务中，batching策略通常是"静态"的：收集一批请求，同时开始推理，等所有请求都完成后一起返回。这叫做静态批处理（static batching）。但LLM推理有个特殊之处——每个请求的生成长度差异巨大。有的请求只生成"是的"两个字，有的请求生成2,000字的一篇文章。在静态批处理中，第一批最长的请求决定了整批的完成时间，短请求生成完后也得干等。这导致GPU利用率曲线像一个"锯齿"：忙碌期满载运行，空闲期全在等IO。

Dynamic Batching / Inflight Batching。Continuous Batching（也叫inflight batching）解决了这个问题。它的核心思想是：每生成一个token就调度一次。不是等一批请求全部完成再开始下一批，而是在每一步（每个token生成后）检查是否有新的请求可以加入、是否有请求已经完成（EOS触发）。这样，GPU每时每刻都在生成token——短请求快速完成退出，新请求立即补上。这使得GPU利用率从静态批处理的50-70%提升到90%以上。

实现细节。Continuous Batching对推理引擎的调度层提出了很高要求。引擎需要维护一个"等待队列"（waiting queue）和"运行集"（running set）。每个step执行以下操作：第一步，从running set中的所有请求各自生成一个token。第二步，检查哪些请求的EOS被触发了，把它们移出running set。第三步，从waiting queue中拉取新的请求，如果batch size还有余量就填满。第四步，预填充（prefill）新请求的KV Cache。推理引擎的处理能力取决于它能不能在一个step周期内完成prefill+decode的混合计算。vLLM的调度器就是做这个的，它的设计保证了prefill和decode阶段在同一个batch中可以共存，互不阻塞。

调度策略对比。三种batching方案的差异很明显。静态批处理实现简单，但在生成长度差异大的场景下效率极低。动态批处理（DCGM监控到批大小动态变化，但仍是"等一批全做完"）比静态好一些，但仍有空闲浪费。Continuous batching实现了真正的"流水线化"，是当前最先进的方案。vLLM、TensorRT-LLM、SGLang等主流引擎都支持continuous batching。

# Continuous Batching调度伪代码
class ContinuousBatchingScheduler:
    def __init__(self, max_batch_size=64):
        self.running = []   # 当前正在生成的请求
        self.waiting = []   # 等待加入的请求
        self.max_batch = max_batch_size

    def step(self, model):
        # 1. 对所有running请求做一次decode
        outputs = model.decode([req.last_token for req in self.running])

        # 2. 处理完成的和EOS的请求
        finished = []
        for req, token in zip(self.running, outputs):
            req.append_token(token)
            if token == EOS or req.reach_max_length():
                finished.append(req)
        self.running = [r for r in self.running if r not in finished]

        # 3. 从waiting中补充新请求（prefill部分）
        slots = self.max_batch - len(self.running)
        new_requests = self.waiting[:slots]
        self.waiting = self.waiting[slots:]
        for req in new_requests:
            model.prefill(req.prompt_tokens)  # 初始化KV Cache
        self.running.extend(new_requests)

        return [req.output for req in finished]

24.5 推理优化全景对比

现在我们把本章讲的所有技术放到一起，从加速比、适用场景、组合效果三个维度做个全面的比较。

技术 单点加速比 主要瓶颈 最佳场景 组合使用建议
PagedAttention 1.5-3x (throughput) 显存碎片 高并发、大batch 任何场景必选
KV Cache量化 1.5-2x (capacity) 显存容量 长序列、大batch 配合PagedAttention效果翻倍
投机解码 1.5-2.5x (latency) GPU利用率 小batch、中等长度生成 与Continuous Batching互补
FlashAttention 2-10x (compute) Attention计算 长序列推理 所有场景必选
Continuous Batching 1.5-3x (throughput) 调度效率 高并发在线服务 任何在线服务必选

组合使用效果。这些技术不是互斥的而是互补的，叠在一起能产生乘数效应。我们来看一个典型的生产环境配置。模型用FP8量化（上一章内容）+ FlashAttention（减少attention计算量）+ PagedAttention（高效管理KV Cache）+ KV Cache INT8量化（进一步压缩KV Cache）+ Continuous Batching（让GPU永远不闲着）+ 投机解码（降低单请求延迟）。这样一套组合拳下来，相对于FP16 + 标准Attention + 静态批处理的baseline，整体吞吐量可以提升5-10倍，延迟降低3-5倍。

但是——组合使用也有陷阱。有一次我在部署中同时开启了投机解码和Continuous Batching，发现加速效果远不如预期。原因是draft model的推理和target model的验证在调度上产生了冲突——Continuous Batching要求每个step所有请求同步推进，但投机解码每个请求的draft长度不同，打乱了同步性。解决办法是把投机解码和Continuous Batching的调度逻辑统一起来：让所有请求使用相同的draft长度K，或者在Continuous Batching的step中对每个请求独立管理其投机状态。vLLM已经支持这种统一调度，但配置时要注意参数配合。

踩坑案例：还是那个AI写作的团队。他们看到我们推荐的优化组合，一口气把所有技术全开了——FP8 + FlashAttention + PagedAttention + KV Cache量化 + Continuous Batching + 投机解码。结果系统变得极其不稳定：偶发的长延迟抖动从500ms涨到了8秒，还有内存泄漏。我们把问题定位到投机解码上——draft model也开启了FP8，但它的精度不够，生成的候选token质量太差，接受率从70%降到了35%。与此同时，Continuous Batching在等待投机验证完成时，waiting queue里的请求越积越多，形成了一层一层backpressure。最终的解决办法是：为draft model保留FP16，关闭draft model的KV Cache量化。系统立刻恢复了稳定。

推理优化是一个"系统工程"，不是开关一通就能躺赢的。每一项技术都有自己的前置条件和适用边界，而它们彼此之间的交互更加复杂。我的建议是：一步一步来，每次只加一个优化，监控好指标变化再决定下一步。最危险的事情就是一上来全开，出了问题根本不知道是哪一个环节导致的。

好了，第24章的内容就到这儿。这一章我们深入了推理优化的关键技术：从KV Cache的显存管理（PagedAttention、Prefix Cache），到投机解码的草稿-验证机制，再到改变attention计算模式的FlashAttention系列，最后是更高效的Continuous Batching调度和全景对比。你会发现，推理优化是一个"组合拳"——没有银弹，每一项技术都在特定的场景下发光。第25章「推理引擎与服务架构」，我们把视野再放大一些，看看这些优化技术是如何在vLLM、SGLang、TensorRT-LLM这些引擎中被实现的，以及如何设计一个生产级的推理服务架构。咱们下章见。
