第18章 网络数据采集

在前17章中，大部分例子都用的是现成的数据集——Zachary的空手道俱乐部、Les Miserables的人物关系网、网络上公开发布的AS级互联网拓扑。但到了实战阶段，你需要面对一个更现实的问题：我要分析的网络数据从哪来？

网络数据的获取和清洗是网络科学实战中最不性感但最重要的环节。很多时候，80%的时间花在了数据采集和清洗上，剩下20%才是分析和建模。这一章我们来系统地梳理网络数据的来源、采集方法和清洗流程——学会这些，你才算真正具备了独立开展网络分析的能力。

网络数据来源

如果你需要网络数据来完成课程作业、做实验验证算法、或者为分析报告寻找素材，下面几个公开数据集是起步的最佳选择。

SNAP（Stanford Network Analysis Project）是斯坦福大学维护的网络数据集宝库，收录了超过100个来自不同领域的大规模网络——社交网络（Facebook、Twitter）、引文网络、道路网络、协作网络等。格式统一，文档齐全，是网络科学领域引用最多的数据集。所有数据均可通过http://snap.stanford.edu/data/ 免费下载。

Konect（Koblenz Network Collection）由科布伦茨大学维护，收录了超过250个网络数据集，按领域分类（社交、生物、经济、通信等）。它还提供了一些其他平台上不常见的独特网络，比如维基百科的编辑网络、Stack Overflow的用户互动网络。

Network Repository是一个交互式的网络数据集和图分析平台，提供了超过3000个网络的可视化和交互式查询功能。在做大规模网络的算法测试时，你可以在线运行基本的网络分析，而无需将全部数据下载到本地。

下表总结了这三个主要数据源的特点：

数据源 数据集数量 数据格式 特色 适合场景
SNAP 100+ 边列表（.txt） 大规模网络，文档详细 学术研究、算法测试
Konect 250+ 边列表 + GEXF 领域覆盖面广 跨领域比较分析
Network Repository 3000+ 多种格式 + 在线查询 数量最多，交互式探索 算法基准测试

除了这些公开数据集，真实世界的网络分析往往需要你自己动手采集数据。下面介绍两种最常用的方法：API获取和网页爬虫。

社交媒体API

社交平台通常会提供应用程序编程接口（API），允许开发者获取用户在平台上的公开数据。这些API是构建社交网络数据的重要渠道。但在开始之前，请记住最重要的一点：遵守平台的使用条款和API限制。过快的请求频率会触发速率限制（rate limiting），过度抓取甚至可能导致你的账号被封禁。

Twitter/X API

Twitter（现X）的API是网络科学研究中最常用的数据来源之一。通过API可以获取用户的粉丝列表、关注列表、推文互动关系等——这些都可以用来构建和分析社交网络。

# 使用Tweepy库获取Twitter关注关系网络
import tweepy

# 使用v2 API端点（需要先申请Bearer Token）
client = tweepy.Client(bearer_token="YOUR_BEARER_TOKEN")

# 获取指定用户的粉丝和关注列表
user_id = 123456789  # 替换为目标用户的ID
followers = client.get_users_followers(id=user_id, max_results=100)
following = client.get_users_following(id=user_id, max_results=100)

# 构建用户关系网络（简化示例）
followers_ids = [u.id for u in followers.data] if followers.data else []
following_ids = [u.id for u in following.data] if following.data else []

# 对于每个粉丝，进一步获取其关注列表（BFS方式扩展）
# 注意：需要控制请求速率，避免触发rate limit
print(f"用户 {user_id} 有 {len(followers_ids)} 个粉丝和 {len(following_ids)} 个关注")
踩坑记录：Twitter API的速率限制非常严格——免费版的v2 API每15分钟只能获取500个粉丝。我有一次写了一个"爬取2000名用户社交图谱"的脚本，满心欢喜地跑了半个小时回来一看，只爬了200个用户就被限流了。解决方案是：实现一个智能的速率限制管理器，计算每次API调用的剩余配额，动态调整休眠时间。另外，一定要在代码里捕获HTTP 429错误（Too Many Requests），优雅地等待配额恢复。

GitHub API

GitHub的API对网络科学研究来说是一个宝藏。GitHub上的协作网络——谁关注了谁、谁给谁的仓库点星、谁在谁的代码上提PR——这些都是天然的社交网络结构。

# 使用PyGithub获取GitHub协作网络
from github import Github

g = Github("YOUR_GITHUB_TOKEN")

# 获取一个仓库的贡献者列表，构建Star关系网络
repo = g.get_repo("networkx/networkx")
stargazers = repo.get_stargazers().get_page(0)  # 第一页

# 对每个star用户，获取其关注的其他用户
# 这可以构建一个"开发者-关注"的有向图
for star_user in stargazers[:10]:  # 限制数量，避免耗时过长
    print(f"用户: {star_user.login}")
    try:
        following = star_user.get_following()
        for follow in following[:5]:
            print(f"  关注了: {follow.login}")
    except Exception as e:
        print(f"  获取关注列表失败: {e}")

GitHub API的一大优势是它的速率限制相对宽松——未经认证的请求每小时60次，使用Personal Access Token后提高到每小时5000次。对于大多数研究项目来说，这个配额已经非常充裕。

网页爬虫基础

不是所有网站都提供方便的API。有时候，你需要的数据只以HTML页面的形式存在。这时候，网页爬虫（web scraping）就是必要的技术手段。

在Python中，最常用的爬虫组合是requests + BeautifulSoup。前者负责下载网页内容，后者负责从HTML中提取数据。一套基本的爬虫流程如下：

import requests
from bs4 import BeautifulSoup
import time

# 步骤1：发送HTTP请求
url = "https://example.com/some-page"
headers = {
    "User-Agent": "Mozilla/5.0 (compatible; NetworkScienceCourse/1.0)"
}
response = requests.get(url, headers=headers)

if response.status_code == 200:
    # 步骤2：解析HTML
    soup = BeautifulSoup(response.text, "html.parser")

    # 步骤3：提取链接关系
    links = []
    for a_tag in soup.find_all("a", href=True):
        href = a_tag["href"]
        text = a_tag.get_text(strip=True)
        links.append((text, href))

    print(f"从页面中提取了 {len(links)} 个链接")

    # 步骤4：BFS方式爬取更深层的页面
    # 关键：设置合理的爬取间隔！
    time.sleep(1)  # 每次请求间隔1秒，避免对服务器造成压力
else:
    print(f"请求失败: HTTP {response.status_code}")
爬虫礼仪：不要做一个"野蛮"的爬虫。第一，检查robots.txt文件（在网站的根目录下），这个文件告诉爬虫哪些页面允许访问、哪些不允许。第二，在请求头中设置合理的User-Agent，让网站知道你是谁。第三，控制请求频率——在两次请求之间至少间隔1秒，最好能模拟人类浏览的速度。第四，如果网站提供API，优先使用API而不是爬虫。

在构建网络数据时，网页爬虫的一个典型用途是爬取维基百科的链接关系：从一篇文章出发，爬取它的所有超链接，再从这些链接指向的文章出发继续爬取——这就是一个典型的"网络爬虫 + 网络构建"复合任务。爬取完成后，你就得到了一个"维基百科文章引用网络"。

数据清洗

不管你从哪个来源获得数据，原始数据几乎总是"脏"的。数据清洗是网络科学实践中不可或缺的一步。常见的脏数据问题包括：

重复边。数据集合中可能存在多条相同的边（比如因为数据抓取过程中的多次记录）。重复边的处理很简单——在构建网络时去重即可。用pandas的drop_duplicates()或NetworkX的nx.Graph()默认行为（自动去重）都可以解决。

自环。在大部分网络分析场景下，自环（self-loop，节点指向自己的边）没有意义——一个人不会和自己构成社交关系。清理时直接移除。

孤立节点。没有任何连接的节点往往是被错误引入的，或者已经退出了系统的活跃部分。根据分析目标不同，可以选择保留或移除。

节点标识不一致。这是最棘手的问题之一。同一个实体在不同的数据源中可能使用了不同的ID（比如"John Smith"、"J. Smith"、"john.smith@email.com"都指向同一个人）。节点去重需要结合实体解析（entity resolution）技术，通常是一个独立的研究领域。

import pandas as pd
import networkx as nx

# 原始边列表（通常从API或爬虫获得）
raw_edges = pd.DataFrame({
    "source": ["A", "B", "A", "C", "A", "D"],
    "target": ["B", "A", "B", "D", "C", "A"],
    "weight": [1, 1, 1, 1, 1, 1]
})

# 步骤1：去重（对无向图，移除(A,B)和(B,A)的重复）
edges_dedup = raw_edges.copy()
edges_dedup["sorted_pair"] = edges_dedup.apply(
    lambda row: tuple(sorted([row["source"], row["target"]])), axis=1
)
edges_dedup = edges_dedup.drop_duplicates(subset="sorted_pair")

# 步骤2：移除自环
edges_clean = edges_dedup[edges_dedup["source"] != edges_dedup["target"]]

# 步骤3：移除空值
edges_clean = edges_clean.dropna(subset=["source", "target"])

print(f"原始边数: {len(raw_edges)}")
print(f"去重后边数: {len(edges_dedup)}")
print(f"清洗后边数: {len(edges_clean)}")

# 步骤4：清理节点列表
all_nodes = set(edges_clean["source"]) | set(edges_clean["target"])
print(f"节点总数: {len(all_nodes)}")

一张好的数据清洗检查清单：

检查项 检查方法 处理方式
重复边 统计 (source, target) 出现次数 保留第一条或合并权重
自环 查询 source == target 的行 直接删除
缺失值 检查 source/target 列的NaN 删除该行或尝试补全
孤立节点 计算每个节点的度 视情况保留或删除
度异常 查看度分布统计数据 检查数据采集过程是否出错

导入NetworkX

数据清洗完毕之后，最后一步就是将数据导入NetworkX（或其他图分析工具）进行分析。NetworkX支持多种数据格式的导入，我们介绍几种最常用的方式。

从边列表导入

边列表是最简单的网络数据格式——每一行代表一条边，通常包含源节点和目标节点，第三列可以包含权重。

import networkx as nx

# 从CSV格式的边列表导入
# 假设network.csv有两列: source, target
G = nx.read_edgelist("network.csv", delimiter=",", nodetype=str)

# 如果边列表包含权重
G_weighted = nx.read_weighted_edgelist("weighted_network.csv", delimiter=",")

# 从pandas DataFrame导入
import pandas as pd
df = pd.read_csv("edges.csv")
G_from_df = nx.from_pandas_edgelist(df, source="source", target="target",
                                     edge_attr="weight")

从邻接矩阵导入

如果你的数据是以邻接矩阵形式存在的（例如，行和列都是节点，单元格表示连接关系），可以直接导入为numpy数组或scipy稀疏矩阵：

import numpy as np

# 假设我们有一个3x3的邻接矩阵
adj_matrix = np.array([
    [0, 1, 1],
    [1, 0, 0],
    [1, 0, 0]
])

G = nx.from_numpy_array(adj_matrix)
# 或从scipy稀疏矩阵导入
# G = nx.from_scipy_sparse_array(sparse_matrix)

# 为节点添加标签
node_labels = {0: "Alice", 1: "Bob", 2: "Charlie"}
G = nx.relabel_nodes(G, node_labels)

从GraphML/GML导入

如果你从SNAP或Gephi等工具中导出了标准的图文件格式，NetworkX也支持直接读取：

# 读取GraphML格式（Gephi的默认导出格式之一）
G = nx.read_graphml("network.graphml")

# 读取GML格式
G = nx.read_gml("network.gml")
性能提示：当你处理大规模网络（节点数超过10万）时，使用nx.read_edgelist()而不是读取GraphML会快得多。GraphML虽然携带更丰富的节点/边属性信息，但文件解析的开销也更大。对于大数据集，我建议将节点属性和边结构分开存储：结构用边列表，属性用单独的CSV文件，然后在Python中通过DataFrame合并。

数据采集和清洗是网络科学中最"接地气"但也最容易被忽视的环节。很多人急于跳到分析建模阶段而轻视数据准备，结果后续所有的分析结果都可能因为数据质量问题而被质疑。记住一句老话：garbage in, garbage out。花在数据清洗上的时间永远值得。
