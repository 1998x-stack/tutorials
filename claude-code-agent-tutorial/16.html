第16章 实现权限控制 🔨

大家好，我是你们的讲师。上一章我们拆解了 Claude Code 的权限系统架构——七种模式、规则链引擎、ML 分类器、Bubble 机制。今天我们要进入 BUILD 模式，亲手实现一个权限控制模块，把它集成到我们一直在构建的 Agent 中。

为什么会有一个独立的"实现权限控制"章节？因为权限系统不是某种锦上添花的功能——它是 Agent 安全运行的地基。如果你跳过了权限控制，Agent 就会有"指哪打哪"的能力——你让它读个文件，它顺手把整个目录删了，权限系统是唯一能拦住它的东西。而且，权限系统的设计质量直接影响用户体验：做得太松不安全，做得太紧用户想摔键盘。

另外分享一个我的亲身经历。我之前参与过一个 Agent 项目，上线第一天就出了安全问题——一个测试同学在 prompt 里写了"help me clean up the project directory"，Agent 直接执行了 rm -rf .。因为当时没有权限控制，这个操作被即时执行了。幸好那只是个测试环境，但也够我们喝一壶的。那个下午，我写下了第一版 PermissionManager——就是从那时起，我深信权限系统应该是 Agent Harness 的第一优先级模块，而不是最后才想起来的"附加功能"。

权限系统的接口设计

我们先把接口定下来。一个好的接口应该让调用方在 5 秒钟内理解如何使用，同时具备足够的灵活性来应对不同的安全策略。

核心 API 只有一个方法：

class PermissionManager:
    def check(self, tool_name: str, params: dict) -> dict:
        \"\"\"
        检查工具调用是否被允许。

        返回:
            allowed: bool              — 是否允许执行
            reason: str                — 原因说明
            requires_confirmation: bool — 是否需要用户确认
        \"\"\"

你可能会问：为什么不做成返回 True/False 的简单接口？两个原因。第一，我们需要 reason 字段——当操作被拒绝时，Agent 需要知道原因，这样才能告知用户"为什么这个操作不被允许"。第二，requires_confirmation 字段实现了我们上一章讨论的"中间地带"——不是简单的允许/拒绝，而是"可以执行但需要你确认一下"。

接下来我们定义四种模式，覆盖从最安全到最高效的整个光谱：

模式 读取操作 写入操作 命令执行 适用场景
safe 自动允许 拒绝 拒绝 代码审查、文件浏览
normal 自动允许 需确认 需确认 日常开发，默认模式
dangerous 自动允许 自动允许 需确认 高强度编码，信任 Agent 编辑
paranoid 需确认 需确认 需确认 + 超时 审计、生产操作、高风险环境

注意 paranoid 模式甚至对读取操作都要确认——你可能觉得这有点过头，但在某些场景下（比如读取包含密钥的配置文件），这确实有道理。它还引入了超时机制：如果用户在一定时间内没有响应确认请求，操作自动被拒绝（而不是默认通过）。

工具白名单与黑名单

有了模式，我们还需要更精细的控制粒度——具体到单个工具层面。这就是白名单和黑名单发挥作用的地方。

白名单模式：只有列在清单中的工具可以被使用。任何不在白名单中的工具调用都会直接被拒绝。这种模式的默认行为是"拒绝一切"，安全等级最高。

黑名单模式：除了列在清单中的工具，其他都可以使用。这种模式的默认行为是"允许一切"，灵活性最高。

在我们的实现中，两种模式同时存在——白名单优先于黑名单。也就是说，一个工具只有在通过了白名单检查（如果在白名单模式下）且不在黑名单中时，才会继续进入下一步的权限评估。

白名单和黑名单的匹配使用 glob 模式——这比精确匹配灵活得多，又比正则表达式更容易理解。例如 Read* 匹配所有以 Read 开头的工具，Write:* 匹配 Write 命名空间下的所有子工具。

import fnmatch
from enum import Enum
from dataclasses import dataclass, field
from typing import Optional


class PermissionMode(Enum):
    SAFE = "safe"
    NORMAL = "normal"
    DANGEROUS = "dangerous"
    PARANOID = "paranoid"


@dataclass
class ToolPolicy:
    """单个工具的安全策略配置"""
    default_action: str = "ask"         # allow / deny / ask
    danger_level: str = "low"           # low / medium / high / critical
    requires_confirmation: bool = True


# 工具安全分类
TOOL_SAFETY_MAP = {
    "Read": ToolPolicy("allow", "low", False),
    "Grep": ToolPolicy("allow", "low", False),
    "ReadFile": ToolPolicy("allow", "low", False),
    "ListDir": ToolPolicy("allow", "low", False),
    "TodoWrite": ToolPolicy("allow", "low", False),
    "TodoList": ToolPolicy("allow", "low", False),
    "WebSearch": ToolPolicy("ask", "medium", True),
    "WebFetch": ToolPolicy("ask", "medium", True),
    "Write": ToolPolicy("ask", "medium", True),
    "Edit": ToolPolicy("ask", "medium", True),
    "WriteFile": ToolPolicy("ask", "high", True),
    "Bash": ToolPolicy("ask", "high", True),
    "RunCommand": ToolPolicy("ask", "high", True),
    "DeleteFile": ToolPolicy("deny", "critical", True),
    "BatchWrite": ToolPolicy("deny", "critical", True),
}


class PermissionManager:
    def __init__(self, mode: PermissionMode = PermissionMode.NORMAL):
        self.mode = mode
        self._whitelist: list[str] = []
        self._blacklist: list[str] = []
        self._configure_for_mode(mode)

    def _configure_for_mode(self, mode: PermissionMode):
        """根据模式设置默认策略"""
        if mode == PermissionMode.SAFE:
            self._whitelist = ["Read*", "Grep*", "Todo*", "ListDir"]
            self._blacklist = ["*"]
        elif mode == PermissionMode.NORMAL:
            self._whitelist = []       # 允许全部（通过逐项检查控制）
            self._blacklist = ["Delete*", "Batch*"]
        elif mode == PermissionMode.DANGEROUS:
            self._whitelist = []
            self._blacklist = ["Delete*", "Batch*"]
        elif mode == PermissionMode.PARANOID:
            self._whitelist = ["Read*", "Grep*", "ListDir"]
            self._blacklist = ["Bash*", "Write*", "Edit*", "Delete*"]

    def check(self, tool_name: str, params: dict) -> dict:
        # 白名单检查：如果有白名单，工具必须在白名单中
        if self._whitelist:
            if not any(fnmatch.fnmatch(tool_name, p) for p in self._whitelist):
                return {
                    "allowed": False,
                    "reason": f"工具 {tool_name} 不在白名单中",
                    "requires_confirmation": False,
                }

        # 黑名单检查：如果工具在黑名单中，拒绝
        if self._blacklist:
            if any(fnmatch.fnmatch(tool_name, p) for p in self._blacklist):
                return {
                    "allowed": False,
                    "reason": f"工具 {tool_name} 在黑名单中",
                    "requires_confirmation": False,
                }

        # 工具安全策略检查
        policy = TOOL_SAFETY_MAP.get(tool_name)
        if policy is None:
            # 未注册的工具：paranoid 模式拒绝，其他模式询问
            if self.mode == PermissionMode.PARANOID:
                return {"allowed": False, "reason": "未知工具",
                        "requires_confirmation": False}
            return {"allowed": True, "reason": "未注册工具，按默认策略处理",
                    "requires_confirmation": True}

        if policy.default_action == "deny":
            return {"allowed": False, "reason": f"{tool_name} 默认不允许",
                    "requires_confirmation": False}

        if policy.default_action == "allow":
            if self.mode == PermissionMode.PARANOID and\
               policy.danger_level != "low":
                return {"allowed": True, "reason": "paranoid 模式下需确认",
                        "requires_confirmation": True}
            return {"allowed": True, "reason": "自动允许",
                    "requires_confirmation": False}

        # default_action == "ask"
        return {"allowed": True, "reason": f"需要用户确认",
                "requires_confirmation": True}

    def set_whitelist(self, patterns: list[str]):
        self._whitelist = patterns

    def set_blacklist(self, patterns: list[str]):
        self._blacklist = patterns

用户确认回调实现

当权限检查返回 requires_confirmation=True 时，我们需要在工具执行前插入一个确认步骤。这是典型的 PreToolUse Hook 模式——在工具执行前，调用一个钩子函数来决定是否继续。

我们的实现包括三种确认选项：

  • y — 确认执行本次操作
  • n — 拒绝本次操作
  • q — 退出当前任务（拒绝所有后续操作）

还有一个重要的细节：超时机制。在 paranoid 模式下，如果用户在 30 秒内没有响应，操作自动被拒绝。这不只是为了用户体验——更重要的是防止 Agent 在用户离开时静默执行危险操作。

import sys
import select
import time


class ConfirmationCallback:
    def __init__(self, timeout: int = 30):
        self.timeout = timeout          # 超时秒数，0 表示不超时
        self.session_trusted: set = set()  # 当前会话中已信任的工具
        self.always_allow_session = False

    def confirm(self, tool_name: str, params: dict, reason: str) -> bool:
        if self.always_allow_session:
            return True

        # 会话级信任：同一个工具同一类操作不再重复确认
        trust_key = f"{tool_name}:{type(params)}"
        if trust_key in self.session_trusted:
            return True

        # 显示确认信息
        print(f"\n⚠️  需要确认操作:")
        print(f"  工具: {tool_name}")
        print(f"  参数: {params}")
        print(f"  原因: {reason}")
        print(f"  [y] 允许  [n] 拒绝  [q] 退出任务  [a] 会话中全部允许")
        print(f"  ({self.timeout}s 超时后自动拒绝)", end=" ")

        # 带超时的输入读取
        start = time.time()
        while True:
            elapsed = time.time() - start
            remaining = self.timeout - int(elapsed)
            if self.timeout > 0 and remaining <= 0:
                print("\n  ⏱ 超时，操作已拒绝")
                return False

            try:
                if sys.stdin in select.select([sys.stdin], [], [], 0.1)[0]:
                    choice = sys.stdin.readline().strip().lower()
                    if choice == "y":
                        return True
                    elif choice == "n":
                        return False
                    elif choice == "q":
                        sys.exit("用户退出任务")
                    elif choice == "a":
                        self.always_allow_session = True
                        return True
                    else:
                        print(f"  输入错误，请选择 y/n/q/a: ", end="")
            except (EOFError, KeyboardInterrupt):
                print("\n  输入中断，操作已拒绝")
                return False

危险操作检测

除了工具的权限级别，我们还需要对操作参数进行深度检查——尤其是在 Bash 命令这个高危领域。即使权限系统允许 Bash 执行，某些命令模式也应该被自动拦截。

这就是危险操作检测层。它使用正则表达式匹配已知的危险模式，并检查路径是否在安全边界内。

import re
import os


class DangerousOperationDetector:
    """危险操作检测——正则匹配 + 路径检查"""

    # 危险命令模式
    DANGEROUS_PATTERNS = [
        (r'\brm\s+-rf\s+/\s', '高危: 尝试删除根目录'),
        (r'\brm\s+-rf\s+/\*', '高危: 尝试删除根目录下所有文件'),
        (r'\bsudo\s+', '高危: 使用 sudo 提权'),
        (r'\bchmod\s+777\b', '高危: 将文件权限设为 777'),
        (r'\bchown\b', '高危: 修改文件所有者'),
        (r'\bmount\b', '高危: 挂载操作'),
        (r'>\s*/dev/sd[a-z]', '高危: 直接写入块设备'),
        (r'\bdd\s+if=', '高危: dd 写入磁盘'),
        (r'\bcurl\s+.*\|\s*(ba|sh)\b', '高危: curl 管道执行 shell'),
        (r'\bwget\s+.*-O-\s*\|\s*(ba|sh)\b', '高危: wget 管道执行 shell'),
        (r'\b(?:mkfs|fdisk|parted)\b', '高危: 磁盘分区/格式化操作'),
        (r'`[^`]*rm\s+-rf[^`]*`', '高危: 反引号中嵌入 rm -rf'),
        (r'\$\s*\([^)]*rm\s+-rf[^)]*\)', '高危: 命令替换中嵌入 rm -rf'),
    ]

    # 限制写入的路径前缀
    PROTECTED_PATHS = [
        '/etc',
        '/usr',
        '/bin',
        '/sbin',
        '/boot',
        '/dev',
        '/proc',
        '/sys',
        # Windows 路径也可以加，但以 Linux 为主
    ]

    def __init__(self, project_root: str = "."):
        self.project_root = os.path.abspath(project_root)

    def check_command(self, command: str) -> Optional[dict]:
        """检查 Bash 命令是否包含危险操作"""
        for pattern, reason in self.DANGEROUS_PATTERNS:
            if re.search(pattern, command):
                return {"allowed": False, "reason": reason}
        return None

    def check_path(self, path: str) -> Optional[dict]:
        """检查路径是否在安全范围内"""
        abs_path = os.path.abspath(path)

        # 检查是否在项目目录内
        if abs_path.startswith(self.project_root):
            return None

        # /tmp 是允许的
        if abs_path.startswith('/tmp'):
            return None

        # 检查是否在受保护路径中
        for protected in self.PROTECTED_PATHS:
            if abs_path.startswith(protected):
                return {"allowed": False,
                        "reason": f"写入受保护路径: {protected}"}

        # 项目目录之外的其他路径需要警告
        return {"allowed": True,
                "reason": "目标路径在项目目录之外，需要确认",
                "requires_confirmation": True}

    def check(self, tool_name: str, params: dict) -> Optional[dict]:
        """综合检查——根据工具类型调用对应的检查"""
        if tool_name in ("Bash", "RunCommand"):
            cmd = params.get("command", params.get("cmd", ""))
            return self.check_command(cmd)

        if tool_name in ("Write", "WriteFile", "Edit", "DeleteFile"):
            path = params.get("path", params.get("file_path", ""))
            return self.check_path(path)

        return None

集成到 Agent

有了 PermissionManager 和 DangerousOperationDetector，我们把它们集成到 Agent 的主循环中。集成点只有一个：在工具执行之前调用权限检查。

class AgentWithPermissions:
    def __init__(self, mode: PermissionMode = PermissionMode.NORMAL):
        self.permission = PermissionManager(mode)
        self.detector = DangerousOperationDetector(project_root="/home/user/project")
        self.confirmer = ConfirmationCallback(timeout=30)

    def execute_tool(self, tool_name: str, params: dict) -> str:
        # 第1步：检查危险操作
        danger_result = self.detector.check(tool_name, params)
        if danger_result and not danger_result.get("allowed", True):
            return f"❌ 危险操作被拦截: {danger_result['reason']}"

        # 第2步：权限检查
        perm_result = self.permission.check(tool_name, params)
        if not perm_result["allowed"]:
            return f"❌ 权限被拒绝: {perm_result['reason']}"

        # 第3步：需要用户确认
        if perm_result.get("requires_confirmation", False):
            confirmed = self.confirmer.confirm(
                tool_name, params, perm_result["reason"]
            )
            if not confirmed:
                return f"❌ 用户取消了操作"

        # 第4步：执行工具
        return f"✅ 执行 {tool_name}({params})"

在 normal 模式下实际运行的效果：

  • 用户说"读取 main.py" → Agent 调用 Read("main.py") → 自动允许，直接执行
  • 用户说"修改 config.py 中的端口号" → Agent 调用 Edit("config.py") → 弹确认框，用户确认后执行
  • 用户说"删掉临时目录" → Agent 调用 Bash("rm -rf /tmp/cache") → 危险检测通过（在 /tmp 内），但 Bash 需要确认 → 弹确认框
  • 用户说"清理系统" → Agent 调用 Bash("rm -rf /var/log") → 危险检测拦截（/var 是受保护路径），拒绝执行
渐进信任的最佳实践：你可能注意到 ConfirmationCallback 支持"会话级信任"——当你对同一个工具确认了多次后，可以自动放行。这解决了 Permission Fatigue 问题。实践中，我们建议这样配置渐进信任策略：前 3 次同类操作需要确认 → 之后自动放行 → 但如果操作参数包含从未见过的模式（比如 Bash 中出现了一个新的危险命令），重新要求确认。这种"信任但验证"的模式，在安全性和效率之间取得了很好的平衡。

至此，我们拥有了一个功能完整的权限控制系统。它支持四种安全模式、工具白名单/黑名单、逐工具安全策略、危险操作检测、以及带超时的用户确认回调。下一章我们将把目光转向所有工具中最危险的那个——Bash 沙箱。我们会深入讨论如何在 Agent 中安全地执行 Shell 命令，以及 Claude Code 的四层纵深防御体系是如何设计的。
