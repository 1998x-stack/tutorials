第二十七章：多模态大模型

各位同学，到今天为止我们一直在跟文本打交道。分词、词向量、BERT、GPT……所有输入都是文字。但人类感知世界靠的是五感——我们看图片、听声音、读文字，大脑天然就是"多模态"的。

你有没有想过一个问题：给ChatGPT发一张照片，它说"我看不到图片，我是一个文本模型"——但在2026年的今天，GPT-4o、Claude 3.5、Gemini这些模型不仅能"看懂"图片，还能"听懂"语音，甚至能理解视频帧。NLP的边界被彻底打破了，或者说，NLP已经进化成了"多模态大模型"。

这一章我们来看看：视觉、语音、文本这三种模态是如何融合到同一个模型中的。这是整个NLP教程的倒数第二章，也是最"烧脑"的一章——因为我们要跳出纯文本的舒适区，看看更大的世界。

27.1 CLIP：打通视觉与语言的桥梁

多模态学习领域的第一个里程碑是OpenAI在2021年发布的CLIP（Contrastive Language-Image Pre-training）。在CLIP之前，计算机视觉模型只能用标注好的类别名做分类（比如ImageNet的1000个类别），既死板又昂贵——标注百万张图片的成本是天文数字。

CLIP的核心理念极其优雅：用自然语言作为监督信号来训练视觉模型。它不再说"这张图属于'狗'这个类别"，而是说"这张图和'一只金毛叼着飞盘'这句话是匹配的"。这样训练出来的模型，你告诉它任何一个类别名（不需要是训练时见过的），它都能判断图片是否匹配。

具体做法是双塔结构：一个文本编码器（Text Encoder）和一个图像编码器（Image Encoder），分别把文本和图像映射到同一个向量空间。训练目标是最简单的对比学习（Contrastive Learning）——在同一条样本里，让匹配的图文对在向量空间中靠近，不匹配的相互远离：

import torch
import torch.nn.functional as F
from transformers import CLIPProcessor, CLIPModel
from PIL import Image
import requests

# 加载 CLIP 模型和处理器
model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-base-patch32")

# 准备图片和候选文本
url = "http://images.cocodataset.org/val2017/000000039769.jpg"
image = Image.open(requests.get(url, stream=True).raw)
texts = ["两只猫在沙发上睡觉", "一只狗在公园奔跑", "雨天的城市街景"]

# 预处理
inputs = processor(text=texts, images=image, return_tensors="pt", padding=True)

# 前向推理
outputs = model(**inputs)

# 计算相似度（logits_per_image 是图像与各文本之间的相似度分数）
logits_per_image = outputs.logits_per_image  # shape: [1, 3]
probs = logits_per_image.softmax(dim=-1)

for text, prob in zip(texts, probs[0]):
    print(f"「{text}」: {prob.item():.2%}")
# 输出接近： 「两只猫在沙发上睡觉」: 99.87%

这段代码只有十几行，却展示了CLIP最强大的能力——zero-shot图像分类。你不需要为每个新任务重新训练模型，只需要提供候选文本描述，CLIP就能告诉你哪个描述最匹配图片。

CLIP的突破意义在于：它第一次证明了对比学习可以大规模对齐视觉和语言两个模态。训练数据是互联网上抓取的4亿图文对，不需要人工标注。训练好之后，CLIP的视觉编码器成了无数后续工作的"视觉主干"——包括下面要讲的LLaVA。

实践要点：使用CLIP时，文本模板（prompt template）对结果影响很大。不要在CLIP里直接输入"猫"，而要写成"一只猫的照片"、"a photo of a cat"——因为它的训练数据中图片描述都是完整句子。这个细节我在项目里踩过坑，不加模板直接输入类别名，准确率掉了将近15个百分点。

27.2 视觉语言模型：教会LLM"看"世界

CLIP只能说"这个图片匹配哪个文本描述"，但它不会生成图文混合的内容。2023年，LLaVA（Large Language and Vision Assistant）的出现开启了一个新时代——视觉指令微调（Visual Instruction Tuning）。

LLaVA的思路很直接：把CLIP的视觉编码器拿过来，加一个简单的线性投影层（Projection Layer），把图像特征映射到语言模型的输入空间，然后端到端微调。这样就形成了一个"看得见"的大语言模型。

整个结构可以分为三块：

  • 视觉编码器（Vision Encoder）：使用CLIP的ViT-L/14，负责把图片变成特征向量。
  • 投影矩阵（Projection Matrix）：一个简单的线性层，把视觉特征映射到LLM的词嵌入空间。
  • LLM主干（Language Model）：使用Vicuna（基于LLaMA微调的对话模型），负责理解图文混合输入并生成回复。

训练数据是LLaVA团队用GPT-4生成的"图文对话"数据——给GPT-4看图片的caption和bounding box信息，让它假装"看到了图片"，然后生成各种问答对。这种用大模型生成训练数据来训练小模型的做法，在当时是非常巧妙的。

一个典型的LLaVA推理流程：

# 伪代码：LLaVA 推理流程
image_features = vision_encoder(image)          # 视觉编码
image_embeds = projection_layer(image_features) # 映射到文本空间

# 构造多模态输入序列：[IMG] + image_embeds + [人类问题]
input_embeds = [word_embeds("<Image>"), image_embeds, word_embeds(user_query)]

# LLM 自回归生成回复
output = llm.generate(input_embeds, max_length=512)

LLaVA成功后，视觉语言模型（VLM）领域迅速爆发：Qwen-VL、InternVL、CogVLM、LLaVA-NeXT……每个模型都在某些维度上做了改进。到2025-2026年，VLM已经能够理解复杂的图表、识别手写公式、阅读小票上的模糊文字，甚至能指出图中物体之间的空间关系。

27.3 语音转文本：Whisper与多语言ASR

说完"看"，我们再说"听"。2022年9月，OpenAI发布了Whisper——一个开源的多语种语音识别（ASR）模型，支持99种语言的语音转文字。

Whisper的特殊之处不仅仅是支持语言多，而是它的训练方式：它把多语种ASR当作一个序列到序列的任务来处理，输入是80通道的log-Mel频谱图（80×3000的矩阵），输出是文本token序列。模型结构是标准的Encoder-Decoder Transformer，但Encoder处理的是二维频谱（类似ViT处理图片的方式），Decoder自回归生成文本。

Whisper有五种尺寸（tiny/base/small/medium/large），large版本有1.55B参数。它的一个巧妙设计是多任务输出格式：输出的文本token序列中包含了任务标记，比如"<|en|>"表示输出语言是英语，"<|transcribe|>"表示做语音转文字任务。这样同一个模型既能做语音识别，又能做语音翻译（直接输出另一种语言的文字）。

Whisper的int8量化版在CPU上跑medium模型延迟不到1秒，很适合做本地语音转录。我自己的播客笔记就是用Whisper local跑的，不用上传任何音频到云端，隐私友好。

多模态设计模式：你有没有注意到CLIP和Whisper的共同点？都是把非文本信号（图像像素/声音频谱）编码成向量，然后与文本模态对齐。CLIP用对比学习做对齐，Whisper用交叉熵做翻译式对齐。殊途同归——所有多模态模型的核心问题都是"怎么把不同模态的表征映射到同一个语义空间"。

27.4 多模态融合方法

前面的CLIP和Whisper都是双模态（文本+图片 或 文本+语音），那如果我们想把文本、图像、语音三个模态都融合到一个模型里呢？这就涉及到多模态融合的核心问题了。

目前主流的融合策略有三种：

早期融合（Early Fusion）

在输入端就把不同模态的特征拼在一起。比如把文本token embedding和图像patch embedding拼接成一个长序列，然后一起输入Transformer处理。优点是模态之间的交互在每一层都能发生，缺点是序列长度暴涨，计算量大。Gemini和GPT-4o在视觉理解上采用了类似策略。

晚期融合（Late Fusion）

每个模态各自独立编码到最后一层，再把最后的表征拼接或加权融合。双塔模型（如CLIP的双编码器）就属于晚期融合。优点是各模态可以独立训练、灵活性高，缺点是模态间的交互只在最后一层发生，容易丢失细粒度对齐信息。

交叉注意力融合（Cross-Attention Fusion）

在Transformer的中间层引入交叉注意力模块，让一个模态的表征去"关注"另一个模态的表征。Florence-2和Flamingo采用了这种方式。交叉注意力的好处是在计算量和交互深度之间取得了很好的平衡，是目前工业界最流行的方案。

融合策略代表模型优点缺点
早期融合Gemini, GPT-4o模态交互充分，对齐精度高序列长，算力需求大
晚期融合CLIP, ImageBind结构简单，可独立训练各编码器模态交互弱，细粒度任务吃力
交叉注意力Flamingo, Florence-2平衡交互与效率，扩展性好实现复杂，训练需要多阶段

值得一提的是Meta的ImageBind——它不做显式融合，而是用"绑定模态"的思路：不直接学习A和C的关系，而是通过A→B和B→C来间接对齐。比如通过"音频"这个中介来对齐"图像"和"文本"，因为图像-音频对比图像-文本对更容易获得。

27.5 通向全模态之路

多模态大模型的终极目标是"全模态"（Omnimodal）——一个模型同时处理文本、图像、音频、视频、代码、3D点云、触觉信号……就像人类一样。

2024-2026年，这个方向有了实质性进展：

  • GPT-4o：原生多模态训练，可以同时输入文本+图像+音频，输出文本或语音。延迟低到可以实时对话。
  • Gemini 2.0：原生多模态输出，不仅能"看懂"视频帧，还能生成图像和音频。
  • Meta Spirit LM：一个统一的多模态语言模型，文本和语音共享同一套tokenizer和模型参数。
  • ImageBind-LLM：用ImageBind作为"对齐枢纽"，把六种模态（图像、文本、音频、深度、热成像、IMU数据）统一到一个LLM中。

这些模型有一个共同的趋势：从"拼接式"多模态走向"原生式"全模态。早期的MLLM是"视觉编码器+LLM"的拼凑，视觉部分和语言部分本质上还是两个模型、两套表征。而GPT-4o和Gemini是在训练的一开始就让所有模态的数据流经同一个Transformer——这才是真正的"全模态"。

下面这张表对比了三个具有代表性的多模态模型，帮你建立直观印象：

可以看到，从CLIP到LLaVA到GPT-4o，多模态模型沿着"双塔拼接原生端到端"的路径快速演化。如果2023年多模态还只是"视觉+文本"的简单拼接，那么到2026年，"语音+视觉+文本"的三模态原生模型已经走进日常生活。

踩坑故事：多模态模型的应用陷阱

讲一个我自己的惨痛经历。2024年我给一个智能客服项目做多模态升级——用户拍了产品照片发过来，系统自动识别产品并回答售后问题。我们选了当时最好的开源VLM，跑了几百张测试图，准确率93%，果断上线。

结果上线第三天就出事了：一张用户拍的模糊产品说明书，系统不仅识别错了型号，还自信满满地给出了错误的安装步骤。用户照着做了，差点把设备弄坏。

事后复盘发现三个问题：第一，我们的测试集全是清晰的、光照好的图片，而真实用户拍的照片五花八门——逆光、手抖、遮挡；第二，VLM在面对模糊输入时不会说"我不确定"，而是自信地"幻觉"；第三，没有设置置信度阈值，模型输出95%的置信度其实不可信。

教训是什么？多模态模型在"不确定"时比纯文本模型更容易产生严重幻觉——因为它多了一个模态的输入，错误会在视觉编码→投影→文本生成三个环节中层层放大。给多模态模型做产品化时，一定要加一个"拒答"机制：当视觉特征模糊或文本生成置信度低时，直接说"抱歉，我无法准确识别"。

本章小结

多模态大模型的发展可以用一句话概括：从各自为政到统一表征。CLIP证明了视觉和文本可以在对比学习中对齐，Whisper证明了语音频谱可以像翻译一样被Transformer处理，LLaVA证明了LLM可以被"教会"看图片，而GPT-4o证明了上述所有可以在同一个原生模型中实现。

对于NLP从业者来说，多模态意味着两个重要转变：第一，你的模型不再只能读文字，还能"看"和"听"了；第二，多模态评估比纯文本评估复杂得多——一张模糊图片、一段嘈杂录音、一个有误导性的文图组合，都可能让模型输出灾难性结果。
