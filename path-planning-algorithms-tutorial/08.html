大家好！上一章我们深入剖析了 A* 算法，理解了它如何通过启发式函数 f(n)=g(n)+h(n) 实现比 Dijkstra 更高效的搜索。但 A* 只是一个起点——在实际工程中，我们需要根据具体场景对 A* 进行各种改造。这一章我们来看看 A* 的四大实用变体：加权 A*、双向 A*、ARA* 和分层 A*。每个变体都在某个维度上突破了经典 A* 的限制。

加权 A*（WA*）——用最优性换速度

经典 A* 保证找到最短路径。但有时候，"足够好"的路径就已经够用了——尤其是在实时性要求极高的场景中。想象一下一个在仓库中穿梭的机器人：它需要在 50 毫秒内做出决策，此时花 200 毫秒去找那条短 3% 的路径毫无意义。

加权 A*（Weighted A*，简称 WA*）的思路极其简单：给启发式函数 h(n) 乘上一个权重 w > 1。

def weighted_astar(start, goal, grid, w=1.5):
    open_set = [(0, 0, start)]  # (f, g, node)
    g_score = {start: 0}
    came_from = {}

    while open_set:
        _, current_g, current = heapq.heappop(open_set)

        if current == goal:
            return reconstruct_path(came_from, current)

        for neighbor, step_cost in get_neighbors(current, grid):
            tentative_g = g_score[current] + step_cost
            if tentative_g < g_score.get(neighbor, float('inf')):
                came_from[neighbor] = current
                g_score[neighbor] = tentative_g
                f = tentative_g + w * heuristic(neighbor, goal)
                heapq.heappush(open_set, (f, tentative_g, neighbor))

    return None

你看，改动只有一行：f = tentative_g + w * heuristic(neighbor, goal)。但这一行的影响是巨大的。

当 w = 1 时，WA* 退化为标准 A*，保证最优性。随着 w 增大，算法越来越"贪心"——它更倾向于朝着目标方向搜索，减少不必要的分支探索。但代价是：找到的路径可能不是最优的。理论上有界次优性（bounded suboptimality）：WA* 找到的路径代价不超过最优路径代价的 w 倍。

w 值 展开节点数（相对 A*） 路径代价（相对最优） 适合场景
1.0 100% 100%（最优） 需要严格最优解
1.2 ~65% ~101% 导航、仓储机器人
1.5 ~40% ~104% 游戏 AI 寻路
2.0 ~25% ~108% 实时策略游戏（RTS）
5.0 ~10% ~125% 极其紧迫的时间预算
真实故事：我曾经给一个仓储机器人项目调参，团队一心想用标准 A* 保证最优路径。结果机器人在路口常常"思考"超过 100 毫秒，导致后方机器人急刹车，整个仓库的吞吐量上不去。我们把 w 设到 1.2 后，规划时间降到了 30 毫秒以内，路径只长了 1.8%。这个 trade-off 太值了——运维同事说 "这 2% 的电费，我们出得起"。记住：在工程中，"足够好且足够快"往往胜过"完美但慢吞吞"。

双向 A*——两头并进

双向 A*（Bidirectional A*）的思路很直观：从起点和终点同时向中间搜索，两个搜索前沿在中间相遇。想象两个人分别从隧道两端向中间挖掘——显然比一个人从一端挖到另一端快得多。

前置条件

双向搜索要成功，两个搜索前沿必须满足"相遇"条件。经典做法是：正向搜索从起点向终点扩展，反向搜索从终点向起点扩展。当某个节点被双方都访问到时，就找到了路径。

但这里有一个陷阱：启发式函数必须一致。正向用 h(n, goal)，反向用 h(n, start)。如果启发式不满足可采纳性（admissible），双向 A* 可能比单向还要慢——因为双方都在错误的方向上浪费计算。

什么时候用？

双向 A* 在以下场景表现优异：

  • 起点和终点距离很远——搜索空间巨大，双向几乎把搜索半径减半（圆的面积是 πr²，两个半圆加起来才 2π(r/2)² = πr²/2，仅为一半）
  • 启发式函数信息量充足——比如欧几里得距离在开放空间中
  • 地图是对称的——正向和反向搜索难度相当

但在迷宫中，双向 A* 的优势会大打折扣——因为反向搜索同样会走入死胡同。

Anytime Repairing A*（ARA*）——先凑合，再优化

现实世界中，规划器往往面临一个矛盾：你既想要快速得到一个可用路径，又想在时间允许时获得更优解。这就是 ARA*（Anytime Repairing A*）的出现背景。

ARA* 的思路：先用一个较大的 w 快速得到一个次优路径，然后随着计算时间的推移逐步降低 w，利用之前的搜索结果来改进路径。它"随时"（anytime）都能给你一个答案，但等待越久，答案越好。

ARA* 的核心机制：

  • 初始使用高权重（如 w=5）快速搜索
  • 记录搜索过程中的所有"不一致"状态（inconsistent states）
  • 降低权重后，只需要修复这些不一致状态的代价估计，不需要重新搜索整个图
  • 重复直到 w=1（最优）或时间耗尽

想象一个自动驾驶汽车在十字路口前需要决策：它必须在 50ms 内给出一个方向（走哪条车道），但可以用接下来的 200ms 优化具体轨迹。ARA* 完美适配这类"截止时间驱动"的场景。

分层 A*——先粗后细

分层 A*（Hierarchical A*）解决的是大规模地图搜索的效率问题。核心思想：不要在一张高分辨率地图上直接搜索，而是先构建一个低分辨率的抽象图，在抽象层找到粗略路径，再在低层细化。

HAA*（Hierarchical Annotated A*）

具体做法是把原始栅格地图分块聚合。一个 1000x1000 的栅格地图有 100 万个节点。如果我们将每个 10x10 的块聚合成一个"区域节点"，则抽象图只有 1 万个节点。在抽象层上跑 A* 几乎是瞬时的，然后再在抽象路径经过的区域内做细粒度搜索。

在游戏 AI 中，分层 A* 是标配。想想《文明》这种大地图策略游戏，单位需要在几千个格子的地图上寻路——如果不做分层，每帧几千个单位的寻路计算会把 CPU 吃光。实际上，几乎所有现代游戏引擎（如 Unreal Engine 的 NavMesh）都采用了分层思想。

踩坑提醒：分层 A* 的致命问题是"局部最优陷阱"。如果抽象层的路径选择了区域 A 到区域 B，但进入区域 A 后发现实际上最优路径需要经过区域 C——这被称为"边界效应"。解决方法是让相邻区域之间有重叠（overlap），或者在细化阶段允许一定程度的跨区域调整。不给抽象层留冗余，就相当于让一个近视眼摘了眼镜找路——大方向也许对，但细节处处是坑。

这四种 A* 变体各有擅长场景：WA* 适合时间敏感的场景，双向 A* 适合远距离规划，ARA* 适合有时间截止压力的环境，分层 A* 适合超大规模地图。在实际系统中，它们经常被组合使用——比如在自动驾驶中，全局规划用分层 A*，局部规划用 WA* 保证实时性。

下一章我们将进入另一个维度的 A* 变体——D* Lite 增量搜索。当环境发生变化时（比如突然出现障碍物），A* 需要从头重新搜索，而 D* Lite 能够高效地"复用"之前的搜索结果，只重新计算受影响的部分。这在火星车探索未知地形时可是保命级的特性。
