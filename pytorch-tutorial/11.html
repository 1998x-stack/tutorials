第11章 模型保存与加载

大家好，我是你们的讲师。

到目前为止，我们训练的所有模型都在内存里——一旦程序退出，模型参数就灰飞烟灭了。训练一个模型少则几小时多则数天，如果每次都得从头开始，那这个框架就没法用了。模型保存和加载看起来是件小事，但一旦出了问题，损失可能是灾难性的。

2019 年，我参加 Kaggle 的一个比赛，训练了整整 72 小时，模型在验证集上达到了当时的最优成绩。我随手用 torch.save(model, 'model.pth') 保存了模型，然后心满意足地去睡觉。第二天起来想继续训练——发现模型加载后完全没办法接着训练，因为优化器的状态和 epoch 信息都没保存。更要命的是，我保存模型的那个环境有 GPU，但我当天想在一台新机器上做推断，那台机器只有 CPU——结果一加载就报 CUDA 错误。这个故事告诉我们：模型保存不是存个 .pth 文件那么简单。这一章我们来系统性地解决这些问题。

11.1 state_dict() vs 完整模型保存

PyTorch 提供了两种保存方式，适用场景完全不同：

# 方式一：保存完整模型（不推荐）
torch.save(model, 'model_complete.pth')
model = torch.load('model_complete.pth')

# 方式二：保存 state_dict（推荐）
torch.save(model.state_dict(), 'model_state.pth')
model.load_state_dict(torch.load('model_state.pth'))
维度 完整模型保存 state_dict 保存
存储内容 模型结构 + 参数 + 训练配置 仅参数字典（层名 → 权重张量）
文件大小 略大（包含序列化的类定义） 较小（仅权重）
加载方式 一行代码，无需定义模型类 需要先实例化模型再 load_state_dict
安全性 潜在 pickle 漏洞（任意代码执行） 较安全（仅加载张量数据）
可移植性 差（PyTorch 版本、代码结构变化会出错） 好（只要模型定义兼容即可）
推荐场景 快速原型、临时实验 生产部署、长期保存、分享给他人
永远不要在正式项目中使用完整模型保存。完整模型使用 Python 的 pickle 序列化，它保存的是 Python class 的引用路径。一旦你修改了模型代码（改了类名、换了目录），之前保存的权重就无法加载了。更严重的是，pickle 格式存在安全风险——恶意的 .pth 文件可以在你加载时执行任意代码（我们稍后会讲到 weights_only=True）。

11.2 检查点（Checkpoint）设计模式

在正式的训练项目中，一个 checkpoint 应该包含所有需要恢复训练的信息。标准模式是把所有状态打包成一个字典：

import torch

def save_checkpoint(model, optimizer, scheduler, epoch, loss, filepath):
    checkpoint = {
        'epoch': epoch,
        'model_state_dict': model.state_dict(),
        'optimizer_state_dict': optimizer.state_dict(),
        'scheduler_state_dict': scheduler.state_dict(),
        'loss': loss,
        # 可选：保存随机状态以便复现
        'rng_state': torch.get_rng_state(),
    }
    torch.save(checkpoint, filepath)
    print(f"Checkpoint saved at epoch {epoch}, loss={loss:.4f}")

加载时，你可以从 checkpoint 中恢复全部训练状态：

def load_checkpoint(filepath, model, optimizer, scheduler):
    checkpoint = torch.load(filepath)
    model.load_state_dict(checkpoint['model_state_dict'])
    optimizer.load_state_dict(checkpoint['optimizer_state_dict'])
    scheduler.load_state_dict(checkpoint['scheduler_state_dict'])
    start_epoch = checkpoint['epoch'] + 1  # 从下一 epoch 继续
    best_loss = checkpoint['loss']
    return start_epoch, best_loss

这个模式的好处：你只需要一个文件就包含了"快照"时刻的所有状态。而且不管训练中断多少次，都能接上。在实际项目中，我通常还会保留最近 3 个 checkpoint 和最佳的一个：

def save_with_rotation(model, optimizer, scheduler, epoch, loss, base_dir):
    # 总是保存最新的 checkpoint
    save_checkpoint(model, optimizer, scheduler, epoch, loss,
                    f"{base_dir}/checkpoint_latest.pt")
    # 保存最佳 checkpoint
    if loss < best_loss:
        save_checkpoint(model, optimizer, scheduler, epoch, loss,
                        f"{base_dir}/checkpoint_best.pt")
    # 每 N 个 epoch 保存一次（历史快照）
    if epoch % 10 == 0:
        save_checkpoint(model, optimizer, scheduler, epoch, loss,
                        f"{base_dir}/checkpoint_epoch_{epoch:03d}.pt")

11.3 map_location：跨设备加载

上面的"踩坑"故事里，那个在 CPU 机器上加载 GPU 模型的惨痛经历，实际上有一个标准解决方案——map_location 参数：

# GPU 上训练 → CPU 上加载
checkpoint = torch.load('checkpoint.pt', map_location='cpu')
model.load_state_dict(checkpoint['model_state_dict'])

# CPU 上训练 → GPU 上加载
checkpoint = torch.load('checkpoint.pt', map_location='cuda:0')

# GPU 0 上训练 → GPU 1 上加载
checkpoint = torch.load('checkpoint.pt', map_location='cuda:1')

# 更灵活的方式：手动映射
checkpoint = torch.load('checkpoint.pt',
    map_location=lambda storage, loc: storage.cuda(1))
保存设备 加载设备 map_location 值 说明
GPU (cuda:0) CPU 'cpu' 最常见的场景：生产部署机器没有 GPU
CPU GPU (cuda:0) 'cuda:0' 先在小规模 CPU 上训练，再迁移到 GPU 推理
GPU (cuda:0) GPU (cuda:1) 'cuda:1' 多卡环境下的设备重映射
GPU (cuda:0) MPS (Apple Silicon) 'mps' 跨硬件平台加载（需 PyTorch 2.0+）

11.4 weights_only=True 与安全

2023 年，安全研究人员发现了一个令人不安的事实：恶意的 .pth 文件可以利用 pickle 的任意代码执行漏洞，在加载模型时植入病毒。攻击者可以训练一个完全正常的模型，但在保存时通过特殊的 pickle 操作插入恶意代码。你加载这个模型时，恶意代码就会被执行。

幸运的是，PyTorch 从 2.1 版本开始引入了 weights_only=True 参数：

# 安全加载（推荐）
checkpoint = torch.load('model.pth', weights_only=True)

# 从不可信的来源加载时，务必使用这个参数
# 如果是自己训练的模型，且环境可信，可以不用
Security 最佳实践：
  • 只加载自己训练的或来自可信来源的模型文件
  • 加载第三方模型时始终使用 weights_only=True
  • 不要在公开环境中轻易使用 torch.load('unknown_model.pth')
  • Hugging Face Hub 上大多数模型使用 safetensors 格式（天生安全）

11.5 版本兼容性陷阱

PyTorch 不同版本之间保存的权重文件可能不兼容。这是一个容易被忽视的问题：

  • PyTorch 1.x → 2.x：state_dict 格式兼容，但序列化格式有变化。直接加载通常没问题，但复杂场景（如自定义 C++ 扩展、TorchScript）可能出错。
  • Python 版本差异：pickle 序列化在 Python 3.8 和 3.12 之间基本兼容，但自定义类的 __getstate__/__setstate__ 行为可能不同。
  • CUDA 版本差异：同一个 PyTorch 版本但不同 CUDA 编译版本加载，通常没有问题。但如果 GPU 架构差异太大（如 V100 → H100），某些 CUDA 扩展可能需要重新编译。
版本兼容的最佳实践：
  • 在保存 checkpoint 时，把 PyTorch 版本号也作为元数据存进去
  • 长期保存的模型建议定期使用最新 PyTorch 版本重新保存一次
  • 如果分发模型给他人，用 torch.save(model.state_dict(), ...) 而非完整模型

这一章我们学习了模型保存与加载的核心知识：state_dict 和完整保存的优劣对比、checkpoint 的标准设计模式、跨设备加载的 map_location 方案、weights_only 的安全保护，以及版本兼容性问题。掌握了这些，你就能安全地在不同环境间迁移训练任务，也能自信地和他人分享你的模型权重。下一章我们来看训练技巧——学习率调度、初始化策略、梯度裁剪等实战调参经验。
