第17章 上下文窗口扩展

想象一个场景：你把一本300页的小说输入给LLM，让它总结整本书的情节脉络——但模型告诉你它只能看到前10页的内容。这就是LLaMA 1时代用户的日常。在2023年初，LLaMA 1的上下文窗口只有2048个tokens，约等于1500个汉字或半页英文文档的篇幅。对于"总结整本书"这样的任务来说，这点上下文连一本书的序言都塞不下。

时至今日，LLaMA 3.1支持了128K tokens的上下文窗口——整整64倍的增长。从2K到128K的跨越，不只是简单的"加长"那么简单。它涉及注意力机制的计算复杂度优化、位置编码的扩展能力、训练策略的调整，以及长文本场景下的评估方法等一系列挑战。这一章我们就来逐层拆解LLaMA上下文扩展背后的技术。

17.1 上下文窗口的瓶颈

为什么扩展上下文窗口这么难？直接原因有两个：注意力机制的计算复杂度和显存消耗。

标准Transformer中，自注意力（self-attention）的计算复杂度是 O(seq² × d)，其中 seq 是序列长度，d 是隐藏维度。当序列长度翻倍时，注意力计算量翻4倍。这还不算最要命的——真正限制序列长度的是KV Cache。推理时，每个Transformer层需要为每个已生成的token缓存Key和Value向量。对于LLaMA 70B（80层、d=8192、8个KV头），KV Cache的公式是：

# KV Cache显存占用估算
# 70B模型推理时的KV Cache计算

layers = 80
kv_heads = 8             # GQA中的KV头数
d_head = 128             # 每个注意力头的维度
dtype_bytes = 2          # BF16 = 2字节

# 每token每层KV Cache大小
per_token_per_layer = kv_heads * d_head * 2 * dtype_bytes
# = 8 * 128 * 2 * 2 = 4096 bytes = 4 KB

# 总KV Cache (seq_len = 4096)
total_kv_cache = layers * per_token_per_layer * seq_len
# = 80 * 4KB * 4096 = 1.31 GB

# 总KV Cache (seq_len = 128K)
total_kv_cache = 80 * 4KB * 131072 = 40.96 GB

看到了吗？在128K上下文下，70B模型仅KV Cache就需要41GB显存——比模型权重本身（140GB的BF16版）的1/4还多。这也解释了为什么GQA分组查询注意力对长上下文如此重要：GQA通过减少KV头的数量，将KV Cache的线性系数从全注意力（full attention）的H个头降低到GQA的g个头，通常g=8而H=64，节省8倍显存。

核心矛盾： LLM的推理需要KV Cache来避免重复计算，但KV Cache的大小与序列长度成正比。序列越长，KV Cache越大，最终会吃掉所有显存。这就好比你有一个不断写入的记事本——页面越多，记事本就越厚，直到你的口袋装不下。

17.2 LLaMA三代上下文扩展历程

LLaMA的上下文窗口扩展不是一蹴而就的。每一代LLaMA都在上下文长度上取得了实质性进展：

版本 上下文长度 所用技术 训练数据 备注
LLaMA 1 2,048 RoPE（基础实现） 1.0T tokens RoPE论文本身支持外推，但实际效果有限
LLaMA 2 4,096 RoPE + GQA 2.0T tokens GQA减少了KV Cache，为更长上下文留出空间
LLaMA 3 8,192 RoPE改进 + 训练数据调整 15.6T tokens Base模型8K，经过继续训练可扩展
LLaMA 3.1 131,072 NTK-aware缩放 + 局部注意力 ~800B tokens（长文本继续训练） 128K上下文实际效果可用

注意到一个有趣的现象：LLaMA 1到LLaMA 2，上下文只翻了一倍（2K→4K）。但在LLaMA 3到LLaMA 3.1，上下文翻了16倍（8K→128K）。为什么后面可以加速扩展？因为前面几年的积累——RoPE的改进、GQA的引入、更好的训练策略——为长上下文提供了基础。

17.3 位置编码扩展方法

RoPE（Rotary Position Embedding）本身是支持位置外推（extrapolation）的——理论上它可以通过旋转矩阵的自然周期性来处理比训练时更长的序列。但实践表明，直接外推的效果很差。原因在于训练时学到的频率分布和未见过位置上的频率产生了"不一致"。为了解决这个问题，学术界提出了多种基于RoPE的扩展方法。

PI（Positional Interpolation）

PI是2023年6月由Meta提出的一种最直观的方法：既然模型没见过大于2048的位置，那就把所有位置索引"压缩"到模型见过的范围内。比如要把上下文从2K扩展到8K，PI的做法是将位置索引乘以 2048/8192 = 0.25，让新的位置 0..8191 被压缩到 0..2047 的范围内。

PI的实现非常简单——只需要修改RoPE计算的输入位置索引：

# Positional Interpolation 的核心实现
# 将位置索引 linear 压缩到训练范围内

def apply_rotary_pos_emb_pi(x, position_ids, scale=4.0):
    """
    x: 输入张量 [batch, seq_len, num_heads, d_head]
    position_ids: 原始位置索引 [batch, seq_len]
    scale: 扩展倍数 (新长度 / 训练长度)
    """
    # 关键步骤：将位置索引除以缩放因子
    scaled_pos = position_ids.float() / scale

    # 其余和标准RoPE一样
    sin, cos = precompute_freqs_cis(d_head, max_len)
    return x * cos[scaled_pos] + rotate_half(x) * sin[scaled_pos]

PI的好处是简单、稳定，经过少量微调就能适应很长的上下文。但它的缺点也很明显——线性压缩导致位置之间的区分度降低（尤其是在短距离上），模型对局部位置关系的感知能力被削弱。这就像你在1:100万的地图上找两个相距1厘米的点，但这两个点实际相距10公里——地图太小，细节丢失了。

NTK-aware扩展

NTK-aware扩展（又称NTK-aware Scaled RoPE）来自社区（Reddit用户bloc97等），它基于神经正切核（Neural Tangent Kernel）理论的洞见：RoPE中的不同频率维度应该有不同的缩放系数。

RoPE是通过一组不同频率的正弦和余弦波来编码位置的。高频波（周期短）负责编码局部位置关系，低频波（周期长）负责编码全局位置关系。高频波在训练时见过足够多的数据，外推能力更强；而低频波更容易受到长度变化的影响。

NTK-aware扩展的策略是：对低频维度做更强的插值，对高频维度做较轻的插值甚至保留原始频率。具体实现是通过修改RoPE的频率计算公式——引入一个温度系数（temperature）来调整不同维度的缩放程度：

# NTK-aware Scaled RoPE 的频率计算
# 高频维度几乎不变，低频维度被压缩

def ntk_aware_frequencies(d_head, base=10000.0, scale=4.0):
    """
    计算NTK-aware缩放的频率
    d_head: 每个注意力头的维度
    scale: 目标扩展倍数
    """
    # 标准RoPE频率
    freqs = 1.0 / (base ** (torch.arange(0, d_head, 2) / d_head))

    # NTK-aware调整：对每个维度应用不同的缩放
    # 高频维度 (i接近0) 几乎保持不变
    # 低频维度 (i接近d_head/2) 被压缩
    scaled_freqs = freqs * (scale ** (torch.arange(0, d_head, 2) / d_head))

    return scaled_freqs

NTK-aware扩展的优点是：模型在短上下文任务上的表现几乎不受影响，同时在长上下文任务上比PI好得多。因为它保留了局部位置的精细区分能力。

YaRN（Yet another RoPE extensioN method）

YaRN是目前LLaMA 3.1长上下文扩展的核心方法。它在NTK-aware的基础上进一步做了两件事：

  • 更精确的频率缩放： 引入一个"波长"的概念——对于每个RoPE维度，计算它在原始序列长度下的波长（即该维度的正弦波完成一个完整周期的token数）。如果波长小于序列长度，说明模型在这一维上"见过完整的周期"，外推能力好，可以不做插值。如果波长超过了序列长度，说明模型在这一维上没见到完整周期，需要插值。
  • 注意力温度调节（temperature tuning）： 在长上下文中，注意力分布的熵会发生变化——softmax后的概率分布变得更扁平（因为Q和K的累积误差导致注意力分数分散）。YaRN通过调整softmax的温度（乘以一个小于1的系数）来恢复注意力的尖锐性。
YaRN在LLaMA上的效果： 在LLaMA 2 7B模型上使用YaRN进行上下文扩展（2K→8K），只需要在8000个长文本序列上做约400 step的微调，就能在长上下文任务上实现接近甚至超过完整训练的效果。相比之下，直接从头的完全训练需要多两个数量级的数据。这就是"连续训练"（continued training）的威力。

17.4 长上下文训练策略

位置编码的改进只是上下文扩展的一个方面。另一个同等重要的方面是训练策略——如何让模型在超长序列上稳定训练而不发生OOM或训练崩溃。

LLaMA 3.1训练128K上下文时采用了以下策略：

  • 渐进式长度增长（Progressive Length Scaling）：先从8K的上下文开始训练，然后逐步扩展到32K、64K、128K。每个阶段持续几百到几千步。这样做的好处是避免了"从短直接跳到超长"导致的训练不稳定。
  • 局部/全局注意力混合：每个Transformer层采用不同的注意力范围。前面的层使用全局注意力（关注所有token），后面的层使用滑动窗口注意力（只关注附近4096个token）。这大幅降低了计算量。
  • FlashAttention-2/3：使用分块计算的FlashAttention算法，避免一次性计算完整的注意力矩阵，将显存复杂度从O(seq²)降低到O(seq)。对于128K序列来说，这是必要的——否则光是注意力矩阵就需要128K×128K×2字节≈32GB显存，远超H100的80GB上限。
# FlashAttention的显存优势 (128K序列, 70B模型)
#
# 标准注意力:
#   注意力分数矩阵 = Q @ K^T  形状 [8, 128K, 128K]
#   显存 = 8 × 128K × 128K × 2B = 256 GB ✗ (OOM)
#
# FlashAttention:
#   分块计算，每块大小 128×128
#   显存 = 8 × 128 × 128 × 2B ≈ 256 KB ✓

17.5 LongContext评估

上下文长度扩展了，但模型真的"用得上"这些长上下文吗？这要问两个问题：第一，模型能否从长上下文中"检索"到相关信息？第二，模型能否利用长上下文中的信息来改善推理？

Needle-in-a-Haystack 测试是目前最主流的评估指标之一。这个测试的实验设计非常简单但巧妙：在文档中的某个随机位置插入一句无关的陈述（比如"旧金山的气温是73华氏度"），然后向模型提问"旧金山气温是多少？"。如果模型能成功回答，说明它在长上下文中仍然具备最基本的信息检索能力。

模型 4K 8K 16K 32K 64K 128K
LLaMA 2 70B 99.8% 65.2% 31.5% — — —
LLaMA 3 70B (8K训练) 99.9% 98.5% 70.1% 38.2% 12.4% —
LLaMA 3.1 70B (128K训练) 99.9% 99.8% 99.5% 98.8% 97.2% 93.5%
LLaMA 3.1 405B 100% 99.9% 99.9% 99.6% 98.9% 97.1%

从这张表可以看出，只有经过专门的长上下文训练，模型才能在128K上下文中保持90%以上的检索准确率。未经长上下文训练但使用了位置扩展技术的模型（如LLaMA 3 70B加NTK-aware），在32K以下效果尚可，但超过32K后准确率急剧下降。

一个重要的发现： "大海捞针"测试只测试了信息检索能力，并没有测试多步推理能力。一个模型可能在128K上下文中准确回忆起一句话的位置，但无法利用长上下文中的多个信息点来完成复杂推理。这就是"大海捞针"测试的局限性——它给的结果往往比模型的实际长上下文能力更乐观。LLaMA 3.1论文中特别强调，他们在长上下文场景中还测试了多文档QA、摘要、代码库理解等更复杂的任务。

17.6 踩坑故事：位置编码崩溃

最后分享两个我在长上下文实践中遇到（或见证）的真实痛苦经历。

故事一：超出边界的灾难

使用PI扩展LLaMA 2上下文时，有一个很隐蔽的问题：当某段文本恰好出现在训练时的最大长度（4K）边界附近时，模型可能完全崩溃。具体表现是——在4K附近的token位置上，模型生成的内容开始出现语义断裂、重复、或者突然切换到另一个话题。这是因为边界附近的RoPE频率已经发生了"变形"（插值比例最大），模型学习到的位置编码分布在这些位置上出现了严重的偏差。

解决方法其实很简单：不要锚定在边界上。在微调长上下文时，应该让训练数据的长度分布尽可能均匀——既有短的（几百tokens），也有接近最大长度的。这样模型在每个长度区间都有充分的训练。

故事二："我忘记了"综合征

另一个更令人沮丧的场景是：模型在128K上下文的前半段表现出色，但从中间的某个位置开始，它突然"忘记"了之前的信息。调查发现，原因是KV Cache的精度损失——当序列非常长时，远距离token的Key/Value经过多层传播后，浮点精度误差逐渐累积，最终导致模型无法区分"这是一个合理的查询"和"这是一段无关的噪声"。

这个问题的缓解方案是使用更高的KV Cache精度（FP16→BF16，甚至FP32），或者在注意力计算中引入上下文压缩（context compression）——对远距离的历史信息做摘要式的压缩表示，而不是保留完整的token级KV Cache。

关键教训： 长上下文并非"把序列加长"那么简单。每一个token位置上的位置编码、注意力计算、KV Cache存储都在考验着模型的数学设计极限。当你在生产环境中使用128K上下文时，务必进行充分的压力测试——包括在极限长度上的检索准确率、推理速度衰减、以及生成质量的一致性。不要被NIAH测试的高分迷惑。

上下文窗口扩展技术还在飞速发展。从位置编码的改进到注意力机制的优化，从训练策略到推理加速，每个方向都在不断涌现新成果。对于LLaMA来说，128K远不是终点——学术界已经在探索百万级甚至无限长度的上下文了。而在下一章中，我们会转头关注另一个同样重要的话题：如何科学地评估LLaMA模型的能力。没有好的评估，再长的上下文也只是空谈。
