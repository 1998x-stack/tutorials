第5章 配置系统：settings与命令行

大家好，我是你们的讲师。上一章我们走了一遍 main() 函数的启动流程，其中一半的代码都在做一件事：解析配置。这一章我们就来深入剖析 Memcached 的配置系统——从 struct settings 的结构体定义，到命令行参数的解析逻辑，再到 -o 扩展选项的设计思路。

你可能觉得"配置系统有什么好看的？不就是解析一下参数吗？"——其实不然。Memcached 的配置系统有几个很好的设计点：一是所有配置集中在一个结构体中，清晰易查；二是 growth_factor 这样的参数直接影响内存布局，错误的配置可能导致严重的性能灾难；三是 -o 扩展选项的设计给了新特性一个"不破坏现有命令行接口"的接入方式。好，我们从结构体开始。

struct settings：30+ 字段全解

struct settings 定义在 memcached.h 中，是整个系统配置的中枢。下面我把核心字段按功能分类逐一说明：

分类 字段名 默认值 说明
内存 maxbytes 64 * 1024 * 1024 最大可用内存（字节），对应 -m
chunk_size 48 最小 chunk 大小（字节），对应 -n
growth_factor 1.25 相邻 slab class 之间的尺寸增长因子，对应 -f
网络 port 11211 TCP 监听端口，对应 -p
udpport 11211 UDP 监听端口，对应 -U
backlog 1024 listen 的 backlog 大小
并发 maxconns 1024 最大并发连接数，对应 -c
num_threads 4 worker 线程数，对应 -t（建议 <= CPU 核数）
maxconns_fast 0 超过最大连接后是否立即关闭新连接
淘汰 evict_to_free 1 内存不足时是否允许淘汰旧数据
lru_crawler 0 是否启用 LRU 爬虫线程
lru_maintainer 0 是否启用 LRU 维护线程（热点/冷热分离）
日志 verbose 0 日志级别（0=无, 1=-v, 2=-vv, 3=-vvv）
log_file NULL 日志文件路径，对应 -o log_file=
logger_watcher_buf_size 512 * 1024 日志 watcher 缓冲区大小
扩展 ext_page_size 0 extstore page 大小（启用 extstore 后生效）
ext_wbuf_size 0 extstore 写缓冲区大小
slab_automove 0 是否自动进行 slab rebalance
安全 sasl 0 是否启用 SASL 认证
enable_seccomp 0 是否启用 seccomp 沙箱

这个结构体在 1.6.41 版本中一共有 30+ 个字段。我没有全部罗列在上面——有些是遗留配置（如 binding_protocol），有些是内部维护用的计数器，读者不需要全部记住。重点记住内存、并发、淘汰三组字段即可，它们是对系统行为影响最大的。

growth_factor 的数学推导

growth_factor（简称 factor）是 Memcached 最重要的配置之一，没有"之一"。它决定了各个 slab class 的 chunk 大小如何增长。假设最小 chunk 为 C0 = 48（settings.chunk_size），增长因子为 f = 1.25，则第 i 个 slab class 的 chunk 大小为：

chunk_size[i] = C0 * f^i    (i = 0, 1, 2, ..., 63)

代入数值:
class 0: 48
class 1: 48 * 1.25 = 60
class 2: 48 * 1.25^2 = 75
class 3: 48 * 1.25^3 = 93
...
class 63: 48 * 1.25^63 ≈ 137,438,953  → 约 131MB

这里的关键问题是：当存储一个 80 字节的 Item 时，它会被分配到哪个 class？

Memcached 的查找算法是 slabs_clsid()：

unsigned int slabs_clsid(const size_t size) {
    int res = POWER_SLAB_START;  // 通常为 1（跳过 class 0）
    // 从 class 1 开始查找，找到第一个 chunk_size >= size 的 class
    if (size == 0) return 0;
    while (size > slabclass[res].size) {
        if (res++ == power_largest) return 0;
    }
    return res;
}

80 字节的 Item 会落到 class 2（chunk_size = 93），但实际只用了 80 字节，浪费了 13 字节（约 14%）。这就是内部碎片——每个 chunk 内部的未使用空间。不同的 growth_factor 会直接影响内部碎片的比例：

growth_factor slab class 数量 最大 chunk（字节） 最坏情况内部碎片 适用场景
1.10 63 ~1.6M ~9% Value 大小差异小
1.25 48 ~131M ~20% 通用推荐（默认）
1.50 32 ~24G ~33% Value 大小差异很大
2.00 24 ~402G ~50% 不推荐（碎片太多）

growth_factor = 1.25 是官方推荐的默认值：它在 slab class 数量（覆盖能力）和内部碎片（内存利用率）之间取得了良好的平衡。如果你的业务中 value 大小分布非常集中（比如大多在 100-200 字节），可以尝试降低 factor 来减少碎片。反之，如果 value 从几十字节到几十 KB 都有，保持默认即可。

踩坑实录：我曾经见过一个运维同事把 growth_factor 设成了 1.01——他认为"越精细的内存分配越省内存"。结果呢？slab class 数量直接达到了上限 63 个，每个 class 能分到的 page 数量极少，导致频繁的 slab rebalance，性能反而下降了 30%。而且由于 class 太多，LRU 链表被切割得过于细碎，热点数据命中率也大幅下降。growth_factor 不是越小越好，它是一个需要综合考虑的工程参数。

-o 扩展选项

随着 Memcached 功能不断增多，简单的单字母参数已经不够用了。-o 选项支持用逗号分隔的 key=value 形式来传递扩展配置：

memcached -o "modern,lru_crawler,slab_automove=2,ext_page_size=65536"

-o 选项的解析逻辑在 main() 中的 subopts 分支：

case 'o':
    subopts = optarg;
    while (*subopts != '\0') {
        switch (getsubopt(&subopts, tokens, &value)) {
            case SUBOPT_MODERN:
                settings.modern = true;
                break;
            case SUBOPT_SLAB_AUTOMOVE:
                settings.slab_automove = atoi(value);
                break;
            case SUBOPT_LRU_CRAWLER:
                settings.lru_crawler = true;
                break;
            case SUBOPT_EXT_PAGE_SIZE:
                settings.ext_page_size = atoi(value);
                break;
            // ...
        }
    }
    break;

常用的 -o 扩展选项包括：

  • modern —— 启用现代模式（同时开启 LRU maintainer、slab automove 等推荐功能，是一组"一键优化"配置）
  • slab_automove=N —— 设置 slab 自动迁移的策略（0=关闭, 1=温和, 2=积极）
  • lru_crawler —— 启用 LRU 爬虫
  • lru_maintainer —— 启用 LRU 维护线程
  • log_file=PATH —— 日志文件路径
  • config_file=PATH —— 配置文件路径
  • ext_page_size=N —— extstore page 大小

配置文件与环境变量

除了命令行参数，Memcached 还支持从配置文件中读取设置。配置文件路径默认为 /etc/memcached.conf，可以通过 -o config_file= 指定：

# /etc/memcached.conf 示例
-m 2048
-p 11211
-t 8
-c 4096
-o modern,lru_crawler,slab_automove=1

配置文件的解析逻辑在 save_line() 函数中——它会读取文件的每一行，忽略以 # 开头的注释行，然后将每行内容当作命令行参数来解析。这意味着配置文件中的语法和命令行完全一致。

此外，1.6 版本还增加了环境变量的支持。所有配置项都可以通过 MEMCACHED_* 环境变量来覆盖：

export MEMCACHED_PORT=11211
export MEMCACHED_MAX_MEMORY=2048
export MEMCACHED_THREADS=8
memcached

环境变量的优先级最低：命令行参数 > 配置文件 > 环境变量 > 默认值。这个优先级设计是合理的——环境变量适合在容器化场景（Docker/K8s）中注入配置，命令行参数适合手动调试。

源码位置提示：环境变量解析的代码位于 memcached.c 的 main() 函数中，在 settings_init() 之后、getopt 之前。它的实现方式是调用 getenv() 逐个检查环境变量，如果存在则覆盖 settings 中的对应字段。代码非常直白，建议你自己看一下。

配置系统的设计启示

回顾 Memcached 的配置系统，有几点设计思路值得学习：

  • 单一事实来源：所有配置集中在一个结构体中，没有分散在各模块的自定义配置。这让代码的可维护性大大提高——想看某个配置的作用，只需查 struct settings 的定义和它的引用点。
  • 分层覆盖：默认值 → 环境变量 → 配置文件 → 命令行参数，形成一个清晰的优先级链路。每一层只覆盖不需要默认值的字段，保持配置的"增量"性质。
  • 向后兼容：-o 扩展选项的引入使得 Memcached 可以在不破坏现有命令行接口的前提下，不断添加新功能。老用户继续用 -m -p 不会受影响，新用户可以用 -o modern 一键启用推荐配置。
