从"Rust不适合做Web"到"真香"

提起Rust的Web生态，很多人还停留在五年前的印象："Rust做Web？框架不成熟，生态不完善，别折腾了。"如果你也这么想，那2026年的Rust Web生态会让你大吃一惊。

我印象最深的一个转折点是2023年，我所在的公司要把一个高吞吐的消息推送服务从Go重写到Rust。当时的Go实现已经碰到瓶颈：单个节点的goroutine调度开销虽然低，但在每秒10万级别的消息路由场景下，GC暂停的毛刺（latency spike）已经到了不可接受的程度。我们用Axum + Tokio重写后，P99延迟从120ms降到了8ms，资源占用减少了40%。从那以后，我开始对Rust做Web这件事彻底改观。

这一章，我们搭建一个完整的RESTful Todo API。选这个场景是因为它的业务逻辑简单（CRUD），但足以覆盖一个生产级Web后端的所有关键环节：路由、数据库、中间件、错误处理、测试和部署。

Axum的杀手锏：Axum是Tokio团队打造的Web框架，从2022年发布以来迅速成为Rust最受欢迎的Web框架。它的核心优势在于——利用Rust的trait系统和tower中间件生态，实现了"类型安全的路由定义"。也就是说，如果路由和处理函数签名不匹配，编译阶段就报错——这在Go和Node.js中是不可能做到的。

项目结构

我们采用分层架构，将路由、处理器、数据模型和数据库操作分开：

todo-api/
├── Cargo.toml
├── migrations/
│   └── 20240101000001_create_todos.sql
├── src/
│   ├── main.rs        # 入口：启动服务器、组装路由
│   ├── router.rs      # 路由定义
│   ├── handler/       # HTTP处理器
│   │   ├── mod.rs
│   │   └── todo.rs
│   ├── model/         # 数据模型
│   │   ├── mod.rs
│   │   └── todo.rs
│   └── db/            # 数据库操作
│       ├── mod.rs
│       └── todo.rs

这种分层的好处是职责清晰：handler处理HTTP请求/响应，model定义数据结构，db封装SQL操作。任何一层的变化都不会"侧漏"到其他层。

数据库模型

我们先定义SQL迁移文件migrations/20240101000001_create_todos.sql：

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

CREATE TABLE todos (
    id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    title       VARCHAR(255) NOT NULL,
    completed   BOOLEAN NOT NULL DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_todos_created_at ON todos(created_at DESC);

在src/model/todo.rs中定义Rust数据结构：

use chrono::{DateTime, Utc};
use serde::{Deserialize, Serialize};
use sqlx::FromRow;
use uuid::Uuid;

/// 数据库行映射（包含所有字段）
#[derive(Debug, Serialize, Deserialize, FromRow)]
pub struct Todo {
    pub id: Uuid,
    pub title: String,
    pub completed: bool,
    pub created_at: DateTime<Utc>,
    pub updated_at: DateTime<Utc>,
}

/// 创建请求（不含ID和时间戳）
#[derive(Debug, Deserialize)]
pub struct CreateTodoRequest {
    pub title: String,
}

/// 更新请求（所有字段可选）
#[derive(Debug, Deserialize)]
pub struct UpdateTodoRequest {
    pub title: Option<String>,
    pub completed: Option<bool>,
}

这里sqlx::FromRow自动将查询结果映射为Todo结构体——只要字段名和数据库列名匹配即可，无需手写映射代码。

数据库操作层

src/db/todo.rs封装所有SQL操作，使用sqlx::query_as在编译时检查SQL语法的正确性：

use sqlx::PgPool;
use uuid::Uuid;
use crate::model::todo::{Todo, CreateTodoRequest, UpdateTodoRequest};

pub async fn list_todos(pool: &PgPool) -> anyhow::Result<Vec<Todo>> {
    let todos = sqlx::query_as::<_, Todo>(
        "SELECT id, title, completed, created_at, updated_at FROM todos ORDER BY created_at DESC"
    )
    .fetch_all(pool)
    .await?;
    Ok(todos)
}

pub async fn create_todo(pool: &PgPool, req: &CreateTodoRequest) -> anyhow::Result<Todo> {
    let todo = sqlx::query_as::<_, Todo>(
        "INSERT INTO todos (title) VALUES ($1) RETURNING id, title, completed, created_at, updated_at"
    )
    .bind(&req.title)
    .fetch_one(pool)
    .await?;
    Ok(todo)
}

pub async fn update_todo(
    pool: &PgPool,
    id: Uuid,
    req: &UpdateTodoRequest,
) -> anyhow::Result<Option<Todo>> {
    let todo = sqlx::query_as::<_, Todo>(
        "UPDATE todos SET
            title = COALESCE($1, title),
            completed = COALESCE($2, completed),
            updated_at = NOW()
         WHERE id = $3
         RETURNING id, title, completed, created_at, updated_at"
    )
    .bind(&req.title)
    .bind(req.completed)
    .bind(id)
    .fetch_optional(pool)
    .await?;
    Ok(todo)
}

pub async fn delete_todo(pool: &PgPool, id: Uuid) -> anyhow::Result<bool> {
    let result = sqlx::query("DELETE FROM todos WHERE id = $1")
        .bind(id)
        .execute(pool)
        .await?;
    Ok(result.rows_affected() > 0)
}
连接池大小的考量：很多人第一次写Rust Web API时直接用PgPoolOptions::new().max_connections(10)硬编码一个值。实际生产中的经验公式是：连接数 ≈ (CPU核心数 × 2) + 有效磁盘数。对于PostgreSQL，过大的连接池反而会降低性能——因为PG为每个连接分配了独立的进程和内存。一般来说，4-8个连接对大多数Web应用已经足够。用sqlx::PgPoolOptions::new().max_connections(5)作为起点是合理的。

API端点

我们在src/handler/todo.rs中定义HTTP处理器：

use axum::{
    extract::{Path, State},
    http::StatusCode,
    Json,
};
use uuid::Uuid;
use crate::db;
use crate::model::todo::{CreateTodoRequest, UpdateTodoRequest, Todo};
use crate::AppState;

/// GET /todos — 列出所有任务
pub async fn list_todos(
    State(state): State<AppState>,
) -> Result<Json<Vec<Todo>>, (StatusCode, String)> {
    let todos = db::todo::list_todos(&state.pool).await
        .map_err(|e| (StatusCode::INTERNAL_SERVER_ERROR, e.to_string()))?;
    Ok(Json(todos))
}

/// POST /todos — 创建新任务
pub async fn create_todo(
    State(state): State<AppState>,
    Json(payload): Json<CreateTodoRequest>,
) -> Result<(StatusCode, Json<Todo>), (StatusCode, String)> {
    if payload.title.trim().is_empty() {
        return Err((StatusCode::BAD_REQUEST, "标题不能为空".into()));
    }
    let todo = db::todo::create_todo(&state.pool, &payload).await
        .map_err(|e| (StatusCode::INTERNAL_SERVER_ERROR, e.to_string()))?;
    Ok((StatusCode::CREATED, Json(todo)))
}

/// PUT /todos/:id — 更新任务
pub async fn update_todo(
    State(state): State<AppState>,
    Path(id): Path<Uuid>,
    Json(payload): Json<UpdateTodoRequest>,
) -> Result<Json<Todo>, (StatusCode, String)> {
    let todo = db::todo::update_todo(&state.pool, id, &payload).await
        .map_err(|e| (StatusCode::INTERNAL_SERVER_ERROR, e.to_string()))?;
    todo.ok_or((StatusCode::NOT_FOUND, format!("任务 {} 不存在", id)))
}

/// DELETE /todos/:id — 删除任务
pub async fn delete_todo(
    State(state): State<AppState>,
    Path(id): Path<Uuid>,
) -> Result<StatusCode, (StatusCode, String)> {
    let deleted = db::todo::delete_todo(&state.pool, id).await
        .map_err(|e| (StatusCode::INTERNAL_SERVER_ERROR, e.to_string()))?;
    if deleted {
        Ok(StatusCode::NO_CONTENT)
    } else {
        Err((StatusCode::NOT_FOUND, format!("任务 {} 不存在", id)))
    }
}

然后，在src/router.rs中组装路由：

use axum::{
    routing::{delete, get, post, put},
    Router,
};
use crate::handler;
use crate::AppState;

pub fn build() -> Router<AppState> {
    Router::new()
        .route("/todos", get(handler::todo::list_todos))
        .route("/todos", post(handler::todo::create_todo))
        .route("/todos/{id}", put(handler::todo::update_todo))
        .route("/todos/{id}", delete(handler::todo::delete_todo))
}
Axum Router的亮点：注意这里的Router<AppState>泛型参数。Axum让你将共享状态作为类型参数传递给Router——这意味着你的handler必须使用和Router同一类型的State提取器才能编译通过。这种"类型级的路由约束"杜绝了你在运行时忘记传递共享状态的可能。对比一下Express.js的req.app.get(...)或者Go中的context.WithValue——同样的需求在动态语言中只能在运行时才能发现bug。

组装入口

main.rs把一切串联起来：

use axum::http::Method;
use sqlx::postgres::PgPoolOptions;
use std::net::SocketAddr;
use tower_http::cors::{Any, CorsLayer};
use tower_http::trace::TraceLayer;
use tracing_subscriber::EnvFilter;

#[derive(Clone)]
pub struct AppState {
    pub pool: sqlx::PgPool,
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    // 初始化日志
    tracing_subscriber::fmt()
        .with_env_filter(EnvFilter::try_from_default_env()
            .unwrap_or_else(|_| "todo_api=debug,tower_http=debug".into()))
        .init();

    // 数据库连接池
    let database_url = std::env::var("DATABASE_URL")
        .expect("DATABASE_URL 环境变量未设置");
    let pool = PgPoolOptions::new()
        .max_connections(5)
        .connect(&database_url)
        .await?;

    // 运行数据库迁移
    sqlx::migrate!().run(&pool).await?;

    // 构建应用
    let state = AppState { pool };
    let app = crate::router::build()
        .with_state(state)
        .layer(TraceLayer::new_for_http())
        .layer(
            CorsLayer::new()
                .allow_origin(Any)
                .allow_methods([Method::GET, Method::POST, Method::PUT, Method::DELETE])
                .allow_headers(Any),
        );

    let addr = SocketAddr::from(([0, 0, 0, 0], 3000));
    tracing::info!("服务启动于 {}", addr);

    let listener = tokio::net::TcpListener::bind(addr).await?;
    axum::serve(listener, app).await?;

    Ok(())
}

中间件

Axum的中间件基于tower的Layer trait。我们在上面已经添加了三个中间件：

  • TraceLayer：自动记录每个请求的方法、路径、状态码和耗时，对调试和监控非常重要
  • CorsLayer：允许跨域请求——在生产环境中你应当限制allow_origin为特定域名
  • 错误处理：handler返回Result<T, (StatusCode, String)>时，Axum会自动将错误转化为JSON响应

如果需要自定义错误响应格式，可以实现自己的IntoResponse：

use axum::response::{IntoResponse, Response};
use axum::Json;
use serde_json::json;

pub struct ApiError {
    pub status: StatusCode,
    pub message: String,
}

impl IntoResponse for ApiError {
    fn into_response(self) -> Response {
        let body = Json(json!({
            "error": self.message,
            "status": self.status.as_u16(),
        }));
        (self.status, body).into_response()
    }
}

测试与部署

HTTP测试

Axum提供了axum::test辅助进行HTTP集成测试，无需启动真正的HTTP服务器：

#[cfg(test)]
mod tests {
    use axum::{
        body::Body,
        http::{Request, StatusCode},
    };
    use tower::ServiceExt;

    use crate::router;
    use crate::AppState;
    use sqlx::PgPool;

    #[sqlx::test]
    async fn test_list_todos(pool: PgPool) {
        let state = AppState { pool };
        let app = router::build().with_state(state);

        let response = app
            .oneshot(
                Request::builder()
                    .uri("/todos")
                    .method("GET")
                    .body(Body::empty())
                    .unwrap(),
            )
            .await
            .unwrap();

        assert_eq!(response.status(), StatusCode::OK);
    }
}

Docker多阶段构建

将Rust Web API容器化是常见的部署方式。多阶段构建能显著减小镜像体积：

# 构建阶段
FROM rust:1.96-slim-bookworm AS builder
WORKDIR /app
COPY . .
RUN cargo build --release

# 运行阶段 — 仅包含二进制文件
FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y ca-certificates && rm -rf /var/lib/apt/lists/*
COPY --from=builder /app/target/release/todo-api /usr/local/bin/todo-api
ENV DATABASE_URL=postgres://user:pass@host:5432/todos
EXPOSE 3000
CMD ["todo-api"]

最终镜像大小通常只有30-50MB——远小于一个包含Node.js + npm依赖的镜像（通常在200MB以上）。

一个常见的陷阱：我在第一次用Axum写CRUD时，忽略了对PATCH和PUT请求的请求体大小限制。在tower-http中，RequestBodyLimit中间件可以防止恶意的大请求体攻击。虽然对Todo API来说不是致命问题，但生产级服务一定要加上.layer(RequestBodyLimitLayer::new(1024 * 1024 /* 1MB */))。
