第18章 实时推荐系统

设想一个场景：你刚在电商App上点了几下——搜索"蓝牙耳机"、点进了前三款、把其中一款加入了购物车。然后你回到首页推荐流，猜猜会看到什么？

在做得好的推荐系统里，首页立刻会多出蓝牙耳机的推荐、耳机保护套、甚至是"买了这款耳机的用户也买了"的相关商品。用户几秒前的行为，已经在改变推荐结果了。但在做得不好的系统里，首页推荐还是半小时前那一套——推荐的还是你上周看过的笔记本电脑。

这就是实时推荐系统要解决的问题：缩短"用户行为发生"到"推荐结果响应"之间的延迟。这个延迟越短，推荐就越"懂你"。

1. 实时推荐的核心思想

实时推荐不是指模型的训练是实时的（模型训练实时化是另一个更难的问题，我们稍后会聊），而是指特征和推理的实时化。用户的一个行为发生后，系统需要在毫秒级的时间内把这个行为反映到推荐特征中，让下一次刷新推荐时能看到它的影响。

为了实现这一点，工业界通常将实时推荐拆解为"三条流水线"：

第一条：行为流水线（Behavior Stream）

用户的每一次点击、滑动、点赞、收藏、购买等行为，都被埋点系统捕捉并发送到消息队列中。消息队列是整条流水线的入口，业界主流方案是Apache Kafka或Pulsar。Kafka在这里扮演的角色就像一个大坝——它接收四面八方涌来的行为数据，然后以可控的速率分发给下游消费者。消息队列最重要的是保证不丢数据和至少一次投递。

第二条：特征流水线（Feature Stream）

原始的用户行为日志需要被加工成模型可用的统计特征。比如"用户过去5分钟点击了多少个商品"、"用户过去1小时在哪个类目上停留最久"。这些特征计算是典型的流式计算任务——用Apache Flink或Spark Streaming定义滑动窗口，在数据到达时实时计算窗口内的聚合指标。

第三条：模型流水线（Model Stream）

实时特征写入Redis等高速在线存储后，推荐服务在收到用户请求时，从Redis读取最新特征，拼装成模型输入，执行推理。整个推理链路同样有严格的时间预算。

这三条流水线串起来，就构成了实时推荐的核心骨架。我们来深入看看每条流水线的具体实现。

2. 实时特征引擎

实时特征引擎是实时推荐系统的"发动机"。它的核心挑战在于：如何在用户行为的洪流中，毫秒级地完成特征计算和更新。

消息队列选型：Kafka vs Pulsar

Kafka是目前最广泛使用的消息队列方案，生态成熟，社区庞大。Pulsar是后起之秀，支持多租户、存算分离、更低的端到端延迟。对于推荐场景，两者都能胜任。选谁主要看团队的技术栈积累——用Kafka的公司远多于用Pulsar的。

流式计算：Flink滑动窗口

实时特征中最常见的一类操作就是"在滑动窗口内做聚合"。比如计算"用户近10分钟的点击次数"。

-- Flink SQL 实现滑动窗口聚合
-- 场景：计算用户在过去5分钟、30分钟、1小时内的点击和曝光统计

CREATE TABLE user_behavior (
    user_id STRING,
    item_id STRING,
    behavior_type STRING,    -- 'click', 'expose', 'like', 'buy'
    event_time TIMESTAMP(3),
    WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
) WITH (
    'connector' = 'kafka',
    'topic' = 'user-behavior',
    'properties.bootstrap.servers' = 'kafka:9092',
    'format' = 'json'
);

-- 5分钟滑动窗口：每30秒触发一次
CREATE VIEW user_click_5min AS
SELECT
    user_id,
    COUNT(*) AS click_cnt,
    COUNT(DISTINCT item_id) AS unique_item_cnt,
    TUMBLE_END(event_time, INTERVAL '5' MINUTE) AS window_end
FROM user_behavior
WHERE behavior_type = 'click'
GROUP BY
    user_id,
    TUMBLE(event_time, INTERVAL '5' MINUTE);

-- 结果写入Redis在线特征存储
-- （实际通过Flink Redis Sink 或 自定义RichSinkFunction 实现）

需要注意滑动窗口不是真正的"滑动"——Flink的TUMBLE是滚动窗口（Tumbling Window），真正的滑动窗口需要用HOP函数。在上面的例子中，如果你需要每30秒更新一次近5分钟的统计，应该使用HOP窗口：

-- 真正的滑动窗口：窗口大小5分钟，滑动步长30秒
SELECT
    user_id,
    COUNT(*) AS click_cnt,
    HOP_END(event_time, INTERVAL '30' SECOND, INTERVAL '5' MINUTE) AS window_end
FROM user_behavior
WHERE behavior_type = 'click'
GROUP BY
    user_id,
    HOP(event_time, INTERVAL '30' SECOND, INTERVAL '5' MINUTE);

在线特征存储：Redis

Flink算好的特征需要写到某个地方，供推荐服务查询。Redis是这里的事实标准——它把数据存在内存中，单次查询延迟可以做到p99 < 2ms。推荐的实时特征通常以用户ID为key，以Hash或JSON格式存储多个统计指标。Redis的过期时间（TTL）机制天然适合管理特征的时效性——如果一个用户半小时没有新行为，他的实时特征可以自动过期释放内存。

3. 延迟预算管理

实时推荐最容易被忽略的问题是：虽然每个组件都在追求低延迟，但推荐系统的端到端延迟是各个组件延迟的和。任何一个组件的延迟抖动，都会直接拖累整个链路。

一份典型的实时推荐延迟预算分配表：

组件 典型延迟 SLO（服务等级目标） 超出后会发生什么
Kafka 消息传输 ~100ms p99 < 200ms 特征更新延迟，推荐"滞后"于用户行为
Flink 滑动窗口计算 ~50ms p99 < 100ms 窗口聚合结果延迟产出
Flink → Redis 写入 ~50ms p99 < 100ms 特征写入积压，旧特征未被覆盖
Redis 在线查询 ~2ms p99 < 5ms 推荐服务耗时增加，用户等待超时
模型推理 ~30ms p99 < 50ms 用户刷新推荐页需要更长时间加载
端到端总延迟 ~200ms p99 < 400ms 用户体验劣化，用户流失风险增加

注意上表中的"端到端总延迟"不是简单求和——Kafka到Redis的链路（约200ms）和Redis到推理完成（约32ms）是串行的，所以总延迟大约是232ms。这里需要指出一个容易被忽视的关键点：实时特征是在用户行为发生后异步更新的，不是在用户请求时同步更新的。也就是说，用户刷新推荐列表时，读取的是"在此之前已经被更新好的特征"，而不是现场去跑Flink算特征。这保证了推荐服务的推理延迟不会被实时特征链路拖累。

但是，这也意味着实时推荐存在一个"新鲜度窗口"——用户行为发生后，需要等待约200ms，新特征才会在Redis中就绪。这200ms内的推荐请求仍然使用的是旧特征。对大部分场景来说，200ms的延迟是完全可以接受的——用户刷新推荐页的间隔通常是几分钟甚至更长。

工业小贴士：监控实时推荐系统的延迟，不要只看平均值。p99（或p999）才是关键指标。平均值可能只有50ms，但如果每100次请求中有1次超过了500ms，用户体验就会被那1%的"卡顿"毁掉。建议对端到端链路的每个组件都设置p99监控告警。

4. 在线学习到底要不要做？

这是实时推荐领域争议最大的问题之一。在线学习（Online Learning）指的是模型参数随新数据实时更新——用户点一个商品，模型立即更新对这个商品embedding的梯度。听起来很美好对吧？但工业界的现实是：99%的团队不需要在线学习。

真实故事：六个月白干，换来0.5% CTR提升

某中大型互联网公司的推荐团队（约15人，支撑千万级DAU），花了整整六个月搭建在线学习系统。他们实现了Flink实时消费行为日志 → 实时拼接训练样本 → 实时更新模型参数的全链路。所有基础设施都是自研的——因为开源的在线学习框架在百万级特征规模下扛不住。六个月后上线了，AB测试CTR提升了0.5%。然后噩梦开始了：模型参数在线上频繁震荡，有时候上午表现好的模型下午就崩了。他们又花了六个月来修复稳定性问题：加参数约束、加梯度裁剪、加回滚机制、加模型版本管理。最终复盘时发现，如果把这12个月投入到"更好的实时特征 + 小时级增量训练"上，CTR提升可能不止0.5%，而且运维成本低得多。后来他们的架构变成了：实时特征（Flink + Redis）+ 每小时增量重训（Spark + GPU集群），简单、稳定、效果不差。

为什么在线学习在工业界这么"不香"？主要有三个原因：

  • 模型稳定性：在线学习意味着模型参数在不断变化。推荐系统的数据分布本身就是非平稳的（受到季节性、促销、突发新闻等影响），再加上参数在学习，你很难区分"模型正在适应新分布"和"模型坏了"。参数震荡是线上事故的常见原因。
  • 灾难性遗忘：模型过度拟合最近的几条训练样本，忘记了从长期数据中学到的模式。你今天点了一个微波炉食谱，模型就只给你推荐微波炉食谱，忘了你还喜欢户外运动。这是在线学习最典型的失败模式。
  • 工程复杂度：在线学习需要一套完全独立的训练基建：实时样本拼接、实时梯度计算、参数同步与一致性保证、回滚机制、模型版本管理……每一样都不好做。这些工程投入足够养活一个专门的infra团队。

那什么情况下才应该做在线学习？一个粗略的判断标准：如果你的日活用户数低于1亿，大概率不需要在线学习。因为在线学习带来的边际收益，远不如把实时特征做好、把离线重训频率提高来得大。

更务实的做法是"三步走"：

  1. 第一步（大多数团队应该止步于此）：实时特征 + 每日全量重训。实现行为日志的实时流处理和特征的秒级更新，但模型是每天从零开始重训的。这套方案能捕获用户行为的短期变化，工程成本可控。
  2. 第二步（推荐给高活跃平台）：实时特征 + 小时级增量重训。用前一个epoch的模型参数做热启动，在增量数据上微调。相比每日全量重训，模型更新频率更高、更贴合最新分布。
  3. 第三步（仅限超大规模平台）：真正的大规模在线学习。实时梯度更新模型参数。需要完善的监控、回滚、约束机制。只在DAU > 1亿时值得认真考虑。

最后总结一下实时推荐的核心要义：实时特征 > 实时训练。对大部分推荐系统来说，用户行为后的"秒级特征更新"已经足够产生显著的体验提升。用实时特征捕捉用户的即时兴趣变化，用频繁的模型重训保证模型精度，这是工业界验证过的最稳健路径。

好了，实时推荐系统到这里就结束了。实时推荐解决了"已知用户"的问题——有了足够的交互数据，我们能实时追踪用户的兴趣变化。但有一个更棘手的问题一直悬而未决：当一个新用户第一次来到你的平台，没有任何行为数据时，你该怎么办？这正是我们下一章要深入探讨的话题——冷启动问题。
