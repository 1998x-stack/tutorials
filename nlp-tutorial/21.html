第二十一章：检索增强生成

同学们好！今天我们来聊一个在 LLM 时代最接地气、最实用的技术——检索增强生成（Retrieval-Augmented Generation，RAG）。如果说上一章的提示工程是"教模型怎么说"，那 RAG 就是"给模型资料让它照着说"。

为什么要 RAG？其实原因很简单：大模型的知识截止在训练数据的那一刻。你用 ChatGPT 问 2025 年的新闻，它要么编造，要么告诉你"我不知道"。更糟糕的是，你公司的内部文档、你产品的用户手册——这些私有数据模型根本不可能见过。RAG 的核心理念就是：不为难模型记住一切，而是在它回答问题时，实时从外部知识库中检索相关文档，然后让模型基于这些文档进行回答。不需要重新训练，不需要微调，插上检索，模型立即可用。

这章我们从最基础的架构讲起，然后深入向量数据库、分块策略、混合检索这些工程上必须啃的硬骨头。内容很干，建议你边读边打开电脑跟着代码试。

21.1 RAG 架构：检索器 + 生成器 流水线

经典的 RAG 流水线由三个环节组成，环环相扣：

  1. 索引（Indexing）：把知识文档切成小块，用嵌入模型转成向量，存入向量数据库。
  2. 检索（Retrieval）：收到用户问题后，把问题也转为向量，在数据库中检索最相似的 K 个文档块。
  3. 生成（Generation）：把检索到的文档块拼到提示词中，让 LLM 基于这些上下文生成回答。
# RAG 流水线的伪代码

# for chunk in chunks:
#     vector = embedding_model.encode(chunk)

就这么简单？对，原理就是这么简单。但魔鬼全在细节里——用什么嵌入模型、切多长的块、向量数据库选哪个、检索结果怎么用——每一个选择都在深刻影响最终的问答质量。咱们依次来看。

21.2 向量数据库：如何选择

向量数据库是 RAG 的核心基础设施，它负责存向量、比相似度、取最近邻。市面上选择很多，但核心就几个维度需要对比：部署模式、扩展性、一致性、以及——对很多人来说最重要——免费不免费。

怎么选？我建议这样判断：原型阶段无脑上 Chroma，5 分钟就能跑起来；到了产品阶段，如果数据量在百万内且不想管运维，用 Pinecone；如果数据量千万级、或者在自建基础设施上有经验，上 Milvus。不要在一开始过度工程化——RAG 的瓶颈通常不在向量数据库，而在你的分块和检索策略。

21.3 嵌入策略：选对模型是关键

嵌入模型负责把文本变成向量，RAG 的整个检索质量全靠它。如果你用的嵌入模型分不清"苹果手机"和"苹果水果"的区别，后面再怎么优化也是白搭。

当前主流的嵌入模型选型思路：

  • OpenAI text-embedding-3-small / 3-large：闭源但效果最好之一。text-embedding-3-small 维度 1536，3-large 维度 3076。开箱即用，但注意数据隐私（数据要发给 OpenAI）。
  • BGE系列（BAAI/bge-*）：开源首选。BGE-large-en-v1.5 在 MTEB 榜单上和 OpenAI 不相上下，1024 维。支持中文（bge-large-zh-v1.5），社区活跃。
  • E5 / E5-mistral：微软出品。效果非常好，但模型稍大。E5-mistral-7b-instruct 在复杂检索任务上表现出色。
  • Cohere Embed v3：优秀的云服务方案，支持多语言，API 设计对 RAG 场景做了专门优化。
  • Jina Embeddings v2：支持 8k token 长度的输入，适合处理长文档。

维度取舍：高维度向量携带更多信息，但也意味着更大的存储开销和更慢的检索速度。经验法则：1024～1536 维是最佳平衡点。低于 384 维虽然快但信息损失明显；高于 3076 维收益递减、成本陡增。实测中 BGE-large (1024 维) 在大部分中文/英文任务上性价比最高。另外注意——降维不是万能药，PCA 降 30% 维度可能损失 5～10% 的检索准确率。

21.4 分块策略：分块比你想的难得多

这章我安排了一个重要环节——踩坑分享。问题是关于分块（Chunking）的。

分块就是把文档切成小段。你说切个段还不好办？有多少人就是这么想法然后掉了坑——我来给你讲讲我的亲身经历。

踩坑实录：一个 chunk size 引发的血案

有次我们做客服知识库 RAG，用的是公司 SOP 文档，每页大概 500 字。我图省事，在 LangChain 里设了 chunk_size=500，想着"一页一块，刚刚好"。上线后客户问 "退货流程中有哪几步？"——模型答出来的步骤缺斤少两，明明文档里写了 7 步，模型只捞出来 3 步。

查了半天发现：SOP 的退货部分刚好跨了三页。分块后第三步写在第一块末尾，第四步写在第二块开头。问 "退货流程" 时，语义搜索只命中了第一块（包含步骤 1-3），后面的内容根本没进入上下文。不完整的分块导致了不完整的回答。

血泪教训：chunk_size 和 chunk_overlap 必须一起调。固定 500 的块大小加 50 的 overlap 根本不够——当信息跨越块时至少要留 10-20% 的重叠。更优雅的解法是用语义分块（Semantic Chunking）：用 embedding 检测自然段落边界，在话题转折处分块，而不是死板地每 N 个字一刀切。

目前常见分块策略对比：

  • 固定大小分块：最简单的做法，每 N 个字符 / token 一刀切。优点是快，缺点是信息碎片化严重。适合结构简单的文档（如 FAQ 列表）。
  • 语义分块：基于 embedding 相似度检测话题转变，在"自然边界"处分块。效果好但计算成本高。
  • 递归分块：LangChain 的 RecursiveCharacterTextSplitter——按段落 -> 句子 -> 词的优先级切分，尽量保持语义完整。这是目前最常用的工程实践。
  • 文档结构分块：对于 Markdown / HTML，按标题（## /

    ）层级分块。特别适合说明书、技术文档。

  • Agentic Chunking：用 LLM 自己决定哪里是合适的断点。效果最好但最慢最贵。
# LangChain 递归分块示例
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,       # 每块 512 token
    chunk_overlap=100,    # 块间重叠 100 token
    separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""]
    # 中文场景一定要加中文字符做分隔符！
)

chunks = splitter.split_text(长文档内容)
# 输出：7 个文档块，相互之间有 100 token 重叠

21.5 混合检索：取长补短

纯向量检索（Dense Retrieval）有两个经典问题：

  • 遇到 OOV（Out-of-Vocabulary）词——比如企业内部的专有名词 "SPC-2000 传感器校准"——嵌入模型没怎么见过，向量可能"跑偏"。
  • 精确匹配场景——比如搜索工单编号 "WO-2024-08923"——语义相似度对这种纯 ID 文本几乎失效。

这时候 BM25（Sparse Retrieval） 回来了。BM25 是经典的关键词匹配算法，基于 TF-IDF 的改进版，对精确匹配和罕见词特别有效。它的缺点是：不认识同义词，比如搜"汽车"找不到"车辆"。

混合检索（Hybrid Retrieval）的核心思路：两种方法都用，得票最多的结果胜出。

# 混合检索（Dense + Sparse）示例
from langchain.retrievers import EnsembleRetriever
from langchain_community.retrievers import BM25Retriever
from langchain_community.vectorstores import Chroma

# 构建两种检索器
bm25_retriever = BM25Retriever.from_texts(chunks)
bm25_retriever.k = 5

vector_retriever = Chroma.from_texts(
    chunks, embedding_model
).as_retriever(search_kwargs={"k": 5})

# 混合 = 加权融合
ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.3, 0.7]  # BM25 30%，向量 70%，可调
)

results = ensemble.get_relevant_documents("SPC-2000 校准流程")
# 混合检索会同时匹配关键词"SPC-2000"和语义"校准流程"

权重的选择要靠实验调整。一般来说，如果数据中精确 ID / 代码多，提高 BM25 权重；如果数据是自然语言文本为主，提高向量检索权重。还有一些框架（如 LlamaIndex）提供了 Reciprocal Rank Fusion (RRF) 方式来做混合排序，不需要调权重，效果也相当不错。

21.6 高阶 RAG 模式

基础的"检索-生成"模式已经能解决 80% 的问题，但还有 20% 的场景需要更聪明的架构：

1. Self-Querying（自查询检索）

用户问"去年的退款订单有多少"——这不仅仅是语义检索，还隐含了时间过滤。Self-querying retriever 让 LLM 先把问题解析成两部分：语义查询向量（"退款订单"）+ 元数据过滤器（"date > 2025-01-01"），然后检索系统同时走语义搜索和元数据过滤。

2. Multi-Hop RAG（多跳检索）

用户问"2024 年诺贝尔物理学奖得主所在大学位于哪个国家？"——你要先检索"2024 年诺贝尔物理学奖得主是谁"（John Hopfield 和 Geoffrey Hinton），再检索"John Hopfield 的所在大学"（普林斯顿），再检索"普林斯顿所在国家"（美国）。每个问题依赖前一个答案。多跳 RAG 的核心是让模型在检索-思考-再检索之间循环，而不是一次检索就下定论。

3. Corrective RAG (CRAG)

检索不是万能的。如果检索到的文档质量很差、和问题无关，强行注入反而降低回答质量。CRAG 的做法是：让模型先评估检索结果的置信度。如果检索内容不够相关，触发二次检索（换检索源、换关键词）；如果检索内容完全不相关，直接让模型用自身知识回答。简单来说就是——给检索加一个"质检员"。

4. Agentic RAG

目前最前沿的做法：给模型一个工具盒（检索工具、计算器、数据库查询器），由模型自主决定"什么时候需要检索、检索什么、是否需要多次检索"。这已经超越了传统 RAG，进入了 Agent 的范畴。有兴趣的同学可以关注 LlamaIndex 的 Agent 模式和 LangGraph 的 StateGraph+RAG。

# 使用 LlamaIndex 的 RAG 完整示例
# pip install llama-index llama-index-embeddings-huggingface

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader
from llama_index.embeddings.huggingface import HuggingFaceEmbedding
from llama_index.core.node_parser import SentenceSplitter

# 1. 加载文档
documents = SimpleDirectoryReader("知识库/").load_data()

# 2. 分块
parser = SentenceSplitter(chunk_size=512, chunk_overlap=100)

# 3. 嵌入式模型
embed_model = HuggingFaceEmbedding(
    model_name="BAAI/bge-large-zh-v1.5"
)

# 4. 构建索引
index = VectorStoreIndex.from_documents(
    documents,
    embed_model=embed_model,
    transformations=[parser],
)

# 5. 创建问答引擎
query_engine = index.as_query_engine(similarity_top_k=3)

# 6. 提问
response = query_engine.query("退货流程有几步？")
print(response)

# 还可以看看模型引用了哪些文档块
for node in response.source_nodes:
    print(f"来源：{node.metadata['file_name']}，得分：{node.score}")

生产环境 checklist：

1. 文档版本管理——知识库会更新，你需要在更新后重建索引，或者在旧索引上打版本标签。

2. 实时性需求——如果你需要"秒级"看到新增文档的效果，考虑用 Pinecone 等实时索引的数据库。

3. 评估不可少——RAG 的效果评估需要一套标准的 QA 测试集。别凭感觉调参，建一套 RAGAS 评估管线（比如用 LlamaIndex 的 Evaluation 模块）。

4. 保护隐私——如果文档包含敏感信息，确保嵌入和生成模型都是本地部署的开源方案（如 BGE + Llama 3），不要走云端。

RAG 是 2024-2025 年大模型落地最成功的模式之一。它不要求你重新训练模型，不需要你有海量 GPU，只要你有知识文档，就能做出一个"懂得你业务"的 AI 助手。而且 RAG 和下一章的内容有天然联系——多语言 NLP。因为你可以把不同语言的文档都嵌入到一个向量空间中，用户用中文提问，检索系统能从英文文档里找到答案，然后用中文生成。这就是跨语言 RAG 的思路。下一章我们专门讲怎么处理多语言场景——包括如何训练多语言嵌入模型、跨语言检索的挑战，以及像 XLM-R、mT5 这些多语言模型的正确使用方法。准备好了吗？
