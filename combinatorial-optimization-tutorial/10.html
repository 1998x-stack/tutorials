开篇：一次糟糕的相亲安排

大家好，我是你们的讲师。今天的故事从一个让我哭笑不得的周末说起。

2022年，一位朋友在一个"大厂联谊"活动中负责配对。活动规则简单得不能再简单：有20位男生和20位女生，每人提前提交了一份"希望见面的异性名单"。组织者需要安排一对一见面，尽量让更多人的愿望得到满足——也就是说，尽量安排最多的互相感兴趣的对子。

我的朋友拍着胸脯说："这还不简单！"他用 Excel 做了一个贪心算法——先把男生的名字排好，然后每个男生选第一个还没被安排的、也对他感兴趣的女生。20分钟搞定的结果，他配成了15对。

然后有人告诉他："你知道吗，这个数据结构在图上叫做二分图（Bipartite Graph），你的问题叫做最大二分图匹配（Maximum Bipartite Matching），1955年就有人提出了多项式时间算法。你那个贪心方法——15对就是最优了吗？"

他重新检查后发现，确实有办法配到18对——但因为贪心的早期错误选择，白白错失了3对。更惨的是，配错的几对在活动中全程冷场——完全没话聊。那个周末他躲在角落里喝闷酒，发誓以后再也不拍脑袋写贪心了。

这个故事的教训是：二分图匹配看起来很简单——不就是"谁和谁配对"吗？但正是因为看起来简单，人们往往低估了它的难度和深度。事实上，二分图匹配是组合优化中应用最广泛、理论最丰富的主题之一。

本章核心：二分图匹配不仅在理论上优雅——匈牙利算法的 O(V·E) 运行时间让它成为真正的"实用算法"——而且在应用上极其广泛：从网约车司乘匹配到论文审稿人分配，从广告展示优化到计算生物学中的基因比对。更重要的是，它是理解更复杂匹配理论（一般图匹配、带权匹配、在线匹配）的基石。

匹配基本概念

让我们从最基础的定义开始。

给定一个图 G = (V, E)，一个匹配（Matching） M 是 E 的一个子集，其中任意两条边没有公共顶点。换句话说，匹配中的边是"不相交"的——每个顶点最多出现在一条匹配边中。

最大匹配（Maximum Matching）：边数最多的匹配。

完美匹配（Perfect Matching）：覆盖所有顶点的匹配。对于有 2n 个顶点的图，完美匹配恰好有 n 条边。

交替路与增广路

匹配理论中有两个最关键的概念：

  • 交替路（Alternating Path）：相对于一个匹配 M，一条路径上的边交替出现在 M 中和不在 M 中。通俗来说，就是"匹配边→非匹配边→匹配边→非匹配边……"这样交替的路径。
  • 增广路（Augmenting Path）：是一条交替路，且它的两端都是未匹配的顶点。

增广路的重要性在于：如果存在一条增广路，我们可以通过"翻转"它来增大匹配——把路径上的匹配边变成非匹配边，非匹配边变成匹配边。这样匹配的大小增加1。反之，如果不存在增广路，则匹配是最大匹配。

这个性质被称为Berge 引理：匹配 M 是最大匹配 ⇔ 不存在关于 M 的增广路。这个引理是所有匹配算法的理论基础。

增广路示例（Matching M = {b-c}，用 === 表示匹配边）：

    a ── b === c ── d

    // 交替路: a─b─c─d （非匹配─匹配─非匹配）
    // 这是增广路吗？ 是的！a 和 d 都是未匹配顶点

    翻转后：a === b ── c === d

    // 匹配大小从 1 变成了 2
关于增广路的直觉理解：在现实配对的场景中，增广路就像一个"重新配对"的方案。假设 A 当前和 B 配对、C 和 D 配对，但发现 A 和 C 更聊得来、B 和 D 也能聊得来。增广路 a─b─c─d 就代表了"让 A 和 C 配对、B 和 D 配对"的重新安排——通过"交换舞伴"的方式使总配对数+1。这正是匈牙利算法的灵魂。

Hall婚姻定理

现在让我们进入二分图匹配理论中最经典的定理。Hall婚姻定理（Hall's Marriage Theorem）回答了一个基本问题：什么时候二分图存在完美匹配？

设 G = (L ∪ R, E) 是一个二分图，其中 L 是"左部"顶点，R 是"右部"顶点。对于任意子集 S ⊆ L，定义 N(S) 为 S 中顶点的邻居集合（在 R 中）。

Hall 定理：G 中存在覆盖 L 的匹配（即 L 中每个顶点都被匹配）⇔ 对任意 S ⊆ L，有 |N(S)| ≥ |S|。

换句话说：左部的任意 k 个顶点，它们的右部邻居必须至少有 k 个。如果某个 S 违反了条件（|N(S)| < |S|），那么这 |S| 个左部顶点只能和少于 |S| 个右部顶点配对——注定有人落单。

把 Hall 定理应用到相亲问题：如果有 5 个男生想去见女生，但他们感兴趣的女生总共只有 3 个不同的名字——这就是 Hall 条件的违反。不管怎么安排，这 5 个男生中至少有 2 个会落单，因为对面只有 3 个"目标"可供选择。就这么简单。

Hall 定理的证明并不复杂。从左到右的推倒是显然的（每个左顶点需要不同的右顶点与其匹配）。从右到左的推倒需要用归纳法或从"临界图"的角度论证——这里就不展开了，但结论非常深刻。

推论（完美匹配的 Hall 条件）：当 |L| = |R| 时，G 存在完美匹配 ⇔ 对任意 S ⊆ L，|N(S)| ≥ |S|。

虽然 Hall 条件很优美，但它不能直接用作算法——检查所有 2^|L| 个子集是指数级的。它的价值在于理论分析：证明某些图存在完美匹配，或者分析匹配问题的结构性质。

匈牙利算法

匈牙利算法（Hungarian Algorithm）由 Harold Kuhn 在1955年提出，他是受匈牙利数学家 Dénes Kőnig 和 Jenő Egerváry 的工作启发命名的。这个算法用于求解二分图的最大匹配问题。

核心思想

从空匹配开始，不断地：

  1. 选择一个未匹配的左部顶点 u
  2. 从 u 出发尝试找一条增广路（DFS 或 BFS）
  3. 如果找到增广路 P，则翻转 P 上的所有边，匹配大小+1
  4. 如果找不到，则 u 就是"无法被匹配"的顶点

关键问题是如何有效地找增广路。标准实现使用 DFS：从 u 出发，遍历它的所有邻居 v。如果 v 未匹配，则 u-v 就是一条增广路（长度为1的交替路）。如果 v 已匹配给某个左顶点 w，则尝试从 w 递归地找增广路——这就是"让路"的思想。

匈牙利算法伪代码：

    函数 find_augment(u, visited):
        visited[u] = True
        对于 u 的每个邻居 v:
            if not seen[v]:
                seen[v] = True
                if matchR[v] == -1 或  // v 未匹配
                   find_augment(matchR[v], visited):
                    // 找到增广路，翻转
                    matchR[v] = u
                    return True
        return False

    主循环：
        matchR = [-1, -1, ..., -1]  // R 中每个顶点匹配的 L 顶点
        对于每个 u ∈ L:
            seen = [False, ..., False]
            if find_augment(u, seen):
                匹配数 += 1

    返回 匹配数

注意这里的关键技巧：visited 标记左顶点避免递归环，seen 标记右顶点避免重复尝试同一个右顶点——在一次增广搜索中，每个右顶点最多被访问一次。

一个常见的 bug：初学者往往在 DFS 实现中忘记在递归失败后"清除状态"。每次从新的左顶点开始搜索时，必须重置 seen 数组（与 visited 不同，seen 的作用范围仅限于单次搜索）。我曾经在一个项目中因为这个 bug 调试了整整一个下午——结果是匹配结果不稳定，时对时错。如果你自己实现匈牙利算法，请务必注意这个细节。

复杂度分析

匈牙利算法每次成功的增广增加匹配大小1，最多增广 V 次。每次增广搜索需要遍历 O(E) 条边（在最坏情况下）。因此总时间复杂度 O(V·E)。如果用邻接矩阵实现（每条边 O(1) 查询），复杂度为 O(V³)。

实现方式 时间复杂度 空间复杂度 推荐场景
邻接表 + DFS O(V·E) O(V+E) 稀疏图（大多数实际场景）
邻接矩阵 + DFS O(V³) O(V²) 稠密小图（V ≤ 500）
网络流转化 O(V·√E) 霍普克洛夫特-卡普 O(V+E) 大规模二分图（竞赛常用）
Kuhn-Munkres 的命名争议：很多人不知道，"匈牙利算法"这个名字有时候也指代求解指派问题（Assignment Problem）的算法——它其实是同一个 Kuhn 在1955年论文中同时解决的两类问题。不过随着时间推移，学界倾向于用"匈牙利算法"指代最大匹配，用"Kuhn-Munkres 算法"或"KM 算法"指代带权匹配。搞清楚这个命名上的小混乱，可以避免你在读论文时产生困惑。

Kőnig定理

现在介绍二分图匹配理论中最优美的定理。Kőnig 定理说：在二分图中，最大匹配的大小等于最小顶点覆盖的大小。

回忆一下顶点覆盖（Vertex Cover）的定义：顶点集 C ⊆ V 称为顶点覆盖，如果每条边至少有一个端点属于 C。

Kőnig 定理等价于：

ν(G) = τ(G)

其中 ν(G) = 最大匹配大小
     τ(G) = 最小顶点覆盖大小

这个等式的证明很有启发性——从最大匹配出发构造最小顶点覆盖：

  1. 从最大匹配 M 出发，从左部所有未匹配的顶点开始做 DFS 遍历
  2. 令 Z 为所有被访问到的顶点集合
  3. 令 C = (L \ Z) ∪ (R ∩ Z)
  4. 可以证明 C 是一个顶点覆盖，且 |C| = |M|

这个构造非常巧妙。它实际上就是在匈牙利算法的 DFS 过程中留下的"标记"——如果你实现了匈牙利算法，只需要在最后额外做一次 BFS/DFS 就能得到最小覆盖。

Kőnig 定理为什么重要？首先，它是"对偶性"在这个系列问题中的第一个体现——最大化匹配等价于最小化覆盖。其次，它给出了一个实用的推论：在某些应用中，顶点覆盖是一个更容易建模和操作的量。例如在第9章我们提到，二分图的最小点覆盖可以通过最小割来求解——现在我们知道它也等价于最大匹配。

Kőnig 定理还有几个重要推论：

  • 最大独立集 = V - 最小点覆盖：最大独立集（两两不相邻的顶点集）的大小 = V - ν(G)。在二分图中，这又是另一个对偶关系。
  • 最小边覆盖 = V - 最大匹配：边覆盖（覆盖所有顶点的边集）的最小大小 = V - ν(G)。
  • Dilworth 定理：在偏序集中，最大反链大小 = 最小链划分的大小。虽然形式上不同，但它与 Kőnig 定理有深刻的联系。

带权匹配

如果每一对 (u, v) 有不同的"满意度"或"代价"，我们不仅要最大化匹配的对数，还要最大化（或最小化）总权值。这就是带权二分图匹配问题，其中最著名的是指派问题（Assignment Problem）。

指派问题

有 n 个工人和 n 项任务。每个工人 i 完成每项任务 j 的成本是 c_{ij}。需要给每个工人分配恰好一项任务（每项任务也只分配给一个工人），求总成本最小的分配方案。

这显然是一个完美匹配问题，只是边上有权值。可以用匈牙利算法的扩展版本——Kuhn-Munkres 算法（简称 KM 算法）来求解，复杂度 O(n³)。

KM算法的核心思想

KM 算法引入了顶点标号（Vertex Labeling）和相等子图（Equality Subgraph）的概念：

  • 每个顶点 i 有一个标号 l(i)，左部和右部标号之和称为"可行标号"如果 l(i) + l(j) ≥ w(i, j)（对于最大权匹配）
  • 相等子图 G_l 只包含满足 l(i) + l(j) = w(i, j) 的边
  • 核心定理：如果相等子图 G_l 有完美匹配，那么这个完美匹配就是原图的最大权完美匹配

算法迭代地扩展相等子图：在 G_l 中找最大匹配，如果覆盖了所有顶点就返回；否则调整顶点标号（降低某些标号、提高另一些），使得新的边加入 G_l，然后继续。

KM算法核心框架（最大化权值）：

    初始化 l(i) = max_j w(i,j) （左部）, l(j) = 0 （右部）
    while True:
        在相等子图 G_l 中找最大匹配 M
        if M 是完美匹配：
            return M
        else:
            调整标号：δ = min{ l(i)+l(j)-w(i,j) | i∈S, j∉T }
            对 i∈S: l(i) -= δ
            对 j∈T: l(j) += δ
            // 新的边加入 G_l

其中 S 是左部中"已探索"的顶点集，T 是右部中"已探索"的顶点集（从当前未匹配的左部顶点出发做匈牙利搜索能到达的部分）。

KM 算法的优美之处：它把带权匹配问题"降维"成了无权匹配问题。通过巧妙地维护顶点标号，算法保证只在"最佳候选边"组成的子图中搜索匹配，逐步放松条件直到完美匹配置信出现。这种"逐步收紧"的思想在组合优化中反复出现。

实际应用：不只是指派

带权二分图匹配的应用远远超出了"工人-任务"：

  • 多目标跟踪：在前一帧检测到的目标和当前帧的目标之间建立匹配，权值基于位置相似度。这是自动驾驶中行人/车辆跟踪的核心模块。
  • 计算生物学：蛋白质结构比对、DNA 序列比对中，将两个序列的元素匹配，权值基于相似度分数。
  • 搜索引擎广告分配：广告与广告位的匹配，权值为预期的点击率（CTR）。

我2020年参与过一个物流调度项目，需要将100个订单分配给20辆配送车，每辆车至少分到3个订单、最多5个，且要最小化总配送里程。这个约束"每辆车的订单数要求"其实可以用"复制顶点"技巧转化为一个标准的指派问题——把一辆车复制成多个"分身"，每个分身负责一个订单槽位。这种"建模大招"我们会在第16章中系统讲解。
