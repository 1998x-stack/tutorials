第24章 大模型驱动的新一代推荐

从第1章到第22章，我们谈论的所有模型都有一个共同特点：判别式（Discriminative）。它们学习的是一个打分函数：给定用户和物品的特征，输出一个点击率或转化率的预测值。你给它一万个候选，它给每个候选打个分，你选分数最高的那几个。这种范式在过去二十年里统治了推荐系统。

但现在，情况正在发生根本性的变化。大语言模型（LLM）的出现，正在把推荐系统从"判别式打分"推向生成式推荐（Generative Recommendation）。这不仅仅是技术路线的变化——它改变了推荐系统"能做"和"不能做"的边界。

1. 为什么LLM改变了推荐

传统推荐模型有一个根本性的限制：它们把一切压缩成了ID和Embedding。一个新商品上架了，传统模型看到的只是一个新的item_id和一个随机初始化的向量——它对这个商品一无所知。而LLM可以"阅读"这个商品的标题、描述、用户评论，它已经对这个商品有了丰富的理解，甚至不需要看到任何用户行为数据。这就是所谓的世界知识（World Knowledge）。

LLM带来的几个根本性变化：

  • 语义理解：LLM理解物品的语义，不只是ID映射。它可以理解"iPhone 15 Pro Max"和"华为Mate 60 Pro"之间的关系——它们是竞品，价格段相近，目标用户群重叠。传统模型需要足够的用户行为数据才能学到这种关系。
  • 推理能力：LLM可以解释"为什么推荐这个"。"因为用户最近搜索了登山鞋，且当前是夏季，所以推荐透气性好的登山鞋。"——这种推理链条传统模型完全无法产生。
  • 零样本推荐：完全不依赖用户行为数据就可以做推荐。这对冷启动问题来说是革命性的。

范式的转变：从评判到创造

传统推荐系统的工作方式可以类比为"高考阅卷"：给定一堆答案（候选物品），你给每个答案打分，选分数最高的。生成式推荐的工作方式更像"写作文"：根据对用户的了解，直接"写"出最合适的推荐结果。前者是筛选，后者是创造。这个转变的深远影响怎么强调都不为过——当推荐系统开始"创造"推荐结果而不是"选择"推荐结果时，很多我们认为理所当然的限制（比如候选集大小、ID体系）就消失了。

2. LLM as Ranker

LLM进入推荐系统最直接的方式就是替代精排模型。但这里有一个问题：LLM的推理速度很慢，不可能像传统模型那样对几百个候选逐一打分。所以LLM as Ranker的典型做法是：先用传统模型召回并粗排到百级候选，然后用LLM做精排或重排。

LLM做排序有三种常见的提示方法：

  • Point-wise（逐点打分）：给LLM一个用户信息和候选物品，让它输出一个分数。简单直接，但效率低（对每个候选需要一次推理）。
  • Pair-wise（两两比较）：给LLM两个候选，让它选更优的那个。然后通过排序算法（如MergeSort）得到全局排序。效果通常比point-wise好。
  • List-wise（列表排序）：给LLM一个候选项列表，让它直接输出排序后的结果。效率最高，但受限于LLM的上下文窗口长度，且对LLM的排序能力要求更高。

下面是一个list-wise排序的Prompt模板示例：

import openai

def llm_rerank(user_profile, candidates, top_k=10):
    """
    Use LLM to re-rank candidate items.
    """
    items_text = "\n".join([
        f"[{i+1}] {item['title']} | 类别: {item['category']} | 标签: {', '.join(item['tags'])}"
        for i, item in enumerate(candidates)
    ])

    prompt = f"""
你是一个推荐系统的排序专家。请根据用户画像，对以下候选物品进行排序。

=== 用户画像 ===
{user_profile}

=== 候选物品 ===
{items_text}

请按推荐优先级从高到低输出排序后的序号列表（只输出序号，用逗号分隔），
并在最后一句给出排序理由。

排序结果:
"""
    response = openai.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.1,
        max_tokens=200
    )

    # Parse response to get ranked item indices
    ranked_indices = parse_llm_response(response.choices[0].message.content)
    ranked_items = [candidates[i-1] for i in ranked_indices if 1 <= i <= len(candidates)]
    return ranked_items[:top_k]

LLM做排序的效果很大程度上取决于Prompt模板的设计——包括用户画像的表示方式、候选物品的描述格式、输出格式约束等。实践中有很多精细的调优空间。

不同LLM排序方案的对比：

方案 模型规模 单次推理延迟 效果 vs 传统精排 最佳使用场景
Point-wise 7B-70B 500-2000ms 略好或持平 冷启动物品排序
Pair-wise 7B-70B 300-1000ms / 对 明显更好 Top候选的精细排序
List-wise 7B-70B 500-2000ms / 列表 最好（受窗口限制） 重排阶段，10-30个候选
Fine-tuned小模型 1B-7B 50-200ms 接近甚至超越大模型 高吞吐在线排序

延迟：LLM推荐的头号敌人

让我们算一笔账：传统精排模型（如DeepFM）对100个候选打分的延迟在20-50ms。而LLM对100个候选做list-wise排序，即使是最快的7B模型也需要500ms以上——10倍到20倍的差距。在推荐系统中，每100ms的额外延迟都会明显影响用户体验和业务指标（Amazon曾报告每100ms延迟导致1%的销售额下降）。这意味着你不能在推荐漏斗的每一层都用LLM。正确的策略是：传统模型做90%的工作，LLM在关键的10%环节介入——比如做最终重排、生成推荐理由、处理冷启动场景。这不仅是一个技术决策，更是一个成本决策——LLM推理的成本是传统模型的100倍以上。

3. RAG推荐

RAG（Retrieval-Augmented Generation）是LLM领域最火热的技术之一，它的核心思想是：不依赖模型的"死知识"，而是先从外部知识库中检索相关信息，再把检索结果作为上下文输入给LLM，让LLM基于最新、最相关的信息生成回答。

这个思路在推荐系统中同样适用。RAG推荐的基本流程是：

  • 检索：从外部知识库（可以是物品知识图谱、用户画像库、相似用户行为库等）中检索与当前用户最相关的信息。
  • 增强：将检索到的信息作为上下文注入到推荐Prompt中。
  • 生成：LLM基于增强后的上下文生成推荐结果，并附上推荐理由。

学术上有一个非常优雅的闭环框架叫RPRAG-LLM-RS（Retrieval, Planning, Reasoning, and Action Generation — LLM — Recommendation System）：检索外部知识 → 规划推荐策略 → 推理具体推荐项 → 生成最终结果。这个框架的核心创新是引入了R-Critic——一个评估推荐结果的反馈模型，它会检查LLM生成的推荐是否合理，如果不合理则触发重新检索和生成。

真实故事：RAG推荐搞定冷启动

一家电商平台尝试用RAG推荐解决冷启动问题。当新商品上架时，传统模型完全无法推荐它——因为没有任何用户行为数据。他们的方案是：为新商品构建一个"相似物品知识库"，当需要推荐这个新商品时，检索知识库中与该商品最相似的3-5个老商品的描述和表现数据，连同新商品的标题、图片描述一起作为上下文输入LLM。LLM由此"理解"了这个新商品适合推荐给什么样的用户群体。结果是：冷启动商品的曝光量提升了380%，点击率从传统模型的几乎为零提升到了正常商品的70%。这个故事告诉我们：RAG不是替代传统推荐，而是用外部知识补上了传统推荐的盲区。

4. 对话式推荐系统

你有没有想过——如果推荐系统不只是"猜你喜欢"，而是直接和你"聊天"，效果会不会更好？这就是对话式推荐系统（Conversational Recommendation, CRS）的核心理念。

CRS有三种主流范式：

  • LLM as Recommender：LLM直接扮演推荐者的角色。用户说"我想看一部科幻片"，LLM回答"推荐《星际穿越》，它讲述……"。这种方式体验最好，但对LLM的幻觉控制和知识实时性要求高。
  • LLM-Augmented CRS：推荐系统的核心还是传统模型，但LLM负责增强各个组件——比如用LLM做用户意图理解、用LLM生成推荐解释、用LLM做多轮对话管理。这是当前工业界最务实的方式。
  • LLM-Assisted Evaluation：用LLM模拟用户行为来评估推荐系统。这在离线评估中有巨大价值——你可以让LLM扮演不同类型的用户，对新推荐策略做出反馈，而不需要上线做A/B测试。

在中国，阿里巴巴的千问APP是这个方向的标杆产品。千问APP的MAU已经超过1亿，它的推荐能力覆盖了高德（推荐出行目的地）、淘宝（推荐商品）、飞猪（推荐旅游产品）等多个阿里系APP。用户可以在千问中通过自然语言表达需求，LLM理解意图后在各个场景中完成推荐。比如用户说"周末带家人去个不太远的地方玩"，千问可以自动调用飞猪的景点数据、高德的交通数据、淘宝的户外装备数据，综合生成一个完整的出行推荐方案。

5. Agent推荐：ChainRec

如果说CRS是推荐系统的"对话界面"，那Agent推荐就是推荐系统的"自主能力"——推荐系统不再是被动等待请求，而是主动规划、工具调用、多步推理来完成推荐任务。

2026年最受关注的Agent推荐框架是ChainRec。它的核心理念是：来自动化推荐系统中的重复性决策，让LLM像一个经验丰富的推荐工程师一样思考。

ChainRec的工作流程：

  • 从专家轨迹中构建Tool Agent Library——每个Tool Agent封装了一个推荐子任务（如：I2I召回、向量检索、多样性打散、过滤已读内容等）。
  • 训练一个Planner模型，它能根据用户请求和当前场景动态选择需要调用的Tool Agents以及调用的顺序。
  • 执行过程中，Planner观察到每个Tool的输出后，决定下一轮该调什么、是否需要终止。

这种方式在冷启动和兴趣演化场景中表现尤为出色。比如一个用户之前喜欢户外运动，最近开始关注摄影，传统模型需要一段时间收集数据才能"感知"到兴趣变化。但ChainRec可以通过Planner发现用户画像中的新兴趣点，主动调用"相似标签物品召回"Tool和"新类别探索"Tool，快速适应用户的兴趣漂移。

Agent推荐代表着推荐系统的下一个进化方向——从被动响应到主动规划，从单一模型到多Agent协作。虽然目前还处于早期阶段，但发展速度非常快。

LLM给推荐系统带来的变革是全方位的。从特征理解到排序决策，从冷启动到对话交互，从单一模型到Agent协作，每一个环节都在被LLM重新定义。但坦率地说，目前大部分LLM推荐方案还停留在论文和实验阶段——真正落地到大规模生产系统面临着一系列工程挑战。
