平衡——从"偏科"到"全面发展"

上一章的结尾，我们看到了一个令人揪心的场景：一棵精心构造的BST，在插入有序数据后变成了"歪脖子树"，从 O(log n) 退化成了 O(n)。这就好比你花重金请了一位"天才教师"，结果他只擅长教一种学生——成绩好的。遇到成绩一般的学生，他就完全不知所措了。

平衡二叉树（Balanced Binary Tree）就是为了解决这个问题而生的。它给BST加了一条"班规"：任何节点的左右子树高度差不能太大。在插入和删除时，如果这条班规被违反，树就自动进行"自我调整"（旋转），让树重新恢复平衡。

在这一章中，我们会介绍两种最重要的平衡二叉树——AVL树和红黑树。前者是"严格班长"，后者是"宽松班长"。

为什么需要平衡？

在BST最坏情况下（树退化为链表），查找一个元素需要 O(n)时间。如果树有100万个节点，最坏情况下需要100万次比较。但如果树是平衡的（高度≈log₂ n ≈ 20），只需要约20次比较！

这个差距巨大的原因在于对数函数增长极其缓慢：

节点数 n 平衡树高度 (≈log₂ n) 退化树高度 (n) 性能差距
1,000 ≈ 10 1,000 100x
1,000,000 ≈ 20 1,000,000 50,000x
1,000,000,000 ≈ 30 1,000,000,000 33,000,000x

当 n = 10 亿时，平衡树只需要30次比较，而退化的链表需要10亿次。这就是"平衡"真正的价值。在数据库、操作系统、网络协议栈这些需要处理海量数据的系统中，"保持平衡"不是一个可选项——它是必需品。

衡量平衡的指标——平衡因子（Balance Factor）：

# 某节点的高度 = 1 + max(左子树高度, 右子树高度)
# 某节点的平衡因子 = 左子树高度 - 右子树高度
# 平衡要求：所有节点的平衡因子 ∈ {-1, 0, 1}

AVL树——严格的平衡使者

AVL树以其发明者 Adelson-Velsky 和 Landis 命名（1962年），是最早被发明的自平衡二叉搜索树。它的核心思想很简单：每次插入或删除后，检查路径上节点的平衡因子，如果不满足 ∈ {-1, 0, 1}，就进行旋转。

四种旋转

AVL树通过四种旋转来恢复平衡。我们用 ASCII 图来直观展示：

1. LL 旋转（右旋）——新节点插入在"左孩子的左子树"上

    失衡的 z               旋转后的 y
       / \                  / \
      y   T4     ===>      x   z
     / \                  /   / \
    x   T3               T1  T3 T4
   / \
  T1 T2

# 步骤：y 成为新根，z 变成 y 的右孩子
# y 原来的右子树 T3 变成 z 的左孩子

2. RR 旋转（左旋）——新节点插入在"右孩子的右子树"上

    失衡的 z             旋转后的 y
       / \                / \
     T1   y     ===>     z   x
         / \            / \   \
       T2   x          T1 T2  T3
           / \
         T3  T4

3. LR 旋转（先左旋后右旋）——新节点插入在"左孩子的右子树"上

    失衡的 z              先对 y 左旋         再对 z 右旋
       / \                / \                / \
      y   T4     ===>    x   z     ===>     x   z
     / \                /   / \            / \   \
    T1  x              T1  T3 T4          T1 T2  T4
       / \
     T2  T3

4. RL 旋转（先右旋后左旋）——新节点插入在"右孩子的左子树"上

    失衡的 z              先对 y 右旋         再对 z 左旋
       / \                / \                / \
     T1   y     ===>     z   x     ===>     z   x
         / \            / \   \            / \   \
        x  T4          T1 T2  T4          T1 T2 T3 T4?
       / \
     T2  T3

记忆口诀："LL右旋，RR左旋，LR左右双旋，RL右左双旋"——哪边高了就往反方向旋转。

def get_height(node):
    return node.height if node else 0

def get_balance(node):
    return get_height(node.left) - get_height(node.right) if node else 0

def right_rotate(z):
    """LL情况：右旋"""
    y = z.left
    T3 = y.right
    y.right = z
    z.left = T3
    z.height = 1 + max(get_height(z.left), get_height(z.right))
    y.height = 1 + max(get_height(y.left), get_height(y.right))
    return y

def left_rotate(z):
    """RR情况：左旋"""
    y = z.right
    T2 = y.left
    y.left = z
    z.right = T2
    z.height = 1 + max(get_height(z.left), get_height(z.right))
    y.height = 1 + max(get_height(y.left), get_height(y.right))
    return y

def insert_avl(root, val):
    """AVL插入：BST插入 + 平衡维护"""
    if not root:
        return TreeNode(val)

    # 1. 标准BST插入
    if val < root.val:
        root.left = insert_avl(root.left, val)
    elif val > root.val:
        root.right = insert_avl(root.right, val)
    else:
        return root  # 不插入重复值

    # 2. 更新高度
    root.height = 1 + max(get_height(root.left), get_height(root.right))

    # 3. 获取平衡因子，判断是否需要旋转
    balance = get_balance(root)

    # 四种不平衡情况
    if balance > 1 and val < root.left.val:
        return right_rotate(root)          # LL
    if balance < -1 and val > root.right.val:
        return left_rotate(root)           # RR
    if balance > 1 and val > root.left.val:
        root.left = left_rotate(root.left) # LR
        return right_rotate(root)
    if balance < -1 and val < root.right.val:
        root.right = right_rotate(root.right) # RL
        return left_rotate(root)

    return root
AVL树面试要点：面试中通常不会要求手写AVL的完整插入（四种旋转太容易出bug），但你需要能画图展示LL/RR/LR/RL四种旋转的过程，并能分析AVL树的高度为什么严格 ≤ 1.44 × log₂(n+2)。能说清楚"为什么AVL树是严格平衡的"就已经及格了。

红黑树入门

红黑树（Red-Black Tree）是实际工程中使用最广泛的平衡二叉树。它不像AVL那样严格（AVL要求平衡因子绝对值 ≤ 1），而是采用一种"松弛平衡"策略——从根到叶的最长路径不超过最短路径的2倍。

红黑树的五个性质

  1. 每个节点要么是红色，要么是黑色。
  2. 根节点是黑色的。
  3. 所有叶节点（NIL）是黑色的。（这里的叶节点指空节点）
  4. 红色节点的子节点必须是黑色的。（不能有两个连续的红色节点）
  5. 从任意节点到其所有后代叶节点的路径上，黑色节点数量相同。（黑色高度相同）

性质4和性质5共同保证了"最长路径 ≤ 2 × 最短路径"——因为最短路径是全黑，而最长路径最多是"黑-红-黑-红"交替，黑色节点数相同但红色节点不能连续。

红黑树与2-3-4树的等价关系

理解红黑树的一个绝佳视角是把它看作一棵2-3-4树的二叉树表示。2-3-4树是一种多叉搜索树，每个节点可以有2个、3个或4个孩子。红黑树中的"红色"节点本质上表示了"这个节点和父节点在2-3-4树中属于同一个节点"。

  • 2节点（一个值，两个孩子）→ 黑色节点。
  • 3节点（两个值，三个孩子）→ 一个黑色节点带着一个红色子节点。
  • 4节点（三个值，四个孩子）→ 一个黑色节点带着两个红色子节点。

这个等价关系是理解红黑树插入过程中各种"变色+旋转"操作的钥匙。插入一个红色节点后如果破坏了性质4（两个连续红色），就通过旋转和变色来"修复"——本质上就是2-3-4树中的节点分裂和提升操作。

AVL vs 红黑树：工程选型

特性 AVL树 红黑树
平衡标准 严格（|平衡因子| ≤ 1） 松弛（最长 ≤ 2×最短）
查找效率 更快（更严格的平衡） 稍慢（但仍是 O(log n)）
插入/删除效率 更慢（更多旋转） 更快（平均更少的旋转）
实现复杂度 较低（四种旋转清晰） 较高（五种情况+变色）
典型工程应用 数据库索引（查询密集型） STL map/set、TreeMap（增删查均衡）

等等——我们刚才提到一个有趣的现象：AVL比红黑树更难实现（不对，其实红黑树更复杂）。是的，AVL的旋转逻辑非常清晰——四种旋转，对号入座。而红黑树的插入有5种情况，删除有6种情况，每种情况要配合变色和旋转。所以实现复杂度上红黑树 > AVL树。

那为什么几乎所有语言的标准库都选择红黑树（而非AVL）？因为在频繁插入/删除的场景下，红黑树需要旋转的次数更少，整体性能更好。而AVL树在插入/删除时可能需要从插入点一路旋转到根节点（O(log n)次旋转），红黑树最多只需要2次旋转（插入）或3次旋转（删除）。

面试口诀："增删多用红黑树，查询多用AVL树。"——如果你能说出红黑树在STL/TreeMap中的应用，以及AVL树更严格的平衡对查询更有利，面试官会认为你真正理解了平衡二叉树的设计哲学。

面试中的平衡树

你可能会有疑问："面试中平衡树到底考什么？难道要我手写红黑树的插入吗？" 好消息是，绝大多数面试不会让你手写红黑树——那东西自己实现至少要100多行代码，太容易出bug了。以下是面试官真正关心的：

考察层级 需要掌握什么 典型问题
基础 平衡树解决了什么问题，复杂度分析 "BST最坏情况下的复杂度是多少？如何改进？"
进阶 AVL的四种旋转，能画图说清楚 "给定一个序列，构建AVL树并展示旋转过程"
高阶 红黑树五条性质，与AVL的区别，工程选型依据 "TreeMap的底层是什么？为什么选红黑树而不是AVL？"
专家 红黑树与2-3-4树的等价关系，删除的复杂情况 "红黑树的删除最多需要几次旋转？为什么？"

对于大多数面试（包括大厂），掌握到"进阶"层级就足够了。画清楚AVL的四种旋转 + 说出红黑树的五条性质和应用场景，就能超过90%的候选人。

不要掉进"代码细节"的陷阱：我曾经见过一位候选人，花了20分钟试图在白板上写出红黑树的插入代码，结果写到一半自己都绕晕了，最终连"为什么需要红黑树"这个基本问题都没答好。面试官要考察的是你对数据结构的设计思想和复杂度分析的理解，不是你能不能背出红黑树的五种插入情况。如果面试官问红黑树，能画图解释概念远重要于能默写代码。
