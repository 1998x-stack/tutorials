第20章 推荐系统消偏

我有一个朋友，曾经兴奋地告诉我，他们团队开发了一个"划时代"的精排模型，AUC提升了整整3个百分点——这在推荐领域是惊人的提升。所有人都以为这个模型会带来业务的爆发式增长。结果上线A/B测试后，业务指标纹丝不动。

调查了一个星期，真相让人哭笑不得：模型提升的3% AUC，全部来自学会了"排在越靠前的位置，点击率越高"这个规律。它并没有更好地理解用户兴趣，只是把位置信息变成了一个高效的预测作弊器。当位置特征被移除后，新模型表现和旧模型几乎一模一样。

这个故事引出了推荐系统中一个极其隐蔽、但又无处不在的问题：偏差（Bias）。

1. 推荐系统的七种偏差

推荐系统收集到的数据不是自然界随机产生的——它是推荐系统自身行为的产物。换句话说，数据本身就是有偏的（Data is biased by design）。这导致了一个根本性问题：你用有偏的数据训练模型，模型学到的也是偏差，而不是用户的真实偏好。

学术界将推荐系统中的偏差归纳为七大类：

偏差类型 产生原因 业务影响 主要缓解手段
位置偏差 (Position Bias) 用户在列表顶部更容易点击 过度高估顶部位置物品的真实质量 位置特征、IPW、RandPair
选择偏差 (Selection Bias) 训练数据只包含被曝光的物品 模型无法学习未曝光物品的信息 IPS、曝光建模、缺失值处理
流行度偏差 (Popularity Bias) 热门物品获得更多曝光和点击 马太效应，长尾内容被忽视 流行度负采样、因果消偏、校准
从众偏差 (Conformity Bias) 用户倾向于和大家保持一致 用户真实偏好被群体意见掩盖 因果分解、显式建模从众效应
曝光偏差 (Exposure Bias) 系统只推荐已知感兴趣的类别 信息茧房，推荐多样性下降 探索策略、反事实推理
展现偏差 (Presentation Bias) 封面图、标题影响点击决策 模型混淆内容质量和展现质量 展现特征、对比测试
历史偏差 (Historical Bias) 旧模型产出的数据训练新模型 模型自我强化，偏差代际传递 数据增强、A/B分流、重采样

这七种偏差不是孤立存在的——它们相互缠绕、相互放大。更糟糕的是，很多偏差在数据中看起来像是"有效信号"，模型学得越"好"，偏差被放大的越严重。

2. 位置偏差：用户点的是位置，不是内容

位置偏差是推荐系统中最普遍、也最需要优先解决的偏差。它的数学表达非常简洁：

P(click | item, position) = P(relevant | item) × P(seen | position)

用户点击一个物品的概率，可以被分解为两个独立事件的乘积：(1) 用户对物品感兴趣的概率，乘以 (2) 用户在这个位置上注意到物品的概率。注意，P(seen | position) 只和位置有关，和物品本身无关——排在第一位的物品被看到的概率远远高于排在第十位的。

这个分解看似简单，但它揭示了一个深刻的含义：我们不能直接用点击数据来训练模型，因为点击并不等于兴趣。一个排在第一位的低质量物品，可能因为被看到的机会多而获得了大量的点击，而这些点击信号会让模型误以为用户很喜欢它。

工业界主要用三种方法来解决位置偏差：

方法一：位置作为特征（Position as Feature）。在训练时，把物品的展示位置作为一个输入特征喂给模型。模型可以学到位置对点击率的影响。但这里有一个大坑：推理时位置是未知的（你正在预测的是未排序的结果，怎么可能知道最终位置？）。解决方案是推理时将位置特征设为一个固定值（如0），或在模型结构上做特殊设计使位置特征仅在训练时生效。

方法二：逆倾向加权（Inverse Propensity Weighting, IPW）。这是目前学术界和工业界最通用的方法。核心思路是：每个训练样本按照其被观察到的概率进行加权。如果某条样本在位置1被展示（被看到的概率高），它应该被少加权；如果在位置10被展示（被看到的概率低），它应该被多加权。这个权重就是1/P(seen|position)。

方法三：RandPair。这是Google在2019年提出的一种巧妙方法。思路是：在推荐列表中随机交换两个物品的位置，观察点击率的变化。如果交换后点击率发生显著变化，说明位置偏差影响了点击决策，而不仅仅是物品本身的质量。通过大量随机交换的实验数据，可以估计出位置对点击率的真实影响。

工业实践：在大多数大厂的推荐系统中，位置消偏是"标配"而非"选配"。阿里、字节、Google都在线上部署了位置消偏模块。常用的方案是将位置特征和IPW结合使用——位置特征吸收掉大部分偏差，IPW处理剩余的部分。

回到开头的故事：那个团队移除了位置特征后，新模型和旧模型的AUC变得几乎一致。这意味着整个3%的"提升"全是虚假信号。更扎心的是，类似的"虚假提升"在推荐系统领域屡见不鲜——有团队测算过，位置偏差在某些场景下可以解释高达40%的"模型提升"。消偏不是锦上添花，是模型评估的前提条件。

3. 流行度偏差：富者越富

流行度偏差（Popularity Bias）是推荐系统中另一个根深蒂固的问题。它的表现形式很直观：热门物品被推荐得多 → 获得更多点击 → 变得更热门。冷门物品被推荐得少 → 获得更少点击 → 变得更加冷门。这就是推荐领域的马太效应（Matthew Effect）。

为什么这是一个问题？如果热门物品确实质量更好，推热门有什么不对吗？问题在于：热门不一定等于优质。一个物品热门可能只是因为它在早期获得了更多曝光，或者是因为营销推广，或者是偶然因素。很多高质量的冷门内容只是因为缺少初始曝光而被永久埋没了——这对内容生态是巨大的损失。从用户角度来说，只看热门内容的信息流也是同质化的、缺乏惊喜感的。

缓解流行度偏差的几种常见手段：

流行度负采样（Popularity-based Negative Sampling）。在构造训练样本时，对热门物品做更频繁的负采样，减少它们对模型的影响。具体来说，热门物品被采样为负样本的概率和它的流行度的平方根成正比。这样模型不会过度关注热门物品。

校准（Calibration）。确保推荐结果中长尾内容的比例和它们在候选池中的比例大体一致。假设你的内容库中有30%是长尾内容，那么推荐结果中长尾内容也不应低于20-30%。这种方法不改变模型，而是通过后处理规则来保证分布公正性。

因果消偏。这是更根本的做法——用因果推断来区分"推荐导致的流行"和"用户真正喜欢的内容"。一个物品被点击可能是因为用户喜欢它，也可能仅仅是因为它被推荐给了很多人。因果消偏试图通过后门调整（Backdoor Adjustment）来隔离推荐系统自身带来的偏差。这是目前学术界最活跃的方向之一，但工程实现较为复杂。

注意事项：流行度偏差和"推荐热门内容"不是一回事。推荐热门内容本身没有错——在很多场景下，新用户确实更希望看到大众认可的内容。问题在于，如果你的模型把所有流量都倾斜给头部内容，导致长尾生态恶化，这才是需要警惕的。消偏的目标不是让推荐系统"不推热门"，而是让推荐系统"既能看到热门，也能发现长尾"。

4. 纠偏技术体系

把各种消偏技术放在一起比较，可以帮助你理解它们各自的定位和适用场景。

方法 主要解决的偏差 数学基础 实现复杂度 工业采用度 主要局限
位置特征 (Position Feature) 位置偏差 特征工程 低 极高 训练-推理不一致
IPW (Inverse Propensity Weighting) 位置偏差、选择偏差 倾向性估计 + 加权 中 高 倾向性分数估计困难；方差大
Doubly Robust (DR) 位置偏差、选择偏差 IPW + 结果模型插补 高 中 需要两个模型，实现复杂
流行度负采样 流行度偏差 采样修正 低 高 可能过度压制热门内容
校准 (Calibration) 流行度偏差、曝光偏差 分布对齐 低 高 表面修正，不解决根本
因果图方法 (Causal Graph) 所有偏差 结构因果模型 极高 低 实施难度大，需要领域知识

让我们看一下IPW在PyTorch中是如何实现的：

import torch
import torch.nn as nn

class DebiasedCTRModel(nn.Module):
    """Simplified CTR model with IPW debiasing."""
    def __init__(self, feature_dim, hidden_dim=64):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(feature_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, 1),
            nn.Sigmoid()
        )

    def forward(self, x):
        return self.net(x)


def ipw_loss(pred, label, propensity, clip_min=0.05, clip_max=0.95):
    """Inverse Propensity Weighted Binary Cross-Entropy Loss.

    Args:
        pred: predicted CTR (batch_size, 1)
        label: ground truth click label (batch_size, 1)
        propensity: P(seen | position), clipped to avoid explosion

    Returns:
        scalar loss weighted by inverse propensity
    """
    # Clip propensity to prevent extreme weights
    propensity = torch.clamp(propensity, clip_min, clip_max)
    weight = 1.0 / propensity

    bce = nn.BCELoss(reduction='none')
    loss = bce(pred, label)

    # Weight each sample by inverse propensity
    weighted_loss = loss * weight
    return weighted_loss.mean()


# Training loop
model = DebiasedCTRModel(feature_dim=128)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for batch in dataloader:
    features, labels, positions = batch
    preds = model(features)

    # Estimate propensity from position (simplified)
    # In practice, train a separate propensity model
    propensity = torch.sigmoid(10.0 - positions.float())  # higher position = lower seen prob

    loss = ipw_loss(preds, labels, propensity, clip_min=0.1)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

上面的代码中有一个关键细节：clip_min和clip_max。这不是可有可无的——IPW的倾向性分数如果非常小，权重会爆炸（比如 propensity=0.01 时权重=100），导致训练不稳定。在实际工业场景中，倾向性分数经常会出现极端值，所以必须做裁剪（Clipping）。

踩坑提醒：IPW看起来简单优雅，但倾向性模型（Propensity Model）的估计质量直接影响消偏效果。如果你的倾向性估计很差，IPW不仅不会消偏，反而可能引入新的偏差。在实际项目中，建议先用简单的倾向性模型（如基于位置的统计估计），逐步验证后再引入更复杂的模型。另外，IPW的方差通常很大，需要配合大量的训练数据才能稳定。

这里有一个技术选型的建议：如果你的团队规模不大（比如3-5人），优先级应该是：

  • 第一步：位置特征。这是最简单有效的消偏手段，几乎零成本。
  • 第二步：流行度负采样。处理"富者越富"的问题，代码改动很小。
  • 第三步：IPW。当你有准确的位置日志和足够的样本量时引入。
  • 第四步：Doubly Robust或因果图。只有在大规模团队且消偏需求非常明确时才考虑。

5. 小结

推荐系统中的偏差是一个"看不见的敌人"——它无时无刻不在影响你的数据、模型和业务指标，但你很难直观地感知到它的存在。位置偏差让你误以为模型更好了（实际上只是学会了作弊），流行度偏差让头部内容统治一切，选择偏差让你对未曝光物品毫无信息。

消偏不是一个可选的优化项，而是推荐系统工程的基本功。如果你的模型评估没有经过消偏处理，你所有的指标提升都可能是虚假的。一个健康的推荐系统，应该首先确保它能"看"到用户的真实兴趣，而不是数据中的偏差信号。

从技术路线来看，消偏正从手工规则向因果推断演进。IPW是当前工业界的主流方案，但Doubly Robust和因果图等更先进的方法正在从学术界走向工业界。未来随着数据生成过程的更精细化建模，推荐系统的"视力"会越来越好。

但消偏解决的是"用户已经看到什么"的问题，它无法回答"用户未来会喜欢什么"——这是一个动态决策问题。当你既要考虑用户的长期满意度，又要权衡短期收益时，推荐问题就升级成了序贯决策（Sequential Decision Making）问题。这就是我们下一章的主题：强化学习在推荐中的应用。
