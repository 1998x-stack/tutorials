第7章 完整训练流程

大家好，我是你们的讲师。经过前面六章的积累，我们已经掌握了 PyTorch 的全部核心零件：Tensor、Autograd、nn.Module、损失函数、优化器。现在，是时候把这些零件组装成一条完整的、可运行的训练流水线了。

这一章的目标非常明确：写一个可复用的训练模板——你未来的所有项目都可以基于这个模板来开始。我们还会讨论训练中的各种最佳实践，以及那些不经历踩坑就永远不会知道的细节。

训练集、验证集、测试集

在开始写代码之前，先聊聊数据划分。很多初学者会犯一个低级但致命的错误：只用一份数据既做训练又做评估。

想象一下，你给学生一份试卷让他做，做完后你拿着同一份试卷来打分——这公平吗？当然不公平，因为他已经"记住"了题目和答案。同样，如果模型在训练数据上评估，它只是把背下来的东西复述出来，根本无法反映真实能力。

规范的做法是把数据分成三份：

数据集 占比（典型） 用途 注意事项
训练集（Train） 60-80% 模型学习参数（梯度下降更新） 必须打乱顺序（shuffle）
验证集（Validation） 10-20% 调超参数、早停、模型选择 每个 epoch 后评估，不参与梯度更新
测试集（Test） 10-20% 最终模型性能的"公证"评估 只在最后用一次，绝不用于调参
关于数据泄露（Data Leakage）的提醒：

在划分数据时，一定要确保验证集和测试集在训练过程中完全不可见。如果训练数据有"Noisy student"或者伪标签等半监督方法，验证集绝对不能参与任何训练步骤。更隐蔽的泄漏包括：在划分之前对整个数据集做了归一化（应该先划分，再对训练集算 mean/std，然后应用到验证/测试集），或者在做时间序列预测时用了未来的数据来训练。

模板化的训练循环

下面我给出一个工业级的训练模板。这个模板可以复用到 90% 以上的训练任务中。

import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, TensorDataset

def train_one_epoch(model, dataloader, loss_fn, optimizer, device):
    """
    训练一个 epoch 的标准函数
    """
    model.train()  # ⚠️ 切换到训练模式
    total_loss = 0.0
    num_batches = len(dataloader)

    for batch_idx, (inputs, targets) in enumerate(dataloader):
        inputs, targets = inputs.to(device), targets.to(device)

        # 前向 + 损失
        outputs = model(inputs)
        loss = loss_fn(outputs, targets)

        # 反向 + 优化
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

        total_loss += loss.item()

    avg_loss = total_loss / num_batches
    return avg_loss


def evaluate(model, dataloader, loss_fn, device):
    """
    验证/测试的标准函数
    """
    model.eval()  # ⚠️ 切换到评估模式
    total_loss = 0.0
    correct = 0
    total = 0
    num_batches = len(dataloader)

    with torch.no_grad():  # ⚠️ 禁用梯度追踪
        for inputs, targets in dataloader:
            inputs, targets = inputs.to(device), targets.to(device)
            outputs = model(inputs)
            loss = loss_fn(outputs, targets)

            total_loss += loss.item()

            # 分类任务计算准确率
            _, predicted = torch.max(outputs, 1)
            total += targets.size(0)
            correct += (predicted == targets).sum().item()

    avg_loss = total_loss / num_batches
    accuracy = correct / total
    return avg_loss, accuracy
这个模板的关键设计点：
  • model.train() 和 model.eval() 明确切换模式 — 绝对不要省略
  • 验证/测试用 torch.no_grad() — 节省显存和计算时间
  • 数据显式搬到 device — 避免 CPU/GPU 不匹配的 bug
  • 返回平均 loss（不是总和）— 便于不同大小的 dataloader 之间比较

model.train() vs model.eval() 的血泪教训

现在讲一个我亲眼见过的最经典的 bug 之一。

2021 年，隔壁组的一个同事训练了一个图像分类的 ResNet，在训练集上准确率达到了 98%，在验证集上也有 95%。一切看起来都很完美。他兴高采烈地把模型部署到测试集上——准确率只有 10%。

你没看错：10%。对于 10 分类任务，随机猜都有 10%。也就是说模型跟没训练一样。

他花了三天时间排查：模型过拟合？——加了 dropout 和 weight decay，没用。数据分布不一样？——确认过测试集和训练集来自同分布。代码有 bug？——一行行检查了，没发现问题。

最后，他偶然发现问题的根源：测试代码中没有调用 model.eval()。

他的模型使用了 nn.BatchNorm2d。在训练模式下，BatchNorm 会使用当前 batch 的均值和方差来归一化。但在推理时，它应该使用训练阶段累积的全局 running statistics。如果没有调 model.eval()，BatchNorm 会继续使用 batch 统计量——而测试时的 batch size 可能很小（甚至为 1），导致每个 batch 的统计量波动极大，输出完全乱掉。

哪些层受 train/eval 模式影响？
  • nn.BatchNorm1d/2d/3d — 训练用 batch stats，推理用 running stats
  • nn.Dropout — 训练时随机丢弃，推理时全部保留（且 scaling）
  • nn.LayerNorm — 不受影响（它总是用当前数据的统计量）
所以即使你没有显式用 Dropout 或 BatchNorm，养成调用 model.train() / model.eval() 的习惯是一个很好的防守型编程策略。

过拟合一个小批量（Sanity Check）

在正式训练之前，有一个非常重要的"冒烟测试"——过拟合一个小批量数据。具体做法是：只取少量样本（比如 4-8 个），训练模型让 loss 降到接近 0。如果模型连这几个样本都记不住，那说明代码有 bug，或者模型没有学习能力。

# 冒烟测试：过拟合一个小批量
def sanity_check(model, loss_fn, optimizer, device, num_iterations=50):
    """
    在小批量数据上测试模型能否过拟合
    """
    # 取 4 个样本
    sample_inputs = torch.randn(4, 784).to(device)
    sample_targets = torch.randint(0, 10, (4,)).to(device)

    model.train()
    for i in range(num_iterations):
        optimizer.zero_grad()
        outputs = model(sample_inputs)
        loss = loss_fn(outputs, sample_targets)
        loss.backward()
        optimizer.step()

        if (i + 1) % 10 == 0:
            _, predicted = torch.max(outputs, 1)
            acc = (predicted == sample_targets).sum().item() / 4
            print(f"Iter {i+1}, Loss: {loss.item():.6f}, Acc: {acc:.0%}")

    final_loss = loss.item()
    if final_loss < 0.01:
        print("✅ Sanity check passed！模型能记住小批量数据。")
    else:
        print("❌ Sanity check failed！loss 没有降到接近 0，检查代码。")

# 使用示例
model = nn.Sequential(
    nn.Linear(784, 256),
    nn.ReLU(),
    nn.Linear(256, 128),
    nn.ReLU(),
    nn.Linear(128, 10),
)
loss_fn = nn.CrossEntropyLoss()
optimizer = optim.Adam(model.parameters(), lr=1e-3)
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

sanity_check(model, loss_fn, optimizer, device)
我个人的项目管理习惯：在每个新项目的前几天，我会反复运行这个 sanity check。——在投入大量时间做完整训练之前，先用 2 分钟确认模型和数据通路没问题。这个习惯帮我至少避免了十几次因数据预处理 bug 或模型结构错误而导致的"训练了一天发现 loss 不降"的悲剧。

学习率调度器（Scheduler）

在训练过程中，通常不会使用固定的学习率。更好的策略是一开始学习率大一些，快速接近最优区域；然后逐步减小学习率，精细地收敛到最优解。

PyTorch 提供了多种学习率调度策略：

调度器 策略 适用场景 优点 缺点
StepLR 每隔固定步数乘以 gamma CV 传统任务 简单直观 需要手动设置 step_size
CosineAnnealingLR 余弦曲线从 lr 下降到 0 大部分现代任务 平滑下降，没有"断崖" 需要知道总的 epoch 数
ReduceLROnPlateau 验证 loss 不再下降时降低 lr 不确定训练长度的场景 自适应，不需要预设步数 依赖验证集，多了一个超参数 patience
OneCycleLR 先升后降的单周期策略 快速训练、超收敛 可以在更少的 epoch 达到好效果 参数较多，需调 
import torch.optim as optim
from torch.optim.lr_scheduler import StepLR, CosineAnnealingLR, ReduceLROnPlateau

model = nn.Linear(10, 2)
optimizer = optim.Adam(model.parameters(), lr=1e-3)

# StepLR: 每 10 个 epoch 学习率乘以 0.1
scheduler = StepLR(optimizer, step_size=10, gamma=0.1)

# CosineAnnealingLR: 余弦退火（推荐）
scheduler = CosineAnnealingLR(optimizer, T_max=50)  # T_max = 总的 epoch 数

# ReduceLROnPlateau: 验证 loss 不下降时自动降低（需要 monitor 指标）
scheduler = ReduceLROnPlateau(optimizer, mode='min', factor=0.5, patience=5)

# 在训练循环中使用 scheduler
for epoch in range(num_epochs):
    train_loss = train_one_epoch(...)
    val_loss, val_acc = evaluate(...)

    # StepLR 和 CosineAnnealingLR 在每个 epoch 后 step
    scheduler.step()

    # ReduceLROnPlateau 需要传入监控指标
    # scheduler.step(val_loss)

    # 打印当前学习率
    current_lr = scheduler.get_last_lr()[0]
    print(f"Epoch {epoch}, LR: {current_lr:.6f}")
学习率调度器的最佳实践：
  • 大部分情况选 CosineAnnealingLR，它平滑且效果好。把 T_max 设为总的 epoch 数。
  • 如果训练中不确定合适的 epoch 数，用 ReduceLROnPlateau 配合 early stopping。
  • 最新的 PyTorch 2.x 推荐使用 torch.optim.lr_scheduler.CosineAnnealingLR 配合 torch.optim.lr_scheduler.LinearLR 做 warmup。

完整的 MNIST 训练示例

现在，让我们把所有内容整合成一个完整的、端到端的 MNIST 训练脚本：

import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader
from torchvision import datasets, transforms
from torch.optim.lr_scheduler import CosineAnnealingLR

# ---------- 1. 超参数 ----------
BATCH_SIZE = 64
EPOCHS = 10
LR = 1e-3
DEVICE = torch.device("cuda" if torch.cuda.is_available() else "cpu")

# ---------- 2. 数据 ----------
transform = transforms.Compose([
    transforms.ToTensor(),
    transforms.Normalize((0.1307,), (0.3081,)),
])

train_dataset = datasets.MNIST("./data", train=True, download=True, transform=transform)
test_dataset = datasets.MNIST("./data", train=False, download=True, transform=transform)

# 从训练集中切出验证集
train_dataset, val_dataset = torch.utils.data.random_split(
    train_dataset, [50000, 10000]
)

train_loader = DataLoader(train_dataset, batch_size=BATCH_SIZE, shuffle=True)
val_loader = DataLoader(val_dataset, batch_size=BATCH_SIZE, shuffle=False)
test_loader = DataLoader(test_dataset, batch_size=BATCH_SIZE, shuffle=False)

# ---------- 3. 模型 ----------
class MNISTModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(
            nn.Flatten(),
            nn.Linear(784, 256),
            nn.ReLU(),
            nn.Dropout(0.2),
            nn.Linear(256, 128),
            nn.ReLU(),
            nn.Dropout(0.2),
            nn.Linear(128, 10),
        )

    def forward(self, x):
        return self.net(x)

model = MNISTModel().to(DEVICE)

# ---------- 4. 损失和优化器 ----------
loss_fn = nn.CrossEntropyLoss()
optimizer = optim.Adam(model.parameters(), lr=LR)
scheduler = CosineAnnealingLR(optimizer, T_max=EPOCHS)

# ---------- 5. 训练循环 ----------
print(f"设备: {DEVICE}")
print(f"训练集大小: {len(train_dataset)}")
print(f"验证集大小: {len(val_dataset)}")
print(f"测试集大小: {len(test_dataset)}")

for epoch in range(1, EPOCHS + 1):
    train_loss = train_one_epoch(model, train_loader, loss_fn, optimizer, DEVICE)
    val_loss, val_acc = evaluate(model, val_loader, loss_fn, DEVICE)

    scheduler.step()
    current_lr = scheduler.get_last_lr()[0]

    print(f"Epoch {epoch:2d} | "
          f"Train Loss: {train_loss:.4f} | "
          f"Val Loss: {val_loss:.4f} | "
          f"Val Acc: {val_acc:.2%} | "
          f"LR: {current_lr:.2e}")

# ---------- 6. 最终测试 ----------
test_loss, test_acc = evaluate(model, test_loader, loss_fn, DEVICE)
print(f"\n最终测试结果 — Loss: {test_loss:.4f}, Acc: {test_acc:.2%}")

跟踪训练指标

上面我们只跟踪了 loss 和准确率。在实际项目中，你可能需要跟踪更多指标来全面监控训练状态：

# 一个更完善的指标跟踪方案
from collections import defaultdict

class MetricsTracker:
    """跟踪训练过程中的各种指标"""
    def __init__(self):
        self.history = defaultdict(list)

    def log(self, phase, **kwargs):
        """记录一个步骤的指标"""
        for key, value in kwargs.items():
            self.history[f"{phase}_{key}"].append(value)

    def get_last(self, phase, key):
        return self.history[f"{phase}_{key}"][-1]

    def plot(self, metric="loss"):
        """用 matplotlib 绘制指标曲线（需要安装 matplotlib）"""
        try:
            import matplotlib.pyplot as plt
            train_key = f"train_{metric}"
            val_key = f"val_{metric}"
            if train_key in self.history:
                plt.plot(self.history[train_key], label=f"train_{metric}")
            if val_key in self.history:
                plt.plot(self.history[val_key], label=f"val_{metric}")
            plt.legend()
            plt.show()
        except ImportError:
            print("请安装 matplotlib 以使用绘图功能")

# 使用
tracker = MetricsTracker()
tracker.log("train", loss=0.35, accuracy=0.90)
tracker.log("val", loss=0.16, accuracy=0.95)
print(tracker.get_last("train", "loss"))  # 0.35
跟踪指标时的一个常见错误：不要在每个 batch 结束时 .item() 提取 loss 存为 Python float，然后在 epoch 结束时取平均——这样确实可以。但要注意：如果你在 loss.backward() 之前就调用了 loss.item()，没问题。但如果你在 loss.backward() 之后还想用 loss 做其他操作（比如打印），一定要先 .item() 再打印，因为 print 也会保持对计算图的引用，可能导致显存泄漏。养成 loss.item() 的习惯。
