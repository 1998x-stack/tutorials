第19章 AXI4总线协议深度解析

想象一下，你设计了一个高性能 FIR 滤波器和一块 DMA 引擎。你用自定的握手方式——一个 valid 一个 ready，再加上几个控制寄存器——把它们连在了一起，跑通了。然后你加了第三个 IP，第四个。每个人定义握手信号的极性都不同——有人 active high，有人 active low，有人甚至用上升沿触发做握手。整个系统的连线乱成一团，时序收敛变成了噩梦。

这就是 ARM 公司 1996 年推出 AMBA（Advanced Microcontroller Bus Architecture） 协议的初衷——给 SoC 内部的 IP 核定一套统一的"交通规则"。本章我们深入其中最核心的成员——AXI4，理解它的通道结构、握手规则，并亲手写一个符合规范的 AXI4-Lite 从设备。

19.1 为什么需要 AXI——AMBA 协议族的演变

AMBA 协议族从诞生到今天经历了五代演变：APB（1998）→ AHB（1999）→ AXI3（2003）→ AXI4（2010）→ ACE（2011）→ CHI（2013，持续更新）。每一代的驱动力都是同一个：带宽需求永远在涨。下面这张表比较了目前还在广泛使用的几个成员：

协议数据宽度通道数仲裁方式关键特性典型场景
APB8/16/321（地址数据共用）集中式两状态 FSM，无流水线低速控制寄存器、UART/GPIO
AHB32/64/1282（地址+数据分离）集中式，多 Master 需要 Arbiter地址流水线、突发传输、Split 传输SoC 内部总线、DMA
AXI4-Lite32/645（独立通道）分散式不支持突发，地址和写数据可独立发出寄存器配置、控制状态访问
AXI4-Full32~10245分散式支持可变长度突发（1~256 beats）DDR 控制器、PCIe、DMA
AXI4-Stream8~10241（仅数据流）无（点到点）无地址，无限突发，TUSER/ TKEEP 边带信号视频流、A/D 数据、DSP 处理链
ACE32~10245 + Snoop 通道分散式 + 一致性缓存一致性协议（MESI）多核 ARM 处理器集群
CHI32~1024报文式（无传统通道）NoC 路由基于 flit 的片上网络，支持事务 ID 重排序大规模 SoC（如 Versal AI Engine）

从上表可以看出一个重要趋势：从 APB 到 CHI，协议越来越"去中心化"——AXI 的五个独立通道已经允许读写完全并行，而 CHI 更进一步，把总线模型彻底变成了网络模型。

19.2 AXI4 的五个独立通道

AXI4 最显著的特征是五个完全独立的通道，读写分离。每个通道都是单向的，只在一个方向上传输数据，这对时序收敛非常友好——减少了扇出和组合逻辑深度。五个通道分别是：写地址（AW）、写数据（W）、写响应（B）、读地址（AR）、读数据（R）。下表列出了每个通道的关键信号：

所属通道信号名方向宽度描述
写地址 (AW)AWADDRM→S32/64写操作起始地址
AWVALIDM→S1地址有效（Maser 驱动）
AWREADYS→M1Slave 准备好接受地址
AWLENM→S8突发长度（AXI4-Lite 省略）
AWSIZEM→S3每拍字节数（AXI4-Lite 省略）
写数据 (W)WDATAM→S32/64/128写数据
WVALIDM→S1数据有效
WREADYS→M1Slave 准备好接收数据
WSTRBM→SDATA_W/8字节使能（哪几个字节有效）
WLASTM→S1突发最后一拍（AXI4-Lite 恒为 1）
写响应 (B)BRESPS→M2写响应：OKAY=00, EXOKAY=01, SLVERR=10, DECERR=11
BVALIDS→M1响应有效
BREADYM→S1Master 准备好接收响应
读地址 (AR)ARADDRM→S32/64读操作起始地址
ARVALIDM→S1地址有效
ARREADYS→M1Slave 准备就绪
ARLENM→S8突发长度（AXI4-Lite 省略）
读数据 (R)RDATAS→M32/64/128读数据
RVALIDS→M1数据有效
RREADYM→S1Master 准备好接收
RRESPS→M2读响应编码
有意思的细节：AXI4 的写数据和写地址是独立通道。这意味着 Master 可以先发地址、后发数据，也可以先发数据、后发地址，甚至可以同时发——只要各自的 VALID/READY 握手满足即可。AXI4-Lite 通常要求地址和数据之间没有依赖关系，但 Master 的实现自由度很高。这个自由度给 Slave 实现带来了第一个设计决策：当 AW 和 W 不同时到达时怎么办？答案是：Slave 需要缓存 AWADDR 或 WDATA 直到两者都收到。

19.3 VALID/READY 握手机制——整个 AXI 的基石

AXI4 的每个通道都使用相同的VALID/READY 两点握手机制。理解这个机制是所有 AXI 设计的基础。规则只有四条：

  • 规则一：发送方（Source）拉高 VALID，表示数据已准备好。VALID 拉高后，在握手完成之前不能拉低。
  • 规则二：接收方（Destination）拉高 READY，表示可以接收数据。
  • 规则三：当 VALID 和 READY 同时为高时，传输发生，数据在时钟上升沿被采样。
  • 规则四（重要）：发送方不能等待 READY 拉高后才拉高 VALID。VALID 必须只依赖于发送方自己的内部状态，不依赖于接收方的 READY。
为什么规则四如此重要？如果双方都等对方先"举手"——Master 等 Slave 的 READY 再拉 VALID，Slave 等 Master 的 VALID 再拉 READY——系统陷入死锁，总线永远无法完成第一次传输。这正是下面踩坑故事的主角。

VALID/READY 握手有三种典型时序模式：（1）VALID 先到，READY 后到（最常见）；（2）READY 先拉高，VALID 后到（接收方预先准备好）；（3）两个同时拉高（零等待，最快传输）。Slave 设计必须对所有三种情况都能正确处理。

19.4 AXI4-Lite vs AXI4 vs AXI4-Stream：怎么选？

同一个 AXI 家族，三个成员各有专攻：

  • AXI4-Lite：是 AXI4 的"精简版"。取消了突发（AWLEN/AWLEN 信号省略），每笔传输 1 拍数据。地址和数据宽度固定为 32 或 64 位。这是配置寄存器最常用的接口——处理器的 APB 桥后面通常接 AXI4-Lite 设备。
  • AXI4-Full：是完整版。支持 1~256 拍的突发传输，每拍字节数（AWSIZE）可以设定。适合搬运大数据块的场景——DDR 控制器、PCIe DMA、帧缓冲。
  • AXI4-Stream：最简洁的成员。没有地址通道，没有读写之分，只有单一的 forward 数据通道（TDATA）+ optional 边带信号（TKEEP、TLAST、TUSER、TID、TDEST）。用于连续数据流——ADC 采样流、视频像素流、DSP 处理链。

选型原则很简单：只做寄存器配置 → AXI4-Lite；搬运大块数据 → AXI4-Full；连续流处理 → AXI4-Stream。在实际工程中，三个经常混合使用——一个 AXI4-Full 通 DDR，一条 AXI4-Stream 传视频，一组 AXI4-Lite 配寄存器。

19.5 写一个 AXI4-Lite 从设备

纸上得来终觉浅。我们来写一个完整的 AXI4-Lite Slave，实现 4 个 32 位寄存器——CTRL（0x00）、STATUS（0x04）、DATA_IN（0x08）、DATA_OUT（0x0C）。这是 FPGA 开发者最常见的任务：把你设计的功能模块包装成一个 AXI4-Lite 外设，挂到处理器的总线上。

// ============================================================
// AXI4-Lite Slave — 4 x 32-bit registers
// Address map: 0x00=CTRL, 0x04=STATUS, 0x08=DATA_IN, 0x0C=DATA_OUT
// ============================================================
module axi_lite_slave #(
    parameter ADDR_W = 4,
    parameter DATA_W = 32
)(
    input  wire               ACLK,
    input  wire               ARESETN,
    // -- Write Address Channel (AW) --
    input  wire [ADDR_W-1:0]  AWADDR,
    input  wire               AWVALID,
    output reg                AWREADY,
    // -- Write Data Channel (W) --
    input  wire [DATA_W-1:0]  WDATA,
    input  wire               WVALID,
    output reg                WREADY,
    // -- Write Response Channel (B) --
    output reg  [1:0] BRESP,
    output reg         BVALID,
    input  wire         BREADY,
    // -- Read Address Channel (AR) --
    input  wire [ADDR_W-1:0]  ARADDR,
    input  wire                ARVALID,
    output reg                 ARREADY,
    // -- Read Data Channel (R) --
    output reg  [DATA_W-1:0] RDATA,
    output reg  [1:0] RRESP,
    output reg         RVALID,
    input  wire         RREADY
);

    // --- User Registers ---
    reg [DATA_W-1:0] reg_ctrl;
    reg [DATA_W-1:0] reg_status;
    reg [DATA_W-1:0] reg_data_in;
    reg [DATA_W-1:0] reg_data_out;

    // --- Write Address Channel: always ready to accept ---
    always @(posedge ACLK) begin
        if (!ARESETN) AWREADY <= 1'b1;
        else          AWREADY <= 1'b1;
    end

    // --- Write Data Channel: always ready to accept ---
    always @(posedge ACLK) WREADY <= 1'b1;

    // --- Register Write Logic ---
    always @(posedge ACLK) begin
        if (!ARESETN) begin
            reg_ctrl     <= {DATA_W{1'b0}};
            reg_status   <= {DATA_W{1'b0}};
            reg_data_in  <= {DATA_W{1'b0}};
            reg_data_out <= {DATA_W{1'b0}};
        end else if (AWVALID && WVALID) begin
            case (AWADDR[3:2])
                2'b00: reg_ctrl     <= WDATA;
                2'b01: reg_status   <= WDATA;
                2'b10: reg_data_in  <= WDATA;
                2'b11: reg_data_out <= WDATA;
                default: ;
            endcase
        end
    end

    // --- Write Response (B): 2-state FSM ---
    reg [1:0] wstate;
    localparam WIDLE = 2'b00, WRESP = 2'b01;

    always @(posedge ACLK) begin
        if (!ARESETN) begin
            wstate <= WIDLE;
            BVALID <= 1'b0;
            BRESP  <= 2'b00;
        end else begin
            case (wstate)
                WIDLE: if (AWVALID && WVALID) begin
                    BVALID <= 1'b1;
                    BRESP  <= 2'b00;    // OKAY response
                    wstate <= WRESP;
                end
                WRESP: if (BREADY) begin
                    BVALID <= 1'b0;
                    wstate <= WIDLE;
                end
            endcase
        end
    end

    // --- Read Address (AR) + Read Data (R): registered read ---
    always @(posedge ACLK) begin
        if (!ARESETN) begin
            ARREADY <= 1'b1;
            RVALID  <= 1'b0;
            RDATA   <= {DATA_W{1'b0}};
            RRESP   <= 2'b00;
        end else begin
            // Back-pressure: de-assert ARREADY while read data pending
            ARREADY <= (RVALID && RREADY) ? 1'b1 :
                       (ARVALID && ARREADY) ? 1'b0 : ARREADY;
            // Read address accepted: register read data, assert RVALID
            if (ARVALID && ARREADY) begin
                RVALID <= 1'b1;
                case (ARADDR[3:2])
                    2'b00:   RDATA <= reg_ctrl;
                    2'b01:   RDATA <= reg_status;
                    2'b10:   RDATA <= reg_data_in;
                    2'b11:   RDATA <= reg_data_out;
                    default: RDATA <= {DATA_W{1'b0}};
                endcase
                RRESP <= 2'b00;
            end else if (RVALID && RREADY) begin
                // Read data consumed by master
                RVALID <= 1'b0;
            end
        end
    end

endmodule
代码要点解读：AWREADY 和 WREADY 恒为高，意味着这个 Slave 永远准备好接受地址和数据——这是最简单的实现。更复杂的 Slave 可能在忙时拉低 READY（例如内部 FIFO 满）。写响应通道使用了一个两状态 FSM：WIDLE 等写完成，WRESP 等 Master 取走响应。读地址通道实现了反压——当 RDATA 尚未被 Master 消费时，ARREADY 拉低，阻止新的读请求。

19.6 一个真实的踩坑故事：VALID/READY 死锁

很多年前我第一次写 AXI4-Lite Slave，用一个简单的状态机处理写地址 + 写数据。由于我习惯了传统协议的"请求-应答"模式（中断 + 等待），我不自觉地写了一个 Master 逻辑：先等 Slave 的 READY 拉高，再拉 VALID。而在 Slave 端，我又等了 Master 的 VALID 后才拉 READY。结果呢？

双方都在等对方先举手——死锁。整个总线在第一次传输就永远停住了。仿真波形上 AWVALID 和 AWREADY 始终为低，Master 在状态机里无限循环等待。我花了整整一天才意识到问题出在自己对规则四的违反——VALID 不能依赖 READY。

正确做法：AXI 协议规定 VALID 必须只由发送方内部条件决定，一旦拉高必须保持到握手完成。READY 可以等 VALID（Slave 可以选择在接收地址前准备好，也可以等到有地址时才准备）。但 VALID 绝对、绝对不能等 READY。记住这个规则设计就不会再犯死锁错误。
