第17章 DPO/PPO/ORPO数据适配

大家好，我是你们的讲师。上一章我们介绍了四大偏好数据标杆，你手上已经有了一份质量不错的原始偏好数据。但一个核心问题来了：同一份原始数据，你的 DPO 训练脚本认它，KTO 训练脚本可能不认识它。不同的偏好对齐算法，对数据的格式、结构和信息量有着截然不同的要求。

这一章，我们将逐一拆解主流偏好算法（DPO、SimPO、KTO、ORPO）对数据的"喜好"，并给出从原始数据到各算法所需格式的转换实战代码。

同一份偏好数据，不同算法需要不同格式

先把结论放在前面：你选择的算法，决定了你应该准备什么格式的数据。如果把一个算法需要的数据喂给另一个算法，轻则训练崩溃，重则模型学偏。下面这张图清晰地展示了不同算法对数据需求的差异：

原始偏好数据（chosen / rejected 成对）
    │
    ├──→ DPO / SimPO：直接使用成对数据
    │
    ├──→ KTO：需要反解为「正/负」二值信号
    │
    ├──→ ORPO：需要成对数据 + SFT 信号
    │
    └──→ PPO / GRPO：需要奖励信号（或 RM 打分）

关键决策点：
- 有成对数据吗？ → DPO / SimPO / ORPO
- 只有点赞/踩信号？ → KTO
- 有奖励模型或评分？ → PPO / GRPO
一个需要理解的重要区别：DPO 和 ORPO 虽然都使用成对数据，但训练目标不同。DPO 只在偏好对上进行优化，ORPO 同时优化 SFT 损失和偏好对比损失。这意味着如果你选择 ORPO，你的数据转换 Pipeline 需要额外保留 SFT 格式的副本。

DPO（Direct Preference Optimization）— 最常用的选择

DPO（Direct Preference Optimization）是 2024 年以来最主流的偏好对齐方法，它的核心贡献在于：无需显式训练奖励模型，直接通过偏好对优化策略模型。

DPO 对数据格式的要求非常直接——就是我们在第 15 章介绍的标准三元组：

{
  "prompt": "请列举三种减少塑料污染的方法",
  "chosen": "1. 使用可重复使用的购物袋...（详细、具体的回答）",
  "rejected": "少用塑料就行了。"
}

DPO 的 Loss 函数会同时做两件事：增加 chosen 回答的生成概率，同时降低 rejected 回答的生成概率（以参考模型为基准）。因此，chosen 和 rejected 之间的区分度至关重要——如果两者只有微小差别，DPO 的梯度信号将非常微弱。

在 HuggingFace TRL 库中使用 DPO 训练时，数据需要以以下格式加载：

from datasets import Dataset
from trl import DPOTrainer

# DPO 要求的数据格式
data = Dataset.from_list([
    {
        "prompt": "用简单的语言解释机器学习",
        "chosen": "机器学习是让计算机从数据中学习...（优秀回答）",
        "rejected": "机器学习就是用数据训练模型。（过于简单）"
    },
    {
        "prompt": "Python 中列表和元组的区别",
        "chosen": "主要区别在于：列表是可变的...（全面回答）",
        "rejected": "列表可变，元组不可变。（不够详细）"
    }
])

DPO 训练的关键超参数包括 β（KL 散度系数，控制与参考模型的偏离程度，默认 0.1）和损失函数类型（sigmoid / IPO / hinge）。数据层面，建议在训练前检查 chosen 和 rejected 之间的平均 token 长度差异——如果差异过大（比如 chosen 比 rejected 长 3 倍以上），模型可能会学到"长就是好"的偏差。

SimPO — 更轻量的 DPO 变体

SimPO（Simple Preference Optimization）是 2024 年底提出的 DPO 改进版本。它的核心洞察是：既然 DPO 需要参考模型来计算 KL 散度，为什么不直接移除它？

SimPO 的简化思路是：直接用模型对回答的平均对数概率作为"隐式奖励"，比较 chosen 和 rejected 的平均对数概率大小。这意味着：

  • 内存减半——不需要同时加载策略模型和参考模型，OOM 风险大大降低
  • 训练速度提升 30-50%——少了参考模型的前向传播
  • 数据格式与 DPO 完全兼容——你不需要为 SimPO 准备任何额外数据

换句话说，如果你的数据已经按 DPO 格式准备好了，切换到 SimPO 不需要改动一行数据代码，只需要在训练框架中将算法类型从 DPO 改为 SimPO。

实际效果对比：在 MT-Bench 和 AlpacaEval 等主流评测上，SimPO 通常能达到与 DPO 相当甚至略优的效果。SimPO 的论文还发现，SimPO 对 β 超参数的敏感度更低，训练更稳定。如果你的 GPU 资源紧张（比如只负担得起 1 张 A100 来训练 7B 模型），SimPO 的更低内存需求是一个实实在在的优势。

KTO — 只需二值反馈

KTO（Kahneman-Tversky Optimization）的命名灵感来自行为经济学家 Kahneman 和 Tversky 的前景理论。它的最大特点是：不需要偏好对，只需要每个回答的"好/坏"二值标签。

这意味着你可以直接从生产环境获取训练信号：

  • 点赞 👍 → desirability = true（正样本）
  • 点踩 👎 → desirability = false（负样本）
  • 没有用户行为 → 中性样本，可以选择忽略或纳入训练

KTO 的数据格式非常简洁：

{
  "prompt": "帮我写一封求职邮件",
  "response": "尊敬的招聘经理...（一封完整的求职邮件）",
  "is_desirable": true   // 用户点赞了
}

{
  "prompt": "今天是星期几？",
  "response": "根据我的计算，今天是周三，不过我不确定...",
  "is_desirable": false  // 用户点踩了 > 回答太犹豫
}

KTO 在真实生产环境中有巨大的优势——不需要额外的标注成本。用户的点赞/踩行为已经在发生，你只需要把它们收集起来，转换成 KTO 格式即可。当然，KTO 也有代价：它的训练收敛速度通常比 DPO 和 SimPO 慢，需要更多的训练步数才能达到相同的对齐效果。

KTO 的一个陷阱：生产环境中的用户反馈存在严重的"选择偏差"——用户更倾向于给那些"出乎意料"的回答点赞或点踩，而对"按预期工作"的回答保持沉默。这意味着你收集到的正/负样本并不能代表真实的数据分布。在实践中，建议定期用人工标注补充 KTO 的训练数据，确保训练信号的代表性。

ORPO — SFT 和偏好对齐的合二为一

ORPO（Odds Ratio Preference Optimization）是 2024 年提出的一种有意思的方法。它试图回答一个根深蒂固的问题：为什么要先做 SFT，再做偏好对齐？能否一次训练完成两个目标？

ORPO 的训练数据非常特别：它需要一份特殊的"成对"数据，其中 chosen 既是"好的回答"，也是 SFT 的目标回答；而 rejected 仅用于偏好对比损失：

{
  "prompt": "解释什么是递归函数",
  "chosen": "递归函数是指调用自身的函数...（这是一个好的回答，同时也作为 SFT 目标）",
  "rejected": "就是自己调自己。（质量明显较低）"
}

在数据转换时，需要注意：ORPO 对 chosen 的质量要求比 DPO 更高——因为 chosen 同时承担了两个角色（SFT 目标 + 偏好正例），如果 chosen 的质量不够好，SFT 部分会把模型往错误方向拉。推荐的策略是：只保留那些 chosen 回答质量确实很高（比如专家人工编写或经过多次迭代优化）的样本。

格式转换实战 — 从 UltraFeedback 构造多种格式

理论讲了很多，现在上代码。这是我们的核心实战环节——以 UltraFeedback 的多维度评分数据为原料，构造 DPO、SimPO 和 KTO 三种格式的数据：

import json
from datasets import load_dataset

# 1. 加载 UltraFeedback 原始数据
dataset = load_dataset("openbmb/UltraFeedback", split="train")

def convert_to_dpo_format(example):
    """构造 DPO/SimPO 格式：取总分最高的 vs 最低的"""
    responses = example["responses"]
    # 计算每个回答的总分
    for r in responses:
        r["total_score"] = sum(r["ratings"].values())
    # 排序：最高分 = chosen, 最低分 = rejected
    sorted_responses = sorted(responses,
        key=lambda x: x["total_score"], reverse=True)
    return {
        "prompt": example["instruction"],
        "chosen": sorted_responses[0]["content"],
        "rejected": sorted_responses[-1]["content"]
    }

def convert_to_kto_format(example):
    """构造 KTO 格式：每条回答变成一条独立样本"""
    samples = []
    responses = example["responses"]
    for r in responses:
        total = sum(r["ratings"].values())
        # 阈值法：总分 >= 16 (4维度×4分) 为正，<= 10 为负
        if total >= 16:
            samples.append({
                "prompt": example["instruction"],
                "response": r["content"],
                "is_desirable": True
            })
        elif total <= 10:
            samples.append({
                "prompt": example["instruction"],
                "response": r["content"],
                "is_desirable": False
            })
    return samples  # 返回列表，每条指令可产生多条 KTO 样本

# 2. 批量转换
dpo_data = dataset.map(convert_to_dpo_format, remove_columns=dataset.column_names)
kto_data = dataset.map(convert_to_kto_format, remove_columns=dataset.column_names)
# 展平 KTO 数据（每条指令产生多条）
kto_data = kto_data.flatten()

# 3. 保存
dpo_data.to_json("ultrafeedback_dpo.jsonl")
kto_data.to_json("ultrafeedback_kto.jsonl")
关于 SimPO 格式的说明：从上面的代码可以看到，我们只实现了 DPO 和 KTO 两种格式的转换。SimPO 的格式与 DPO 完全一致，所以 dpo_data 可以直接用于 SimPO 训练。你在训练脚本中只需要将 loss_type 参数从 "sigmoid" （DPO）改为 "simpo" 即可——数据不需要任何变动。

这个转换过程的几个关键决策点值得注意：

  • DPO 格式中，选择"最高分 vs 最低分"——确保 chosen 和 rejected 之间有足够的区分度。你也可以选择"平均分以下的随机选一个"，但我们的经验是"极端对比"的效果更稳定。
  • KTO 格式中，阈值的选择——我们用了总分 ≥ 16 为正、≤ 10 为负。中间的灰色地带（11-15 分）被丢弃了，因为这些样本的反馈信号不够明确。阈值需要根据实际的数据分布来调整。
  • 数据平衡——KTO 转换后，正样本和负样本的数量可能严重不均衡。建议在训练前对多数类进行下采样，保持正负比例在 1:1 到 3:1 之间。

掌握了这些数据转换技巧，你就能灵活地在不同算法之间切换，根据计算资源和数据可得性选择最合适的方案。第 18 章，我们将进入一个不能回避的话题——安全对齐。当模型面对有害请求时，它应该学会"有所不为"。
