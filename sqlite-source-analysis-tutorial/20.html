第20章 虚拟表与扩展

大家好，我是你们的讲师。上一章我们学习了内置函数机制——通过注册回调函数来扩展 SQLite。这一章我们要聊一个更强大的扩展体系——虚拟表（Virtual Table）。

函数给 SQLite 加"新公式"，虚拟表则给 SQLite 加"新数据源"。通过虚拟表接口，你可以用 SQL 查询任意数据——内存数组、CSV 文件、HTTP 接口、甚至另一个数据库。你写 SELECT * FROM my_vtab WHERE id = 42，背后调用的是你自己实现的 C 函数。

一句话： 虚拟表接口是 SQLite 的"存储引擎插件系统"，把存储和查询的底层实现完全开放给开发者。

虚拟表接口 — sqlite3_module

虚拟表的心脏是 sqlite3_module 结构体，定义在 sqlite3.h 中。它包含虚拟表需要实现的全部回调方法：

struct sqlite3_module {
  int iVersion;
  /* 生命周期 */
  int (*xCreate)(sqlite3*, void *pAux, int argc, const char *const*argv,
                 sqlite3_vtab **ppVTab, char **pzErr);
  int (*xConnect)(sqlite3*, void *pAux, int argc, const char *const*argv,
                  sqlite3_vtab **ppVTab, char **pzErr);
  int (*xBestIndex)(sqlite3_vtab *pVTab, sqlite3_index_info*);
  int (*xDisconnect)(sqlite3_vtab *pVTab);
  int (*xDestroy)(sqlite3_vtab *pVTab);
  /* 游标操作（必须实现） */
  int (*xOpen)(sqlite3_vtab *pVTab, sqlite3_vtab_cursor **ppCursor);
  int (*xClose)(sqlite3_vtab_cursor*);
  int (*xFilter)(sqlite3_vtab_cursor*, int idxNum, const char *idxStr,
                 int argc, sqlite3_value **argv);
  int (*xNext)(sqlite3_vtab_cursor*);
  int (*xEof)(sqlite3_vtab_cursor*);
  int (*xColumn)(sqlite3_vtab_cursor*, sqlite3_context*, int);
  int (*xRowid)(sqlite3_vtab_cursor*, sqlite3_int64 *pRowid);
  /* DML 可选 */
  int (*xUpdate)(sqlite3_vtab*, int, sqlite3_value**, sqlite3_int64*);
};

在这众多回调中，6 个是必须实现的：

回调时机职责
xCreateCREATE VIRTUAL TABLE创建并初始化虚拟表
xConnect打开数据库时连接已有虚拟表
xBestIndex查询规划报告支持哪些过滤条件
xOpen每次查询开始分配游标
xClose查询结束释放游标
xDisconnect关闭数据库断开连接

xCreate 和 xConnect 的区别在于：xCreate 在第一次 CREATE VIRTUAL TABLE 时被调用，同时负责"创建"和"连接"；xConnect 在后续打开数据库时只负责"连接"。对于只读虚拟表（如 CSV 读取器），两者可以完全相同。

虚拟表生命周期

虚拟表从创建到销毁经历完整的生命周期：

/*
  CREATE VIRTUAL TABLE → xCreate()
  SELECT FROM vtab      → xConnect() → xBestIndex()
                         → xOpen() → xFilter()
                         → LOOP: xEof()? xColumn()+xNext()
                         → xClose()
  DROP TABLE            → xDisconnect() → xDestroy()
*/

最关键的环节是 xBestIndex + xFilter 这对组合。xBestIndex 在查询规划时被调用，接收 sqlite3_index_info 结构，其中包含 WHERE 子句中所有可用条件。你根据这些条件选择最优策略，设置 estimatedCost（估计成本）。SQLite 会选择成本最低的方案。

💡 虚拟表的第一原则：永远不要在 xOpen 中做重活儿。xOpen 只分配游标内存，真正的初始化放到 xFilter 中。否则查询规划器在试探阶段就会做大量无用功。

carray 扩展源码

来看一个简洁但完整的虚拟表实现——carray（ext/misc/carray.c），它把 C 语言的数组暴露为 SQL 表。

/* carray 虚拟表的核心结构 */
typedef struct carray_cursor {
  sqlite3_vtab_cursor base;
  sqlite3_int64 iRowid;     /* 当前行号 */
  const void *pPtr;         /* 数组指针 */
  int nVal;                 /* 数组长度 */
  int eType;                /* 元素类型 */
} carray_cursor;

/* xConnect — 声明表模式 */
static int carrayConnect(sqlite3 *db, void *pAux, int argc,
                         const char *const*argv,
                         sqlite3_vtab **ppVTab, char **pzErr){
  carray_vtab *pNew = sqlite3_malloc(sizeof(carray_vtab));
  memset(pNew, 0, sizeof(*pNew));
  sqlite3_declare_vtab(db, "CREATE TABLE x(value ANY, ptr ANY, count INT)");
  *ppVTab = (sqlite3_vtab*)pNew;
  return SQLITE_OK;
}

/* xFilter — 通过参数获取数组指针、长度和类型 */
static int carrayFilter(sqlite3_vtab_cursor *pVtabCursor, int idxNum,
                        const char *idxStr, int argc, sqlite3_value **argv){
  carray_cursor *pCur = (carray_cursor*)pVtabCursor;
  pCur->pPtr = (void*)(sqlite3_value_int64(argv[0]));
  pCur->nVal = sqlite3_value_int(argv[1]);
  pCur->eType = argc>=3 ? sqlite3_value_int(argv[2]) : 1;
  pCur->iRowid = 1;
  return SQLITE_OK;
}

使用方式是通过指针绑定将 C 数组指针传给 SQL：

int aVals[] = {10, 20, 30, 40, 50};
sqlite3_bind_pointer(pStmt, 1, aVals, "carray", 0);
sqlite3_bind_int(pStmt, 2, 5);
/* SELECT * FROM carray(?1, ?2); → 返回 5 行 */

generate_series 内置虚拟表

从 3.32.0 开始，generate_series 作为内置虚拟表提供（需 -DSQLITE_ENABLE_GENERATE_SERIES 编译）。它的核心在于 xBestIndex 识别 START/STOP/STEP 约束：

static int seriesBestIndex(sqlite3_vtab *pVTab, sqlite3_index_info *pIdxInfo){
  for(int i=0; i<pIdxInfo->nConstraint; i++){
    if( !pIdxInfo->aConstraint[i].usable ) continue;
    int iCol = pIdxInfo->aConstraint[i].iColumn;
    if( iCol==0 && pIdxInfo->aConstraint[i].op==SQLITE_INDEX_CONSTRAINT_EQ ){
      pIdxInfo->aConstraintUsage[i].argvIndex = 1; /* START */
      pIdxInfo->idxNum |= 1;
    }else if( iCol==1 && ... ){ /* STOP */  pIdxInfo->aConstraintUsage[i].argvIndex = 2; }
    else if( iCol==2 && ... ){ /* STEP */  pIdxInfo->aConstraintUsage[i].argvIndex = 3; }
  }
  pIdxInfo->estimatedCost = 100;
  return SQLITE_OK;
}
/* 使用：SELECT value FROM generate_series(1, 10, 2); */
设计启发： generate_series 不预生成全部数据，而是在 xNext 中逐步计算下一个值。这种"按需计算"的懒加载模式让虚拟表可以处理理论上无限的数据流。

最佳实践与踩坑故事

实践好做法坏做法
xOpen/xClose仅分配/释放游标内存在 xOpen 中扫描数据或打开文件
xBestIndex根据实际能力填写 cost总是返回固定 cost 0
xFilter只定位到第一个匹配行在 xFilter 中加载全部数据到内存
xColumn只返回当前行的列值在 xColumn 中推进游标位置
⚠️ 真实踩坑： 我曾为内部工具编写 CSV 虚拟表，在 xOpen 中直接解析了整个 CSV 文件。小文件没事，但某天同事导入了一个 500MB 的日志 CSV——SQLite 直接 OOM。修复方式是把解析逻辑移到 xNext 中逐行读取，xFilter 只打开文件句柄。修复后同样的 500MB 文件在 64MB 内存的嵌入式设备上跑得稳稳当当。

实战：内存键值存储虚拟表

最后我们来设计一个内存键值存储虚拟表 kvstore。它使用哈希表存储键值对，在 xBestIndex 中区分"点查"（直接哈希定位）和"全表扫描"两种策略：

/* 核心：xBestIndex 中判断是否有点查条件 */
static int kvBestIndex(sqlite3_vtab *pVTab, sqlite3_index_info *pIdxInfo){
  for(int i=0; i<pIdxInfo->nConstraint; i++){
    if( pIdxInfo->aConstraint[i].usable &&
        pIdxInfo->aConstraint[i].iColumn==0 &&  /* key 列 */
        pIdxInfo->aConstraint[i].op==SQLITE_INDEX_CONSTRAINT_EQ ){
      pIdxInfo->aConstraintUsage[i].argvIndex = 1;
      pIdxInfo->idxNum = 1;          /* 点查模式 */
      pIdxInfo->estimatedCost = 1.0;  /* 成本极低 */
      pIdxInfo->estimatedRows = 1;
      return SQLITE_OK;
    }
  }
  pIdxInfo->estimatedCost = 100.0;   /* 全表扫描，成本高 */
  return SQLITE_OK;
}
/* 注册：sqlite3_create_module(db, "kvstore", &kvModule, 0); */
