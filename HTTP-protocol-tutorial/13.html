第十三章：内容编码与传输

各位同学好。上一章我们聊了CORS——一种跨域通信的许可机制。这一章我们换个角度，聊聊HTTP在传输效率方面的设计：内容怎么压缩节省带宽、大文件怎么分段传输、断点续传是怎么实现的。这些机制看起来不起眼，但直接影响你的网站加载速度。

13.1 内容编码（Content-Encoding）

内容编码指的是服务端在发送响应之前，对响应体进行的压缩编码。浏览器会在请求时通过 Accept-Encoding 告诉服务器自己支持哪些压缩算法，服务器选择一种后，在 Content-Encoding 响应头中声明使用了哪种编码。

// 请求：浏览器声明支持的压缩算法
GET /index.html HTTP/1.1
Host: www.example.com
Accept-Encoding: gzip, deflate, br

// 响应：服务器选择了 brotli 压缩
HTTP/1.1 200 OK
Content-Encoding: br
Content-Type: text/html; charset=utf-8
Vary: Accept-Encoding

目前主流的三种压缩算法：

算法Accept-Encoding值压缩比压缩速度解压速度浏览器支持
Gzipgzip较好（通常60-70%）快快所有浏览器
Brotlibr优秀（比gzip高10-20%）中等快Chrome/Firefox/Safari/Edge
Deflatedeflate与gzip相近快快支持但较少使用

Gzip 是最经典的选择，几乎所有的Web服务器和CDN都支持。它使用DEFLATE算法（和Deflate编码不是一回事），对文本资源（HTML、CSS、JS、JSON）压缩效果很好，但对图片、视频等已经压缩过的资源没什么效果。

Brotli 是Google开发的算法，在HTTP/2时代开始流行。它在压缩比上比gzip好10-20%，而且解压速度很快——这意味着用户可以更快地拿到完整的响应体。Brotli有11个压缩等级（0-11），等级越高压缩比越大但压缩时间也越长，服务器上通常设置为等级4-6以平衡CPU和压缩比。

什么时候不要压缩？

  • 已压缩过的文件：JPEG、PNG、GIF、MP4、ZIP等——尝试压缩它们只会浪费CPU
  • 小文件（小于1KB）：压缩产生的额外头部可能让文件更大
  • 流媒体：实时传输的场景不适合等待压缩完毕再发送

实战配置：Nginx开启gzip/brotli压缩很简单。注意 gzip_types 要准确，只对文本类资源开启压缩，避免对图片和视频做无意义的CPU消耗。brotli需要额外安装模块 ngx_brotli，但性能收益值得投入。

13.2 传输编码（Transfer-Encoding）

内容编码解决的是"内容怎么压缩"，传输编码解决的是"数据怎么分段发送"。它们是不冲突的——一个响应可以同时使用内容编码和传输编码：先压缩内容，再分块传输。

Chunked Transfer（分块传输）是HTTP/1.1引入的核心机制。它的场景是：服务器在开始发送响应时，还不知道响应体的大小——比如动态生成的内容、数据库查询结果流式返回。

没有分块传输时，服务器必须在响应头里设置 Content-Length，告诉浏览器响应体有多少字节。但如果内容没法提前计算大小怎么办？有两种方案：

  • 先在内存中缓冲整个响应，计算大小，再发送——浪费内存，增加延迟
  • 使用 Transfer-Encoding: chunked，一块一块地发

分块传输的报文格式：

HTTP/1.1 200 OK
Content-Type: text/plain
Transfer-Encoding: chunked

1e
这是第一块数据，一共30个字节
1a
这是第二块数据，一共26个字节
0
// 空行表示传输结束

每个块的格式：[块大小(十六进制)]\r\n[块数据]\r\n。最后以 0\r\n\r\n 结束。块大小的十六进制表示不需要前导零。

Content-Length vs Chunked：

  • 有 Content-Length 时，浏览器知道完整大小——可以显示进度条（比如下载文件）
  • 使用 Transfer-Encoding: chunked 时，浏览器不知道总大小——进度条无法显示百分比
  • 两者不能同时存在（按HTTP/1.1规范，Transfer-Encoding: chunked 优先级高于 Content-Length）

理解传输编码 vs 内容编码：传输编码是"传输层的包装"——数据先被切成块，加上块大小标注，再发送。内容编码是"应用层的包装"——数据先被gzip压缩，再放到响应体里。两者可以同时使用：数据先压缩（Content-Encoding: gzip），再分块（Transfer-Encoding: chunked）。客户端接收时，先拼接块，再解压。

13.3 范围请求（Range Requests）

范围请求是HTTP提供的断点续传和多线程下载的基础。它的核心思想是：客户端可以只请求资源的一部分，而不是整个文件。

流程很简单：

  1. 客户端先发一个HEAD请求或者GET请求，看服务器是否支持范围请求（响应头中有 Accept-Ranges: bytes）
  2. 如果支持，客户端通过 Range 请求头指定需要哪段字节
  3. 服务器返回 206 Partial Content，带上 Content-Range 头表明返回的是哪一段
// 请求：只要 1024 到 2047 字节
GET /bigfile.zip HTTP/1.1
Host: download.example.com
Range: bytes=1024-2047

// 响应
HTTP/1.1 206 Partial Content
Accept-Ranges: bytes
Content-Range: bytes 1024-2047/10485760
Content-Length: 1024

Range的几种格式：

  • Range: bytes=0-499 —— 前500字节
  • Range: bytes=500-999 —— 第500到999字节
  • Range: bytes=-500 —— 最后500字节（省略起始位置）
  • Range: bytes=500- —— 从第500字节到文件末尾（省略结束位置）

范围请求最典型的场景：

  • 视频拖拽播放：你看视频时拖动进度条到一个新位置，浏览器只请求从那个位置开始的视频数据，不需要重新下载整个视频
  • 暂停续传：下载到一半断了，下次只下载缺失的那部分
  • 多线程下载：下载管理器把文件分成几段，同时下载，最后合并——每个线程请求不同的Range

调试技巧：用 curl 测试范围请求很方便。curl -H "Range: bytes=0-99" https://example.com/file -I 可以快速验证服务器是否支持Range请求。如果服务器返回200而不是206，说明它不支持范围请求，返回的是完整文件。

13.4 断点续传实战

理解了Range请求的基础后，我们来看一个完整的多范围请求（Multi-part Range）场景——下载管理器如何同时从多个线程下载同一个文件的不同部分。

客户端可以一次性请求多个不连续的Range：

// 请求三段数据
GET /bigfile.zip HTTP/1.1
Host: download.example.com
Range: bytes=0-999, 5000-5999, 10000-10999

服务器返回的 Content-Type 会变成 multipart/byteranges，每个段用MIME分隔符隔开：

HTTP/1.1 206 Partial Content
Content-Type: multipart/byteranges; boundary=THIS_STRING_SEPARATES

--THIS_STRING_SEPARATES
Content-Type: application/zip
Content-Range: bytes 0-999/10485760

[前1000字节的数据]
--THIS_STRING_SEPARATES
Content-Type: application/zip
Content-Range: bytes 5000-5999/10485760

[5000-5999字节的数据]
--THIS_STRING_SEPARATES
Content-Type: application/zip
Content-Range: bytes 10000-10999/10485760

[10000-10999字节的数据]
--THIS_STRING_SEPARATES--

实际下载管理器的策略更精细：

  • 启动时先发HEAD请求获取文件总大小（通过 Content-Length）
  • 检查 Accept-Ranges 是否支持Range请求
  • 如果不支持，退化为单线程完整下载
  • 如果支持，将文件分成N段（通常4-8段），用N个线程同时下载
  • 每个线程通过Range请求获取自己的那一段，写入临时文件的不同位置
  • 所有线程完成后，合并（或者本身就是稀疏文件，不需要合并）

踩坑故事：早年做一个文件上传下载系统，用户下载大文件时经常断连，我们加入了断点续传功能。上线后发现一个诡异的问题：某些用户下载的文件校验和不对。排查了很久才发现，问题出在多线程下载的合并阶段——不同线程下载速度不同，完成顺序不确定，但我们的合并逻辑假设线程按顺序完成。更要命的是，有些CDN节点对Range请求的处理不一致，有的节点返回了正确的Range，有的节点忽略Range头直接返回了完整文件。从那以后我们加了两层防护：第一，收到每个Range响应后校验Content-Range是否匹配；第二，下载完成后对整个文件做MD5校验，不通过就重新下载。

好了，第十三章的内容就到这里。我们学习了HTTP的内容编码（gzip/brotli压缩）、传输编码（chunked分块传输）、以及范围请求与断点续传。这些机制共同构成了HTTP高效传输数据的基础。下一章我们将进入重定向与URL重写，看看HTTP是如何优雅地把请求从一个URL"转交"到另一个URL的。
