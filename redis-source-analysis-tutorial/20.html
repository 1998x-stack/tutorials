第20章：主从复制（上）—— 全量与部分同步

大家好，我是你们的讲师。前面十几章我们一直围绕"一台 Redis 服务器"来讲解：内存管理、数据结构、命令实现、持久化、过期淘汰。但在真实的生产环境中，Redis 从来不会单枪匹马地工作。

为什么？因为单点故障是分布式系统的大敌。你的机器可能会宕机、网络可能会中断、甚至机房可能会断电。如果只有一台 Redis，无论它跑得多快，一旦挂了就是"全站瘫痪"。主从复制（Replication）就是为了解决这个问题而存在的——它让数据在多个节点之间保持一致，实现数据冗余。

不仅如此，主从复制还带来了读能力水平扩展：你可以在多个从库之间分摊读流量，让一台主库专注于处理写请求。今天我们就从源码层面完整剖析 Redis 的复制机制，重点理解全量同步和 PSYNC 部分同步的原理。

复制架构概览

Redis 的复制是典型的一主多从（Leader-Follower）架构。主库（Master）负责处理写请求，从库（Replica/Slave）从主库同步数据，对外提供只读服务。这种模型有几个核心特征：

  • 非对称：只有主库可以写入，从库默认是只读的。
  • 异步：主库在执行写入命令后，不会等待从库确认就返回客户端。这意味着从库的数据可能落后于主库（秒级甚至更大），属于最终一致性模型。
  • 全量 + 增量：从库首次连接时触发全量同步（RDB 快照传输），后续断线重连时尝试增量同步（PSYNC）。

复制建立的流程从一条 REPLICAOF（或旧版 SLAVEOF）命令开始：

/* replication.c - replicaofCommand 入口 */
void replicaofCommand(client *c) {
    /* 解析 host 和 port */
    /* 调用 replicationSetMaster() 设置主库信息 */
    replicationSetMaster(c->db->id, host, port);
    /* 这时 server.masterhost 被设定，server.master 为 NULL（尚未连接） */
    /* 下一次 serverCron 会调用 replicationCron() 开始连接 */
}

replicationSetMaster() 只是设置了 server.masterhost 和 server.masterport，实际连接是由定时任务 replicationCron() 发起的。这个函数每秒执行一次（由 server.hz 控制），检查是否需要连接或重连主库。

全量同步

当从库第一次连接主库，或者 PSYNC 部分同步无法进行时，就触发了全量同步（Full Sync）。全量同步是整个复制过程中代价最高的操作，我们一步一步看它发生了什么：

第一步：连接建立与握手

从库通过 connectWithMaster() 建立 TCP 连接，然后经过一套 PING-PONG 握手流程，确认主库可用后，发送 PSYNC ? -1——其中的 ? 表示没有 replid，-1 表示偏移量未知，这等价于告诉主库"我需要全量同步"。

第二步：主库生成 RDB

主库收到 PSYNC 请求后，发现无法进行部分同步（因为从库没有任何历史信息），于是触发 rdbSaveToSlavesSockets() 或 rdbSave()：

/* replication.c - 主库处理全量同步请求 */
void replicationCron(void) {
    /* ... 省略连接管理逻辑 ... */
    /* 如果复本请求全量同步，主库开始 BGSAVE */
    if (server.rdb_child_pid == -1) {
        /* 没有 RDB 子进程在跑，启动 BGSAVE */
        startBgsaveForReplication(c->slave_capa);
    }
    /* 如果已经在 BGSAVE，检查是否是给这个从库用的 */
    /* 如果是，把这个从库加入等待列表 */
}

主库把自己的 RDB 子进程（rdbSaveBackground()）生成的快照发送给从库。如果开启了无盘复制（下一章讲），RDB 数据通过 socket 直接传输，不落盘。

第三步：从库清空与加载

从库收到 RDB 文件后，首先清空自己当前的所有数据，然后加载 RDB 文件。这是一个非常暴力的操作——如果从库上本来有 10GB 数据，一下全没了，全部替换为主库的数据。

/* replication.c - 从库读取 RDB 的核心逻辑 */
void readSyncBulkPayload(aeEventLoop *el, int fd, void *privdata, int mask) {
    /* ... 读取 socket 上的 RDB 数据流 ... */

    if (eof_reached) {
        /* RDB 文件接收完毕 */

        /* 1. 清空当前数据库 */
        emptyDb(-1, EMPTYDB_NO_FLAGS, replicationEmptyDbCallback);

        /* 2. 将 RDB 数据加载到内存 */
        if (rdbLoad(ri, RDBFLAGS_REPLICATION) != C_OK) {
            serverLog(LL_WARNING, "失败读取 RDB");
            freeClient(c);
            return;
        }

        /* 3. 加载完成后，从库的身份变为"在线从库" */
        server.repl_state = REPL_STATE_CONNECTED;
        /* 标记需要发送 ACK */
        server.repl_send_ack = 1;
    }
}

第四步：后续数据回放

你可能会问：主库在 BGSAVE 期间，新的写请求怎么处理？答案是通过 replication backlog 缓冲区暂存。RDB 生成完毕后，主库把 backlog 中的增量数据追加发送给从库。这样从库的数据才能"追上"主库。

整个全量同步期间，主库的 CPU 和磁盘 IO 消耗都非常大——子进程 fork 后，虽然用了 COW（写时复制）机制，但如果写入量大，内存页复制仍可能造成数十毫秒的延迟尖刺。

PSYNC 部分重同步

全量同步的开销太大了：如果从库只是断网了 5 秒钟重新连上，就要把几十 GB 的数据重新传输一遍，这显然不能忍。Redis 2.8 引入的 PSYNC（Partial SYNC）就是为了解决这个问题的。

PSYNC 的核心思想是：记录主库的唯一标识符（replication ID, replid）和数据流中的位置（offset），重新连接时，只需要传输从库断开期间缺失的那部分数据。

每个 Redis 主库在启动时都会生成一个 40 位十六进制的随机字符串作为 replid（replication ID）。这个 ID 唯一标识了主库的数据版本。如果主库重启后数据变了（比如清空了），replid 也会变更。从库在初次同步时拿到这个 ID，然后每次断线重连时，发送 PSYNC <replid> <offset>：

/* replication.c - 从库发送 PSYNC */
void slaveTryPartialResynchronization(int fd) {
    /* 构造 PSYNC 命令 */
    char psync_cmd[256];
    psync_len = snprintf(psync_cmd, sizeof(psync_cmd),
                         "PSYNC %s %lld",
                         server.master_replid,       /* 上次同步的 replid */
                         server.master_initial_offset); /* 上次同步的偏移量 */

    /* 发送给主库 */
    write(fd, psync_cmd, psync_len);
    /* 等待主库响应 */
}

主库收到 PSYNC 命令后，在 masterTryPartialResynchronization() 中做判断：

/* replication.c - 主库判断能否部分同步 */
int masterTryPartialResynchronization(client *c) {
    /* 1. 检查 replid 是否匹配 */
    if (strcasecmp(c->slave_replid, server.replid) &&
        strcasecmp(c->slave_replid, server.replid2)) {
        /* replid 不匹配，必须全量同步 */
        goto need_full_resync;
    }

    /* 2. 检查 offset 是否还在 backlog 中 */
    if (!server.repl_backlog ||
        c->slave_repl_offset < server.repl_backlog->offset ||
        c->slave_repl_offset > (server.repl_backlog->offset +
                                  server.repl_backlog->histlen))
    {
        /* offset 在 backlog 范围之外，必须全量同步 */
        goto need_full_resync;
    }

    /* 3. 可以部分同步！从 backlog 中取出缺失数据发送 */
    /* ... 发送 backlog 中从 offset 开始的数据 ... */
    return C_OK;

need_full_resync:
    /* 回复 +FULLRESYNC <replid> <offset> */
    return C_ERR;
}

PSYNC 的关键在于：replid 匹配且 offset 在 backlog 范围内，就能做部分同步。只要有任何一个条件不满足，就只能降级为全量同步。

Backlog 机制

backlog（复制积压缓冲区）是实现部分同步的核心数据结构。它是一个环形缓冲区，主库上所有写入命令的传输数据都会被写入这个缓冲区。当从库断线重连时，只要它的 offset 还在缓冲区的有效范围内，就能通过 backlog 找到缺失的数据。

/* server.h - backlog 相关结构 */
struct repl_backlog {
    char *buf;              /* 环形缓冲区内存指针 */
    long long offset;       /* backlog 中最早数据的全局偏移量 */
    long long histlen;      /* 当前有效数据长度 */
    long long idx;          /* 下一次写入的位置（在 buf 中的索引） */
};

/* server.h - server 结构中的相关字段 */
struct redisServer {
    /* ... */
    char replid[CONFIG_RUN_ID_SIZE+1];       /* 当前 replid */
    char replid2[CONFIG_RUN_ID_SIZE+1];      /* 上一个 replid */
    long long master_repl_offset;             /* 当前全局复制偏移量 */
    struct repl_backlog *repl_backlog;        /* backlog 指针 */
    long long repl_backlog_size;              /* backlog 大小（默认 1MB） */
    long long repl_backlog_histlen;           /* 实际有效数据长度 */
    long long repl_backlog_idx;               /* 写指针位置 */
    time_t repl_backlog_time_limit;           /* 无从库时保留时间 */
    /* ... */
};

backlog 的大小通过 repl-backlog-size 配置（默认 1MB）。它是一个环形缓冲区：当写入数据超过缓冲区大小时，最早的数据会被覆盖。这正是"offset 超出范围"的本质原因——从库落后太多，它在等的那部分数据已被新数据覆盖了。

backlog 的写入由 feedReplicationBuffer() 完成，后面第21章会有详细源码分析。这里先记住它的核心特性：

  • 只记录命令传播的数据（即写入命令导致的数据变更），不记录全量同步时的 RDB 内容。
  • 如果当前没有从库连接，但配置了 repl-backlog-size，backlog 会保留 repl-backlog-ttl（默认 3600 秒）后释放。
  • backlog 大小不能动态缩小，只能通过重启或者修改配置后 CONFIG SET 调整。
踩坑实录：生产环境中最常见的一个"悲剧场景"是这样的：A 团队上线了 Redis 主从架构，从库连接正常，一切似乎很完美。但某个夜晚，网络抖动导致从库断线重连——从库发送 PSYNC，请求部分同步。然而，因为 backlog 使用默认的 1MB，而主库的写入量是每秒 5MB，1MB 的缓冲区在短短 200ms 内就被新数据填满了。从库断线的 3 秒期间产生的 15MB 数据早已覆盖了它需要的部分。于是从库的 PSYNC 被拒绝，回退到全量同步。全量同步期间，主库 fork 子进程产生 10GB 的 RDB，CPU 飙升到 80%，latency 从 1ms 涨到 200ms——连锁反应导致其他从库也超时断线，然后它们也请求全量同步，形成了"复制雪崩"。

教训：backlog 大小要根据"业务写入峰值 × 可接受的最大断线时间"来估算。对于写密集场景（每秒上百 MB 写入），建议将 repl-backlog-size 设置为 100MB~512MB，确保从库断线数分钟内仍能部分同步。

全量同步 vs 部分同步

为了帮助你直观地理解两种同步方式的差异，我们用一个对比表总结：

对比维度 全量同步（Full Sync） 部分同步（Partial Sync）
触发条件 首次连接、replid 不匹配、offset 落后超过 backlog 断线重连，且 offset 在 backlog 范围内
数据传输量 整个数据集（GB ~ TB 级） 仅断线期间的增量数据（KB ~ MB 级）
主库开销 fork 子进程、BGSAVE、大量磁盘 IO / 网络带宽 直接读取 backlog 发送，几乎无额外开销
从库影响 清空所有数据，重新加载 RDB，期间不可服务 在线接收增量数据，对服务几乎无影响
耗时 分钟~小时级（取决于数据大小和网络带宽） 毫秒~秒级
网络带宽 极高。一次全量同步可能耗尽机房带宽 极低。仅传输增量命令

从表可以看得很清楚：全量同步是最后的无奈选择。生产环境的优化目标应该是"让 PSYNC 成功率尽可能接近 100%"——这需要通过合理配置 backlog 大小、控制网络抖动、以及使用稳定的网络基础设施来实现。

监控指标：通过 INFO replication 可以看到 sync_partial_ok 和 sync_partial_err 两个计数器，分别表示部分同步成功和失败的次数。如果 sync_partial_err 持续增长，说明你的 backlog 太小或者网络抖动太频繁，需要针对性处理。
