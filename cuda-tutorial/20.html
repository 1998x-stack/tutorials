多GPU编程

大家好，我是你们的讲师。欢迎来到第20章。到目前为止，我们的所有讨论都基于"单张GPU"这个假设。但在真实的生产环境中，一张GPU几乎永远不够用——训练GPT-4需要数万张GPU跑上几个月；即使是一个中等规模的推荐系统推理服务，也常常需要4到8张GPU才能满足延迟和吞吐要求。

这一章，我们会从硬件拓扑讲起，一步步带你掌握多GPU环境下的核心编程技术：设备管理、Peer-to-Peer访问、NCCL集体通信，以及三种主流的并行策略。学完本章，你会知道"如何让多张GPU不仅不打架，而且高效协作"。

多GPU拓扑结构

多GPU系统的性能瓶颈往往不是GPU本身，而是GPU之间的通信。如果你把两张GPU插在PCIe插槽上，它们之间通过PCIe交换数据，带宽和延迟都远不如通过NVLink直接互联。了解硬件拓扑是写出高效多GPU代码的第一步。

互连方式 典型带宽 延迟 拓扑 适用场景
PCIe Gen4 x16 ~32 GB/s ~1微秒 树状（通过CPU Root Complex） CPU-GPU通信、低成本多卡
PCIe Gen5 x16 ~64 GB/s ~0.8微秒 树状 中端多GPU系统
NVLink 2.0 (Volta) ~300 GB/s (6 links) ~0.2微秒 混合立方网格 V100集群、P2P频繁访问
NVLink 3.0 (Ampere) ~600 GB/s (12 links) ~0.15微秒 全互联（A100 8-GPU） A100 DGX系统、大模型训练
NVLink 4.0 (Hopper) ~900 GB/s (18 links) ~0.1微秒 NVSwitch全互联 H100 DGX、大规模分布式训练
NVSwitch 聚合可达 7.2 TB/s ~0.1微秒 全交换（所有GPU任意互联） DGX Pod、超大规模集群

这张表的核心信息是：NVLink比PCIe快一个数量级。如果你的多GPU工作负载有大量跨卡通信，一定尽量让通信走NVLink。很多人在8卡机器上训练时发现一个GPU的梯度同步比计算还慢，排查到最后的根因是——驱动没有启用P2P访问，数据被迫绕道CPU内存走PCIe，带宽从600 GB/s降到了32 GB/s。这个问题我们马上会讲到如何避免。

设备管理与选择

多GPU编程的基本操作非常简单——CUDA用整数索引标识每张GPU。核心API就三个：cudaGetDeviceCount 查询设备数量，cudaSetDevice 选择当前设备，cudaGetDevice 获取当前设备ID。每个host线程都有自己当前的device上下文，互不干扰。

// 多GPU初始化模板

int main() {
    int deviceCount;
    cudaGetDeviceCount(&deviceCount);
    printf("检测到 %d 张GPU\n", deviceCount);

    // 为每张GPU分配独立的内存和stream
    const int N = 1 << 20;
    float **d_data = (float**)malloc(deviceCount * sizeof(float*));
    cudaStream_t *streams = (cudaStream_t*)malloc(deviceCount * sizeof(cudaStream_t));

    for (int dev = 0; dev < deviceCount; dev++) {
        cudaSetDevice(dev);  // 切换到第dev张GPU

        // 查询设备属性
        cudaDeviceProp prop;
        cudaGetDeviceProperties(&prop, dev);
        printf("GPU %d: %s, 显存 %zu MB, SM数量 %d\n",
               dev, prop.name, prop.totalGlobalMem / (1024*1024), prop.multiProcessorCount);

        // 在当前设备上分配显存和创建stream
        cudaMalloc(&d_data[dev], N * sizeof(float));
        cudaStreamCreate(&streams[dev]);
    }

    // 每个设备上的操作
    for (int dev = 0; dev < deviceCount; dev++) {
        cudaSetDevice(dev);
        myKernel<<<grid, block, 0, streams[dev]>>>(d_data[dev], N);
    }

    // 同步所有设备
    for (int dev = 0; dev < deviceCount; dev++) {
        cudaSetDevice(dev);
        cudaStreamSynchronize(streams[dev]);
    }

    // 清理
    for (int dev = 0; dev < deviceCount; dev++) {
        cudaSetDevice(dev);
        cudaStreamDestroy(streams[dev]);
        cudaFree(d_data[dev]);
    }
    free(d_data);
    free(streams);
    return 0;
}
黄金法则： 在任何cudaMalloc、cudaMemcpy或kernel launch之前，确保已经通过 cudaSetDevice 切换到了正确的GPU。一个常见的习惯是——在每个线程/循环的开头就显式设置device，不要依赖"当前device没变"这种假设。显式总比隐式安全。

Peer-to-Peer（P2P）访问

多GPU编程中最强大的特性之一就是直接内存访问——让一张GPU直接读写另一张GPU的显存，而不需要经过CPU做中转。启用方式很简单：

// Peer-to-Peer (P2P) 访问示例

int main() {
    int devCount;
    cudaGetDeviceCount(&devCount);

    // 启用所有GPU之间的P2P访问
    for (int i = 0; i < devCount; i++) {
        for (int j = 0; j < devCount; j++) {
            if (i == j) continue;

            int canAccessPeer = 0;
            cudaDeviceCanAccessPeer(&canAccessPeer, i, j);

            if (canAccessPeer) {
                cudaSetDevice(i);
                cudaDeviceEnablePeerAccess(j, 0);
                printf("GPU %d 与 GPU %d 已建立P2P连接\n", i, j);
            } else {
                printf("GPU %d 无法直接访问 GPU %d（通过PCIe绕行）\n", i, j);
            }
        }
    }

    // P2P memcpy —— 直接从GPU 0拷贝到GPU 1，不经过CPU
    const int N = 1 << 24;  // 64M floats = 256MB
    float *d_ptr0, *d_ptr1;

    cudaSetDevice(0);
    cudaMalloc(&d_ptr0, N * sizeof(float));

    cudaSetDevice(1);
    cudaMalloc(&d_ptr1, N * sizeof(float));

    // GPU 0 → GPU 1 直接拷贝
    cudaMemcpyPeer(d_ptr1, 1, d_ptr0, 0, N * sizeof(float));
    // 等价于不经过CPU的 cudaMemcpy —— 如果P2P已启用

    // 还可以直接在kernel中访问另一张GPU的内存（通过指针）
    // 在GPU 1的kernel中读写d_ptr0指向的GPU 0的显存
    cudaSetDevice(1);
    p2pKernel<<<grid, block>>>(d_ptr1, d_ptr0, N);  // 直接读写GPU 0的内存

    cudaDeviceSynchronize();

    cudaSetDevice(0);
    cudaFree(d_ptr0);
    cudaSetDevice(1);
    cudaFree(d_ptr1);
    return 0;
}
重要提醒： P2P访问不是在所有GPU组合之间都可用的。两个GPU必须属于同一棵PCIe树，或者通过NVLink直接相连。另外，P2P一旦启用，两张GPU的上下文就会被绑定——直到进程退出前，你无法单独销毁其中一个。所以建议在程序初始化时统一启用所有需要的P2P连接，不要在运行中动态开启和关闭。

NCCL入门

如果说P2P是"点对点"通信的脚手架，那NCCL（NVIDIA Collective Communications Library）就是多GPU通信的"预制装配式房屋"。NCCL提供了AllReduce、Broadcast、AllGather、ReduceScatter等集体通信原语，经过深度优化，可以充分利用NVLink和NVSwitch的带宽。当今所有主流深度学习框架（PyTorch DDP、Horovod、DeepSpeed）的多GPU通信底层都依赖NCCL。

// NCCL AllReduce 示例

#include <nccl.h>
#include <cuda_runtime.h>

int main() {
    int nDevices = 4;
    int dataSize = 1 << 20;  // 1M floats per GPU

    float **d_send = (float**)malloc(nDevices * sizeof(float*));
    float **d_recv = (float**)malloc(nDevices * sizeof(float*));
    ncclComm_t *comms = (ncclComm_t*)malloc(nDevices * sizeof(ncclComm_t));
    cudaStream_t *streams = (cudaStream_t*)malloc(nDevices * sizeof(cudaStream_t));

    // 初始化每张GPU
    for (int dev = 0; dev < nDevices; dev++) {
        cudaSetDevice(dev);
        cudaMalloc(&d_send[dev], dataSize * sizeof(float));
        cudaMalloc(&d_recv[dev], dataSize * sizeof(float));
        cudaStreamCreate(&streams[dev]);
    }

    // 初始化NCCL通信器 —— 所有GPU联合创建
    ncclCommInitAll(comms, nDevices, NULL);

    // 执行AllReduce：每张GPU发送d_send中的数据，
    // 所有GPU都收到各卡数据求和后的结果到d_recv
    for (int dev = 0; dev < nDevices; dev++) {
        cudaSetDevice(dev);
        ncclAllReduce((const void*)d_send[dev],
                      (void*)d_recv[dev],
                      dataSize,
                      ncclFloat,
                      ncclSum,
                      comms[dev],
                      streams[dev]);
    }

    // 同步所有stream和NCCL操作
    for (int dev = 0; dev < nDevices; dev++) {
        cudaSetDevice(dev);
        cudaStreamSynchronize(streams[dev]);
    }

    // 清理
    for (int dev = 0; dev < nDevices; dev++) {
        ncclCommDestroy(comms[dev]);
        cudaSetDevice(dev);
        cudaStreamDestroy(streams[dev]);
        cudaFree(d_send[dev]);
        cudaFree(d_recv[dev]);
    }
    free(d_send); free(d_recv); free(comms); free(streams);
    return 0;
}

注意NCCL的一个关键点：AllReduce是同步阻塞的——所有参与卡必须同时调用，否则会出现死锁。这也是为什么NCCL通常和MPI或分布式框架配合使用，确保所有rank同步推进。

多GPU编程模式

了解了底层通信工具后，我们来谈谈上层策略。多GPU并行编程主要有三种模式，下面这张表总结了它们的核心差异：

并行模式 核心思想 通信需求 典型场景 扩展性
数据并行 每张GPU持完整模型副本，处理不同数据分片，梯度同步 高（每步AllReduce梯度） 大多数深度学习训练 好（近线性加速，但受限于batch size）
模型并行 模型的不同层分布在不同的GPU上，数据在层间流水 中（每层之间传递activations） 模型太大放不进一张卡（如GPT-3） 中（存在"短板GPU"问题）
流水线并行 模型按层切分到多卡，数据分micro-batch流水线执行 低-中（仅边界传递） 大模型训练，如GPipe、PipeDream 中-好（需平衡各阶段计算量）

数据并行是最简单也最常用的模式。PyTorch的 DistributedDataParallel 就是它的实现——每张GPU上跑一份完整的模型，每个step处理不同的数据，然后用AllReduce同步梯度。它的扩展性极好（8卡接近8倍加速），但如果单张GPU放不下模型，就需要模型并行或流水线并行。

模型并行把模型的不同层放在不同GPU上——比如Transformer的前6层在GPU 0，后6层在GPU 1。问题是GPU 0算的时候GPU 1在空等，GPU利用不充分。流水线并行在模型并行的基础上引入了micro-batch——把一个大batch切成多个小batch流水线式地送入。这样每张GPU都在持续工作而不是空闲等待。

真实的踩坑故事： 我们团队曾经在一台DGX A100（8张GPU）上训练一个推荐模型。训练一直很稳定，但隔几天就会出现一次loss spike，然后模型发散。排查了数据、学习率、梯度裁剪……所有常见的原因都排除了。后来我们加了详细的日志，发现每次loss spike都伴随着cudaSetDevice的调用。最终定位到：代码中有一个预处理线程在某个罕见路径下没有调用cudaSetDevice，导致它默认使用了device 0，而device 0的主训练线程正好在做forward——两个线程同时往同一张GPU上提交kernel，产生了干扰。这个bug的修复只有一行代码：在预处理线程的开头加上 cudaSetDevice(local_rank)。一个无声的bug，两个星期的排查。多GPU编程中，每一行看起来"多余"的设备管理代码，背后都是前人踩过的坑。
