第9章 Lemon解析器与语法

大家好，我是你们的讲师。上一章我们看了 SQLite 的"分词器"是怎么把 SQL 文本拆成 token 的。但 token 只是一堆离散的碎片——谁来把这些碎片按语法规则组装成有意义的结构呢？答案就是这一章的主题：Lemon 解析器。

在编译器领域，YACC（Yet Another Compiler-Compiler）和 Bison 是解析器生成器的"行业标准"。但 SQLite 做了一个与众不同的选择——它使用了自己的自研解析器生成器 Lemon，也用同一工具生成自己的 SQL 文法解析器。

Lemon 是 Dr. Hipp 在 1990 年代早期开发的，比 SQLite 本身还要年长。它的设计目标非常明确：可重入（reentrant）、线程安全、零全局变量。这些特性恰好是 SQLite 需要的——因为 SQLite 要支持多线程环境下的并发数据库连接访问。

为什么选择 Lemon

你可能要问：既然 YACC/Bison 那么成熟、文档那么丰富，为什么 SQLite 非要另起炉灶？答案藏在编译器设计的几个"痛点"里。

特性LemonYACC / Bison
可重入性原生支持，无全局变量需要 %pure-parser 声明
内存分配调用者提供 malloc，完全控制内部 malloc，难以注入
代码风格生成的 C 代码可读性强紧凑但难读
错误处理内置 error token 机制需要手写 yyerror
模板引擎最终 action 代码保留在生成文件中宏展开，调试更难
依赖性零外部依赖需要配套运行时库

最关键的一点是 可重入性。在 SQLite 中，同一个进程可能有多个线程同时执行 sqlite3_prepare_v2()。如果解析器使用全局变量来维护解析状态，就会出现竞态条件。Lemon 从一开始就设计为无全局状态——所有状态都存储在传入的 yyParser 指针中，解析器函数本身是纯函数的。

另外，Lemon 生成的 C 代码可读性相当不错。你打开 src/parse.c 看看会发现变量命名清晰、逻辑分段明确。这对于 SQLite 长期维护来说极其重要——毕竟如果生成的代码人类完全看不懂，那出了问题根本没法调试。

💡 Lemon 的另一个特色是它的 fallback token 机制。你可以为某个 token 指定 fallback：当解析器遇到一个无法处理的 token 时，如果该 token 有 fallback 定义，解析器会尝试用 fallback token 来匹配文法。这为实现 SQL 的"关键字可以作为标识符"功能提供了优雅的解决方案。

parse.y 文件结构

SQLite 的文法定义在 src/parse.y 中，这个文件约有 1500 行，是 Lemon 的输入源文件。它的结构遵循 Lemon 输入格式：

/* 第一部分：类型声明和选项 */
%token_prefix TK_
%token_destructor { sqlite3FreeTokens(pParse, $$); }
%name sqlite3Parser
%include {
    /* #include 头文件，类型定义 */
}

/* 第二部分：类型定义 */
%type select {Select*}
%type expr {Expr*}
%type oneselect {Select*}
%type id {Token}

/* 第三部分：文法规则与 action 代码 */
select(A) ::= SELECT select_opts selcollist from_clause(From) \
              where_clause(Where) groupby_clause group_clause_opt \
              having_clause orderby_clause limit_clause. {
    A = sqlite3SelectNew(pParse, ...);
}

expr(A) ::= expr(X) PLUS expr(Y). {
    A = sqlite3ExprAlloc(pParse, TK_PLUS, &X, &Y, 0);
}

cmd ::= CREATE TABLE ifnotexists nm(E) database_kw_opt . {
    sqlite3StartTable(pParse, &E, 0);
}

文法规则的每个 production 后面跟着用大括号包裹的 action 代码——当这条规则被归约（reduce）时执行的 C 代码。action 代码中可以使用 $$ 表示当前非终结符的值，$1、$2 表示各符号的值。Lemon 没有 YACC 的 $$ = $1 这种隐式赋值，而是要求你在 action 中显式赋值，这让生成的代码更清晰。

在 parse.y 中还定义了 token 的优先级和结合性，用于解决 Shift/Reduce 冲突：

/* 优先级定义，从低到高 */
%left OR.
%left AND.
%right NOT.
%left EQ NE IS ISNULL NOTNULL IN LIKE GLOB BETWEEN REGSTR.
%left GT GE LT LE.
%left PLUS MINUS.
%left STAR SLASH REM.
%left CONCAT.
%left UMINUS UPLUS.

通过这样的优先级声明，Lemon 可以自动处理像 1 + 2 * 3 这样的运算符优先级问题——STAR 的优先级高于 PLUS，所以解析器知道要先归约乘法。

Lemon 的工作流程

Lemon 自身的代码在 tool/lemon.c 中，约 12000 行。它的工作可以概括为"三步走"：

  1. 读入 .y 文件 — 解析文法定义，构建文法规则的内部表示
  2. 构建 LALR(1) 分析表 — 计算 First 集、Follow 集、构建 LR 状态机
  3. 生成输出文件 — 输出 parse.c（解析器实现）和 parse.h（token 枚举定义）

构建过程由 Makefile 驱动。在源码树的根目录运行 make 时，main.mk 中的规则会先编译 Lemon，然后运行 Lemon 处理 parse.y：

# main.mk 中的相关规则
lemon: $(TOP)/tool/lemon.c
    $(CC) -o lemon $(TOP)/tool/lemon.c

parse.c parse.h: $(TOP)/src/parse.y lemon
    ./lemon $(OPTIONS) $(TOP)/src/parse.y

输出文件 parse.c 是一个自包含的 LALR(1) 解析器，包含完整的分析表（action table + goto table）和解析引擎（sqlite3Parser 系列函数）。它不依赖 Lemon 运行时，可以直接编译进 SQLite 库。

Lemon 的代码生成能力很强——parse.c 中嵌入的分析表经过精心编码，使用压缩编码方式存储（利用 Lemon 的 compress 选项），使得生成的表大小最小化。对于嵌入式场景，这种省出来的几百 KB 可能是决定性的。

Shift/Reduce 解析过程

Lemon 在运行时维护一个状态栈，栈上的每个元素代表一个 LR 状态。对于每个输入 token，解析器执行以下操作：

  1. 查表 — 在当前状态下，根据输入 token 的类别查 action 表
  2. Shift — 如果查到的是 "shift"，将当前 token 和下一个状态压栈，继续读入下一个 token
  3. Reduce — 如果查到的是 "reduce"，弹出栈顶的 N 个符号（对应文法的右部），执行 action 代码，将左部非终结符和新状态压栈
  4. Accept — 解析成功。触发条件是归约到起始符号且输入结束。
  5. Error — 查表结果为错误，触发错误处理

让我们用 SELECT 1 + 2 来走一遍解析过程：

状态栈（栈顶在右）       输入 token            动作
────────────────────────────────────────────────────
0                        TK_SELECT              shift, goto 状态3
0 3                      TK_INTEGER(1)          shift, goto 状态X
0 3 X                    TK_PLUS                reduce: expr → INTEGER
0 3                      TK_PLUS                shift, goto 状态Y
0 3 Y                    TK_INTEGER(2)          shift, goto 状态Z
0 3 Y Z                  (结束)                  reduce: expr → expr PLUS expr
0 3                      (结束)                  reduce: select → SELECT expr
0                        (结束)                  accept!

每一步的 shift 或 reduce 动作都是由 Lemon 的分析表决定的。这个分析表在 parse.c 中以压缩形式存储，称为 ParseAction[] 和 ParseReduce[] 两个大数组。

💡 SQLite 中有一个非常巧妙的优化：由于 SQL 语句通常很短，解析过程的递归深度很浅，Lemon 的默认栈大小（约 100 个元素）几乎从不溢出。这让 SQLite 避免了解析时的动态栈扩展开销。

Parse 上下文对象

Lemon 解析器本身只负责"语法是否正确"的判断和语法树的初步构建。实际的语义动作（比如为 SELECT 创建 Select 结构体、为表达式创建 Expr 节点）是在 action 代码中通过 Parse 对象完成的。

Parse 结构体定义在 src/sqliteInt.h 中，是整个编译前端的"上下文对象"。它包含了：

字段用途
db数据库连接指针，用于内存分配和错误报告
pVdbe当前正在构建的 VDBE 程序（字节码）
nErr解析过程中的错误计数
zErrMsg错误信息字符串
pNewTableCREATE TABLE 时构建的表定义
pNewTriggerCREATE TRIGGER 时构建的触发器定义
sLastToken最后一个 token，用于错误定位
aLabelVDBE label 表，用于跳转地址回填
nLabellabel 数量
ckptSave错误恢复时的检查点状态

当解析器进行 reduce 时，action 代码中调用 sqlite3SelectNew()、sqlite3ExprAlloc() 等函数，它们内部操作的就是 pParse 所指向的 Parse 对象。这些函数还会把生成的 VDBE 指令添加到 pParse->pVdbe 中，实现"一边解析一边生成字节码"的"单通道"编译模式。

这种设计有一个重要后果：如果你在解析器的 action 中写错了代码，会导致生成的字节码有问题，但解析过程本身可以顺利完成。这也是为什么 SQLite 有大量针对 VDBE 程序正确性的测试——解析器的"语法检查"和"代码生成"是紧密耦合的。

实战：添加一条 SQL 语法

现在我们来做一个极有实战价值的练习——尝试在 SQLite 中添加一条新的 SQL 语句。假设我们要添加一个 SHOW TABLES 命令（MySQL 风格，虽然 SQLite 没有原生支持）。

完整的实现路径如下：

  1. 在 parse.y 中添加文法规则：
/* 在 parse.y 的 cmd 规则中加入新分支 */
cmd ::= SHOW TABLES. {
    sqlite3ShowTable(pParse);
}
  1. 添加 token 定义： 由于 SHOW 已经是 SQL 的保留词（用于触发器等上下文），实际上 TK_SHOW 已经存在。但 TABLES 如果还不是关键词，需要添加。
/* 在 keywordhash.h 对应的生成脚本中 */
/* 添加 TABLES 关键词，ID 为 TK_TABLES */
  1. 实现语义函数： 在 src/build.c 或合适的文件中添加：
void sqlite3ShowTable(Parse *pParse){
    sqlite3 *db = pParse->db;
    /* 生成 VDBE 字节码 */
    sqlite3VdbeAddOp4(v, OP_ShowTable, 0, 0, 0, 0, 0);
}
  1. 在 VDBE 中添加对应的 opcode： 在 src/vdbe.c 中实现 OP_ShowTable 的执行逻辑。

从"定义文法"到"YACC 文件中写规则"到"实现 action"到"生成 VDBE 指令"——这就是一条完整的前端到后端链路。理解了这条链路，你就能在 SQLite 上做任何定制化的 SQL 扩展。

⚠️ 踩坑实录：我在尝试给 SQLite 添加自定义语法时踩过一个经典的坑——Shift/Reduce 冲突。新加的语法规则可能会与已有的规则产生歧义，比如 SHOW TABLES 中的 TABLES 如果也是合法的标识符，解析器不知道应该 shift 还是 reduce。解决方法是利用 Lemon 的优先级声明，或调整文法使其无歧义。记得每次修改 parse.y 后查看 Lemon 输出的冲突报告。
