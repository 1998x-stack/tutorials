第11章 实现上下文管理

大家好，我是你们的讲师。上一章我们分析了 Claude Code 的 5 阶段压缩管线——从 Budget Reduction 到 Auto-Compact，每一层都在特定的水位上主动释放上下文空间。但理论归理论，是时候动手了。这一章我们要实现一个完整的上下文管理器，把它集成到我们的 v2 Agent 中。

在开始之前，先讲一个让人哭笑不得的场景。我见过一个团队做的 Agent，为了"防止上下文溢出"，在每轮对话结束后直接把整个消息历史清空，只保留最后一轮。结果模型每轮都"失忆"，同一个问题反复问用户。这就是"过度压缩"的极端例子——上下文管理不是"有了就行"，而是要在保留关键信息和控制 token 预算之间找到平衡。这恰恰是最难的地方。

上下文管理的设计目标

一个优秀的上下文管理器应该同时满足三个目标：

  1. 不超预算——任何时候发送给 API 的消息列表都不能超过模型的最大上下文窗口。这是硬约束，没有商量余地。
  2. 不丢关键信息——压缩可以丢失细节，但不能丢失关键决策、文件路径、错误消息等对任务至关重要的信息。
  3. 对 Agent 透明——Agent 不应该感知到上下文管理器的存在。它不需要操心"上下文满了怎么办"，应该像使用无限内存一样自然地使用上下文。

这三个目标之间存在天然的张力。"不超预算"意味着必须压缩；"不丢关键信息"意味着压缩要有选择性；"对 Agent 透明"则要求压缩不能改变 Agent 的行为。处理这个张力的能力，就是上下文管理器质量的分水岭。

Token 估算实现

第一步是实现可靠的 token 估算。我们用 tiktoken 库配合 cl100k_base 编码来做近似计数。为什么用 tiktoken 而不是 Anthropic 的官方 tokenizer？上一章说过了——速度优先，精度足够。

import tiktoken
import json
from datetime import datetime
from typing import Optional

ENCODING = tiktoken.get_encoding("cl100k_base")

def count_tokens(text: str) -> int:
    return len(ENCODING.encode(text))

def count_messages(messages: list[dict]) -> int:
    """精确计算消息列表的 token 数，含格式开销"""
    total = 0
    for msg in messages:
        total += 4  # 每条消息的元数据开销
        for key, value in msg.items():
            if isinstance(value, str):
                total += count_tokens(value)
                total += count_tokens(key)
            elif isinstance(value, list):
                for item in value:
                    if isinstance(item, dict):
                        total += count_block_tokens(item)
                    elif isinstance(item, str):
                        total += count_tokens(item)
            elif isinstance(value, dict):
                total += count_block_tokens(value)
    return total

def count_block_tokens(block: dict) -> int:
    """计算 content block 的 token 数"""
    total = 0
    for k, v in block.items():
        total += count_tokens(k)
        if isinstance(v, str):
            total += count_tokens(v)
        elif isinstance(v, dict):
            total += count_block_tokens(v)
    return total

滑动窗口截断策略

当上下文接近预算上限时，最简单的策略是丢弃最早的历史消息。但"扔掉最老的"有一个关键改进：保留第一条用户消息。

为什么保留第一条？因为用户最初的问题往往定义了整个任务的目标和范围。如果把它也丢掉了，Agent 可能逐渐偏离最初的意图。想象一个对话：用户第一句说"帮我重构 auth 模块的数据库层"，然后 30 轮后上下文被截断，第一条消息消失了。Agent 从摘要中知道"在做数据库相关的事情"，但忘了"是 auth 模块"——它可能开始重构其他模块的数据库代码。

class SlidingWindowManager:
    """滑动窗口上下文管理器"""

    def __init__(self, max_tokens: int = 200_000,
                 system_prompt: str = "",
                 tool_defs: list = None,
                 safety_margin: float = 0.08):
        self.max_tokens = max_tokens
        self.system_prompt = system_prompt
        self.tool_defs = tool_defs or []
        self.safety_margin = safety_margin
        self.window_messages: list[dict] = []
        self.summary: Optional[str] = None

    def _get_budget(self) -> int:
        """计算可用于消息历史的 token 预算"""
        sys_tokens = count_tokens(self.system_prompt) if self.system_prompt else 0
        tool_tokens = sum(count_tokens(json.dumps(t)) for t in self.tool_defs)
        usable = int(self.max_tokens * (1 - self.safety_margin))
        return usable - sys_tokens - tool_tokens

    def add_message(self, message: dict):
        """追加消息，如果超预算则触发截断"""
        self.window_messages.append(message)
        if count_messages(self.window_messages) > self._get_budget():
            self._truncate()

    def _truncate(self):
        """滑动窗口截断：保留第一条 + 最后 N 条"""
        if len(self.window_messages) < 3:
            return

        first_msg = self.window_messages[0]
        candidates = self.window_messages[1:]  # 去掉第一条

        # 从最老的消息开始移除，直到预算充足
        while candidates and count_messages(
            [first_msg] + candidates
        ) > self._get_budget():
            candidates.pop(0)

        self.window_messages = [first_msg] + candidates

    def get_messages(self) -> list[dict]:
        """获取当前消息列表"""
        if self.summary:
            return [{"role": "system",
                     "content": f"[对话摘要]\n{self.summary}"}] + self.window_messages
        return self.window_messages

自动摘要生成

滑动窗口虽然简单有效，但它会丢失被丢弃消息中的所有信息。为了弥补这个缺陷，我们引入自动摘要生成——当窗口截断发生时，用一个便宜的模型把丢弃的内容压缩为结构化摘要。

关键设计决策：使用 Haiku 而不是 Sonnet 来做摘要。Haiku 的价格只有 Sonnet 的十分之一，速度却快得多。对于"理解并总结"这个任务，Haiku 的表现和 Sonnet 差距很小——因为摘要不需要创造性推理，只需要忠实提取关键信息。

class ContextManager:
    """完整的上下文管理器：滑动窗口 + 自动摘要 + 持久化"""

    def __init__(self, model_max: int = 200_000,
                 summary_model: str = "claude-haiku-3-5-20251022"):
        self.window = SlidingWindowManager(max_tokens=model_max)
        self.summary_model = summary_model
        self.summary: Optional[str] = None
        self.is_summarized: bool = False

    async def add_and_check(self, message: dict) -> list[dict]:
        """追加消息，并在必要时触发摘要"""
        self.window.add_message(message)
        current_tokens = count_messages(self.window.window_messages)

        # 如果超过 85% 预算，触发摘要
        budget = self.window._get_budget()
        usage_ratio = current_tokens / budget if budget > 0 else 1.0

        if usage_ratio > 0.85 and not self.is_summarized:
            await self._summarize_early_turns()
            self.is_summarized = True

        return self.window.get_messages()

    async def _summarize_early_turns(self):
        """使用 Haiku 对早期对话生成结构化摘要"""
        messages = self.window.window_messages
        if len(messages) < 4:
            return

        # 前半部分需要摘要，后半部分保留原始
        split_point = len(messages) // 2
        early_part = messages[:split_point]
        late_part = messages[split_point:]

        # 构造摘要请求
        summary_prompt = (
            "请为以下 AI 对话历史生成结构化摘要。"
            "保留所有重要的技术决策、文件路径、错误消息和未完成任务。\n\n"
            + json.dumps(early_part, ensure_ascii=False)
        )

        # 使用 Haiku 生成摘要（伪代码）
        summary_response = await llm_call(
            model=self.summary_model,
            messages=[{"role": "user", "content": summary_prompt}],
            max_tokens=2000
        )
        self.summary = summary_response.content

        # 用摘要 + 后半部分替换完整窗口
        self.window.window_messages = late_part
        self.window.summary = self.summary

    def get_context(self) -> list[dict]:
        """返回最终发送给 API 的上下文"""
        return self.window.get_messages()

对话持久化

除了实时管理，上下文管理器还有一个容易被忽视的职责：对话持久化。如果 Agent 运行了 30 分钟然后在最后一步崩溃了，所有中间状态就丢失了——这太浪费了。

持久化的核心设计原则是 append-only。每轮对话的每条消息都追加写入 JSONL 文件，不修改、不删除。这样即使进程崩溃，最后一条完整写入的消息就是恢复点。

import json
from pathlib import Path

class ConversationLogger:
    """对话持久化：append-only JSONL"""

    def __init__(self, log_dir: str = ".agent/logs"):
        self.log_dir = Path(log_dir)
        self.log_dir.mkdir(parents=True, exist_ok=True)
        session_id = datetime.now().strftime("%Y%m%d_%H%M%S")
        self.log_file = self.log_dir / f"session_{session_id}.jsonl"

    def log_message(self, role: str, content, metadata: dict = None):
        """追加写入一条消息"""
        entry = {
            "timestamp": datetime.now().isoformat(),
            "role": role,
            "content": content,
            "metadata": metadata or {}
        }
        with open(self.log_file, "a", encoding="utf-8") as f:
            f.write(json.dumps(entry, ensure_ascii=False) + "\n")

    def replay(self, session_id: str) -> list[dict]:
        """回放指定 session 的对话，用于调试"""
        path = self.log_dir / f"session_{session_id}.jsonl"
        if not path.exists():
            raise FileNotFoundError(f"Session {session_id} not found")
        messages = []
        with open(path, "r", encoding="utf-8") as f:
            for line in f:
                messages.append(json.loads(line))
        return messages

集成测试

让我们对比一下不同上下文管理策略的效果：

策略 实现复杂度 Token 利用率 信息保留度 Agent 行为稳定性 适用场景
无管理 低 低（容易溢出） 高 低（溢出后崩溃） 简单、轮次少的任务
滑动窗口（仅最后 N 条） 低 中 中（丢失早期关键信息） 中 对话逻辑简单、状态可重置的任务
滑动窗口 + 摘要 中 高 高（关键信息保留在摘要中） 高 大多数场景
混合（窗口 + 摘要 + Auto-Compact） 高 极高 极高 极高 复杂长任务、生产环境

我们实现的 ContextManager 属于第三行——"滑动窗口 + 摘要"策略。它在复杂度和效果之间取得了最佳平衡。要升级到第四行的"混合策略"，只需要将第10章描述的 Auto-Compact 作为摘要生成的后备方案接入即可。

集成到 Agent 后，我们来做一个简单的测试：让 Agent 执行一个 30 轮以上的长任务，然后检查上下文是否始终在预算之内，Agent 行为是否在压缩后保持一致。

工程经验：摘要模型的选择很关键。Haiku 比 Sonnet 便宜 10 倍，速度也快得多。在摘要场景中，关键不在于模型的"创造力"而在于"信息提取能力"——Haiku 在这方面表现非常好。如果你用 Sonnet 做摘要，相当于用航天发动机驱动自行车——浪费。坚持"大材小用"的反面：给任务匹配最便宜的模型。

现在，我们的 v2 Agent 已经有了引擎（queryLoop）、导航（TodoWrite）、空间管理（ContextManager）。这三件套足够应对大多数单 Agent 任务了。但 Claude Code 还有一个更强大的武器——Sub-Agent。它能像人类团队一样分工协作：一个 Agent 读代码，一个 Agent 写代码，一个 Agent 跑测试。下一章我们就来分析 Sub-Agent 的架构设计。
