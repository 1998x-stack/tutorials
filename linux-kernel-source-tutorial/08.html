大家好，我是你们的讲师。上一章我们学习了x86保护模式下的分段与特权级机制，弄清楚了GDT、LDT、DPL/RPL/CPL这些核心概念。但保护模式还有一个至关重要的机制我们还没有展开——那就是中断与异常。

如果说CPU是操作系统的"大脑"，那么中断就是驱动这颗大脑运转的"神经信号"。没有中断，CPU只能轮询一切——键盘、鼠标、硬盘、定时器——轮询的效率有多低？想象一下你在厨房盯着电水壶看它什么时候烧开，而不是去干点别的，等水壶"哔"一声再过来。中断就是那声"哔"。

引言——中断是操作系统的事件驱动引擎

在学习中断机制之前，我们需要先区分两个容易混淆的概念：中断(Interrupt)和异常(Exception)。

中断是由外部硬件设备触发的，完全是异步的——CPU在执行某条指令时，并不知道下一秒会有一个键盘按键被按下。时钟中断、键盘中断、硬盘中断都属于这一类。

异常则是由正在执行的指令本身触发的，是同步的。除零错误、缺页异常、非法指令都属于异常——它们是CPU在执行指令过程中发现"出问题了"，主动产生的事件。

虽然触发来源不同，但CPU处理中断和异常的机制是高度一致的：保存当前执行现场、跳转到对应的处理函数、处理完成后恢复现场继续执行。理解了这个统一的处理框架，你就理解了中断系统的核心。

关键区别：中断是异步且可以被屏蔽的（IF标志位控制），异常是同步且不可屏蔽的（如果CPU无法继续执行的话）。外设中断不影响指令的完成，而异常（如缺页）必须在当前指令处理完成后才能返回——因为异常发生在当前指令执行过程中。

x86中断向量表

x86 CPU硬件定义了一张中断向量表(IVT, Interrupt Vector Table)——实际上是一组从0到255的编号，每个编号对应一个中断或异常。CPU拿到编号后，通过IDT（中断描述符表）找到对应的处理函数入口。

这256个向量分配如下：

  • 0-31：Intel保留的异常和NMI（不可屏蔽中断）
  • 32-255：用户可定义的中断（通常用于外设中断和系统调用）

下面是Linux 0.11中用到的一些关键中断向量：

向量号名称类型说明
0#DE异常除零错误（Divide Error）
1#DB异常调试异常（Debug）
3#BP异常断点异常（Breakpoint）——int 3指令
6#UD异常非法指令（Invalid Opcode）
8#DF异常双重错误（Double Fault）
13#GP异常通用保护错误（General Protection Fault）
14#PF异常缺页异常（Page Fault）
16#MF异常x87浮点异常
32IRQ0中断时钟中断（8259定时器）
33IRQ1中断键盘中断
36IRQ4中断串口1
38IRQ6中断软盘驱动
39IRQ7中断并口/打印机
44IRQ12中断PS/2鼠标
46IRQ14中断硬盘IDE主通道
0x80int 0x80中断Linux系统调用入口

注意0x80——这个在Linux中具有特殊地位的向量号。Linus选择0x80作为系统调用入口，是因为它在用户空间和内核空间都有明确的定义，不会与硬件中断冲突。

IDT与门描述符

在保护模式下，中断/异常的处理入口不再像实模式那样通过IVT（中断向量表）查地址，而是通过IDT（中断描述符表，Interrupt Descriptor Table）。IDT中每个表项是一个8字节的门描述符，指向一个代码段中的处理函数。

IDT有三种类型的门描述符：

门类型作用自动关IF?使用场景
中断门（Interrupt Gate）进入处理函数时自动关IF，防止嵌套是硬件中断处理
陷阱门（Trap Gate）进入处理函数时不关IF否异常处理、系统调用
任务门（Task Gate）通过TSS切换到另一个任务处理—很少使用（Linux几乎不用）

中断门和陷阱门的核心区别就在于IF标志位的处理。中断门会在进入处理函数时自动将IF清零（CLI），从而屏蔽后续中断，防止中断嵌套。陷阱门则保持IF不变——这对于系统调用特别重要，因为系统调用中可能需要中断来处理其他事件。

IDT的地址由IDTR寄存器保存，通过lidt指令加载。在Linux 0.11的boot/head.s中，你可以看到IDT被初始化在物理地址0x0000处，256个门描述符占2,048字节（256 * 8 = 2048）。

! boot/head.s — 初始化IDT
!
! 将IDT设置在物理地址0x0000处
! 256个中断门，每个8字节，共2048字节

lidt  idt_descr      ! 加载IDT寄存器
...
idt_descr:
    .word 256*8-1    ! IDT的limit（2047）
    .long 0x0000     ! IDT的基地址（0x0000）

8259A中断控制器

在x86体系中，硬件中断不是直接送给CPU的，而是经过一个中断控制器统一管理。PC/AT架构使用的是两块级联的8259A可编程中断控制器（PIC）。

为什么需要8259A？因为CPU的INTR引脚只有一个，但外设可能有十几个。8259A的作用就是收集多个外设的中断请求，按优先级排序后，统一向CPU的INTR引脚发送中断信号。

Linux 0.11中，主从两个8259A级联工作：

  • 主片8259A：管理IRQ0-IRQ7，其中IRQ2用于连接从片
  • 从片8259A：管理IRQ8-IRQ15，通过IRQ2连接到主片

8259A的编程通过ICW（初始化命令字）和OCW（操作命令字）完成。在Linux 0.11的kernel/init/main.c中，我们可以看到初始化代码：

/* kernel/init/main.c — 8259A初始化 */

/* 重新对8259A中断控制器编程，把中断向量号映射到32之后 */
void init_8259A(void)
{
    /* 主片ICW1：触发方式=边沿触发，需要ICW4 */
    outb_p(0x11, 0x20);        /* 0x20 = 主片命令口 */
    /* 从片ICW1 */
    outb_p(0x11, 0xA0);        /* 0xA0 = 从片命令口 */

    /* 主片ICW2：中断向量基地址 = 0x20（即IRQ0对应向量32） */
    outb_p(0x20, 0x21);        /* 0x21 = 主片数据口 */
    /* 从片ICW2：中断向量基地址 = 0x28 */
    outb_p(0x28, 0xA1);        /* 0xA1 = 从片数据口 */

    /* 主片ICW3：IRQ2连接从片 */
    outb_p(0x04, 0x21);
    /* 从片ICW3：级联标识 */
    outb_p(0x02, 0xA1);

    /* 主片ICW4：x86模式 */
    outb_p(0x01, 0x21);
    outb_p(0x01, 0xA1);

    /* 屏蔽所有中断（先全部关掉，后续按需打开） */
    outb_p(0xFF, 0x21);
    outb_p(0xFF, 0xA1);
}

注意ICW2中设置的中断向量基地址是0x20（32）——这就是为什么时钟中断（IRQ0）对应向量32的原因。

为什么不从0开始？向量0-31被Intel保留给CPU异常，如果硬件中断也用这个范围，CPU将无法区分是异常还是外设中断。Linus将8259A映射到32开始的向量，完美避开了保留区域。系统调用也没有使用硬件中断，而是单独使用了向量0x80——从0.11开始就是如此。

asm.s源码分析——通用中断入口

当中断发生时，CPU需要保存当前执行现场，跳转到处理函数，处理完毕后再恢复现场。Linux 0.11在kernel/asm.s中实现了一套优雅的通用中断入口。

每个硬件中断都对应一个汇编入口函数，格式高度统一：

! kernel/asm.s — 硬件中断通用入口

! 时钟中断入口
_timer_interrupt:
    push %ds        ! 保存数据段寄存器
    push %es
    push %fs
    pushl %edx      ! 保存通用寄存器
    pushl %ecx
    pushl %ebx
    pushl %eax
    pushl %ebp
    pushl %esi
    pushl %edi
    movl $0x10, %eax   ! 重置数据段寄存器为内核数据段
    mov %ax, %ds
    mov %ax, %es
    movl $0x17, %eax   ! 重置为内核FS段（用户数据段）
    mov %ax, %fs
    incl _jiffies      ! 更新全局时钟滴答计数
    pushl $0x20        ! 压入中断向量号（时钟=32=0x20）
    call _do_IRQ        ! 调用C函数do_IRQ
    addl $4, %esp      ! 清理栈
    popl %edi
    popl %esi
    popl %ebp
    popl %eax
    popl %ebx
    popl %ecx
    popl %edx
    pop %fs
    pop %es
    pop %ds
    iret               ! 中断返回

这段汇编代码的模式非常清晰：

  1. SAVE_ALL（保存所有寄存器）：将段寄存器（ds, es, fs）和通用寄存器（eax, ebx, ecx, edx, ebp, esi, edi）压栈保存
  2. 重置段寄存器：将ds、es设置为内核数据段（0x10），将fs设置为用户数据段（0x17）
  3. 更新jiffies：对于时钟中断，递增全局变量_jiffies
  4. 调用do_IRQ：压入中断向量号作为参数，调用C函数do_IRQ
  5. RESTORE_ALL（恢复所有寄存器）：从栈中恢复所有保存的寄存器
  6. iret返回：CPU自动从栈中恢复CS、EIP和EFLAGS

这个模式——保存现场、处理中断、恢复现场——是所有操作系统中断处理的核心模式，直到今天也没有改变过。

异常处理

异常处理与硬件中断处理的结构类似，但异常处理函数通常需要更多信息来判断错误原因。Linux 0.11在kernel/traps.c中初始化了所有异常处理函数：

/* kernel/traps.c — 异常处理初始化 */

void trap_init(void)
{
    int i;
    /* 设置IDT表项：异常0-31 */
    set_trap_gate(0, &divide_error);
    set_trap_gate(1, &debug);
    set_trap_gate(2, &nmi);
    set_trap_gate(3, &int3);        /* int 3 — 断点 */
    set_trap_gate(4, &overflow);
    set_trap_gate(5, &bounds);
    set_trap_gate(6, &invalid_op);    /* 非法指令 */
    set_trap_gate(7, &device_not_available);
    set_trap_gate(8, &double_fault);  /* 双重错误 */
    set_trap_gate(9, &coprocessor_segment_overrun);
    set_trap_gate(10, &invalid_TSS);
    set_trap_gate(11, &segment_not_present);
    set_trap_gate(12, &stack_segment);
    set_trap_gate(13, &general_protection);  /* 通用保护异常 */
    set_trap_gate(14, &page_fault);           /* 缺页异常 */
    set_trap_gate(15, &reserved);
    set_trap_gate(16, &coprocessor_error);
    /* 17-31保留 */
    for (i = 17; i < 32; i++)
        set_trap_gate(i, &reserved);
}

每个异常对应的处理函数在kernel/asm.s中定义。以除零错误为例：

! kernel/asm.s

_divide_error:
    pushl $_do_divide_error  ! 压入处理函数地址
    jmp error_code           ! 跳转到通用异常处理代码

! 通用异常处理代码
error_code:
    pushl %ds        ! 保存所有寄存器
    pushl %eax
    xorl %eax, %eax
    pushl %ebp
    pushl %esi
    pushl %edi
    ...
    call *%eax        ! 调用具体的异常处理函数

当异常严重到无法恢复时，内核会调用die()函数——这是Linux内核最初的Panic机制。die()会打印异常信息、寄存器状态，然后进入死循环或重启：

/* 内核Panic */
void die(char *str, ...)
{
    printk("Kernel panic: %s\n", str);
    /* 打印寄存器现场 */
    __asm__("cli");   /* 关闭中断 */
    while(1);         /* 死循环 */
}
注意：内核Panic意味着操作系统已经无法继续安全运行。在早期内核中，die()只是简单地打印信息然后死循环——没有优雅的关机，没有文件系统同步。这是因为0.11的定位是"功能原型"，容错机制后来才逐渐完善。直到今天，"Kernel panic"仍然是Linux遇到不可恢复错误时的最终处理方式。

踩坑故事——EOI丢失导致的中断"静默"

这是早期PC系统中一个非常经典的Bug，即使是经验丰富的内核开发者也可能中招。

故事是这样的：某位开发者写了一个简单的设备驱动，在中断处理函数中做了些常规操作——读取设备寄存器、处理数据、返回。在测试时，他发现一个奇怪的现象：时钟中断偶尔会"消失"几秒钟，系统时间变慢，调度器行为异常。

他花了整整三天调试，最终发现问题出在EOI（End of Interrupt，中断结束）命令上。

8259A中断控制器在收到一个中断后，会屏蔽同优先级和更低优先级的中断，直到收到EOI命令才会重新打开。如果驱动在处理中断时没有向8259A发送EOI（通过向0x20端口写0x20），8259A就会认为中断还在处理中，从而阻塞后续所有中断。

/* 错误写法：缺少EOI */
void timer_interrupt_bad(void)
{
    do_timer();       /* 处理定时器 */
    /* 忘记发送EOI！8259A会一直屏蔽中断 */
}

/* 正确写法：必须发送EOI */
void timer_interrupt_good(void)
{
    do_timer();       /* 处理定时器 */
    outb_p(0x20, 0x20);  /* 向主片8259A端口0x20发送EOI命令(0x20) */
}

Linux 0.11的do_IRQ函数中，在调用具体处理函数之后，会检查中断是否来自从片（向量号 >= 0x28），如果是则先向从片发送EOI，然后再向主片发送EOI：

/* kernel/irq.c — do_IRQ */
void do_IRQ(unsigned int irq)
{
    /* 调用注册的中断处理函数 */
    if (call_irq(irq))
        return;

    /* 发送EOI */
    if (irq >= 8)        /* 从片的中断 */
        outb_p(0x20, 0xA0);  /* 向从片发EOI */
    outb_p(0x20, 0x20);      /* 向主片发EOI */
}

这个Bug给所有写中断驱动的开发者上了一课：中断处理和硬件之间有一个严格的协议，任何一步遗漏都可能导致整个系统行为异常。而且这类问题特别难调试——因为症状不是系统崩溃，而是"间歇性行为异常"——时断时续的响应、偶发的延迟、间歇性的数据传输失败。碰到这种问题，先检查EOI——这几乎成了内核调试的一条"铁律"。

理解了这个故事，你就会明白为什么Linux 0.11中的中断处理代码看起来如此简洁，却又处处透着稳健——每一行代码背后都是用血的教训换来的工程经验。
