📡 第24章 开源5G平台实战

大家好，我是你们的讲师。上一章我们用MATLAB做了完整的5G NR链路级仿真，在虚拟空口中计算了BER曲线。但仿真终究是仿真——当你把算法部署到真实硬件上时，驱动问题、时钟不同步、实时性约束、协议栈状态机交互，每一个都是新的"坑"。如果你只做过仿真没碰过真实系统，面试时被问到"USRP初始化失败怎么办"这样的问题，可能会直接卡住。

开源5G平台正好填补了"仿真"和"商用设备"之间的巨大鸿沟。它们用通用处理器和SDR硬件实现了完整或准完整的5G NR协议栈，成本只有商用基站的几十分之一，但功能足以让你跑通一次端到端呼叫、分析协议消息、甚至做性能优化研究。截至2026年，开源5G生态已经非常成熟，形成了以OpenAirInterface(OAI)、srsRAN、UERANSIM和Open5GS为核心的完整工具链。

24.1 开源5G生态全景

项目 类型 主导组织 协议栈覆盖 是否需要SDR
OpenAirInterface (OAI) 完整gNB + UE + 5GC EURECOM / OAI Alliance L1-L3完整NR SA 是（USRP等）
srsRAN Project gNB + UE SRS（原Soft-RAN） L1-L3完整NR SA 是（USRP等）
UERANSIM 纯软件UE模拟器 Aligung（个人主导） NAS + RRC 否（纯软件）
Open5GS 5G核心网(5GC) Open5GS社区 AMF/SMF/UPF/AUSF/UDM 否

这四个项目之间的关系是互补的：OAI和srsRAN提供gNB（基站）+ UE（终端）功能，Open5GS提供5G核心网，UERANSIM则作为纯软件UE替代真实SDR UE。你可以根据硬件条件和实验目标自由组合它们。

24.2 OpenAirInterface (OAI)

OAI是开源5G领域最具影响力的项目，由法国EURECOM研究所发起，至今已有超过15年的历史。OAI支持3GPP Release 15/16的5G NR SA模式，包含完整的L1/L2/L3协议栈。

OAI架构

OAI的gNB部署分为三大组件：

  • OAI gNB (nr-softmodem)：运行在通用x86服务器上，通过USRP等SDR硬件发射5G NR信号，支持FR1（sub-6 GHz）和FR2（毫米波）
  • OAI 5GC (oai-cn5g)：轻量级5G核心网，包含AMF/SMF/UPF/AUSF/UDM/NRF等网元
  • OAI NR UE (nr-uesoftmodem)：模拟5G终端，需要另一套SDR硬件

硬件需求

运行OAI对硬件有一定要求。USRP是最常用的SDR平台：

SDR型号 频率范围 带宽 连接方式 适用场景
USRP B210 70 MHz - 6 GHz 56 MHz USB 3.0 入门级gNB/UE，FR1低带宽
USRP X310 DC - 6 GHz 160 MHz 10GbE / PCIe 5G NR 100 MHz带宽
USRP N310 300 MHz - 6 GHz 100 MHz 1GbE / SFP+ 分布式部署，GPSDO内置
ADI Pluto 325 MHz - 3.8 GHz 20 MHz USB 2.0 教学演示，带宽受限

注意：OAI的编译和运行对Linux内核版本、UHD驱动版本、CPU支持（必须支持AVX2指令集）都有严格依赖。建议使用Ubuntu 22.04 LTS + OAI官方推荐的Linux内核版本（如5.15或6.2）。不要试图在VMware或VirtualBox中运行OAI——实时信号处理对中断延迟极其敏感，虚拟化环境会引入不可预测的延迟抖动，导致gNB的OFDM符号定时漂移，轻则吞吐量下降，重则UE无法同步。

24.3 srsRAN

srsRAN由SRS（前身为Soft-RAN）公司主导开发，是OAI之外最主流的开源5G实现。srsRAN的一个显著优势是代码结构更清晰、部署更轻量。在2024-2026年间，srsRAN经历了重要的品牌重组，现在分为两个主要项目：

  • srsRAN Project：面向5G SA的gNB和UE实现，支持完整的NR协议栈
  • srsRAN 4G：继续维护LTE实现（原来叫srsLTE）

相比OAI，srsRAN Project的代码更简洁、读起来更友好。很多学术论文的原型验证选择srsRAN而不是OAI，原因就是定制化的学习成本更低。

部署步骤：Docker Compose快速搭建

最快速的部署方式是使用srsRAN Project官方提供的Docker Compose配置:

# docker-compose.yml — srsRAN gNB + 核心网
version: '3.8'

services:
  # 5G核心网 (使用Open5GS)
  open5gs:
    image: free5gc/ueransim:latest
    network_mode: "host"

  # srsRAN gNB
  srsgnb:
    image: srsran/srsgnb:latest
    network_mode: "host"
    volumes:
      - ./config/gnb.yaml:/etc/srsran/gnb.yaml
    depends_on:
      - open5gs

  # UERANSIM (模拟UE)
  ueransim:
    image: free5gc/ueransim:latest
    network_mode: "host"
    volumes:
      - ./config/ue.yaml:/etc/ueransim/ue.yaml
    depends_on:
      - srsgnb

对应的 gnb.yaml 配置文件核心部分：

# gnb.yaml — srsRAN gNB配置
gnb_id: 1
gnb_id_bit_length: 22
cell_id: 0x01
nci: 0x00000001

# 频段配置
band: 78                # n78 (3.5 GHz)
dl_arfcn: 630000        # 对应3.5 GHz
channel_bandwidth: 40   # 40 MHz

# 核心网连接
amf:
  addr: 127.0.0.1
  port: 38412

# USRP配置
device_args: "type=usrp_b200"

经验谈：如果这是你第一次接触开源5G，我建议从 srsRAN + Open5GS + UERANSIM 这个组合开始，而不是直接上OAI。原因有三：① srsRAN的Docker映像开箱即用，不需要从源码编译；② UERANSIM是纯软件UE，跑在你的开发机上就能模拟终端，不需要第二台SDR；③ 排错时Docker日志清晰，docker-compose logs -f 就能看到所有组件输出。等你熟悉了5G协议栈的交互流程，再换OAI做更底层的物理层优化也不迟。

24.4 UERANSIM：纯软件UE仿真器

UERANSIM是一个特殊的存在——它不需要任何SDR硬件，直接在Linux上通过Socket与gNB的RRC/NAS层通信。它的典型用法是：连接一个真实或仿真的gNB（比如srsRAN gNB），模拟UE的NAS层注册流程、PDU会话建立流程、数据业务传输。

UERANSIM能做什么：

  • 完整的NAS层信令：注册请求、身份认证、安全模式控制
  • PDU会话建立：支持IPv4/IPv6、以太网、非结构化PDU
  • 移动性管理：小区选择/重选、切换
  • 数据面通道：支持ping/iperf测试

UERANSIM不能做什么：它不处理L1/L2物理层，所以无法测试射频性能、EVM、链路自适应等物理层相关指标。它本质上是"在IP层面模拟了一个5G终端的NAS+RRC行为"。

24.5 Open5GS：轻量级5GC

Open5GS是目前最流行的开源5G核心网实现之一，支持5G SA模式的完整网元集合：

  • AMF：接入和移动性管理功能，处理UE注册和连接管理
  • SMF：会话管理功能，处理PDU会话建立和IP分配
  • UPF：用户面功能，负责数据包的转发和QoS执行
  • AUSF/UDM：认证和用户数据管理
  • NRF：网络仓库功能，服务发现

Open5GS的部署非常轻量，一个4核8 GB RAM的云服务器就能跑起完整的5GC。它不涉及任何底层射频处理，完全工作在N1/N2/N3/N4/N6等3GPP接口的SCTP/HTTP2/GTP-U协议层面。

24.6 实战项目：端到端5G实验网

下面是一个推荐给初学者的一站式实验方案——用srsRAN + Open5GS + UERANSIM搭建纯软件端到端5G系统：

步骤 操作 验证手段
1. 安装Docker sudo apt install docker.io docker-compose docker --version
2. 启动5GC docker run -d --name open5gs --network host open5gs/open5gs docker logs open5gs 确认AMF/SMF/UPF启动
3. 启动gNB docker run -d --name srsgnb --network host -v $(pwd)/gnb.yaml:/etc/srsran/gnb.yaml srsran/srsgnb docker logs srsgnb 看到 "gNB setup OK"
4. 启动UE docker run -d --name ue --network host -v $(pwd)/ue.yaml:/etc/ueransim/ue.yaml free5gc/ueransim docker logs ue 看到 "Registration complete"
5. 测试连通性 在UE容器内执行 ping 192.168.0.1（UPF地址） ping通即表示端到端数据面正常
6. 性能测试 iperf3 -c 192.168.0.1 -t 30 记录吞吐量结果

关键洞察：当你在Docker上完成这个"端到端5G呼叫"后，回头看第8章和第9章讲的核心网架构和gNB协议栈，会有完全不同的感受——协议文本中的"AUSF处理认证向量"、"SMF分配UPF隧道信息"、"NGAP建立NG接口"，每一句抽象的描述都在你刚才的日志里变成了具体的一行输出。这就是做实战项目的最大价值：把标准文本变成体内记忆。

24.7 踩坑实录

所谓"无坑不实战"。这里分享一个我亲身经历的教训：

踩坑故事：有一次在我的实验室搭建OAI端到端系统，使用USRP B210做gNB射频前端。部署了整整两天，OAI gNB启动正常、5GC正常运行，但UE就是搜不到SSB信号。看OAI日志，gNB一直打印"Waiting for sync"——UE收不到PSS/SSS，无法完成时频同步。我用频谱仪测了B210的射频输出口，有信号发射，功率正常。问题到底在哪？

排查了整整一个下午，最后发现：USRP B210内部没有GPSDO（GPS驯服晶振），它的本地振荡器(LO)频率与UE的LO之间存在约5 ppm的偏差。在3.5 GHz频段上，5 ppm相当于17.5 kHz的频率偏移——远超过了NR UE的初始频率搜索范围（通常约2-3 kHz）。UE的PSS搜索算法在这种频偏下完全失效，自然无法同步。

解决方案：要么给B210外接GPSDO参考时钟（B210有外部10 MHz参考输入口），要么在配置中启用UE端的"频率自动校正"功能（OAI的UE支持 --usrp-args "clock_source=external" 参数）。我后来加了一台OctoClock-G时钟分配器，问题解决。

教训：SDR的时钟精度是开源5G系统中最容易被低估的问题。B210/B200的TCXO精度只有±2.5 ppm，X310/N310可以配OCXO（±0.01 ppm）。如果需要两台以上的设备协同工作，GPSDO或外参考几乎是强制的。

这个案例告诉我们：开源5G系统不比商用设备"好搞"，恰恰相反，它把商用设备中已经替你处理好的"脏活"（时钟同步、RF校准、中断调度）全部暴露了出来。搞开源5G的过程，就是加深对这些"脏活"理解的过程。

从软件仿真到SDR硬件，我们已经开始在真实空口中"跑"5G了。但SDR平台可以验证协议栈的正确性，却无法做标准化的射频指标测量——发射功率是否合规？ACLR是否超标？EVM是否在限值以内？这些需要专业的射频仪表来回答。下一章我们走进仪表测试与路测实战，看看精密仪器是怎么给通信系统"把脉"的。
