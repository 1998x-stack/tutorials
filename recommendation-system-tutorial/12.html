第12章 DIEN与序列模型

上一章的DIN是一个重要的里程碑——它用注意力机制让模型能够根据候选物品自适应地构建用户兴趣表示。但DIN有一个根本性的缺陷：它把用户的行为序列当成了一个集合。

什么意思呢？在DIN中，注意力权重是在"当前行为"和"候选物品"之间一次性计算出来的，行为之间的先后顺序被完全忽略了。用户"先看了手机壳、再看了手机、最后看了耳机"和用户"先看了耳机、再看了手机、最后看了手机壳"——这两条序列在DIN看来是完全一样的。

但现实世界中，用户的兴趣是动态演化的。今天的兴趣可能源自昨天的探索，明天的购买决策可能来自今天的比较。序列中的顺序信息，恰恰是理解用户兴趣演化的关键。

DIEN（Deep Interest Evolution Network）就是阿里巴巴在DIN之后推出的"进化版"——它把序列建模引入了推荐系统。

1. DIN的局限：只看相关性，不看演化

让我们用一个具体的例子来说明DIN的盲区。

小张是一个科技爱好者。他在过去一个月里的行为序列是：

  • 第1天：搜索"iPhone 15 评测"
  • 第7天：点击"iPhone 15 手机壳"
  • 第14天：查看"iPhone 15 Pro Max 价格"
  • 第21天：点击"MagSafe 充电器"
  • 第28天：购买"AirPods Pro 2"

现在系统要推荐"MagSafe 电池背夹"。DIN会把最后5个行为打散成一个集合，分别计算每个行为与"MagSafe 电池背夹"的attention分数。手机壳和充电器的得分应该较高，但这只是"相关性"的捕捉。

但如果我们仔细看这个序列，会发现一条清晰的兴趣演化路径：对iPhone产生兴趣 → 购买配件 → 扩展配件生态 → 购买无线耳机。这是一个"从一个兴趣点出发、逐步拓展到周边生态"的过程。这种演化模式对于预测"接下来用户可能需要什么"非常有价值——而DIN完全无法捕捉。

DIEN要解决的核心问题正是：如何同时建模行为的相关性 AND 行为的演化性。

2. DIEN：兴趣提取层+兴趣演化层

DIEN的架构分为两个核心模块：兴趣提取层（Interest Extractor Layer）和兴趣演化层（Interest Evolution Layer）。

兴趣提取层使用GRU（Gated Recurrent Unit）对用户的行为序列进行编码。GRU的每个时间步接收一个行为的embedding，输出一个隐藏状态 h_t，理论上代表了"截至时间 t 的用户兴趣状态"。

但这里有一个关键问题：GRU的训练目标是"拟合输入序列"，而不是"提取兴趣"。DIEN引入了一个非常聪明的辅助损失（Auxiliary Loss）来解决这个问题。

辅助损失的核心思想：在每一个时间步 t，用当前GRU的隐藏状态 h_t 去预测用户的下一个行为 b_{t+1}。这相当于让GRU的隐藏状态"被迫"学习到——什么是用户行为的演化趋势。这个辅助损失和主任务的CTR损失一起联合优化。

L_total = L_CTR + α * L_aux

α 是辅助损失的权重系数（论文中建议设为1）。辅助损失的引入极大地提升了隐藏状态的信息质量——如果没有它，GRU的隐藏状态可能会被CTR任务"扭曲"到只关注与CTR最相关的信号上，而损失了兴趣演化的全貌。

深度解读：辅助损失的引入是DIEN最精妙的设计。它相当于告诉模型——"GRU的每个隐藏状态不仅要帮助最终CTR预测，还要自己有能力预测下一个行为"。这个额外的监督信号让兴趣提取更加充分，而且不增加任何线上推理的计算成本（辅助网络只在训练时使用，inference时可以丢弃）。

兴趣演化层在GRU的基础上进一步引入了注意力机制。用户的兴趣演化不是匀速的——当候选物品与当前兴趣强相关时，隐藏状态的更新应该更"激进"；当不相关时，更新应该更"保守"，更多地保留之前的状态。DIEN使用AUGRU（Attention Update GRU）来实现这个机制。

3. AUGRU机制详解

AUGRU是DIEN的核心创新。我们先回顾一下标准GRU的更新门：

z_t = σ(W_z · [h_{t-1}, x_t])

z_t 是一个0到1之间的值，控制"多少过去的信息被保留、多少新信息被加入"。z_t 越接近1，新信息的影响越大。

AUGRU的修改就发生在这个更新门上：

z'_t = a_t * z_t

其中 a_t 是注意力得分——当前行为与候选物品的相关性。这个修改的含义非常优雅：

  • 如果当前行为与候选物品高度相关（a_t 接近1），更新门保持原样，GRU正常更新——兴趣状态向当前行为的方向演变
  • 如果当前行为与候选物品几乎无关（a_t 接近0），更新门被压低到接近0，GRU几乎不做更新——兴趣状态保持之前的演化轨迹

用一句话总结：AUGRU在GRU的基础上，用注意力来控制"兴趣演化应该在多大程度上受当前行为影响"。

下面是简化版的AUGRU实现：

import torch
import torch.nn as nn
import torch.nn.functional as F

class AUGRUCell(nn.Module):
    """Attention Update GRU Cell"""
    def __init__(self, input_dim, hidden_dim):
        super().__init__()
        self.hidden_dim = hidden_dim
        # GRU门控参数
        self.W_z = nn.Linear(input_dim + hidden_dim, hidden_dim)
        self.W_r = nn.Linear(input_dim + hidden_dim, hidden_dim)
        self.W_h = nn.Linear(input_dim + hidden_dim, hidden_dim)

    def forward(self, x_t, h_prev, att_score):
        """
        x_t: 当前输入 (batch, input_dim)
        h_prev: 上一个隐藏状态 (batch, hidden_dim)
        att_score: 注意力得分 (batch, 1) 范围 [0,1]
        """
        combined = torch.cat([x_t, h_prev], dim=-1)

        # 标准GRU计算
        z_t = torch.sigmoid(self.W_z(combined))   # 更新门
        r_t = torch.sigmoid(self.W_r(combined))   # 重置门
        h_tilde = torch.tanh(self.W_h(torch.cat([x_t, r_t * h_prev], dim=-1)))

        # AUGRU核心: 注意力加权更新门
        z_t_hat = att_score * z_t

        # 更新隐藏状态
        h_t = (1 - z_t_hat) * h_prev + z_t_hat * h_tilde
        return h_t

注意，这个AUGRU和标准GRU只有一行代码的差别—— z_t_hat = att_score * z_t。但这一行代码的改动带来了显著的线上效果提升。阿里巴巴在DIEN论文中报告：与DIN相比，DIEN在展示广告CTR上又提升了约5%。

实用性建议：虽然DIEN在理论上优雅且效果出色，但它的工程实现比DIN复杂不少——需要维护GRU的序列依赖，训练时需要额外的辅助损失计算，序列较长时训练速度会明显下降。对大多数团队来说，DIN是"性价比更高"的选择。我一般建议：如果行为序列较短（<50）且时序演化模式明显（如电商购物决策周期），可以尝试DIEN；如果序列很长或行为较为随机，DIN可能就足够了。

4. 序列推荐模型全景图

DIEN是"用RNN做序列推荐"的代表，但序列推荐的世界远不止于此。近年来学术界提出了多种基于不同技术路线的序列推荐模型——从Self-Attention到Transformer，从双阶段检索到多模态融合。

模型 年份 核心架构 最大序列长度 关键创新
GRU4Rec 2015 GRU + Session-based ~50 首次将RNN用于会话推荐
DIEN 2018 GRU + AUGRU + Aux Loss ~100 兴趣提取与演化分离
SASRec 2018 Self-Attention + Transformer ~200 自注意力捕获序列内相关性
BERT4Rec 2019 Cloze Mask + Transformer ~200 双向上下文、masked item预测
SIM 2020 两阶段检索 + Attention ~1000+ 长序列检索 + 精排双重架构
SR‑GNN 2019 Graph + GNN ~50 将会话序列建模为图结构

这里特别提一下阿里巴巴的SIM（Search-based Interest Model）。DIEN的GRU在序列长度超过100后，训练速度和内存消耗都会大幅增加。SIM的思路非常粗暴且有效——先用候选物品的类别/属性对用户的长序列进行检索过滤，只保留最相关的部分，然后再做attention。这个两阶段设计让DIEN/SIM可以处理长达数千甚至上万的行为序列。想象一下，一个淘宝用户在三年内可能有上万次点击行为，SIM通过检索将其压缩到几十个最相关的行为，再进行精细化建模。

从DIN到DIEN到SIM，这个系列的演进告诉我们一个道理：在推荐系统中，序列的"长度"和"演化信息"是两个不同维度的问题。DIN解决了"相关性"但忽略了"演化"；DIEN引入了"演化"但受限于"长度"；SIM则在特定方向上突破了"长度"的限制。在实际工业场景中，这三个模型各有用武之地——如果你需要按顺序阅读论文，我建议的优先级是：DIN → DIEN → SIM，核心原因后面章节会逐渐展开。

到此为止，我们用了整整五章来讨论推荐系统"排序阶段"的模型。但你有没有想过——所有这些精排模型，处理的是百级规模的候选集。但在这百级候选进入精排之前，是谁从千万甚至上亿的物品中把它们捞出来的？这是召回（Recall）阶段的工作。下一章开始，我们将进入召回篇——从双塔模型到向量检索，从多兴趣到图召回，看看推荐的"第一道漏斗"是如何设计的。
