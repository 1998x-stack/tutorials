第21章 5G QoS与策略控制

大家好，我是你们的讲师。到目前为止，我们已经学习了网络切片（建专用车道）和MEC（在路边建服务站）。但还有一个关键问题没解决：在这条"专用车道"上，谁先走？谁走快车道？谁可以插队？

这就是本章要讲的5G QoS（Quality of Service）与策略控制。在5G网络里，不同的业务对带宽、时延、可靠性的要求天差地别——微信语音丢了几个包最多听不清，但远程手术丢一个包可能就是人命关天。QoS机制就是给网络一个能力，让它能区分"哪个包更重要"，并为不同重要级别的包提供差异化的处理待遇。

21.1 从4G QoS到5G QoS的演进

4G时代也有QoS，叫作QCI（QoS Class Identifier），是一个标准化的一维标识符，取值范围1-9，每个值对应一组QoS参数（优先级、时延、丢包率等）。4G的QoS体系在5G中得到了全面升级：

维度 4G (QCI) 5G (5QI) 升级说明
标识范围 QCI 1-9（标准化） 5QI 1-127（标准化）+ 128-254（自定义） 5G支持运营商自定义QoS等级
GBR支持 GBR / Non-GBR 两类 GBR / Non-GBR / Delay-Critical GBR 三类 新增"延迟敏感GBR"，满足URLLC
反射QoS 不支持 支持（Reflective QoS） UE可自动学习下行QoS规则用于上行
QoS粒度 EPS承载级 QoS Flow级（更细粒度） 一个PDU会话可包含多个QoS Flow
策略控制 PCRF PCF + SMF + UPF 分布式协同 SBA架构下的更灵活策略控制

一句话总结：如果说4G QoS是"粗颗粒的沙砾"，5G QoS就是"细颗粒的面粉"——能更精细地区分业务类型、更灵活地配置策略参数。

21.2 QoS Flow与5QI

5G引入了一个重要概念——QoS Flow。QoS Flow是5G网络中QoS处理的最小粒度单元，一个PDU会话（相当于用户的网络连接）中可以包含多个QoS Flow，每个Flow承载不同QoS要求的业务流量。

每个QoS Flow由以下要素定义：

  • QFI（QoS Flow ID）：6位标识符，唯一标识PDU会话内的一个QoS Flow
  • 5QI（5G QoS Identifier）：指向一组预定义的QoS特性
  • QoS Profile：包含5QI、ARP、MBR/GBR、AMBR等QoS参数

3GPP TS 23.501定义了标准化的5QI值，下面是部分常用值：

5QI 资源类型 优先级 PDB（包时延预算） PER（包错误率） 典型业务
1 GBR 2 100ms 10⁻² IMS语音（VoNR）
2 GBR 4 150ms 10⁻³ IMS视频通话
5 Non-GBR 1 100ms 10⁻⁶ IMS信令
6 Non-GBR 6 300ms 10⁻⁶ 视频流（TCP）
7 Non-GBR 7 100ms 10⁻³ 语音实时交互、直播
8 Non-GBR 8 300ms 10⁻⁶ 普通网页浏览
9 Non-GBR 9 300ms 10⁻⁶ 默认尽力而为
82 Delay-Critical GBR 19 10ms 10⁻⁵ URLLC离散自动化
83 Delay-Critical GBR 22 10ms 10⁻⁵ URLLC智能交通
84 Delay-Critical GBR 24 30ms 10⁻⁵ URLLC一般工业

解释一下PDB（Packet Delay Budget）：PDB不是"端到端时延"，而是"单跳时延预算"——从数据包到达UPF到它被gNB成功发送到空口之间的时间（N3+N2+空口）。实际端到端时延还需要加上应用服务器到UPF的传输时延。不过工程上经常用PDB作为时延规划的参考依据。

21.3 GBR与Non-GBR QoS Flow

5G定义了三种资源类型，决定了QoS Flow的带宽保障方式：

  • GBR（Guaranteed Bit Rate）：保证比特率，网络承诺为这个Flow提供最低保障带宽。即使网络拥塞，GBR Flow也能获得其保证的带宽。典型应用：VoNR语音、实时视频通话。
  • Non-GBR（Non-Guaranteed Bit Rate）：不保证比特率，网络尽最大努力传输，没有带宽承诺。典型应用：网页浏览、文件下载、大部分OTT业务。
  • Delay-Critical GBR：延迟关键型GBR，5G新增类型。不仅保证带宽，还保证时延不超标。如果时延超过PDB，gNB可以选择丢弃该包（丢包优于晚到）。典型应用：URLLC工业控制、自动驾驶。

21.4 反射QoS（Reflective QoS）

5G引入了4G没有的一个重要能力——反射QoS。它的机制很有意思：UE通过观察下行数据包的QoS标记，自动推导出上行数据包应该使用的QoS规则，不需要网络侧显式地配置上行QoS规则。

下行路径（触发反射QoS学习）:
  应用服务器 → UPF(标记QFI=6) → gNB → UE
                                        |
                         UE自动学习: 上行也用QFI=6
                                        |
上行路径（UE自主应用）:
  UE → gNB → UPF(检查QFI=6) → 应用服务器

反射QoS的优点是减少了信令开销——网络不需要为每个UE配置上行QoS规则，UE自己"看样学样"就够了。但它的缺点是：UE必须收到一个下行包之后才能确定上行QoS规则，所以对第一个上行包不适用（一般用默认QoS规则兜底）。

适用场景：反射QoS特别适合非对称业务——比如视频直播，主播的上行流量大（推流）、下行流量小（看弹幕），如果下行发一个包UE就能学会上行QoS规则，比SMF单独下发一个上行规则高效得多。

21.5 PCF策略控制架构

5G的策略控制体系由PCF（Policy Control Function）主导，整体架构如下：

+------+      +------+      +------+
|  AF  |      |  PCF |      |  UDR |
| (应用 |----->| (策略 |<---->| (策略 |
|  功能)|      |  控制)|      |  数据)|
+------+      +------+      +------+
                |
    +-----------+-----------+
    |           |           |
+------+  +------+  +------+
| SMF  |  | AMF  |  | NSSF |
| (会话|<->| (移动|<->| (切片|
|  管理)|  |  管理)|  |  选择)|
+------+  +------+  +------+
    |
+------+
| UPF  | ← 最终执行QoS策略的网元
| (用户|
|  面)|
+------+

流程简述：

  • AF（Application Function）：应用侧告知PCF业务需要什么样的QoS（比如"这是VIP用户的视频流，需要优先保障"）
  • PCF：根据AF请求、UE签约数据和运营商策略，生成PCC（Policy and Charging Control）规则
  • SMF：从PCF获取PCC规则，将其转化为QoS Profile，通过N4接口下发到UPF执行
  • UPF：数据面最终执行——打QFI标记、做速率限制、GBR带宽保障、包过滤等

21.6 端到端QoS映射

从应用程序到空口，QoS需要经历多层映射：

应用层: App 识别业务类型 (语音/视频/数据)
   ↓
QoS Flow层: SMF/PCF 分配 QFI + 5QI + QoS Profile
   ↓
DRB层: gNB 将 QoS Flow 映射到 DRB（数据无线承载）
   ↓
空口: MAC 层调度器根据 5QI 优先级执行资源分配

gNB在其中的角色至关重要——它需要将一个或多个QoS Flow映射到合适的DRB上。同一条DRB上的所有数据包在空口侧获得相同的调度优先级。一般来说，相同或相似5QI值的QoS Flow会被映射到同一条DRB以节省空口资源。

工程视角：在5G基站侧配置QoS映射时，最常犯的错误就是"映射粒度过粗"——把所有QoS Flow一股脑塞进同一条DRB，导致URLLC和eMBB业务在同一条DRB里竞争资源。正确的做法是至少为URLLC和eMBB各创建一条独立的DRB，这样才能在MAC调度层实现差异化。

21.7 踩坑实录：5QI配置错误导致视频卡顿

讲一个亲身经历的真实问题。2023年底，某视频直播平台迁移到5G专网。直播流的QoS配置为5QI=6（Non-GBR，视频流TCP传输），推流端用的是有线+5G双链路聚合。上线后用户反馈：在移动场景下（主播在移动直播车上），视频流频繁卡顿，甚至出现30秒以上的冻屏。

我们首先排除了无线侧的问题——参考信号接收功率（RSRP）在-85dBm以上，信噪比（SINR）在20dB以上，空口条件非常好。然后查UPF侧——带宽也没有打满。最后查了SMF下发的QoS Profile，发现问题出在5QI的PDB配置上。

原来，5QI=6的PDB是300ms，但这个值对实时直播推流来说太宽松了。直播推流用的是TCP协议，当网络出现短暂抖动时，TCP会启动重传。但因为PDB=300ms，gNB的MAC调度器认为300ms以内到达都算"合格"，不会优先调度重传包。结果就是：轻微的丢包触发TCP重传，而重传包在gNB调度队列里等了200多毫秒才被发出去——导致视频流累计了大量延迟，最终Player端因为缓冲区耗尽而卡顿。

最终的解决方案是将直播推流的5QI从6改为5QI=7（PDB=100ms，优先级更高）。5QI=7的PDB更严格，gNB调度器会优先处理这个Flow的包，包括重传包。配置修改后，直播流的卡顿完全消失。

教训：5QI的选择不仅影响"带宽和优先级"，还影响gNB调度器的丢包和重传策略。对实时性敏感的TCP业务，建议选择PDB<100ms的5QI值。如果使用运营商的5QI自定义范围（128-254），一定要先做充分的路测验证再上线。我们事后复盘时发现，如果配置前做一次"不同5QI下的FTP下载对比测试"，这个问题在测试阶段就能发现。

QoS策略控制是5G网络从"尽力而为"走向"可承诺体验"的核心能力。它和网络切片、MEC一起构成了5G高阶能力的"铁三角"。有了这三件套，运营商才能真正对行业客户说："我们不仅提供带宽，还提供可测量的SLA保障。"下一章，我们将转向一个所有工程师都必须重视的话题——5G安全架构。好网络不仅要快，还要安全可靠。
