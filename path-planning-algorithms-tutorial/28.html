我先说一个真实的故事。我在2019年带过一个实习生，他花了整整三周时间从零搭建了一套导航栈：A* 全局规划 + DWA 局部规划 + 三次样条平滑。他的代码在仿真环境里跑得很漂亮。但在最后的评测中，我们把机器人放到一个布满碎纸屑的地板上——DWA 的评价函数瞬间被摩擦系数变化搞乱了，机器人在原地打转了整整两分钟。他当时非常沮丧，但这个经历教会了他一件事：仿真和真实世界之间永远有一道你意想不到的鸿沟。也正因为这道鸿沟，你的能力才能真正得到锻炼。

项目目标

这个综合项目的目标是：搭建一个完整的自主导航栈，使其能够在包含静态和动态障碍物的环境中，从起点安全、高效地导航到终点。

具体来说，你的系统需要包含以下三个核心模块：

  • 全局规划器（Global Planner）——在已知的栅格地图上，计算一条从起点到终点的无碰撞粗略路径。推荐实现：A*（追求最优性）或 RRT*（追求高维适应性和渐进最优性）。
  • 局部规划器（Local Planner）——沿全局路径前进，同时利用局部传感器信息避开未在全局地图中标记的动态障碍物。推荐实现：DWA（动态窗口法）。
  • 路径平滑模块（Path Smoothing）——对生成的路径做后处理，消除不必要的急转弯和抖动。推荐实现：三次样条插值或梯度下降平滑。

全局规划器实现

你的全局规划器应当接收一个代价地图（costmap）和起点终点坐标，输出一系列离散路径点。

如果你选择 A* 作为全局规划器，你已经在前面的章节中掌握了它的实现。关键设计决策是：

  • 启发式函数——对于栅格地图，Euclidean distance 通常是最安全的选择。如果你的地图有特殊结构（如大量直线走廊），可以考虑对角线距离。
  • 邻域扩展——4 邻域足够吗？8 邻域？还是应该支持任意角度的扩展（Any-angle A* / Theta*）？我建议从 8 邻域开始。
  • 代价定义——最简单的做法是基于距离。但如果地图包含不同地形（如碎石路 vs 柏油路），可以用加权代价。

如果你选择 RRT*，你需要在连续空间中采样，并且支持运动学约束（如车辆的最小转弯半径）。对于教学目的，我建议先做 A*，再用 RRT* 作为进阶挑战。

局部规划器实现

DWA（Dynamic Window Approach）是局部规划器的首选，原因有二：它直观、容易实现，而且效果好。

DWA 的核心逻辑是：在速度空间 (v, ω) 中采样，用运动学模型前向仿真一小段轨迹，然后用评价函数选择最优轨迹。评价函数通常包含三个分量：

  • 方位角代价（heading cost）——轨迹终点与目标方向的偏差角。鼓励朝向目标。
  • 间隙代价（clearance cost）——轨迹与最近障碍物的距离。鼓励远离障碍物。
  • 速度代价（velocity cost）——鼓励高速而不是低速爬行。

你需要在 config.yaml 中暴露关键参数，以便后续调优：速度采样范围、加速度限制、前向仿真时间、评价函数权重。

系统集成与测试

这是整个项目最关键的部分——如何把全局路径的输出变成局部规划器的输入。

全局规划器输出的是一条稀疏路径（比如在 100x100 的栅格地图上每 10 个像素一个路径点）。局部规划器需要一个局部目标点——不是全局终点，而是全局路径上的"前方某点"。这个"前瞻距离"（lookahead distance）是一个非常关键的参数。太短：机器人会频繁重新选向，路径不平滑；太长：局部规划器可能"看到"远处的转角，做出不合理的加速决策。

下面是一个集成的伪代码骨架——这是你整个导航栈的核心循环：

import numpy as np
from planners.global_planner import AStarPlanner
from planners.local_planner import DWAPlanner
from smoothing.path_smoother import spline_smooth

class NavigationStack:
    def __init__(self, costmap, config):
        self.costmap = costmap
        self.global_planner = AStarPlanner(costmap)
        self.local_planner = DWAPlanner(config)
        self.lookahead_dist = config["lookahead_distance"]
        self.goal_tolerance = config["goal_tolerance"]

    def set_goal(self, start, goal):
        # Step 1: 全局规划
        raw_path = self.global_planner.plan(start, goal)
        if raw_path is None:
            raise RuntimeError("Global planner failed: no path found")
        # Step 2: 路径平滑
        self.global_path = spline_smooth(raw_path)
        self.path_index = 0

    def compute_command(self, robot_pose, laser_scan):
        # Step 1: 在全局路径上找到前瞻点
        lookahead_pt = self._get_lookahead_point(robot_pose)
        if lookahead_pt is None:
            return (0.0, 0.0)  # 到达终点

        # Step 2: 局部规划——以前瞻点为目标
        cmd_vel = self.local_planner.plan(
            pose=robot_pose,
            goal=lookahead_pt,
            scan=laser_scan,
            costmap=self.costmap
        )
        return cmd_vel  # (linear_vel, angular_vel)

    def _get_lookahead_point(self, robot_pose):
        """在全局路径上找到距离机器人最近的前瞻点"""
        min_dist = float("inf")
        nearest_idx = 0
        for i, pt in enumerate(self.global_path):
            d = np.linalg.norm(robot_pose[:2] - pt)
            if d < min_dist:
                min_dist = d
                nearest_idx = i
        # 向前偏移 lookahead_dist 距离
        accumulated = 0.0
        for j in range(nearest_idx, len(self.global_path) - 1):
            seg_len = np.linalg.norm(
                self.global_path[j+1] - self.global_path[j])
            accumulated += seg_len
            if accumulated >= self.lookahead_dist:
                return self.global_path[j+1]
        return self.global_path[-1]

    def is_goal_reached(self, robot_pose):
        return (np.linalg.norm(robot_pose[:2] -
                self.global_path[-1]) < self.goal_tolerance)

你需要至少准备 3 个测试场景：

  • 简单房间——没有障碍物，验证基本功能是否正常。
  • 迷宫——验证全局规划器能否正确绕障。
  • 动态障碍物——在路径中间放置一个突然出现的障碍物，验证局部规划器的避障能力。
调试建议：从一开始就实现可视化。把全局路径、DWA 采样轨迹、前瞻点、以及评价函数的热力图全部画在 Matplotlib 上。你会发现 80% 的 bug 在可视化后会一目了然——比如"为什么机器人撞上了障碍物？"可能只是因为你的碰撞检测用了世界坐标而不是机器人坐标系。

算法选型决策

完成项目后，请认真写下你的选型理由。这是一个很好的面试题——比你想象中更常被问到。

下面是一些你应该能回答的问题：

  • 为什么选择 A* 而不是 Dijkstra？——因为你的地图是已知的、静态的，A* 的启发式搜索可以显著减少扩展节点数。但如果地图非常大（比如 2000x2000），你可能会考虑用 Jump Point Search 来进一步加速。
  • 为什么选择 DWA 而不是 TEB 或 MPC？——因为 DWA 的计算开销低、实现简单、且对于差速驱动底盘效果足够好。TEB 和 MPC 在高速车辆场景中更受青睐，但它们的参数调优成本也更高。
  • 前瞻距离多大？为什么？——这个值取决于机器人的速度和转弯能力。一个经验法则是：前瞻距离至少是机器人长度的 2-3 倍。你可以在 config.yaml 中暴露这个参数，然后通过实验来确定最优值。
  • 你的系统在大规模场景中能实时运行吗？——如果不能，瓶颈在哪里？是 A* 的搜索耗时？还是 DWA 的采样数量太多？你做了哪些优化？（比如 A* 的二叉堆、DWA 的并行轨迹评估）

这些问题的质量直接决定了你的项目从"能跑"到"优秀"之间的距离。

扩展方向

在你完成了基础版本之后——如果你还有精力——这里有一些有意义的扩展方向：

  • 动态障碍物——在 DWA 的基础上加入运动障碍物的速度预测（恒定速度模型或更复杂的 Social LSTM 预测器）。
  • 多机器人——扩展你的导航栈，支持多台机器人在地图中同时运行。你需要实现第21章讨论的优先级调度或死锁检测。
  • 学习型组件——在局部规划器的评价函数中引入一个学习型权重调整模块。比如用 PPO 训练一个策略，根据环境密度动态调整 heading/clearance/velocity 的权重。
  • ROS2 集成——把你的导航栈封装为 ROS2 节点，接入 Nav2 的 lifecycle 管理。这会让你的项目看起来像一个"真正的"产品级代码。
  • 语义导航——加入视觉 foundation model，让机器人能够理解"去那个蓝色椅子旁边"这样的高级指令。
