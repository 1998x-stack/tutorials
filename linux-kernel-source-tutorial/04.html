大家好，我是你们的讲师。上一章我们搭建了开发环境——Bochs已经能在你的电脑上运行Linux 0.11了。从本章开始，我们将真正进入代码世界，逐行阅读那些构成操作系统的核心源码。

这是整个课程中我最喜欢的部分——引导启动。为什么？因为引导启动是内核的"创世纪"：从你按下电源键到屏幕上出现命令行提示符，CPU经历了从实模式到保护模式、从汇编到C语言的两次"质变"。读懂引导代码，你就理解了x86计算机最底层的运行机制。

这一章我们分析引导流程的前半部分——bootsect.S和BIOS之间的交互。下一章再完成后半部分——进入保护模式。我们把"从实模式到保护模式"这个巨大的主题分成两章来消化。

引言——从按下电源键到内核第一条指令

x86计算机的启动过程，是一部从"硬"到"软"的接力赛。按下电源键之后，发生的事情如下：

1. 按下电源键
2. CPU加电 -> 内部逻辑将CS:IP强制设为 0xF000:0xFFF0
   (物理地址 = 0xF000 * 16 + 0xFFF0 = 0xFFFFFFF0)
3. 这个地址映射到BIOS ROM，CPU从这里执行第一条指令
   (这是一条跳转指令，跳到BIOS真正的POST代码入口)
4. BIOS执行POST(Power-On Self Test)
   - 检测CPU、内存、芯片组
   - 初始化中断向量表(IVT)到0x00000开始的内存区域
   - 初始化各种硬件控制器
5. BIOS扫描启动设备顺序(在CMOS中配置)
   - 检查软盘0是否有引导扇区
   - 检查硬盘MBR
   - 检查CD-ROM等
6. 找到启动设备后，读取第一个扇区(512字节)到 0x7C00
7. 跳转到 0x7C00 执行——控制权交给引导扇区

当BIOS的最后一个动作——jmp 0x0000:0x7C00——执行完毕时，控制权就正式交给了软盘上读入的引导扇区代码。这个引导扇区，在Linux 0.11中就是boot/bootsect.S编译后的代码。

为什么是0x7C00？ 这个地址是IBM PC设计时的一个历史选择。当时的考虑是：BIOS需要尽可能地给引导程序留出足够的连续内存空间。0x7C00这个位置在实模式1MB地址空间的低端区域，留给引导程序约30KB的自由空间。这个地址后来成了x86平台的事实标准，一直沿用至今。

CPU加电与BIOS

让我们深入第一步：当CPU加电复位时，它到底在做什么？

x86 CPU有专门的上电复位逻辑。当RESET引脚收到高电平信号后，CPU内部的所有寄存器会被设置成预定义的初始值。其中最关键的是CS和IP寄存器：

上电复位后x86 CPU的寄存器状态:
  CS    = 0xF000     (代码段寄存器)
  IP    = 0xFFF0     (指令指针寄存器)
  DS    = 0x0000
  ES    = 0x0000
  SS    = 0x0000
  FLAGS = 0x0002     (IF=0, 中断关闭)

  物理地址 = CS * 16 + IP = 0xF0000 + 0xFFF0 = 0xFFFFFFF0

  这个地址位于4GB地址空间的最顶端，被映射到BIOS ROM

0xFFFFFFF0这个地址通常被称为"复位向量"（Reset Vector）。BIOS固化ROM的时候，会在这个地址放一条jmp指令，跳转到BIOS自检代码的真正入口。注意，CPU刚上电时仍然处于实模式，但实模式按理说只能访问1MB地址空间（0x00000 ~ 0xFFFFF）。那CPU怎么访问0xFFFFFFF0呢？这是因为CPU在复位后有一个特殊状态——指令预取队列和执行逻辑的某些部分被设计为允许从高地址取第一条指令，此后就回到正常的实模式地址限制。

BIOS完成POST之后，会填写中断向量表（IVT）。IVT位于物理地址0x00000开始的内存区域，共有256个中断向量，每个向量4字节（段地址:偏移量），总共占用1KB（0x00000 ~ 0x003FF）。BIOS int 0x13（磁盘读写服务）就是通过IVT提供的。

接着，BIOS读取引导扇区——将启动设备的第0个扇区（LBA 0）读入内存0x7C00处。如果这是一个有效的引导扇区（最后两个字节是0x55AA），BIOS就跳转到0x7C00去执行。

bootsect.S源码分析

现在，让我们打开boot/bootsect.S来看。这是内核的第一个文件，也是内核执行的第一段代码。它完全由x86实模式汇编写成。

我们先看它的整体逻辑，再逐段分析：

! bootsect.S 的完整调用流程:
!
! 0x7C00 ── 入口(BIOS跳转至此)
!   │
!   1. 搬移自身: 将 0x7C00 ~ 0x7DFF 复制到 0x90000 ~ 0x901FF
!      (因为后面要把内核加载到0x00000，不想和自己冲突)
!   │
!   2. 设置段寄存器: DS=ES=SS=0x9000, SP=0xFF00
!      (SP指向0x9000:0xFF00 = 0x9FF00，在搬移后的代码上方)
!   │
!   3. 加载setup模块: 通过INT 0x13读取磁盘第2~5个扇区到0x90200
!      (setup.s编译后大约2KB，需要4个扇区)
!   │
!   4. 加载system模块: 通过INT 0x13读取后续扇区到0x10000 ~ 0x8FFFF
!      (system模块包括head.s + 所有内核C代码, 约130KB)
!   │
!   5. 在屏幕上打印 "Loading system ..."
!   │
!   6. 跳转到 0x90200 ── setup.S 的入口

! 搬移指令的关键代码:
start:
    mov ax,#BOOTSEG    ! BOOTSEG = 0x07C0 (对应物理地址0x7C00)
    mov ds,ax
    mov ax,#INITSEG    ! INITSEG = 0x9000 (对应物理地址0x90000)
    mov es,ax
    mov cx,#256        ! 搬移256个字 = 512字节
    sub si,si          ! DS:SI = 0x07C0:0x0000 = 0x7C00
    sub di,di          ! ES:DI = 0x9000:0x0000 = 0x90000
    rep movw           ! 重复执行movw，每次搬移一个字
    ! 此时代码已经运行在0x90000处

    jmpi go,INITSEG    ! 远跳转到 0x9000:go
go: mov ax,cs          ! 重新设置段寄存器指向新位置
    mov ds,ax
    mov es,ax
    mov ss,ax
    mov sp,#0xFF00     ! 设置栈顶

注意rep movw这个指令——它是x86汇编中非常高效的块复制指令。rep前缀让后面的movw重复执行cx次，每次自动递减cx并调整si/di指针。256个字 × 2字节 = 512字节，正好是一个扇区的完整大小。

为什么需要搬移？ 引导扇区加载到0x7C00后，接下来要加载内核system模块到0x00000开始的位置。0x7C00和0x00000之间的距离只有约31KB。system模块大约130KB，加载过程中必然会覆盖0x7C00处的引导代码。因此bootsect必须先把自己搬到更高的地址（0x90000）去运行，避免被后续加载的数据覆盖。

加载setup模块

bootsect搬移自身的下一步，是用BIOS int 0x13加载setup模块。setup模块大约2KB，由4个扇区组成：

! 加载setup模块 —— 从磁盘第2个扇区开始读4个扇区到0x90200
load_setup:
    mov dx,#0x0000      ! DH=磁头0, DL=驱动器号0(软盘)
    mov cx,#0x0002      ! CH=柱面0, CL=起始扇区2
    mov bx,#0x0200      ! ES:BX = 0x9000:0x0200 = 0x90200
    mov ax,#0x0204      ! AH=0x02(读扇区), AL=0x04(读4个扇区)
    int 0x13            ! 调用BIOS磁盘中断
    jnc ok_load_setup   ! 如果CF=0(成功), 跳转
    mov dx,#0x0000      ! 出错, 复位磁盘驱动器
    mov ax,#0x0000      ! AH=0x00(复位)
    int 0x13
    jmp load_setup      ! 重试
ok_load_setup:
    ! 此时setup已在0x90200

这段代码值得仔细看，因为它展示了BIOS int 0x13调用的标准用法：

  • AH = 0x02（读扇区功能号）
  • AL = 要读取的扇区数（这里是4）
  • CH = 柱面号低8位（这里是0）
  • CL = 起始扇区号（位0-5） + 柱面号高2位（位6-7）（这里是2，表示从第2个扇区开始——第1个扇区是bootsect自身）
  • DH = 磁头号（这里是0）
  • DL = 驱动器号（0表示第一个软盘驱动器）
  • ES:BX = 数据存放的内存地址

返回时，如果CF=0表示成功，AL中返回实际读取的扇区数。如果CF=1，则AH中包含错误码。

CHS寻址的局限：BIOS int 0x13使用CHS（柱面-磁头-扇区）寻址方式，最大只能访问约8GB的磁盘。后来的扩展int 0x13（AH=0x42）使用LBA（逻辑块地址）方式，突破了这一限制。但在1991年，8GB已经是大到不可想象的磁盘容量了——Linus根本不需要考虑这个问题。

加载完setup之后，bootsect接着用类似的方式加载system模块。system模块的大小在编译时就已经确定，以扇区数为单位记录在bootsect.S的一个全局变量中。加载完system后，bootsect在屏幕上打印"Loading system ..."（通过int 0x10调用BIOS字符显示服务），然后执行jmpi 0, SETUPSEG跳转到0x90200处的setup代码。

! 最终跳转到setup模块
    jmpi    0,SETUPSEG    ! SETUPSEG = 0x9020
                          ! 跳转到 0x9020:0x0000 = 0x90200

实模式编程要点

如果你之前没有接触过x86实模式汇编，这里有几个核心概念需要先掌握。理解它们，你才能跟上bootsect.S的每一行代码。

1. 段寄存器与物理地址的计算

在实模式下，物理地址 = 段寄存器值 × 16 + 偏移量。因为x86的寄存器都是16位的，最大可表示地址为0xFFFF × 16 + 0xFFFF = 0x10FFEF，大约是1MB + 64KB - 16字节。但实际上，由于早期8086只有20根地址线，超过0xFFFFF（1MB）的地址会"回绕"到低地址——这就是后面A20门问题的根源。

  段地址:偏移   计算过程           物理地址
  ============  ================  ==========
  0x07C0:0x0000  0x07C0*16+0x0000 = 0x7C00
  0x9000:0x0000  0x9000*16+0x0000 = 0x90000
  0x9000:0x0200  0x9000*16+0x0200 = 0x90200
  0x9000:0xFF00  0x9000*16+0xFF00 = 0x9FF00

2. 寄存器约定

汇编代码和BIOS调用之间通过寄存器传递参数。在bootsect.S中你主要会遇到这些寄存器：

寄存器用途
AX (AH/AL)BIOS功能号(AH)和参数(AL)，比如AH=0x02表示读扇区
BX通常作为内存地址的低16位（与ES组合）
CX (CH/CL)CHS寻址中的柱面号和扇区号
DX (DH/DL)磁头号(DH)和驱动器号(DL)
SI/DI字符串操作中的源地址和目标地址索引
CS:IP代码段:指令指针，决定CPU从哪里取指令
SS:SP栈段:栈指针，push/pop操作的内存位置
FLAGS (标志寄存器)CF进位标志(BIOS调用返回时CF=0成功/CF=1失败)，IF中断标志

3. 实模式1MB地址空间

实模式的1MB空间有着固定的用途划分。了解这个布局对于理解引导过程非常重要：

  0x000000 ~ 0x0003FF: 中断向量表(IVT), 256个中断向量
  0x000400 ~ 0x0004FF: BIOS数据区
  0x000500 ~ 0x007BFF: 自由内存区(~30KB)
  0x007C00 ~ 0x007DFF: 引导扇区加载位置(512字节)
  0x007E00 ~ 0x009FFF: 自由内存(~8.5KB)
  0x00A000 ~ 0x00BFFF: 显存(VGA显卡)
  0x00C000 ~ 0x00FFFF: 扩展ROM/BIOS

  ---- 加载后的Linux 0.11布局 ----
  0x000000 ~ 0x08FFFF: system模块(内核代码和数据)
  0x090000 ~ 0x0901FF: bootsect(临时驻留)
  0x090200 ~ 0x0903FF: setup(临时驻留)
  0x100000 ~ 0x17FFFF: 内核最终位置(通过分页映射)

这个布局很关键——后续的bootsect.S、setup.S和head.S都围绕着这个地址规划来工作。

踩坑故事——BIOS int 0x13磁盘读取的不可靠性

早期的BIOS实现在int 0x13上有不少Bug。其中一个让人头疼的问题是：某些BIOS不支持跨磁头读取。

你可能注意到了，bootsect.S加载setup时只读取了4个扇区，而且这4个扇区都在同一个磁头（磁头0）、同一个柱面（柱面0）上。这是有意的——Linus知道某些古董级BIOS在处理"跨磁头"的读取时会出错。

更具体地说，当读取请求跨越磁头边界（即从磁头0切换到磁头1）时，一些BIOS会错误地返回数据，或者返回正确数据但搞错扇区编号。这个问题在IBM PC/AT兼容机上非常普遍——因为当时很多厂商对BIOS int 0x13的实现并不严格遵循标准。

Linux 0.11的处理方式很简单：能在一个CHS范围内读完的就尽量读完。system模块的加载更加谨慎——它通过循环逐段加载，而不是一次性要求读取大量扇区。每次调用int 0x13后都检查CF标志位，失败则重试。

经验教训：在底层硬件接口编程中，"信任但要验证"是黄金法则。BIOS虽然提供了标准接口，但不同厂商的实现在细节上差异很大。Linux内核早期代码中对各种BIOS的兼容性处理，就是一个不断发现Bug、填补特例的过程。Linus为此在代码中添加了大量重试逻辑和边界检查——这些代码虽然看起来啰嗦，但都是真实世界的血泪教训。

还有一个有趣的故事：早期Linux的引导软盘在不同品牌PC上的成功率差异很大。Linus自己有一台386/DX33，他在上面测试引导完全没问题。但当他发布v0.02后，有用户报告在某些IBM兼容机上引导失败。最终排查发现是BIOS的int 0x13在读取高柱面时出错——Linus的电脑恰好没有触发这个Bug。他后来在代码中加了一个"先测试后读取"的机制，这个问题才得到解决。

好了，bootsect.S的工作到这里就结束了。它把setup模块和system模块都加载到了内存中，自己完成了历史使命。接下来，CPU跳转到0x90200，开始执行setup.S的代码。
