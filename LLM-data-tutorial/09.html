粒度决定效果

上一章我们学会了用 MinHash 找出"长得像"的文档。但有一个问题你可能还没意识到：去重应该发生在哪个层次上？

举个具体的例子。你有一篇技术博客介绍"如何用 PyTorch 实现 Transformer"，另一篇文章是某课程笔记，里面完整引用了这篇博客的"注意力机制"部分，但加了自己的实验分析和代码。这时候如果做文档级去重，这两篇可能因为相似度过高而被判定为重复——于是课程笔记被删掉了，但删掉的那部分恰恰是笔记作者自己写的实验分析，这些内容是博客里没有的。

这就是单粒度去重的固有问题：要么漏掉重复，要么误删独有内容。

多粒度去重策略的核心思想是：在"整篇文档"、"段落"和"行"三个层次上分别做去重，每个层次解决不同的问题。Dolma 语料库的构建过程就采用了这种多粒度策略。他们发现单一粒度的去重总会遗漏一些重复模式，而组合使用三种粒度可以在保持数据多样性的同时最大化去除冗余。

文档级去重

文档级去重是最简单、也是应用最广的粒度。无论是精确去重还是 MinHash 模糊去重，大多数流水线的默认操作单元就是"一篇文档"。它的逻辑很直观：如果两篇文档足够相似，就只保留一篇。

The Pile（EleutherAI, 2020）在构建 22 个来源的多源语料时，就是用文档级 MinHash 做全局去重的。他们的做法是对所有来源的文档统一计算 MinHash 签名，然后通过 LSH 找到近似重复的文档对。一篇来自 PubMed 的论文摘要如果在某个博客中被完整复制，就会被识别出来并只保留其中一个版本。

但文档级去重有一个隐藏的痛点：内容类似的文档未必是"重复"。比如一个代码教程的不同章节可能都包含了一段相同的 import 语句和模型定义代码，但核心教学内容完全不同。文档级去重会错误地把它们标记为重复。

实践要点： 文档级去重适合作为"第一道防线"。它的好处是简单高效，能快速去除绝大部分明显的重复内容。对于 Web 语料来说，文档级去重通常能减少 20-40% 的冗余。但如果你的语料中有很多"引用型"内容（如论文引用、代码复用），就需要更细的粒度来补充。

段落级去重

段落级去重提供了更精细的控制。它不是把整篇文档当成一个整体，而是把文档拆分成段落（通常用空行或标题分隔），然后对每个段落单独做去重。

CCNet 最早在工业级流水线中引入了段落级去重。他们的做法很简单：对每个段落计算 SHA-1 哈希，然后在整个语料中查找完全相同的段落。如果一个段落出现在多篇文档中，只保留它在第一篇文档中的出现，从其他文档中移除该段落。

这种做法的好处是：可以保留每篇文档中的"独有部分"，同时剔除那些被到处复制粘贴的"通用段落"。回到开头的例子——课程笔记中引用的"注意力机制"段落如果和原博客完全一致，这个段落会被去重；但笔记作者自己写的实验分析部分是独特的，会被保留下来。

粒度去重单元典型方法适用场景误删风险

行级去重

行级去重是最激进的粒度。它把每个文本行作为去重的基本单元，只要某一行的内容和语料中其他行的内容一模一样，就将该行标记为重复并移除。

这种粒度的主要应用场景是代码语料和表格数据。比如在 GitHub 上，无数项目都包含了相同的开源协议头部声明（如 Apache 2.0 License block），或者大量 Python 文件都有完全相同的 import 语句。文档级去重抓不到这种"局部重复"，段落级去重也未必覆盖，但行级去重可以精确识别并移除。

不过，行级去重有一个大坑。

血的教训： 对代码语料做行级去重要非常非常小心！我在处理一个代码数据集时，发现模型生成的代码总是缺少 import 语句——debug 了好几天才发现是行级去重把所有 import 行当成"重复"删掉了。两行一模一样的 "import torch" 是合法的——每个 Python 文件都需要它，它不该被当成"重复内容"。

解决方案是什么？要么对代码语料不做行级去重，要么在做行级去重时设置一个"白名单"（如 import、def、class 开头的行排除在外），要么只对内容行（非语法关键字行）做去重。

跨语料去重

当你从多个来源收集数据时，跨语料去重就变得至关重要。想想看：一篇 arXiv 论文可能同时出现在 arXiv 原始源、某人的博客摘要、某课程的参考书目、以及维基百科的引用中。如果不用跨语料去重，这篇论文的关键内容可能被重复计算 4 次，不仅浪费 token 预算，还会在训练中产生不自然的偏向。

The Pile 采用了一种非常系统化的跨语料去重策略：把所有 22 个来源的文档合并成一个巨大的语料库，然后用 MinHash + LSH 做统一的跨源去重。不管是来自哪个源的文档，只要内容相似度超过阈值，就只保留一个版本。

{
  "dedup_config": {
    "strategy": "cross_corpus_minhash",
    "threshold": 0.87,
    "shingle_size": 13,
    "num_hashes": 128,
    "lsh_bands": 20,
    "lsh_rows_per_band": 6,

    "source_priority": [
      "arxiv",        # arXiv 版本优先（通常是最新）
      "books3",       # 书籍版本次之
      "pubmed",       # 论文摘要
      "stackexchange",
      "github",
      "wikipedia",
      "web_crawl"     # Web 抓取优先级最低
    ],

    "preserve_if_unique": true
  }
}

上面的配置展示了跨语料去重的一个重要设计维度——来源优先级。当检测到两篇近似重复的文档（比如一篇来自 arXiv，另一篇来自 Web 抓取），我们不是简单地随机保留一篇，而是根据来源的权威性和质量决定保留哪个。一般规则是：权威版本（arXiv、PubMed）优先于引述版本（博客、论坛），原始版本优先于转载版本。

来源优先级策略： 跨语料去重不只是比较文本相似度。你还需要考虑——哪个来源的版本质量更高？arXiv 论文通常是最完整、最准确的版本；博客转载可能加了错误的理解；Wikipedia 的摘要虽然简洁但经过了编辑审核。在去重时保留最权威的版本，是提升整体语料质量的一个"隐含杠杆"。

总结一下多粒度去重：文档级负责"大局"，段落级负责"局部"，行级负责"细节"，跨语料负责"全局一致性"。这四种粒度组合使用，形成了一张覆盖各种重复模式的网。在实际工程中，通常的执行顺序是：先做文档级去重（效率最高，能去除大部分重复），再做段落级去重（精细处理引用型内容），最后按需决定是否做行级去重。跨语料去重则贯穿整个过程。

从下一章开始，我们进入一个新的环节——质量过滤。去重解决了"重复"的问题，但"低质量"是另一回事。我们将从困惑度这个经典指标讲起，看看如何用语言模型自己来评判文本质量。
