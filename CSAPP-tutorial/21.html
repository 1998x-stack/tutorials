第二十一章：网络编程——进阶

上一章我们手写了一个 echo 服务器，它能接收客户端的连接请求，读取数据并回写响应。但这个服务器有一个致命的弱点：它一次只能服务一个客户端。为什么？因为在 while 循环中，accept() 等待新连接是阻塞的，read() 读取客户端数据也是阻塞的。如果你正在读客户端 A 的数据，客户端 B 的连接请求就会排队等待——等 A 处理完了 B 才能得到响应。在真实的互联网上，用户成千上万，这样的"迭代服务器"毫无可用性。

如何让一个服务器同时服务成千上万的客户端？这就是本章要回答的核心问题。我们将从最基本的 fork-per-client 模型讲起，逐步引入更高效的 I/O 多路复用和事件驱动模型，还会介绍 CGI 等经典的 Web 后端技术——以及一个在面试和生产环境中反复出现的话题：并发服务器设计。

21.1 从迭代到并发：三种服务器模型

并发服务器的核心思想就是"不等待"。当服务器在处理一个客户端时，它不能阻塞其他客户端的连接。实现这一目标有多种策略：

模型 工作原理 优点 缺点 适用场景
迭代服务器 单进程，依次处理每个客户端 实现简单，无竞态条件 一个慢客户端会阻塞所有人 开发调试、内部工具
fork-per-client 每个连接 fork 一个子进程处理 进程隔离，一个请求崩溃不影响其他 进程开销大，fork 成本高（毫秒级） 小型服务、CGI 程序
预派生进程池 启动时 fork 固定数量的子进程，共享 listen_fd 避免 fork 开销，控制并发度 进程数固定，负载高时可能不够 Apache 的 prefork 模式

fork-per-client 是最容易理解的并发模型。下面是一个基本实现：

int listen_fd = Open_listenfd(8080);

while (1) {
    struct sockaddr_in client_addr;
    socklen_t addr_len = sizeof(client_addr);
    int conn_fd = accept(listen_fd,
                         (SA *)&client_addr, &addr_len);

    if (fork() == 0) {
        // 子进程：处理客户端请求后退出
        close(listen_fd);  // 子进程不需要监听 fd
        handle_client(conn_fd);
        close(conn_fd);
        exit(0);
    }

    // 父进程：关闭 conn_fd，继续 accept
    close(conn_fd);
}

注意两个微妙但关键的地方。第一，父进程在 fork 后立即 close(conn_fd)——子进程持有 conn_fd 的副本，父进程不关闭就会泄漏文件描述符。第二，子进程 close(listen_fd)——监听套接字是父进程的职责，子进程不需要。fork 会复制进程的文件描述符表，所以父子进程共享所有打开的文件描述符。每个文件描述符都有一个引用计数，close 只是减少引用计数，真正关闭文件是在引用计数降为 0 的时候。

fork-per-client 的优点是简单、隔离性好。缺点是每个连接都要 fork——fork 本身开销很大（几十到几百微秒），而且子进程退出时会变成僵尸进程，需要父进程调用 wait()/waitpid() 来回收。

真实踩坑故事：某次线上服务升级后，发现服务器响应越来越慢，最终完全无响应。排查后发现是 fork-per-client 模型没有正确处理 SIGCHLD 信号——子进程退出后变成僵尸，父进程没有 wait 回收。随着僵尸进程越积越多，系统进程表满了，fork() 返回 EAGAIN。修复方案就是注册 SIGCHLD 信号处理函数，在 handler 中循环调用 waitpid(-1, NULL, WNOHANG) 回收所有已退出的子进程。

21.2 CGI——动态内容的经典方案

早期的 Web 服务器只能返回静态文件——你把一个 index.html 放在服务器上，客户端请求什么文件就返回什么。但很快人们就不满足于此了：我们需要动态内容——根据用户的输入生成不同的页面。CGI（通用网关接口，Common Gateway Interface）应运而生。

CGI 的接口非常简单：服务器收到一个动态请求时，fork 一个子进程，通过环境变量将请求信息传递给 CGI 程序，并通过 execve() 执行 CGI 程序。CGI 程序的 stdout 被重定向到客户端的 socket，因此它只要 printf() 输出 HTML 内容即可。

具体来说，服务器设置了以下环境变量：

  • QUERY_STRING：URL 中 ? 后面的部分（如 name=alice&age=25）
  • REQUEST_METHOD：GET 或 POST
  • CONTENT_LENGTH：POST 请求的主体长度
  • GATEWAY_INTERFACE：固定为 CGI/1.1

下面是一个 CGI handler 的伪代码：

void handle_cgi_request(int conn_fd, const char *uri) {
    char *query_pos = strchr(uri, '?');
    char *cgi_path = "/usr/lib/cgi-bin/hello";

    if (fork() == 0) {
        // 子进程：设置环境变量并执行 CGI 程序

        // 设置 QUERY_STRING
        char query_buf[1024];
        if (query_pos != NULL) {
            snprintf(query_buf, sizeof(query_buf),
                     "QUERY_STRING=%s", query_pos + 1);
            putenv(query_buf);
        }

        // 将 stdout 重定向到客户端连接
        dup2(conn_fd, STDOUT_FILENO);

        // 执行 CGI 程序（替换子进程映像）
        execl(cgi_path, cgi_path, NULL);

        // 如果 execl 返回，说明出错了
        printf("Content-Type: text/html\r\n\r\n");
        printf("<h1>CGI Error</h1>");
    }
    wait(NULL);  // 等待 CGI 进程结束
}

CGI 程序本身可以是用 C、Perl、Python 等任何语言编写的可执行文件。一个最简单的 CGI 程序（C 语言）：

#include <stdio.h>
#include <stdlib.h>

int main() {
    char *query = getenv("QUERY_STRING");
    printf("Content-Type: text/html\r\n\r\n");
    printf("<html><body>");
    printf("<h1>Hello from CGI!</h1>");
    printf("<p>Query: %s</p>", query ? query : "(none)");
    printf("</body></html>");
    return 0;
}

CGI 的最大问题就是效率——每个请求都要 fork 一个进程，开销极大。这就是后来 FastCGI、mod_perl、WSGI 等更高效方案出现的原因。但理解 CGI 仍然是理解 Web 后端技术发展脉络的关键一步。

21.3 非阻塞 I/O 与 I/O 多路复用

fork-per-client 解决了并发问题，但引入了新的问题：进程的开销。每个进程都有自己的地址空间、文件描述符表、信号处理等，开销很大。能不能在单进程内同时处理多个客户端？答案是使用 I/O 多路复用。

先来说说非阻塞 I/O。默认情况下，read() 从没有数据到达的 socket 上读取时会阻塞，直到有数据到来。如果我们将 socket 设置为非阻塞模式，read() 在没有数据时会立即返回 -1，errno 被设置为 EAGAIN 或 EWOULDBLOCK：

#include <fcntl.h>

int flags = fcntl(conn_fd, F_GETFL, 0);
fcntl(conn_fd, F_SETFL, flags | O_NONBLOCK);

但非阻塞 I/O 还不够——你怎么知道什么时候该读哪个 socket 呢？总不能在 while 循环里"轮询"所有 socket 吧？那样 CPU 会被白白浪费。这就是 select() 和 poll() 系统调用的由来。它们让你能够同时监视多个文件描述符，只有当某个 fd 就绪（可读/可写/异常）时才会返回。

来看一个基于 select() 的 echo 服务器核心逻辑：

#include <sys/select.h>

int main() {
    int listen_fd = open_listenfd(8080);

    fd_set read_set, ready_set;
    FD_ZERO(&read_set);
    FD_SET(listen_fd, &read_set);  // 监视 listen_fd

    int max_fd = listen_fd;

    while (1) {
        ready_set = read_set;            // select 会修改这个集合
        select(max_fd + 1, &ready_set,   // 监视可读
               NULL, NULL, NULL);        // 不监视可写和异常

        // 检查哪些 fd 就绪了
        for (int fd = 0; fd <= max_fd; fd++) {
            if (FD_ISSET(fd, &ready_set)) {
                if (fd == listen_fd) {
                    // 有新连接
                    int conn_fd = accept(listen_fd, NULL, NULL);
                    FD_SET(conn_fd, &read_set);  // 加入监视
                    if (conn_fd > max_fd)
                        max_fd = conn_fd;
                } else {
                    // 现有客户端发来了数据
                    char buf[1024];
                    int n = read(fd, buf, sizeof(buf));
                    if (n == 0) {
                        close(fd);               // 客户端关闭
                        FD_CLR(fd, &read_set);
                    } else {
                        write(fd, buf, n);        // echo 回显
                    }
                }
            }
        }
    }
}

核心洞察：select 模型让一个单进程线程就能管理数百个并发连接。它的工作模式是"事件驱动"的——你告诉内核你关心哪些 fd 上的哪些 I/O 事件，内核在事件发生时通知你。这比"为每个连接开一个线程/进程"要高效得多。但 select 也有局限：监视的 fd 数量上限是 FD_SETSIZE（通常是 1024），而且每次调用都需要 O(n) 的遍历。现代高性能服务器（如 Nginx、Redis）使用的是 epoll（Linux）或 kqueue（macOS），它们的事件通知是 O(1) 的。

21.4 实用网络调试工具集

网络编程中，你迟早会遇到各种奇怪的问题：连接被拒绝、数据发不出去、端口被占用。掌握下面这几个工具，能让你在网络排错时事半功倍。

netstat：查看网络连接状态的经典工具。

# 查看所有 TCP 连接及其状态
$ netstat -tan
Proto Recv-Q Send-Q Local Address    Foreign Address  State
tcp   0      0      0.0.0.0:8080     0.0.0.0:*        LISTEN
tcp   0      0      127.0.0.1:8080   127.0.0.1:54321  ESTABLISHED
tcp   0      0      127.0.0.1:54321  127.0.0.1:8080   TIME_WAIT

tcpdump / Wireshark：抓包工具，可以让你看到线缆上传输的原始数据包。对于分析协议细节、排查 TCP 问题极其有用。

# 抓取 localhost:8080 上的所有包，显示详细内容
$ sudo tcpdump -i lo -X port 8080

strace：追踪系统调用的终极武器。当一个网络程序行为异常（比如客户端连接不上）时，strace 可以告诉你程序到底做了哪些系统调用，哪个调用返回了错误。

# 追踪程序的系统调用，只看 socket 相关的
$ strace -e trace=network ./my_server

真实踩坑故事：有一次我在开发一个 HTTP 代理服务器时，重启后 bind 总是失败，返回"Address already in use"。用 netstat -tan | grep 8080 一看，发现大量处于 TIME_WAIT 状态的连接。TIME_WAIT 是 TCP 协议关闭连接时的正常状态，持续 2MSL（约 1-4 分钟）。在这期间，端口不能被重新 bind。根本原因是代码中没有设置 SO_REUSEADDR 套接字选项。加上 setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, ...) 之后，问题解决。后来我养成了习惯：每个 TCP 服务器代码中的前几行一定是设置 SO_REUSEADDR。

进阶的网络编程不仅是写代码，更是用工具和协议栈对话。当你看到 TIME_WAIT 时，你要知道这是 TCP 在保证最后一个 ACK 不丢失；当你看到 SYN_SENT 长时间不变成 ESTABLISHED，你要想到是不是防火墙拦截了 SYN 包。这些知识只能通过动手实践和"踩坑"来积累。

好了，网络编程的进阶之路我们走完了。从 fork-per-client 到 select 的事件驱动模型，从 CGI 到调测工具——你现在已经具备了编写一个真实并发 Web 服务器的理论基础。如果要做 Proxy Lab，这些知识就是你最重要的武器。下一章，我们将进入一个全新的主题——并发编程。线程、互斥锁、条件变量——这些概念是现代高性能服务器的基础，也是系统编程中最容易出错、最有挑战性的领域。
