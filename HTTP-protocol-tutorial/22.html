第二十二章：HTTP性能优化

网页加载速度直接影响了用户体验、转化率和SEO排名。Google的研究表明：加载时间从1秒增加到3秒，跳出率增加32%；再增加到5秒，跳出率增加90%。

HTTP性能优化的目标很明确：让浏览器尽快拿到所有需要的资源。这涉及到两个维度——减少请求数量和减少传输体积。但HTTP/2和HTTP/3的出现，让这两个维度的优先级发生了根本性变化。

22.1 请求数量优化

在HTTP/1.1时代，浏览器对每个域名有并发连接数限制（通常是6个）。超过限制的请求必须排队等待。这意味着：一个网页加载50个资源，很多时间花在了连接管理和排队上。

所以HTTP/1.1时代的优化策略核心是减少请求数：

CSS Sprites（雪碧图）：把多个小图标合并到一张大图上，通过 background-position 定位显示不同部分。把一个页面上几十个图标请求合并为1个图片请求。

/* 雪碧图示例 */
.icon-home { background: url(sprite.png) 0 0; width: 24px; height: 24px; }
.icon-user { background: url(sprite.png) -24px 0; width: 24px; height: 24px; }
.icon-search { background: url(sprite.png) -48px 0; width: 24px; height: 24px; }

图标字体（Icon Fonts）：Font Awesome、Glyphicons等把图标做成字体文件。一个字体文件包含几十上百个图标，只需一个HTTP请求。缺点是：加载了用不到的图标、可访问性差、在Retina屏幕上可能模糊。

SVG图标：现代的方案是用内联SVG或SVG Sprite。SVG是矢量图，清晰度高、可缩放、可用CSS控制颜色。推荐用SVG Sprite机制——把多个SVG合并到一个 svg-sprite.svg 文件中，通过 <use> 引用。

JS/CSS打包（Bundling）：把多个JavaScript文件合并成一个，多个CSS文件合并成一个。Webpack、Rollup、Vite等构建工具都有这个功能。

HTTP/2改变了什么：HTTP/2的多路复用允许在一个连接上同时传输多个资源，不再受并发连接数限制。所以HTTP/2环境下，反而不建议过度打包（Bundling）——因为打包会导致缓存失效的粒度变粗。比如你改了1个小组件的代码，整个大bundle的缓存都失效了。正确做法是拆成多个小文件，各自独立缓存。

22.2 资源体积优化

无论什么协议版本，减少传输体积永远是有效的优化手段。

代码压缩（Minification）：移除代码中的注释、空格、换行、缩短变量名。以JavaScript为例，原始代码可能100KB，压缩后降到40KB。

// 压缩前
function calculateTotal(price, quantity, discount) {
    // 计算折扣后的总价
    const subtotal = price * quantity;
    const finalPrice = subtotal * (1 - discount);
    return finalPrice.toFixed(2);
}

// 压缩后（手动模拟，实际压缩工具更彻底）
function calculateTotal(a,b,c){return (a*b*(1-c)).toFixed(2)}

文本压缩：在HTTP层面使用 Content-Encoding: gzip 或 br（Brotli）压缩响应体。Brotli的压缩率比gzip高20-30%，但CPU开销也略高。对于静态资源，预压缩（提前生成 .gz/.br 文件）可以避免每次重复压缩。

图片优化：图片通常是页面体积的"大头"。

格式特点适用场景
JPEG有损压缩，不支持透明照片、渐变色图片
PNG无损压缩，支持透明图标、截图、需要透明的图
WebP有损/无损，支持透明，比JPEG小25-35%网页图片首选（除Safari旧版）
AVIF比WebP压缩率更高，但编码较慢新兴格式，适配还在路上
SVG矢量格式，无限缩放图标、Logo、插画

Tree Shaking：在打包时移除未被引用的代码。比如你从 Lodash 只引用了 _.debounce，打包工具可以只包含 debounce 的代码，而不是整个 Lodash 库（300KB+）。

实战建议：对图片做响应式加载——通过 <picture> 标签或 srcset 属性，让移动设备只加载小尺寸图片，桌面设备加载高清图。同时配合 loading="lazy" 实现图片懒加载。另外，图片CDN（如Imgix、Cloudinary）可以实时调整图片尺寸和质量，省去手动准备多版本的工作。

22.3 加载策略

请求数量和体积都优化了，还有一个关键问题：什么时候加载什么资源。

Script加载：async vs defer

普通的 <script> 标签会阻塞HTML解析。也就是说，浏览器遇到 <script> 就得停下来下载并执行脚本，然后才能继续解析后面的HTML。

<!-- 默认（阻塞解析） -->
<script src="app.js"></script>

<!-- async：下载不阻塞HTML解析，下载完立即执行（执行可能中断解析） -->
<script src="app.js" async></script>

<!-- defer：下载不阻塞HTML解析，在HTML解析完成后、DOMContentLoaded事件前执行 -->
<script src="app.js" defer></script>

async 适合独立脚本（如分析工具、广告脚本），不依赖其他脚本。defer 适合需要在DOM就绪后执行的脚本（如操作DOM的代码），并且多个defer脚本按顺序执行。

关键CSS内联（Critical CSS）

首屏渲染所需的CSS如果放在外部CSS文件中，需要先下载CSS文件才能渲染页面。解决方案：把首屏CSS直接内联到HTML的 <head> 中，让浏览器立即开始渲染。

<head>
    <style>
        /* 首屏关键CSS直接内联 */
        header { background: #fff; padding: 20px; }
        .hero { font-size: 2rem; margin: 0 auto; }
    </style>
    <link rel="stylesheet" href="full.css" media="print" onload="this.media='all'">
</head>

上面这个技巧叫"加载非关键CSS"——先用 media="print" 让浏览器低优先级下载完整CSS，下载完成后切换回 media="all" 应用样式。

懒加载（Lazy Loading）

<!-- 原生懒加载：浏览器原生支持 -->
<img src="photo.jpg" loading="lazy" alt="...">
<iframe src="widget.html" loading="lazy"></iframe>

loading="lazy" 是现代浏览器原生支持的懒加载属性。图片和iframe只在即将进入视口时才加载。它的阈值是浏览器自动管理的（通常是视口外1250px），不需要JavaScript库。

踩坑故事：有次我们给一个电商详情页加懒加载，商品图片全部设置了 loading="lazy"。上线后发现首屏最大内容绘制（LCP）指标变差了。排查发现：首屏的几张商品主图也在视口内，但浏览器把它们标记为"可能不需要立即加载"，导致它们延迟出现了。教训是：首屏可见的图片不要加懒加载，只给"折叠线以下"的图片加。而且 loading="lazy" 不适用于 <picture> 中作为首屏内容的图片。

22.4 资源提示

资源提示（Resource Hints）是浏览器提供的一组声明式指令，告诉浏览器未来可能需要的资源，让浏览器提前做一些准备工作。

提示作用示例
dns-prefetch提前解析DNS<link rel="dns-prefetch" href="//api.example.com">
preconnect提前建立TCP+TLS连接（含DNS解析）<link rel="preconnect" href="https://api.example.com">
prefetch提前下载下一个页面需要的资源（低优先级）<link rel="prefetch" href="next-page.js">
preload提前下载当前页面关键资源（高优先级）<link rel="preload" href="font.woff2" as="font">

preconnect vs dns-prefetch：preconnect 做了 DNS 解析 + TCP 握手 + TLS 握手，比 dns-prefetch 更激进。对于关键的第三方源（比如CDN、支付API），用 preconnect；对于非关键但可能需要的，用 dns-prefetch。

preload vs prefetch：preload 告诉浏览器"当前页面立即需要这个资源"，浏览器会以高优先级加载它。prefetch 告诉浏览器"下一个页面可能需要这个资源"，浏览器在空闲时低优先级加载。

一个实用的preload例子——字体加载优化：

<!-- 预加载关键字体，避免FOUT（Flash of Unstyled Text） -->
<link rel="preload" href="/fonts/inter-var.woff2" as="font" type="font/woff2" crossorigin>

<!-- 关键API的预连接 -->
<link rel="preconnect" href="https://api.example.com">

<!-- 下一页可能需要的资源 -->
<link rel="prefetch" href="/product-list.html">
<link rel="prefetch" href="/product-list.js">

资源提示选择指南： 1. 你的页面依赖的第三方源（CDN、字体服务、API）→ 用 preconnect 2. 当前页面的关键资源（首屏字体、hero图片、关键CSS）→ 用 preload 3. 用户下一步很可能访问的页面资源 → 用 prefetch 4. 别把所有资源都加上 preload——这会降低真正的关键资源的优先级。preload要用在刀刃上。

22.5 HTTP缓存策略

第九章已经详细讲了HTTP缓存机制。这里我们聚焦在性能优化场景下的缓存策略。

Hash文件名——不可变缓存

这是最强大的缓存策略：给静态资源文件名加上内容哈希值（如 app.a3b4c5.js），然后设置极长的 Cache-Control，告诉浏览器"永远不会过期"：

# 对带hash的文件名，设置1年的强缓存
location ~* \.[a-f0-9]{8}\.(js|css|png|jpg|webp)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
}

这个技巧的妙处在于：内容变了 → 哈希变了 → 文件名变了 → 浏览器把它当作新资源重新下载。旧版本永久缓存，永不失效（除非用户手动清除缓存）。

Cache-Control 细粒度控制

资源类型推荐策略说明
带hash的JS/CSS/图片public, max-age=31536000, immutable一年强缓存，永不重新验证
HTML页面no-cache每次都回源验证（ETag或Last-Modified）
API响应（动态数据）no-cache 或 max-age=0, must-revalidate确保数据新鲜
用户头像public, max-age=86400缓存1天，允许CDN缓存
配置/清单文件public, max-age=3600缓存1小时

服务端缓存

除了客户端缓存，服务端的缓存策略同样重要：

  • CDN缓存：用 s-maxage 给CDN设置独立的缓存时间，可以比客户端缓存时间长。
  • 反向代理缓存：Nginx的 proxy_cache，对后端API响应做缓存。
  • 应用层缓存：Redis/Memcached缓存数据库查询结果。
# Nginx反向代理缓存配置示例
proxy_cache_path /data/nginx/cache levels=1:2 keys_zone=mycache:10m max_size=1g;

location /api/ {
    proxy_cache mycache;
    proxy_cache_key "$scheme$request_method$host$request_uri";
    proxy_cache_valid 200 5m;
    proxy_cache_use_stale error timeout updating;
    add_header X-Cache-Status $upstream_cache_status;
}

缓存链整体设计：一个请求可能经过 客户端 → CDN → 反向代理 → 应用服务器 → 数据库。每一层都可以做缓存。好的缓存策略是"层层衰减"的——离用户越近的层，缓存时间越长；越靠近源的层，缓存时间越短。这样既保证了性能，又保证了数据新鲜度。

22.6 HTTP/2与HTTP/1.1优化策略的差异

这是很多开发者容易忽略的问题。HTTP/2的多路复用、头部压缩、服务器推送等特性，让HTTP/1.1时代的一些"最佳实践"变成了反模式。

优化策略HTTP/1.1HTTP/2
文件合并（Bundling）推荐 —— 减少请求数不推荐 —— 增大缓存失效粒度
雪碧图（CSS Sprites）推荐 —— 减少图片请求不推荐 —— 单个小图独立缓存更灵活
域名分片（Domain Sharding）推荐 —— 突破6连接限制反模式 —— 破坏了多路复用
资源内联（Inline）对首屏有用需要谨慎 —— 影响缓存和优先级
资源预加载很重要依然重要
压缩重要依然重要

域名分片（Domain Sharding）特别值得一提。在HTTP/1.1时代，浏览器对每个域名只能建立6个并发连接。为了突破这个限制，开发者把资源分布到多个子域名上（如 css.example.com, img.example.com），每个子域名独立建立6个连接，总并发能力翻倍。

但在HTTP/2下，域名分片完全是一个反模式。因为HTTP/2在一个连接上复用所有流，多个连接反而增加了不必要的TCP开销和TLS握手开销。实际测试表明，对HTTP/2的页面做域名分片，性能反而下降10-30%。

HTTP/2下的推荐策略：

  • 把所有资源放在同一个域名下，最大化多路复用的收益
  • 文件拆分成粒度较小的单元，利用独立缓存
  • 用服务器推送（Server Push）来主动推送关键资源——不过这个功能浏览器支持度有限，很多场景下不如preload
  • 继续做好压缩、缓存、资源提示等工作

踩坑故事：有一个项目从HTTP/1.1迁移到HTTP/2后，性能反而变差了。排查后发现两个问题： 1. 他们把JS打包成了一个巨大的 bundle.js（2.5MB），HTTP/2的多路复用虽然让这个文件能传输了，但解析和执行时间还是那么长 2. 域名分片没有去掉，导致浏览器建立了3个独立的HTTP/2连接，每个连接的多路复用能力都被浪费了 把大bundle拆成按路由分块的多个小chunk、取消域名分片后，LCP从4.2秒降到了1.8秒。这个案例说明：从HTTP/1.1到HTTP/2的迁移不是简单地换个协议版本，而是要重新思考优化策略。
