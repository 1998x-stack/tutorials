# 生成与引用溯源

## 1. 可控生成

RAG 的生成阶段需要让 LLM"忠实于上下文"而非自由发挥。

### 1.1 生成参数控制

| 参数 | RAG 推荐值 | 原因 |
|------|-----------|------|
| temperature | 0.0-0.3 | 降低随机性，减少编造 |
| top_p | 0.9-0.95 | 保留一定多样性 |
| frequency_penalty | 0.0 | 不需要惩罚重复 |
| presence_penalty | 0.0 | 避免偏离上下文 |
| max_tokens | 根据场景 | 一般 500-2000 |

```python
response = client.chat.completions.create(
    model="gpt-4o",
    temperature=0.1,  # 低温度减少幻觉
    messages=[
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": f"参考资料：{context}\n\n问题：{query}"}
    ]
)
```

### 1.2 基于上下文约束的生成

在 System Prompt 中强制约束：
```
你是一个事实型问答助手。请严格遵守以下规则：

1. 只能使用"参考资料"中明确出现的信息来回答问题
2. 如果资料中没有相关信息，回答："根据现有资料，无法回答此问题"
3. 不要使用你的内部知识来补充或修正参考资料
4. 不要推测、假设或联想资料中不存在的信息
5. 直接引用原文时，使用双引号标注
```

## 2. 引用标记与来源追溯

### 2.1 内联引用格式

生成答案时标注每个声明对应的来源：
```
用户: iPhone 15 的摄像头参数？

RAG 答案:
iPhone 15 主摄像头为 4800 万像素 [1]，支持 2 倍光学变焦 [2]。
超广角摄像头为 1200 万像素 [1]，支持微距摄影 [3]。

来源:
[1] Apple官网-iPhone15规格页
[2] iPhone15相机技术文档-第3章
[3] WWDC2024-相机新特性介绍
```

### 2.2 引用机制的实现

```python
def generate_with_citations(query, context_docs, llm):
    # 构建带编号的上下文
    numbered_docs = []
    for i, doc in enumerate(context_docs, 1):
        numbered_docs.append(f"[{i}] 来源: {doc.metadata['source']}\n{doc.content}")

    context = "\n\n---\n\n".join(numbered_docs)

    prompt = f"""基于以下参考资料回答问题。在答案中使用 [数字] 标注信息来源。

## 参考资料
{context}

## 要求
- 每个关键声明后标注来源编号，如"xxx [1]"
- 如果信息来自多个来源，标注 [1][3]
- 如果资料无法回答，明确说明
- 在回答末尾列出所有引用来源

## 用户问题
{query}

## 回答（含引用）:"""

    response = llm.generate(prompt)
    return response
```

## 3. 置信度标注

对于不同确定性的信息，使用不同置信度：

```
高置信度 ✅: iPhone 15 主摄像头为 4800 万像素 [1]
中置信度 ⚠️: 下一代 iPhone 可能采用 4 摄方案（行业分析师预测，非官方信息）[2]
低置信度 ❌: 建议等待官方发布确认
```

某些场景可以让 LLM 输出结构化格式（JSON）：

```json
{
  "answer": "iPhone 15 主摄像头为 4800 万像素，支持 2 倍光学变焦。",
  "citations": [
    {"source": "Apple官网", "confidence": "high", "text": "4800万像素主摄"},
    {"source": "技术文档", "confidence": "high", "text": "2倍光学变焦"}
  ],
  "uncertain_parts": [
    {"text": "未来型号可能新增潜望式长焦", "reason": "基于行业分析师预测，非官方信息"}
  ]
}
```

## 4. 生成效果评估初探

### 4.1 基础指标
- **忠实度（Faithfulness）**：答案是否忠于检索到的上下文
- **答案相关性（Answer Relevance）**：答案是否直接回答了问题
- **流畅度（Fluency）**：语言表达是否自然通顺

### 4.2 简单自检 Prompt
```
请评估以下 AI 生成的回答质量：

回答：{generated_answer}
参考资料：{context}
问题：{query}

请从以下维度评分（1-5分）：
- 忠实度：回答中是否有参考资料不支持的声明？
- 完整性：回答是否覆盖了问题的所有方面？
- 准确性：回答中的事实是否与参考资料一致？

对每个低于5分的维度，说明扣分原因。
```

## 5. 生成失败的常见模式

| 失败模式 | 示例 | 根因 | 解决方案 |
|---------|------|------|---------|
| 编造事实 | "该产品于2023年发布"（资料中无此信息） | temperature 过高 | 降低温度，加强约束 |
| 忽略上下文 | 不使用检索结果，按自身知识回答 | Prompt 约束不足 | 在 Prompt 开头强调"仅使用参考资料" |
| 信息遗漏 | 资料包含3个要点但只用了1个 | 上下文窗口有限 | 增加检索数量或上下文压缩 |
| 过度引用 | "根据提供的信息，[完整复制原文]" | 缺乏合成指令 | 要求"总结和归纳"而非"引用" |
| 矛盾回答 | 回答包含两个相互矛盾的声明 | 检索到的文档本身矛盾 | 要求 LLM 标注不同来源的分歧 |
