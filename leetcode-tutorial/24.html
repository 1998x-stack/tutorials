并发编程为什么重要？

大家好，欢迎来到多线程与并发编程的章节。如果你觉得"并发题在 LeetCode 上占比不到 5%，不值得花时间"，那我必须纠正你——确实，LeetCode 题库中并发题只有大约 15 道，但你知道吗？几乎所有一线大厂的面试中都有一轮系统设计或并发编程的考察。尤其是 Google、Apple、Uber 这些公司，onsite 中经常出现多线程编码题。就算你面的不是并发岗位，下面这个问题几乎必问："这段代码在多线程环境下有什么问题？"如果你的回答是"加个锁"，那面试官会追问："用什么锁？锁的粒度是什么？可能产生死锁吗？"

Python 的多线程常被误解——很多人觉得 GIL 的存在让多线程变得毫无意义。但请注意两点：第一，LeetCode 的并发题几乎都是基于 Java 或 C++ 的线程模型来设计的，考察的是并发思维而不是语言特性；第二，Python 的 threading 模块在 I/O 密集型场景下依然很有用，而且 Python 3 的 asyncio 也是大厂面试的高频话题。所以我们这章统一使用 Python 的 threading 和 concurrent.futures 来解题，重点是并发控制的思想本身。

核心思维转变：要写好并发程序，你需要从"顺序执行"切换到"可能在任何时候被中断"的思维方式。每个共享变量都可能是雷，每个没有锁的代码块都可能是 bug 的温床。

并发基础

在开始刷题之前，我们需要建立几个基本概念。如果这些你都已经清楚了，可以跳过这段直接看题；如果还不清楚，请认真看。

线程安全

当多个线程同时访问一个共享变量，且至少有一个线程在写时，如果没有同步机制，结果就是数据竞争（Data Race）。例如两个线程同时执行 counter += 1，这条语句在底层对应三条指令：读值、加一、写回。两个线程交叉执行可能导致最终只加了一次。线程安全就是通过同步机制保证无论线程怎么调度，结果都是正确的。

锁（Lock）

Python 的 threading.Lock 是最基本的同步原语。线程在进入临界区前 acquire()，离开后 release()。也可以用 with lock: 上下文管理器，更安全。

信号量（Semaphore）

信号量维护一个计数器，表示可用资源的数量。acquire() 减少计数器（计数器为 0 时阻塞），release() 增加计数器。Python 中 threading.Semaphore 就是这个语义。

屏障（Barrier）

threading.Barrier(n)：当 n 个线程都调用了 barrier.wait() 后，所有线程才同时继续执行。这很适合"所有线程准备就绪后才开始"的场景。

同步原语 Python 类 适用场景
互斥锁 threading.Lock 保护共享资源，一次一个线程访问
可重入锁 threading.RLock 同一线程可多次获取，避免死锁
条件变量 threading.Condition 一个线程等待某个条件，另一个线程通知它
信号量 threading.Semaphore 限制并发数量、资源池
屏障 threading.Barrier 多线程同步到同一个时间点
事件 threading.Event 线程间信号传递（一发多收）
Python 的 GIL 不影响 LeetCode 并发题的练习：虽然 GIL 让 Python 线程无法真正并行执行 CPU 密集型任务，但 LeetCode 的并发题考察的是"线程间的协作与同步"——这完全可以用 Python 的 threading 模块来模拟。关键是掌握 wait/notify、锁、信号量的使用模式。

经典并发问题

现在进入 LeetCode 的并发题目。这部分题目数量不多但非常经典，几乎每道题都对应一个实际的并发场景。

按序打印 — LeetCode 1114

题目要求：有三个线程分别打印 "first"、"second"、"third"，要求按序打印。也就是说，不管线程启动的顺序如何，输出必须是 "firstsecondthird"。

这道题是并发入门第一题，非常直接。可以用多种同步原语实现。这里我们用事件（Event）来实现——两个 Event 分别控制 first→second 和 second→third 的推进。

from threading import Event

class Foo:
    def __init__(self):
        self.event_12 = Event()
        self.event_23 = Event()

    def first(self, printFirst):
        printFirst()
        self.event_12.set()

    def second(self, printSecond):
        self.event_12.wait()
        printSecond()
        self.event_23.set()

    def third(self, printThird):
        self.event_23.wait()
        printThird()

这个模式叫"信号链"（Signal Chain）。每个阶段完成后发送信号，唤醒下一个阶段。用 Event 还是 Condition 还是 Semaphore 都可以，核心思想是一样的。来，我们看一个踩坑故事。

我第一次写这道题时，犯了一个天真的错误：我在 first 里先 set 后调用 printFirst。结果在某些调度下，second 会在 first 打印之前就执行了——因为 Event.set() 之前的代码还没执行完，第二个线程已经通过了 wait()。教训：信号发送必须在所有操作完成之后。

打印零与奇偶数 — LeetCode 1116

题目要求：三个线程——zero 线程打印 0，odd 线程打印奇数，even 线程打印偶数。给定 n，输出 01020304...0n（0 和数字交替，数字分奇偶）。这道题的复杂性在于三个线程之间的两两同步。

from threading import Lock

class ZeroEvenOdd:
    def __init__(self, n):
        self.n = n
        self.zero_lock = Lock()
        self.even_lock = Lock()
        self.odd_lock = Lock()
        self.even_lock.acquire()
        self.odd_lock.acquire()

    def zero(self, printNumber):
        for i in range(1, self.n + 1):
            self.zero_lock.acquire()
            printNumber(0)
            if i % 2 == 1:
                self.odd_lock.release()
            else:
                self.even_lock.release()

    def even(self, printNumber):
        for i in range(2, self.n + 1, 2):
            self.even_lock.acquire()
            printNumber(i)
            self.zero_lock.release()

    def odd(self, printNumber):
        for i in range(1, self.n + 1, 2):
            self.odd_lock.acquire()
            printNumber(i)
            self.zero_lock.release()

关键点分析：

  • 三把锁的初始状态：只有 zero_lock 是未锁的，odd_lock 和 even_lock 在一开始都是 acquire 状态。这就保证了第一个执行的必然是 zero 线程。
  • zero 线程充当调度器：它根据当前数字的奇偶性，释放 odd_lock 或 even_lock，通知对应的线程执行。
  • 奇偶线程执行完后释放 zero_lock：这样 zero 线程才能继续打印下一个 0。
死锁风险：如果你用 with lock: 上下文管理器的写法，一定要确保锁的释放顺序正确。在这道题中，锁的 cross-thread 传递（zero 线程的锁由 odd/even 释放）很容易导致死锁。一个常见调试方法：加一个 timeout，看哪个线程卡住。

生产者消费者问题

"生产者-消费者"是并发编程中最经典的模式，没有之一。一个或多个生产者往缓冲区放数据，一个或多个消费者从缓冲区取数据，缓冲区有大小限制。LeetCode 有直接对应的题目。

阻塞队列实现 — LeetCode 1188

设计一个阻塞队列（Bounded Blocking Queue），支持 enqueue(element) 和 dequeue()。队列满时 enqueue 阻塞，队列空时 dequeue 阻塞。

from threading import Lock, Condition

class BoundedBlockingQueue:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.queue = []
        self.lock = Lock()
        self.not_full = Condition(self.lock)
        self.not_empty = Condition(self.lock)

    def enqueue(self, element: int) -> None:
        with self.not_full:
            while len(self.queue) >= self.capacity:
                self.not_full.wait()
            self.queue.append(element)
            self.not_empty.notify()

    def dequeue(self) -> int:
        with self.not_empty:
            while len(self.queue) == 0:
                self.not_empty.wait()
            val = self.queue.pop(0)
            self.not_full.notify()
            return val

注意这里使用 Condition 而不是单纯用 Lock。Condition 相当于"锁 + 等待队列"的组合——线程在条件不满足时释放锁并等待，满足时被唤醒并重新获取锁。它比轮询（busy waiting）高效得多。

我在美团二面时就被问到这道题。我写完基础版本后，面试官问了三个 follow-up："如果多个生产者和多个消费者，代码需要改吗？"（不需要，Condition 的设计本身就是支持多对多的）。"如果生产速度远大于消费速度，怎么解决？"（可以丢弃或合并请求，或者给生产者加反压机制）。"如果队列用链表而不是数组，有什么优缺点？"（链表没有扩容问题，但节点分配有开销）。这些问题比你写下代码重要得多。

H2O 生成 — LeetCode 1117

题目要求：两个氢线程和一个氧线程分别调用 hydrogen() 和 oxygen()，并确保最终生成的水分子（H₂O）是按"两个 H 一个 O"的规律组合起来的。这是一个典型的多线程协作模式。

from threading import Barrier, Lock, Semaphore

class H2O:
    def __init__(self):
        self.h_sem = Semaphore(2)    # H 可以有两个同时打印
        self.o_sem = Semaphore(1)    # O 只能有一个打印
        self.barrier = Barrier(3)    # 三个线程到齐后一起释放

    def hydrogen(self, releaseHydrogen):
        self.h_sem.acquire()
        self.barrier.wait()
        releaseHydrogen()
        self.h_sem.release()

    def oxygen(self, releaseOxygen):
        self.o_sem.acquire()
        self.barrier.wait()
        releaseOxygen()
        self.o_sem.release()
设计思路：Barrier(3) 保证三个线程（两个 H、一个 O）都到达后才同时执行各自的打印操作——这确保了 H 和 O 打印的相对顺序是随机的（在同一个屏障点后谁先抢到 CPU 谁先打印），但分子的组成一定是 2H + 1O。这个设计非常优雅——用信号量控制配额，用屏障同步组队。

Web Crawler 多线程设计

Web Crawler Multithreaded（LeetCode 1242）：设计一个多线程网页爬虫，给定起始 URL，爬取所有符合特定条件的 URL。这道题是 LeetCode 上最有工程感的并发题——它模拟了一个真实的多线程爬虫场景。

这道题的核心难点是：

  • 多个爬虫线程并发抓取
  • 需要去重（同一个 URL 不重复爬）
  • 需要知道什么时候所有线程都完成了（终止条件）
from threading import Thread, Lock
from collections import deque

class CrawledThread:
    def __init__(self, startUrl, htmlParser):
        self.htmlParser = htmlParser
        self.hostname = startUrl  # 简化处理
        self.visited = set([startUrl])
        self.lock = Lock()
        self.pending = deque([startUrl])
        self.active_workers = 0
        self.all_done = False

    def crawl(self, url: str) -> List[str]:
        # 主爬虫方法 — 使用多线程
        NUM_WORKERS = 4
        self.active_workers = NUM_WORKERS

        def worker():
            while True:
                with self.lock:
                    if not self.pending:
                        self.active_workers -= 1
                        if self.active_workers == 0:
                            self.all_done = True
                        return
                    current_url = self.pending.popleft()
                    self.active_workers += 1

                urls = self.htmlParser.getUrls(current_url)

                with self.lock:
                    for u in urls:
                        if u not in self.visited and self._same_host(u):
                            self.visited.add(u)
                            self.pending.append(u)

        threads = [Thread(target=worker) for _ in range(NUM_WORKERS)]
        for t in threads:
            t.start()
        for t in threads:
            t.join()

        return list(self.visited)

    def _same_host(self, url: str) -> bool:
        # 判断是否属于同一主机
        return self.hostname in url
爬虫并发设计要点：（1）"任务队列 + 工作线程"是标准的并发模型；（2）visited 集合需要加锁保护——多个线程不能同时修改，否则可能出现重复爬取；（3）终止条件需要小心处理——所有线程都空闲且任务队列为空时才能结束。这个模式在后续的"并发任务调度"中会反复使用。

这个爬虫模型是真实生产环境的极度简化版。真实的爬虫还需要考虑：robots.txt 协议、请求频率控制（对同一个域名的请求间隔不能太短）、断点续爬、去重机制的持久化（Bloom Filter）等。但 LeetCode 这道题已经很好地抓住了并发爬虫的核心矛盾。

踩坑与总结

多线程编程是 bug 率最高的编程范式之一。即使是有多年经验的工程师，也难免写出隐藏的并发 bug。下面是我整理的最常见的坑。

死锁

两个线程各自持有一把锁，同时等待对方释放——这就是死锁。经典的"哲学家就餐问题"就是这个场景。避免死锁的最简单方法：全局固定锁获取顺序。如果要获取多把锁，所有线程必须以同样的顺序获取。

活锁

活锁中线程并没有阻塞，但一直在做无用功。比如两个线程检测到冲突后，各自"谦让"释放资源，然后重新尝试——结果又冲突了，如此循环。解决方法：引入随机退避（Exponential Backoff），避免所有线程在同一时间重试。

线程池大小设置

这是一个没有标准答案的问题，但有一个经验公式：

任务类型 线程池大小计算公式
CPU 密集型 CPU核心数 + 1（加一个补偿页缺失）
I/O 密集型 CPU核心数 * (1 + 等待时间/计算时间)

Python 中用 concurrent.futures.ThreadPoolExecutor 可以方便地管理线程池。但请注意：GIL 导致 Python 的 CPU 密集型多线程无法并行——对于 CPU 密集型任务，请使用 multiprocessing 模块。

Python GIL 的实际情况：Python 3 对 GIL 做了很多优化（比如 GIL 切换从 100 个字节码指令改为基于超时），但 GIL 本质没有消失。面试时如果被问到 GIL，要这样回答："GIL 是 CPython 解释器中的一把全局锁，保证了同一时刻只有一个线程在执行 Python 字节码。它简化了 C 扩展的编写，但限制了多核并行。对于 I/O 密集型任务影响不大——执行 I/O 操作时会释放 GIL。Python 3 的 GIL 经过改进，在大多数场景下表现良好。"

这里还有一个我自己亲身经历的惨痛教训。有一年我做了一个分布式爬虫系统，上线后在低并发环境运行正常，一到高峰期就随机报错。排查了整整两天，最后发现是一个 Counter 对象在多个线程中同时自增——我没有加锁。counter += 1 不是原子的！这个看似人畜无害的表达式包含了读取、增加、写入三个步骤，在线程切换的间隙就会出现竞态条件。从那以后，我所有多线程代码里的计数器都换成了 threading.Lock 保护的版本，或者直接用 threading.Atomic 类（可惜 Python 没有原生的 AtomicInteger，需要使用 multiprocessing.Value）。

多线程与并发的学习路径建议：从 LeetCode 1114（按序打印）开始，掌握基础的 wait/notify 模式；然后做 1115（交替打印 FooBar），理解两线程交替；接着做 1116（零与奇偶），体会三线程协作；最后用 1242（多线程爬虫）收尾，感受真实的并发工程。完成这四道题，你在并发面试题上至少不会慌了。
