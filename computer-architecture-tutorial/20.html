第20章 多核与多处理器架构

同学们好！上一章我们见识了在单核上"压榨"性能的极致手段——超标量、乱序执行、前瞻执行。但单核性能是有极限的。2005年左右，整个行业撞上了一堵墙——功耗墙。从那以后，处理器的进化方向彻底改变了。今天我们就来聊聊这个改变一切的技术：多核（Multi-Core）与多处理器（Multi-Processor）架构。

20.1 为什么我们需要多核

2004年，Intel的Pentium 4 Tejas架构被取消——它原计划达到10GHz，结果功耗已经飙升到150W以上，无法冷却。这正是Dennard Scaling（登纳德缩放定律）失效的后果：过去每当工艺节点缩小，晶体管尺寸缩小、电压降低、频率可以提升，而功率密度保持不变。但大约在130nm节点之后，漏电流剧增，电压无法再等比例下降——频率提升带来的功耗增长变得不可接受。

这就是著名的功率墙（Power Wall）。如果不能继续提高单核频率，唯一的选择就是增加核心数——用多核并行来换取吞吐量的提升。Intel Core 2 Duo（2006年）是x86双核时代的标志性产品，随后四核、六核、八核乃至更多核心迅速铺开。

多核的另一个驱动力来自暗硅（Dark Silicon）理论：在先进工艺下，芯片上不能同时让所有晶体管都全速工作，否则会过热。因此，与其做一个巨大的复杂核心，不如做多个较小的简单核心，在功耗预算内轮流使用不同的部分。今天一个服务器CPU可以轻松拥有64核甚至更多，而手机SoC中也有大小核搭配（big.LITTLE）来实现功耗和性能的平衡。

20.2 SMP与NUMA架构

多核系统有两种主要的内存架构：

  • SMP（Symmetric Multi-Processing，对称多处理）：所有核心通过共享总线或交叉开关连接到同一内存，所有核心看到统一的内存访问延迟。优点是编程模型简单——任意核心访问任意内存地址的速度相同。缺点是可扩展性差，当核心数超过8-16个时，共享总线会成为瓶颈。
  • NUMA（Non-Uniform Memory Access，非统一内存访问）：每个CPU有自己的本地内存（访问快速），也可以访问其他CPU的远程内存（慢30-60%）。这对于大型服务器系统（32核以上）是必需的。

现代处理器在NUMA基础上还有更精细的子NUMA聚类（Sub-NUMA Clustering, SNC）：将芯片内部的L3 Cache和内存控制器划分为多个域，每个域对本地内存访问更快。Intel的至强处理器在胶水方案（多个die封装在一起）中大量使用了类似技术。

讲一个真实的踩坑故事。我参与过一个数据库性能优化项目，部署在一台4路服务器上（4个CPU socket，每个16核共64核）。数据库的吞吐量始终上不去，只达到预期的60%。排查了索引优化、查询优化、磁盘IO，都没用。最后用numactl --hardware一看，发现操作系统的线程调度器把数据库的关键线程和内存分配器分配在了不同的NUMA节点上——一个线程在Socket 0上运行，但它分配的内存却在Socket 2上。每次内存访问都要跨NUMA跳转，延迟增加了40%。解决方案是强制线程和内存绑定到同一NUMA节点（numactl --membind --cpunodebind），性能立刻恢复了。这个经验告诉我们：在多核系统中，理解NUMA拓扑是性能调优的基本功。

20.3 缓存一致性协议深入

多核系统面临一个根本问题：每个核心都有自己的L1/L2 Cache，如果Core 0和Core 1都缓存了内存地址A的数据，Core 0修改了它的副本，Core 1怎么知道自己应该失效并重新加载？答案是缓存一致性协议（Cache Coherence Protocol）。

MESI协议是最经典的协议，定义了四种状态：

  • M（Modified）：数据已修改且只在本Cache
  • E（Exclusive）：数据干净且只在本Cache
  • S（Shared）：数据干净且多份拷贝
  • I（Invalid）：数据失效

当一个核心要对处于S状态的行做写操作时，必须发送总线嗅探（Bus Snoop）让其他核心的行失效。这种基于总线的嗅探协议在核心数少时工作良好，但在大规模系统中，嗅探的广播开销变得不可接受。

于是有了扩展协议：

  • MOESI：增加Owned状态，允许一个Cache持有修改后的数据却不写回主存，其他Cache可以看到这个"脏共享"数据。
  • MESIF（Intel）：增加Forward状态，当多个Cache都有相同数据时，指定一个作为转发者，减少嗅探流量。

在物理实现上，Intel从Sandy Bridge到Broadwell使用环总线（Ring Bus）——所有核心、L3 Cache切片和系统代理挂在一个多环网络上。从Skylake-SP开始，Intel切换到了网格（Mesh）互连，延迟更低、带宽更高。AMD则使用Infinity Fabric——一种基于高速SerDes的片上互连技术，连接CCX（Core Complex）和I/O die。

对于超大规模系统（例如256核以上），嗅探的广播代价已经无法接受。这时需要使用目录式一致性（Directory-based Coherence）：每个内存块有一个"家节点"（Home Node），记录哪些核心的Cache持有该数据。写操作只需要通知家节点，家节点再通知共享者——避免了全局广播。

20.4 内存一致性模型

缓存一致性保证所有核心看到同一地址的写入是一致的，但它不保证不同地址的写入顺序在各核心看来是一样的。这就是内存一致性模型（Memory Consistency Model）要解决的问题。

模型 典型实现 核心规则 性能
SC (Sequential Consistency) 理想模型，无商用处理器 所有内存操按程序顺序全局排列 低
TSO (Total Store Order) x86/x86-64 (Intel, AMD) 写可被缓冲，读可绕过之前的写 中等
Relaxed (弱一致性) ARM, RISC-V, PowerPC 内存操作可任意重排，需显式fence 高

x86的TSO模型意味着：核心可以先将写入结果放到Store Buffer，在后续的读操作中，它可以直接从Store Buffer读取最新的值（Store-to-Load forwarding）。但另一个核心可能还没有看到这个写入——这就是为什么x86多核编程中仍然需要mfence指令来保证可见性。

ARM和RISC-V采用弱一致性（Weakly Ordered）模型，更加激进。编译器或硬件可以自由重排内存访问，除非遇到显式的屏障指令。ARM上使用dmb（Data Memory Barrier），RISC-V使用fence。弱模型给了硬件和编译器更大的优化空间，但编程难度更高。

这里有一个著名的双检测锁（Double-Checked Locking, DCL）bug：

// 错误的DCL — 在弱一致性模型上会崩溃！
class Singleton {
    static Singleton* instance;
    static std::mutex mtx;
public:
    static Singleton* getInstance() {
        if (!instance) {              // 第一次检查
            std::lock_guard<std::mutex> lock(mtx);
            if (!instance) {          // 第二次检查（带锁）
                instance = new Singleton(); // 问题在这里！
            }
        }
        return instance;
    }
};

// 问题: "new Singleton()" 在汇编层面包含：
// 1. 分配内存
// 2. 在内存上构造对象（调用构造函数）
// 3. 将指针赋给instance
// 在弱一致性模型下，步骤2和3可能被重排！
// 线程A执行步骤3（instance非空）但步骤2还没执行
// 线程B看到instance非空，直接使用——访问了未初始化的对象！

// 正确的做法 — 使用 std::atomic 和 memory_order
class SingletonCorrect {
    static std::atomic<Singleton*> instance;
    static std::mutex mtx;
public:
    static Singleton* getInstance() {
        Singleton* p = instance.load(std::memory_order_acquire);
        if (!p) {
            std::lock_guard<std::mutex> lock(mtx);
            p = instance.load(std::memory_order_relaxed);
            if (!p) {
                p = new Singleton();
                instance.store(p, std::memory_order_release);
            }
        }
        return p;
    }
};

这个bug在Java中同样存在（直到JDK 5引入volatile语义才修复），在C++中则直到C++11引入标准内存模型和高层原子操作才得到正确解决。每次我跟学生讲这个故事，都会强调：多核编程中的"看似正确"往往只是"在x86上碰巧正确"。

20.5 多核编程的性能陷阱

最后我们来聊聊多核编程最核心的性能理论——Amdahl定律（Amdahl's Law）。公式非常简单：

Speedup = 1 / (s + p/N)

其中s是串行部分比例，p是并行部分比例，N是核心数。关键洞察：即使并行部分可以无限加速，整体加速极限也受制于1/s。一个程序95%的代码可以并行（p=0.95, s=0.05），在64核上的加速上限只有1/(0.05+0.95/64) ≈ 14.5倍——不到16倍的加速，尽管你有64个核心！

🔑 Amdahl定律的启示：Amdahl定律是悬在并行计算头顶的达摩克利斯之剑——只要一点点串行代码，就能让再多核的性能提升归于零。这就是为什么多核编程需要从算法层面重新思考，把串行部分压缩到极致。很多时候，并行化的收益不是由你能并行多少决定的，而是由你"不能并行的那部分"决定的。

实际多核编程中常见的性能陷阱还包括：

  • 伪共享（False Sharing）：两个线程操作不同但位于同一Cache行的变量，每次写都导致整行缓存失效，引发"乒乓"效应。
  • 同步开销：锁竞争、原子操作的CAS循环、屏障等待——同步的成本随着核心数的增加反而可能增长（因为更激烈的竞争）。
  • 缓存一致性流量：频繁读写的共享数据导致大量的MESI协议消息在核心间传递，消耗带宽和延迟。

应对这些挑战需要程序员从数据并行（Data Parallelism）和任务并行（Task Parallelism）的角度重新设计算法。数据并行（如对一个大数组分块处理）通常扩展性更好，因为数据碎片之间的通信很少。而任务并行（如流水线）由于任务间的依赖关系，往往更难做到线性加速。

好了，多核的故事就讲到这里。下一章我们将探讨另一种并行化的思路——不是多核并行（多指令多数据流），而是SIMD（单指令多数据流）和向量处理器的世界。如果说多核是为了吞吐量，那么SIMD就是为了让单条指令能处理更多数据。我们下章见！
