🌾 第28章 综合实战项目：智慧农业传感网

大家好，我是你们的讲师。经过前面27章的学习，你已经掌握了从传感器原理、无线通信、网络协议到AI融合、新一代传感网的完整知识体系。现在是时候把所有知识串起来了——用一个真实的综合项目，把每一章学过的东西都落实到代码和硬件上。

项目背景来自我2025年参与的一个实际工程：某农业科技公司要在山东寿光的100亩温室大棚中部署一套传感网系统。这个项目甲方明确提出了几个硬性要求：系统必须电池供电且续航一年以上、数据要能远程查看、部署成本不能超过15元/平方米。来，我们一步步走完这个项目的全过程。

28.1 需求分析

项目范围是100亩连栋温室大棚，分为40个标准温室单元，每个单元约2.5亩（约1667平方米）。每个温室单元需要监测以下参数：

  • 🌡️ 温湿度：精度±0.3°C / ±3% RH，每5分钟采集一次
  • ☀️ 光照强度：0-65535 lux，每10分钟采集一次
  • 🧪 土壤pH值：精度±0.1pH，每30分钟采集一次
  • 🌬️ CO₂浓度：0-5000ppm，每15分钟采集一次

每个温室单元部署3个传感器节点（覆盖面积约550m²/节点），共计120个终端节点。加上3个集中器网关（每13-14个节点对应一个网关），形成星型拓扑。

28.2 协议选型决策

在协议选型阶段，团队内部发生过激烈的争论。候选方案有三个：

对比维度 LoRaWAN ZigBee NB-IoT
通信距离 2-5km（郊区/农村） 10-100m 1-3km（依赖基站覆盖）
穿透能力（温室大棚） 强（868MHz穿透好） 一般（2.4GHz衰减大） 较强（低频段）
终端功耗 极低（休眠2μA） 低（休眠3μA） 中等（需维持连接）
电池寿命（2000mAh） >18个月 8-12个月 6-8个月
数据速率 0.3-50kbps 250kbps 60-200kbps
网关成本 ¥800-1500/台 ¥500-800/台 无需自建（SIM卡月租）
部署独立性 完全自建 完全自建 依赖运营商基站

最终选型是LoRaWAN，理由有三：第一，温室大棚的钢架结构对2.4GHz信号衰减很明显，868MHz的LoRa穿透力好得多；第二，甲方要求电池寿命一年以上，只有LoRaWAN的ADR（自适应数据速率）和省电模式能确保这个指标；第三，寿光郊区NB-IoT基站覆盖不完整，自建LoRaWAN网关可以消除对运营商的依赖。

💡 选型教训：项目初期曾有工程师坚持用ZigBee，理由是"大家都用ZigBee做农业"。但实测发现，在温室大棚中，ZigBee节点之间的多跳路径非常不稳定——番茄植株长大后的叶片对2.4GHz信号的吸收非常严重。这就是典型的"技术路线选择应基于物理层实测，而非业界潮流"的案例。

28.3 硬件设计

节点主控选择了STM32WLE5——这是一颗集成了Cortex-M4内核和LoRa收发器（SX126X兼容）的SoC。为什么选它？因为MCU和LoRa射频集成在一颗芯片上，PCB面积从40×60mm缩小到25×35mm，BOM成本降低约35%，而且芯片级的匹配网络比分立方案更稳定。

传感器选型：

  • SHT30：温湿度传感器，I²C接口，±0.3°C精度，休眠功耗0.2μA
  • BH1750：光照传感器，I²C接口，0-65535 lux，休眠功耗1μA
  • CS1503：土壤pH值传感器，模拟输出，通过STM32WLE5内部ADC采集
  • MH-Z19C：CO₂传感器，UART接口，0-5000ppm，加热功耗85mA（峰值1秒）

电源部分：两节ER14505（AA型）锂亚硫酰氯电池，标称电压3.6V，容量2700mAh。搭配一个Torex XC6219 LDO，静态电流仅1μA。这个组合的理论待机时间约为2.4年（按每日数据量计算）。

28.4 节点固件开发

固件基于Contiki-NG操作系统编写——这是专门为资源受限物联网设备设计的轻量级OS，内置了完整的LoRaWAN协议栈。以下是一段完整的传感器节点主循环代码，它实现了"传感器采集→LoRaWAN发送→低功耗休眠"的完整闭环：

// 智慧农业传感器节点主循环
// 平台：STM32WLE5 + Contiki-NG
// 传感器：SHT30 (温湿度) + BH1750 (光照)

#include "contiki.h"
#include "dev/leds.h"
#include "sht30.h"
#include "bh1750.h"

PROCESS(sensor_node_process, "Sensor Node");
AUTOSTART_PROCESSES(&sensor_node_process);

// LoRaWAN 发送缓冲区
static uint8_t tx_buffer[51];
static uint8_t app_data[11];

// 传感器数据读取
static void read_sensors(void) {
    uint16_t temp_raw, hum_raw;
    uint16_t lux_raw;

    // 1. 读取SHT30 (I²C, 耗时约15ms)
    sht30_read(&temp_raw, &hum_raw);
    // 2. 读取BH1750 (I²C, 耗时约180ms in high-res mode)
    bh1750_read(&lux_raw);

    // 3. 打包应用层数据 (Big Endian)
    app_data[0] = (temp_raw >> 8) & 0xFF;   // 温度高字节
    app_data[1] = temp_raw & 0xFF;           // 温度低字节
    app_data[2] = (hum_raw >> 8) & 0xFF;    // 湿度高字节
    app_data[3] = hum_raw & 0xFF;            // 湿度低字节
    app_data[4] = (lux_raw >> 8) & 0xFF;    // 光照高字节
    app_data[5] = lux_raw & 0xFF;            // 光照低字节

    // 4. 电池电压 (通过内部VREF测量)
    uint16_t vbat = batt_measure_mv();
    app_data[6] = (vbat >> 8) & 0xFF;
    app_data[7] = vbat & 0xFF;
}

PROCESS_THREAD(sensor_node_process, ev, data) {
    static struct etimer timer;

    PROCESS_BEGIN();

    // Phase 1: 初始化 (对应第24章 硬件设计实战)
    lora_init();
    sht30_init();
    bh1750_init();

    // Phase 2: LoRaWAN Join (对应第19章 LPWAN)
    lora_join_otaa();  // Over-The-Air Activation
    printf("[BOOT] Joining LoRaWAN network...\n");
    while (!lora_joined()) {
        PROCESS_YIELD();
    }
    printf("[BOOT] Joined! DevAddr: 0x%08lX\n", lora_get_devaddr());

    // Phase 3: 主循环 (对应第14章 能量管理)
    while (1) {
        // 设置采集间隔 (5分钟 = 300秒, 对应ADR Class A)
        etimer_set(&timer, CLOCK_SECOND * 300);

        PROCESS_WAIT_EVENT_UNTIL(etimer_expired(&timer));

        // 唤醒 → 采集
        printf("[WAKE] Battery: %d mV\n", batt_measure_mv());
        read_sensors();

        // 发送上行数据 (LoRaWAN, SF12 → SF7 自适应)
        memcpy(tx_buffer, app_data, 8);
        lora_send(tx_buffer, 8, LORA_PORT_USER);

        printf("[SEND] Temp=%d, Hum=%d, Lux=%d\n",
               (app_data[0] << 8 | app_data[1]),
               (app_data[2] << 8 | app_data[3]),
               (app_data[4] << 8 | app_data[5]));

        // 监听下行 (RX1/RX2窗口, 约2秒)
        // Contiki-NG自动处理, 之后自动进入低功耗模式

        // 进入休眠: MCU Stop Mode, LoRa Sleep, RTC唤醒
        // 功耗: ~5μA (含RTC运行)
    }

    PROCESS_END();
}

这段代码有个细节值得说。很多刚入行的工程师会把"采集"和"发送"放在同一个延时节拍里，但我们的实际做法是把采集提前到休眠唤醒后立即执行，然后利用LoRaWAN的RX1/RX2接收窗口等待期间做本地数据处理，而不是CPU空转。这就是第14章讲过的"功耗解耦"——把计算密集型操作分布在通信等待的间隙中完成，而不是叠加在唤醒时间内。

28.5 网关开发

网关基于树莓派4B + LoRa HAT（基于SX1302），运行ChirpStack开源LoRaWAN服务器栈。网关在整个系统中的角色是协议转换和数据中继——它接收所有120个节点的LoRa上行帧，解包后通过MQTT协议转发到云端。

树莓派上运行的服务栈如下：

  • Packet Forwarder：SX1302的底层驱动，负责LoRa物理层的帧接收/发送
  • ChirpStack Gateway Bridge：将Packet Forwarder的UDP数据转换为MQTT消息
  • ChirpStack Network Server：LoRaWAN网络服务器，处理ADR、MAC命令、节点认证
  • ChirpStack Application Server：应用层服务器，负责上下行数据的业务逻辑
  • Mosquitto MQTT Broker：本地MQTT代理，向云端转发数据

3个网关分别覆盖温室的东、中、西三个区域。网关之间通过有线以太网连接（温室内部预埋了网线），避免了网关之间再用LoRa通信造成的干扰和带宽瓶颈。

28.6 云端对接

云端数据流设计为：LoRaWAN → MQTT → Telegraf → InfluxDB → Grafana

网关将MQTT消息发到云端的EMQX集群，Telegraf作为数据采集代理解析JSON Payload，写入InfluxDB时序数据库，Grafana负责前端可视化。整个管道的端到端延迟在3-8秒之间——对于农业场景来说完全足够。

Grafana仪表盘包含以下面板：

  • 全局总览：40个温室单元的温湿度热力图
  • 单棚详情：选定温室的时间序列曲线（温/湿/光/CO₂ × 3个节点）
  • 电池健康：所有节点电池电压的时间趋势，低于3.0V时高亮告警
  • 数据完整率：每个节点理论上报次数 vs 实际收到次数

28.7 系统测试

功耗测试

功耗测试是项目中最关键的测试环节。我们使用Keysight N6705C 直流电源分析仪串接在电池和节点之间，精确测量各状态的电流：

工作状态 实测电流 持续时间 单次能耗
深度休眠（Stop Mode） 4.8 μA ~299秒 1.44 μAh
传感器采集 2.3 mA ~200 ms 0.13 μAh
LoRa发送（SF10, 14dBm） 42.0 mA ~180 ms 2.10 μAh
RX1/RX2接收窗口 14.5 mA ~2.0 s 8.06 μAh
单周期总能耗（5分钟） 11.73 μAh

按这个数据计算：2700mAh / 11.73μAh × 5分钟 = 约773天，折合25.7个月，满足甲方要求的12个月以上。实际部署中考虑到电池自放电（锂亚电池年自放电率约1%）和极端低温（冬季寿光大棚夜间最低5°C）的影响，预期实际寿命约为18-22个月。

通信可靠性测试

我们在一个标准温室单元内测试了不同位置的RSSI和Packet Delivery Ratio：

  • 距网关50米内（无遮挡）：RSSI -85至-95dBm，PDR 99.8%
  • 距网关100米（穿越2层塑料膜）：RSSI -100至-110dBm，PDR 98.2%
  • 距网关150米（穿越3层塑料膜+番茄植株）：RSSI -115至-120dBm，PDR 95.1%

PDR低于99%的区域，我们通过调整ADR将SF从10调整到7，数据速率提升、空中时间缩短、碰撞概率降低，PDR回升到99%以上。这也是LoRaWAN相比其他协议的独特优势——ADR的闭环控制让网络可以在部署后自动优化。

28.8 踩坑总结

这个项目从头到尾做了8个月，踩的坑说多不多说少不少，我挑几个最值得分享的：

⚠️ 踩坑1：天线选型
项目初期采购了一批PCB板载天线（陶瓷贴片天线），价格便宜（¥0.8/颗）。实测发现，在贴片天线距离STM32WLE5的屏蔽罩只有5mm的布局下，天线效率从标称的45%掉到了11%。换用外置1/4波长鞭状天线（¥3.5/根）后，RSSI平均提升了12dB。教训：射频设计中的"天线周边净空区"不可妥协，尤其是在金属温室钢架环境中。
⚠️ 踩坑2：休眠唤醒时序
固件第一版有个低级错误——传感器I²C总线的初始化放在了每次唤醒后的read_sensors()中，但SHT30的上电启动时间（Typ. 1ms）被忽略了。结果是传感器刚上电就发I²C命令，偶尔返回0xFF 0xFF。修复方案：在唤醒后先给传感器供电压100μs的稳定时间，然后再发起始命令。
⚠️ 踩坑3：传感器校准
BH1750在出厂时校准过了，但发现不同节点的光照读数在相同环境下最大差了15%。原因不是BH1750不准，而是传感器安装朝向不同，导致入射角差异。解决方案：在部署时统一将传感器朝南安装，并进行角度补偿标定。
⚠️ 踩坑4：网关天线高度
东区网关架设在温室外侧2米处，西区网关在温室内部1.5米处。结果东区网关的覆盖范围比西区大了3倍。原因是温室大棚的钢架本身就是一个巨大的屏蔽笼。最终所有网关都安装到温室外部，天线高度统一为4米（使用4米玻璃钢管），覆盖面积一致。

28.10 课程总结

到这里，同学们，我们28章的传感网技术教程就全部结束了。让我带着你回顾一下这趟旅程：

  • 第1-7章：我们从传感网的"是什么"开始，学习了传感器的工作原理、无线通信基础、网络拓扑和协议栈架构，搭建了完整的开发环境。这些是地基。
  • 第8-13章：深入到协议内部——MAC层怎么避免碰撞、路由协议怎么找路、时间同步怎么做、定位技术怎么算坐标、数据融合怎么减少冗余。这些是传感网的"操作系统"。
  • 第14-18章：聚焦工程实践——能量管理决定了电池能用多久，安全机制决定了系统会不会被攻破，标准化决定了你的节点能不能和别人对接。一个技术方案，如果只有性能没有工程可靠性，那就是空中楼阁。
  • 第19-23章：拓展边界——LPWAN解决远距离覆盖，工业WSN解决实时性，移动/水下/多媒体传感网解决特殊场景，仿真工具解决设计验证。这些是把WSN从"实验室玩具"变成"工业产品"的关键能力。
  • 第24-25章：回归到"造"——硬件设计、协议栈实现，真正把一颗芯片和一个协议栈变成能跑的产品。
  • 第26-27章：展望未来——AI+传感网让传感器有了"大脑"，无源物联网和通感一体化让传感器有了"永生"的可能。
  • 第28章（本章）：把所有知识串成一个完整的智慧农业项目——从需求到硬件，从固件到云端，从测试到部署。
🔑 送给你的最后一句话：传感网这门技术，表面上是硬件和协议，骨子里是"在约束条件下做出最优工程决策"的能力——功耗约束、带宽约束、成本约束、可靠性约束。你在做每一个设计决策时，心里都要有一本账：这个选择节省了什么、牺牲了什么。能算清楚这笔账的工程师，才是真正能独当一面的传感网专家。
