开篇：一次数据竞争的"编译期救赎"

2019年，我在一个Go微服务项目中遇到一个经典的数据竞争问题：一个全局缓存map在没有锁的情况下被多个goroutine并发读写，一周出现两三次偶发panic，core dump看半天才能定位到原因。Go的race detector能发现一部分问题，但不是100%——毕竟它在运行时检测，总有漏网之鱼。

切到Rust后，同样的并发场景——多个线程并发读取配置、偶尔写入更新——你猜怎样？第一次编译时，Rust编译器直接拒绝了代码，告诉我数据访问可能存在竞争。我不需要花费数小时在竞态条件调试上，因为编译器已经把所有的数据竞争消灭在了代码提交之前。这就是Rust的"Fearless Concurrency"——无畏并发。

Rust的安全承诺：Rust的所有权系统在编译期就保证了数据竞争（Data Race）不会发生。数据竞争被定义为：两个或更多线程同时访问同一内存位置，其中至少一个是写操作，且没有同步机制。Rust的类型系统加Send/Sync trait从根源上杜绝了这种可能。

线程基础 — std::thread

Rust标准库提供了std::thread模块，让你以最直接的方式创建操作系统线程。不同于Go的goroutine（M:N模型）或Erlang的进程，Rust的线程默认是1:1模型——一个Rust线程对应一个OS线程。

spawn和join

use std::thread;
use std::time::Duration;

fn main() {
    // 创建一个新线程
    let handle = thread::spawn(|| {
        for i in 0..5 {
            println!("子线程: 计数 {}", i);
            thread::sleep(Duration::from_millis(10));
        }
    });

    // 主线程继续执行
    for i in 0..3 {
        println!("主线程: 计数 {}", i);
        thread::sleep(Duration::from_millis(10));
    }

    // 等待子线程结束
    handle.join().unwrap();

    println!("两个线程都执行完毕！");
}

thread::spawn接收一个闭包作为线程入口。join()阻塞当前线程直到目标线程结束——如果忘记调用join，主线程可能在子线程执行完之前退出，导致子线程被强制终止。

move闭包与所有权转移

当子线程需要捕获外部数据时，编译器会强制你使用move关键字——将数据所有权从主线程转移到子线程：

use std::thread;

fn main() {
    let data = vec![1, 2, 3, 4, 5];

    // 不加 move：编译错误！
    // 闭包可能比 data 活得更久，但 data 的所有权在主线程
    // let handle = thread::spawn(|| {
    //     println!("{:?}", data);
    // });

    // ✅ 用 move 将 data 所有权转移到子线程
    let handle = thread::spawn(move || {
        println!("子线程看到的 data: {:?}", data);
        // data 在此处被消耗
    });

    // 此时主线程无法再访问 data——所有权已被转移
    // println!("{:?}", data); // ❌ 编译错误：data 已被 move

    handle.join().unwrap();
}

这个设计非常巧妙：所有权转移保证了同一份数据绝对不会同时被两个线程访问，从根本上杜绝了数据竞争。如果你想在多个线程之间共享数据，需要显式使用Arc（我们稍后会看到）。

消息传递 — mpsc通道

Go的并发哲学说"不要通过共享内存来通信，而要通过通信来共享内存"。Rust标准库提供了mpsc（Multi-Producer, Single-Consumer）通道来支持这种模式：

use std::sync::mpsc;
use std::thread;

fn main() {
    // 创建一个通道：发送端和接收端
    let (tx, rx) = mpsc::channel();

    // 创建3个生产者线程
    for id in 0..3 {
        let tx_clone = tx.clone();
        thread::spawn(move || {
            // 每个线程发送多条消息
            for i in 0..3 {
                let msg = format!("线程{}的消息 #{}", id, i);
                tx_clone.send(msg).unwrap();
                thread::sleep(std::time::Duration::from_millis(10));
            }
        });
    }

    // 丢弃原始发送端——否则接收端会一直等待
    drop(tx);

    // 接收端迭代接收所有消息
    for received in rx {
        println!("收到: {}", received);
    }

    println!("所有消息接收完毕！");
}

核心要点：

  • mpsc::channel()返回一对(Sender, Receiver)
  • Sender实现了Clone——支持多个生产者
  • Receiver是单消费者——不能Clone
  • send()返回Result——当接收端被drop时返回错误
  • rx实现了Iterator——可以用for循环接收直到所有Sender都drop
通道缓冲区：mpsc::channel()创建的是异步通道——发送端不阻塞。标准库也提供mpsc::sync_channel(n)创建同步通道，缓冲区为n条消息，满了之后发送端会阻塞等待接收端消费。

共享状态 — Mutex、Arc、RwLock

消息传递不是万能的。有些场景下共享状态更自然——比如多个线程更新同一个计数器。Rust提供了互斥锁和读写锁来安全地共享可变数据。

Mutex<T> — 互斥锁

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    // Arc 提供跨线程共享，Mutex 提供互斥访问
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
            // 锁在 num 离开作用域时自动释放
        }));
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("最终计数: {}", *counter.lock().unwrap());
    // 输出：最终计数: 10（100%确定，无数据竞争！）
}

Mutex的lock()返回一个MutexGuard——这是一个智能指针，实现了Deref和Drop。当MutexGuard离开作用域，锁自动释放。如果锁被其他线程持有且发生了panic，lock()会返回PoisonError——Rust用"中毒（poisoning）"机制确保不安全状态不会被静默传播。

死锁风险：Rust不能防止死锁！如果你同时持有两个锁（A和B），线程1锁A再锁B，线程2锁B再锁A——经典的死锁就发生了。Rust不会在编译期检测死锁，这是运行时逻辑问题。预防策略：
  • 始终按相同的顺序获取多个锁
  • 尽量缩短持有锁的时间——只在临界区内操作
  • 考虑用std::sync::try_lock采用"尝试-回退"策略

RwLock<T> — 读写锁

当读操作远多于写操作时，RwLock比Mutex更高效——它允许多个读取者同时访问，写操作则独占：

use std::sync::{Arc, RwLock};
use std::thread;

fn main() {
    let data = Arc::new(RwLock::new(vec![1, 2, 3]));

    let mut handles = vec![];

    // 多个读取者
    for i in 0..5 {
        let data = Arc::clone(&data);
        handles.push(thread::spawn(move || {
            let read = data.read().unwrap();
            println!("读者 {} 看到: {:?}", i, *read);
        }));
    }

    // 一个写入者
    {
        let data = Arc::clone(&data);
        handles.push(thread::spawn(move || {
            let mut write = data.write().unwrap();
            write.push(4);
            println!("写入者添加了元素");
        }));
    }

    for handle in handles {
        handle.join().unwrap();
    }
}

RwLock vs Mutex：读多写少用RwLock，读写相当或写多读少用Mutex。RwLock有额外的管理开销，在写密集场景下性能可能不如Mutex。

Send与Sync — 自动trait

Rust中并发安全的核心是两个标记trait（没有方法的trait）：

Trait 含义 自动实现
Send 类型的所有权可以跨线程转移 绝大多数类型（除Rc<T>、裸指针等）
Sync 类型的不可变引用可以跨线程共享，即&T: Send 绝大多数类型（除RefCell<T>、Cell<T>等）

为什么会自动实现？因为编译器根据字段推导：如果一个结构体的所有字段都是Send，那么该结构体自动Send。如果一个类型的所有字段都是Sync（即&T: Send），那么该类型自动Sync。

Send/Sync 自动实现的条件：你几乎不需要手动实现它们。编译器根据类型组成自动推导。如果你发现某个类型不能跨线程（比如Rc<T>），是编译器在保护你——而不是"编译器太笨"。Rc的非原子引用计数在多线程下会数据竞争，所以它不是Send，也不是Sync。要用Arc替换。

手动实现Send或Sync需要使用unsafe——因为编译器无法验证你的实现是否正确。99.9%的情况下你不需要手动实现它们。

原子类型 — AtomicBool / AtomicUsize

对于简单的整数或布尔标志，互斥锁太重了。Rust提供了原子类型——利用CPU的原子指令实现无锁并发操作：

use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::Arc;
use std::thread;

fn main() {
    let running = Arc::new(AtomicBool::new(true));
    let running_clone = Arc::clone(&running);

    let worker = thread::spawn(move || {
        while running_clone.load(Ordering::Relaxed) {
            // 执行工作...
            println!("工作中...");
            thread::sleep(std::time::Duration::from_millis(100));
        }
        println!("收到停止信号，退出。");
    });

    // 主线程等待一会儿后发出停止信号
    thread::sleep(std::time::Duration::from_millis(300));
    running.store(false, Ordering::Relaxed);

    worker.join().unwrap();
}

Rust标准库提供的原子类型包括：AtomicBool、AtomicI8/AtomicI16/AtomicI32/AtomicI64/AtomicIsize、AtomicU8/AtomicU16/AtomicU32/AtomicU64/AtomicUsize，以及AtomicPtr<T>。

Ordering内存序

Ordering参数控制原子操作的内存顺序保证，直接影响性能和正确性：

Ordering 保证级别 性能 推荐场景
Relaxed 无顺序保证 最快 单调计数器、统计指标
Release/Acquire 写-读同步 中等 标志位、锁实现
AcqRel Acquire+Release 中等 Read-Modify-Write操作
SeqCst 全局顺序一致 最慢 复杂同步逻辑（默认值，但谨慎使用）
选择指南：新手通常从SeqCst开始（最安全但最慢），然后通过性能分析确定瓶颈后放松到Acquire/Release。Relaxed只在计数器等不依赖顺序的场景下使用。内存序是复杂的话题，理解错了可能写出比不用原子更糟糕的bug。对于大多数场景，Arc<Mutex<T>>已经足够。

让我讲完这个故事：我之前提到的那个Go缓存项目，后来用Rust重写了——使用Arc<RwLock<HashMap>>加上一个AtomicBool做定期清理的标志。重写后的版本跑了18个月没出过一次并发相关的bug。在大多数公司里，一次严重的数据竞争事故造成的损失，就超过了整个团队学习Rust的成本。这就是"无畏并发"的商业价值。

并发原语 适用场景 共享方式 编译期保障
thread::spawn 独立并行任务 move闭包传所有权 所有权转移保证独占访问
mpsc::channel 多生产者-单消费者 消息传递 Send trait保证类型安全
Arc<Mutex<T>> 共享可变状态 互斥锁 Arc：Sync；Mutex：互斥
Arc<RwLock<T>> 读多写少的共享状态 读写锁 同上
Atomic* 简单共享变量 CPU原子指令 类型安全，无锁

本章小结

Rust的并发模型不是最激进的（看看Erlang的Actor模型或Go的goroutine），但它提供了在系统编程语言中独一无二的安全保障。我们从std::thread的线程创建开始——这是最基础但也最可控的并发方式。然后我们通过mpsc通道实践了消息传递模式，又用Mutex和RwLock处理了共享状态，最后理解了Send/Sync两个核心trait如何从编译期保证并发安全。原子类型为轻量级同步提供了无锁方案。

并发编程是你成为Rust高级开发者的必经之路。标准库线程是基础，但如果你的应用需要在I/O密集型场景下处理成千上万的并发任务，OS线程的开销就成了瓶颈——这时候我们需要异步编程。下一章我们将深入async/await的世界，学习如何用少量线程处理海量并发。打开第21章，继续前行。
