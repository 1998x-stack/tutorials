第16章 分布式训练策略

通过上一章的学习，我们掌握了如何在单卡甚至CPU上跑LLaMA推理。但推理是一回事，训练是另一回事。LLaMA 3 405B的参数量相当于4050亿个浮点数——如果全部用FP16存储就是810GB，而目前最大的单GPU显存（H100 80G）也放不下它的零头。要把这样的庞然大物训练出来，分布式训练不是可选项，而是必选项。

分布式训练的核心问题是：如何把一个大模型切分成多份，分配到多个GPU上，让它们协同工作得像一个GPU一样？这个问题的答案涉及数据并行、张量并行、流水线并行三种基本策略，以及它们组合而成的"3D并行"。这一章我会逐一拆解每种策略的原理、优缺点，以及Meta在LLaMA训练中的具体选择。

16.1 数据并行与ZeRO优化

数据并行（Data Parallelism）是分布式训练最朴素的想法：每个GPU上都有一份完整的模型副本，但分别处理不同的数据批次（micro-batch）。每个GPU独立计算梯度，然后通过all-reduce通信操作把所有梯度同步到所有GPU上。这样每个GPU上的梯度都等于全局平均梯度，再用它去更新模型参数。

但这个朴素方案有一个致命问题：每个GPU都要存一份完整的模型参数、梯度、优化器状态。以LLaMA 2 70B为例，AdamW优化器需要为每个参数维护两个动量值（一阶和二阶动量），所以总内存占用是：参数（BF16）140GB + 梯度（BF16）140GB + 优化器状态（FP32）840GB = 1120GB。即使分布到8张H100上，每张也要存140GB——远超80GB的显存容量。

ZeRO（Zero Redundancy Optimizer）正是为了解决这个问题而生的。ZeRO有三种优化级别：

级别 分片内容 内存节省 通信量 说明
ZeRO-1 优化器状态 4× 与DP相同 只分片optimizer state，最简单
ZeRO-2 优化器状态 + 梯度 8× 与DP相同 梯度在反向后立即分片
ZeRO-3 优化器状态 + 梯度 + 参数 N×（N为GPU数） 1.5×DP 参数在计算前动态all-gather

ZeRO-3是最激进的优化级别：它不仅分片优化器状态和梯度，还把模型参数也分片了。在每次前向传播之前，ZeRO-3通过all-gather操作从所有GPU上收集所需的参数分片；计算完成后立即释放不需要的分片。这样，每个GPU只在特定时刻持有它需要的参数块，训练70B模型时每张GPU的显存占用可以从140GB降到20GB以下。

实际对比： 对于70B模型的训练，使用ZeRO-3（配合BF16）可以在64张A100 80G上高效运行，每张GPU的显存峰值约65GB。如果不用ZeRO，同样的硬件配置连模型参数都塞不下。ZeRO-3的通信开销比普通DP高出约50%，但节省的数倍显存让训练大模型变得可能。

16.2 张量并行

如果说数据并行是"按数据维度切分"，那张量并行（Tensor Parallelism，简称TP）就是"按模型维度切分"。TP把一个Transformer层内部的矩阵运算切分成多个GPU来执行。具体怎么做呢？核心思想来自Megatron-LM项目：把线性层按列或行切分。

考虑LLaMA前馈网络中的一个标准线性层 Y = XW，其中输入形状为 [B, S, d_in]，权重形状为 [d_in, d_out]。在TP中，我们可以：

  • 列切分（Column-wise）： 把权重W沿列方向切成 [d_in, d_out/t] 分配给t个GPU，每个GPU计算部分结果，然后通过all-reduce合并
  • 行切分（Row-wise）： 把权重W沿行方向切成 [d_in/t, d_out]，每个GPU计算部分，然后通过all-reduce累加

在Transformer中，TP的典型应用是一个网格模式。对于多头注意力中的QKV投影层，使用列切分，让不同GPU计算不同的注意力头。对于输出投影层，使用行切分。这样，每个GPU之间只需要一次all-reduce通信，就能完成一个完整Transformer层的计算。

# Megatron-LM张量并行的核心切分模式
#
# 注意力层 (t=2个GPU):
#   GPU-0: QKV投影 (列切分) → 计算头1..H/2 → 输出投影 (行切分)
#   GPU-1: QKV投影 (列切分) → 计算头H/2+1..H → 输出投影 (行切分)
#   通信: 输出投影前一次all-reduce
#
# FFN层 (t=2个GPU):
#   GPU-0: gate_proj (列切分) + up_proj (列切分) → SwiGLU → down_proj (行切分)
#   GPU-1: gate_proj (列切分) + up_proj (列切分) → SwiGLU → down_proj (行切分)
#   通信: down_proj前一次all-reduce

TP的优点是每层只需要一次all-reduce，通信开销相对可控。但它有一个硬性要求：TP节点内的GPU必须通过NVLink/NVSwitch等高速互联连接。因为每次训练step中TP需要传输大量中间激活值，如果GPU之间是普通的InfiniBand连接，通信延迟会吃掉所有性能收益。在实践中，TP通常限制在单节点内部（通常是8卡节点），不跨节点。

16.3 流水线并行

流水线并行（Pipeline Parallelism，简称PP）是另一种"按模型维度切分"的思路，但切分方式不同：PP把Transformer的层（layers）按深度方向分配到不同的GPU上。比如LLaMA 70B有80层，分配到4个GPU上，GPU-0负责层0-19，GPU-1负责层20-39，以此类推。

但朴素的PP有一个严重问题：在任何一个时刻，只有1/t的GPU在工作（其他都在等待前一个GPU完成计算），利用率极低。这就是经典的流水线气泡（pipeline bubble）问题。假设有4个GPU，每层前向计算时间为1单位，那么流水线填充需要4单位时间，排空也需要4单位时间，总共12单位时间内只有8单位在做有效计算，气泡占比高达33%。

为了解决气泡问题，现代PP实现引入了微批次（micro-batch）的概念。将一个大batch分成多个micro-batch，让流水线中同时有多个micro-batch在流动。GPipe和1F1B（One-Forward-One-Backward）是两种主流的调度策略。

# 微批次流水线调度示意图（1F1B策略，t=4个GPU，m=8个micro-batch）
#
# 时间→
# GPU-0: F1 F2 F3 F4 B7 B8 B4 B5 B6 B1 B2 B3
# GPU-1:    F1 F2 F3 F4 B7 B8 B4 B5 B6 B1 B2
# GPU-2:       F1 F2 F3 F4 B7 B8 B4 B5 B6 B1
# GPU-3:          F1 F2 F3 F4 B7 B8 B4 B5 B6
#
# 1F1B策略让每个GPU在处理完一个micro-batch的前向后立即开始反向
# 减少了激活值存储压力，同时保持高利用率
# 气泡占比 ≈ (t-1)/(m) = 3/8 = 37.5%  → 当m增大时占比降低

增加micro-batch数量m可以降低气泡占比。一个经验规则是：m ≥ 4t时，气泡占比可以降到10%以下。但m不能太大，因为每个micro-batch需要单独存储激活值供反向传播使用，这会大幅增加显存压力。LLaMA 3训练中，Meta在PP方面采用了一种折中策略：在保证训练吞吐的前提下，将micro-batch数控制在合理范围内。

踩坑故事： 我第一次尝试实现流水线并行时犯了一个经典错误——没有匹配好前向和反向的micro-batch顺序。我用了GPipe的调度策略，先做完所有micro-batch的前向，再统一做反向。这就意味着在最后半个流水线周期中，所有GPU的显存里同时存着所有micro-batch的激活值。对LLaMA-13B来说，这直接导致了OOM。后来换用1F1B策略后，激活值峰值立即降到1/5。这个教训是：分布式训练中，算法选择和内存管理同样重要。

16.4 3D并行组合策略

数据并行、张量并行、流水线并行各有所长。但在LLaMA这样的超大规模模型训练中，单一策略无法满足需求。Meta在训练LLaMA 65B时使用了3D并行——三种策略组合使用，在不同的维度上解决不同的问题。

具体的分工是这样的：

  • 张量并行（TP）： 在单节点内部（8 GPU），通过NVLink高速互联处理单个Transformer层的计算切分。TP的通信量最大，所以必须放在最快的互联上。
  • 流水线并行（PP）： 将模型层分配到不同的GPU组（每组一个TP组），减少"每张GPU都存完整激活值"的压力。PP的通信量较小，可以跨节点。
  • 数据并行（ZeRO）： 在更大规模的GPU集群上做数据并行复制，处理不同batch的数据。ZeRO的通信主要是梯度同步。

Meta在LLaMA 65B上的具体配置是：TP=8（单节点内），PP=4（4个节点），DP=64（64个数据并行副本），总计 8×4×64 = 2048 GPU。这个配置下，训练吞吐量达到了约150 TFLOPS/GPU的利用率。

一个直观的类比： 训练一个大模型就像写一本科教书。数据并行是你雇佣多个作者写不同的章节——每个作者都有一份完整的参考资料。张量并行是你把一章交给多个作者合写——每人负责一段，最后拼接。流水线并行是你把写作流程分成"起草-审稿-定稿"——每个环节由不同的人负责，批量化流转。最好的效率来自这三种模式同时使用。

16.5 FSDP与ZeRO-3对比

FSDP（Fully Sharded Data Parallel）是PyTorch原生的分布式训练方案。它的设计理念和ZeRO-3几乎一致——将模型参数、梯度、优化器状态分片到所有GPU上，在计算时动态收集。但FSDP和DeepSpeed ZeRO-3在实现细节上有一些值得注意的区别。

特性 DeepSpeed ZeRO-3 PyTorch FSDP
分片策略 参数+梯度+优化器状态 参数+梯度+优化器状态
通信后端 定制NCCL包装 原生NCCL
混合精度 AMP + BF16自动 需手动配置
CPU offload 原生支持（效率高） 支持但效率较低
梯度累积 内置优化 需手动处理
与TP/PP集成 高（Megatron-DeepSpeed成熟） 中（较新特性）
易用性 中等（配置复杂） 高（原生API）
性能（同规模） 略高（优化更精细） 接近（近期版本追赶）

选哪个？我的建议是：如果只需要ZeRO-3而不需要TP/PP，FSDP就足够了——它的API更简洁，和PyTorch生态的集成更自然。但如果需要组合TP+PP+DP进行3D并行训练，DeepSpeed是更好的选择，因为DeepSpeed对Megatron-LM的集成已经非常成熟，而FSDP的TP/PP支持还在完善中。

16.6 LLaMA 3 405B训练集群

LLaMA 3 405B的训练规模放眼整个AI行业都是数一数二的。Meta公开了训练集群的详细信息，我们来看看这个"超级工程"是怎么搭建的。

硬件配置： 训练使用了16384张NVIDIA H100 GPU（约值2.5亿美元），分布在16个集群节点上（每个节点1024 GPU）。每个节点内部采用NVSwitch全互联，任意GPU之间都有1600 GB/s的带宽。节点之间通过InfiniBand NDR400（400 Gbps）连接。

并行策略： 405B采用的是"TP=8，PP=16，DP=128"的3D并行配置。TP=8意味着每个节点内部做张量并行，充分利用NVSwitch带宽。PP=16将405B的层分配到16组TP组上，每组处理约2.5B参数的层。DP=128意味着128路数据并行副本，各自处理不同的数据。

通信优化： Meta在通信层面做了大量优化。关键的一点是使用了计算与通信重叠（computation-communication overlap）技术——在计算当前层的前向时，提前发起下一层的参数all-gather。此外，Meta还引入了通信压缩技术，在不影响训练质量的前提下减少通信量。

# LLaMA 3 405B训练并行策略配置
# 从Meta公开的论文中提取

TOTAL_GPUS = 16384
TP_SIZE = 8        # 单节点内张量并行，利用NVSwitch
PP_SIZE = 16       # 16个流水线阶段
DP_SIZE = 128      # 数据并行副本数
# 总计: 8 × 16 × 128 = 16384

# 每张GPU上的显存分布 (H100 80G):
# - 模型参数分片: ~3.0 GB
# - 优化器状态: ~4.5 GB
# - 激活值 (train): ~45 GB
# - 临时buffer: ~15 GB
# - 总计: ~67.5 GB (预留余量)

注意到每张H100上只存了3GB的模型参数分片——这就是ZeRO-3和3D并行的威力。即使是405B的巨无霸模型，通过精细的切分和调度，也可以让单GPU的显存占用降到合理水平。

16.7 通信瓶颈分析

分布式训练中，"计算"的速度往往不是瓶颈——GPU在计算上是极快的。真正的瓶颈经常出现在通信环节。理解不同通信原语的性能特征，是优化分布式训练的基础。

All-Reduce 是数据并行中最常用的通信操作。假设有N个GPU，每个GPU上有一个梯度向量，all-reduce会将所有梯度向量求和并广播给所有GPU。用NCCL的Ring All-Reduce算法，通信成本为 2×(N-1)/N × 数据量（带宽上界）。对于大梯度向量，all-reduce非常高效——因为它只对求和后的结果做一次传输。

Reduce-Scatter + All-Gather 是ZeRO-3的核心通信模式。Reduce-Scatter先将每个GPU上的数据分为N份，每份做归约后分别保存到对应的GPU上。All-Gather再从每个GPU上收集归约后的完整数据。这两个操作加起来的总通信量恰好等于一次全量all-reduce。

那么为什么ZeRO-3反而比DP慢50%呢？问题是出在通信频率上。DP只需要在每个训练step结束时做一次all-reduce（通信整个梯度向量）。而ZeRO-3需要在每个Transformer层的前向和后向各做一次all-gather和reduce-scatter——对于LLaMA 70B的80层来说，这意味着每个step有320次通信操作。每次通信操作都有固定的延迟开销（latency），即使数据量小，累积起来也很可观。

优化技巧： DeepSpeed实现了梯度累积通信合并（gradient accumulation communication coalescing）——将多个小通信操作合并成一个大通信操作，减少延迟开销。此外，DeepSpeed ZeRO-3还支持预取（prefetch），在前向计算当前层时提前发起下一层参数的all-gather。这些优化可以将ZeRO-3的速度提升到DP的95%以上，同时显存节省数倍。

16.8 训练框架对比

选择分布式训练框架是LLaMA训练中的一个重要决策。下面这张表总结了三大主流框架的特点，供你在实际项目中参考：

维度 Megatron-LM DeepSpeed PyTorch FSDP
开发者 NVIDIA Microsoft PyTorch社区
核心并行策略 TP + PP + DP ZeRO-1/2/3 + PP FSDP (ZeRO-3风格)
TP支持 原生实现（最成熟） 通过Megatron集成 实验性支持
PP支持 原生（1F1B/V-K折叠） 原生（PipeDream） 有限
混合精度训练 BF16/FP16原生 AMP自动混合精度 需torch.cuda.amp
序列并行 原生支持 需自定义 不支持
最佳场景 超大规模TP+PP（>1000 GPU） ZeRO-3或中等规模PP（100-1000 GPU） 纯ZeRO-3场景（<256 GPU）
学习曲线 陡峭（配置复杂） 中等 平缓

对于LLaMA训练来说，如果你的GPU数量少于256张且不需要TP/PP，直接使用PyTorch FSDP是最省心的方案。如果需要更复杂的并行策略，DeepSpeed + Megatron的组合是目前事实上的标准选择。而如果你要复现LLaMA 3 405B级别的训练（数万GPU），那么Megatron-LM是唯一经过大规模验证的框架。

分布式训练是一个实践远重于理论的领域。即使你理解了所有理论，实际部署时仍然会遇到各种意想不到的问题——网络拓扑配置错误、NCCL版本不兼容、显存分配碎片化等。但这些挑战正是大模型训练工程师价值所在。下一章中，我们会回到模型推理层面，讨论一个在LLaMA 3中取得重大突破的方向——上下文窗口扩展。
