第12章 异常与中断处理

同学们好！到目前为止，我们一直假设程序是线性执行的 ——指令从存储器中顺序取出，即使有分支也是在预测的路径上。 但在现实世界中，程序运行时会出现各种"意外"： 除零错误、访问非法内存地址、外部设备请求服务等。 处理器如何优雅地处理这些意外，然后无缝地回到被中断的程序？ 这就是本章的主题——异常与中断处理。

异常 vs 中断

在计算机体系结构中， "异常"和"中断"这两个术语经常被混用，但它们有本质区别：

  • 异常（Exception）： 由正在执行的指令触发，是同步的。 例如：非法操作码、除零错误、系统调用（ecall）、 页错误、访问未对齐的内存地址等。 这些事件与程序的执行流直接相关，发生在可预测的时刻。
  • 中断（Interrupt）： 由处理器外部的设备触发，是异步的。 例如：网卡收到数据包、定时器到期、键盘被按下、 磁盘完成读写等。 中断与当前执行的指令无关，可以在任何时刻发生。

在RISC-V术语中，两者统称为"Trap"（陷入）。 但系统工程师通常还是区分"异常"和"中断"， 因为它们处理的流程有微妙差异。 一个重要的区别是： 异常发生时，处理器可以精确地知道哪条指令出了问题， 而中断发生时，处理器只知道"有个外部事件需要处理"。

以下是RISC-V定义的主要异常和中断代码 （mcause寄存器的值）：

精确异常处理

在流水线处理器中，异常处理面临一个独特的难题： 当一条指令在EX阶段触发异常时， 后面几条指令可能已经进入了流水线的不同阶段。 如果直接清空流水线，后面的指令会丢失它们已经部分执行的副作用。 如果不清空，这些指令的结果可能会被错误地提交。

精确异常（Precise Exception） 要求异常发生时：

  • 异常指令之前的所有指令都已完成（committed）， 它们的执行结果是可见的。
  • 异常指令及其之后的所有指令都没有执行， 或它们的执行结果被丢弃。
  • 保存异常指令的PC， 让异常处理程序知道问题出在哪里。

实现精确异常的硬件机制是 按序提交（In-Order Commit）： 每一条指令必须严格按照程序顺序提交其执行结果。 在五级流水线中，这意味着WB阶段不能提前写回结果 ——必须等到前面的指令都完成WB后才能进行。 这实际上是乱序执行处理器中Reorder Buffer（ROB）的前身。 如果没有精确异常保证，操作系统的调试将是一场灾难 ——你无法知道程序崩溃时到底执行到了哪条指令。 更糟糕的是，操作系统无法正确地杀死一个非法进程， 因为无法确定进程的哪些指令已经执行、哪些没有。

不精确的异常是调试的噩梦 ——这也是为什么现代处理器宁愿增加复杂度 也要保证精确异常。早期的SPARC处理器 支持非精确异常的"快速模式"， 结果操作系统开发者发现调试器几乎无法工作， 因为异常返回地址指向的不是触发异常的指令， 而是之后的某条指令。 这个教训促使所有现代处理器都优先保证精确异常。

RISC-V的异常处理机制

RISC-V将异常处理机制设计得非常清晰和精简。 与x86的复杂分段和特权级不同， RISC-V使用一组控制状态寄存器 （Control and Status Registers, CSRs） 来管理异常处理。关键的CSR包括：

  • mtvec（Machine Trap Vector）： 保存异常处理程序的入口地址。 当异常发生时，处理器跳转到这个地址开始执行。
  • mepc（Machine Exception PC）： 保存触发异常的指令的PC值。 异常处理完毕后，用mret指令将mepc写回PC，恢复执行。
  • mcause（Machine Cause）： 保存异常的原因编号（如上表所示）。 最高位指示是中断还是异常，低位置指示具体原因。
  • mtval（Machine Trap Value）： 保存与异常相关的附加信息。 例如地址访问错误时保存非法地址， 非法指令时保存指令编码。
  • mstatus（Machine Status）： 保存处理器的状态信息， 包括全局中断使能位（MIE）、 之前的中断使能位（MPIE）和特权级（MPP）。

RISC-V的异常处理流程如下：

  1. 将当前PC保存到mepc
  2. 将异常原因写入mcause，附加信息写入mtval
  3. 保存当前中断使能位和特权级到mstatus的MPIE和MPP字段， 然后清除MIE（禁止嵌套中断）
  4. 跳转到mtvec指定的地址开始执行异常处理程序
  5. 异常处理程序执行完毕后，执行mret指令： 从mepc恢复PC，从mstatus恢复中断使能位和特权级

下面是一个简单的RISC-V陷阱处理程序框架：

# RISC-V Trap Handler (机器模式)
.section .text.trap
.align 2
trap_handler:
    # 保存现场 (保存被中断程序的上下文)
    csrrw sp, mscratch, sp  # 交换sp和mscratch
    addi sp, sp, -32*8      # 分配保存区域
    sd x1, 0(sp)            # 保存ra
    sd x2, 8(sp)            # 保存sp
    sd x3, 16(sp)           # 保存gp
    # ... 保存其他寄存器 ...

    # 判断异常类型
    csrr t0, mcause
    bltz t0, handle_interrupt   # mcause最高位=1 → 中断
    # 处理异常
    li t1, 8                # ecall的mcause值
    beq t0, t1, handle_ecall
    li t1, 2                # 非法指令
    beq t0, t1, handle_illegal
    j handle_default

handle_interrupt:
    # 根据中断类型分派
    andi t0, t0, 0xFF       # 取低8位
    li t1, 11               # 机器外部中断(MEI)
    beq t0, t1, handle_external
    # ... 其他中断处理 ...

handle_ecall:
    # 系统调用处理
    csrr t0, mepc
    addi t0, t0, 4          # 跳过ecall指令(4字节)
    csrw mepc, t0           # 更新mepc
    # ... 执行系统调用 ...
    j trap_return

trap_return:
    # 恢复现场
    ld x1, 0(sp)
    ld x2, 8(sp)
    ld x3, 16(sp)
    # ... 恢复其他寄存器 ...
    addi sp, sp, 32*8
    csrrw sp, mscratch, sp
    mret

这个陷阱处理程序框架展示了RISC-V异常处理的核心模式： 保存现场 → 判断原因 → 分派处理 → 恢复现场 → mret返回。 所有具体的系统调用处理、设备驱动逻辑 都是在这个框架中由软件实现的。

中断控制器与优先级

在实际系统中，可能有几十个甚至上百个外部设备 能够产生中断。如何管理这些中断的优先级、路由和嵌套？ RISC-V体系结构中，中断控制功能由 平台级中断控制器 （Platform-Level Interrupt Controller, PLIC）负责。

PLIC的核心功能包括：

  • 中断聚合：将所有外部中断源 （GPIO、UART、以太网等）汇聚到一条或几条处理器中断线。
  • 优先级管理：为每个中断源分配优先级。 当多个中断同时发生时， PLIC将最高优先级的中断提交给处理器。
  • 中断屏蔽：允许软件动态启用 或禁用特定中断源。
  • 中断嵌套：高优先级中断可以抢占 低优先级中断的处理过程。 RISC-V通过mstatus.MIE位控制嵌套： 在中断处理程序中重新使能MIE即可允许嵌套中断。

中断的触发方式有两种：

  • 边沿触发（Edge-Triggered）： 检测信号的上升沿或下降沿。 优点是不会丢失中断（即使处理器正忙，边沿信号会被锁存）。 缺点是容易因抖动产生虚假中断。
  • 电平触发（Level-Triggered）： 检测信号的高电平或低电平。 优点是简单可靠， 但要求处理器在中断处理结束前清除中断源的电平信号， 否则会重复触发。

Syscall与陷入

系统调用（System Call）是用户程序 进入操作系统内核的标准途径。在RISC-V中， 系统调用通过ecall指令触发。 让我们以Linux的write()系统调用为例， 完整走一遍user→kernel→user的旅程：

  1. 用户程序调用 write(fd, buf, count)， C库将其转换为RISC-V ABI调用： 设置a7=64（write的syscall编号）、a0=fd、 a1=buf、a2=count，然后执行ecall指令。
  2. ecall触发异常（mcause=8）， 处理器进入机器模式（M-mode）， 跳转到mtvec指定的陷阱处理程序。
  3. 陷阱处理程序保存上下文，从mcause判断这是ecall， 检查a7中的系统调用编号， 分派到对应的内核服务函数。
  4. 内核执行实际的写操作—— 这涉及到文件系统、VFS层、设备驱动等 复杂的软件栈。
  5. 内核将返回值写入a0， 恢复用户程序的上下文，执行mret返回用户模式。
  6. 用户程序从ecall之后的下一条指令继续执行， 读取a0中的返回值。

这个过程中，处理器硬件只做了两件事： 捕获ecall并跳转到陷阱处理程序， 以及执行mret返回。 所有系统调用的分派、权限检查、参数验证 都是由软件完成的。 这种"硬件做最少的事情"的设计哲学 贯穿了RISC-V的整个异常处理系统。 相比之下，x86的sysenter/sysexit机制 在硬件中做了更多的事情 （如自动切换栈、加载段寄存器等）， 但换来的是更高的复杂度和灵活性损失。

异常和中断处理是操作系统和处理器硬件的交汇点。 理解了异常处理，你就理解了操作系统 如何在硬件层面上获得控制权。 下一章我们将视角转回性能优化—— 在解决了单指令执行效率、流水线冲突、分支预测 和异常处理之后，是时候探索指令级并行（ILP） 的最高境界了：静态多发射与VLIW架构。
