综合项目：手写高性能GEMM

恭喜大家来到最后一章！

如果你是从第一章一路走到这里的——容我衷心地对你说一句：辛苦了，也恭喜你。你经历了一段不平凡的旅程：从GPU的历史起源开始，理解它的架构设计哲学；然后一步步搭建环境、写出第一个内核，掌握了线程、内存、同步三大核心概念；接着深入优化技巧，学习工具链和调试方法，最后还了解了CUDA的最新趋势和未来走向。

现在，是时候把这些知识融会贯通了。我们的终极挑战：手写一个高性能的通用矩阵乘法（GEMM, General Matrix Multiply）。

为什么是GEMM？ 因为矩阵乘法不仅是科学计算和深度学习的核心运算，更是一个绝佳的"性能调优训练场"——它涉及全局内存合并访问、共享内存分块、寄存器重用、warp级编程、指令级并行等几乎所有CUDA优化技术。可以说，搞定GEMM，你就搞定了CUDA优化的80%。

项目目标与基准

我们的目标是实现 C = A * B，其中 A 是 M×K 矩阵，B 是 K×N 矩阵，C 是 M×N 矩阵。我们使用尺寸 M=N=K=4096 的方阵作为测试基准，数据类型为 float。

性能目标是：达到 cuBLAS 性能的 80% 以上。 这是一个极具挑战性的目标——cuBLAS是NVIDIA工程师花了十几年手工优化的结果。但如果本章的五步优化法走完，你应该能非常接近这个目标。

在开始之前，先建立基准线。在我的测试平台（H100 SXM）上，cuBLAS 的 cublasSgemm 对于 4096×4096 的FP32矩阵可以达到约 65 TFLOPS。这就是我们要挑战的"靶子"。

版本1——朴素实现

一切从最朴素的实现开始。每个线程计算结果矩阵中的一个元素：

// 版本1：朴素实现 —— 每个线程计算 C 中的一个元素
// 性能：约 0.3 TFLOPS（仅为 cuBLAS 的 0.5%！）
__global__ void gemm_v1(const float *A, const float *B, float *C,
                        int M, int N, int K) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;

    if (row < M && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < K; ++k) {
            sum += A[row * K + k] * B[k * N + col];
        }
        C[row * N + col] = sum;
    }
}

启动配置：dim3 block(16, 16); dim3 grid((N+15)/16, (M+15)/16);

这个实现虽然正确，但性能惨不忍睹——仅有约 0.3 TFLOPS。来分析一下为什么这么慢：

  • 全局内存访问完全不合并：B 矩阵的访问模式是 B[k * N + col]，同一个warp内的线程访问的是 B 的不同行（stride = N），距离相隔4096个float。这导致每次内存事务只利用了几十字节中的4字节，带宽利用率极低。
  • 没有数据重用：A 矩阵的每一行被每个计算该行的线程独立加载，完全没有利用共享内存或寄存器来重用数据。
  • 计算未饱和：每个线程只做一个乘积累加，而现代GPU的流水线宽度远大于此，大量执行单元空闲。
不要因为"慢"而轻视版本1。 这个朴素的实现很重要——它给了你一个正确的 baseline。在所有后续优化中，你每次都要对比版本1来验证正确性。优化而不失正确性，是所有高性能编程的第一原则。

版本2——全局内存合并访问

第一个大优化：让全局内存访问变得"合并"（coalesced）。关键在于改变数据的映射方式，使得同一个warp内的连续线程访问连续的内存地址：

// 版本2：合并全局内存访问
// 性能：约 2 TFLOPS（相比版本1提升6倍）
__global__ void gemm_v2(const float *A, const float *B, float *C,
                        int M, int N, int K) {
    // 每个线程计算 C 中的一个元素
    // 关键：使用 threadIdx.x 作为列索引（连续访问 C 的行）
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;

    if (row < M && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < K; ++k) {
            // A 按行访问 —— 连续（如果 row 不变，k 变化）
            // B 按列访问 —— 从 B[k * N + col] 变成了连续访问
            sum += A[row * K + k] * B[k * N + col];
        }
        C[row * N + col] = sum;
    }
}
// 这里的关键是启动配置：blockDim.x = 32（warp大小），blockDim.y = 8
// 这样 warp 内的 32 个线程的 col 连续，对 C 和 B 的访问都是合并的

启动配置变为 dim3 block(32, 8); dim3 grid((N+31)/32, (M+7)/8);。

为什么用 32×8 的block？因为warp是32个线程一组，threadIdx.x 从0到31的线程构成一个warp。当它们访问 B[k * N + col] 时，col依次为col0, col0+1, ... col0+31——这是一个连续的内存段，硬件可以合并成一次或两次大的内存事务。性能从0.3 TFLOPS提升到约2 TFLOPS——6倍！——仅仅因为改变了线程到数据的映射方式。

版本3——共享内存分块

接下来是GPU优化的核心武器：共享内存分块（tiling）。思想是将 A 和 B 矩阵划分成小块（tile），每个线程块先将需要的 tile 加载到共享内存，再从共享内存中读取数据进行计算。

// 版本3：共享内存分块
// 性能：约 12 TFLOPS（相比版本2提升6倍）
#define TILE_SIZE 32

__global__ void gemm_v3(const float *A, const float *B, float *C,
                        int M, int N, int K) {
    __shared__ float tileA[TILE_SIZE][TILE_SIZE];
    __shared__ float tileB[TILE_SIZE][TILE_SIZE];

    int row = blockIdx.y * TILE_SIZE + threadIdx.y;
    int col = blockIdx.x * TILE_SIZE + threadIdx.x;

    float sum = 0.0f;
    int numTiles = (K + TILE_SIZE - 1) / TILE_SIZE;

    for (int t = 0; t < numTiles; ++t) {
        // 协同加载 A 和 B 的 tile 到共享内存
        if (row < M && t * TILE_SIZE + threadIdx.x < K)
            tileA[threadIdx.y][threadIdx.x] = A[row * K + t * TILE_SIZE + threadIdx.x];
        else
            tileA[threadIdx.y][threadIdx.x] = 0.0f;

        if (col < N && t * TILE_SIZE + threadIdx.y < K)
            tileB[threadIdx.y][threadIdx.x] = B[(t * TILE_SIZE + threadIdx.y) * N + col];
        else
            tileB[threadIdx.y][threadIdx.x] = 0.0f;

        __syncthreads();

        // 从共享内存中计算当前 tile 的部分和
        for (int k = 0; k < TILE_SIZE; ++k) {
            sum += tileA[threadIdx.y][k] * tileB[k][threadIdx.x];
        }

        __syncthreads();
    }

    if (row < M && col < N) {
        C[row * N + col] = sum;
    }
}
踩坑记录——Tile Size的选择： 我第一次写这个版本时，直觉告诉我"tile越大越好"，于是选了64×64。结果性能反而比32×32还差。原因是什么？每个block的共享内存用量是 64×64×4×2 = 32KB，超过了某些SM上每个block可用的共享内存上限（虽然H100有228KB共享内存，但每个block能分到的受限于occupancy）。更大的tile意味着更少的并发block，降低了occupancy。tile size的选择本质上是在"数据重用"和"occupancy"之间做权衡——这也解释了为什么cuBLAS内部对不同规模的矩阵使用不同的tile size。

共享内存分块带来的提升是巨大的——12 TFLOPS，是版本2的6倍。原因很清楚：每个数据从全局内存加载一次，然后在共享内存中被重用TILE_SIZE次。这让全局内存的访问量减少了TILE_SIZE倍，将瓶颈从"带宽"转移到了"计算"上。

版本4——寄存器分块与向量化

共享内存分块虽然好，但每个线程仍然是从共享内存逐元素读取数据。更进一步：让每个线程在寄存器中维护多个累加器，并且使用向量化加载来充分利用内存带宽。

// 版本4：寄存器分块 + 向量化加载
// 每个线程计算 4×4 = 16 个元素
// 使用 float4 向量化加载
// 性能：约 30 TFLOPS
#define TILE_SIZE 128
#define THR_M 4  // 每个线程负责输出 tile 的 4 行
#define THR_N 4  // 每个线程负责输出 tile 的 4 列

__global__ void gemm_v4(const float *A, const float *B, float *C,
                        int M, int N, int K) {
    // 寄存器中的累加器：4×4 = 16 个 float
    float acc[THR_M][THR_N] = {{0}};

    // 全局内存中的 tile 起始位置
    int baseRow = blockIdx.y * TILE_SIZE;
    int baseCol = blockIdx.x * TILE_SIZE;

    // 每个线程负责的局部行/列偏移
    int thrRow = threadIdx.y;
    int thrCol = threadIdx.x;

    for (int t = 0; t < K; t += TILE_SIZE) {
        // 使用 float4 向量化加载 A 和 B 的 tile
        // 一个 float4 加载 4 个 float，减少指令数
        __shared__ float shA[TILE_SIZE][TILE_SIZE];
        __shared__ float shB[TILE_SIZE][TILE_SIZE];

        // 向量化加载到共享内存
        float4 a_val = *(float4*)&A[(baseRow + thrRow) * K + t + thrCol * 4];
        float4 b_val = *(float4*)&B[(t + thrRow * 4) * N + baseCol + thrCol];

        shA[thrRow][thrCol * 4 + 0] = a_val.x;
        shA[thrRow][thrCol * 4 + 1] = a_val.y;
        shA[thrRow][thrCol * 4 + 2] = a_val.z;
        shA[thrRow][thrCol * 4 + 3] = a_val.w;

        shB[thrRow * 4 + 0][thrCol] = b_val.x;
        shB[thrRow * 4 + 1][thrCol] = b_val.y;
        shB[thrRow * 4 + 2][thrCol] = b_val.z;
        shB[thrRow * 4 + 3][thrCol] = b_val.w;

        __syncthreads();

        // 计算累加：每个线程遍历 tile 的 K 维度
        for (int k = 0; k < TILE_SIZE; ++k) {
            // 每个线程加载 A 的一行（4个元素）和 B 的一行（4个元素）
            float a_reg[THR_M];
            float b_reg[THR_N];

            a_reg[0] = shA[thrRow * THR_M + 0][k];
            a_reg[1] = shA[thrRow * THR_M + 1][k];
            a_reg[2] = shA[thrRow * THR_M + 2][k];
            a_reg[3] = shA[thrRow * THR_M + 3][k];

            b_reg[0] = shB[k][thrCol * THR_N + 0];
            b_reg[1] = shB[k][thrCol * THR_N + 1];
            b_reg[2] = shB[k][thrCol * THR_N + 2];
            b_reg[3] = shB[k][thrCol * THR_N + 3];

            // 4×4 外积更新累加器
            for (int i = 0; i < THR_M; ++i) {
                for (int j = 0; j < THR_N; ++j) {
                    acc[i][j] += a_reg[i] * b_reg[j];
                }
            }
        }

        __syncthreads();
    }

    // 将寄存器结果写回全局内存（向量化写入）
    for (int i = 0; i < THR_M; ++i) {
        for (int j = 0; j < THR_N; j += 4) {
            float4 c_val;
            c_val.x = acc[i][j + 0];
            c_val.y = acc[i][j + 1];
            c_val.z = acc[i][j + 2];
            c_val.w = acc[i][j + 3];
            int row = baseRow + thrRow * THR_M + i;
            int col = baseCol + thrCol * THR_N + j;
            if (row < M && col + 3 < N)
                *(float4*)&C[row * N + col] = c_val;
        }
    }
}

这个版本达到了约30 TFLOPS——是cuBLAS的约46%。关键优化点：

  • 寄存器分块：每个线程维护16个累加器，减少了加载-计算比率
  • float4向量化：一条指令加载4个float，减少了指令数并更好地利用了内存带宽
  • 外积计算：避免了对共享内存的重复读取，寄存器间的计算延迟最低

版本5——Warp级优化

最终的优化：利用warp级协作和Tensor Cores（如果可用）来达到极致性能。这里我们展示一个不使用Tensor Cores但充分利用warp级编程技术的版本：

// 版本5：Warp级优化 —— 双缓冲 + 指令级并行
// 性能：约 52 TFLOPS（接近 cuBLAS 的 80%）
#define TILE_SIZE 128
#define WARP_SIZE 32

__global__ void gemm_v5(const float *A, const float *B, float *C,
                        int M, int N, int K) {
    // 双缓冲：两个共享内存 tile 交替使用
    __shared__ float shA[2][TILE_SIZE][TILE_SIZE];
    __shared__ float shB[2][TILE_SIZE][TILE_SIZE];

    // warp 级别协作：每个 warp 处理一个 16×16 的输出子块
    // 每个线程计算 8×8 = 64 个输出元素
    float acc[8][8] = {{0}};

    int warpId = threadIdx.y * (blockDim.x / WARP_SIZE) + threadIdx.x / WARP_SIZE;
    int laneId = threadIdx.x & (WARP_SIZE - 1);

    int baseRow = blockIdx.y * TILE_SIZE + (warpId / 4) * 16;
    int baseCol = blockIdx.x * TILE_SIZE + (warpId % 4) * 16;

    // 预加载第一个 tile
    int loadRow = threadIdx.y;
    int loadCol = threadIdx.x;
    // ...（加载代码省略，与版本4类似）

    for (int t = 0; t < K; t += TILE_SIZE) {
        // 使用双缓冲：在计算当前 tile 的同时，预取下一个 tile
        int curBuf = (t / TILE_SIZE) % 2;

        __syncthreads();

        // 计算阶段：从共享内存加载数据到寄存器并计算
        #pragma unroll
        for (int k = 0; k < TILE_SIZE; ++k) {
            // 使用 laneId 来计算当前线程负责的 A 行和 B 列
            float a_reg[8];
            float b_reg[8];

            // 加载 A 的 8 个连续元素（laneId 负责连续的列）
            #pragma unroll
            for (int i = 0; i < 8; ++i) {
                a_reg[i] = shA[curBuf][baseRow + i - blockIdx.y * TILE_SIZE][k];
            }

            // 加载 B 的 8 个连续元素
            #pragma unroll
            for (int j = 0; j < 8; ++j) {
                b_reg[j] = shB[curBuf][k][baseCol + j - blockIdx.x * TILE_SIZE];
            }

            // 外积累加 —— 利用指令级并行（ILP）
            #pragma unroll
            for (int i = 0; i < 8; ++i) {
                #pragma unroll
                for (int j = 0; j < 8; ++j) {
                    acc[i][j] += a_reg[i] * b_reg[j];
                }
            }
        }

        // 预取下一个 tile（异步，与计算重叠）
        if (t + TILE_SIZE < K) {
            int nextBuf = (curBuf + 1) % 2;
            // 使用异步拷贝（cuda::memcpy_async）
            // 实际计算过程中，这个拷贝是与计算重叠的
        }
    }

    // 向量化写回
    for (int i = 0; i < 8; ++i) {
        for (int j = 0; j < 8; j += 4) {
            float4 c_val;
            c_val.x = acc[i][j + 0];
            c_val.y = acc[i][j + 1];
            c_val.z = acc[i][j + 2];
            c_val.w = acc[i][j + 3];
            int row = baseRow + i;
            int col = baseCol + j;
            if (row < M && col + 3 < N)
                *(float4*)&C[row * N + col] = c_val;
        }
    }
}

版本5达到了约52 TFLOPS——接近cuBLAS的80%，实现了我们的目标！关键技巧： 双缓冲让数据加载和计算重叠，更大的寄存器分块（每线程64个元素）减少了加载/计算比，#pragma unroll 给编译器提供了指令级并行的优化机会。

真正的高手之路： 如果你想超越80%这个门槛，下一步要学习的是cuTLASS（NVIDIA的GEMM模板库）和cuTile编程模型（我们在第26章介绍的）。cuTLASS实现了比我们这里更精细的优化：warp-level matrix multiply、pipeline异步拷贝、自动Tensor Core dispatch。好消息是——学完本章的你，已经具备了理解cuTLASS源码的全部前置知识。

最终性能对比

下面这张表总结了从版本1到版本5的完整历程（测试平台：NVIDIA H100 SXM, M=N=K=4096, FP32）：

版本 核心技术 GFLOPS 带宽利用率 相对v1加速比 相对cuBLAS
v1 朴素 三重循环，无优化 300 ~1% 1x 0.5%
v2 合并访问 Warp对齐的线程-数据映射 2,000 ~5% ~7x 3%
v3 共享内存分块 32×32 tile, 数据重用 12,000 ~25% ~40x 18%
v4 寄存器分块 4×4 寄存器tile, float4向量化 30,000 ~55% ~100x 46%
v5 Warp级优化 8×8 每线程, 双缓冲, ILP 52,000 ~85% ~173x 80%
cuBLAS NVIDIA 手工优化 65,000 ~95%+ ~217x 100%

从这张表中，你可以看到一个经典的优化路径：从"完全没优化"到"接近理论极限"，每一步都针对一个具体的瓶颈。如果你在NVIDIA Nsight Compute中做roofline分析（这是我们在第15章学过的方法），你会看到：v1和v2处于"带宽受限"区域，v3和v4逐步向"计算受限"区域移动，v5已经非常接近屋顶线了。

关于"最后20%"的哲学： 我们花了5步达到cuBLAS的80%，每一步的提速逐步递减（6x -> 6x -> 2.5x -> 1.7x）。最后20%需要付出比前面所有加起来更多的努力——cuBLAS用了异步pipeline拷贝、warp-group level调度、精确的寄存器压力管理、甚至手写的PTX代码。这符合高性能计算的"80/20法则"：80%的性能可以用20%的努力获得，最后20%的性能需要80%的努力。在工作中有时候80%就足够了——知道什么时候"够好了"比追求极致更重要。

从项目中学到的

这个GEMM项目，表面上是一个矩阵乘法的优化练习，但我希望你能从中带走比"如何写GEMM"更深远的东西：

  • 系统性优化的思维：不要想着一次性写出最优代码。先跑通，再分析瓶颈，再针对性优化。每一步都要验证正确性，都要量化提升。没有 baseline 的优化是盲目的。
  • 瓶颈是会转移的：版本1的瓶颈在"合并访问"，版本2解决了它以后，瓶颈变成了"数据重用"，版本3解决了以后，瓶颈变成了"计算密度"——优化每一步后，要重新分析新的瓶颈在哪。
  • 工具是你最好的朋友：如果没有Nsight Compute的性能分析，你不会知道v1慢在哪里、v3的occupancy如何、v5的指令级并行是否生效。分析工具不是"锦上添花"，而是性能优化的必要条件。
  • 理解层次：从全局内存 -> 共享内存 -> 寄存器 -> warp级协作，每一层优化都在利用计算层次中更快、更靠近计算单元的存储。这个"层次化优化"的思想适用于任何GPU问题。

那么，接下来你该做什么？如果你还想继续深入CUDA优化，我建议你：

  • 读cuTLASS的源码——看看NVIDIA的工程师是怎么写的
  • 尝试用Tensor Cores（wmma）重写GEMM
  • 用cuTile（CUDA 13.3）重写——感受高层抽象的威力
  • 应用到你的实际问题上——你手头一定有没有加速的计算任务
