第14章 时序约束与分析基础

前几章我们做了很多"正确"的事情——Testbench 覆盖率足够、CDC 路径全部加了同步器、流水线也设计好了。但你的设计真的能在目标时钟频率下稳定工作吗？如果不确定，那你的 FPGA 开发还差最后一步——时序约束和分析。

时序约束（Timing Constraints）是你告诉综合和布局布线工具"设计应该以多快的速度运行"的方式。没有约束，工具会默认尝试最大化性能——但可能在你不需要的地方浪费资源；或者默认约束过于宽松，导致关键路径没有被充分优化。更糟的是：没有约束的时序报告是毫无意义的——"fmax = 325 MHz" 这个数字只有在约束正确的前提下才算数。

14.1 时序分析的基本模型

所有静态时序分析（Static Timing Analysis, STA）都基于同一个基础模型。考虑一个简单的触发器到触发器（register-to-register）路径：

在时钟上升沿（称它为 Launch Edge），源触发器的输出 Q 开始变化，经过 Tclk-q 延迟后到达组合逻辑。组合逻辑经过 Tlogic 延迟后到达目的触发器的 D 输入。下一个时钟上升沿（Capture Edge）到来时，目的触发器采样 D 输入。采样时，D 输入必须在建立时间 Tsetup 之前到达并稳定。

所以 setup slack 的计算公式是：

Setup Slack = Tperiod - (Tclk-q + Tlogic + Tsetup - Tskew)

其中 Tskew 是 Capture Clock 减去 Launch Clock 的偏斜（skew），正值表示 Capture 比 Launch 晚到（对 setup 有利，对 hold 有害）。如果 Slack 为正，说明时序有余量；如果为负，则存在时序违例（violation），设计在目标频率下可能无法正常工作。

Hold time 的分析同样重要——它确保触发器输入在时钟沿之后保持足够长的时间。Hold slack = Tclk-q + Tlogic - Thold + Tskew。Hold 违例通常比 setup 违例更难修复，因为它不能通过降低时钟频率来解决——你需要在物理上增加数据路径的延迟（插入 buffer 或增加路径长度）。

Setup vs Hold 的本质区别：Setup 关心"数据能否在下一个时钟沿之前到达"，它与时钟周期相关——降低频率（增加 Tperiod）就可以修复 setup 违例。Hold 关心"数据在上一个时钟沿之后是否能保持足够长的时间"，它与时钟频率无关——降低频率对 hold 违例毫无帮助。这也是为什么在芯片老化或温度变化时，hold 违例更致命——它不受你"降频"的影响。

14.2 时钟约束

时钟约束是时序分析的起点——没有定义时钟，工具没法判断"一个周期"是多长。

主时钟约束（create_clock）：

# 定义一个 100MHz 的主时钟（周期 10ns），占空比 50%
create_clock -name sys_clk -period 10.000 [get_ports clk]

# 定义一个有抖动的时钟（如果 PLL 输出）
create_clock -name pll_clk -period 5.000 [get_pins mmcm/inst/CLKOUT0]

派生时钟约束（create_generated_clock）：当 PLL/MMCM 输出分频或倍频后的时钟，你需要告诉工具它的相位和频率关系：

# 定义 PLL 输出 200MHz （相对于输入 100MHz，乘以 2）
create_generated_clock -name clk_200m -source [get_pins mmcm/inst/CLKIN] \
    -multiply_by 2 -divide_by 1 [get_pins mmcm/inst/CLKOUT0]

# 定义 SPI SCK（输入时钟的分频输出）
create_generated_clock -name spi_sck -source [get_pins spi_top/spi_master/sck_reg] \
    -divide_by 10 [get_ports sck]

时钟组（set_clock_groups）：对于不同时钟域之间的路径，如果你已经做了 CDC 同步（比如异步 FIFO），应该在约束中将它们声明为异步关系——这样工具就不会分析这些路径的 setup/hold（它们由你设计的同步器保证安全），可以显著减少布局布线工具的负担：

# 声明两个时钟组为异步关系
set_clock_groups -asynchronous -group [get_clocks sys_clk] \
    -group [get_clocks pll_clk]

14.3 I/O 约束

FPGA 不是独立存在的——它总是连接到外部的芯片（ADC、DAC、DDR、PHY、MCU）。这些外部器件也有自己时序要求，你需要告诉工具它们对 FPGA 的输入输出施加了什么样的时间窗口。

set_input_delay：定义外部器件在 FPGA 输入引脚上提供数据的时间窗口。最大值（max）用于 setup 分析——表示数据在时钟沿之后最晚什么时候到达；最小值（min）用于 hold 分析——表示数据最早什么时候到达。

# ADC 在时钟上升沿之后 2ns 开始提供数据，最多 5ns 后数据失效
set_input_delay -clock [get_clocks adc_clk] -max 5.000 [get_ports adc_data*]
set_input_delay -clock [get_clocks adc_clk] -min 2.000 [get_ports adc_data*]

set_output_delay：定义外部器件在 FPGA 输出引脚上采样数据的要求。同样分 max 和 min——max 用于 setup 分析（数据需要在多长时间内到达外部器件），min 用于 hold 分析（数据需要保持多久）。

# 外部 DAC 在时钟沿之前 3ns 需要数据稳定，之后保持 1ns
set_output_delay -clock [get_clocks dac_clk] -max 3.000 [get_ports dac_data*]
set_output_delay -clock [get_clocks dac_clk] -min 1.000 [get_ports dac_data*]
I/O 延迟中最常见的错误：新手常常把 set_input_delay 和 set_output_delay 的值设成 FPGA 内部的延迟——但这两个约束描述的是芯片外部的时序，不是 FPGA 内部的。例如，一个 ADC 的数据手册上说"在随路时钟上升沿之后 2-5ns，数据有效"——那么 set_input_delay -min 2 -max 5 就是正确的。如果你搞反了方向，把 setup/hold 时间写成了 FPGA 触发器的 Tsu/Th，那么工具计算出的 slack 将会完全错误。

14.4 时序违例诊断与修复

当你在 Vivado 中运行完布局布线并查看时序报告时，以下几个指标决定了设计是否收敛：

  • WNS（Worst Negative Slack）：所有路径中最差的 setup slack。这是最重要的指标——WNS ≥ 0 表示 setup 收敛。
  • TNS（Total Negative Slack）：所有负 slack 路径的总和。TNS 接近 0 意味着违例路径很少；TNS 很大意味着大量路径都需要修复。
  • WHN（Worst Hold Slack）：最差的 hold slack。同样要求 ≥ 0。
  • THN（Total Hold Slack）：所有 hold 违例的总和。

当遇到时序违例时，下面是常见的原因和修复方法：

违例类型常见原因修复方法
Setup 违例（长路径）组合逻辑太深——多级加法器、乘法器、大 MUX 链插入流水线寄存器切割路径
Setup 违例（高扇出）一个信号驱动了太多负载（复位、使能、时钟使能）手动复制寄存器（/* synthesis keep */ + duplicate）
Setup 违例（跨区域）路径跨越了芯片的远端区域，线延迟过大在 floorplan 中将相关模块放置在一起（Pblock）
Hold 违例数据路径比时钟路径快太多在数据路径中插入 buffer 或增加逻辑级数
Hold 违例（I/O）输出数据在外部器件采样后变化在 IOB 中插入输出寄存器（IOB = 1）或增加 output delay
全是违例（约束错误）时钟约束错误、async path 未标记检查 create_clock/create_generated_clock 是否正确定义

多周期路径约束（set_multicycle_path）：有些路径天生就不需要在一个时钟周期内完成。比如一个计数器溢出后驱动的使能信号——你可以允许它在两个周期内到达目的触发器：

# 使能信号可以在 2 个时钟周期后到达
set_multicycle_path -setup 2 -from [get_pins cntr/overflow_reg] -to [get_pins en_reg]
set_multicycle_path -hold 1 -from [get_pins cntr/overflow_reg] -to [get_pins en_reg]

假路径约束（set_false_path）：对于那些你确定不需要时序分析的路径——比如异步复位引脚、测试模式信号、静态配置信号——使用 set_false_path 可以节省工具的分析时间并减少违例报告中的噪音：

# 所有经过 CDC 同步器的路径不需要时序分析
set_false_path -from [get_clocks clk_a] -to [get_clocks clk_b]

# 复位信号
set_false_path -to [get_pins * -filter {NAME =~ */*RST*}]
时序 vs 功能：一个常见的陷阱是：当你发现一个时序违例很难修复时，不要轻易使用 set_false_path 或 set_multicycle_path 来"骗"工具让它出过一个干净的报告。每条被你标记为 false 或 multicycle 的路径都是你在手工向工具承诺"这条路径在功能上没问题"——而你作为人类，犯错的概率远大于工具。每次添加这些约束时，在代码注释中写清楚为什么这条路径不需要一个周期的约束，并由第二位工程师 review 确认。

14.5 完整的 XDC 约束文件示例

下面是项目中一个典型的 XDC（Xilinx Design Constraints）文件——覆盖了时钟、I/O、异步路径和特殊约束：

# ============================================================
# clock.tcl — 时序约束文件
# Project: data_acquisition_system
# Device: XC7A35T-CPG236-1
# ============================================================

# --- 主时钟 ---
create_clock -name sys_clk -period 10.000 [get_ports clk_100m]
create_clock -name adc_clk -period 20.000 [get_ports adc_clk_p]

# --- PLL 生成时钟 ---
create_generated_clock -name clk_200m \
    -source [get_pins clk_gen/mmcm/CLKIN1] \
    -multiply_by 2 -divide_by 1 \
    [get_pins clk_gen/mmcm/CLKOUT0]

create_generated_clock -name clk_50m \
    -source [get_pins clk_gen/mmcm/CLKIN1] \
    -multiply_by 1 -divide_by 2 \
    [get_pins clk_gen/mmcm/CLKOUT1]

# --- 异步时钟域 ---
set_clock_groups -asynchronous \
    -group [get_clocks sys_clk] \
    -group [get_clocks adc_clk]

# --- ADC 输入约束（16 位并行 ADC） ---
# ADC datasheet: tSU=2ns, tHD=1.5ns (relative to adc_clk rising edge)
set_input_delay -clock adc_clk -max 5.0 [get_ports {adc_data[*]}]
set_input_delay -clock adc_clk -min 1.5 [get_ports {adc_data[*]}]

# --- DAC 输出约束（12 位并行 DAC） ---
# DAC datasheet: tSU=3ns, tHD=1ns (relative to sys_clk rising edge)
set_output_delay -clock sys_clk -max 6.0 [get_ports {dac_data[*]}]
set_output_delay -clock sys_clk -min 1.0 [get_ports {dac_data[*]}]

# --- 复位信号（组合路径被忽略） ---
set_false_path -to [get_pins *RST*]

# --- SPI 时钟是生成的，且是慢速 ---
create_generated_clock -name spi_sck \
    -source [get_pins spi_ctrl/inst/sck_reg] \
    -divide_by 100 [get_ports spi_sck]
set_multicycle_path -setup 20 -from [get_clocks sys_clk] -to [get_clocks spi_sck]

# --- 跨时钟域路径（异步 FIFO 内的，不分析） ---
set_false_path -from [get_clocks clk_50m] -to [get_clocks sys_clk]

14.6 踩坑：约束使用了错误的时钟周期

有一个我参与的项目，设计在实验室环境中完美通过了所有测试——100MHz 时钟、功能正确、温度循环也没出问题。但客户部署到现场后，大概每工作 2-3 天就会出现一次数据错误。更诡异的是——现场的温度、湿度、电源电压都在规格范围内，完全不应该出现问题。

我们把设计从客户现场取回来，重新做了完整的时序分析。发现 setup slack 在 WCS（Worst-Case Slow）corner 下只有 +12ps——虽然为正，但零余量接近极限。但 WCS 是芯片在 85°C、0.95V 供电、慢工艺角的最坏情况——实验室的 25°C、1.0V 供电条件下，实际可用 slack 大于 500ps，所以测不出问题。

但根源不在这里——让我们继续追踪。客户现场的 FPGA 使用的是 100MHz 振荡器，规格为 ±50ppm 精度。我们的约束文件中写的是 create_clock -period 10.000——即精确的 10ns 周期。但实际振荡器的周期可能是 10.000 ± 0.0005 ns。在 85°C 高温时，电源电压可能会有 ±5% 的波动——这些微小的偏差在实验室里完全不影响功能，但在 WCS corner 下、slack 只剩下 12ps 时，额外的 0.5ps 周期偏差就决定了设计能否稳定工作。

修复方案不是什么复杂的工程——在约束文件中将目标频率从 100MHz（10.0ns）改为 83MHz（12.0ns），留出足够的余量。但排查过程告诉我们：约束中的时钟周期应当包含余量——不要写 10.0ns，写 9.5ns（相当于目标频率 105MHz），这样即使晶振频率和温度电压有偏差，设计也能容错。余量留多少取决于你的应用场景：消费级产品 5-10%，工业级 15-20%，航空航天级 30-50%。

时序约束是 FPGA 设计中从"业余"到"专业"的分界线。正确理解和编写约束，意味着你不再仅仅是"能跑出比特流"，而是真正理解了你设计的时序行为。掌握了本章的内容后，你已经具备了 FPGA 开发中"RTL 设计"和"验证"两大支柱的知识。从下一章开始，我们将进入第三部分——实战工具链：Vivado、Quartus Prime、ModelSim 等工具的使用方法和工程最佳实践。真正的工业级 FPGA 开发流程将从那里开始。
