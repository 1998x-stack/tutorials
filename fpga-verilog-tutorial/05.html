第5章 组合逻辑设计实战

大家好，我是你们的讲师。前两章我们学了 Verilog 的语法，现在是时候真正写电路了。本章聚焦于组合逻辑设计——也就是那些"输入一变输出跟着变"的纯逻辑电路：MUX、译码器、编码器、ALU。

组合逻辑是 FPGA 设计的呼吸——它无处不在。即使是纯时序设计（比如一个计数器），也需要组合逻辑来决定"下一个状态是什么"、"使能信号怎么产生"。可以说，组合逻辑写得好不好，直接决定了你的设计能跑多快、占多大面积。

5.1 多路选择器的多种写法

MUX 是 FPGA 设计中使用频率最高的组合逻辑结构。同一功能有至少三种写法，综合结果略有差异。

5.1.1 assign + 条件运算符

// 2:1 MUX——最简洁
assign y = sel ? b : a;

// 4:1 MUX——嵌套条件运算符
assign y = (sel == 2'b00) ? a :
           (sel == 2'b01) ? b :
           (sel == 2'b10) ? c : d;

嵌套的条件运算符综合为级联 MUX 树——延迟与嵌套深度成正比。

5.1.2 always + case

// 4:1 MUX——case 风格，综合为并行 MUX
always @(*) begin
    case (sel)
        2'b00:   y = a;
        2'b01:   y = b;
        2'b10:   y = c;
        2'b11:   y = d;
        default: y = 'bx;
    endcase
end

case 风格综合出的 MUX 往往延迟比级联条件运算符更优，因为综合器可以优化为平衡的选择树。

5.1.3 FPGA 中 MUX 的物理实现

在 FPGA 架构层面，MUX 的实现非常有特色：

  • 2:1 MUX = 1 个 3-LUT（2 个数据输入 + 1 个选择输入）
  • 4:1 MUX = 2 个 LUT + 1 个 2:1 MUX（利用 FPGA Slice 内部的 F7AMUX/F7BMUX）
  • 8:1 MUX = 4 个 LUT + F7MUX + F8MUX（Xilinx 7 系列专用结构）

Xilinx 7 系列 Slice 中有一个技巧——MUXF7 和 MUXF8 原语，它们专门用于构建大扇入的 MUX。当一个设计中有大量 MUX 时，使用这些原语可以显著降低 LUT 消耗。Vivado 综合器的 -mux_extract 选项可以自动推断。

实战建议：对于小规模 MUX（2:1 ~ 8:1），任何风格都能综合出高效的电路。但对于大规模 MUX（16:1 以上），建议显式声明为 case 结构，并确保综合器的 mux_extract 和 mux_style 选项设置为自动优化（Vivado 默认即为 auto）。

5.2 译码器与编码器

5.2.1 3-8 译码器

// 3-to-8 decoder
module decoder_38 (
    input  wire [2:0] addr,
    output reg  [7:0] sel
);
    always @(*) begin
        case (addr)
            3'b000: sel = 8'b00000001;
            3'b001: sel = 8'b00000010;
            3'b010: sel = 8'b00000100;
            3'b011: sel = 8'b00001000;
            3'b100: sel = 8'b00010000;
            3'b101: sel = 8'b00100000;
            3'b110: sel = 8'b01000000;
            3'b111: sel = 8'b10000000;
            default: sel = 8'b00000001;
        endcase
    end
endmodule

综合后每个输出 bit 由 1 个 3-LUT 实现，一共 8 个 LUT。也可以写为 assign sel = 1 << addr;——这种写法仅需 1 个 3-LUT + 布线（左移是纯布线），综合后更紧凑。

5.2.2 优先编码器

优先编码器在中断控制器中很常用——多个输入同时有效时，输出优先级最高的信号对应的编码：

// 4-to-2 priority encoder（优先级：in[3] > in[2] > in[1] > in[0]）
module priority_encoder_42 (
    input  wire [3:0] in,
    output reg  [1:0] code,
    output reg        valid
);
    always @(*) begin
        casez (in)
            4'b1???: {code, valid} = {2'b11, 1'b1};
            4'b01??: {code, valid} = {2'b10, 1'b1};
            4'b001?: {code, valid} = {2'b01, 1'b1};
            4'b0001: {code, valid} = {2'b00, 1'b1};
            default: {code, valid} = {2'b00, 1'b0};
        endcase
    end
endmodule

casez 在这里非常适合，因为它可以直接表达"不关心"位。综合器会为优先级最高的 bit 生成最短的路径。

5.2.3 BCD-七段码译码器

这是 FPGA 入门课的标准练习——将一个 4-bit BCD 码转换为七段数码管的驱动信号：

// BCD to 7-segment decoder (common anode, active low)
module bcd7seg (
    input  wire [3:0] bcd,
    output reg  [6:0] seg  // {a,b,c,d,e,f,g}
);
    always @(*) begin
        case (bcd)
            4'h0: seg = 7'b1000000;
            4'h1: seg = 7'b1111001;
            4'h2: seg = 7'b0100100;
            4'h3: seg = 7'b0110000;
            4'h4: seg = 7'b0011001;
            4'h5: seg = 7'b0010010;
            4'h6: seg = 7'b0000010;
            4'h7: seg = 7'b1111000;
            4'h8: seg = 7'b0000000;
            4'h9: seg = 7'b0010000;
            default: seg = 7'b1111111;  // 全灭
        endcase
    end
endmodule

综合后消耗 7 个 4-LUT（每个输出段对应一个 LUT 的 SOP 最小项）。如果想进一步化简，可以用 Vivado 的 resource_sharing 选项。

5.3 算术运算单元（ALU）设计

5.3.1 加减法的硬件实现

FPGA 中的加法不是用 LUT 做二极管的——每个 Slice 都有专用的进位链（Carry Chain）。Xilinx 7 系列的 CARRY4 原语就是一个专用的快速进位链单元，它可以在约 0.1ns/bit 的延迟上完成加法。所以：加减法在 FPGA 上是非常快的操作。

减法是加法器加一个补码：a - b = a + (~b) + 1。综合器会自动将减法映射到同一个 CARRY4 链上。

5.3.2 乘法的综合结果

乘法就不同了。乘法会映射到 DSP slice（如 Xilinx 7 系列的 DSP48E1）——而不是 LUT。一个 25×18 的有符号乘法只需要 1 个 DSP48E1。但如果位宽超出 DSP 的原生宽度，综合器会用多个 DSP 拼接，或回退到 LUT + CARRY4 的分布式乘法——后者资源消耗会急剧增加。

5.3.4 完整 ALU：8 种操作

// 8 功能 ALU
module alu_8op #(
    parameter WIDTH = 8
) (
    input  wire [WIDTH-1:0] a, b,
    input  wire [2:0]       op,    // 3-bit 操作码
    output reg  [WIDTH-1:0] result,
    output reg              zero,  // 结果为 0
    output reg              carry  // 进位/借位
);
    wire [WIDTH:0] sum  = a + b;
    wire [WIDTH:0] diff = a - b;

    always @(*) begin
        case (op)
            3'b000: {carry, result} = {1'b0, a & b};          // AND
            3'b001: {carry, result} = {1'b0, a | b};          // OR
            3'b010: {carry, result} = {1'b0, a ^ b};          // XOR
            3'b011: {carry, result} = sum;                    // ADD
            3'b100: {carry, result} = diff;                   // SUB
            3'b101: {carry, result} = {1'b0, a << b[2:0]};   // SLL
            3'b110: {carry, result} = {1'b0, a >> b[2:0]};   // SRL
            3'b111: {carry, result} = {1'b0, ~(a | b)};      // NOR
            default: {carry, result} = {2{1'b0}};
        endcase
        zero = (result == {WIDTH{1'b0}});
    end
endmodule

这个 ALU 在综合后会生成：一个 WIDTH-bit 加法器（含进位链）、一个 WIDTH-bit 减法器（复用同一进位链的变体）、各按位运算逻辑、一个桶形移位器（仅当 WIDTH > 4 时）以及一个 8:1 的输出 MUX。所有运算并行执行，结果由 MUX 选择——这是组合逻辑设计的基本模式。

资源共享优化：注意我们用了 wire [WIDTH:0] sum = a + b; 这样加法器只例化一份，所有需要加法结果的操作（SUB = 加法器 + 补码）共享同一个进位链。如果每个操作各自写一行 result = ...，综合器也能做资源共享，但显式声明中间变量可以给综合器更明确的约束。Vivado 的 -resource_sharing 选项控制这一行为，建议设为 auto。

5.4 组合逻辑的竞争与冒险

5.4.1 毛刺的产生机制

组合逻辑的每个门都有传播延迟。当多个输入同时变化时，由于路径延迟不同，输出在稳定之前可能出现短暂的错误脉冲——这就是毛刺（Glitch）。

一个经典例子：y = a & b，当 a=1 保持不变，b 从 1 变为 0。如果布线延迟导致 b 的反相器输出到达 AND 门晚了几个皮秒，输出可能出现一个短暂的"1"脉冲。这就是静态冒险。

5.4.2 什么时候需要关心毛刺

关键问题：毛刺会出问题吗？答案是取决于毛刺的目的地：

  • 毛刺进入触发器 D 输入——只要在时钟沿采样时刻 D 是稳定的，毛刺就无害。这是最常见的情况。
  • 毛刺进入时钟、复位、使能引脚——危险！可能导致 FF 误触发或异步复位。
  • 毛刺进入计数值（组合逻辑驱动计数器 load）——危险！可能导致多计一次。
  • 毛刺进入异步逻辑（跨时钟域握手信号）——可能导致亚稳态。

5.4.3 用寄存器输出消除毛刺

消除毛刺最可靠的方法：在组合逻辑的输出端加一级寄存器（FF）。FF 只在时钟沿采样，毛刺只要不在采样窗口内出现，就会被过滤掉。这就是"组合逻辑 + 一级寄存器"设计模式的原理——也是为什么 FPGA 的每个 Slice 都包含 LUT + FF 对。

// 在输出端加寄存器消除毛刺
module glitch_free_adder (
    input  wire        clk,
    input  wire [7:0]  a, b,
    output reg  [7:0]  sum_sync  // 寄存器输出，无毛刺
);
    wire [7:0] sum_comb;

    assign sum_comb = a + b;    // 组合加法（可能有毛刺）

    always @(posedge clk) begin
        sum_sync <= sum_comb;   // 寄存器采样，滤除毛刺
    end
endmodule

当然，代价是多了一级时钟周期的延迟（流水线效果）。这个权衡——延迟 vs 无毛刺——是数字设计中最基本的取舍之一。

5.5 一个真实的踩坑故事：加法器毛刺让计数器多计数

2019 年我做一个高速数据采集项目，ADC 以 100 MSPS 的速率输出 12-bit 数据。我们需要对连续的 16 个采样值求和，然后再除以 16 取平均（滑动平均滤波器）。核心逻辑大致如下：

// 简化的代码结构——有问题的版本
wire [15:0] sum = sample_window[0] + sample_window[1] + ... + sample_window[15];
// 16 个数的加法树

always @(posedge clk) begin
    if (sum > threshold) begin
        over_count <= over_count + 1;  // 超阈值计数
    end
end

在 FPGA 板上运行时，over_count 的数值比预期高出约 2%。一开始以为是 ADC 噪声，但用信号发生器输入已知幅度的正弦波后，误差仍然存在。用 ILA 抓取 sum > threshold 的比较器输出——发现一个时钟周期内这个信号多次跳变。

原因分析：16 个数相加的加法树有四级加法器级联。输入数据同时变化时，每级加法器的进位链传播延迟不一致，导致 sum 在稳定之前有"毛刺"——瞬间超过阈值，触发计数器递增。虽然计数器的读操作在时钟沿采样，但 if (sum > threshold) 是一个组合比较器，它的结果变化比 sum 更快——毛刺直接驱动了计数器的使能逻辑。

修复方案是在比较器之前加一级寄存器：

// 修复：加一级寄存器
reg [15:0] sum_reg;

always @(posedge clk) begin
    sum_reg <= sum;  // 把 sum 寄存一拍
    if (sum_reg > threshold) begin
        over_count <= over_count + 1;
    end
end

仅仅多了这一行寄存器，毛刺消灭了，计数结果准确了。这个经验告诉我的道理很简单：不要用组合逻辑的输出直接驱动计数器的使能或复位信号——加一级寄存器，成本极低，收益极大。
