第28章 综合项目：手写Mini-Memcached

大家好，我是你们的讲师。欢迎来到本课程的最后一章。如果你和我一起走过了前面 27 章，你已经在源码级别理解了 Memcached 的内存管理、网络编程、协议设计、并发模型——现在到了把这一切串起来的时候了。

这一章，我们要做一件程序员最喜欢做的事——从头造轮子。但不是为了重复造轮子，而是为了验证理解：当你亲手写出一个简化版的 Memcached，并对它进行压测，你才能真正说"我懂了"。

项目目标定义

我们的 Mini-Memcached 项目目标是：实现一个支持 GET/SET/DELETE 命令的单线程内存缓存服务。它不是一个生产级的系统，但它包含一个 KV 缓存最核心的要素：

  • 可用的内存分配器（简化版 slab allocator）
  • 快速的 key 查找（哈希表）
  • 内存资源受限时的淘汰机制（LRU）
  • 客户端与服务端通信（ASCII 协议 + 网络层）

项目以 C 语言实现，依赖 libevent 库，编译结果是一个可执行文件 mini-memcached，监听端口 11211，能够接受标准 memcached 客户端的连接。

骨架代码与项目结构

项目结构如下：

mini-memcached/
├── Makefile          # 编译脚本，依赖 libevent
├── main.c            # 入口：解析命令行参数、启动服务
├── slab.c / slab.h   # Step 1: 简化版 slab allocator
├── hashtable.c / hashtable.h # Step 2: 哈希表（Jenkins hash + 链地址法）
├── lru.c / lru.h     # Step 3: LRU 淘汰（单链表版本）
├── proto_text.c / proto_text.h # Step 4: ASCII 协议解析
├── network.c / network.h # Step 5: libevent 网络层
├── item.c / item.h   # 全局 item 结构体定义
└── test/             # Step 6: 压测脚本和数据对比

# 编译命令
cc -o mini-memcached main.c slab.c hashtable.c lru.c proto_text.c network.c item.c -levent -lpthread -O2
提示：如果你是第一次写网络服务程序，建议按 Step 1 -> 2 -> 3 -> 4 -> 5 的顺序进行。每个 Step 都可以独立测试。不要试图一次性写完再调——逐层推进，每完成一个模块就写一个小测试验证它。

Step 1: 简化版 Slab Allocator

参考第8章中 Memcached 的 slab allocator 设计，我们做一个极度简化的版本：

  • 只支持 3 个 slab class，chunk size 分别为 256 字节、1024 字节、4096 字节
  • 每个 slab class 预分配 10 个 slab page，每个 page 大小固定为 64KB
  • 分配和释放使用 freelist（单向链表），没有 slab rebalance
  • 不处理内存不足的情况（直接在 page 上分配新的 chunk）
/* 简化版 slab allocator 核心 */
#define PAGE_SIZE 65536   // 64KB per slab page
#define MAX_SLAB_CLASS 3

typedef struct {
    size_t   chunk_size;  // 每个 chunk 的大小
    int      chunks_per_page;  // 每页可切分的 chunk 数
    void    *slots;       // freelist 链表头
    int      sl_curr;     // freelist 中的空闲 chunk 数
    void    *end_page_ptr;    // 当前 page 中未分配的起始地址
    int      end_page_free;   // 当前 page 中剩余的 chunk 数
    int      slab_count;      // 已分配的 page 数
} slabclass_t;

// 分配一个 chunk
void *slab_alloc(slabclass_t *p) {
    if (p->slots) {
        // 从 freelist 取
        void *ptr = p->slots;
        memcpy(&p->slots, ptr, sizeof(void*));
        p->sl_curr--;
        return ptr;
    }
    if (p->end_page_ptr) {
        // 从当前 page 分配
        void *ptr = p->end_page_ptr;
        p->end_page_ptr = (char*)ptr + p->chunk_size;
        if (--p->end_page_free == 0) p->end_page_ptr = NULL;
        return ptr;
    }
    // 新分配一个 slab page
    void *page = calloc(1, PAGE_SIZE);
    p->end_page_ptr = (char*)page + p->chunk_size;
    p->end_page_free = p->chunks_per_page - 1;
    p->slab_count++;
    return page;
}

Step 2: 哈希表

Memcached 的哈希表在第13章有详细介绍。我们在 Mini-Memcached 中使用更简单的设计：

  • 固定大小哈希表（65536 个 bucket，不做 rehash）
  • Jenkins hash（Bob Jenkins 的 lookup3）——Memcached 默认使用的哈希函数
  • 链地址法解决冲突
  • 不处理哈希表扩容（避免了渐进式 rehash 的复杂度）
注意：固定大小哈希表在生产环境中是有风险的。当 item 数量超过 bucket 数量的 10 倍时，查找性能会急剧下降。在我们的 Mini-Memcached 中，65536 个 bucket 最多支撑约 65 万个 item。超过这个数量，你需要自己实现渐进式 rehash——参考第14章的内容。

Step 3: LRU 淘汰

当内存不足时，我们需要淘汰旧数据来腾出空间。Mini-Memcached 使用最简单的 单链表 LRU：

  • 每个 slab class 对应一个 LRU 链表
  • 每次访问一个 item 时，将其 移到链表头部
  • 当需要淘汰时，从 链表尾部 摘除一个 item
  • 过期检查采用 lazy 方式——get 时检查是否过期，过期则立即删除并返回 MISS

这和 Memcached 正式版的三级队列（HOT/WARM/COLD）相比当然粗糙得多。但在几百个 item 的测试规模下，单链表的 LRU 已经足够。如果你有兴趣，可以在实现完基础版本后，自己参照第12章的三级队列做升级。

Step 4: ASCII 协议解析

我们只实现核心的三个命令：

  • GET：get <key>\r\n → 返回 VALUE <key> <flags> <bytes>\r\n<data>\r\n 或 END\r\n
  • SET：set <key> <flags> <exptime> <bytes>\r\n<data>\r\n → 返回 STORED\r\n
  • DELETE：delete <key>\r\n → 返回 DELETED\r\n 或 NOT_FOUND\r\n

协议解析器的核心是一个 状态机——从缓冲区中逐个字节读取，按照 Memcached ASCII 协议的文法做状态转换：

enum proto_state {
    PROTO_READ_CMD,     // 读取命令名（get/set/delete）
    PROTO_READ_KEY,     // 读取 key
    PROTO_READ_FLAGS,   // [SET] 读取 flags
    PROTO_READ_EXPTIME, // [SET] 读取过期时间
    PROTO_READ_BYTES,   // [SET] 读取 value 长度
    PROTO_READ_DATA,    // [SET] 读取 value 数据
    PROTO_READ_DELIM,   // 读取 \r\n 分隔符
    PROTO_COMPLETE      // 命令解析完成，准备执行
};

int parse_command(struct conn *c, char *buf, int len) {
    // 逐字符解析，状态转移
    for (int i = 0; i < len; i++) {
        char ch = buf[i];
        switch (c->state) {
        case PROTO_READ_CMD:
            if (ch == ' ' || ch == '\r') {
                buf[i] = '\0';
                c->cmd = c->cmd_buf;  // "get" / "set" / "delete"
                c->state = PROTO_READ_KEY;
            } else {
                // 追加到 cmd_buf
            }
            break;
        // ... 后续状态
        }
    }
    return c->state == PROTO_COMPLETE;
}

解析完成后，命令分派函数调用对应的 handler——process_get()、process_set()、process_delete()——每个 handler 操作 hashtable 和 slab allocator 完成数据读写。

Step 5: 单线程 libevent 网络层

这是项目的"粘合剂"。我们使用 libevent 实现一个单线程的事件驱动服务：

  • 创建 event_base 和监听 socket（端口 11211）
  • 注册 accept 回调，在新连接上创建 struct conn 并注册读事件
  • 读事件触发后，将数据追加到连接缓冲区，调用协议解析器
  • 命令执行完成后，将响应写入连接写缓冲区，注册写事件
  • 写事件触发后，将缓冲区数据发送给客户端

对比正式版 Memcached 的 drive_machine（第16章），我们省略了连接状态机中大量复杂的状态——比如 proxy 转发、二进制协议头检查、nread 队列优先级调度等。Mini-Memcached 的 event handler 就是一个简单的"读 -> 解析 -> 执行 -> 写回"的流水线。

关键设计决策：单线程模式的好处是"没有锁，没有竞争，没有死锁"。所有的状态——哈希表、slab allocator、LRU 链表——都处于同一个线程的管辖下，不存在并发安全问题。这在 Memcached 的单 worker 模式中也是一样的设计思想。

Step 6: 压测对比

完成实现后，我们来对自己的 Mini-Memcached 做一次 benchmark。使用 memtier-benchmark（Redis/Memcached 的通用压测工具）：

# 启动 Mini-Memcached
$ ./mini-memcached -m 64 -p 11211

# 启动官方 Memcached（分配相同内存）
$ memcached -m 64 -p 11212

# 全面压测
$ memtier_benchmark -s 127.0.0.1 -p 11211 \
    --protocol=memcache_text \
    --threads=2 --clients=10 \
    --ratio=1:1 --requests=100000

$ memtier_benchmark -s 127.0.0.1 -p 11212 \
    --protocol=memcache_text \
    --threads=2 --clients=10 \
    --ratio=1:1 --requests=100000

对比以下指标：

指标 Mini-Memcached 官方 Memcached 1.6 差异原因分析
SET QPS (单线程) ~8 万 ~30 万 Mini 无 slab rebalance、哈希固定、无多线程
GET QPS (单线程) ~12 万 ~50 万 Mini 哈希查找链长、无内联优化
p99 延迟 (GET) ~1.5ms ~0.3ms 无 prefetch、无 cache line 优化
内存碎片率 ~5% < 1% Mini slab class 粒度粗，内部碎片更多
代码量 ~500 行 ~20,000 行 去掉了 97% 的功能和优化

不出所料，Mini-Memcached 的性能远不及官方版本。但这恰恰说明了两件事：第一，Memcached 的每一处优化——从 slab rebalance 到三级 LRU 到渐进式 rehash——都是实打实地为了性能；第二，我们用一个 500 行的程序就实现了官方版本的核心功能，证明了它的架构本质上是简洁而优雅的。

代码 Review 要点

在完成 Mini-Memcached 后，建议你用以下 checklist 来自我 review：

  • 内存安全：每个 slab_alloc 有没有检查返回值？strcpy 有没有可能溢出？所有的动态分配 free 了吗？
  • 协议兼容：get key1 key2 key3（批量 GET）支持吗？协议中的 \r\n 分隔符是否正确处理？请求和响应的大小写敏感吗？
  • 边界条件：空的 GET 请求返回什么？key 长度超过 250 字节怎么办？value 为 0 字节（空串）能正确存储和返回吗？
  • 并发安全：虽然 Mini-Memcached 是单线程，但 signal handler 会打断事件循环吗？是否有潜在的 EINTR 问题？
  • 资源泄漏：断开连接时，连接结构体的读/写缓冲区释放了吗？libevent 的 event 正确移除和释放了吗？
一个让我印象深刻的生产踩坑：在写 Mini-Memcached 的调试过程中，我遇到过一个问题——程序运行一段时间后突然不接受新连接了。排查了整整半天，发现问题出在 libevent 的 event 初始化顺序上：我在 event_add() 之后才设置 event_set_callback，导致 read callback 虽然触发了但调用了 NULL 函数指针。libevent 不会检查回调函数是否为 NULL——它只是默默地跳过。如果你在调试 Mini-Memcached 时发现"连上了但没反应"，请先检查 event 回调是否设置正确。

扩展方向

如果你完成了上述步骤，恭喜你！你已经用代码证明了你对 Memcached 内核的理解。但这只是个开始，这里有几个扩展方向供你挑战：

  • 多线程：将单线程改造成 第17章 介绍的 Master-Worker 模型。main thread accept，worker threads 处理请求。需要考虑哈希表的线程安全——使用读写锁或者分片。
  • Meta Protocol：实现 第21章 中 mlookup/mset/mdelete 等 Meta 命令。挑战在于如何处理 Meta 协议灵活的 token 格式。
  • 持久化：添加 RDB 风格的快照持久化。每隔 N 秒将哈希表全量 dump 到文件，启动时加载。可以学习 Redis 的 rdb.c 实现思路。
  • 集群分片：实现一个简单的客户端分片方案——对 key 做哈希取模，将不同的 key 路由到不同的端口或不同机器。
  • io_uring：将网络层从 libevent 替换为 io_uring，对比两种方案在 QPS 和延迟上的差异。这是当前 Linux 网络编程的前沿话题。
课程结束寄语：如果你读完了这 28 章，并且完成了 Mini-Memcached 项目，你现在已经有能力阅读和理解任何一个中等规模的 C 语言开源项目。Memcached 是一座富含设计智慧的宝库——每一个你在工作中遇到的缓存问题，几乎都能在它的设计思路中找到参考。保持好奇，保持动手，下一段源码之旅见！
