第11章 存储器设计与Block RAM

在 FPGA 中实现任何一个非平凡的设计，都离不开存储器。你至少需要一个 FIFO 做跨时钟域数据缓冲、一个 lookup table 存正弦波系数、一个 RAM 做视频帧缓存、或者一个 register file 实现处理器的通用寄存器组。到目前为止，我们在前几章的代码中主要用了触发器和查找表（LUT）来存储状态——但这对于需要存储大量数据的场景来说，不仅浪费逻辑资源，还会把设计的时序性能拖垮。

FPGA 提供了两种内置的存储资源：分布式 RAM（Distributed RAM）和Block RAM（BRAM）。理解它们的差异、优缺点和正确的使用方式，是 FPGA 设计从"能跑"到"优秀"的关键一步。

11.1 FPGA 内部存储结构

分布式 RAM（Distributed RAM）其实是用 LUT 的 SRAM 单元来实现存储器。LUT 本质上是一个 6 输入的单端口小存储器（1 个 6 输入 LUT 可以存储 64 bit）。当综合工具检测到你的 Verilog 中声明了一个 reg [7:0] mem [0:15] 这样的小容量数组时，它会自动用 LUT 来实现。分布式 RAM 的优点是没有额外的延迟（读操作与 LUT 的传播延迟相同）、粒度小（可以精确配置为任意深度和宽度）。缺点是容量有限——FPGA 的 LUT 数量虽然多，但每个 LUT 只有几十 bit，大规模存储器会迅速消耗殆尽。

Block RAM（BRAM）是 FPGA 芯片中的专用存储硬核。在 Xilinx 7 系列中，每个 BRAM 是 36Kb 的空间，可以配置为两个独立的 18Kb RAM。在 Intel/Altera 的 Cyclone V 中，M10K 块也是 10Kb。BRAM 的优点是容量大、速度快（读取有独立的输出寄存器，可以达到几百 MHz）、功耗相对更低。缺点是粒度大——你深度不想用满 36Kb，但用了 BRAM 就至少占一个物理块，对于超小容量的存储器来说效率低。

维度分布式 RAMBlock RAM

选型经验：经验法则是——容量 ≤ 64 字 × 8 bit 的存储器，用分布式 RAM；更大的存储器显式地用 BRAM。但有些情况下即使容量小也建议用 BRAM：比如这个存储器所在的数据路径需要跑 300MHz 以上，或者 LUT 资源已经非常紧张（LUT 利用率超过 70% 时布局布线压力很大）。始终在综合后在资源报告里检查你的存储器被映射到了什么上。

11.2 单端口与双端口 RAM

Verilog 中描述 RAM 最直观的方式是用二维寄存器数组。综合工具（Vivado/Quartus/Yosys）会识别这种 pattern 并推断为 BRAM：

单端口 RAM：同一时钟域通过同一端口访问，读写共用一个地址总线。不适合同时读写。

简单双端口 RAM（Simple Dual-Port, SDP）：一个端口只写、另一个只读。这是最常用的配置——写端口处理输入数据流，读端口处理输出数据流。

真双端口 RAM（True Dual-Port, TDP）：两个端口都可以独立读写。适用于两个处理器核共享数据、或者一个端口被主机访问另一个被用户逻辑访问的场景。

// 推断为真双端口 Block RAM (Read-Before-Write)
module dpram #(
    parameter DATA_WIDTH = 8,
    parameter ADDR_WIDTH = 10  // 1024 深度
)(
    input  wire                    clka,
    input  wire                    clkb,
    input  wire                    wea,    // port A 写使能
    input  wire [ADDR_WIDTH-1:0]  addra,
    input  wire [DATA_WIDTH-1:0]  dina,
    output reg  [DATA_WIDTH-1:0]  douta,
    input  wire                    web,    // port B 写使能
    input  wire [ADDR_WIDTH-1:0]  addrb,
    input  wire [DATA_WIDTH-1:0]  dinb,
    output reg  [DATA_WIDTH-1:0]  doutb
);

    reg [DATA_WIDTH-1:0] ram [0:(1<
读写冲突（Read-During-Write）：当两个端口同时操作同一个地址时，BRAM 的行为取决于器件和配置。Xilinx BRAM 的默认行为是"写优先"（write-first/write-through）——写入的数据同时出现在该时钟周期的读输出上。另一种模式是"读优先"（read-first）——读输出返回旧值，新值从下一个周期开始可见。如果你的设计依赖特定冲突行为（比如双端口用作共享缓存），必须在 IP 核配置中显式指定，或者在 RTL 中通过综合属性声明。不明确的做法会让你在芯片跑到特定工作条件时发现数据不连贯。

11.3 ROM 实现

ROM（只读存储器）在 FPGA 中有两种实现方式：

方法一：case 语句综合为 ROM。综合工具会将一个大的 case 语句（输入是地址，输出是数据）优化为 LUT ROM 或者 BRAM（取决于大小）。优点是完全可移植（不依赖任何厂商原语），适合中等大小的查找表。

方法二：用 $readmemh 初始化 BRAM。这是最推荐的工业做法。将 ROM 数据保存在一个外部文本文件（.mem 或 .hex）中，通过 $readmemh 在仿真和综合时加载到 BRAM 中：

// 推断为 Block ROM（用 $readmemh 初始化）
module sin_lut #(
    parameter ADDR_WIDTH = 10,
    parameter DATA_WIDTH = 16
)(
    input  wire                    clk,
    input  wire [ADDR_WIDTH-1:0]  addr,
    output reg  [DATA_WIDTH-1:0]  dout
);

    reg [DATA_WIDTH-1:0] rom [0:(1<

sin_lut.mem 文件的格式很简单——每行一个十六进制数，对应一个地址的数据：

0000
00C9
0192
025B
...
FFFF
IP 核 vs RTL 推断：很多初学者—上来就用 IP 核生成器（Vivado IP Catalog / Quartus Platform Designer）产生 BRAM 或 ROM。这当然可以工作，但我建议你优先尝试在 RTL 中"描述"它，让综合工具去推断。原因有三：(1) RTL 代码可移植（可以在 Vivado、Quartus、Yosys 之间无缝切换）；(2) 代码可读性强，团队 review 更容易；(3) 综合工具对标准 RTL pattern 的 BRAM 推断已经非常成熟（99% 的情况都正确）。只有在需要用到 BRAM 的特殊功能（ECC、字节使能、输出寄存器级数控制）时才应该使用 IP 核。

11.4 FIFO 设计

FIFO（First-In-First-Out）是 FPGA 设计中使用频率最高的存储器结构——它协调两个速率不同的数据流，或者跨越不同的时钟域。

同步 FIFO（同一时钟域）：读写指针在同一时钟域，空满标志的生成比较简单——当读写指针相等时，要么是满（写指针追上了读指针一圈），要么是空（两者指在同一个元素上）。区分的方法是使用额外的一位指针宽度来标识是否绕圈。

// 参数化同步 FIFO
module sync_fifo #(
    parameter DATA_WIDTH = 8,
    parameter ADDR_WIDTH = 8  // 256 深度
)(
    input  wire                clk,
    input  wire                rst_n,
    input  wire                wr_en,
    input  wire [DATA_WIDTH-1:0] wr_data,
    output reg                 full,
    input  wire                rd_en,
    output reg  [DATA_WIDTH-1:0] rd_data,
    output reg                 empty,
    output reg  [ADDR_WIDTH:0] fifo_count  // 用于水位线判断
);

    reg [DATA_WIDTH-1:0] mem [0:(1<

异步 FIFO（跨时钟域）：读写时钟不同，需要解决两个关键问题：(1) 指针在两个时钟域之间传递时避免亚稳态（使用格雷码和两级同步器）；(2) 空满标志的正确生成（写指针同步到读时钟域判断空，读指针同步到写时钟域判断满）。异步 FIFO 是跨时钟域设计中最常见的解决方案，我们将在第 13 章深入展开。

11.5 Block RAM 的原语例化 vs 推断

当你的需求超出了标准 RTL pattern 的范围，就需要直接例化厂商原语。以 Xilinx 7 系列为例，BRAM 的原语是 RAMB36E1（36Kb）和 RAMB18E1（18Kb）。它们支持的特性包括：

  • 字节使能（Byte-Write Enable）：按字节粒度控制写入，对以太网帧处理等场景至关重要
  • 输出寄存器级数（Output Register）：可选择 1 级或 2 级寄存器输出，影响 peformance vs latency 的权衡
  • ECC（Error Correction Code）：支持单纠错双检错，用于太空/辐射环境
  • 级联（Cascade）：将多个 BRAM 拼接为更大的存储器

在 Intel/Altera 器件中，对应的原语是 altsyncram（通过 Quartus MegaWizard 生成），参数化方式与 Xilinx 不同但功能等效。

关于"寄存器推断"的建议：在对厂商工具链不够熟悉时，写 RTL 让工具推断 BRAM 是最安全的方式。根据 Amd 的白皮书（UG901），Vivado 支持超过 20 种 BRAM 推断 pattern。如果你在 Vivado 综合后的报告中看不到 BRAM（显示 LUTRAM），通常原因有：(1) 复位逻辑不当（BRAM 不支援异步复位读操作）；(2) 读写使能写成了组合逻辑而非时序逻辑；(3) 读取地址在复位时被赋予了非零值。

11.6 踩坑：双端口 RAM 读-写碰撞导致视频帧缓存花屏

我曾经参与过一个 FPGA 视频处理项目，需要在帧缓存（dual-port BRAM）中同时写入来自 camera 的下一帧数据和读取当前帧送显示。设计很直接——写端口对着 camera 的像素流，读端口对着 HDMI 控制器。仿真完美，读写的地址完全分开（写端口写入 frame N+1，读端口读出 frame N），看起来不会有冲突。

但实际板子上显示出现了周期性的花屏——每隔几十帧，画面上出现横条纹干扰。花了整整两天才定位到问题：当 camera 的像素时钟和 HDMI 的像素时钟有微小的频率偏差时，两个端口的地址指针会慢慢漂移——直到在某个时刻指向了同一个 BRAM 地址。默认的 BRAM 写优先模式下，写端口在写入新像素的同时，读端口读到了正在被覆盖的中间值——半旧半新的像素数据被送到了 HDMI 显示。

修复方案不是改 RTL，而是在 BRAM 配置中将模式从"写优先"改为"读优先"——这样在发生碰撞时读端口始终返回旧数据，写端口写入新数据从下一个周期开始可见。虽然碰撞的那一个像素是旧值，但旧版本帧和新版本帧之间的单像素差异人眼完全无法察觉——花屏立刻消失了。

这个经历告诉我两个道理：一是永远不要假设两个时钟域的地址不会碰撞，哪怕逻辑上看起来永远不会；二是BRAM 的碰撞行为是可以利用的——根据应用场景选择合适的模式。
