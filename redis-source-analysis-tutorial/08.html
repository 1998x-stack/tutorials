第8章：redisObject 与编码转换

大家好，我是你们的讲师。前几章我们逐个拆解了 Redis 的底层数据结构：SDS、链表、压缩列表、listpack、字典、跳表、整数集合。这些数据结构各自有各自的擅长领域——有的节省内存、有的支持快速查找、有的擅长顺序访问。但 Redis 对外暴露的只有五种类型（String、List、Hash、Set、ZSet），底层却用了十多种编码。用户存一个数字"1"和一个长文本，背后可能用了完全不同的内存布局。

这个"多种编码对用户透明"的魔法，就藏在 redisObject 这个看似普通的包装结构里。它是 Redis 对象系统的基石，也是理解命令实现和内存优化的前提。这一章我们就把它彻底拆解明白。

统一对象模型

为什么 Redis 需要一个 redisObject 包装层？想象一下，如果没有这个统一对象，每个命令函数都得用 void* 加 type 标签来手动判断类型，代码会变成一锅粥。antirez 用了一个非常经典的 C 面向对象手法——用一个结构体作为所有类型的"基类"：

/* server.h */
typedef struct redisObject {
    unsigned type:4;       /* 对象类型：OBJ_STRING/OBJ_LIST/OBJ_SET/OBJ_ZSET/OBJ_HASH */
    unsigned encoding:4;   /* 编码方式，见下方 OBJ_ENCODING_* */
    unsigned lru:LRU_BITS; /* LRU 时间戳或 LFU 计数，LRU_BITS = 24 */
    int refcount;          /* 引用计数 */
    void *ptr;             /* 指向底层数据结构的指针 */
} robj;

这个结构体只有 16 字节（在 64 位系统上，4+4+24 位 + 4 字节 int + 8 字节指针，对齐后刚好 16 字节）。它实现了三个核心设计目标：多态（统一操作接口）、内存优化（灵活的编码切换）、资源管理（引用计数共享）。

redisObject 结构体逐字段分析

type —— 5 种类型

type 字段占用 4 位，取值在 server.h 中定义：OBJ_STRING (0)、OBJ_LIST (1)、OBJ_SET (2)、OBJ_ZSET (3)、OBJ_HASH (4)。当你执行 TYPE key 时，Redis 返回的就是这个字段的字符串映射。

encoding —— 10+ 种编码

encoding 同样占 4 位，但它的取值在不断扩充。到 Redis 7.2 为止，有以下编码：

常量名 值 对应数据结构 使用场景
OBJ_ENCODING_RAW 0 SDS 长度超出阈值的 String
OBJ_ENCODING_INT 1 long 整型 可解析为整数的 String
OBJ_ENCODING_HT 2 dict（字典） 大 Hash / 大 Set
OBJ_ENCODING_ZIPLIST 5 ziplist 小 Hash / 小 ZSet（旧）
OBJ_ENCODING_INTSET 6 intset 整数小 Set
OBJ_ENCODING_SKIPLIST 7 zskiplist + dict 大 ZSet
OBJ_ENCODING_EMBSTR 8 嵌入式 SDS 短 String（≤ 44 字节）
OBJ_ENCODING_QUICKLIST 9 quicklist List 始终使用
OBJ_ENCODING_STREAM 10 rax + listpack Stream 始终使用
OBJ_ENCODING_LISTPACK 11 listpack 小 Hash / 小 ZSet（新版）

refcount —— 引用计数

refcount 是一个 4 字节的 int，记录当前对象被引用的次数。当一个对象的 refcount 降为 0 时，Redis 调用 freeObject() 释放其内存。引用计数机制让 Redis 可以在多个地方共享同一个对象，而无需深拷贝。

lru —— 24 位 LRU 时钟

lru 字段在不同的淘汰策略下有不同的含义：在 LRU 模式下，它存储的是服务器时钟的低 24 位（以秒为精度）；在 LFU 模式下，高 16 位存储上次递减时间，低 8 位存储访问频次（对数计数器）。OBJECT IDLETIME 命令就是通过计算当前时间与 lru 字段的差值来估算键的空闲时间的。

设计细节点：LRU 时钟使用 24 位意味着大约 194 天会回绕一次（2^24 秒）。estimateObjectIdleTime() 函数中考虑了回绕情况，通过取差值 (unsigned long) 强制转无符号来正确处理溢出。

编码转换规则

编码转换是 Redis 内存优化的核心策略。每种类型都遵循"小数据用紧凑编码，大数据用高效编码"的原则。

String：int → embstr → raw

当 SET key 100 时，Redis 检查字符串能否被 string2ll() 解析为 long long。如果可以，就用 OBJ_ENCODING_INT 编码，直接将值存储在 ptr 字段中（强制类型转换为 void*），不额外分配内存。当字符串长度不超过 44 字节时，使用 OBJ_ENCODING_EMBSTR——一个一次性分配的连续内存块，同时包含 redisObject 和 sdshdr，减少内存碎片。超过 44 字节则使用 OBJ_ENCODING_RAW，单独分配 SDS。

Hash：listpack → dict（旧版 ziplist → dict）

当 Hash 中字段数量少于 hash-max-listpack-entries（默认 512）且每个字段名称和值的长度都小于 hash-max-listpack-value（默认 64）时，底层使用 listpack。超出任一阈值则自动转换为 dict 编码。

List：始终使用 quicklist

Redis 3.2 之后 List 底层统一使用 quicklist（双向链表 + listpack 节点）。list-max-listpack-size 控制每个节点的 listpack 最大大小，list-compress-depth 控制两端不压缩的节点数。

Set：intset → dict

当 Set 中的所有元素都是整数且数量不超过 set-max-intset-entries（默认 512）时，使用 intset 编码。否则转为 dict 编码，元素作为 dict 的 key，value 统一为 NULL。

ZSet：listpack → skiplist + dict

当 ZSet 元素数量少于 zset-max-listpack-entries（默认 128）且每个元素长度小于 zset-max-listpack-value（默认 64）时，使用 listpack。超过任一阈值则转换为 skiplist + dict 的双重结构。

踩坑实录：有一次线上业务使用 Hash 存储用户画像数据，每个 Hash 约 300 个字段，每个字段值大约 80 字节。看起来 300 < 512（entries 阈值），但 80 > 64（value 阈值），所以编码是 dict 而非 listpack。DBA 尝试调大 hash-max-listpack-value 到 128 后，内存立刻下降了 40%。后来排查发现，ziplist/listpack 对中等长度字符串的存储密度远高于 dict——dict 每个 entry 需要两个 dictEntry（field 和 value 各一个）+ 两套 SDS 头，而 listpack 只需紧凑的连续存储。调参前一定要先 DEBUG OBJECT key 查看当前的编码类型。

引用计数与共享对象

Redis 在启动时预创建了 0 到 9999 的整数字符串对象（shared.integers[10000]），这些对象的 refcount 初始值极高且永远不会被释放。当你执行 SET key1 100 时，createObject() 会发现值可以解析为 100，于是直接返回 shared.integers[100]，refcount 加 1。这意味着所有值为 0-9999 的字符串键，在内存中共享同一份对象。

/* object.c */
robj *createObject(int type, void *ptr) {
    robj *o = zmalloc(sizeof(*o));
    o->type = type;
    o->encoding = OBJ_ENCODING_RAW;
    o->ptr = ptr;
    o->refcount = 1;
    o->lru = 0; /* 将在后续被设置 */
    return o;
}

robj *createStringObject(const char *s, size_t len) {
    if (len <= 44)
        return createEmbeddedStringObject(s, len);
    else
        return createRawStringObject(s, len);
}

/* tryObjectEncoding 中检查整型共享对象 */
if (o->encoding == OBJ_ENCODING_RAW &&
    sdsEncodedObject(o)) {
    /* 如果 sds 可以解析为 long long，尝试转 INT 编码 */
    if (string2ll(s, len, &value)) {
        /* 如果值在 0-9999 之间，直接返回共享对象 */
        if (value >= 0 && value < OBJ_SHARED_INTEGERS) {
            decrRefCount(o);
            return shared.integers[value];
        }
        /* 否则用 INT 编码 */
        o->encoding = OBJ_ENCODING_INT;
        o->ptr = (void*)((long)value);
    }
}

refcount 的增减操作在 incrRefCount() 和 decrRefCount() 中处理。特别地，对于共享对象（refcount 初始值极大），decrRefCount() 会特殊处理——只有 refcount == 1 时才真正释放。

/* object.c */
void decrRefCount(robj *o) {
    if (o->refcount == 1) {
        /* 唯一引用，可以释放 */
        freeObject(o);
    } else {
        if (o->refcount <= 0) return; /* 共享对象，永不释放 */
        o->refcount--;
    }
}

void incrRefCount(robj *o) {
    if (o->refcount < OBJ_SHARED_REFCOUNT) /* 大于 1 的非共享对象 */
        o->refcount++;
}
注意：OBJ_SHARED_REFCOUNT 定义为一个极大值（INT_MAX / 2），共享对象的 refcount 初始设为这个值。因此 incrRefCount 不会对共享对象做自增，decrRefCount 也不会自减——它们永远"活着"。

OBJECT 命令实现

OBJECT 命令是 Redis 源码调试的利器。它让你能直接查看一个键对应 redisObject 的内部状态。其实现集中在 object.c 的 objectCommand() 函数中：

/* object.c */
void objectCommand(client *c) {
    robj *o;

    if (!strcasecmp(c->argv[2]->ptr, "refcount")) {
        /* OBJECT REFCOUNT key */
        o = lookupKeyReadOrReply(c, c->argv[1], shared.null[c->resp]);
        if (o == NULL) return;
        addReplyLongLong(c, o->refcount);

    } else if (!strcasecmp(c->argv[2]->ptr, "encoding")) {
        /* OBJECT ENCODING key */
        o = lookupKeyReadOrReply(c, c->argv[1], shared.null[c->resp]);
        if (o == NULL) return;
        addReplyBulkCString(c, strEncoding(o->encoding));

    } else if (!strcasecmp(c->argv[2]->ptr, "idletime")) {
        /* OBJECT IDLETIME key */
        o = lookupKeyReadOrReply(c, c->argv[1], shared.null[c->resp]);
        if (o == NULL) return;
        addReplyLongLong(c, estimateObjectIdleTime(o) / 1000);

    } else if (!strcasecmp(c->argv[2]->ptr, "freq")) {
        /* OBJECT FREQ key — 仅在 LFU 策略下可用 */
        o = lookupKeyReadOrReply(c, c->argv[1], shared.null[c->resp]);
        if (o == NULL) return;
        addReplyLongLong(c, LFUDecrAndReturn(o));
    }
}

各子命令的作用：

  • OBJECT REFCOUNT —— 查看对象的引用计数，检查是否为共享整数对象。返回共享对象时通常为 INT_MAX 级别的值。
  • OBJECT ENCODING —— 查看底层编码类型，这是判断是否需要调优配置参数的关键依据。
  • OBJECT IDLETIME —— 估算空闲秒数，结合 maxmemory-policy volatile-lru 验证淘汰效果。
  • OBJECT FREQ —— 仅在使用 LFU 淘汰策略时可用，返回对象的访问频率估算值。
调试技巧：DEBUG OBJECT key 是 OBJECT 命令的"增强版"，它直接输出 redisObject 的原始字段值，包括 type、encoding、refcount、lru（原始时钟值）、lru_seconds_idle 等。在分析内存问题时，先用 DEBUG OBJECT 快速确认编码类型，再决定是否需要调整配置参数。

编码转换的触发时机

编码转换的检测并不是每次命令执行都触发——那样太浪费 CPU。Redis 采用按需转换策略：在创建新对象时选择初始编码，在操作过程中如果发现当前编码不再适合，才触发转换。例如 hsetCommand 调用 hashTypeSet() 时，如果发现 listpack 中的元素数量或值长度超过了配置阈值，就调用 hashTypeConvert() 转换为 dict。

/* t_hash.c */
void hashTypeSet(robj *o, sds field, sds value, int flags) {
    int update = 0;

    if (o->encoding == OBJ_ENCODING_LISTPACK) {
        update = hashTypeSetListpack(o, field, value, flags);
        /* 检查是否需要转换 */
        if (hashTypeLength(o) > server.hash_max_listpack_entries ||
            sdslen(value) > server.hash_max_listpack_value)
        {
            hashTypeConvert(o, OBJ_ENCODING_HT);
        }
    } else if (o->encoding == OBJ_ENCODING_HT) {
        update = hashTypeSetHT(o, field, value, flags);
    }
}

这种"写时转换"的机制避免了在每次读取时做类型判断的开销，同时确保了紧凑编码能持续使用到最后一刻。

踩坑实录：embstr 编码有一个容易被忽略的特性——它是只读的。当你对一个 embstr 字符串执行 APPEND 操作时，Redis 不会在原地址追加（因为 embstr 是一次性分配的连续内存块，没有预留额外空间），而是创建一个新的 raw 编码对象，将原内容复制过去，然后追加。这意味着 APPEND 一个很小的数据到 embstr 字符串上也会触发 malloc + free + memcpy 的完整开销。在循环中执行 APPEND 的代码，性能会比预分配好 SDS 再用 SET 差一个数量级。所以，如果需要频繁拼接字符串，客户端拼好再 SET，远比用 APPEND 一个个追加要高效。
