第13章：数据分片与分布式训练

你可能遇到过这种情况：买了 4 张 GPU 的机器，跑训练脚本时发现显存 OOM 了。排查半天发现是个"低级错误"——4 个进程各自加载了完整的训练数据。每个进程以为自己是"唯一"的，结果同一份数据被复制了 4 份，4 倍的冗余加载直接撑爆了显存。

分布式训练中的数据管理是 AI 工程里最容易被低估的复杂性来源。模型并行（Model Parallelism）和流水线并行（Pipeline Parallelism）比较复杂，但在数据并行（Data Parallelism）场景下，你唯一需要解决的就是：确保每个进程只看到属于自己的那部分数据。听起来简单，但做错的人比你想象的要多得多。

核心认知：分布式训练的第一原则——每个进程只加载自己负责的那一块数据。全量加载 + 分布式训练 = 灾难。这个公式几乎每周都在某个团队的工单系统中被重新验证一次。

分片基础：shard()

datasets 库提供了最直接的分片方法——shard()。它的名字可能会和 PyTorch 的 Dataset.shard 混淆，但理解起来很简单：

from datasets import load_dataset

dataset = load_dataset("imdb", split="train")
print(f"总样本数: {len(dataset)}")  # 25000

# 将数据集分为 4 片，取第 0 片
shard_0 = dataset.shard(num_shards=4, index=0)
print(f"第0片样本数: {len(shard_0)}")  # 6250

# 取第 1 片
shard_1 = dataset.shard(num_shards=4, index=1)
print(f"第1片样本数: {len(shard_1)}")  # 6250

# 验证分片不重叠
shard_0_indices = set(shard_0["label"])  # 简化验证
shard_1_indices = set(shard_1["label"])

# 在分布式训练中的典型用法
def get_shard(dataset, rank, world_size):
    """根据进程 rank 获取对应分片"""
    return dataset.shard(
        num_shards=world_size,
        index=rank,
        contiguous=False,  # 不连续分片（更均匀的分布）
    )

# contiguous=True vs False
# True:   [0,1,2,...,6249] 给 rank0, [6250,...,12499] 给 rank1
# False:  [0,4,8,...] 给 rank0, [1,5,9,...] 给 rank1（交替分配）

contiguous=False 在大多数场景下更优。因为如果数据集有某种顺序结构（比如按类别排列），连续分片可能导致某些进程分到的数据分布有偏——rank0 全是类别 A，rank1 全是类别 B。

踩坑故事：某团队在进行 4 卡分布式微调时，每个训练脚本都写了 load_dataset("imdb", split="train")，然后直接送进了 PyTorch 的 DataLoader。结果每个进程都加载了完整的 25000 条数据——因为没有任何人告诉数据集"你只需要加载 1/4"。4 个进程各吃了 25000 条数据，总共 100000 条数据被加载到了 GPU 显存里。对于 BERT-base 这样的模型，即使本身参数量不大，但 dataset 的 batch 处理依然会导致显存迅速耗尽。

更隐蔽的问题是：即使显存没爆，每个 epoch 中 4 个进程看到了完全相同的 25000 条数据，每个 batch 的梯度完全相同，分布式训练退化成了 4 倍浪费的"伪单卡训练"。模型没有从分布式训练中得到任何加速或泛化收益。

分布式数据并行（DDP）集成

PyTorch 的 DistributedSampler 是最主流的分布式数据分发方案。它自动为每个进程分配一个不重叠的数据子集，并在每个 epoch 重新洗牌：

import torch
import torch.distributed as dist
from torch.utils.data import DataLoader
from torch.utils.data.distributed import DistributedSampler
from datasets import load_dataset

# 初始化分布式环境
dist.init_process_group(backend="nccl")
rank = dist.get_rank()        # 当前进程编号 (0, 1, 2, 3)
world_size = dist.get_world_size()  # 总进程数

# 方式一：使用 shard 手动分片（推荐）
dataset = load_dataset("imdb", split="train")
my_shard = dataset.shard(num_shards=world_size, index=rank, contiguous=False)

# 将 HF Dataset 转为 PyTorch Dataset
class HFDataset(torch.utils.data.Dataset):
    def __init__(self, hf_dataset):
        self.dataset = hf_dataset

    def __len__(self):
        return len(self.dataset)

    def __getitem__(self, idx):
        example = self.dataset[idx]
        return {
            "input_ids": torch.tensor(example["input_ids"]),
            "attention_mask": torch.tensor(example["attention_mask"]),
            "label": torch.tensor(example["label"]),
        }

pt_dataset = HFDataset(my_shard)

# 使用 DistributedSampler
sampler = DistributedSampler(
    pt_dataset,
    num_replicas=world_size,
    rank=rank,
    shuffle=True,
    seed=42,
)

dataloader = DataLoader(
    pt_dataset,
    batch_size=16,
    sampler=sampler,
)

# 训练循环
for epoch in range(3):
    sampler.set_epoch(epoch)  # 每个 epoch 重新洗牌
    for batch in dataloader:
        # 每个进程只处理自己的 batch
        pass
小技巧：使用 DistributedSampler 时，一定要在每个 epoch 开始前调用 sampler.set_epoch(epoch)。这个操作会基于当前 epoch 的 seed 重新生成随机索引，确保每个 epoch 中每个进程看到的数据顺序不同。忘记这一步是最常见的分布式训练 bug 之一。

跨节点数据分发

单机多卡的场景我们解决了。但当你需要多机（多节点）训练时——比如 4 台机器，每台 8 张 GPU，总共 32 个进程——数据分片的挑战升级了。

架构 数据加载方式 优点 缺点
共享文件系统（NFS） 所有节点访问同一存储 管理简单，数据一致 NFS 可能成为 I/O 瓶颈
对象存储（S3/MinIO） 每个节点从 S3 拉取 扩展性好，云原生 网络带宽敏感
数据本地化 每个节点本地磁盘存一份 读取最快 存储空间浪费，同步困难 
# 多节点场景：每个节点独立加载，用 shard 保证数据不重叠
import os
from datasets import load_dataset

# 假设：4 个节点，每节点 8 GPU
node_rank = int(os.environ.get("NODE_RANK", 0))   # 当前节点编号 0-3
local_world_size = 8                                # 每节点 GPU 数
global_world_size = 32                              # 总 GPU 数

# 方式二：全局 rank 计算
rank = node_rank * local_world_size + local_rank  # local_rank 由 torchrun 提供

# 每个进程只加载自己的一份
dataset = load_dataset("imdb", split="train")
my_shard = dataset.shard(num_shards=global_world_size, index=rank)

# 流式模式下的分布式加载
stream_dataset = load_dataset(
    "c4", "en",
    split="train",
    streaming=True,
)
# 跳转到本进程负责的数据范围
# 每个进程跳过前 rank * per_worker_samples 条
per_worker = 100000  # 每个进程处理的样本量
my_stream = stream_dataset.skip(rank * per_worker).take(per_worker)
性能建议：在分布式训练中，数据加载往往是第一个瓶颈。如果数据存储在远程（如 S3），确保数据本地化——在训练开始前把数据拷贝到每台机器的本地 SSD 上。网络带宽在高吞吐训练中是稀缺资源，不要让数据加载和梯度通信抢带宽。实测显示，从 S3 流式加载 vs 本地 NVMe SSD 加载，速度差距可以达到 3-5 倍。

分片策略设计

分片不仅仅是"把数据切成等份"那么简单。不同的分片策略对训练效果和性能有显著影响。

按样本 shard vs 按文件 shard

# 按样本 shard（推荐）：打乱后再分片，分布均匀
dataset = dataset.shuffle(seed=42)
shard = dataset.shard(num_shards=world_size, index=rank)

# 按文件 shard：适合流式场景，每个 shard 对应不同的原始文件
# 假设数据集有 100 个 Parquet 文件
files_per_worker = 100 // world_size
start_file = rank * files_per_worker
end_file = start_file + files_per_worker

# 只加载分配给自己的文件
my_files = [f"data/train-{i:05d}.parquet" for i in range(start_file, end_file)]
dataset = load_dataset("parquet", data_files=my_files, split="train")

按文件 shard 适合流式大数据场景，因为每个进程只打开自己负责的文件，不需要全局扫描。但缺点是如果文件大小不均匀，分片可能不平衡。

验证分片不重叠永远是一个好习惯：

def verify_no_overlap(dataset, num_shards):
    """验证各分片之间没有重叠"""
    all_indices = []
    for i in range(num_shards):
        shard = dataset.shard(num_shards=num_shards, index=i)
        # 取每个样本的唯一标识（这里用文本的前50字符）
        ids = [ex["text"][:50] for ex in shard]
        all_indices.extend(ids)

    total = len(all_indices)
    unique = len(set(all_indices))
    overlap = total - unique
    print(f"总样本: {total}, 唯一样本: {unique}, 重叠: {overlap}")
    assert overlap == 0, f"发现 {overlap} 条重叠数据!"

PyTorch Lightning / HF Trainer 集成

现代训练框架（PyTorch Lightning 和 Hugging Face Trainer）已经封装了大部分分布式逻辑。你只需要知道正确的使用方式：

# === HF Trainer 集成 ===
from transformers import Trainer, TrainingArguments
from datasets import load_dataset

dataset = load_dataset("imdb", split="train")

training_args = TrainingArguments(
    output_dir="./results",
    per_device_train_batch_size=16,
    num_train_epochs=3,
    ddp_find_unused_parameters=False,  # 加速 DDP
    dataloader_drop_last=True,         # 丢弃 last incomplete batch
)

# Trainer 内部自动处理 DistributedSampler
# 你只需要传入完整的数据集
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=dataset,
)

# 注意：Trainer 会自动调用 dataset.shard()
# 你不需要手动分片！

# === PyTorch Lightning 集成 ===
import pytorch_lightning as pl
from torch.utils.data import DataLoader
from torch.utils.data.distributed import DistributedSampler

class DataModule(pl.LightningDataModule):
    def setup(self, stage=None):
        # prepare_data 只在 rank 0 上执行一次
        # setup 在所有进程上执行
        self.dataset = load_dataset("imdb", split="train")

    def train_dataloader(self):
        sampler = DistributedSampler(
            self.dataset,
            num_replicas=self.trainer.world_size,
            rank=self.trainer.global_rank,
        )
        return DataLoader(
            self.dataset,
            batch_size=16,
            sampler=sampler,
        )

使用高级框架时，最常见的错误配置：

错误 现象 解决方案
忘记设置 ddp_find_unused_parameters=False DDP 训练慢 30% 显式设置该参数
prepare_data 中加载了全部数据 rank 0 耗尽所有内存 只在该方法中下载数据，不要加载到内存
每个进程没有独立的 seed 所有进程看到相同洗牌顺序 使用 pl.seed_everything(seed, workers=True)
batch_size 未按总进程数缩放 学习率与 batch 大小不匹配 total_batch = per_gpu_batch * num_gpus，线性调整 lr
流式模式下使用 len(dataset) 报错或返回无穷大 流式数据集没有 __len__，需改用 take(n)
实践建议：从单卡迁移到多卡训练时，分步验证。先跑 1 卡确认没问题，再跑 2 卡，再跑 4 卡。每次增加卡数时，检查 loss 曲线的形态——如果多卡的 loss 和单卡的 loss 在相同 step 位置上形态不一致，说明数据分片或学习率缩放出了问题。这个"分步验证"策略可以帮你在 10 分钟内定位到问题，而不是在一整轮训练后才发现。

数据分片解决了"怎么把数据分给多张 GPU"的问题。但还有一个更基础的工程问题没有解决——怎么保证数据不被不小心改坏？当你团队里有 5 个人都在处理同一个数据集，有人改了一列，有人删了一些样本，有人重新标注了一批数据——几周后谁也说不清训练用的数据到底是哪个版本。下一章的数据版本管理正是为了解决这个问题而生的。
