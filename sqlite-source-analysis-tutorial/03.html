第3章 源码整体架构

大家好，我是你们的讲师。经过前两章的"热身"——历史概览和编译体系——现在我们终于要正式走进 SQLite 的核心源码了。这一章我们站在高处，把 SQLite 的整个架构"拍一张卫星照片"，这样后续每深入一个模块时，你都知道它在大图中的位置。

优雅的分层架构

SQLite 最让我佩服的设计特点之一就是它的 分层架构。整个数据库引擎从 SQL 语句输入到数据落盘，中间经过恰好 10 层管道，每层各司其职，层与层之间有清晰的接口边界。

我们从顶层到底层来认识这 10 层：

┌──────────────────────────────────────────────┐
│                用户 SQL 语句                    │
├──────────────────────────────────────────────┤
│ 1. Tokenizer   词法分析器 (tokenize.c)        │
│    ├ 将 "SELECT * FROM t" 拆成 token 流       │
│ 2. Parser      Lemon解析器 (parse.y)          │
│    ├ 将 token 流构建为语法分析树 (Parse Tree)  │
│ 3. Code Gen   代码生成器 (expr.c, where.c)    │
│    ├ 将语法树编译为 VDBE 字节码               │
│ 4. VDBE        虚拟机执行引擎 (vdbe.c)        │
│    ├ 逐条执行字节码指令 (Op)                  │
│ 5. B-Tree      存储引擎 (btree.c)             │
│    ├ 管理数据页的增删改查 (表 + 索引)          │
│ 6. Pager       页缓存与事务 (pager.c)         │
│    ├ 管理脏页刷新、事务提交与回滚              │
│ 7. VFS        虚拟文件系统 (os_unix.c等)      │
│    ├ 抽象 read/write/sync 系统调用            │
│ 8. OS          操作系统层                     │
├──────────────────────────────────────────────┤
│              磁盘上的数据库文件                 │
└──────────────────────────────────────────────┘

这套架构最核心的设计哲学是 "每层只知下层，不越级调用"。VDBE 不知道 SQL 语法是什么，它只管执行 Op 指令；B-Tree 不关心事务的具体细节，它只负责把数据页塞给 Pager。这种严格的接口隔离，使得每一层都可以独立测试、独立优化、独立替换。

举个例子：如果你想替换 SQLite 的存储引擎（比如用 LevelDB 替代 B-Tree），理论上你只需要把 B-Tree 层替换掉，上面的 VDBE 和 Parser 完全不用动。虽然实际上很少有人这么做，但这种"可插拔"的设计思想让 SQLite 的架构非常干净。

理解分层架构是读懂 SQLite 源码的钥匙。 当你看到一个函数调用链时，如果能在脑中映射出"这是从第几层到第几层的调用"，你就能快速判断它是核心逻辑还是边缘路径。这种"层感"需要刻意练习，但一旦建立，阅读效率会有质的飞跃。

src/ 目录文件映射

第 1 章我们简单浏览了目录结构，现在我们来深入 src/ 目录，了解每个核心文件的具体职责。把这些文件按照前面提到的"10 层管线"来归类，就是下面这张表：

分层文件代码行数（约）核心职责
Tokenizertokenize.c~8KSQL 关键字与标识符的词法分析
Parserparse.y~10KLemon 语法文件，定义 SQL 语法规则
Code Genexpr.c~65K表达式编译——WHERE、SET 等子句的字节码生成
Code Genwhere.c~100K查询规划器——生成最优数据访问路径
Code Geninsert.c~30KINSERT/UPDATE/DELETE 的字节码生成
Code Genselect.c~45KSELECT 语句的字节码生成
VDBEvdbe.c~230K字节码虚拟机——整个库最大的文件
VDBEvdbeapi.c~25KVDBE 对外 API——sqlite3_step 等
VDBEvdbeaux.c~30KVDBE 辅助——程序准备、绑定参数
B-Treebtree.c~215KB-Tree 存储引擎——表与索引的核心
Pagerpager.c~70K页缓存、事务提交/回滚、WAL 模式
Pagerwal.c~55KWAL 写前日志的独立实现
VFSos_unix.c~40KUnix/Linux 平台 VFS 实现
VFSos_win.c~45KWindows 平台 VFS 实现
共享sqliteInt.h~12K内部数据结构与宏定义——一切的基础
共享malloc.c~20K内存分配器——Lookaside/Scratch pool

从行数上你能看出一些有趣的事情：vdbe.c 是当之无愧的"巨无霸"，230K 行里塞满了各种 opcode 的实现逻辑；where.c 的 100K 行则反映了查询规划这个问题的复杂度；而 tokenize.c 只有 8K 行——词法分析器在所有模块里是最简单的。

💡 阅读顺序建议： 先读短文件（tokenize.c → select.c → insert.c），再读长文件（expr.c → where.c → btree.c），最后啃最难的（pager.c → vdbe.c）。这和"先易后难"的学习节奏一致，也能让你更快地获得正反馈。

头文件依赖网络

SQLite 的头文件组织非常精妙。它没有把所有东西塞进一个"超级头文件"（尽管 amalgamation 最终会合并所有 .c，但 .h 仍保持分离），而是分成了三个层级的头文件体系。

第一层：sqlite3.h（公共 API 头文件）

这是 SQLite 的对外接口。所有你作为 SQLite 使用者能调用的函数——sqlite3_open()、sqlite3_prepare_v2()、sqlite3_step()、sqlite3_finalize()——都在这。它不包含任何内部实现细节，只暴露了稳定的 API 签名。这个文件是 sqlite.h.in 模板经过 configure 替换版本号之后生成的。

第二层：sqliteInt.h（内部实现头文件）

这是 SQLite 源码阅读的 重中之重。几乎所有核心数据结构的定义都在这里：sqlite3（数据库连接）、Table（表元数据）、Index（索引元数据）、Select（SELECT 语句）、Expr（表达式树节点）、Parse（编译上下文）等等。它还包括了大量辅助宏、工具函数声明、全局变量声明。如果你要改 SQLite 的行为，八成要从修改 sqliteInt.h 开始。

第三层：模块内部头文件

每个核心模块有自己的内部头文件，该模块私有的数据结构定义在其中：

┌────────────────────────────────────────────┐
│               sqlite3.h                     │
│            (公共API, 对外接口)               │
├────────────────────────────────────────────┤
│               sqliteInt.h                   │
│   (核心数据结构: sqlite3, Table, Expr...)    │
├─────────────┬───────────────┬───────────────┤
│  vdbeInt.h  │  btreeInt.h   │  pagerInt.h   │
│  (Vdbe, Op) │ (BtShared,    │ (Pager,       │
│  (Mem,      │  MemPage,     │  PgHdr)       │
│  VdbeCursor)│  BtCursor)    │               │
├─────────────┴───────────────┴───────────────┤
│               os.h / mutex.h / pcache.h     │
│           (VFS抽象层, 互斥锁, 页缓存)         │
└─────────────────────────────────────────────┘
  • vdbeInt.h — VDBE 内部结构：Vdbe（虚拟机）、Op（字节码指令）、Mem（内存值）、VdbeCursor（游标）。
  • btreeInt.h — B-Tree 内部结构：BtShared（共享缓存状态）、MemPage（内存页）、BtCursor（B-Tree 游标）。
  • pager.h — Pager 结构：Pager（页面管理器）、PgHdr（页头部元数据）。

生成文件的角色

SQLite 源码树中有几个文件不是手写的，而是由代码生成器（code generator）自动产生的。理解这些生成文件，有助于你判断"哪些代码可以直接读，哪些代码需要去看生成脚本"。

生成文件源文件/脚本生成方式作用
parse.c / parse.hparse.y + tool/lemon.cLemon LALR(1) 解析器生成SQL 语法解析表
opcodes.h / opcodes.cvdbe.c + tool/mkopcodeh.tclTCL 脚本提取 opcode 定义VDBE 指令集枚举
keywordhash.htool/mkkeywordhash.cC 程序生成完美哈希表SQL 关键字的快速查找
sqlite3.hsqlite.h.in + configureAutoconf 替换版本宏公共 API 头文件
sqlite3.ctool/mksqlite3c.tclTCL 脚本合并分散源码Amalgamation 合并版

最有意思的是 opcodes.h 的生成方式。它不是手写的，而是通过 tool/mkopcodeh.tcl 从 vdbe.c 的源码注释中提取出来的。在 vdbe.c 中，每个 opcode 的实现前面都有一段注释，格式类似于：

/* Opcode: Column P1 P2 P3 P4 P5 *//* Synopsis: ...
**
** 从第 P1 行读取列 P2 的值。
** 将结果写入栈顶寄存器。
*/
case OP_Column: {
    // 实现代码...
    break;
}

生成脚本解析这些注释，自动提取 opcode 名称、参数数量、操作数含义，然后生成 opcodes.h 中的枚举定义和 opcodes.c 中的名称字符串表。这种"代码即文档、文档即代码"的做法，保证了 opcode 定义和实现之间永远同步。

一个小发现： keywordhash.h 是用 C 程序生成代码的典型案例。mkkeywordhash.c 的原理是分析当前 SQLite 支持的所有关键词，计算一个完美哈希函数（perfect hash function）——使得每个关键词映射到哈希表的唯一位置，没有冲突。这意味着 SQLite 的关键字查找是 O(1) 的。如果你添加了一个新的 SQL 关键字，需要重新运行这个程序生成新的哈希表。

如何入手阅读源码

有了架构全景图，现在我们来看一个实际的 SQL 执行流，把这些概念串起来。假设你执行了这样一条 SQL：

SELECT name FROM users WHERE age > 18;

这条语句在 SQLite 内部经历了以下旅程：

  1. sqlite3_prepare_v2() — 编译 SQL。内部调用 sqlite3ParserParse()（Lemon 生成的解析器），生成语法树。然后调用 sqlite3Select() → sqlite3WhereBegin() 等代码生成函数，把语法树翻译成 VDBE 字节码。
  2. sqlite3_step() — 逐行执行。VDBE 虚拟机从程序计数器（pc）指向的第一条 opcode 开始，逐条解释执行。遇到 OP_Column 从行中提取列值，遇到 OP_Ge 做比较判断。
  3. VDBE 调用 B-Tree — 当需要读取表数据时，VDBE 通过 sqlite3BtreeCursor() 打开游标，通过 sqlite3BtreeNext() 逐行遍历。
  4. B-Tree 调用 Pager — 当需要读取磁盘页时，B-Tree 通过 sqlite3PagerGet() 请求 Pager 返回指定页号的数据。
  5. Pager 调用 VFS — 如果页不在缓存中，Pager 通过 VFS 层的 xRead() 方法发起磁盘 I/O 请求。

你看，一条简单的 SQL 从顶层的 SQL 语句到底层的磁盘 I/O，恰好穿过了我们之前说的所有 8 层（实际涉及了 Tokenizer → Parser → Code Gen → VDBE → B-Tree → Pager → VFS → OS）。

对于刚刚开始阅读源码的同学，我建议你从以下四个关键入口函数开始追踪：

函数所在文件作用适合理解
sqlite3_prepare_v2()prepare.cSQL 编译入口前端编译流程
sqlite3_step()vdbeapi.c逐行执行入口VDBE 执行模型
sqlite3BtreeOpen()btree.cB-Tree 引擎初始化存储引擎启动
sqlite3PagerOpen()pager.cPager 初始化页缓存与事务系统
💡 实战建议： 打开你本地的源码，用 gdb 或 lldb 在上述四个函数上设置断点，然后执行一条简单的 SQL。观察调用栈（bt/backtrace 命令），亲眼验证我们说的这个"10层管线"是怎么跨文件调用的。一次"断点追踪"比读十遍架构图都有效。

好了，第 3 章我们完成了 SQLite 源码的"地形勘察"。从 10 层管线到 src 目录的文件映射，从头文件的依赖网络到生成文件的角色，再到实际的 SQL 执行流追踪——你现在应该对整个代码库有了一个立体的、有层次感的认识。
