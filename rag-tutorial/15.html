第十五章：幻觉检测与事实核查

大家好，我是你们的讲师。今天我们要谈论一个让所有RAG开发者夜里睡不踏实的话题——幻觉。我知道你们心里在想什么："RAG不是已经解决了幻觉问题吗？" 事实上，RAG大幅减少了模型凭空编造的概率，但它并没有、也永远无法彻底消除幻觉。根据2024年Anthropic的一份内部研究报告，即使是经过RAG增强的模型，在面对领域专业度极高或文档质量较差的知识库时，依然有大约15%-25%的生成内容包含不同程度的事实偏离。

想象一下这个场景：你是一家医疗科技公司的算法工程师，你们的RAG系统为医生提供临床决策支持。系统检索了最新的医学指南，并且引用了具体段落。但大模型在生成回答时，不小心将"禁忌症"列表中的一个药物剂量"合并"到了另一个药物的用法说明中。这个看似微小的"串位"错误，可能导致严重的临床后果。这就是幻觉检测和事实核查如此重要的原因——它不是锦上添花，而是RAG系统走向生产环境的最后一道安全闸门。

5.1 RAG为何仍然会产生幻觉？——五大失败模式

要检测和消除幻觉，首先要理解RAG系统中幻觉的根源。我将它们归纳为五种典型模式：

1. 上下文冲突（Context Conflict）： 检索到的文档之间本身就存在矛盾信息。比如三份文档分别说"API v2已在2025年弃用"、"API v2仍受支持到2026年底"、"API v2正在逐步淘汰中"。模型被迫在这些矛盾的上下文之间"选边站"，选错了就会产生幻觉。

2. 上下文不完整（Incomplete Context）： 检索返回的内容缺乏关键限定条件。例如返回了"该药物在临床试验中有效率为78%"，但忽略了原文紧跟着的"但仅限于EGFR突变阳性患者"这个关键主语限定。

3. 过度外推（Over-extrapolation）： 模型在检索到的信息基础上"合理"地补充了细节，但这些细节并不存在于原文中。这是最隐蔽的幻觉形式，因为补充的内容在逻辑上看似合理。

4. 事实漂移（Factual Drift）： 模型在生成过程中，从正确的起点开始，但随着生成token增多，逐渐偏离了原始事实。就像一个故事经过多次转述后变得面目全非。

5. 知识边界混淆（Knowledge Boundary Confusion）： 模型自身的预训练知识与检索到的外部知识发生冲突。如果模型"坚信"自己知道某件事的答案，即使检索到的信息是正确的，它也可能倾向于用自己的知识替代。

幻觉类型 表现 检测难度 常见场景
上下文冲突 回答与检索文档部分矛盾 中 多文档聚合回答
上下文不完整 缺失限定条件导致误导 高 长文档摘要
过度外推 补充了文档中没有的细节 极高 推理型问答
事实漂移 从正确逐渐偏离到错误 中 多轮对话、长文本生成
知识边界混淆 预训练知识干扰检索知识 高 模型擅长的领域

5.2 NLI一致性检查——幻觉检测的主力方法

自然语言推理（NLI）是目前工业界最主流的幻觉检测手段。核心思想非常简单：将"检索到的上下文"作为前提（Premise），将"模型生成的回答"作为假设（Hypothesis），用NLI模型判断两者之间的逻辑关系——是蕴含（Entailment）、矛盾（Contradiction）还是中立（Neutral）。如果判断结果为矛盾或中立，则说明回答可能包含了幻觉。

下面是一个使用HuggingFace上的NLI模型进行事实一致性检查的示例：

from transformers import pipeline
import json

# 加载NLI模型（可以使用开源模型如BART-large-MNLI）
nli_pipeline = pipeline(
    "text-classification",
    model="facebook/bart-large-mnli",
    device=0  # 使用GPU
)

def check_factual_consistency(context, claim):
    """
    检查一个陈述(claim)是否被上下文(context)所支持。
    返回蕴含(entailment)、矛盾(contradiction)或中立(neutral)。
    """
    result = nli_pipeline(
        f"{context} <sep> {claim}",
        return_all_scores=True
    )
    label_map = {
        "ENTAILMENT": "蕴含（支持）",
        "NEUTRAL": "中立（无足够证据）",
        "CONTRADICTION": "矛盾（不符事实）"
    }
    scores = {r["label"]: r["score"] for r in result[0]}
    predicted = max(scores, key=scores.get)
    return {
        "claim": claim,
        "judgment": label_map[predicted],
        "confidence": scores[predicted],
        "all_scores": scores
    }

# 使用示例
context = "辉瑞-BioNTech新冠疫苗在预防有症状感染方面显示出95%的有效率，" \
          "但该数据基于临床试验，实际效果可能因变异株和人群而异。"

claims = [
    "辉瑞疫苗对预防有症状感染的有效率是95%。",
    "辉瑞疫苗对所有变异株的防护效果都完全相同。",
    "该疫苗的临床试验是在2023年进行的。"
]

for claim in claims:
    result = check_factual_consistency(context, claim)
    print(f"陈述: {result['claim']}")
    print(f"判定: {result['judgment']} (置信度: {result['confidence']:.2%})")
    print("-" * 60)
实践建议： NLI模型的阈值设置非常关键。建议将蕴含（Entailment）的置信度阈值设在0.7以上，低于该阈值的回答应触发人工复核。不要使用单一阈值一刀切——对高风险领域（医疗、金融），建议将阈值提高到0.85以上。

5.3 声明提取与分步验证管线

将整个回答作为一个整体来判断是否一致，粒度太粗了。实际生产系统中，我们需要对回答进行"原子化"处理——将回答拆解为独立的声明（Claim），然后逐一验证每个声明是否被上下文支持。这个管线通常包含三个步骤：

第一步——声明提取： 使用一个专门的提取模型或prompt，将模型的回答分解为可验证的原子声明。例如回答"GPT-4有1.8万亿参数，训练于2023年完成"应被分解为"GPT-4有1.8万亿参数"和"GPT-4训练于2023年完成"两个声明。

第二步——证据检索： 对每个声明，从知识库中检索最相关的证据段落。注意，这里的检索可能和原始RAG检索不同——我们需要的是与声明最直接相关的证据，而不仅仅是回答原始问题所需的上下文。

第三步——逐声明验证： 使用NLI模型或LLM裁判，对每个声明做出支持/矛盾/无证据的判断。最终汇总所有声明的验证结果，给出整体的"事实一致性分数"。如果任何一个声明被判定为矛盾，整个回答就应被标记为可疑。

5.4 SelfCheckGPT——无参考的幻觉检测

有时候我们根本没有"标准上下文"可用——比如在纯生成场景中。SelfCheckGPT是一种创新的无参考幻觉检测方法，核心理念是：如果模型对某个事实是"真正知道"的，那么多次采样生成的结果应该保持一致；如果某段内容属于幻觉，不同采样之间的描述往往会互相矛盾。

具体做法是：对同一个prompt进行N次采样（通常N=5~10），然后将每次采样中的句子与其他采样中的对应部分进行一致性比较。常用的比较方法包括：基于BERTScore的语义相似度、基于NLI的矛盾检测、或者是基于QA模型的一致性验证（从一句生成问题，用另一句回答，看答案是否匹配）。

SelfCheckGPT的好处是不需要外部知识库作为参考，完全依赖于模型自身的一致性。缺点是对采样次数敏感，且计算开销较大——每次查询需要生成N次，并做O(N²)次比较。

5.5 DeepEval框架的幻觉检测实践

DeepEval是一个开源的LLM评估框架，内置了幻觉检测能力。它将前面介绍的NLI验证和声明提取整合为开箱即用的接口：

from deepeval import evaluate
from deepeval.metrics import HallucinationMetric
from deepeval.test_case import LLMTestCase

# 定义测试用例
test_case = LLMTestCase(
    input="阿司匹林的推荐剂量是多少？",
    actual_output=(
        "对于成人的发热和疼痛，阿司匹林的推荐剂量是每次325mg，"
        "每4-6小时一次。对于儿童，推荐剂量为每公斤体重10mg。"
    ),
    context=[
        "成人阿司匹林推荐剂量：每次325-650mg，每4-6小时一次，"
        "每日不超过4000mg。",
        "儿童使用阿司匹林需谨慎，推荐剂量为每公斤体重5-10mg，"
        "每4-6小时一次，但需在医生指导下使用。"
    ]
)

# 使用幻觉检测指标
hallucination_metric = HallucinationMetric(threshold=0.5)
result = evaluate(
    [test_case],
    metrics=[hallucination_metric]
)

print(f"是否通过: {result.test_results[0].is_passed}")
print(f"幻觉分数: {result.test_results[0].metrics_data[0].score}")
print(f"失败原因: {result.test_results[0].metrics_data[0].error}")
# 输出中会标明哪些具体声明未被上下文支持

⚠ 重要警示：过度推理导致的反噬： 在实际项目中我们曾经遇到一个教训——对某金融咨询RAG系统的幻觉检测设置得过于激进，导致系统频繁拒绝回答（false positive率超过30%）。用户满意度急剧下降，甚至比系统偶尔给出错误回答时更糟糕。请记住：幻觉检测的终极目标是平衡"不犯错"和"能回答问题"之间的关系。如果你的系统因为过度检测而变得几乎不可用，那说明阈值设置出了问题。务必在你的评估指标中加入"拒绝率"或"可用率"作为辅助指标。

5.7 小结与下章预告

幻觉检测是一个典型的"写容易、做好极难"的工程问题。我们介绍了NLI一致性检查、声明分解验证和SelfCheckGPT等核心方法，以及DeepEval等实用工具。在实际生产中，建议采用分层策略：先用轻量级NLI做快速过滤，对置信度不足的回答再启动声明分解做深度验证。仅对高风险场景启用SelfCheckGPT或多采样验证。

检测出幻觉之后怎么办？除了直接拒绝回答，我们还希望在保持系统可用性的同时过滤掉不安全的内容。下一章，我们将进入RAG系统的安全领域——内容安全护栏。你会学到如何在输入、检索、输出三个环节构建多层防御，以及如何满足2026年欧盟AI法案的合规要求。
