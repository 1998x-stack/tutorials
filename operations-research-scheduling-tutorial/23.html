第23章 强化学习调度

大家好，我是你们的讲师。在之前十几章里，我们学习了精确求解方法、元启发式算法、约束规划——这些方法各有所长，但它们共享一个前提：问题模型是静态的，输入给定之后算法跑一遍，输出一个方案，完事。但现实世界不是这样的——机器会突然故障，新订单会随时插单，原材料可能延迟到货。当你面对一个不断变化的环境，你需要一种能边做边学、动态调整的方法。

这正是强化学习（Reinforcement Learning, RL）登场的时刻。RL 不是"另一种优化算法"，它提供的是完全不同的范式：智能体（Agent）与环境（Environment）交互，通过试错学习最优决策策略。2016年 AlphaGo 击败李世石就是 RL 的高光时刻——它没有背谱，没有预存棋路，纯粹靠自我对弈学到了超越人类的策略。在调度领域，RL 正在复制类似的故事。

为什么RL适合调度 — 序贯决策的本质

我们回想一下调度问题的本质：有一批工件要加工，有若干台机器可用，你需要决定谁先做、在哪做、何时做。这天然是一个序贯决策（Sequential Decision Making）问题——你不需要一次性决定全年的排程，而是需要在每个决策时刻，根据当前车间状态做出下一步选择。

传统方法把调度建模为一次性优化问题：输入所有工件的全部信息，求解器一口气输出完整的甘特图。这在理论上是优雅的，但在实践中面临三个致命问题：

  • 计算延迟——对于 NP-hard 问题，求解器可能需要数小时才能找到高质量解，而车间等不了那么久。
  • 动态扰动——当机器故障或紧急插单发生时，已算好的甘特图瞬间作废，需要重新求解。
  • 重复模式——工厂每天面临的调度问题高度相似，但传统方法每次从零求解，不会"积累经验"。

RL 恰好能解决这些问题。它把调度看作一个序贯决策过程：在每个决策时刻，智能体观察当前状态（机器忙闲、工件队列、截止日期等），然后选择一个动作（比如"把工件 A 放到机器 2 上加工"），环境反馈一个奖励并转移到下一个状态。智能体的目标是在一个完整的 episode（比如一班生产）里最大化累计奖励。

这个框架天然支持动态环境——机器故障了，状态变了，智能体根据新状态重新决策就好，不需要从头求解。而且智能体可以在大量相似实例上训练，学到的策略可以快速泛化到新实例上，这正是传统方法做不到的。

核心认知：传统优化方法追求的是"给定实例下的最优解"；RL 追求的是"给定分布下的最优策略"。前者是instance-level optimization，后者是policy-level optimization。在工业调度中，后者往往更实用——因为工厂不会只做一个批次的生产。

调度问题的MDP建模

要用 RL 解决调度问题，第一步是把调度问题形式化为马尔可夫决策过程（Markov Decision Process, MDP）。MDP 由五元组 (S, A, P, R, γ) 定义，其中最关键的是状态、动作和奖励的设计。下面我们逐一展开。

状态设计：机器状态、工件状态、全局特征的拼接

在调度问题中，状态表示智能体在某一时刻"看到的"全部信息。设计好的状态是 RL 调度成功的基石——状态信息不够，智能体学不到好的策略；状态过于复杂，训练又变得困难。

常见的状态设计包含三个部分：

  • 机器状态——每台机器的当前状态（空闲、正在加工、故障）、剩余加工时间、累计负载等。这是高维连续/离散信息的混合。
  • 工件状态——每个待加工工件的属性（剩余工序数、各工序加工时间、截至日期、已等待时间等）。对于大批量问题，工件数量可能成百上千，需要合理的聚合或采样策略。
  • 全局特征——车间的整体统计数据，如所有工件的平均剩余加工时间、队列长度分布、当前完工进度百分比等。

一个典型的状态向量可能看起来像这样：

# 一个 Job Shop 调度问题的状态表示（示意）
state = {
    "machines": [
        {"status": "busy", "remaining_time": 12.5, "load": 85.0},
        {"status": "idle", "remaining_time": 0.0, "load": 42.0},
        ...  # 共 m 台机器
    ],
    "jobs": [
        {"id": 0, "next_op": 2, "proc_time": 8.0, "due": 120.0, "wait": 5.3},
        {"id": 1, "next_op": 0, "proc_time": 15.0, "due": 90.0, "wait": 12.1},
        ...  # 当前所有待加工的工件
    ],
    "global": {
        "makespan_so_far": 63.0,
        "remaining_ops": 47,
        "avg_machine_load": 0.72,
        "num_late_jobs": 3
    }
}

这里有一个实操中容易踩的坑：状态维度高度可变。在 10 个工件的实例上训练好的网络，输入维度定死了，遇到 100 个工件的实例怎么办？常用的解法包括：固定最大工件数并用 padding 填充、使用图神经网络处理变长输入（下章详述）、或者使用注意力机制自动加权聚合。

动作空间：选择下一个加工工件、或选择派工规则

动作空间的设计有两种主流范式：

  • 直接选择工件（Job Selection）——智能体直接从当前等待队列中选择一个工件安排到空闲机器上。动作空间的大小等于等待工件的数量。这种方式的优点是决策精细，缺点是当工件数量大时动作空间也大，且动作数动态变化。
  • 选择派工规则（Rule Selection）——智能体不是直接选工件，而是从一组预定义的派工规则中选择一条（如 SPT、EDD、MWKR、FIFO 等），然后由规则自动决定下一个加工哪个工件。动作空间固定（等于规则库的大小），但决策粒度较粗。

两种方式各有千秋，目前的主流趋势是先选规则再具体指派的两阶段架构，既保持了决策的可解释性，又控制了动作空间的规模。

奖励函数设计

奖励函数是 RL 调度中最微妙的部分。设计不当的奖励函数会让智能体学到完全错误的策略。以下是一些常见的设计模式：

  • -1 per step——每走一步给一个固定的负奖励，鼓励智能体尽快完成任务。这种设计简单有效，但只有稀疏的全局信号，学习效率较低。
  • 延迟惩罚——每当一个工件超过截止日期，给予一个与延迟时间成正比的负奖励。这能让智能体更敏感于截止日期约束。
  • 完工时间差——每一步的奖励 = 前一步的最大完工时间 - 当前步的最大完工时间（即 C_max 的减少量）。这样智能体每一步都能感受到进展，学习效率更高。
  • 混合奖励——将多个目标综合考虑，如 r = -ΔC_max - α * Σmax(0, C_j - d_j)，其中 α 控制延迟惩罚的权重。

踩坑经验：我曾经做一个 Flow Shop RL 调度项目，用了"每步 -1"的稀疏奖励，训练了 200 万步后智能体还是一团浆糊。后来改成基于完工时间差的稠密奖励，训练 20 万步就收敛了。教训是：奖励密度比奖励大小更重要——让智能体每一步都获得反馈，比给它一个巨大的最终奖励要有效得多。

深度强化学习调度方法

有了 MDP 形式化之后，我们需要用具体的 RL 算法来求解。以下三种是调度领域最常用的深度 RL 方法。

DQN：用神经网络逼近Q值

Deep Q-Network（DQN）是 DeepMind 在 2013 年提出的经典方法，首次将深度神经网络与 Q-learning 结合。在调度场景中，DQN 学习一个 Q 函数 Q(s, a)，表示在状态 s 下执行动作 a 后能得到的期望累计奖励。算法维护两个网络（online network 和 target network），使用经验回放（experience replay）来稳定训练。

DQN 适合离散动作空间的调度问题，如上面的"选择派工规则"范式。它已经在单机调度、Job Shop 调度等场景中取得了不错的效果。但它无法处理连续动作空间，且在大动作空间下性能会显著下降。

PPO：连续动作空间与稳定训练

Proximal Policy Optimization（PPO）是 OpenAI 在 2017 年提出的策略梯度方法，也是目前工业界最主流的 RL 算法。PPO 的核心思想是：每次更新策略时不要改变太多，通过一个 clip 机制限制新旧策略的差异，从而避免训练崩溃。

在调度中，PPO 可以输出连续的动作值（比如"优先级分数"），然后根据分数排序来实际决定加工顺序。这本质上是一种learning to rank的方法。PPO 的稳定性使其非常适合调度这种 long-horizon 问题——训练过程不会因为一步更新过大而把策略彻底毁掉。

Actor-Critic for 调度

Actor-Critic 方法同时学习一个策略（Actor）和一个值函数（Critic）。Actor 负责决定做什么，Critic 负责评估做得怎么样。这种架构天然适合调度问题：Actor 输出排程决策，Critic 评价当前排程方案的优劣。

在调度领域，A2C/A3C 方法已被成功应用于多种调度问题。例如，Park 等人（2021）使用 A3C 训练了一个端到端的 Job Shop 调度器，在 15×15 以下的实例上达到了 OR-Tools CP-SAT 相当的性能，但决策速度快了两个数量级。

RL调度的实际挑战

虽然 RL 调度看起来前景光明，但在实际落地中仍然面临三大硬骨头。

第一个挑战：泛化性。这是一个让 RL 调度研究者夜不能寐的问题。你在 10 台机器 × 10 个工件的小实例上训练了一个智能体，它表现得很好。但当你把它放到 100 台机器 × 100 个工件的实际场景中，性能可能直接崩盘——因为状态空间的分布已经完全变了。目前的研究方向包括领域随机化（domain randomization）和元学习（meta-learning），但这些方法离工业级泛化还有距离。

第二个挑战：动作空间爆炸。在 Job Shop 调度中，等待队列中的工件数可能达到几十甚至上百个。直接选择工件的动作空间规模等于队列长度，这会让 DQN 等方法的 Q 值估计变得极不准确。解决方案包括：使用参数量共享的 dueling architecture、使用 pointer network 输出动作概率、或者退回到"选择规则"的范式来压缩动作空间。

第三个挑战：奖励稀疏。特别是当目标函数是 makespan 时，智能体只有在整个 episode 结束时才知道最终 C_max 是多少。中间步骤的奖励几乎为零，这导致智能体很难将最终结果归因到具体的决策上。前文提到的"完工时间差"奖励可以在一定程度上缓解这个问题，但在大规模问题上，中间奖励信号仍然非常微弱。

论文复现的常见坑：如果你尝试复现某篇 RL 调度论文的结果，第一个要检查的是随机种子。我曾在复现一篇 AAAI 论文时，无论怎么调参都无法复现论文报告的指标。折腾了两周后偶然发现：论文作者没有固定 Gym 环境的随机种子，导致每次跑的结果差异极大；他们报告的可能是 10 次运行中的最好结果。自此之后，我养成了一个习惯——所有 RL 调度实验必须固定至少 5 个不同的随机种子，报告均值和标准差，而不是仅报告最优值。

下面是一个简易的调度环境 Gym 接口定义，展示了 RL 调度环境的基本骨架。你可以基于这个模板实现自己的调度环境：

import gym
from gym import spaces
import numpy as np

class SchedulingEnv(gym.Env):
    """一个简化的 Job Shop 调度 Gym 环境"""

    def __init__(self, num_jobs=10, num_machines=5):
        super().__init__()
        self.num_jobs = num_jobs
        self.num_machines = num_machines

        # 动作空间：选择下一个要加工的工件（离散）
        self.action_space = spaces.Discrete(num_jobs)

        # 状态空间：机器状态 + 工件状态 + 全局特征
        # 简化版：每个机器一个 float（剩余时间），每个工件一个 float（剩余工序数）
        obs_dim = num_machines + num_jobs + 3
        self.observation_space = spaces.Box(
            low=0, high=1e6, shape=(obs_dim,), dtype=np.float32
        )

    def reset(self):
        """重置环境到初始状态"""
        # 生成随机工件数据
        self.jobs = np.random.randint(1, 20, (self.num_jobs, self.num_machines))
        self.machine_remaining = np.zeros(self.num_machines)
        self.current_time = 0.0
        self.completed_jobs = 0
        return self._get_obs()

    def step(self, action):
        """执行动作：将某个工件安排到可用机器上"""
        job_id = action
        # 这里省略了具体的调度逻辑
        # ...
        obs = self._get_obs()
        reward = self._compute_reward()
        done = (self.completed_jobs == self.num_jobs)
        return obs, reward, done, {}

    def _get_obs(self):
        """拼接当前状态向量"""
        machine_state = self.machine_remaining.copy()
        job_state = np.sum(self.jobs > 0, axis=1).astype(float)
        global_feat = np.array([
            self.current_time,
            self.completed_jobs / self.num_jobs,
            np.mean(self.machine_remaining)
        ])
        return np.concatenate([machine_state, job_state, global_feat])

    def _compute_reward(self):
        """计算奖励（简化版：每步 -1 + 完工时的奖励）"""
        return -1.0  # 占位，实际实现会更复杂

这个接口遵循 OpenAI Gym 标准，可以直接与 Stable-Baselines3 等主流 RL 库对接。当然，真实环境远比这个复杂——你需要实现工件工序约束、机器阻塞、setup time 等细节。但骨架是一样的。
