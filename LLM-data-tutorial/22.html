通用 vs 领域模型的数据差异

前面所有的章节，我们讨论的几乎都是"通用数据"——从 Common Crawl 中清洗出的多领域、多语言文本。这种思路对通用大模型是正确的：你希望模型知道得越多越好，覆盖面越广越好。但是，当目标从"一个什么都知道的通用助手"变成"一个精通医学诊断的 AI 医生"时，数据策略需要一场彻底的改变。

这背后是数据工程中一个根本性的权衡：广度 vs 深度。通用模型需要数据广度——覆盖尽可能多的主题、风格和领域。领域模型需要数据深度——在同一主题上拥有足够的密度和颗粒度，让模型能够理解领域内的细微差别和专业术语。

2024 年，某知名医疗 AI 团队做了一个有趣的实验：他们在一个通用模型的基础上，用 200 亿 token 的医学文献进行继续预训练（continue pre-training），获得了专精医学的领域模型。但在评测时发现，这个领域模型在医学问答上表现优异，但在普通对话场景中反而比原版通用模型更差——因为医学文献的语体风格（严谨、正式、充满术语）"污染"了模型的通用表达能力。这个案例说明了领域数据策略的一个核心原则：领域适配必须在通用能力和专业能力之间找到平衡。单纯堆砌领域数据而不做策略设计，会带来灾难性的知识遗忘（catastrophic forgetting）。

在实践中，领域数据策略通常包含三个环节：领域数据采集与清洗（找到高质量的专业语料）、通用数据保持（在继续预训练中混入一定比例的通用数据，防止知识遗忘）、以及领域配比优化（确定通用与领域数据的最佳混合比例）。这三个环节在不同领域的具体实现差异巨大。

代码领域

代码是 LLM 最早的"领域成功故事"之一。GitHub Copilot、Code Llama、DeepSeek-Coder 等代码模型证明了专用数据策略的巨大价值。但代码数据的清洗和处理方式与通用文本有着本质区别。

第一个挑战是 自动生成的代码 vs 人工编写的代码。GitHub 上有大量由 IDE 自动生成、模板填充、或者从其他库复制粘贴的代码。这些代码虽然在语法上正确，但通常质量低下——缺乏注释、不符合编码规范、没有考虑边界情况。一个简单的判定方法：如果一段代码的 commit message 是 "auto-generated" 或 "initial commit"，它大概率是低质量的自动生成代码。

第二个挑战是 代码去重。你不能用清洗自然语言文档的方法去重代码。自然语言去重关注的是文本级别的精确或近似重复；但代码中，两段在文本上几乎一样的代码（仅变量名不同）可能就是完全不同的函数实现。反过来，两段文本完全不同的代码可能实现了相同的功能（一个用 for 循环，一个用 while 循环）。正确的做法是使用抽象语法树（AST）级别的去重——将代码解析为语法树，比较树结构的相似性。

代码质量标注方面，DCLM（DataComp for Language Models）在 2024 年提出了一种高效方法：他们用代码仓库的 GitHub Star 数作为代理标签（proxy label），训练一个分类器来预测代码质量。Star 数超过 1000 的项目中的代码被视为正样本，Star 数为 0 的视为负样本。虽然这个指标并不完美（好的代码也可能在冷门仓库中），但在大规模筛选中它出人意料地有效——使用这个分类器筛选后的代码数据训练出的模型，在 HumanEval 上的 pass@1 提升了 11%。

医学和科学领域

医学和科学领域的数据清洗方法与通用领域有显著不同。通用领域追求"去除噪声"，而科学领域追求"保留精确的专业结构"。一篇医学论文中的表格、化学式中的下标、基因序列中的特定模式——这些在通用数据清洗中可能被视为"难以处理"的内容被丢弃的东西，恰好是领域模型中最重要的信息载体。

在 2025 年的 FineMed 研究中，研究人员发现了一个引人深思的结果：对医学领域来说，医学术语密度过滤比通用教育质量过滤效果高出 2 倍。也就是说，与其用通用的"教育质量评分"来判断一篇文档是否适合训练医学模型，不如直接计算文档中医学术语的密度——包含更多医学术语的文档更有价值。

# 医学术语密度过滤实现
import re
from typing import Set

# 医学术语词典（从 UMLS、MeSH 中提取）
MEDICAL_TERMS: Set[str] = {
    # 疾病名称（示例子集）
    "myocardial infarction", "diabetes mellitus", "hypertension",
    "osteoarthritis", "pneumonia", "schizophrenia",
    # 药物名称
    "metformin", "atorvastatin", "lisinopril", "omeprazole",
    # 解剖学术语
    "myocardium", "endothelium", "cerebellum", "epidermis",
    # 临床概念
    "biopsy", "angiography", "electrocardiogram", "hemoglobin A1c",
    # 中文医学术语（示例）
    "心肌梗死", "糖尿病", "高血压", "骨关节炎", "肺炎",
    "二甲双胍", "阿托伐他汀", "心肌", "内皮",
}

MIN_TERM_DENSITY = 0.02  # 最低医学术语密度阈值（2%）

def medical_term_density_filter(doc: str) -> bool:
    """
    医学术语密度过滤器。
    保留术语密度高于阈值的文档。
    """
    total_words = len(doc.split())
    if total_words == 0:
        return False

    doc_lower = doc.lower()
    term_count = 0
    for term in MEDICAL_TERMS:
        # 使用词边界匹配，避免部分匹配
        pattern = r'\b' + re.escape(term) + r'\b'
        matches = re.findall(pattern, doc_lower)
        term_count += len(matches)

    density = term_count / total_words
    return density >= MIN_TERM_DENSITY

def filter_medical_corpus(documents: list[str]) -> list[str]:
    """对整个医学语料库执行过滤"""
    kept = 0
    total = len(documents)
    filtered = []
    for doc in documents:
        if medical_term_density_filter(doc):
            filtered.append(doc)
            kept += 1
    print(f"保留率: {kept/total*100:.1f}% ({kept}/{total})")
    return filtered

类似的逻辑也适用于科学领域的其他分支。物理学论文中的费曼图术语、生物学论文中的基因名称、化学论文中的化合物命名——每种科学领域都有自己的专业词汇特征。基于专业词汇密度进行过滤，比通用的文本质量指标更能识别出对该领域有价值的内容。这种方法的核心洞察是：领域知识的特征不在于"写得好不好"，而在于"用了多少专业术语"。

PubMed 和 ArXiv 是医学和科学领域的两个核心数据源。但它们的数据需要专门处理：PDF 格式的论文需要转换为可解析的文本（使用 GROBID、ScienceBeam 等工具），数学公式需要保留 LaTeX 表示而非渲染成图片，引用关系可以提取出来作为额外的训练信号。2025 年的一篇论文发现，将论文中的引用关系（"根据 Smith et al. (2023)...") 显式标记并保留，而不是像通用清洗那样当作无意义段落丢弃，可以使模型在科学问答上的准确率提升 7%。

多语言领域适配

如果说代码和医学领域的挑战是"数据质量"，那么多语言领域的挑战首先是"数据可得性"。在 Common Crawl 的数据分布中，英文占比约 45%，而很多语言的占比不到 0.01%。这种严重不平衡导致了一个残酷的现实：即使你用最先进的过滤技术处理撒哈拉以南非洲地区的本地语言数据，你最终能得到的数据量可能还不够填满一个 GPU 的显存。

CCNet 项目对 174 种语言的覆盖是一个标志性工作。Meta AI 团队为每种语言训练了独立的 fastText 语言识别模型，并针对每种语言的数据量设定了不同的处理策略——对于高资源语言（英文、中文、西班牙文），使用复杂的三阶段清洗；对于低资源语言（如斯瓦希里语、祖鲁语），则只做基础的语言验证和暴力过滤，以尽可能保留数据。

2026 年的一个研究进一步提出了"语言适应性数据策略"的概念：对于每种语言，先评估其"数据可利用性"（data availability），然后动态调整清洗强度。对于可用数据极少的语言，清洗标准应该极度宽松——宁可保留一些噪声，也不要因为过度清洗而丢掉稀缺的语料。这背后的工程哲学是：低资源语言的第一要务是"有数据"，然后才是"有好数据"。

低资源语言的数据策略： 对于数据量少于 10GB 的语言，清洗流水线的重点应该从质量过滤转移到语言识别精度。因为这类语言的最大问题不是"数据不够干净"，而是"数据中混入了太多其他语言的文本"。提高语言识别模型的精度（例如使用更大的 n-gram 特征）、引入语言特有的停用词过滤、甚至使用人工标注的种子数据训练专用识别器，这些投入的 ROI 远高于通用质量过滤。换句话说：对于低资源语言，先把"是不是这种语言"搞清楚，再去纠结"这数据质量好不好"。

语言平衡策略是另一个关键议题。在多语言数据配比中，常见的做法是对低资源语言进行上采样（upsampling）——人为提高它们在训练数据中的占比。比如，斯瓦希里语在 Common Crawl 中只占约 0.005%，但在训练配比中可能被上采样到 1%。但这个做法有一个上限：如果上采样倍数过大，模型会在低资源语言上出现"死记硬背"（overfitting）的现象，反而损害泛化能力。

2026 年的一篇综述性论文给出了一个经验法则：对于多语言数据配比，各语言的 token 数量与对应语言使用人口的比例不应相差超过一个数量级。也就是说，如果一个语言的使用人口是英文的 1/100，它的训练数据量不应低于英文的 1/1000。这个比例既能保证各语言都有足够的训练信号，又不会因过度上采样而损害模型的多语言泛化能力。
