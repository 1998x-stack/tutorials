从困惑度到分类器

上一章我们讲到困惑度过滤有一个根本性的局限：它用"像不像 Wikipedia"来衡量质量，而高质量文本的形态远不止一种。就像你不能用一把尺子量所有东西一样，单纯依赖困惑度去判断文本质量，难免会有偏差。

那怎么办？一个自然的思路是：与其用一个固定的参考语料来间接判断，不如直接训练一个分类器，来学习"什么文本是有教育价值的"。

这就是从困惑度到分类器的范式升级——从无监督到有监督。困惑度不需要标注数据，但不够精准；分类器需要标注数据，训练成本高，但分类精度远超困惑度。2024 年 HuggingFace 的 FineWeb-Edu 论文把这条路走到了极致：他们用 Llama-3-70B 标注了 50 万条 Web 文档的教育质量分数，训练了一个轻量级的 fastText 分类器，然后用于过滤整个 FineWeb 语料。

fastText 分类器的工作原理

你可能以为做文本质量分类需要 BERT、RoBERTa 这样的 Transformer 模型。但 FineWeb-Edu 选择了另一个工具——Facebook 的 fastText。

fastText 本质上是一个基于 n-gram 特征的线性分类器。它的工作流程是：把文本拆成字符 n-gram（例如 "transformer" 拆成 "tra"、"ran"、"ans" 等 3-gram），把这些 n-gram 作为特征输入一个浅层神经网络（通常只有一层 hidden layer）。训练速度极快——在 50 万样本上只需要 10 分钟就能收敛。推理速度更是快得惊人——单 CPU 核每秒可以处理上百万篇文档。

import fasttext

# 准备标注数据（FastText 格式）
# 每行：__label__分数 + 文档文本
# __label__5 这是一篇高质量的学术论文...
# __label__1 这是一个SEO堆砌关键词的页面...

with open('edu_train.txt', 'w') as f:
    for doc in high_quality_docs:
        f.write(f"__label__5 {doc}\n")
    for doc in low_quality_docs:
        f.write(f"__label__1 {doc}\n")

# 训练教育质量分类器
model = fasttext.train_supervised(
    input='edu_train.txt',
    lr=1.0,              # 学习率
    epoch=25,            # 训练轮次
    wordNgrams=3,        # 词级 3-gram 特征
    dim=100,             # 词向量维度
    loss='softmax',      # 多分类损失
    thread=32            # 多线程训练
)

# 保存模型
model.save_model('edu_classifier.ftz')

# 对百万级文档批量推理
scores = model.predict(web_documents, k=1)
# 输出: [('__label__3', 0.85), ('__label__4', 0.72), ...]
为什么 fastText 够用？ 教育质量的判断本质上是一个"文本模式识别"问题——高质量学术文章有明显的词汇和句式特征（如引用格式、专业术语、正式语体），低质量 SEO 页面也有明显的特征（关键词堆砌、短句重复、广告用语）。fastText 的 n-gram 特征恰好能捕捉这些模式，而且它不需要深度学习那样的海量数据。在 FineWeb-Edu 的实验中，fastText 的分类准确率达到了 85% 以上，与轻量级 Transformer 相当。

FineWeb-Edu 的方法

FineWeb-Edu 的完整流程是这样的：第一步，从 FineWeb 语料中随机抽取 50 万篇文档；第二步，用 Llama-3-70B 对这些文档进行教育质量评分（0-5 分）；第三步，用这些标注数据训练 fastText 分类器。分类器的输出也是一个 0-5 分的教育评分。

在标注阶段，FineWeb-Edu 选择的训练数据很有讲究：正样本是 Wikipedia、ArXiv、高质量的学术和技术网站；负样本是 SEO 垃圾页面、机器生成内容、低质量论坛帖子。他们特意确保正负样本都覆盖了足够多的主题和文本类型，让分类器学会判断"教育价值"而非简单地"像学术文章"。

重磅发现： FineWeb-Edu 证明了 1.3T 高质量 token 胜过 15T 原始数据——在 MMLU 上，用 FineWeb-Edu 的 1.3T token 训练的模型比用完整 FineWeb（15T token）训练的模型高出近 5 个百分点。这是"质量优先"理念最有说服力的证据。

分类器的工程部署

在工程层面，教育质量分类器的部署非常直接。它被集成到 DataTrove 这样的流水线框架中，作为数据过滤流水线的一个"过滤器"（filter）组件。每篇文档流入后，分类器给出一个教育评分，流水线根据预设的阈值决定保留还是丢弃。

常见的做法是只保留评分 > 3 分的文档，这样通常保留 5-15% 的原始数据。但这个比例可以根据实际需求调整：

  • 数据量优先：阈值降低到 2 分，保留约 50% 的数据，虽然混入了一些低质量内容，但数据量大大增加
  • 质量优先：阈值提升到 4 分，只保留 5-8% 的"精华"数据
  • 平衡模式：阈值设为 3 分，保留约 10-15% 的数据

在 FineWeb-Edu 实验中，他们选择了最激进的 4-5 分策略，只保留 8.7% 的数据。这个选择是基于他们做的一个消融实验：不同阈值下训练模型在 benchmark 上的表现。结果显示 4 分阈值是最优的。

工程选择： 分类器不是越复杂越好。FastText 的线性分类器效果已经远超随机，在 50 万训练样本上达到 85%+ 的准确率。像 Llama-3-70B 这样的 LLM 做分类器虽然精度更高（约 92%），但推理成本贵 1000 倍——处理 1 亿篇文档，用 fastText 只需要 10 分钟和一台机器，用 Llama-3-70B 需要几十台 GPU 跑好几个星期。在数据工程中，规模化效率压倒一切。

在实际部署中还有一个容易被忽略的坑：分类器的时效性。Web 内容的语言模式会随时间演变——2023 年的 SEO 内容和 2025 年的 SEO 内容使用了不同的关键词策略。我见过一个团队两年前训练的分类器在最新语料上的准确率从 85% 掉到了 68%。建议的做法是定期（比如每季度）用最新语料中的一小批数据重新评估分类器效果，必要时重新训练。

总结一下这一章的核心：教育质量分类器是困惑度过滤的"进化版"——从无监督到有监督，从"单一参考标准"到"学习什么是有质量的文本"。FineWeb-Edu 用 fastText + LLM 标注的组合，在成本和精度之间找到了一个极佳平衡点。但这里有一个关键问题没有展开：标注数据从哪来？下一章我们将深入 LLM 辅助数据标注——如何用大模型自己来标注训练数据，这背后的 prompt 工程设计、蒸馏流程、以及潜在的偏差问题，都是值得深入探讨的话题。
