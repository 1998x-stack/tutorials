第24章 Chiplet与先进封装

同学们好！上一章我们讨论了RISC-V如何在ISA层面打破传统垄断，今天我们来聊一个在物理层面同样具有革命性的趋势——Chiplet（芯粒）与先进封装。如果你用近两年的高端CPU或GPU（比如AMD的Ryzen或NVIDIA的H100），你手里其实已经拿了一个"拼接怪"——里面根本不是一整块硅，而是多个小芯片通过高速接口拼在一起的。这个从"单体"到"乐高"的转变，是后摩尔时代芯片设计最重要的范式转移之一。

24.1 从单片到芯粒：范式的转移

在理解Chiplet之前，我们先看看"单片"芯片为什么走到了尽头。

第一个问题是成本。一块芯片的成本 = 晶圆成本 ÷ （每片晶圆的芯片数 × 良率）。良率随着芯片面积的增大呈指数级下降——这就是集成电路领域著名的"Murphy's Law of Silicon"（硅片墨菲定律）。一块858mm²的芯片（光刻机的reticle limit，即单次曝光最大面积），良率可能低到只有20-30%。这意味着你生产出来的大部分都是废品。

换句话说，如果你想造一个巨大的芯片，你得为那些报废品付钱——很多很多钱。NVIDIA H100的Die尺寸约为814mm²，几乎达到了reticle limit。从经济学角度看，再做大已经不划算了。

Chiplet的哲学很简单：别造一个巨大的芯片，把它拆成多个小芯片，然后用高速信号把它们连接起来。这样做有四个直接的好处：

  • 良率提升——每个小芯片的面积小得多，良率显著提高。比如一个200mm²的芯片良率80%，拆成4个50mm²的芯片（假设良率分别为95%），通过测试筛选可以把坏的部分丢掉。
  • 混合工艺节点——CPU核心需要最先进工艺（如3nm）以获得最高性能，而I/O接口、内存控制器可以用成熟工艺（如12nm）。把两者做在不同芯片上，再封装到一起，既保证了性能又控制了成本。
  • 更快的上市时间——设计小芯片比设计大芯片简单得多。模块化设计让不同团队可以并行工作。
  • 复用——同一款I/O Chiplet可以用在不同产品系列中。AMD的I/O Die就用于了其所有Zen架构的EPYC和Ryzen处理器。

AMD EMOPYC Rome（2019）是Chiplet路线最经典的案例。Rome处理器包含9个芯片：1个中央I/O Die（12nm工艺）和8个CPU Chiplet（CCD，7nm工艺），每个CCD包含8个Zen 2核心。这种"中心化I/O + 计算Chiplet"的架构成为后续所有AMD处理器的模板。EPYC Rome的成功让整个行业看到了Chiplet的商业价值——AMD从此在服务器市场开始收复失地。

24.2 UCIe标准：芯粒的"USB"

Chiplet的瓶颈很快就从"能不能做"变成了"怎么连"——不同公司生产的Chiplet之间需要一种标准化的互连协议。这就像USB对于外设的意义一样。2022年，Intel联合AMD、Arm、TSMC、Samsung等巨头，共同推出了UCIe（Universal Chiplet Interconnect Express）标准。

物理层：UCIe定义了两种物理层选项——标准型（16 GT/s per lane）和高级型（32 GT/s per lane）。标准型可以用普通的organic substrate（有机基板）实现，成本较低；高级型需要silicon interposer（硅中介层）或桥接技术，带宽密度更高。

协议层：UCIe支持PCIe、CXL以及原始的die-to-die流式传输。这让Chiplet之间的通信既可以利用成熟的PCIe生态系统，也可以使用轻量级的自定义协议进行低延迟通信。

关键指标：延迟<2ns（先进封装），能效<0.5 pJ/bit，带宽密度高达1.3 TB/s per mm of die edge。这些数字意味着什么？Chiplet之间的数据传输几乎可以被视为"片上通信"——延迟和功耗远远低于片间PCB走线，接近片内连线。

2024年发布的UCIe 2.0进一步增加了对3D封装的支持、汽车可靠性等级（ASIL）以及可测试性（DFT）特性。UCIe的发展速度远超预期，已经成为Chiplet生态系统的事实标准。

一个"踩坑"故事：我在2022年参与了一个内部Chiplet项目，当时UCIe还处于草案阶段，我们决定用私有协议。结果一年后UCIe正式发布，我们的私有协议虽然性能绰绰有余，但无法与任何第三方Chiplet互操作。老板气得直拍桌子："你们这等于在造一个只能跟自家设备通信的USB！"——这就是标准的威力，宁可用一个"够用"的标准，也不要用一个"最优"的私有方案。

24.3 先进封装技术全景

UCIe定义了逻辑层面的互连，但物理上如何把多个Chiplet拼在一起，则是先进封装技术的领域。这里的技术栈非常丰富，让我一一为你们道来。

2.5D封装——Silicon Interposer（硅中介层）：最经典的2.5D方案。一个硅中介层放在Chiplet和封装基板之间，中介层上用硅的微细布线连接Chiplet。TSMC的CoWoS（Chip-on-Wafer-on-Substrate）是2.5D封装的行业标杆。NVIDIA从Pascal时代的GP100就开始使用CoWoS，到H100/B100时代已经是第二代CoWoS-S。CoWoS-S可以支持最多2倍reticle size的interposer，是当前AI训练GPU的首选封装方案。

2.5D封装——EMIB（Intel的嵌入式桥接技术）：Intel的EMIB（Embedded Multi-Die Interconnect Bridge）走了一条不同的路——不用大面积的硅中介层（那很贵！），而是在有机基板中嵌入小块的高密度硅桥，仅在Chiplet之间需要高速通信的位置使用。这比CoWoS便宜得多，但带宽密度略低。

3D封装——Foveros（Intel）和SoIC（TSMC）：3D封装更进一步，直接把芯片堆叠在一起。Intel的Foveros采用face-to-face bonding，芯片之间通过密集的microbump（微凸点）连接。TSMC的SoIC（System on Integrated Chips）则更加激进——直接使用hybrid bonding（混合键合），不需要microbump，把铜对铜直接焊接在一起，间距可以做到小于1μm！

AMD 3D V-Cache是3D堆叠的一个精彩应用。在Zen 3的CCD（计算Chiplet）上，AMD直接堆叠了一个额外的64MB L3 Cache芯片。这使得每个CCD的总L3 Cache从32MB暴增到96MB。在游戏场景中，这个"缓存堆叠"带来了平均15%的性能提升——原因很简单，更大的L3可以减少对内存的访问，而内存访问是最大的性能瓶颈之一。有趣的是，AMD实际上是把一个"缓存Chiplet"用3D方式放在计算Chiplet上面，两者通过hybrid bonding连接。这是Chiplet理念的终极体现——不只是"CPU + I/O分离"，连缓存都可以做成独立的Chiplet。

24.4 开源Chiplet：Occamy及未来

就像RISC-V在ISA层面推动"开源硬件"一样，Occamy项目在Chiplet层面也做着同样的事情。

Occamy——由苏黎世联邦理工学院（ETH Zurich）和博洛尼亚大学（University of Bologna）联合开发。这是一个双Chiplet设计，每个Chiplet包含216个RISC-V核心（Snitch核心，专门针对整数和浮点运算优化的微型核心），共432个核心。采用12nm工艺，双精度浮点性能达到768 GFLOPS。整个设计从RTL到物理设计到固件全部开源——这可能是目前最复杂的开源芯片设计项目。

Ramora——Occamy的继承者，双Chiplet设计，采用mesh NoC（片上网络）互连。还在设计阶段，目标是将频率提高到1GHz以上。

Ogopogo——更加宏大的四Chiplet设计，采用7nm工艺，目前处于概念验证阶段。如果成功，将成为首个完全开源的7nm四Chiplet处理器。

Occamy项目组有一个更大胆的愿景——Chiplet Marketplace（芯粒市场）。想象一下这样的场景：你想设计一个AI服务器加速卡，不用从头做起。你去Chiplet Market挑选：从A公司买一个RISC-V CPU Chiplet，从B公司买一个AI加速Chiplet，从C公司买一个HBM内存控制器Chiplet，然后找一家封装厂把它们拼在Interposer上。整个过程就像在淘宝上买零件组装电脑一样。

Chiplet的"乐高"比喻：Chiplet架构正在将芯片设计从"艺术"变成"乐高"——这个行业的未来不在于造出更大的芯片，而在于找出更聪明的方式把芯片组合在一起。当你可以在一个季度内拼出一款AI加速芯片，而不是花18个月从零设计时，整个行业的创新节奏将被彻底改变。

24.5 Chiplet的工程挑战

Chiplet虽然前景光明，但工程落地的挑战也不容小觑：

热管理：3D堆叠将多个发热源堆在一起，功率密度成倍增加。AMD的3D V-Cache本身几乎不发热（SRAM），所以堆叠相对容易。但如果把计算核心堆叠在一起（比如Intel的Foveros Lakefield），散热问题就非常棘手——热量无法有效传导到散热器。Intel在Lakefield中采用"热de-rating"（降频）策略来应对。

信号完整性：Chiplet之间的高速信号经过C4 bump、interposer走线、再到目标芯片，跨越了多种介质的界面。25Gb/s以上的信号对阻抗匹配和时钟恢复提出了极高的要求。这也是UCIe为什么这么重要——标准化的PHY设计可以让信号完整性问题在IP层面就得到解决。

已知合格裸片（Known-Good-Die）测试：在组装Chiplet之前，必须先确保每个Chiplet是"好"的——否则10个Chiplet里有一个坏的，整个封装就是废品。但Chiplet测试面临一个两难：测试太粗（只能测基本功能），组装后可能出问题；测试太细（要在full-speed下测所有IO端口），成本太高。行业正在开发的方案是"built-in self-test"（BIST），让Chiplet在上电时自我检测，报告健康状态。

标准的荒漠：UCIe解决了物理层和链路层的互操作问题，但更高层次的标准仍需完善——如何发现Chiplet的能力？如何管理电源？如何保证安全性（防止第三方Chiplet窃取数据）？RISC-V和DSA社区正在推进一个名为"Open Compute Project - Chiplet"的倡议，目标就是填补这些空白。

让我分享一个亲身经历的"踩坑"故事。我曾参与一个内部Chiplet项目，规划得相当完美：UCIe接口、混合工艺、复用I/O Chiplet。但当我们拿到第一批封装好的样品时，发现Chiplet启动时UCIe链路的training phase总是失败。排查了整整三周，最后发现原因让人哭笑不得——我在设计Chiplet A时的参考时钟频率与Chiplet B差了25ppm（百万分之二十五）。对于低速设计，25ppm根本不影响；但在UCIe的32Gbps链路上，25ppm的频率偏差会导致时钟数据恢复（CDR）电路无法锁定。从那以后，我们的Checklist上多了一条铁律：所有Chiplet必须共享同一个参考时钟源，或者采用独立的clock recovery机制。

// UCIe Streaming Interface 简化伪代码
// 模拟Die-to-Die数据传输的基本握手

module ucie_stream_tx (
    input  logic        clk,
    input  logic        rst_n,
    input  logic [63:0] tx_data,
    input  logic        tx_valid,
    output logic        tx_ready,
    // Die-to-Die 物理接口
    output logic [15:0] tx_d2d_data,
    output logic        tx_d2d_clk
);

    logic [63:0] data_reg;
    logic [2:0]  beat_count;
    logic        sending;

    // UCIe 以64B flit为单位传输
    // 物理层把64B分解为16个16-bit beatx

    always_ff @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            tx_ready   <= 1'b1;
            beat_count <= 3'd0;
            sending    <= 1'b0;
        end else begin
            if (tx_valid && tx_ready) begin
                data_reg  <= tx_data;
                sending   <= 1'b1;
                beat_count <= 3'd0;
                tx_ready  <= 1'b0;  // 数据被接收，开始发送
            end

            if (sending) begin
                tx_d2d_data <= data_reg[15:0];
                data_reg    <= {16'b0, data_reg[63:16]};
                if (beat_count == 3'd15) begin
                    sending   <= 1'b0;
                    tx_ready  <= 1'b1;  // 16个beat发送完毕
                end else begin
                    beat_count <= beat_count + 1;
                end
            end
        end
    end

    assign tx_d2d_clk = clk;

endmodule

// 实际UCIe PHY处理包括：CRC、加扰、均衡、CDR、ECC纠错等
// 但核心思想的精髓都在这里——把宽数据serialize为高速窄通道
