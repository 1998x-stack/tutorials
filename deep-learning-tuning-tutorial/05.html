批大小——不止是"调到显存装满为止"

大家好，我是你们的讲师。如果你问一个深度学习工程师"batch size怎么设"，最常见的回答是："调到显存装满为止。"这是对的，但是不够。批大小不仅是显存利用率的参数，它从根本上决定了梯度估计的精度、训练的动态过程、以及模型的最终泛化能力。

我们先讲一个真实的故事。某实验室在做对比学习（SimCLR），按照论文复现，batch size设成了4096。但实验室只有4张卡，每张卡最多塞512张图——这样需要梯度累积2步才能到4096。跑了两周，模型就是不收敛，效果比随机还好不了多少。后来发现，问题出在batch size上——对比学习依赖大量的负样本来做正负区分，而微调阶段的"effective batch size"太小（实际上是每步只有4×512=2048），不足以提供足够的负样本。把梯度累积步数改成4步后，模型立刻就开始收敛了。

教训：batch size不是"显存能塞多少就设多少"——你需要弄清楚effective batch size到底是多少，以及你的任务对batch size有没有硬性要求。

批大小的统计学本质

从统计学的角度看，批大小的作用非常清晰。全批量梯度下降使用整个数据集计算精确梯度，但计算量太大。小批量SGD使用B个样本来估计梯度：

∇f_B(θ) = (1/B) · Σ ∇f_i(θ)   ≈   ∇f(θ)   （全量梯度）

小批量梯度是真实梯度的无偏估计——期望等于真实梯度，但存在方差：

Var(∇f_B(θ)) = Var(∇f(θ)) / B

梯度估计的方差与batch size成反比。batch扩大一倍，方差减半。这就是批大小的统计学本质：你在用计算量换取梯度估计的精度。

从中心极限定理的角度看，小批量梯度服从近似正态分布：

∇f_B(θ) ~ N(∇f(θ), Σ / B)

这意味着batch越大，梯度估计越集中在真实梯度附近，更新方向越稳定。但同时也意味着"噪声正则化"效应减弱了——因为方差的减小削弱了随机性的正则化作用。

在多卡训练中，理解effective batch size至关重要。很多论文里写"batch size = 1024"，但实际上是4卡 × 每卡64 × 4步梯度累积。如果你在自己的环境下复现，每张卡batch size设成1024，显存直接炸了。原因就是你没理解这个公式：

effective_batch_size = per_device_batch × num_gpus × accumulation_steps
复现实验的关键："batch size"不是一个数字——至少应该是三个数字的乘积。BugBuster喵在知乎上专门强调过："调参时要记录清楚——单卡batch、卡数、累积步数、总batch到底是多少。很多复现实验复现不出来，就是这里没说清。"每次做实验，请在日志里明确记录这三项。

大批量泛化退化之谜

虽然大batch的梯度估计更准确、训练更快，但大量实验发现了一个反直觉的现象：batch size超过某个阈值后，泛化性能开始下降。

这是为什么？从优化景观的角度来看：

  • 小batch的噪声起到了正则化作用：梯度噪声让参数不会精确收敛到某个极小值的谷底，而是在谷底附近"徘徊"。这种徘徊帮助模型避开尖锐的极小值（tiny bump），因为噪声会把参数从狭窄的尖坑里弹出来。最终模型收敛到更平坦、更宽的谷底——泛化更好。
  • 大batch的梯度更准确：但准确的梯度带参数直奔最近的极小值，如果有多个极小值时，大batch倾向于收敛到离起点最近的那个——往往是尖锐的极小值。

Flat Minima vs Sharp Minima 的理论我们在第三章已经介绍过了。在batch size的语境下，batch越小，梯度噪声越大，越容易逃离尖锐极小值；batch越大，梯度越稳定，越容易被尖锐极小值捕获。这就是大批量泛化退化的核心原因。

但是，这并不意味着大batch永远不好。如果配合合适的学习率缩放和warmup策略，大batch也能达到与标配batch相近甚至更好的泛化效果。关键是要找到正确的学习率和batch size的组合。

注意区分："大批量泛化退化"主要是在固定epoch数的情况下观察到的。如果给大batch模型更多epoch，让它多走几步，通常可以弥补泛化差距。在实践中，batch size翻倍时epoch也应该适当增加——也就是用更多的计算量来换取更高的精度。

梯度累积的工作原理

梯度累积（Gradient Accumulation）是在显存不足时等效扩大batch size的核心技术。它的原理非常简单：把一次大batch的梯度计算拆成多次小batch的梯度计算，累积起来，然后用累积的梯度一次更新参数。

完整的PyTorch实现如下：

accumulation_steps = 4  # 累积4次后更新一次
        optimizer.zero_grad()

        for i, (inputs, labels) in enumerate(dataloader):
            outputs = model(inputs)
            loss = criterion(outputs, labels)

            # 关键：loss必须除以累积步数
            loss = loss / accumulation_steps
            loss.backward()

            if (i + 1) % accumulation_steps == 0:
                # 累积够了，更新参数
                optimizer.step()
                optimizer.zero_grad()

这个实现有两个关键细节：

细节一：loss / accumulation_steps。这个除法是必须的。如果不除，在 accumulation_steps 次前向后，累积的梯度是单次梯度的 accumulation_steps 倍。等效于batch size放大了accumulation_steps倍，但梯度量级也放大了同样的倍数——相当于LR放大了accumulation_steps倍，训练不稳定。除以accumulation_steps后，每次micro-batch的梯度贡献归一化，累积梯度等效于一个batch的梯度。

细节二：BN统计量问题。梯度累积模式下，BN的均值和方差是基于每个micro-batch（小batch）计算的，而不是基于effective batch。如果micro-batch太小（比如每卡只塞2张图），BN统计量会有较大偏差，导致训练效果不如直接使用大batch。解决方案：用SyncBN同步所有卡上BN统计量，或者换用不依赖batch维度的归一化方法（如LayerNorm、GroupNorm）。

梯度累积的适用场景：

  • 显存OOM，但任务需要大batch（对比学习、PPO）
  • 设备不够，用时间换空间
  • 想用大batch但不想改dataloader代码

梯度累积的局限性：

  • 每个micro-batch只做前向+反向，不做参数更新——训练速度变慢（wall time增加）
  • BN的统计量偏差问题
  • 梯度累积和真正的大batch不完全等价——优化轨迹不同

Batch Size与Learning Rate的线性缩放法则

batch size和学习率之间有一个重要的耦合关系——线性缩放法则（Linear Scaling Rule）。

由Goyal等人在2017年的论文中系统化提出：当batch size乘以k时，学习率也乘以k，前提是配合warmup策略。公式很简单：

lr_new = lr_old × (batch_size_new / batch_size_old)

一个被广泛验证的经验锚点：batch size 从64扩到256（4倍）时，LR从0.1扩到0.4。

这个缩放规则背后的直觉：batch size扩大k倍后，每步梯度估计的方差缩小了k倍，梯度更准确。因此可以用更大的步长来利用更准确的梯度信息。如果不增大LR，每步更新幅度不变但样本效率降低——相当于"浪费"了更准确梯度带来的信息量。

这个规则的适用条件：

  • 主要适用于SGD和SGD+Momentum
  • 对Adam不完全适用——Adam有自适应学习率机制，batch变化的影响已被部分吸收
  • 批量差距过大时（如64→8192），线性缩放给出的LR过大，需要sub-linear缩放或更长的warmup
  • 必须配合warmup，否则大LR在训练初期直接发散

DeepSeek的Scaling Law：在LLM预训练中，DeepSeek给出了更精确的幂律公式：

η_opt = 0.3118 · C^(-0.1250)    # 最优LR随计算量幂律下降
        B_opt = 0.2920 · C^(0.3271)     # 最优batch size随计算量幂律上升

其中C是计算预算（FLOPs）。这个公式告诉我们三件事：一是计算量增大时，batch size应该变大，LR应该变小；二是LR随计算量下降得非常慢（C^(-0.125)），所以在大规模训练中LR基本稳定；三是给定计算预算后，大多数超参固定，只需要调batch size和LR——这相当于把高维调参降到了二维。

不同任务的Batch Size推荐范围

任务类型 推荐batch size范围 典型值 说明
CV分类（CIFAR-10/100） 64 ~ 512 128 小数据集用更小batch
CV分类（ImageNet） 256 ~ 1024 256 多卡训练更大
CV检测/分割 8 ~ 32（每卡） 16 高分辨率图片限制batch
NLP Bert微调 16 ~ 64 32 序列长度padding影响
Transformer（WMT从头训） 4096 ~ 65536（token） 浮点 看token数而非句子数
LLM预训练 512 ~ 4096（序列） 随算力增长 满足DeepSeek幂律
对比学习（SimCLR/MoCo） 4096 ~ 16384 4096 越大越好，用梯度累积
推荐系统 32 ~ 512 256 超过512收益递减
PPO/RLHF 512 ~ 4096 1024 宁大勿小
LoRA微调 4 ~ 16（每卡） 8 小batch用梯度累积扩大
调参优先级：先确定LR，再确定batch size，然后根据batch size重新确认LR（线性缩放调整）。把batch size和LR当作一对耦合参数来调，不要独立调。一个系统化的做法是：在默认batch size上用LR Range Test确定LR，然后batch size改变时用线性缩放的LR作为起点，再做微调。

这一章我们理解了批大小的本质——它是梯度估计精度的控制旋钮。batch越大，梯度越准，训练越快，但噪声的正则化效应减弱、泛化可能退化。我们学习了梯度累积的工作原理（以及那个关键的"loss除以累积步数"的细节）、Batch Size与LR的线性缩放法则、以及DeepSeek的幂律公式在不同计算预算下的指导意义。
