第二十三章：Gymnasium环境实战

同学们好，欢迎来到第二十三章。

前二十二章我们一直在讲理论——MDP、贝尔曼方程、值函数、策略梯度、PPO、逆强化学习……如果你一路跟下来，现在脑子里应该装满了各种算法。但有一点可能让你心里发虚："说了这么多，我到底该在哪写代码？"

这就是本章要解决的问题。从今天开始，我们正式进入强化学习工程实践的阶段。本章的主角是 Gymnasium——强化学习领域最通用的环境接口标准。不管你是用 DQN 玩 Atari，还是用 PPO 训练机器人走路，第一步永远是一样的：创建一个环境，看一看它的状态空间和动作空间，然后开始交互。

学完本章，你将掌握 Gymnasium 的完整用法，包括经典环境的使用、自定义环境的编写、包装器的组合，以及如何为自己的问题选择最合适的环境。更重要的是，这是通往下一章 Stable-Baselines3 的必经之路——SB3 的全部示例都基于 Gymnasium 接口。

23.1 Gymnasium 简介与安装

说起强化学习的环境库，老读者一定听过 OpenAI Gym。它诞生于 2016 年，在长达七年的时间里一直是 RL 社区的事实标准。2022 年底，Gym 的维护权从 OpenAI 移交到了 Farama Foundation，并更名为 Gymnasium。

这次交接带来了一个非常重要的 API 变更。在 Gym v0.25 及以前，env.step(action) 返回四个值：obs, reward, done, info。但从 v0.26 开始（也就是 Gymnasium），done 被拆成了 terminated 和 truncated：

# 旧版 Gym API（v0.25 及以前）
obs, reward, done, info = env.step(action)

# Gymnasium API（v0.26+）
obs, reward, terminated, truncated, info = env.step(action)

# terminated = True  → 智能体达到了终止状态（如：杆子倒了）
# truncated  = True  → 智能体被截断终止（如：超过最大步数）
# done = terminated or truncated

这个拆分的意义在于：terminated 和 truncated 的后续处理不同。如果是 terminated（真的走到了终端状态），下一轮应该调用 env.reset() 重新开始。如果是 truncated（只是步数到了），有些算法会认为智能体的表现仍然是好的，不需要完全重置策略状态。这个区分在实现某些高级算法时很重要。

安装 Gymnasium 非常简单：

# 最小安装（只含经典控制类环境）
pip install gymnasium

# 完整安装（含 Box2D、MuJoCo、Atari 等）
pip install "gymnasium[all]"

# 按需安装特定环境族
pip install "gymnasium[box2d]"      # LunarLander, BipedalWalker
pip install "gymnasium[mujoco]"     # MuJoCo 连续控制环境
pip install "gymnasium[atari]"      # Atari 游戏环境

来，我们写第一个 Gymnasium 程序，感受一下它的工作方式：

import gymnasium as gym

# 创建环境
env = gym.make("CartPole-v1")

# 重置环境，返回初始观测和 info
obs, info = env.reset()
print(f"初始观测: {obs}")
print(f"观测空间: {env.observation_space}")
print(f"动作空间: {env.action_space}")

# 运行一个随机策略的 episode
terminated = truncated = False
total_reward = 0
step_count = 0

while not (terminated or truncated):
    action = env.action_space.sample()   # 随机动作
    obs, reward, terminated, truncated, info = env.step(action)
    total_reward += reward
    step_count += 1

print(f"总步数: {step_count}, 总奖励: {total_reward:.2f}")

env.close()   # 记得关闭环境释放资源

这就是 Gymnasium 的核心工作流：make → reset → step(循环) → close。所有环境，无论简单还是复杂，都用这同一个接口。你学会了这一个，就学会了所有。

23.2 经典环境详解

Gymnasium 内置了几十个标准环境，按复杂度大致分为三个梯队。我们逐个来看。

第一梯队：经典控制类（Classic Control）

这类环境状态维度低、动作空间简单、训练速度快，最适合算法调试和学习入门。

CartPole-v1：平衡小车上的杆子。4 维连续观测（小车位置、速度、杆子角度、角速度），2 个离散动作（向左推 / 向右推）。奖励：每步 +1，直到杆子倾斜超过 15 度或小车移出边界。这是 RL 界的 "Hello World"，任何算法的第一个测试都应该在这里跑。

MountainCar-v0：小车需要借助来回摆动的惯性冲上山顶。2 维观测（位置、速度），3 个离散动作（向左加速、不动、向右加速）。难点在于稀疏奖励——到达山顶之前奖励一直是 -1，智能体必须学会"为了长远目标忍受短期负收益"。这是检验探索能力的经典环境。

Acrobot-v1：双关节摆杆，要将末端摆到指定高度。6 维观测，3 个离散动作。类似于 CartPole 的难版，适合测试算法在稍高维度上的表现。

第二梯队：Box2D 环境

这些环境基于 Box2D 物理引擎，状态空间更丰富，需要更强的控制能力。

LunarLander-v2：月球着陆器降落。8 维连续观测（坐标、速度、角度、角速度、两条腿是否触地），4 个离散动作（主引擎、左引擎、右引擎、不动）。奖励设计非常细致——引擎有燃料惩罚，着陆平稳有额外奖励，坠毁则大额惩罚。这是 DQN 系列算法的标准验证平台。

BipedalWalker-v3：双足机器人行走。24 维连续观测，4 个连续动作（每个关节的扭矩）。这是从离散动作到连续动作的过渡环境，适合测试 PPO、SAC 等连续控制算法。

第三梯队：MuJoCo 与 Atari

这些是高维环境，需要更强的计算资源。

MuJoCo 环境族（如 HalfCheetah-v4、Hopper-v4、Walker2d-v4、Ant-v4、Humanoid-v4）：基于 MuJoCo 物理引擎的连续控制任务。观测维度从 17（Hopper）到 376（Humanoid），动作空间从 3 到 17 维连续。这些是连续控制算法（DDPG、TD3、SAC）的标准 benchmark。

Atari 环境族（如 PongNoFrameskip-v4、BreakoutNoFrameskip-v4）：以屏幕像素（210×160×3 的 RGB 图像）作为输入，通常需要 Frame stacking 和灰度化预处理。这是深度强化学习（DQN、Rainbow、IMPALA）的核心战场。

来看一下如何查看环境的观测和动作空间：

import gymnasium as gym

env_names = ["CartPole-v1", "MountainCar-v0", "LunarLander-v2",
             "HalfCheetah-v4", "ALE/Breakout-v5"]

for name in env_names:
    env = gym.make(name)
    obs_space = env.observation_space
    act_space = env.action_space
    print(f"{name:30s}  obs: {str(obs_space):35s}  act: {str(act_space)}")
    env.close()

输出大致如下（各版本可能略有差异）：

CartPole-v1                      obs: Box(4,)                          act: Discrete(2)
MountainCar-v0                   obs: Box(2,)                          act: Discrete(3)
LunarLander-v2                   obs: Box(8,)                          act: Discrete(4)
HalfCheetah-v4                   obs: Box(17,)                         act: Box(6,)
ALE/Breakout-v5                  obs: Box(210, 160, 3)                 act: Discrete(4)

动手试试：建议你把上面这些环境逐个创建出来，随便跑几步看看 obs 的形状和数值范围。这种"亲手摸一摸"的体验比看文档管用十倍。你会发现，HalfCheetah 的观测里大部分值都在 -1 到 1 之间，而 CartPole 的位置和速度的数值范围完全不同——这对后续做观测归一化是个重要线索。

23.3 自定义环境

内置环境终归是别人设计好的。真正的强化学习项目里，你八成需要自己定义环境。好消息是，Gymnasium 让你可以轻松地继承 gym.Env 来实现自己的环境。

一个自定义环境需要实现以下方法：

  • __init__(self)：定义观测空间 self.observation_space 和动作空间 self.action_space，调用 super().__init__()。
  • reset(self, seed=None, options=None)：重置环境到初始状态，返回 (obs, info)。
  • step(self, action)：执行一步，返回 (obs, reward, terminated, truncated, info)。
  • render(self)（可选）：渲染当前环境状态。Gymnasium 的新版渲染推荐用 render_mode 参数在 __init__ 中指定。

下面是一个完整的 GridWorld 自定义环境示例。智能体在 n×n 网格中移动，从起点走到目标点并避开障碍物：

import gymnasium as gym
from gymnasium import spaces
import numpy as np

class GridWorldEnv(gym.Env):
    """一个简单的网格世界环境。

    智能体在 n×n 网格中移动，目标是到达终点 (goal)，
    同时避开障碍物 (obstacles)。每走一步奖励 -1，
    到达终点奖励 +50，撞到障碍物奖励 -10 并终止。
    """

    def __init__(self, size=5, start=(0, 0), goal=(4, 4),
                 obstacles=None, render_mode=None):
        super().__init__()

        self.size = size          # 网格大小 n×n
        self.start = np.array(start, dtype=np.int32)
        self.goal = np.array(goal, dtype=np.int32)
        self.obstacles = obstacles if obstacles else []
        self.render_mode = render_mode

        # 观测：智能体的 (x, y) 坐标，范围 [0, size-1]
        self.observation_space = spaces.Box(
            low=0, high=size - 1, shape=(2,), dtype=np.int32
        )

        # 动作：0=上, 1=右, 2=下, 3=左
        self.action_space = spaces.Discrete(4)

        # 动作到坐标偏移的映射
        self._action_to_delta = {
            0: np.array([-1, 0]),   # 上
            1: np.array([0, 1]),    # 右
            2: np.array([1, 0]),    # 下
            3: np.array([0, -1]),   # 左
        }

        self._agent = None  # 智能体当前位置

    def _is_obstacle(self, pos):
        """检查 pos 是否是障碍物"""
        return any(np.array_equal(pos, obs) for obs in self.obstacles)

    def reset(self, seed=None, options=None):
        super().reset(seed=seed)
        self._agent = self.start.copy()
        obs = self._agent.copy()

        info = {"distance": np.linalg.norm(self._agent - self.goal)}

        if self.render_mode == "human":
            self._render_frame()

        return obs, info

    def step(self, action):
        # 计算新位置
        delta = self._action_to_delta[action]
        new_pos = self._agent + delta

        # 边界约束：不能超出网格
        new_pos = np.clip(new_pos, 0, self.size - 1)

        # 检查是否撞到障碍物
        if self._is_obstacle(new_pos):
            self._agent = new_pos
            terminated = True
            truncated = False
            reward = -10.0
            info = {"cause": "hit obstacle"}
            obs = self._agent.copy()
            return obs, reward, terminated, truncated, info

        self._agent = new_pos
        obs = self._agent.copy()

        # 检查是否到达终点
        if np.array_equal(self._agent, self.goal):
            terminated = True
            reward = 50.0
            info = {"cause": "reached goal"}
        else:
            terminated = False
            reward = -1.0  # 每步的"时间成本"

        truncated = False
        info["distance"] = np.linalg.norm(self._agent - self.goal)

        if self.render_mode == "human":
            self._render_frame()

        return obs, reward, terminated, truncated, info

    def render(self):
        if self.render_mode == "rgb_array":
            return self._render_frame()

    def _render_frame(self):
        """用字符画打印网格"""
        grid = np.full((self.size, self.size), ".", dtype=str)
        for obs in self.obstacles:
            grid[obs[0], obs[1]] = "X"
        grid[self.goal[0], self.goal[1]] = "G"
        grid[self._agent[0], self._agent[1]] = "A"

        print("-" * (self.size * 2 + 1))
        for row in grid:
            print("|" + " ".join(row) + "|")
        print("-" * (self.size * 2 + 1))


# ========== 使用示例 ==========
if __name__ == "__main__":
    obstacles = [(1, 1), (2, 2), (3, 1)]
    env = GridWorldEnv(size=5, start=(0, 0), goal=(4, 4),
                       obstacles=obstacles, render_mode="human")

    obs, info = env.reset()
    print(f"初始位置: {obs}")

    # 手动走几步看看
    actions = [1, 1, 1, 1, 2, 2, 2, 2]  # 右右右右 下下下下
    for a in actions:
        obs, reward, terminated, truncated, info = env.step(a)
        print(f"动作: {a}, 位置: {obs}, 奖励: {reward}, 终止: {terminated}")
        if terminated:
            break

    env.close()

踩坑故事——Step 返回值的"五元组陷阱"

有一回我带学生做课程项目，一个组用 Gymnasium 写的代码，另一个组用旧版 Gym 写的。两个组的代码要合并，结果跑了半天全是莫名其妙的问题。最后排查了两天才发现：A 组代码用 obs, reward, done, info = env.step(action)（旧版解包），B 组用 obs, reward, terminated, truncated, info = env.step(action)（新版解包），两个一合并，done 变量拿到的其实是 truncated 的值！

更隐蔽的是：旧版 done 和新版 terminated 的含义不完全等价——在 Gymnasium 中，terminated 只是"真的终止"（比如撞墙、杆子倒了），而旧版 done 包含 terminated or truncated。如果你用旧版代码写了 if done: env.reset()，在 Gymnasium 里应该写成 if terminated or truncated: env.reset() 才不会漏掉截断的情况。

教训：一个项目组永远统一用同一个版本的 Gymnasium。如果你接手旧代码，第一件事就是全局搜索 done，全部改成 terminated, truncated 逻辑。这看起来是小事，但踩一次坑等于浪费一周时间。

写自定义环境时，有几点需要特别注意：

  • 观测空间和动作空间的边界要合理——如果你的观测是图像，用 spaces.Box(low=0, high=255, shape=(H,W,3), dtype=np.uint8)，千万别用 float32 浪费内存。
  • 种子要传下去——在 reset 中调用 super().reset(seed=seed)，这样 env.action_space.seed() 才能和全局种子一致，实验结果才可复现。
  • Terminated 和 Truncated 要分清楚——"真的赢了/输了"是 terminated，"到了最大步数/timeout"是 truncated。如果你不区分，某些算法（如 GAIL 中的 discriminator 计算）会出错。

23.4 环境包装器（Wrappers）

现实中的强化学习从来不是"创建环境 -> 直接开跑"这么简单。你经常需要对环境做一些预处理，比如：把图像缩放到 84×84、累积最近 4 帧、对奖励做缩放、记录训练日志……如果每次都要修改环境本身的代码，那就乱套了。

Wrapper（包装器）就是解决这个问题的设计模式。它使用装饰器模式，在不修改环境本身的前提下，在 reset 和 step 之间"插入"自定义逻辑。Gymnasium 提供了三个基类包装器：

  • ObservationWrapper：改写 observation(obs) 方法，对观测做变换。
  • ActionWrapper：改写 action(act) 方法，对智能体输出的动作做变换。
  • RewardWrapper：改写 reward(rew) 方法，对环境返回的奖励做变换。

来看一个实际的例子。假设我们希望：1) 奖励乘以一个缩放因子（调学习率时常用）；2) 把观测从 (x, y) 坐标归一化到 [0, 1] 区间：

import gymnasium as gym
from gymnasium import RewardWrapper, ObservationWrapper
import numpy as np

class ScaledRewardWrapper(RewardWrapper):
    """将奖励乘以缩放因子"""
    def __init__(self, env, scale=0.1):
        super().__init__(env)
        self.scale = scale

    def reward(self, reward):
        return reward * self.scale


class NormalizedObsWrapper(ObservationWrapper):
    """将观测归一化到 [0, 1] 区间"""
    def __init__(self, env):
        super().__init__(env)
        low = env.observation_space.low
        high = env.observation_space.high
        self._range = high - low
        # 避免除零
        self._range[self._range == 0] = 1.0

    def observation(self, obs):
        return (obs - self.env.observation_space.low) / self._range


# 使用示例：包装我们的 GridWorld
base_env = GridWorldEnv(size=5)
wrapped_env = ScaledRewardWrapper(base_env, scale=0.5)
# 现在所有奖励都被减半了

# 也可以链式包装
env = gym.make("CartPole-v1")
env = NormalizedObsWrapper(env)
env = ScaledRewardWrapper(env, scale=0.1)

obs, info = env.reset()
# obs 现在在 [0, 1] 范围内，奖励被缩放到原来的 1/10

Gymnasium 还提供了一些非常实用的内置包装器：

包装器 作用 典型用途
RecordVideo 将 episode 渲染成视频文件 训练完成后录制智能体表现，用于展示和日志存档
RecordEpisodeStatistics 自动记录每 episode 的总奖励和长度 训练时监控奖励曲线，比手动计算更方便
TimeLimit 限制 episode 的最大步数，超时触发 truncated 几乎所有环境都需要，防止死循环
NormalizeObservation 在线统计观测的均值和方差并做标准化 训练时防止观测值范围差异过大
NormalizeReward 在线统计奖励的均值和方差并做标准化 稳定训练，特别是奖励尺度差异大时
FrameStack 堆积最近 k 帧作为观测（Atari 必备） 给智能体提供运动信息，单帧无法感知速度和方向
GrayScaleObservation 将 RGB 图像转为灰度图 降低 Atari 输入维度，加速训练
ResizeObservation 将图像缩放到指定尺寸 Atari 从 210×160 缩放到 84×84

一个典型的 Atari 环境预处理链是这样的：

import gymnasium as gym
from gymnasium.wrappers import ResizeObservation, GrayScaleObservation, FrameStack

env = gym.make("ALE/Breakout-v5", render_mode="rgb_array")
env = ResizeObservation(env, (84, 84))        # 缩放
env = GrayScaleObservation(env)               # 转灰度
env = FrameStack(env, 4)                      # 堆积 4 帧

obs, info = env.reset()
print(obs.shape)  # (4, 84, 84)  — 4 张灰度图

# 现在 obs 就是 Atari DQN 论文里用的标准输入格式

包装器的哲学：不要修改环境本身，而是用包装器堆叠出你需要的功能。这样每个组件职责单一，可以自由组合，而且可以随时"取下"某个包装器来调试。如果以后 Gymnasium 升级了，你也只需要确认包装器接口是否兼容——不用重写环境代码。

23.5 环境选择指南

面对这么多环境，初学者最容易犯的错误就是"一上来就挑最难的环境"。比如刚学会 DQN 就打开 Atari Breakout，结果训练三天不收敛，信心全无。

我是这样建议选择的：

你的目标 推荐环境 理由
验证算法是否写对了 CartPole-v1 最简单的环境，随机策略奖励约 20，DQN 几分钟内应达到 500（满分）。如果 CartPole 都跑不好，别急着换复杂环境。
测试离散动作空间算法 LunarLander-v2 8 维观测、4 个动作，难度适中。随机策略约 -100，DQN 应能稳定到 200+。奖励设计丰富，能测试算法对稠密和稀疏奖励的适应能力。
验证连续动作空间算法 HalfCheetah-v4 或 Hopper-v4 标准 MuJoCo 环境。SAC/TD3 在这个环境上有公认的 benchmark 曲线，方便你对照结果。
研究基于像素的深度 RL ALE/Pong-v5（先）→ Breakout-v5 Pong 是 Atari 里最简单的之一，动作只有上下两个方向。从 Pong 开始验证 CNN 策略网络，再过渡到 Breakout 等更复杂的游戏。
多智能体/元学习/课程学习实验 自定义环境 内置环境无法满足特定需求时，用我们 §23.3 的方法自己写一个。很多多智能体研究用的是自制的粒子世界或网格世界。
快速调试算法超参数 Acrobot-v1 2 秒跑完一轮评估，适合做大量消融实验。状态维度适中，能暴露算法稳定性问题。

另外还有一些重要的注意事项：

  • 版本后缀很重要——CartPole-v1 和 CartPole-v0 的最大步数不同，HalfCheetah-v3 和 HalfCheetah-v4 的奖励函数也有微调。用最新版本总没错。
  • Atari 的版本选择——推荐用 ALE/环境名-v5 系列，它是最新维护的版本，支持 render_mode 参数。旧版 环境名NoFrameskip-v4 仍然能用，但更推荐迁移到 v5。
  • 不要忽视 "v0" 和 "v1" 的差异——哪怕是同一套环境名，不同版本的奖励函数、最大步数、甚至物理参数都可能不同。跑论文复现时一定要注意版本号匹配。

学习路线建议：如果你刚开始学 RL 编程，我推荐你按这个顺序练习：(1) CartPole 跑 DQN 拿到满分；(2) LunarLander 跑 DQN 看到稳定收敛曲线；(3) HalfCheetah 跑 SAC 看到四足机器人学会奔跑；(4) 自定义一个自己的 GridWorld 环境，跑 PPO。每一步都会让你更深入理解 Gymnasium 的设计哲学。

学完本章，你应该已经掌握了 Gymnasium 的核心用法。但手动实现 DQN、PPO 这些算法——即使是调包装器——也还是有不少重复工作。好消息是，下一章我们介绍的 Stable-Baselines3（SB3） 把这些算法的工业级实现都打包好了，你只需要三行代码就能跑 PPO。而 Gymnasium 学到的所有知识——空间定义、包装器、环境创建——在 SB3 里全部直接适用。
