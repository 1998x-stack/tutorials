第八章：连接管理

各位同学好，欢迎来到第八章。前面七章我们主要讨论了HTTP协议的应用层特性——报文结构、方法、状态码、头部字段、内容协商。现在我们要往下走一层，看看HTTP和底层TCP连接之间的关系。坦白说，连接管理是Web性能优化的第一块基石——不懂连接，谈性能就是空中楼阁。

你可能有过这种体验：明明网速很快，加载一个页面却要好几秒。问题往往出在连接建立上，而不仅仅是带宽。今天我们就把HTTP的连接管理彻底搞明白。

8.1 短连接与长连接

在HTTP的早期版本（0.9和1.0）中，每个HTTP请求都要创建一个新的TCP连接。流程是这样的：

  1. 浏览器建立TCP连接（三次握手）
  2. 发送HTTP请求
  3. 服务器返回HTTP响应
  4. 关闭TCP连接（四次挥手）

你看，一个请求就要经历一次完整的TCP生命周期。如果网页有10个资源（CSS、JS、图片等），那就得建10次TCP连接。这在1996年还行，因为那时候一个网页也就三五个资源。但放到今天——一个电商首页轻松上百个请求——这种模式就完全不可接受了。

HTTP/1.1引入了持久连接（Persistent Connections），也叫长连接。它允许在同一个TCP连接上发送多个HTTP请求和响应，连接不会在每个请求结束后立即关闭。浏览器在连接空闲一段时间后才会关闭它，或者当连接数达到上限时才回收。

核心数据：建立一次TCP连接的开销大约是1-3个RTT（Round-Trip Time）。如果客户端到服务器的物理距离是200ms延迟（比如从中国到美国西海岸），三次握手就花掉至少200ms。短连接下每个资源都要额外等这个时间，而长连接只需在第一次请求时付出这个代价。

8.2 Keep-Alive机制

持久连接是通过 Connection: keep-alive 头实现的。在HTTP/1.0中，你需要显式发送这个头来启用长连接。到了HTTP/1.1，持久连接变成了默认行为，除非你发送 Connection: close 明确要求关闭。

服务器端通常可以配置keep-alive的参数：

# Nginx配置示例
keepalive_timeout  65;  # 连接空闲65秒后关闭
keepalive_requests 100; # 一个连接最多处理100个请求

浏览器的6连接限制——这是一个真实存在的"潜规则"。HTTP/1.1规范建议每个客户端对同一个域名建立最多2个持久连接。但所有主流浏览器都无视了这条建议，统一将限制设为6个。这意味着什么？Chrome打开一个页面，对同一个域名（比如 assets.example.com）最多同时发起6个TCP连接，每个连接上可以串行发送多个请求。

踩坑故事：有次做一个大流量图片站，发现页面加载特别慢。查了半天，发现页面里有超过30张图片，全部来自同一个CDN域名。浏览器一次只能开6个连接，每个连接串行拉取图片。30张图片要串好几轮才下完。解决方案很简单——把图片分散到两个域名（图片分片，Domain Sharding），连接数上限变成12，加载速度快了一倍。后来HTTP/2的多路复用彻底解决了这个问题，一个连接就够了。

版本连接策略每域名连接数连接复用方式
HTTP/1.0短连接（默认）无限制（每个请求建新连接）无
HTTP/1.1长连接（默认）浏览器限制6个串行复用
HTTP/2多路复用1个连接并行复用

8.3 HTTP管线化

在讲管线化（Pipelining）之前，我们先理解一下HTTP/1.1的基本工作模式——请求-应答。在同一个连接上，浏览器必须先发一个请求，等收到响应后才能发下一个请求。这叫串行，或者叫"停等协议"。

管线化的想法是：不等响应，连续发送多个请求。看起来很美对吧？但现实很骨感。

管线化的核心问题在于队头阻塞（Head-of-Line Blocking）：如果连续发送了请求A、B、C，服务器的处理顺序也必须是A→B→C。但如果A处理得特别慢（比如A是一个数据库慢查询），B和C的响应即使已经准备好了，也必须排队等A处理完才能发送。这就跟只有一个收银台的超市一样——前面的人买东西磨磨蹭蹭，后面的人只能干等。

为什么浏览器禁用了管线化？ 因为管线化在实际部署中问题太多了：1）队头阻塞导致整体延迟更差；2）很多代理服务器和中间设备不支持或实现有bug；3）如果连接中途断开，所有已发送但未收到响应的请求必须重发。Chrome在2012年尝试过启用管线化，发现延迟没有改善反而恶化，最后在2015年彻底禁用了。直到HTTP/2的多路复用，才算真正解决了这个问题。

8.4 TCP连接开销

很多人低估了建立TCP连接的实际开销。我们拆开来看：

三次握手：

  1. 客户端发SYN（同步序列号）→ 服务端
  2. 服务端回SYN-ACK → 客户端
  3. 客户端发ACK → 服务端

这需要1.5个RTT。假设客户端在上海，服务器在北京，网络延迟约30ms，那就是45ms。

TLS握手： 在HTTPS连接中，TCP握手之后还要进行TLS握手。TLS 1.3需要1个RTT，TLS 1.2需要2个RTT。加上TCP的1.5个RTT，一个HTTPS请求在数据开始传输前已经花掉了2.5到3.5个RTT。

慢启动（Slow Start）： TCP连接建立后，不会立刻使用全部带宽。TCP拥塞控制算法会从较小的拥塞窗口（通常10个TCP段）开始，每收到一个ACK就增加窗口大小。这意味着前几个HTTP请求的数据包传输速度远低于带宽上限。一个典型的新连接需要经过几个往返后才能达到全速。

真实数据：一个从中国访问美国西海岸的HTTPS请求，TCP+TLS握手大约需要200-300ms。加上慢启动的影响，一个10KB的响应可能要400-500ms才能完整下载。而如果复用已有的连接，完全绕过了握手和慢启动，同样的请求可能只需100ms。这就是为什么连接复用如此重要的原因。

8.5 连接池实战

理解了连接的开销后，我们来看看在实际系统中如何管理连接。

浏览器端的连接池：

浏览器的连接池是自动管理的，你不需要写任何代码。但了解它的行为对性能优化很有帮助：

  • 每个域名最多6个并发连接（HTTP/1.1）
  • 连接空闲一段时间后自动关闭（通常是60-300秒）
  • 连接出错时会尝试重试（通常最多3次）
  • 用完的连接放回池中供后续请求复用

服务端Nginx配置最佳实践：

http {
    # 上游（后端应用服务器）的keep-alive配置
    upstream backend {
        server 127.0.0.1:8080;
        keepalive 32;  # 保留32个空闲连接
    }

    server {
        listen 80;
        # 客户端的keep-alive配置
        keepalive_timeout  65s;
        keepalive_requests 1000;

        location / {
            proxy_http_version 1.1;
            proxy_set_header Connection "";
            proxy_pass http://backend;
        }
    }
}

这段配置的几个关键点：

  • keepalive 32：Nginx会保持32个到后端服务器的空闲连接，避免每次请求都重新建连
  • proxy_http_version 1.1 + proxy_set_header Connection ""：让代理也使用持久连接，不走短连接
  • keepalive_requests 1000：一个连接最多服务1000个请求后关闭，防止内存泄漏累积

后端编程语言的连接池：

如果你用Java的HttpClient、Python的requests（使用Session）、Node.js的http.Agent，底层都有连接池管理。关键配置参数：

  • 最大连接数（maxConnections）：限制总连接数，防止资源耗尽
  • 每路由最大连接数（maxConnectionsPerRoute）：限制对单个后端的连接数
  • 空闲超时（idleTimeout）：连接空闲多久后关闭
  • 连接存活时间（ttl）：连接最多存活多久，不管是否空闲

踩坑故事：之前维护过一个Java微服务，A服务调用B服务时经常报"连接超时"。查了日志发现B服务本身很快（平均10ms），但A服务的HTTP连接池把maxConnectionsPerRoute设成了2。当并发超过2时，新的请求就要排队等空闲连接。后来升级到20，并发能力直接提升了10倍。所以排查"慢"的问题时，如果你发现后端响应快但客户端调用慢，第一反应应该是——连接池满了。

好了，第八章的内容就到这里。我们从短连接的历史讲到Keep-Alive和管线化的实践，再到TCP连接开销的细节分析。理解连接管理是理解HTTP性能的关键——无论是浏览器加载页面还是微服务间调用，连接复用的设计直接影响整体效率。下一章我们进入另一个对性能至关重要的主题——HTTP缓存。
