第20章 自定义层与算子

大家好，我是你们的讲师。在上一章我们学会了用 torch.compile 榨取性能，但你有没有想过这样一个问题：当你需要实现一个PyTorch没有内置的操作时该怎么办？或者内置的实现对你来说不够快、显存占用太高？这时候，你就需要掌握自定义层和算子的能力了。

自定义层和算子是区分"PyTorch用户"和"PyTorch高级用户"的分水岭。前者只会组合已有的积木，后者能自己造积木。这一章我们从最简单的 nn.Module 自定义开始，一步步深入到 autograd.Function 的反向传播自定义，最后到C++/CUDA扩展的编写。

20.1 自定义 nn.Module 层

大部分时候，你不需要从零写一个算子。PyTorch提供了上百个基础操作，你只需要把它们组合起来。自定义 nn.Module 就是这种组合的正式方式。

import torch
import torch.nn as nn

class MyDenseLayer(nn.Module):
    """一个带LayerNorm和Dropout的自定义全连接层"""
    def __init__(self, in_features, out_features, dropout=0.1):
        super().__init__()
        self.linear = nn.Linear(in_features, out_features)
        self.layernorm = nn.LayerNorm(out_features)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x):
        x = self.linear(x)
        x = self.layernorm(x)
        x = torch.relu(x)
        return self.dropout(x)

什么时候该写自定义 nn.Module？总结三条：

  • 你需要将一组操作打包成一个逻辑单元（比如上面的"线性+归一化+激活+Dropout"）。
  • 你需要在 forward 中加入非标准的控制流（比如条件分支，但要小心graph break）。
  • 你需要在 __init__ 中注册可学习参数（nn.Parameter）。
小贴士：尽量用现有op组合来实现新功能。PyTorch的aten操作经过了充分优化，组合的开销通常远小于自己写一个naive实现。只有在确定组合实现确实成为瓶颈时，才考虑走下面的路。

20.2 自定义反向传播：autograd.Function

当你需要自定义反向传播逻辑时——比如你想实现一个内存效率更高的梯度计算，或者在forward中用了某个不支持自动求导的外部操作——就需要使用 torch.autograd.Function。

下面是一个自定义ReLU的实现，虽然PyTorch有内置的，但用来演示 autograd.Function 的接口非常清晰：

import torch

class MyReLU(torch.autograd.Function):
    @staticmethod
    def forward(ctx, input):
        """前向传播：计算ReLU，保存mask用于反向"""
        ctx.save_for_backward(input)
        return input.clamp(min=0)

    @staticmethod
    def backward(ctx, grad_output):
        """反向传播：mask掉梯度中对应输入为负的位置"""
        input, = ctx.saved_tensors
        grad_input = grad_output.clone()
        grad_input[input < 0] = 0
        return grad_input

# 使用方式
input = torch.randn(4, 4, requires_grad=True)
output = MyReLU.apply(input)  # 注意是 .apply 不是直接调用
output.sum().backward()

ctx 对象（context）至关重要：

  • ctx.save_for_backward() 保存反向传播需要的中间张量。
  • ctx.mark_non_differentiable() 标记某些输出不需要梯度。
  • ctx.set_materialize_grads(False) 在反向传播中节省显存（高级用法）。

什么时候需要自定义backward？常见场景包括：

  • 显存优化：forward时丢弃某些中间结果，backward时重新计算（时间换空间）。这在训练超大模型时非常关键。
  • 数值技巧：重新排列梯度计算顺序以获得更好的数值稳定性。
  • 不可微操作：在forward中使用了采样等不可微操作，需要在backward中用替代梯度（如Straight-Through Estimator）。

20.3 C++/CUDA 扩展

当纯Python实现无法满足性能要求时，就需要用C++或CUDA来写算子了。PyTorch提供了 torch.utils.cpp_extension 让这个过程变得相当简单。

最轻量的方式是使用 load_inline——把C++/CUDA源码以字符串形式嵌入Python脚本，即时编译：

from torch.utils.cpp_extension import load_inline

# C++源码
cpp_source = """
#include <torch/extension.h>

torch::Tensor custom_add(torch::Tensor a, torch::Tensor b) {
    return a + b;
}
"""

# 编译并加载
custom_ops = load_inline(
    name="custom_ops",
    cpp_sources=[cpp_source],
    functions=["custom_add"],
    verbose=True
)

# 使用
result = custom_ops.custom_add(a, b)

对于需要GPU加速的场景，可以写CUDA kernel并通过 load_inline 的 cuda_sources 参数传入：

cuda_source = """
#include <torch/extension.h>

__global__ void vec_add_kernel(float* out, const float* a, const float* b, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) out[idx] = a[idx] + b[idx];
}

torch::Tensor vec_add_cuda(torch::Tensor a, torch::Tensor b) {
    auto out = torch::zeros_like(a);
    int n = a.numel();
    vec_add_kernel<<<(n + 255) / 256, 256>>>(
        out.data_ptr<float>(), a.data_ptr<float>(), b.data_ptr<float>(), n
    );
    return out;
}
"""
方法 开发成本 性能上限 适用场景
纯Python组合（nn.Module） 低 中等 大部分日常需求
autograd.Function 中等 高（可优化显存） 需要自定义梯度的场景
load_inline C++ 扩展 中高 高 CPU算子融合，调用第三方C库
完整CUDA kernel 高 最高 高性能GPU算子，融合kernel

20.4 让自定义算子兼容 torch.compile

如果你的自定义算子要和 torch.compile 配合使用，需要做一些额外的工作。默认情况下，TorchDynamo不认识你的自定义操作，会直接产生graph break。为了让编译图认识它们，需要注册fake tensor实现和autograd规则：

import torch.library

# 注册一个fake tensor实现（告诉编译器这个操作的输出shape/type）
@torch.library.register_fake("myops::custom_relu")
def _(input):
    return input.new_empty(input.shape)  # 输出shape与输入相同

# 注册autograd规则
@torch.library.register_autograd("myops::custom_relu", backward_op="myops::custom_relu_backward")
def _(ctx, input):
    ctx.save_for_backward(input)
    return input.clamp(min=0)

有了这些注册信息，torch.compile 就能在你的自定义算子周围正确构建计算图，而不会触发graph break。这样你既能享受自定义算子的性能优势，又能享受 torch.compile 的全局优化。

踩坑故事：之前有个团队做推荐系统的特征交叉模块，发现PyTorch的EmbeddingBag实现不够快。他们写了一个自定义CUDA kernel，把多个特征查找和求和融合在一起，速度直接提升了10倍！但这个kernel没有注册fake tensor实现，导致一用 torch.compile 就graph break。最后加上 register_fake 和 register_autograd 后，既拿到了自定义kernel的10x加速，又享受了编译图的额外优化，整体加速比达到了12x。这个故事的教训是：自定义算子不是最终答案，和编译框架的配合才是。
