第2章 环境搭建与逆向工具链

大家好，我是你们的讲师。在上一章我们聊了很多理论——Agent 范式、Harness 概念、行业趋势。但说句实话，看别人拆过的代码和自己亲手拆一次，完全是两码事。

这一章的定位很明确：教会你如何安全、高效地"解剖"Claude Code。我们会从安装开始，到格式化混淆源码，到用 Chrome DevTools 动态调试，再到用 Monkey Patch 在运行时拦截 API 请求。学完这一章，你将拥有一个完整的逆向工具箱——我们之后每一章的"源码拆解"内容，都建立在你已经掌握本章工具链的假设之上。

顺便说一句，我当年第一次尝试拆解一个 Node.js 闭源工具时，天真地以为直接扔进 IDE 就能看懂。结果打开一个 8 万行的 bundle 文件，所有变量名都是 _0x3f2a8b 这种格式，我看了一个小时连入口在哪都没找到。从那以后，我学会了——逆向的第一个原则是：不要硬看，用工具。

安装 Claude Code 与开发环境准备

首先，你需要安装 Claude Code 本身。如果你已经在用，可以跳过安装步骤直接看后面的逆向部分。安装命令很简单：

npm install -g @anthropic-ai/claude-code

安装完成后，运行 claude 或 claude --help 确认安装成功。接下来，我们需要找到 Claude Code 的安装路径。Node.js 全局包通常安装在：

# 查看 claude 命令的真实路径
which claude
# 输出示例: /usr/local/bin/claude

# 跟进符号链接找到真实文件
ls -la $(which claude)
# 你会看到它指向一个 shim 脚本或 .js 文件

Claude Code 的代码是经过混淆（obfuscated）的——Anthropic 使用了商业级的 JavaScript 混淆器对源码进行了处理。但这并不意味着我们完全无法阅读。你找到的 cli.js 文件是一个打包好的 bundle，包含了几乎所有核心逻辑。后续的静态分析就是围绕这个文件展开的。

接下来安装我们的主力逆向工具——js-beautify。这个美化成千上万的 JavaScript 开发者每天都在用的工具，是我们对付混淆代码的第一利器：

npm install -g js-beautify

最后，建议你配置一个专门的 VS Code 工作区：

  • 将 Claude Code 的安装目录添加为工作区根目录之一
  • 安装 Prettier 插件（作为 js-beautify 的辅助）
  • 启用"Large File"支持（VS Code 默认对大文件有高亮限制，需要调整）
  • 建议关闭自动保存——因为我们后面可能会修改 cli.js 来注入调试代码
踩坑提示：Claude Code 的 cli.js 通常超过 5MB。VS Code 在某些版本中会对超过 5MB 的文件禁用语法高亮和格式化。如果你发现格式化没反应，先确认文件大小——有时需要多用几次 js-beautify 才行，或者直接在工作区设置中调高 editor.largeFile 阈值。

静态分析：格式化与阅读混淆代码

静态分析是我们最常用的逆向方法——不上手运行，只靠读代码来理解结构。对于 Claude Code 这样的大型 bundle 来说，静态分析的主要任务是定位关键模块的入口。

第一步，格式化原始文件：

js-beautify cli.js -o cli.beautified.js

这个命令会输出一个格式化的版本（通常体积膨胀 20-30%，但可读性提升 10 倍）。注意 -o 参数直接写入新文件，不会修改原文件。

第二步，搜索关键字符串来定位核心模块。混淆代码虽然变量名被替换了，但字符串常量（API endpoints、错误消息、函数名等）通常会保留，因为它们是运行时需要的：

# 搜索 API 端点
grep -n "api.anthropic.com" cli.beautified.js

# 搜索 Tool Use 相关代码
grep -n "tool_use" cli.beautified.js

# 搜索 stop_reason 处理逻辑
grep -n "stop_reason" cli.beautified.js

# 搜索系统提示词构造
grep -n "system_prompt" cli.beautified.js

每个搜索结果会告诉你核心逻辑在什么位置附近。例如，"api.anthropic.com" 会出现在 HTTP 客户端初始化和请求发送两个地方；"tool_use" 会出现在响应解析和工具调度两个模块。通过交叉引用这些字符串的位置，你可以逐步还原出代码的模块边界。

第三步，理解文件结构。格式化后的 cli.beautified.js 通常遵循这样的模式：

  • 开头 1000-3000 行：运行时 polyfill 和助手函数
  • 中间部分：核心业务逻辑（工具定义、LLM 调用、状态管理）
  • 末尾：入口函数和 CLI 参数解析
  • 散布的 IIFE（立即执行函数表达式）：每个 IIFE 通常对应一个模块

静态分析的局限在于：你看不到运行时状态。变量在混淆后被赋予了无意义的名称，很多逻辑只有在实际运行时才能理解其意图。所以我们还需要动态调试。

动态调试：Chrome DevTools 连接

当你需要理解一个复杂逻辑在运行时到底走了哪条分支时，静态分析就力不从心了。这时候需要上动态调试——给 Node.js 进程挂一个调试器。

Node.js 内置了 V8 Inspector 协议，通过 --inspect 或 --inspect-brk 标志启动：

# --inspect-brk 会在第一行代码处暂停，等待调试器连接
NODE_OPTIONS="--inspect-brk=9229" claude

然后打开 Chrome 浏览器，地址栏输入 chrome://inspect，你会看到 "Remote Target" 列表中出现了你的 Node.js 进程。点击 "inspect" 即可打开 DevTools 调试界面。

--inspect-brk 和 --inspect 的区别：前者在第一行代码处暂停（适合从入口开始跟踪），后者不暂停（适合在你已经知道断点位置后使用）。

有了 DevTools 界面，你可以：

  • 设置断点：在格式化后的源码中点击行号，程序执行到该行时会暂停
  • 观察调用栈：查看当前函数是谁调用的
  • 查看作用域变量：即使变量名是混淆的，你也能看到它们的实际值
  • 条件断点：在特定条件下暂停，例如当 stop_reason === "tool_use" 时

最常用的断点位置包括：HTTP 请求发送前（拦截请求体）、HTTP 响应回来后（解析响应体）、工具调用分发处、错误处理分支。这些是理解 Agent 工作流的关键节点。

踩坑实录：我第一次用 DevTools 调试 Claude Code 时遇到了一个问题——DevTools 在加载 5MB 以上的 JavaScript 文件时会非常卡，甚至经常崩溃。原因在于 DevTools 的解析器会尝试对混淆代码进行语法高亮和格式化，这对大文件是灾难。解决方案是：先用 js-beautify 格式化代码，然后在 DevTools 中直接打开格式化后的版本（在 Sources 面板中切换到 cli.beautified.js）。但仍然需要注意，每次 Claude Code 版本更新后文件内容会变，需要重新格式化和重新定位断点。

运行时拦截：Monkey Patch API 请求

静态分析和动态调试都是"只读"操作。有时你还需要"写入"——即修改 Claude Code 的运行时行为，拦截其 API 请求和响应，记录下完整的交互数据。这对于理解 Tool Use 协议的细节特别有用。

Monkey Patch（猴子补丁）的思路很简单：在 cli.js 中找到发起 HTTP 请求的函数，用我们的函数将其替换，在调用原始函数之前/之后插入日志记录逻辑。

下面是一个典型的 Monkey Patch 注入脚本，用于拦截 Anthropic API 调用：

// monkey-patch.js — 注入到 claude 进程
// 用法: node -r ./monkey-patch.js $(which claude)

const originalFetch = globalThis.fetch;

globalThis.fetch = async function patchedFetch(url, options) {
    // 只拦截 Anthropic API 的请求
    if (typeof url === 'string' && url.includes('api.anthropic.com')) {
        const body = options?.body ? JSON.parse(options.body) : null;

        console.error('\n[Monkey Patch] === API Request ===');
        console.error('[Monkey Patch] URL:', url);
        if (body) {
            console.error('[Monkey Patch] Model:', body.model);
            console.error('[Monkey Patch] Tools count:', body.tools?.length ?? 0);
            console.error('[Monkey Patch] Messages count:', body.messages?.length ?? 0);
            console.error('[Monkey Patch] System length:', body.system?.length ?? 0);
        }

        const startTime = Date.now();
        const response = await originalFetch(url, options);
        const elapsed = Date.now() - startTime;

        // 克隆 response 以便读取 body（response 是 single-use stream）
        const cloned = response.clone();
        const responseBody = await cloned.json();

        console.error('[Monkey Patch] === API Response ===');
        console.error('[Monkey Patch] Elapsed:', elapsed + 'ms');
        console.error('[Monkey Patch] Stop reason:', responseBody.stop_reason);
        console.error('[Monkey Patch] Usage:', JSON.stringify(responseBody.usage));

        // 记录到文件
        const fs = require('fs');
        fs.appendFileSync('messages.log', JSON.stringify({
            timestamp: new Date().toISOString(),
            request: body,
            response: responseBody,
            elapsed
        }) + '\n');

        return response;
    }

    return originalFetch(url, options);
};

console.error('[Monkey Patch] Loaded — intercepting Anthropic API calls');

使用方式：在终端中运行 node -r ./monkey-patch.js $(which claude)。这里的 -r 标志告诉 Node.js 在加载主模块之前先加载并执行我们的补丁脚本。一旦脚本运行，所有通过 globalThis.fetch 发出的 HTTP 请求都会被拦截。

拦截到的数据会以 JSONL 格式写入 messages.log 文件——每一行是一个完整的请求-响应周期。之后我们可以用任何 JSON 处理工具来分析这些数据。

社区中已经有了基于这种思路的图形化工具——claude-code-reverse（GitHub 上可搜索到）。它提供了一个 Web UI 来查看拦截到的 API 调用，包括请求体、响应体、Token 消耗的可视化。如果你不想自己写 Monkey Patch 脚本，可以直接使用这个工具。

三种逆向方法对比

下面这个表格总结了三种逆向方法的特点，你可以根据当前的分析目标选择最合适的工具：

方法 主要工具 优势 局限 适用场景
静态分析（格式化 + 搜索） js-beautify, grep, VS Code 不运行程序，安全无副作用；适合全局理解 混淆变量名难以追踪；看不到运行时值 定位模块边界、搜索关键功能入口
动态调试（DevTools） Chrome DevTools, --inspect 能看到运行时状态和调用栈；可设条件断点 大文件卡顿；需手动操作浏览器 追踪特定逻辑分支、理解运行时数据流
运行时拦截（Monkey Patch） -r 注入, fetch 拦截 自动记录所有 API 交互；可输出结构化数据 有副作用（修改了全局状态）；需要维护补丁 采集 API 交互数据、理解 Tool Use 协议细节
