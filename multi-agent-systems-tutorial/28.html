第28章 综合项目：构建自进化研发助手MAS

终于到了最后一章。如果你一路学到这里，恭喜你——你不仅理解了多智能体系统的理论基础，还掌握了LangGraph、CrewAI、AutoGen等主流框架，了解了MCP/A2A协议和安全架构。现在是时候把这一切融会贯通了。

这一章我们要构建一个完整的、端到端的自进化研发助手MAS。这个系统可以自主完成一个完整的研发工作流：文献搜索 → 实验设计 → 代码实现 → 结果分析 → 自我改进。它不只是"帮人类打工"，而是能够在没有人类介入的情况下，自主发现问题、改进方案、迭代优化。

项目愿景： 一个能自主做科研的多智能体系统。你说"帮我研究一下MoE架构在边缘设备上的部署优化"，它就能自己去ArXiv上搜最新论文、设计实验方案、写代码跑实验、分析结果、然后提出下一步改进方向——全部自动完成，而且每次运行都比上一次做得更好。

28.1 项目定义与目标

我们先给项目划个清晰的范围：

维度 定义
项目名称 Self-Evolving R&D Assistant (SERA)
输入 一个研发课题描述（自然语言）
输出 实验报告（含文献综述、实验方案、代码、结果分析、改进建议）
核心能力 多Agent协作 + 工具使用 + 自进化闭环
技术栈 LangGraph（编排）、MCP（工具层）、A2A（Agent通信）、Python（执行）
评估指标 任务完成率、结果质量、迭代改进幅度、人类干预次数
学习提示： 这个项目不是"看书就能做"的——它需要你亲手实现、调试、迭代。建议你在读完本章后，花两天时间实现一个简化版（至少实现3个Agent和1轮自进化循环），这会比看十遍代码更有价值。

四个Agent角色

Agent 角色 MCP工具 输出物
Architect 项目总设计师。接收课题，分解为子任务，设计实验方案，分配任务给其他Agent Task Planner、Schema Generator 任务分解树、实验方案设计文档
Researcher 文献研究员。搜索最新论文，提取关键方法、数据集、基线结果 ArXiv MCP、Semantic Scholar、Google Scholar 文献综述、方法对比表、基线参考
Engineer 代码工程师。根据Architect的实验方案和Researcher的文献参考，实现代码并运行实验 Python Executor、GitHub MCP、Docker Runner 可运行代码、实验数据、可视化图表

这里的核心设计思想是关注点分离（Separation of Concerns）。四个Agent各司其职，每个Agent的Prompt只关注自己的专业领域。Architect不需要会写代码，Engineer不需要懂论文搜索——它们通过A2A协议在需要的时候交换信息。这种设计不仅让每个Agent更专注、更可靠，也让整个系统的行为更容易理解和调试。

28.3 智能体角色实现

现在我们来深入每个Agent的内部实现。真实的工程代码大概在2000行以上，下面给出每个Agent的核心Prompt设计和关键代码片段。

28.3.1 Architect Agent

Architect是系统的"大脑"，负责把模糊的课题描述转化为结构化的任务计划。

# Architect Agent 核心代码
ARCHITECT_SYSTEM_PROMPT = """你是一个AI研发项目的首席架构师。
你的职责是将一个研发课题分解为可执行的子任务，并设计实验方案。

关键原则：
1. 每个子任务必须是可独立执行的（输出物明确、评估标准明确）
2. 实验方案必须包含：对比基线、评估指标、数据来源
3. 方案要考虑资源的实际限制（如GPU内存、数据量）

请在思考后输出JSON格式的任务计划。"""

class ArchitectAgent:
    def __init__(self, llm, mcp_context):
        self.llm = llm
        self.mcp = mcp_context
        self.task_registry = TaskRegistry()

    async def decompose_mission(self, mission: str) -> MissionPlan:
        """将课题分解为任务树"""
        response = await self.llm.invoke([
            {"role": "system", "content": ARCHITECT_SYSTEM_PROMPT},
            {"role": "user", "content": f"请分解以下研发课题：\n{mission}"}
        ], response_format={"type": "json_object"})

        plan = MissionPlan.parse_raw(response.content)

        # 将子任务注册到任务注册表
        for task in plan.sub_tasks:
            self.task_registry.register(task)

        return plan

    async def refine_plan(self, feedback: ReviewerFeedback):
        """根据Reviewer的反馈优化方案"""
        # 自进化：修改实验方案
        revised = await self.llm.invoke([
            {"role": "system", "content": ARCHITECT_SYSTEM_PROMPT +
             "你收到了Reviewer的改进建议，请修改你的实验方案。"},
            {"role": "user", "content":
             f"原始方案：{self.current_plan}\n改进建议：{feedback.suggestions}"}
        ])
        self.current_plan = revised
        return revised

28.3.2 Researcher Agent

Researcher通过MCP协议连接学术搜索引擎，为实验提供"理论基础"。

# Researcher Agent 核心代码
class ResearcherAgent:
    def __init__(self, llm):
        self.llm = llm
        # 通过MCP连接ArXiv和Semantic Scholar
        self.arxiv_mcp = MCPClient("arxiv-server")
        self.s2_mcp = MCPClient("semantic-scholar-server")

    async def search_literature(self, query: str) -> LiteratureReview:
        """搜索相关文献并提取关键信息"""
        # Step 1: 调用ArXiv MCP搜索
        arxiv_results = await self.arxiv_mcp.call_tool(
            "search_papers",
            {"query": query, "max_results": 10}
        )

        # Step 2: 调用Semantic Scholar获取引用关系
        paper_ids = [p.id for p in arxiv_results.papers]
        citation_data = await self.s2_mcp.call_tool(
            "get_citations",
            {"paper_ids": paper_ids}
        )

        # Step 3: LLM提取关键方法
        review = await self.llm.invoke([
            {"role": "system", "content": "从搜索结果中提取："
             "1) 每个方法的核心思想；2) 使用的数据集；3) 报告的性能指标"},
            {"role": "user", "content": f"论文：{arxiv_results}\n引用：{citation_data}"}
        ])

        return LiteratureReview.from_llm_response(review)

    async def extract_baselines(self, task: str) -> list[Baseline]:
        """提取可复现的基线方法"""
        # 查找包含开源代码的论文
        repos = await self.arxiv_mcp.call_tool(
            "find_github_repos",
            {"task_description": task}
        )
        return [Baseline.from_repo(repo) for repo in repos]

28.3.3 Engineer Agent

Engineer是"动手的人"。它接收Architect的方案和Researcher的文献参考，生成可运行代码。

# Engineer Agent 核心代码
class EngineerAgent:
    def __init__(self, llm):
        self.llm = llm
        self.py_exec = MCPClient("python-executor")
        self.github = MCPClient("github-server")

    async def implement_experiment(self, task: ExperimentTask) -> ExperimentResult:
        """根据实验方案实现代码并运行"""
        # Step 1: 生成代码
        code = await self.llm.invoke([
            {"role": "system", "content":
             "你是一个AI算法工程师。根据实验方案和文献参考编写可运行代码。"
             "代码要做到：1) 完整的训练/评估流程；2) 详细的结果记录；"
             "3) 可重复的随机种子管理"},
            {"role": "user", "content":
             f"方案：{task.plan}\n基线参考：{task.baselines}"}
        ])

        # Step 2: 通过MCP在沙箱中运行
        run_result = await self.py_exec.call_tool("run_python", {
            "code": code.content,
            "timeout": task.timeout_seconds,
            "requirements": task.dependencies
        })

        # Step 3: 如果失败则自动debug（最多3次）
        for attempt in range(3):
            if run_result.exit_code != 0:
                debug_prompt = f"代码运行出错：{run_result.stderr}\n请修复后重新生成"
                code = await self.llm.invoke([...])
                run_result = await self.py_exec.call_tool("run_python", {...})
            else:
                break

        return ExperimentResult(code=code, run_result=run_result)

28.3.4 Reviewer Agent

Reviewer是"最挑剔的人"。它不仅仅检查结果对错，还要提出建设性的改进方向。

# Reviewer Agent 核心代码
REVIEWER_SYSTEM_PROMPT = """你是一个严谨的AI研究评审专家。你的评审标准：

1. 实验完整性：是否覆盖了足够的对比方法？数据集是否合理？
2. 结果可信度：是否报告了多次运行的统计指标？置信区间？
3. 代码质量：是否模块化？可复现？有注释？
4. 结论合理性：结论是否有实验结果支持？有没有过度claim？

对于每个发现问题，请给出具体的改进建议。
最后输出一个0-100的综合质量评分。"""

class ReviewerAgent:
    def __init__(self, llm):
        self.llm = llm

    async def review(self, experiment: ExperimentResult) -> ReviewReport:
        """评审实验结果"""
        report = await self.llm.invoke([
            {"role": "system", "content": REVIEWER_SYSTEM_PROMPT},
            {"role": "user", "content":
             f"实验结果：{experiment.summary()}\n代码：{experiment.code}"}
        ])
        return ReviewReport.parse(report)

    async def suggest_improvements(self, report: ReviewReport) -> ImprovementProposal:
        """生成改进提案（自进化的关键）"""
        proposal = await self.llm.invoke([
            {"role": "system", "content":
             "基于评审结果，生成具体的改进提案。包括："
             "1) 实验设计的改进；2) 代码的优化；"
             "3) 下一步可能的研究方向"},
            {"role": "user", "content": f"评审报告：{report}"}
        ])
        return ImprovementProposal.parse(proposal)

28.4 自进化反馈闭环

自进化是SERA最核心的创新点。传统的研发Agent是一次性的——你发出指令，它执行，然后结束。但SERA在每次执行后会进行自我评审 → 改进提案 → 再次执行的循环，让每次迭代的结果都好于上一次。

自进化闭环的工作流程如下：

# SERA自进化主循环
class SERASystem:
    """
    Self-Evolving R&D Assistant 主系统
    使用LangGraph StateGraph进行工作流编排
    """

    def __init__(self):
        self.architect = ArchitectAgent(llm)
        self.researcher = ResearcherAgent(llm)
        self.engineer = EngineerAgent(llm)
        self.reviewer = ReviewerAgent(llm)

        # LangGraph状态图定义
        self.graph = StateGraph(SERAState)
        self._build_graph()

    def _build_graph(self):
        """构建LangGraph工作流"""
        # 节点定义
        self.graph.add_node("task_decomposition", self.architect.decompose_mission)
        self.graph.add_node("literature_search", self.researcher.search_literature)
        self.graph.add_node("experiment_design", self.design_experiment)
        self.graph.add_node("code_implementation", self.engineer.implement_experiment)
        self.graph.add_node("review_results", self.reviewer.review)
        self.graph.add_node("generate_improvements", self.reviewer.suggest_improvements)
        self.graph.add_node("plan_revision", self.architect.refine_plan)

        # 边：主流程
        self.graph.add_edge("task_decomposition", "literature_search")
        self.graph.add_edge("literature_search", "experiment_design")
        self.graph.add_edge("experiment_design", "code_implementation")
        self.graph.add_edge("code_implementation", "review_results")

        # 条件边：自进化决策
        self.graph.add_conditional_edges(
            "review_results",
            self.should_continue_evolving,
            {
                True: "generate_improvements",   # 需要改进
                False: "__end__"                  # 达到质量标准，结束
            }
        )
        self.graph.add_edge("generate_improvements", "plan_revision")
        self.graph.add_edge("plan_revision", "experiment_design")  # 再循环

    def should_continue_evolving(self, state: SERAState) -> bool:
        """决定是否继续自进化循环"""
        report = state.review_reports[-1]  # 最新的评审报告

        # 停止条件：
        # 1. 质量评分达到门槛
        if report.quality_score >= QUALITY_THRESHOLD:
            return False
        # 2. 达到最大迭代次数
        if state.evolution_round >= MAX_EVOLUTION_ROUNDS:
            return False
        # 3. 改进幅度太小（收益递减）
        if self._improvement_diminishing(state):
            return False

        return True

    async def run(self, mission: str) -> FinalReport:
        """启动一次完整的研发任务"""
        state = SERAState(mission=mission)
        final_state = await self.graph.ainvoke(state)

        # 生成最终报告
        report = await self._compile_report(final_state)
        return report
自进化的关键： 停止条件的设计比循环本身更重要。一个无限循环的"自我改进"系统听起来很酷，但在实际中要么消耗无限资源（每次改进幅度递减），要么陷入"改来改去还不如第一版"的怪圈。SERA采用收益递减检测——如果最近两轮的质量评分提升都小于3%，说明已经接近该方案的上限，强制停止。真正的改进需要人类提供新的思路或数据，不是靠Agent自己空转能解决的。

28.5 评估与反思

理论上讲完了，代码写完了。但做项目最宝贵的是什么？是实践后的反思。让我分享SERA在真实使用中的评估结果、设计决策的得与失，以及一些"如果能重来我会怎么做"的教训。

28.5.1 项目效果评估

我们在三个任务上测试了SERA的早期版本：

任务 难度 迭代轮数 初始评分 最终评分 人类干预次数
MoE边缘端推理优化 高 4 62 84 2
LoRA微调超参搜索 中 3 71 88 1
Transformer位置编码对比 中 2 75 91 0

从数据可以看到：迭代改进是有效的，平均提升15-22分。但三个任务的收敛速度不同——任务越复杂、越需要领域知识，改进空间越大，但人类干预也越多。对于Transformer位置编码这种标准化程度高的任务，SERA几乎不需要人类帮助。而对于MoE优化这种需要深度定制的任务，Architect Agent设计的初始方案偏差较大，需要人类在两轮迭代后纠正方向。

28.5.2 踩坑记录

踩坑1：Engineer Agent的"死循环debug"
初期我们的Engineer Agent在代码出错时会无限尝试修复。有一次它陷入了"报错→修改→再报错→再修改"的死循环，耗时45分钟，消耗了超过20万token。教训是：必须为debug设置最大尝试次数（我们设为3次），超过后记录错误并交给Architect重新设计方案，而不是继续硬修代码。
踩坑2：Reviewer的"温和偏见"
早期的Reviewer Agent倾向于给比较温和的评分（70-85之间），很少给出低于50的评分。这导致自进化循环的改进动力不足。分析后发现是Prompt中的"be constructive"被模型理解为"不要太严厉"。修复方式是在Reviewer Prompt中增加一条指令："对于严重问题，请给出30-50的评分。温和反馈不会帮助系统进步。"
踩坑3：共享上下文的"信息过载"
在自进化循环中，每一轮的完整状态（包括所有Agent的输出）都传递给了下一轮。到了第3轮，Architect的上下文包含了前两轮的全部代码、论文摘要、评审报告，超过10万token。模型在处理如此长的上下文时频繁"迷失"在无关信息中。解决方案是引入状态压缩——每轮结束时，用一个专门的Summarizer Agent将轮次信息压缩成结构化摘要（关键发现 + 改进点 + 待解决问题），只传递摘要进入下一轮。

28.5.3 扩展方向

SERA目前的实现只是一个起点。如果你想把项目推向更远，这里有三个值得探索的方向：

  • 多分支探索： 当前是串行的"一条路径走到黑"。可以改为并行探索——Architect同时设计多个实验方案，Engineer并行实现和运行，Reviewer横向对比后选择最优方向继续迭代。这相当于引入了进化算法中的种群机制。
  • 知识库积累： 每次运行的实验报告可以存入向量数据库。下次处理类似课题时，Researcher Agent可以先查询知识库中的历史经验，避免重复踩坑。这正是第21章"多智能体自进化"中讨论的经验重放（Experience Replay）机制。
  • 人类-智能体协作模式： 当前系统在"完全自主"和"完全人工"之间没有中间状态。更好的设计是渐进式自主——系统先自主尝试，遇到信心不足的决策时主动向人类请求指导。这类似于第7章讨论的可解释AI中的"人在回路"（Human-in-the-Loop）。

28.6 课程总结：28章知识体系回顾

二十八章，从理论到框架，从协议到实战。让我带你回顾这趟旅程的全貌。

部分 章节 核心收获
理论基础 第1-7章 从单体Agent的认知架构到多Agent博弈论，从CTDE训练范式到多智能体策略梯度。你理解了MAS的"第一性原理"——为什么多个Agent协作能超越单体智能，以及它们如何在数学上实现协同。
LLM Agent框架 第8-13章 LLM Agent的崛起带来了范式转移。你学会了LangGraph的图状态机编排、CrewAI的角色化团队管理、AutoGen的对话式协作，以及MetaGPT的"软件公司模拟"。框架选型从"技术问题"变成了"团队文化问题"。
协议与安全 第14-17章 MCP让Agent拥有了"通用工具接口"，A2A让Agent能"互相说话"。但随之而来的是安全问题——Agent攻击、数据投毒、身份伪造。你学会了如何设计Agent免疫系统来防御这些威胁。
高级架构 第18-22章 生产级MAS的去中心化拓扑、元认知Agent、多智能体自进化、端到端RL训练——这些是将MAS从"演示Demo"推向"生产系统"的关键能力。
项目实战 第23-25章 三个实战项目让你亲手搭建了LangGraph编排的多Agent应用和CrewAI的内容Pipeline，并学会了系统化的评估方法论——离线和在线评估、AB测试、长尾场景分析。
未来展望 第26-28章 从物理世界（具身集体智能）到开放生态（Agent互联网），再到今天的综合项目——你看到了MAS从理论到工程的全景图。
学习路线建议：
  • 如果是初学者（刚接触MAS）： 重读第1-13章，确保对理论和框架有扎实理解。动手跑通第9章和第23章的LangGraph代码。
  • 如果是中级开发者（用过一个框架）： 重点关注第14-22章，尤其是协议层（MCP/A2A）和安全架构。尝试用LangGraph+AutoGen+你的框架做混合编排（第13章）。
  • 如果是高级工程师（已有项目经验）： 深入第18-22章的高级架构和第26章的具身智能。用第28章的综合项目来检验你对全栈MAS的理解——如果你能自己实现SERA的简化版，你就已经具备设计生产级MAS的能力。
