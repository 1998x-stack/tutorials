第二十章：负载均衡与限流

这一章我们继续深入架构层的HTTP话题。上一章讲了代理和CDN，这一章聚焦在流量分配和流量控制两个核心问题上。

简单来说：负载均衡解决的是"来了100万个请求，怎么分给10台服务器"的问题；限流解决的是"来了1000万个请求，超过了系统的承受能力，怎么办"的问题。这两个机制配合使用，是构建高可用系统的基础。

20.1 负载均衡算法

负载均衡器（无论是Nginx、HAProxy还是云服务商的LB）的核心任务就是决定"下一个请求发给谁"。不同的算法有不同的适用场景。

算法原理适用场景缺点
轮询（Round-Robin）按顺序轮流分配请求到每台服务器服务器配置相同、请求处理时间相近不区分服务器负载差异
加权轮询（Weighted RR）给高性能服务器更大的权重，按比例分配服务器配置不同、需要差异化分配权重配置不合理时效果差
最少连接（Least Connections）把请求发给当前活跃连接最少的服务器请求处理时间差异大、长连接场景需要维护连接状态
IP Hash对客户端IP做哈希，同一IP始终分配到同一台服务器需要会话保持的场景IP变化或很少客户端时分布不均匀
随机（Random）随机选择一台服务器请求量极大、分布需求简单短期可能不均匀

在Nginx里配置这些算法：

# 轮询（默认）
upstream backend {
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
}

# 加权轮询
upstream backend {
    server 10.0.0.1:8080 weight=3;
    server 10.0.0.2:8080 weight=1;
}

# 最少连接
upstream backend {
    least_conn;
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
}

# IP Hash
upstream backend {
    ip_hash;
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
}

选择建议：如果你的后端是无状态服务（推荐架构），优先使用最少连接或加权轮询。IP Hash只在需要会话保持时才使用，而且会带来负载不均的风险——比如一个公司所有员工都通过同一个出口IP访问，那这个IP的哈希值会一直打到同一台服务器上。

20.2 Nginx upstream配置

upstream 块是Nginx负载均衡的配置核心。这里整理一些进阶用法：

upstream backend {
    # 加权分配
    server 10.0.0.1:8080 weight=5;
    server 10.0.0.2:8080 weight=3;

    # 备份服务器（仅在其他服务器不可用时使用）
    server 10.0.0.3:8080 backup;

    # 标记为"down"，从负载池中移除但不下线（用于灰度发布）
    server 10.0.0.4:8080 down;

    # 最大失败次数和失败超时
    server 10.0.0.5:8080 max_fails=3 fail_timeout=30s;

    # 最大连接数（配合 least_conn 使用）
    server 10.0.0.6:8080 max_conns=100;

    # 慢启动（让新加入的服务器"热身"）
    server 10.0.0.7:8080 slow_start=30s;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend;
        proxy_next_upstream error timeout http_500 http_502 http_503;
        proxy_next_upstream_tries 3;
        proxy_connect_timeout 5s;
        proxy_read_timeout 10s;
    }
}

关键配置解读：

  • max_fails 和 fail_timeout：配合实现被动健康检查。如果一台服务器在30秒内失败了3次，Nginx会将其标记为不可用，不再转发流量给它。
  • backup：备胎服务器。平时不处理请求，只在其他服务器都挂掉时顶上。适用于"保底可用"的场景。
  • proxy_next_upstream：当前服务器返回错误（超时、5xx等）时，自动重试下一台服务器。但注意：幂等性问题——如果请求是POST写操作，重试可能导致重复写入，需要后端做幂等处理。
  • slow_start：新加入或恢复的服务器不会瞬间接收全量请求，而是逐步增加权重，避免"冷启动"问题。

踩坑故事：有次我们做服务扩容，凌晨加了5台新服务器到Nginx upstream里。上线后发现新服务器的CPU飙到100%，大量请求超时。原因就是没有用slow_start。新服务器上的应用进程在初始化时需要加载大量数据到内存（冷缓存状态），瞬间涌入的流量导致所有请求都在等数据加载，死锁了。从那以后，我们对所有新上线的服务器都配置了 slow_start=60s，让流量缓慢爬升，给服务预热的时间。

20.3 会话保持（Sticky Sessions）

HTTP是无状态的，但很多应用是有状态的——一个用户登录后的会话数据存在某台服务器的内存中。如果下一个请求被负载均衡器分到了另一台服务器，用户就"掉登录"了。

解决这个问题的方案有几种：

方案原理优点缺点
IP Hash同一客户端IP始终发到同一台服务器无需额外配置NAT环境下多个用户共享IP
Cookie插入LB在响应中插入一个Cookie，记录后端服务器ID精确到用户级别需要LB支持、可能有安全风险
会话复制会话数据在服务器之间同步任何服务器都能处理网络开销大、实现复杂
集中式会话存储会话数据存放在Redis等外部存储服务器无状态、弹性伸缩增加了外部依赖和延迟

Nginx的sticky cookie配置：

upstream backend {
    sticky cookie srv_id expires=1h domain=.example.com path=/;
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
}

虽然sticky session能解决问题，但我强烈建议你优先考虑集中式会话存储（方案4）。为什么呢？因为sticky session破坏了负载均衡的弹性——如果某台服务器挂了，它上面的所有会话都丢失了；而且服务器增删时需要手动调整负载均衡策略。

架构建议：如果你的应用正在考虑sticky session，不如花时间把Session迁移到Redis或数据库中。在微服务时代，"无状态"是服务设计的目标。任何一台服务器都应该能处理任何用户的请求。这样你才能自由扩缩容，而不受"session粘在哪个节点上"的约束。

20.4 限流策略

限流是保护系统不被流量冲垮的最后一道防线。如果说负载均衡是"怎么分配流量"，限流就是"流量太多时怎么拒绝"。

常见的限流算法：

令牌桶（Token Bucket）：以固定速率向桶里放令牌，每个请求消耗一个令牌。桶有容量上限，允许突发流量消耗累积的令牌。这是最常用的算法，也是Nginx和Guava RateLimiter的实现基础。

漏桶（Leaky Bucket）：请求先进入队列，以固定速率出队处理。队列满了就丢弃。和令牌桶的区别：令牌桶允许一定程度的突发，漏桶则严格平滑。

计数器（固定窗口）：在固定时间窗口内计数，超过阈值就拒绝。缺点是在窗口边界可能发生"突发"——前一秒末和后一秒初各允许一次突发，合在一起就超过了限制的两倍。

滑动窗口：把时间窗口细化成多个小格子，拒绝粒度和精度更好。

Nginx的限流配置：

# 定义限流区域：每个IP在10MB内存空间中记录状态
# rate=10r/s 表示每秒最多10个请求
limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;

# 可选的：针对特定URL的限流
limit_req_zone $uri zone=static:10m rate=100r/s;

server {
    location /api/ {
        # burst=20 允许突发20个请求的排队（令牌桶容量20）
        # nodelay 表示排队中的请求不额外增加延迟（立即处理）
        limit_req zone=mylimit burst=20 nodelay;

        # 超过限制时的响应
        limit_req_status 429;

        proxy_pass http://backend;
    }
}

当请求被限流时，Nginx会返回 429 Too Many Requests。客户端看到429后，可以通过 Retry-After 响应头知道需要等待多久再重试。

限流设计要点： 1. 区分用户：对普通用户的限流阈值应该低于VIP用户。可以通过API Key、Token或IP来区分。 2. 分层限流：在网关/反向代理层做粗粒度限流，在应用层做细粒度限流（比如每个用户的限流）。 3. 限流降级：被限流的请求不要直接丢弃，可以考虑降级处理（返回缓存数据、返回简化版响应等）。 4. 过载保护：限流不仅是限制客户端，也要保护下游——比如数据库连接池满了，就应该在API层限流。

20.5 健康检查

负载均衡器必须知道哪些后端服务器是"活的"，才能把请求只转发给健康的服务器。这就是健康检查（Health Check）。

主动健康检查（Active Check）：负载均衡器定期向后端服务器发送探测请求（比如GET /health），根据响应判断服务器是否健康。

Nginx Plus（商业版）支持主动检查。开源版Nginx可以通过 nginx_upstream_check_module 或第三方的健康检查方案实现。

HAProxy的主动健康检查：

backend servers
    server app1 10.0.0.1:8080 check inter 3s fall 3 rise 2
    server app2 10.0.0.2:8080 check inter 3s fall 3 rise 2

被动健康检查（Passive Check）：不额外发请求，通过观察正常流量中的错误来判断。Nginx开源版通过 max_fails 和 fail_timeout 实现。

主动检查被动检查
额外流量有（探测请求）无
发现故障时间快（探测间隔可控）慢（需要等到有流量）
适用场景高可用要求严格流量均匀、资源紧张
实现复杂度需要额外配置基本零配置

合理的健康检查需要配置失败阈值（fall）和恢复阈值（rise）：

  • 不要因为一次超时就标记服务器为"挂掉"——网络抖动是正常的。一般建议连续失败3次以上再摘除。
  • 恢复也要谨慎——服务器刚重启时可能还在预热，连续成功2次再放回流量池。
  • 状态变更要记录日志，方便排查问题。

健康检查最佳实践：/health 端点不仅要返回200，还应该检查依赖的服务是否正常（数据库连接、缓存连接等）。但注意：如果检查得太深（比如每次查数据库），健康检查本身就成了额外的压力。我建议做一个分层健康检查：LB层的检查只测TCP端口通不通和HTTP 200；应用内部再有一个深度的健康检查用于诊断，不暴露给外部的LB。
