第2章 推荐系统简史

大家好，我是你们的讲师。上一章我们了解了推荐系统的全貌——它是什么，它的架构长什么样，它如何改变互联网产品。但你有没有想过，推荐系统不是一天建成的。从1990年代学术界的一个朴素想法，到今天大模型驱动的主流技术，这条路走了整整三十年。

这一章，我们就沿着时间线，梳理推荐系统发展的四个关键时期。我会把每个时期的标志性论文、关键公司和核心技术讲清楚。这不仅是历史课——理解了"为什么某个技术在那个时间点出现"，你才能真正理解推荐系统"为什么是今天这个样子"。

为什么了解历史重要？

推荐系统的每一次跃进都不是凭空出现的：协同过滤诞生于数据匮乏时代，矩阵分解爆发于Netflix百万美元竞赛，深度学习崛起于算力成本下降，LLM主导推荐则源自大语言模型的通用推理能力。每个时代的"最优方案"都是特定约束下的产物。不理解这些约束，你会在错误的地方寻找答案。

1. 1994-2010：协同过滤与矩阵分解的黄金时代

1.1 GroupLens：协同过滤的起点

推荐系统的学术起点公认是1994年GroupLens研究小组的工作。当时明尼苏达大学的Paul Resnick等人发表了论文《GroupLens: An Open Architecture for Collaborative Filtering of Netnews》。这是学术界第一次提出用自动化协同过滤（Collaborative Filtering）算法来帮用户过滤Usenet新闻组中的内容。

核心思想在今天看来非常朴素：如果用户A和用户B过去对某些新闻的评价相似，那么A喜欢的新闻，B也可能喜欢。这个思想后来演变成了User-based Collaborative Filtering（UserCF）。1994年的GroupLens系统虽然没有大规模商业部署，但它证明了"让算法代替用户做信息筛选"是可行的。

1.2 Amazon Item-to-Item：协同过滤的商业化里程碑

如果说GroupLens是"提出了想法"，那Amazon则是在2003年把想法变成了真金白银。Amazon的Greg Linden等人在论文《Amazon.com Recommendations: Item-to-Item Collaborative Filtering》中提出了Item-based Collaborative Filtering（ItemCF）。

区别于UserCF的"找相似用户"，ItemCF的核心是"找相似物品"——"购买此商品的客户也购买了……"这个今天所有电商平台标配的功能，就是ItemCF的产物。ItemCF相对于UserCF的一个巨大优势是可扩展性：在用户量远大于商品量的场景下（比如Amazon早期，"书"的数量增长慢于"买书的人"），物品间相似度可以离线计算，在线查询效率极高。

这个算法效果有多好？我们上一章提到过——Amazon的核心营收中，约35%来自推荐系统，而这个比例的起点就是ItemCF。

1.3 Netflix Prize：一个改变行业格局的比赛

2006年10月，Netflix宣布启动Netflix Prize竞赛——只要有人能将Netflix的评分预测算法RMSE（Root Mean Square Error）降低10%，就能获得100万美元的奖金。这个比赛彻底改变了推荐系统的研究方向。

比赛持续了近三年。期间，全球4000多个团队提交了超过44000个解决方案。早期的领先方案基本都是矩阵分解（Matrix Factorization）和隐语义模型（Latent Factor Model）。耶胡达·科伦（Yehuda Koren）成为了这个时期的关键人物——他后来在论文中系统性地提出了基于SVD的矩阵分解推荐方法，成为了推荐系统最经典的算法之一。

Netflix Prize的戏剧性结局：2009年6月，距离比赛结束不到24小时，一支叫BellKor's Pragmatic Chaos的团队提交了最终方案——这是一个由超过100个模型组成的超级集成（Ensemble），包括矩阵分解、受限玻尔兹曼机（RBM）、邻域模型等多种算法的加权融合。他们以1.0%的优势击败了第二名，赢得了100万美元。但这个方案在工程上是近乎"不可用"的——100多个模型同时做预测，单次推荐的计算成本高得惊人。

更值得思考的是，Netflix最终并没有部署这套百万美元的获奖方案。原因很简单：工程成本和实时性要求。比赛要求的是离线RMSE指标，但Netflix产品需要的是快速、稳定的在线推荐。这个问题——"离线SOTA不等于在线最优"——成为贯穿整个推荐系统历史的永恒话题。

尽管如此，Netflix Prize的深远影响不可否认：它把矩阵分解和隐语义模型推到了聚光灯下，吸引了一大批顶级研究人员进入推荐系统领域。今天我们使用的很多技术，源头都可以追溯到那场比赛。

2. 2010-2016：逻辑回归与FM的工业落地

2010年前后，推荐系统的重心从学术界转向了工业界。互联网公司开始面临真正的"大厂挑战"：用户数亿、物品千万级、需要在毫秒级别完成推荐。矩阵分解虽然漂亮，但面对工业级的稀疏特征和海量样本，它的可扩展性和特征灵活性都捉襟见肘。

2.1 Google的LR + 稀疏特征范式

2010-2012年间，Google在广告点击率预估（CTR Prediction）上的工作奠定了工业推荐的基础范式。他们使用逻辑回归（Logistic Regression）作为核心模型，配合大规模稀疏特征（用户ID、广告ID、Query等特征经过One-Hot编码后达到数十亿维），在大规模分布式系统（Google的DistBelief框架）上进行训练。

LR的优势在于：可解释性强、训练速度快、分布式扩展性好。但它有一个致命缺陷——缺乏特征交叉能力。LR本质上是对每个特征做线性加权，如果不手动做特征交叉（Feature Crossing），模型就学不到"年轻用户对运动类商品更感兴趣"这种特征组合模式。

2.2 Facebook的GBDT+LR

2014年，Facebook发表了论文《Practical Lessons from Predicting Clicks on Ads at Facebook》，提出了GBDT+LR的组合模型。思路非常巧妙：用GBDT（梯度提升决策树）将原始特征空间自动转换为高维稀疏特征（GBDT的叶子节点路径），再将转换后的特征输入LR做CTR预估。

这篇论文之所以经典，是因为它用一个"浅层模型组合"达到了当时接近深度学习的效果，而且工程实现简单。但它的局限性也很明显：GBDT对高维稀疏特征的处理并不理想，树模型天然不太擅长处理大量离散特征。

2.3 Rendle的FM

2010年，Steffen Rendle发表了因子分解机（Factorization Machines，FM）论文。FM巧妙地解决了LR无法做特征交叉的问题——它为每个特征学习一个隐向量（类似矩阵分解中的user/item向量），特征交叉的权重由隐向量内积计算。这样既能捕捉交叉特征，又能处理稀疏数据，并且预测复杂度是线性的。

FM的诞生标志着推荐系统从"人工特征工程+线性模型"向"自动特征交叉"迈出了重要一步。后来出现的FFM（Field-aware FM）、DeepFM等模型，都是在FM这个地基上盖起来的高楼。

3. 2016-2022：深度学习的全面渗透

2016年是推荐系统深度学习的"元年"。这一年，Google同时上线了Wide&Deep和YouTube DNN推荐系统，向业界展示了深度学习在推荐场景中的巨大潜力。

3.1 Wide&Deep与DNN范式

Google在2016年发表的《Wide & Deep Learning for Recommender Systems》成为了业界最具影响力的推荐论文之一。核心思想是"记忆与泛化的结合"：Wide部分（LR）负责记住高频特征组合（比如"安装了某个app的用户通常会下载某个游戏"），Deep部分（DNN）负责学习低维稠密表示，泛化到未见过的特征组合。

这个框架的工业意义极其深远：它提供了一种可复用的深度推荐架构模板。后来的DeepFM、DCN、xDeepFM等模型，本质上都在回答同一个问题——"如何用深度网络实现更有效的特征交叉？"

3.2 Alibaba DIN/DIEN：关注用户行为序列

2017-2018年，阿里巴巴发表了深度兴趣网络（Deep Interest Network，DIN）和深度兴趣进化网络（Deep Interest Evolution Network，DIEN）。这两篇论文的核心洞察是：用户的兴趣不是静态的，而是受"候选物品"影响的动态过程。用户在浏览某个商品时，历史行为中与该商品相关的部分应该被赋予更高权重——这就是Attention机制在推荐中的应用。

DIN提出的"目标物品自适应注意力"（Target Attention）框架，后来成为了推荐系统处理用户行为序列的标准范式。

3.3 YouTube DNN与双塔模型

Google在2016年发表的《Deep Neural Networks for YouTube Recommendations》首次将深度学习应用于大规模视频推荐。更关键的是，它提出了"双塔模型"（Two-Tower Model）的雏形：用两个独立的DNN分别将用户和物品映射到同一个隐空间，通过向量内积衡量匹配度。这种架构在召回阶段效果极佳，成为了后续工业界最主流的召回范式。

双塔模型的成功让向量检索（Vector Search / ANN）成为推荐系统的基础设施。Facebook的Faiss、Google的ScaNN、以及各种近似最近邻搜索库，都是在这个需求下蓬勃发展起来的。

一个时代的缩影：2016-2022这六年，推荐系统从一个"用LR就够用"的领域变成了"没有DNN就不好意思说自己是推荐系统"的领域。几乎所有头部互联网公司都在这个时期重建了自己的推荐架构，从Google到Meta，从字节跳动到阿里。这也是推荐系统从业者的黄金时代——人才缺口巨大，薪资水涨船高。

4. 2023-2026：LLM重塑推荐范式

2023年，随着ChatGPT引爆大语言模型浪潮，推荐系统迎来了新一轮范式革命。与前几次"渐进式进化"不同，这一轮变革的核心命题是——推荐系统能否从"判别式"走向"生成式"？

传统的推荐系统核心是一个判别（Discriminative）任务：给定用户特征和候选物品，模型判断用户点击/购买的概率。而生成式推荐（Generative Recommendation）的思路是：让模型直接生成用户可能会喜欢的物品ID或内容描述。

4.1 Meta的生成式召回

2023年，Meta发表了论文《Actions Speak Louder than Words: Trillion-Parameter Sequential Transducers for Generative Recommendations》（GR模型）。这篇论文的震撼之处在于：它直接将整个推荐任务建模为序列生成问题，用Transformer架构对用户行为序列建模，通过自回归方式生成下一个物品的ID。

虽然GR模型在工程落地层面仍然面临巨大的推理成本挑战，但它证明了"生成式推荐"至少在大规模数据上是可行的——这为整个行业打开了新的想象空间。

4.2 Kuaishou OneRec与Alibaba LUM

快手在2024年提出了OneRec框架，核心思想是将多阶段推荐（召回+粗排+精排+重排）统一为一个端到端的生成式模型。传统四阶段架构的优势是计算效率高，但缺点是各阶段优化的目标不一致，存在"信息损耗"（比如召回阶段筛掉的内容，精排阶段永远看不到）。OneRec用一个大模型直接生成最终推荐列表，从根本上避免了这个问题。

阿里巴巴的LUM（Large User Model）则是从另一个角度切入：将用户的所有行为序列编码为一个庞大的用户状态向量，然后用这个向量直接"生成"用户下一步可能会交互的物品。这和传统的"用户特征+候选集合+打分排序"的框架有本质区别。

4.3 范式转变的意义

LLM驱动的推荐范式转变，不仅体现在模型架构上，更体现在整个系统设计理念上：

  • 从匹配到生成：不再"从候选池中挑出最好的"，而是"从零生成最合适的推荐"
  • 从多阶段到端到端：四阶段漏斗可能被单一生成模型替代
  • 从专有模型到通用能力：LLM的常识推理和跨域迁移能力，有望缓解推荐系统的冷启动和长尾问题
  • 从行为信号到语义理解：大模型能理解内容的语义，而不仅仅是行为统计信号

不过也要清醒地看到，生成式推荐在2026年的今天依然面临巨大挑战：推理成本比传统双塔模型高出几个数量级、延迟控制艰难、可解释性和可控性有待验证。未来几年，我们大概率会看到"传统召回+LLM精排"的混合架构，而不是完全的范式替换。

5. 发展脉络总览

让我们用一张表格来总结这段跨越三十年的推荐系统进化史：

1994-2010 UserCF, ItemCF, SVD, MF GroupLens (UMN), Amazon, Netflix Prize 协同过滤思想的确立；隐语义模型的提出；推荐系统进入工业界

# 推荐系统关键技术时间线
timeline = [

    (2006, "Netflix Prize启动 - 矩阵分解时代开启"),
    (2009, "BellKor赢得Netflix Prize - 集成学习的巅峰"),
    (2010, "FM提出 - 特征交叉自动化的开端"),
    (2014, "Facebook GBDT+LR - CTR预估的工业范本"),
    (2016, "Wide&Deep + YouTube DNN - 深度学习元年"),
    (2017, "DIN - Attention机制引入推荐"),
    (2018, "DIEN - 用户兴趣演化建模"),
    (2020, "双塔模型全面普及 - 向量检索成为基建"),
    (2023, "Meta GR - 生成式推荐的理论验证"),
    (2024, "Kuaishou OneRec + Alibaba LUM - 端到端生成推荐"),
    (2026, "LLM推荐进入工业落地深水区"),
]

for year, event in timeline:
    bar = "█" * ((year - 1990) // 2)
    print(f"{year} | {bar} {event}")

一个重要的历史教训：回顾这三十年的发展，有一个规律反复出现——每次新技术的出现，都会引发"旧技术已死"的论调，但最终总是新旧技术共存。深度学习没有杀死LR（LR仍然在重排和解释性场景中广泛使用），LLM也不太可能完全杀死传统推荐模型。推荐系统的进化不是"替代史"，而是"扩展史"——工具箱里的工具越来越多，而不是越换越新。如果你是新手入行，务必打好传统推荐方法的功底，而不要只看最前沿的LLM技术。

好了，以上就是完整的推荐系统简史。我们从1994年的GroupLens一路走到了2026年的LLM推荐，看到了每个时代的代表性技术、关键突破和标志性故事。对历史脉络有了清晰认知之后，下一个问题是：我们怎么判断一个推荐系统好还是不好？离线指标怎么做，在线指标怎么设，A/B测试怎么设计才能避免"假阳性"？这些评估体系的问题，正是第三章要深入讨论的内容。
