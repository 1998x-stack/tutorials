第12章 流水线设计技术

想象你有一个工厂生产线。一个工人完成全部工序——从零件加工到组装到质检——需要 10 分钟。那么这条线每分钟只能产出一个产品。现在你把工序分成 10 个工位，每个工位完成一道工序，每道工序 1 分钟。第一分钟，第一个工位在加工第一个零件；第二分钟，第一个工位加工第二个零件，第二个工位处理第一个零件……十分钟后，第一个成品下线，然后每分钟下线一个成品。这就是流水线——用轻微的"启动延迟"换取了 10 倍的吞吐率。

FPGA 中的流水线（pipeline）设计就是这个道理。通过在组合逻辑路径中插入寄存器（pipeline registers），将一条长的组合路径切割成若干短的"流水级"。每级的最大延迟减少了，所以时钟可以跑得更快——这就是流水线提高吞吐率的本质。代价是额外的几个时钟周期的 latency，以及额外的触发器资源消耗。在大多数高性能设计中，这点代价微不足道。

12.1 流水线的本质

流水线的理论基础可以用一个简单的公式表达。一个同步数字电路的最大时钟频率 fmax 受限于最长的组合逻辑路径延迟：

fmax = 1 / (Tclk-q + Tlogic + Tsetup - Tskew)

其中 Tclk-q 是触发器的时钟到输出延迟，Tlogic 是组合逻辑的传播延迟，Tsetup 是触发器的建立时间，Tskew 是时钟偏斜（skew）。在一段设计中，Tlogic 往往是最大的变量——一个复杂的 32 位加法器可能有几十级 LUT 级联，延迟达到几纳秒甚至十几纳秒。

流水线的核心思想就是切割 Tlogic：在组合路径中间插入寄存器，把一段 N ns 的组合逻辑切成 M 段，每段最长不超过 N/M ns（更准确地说是 fmax 由最长段决定）。代价是增加了 (M-1) 个周期的 latency——输入到输出需要等待 M 个时钟周期才能看到结果。

吞吐率 vs 延迟：这是 FPGA 设计中最重要的两个性能指标。流水线提高吞吐率（每时钟周期可以处理一个新输入），但增加延迟（从输入到输出经过流水级的时钟周期数）。在信号处理中，大部分应用对延迟不敏感——你关心的是每秒钟能处理多少数据，而不是第一个数据晚了几纳秒出来。但控制回路（比如电源反馈环路、电机伺服）对延迟敏感，流水线就需要谨慎使用。选不选流水线、选多少级，永远取决于你的应用场景。

12.2 流水线深度的选择

流水线并非越深越好。增加一级流水线需要增加一组触发器，也增加一个周期的 latency。虽然这通常很便宜，但存在"收益递减"的效应——当组合路径已经被切到足够短时，再切下去，时序报告中的瓶颈变成了 Tclk-q + Tsetup（触发器的自身开销），此时继续增加流水级也提高不了 fmax。

典型的设计指导原则是：

  • 在顶层拍平的关键路径上选择切割点——不要把时间花在非关键路径上
  • 切割点应选在组合逻辑最密集的位置（从时序报告中的最差路径入手）
  • 数据路径和控制信号必须同时流水——如果数据有 4 级流水但 valid 信号没有延迟，下游模块会采到错误的数据
  • 重新收敛的信号（fanout 重汇聚）要特别注意——不同的流水级延迟会导致数据偏差

12.3 流水线冒险与解决

说到流水线，学计算机体系结构时你一定听过"冒险"（Hazard）这个词。在 FPGA 的流水线设计中，也存在三类冒险：

数据冒险（Data Hazard）：流水线的后一级需要用到前一级的结果，但结果要等时钟周期之后才能出来。在 CPU 中，解决方法是 forwarding（旁路/前推）——将 ALU 的输出直接送到同一周期中需要它的位置，不等待写回寄存器。在 FPGA 的数据路径设计中，同样的方法适用：如果第 2 级需要一个第 1 级的计算中间结果，不要等它写到寄存器再读——直接从第 1 级的组合输出连接到第 2 级的输入。

控制冒险（Control Hazard）：当流水线遇到条件分支时，你不知道下一个输入应该走哪条路径。在通用 CPU 中，这引出分支预测、预测错误后的流水线冲刷（flush）等复杂机制。在 FPGA 数据路径中，这个问题通常更简单——因为大部分信号处理数据流是确定的。如果你确实需要条件选择，可以用多路选择器（mux）在流水级中传递条件标志，到输出端再选择正确的结果。

结构冒险（Structural Hazard）：两个流水级需要同时访问同一个硬件资源（如同一个 BRAM 端口）。解决方法：（1）在 BRAM 中使用双端口（这也是为什么几乎所有的 FPGA BRAM 都至少是双端口的）；（2）将资源复制一份（duplication）；（3）在流水线中插入气泡（stall），但这会降低吞吐率。

12.4 实战案例：四级流水线乘法器

假设我们需要一个 16×16 的无符号乘法器，目标是在 FPGA 上实现 400MHz 的工作频率（对于 Artix-7 这样的中端器件，这是一个激进的但可达的 target）。未经流水线的组合乘法器，在 Artix-7 上的 LUT 延迟大约 8-10 ns，最高只能跑到 100-125 MHz。通过四级流水线，我们可以将每级的组合逻辑深度压缩到 2-3 级 LUT，接近 400MHz。

在这个例子中，我们不用 DSP slice 的硬件乘法器（DSP48），而是用 LUT 实现——目的是清楚地展示流水线的结构。

// 四级流水线乘法器 (16×16 → 32)
module pipelined_mult #(
    parameter A_WIDTH = 16,
    parameter B_WIDTH = 16
)(
    input  wire                 clk,
    input  wire                 rst_n,
    input  wire                 valid_in,
    input  wire [A_WIDTH-1:0]   a,
    input  wire [B_WIDTH-1:0]   b,
    output reg  [A_WIDTH+B_WIDTH-1:0] result,
    output reg                  valid_out
);

    // 流水级 1：部分积生成
    reg [A_WIDTH-1:0] a_stage1, b_stage1;
    reg valid_stage1;
    reg [A_WIDTH+B_WIDTH-1:0] pp [0:A_WIDTH-1];  // 部分积数组

    // 流水级 2：部分积缩减（第一级 3:2 压缩）
    reg [A_WIDTH+B_WIDTH-1:0] sum_stage2, carry_stage2;
    reg valid_stage2;

    // 流水级 3：部分积缩减（第二级 3:2 压缩 + 最终加法）
    reg [A_WIDTH+B_WIDTH-1:0] sum_stage3, carry_stage3;
    reg valid_stage3;

    integer i;

    // Stage 1: 生成部分积 + 锁存输入
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            a_stage1    <= 0;
            b_stage1    <= 0;
            valid_stage1<= 0;
        end else begin
            a_stage1    <= a;
            b_stage1    <= b;
            valid_stage1<= valid_in;
            for (i = 0; i < A_WIDTH; i = i + 1)
                pp[i] <= (b[i]) ? (a << i) : 0;
        end
    end

    // Stage 2: Wallace tree 第一级 3:2 压缩
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            sum_stage2   <= 0;
            carry_stage2 <= 0;
            valid_stage2 <= 0;
        end else begin
            valid_stage2 <= valid_stage1;
            // 简化的 Wallace tree：将 16 个部分积压缩为 sum + carry
            // 此处展示核心思路，实际 Wallace tree 需要更复杂的连接
            sum_stage2   <= pp[0] ^ pp[1] ^ pp[2] ^ pp[3];
            carry_stage2 <= (pp[0] & pp[1]) | (pp[1] & pp[2]) | (pp[0] & pp[2]);
            // 在实际项目中，这里会展开完整的 Wallace 压缩树
        end
    end

    // Stage 3: 继续压缩
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            sum_stage3   <= 0;
            carry_stage3 <= 0;
            valid_stage3 <= 0;
        end else begin
            valid_stage3 <= valid_stage2;
            sum_stage3   <= sum_stage2;
            carry_stage3 <= carry_stage2;
        end
    end

    // Stage 4: 最终加法
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            result    <= 0;
            valid_out <= 0;
        end else begin
            valid_out <= valid_stage3;
            result    <= sum_stage3 + carry_stage3;
        end
    end

endmodule

上面这个乘法器如果不用流水线，fmax 大约 100-125 MHz；加了四级流水线后，可以在 Artix-7 上达到 350-400MHz。代价是 4 个周期的 latency 和大约多出 200 个触发器。但注意——valid_in/valid_out 信号也必须跟着流水，否则下游模块会以为新数据在第 1 个周期就有效了。这是流水线设计中最容易犯的错误之一。

更实用的乘法器：在实际项目中，你几乎永远不会自己写 Wallace Tree 乘法器——直接使用 FPGA 的 DSP slice 或例化 IP 核（Xilinx Mult_gen / Intel lpm_mult），它们在面积和速度上都远超 LUT 实现。上面的例子纯粹是为了教学目的——展示流水线的每个阶段的寄存器插入和 valid 信号的同步传递。真正工作时，用 assign result = a * b; 让综合工具决定是否用 DSP slice。

12.5 实战案例：流水线 AES-128 加密引擎的数据路径

AES-128 加密算法包含 10 轮（round）运算，每轮包括 SubBytes、ShiftRows、MixColumns 和 AddRoundKey 四个步骤。在 FPGA 中实现 AES 加密器时，流水线是常用的优化手段——但不同的流水线策略会影响面积和吞吐率的平衡：

  • 全展开流水线（Fully Unrolled Pipeline）：将 10 轮全部展开为 10 个流水级，每级完成一轮运算。实现最高吞吐率——每时钟周期可以输出一个加密块。代价是面积很大——需要 10 组 S-Box（总共 160 个查找表 ROM），资源消耗约 2-3 万个 LUT。
  • 循环结构（Iterative/Loop Architecture）：只实现一轮的硬件，状态机控制回环运行 10 次。面积很小（约 2000 LUT），但输出一个块需要 10-11 个时钟周期。
  • 部分流水线（Partially Pipelined）：折中方案——流水级内做 2-3 轮，或者用 5 级流水线，每级做 2 轮。

选择哪种模式取决于你的应用：如果是磁盘加密（需要持续的高吞吐率），全展开流水线是最好的。如果是 I2C 加密（数据率才几百 Kbps），循环结构最经济。

12.6 踩坑：加了流水线但忘了延迟 data-valid 信号

几年前我接到一个任务：优化一个 FIR 滤波器模块的时序。它在 Spartan-6 上只能跑到 80MHz，但系统的其他模块需要 150MHz。原因是 FIR 滤波器内部的乘加树组合路径太长，成为全系统的瓶颈。

解决方案很直接——在乘加树的中间插入三级流水线寄存器。修改完后仿真过的，时序报告显示 fmax 提升到了 180MHz。下载到板子上——输出波形完全是错误的，前几个采样点的数据根本不连续。

排查了整整一个下午才发现：FIR 滤波器的数据输入有一个 data_in_valid 信号，我给它加了三组流水寄存器，所以它延迟了 3 个时钟周期。但滤波器的数据输出 data_out_valid 我只配置了一级寄存器——它是从最后一级的组合逻辑直接驱动的！于是下游模块在第 3 个时钟周期收到 data_out_valid 时，以为输出数据已经准备好了，但其实流水线里的第三级还在处理第三个采样点，输出端口上才是刚刚通过第一级的第一个采样点的中间结果。

修复很简单——保证 data_out_valid 经过和主数据路径完全相同的流水级数（即同样的寄存器延迟）。将 valid 信号像数据信号一样逐级寄存，每一级都增加一个寄存器位。

这是流水线设计中最容易犯、也最容易忽视的错误——控制信号（valid、ready、last）必须和数据路径保持完全相同的流水深度。一个好习惯是将控制信号与数据信号在同一个 always 块中一起寄存——这样永远不会不同步。

流水线技术是 FPGA 高性能设计的基础——没有流水线，你的设计就只能跑在几十 MHz。但它带来的更高工作频率也意味着新的挑战：当同一个设计中有多个不同的时钟速度的模块时，它们如何交换数据？这就引出了 FPGA 设计中最棘手的问题，也是下一章的主题——跨时钟域（CDC）处理。
