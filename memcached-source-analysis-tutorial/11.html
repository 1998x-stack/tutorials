第11章 Item生命周期：分配、访问与删除

大家好，我是你们的讲师。上一章我们解剖了 item 结构体——这具"尸体"的每个字段我们都看清楚了。但结构体本身是静态的，真正让系统活起来的是 item 在内存中经历的生老病死。这一章我们就追踪一个 item 的完整生命周期：从 set 命令触发的分配，到 get 命令触发的访问，再到过期或淘汰导致的删除。

item 的生命周期涉及三个核心模块的协作：slab allocator（提供物理内存）、assoc（哈希表——提供 key 到 item 的映射）、LRU（提供淘汰策略）。我们已经在第 7-9 章详细讨论了 slab allocator，本章会简要涉及 assoc 和 LRU 的部分接口，但它们的内部实现要留到第 12、13 章再深入展开。

do_item_alloc()：item 的"出生"

当客户端发送 set 命令时，协议解析完成后会调用 do_item_alloc()。这个函数是 item 生命周期的起点，包含三条"备降路径"：

item *do_item_alloc(const char *key, const size_t nkey,
                    const unsigned int flags, const rel_time_t exptime,
                    const int nbytes) {
    uint8_t clsid;
    item *it = NULL;

    // 第一步：计算 item 总大小，找到合适的 slab class
    const size_t ntotal = ITEM_ntotal(nkey, nbytes);
    clsid = slabs_clsid(ntotal);
    if (clsid == 0) return NULL;  // item 太大，超过 item_size_max

    // 路径一：尝试从 slab allocator 直接分配
    it = (item *)do_slabs_alloc(clsid, 0, 0);

    if (it == NULL) {
        // 路径二：尝试 LRU 淘汰后重试
        it = do_item_alloc_pull(ntotal, clsid);
    }

    if (it != NULL) {
        // 初始化 item 的各个字段
        ITEM_key(it) = key;
        it->nkey = nkey;
        it->exptime = exptime;
        it->nbytes = nbytes;
        it->it_flags = 0;       // 尚未 link
        it->refcount = 1;       // 初始引用计数为 1
        it->slabs_clsid = clsid;
        it->next = it->prev = it->h_next = NULL;
        // ... 复制 flags 到 suffix 等
    }

    return it;
}

三条路径的优先级很清晰：

路径 条件 代价 成功概率
1. do_slabs_alloc() 直接分配 slots 空闲链表不为空 O(1)，无锁或轻量锁 通常 > 99%（正常负载下）
2. LRU 淘汰后重试 slots 为空但内存未超限 需要驱逐一个旧 item，涉及哈希表删除 + slab free 中等（取决于内存压力）
3. 返回 NULL 所有路径都失败 客户端看到 SERVER_ERROR out of memory 极少发生

do_item_alloc_pull() 是路径二的入口。它的逻辑是：从 LRU 尾部淘汰一个 item，释放其 chunk，然后重新尝试 do_slabs_alloc()。如果一次不够，它会循环淘汰直到成功或确认没有可淘汰的 item。

item_link() 与 do_item_unlink()：连接与断开

分配后 item 还只是一块"初始化过的内存"，对客户端不可见。需要调用 item_link() 将 item 加入哈希表和 LRU 链表，客户端才能通过 get 找到它。

int item_link(item *it) {
    // 1. 加入哈希表，使其可通过 key 被查找
    assoc_insert(it);

    // 2. 加入 LRU 链表（具体插入逻辑下一章详解）
    item_link_q(it);  // 将 item 插入到 HOT LRU 头部

    // 3. 更新统计
    stats.curr_items++;
    stats.total_items++;

    // 4. 设置 LINKED 标志
    it->it_flags |= ITEM_LINKED;
    it->it_flags &= ~ITEM_SLABBED;

    return 1;
}

与 item_link() 对称的是 do_item_unlink()，在删除 item 时调用：

void do_item_unlink(item *it) {
    // 1. 从 LRU 链表中移除
    item_unlink_q(it);

    // 2. 从哈希表中移除
    assoc_delete(ITEM_key(it), it->nkey);

    // 3. 更新统计
    stats.curr_items--;

    // 4. 清除 LINKED 标志
    it->it_flags &= ~ITEM_LINKED;

    // 5. 引用计数减 1，如果归零则释放内存
    item_free_or_reuse(it);
}

注意 do_item_unlink() 的最后一步：item_free_or_reuse(it)。这里触发了引用计数的递减和可能的内存释放——但释放不一定发生，因为可能还有其他代码在引用这个 item（比如正在响应 get 命令）。

refcount 引用计数：为什么不能直接释放

引用计数是 item 生命周期中最重要的安全机制。基本规则是：

  • 计数 > 0：item 正在被使用，不能释放
  • 计数 == 0：没有人在使用，可以归还给 slab allocator

引用计数的变化场景：

// do_item_alloc() 之后：refcount = 1（刚刚分配）
// do_item_link() 之后：refcount = 2（分配 + 哈希表引用）
// 客户端 get 命中时：refcount++（正在被读取，防止释放）
// do_item_get() 返回后：refcount--（读取完毕）
// 客户端 delete 时：do_item_unlink() → refcount--（去掉哈希表引用）
//                    如果 refcount 归零 → 释放 chunk

为什么需要这个机制？考虑一个并发场景：客户端 A 正在 get 一个 item（已经在响应中读取这个 item 的数据），同时客户端 B delete 了这个 key。如果没有引用计数，B 的 delete 操作会直接把 item 的 chunk 释放回 slab allocator，然后 slab allocator 可能立即把这个 chunk 分配给另一个新 item——A 继续读取的就是"别人的数据"。这就是经典的 use-after-free bug。

有了引用计数后，do_item_unlink() 会在 B 请求时把 item 从哈希表和 LRU 中移除（所以新的 get 请求不会再找到它），但真正的内存释放要等到 A 完成读取（refcount 归零）后才进行。

设计哲学：Memcached 的引用计数是一种"乐观锁"——它不阻止并发访问，但在访问期间保证内存不会被回收。这与 Redis 的"单线程事件循环"（天然不需要引用计数）形成了有趣的对比。Memcached 选择多线程 + 引用计数，Redis 选择单线程 + 无锁。两种策略各有优劣，但都保证了数据一致性。

do_item_get()：查找并更新访问时间

当客户端执行 get mykey 命令时，do_item_get() 被调用：

item *do_item_get(const char *key, const size_t nkey) {
    item *it;

    // 1. 在哈希表中查找
    it = assoc_find(key, nkey);

    if (it != NULL) {
        // 2. 检查是否过期
        if (it->exptime != 0 && current_time > it->exptime) {
            // 过期了！不返回，但也不在这里删除——交给 LRU crawler
            do_item_unlink(it);
            it = NULL;
        } else {
            // 3. 未过期：更新访问时间 + LRU 提升
            it->refcount++;
            it->time = current_time;
            it->it_flags |= ITEM_ACTIVE;
            // LRU bump：将 item 在 LRU 队列中前移
            // （下一章详细讲）
        }
    }

    return it;
}

这里有一个关键设计：lazy expiration（惰性过期）。Memcached 不设专门的后台线程扫描过期 item，而是在 do_item_get() 中"碰巧"发现某个 item 过期了，才把它删除。这种策略的好处是不需要定时扫描的开销——如果 99% 的过期 item 在被访问之前就已经被 LRU 淘汰掉了，那专门做一遍过期扫描就是浪费 CPU。

当然，纯惰性过期也有问题：如果大量 item 设置了 TTL 但再也没有被访问过，它们会一直占据内存直到 LRU 链表的尾部被淘汰。为了解决这个问题，Memcached 引入了 LRU crawler 后台线程（下一章会详细讲），它会周期性地扫描 LRU 链表，回收那些已过期但未被访问的 item。

踩坑：refcount 泄漏导致的内存不可回收

现在来分享一个我在 debug 过程中遇到的经典 case。有一次我们的 Memcached 实例 stats 显示 curr_items 只有 50 万，但 RSS（实际物理内存占用）已经达到 8GB，远超理论估算值——每个 item 平均 200 字节，50 万 × 200B ≈ 100MB，就算加上 slab allocator 的预分配，也不应该到 8GB。

用 stats slabs 一看，free_chunks 几乎为零，但很多 item 的 refcount 异常偏高——有三四千个 item 的 refcount 一直在 3-5 之间，从未降回 0。这意味着这些 item 的 chunk 永远无法释放。

经过漫长的代码审查，最终发现是在我们定制的一个统计分析模块中，有一段代码在 do_item_get() 返回 item 后做了额外的处理，但在处理完成后忘记调用 item_remove() 来递减 refcount。这样就造成了"引用泄漏"——每次 get 命中，refcount 就 +1 但不 -1，日积月累，refcount 永远无法归零。

教训总结：引用计数机制虽然安全，但它要求调用链上的每一个环节都严格遵守"配对"规则——item_get() 必须配对 item_remove()，do_item_link() 必须配对 do_item_unlink()。任何一处的泄漏都会导致 item 变成"僵尸"——明明已经不可见了（哈希表中已删除），但 chunk 无法被回收。建议在开发扩展功能时，用 stats items 命令定期检查 refcount 分布，如果发现大量 item 的 refcount > 2，需要警惕。

另外，Memcached 源码本身也提供了一个"应急"手段：lru_crawler 在 meta 模式下会报告 refcount 异常的 item。在 1.6.41 版本中，甚至可以设置 expirezero_old 强制回收 refcount 为正但 ITEM_LINKED 标志已清除的 item。

生命周期全景图

现在用一张图来总结 item 的生命周期：

  客户端 set 命令
        │
        ▼
  do_item_alloc()      ──→ 从 slab allocator 获取 chunk
        │                     （先查 slots → 再 LRU 淘汰 → 最后 new page）
        ▼
  item_link()           ──→ assoc_insert() + item_link_q()
        │                     （插入哈希表 + 插入 LRU HOT 队列）
        ▼
  ┌──────────────┐
  │   活跃状态     │   ←── 客户端可 get/gets 命中
  │ ITEM_LINKED   │
  │ refcount >= 1 │
  └──────┬───────┘
         │
    ┌────┴────┐
    │         │
 过期      删除 / 淘汰
    │         │
    ▼         ▼
  惰性过期    do_item_unlink()
  (get 时)     │
    │         ├── assoc_delete() 从哈希表移除
    │         ├── item_unlink_q() 从 LRU 移除
    │         └── item_free_or_reuse()
    │                 │
    ▼                 ▼
  LRU crawler    refcount == 0?
  扫描回收           │
                是     否
                 │      │
                 ▼     ┌┘
           do_slabs_free()   等待 refcount 归零
                 │
                 ▼
           chunk 回到 slots 链表
           （等待下一次 do_item_alloc）

这张图反映了我们在本章讨论的所有内容：从分配、到链接、到访问、再到回收的完整闭环。
