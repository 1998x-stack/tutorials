第8章 Slab Allocator初始化与分配

大家好，我是你们的讲师。上一章我们从设计哲学层面理解了 slab allocator 为什么选择"预分配+固定 chunk size"这条路——避免内存碎片、分配效率高、可预测性强。但纸上得来终觉浅，哲学层面的理解终究是空中楼阁。今天我们就直接打开 slabs.c，从 slabs_init() 出发，一行一行追踪 slab allocator 从出生到第一次分配内存的全过程。

slabs.c 总共大约 700 行，是 Memcached 源码中最核心也最精巧的模块之一。这一章我们会把 slabs.c 的前半部分讲透——初始化、page 分配、chunk 切割。第 9 章再讲回收和 rebalance。好，我们直接开始读代码。

slabs_init()：启动的起点

slabs_init() 是整个 slab allocator 的入口。它在 memcached.c 的 main() 函数中早期被调用，在参数解析之后、网络连接打开之前。它的函数签名如下：

void slabs_init(const size_t limit, const double factor, const bool prealloc)

三个参数的含义非常清晰：

  • limit：最大内存上限（bytes），对应命令行 -m 参数，默认 64MB
  • factor：chunk 增长因子，对应命令行 -f 参数，默认 1.25
  • prealloc：是否预分配所有内存，对应 -L 参数，默认 false

函数的核心是一个循环，从 slab class 1（POWER_SMALLEST）初始化到 slab class 最大编号（POWER_LARGEST），直到 chunk size 超过 item_size_max / factor 为止。我们来看简化后的代码：

void slabs_init(const size_t limit, const double factor, const bool prealloc) {
    int i = POWER_SMALLEST - 1;
    unsigned int size = sizeof(item) + CHUNK_ALIGN_BYTES;

    memset(slabclass, 0, sizeof(slabclass));

    while (++i < POWER_LARGEST && size <= settings.item_size_max / factor) {
        if (size % CHUNK_ALIGN_BYTES)
            size += CHUNK_ALIGN_BYTES - (size % CHUNK_ALIGN_BYTES);

        slabclass[i].size = size;
        slabclass[i].perslab = settings.item_size_max / size;

        // 更新 size 为下一个 class 的 chunk 大小
        size *= factor;
    }

    if (prealloc) {
        // 预分配全部内存，每个 slab class 分配一个 page
        for (i = POWER_SMALLEST; i < slabclass_max; i++) {
            do_slabs_newslab(i);
        }
    }
}

这段代码虽然不长，但有几个关键细节值得反复品味。我们逐个来看。

起始大小的确定

第一个 slab class 的 chunk size 是 sizeof(item) + CHUNK_ALIGN_BYTES。sizeof(item) 在 64 位系统上通常是 56 字节，加上 8 字节对齐余量得到 64 字节，再经过 8 字节对齐后，实际是第一级 slab class 的 chunk size —— 在经典配置下最终对齐到 96 字节。这个大小刚好可以存放一个很小的 key-value 对（key 最长 250 字节，但 value 可以很短）。

为什么从 96 字节而不是更小开始？因为 item 结构体本身需要占用大约 56 字节，再加上 key、flags、CAS 等元数据，小于 96 字节的 chunk 几乎放不下任何有用的数据。从 96 起步是一个工程上的折中。

chunk_size × factor：增长算法的数学

这是 slab allocator 最精妙的地方之一。每次循环结束后，size *= factor，下一级 slab class 的 chunk 大小是上一级的 factor 倍。默认 factor=1.25，我们来算一算前几级 slab class 的实际大小：

注意到两个有意思的现象：

第一，实际倍数不是精确的 1.25——因为每次都要做 CHUNK_ALIGN_BYTES（8 字节）对齐，导致一些 slab class 的倍数有微小波动。比如 class 3 是 152 / 120 ≈ 1.267，class 4 是 192 / 152 ≈ 1.263。这是因为 120 × 1.25 = 150，对齐到 8 变成 152。这个波动是正常现象，不影响整体设计。

第二，perslab 从一万多在 class 1 急剧下降到 class 49 的 1。这意味着大 slab class 的每个 page 只能切出很少的 chunk，甚至只有 1 个。道理很简单：一个 page 固定 1MB，一个 chunk 就占 640KB，那一个 page 只能放 1 个 chunk（实际上还会剩不少空间，但不会跨 page 使用）。

关键理解：factor=1.25 时，从 96B 到 1MB 大约会生成 49 个 slab class。如果你调大 factor（比如 2.0），slab class 数量减少（大约 21 个），但每个 class 内部的 chunk 大小增长更快，内存浪费会增加。反之，调小 factor（比如 1.1）会增加 slab class 数量到上百个，每个 class 获得的内存页面更少，但内存利用率更高。默认值 1.25 是经过大量生产验证的"甜点"。

CHUNK_ALIGN_BYTES：8 字节对齐的秘密

你可能觉得内存对齐是一个低级到不值得关注的问题。但在高性能场景下，不对齐的内存访问会导致额外的 cache line 访问、甚至在某些架构上触发 SIGBUS。Memcached 的做法是：每个 chunk 的起始地址和大小都按照 8 字节对齐。

#define CHUNK_ALIGN_BYTES 8

// 对齐宏：将 size 向上对齐到 8 的倍数
if (size % CHUNK_ALIGN_BYTES)
    size += CHUNK_ALIGN_BYTES - (size % CHUNK_ALIGN_BYTES);

为什么是 8 字节而不是 4 字节或 16 字节？

  • 4 字节：现代 CPU 的 cache line 通常是 64 字节，4 字节对齐对 cache 性能帮助不大
  • 8 字节：正好对齐到 64 位 CPU 的原生字长。大多数指针和 64 位整数在 8 字节对齐时访问效率最高
  • 16 字节：SSE/AVX 指令集偏好 16 字节对齐，但 Memcached 并不使用 SIMD 指令处理 item 数据，16 字节对齐反而会浪费更多空间

所以 8 字节是对齐收益和空间浪费之间的最优解。别忘了 slab allocator 本身就是为减少内存碎片设计的，如果对齐粒度过大，反而会制造"内部碎片"。

do_slabs_newslab()：申请新的 Page

初始化完成后，slab class 有了配置信息但还没有实际内存。真正的内存分配发生在 do_slabs_newslab() 中：

static int do_slabs_newslab(const unsigned int id) {
    slabclass_t *p = &slabclass[id];
    int len = settings.slab_page_size;  // 默认 1MB
    unsigned int count;

    // 从 OS 申请一个 page 大小的内存
    void *ptr = memory_allocate((size_t)len);
    if (ptr == NULL) return 0;

    // 将这个 page 切分成 chunk，串入空闲链表
    count = split_slab_page_into_freelist(ptr, id);

    // 更新统计
    p->slabs++;           // 已使用的 page 数 +1
    p->total_pages++;     // 总 page 数 +1
    p->free_chunks += count;
    p->free_chunks_total += count;

    return 1;
}

这个函数做了三件清晰的事情：

  1. 从操作系统申请 1MB 连续内存（对应 settings.slab_page_size）
  2. 把这一整块内存切分成固定大小的 chunk
  3. 更新 slab class 的统计信息

memory_allocate() 内部使用 mmap()（Linux）或 VirtualAlloc()（Windows）进行大块内存分配。当 -L 启用预分配时，这个过程在 slabs_init() 中一次性完成全部 slab class 的 page 分配。否则，page 是懒分配的——只有当一个 slab class 的 free list 用尽时，才调用 do_slabs_newslab()。

split_slab_page_into_freelist()：将 Page 切割成 Chunk

这是最底层的内存整理函数。它把一整块 1MB 的连续内存，按 slabclass[id].size 切成等大小的块，并将这些块的起始地址串成一条"隐式空闲链表"：

static int split_slab_page_into_freelist(void *ptr, const unsigned int id) {
    slabclass_t *p = &slabclass[id];
    int count = 0;

    for (int i = 0; i < p->perslab; i++) {
        // 将当前 chunk 的起始地址通过指针强制转换
        // 把 chunk 当作 item 来看，用它的 next 指针串成链表
        item *it = (item *)((char *)ptr + i * p->size);
        it->slabs_clsid = (uint8_t)id;

        // 头插法：新 chunk 指向当前空闲链表头
        it->next = p->slots;
        p->slots = it;
        count++;
    }

    return count;
}

这里有一个非常精巧的设计：空闲内存块不需要额外的空闲链表节点。因为每个 chunk 都是"空闲"的，我们直接借用 item 结构体的 next 指针字段来形成链表。当这个 chunk 被分配出去时，next 指针会被 item 的正常使用覆盖——但没关系，因为那时它已经不在空闲链表里了。

这种"内存即元数据"的设计在操作系统内核中很常见（比如 Linux slab allocator），但在用户态程序中并不普遍。Memcached 把它运用到了极致。

设计精髓：slab allocator 的 slots 空闲链表是一个"隐式链表"——它不额外占用任何内存。每个空闲 chunk 的头部几个字节（恰好是 item->next 指针的位置）被重用来指向下一个空闲 chunk。分配时 O(1) 出栈，释放时 O(1) 入栈。这正是 slab allocator 比通用 malloc 快得多的根本原因之一。

do_slabs_alloc()：分配的核心入口

当 Memcached 需要存储一个新 item 时，最终会调用 do_slabs_alloc()：

void *do_slabs_alloc(const unsigned int id) {
    slabclass_t *p = &slabclass[id];

    // 1. 先尝试从空闲链表拿
    if (p->slots != NULL) {
        p->slots_cur--;
        item *it = p->slots;
        p->slots = it->next;  // slots 指针移到下一个空闲 chunk
        it->next = NULL;      // 清空 next，交给上层使用
        return it;
    }

    // 2. slots 空了，尝试申请新 page
    if (p->slab_current != NULL || do_slabs_newslab(id) != 0) {
        if (p->slab_current != NULL) {
            // 从当前正在使用的 page 中切一块
            // ...
        }
    }

    // 3. 内存用尽，返回 NULL（触发 LRU 淘汰）
    return NULL;
}

分配策略的分层非常清晰：

  1. 第一优先：空闲链表（slots）——这是最快路径，O(1) 出栈，无任何系统调用
  2. 第二优先：当前 page 的未使用空间——从 slab_current 指向的 page 中切出下一个 chunk
  3. 第三优先：申请新 page——调用 do_slabs_newslab() 从操作系统（或预分配池）中获取 1MB 内存
  4. 最后手段：返回 NULL——触发上层调用 LRU 淘汰，腾出空间

注意一个细节：步骤 1 中，p->slots_cur-- 的递减发生在取出 chunk 之前。这是因为 slots_cur 在初始化时就已经等于 free_chunks 的总数。如果先取 chunk 再递减，在并发场景下可能会有短暂的不一致——不过从 1.6 版本的代码来看，这里已经是单线程访问（有全局锁保护），所以没有问题。

踩坑：factor=1.25 引发的 OOM 血案

到这里，我想分享一个我在生产环境中亲身经历的教训。有一次，我们有一个 Memcached 集群运行在 256MB 的容器中，所有配置看起来都很正常——默认 factor=1.25，默认 item_size_max=1MB。上线后运行平稳了几个月，直到某天突然开始 OOM，容器被 kill 重启。

排查过程是这样的：我们用 stats slabs 查看 slab class 分布，惊讶地发现竟然创建了 49 个 slab class。每个 slab class 至少要分配 1 个 page（1MB）。也就是说，仅仅是 slab allocator 初始化阶段，就已经占用了 49MB 内存。加上每个 slab class 在运行时按需分配更多的 page，实际内存占用了 200+ MB——已经逼近了 256MB 的限制。

真正致命的是什么呢？是"踩踏效应"：当某个 slab class 内存耗尽时，它会触发 LRU 淘汰，但淘汰的 item 可能属于另一个 slab class，导致那个 class 的 page 被释放、重新分配——这个过程中额外分配了新的 page，踩过了内存上限。

教训总结：factor=1.25 在大多数场景下是最优选择，但它在内存受限的环境中会产生 49 个 slab class，每个至少占 1 个 page。如果你的 -m 设置得很小（比如 256MB），建议调大 factor（比如 1.5 甚至 2.0）来减少 slab class 数量，或者降低 item_size_max（比如 512KB）来减少每个 page 的浪费。不要天真地以为 -m 64 就只占用 64MB——slab allocator 的最小"启动成本"可能比你以为的要高得多。

另外，还有一个相关的坑：slab_automove（我们下一章会详细讲）在低内存环境中可能频繁触发 page 迁移，进一步加剧内存压力。如果你的容器内存真的很紧张，建议显式设置 -o slab_automove=0 禁用自动迁移。

小结与动手实验

到现在为止，我们完成了 slab allocator 从初始化到分配的全链路分析。你可以打开源码，对照着 slabs_init() → do_slabs_newslab() → split_slab_page_into_freelist() → do_slabs_alloc() 这条路径，一行一行地读，每一步都对应到我们的讲解。

建议你在 GDB 中给 slabs_init() 设一个断点，启动后单步跟踪：

cgdb memcached
(gdb) b slabs_init
(gdb) r -m 64 -f 1.25 -vv
(gdb) n  # 逐行观察 slabclass 数组的填充过程

观察每执行一行循环后，slabclass[i].size 和 slabclass[i].perslab 的值变化，这会让你对 slab class 的增长规律建立直观感受。
