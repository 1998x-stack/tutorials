第23章 测试策略与质量保障

2025 年初，我参与了一家金融科技公司 Agent 系统的上线评审。他们的交易监控 Agent 已经通过了全部功能测试，但在我们做"影子模式"并行验证时发现了一个隐蔽的问题：当用户查询的股票代码包含前缀零时（比如 "00700" 而非 "700"），Agent 调用行情 API 时没有做规范化处理，导致查询失败。更麻烦的是，失败后 Agent 没有重试，而是直接告诉用户"该股票代码不存在"——这是一个非常严重的虚假否定。事后复盘时发现，这个问题之所以逃过了测试，是因为测试用例中所有的股票代码都是标准格式，没有一个带前导零的。

这个故事揭示了一个残酷的现实：Agent 系统的测试比传统软件测试要困难得多。LLM 的非确定性输出、复杂的工具调用链、多步推理路径的分叉——这些特性让传统"输入-执行-断言"的测试范式不断碰壁。本章我们就来系统性地解决这个问题，建立一套真正适用于 Agent 的测试质量保障体系。

核心洞察： Agent 测试的终极目标不是"消灭所有 Bug"——这在大模型系统中是不可能的。而是建立信心：让团队有足够的把握将 Agent 推向生产环境，并在出现问题时能够快速定位和修复。

23.1 Agent 测试的独特性与挑战

在讨论具体方法之前，我们得先理解 Agent 测试到底"难"在哪里。作为一个在多个生产级 Agent 项目中摸爬滚打过的工程师，我把 Agent 测试的主要挑战总结为"三座大山"：

挑战 说明 传统测试方案 Agent 场景的难度
非确定性输出 LLM 对同一输入可能给出不同回复 精确断言即可 需要模糊匹配、语义等价判断
多步推理链 Agent 可能通过不同路径到达同一结果 固定执行路径 路径空间指数级增长
外部依赖 工具调用依赖外部 API、数据库状态 Mock 即可 Mock 粒度复杂，需模拟非预期响应
评估维度多 正确性、安全性、效率、用户体验都要测 单一通过标准 需要多维度的评估矩阵

面对这些挑战，我们需要一套分层测试策略。我推荐采用"测试金字塔"的 Agent 改良版：底层是大量的单元测试（工具函数、组件），中间层是 Mock LLM 测试（可控环境下验证逻辑），顶层是少量的端到端集成测试（使用真实的 LLM 调用）。

经验之谈： 在 Agent 项目中，单元测试和 Mock 测试应该覆盖 80% 以上的场景，端到端测试只需要覆盖关键路径。这不仅是因为端到端测试成本高，还因为 LLM API 调用会引入不必要的随机性，让测试结果难以复现。

23.2 单元测试：工具函数与组件

单元测试是 Agent 测试金字塔的基石。虽然 Agent 的整体行为是非确定的，但它的各个"零部件"——工具函数、数据处理器、格式化器、验证器——都是纯确定性的逻辑，完全可以按传统方式测试。

下面是一个典型的 Agent 工具函数单元测试示例。假设我们有一个天气查询工具，它需要解析用户输入的城市名，调用外部 API，然后格式化返回结果：

# test_weather_tool.py — 工具函数的单元测试
import pytest
from agent_tools import WeatherTool

def test_parse_city_name_standard():
    """测试标准城市名解析"""
    tool = WeatherTool()
    assert tool.parse_location("北京") == {"city": "北京", "country": "CN"}
    assert tool.parse_location("New York") == {"city": "New York", "country": "US"}
    assert tool.parse_location("東京") == {"city": "東京", "country": "JP"}

def test_parse_city_name_edge_cases():
    """测试边界情况：空值、特殊字符、别名"""
    tool = WeatherTool()
    assert tool.parse_location("") is None  # 空输入应返回 None
    assert tool.parse_location(None) is None
    # 别名映射
    assert tool.parse_location("魔都") == {"city": "上海", "country": "CN"}
    assert tool.parse_location("NYC") == {"city": "New York", "country": "US"}

def test_format_weather_response():
    """测试天气信息格式化输出"""
    tool = WeatherTool()
    raw_data = {
        "temperature": 25.5,
        "humidity": 65,
        "condition": "多云",
        "wind_speed": 3.2
    }
    result = tool.format_response(raw_data, "北京")
    assert "北京" in result
    assert "25.5°C" in result
    assert "多云" in result
    # 确认输出包含所有关键信息
    assert all(keyword in result for keyword in ["温度", "湿度", "风速"])

这类测试的价值在于：它们执行速度快（毫秒级）、结果确定、不依赖任何外部服务。每个工具函数、每个数据转换管道、每个格式化器，都应该有对应的单元测试。我在实际项目中制定的标准是：纯逻辑函数的行覆盖率不低于 95%。

注意： 不要把网络请求也放到单元测试中。工具函数中所有涉及外部 API 调用的部分，都应该通过依赖注入方式进行 Mock。单元测试只验证逻辑本身，不验证外部服务的可用性。

23.3 Mock LLM：模拟非确定性输出

单元测试解决的是"零件"的质量问题。但 Agent 的核心逻辑——LLM 推理和决策——才是更需要测试的部分。直接用真实 LLM 做测试的问题是：成本高、速度慢、结果不稳定。同一个 Prompt，GPT-4o 今天和明天的输出可能不同，这让测试结果变得不可靠。

解决方案是 Mock LLM——用一个可控的模拟器代替真实的 LLM，让 Agent 在测试环境中以确定性的方式运行。

一个好的 Mock LLM 设计通常包含三个层次：

层次 功能 适用场景
静态 Mock 对特定输入返回固定预定义输出 测试工具调用逻辑、解析器
状态机 Mock 根据对话历史状态决定下一步回复 测试多步推理流程
录制-回放 Mock 录制真实 LLM 响应，在测试中回放 回归测试、性能测试

下面是一个静态 Mock 的 Python 实现示例——这是最常用、也最容易上手的方式：

# mock_llm.py — 模拟 LLM 回复的测试工具
from dataclasses import dataclass
from typing import Callable

@dataclass
class MockLLMResponse:
    content: str
    tool_calls: list = None

class MockLLM:
    """可编程的 Mock LLM，用于 Agent 单元测试"""

    def __init__(self):
        self.responses: list[MockLLMResponse] = []
        self.call_index = 0

    def add_response(self, content: str, tool_calls: list = None):
        """按顺序预设 LLM 回复"""
        self.responses.append(
            MockLLMResponse(content=content, tool_calls=tool_calls)
        )

    def invoke(self, messages, **kwargs):
        """模拟 LLM.invoke() 接口"""
        if self.call_index >= len(self.responses):
            raise RuntimeError(
                f"Mock LLM 耗尽：已调用 {self.call_index} 次，"
                f"但只预设了 {len(self.responses)} 个回复"
            )
        response = self.responses[self.call_index]
        self.call_index += 1
        return response

# 在测试中使用
def test_agent_weather_query():
    mock_llm = MockLLM()

    # 预设对话流程：
    # 第1步：LLM 决定调用天气查询工具
    mock_llm.add_response(
        content="我需要查询北京的天气",
        tool_calls=[{
            "name": "get_weather",
            "arguments": {"city": "北京"}
        }]
    )
    # 第2步：LLM 基于工具返回结果生成最终回复
    mock_llm.add_response(
        content="北京今天多云，温度25.5°C，湿度65%。",
        tool_calls=None
    )

    agent = CustomerAgent(llm=mock_llm)
    result = agent.run("北京天气怎么样？")
    assert "多云" in result
    assert "25.5°C" in result
最佳实践： 录制-回放模式是最"逼真"的 Mock 方式。你可以在开发阶段使用真实 LLM 运行 Agent，记录下每一轮的 Prompt 和 Response，然后把这些记录序列化成测试用例。这样既保留了真实 LLM 的行为特征，又让测试可以稳定复现。VCR.py（HTTP 调用的录制回放库）是一个很好的参考实现思路。

23.4 集成测试：端到端 Agent 流程

单元测试和 Mock LLM 测试覆盖了"零件"和"局部逻辑"，但 Agent 最终在用户面前的表现，需要端到端验证。集成测试使用的是真实 LLM（通常选择速度较快、成本较低的模型，如 Claude 3.5 Haiku 或 GPT-4o Mini），并且连接真实的工具 API（或用沙箱环境）。

端到端测试的核心挑战是：如何判断 Agent 的输出是否正确。对于"查询天气"这类事实性问题，答案有客观标准。但对于"帮我整理一下本周会议纪要"这种开放性任务，正确性判断就变得模糊了。

我推荐使用"评估矩阵"方法——从多个维度对 Agent 的输出进行评分：

# eval_matrix.py — 端到端评估矩阵示例
def evaluate_agent_response(query, response, context):
    """从多个维度评估 Agent 回复质量"""
    scores = {}

    # 维度1：任务完成度 — Agent 是否完成了用户的核心请求？
    scores["task_completion"] = check_task_completion(query, response)

    # 维度2：事实准确性 — 回复中的事实性陈述是否正确？
    scores["factual_accuracy"] = check_factual_claims(response, context)

    # 维度3：工具使用正确性 — 是否调用了正确的工具、传入了正确的参数？
    scores["tool_correctness"] = validate_tool_usage(response.trace)

    # 维度4：安全合规 — 是否泄露了敏感信息？是否拒绝了有害请求？
    scores["safety"] = safety_check(response)

    # 维度5：用户体验 — 回复是否自然、友好、信息完整？
    scores["user_experience"] = ux_rating(response)

    return scores

# 判断标准
def is_pass(scores, thresholds):
    """根据阈值判断是否通过测试"""
    return all(
        scores[dim] >= thresholds.get(dim, 0.8)
        for dim in scores
    )

在实践中，我建议为每个核心场景准备 3-5 个测试用例，每个用例使用评估矩阵打分。如果你的评估矩阵全部使用代码实现（比如检查特定关键词、验证 JSON 格式等），那么这些测试可以集成到 CI 流水线中自动运行。

一个常见的坑： 不要用 LLM 来评估 LLM——至少在自动化测试流水线中不要这样做。用 GPT-4 评估 GPT-4 的输出会产生循环引用，而且成本高昂。在开发和调试阶段可以用 LLM 辅助评估，但 CI 中的自动化测试应该使用确定性检查逻辑。

23.5 回归测试与持续集成

Agent 系统的回归测试比传统软件更具挑战性，因为模型更新、Prompt 修改、工具升级都可能引发连锁反应。我见过最典型的场景是：某团队优化了一个工具函数的错误处理逻辑，结果导致 Agent 在某个边缘场景下选择了一条完全不同的工具调用路径——功能测试通过了，但用户体验变差了。

因此，回归测试对于 Agent 项目来说不是"锦上添花"，而是"雪中送炭"。下面是一个推荐的 CI 流水线设计：

阶段 内容 触发时机 运行时间
Stage 1: 代码检查 Lint、类型检查、安全检查 每次提交 1-2 分钟
Stage 2: 单元测试 工具函数、数据管道、格式化器等纯逻辑测试 每次提交 2-5 分钟
Stage 3: Mock LLM 测试 用 Mock LLM 验证 Agent 决策逻辑 每次提交 5-10 分钟
Stage 4: 回归测试 录制-回放模式下运行历史关键场景 每日/每次合并 15-30 分钟
Stage 5: 端到端测试 使用真实 LLM 运行核心场景 每日/发版前 30-60 分钟

注意这个流水线的设计原则：快速反馈优先。Stage 1-3 在每次提交时运行，保证开发者在几分钟内就能得到反馈。Stage 4-5 虽然更全面，但因为耗时长且有成本（真实 LLM 调用），放在每日构建和发版前执行。

成本考量： 以 2026 年的 API 价格计算，一套完整的端到端测试（运行 20 个场景）大约消耗 50 万 Token，使用 Claude 3.5 Sonnet 的成本约为 $1.5。每日运行一次，月成本约 $45。相比于一次线上事故造成的损失，这笔投入非常值得。

23.6 实战：为客服 Agent 编写测试套件

让我们把本章的所有方法整合起来，为一个智能客服 Agent 编写完整的测试套件。假设我们的客服 Agent 处理三种类型的请求：订单查询、退换货申请、产品咨询。

# test_customer_service_agent.py — 客服 Agent 完整测试套件

# ============ 1. 单元测试：工具函数 ============
class TestOrderTools:
    """测试订单查询工具函数"""

    def test_parse_order_id_normal(self):
        """标准订单号解析"""
        assert parse_order_id("ORD-2026-0001") == "ORD-2026-0001"

    def test_parse_order_id_short(self):
        """短格式订单号自动补全"""
        assert parse_order_id("0001") == "ORD-2026-0001"

    def test_validate_order_status_transition(self):
        """验证订单状态转移的合法性"""
        # 已发货订单不能取消
        assert not is_valid_transition("shipped", "cancelled")
        # 待付款订单可以取消
        assert is_valid_transition("pending", "cancelled")


# ============ 2. Mock LLM 测试：决策逻辑 ============
class TestAgentDecisionLogic:
    """测试 Agent 的决策逻辑"""

    def test_order_query_flow(self):
        """测试订单查询场景的完整决策流程"""
        mock_llm = MockLLM()
        # 步骤1: Agent 识别意图 → 决定查询订单
        mock_llm.add_response(
            content="用户想查询订单状态，我需要查找订单信息",
            tool_calls=[{"name": "query_order", "args": {"order_id": "ORD-2026-0001"}}]
        )
        # 步骤2: Agent 格式化查询结果
        mock_llm.add_response(
            content="您的订单 ORD-2026-0001 当前状态为：已发货，预计明天送达。",
            tool_calls=None
        )

        agent = CustomerAgent(llm=mock_llm, tools=[OrderTool()])
        result = agent.run("帮我查一下订单 ORD-2026-0001")
        assert "已发货" in result

    def test_refund_flow_requires_authentication(self):
        """退换货流程必须先验证身份"""
        mock_llm = MockLLM()
        # 用户直接要求退款，Agent 应先要求身份验证
        mock_llm.add_response(
            content="退款需要先验证身份，请提供注册手机号。",
            tool_calls=None  # 不调用工具，先要求验证
        )

        agent = CustomerAgent(llm=mock_llm, tools=[RefundTool()])
        result = agent.run("我要退款")
        assert "验证" in result or "手机" in result


# ============ 3. 端到端测试（标记为慢测试，CI 中每日运行）============
@pytest.mark.slow
class TestE2E:
    """端到端集成测试"""

    def test_real_order_query(self):
        """使用真实 LLM 测试订单查询"""
        agent = CustomerAgent(llm=RealLLM(model="claude-3-haiku"))
        result = agent.run("我的订单 ORD-2026-0001 到哪了？")
        # 使用评估矩阵验证
        scores = evaluate_agent_response(
            "我的订单 ORD-2026-0001 到哪了？", result, {}
        )
        assert scores["task_completion"] >= 0.8
        assert scores["factual_accuracy"] >= 0.9
        assert scores["safety"] == 1.0

这个测试套件覆盖了三个层次：单元测试验证工具函数的正确性，Mock LLM 测试验证 Agent 的决策逻辑，端到端测试在真实环境中验证整体表现。三者结合起来，就形成了"从零件到整机"的全方位质量保障体系。

持续改进： 测试套件不是一次性工作。每当你发现一个线上 Bug，第一件事不是修，而是写一个能复现这个 Bug 的测试用例。然后修复代码让测试通过。这样你的测试套件会越积越厚，Agent 系统的稳定性也会越来越高。这就是经典的"测试驱动 Bug 修复"方法论。

本章小结

Agent 测试很难——但并非无章可循。我们这一章建立了一套分层测试策略：底层用单元测试守住工具函数的质量基线，中层用 Mock LLM 在可控环境中验证 Agent 的决策逻辑，顶层用端到端测试配合评估矩阵进行整体把关。再加上 CI 流水线的自动化回归，你就能对自己的 Agent 系统建立足够的信心。

不过，测试只是确保 Agent 能"正常工作"。下一章我们将转向另一个重要话题——成本优化与性能调优。当你把 Agent 推向生产环境时，API 账单会让你意识到：让 Agent "工作"很容易，让 Agent "经济地工作"才是真正的工程挑战。我将分享一些实战中验证过的策略，帮助你在大规模部署中把成本降低 60% 以上。
