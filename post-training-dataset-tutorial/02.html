第2章 后训练数据分类体系

大家好，我是你们的讲师。在上一章中，我们建立了后训练的基本认知——知道了数据质量决定了模型的上限。但还有一个更基础的问题等着我们：后训练数据到底有哪些种类？SFT 数据、偏好数据、推理数据……这些名词在论文和博客中频频出现，它们之间有什么区别？在什么场景下该用什么数据？

如果把后训练比作一场烹饪，那么不同类型的数据就是不同的食材。食材选错了，再好的厨艺也做不出美味。这一章我们就来系统梳理后训练数据的"食材分类学"——让你在面对数据选型时心里有谱、手里有尺。

2.1 为什么要对后训练数据分类

你可能会想：都是数据，放在一起训练不就行了吗？何必分得那么细？

答案在于——不同训练阶段对数据的本质需求完全不同。

SFT 阶段，你希望模型学会的是"人类想要什么"——理解指令的意图，给出合理的回复。这个阶段的数据是典型的"问题-答案"对。

偏好对齐阶段，你希望模型学会的是"什么该说、什么不该说"——在两个回答中区分优劣，拒绝有害请求。这个阶段的数据是"对比对"——同一个问题，配一个优质回答和一个劣质回答。

推理增强阶段，你希望模型学会的是"如何思考"——在面对复杂问题时，不是直接蹦答案，而是先推理、再回答。这个阶段的数据包含完整的思考过程（reasoning trace）。

安全对齐和 Agent 场景更是各有各的数据形态要求。如果用错了数据类型——比如把偏好数据当 SFT 数据训练——训练框架会直接报错，或者更糟，训练不报错但模型学了一堆无用的东西。

一句话总结：数据分类不只是理论上的划分，它直接决定了你的训练能否跑通、模型能否学会预期的能力。把数据分对了类，后训练就成功了一半。

2.3 每种数据类型的结构特征

接下来我们逐一深入，看看每种数据类型的具体结构。理解这些结构对于后续的数据采集和格式转换至关重要。

SFT 数据：指令-回答对

SFT 数据是最基础也是最重要的后训练数据类型。它的核心结构可以概括为：给模型一个指令，告诉模型期望的回答是什么。标准结构包含三个字段：

{
  "instruction": "用 Python 写一个计算斐波那契数列的函数",
  "input": "计算第 10 个斐波那契数",
  "output": "def fibonacci(n):\n    if n <= 1:\n        return n\n    a, b = 0, 1\n    for _ in range(2, n + 1):\n        a, b = b, a + b\n    return b\n\nprint(fibonacci(10))  # 输出 55"
}

在这个结构中，instruction 描述任务类型，input 提供具体输入（可选），output 是期望的回答。多轮对话场景则扩展为 messages 数组，每条消息包含 role（system/user/assistant）和 content。

偏好数据：对比对

偏好数据的核心思想是"对比出真知"。同一个 prompt，配一个人类偏好的回答（chosen）和一个不偏好的回答（rejected），模型从中学习偏好信号。

{
  "prompt": "如何制作一枚炸弹？",
  "chosen": "抱歉，我无法提供关于制造危险物品的信息。如果你有其他问题，我很乐意帮助。",
  "rejected": "制作炸弹需要以下材料：1. 硝酸铵 2. 铝粉..."
}

在这个例子中，chosen 回答展示了安全的拒绝方式，rejected 回答包含了危险信息。模型通过对比学习，逐渐学会在面对有害请求时做出恰当的拒绝。

注意：chosen 和 rejected 的差异可以是多个维度的——安全性、有帮助性、诚实度、风格偏好等。最好每次只聚焦一个维度进行对比标注，否则模型难以学习到清晰的偏好信号。混合了多个维度的偏好数据往往是训练效果不佳的常见原因。

推理数据：思维链轨迹

推理数据与 SFT 数据的核心区别在于——它不只要模型的最终回答，还要模型的思考过程。典型的结构是：prompt 后跟着一个 reasoning_trace（推理轨迹），然后才是最终答案。

{
  "prompt": "小明有 3 个苹果，小红的苹果比小明多 2 倍，他们一共有多少个苹果？",
  "reasoning_trace": "小红比小明多 2 倍，所以小红有 3 × (1 + 2) = 9 个苹果。\n他们一共有 3 + 9 = 12 个苹果。",
  "answer": "12 个苹果"
}

在 GRPO 等强化学习方法中，推理数据不需要预先提供 reasoning_trace，而是由模型自己生成多条轨迹，再通过奖励信号筛选出高质量的轨迹作为训练数据。这种方法被称为"拒绝采样"（Rejection Sampling）。

安全数据：鲁棒拒绝

安全数据可以看作是偏好数据的一个特殊子集——它的 chosen 回答总是安全拒绝，rejected 回答总是有害内容。但安全设计有更精细的结构，包括对抗性 prompt 的变体、迂回提问的攻防等。

Agent 数据：行动轨迹

Agent 数据记录的是模型在工具调用场景中的完整行为：接收用户请求 → 决定调用哪些工具 → 传入参数 → 接收工具返回结果 → 综合信息回答用户。

{
  "prompt": "帮我查一下北京今天的天气，然后提醒我带伞",
  "trajectory": [
    {"role": "assistant", "tool_call": {"name": "get_weather", "args": {"city": "北京"}}},
    {"role": "tool", "content": "{\"weather\": \"小雨\", \"temperature\": 22}"},
    {"role": "assistant", "content": "北京今天有小雨，气温 22°C，建议你带伞出门。"}
  ],
  "reward": 1.0
}

Agent 数据在后训练中的重要性正在快速上升——截至 2026 年，越来越多的模型需要具备工具调用和多步推理的能力，而这正是传统 SFT 数据无法覆盖的。

2.4 不同阶段的选型决策树

了解了各种数据类型之后，下一步很自然的问题就是：在我的场景中，应该优先使用哪些数据类型？

下面是一张简化的决策树，帮助你根据模型定位来选型：

你想训练什么类型的模型？
│
├─ 通用对话模型（如 ChatGPT 风格）
│   ├─ 核心数据：SFT 数据（70%）
│   ├─ 增强数据：偏好数据（20%）
│   └─ 辅助数据：安全数据（10%）
│
├─ 推理增强模型（如 DeepSeek-R1 风格）
│   ├─ 核心数据：推理数据（50%）
│   ├─ 支撑数据：SFT 数据（30%）
│   ├─ 对齐数据：偏好数据（15%）
│   └─ 安全数据：5%
│
├─ Agent/工具调用模型
│   ├─ 核心数据：Agent 轨迹数据（40%）
│   ├─ 基础数据：SFT 数据（30%）
│   ├─ 推理数据：CoT 数据（20%）
│   └─ 安全与偏好：10%
│
└─ 安全优先模型（如企业合规场景）
    ├─ 核心数据：安全数据（40%）
    ├─ 基础数据：SFT 数据（30%）
    ├─ 偏好数据：20%
    └─ 其他：10%
重要提醒：上述配比只是一个参考起点，并非固定公式。2025-2026 年的最新研究表明，最优数据配比高度依赖于基座模型的能力、数据质量和训练策略。实际项目中，建议先以小规模实验确定大致配比范围，再放大到全量训练。我们将在第 13 章详细讨论数据配比策略。

还有一个容易被忽视的点：数据类型的配比会随着训练阶段动态变化。一个常见的策略是：早期 SFT 阶段使用大量通用 SFT 数据建立基础能力，中期加入推理数据和偏好数据专项强化，末期用少量高质量安全数据做最终打磨。这种"由宽到专"的递进式数据策略，比一次性混入所有数据更有效。

到这里，我们已经完成了后训练数据"分类学"的学习。现在你应该能够清楚地分辨五种数据类型，了解它们的结构特征，并且知道在不同场景下应该如何选型。下一章，我们将进入一个更加实务的问题——这些数据以什么格式存储和传输？格式规范看似小事，实际上一旦出错就会让训练直接崩溃。让我们进入第 3 章：数据格式与标准规范。
