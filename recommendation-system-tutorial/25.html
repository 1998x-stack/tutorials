第25章 LLM推荐的工业实践

上一章我们聊了LLM驱动推荐的各种前沿方案——LLM as Ranker、RAG推荐、对话式推荐、Agent推荐。读完之后你可能会有一个感觉：这些听起来都很美好，但到底有多少真的在大规模生产系统里跑起来了？

这一章，我们就来实地考察几家中国互联网巨头的LLM推荐落地实践。为什么聚焦中国公司？因为中国互联网公司在LLM推荐这个方向的投入是全世界最激进的——这里有最丰富的业务场景（电商、短视频、生活服务、出行），有最多的用户行为数据，更重要的是，有决心把新技术转化为业务增长的动力。

1. 快手OneRec系列：从判别到生成

快手在2023到2025年期间发布了一系列论文，完整展示了他们从传统推荐向生成式推荐转型的路线图，这个系列被称为OneRec。

OneRec：第一个生成式推荐系统

2023年，快手发布了OneRec——这是一个Encoder-Decoder架构的生成式推荐模型。它的创新点在于：不依赖传统推荐中的ID Embedding，而是使用一个多模态Token化器，将物品的标题、封面图、标签等多模态信息编码成一系列"语义Token"（Semantic IDs），然后像语言模型一样auto-regressively地生成推荐结果。

这意味着什么？想象一下，传统模型是把每个视频映射成一个稠密向量，然后在向量空间中搜索近邻；OneRec是把"推荐"这件事变成了"根据用户历史序列，生成下一个最可能感兴趣的物品的语义描述"。后者更接近人类的推荐方式——我们不会说"这个用户的向量和那个视频的向量的余弦相似度是0.87"，我们会说"用户喜欢搞笑视频，这个视频是搞笑的，所以推荐它"。

OneReason：让模型学会推理

OneRec解决了"生成"的问题，但还没有解决"推理"的问题。2024年，快手推出了OneReason——一个让推荐模型能够进行链式推理（Chain-of-Thought）的系统。

OneReason的核心是一个CoT数据系统：它从海量用户行为日志中自动提取推理路径。举个例子：一个用户先搜索了"露营帐篷"，然后看了"户外折叠椅"的商品详情页，最后购买了"便携炉具"——这条行为序列背后隐含的一个推理路径是"用户计划露营 → 需要帐篷 → 还需要椅子 → 还需要炉具"。OneReason从数据中自动学习这些推理模式，然后将它们编码到推荐模型中。

更巧妙的是，OneReason构建了一个RL推理数据飞轮：推荐模型生成的推理路径 → 上线验证 → 用户反馈数据 → 强化学习优化推理能力 → 更好的推理路径。这个闭环让模型的推理能力持续提升。

Pool-Rec：计算池化解决效率问题

生成式推荐的计算开销是巨大的。为了解决这个问题，快手提出了Pool-Rec——一种计算池化方案。它将推荐计算划分为多个可复用的子计算（如用户特征编码、物品特征编码、交叉特征计算等），在跨请求之间共享计算结果。Pool-Rec使得OneRec的推理成本下降到了传统方案可接受的范围。

从研究到生产的距离：快手的OneRec系列特别值得学习的一点是它的"落地节奏"——不是一次性推翻所有东西，而是每一步都保持向后兼容。OneRec先作为一路召回的补充上线，OneReason先作为精排模型的一个辅助模块，Pool-Rec先在小流量上验证。每一步都确保了如果新方案效果不好，可以快速回滚。这种务实的演进策略，是LLM推荐落地的教科书级范例。

2. 阿里LUM：大型用户模型

如果快手是在"生成式推荐"这条路上走得最坚决的，那阿里就是在"用户理解"这个方向上做得最深的。

阿里在2024年提出了LUM（Large User Model）——一个真正意义上的"大模型"级别的用户模型。它的核心思想是：传统推荐模型把用户表示成一个固定维度的Embedding向量，这个向量维度不可能太大（通常几百维）。但LUM把用户的表示空间扩大了100倍以上——用数亿参数来编码用户的行为模式、兴趣偏好、消费习惯。

LUM验证了一个在推荐领域极其重要的发现：推荐系统同样存在Scaling Law。阿里的实验表明，随着用户模型参数从千万级增加到百亿级，推荐效果（AUC、GAUC）持续提升，没有出现明显的边际递减。这和NLP/CV领域的发现是一致的——参数越多，性能越好。关键在于，这个Scaling Law的前提是你要有足够多的用户行为数据——阿里恰恰不缺这个。

LUM最实用的创新是动态跨域推理链：当一个用户在淘宝上搜索了"婴儿推车"，LUM不仅更新了淘宝场景下的用户表示，它还会生成一段自然语言描述的用户画像（例如"该用户近期有育儿需求，可能对母婴类商品感兴趣"），然后通过LoRA微调机制将这段描述传播到高德（推荐附近母婴店）、飞猪（推荐亲子游产品）等其他阿里系APP。

这个传播机制的效率让人印象深刻——它是分钟级LoRA微调。每次用户行为变化后，LUM通过LoRA（只需要更新不到1%的参数）快速微调，将最新的用户理解传播到所有下游场景。这比传统方案中每个场景各自维护用户Embedding的做法高效太多了。

3. 美团MTGR与京东跨域推理

美团的推荐场景与快手和阿里有本质不同：它是LBS+即时消费。用户打开美团，关心的是"附近有什么好吃的"和"现在能不能配送"，而不是"给我推荐一个可能的兴趣点"。这个特点决定了美团的LLM推荐策略必须同时融合图神经网络和生成式排序。

MTGR（Multi-Task Graph Ranking）是美团在2024年的核心推荐升级。它将商家的地理位置信息、配送时效、用户历史消费记录构建成一个异构图，在这个图上用GNN学习用户-商家-位置的表示，然后用生成式的方式综合排序。MTGR的升级版本EGA_V2更是将LLM的能力整合到了美团的"全链路广告"中——从搜索广告的Query理解，到推荐广告的商品排序，到品牌广告的创意生成，全部由LLM增强。

京东的路线和阿里类似，但重点放在了跨域推理。京东有一个独特的优势：它同时拥有电商数据和物流数据。京东的LLM推荐系统能理解用户的购买意图不仅包括"买什么"，还包括"什么时候送到"、"送到哪里"——这些额外信息对于推荐精准度极为重要。京东的DMP（数据管理平台）结合LLM的意图识别能力，实现了业界领先的智能人群定向——广告主只需要用自然语言描述目标人群（如"25-35岁、住在一线城市、最近搜索过跑步装备的男性"），系统就能自动圈选并生成个性化推荐创意。

中国互联网公司的LLM推荐实践模式：仔细观察快手、阿里、美团、京东的案例，你会发现一个有趣的规律：每家公司都沿着自己最独特的优势数据往前走。快手强在短视频内容理解，所以走生成式推荐；阿里强在跨域生态和海量用户行为，所以走Large User Model；美团强在LBS和即时配送网络，所以走图+生成式融合；京东强在供应链数据，所以走跨域推理。没有一家公司试图复制另一家的路。这个现象告诉我们：LLM推荐没有银弹，成功的落地策略一定是跟你的数据优势和业务场景深度绑定的。

4. 端侧推理与轻量化

所有LLM推荐的工业实践都面临同一个紧箍咒：延迟。云端的LLM推理再快也要几百毫秒，而推荐系统的延迟预算往往只有几十毫秒。千亿参数的LLM不可能在每个推荐请求中都跑一次。解决方案是什么呢？

BitNet：1比特量化

微软提出的BitNet技术将LLM的权重从FP16压缩到了1比特——也就是每个参数只用一个bit表示。想象一下：一个原本需要16GB显存的7B模型，用BitNet后只需要不到1GB。虽然精度有所损失，但对于推荐场景中的排序任务来说，这个损失往往可以接受。BitNet使得在手机端侧运行LLM进行轻量级推荐成为可能。

模型蒸馏

知识蒸馏（Knowledge Distillation）是另一种高效的轻量化方案。用一个大规模的"教师"LLM生成推荐结果和推荐理由，然后用这些数据训练一个小规模的"学生"模型。学生模型（通常1B以下）可以达到教师模型90%以上的效果，但推理速度提升10倍以上。京东和快手都有公开的技术分享描述了这种方案的成功实践。

推测解码（Speculative Decoding）

生成式推荐的瓶颈在解码过程——每生成一个Token都需要一次前向推理。推测解码的思路是：用一个轻量的草稿模型预测多个候选Token序列，然后用原始LLM一次性验证所有候选。如果草稿模型够准，可以大幅减少LLM的前向推理次数。在实践中，推测解码可以将生成式推荐的推理速度提升2-3倍。

端云协同推理

这可能是最现实的工程方案：将推荐推理分为"云端"和"端侧"两部分。云端运行大型LLM做全局优化和复杂推理（处理冷启动、兴趣演变等长尾场景），端侧运行量化后的小模型做日常的快速推荐和rerank。端侧模型定期从云端同步更新的知识。这种架构既保证了大部分请求的低延迟，又保留了云端大模型处理复杂场景的能力。

下面是一个使用PEFT库对推荐LLM做LoRA微调的代码示例：

import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model, TaskType

# Load base model (e.g., 1B parameter LLM)
model_name = "Qwen/Qwen2.5-1.5B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.bfloat16,
    device_map="auto"
)

# LoRA configuration for recommendation fine-tuning
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=8,              # LoRA rank
    lora_alpha=32,    # Scaling factor
    lora_dropout=0.1,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    bias="none"
)

# Wrap with LoRA (trainable params < 1%)
model = get_peft_model(model, lora_config)
print(f"Trainable params: {model.num_parameters(only_trainable=True):,}")
print(f"Total params: {model.num_parameters():,}")

# Prepare recommendation training data
# Format: (user_history, candidate_items, relevance_label)
rec_dataset = [
    {
        "input": "用户历史: [露营帐篷, 户外折叠椅] 候选商品: [便携炉具, 羽绒睡袋, 烧烤架]",
        "output": "推荐: 便携炉具 (理由: 用户正在配置露营装备, 炉具是露营必备)"
    },
    # ... more training examples
]

5. 落地路线图

综合来看，LLM在推荐系统中的落地是一个渐进的过程。根据风险等级和收益预期的不同，我将其分为三个阶段：

阶段 LLM角色 延迟容忍度 风险等级 已实践团队
Phase 1：特征增强 离线生成高质量特征（物品描述、用户画像、内容标签） 无延迟要求（离线） 低 几乎所有头部公司
Phase 2：精排/重排增强 LLM做精排rerank、推荐理由生成 50-200ms 中 快手、阿里、京东
Phase 3：全链路生成式推荐 LLM替代传统召回+排序全链路，端到端生成推荐结果 20-100ms（需优化） 高 快手（小范围）、学术前沿

2026年的核心洞察：如果你问我现在LLM在推荐系统里最value-for-money的应用是什么——我的答案不是"用LLM替代推荐系统"，而是"用LLM生成更好的特征，喂给传统推荐模型"。这个方案风险最低、回报最确定、部署最难。具体来说：用LLM离线生成商品的语义Embedding、生成用户兴趣摘要、生成内容标签体系——这些特征一旦进入传统模型，效果提升立竿见影。阿里和京东内部都有公开数据展示这个方案带来的显著收益。我见到太多团队一上来就想用LLM替换整个推荐系统，结果卡在延迟和成本上几个月出不了成果。稳扎稳打从特征增强开始，才是LLM推荐落地的"最小阻力路径"。
