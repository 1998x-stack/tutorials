第15章 复制与一致性

想象一下这个场景：你刚在社交媒体上发了一条动态，刷新页面——动态消失了。再刷新，又出来了。再刷新，又没了。你是不是想摔手机？这种"数据幽灵"现象，正是分布式复制系统的经典产物。

复制（Replication）是分布式系统最基础的容错手段——把数据复制到多台机器上，一台挂了还有另一台顶着。但说来讽刺，复制同时也是分布式系统里坑最多的地方。它就像一把双刃剑：用好了，系统高可用、高性能；用不好，数据不一致、脑裂、丢数据，能把运维同学折腾到下半夜。本章我们就来把这把剑玩明白。

15.1 复制拓扑结构

根据数据流向和写入权责，复制拓扑大致分为三类：主从复制（Master-Slave / Leader-Based）、多主复制（Multi-Master）、无主复制（Leaderless）。它们的核心区别在于：谁有权力决定写入顺序。

主从复制（Master-Slave）

这是最经典的架构。一台节点为 Master（主节点），负责处理写请求，把写操作顺序记录并发送给一个或多个 Slave（从节点）。Slave 只响应读请求。优点是实现简单、写顺序天然一致；缺点是 Master 是单点瓶颈，挂了需要选举新 Master。

多主复制（Multi-Master）

多个节点都接受写入，各自将写入异步复制给其他节点。常见于多数据中心场景（例如用户在上海写入，同时同步到纽约）。最大挑战是写冲突解决——同一行数据在上海被改成 A，在纽约被改成 B，最终该是什么？常见的策略有 Last-Write-Win（LWW，根据时间戳）、CRDT（无冲突数据类型）、或用户自定义合并。

无主复制（Leaderless）

任何节点都接受读写，没有固定的 Leader。客户端将写请求发到所有副本节点，读请求也从多个副本读取，用版本号判断最新值。DynamoDB 和 Cassandra 就是这种风格。读写的"法定人数"（Quorum）决定了一致性强度。

特性 主从复制 多主复制 无主复制
写入点 单点（Master） 多点 任意节点
写顺序保证 天然有序 需冲突解决 需版本向量
可用性（单节点故障） 需选举 高（其他 Master 继续服务） 高（任意节点均可）
复杂性 低 中 高
典型代表 MySQL、PostgreSQL、Redis Sentinel MySQL Group Replication、CouchDB Cassandra、DynamoDB、Riak

15.2 同步复制 vs 异步复制

假设 Master 收到一个写请求。在回复客户端"写入成功"之前，它是否需要等 Slave 也写入完毕？这个选择决定了系统在写延迟、数据一致性、可用性之间的三角权衡。

同步复制（Synchronous Replication）：Master 必须等至少一个 Slave 确认写入完成，才返回客户端成功。好处是 Master 挂了，Slave 一定有最新数据，不会丢数据。坏处是写延迟增加了网络来回的时间；如果 Slave 挂了，Master 就得一直等着——除非配置超时降级。

异步复制（Asynchronous Replication）：Master 写入本地就回复客户端，Slave 的数据随后慢慢追。延迟低、吞吐高。坏处是 Master 突然挂了，还没同步到 Slave 的数据就丢了。

真实案例：MySQL 的异步复制是最常用的模式。但 2018 年 GitHub 的 MySQL 主从切换事故中，异步复制导致数十秒的数据不一致，最终通过重新设计复制拓扑和半同步复制（Semi-Sync Replication）解决。我们在 15.6 节再详细复盘。

半同步复制（Semi-Sync Replication）是 MySQL 5.7 引入的中间方案——Master 等至少一个 Slave 确认写入，但不等全部的 Slave。兼顾了同步的安全性和异步的性能。实际部署中，Semi-Sync 是 MySQL 高可用的"标准配置"。

模式 数据安全 写延迟 可用性影响 适用场景
全同步 最高（零丢失） 高（等所有副本） 一个副本慢就拖慢全局 金融核心交易
半同步 较高（至少一个副本确认） 中等 降级为异步若超时 MySQL 高可用标配
异步 低（可能丢数据） 低 无 日志、监控、可容忍丢失的场景

15.3 复制延迟问题

异步复制最大的隐患不是丢数据——丢数据毕竟概率低——而是复制延迟带来的各种一致性异常。一个用户刚写完的数据，读请求打到还没来得及同步的副本上，就读到了旧数据。这种"时间穿越"问题在用户侧体验极差。

读己之写（Read-Your-Writes）

用户提交评论后刷新页面，自己的新评论应该出现——这是最基本的期望。但如果是异步复制，刷新可能落到尚未同步的副本上，新评论"消失"了。解决方案是读己之写一致性（Read-Your-Writes Consistency）：对"自己"的写入，读请求必须路由到包含最新数据的副本。常见的做法是把用户自己的请求始终路由到 Master，或使用时间戳判断副本是否够新。

单调读（Monotonic Reads）

用户在副本 A 读到了一条新评论（已同步），刷新后请求落到副本 B（尚未同步），那条评论"消失了"。这是单调读不一致——同一用户多次读到同一数据，值却来回跳变。解决方案是让同一用户始终路由到同一副本（Sticky Session），或跟踪时间戳确保只往前读。

前缀一致性（Consistent Prefix Reads）

假设用户先发了一条评论 A，再发评论 B。如果某个副本先同步了 B 再同步 A，用户将看到"B 在 A 之前"——这个顺序违背了因果关系。解决这个问题需要前缀一致性：保证观察到的写顺序与原始提交顺序一致。这在分片跨多个副本的场景尤其棘手。

运维建议：在实际生产环境中，这些一致性异常很难 100% 消除。务实的做法是：在用户侧感知不到的范围内容忍弱一致性（比如朋友圈的点赞数），对关键业务路径（支付、订单）则使用强一致性复制或读主库。

15.4 无主复制与 Quorum

Dynamo 风格的无主复制是 Amazon 2007 年提出的设计，后来被 Cassandra、Riak 继承。它的核心思想是：没有 Master，人人平等。

W + R > N 法则

在无主复制中，数据被复制到 N 个节点。客户端写入时，至少等待 W 个节点确认；读取时，至少读取 R 个节点并取最新的版本。只要 W + R > N，读写集合必然有重叠，从而保证读到最新值。典型的配置是 N=3, W=2, R=2——既容忍一个节点故障，又保证强一致性。

算一笔账：N=3, W=3, R=1 意味着写必须等所有 3 个节点确认（容错为 0），读只需 1 个节点。这叫"写优先"配置，适合读多写少的场景。反过来 N=3, W=1, R=3 是"读优先"，适合写多读少的场景。

Sloppy Quorum 与 Hinted Handoff

在 Dynamo 中，如果某个副本节点宕机了，不严格需要它参与 Quorum——临时将数据写入另一个健康的节点（即使它不在原始 N 个节点的列表中），等原节点恢复后再把数据转发回去。前者叫 Sloppy Quorum（宽松法定人数），后者叫 Hinted Handoff（提示移交）。这让系统在节点故障时依然能写入，大大提高了可用性——但代价是牺牲了一致性（因为数据可能最终不在正确的节点上）。

读修复（Read Repair）

读取时如果发现某个副本的版本比其它副本旧（版本号更低），客户端在返回结果的同时，异步地把新版本写回旧副本。这个机制叫读修复。它保证了读操作具有副作用——读得越多，数据越一致。Cassandra 的 Read Repair Chance 参数控制了概率。

15.5 真实案例：GitHub MySQL 主从切换事故

2018 年 10 月 21 日，GitHub 进行一次计划内的 MySQL 主从切换——旧 Master 需要逐步下线，新 Master 接管服务。切换过程本来有标准流程：等待所有从库追平主库的 Binlog，然后切换 VIP（虚拟 IP）。但问题出在备库的 Binlog 追平延迟随时间增长，切换团队等了两个小时它都没有追平。

团队最终决定强行切换，结果发现新 Master 的数据比旧 Master 少了约 40 秒的数据。这 40 秒的数据包括 Issue 评论、PR 更新、Wiki 编辑等。最终花了数小时通过 Binlog 手动恢复。

教训：异步复制下的主从切换，在不彻底追平 Binlog 的情况下强行切换，必然丢数据。GitHub 事后将复制模式升级为 Semi-Sync，并增加了自动化延迟监控和切换前的一致性校验。任何做数据库复制的团队，都应该把这个案例贴在墙上。

GitHub 的事故给了我们一个深刻的启示：复制不是万能的，它的极限在哪里，恰恰需要下一章要讲的分布式事务和共识算法来解决。当复制无法满足数据一致性需求时，我们需要更强的分布式协调机制。下一章，我们就来深入分布式事务——从 2PC 到 Saga，看看这些"大招"在什么场景下值得用。
