# vLLM V1 架构详解

## 1. V0 → V1：为什么需要架构升级

vLLM V0 是一个快速的"原型级"实现，在设计上存在根本问题：

- **单一进程**：所有逻辑混在一个进程中，难以扩展和管理
- **硬编码两阶段**：prefill 和 decode 严格分离，难以统一优化
- **代码耦合严重**：调度、KV 管理、模型执行互相缠绕
- **功能开发困难**：每加一个新特性需要改动大量代码

2025 年 1 月 27 日，vLLM 发布 V1 Alpha。V0 被正式弃用（RFC #18571）。

## 2. V1 四大设计目标

| 目标 | 含义 |
|------|------|
| 简单、模块化 | 代码库易于理解和修改，降低贡献门槛 |
| 近乎零 CPU 开销 | GPU 不应因为 CPU 调度慢而空闲 |
| 关键优化统一 | PagedAttention、prefix cache、chunked prefill 等统一到一个架构 |
| 零配置 | 默认启用所有优化，减少用户调参负担 |

## 3. 核心架构：1+N 多进程

### 3.1 整体结构

```
┌──────────────────────────────────────────────────────┐
│                   Master 进程                         │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐  │
│  │  API 路由     │  │  健康检查     │  │  资源监控   │  │
│  └──────────────┘  └──────────────┘  └────────────┘  │
│                         │                             │
│          ZeroMQ 消息总线（10万+ 请求/秒）              │
│                         │                             │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐  │
│  │  Worker 1    │  │  Worker 2    │  │  Worker 3   │  │
│  │  EngineCore  │  │  EngineCore  │  │  EngineCore │  │
│  │  ├─主线程    │  │  ├─主线程    │  │  ├─主线程   │  │
│  │  └─计算线程  │  │  └─计算线程  │  │  └─计算线程 │  │
│  └──────────────┘  └──────────────┘  └────────────┘  │
└──────────────────────────────────────────────────────┘
```

### 3.2 多进程架构的优势

| 优势 | 说明 |
|------|------|
| 故障隔离 | 单个 Worker 崩溃不影响整体服务（其他 Worker + Master 继续） |
| 资源隔离 | 不同 GPU 分配给不同 Worker |
| 热更新 | 模型版本升级只需重启一个 Worker 轮流进行 |

### 3.3 EngineCore 内部双线程设计

- **主线程**：管理请求队列、资源分配、调度决策
- **计算线程**：GPU 张量运算、模型前向传播

## 4. 统一调度器（Unified Scheduler）

### 4.1 消除 Prefill/Decode 硬分界

V0 有两个独立的调度器用于 prefill 和 decode，V1 用一个统一字典 `{request_id: num_tokens}` 同时管理两者。

```python
# V0: 两阶段硬分离
def step():
    prefill_results = prefill_scheduler.step()
    decode_results = decode_scheduler.step()
    # 两步之间无法相互感知

# V1: 统一调度
def step():
    # 每个 request 告诉调度器"我这步需要处理多少 token"
    # 调度器统一决定如何 mix prefill 和 decode
    batch = scheduler.schedule(waiting, running)
    model.forward(batch)
```

### 4.2 支持的调度特性

- **分块预填充（Chunked Prefill）**：默认启用
- **FCFS + Priority 混合调度**
- **推测解码原生集成**：支持 n-gram、EAGLE、Medusa 草稿方法

## 5. KV Cache Manager 革新

| 特性 | V0 | V1 |
|------|----|----|
| 前缀缓存 | 基本支持，有数据拷贝 | 零开销，指针重定向 |
| CPU 占用 | 较高 | 降低 90% |
| GPU↔CPU 置换 | 支持 | 移除（不再需要） |
| 并发预填充 | 不支持 | 支持多请求并发预填充 |

## 6. 性能对比

| 指标 | V0 | V1 | 提升 |
|------|----|----|------|
| 7B 模型吞吐量 | 1.2K tok/s | 2.1K tok/s | +75% |
| 70B 模型首 token 延迟 | 320ms | 185ms | -42% |
| P99 延迟标准差（高 QPS） | 基准 | -62% | 大幅降低 |
| GPU 利用率 | 65% | 92% | +41% |
| 长上下文推理效率 | 基准 | 2.3× | +130% |
