第10章 路由协议(二)：层次路由

大家好，我是你们的讲师。想象一下管理一个1000人的公司：如果CEO直接给每个员工发指令，CEO会疯掉，员工也会被大量消息淹没。聪明的做法是设立部门——部门经理协调本部门事务，只有需要跨部门协作时才上报CEO。这就是"分治"的力量。

上一章我们讨论了平面路由——所有节点地位平等，这在几十个节点的网络中没问题。但当网络扩展到几百、上千个节点时，平面路由的控制开销和通信瓶颈就会把网络压垮。层次路由（也称为分簇路由）正是为解决这个问题而生。它的核心思路是：选择部分节点担任簇头（Cluster Head, CH），负责协调本簇成员的数据收集和跨簇通信。

本章核心问题： 如何通过"分而治之"的思想让WSN路由扩展到数千个节点？簇头怎么选、怎么轮换？RPL为什么成为物联网事实上的路由标准？学完本章，你将对WSN的层次路由体系有系统的理解。

为什么分簇是关键？

分簇给WSN带来了三大好处：

  • 扩展性：簇内通信限于局部，控制消息不会在全网扩散。网络规模增大时，只需增加簇的数量，路由控制开销的增长是线性的（甚至亚线性的）。
  • 能量均衡：簇头负责更耗能的远距离通信和数据处理，普通成员只需要近距离发送数据给簇头。通过定期轮换簇头，可以避免某些节点因"过劳"而过早死亡。
  • 数据融合：簇头可以对本簇成员的数据进行聚合（如计算平均温度、最大值等），再向Sink发送精简后的结果，大幅减少传输量。

LEACH协议

LEACH（Low-Energy Adaptive Clustering Hierarchy）由MIT的Wendi Heinzelman于1999年提出，是层次路由的"开山之作"，截至2026年已被引用超过2万次。它最关键的设计贡献是随机轮换簇头的思想。

LEACH的工作机制

LEACH将时间划分为若干"轮（Round）"，每一轮包括两个阶段：

阶段一：簇建立阶段。 每个节点生成一个0到1之间的随机数，如果该数小于阈值T(n)，则节点宣布自己是本轮的簇头。阈值由以下公式计算：

T(n) = P / [1 - P × (r mod 1/P)]    （如果n在最近1/P轮中未被选为簇头）

其中P是期望的簇头比例（通常设为5%），r是当前轮次。这个公式确保了每个节点在最近1/P轮（约20轮）中至少被选为一次簇头，实现了能量的均衡消耗。

选出的簇头通过CSMA广播ADV消息宣告自己。其他节点根据接收信号的强度选择加入哪个簇，并向簇头发送加入请求。

阶段二：稳定传输阶段。 簇内成员在自己的TDMA时隙内向簇头发送数据。簇头对数据进行融合后直接发送给Sink。稳定阶段通常持续较长时间（远长于簇建立阶段），以摊薄分簇开销。

LEACH的局限

LEACH虽然经典，但有几个明显的短板：簇头选择完全随机，不考虑节点的剩余能量；假设所有节点都能直接与Sink通信（单跳模型），在大范围网络中不现实；簇头分布不均匀，可能出现某些簇头负责过多节点的情况。

一个需要警惕的陷阱： LEACH的随机簇头选择在理论仿真中表现得还不错——因为仿真假设节点是同构的、均匀分布的。但真实部署中，哪怕轻微的节点位置不均匀，也会导致簇头分布的严重不平衡。某智慧农业项目中，我们部署了200个节点，结果一轮下来，最"忙"的簇头管理了47个成员节点，而最"闲"的只有3个。忙的那个簇头在下一轮开始前就耗尽了能量。

LEACH的改进家族

LEACH的不足催生了一系列改进协议：

  • HEED（Hybrid Energy-Efficient Distributed clustering）：将剩余能量作为选簇头的首要因素，解决了LEACH"随机选"的问题。HEED还考虑了簇内通信代价，让节点倾向于加入通信代价最低的簇。
  • TEEN（Threshold-sensitive Energy Efficient sensor Network）：专为响应型应用设计。引入了硬阈值和软阈值两个参数——只有读数超过硬阈值或相对于上次报告的变化超过软阈值时，节点才传输数据。非常适合突发事件监测。
  • APTEEN（Adaptive Periodic TEEN）：在TEEN基础上增加了周期性上报能力。节点既按周期上报数据，又在发生突发事件时触发式上报，兼顾了"监测"和"告警"双重需求。

RPL协议深度解析

RPL（Routing Protocol for Low-Power and Lossy Networks）是IETF ROLL工作组在2012年发布的标准化路由协议（RFC 6550），专为LLN（低功耗有损网络）设计。它是目前物联网领域应用最广泛的路由协议之一，被ZigBee IP、Thread、6LoWPAN等技术体系广泛采用。

核心概念：DODAG

RPL的核心数据结构是DODAG（Destination-Oriented Directed Acyclic Graph，面向目的地的有向无环图）。简单来说，它就是一棵以Sink（或称为根节点）为根的多跳路由树，但允许多个父节点存在（有向无环图比树更灵活）。

DODAG的构建过程的核心消息是DIO（DODAG Information Object，DODAG信息对象）：

// RPL DIO消息主要字段与DODAG构建过程

// DIO消息格式（精简描述）
struct DIO_message {
    version:        uint8   // DODAG版本号
    rank:           uint16  // 节点在DODAG中的等级
    dodag_id:       IPv6    // DODAG根节点标识
    objective_code: uint8   // 目标函数标识
    ground:         bool    // 是否支持到Sink的路径
    prefix_info:    ...     // 前缀信息（用于IPv6地址自动配置）
}

// DODAG构建过程
function build_DODAG(sink_node):
    // 1. 根节点发送初始DIO消息，rank=0
    dio = {version=1, rank=0, dodag_id=sink_addr, objective_code=OF0}
    broadcast(dio)

    // 2. 收到DIO的节点计算自己的rank
    //    Rank = Parent_Rank + Rank_Increase
    //    Rank_Increase取决于链路质量（ETX）
    on_receive_DIO(dio):
        parent_rank = dio.rank
        link_quality = estimate_ETX(dio.sender)
        my_rank = parent_rank + rank_increase(link_quality)

        // 维护候选父节点列表，选最优的作为首选父节点
        parent_list.add(dio.sender, my_rank, link_quality)
        preferred_parent = select_best_parent(parent_list)

        // 发送更新后的DIO，携带自己的rank
        my_dio = {rank=my_rank, dodag_id=dio.dodag_id, ...}
        broadcast(my_dio)

    // 3. 循环：DIO逐跳扩散，所有节点获得rank
    //    形成以根为最低rank的有向无环图
    //    数据包沿着rank递减的方向流向根节点

    // 4. 通过DAO(Destination Advertisement Object)消息
    //    子节点向父节点注册可达性信息
    //    支持P2P和P2MP通信

Objective Function与Trickle计时器

RPL的目标函数（Objective Function, OF）决定了节点如何根据路由度量计算rank和选择父节点。常见的OF有：

  • OF0（RFC 6552）：使用最小跳数作为度量，简单直接
  • MRHOF（RFC 6719）：使用期望传输次数（ETX）作为度量，选择链路质量更好的路径

Trickle计时器是RPL能效的关键。它的核心策略是"趋于稳定时减少通信，检测到变化时增加通信"：当网络稳定时，DIO的发送间隔指数级增长（从几百ms到几分钟甚至几小时）；当检测到环路或拓扑变化时，重置计时器，立即高频发送DIO。这种机制让RPL在稳定状态下的控制开销极低。

前沿：生物启发路由（BDSR）

2025年，一种受到生物群体智能启发的路由协议——BDSR（Bee-inspired Dynamic Swarm Routing）在学术界引起了广泛关注。BDSR模拟蜜蜂采蜜的行为：数据包就像"工蜂"，在从源到Sink的路径上留下信息素（Pheromone）标记；后续的数据包根据信息素浓度选择路径，浓度高的路径（意味着路径质量好）被更多数据包选择；同时信息素会随时间蒸发，避免过时路径持续被使用。

BDSR的创新之处在于引入了自适应环形同步（Adaptive Ring Synchronization）机制：网络以Sink为中心划分多个"环"（Ring），每个环内的节点同步维护路由信息。这种环形分簇结构与DODAG的层次结构有异曲同工之妙，但BDSR通过群体的自组织行为替代了中心化的DODAG构建过程。

实验数据表明，在500节点规模的测试中，BDSR相比RPL可以提升约47%的网络吞吐量，同时将端到端延迟降低约28%。不过BDSR目前仍处于学术研究阶段，尚未有大规模商业部署的案例。

分簇路由协议全方位对比

协议 簇头选择 通信模型 能量感知 扩展性 适用场景
LEACH 随机阈值 单跳（簇头→Sink） 否 中（N≈500） 均匀部署、同构节点
HEED 剩余能量+通信代价 单跳/多跳混合 是 中高（N≈1000） 节点能量异构
TEEN 同LEACH（改进版同HEED） 单跳/多跳 是 中 突发事件监测
APTEEN 同HEED 单跳/多跳 是 中高 周期+事件混合
RPL DODAG构建（OF驱动） 多跳（P2MP/MP2P） 可配置 高（N≈10000+） 物联网、6LoWPAN、工业
BDSR（前沿） 群体智能+信息素 多跳+环形同步 是 高 大规模动态拓扑

分簇算法的选择决策树

面对这么多分簇路由协议，如何选择？下面是一个实用的决策思路：

  • 网络规模 < 100节点 → 简单分簇（LEACH）或甚至平面路由（SPIN/DD）即可
  • 100-1000节点 → HEED或APTEEN（根据是周期上报还是事件触发）
  • 1000-10000节点 → RPL（标准化、多跳、IP兼容）
  • 10000+节点 → 可能需要多级分簇或结合地理路由

另外还要考虑：需要与互联网无缝连接吗？需要 → RPL（IPv6原生支持）。节点有移动性吗？有 → 可以考虑BDSR或移动感知的RPL扩展。主要流量方向是"多对一"吗？是的 → RPL和LEACH都适合。

实践建议： 在实际项目中用RPL时，请一定仔细配置目标函数（OF）和Trickle参数。默认的OF0（最小跳数）可能会导致网络中所有节点都选择"看起来最短但实际上链路质量很差的"路径。推荐使用MRHOF（ETX度量），同时根据网络规模和流量特征调整Trickle的Imin和Imax参数。

一个真实的"踩坑"故事

2021年，我参与了某智慧农业项目——在一个大型温室大棚里部署了200个土壤湿度和养分传感器，使用LEACH协议进行分簇路由。大棚面积约6000平方米，传感器均匀分布在种植区。

部署后第二周，我们发现大棚东侧约40个节点的数据全部丢失，但节点本身硬件正常（断电重启后能工作片刻）。排查发现，这40个节点被分配给了同一个簇头，而那个簇头位于大棚边缘。原因很简单：LEACH的随机簇头选择在整个网络中是均匀的——东侧区域的节点少，但簇头选择概率相同，导致东侧区域"有资格"当簇头的节点很少。当该区域唯一的簇头耗尽能量后，整个区域相当于"失联"了。

解决方法是将HEED的剩余能量感知机制引入到LEACH中——我们修改了簇头选择阈值，增加了节点剩余能量的权重。同时手动将网络划分为4个地理区域，每个区域独立运行LEACH算法，保证每个区域都有足够的簇头候选者。

这个教训让我深刻理解了：WSN协议不是"开箱即用"的。LEACH成功的前提假设是"节点均匀分布"——但真实的物理部署几乎永远不会满足这个假设。任何路由协议在部署前都需要根据实际的节点布局进行调整和验证。

好了，从平面路由到层次路由，我们走完了WSN路由协议的主要路线。路由让数据找到了路，但这条路走得"快不快"、"准不准"还需要好用的"钟表"和"指南针"。下一章和第12章将分别介绍时间同步和定位技术——这两个技术一个是让全网节点"对准表"，一个是让网络知道自己"在哪"。没有它们，很多WSN应用就是纸上谈兵。
