第28章：综合实战项目

"如果让我重做这个项目，我会把 80% 的时间花在数据上。"这句话出自我们团队法律 AI 项目复盘时首席科学家之口。我们花了一个半月搭模型架构、调超参数，最后模型在 NER 任务上的 F1 分数一直卡在 76% 上下。回过头去排查，发现数据标注中实体边界不一致、法条编号格式混乱、刑期字段存在大量空值和异常值——数据清洗阶段的任何一个问题，如果在早期被系统化处理，都不至于到模型训练阶段才暴露。如果一开始就有这 28 章的知识，至少能省一半的时间。

这个项目的完整复盘，就是本章的内容。我们将一步步构建一个中文法律文书命名实体识别（NER）数据集，涵盖从数据采集、清洗标注、管线构建、模型训练到数据集发布的完整流程。这不是理论推演——每一步都有可执行的代码、具体的决策理由，以及我们在实际项目中踩过的坑。你完全可以把这个项目当作你的第一个数据工程实践模板。

目标声明：本章结束时，你将拥有一个可复现的中文法律 NER 数据管线、一个发布在 Hugging Face Hub 上的数据集、以及一个基于 BERT 的基线模型。更重要的是，你将掌握一套可迁移到任意领域的"数据集工程方法论"。

项目定义：中文法律 NER 数据集工程

在开始动手之前，先明确我们要做什么：

  • 场景：构建一个中文法律文书命名实体识别数据集，用于训练能够自动从裁判文书中提取关键信息的模型。
  • 数据来源：中国裁判文书网公开的裁判文书（已脱敏、无个人隐私信息）。
  • 目标标签类型：共 5 种实体类型——人名/被告、机构（法院/检察院/律所）、法条引用、罪名、刑期。
  • 项目范围：采集约 5000 份文书→清洗去重→手工标注+BIO格式→构建 datasets 管线→训练基线模型→发布到 HF Hub。
  • 团队角色：1 名数据工程师（你）+ 2 名法律领域标注员。

阶段 1：数据采集与探索

第一阶段的目标是获取原始文本并理解数据的基本面貌。不要跳过 EDA 直接进清洗——你在 EDA 中发现的问题模式，会直接影响后续所有的清洗策略。

从公开 API 采集原始文本

中国裁判文书网提供了公开的检索接口。我们使用基于 requests 的爬虫获取文书 HTML，提取正文文本，并存储为 JSONL 格式：

import requests
import json
from bs4 import BeautifulSoup

def fetch_judgment(document_id):
    """获取单篇裁判文书文本"""
    url = f"https://wenshu.court.gov.cn/website/wenshu/{document_id}.html"
    resp = requests.get(url, headers={"User-Agent": "Mozilla/5.0"})
    soup = BeautifulSoup(resp.text, "html.parser")
    text = soup.select_one(".main-content").get_text(strip=True)
    return text

# 采集 5000 份文书
raw_data = []
for doc_id in range(5000):
    try:
        text = fetch_judgment(f"doc_{doc_id:05d}")
        raw_data.append({"id": doc_id, "text": text, "source": "wenshu"})
    except Exception as e:
        print(f"Failed to fetch {doc_id}: {e}")

# 保存为 JSONL
with open("raw_judgments.jsonl", "w", encoding="utf-8") as f:
    for item in raw_data:
        f.write(json.dumps(item, ensure_ascii=False) + "\n")
实用建议：实际项目中不要用单线程爬取。用 concurrent.futures.ThreadPoolExecutor 或者 aiohttp 做异步并发，采集速度可以提升 5-10 倍。但注意控制并发数（建议 5-10），避免被目标服务器限流。

阶段 2：数据清洗与标注

这是整个项目中最耗时、最需要细心的环节，没有之一。也是我们在真实项目中犯错最多、付出代价最大的阶段。

去重：MinHash 与 SimHash

裁判文书存在大量重复——同一个案件的"一审"和"二审"文书内容高度相似。我们需要做近似去重而非精确去重，因为两份文书可能只有判决结果不同。我们使用 SimHash 做近似去重：

from simhash import Simhash

def compute_simhash(text):
    """计算文本的 SimHash 指纹"""
    return Simhash(text).value

def dedup_by_simhash(df, threshold=0.85):
    """基于 SimHash 相似度去重"""
    df["simhash"] = df["text"].apply(compute_simhash)
    deduped = []
    seen = set()
    for _, row in df.iterrows():
        # 检查是否有相似度过高的已保留文档
        is_dup = any(
            abs(row["simhash"] - s) < threshold * 64
            for s in seen
        )
        if not is_dup:
            deduped.append(row)
            seen.add(row["simhash"])
    return pd.DataFrame(deduped)

df_deduped = dedup_by_simhash(df, threshold=0.85)
print(f"去重前: {len(df)}, 去重后: {len(df_deduped)}")
# 去重前: 4987, 去重后: 4213

去噪：过滤短文本与异常字符

EDA 阶段发现的短文本（<500 字）需要过滤。此外，一些文书包含爬取引入的 HTML 残留标签、乱码字符等：

import re

def clean_text(text):
    """清洗文本：去 HTML 标签、去空白过载、标准化标点"""
    # 去 HTML 标签
    text = re.sub(r"<[^>]+>", "", text)
    # 去多余空白
    text = re.sub(r"\s+", " ", text).strip()
    # 标准化中文引号
    text = text.replace("“", "\"").replace("”", "\"")
    return text

def filter_noisy(df, min_length=500):
    """过滤低质量样本"""
    df["clean_text"] = df["text"].apply(clean_text)
    df["clean_length"] = df["clean_text"].apply(len)

    # 过滤短文本和异常长文本
    mask = (df["clean_length"] >= min_length) & (df["clean_length"] < 20000)
    # 过滤乱码占比过高的文本
    def garbled_ratio(text):
        non_chinese = sum(1 for c in text if ord(c) > 127)
        return non_chinese / len(text) if len(text) > 0 else 1

    df["garbled_ratio"] = df["clean_text"].apply(garbled_ratio)
    mask = mask & (df["garbled_ratio"] < 0.3)

    return df[mask].copy()

df_clean = filter_noisy(df_deduped, min_length=500)
print(f"清洗后样本数: {len(df_clean)}")
# 清洗后样本数: 3718

BIO 标注格式构建

NER 标注采用标准的 BIO 格式（Begin/Inside/Outside）。对于一个实体"张三"（PER 类型），标注为：

Token 标签
被 O
告 O
张 B-PER
三 I-PER
因 O
盗 B-CHARGE
窃 I-CHARGE
罪 I-CHARGE

由于我们是从零开始的标注项目，需要让标注员在标注工具中标记实体 span。我们选择开源的 LabelStudio 作为标注平台：

# 将清洗后的文本导入 LabelStudio 格式
import json

labelstudio_tasks = []
for idx, row in df_clean.iterrows():
    task = {
        "id": idx,
        "data": {"text": row["clean_text"]},
        "annotations": []  # 标注员填充

    labelstudio_tasks.append(task)

with open("labelstudio_import.json", "w", encoding="utf-8") as f:
    json.dump(labelstudio_tasks, f, ensure_ascii=False, indent=2)

标注一致性验证

两个标注员独立标注同一批样本后，需要验证标注一致性。NER 领域常用 Cohen's Kappa 系数评估标注者之间的一致性：

from sklearn.metrics import cohen_kappa_score

def compute_ner_kappa(annotator1, annotator2):
    """
    计算两个标注员在 token 级别标签上的 Kappa 系数
    annotator1/2: 每个 token 的标签列表
    """
    # 展平所有 token 的标签
    flat_1 = [tag for seq in annotator1 for tag in seq]
    flat_2 = [tag for seq in annotator2 for tag in seq]
    return cohen_kappa_score(flat_1, flat_2)

# 假设有 100 条双标注样本
kappa = compute_ner_kappa(ann1_labels, ann2_labels)
print(f"标注者一致性 Kappa: {kappa:.3f}")
# 通常 kappa > 0.8 表示一致性良好
# 如果 kappa < 0.7，需要回顾分歧，更新标注规范
血泪教训：在标注阶段节省的时间，会在模型训练阶段加倍偿还。我们第一轮标注的一致性 Kappa 只有 0.63——分歧集中在"法条引用"和"刑期"两个实体类型上。一位标注员把"根据《刑法》第264条"的"第264条"标记为 LAW 实体，另一位认为只标记"《刑法》"就够了。这种规范分歧如果在标注前用"标注手册"明确下来，可以避免后期大量的重标工作。

用 cleanlab 检测噪声标签

标注完成后，可以使用 cleanlab 库来检测可能的标注错误。cleanlab 通过训练一个初步分类模型，找出那些模型预测和人工标注不一致的样本，辅助标注员复查：

    pred_probs=model_probs,   # 模型预测概率

效率策略：不要试图一次性把 3718 篇文书全部标完。采用"先标 200 篇→训练初版模型→用模型预标注剩余样本→人工修正"的迭代标注策略。每轮迭代可以节省 60-70% 的标注时间，同时保证标注质量。

阶段 3：构建可复现数据管线

有了带 BIO 标注的文本数据后，我们用它构建一个标准的 Hugging Face datasets 管线。管线的核心目标是：任何人、在任何机器上、输入同一个 git commit，都能拿到完全一样的数据。

定义 Features Schema

from datasets import Features, Value, Sequence, ClassLabel

# 定义 NER 标签列表（BIO 格式）
ner_tags = [
    "O", "B-PER", "I-PER",
    "B-ORG", "I-ORG",
    "B-LAW", "I-LAW",
    "B-CHARGE", "I-CHARGE",
    "B-SENTENCE", "I-SENTENCE"
]

features = Features({
    "id": Value("string"),
    "tokens": Sequence(Value("string")),
    "ner_tags": Sequence(ClassLabel(names=ner_tags)),
    "text": Value("string"),
    "source": Value("string"),
})

构建 dataset 对象

from datasets import Dataset

def build_ner_dataset(annotated_data, features):
    """将标注数据构建为 datasets.Dataset 对象"""
    return Dataset.from_list(annotated_data, features=features)

# 假设 annotated_data 是经过前两步处理好的样本列表
# 每条包含 id, tokens, ner_tags, text, source 五个字段
dataset = build_ner_dataset(annotated_data, features)

map/filter 处理流水线

def filter_empty_entities(example):
    """过滤掉没有任何实体的样本"""
    tags = example["ner_tags"]
    return any(t != 0 for t in tags)  # 0 = "O"

def add_entity_counts(example):
    """添加每个样本的实体数量统计（用于后续分析）"""
    from collections import Counter
    tags = example["ner_tags"]
    # ner_tags 是整数编码，需要映射回标签名
    tag_names = [ner_tags[t] for t in tags if t != 0]
    example["entity_counts"] = dict(Counter(tag_names))
    return example

# 完整管线
pipeline = (
    dataset
    .filter(filter_empty_entities)
    .map(add_entity_counts)
)

print(f"过滤前: {len(dataset)}, 过滤后: {len(pipeline)}")
# 过滤前: 3718, 过滤后: 3402

使用 DVC（Data Version Control）对 Parquet 格式的数据集做版本管理：

管线设计原则：数据管线应该设计成"无状态"的——输入是原始 JSONL 文件，输出是清洗后的 dataset 对象。中间不产生任何手动修改后不可追溯的文件。每次管线修改都通过 git commit 记录变更，实现完整的审计追踪。

Train/Val/Test 分层切分

按实体类型进行分层抽样，确保三个子集的标签分布一致：

阶段 4：模型训练与数据集迭代

管线就绪后，用 BERT-base-Chinese 训练一个基线 NER 模型。这一阶段的核心循环是"训练→评测→错误分析→回溯到数据集→再训练"。

训练基线模型

from transformers import (
    AutoTokenizer, AutoModelForTokenClassification,
    TrainingArguments, Trainer
)

# 加载 BERT-base-Chinese
model_name = "bert-base-chinese"
tokenizer = AutoTokenizer.from_pretrained(model_name)

def tokenize_and_align_labels(examples):
    """将 tokens 和 NER 标签对齐到 BERT 的 subword tokenization"""
    tokenized_inputs = tokenizer(
        examples["tokens"], truncation=True, padding="max_length",
        max_length=512, is_split_into_words=True
    )
    labels = []
    for i, label in enumerate(examples["ner_tags"]):
        word_ids = tokenized_inputs.word_ids(batch_index=i)
        previous_word_idx = None
        label_ids = []
        for word_idx in word_ids:
            if word_idx is None:
                label_ids.append(-100)
            elif word_idx != previous_word_idx:
                label_ids.append(label[word_idx])
            else:
                label_ids.append(-100)  # subword 不参与 loss 计算
            previous_word_idx = word_idx
        labels.append(label_ids)
    tokenized_inputs["labels"] = labels
    return tokenized_inputs

# Tokenize 数据集
tokenized_splits = {
    k: v.map(tokenize_and_align_labels, batched=True)
    for k, v in splits.items()
}

model = AutoModelForTokenClassification.from_pretrained(
    model_name, num_labels=len(ner_tags)
)

training_args = TrainingArguments(
    output_dir="./legal-ner-bert",
    evaluation_strategy="epoch",
    save_strategy="epoch",
    per_device_train_batch_size=16,
    num_train_epochs=5,
    logging_steps=50,
    seed=42,
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized_splits["train"],
    eval_dataset=tokenized_splits["validation"],
)
trainer.train()

评估指标：Span-Level F1

NER 的标准评测指标是 span-level F1——模型预测的实体 span（起始位置+标签类型）与真实标注完全一致才算正确。使用 seqeval 库计算：

from seqeval.metrics import classification_report
from seqeval.scheme import IOB2

# 推理
predictions = trainer.predict(tokenized_splits["test"])
pred_labels = predictions.predictions.argmax(-1)
true_labels = predictions.label_ids

# 将 id 映射回标签名（忽略 -100）
true_tags = [
    [ner_tags[t] for t in seq if t != -100]
    for seq in true_labels
]
pred_tags = [
    [ner_tags[p] for i, p in enumerate(seq)
     if true_labels[j][i] != -100]
    for j, seq in enumerate(pred_labels)
]

# span-level F1 报告
report = classification_report(true_tags, pred_tags, scheme=IOB2)
print(report)
#               precision    recall  f1-score   support
# PER           0.882      0.891     0.886      1205
# ORG           0.794      0.812     0.803       687
# LAW           0.763      0.745     0.754       521

首次训练的 F1 约 0.828，不算差但远非完美。接下来要做错误分析。

错误分析与数据集迭代

我们逐一检查模型预测错误的样本，发现三类典型问题：

  1. 法条引用边界不一致：标注员把"《刑法》第264条"标为 LAW（正确），但有些只标了"《刑法》"，漏掉了"第264条"。→ 回溯修复标注规范。
  2. 嵌套实体未处理："北京市海淀区人民法院"既是 ORG，又包含地名"北京市"。我们的标签方案未支持嵌套 NER，导致模型混淆。→ 暂时维持平面 BIO，记录在已知局限中。
  3. 刑期格式多样性："有期徒刑三年"、"拘役两个月"、"罚金人民币一万元"——刑期的表述方式超过 20 种。→ 建议标注员在标注规范中做统一化处理。
迭代节奏：不要等到所有数据都标注完美了才训模型。我们建议"三明治迭代"——训练→错误分析→修复数据→再训练，每轮迭代 1-2 天。三轮迭代后，数据集从 v1 进化到 v4，F1 从 0.828 提升到 0.876。关键不是一次性把数据做到完美，而是"训-析-改"这个循环本身。

生产管线部署与监控

最后，将整条数据管线和模型部署到生产环境。核心组件包括：

  • 数据采集模块：每周自动从裁判文书网增量采集新文书（Airflow DAG）
  • 数据清洗模块：封装为 clean_pipeline.py 脚本，输入原生文本，输出清洗后的 BIO 数据集
  • 模型推理端点：使用 Hugging Face Inference Endpoints 部署 NER 模型
  • 数据漂移监控：每周对比新数据的标签分布与训练数据的分布，如果 JS 散度超过阈值（如 0.1），触发重新训练
# 数据漂移监控示例
from scipy.spatial.distance import jensenshannon

def monitor_drift(reference_dist, production_dist, threshold=0.1):
    """监控标签分布漂移"""
    js_div = jensenshannon(reference_dist, production_dist)
    if js_div > threshold:
        print(f"警告：JS 散度 {js_div:.4f} 超过阈值 {threshold}")
        # 触发重新训练
        return True
    return False
生产环境提醒：数据漂移不是"如果发生"而是"何时发生"。法律文本的变化是有明确模式的——2026 年新出台的司法解释会让某些法条引用的出现频率突变。一定要在生产管线中嵌入数据漂移监控，否则模型精度会在你不知情的情况下逐渐劣化。
