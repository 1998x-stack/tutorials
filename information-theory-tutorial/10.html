大家好，我是你们的讲师。

Huffman编码有一个"痛处"：每个符号必须分配整数个比特。如果某个符号的概率是2的负整数次幂，那是完美的——比如概率0.5就是1位，0.25就是2位。但如果概率是0.9呢？自信息 -log₂(0.9) ≈ 0.152比特——Huffman只能给它1比特，浪费了5倍多。

你可能会说："那就把多个符号打包编码呗。"没错，Huffman块扩展确实可以缓解这个问题——把k个符号拼成一个"超级符号"，每个超级符号的码长渐近趋近于熵。但代价是码表大小从|X|暴涨到|X|^k，对于稍微大一点的字母表（比如文本的256个字符），k稍微大一点就完全不可行了。

算术编码（Arithmetic Coding）用一种非常优雅的方式绕过了这个困境：它不把每个符号单独编码，而是把整条消息编码成一个[0,1)之间的实数。单个符号可以"占用"不到1比特的位置——分数比特不再是个问题。

我第一次理解算术编码的时候，脑子里浮现的是一把尺子——你把[0,1)区间不断细分，每多知道一个符号，就在更精确的位置上标记。最后，整个消息对应着尺子上一个独一无二的小区间。

区间编码的原理

算术编码的核心思想可以用一句话概括：

一条消息被表示为[0,1)上的一个子区间，区间长度等于消息的概率。消息越长，区间越窄，需要的比特数就越多（精确指定区间的位置）。

具体过程如下：

  1. 从整个区间 [low=0, high=1) 开始
  2. 对于信源中的每个符号，将当前区间按照符号概率分布进行细分
  3. 选择对应符号的那个子区间作为新的当前区间
  4. 处理完所有符号后，在最终区间内选择任何一个数来代表整条消息

解码时，接收端重复这个细分过程：给定一个数（它落在某个子区间内），就能判断出对应的符号。然后在这个子区间内重复细分，逐符号解码。

注意一个微妙的点：我们实际上不需要传输整个实数——只要传输足够多的比特来区分最终区间与其他所有可能的区间就可以了。需要多少比特？大约 -log₂(区间长度) = -log₂(消息的概率)。这正好等于消息的熵！

漂亮之处：算术编码避免了整数码长的限制——它可以"累积"多个低概率符号的小分数比特，最终用一个紧凑的二进制串表示。理论上，对于足够长的消息，算术编码的冗余度可以任意小。

算术编码的编码过程

让我们用一个具体例子来理解。假设信源有三个符号：A、B、C，概率分别为0.6、0.3、0.1。

初始区间：[0, 1)

每个符号的细分区间（始终按概率比例划分）：
  A: [0, 0.6)
  B: [0.6, 0.9)
  C: [0.9, 1.0)

消息："BAC"

逐步编码过程：

步骤1：编码符号 B（当前区间 [0, 1)）
  → B 对应 [0.6, 0.9)
  → 新区间: [0.6, 0.9)
  → 区间宽度 = 0.3

步骤2：编码符号 A（当前区间 [0.6, 0.9)）
  → A 在信源中占 [0, 0.6)，按比例映射到当前区间
  → 新区间 low = 0.6 + 0.0 * 0.3 = 0.6
            high = 0.6 + 0.6 * 0.3 = 0.78
  → 新区间: [0.6, 0.78)
  → 区间宽度 = 0.18

步骤3：编码符号 C（当前区间 [0.6, 0.78)）
  → C 在信源中占 [0.9, 1.0)，按比例映射
  → 新区间 low = 0.6 + 0.9 * 0.18 = 0.762
            high = 0.6 + 1.0 * 0.18 = 0.78
  → 最终区间: [0.762, 0.78)
  → 区间宽度 = 0.018

消息 "BAC" 对应的最终区间: [0.762, 0.78)

我们只需要传输一个位于这个区间中的数字——比如0.77——解码器就能唯一确定原始消息"BAC"。

理论上需要多少位？区间概率 = p(B) × p(A) × p(C) = 0.3 × 0.6 × 0.1 = 0.018。所以需要 -log₂(0.018) ≈ 5.8 bits。相比之下，Huffman编码给A、B、C分别分配1、2、3位（需要构造树），编码"BAC"需要2+1+3=6位。算术编码略优。

而且随着消息变长，Huffman的整数码长浪费会持续累积，而算术编码可以任意接近理论极限。

解码过程与精度问题

解码是编码的逆过程。假设收到了编码值 0.77：

解码步骤1：0.77 落在哪个符号区间？
  [0, 0.6) → A? 0.77 ≥ 0.6，不是
  [0.6, 0.9) → B? 0.77 ∈ [0.6, 0.9)，是！
  解码出 B，当前区间更新为 [0.6, 0.9)

解码步骤2：将 0.77 映射到当前区间内的位置
  pos = (0.77 - 0.6) / (0.9 - 0.6) = 0.567
  0.567 落在哪个符号区间？
  [0, 0.6) → A! pos = 0.567 ∈ [0, 0.6)
  解码出 A，当前区间更新为 [0.6, 0.78)

解码步骤3：再次映射
  pos = (0.77 - 0.6) / (0.78 - 0.6) = 0.944
  0.944 落在哪个符号区间？
  [0.9, 1.0) → C!
  解码出 C

解码完成：BAC

看起来很简单对不对？但实际实现时，有两个严峻的问题：

  • 精度问题：编码长消息时区间会变得极其狭窄。处理100个符号后，区间宽度可能是10^{-30}量级——标准浮点数根本无法表示。算术编码不可能真的用一个double逐次缩小区间。
  • 输出时机：需要等待所有符号处理完才能输出吗？那样编码器需要知道完整消息长度。

解决方案是重新归一化（Renormalization）——一个天才的工程技巧：

实际算术编码的关键优化：

当区间完全落在 [0, 0.5) 内时：
  - 输出 bit "0"
  - 将区间放大2倍：low ← 2*low, high ← 2*high
当区间完全落在 [0.5, 1) 内时：
  - 输出 bit "1"
  - 将区间缩放：low ← 2*(low-0.5), high ← 2*(high-0.5)
当区间跨过0.5时：
  - 继续等待，不输出

这样就实现了"边编码边输出"的效果，
并且始终用固定精度的整数算术（如32位或64位）。
实现算术编码时需要注意浮点精度问题！实际系统使用整数算术配合重新归一化（renormalization）来避免精度损失。标准的实现用32位或64位无符号整数表示区间的low和high，再加上"pending bits"机制处理区间跨0.5的情况。如果直接用浮点数实现，最多编码几十个符号就会因为精度不够而出错。

我记得自己第一次尝试实现算术编码时就踩了这个坑——用Python的float跑了一百个符号的编码，解码出来全是错的。查了半天，才发现区间宽度已经缩小到10^{-35}，远远低于浮点数的精度极限。改用整数算术后，一切才正常起来。

算术编码 vs. Huffman编码

既然算术编码和Huffman编码都能做无损压缩，它们的区别到底在哪？

对比维度 Huffman编码 算术编码
每符号码长 整数 分数（整数+分数总和）
理论极限 ≥ H(X)，差距 ≤ 1 bit 可任意接近 H(X)
符号概率为0.9时 浪费约0.85 bits 几乎无浪费
自适应能力 需要重建整棵树 只需更新概率表
编码/解码速度 快（查表即可） 慢（需要区间乘除法）
复杂度 简单 中等
专利历史 无专利（1952年） 曾经有多个专利
实际应用 JPEG（默认）、gzip H.264/H.265（CABAC）、JPEG 2000

从上表可以看出：算术编码在压缩率上优于Huffman，但代价是速度更慢、实现更复杂。所以工程上的选择很明确：

  • 如果你需要高速度且压缩率要求不是极端苛刻 → 选Huffman
  • 如果你需要最优压缩率 → 选算术编码

现代视频编码标准（H.264/H.265）中的CABAC（Context-Adaptive Binary Arithmetic Coding）就是算术编码的高阶形态——它结合了上下文建模和自适应概率估计，是目前视频压缩效率最高的熵编码方案。

让我们看一个Python的算术编码简单实现（仅用于教学，未做完整的重归一化优化）：

def arithmetic_encode(message, prob_table):
    """
    算术编码 - 教学版（用高精度Decimal避免部分精度问题）
    prob_table: {symbol: probability}
    返回编码后的区间 [low, high)
    """
    low, high = 0.0, 1.0

    for sym in message:
        span = high - low
        # 找到当前符号的累积概率区间
        cum = 0.0
        for s, p in prob_table.items():
            if s == sym:
                break
            cum += p
        # 缩小区间到该符号
        high = low + span * (cum + prob_table[sym])
        low = low + span * cum

    return (low + high) / 2  # 取区间中点

def arithmetic_decode(encoded_val, prob_table, msg_length):
    """
    算术解码 - 教学版
    """
    low, high = 0.0, 1.0
    result = []

    # 构建累积概率列表
    symbols = list(prob_table.keys())
    cum_probs = []
    c = 0.0
    for s in symbols:
        cum_probs.append(c)
        c += prob_table[s]

    for _ in range(msg_length):
        span = high - low
        # 确定编码值落在哪个符号区间
        pos = (encoded_val - low) / span
        for i, s in enumerate(symbols):
            if cum_probs[i] <= pos < cum_probs[i] + prob_table[s]:
                result.append(s)
                low = low + span * cum_probs[i]
                high = low + span * prob_table[s]
                break

    return ''.join(result)

# 测试
prob = {'A': 0.6, 'B': 0.3, 'C': 0.1}
msg = "BAC"
code = arithmetic_encode(msg, prob)
decoded = arithmetic_decode(code, prob, len(msg))
print(f"原消息: {msg}")
print(f"编码值: {code:.10f}")
print(f"解码后: {decoded}")
print(f"需要比特数: {-math.log2(0.018) if 'math' in dir() else '约5.8 bits'}")
# 输出：
# 原消息: BAC
# 编码值: 0.7700000000
# 解码后: BAC
工程经验：上面的教学版代码用浮点数实现，只适用于演示原理。在实战中，务必使用整数算术实现重归一化。Python有现成的arithmetic库可用，C++的cppcodec也提供了工业级实现。自己实现时，注意32位无符号整数的溢出边界。

算术编码的优雅在于，它从根本上解决了"整数码长"的限制。但它的代价是需要事先知道概率分布，而且要传输或者和编码数据一起存储这个分布信息。这就引出了一个自然的问题：有没有一种方法，不用事先知道概率分布，也能达到接近熵的压缩效果？

1977年，两位以色列研究者Lempel和Ziv给出了答案——他们通过"边压缩边学"的方式，创造了一种不需要先验知识的通用压缩方法。这就是下一章的内容：字典编码。
