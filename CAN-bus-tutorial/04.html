第4章 CAN总线错误处理：错误类型与状态机

各位同学，咱们今天聊聊CAN总线的错误处理。说实话，这是很多工程师容易忽略的部分。我刚入行时也觉得，只要报文能发出去不就行了？直到有一次在项目现场，整车的ECU突然集体“罢工”，我才意识到——错误处理机制，才是CAN总线稳定运行的基石。

4.1 为什么需要错误处理？

你想想看，汽车在高速上跑着，电磁干扰、温度变化、线束老化，这些都会让总线上的信号出问题。如果没有一套完善的错误处理机制，一个节点发错了数据，整个网络都可能被污染。

CAN总线的设计者很聪明。他们搞了一套“分布式错误检测”机制。每个节点都能检测错误，而且一旦发现错误，会立刻通知其他节点。说白了，就是让所有节点互相监督。

4.2 五种错误类型

CAN协议定义了五种错误类型。我在项目中基本都遇到过，一个一个说。

4.2.1 位错误（Bit Error）

这个最好理解。节点往总线上发了一个显性位（0），结果自己读回来发现是隐性位（1）。或者反过来。这就叫位错误。

为什么会这样？可能是总线被其他节点拉低了，也可能是信号反射导致的。我记得有一次调试，发现某个节点总是在特定温度下报位错误。查了半天，原来是PCB走线太长，信号质量不行。

关键点：发送节点会在发送每一位后，立即回读总线电平。如果读到的和自己发的不一致，就报位错误。但仲裁期间和ACK时隙除外——这两个场景下，位电平不一致是正常的。

4.2.2 填充错误（Stuff Error）

CAN总线用了NRZ编码。如果连续出现5个相同电平，接收节点可能会失去同步。所以协议规定：连续发送5个相同位后，必须插入一个相反电平的位。

如果接收节点发现连续6个相同位，就报填充错误。嗯，这里要注意：CRC界定符、ACK界定符这些固定格式的字段，是不做填充的。

我的经验：我曾经遇到一个案例，某款MCU的CAN控制器在接收时，偶尔会报填充错误。后来发现是晶振精度不够，导致采样点偏移。换了个高精度晶振，问题就解决了。

4.2.3 CRC错误（CRC Error）

CRC（循环冗余校验）是检查数据完整性的。发送节点计算CRC值，放在CRC字段里。接收节点重新计算一遍，如果对不上，就报CRC错误。

CAN的CRC算法是15位的，生成多项式是x¹⁵ + x¹⁴ + x¹⁰ + x⁸ + x⁷ + x⁴ + x³ + 1。你不需要记住这个，但要知道它的检错能力很强。

// CRC计算示例（伪代码）
uint16_t calc_can_crc(uint8_t *data, uint8_t len) {
    uint16_t crc = 0;
    for (int i = 0; i < len; i++) {
        crc ^= (data[i] << 7);
        for (int j = 0; j < 8; j++) {
            if (crc & 0x8000) {
                crc = (crc << 1) ^ 0x4599;  // 生成多项式
            } else {
                crc <<= 1;
            }
        }
    }
    return crc & 0x7FFF;  // 取15位
}

4.2.4 格式错误（Form Error）

CAN帧中有一些固定格式的位段。比如CRC界定符必须是隐性位，ACK界定符也必须是隐性位。如果检测到这些位置的电平不对，就报格式错误。

这个错误类型我遇到的比较少。但有一次，某个节点在发送EOF（帧结束）时，把最后一个隐性位拉成了显性。结果整个网络都在报格式错误。查到最后，是那个节点的CAN控制器驱动代码有bug。

4.2.5 应答错误（ACK Error）

发送节点在ACK时隙会释放总线（输出隐性位）。如果至少有一个接收节点正确收到了报文，它会把ACK时隙拉成显性。如果发送节点读到的ACK时隙是隐性，就说明没有节点正确接收，报应答错误。

注意：应答错误不代表总线坏了。可能是总线上只有发送节点自己，没有其他节点在监听。我在做单节点测试时就经常遇到这个情况，属于正常现象。

4.3 错误状态机

每个CAN节点都有两个计数器：发送错误计数器（TEC）和接收错误计数器（REC）。根据这两个计数器的值，节点会处于三种状态之一。

状态 条件 行为
主动错误 TEC < 128 且 REC < 128 正常通信，检测到错误时发送主动错误帧（6个显性位）
被动错误 TEC ≥ 128 或 REC ≥ 128 可以收发，但检测到错误时只能发送被动错误帧（6个隐性位）
总线关闭 TEC > 255 完全断开与总线的连接，不参与任何通信

4.3.1 主动错误状态

这是正常状态。节点可以自由收发报文。如果检测到错误，它会发送一个主动错误帧——6个连续的显性位。这个帧会破坏当前报文，让所有节点都知道出错了。

主动错误帧的优先级很高。因为它全是显性位，会覆盖总线上其他节点的隐性位。说白了，就是“我发现了错误，大家都停下来重传”。

4.3.2 被动错误状态

当TEC或REC超过127时，节点进入被动错误状态。这时候它还能收发报文，但发送错误帧时只能发6个隐性位。

隐性位会被显性位覆盖。所以如果总线上有其他主动错误节点，被动错误节点的错误帧根本起不了作用。只有等所有节点都变成被动错误，被动错误帧才能生效。

避坑指南：我曾经遇到一个项目，某个节点频繁进入被动错误状态。它发送的报文经常被其他节点覆盖，导致通信不稳定。后来发现是它的CAN收发器驱动能力不足，换了个收发器就好了。

4.3.3 总线关闭状态

如果TEC超过255，节点就彻底关闭了。它不再参与总线通信，直到用户手动恢复，或者硬件自动恢复（取决于配置）。

这个状态很危险。一个节点总线关闭，意味着它完全失联了。在汽车上，这可能导致某个功能彻底失效。

重要提醒：总线关闭后，节点需要等待128个连续的11个隐性位（即128个总线空闲状态）才能重新进入主动错误状态。这个机制是为了防止频繁出错的节点反复干扰总线。

4.4 错误计数规则

错误计数器的增减规则，我简单总结一下：

  • 发送成功：TEC减1（前提是TEC > 0）
  • 接收成功：REC减1（前提是REC > 0）
  • 发送错误：TEC加8
  • 接收错误：REC加1
  • 主动错误帧发送后：TEC加8
  • 被动错误帧发送后：TEC加8

这个规则设计得很巧妙。发送错误的惩罚比接收错误重得多。因为发送错误往往意味着节点本身有问题，而接收错误可能是临时的干扰。

4.5 实际项目中的经验

最后分享几个我在项目中踩过的坑：

  1. 不要忽视REC的增长。如果某个节点的REC持续增长，说明它接收到了很多错误帧。这可能是总线干扰，也可能是其他节点在疯狂发错误帧。
  2. 总线关闭后不要立即恢复。我见过有人写代码，检测到总线关闭就立刻复位CAN控制器。结果节点反复关闭、恢复，把整个总线都搞乱了。
  3. 错误状态要上报。在汽车电子项目中，ECU的错误状态应该通过诊断报文上报。这样工程师在售后阶段才能分析问题。

我的习惯：在开发阶段，我会把TEC和REC的值通过串口打印出来。这样能直观地看到节点的健康状态。量产时再把这个功能关掉，避免影响总线负载。
