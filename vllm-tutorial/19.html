第19章 张量并行

从这一章开始，咱们进入"高级专题篇"。张量并行（Tensor Parallelism, TP）是 vLLM 解决"模型太大，单 GPU 装不下"的核心方案。它基于 Megatron-LM 提出的算法，将单个 Transformer 层的权重矩阵切分到多个 GPU 上。

19.1 为什么需要张量并行

模型大小 vs GPU 显存:
┌────────────────┬──────────────┬──────────────────┐
│ 模型            │ FP16 显存需求 │ 单 GPU 能装下吗？ │
├────────────────┼──────────────┼──────────────────┤
│ Llama-3-8B     │ ~16 GB       │ ✅ A100-80GB     │
│ Llama-3-70B    │ ~140 GB      │ ❌ 需要 2×A100   │
│ DeepSeek-R1    │ ~140 GB(激活)│ ❌ 需要 4×A100   │
│ Llama-3-405B   │ ~810 GB      │ ❌ 需要 11×A100  │
└────────────────┴──────────────┴──────────────────┘

张量并行的作用: 把 N 张 GPU 变成"1 张更大的 GPU"

19.2 张量并行的原理

核心思想：把一个层的权重矩阵切分到多个 GPU，每 GPU 只存一部分，用通信同步中间结果。

Attention 层的分片 (TP=4, 32 heads):

GPU 0: heads[0:8]  - W_Q_0, W_K_0, W_V_0, W_O_0[0:8]
GPU 1: heads[8:16] - W_Q_1, W_K_1, W_V_1, W_O_1[8:16]
GPU 2: heads[16:24] - W_Q_2, W_K_2, W_V_2, W_O_2[16:24]
GPU 3: heads[24:32] - W_Q_3, W_K_3, W_V_3, W_O_3[24:32]

前向传播:
1. 输入 X 复制到所有 GPU (每个 GPU 都有完整副本)
2. 每个 GPU 独立计算自己的 8 个 heads
3. All-Reduce: 同步合并 O = concat(O_0, O_1, O_2, O_3)
4. 进入下一层...

MLP 层的分片 (TP=4):
GPU 0: W_up 的 1/4 列, W_gate 的 1/4 列, W_down 的 1/4 行
...类似逻辑

19.3 通信开销：张量并行的代价

每层需要 2 次 All-Reduce（Attention 后一次，MLP 后一次），通信量取决于 GPU 间带宽：

互联方式带宽TP 推荐度场景
NVLINK (A100)600 GB/s⭐⭐⭐单节点 TP=8
NVLINK (H100)900 GB/s⭐⭐⭐单节点 TP=8
PCIe 4.0 ×16~32 GB/s⭐⭐TP≤2 可接受, TP=4 降速
PCIe 5.0 ×16~64 GB/s⭐⭐TP≤4 可接受
InfiniBand (跨节点)25-50 GB/s⭐不推荐 TP 跨节点
以太网 (跨节点)1-10 GB/s❌不要 TP 跨节点

黄金法则：张量并行在同一节点内（有 NVLINK）表现最好。跨节点一定要用流水线并行而不是张量并行。NVLINK 带宽是 PCIe 的 10-30 倍，是跨节点网络的 20-100 倍。

19.4 使用方式

# 2 GPU 张量并行
vllm serve meta-llama/Llama-3-70B --tensor-parallel-size 2

# 4 GPU 张量并行
vllm serve meta-llama/Llama-3-70B --tensor-parallel-size 4

# Python API
from vllm import LLM
llm = LLM("meta-llama/Llama-3-70B", tensor_parallel_size=4)

# 验证是否用了 NCCL
# 日志中应看到:
# NCCL INFO: Using network: NET/IB/GDRDMA  ← 好，用了 RDMA
# NCCL INFO: Using network: NET/Socket     ← 差，回退 TCP

踩坑：Docker 里 TP 通信经常失败，两个常见原因：1) --ipc=host 没加 — NCCL 用共享内存做初始化；2) --shm-size 太小 — 多 GPU NCCL 需要大量共享内存。建议至少 8GB。

19.5 性能分析

Llama-3-70B, A100-80GB, FP16:

TP=1: ❌ OOM (140GB 权重大于 80GB 显存)
TP=2: 吞吐 ~800 tok/s, 每 GPU 显存 ~70GB (权重 70GB + KV cache)
TP=4: 吞吐 ~600 tok/s, 每 GPU 显存 ~50GB (权重 35GB + KV cache)
       ↑ 吞吐下降是因为 4 GPU 的 All-Reduce 通信开销增大

结论：TP=2 在这个配置下最优（最小通信开销的可用方案）
