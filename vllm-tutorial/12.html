第12章 V1架构上篇：多进程与EngineCore

2025 年 1 月，vLLM 发布了 V1 Alpha——这是一次彻底的架构重写。V0 被正式弃用。这一章咱们理解 V1 最核心的架构变化：多进程引擎和执行核心（EngineCore）。

12.1 为什么要重写整个架构

V0 是一个"原型级"实现——它能跑、性能也好，但随着功能增加，代码库变得越来越难维护：

  • 单进程：所有逻辑（API 服务、调度、模型执行）挤在一个进程里
  • 硬编码两阶段：prefill 和 decode 的调度器是两个独立实现，代码里有大量 if-else
  • 耦合严重：改 KV cache 管理可能影响 API 路由，改调度器可能影响模型加载
  • 新功能开发困难：每加一个新特性（如推测解码、多模态 encoder），要在多个模块间穿线

重写决策：V1 的 RFC #18571 讨论了 3 个月。最终结论是——与其在 V0 上修修补补，不如重写。目标是建立一个能支撑未来 5-10 年发展的架构。

12.3 故障隔离：最大的架构收益

在 V0 的单进程架构中，任何未捕获异常都会导致整个服务崩溃——API 端口关闭，所有请求丢失，需要重启。

在 V1 中：

Worker 2 崩溃 (OOM / CUDA error / segfault):
  ├── Master 检测到 Worker 2 不响应
  ├── 自动将 Worker 2 的请求重新路由到 Worker 1/3
  ├── 重启 Worker 2（加载模型需要 30s-2min）
  └── Worker 2 恢复后重新加入服务

在此期间：
  ✅ API 端口正常运行
  ✅ Worker 1/3 继续服务
  ✅ 用户请求丢到已崩溃 Worker 2 的可能失败，但整体服务没挂

生产价值：这个故障隔离对在线服务极其重要。我在 V0 时代见过一个 CUDA OOM 导致整个 API 服务挂了 3 分钟——所有用户都受影响。V1 中同样的 OOM 只影响一个 Worker，系统自动恢复。

12.4 EngineCore 双线程设计

EngineCore 内部：

主线程（控制面）:
  ├── 接收 Master 发来的请求
  ├── 运行调度器（决定 batch 组成）
  ├── 管理 Block Table（分配/回收）
  └── 将准备好的 batch 交给计算线程

计算线程（数据面）:
  ├── GPU 模型前向传播
  ├── CUDA Graph 回放
  └── 将结果返回给主线程 → Master → 用户

双线程的好处：主线程在做"下一轮怎么调度"的决策时，计算线程在并行跑当前的 GPU 推理。这就是 V1 的"近乎零 CPU 开销"——GPU 不用等 CPU 做调度决策。

如果 N 个 Worker 跑同一个模型，每个 Worker 需要独立加载模型吗？不——vLLM 支持同一进程组的模型共享：
