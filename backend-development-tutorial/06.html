开篇：算法不是面试题，是生产环境的"救命稻草"

很多后端工程师对数据结构和算法的印象停留在"面试造火箭"上——LeetCode 刷了几百道题，进了公司后发现日常工作就是 CRUD，算法好像根本用不上。这种认知其实是一种错觉，区别在于：面试题里的算法是你主动写的，而生产环境的算法是被动藏在框架和数据库内部的。

我亲身经历过一次"算法差距"带来的性能天堑。那时我负责维护一个用户标签系统，需求是根据标签组合来查询匹配的用户。最初版本的实现非常简单：用 Redis 的 Set 存储每个标签下的用户 ID 集合，查询时对多个 Set 做 SINTER（交集）操作。线上运行了几个月都很正常，直到标签数量从几百增长到几万，每个标签下的用户从几千涨到几百万。SINTER 操作的耗时从几毫秒飙升到了十几秒。Redis 的 Set 交集算法在最坏情况下需要 O(N * M) 的时间——当集合足够大时，这个复杂度变得不可接受。

最终我们重建了数据模型：改用 Redis Bitmap 存储用户标签矩阵，交集用 BITOP AND 实现。Bitmap 的按位与操作在 CPU 层面就是一条指令，无论 Bitmap 多大（在合理范围内），速度都比 Set 交集快几个数量级。同一个查询，从 12 秒降到了 30 毫秒。这就是"选对数据结构"和"选错数据结构"的差距——不是 10%，是 400 倍。

🔑 核心认知：数据结构的选择决定了系统性能的"天花板"。数据库选型、缓存策略、搜索引擎——所有这些底层系统的性能，根本上都是由它们使用的数据结构决定的。理解数据结构的原理，让你能在技术选型和性能优化时做出更好的判断，而不是停留在"调个参数试试"的层面。

哈希表原理与 Redis 中的应用

哈希表（Hash Table / Dict）是后端开发中使用频率最高的数据结构。你可能每天都在用——Go 的 map、Python 的 dict、Redis 的 Hash，但它的内部机制远比"key-value 查找"要复杂。

核心机制：哈希与冲突解决

哈希表的核心思想是将 key 通过哈希函数映射到一个数组索引上。当两个 key 映射到同一索引时（哈希冲突），最常见的解决方式是链地址法（同一槽位用链表链接多个 entry）。Redis 的 dict 使用的就是链地址法，但它有一个非常精巧的设计——渐进式 rehash。

// Redis dict 数据结构（简化版）
struct dict {
    dictEntry **ht_table[2];  // 两张哈希表，用于渐进式 rehash
    long rehashidx;           // rehash 进度，-1 表示未进行
    unsigned long used;       // 已使用的 entry 数量
};

struct dictEntry {
    void *key;
    void *val;
    dictEntry *next;  // 链地址法的下一个 entry
};

// 渐进式 rehash 过程（伪代码）
// 每次对 dict 执行增删改查时，顺便迁移 rehashidx 指向的桶
void dictRehash(dict *d, int n) {
    while (n--) {
        if (d->ht_table[0][d->rehashidx] == NULL) {
            d->rehashidx++;
            continue;
        }
        // 将 ht_table[0] 当前桶的全部 entry 迁移到 ht_table[1]
        dictEntry *entry = d->ht_table[0][d->rehashidx];
        while (entry) {
            dictEntry *next = entry->next;
            int idx = hash(entry->key) & (ht_size[1] - 1);
            entry->next = d->ht_table[1][idx];
            d->ht_table[1][idx] = entry;
            entry = next;
        }
        d->ht_table[0][d->rehashidx] = NULL;
        d->rehashidx++;
    }
}

渐进式 rehash 的精妙之处在于：它将一次性的、可能耗时很长的扩容操作，分散到了后续的每一次增删改查中。Redis 是单线程事件循环架构，如果 rehash 一次做完，大量 key 迁移会导致服务在几十到几百毫秒内无法处理其他请求。渐进式 rehash 保证了每次操作的延迟都是可控的，不会因为扩容而产生"抖动"。这一设计哲学对后端系统设计有普遍借鉴意义——不要把大操作集中在某个时间点，而要将其摊销到时间轴上。

📖 延伸思考：Go 的 map 使用"增量扩容"思路，每次写入操作最多迁移 2 个 bucket。Java 的 HashMap 则是"一次性扩容"，所以提前预估 HashMap 的初始容量可以避免扩容带来的性能损失。不同语言对同一数据结构的实现取舍，反映的是它们对延迟敏感度的不同假设。

B+Tree 与数据库索引

为什么关系型数据库（PostgreSQL、MySQL、SQLite）几乎都使用 B+Tree 作为默认索引结构，而不是二叉搜索树、红黑树或者哈希表？答案是：磁盘 I/O 的代价决定了索引结构的设计。

后端服务的数据通常存储在磁盘上——内存贵且易失。磁盘随机 I/O 比顺序 I/O 慢几个数量级（机械硬盘上万倍，SSD 也差一个数量级）。因此，索引设计的核心优化目标是"减少磁盘 I/O 次数"，而不是"减少 CPU 操作次数"。

数据结构 树高（百万数据） 每次搜索 I/O 次数 为什么不适合做磁盘索引
二叉搜索树 ~20 20 次随机 I/O 每个节点对应一次磁盘 I/O，树高太高
红黑树 ~20 20 次随机 I/O 平衡了但仍然太深
B-Tree ~3-4 3-4 次随机 I/O 每个节点存储多个 key，但范围查询需要回溯父节点
B+Tree ~3-4 3-4 次随机 I/O 所有数据在叶子节点，叶子节点通过指针串联，范围查询只需一次顺序扫描
哈希表 ~1 1-2 次随机 I/O 不支持范围查询，不支持排序，不支持前缀匹配

B+Tree 相比 B-Tree 的关键改进是：只有叶子节点存储数据，非叶子节点只存储 key 和子节点指针。这意味着：第一，非叶子节点可以容纳更多 key，降低了树高；第二，叶子节点之间通过指针形成一个有序链表，范围查询（如 WHERE age BETWEEN 20 AND 30）找到一个起始叶子节点后，沿着指针顺序扫描即可，不需要回溯到父节点。

💡 索引优化启示：理解 B+Tree 的工作原理后，很多"最佳实践"就不再只是死记硬背。比如"联合索引的最左前缀原则"——B+Tree 的 key 是按定义的列顺序组合排序的，所以跳过最左边列直接查后面的列无法利用索引。再比如"不要对大文本列建索引"——B+Tree 节点大小通常是 16KB（InnoDB 的一个 page），每个 key 越小，节点能容纳的 key 越多，树越矮 I/O 越少。

跳表（Skip List）：Redis 有序集合的底层实现

跳表（Skip List）是一种"用概率换性能"的数据结构，它的代码实现比平衡树简单许多，但性能却可以媲美红黑树（O(log N) 的搜索、插入、删除）。

跳表的核心思想是：在一个有序链表的基础上，为每个节点按概率（通常 1/2）创建"高层指针"。高层指针可以"跳过"大量中间节点，实现类似二分查找的效果。Redis 的 sorted set（ZSET）就是用跳表作为底层实现的——当你需要同时支持"按 score 排序"和"快速按 key 查找 score"时，跳表 + 哈希表的组合是一个经典架构。

🔍 工程选择逻辑：Redis 作者为什么用跳表而不是平衡树实现 ZSET？作者的官方回答是：跳表实现更简单、更容易调试、内存占用相当、范围查询更容易。这个选择背后是一个重要的工程原则——在满足性能要求的前提下，选最简单的实现。跳表的代码量大约是红黑树的 1/3，这意味着更少的 bug、更低的维护成本。

LRU/LFU 缓存淘汰：实现原理与 Redis 内存策略

缓存是后端性能优化的第一手段。但缓存的核心问题是：当内存满了，淘汰哪些数据？

  • LRU（Least Recently Used）：淘汰最近最少使用的数据。实现方式通常是"哈希表 + 双向链表"——哈希表提供 O(1) 查找，双向链表维护访问顺序。每次访问一个 key 就将其移至链表头部，淘汰时从链表尾部移除。这是最常用的缓存淘汰策略。
  • LFU（Least Frequently Used）：淘汰访问频率最低的数据。需要维护每个 key 的访问计数。问题在于：一个数据刚被高频访问后就不再被访问，其计数器仍然很高，导致"缓存污染"。Redis 的 LFU 实现用"衰减因子"解决这个问题——计数会随时间衰减。
// 实现一个简单的 LRU 缓存（Go 语言）
type LRUCache struct {
    capacity int
    cache    map[int]*list.Element
    list     *list.List  // 标准库的双向链表
}

type entry struct {
    key   int
    value int
}

func NewLRUCache(capacity int) *LRUCache {
    return &LRUCache{
        capacity: capacity,
        cache:    make(map[int]*list.Element),
        list:     list.New(),
    }
}

func (c *LRUCache) Get(key int) int {
    if elem, ok := c.cache[key]; ok {
        c.list.MoveToFront(elem)           // 移到头部 = 最近使用
        return elem.Value.(*entry).value
    }
    return -1
}

func (c *LRUCache) Put(key, value int) {
    if elem, ok := c.cache[key]; ok {
        elem.Value.(*entry).value = value
        c.list.MoveToFront(elem)
        return
    }
    if c.list.Len() >= c.capacity {
        tail := c.list.Back()               // 淘汰最久未使用的
        delete(c.cache, tail.Value.(*entry).key)
        c.list.Remove(tail)
    }
    elem := c.list.PushFront(&entry{key, value})
    c.cache[key] = elem
}

Redis 支持多种淘汰策略，2026 年推荐在大多数场景下使用 allkeys-lru（对所有 key 执行 LRU 淘汰）。对于有明确冷热分层的数据，volatile-lru（只对设置了 TTL 的 key 淘汰）也很常用。如果你的缓存数据有明显的访问频率差异且存在"突发热点"，allkeys-lfu 可能是更好的选择。

⚠️ 踩坑实录：我们曾有一个 Redis 缓存大量用户会话数据的场景。上线时配置了 allkeys-lru，运行一段时间后发现活跃用户的会话频繁被淘汰，非活跃用户的会话反而一直留着。排查发现这是因为我们的数据访问模式是"周期性批量扫描"——后台定时任务每隔 5 分钟扫描一次全量用户数据，把每个用户的记录都 Get 了一遍，导致所有 key 在 LRU 视角下都变成了"最近使用过的"。解决方案是把淘汰策略换成 allkeys-lfu，因为批量扫描虽然会访问所有 key，但每个 key 的访问频率远低于实际活跃用户的频率。

实战案例：用正确的数据结构把查询从 3 秒降到 3 毫秒

让我们看一个综合案例。假设你有一个电商后台系统，需要实现一个功能：根据用户的地理位置（经纬度）查询附近的商铺。

如果你把所有商铺坐标存到关系型数据库中，用 WHERE lat BETWEEN ? AND ? AND lng BETWEEN ? AND ? 来做范围查询，数据量到 100 万条时查询时间会轻松超过 1 秒。如果不加合适的索引，3 秒甚至更久都很正常。

选择正确的数据结构可以彻底改变结果：

  • GeoHash：将二维经纬度编码为一维字符串，本质上是空间数据的 B+Tree 索引。前缀匹配越长的 GeoHash 代表距离越近。PostgreSQL 的 PostGIS 扩展就是用类似原理实现的。
  • R-Tree：专门为空间数据设计的多维索引结构，是大多数空间数据库的底层实现。
  • Redis Geo：使用 GeoHash + 跳表（ZSET）实现，可以在毫秒级完成"查找附近 5 公里的商铺"操作。

最终，用 Redis Geo 将这个查询从 3 秒降到了 3 毫秒——这不是"优化"出来的，是"选对了数据结构"得来的。数据结构的差距，就是 1000 倍的差距。
