第11章 CAN报文接收：接收FIFO机制、接收过滤器匹配、接收中断处理、接收超时处理

好，咱们今天聊聊CAN报文的接收。说实话，很多初学者觉得发送比接收难，其实恰恰相反。接收才是真正考验系统稳定性的地方。我见过太多项目，发送跑得挺欢，一收数据就丢帧、乱序、甚至死机。嗯，这里面的门道，咱们一个一个掰开讲。

11.1 接收FIFO机制：为什么需要它？

先问个问题：CPU处理一条报文需要多久？假设你的主频是72MHz，处理一条中断服务函数大概要几十微秒。而CAN总线在500kbps速率下，一条标准帧也就134微秒左右。你想想看，如果每来一条报文就中断一次CPU，那CPU基本啥也别干了，光顾着响应中断了。

所以，硬件上引入了FIFO（先进先出队列）。说白了，就是一个硬件缓冲区。报文来了，先存到FIFO里，CPU有空了再慢慢取。这样CPU就不用被频繁打断，系统吞吐量自然就上去了。

FIFO的核心作用：

  • 缓冲多条报文，防止丢帧
  • 降低CPU中断频率
  • 提供时间戳，方便超时判断

我在项目中遇到过一个问题：某款MCU的FIFO深度只有3个槽位，而总线负载率高达70%。结果就是，连续来4条报文，第4条直接覆盖了第1条。后来我换了个FIFO深度8的芯片，问题才解决。所以选型时，FIFO深度是个硬指标，别忽视。

11.2 接收过滤器匹配：别让CPU处理无关数据

CAN总线上挂着几十个节点，每个节点只关心特定ID的报文。如果所有报文都进FIFO，CPU得一个个过滤，效率太低了。所以硬件提供了接收过滤器。

接收过滤器的工作原理，说白了就是：硬件帮你做ID匹配。匹配上了，报文进FIFO；匹配不上，直接丢弃。CPU根本不知道有这回事。

常见的过滤器模式有两种：

模式 说明 适用场景
掩码模式 通过掩码指定哪些位需要匹配 需要接收一组连续ID
列表模式 列出所有需要接收的ID 只接收几个特定ID

我个人习惯用掩码模式。比如我需要接收ID范围为0x100到0x10F的报文，掩码设为0x7F0，ID设为0x100，这样高7位匹配即可。省事，也省过滤器资源。

小技巧：配置过滤器时，记得把IDE位和RTR位也考虑进去。我曾经因为没配IDE位，导致扩展帧和标准帧混在一起，排查了半天才发现问题。

11.3 接收中断处理：别在中断里干重活

报文匹配成功进入FIFO后，硬件会触发接收中断。这时候CPU该干嘛？很多新手会直接在中断里处理报文——解析、校验、甚至调用上层函数。这是大忌！

为什么？因为中断服务函数执行时间越长，其他中断被阻塞的风险就越大。CAN总线本身就有实时性要求，你这边处理报文花了500微秒，那边发动机转速信号可能就丢了。

正确的做法是：中断里只做一件事——把报文从FIFO搬到内存缓冲区。然后置一个标志位，退出中断。主循环里再慢慢处理。

// 接收中断服务函数（精简版）
void CAN_RX_IRQHandler(void)
{
    CanRxMsg rxMsg;
    
    // 从FIFO读取报文
    CAN_Receive(CAN1, CAN_FIFO0, &rxMsg);
    
    // 存入环形缓冲区
    if (RingBuf_Write(&g_rxBuf, &rxMsg) == 0)
    {
        // 缓冲区满了，记录丢帧
        g_rxOverflowCount++;
    }
    
    // 清除中断标志
    CAN_ClearITPendingBit(CAN1, CAN_IT_FMP0);
}

你看，这个中断函数只做了三件事：读FIFO、写缓冲区、清标志。耗时不超过10微秒。主循环里再慢慢解析报文，这样就不会影响其他中断了。

注意：环形缓冲区一定要设计成无锁的。如果主循环和中断同时操作同一个缓冲区，可能会出问题。我一般用读写指针分离的方式，中断只写，主循环只读，互不干扰。

11.4 接收超时处理：别让报文烂在缓冲区里

报文进了缓冲区，不代表万事大吉。如果主循环处理不过来，或者某个任务被阻塞了，报文就会一直积压在缓冲区里。时间一长，这些报文就失去了时效性。

比如，车速信号每10ms发一次，你20ms后才处理，那这数据就没意义了。所以，我们需要超时处理机制。

具体做法：每条报文入缓冲区时，记录一个时间戳。主循环处理时，检查当前时间减去时间戳是否超过阈值。如果超时，直接丢弃，不处理。

// 报文结构体，带时间戳
typedef struct {
    uint32_t id;
    uint8_t data[8];
    uint8_t len;
    uint32_t timestamp;  // 入队时的时间戳
} CanMsgWithTs;

// 主循环处理函数
void ProcessRxMessages(void)
{
    CanMsgWithTs msg;
    uint32_t now = GetSysTick();
    
    while (RingBuf_Read(&g_rxBuf, &msg) == 1)
    {
        // 检查是否超时（阈值设为50ms）
        if ((now - msg.timestamp) > 50)
        {
            g_rxTimeoutCount++;
            continue;  // 直接丢弃
        }
        
        // 正常处理报文
        HandleCanMessage(&msg);
    }
}

我曾经在一个项目中，因为某个任务被高优先级任务抢占，导致报文处理延迟了200ms。结果就是，ECU根据过时的车速数据算出了错误的喷油量，发动机抖得跟筛子似的。从那以后，我每个接收模块都加了超时检查，再也没出过类似问题。

避坑指南：超时阈值设多少合适？我一般按报文周期的1.5倍来设。比如10ms周期的报文，阈值设15ms。太短容易误丢，太长又失去意义。你可以根据实际负载调整。

11.5 综合实战：一个完整的接收流程

好了，咱们把上面几个知识点串起来，看看一个完整的接收流程长什么样：

  1. 硬件过滤：报文进入CAN控制器，过滤器匹配ID。不匹配的直接丢弃。
  2. FIFO缓冲：匹配成功的报文进入FIFO。如果FIFO满了，新报文会覆盖最旧的报文（取决于芯片设计）。
  3. 中断触发：FIFO非空时，硬件触发接收中断。中断服务函数将报文从FIFO搬到环形缓冲区，并记录时间戳。
  4. 主循环处理：主循环检查缓冲区是否有新报文。有则读取，检查超时，未超时则解析处理。
  5. 错误统计：记录丢帧次数、超时次数，用于系统健康监测。

这个流程看起来简单，但每个环节都可能出问题。我建议你在调试阶段，把丢帧计数和超时计数通过串口打印出来。如果发现丢帧，先看FIFO深度够不够；如果超时多，说明主循环处理不过来，得优化代码或者提高优先级。

最后说一句：接收这块，硬件能做的尽量让硬件做，CPU只做最核心的解析工作。别想着用软件去弥补硬件的不足，那只会让系统越来越复杂，越来越不稳定。
