第五章：文本分块策略

大家好，欢迎来到RAG课程中最容易被低估的一章。如果你问我RAG系统中哪个环节对最终效果影响最大但投入精力最少，我的回答一定是——文本分块（Chunking）。

为什么？因为分块决定了检索的"颗粒度"。块太大，一个chunk包含太多主题，检索出来噪音大，LLM不知道重点在哪里；块太小，单个chunk信息不完整，无法回答需要综合理解的问题。分块的质量直接决定了Recall的上限——如果正确答案被切成了两半，再牛的检索算法也找不回完整的答案。

一、为什么分块如此重要

分块的核心矛盾是：我们希望每个chunk包含的信息足够完整，但又希望它足够聚焦。一个理想的chunk应该像一个维基百科条目——独立、自包含、聚焦于一个主题。

我在2024年做过一次实验：同样一份50页的技术文档，用不同分块策略做RAG，在同一个测试集上Recall的差距可以达到30%以上。分块策略几乎是最"性价比"高的RAG优化手段——不涉及模型更换、不涉及复杂的算法调参，但效果提升立竿见影。

二、主流分块方式对比

分块策略 原理 优点 缺点 适用场景
Fixed-size 按固定字符数或token数切割，可选overlap 实现简单，速度最快 容易切断句子/段落，语义断裂 快速原型，内容统一的文档
Semantic 检测语义边界（段落/主题变化）进行切割 保持语义完整性，检索质量高 依赖语义边界检测质量，速度稍慢 知识问答、技术文档
Hierarchical 多层级分块（文档→章节→段落→句子） 支持不同粒度的检索，可逐层递进 索引复杂，检索时需要多级匹配 长文档、教科书级内容
Proposition 将文本拆解为原子化命题（assertion） 信息原子化，精确匹配 需要LLM辅助拆分，成本高 法律条文、事实性问答
Agentic 用LLM动态决定分块边界 最灵活，能按内容逻辑分块 高成本、延迟、不可控 高质量要求、复杂文档
关键洞察：没有"最好的分块策略"，只有"最适合你的数据和场景"的策略。我的建议是：从Semantic Chunking开始，用Fixed-size作为Baseline——先跑通，再优化。Semantic Chunking是一个质量/成本的甜蜜点，值得作为首选。

三、Semantic Chunking实战

下面是用LangChain的语义分块器实现的示例。核心思路是：先按标点符号和段落切分句子，然后基于句子间的语义相似度合并成块。

from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_experimental.text_splitter import SemanticChunker
from langchain_community.embeddings import HuggingFaceEmbeddings

# ——— 方法1: RecursiveCharacterTextSplitter（基于规则的语义近似） ———
recursive_splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=128,
    separators=["\n\n", "\n", "。", "！", "？", "；", "，", " ", ""],
    length_function=len,
)

# ——— 方法2: Semantic Chunker（基于语义相似度） ———
embedding_model = HuggingFaceEmbeddings(
    model_name="BAAI/bge-small-zh-v1.5"
)

semantic_splitter = SemanticChunker(
    embeddings=embedding_model,
    breakpoint_threshold_type="percentile",  # percentile/standard_deviation/interquartile
    breakpoint_threshold_amount=85,  # 百分位阈值
)

long_text = """
第一章 总则
第一条 为了规范人工智能技术的应用，保护公民合法权益，制定本条例。
第二条 本条例适用于中华人民共和国境内从事人工智能技术研发和应用的活动。
第三条 人工智能技术应用应当遵循安全可控、公平公正、透明可释的原则。

第二章 技术规范
第四条 人工智能系统应当具备可解释性，能够对其决策过程进行合理解释。
第五条 开发者应当对训练数据的合法性、准确性、完整性负责。
第六条 人工智能系统运行过程中应当记录运行日志，日志保存期限不少于三年。
"""

chunks = semantic_splitter.split_text(long_text)
for i, chunk in enumerate(chunks):
    print(f"Chunk {i}: {chunk[:60]}...")
    print("---")

四、Overlap策略与"半句话"问题

Fixed-size分块最典型的悲剧就是把一句话从中间切断。假设我们的chunk_size=200字符，但某个关键结论正好在第199个字符和第201个字符之间被切断了——检索时，两个半截chunk可能都排在很后面，LLM一个完整的上下文都拿不到。

解决方案是使用Chunk Overlap（块重叠）。在切割时，让相邻的chunk共享一部分内容（通常10%~20%）。这样即使关键信息落在边界附近，至少有一个chunk包含了完整信息。

血的教训：我曾经在处理中文法律条文时，发现模型对法律条款的理解总是出错。排查后发现，一个完整的法律条款（"当事人一方不履行合同义务或者履行合同义务不符合约定的，应当承担继续履行、采取补救措施或者赔偿损失等违约责任"）被从中间切成了两半——前半句存一个chunk，后半句存另一个。检索时系统只找到前半句，LLM基于"不完整的法律条文"给出了完全错误的判断。对于法律法规、技术规范这类"一句话决定生死"的内容，必须确保句子完整性。

五、分块大小的选择

分块大小影响检索质量、存储成本和生成质量，是一个需要仔细平衡的超参数：

  • 小chunk（~128 tokens）：适合精确的事实问答（"公司CEO是谁？"），检索精度高，但缺乏上下文。
  • 中chunk（~256-512 tokens）：最常见的配置，适合大多数场景，兼顾精度和上下文。
  • 大chunk（~1024+ tokens）：适合需要综合理解的问答（"总结第三章的主要内容"），但检索噪音增加。

一个实用的策略是Multi-size Chunking：对小chunk做向量检索、对大chunk做rerank时的上下文补充。

六、前沿：SCAR（Semantic Continuity-Aware Retrieval）

2026年6月，社区提出了SCAR（语义连续性感知检索）框架。它认为传统分块策略有一个根本性的缺陷：把连续的语义流硬性打断，导致chunk边界破坏了语义自然流动。SCAR的创新在于：

  • 不再预先固定分块边界，而是在检索时动态确定语义片段的起止。
  • 利用嵌入向量的连续性特征，检测"语义漂移点"来自然切割。
  • 对跨chunk的信息实现无缝拼接，极大降低了"半句话"问题的影响。

虽然SCAR目前还处于学术前沿，但它指向了一个清晰的方向：未来的分块将不再是索引阶段的"一刀切"，而是检索阶段"按需切割"。我们在设计系统时应保持架构的灵活性，为这类新技术预留扩展空间。

实用建议：如果你的RAG系统刚起步，建议采用"分层"分块方案——将文档同时存储两种粒度的chunk：小chunk（256 tokens + 25% overlap）用于精确语义检索，大chunk（512 tokens）作为检索到后的上下文补充。先用小chunk找"哪块最相关"，再用大chunk提供"完整的上下文套给LLM"。这种方案在多个生产项目中被验证是效果和成本的最佳平衡点。

本章小结

分块是RAG系统中最容易被忽视但影响最大的环节。我们对比了五种主流分块策略，实践了Semantic Chunking，探讨了overlap的必要性，并了解到了SCAR这样的前沿思想。记住：算法决定检索的下限，分块决定检索的上限。下一章，我们将进入"把chunk变成机器能理解的语义表示"的关键环节——向量嵌入技术。
