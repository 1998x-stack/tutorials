第24章 网络性能测试与基准

大家好，欢迎来到第24章。上一章我们学会了用Wireshark"看"网络的质量——看有没有重传、看延迟、看TCP窗口。但"看"是定性的，"测"才是定量的。很多工程师容易陷入一个误区：看到交换机端口指示灯亮绿灯就觉得"网络没问题"，看到流量跑起来了就觉得"带宽是够的"。但实际呢？"网络很慢"到底有多慢？"网络很快"到底有多快？这些都需要用数据说话。

这就是网络性能测试要做的事。它不是简单测一个"宽带速度"，而是系统性地测量网络四个核心维度：吞吐量（Throughput）、时延（Latency）、抖动（Jitter）和丢包率（Packet Loss）。这四个指标构成了网络性能的完整画像，缺一不可。

24.1 网络性能的四个维度

吞吐量（Throughput）是最直观的指标——单位时间内能传输的数据量，通常以bps（bits per second）衡量。千兆以太网的理论吞吐量上限大约是940 Mbps（扣除帧头、帧间隙和前导码的开销后），万兆以太网大约是9.4 Gbps。吞吐量反映了网络链路能"喂饱"上层的最大能力。

时延（Latency）是数据包从发送端到接收端所花费的时间。在一台交换机内，线速转发（cut-through）的时延通常在几微秒量级；存储转发（store-and-forward）则取决于帧长——1518字节的帧在千兆链路上约为12微秒。穿越广域网的时延则受光速和路由跳数限制：上海到北京约1300公里，光纤中的光速约为200,000公里/秒，纯传播时延就是6.5毫秒，加上交换机和路由器的处理时间，通常在10-20毫秒之间。

抖动（Jitter）是连续数据包之间时延的变化程度。对于视频会议和VoIP这类实时应用，抖动比绝对时延更致命——试想一下，如果每个包的到达间隔忽长忽短，视频就会卡顿、声音就会变调。抖动通常用连续包之间的时延差值的标准差来量化，良好的网络环境中抖动应该控制在1毫秒以下。

丢包率（Packet Loss）是传输过程中丢失的数据包占总发送包的百分比。在正常运转的有线以太网中，丢包率应该接近于零（绝大多数场景下小于0.01%）。如果出现高于1%的丢包率，通常意味着链路拥塞、硬件故障或光模块信号质量下降。

关键认知：这四个维度是相互影响的。当链路利用率接近饱和时，交换机队列开始填满，导致时延增加（排队时延），进而触发TCP拥塞控制减少发送窗口——最终表现为吞吐量下降。所以任何一个维度的恶化，都可能引发其他维度的连锁反应。性能测试的关键就在于找到这四个维度之间的平衡点。

24.2 iperf3详解

iperf3是目前最流行的网络性能测量工具，由ESnet（美国能源部科学网络）开发和维护。它采用客户端-服务器架构，能测量TCP和UDP两种模式下的吞吐量。它的设计哲学是"简单、专注、可重复"——不像一些商业测试工具那样功能繁杂，但它的测试结果在行业内被广泛认可。

TCP模式：iperf3在TCP模式下会发送尽可能多的数据，模拟大文件传输的场景。默认使用单流（single stream），测试结果反映的是单条TCP连接所能达到的最大吞吐量。但很多时候，单流的性能受到TCP拥塞控制和接收窗口大小的限制，远远达不到链路带宽上限。这时候就需要用多流（parallel streams）来压满链路：

# 服务端（在192.168.1.100上运行）
iperf3 -s

# 客户端：单流TCP测试，持续30秒，每2秒报告一次
iperf3 -c 192.168.1.100 -t 30 -i 2

# 客户端：4条并行流的TCP测试
iperf3 -c 192.168.1.100 -P 4 -t 30 -i 2

# 客户端：反向测试（服务器向客户端发送数据）
iperf3 -c 192.168.1.100 -R -t 30 -i 2

# 双向同时测试
iperf3 -c 192.168.1.100 --bidir -t 30 -i 2

# 设置TCP窗口大小为512KB
iperf3 -c 192.168.1.100 -w 512K -t 30

UDP模式：UDP模式不需要拥塞控制，可以以固定的速率发送指定大小的数据包。它特别适合模拟实时音视频流量（VoIP、RTP），以及用来测量实际链路的容量上限：

# 客户端：UDP模式，速率100Mbps，包长1400字节
iperf3 -c 192.168.1.100 -u -b 100M -l 1400 -t 30 -i 2

# 客户端：逐渐增加速率，找到链路瓶颈
iperf3 -c 192.168.1.100 -u -b 500M -l 1400 -t 10
iperf3 -c 192.168.1.100 -u -b 1G -l 1400 -t 10
iperf3 -c 192.168.1.100 -u -b 2G -l 1400 -t 10

# 服务端侧查看丢包报告
# (UDP模式下iperf3会报告服务端收到的包数、丢失的包数和抖动情况)
iperf3 -s
# [ ID] Interval       Transfer     Bandwidth       Jitter    Lost/Total
# [ 4] 0.00-10.00 sec  1.09 GBytes   940 Mbits/sec  0.023 ms  0/813951 (0%)

在UDP模式下，当发送速率超过链路容量时，服务端会报告大量的丢包（比如Total=1000000, Lost=250000，意味着25%的包丢失了）。这个"开始丢包的点"就是链路的实际极限容量——它可能远远低于接口的协商速率。

实用技巧：测试前一定要确认服务端和客户端的系统参数已经优化过。特别是TCP缓冲区大小——如果默认的socket buffer太小（通常Linux默认只有128KB），高带宽长距离链路的性能会严重受限。一个经验法则是：带宽延迟积（BDP）决定了所需的最小缓冲区大小。BDP = 带宽 × RTT。例如千兆链路、RTT=20ms，BDP = 1Gbps × 20ms = 20Mb ≈ 2.5MB。这意味着你需要将TCP窗口至少设置为2.5MB才能填满这条链路。可以用sysctl -w net.core.wmem_max=8388608来调整，配合iperf3的-w参数使用。

24.3 RFC 2544标准测试

如果你需要更正式的网络性能评估，那就绕不开RFC 2544。这个RFC由IETF在1999年发布，定义了网络互连设备性能测试的标准化方法。虽然已经发布了快三十年，但它仍然是网络设备测试领域的黄金标准。RFC 2544定义了四个核心测试项目：

吞吐量测试（Throughput）：以不同的发送速率（从高到低逐渐降低）发送固定数量的包，找到"零丢包"的最大速率。这个值才是网络设备真正的"额定吞吐量"。很多设备标称"千兆交换机"，但在64字节小包场景下吞吐量可能只有线速的百分之七十——因为小包转发需要更多的CPU/ASIC处理能力。RFC 2544要求分别测试64、128、256、512、1024、1280、1518字节共7种帧长。

时延测试（Latency）：在特定负载下（通常是吞吐量的70%）发送带时间戳的测试包，测量包从源到目的的传播时间。RFC 2544区分"存储时延"和"转发时延"，前者是设备收到完整帧到开始转发的时间，后者是设备决策到数据出现在输出端口的时间。

丢帧率测试（Frame Loss Rate）：在不同负载下，测量设备丢弃的帧数占总发送帧数的百分比。这个结果通常画成一条"负载-丢帧率"曲线，可以看到设备在哪个负载点开始"撑不住"。

背对背测试（Back-to-Back）：以线速发送连续的最短帧（通常是64字节），测量设备在不丢包的情况下能承受的最大连续帧数量。这个测试考察的是设备缓冲区的大小和处理能力——缓冲区太小的设备在突发流量面前会大量丢包。

实施RFC 2544的注意事项：在实际测试中，必须确保测试仪本身不是瓶颈。如果测试主机是一台普通的x86服务器，你可能需要以下优化：1) 关闭中断合并（ethtool -C eth0 rx-usecs 0 tx-usecs 0）；2) 将测试进程绑定到特定的CPU核心（taskset）；3) 使用DPDK或AF_XDP绕过内核协议栈以获得更精准的微秒级时间戳。如果这些条件都不具备，至少也要确认测试主机的性能上限远高于被测链路的标称速率——否则你测出来的不是网络的性能，而是自己主机的性能上限。

24.4 测试中的常见陷阱

做网络性能测试看起来简单（"不就是跑个iperf3吗？"），但实际上有很多隐蔽的陷阱。我见过无数工程师因为没注意这些细节得出了错误的结论：

陷阱一：单流测试的误导。很多人跑一次iperf3 -c server看到结果是350Mbps，就得出结论"千兆链路只有350Mbps"。但实际上，单条TCP流的性能受到很多因素限制：TCP窗口大小、延迟、CPU单核处理能力。正确的做法是先跑多流测试（-P 4甚至-P 8），如果多流能跑满接近千兆但单流不能，那问题出在TCP参数上（窗口大小、拥塞控制算法），而不是网络链路本身。

陷阱二：忘记测试反向。链路的性能通常是非对称的。上游拥挤、下游空闲的情况很常见。比如某个场景中，从A到B能达到900Mbps，但从B到A只有200Mbps——原因可能是B的出口交换机做了端口限速、或者B的上行链路过载。所以每次测试都要做正反两个方向。

陷阱三：忽略CPU瓶颈。iperf3在高带宽测试下会消耗大量CPU资源。如果在10G测试中CPU占用率达到100%，那么测出来的结果反映的是CPU能力而非网络能力。使用-Z（零拷贝）和绑定CPU核心可以缓解这个问题。在40G/100G测试中，通常需要使用多队列网卡配合RSS（Receive Side Scaling），让多个CPU核心分担报文处理。

陷阱四：中断合并（coalescing）导致的假象。现代网卡默认启用中断合并——将多个小包的接收中断合并为一个中断，以减少CPU开销。这个机制在正常工作负载下非常有效，但在性能测试中会引入额外的毫秒级延迟，让你以为网络延迟很大。测试时可以用ethtool -C eth0 rx-usecs 0 tx-usecs 0关闭中断合并。

24.5 一个真实的"踩坑"故事

这个故事发生在我参与的一个数据中心迁移项目中。新数据中心布好线缆后，机房团队拿着Fluke测试仪通过了所有链路的"通断测试"——物理层没有问题。然后我们上服务器做"千兆验收测试"。

第一次测试：两台服务器直连新交换机，跑iperf3单流TCP，结果只有350Mbps。"千兆怎么可能才350Mbps？"项目组炸锅了。有人怀疑网线质量不行，有人怀疑交换机配置有问题，还有人怀疑服务器网卡坏了。

我让团队先换个测试方法——用-P 4跑四流并行，结果勉强到了850Mbps。这排除了物理链路的问题（四流能到850说明链路本身容量没问题），也排除了交换机硬件故障。问题出在单流性能上。但为什么同样是千兆，旧机房的交换机跑单流就能到930Mbps呢？

我检查了两台交换机的配置对比。在接口配置中，发现新交换机上多了一行配置：storm-control broadcast level 10，这是广播风暴控制。但这个配置只会影响广播包，不应该破坏TCP吞吐啊。继续深挖，注意到交换机的配置中还有一句：qos queue-group egress-queue-limit 100——新交换机的出端口队列缓冲区被手动配置成了只有100个包！

标准的千兆交换机默认出端口缓冲区通常在几百KB到数MB之间。改成100个包后，以1518字节MTU计算，缓冲区只能容纳约150KB的突发数据。当iperf3的TCP发送窗口增速超过交换机缓冲区时，队列瞬间填满，触发丢包，TCP拥塞控制立刻砍半窗口——这正是"单流只能跑350Mbps"的根本原因。多流测试因为每个流的窗口独立增长，互相错开爆发高峰，所以总吞吐量要好一些。但本质上，交换机的缓冲区配置错误导致了严重的性能瓶颈。

找到原因后，机房团队查看了一圈，发现是部署新交换机时，一位工程师为了防止广播泛洪，参考了某个网上教程随意修改了队列深度参数。恢复默认缓冲区配置后，单流iperf3测试立即从350Mbps跳到了940Mbps。这个故事的教训是：千万不要随意修改交换机的队列缓冲区参数，尤其是在你不太清楚它在干什么的情况下。而性能测试的价值恰恰在于——它能暴露那些"看起来没问题但实际有问题"的配置。

网络性能测试是网络工程的基石。无论是验收新设备、排查性能瓶颈、还是验证配置变更的效果，一套标准化的测试流程能帮你节省大量时间。记住这句广告词般的话：没有数据就没有真相，没有基准就无法优化。
