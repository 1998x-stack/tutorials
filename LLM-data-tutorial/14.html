为什么混合重要

想象一下，你要培养一个"通才"学生。你给他看的书应该有哪些？全是数学教材？他数学会很好，但语文一塌糊涂。全是小说？语言表达能力一流，但逻辑推理是短板。全是百科全书？知识面很广，但深度不够。

训练 LLM 也是同样的道理。单一来源的数据无论质量多高，都无法覆盖模型需要的全部能力维度。Web 数据覆盖面广但偏杂，代码逻辑性强但缺乏常识，书籍内容有深度但时效性差。关键是把这些来源按合适的比例融合在一起，让模型在各维度上均衡发展。

The Pile（EleutherAI, 2020）是第一个系统研究这个问题的工作。在此之前，大多数模型只用 Wikipedia + Books 或者单一 Web 抓取。The Pile 首次构建了包含 22 个数据源的大规模语料，并逐一分析了每个数据源对下游任务的具体贡献。他们发现，不同数据源的"能力贡献曲线"差异巨大——某些数据源对常识推理帮助大，某些对代码生成帮助大，某些几乎在所有任务上都没什么用。

The Pile 的多源方法论

The Pile 的 22 个数据源涵盖了 PubMed 论文摘要、ArXiv 论文、GitHub 代码、Project Gutenberg 书籍、Wikipedia、StackExchange 问答、YouTube 字幕、USENET 新闻组等等。每个数据源都经过独立的清洗流程，然后做全局去重（跨语料去重，第 9 章讲过），最后按特定比例混合。

The Pile 的贡献不仅仅在于"混合了多个数据源"，更在于他们做了系统的消融实验——在保持总 token 数不变的前提下，分别去掉某一个数据源，观察模型在哪些任务上表现下降。通过这种"源归因"分析，他们得出了一些重要结论：

数据源混合占比贡献的能力领域移除后影响最大的任务
PubMed Central17.2%生物医学知识、长文本理解BIOMED 系列任务下降 8-12%
ArXiv13.0%数学推理、科学论文理解MathQA、SciQ 下降 5-10%
GitHub10.4%代码生成、结构化内容代码补全任务下降 15-20%
Books311.5%长文本连贯性、叙事理解故事生成、长距离推理下降
Wikipedia4.6%常识知识、事实准确性MMLU、NQ 下降 3-5%
StackExchange5.1%逻辑问答、技术知识推理类任务下降 4-7%
其他 16 个源38.2%多样性、特定领域知识各任务略有下降

2026 年的典型数据配比

从 The Pile（2020）到 2026 年，数据配比的格局发生了显著变化。最大的变化是合成数据的崛起。2024 年合成数据在训练语料中还只占不到 2%，到 2026 年这个比例已经涨到了 5-15%，且仍在快速增长。

Web 数据（过滤后）70-80%65-75%60-70%缓慢下降
代码数据8-12%10-15%10-15%稳定
学术论文3-5%5-8%5-10%略有增加
书籍数据5-10%5-8%5-8%稳定
Wikipedia3-5%3-5%3-5%稳定
合成数据<1%2-5%5-15%快速增长
合成数据的崛起： 合成数据占比从 2024 年的 2% 增长到 2026 年的 15%，这是过去两年数据工程领域最显著的变化趋势。背后的驱动因素包括：1) 真实高质量 Web 数据增长放缓（互联网的新增内容质量在下降）；2) LLM 生成能力大幅提升（GPT-4o、Claude-4 等模型生成的文本质量已接近人类）；3) 合成数据可以"定向生产"——想要更多数学推理数据？让模型生成就是了。但这也带来了语料单一化的隐忧——我们前面在 LLM 标注的偏差中已经提到过。

当然，这些数字只是行业平均值。不同团队有不同的哲学：HuggingFace 的 FineWeb-Edu 偏重 Web 数据进行精细清洗；Codestral 和 DeepSeek-Coder 等代码模型大幅提高代码数据的比例（可达 30-40%）；BloombergGPT 在金融领域训练时几乎全部使用金融文本。

上采样 vs 降采样策略

多源混合的核心工程挑战是：每种数据源在原始语料中的数量和它的质量密度往往不匹配。Wikipedia 只有 40 亿 token，但质量极高；Common Crawl 有数万亿 token，但大部分是噪音。

解决这个不匹配的策略是上采样（upsampling）高质量源和降采样（downsampling）低质量源。具体来说：

  • 上采样：对 Wikipedia、教科书、优秀代码库等高质量来源，在训练时让它们的数据被"重复使用"更多次。比如 The Pile 的 Wikipedia weight=3，意味着每个 Wikipedia token 在训练中出现的频率是普通 Web 数据的 3 倍。
  • 降采样：对低质量 Web 数据，随机丢弃部分内容以减少训练中的占比。FineWeb-Edu 的教育质量分类器本质上就是一个降采样策略——把 91.3% 的低分文档直接丢弃。

更高级的策略是课程学习（Curriculum Learning）式的动态配比：在训练的不同阶段使用不同的数据配比。早期的 LLM 训练通常从大量 Web 数据开始（建立广泛的语言知识），中期逐步增加代码和推理数据（提升逻辑能力），后期使用高质量精挑细选的数据（收尾打磨）。不过这种策略工程复杂度很高，目前只有少数大厂在实践。

说到这里，我想起了 RefinedWeb 的一个有趣发现。TII 团队做了一个"纯粹"的实验：只使用经过精细清洗的 Web 数据（完全不用书籍、论文等其他来源），结果发现清洗充分的 Web 数据可以达到甚至超越多源混合的效果。他们的结论是：Web 数据本身已经包含了几乎所有类型的知识——论文、代码、书籍在 Web 上都有镜像——关键在于如何把 Web 数据中的"精华"提取出来。

RefinedWeb 的关键洞察： 精细清洗比多源混合更重要。如果你能把 Web 数据清洗到极致——去重、过滤、质量打分做到位——那你可能不需要费心去收集和混合多源数据。当然，这并不意味着多源混合没有价值；对于特定领域能力（如代码生成）来说，针对性获取该领域的高质量数据仍然是最有效的策略。两种方法各有侧重，可以互补。

数据配比是理论层面的顶层设计。配比方案定了，下一步就是在实际的大规模分布式环境中实现这些处理。单机处理数十 TB 数据显然不现实——需要分布式计算框架、高效的数据存储、以及可靠的任务调度。从下一章开始，我们进入第四部分：系统架构与工程实战。
