第二十章：GraphRAG与知识图谱增强

大家好，我是你们的讲师。前面的章节中我们反复提到一个概念：向量检索擅长找"相似"的内容，但"相似"不等于"相关"。想象你在一个公司内部知识库中提问："去年从市场部离职去了竞品公司的员工有哪些？" 传统的向量检索会找到离职相关的文档、市场部相关的文档和竞品公司相关的文档——但它无法理解这三者之间的逻辑关系。它不知道需要取"市场部"和"离职"和"去了竞品公司"三个条件的交集。这就是向量检索的盲点：它没有真正的"关系"理解能力。

让我讲一个真实的项目教训。一个大型制药企业使用向量RAG来加速新药研发的知识检索。研究团队问了一个非常简单的问题："哪些化合物同时抑制了EGFR和VEGFR靶点，并且已进入二期临床？" 向量检索返回了大量关于EGFR抑制剂、VEGFR抑制剂和二期临床的文档，但没有任何一篇文章同时包含这三个条件。团队花了几个小时手动翻阅了40多篇检索到的论文才找到了答案。如果系统背后有一个知识图谱，这个查询只需一次Cypher查询即可在几十毫秒内得到精确答案。这个教训直接促成该企业启动了GraphRAG项目。

10.1 向量检索的盲点

向量检索在以下场景中表现不佳：

全局性问题： "总结公司过去三年的研发投入趋势"——这需要聚合和分析多个文档的数据，而非找一篇相似的文章。

多跳推理： "A部门和B部门合作的项目的负责人是谁？"——需要从A部门的文档找到项目X，再从项目X的文档找到负责人，涉及两次跨文档的信息链接。

精确匹配： "2025年Q3的营收是多少？"——向量检索返回的可能是"2025年营收表现"或"近几个季度的财务数据"这类相似的文档，而不是精确的数字。

关系理解： "哪些人同时参与了项目Alpha和项目Beta？"——向量检索找不到"同时参与"这个关系。

知识图谱正是为了解决这些盲点而引入RAG的。

10.2 知识图谱构建：实体与关系提取

将非结构化文档转换为知识图谱的核心步骤是从文本中提取实体和关系。在2026年，这一步骤主要依赖LLM来完成——使用精心设计的prompt从文档中提取出结构化的三元组（实体-关系-实体）。

典型的知识图谱构建流程如下：

  1. 文档分块： 和传统RAG一样，将文档分割为可处理的块。
  2. 实体提取： 对每个块使用LLM提取命名实体——人、组织、地点、项目、产品、日期等。
  3. 关系提取： 识别实体之间的语义关系（隶属于、参与、位于、导致等）。
  4. 共指消解： 将不同文档中指向同一实体的名称统一（如"苹果公司"和"Apple Inc."应合并）。
  5. 图谱存储： 将实体和关系存入图数据库中（如Neo4j、Amazon Neptune）。

以下是一个使用Neo4j查询知识图谱中实体关系的示例：

// Cypher查询：查找同时参与了多个项目的员工
// 并展示他们与项目、部门之间的关系

MATCH (p:Project)<-[:PARTICIPATES_IN]-(e:Employee)-[:PARTICIPATES_IN]->(p2:Project)
MATCH (e)-[:BELONGS_TO]->(d:Department)
WHERE p.name <> p2.name  // 确保是不同项目
RETURN e.name AS 员工姓名,
       d.name AS 所属部门,
       collect(DISTINCT p.name) AS 参与项目列表,
       count(DISTINCT p) + count(DISTINCT p2) AS 项目数量
ORDER BY 项目数量 DESC
LIMIT 10

// 查询结果示例：
// ╒══════════╤══════════╤══════════════════════╤══════════╕
// │员工姓名   │所属部门   │参与项目列表          │项目数量  │
// ╞══════════╪══════════╪══════════════════════╪══════════╡
// │"张明"    │"研发部"  │["Alpha","Beta"]     │2         │
// │"李芳"    │"产品部"  │["Alpha","Gamma"]     │2         │
// └──────────┴──────────┴──────────────────────┴──────────┘

// 另一种常见查询：查找最短路径
// 例如，两个看似不相关的实体之间有何关联？
MATCH path = shortestPath(
    (a:Person {name: "Alice"})-[:*..4]-(b:Person {name: "David"})
)
RETURN path

10.3 Microsoft GraphRAG与层级社区

2024年微软研究院发布的GraphRAG论文提出了一种革命性的方法，从根本上解决了传统RAG在处理全局性问题时的不足。GraphRAG不是简单地用知识图谱替代向量检索，而是构建了一个层级化的知识结构：

层级1——实体和关系： 最底层，从原始文档中提取的原子事实，存储在知识图谱中。

层级2——社区检测： 使用图算法（如Leiden算法）将紧密关联的实体聚合成社区。例如，所有与"2025年营销计划"相关的实体可能形成一个社区。

层级3——社区摘要： 对每个社区使用LLM生成自然语言摘要，描述该社区的整体知识。

层级4——全局索引： 将所有社区的摘要组织成一个可全局搜索的结构。

这种层级结构的效果是：当用户问一个全局性问题时（如"公司的主要风险有哪些？"），系统不需要遍历所有文档，而是直接检索高层社区的摘要，然后根据摘要追踪到具体的实体和关系作为证据。

10.4 GraphRAG + VectorRAG：不是替代，是互补

一个常见的误解是GraphRAG要取代传统的向量RAG。实际上，两者是互补关系：

对比维度 VectorRAG（向量RAG） GraphRAG（图谱RAG）
检索机制 语义相似度匹配 图结构遍历+关系匹配
最适合的查询 开放式、探索性问题 精确、结构化、多跳问题
全局性问题 差 极好（通过社区摘要）
多跳推理 差 极好（通过图遍历）
相似内容查找 极好 差
构建成本 低（一次嵌入） 高（多步LLM处理）
更新成本 低（增量嵌入） 高（可能需要重构图谱）
推荐策略 日常查询首选 复杂逻辑查询使用

在工业实践中，大多数团队采用"混合路由"策略：先判断查询类型，简单查询走向量检索，复杂逻辑查询走图谱检索，两者结果融合后交给生成模型。这种混合架构通常在整体准确率上比单一方式高出15-30%。

⚠ 成本警示：GraphRAG的索引构建成本不容低估 GraphRAG的强大能力伴随着高昂的索引构建成本。根据2025年多家企业的实践报告，对于一个100万文档的知识库，使用GPT-4级别的模型进行实体提取、关系识别和社区摘要，单次全量构建的LLM调用成本可能在5000-20000美元之间。如果你的知识库频繁更新（每天改动超过5%的内容），维护成本会更高。在决定是否采用GraphRAG之前，建议先进行成本收益分析：如果用户查询中超过30%是多跳推理或全局性问题，GraphRAG是值得的投资；如果90%以上的查询是简单的单文档问答，优先优化向量RAG是更经济的选择。

10.5 TechGraphRAG的13步管线

在GraphRAG的基础上，微软和社区进一步推出了TechGraphRAG（MEGAVARSHINI等，2025），将知识图谱构建和检索过程标准化为13个可组合的步骤：文档解析→分块→实体提取→关系提取→实体解析→图谱构建→社区检测→社区摘要→摘要索引→查询分类→图谱检索→摘要检索→生成融合。每个步骤都可以根据具体需求进行定制和替换。

TechGraphRAG的关键创新在于引入了"查询分类器"步骤——在检索之前先判断查询类型，然后根据类型动态选择检索策略。这解决了"一刀切"的问题：一个查询分类器可以区分出事实型问题、推理型问题、总结型问题和探索型问题，分别路由到最合适的检索通道。

10.6 小结与下章预告

GraphRAG通过引入知识图谱的结构化知识表示，弥补了向量RAG在关系理解和多跳推理方面的先天不足。我们学习了知识图谱的构建流程、Neo4j的Cypher查询语言、微软GraphRAG的层级社区方法，以及如何将GraphRAG和VectorRAG组合使用。虽然GraphRAG的构建成本较高，但对于需要深度推理、全局分析和精确查询的场景，它的价值无可替代。请记住：GraphRAG和VectorRAG不是二选一，而是根据查询类型灵活切换的双引擎架构。完成了知识图谱的增强之后，下一章我们将进入RAG的另一个前沿方向——Agentic RAG。当RAG不再是被动地"查了再答"，而是主动地规划、执行、反思和迭代时，系统将展现出前所未有的智能水平。
