开篇：当 REST 不够快——gRPC 的诞生

几年前我参与了一个广告推荐系统，后端由十几个微服务组成。服务之间的通信全部使用 RESTful JSON API。广告请求需要在 50ms 内返回结果，但每次请求都要经过 5-6 个服务链式调用。每个服务之间的 JSON 序列化/反序列化、HTTP 连接建立、请求头的开销积累起来，光通信就占了 30ms。留给业务逻辑的时间只剩下 20ms——完全不够用。

后来我们把服务间通信全部切换到了 gRPC。同样的链式调用，通信时间从 30ms 降到了 8ms。性能的提升并非来自某一种魔法——而是来自 Protobuf 的紧凑二进制编码、HTTP/2 的多路复用、以及强类型接口带来的代码生成。

gRPC 是 Google 在 2015 年开源的 RPC 框架，基于 HTTP/2 协议和 Protobuf 序列化。它不是要取代 REST——事实上，gRPC 和 REST 各有适用的领域。但如果你需要服务之间的高性能通信，gRPC 是目前最好的选择之一。

🔑 一句话总结：REST 是"给外部用的"（浏览器、移动端、第三方），gRPC 是"给自己人用的"（微服务间内部通信）。选择工具不是看谁更"先进"，而是看谁更适合场景。

Protobuf 序列化原理：比 JSON 快 3-10 倍的秘密

Protobuf（Protocol Buffers）是 gRPC 的默认序列化格式。它比 JSON 快 3-10 倍、体积小 3-5 倍，这个性能差距来自几个关键设计：

  • 二进制编码：JSON 是文本格式，每个字段名都需要重复传输（比如 "user_id": 12345 的字段名就占了 10 个字节）。Protobuf 用字段编号替代字段名，在传输中只传输编号和值。
  • Varint 编码：小整数用更少的字节表示。数值 1 在 JSON 中是一个字节的 1（加上分隔符上下文更大），在 Protobuf 中也是 1 字节。但数值 300 在 Protobuf 中用 2 字节，在 JSON 中是 3 个字符。
  • 强类型：不需要像 JSON 那样用额外的类型标记，因为 Schema 在编译时就确定了。
  • 无冗余结构：没有 JSON 的 {} 括弧、逗号分隔符、冒号等语法元素。
// protobuf 定义文件：order.proto
syntax = "proto3";

package order_service;

option go_package = "./proto;proto";

// 订单服务定义
service OrderService {
    rpc GetOrder(GetOrderRequest) returns (Order);
    rpc ListOrders(ListOrdersRequest) returns (ListOrdersResponse);
    rpc CreateOrder(CreateOrderRequest) returns (Order);
    rpc StreamOrderUpdates(StreamRequest) returns (stream Order);  // 服务端流
}

message GetOrderRequest {
    string order_id = 1;
}

message Order {
    string order_id   = 1;
    int64  user_id    = 2;
    double amount     = 3;
    int32  status     = 4;  // 1=pending, 2=paid, 3=shipped
    string created_at = 5;
    repeated OrderItem items = 6;  // 嵌套消息 + 数组
}

message OrderItem {
    string product_id = 1;
    string name       = 2;
    int32  quantity   = 3;
    double price      = 4;
}

message ListOrdersRequest {
    int32  page    = 1;
    int32  size    = 2;
    int32  status  = 3;  // 可选过滤条件
}

message ListOrdersResponse {
    repeated Order orders = 1;
    int32  total_count    = 2;
}

message CreateOrderRequest {
    int64  user_id       = 1;
    repeated OrderItem items = 2;
}

message StreamRequest {
    string user_id = 1;
}
💡 开发体验的提升：定义好 .proto 文件后，运行 protoc 命令即可自动生成客户端和服务端的代码，包括序列化、反序列化、网络通信的全部逻辑。开发者只需实现业务接口——比手写 REST 接口的开发效率更高，而且不会出现字段拼写错误这种低级 bug。

gRPC 四种调用模式

gRPC 支持四种通信模式，从简单到复杂覆盖了不同的交互需求。

模式 描述 适用场景 类比 REST
Unary（一元） 客户端发送一个请求，服务端返回一个响应 常规 CRUD，查询/创建 GET / POST
Server Streaming 客户端发送一个请求，服务端返回多个响应流 推送通知、订阅事件、日志流 WebSocket / SSE
Client Streaming 客户端发送多个请求流，服务端返回一个响应 批量上传、大文件分段提交 Chunked upload
Bidirectional Streaming 双方独立发送多个消息，全双工通信 实时聊天、协同编辑、AI 对话 WebSocket 
// 四种调用的 Go 代码示意

// 1. Unary：最简单
resp, err := client.GetOrder(ctx, &pb.GetOrderRequest{OrderId: "ORD123"})

// 2. Server Streaming：服务端持续推送
stream, _ := client.StreamOrderUpdates(ctx, &pb.StreamRequest{UserId: 1001})
for {
    order, err := stream.Recv()
    if err == io.EOF { break }
    fmt.Println("收到更新:", order.Status)
}

// 3. Client Streaming：客户端批量上传
uploadStream, _ := client.BatchUpload(ctx)
for _, item := range items {
    uploadStream.Send(&pb.UploadRequest{Data: item})
}
summary, _ := uploadStream.CloseAndRecv()

// 4. Bidirectional Streaming：双方同时收发
chatStream, _ := client.Chat(ctx)
go func() {
    for { chatStream.Send(chatMsg) }  // 持续发送
}()
for {
    reply, _ := chatStream.Recv()     // 持续接收
    fmt.Println(reply.Text)
}

gRPC vs REST 性能对比

很多人关心 gRPC 到底比 REST 快多少。但这个问题的答案取决于你的场景。下面是一组典型的基准测试数据（服务间通信、相同硬件、相同网络环境）：

维度 REST (JSON over HTTP/1.1) gRPC (Protobuf over HTTP/2)
单次请求延迟（P50） ~3ms ~0.8ms
单次请求延迟（P99） ~15ms ~3ms
消息体大小（100 个对象） ~12KB ~3KB
吞吐量（单连接） ~5,000 req/s ~40,000 req/s
序列化耗时（1KB 数据） ~3μs ~0.5μs
浏览器原生支持 是 需 gRPC-Web 代理

但注意：这些数据只代表服务间通信的对比。如果客户端是浏览器或移动端，REST 仍然是最友好的选择。如果内部服务之间需要高吞吐低延迟的通信，gRPC 的优势是压倒性的。

📌 选型决策树：
• 对外 API（浏览器/移动端）→ REST / GraphQL
• 服务间内部通信 → gRPC
• 需要双向流或实时推送 → gRPC Streaming
• 需要支持多种语言客户端 → gRPC（自动生成多语言代码）
• 对性能不敏感的内部调用 → 用你最熟悉的即可

服务发现与负载均衡

使用 gRPC 进行微服务通信时，一个核心基础设施问题是：客户端怎么知道调用哪个服务实例的哪个 IP 和端口？这就是服务发现要解决的问题。

DNS 方式：最简单——通过 DNS 解析服务名到多个 IP，客户端自带简单的轮询负载均衡。适合小规模部署。

Client-side 服务发现：客户端通过服务注册中心（Consul、etcd、Eureka）获取可用实例列表，自己负责负载均衡。这种方式减少了网络跳数，但需要客户端集成服务发现逻辑。gRPC 原生支持 pick_first 和 round_robin 两种负载均衡策略。

Proxy-side 服务发现：使用 Sidecar Proxy（如 Envoy、Linkerd），客户端将请求发送给本地的 Sidecar Proxy，由 Proxy 负责服务发现和负载均衡。这是 Service Mesh 的标准模式，在 Kubernetes 生态中最为流行。gRPC 与 Envoy 集成良好，Envoy 支持 gRPC 的 HTTP/2 长连接、重试、超时等高级特性。

2026 年的推荐方案：在 K8s 上使用 gRPC + Envoy Sidecar。如果还没上 Service Mesh，K8s 的 DNS 轮询 + round_robin 策略对大多数团队已经够用。

实战：用 gRPC 构建一个高性能内部服务

// 服务端实现（Go）
type orderServer struct {
    pb.UnimplementedOrderServiceServer
}

func (s *orderServer) GetOrder(ctx context.Context, req *pb.GetOrderRequest) (*pb.Order, error) {
    // 从数据库查询订单
    order, err := db.FindOrder(req.OrderId)
    if err != nil {
        return nil, status.Errorf(codes.NotFound, "订单 %s 不存在", req.OrderId)
    }
    return toProtoOrder(order), nil
}

func (s *orderServer) StreamOrderUpdates(req *pb.StreamRequest, stream pb.OrderService_StreamOrderUpdatesServer) error {
    // 从消息队列订阅订单变更事件
    ch := subscribeOrderEvents(req.UserId)
    for event := range ch {
        order := fetchOrder(event.OrderId)
        if err := stream.Send(toProtoOrder(order)); err != nil {
            return err
        }
    }
    return nil
}

func main() {
    lis, _ := net.Listen("tcp", ":50051")
    s := grpc.NewServer(
        grpc.UnaryInterceptor(otelInterceptor),        // 可观测性拦截器
        grpc.ChainStreamInterceptor(authStreamInterceptor),
    )
    pb.RegisterOrderServiceServer(s, &orderServer{})

    // gRPC 使用 HTTP/2，TLS 可选（内部网络可省略）
    reflection.Register(s)  // 启用 gRPC reflection 方便调试
    s.Serve(lis)
}

// 客户端调用
conn, _ := grpc.Dial("order-service:50051",
    grpc.WithInsecure(),
    grpc.WithDefaultServiceConfig(`{"loadBalancingConfig": [{"round_robin": {}}]}`),
)
defer conn.Close()

client := pb.NewOrderServiceClient(conn)
order, _ := client.GetOrder(ctx, &pb.GetOrderRequest{OrderId: "ORD123"})
⚠️ 踩坑故事：第一次上线 gRPC 服务时，我们忘了配置 Keepalive。结果一个实例宕机了，负载均衡器还在继续往它转发请求，因为 HTTP/2 的长连接一直没断开。客户端收到了大量的 UNAVAILABLE 错误，花了很长时间才触发连接重连。教训：gRPC 必须配置合理的 Keepalive 和 Health Check——客户端的 keepaliveTime 设为 10 秒，服务端暴露 Health 服务供负载均衡器探测。

gRPC 让服务间通信变得高效、可维护、类型安全。但它解决的是"后端和后端"之间的问题。当一个请求到达前端/移动端时，你面对的是完全不同的挑战——不同的客户端可能需要不同的数据格式、不同的字段组合、不同的查询深度。这就是我们下一章的主题——GraphQL 与 BFF 模式，看它们如何让前后端之间的数据交换更加灵活和高效。
