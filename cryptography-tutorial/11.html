第11章 数字签名与认证

同学们，想象一下这个场景：你收到一封来自老板的邮件，写着"立即给这个账户转账10万元"。你如何确认这封邮件确实是老板发的？如果老板事后说"我没发过这条消息"，你怎么证明他确实发了？

在现实世界中，我们有手写签名和印章来解决这些问题。但在数字世界中，签名文件只是一串比特——复制一份和原件一模一样。你需要一种方式让签名可以验证、不可伪造、不可抵赖。这就是数字签名（Digital Signature）的任务。

🔑 数字签名解决的三个问题：（1）真实性（Authentication）——消息确实来自声称的发送者；（2）完整性（Integrity）——消息在传输中没有被篡改；（3）不可否认性（Non-repudiation）——发送者不能否认自己签署过该消息。这三个目标合在一起，构成了信任的数字基础。

11.1 数字签名的工作原理

数字签名的基本逻辑其实很简单：

  • 签名者使用私钥对消息进行签名，生成一个数字签名值
  • 验证者使用签名者的公钥来验证签名是否匹配
  • 只有持有私钥的人才能生成有效的签名
  • 签名与消息内容绑定——消息的任何改变都会使签名无效

实际签名过程并不是直接对消息本身签名——那太慢了（公钥操作很昂贵）。通常的流程是：先对消息计算哈希值，然后对哈希值签名。这样无论消息多大，签名都只涉及一个固定长度的哈希值。

签名流程：
1. h = HASH(message)            —— 先哈希
2. sig = SIGN(private_key, h)    —— 再签名
3. 输出 (message, sig)

验证流程：
1. h = HASH(message)            —— 同样计算哈希
2. result = VERIFY(public_key, h, sig)  —— 验证签名
3. 如果 result == True，签名有效；否则无效

11.2 签名 vs 加密：先签后加密还是先加密后签名？

这是一个经典的设计问题。假设Alice要给Bob发送一条经过认证（签名）和保密（加密）的消息。有两个顺序选择：

方案 操作 风险
先签名后加密 Alice: sig = Sign(skA, m), c = Enc(pkB, m||sig)
Bob: m||sig = Dec(skB, c), Verify(pkA, sig, m)		 正确方式。Bob知道谁发的，且消息保密
先加密后签名 Alice: c = Enc(pkB, m), sig = Sign(skA, c)
Bob: Verify(pkA, sig, c), m = Dec(skB, c)		 ⚠️ 风险：Bob解密密文后得到m，但没有任何签名绑定m。更糟的是，第三方可以"剥离"签名后用自己签名替换

为什么先加密后签名有问题？假设Alice加密消息给Bob，并签名了密文。Bob收到后验证签名、解密，得到消息。但Alice也可以把这个签名过的密文发送给Charlie——Charlie知道这是Alice加密的，但他打不开（因为是用Bob的公钥加密的）。更危险的是，如果Bob把解密后的原始消息转发给Charlie，Charlie无法验证这条消息确实来自Alice——因为签名附着在密文上，而不是明文上。

标准做法是：先签名、再加密（Sign-then-Encrypt）。这样接收者先解密、再验证签名，确保消息的来源和完整性都可验证。

11.3 主流签名算法

目前业界有多种数字签名算法，各有千秋：

算法 基础 签名大小 特点
RSA-PSS 大整数分解 与密钥等长（通常256-512字节） 广泛兼容，填充方案安全
DSA 离散对数 约40-64字节 传统标准，已被ECC取代
ECDSA 椭圆曲线 约64-72字节 比RSA短，性能好，但需要好随机数
EdDSA (Ed25519) 扭曲爱德华曲线 64字节 确定性签名，快速，实现简单
SM2 椭圆曲线（国密） 约64字节 中国国家标准

RSA-PSS（Probabilistic Signature Scheme）是RSA签名的推荐方案，替代了旧的PKCS#1 v1.5签名。与OAEP类似，PSS在签名中加入了随机化元素，提供了严格的安全性证明。

EdDSA（特别是Ed25519）近年来获得了大量关注。它是确定性的——不需要可靠的随机数源，消除了一个主要的安全隐患。它的曲线参数经过精心设计，实现了"防弹"级别的常数时间实现，抵抗侧信道攻击。目前SSH、OpenPGP、TLS 1.3都支持Ed25519。

11.4 消息认证码（MAC）

数字签名使用非对称密码，但如果我们只是想在共享密钥的双方之间验证消息的完整性，用对称密码就能实现——这就是消息认证码（Message Authentication Code, MAC）。

MAC与数字签名的区别：

  • MAC：双方共享同一个密钥，发送方用密钥生成MAC标签，接收方用相同密钥验证。所以MAC可以证明消息来自持有密钥的某个人，但不能证明是哪一个人——因为双方都有密钥。
  • 数字签名：私钥只有签名者一人持有。任何人都可以用公钥验证。这提供了不可否认性——只有私钥持有者才能生成该签名。

最常见的MAC构造是HMAC（Hash-based MAC）：

HMAC(K, m) = H( (K' ⊕ opad) || H( (K' ⊕ ipad) || m ) )

其中：
K' 是经过长度调整的密钥
opad 是外部填充码（0x5c重复）
ipad 是内部填充码（0x36重复）
|| 表示拼接

HMAC的设计很巧妙：即使底层的哈希函数（如MD5）被发现有碰撞攻击，HMAC的安全性仍然保持，因为密钥使得攻击者无法预计算碰撞对。当然，实际应用中还是推荐使用SHA-256等强哈希函数来构建HMAC。

另一种常见的MAC是CMAC（基于分组密码的MAC），它使用AES等分组密码来生成认证码，在需要硬件加速的场景中很常用。

💡 实际选择建议：需要通过公钥进行身份验证或需要不可否认性 → 使用数字签名（Ed25519或ECDSA）。只是在两个服务之间认证消息（共享密钥已通过其他方式分发）→ 使用HMAC-SHA256。HMAC比签名快一到两个数量级。

11.5 踩坑故事：索尼PS3的ECDSA签名惨案（再探）

在上一章ECC中我们简要提到了索尼PS3的ECDSA漏洞，现在让我们深入这个经典案例的技术细节——因为它完美展示了签名算法实现中"一个看似微小的错误"如何导致整个系统的崩溃。

索尼PS3使用ECDSA签名来验证游戏软件是否为正版。如果用户尝试运行未经索尼授权的软件，PS3会检查数字签名，如果签名无效则拒绝运行。这套系统的安全性完全依赖于索尼私钥的保密性。

2010年12月29日，黑客团队fail0verflow在混沌通信大会上宣布他们获得了索尼的ECDSA私钥。他们的发现过程是这样的：

在ECDSA中，签名 (r, s) 的计算方式为：
s = k⁻¹ × (H(m) + d × r) mod n

如果同一个 k 被用来签署两个不同的消息 m₁ 和 m₂：
s₁ = k⁻¹ × (H(m₁) + d × r) mod n
s₂ = k⁻¹ × (H(m₂) + d × r) mod n

两式相减：
s₁ - s₂ = k⁻¹ × (H(m₁) - H(m₂)) mod n
k = (H(m₁) - H(m₂)) / (s₁ - s₂) mod n

一旦 k 被恢复，私钥 d 就能从任一签名中计算：
d = (s × k - H(m)) / r mod n

索尼的ECDSA实现中，k被硬编码为固定值4。这意味着每个PS3游戏的签名都使用了相同的k值。黑客只需要两个不同的游戏（或两个不同的签名），用上面的公式就能轻松恢复出索尼的私钥。

⚠️ 关键教训：索尼的工程师当然知道"k值必须随机"这条规则——密码学教材上写得明明白白。但他们在实际实现中选择了一个"看起来无害"的捷径。这告诉我们：密码学安全不是只靠选择正确算法就能保证的。一个错误的实现可以把最安全的算法变成纸老虎。这也是为什么现代密码学越来越推崇"默认安全"的设计（如EdDSA的确定性签名）——不要让开发者有机会犯这种错误。

这次事件给整个行业的启示是深远的。如今，许多系统已经转向EdDSA，它通过确定性方式消除k值随机性问题。同时，形式化验证工具（如ProVerif、Tamarin）开始被用来形式化验证密码协议实现的正确性——从数学上保证实现没有漏洞。

掌握了数字签名的原理，下一章我们将进入一个更宏观的视角：PKI（公钥基础设施）与证书体系。数字签名解决了"这个签名是否有效"的问题，但PKI解决了更根本的问题——"这个公钥到底属于谁？"如果没有PKI，你手里的公钥可能是钓鱼网站发来的。让我们一起看看，全球信任网络是如何通过CA证书链建立起来的。
