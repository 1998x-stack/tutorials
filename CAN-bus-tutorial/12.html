第12章 CAN中断编程：中断优先级配置、中断服务函数编写、中断调试技巧

说到CAN通信，中断处理绝对是绕不开的核心话题。我见过不少新手，CAN配置得挺顺，但一上中断就翻车。说白了，中断就像汽车的刹车系统——平时用不上，关键时刻必须可靠。

这一章，我带你把CAN中断的优先级配置、服务函数编写、还有调试技巧，一次性讲透。

12.1 中断优先级配置——别让优先级坑了你

先问个问题：你的MCU里，CAN中断该排第几？

我个人习惯，把CAN中断放在中等偏高的优先级。为什么？因为CAN通信对实时性有要求，但也不是最高优先级——你总不能让CAN中断把系统滴答定时器给抢了吧？

以STM32为例，NVIC配置是这样的：

// 配置CAN接收中断优先级
NVIC_InitTypeDef NVIC_InitStructure;

NVIC_InitStructure.NVIC_IRQChannel = CAN1_RX0_IRQn;  // CAN1接收中断
NVIC_InitStructure.NVIC_IRQChannelPreemptionPriority = 2;  // 抢占优先级
NVIC_InitStructure.NVIC_IRQChannelSubPriority = 0;         // 子优先级
NVIC_InitStructure.NVIC_IRQChannelCmd = ENABLE;
NVIC_Init(&NVIC_InitStructure);

这里有个坑，我踩过。抢占优先级和子优先级，很多人搞混。抢占优先级决定能否打断正在执行的中断，子优先级决定同抢占优先级下的响应顺序。

⚠️ 我曾经犯过的错： 把CAN中断优先级设得和定时器中断一样，结果两个中断互相抢，系统直接卡死。记住：同优先级的中断不会互相打断，只能排队。

我建议你遵循这个原则：

  • 时间敏感的中断（如电机控制PWM）→ 最高优先级
  • CAN通信中断 → 中等优先级
  • 非实时任务（如按键扫描）→ 低优先级

12.2 中断服务函数编写——短小精悍是王道

中断服务函数（ISR）怎么写？四个字：短小精悍。

你想想看，中断服务函数执行时间越长，其他中断被阻塞的时间就越长。我在项目中遇到过，有人把CAN报文解析逻辑直接写在ISR里，结果CAN总线负载一高，系统直接崩了。

正确的做法：ISR里只做最必要的事，比如把数据搬到缓冲区，然后置一个标志位。真正的处理逻辑，放到主循环里做。

// 正确的CAN接收中断服务函数
void CAN1_RX0_IRQHandler(void)
{
    CanRxMsg RxMessage;
    
    // 1. 读取接收到的报文
    CAN_Receive(CAN1, CAN_FIFO0, &RxMessage);
    
    // 2. 把数据存入环形缓冲区（快速操作）
    if (ring_buffer_write(&can_rx_buf, &RxMessage) == 0)
    {
        // 缓冲区满了？记录溢出标志
        can_overflow_flag = 1;
    }
    
    // 3. 清除中断标志
    CAN_ClearITPendingBit(CAN1, CAN_IT_FMP0);
    
    // 注意：这里没有报文解析，没有协议处理！
}
💡 我的小技巧： ISR里尽量不用printf、不用延时、不用malloc。这些函数执行时间不可控，而且printf本身可能触发中断，搞不好就递归了。

嗯，这里还要注意一点：CAN有多个中断源——接收中断、发送中断、错误中断。每个中断源最好单独写一个ISR，别图省事写在一起。否则调试的时候，你根本分不清是哪个事件触发了中断。

12.3 中断调试技巧——没有示波器也能定位问题

中断调试，说实话比普通代码调试难。为什么？因为中断是异步的，你打断点的时候，中断可能已经跑飞了。

我总结了几招，很实用：

  1. 用GPIO翻转做“软示波器”

在ISR入口和出口各翻转一次GPIO，用逻辑分析仪或者示波器看这个引脚。你能直观看到：中断多久触发一次？ISR执行了多长时间？有没有嵌套？

void CAN1_RX0_IRQHandler(void)
{
    GPIO_SetBits(GPIOB, GPIO_Pin_0);  // 入口拉高
    
    // ... 中断处理 ...
    
    GPIO_ResetBits(GPIOB, GPIO_Pin_0); // 出口拉低
}
  1. 统计中断触发次数

定义一个全局变量，每次进ISR就加1。主循环里打印这个值。如果实际值和预期不符，说明中断配置有问题。

volatile uint32_t can_irq_count = 0;

void CAN1_RX0_IRQHandler(void)
{
    can_irq_count++;
    // ... 其他处理 ...
}
  1. 检查中断标志位

有时候中断没响应，不是因为没触发，而是标志位没清干净。我建议你在ISR开头先读一下中断状态寄存器，确认是哪个事件触发的。

🔑 核心要点： CAN中断调试，80%的问题出在优先级配置和标志位处理上。先检查这两个，别一上来就怀疑硬件。

12.4 实战中的那些坑

讲几个我亲身踩过的坑，你遇到了能少走弯路：

问题现象 原因 解决方案
CAN中断只触发一次 中断标志位没清除 ISR末尾调用清除函数
系统卡死，无法响应 ISR里调用了阻塞函数 ISR只做数据搬运
中断响应延迟大 优先级设置太低 提高抢占优先级
中断频繁触发 CAN总线错误导致错误中断 检查总线终端电阻和电平

我曾经遇到一个特别诡异的bug：CAN接收中断偶尔丢包。查了两天，最后发现是ISR里用了一个全局变量，而这个变量在主循环里被修改了——没有加volatile关键字。编译器优化后，ISR读到的永远是旧值。

所以，记住：ISR和主循环共享的变量，一定要加volatile！

12.5 总结一下

CAN中断编程，说白了就三件事：

  • 优先级配好——别太高也别太低
  • ISR写短——只做搬运工
  • 调试有招——GPIO翻转+计数统计

你把这些掌握了，CAN中断这块基本就稳了。下一章，我们聊聊CAN的错误处理——那才是真正考验功底的地方。
