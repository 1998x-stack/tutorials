第22章 时序交叉验证

大家好。前面三章我们学了特征工程、树模型和自动化预测工具。这一章要聊的话题，比前面任何一个模型或特征都更加重要——如何正确地评估你的时序模型。

你可能觉得评估很简单：划分训练集和测试集，算个 MAE 不就好了？但时序领域的评估暗藏杀机。一个看似合理的评估流程可能给你虚高的分数，让你满怀信心地把模型部署到生产环境，然后看着真实效果惨不忍睹。

这章的核心只有一个问题：你凭什么相信你的模型真的能预测未来？

22.1 为什么不能随机 Shuffle

先讲一个真实的案例。几年前，一家金融科技公司的数据科学家用随机森林预测用户违约风险，在交叉验证中 AUC 达到了 0.96。模型上线后，实际 AUC 掉到了 0.52——跟抛硬币差不多。

问题出在哪里？他们用 cross_val_score 的默认 K-Fold。K-Fold 是随机划分数据的，而他们的数据是按时间排列的——用户的信贷行为在时间上是高度自相关的。随机划分意味着每个 fold 的测试集中，大量样本的时间"邻居"在训练集中。模型没有在预测未来，而是在做"近邻记忆"。

这个故事的教训是：时序数据的交叉验证，永远不能随机 shuffle。你的训练集必须全部在测试集之前。

踩坑警告：我曾经见过有人用 ShuffleSplit 做时序验证得到 R² = 0.95，换成时序划分后 R² = 0.31。团队差点拿着 0.95 的结果去给投资人做演示。这个 0.95 完全是无意义的数字——它衡量的是"模型能在相似的历史中做模式匹配"，而不是"模型能预测未来"。

22.2 两种时序验证策略

时序交叉验证有两种主流形式：

扩展窗口：训练集从起始点不断向后扩展，测试集固定在训练集之后的一个固定长度窗口。早期 fold 训练数据少、验证时间长，后期 fold 训练数据多、验证时间短。

滚动窗口：训练集固定一个窗口长度（如最近 365 天），随着时间向前滚动。测试集紧跟在训练窗口之后。所有 fold 的训练数据量保持一致。

特性扩展窗口滚动窗口
训练数据量逐步增加固定不变
适合场景数据量小，想用全部历史数据量大，或数据分布随时间变化
计算成本递增（后期 fold 训练慢）稳定
对概念漂移的适应性差（早期数据拖累模型）好（只关注近期模式）
更接近线上场景不一定通常更接近（只用近期数据预测）

用代码实现两种策略：

import numpy as np
import pandas as pd
from sklearn.metrics import mean_absolute_error

def expanding_window_cv(df, n_splits=5, test_size=30):
    """扩展窗口交叉验证"""
    scores = []
    total = len(df)
    start = total - n_splits * test_size

    for i in range(n_splits):
        train_end = start + i * test_size
        test_start = train_end
        test_end = test_start + test_size

        train = df.iloc[:train_end]
        test = df.iloc[test_start:test_end]

        # 这里训练你的模型
        # model.fit(train[features], train['y'])
        # preds = model.predict(test[features])
        # scores.append(mae(test['y'], preds))
        print(f'Fold {i+1}: train=0:{train_end}, test={test_start}:{test_end}')

    return scores

def rolling_window_cv(df, window=365, test_size=30, n_splits=5):
    """滚动窗口交叉验证"""
    scores = []
    for i in range(n_splits):
        test_end = len(df) - i * test_size
        test_start = test_end - test_size
        train_start = test_start - window
        train_end = test_start

        train = df.iloc[train_start:train_end]
        test = df.iloc[test_start:test_end]

        print(f'Fold {i+1}: train={train_start}:{train_end}, test={test_start}:{test_end}')

    return scores

22.3 步长选择：一个容易被忽视的参数

时序交叉验证中，步长（step size / stride）是指每个 fold 向前移动多少步。理论上步长应该等于你的预测周期。如果你预测未来 30 天，步长应该是 30。但实践中，步长对评估结果的影响比你想象的大。

步长太小（比如预测未来 30 天，但步长设为 1）：相邻 fold 的测试集几乎重叠，评估结果高度相关，你得到的"多次验证平均分数"其实只代表了一小段时间的表现。

步长太大（比如预测未来 30 天，但步长设为 365）：验证次数太少，分数方差大，调参时容易过拟合 validation set。

一个实用的折中：步长设为预测周期的 1-3 倍，验证次数不少于 5 次。比如预测未来 30 天，步长设为 30-90，做 5-10 个 fold。

22.4 Purging 和 Embargo：防止泄露的最后防线

即使你用了时序交叉验证，依然可能存在一种隐蔽的数据泄露。当你的特征包含滞后值时，测试集的一个样本和训练集中某个样本可能在时间上非常接近（甚至只差一天），导致信息通过"近邻相似性"泄露。

解决方法分别是 purging 和 embargo，这两个概念来自 Marcos López de Prado 的《Advances in Financial Machine Learning》。

  • Purging：从训练集中移除那些和测试集时间上重合的样本。适用于特征中含有长期滞后项的场景。
  • Embargo：在训练集和测试集之间留出一段"隔离带"。例如，设定 embargo 为 5 天，意味着训练集只包含距离测试集至少 5 天之前的数据。
关键洞察：Embargo 的大小取决于特征中用到的最大滞后阶数。如果你的最大滞后是 lag_28，那么至少需要 28 天的 embargo，否则测试集的 t 时刻样本会包含训练集 t-1 时刻的信息。这个"自我重复"的微妙泄露，在金融时序中足以让你构建一个看起来很赚钱但实际上毫无预测能力的策略。

22.5 模型集成（Ensemble）与时序

在正确验证的基础上，我们可以进一步通过集成提升预测效果。时序模型集成和普通 ML 集成有一个关键区别：时序集成必须考虑时间漂移。

两种主流的时序集成方式：

Blending（加权平均）：在验证集上计算每个模型的权重，然后用固定权重做预测。简单、稳定、不容易过拟合。权重可以用验证集上的误差（如 MAE）的倒数来确定。

Stacking（堆叠）：用一个元模型学习如何组合各基础模型的预测。更灵活，但也更容易过拟合。在时序场景下，Stacking 的元模型训练必须严格使用时序交叉验证——把每个 fold 的基础模型预测作为元模型的特征。

import numpy as np
from sklearn.linear_model import Ridge

def timeseries_blending(predictions_dict, y_true):
    """
    基于验证集表现做简单加权平均
    predictions_dict: {'model_name': np.array of predictions}
    """
    errors = {}
    for name, preds in predictions_dict.items():
        errors[name] = np.mean(np.abs(preds - y_true))

    total = sum(1 / e for e in errors.values())
    weights = {name: (1 / e) / total for name, e in errors.items()}
    return weights

def timeseries_stacking(base_preds_train, y_train,
                         base_preds_test, meta_model=None):
    """
    时序堆叠集成
    base_preds_train: (n_train, n_models)
    """
    if meta_model is None:
        meta_model = Ridge(alpha=1.0)
    meta_model.fit(base_preds_train, y_train)
    return meta_model.predict(base_preds_test)

22.6 完整的基准对比框架

将以上所有内容综合起来，一个严谨的时序模型评估和对比框架应该包含：

  1. 固定一个统一的时序交叉验证流程（5-fold 滚动窗口，步长 = 预测周期 × 2）
  2. 每个 fold 内执行完整的特征工程（不使用未来数据，包括不会跨越 fold 边界的滚动特征）
  3. 对比至少 3 个候选模型（如 ARIMA baseline、LightGBM、mlforecast）
  4. 使用 purging 和 embargo 消除近邻泄露
  5. 报告每个 fold 的分数分布（不只是平均分，还包括方差）
  6. 做一次额外的"最终验证"：留出最后一段完全未见过的数据做终验 ——这段数据不参与任何调参和模型选择决策
小贴士：严谨的评估框架看起来繁琐，但它是你和"数据骗子"之间的分界线。一个简单的建议：把你做评估的全部代码写成一个函数，每次评估新模型都用同一个函数。这样你永远不会"不小心"用了随机划分，也不会"不小心"让特征泄露了未来信息。
