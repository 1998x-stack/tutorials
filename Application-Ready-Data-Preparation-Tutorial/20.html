数据画像：理解你拥有的数据

大家好。今天我们要聊一个听起来很"老派"的话题——数据画像。你可能会想：数据仓库时代就在讲数据画像了，现在都LLM时代了，有什么好说的？但恰恰是LLM时代的到来，让数据画像从一个"可有可无的辅助工作"变成了"决定AI项目成败的关键环节"。

原因不难理解。当你准备用LLM处理数据时，你首先需要回答几个基本问题：我的数据集中有哪些列？每列数据的语义含义是什么？哪些列是冗余的？哪些列代表了同一个实体？这些问题看起来简单，但在真实的大规模数据湖中，答案往往出人意料。一个典型的中型企业数据湖可能有数千张表、数十万列，列名缩写晦涩、语义模糊、值分布诡异——如果没有系统性的数据画像，后面的所有数据操作都是在黑暗中摸索。

关键认识： 数据画像不是一次性的"体检"，而是一个持续的过程。随着新数据的接入、业务语义的演化、数据质量的变化，画像结果需要不断更新。在LLM驱动的数据准备流水线中，数据画像扮演着"导航仪"的角色——它告诉我们数据在哪里、是什么、有什么问题，然后下游的清洗、标注、转换模块才能有针对性地工作。

三级画像策略

在实际工程中，我们把数据画像分为三个层次。单看任何一个层次都不足以全面理解数据，但三者结合起来就能形成一个完整的数据认知。这种三级分层策略最早由AutoDDG（Automatic Data Description Generation）框架提出，后来被多家大厂采纳为内部标准。

画像层级 分析对象 输出内容 典型技术
表级画像 整张表 行数、列数、主键候选、外键关系、表间关联 PK/FK检测、基数估计、关联规则
列级画像 单列 数据类型、缺失率、唯一值比率、值分布、语义类型 统计摘要、直方图、语义类型分类
模式级画像 跨表结构 模式相似度、列映射、数据依赖、异常模式 嵌入相似度、函数依赖挖掘、图匹配

三级的执行顺序通常是自顶向下的：先做表级画像了解整体轮廓，然后深入到列级分析每一列的具体特征，最后在模式级发现跨表的连接点和数据依赖关系。在实际操作中，表级和列级可以并行执行，但模式级依赖于前两者的输出。

语义类型自动发现

语义类型发现（Column Type Annotation, CTA）是数据画像中最具挑战性也最有价值的环节之一。传统方法依赖正则表达式和字典匹配——看到"2024-01-01"就猜是日期，看到"13800138000"就猜是手机号。这种方法的局限性很明显：一是无法覆盖长尾类型，二是无法区分格式相似的列（例如"身份证号"和"手机号"在格式上有重叠）。

Sherlock是2019年提出的一个经典CTA模型，它从列数据中提取字符分布、词频、嵌入等特征，训练一个多分类器来预测语义类型。Sherlock在WebTable数据集上取得了不错的效果，但它的局限性同样明显——分类器是预先训练好的，无法灵活扩展到用户自定义的类型体系。

LLM的到来彻底改变了这个局面。LLM不需要预先训练，只需要在Prompt中描述候选类型和判定规则，就能零样本地完成语义类型分类。更重要的是，LLM可以"理解"列名的语义——例如一个名叫"c_id"的列，传统方法只能看到两个字符的分布，但LLM能猜出它可能是"customer_id"的缩写。

# LLM驱动的语义类型发现
from openai import OpenAI
import pandas as pd

client = OpenAI()

def discover_column_types(df, candidate_types=None):
    """对DataFrame的每一列进行语义类型检测"""
    if candidate_types is None:
        candidate_types = [
            "人名", "组织名", "地址", "电子邮件",
            "电话号码", "日期", "时间", "金额",
            "百分比", "身份证号", "邮政编码",
            "产品名称", "订单编号", "客户ID",
            "城市名", "国家/地区", "IP地址",
            "URL链接", "枚举值", "布尔值", "文本描述"
        ]

    column_types = {}

    for col in df.columns:
        # 抽样该列的数据值
        sample_values = df[col].dropna().sample(min(5, len(df))).tolist()
        sample_str = "\n".join([str(v) for v in sample_values])

        # 统计基本信息
        null_ratio = df[col].isnull().mean()
        unique_ratio = df[col].nunique() / len(df)

        prompt = f"""分析以下数据列的语义类型。

列名: {col}
唯一值比率: {unique_ratio:.2%}
缺失率: {null_ratio:.2%}
样本值:
{sample_str}

从以下候选类型中选择最匹配的一个（如果都不匹配，请选择"其他"）：
{', '.join(candidate_types)}

请以JSON格式输出：
{{
    "semantic_type": "判定的语义类型",
    "confidence": 置信度(0-1),
    "reasoning": "简要推理过程"
}}"""

        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": prompt}],
            response_format={"type": "json_object"}
        )

        import json
        result = json.loads(response.choices[0].message.content)
        column_types[col] = result

    return column_types
工程建议： LLM做CTA虽然灵活，但对于高吞吐场景成本偏高。一个实用的分层策略是：先使用基于规则的匹配器（覆盖常见类型如日期、电话、邮箱）处理80%的列，只有规则匹配不到的"疑难列"才交给LLM。这样可以将总体成本降低到纯LLM方案的十分之一。

层次分类与语义树构建

语义类型发现的下一步是构建层次化的分类体系。一个电商数据集中，"价格"、"折扣价"、"原价"属于"金额"这个大类下的子类；"收货地址"、"公司地址"、"发货地址"属于"地址"大类下的子类。如果只做扁平化分类，这些语义关系就丢失了。

LLM-HTS（LLM-based Hierarchical Type System）是解决这个问题的代表性方法。它的核心流程是：先用LLM对列进行初步语义分类，然后用嵌入聚类自动发现类型之间的层次结构（例如"折扣价"距离"原价"近，说明它们应该是兄弟节点），最后让LLM为每个层次节点生成自然语言描述。最终的输出是一个语义树，树的每一层代表了不同粒度的语义抽象——从最顶层的"金额类"到最底层的"满减优惠价格"。

# 基于嵌入聚类的语义层次构建
from openai import OpenAI
import numpy as np
from sklearn.cluster import AgglomerativeClustering

client = OpenAI()

def build_semantic_hierarchy(columns, samples_by_column):
    """为列集合构建层次化语义树"""
    embeddings = {}
    for col in columns:
        # 用列名+样本值拼成文本，生成嵌入
        text = f"{col}: {' '.join(str(s) for s in samples_by_column[col][:3])}"
        resp = client.embeddings.create(
            model="text-embedding-3-small",
            input=text
        )
        embeddings[col] = np.array(resp.data[0].embedding)

    # 层次聚类
    emb_matrix = np.array(list(embeddings.values()))
    clustering = AgglomerativeClustering(
        n_clusters=None,
        distance_threshold=0.5,
        metric="cosine",
        linkage="average"
    )
    labels = clustering.fit_predict(emb_matrix)

    # 按聚类分组输出
    clusters = {}
    for col, label in zip(columns, labels):
        clusters.setdefault(int(label), []).append(col)

    # 用LLM为每个簇生成语义标签
    for cluster_id, cluster_cols in clusters.items():
        cols_str = ", ".join(cluster_cols)
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{
                "role": "user",
                "content": f"以下列属于同一个语义组: {cols_str}\n"
                           f"请用2-4个字概括这个组的共同语义类型："
            }]
        )
        cluster_label = resp.choices[0].message.content.strip()
        print(f"簇 {cluster_id}: {cluster_label} -> {cluster_cols}")

    return clusters

实战案例：列名"ID"背后隐藏的30种语义

这是一个让我印象极深的踩坑经历。某数据中台团队在整合多个业务系统的数据时，发现几乎所有表都有一列叫"ID"的字段。数据工程师理所当然地认为这列在不同表中代表的是同一种实体标识，就直接用"ID"列做了跨表关联。结果呢？数据混乱持续了整整半年——订单量统计对不上、用户画像数据错位、报表指标打架。

后来他们做了一个系统性的数据画像分析，才发现"ID"这个列名在不同的业务系统中至少对应了30种不同的语义：有的是用户ID，有的是订单ID，有的是设备ID，甚至有的是"数据库自增ID"（毫无业务含义）。最夸张的是，在同一个数据库中，有3张表的"ID"列分别代表三种完全不同的实体，但因为列名都叫"ID"，被ETL工程师统一当成了同一个东西处理。

深刻的教训： 列名是人类给数据贴的第一个标签，但也是最不可靠的标签。一个"ID"可以代表30种含义，"name"可以是人名、产品名、文件名。在LLM时代，我们有了强大的语义理解能力，但工具越强大，越不能丢掉"先画像再做操作"的基本原则。数据画像不是锦上添花——它是数据集成前的必经之路，不经过画像就直接做数据操作，相当于蒙着眼睛开车。

好，数据画像就讲到这里。这一章我们学习了三级画像策略、语义类型自动发现、以及层次语义树的构建方法。画像让我们理解了数据的"静态结构"，但数据真正的价值在于它承载的语义关系。下一章，我们将把这些语义关系显式地建模出来——用知识图谱来增强数据处理能力。我们下一章见。
