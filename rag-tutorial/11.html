第十一章：重排序技术详解

大家好，我是你们的讲师。上一章我们讲了混合检索，通过融合BM25和稠密检索的结果大大提升了召回质量。但这里有个微妙的问题：混合检索的"底层评分"——无论是BM25的TF-IDF分数还是向量的余弦相似度——其实都只是粗糙的近似信号。

想想看，向量相似度衡量的是两个文本的"整体语义接近程度"，但用户的查询可能只关心某个具体方面。例如搜索"苹果公司的财务状况"，一个关于"苹果公司历史"的文档可能有很高的语义相似度，但它根本不包含财务信息。这时候，就需要重排序（Reranking）出场了。

1. 为什么需要重排序？

重排序的核心逻辑是：在第一阶段（Bi-Encoder检索）快速筛出候选集（比如Top-100），然后在第二阶段用更强的模型（Cross-Encoder）对候选集做精细排序，选出真正相关的Top-5到Top-10送到生成模块。

这种两阶段架构的好处是：

  • 效率：Bi-Encoder可以预计算所有文档的向量，检索时只需计算一次查询向量，速度快。
  • 精度：Cross-Encoder让查询和文档直接交互，能捕捉到细粒度的匹配信号。
  • 互补：第一段"广撒网"保证高召回，第二段"精挑细选"保证高精度。

2. Cross-Encoder vs Bi-Encoder

对比维度 Bi-Encoder Cross-Encoder
编码方式 查询和文档分别独立编码 查询和文档拼接后一起编码
交互程度 无交互（仅有最后的向量对比） 深度交互（Attention跨越两个文本）
预计算 文档向量可预计算 不可预计算，需要实时推理
速度 极快（适合检索阶段） 慢（适合重排序阶段，只处理少量结果）
精度 中等 高（通常比Bi-Encoder高5-15% MRR）
💡 小贴士：批次处理降低延迟
在使用Cross-Encoder做重排序时，一个常见的性能问题是：如果候选集有50个文档，就需要50次独立推理，延迟极高。解决方案是批量处理（Batch Inference）：将50个(query, doc)对打包成一个batch一次性送入Cross-Encoder。通常batch_size=16-32可以在精度无损的情况下将延迟降低5-10倍。注意要根据你的GPU显存选择合适的batch_size。

3. 主流的重排序模型

目前社区中有几个非常优秀的重排序模型：

模型 开发者 特点 推荐场景
BGE-Reranker-v2-m3 BAAI（北京智源研究院） 支持多语言、多粒度（token/sentence/doc），1024k上下文 中文场景首选，支持长文档

4. 两阶段检索架构

一个标准的RAG两阶段检索流程如下：

  1. 第一阶段——检索阶段：使用Bi-Encoder（如BGE嵌入模型）或混合检索（BM25+稠密）从全量知识库中快速召回Top-N（通常N=50-200）。
  2. 第二阶段——重排序阶段：使用Cross-Encoder（如BGE-Reranker-v2-m3）对Top-N候选文档进行精细评分，重新排序后取Top-K（通常K=3-10）。
  3. 送入生成：将重排序后的Top-K文档作为上下文送入LLM进行回答生成。

这个架构的关键是N >> K。N越大，第一阶段的召回率越高，但第二阶段的延迟也越大。合理的经验值：N取100，K取5，可以在召回率和延迟之间取得良好平衡。

5. 代码实战：使用FlagEmbedding做重排序

from FlagEmbedding import FlagReranker

# 加载BGE重排序模型
reranker = FlagReranker('BAAI/bge-reranker-v2-m3', use_fp16=True)

# 查询和候选文档
query = "如何治疗上呼吸道感染？"
candidates = [
    "感冒是由病毒引起的上呼吸道感染，通常一周可自愈。",
    "肺炎是肺部感染，需要抗生素治疗。",
    "上呼吸道感染包括感冒、咽炎、扁桃体炎等。",
    "高血压患者应注意低盐饮食和规律运动。",
    "上呼吸道感染的治疗包括休息、多喝水和对症用药。",
]

# 构造(query, doc)对
pairs = [(query, doc) for doc in candidates]

# 批量计算相关性分数（batch推理）
scores = reranker.compute_score(pairs, batch_size=4)

# 排序
scored_docs = list(zip(candidates, scores))
scored_docs.sort(key=lambda x: x[1], reverse=True)

# 输出结果
print("重排序结果：")
for doc, score in scored_docs:
    print(f"  [{score:.4f}] {doc[:40]}...")

# 输出：
# [0.9821] 上呼吸道感染包括感冒、咽炎、扁桃体炎等...
# [0.9734] 上呼吸道感染的治疗包括休息、多喝水和对症用药...
# [0.8512] 感冒是由病毒引起的上呼吸道感染，通常一周可自愈...
# [0.3123] 肺炎是肺部感染，需要抗生素治疗...
# [0.1247] 高血压患者应注意低盐饮食和规律运动...

踩坑故事："相关但不是答案"——重排序的陷阱

曾经做一个医疗问答系统，发现一个奇怪的现象：无论问什么症状，重排序模型都把"症状概述"类文档排在第一位。比如用户问"头痛怎么办？"，排第一的永远是"头痛是一种常见的临床症状，可能由多种原因引起……"这类概述文档，而不是给出具体建议的文档。

排查后发现：Cross-Encoder的评分是基于相关性（Relevance）而非有用性（Usefulness）。概述类文档几乎和任何查询都高度相关，因为它们覆盖面广。但你真正想要的是能回答问题的具体干货。

解决方案：我们在训练重排序模型时，需要特别标注"有用性"标签——不仅问"这个文档和查询相关吗"，还要问"这个文档能帮助回答这个问题吗"。同时，在重排序之后，我们增加了一个基于信息密度的二次过滤，过滤掉过于泛泛而谈的文档。

6. 重排序的延迟分析

很多团队担心引入重排序会增加系统延迟。我们来看实际的延迟数据（基于BGE-Reranker-v2-m3，A100 GPU）：

候选集大小 单样本延迟 Batch=16延迟 Batch=32延迟
20 ~400ms ~30ms ~20ms
50 ~1000ms ~70ms ~45ms
100 ~2000ms ~130ms ~80ms

可以看到，使用batch推理可以将延迟降低一个数量级。对于大部分的RAG场景，重排序的整体延迟在30-100ms之间，是完全可以接受的开销。

本章小结

重排序是RAG检索流程中的"质变"环节。通过Cross-Encoder对候选文档进行精细评分，能有效过滤噪声、提升Top-K结果的相关性。Bi-Encoder负责广撒网，Cross-Encoder负责精挑细选，两阶段架构兼顾了效率和精度。BGE-Reranker-v2-m3是目前中文场景的最佳选择之一。
