第7章 数据准备评估体系

大家好，我是你们的讲师。过去五章，我们学习了从提示工程到微调的完整技术栈。但一个关键问题始终悬而未决：我们如何知道这些方法到底好不好？只看演示案例往往失真——演示时选的都是模型做对了的例子。真正的问题隐藏在那些模型做错的时候。

我见过太多团队在数据准备工具的评估上走了弯路：有的只看整体F1-score，结果忽视了模型在关键子类上的严重退步；有的用自己收集的测试集评估，结果测试集本身就有标注错误；还有的用GPT-4做裁判评估另一个模型，结果陷入了"测量仪器本身就不可靠"的困境。本章我们将系统性地建立一套数据准备评估体系：从核心指标、基准数据集到实验设计原则，让你能够科学地衡量和比较不同方法的效果。

本章核心知识点脉络： 评估的核心困难 → 评测指标全解 → 基准数据集大全 → 实验设计原则 → 指标陷阱实战案例

为什么评估如此困难

数据准备任务的评估之所以困难，根本原因在于任务的多样性。不像ImageNet有明确的分类指标，或者机器翻译有BLEU评分，数据准备涵盖了几种本质不同的任务类型，每种任务需要不同的评估策略。

  • 分类型任务（错误检测、数据标注、实体匹配）：需要精确/召回/F1等分类指标。
  • 生成型任务（数据清洗、格式转换、缺失值填补）：需要衡量输出与标准答案的相似度。
  • 检索引擎任务（去重、相似度排序）：需要衡量排序质量（MRR、NDCG）。
  • 端到端任务（清洗后对下游模型的影响）：需要衡量下游任务指标的变化。

更麻烦的是，同一个任务的不同侧面可能需要不同的指标。一个实体匹配模型可能在"精确率"上很高（它很少误判匹配），但"召回率"很低（它漏掉了大量真正的匹配）。只看一个指标会得出完全不同的结论。

评估第一原则： 永远不要在论文或报告中使用单一指标来声称你的方法"更好"。一个负责任的评估至少需要报告精确率、召回率、F1-score，并且按数据子类分组展示结果。如果可能，还需要报告统计显著性检验的结果。

核心评测指标

以下是数据准备任务中最常用的评估指标，按任务类型分类。

指标 公式 适用任务 说明
Precision TP / (TP + FP) 错误检测、EM、标注 模型判断为"正"的结果中，有多少是真正正确的
Recall TP / (TP + FN) 错误检测、EM、标注 所有真正的"正"例中，模型找出了多少
F1-score 2PR / (P + R) 所有分类任务 精确率与召回率的调和平均数
ROC-AUC ROC曲线下面积 二分类任务 模型排序能力，不受阈值影响
MRR 1/k的平均值 去重、相似度排序 第一个正确答案出现在第k个位置的倒数
P@k Top-k中正确比例 匹配候选排序 在前k个候选中包含正确匹配的比例
Exact Match 完全正确占比 格式转换、清洗 输出与标准答案完全一致的比例
Edit Distance 编辑距离/长度 近似清洗 清洗结果与标准答案的字符级差异
NDCG 归一化折损累计增益 排序类任务 考量排序顺序加权的指标

在实践中，对于大多数数据准备任务，F1-score是最核心的单一指标。但有两个重要补充：

  1. 按类别分组的F1：不仅报告macro-F1，还要报告每个子类的F1。可能模型在整体上表现不错，但在某个稀有类别上完全失败。
  2. 精确率-召回率曲线：F1只是一个点的值，PR曲线展示了模型在不同阈值下的表现全貌。
# 分组评估示例：按数据类型报告F1
from sklearn.metrics import precision_recall_fscore_support

def evaluate_by_category(y_true, y_pred, categories):
    """
    按类别分组评估数据准备任务的效果。
    """
    results = {}
    for cat in set(categories):
        mask = [c == cat for c in categories]
        cat_true = [y for y, m in zip(y_true, mask) if m]
        cat_pred = [y for y, m in zip(y_pred, mask) if m]

        if len(set(cat_true)) < 2:
            continue  # 该类别只有一种标签，跳过

        p, r, f1, _ = precision_recall_fscore_support(
            cat_true, cat_pred, average="binary"
        )
        results[cat] = {"precision": p, "recall": r, "f1": f1}

    # 汇总行
    p_macro, r_macro, f1_macro, _ = precision_recall_fscore_support(
        y_true, y_pred, average="macro"
    )
    results["macro_avg"] = {"precision": p_macro, "recall": r_macro, "f1": f1_macro}

    return results

# 例子：评估实体匹配在不同品类上的效果
categories = ["电子产品"] * 200 + ["服装"] * 150 + ["食品"] * 100
eval_report = evaluate_by_category(y_true, y_pred, categories)
for cat, metrics in eval_report.items():
    print(f"{cat}: P={metrics['precision']:.2%}, R={metrics['recall']:.2%}, F1={metrics['f1']:.2%}")

基准数据集全景

要评估数据准备方法，你需要标准化的基准数据集。以下是数据准备领域最常用的公开基准数据集，覆盖了错误检测、实体匹配、清洗等核心任务。

数据集名称 任务类型 领域 规模 特点
Chicago Food Inspections 错误检测 食品安全 15万+条 真实脏数据，含各种类型错误
Hospital Data 错误检测、清洗 医疗 ~1000条 小规模，人工标注错误类型，适合快速验证
Beers 错误检测、清洗 食品饮料 ~2400条 多种错误类型，含时序标注
WDC Products 实体匹配 电商 1600万+实体 大规模跨站产品匹配，含gold标准
MIMIC-III/IV 错误检测、EM 医疗 5万+记录 真实EMR数据，隐私脱敏
DBPedia 实体匹配 通用知识图谱 数百万实体 跨来源匹配（DBpedia vs Wikipedia等）
OpenData Schema匹配 政府开放数据 ~1000组 不同政府门户的表格Schema匹配
GDC 数据清洗综合 混合 10万+条 包含11种错误类型的合成+真实数据
PrepBench 综合评测 混合 多子集 专为LLM数据准备设计的benchmark，含8+子任务
AB公司人工标注 错误检测、EM 混合 ~2000条 5轮独立标注，含置信度分数

在这些基准数据集中，PrepBench 是2024年专门为评估LLM数据准备能力而设计的新基准，由UC Berkeley和Google Research联合发布。它包含了错误检测、实体匹配、Schema匹配、数据清洗等8个子任务，使用GPT-4和人工标注联合构建ground truth。如果你想评估一个LLM数据准备系统，PrepBench应该是首选。

基准数据集选择警告： 不要在一个小数据集上反复调优你的方法。Hospital数据集只有1000条记录，如果你在这个数据集上反复调整prompt或训练参数，最终得到的结果很可能是过度拟合到这个特定数据集上——在Hospital上表现优异，换到真实生产数据上表现一塌糊涂。正确的做法是：选择3个以上的数据集覆盖不同领域，并在最终模型上使用完全未见过的数据集做最终评估。

实验设计原则

好的评估不仅仅是选择正确的指标和数据集，还需要严谨的实验设计。以下是我从多次评估中总结的核心原则：

原则一：控制变量法

比较方法A和方法B时，确保只有一个变量不同。例如，比较"GPT-4o + ICL"和"微调Qwen2.5-3B"时，两个方法的评估数据、评估指标、评估脚本必须完全一致。看似简单，但我见过太多论文比较时使用不同的测试集甚至不同的标注标准。

原则二：消融实验（Ablation Study）

当你的方法包含多个组件（如CoT + ICL + 自纠正），你需要逐一移除组件来验证每个组件的贡献。消融实验能回答"这个组件到底带来了多少提升"的问题。

原则三：统计显著性检验

方法A的F1=0.92，方法B的F1=0.91——这是真正的提升还是随机波动？在评估数据量不够大时，这种差异完全可能是随机带来的。使用McNemar's test或Bootstrap test来判断差异是否显著。

# Bootstrap统计显著性检验示例
import numpy as np
from sklearn.metrics import f1_score

def bootstrap_significance_test(y_true, pred_a, pred_b, n_iterations=5000):
    """
    使用Bootstrap方法检验方法A和B的F1差异是否显著。
    """
    n = len(y_true)
    f1_a = f1_score(y_true, pred_a, average="macro")
    f1_b = f1_score(y_true, pred_b, average="macro")
    observed_diff = f1_a - f1_b

    # Bootstrap: 从样本中有放回抽样
    boot_diffs = []
    for _ in range(n_iterations):
        indices = np.random.choice(n, n, replace=True)
        boot_f1_a = f1_score(np.array(y_true)[indices], np.array(pred_a)[indices], average="macro")
        boot_f1_b = f1_score(np.array(y_true)[indices], np.array(pred_b)[indices], average="macro")
        boot_diffs.append(boot_f1_a - boot_f1_b)

    # 计算p值：观察到的差异在零分布中的位置
    boot_diffs = np.array(boot_diffs)
    p_value = np.mean(np.abs(boot_diffs) >= np.abs(observed_diff))

    return {
        "f1_a": f1_a,
        "f1_b": f1_b,
        "diff": observed_diff,
        "p_value": p_value,
        "significant": p_value < 0.05
    }

# 使用
result = bootstrap_significance_test(y_true, pred_a, pred_b)
print(f"差异显著: {result['significant']} (p={result['p_value']:.4f})")

原则四：评估数据与训练数据严格分离

这条原则看似是常识，但在实际中经常被违反。特别是对于ICL方法——ICL的示例本身也是从标注数据中选取的。如果你从测试集中选取ICL示例，那评估结果就失去了意义。正确做法：将数据分为训练集(60%)、验证集(20%)和测试集(20%)，ICL示例只能从训练集中选取。

实验设计检查清单： 1) 测试集与训练/验证集完全无交叉；2) ICL示例来自训练集，非测试集；3) 至少使用2个独立数据集做评估；4) 报告置信区间或p-value；5) 消融实验确认每个组件的贡献；6) 基线方法使用与你的方法相同的评估框架；7) 在最终report之前，不要用测试集调优任何参数。

实战案例：高F1却生产翻车

最后，分享一个我自己犯过的严重错误。2023年底，我为一个电商平台的商品匹配系统开发了一个实体匹配模型。在离线评估中，模型在WDC Products测试集上取得了F1=0.95的优异成绩。团队信心满满地部署到生产环境。结果上线第一天就翻车了——匹配准确率只有62%，远低于离线测试的95%。

崩溃排查后，问题的根源很清晰：

# 线下测试 vs 线上真实数据的差异
# 1. 数据分布偏移
wdc_test_data = {
    "source": "结构化电商数据（XML格式）",
    "字段完整度": "95%+ 字段完整",
    "噪声水平": "低（经过清洗的公开数据）",
    "匹配难度": "中等（同平台商品变体）"
}

production_data = {
    "source": "用户上传的非结构化文本 + OCR识别",
    "字段完整度": "约60%（大量缺失品牌、型号字段）",
    "噪声水平": "极高（包含错别字、表情符号、乱码）",
    "匹配难度": "非常高（不同平台、不同命名习惯）"
}

# 2. 评估指标的陷阱
# 我们在离线测试中使用了 "宏观平均F1" (macro-F1)
# 但线上数据的类别分布与测试集完全不同
# 模型在"高频品类"上表现尚可（F1=0.88），但在"长尾品类"上惨不忍睹（F1=0.35）
# macro-F1掩盖了长尾品类的糟糕表现

# 3. 问题复现
def diagnose_failure():
    offline_results = {
        "电子产品": {"f1": 0.96, "占比": 0.35},
        "服装": {"f1": 0.94, "占比": 0.30},
        "食品": {"f1": 0.97, "占比": 0.20},
        "长尾品类": {"f1": 0.93, "占比": 0.15},  # 离线测试有标注的长尾品类较少
    }
    # macro-F1 = (0.96 + 0.94 + 0.97 + 0.93) / 4 = 0.95 ✓（看起来很好）

    online_results = {
        "电子产品": {"f1": 0.88, "占比": 0.25},
        "服装": {"f1": 0.85, "占比": 0.20},
        "食品": {"f1": 0.90, "占比": 0.15},
        "长尾品类": {"f1": 0.35, "占比": 0.40},  # 线上长尾品类实际占比40%！
    }
    # 实际的线上F1 = 0.88*0.25 + 0.85*0.20 + 0.90*0.15 + 0.35*0.40 = 0.62 ✗

diagnose_failure()
print("教训：只报告macro-F1会掩盖分布偏移带来的性能退化")
print("解决方案：按业务占比加权的weighted-F1 + 实际分布下的测试集")

这个案例给了我们三个重要教训：

  1. 基准数据集的分布必须接近实际生产分布。不要在精心清洗的公开数据集上训练模型，然后直接用在充满噪声的生产数据上。应该在生产数据上采样构建测试集。
  2. 按业务价值加权评估。不要只看macro-F1，要根据业务中各类数据的实际占比计算weighted-F1。如果长尾品类在业务中占比40%，那它在评估中的权重也应该是40%。
  3. 评估不是一次性的。生产环境的数据分布会随时间漂移，定期重新评估模型的真实表现是必需的。我们后来建立了"线上评估dashboard"，实时监控模型在每个品类上的表现，一旦发现漂移立即触发重新训练。

本章核心总结： 一个严谨的评估体系 = 匹配任务类型的指标 + 至少3个基准数据集 + 消融实验 + 统计检验 + 与实际分布对齐的评估数据。不要只让你"看起来好"的指标，要去理解模型在真实场景下会碰到什么数据。离线评估是安全网，但不是免死金牌。
