第8章 5G核心网架构

大家好，我是你们的讲师。今天我们来聊一个通信圈内公认"最颠覆性"的变革——5G核心网架构。

如果你拆开一台4G时代的EPC（Evolved Packet Core）设备，你会看到一堆定制的硬件板卡：信号处理板、转发板、控制板，每一块都是厂商专有的"黑盒子"。扩容？加板卡。升级？换硬件。这种模式从2G延续到了4G，用了将近30年。但在5G时代，这一切被彻底推翻了。5G核心网（5GC）不再依赖专用硬件，而是像互联网应用一样跑在通用服务器上，用软件实现网络功能。用一句话概括：核心网从"硬件盒子"变成了"软件服务"。

截至2026年，全球已有超过200张5G SA（Standalone）核心网投入商用，中国三大运营商全部采用SA架构。3GPP Release 20进一步增强了5GC的服务化架构，为6G核心网的"AI原生"设计铺路。理解5GC，你就抓住了下一代通信网络的中枢神经。

8.1 从4G EPC到5GC：控制与用户面分离

回顾4G EPC，它由MME（移动性管理）、S-GW（服务网关）和P-GW（PDN网关）三个主要网元组成。MME管信令，S-GW/P-GW管数据，但它们之间的分工比较模糊——S-GW既有控制面功能也有用户面功能，导致扩容和维护都不灵活。

5GC解决这个问题的思路非常彻底：控制面（CP）和用户面（UP）完全分离，即CUPS（Control and User Plane Separation）架构。控制面网元负责信令处理和会话管理，用户面网元只负责数据包转发。这样一来，控制面可以集中部署，用户面则可以灵活下沉到靠近用户的位置（比如基站侧或园区内），大幅降低时延。

直观理解：控制面就像快递公司的调度中心，只负责接单、规划路线、跟踪包裹；用户面就是那些送货卡车，只管把包裹从A运到B。调度中心可以只有一个，但卡车可以根据需要在各地灵活部署——这就是CUPS的精髓。

8.2 5GC关键网元详解

5GC采用了"服务化"的设计思想，将传统的一体化网元拆解为多个独立的功能模块，每个模块提供标准化的服务。下面是5GC的核心网元一览表：

网元 英文全称 核心职责
AMF Access and Mobility Management Function 接入与移动性管理：处理UE注册、连接管理、移动性管理、NAS信令终结。UE附着网络后第一个接触的控制面网元
SMF Session Management Function 会话管理：负责PDU会话的建立、修改、释放，UE IP地址分配，UPF选择与控制
UPF User Plane Function 用户面功能：数据包路由转发、QoS执行、流量计量、数据包检测。唯一的用户面网元，是数据流的"高速公路"
UDM Unified Data Management 统一数据管理：存储用户签约数据、鉴权凭证、服务授权。相当于用户的"身份证数据库"
AUSF Authentication Server Function 认证服务功能：处理SEAF与UDM之间的鉴权流程，支持5G-AKA和EAP-AKA'两种鉴权方式
PCF Policy Control Function 策略控制功能：制定QoS策略、接入控制策略、计费策略。指导SMF和AMF如何管理会话和移动性
NEF Network Exposure Function 网络开放功能：将核心网能力以API形式开放给外部应用（如MEC平台、企业应用），是网络与业务之间的"桥梁"
NRF Network Repository Function 服务注册与发现：维护所有网元的服务配置信息，支持服务注册、注销、发现和状态监测——5GC的"服务黄页"

记忆口诀：AMF管接入、SMF管会话、UPF管转发、UDM管数据、AUSF管认证、PCF管策略、NEF管开放、NRF管发现。八个网元各司其职，共同构成5GC的完整服务体系。

8.3 服务化架构SBA

5GC最标志性的创新就是SBA（Service-Based Architecture）。在4G中，网元之间通过预定义的接口（如S1、S11、S5）进行通信，接口类型固定，升级一个网元可能影响多个接口。5GC彻底抛弃了这种"接口导向"的模式，改为"服务导向"——每个网元把自己能做的事情注册为服务，其他网元通过API调用这些服务。

SBA的核心技术栈是：

  • HTTP/2：作为服务化接口的传输协议，取代了传统的SCTP和Diameter。HTTP/2支持多路复用、头部压缩、服务端推送，比HTTP/1.1更适合核心网的高频通信场景。
  • JSON/Protocol Buffers：信令消息采用JSON或Protobuf序列化，取代了ASN.1编码，开发和调试更加方便。
  • RESTful API：每个服务通过RESTful API暴露，接口风格统一，第三方应用也可以通过NEF调用这些API。

在SBA中，NRF是"服务注册中心"。每个网元启动时向NRF注册自己的服务信息（服务名称、IP地址、API版本、支持的特性等）。其他网元需要调用服务时，先向NRF查询，获取目标服务的地址后直接调用。这个过程和微服务架构中的Service Registry（如Consul、Eureka）如出一辙。

8.4 NFV与SDN：虚拟化的双引擎

SBA是软件架构层面的创新，而NFV（Network Functions Virtualization）和SDN（Software-Defined Networking）是基础设施层面的支撑。两者经常被放在一起讨论，但分工完全不同：

  • NFV：把网络功能从专用硬件中剥离，运行在通用服务器上的虚拟机或容器中。AMF、SMF、UPF等网元都以虚拟化网元（VNF/CNF）的形式部署。NFV的关键组件包括VIM（虚拟化基础设施管理器，如OpenStack、Kubernetes）、VNFM（VNF管理器）和NFVO（NFV编排器）。
  • SDN：把网络设备的控制面和数据面分离，控制面集中到SDN控制器，数据面设备只做简单的转发。在5GC中，SDN主要用于UPF的转发面控制和核心网内部的数据中心网络。

简单来说：NFV解决了"网元跑在哪里"的问题，SDN解决了"流量怎么走"的问题。两者结合，让5GC从"铁盒子机房"变成了"数据中心里的软件"。中国移动、中国电信和中国联通在2021-2024年陆续完成了5GC的云化部署，全网NFV覆盖率已超过95%。

重要区别：NFV和SDN不是一回事。一个常见的面试题就是"请解释NFV和SDN的区别"。NFV关注网络功能的虚拟化（从专用硬件迁移到通用服务器），SDN关注网络架构的控制与转发分离（集中控制、简化转发）。5GC同时需要两者，但它们解决的是不同层次的问题。

8.5 云原生核心网

NFV的早期实现是基于虚拟机的（如OpenStack），但虚拟机启动慢、资源开销大、弹性伸缩能力有限。随着容器技术的成熟，5GC正在向"云原生"演进——网元以容器（Container）形式部署，由Kubernetes进行编排管理。

云原生5GC的优势非常明显：

  • 快速部署：容器启动只需秒级，而虚拟机需要分钟级。在网络切片场景中，容器化核心网可以在用户请求到达时快速实例化一个新的核心网切片。
  • 弹性伸缩：Kubernetes的Horizontal Pod Autoscaler可以根据CPU/内存/信令负载自动扩缩容。节假日流量高峰时自动增加AMF/SMF实例，流量回落后自动回收资源。
  • 灰度升级：采用"蓝绿部署"或"金丝雀发布"策略，先升级少量实例验证，确认无误后再全量升级，极大降低了升级风险。

截至2026年，中国电信已经在部分省份完成了5GC的容器化改造，核心网网元的部署密度提升了3倍以上，资源利用率从虚拟机的约40%提升到了容器的约70%。

8.6 踩坑实录：NRF发现失败导致全业务中断

讲一段真实的故事。2023年，某运营商在试点5GC SA网络时，遇到了一个令人印象深刻的故障。

那天凌晨，工程师对核心网进行例行升级，在升级NRF节点后重启了所有网元。重启完成后，AMF、SMF、PCF等网元都成功启动并注册到了NRF。但诡异的是——UPF始终无法被SMF发现。工程师检查UPF的日志，发现UPF向NRF发送了注册请求，NRF也返回了200 OK，但SMF查询可用UPF列表时，结果却是空的。

排查了几个小时后，问题终于定位：UPF注册时填写的"nfType"字段写成了小写的"upf"，而SMF查询时请求的是大写的"UPF"。NRF的字符串匹配是大小写敏感的，两者不一致，导致SMF永远发现不了UPF。所有本应经过UPF的数据流全部中断，用户无法上网、无法打电话。

教训：SBA看起来很美，但服务注册/发现环节的每一个字段都必须严格对齐。一个小写字母的差异就能导致全网数据面瘫痪。生产环境的配置校验和自动化测试必不可少——千万不能"人肉"操作。这个故障之后，该运营商在CI/CD流水线中增加了NRF注册/发现的一致性测试用例。
