🔨 第3章 从零实现最简 Agent Loop

大家好，我是你们的讲师。在上一章我们搭建了逆向工具链，算是拿到了"解剖刀"。但说实话，光会拆不会建，理解和学习都是不完整的。这一章我们来换个角度——从零开始写一个最简的 Agent，亲手把那个核心循环跑通。

2025 年 Arize AI 的那份报告提到一个让我印象深刻的数字：一个完整 Coding Agent 里，调用模型的代码只有大约 50 行。今天我们就要亲手验证这一点——用 50 行 Python 代码，实现一个能真正工作的 Agent。不是说压缩到 50 行是面试题，而是说，去掉所有工程化包装后，Agent 最本质的那个循环真的就这么简单。

Agent Loop 的最小定义

在动手写代码之前，我们先搞清楚一个问题：到底什么才算一个 Agent？

学术界和工业界有各种复杂的定义，但作为一个务实的工程师，我的定义很简单：一个系统只要满足三个条件，就是 Agent——

  • 接收输入：理解人类的自然语言指令
  • 调用 LLM：基于当前上下文生成下一步行动决策
  • 执行行动：根据模型的输出，调用工具、观察结果、形成新上下文

把三件事串成一个循环，就是 Agent Loop。伪代码如下：

# Agent Loop 最低限度伪代码
messages = [{"role": "user", "content": task}]

while not done:
    # 1. 调用 LLM，获取响应
    response = client.messages.create(
        model="claude-sonnet-4",
        messages=messages,
        tools=[...]  # 注册可用工具
    )

    # 2. 检查响应：是工具调用还是最终回答
    if response.stop_reason == "tool_use":
        for block in response.content:
            if block.type == "tool_use":
                # 3. 执行工具
                result = execute_tool(block.name, block.input)
                # 4. 将结果附加到消息列表
                messages.append(tool_result_block(result))
    else:
        # 模型给出了最终回答，循环结束
        done = True
        print(response.content[0].text)

就是这样。一个 while 循环，一个 if-else 分支，两端 API 调用（模型端 + 工具执行端）。Agent 的神秘面纱瞬间就揭开了。

实现 v0：50 行 Bash Agent

好，到了真正写代码的时候了。下面这个 v0_agent.py 只做一件事：它可以听懂你的自然语言指令，然后通过执行 Bash 命令来完成它。

"""v0_agent.py — 最简 Bash Agent (50 行)"""

import os
import subprocess
import json
from dotenv import load_dotenv
from anthropic import Anthropic

load_dotenv()
client = Anthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))

# ===== 工具定义 =====
TOOLS = [
    {
        "name": "bash",
        "description": "执行 shell 命令，支持任意合法的 bash 命令",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {
                    "type": "string",
                    "description": "要执行的 bash 命令"
                }
            },
            "required": ["command"]
        }
    }
]

# ===== 工具执行 =====
def execute_tool(name, input_data):
    if name == "bash":
        result = subprocess.run(
            input_data["command"],
            shell=True,
            capture_output=True,
            text=True,
            timeout=30
        )
        output = ""
        if result.stdout:
            output += result.stdout
        if result.stderr:
            output += f"[stderr]\n{result.stderr}"
        if result.returncode != 0:
            output += f"\n[exit code: {result.returncode}]"
        return output
    raise ValueError(f"Unknown tool: {name}")

# ===== Agent Loop =====
def agent_loop(task, max_turns=10):
    messages = [{"role": "user", "content": task}]

    for turn in range(max_turns):
        print(f"\n--- Turn {turn + 1} ---")

        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=4096,
            system="你是一个 Bash 助手。使用 bash 工具执行命令来完成任务。"
                  "每次调用工具后观察结果，决定下一步。完成后直接给出总结。",
            messages=messages,
            tools=TOOLS
        )

        if response.stop_reason == "tool_use":
            for block in response.content:
                if block.type == "tool_use":
                    print(f"🧰 调用工具: {block.name}({json.dumps(block.input)})")
                    result = execute_tool(block.name, block.input)
                    print(f"📋 结果:\n{result[:500]}")
                    messages.append({
                        "role": "assistant",
                        "content": [{"type": "tool_use", **block.model_dump()}]
                    })
                    messages.append({
                        "role": "user",
                        "content": [
                            {"type": "tool_result",
                             "tool_use_id": block.id,
                             "content": result}
                        ]
                    })
        else:
            # stop_reason 为 end_turn 或其他，视为最终回答
            for block in response.content:
                if block.type == "text":
                    print(f"🤖 {block.text}")
            break
    else:
        print("⚠️ 达到最大对话轮次，强制结束")

if __name__ == "__main__":
    task = input("请输入任务: ")
    agent_loop(task)

这段代码其实做了四件简单的事：

  • 工具定义：通过 JSON Schema 告诉模型它可以调用哪些工具（目前只有一个 bash）
  • 工具执行：接收模型的工具调用指令，用 subprocess.run 执行命令
  • 循环管理：把模型响应（tool_use blocks）和执行结果（tool_result blocks）追加到消息列表，形成持续对话
  • 终止检测：当 stop_reason 不是 "tool_use" 时，认为模型给出了最终回答，结束循环

首次运行与行为分析

保存文件后运行：

python v0_agent.py

输入一个简单的任务测试，比如 "列出当前目录的文件"。你会看到类似这样的输出：

--- Turn 1 ---
🧰 调用工具: bash({"command": "ls -la"})
📋 结果:
总用量 32
drwxr-xr-x  4 user user  128 Jun 30 10:00 .
drwxr-xr-x 10 user user  320 Jun 30 09:55 ..
-rw-r--r--  1 user user   51 Jun 30 09:55 .env
drwxr-xr-x  2 user user 4096 Jun 30 09:55 __pycache__
-rw-r--r--  1 user user 2460 Jun 30 10:00 v0_agent.py

--- Turn 2 ---
🤖 当前目录下共有以下文件：
- .env（环境配置文件）
- v0_agent.py（我们的 Agent 脚本）
- __pycache__（Python 缓存目录）

这些就是目录中的全部内容。

注意观察 Agent Loop 是如何工作的：

  • Turn 1：模型收到"列出文件"的指令，决定调用 bash 工具，参数 {"command": "ls -la"}。工具执行后返回了目录列表
  • Turn 2：模型看到了工具执行的结果，认为任务已完成，于是直接输出文本回答，不再调用工具。循环结束

这就是一个完整的 ReAct 循环——思考（要执行 ls 命令）、行动（执行 bash）、观察（看输出）、总结（给出回答）。只有两个回合，但麻雀虽小五脏俱全。

再试一个更复杂的任务，比如 "创建一个 test.txt 文件，写入 Hello World，然后读取它"。你会看到更多回合：

--- Turn 1 ---
🧰 调用工具: bash({"command": "echo 'Hello World' > test.txt"})
📋 结果:                (空——命令成功，无输出)

--- Turn 2 ---
🧰 调用工具: bash({"command": "cat test.txt"})
📋 结果:
Hello World

--- Turn 3 ---
🤖 已完成。创建了 test.txt 文件并写入 "Hello World"，读取验证成功。

这里模型做了一个多步规划：先创建文件，再读取验证。它没有试图两步并一步，而是拆成了两个工具调用——中间观察了第一次的执行结果后，才决定进行第二次验证。这是 ReAct 范式的核心优势：每一步都基于上一步的实际结果做决策。

v0 的局限与下一站

我们这个 50 行的 v0 Agent 能做什么？能执行任何 shell 命令。这听起来已经很厉害了，但看看它不能做什么：

能力维度 能做 ✅ 不能做 ❌
命令执行 执行任意 bash 命令 没有超时限制（除了 subprocess 默认值）
文件操作 通过 bash 间接读写文件 没有专门的 Read/Write 工具，依赖 cat/echo
代码编辑 通过 bash 使用 sed 没有精确的编辑工具，sed 容易出错
上下文管理 消息列表不断累积 没有压缩机制，上下文会无限增长
错误恢复 能看到 stderr 命令失败后没有自动纠错逻辑
任务规划 模型自行决定步骤 没有显式的 Todo 列表，模型可能遗漏
权限控制 无 Agent 可以执行任何命令，包括 rm -rf /
多工具 只有 bash 没有 Read/Write/Edit 等专用工具

最核心的局限是什么？只有 bash 一个工具。这意味着模型只能用 echo、cat、sed 等命令来读写文件，用 find、grep 来搜索代码。这些命令的输出格式不是为机器阅读优化的，模型需要花额外的 token 去 "parse" 它们。而且在编辑代码时，用 sed 做精确替换是非常容易出错的——你试过用 sed 替换一个包含特殊字符的 Python 多行字符串吗？那是噩梦。

另外，我们也没有处理一些工程细节：API 调用可能失败（网络问题、限流）、工具执行可能超时、上下文可能超过模型限制。这些都是设计一个生产级 Agent 必须解决的问题。

关于限流的提示：Anthropic API 有速率限制（Rate Limit），免费层尤其严格。如果你频繁运行 Agent，可能会遇到 429 错误。一个简单的应对方案是加入指数退避（Exponential Backoff）：检测到 429 后等待 2^n 秒再重试（n 是重试次数）。更完整的做法可以参考 anthropic Python SDK 中内置的重试逻辑——它默认已经实现了自动重试，但在高并发场景下你仍然需要手动管理配额。

本章小结

这一章我们亲手实现了一个最简的 Agent Loop——50 行代码，一个 while 循环，两个 API 调用方向（模型 + 工具），一个 stop_reason 的 if-else 判断。你已经写出了一个真正的 AI Coding Agent 的核心骨架。

v0 看起来简单，但它证明了核心洞察：Agent Loop 本身并不复杂，复杂的是围绕它构建的工程系统。50 行代码就能跑通 Agent Loop，但 Claude Code 的 60 万行代码都加在了哪里？下一章我们拆解顶层架构，看看那些加出来的代码到底在解决什么问题。
