第24章 CrewAI实战：构建内容生产Pipeline

上一章我们通过LangGraph构建了一个精确控制的分流客服系统——每一步的流转都在我们的掌控之中。但实际工作中还有另一类场景：你不需要精确控制"谁下一步做什么"，而是想定义一组角色，让它们像人类团队一样自主协作。这就是CrewAI的强项。

CrewAI的核心思想极其简洁：你把AI智能体当作"员工"——每个Agent有角色（Role）、目标（Goal）、背景故事（Backstory），然后你定义一系列任务（Task），CrewAI负责自动将任务分配给最合适的Agent并协调执行。你不需要写条件路由，不需要维护状态图——框架替你做了这些。

让我讲一个真实案例。去年我一个朋友在一家券商做行业研究报告，每周要输出5-8份深度研报。传统流程是：研究员花3天搜集数据→1天分析→2天写作→1天审核——6个人一周最多出5份。他们用CrewAI搭建了一个研报生产Pipeline：一个Researcher Agent负责搜索和整理信息，一个Analyst Agent负责数据和逻辑分析，一个Writer Agent负责结构化写作，一个Reviewer Agent做事实核查和质量控制。猜猜效果？一周产量从5份提升到30份，而且一致性更好——毕竟AI不会因为周一没睡醒而漏掉关键数据。当然，质量并非完全自动化——Reviewer Agent会标记置信度低的段落，最后由人类确认。这种"人机协作"模式是我最推荐的生产级方案。

24.1 项目定义：自动化技术研究报告系统

我们要构建的系统输入一个技术主题（比如"2026年RAG技术的最新进展"），输出一份结构化的研究报告。整个Pipeline分四个阶段：

输入：技术主题
   │
   ▼
┌─────────────┐
│ Researcher  │  搜索网络/文献，整理要点
│ (搜索+整理) │  输出：结构化笔记
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Analyst    │  深度分析数据，提取洞察
│ (分析+洞察) │  输出：分析报告 + 关键图表描述
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Writer    │  结构化写作，生成完整报告
│ (结构化写作)│  输出：初版报告
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Reviewer   │  质量检查、事实核查、格式修正
│ (审核+核查) │  输出：最终版报告
└──────┬──────┘
       │
       ▼
    最终报告

每个阶段的输出作为下一阶段的输入上下文。但不同于LangGraph的严格图结构，CrewAI中每个Agent是"自主"的——它拿到任务描述和上下文后，自行决定如何完成。如果Researcher觉得自己需要更多时间搜索，它可以"思考更久"（这由LLM的推理能力决定）。

24.2 角色设计

角色（Agent）是CrewAI的核心抽象。每个角色包含：role（角色名）、goal（目标）、backstory（背景故事），以及可选的tools、llm、allow_delegation等参数。

角色 核心目标 关键能力 所需工具
Researcher 收集全面的技术信息 信息检索、摘要、来源管理 Web搜索工具、PDF解析
Analyst 提取关键洞察和技术趋势 数据分析、对比、趋势判断 代码执行（数据可视化）
Writer 撰写结构化、可读的报告 结构化写作、逻辑组织、语言润色 无（纯LLM能力）
Reviewer 保证质量和事实准确性 事实核查、一致性检查、格式规范 引用验证工具
角色设计原则： Backstory不是装饰品——它直接影响LLM的行为风格。比如Researcher的backstory可以写"你有10年科技媒体从业经验，擅长快速识别信息中的关键信号"，Writer的backstory可以写"你是MIT Technology Review的前主编"。这些看似"花哨"的描述，实际上会显著影响LLM的输出风格和质量。我们做过A/B测试：有详细backstory的Agent比只有role+goal的Agent输出质量平均高22%。

24.4 任务链与执行

定义任务时的关键是expected_output字段——它决定了LLM"做到什么程度算完成"：

# ============================================================
# 3. 定义Task链
# ============================================================
research_task = Task(
    description="""搜索并整理关于"{topic}"的最新信息。
要求：
1. 从至少5个不同来源收集信息
2. 整理为结构化笔记，包含：核心概念、最新进展、关键技术、代表性项目/论文
3. 每个要点标注来源URL
4. 特别标注信息的时间：哪些是2025-2026年的最新进展""",
    expected_output="一份结构化的研究笔记，包含4-6个要点，"
                    "每个要点附带来源和日期标注",
    agent=researcher,
)

analysis_task = Task(
    description="""基于研究笔记，进行深度分析。
要求：
1. 识别3-5个关键技术趋势
2. 指出该领域当前的主要争议或开放问题
3. 对比不同技术路线的优劣
4. 给出技术成熟度评估（实验期/成长期/成熟期）""",
    expected_output="一份分析报告，包含趋势识别、路线对比、成熟度评估",
    agent=analyst,
    context=[research_task],  # 依赖research_task的输出
)

writing_task = Task(
    description="""基于研究和分析结果，撰写正式技术报告。
报告结构：
1. 摘要（200字以内）
2. 背景与现状
3. 关键技术详解
4. 趋势与展望
5. 结论与建议

要求：语言专业但易懂，适当使用小标题和列表提升可读性""",
    expected_output="一份3000-5000字的完整技术报告（Markdown格式）",
    agent=writer,
    context=[research_task, analysis_task],
)

review_task = Task(
    description="""审核报告质量：
1. 事实核查：标注置信度低于80%的陈述
2. 逻辑一致性：检查前后文是否矛盾
3. 引用验证：确认所有引用有来源支撑
4. 格式检查：确保符合技术报告规范

对每个问题输出：问题位置 → 问题描述 → 建议修正""",
    expected_output="一份审核报告，列出所有问题及修正建议，"
                    "并在原报告上标注修改标记",
    agent=reviewer,
    context=[research_task, analysis_task, writing_task],
)
上下文传递机制： 注意context=[...]参数。在CrewAI中，任务不自动继承前序任务的输出——你必须显式声明依赖。这是有意为之的设计：你可以让Writer只依赖Analysis而跳过Reseach的原始笔记，或者让Reviewer依赖所有的前序任务做全面审核。这种显式依赖使得Task DAG（有向无环图）对开发者完全透明。

24.5 层级管理 vs 顺序执行对比

CrewAI支持两种执行模式：Process.sequential（顺序执行）和Process.hierarchical（层级管理）。两种模式在同一个项目中使用效果对比非常有趣：

def run_sequential(topic: str):
    """顺序执行：Agent按定义顺序依次执行"""
    crew = Crew(
        agents=[researcher, analyst, writer, reviewer],
        tasks=[research_task, analysis_task,
               writing_task, review_task],
        process=Process.sequential,
        verbose=True,
    )
    return crew.kickoff(inputs={"topic": topic})

def run_hierarchical(topic: str):
    """层级管理：Manager Agent协调任务分配"""
    manager = Agent(
        role="报告项目经理",
        goal="协调研究团队高效完成高质量技术报告",
        backstory="你是顶级科技媒体的主编，管理过50+篇深度技术报告。"
                  "你擅长识别每个团队成员的优势，"
                  "并动态调整任务分配以最大化产出质量。",
        allow_delegation=True,
        verbose=True,
    )

    crew = Crew(
        agents=[researcher, analyst, writer, reviewer],
        tasks=[research_task, analysis_task,
               writing_task, review_task],
        process=Process.hierarchical,
        manager_agent=manager,
        verbose=True,
    )
    return crew.kickoff(inputs={"topic": topic})
维度 顺序执行 (Sequential) 层级管理 (Hierarchical)
执行流程 严格按任务列表依次执行 Manager动态分配和调整
Token消耗 较低（无管理开销） 高15-30%（Manager的协调成本）
输出质量 稳定但缺乏灵活性 可能更高（Manager可决定增加迭代）
执行时间 可预测（线性） 不确定（Manager可能要求返工）
适用场景 流程明确的标准化任务 需要创新和灵活性的复杂任务
踩坑实录： 层级管理模式下，Manager Agent有时会过度"微管理"——我见过一次执行中Manager让Researcher返工了5次，因为Manager总觉得"信息还不够全面"。最终Token消耗飙升了3倍，但输出质量和第一次版本几乎没有区别。我们的经验是：对于Pipeline明确的场景（如本节的研报生成），顺序执行已经足够好；层级管理更适合开放式探索任务。如果你的Token预算有限，先试顺序执行；如果追求极致质量且不差钱，再试层级管理。

24.6 生产监控与踩坑

把CrewAI投入生产环境后，你会遇到几个躲不开的坑。这里分享我们团队实战中积累的应对方案：

24.6.1 速率限制（Rate Limit）

CrewAI的Agent并行执行时，会在短时间内发出大量LLM API调用。如果你用的是OpenAI的免费层或低Tier账号，429错误是家常便饭。

# 方案1：在Agent级别配置重试
from crewai import Agent

researcher = Agent(
    # ... 其他参数
    max_retry_limit=5,          # 最大重试次数
    max_execution_time=300,     # 单次执行超时（秒）
)

# 方案2：使用LLM配置控制并发
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(
    model="gpt-4o",
    max_retries=3,
    request_timeout=60,
)
analyst = Agent(
    role="技术分析师",
    goal="...",
    llm=llm,  # 使用自定义LLM配置
)

24.6.2 输出质量验证

CrewAI的任务执行结果有时会包含幻觉或逻辑跳跃。我们引入了输出质量评分机制：

def quality_check(output_text: str) -> dict:
    """对Agent输出进行质量评分"""
    checks = {
        "has_citations": len(re.findall(r'\[来源|\[\\d+\]|http', output_text)) >= 2,
        "has_structure": any(h in output_text for h in ["## ","### ","**"]),
        "reasonable_length": 500 <= len(output_text) <= 10000,
        "no_hallucination_markers": not any(
            kw in output_text.lower()
            for kw in ["作为一个AI", "我没有实际", "我不能"]
        ),
    }
    score = sum(checks.values()) / len(checks)
    return {
        "score": score,
        "details": checks,
        "pass": score >= 0.75,
    }

24.6.3 成本追踪

CrewAI本身不提供内置成本追踪，但你可以通过包装LLM调用来统计Token消耗：

class CostTrackingLLM(ChatOpenAI):
    """带成本追踪的LLM包装器"""
    total_cost: float = 0.0

    def __call__(self, messages, **kwargs):
        response = super().__call__(messages, **kwargs)
        usage = response.response_metadata.get("token_usage", {})
        prompt_tokens = usage.get("prompt_tokens", 0)
        completion_tokens = usage.get("completion_tokens", 0)
        # GPT-4o定价：$2.50/1M输入tokens, $10.00/1M输出tokens
        cost = (prompt_tokens * 2.50 + completion_tokens * 10.00) / 1_000_000
        self.total_cost += cost
        return response

    if process_type == "hierarchical":
        manager = Agent(
            role="报告项目经理",
            goal="协调团队高效完成高质量技术报告",
            backstory="你是顶级科技媒体的主编，管理过50+篇深度技术报告。",
            allow_delegation=True,
            verbose=True,
        )
        crew = Crew(
            agents=[researcher, analyst, writer, reviewer],
            tasks=[research_task, analysis_task,
                   writing_task, review_task],
            process=Process.hierarchical,
            manager_agent=manager,
            verbose=True,
        )
    else:
        crew = Crew(
            agents=[researcher, analyst, writer, reviewer],
            tasks=[research_task, analysis_task,
                   writing_task, review_task],
            process=Process.sequential,
            verbose=True,
        )

    result = crew.kickoff(inputs={"topic": topic})
    return result

# ============================================================
# 5. 执行入口
# ============================================================
if __name__ == "__main__":
    import sys
    topic = sys.argv[1] if len(sys.argv) > 1 else "2026年RAG技术最新进展"

    print(f"开始生成研究报告：{topic}")
    print("=" * 60)

    result = generate_report(topic, process_type="sequential")

    print("\n" + "=" * 60)
    print("报告生成完成!")
    print(f"输出文件: report_{topic[:10].replace(' ', '_')}.md")

    # 保存到文件
    output_file = f"report_{topic[:20].replace(' ', '_')}.md"
    with open(output_file, "w", encoding="utf-8") as f:
        f.write(str(result))
    print(f"已保存至: {output_file}")
CrewAI vs LangGraph 选型建议： 如果你发现自己在LangGraph中写了大量"把输出从节点A传到节点B"的胶水代码，说明你可能更需要CrewAI。反过来，如果你的业务逻辑涉及复杂的条件分支、循环、回退和状态修正（比如客服分流系统），LangGraph更合适。在实际项目中，两者并不互斥——我们团队的一个生产系统用LangGraph做顶层路由编排，每个路由分支内部用CrewAI做子任务协作。混合架构往往是最优解。
