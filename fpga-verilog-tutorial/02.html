第2章 数字电路基础回顾

大家好，我是你们的讲师。在开始写 Verilog 之前，我们先花一章回顾数字电路的核心基础。为什么？因为Verilog 本质上就是"用文本描述硬件"——如果你不理解综合器最终会生成什么样的电路，那么你写的代码就是空中楼阁。可以说，数字电路直觉就是你写 Verilog 的底气。

这一章会快速回顾组合逻辑和时序逻辑的根基，重点放在那些与 FPGA 设计最相关的知识点上。如果你已经对这些概念了如指掌，可以快速浏览；如果你有点生疏，建议一边读一边在纸上画门电路——硬件思维是"画"出来的，不是"背"出来的。

2.1 组合逻辑基础

组合逻辑的特点是输出仅取决于当前输入，没有记忆功能。输入一变，经过一段传播延迟后，输出就会稳定到新的值。

2.1.1 基本门电路

所有数字电路都是由三个基本门组成的：与门（AND）、或门（OR）、非门（NOT）。从这三个门可以推导出与非（NAND）、或非（NOR）、异或（XOR）、同或（XNOR）。实际上，FPGA 内部连这些基本门都没有——FPGA 的 LUT（查找表）是用 SRAM 存储真值表，用多路选择器实现的"万能"逻辑单元。一个 6-LUT 就是一个 2^6 × 1 的 ROM，能实现任意 6 输入 1 输出的组合逻辑。但了解基本门电路有助于我们理解 LUT 的输入限制：一个 6-LUT 最多处理 6 个输入，超过就需要级联。

2.1.2 真值表与逻辑表达式

给定一个功能，先列真值表，再写逻辑表达式，这是数字电路的"翻译"过程。标准形式有两种：SOP（积之和，与或式）和POS（和之积，或与式）。在 FPGA 中，SOP 更自然，因为 LUT 本质上就是一个"先与后或"的超大表格——每个 LUT 的输入相当于"与"项，多路选择器相当于"或"。

2.1.3 卡诺图化简

用卡诺图合并相邻最小项，可以化简逻辑表达式，减少 LUT 的使用数量。在手工设计中，卡诺图是最小化逻辑的工具；但在现代综合器中，Vivado 的并行综合引擎会在毫秒级完成比卡诺图复杂百倍的逻辑优化（如布尔代数重写、资源共享、寄存器重定时等）。所以初学者了解卡诺图的原理即可，不必纠结于五变量以上的化简。

2.1.4 常用中规模组合器件

在 FPGA 设计中，以下组合器件几乎每天都在用：

  • MUX（多路选择器）——FPGA 的 LUT 本质上就是一个 2^n:1 的 MUX。综合器会把多路选择映射到 LUT 结构上。
  • 译码器——将 n 位输入映射到 2^n 位输出，例如 3-8 译码器用于地址译码。
  • 编码器——将 2^n 位输入压缩为 n 位输出。优先编码器（Priority Encoder）是设计中断控制器的核心。
器件输入→输出FPGA 映射方式典型场景

MUX 4:14 数据 + 2 选择 → 11 个 LUT（6 输入以下）或 2 个 LUT 级联寄存器文件读端口

2.2 时序逻辑基础

时序逻辑引入了时间维度——输出不仅取决于当前输入，还取决于电路之前的状态。实现"记忆"的基本单元就是触发器（Flip-Flop）。

2.2.1 锁存器 vs 触发器

这是初学者最容易混淆的概念：

  • 锁存器（Latch）——电平触发，在使能信号有效期间，输出随输入变化（透明模式）。用 Verilog 的 always @(*) 或 always @(enable or d) 且缺少 else 分支时会意外产生。
  • 触发器（Flip-Flop）——边沿触发，只在时钟沿采样输入，其他时间保持。用 Verilog 的 always @(posedge clk) 描述。

在 FPGA 中，每个逻辑单元（Slice）都集成了 LUT + FF，FF 是标配而 Latch 是额外支持的——Xilinx 7 系列的 Slice 中每个 FF 都可以配置为 Latch，但会占用额外的资源，且时序分析更复杂。因此在 FPGA 设计中应主动避免 Latch，除非你明确知道自己在做什么（比如高速异步 FIFO 的读操作）。

Latch 的隐藏成本：很多初学者以为 "Latch 也是一个触发器嘛，能用就行"——错了。Latch 在综合后报告中不会明确报出来，但它会导致额外的 LUT 占用、提高静态时序分析的复杂度、并且在 STA 中需要 check timing arc 而不是简单的 setup/hold。在国产 GOWIN 或安路 FPGA 上，Latch 的综合支持甚至不完整，可能导致布局布线失败。所以：写 always 块，先想清楚是 combo（@(*)）还是 seq（@(posedge clk)），不要混淆。

2.2.2 D 触发器与异步复位

D 触发器是 FPGA 设计中使用最频繁的时序单元。一个带异步复位的 D 触发器代码如下：

// D flip-flop with active-low asynchronous reset
module dff_async_rst (
    input  wire       clk,
    input  wire       rst_n,  // 异步复位，低有效
    input  wire       d,
    output reg        q
);

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n)
            q <= 1'b0;        // 复位时清零
        else
            q <= d;            // 时钟上升沿采样
    end

endmodule

注意两点：第一，异步复位必须在敏感列表中用 negedge rst_n 或 posedge rst 列出；第二，同步复位只出现在 always @(posedge clk) 的 if 条件中，敏感列表只有 clk。两者的综合结果不同：异步复位会映射到 FF 的 CLR 引脚，同步复位会消耗额外的组合逻辑。

2.3 建立时间与保持时间

这是面试必考题，也是芯片设计中最核心的物理概念。不理解 setup/hold，你写的 100MHz 电路可能实际只能跑 50MHz——甚至完全不工作。

2.3.1 定义

  • 建立时间（Setup Time, Tsu）：在时钟有效沿到来之前，数据输入必须保持稳定的最短时间。
  • 保持时间（Hold Time, Th）：在时钟有效沿到来之后，数据输入必须保持稳定的最短时间。

为什么需要这两个时间？因为触发器的内部是一个"模拟电路"——内部锁存级需要通过电压差来判决。如果数据在时钟沿附近变化，内部节点可能无法在判决时间内稳定到确定的逻辑电平，从而进入亚稳态（Metastability）。

2.3.2 亚稳态

亚稳态是指触发器输出既不是 0 也不是 1，而是介于两者之间的电压值，持续一段不确定时间后才会"坍塌"到某个稳态。亚稳态的危害在于：

  • 后续级联的逻辑可能把亚稳态电压理解为不同的逻辑值（前级输出为 1.2V，后级有的 FF 读成 0，有的读成 1）——这叫亚稳态传播。
  • MTBF（平均无故障时间）与时钟频率、工艺、电压相关。比如在 200MHz 下，不加同步器的跨时钟域信号的 MTBF 可能只有几微秒。

2.3.3 Setup Slack 与 Hold Slack

时序分析工具（Vivado、Quartus、Yosys + STA）会计算每条路径的 slack：

  • Setup Slack = Tclk - Tco - Tlogic - Tsu + Tskew——正值表示时序裕量足够。如果为负，说明路径太长，需要加流水线或优化逻辑。
  • Hold Slack = Tco + Tlogic - Th - Tskew——如果为负，说明数据变化太快，需要加延迟。
关键认知：setup 是 频率相关的（降低时钟频率可以解决 setup 违例），而 hold 是 频率无关的（即使降到 1Hz，hold 违例依旧存在，必须通过修改电路解决）。在 FPGA 设计中，hold 违例相对少见（因为内部布线延迟足够大），但跨时钟域信号处理不当会引起 hold 问题。Vivado 默认的时序约束只 check setup，需要在约束文件中加 set_false_path 或 set_max_delay 处理跨时钟域路径。

2.4 同步电路设计原则

FPGA 设计的黄金法则是：所有触发器都在同一时钟边沿工作，所有异步输入都先经过同步器。这被称为"同步设计方法论"。

2.4.1 全局时钟网络

FPGA 内部有专用的 全局时钟网络（Global Clock Tree），由专用引脚（如 Xilinx 的 MRCC/SRCC）、PLL/DLL 和时钟布线资源组成。通过全局时钟网络传输的时钟信号：

  • 到达每个 FF 的时钟偏斜（skew）极小（通常在几十皮秒量级）
  • 占空比保持完好
  • 抖动（jitter）可控

一定要在代码中将时钟信号接入全局时钟缓冲区（如 Vivado 中的 BUFG 原语），而不是用普通布线资源。Yosys 综合时也可以用 clkbuf_opt 策略自动插入时钟缓冲。

2.4.2 为什么避免门控时钟和行波计数器

很多从 MCU 转来的工程师习惯用"门控时钟"——用一个使能信号 AND 到时钟线上来产生慢速时钟：

// 错误写法：门控时钟
assign gated_clk = clk & enable;  // 不要这样做！
always @(posedge gated_clk) begin
    // ... 逻辑
end

问题在哪？gated_clk 没有经过全局时钟网络，延迟和 skew 不可控；enable 信号的变化会在时钟沿附近产生毛刺，导致 FF 误触发。正确做法是使用时钟使能：

// 正确写法：时钟使能
always @(posedge clk) begin
    if (enable) begin
        // ... 逻辑
    end
end

同理，行波计数器（Ripple Counter）——前一级的输出作为后一级的时钟——也应该避免。每个触发器的工作时钟都不同，skew 无法保证，且综合工具难以做时序分析。永远只用一个主时钟配合使能信号来做分频。

2.4.3 同步复位的正确写法

// 同步复位示例
always @(posedge clk) begin
    if (rst_n) begin  // 同步复位
        q <= 1'b0;
    end else begin
        q <= d;
    end
end

同步复位的优点是电路更干净（复位不额外占用 FF 的 CLR 引脚），但复位信号必须满足建立时间——如果复位释放离时钟沿太近，会导致亚稳态复位。这就是复位释放时序问题，也是 FPGA 设计中常见的坑。

2.6 一个真实的踩坑故事：门控时钟的惨痛教训

2018 年，我参与一个工业相机项目，用 Xilinx Artix-7 做图像传感器驱动。FPGA 需要从 MT9V034 传感器接收 27MHz 的像素时钟和数据，做简单处理后通过 USB 3.0 上传。Bayer 格式一帧 1280×1024，要求 30fps，总线带宽还算宽裕。

项目进展顺利，Vivado 仿真全部通过——像素数据正确，行场同步信号时序完美。板上验证时，好玩的来了：前 50 帧完全正确，第 51 帧突然图像顶部出现了一行错位的像素。然后可能正常 200 帧，又出现一行。毫无规律。

用 Vivado 的 ILA（集成逻辑分析仪）抓了将近一个星期，最后定位到问题——我在代码里用了门控时钟来选择读传感器的速度。代码大致是这样的：

// （这是有问题的代码——仅供教学参考）
assign slow_clk = pixel_clk & slow_enable;

always @(posedge slow_clk) begin
    pixel_count <= pixel_count + 1;
end

slow_enable 信号是另一个模块组合逻辑生成的，它变化时正好落在 pixel_clk 的上升沿附近——产生了半个周期的毛刺。这个毛刺被 slow_clk 捕获，计数器多计了一次，导致像素地址错位。仿真为什么通过？因为仿真器的毛刺是理想化的——组合逻辑的变化发生在 delta 延迟之后，不会产生"模拟"性质的毛刺边沿。

修复很简单：改成时钟使能模式，所有触发器都用同一个 pixel_clk。

// 修复后
always @(posedge pixel_clk) begin
    if (slow_enable) begin
        pixel_count <= pixel_count + 1;
    end
end

这个教训让我养成了一个习惯：任何设计中，永远不要用组合逻辑产生时钟。时钟只能用 PLL 生成，然后通过全局时钟网络分配。从此我再也没被门控时钟坑过。
