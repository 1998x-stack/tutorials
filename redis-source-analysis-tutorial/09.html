第9章：String 与 Hash 命令实现

大家好，我是你们的讲师。上一章我们拆解了 redisObject，搞清楚了 Redis 如何用统一的对象模型来包装各种底层数据结构。现在问题来了：当你输入 SET key value EX 10 或者 HSET user:1001 name "Alice" 时，这些命令背后到底走了哪些代码路径？从客户端发出命令到 Redis 返回结果，中间经历了什么样的处理链条？这一章我们就从最常用的 String 和 Hash 命令入手，深入到 t_string.c 和 t_hash.c 的源码细节中一探究竟。

String 命令族

setGenericCommand —— SET 的统一入口

所有与 SET 相关的功能（包括 SET、SETEX、PSETEX、SETNX）最终都汇聚到 setGenericCommand 这个核心函数。它的签名为：

/* t_string.c */
void setGenericCommand(client *c, int flags, robj *key, robj *val,
                       robj *expire, int unit, robj *ok_reply, robj *abort_reply) {
    long long milliseconds = 0; /* 过期时间，毫秒 */
    int ret;

    /* 1. 解析过期时间 */
    if (expire) {
        if (getLongFromObjectOrReply(c, expire, &milliseconds, NULL) != C_OK)
            return;
        if (milliseconds <= 0) {
            addReplyError(c, "invalid expire time");
            return;
        }
        if (unit == UNIT_SECONDS) milliseconds *= 1000;
    }

    /* 2. 检查 SET NX/XX 语义 */
    if (flags & OBJ_SET_NX) {
        ret = lookupKeyWrite(c->db, key);
        if (ret != NULL) {
            addReply(c, abort_reply ? abort_reply : shared.null[c->resp]);
            return;
        }
    }
    if (flags & OBJ_SET_XX) {
        ret = lookupKeyWrite(c->db, key);
        if (ret == NULL) {
            addReply(c, abort_reply ? abort_reply : shared.null[c->resp]);
            return;
        }
    }

    /* 3. 执行实际的键值写入 */
    setKey(c->db, key, val, LOOKUP_NOEXPIRE);

    /* 4. 设置过期 */
    if (expire) setExpire(c, c->db, key, milliseconds);

    /* 5. 通知 watch 客户端、发布键空间通知 */
    signalModifiedKey(c->db, key);
    notifyKeyspaceEvent(NOTIFY_STRING, "set", key, c->db->id);

    /* 6. 当使用 SET ... GET 时返回旧值 */
    if (flags & OBJ_SET_GET) {
        addReplyBulk(c, ret ? ret : shared.null[c->resp]);
    } else {
        addReply(c, ok_reply ? ok_reply : shared.ok);
    }
}
源码阅读技巧：setGenericCommand 是典型的"所有分支汇聚到一个主干"模式。不管是 SET 还是 SETEX，最终都调用 setKey() 写入数据库，然后用 setExpire() 设置过期时间。区别只在于参数解析层。

而 setCommand 作为命令入口，主要职责是解析 SET key value [NX | XX] [EX seconds | PX milliseconds] [GET] 这一串复杂的可选参数：

/* t_string.c */
void setCommand(client *c) {
    robj *expire = NULL;
    int flags = OBJ_SET_NO_FLAGS;
    int unit = UNIT_SECONDS;

    /* 从 argv[3] 开始逐个解析可选参数 */
    for (int j = 3; j < c->argc; j++) {
        char *opt = c->argv[j]->ptr;
        if (!strcasecmp(opt, "nx")) {
            flags |= OBJ_SET_NX;
        } else if (!strcasecmp(opt, "xx")) {
            flags |= OBJ_SET_XX;
        } else if (!strcasecmp(opt, "ex")) {
            flags |= OBJ_SET_EX;
            expire = c->argv[j+1];
            unit = UNIT_SECONDS;
            j++;
        } else if (!strcasecmp(opt, "px")) {
            flags |= OBJ_SET_PX;
            expire = c->argv[j+1];
            unit = UNIT_MILLISECONDS;
            j++;
        } else if (!strcasecmp(opt, "get")) {
            flags |= OBJ_SET_GET;
        }
    }

    /* 参数校验后调用通用入口 */
    setGenericCommand(c, flags, c->argv[1], c->argv[2],
                      expire, unit, NULL, NULL);
}

编码选择逻辑

在 setKey() 内部，Redis 最终会调用 createStringObject() 来创建值的 redisObject。创建时依据字符串长度选择编码：

  • 如果可以解析为 long long，使用 OBJ_ENCODING_INT，值存储在 ptr 字段中，零内存分配。
  • 如果长度 ≤ 44 字节，使用 OBJ_ENCODING_EMBSTR，一次 malloc 分配 robj + sdshdr + 数据。
  • 如果长度 > 44 字节，使用 OBJ_ENCODING_RAW，两次 malloc 分别分配 robj 和 sdshdr。
编码 条件 内存布局 malloc 次数
OBJ_ENCODING_INT 可解析为 long long 值直接存在 robj.ptr 中 0 次（创建时）
OBJ_ENCODING_EMBSTR 长度 ≤ 44 robj + sdshdr + 数据 连续 1 次
OBJ_ENCODING_RAW 长度 > 44 robj 与 sdshdr 分离 2 次

INCR/DECR 原子操作

INCR/DECR/INCRBY 是 Redis 最常用的计数操作，它们的关键在于原子性——在单线程的事件循环中，这些操作的执行不会被其他客户端命令中断。

/* t_string.c */
void incrDecrCommand(client *c, long long value) {
    long long incr;
    robj *o, *new;

    /* 1. 查找现有键 */
    o = lookupKeyWrite(c->db, c->argv[1]);
    if (o) {
        /* 2. 检查类型必须是 String */
        if (checkType(c, o, OBJ_STRING)) return;

        /* 3. 尝试解析为 long long */
        if (getLongLongFromObjectOrReply(c, o, &incr, NULL) != C_OK)
            return;

        /* 4. 执行运算（带溢出检测） */
        incr += value;
        if ((value > 0 && incr < value) ||
            (value < 0 && incr > value)) {
            addReplyError(c, "increment or decrement would overflow");
            return;
        }

        /* 5. 尝试共享整数对象 */
        new = createStringObjectFromLongLongWithOptions(incr);
        setKey(c->db, c->argv[1], new, LOOKUP_NOEXPIRE);
    } else {
        /* 键不存在，使用 value 作为初始值 */
        new = createStringObjectFromLongLongWithOptions(value);
        setKey(c->db, c->argv[1], new, LOOKUP_NOEXPIRE);
    }
}
溢出保护：Redis 7.2 的 INCR 会检测溢出。当 incr + value 结果回绕时（即 incr < 0 && value < 0 && incr + value > incr 这种场景），Redis 不会静默回绕，而是返回溢出错误。这个保护在 Redis 6.2 后才引入，早期版本使用 C 语言的带符号整数溢出（undefined behavior），实际行为依赖于编译器和平台。

溢出保护的检测逻辑利用了带符号整数溢出的特点：当两个正数相加为负数，或两个负数相加为正数时，说明发生了溢出。这是 C 语言中检测带符号溢出的标准手法。

Hash 命令族

hsetCommand 与 hashTypeSet

Hash 命令的核心抽象层在 hashTypeSet / hashTypeGet / hashTypeDelete 等函数中。无论底层是 listpack 还是 dict，上层命令都通过这一组接口操作。

/* t_hash.c */
void hsetCommand(client *c) {
    robj *o;
    int updated = 0;

    /* 1. 查找或创建键 */
    o = lookupKeyWrite(c->db, c->argv[1]);
    if (o == NULL) {
        /* 创建空的 listpack 编码的 Hash */
        o = createHashObject();
        dbAdd(c->db, c->argv[1], o);
    } else {
        if (checkType(c, o, OBJ_HASH)) return;
    }

    /* 2. 支持 HSET field value [field value ...] 多字段模式 */
    for (int j = 2; j < c->argc; j += 2) {
        updated += hashTypeSet(o, c->argv[j]->ptr, c->argv[j+1]->ptr, HASH_SET_TAKE_FIELD | HASH_SET_TAKE_VALUE);
    }

    /* 3. 返回新增字段数量（非更新数量） */
    addReplyLongLong(c, updated);
}

int hashTypeSet(robj *o, sds field, sds value, int flags) {
    int update = 0;

    if (o->encoding == OBJ_ENCODING_LISTPACK) {
        /* listpack 编码下的插入 */
        update = hashTypeSetListpack(o, field, value, flags);

        /* 检查是否需要转换为 dict */
        if (hashTypeLength(o) > server.hash_max_listpack_entries ||
            sdslen(field) > server.hash_max_listpack_value ||
            sdslen(value) > server.hash_max_listpack_value)
        {
            hashTypeConvert(o, OBJ_ENCODING_HT);
        }
    } else if (o->encoding == OBJ_ENCODING_HT) {
        /* dict 编码下的插入 */
        update = hashTypeSetHT(o, field, value, flags);
    }

    return update;
}

hashTypeSetListpack 在 listpack 中线性查找字段名（因为 listpack 是无序的紧凑存储），如果找到则替换值，否则追加。线性查找在 512 个字段以内的 Hash 中性能是可以接受的，因为 listpack 的字段长度通常很小，内存局部性好。

hashTypeSetHT 则是直接调用 dictAddOrFind + dictSetVal，时间复杂度 O(1)。

hgetCommand 与 hashTypeGet

/* t_hash.c */
void hgetCommand(client *c) {
    robj *o;

    /* 查找键 */
    if ((o = lookupKeyReadOrReply(c, c->argv[1], shared.null[c->resp])) == NULL
        || checkType(c, o, OBJ_HASH)) return;

    /* 调用抽象层获取字段值 */
    hashTypeGetValue(o, c->argv[2]->ptr, &value, NULL);
    if (value)
        addReplyBulk(c, value);
    else
        addReplyNull(c);
}

hashTypeGetValue 根据编码类型自动选择 listpack 或 dict 的查询路径。listpack 下线性扫描，dict 下 O(1) 哈希查找——调用者无需关心底层实现。

hgetallCommand —— 小心 bigkey

HGETALL 返回 Hash 的所有字段和值。它的实现很简单：遍历整个底层存储，逐个字段添加到回复缓冲区。

/* t_hash.c */
void hgetallCommand(client *c) {
    robj *o;
    if ((o = lookupKeyReadOrReply(c, c->argv[1], shared.emptyarray))
        == NULL || checkType(c, o, OBJ_HASH)) return;

    hashTypeIterator *hi = hashTypeInitIterator(o);
    robj *field, *value;
    unsigned long count = 0;
    addReplyArrayLen(c, 2 * hashTypeLength(o));

    while (hashTypeNext(hi) != C_ERR) {
        hashTypeCurrentFromHash(hi, &field, &value);
        addReplyBulk(c, field);
        addReplyBulk(c, value);
        count++;
    }
    hashTypeReleaseIterator(hi);
}
踩坑实录：有一个线上案例，某业务方把用户的所有行为标签存入一个 Hash，Hash 中字段数超过 10 万。每次查询时使用 HGETALL 获取全量数据，导致 Redis 单线程阻塞 5 秒以上，所有其他请求排队等待。我们建议改用 HSCAN 分批迭代（每次返回 100 个字段），同时推动业务方拆分 Redis 键——每个用户按标签类别拆成多个小的 Hash。改造后 P99 延迟从 3.2 秒降到了 8 毫秒。HGETALL 本身是一个 O(N) 操作，当 N 很大时，它阻塞的不只是自己，而是整个 Redis 事件循环中的所有客户端。

Hash Field Expiration (HFE) —— Redis 8.x 新特性

传统上 Redis 的过期只支持到 key 级别。从 Redis 8.0 开始，Hash 支持每个字段独立的 TTL（HEXPIRE / HPEXPIRE / HTTL 等命令）。这个特性在源码层面涉及较大的改动：

  • 每个 Hash 内部新增一个 dict *metadata，用于存储字段级别的过期时间。
  • HEXPIRE key field seconds 在元数据字典中记录该字段的绝对过期时间戳。
  • HGET / HSCAN 等读取命令在返回前会检查字段是否已过期，过期则视为不存在。
  • 过期字段的清理策略与 key 级过期类似，采用惰性删除 + 定期采样删除。

HFE 的实现难度在于兼容性——已有的 RDB/AOF 格式、主从复制协议、Cluster 迁移都需要为字段级过期增加新的序列化格式。这也是为什么这个特性从 Redis 7.x 就开始规划，直到 8.x 才正式落地。

批量操作优化

MGET 和 MSET 是批量读写多个 String 键的命令。它们的实现非常直接——遍历 key 数组，对每个 key 执行 lookupKeyRead / setKey，然后一次性回复结果。

/* t_string.c */
void mgetCommand(client *c) {
    int j;
    addReplyArrayLen(c, c->argc - 1);
    for (j = 1; j < c->argc; j++) {
        robj *o = lookupKeyRead(c->db, c->argv[j]);
        if (o == NULL) {
            addReplyNull(c);
        } else {
            if (o->type != OBJ_STRING) {
                addReplyNull(c);
            } else {
                addReplyBulk(c, o);
            }
        }
    }
}
Pipeline 与 MGET 的区别：Pipeline 是将多个命令打包一次性发送，但服务器仍然逐个执行每个命令。MGET/MSET 是一个命令内部对多个 key 进行循环操作，省去了命令解析和协议交互的开销。在批量读取 1000 个 key 时，MGET 比 Pipeline 的 1000 个独立 GET 快大约 3-5 倍。对于 Hash 的多字段操作，HMSET 和 HSET 在 Redis 4.0+ 中已经合并——HSET 支持多字段语法，内部循环调用 hashTypeSet 即可。
操作模式 网络往返 命令解析次数 适用场景
N 次独立 SET N 次 RTT N 次 少量随机写入
Pipeline N 次 SET 1 次 RTT N 次 批量写入，但不同命令
MSET 1 次 RTT 1 次 批量写入相同命令
HSET f1 v1 f2 v2 ... 1 次 RTT 1 次 Hash 批量字段写入

String 和 Hash 是"静态"的数据类型，它们的操作要么读写，要么原子增减，不涉及等待。但下一章我们要讨论的 List 则完全不同——BLPOP/BRPOP 引入了一个全新的维度：阻塞等待。当一个列表为空时，Redis 可以让客户端挂起等待，直到有数据到来。这是事件循环与命令实现之间最精妙的配合之一。
