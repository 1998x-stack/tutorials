第17章 锁机制与并发

大家好，我是你们的讲师。前面两章我们分别介绍了 Rollback Journal 和 WAL 两种事务模型。但无论是 journal 还是 WAL，它们都依赖一个更底层的基础设施——锁机制。没有可靠的锁，事务隔离就是一句空话，并发控制也无从谈起。这一章我们就深入 SQLite 的锁系统，看看这个只有 15 万行代码的嵌入式数据库是怎么在文件系统之上实现精细的并发控制的。

SQLite 的锁机制有一点和大型数据库非常不同：它没有中心化的锁管理器，没有死锁检测器，也没有复杂的锁升级协议。它依赖的是操作系统提供的文件锁（POSIX 的 fcntl 或 Windows 的 LockFileEx），在这之上实现了一个五级锁状态机。你可能会觉得这听起来很简陋——但正是这种"从简"的设计，让 SQLite 在数十亿设备上稳定运行了二十多年。

文件锁的五级状态

SQLite 对数据库文件定义了五级锁状态，构成了一个严格的升级链：

锁级别名称持有者数量允许的操作是否阻塞其他操作
0NONE任意未连接或空闲—
1SHARED多个读取数据库不阻塞其他 SHARED，阻塞 EXCLUSIVE
2RESERVED一个预留写权限，可以修改缓存中的页允许其他 SHARED 继续，阻塞其他 RESERVED
3PENDING一个等待所有 SHARED 释放，准备升级到 EXCLUSIVE阻塞新的 SHARED 请求
4EXCLUSIVE一个写入数据库文件阻塞所有其他操作

这个状态机的设计非常精巧。我们沿着一条典型的写操作路径来看看它是怎么流转的：

/* 写事务的锁升级路径 */
NONE
  │
  ▼
SHARED     ← BEGIN TRANSACTION（读操作也需要）
  │
  ▼
RESERVED   ← 第一次修改数据（UPDATE/INSERT/DELETE）
  │         此时其他读者仍然可以读取数据库
  │
  ▼
PENDING    ← 准备提交，不再允许新读者进入
  │         等待所有现有读者释放 SHARED 锁
  │
  ▼
EXCLUSIVE  ← 开始将脏页写回数据库文件
  │         其他任何操作都无法进行
  │
  ▼
NONE       ← 事务结束，释放锁

这里的关键洞察是 RESERVED 锁的设计：它让写事务可以"预留"写入权，同时不阻塞读者。写事务在 RESERVED 状态下可以修改内存中的页缓存（脏页），修改可以积累多个页，直到提交时才升级到 PENDING/EXCLUSIVE。这意味着读操作在写事务的整个执行阶段（可能执行多个 SQL 语句）都不会被阻塞，只有在提交的瞬间才需要短暂的排他访问。

POSIX 咨询锁实现

在 Unix/Linux 平台上，SQLite 使用 fcntl() 系统调用的咨询锁（advisory locking）来实现文件锁。对应的实现在 os_unix.c 文件中。

POSIX 的 fcntl 锁有几个重要的特性：

  • 咨询锁 (advisory) — 不是强制锁。不遵守锁协议的进程仍然可以读写文件。SQLite 依靠所有 SQLite 进程都遵守相同的锁协议来保证正确性。
  • 与进程关联 — 当持有锁的进程退出（无论正常还是异常）时，内核自动释放所有锁。这正是 SQLite 崩溃恢复的基础——崩溃的写者释放了锁，新的会话就可以获得 EXCLUSIVE 锁并执行恢复。
  • 不是与线程关联 — 同一个进程内的多个线程共享文件描述符，所以 fcntl 锁是进程级别的。
/* os_unix.c 中的锁操作核心代码 */
/* 文件: src/os_unix.c */

/*
 * unixLock() — 尝试在文件描述符上获取指定级别的锁
 * eFileLock: 目标锁级别 (SHARED, RESERVED, PENDING, EXCLUSIVE)
 */
static int unixLock(unixFile *pFile, int eFileLock){
    struct flock lock;
    int rc = SQLITE_OK;

    /* 1. 根据目标锁级别设置锁参数 */
    assert( eFileLock <= SHARED || pFile->eFileLock >= SHARED );
    lock.l_type = (eFileLock == SHARED) ? F_RDLCK : F_WRLCK;
    lock.l_whence = SEEK_SET;

    switch( eFileLock ){
        case SHARED:
            lock.l_start = 0;
            lock.l_len = 1;           /* 范围: 字节 0, 用于 SHARED 锁 */
            break;
        case RESERVED:
            lock.l_start = 1;
            lock.l_len = 1;           /* 范围: 字节 1, 用于 RESERVED 锁 */
            break;
        case PENDING:
            lock.l_start = 2;
            lock.l_len = 1;           /* 范围: 字节 2, 用于 PENDING 锁 */
            break;
        case EXCLUSIVE:
            lock.l_start = 3;
            lock.l_len = 1;           /* 范围: 字节 3, 用于 EXCLUSIVE 锁 */
            break;
        default:
            assert(!"Invalid lock level");
    }

    /* 2. 调用 fcntl 设置锁 */
    /* F_SETLK: 非阻塞尝试 */
    /* F_SETLKW: 阻塞等待 */
    if (sqlite3_os_type == SQLITE_OS_UNIX){
        rc = fcntl(pFile->h, F_SETLK, &lock);
        if (rc < 0){
            /* 锁冲突，返回 SQLITE_BUSY */
            return SQLITE_BUSY;
        }
    }

    /* 3. 更新内部状态 */
    pFile->eFileLock = eFileLock;
    return SQLITE_OK;
}

/*
 * unixUnlock() — 释放文件锁，降低锁级别
 */
static int unixUnlock(unixFile *pFile, int eFileLock){
    struct flock lock;
    lock.l_type = F_UNLCK;
    lock.l_whence = SEEK_SET;
    lock.l_start = 0;
    lock.l_len = 4;      /* 解锁所有 4 个字节区域 */

    fcntl(pFile->h, F_SETLK, &lock);
    pFile->eFileLock = eFileLock;
    return SQLITE_OK;
}

一个很有趣的实现细节是：SQLite 用文件中不同的字节范围来表示不同的锁级别。字节 0 用于 SHARED 锁（读锁），字节 1 用于 RESERVED，字节 2 用于 PENDING，字节 3 用于 EXCLUSIVE。通过这种"分区域锁"的方式，同一个 fd 上可以同时持有多个不同区域的锁——这就是为什么一个连接可以同时持有 SHARED 锁（字节 0）和 RESERVED 锁（字节 1）。

💡 如果你想在运行中的系统上观察 SQLite 的锁状态，可以使用 lsof 命令查看数据库文件的文件描述符，或者用 strace -e fcntl 来追踪所有 fcntl 系统调用。我曾经用 strace -p <pid> -e trace=fcntl -c 来统计一个高负载应用中 SQLite 锁调用的频率——结果发现每秒有超过 2000 次 fcntl 调用，这对性能的影响不容忽视。

Windows 锁实现

Windows 平台使用 LockFileEx() 和 UnlockFileEx() 来实现锁，实现在 os_win.c 中。在 Windows 上锁的行为有一些重要的差异：

  • Windows 的锁是 强制锁 (mandatory) — 不遵守锁协议的进程会被操作系统直接拒绝访问，而不是仅仅得到一个错误码。
  • 锁与句柄关联 — 与 POSIX 的"与进程关联"不同，Windows 锁与特定的文件句柄关联。这意味着同一个进程中的不同线程如果打开了不同的文件句柄，它们之间也需要 lock 协调。

在 3.54.0 版本中，SQLite 做了一项重要的性能优化：用 SRWLOCK (Slim Reader/Writer Lock) 替换了传统的 CRITICAL_SECTION。这个优化的背景是：在多核心高并发的 Windows 系统上，CriticalSection 的内核态切换成本越来越高，而 SRWLOCK 是用户态的轻量级读写锁，特别适合 WAL 模式下大量 SHARED 锁获取/释放的场景。

/* os_win.c — SRWLOCK 使用 (3.54.0+) */
/* 文件: src/os_win.c */

/* 初始化 SRWLOCK */
void winInitLock(winLock *pLock){
    InitializeSRWLock(&pLock->srw);
}

/* 获取共享锁 (读) */
void winReadLock(winLock *pLock){
    AcquireSRWLockShared(&pLock->srw);
}

/* 获取排他锁 (写) */
void winWriteLock(winLock *pLock){
    AcquireSRWLockExclusive(&pLock->srw);
}

/* 释放共享锁 */
void winReadUnlock(winLock *pLock){
    ReleaseSRWLockShared(&pLock->srw);
}

/* 释放排他锁 */
void winWriteUnlock(winLock *pLock){
    ReleaseSRWLockExclusive(&pLock->srw);
}

根据 SQLite 团队的基准测试，SRWLOCK 替换 CriticalSection 后，Windows 平台上的多线程并发读性能提升了约 15-20%，在高核心数（32+ 核）的系统上提升更为明显。

死锁预防

SQLite 没有死锁检测器——它采用了一个更简单粗暴的策略：只要遵守固定的锁获取顺序，就不会发生死锁。

SQLite 的约定是：锁只能从低级别向高级别逐级升级，不能跳级，不能反向降级（除了直接回到 NONE）。这个单向升级的规则本质上就是一个 锁排序协议——如果所有参与者都遵守 NONE → SHARED → RESERVED → PENDING → EXCLUSIVE 的线性顺序，那么循环等待就不可能发生。

但实践中有一个常见的死锁场景：

/* 死锁场景：两个连接交叉获取锁 */
/* 连接 A */                         /* 连接 B */
BEGIN IMMEDIATE;                     BEGIN IMMEDIATE;
-- 获取 RESERVED 锁成功              -- 获取 RESERVED 锁成功
UPDATE t1 SET x=1;                   UPDATE t2 SET y=2;
-- 尝试升级到 PENDING —             -- 尝试升级到 PENDING —
-- 等待连接 B 释放 SHARED           -- 等待连接 A 释放 SHARED
-- 死锁！                           -- 死锁！

对于这种情况，SQLite 的处理方式是：返回 SQLITE_BUSY。当 unixLock() 中的 fcntl(F_SETLK) 返回 EACCES 或 EAGAIN 时，SQLite 不会一直重试，而是立即返回 SQLITE_BUSY。调用者需要在应用层进行重试（通常是通过注册 sqlite3_busy_handler 或设置 sqlite3_busy_timeout）。

SQLite 的"反死锁"哲学： 与其在数据库内核中实现复杂的死锁检测和恢复（代价是增加代码复杂度和运行时开销），不如在 API 层面简单返回 BUSY，把决策权交给应用层。这种"让上层做决定"的设计思路贯穿 SQLite 始终——内核保持简单，复杂度推到边界。

共享缓存模式

SQLite 还提供了一种特殊的并发模式：共享缓存模式 (Shared Cache Mode)。当你通过 sqlite3_enable_shared_cache(1) 启用后，同一个进程内的多个数据库连接可以共享同一个 page cache。

在共享缓存模式下，锁的粒度更细了。不再只是文件级别的五级锁，还有一系列内部锁来保护共享的数据结构：

内部锁保护的内容类型
sqlite3.mutex整个数据库连接递归互斥锁
Btree.db->mutexB-Tree 结构（页拆分、合并）互斥锁
Btree.pBt->mutex单个 B-Tree 实例互斥锁
PCache.mutexPage Cache 的 LRU 链表互斥锁

共享缓存模式可以显著减少内存使用（多个连接共享同一份缓存的页），但也带来了更大的锁竞争。在实践中，除非你的应用有大量短连接且内存受限，否则不推荐使用共享缓存模式——每个连接独立缓存是更安全、更可预测的选择。

SQLITE_BUSY_SNAPSHOT

在 WAL 模式下，有一个特殊的错误码 SQLITE_BUSY_SNAPSHOT（从 3.41.0 开始可用）。这个错误发生在以下场景：

  • 连接 A 开始一个读事务，获取了一个 WAL 快照（相当于"我在这个时间点看到的数据"）
  • 连接 B 提交了一个写事务，WAL 向前推进了
  • 连接 A 尝试执行一个写操作，但它持有的快照已经"过时"了——在 A 读取数据之后，数据已经被 B 修改
  • SQLite 不会让 A 基于过时的快照进行写操作，因为那样会违反快照隔离的语义
  • 于是返回 SQLITE_BUSY_SNAPSHOT，告诉 A："你需要重新开始一个事务，读取最新的数据"

这个错误码的引入解决了 WAL 模式下一个常被忽略的语义问题：读事务不能升级为写事务，如果它开始之后数据已经被其他写者修改了。应用层收到这个错误后，应该回滚当前事务，开启一个新事务重新读取数据再尝试写入。

真实故事：锁顺序导致的生产事故

最后，分享一个我亲身经历的教训。

几年前，我所在的团队开发一个嵌入式支付终端系统。系统使用 SQLite 存储交易记录，多个进程（支付进程、日志进程、监控进程）同时访问同一个数据库。我们使用了 WAL 模式来避免读写冲突。在实验室测试了数千笔交易都正常，于是信心满满地部署了。

上线当天晚上就出事了——系统在高频交易压力下频繁返回 SQLITE_BUSY，导致一笔交易需要重试 3-4 次才能成功，客户体验极差。

排查过程非常曲折。我们用 strace 捕获了锁操作的时序，发现一个诡异的模式：每次失败都在 fcntl(F_SETLK, F_WRLCK) 上返回 EAGAIN，但谁持有了锁呢？同一个进程里的其他线程！

原来，我们的日志进程在主线程中打开了一个 SQLite 连接并持有 SHARED 锁用于轮询查询。然后在同一个进程中另一个线程（支付线程）尝试打开同一个数据库文件开始写事务。问题出在 POSIX 锁的特性上：同一个进程内的 fcntl 锁是共享的。如果两个线程使用同一个文件描述符（或者通过 fork 继承的 fd），fcntl 锁会互相覆盖。但我们的情况更微妙——两个线程各自调用了 sqlite3_open()，打开了不同的文件描述符。问题是：POSIX 规定，一个进程内的多个 fd 指向同一个文件时，fcntl 锁是进程级别的。即：线程 A 的 fd 持有 SHARED 锁，线程 B 的 fd 请求 EXCLUSIVE 锁会被自己阻塞——因为内核看到的是同一个进程已经持有了冲突的锁。

解决方案也很简单：将日志查询和支付写入放在同一个线程中串行执行，或者使用 PRAGMA journal_mode=wal2（3.54.0 实验特性）来缓解锁竞争。最终我们选择了后者，同时为每个进程只打开一个 SQLite 连接，所有操作通过消息队列在单一线程中处理。

这个事故的关键教训： POSIX 文件锁是 进程级别 的，不是文件描述符级别的。同一进程内的多个线程使用 fcntl 时会发生自锁 (self-deadlock)。如果你的应用是多线程的，确保每个线程使用独立的 SQLite 连接，并且不要在同一个线程中同时持有读锁和等待写锁。最佳实践是将所有数据库操作集中到一个线程中串行化，或者使用 sqlite3_busy_timeout(1000) 让 SQLite 自动重试。
