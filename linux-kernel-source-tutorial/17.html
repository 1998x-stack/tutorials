大家好，我是你们的讲师。上一章我们学习了MINIX文件系统的磁盘布局和底层数据结构——超级块、inode、位图。这些是文件系统的"静态"部分。本章我们将进入"动态"部分：文件系统是如何运转起来的。从你在shell里输入一个路径名开始，到最终读取到文件内容，中间经历了怎样的一条代码链路？这就是我们本章要回答的问题。

引言——从文件名到文件数据

当你执行 cat /usr/bin/sh 时，发生了什么事情？看起来只是一个命令读取了一个文件，但内核需要完成以下一连串的工作：

  1. 解析路径：将字符串 /usr/bin/sh 拆解为组件：/ → usr → bin → sh。
  2. 逐级查找：从根inode出发，读取根目录的内容，找到 usr 的inode号；读取 usr 目录的内容，找到 bin 的inode号；读取 bin 目录的内容，找到 sh 的inode号。
  3. 打开文件：将文件路径映射到一个文件描述符——一个整数，后续操作都通过这个整数来引用文件。
  4. 读取数据：根据sh的inode中的 i_zone[] 数组，找到文件数据所在的磁盘块，通过缓冲区读取到内存，再复制到用户空间。

整个过程就是一条三层映射链：文件名 → inode → 磁盘块。本章将沿着这条链，从目录项结构开始，一步步走到最终的I/O完成。

目录项结构

在MINIX文件系统中，目录本质上是一个特殊类型的文件——它的"文件内容"是一组连续的目录项（directory entry）。每个目录项将一个文件名映射到一个inode号。

struct dir_entry {
    unsigned short inode;  /* inode号（2字节），0表示空闲目录项 */
    char name[14];         /* 文件名（14字节），以'\0'结尾 */
};

每个目录项固定占用16字节。这意味着一个磁盘块（1,024字节）恰好可以存放64个目录项。在MINIX V1中，文件名最长14个字符——这是早期UNIX文件系统的惯例（System V的14字节文件名限制的继承）。

根目录（inode号为1）的i_size决定了这个目录下有多少个目录项。例如，如果根目录的i_size = 1024，则它有 1024 / 16 = 64 个目录项，恰好占满一个磁盘块。

目录项的两种特殊条目：每个目录下都有两个特殊的目录项——.（当前目录）和..（父目录）。前者指向自身inode，后者指向父目录的inode。根目录的..特殊之处在于它指向自身（因为根目录没有父目录）。这两个特殊条目极大地简化了namei()中处理.和..的逻辑。

namei()——路径解析的核心函数

namei()是文件系统中最重要的函数之一。它接受一个路径名字符串，返回对应的inode指针。路径解析的整体逻辑如下：

// fs/namei.c — 简化后的namei()逻辑

        struct inode * namei(const char * pathname)
        {
            struct inode * inode;

            // 1. 确定起始inode
            if (pathname[0] == '/')
                inode = iget(dev, ROOT_INO); // 绝对路径 → 从根开始
            else
                inode = current->pwd;         // 相对路径 → 从当前目录开始

            // 2. 逐组件解析
            while (1) {
                // 跳过开头的'/'
                while (*pathname == '/') pathname++;
                if (!*pathname) break; // 已到末尾

                // 3. 提取路径组件（如 "usr"）
                char * component = get_next_component(&pathname);

                // 4. 在当前目录中查找该组件
                int inr = find_entry(inode, component);
                if (!inr) { /* ENOENT: 文件不存在，返回错误 */ }

                // 5. 获取子inode
                inode = iget(dev, inr);

                // 6. 如果还有下一级，当前inode必须是目录
                if (is_last_component) break;
                if (!S_ISDIR(inode->i_mode))
                    return error(ENOTDIR);
            }

            return inode;
        }

让我们用具体的例子来追踪路径 /usr/bin/sh 的解析过程：

步骤0: 输入 pathname = "/usr/bin/sh"
        以 '/' 开头 → 从根目录 inode(1) 开始

        步骤1: 提取组件 "usr"
        → 读取根目录的数据块（inode(1) 的 i_zone[0]）
        → 在根目录的目录项中搜索 "usr"
        → 发现 inode = 128
        → 读取 inode(128)（发现是目录，继续）

        步骤2: 提取组件 "bin"
        → 读取 inode(128) 的数据块
        → 在 /usr 的目录项中搜索 "bin"
        → 发现 inode = 256
        → 读取 inode(256)（发现是目录，继续）

        步骤3: 提取组件 "sh"
        → 读取 inode(256) 的数据块
        → 在 /usr/bin 的目录项中搜索 "sh"
        → 发现 inode = 384
        → 读取 inode(384)（发现是文件，是最后组件）

        步骤4: 返回 inode(384)

        整个流程使用了 3 次目录读取 + 4 次 inode 读取，
        每次读取都通过 bread() 走缓冲区缓存。
        如果文件被频繁访问，这些读取全部命中缓存。

这个例子揭示了文件系统性能的关键：路径解析过程中，目录项数据被记录在目录文件的数据块中，而这些数据块是通过缓冲区的bread()读取的。目录的i_zone[]指向包含目录项的数据块，内核逐块扫描查找匹配的文件名。这虽然没有现代文件系统的目录索引（如ext4的Htree或B树）那样高效，但对于目录项数量不多的场景（每个块扫描64个目录项），性能是可以接受的。

file_table[] 与文件描述符

路径解析完成后，我们得到了文件对应的inode。但inode只是一个"索引"，要从中读写数据，还需要建立进程和文件之间的关联——这就是文件描述符和打开文件表的职责。

Linux 0.11中维护了三级映射关系：

进程控制块(PCB)          文件表(file_table[])         inode表(inode_table[])
        ┌─────────────┐        ┌──────────────┐        ┌──────────────────┐
        │ filp[0] ──────────→  │ f_mode       │        │ i_mode           │
        │ filp[1] ──────────→  │ f_pos        │        │ i_uid            │
        │ filp[2] ──────────→  │ f_flags      │        │ i_size           │
        │ ...           │        │ f_inode ─────────→  │ i_zone[0..8]     │
        │ filp[19]     │        │ f_count     │        │ i_nlinks         │
        └─────────────┘        └──────────────┘        └──────────────────┘
        fd(整数索引)          file_struct          inode(元数据+数据索引)

每个进程可以在 filp[] 数组中最多打开20个文件（NR_OPEN=20），而系统全局的 file_table[] 最多可以容纳64个打开的文件句柄（NR_FILE=64）。filp[] 中的每个元素是一个指向 file_table[] 的指针——为什么需要两层？因为这样可以让多个文件描述符指向同一个打开的文件（比如 dup() 和 fork() 后的父子进程共享）。

file_struct 中的 f_pos 字段记录了当前读写位置——这就是为什么每个打开的文件句柄可以独立追踪读写位置。当你调用 read(fd, buf, 10) 读取10个字节后，下次再调用 read(fd, buf, 10) 时，文件指针已经自动推进了10个字节。

层级数据结构容量作用
进程级task_struct.filp[20]20个fd进程私有的文件描述符数组
系统级file_table[64]64个打开文件全局打开文件表，记录偏移和引用计数
全局inode_table[32]32个内存inodeinode缓存，所有文件系统共用
f_count引用计数：这是避免"使用中释放"的关键机制。当多个进程共享同一个文件句柄时（例如fork()后），每次close()只是将f_count减1，只有减到0时才真正释放file_table项。这就是Linux中"引用计数"设计模式的最早应用之一。

文件读写完整路径

假设你已经打开了 /usr/bin/sh 并得到了文件描述符3。现在执行 read(3, buf, 1024)，这个系统调用会经历怎样的链条？

用户态调用:
    n = read(3, buf, 1024);  // 用户态

         │
         ▼
    系统调用入口 (kernel/system_call.s)
         │
         ▼
    sys_read() (fs/read_write.c)
        │ fd → 获取当前进程的 filp[fd]
        │ filp[fd] → 获取 file_table[f_inode]
         │
         ▼
    file_read() (fs/file_dev.c)
        │ 根据 f_pos 计算要读取的块号
        │ 例如 f_pos=2048 → 块号 = 2048 / 1024 = 2 (从第0块开始偏移2)
         │
         ▼
    minix_file_read() (fs/minix/file.c, 注意: 0.11中直接实现)
        │ 调用 bread() 读取数据块
        │ 使用 inode.i_zone[] 将文件逻辑块号 → 磁盘物理块号
         │
         ▼
    bread() 返回 buffer_head (fs/buffer.c)
        │ 数据在 bh->b_data 中
         │
         ▼
    copy_to_user()  (mm/memory.c)
        │ 将 bh->b_data 中的数据复制到用户空间的 buf
        │ 注意：这涉及内核态→用户态的数据拷贝
         │
         ▼
    brelse() — 释放缓冲区
    更新 f_pos += 实际读取的字节数
    返回读取的字节数给用户态

这条调用链中，file_read() 函数内部有一个重要的逻辑——将文件逻辑块号映射为磁盘物理块号。这个映射就是通过上一章讲的 i_zone[] 数组来完成的：

  • 如果逻辑块号 block < 7，直接取 i_zone[block]（直接块）；
  • 如果 7 <= block < 7+256，通过 i_zone[7] 的一级间接块来查找；
  • 如果 7+256 <= block，通过 i_zone[8] 的二级间接块来查找。

注意，copy_to_user() 这一步不是可选的——内核不能直接把内核空间的b_data地址返回给用户态程序。因为内核地址空间和用户地址空间是隔离的（通过页表），用户态程序不能直接访问内核的内存。所以必须进行一次数据拷贝。这就是为什么 read() 系统调用在读取小文件时开销主要来自"数据拷贝"而非"磁盘I/O"——因为缓存命中时没有真正的磁盘访问，但数据拷贝无论如何都要进行。

sys_open()/sys_close()——打开与关闭

sys_open()

当我们调用 open("/usr/bin/sh", O_RDONLY) 时，内核执行以下步骤：

int sys_open(const char * filename, int flags, int mode)
{
    struct file * f;
    int fd, i;

    // 1. 在当前进程的 filp[] 中找一个空闲位置
    fd = find_empty_filp(current);

    // 2. 在系统 file_table[] 中找一个空闲项
    f = get_empty_file();

    // 3. 建立关联
    current->filp[fd] = f;

    // 4. 解析路径名 → 获取inode
    f->f_inode = open_namei(filename, flags, mode);

    // 5. 根据flags做处理
    if (flags & O_TRUNC) {
        // 截断文件：将文件大小设为0，释放所有数据块
        truncate_inode(f->f_inode);
    }

    // 6. 初始化文件结构
    f->f_mode = flags;     // 读写模式
    f->f_pos = 0;          // 读写位置从0开始
    f->f_count = 1;        // 引用计数

    return fd;              // 返回文件描述符整数
}

flags 参数支持以下模式：O_RDONLY（0，只读）、O_WRONLY（1，只写）、O_RDWR（2，读写）、O_CREAT（不存在时创建）、O_TRUNC（截断）、O_EXCL（与O_CREAT配合使用，文件已存在时报错）。这些标志位的组合构成了UNIX文件操作的基础API。

sys_close()

int sys_close(unsigned int fd)
{
    struct file * filp;

    // 检查fd是否有效（0 ≤ fd < NR_OPEN）
    if (fd >= NR_OPEN)
        return -EINVAL;

    filp = current->filp[fd];
    if (!filp)
        return -EINVAL;  // 这个fd没有被打开

    // 将进程的fd槽位置空
    current->filp[fd] = NULL;

    // 减少文件结构的引用计数
    // 如果减到0，真正释放全局file_table项
    if (--filp->f_count == 0) {
        // 释放对应的inode（完成写回等）
        iput(filp->f_inode);
        // file_table项被回收
    }

    return 0;
}

注意 close() 的引用计数管理是非常精妙的设计——一个文件句柄可以被多个进程共享（例如 fork() 后父子进程共享了所有打开的文件句柄），只有最后一个使用者关闭它时，内核才真正释放资源。

踩坑故事——文件描述符泄漏

2008年，我维护的一个网络服务器程序在线上运行了大约两周后，突然开始疯狂输出错误日志，全是 "Too many open files"（EMFILE错误）。更糟糕的是，syslogd 本身也打不开日志文件了，因为它也需要打开文件描述符来写入日志。结果就是：服务器虽然进程还在，但完全无法写入任何日志，相当于"失明"了。

问题出在一段看似人畜无害的代码中：

// 有bug的代码示例
        for (int i = 0; i < 10000; i++) {
            int fd = open("/proc/uptime", O_RDONLY);
            // 读一点数据...
            char buf[64];
            read(fd, buf, 10);
            // 处理数据...
            // 哦——忘了 close(fd)！！！
        }

这个循环在一个请求的处理路径中被调用，每次处理请求时打开 /proc/uptime 读取系统运行时间。每次打开都会分配一个文件描述符，但程序员忘记调用 close()。每个请求泄漏一个fd，当并发请求足够多时，进程的 filp[] 表中的20个槽位（Linux 0.11限制；现代Linux默认1024，但可以调整）被全部占满。

随后所有 open() 调用都返回 -1 并设置 errno = EMFILE（"The per-process limit on the number of open file descriptors has been reached"）。不仅仅是你的业务逻辑打不开文件——syslog 也打不开，新的网络连接也 accept() 不出来，因为 accept() 返回的socket本质上也是一个文件描述符！

教训：在系统编程中，每一个 open() 都必须配对 close()。这个原则和 malloc()/free() 同样重要。现代C/C++的最佳实践是使用RAII（Resource Acquisition Is Initialization）模式——在对象的析构函数中自动释放资源。在Linux内核源码中，你可以看到每一对 open()/close()、getblk()/brelse() 的严格对称——这是Linus从一开始就坚持的编码风格。

文件描述符泄漏在Linux 0.11中更加致命，因为每个进程最多只能打开20个文件，全局也只有64个文件句柄。一旦泄漏，系统可能会在几分钟内就耗尽可用资源。今天的Linux虽然默认限制更高（可以通过 ulimit -n 查看和调整），但问题的本质是一模一样的。

下一章我们将进入另一个精彩的主题——管道与exec。管道让数据可以在不同进程之间流动，而exec让一个进程可以"变身"成另一个程序。这两个机制结合起来，就构成了Unix/Linux最强大的编程范式：组合小程序完成复杂任务。
