第5章 矩阵分解与隐语义模型

大家好，我是你们的讲师。上一章我们看了协同过滤——它简洁、优雅，但面临稀疏性问题：当用户-物品矩阵极度稀疏时，用户间或物品间很难找到足够的"共同评分"，推荐效果急剧下降。这一章我们要讲的方法——矩阵分解（Matrix Factorization, MF）——正是为了解决这个问题而生。

矩阵分解是推荐系统历史上最重要的算法之一。它不仅直接解决了协同过滤的稀疏性难题，更重要的是，它开启了"隐语义表示学习"这个方向——让模型自动从数据中学习用户和物品的"特征"，而不需要人工设计。这个思想直接影响了后续所有深度学习推荐模型。

1. 从评分矩阵到隐向量

1.1 为什么需要矩阵分解？

在协同过滤中，我们用用户-物品评分矩阵直接做相似度计算。但问题是：

  • 矩阵极度稀疏（99%+的元素是0或未知）
  • 计算相似度的依据是"共同评分项"——在没有共同评分的情况下，两个用户或两个物品就无法建立联系
  • 没有"泛化能力"——模型只能记住历史交互模式，无法推断未出现过的组合

矩阵分解的解决方案是为每个用户和每个物品学习一个低维稠密向量（Latent Factor Vector / Embedding）。用户向量表示用户的"口味偏好"（比如对科幻片的喜爱程度、对爱情片的喜爱程度），物品向量表示物品的"属性特征"（比如电影包含科幻元素的多少）。用户对物品的预测评分 = 用户向量与物品向量的内积。

这种做法的精妙之处在于：即使两个用户没有"共同评分"的物品，只要他们的隐向量在隐空间中是相似的，系统依然知道他们是"相似用户"。这就绕过了CF的稀疏性限制。

1.2 隐语义的直觉

我们用一个直观的例子来理解"隐语义"到底是什么。

假设我们有三个用户对三部电影的评分：

  • 用户A给《泰坦尼克号》5分，给《阿凡达》4分，给《星际穿越》1分
  • 用户B给《泰坦尼克号》4分，给《阿凡达》5分，给《星际穿越》2分
  • 用户C给《泰坦尼克号》1分，给《阿凡达》2分，给《星际穿越》5分

从这些评分数据中，矩阵分解可能会自动"发现"两个隐语义维度："浪漫爱情元素"和"科幻硬核元素"。《泰坦尼克号》和《阿凡达》在第一个维度上得分很高（浪漫），《星际穿越》在第二个维度上得分很高（硬科幻）。用户A和B的隐向量在第一个维度上权重高（他们喜欢浪漫片），用户C的隐向量在第二个维度上权重高（喜欢科幻片）。

这些"隐语义维度"不是人工标注的（不像"类型：爱情/科幻"这种显式标签），而是模型从用户的行为模式中自动学习出来的。它可能对应着人类可理解的语义（比如"浪漫""硬核"），也可能只是一些数学上有效的抽象模式。

2. SVD与矩阵分解的数学直觉

矩阵分解的数学基础是奇异值分解（Singular Value Decomposition, SVD）。当然，推荐系统中实际用的SVD和数学课本里的SVD有所不同——我们重点看直觉。

2.1 SVD的核心思想

从数学上看，SVD说的是：任何一个 m×n 的矩阵 R 都可以分解为三个矩阵的乘积：R = U·Σ·Vᵀ。其中U是m×m的正交矩阵（左奇异向量），V是n×n的正交矩阵（右奇异向量），Σ是m×n的对角矩阵（对角线上的值叫奇异值，代表每个隐维度的"重要性"）。

在推荐场景中，这意味着：我们可以把用户-物品评分矩阵 R 分解为用户因子矩阵 U·Σ^(1/2) 和物品因子矩阵 Σ^(1/2)·Vᵀ，每个用户和物品对应一个k维向量（k < m, n）。这个k就是隐维度的数量。

2.2 为什么用k个维度？

选择k的本质是"降维"——我们相信用户的偏好虽然表面上千差万别，但本质上只由少数几个"隐因子"决定（比如喜欢动作元素多少、喜欢文艺元素多少、价格敏感程度等等）。相比原始的m维用户向量（m可能是一亿），k通常只有8到256维。

降维带来的好处：泛化能力和抗噪能力。原始的评分矩阵充满了噪声（用户偶尔手滑给了高分、网络水军刷分等），k维隐向量强制模型只保留"最有解释力的模式"，忽略随机噪声。

3. ALS交替最小二乘法

3.1 ALS vs SGD

矩阵分解的参数学习有两种主流方法：SGD（随机梯度下降）和ALS（交替最小二乘法，Alternating Least Squares）。

  • SGD：每次随机采样一个评分数据，计算预测误差，沿着梯度方向更新用户向量和物品向量。简单高效，适合隐式反馈数据。
  • ALS：固定物品向量，求解用户向量的最优闭式解（这是一个最小二乘问题）；然后固定用户向量，求解物品向量的最优闭式解。交替迭代直到收敛。

ALS最大的优势是可并行化：求解每个用户向量的过程是相互独立的，可以分布到不同机器上并行计算。这也是为什么ALS成为Spark MLlib中矩阵分解的默认实现。

3.2 ALS对隐式反馈的处理

隐式反馈（Implicit Feedback，如点击、观看时长、购买）不同于显式评分（1-5星）。在隐式反馈场景中，我们只有"用户是否交互了某个物品"，没有用户明确的喜好程度。此外，"未交互"不等于"不喜欢"——用户可能根本没看到这个商品。

ALS通过给"未观测"样本分配低置信度的权重来处理这个问题：

  • 交互过的样本：高置信度权重（比如1.0），目标值设为1
  • 未交互的样本：低置信度权重（比如0.01 or 0.1），目标值设为0

这种加权策略让模型学到"交互过的物品比没交互过的更相关"，而不至于把所有没交互过的都当成负样本。

3.3 PySpark ALS代码示例

from pyspark.ml.recommendation import ALS
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("MF").getOrCreate()

# 准备数据：需要 (userId, itemId, rating) 三列
# 对于隐式反馈，rating可以是点击次数、播放时长等
data = spark.createDataFrame([
    (1, 101, 5.0), (1, 102, 3.0), (1, 103, 4.0),
    (2, 101, 4.0), (2, 104, 5.0), (2, 105, 2.0),
    (3, 102, 2.0), (3, 103, 5.0), (3, 104, 4.0),
    (4, 101, 5.0), (4, 104, 3.0), (4, 105, 4.0),
], ["userId", "itemId", "rating"])

# 训练ALS模型
als = ALS(
    maxIter=10,
    rank=10,            # 隐向量维度k
    regParam=0.1,       # 正则化系数
    userCol="userId",
    itemCol="itemId",
    ratingCol="rating",
    coldStartStrategy="drop"  # 冷启动策略
)
model = als.fit(data)

# 给每个用户推荐top-3物品
user_recs = model.recommendForAllUsers(3)
user_recs.show(truncate=False)

# 查看学习到的用户因子
user_factors = model.userFactors
user_factors.show(truncate=False)

# 查看学习到的物品因子
item_factors = model.itemFactors
item_factors.show(truncate=False)

4. Bias MF：偏置项的意义

基本的矩阵分解公式是：预测评分 = 用户向量 · 物品向量（内积）。但在实际数据中，用户的"评分行为"受到很多系统性的偏差影响：

  • 全局偏差（Global Bias）：所有用户对所有物品的平均评分。比如某个平台的全体平均分是3.5。
  • 用户偏差（User Bias）：某些用户倾向于给高分（"老好人"永远4-5星），某些用户倾向于给低分（"严格派"很少有4星以上）。
  • 物品偏差（Item Bias）：某些物品本身质量超群所以普遍得分高（比如《肖申克的救赎》），或者某些物品因为营销原因获得了偏高评分。

引入偏置项的Bias MF模型（由Koren在Netflix Prize期间提出）的预测公式为：

# r̂_u,i = μ + b_u + b_i + p_u · q_i
# 其中：
# μ = 全量数据的平均评分
# b_u = 用户u的偏差（用户u的平均评分 - μ）
# b_i = 物品i的偏差（物品i的平均评分 - μ）
# p_u = 用户u的隐向量
# q_i = 物品i的隐向量
# p_u · q_i = 用户向量与物品向量的内积（捕获"个性匹配"的部分）

加入偏置后的模型效果通常比不加偏置好很多——因为偏置项捕获了"系统性的评分趋势"，让隐向量专注于学习真正的"匹配关系"。

5. Netflix Prize深度故事

这一章我们已经多次提到矩阵分解与Netflix Prize的关系，现在让我们完整地讲这个故事——因为它可能是推荐系统历史上最有启发意义的一段历史。

5.1 竞赛的起源

2006年，Netflix的推荐系统还非常朴素——主要是一个Cinematch系统，基于简单的协同过滤。Netflix当时的CEO Reed Hastings决定做一个大胆的事情：悬赏100万美元，奖励任何能把Cinematch的RMSE降低10%的人。

消息一出，全球4000多支队伍加入了比赛。学术界和工业界的研究人员、大学生、数据分析爱好者……只要能写代码的人都在尝试。

5.2 关键进展：矩阵分解登台

比赛早期（2006-2007），领先的技术方案主要是矩阵分解。Yehuda Koren（当时在贝尔实验室）发表的系列论文证明了SVD和隐语义模型在评分预测上的强大效果。他提出的"Bias SVD"、"SVD++"（融合隐式反馈）等方法，成为了推荐系统的经典算法。

到了2008年，比赛进入了白热化阶段。各支领先队伍都发现：单一的矩阵分解模型已经到瓶颈了，真正有效的方案是"模型集成"（Ensemble）——把矩阵分解、邻域模型、RBM（受限玻尔兹曼机）等不同算法的预测结果加权平均，效果会好于任何一个单一模型。

5.3 最后24小时的绝杀

2009年6月，比赛进入最后冲刺。BellKor团队（Koren所在的团队）和The Ensemble团队（由AT&T研究院的研究员组成）交替领先。距离截止不到24小时，BellKor团队合并了另一支队伍Pragmatic Chaos的模型，组成了"BellKor's Pragmatic Chaos"——一个由107个独立模型组成的超级集成。

最终，BellKor's Pragmatic Chaos以10.06%的RMSE提升率（刚好超过10%的门槛）赢得了比赛。第二名的The Ensemble成绩是8.43%。差距非常微小。

5.4 Netflix为什么没有部署

这场比赛在技术上取得了辉煌的成就，但Netflix最终做了一个令人意外的决定——没有部署获奖方案。原因有三：

  • 工程成本极高：107个模型做集成，预测一次需要数秒甚至更久，而Netflix需要在毫秒级别做出推荐
  • 离线指标 ≠ 在线收益：RMSE优化的是评分预测精度，而Netflix实际关心的是用户观看时长和留存率——这两者并不完全相关
  • 算法环境变了：Netflix上线后不久产品形态就发生了变化（从DVD租赁转向在线流媒体），原始的数据集和业务场景已经不同了

Netflix Prize的最大遗产：

虽然Netflix没有上线那套107模型的集成系统，但Netflix Prize的影响远超技术本身。它第一次向全行业证明了：推荐系统值得投入巨大的资源来优化，因为即使是1%的RMSE提升，对应的商业价值也是天文数字。这场竞赛吸引了一代顶尖人才进入推荐系统领域，Koren后来入职Google继续从事推荐研究，他的矩阵分解论文成为了领域必读。更重要的是，它留下的经验——"单纯优化离线指标是不够的，必须和业务目标对齐"——今天读来仍然振聋发聩。

6. 不同矩阵分解方法的比较

方法 可扩展性 隐式反馈支持 冷启动 训练速度 典型应用
SVD（标准） 差（SVD分解 O(min(nm², n²m))） 不支持（需稠密矩阵） 差 慢（不可扩展） 小规模实验
SGD-MF 好（O(非零元素数 × k)） 支持（加权策略） 差 中等（难以完全并行） 中小规模，早期工业应用
ALS 非常好（完全可并行化） 支持（核心设计） 差 快（分布式并行） 大规模工业系统（Spark MLlib）
Bias MF 好（同SGD-MF） 支持 差 中等 评分预测场景（如Netflix Prize）

可以看到，矩阵分解虽然解决了稀疏性问题，但在冷启动上依然是短板——新用户和新物品没有历史交互，自然也就没有隐向量。这也是为什么工业推荐系统不可能只靠矩阵分解，还需要内容特征、用户画像等"非协同"信号的补充。

好了，这一章的内容就到这里。矩阵分解是推荐系统从"记忆"走向"泛化"的关键一步——它让模型不再只是记住过去的交互模式，而是通过隐向量学习到了用户和物品的"本质特征"。但隐向量只是特征的一种形式。下一章，我们将进入推荐系统中工程量最大、也是"性价比"最高的部分——特征工程。你会发现，很多时候"加一个有用的特征"比"换一个更先进的模型"带来的提升更大。
