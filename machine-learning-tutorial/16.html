16.1 当"大多数"淹没了"少数"

现实世界中最有趣的预测问题几乎都是不平衡的：欺诈交易（0.1%）、疾病筛查（1%）、用户转化（3%）、设备故障（0.5%）。在这些场景中，一个"全预测正常"的模型就能拿到99%+的准确率——毫无意义。

我在金融科技公司做反欺诈的时候，正样本（欺诈）只占0.15%。一开始用默认的随机森林，recall只有不到10%——模型几乎找不到任何欺诈交易。后来经过一系列处理，recall提到了85%且precision保持在30%。这个经历让我对不平衡问题有了切实的体会：不平衡不是一个"调调参数"的小问题，它决定了你的模型是否有实用价值。

16.2 重采样：最简单直接的方案

方法做法风险
欠采样（Under-sampling）减少多数类样本，使两类数量接近丢弃大量有价值数据
过采样（Over-sampling）复制少数类样本严重过拟合（模型记住重复样本）

朴素的重采样问题很大。但有两个改进版非常有效：

SMOTE

SMOTE不是简单复制少数类样本，而是在两个少数类样本之间"插值"生成新的合成样本。这样生成的样本不是原始样本的精确副本，增加了多样性。

Tomek Links / Edited Nearest Neighbors

欠采样的改进版：只删除那些"可能是噪声"的多数类样本（比如周围全是少数类的多数类样本），而不是盲目随机删除。

from imblearn.over_sampling import SMOTE
from imblearn.under_sampling import TomekLinks
from imblearn.combine import SMOTETomek
from imblearn.pipeline import Pipeline as ImbPipeline

# SMOTE + Tomek Links组合（先过采样，再清洗边界）
smote_tomek = SMOTETomek(smote=SMOTE(random_state=42),
                         tomek=TomekLinks())

# 必须只在训练集上做重采样！
X_train_resampled, y_train_resampled = smote_tomek.fit_resample(X_train, y_train)
print(f"原始训练集: {dict(zip(*np.unique(y_train, return_counts=True)))}")
print(f"重采样后: {dict(zip(*np.unique(y_train_resampled, return_counts=True)))}")

# 使用Pipeline确保重采样只在训练时发生
pipeline = ImbPipeline([
    ('smote', SMOTE(random_state=42)),
    ('classifier', RandomForestClassifier(random_state=42))
])
pipeline.fit(X_train, y_train)
⚠️ 绝对不要对测试集做重采样！测试集必须保持原始分布。重采样的目的是改变训练数据的分布让模型更关注少数类，但评估必须反映真实世界的数据分布。使用imblearn的Pipeline可以自动保证重采样只在fit时生效，predict时不生效。

16.3 类别权重：不改变数据，改变关注度

另一种思路：不改变数据分布，而是让模型在计算损失时对少数类的错误给予更大的惩罚。大多数sklearn模型支持class_weight='balanced'参数。XGBoost/LightGBM用scale_pos_weight。

方法优点缺点
重采样（SMOTE等）对任何模型有效，改变数据分布直观增加训练数据量和时间，可能引入噪声
类别权重不增加训练量，实现简单部分模型不支持，权重选择需要经验

实战中我通常两种方法都试：先用class_weight='balanced'快速跑一个基线，如果效果不够好，再上SMOTE。很多情况下，类别权重 + 合适的阈值调优就已经足够解决大部分不平衡问题。

16.4 异常检测视角：换个思路

如果少数类极端稀少（< 0.1%），与其把它们当"少数类"用分类模型，不如把它们当"异常值"用异常检测算法。孤立森林（Isolation Forest）、LOF（局部离群因子）、One-Class SVM——这些算法专门训练在"正常样本"上，然后识别偏离正常的样本。

异常检测的哲学和分类不同：分类试图学习"X → 正常/异常"的映射，异常检测试图学习"什么是正常"，然后说"不符合这个模式的都是异常"。当异常模式多种多样（比如欺诈手法不断变化），异常检测往往比分类更鲁棒。

16.5 评估指标：请抛弃Accuracy

在不平衡数据上，accuracy完全不可用。应该用什么取决于你的业务目标：

业务场景核心指标为什么
癌症筛查Recall（召回率）漏诊一个癌症患者后果严重
垃圾邮件过滤Precision（精确率）误删重要邮件用户会发疯
欺诈检测F1或Fβ (β=0.5)兼顾误报（人工审核成本）和漏报（损失）
任何不平衡AUC-ROC, AUC-PR综合评估排序能力
🔑 实战建议：处理不平衡数据的推荐顺序：(1) 先设定正确的评估指标（不是accuracy）；(2) 用class_weight训练基线；(3) 如果少数类样本≥几千，尝试SMOTE及其变体（BorderlineSMOTE, ADASYN）；(4) 调阈值，找到P/R的最佳平衡点；(5) 如果少数类极少（<100），考虑异常检测或数据增强。记住：永远先在原始测试集上评估，再做任何处理。
