第15章 排序模型深入

大家好。前几章我们聊完了召回——从协同过滤到双塔模型，从多兴趣到图召回，把"从百万级候选里捞出千级"这件事讲了个遍。但召回只是推荐的第一道关卡。接下来我们要面对一个更关键的问题：这上千个物品，到底哪个排第一？哪个排第十？哪个压根不应该出现？这就是排序阶段要解决的事。

很多人以为排序就是"拿一个模型算CTR，按分数排"，但工业界的排序远比这复杂。它不是单个模型，而是一条精心设计的流水线——粗排、精排、重排层层递进。每一层都有它的使命，每一层都在跟时间赛跑。今天我们就来深入聊聊排序模型的那些门道。

1. 粗排与精排的分工

为什么需要粗排？答案是：精排模型太慢了。一个现代精排模型（比如DeepFM或DCN）可能包含几十个特征、数十亿参数，对单个样本做一次推理就要花几毫秒。如果召回了2000个物品，精排全部处理可能要10秒钟——用户不可能等那么久。

粗排存在的意义，就是在召回到精排之间架一座桥。它的目标不是"最准"，而是"够快"。用轻量级模型快速把千级候选压缩到百级，让精排模型从容处理。

维度 粗排（Pre-Ranking） 精排（Ranking）
输入候选数 1000-5000 100-500
模型参数量 数百万级 数千万到数亿级
特征数量 10-30个核心特征 50-200个全量特征
延迟预算 < 10ms 30-100ms
训练数据采样 负样本可能大幅降采样 全量或轻度采样
典型模型 双塔内积、LightGBM、轻量DNN DeepFM、DCN、MMoE、DIN

上表让你直观感受两者的差异。粗排要处理5-10倍于精排的候选量，但延迟预算只有精排的十分之一。也就是说，粗排必须在每毫秒内处理比精排多50-100倍的工作量。这种极致的不对称，决定了粗排必须采用完全不同的设计哲学。

2. 粗排模型设计

粗排模型的设计，本质上是在速度和精度之间找平衡。业界常见的方案有以下几种：

方案一：双塔模型 + 向量内积

这是最常见也最轻量的方案。用户塔和物品塔分别计算用户向量和物品向量，粗排时只需要做一次内积运算。双塔的优点是计算量与候选数线性相关，易于扩展到更大的候选集。缺点是用户和物品之间没有特征交叉，精度天花板较低。

方案二：LightGBM

不少人觉得"都2025年了还用GBDT？"但在粗排场景下，LightGBM依然有它的优势：推理速度极快、特征重要性可解释、不需要GPU。很多中型团队在粗排阶段仍然使用LightGBM，效果不输轻量级DNN。搭配上精心设计的特征集，LightGBM可以在2-3ms内完成对数百个候选的评分。

方案三：知识蒸馏

这是近年来最受关注的粗排方案。核心思想是让粗排模型（学生）去"学习"精排模型（老师）的打分结果，而不是直接学习用户点击标签。为什么这样做？因为点击标签是稀疏且嘈杂的，而精排模型的打分包含了更丰富的排序信息——它不仅知道"点击了"，还知道"点击概率是0.87还是0.52"。

蒸馏过程大致如下：

1. 用全量特征训练精排模型（老师），在线上收集打分
2. 构建蒸馏训练集：(user, item, 老师预测的CTR)
3. 用轻量特征训练粗排模型（学生），最小化MSE：
   Loss = MSE(学生(x_light), 老师(x_full))
4. 可选：加入点击标签做辅助损失
   Loss = alpha * MSE + (1-alpha) * BCE(学生(x_light), y_click)

蒸馏的好处是学生不需要看到全量特征，也不需要学到那么复杂的模式——它只需要"模仿"老师的输出就好。这比直接学习点击标签要高效得多。

工业小贴士：粗排阶段的特征选择是一门艺术。有些特征虽然对精度提升很大，但计算成本太高（比如实时统计特征需要查Redis），在粗排阶段可能不值得引入。一个实用的方法论：对每个候选特征，做一次A/B测试——引入该特征后AUC提升 vs 延迟增加。如果边际收益小于成本，果断放弃。

3. 精排模型的实际考量

精排模型是推荐系统的"心脏"。前面几章我们深入讨论了各种精排架构——从Wide&Deep到DeepFM，从DCN到DIN和DIEN。但在工业界，模型架构只是精排工作的一部分。还有几个同样关键但经常被忽视的问题。

特征新鲜度：你用的是今天的数据还是昨天的？

精排模型使用的特征可以分为两类：静态特征（用户画像、物品属性）和动态特征（用户近30分钟的点击序列、物品过去1小时的曝光量）。动态特征对精排模型的贡献巨大——据字节跳动公开分享，仅"物品近1小时点击率"这一个特征，就能提升精排AUC约2-3%。但代价是：你必须在线上实时计算这些特征，或者将它们以微批的方式更新到在线特征存储中。

训练数据：多久之前的数据还能用？

推荐系统的数据分布是不断漂移的。一周前的数据分布可能已经和今天大不相同（尤其是遇到节假日、促销活动、热门事件）。业界实践通常是用最近7-14天的数据训练模型，有些甚至只使用最近3天。更激进的做法是"滚动训练"：每天用最新一天的数据增量更新模型，而不是完全重训。

负样本采样：一个容易被忽略的坑

精排模型的训练数据中，正样本是用户的点击（或购买），负样本是曝光未点击。但问题在于：曝光数据本身就是有偏的——它已经是经过精排模型筛选后的结果。那些"精排认为很不相关所以没排上来"的内容，根本没有出现在曝光日志中。如果只用曝光数据训练，模型会越来越"乐观"——它只看到了被它筛选过的内容，误以为整个世界就是这样。解决方案是引入随机负样本（从全库中随机采样）或使用importance sampling来校正分布。

模型校准：你的CTR是真实CTR吗？

踩坑故事：AUC漂亮，CTR翻了三倍

某电商团队花了一个月时间优化精排模型，离线AUC从0.78提升到了0.83，团队上下非常兴奋。上线A/B测试，实验组的页面点击率果然提升了15%。运营部门看到"点击率大幅提升"的报告后，基于这个数据调整了广告投放预算。结果一个月后复盘发现：实验组的CTR确实高了，但这是"虚高"——模型预测的点击概率整体偏高了约3倍。原来他们在训练时对负样本做了大幅降采样，但没有对模型输出做calibration。离线AUC确实提升了（序没乱），但模型输出的概率值不再是真实概率。运营用这些虚高的CTR数据做预算决策，导致广告ROI大幅下滑。教训：AUC只关心排序质量，不关心概率校准。如果你的CTR预测值会被其他系统使用，一定要做calibration。

模型校准的常用方法包括Platt scaling（用逻辑回归拟合预测值和真实值之间的关系）和Isotonic regression（保序回归）。对于CTR预估任务，经验法则是：如果训练时做了负样本降采样，需要根据采样率对预测值进行校正。

4. 从GBDT+LR到端到端深度学习

2014年，Facebook发表了一篇影响了整个推荐系统行业的论文——"Practical Lessons from Predicting Clicks on Ads at Facebook"。文中提出了一个现在看来简单但当时极具革命性的方案：GBDT + LR。

这个方案的核心思路是：用GBDT做自动特征转换，将原始特征映射为高维稀疏的叶子节点ID序列，然后把这些ID作为LR模型的特征输入。

为什么它如此重要？因为在此之前，特征工程主要靠人工——你需要手动做特征交叉、分箱、组合，非常依赖领域知识。GBDT + LR提供了一种自动化的特征转换方法：GBDT的每棵树都在学习特征之间的交互模式，它的叶子节点天然编码了高阶特征交叉信息。你不再需要手动设计交叉特征了。

下面是一个简化版的GBDT + LR训练流程：

import lightgbm as lgb
from sklearn.linear_model import LogisticRegression

# 第一步：训练GBDT做特征转换
gbdt = lgb.LGBMClassifier(
    n_estimators=100, max_depth=6,
    learning_rate=0.1, num_leaves=63
)
gbdt.fit(X_train, y_train)

# 第二步：提取GBDT叶子节点索引作为新特征
# 每棵树贡献一个叶子节点ID，组成特征向量
def gbdt_feature_transform(model, X):
    # apply() 返回每个样本在每棵树的叶子节点索引
    leaf_indices = model.apply(X, pred_leaf=True)
    # shape: (n_samples, n_trees)
    return leaf_indices

X_train_transformed = gbdt_feature_transform(gbdt, X_train)
X_test_transformed = gbdt_feature_transform(gbdt, X_test)

# 第三步：用转换后的特征训练LR
lr = LogisticRegression(C=0.1, solver='liblinear')
lr.fit(X_train_transformed, y_train)

# 线上推理时，先走GBDT得到叶子索引，再走LR预测CTR

这套方案在2014-2017年间是工业推荐系统的标配。即使今天有了各种深度学习模型，GBDT + LR的思想仍然在粗排和特征工程中发光发热。

端到端深度学习的崛起

GBDT + LR的局限在于：GBDT的树结构是固定的，不能与LR的目标联合优化。也就是说，GBDT的叶子节点不一定对最终的CTR预估最优。端到端深度学习（如Wide&Deep、DeepFM）解决了这个问题——它们让特征表示学习和预测目标联合优化，理论上能达到更高的精度天花板。

那么，我们失去了什么？主要是两点：简单性和可解释性。GBDT + LR是完全可解释的——你可以精确知道每个特征对最终预测的贡献。而深度模型更像黑盒。此外，GBDT + LR的训练和部署都极其简单，不需要GPU。

关键洞察：不要因为有了更先进的模型就放弃简单的方案。如果你的推荐系统只有几百万用户，GBDT + LR可能比DeepFM更合适——训练快、部署简单、效果不差。"先用简单方案跑通全链路，再逐步迭代到更复杂的模型"是工业界最务实的技术路线。

从粗排的轻量级模型到精排的深度网络，排序这条流水线上的每一个环节都有自己的权衡和智慧。粗排追求极致的速度，精排追求极致的精度，而模型校准确保输出值的可靠性。但排序只是推荐链路中的一环——当你需要对多个目标同时优化时（CTR、CVR、点赞、转发……），简单的单任务模型就不够用了。下一章我们就来聊聊多任务学习，看看如何让一个模型同时做好几件事。
