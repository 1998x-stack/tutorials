大家好，经过前面二十四章的学习，你的算法已经在 notebook 中跑通了、可视化效果完美、基准测试数据也令人满意。现在你面临真正的考验：把它放到一台真实机器人上，看它能不能在现实的物理世界中工作。

这一步，是所有路径规划学习者的"成年礼"。仿真中的完美表现可能在真实机器人上完全失效——你的优雅算法会碰到传感器噪声、执行器延迟、地面摩擦力不均、电池电压波动等一系列仿真环境中从未出现过的问题。我自己在第一次把路径规划系统部署到真实机器人上时，经历了整整三周的"崩溃-修复-再崩溃"循环。最终它跑起来的那一刻，实验室的师兄弟们都围过来鼓掌——那是我整个研究生生涯最有成就感的瞬间之一。

仿真的价值与局限

在讨论 sim-to-real 问题之前，首先要明确：仿真不是可选项，它是路径规划开发的必备工具。几乎所有的现代机器人系统都依赖于仿真进行开发迭代。但我们需要清楚仿真的边界在哪。

仿真做什么做得很好

  • 快速迭代算法逻辑——在仿真中你可以 1 秒跑完真实世界中需要 1 小时的实验。这意味着你可以在短时间内尝试几十种参数组合。
  • 复现和调试bug——仿真环境是完全可复现的：相同输入一定得到相同输出。在真实机器人上，一次实验的条件几乎不可能完全复现。
  • 测试极端场景——在仿真中你可以安全地测试传感器失效、通信延迟、极端地形——这些场景在实体上测试要么太危险、要么太昂贵。
  • 大规模并行测试——你可以同时启动几百个仿真实例，在一夜之间完成真实世界中几个月才能积累的测试里程。

仿真永远教不会你的

  • 传感器噪声的真实分布——仿真中的噪声模型通常是高斯分布的简化假设，而真实传感器的噪声往往包含系统性偏差和长尾异常值。
  • 硬件的非理想特性——电机回差、轮胎打滑、履带震动、机械结构柔性——这些物理效应极其复杂，几乎不可能完全建模。
  • 系统集成的"最后一公里"——把所有软件模块（感知、规划、控制、通信、状态机）连成一个完整系统后出现的偶发问题，仿真几乎无法预测。
我的原则：在仿真中完成 95% 的开发工作，但永远为最后的 5% 预留至少 30% 的工期。这不单纯是经验之谈——sim-to-real gap 的本质决定了最后 5% 的工程工作往往比前 95% 更耗时。

传感器噪声建模

从仿真到现实的第一步，是理解你的机器人上的传感器实际上在给你什么数据。每一个传感器都引入误差。如果你在仿真中假设传感器是完美的，你的规划器必然无法应对真实世界。下面我们来看如何对最常见的几种传感器噪声进行建模：

import numpy as np
from dataclasses import dataclass


@dataclass
class LiDARNoiseModel:
    """激光雷达噪声模型"""
    range_std: float = 0.02       # 距离标准差 (m)
    angular_std: float = 0.003     # 角度标准差 (rad)
    drop_out_rate: float = 0.01    # 丢点率
    max_range: float = 30.0        # 最大量程 (m)
    min_range: float = 0.1         # 最小量程 (m)

    def apply(self, ranges: np.ndarray, angles: np.ndarray) -> np.ndarray:
        """对完美激光雷达数据施加噪声"""
        noisy = ranges.copy()

        # 1. 高斯测距噪声
        noisy += np.random.normal(0, self.range_std, len(noisy))

        # 2. 随机丢点（返回 inf 表示无回波）
        drop_mask = np.random.random(len(noisy)) < self.drop_out_rate
        noisy[drop_mask] = np.inf

        # 3. 量程截断
        noisy = np.clip(noisy, self.min_range, self.max_range)

        return noisy


@dataclass
class OdometryNoiseModel:
    """里程计漂移噪声模型"""
    trans_std: float = 0.01        # 平移漂移 (m/m)
    rot_std: float = 0.005         # 旋转漂移 (rad/rad)
    bias_drift: float = 0.001      # 偏置随时间漂移 (m/s)

    def __post_init__(self):
        self.trans_bias = 0.0
        self.rot_bias = 0.0

    def apply(self, dx: float, dy: float, dtheta: float,
              dt: float) -> tuple:
        """对里程计增量施加累积性噪声"""
        # 偏置随机游走
        self.trans_bias += np.random.normal(0, self.bias_drift * dt)
        self.rot_bias += np.random.normal(0, self.bias_drift * dt * 0.5)

        # 比例噪声 + 偏置
        noise_dx = dx * (1 + np.random.normal(0, self.trans_std)
                         + self.trans_bias * dt)
        noise_dy = dy * (1 + np.random.normal(0, self.trans_std)
                         + self.trans_bias * dt)
        noise_dtheta = dtheta * (1 + np.random.normal(0, self.rot_std)
                                  + self.rot_bias * dt)

        return noise_dx, noise_dy, noise_dtheta


@dataclass
class GPSNoiseModel:
    """GPS 噪声模型"""
    xy_std: float = 0.5            # 水平位置标准差 (m)
    update_rate: float = 10.0      # 更新频率 (Hz)
    outage_prob: float = 0.001     # 信号丢失概率
    multipath_prob: float = 0.02   # 多径效应概率
    multipath_offset: float = 3.0  # 多径偏移量 (m)

    def apply(self, true_position: np.ndarray, dt: float) -> np.ndarray:
        """GPS 测量模型"""
        if np.random.random() < self.outage_prob:
            return None  # 信号丢失

        noisy = true_position.copy()

        # 基础高斯噪声
        noisy += np.random.normal(0, self.xy_std, 2)

        # 多径效应（突发大偏移）
        if np.random.random() < self.multipath_prob:
            angle = np.random.uniform(0, 2 * np.pi)
            noisy += np.array([
                self.multipath_offset * np.cos(angle),
                self.multipath_offset * np.sin(angle),
            ])

        return noisy


# ---- 噪声注入测试 ----
def simulate_sensor_noise():
    """模拟机器人直线运动时的传感器噪声"""
    lidar = LiDARNoiseModel()
    odom = OdometryNoiseModel()
    gps = GPSNoiseModel()

    # 模拟 100 步直线运动
    true_path = []
    odom_path = []
    gps_path = []

    x, y, theta = 0.0, 0.0, 0.0
    dt = 0.1

    for step in range(100):
        # 真实运动：以 0.5 m/s 直线前进
        dx = 0.5 * dt * np.cos(theta)
        dy = 0.5 * dt * np.sin(theta)
        dtheta = 0.0  # 直线

        x += dx
        y += dy
        true_path.append((x, y))

        # 带噪声的里程计
        dx_n, dy_n, dt_n = odom.apply(dx, dy, dtheta, dt)
        # odom 是累积的，所以保持姿态估计
        theta_n = theta + dt_n
        if step == 0:
            ox, oy = dx_n, dy_n
        else:
            ox += dx_n
            oy += dy_n
        odom_path.append((ox, oy))

        # 带噪声的 GPS
        gps_meas = gps.apply(np.array([x, y]), dt)
        if gps_meas is not None:
            gps_path.append(gps_meas)

        theta = theta_n  # 实际控制用里程计更新的姿态

    return true_path, odom_path, gps_path

这段代码的关键在于：里程计的误差是累积的。随着机器人移动，里程计的偏置会不断漂移——这就是为什么只靠里程计的机器人在移动几十米后位置估计会严重偏离真实值。而 LiDAR 的误差是每次测量独立的（虽然也有系统性偏差），GPS 则偶尔会有多径效应导致的"跳变"。在你的规划器中，你必须考虑到这些不确定性——规划器的"世界模型"永远是不完美的。

Sim-to-Real Gap

Sim-to-real gap（仿真到现实的鸿沟）是机器人领域最棘手的问题之一。它指的是：在仿真中表现完美的系统，搬到真实世界后性能急剧下降的现象。这不是某一两个参数的问题，而是多方面的系统性差异：

动力学失配

仿真中的物理模型（摩擦力、惯性、弹性形变）是对真实物理的近似。你为仿真中的机器人规划的路径，假设它能以某个速度拐弯——但真实机器人的轮胎在地毯上和瓷砖上的摩擦力系数可能相差 30%。这意味着你规划的曲率半径在真实世界中可能是错误的。

传感器真实性

仿真中的 LiDAR 返回的是精确的射线-三角面片交点。真实的 LiDAR 点云包含混合像素（一个激光脉冲打到两个物体的边缘）、反射率导致的测量偏差、运动畸变（扫描过程中机器人移动了位置）。你的规划器可能完美地利用了仿真中的"干净"数据，但在面对真实点云时完全不知所措。

环境多样性

仿真环境通常只包含有限的场景变化。真实世界有无穷的变化：阳光直射导致视觉传感器过曝、雨滴反射导致 LiDAR 产生噪点、地毯边缘高出地面 1 毫米导致机器人颠簸。这些"小"事叠加起来，足以让最精巧的规划器崩溃。

踩坑故事：我曾经参与一个 AGV（自动导引车）项目，所有算法在 Gazebo 仿真中跑得完美无缺——路径平滑、避障灵敏、停位精度 ±1 厘米。我们兴高采烈地把系统部署到实车上，结果第一圈就跑偏了 30 厘米。排查了两天发现罪魁祸首是：仿真中的轮胎摩擦模型假设了理想库伦摩擦，而真实仓库地面有微小的油渍残留，导致机器人在转弯时侧滑了约 5%。这个侧滑在单次转弯中微乎其微，但在经过 6 个连续转弯后累积了 30 厘米的偏差。我们的路径规划器完全不知道"轮胎可能打滑"这件事——它假设机器人可以精确跟踪任意路径。最后的解决方案是通过实车标定获取侧滑模型参数，并在规划器中的运动学约束里加入安全裕度。教训是：永远不要相信仿真中的动力学参数——它们永远是"接近真实"而非"等于真实"。

域随机化与鲁棒规划

既然 sim-to-real gap 无法完全消除，我们需要系统性的方法来缩小它。两种最有效的方法是域随机化和鲁棒规划。

域随机化（Domain Randomization）

域随机化的思路很简单：在仿真中随机化各种参数（摩擦系数、质量、传感器噪声水平、光照条件等），让规划器在多样化的条件下训练，从而在真实世界中更具鲁棒性。它的哲学是：如果你见过足够多的"假真实"，你就不会被任何一个"特定假真实"所误导。

常用的域随机化参数包括：

  • 物理参数——质量 ±20%、摩擦系数 ±50%、阻尼系数 ±30%
  • 传感器参数——噪声标准差 ±50%、采样频率 ±10%、丢包率 0-5%
  • 环境参数——光照强度 100-1000 lux、地面纹理、随机障碍物位置
  • 执行器参数——电机延迟 0-50ms、最大速度 ±10%、最大力矩 ±20%

鲁棒规划策略

即使不做完整的域随机化训练，你也可以在规划器中加入鲁棒性设计：

  • 安全裕度——在代价地图中将障碍物膨胀半径比实际值大 10-20%。仿真中的完美路径在真实机器人上可能根本走不出来——永远留 30% 的安全裕度。
  • 不确定性感知规划——将定位不确定性建模到代价函数中：不确定性越大的区域，代价越高。机器人在狭小空间中应该走得更慢、更谨慎。
  • 保守的速度规划——在靠近障碍物时主动减速。理想路径可能贴着障碍物边缘通过，但加上执行器延迟和定位误差后，这无异于自杀式导航。
  • 频繁重规划——不要期望一次规划管到底。每 100ms 重规划一次，让系统能够及时纠正偏差。
核心要点：仿真中的完美路径在真实机器人上可能根本走不出来——永远留 30% 的安全裕度。具体来说：在代价地图中将所有障碍物额外膨胀 1-2 个栅格、将最大速度限制在理论值的 70%、让规划路径与最近的障碍物保持至少 20 厘米的距离。这些看起来"保守"的做法，恰恰是让系统在真实世界中可靠运行的关键。

最后分享一个我亲身经历的 sim-to-real 故事。2020 年，我们团队为某制造工厂部署一批 AGV。仿真测试中，每台 AGV 在 4000 平米的车间内路径规划成功率接近 100%，平均路径长度仅比理论最短长 3%。但上线第一天就出事了：一台 AGV 在通过一个门洞时，由于地面有一条 3 毫米高的电缆槽盖板，机器人碾过时发生了轻微颠簸。这一颠簸导致 LiDAR 的一帧扫描产生了运动畸变，定位模块短暂跳变，航向角估计偏离了 2 度。2 度看起来极小，但结合机器人 0.8m/s 的速度和规划器的 500ms 重规划周期，这 2 度的航向偏差被放大成了 14 厘米的位置误差。机器人擦到了门框，触发了急停。整个车间的 AGV 因为交通管制策略的连锁反应全部停摆。根本原因不是规划算法本身有问题，而是我们的重规划周期和定位频率之间的时间解耦没有考虑到传感器噪声在实际物理环境中的传播效应。后来我们在规划器中加入了基于定位协方差的自适应安全距离：当定位不确定性增大时，自动扩大障碍物膨胀半径、降低最大速度。这个改动只花了 2 天时间，却彻底解决了这类问题——后来的 8 个月部署中再没有出现过类似事故。

这个故事告诉我们：sim-to-real 不是一个"可以解决"的问题，而是一个"需要持续管理"的风险。理解这个 gap 的存在，尊重它的复杂性，然后在系统设计中为它留出足够的空间。做好了这一步，你的路径规划系统才能从"玩具"变成"产品"。
