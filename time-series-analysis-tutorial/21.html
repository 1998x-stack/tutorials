第21章 Prophet与自动预测

大家好。前面两章我们走的是"手工打造"路线——自己构造滞后特征、滚动统计量、日历编码，再塞进 XGBoost。这条路线效果好，但代价高：每个新数据集都要重新设计特征，每个模型都要手动调参。

有没有更轻量的方式？如果我只是想快速得到一个合理的预测，不想在特征工程上花太多时间呢？

这就是 Prophet 和 Nixtla 生态登场的时刻。这两个工具代表了时序预测的"自动挡"方向。它们封装了经过工业验证的预测策略，让你用几行代码就能跑出不错的 baseline，甚至直接上线。

21.1 Prophet：来自 Meta 的"预测工具箱"

Prophet 是 Meta（当时还叫 Facebook）在 2017 年开源的时间序列预测工具。它的核心思想是加法分解：

y(t) = trend(t) + seasonality(t) + holidays(t) + changepoints(t) + error

这个公式看似简单，但 Prophet 的精妙之处在于每个组件都可以独立控制：

  • 趋势（trend）：分段线性或逻辑增长曲线，自动检测趋势变化点（changepoints）
  • 季节性（seasonality）：通过傅里叶级数建模年、周、日等多周期模式
  • 节假日（holidays）：你只需要提供一个包含节假日日期的列表，Prophet 自动为每个节假日及其前后窗口创建指示特征
  • 变化点（changepoints）：自动检测趋势发生结构性变化的时间点，默认识别前 80% 数据中的变化
import pandas as pd
from prophet import Prophet

df = pd.read_csv('sales.csv')
df.columns = ['ds', 'y']  # Prophet 要求列名必须是 ds 和 y

model = Prophet(
    yearly_seasonality=True,
    weekly_seasonality=True,
    daily_seasonality=False,
    changepoint_prior_scale=0.05  # 默认 0.05，越大越容易跟随趋势变化
)

# 添加自定义节假日
christmas_dates = pd.DataFrame({
    'holiday': 'christmas',
    'ds': pd.to_datetime(['2023-12-25', '2024-12-25']),
    'lower_window': -3,
    'upper_window': 3
})
model.add_regressor('promotion_flag')  # 还可以添加额外回归变量
model.fit(df)

future = model.make_future_dataframe(periods=30)
forecast = model.predict(future)
model.plot(forecast)
小贴士：changepoint_prior_scale 是 Prophet 最重要的超参数。设为接近 0 时，趋势几乎不变（类似于全局线性趋势）。设为较大值（如 0.5）时，趋势会紧贴数据的每个波动，容易过拟合。一个实用策略是用时序交叉验证（第 22 章）在这个参数上做网格搜索，通常 0.01-0.5 之间能找到不错的点。

21.2 Prophet 的优势与局限

Prophet 最大的优势是易用性和解释性。你不需要懂特征工程，不需要操心缺失值（Prophet 直接传 NaN 不会报错），不需要处理异常值（Prophet 对异常值容忍度较高）。输出结果直接包含趋势、季节性和节假日各分量的可视化，业务方看了也能理解。

但 Prophet 也有明显局限：

方面优势局限
数据量小样本（几十个点）也能跑大数据量（百万级）训练慢
多周期季节性支持年/周/日，傅里叶项可调不支持子周期（如每两周一次）
外生变量支持 add_regressor不支持大量外生变量（树模型做更好）
不确定性量化原生支持预测区间区间假设基于 MCMC，在小样本下不可靠
速度小数据快100 万行以上数据比 LightGBM 慢 10 倍以上
交互效应加法模型，一目了然无法捕捉特征间的交互

21.3 Nixtla 生态：工业级预测工具链

如果说 Prophet 是"家用轿车"，那 Nixtla 生态就是"工程车队"。Nixtla 是一套 Python 时序预测库的组合，由一位前 Amazon 经济学家主导开发。它包含三个核心库：

statsforecast——经典统计模型的规模化实现。它用 Cython 和 numba 重写了 AutoARIMA、AutoETS、AutoTheta、TBATS、MSTL 等经典模型，速度比 R 语言的 forecast 包快 50-100 倍。

from statsforecast import StatsForecast
from statsforecast.models import AutoARIMA, AutoETS, TBATS

sf = StatsForecast(
    models=[AutoARIMA(season_length=7),
            AutoETS(season_length=7),
            TBATS(season_length=[7, 365])],
    freq='D',
    n_jobs=-1
)

sf.fit(df)
forecasts = sf.predict(h=30)
# 一行代码同时跑三个模型，返回 30 天预测

mlforecast——ML 特征工程自动化。这个库把第 19 章学的那些特征工程步骤（滞后特征、滚动窗口、日历特征）全部自动化了。你只需要指定想要什么模型（XGBoost、LightGBM 都行），它会自动帮你做滑窗变换。

from mlforecast import MLForecast
from mlforecast.target_transforms import Differences
import lightgbm as lgb

fcst = MLForecast(
    models=lgb.LGBMRegressor(),
    freq='D',
    lags=[1, 2, 3, 7, 14, 28],
    lag_transforms={
        1: [('rolling_mean', 7), ('rolling_std', 7)],
        7: [('rolling_mean', 14)]
    },
    date_features=['dayofweek', 'month', 'year'],
    differences=[1]
)

fcst.fit(df)
predictions = fcst.predict(30)

注意看上面的代码：你不需要自己写 shift()、rolling()，甚至连差分都不需要手动做。mlforecast 自动处理了所有这些变换，而且它内部的实现是安全的——不会出现我们在第 19 章警告的那种 look-ahead 泄露，因为它在每一步迭代中只使用截止到当前时间点的数据。

21.4 决策指南：我应该用哪个？

面对这么多工具，做选择是有章可循的：

场景推荐工具理由
快速探索、做 baselineProphet3 行代码出结果，可视化好
经典模型 baseline（AutoARIMA 等）statsforecast速度比 R 快 50 倍，API 统一
需要 ML 模型但不想手动做特征工程mlforecast自动滞后、滚动、日历特征
需要解释性，面对业务方展示Prophet分解图直观，非技术人员也能理解
竞赛或追求极致准确率mlforecast + LightGBM / XGBoost特征工程可控 + 树模型强大
百万级以上的大规模预测任务statsforecast / mlforecastProphet 在这个量级太慢
需要同时预测上千条序列statsforecast原生支持多序列并行
关键洞察：不要神话任何一个工具。Prophet 不是万能的，mlforecast 也有局限性。实战中的黄金准则是：同时跑 3-4 个候选方法，用下一章要学的时序交叉验证来做选择。我见过很多团队花了两周调 Prophet 没调好，换成 mlforecast 一天就出了结果——反过来也成立。工具只是工具，验证才是裁判。

21.5 一个真实的生产故事

我曾经参与过一个零售需求预测项目。业务方要求预测 5000 个 SKU 在每个门店的日销量——总共涉及约 20 万条独立时间序列。最开始我们用 Prophet，但它单个序列就要跑 3-5 秒，20 万条算下来要 200 多个小时。

切换到 statsforecast 后，同样的 AutoARIMA 模型，因为 C 级别的优化和并行化，20 万条序列在 2 小时内跑完了。这是 100 倍的速度差距。这件事告诉我们：选工具一定要考虑规模。在小数据上好用的工具，到了大规模场景可能完全不可用。反过来，像 mlforecast 这样为大规模设计的工具，在小数据上又可能过度工程化。

下一章，我们将学习如何正确地评估和验证时序模型。不管你是用手工特征工程 + XGBoost 还是自动工具，评估方法如果不正确，结论就毫无意义。时序交叉验证——这门课最重要的一章之一——即将开始。
