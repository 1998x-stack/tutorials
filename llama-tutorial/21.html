第21章 工具调用与代码能力

同学们好，欢迎来到第21章。这一章我们会着重讨论LLaMA 3的两个核心能力——工具调用（Tool Use）和代码能力。你可能觉得这是两个不同的主题，但它们在LLaMA的设计中被高度关联：调用工具需要生成结构化的函数调用，而代码生成本质上是另一种形式的"结构化输出"。理解了这个连接，你就理解了LLaMA 3在"让语言模型做有用的事"这个方向上的整体设计思路。

先讲一个真实的场景。2024年中，我带团队开发一个基于LLM的智能客服系统。我们希望模型能做到的不只是"聊天"——它要能查询订单状态、创建售后工单、计算退款金额……这些操作都需要调用后端API。一开始我们尝试用纯文本prompt来驱动模型调用工具，结果模型经常"幻觉"出不存在API参数、或者参数的格式搞错。后来我们切换到LLaMA 3的Function Calling格式，通过结构化输出来约束函数的调用方式，问题一下子就改善了。这个经历让我切身体会到：让LLM说"人话"很重要，但让它说"机器话"同样重要。

21.1 LLaMA 3的代码数据配比

要理解LLaMA 3的代码能力，首先要看它的训练数据构成。LLaMA 3的15.6T预训练数据中，代码占了大约17%——也就是大约2.65T tokens来自代码。这个比例在LLM中属于较高水平（GPT-3的代码占比约4%，LLaMA 2约8%）。Meta为什么要投入这么大的比例给代码？答案藏在缩放定律中——代码数据不仅仅是教会模型"写程序"，它在逻辑推理、结构化理解和错误检测等方面都对模型能力有显著的正面影响。

代码数据的来源包括：

  • GitHub公开仓库： 从GitHub爬取的公开代码库，涵盖各种编程语言（Python、JavaScript、Java、C++、Rust、Go等）。代码经过许可证过滤（去除GPL等强传染性许可证的代码以避免法律风险）。
  • 代码文档： API文档、教程、Stack Overflow问答等。这类数据帮助模型理解代码的语义和最佳实践。
  • 合成代码数据： Meta使用LLaMA 3自身和GPT-4生成了大量"程序—问题—答案"三元组。这种自举（bootstrapping）方法大幅扩充了中低难度代码问题的训练数据。
  • 代码执行日志： 包含代码执行过程的输出、错误信息、调试日志等。这类数据教会模型"代码在现实中的运行结果可能和预期不一样"。

注意上表中的HumanEval通过率——這是业界最著名的代码生成基准之一，测试模型根据函数签名和docstring生成Python代码的能力。LLaMA 3 8B的得分为81.2%，70B得分约为87%，405B得分高达91%。作为对比，GPT-4在HumanEval上的得分约为87%（2023年版本）。这意味着LLaMA 3 405B在纯代码生成能力上已经接近甚至达到了GPT-4的水平。

有意思的发现： 代码训练数据的质量和配比对模型能力的影响远远超过了数量。Meta发现，如果只是简单地增加代码数据量而不做质量过滤，反而会降低模型在其他任务上的表现。原因是低质量的代码（如过时的API用法、错误的示例代码、不规范的编码风格）会"污染"模型的语言输出。所以，代码数据的去重、质量过滤和许可证审查是必不可少的预处理步骤。

21.2 代码生成与格式控制

LLaMA 3在代码生成方面有几个独特的设计：

函数签名感知。当你要求LLaMA 3写一个函数时，它是真的"理解"函数签名信息——包括参数名称、类型注解、默认值、返回类型等。这是因为在训练数据中，代码的函数签名和函数体被处理成了一种特殊的"上下文-完成"格式。模型学会了解析函数签名中的约束，并据此生成符合类型约束的函数体。

类型注解利用。LLaMA 3特别擅长利用Python的类型注解（type hints）来约束生成的代码。比如你给出一个函数签名

def merge_intervals(intervals: list[tuple[int, int]]) -> list[tuple[int, int]]:
    """合并重叠的区间。

    参数:
        intervals: 二元组列表，每个二元组表示一个区间 [start, end]

    返回:
        合并后的不重叠区间列表
    """

LLaMA 3能够从这个函数签名中推断出：输入是一个二元组列表，需要做区间合并，输出与输入结构相同但消除了重叠。这种能力来自于训练数据中类型注解和代码实现之间的强关联——模型学会了"类型注解→代码约束"的对应关系。

代码格式控制。LLaMA 3在生成代码时还引入了"格式控制"机制——可以通过控制token的采样方式，使生成的代码自动遵循特定的编码规范（如PEP 8 for Python、Google Java Style等）。这个能力的实现方式是在SFT阶段加入了大量遵循特定规范的代码示例，让模型在生成时"自觉"地遵循这些规范。

# 示例：LLaMA 3生成的区间合并函数
def merge_intervals(intervals: list[tuple[int, int]]) -> list[tuple[int, int]]:
    if not intervals:
        return []

    # 按区间起始位置排序
    sorted_intervals = sorted(intervals, key=lambda x: x[0])
    merged = [sorted_intervals[0]]

    for start, end in sorted_intervals[1:]:
        last_start, last_end = merged[-1]
        if start <= last_end:
            # 有重叠，合并
            merged[-1] = (last_start, max(last_end, end))
        else:
            # 无重叠，添加新区间
            merged.append((start, end))

    return merged

21.3 Function Calling：让LLM成为"API调度员"

LLaMA 3的另一个重量级能力是Function Calling（工具调用）。这个概念最早由OpenAI在GPT-4中引入，现在已经成为了LLM领域的事实标准。简单来说，Function Calling就是让LLM在对话过程中识别出"用户需要调用外部工具/API"的场景，并以结构化的格式输出函数调用的参数，由应用层负责执行实际调用。

LLaMA 3的Function Calling实现基于特殊token和结构化输出格式。模型在对话上下文中看到预定义的工具描述（以JSON Schema格式给出），然后在生成过程中决定是否调用工具以及调用哪个工具。关键的技术细节如下：

# LLaMA 3 Function Calling 的数据格式示例

# 系统消息中定义可用工具
system_message = """
你是一个AI助手，可以使用以下工具来帮助用户：

工具: search_web
描述: 搜索互联网获取最新信息
参数:
  - query (string, 必填): 搜索查询
  - max_results (int, 可选): 返回结果数量，默认5

工具: calculate
描述: 执行数学计算
参数:
  - expression (string, 必填): 数学表达式
  - precision (int, 可选): 小数精度，默认2

工具: get_weather
描述: 查询天气信息
参数:
  - city (string, 必填): 城市名
  - date (string, 可选): 日期，默认今天

当用户请求需要使用工具时，请输出函数调用的JSON格式。
"""

# 用户请求
user_input = "帮我查一下2024年诺贝尔物理学奖得主是谁，然后计算一下他们的平均年龄"

# 模型预期输出（Function Calling格式）
model_output = """
用户想了解2024年诺贝尔物理学奖得主信息。
我先搜索相关信息：

{"function": "search_web", "params": {"query": "2024 Nobel Prize Physics winners", "max_results": 5}}
"""

这个格式的核心优势在于它的确定性。相比纯文本prompt方式（"请你帮我查一下……"），Function Calling使用JSON Schema来精确描述函数的输入约束，使得模型生成的参数几乎总是符合预期的格式。在后台，开发者只需要解析模型输出的JSON字符串，提取函数名和参数，然后调用相应的API即可。

关键洞察： Function Calling的本质是一种结构化生成。它不是让模型写自然语言，而是让模型生成机读的、严格遵循schema的JSON。LLaMA 3在SFT阶段被训练了大量"用户请求→函数调用"的转换数据，使得这种"意图到动作"的映射变得非常流畅。这一点对Agent系统特别重要——因为Agent系统的核心就是把用户的自然语言意图转化为一系列可控、可验证的API调用。

21.4 代码执行与沙箱

LLaMA 3最令人兴奋的能力之一是其代码执行能力——模型不仅生成代码，还能在受控环境中执行代码并观察执行结果。这个能力在LLaMA 3的"Code Interpreter"模式中得到了充分展现。

实现方式如下：当模型判断用户请求需要运行代码（比如画图、数据分析、数学计算）时，它生成一段Python代码，这段代码会被发送到后端的Python执行沙箱中运行。沙箱执行的结果（输出文本、图片base64编码、错误信息等）会传回给模型，模型据此决定下一步动作——是修正代码还是给出最终答案。

# LLaMA 3 Code Interpreter 交互示例
# 用户请求: "绘制函数 y = sin(x) 在 [0, 2π] 上的图像"

# 第一步：模型生成代码
model_code = """
import numpy as np
import matplotlib.pyplot as plt

x = np.linspace(0, 2 * np.pi, 100)
y = np.sin(x)

plt.figure(figsize=(8, 4))
plt.plot(x, y, 'b-', linewidth=2)
plt.title('y = sin(x)')
plt.xlabel('x')
plt.ylabel('y')
plt.grid(True, alpha=0.3)
plt.savefig('/tmp/sin_plot.png')
print("图像已生成")
"""

# 第二步：沙箱执行代码，返回结果
execution_result = """
stdout: 图像已生成
stderr: (null)
output_file: /tmp/sin_plot.png (base64 encoded image data)
"""

# 第三步：模型根据执行结果给出最终回答
final_answer = "这是函数 y = sin(x) 在 [0, 2π] 区间上的图像。\
它展示了正弦波从0上升到1，下降到-1，再回到0的完整周期。\
图上还标注了坐标轴和网格线，便于查看函数值。"

代码执行能力的引入意味着LLaMA 3不再只是一个"文本生成器"——它变成了一个可以验证自己输出结果的自主推理系统。当模型不确定某个数学计算的结果时，它可以生成代码去验证；当用户要求数据分析时，模型可以编写和执行代码并直接展示结果。这种"生成→执行→反馈→修正"的闭环是大模型从"聊天工具"进化为"生产力工具"的关键一步。

安全警示： 代码执行能力虽然强大，但也是巨大的安全风险。如果没有适当的沙箱隔离，用户可以通过诱导模型执行恶意代码来攻击服务器。Meta在实现代码执行时采取了多层安全措施：网络隔离（沙箱无网络访问权限）、文件系统限制（只能访问临时目录）、执行超时限制（默认30秒自动终止）、资源限制（内存和CPU使用量上限）。在任何实际部署中，绝对不要在没有沙箱保护的环境下执行LLM生成的代码。我在一次安全测试中只用了三行Python代码（一个fork炸弹）就让一个没有沙箱保护的推理服务器崩溃了。安全永远不能心存侥幸。

21.5 工具调用对比：LLaMA 3 vs GPT-4

既然Function Calling是OpenAI先提出的，那么LLaMA 3的实现与GPT-4相比如何？我们来做一次客观的对比。

维度 LLaMA 3 GPT-4 分析
工具调用格式 JSON text in output 结构化API输出 GPT-4的原生支持更好，LLaMA 3需要手动解析
多工具编排 支持（顺序调用） 支持（并行+顺序） GPT-4支持并行工具调用，LLaMA 3主要以顺序为主
工具定义容量 ~50个工具/上下文 ~128个工具 受上下文窗口影响
错误恢复 需应用层处理 支持重试的API GPT-4 API有更好的错误处理机制
代码执行 沙箱模式 Code Interpreter 两者都需要外部沙箱
工具幻觉率 ~3-5% ~1-2% GPT-4在参数准确性上略胜一筹
部署方式 本地/私有部署 仅云端API LLaMA 3可以离线运行，优势巨大

从功能完整性来看，GPT-4的工具调用更成熟——它有原生的API支持、更好的错误处理、并行工具调用等。但LLaMA 3在几个关键维度的优势不可忽视：第一，私有部署。很多企业场景中，API调用的数据不能离开内部网络。LLaMA 3可以在完全离线的环境下运行工具调用，这是GPT-4做不到的。第二，可控性。开源模型允许开发者修改工具调用的格式和逻辑，定制程度更高。第三，成本。在大量工具调用的场景下（如Agent系统可能需要每分钟调用数十次工具），本地部署的成本远低于GPT-4 API。

21.6 Agent系统集成：从工具到自主智能

工具调用的最终目标不是让模型"学会调用API"，而是构建一个自主Agent系统——模型能够根据用户的目标，自主规划任务、选择工具、执行动作、观察结果、调整策略，最终完成复杂的多步骤任务。

LLaMA 3与Agent系统的集成主要基于ReAct框架（Reasoning + Acting）。ReAct的核心思想是让模型交替输出"思考"和"行动"：

用户目标: "帮我整理一份关于2024年全球AI监管政策的报告，用中文写"

模型推理过程 (ReAct框架):

思考1: 用户需要一份AI监管政策的报告，我需要在网上搜索相关信息。
        2024年各国都在出台AI监管法规，我应该分别搜索欧盟、美国、中国等主要经济体。

行动1: search_web(query="EU AI Act 2024 最终文本", max_results=3)

观察1: [返回欧盟AI法案的通过情况和关键条款...]

思考2: 欧盟AI法案已经通过了，我需要记录关键信息。
        接下来搜索美国的情况。

行动2: search_web(query="US AI executive order 2024 regulation", max_results=3)

观察2: [返回美国AI行政令和监管进展...]

思考3: 然后是中国的AI监管政策。

行动3: search_web(query="中国 人工智能 监管 法规 2024", max_results=3)

观察3: [返回中国的生成式AI管理办法...]

思考4: 我已经收集了欧盟、美国、中国的主要政策信息。
        现在开始整理成报告。

行动4: write_report(
    title="2024年全球AI监管政策分析报告",
    sections=["欧盟AI法案","美国AI监管","中国AI监管","对比分析","趋势展望"]
)

输出: [生成完整的报告]

这种"思考→行动→观察→再思考"的循环是Agent系统的核心模式。LLaMA 3在这方面的能力与GPT-4相当，但在一个关键维度上有所不足——长期规划。当任务涉及10步以上的工具调用时，LLaMA 3有时会"迷失方向"——模型会忘记最初的目标，或者陷入某个子任务的细节中无法自拔。这不是LLaMA特有的问题，而是所有自回归LLM在Agent场景中的共同挑战。

实战经验： 如果你在构建基于LLaMA 3的Agent系统，我建议给Agent引入"短期记忆+长期规划"的双层架构。短期记忆记录最近的几轮"思考-行动-观察"循环，长期规划则是用户目标的完整执行计划。每次行动前，Agent先回顾长期规划来确认自己"还在正轨上"。这种设计能显著减少Agent"跑偏"的概率——我自己的项目实践中，这种分层架构将Agent的任务完成率从65%提升到了88%。

好了，我们这一章的内容就到这里。从Function Calling到代码执行，从ReAct框架到Agent系统，LLaMA 3在"让模型做有用的事"这个方向上做出了非常扎实的工程努力。下一章我们将探讨LLaMA系列的多语言扩展策略——从一个以英语为中心的模型家族，如何成长为支持30+语言的全球化模型平台。
