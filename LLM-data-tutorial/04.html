为什么需要语言识别

大家好，上章我们学会了如何从 HTML 中提取出干净的正文。但现在面临一个新问题：Common Crawl 是一个全球性的互联网爬取数据集，里面包含了 100 种以上的语言。你训练一个中文模型，结果喂进去的是英文、日文、阿拉伯文、甚至少量斯瓦希里文——这显然不行。

语言识别（Language Identification）是数据清洗流水线中不可或缺的一环。它解决的是一个看起来简单的问题："这段文本是什么语言？"但在 TB 级数据规模下，这个问题却有大量的工程挑战：要足够快、足够准、而且要对 100+ 种语言都有基本识别能力。

CCNet（2020年，Meta AI）是第一篇将语言识别作为大规模语料清洗核心步骤的里程碑论文。他们观察到，Common Crawl 中英文内容的比例大约在 40-50% 左右，其余都是其他语言或混合语言的页面。如果不做语言过滤，你的训练数据将是一个"语言大杂烩"——混合语言的数据不仅会浪费 token 预算，还会稀释模型在目标语言上的能力。

有趣的观察： CCNet 团队发现 Common Crawl 中排名前 10 的语言占比约为 85%，剩余 90+ 种语言共享 15% 的份额。长尾语言的数据极度稀缺，对它们的识别准确率也相应较低。这意味着如果你需要为低资源语言构建数据集，需要有针对性的策略——不能简单地套用英语数据的处理流程。

fastText 语言分类器

CCNet 选择的语言识别工具是 Facebook Research 开源的 fastText。准确来说，是 fastText 的预训练语言识别模型 lid.176.bin，它基于字符 n-gram 的线性分类器，体积小、速度快、覆盖语种多。

为什么选 fastText 而不是更复杂的方案？因为在大规模数据处理中，"快"比"准"有时候更重要。fastText 的语言分类器在 CPU 上单线程每秒可以处理数千个文档，而像基于神经网络的方案（如 XLM-R 的 zero-shot 分类）虽然精度略高，但每秒只能处理几十个文档。在 PB 级数据处理中，这种速度差异决定了是花 1 天还是花 1 个月。

截至 2025 年，fastText 的 lid.176.bin 模型支持 176 种语言。它的原理非常优雅：将文本拆分成字符级别的 n-gram（包括 unigram 到 pentagram），然后通过一个线性分类器来预测语言。由于只依赖字符特征，它不需要任何 NLP 预处理（不需要分词、不需要词表），真正做到了一行代码完成语言识别。

import fasttext

# 加载预训练模型（约 126MB）
model = fasttext.load_model("lid.176.bin")

# 单条文档的语言识别
text = "Natural language processing is a subfield of AI."
predictions = model.predict(text, k=3)  # 返回 top-3 预测

# 输出示例：
# (('__label__en', '__label__fr', '__label__de'),
#  (0.9921, 0.0032, 0.0015))
# 第一语言：英语，置信度 99.2%

# 批量处理：将结果映射为简洁格式
def detect_language(text, threshold=0.5):
    labels, scores = model.predict(text.replace("\n", " "), k=1)
    lang = labels[0].replace("__label__", "")
    score = scores[0]
    if score < threshold:
        return "unknown", score
    return lang, score

上面的代码就是一个工业级语言识别流水线的核心。CCNet 的处理流程就是对每个文档调用这个函数，如果置信度低于阈值就标记为"unknown"（通常是混合语言或噪音文档）。

置信度阈值设计

阈值的选择直接影响语言过滤的效果。CCNet 将阈值设为 0.5，这意味着置信度低于 50% 的文档会被丢弃。但这个阈值并非放之四海而皆准，不同的语言和数据源会有不同的最优阈值。

下面是 CCNet 在不同语言上的识别表现（基于他们的内部测试）：

语言精确率覆盖率CCNet 阈值 0.5 下的保留率备注
英语99.8%98.5%~45%最大语种，识别效果最好
中文99.2%97.3%~8%字符特征明显，识别效果也很好
阿拉伯语98.5%95.1%~3%文字系统独特，区分度高
印地语97.3%93.8%~1.5%有足够的训练语料
斯瓦希里语85.2%76.4%~0.2%资源有限，部分与邻近语言混淆
威尔士语72.1%60.3%<0.1%训练数据极有限，阈值 0.5 可能过于严格
阈值策略的权衡： 阈值设得越高，语言纯度越高，但会丢弃更多文档（尤其是短文档和低资源语言的文档）。如果你想构建一个多语言模型（如 XLM-R 的 100 语种覆盖），就要小心阈值不能太高，否则低资源语言的数据会完全消失。一种常见的改进方案是对不同语言设置不同的阈值——英语 0.5，低资源语言 0.3。

其他语言识别方案

除了 fastText，社区中还有几种常用的语言识别工具。我整理了一个速度对比表：

工具开发方覆盖语言速度（文档/秒/CPU）模型大小特点
fastTextMeta176~5000126MB基于字符 n-gram，极致速度
CLD3Google107~3000~2MB（内置）基于神经网络，体积小但覆盖语言少
langdetect社区维护55~800几 MB基于字符 n-gram + Naive Bayes，纯 Python
langid社区维护97~600~20MB也是基于 n-gram，但已经较少维护

对于大规模数据处理，fastText 是目前的不二之选。它的速度优势在高吞吐场景下体现得尤为明显。CLD3 虽然体积很小，适合嵌入移动端或浏览器，但覆盖的语言种类太少。langdetect 的精度不错，但速度慢了将近一个数量级，不适合 TB 级别的数据处理。

一个小技巧：在实际使用中，我们可以用 CLD3 做快速预筛选（以节约 API 调用成本），对 CLD3 无法确定的文本再用 fastText 做二次判定。这种级联策略可以在几乎不影响精度的情况下，将整体吞吐量再提升 30-50%。

多语言场景的数据配比

如果你的目标是训练一个多语言模型，语言识别不仅仅是"过滤"，更是"配比"的依据。一个经典的案例是 XLM-R（Facebook 的跨语言模型），它的训练数据集 CCNet-100 包含了 100 种语言。

在多语言场景下，语言识别面临一个核心矛盾：高资源语言数据太多，低资源语言数据太少。CCNet-100 的数据分布极度不均衡——英语、中文、俄语占据了超过 60% 的数据量，而最底层的 30 种语言合起来不到 1%。如果不对语言分布做人为干预，模型将主要学会英语能力，而对低资源语言几乎"失明"。

一个常见的错误： 很多团队在构建多语言训练数据时，简单地用语言分类器把所有非目标语言都过滤掉。如果你只需要做单语言模型，这没问题。但如果你的目标是多语言能力，这种做法会严重削弱模型的跨语言泛化能力。一个更好的策略是：先识别语言，然后按预定的语言比例进行采样（而不是过滤）。比如，英语数据虽然多，但限制每个训练 epoch 只使用 60% 的英语数据，剩下的 40% 留给 99 种其他语言。

2024 年的 FineWeb 论文在这方面做了一个有趣的发现：即使你的目标是训练英文模型，保留少量其他语言的"噪音"反而可能有助于提高模型在代码混合（code-switching）场景下的表现。但这个结论是否普遍成立，目前学术界还有争议。

语言识别完成后，我们就知道每个文档属于哪种语言。但同一种语言内部，文档质量仍然天差地别——有些是精心撰写的维基百科条目，有些是毫无价值的机器生成垃圾。下一章，我们将学习如何使用启发式规则来识别并过滤这些低质量内容。
