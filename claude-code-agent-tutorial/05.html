第5章 Tool Use 协议深度解析

大家好，我是你们的讲师。上一章我们建立了 Claude Code 的整体架构坐标系。这一章我们要扎进一个最关键的细节层——Tool Use 协议。

你可能已经在第 3 章的 v0 Agent 中见过 Tool Use 的基本模样——模型返回一个 tool_use block，我们把它分发给对应的工具执行函数。但生产级的 Tool Use 比这复杂得多：JSON Schema 怎么写才能让模型正确理解工具的用法？并行工具调用怎么处理？Streaming 模式下 content_block_start 和 content_block_delta 的事件顺序是什么？这些问题如果搞不明白，写出来的 Agent 就会在边界情况上频频出错。

有一次我在做一个 Refactoring Agent 时，连续遇到了 5 次 stop_reason = "tool_use" 但 response 里没有任何 tool_use block 的奇怪现象。当时我盯着终端愣了 10 秒钟，然后意识到——我对 Tool Use 协议的理解太粗浅了。这一章就是为了填补那些"你以为你知道，但其实不知道"的知识盲区。

Tool Use 的 API 契约

Tool Use 的本质是一种消息格式约定：模型在响应中请求调用工具，我们的代码在下一轮请求中将工具的返回结果回传。整个过程遵守一套严格的消息交替规则。

来看核心的数据结构。模型响应中的 content 数组可以包含两种类型的 block：

// Assistant 发送的两种内容块
{
  "type": "text",
  "text": "让我先看看这个文件的内容..."      // 模型在思考或给出文本回答
}

{
  "type": "tool_use",
  "id": "toolu_abc123",                      // 工具调用的唯一 ID
  "name": "read",                             // 要调用的工具名
  "input": { "file_path": "src/main.py" }     // 调用参数
}

我们的代码将 tool_result 回传给模型的格式：

// User 发送的内容块（注意这里"role": "user"）
{
  "type": "tool_result",
  "tool_use_id": "toolu_abc123",              // 关联到之前的 tool_use
  "content": "def main():\n    print('hello')", // 工具执行结果
  "is_error": false                           // 是否出错
}

交替规则非常严格：

  • 消息列表必须以 user 角色开头或结尾（不能以 assistant 结尾）
  • 每次 tool_use 之后必须在下一轮用户消息中给出对应的 tool_result
  • tool_use_id 必须与之前 assistant 返回的 id 精确匹配
  • 一轮中可以有多个 tool_use block，每个都需要独立的 tool_result
常见陷阱：新手最容易犯的错误是忘记把 assistant 的完整 content 数组回填到消息列表中。你不仅需要回填 tool_use block，还需要回填同次响应中的 text block（如果有的话）。如果你只追加了 tool_use 而没有追加对应的 text，模型的上下文就会不完整，可能导致"失忆"或行为异常。

JSON Schema 定义最佳实践

工具定义的质量直接决定了 Agent 的行为质量。一个定义糟糕的工具会让模型频繁误用、调用参数错误、或者根本不知道什么时候该用这个工具。请记住以下原则：

好的 description 告诉模型 WHEN 和 WHY 使用该工具，而不是仅仅描述 WHAT 它做。对比下面两个定义：

// 坏的例子：太平淡
{
  "name": "read",
  "description": "读取文件",
  "input_schema": {
    "type": "object",
    "properties": {
      "file_path": { "type": "string" }
    },
    "required": ["file_path"]
  }
}

// 好的例子：告诉模型何时使用
{
  "name": "read",
  "description": "读取指定文件的全部或部分内容。当你需要查看源代码、配置文件、日志文件或任何文本文件时使用。支持行号偏移和行数限制，适合查看大文件的部分内容。对于二进制文件会自动检测并返回提示信息。",
  "input_schema": {
    "type": "object",
    "properties": {
      "file_path": {
        "type": "string",
        "description": "要读取的文件的绝对路径或相对于工作目录的路径"
      },
      "offset": {
        "type": "number",
        "description": "从第几行开始读取（从 0 开始），不传则从文件开头读取"
      },
      "limit": {
        "type": "number",
        "description": "最多读取多少行，不传则读取全部"
      }
    },
    "required": ["file_path"]
  }
}

关键要点：

  • 工具名必须是唯一的、语义清晰的动词（如 read、write、edit），不要用模糊的名词（如 file_operation）
  • 描述中要用场景化的语言告诉模型"什么情况下该用这个工具"
  • 参数描述要给出具体的值格式提示（如"绝对路径或相对路径"、"从 0 开始的行号"）
  • 必填参数要精确——不要把可选参数设置成必填，这会导致模型在不必要的时候被迫提供值

并行工具调用机制

Anthropic API 支持在单次响应中返回多个 tool_use block。这对于需要同时执行多个独立操作的场景非常有用——比如同时查看多个文件的内容、在多个目录中执行搜索。

当模型返回多个 tool_use block 时，我们的处理逻辑应该是并行执行它们：

import asyncio

async def handle_parallel_tools(response):
    """并行执行同一个响应中的多个工具调用"""
    tool_blocks = [b for b in response.content if b.type == "tool_use"]

    if not tool_blocks:
        return

    # 将所有工具调用打包成异步任务
    async def execute_one(block):
        result = await execute_tool(block.name, block.input)
        return (block.id, result)

    tasks = [execute_one(b) for b in tool_blocks]
    results = await asyncio.gather(*tasks, return_exceptions=True)

    # 构造 tool_result block
    tool_results = []
    for block, (tool_id, result) in zip(tool_blocks, results):
        if isinstance(result, Exception):
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tool_id,
                "content": f"Error: {str(result)}",
                "is_error": True
            })
        else:
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tool_id,
                "content": result
            })

    return tool_results

并行执行的关键注意事项：

  • 独立工具才能并行：如果工具 A 的输出是工具 B 的输入（比如先创建目录再写文件），则必须串行执行。目前的 API 不会告诉你哪些调用有依赖关系——需要模型自己通过消息顺序来管理
  • 部分失败的处理：一个工具调用失败不应该影响其他工具的执行。return_exceptions=True 确保一个失败不会引发 gather 整体崩溃
  • 并发限制：如果同时有 20 个工具调用，你的系统能处理吗？需要设置信号量限制并发数

Streaming 模式下的工具调用处理

Claude Code 默认使用 Streaming 模式——你不必等到完整响应生成后再处理，而是可以逐块处理到达的事件。这对长响应特别重要（比如模型在返回大量工具调用的同时输出很长的分析文本）。

Streaming 模式下，工具调用相关的事件序列如下：

# Streaming 事件序列（工具调用场景）
1. message_start            # 消息开始，包含消息级别的元数据
2. content_block_start      # 第一个内容块开始（可能是 text 或 tool_use）
   ├─ index: 0, type: "text"
3. content_block_delta      # 文本内容逐块到达
   ├─ index: 0, delta: { text: "让我查..." }
   ...
4. content_block_stop       # 第一个内容块结束
5. content_block_start      # 第二个内容块开始
   ├─ index: 1, type: "tool_use", id: "toolu_xxx", name: "read"
   └─ input: {}              # input 初始为空对象
6. content_block_delta      # input JSON 逐块到达
   ├─ index: 1, delta: { partial_json: '{"file' }
   ...
7. content_block_stop       # 第二个内容块结束（input 完整了）
8. message_delta            # 消息级别更新（stop_reason, usage）
   ├─ stop_reason: "tool_use"
9. message_stop             # 消息结束

处理 streaming 工具调用的关键是要实现一个状态机来跟踪每个 content block 的构建进度：

class StreamingToolParser:
    """解析 streaming 响应中的工具调用"""

    def __init__(self):
        self.blocks = {}       # index -> block 状态
        self.active_index = None

    def handle_event(self, event):
        event_type = event.type

        if event_type == "content_block_start":
            idx = event.index
            block = event.content_block
            if block.type == "tool_use":
                self.blocks[idx] = {
                    "type": "tool_use",
                    "id": block.id,
                    "name": block.name,
                    "input": block.input or {},
                    "partial_input": ""   # 累积 partial_json
                }
            else:
                self.blocks[idx] = {
                    "type": "text",
                    "text": ""
                }
            self.active_index = idx

        elif event_type == "content_block_delta":
            idx = event.index
            delta = event.delta
            block = self.blocks[idx]

            if block["type"] == "tool_use":
                # 累积 partial JSON
                block["partial_input"] += delta.partial_json
            elif block["type"] == "text":
                block["text"] += delta.text

        elif event_type == "content_block_stop":
            idx = event.index
            block = self.blocks[idx]
            if block["type"] == "tool_use":
                # 解析完整的 JSON input
                if block["partial_input"]:
                    block["input"] = json.loads(block["partial_input"])
                block.pop("partial_input")
            self.active_index = None

        elif event_type == "message_delta":
            # 最终确认 stop_reason
            self.stop_reason = event.delta.stop_reason
            self.usage = event.usage

    def get_tool_calls(self):
        """返回所有完成的 tool_use blocks"""
        return [
            b for b in self.blocks.values()
            if b["type"] == "tool_use"
        ]

注意一个关键细节：tool_use 的 input 是以 partial JSON 流式到达的。你不能在收到 content_block_start 事件时就读取 block.input——它在开始时刻是空对象 {}。你必须等待所有 content_block_delta 事件完成后，在 content_block_stop 时才能拿到完整的 input。

工具调用中的踩坑实录

以下是我在实际开发 Agent 过程中遇到过的几个最棘手的问题，每个都花了至少半天才定位到根因。

坑 1：tool_use_id 不匹配

tool_use_id 由模型在响应中生成，你必须原样返回。如果你在构造 tool_result 时拼写错误（例如复制了错误的 ID），API 会返回一个 Invalid Request 错误。这个错误消息有时候不太明确，很容易让人误以为是别的 bug。

坑 2：tool_result 内容过大

某些工具调用会返回大量数据——比如读取一个几千行的日志文件、或者递归列出整个项目目录。Anthropic API 对 tool_result 的内容大小有限制（约 5MB）。如果超过限制，API 会返回错误。解决方案是在工具执行端就做好截断，确保返回给模型的数据不超过 1-2MB。模型通常不需要看到完整的 5000 行文件——给它前 100 行和后 100 行加上"中间省略了 N 行"的提示就够了。

踩坑实录：有一次我的 Agent 在处理一个大型前端项目时，Read 工具返回了整个 node_modules 的 package.json 文件内容。这个文件虽然不大，但 grep 匹配到的结果列表特别长。tool_result 瞬间飙到了 8MB，API 直接返回了 413 Payload Too Large。更糟的是，这个错误发生在 Agent 跑了 20 多轮之后，所有之前的上下文都浪费了。从那以后，我强制所有工具的输出上限为 100KB——宁可让模型多问一次，也不冒丢失上下文的风险。

坑 3：stop_reason 是 "tool_use" 但 content 里没有 tool_use block

这种情况通常发生在 response 中只包含 text block 但 stop_reason 仍然是 "tool_use" 的时候。虽然罕见（据观察大约 1/500 次调用），但确实会发生。你的代码必须处理这种情况，否则会陷入死循环——一直以为有 tool_use 但永远找不到。

Tool Use API 字段参考

Block 类型 字段 类型 说明
tool_use id string 唯一标识符，格式 toolu_xxx，用于关联 tool_result
tool_use name string 工具名称，必须匹配工具定义中的 name
tool_use input object 工具调用参数，必须符合工具定义的 input_schema
tool_result tool_use_id string 关联的 tool_use 的 id
tool_result content string | array 执行结果，可以是文本字符串或内容块数组
tool_result is_error boolean 标记工具执行是否出错，默认为 false
