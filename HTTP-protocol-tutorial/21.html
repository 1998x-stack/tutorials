第二十一章：RESTful API设计

前面二十章我们把HTTP协议从底层到架构层都聊了一遍。从这一章开始，我们进入应用层设计——如何基于HTTP协议设计出优雅、健壮、易用的API。

REST（Representational State Transfer）是Roy Fielding在2000年的博士论文中提出的架构风格。它不是一个标准，而是一组设计约束。如果你理解了这些约束背后的"为什么"，你就理解了REST的精髓。

重要提醒：业界说的"RESTful API"和Fielding论文里定义的REST差距很大。绝大多数自称RESTful的API只做到了"使用HTTP方法+JSON"，连REST的基本约束都没有满足。这不一定是坏事——实用主义胜过教条主义。但理解完整的REST约束能帮你做出更好的设计决策。

21.1 REST约束

REST定义了六个架构约束，少了任何一个就不是真正的REST。我们逐一来看：

1. 客户端-服务器（Client-Server）

分离用户界面和数据存储。客户端不关心数据怎么存的，服务器不关心界面怎么渲染的。这种分离让两端可以独立演进。

2. 无状态（Stateless）

每个请求包含服务器处理所需的所有信息。服务器不保存任何客户端上下文。这意味着：认证信息（Token）每次都要带，会话状态存在客户端。

好处是：任意服务器都能处理任意请求，方便水平扩展。坏处是：每个请求会变大（重复传输上下文）。

3. 可缓存（Cacheable）

响应必须显式或隐式地标记为可缓存或不可缓存。如果响应是可缓存的，客户端和中间代理可以复用缓存，减少延迟和网络开销。

4. 统一接口（Uniform Interface）

这是REST最核心也最容易被误解的约束。它分四个子约束：

  • 资源标识：每个资源通过URI唯一标识。
  • 通过表示操作资源：客户端通过资源的表示（JSON、XML）来操作资源，而不是直接操作资源本身。
  • 自描述消息：每个消息包含足够的信息来描述如何处理它（通过Content-Type、Content-Language等）。
  • HATEOAS：超媒体作为应用状态的引擎（我们会在21.6单独讲）。

5. 分层系统（Layered System）

客户端不知道它直接连接的是最终服务器还是中间代理。每层只知道相邻层，不能跨层。这种分层解耦允许在中间插入负载均衡器、缓存、网关等。

6. 按需代码（Code on Demand，可选）

服务器可以传输可执行代码到客户端执行（比如JavaScript、Java Applet）。这是REST中唯一可选的约束。在现代Web中，JavaScript文件本质上就是一种"按需代码"。

21.2 资源建模

REST的核心概念是资源（Resource）。资源建模就是决定"什么东西应该是一个URL"以及"URL应该长什么样"。

名词 vs 动词

REST API的URL应该只用名词来表示资源，用HTTP方法来表动词。这是最基本的原则，但被违反得最多：

// 好的：使用名词，HTTP方法表示动作
GET    /users          — 获取用户列表
POST   /users          — 创建用户
GET    /users/42       — 获取用户42
PUT    /users/42       — 替换用户42
PATCH  /users/42       — 部分更新用户42
DELETE /users/42       — 删除用户42

// 不好的：在URL中使用动词
GET    /getUsers
POST   /createUser
GET    /deleteUser?id=42
POST   /updateUser
POST   /getUserData

集合 vs 单例

资源通常分为集合（Collection）和单例（Item）：

  • /users 是集合，GET返回列表，POST创建新成员
  • /users/42 是单例，GET/PUT/PATCH/DELETE操作单个资源

嵌套资源

当资源之间有明确的从属关系时，用嵌套URL表达：

GET    /users/42/orders          — 用户42的订单列表
GET    /users/42/orders/5        — 用户42的订单5
POST   /users/42/orders          — 为用户42创建订单

但嵌套层级不要超过3层。层级太深说明资源模型可能有问题，或者可以通过查询参数来简化：

// 嵌套太深
GET /organizations/1/projects/2/sprints/3/tasks

// 可以简化
GET /tasks?sprint_id=3

什么时候该/api前缀？ 加 /api/v1 前缀是好的实践，它把API路径和前端页面路径区分开，不会冲突。

命名惯例：推荐用复数名词（/users，/orders），全小写，单词之间用连字符分隔（/blog-posts），避免下划线和驼峰（/blog_posts 或 /blogPosts）。这没有严格的REST规定，但符合URL的通用设计习惯。

21.3 HTTP方法映射

REST利用HTTP方法来表示对资源的操作。这是REST设计中最直观但也最容易被滥用的部分。

HTTP方法CRUD语义幂等安全（只读）请求体响应体
GET读取资源是是无资源表示
POST创建资源否否新资源数据已创建资源
PUT替换资源是否完整资源数据更新后的资源
PATCH部分更新否（可能）否部分字段更新后的资源
DELETE删除资源是否无204或已删除资源
HEAD获取响应头是是无无
OPTIONS查询支持的方法是是无Allow头

幂等（Idempotent）意味着无论执行多少次，结果都一样。多次PUT同一个资源结果不变（都是替换成同样的数据），所以PUT是幂等的。POST创建资源——多次POST会创建多个资源——不是幂等的。PATCH理论上可以是幂等的，但取决于实现，所以通常视为非幂等。

一个常见的选择困难：PUT vs PATCH。简单规则：如果客户端传了完整资源，用PUT；如果只传需要变更的字段，用PATCH。

踩坑故事：早期的项目里，我们设计了一个API用PUT来做部分更新。结果有一天产品说"用户信息被清空了"。排查发现：前端在更新用户昵称时，用PUT只发了 {"nickname": "新名字"}，没有发 email、phone等字段。后端收到PUT请求后直接替换了整条记录——其他字段变成了NULL。从那以后，我们严格规定：PUT必须传完整资源，PATCH才传部分字段。这个坑太经典了，在业界被称为"partial PUT anti-pattern"。

21.4 状态码的正确使用

我在第五章详细讲过HTTP状态码的含义。在REST API设计中，正确使用状态码能让API的语义更加清晰。这里我挑几个最常见的好例子和坏例子：

好的实践：

// 创建成功 → 201 Created
POST /users
→ 201 Created
Location: /users/42

// 异步任务已接受 → 202 Accepted
POST /reports/export
→ 202 Accepted
{ "status_url": "/tasks/export-123" }

// 请求成功但没有内容 → 204 No Content
DELETE /users/42
→ 204 No Content

// 请求参数错误 → 400 Bad Request
POST /users
{"name": ""}
→ 400 Bad Request
{"error": "name is required"}

// 未认证 → 401 Unauthorized
GET /users/me (without token)
→ 401 Unauthorized
{"error": "missing auth token"}

// 无权限 → 403 Forbidden
GET /admin/users (as normal user)
→ 403 Forbidden
{"error": "insufficient permissions"}

// 资源不存在 → 404 Not Found
GET /users/99999
→ 404 Not Found

// 请求冲突 → 409 Conflict
PUT /users/42 (with outdated version)
→ 409 Conflict
{"error": "version conflict", "current_version": 7}

// 资源已被删除 → 410 Gone
GET /old-promotions/5
→ 410 Gone

坏的实践（常见错误）：

  • 所有成功都返回200：不管是创建成功还是删除成功，全部200。这让客户端无法区分操作结果。
  • 认证失败返回200 + 错误体：比如 200 {"code": -1, "msg": "no auth"}。正确的做法是用 401。
  • 参数错误返回500：客户端传了无效参数，服务器返回500 Internal Server Error。这是服务器的错还是客户端的错？显然是客户端的错，应该返回4xx。
  • 所有错误都返回400：不管是"未认证"还是"参数无效"还是"资源不存在"，一律400。这让排查问题变得困难。

实用原则：HTTP状态码是为HTTP协议层设计的语义，不应该用业务错误码替代。正确的方式是：HTTP状态码表示"请求本身的结果"（成功、客户端错误、服务器错误），响应体里的业务错误码表示"业务逻辑的结果"。两者是互补的，不是二选一的。

21.5 API版本控制

API发布后，难免要升级——新增字段、修改参数、彻底重构。怎么让旧客户端不受影响？答案是版本控制。

常见的版本控制方案：

方案示例优点缺点
URL路径版本/api/v1/users最直观、容易路由、缓存友好URL"膨胀"、版本号污染路径
查询参数版本/api/users?version=1URL保持不变不够显眼、容易被忽略
自定义请求头Accept: application/vnd.myapp.v1+jsonURL纯净、符合HTTP语义调试不方便、无法直接浏览器访问
内容协商Accept: application/json;version=1最符合REST哲学实现复杂、客户端支持要求高

在实际项目中，URL路径版本是最常用的方案。它简单直观，任何人都能一眼看出在调用哪个版本的API。而且反向代理和缓存层可以基于URL路径做规则匹配。

// URL路径版本控制
GET /api/v1/users/42
GET /api/v2/users/42

版本管理建议：不要对每个小改动都升级版本号。向后兼容的改动（新增字段、新增可选参数）不需要新版本。只在破坏性变更时才升级版本号。同时给每个版本设置废弃时间线——比如v1上线后，v1.1发布时v1进入"维护期"，v2发布后v1进入"倒计时"，3个月后正式关闭。不要让旧版本永远存在。

21.6 HATEOAS

HATEOAS（Hypermedia As The Engine Of Application State）是REST约束中最被忽视的一个。它的核心思想是：服务器在响应中提供链接，客户端通过这些链接导航，而不是通过硬编码的URL。

换句话说：一个真正的REST API应该是"可浏览的"——就像你在网站上点链接一样，不需要预先知道URL结构。

举个例子：

// 不遵循HATEOAS的响应
GET /orders/123
→ 200 OK
{
    "id": 123,
    "status": "shipped",
    "items": [...]
}

// 遵循HATEOAS的响应
GET /orders/123
→ 200 OK
{
    "id": 123,
    "status": "shipped",
    "items": [...],
    "links": {
        "self": { "href": "/orders/123" },
        "customer": { "href": "/customers/42" },
        "cancel": { "href": "/orders/123/cancel", "method": "POST" },
        "return": { "href": "/orders/123/return", "method": "POST" },
        "tracking": { "href": "/tracking/abc-xyz" }
    }
}

HATEOAS的好处是：

  • 客户端不需要硬编码URL模式——只需从响应中提取链接。
  • 服务器可以改变URL结构而不破坏客户端（只要链接关系名不变）。
  • API变成"自文档化"的。

但为什么大多数API不实现HATEOAS？原因很现实：

  • 实现成本高——每个响应都要动态生成链接。
  • 大多数客户端还是需要知道"有哪些资源"——光靠链接导航效率低。
  • 对非浏览器客户端（移动App、CLI工具）来说，HATEOAS的价值大大降低。

我的观点：HATEOAS在工作流复杂、状态转移多的场景下很有价值——比如订单系统的"已支付→已发货→已签收"状态流转，通过链接告诉客户端下一步能做什么。但对于简单的CRUD API，硬编码URL足够了。你可以把它当作一个进阶技巧，在需要的时候再用，不必在所有API上都强求。Fielding本人可能会不同意我说这话，但这就是业界的现实。
