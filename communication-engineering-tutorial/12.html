第12章 5G物理层：信道与信号

大家好，我是你们的讲师。前面几章我们学习了OFDM帧结构和Massive MIMO——一个提供了"时频资源格"，另一个提供了"空间维度"。现在，我们要在这个资源格上铺上实际的"管道"和"路标"。这些管道就是物理层信道，路标就是参考信号。

想象一个巨大的物流中心（时频资源格），里面有成千上万个格子。有些格子被标记为"收发室"（控制信道），专门用来传指令和通知；有些格子是"货运通道"（数据信道），用来搬运用户数据；还有一些格子是"定位标签"（参考信号），帮助接收端判断自己在什么位置、信道好不好。5G物理层的全部工作，就是把这些格子管理好、分配好、使用好。

截至2026年，3GPP Release 20在物理层引入了更多AI增强技术，包括AI/ML辅助的信道估计和CSI压缩。物理层的智能化正在成为5G-Advanced和6G的重要趋势。

12.1 物理层信道全景

5G NR定义了6个物理层信道，分为下行和上行两大类：

信道 全称 方向 功能
PDSCH Physical Downlink Shared Channel 下行 承载用户数据（下行用户面数据）和部分系统消息（如寻呼、RAR）。是下行吞吐量的主力信道
PDCCH Physical Downlink Control Channel 下行 承载DCI（下行控制信息），包括调度分配、上行授权、功率控制等。UE接收数据前必须先解出PDCCH
PBCH Physical Broadcast Channel 下行 承载MIB（主信息块），包含SSB的时频位置、系统帧号、子载波间隔等小区基本信息
PUSCH Physical Uplink Shared Channel 上行 承载用户上行数据和控制信息（如CSI报告、HARQ ACK）。上行吞吐量的主力
PUCCH Physical Uplink Control Channel 上行 承载上行控制信息（UCI）：包括HARQ ACK/NACK、SRI（调度请求）、CSI报告。多种格式支持不同的负载大小
PRACH Physical Random Access Channel 上行 承载随机接入前导码（Preamble），用于初始接入、切换、RRC重建、波束失败恢复等场景

类比理解：PDSCH/PUSCH是"高速公路"（运货多），PDCCH是"交通指示牌"（通知前方怎么走），PBCH是"路标"（告诉你在哪条路上），PUCCH是"收费站反馈"（告诉调度器路好不好走），PRACH是"入口匝道"（车辆如何进入高速）。

12.2 传输信道到物理信道的映射

在5G协议栈中，MAC层向物理层传递的是"传输信道"（Transport Channel），物理层将传输信道映射到物理信道上进行实际传输。这种分层设计使得上层协议与具体的物理传输方式解耦。

下行方向：DL-SCH（下行共享信道）→ PDSCH，BCH（广播信道）→ PBCH，PCH（寻呼信道）→ PDSCH（承载寻呼消息）。上行方向：UL-SCH（上行共享信道）→ PUSCH，RACH（随机接入信道）→ PRACH。

12.3 参考信号体系

5G NR的参考信号比4G LTE丰富得多。每种参考信号都有特定的用途：

  • DMRS（解调参考信号）：伴随数据信道和控制信道一起传输，用于信道估计和解调。PDSCH DMRS、PUSCH DMRS、PDCCH DMRS、PBCH DMRS各自独立配置。DMRS的位置和密度可以根据信道条件动态调整——高速场景增加密度，静止场景降低密度。
  • SRS（探测参考信号）：UE发送给基站的上行参考信号，用于基站估计上行信道质量。在TDD系统中，由于信道互易性，SRS也可以用来间接估计下行信道。SRS的频域位置可以灵活配置，支持"跳频"方式测量全带宽的信道。
  • CSI-RS（信道状态信息参考信号）：基站发送给UE的下行参考信号，用于UE测量CSI（CQI/PMI/RI/LI）。CSI-RS可以配置多个资源集，支持波束管理和信道测量。在Massive MIMO中，CSI-RS的波束扫描是波束管理的基础。
  • PT-RS（相位跟踪参考信号）：用于补偿相位噪声。在高频段（毫米波）和256QAM调制下，相位噪声非常严重，PT-RS帮助接收端跟踪和校正相位误差。

5G vs 4G：4G LTE只有CRS（小区特定参考信号）和UE-RS（UE特定参考信号）。CRS始终全带宽广播，造成大量不必要开销。5G NR完全取消了CRS，所有参考信号都是"按需发送"的——DMRS只在有数据传输时出现，CSI-RS只在配置了CSI测量时发送。这种"按需参考信号"的设计，让5G NR在低负载时非常省电，在高负载时把资源全部留给数据。

12.4 资源网格

OFDM的时频资源以"资源网格"的形式组织：

  • RE（Resource Element）：最小的时频资源单元，对应1个子载波 × 1个OFDM符号。
  • RB（Resource Block）：频域上12个连续的子载波，时域上通常为14个OFDM符号（1个时隙）。RB是资源分配的最小粒度。
  • BWP（Bandwidth Part）：频域上一组连续的RB，是UE实际工作的带宽范围。一个小区可以配置多个BWP，但一个UE同一时间只在激活的BWP上工作。

下面是一个简化的资源分配示例：

// 5G NR PDSCH资源分配示例（频域Type 0）
// 资源指示方式：Bitmap + RBG Size
// 系统带宽：100 MHz, SCS=30 kHz => 273个RB

#define N_RB 273
#define RBG_SIZE 16  // 资源块组大小（取决于BWP Size和配置）
#define N_RBG (N_RB / RBG_SIZE)  // 18个RBG

// RBG Bitmap: 哪个RBG被分配给UE，哪个就是1
uint8_t rbg_bitmap[N_RBG] = {
    1, 1, 0, 1, 1, 1, 0, 0, 1,  // RBG 0-8
    1, 0, 1, 1, 0, 1, 1, 1, 0   // RBG 9-17
};

// 计算分配的RB总数
int total_rb = 0;
for (int i = 0; i < N_RBG; i++) {
    if (rbg_bitmap[i]) {
        total_rb += RBG_SIZE;
    }
}
// 该UE被分配了 13 * 16 = 208 个RB
// 对应频域带宽 = 208 * 12 * 30kHz = 74.88 MHz

// 时域上：起始符号位置S + 长度L (SLIV编码)
int S = 2;  // 从第2个OFDM符号开始
int L = 12; // 连续12个符号
// SLIV = 14 * (L - 1) + S = 14 * 11 + 2 = 156

12.5 SSB结构

SSB（Synchronization Signal Block）是UE在初始接入时检测的第一个信号，包含三个组成部分：

  • PSS（主同步信号）：占用127个RE，使用m序列，携带物理小区ID的模3信息（0,1,2）。UE通过PSS实现时域和频域的初步同步。
  • SSS（辅同步信号）：也占用127个RE，使用m序列，与PSS一起确定物理小区ID（共1008个）。
  • PBCH DMRS + PBCH：承载MIB，包括系统帧号（SFN）、SSB索引、子载波间隔、CORESET 0的配置等关键信息。PBCH使用QPSK调制和Polar码。

SSB在时域上占据4个OFDM符号，频域上占据20个RB（240个RE，包括保护间隔）。SSB的周期性可配置为5ms、10ms、20ms、40ms、80ms或160ms，默认是20ms。UE开机后就在中心频率上搜索SSB——如果找到了，就能获取小区的所有基本配置。

12.6 CORESET与搜索空间

UE如何知道PDCCH在哪里？答案是CORESET（Control Resource Set）和搜索空间（Search Space）。

CORESET定义了PDCCH在时频域上的资源位置：频域上是一组连续的RB，时域上是1-3个连续的OFDM符号（通常在第1-3个符号，即"符号头"位置）。CORESET内的PDCCH以CCE（Control Channel Element）为单位组织，每个CCE对应6个REG（Resource Element Group），每个REG等于1个RB × 1个OFDM符号。

搜索空间（Search Space）定义了UE在哪些时间点去CORESET中监听PDCCH。搜索空间分为Common Search Space（所有UE共用，用于寻呼、RAR、SIB1等公共消息）和UE-specific Search Space（每个UE独有，用于专属调度）。UE通过搜索空间周期性（如每1个时隙、每2个时隙）监听PDCCH，检查DCI中的RNTI（无线网络临时标识）是否匹配自己的标识。

BD次数限制：UE在每个时隙内能做的PDCCH盲检测（Blind Decode，BD）次数是有限制的（例如SCS=30kHz时最多36次）。这意味着基站在设计CORESET时不能无限制地增加CCE数量和聚合等级——UE算不过来。这也是为什么CORESET和Search Space的配置需要精心计算，不能随意设置。

12.7 踩坑实录：SRS配置错误导致波束赋形失效

2025年初，某芯片厂商在测试自家5G终端芯片时发现了一个令人困惑的问题：终端在实验室的衰落模拟器测试中，下行吞吐量始终比参考终端低30%。物理层各项指标检查下来，PDSCH的BLER偏高，说明解调质量不好。

仔细分析了基站侧的信道估计结果后发现：基站使用SRS估计出来的上行信道与下行信道的实际到达角不一致。在TDD系统中，理论上行和下行信道是互易的（同一个频率，不同时隙），基站应能用SRS准确估计下行信道方向。但现在上行估计的到达角是30度，而下行最优波束方向却是45度——差了15度！

最终定位到根因：SRS资源在频域上的配置过于集中——只配置了BWP中心位置的8个RB。由于信道的频率选择性，这8个RB上的信道响应不能代表整个BWP的信道特征。而基站的波束赋形是针对全带宽计算的，用局部信道的估计值来做全带宽波束赋形，方向当然不准。

解决方案：增大SRS的带宽配置，从8个RB扩展到48个RB，并启用SRS跳频（在多个时隙内测量全带宽的信道）。调整后，下行波束方向误差从15度降到了1度以内，吞吐量恢复正常。

教训：TDD的"信道互易性"并不是天然的数学对等——它的前提是"足够多的信道采样"。配置SRS时，一定要保证SRS的频域覆盖范围至少等于BWP宽度的1/4以上。特别是在信道频率选择性很强的场景（如室内多径丰富），局部SRS测量得出的波束方向很可能是错的。记住一个原则：测量宽度决定波束精度。

物理层提供了传输的"管道"和"路标"，但谁来告诉物理层"什么时候用什么管道、怎么用、给谁用"？答案是MAC层。第13章我们进入5G协议栈的MAC和RLC层——这些"管理者"负责调度、排序、重传，确保数据有序可靠地传输。
