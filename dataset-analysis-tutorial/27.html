第27章：前沿基准数据集巡览

2025 年，某顶级 AI 实验室在 MMLU 上取得了 SOTA 级别的成绩——这在内部被视为"对标 GPT-5"的标志性节点。团队欢欣鼓舞，PR 稿都写好了。但当他们把同一个模型放到 MathNet——一个全新的奥数推理基准上——准确率暴跌到 55%。不是代码 bug，不是模型退化，而是模型在 MMLU 上"记住"了答案模式（MMLU 的题目是静态的、有标准答案的、在社区广泛流传的），却在真正需要推理的题目面前现了原形。

这个故事的教训是深刻的：你选择的基准决定了你看到的是真相还是幻觉。如果只盯着几个"热门"基准，你很可能在优化一个"排名游戏"而非真正的能力提升。这也是为什么我们需要一章专门来讨论前沿基准数据集——不是为了追逐榜单，而是为了理解如何用正确的尺子丈量模型的能力。

基准幻觉：不要用单一的基准分数来判断模型好坏。MMLU 89% 和 MathNet 55% 可能出自同一个模型——前者反映的是知识记忆能力，后者反映的是推理能力。你在选择基准时，实际上是在选择"我要看到模型的哪一面"。

2025-2026 新兴基准全景

2025-2026 年的基准设计呈现出几个明显的趋势转变：

  • 从大规模到抗污染：静态数据集（如 MMLU、HellaSwag）的"寿命"越来越短——模型发布一个月后，评测分数就开始因数据泄露而虚高。新的基准引入动态生成、非公开数据和计时机制。
  • 从表面匹配到推理深度：GLUE/SuperGLUE 时代的"模式匹配"任务正在被真正的推理任务取代。模型需要展示多步推理能力，而非简单的语义相似度匹配。
  • 从英语中心到多语言覆盖：非英语语言（尤其是中文、阿拉伯语、东南亚语言）的基准数量在 2025 年增长了近 3 倍。
  • 从静态到时间演化：部分基准开始引入"时间维度"，评测模型处理时效性信息（新闻、实时事件）的能力。

推理深度基准

推理能力是 2026 年衡量大模型的核心维度。单纯的语言建模能力已经被充分 solve，真正的分野在于模型能否进行多步逻辑推理。

MathNet：最硬的数学推理试金石

MathNet 在 ICLR 2026 上发表，立刻成为了"模型能力的试金石"。它包含了来自 47 个国家、17 种语言的奥数竞赛题目，并且设计了专门的抗污染机制——每道题都有多个"语义等价的变体"，模型不能通过记忆答案来得分。GPT-5 只能拿下 69.3%，Claude 4 Opus 约 65%，开源模型的最佳成绩（Llama 4）约为 52%。MathNet 上每提升 1 个百分点，都是模型推理能力的真正进步。

from datasets import load_dataset

# 加载 MathNet 数据集
mathnet = load_dataset("mathnet/mathnet", split="test")
sample = mathnet[0]

print(sample["question"][:200])
# "设 a, b, c 是实数，满足 a+b+c=1 且 a²+b²+c²=2。
#  求 a³+b³+c³ 的最小可能值。"

print(sample["difficulty"])   # "hard"
print(sample["origin"])       # "中国数学奥林匹克 2022"
print(sample["variants"])     # 3 个等价变体
实用技巧：MathNet 的"变体机制"是一个很好的抗污染设计参考。它的做法是：对每道母题，通过替换数字、重排条件、变换等价表述生成多个变体。评测时从变体池中随机抽取一个，确保模型即使见过母题也无法直接背答案。如果你在构建自己的评测集，可以借鉴这个思路。

RMIR：推理式图像检索

RMIR（Reasoning-based Multi-modal Image Retrieval）在 CVPR 2026 上发布。与传统的图文检索（给定文本查询→返回最相关图片）不同，RMIR 要求模型先推理出查询中的隐含语义，再去检索。例如，查询可能是"一张照片，拍摄者正在下雨的街道上撑伞，但照片的主题不是人而是建筑"——模型需要理解拍摄者、撑伞、下雨街道是场景描述，而真正的检索目标是建筑。

截至 2026 年 6 月，最佳模型在 RMIR 上的准确率仅为 46.53%（随机猜测为 12.5%），说明这个基准远未被 solve。

PlantInquiryVQA：诊断级植物病理

PlantInquiryVQA 是一个农业 AI 领域的专业基准。与通用 VQA 不同，它要求模型回答（1）植物病变类型，（2）病变严重程度，（3）推荐治疗方案。有些问题需要模型结合多张图片（健康叶片 vs 病变叶片）进行对比推理。所有答案都经过植物病理学专家验证——不是简单的"匹配"，而是诊断级别的 QA。

启发：如果你的模型在 MathNet 上表现不错但在 PlantInquiryVQA 上很差，不要急着认为是农业领域知识不足。检查你的模型是否真的擅长"对比推理"——PlantInquiryVQA 中三分之二的问题需要同时分析两张图片，而很多多模态模型在"对比两幅图"这类任务上仍然表现不佳。

Agent 与代码基准

Agent 和代码能力是 2026 年评测的另一大核心维度。这些基准评测的不再是"模型知不知道"，而是"模型能不能做事"——能否理解复杂指令、能否正确调用工具、能否在真实环境中完成工程任务。

Terminal-Bench 2.0

Terminal-Bench 2.0 是一个专门评测 Agent 终端操作能力的基准，包含 89 个真实的 CLI 任务——从文件操作、Git 命令、Docker 管理，到数据库查询、网络诊断。Agent 需要在真实的 Linux 环境中执行命令并验证结果。

# Terminal-Bench 2.0 的一个典型任务
{
  "task_id": "tb2_047",
  "instruction": "在当前目录下找到所有 .log 文件，
                  将它们打包成 archive.tar.gz，
                  然后删除原始 .log 文件。",
  "setup": "预先生成 15 个分散在不同子目录的 .log 文件",
  "validation": "验证 archive.tar.gz 存在且包含所有 .log 文件,
                 原始 .log 文件已被删除",
  "timeout": 120,
  "category": "文件操作"
}

截至 2026 年中，Claude 4 以 61% 的通过率领先，GPT-5 约为 57%，开源模型普遍在 30-45% 之间。Terminal-Bench 的高难度部分在于：错误的命令执行可能是不可逆的（比如误删文件），因此评估不只是"能否完成"，还有"能否安全地完成"。

SWE-Bench Pro

SWE-Bench 家族在 2026 年 1 月迎来了 Pro 版本。相比于原始的 SWE-Bench（基于 Django、Flask 等 Python 仓库的 issue 修复），SWE-Bench Pro 做了三个重大升级：

  • 多语言：从纯 Python 扩展到 Python、JavaScript、TypeScript、Rust、Java、Go 共 6 种语言
  • 任务量翻倍：从 494 个任务增长到 1865 个任务
  • 抗污染设计：所有测试用例在 2025 年 6 月之后新收集，训练数据截止日期前未公开

即便最强模型（GPT-5）也只能解决 SWE-Bench Pro 上 43.8% 的任务，说明自动编程修复仍然是一个远未被解决的问题。

BFCL v3（Berkeley Function Calling）

BFCL 是伯克利大学维护的函数调用基准，v3 版本在 2026 年 2 月发布。它测试的是模型理解 API 文档、生成正确的函数调用参数、处理嵌套函数调用和错误恢复的能力。得益于函数调用在 Agent 场景中的核心地位，BFCL 的分数与实际 Agent 系统的效果有很高的相关性。

# BFCL v3 的一个典型样本（简化格式）
{
  "question": "明天下午3点给张三发邮件，提醒他参加5点的项目评审会，
              同时在我的日历上创建一个会议事件。",
  "expected_function_calls": [
    {
      "name": "send_email",
      "arguments": {
        "to": "张三",
        "subject": "提醒：项目评审会",
        "body": "张三，提醒你明天下午5点有项目评审会，请准时参加。",
        "schedule": "2026-07-01T15:00:00"
      }
    },
    {
      "name": "create_calendar_event",
      "arguments": {
        "title": "项目评审会",
        "start_time": "2026-07-01T17:00:00",
        "end_time": "2026-07-01T18:00:00"
      }
    }
  ]
}

SWE-Bench Multilingual

与 SWE-Bench Pro 同家族的另一个重要成员专门测试模型在非英语语言环境下理解代码任务的能力。Issue 描述可能是中文、日文、西班牙文或阿拉伯文，但代码本身是英文关键词。这测试的是模型的跨语言理解能力——理解用中文描述的特殊逻辑，然后在 Python 中正确实现。

选型提醒：在选择基准评测 Agent 能力时，不要只看 SWE-Bench 分数。SWE-Bench 偏向代码调试场景，如果你的 Agent 主要做的是 CRUD 操作、数据查询、报告生成（而非代码修复），那么 Terminal-Bench 和 BFCL 的分数更有参考价值。一定要根据实际业务场景选择基准。

多模态前沿基准

多模态模型的评测从 2025 年的"能不能看图说话"进化到了"能不能跨模态推理"：

VT-Bench：视觉+表格的跨界考验

VT-Bench（Visual-Table Benchmark）测试模型同时理解图表图像和结构化表格数据的能力。例如，给模型一张销售趋势折线图和一张对应的产品表格，要求模型回答"Q3 销售额超过 100 万的品类有哪些"——这需要模型既看懂折线图的趋势，又能从表格中定位具体品类。

VT-Bench 覆盖了 14 个数据集、9 个业务领域（医疗、金融、物流、零售等），是目前覆盖最广的视觉+表格评测基准。最佳模型准确率仅 52.1%，人类基准约为 89%。

MIRAGE：专家咨询仿真

MIRAGE（Multimodal Instruction-following Reasoning Agent Evaluation）在 NeurIPS 2026 上发表。它的独特设定是：模型扮演一个"虚拟专家"，用户（评测员）带着一组图片、文档和问题来咨询。模型需要主动追问、推理、最后给出建议。这不仅测试多模态理解，还测试对话管理和长期记忆能力。

前瞻：MIRAGE 的评测范式可能会影响未来很多垂类基准的设计——"被动回答"正在被"主动咨询"取代。如果你的产品涉及客服、医疗问诊、法律咨询等场景，建议多关注这个方向。

AgMMU：农业多模态理解

AgMMU 是一个农业领域的专业多模态基准。它覆盖了从作物识别、病虫害检测到农业生产决策的全流程。与通用的农业数据集不同，AgMMU 特别强调"时序推理"——比如，根据连续三周的叶片图像，判断病情是好转还是恶化。这种时序多模态推理的能力，与通用视觉-语言模型的评测关注点很不一样。

# 加载 AgMMU 数据集
from datasets import load_dataset

agmmu = load_dataset("agmmu/agmmu", split="test")
sample = agmmu[0]

# 样本包含多张时序图片
print(len(sample["images"]))     # 3（连续三周拍摄）
print(sample["question"])
# "根据以上三张图像，判断该作物的病害是在好转还是恶化？
#  并给出你的判断依据。"

print(sample["expert_answer"])
# "恶化。第三张图像显示叶斑面积较前两周扩大约 40%，
#  且出现新的褐色坏死斑，建议立即喷施代森锰锌。"

MMMU Pro

MMMU（Massive Multi-discipline Multimodal Understanding）在 2026 年推出了 Pro 版本。相比于原版 MMMU，Pro 版本做了两件事：去除所有可能被模型记忆的题目（抗污染），以及将每道题目的推理步骤从 2-3 步增加到 5-8 步。Gemini 2.5 在 MMMU Pro 上取得了 72.1% 的目前最佳成绩——比人类专家平均 88% 仍有明显差距。

# 加载 MMMU Pro 数据集
from datasets import load_dataset

mmmu_pro = load_dataset("mmmu/mmmu_pro", split="validation")
sample = mmmu_pro[0]

print(sample["question"][:150])
# "结合以下三张电路图和电阻值表，计算 AB 两点之间的等效电阻。
#  步骤一：识别串并联关系；步骤二：简化电路拓扑；..."

print(sample["num_reasoning_steps"])  # 6
print(sample["domain"])              # "电气工程"
评测效率建议：多模态基准的评测成本通常很高（需要 GPU 推理、处理图片/视频等）。建议你列一个"评测优先级"清单：先跑轻量级基准（如 MMMU Pro 的文字部分），如果表现合格，再升级到耗资源的基准（如 VT-Bench、MIRAGE）。这样可以在资源预算内最大化评测覆盖面。

如何跟进前沿基准

前沿基准更新极快——几乎每周都有新的基准发布、旧的基准被解构。作为一个 AI 工程师，如何在不被信息淹没的前提下保持跟进？

三大信息源

  • Papers With Code：按任务/领域分类的基准排行榜，可以看到每个基准的历史趋势和 SOTA 演进路径。它的"Task"页面比"Benchmark"页面更有信息密度。
  • Hugging Face Open LLM Leaderboard v2：经过标准化的一组核心基准评测，覆盖知识、推理、代码、Agent 等多个维度。2026 年的 v2 版本已经包含了 MathNet、BFCL 等新基准，这是关注模型能力的"一站式"参考。
  • arXiv（CS.CL / CS.AI / CS.LG）：每天花 10 分钟扫一眼新论文标题，重点关注标题中包含 "Benchmark"、"Dataset"、"Evaluation" 的论文。不需要每篇都读，但保持敏感度很重要。

如何评估一个新基准是否值得关注

面对一个新发布的基准，我通常会问自己五个问题：

  1. 这个基准评测的能力，是我们关心的吗？——如果基准评测的是"3D 物体渲染质量"，而我们的产品是文本分类，那它再热门也不应该分心。
  2. 基准数据的构建过程透明吗？——数据来源是什么？标注者是谁？标注一致性如何？如果基准文档里这些信息缺失，分数可能不可靠。
  3. 基准有抗污染设计吗？——如果基准全是公开的静态数据，那么在发布后的 3-6 个月内就会失效，不值得投入精力去追踪。
  4. 基准的"区分度"够吗？——如果所有模型都在 90-95% 之间（天花板效应），或者全部在 20-30% 之间（地板效应），这个基准对模型选型帮助不大。
  5. 基准的测试集够大吗？——少于 500 个样本的基准，置信区间会非常宽，一次运行的结果可能不可靠。

选基准的 ROI 思考

最后，一个关于基准选择的"元认知"：选择基准不只是技术决策，更是战略决策。你选择的基准，决定了你的团队的优化方向。如果你的团队整天盯着 MMLU 的分数，那潜意识里在优化的就是"知识记忆"模型而非"推理能力"模型。如果你的 KPI 是 SWE-Bench 分数，那产品方向就会向代码修复倾斜——哪怕核心用户是数据分析师。选基准的 ROI 思考公式是：

基准 ROI = 评测能力与业务需求的相关性 × 基准的抗污染性 / 评测成本。 跑分最高的基准不一定是最好的基准。对你的业务场景最有"信息量"的基准，才是值得你投入时间和算力的基准。
