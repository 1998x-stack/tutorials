第25章 协议栈实现

大家好，我是你们的讲师。想象这样一个场景：你设计了一款非常漂亮的传感器节点硬件——PCB天线匹配完美、功耗低至微安级别、所有的GPIO都分配合适。你信心满满地把它部署到现场，然后发现——它和一个多月前部署的另一款节点的协议栈完全不兼容，两个系统在同一个频段上互相干扰，数据包全部碰撞掉了。你老板问你"为什么不能像手机一样自动连上WiFi"，而你只能苦笑——WSN的协议栈实现远比大多数人想象的复杂。

WSN的软件栈是连接硬件能力和网络功能的桥梁。没有好的协议栈实现，再好的硬件也只是"昂贵的砖头"。本章我们将深入四款主流的WSN操作系统和协议栈，然后动手实现一个实际可用的CoAP温度传感器系统，让你真正理解"从节点到云端"的完整数据通路。

本章的实践目标： 学完本章后，你将能够独立选择适合自己项目的WSN操作系统，理解不同协议栈的架构设计哲学，并能在RIOT OS上实现一个端到端的CoAP传感器应用。

四大WSN操作系统全景对比

WSN对操作系统的要求非常特殊：既要能在KB级内存中运行，又要提供网络协议栈、传感器驱动、任务调度和功耗管理。传统的Linux或FreeRTOS（虽然轻量但缺少标准的网络协议栈）很难满足全部需求，于是出现了专门为WSN设计的操作系统。目前最主流的四个选择是：

从这张表格可以看出，这四个系统各自有不同的设计哲学和适用场景，不存在"万能"的选择。但有一个明显的趋势：RIOT OS在近两年发展非常快，得益于它的POSIX兼容性和活跃的社区支持，越来越多的工业IoT项目选择它作为底层操作系统。

Contiki-NG深度解析

Contiki-NG（Next Generation）是原Contiki OS的继任者，也是目前WSN领域最成熟的轻量级操作系统之一。它的核心设计哲学是"在极度受限的资源下提供标准的互联网协议栈"。

Protothread协程机制

Contiki最独特的设计是Protothread——一种轻量级的协程实现。Protothread不同于完整的线程（thread），它不需要独立的栈空间（只需要一个局部静态变量保存状态），因此开销极小。一个Protothread只占用2个字节的状态存储。

Protothread的写法看起来像带阻塞的C函数，但实际上是通过宏和switch-case实现的"伪栈"。下面是一个典型的Contiki protothread：

// Contiki-NG Protothread示例：周期读取温度传感器并发送
PROCESS(temp_sensor_process, "Temperature Sensor Process");
AUTOSTART_PROCESSES(&temp_sensor_process);

PROCESS_THREAD(temp_sensor_process, ev, data) {
    static struct etimer timer;
    static int temp_value;

    PROCESS_BEGIN();

    // 初始化传感器
    sht11_init();

    while (1) {
        // 设置定时器，每30秒触发一次
        etimer_set(&timer, CLOCK_SECOND * 30);

        // 等待定时器事件——这里阻塞但不消耗CPU
        PROCESS_WAIT_EVENT_UNTIL(etimer_expired(&timer));

        // 读取温度
        temp_value = sht11_read_temperature();

        // 通过UART打印
        printf("Temp: %d°C\n", temp_value);

        // 存入REST Engine资源（供CoAP查询）
        rest_set_resource_value("sensor/temp", temp_value);
    }

    PROCESS_END();
}

看到上面的PROCESS_WAIT_EVENT_UNTIL了吗？这一行就是Protothread的精髓——函数执行到这里会"挂起"，但不会像阻塞式线程那样占用栈空间。当定时器到期时，进程被重新调度执行，继续往下走。正是这种设计让Contiki能在仅有2KB RAM的芯片上同时运行十几个并发任务。

uIP/uIPv6协议栈

Contiki内置的uIP协议栈是嵌入式互联网的先驱。它实现了完整的IPv6、TCP、UDP和ICMPv6，同时将内存占用压缩到极致——uIPv6的最小配置只需要约1.5KB RAM。此外，Contiki还提供了6LoWPAN适配层（sicslowpan），将IPv6包在IEEE 802.15.4的最大127字节帧上做分片和头部压缩。

TinyOS与nesC

TinyOS是WSN领域最早的专用操作系统之一，由UC Berkeley在Smart Dust项目期间开发。如果说Contiki的设计哲学是"用标准的C写受限的代码"，那么TinyOS的哲学就是"用专门的语言造专门的操作系统"。

TinyOS使用nesC（Network Embedded Systems C）编程语言，它是C的一个扩展，引入了组件（Component）、接口（Interface）、配置（Configuration）和任务（Task）等概念。nesC的组件模型将硬件驱动、协议模块和应用逻辑封装成独立的"组件"，通过接口连接（Wiring）的方式组装成完整的系统——有点像硬件工程师连接芯片引脚的感觉。

TinyOS的另一个核心概念是Split-Phase操作：一个耗时的操作（如发送数据包或读取ADC）被拆分成"发起"和"完成回调"两个阶段，中间CPU可以处理其他任务。这本质上是一种协作式多任务调度。举个例子——当你调用call Send.send()发送数据包时，函数立即返回，发送完成后系统回调你的event Send.sendDone()事件处理函数。

RIOT OS：现代IoT操作系统的标杆

如果说Contiki是"从零打造一个极简OS"，RIOT OS就是"把标准Linux的最佳实践移植到MCU上"。RIOT OS的设计目标非常明确：让嵌入式开发者能用他们熟悉的POSIX API编程。

这意味着什么？你在RIOT OS上可以使用真正的多线程（不是Protothread）、mutex信号量、消息队列、标准socket接口——所有你在Linux上熟悉的编程模型，在RIOT OS上几乎都能用。它的微内核设计只需要约5KB RAM，却能提供完整的多线程能力和网络协议栈。

为什么要关注RIOT OS？ 2024-2025年，RIOT OS的社区活跃度已经超过Contiki-NG，成为Github上最活跃的WSN操作系统项目。它的POSIX兼容性意味着你可以把应用层的网络代码在Linux上先调试通过，然后直接交叉编译到嵌入式目标板上，大幅缩短了开发周期。

CoAP协议：WSN的HTTP

CoAP（Constrained Application Protocol, RFC 7252）是专门为受限节点设计的应用层协议，常被称为"嵌入式HTTP"。CoAP基于UDP（而不是TCP），使用RESTful架构风格（GET、PUT、POST、DELETE），但设计上比HTTP轻量得多：最小CoAP消息头只有4个字节。

CoAP的几个关键特性：

  • 请求/响应模型：客户端发送CON（Confirmed，需确认）或NON（Non-confirmed，无需确认）请求。CON请求通过ACK重传机制保证可靠性。
  • 资源发现：CoAP支持".well-known/core"路径，客户端可以通过GET请求发现服务器上所有可用的资源列表。
  • Observe模式：客户端可以"订阅"一个资源，服务器在有新数据时主动推送（而不是客户端反复轮询）。这对WSN的能效至关重要——不浪费电池来轮询无变化的数据。

CoAP和HTTP之间还有Cross-Proxy机制——一个CoAP-to-HTTP代理可以在CoAP网络中接收传感器数据，然后通过HTTP转发到云端。这也就是你在实际IoT架构中常见的"传感器节点 ↔ 网关 ↔ 云平台"的通路。

MQTT-SN：MQTT的传感器版

如果CoAP是"嵌入式HTTP"，那么MQTT-SN就是"嵌入式MQTT"。MQTT-SN（MQTT for Sensor Networks, OASIS标准）专门针对WSN和低功耗网络进行了优化，最重要的改进是支持休眠客户端（Sleeping Client）。

在标准MQTT中，客户端需要维持一个长时间的TCP连接——这对WSN节点来说代价太高了。MQTT-SN引入了网关中间人模式：节点在休眠前向网关发送一条DISCONNECT消息，网关缓存所有订阅的消息，节点醒来时通过SEARCHGW寻找网关并接收缓存的消息。这种机制让节点可以在99%的时间处于深度休眠状态，只在必要时短暂唤醒通信。

实战：基于RIOT OS的CoAP温度传感系统

下面我们动手编写一个完整的传感器节点程序。这个程序运行在RIOT OS上，提供一个CoAP资源/temp，每次被GET请求访问时返回当前温度值。我们将它设计为一个工程级的可复用系统。

/*
 * riot_coap_temp_server.c
 * RIOT OS CoAP温度传感器服务器
 * 支持: GET /temp -> 返回当前温度
 *       GET /.well-known/core -> 资源发现
 * 硬件: 任何支持RIOT OS的板子 + 温度传感器(如DHT22或BME280)
 */

#include <stdio.h>
#include <string.h>
#include "board.h"
#include "ztimer.h"
#include "fmt.h"
#include "net/nanocoap_sock.h"
#include "net/gcoap.h"
#include "phydat.h"
#include "saul_reg.h"

#define COAP_INBUF_SIZE      (256U)
#define COAP_TEMP_PORT       (5683U)

static uint8_t coap_buf[COAP_INBUF_SIZE];
static char temp_str[16];

/* 温度传感器资源的回调函数 */
static ssize_t temp_handler(coap_pkt_t *pkt, uint8_t *buf,
                            size_t buf_len, coap_request_ctx_t *ctx) {
    phydat_t res;
    saul_reg_t *dev = saul_reg_find_type(SAUL_SENSE_TEMP);

    if (dev == NULL) {
        return gcoap_response(pkt, buf, buf_len, COAP_CODE_INTERNAL_SERVER_ERROR);
    }

    /* 读取传感器 */
    if (saul_reg_read(dev, &res) < 0) {
        return gcoap_response(pkt, buf, buf_len, COAP_CODE_INTERNAL_SERVER_ERROR);
    }

    /* 格式化温度值：例如 "25.3°C" */
    size_t n = fmt_s16_dfp(temp_str, res.val[0], -2);
    temp_str[n] = '\0';

    /* 构造CoAP 2.05响应 */
    size_t resp_len = gcoap_response(pkt, buf, buf_len, COAP_CODE_CONTENT);
    if (resp_len) {
        coap_hdr_set_content_format(pkt, COAP_FORMAT_TEXT_PLAIN);
        uint8_t *payload_ptr = coap_pkt_get_payload(pkt);
        if (payload_ptr) {
            memcpy(payload_ptr, temp_str, strlen(temp_str));
            pkt->payload_len = strlen(temp_str);
            resp_len += pkt->payload_len;
        }
    }
    return resp_len;
}

/* 注册CoAP资源 */
static coap_resource_t resources[] = {
    { "/temp", COAP_GET, temp_handler, NULL },
};

/* 主函数 */
int main(void) {
    puts("RIOT OS CoAP Temperature Server");
    puts("Initializing sensors...");

    /* 初始化SAUL传感器抽象层 */
    saul_reg_init();

    /* 启动CoAP服务器 */
    gcoap_init_listen();
    gcoap_register_resources(resources, ARRAY_SIZE(resources));

    char addr_str[64];
    ipv6_addr_t addr = { 0 };
    netif_get_global(1, &addr, NULL, 0);
    ipv6_addr_to_str(addr_str, &addr, sizeof(addr_str));
    printf("CoAP server listening on [%s]:%d\n", addr_str, COAP_TEMP_PORT);

    /* 主循环：打印传感器数据，等待CoAP请求 */
    while (1) {
        ztimer_sleep(ZTIMER_MSEC, 5000);      // 每5秒主动打印一次
        phydat_t sample;
        saul_reg_t *dev = saul_reg_find_type(SAUL_SENSE_TEMP);
        if (dev && saul_reg_read(dev, &sample) == 0) {
            printf("  Temp: %d.%02d °C\n", sample.val[0] / 100,
                   abs(sample.val[0] % 100));
        }
    }
    return 0;
}
/*
 * riot_coap_temp_client.c — 配套的CoAP客户端
 * 查询远程节点的/temp资源
 * 编译: BOARD=<your_board> make -C . all term
 */

#include <stdio.h>
#include "net/nanocoap_sock.h"
#include "net/gnrc/ipv6/netif.h"

#define COAP_SERVER_ADDR  "2001:db8::1"
#define COAP_TEMP_PORT    "5683"
#define COAP_TEMP_URI     "/temp"

int main(void) {
    printf("CoAP Temp Client: fetching %s\n", COAP_TEMP_URI);

    /* 构造远程URI */
    char uri[64];
    snprintf(uri, sizeof(uri), "coap://[%s]:%s%s",
             COAP_SERVER_ADDR, COAP_TEMP_PORT, COAP_TEMP_URI);

    /* 发起CoAP GET请求 */
    uint8_t buf[128];
    ssize_t len = nanocoap_get(uri, buf, sizeof(buf));
    if (len < 0) {
        printf("Error: CoAP GET failed (%d)\n", (int)len);
        return 1;
    }

    /* 打印响应内容 */
    buf[len] = '\0';
    printf("Response: %s\n", (char *)buf);
    return 0;
}

这个系统实现了以下功能：客户端通过CoAP协议查询远程传感器节点的温度值，节点返回类似于25.3的纯文本响应。你可以把客户端运行在另一个节点上、一个树莓派网关上、甚至直接在你的电脑上——只要网络可达，CoAP协议的跨平台兼容性非常出色。

踩坑故事：Contiki内存泄漏的48小时

讲一个我亲身经历的故事。几年前我用Contiki-NG做一个环境监测项目，20个节点布在外面采集温湿度。刚开始测试的12小时一切正常——数据稳定上报，RPL路由也收敛得很好。结果到了第24小时左右，问题开始出现了：部分节点突然停止响应，重启后恢复正常，但再过十几个小时又死了。

这个问题断断续续排查了整整两天。我开始怀疑是硬件问题——是不是哪颗电容坏了？射频功放过热了？甚至在示波器上蹲了俩小时看波形。后来切换到软件日志模式，用了Contiki的energest功耗追踪模块，发现节点在死之前的剩余内存（free memory）在持续下降。原来是一个同事在代码中每次发送数据包时调用malloc分配了一个缓冲区，却忘记对应添加free。每次发送泄漏几十字节，24小时大约发了5000次，泄漏了约200KB——而CC2538的RAM只有32KB，每一个小时左右就要溢出一次，只是数据在Flash里部分缓存掩盖了前几次崩溃。找到原因后把malloc换成静态分配，节点稳定运行了90天不间断。教训是：在没有MMU的嵌入式系统中，动态内存分配是万恶之源——能用静态分配就别用动态分配，必须用的时候一定要配对free，最好配一个内存池（Memory Pool）管理机制。

黄金法则： WSN嵌入式开发中，永远不要在中断服务函数（ISR）中调用malloc。永远不要在热路径（每包都走的代码路径）中调用malloc。如果一定要用动态内存，用static数组预分配pool，实现自己的分配器。

协议栈选择的决策框架

面对这么多选择，如何为你的项目选出正确的协议栈？以下是我的决策框架：

  • 你需要IPv6吗？ 如果节点需要直接与互联网通信（端到端IP），选Contiki-NG或RIOT OS。如果数据先汇聚到网关再上云，FreeRTOS+自定义足以。
  • 你的RAM有多少？ 少于8KB？TinyOS或裸机+手写协议可能是唯一选择。8-32KB？Contiki-NG是最佳平衡。大于64KB？RIOT OS的POSIX接口优势非常明显。
  • 你的团队熟悉什么？ 如果团队有Linux/POSIX背景，RIOT OS的学习曲线最平；如果有嵌入式C背景但不熟悉OS，FreeRTOS最稳妥；如果纯学术界，TinyOS仍然有大量参考代码。
  • 你需要什么应用层协议？ 需要CoAP？Contiki和RIOT都有原生支持。需要MQTT-SN？最好在FreeRTOS或RIOT上集成第三方库。需要自定义二进制协议？选什么OS都行，裸机也能做。
我的推荐（截至2026年）： 新项目首选RIOT OS + nanocoap/gcoap，这套组合的POSIX兼容性、社区活跃度和协议栈成熟度是目前WSN领域最平衡的选择。如果你在做一个超低成本的ZigBee产品且RAM极度受限，CC2538 + Z-Stack（TI的专有协议栈）比任何开源方案都省心。
