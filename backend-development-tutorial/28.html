开篇：学完 28 章，我们来"交作业"

到这里，我们共同走过了 27 章的知识旅程。从第一天的"什么是后端开发"到上周的"云原生前沿技术"，你已经接触了后端工程师所需的核心知识图谱——网络协议、数据库、缓存、消息队列、安全、可观测性、容器编排、AI 集成，再到最新的 Wasm 和 eBPF。

但中国有句老话叫"纸上得来终觉浅"。这一章我们不学新概念，我们来"交作业"——把前面 27 章的知识串联起来，从零搭建一个完整的微服务后端系统。

我们要做的项目是一个"在线教育平台"的后端系统。我之所以选这个场景，是因为它包含了大多数后端系统都会遇到的典型问题：用户管理、内容管理、订单支付、实时互动、推荐搜索、数据分析——几乎覆盖了本课程讨论过的每一个技术维度。

我自己职业生涯中做过的最复杂的一个项目也是一个教育平台。当时团队 6 个人用了 4 个月上线了第一版，然后又花了两年不断重构。踩过最大的坑是"领域划分不合理"——把"课程"和"直播"放在同一个服务里，结果直播流量暴涨时影响了整个课程服务的可用性。如果当时我们按照下面这套架构来设计，至少能省下半年的重构时间。

项目需求分析

核心功能需求

模块 功能 优先级 涉及技术
用户服务 注册/登录/角色管理/个人中心 P0 JWT、OAuth 2.0、PostgreSQL
课程服务 课程CRUD/章节管理/视频上传 P0 PostgreSQL、Redis 缓存、对象存储
订单服务 下单/支付/退款/发票 P0 Kafka、分布式事务、支付网关
学习服务 学习进度/笔记/测验 P1 MongoDB（灵活 schema）、Redis
直播服务 实时课堂/聊天/白板 P1 WebSocket、RTMP、gRPC
搜索推荐 课程搜索/个性化推荐 P2 Elasticsearch、向量数据库
数据分析 学习统计/转化分析/报表 P2 ClickHouse、Flink

非功能需求

  • 可用性：核心链路 99.95%（课程浏览、下单支付），允许用户无感知的故障切换
  • 性能：课程列表 API P99 < 200ms，直播延迟 < 2s
  • 可扩展：支持万人同时在线听课，服务可按需水平扩展
  • 安全：所有 API 必须鉴权，支付敏感信息加密存储，日志脱敏
  • 可观测：全链路追踪 + 黄金指标告警 + 统一日志

架构设计：微服务拆分、API 网关、事件驱动

整体架构采用"API 网关 + 微服务 + 事件驱动"的模式。每个服务独立部署、独立数据库、通过 API 网关对外暴露统一的入口。

┌─────────────────────────────────────────────────────┐
│                   客户端 (Web / App)                   │
└──────────────────┬──────────────────────────────────┘
                   │ HTTPS
┌──────────────────▼──────────────────────────────────┐
│               API Gateway (Kong / Envoy)              │
│     ┌──────────────────────────────────────────┐     │
│     │  限流 / 鉴权 / 路由聚合 / 协议转换 / WAF  │     │
│     └──────────────────────────────────────────┘     │
└──┬─────────┬─────────┬─────────┬─────────┬──────────┘

┌──▼──┐ ┌───▼───┐ ┌───▼───┐ ┌───▼───┐ ┌──▼──────┐
│用户  │ │课程   │ │订单   │ │学习   │ │直播     │
│服务  │ │服务   │ │服务   │ │服务   │ │服务     │
└──┬──┘ └───┬───┘ └───┬───┘ └───┬───┘ └──┬───────┘

              │ (ClickHouse) │     │ (ES + 向量库)  │
              └─────────────┘     └───────────────┘

核心设计决策：

  • 服务间通信：同步调用走 gRPC（内部通信高效），异步解耦走 Kafka（事件驱动）。比如"用户下单"这个操作是同步返回支付 URL，但"下单成功后的积分发放、课程开通、通知推送"都是异步事件。
  • API 网关职责：统一认证鉴权、速率限制、路由转发、请求/响应转换。业务服务不需要重复实现这些横切关注点。
  • 数据库隔离：每个服务拥有自己的数据库实例（或 schema），禁止跨服务直接访问数据库——必须通过服务 API。这是微服务架构的红线。
血泪教训：我见过一个团队在微服务上线半年后，因为"跨服务查数据太麻烦"而在订单服务中直接连接了用户数据库。三个月后用户库做了分库分表，订单服务直接宕机了两天。记住：微服务之间共享数据库就是回到单体——所有的好处（独立部署、独立扩展、独立技术栈）全都没了。如果跨服务查询很频繁，考虑通过事件同步一份"只读副本"或使用 CQRS 模式。

容器化与编排：Docker multi-stage build + K8s Manifest

每个微服务都用 Docker multi-stage build 构建，确保最终镜像最小化、不包含任何构建工具或源码。

# Dockerfile — 以课程服务为例（Go 实现）
# ===== Stage 1: Build =====
FROM golang:1.24-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server ./cmd/course-service

# ===== Stage 2: Runtime =====
FROM alpine:3.21
RUN apk --no-cache add ca-certificates tzdata
WORKDIR /app
# 只复制编译好的二进制
COPY --from=builder /app/server .
# 复制配置文件
COPY --from=builder /app/configs/prod.yaml ./configs/
EXPOSE 8080
USER nobody
ENTRYPOINT ["./server", "--config=configs/prod.yaml"]
# deployment.yaml — K8s 部署清单（课程服务）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: course-service
  namespace: edu-platform
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0    # 零停机部署
      maxSurge: 1
  selector:
    matchLabels:
      app: course-service
  template:
    metadata:
      labels:
        app: course-service
    spec:
      containers:
      - name: course-service
        image: registry.edu.cn/course-service:v1.0.0
        ports:
        - containerPort: 8080
          name: grpc
        - containerPort: 9090
          name: metrics
        env:
        - name: DB_DSN
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: course-db-dsn
        resources:
          requests:
            cpu: "500m"
            memory: "512Mi"
          limits:
            cpu: "1000m"
            memory: "1Gi"
        livenessProbe:
          grpc:
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 15
        readinessProbe:
          grpc:
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
生产技巧：使用 maxUnavailable: 0 和 maxSurge: 1 实现滚动更新零停机。再加上 PodDisruptionBudget（minAvailable: 2）保障集群节点维护时服务的可用性。这是生产级部署的基本配置。

CI/CD 流水线：GitHub Actions → Docker Build → ArgoCD → K8s

我们用 GitOps 的方式管理部署：代码推送到主分支后自动构建镜像、推送镜像仓库、更新 GitOps 仓库中的 manifest，ArgoCD 自动同步到 K8s 集群。

# .github/workflows/deploy.yaml
name: Build & Deploy

on:
  push:
    branches: [main]
    paths:
      - 'services/course/**'

env:
  REGISTRY: registry.edu.cn
  IMAGE_NAME: course-service

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.24'
      - run: go test ./services/course/... -v -race -cover

  build-and-push:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ secrets.REGISTRY_USER }}
          password: ${{ secrets.REGISTRY_PASS }}

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: services/course
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest

      - name: Generate SBOM
        uses: anchore/sbom-action@v0
        with:
          image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

      - name: Update GitOps manifest
        run: |
          # 自动更新 GitOps 仓库中的镜像版本
          git clone https://x-access-token:${{ secrets.GITOPS_TOKEN }}@github.com/org/edu-gitops.git
          cd edu-gitops
          sed -i "s|image:.*course-service.*|image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}|g" \
            services/course/deployment.yaml
          git config user.name "CI Bot"
          git config user.email "ci@edu.cn"
          git commit -am "chore: bump course-service to ${{ github.sha }}"
          git push
# ArgoCD Application manifest（应用自动同步）
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: course-service
  namespace: argocd
spec:
  project: edu-platform
  source:
    repoURL: https://github.com/org/edu-gitops.git
    targetRevision: main
    path: services/course
  destination:
    server: https://kubernetes.default.svc
    namespace: edu-platform
  syncPolicy:
    automated:
      prune: true      # 自动清理废弃资源
      selfHeal: true   # 自动修复手动修改
    syncOptions:
    - CreateNamespace=true

这套流水线的核心优势在于"不可变部署"：每次部署的镜像版本由 Git commit SHA 唯一标识，你可以随时追溯到构建出这个镜像的代码版本、GitHub Actions 日志、以及测试结果。当线上出问题时，回滚只需要在 GitOps 仓库中 revert 一次 commit。

可观测性接入：OpenTelemetry → Grafana Stack

没有可观测性的系统就是"黑箱"。我们为整个平台统一接入 OpenTelemetry，将指标、日志和链路追踪全部上报到 Grafana Stack。

// Go: 使用 OpenTelemetry 为课程服务添加全链路追踪
import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/trace"
)

var tracer = otel.Tracer("course-service")

func GetCourseDetail(ctx context.Context, courseID string) (*Course, error) {
    ctx, span := tracer.Start(ctx, "GetCourseDetail",
        trace.WithAttributes(
            attribute.String("course.id", courseID),
        ),
    )
    defer span.End()

    // 从缓存读取
    cacheSpan := trace.SpanFromContext(ctx)
    cached, err := cache.Get(ctx, "course:"+courseID)
    if err == nil && cached != nil {
        span.AddEvent("cache_hit")
        return cached.(*Course), nil
    }
    span.AddEvent("cache_miss")

    // 从数据库读取
    dbSpan, dbCtx := tracer.Start(ctx, "db.query")
    course, err := db.GetCourse(dbCtx, courseID)
    dbSpan.End()
    if err != nil {
        span.RecordError(err)
        span.SetStatus(codes.Error, err.Error())
        return nil, err
    }

    // 写入缓存
    cache.Set(ctx, "course:"+courseID, course, 5*time.Minute)
    return course, nil
}

接入 OpenTelemetry 后，你可以在 Grafana 中看到：

  • Grafana Tempo：每个请求的完整调用链——从 API 网关→课程服务→Redis→PostgreSQL，每一跳的耗时和状态一目了然
  • Grafana Loki：聚合所有服务的日志，通过 trace ID 将日志和链路关联起来
  • Grafana Mimir（或 Prometheus）：采集 Redis 命中率、DB 连接池使用率、P99/P95 延迟等黄金指标，配上告警规则
起步建议：不要试图一次接入所有可观测性组件。建议按这个顺序：指标 → 告警 → 日志 → 链路追踪。先让你能够回答"系统正常吗"和"出问题时哪里出错了"，再精细化地做链路分析。很多人一上来就上全链路追踪，结果在数据噪音中迷失方向。

课程总结与下一步学习建议

28 章，到此画上句号。

让我先和你一起回头看看我们走过的路：

  • Part 1（01-07章）：我们从后端开发的全景图出发，打下了网络协议、Linux、版本控制、编程语言、数据结构和设计模式的基础。这是后端工程师的"基本功"——不管做多少年，这些知识都是你的底牌。
  • Part 2（08-14章）：进入了 API 和数据的世界——RESTful、认证授权、关系型数据库、NoSQL、消息队列、gRPC、GraphQL。这是后端工程师的"日常战场"。
  • Part 3（15-18章）：关注质量和安全——测试策略、安全攻防、可观测性、CI/CD。这些是区分"能写代码"和"能交付可靠软件"的分水岭。
  • Part 4（19-22章）：容器化和编排——Docker 和 Kubernetes。这是现代后端基础设施的"标配"。
  • Part 5（23-25章）：运维进阶——GitOps、可观测性进阶、Serverless。让你从"能用 K8s"进化为"高效运维"。
  • Part 6（26-28章）：前沿与实战——AI 驱动开发、云原生前沿技术，以及刚刚完成的这个综合项目。这是面向未来的视野和"毕业设计"。
给读者的最后几句话：第一，不要被知识广度吓到。你不需要"熟练掌握"这 28 章里的每一个技术细节才能称自己是后端工程师。精通 3-4 个领域、了解 10-15 个领域、对剩下的有基本认知——这才是真实世界中资深工程师的知识分布。第二，持续动手。这门课程给了你地图和路线，但路要你自己走。尝试用不同的编程语言重写第 28 章的项目，尝试把它部署到真正的云环境上，尝试给它加上第 26 章学到的 AI 推荐功能。第三，保持好奇心。我在第 1 章说过，后端开发是"看不见的冰山"。做这一行最大的乐趣在于：你永远看得不够深。每一次挖下去，都能发现新的世界——从理解一个 HTTP 请求的背后发生了什么，到理解 CPU 的分支预测如何影响你的代码性能。这条路没有终点。
