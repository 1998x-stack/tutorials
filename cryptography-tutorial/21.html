第21章 安全多方计算

假设有两个亿万富翁，Alice 和 Bob。他们想在不动用自己真实财富的前提下，比较一下谁更有钱。但问题来了——Alice 不想告诉 Bob 自己有多少亿，Bob 也不想告诉 Alice。他们能不能只交换一些信息，就确定到底谁的财富数字更大，同时不泄露任何一方的具体金额？这就是安全计算史上最著名的故事——姚氏百万富翁问题（Yao's Millionaires' Problem）。

安全多方计算（Secure Multi-Party Computation, MPC）从这个问题出发，发展成为密码学中最实用的隐私计算框架之一。它回答的核心问题是：多个互不信任的参与方如何在不泄露各自私有输入的前提下，协同计算一个约定的函数？如果说同态加密是把计算交给"不信任的服务器"，那安全多方计算就是让多个"互不信任的对等方"共同完成计算。

🌟 MPC 的精髓：每个参与方都有自己的隐私输入，大家在一起执行一个协议后，每个人只得到输出结果，而无法获知其他方的输入。就像你和三个同事一起计算平均工资——最后知道平均数是 35K，但谁也不知道谁拿了多少。

21.1 姚氏百万富翁问题与乱码电路

1982 年，华裔计算机科学家姚期智（Andrew Chi-Chih Yao）提出了百万富翁问题，并给出了一种基于 乱码电路（Garbled Circuit） 的解决方案。这个故事被认为是安全多方计算的起源。

姚期智的方案——今天称为 Yao's Protocol 或混淆电路协议——适用于两方场景，具体工作方式如下：

// Yao's Garbled Circuit (两方场景)
//
// 假设 Alice 和 Bob 要比较谁更富:
// Alice 有财富 a (比如 4 亿), Bob 有财富 b (比如 2 亿)
// 他们想计算 f(a, b) = (a > b ? "Alice" : "Bob")
//
// Alice (电路生成者):
// 1. 将比较函数转换为布尔电路（逐位比较器）
// 2. 为电路中的每一根导线生成两个随机密钥:
//    - 标签 0: 表示该导线为 0 的随机值
//    - 标签 1: 表示该导线为 1 的随机值
// 3. 对每个门电路，用真值表中的值加密对应的导线标签
//    形成"乱码表"（Garbled Table）
// 4. 将整个乱码电路发送给 Bob
// 5. 将自己的输入 a 的标签（通过混淆后的值）发给 Bob
//
// Bob (电路计算者):
// 1. 通过不经意传输得到自己输入 b 的标签
// 2. 用收到标签逐步解密乱码门电路
// 3. 最终得到比较结果的明文（一个比特）
// 4. 把结果告诉 Alice
//
// Bob 在计算过程中完全看不到电路的实际值，
// 只能看到随机的标签在流动。

这个方案的天才之处在于：Bob 可以用 Alice 生成的"乱码电路"正确计算结果，但从 Bob 的视角看，所有数据都是随机标签，没有任何有意义的信息泄露。而 Alice 则完全不知道 Bob 用的是什么输入标签——因为 Bob 是通过不经意传输获取的，Alice 不知道 Bob 选择了哪个标签。

21.2 秘密共享

如果说乱码电路是 MPC 的一条路线，那么秘密共享（Secret Sharing）就是另一条同样重要的技术路线。秘密共享的思想更简单直观：把一个秘密拆成多份，分别交给不同的参与方持有，任何单一参与方都无法恢复出完整的秘密。

Shamir 秘密共享：1979 年 Adi Shamir（RSA 中的 S）提出了一种基于多项式插值的秘密共享方案。它的核心思想是：给定一个 t-1 次多项式，你需要至少 t 个点才能唯一确定它。如果你想将秘密 s 分给 n 个人，要求其中任意 t 个人可以恢复秘密（少于 t 人无法恢复）：

// Shamir (t, n)-门限秘密共享
//
// 1. 选取一个大素数 p，秘密 s ∈ Zₚ
// 2. 随机构造一个 t-1 次多项式 f(x):
//    f(x) = s + a₁·x + a₂·x² + ... + aₜ₋₁·xᵗ⁻¹ (mod p)
//    其中 a₁, a₂, ..., aₜ₋₁ 是随机数
// 3. 为每个参与方 i 计算份额:
//    share_i = f(i) (mod p)
// 4. 恢复秘密时，任意 t 个份额通过拉格朗日插值:
//    s = Σ(share_i · Π(j≠i) (0-xⱼ)/(xᵢ-xⱼ)) (mod p)

加法秘密共享：一种更简单的共享方式。对于秘密 s，将其拆分为 s₁ + s₂ + ... + sₙ = s。每个参与方持有一个份额。所有份额相加即可恢复原值。加法共享特别适合与 MPC 协议配合——如果每个参与方持有自己数据的加法份额，大家可以在共享域上执行加法计算，而泄漏任何一方的原始数据。

共享方案 原理 恢复方式 典型应用
Shamir 秘密共享 多项式插值 Lagrange 插值（t 个份额） 密钥管理、门限签名
加法秘密共享 算术拆分 求和（全部份额） MPC 协议、安全聚合
Blakley 秘密共享 超平面交点 求解线性方程组 （较少使用）

21.3 SPDZ 协议

乱码电路适合两方计算，但扩展到多方时效率迅速下降。对于三方可或更多参与方的场景，SPDZ 协议成为了当前最重要的 MPC 方案之一。SPDZ 的名称来自其发明者名字的首字母（Damgård, Pastro, Smart, Zakarias）以及后续改进版（Overdrive 等）。

SPDZ 的核心思想是在线/离线分离——将协议分为两个阶段：

  • 离线阶段（Offline Phase）：预先生成大量公共的"乘法三元组"（Beaver Triples）。这个阶段不依赖具体的输入数据，可以在闲时完成。乘法三元组是一组随机值 (a, b, c)，满足 c = a · b。这些三元组被秘密共享给所有参与方。
  • 在线阶段（Online Phase）：利用离线阶段准备好的三元组，参与方只需轻量级的通信即可完成任意计算。加法只需本地计算，乘法则需要一轮通信来消耗一个三元组。

SPDZ 的一个关键特性是它提供了主动安全性（Active Security）——即使恶意参与方偏离协议行为（比如发送虚假数据），SPDZ 也能检测并阻止攻击。这是通过与消息认证码（MAC）结合实现的：每个共享值都附带一个认证标签，任何篡改都会被立即发现。

💡 Beaver Triple 的直觉：想象多位厨师合作烘焙一个蛋糕，但每个人都不能看到完整的配方。离线阶段，他们提前准备好了一些"预混合料包"（三元组）。在线阶段，他们只需要用这些料包来"组合"各自手中的秘密配料。每个料包只用一次，用完即弃——就像一次性的砧板垫。

21.4 隐私集合交集（PSI）

安全多方计算的一个特例问题是：两家公司各自有一个客户名单，他们想知道两家公司共有的客户（即交集），但都不想向对方透露自己的全部客户名单。这就是隐私集合交集（Private Set Intersection, PSI）。

PSI 有多种实现方式，其中一种效率较高的是基于不经意传输扩展（OT Extension）的 PSI 方案。简单的思路如下：

假设 Alice 有集合 A，Bob 有集合 B。Alice 把她的每个元素通过哈希映射到一组随机数，然后用这些随机数构造一个布隆过滤器（Bloom Filter）。Bob 对他的每个元素进行同样的哈希操作，然后通过不经意传输查询 Alice 的布隆过滤器，判断自己的元素是否在交集中。最终只有 Bob 能得到交集结果——如果他想要 Alice 也得到结果，就需要增加额外的轮次。

21.5 安全多方计算的应用

MPC 已经从学术论文走向了实际生产环境。以下是一些引人注目的案例：

  • 丹麦甜菜拍卖（2013）：这是 MPC 史上最著名的真实世界应用之一。丹麦的甜菜农需要向糖厂出售甜菜，但传统的拍卖方式存在价格操纵的问题。研究人员部署了一个基于 MPC 的拍卖系统，让 1000 多名农民各自提交加密的报价，系统在不泄露任何单个报价的情况下自动计算出最优价格和分配方案。这个系统运行了多年，证明了 MPC 在大规模真实场景中的可行性。
  • 联合数据分析：多家医院可以联合训练疾病预测模型，而每家医院的病患数据始终留在本地。通过 MPC 协议，各家医院共享加密的梯度更新，协同优化模型，但不泄露任何患者的具体病例数据。
  • 密钥管理：高安全性系统的私钥可以拆分为多份，分别存储在多个管理员处。只有当足够多的人一起参与时，才能恢复并使用密钥——这防止了单点故障和内部人员作恶。
  • 基因组隐私：两个研究机构可以联合计算基因序列的相似度，或一起做关联分析（GWAS），而不暴露各自的基因数据库。这在生物医学领域有着巨大的应用前景。
⚠️ 踩坑故事：MPC 系统实现中的"把戏"
在某次知名的隐私计算竞赛中，有两支 MPC 团队给出的"最佳"成绩后来被发现在安全性上存在严重缺陷。问题出在哪里？一个团队使用了"固定随机数种子"来加速不经意传输——这实际上完全破坏了安全性，因为攻击者可以重现所有随机数。另一个团队在实现秘密共享时，不小心将某个参与方的份额以明文形式写入了日志文件。

这两个故事说明了一个重要的原则：算法安全 ≠ 实现安全。MPC 的工程实现比普通加密协议更加微妙。一个不恰当的性能优化（比如重用三元组、固定随机数）就可能彻底摧毁安全性。即使协议在理论上被证明是安全的，错误的实现方式能让所有理论保证化为乌有。
📊 MPC vs 同态加密：如果我们将第 20 章的同态加密与本章的 MPC 做一个对比：
  • HE：数据拥有者加密数据后发给计算方，计算方独立完成所有运算。适合"一方持有数据、一方提供算力"的场景，通信开销极低但计算开销很高。
  • MPC：多方持有各自的秘密，通过多轮通信协作完成计算。通信是瓶颈，但计算通常比 HE 快得多。
  • 混合方案：实际应用中，将 HE 和 MPC 结合使用往往效果最好。例如，使用 HE 来优化 MPC 中不经意传输的效率，或者用 MPC 来弥补 HE 中无法处理比较运算的短板。

安全多方计算从一个学术趣味问题——姚氏百万富翁问题——发展成了如今价值数十亿美元的隐私计算产业的核心技术。从 1000 人参与的丹麦甜菜拍卖，到跨国医疗机构的联合数据分析，MPC 证明了隐私不一定是效率的对立面。然而，我们也要认识到，MPC 的通信开销在参与方数量增多时快速增长，协议的正确实现也需要深厚的密码学功底。下一章，我们将从"联合计算"过渡到"数据发布"场景，来看一种完全不同但同样强大的隐私保护技术——差分隐私。
