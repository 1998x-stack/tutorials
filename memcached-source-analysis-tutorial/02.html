第2章 环境搭建与源码获取

大家好，我是你们的讲师。上一章我们聊了为什么要读 Memcached 源码——它的代码量适中、设计思想经典、工程实践丰富。但 "纸上得来终觉浅"——要想真正理解一个系统，光看别人写的分析文章是不够的，你需要在自己的机器上把代码跑起来、打断点、看变量、甚至改几行代码试效果。这一章我们就来做这件事：从零搭建 Memcached 源码阅读环境。

我见过太多人卡在环境搭建这一步——不是 libevent 版本不对，就是 autotools 工具链缺失，折腾半天连 make 都没跑过就放弃了。这一章我会把每一步拆细，包括常见坑的解法，让你一次把事情做对。好，我们直接开始。

获取源码：克隆 1.6.41

Memcached 的源码托管在 GitHub 上，直接 clone 最新稳定版。本课程所有分析基于 1.6.41 版本——这是 2025 年发布的版本，包含了 extstore、proxy 等新特性，同时又足够稳定。

git clone https://github.com/memcached/memcached.git
cd memcached
git checkout 1.6.41

如果你克隆时看到的是 master 分支，不要慌——master 是开发分支，可能包含尚未发布的实验性代码。我们用 git checkout 1.6.41 切到稳定标签。建议你也锁在这个版本，方便跟我后面的章节一一对应。

版本选择建议：1.6.41 是目前最稳定的长期支持版本。如果你在生产环境使用，建议选择 1.6.x 系列的最新小版本。不要用 1.5.x——那个版本缺少 extstore 和 proxy 等重要的新特性。

编译安装：autotools 构建流程

Memcached 使用经典的 autotools 构建系统。如果你来自 Go 或 Rust 生态，可能会觉得这套工具链有点"复古"——但 autotools 在 C 项目中仍然是事实标准，尤其是需要跨平台兼容的项目。

安装依赖

在 Ubuntu/Debian 系统上，首先安装编译工具链和 Memcached 的运行时依赖：

sudo apt update
sudo apt install build-essential autoconf automake libtool pkg-config
sudo apt install libevent-dev libssl-dev

这里的两个关键依赖是：

  • libevent (2.1+)：Memcached 的网络事件驱动引擎。Memcached 依赖 libevent 来管理 socket 事件、定时器等。libevent 1.x 和 2.x 的 API 有较大差异，Memcached 1.6 要求 libevent >= 2.1。
  • libssl-dev：用于 SASL 认证支持，不是必需但建议安装，否则编译时某些特性会被禁用。

autoreconf 与 configure

克隆下来的源码中，configure 脚本可能没有预生成，需要先运行 autoreconf：

autoreconf -fi
./configure --prefix=/usr/local/memcached

autoreconf -fi 的作用是调用 autoconf、automake、libtool 等工具链重新生成 configure 脚本和 Makefile.in 模板。其中 -f（force）表示强制重新生成所有文件，-i（install）表示自动复制缺失的辅助文件（如 config.sub、install-sh）。如果你看到 "possibly undefined macro" 错误，多半是缺少某个 autotools 工具或 pkg-config 没装。

踩坑实录：我第一次编译 Memcached 时，运行 autoreconf -fi 后报错 "libtool: command not found"。问题出在 ubuntu 上 libtool 和 libtool-bin 是两个不同的包，apt install 时我只装了前者而忘了装后者。解决方案很简单：sudo apt install libtool-bin。

make 与 install

make -j$(nproc)
sudo make install

编译过程大约需要 1-2 分钟。如果一切顺利，你会看到一个干净的编译输出，没有 warning。Memcached 的代码质量很高，编译时几乎不会产生警告——这也是其代码风格严谨的一个佐证。

调试环境配置

仅仅能跑起来是不够的——阅读源码最重要的是能设断点、看变量、单步跟踪。我推荐两种调试方式，你可以根据自己的喜好选择。

方案一：GDB + CGDB（推荐终端党）

GDB 是 C 代码事实上的调试标准。CGDB 是 GDB 的一个 TUI 前端，在终端中同时显示源码和调试命令，比纯 GDB 体验好很多。

sudo apt install gdb cgdb

# 编译时保留调试信息（默认 Makefile 已经带 -g）
./configure --prefix=/usr/local/memcached CFLAGS="-g -O0"

make -j$(nproc)
sudo make install

# 启动 CGDB
cgdb /usr/local/memcached/bin/memcached

注意这里 CFLAGS="-g -O0"：-g 生成调试符号，-O0 关闭优化。关闭优化很重要，否则 GCC 会内联函数、重排指令，导致你单步跟踪时看到"跳跃"的执行顺序。调试用 -O0，线上用 -O2，不要搞混。

在 CGDB 中按 F7 进入源码模式，然后就可以：

  • b main —— 在 main 函数设断点
  • r —— 运行程序
  • n —— 下一行（step over）
  • s —— 进入函数（step into）
  • p settings.maxbytes —— 打印变量值

方案二：VS Code / CLion（IDE 党）

如果你习惯图形化界面，VS Code 搭配 C/C++ 扩展也可以很好地调试 C 项目。打开 Memcached 源码目录后，配置 .vscode/launch.json：

{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Debug memcached",
            "type": "cppdbg",
            "request": "launch",
            "program": "${workspaceFolder}/memcached",
            "args": ["-m", "64"],
            "stopAtEntry": true,
            "cwd": "${workspaceFolder}",
            "environment": [],
            "externalConsole": false,
            "MIMode": "gdb"
        }
    ]
}

CLion 用户更简单——直接用"Open"打开项目根目录，CLion 会自动识别 CMake 或 Makefile 项目。如果是纯 Makefile 项目，CLion 会提示你配置编译设置，选择"使用 Makefile 编译"即可。

跑通测试套件

一个能让你安心修改代码的保障是：测试全过。Memcached 的测试套件位于 t/ 目录下，是基于 Perl 的测试框架：

make test

这个命令会启动所有测试用例，包括功能测试、协议测试、压力测试等。全部跑完大约需要 5-10 分钟，取决于你的机器性能。如果所有测试通过，你会看到类似这样的输出：

All tests successful.
Files=NN, Tests=MM, X wallclock secs
Result: PASS

如果某些测试失败，可以先检查是否有权限问题、端口冲突或 libevent 版本不匹配。最常见的失败原因是你的机器上已经运行了一个 memcached 实例占用了默认端口（11211）：

踩坑实录：make test 第一次跑时我遇到了一个诡异的失败——某个二进制协议测试总是超时。查了半天发现是我在 docker 里跑的，容器的 net.core.somaxconn 设置太低导致连接队列满。解决方案：sysctl -w net.core.somaxconn=1024。

编译选项详解

除了默认编译，Memcached 还提供了几个重要的 configure 选项，用于启用特定功能：

编译选项 说明 依赖 建议
--enable-seccomp 启用 seccomp BPF 沙箱，限制系统调用 Linux 内核 3.5+ 生产环境强烈建议开启
--enable-extstore 启用外部存储（SSD/NVMe）支持 无额外依赖 分析 extstore 模块时需要
--enable-proxy 启用内置 Proxy（Lua 路由） liblua 5.3+ 分析 proxy 模块时需要
--enable-sasl 启用 SASL 认证 libsasl2-dev 需要认证时开启
--enable-tls 启用 TLS/SSL 加密传输 libssl-dev 安全敏感场景
--enable-dtrace 启用 DTrace 动态跟踪 DTrace 工具 性能分析用途

对于本课程的阅读目的，建议你这样编译：

./configure --enable-extstore --enable-proxy --enable-seccomp CFLAGS="-g -O0"
make -j$(nproc)

这样所有模块的符号都可用，我们可以随时切到任何模块打点分析。

验证安装

编译安装完成后，验证一下 Memcached 是否能正常启动：

memcached -m 64 -p 11211 -d
echo "stats" | nc localhost 11211

如果看到一堆 stats 输出（包括 version、uptime、curr_items 等字段），恭喜你——环境搭建成功。

动手练习：尝试用 memcached -h 查看所有命令行参数。重点关注 -m（最大内存）、-p（端口号）、-t（工作线程数）、-c（最大连接数）、-v（详细日志输出）。下一章我们会深入分析这些参数在源码中的处理逻辑。

环境搭建是一个看似琐碎但至关重要的环节。我建议你花半天时间把这一章的步骤完整走一遍，确保你能在自己的机器上完成编译、调试、跑测试的全流程。有了这个基础，后续每一章的分析你都能亲手复现——这才是阅读源码的正确姿势。
