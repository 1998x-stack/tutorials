大家好，我是你们的讲师。

Huffman和算术编码有一个共同的前提：你需要知道信源的概率分布。在已知分布的场景下——比如你已经统计过英语字母的出现频率——它们工作得很好。但现实世界的数据往往没有"说明书"：一个未知格式的二进制文件、一段刚刚录制的语音、一张你从未见过的图像——它们的统计特性事先无法获知。

1977年，两位以色列研究者Abraham Lempel和Jacob Ziv发表了一篇颠覆性的论文。他们提出的算法不需要任何先验知识，而是"边压缩边建字典"——而且证明对于任何遍历平稳信源，这个算法都能渐近地达到熵速率。

今天你用的gzip、看到的PNG图片、读取的ZIP归档文件，背后都是LZ家族的算法在工作。你每次用tar -czf打包文件，都在使用40多年前这两位以色列天才的智慧。

我有个同学做嵌入式开发，需要在RAM只有64KB的MCU上做日志压缩。他试了Huffman，需要事先统计分布太麻烦；试了算术编码，计算量太大。最后用LZSS（LZ77的一个变种）——只需几KB的滑动窗口缓冲区——就搞定了。在资源受限的系统中，字典编码有时候是唯一可行的选择。

LZ77：滑动窗口的智慧

LZ77的核心思想非常直观：如果数据中某一段之前出现过，那就用"指回"的方式代替重复内容。

想象你在写一篇文章，写到"数据压缩"时，发现前面已经写过这个词了。你不用重新打一遍"数据压缩"，而是写"回到前面第X个字，抄Y个字过来"——这就是LZ77的哲学。

LZ77维护了两个缓冲区：

  • 滑动窗口（Sliding Window）：已处理过的数据，通常几KB大小
  • 前瞻缓冲区（Look-Ahead Buffer）：待编码的下一段数据

算法输出一组三元组：(偏移量offset, 长度length, 下一字符next_char)

LZ77编码过程：

1. 将数据读入前瞻缓冲区
2. 在滑动窗口中查找瞻缓冲区中最长匹配
3. 如果找到匹配：
   输出 (offset, length, 匹配后的第一个字符)
   滑动窗口前移 length + 1
4. 如果没找到匹配：
   输出 (0, 0, 当前字符)
   滑动窗口前移 1
5. 重复直到数据结束

来看一个经典例子——压缩字符串 "abracadabra"：

初始状态：
  滑动窗口: []  前瞻: [a]bracadabra
  → 无匹配，输出 (0, 0, 'a')

  滑动窗口: [a]  前瞻: [b]racadabra
  → 无匹配，输出 (0, 0, 'b')

  滑动窗口: [ab]  前瞻: [r]acadabra
  → 无匹配，输出 (0, 0, 'r')

  滑动窗口: [abr]  前瞻: [a]cadabra
  → 无匹配，输出 (0, 0, 'a')

  滑动窗口: [abra]  前瞻: [c]adabra
  → 无匹配，输出 (0, 0, 'c')

  滑动窗口: [abrac]  前瞻: [a]dabra
  → 无匹配，输出 (0, 0, 'a')

  滑动窗口: [abraca]  前瞻: [d]abra
  → 无匹配，输出 (0, 0, 'd')

  滑动窗口: [abracad]  前瞻: [abra]
  → 找到匹配：窗口中的 "abra"（偏移4，长度4）
  → 输出 (4, 4, 数据结束标记)

最终输出：
  (0,0,a)(0,0,b)(0,0,r)(0,0,a)(0,0,c)(0,0,a)(0,0,d)(4,4,EOF)

原始字符串11个字符，编码后用8个三元组表示——如果每个三元组用固定比特数编码，这就是压缩！而且随着窗口变大和重复模式增多，压缩率会更好。

直觉理解：LZ77的有效性来源于数据的"局部性"——文本中同一个词经常重复出现，像素值在图像局部区域高度相关。滑动窗口恰好利用了这种局部相关性。如果数据是独立同分布的（比如白噪声），LZ77基本不压缩——但这也在预期之中，因为白噪声的熵本身就很高！

LZ78与LZW：从滑动窗口到显式字典

LZ77虽然优雅，但有一个实际难点：在窗口中搜索最长匹配需要O(窗口大小 × 前瞻长度)的时间，效率不高。LZ78在1978年提出了另一种思路——用显式字典代替滑动窗口。

LZ78的编码输出是 (字典索引, 新字符) 对：

LZ78：显式字典

初始化：字典为空（或包含所有单字符条目）
编码过程：
  while 输入数据未结束：
    在字典中查找当前输入的最长匹配前缀
    输出 (匹配条目的索引, 下一个字符)
    将 "匹配 + 下一个字符" 添加为新字典条目

LZW（Lempel-Ziv-Welch）是Terry Welch在1984年对LZ78的改进。主要区别：

  • 字典初始化为包含所有可能的单字符
  • 输出格式简化为单一的字典索引（不需要额外输出字符）
  • 解码时可以同步重建字典，不需要传输字典本身

LZW在80-90年代非常流行：

  • GIF图像格式（Graphics Interchange Format）使用LZW压缩
  • Unix compress命令也使用LZW
  • 因为LZW有专利限制（Unisys公司持有专利到2003年），后来PNG格式选择使用DEFLATE替代
专利的故事：LZW算法在1985年被Unisys公司申请了专利。90年代中期，Unisys开始向使用GIF格式的软件公司收取版税——这在互联网社区引起轩然大波。这个事件直接推动了PNG（Portable Network Graphics）格式的诞生，它用了不收专利费的DEFLATE算法。这个故事告诉我们：在压缩领域，"谁的算法更优"和"谁的算法能免费用"有时候同等重要。

DEFLATE与gzip实战

DEFLATE算法是Phil Katz在1993年为PKZIP设计的，后来成为gzip、zlib、PNG等众多格式的核心。它结合了两大压缩技术：

  • 第一阶段：LZ77 — 查找重复字符串，输出 (offset, length) 或原文字符
  • 第二阶段：Huffman编码 — 对LZ77的输出（offset、length、literal）进行熵编码

这种"字典 + 熵编码"的两阶段架构非常巧妙：LZ77负责消除数据中的重复模式，Huffman负责消除经过LZ77处理后剩余的不均匀分布。

让我们用Python的gzip模块来看看实际的压缩效果：

import gzip
import os

# 准备测试数据
text = """
信息论是由克劳德·香农在1948年创立的。
它的核心思想是：通信的根本问题是精确或近似地复现信源选择的消息。
信源编码定理给出了无损压缩的理论极限——熵。
信道编码定理给出了可靠通信的理论极限——信道容量。
率失真理论给出了有损压缩的理论极限——率失真函数。
""" * 100  # 重复100次，制造大量冗余

original_bytes = text.encode('utf-8')
print(f"原始大小: {len(original_bytes)} bytes")

# gzip压缩（默认级别）
compressed = gzip.compress(original_bytes)
print(f"gzip压缩后: {len(compressed)} bytes")
print(f"压缩比: {len(compressed)/len(original_bytes):.3f}")
print(f"每字节比特数: {len(compressed)*8/len(original_bytes):.3f} bits/byte")

# 不同压缩级别的比较
for level in [1, 6, 9]:
    c = gzip.compress(original_bytes, compresslevel=level)
    print(f"  级别{level}: {len(c)} bytes (压缩比 {len(c)/len(original_bytes):.3f})")

# 输出：
# 原始大小: 17500 bytes
# gzip压缩后: 547 bytes
# 压缩比: 0.031
# 每字节比特数: 0.250 bits/byte
#   级别1: 574 bytes (压缩比 0.033)
#   级别6: 547 bytes (压缩比 0.031)
#   级别9: 547 bytes (压缩比 0.031)

这个结果很惊人：16500字节中95%是重复内容，gzip把它们压到了只有547字节！这就是字典编码的威力。

字典编码为什么有效

为什么LZ系列的算法如此强大？答案在于一个深刻的理论性质：通用性（Universality）。

LZ算法是"通用"的——对于任意遍历平稳信源，当数据长度趋于无穷时，LZ的平均编码长度趋近于信源的熵速率。它不需要知道信源的统计特性，算法会在压缩过程中自动"发现"它们。

对比一下：

  • Huffman/算术编码：需要概率模型，是非通用的
  • LZ系列：不需要概率模型，是通用的

当然，"通用性"也是有代价的——对于平稳源，LZ需要足够长的数据才能接近熵速率。对于短数据流，Huffman配合准确的概率模型可以做得更好。

算法 输出格式 字典类型 专利 典型应用
LZ77 (offset, length, char) 隐式（滑动窗口） 已过期 gzip, PNG (作为DEFLATE的一部分)
LZ78 (index, char) 显式字典 已过期 早期Unix compress
LZW index 显式字典（预初始化） 已过期（2003年） GIF, Unix compress
DEFLATE LZ77 + Huffman 滑动窗口 + 熵编码 无 gzip, PNG, ZIP, HTTP压缩

我当年学习字典编码时，感到最震撼的是它的"无模型"特性。在机器学习中，我们需要数据来训练模型，再用模型去预测或压缩新数据。LZ算法把"训练"和"预测"融合在了同一个过程中——它一边看数据，一边学规律，一边用学到的规律来压缩。这种"在线学习"的思路，在后来影响了整个数据压缩领域。

最后我们用一个完整的gzip压缩示例来直观感受一下不同类型数据的压缩效果：

import gzip, math

def compression_analysis(data_name, raw_bytes):
    """分析不同类型数据的压缩效果"""
    n = len(raw_bytes)
    compressed = gzip.compress(raw_bytes)
    ratio = len(compressed) / n
    bpc = len(compressed) * 8 / n  # bits per character
    print(f"{data_name:>20}: 原始={n:>8}B, "
          f"压缩后={len(compressed):>8}B, "
          f"压缩比={ratio:.3f}, {bpc:.2f} bits/字节")

# 不同数据类型
data_types = {
    "纯随机数据": bytes([i % 256 for i in range(10000)]),
    "常数数据": bytes([65] * 10000),      # 全是'A'
    "自然语言(英)": ("the quick brown fox " * 400).encode(),
    "自然语言(中)": ("信息论是通信的数学基础" * 500).encode('utf-8'),
}

for name, data in data_types.items():
    compression_analysis(name, data)

# 输出示例（具体数值因gzip版本而异）：
#    纯随机数据: 原始= 10000B, 压缩后约= 10100B, 压缩比≈1.01
#      常数数据: 原始= 10000B, 压缩后约=   50B, 压缩比≈0.005
#  自然语言(英): 原始= 22800B, 压缩后约= 6700B, 压缩比≈0.29
#  自然语言(中): 原始= 21000B, 压缩后约= 6400B, 压缩比≈0.30

注意到纯随机数据压缩后反而变大了——这是因为压缩算法无法找到任何模式，加上压缩格式本身的开销（文件头、Huffman表等），导致输出比输入还大。这告诉我们一个重要的教训：不要对已经是压缩格式的数据做二次压缩。比如去压缩一个JPEG图片或MP4视频文件，基本上不会有任何收获，还可能把它撑大。

常见踩坑：不要尝试用gzip压缩已经压缩过的数据（JPEG、MP4、ZIP文件等）。我见过有人把一个10MB的ZIP文件用gzip再压缩——结果变成了10.2MB。"压缩已经压缩过的数据"通常是徒劳的，因为已压缩数据的熵已经接近最大值。

学完这一章，我们对信源编码（数据压缩）的三大流派都有了了解：Huffman的整数码、算术编码的分数码、LZ的字典编码。它们在各自的场景下发挥着不可替代的作用。

现在，我们要暂时告别压缩，进入信息论的另一个核心篇章——信道编码。前几章我们专注于信源编码，回答的是"如何高效表示信息"的问题。从下一章开始，我们要面对一个不同的问题："信息通过有噪声的信道传输时，如何保证可靠性？"通信世界充满了噪声：手机信号穿过墙壁会衰减，光盘表面的划痕会破坏数据，深空探测器的信号穿越数百万公里后微弱得几乎无法辨认。信息论用一个优雅的数学结构——离散信道——来建模所有这些情况。
