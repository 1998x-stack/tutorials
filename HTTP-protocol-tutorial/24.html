第24章：Web服务器配置

大家好。之前我们学的HTTP协议都是站在客户端视角。但从今天开始，我们要走到服务器这一侧，看看HTTP请求到达服务器后发生了什么。本章以Nginx为例，因为它是目前最流行的Web服务器兼反向代理，市场占有率超过30%。

24.1 Nginx基础配置

Nginx的配置文件通常位于 /etc/nginx/nginx.conf，核心概念就是四个字：指令和上下文。上下文是指令的作用域，主要有：http、server、location、upstream。

一个最小的Nginx配置长这样：

http {
    server {
        listen 80;
        server_name example.com;

        root /var/www/example;
        index index.html;

        access_log /var/log/nginx/example_access.log;
        error_log /var/log/nginx/example_error.log warn;
    }
}

解释几个核心指令：

  • server — 定义一个虚拟主机块，里面所有指令都会影响该虚拟主机下的请求处理。
  • listen — 监听的IP地址和端口。可以写成 listen 80（所有IP的80端口）或 listen 192.168.1.1:80（指定IP）。
  • root — 文档根目录。当请求 /images/logo.png 时，Nginx会返回 /var/www/example/images/logo.png。
  • location — URL路径匹配规则，后面会细说。
  • access_log / error_log — 日志文件路径和日志级别。

location指令是Nginx最灵活也最容易出错的部分。它支持多种匹配方式：

匹配方式示例说明
精确匹配location = /完全等于指定路径
前缀匹配location /images/以指定前缀开头
正则匹配（区分大小写）location ~ \.php$匹配以.php结尾的URL
正则匹配（忽略大小写）location ~* \.(jpg|png)$匹配图片文件
优先前缀匹配location ^~ /static/匹配后不再检查正则

匹配顺序是：先精确匹配（=），再优先前缀匹配（^~），再正则匹配（~/~*），最后普通前缀匹配。很多人在这里翻车，因为正则会覆盖前缀匹配。下面这个坑我栽过不止一次：

踩坑故事：有次我写了一个 location /api/ 来处理后端API代理，又写了一个 location ~ \.php$ 来处理PHP请求。结果发现 /api/user.php 的请求被PHP location处理了，根本没有经过API代理。原因是正则匹配的优先级高于普通前缀匹配。解决方案是用 location ^~ /api/，告诉Nginx即使有正则匹配，这个前缀也优先。那次翻车让我花了一下午在线上排查"为什么API返回乱码"。结论：location匹配优先级一定要记牢。

24.2 虚拟主机

虚拟主机（Virtual Host）就是用一台服务器托管多个网站。Nginx通过server块来实现虚拟主机，有两种方式：

基于名称（Name-based）——根据请求的 Host 头部区分。一台服务器一个IP，可以跑上百个网站：

server {
    listen 80;
    server_name blog.example.com;
    root /var/www/blog;
}

server {
    listen 80;
    server_name shop.example.com;
    root /var/www/shop;
}

基于IP（IP-based）——不同网站绑定不同IP地址，现在用得比较少了：

server {
    listen 203.0.113.1:80;
    root /var/www/site1;
}
server {
    listen 203.0.113.2:80;
    root /var/www/site2;
}

server_name 匹配规则——Nginx按以下顺序匹配：精确名称 → 最长的通配符前缀（*.example.com）→ 最长的通配符后缀（www.example.*）→ 正则匹配（~^www\d+\.example\.com$）。如果都不匹配，则使用 default_server：

server {
    listen 80 default_server;
    return 444;  # 直接关闭连接
}

良好实践：强烈建议配置一个 default_server 来拒绝未匹配的请求。我曾经见过一个没配 default_server 的Nginx，被人用IP直接访问，暴露了内部管理后台。配置 default_server 返回 444 或 403 是最基本的防护。

24.3 HTTPS配置

HTTPS配置的核心是证书管理和安全参数。一个完整的HTTPS server块：

server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate     /etc/ssl/certs/example.com.pem;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    # 现代安全配置
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers on;

    # 会话缓存加速
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1h;
}

HTTP 自动跳转 HTTPS——这是标配：

server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}

几点配置建议：

  • 只启用 TLSv1.2 和 TLSv1.3，禁用 SSLv3 和 TLSv1.0/1.1，这些老协议有已知漏洞（如POODLE、BEAST）。
  • 密码套件优先选择 AEAD 算法（GCM 或 ChaCha20），禁用 RC4、3DES 等不安全算法。
  • ssl_session_cache 可以显著减少TLS握手的开销——客户端复用会话时不需要完整的握手。

检查工具：配置完HTTPS后，用 SSL Labs 测试你的服务器评分。目标是拿到 A+ 评分。如果某个密码套件导致评分下降，可以参考 Mozilla SSL Configuration Generator 推荐的安全配置。

24.4 静态文件服务优化

Nginx处理静态文件非常高效，但默认配置还有优化空间。下面是几个关键优化项：

location /static/ {
    root /var/www/example;

    # 零拷贝文件传输，减少用户态/内核态切换
    sendfile on;

    # 在发送文件头时尽量攒包，减少小包数量
    tcp_nopush on;

    # 启用gzip压缩
    gzip_static  on;
    gzip_types   text/css application/javascript image/svg+xml;

    # 强制客户端缓存，减少重复请求
    expires 30d;
    add_header Cache-Control "public, immutable";

    # 防目录遍历
    autoindex off;
}

各指令的作用：

  • sendfile — 启用零拷贝，数据直接从磁盘到网卡，不经过用户态内存。对静态文件服务器来说这是最重要的性能优化。
  • tcp_nopush — 与 sendfile 配合使用，将多个数据包合并发送，减少网络小包数量。
  • gzip_static — 如果预先生成了 .gz 文件，Nginx直接发送压缩包而不实时压缩。减少了CPU开销。
  • expires — 设置 Cache-Control: max-age 头部，让浏览器强缓存资源。30天是静态资源的一个合理值。

注意：sendfile 在某些文件系统（如某些网络文件系统）上可能有问题，会在高并发下导致连接挂起。如果发现请求偶尔卡死、日志里没有错误，试试关掉 sendfile 看是否复现。我就在一个NFS挂载的静态资源服务器上踩过这个坑。

24.5 日志分析

日志是排查问题的重要依据。Nginx默认的日志格式包含的信息有限，建议自定义日志格式：

http {
    log_format main '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    '$request_time $upstream_response_time';

    # 还可以加一个带更多信息的格式
    log_format detail '$remote_addr - [$time_local] '
                      '$host "$request" $status $body_bytes_sent '
                      '"$http_referer" "$http_user_agent" '
                      '$request_time $upstream_addr $upstream_status '
                      '$upstream_response_time $connection $connection_requests';

    access_log /var/log/nginx/access.log main;
}

关键变量说明：

  • $request_time — 请求处理总时间（秒），包含接收请求和发送响应的时间。
  • $upstream_response_time — 后端（upstream）处理请求的时间。
  • $upstream_addr / $upstream_status — 上游服务器的地址和状态码。

GoAccess——这是一款轻量级的日志分析工具，可以直接在终端里生成实时的访问统计：

goaccess /var/log/nginx/access.log --log-format=COMBINED

它会展示：访问量排名、404/500错误分布、访问高峰时段、客户端IP排名、请求方法分布等。用这个工具可以快速定位问题：

  • 慢请求：按 $request_time 排序，找出处理时间最长的请求。
  • 404错误频发：可能是有前端请求了错误的资源路径。
  • 5xx错误激增：后端服务可能出了问题。
  • 异常流量：某个IP突然狂刷请求，可能是爬虫或攻击。

我曾经用 GoAccess 发现了一个奇怪的现象：每天凌晨3点服务器负载飙升。查日志发现一个爬虫在这个时段疯狂抓取。后来加了个限流配置（limit_req_zone）就解决了。所以说，日志分析不仅仅是事后复盘——主动监控能帮你发现很多意料之外的问题。
