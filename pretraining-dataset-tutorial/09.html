找对语言，筛掉垃圾：清洗流水线的第一关

大家好，我是你们的讲师。上一章我们把 WARC 文件里的 HTML 解析成了原始文本——但这只是万里长征第一步。打开一个刚解析完的文本文件夹，你会发现里面什么都有：英文、中文、阿拉伯文、乱码、只有标点符号的页面、全是 URL 列表的页面……最夸张的一次，我看到一个"文章"居然是一个 YouTube 视频的 2,000 条自动翻译评论的拼接体——中文夹杂着泰语，中间还混了一段 Python traceback。

如果把这堆东西直接喂给模型，后果不堪设想。这就是为什么在数据流水线的第二个阶段，我们需要做两件事：语言识别（确定每篇文档的语言并筛选目标语言）和初步清洗（用一系列启发式规则快速剔除明显的低质量文本）。

业界经过多年实践，已经沉淀出一套比较成熟的"过滤管道"——从 fastText 语言分类器到 Gopher/C4/FineWeb 的启发式过滤器。这一章我们来逐一拆解。

9.1 fastText 语言分类器

在预训练数据处理中，fastText 是使用最广泛的语言识别工具。它由 Facebook AI Research 开发，核心优势在于：

  • 覆盖 176 种语言的预训练分类模型
  • 基于字符 n-gram，对拼写错误和形态变化有很好的鲁棒性
  • 推理速度极快——在 CPU 上每秒可处理数千条短文本
  • 对中文、日文等 CJK 语言的分辨率相当高，很少发生中日韩语之间的误判

使用 fastText 做语言检测非常简单：

import fasttext

# 加载预训练语言分类模型（约 900MB）
# 可以从 https://fasttext.cc/docs/en/language-identification.html 下载
model = fasttext.load_model('lid.176.bin')

def detect_language(text, threshold=0.5):
    """返回 (语言代码, 置信度)"""
    labels, scores = model.predict(text.replace('\n', ' '), k=1)
    lang = labels[0].replace('__label__', '')
    score = scores[0]
    return (lang, score)

# 测试
print(detect_language("Hello, how are you?"))      # ('en', 0.98)
print(detect_language("今天天气真不错"))            # ('zh', 0.99)
print(detect_language("こんにちは"))                # ('ja', 0.97)

# 实际过滤逻辑：只保留中文文档
text = "这里是某篇文章的正文..."
lang, score = detect_language(text)
if lang == 'zh' and score > 0.7:
    print("保留：置信度高的中文文档")
else:
    print("丢弃：非中文或置信度不足")

在实际流水线中，一个常见的配置是：设置最小置信度阈值为 0.5（对于长文档）或 0.7（对于短文档），置信度低于阈值的文档直接丢弃。为什么不用降低阈值来多保留一些数据？因为低置信度通常意味着文档是混合语言、乱码或内容太短——这些对预训练都是有害的。

性能优化：fastText 模型文件约 900MB，在分布式处理中每个 Worker 加载一份会消耗大量内存。优化方法是：使用 fasttext.FastText._load_model_bytes() 预先将模型加载到共享内存中，或者使用内存映射（mmap）方式加载。在 Spark/Datatrove 中，通常每个 Executor 只加载一次模型，在所有 Task 之间共享。

9.2 CLD3：Google 的轻量级替代方案

如果你对推理速度有极致要求，或者不能接受 900MB 的模型体积，Google 的 CLD3（Compact Language Detector v3） 是一个优秀的选择。CLD3 基于字符 n-gram 的神经网络，模型大小仅约 2MB，支持 107 种语言，推理速度是 fastText 的 3-5 倍。

import cld3

def detect_language_cld3(text):
    """使用 CLD3 检测语言"""
    result = cld3.get_language(text)
    return (result.language, result.probability)

# CLD3 对短文本的检测效果通常优于 fastText
print(detect_language_cld3("Hello world"))           # ('en', 0.99)
print(detect_language_cld3("深度学习是机器学习的一个分支"))  # ('zh', 0.98)

CLD3 的缺点是对高度混合语言的文本和有大量专有名词的文档（如包含大量英文技术术语的中文文章）精度稍低。实际做法是：在需要高吞吐的场景下用 CLD3 做一轮粗筛，对边界案例再交给 fastText 做二次确认。

对比维度 fastText CLD3
模型大小 ~900 MB ~2 MB
支持语言数 176 107
长文本精度 优秀 良好
短文本精度 良好 优秀
推理速度 快 极快（3-5x fastText）
混合语言文本 较好 一般

9.3 Gopher 质量过滤器（DeepMind）

语言识别之后，我们需要用一套启发式规则来快速剔除那些"一看就不是好数据"的文档。DeepMind 在训练 Gopher（2022）时定义了一套经典的质量过滤规则，后来被几乎所有后续项目沿用和改进。

Gopher 过滤器的主要规则包括：

  • 最小/最大词数：丢弃词数少于 50 或多于 100,000 的文档。太短的文档信息量不足，太长的通常是拼接产物。
  • 重复段落比例：计算文档中最长公共子串的比例。如果任何一行出现了 30 次以上，或文档的重复行占比超过 30%，丢弃。
  • 字母字符占比：文档中字母字符（对于英文）或 Unicode 字符（对于通用）占比应超过 70%。这能有效过滤掉乱码页面和纯表格数据。
  • 符号占比：文档中非字母、非数字、非空格的字符占比不应超过 30%。过滤掉大量特殊符号的页面（如表情符号列表、ASCII art）。
  • 停用词覆盖率：文档必须包含至少一定比例的停用词（常见词如 "the"、"a"、"is"）。这能有效筛掉关键词堆砌的垃圾页面。
import re
from collections import Counter

def gopher_quality_filter(text: str) -> bool:
    """实现 Gopher 的启发式过滤规则"""
    words = text.split()
    num_words = len(words)

    # 规则 1: 词数范围
    if num_words < 50 or num_words > 100000:
        return False

    # 规则 2: 重复行检测
    lines = text.split('\n')
    line_counts = Counter(lines)
    repeated_ratio = sum(c for l, c in line_counts.items() if c > 1) / len(lines)
    if repeated_ratio > 0.3:
        return False

    # 规则 3: 字母字符占比
    alpha_chars = sum(1 for c in text if c.isalpha())
    total_chars = len(text)
    if alpha_chars / total_chars < 0.7:
        return False

    return True

# 测试
good_text = "这是一个正常的文档，包含足够的信息量。"
bad_text = "!!@@##$$%%^^&&**(())__++==  " * 100
print(gopher_quality_filter(good_text))  # True
print(gopher_quality_filter(bad_text))   # False

9.4 C4 质量过滤器（Google）

Google 在构建 C4（Colossal Clean Crawled Corpus）数据集时，在 Gopher 的基础上增加了一批针对网页文本特点的过滤规则，使过滤后的数据更适合预训练场景。C4 过滤器的独特之处在于：

  • 终端标点要求：文档的最后一句必须以句号、感叹号、问号或引号结束。这确保文档是"完整"的，而不是被截断的。
  • "脏词"过滤：如果文档包含一个预定义的脏词列表中的任何词，直接丢弃。这个列表包含约 400 个英语中的歧视性、冒犯性和粗俗词汇。
  • JavaScript 警告移除：丢弃带有 "Javascript"、"lorem ipsum" 等占位文本的页面。
  • 花括号占比：花括号占比过高的文档通常是 JSON 或代码片段，不是正常文本。
def c4_additional_filters(text: str) -> bool:
    """C4 额外的过滤规则"""
    # 规则 1: 终端标点
    stripped = text.strip()
    if not stripped.endswith(('.', '!', '?', '"', '。', '！', '？')):
        return False

    # 规则 2: 脏词检测（简化版本）
    profanity_list = ['badword1', 'badword2']  # 实际有 ~400 词
    text_lower = text.lower()
    for word in profanity_list:
        if word in text_lower:
            return False

    # 规则 3: 花括号占比
    curly_brace_count = text.count('{') + text.count('}')
    if curly_brace_count / max(len(text), 1) > 0.01:
        return False

    return True
需要注意：C4 的脏词列表是针对英语文化的。如果你构建的是中文数据集的过滤管道，不能直接套用——你需要构建自己的中文敏感词和脏词列表。此外，终端标点要求对于中文页面也应当调整：中文文档经常没有终端句号吗？事实上是的——很多中文新闻和博客的结尾并不带句号。在中文场景下，可以考虑放宽这个条件，或者对中文文档单独设置规则。

9.5 FineWeb 质量过滤器：集大成者

2024 年，Hugging Face 发布了 FineWeb 数据集（已被广泛认为是目前最优的公开预训练数据集之一），其过滤器的设计融合了 Gopher 和 C4 的经验，并做了多项改进。FineWeb 的思路是：宁缺毋滥，宁可丢弃可疑的好数据，也不要让坏数据通过。

FineWeb 在 Gopher + C4 基础上的主要改进包括：

  • 更严格的重复阈值：将重复行比例阈值从 30% 收紧到 20%
  • 最小词数提升：从 50 词提升到 100 词
  • 长线检测（Line End Detection）：检测文档中是否存在只有 1-2 个单词的"短线"——这通常是导航栏、页脚的标志
  • 困惑度预筛选：在应用模型评分之前，先用一个轻量级的 n-gram 语言模型计算文档的困惑度，去除那些困惑度异常高（可能是乱码）或异常低（可能是极度重复）的文档
def fineweb_quality_filter(text: str) -> bool:
    """FineWeb 的增强过滤规则"""
    lines = text.split('\n')
    words = text.split()

    # 1. 基础 Gopher 规则（更严格）
    if len(words) < 100 or len(words) > 100000:
        return False

    # 2. 重复行比例（收紧到 20%）
    from collections import Counter
    line_counter = Counter(lines)
    dup_ratio = sum(c for c in line_counter.values() if c > 1) / len(lines)
    if dup_ratio > 0.2:
        return False

    # 3. 短线检测：超过 30% 的行是短线（1-2 个单词），丢弃
    short_lines = sum(1 for l in lines if 0 < len(l.split()) <= 2)
    if short_lines / max(len(lines), 1) > 0.3:
        return False

    # 4. C4 终端标点（FineWeb 对中英文分别处理）
    if not text.strip()[-1] in '.!?"\'。！？）':
        return False

    return True

9.6 启发式过滤链的实际效果

在真实的大规模数据流水线中，这些过滤器的数据淘汰率是相当惊人的。以下是我们从一次 FineWeb 风格的流水线运行中记录的数据缩减情况：

过滤阶段 通过率 累计保留率 主要淘汰原因
原始 HTML 抽取 100% 100% —
语言识别（仅保留中文） ~30% ~30% Common Crawl 中英语占 ~50%，中文占 ~3-5%
词数过滤（50-100K） ~85% ~25.5% 过短页面（导航页、占位页）
重复行检测 ~90% ~23% 模板化页面、拼接文章
终端标点检查 ~80% ~18.4% 截断页面、垃圾内容
字母/符号比率 ~85% ~15.6% 乱码、纯表格/列表数据
短线检测 ~90% ~14% 评论区、导航页

从原始数据到通过初步清洗——你大约只会保留 10-15% 的原始数据。这不是浪费，而是必需的筛选。保留率越低，最终进入训练集的数据质量就越高。

关键洞察：很多做预训练的新手团队对清洗过程的"数据损失率"感到恐慌——觉得丢了 85% 的数据太浪费了。但经验表明：最终进入训练集的 token 数量不是由原始数据量决定的，而是由有效信息密度决定的。你用 1TB 高质量数据训练的模型，效果大概率优于用 5TB 低质量数据训练的模型。清洗不是损失，而是提纯。

本章小结

语言识别和启发式清洗是数据处理流水线中最关键、数据淘汰率最高的阶段。fastText 和 CLD3 各有所长，Gopher、C4、FineWeb 的过滤器设计理念一脉相承但层层加码。把这些过滤器组合成一个可配置的过滤链，你就有了一个稳健的数据初筛系统。

经过这一轮的清洗，我们手里已经有了"干净的文本"——但还有一个巨大的问题没有解决：重复数据。同样的文章可能出现在多个 URL 上，被多次爬取；不同文章之间可能在段落级别高度雷同。这就是下一章的主题——精确去重。
