数据结构是系统设计的骨架

大家好，欢迎来到第25章。如果你之前一直在LeetCode的世界里打转，那么这一章将帮你打开一扇新的大门——真实的系统世界。

在系统设计面试中，面试官不会直接问你"请实现一个哈希表"，而是问你"请设计一个短网址系统"。你可能会想：这跟数据结构有什么关系？关系太大了。短网址系统的核心就是哈希函数的选择和冲突处理策略——这本质上是哈希表在分布式环境下的应用。

事实上，几乎所有系统设计问题都可以拆解为：选择正确的数据结构 + 设计合理的接口 + 处理边界条件。数据结构是骨架，算法是血肉。设计一个高并发系统，你对数据结构的理解深度决定了系统的天花板。

一个核心观点：系统设计面试不问"用什么数据结构"，而是问"为什么选这个"。你要能说出trade-off——为什么选Redis的Sorted Set而不是MySQL的B+树？为什么用跳表而不是平衡树？如果你想不明白这些，那么你对数据结构的理解只是纸上谈兵。

短网址系统 (TinyURL)

短网址是系统设计面试的"Hello World"——几乎所有面经中都会出现。它的功能很简单：把一个长URL映射成一个短字符串（如 https://tinyurl.com/abc123），访问短链接时重定向到长URL。

核心问题就是：怎么生成这个短字符串？

方案一：哈希函数 + Base62编码

对长URL取MD5哈希（128位），取前6个字节，转换为Base62编码（62个字符：a-z, A-Z, 0-9），得到6-7位的短字符串。

import hashlib
import base62  # 注意：base62需要额外安装或用自定义函数

def generate_short_url(long_url: str) -> str:
    """基于MD5+Base62的短链接生成"""
    # 取MD5前6个字节（48位）
    md5_hash = hashlib.md5(long_url.encode()).digest()[:6]
    # 转为整数，再转为Base62
    num = int.from_bytes(md5_hash, byteorder='big')
    return base62_encode(num)

def base62_encode(num: int) -> str:
    """将整数编码为Base62字符串"""
    chars = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789"
    if num == 0:
        return chars[0]
    result = []
    while num > 0:
        result.append(chars[num % 62])
        num //= 62
    return ''.join(reversed(result))

# 示例
print(generate_short_url("https://example.com/very/long/url/here"))
# 输出类似: "d3g8xQ"

哈希冲突怎么处理？

  • 再哈希法：冲突时在长URL末尾加一个固定盐值(如时间戳)重新哈希。
  • 预生成法：预先用随机数生成一批短码，使用时直接分配——避免了在线冲突检测的性能开销。这是工业界的常用做法。

方案二：自增ID + Base62编码

用分布式ID生成器（如雪花算法Snowflake）生成唯一自增ID，然后Base62编码。好处是不会冲突，而且ID天然有序，方便分库分表。

分库分表策略

当短码数量超过单库容量时，需要分库分表。常用策略是按短码的首字符分片——因为Base62的前缀分布均匀，天然适合做分片键。

维度 哈希法 自增ID法
冲突概率 有（需处理） 无（天然唯一）
短码长度 固定（6-7位） 随ID增长而变长
可预测性 不可预测（安全） 可预测（可能被遍历）
性能 无需中心化ID生成器 依赖分布式ID生成器
工程复杂度 低 中
面试回答框架："我会采用自增ID+Base62的方案，用雪花算法生成全局唯一ID，然后转换为Base62短码。使用MySQL存储映射关系，短码作为主键。当数据量超过单库容量时，按短码首字符水平分片。同时用Redis缓存热点短码，降低数据库查询压力。" —— 这就叫"有数据结构的系统设计"。

实时排行榜

实时排行榜几乎是每个互联网产品的标配——游戏战力榜、直播热度榜、电商销量榜。核心需求是：大量用户的分数实时更新，我们需要快速查询Top-K排名。

方案一：跳表实现 Sorted Set

Redis的ZSet（Sorted Set）背后就是跳表（Skip List）。跳表支持O(log n)的插入、删除和范围查询——完美的排行榜需求。每个用户的分数更新只需要O(log n)时间，查询Top-K也只需要O(k + log n)。

为什么Redis选择跳表而不是平衡树？因为跳表的实现更简单，范围查询更高效（平衡树需要中序遍历），而且通过概率平衡避免了复杂的旋转操作。

方案二：堆实现 Top-K

如果系统只需要Top-K（比如Top 10），不需要知道全部排名，那么用最小堆（Min-Heap）维护Top-K是更优的选择。时间复杂度O(n log k)，空间O(k)。

import heapq

def top_k_scores(scores: dict, k: int) -> list:
    """从用户分数映射中找出Top-K"""
    heap = []
    for user, score in scores.items():
        if len(heap) < k:
            heapq.heappush(heap, (score, user))
        elif score > heap[0][0]:
            heapq.heapreplace(heap, (score, user))

    # 按分数降序返回
    return sorted(heap, key=lambda x: -x[0])

# 示例
user_scores = {"Alice": 950, "Bob": 880, "Charlie": 990,
               "David": 720, "Eve": 1050, "Frank": 810}
print(top_k_scores(user_scores, 3))
# 输出: [(1050, 'Eve'), (990, 'Charlie'), (950, 'Alice')]

时间分片策略

对于"小时榜"和"日榜"这类需求，可以用时间分片：

  • 时榜：用Redis Key leaderboard:20260630:14 存储每小时的排行榜数据。
  • 日榜：每天凌晨将24个小时榜合并为日榜。
  • 历史榜：归档到MySQL/时序数据库，提供历史查询。

这个策略的核心思想是：把大时间窗口拆成小窗口，每个窗口独立计算，降低单次计算的数据量。这是系统设计中非常通用的方法。

限流器 (Rate Limiter)

限流器是保护后端系统不被流量冲垮的第一道防线。常见的限流算法有四种，数据结构的角色各不相同。

固定窗口

每个时间窗口（如1秒）内计数，到达阈值则拒绝。简单但有"毛刺"问题——窗口切换时可能突增双倍流量。

滑动窗口（用 deque 实现）

解决了固定窗口的毛刺问题——不再按固定时间点重置，而是记录每个请求的时间戳，窗口是一个滑动的时间段。

from collections import deque
import time

class SlidingWindowRateLimiter:
    """用双端队列实现滑动窗口限流"""

    def __init__(self, max_requests: int, window_seconds: int):
        self.max_requests = max_requests
        self.window_seconds = window_seconds
        self.requests = deque()  # 存储请求的时间戳

    def allow_request(self) -> bool:
        now = time.time()
        # 移除窗口外的旧请求
        while self.requests and self.requests[0] <= now - self.window_seconds:
            self.requests.popleft()

        if len(self.requests) >= self.max_requests:
            return False  # 限流

        self.requests.append(now)
        return True

"""
为什么用deque？
- 需要频繁从队首移除过期请求 —— deque的popleft是O(1)
- 需要从队尾添加新请求 —— append是O(1)
- 如果用list，pop(0)是O(n)，会成为性能瓶颈

复杂度：每次调用平均O(1)，偶尔触发清理需要O(k)（k为过期的请求数）
但每个请求最多被移除一次，所以均摊O(1)。
"""

令牌桶 (Token Bucket)

系统以固定速率向桶中放入令牌，请求到来时从桶中取令牌，取到则放行，取不到则拒绝。允许一定程度的突发流量——桶空时拒绝，但桶会持续补充。

import time

class TokenBucket:
    """令牌桶限流器"""

    def __init__(self, rate: float, capacity: int):
        """
        rate: 令牌放入速率（个/秒）
        capacity: 桶容量（最大突发量）
        """
        self.rate = rate
        self.capacity = capacity
        self.tokens = capacity
        self.last_refill = time.time()

    def allow_request(self) -> bool:
        now = time.time()
        # 计算需要补充的令牌数
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_refill = now

        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

漏桶 (Leaky Bucket)

请求以任意速率进入桶中，但桶以恒定速率"漏出"处理请求。桶满了则新请求被丢弃。相比令牌桶，漏桶更加严格——它强制了处理速率恒定，不能突发。

算法 核心数据结构 是否允许突发 适用场景
固定窗口 计数器(int) 是（窗口边界有毛刺） 简单需求，精度要求不高
滑动窗口 双端队列(deque) 是（更平滑） 大多数通用场景
令牌桶 时间戳+令牌数 是（桶容量控制最大突发） 允许突发的API限流
漏桶 队列+定时器 否（严格恒定速率） 需要平滑流量的场景，如数据库写入
一个真实的踩坑故事：
我之前参与的一个项目，使用固定窗口限流器保护支付接口。上线后一切正常，直到某次大促活动开始。在每分钟切换的瞬间（比如从10:00:59到10:01:00），窗口重置导致流量瞬间暴涨——客户在最后一秒发了1000个请求（窗口内最大），下一秒钟又发了1000个请求。1秒内2000个请求涌入，支付服务直接被打挂。

事后复盘：固定窗口限流的"边界毛刺"是一个经典问题。我们后来改成了滑动窗口限流（用Redis的Sorted Set实现），虽然多消耗了一点内存，但彻底解决了这个问题。从此我学到：选择限流算法时，一定要考虑业务流量的特点。

Feed流推送

Feed流（Timeline/News Feed）是社交产品的核心功能——关注的人发布内容后，你如何看到？这里涉及三种架构模式，每种模式对数据结构有不同的要求。

推模式 (Write Fanout)

发布内容时，将内容写入所有关注者的Timeline中。Timeline用Redis的List或Sorted Set存储（按时间排序）。

  • 优点：读时快（只需拉取自己的List）
  • 缺点：写时慢（大V发一条内容，要写入数百万粉丝的Timeline）
  • 适用：普通用户为主，没有超级大V的场景（如微信朋友圈）

拉模式 (Read Fanout)

发布内容只写入自己的发件箱。阅读时，拉取所有关注者的发件箱，做归并排序。

  • 优点：写时极快，大V没有任何写入压力
  • 缺点：读时慢（需要K路归并，K是关注人数）
  • 适用：大V粉丝极多的场景（如微博）

推拉结合

工业界的常见做法：普通用户用推模式，大V用拉模式。具体来说，当一个用户发布内容时：

  • 如果粉丝数小于阈值（如1000），直接推送到所有粉丝的Timeline。
  • 如果粉丝数大于阈值，只写入自己的发件箱，粉丝在拉取时"顺带"拉取大V的内容。

这里的数据结构选择非常关键：

组件 推荐数据结构 为什么
Timeline缓存 Redis Sorted Set 按时间排序，支持范围查询，O(log n)插入
发件箱 Redis List 按时间顺序存储用户自己发的推文，LTRIM可以截断
关注关系 MySQL + Redis Set Set支持O(1)判断是否关注，MySQL做持久化存储
未读计数 Redis Hash Hash的field是每个关注者的ID，value是未读数

即时通讯 (IM)

IM系统是数据结构在真实系统中应用最为淋漓尽致的领域之一。

消息存储：时间线模型

IM的消息存储通常采用时间线模型——每个对话维护一个按时间排序的消息列表。存储上可以使用：

  • 消息ID：用雪花算法生成全局唯一自增ID——天然有序。
  • 存储结构：每个消息用一行MySQL记录或一个Redis Sorted Set元素。
  • 分页查询：根据消息ID偏移量查询，而不是LIMIT OFFSET——避免深度分页的性能问题。

未读计数

每个用户需要知道"我有多少条未读消息"。最直接的数据结构是哈希表：

# Redis存储结构
# Key: "unread:{user_id}"  Hash
# Field: "{conversation_id}"
# Value: 未读消息数量（整数）

# 发送消息时，给对话中所有其他用户的未读数+1
# 用户打开对话时，将该对话的未读数置0

# 伪代码
def send_message(sender, conversation_id, content):
    # 存储消息
    msg_id = snowflake.next_id()
    redis.zadd(f"conversation:{conversation_id}:messages",
               {msg_id: content})
    # 更新所有其他参与者的未读计数
    members = get_conversation_members(conversation_id)
    for uid in members:
        if uid != sender:

def open_conversation(user_id, conversation_id):
    # 将未读数置0
    redis.hdel(f"unread:{user_id}", conversation_id)
    # 拉取最近的消息
    return redis.zrevrange(f"conversation:{conversation_id}:messages",
                           0, 19, withscores=True)

在线状态：心跳+位图

在线状态的判断通常使用心跳机制——客户端每隔30秒发送一次心跳，服务端记录最后心跳时间。当时间差超过阈值（如70秒），判定为离线。

对于上亿用户的在线状态，内存消耗是一个大问题。一个巧妙的优化是用位图（Bitmap）：

# Redis位图存储在线状态
# Key: "online:20260630:14"  — 按小时分片
# 用户ID作为offset，1=在线，0=离线
# 每个用户只占1个bit，1亿用户仅需约12MB

# 标记用户在线
redis.setbit(f"online:{today_hour}", user_id, 1)

# 批量查询在线状态
def is_online(user_ids):
    results = {}
    for uid in user_ids:
        results[uid] = redis.getbit(f"online:{today_hour}", uid)
    return results

这个位图方案可以做一个粗略估算：1亿用户在线状态只需要 100,000,000 / 8 / 1024 / 1024 ≈ 12MB。对比哈希表（每个用户至少几十字节），内存效率提升了几个数量级。

系统设计面试中的数据结构和算法决策框架：
每当你在系统设计面试中面对需要选择数据结构时，问自己四个问题：
1. 读写比例是多少？ 读多写少→索引优化；写多读少→日志结构。
2. 数据量有多大？ 百万级→单机即可；十亿级→必须分片。
3. 一致性要求有多高？ 强一致→选关系型；最终一致→选NoSQL。
4. 延迟要求是什么？ 毫秒级→内存结构；秒级→磁盘结构。

面试官真正在考察什么？
系统设计面试中，当你提到一个数据结构时，面试官大概率会追问——不要慌，这是展示你思考深度的机会。比如：

"你提到用Redis Sorted Set做排行榜，为什么不用MySQL的B+树？"
你可以回答：
"B+树在磁盘上表现很好，但每个分数更新都需要一次磁盘I/O——对于游戏排行榜这种秒级更新的场景，磁盘I/O的延迟(<10ms)远高于Redis的内存操作(<1ms)。而且Sorted Set的跳表实现使得范围查询（也就是Top-N查询）非常高效，只需要遍历链表即可，而B+树需要多次磁盘寻道。所以在这个低延迟、高并发的场景下，Redis Sorted Set是更合适的选择。"

这就是"为什么选这个"的回答框架：延迟对比 + 数据结构特性 + 场景匹配。
