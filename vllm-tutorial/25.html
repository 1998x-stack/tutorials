第25章 性能基准测试

优化需要基准线。不知道现在的性能是多少，就不知道优化有没有效果。这一章咱们建立一个系统的 vLLM 性能测试方法论。

25.1 测试维度

维度指标怎么测
吞吐量tokens/s, requests/s并发压力测试，逐步加 QPS
延迟TTFT (P50/P95/P99), TPOT固定 QPS，记录延迟分布
显存效率KV cache 利用率, 最大并发数观察 free_blocks 随并发变化
稳定性长时间运行无 OOM/内存泄漏/性能退化至少跑 2hr 持续负载

25.2 压测工具

# 1. vLLM 自带的 benchmark 工具
python -m vllm.entrypoints.openai.run_batch \
    --model meta-llama/Llama-3-8B \
    --prompts prompts.jsonl

# 2. 使用 LLMPerf / vllm-bench
pip install vllm-bench
vllm-bench \
    --host localhost \
    --port 8000 \
    --model meta-llama/Llama-3-8B \
    --num-prompts 1000 \
    --request-rate 10  # 每秒 10 个请求

# 3. 自定义压测脚本 (Python)
import asyncio, aiohttp, time

async def benchmark(url, model, num_requests, concurrency):
    sem = asyncio.Semaphore(concurrency)
    async def send_request():
        async with sem:
            async with aiohttp.ClientSession() as session:
                start = time.time()
                async with session.post(f"{url}/v1/chat/completions", json={
                    "model": model,
                    "messages": [{"role": "user", "content": "写一篇100字的短文。"}],
                    "max_tokens": 200
                }) as resp:
                    await resp.json()
                    return time.time() - start

    tasks = [send_request() for _ in range(num_requests)]
    latencies = await asyncio.gather(*tasks)
    return latencies

25.4 性能调优 Checklist

☐ --enable-prefix-caching             # 如果大部分请求有共享前缀
☐ --kv-cache-dtype fp8                # 显存有限时的首选
☐ --gpu-memory-utilization 0.95       # 尽量高，留 5% buffer
☐ --max-num-seqs 合理值               # 不是越大越好，观察 free_blocks
☐ --max-model-len 按需设置             # 不要设得超过业务需要的最大值
☐ --disable-log-requests              # 关闭请求日志（减少 CPU I/O）
☐ 不用 --enforce-eager                # 让 CUDA Graph 生效（decode 阶段加速）
☐ TP size = 满足显存需求的最小值        # TP 越大通信开销越大
☐ speculative decoding (n-gram)       # 如果 decode 是瓶颈且 temperature 低

基准测试第一原则：一次只改一个变量。如果你同时改了 FP8、启用了 prefix cache、调整了 max-num-seqs，完全无法判断哪个改动带来了多少收益。每个优化独立验证后再叠加。

25.5 解读结果的要点

  • 不要只看平均吞吐：P99 延迟可能已经飙到不可用。同时看 P50/P95/P99
  • 预热很重要：首次推理有 CUDA kernel 编译、权重缓存的开销，前几十个请求的延迟不具有代表性
  • 测试负载要贴近真实：如果你的线上 prompt 平均 500 token，benchmark 别用 100 token 的短 prompt
  • 长时间稳定测试：至少跑 1-2 小时。有些问题（如缓慢显存泄漏）几分钟看不出来
