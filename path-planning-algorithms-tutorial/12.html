大家好！上一章的 RRT 给了我们在高维空间中搜索的能力，但它有一个致命缺陷——找到的路径质量很差。锯齿、绕路、不必要的回头——RRT 的路径让你怀疑"这真的是一条可行的路径吗？"。如果你让一个 RRT 多跑几次，每次得到的路径还不一样。对于追求品质的工程师来说，这不可接受。

RRT*（又称 RRT-Star）由 Sertac Karaman 和 Emilio Frazzoli 于 2010 年提出，解决了这个问题。它在 RRT 的基础上只增加了两个操作——ChooseParent 和 Rewire——但却带来了革命性的性质：渐进最优性（asymptotic optimality）。

RRT 的致命缺陷

为什么 RRT 的路径这么差？原因不在于 RRT 的 "探索" 逻辑有问题，而在于它完全没有"优化"的逻辑。当一个新节点被加入树中时，RRT 简单地把它挂到最近的节点下面。这样就产生了一个问题：

  • 新节点可能通过另一个更远的父节点获得更短的路径——但 RRT 从不检查这种可能性
  • 新节点可能成为它"附近"已有节点的更好父节点——但 RRT 也不做这种反向调整
  • 随着树的生长，早期生成的连接方式可能不再是最优的——但 RRT 从不回溯修正

你可以把 RRT 想象成一个"只管铺路、从不修路"的工程队——路是铺出来了，但全是弯弯绕绕的土路。RRT* 就是那个愿意回头把土路修成高速公路的工程队。

Rewire 操作——核心创新

RRT* 在 RRT 的框架上做了两个关键的改动：

ChooseParent：为新节点选择最佳父节点

当新节点 x_new 加入时，不找最近的节点作为父节点，而是在一个半径 r 的邻域内遍历所有近邻节点，选择那个能使 x_new 的路径代价最小的节点作为父节点。

Rewire：重连近邻

然后反过来：检查 x_new 的所有近邻节点，看能否让 x_new 成为它们的父节点从而缩短路径。如果可以，就切断它们和原父节点的连接，连到 x_new 上来。

这个"双向检查"就是 Rewire 的全部。但它的效果令人惊叹——随着采样增加，路径会逐步趋向最优。

手写 RRT*

在上一章 RRT 代码的基础上，RRT* 只需要添加两个函数：

import numpy as np
from scipy.spatial import KDTree

class RRTStar(RRT):  # 继承自上一章的 RRT
    def __init__(self, start, goal, obstacle_check, bounds,
                 max_iter=1000, step_size=5.0, goal_tolerance=5.0,
                 search_radius=15.0):  # 新增：重连搜索半径
        super().__init__(start, goal, obstacle_check, bounds,
                         max_iter, step_size, goal_tolerance)
        self.search_radius = search_radius
        self.cost = {0: 0.0}  # 从起点到每个节点的代价

    def plan(self):
        tree = KDTree([self.start])
        for i in range(self.max_iter):
            x_rand = self.sample()
            _, idx = tree.query(x_rand)
            x_near = self.nodes[idx]
            x_new = self.steer(x_near, x_rand)
            if not self.obstacle_check(x_near, x_new):
                continue

            # 找邻域内的所有近邻
            indices = tree.query_ball_point(x_new, self.search_radius)
            self.nodes.append(x_new)

            # ChooseParent：选择最佳父节点
            best_parent = None
            best_cost = float('inf')
            for i_idx in indices:
                if not self.obstacle_check(self.nodes[i_idx], x_new):
                    continue
                cand_cost = self.cost[i_idx] + np.linalg.norm(
                    x_new - self.nodes[i_idx])
                if cand_cost < best_cost:
                    best_cost = cand_cost
                    best_parent = i_idx

            if best_parent is None:
                self.nodes.pop()  # 没有合法父节点，丢弃
                continue

            new_idx = len(self.nodes) - 1
            self.parents[new_idx] = best_parent
            self.cost[new_idx] = best_cost

            # Rewire：尝试重连近邻
            for i_idx in indices:
                if i_idx == new_idx:
                    continue
                # 检查能否通过 x_new 缩短近邻的路径
                old_cost = self.cost[i_idx]
                new_cost = self.cost[new_idx] + np.linalg.norm(
                    self.nodes[i_idx] - x_new)
                if new_cost < old_cost and \
                   self.obstacle_check(x_new, self.nodes[i_idx]):
                    self.parents[i_idx] = new_idx
                    self.cost[i_idx] = new_cost

            tree = KDTree(self.nodes)
            if np.linalg.norm(x_new - self.goal) < self.goal_tolerance:
                return self._reconstruct_path(new_idx)
        return None
那个"啊哈"时刻：我第一次用 matplotlib 可视化 RRT* 时，看到了一个让我终生难忘的画面。算法运行了 100 次迭代，路径还是扭曲的——跟 RRT 差不多。200 次迭代，路径开始变直。500 次迭代，路径几乎是一条完美的直线穿过障碍物之间的缝隙。这个过程就像你在看着一个雕塑家不断修正他的作品——每次 Rewire 都是一次精修。当 1000 次迭代结束时，路径已经稳定在最优值附近，后续只是在做微调。这种"看着路径逐渐变好"的体验，跟 RRT 那种"一次成型、粗糙永不变"形成了鲜明对比。如果你也想体验这种感觉，强烈建议你亲自跑一遍，把每次迭代的路径画出来。

Informed RRT*——用椭球加速收敛

RRT* 虽然能渐进最优，但收敛速度可能很慢——尤其是在高维空间中。这是因为即使已经找到了一条不错的路径，RRT* 仍然在整个自由空间中均匀采样——很多采样点对改善当前路径毫无贡献。

Informed RRT* 的思路：一旦找到一条初始路径（代价为 c_best），任何可以改善这条路径的点都必须满足 h(start, x) + h(x, goal) < c_best——这个不等式在几何上定义了一个椭球。Informed RRT* 严格在这个椭球内部采样，大幅提升了采样效率。

效果有多显著？在典型场景中，Informed RRT* 达到同样路径质量所需的采样次数是原始 RRT* 的 1/5 到 1/3。这也是为什么在实际项目中，很少人用原始的 RRT*——大家都在用 Informed 版本。

渐进最优性

RRT* 最引人注目的理论贡献是渐进最优性的证明。Karaman 和 Frazzoli 证明了：当采样点数量 n → ∞ 时，RRT* 返回的路径代价以概率 1 收敛到全局最优解的代价。

这个结论的成立依赖于一个关键设计：重连半径 r 必须满足 r = γ · (log(n)/n)^(1/d)，其中 d 是空间维度，γ 是环境相关的常数。这个半径设计保证了树中连接的"密度"足以覆盖全局最优路径。在实践中，我们通常取一个固定较大的半径值，虽然会牺牲一些计算效率，但收敛性在工程意义上已经足够好了。

下面是 RRT、RRT* 和 Informed RRT* 的对比：

属性 RRT RRT* Informed RRT*
概率完备性 ✓ ✓ ✓
渐进最优性 ✗ ✓ ✓
首次找到路径的速度 快 中等 中等
收敛到最优的速度 N/A 慢 快
实现复杂度 低 中 中高
代码行数（核心） ~50 ~80 ~100

RRT* 家族的另一个重要变体是 RRT#（RRT-Hash）——它用类似 D* Lite 中的 LPA* 机制来维护树的一致性，每次重连后传播代价更新。在需要实时重规划的场景中，RRT# 比 RRT* 更高效。

下一章我们将介绍采样方法的另一大流派——PRM（Probabilistic Roadmap，概率路线图）。如果说 RRT 是用于"单次查询"（single-query）的探索树，那么 PRM 就是用于"多次查询"（multi-query）的路线图。它花时间构建一张覆盖整个自由空间的图，然后可以在这张图上运行任意多次查询——这对静态环境中的多次规划任务来说是理想选择。
