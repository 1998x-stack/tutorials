第2章 vLLM概述

上一章咱们聊了 LLM 推理的基础和痛点。这一章正式认识今天的主角——vLLM。先说说这个名字：vLLM 是 "Very Large Language Model" 的缩写，但它的核心贡献不是"大"，而是"快"和"省"。

2.1 vLLM 的诞生

vLLM 诞生在 UC Berkeley 的 Ion Stoica 实验室（没错，就是那个做 Spark、Ray 的实验室）。2023 年 6 月，几位博士生在实验室里发现了一个问题：明明 GPU 显存还有空余，却没法容纳更多并发请求——因为显存是碎的。

这几位博士生有深厚的系统背景（OS、分布式系统），他们一看这场景就想到了操作系统教科书里的经典解法：虚拟内存分页。于是 PagedAttention 诞生了。同年 10 月，论文在 SOSP 2023（操作系统原理大会，CS 顶会）正式发表。

跨领域洞察：vLLM 是用 OS 的系统思想解决 ML 系统问题。60 年代的分页技术，在 GPU 显存管理上大放异彩。这也是为什么建议做 ML 系统的同学多读一些经典 OS/分布式论文——灵感往往来自跨领域。

2.2 传统推理为什么"慢"

问题原因后果
KV Cache 碎片连续分配，预分配固定长度，分配/释放交错60-80% 显存浪费
静态批处理batch 组好后不能中途换入新请求GPU 利用率 35-58%
重复计算相同前缀的 system prompt 多次计算显存和计算双重浪费
调度粗糙prefill/decode 两阶段硬分离延迟抖动大

这四个问题，vLLM 用四个核心特性逐一击破：

问题vLLM 解法效果
KV Cache 碎片PagedAttention（分页注意力）碎片率 <4%，显存利用率 96%+
静态批处理Continuous Batching（连续批处理）GPU 利用率 85-95%
重复计算Prefix Caching（前缀缓存）RAG 场景成本降低 50-80%
调度粗糙Unified Scheduler（统一调度器）P99 延迟标准差降低 62%

2.3 vLLM 为什么这么流行

vLLM 的日均安装量超过 10 万次，是 GitHub 上 Star 数最多的 LLM 推理项目。为什么？

第一，安装简单。 不需要编译 C++ 项目，不需要配 Docker 环境：

pip install vllm
vllm serve meta-llama/Llama-3-8B

相比 TensorRT-LLM 的需要配置编译环境、HuggingFace TGI 的需要 Docker，vLLM 的门槛低了一个数量级。

第二，性能真的好。 相比 HuggingFace Transformers，vLLM 可以达到 2-4× 的吞吐提升。而且不用调参——默认配置就很能打。

第三，OpenAI 兼容 API。 现有用 OpenAI SDK 写的代码，改一行 base_url 就能切到 vLLM。迁移成本几乎为零。

第四，社区和生态。 Databricks、AWS、Anthropic 等大厂在使用和贡献。支持的模型列表超过 50 个架构，NVIDIA/AMD/Intel GPU + TPU + CPU 全覆盖。

学习建议：如果你是 2024 年开始用 vLLM 的老用户，注意 V1 和 V0 有些 API 差异（比如 best_of 被移除、logprobs 语义变化）。本教程基于 V1 讲解，V0 残留知识需要更新。

2.6 谁在用 vLLM

  • Stripe：迁移后推理成本降低 73%，1/3 GPU 集群处理 5000 万次日均 API 调用
  • Anthropic：内部推理基础设施广泛使用
  • Databricks：集成到 MosaicML 产品线
  • AWS：SageMaker 和 Bedrock 均提供 vLLM 部署选项
  • 大量创业公司：vLLM 已成为 LLM 推理的默认选择

注意：虽然 vLLM 很强大，但它不是银弹。如果你的场景是单条请求异步生成（不需要并发），HuggingFace Transformers 可能更简单；如果你只跑 NVIDIA GPU 且追求极致性能，TensorRT-LLM 在一些 benchmark 上可能更优。选择工具要结合场景。
