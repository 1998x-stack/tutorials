第21章：语音与多模态数据集

想象这样一个场景：你的模型需要同时理解一段视频中的语音内容、画面中的物体、背景音效和字幕文字，然后回答一个关于这段视频的问题。每一个"模态"都有自己的数据格式、采样率、编码方式，而你需要把它们对齐到同一个时间轴上。这就是多模态AI的日常——也是为什么多模态数据工程比单一模态复杂一个数量级。

语音和多模态数据集的共同特点是：数据结构复杂。一个语音样本不只是"音频文件 + 文本转录"，它还涉及采样率、声道数、信噪比、语速等无数变量。一个图文对不只是"图片 + 文字"，还要考虑图文匹配度、描述质量、时空对齐。本章将从语音识别（ASR）和语音合成（TTS）数据集入手，再扩展到通用音频、图文多模态和视频多模态数据集，带你全面了解非文本数据的加载与使用。

核心认知：语音和多模态数据的"加载"不是load_dataset就结束了。你还得处理音频重采样、视频帧提取、模态对齐、特征提取等一系列工程问题。数据的规模从GB级（语音）到TB级（视频）不等，streaming不是选项而是必需品。

语音识别（ASR）数据集

ASR（Automatic Speech Recognition）数据集的结构很清晰：一段音频对应一段文本转录。但"音频"本身有很多参数——采样率、位深、声道数、编码格式——这些参数如果不一致，你的训练管线就会崩。

LibriSpeech — ASR的MNIST

LibriSpeech 由Google和卡内基梅隆大学联合发布，包含约960小时的英文有声书录音。虽然不是最大的ASR数据集，但它的清洗质量极高——所有音频都是16kHz单声道、WAV格式，文本由专业配音演员朗读，几乎没有背景噪音。这使得LibriSpeech成为ASR实验的标准起点。

from datasets import load_dataset

# LibriSpeech 默认返回 Audio 特征
libri = load_dataset("librispeech_asr", "clean", split="train.100")
print(libri[0])
# {'file': '.../path/to/audio.flac',
#  'audio': {'path': '...', 'array': array([...]), 'sampling_rate': 16000},
#  'text': 'CHAPTER ONE MISSUS RACHEL LYNDE IS SURPRISED',
#  'speaker_id': 1272, 'chapter_id': 36, 'id': '1272-128104-0000'}

# audio['array'] 是 numpy array，可以直接送入模型
# sampling_rate 固定为 16000 Hz

注意 audio 字段——datasets库的 Audio 特征类型会自动将音频解码为numpy数组。这是datasets库对比纯粹的文件路径列表的一大优势：你不需要手动用 librosa 或 soundfile 读音频。

Common Voice — 众包的全球语料

Common Voice 是Mozilla发起的众包语音项目，截至2026年6月已覆盖100+种语言，总时长超过2万小时。它的独特价值在于多语言覆盖——如果你在做小语种或方言的ASR，Common Voice几乎是唯一的选择。

# 加载粤语（Cantonese）子集
cv_yue = load_dataset("mozilla-foundation/common_voice_16_0", "yue", split="train")
print(cv_yue[0].keys())
# dict_keys(['client_id', 'path', 'audio', 'sentence', 'up_votes',
#            'down_votes', 'age', 'gender', 'locale', ...])

# Common Voice 有多语言的客户端ID和投票信息
# 可以用 up_votes/down_votes 过滤低质量样本
cv_filtered = cv_yue.filter(lambda x: x["up_votes"] >= 2)
踩坑实录：某团队用Common Voice训练印地语（Hindi）ASR模型，训练了4个epoch后WER（词错误率）高达60%。排查了整整一个月——模型架构没问题、训练参数没问题、数据量也够。最后发现罪魁祸首是tokenizer：他们用的是英文Whisper的tokenizer，而印地语的Devanagari文字（天城文）在英文tokenizer中的覆盖率极低。模型根本不是"听错了"，而是"看到的天城文编码全是"。换成多语言tokenizer后WER直接降到18%。ASR数据预处理中最容易被忽视的环节：确保tokenizer覆盖你的目标语言的所有字符。

WenetSpeech — 中文ASR的旗舰

如果你是做中文语音识别，WenetSpeech 是目前最大的公开中文ASR数据集，包含超过1万小时的音频，涵盖阅读、演讲、对话、综艺等多种场景。它的多场景设计使得训练出的ASR模型对实际应用中的各种音频类型都有不错的鲁棒性。

wenet = load_dataset("wenetspeech", split="train")
# 注意：WenetSpeech很大，建议 streaming
print(wenet[0])
# {'audio': {'array': array([...]), 'sampling_rate': 16000},
#  'text': '...',
#  'segment_id': '...', 'source': '...'}

# WenetSpeech 包含 source 字段标明音频来源
# 'source' 可以是 'audiobook', 'podcast', 'talk', 'voip' 等

三大ASR数据集的核心参数对比如下：

数据集 总时长 语言 采样率 场景覆盖 推荐用途
LibriSpeech 960小时 英文 16kHz 有声书（干净语音） ASR入门、算法对比
Common Voice 2万+小时 100+语言 48kHz（多数） 多语言众包（带噪音） 小语种ASR
WenetSpeech 1万+小时 中文 16kHz 阅读/演讲/对话/综艺 中文ASR
音频加载最佳实践：处理多个ASR数据集时，最头疼的问题是采样率不一致。LibriSpeech是16kHz，Common Voice各有不同（部分语言是48kHz），WenetSpeech是16kHz。在训练之前，一律重采样到模型期望的采样率（Whisper是16kHz，Wav2Vec2是16kHz，Conformer可能有不同要求）。用datasets的 .cast_column("audio", Audio(sampling_rate=16000)) 可以自动完成重采样。

语音合成（TTS）数据集

TTS数据集与ASR数据集看起来相似（都是"音频+文本"），但要求完全不同。ASR关注的是泛化性——大量说话人、各种口音和噪声环境。TTS关注的是音质——极低噪声、均匀语速、标准发音。一个优秀的TTS数据集可能需要数百小时的精心录制和筛选。

LJSpeech — TTS入门的唯一选择

LJSpeech 是最广泛使用的英文TTS数据集。它只有24小时的单人女声录音，来自LibriVox项目的一位朗读者。24小时听起来不多，但它的质量极高：48kHz采样率、无背景噪声、语速平稳。几乎所有开源TTS模型（Tacotron、FastSpeech、VITS）的论文里都用了LJSpeech。

ljspeech = load_dataset("ljspeech", split="train")
print(ljspeech[0])
# {'audio': {'array': array([...]), 'sampling_rate': 22050},
#  'text': '...', 'normalized_text': '...', 'id': 'LJ001-0001'}

# 注意：LJSpeech提供 raw_text 和 normalized_text 两个版本
# normalized_text 已经做过数字展开等标准化
# 建议使用 normalized_text 作为TTS训练的文本目标

VCTK — 多说话人的选择

如果你需要多说话人的TTS训练数据，VCTK（Voice Cloning Toolkit）收录了109位英语说话人的录音，每人约400句，总计约44小时。说话人覆盖了不同口音和年龄段，适合训练语音克隆或多说话人TTS模型。

vctk = load_dataset("vctk", split="train")
print(vctk[0])
# {'audio': {'array': array([...]), 'sampling_rate': 48000},
#  'text': '...', 'speaker_id': 'p225', 'id': 'p225_001'}

# speaker_id 是说话人标识，如 'p225', 'p226'
# 训练多说话人模型时，需要将 speaker_id 映射为embedding
TTS数据注意：音频质量比数量更重要。一个TTS模型在10小时高质量数据上训练的合成效果，往往好过在100小时带噪数据上训练的模型。如果你有预算，建议优先投资录音棚级别的数据采集，而不是在网络上抓取大量低质音频。

通用音频分类

除了语音（人声），AI系统还需要理解非语音音频——门铃声、狗叫声、警笛声、机器运转声。这类环境音分类在智能家居、工业检测、安全监控等领域有广泛的应用。

AudioSet — 谷歌的音频分类航母

AudioSet 由Google发布，包含208万段10秒音频，标注了527个事件类别的层次化标签（从"动物"到"交通工具"到"乐器"）。它的规模和质量使它成为音频分类的标准预训练数据集。

audioset = load_dataset("google/audioset", "balanced", split="train", streaming=True)
print(audioset[0].keys())
# dict_keys(['segment_id', 'start_time', 'duration',
#            'audio', 'labels', 'label_names'])

# labels 是多标签格式：一段音频可能同时包含多个声音事件
# 例如：'Speech', 'Music', 'Vehicle' 可以在同一段中出现

ESC-50 — 轻量环境音

ESC-50（Environmental Sound Classification）是一个小而精的环境音数据集：2000段5秒音频，50个类别，每类40段。虽然规模小，但它的标注质量高、类别定义清晰，是音频分类模型开发的优秀"调试沙盒"。

esc50 = load_dataset("esc50", split="train")
print(esc50[0])
# {'audio': {'array': array([...]), 'sampling_rate': 44100},
#  'label': 0, 'category': 'dog', 'src_file': '...', 'fold': 1}

# fold 字段用于5折交叉验证的标准划分
# ESC-50的论文建议使用交叉验证而非固定train/val split

从音频数据中提取有用的特征通常需要经过以下流程：

import numpy as np
import librosa

def extract_mfcc(audio_array, sr=16000, n_mfcc=13):
    """将原始音频转换为MFCC特征"""
    # 预加重
    pre_emphasis = 0.97
    emphasized = np.append(audio_array[0],
                           audio_array[1:] - pre_emphasis * audio_array[:-1])

    # 提取MFCC
    mfccs = librosa.feature.mfcc(y=emphasized, sr=sr,
                                  n_mfcc=n_mfcc, n_fft=1024, hop_length=512)
    return mfccs.mean(axis=1)  # 沿时间轴平均
Audio特征加载技巧：datasets的 Audio 特征类型支持指定 sampling_rate，它会自动用librosa做重采样。如果你有多个不同采样率的数据集，用 .cast_column("audio", Audio(sampling_rate=16000)) 统一采样率，这样可以省去手动重采样的麻烦。不过注意，降采样（48kHz→16kHz）几乎无损，但升采样（16kHz→48kHz）不会增加信息量，只是把数据点插值变多而已。

图文多模态数据集

图文多模态是2024-2026年发展最快的AI领域之一。GPT-4V、Gemini、Claude 3.5等模型展示了强大的图文理解能力，而Stable Diffusion、Midjourney、DALL·E 3代表了文生图的巅峰。这些模型都依赖大规模、高质量的图文对数据。

Flickr30k — 精细化图文描述

Flickr30k 包含3.1万张Flickr图片，每张图片有5个独立的人工描述。数量虽然不大，但描述质量极高——每个描述都是独立的完整句子，覆盖了图片中的不同细节和角度。它经常被用作图文检索（image-text retrieval）的标准评测基准。

flickr30k = load_dataset("flickr30k", split="train")
print(flickr30k[0].keys())
# dict_keys(['img', 'sent_ids', 'sentences', 'split'])

# sentences 包含5个描述
for sent in flickr30k[0]["sentences"]:
    print(sent["raw"])
# 'A man with a red helmet on a small moped on a dirt road.'
# 'Man riding a motorized bike on a dirt road.'
# ...

Conceptual Captions — 弱监督的大规模替代

Conceptual Captions（CC3M）包含约330万图文对，不同于Flickr30k的人工标注，它是从网页中自动抓取和过滤的——利用图片的alt-text属性和HTML上下文，再经过一系列清洗规则得到。虽然单个样本的质量不如Flickr30k，但规模优势让它成为预训练阶段的理想选择。

cc3m = load_dataset("googleai/conceptual_captions", split="train")
print(cc3m[0])
# {'image_url': '...', 'caption': '...'}

# 注意：和LAION一样，CC3M只包含URL和标题
# 图片需要单独下载，且部分URL可能已失效
图文数据质量金字塔：人工标注（Flickr30k、COCO Captions）> 弱监督清洗（CC3M）> 无监督爬取（LAION）。越往下的数据量越大，但图文匹配度和描述多样性越低。推荐的训练策略：用LAION做广泛预训练（几十亿对），用CC3M做中等精调（几百万对），用COCO Captions/Flickr30k做最终对齐（几万对）。这也是OpenAI在CLIP训练中采用的分层策略。

视频多模态数据集

视频数据是图文数据的"升级版"——加入了时间维度。一个视频不是几十张独立图片的堆叠，而是有时序上下文、运动信息和因果逻辑的连续序列。处理视频数据的核心挑战在于：如何在不丢失时序信息的前提下，有效管理和采样海量的帧数据。

WebVid-10M — 大规模视频描述

WebVid-10M 包含约1000万个短视频及其对应的文本描述，总时长约5万小时。视频来自Shutterstock等素材网站，文本是alt-text。它是训练文生视频模型（如Sora、Runway Gen）的常用预训练数据。

webvid = load_dataset("MaxJeblick/webvid-10m", split="train", streaming=True)
print(webvid[0].keys())
# dict_keys(['videoid', 'url', 'duration', 'caption', 'page_dir'])

# WebVid-10M 同样只提供URL，视频需要下载
# 平均时长约18秒，总时长~5万小时

# 视频帧采样函数（示例）
def sample_frames(video_path, num_frames=8):
    """从视频中均匀采样 num_frames 帧"""
    import cv2
    cap = cv2.VideoCapture(video_path)
    total_frames = int(cap.get(cv2.CAP_PROP_FRAME_COUNT))
    indices = np.linspace(0, total_frames-1, num_frames, dtype=int)

    frames = []
    for idx in indices:
        cap.set(cv2.CAP_PROP_POS_FRAMES, idx)
        ret, frame = cap.read()
        if ret:
            frames.append(frame)
    cap.release()
    return frames

MSR-VTT — 视频检索的标准基准

MSR-VTT（Microsoft Research Video to Text）包含1万个YouTube视频片段，每个视频有20条人工编写的描述。总描述量约20万条。它是图文检索和视频理解领域最广泛使用的评测基准之一。

msrvtt = load_dataset("MSR-VTT", split="train")
print(msrvtt[0])
# {'video_id': 'video0', 'caption': ['...描述1...', '...描述2...', ...],
#  'url': '...', 'video_path': '...'}

# 每个视频有20条描述，覆盖了不同的角度和措辞
print(f"每个视频的描述数: {len(msrvtt[0]['caption'])}")
# 20
视频数据处理提醒：视频数据的体积是巨大的。一个10秒的1080p视频未经压缩约有500MB，即使经过H.264压缩也在10-50MB之间。处理视频数据时请注意：1）千万不要把所有帧都加载到内存中——每帧的读取应该按需进行；2）使用 帧采样策略（均匀采样、随机采样、关键帧提取）控制输入规模；3）训练文生视频模型时，视频片段的分辨率、时长和帧率必须标准化，否则batch内无法对齐。建议先用 streaming=True 做一个采样子集的分析，再设计完整的预处理管线。
多模态数据处理要点：无论语音还是视频，数据加载后的特征对齐是最容易出bug的环节。语音（Audio）的array与文本的长度不对应、视频帧数与字幕条数不匹配、图文对中的图片URL失效——这些都是多模态数据工程中常见的"隐形成本"。建议在正式训练前，先用 .map() 跑一个"健康检查"函数，验证每个样本的关键字段格式正确。

从语音到图文再到视频，多模态数据的世界虽然复杂，但也正是AI最有潜力的方向之一。第22章我们将进入两个"非主流"但同样重要的数据领域——图神经网络（GNN）和强化学习（RL）。它们的结构既不是文本、不是图像、也不是音频，而是节点与边的拓扑关系和状态转移四元组。一个新的世界正在展开。
