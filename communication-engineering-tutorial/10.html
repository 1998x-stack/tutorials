第10章 OFDM与5G帧结构

大家好，我是你们的讲师。如果只能用一个技术来定义4G和5G的物理层，那毫无疑问是OFDM（Orthogonal Frequency Division Multiplexing，正交频分复用）。OFDM不是5G才有的——4G LTE就用它，WiFi 6/7也用，数字广播DVB-T也用。但5G把OFDM推到了全新的高度：灵活的子载波间隔、动态的时隙配置、带宽部分（BWP）等创新，让OFDM从一个"调制技术"演变成了一个"系统设计框架"。

理解OFDM是理解5G物理层的起点。如果你搞懂了OFDM的原理和5G帧结构的设计逻辑，那么物理层的信道映射、Massive MIMO的预编码、波束赋形的时序安排——所有这些都能迎刃而解。这一章我们不讲复杂的数学推导，而是用直观的方式让你理解OFDM"为什么好"以及5G帧结构"为什么这么设计"。

10.1 OFDM的核心思想

OFDM的基本想法其实很朴素：把高速数据流分解成多个低速数据流，并行地在多个子载波上传输。

为什么需要这样做？因为在无线信道中，多径效应会导致符号间干扰（ISI）。假设你以100 Mbps的速率发送数据，每个符号的持续时间只有约10 ns。多径信号之间的时延差可能有几微秒——也就是说，一个符号的延迟版本会严重干扰下一个符号。但如果把数据分成1000路并行传输，每路的速率就只有100 kbps，符号持续时间变为10 μs，远大于多径时延，ISI问题就大大缓解了。

OFDM的关键在于"正交"二字。子载波之间的频率间隔Δf = 1/Tsymbol，即等于符号周期的倒数。在这个条件下，任意两个子载波在符号周期内的积分结果为0——也就是说，尽管子载波在频谱上有重叠，但接收端可以完美地区分每个子载波上承载的数据。这就是正交性的威力：频谱重叠但不干扰，极大地提升了频谱效率。

直观理解：想象一个合唱团，每个歌手唱不同的音调。虽然声音在空气中混合在一起，但你的耳朵（相当于接收端的FFT）可以分辨出每个歌手的声音——因为不同音调的声波是"正交"的。OFDM的原理与此类似，只不过它用的是电波而不是声波。

OFDM的调制和解调通过IFFT（逆快速傅里叶变换）和FFT高效实现：

/* 简化的OFDM发射机：IFFT调制 */
#include <complex.h>
#include <math.h>

#define NFFT 1024  // FFT点数
#define N_SC 600   // 有效子载波数

void ofdm_modulate(complex double *mod_symbols,   // QAM调制后的频域符号
                   complex double *time_samples) { // 输出的时域采样
    complex double freq_domain[NFFT] = {0};

    // 将调制符号映射到有效子载波（直流DC位置留空）
    for (int k = 0; k < N_SC/2; k++)
        freq_domain[k] = mod_symbols[k];
    for (int k = NFFT - N_SC/2; k < NFFT; k++)
        freq_domain[k] = mod_symbols[k - (NFFT - N_SC/2)];

    // IFFT：将频域信号转换为时域信号
    // 实际协议中使用优化的IFFT实现
    for (int n = 0; n < NFFT; n++) {
        time_samples[n] = 0;
        for (int k = 0; k < NFFT; k++) {
            time_samples[n] += freq_domain[k] *
                cexp(2 * M_PI * I * k * n / NFFT);
        }
        time_samples[n] /= NFFT;
    }
}

10.2 循环前缀CP

即使有了OFDM，多径效应仍然会造成子载波间的正交性破坏——多径延迟会导致符号边界错位。解决方案是在每个OFDM符号前面加一个"循环前缀"（Cyclic Prefix，CP）：把符号末尾的一段采样复制到开头。

CP的作用有两个：

  • 对抗多径：只要多径时延不超过CP长度，所有多径分量就都落在CP范围内，FFT窗口可以正确对齐，正交性得以保持。
  • 保护间隔：CP充当了符号间的保护间隔，吸收前一个符号的延迟到达分量，避免ISI。

5G NR支持两种CP配置：Normal CP（每个时隙14个符号，前2个符号的CP略长）和Extended CP（每个时隙12个符号，CP长度固定为512 Ts，仅用于60 kHz SCS）。

10.3 5G NR Numerology

5G NR比4G LTE更灵活的地方在于：子载波间隔（Subcarrier Spacing，SCS）不是固定的15 kHz，而是可以按需选择。3GPP 38.211定义了多种Numerology（参数集），通过参数μ来索引：

μ SCS (kHz) 符号长度 (μs) CP长度 (μs) 每时隙符号数 典型频段
0 15 66.67 4.69 14 Sub-1GHz（与LTE兼容）
1 30 33.33 2.34 14 2.6 GHz / 3.5 GHz（主流5G中频）
2 60 16.67 1.17 14 4-6 GHz / 毫米波（支持Extended CP）
3 120 8.33 0.57 14 毫米波（24-52 GHz）
4 240 4.17 0.29 14 毫米波（用于同步信号，非数据信道）

SCS越大，符号越短，时延越低，但对多径的容忍度也越低（CP更短）。这就是为什么毫米波（高频段）使用大SCS——高频段的多径时延本来就小，而低时延需求更迫切。中低频段则用小SCS，覆盖范围更广。

设计思路：5G的Numerology设计体现了"按需配置"的哲学。eMBB场景用30 kHz SCS兼顾覆盖和容量，uRLLC场景用60或120 kHz SCS降低时延，毫米波用120 kHz SCS应对大带宽需求。一个5G基站可以根据配置同时支持多种Numerology——这在4G LTE中是完全做不到的。

10.4 帧结构：无线帧、子帧、时隙

5G NR的帧结构沿用了LTE的层次化设计，但加入了更多灵活性：

  • 无线帧（Radio Frame）：10 ms，由10个子帧组成。
  • 子帧（Subframe）：1 ms，包含的时隙数取决于SCS。SCS=15 kHz时有1个时隙，SCS=30 kHz时有2个时隙，依此类推（时隙数 = 2μ）。
  • 时隙（Slot）：固定包含14个OFDM符号（Normal CP）或12个符号（Extended CP）。时隙的绝对时间长度 = 1 ms / 2μ。

以主流的中频5G（SCS=30 kHz，μ=1）为例：每10ms无线帧包含20个时隙，每个时隙长0.5ms，每个时隙14个OFDM符号，每个符号约33.33 μs。整个时间结构非常清晰。

10.5 灵活时隙格式：D/U/F动态配置

4G LTE的帧结构是固定的——每个子帧要么全下行，要么全上行。但5G的流量是动态的：白天浏览网页（下行多），晚上直播上传（上行多）。如果帧结构固定，就无法适应这种流量变化。

5G NR引入了灵活的时隙格式（Slot Format），每个时隙中的每个OFDM符号可以单独配置为：

  • D（Downlink）：下行符号
  • U（Uplink）：上行符号
  • F（Flexible）：灵活符号（可由RRC信令或DCI动态决定用作上行或下行）

3GPP TS 38.213定义了多达256种时隙格式（Slot Format Indicator，SFI）。基站可以通过RRC信令配置半静态的时隙格式，也可以通过DCI 2_0动态调整。典型的TDD配置是"7D+2F+5U"（7个下行符号，2个灵活符号，5个上行符号）或"10D+2F+2U"等。

注意：灵活的时隙格式虽然带来了便利，但也引入了新的干扰问题。如果相邻两个基站的上下行配比不一致，就会产生"交叉链路干扰"——隔壁基站的强下行发射会干扰本基站的上行接收。这就需要通过Xn接口协调相邻基站的时隙格式，或者使用统一的区域配置。

10.6 BWP：带宽部分

BWP（Bandwidth Part）是5G NR引入的一项创新功能。在4G LTE中，UE必须始终在整个系统带宽上工作，对于只需要低速率业务的UE（如NB-IoT传感器），这造成了极大的功耗浪费。5G NR允许给UE配置一个子集带宽——即BWP，UE只在BWP内收发数据。

BWP的主要用途：

  • 节能：UE没有大数据业务时，切换到小带宽BWP（例如5 MHz），关闭部分射频链路，功耗降低30-50%。
  • 频谱灵活性：不同Numerology的BWP可以在同一个小区内共存。比如系统带宽100 MHz，一部分BWP用30 kHz SCS做eMBB，另一部分用15 kHz SCS做覆盖增强。
  • 初始接入：UE在初始接入阶段使用Initial BWP（由MIB/SIB1配置），接入后再切换到专用BWP。

BWP切换通过DCI动态触发，切换时延在1-2 ms以内，用户基本感受不到。

10.7 踩坑实录：子载波间隔配置错误导致基站无法解调UE信号

2023年，某厂商在实验室验证5G小站时遇到了一个"诡异"的问题：基站能正常发送SSB和系统消息，UE也能完成下行同步，但UE发送的上行信号基站完全解不出来——PUSCH和PUCCH的BLER都在50%以上，几乎等于随机猜测。

工程师检查了上行功率、定时提前、频偏估计，全部正常。反复对比UE侧和基站侧的配置参数后，终于发现了问题：上行BWP配置的子载波间隔是30 kHz，但基站的接收链路的FFT处理是按照60 kHz SCS初始化的。UE按照30 kHz SCS发送的上行信号，基站的FFT窗口宽度不对，自然解调不出来。

原因是什么？配置文件中的一个枚举值写错了——SCS的配置参数在XML文件中被误写成了"kHz60"，而其他部分都是"kHz30"。这个笔误在生产环境可能不会出现，但在实验室的快速原型验证中极其常见。

教训：5G的Numerology参数虽然灵活，但配置的一致性要求极高。上行和下行、基站和UE、BWP和载波——每一处配置都必须严格对齐。建议在开发阶段使用自动化配置校验工具，定期比对基站侧和UE侧的参数一致性表。一个小小的SCS笔误，浪费了整个团队三天的排查时间。
