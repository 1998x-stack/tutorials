开篇：一个文件撑破天的噩梦

几年前我入职一家创业公司，接手一个用Rust写的后端服务。打开项目一看，src/目录下只有一个文件：lib.rs，整整4300行。一个函数调用另一个函数，全靠CTRL+F搜索。我当时就想：这代码写得，跟把整个家当塞进一个抽屉有什么区别？

这就是典型的"没有模块管理"的惨状。Rust提供了强大的模块系统——从代码组织到可见性控制，再到跨crate依赖管理，一整套体系。很多人学了Rust基础语法后就卡在"怎么组织代码"这一步。

这一章我们就把这些理念讲清楚。学完之后，你将知道如何把一个大型项目拆成恰如其分的小单元，而不是把所有东西塞进一个文件。

包与Crate

在Rust中，代码组织的最小单位是什么？三个概念——Package（包）、Crate（箱）、Module（模块）——层层递进。

Crate：编译单元

Crate是Rust编译器一次编译的最小单位。每一个crate对应一个编译单元，最终生成一个.rlib（库文件）或可执行文件。Rust中有两种crate：

  • Binary Crate：包含fn main()入口，编译为可执行文件
  • Library Crate：提供共享功能，编译为库文件，没有main函数

Package：Cargo项目

Package是Cargo.toml定义的一个项目。一个package可以包含多个crate——最多一个library crate，加上任意多个binary crate。默认的目录结构：

my_project/
├── Cargo.toml
└── src/
    └── main.rs      # 默认binary crate入口
    └── lib.rs       # library crate入口（可选）

如果同时有main.rs和lib.rs，这个package就同时包含一个binary crate和一个library crate（名称与package名相同）。

如果你想有多个binary crate，在src/bin/下放文件即可：

my_project/
├── Cargo.toml
├── src/
│   ├── main.rs
│   ├── lib.rs
│   └── bin/
│       ├── tool_one.rs
│       └── tool_two.rs
└── ...

每个bin/下的.rs文件都单独编译为一个可执行文件。运行特定二进制：cargo run --bin tool_one。

包名与crate名：在Rust中，crate的名字默认就是package名字的snake_case形式。如果package名包含连字符（如my-crate），在代码中引用时用下划线（my_crate）。

模块系统

当你开始把代码拆分到多个文件时，就需要模块（Module）来组织这些文件了。

mod关键字

在src/lib.rs中声明一个模块：

mod network {
    fn connect() {
        println!("connecting...");
    }
}

模块可以嵌套：

mod network {
    pub fn connect() {}

    mod server {
        fn listen() {}
    }
}

模块内部的项默认是私有的（private），只有父模块和显式标记为pub的项才能被外部访问。

文件层次：模块即文件

当模块代码变长时，你可以把它放到单独的文件中：

// src/lib.rs
pub mod network;
pub mod database;

// src/network.rs
pub fn connect() {}

// src/database.rs
pub fn query() {}

对于带子模块的模块，使用目录：

// src/network/mod.rs
pub mod tcp;
pub mod udp;

// src/network/tcp.rs
pub fn connect() {}

// src/network/udp.rs
pub fn bind() {}

从Rust 2018 Edition开始，也支持src/network.rs配合src/network/目录的模式——编译器会优先找network.rs和network/目录并将它们合并。

Rust 2024 Edition变化：从Rust 2024开始，mod.rs惯例被逐步淘汰。强烈建议使用同名的.rs文件加上同名目录的模式。例如network.rs + network/，而不是network/mod.rs。

路径与可见性

有了模块，如何在代码中引用模块中的项？使用路径（Path）。

绝对路径与相对路径

关键字 含义 示例
crate 当前crate的根 crate::network::connect()
self 当前模块自身 self::helper::format()
super 父模块 super::network::connect() 
// src/lib.rs
pub mod network {
    pub fn connect() {}
}

pub mod client {
    // 绝对路径
    fn run() {
        crate::network::connect();
    }

    // 相对路径
    fn run2() {
        super::network::connect();
    }
}

可见性控制：pub的进阶用法

pub关键字不止有"公开/私有"的二元选择，还可以细化可见性范围：

  • pub —— 对所有模块公开
  • pub(crate) —— 仅对当前crate公开
  • pub(super) —— 仅对父模块公开
  • pub(in path) —— 仅对指定路径及其子模块公开
常见陷阱：很多初学者把结构体的所有字段都设为pub，完全违背了封装的初衷。Rust允许结构体本身公开而字段私有——利用这一点，你可以通过公开方法控制对内部状态的修改。只有在确实需要外部直接读写时才用pub字段。

use关键字：路径别名

每次写完整路径太啰嗦，use可以引入路径的快捷方式：

use crate::network::connect;
connect();  // 直接调用

// 引入多个同名项时使用 as 别名
use std::fmt::Result as FmtResult;
use std::io::Result as IoResult;

嵌套路径：

use std::{cmp::Ordering, io::{self, Write}};

引入模块的所有公开项（谨慎使用）：

use crate::network::*;

外部依赖

Rust项目的强劲之处在于它的依赖管理。没有node_modules深似海的恐惧，没有C++手工下载静态库的繁琐。

crates.io

crates.io是Rust官方的包注册中心，上面有超过20万个公开包（crate）。添加依赖很简单：

[dependencies]
serde = "1.0"         # ^1.0 — 兼容1.0的最新版本
tokio = { version = "1", features = ["full"] }
regex = "1.9.0"       # 精确到次版本
chrono = "0.4.38"     # 精确到补丁版本

Cargo.lock

第一次cargo build时，Cargo会解析[dependencies]中的所有依赖，计算出一组满足所有版本约束的依赖版本，并将实际使用的精确版本写入Cargo.lock文件。这个文件应该提交到版本控制（git），确保所有开发者和CI环境使用完全一致的依赖版本。

如果你想升级依赖到最新兼容版本，运行：

cargo update

这会忽略Cargo.lock，重新解析。Cargo.toml中的版本约束仍然生效——cargo update不会跨越^指定的范围升级主版本。

语义版本速查：^1.2.3 表示 ≥1.2.3 且 <2.0.0；~1.2.3 表示 ≥1.2.3 且 <1.3.0；=1.2.3 锁定精确版本；1.2.* 表示补丁版本通配。

依赖来源

除了crates.io，Rust支持多种依赖来源：

[dependencies]
# Git仓库
my_crate = { git = "https://github.com/user/my_crate.git", branch = "main" }

# 本地路径（开发调试用）
my_crate = { path = "../my_crate" }

Workspace：多crate项目管理

随着项目增长，你可能会想把代码拆成多个互相依赖的crate。例如一个web_app项目，拆成core（核心逻辑）、api（HTTP接口）、cli（命令行工具）三个crate。

这时候就需要Workspace（工作空间）。根目录的Cargo.toml中声明子crate：

[workspace]
resolver = "2"
members = [
    "core",
    "api",
    "cli",
]

目录结构：

web_app/
├── Cargo.toml          # workspace配置
├── core/
│   ├── Cargo.toml
│   └── src/
│       └── lib.rs
├── api/
│   ├── Cargo.toml
│   └── src/
│       └── main.rs
├── cli/
│   ├── Cargo.toml
│   └── src/
│       └── main.rs
└── Cargo.lock          # 全局锁文件
Workspace的优势：所有子crate共享一个Cargo.lock和target/目录，避免版本冲突和重复编译。你可以用cargo build -p core只构建特定子crate，也可以用cargo test --workspace一次性测试所有crate。

子crate之间的依赖用path引入：

# api/Cargo.toml
[dependencies]
core = { path = "../core" }

这样就建立了本地依赖关系。api可以直接use core::SomeType;。

Workspace的常见坑：每个子crate的Cargo.toml都要有自己的[package]段，不能缺少version和edition。另外，如果子crate之间有循环依赖（A依赖B，B也依赖A），Rust是不允许的——编译器会报错。这迫使你认真考虑模块划分，理清依赖层次。

实战示例：一个真实的项目结构

下面展示一个我参与过的Rust后端项目结构，让你直观感受生产环境中的模块划分：

auth_service/                    # 认证服务
├── Cargo.toml                  # workspace根
├── Cargo.lock
├── core/                       # 核心库
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs              # 公开模块声明
│       ├── user.rs             # 用户模型
│       ├── token.rs            # JWT令牌逻辑
│       ├── password.rs         # 密码哈希验证
│       └── error.rs            # 自定义错误类型
├── api/                        # HTTP API二进制
│   ├── Cargo.toml
│   └── src/
│       ├── main.rs             # 入口：启动服务器
│       ├── routes/
│       │   ├── mod.rs          # pub mod 声明
│       │   ├── auth.rs         # /auth/* 路由
│       │   └── user.rs         # /user/* 路由
│       └── middleware.rs       # 中间件
├── cli/                        # CLI工具
│   ├── Cargo.toml
│   └── src/
│       └── main.rs             # 管理命令入口
└── tests/                      # 集成测试
    └── integration_test.rs

这种结构的好处：核心逻辑与接口分离，单元测试专注于core，集成测试覆盖跨crate交互。每个crate只对外暴露必要的pub接口，内部实现可以自由重构。
