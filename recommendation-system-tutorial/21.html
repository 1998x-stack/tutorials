第21章 强化学习在推荐中的应用

前面18章讲的内容，本质上都在解决同一个问题："用户现在最可能喜欢什么？"。不管是协同过滤、DeepFM还是双塔模型，它们做的都是"单步预测"——给定当前状态，预估用户对每个物品的兴趣分数，然后选最高的。这种范式有一个隐藏的假设：当前的最优选择，不会影响未来的选择空间。

但现实中这个假设不成立。想想看：你今天给用户推荐了一条"震惊！XXX"的标题党内容，用户确实点击了（短期收益），但用户对平台的信任度下降了（长期损失）。下一次推荐时，用户可能就不那么愿意点击了。这意味着：推荐不仅仅是预测，更是决策——你在塑造用户的未来行为。

这就引出了强化学习（Reinforcement Learning, RL）在推荐中的应用。RL不关心"当前做什么最好"，它关心的是"当前做什么能让未来的一系列收益最大化"。

1. 推荐就是序贯决策

让我们把推荐问题重新表述为强化学习的标准框架：

  • 状态（State）：用户的完整历史行为序列 + 当前上下文（时间、设备、场景等）。状态是动态变化的——每次推荐都改变用户的状态。
  • 动作（Action）：推荐给用户的物品（单条或一个列表）。在大型推荐系统中，动作空间通常是百万到亿级的。
  • 奖励（Reward）：用户对推荐的反应——点击、观看时长、点赞、购买、分享、停留时间等。注意，奖励可以是多维度的。
  • 状态转移（Transition）：用户在看到推荐内容后，状态发生变化——他的历史序列被更新了，兴趣可能被强化或改变，注意力可能消耗了。

在这个框架下，传统的监督学习做的是"单步优化"——给定当前状态S，预测每个动作a的即时奖励R，选择最高的那个。而RL解决的是"长期回报最大化"问题——当前选择的动作，不仅决定即时奖励，还会影响未来的状态和未来的奖励空间。

为什么这对推荐如此重要？因为推荐系统中最棘手的几个问题，本质上都是短期收益 vs 长期收益的冲突：

  • 标题党 vs 优质内容：标题党获得即时的高点击率，但损害用户信任。优质内容短期点击可能不如标题党，但长期提升用户留存。
  • 窄推 vs 宽探：持续推荐用户已知喜欢的领域，短期满意度高、但导致信息茧房。探索性推荐短期可能被忽略，但长期提升平台多样性。
  • 广告 vs 自然内容：展示广告获得直接收入，但过多广告导致用户流失。减少广告短期损失收入，但长期保护用户体验。

这些冲突无法在用监督学习框架内解决，因为监督学习根本没有"未来"这个维度。这就是RL进入推荐系统的逻辑起点。

2. 从Bandit到深度RL

上一章我们讨论了Bandit算法。如果把Bandit看作RL的一个特例，它对RL的推广是沿着"状态"这个维度展开的：

多臂老虎机（Multi-Armed Bandit）：无状态。每个"臂"是独立的，拉哪个臂只影响短期收益，不改变"环境"的状态。这个框架太简单了——在推荐中，你推了一个科技类文章，用户就更可能喜欢下一篇科技文章，状态是变化的。

上下文Bandit（Contextual Bandit）：状态 = 上下文特征。每次决策时能感知到上下文（用户特征、时间、设备等），但动作仍然不影响后续状态——每次推荐是独立的。这是目前工业界最广泛使用的RL类方法。

完整马尔可夫决策过程（Full MDP）：状态随动作变化。当前的推荐改变用户状态，进而影响未来的推荐结果。这是真正的强化学习。例如用户连续看了三条体育内容，状态中体育兴趣被"加热"了，下一次推荐应该给他更多体育内容——但同时也要考虑是否应该推点别的来避免同质化。

这三种方法在状态建模、动作影响和实现复杂度上有本质区别：

方法 状态 动作影响状态 实现复杂度 工业成熟度 典型产出
MAB 无状态 否 低 极高 探索-利用策略，冷启动
Contextual Bandit 上下文特征 否 中 高 个性化探索，多样性优化
Full MDP (深度RL) 完整用户状态 是 高 低到中 长期留存优化，内容发现

在深度RL推荐方面，微软的DRN（Deep Reinforcement Network for News Recommendation）是早期最知名的工业级工作。DRN将DQN（Deep Q-Network）应用于新闻推荐：

  • 用Dueling DQN架构，将Q值分解为状态价值V(s)和动作优势A(s,a)
  • 状态编码：用户近期的新闻浏览序列通过GRU编码
  • 奖励设计：点击=1，不点击=0，阅读时长超过阈值额外加分
  • 使用经验回放（Experience Replay）打破数据相关性

DRN在微软新闻产品上的实验表明，相比基于DeepFM的监督学习基线，DRN在用户点击率上的提升有限（约2-5%），但在用户留存和平均阅读时长上有10%以上的提升。这说明RL的收益主要在"长期指标"上——这正是RL区别于监督学习的核心价值。

实用建议：如果你是第一次在推荐系统里尝试RL，不要直接上深度RL。强烈建议从Contextual Bandit开始——它已经能解决"探索-利用"问题，实现成本低，风险小。实际上，很多工业团队宣称的"RL推荐"，本质上就是Contextual Bandit。Contextual Bandit能带来约70%的RL收益，而实现成本不到完整MDP的10%。

3. 离线策略评估

RL在推荐中落地最大的拦路虎之一就是评估。在监督学习中，评估是直截了当的：在测试集上算AUC、LogLoss就完事了。但在RL中，你不能简单地用历史数据来评估一个新策略——因为历史数据是在旧策略下收集的，你观察到的是旧策略的行为和结果，不是新策略的。

这个问题被称为离线策略评估（Off-Policy Evaluation, OPE）。

主要有三种方法：

  • 直接方法（Direct Method, DM）：用历史数据训练一个"环境模型"（Simulator），然后在模拟器上评估新策略。问题是：如果环境模型不准确，评估结果毫无意义。推荐系统的用户行为模拟是出了名的难做——用户太复杂了。
  • 重要性采样（Importance Sampling, IS）：用旧策略和新策略在每条样本上的概率比值来加权。如果新策略选择了一条样本的概率是旧策略的2倍，这条样本的权重就是2。这个方法无偏但方差极大——当新策略和旧策略差异很大时，重要性权重可以变得非常极端。
  • Doubly Robust（DR）：结合DM和IS。先用一个模型估计奖励，然后用IS来修正模型偏差。DR兼具DM的低方差和IS的无偏性，但实现也更复杂。

讲一个真实的故事。我认识的另一个团队花了一个月实现了DQN新闻推荐模型，在离线评估中看起来非常漂亮——模拟用户点击率提升了15%。团队士气高涨，迫不及待地上线了A/B测试。结果：线上指标全面下跌。离线评估和线上表现完全相反。

问题出在哪里？后来复盘发现：离线数据中几乎没有探索过的新内容被推荐。旧策略是保守的（只推热门），所以离线数据中的"用户反馈"全是关于热门内容的。DQN模型"学会"了推荐热门内容的策略——但它"学会"的方式是猜的，因为模型没有见过用户对长尾内容的真实反馈。当模型上线后开始推长尾内容时，用户的真实反应和离线数据的推断完全不一致。

关键教训：离线策略评估最大的陷阱是"数据覆盖不足"。你的离线数据只覆盖了旧策略探索过的区域，但新策略会探索全新的区域——在这些区域，你对用户行为的预测完全建立在"猜测"之上。这就是为什么很多RL推荐的离线评估看起来很好，上线就崩盘。解决方案是：确保离线数据包含足够的探索性曝光，这样策略评估才有意义。

4. 为什么RL推荐落地这么难

诚实地讲，RL在推荐系统中的落地状况远不如它在游戏和机器人领域那么成功。说句实话：对于大多数推荐团队，RL的投入产出比仍然存疑。我们来分析一下原因：

第一个难题：缺乏好的模拟器。AlphaGo的成功离不开围棋这个完美的模拟器——规则明确、状态完全可观、奖励清晰。但在推荐系统中，没有这样的模拟器。每个用户都是独特且复杂的，你无法用一个数学函数来模拟几亿用户的真实行为。没有好的模拟器，RL训练就只能在真实用户上做——成本高、风险大、迭代慢。

第二个难题：离线评估不可靠。如上节所述，OPE是RL推荐的最薄弱环节。你很难确认你的新策略在离线评估中的表现是真实的还是伪装的。在监督学习中，测试集AUC几乎可以相信；在RL中，OPE指标甚至可能会给出完全相反的方向。

第三个难题：在线风险高。一个未经充分验证的RL策略上线，可能短期提升点击率但长期毁掉用户留存——而留存下降的观测周期可能长达数周甚至数月。没有几个团队敢于承担这种风险。

第四个难题：奖励设计难。推荐系统的目标通常是多维度的——点击、时长、留存、多样性、广告收入……如何把它们融合成一个标量奖励？这个"reward shaping"本身就是一门学问，而且选项的微小差异可能导致策略行为的巨大差异。

下面是一个简化版的DQN训练代码，用于理解基本的RL推荐训练流程：

import torch
import torch.nn as nn
import torch.optim as optim
import random
from collections import deque

class DQN(nn.Module):
    """Deep Q-Network for slate recommendation (simplified)."""
    def __init__(self, state_dim, action_dim, hidden_dim=128):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(state_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, action_dim)  # Q-values for each action
        )

    def forward(self, state):
        return self.net(state)


class ReplayBuffer:
    def __init__(self, capacity=10000):
        self.buffer = deque(maxlen=capacity)

    def push(self, state, action, reward, next_state, done):
        self.buffer.append((state, action, reward, next_state, done))

    def sample(self, batch_size):
        return random.sample(self.buffer, batch_size)


def train_dqn(env, state_dim, action_dim, episodes=1000):
    q_net = DQN(state_dim, action_dim)
    target_net = DQN(state_dim, action_dim)
    target_net.load_state_dict(q_net.state_dict())
    target_net.eval()

    optimizer = optim.Adam(q_net.parameters(), lr=1e-3)
    replay = ReplayBuffer(capacity=20000)
    gamma = 0.99  # discount factor
    epsilon = 0.1  # exploration rate
    batch_size = 64

    for episode in range(episodes):
        state = env.reset()
        total_reward = 0
        done = False

        while not done:
            # Epsilon-greedy action selection
            if random.random() < epsilon:
                action = random.randint(0, action_dim - 1)
            else:
                with torch.no_grad():
                    q_values = q_net(torch.FloatTensor(state))
                    action = q_values.argmax().item()

            next_state, reward, done = env.step(action)
            replay.push(state, action, reward, next_state, done)
            state = next_state
            total_reward += reward

            # Training step
            if len(replay.buffer) > batch_size:
                batch = replay.sample(batch_size)
                states, actions, rewards, next_states, dones = zip(*batch)

                states = torch.FloatTensor(states)
                actions = torch.LongTensor(actions)
                rewards = torch.FloatTensor(rewards)
                next_states = torch.FloatTensor(next_states)
                dones = torch.FloatTensor(dones)

                # Current Q-values
                current_q = q_net(states).gather(1, actions.unsqueeze(1))

                # Target Q-values (Double DQN)
                with torch.no_grad():
                    next_actions = q_net(next_states).argmax(1, keepdim=True)
                    next_q = target_net(next_states).gather(1, next_actions)
                    target_q = rewards.unsqueeze(1) + gamma * next_q * (1 - dones.unsqueeze(1))

                loss = nn.MSELoss()(current_q, target_q)
                optimizer.zero_grad()
                loss.backward()
                optimizer.step()

        # Update target network every 10 episodes
        if episode % 10 == 0:
            target_net.load_state_dict(q_net.state_dict())

        if episode % 50 == 0:
            print(f"Episode {episode}, Total Reward: {total_reward:.2f}")

    return q_net

这段代码是标准的DQN框架，但请注意：在真实的推荐系统中，"环境"（env）不是简单的Python函数——它是一个庞大的线上推荐系统，每次step就是一次真实的用户请求，这决定了RL推荐的迭代成本远高于游戏场景。

务实建议：对于绝大多数推荐团队，我建议采用"渐进式RL"策略：(1) 先从Contextual Bandit开始，解决探索-利用问题；(2) 如果效果不错且团队有RL能力，引入短期-长期奖励的加权；(3) 只在大规模团队和有明确长期优化目标的场景下，才考虑完整MDP的深度RL。这个路径可以把风险降到最低，同时让团队逐步建立RL能力。

5. 小结

强化学习为推荐系统提供了一种全新的世界观——推荐不再是"预测用户此刻喜好"的静态问题，而是"如何通过一系列决策最大化用户的长期价值"的动态优化问题。这个视角从根本上改变了我们看待推荐系统的方式。

从Bandit到Contextual Bandit再到深度RL，RL框架在推荐中的表达力逐步增强，但实现的复杂度、评估的难度和上线的风险也同步攀升。目前工业界的主流实践是Contextual Bandit级别——它能解决探索-利用等核心问题，同时实现成本可控。

坦率地说，完整的深度RL推荐在一线大厂的落地案例仍然有限。这个领域的核心瓶颈不在于算法，而在于基础设施——用户行为模拟器、可靠的离线评估方法、低风险的在线验证机制。这些"工程基础设施"的成熟度决定了RL在推荐中的实际应用边界。

不过，RL推荐有一个极大的优势：它能天然地融入物品的内容信息来做决策。当推荐系统不只是依靠ID Embedding，而是真正理解推荐给用户的是什么内容时，RL的策略可以更加"通用"和"可迁移"。这就引出了我们下一章的主题：多模态与内容理解——推荐系统是如何从"猜你喜欢"进化到"懂你喜欢"的。
