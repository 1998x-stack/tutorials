第6章 错误处理与断言

大家好，我是你们的讲师。上一章我们讨论了内存管理——SQLite 是怎么管好它的"家底"的。这一章我们来看看一个同样重要但容易被人忽视的话题：错误处理。

在系统软件中，错误处理代码常常占到一个项目总代码量的 40% 以上。SQLite 也不例外。但 SQLite 的错误处理之所以值得专门花一章来学习，是因为它把"返回码传播"这件事做成了一套干净、一致、可推理的体系。同时，它的 assert 使用哲学也被业界视为 C 语言防御性编程的典范。

SQLite 错误码体系

SQLite 的错误码体系分为两层：基础错误码和扩展错误码。

基础错误码是 SQLite 核心的返回码，每个 API 函数都返回其中之一。最重要的几个包括：

错误码数值含义典型触发场景
SQLITE_OK0操作成功一切正常
SQLITE_ERROR1通用 SQL 错误语法错误、违反约束等
SQLITE_BUSY5数据库文件被锁定另一个连接正在写入
SQLITE_LOCKED6表被锁定同一个连接内表级锁冲突
SQLITE_NOMEM7内存分配失败OOM 或硬限制到达
SQLITE_READONLY8数据库是只读的尝试写入只读文件
SQLITE_INTERRUPT9操作被中断sqlite3_interrupt() 被调用
SQLITE_IOERR10磁盘 I/O 错误磁盘满、设备移除等
SQLITE_CORRUPT11数据库文件损坏B-Tree 页校验失败
SQLITE_FULL13磁盘已满写操作时磁盘空间不足
SQLITE_CANTOPEN14无法打开数据库文件权限不足或路径不存在
SQLITE_EMPTY16空数据库已打开但尚未创建表
SQLITE_SCHEMA17数据库 schema 已变更另一个连接修改了表结构
SQLITE_CONSTRAINT19约束违反UNIQUE、NOT NULL、FK 等
SQLITE_MISMATCH20数据类型不匹配关联的列类型不兼容
SQLITE_MISUSE21API 使用不当在错误的线程调用、已关闭的连接上操作
SQLITE_ROW100有数据行可用sqlite3_step() 返回了一行
SQLITE_DONE101执行完成sqlite3_step() 完成了所有行的处理

注意两个特殊码：SQLITE_ROW 和 SQLITE_DONE。它们虽然数值上不是 0（不是 OK），但语义上并不是"错误"，而是"有意义的非最终状态"。sqlite3_step() 返回 SQLITE_ROW 时，表示调用者可以读取当前行数据并继续调用 sqlite3_step()；返回 SQLITE_DONE 时，表示所有行都已处理完毕。

扩展错误码是在基础错误码上增加了更细粒度的信息。例如 SQLITE_IOERR 可以被细分为：

SQLITE_IOERR_READ          /* 读数据失败 */
SQLITE_IOERR_WRITE         /* 写数据失败 */
SQLITE_IOERR_FSYNC         /* fsync 同步失败 */
SQLITE_IOERR_TRUNCATE      /* ftruncate 截断失败 */
SQLITE_IOERR_ACCESS        /* access 检查失败 */
SQLITE_IOERR_LOCK          /* 文件锁操作失败 */
SQLITE_IOERR_SHMMAP        /* 共享内存映射失败 */
SQLITE_IOERR_DELETE        /* 删除文件失败 */
SQLITE_IOERR_CHECKRESERVEDLOCK /* 检查保留锁失败 */
SQLITE_IOERR_ROLLBACK_ATOMIC   /* 原子回滚失败 */

要启用扩展错误码，需要在打开数据库连接后调用：

sqlite3_extended_result_codes(db, 1);

扩展错误码的设计理念是：让开发者能够准确区分"同类不同因"的错误。一个 SQLITE_IOERR 太模糊了，你无法判断是磁盘坏了还是文件系统锁有问题。但 SQLITE_IOERR_LOCK 和 SQLITE_IOERR_READ 直接就指出了问题所在。

💡 编码惯例： SQLite 内部有一套严格的"返回码作为函数返回值"的惯例。几乎所有内部函数都按照"0 = 成功，非 0 = 错误码"的约定返回值。这种一致性使得调用者可以无脑地用 if( rc ) return rc; 向上传播错误。这在我们的下一节会详细展开。

rc 返回码传播链

SQLite 中最常见的代码模式就是 rc 变量的传播。rc 通常是一个局部 int 变量，用于捕获内层函数的返回值，并在出错时立即向上返回。我们来看一个典型的调用链：

/* 用户调用层 */
int sqlite3_step(sqlite3_stmt *pStmt){
    Vdbe *v = (Vdbe*)pStmt;
    int rc = sqlite3Step(v);          /* 进入 VDBE 执行层 */
    return rc;
}

/* VDBE 执行层内部 */
static int sqlite3Step(Vdbe *v){
    int rc = SQLITE_OK;
    /* ... 执行循环 ... */
    rc = sqlite3BtreeNext(pCur, 0);   /* 进入 B-Tree 层 */
    if( rc ) return rc;               /* 立即传播 */
    /* ... */
    return rc;
}

/* B-Tree 层 */
int sqlite3BtreeNext(BtCursor *pCur, int *pRes){
    int rc;
    rc = sqlite3PagerGet(pCur->pBt->pPager, pgno, &pPage); /* 进入 Pager 层 */
    if( rc ) return rc;
    /* ... 移动游标 ... */
    return SQLITE_OK;
}

/* Pager 层 */
int sqlite3PagerGet(Pager *pPager, Pgno pgno, DbPage **ppPage){
    int rc = sqlite3OsRead(pPager->fd, buf, pPager->pageSize, offset); /* 进入 VFS */
    if( rc ) return rc;
    /* ... */
    return SQLITE_OK;
}

这个链条从 sqlite3_step() 出发，穿过 VDBE、B-Tree、Pager，最终可能落到 VFS 的 xRead() 实现。任何一层返回了非 0 的错误码，都会沿着调用栈逐层向上传播，最终回到用户 API。

这种设计看似简单——不就是"出错就返回"吗？但它的简洁性背后有几个关键的设计决策：

  • 没有异常机制 — SQLite 是纯 C 代码，没有 C++ 的 try/catch。如果采用"设置全局错误码 + 继续执行"的方式，那么每个函数都需要在执行任何操作前检查错误状态。RC 传播是一种轻量级的替代。
  • 错误码是唯一的"带外"通道 — 正常的函数结果通过参数指针传递，错误码通过返回值传递。两者互不干扰。
  • 错误码从不被忽略 — 代码审查的一个重要准则就是检查是否有函数调用的返回值被忽略了。SQLite 不允许任何"不检查返回值的函数调用"在关键路径上。

sqlite3ErrStr() 函数负责将错误码转换为可读的错误消息字符串。它的实现就是一个大型 switch 语句：

const char *sqlite3ErrStr(int rc){
    switch( rc & 0xff ){  /* 只用低 8 位匹配基础错误码 */
        case SQLITE_OK:         return "not an error";
        case SQLITE_ERROR:      return "SQL logic error";
        case SQLITE_BUSY:       return "database is locked";
        case SQLITE_NOMEM:      return "out of memory";
        case SQLITE_IOERR:      return "disk I/O error";
        /* ... 几十个 case ... */
        default:                return "unknown error";
    }
}
为什么 rc 传播链如此重要？ 因为它是 SQLite 错误处理哲学的集中体现：让错误沿着代码的自然路径传播，不引入特殊的"错误跳转"机制。 你只需要追踪 rc 变量的流动，就能理解整个错误处理逻辑。这比使用 setjmp/longjmp 或全局错误标志要直观得多，也更不容易出错。

assert 的艺术

SQLite 源码中散布着成千上万个 assert 调用。它们不是"事后诸葛亮"式的防御性检查，而是贯彻在代码设计中的"契约式"约束。

Assert 在 SQLite 中的定位非常明确： 它不是错误处理。assert 是用来检查"绝对不可能发生的事情"的——如果触发了，说明代码里有 bug，而不是运行时出现了异常情况。

/* 典型的 sqlite3 内部 assert 用法 */
void sqlite3VdbeFreeCursor(Vdbe *p, VdbeCursor *pCx){
    assert( pCx != 0 );           /* 传入 NULL 游标意味着调用者有 bug */
    assert( pCx->eCurType!=0 );   /* 未初始化的游标意味着逻辑遗漏 */
    /* ... 销毁游标资源 ... */
}

int sqlite3BtreeNext(BtCursor *pCur, int *pRes){
    assert( cursorHoldsMutex(pCur) );  /* 调用者必须持有互斥锁 */
    assert( pRes!=0 );                 /* pRes 不能为 NULL */
    /* ... */
}

SQLite 定义了三个特殊的 assert 变体宏，它们提供了比标准 assert() 更丰富的表达能力：

宏含义示例
ALWAYS(expr)断言 expr 永远为真。如果为假（即使非调试模式也会检查），返回假值让程序继续执行。if( ALWAYS(p!=0) ){ ... }
NEVER(expr)断言 expr 永远为假。如果为真（即使非调试模式也会检查），返回假值让程序继续。if( NEVER(p==0) ) return; // 逻辑上不可能
testcase(expr)测试 expr 是否需要被测试覆盖。仅在调试模式下检查。testcase( n==0 ); testcase( n==1 );

这三个宏的设计非常精妙。ALWAYS 和 NEVER 即使在非调试版本中也会保留条件判断，但它们的语义是"向编译器保证"——告诉编译器"这个分支几乎永远走这里，可以优化掉另一个分支"。而 testcase 则纯粹是测试覆盖率的标记——它告诉测试人员"这里有一个边界情况需要测试到"。

在性能关键路径上，assert 的使用需要特别谨慎。看看 SQLite 怎么做的：

/* B-Tree 的页查找——性能关键路径 */
static MemPage *btreePageLookup(BtShared *pBt, Pgno pgno){
    MemPage *p;
    DO_PAGE_LOOKUP(pBt, pgno, p);   /* 哈希查找 */
    assert( p==0 || p->pgno==pgno );/* 调试时验证一致性 */
    return p;
}

注意 assert 出现在哈希查找 之后，而不是 之前。它只做验证，不做分支决策。这意味着在关闭调试的发布版本中，assert 被完全编译掉，不产生任何运行时开销。这是 assert 使用的黄金法则：不要在 assert 中加入有副作用的表达式。

⚠️ 一个真实的 assert 踩坑： 曾经有一个 SQLite 的 bug 是这么被引入的——开发者在 assert 中放置了一个有副作用的函数调用：assert( sqlite3BtreeNext(pCur, &res) == SQLITE_OK ); 在调试模式下一切正常，但在发布模式下 assert 被移除，sqlite3BtreeNext() 不再被调用，导致游标没有前进，查询返回了无限循环的结果。这个 bug 后来被修复为：rc = sqlite3BtreeNext(pCur, &res); assert( rc==SQLITE_OK ); ——先执行，再断言验证。教训是：永远不要在 assert 的参数中包含有副作用的表达式。这也是一条 C 语言的基本规则，但再怎么强调也不为过。

日志与诊断

除了 assert，SQLite 还提供了一个轻量级的日志系统，用于在运行时记录错误和诊断信息。核心机制是 sqlite3_log() 函数和回调系统的组合。

/* 注册日志回调 */
void sqlite3_config(SQLITE_CONFIG_LOG, myLogFunc, (void*)userData);

/* 日志回调的签名 */
void myLogFunc(void *pArg, int iErrCode, const char *zMsg){
    fprintf(stderr, "SQLite(%d): %s\n", iErrCode, zMsg);
}

/* 在源码中触发日志 */
sqlite3_log(SQLITE_WARNING,
    "unable to close due to unfinalized statements in db %s",
    db->zDbName);

sqlite3_log() 在 SQLite 内部被广泛用于记录以下类型的事件：

  • 资源泄漏警告 — 连接关闭时发现未 finalize 的 prepared statement。
  • 性能警告 — 检测到数据库文件碎片可能影响性能。
  • 恢复事件 — 数据库从崩溃中恢复时，记录了哪些事务被回滚。
  • 锁等待 — 在繁忙时的锁等待日志。
  • VFS 错误详情 — 操作系统返回的错误代码（如 errno）的翻译。

对于生产环境的应用，注册一个合理的日志回调是极其重要的。想象一下：你的 SQLite 数据库在某个客户的机器上频繁返回 SQLITE_CORRUPT，但你的应用只是简单地把错误码显示给用户。如果没有日志，你和客户都无法知道到底哪一页损坏了、损坏的原因是什么。但如果注册了日志回调，SQLite 会输出类似 "wrong # of entries in index 'idx_users_email'" 这样的详细信息，直接定位问题所在。

错误处理的哲学总结： SQLite 的整个错误处理体系建立在几个简单原则之上——(1) 函数返回值即错误码，与正常结果分离；(2) 错误沿着调用链自然传播，不引入特殊机制；(3) assert 只检查"绝不可能"的场景，不用于运行时防御；(4) 日志用于记录"不致命但值得注意"的事件。这套体系不花哨，但极度可靠。在系统软件开发中，"朴素但正确"远比"聪明但脆弱"有价值。

好了，第 6 章我们走了一遍 SQLite 的错误处理体系。从基础错误码到扩展错误码，从 rc 传播链到 assert 的使用哲学，再到运行时日志系统——你看到的是一套贯穿全程的、严格一致的错误管理哲学。这种哲学不仅让 SQLite 更健壮，也让它的代码更容易被人类理解。
