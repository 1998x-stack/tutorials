从面试到竞赛的思维跃迁

同学们好，走到第26章，意味着你已经系统性地学完了 LeetCode 上绝大多数的核心算法和数据结构。此时此刻，你刷过的题应该已经有几百道了，常见的套路基本都见过了。但你有没有发现——有些人参加 LeetCode 周赛，四道题一个小时就 AK（All Kill）了；而你吭哧吭哧做两个半小时，第三道题可能还没调出来？

差别在哪？不是知识量，而是竞赛思维。平时刷题你有充足的时间，可以慢慢想、慢慢试。但竞赛或者面试白板编程的本质是在有限时间内做最优决策——读题要快、判断难度要准、分配时间要狠、放弃要果断。这一章我就带你掌握这些在实战中极其宝贵的技巧。

重要认知：竞赛思维不是天赋，而是可以训练的方法论。它包含四个维度——读题策略、时间管理、逆向思维、陷阱识别。每一条都能通过刻意练习提升。

周赛实战策略

LeetCode 双周赛/周赛每场四个题，时长 90 分钟。怎么分配时间？我总结了一套"三分钟读题法"。

读题策略

拿到题目后，用三分钟完成四件事：

  1. 看输入规模——这是最重要的信号。n ≤ 20 暗示回溯/状态压缩；n ≤ 10⁵ 暗示 O(n log n) 或 O(n)；n ≤ 10⁹ 暗示数学公式或 O(log n)。
  2. 看样例——有些题目文字描述很长很绕，但样例一看就明白要求什么。先看样例，再回看描述。
  3. 确定数据结构——看到"连续子数组"想滑动窗口/前缀和；看到"第 k 大/小"想堆或二分；看到"最短路径"想 BFS/Dijkstra。
  4. 打个标签——在心里或纸上标记：Easy / Medium / Hard，预估所需时间。

难度判断与时间分配

我的时间分配原则：

题目位置 典型难度 建议用时 策略
Q1 Easy 5-10 分钟 必须拿下，不要炫技，用最简单的写法
Q2 Medium 10-20 分钟 核心得分点，严谨推导，边界稳一点
Q3 Medium~Hard 20-30 分钟 如果 15 分钟没思路，先跳 Q4 再回来看
Q4 Hard 剩余时间 能做就做，不能做就写暴力部分分
时间管理口诀：Q1 求稳，Q2 求快，Q3 求突破，Q4 求缘分。不要在一道题上死磕超过 30 分钟——那不是坚持，是愚蠢。

思维技巧

竞赛中最值钱的不是你会多少算法，而是你能不能想到那个关键的转折点。下面三个思维技巧是我的压箱底存货。

逆向思维：从答案反推

有一类题目叫"给定条件，求某个值"。正向推导可能需要很复杂的数学变换。但如果你 从可能的答案出发反向验证，常常会简单很多。

经典案例：Koko Eating Bananas（LeetCode 875）。题目是：Koko 要在 h 小时内吃掉 n 堆香蕉，每堆有 piles[i] 根。她每小时吃 k 根，如果一堆少于 k 根就吃完这堆然后等下一小时。求最小的 k。

正向思考：k 的范围从 1 到 max(piles)，怎么找最优的 k？如果你去模拟，太慢了。换个方向：假设我猜一个 k，能否在 h 小时内吃完？——这不就是检查吗？然后把"猜测"换成二分查找。

def min_eating_speed(piles, h):
    def can_finish(k):
        hours = 0
        for pile in piles:
            hours += (pile + k - 1) // k  # 向上取整
        return hours <= h

    left, right = 1, max(piles)
    while left < right:
        mid = (left + right) // 2
        if can_finish(mid):
            right = mid
        else:
            left = mid + 1
    return left

这道题的"反转"在于：不知道 k，但可以验证 k。把"求值问题"变成"判定问题"，再用二分搜索答案空间——这是竞赛中最经典的套路之一。

正难则反

有时候直接求解非常困难，但求它的补集却很容易。这时候就用"总数 - 反面"来得到答案。

一个超级典型的例子：求数组中所有和为某个值的子数组个数。正向枚举显然不现实。但我们有前缀和 + 哈希表的技巧——本质上就是用"部分和相减"来完成任务。

def subarray_sum(nums, k):
    count = 0
    prefix_sum = 0
    sum_count = {0: 1}  # 前缀和为 0 出现了 1 次

    for num in nums:
        prefix_sum += num
        # 如果存在前缀和为 prefix_sum - k，则中间那段和为 k
        count += sum_count.get(prefix_sum - k, 0)
        sum_count[prefix_sum] = sum_count.get(prefix_sum, 0) + 1

    return count

"正难则反"的另一种形式是用总数减去非法情况。比如求"所有子数组中满足某条件的个数"，如果合法条件复杂，你就求"总数 - 不合法子数组个数"，往往简单很多。

增量构造

竞赛中有一类"构造题"——给你一些约束，让你构造出一个满足条件的序列。增量构造的套路是：一个一个元素地加，每一步都保证当前部分满足局部约束。

举个例子，Shuffle the Array（LeetCode 1470）：给定 [x1,x2,...,xn,y1,y2,...,yn]，重排为 [x1,y1,x2,y2,...,xn,yn]。这道题很简单，但它背后的"双指针交替取数"思想在更复杂的构造题里非常有用。

def shuffle(nums, n):
    result = []
    for i in range(n):
        result.append(nums[i])
        result.append(nums[i + n])
    return result

增量构造的关键在于：不追求一步到位，而是每一步都推进一点点，最终自然得到全局解。这在贪心算法中也广泛使用。

常见竞赛陷阱

掌握了技巧，还要避免踩坑。竞赛中的 Bug 很多时候不是算法设计问题，而是对环境的不熟悉。

大数溢出

Python 的 int 是无限精度的，看起来不会溢出——但 CPU 时间和内存不是无限的！当数值变得极大时（比如 Fibonacci 第 10⁶ 项），Python 的大整数运算会变得非常慢。更常见的陷阱是：你以为不会大，但循环中的乘法让你的数变成了 10³⁰⁰⁰。

Python 注意：虽然 Python 不会像 C++ 那样 int 溢出，但大整数运算的耗时是 O(n) 级别的（n 为位数）。一个包含 10000 位大整数的乘法足够让你的代码 TLE。如果题目要求对大质数取模，那你每一步操作后都要模，不要只在最后模。

Python 的递归限制

Python 默认递归深度是 1000。DFS 二叉树还行，但如果是链状树（深度 20000），你的递归会直接 RecursionError。

解决方案有三条：

  1. 改用栈模拟递归（最稳妥）
  2. 调用 sys.setrecursionlimit(1000000) ——但这不是万能的，Python 递归本身开销大，深层递归仍然可能栈溢出
  3. 如果可以用迭代，优先迭代
import sys
sys.setrecursionlimit(1000000)

# 但更好的做法是用栈模拟
def dfs_iterative(graph, start):
    stack = [start]
    visited = set()
    while stack:
        node = stack.pop()
        if node in visited:
            continue
        visited.add(node)
        for neighbor in graph[node]:
            stack.append(neighbor)

TLE 优化 checklist

当你的代码 TLE 时，按以下顺序排查：

  1. 算法复杂度不对——比如该用 O(n log n) 的用了 O(n²)，换算法。
  2. 数据结构选错——列表的 in 操作是 O(n)，换成 set 或 dict。
  3. 不必要的复制——nums[:] 或 sorted(nums) 在循环里调用会炸。
  4. 递归换成迭代——特别是树/图的 DFS，迭代往往比递归快 2-3 倍。
  5. Python 特有的慢操作——for i in range(len(arr)) 改成 for val in arr；slicing 改成索引访问。
调试建议：周赛中如果某个测试用例 TLE 了，不要傻看代码。先加一个 if len(nums) > 10000: return -1 之类的快速判断来缩小问题范围，再针对性优化。

序列与构造题

构造题是竞赛中的常客，但在面试中比较少出现。这类题目考验的是你对数学规律的敏感度。

构造特定序列

常见的构造思路：

  • 奇偶交替——把所有奇数放前面、偶数放后面，或者按奇偶位置交替。
  • 等差/等比——利用等差数列求和公式构造特定目标。
  • 位运算构造——比如用 XOR 性质构造互不相同的数。
  • 贪心构造——每一步都选当前最优，使得最终结果满足条件。
# 构造一个长度为 n 的排列，使得相邻元素差值的绝对值恰好为 1 或 n-1
def construct_array(n):
    # 一个经典构造：头尾交替
    result = []
    left, right = 1, n
    while left <= right:
        result.append(left)
        left += 1
        if left <= right:
            result.append(right)
            right -= 1
    return result

# 示例：n=5 → [1, 5, 2, 4, 3]
# 相邻差值：4, 3, 2, 1 — 都是 [1, n-1] 内的值

模拟题的高级技巧

模拟题（Simulation）是很多人不喜欢的类型——"不就是按题意写代码吗？有什么难的？" 但问题在于，题意可能非常复杂，比如"旋转矩阵"、"螺旋遍历"、"生命游戏"。

模拟题的诀窍是 建立方向向量：

# 螺旋矩阵 (LeetCode 54)
def spiral_order(matrix):
    if not matrix:
        return []
    rows, cols = len(matrix), len(matrix[0])
    # 方向向量：右、下、左、上
    dr = [0, 1, 0, -1]
    dc = [1, 0, -1, 0]
    visited = [[False] * cols for _ in range(rows)]

    result = []
    r = c = di = 0
    for _ in range(rows * cols):
        result.append(matrix[r][c])
        visited[r][c] = True
        nr, nc = r + dr[di], c + dc[di]
        if not (0 <= nr < rows and 0 <= nc < cols and not visited[nr][nc]):
            di = (di + 1) % 4  # 转向
            nr, nc = r + dr[di], c + dc[di]
        r, c = nr, nc
    return result

方向向量法是模拟题的"万能钥匙"。不管题目是走迷宫、螺旋矩阵、还是机器人行走，一个方向数组 + 边界检查 + 转向逻辑就能搞定几乎所有情况。

竞赛中10种常见陷阱及应对

# 陷阱 现象 应对策略
1 大数溢出（C++） int 变负数 用 long long 或 Python 但注意模运算
2 Python 递归深度 RecursionError setrecursionlimit 或用栈模拟
3 全局变量污染 多用例时互相干扰 每次重新初始化，或在函数内定义
4 浮点数精度 0.1 + 0.2 != 0.3 用 Decimal 或转整数比大小
5 数组越界 IndexError 检查 -1 下标、空的输入
6 边界条件遗漏 n=0 或 n=1 时报错 先写边界判断作为习惯
7 哈希表覆盖旧值 相同 key 更新丢失数据 确认是覆盖还是需要 append
8 排序改变原数组 后续操作依赖原序 排序前 copy 或备份索引
9 死循环 程序不结束 while 循环检查终止条件，加计数器
10 读错题目 样例不过 重新读题，关注输入格式约束
最可怕的陷阱：读错题。我见过太多人——包括我自己——在竞赛中花 40 分钟写了一个"解法"，最后发现题目读反了（比如求最大值看成最小值）。我的习惯是：在写代码之前，用自己的话把题目复述一遍，或者在纸上画个样例，确认理解无误再动手。

踩坑与总结

说到踩坑，我必须分享一个印象深刻的故事。有一次 LeetCode 周赛，Q4 是一道 Hard 题，我读了 10 分钟完全没思路，然后开始死磕。30 分钟过去了，我写了各种尝试代码，都不对。45 分钟过去了，我还在挣扎。最终 90 分钟结束，我的 Q4 一分没拿到，而且 Q2 和 Q3 也因为时间不够写得毛毛躁躁，各错了一个 corner case——总排名掉了 1000 名。

赛后我看题解，发现 Q4 的解法我其实会的——就是 二分答案 + 贪心验证，都是学过的套路。但我"卡"在第一道关：没想通"验证函数怎么设计"。如果我当时先跳过去做 Q3，把 Q3 的分稳稳拿到，再回来看 Q4，结果会完全不一样。

止损策略：Hard 题先写暴力，拿部分分。如果暴力都不会写，果断跳过，把时间留给前面题目的正确率和速度。周赛的竞争不是"谁能做出 Hard"，而是"谁在 Mid 上不出错"。相信我，一场比赛中 Q2+Q3 全部一遍 AC 的人，排名一定在前 10%。

从周赛里，你不应该期待每道题都做出来。相反，你要从每场周赛中提取出至少一个通用技巧——这周的 Q3 用了一个巧妙的排序技巧，记下来；下周的 Q4 用了一个状态压缩 DP，记下来。积累 20 场周赛，你的"工具箱"里就多了 20 个武器。

竞赛思维的本质，不是在 90 分钟内创造新的算法，而是在你已有的知识库里快速检索并准确应用。这需要刻意练习——刷周赛、复盘、总结、再刷。没有什么捷径。
