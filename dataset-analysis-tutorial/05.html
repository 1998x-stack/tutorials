第5章：数据类型与格式转换

前面几章我们一直在datasets库的"舒适区"里——加载、查看元信息、理解Arrow结构。但数据工程师的日常不是停留在舒适区。真实场景中，数据最终要喂给PyTorch、TensorFlow、JAX或scikit-learn。Datasets库的Dataset对象在这些框架面前是"异类"——你没法把一个Dataset直接丢进 model.train() 里。你需要把数据格式转换为目标框架可以消费的形式。

这正是 set_format()、with_format() 以及各种 to_*() 方法的用武之地。这一章我们系统性地讲解datasets库的格式转换体系，以及其中的各种"暗坑"。

set_format() 基础

set_format() 是Dataset对象上最核心的格式转换方法。它就地修改Dataset的数据返回格式——此后所有通过索引或切片访问的数据都会按照指定的格式返回。

支持的格式

格式 方法参数 返回类型 适用框架 适用场景
Python原生 （默认） dict / list 通用 数据浏览、调试
numpy format="numpy" dict of np.array numpy, sklearn 数值计算、传统ML
torch format="torch" dict of torch.Tensor PyTorch 深度学习训练
tensorflow format="tensorflow" dict of tf.Tensor TensorFlow 深度学习训练
pandas format="pandas" pd.DataFrame pandas 数据分析、可视化
jax format="jax" dict of jnp.array JAX JAX训练、函数变换
polars format="polars" pl.DataFrame Polars 高性能数据处理

基本用法

from datasets import load_dataset

dataset = load_dataset("imdb", split="train")

# 默认格式：Python原生dict
sample = dataset[0]
print(type(sample["text"]))   # str
print(type(sample["label"]))  # int

# 切换到numpy格式
dataset.set_format(type="numpy")
sample = dataset[0]
print(type(sample["text"]))    # numpy.str_
print(type(sample["label"]))   # numpy.int64

# 批量样本也是numpy数组
batch = dataset[:5]
print(type(batch["label"]))    # numpy.ndarray, shape=(5,)

# 切换到torch格式
dataset.set_format(type="torch")
batch = dataset[:5]
print(batch["label"].shape)    # torch.Size([5])
print(type(batch["label"]))    # torch.Tensor
关键理解：set_format() 并不会修改底层Arrow数据——它只是改变了"读取时"的返回类型。数据始终以Arrow列式格式存储在内存中，格式转换发生在读取时，是一个"视图"而非"副本"。这意味着切换格式的成本极低。

按列指定格式

有时你不需要所有列都转换格式。例如，text列是字符串（保持为Python str即可），只有label列需要转为torch tensor：

dataset.set_format(type="torch", columns=["label"])
# 只转换label列
sample = dataset[0]
print(type(sample["label"]))   # torch.Tensor
print(type(sample["text"]))    # str — 不变

# 也可以指定需要排除的列
dataset.set_format(type="torch", columns=["label", "text"], output_all_columns=True)

with_format() 上下文管理

set_format() 是永久性的——一旦设置，后续所有访问都受影响。这有时会带来麻烦：你在这个cell里设了torch格式，跑到下一个cell要查看文本内容，结果文本也变成tensor了，还得重新切回默认格式。

with_format() 提供了临时性的上下文管理器：

dataset = load_dataset("imdb", split="train")

# 在上下文块内，数据以PyTorch格式返回
with dataset.with_format(type="torch"):
    batch = dataset[:3]
    print(batch["label"].shape)  # torch.Size([3])
    # 在这个块里所有操作都用torch tensor

# 离开上下文后，自动恢复为默认格式
sample = dataset[0]
print(type(sample["label"]))  # int — 恢复默认

# 上下文也可以嵌套
dataset.set_format(type="numpy")  # 全局设为numpy

with dataset.with_format(type="torch"):
    batch = dataset[:3]
    print(type(batch["label"]))  # torch.Tensor — 临时覆盖

# 恢复为numpy
print(type(dataset[0]["label"]))  # numpy.int64

在实际的training loop中，with_format() 比 set_format() 更常用——你不想在全局修改数据格式，只希望在训练循环的上下文内让数据以tensor形式出现。

最佳实践建议：在你自己的代码中，优先使用 with_format() 管理格式转换。它有明确的作用域，不会意外污染后续代码的行为。只有在你确信整个会话阶段都只需要一种格式时，才用 set_format()。

多后端切换实战

在团队协作中，不同成员可能使用不同的深度学习框架。batasets的灵活格式切换让多框架协作变得简单：

# 同一份数据，为不同框架切换格式
dataset = load_dataset("cifar10", split="train")

# PyTorch团队
with dataset.with_format("torch"):
    images = dataset[:4]["img"]
    print(images.shape)  # torch.Size([4, 32, 32, 3])

# TensorFlow团队
with dataset.with_format("tensorflow"):
    images = dataset[:4]["img"]
    print(images.shape)  # (4, 32, 32, 3) — tf.Tensor

to_pandas() / to_polars() 互转

虽然 set_format("pandas") 可以让你在Dataset访问时获得pandas DataFrame，但在很多场景下你需要的是一份"真正的"DataFrame——可以传给seaborn画图、用pandas的groupby做聚合、或者写入Excel报表。这时就需要 to_pandas() 和 to_polars() 方法了。

Dataset → Pandas

import pandas as pd
from datasets import load_dataset

dataset = load_dataset("imdb", split="train")

# 转换为Pandas DataFrame
df = dataset.to_pandas()
print(type(df))           # pd.DataFrame
print(df.shape)           # (25000, 2)
print(df.dtypes)
# text     object
# label     int64

# 现在可以使用所有pandas操作
label_dist = df["label"].value_counts()
print(label_dist)
# label
# 1    12500
# 0    12500

# 文本长度分析
df["text_length"] = df["text"].str.len()
print(df["text_length"].describe())
# count    25000.000
# mean       231.478
# std        192.472
# min          7.000
# max       2470.000
内存警告：to_pandas() 会生成一份完整的数据副本（从Arrow格式转换为Pandas格式）。对于大型数据集（超过可用内存的50%），这可能导致OOM。如果你只是做简单的统计，用 dataset["column"] 直接在Arrow上操作更省内存。需要Pandas功能时，可以先用 select() 取子集再转换。

Pandas → Dataset

# 从Pandas创建Dataset
df = pd.DataFrame({
    "text": ["Hello world", "Goodbye world"],
    "label": [1, 0],
})
dataset = Dataset.from_pandas(df)
print(dataset.features)
# {'text': Value(dtype='string'), 'label': Value(dtype='int64')}

# 类型保真度验证
print(dataset[0]["label"])        # 1 (Python int)
print(type(dataset[0]["label"]))  # int

注意从Pandas转换到Dataset时，类型会经历一次"Arrow化"——pandas的 int64 变成 Value("int64")，object 变成 Value("string")。这通常没有问题，但如果你在pandas中使用了 pd.Categorical 类型，转换后不会自动变成 ClassLabel——你需要手动指定features。

Polars互操作性

datasets v5.0.0 新增了对Polars的原生支持。Polars是一个比Pandas快得多的DataFrame库，基于Arrow原生构建：

# Dataset → Polars（零拷贝！）
pl_df = dataset.to_polars()
print(type(pl_df))  # polars.DataFrame

# Polars → Dataset
dataset_from_pl = Dataset.from_polars(pl_df)

Dataset和Polars之间的转换是零拷贝的——两者都基于Arrow内存格式，所以不需要数据序列化。这是Pandas做不到的（Pandas基于NumPy，需要数据重组）。

to_iterable_dataset()

除了格式转换，数据集的"访问模式"也可以转换。前面几章我们用的都是map-style dataset——可以通过索引随机访问任意样本。但在某些场景下，比如处理无限流数据、或者数据集太大无法全部加载时，就需要可迭代数据集了。

from datasets import load_dataset

dataset = load_dataset("imdb", split="train")

# 转换为可迭代数据集
iter_dataset = dataset.to_iterable_dataset()
print(type(iter_dataset))
# 

# 可迭代数据集不支持随机访问
# iter_dataset[0]  # 会报错！

# 只能顺序遍历
for i, sample in enumerate(iter_dataset):
    if i >= 5:
        break
    print(sample["text"][:50])

to_iterable_dataset() 在你需要按顺序处理大量数据、并且不需要随机访问时非常有用。它的主要优势是内存占用极低——每次只保留一个batch在内存中。

适用场景判断：
  • 需要随机访问（按索引取样本）→ 用map-style Dataset
  • 顺序遍历 + 数据量大 → 用IterableDataset（更省内存）
  • 数据量超过内存 → 用 load_dataset(streaming=True)（根本不会下载到本地）
第12章会详细讲解流式处理和IterableDataset的高级用法。

格式转换中的常见坑

格式转换看起来很简单——不就是设个type参数吗？但实际项目中，这里暗藏着几个非常容易踩中的陷阱。

坑1：图像列转换丢失通道信息

dataset = load_dataset("cifar10", split="train")

# 图像列默认是 PIL Image 对象
sample = dataset[0]
print(type(sample["img"]))  # PIL.PngImagePlugin.PngImageFile
print(sample["img"].size)   # (32, 32)

# 设置为torch格式后
dataset.set_format("torch")
sample = dataset[0]
print(sample["img"].shape)  # torch.Size([3, 32, 32])
# 注意：默认是 CHW 格式（通道在前），而不是HWC！

如果你习惯用OpenCV（HWC格式），这里很容易出问题。PyTorch的CNN期望输入是 (C, H, W)，而大多数图像库返回的是 (H, W, C)。set_format("torch") 默认做了 HWC → CHW 的转置——这个行为是设计好的，但如果你不知道，就会在模型维度不匹配时排查半天。

坑2：嵌套dict列被展平

# 一个有嵌套结构的样本
nested_dataset = Dataset.from_dict({
    "id": [1, 2, 3],
    "nested": [{"a": 1, "b": 2}, {"a": 3, "b": 4}, {"a": 5, "b": 6}],
})

# 默认格式下，访问nested列返回的是Python dict
print(nested_dataset[0]["nested"])  # {'a': 1, 'b': 2}

# 转换为numpy格式
nested_dataset.set_format("numpy")
# 这里可能报错或产生意外结果！
# numpy数组不能存储dict，所以嵌套结构会被展平或报错

一个实践建议：在格式转换之前，先用 map() 把嵌套结构展平为扁平的列（第8章会详细讲 map() 方法）。

坑3：None值处理的差异

不同框架对缺失值的处理方式不同：

# 包含None的数据
dataset = Dataset.from_dict({"value": [1.0, None, 3.0]})

# 默认格式
print(dataset[:3]["value"])  # [1.0, None, 3.0]

# numpy格式
dataset.set_format("numpy")
print(dataset[:3]["value"])
# array([ 1., nan,  3.])  # None → NaN

# torch格式
dataset.set_format("torch")
# 会报错！torch不支持NaN
# TypeError: can't convert np.ndarray of type numpy.float64
# 解决方案：先填充缺失值
dataset_clean = dataset.map(lambda x: {"value": x["value"] if x["value"] is not None else 0.0})
dataset_clean.set_format("torch")
print(dataset_clean[:3]["value"])  # tensor([1., 0., 3.])
最重要的建议：永远在格式转换之前完成数据清洗和缺失值处理。格式转换是数据管线的"最后一步"，它假设数据已经是干净、完整的。如果在格式转换之后才发现数据问题，排查难度会成倍增加。

一个真实的"踩坑"故事

讲一个看似基础但排查了整整两天的问题。

某团队的PyTorch训练脚本突然报错了，错误信息如下：

RuntimeError: expected scalar type Long but found Int

看到这个错误，团队的第一反应是数据类型的锅——label列应该是int64（LongTensor）但被读成了int32。他们检查了datasets的features，确实是 Value("int32")，于是增加了 cast_column 转为int64。但错误依然出现。

后来经过仔细排查，发现根因完全不是类型精度的问题——而是他们根本没有调用 set_format("torch")。DataLoader返回的batch中，每列数据仍然是Python list而不是torch Tensor。当PyTorch的损失函数试图在这些list上执行张量操作时，触发了类型不匹配错误。

修复方法只有一行：

# 在创建DataLoader之前
dataset.set_format("torch")  # 就这一行

dataloader = DataLoader(dataset, batch_size=32, shuffle=True)

看似简单，但团队为什么排查了两天？因为错误信息指向的是"类型不匹配"而不是"格式不正确"——他们一直在调整数据类型，从未想过问题出在格式转换这一步。这个案例告诉我们：当你在PyTorch中使用datasets时，忘记 set_format("torch") 是一个最常见的低级错误，但它的症状看起来像一个高级问题。

调试记忆法：当你的dataloader返回的数据类型不符合预期时，按顺序检查三件事：
  1. 是否调用了 set_format() 或 with_format()？
  2. features中的类型是否正确？
  3. 列名是否与 columns 参数匹配？
90%的格式转换问题出在第一项。

本章小结

格式转换是数据集工程中"最后一公里"的核心环节。这一章我们覆盖了：

  • set_format()：永久性格式转换，支持numpy/torch/tf/jax/pandas/polars
  • with_format()：临时性上下文管理器，推荐优先使用
  • to_pandas() / to_polars()：完整的DataFrame互转，Polars可零拷贝
  • to_iterable_dataset()：转换为可迭代模式，减少内存占用
  • 常见陷阱：图像通道顺序、嵌套展平、None值处理、忘记set_format
