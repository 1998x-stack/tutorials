第四章：知识库与文档解析

大家好，欢迎回来。从这一章开始，我们进入RAG系统的第一个实战环节——知识库建设。如果把RAG比喻成给LLM配一本参考书，那么文档解析就是"如何把原材料变成可查阅的书页"。这一步的质量决定了整个RAG系统的上限。正如一句老话所说：garbage in, garbage out。

很多初学者觉得文档解析很简单，不就是把PDF转成文本吗？但真正做过生产级RAG的人都知道，文档解析是整条链路中最脏最累、坑最多的环节。今天就让我们把这些坑一一填平。

一、多源文档加载

现实世界中的知识来源是极其多样的。你的用户可能有以下所有类型的文档：

  • PDF文件：最常见但也最棘手。有文本型PDF（容易提取）和扫描型PDF（需要OCR）。
  • HTML网页：需要去除导航栏、广告、页脚等噪音，提取正文内容。
  • Word/Excel/PowerPoint：办公文档三件套，格式复杂。
  • Markdown/LaTeX：技术文档常用，结构清晰但数学公式提取有挑战。
  • 数据库：结构化的表格数据，需要考虑如何转化为可检索的文本。
  • API接口：实时数据源，如天气、股票、物流信息。

针对不同类型的源，我们需要不同的解析策略和工具链。

二、文档解析库对比

目前主流文档解析工具有哪些？它们各自擅长什么？我们用一张表来说清楚：

工具/库 支持格式 优势 劣势 适用场景
PyPDF2 / pypdf PDF 轻量、快速、纯Python 对复杂布局支持差，无OCR 简单文本PDF
Unstructured PDF/HTML/Word/PPT/图片等20+ 一站式多格式、支持分区检测、表格提取 依赖多（需要安装detectron2等）、速度较慢 多格式混合的企业文档
LlamaParse PDF/PPT/Word/Excel LlamaIndex官方、使用LLM辅助解析、Markdown输出 需要API key、有调用成本 复杂排版、需要高精度Markdown的场景
PyMuPDF (fitz) PDF 速度极快、支持文本/图片/注释提取 无内置OCR 大规模PDF批处理
Tesseract OCR 图片/扫描PDF 开源OCR标准、支持多语言 对清晰度敏感、速度慢 扫描件和图片型文档
BeautifulSoup / Trafilatura HTML 专注网页正文提取、速度快 仅支持HTML 网页抓取类RAG
Docling PDF/Word/PPT IBM开源、布局分析强、支持表格+图片 新项目、生态还在成长 需要深度文档理解的场景

在实际项目中，我们通常组合使用多种工具：用Unstructured做通用解析，对特殊格式（如扫描件）用LlamaParse或OCR兜底，用Trafilatura处理网页文档。

三、一个典型的文档解析流程

下面用Unstructured库演示一个完整的文档解析流水线：

from unstructured.partition.pdf import partition_pdf
from unstructured.partition.html import partition_html
from unstructured.chunking.title import chunk_by_title

# ===== 解析PDF文档 =====
elements = partition_pdf(
    filename="公司年度报告.pdf",
    strategy="auto",          # auto/hi_res/ocr_only/fast
    extract_images_in_pdf=True,
    infer_table_structure=True,
)

# 查看解析出的元素类型
for i, el in enumerate(elements[:5]):
    print(f"[{i}] {el.category}: {el.text[:80]}...")

# 将解析后的元素按标题分块
chunks = chunk_by_title(
    elements,
    new_after_n_chars=1500,
    combine_text_under_n_chars=500,
)

# ===== 解析HTML网页 =====
html_elements = partition_html(
    url="https://example.com/knowledge-base",
    skip_headers_and_footers=True,
)

print(f"PDF文档解析出 {len(elements)} 个元素，合并为 {len(chunks)} 个块")
print(f"HTML文档解析出 {len(html_elements)} 个元素")

四、表格提取的挑战

表格是文档解析中最棘手的部分。复杂表格（合并单元格、跨页表格、无边框表格）经常被解析器破解成乱码。下面是我的真实经历：

惨痛教训：2024年，我们为一个金融客户做财报分析RAG。客户的PDF中包含大量财务报表表格——有合并单元格、表头和表尾、脚注标注。我们一开始用基础PDF解析，结果表格被拆散成碎片文字，模型把"净利润 1,234万（注：经审计调整后）"中的"审计调整"当成了单独的条目，导致回答中出现了"公司净利润可能被调整"的错误结论。后续我们改用LlamaParse配合表格专门解析，并在分块时保持表格完整性，才解决了这个问题。永远不要低估表格解析的难度。

对于表格，推荐的做法是：

  • 使用专门的表格提取工具（如Table Transformer、Camelot、Tabula）。
  • 保持表格的完整语义——不要把一个表格切到不同chunk中。
  • 对于大表格，考虑用LLM生成表格摘要，将摘要而不是原始表格存入向量库。
  • 保持对表格结构和数值的精确保留，数值错误在金融场景中是致命问题。

五、Metadata管理与版本追踪

每个文档片段（chunk）都应该携带丰富的metadata信息，这对于后面的溯源、过滤、权限管理至关重要：

chunk_metadata = {
    "source_file": "2024年度财报_v3.2.pdf",
    "source_url": "https://internal.finance/reports/2024/Q4",
    "page_range": "12-13",
    "section": "利润表",
    "last_modified": "2025-01-15T10:30:00Z",
    "version": "3.2",
    "author": "张三",
    "department": "财务部",
    "permission_level": "confidential",
    "checksum": "a1b2c3d4e5f6...",
    "embedding_model": "BAAI/bge-large-zh-v1.5",
    "chunk_strategy": "semantic_chunking_v2",
    "parent_doc_id": "doc_finance_2024_annual"
}

有了metadata，我们可以在检索时实现：

  • 权限过滤：只检索用户有权访问的文档。
  • 时间过滤："只检索2025年之后的文档"。
  • 版本管理：当文档更新时，只替换变更的chunk，避免全量重建索引。
  • 溯源：生成答案时返回来源链接和页码。
最佳实践：在文档解析阶段就做好metadata的标准化标注，远比在检索阶段再临时添加高效得多。建议将metadata以JSON形式存储在向量数据库的payload字段中，检索时通过metadata filtering来缩小搜索范围，这可以显著提升检索精度和速度。

六、知识库更新的增量策略

知识库不是"建一次就完事"的。文档会更新、会过期、会新增。增量更新策略通常包括：

  • 定时全量重建：简单粗暴，适合小规模知识库（<10万文档）。
  • 基于文档hash的增量替换：检测文件hash变化，只重新索引变更的文档。
  • 流式更新：知识源通过消息队列推送更新，系统实时处理。
  • 版本化索引：保留多版本索引，支持快速的版本回退。

本章小结

我们从最基础的文档加载开始，走过了不同格式的解析挑战，见识了表格提取的"坑"，学会了metadata管理的重要性。记住：高质量的文档解析是RAG系统的地基——地基不稳，上层再漂亮的检索策略和生成框架也白搭。下一章，我们将进入解析后的关键一步：如何把长文档切割成最适合检索的文本片段——文本分块策略。
