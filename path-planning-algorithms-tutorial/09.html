大家好！上一章我们学习了 A* 的四种实用变体。但这些变体都有一个共同假设：环境是静态的。地图一旦建好就不会变了。但在现实世界中——机器人探索未知地域、自动驾驶汽车遇到前方事故、无人机在阵风中调整航线——环境总是在变。如果每次变化都从头搜索，计算量将是灾难性的。这就是 D* Lite 登场的时刻。

为什么需要增量搜索？

想象一下这个场景：你正沿着 A* 规划的最优路径行驶，突然前方出现了一个新的障碍物——一辆抛锚的卡车。用 A* 的思维，你该怎么办？唯一的办法是在新地图上从头跑一遍 A*。如果地图很大（比如 1000x1000 的栅格），这就是 100 万个节点的重新搜索。

但仔细想想：障碍物的影响是局部的。只有障碍物附近的节点代价发生了变化，远处的节点完全不受影响。增量搜索（incremental search）的精髓就是只重新计算"受影响的部分"。

增量搜索家族中最著名的两个算法是 D*（Dynamic A*，发音为"D star"）和 D* Lite。D* 最早由 Anthony Stentz 在 1994 年提出，而 D* Lite 由 Sven Koenig 和 Maxim Likhachev 在 2002 年提出——后者更简洁、更易实现，且性能相当甚至更优。

火星传奇：D* 算法被用于 NASA 的 Mars 探测车——"勇气号"（Spirit）和"机遇号"（Opportunity）的实际导航系统中。想象一下：你是一个远在 2.25 亿公里之外的火星车，通信延迟长达 20 分钟。你不能每次遇到一个石头就停下来等着地球发指令。火星车必须在自主导航模式下实时处理未知地形——这也是"Dynamic A*"名字中"Dynamic"的真实含义。D* Lite 虽然比 D* 晚提出，但由于其更简洁的设计，后来被广泛应用于各种机器人的动态重规划中。

D* Lite 核心概念

理解 D* Lite 需要掌握三个核心概念：

g 值与 rhs 值

在 D* Lite 中，每个节点维护两个代价估计：

  • g(s) ——从起点到节点 s 的当前最优代价估计（和 A* 中的 g 相同）
  • rhs(s) ——基于父节点计算的"右侧"代价：rhs(s) = min_{s' ∈ pred(s)} (g(s') + c(s', s))。可以理解为"从起点到 s 的第二种估计"

当 g(s) == rhs(s) 时，节点被称为"一致的"（consistent）。当环境发生变化导致边代价改变时，某些节点会变得不一致——此时 D* Lite 就知道需要更新这些节点了。

Key 值

D* Lite 的优先队列使用一个两元素的 key：key(s) = [min(g(s), rhs(s)) + h(s_start, s), min(g(s), rhs(s))]。这是一个精心设计的排序机制——它保证了队列按"谁最需要被处理"排序，且能处理节点状态的动态变化。

D* Lite 算法流程

下面是简化版的 D* Lite 实现（从目标向起点搜索）：

import heapq

class DStarLite:
    def __init__(self, grid, start, goal):
        self.grid = grid
        self.start = start
        self.goal = goal
        self.g = {goal: 0}
        self.rhs = {goal: 0}
        self.U = []  # 优先队列
        self._insert(goal)

    def _key(self, s):
        k1 = min(self.g.get(s, float('inf')),
                 self.rhs.get(s, float('inf'))) + self._h(self.start, s)
        k2 = min(self.g.get(s, float('inf')),
                 self.rhs.get(s, float('inf')))
        return (k1, k2)

    def _h(self, a, b):
        # Manhattan distance or Euclidean
        return abs(a[0] - b[0]) + abs(a[1] - b[1])

    def _insert(self, s):
        heapq.heappush(self.U, (self._key(s), s))

    def compute_shortest_path(self):
        while self.U and self._top_key() < self._key(self.start) \
              or self.rhs[self.start] != self.g.get(self.start, float('inf')):
            k_old, s = heapq.heappop(self.U)
            if k_old < self._key(s):
                heapq.heappush(self.U, (self._key(s), s))
            elif self.g.get(s, float('inf')) > self.rhs.get(s, float('inf')):
                self.g[s] = self.rhs[s]
                for pred in self._predecessors(s):
                    self._update_vertex(pred)
            else:
                self.g[s] = float('inf')
                self._update_vertex(s)
                for pred in self._predecessors(s):
                    self._update_vertex(pred)

    def _update_vertex(self, u):
        if u != self.goal:
            self.rhs[u] = min(
                self.g.get(v, float('inf')) + self._cost(u, v)
                for v in self._successors(u)
            )
        self._remove(u)
        if self.g.get(u, float('inf')) != self.rhs.get(u, float('inf')):
            self._insert(u)

    def update_edge(self, u, v, new_cost):
        # 当传感器检测到边代价变化时调用
        if self._cost(u, v) != new_cost:
            self.grid[u][v] = new_cost
            self._update_vertex(u)
            self.compute_shortest_path()

这个实现的关键在于 compute_shortest_path 只会在代价变化后更新受影响的节点，而不是重新搜索全图。

D* vs D* Lite vs LPA*

这三个增量搜索算法经常被混淆，这里做一个清晰的对比：

算法 提出年份 搜索方向 核心思想 适用场景
LPA* 2001 从起点向目标 增量式地维护从起点到所有节点的最短路径 静态地图上边代价变化
D*（原版） 1994 从目标向起点 首次提出"由目标向起点搜索，再反向跟踪"的策略 机器人探索未知环境
D* Lite 2002 从目标向起点 用 LPA* 的框架重新实现了 D*，代码更简洁 机器人动态重规划（推荐）

从实践角度来说：除非你有特殊的历史遗留系统，否则请直接用 D* Lite。它只有 D* 约一半的代码量，且数学证明更严谨。我见过不少项目还在用原版 D*，仅仅是因为"20 年前就这么写的"——典型的 legacy 技术债。

场景 A*（完全重规划） D* Lite 加速比
随机障碍物（5% 栅格变化） 45 ms 3.2 ms ~14x
局部障碍物（1 个区域堵塞） 45 ms 0.8 ms ~56x
动态障碍物频繁出现 45 ms 每次 平均 1.5 ms ~30x

在实际系统中，D* Lite 并不是完美无缺的。它的主要局限在于：当环境发生大规模变化时（比如整张地图重建），增量更新的优势会消失——此时还不如从头跑 A*。因此聪明的做法是同时运行两种策略：如果变化的节点数超过某个阈值（比如总节点数的 10%），就切换到完全重规划。

说到重规划的效率问题，下一章我们将介绍另一种极致优化——JPS 跳点搜索（Jump Point Search）。它不靠增量更新，而是通过巧妙地跳过栅格地图中的"对称路径"来实现数量级的加速。如果说 D* Lite 是在时间维度上复用信息，JPS 则是在空间维度上消除冗余——两种哲学截然不同，但都能带来惊人的性能提升。
