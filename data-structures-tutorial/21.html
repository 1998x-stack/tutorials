开篇：一个"搜不到"的搜索框

2019年，我参与了一个电商平台的搜索优化项目。我们的搜索框有一个"自动补全"（Autocomplete）功能：用户在搜索框里输入"手"，下面应该弹出"手机"、"手表"、"手工饼干"等建议词。

第一版实现非常简单粗暴——每次用户输入一个字符，后端就从数据库中模糊查询：SELECT keyword FROM suggestions WHERE keyword LIKE '输入%' ORDER BY popularity DESC LIMIT 10。上线后发现，数据库CPU动不动就飙到80%，原因是用户每敲一个字就发一次SQL查询，并发一高数据库就撑不住了。

于是我们改成了内存搜索：把所有关键词加载到一个Python列表里，用户输入时遍历列表做前缀匹配。结果呢？50万个关键词，每次输入都要遍历一遍——虽然是O(n)，但n=500,000，一个输入只要几十毫秒还能接受。但产品经理说："不行，用户打字速度很快，每200ms就输入一个字，我们要在10ms内给出建议！"

于是我们引入了字典树（Trie）。用Trie存储50万个关键词，任何前缀的查找都是O(k)——k是输入字符串的长度，即使用户输入了20个字，也只需要查20次节点指针。速度从几十毫秒降到了微秒级。

这就是Trie的威力。

Trie的概念

Trie，中文叫字典树或前缀树，是一种专门用于高效存储和检索字符串的树形数据结构。它的名字"Trie"来源于"Retrieval"（检索）一词的中间部分——发音是"try"而非"tree"，但大家经常混着读。

Trie的核心思想是：利用字符串之间的公共前缀来节省存储空间和查找时间。

例如，存储"apple"、"app"、"apply"三个单词。如果用哈希表，它们各自独立占用空间。但Trie可以将它们共有的前缀"app"合并存储：

        根节点
          |
          a
          |
          p
          |
          p  ← 此处标记为单词结尾（"app"）
          |
          l
          |
          e  ← "apple" 的结尾
          |
          y  ← "apply" 的结尾

每个节点代表一个字符，从根节点到某个节点的路径就是一个前缀。如果某个节点被标记为"单词结尾"，说明从根到该节点的路径构成了一个完整的单词。

这种结构带来的好处是：任何前缀匹配的操作都不需要遍历所有单词，而只需要沿着树往下走——查找时间与字典大小无关，只与输入字符串的长度有关。

核心直觉：Trie的本质是把所有单词的公共前缀"合并"成一条路径。你可以把它想象成一个"会分叉的字典"——查找时不是一页页翻书，而是顺着字母走迷宫，走到哪算哪。每走一步就缩小了搜索范围，直到找到完整的路径。

Trie的实现

Trie最常见的实现方式是用数组（每个节点包含一个长度为26的指针数组）来存储子节点，针对的是只包含小写英文字母的字符串。对于更通用的场景，可以用哈希表来存储子节点。

class TrieNode:
    def __init__(self):
        # 用字典存储子节点，key为字符，value为TrieNode
        self.children = {}
        self.is_end = False  # 标记是否为单词结尾


class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        """插入一个单词"""
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_end = True  # 标记单词结尾

    def search(self, word):
        """搜索完整单词，返回是否存在"""
        node = self._traverse(word)
        return node is not None and node.is_end

    def starts_with(self, prefix):
        """判断是否存在以prefix开头的单词"""
        return self._traverse(prefix) is not None

    def _traverse(self, s):
        """辅助方法：沿着字符串s走，返回最后一个节点（如果路径不存在则返回None）"""
        node = self.root
        for ch in s:
            if ch not in node.children:
                return None
            node = node.children[ch]
        return node

    def autocomplete(self, prefix):
        """返回所有以prefix开头的单词（自动补全功能）"""
        node = self._traverse(prefix)
        if node is None:
            return []

        results = []
        self._dfs(node, list(prefix), results)
        return results

    def _dfs(self, node, path, results):
        """DFS遍历，收集所有从当前节点开始的完整单词"""
        if node.is_end:
            results.append(''.join(path))
        for ch, child in node.children.items():
            path.append(ch)
            self._dfs(child, path, results)
            path.pop()


# 使用示例
trie = Trie()
trie.insert("apple")
trie.insert("app")
trie.insert("apply")
trie.insert("application")
trie.insert("apt")

print(trie.search("app"))      # True
print(trie.search("apps"))     # False
print(trie.starts_with("app")) # True
print(trie.autocomplete("ap")) # ['app', 'apple', 'apply', 'application', 'apt']
print(trie.autocomplete("app")) # ['app', 'apple', 'apply', 'application']

注意上面代码中的autocomplete方法——它先用_traverse走到前缀路径的末尾，然后做一次DFS来收集所有子路径上的完整单词。这个思路非常重要：Trie + DFS = 自动补全，这是一个天然的组合。

Trie的应用

应用一：自动补全（Autocomplete）

我们开头讲的搜索框自动补全就是Trie最经典的应用。无论是Google搜索、IDE的代码补全、还是手机输入法的联想词，底层都离不开Trie或它的变体。

在工程实现中，通常会在Trie节点上存储一些额外的信息，比如热度值（该前缀被搜索的频率），这样在autocomplete时可以优先返回最热门的前几个建议。

应用二：拼写检查

当用户输入了一个单词，我们检查Trie中是否存在这个单词。如果不存在，可以基于Trie提供"你是不想要找……"的纠错建议。常用的做法是编辑距离（Edit Distance）+ Trie剪枝：在Trie上DFS时，计算当前路径对应的单词与用户输入的编辑距离，如果距离超过阈值则剪枝。

这看起来可能有点复杂，但核心思想很简单：Trie限制了搜索空间——你不需要和字典里的每一个单词去计算编辑距离，只需要沿着Trie探索可能的路径即可。

应用三：IP路由最长前缀匹配

在计算机网络中，路由器需要根据IP地址决定将数据包转发到哪个下一跳。路由表中的每条路由规则都有一个子网掩码，路由器需要找到与目标IP地址匹配且前缀最长的那条规则。

这本质上就是Trie上的最长前缀匹配（Longest Prefix Match）。将二进制形式的IP地址插入Trie（每个节点只有0和1两个分支），然后沿着目标IP的二进制表示在Trie上走，记录下最后一个匹配的路由节点即可。这种Trie在网络上被称为Patricia Trie或Radix Tree。

工程启示：Trie在搜索引擎、路由器、输入法中无处不在。它的核心价值是"用空间换时间"——虽然每个节点都需要存储指向子节点的指针（可能很占内存），但换来的是O(k)的前缀查询速度，其中k远小于字典大小n。在n很大的场景下，这个交换非常值得。

异或最大值（LeetCode 421）

这是Trie的一道经典"非字符串"题目：给定一个整数数组，找出两个数异或（XOR）的最大值。

暴力解法：双重循环枚举所有数对，计算异或值，O(n²)。

优化思路：把每个数的二进制表示看作一个"字符串"，插入到一个二进制Trie中（每个节点只有0和1两个分支）。然后对于每个数，在Trie上尽量走与当前位相反的路径——因为异或运算中，0⊕1=1，所以如果当前位是0，我们想去1的那一侧（这样异或结果当前位为1）。如果1的那一侧不存在，再走0的那一侧。

class BinaryTrieNode:
    def __init__(self):
        self.children = [None, None]  # 0 和 1


class BinaryTrie:
    def __init__(self):
        self.root = BinaryTrieNode()

    def insert(self, num):
        """将一个整数的二进制表示插入Trie"""
        node = self.root
        # 从最高位开始，假设整数范围是 0 到 2^31-1
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            if node.children[bit] is None:
                node.children[bit] = BinaryTrieNode()
            node = node.children[bit]

    def max_xor(self, num):
        """在Trie中查找与num异或最大的数，返回异或值"""
        node = self.root
        xor_val = 0
        for i in range(31, -1, -1):
            bit = (num >> i) & 1
            # 尽可能走与当前位相反的方向
            opposite = 1 - bit
            if node.children[opposite] is not None:
                xor_val |= (1 << i)  # 这一位异或结果为1
                node = node.children[opposite]
            else:
                node = node.children[bit]  # 只能走相同方向
        return xor_val


def find_max_xor(nums):
    trie = BinaryTrie()
    max_val = 0
    # 先插入第一个数
    trie.insert(nums[0])
    for num in nums[1:]:
        # 对每个数，查询它和已插入的数的最大异或值
        max_val = max(max_val, trie.max_xor(num))
        trie.insert(num)
    return max_val


# 测试
nums = [3, 10, 5, 25, 2, 8]
print(find_max_xor(nums))  # 输出 28，因为 5(101) ⊕ 25(11001) = 28(11100)

这个解法的核心是贪心地在Trie上走"相反位"，确保高位尽可能为1。因为高位为1的权重远大于低位，所以这种贪心策略是正确的。时间复杂度O(32n) = O(n)，空间复杂度O(32n)。

思维拓展："异或最大值"这个问题展示了Trie的一个非常重要的通用思路——任何可以表示为"前缀"或"路径"的"匹配"问题，都可以考虑用Trie来优化。把数字的二进制表示看作路径，Trie就能用来优化"按位匹配"的问题。

AC自动机入门

在Trie的基础上，加上KMP的失配指针（Failure Link）思想，就得到了AC自动机（Aho-Corasick Automaton）。这是多模式串匹配的经典算法：给定一个文本串和多个模式串，在O(n + m)时间内找出所有模式串在文本串中的出现位置。

AC自动机的核心思想：

  • 先用所有模式串建立一棵Trie。
  • 然后为Trie中的每个节点计算一个失配指针（fail指针）——指向当前字符串的"最长后缀"在Trie中对应的节点。这和KMP中next数组的思路一模一样。
  • 匹配时，在Trie上沿着文本串的字符走。如果当前节点没有对应的子节点，就沿着fail指针回溯，直到找到匹配或回到根节点。

AC自动机最常见的应用是敏感词过滤：

# 简化的AC自动机思路示意（非完整实现）
# 假设我们要在文本中查找 ["abc", "bcd", "cde"] 这三个模式串

# Step 1: 建立Trie
#    根
#   /  |  \
#  a   b   c
#  |   |   |
#  b   c   d
#  |   |   |
#  c★  d★  e★
# （★标记单词结尾）

# Step 2: 计算fail指针
# - 第一层节点（a,b,c）的fail指向根
# - 节点"ab"的fail指向"b"（最长后缀"b"在Trie中有）
# - 节点"bc"的fail指向"c"
# - 以此类推

# Step 3: 匹配时，沿着文本串走向时，如果遇到失配就用fail指针回溯

AC自动机能在一遍扫描中找出所有模式串，时间复杂度O(n + m + occurrences)，空间复杂度O(m * 字符集大小)。对于大规模的多模式串匹配场景——比如文章查重、网页敏感内容检测——AC自动机是首选方案。

踩坑实录：之前在一个弹幕网站的敏感词过滤系统里，第一版用的是简单的"遍历所有敏感词 + in判断"，敏感词有10万个，每条弹幕都要遍历10万次——CPU直接打满。换成AC自动机后，建立好自动机后每条弹幕只需要一次O(弹幕长度)的扫描，CPU占用从95%降到了8%。多模式串匹配不要用暴力，AC自动机是标准答案。

哈希表 vs Trie

维度 哈希表 Trie
完整单词查找 O(1) 平均 O(k)（k为单词长度）
前缀匹配 O(n) — 需要遍历所有key O(k) — 顺着前缀路径走即可
自动补全 需要额外索引 天然支持（Trie + DFS）
空间占用 O(n) — 存储所有key O(n * k) — 有公共前缀时更省
有序遍历 不支持（无序） 支持（按字母序DFS）
删除操作 O(1) 平均 O(k) — 需要清理空节点

从上表可以看出，哈希表在查找完整单词的场景下更快（O(1) vs O(k)），但Trie在前缀匹配的场景下完胜（O(k) vs O(n)）。Trie的另一个优势是有序性——可以按字母序遍历所有单词，而哈希表是无序的。

选择指南：
  • 如果只做完全匹配（判断某个单词是否存在），哈希表更好。比如"用户输入用户名，判断是否已注册"。
  • 如果需要前缀匹配或自动补全，Trie更好。比如"用户输入'ap'，推荐所有以ap开头的单词"。
  • 如果两者都需要，可以考虑Trie + 哈希表的混合方案：Trie负责前缀匹配，每个Trie节点上挂一个"热词哈希表"用于排序。

本章小结

在这一章中，我们学习了：

  • Trie的基本概念——一种基于前缀合并的树形字符串检索结构，每个节点代表一个字符。
  • Trie的三大操作——insert（插入）、search（搜索完整单词）、startsWith（前缀匹配），均为O(k)时间复杂度。
  • Trie的经典应用——自动补全（搜索框/IDE/输入法）、拼写检查（编辑距离+Trie剪枝）、IP路由最长前缀匹配（二进制Trie）。
  • 异或最大值——LeetCode 421，将整数转为二进制Trie，贪心走相反位，O(n)时间复杂度解决O(n²)的问题。
  • AC自动机入门——Trie + KMP失配指针 = 多模式串匹配，一次扫描找出所有模式串。

Trie是一个"虽然简单但极其实用"的数据结构。它的核心思想——通过合并公共前缀来加速前缀查询——在计算机科学的方方面面都能找到影子：从搜索引擎的倒排索引到路由器的CIDR匹配，从编译器前端的关键字识别到基因序列的匹配算法。
