第1章 AI Coding Agent 前世今生

大家好，我是你们的讲师。在开始动手拆代码之前，我想先讲个小故事。

2022 年初，我所在的团队接到了一个紧急任务：在一个庞大且文档几乎为零的 Java 遗留系统中找到某个内存泄漏的根源。彼时 GitHub Copilot 刚刚公开预览，大家的期待是它能"自动补全"出一些代码片段。没人想到——包括我自己在内——我会在一个深夜的调试 session 中，无意间在注释里写了半行 // this function might leak because，然后 Tab 键补全出了下面这段话："the FileInputStream is never closed in the exception path at line 173"。我打开第 173 行，沉默了——确实有一个 try 块里的 FileInputStream 在异常分支上没有被关闭。那一刻我意识到，自动补全不再是"猜你下一个字符"，它已经在读你的代码、理解你的意图、甚至看到了你没看到的东西。

四年后的今天，我们站在了一个完全不同的时代。Claude Code、Codex CLI、Gemini CLI 等工具已经不再满足于"补全代码"，而是能够自主地规划任务、读写文件、执行命令、调试错误、甚至完成整个 Pull Request 的编写和提交。这场从"自动补全"到"自主编程"的进化，正是本章要追溯的主线。理解这条演进脉络，你才能真正理解 Claude Code 为什么被设计成今天这个样子——以及为什么我们说，Agent 的 Harness 比模型本身更重要。

1.1 三代 AI 编程工具演进

回顾 2021 到 2026 这短短五年，AI 编程工具经历了三次范式跃迁。每次跃迁都不是简单的功能叠加，而是对"人机协作关系"的一次重新定义。

第一代（2021-2022）：补全时代的 "Tab 键魔法"

2021 年 6 月，GitHub Copilot 以 VS Code 插件的形式横空出世。它的核心交互极其简单：你写代码，它猜测你的意图，按 Tab 接受建议。背后的模型是 OpenAI 的 Codex（GPT-3 的代码微调版），上下文窗口只有区区 2K 到 4K tokens。这意味着它只能"看到"当前文件的一部分，跨文件理解几乎不可能。

但这依然足够震撼。对于样板代码（boilerplate）、单元测试、正则表达式、CRUD 接口这类模式化任务，Copilot 的效率提升是 2-3 倍。一时间，"10x 工程师"的说法甚嚣尘上。

不过第一代的局限性也很快暴露出来：没有对话能力、没有文件编辑能力、没有上下文记忆。它是个哑巴工具——只能在你写的时候给建议，你不能跟它说"把这个 for 循环改成 stream"然后期待它做到。

第二代（2023-2024）：聊天的时代——Chat + Edit

ChatGPT 的出现改变了游戏规则。2023 年，Cursor 率先将聊天界面与代码编辑器深度融合，开创了"对话式编程"范式。你不再需要把代码粘贴到网页上问 ChatGPT——你可以在编辑器里直接圈选代码，然后输入"把这个函数改成异步的"，模型会理解上下文并生成 diff。Cline（原 Continue）和 Aider 紧随其后，引入了更完善的编辑协议。

这一代的关键突破有两个：多文件感知（模型可以同时看到项目中多个相关文件）和 应用编辑（模型可以直接修改你的代码文件，而不是让你手动复制粘贴）。上下文窗口也从 4K 跃升到 128K（GPT-4 Turbo），这使得理解整个中等规模项目成为可能。

但第二代仍然有一个根本性的局限：模型是被动的。它不会主动去运行你的代码、检查测试结果、发现 bug 然后修复。所有的"行动"仍然需要人类发出指令，模型只是"回答"和"编辑"。这种模式下，人仍然是计划者，模型只是执行打字工作的打字员。

踩坑实录：我曾经用第二代工具重构一个 Python 项目的数据库层。Aider 非常完美地生成了所有 ORM 映射代码，但它不会主动去 pip install 缺失的依赖、不会运行迁移脚本来验证表结构是否正确、更不会检查 SQLAlchemy 版本兼容性。结果是我花了一个小时修它"看不见"的环境和集成问题。这让我意识到——不主动行动的 Agent，只是一个高级的语法补全器。

第三代（2024-2026）：自治时代——Agent 元年

2024 年底到 2025 年初，一个质变发生了。Anthropic 发布了 Claude Code（最初叫 Claude CLI），OpenAI 紧随其后推出了 Codex CLI，Google 也拿出了 Gemini CLI。这些工具的共同特征是：它们不再是 IDE 插件，而是一个运行在终端的自主 Agent。

第三代工具的核心能力是什么？Tool Use。模型不再只是"说"和"写"，它被赋予了工具：读文件、写文件、执行 Shell 命令、搜索网络、发起 Git 操作、调用 API。模型自己决定"下一步做什么"——它看到编译错误，就自己读日志；找到 bug，就自己编辑文件；改完了，就自己跑测试验证。整个过程你只需要说一句："帮我修好这个 bug。"

这才是真正意义上的"Agent"。不再是人在循环中（Human-in-the-Loop），而是人在监督上（Human-on-the-Loop）。

产品 发布年份 范式 核心能力 上下文窗口 工具数量
GitHub Copilot 2021 Inline Completion Tab 补全 2K-4K 0
Cursor 2023 Chat + Edit 对话编辑、多文件感知 128K 0 (编辑协议)
Aider 2023 Chat + Edit Git-aware 编辑、多文件重构 128K 1 (编辑)
Claude Code 2025 Autonomous Agent Tool orchestration, Sub-Agent, MCP 200K 10+
Codex CLI 2025 Autonomous Agent Sandboxed execution, file editing 128K 5+
Gemini CLI 2025 Autonomous Agent Deep project analysis, tool use 200K 8+

1.2 Agent 范式大观

说完了三代产品，我们再从架构层面审视：这些 Agent 背后到底遵循了什么样的设计范式？目前业界主流的 Agent 架构可以归纳为三种——理解它们各自的优劣，是设计你自己的 Agent 的前提。

ReAct（Reasoning + Acting）

ReAct 是当前最广泛使用的 Agent 范式，也是 Claude Code 的核心架构。它的名字暗示了核心思想：思考与行动交错进行。模型输出一段推理（Thought），然后执行一个动作（Action），观察结果（Observation），再继续推理。这个循环一直持续到任务完成。

这种方式的优势是灵活——Agent 可以在任何时候根据新观察到的信息调整计划。比如它查到一个文件不存在，就可以立即切换策略去搜索其他文件，而不是死板地按原定计划执行。

Debater's Corner：批评者认为 ReAct 是"边想边做"而不是"想好了再做"，容易在复杂任务中陷入局部最优，反复纠结于细节而非全局。但在 Coding Agent 场景中，ReAct 的灵活应变能力远远超过它的缺陷——毕竟代码世界的意外太多了。

一个简化的 ReAct Loop 在代码层面看起来是这样的：

# 最简 ReAct Agent Loop 示意
def agent_loop(task):
    messages = [{"role": "user", "content": task}]
    while not task_complete(messages):
        response = model.generate(messages)            # 模型输出 Thought + Action
        tool_name, args = parse_tool_call(response)     # 解析工具调用
        result = execute_tool(tool_name, args)          # 执行工具
        messages.append(response)                       # 追加思考到上下文
        messages.append({"role": "tool",                 # 追加观察结果
                         "content": result})
    return final_answer(messages)

Plan-Execute

Plan-Execute 范式由两步组成：先制定一个完整的计划，然后按顺序执行。2023 年的 AutoGPT 是这一范式的代表性尝试——它会先生成一个包含多个步骤的任务列表，然后逐个执行。这种方式的好处是可预测性强、便于人类审核计划后再放行。

但缺点也很明显：现实世界的任务很少能精确预测每一步。你计划"打开文件 X，找到函数 Y，修改 Z"，但打开文件后发现结构和你想象的不一样——整个计划就要推翻重来。AutoGPT 在早期演示中看起来很酷，但在实际工程任务中很快就暴露出僵化的问题。

Multi-Agent

第三种范式是多 Agent 协作，代表包括 MetaGPT 和 ChatDev。这种架构将软件开发流程角色化——有产品经理 Agent、架构师 Agent、开发者 Agent、测试 Agent——它们各自扮演一个角色，通过对话协作完成项目。

Multi-Agent 的优势是分工明确，适合大型项目的规划和并行开发。但代价是通信开销巨大——Agent 之间交换的 token 量可能比实际代码还多，而且协调多个 Agent 的上下文一致性是个难题。

务实建议：对于个人开发者或小团队来说，Multi-Agent 架构往往是过度设计。单 Agent + 优秀的 Tool Use 系统（即 Claude Code 的路线）在绝大多数场景下效率更高。只有在需要明确分离职责的大型项目中，Multi-Agent 才有其用武之地。

1.3 为什么 Harness 是新的护城河

2026 年初，Arize AI 发表了一份引起广泛讨论的分析报告。他们的核心发现看似简单，却从根本上改变了我们对 AI Coding Agent 的理解。

报告拆解了一个典型的 Coding Agent 代码库，发现：调用模型 API 的逻辑大约只有 50 行代码，而围绕它构建的整套基础设施——权限管理、上下文压缩、工具调度、错误恢复、任务编排——超过了 500,000 行。这 50 行代码是"模型调用"，剩下的 50 万行就是 Harness。

Arize 团队还做了一个有趣的实验：他们把同一个 Harness 分别接入三个不同的模型——Claude Sonnet 4、GPT-4o、Gemma 4——然后测试它们在标准 coding benchmark 上的表现。结果发现，使用同一个 Harness 时，三个模型的差距缩小到 5% 以内。更惊人的是，同一个模型在不同 Harness 上的差距却可以超过 30%。

这说明什么？说明在 Agent 系统中，模型是 commodity，Harness 是 moat。当你抱怨"这个模型写代码不行"时，问题可能 80% 出在 Harness 上——上下文没有管理好、工具调用协议设计不佳、错误恢复不完善，这些都会让一个优秀的模型看起来像个新手。

具体来说，Harness 的设计有几个关键维度，我们称之为"三种完成模式"：

完成模式 核心策略 平均 Token 消耗 假完成捕获率 任务正确率
Explicit Finish 模型显式声明"我完成了" 低（~2K） ~35% 假阴性 ~62%
Implicit Finish Harness 自动检测完成状态 中（~8K） ~8% 假阴性 ~78%
Adaptive Finish 混合策略 + 中间验证 较高（~15K） ~2% 假阴性 ~91%

Explicit Finish 是大部分简单 Agent 实现的方式——模型调用完工具后说一句"I'm done"，Agent 就停了。这在简单任务上效率很高，但模型经常"自以为做完了"而实际上任务并没有正确完成。Implicit Finish 增加了 Harness 层面的检测逻辑，比如检查文件是否确实被修改、测试是否真的通过。Adaptive Finish 则引入了中间验证节点，在关键步骤后插入检查点，确保任务质量。

Claude Code 采用的是接近 Adaptive Finish 的混合策略，这也是它能够稳定处理数十分钟甚至数小时的长任务而不"幻觉性提前交卷"的关键原因。

1.4 Claude Code 在这个版图中的位置

说了这么多背景，我们把目光聚焦到本课程的主角——Claude Code。

Claude Code 于 2025 年 2 月由 Anthropic 以 Claude CLI 的名字首次发布，最初只是一个运行在终端的实验性工具。谁也没想到，它在短短几个月内引发了开发者社区的狂热。不仅仅是它用起来爽，更因为它第一次证明了"Vibe Coding"模式可以生产级落地——你可以用自然语言描述一个需求，然后让 Agent 自己完成从代码编写、依赖安装、测试验证到 Git 提交的完整流程，而你只需要喝着咖啡偶尔看一眼进度。

Claude Code 在技术上有几个值得铭记的里程碑：

  • 率先开源 MCP 协议（Model Context Protocol），将工具集成从厂商锁定中解放出来，现在已经成长为行业标准，被 JetBrains、VS Code、Sourcegraph 等主流工具采纳。
  • 12 层渐进式 Harness 架构，从底层的 Bash 沙箱到顶层的任务编排，每层都提供了可插拔的设计。这是目前公开资料中最为完整的 Coding Agent 架构。
  • 2025 年中期，Claude Code 的部分源代码意外流出（业内俗称"Leak-gate"），这反而催生了前所未有的社区学习浪潮。开发者们第一次得以窥见一个生产级 Agent 的内部运作——从 Token 计费策略到上下文压缩算法，从 Sub-Agent 调度到 MCP 握手协议。这个课程中的很多分析，也受益于那次泄漏带来的开源社区逆向工程成果。

当然，Claude Code 并非完美。它也有自己的短板——对大项目的上下文管理仍然有改进空间、Sub-Agent 调度在某些复杂场景下效率不高、权限系统的颗粒度还可以更细致。但瑕不掩瑜，它无疑是目前我们能够学习到的最完整的 Coding Agent 范本。

课程定位说明：本课程不是 Claude Code 的用户手册，而是它的建筑蓝图。我们会逐层拆解 Claude Code 的 Harness 架构，理解每一个设计决策背后的权衡，然后用 Python 从零搭建一个功能完整的 AI Coding Agent。到第 27 章，你将拥有一个自己的 Agent——它可能不如 Claude Code 健壮，但它的每一行代码你都懂。
