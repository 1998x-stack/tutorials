第27章 DeepSeek-V4

2026 年 4 月 24 日，DeepSeek 发布了 V4 预览版。消息一出，中文互联网上立刻炸开了锅——不仅仅是因为它 1.6 万亿参数的规模，更因为它的技术路线上写满了"叛逆"：放弃了已经极其成熟的 MLA 注意力机制，换上了全新的 CSA+HCA 混合注意力；放弃了业界标配的 AdamW 优化器，改用了尚在探索期的 Muon；放弃了传统的 FP8 推理，直接压到 FP4。每一项决策单独拿出来都像是在"赌"，但 V4 把它们全部赌赢了。

这一章，我们来深度拆解 DeepSeek-V4 的每一项技术突破——是什么让团队敢于在 V3 系列大获成功后，几乎推倒重来地设计一套全新架构。

27.1 V4 全景：Pro 与 Flash 双变体

V4 发布时提供了两种变体，覆盖了"极致性能"和"极致效率"两个极端：

规格 V4-Pro（旗舰） V4-Flash（高效）
总参数量 1.6 万亿（1.6T） 2,840 亿（284B）
激活参数量 490 亿（49B） 130 亿（13B）
层数 61 43
隐层维度 7,168 4,096
路由专家数 1 共享 + 384 路由 1 共享 + 256 路由
训练 token 数 33+ 万亿 32+ 万亿
上下文窗口 100 万 token（默认） 100 万 token（默认）
最大输出 384K token 384K token
推理模式 Non-Think / Think High / Think Max Non-Think / Think High / Think Max
API 价格（每百万 token） $1.74 输入 / $3.48 输出 $0.14 输入 / $0.28 输出

注意一个关键数字：V4-Flash 的 API 价格只有 Pro 的十二分之一，但在简单 Agent 任务上表现与 Pro 相差无几。这意味着对于大多数实际应用场景，Flash 已经足够——只有最极致的推理任务才需要调用 Pro。这种"分级服务"的策略，与 OpenAI 的 GPT-4o / GPT-4.5 分层思路类似，但 DeepSeek 的价差更大、选择更灵活。

战略意义：V4-Flash 只需要 2-4 张 H100 即可部署推理，而 V4-Pro 需要一个完整的集群。Flash 的 284B/13B 配置恰好卡在了"单机可部署"的临界点上——这对中小型团队来说是一个巨大的利好。

在评测方面，V4-Pro 的成绩同样令人印象深刻：MMLU-Pro 达到 87.5，LiveCodeBench Pass@1 达到 93.5，Codeforces Rating 达到 3,206（超越 GPT-5.4 的 3,168），GPQA Diamond 达到 90.1，SWE-bench Verified 达到 80.6%。这组数据表明，V4-Pro 在编程和推理能力上已经与全球最顶尖的闭源模型站在了同一水平线上。

27.2 CSA+HCA 混合注意力

V4 最令人意外的架构决策，就是放弃了在 V2/V3 系列中表现极为出色的 MLA（Multi-head Latent Attention），取而代之的是一套全新的混合注意力机制：CSA（Compressed Sparse Attention）+ HCA（Heavily Compressed Attention）。

为什么要放弃 MLA？答案很简单：MLA 在 128K 上下文内效率出色，但在百万 token 级别力不从心。MLA 通过低秩压缩减少了 KV Cache，但注意力计算的复杂度仍然是 O(L²)。当 L 达到 100 万时，即使是 MLA 的 KV Cache 用量也会变得难以承受。

CSA 的核心创新在于"压缩加稀疏"的双重降维：

  • 压缩（Compression）——将每 m=4 个相邻的 KV token 压缩成一个"块表示"，然后在这些块级别上执行注意力。这相当于在时间维度上做了一次 4 倍的降采样。
  • 稀疏（Sparse）——用一个闪电索引器（Lightning Indexer）为每个查询打分，只保留分数最高的 1,024 个压缩块。加上一个大小为 128 的局部滑动窗口，确保细粒度的局部信息不丢失。
  • 重叠压缩——相邻两个压缩窗口有重叠，防止信息在块边界处断裂。

HCA 则是一种更加激进的全局压缩通道：它把每 128 个 token 压缩成一个条目，不做稀疏选择，而是对所有压缩条目执行密集注意力。在 100 万 token 上下文中，HCA 只需处理大约 8,000 个压缩条目——计算量可以忽略不计，但提供了一条全局信息通路。

CSA 计算流程示意:

输入序列 (L=1M tokens)
    │
    ├── CSA 路径:
    │   窗口压缩 (m=4, 重叠50%) → 约 500K 压缩块
    │   Lightning Indexer 打分 → Top-1024 选择
    │   在 1024 个块上执行注意力 + 128 个滑动窗口 token
    │   输出: 约 1,152 个 token 级别的注意力
    │
    ├── HCA 路径:
    │   全局压缩 (m'=128) → 约 8K 压缩条目
    │   密集注意力 → 全局信号
    │
    └── 拼接 → 最终注意力输出

效果是惊人的。在 100 万 token 上下文中，V4-Pro 的推理 FLOPs 只有 V3.2 的 27%，KV Cache 用量只有 V3.2 的 10%。V4-Flash 更是将这两个数字压到了 10% 和 7%。换句话说，如果你在 V3.2 上跑一个 100 万 token 的推理请求需要 100 块钱的电费，V4-Flash 只需要 10 块钱。

技术细节：V4 还引入了 Partial RoPE（仅对最后 64 维施加位置编码）、Attention Sink（每个注意力头设置一个专门的"注意力汇聚"token）、以及混合精度 KV Cache（BF16 用于 RoPE 维度，FP8 用于其余维度）等辅助技术，共同构成了百万 token 上下文的全栈优化方案。

27.3 mHC 流形约束超连接

在大模型中，残差连接（Residual Connection）是最基础也是最容易被忽视的组件。它让梯度可以顺利地从输出层"绕过"中间层直接流回输入层，解决了深层网络训练中的梯度消失问题。但传统的残差连接有一个根本性的限制——它只是简单的"加和"操作，缺乏对信息流的精细控制。

DeepSeek 之前的研究中提出过 Hyper-Connections（HC），通过可学习的投影矩阵来替代简单的残差加和。但在实验中发现，当模型深度增加到一定程度时，Plain HC 的数值训练变得不稳定——投影矩阵的谱范数可能远大于 1，导致信号在前向传播中不断放大，最终爆炸。

V4 中提出的 mHC（Manifold-Constrained Hyper-Connections） 解决了这个问题。关键创新是将残差变换矩阵 B_l 约束到一个特殊的数学结构上——Birkhoff 多面体（双随机矩阵，即所有元素非负且每行每列之和均为 1）。

这个约束通过 Sinkhorn-Knopp 迭代（20 步）实现。双随机矩阵的谱范数被严格限制为 1，意味着该变换是"非扩张的"——信息既不会被无端放大，也不会被无端缩小，而是被稳定地传递。再加上将残差流从 d 维"拓宽"到 n_hc × d 维（n_hc=4，相当于 4 条并行通道），模型获得了更强的信息表达能力。

残差方法 参数量 训练稳定性 信息容量 V4 是否采用
标准残差连接 0（无参数） 优秀 低（单通道） 否
Pre-LayerNorm 2d 良好 低 否
Hyper-Connections 3d² 不稳定（深层） 高 否
mHC（本方案） 3n_hc d² 优秀（有界谱范数） 高（4 通道） 是

在 6,144 序列长度的实验中，标准残差连接在 60 层时梯度回传后的信号衰减到初始值的 30% 左右，而 mHC 在 61 层 V4-Pro 上仍然保持了 85% 以上的信号强度。这正是 V4 能够稳定训练 1.6T 参数模型的关键原因之一。

27.4 Muon 优化器

自从 Adam 优化器在 2014 年提出以来，它几乎成了深度学习训练的事实标准。AdamW 更是在大模型训练中占据了统治地位。V4 打破了这一惯例，首次将 Muon 优化器 部署到万亿参数级别的 MoE 模型训练中。

Muon 的核心思想与 AdamW 截然不同。AdamW 对每个参数计算一阶矩（动量）和二阶矩（自适应学习率），然后进行逐元素的参数更新。Muon 则将参数的梯度矩阵化，通过 正交化 操作来更新参数——具体来说，它使用 Newton-Schulz 迭代将动量矩阵"拉"到最接近的正交矩阵。

V4 中实现的 Muon 采用了"混血" Newton-Schulz 策略：8 步激进迭代（系数: 3.4445, −4.7750, 2.0315）+ 2 步保守迭代（系数: 2, −1.5, 0.5），确保奇异值被稳定地收敛到 1。此外还搭配了 Nesterov 动量、权重衰减和 RMS 重新缩放。

但工程上的挑战比算法本身更大。Muon 需要完整的梯度矩阵才能做正交化，这与 ZeRO 优化器将参数分片到不同设备的策略直接冲突。DeepSeek 的工程师用了一种巧妙的方法解决：基于背包问题的桶分配算法 + MoE 参数展平，将额外开销控制在 10% 以下。跨数据并行 ranks 的梯度通信使用 BF16 随机舍入 + all-to-all + 本地 FP32 累加。

Muon vs AdamW 对比:

AdamW 参数更新:
  m_t = β₁·m_{t-1} + (1-β₁)·g_t
  v_t = β₂·v_{t-1} + (1-β₂)·g_t²
  θ_t = θ_{t-1} - η·m_t/(√v_t + ε)

Muon 参数更新:
  M_t = β·M_{t-1} + (1-β)·g_t         // 动量累积
  O_t = NewtonSchulz(M_t)              // 正交化!
  θ_t = θ_{t-1} - η·O_t                // 直接用正交矩阵更新

实验表明，在相同训练 token 数下，Muon 的收敛速度比 AdamW 快约 15%-20%。这意味着要么在同样的算力下训练出更好的模型，要么用更少的算力达到同样的质量。对于 V4 的训练规模而言，这 15% 的效率提升对应着数千万美元的算力节省。

适用性注意：Muon 并非对所有参数类型都有效。V4 中，嵌入层、预测头、mHC 的偏置/门控、以及 RMSNorm 仍然使用 AdamW。只有注意力权重和 FFN 权重等"矩阵型"参数才使用 Muon。如果你的项目想尝试 Muon，注意区分参数类型。

27.5 FP4 QAT

FP8 已经让大模型训练和推理的效率提升了一个台阶。V4 在此基础上再进一步，推出了业界首个超大规模 MoE 模型的 FP4 量化感知训练（QAT）方案。

FP4 采用 E2M1 格式（1 个符号位、2 个指数位、1 个尾数位），每组仅 16 个可表示的值。听起来少得可怜——但通过细粒度的分组缩放因子（group size），FP4 可以在大多数场景下保持与 BF16 接近的精度。

QAT 的关键在于"让模型学会适应低精度"：

  • 混合精度主权重：优化器维护 BF16 主副本，但前向计算使用 FP4 量化后的权重。反向传播的梯度也基于 FP4 权重计算。
  • 分组缩放：注意力权重矩阵使用 group size 16，FFN 权重使用 group size 32——这背后的直觉是 FFN 权重对量化的鲁棒性更高，可以使用更大的分组。
  • 分阶段调度：训练从高精度开始预热，待权重分布稳定后再逐步过渡到全 FP4。MoE 专家权重先量化，注意力权重后量化。

最终效果：FP4 QAT 将内存占用相比 FP8 降低约 2 倍、相比 FP16 降低约 4 倍。在编码任务上与 BF16 基线的精度差距约 1%，数学推理差距约 1-2%——对于绝大多数实际应用来说完全可以接受。更妙的是，V4 也是全球首个深度适配华为昇腾 950 芯片 FP4 推理能力的模型。

战略意义：昇腾 950PR 是国内唯一支持 FP4 低精度推理的商用芯片，单卡 FP4 算力达到 1.56 PFLOPs，单卡算力是 H20 的 2.87 倍。DeepSeek 对昇腾的深度适配，不仅仅是技术选择，更是供应链安全的战略布局。

27.6 1M 默认上下文

V4 的默认上下文窗口从 V3 的 128K 跃升到了 100 万 token。这不仅仅是数字的翻倍——百万 token 上下文打开了全新的应用场景：

  • 可以直接将整个代码仓库（比如 Linux 内核的某个子系统）输入给模型做分析
  • 可以一次性处理数千页的技术文档或法律合同
  • 可以在一个会话中完成数小时的复杂多轮 Agent 任务而无需频繁清理上下文

百万 token 的训练也并非一步到位。DeepSeek 采用了一种渐进式上下文扩展策略：从 4K 开始训练，逐步扩展到 16K → 64K → 1M。而且只有在 64K 以上的阶段才引入 CSA/HCA 的稀疏化——因为团队发现，在短上下文下稀疏化反而会带来不必要的精度损失。

27.7 华为昇腾 950 支持

V4 的另一个里程碑意义在于它是首个深度适配国产芯片的万亿参数级开源模型。华为昇腾 950PR 在 V4 发布当日即实现 Day 0 适配：

适配项 细节
底层算子 基于昇腾 CANN 体系开发定制化融合 Kernel
MoE 异构并行 重新完成专家分片调度和负载均衡调优
长上下文适配 定制 KV Cache 滑窗策略和缓存压缩算法
TileLang 编译 跨硬件平台编译 DSL，降低迁移成本
MegaMoE2 融合内核 DeepSeek 自研，在昇腾平台解决 MoE 通信瓶颈

尽管目前 V4-Pro 的推理服务"受限于高端算力，吞吐量有限"，DeepSeek 在官方公告中明确表示 "预计下半年昇腾 950 超节点批量上市后价格将大幅下调"。在中国 AI 芯片出口管制持续收紧的大背景下，V4 + 昇腾的软硬一体化方案，具有深远的战略意义。

27.8 MIT 开源

最后也是最值得注意的一点：V4 延续了 DeepSeek 的开源传统，以 MIT 许可证 发布模型权重。这意味着：

  • 任何人可以自由下载、使用、修改模型
  • 商用无需授权，无需标注出处
  • 可以基于 V4 开发自己的微调版本、应用、或服务

在一个越来越多玩家走向闭源的时代——GPT-5 闭源、Claude 系列闭源、Gemini 闭源——DeepSeek 选择在 1.6T 参数这个量级上继续开源，本身就是一种鲜明的表态。而 V4 的代码库（包括推理适配和部分训练框架）也已经同步公开在 GitHub 和 Hugging Face 上。

快速上手 V4-Flash： 
pip install transformers
from transformers import AutoModelForCausalLM, AutoTokenizer
model = AutoModelForCausalLM.from_pretrained(
    "deepseek-ai/DeepSeek-V4-Flash",
    trust_remote_code=True,
    device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained("deepseek-ai/DeepSeek-V4-Flash")
prompt = "用 Python 实现一个快速排序算法"
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
outputs = model.generate(**inputs, max_new_tokens=1024)
print(tokenizer.decode(outputs[0]))
只需 2-4 张 H100 就能跑起 V4-Flash 推理！

V4 是一个承前启后的版本——它继承了 V3 系列"效率优先"的基因，但在注意力机制、参数更新、训练稳定性和量化精度上实现了全面的代际升级。最重要的是，V4 证明了开源模型也能同时做到"最大"（1.6T）和"最省"（百万上下文只需 27% FLOPs）。

那么，V4 之后呢？DeepSeek 团队在 V4 技术报告的结尾埋了一个伏笔——"在未来的迭代中，我们将进行更全面、更本质的研究，将架构简化到最核心的部分。" 这句话预示着 V5 的方向。下一章，也是全教程的最后一章，我们走出具体的技术细节，从更高的视角审视 DeepSeek 对整个 AI 行业的深远影响，并展望未来的 V5 路线图。
