🔨 第21章 实现 Skills 系统

大家好，我是你们的讲师。上一章我们深入讨论了 Context Engineering——如何用最少的 token 传递最多的信息。但这里有一个更深层的矛盾：你既希望 Agent 拥有尽可能多的专业知识，又不想把这些知识全部塞进上下文。全部塞进去，上下文爆炸、API 账单上天；不塞进去，Agent 碰到特定领域任务时表现拉胯。

Skills 系统就是 Claude Code 对这个矛盾的优雅解答。我在早期自己写 Agent 的时候，犯过一个典型的错误：我把所有我知道的知识点——Python 最佳实践、Docker 命令、Git 工作流、Kubernetes 操作——全部写进了 System Prompt。结果呢？Agent 在处理一个简单的文件读写任务时，却花了几千 token "思考"要不要用 K8s 命令。这不是 Agent 的问题，是我的问题——我不应该让 Agent 背着整个图书馆去散步，而是应该在它需要一本书的时候才帮它借。

Skills 系统的架构

Skills 系统的核心是 SkillManager，它负责管理技能的完整生命周期：

阶段 触发条件 执行动作 持续时间
Discover Agent 启动 / 手动触发 扫描 .claude/skills/ 目录下的 SKILL.md 文件 一次性
Load 用户指令命中触发词 / 显式调用 SkillTool 解析 SKILL.md，将技能内容注入上下文 当前任务期间
Inject 技能加载完成后，下一次模型调用前 将技能内容作为 SKILL CONTEXT 部分添加到 System Prompt 当前调用 / 持续到任务切换
Evict 任务完成 / Agent 检测到上下文主题切换 从 System Prompt 中移除技能内容 立即

Skills 架构的一个关键设计决策是：技能不在 Agent 启动时加载。启动时只做一次"发现"（discover），记录有哪些技能可用。实际加载发生在技能被匹配到用户指令之后。这种按需加载（lazy loading）策略确保了上下文不被无关知识污染。

SKILL.md 解析器实现

每个技能由一个 SKILL.md 文件定义，放在 .claude/skills/ 目录下（可以用于目录组织子目录）。文件格式遵循 YAML frontmatter + Markdown 正文的结构：

---
name: python-testing
description: Guidelines for writing and running Python unit tests
triggers:
  - "test"
  - "unittest"
  - "pytest"
  - "单元测试"
tools_needed:
  - "bash"
  - "read"
  - "write"
priority: 5
---

## Python Testing Skills

When writing tests in this project:
1. Use pytest as the test runner
2. Test files go in the `tests/` directory
3. Follow the Arrange-Act-Assert pattern
4. Mock external services using `unittest.mock`
...

解析器的实现：使用 Python 的 yaml.safe_load 解析 frontmatter，剩余部分作为 Markdown 正文。然后执行字段验证——确保 name 不为空、triggers 是列表、body 有合理长度。

import yaml
from dataclasses import dataclass, field
from pathlib import Path
from typing import List, Optional


@dataclass
class SkillDef:
    name: str
    description: str
    triggers: List[str] = field(default_factory=list)
    tools_needed: List[str] = field(default_factory=list)
    body: str = ""
    priority: int = 0


def parse_skill_file(path: Path) -> Optional[SkillDef]:
    """Parse a SKILL.md file with YAML frontmatter."""
    content = path.read_text(encoding="utf-8")
    if not content.startswith("---"):
        return None
    _, frontmatter, body = content.split("---", 2)
    meta = yaml.safe_load(frontmatter)
    # Validate
    if not meta.get("name"):
        return None
    if not isinstance(meta.get("triggers"), list):
        meta["triggers"] = []
    return SkillDef(
        name=meta["name"],
        description=meta.get("description", ""),
        triggers=meta["triggers"],
        tools_needed=meta.get("tools_needed", []),
        body=body.strip(),
        priority=meta.get("priority", 0),
    )
技能粒度建议：一个容易犯的错误是把技能粒度弄得太细或太粗。太细（比如"写 print 语句的技能"），你会管理几十个小技能，匹配成本和上下文切换开销很大。太粗（比如"全栈开发技能"），包含了太多不相关的内容，稀释了技能的针对性。黄金法则是：一个技能对应一个任务类型，而不是一个工具。Python 测试是一个技能，Git 工作流是另一个技能，Docker 部署又是一个技能。

触发匹配与按需加载

SkillManager 如何决定何时加载哪个技能？有三种触发方式：

  • 关键词匹配：用户消息中包含技能的 trigger 词。比如用户说"给我写个测试"，触发 python-testing 技能。
  • 上下文匹配：当前任务的上下文（比如工作目录中的文件类型、最近使用的工具）暗示了技能域。这个模式更复杂，通常需要模型参与判断。
  • 显式调用：用户通过 load_skill 工具显式加载技能。这是最直接的触发方式，也适用于新手用户。

匹配算法的核心是一个相关性评分函数：

def relevance_score(skill: SkillDef, query: str) -> float:
    """Score how relevant a skill is to the given query."""
    query_lower = query.lower()
    score = 0.0

    # Exact skill name mention -> high boost
    if skill.name.lower() in query_lower:
        score += 2.0

    # Trigger word matches -> moderate boost per match
    for trigger in skill.triggers:
        if trigger.lower() in query_lower:
            score += 1.0

    # Priority modifier
    score *= (1.0 + skill.priority * 0.1)

    return score

这个评分函数设计上的考虑：显式技能名匹配比 trigger 匹配权重更高（精确性 > 模糊性），priority 作为一个乘法因子而非加法因子，意味着高优先级的技能在同等匹配度下胜出。

一个重要的防误匹配机制：不加载空匹配的技能。如果 query 中没有出现任何 trigger，relevance_score 返回 0，技能不会被加载。这避免了 Agent 因为"看起来有点相关"就加载一堆无用的技能。

优先级合并策略

当多个技能同时匹配时，SkillManager 需要决定哪些技能被加载以及按什么顺序放置。合并策略按以下优先级排序：

  1. 精确性：技能名称直接匹配的 > trigger 匹配的（精确性优先）
  2. 明确性：显式调用加载的 > 关键词匹配的（用户意图优先）
  3. 时效性：最近使用过的 > 长期未使用的（recency boost）
  4. 优先级：高 priority 值的 > 低 priority 值的（权重优先）

合并后的技能内容需要被组装成一个整体，作为 System Prompt 的一部分注入。组装逻辑：

def build_skill_context(loaded_skills: List[SkillDef]) -> str:
    """Build SKILL CONTEXT section from loaded skills."""
    if not loaded_skills:
        return ""

    parts = [""]
    for skill in loaded_skills:
        header = f"--- {skill.name}: {skill.description} ---"
        parts.append(header)
        parts.append(skill.body)
        parts.append("")  # blank line separator
    parts.append("")
    return "\n".join(parts)

动态 Prompt 构建

Skills 注入到 System Prompt 的过程不是简单地在后面追加。它遵循"有界注入"原则——技能内容被包裹在 <SKILL_CONTEXT>...</SKILL_CONTEXT> 标签中，形成一个清晰的上下文边界。这个边界有两个作用：

  • 帮助模型识别：模型看到 <SKILL_CONTEXT> 标签就知道这是一段临时加载的外部知识，不是核心 System Prompt。
  • 便于移除：任务切换时，Harness 不需要重新生成整个 System Prompt，只需要移除 <SKILL_CONTEXT> 块中的内容。

Token 预算管理也是一个关键点。SkillManager 维护一个 _max_skill_tokens（默认 8192）的预算上限。加载技能时，按匹配度从高到低依次加载，直到预算用完。如果单个技能就超过了预算（这种情况应该很少见），它仍然会被加载（因为部分知识总比没有好）。

注意：技能内容会在任务切换时被自动移除。但什么叫"任务切换"呢？Claude Code 使用了一个启发式判断：如果用户连续 3 轮对话中的工具调用分布发生了显著变化（比如从大量文件操作切换到了大量 Git 操作），或者用户显式提到了一个新任务，Harness 就会触发技能上下文重置。

完整实现：SkillManager + SkillTool

以下是 SkillManager 和 SkillTool 的完整实现：

import yaml
from dataclasses import dataclass, field
from pathlib import Path
from typing import Dict, List, Optional


@dataclass
class SkillDef:
    name: str
    description: str
    triggers: List[str] = field(default_factory=list)
    tools_needed: List[str] = field(default_factory=list)
    body: str = ""
    priority: int = 0


class SkillManager:
    """Discovers, loads, and injects skills on demand."""

    def __init__(self, skills_dir: str = ".claude/skills"):
        self._skills_dir = Path(skills_dir)
        self._all_skills: Dict[str, SkillDef] = {}
        self._loaded: Dict[str, SkillDef] = {}
        self._max_skill_tokens = 8192

    def discover(self) -> Dict[str, SkillDef]:
        """Scan skills directory for SKILL.md files."""
        self._all_skills.clear()
        if not self._skills_dir.exists():
            return self._all_skills
        for skill_file in self._skills_dir.rglob("SKILL.md"):
            skill = self._parse_skill_file(skill_file)
            if skill:
                self._all_skills[skill.name] = skill
        return self._all_skills

    def _parse_skill_file(self, path: Path) -> Optional[SkillDef]:
        content = path.read_text(encoding="utf-8")
        if not content.startswith("---"):
            return None
        _, frontmatter, body = content.split("---", 2)
        meta = yaml.safe_load(frontmatter)
        if not meta or not meta.get("name"):
            return None
        if not isinstance(meta.get("triggers"), list):
            meta["triggers"] = []
        return SkillDef(
            name=meta["name"],
            description=meta.get("description", ""),
            triggers=meta["triggers"],
            tools_needed=meta.get("tools_needed", []),
            body=body.strip(),
            priority=meta.get("priority", 0),
        )

    def match(self, query: str) -> List[SkillDef]:
        """Find skills relevant to the query, scored by relevance."""
        query_lower = query.lower()
        scored = []
        for skill in self._all_skills.values():
            score = self._relevance_score(skill, query_lower)
            if score > 0:
                scored.append((score, skill))
        scored.sort(key=lambda x: -x[0])
        return [s[1] for s in scored]

    def _relevance_score(self, skill: SkillDef, query: str) -> float:
        score = 0.0
        if skill.name.lower() in query:
            score += 2.0
        for trigger in skill.triggers:
            if trigger.lower() in query:
                score += 1.0
        score *= (1.0 + skill.priority * 0.1)
        return score

    def load_skills(self, query: str) -> List[SkillDef]:
        """Load skills up to the token budget, highest relevance first."""
        matched = self.match(query)
        loaded = []
        total_tokens = 0
        for skill in matched:
            tokens = len(skill.body.split())
            if total_tokens + tokens <= self._max_skill_tokens:
                self._loaded[skill.name] = skill
                loaded.append(skill)
                total_tokens += tokens
        return loaded

    def build_prompt_section(self) -> str:
        """Build the SKILL CONTEXT section for system prompt injection."""
        if not self._loaded:
            return ""
        parts = [""]
        for skill in self._loaded.values():
            parts.append(f"--- {skill.name}: {skill.description} ---")
            parts.append(skill.body)
        parts.append("")
        return "\n\n".join(parts)

    def evict(self, skill_name: str):
        self._loaded.pop(skill_name, None)

    def clear_loaded(self):
        self._loaded.clear()


class SkillTool:
    """Tool that lets the agent load skills at runtime."""

    name = "load_skill"
    description = "Load a skill by name or topic into the current context"

    def __init__(self, manager: SkillManager):
        self._manager = manager

    async def execute(self, query: str) -> str:
        skills = self._manager.load_skills(query)
        if not skills:
            return "No matching skills found."
        names = [s.name for s in skills]
        return f"Loaded skills: {', '.join(names)}"

集成到 v4 Agent 中很简单：在 Agent 初始化时创建 SkillManager，调用 discover()，然后将 SkillTool 添加到工具注册表中。当 Agent 需要加载技能时，它调用 load_skill 工具，之后的下一次模型调用中，技能内容会自动出现在 System Prompt 中。

至此，你的 Agent 拥有了按需加载专业知识的能力。它不再需要背着一整个图书馆散步了。但这里有一个更深层的问题：这些技能加载进来之后，Agent 如何"知道自己该做什么"？这就引出了我们课程中最重要的一个设计模式讨论——Harness 设计模式。在下一章中，我们将对比分析三种主流的 Harness 完成模式，看看它们各自的优劣和适用场景。
