第9章 5G NR接入网

大家好，我是你们的讲师。上一章我们聊了5G核心网——那个藏在数据中心里、用软件定义的"大脑"。但大脑再好，没有神经末梢去感知和连接用户，也是白搭。这个"神经末梢"就是接入网，也就是我们常说的基站。

在4G时代，基站是一个整体：eNB（Evolved NodeB）集成了所有无线功能，从射频处理到基带处理再到高层协议栈，全部装在一个机柜里。但5G基站完全不同——它被拆成了三个独立的单元：CU（Central Unit）、DU（Distributed Unit）和RU（Radio Unit）。这种拆分不是技术炫技，而是为了满足5G多样化的部署场景：高带宽的密集城区需要分布式部署，低时延的工业场景需要靠近用户部署，广覆盖的农村场景则需要集中式部署。一套架构，三种部署，全凭"拆分"这个设计哲学。

截至2026年，中国5G基站部署量已超过420万站，其中超过60%采用了CU-DU-RU分离架构。O-RAN联盟定义的开放接口标准正在推动接入网从"黑盒"走向"白盒"，运营商的设备采购成本因此下降了约30%。

9.1 gNB架构：CU、DU、RU三层拆分

5G的基站被称为gNB（next Generation NodeB）。一个gNB在逻辑上是一个完整的基站，但物理上可以由一个CU、多个DU和多个RU组成。三者各自承担不同层次的功能：

  • gNB-CU（Central Unit）：处理高层协议栈，包括RRC、PDCP、SDAP层。负责与核心网AMF/UPF的N2/N3接口通信。CU的计算负载相对较轻，但对处理器的单核性能有要求。
  • gNB-DU（Distributed Unit）：处理底层协议栈，包括RLC、MAC、部分PHY（High-PHY）层。负责实时调度、HARQ、随机接入等时延敏感功能。DU对实时性要求极高，通常运行在RTOS（实时操作系统）上。
  • gNB-RU（Radio Unit）：处理射频和Low-PHY功能，包括FFT/ IFFT、数字预失真（DPD）、波束赋形、数字上下变频等。RU直接连接天线，负责射频信号的收发。

9.2 功能拆分选项

CU-DU-RU之间的功能不是随便切的，3GPP和O-RAN定义了多个功能拆分选项。最常用的是Option 2（CU-DU之间的High-Layer Split）和Option 7.2（DU-RU之间的Low-Layer Split）。

拆分选项 拆分点 接口 传输带宽需求 典型部署
Option 2 CU（PDCP）与DU（RLC/MAC/PHY/RF）之间 F1（Midhaul） 约50-200 Mbps C-RAN集中式部署，适合多数宏站场景
Option 7.2 DU（High-PHY）与RU（Low-PHY/RF）之间 eCPRI（Fronthaul） 约10-25 Gbps（100MHz 64T64R） 分布式部署，AAU拉远，适合密集城区
Option 6 MAC与PHY之间 - 极高 较少商用，主要用于O-RAN开放架构
Option 8 RU与天线之间（完全集成） CPRI 最高（100-200 Gbps） 传统4G基站架构，5G已逐步淘汰

关键权衡：拆分点越靠近天线（Option编号越大），RU的处理能力越强，Fronthaul带宽需求也越大。Option 2的F1接口带宽仅需几十到几百Mbps，可以使用常规以太网或微波传输；Option 7.2的eCPRI接口带宽高达10-25 Gbps，必须使用光纤直连。因此实际部署中，CU-DU之间的传输距离可以较远（几十公里没问题），但DU-RU之间的距离通常限制在10-20公里以内。

9.3 F1接口：CU与DU的"连接纽带"

F1接口是3GPP TS 38.470系列规范定义的CU-DU间接口，分为两个部分：

  • F1-C（控制面）：基于SCTP协议，传输UE上下文管理、RRC消息透传、系统信息分发等信令。AMF发来的NAS消息通过CU经过F1-C转发给DU，再由DU通过空口发给UE。
  • F1-U（用户面）：基于GTP-U协议，传输用户面数据。CU从核心网UPF接收到用户数据后，通过F1-U转发给DU，DU再通过空口下达给UE。

F1接口的设计允许一个CU管理数百个DU，也允许一个DU同时连接到多个CU（用于灾备）。CU和DU之间的传输网络可以是IP专线、MPLS VPN甚至微波，只要满足时延和带宽要求即可。

9.4 Xn接口：gNB之间的"握手"

当用户从一个基站移动到另一个基站时，两个gNB之间需要通过Xn接口交互信息。Xn接口也分为控制面（Xn-C，基于SCTP）和用户面（Xn-U，基于GTP-U）。Xn接口的主要用途包括：

  • 切换（Handover）：UE从源gNB移动到目标gNB时，源gNB通过Xn接口将UE上下文转发给目标gNB，支持无缝切换（Break-Before-Make或Make-Before-Break）。
  • 双连接（EN-DC / NR-DC）：UE同时连接两个gNB（或一个LTE eNB加一个NR gNB），数据通过Xn接口聚合，实现更高的用户吞吐量。
  • 干扰协调：相邻gNB通过Xn交换干扰信息，协调波束方向和功率分配，降低小区间干扰。

9.5 O-RAN架构：开放与智能

传统基站设备是"一揽子买卖"——华为、爱立信、诺基亚提供整套设备，运营商无法选配。O-RAN（Open RAN）联盟致力于打破这种局面，通过定义开放的接口标准，让不同厂商的CU、DU、RU可以互操作。

O-RAN架构的关键组件包括：

  • Near-RT RIC（近实时RIC）：运行在CU/DU侧，时延响应在10ms-1s级别。负责负载均衡、切换优化、干扰管理等近实时策略控制。通过E2接口控制gNB的行为。
  • Non-RT RIC（非实时RIC）：运行在O-RAN管理中心，时延响应大于1秒。负责AI/ML模型的训练与下发、网络策略的长期优化。通过A1接口向Near-RT RIC下发策略目标。
  • E2接口：连接RIC和gNB的控制面接口，支持策略下发和UE/小区级别的数据采集。
  • 开放前传（Open Fronthaul）：基于Option 7.2拆分，定义DU-RU之间的eCPRI接口规范，支持多厂商RU和DU的互操作。

行业趋势：中国移动在2024年启动了O-RAN现网试点，在部分省份部署了基于O-RAN架构的5G小站。虽然O-RAN的整体性能与传统一体化基站仍有差距（主要是互操作测试不充分导致的稳定性问题），但在室内覆盖、边缘园区等场景已经具备商用条件。预计到2027年，O-RAN设备将占据全球5G接入网市场的15-20%。

9.6 中国运营商CU/DU部署实践

中国地域辽阔，不同场景对CU/DU的部署方式要求差异很大。以下是三种主流的部署模式：

  • 集中式（C-RAN）：CU和DU集中部署在中心机房，RU拉远到多个基站站点。适合密集城区，光纤资源丰富，便于统一管理和维护。中国移动在TDD 2.6GHz频段的宏站大多采用这种方式。
  • 分布式（D-RAN）：CU集中、DU分布式部署，每个DU管理少量RU。适合郊区或光纤资源紧张的区域，缩短前传距离。
  • 一体化：CU/DU/RU集成在一个设备中（即传统的一体化基站）。适合农村广覆盖、应急通信车等场景，部署最简便。

中国电信和中国联通在共建共享网络中大量采用了集中式C-RAN方案，通过CU共享降低投资成本。据公开数据，CU集中化部署将基站配套成本降低了约20%，同时提升了资源利用率。

9.7 踩坑实录：CU-DU传输时延超标导致RRC连接成功率骤降

2024年某地市运营商新建了一批5G站点，采用C-RAN集中式部署，CU在中心机房，DU分布在各个基站站点，中间通过IPRAN传输网络连接。站点开通后，调测工程师发现一个奇怪的现象：部分站点的RRC连接成功率只有70%左右，远低于正常水平（应大于99%）。

排查过程一波三折。一开始怀疑是空口问题，检查了SSB功率、波束配置、干扰水平，都没问题。接着怀疑是核心网问题，检查了AMF的注册响应，也正常。最后在DU侧抓包分析信令流程，终于发现了端倪——RRC Setup Complete消息从DU发送到CU再返回确认，来回的时延达到了35ms，而3GPP要求的F1接口最大单向时延是10ms。

进一步排查发现，问题出在IPRAN传输网络上：这几个站点的CU-DU之间经过了过多的网络跳数，中间还穿过了一个配置不当的QoS队列，导致信令包被误判为低优先级而排队等待。调整了传输网络的QoS策略，将F1-C信令标记为高优先级后，RRC连接成功率立即恢复到了99.5%以上。

教训：CU-DU分离架构引入了传输网络依赖，传输时延不再是"网络部门的事"而是直接影响空口性能的关键因素。基站工程必须联合传输团队一起做端到端的时延测试，尤其要关注F1-C接口的信令面时延。空口问题可能根子在传输上——做通信工程，要时刻保持"端到端"的视野。

// F1接口CU-DU传输时延验证（简化示例）
// 使用ICMP Ping测量RTT，识别超时延链路节点
$ ping -c 100 -s 1500 cu-du-interface.example.com
--- cu-du-interface.example.com ping statistics ---
100 packets transmitted, 100 received, 0% packet loss
rtt min/avg/max/mdev = 2.145/5.678/38.234/4.123 ms
// 平均时延5.678ms，最大RTT 38ms（单向约19ms）
// 3GPP要求F1-C单向时延 < 10ms → 超标
// 排查结果：IPRAN某节点QoS队列配置错误
