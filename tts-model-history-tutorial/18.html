第18章 前端文本处理

大家好，我是你们的讲师。在TTS系统中，最容易被人忽视但最容易出问题的一个模块就是——前端文本处理。你可能会想：不就是把文字转成拼音吗？有什么难的？

让我给你讲一个真实的故事。2021年，某知名导航App的TTS在播报"行驶至G25长深高速南京入口"时，把"长深高速"读成了"grow deep highway"——拼音处理模块把多音字"长"错误地标注成了zhǎng而不是cháng。一时间，社交媒体上充满了调侃这个问题的段子。更尴尬的是，这个问题在线上运行了整整三个月才被发现，因为测试数据集里恰好没有包含"长深高速"这个地名。从那时起，这家公司的前端处理系统里专门加了一条硬编码规则："长深高速"强制读cháng。

这个故事告诉我们：前端文本处理决定了TTS系统"下限"的高度——声学模型再好、声码器再先进，如果前端把音读错了，合成出来的就是"用极其自然的声音说了一句极其离谱的话"。

TTS的第一步：让机器"读懂"文字

前端文本处理（Text Frontend）的任务，是将原始文本转换为声学模型能够理解的语言学特征。整个流程可以分为几个步骤：文本正则化（TN）→ 分词（Tokenization）→ 字音转换（G2P）→ 韵律预测。每一步都有独特的挑战，尤其是对中文来说。

文本正则化（TN）

文本正则化（Text Normalization）的目标是将输入文本中的非标准格式转换为标准口语形式。考虑以下几个例子：

  • "2024年" → "二零二四年"（不是"两千零二十四年"也不是"two thousand twenty-four"）
  • "3.14" → "三点一四"（小数点后逐位读）
  • "98%" → "百分之九十八"
  • "¥199" → "一百九十九元"
  • "2024/06/30" → "二零二四年六月三十日"
  • "10086" → "幺洞洞八六"（电话号码逐位读，"1"读成"幺"）
  • "32路" → "三十二路"（公交线路）vs. "32号" → "三十二号"（门牌号）

这些规则看起来简单，但组合起来非常复杂。传统的TN方案使用正则表达式 + 有限状态机（FST）来处理：对每一种数字格式写一套规则，按优先级匹配。这种方法的好处是可控性强、可解释性好；但缺点是规则维护成本高、对于格式模糊的输入容易出错。

近年来，基于Transformer的序列到序列模型被用于TN任务。Google在2018年提出的RNN-TN系统直接将原始文本映射到规范化后的文本，用一个模型替代了上千条手工规则。但这种方法的缺点是偶尔会"创造性发挥"——把不该规范的文本也改了。因此工业界通常采用"规则为主、模型兜底"的混合策略。

分词与G2P

对于中文来说，分词（Word Segmentation）是字音转换的前提。因为中文不像英文有天然的空格分隔，同一个词在不同的上下文中可能有完全不同的读法。

G2P（Grapheme-to-Phoneme，字形到音素转换）是将分词后的文本转换为音素序列的过程。在中文TTS中，G2P本质上就是"将汉字转换为拼音"，但这远非查字典那么简单。

中文字音转换有多种方案：

  • 基础拼音方案：使用声母+韵母+声调的方式表示（如"好" → hǎo）。这是中文TTS中最常见的表示方式。
  • IPA方案：使用国际音标（International Phonetic Alphabet），可以更精确地描述发音细节。例如"好"的IPA表示是 /xɑʊ̯˨˩˦/。IPA的优势是可以跨语言统一表示，但在中文TTS中并不常用。
  • 音素级方案：将音节拆解为更细粒度的音素单元。例如"好"拆解为 h + ɑo + 上声。这种方式可以捕捉到协同发音的细微变化。
中文G2P的特殊性：中文的"轻声"和"儿化音"是G2P中的难点。比如"妈妈"的第二个"妈"要读轻声（mā ma，不是mā mā），而"这儿"要读儿化音（zhèr）。这些现象在英文中不存在，但对于中文TTS的自然度至关重要。

多音字消歧：TTS的噩梦

多音字是中文TTS前端处理的最大挑战。同一个汉字在不同上下文中可能有完全不同的发音和含义。下面是中文TTS中最令人头疼的几组多音字：

汉字 常见读音 典型上下文 较罕见读音 典型上下文
行 xíng（行为、行动） "这种行为不可取" háng（银行、行业） "这家银行在行业内排名靠前"
了 le（助词，完成了） "我吃了饭" liǎo（了结、了不起） "这件事终于了结了"
着 zhe（助词，持续态） "他正在看着书" zháo（着急、着凉）、zhāo（着数） "别着急，我还有一招没使"
长 cháng（长度、长久） "这条路很长" zhǎng（生长、长辈） "他长高了，成了长辈"
重 zhòng（重量、重要） "这个问题很重要" chóng（重复、重新） "请你重新说一遍"
的 de（助词） "我的书" dí（的确）、dì（目的） "他的目的的确不单纯"
乐 lè（快乐、乐观） "祝你生日快乐" yuè（音乐、乐队） "他是一名音乐老师"

早期TTS系统使用规则和词典来处理多音字——比如"如果'行'后面跟'银'、'业'、'内'、'业界'，就读háng"。这种方法实现简单、推理速度快，但覆盖面有限、对未知词无能为力。

现代TTS系统通常使用基于BERT的消歧分类器。做法如下：

from transformers import BertForSequenceClassification, BertTokenizer

class PolyphoneDisambiguator:
    """
    基于BERT的多音字消歧分类器
    """

    def __init__(self, model_name="bert-base-chinese"):
        self.tokenizer = BertTokenizer.from_pretrained(model_name)
        # 为每个多音字训练独立的分类头
        # 这里以"行"为例，输出2类：xíng（0）或 háng（1）
        self.model = BertForSequenceClassification.from_pretrained(
            model_name,
            num_labels=2
        )

    def predict(self, text, target_char="行", context_window=5):
        """
        预测目标多音字在文本中的读音

        Args:
            text: 输入文本
            target_char: 需要消歧的汉字
            context_window: 上下文窗口大小（字符数）

        逻辑:
            1. 在text中找到target_char的位置
            2. 取前后context_window个字符作为上下文
            3. 用BERT编码后通过分类头预测读音
        """
        # 找到目标字的位置
        pos = text.find(target_char)
        if pos == -1:
            return None

        # 构建输入：标记目标字位置
        start = max(0, pos - context_window)
        end = min(len(text), pos + context_window + 1)

        # 在目标字前后插入特殊标记
        marked_text = (
            text[start:pos] +
            "[MASK]" + target_char + "[MASK]" +
            text[pos+1:end]
        )

        inputs = self.tokenizer(
            marked_text,
            return_tensors="pt",
            padding="max_length",
            max_length=32,
            truncation=True
        )

        with torch.no_grad():
            outputs = self.model(**inputs)
            probs = torch.softmax(outputs.logits, dim=-1)

        pred_class = torch.argmax(probs, dim=-1).item()
        confidence = probs[0][pred_class].item()

        return {
            "char": target_char,
            "prediction": "háng" if pred_class == 1 else "xíng",
            "confidence": round(confidence, 4),
            "context": text[start:end]
        }

# 测试
disambiguator = PolyphoneDisambiguator()

print(disambiguator.predict("我在银行工作"))
# {'char': '行', 'prediction': 'háng', 'confidence': 0.9987}

print(disambiguator.predict("这种行为不可取"))
# {'char': '行', 'prediction': 'xíng', 'confidence': 0.9963}

基于BERT的方法在多音字消歧上可以达到98%以上的准确率。但剩下的2%就是"噩梦"所在——比如"我在人行道上走着"中的"行"（人行道，应读xíng，但容易被误判为人"行"）。这些边界情况往往需要额外的领域知识和规则来弥补。

韵律结构预测

有了正确的拼音之后，还需要让机器知道在哪里停顿、哪里重读。这是决定TTS输出是否"自然"的关键环节。

中文的韵律结构通常分为三个层级：

  • 韵律词（Prosodic Word, PW）：最小的韵律单元，通常由一个或几个音节组成，内部不停顿。
  • 韵律短语（Prosodic Phrase, PPH）：由几个韵律词组成，之间有较短的停顿（约50-100ms）。
  • 语调短语（Intonational Phrase, IP）：最大的韵律单元，对应句子中的主要停顿或句子边界，停顿较长（100-300ms）。

传统方法使用CART树（分类与回归树）来预测韵律边界：基于词性、词长、句法位置等特征构建决策树。这种方法在数据量有限时效果还不错，但对复杂的韵律现象（如强调重音、对比焦点）表现不佳。

当前主流方法将韵律预测建模为序列标注问题——使用BiLSTM-CRF或Transformer对每个词/音节打上韵律标签：

# 韵律标注格式（以音节为单位）：
# B: 韵律词边界（短停顿）
# M: 韵律短语边界（中等停顿）
# E: 语调短语边界（长停顿）
# O: 词内（无停顿）

# 输入文本： "今天天气真好"
# 分词后：   [今天] [天气] [真] [好]
# 韵律标注： [今天]_B [天气]_B [真]_O [好]_E

# 序列标注模型（简化版）
def predict_prosody(tokens, model):
    """
    预测每个词的韵律边界
    返回每个词后的停顿类型
    """
    # 提取特征：词本身、词性、词长、前后词
    features = extract_prosody_features(tokens)

    with torch.no_grad():
        logits = model(features)  # 输出 [num_tokens, 4]
        labels = torch.argmax(logits, dim=-1)

    label_map = {0: "O", 1: "B", 2: "M", 3: "E"}
    return [label_map[l.item()] for l in labels]

# 韵律预测的典型准确率：
# B 边界（词边界）: ~95%
# M 边界（短语边界）: ~85%
# E 边界（句子边界）: ~98%

中英文前端差异

在结束本章之前，来对比一下中文和英文TTS前端处理的差异，这能帮助你理解为什么中文TTS的前端实现比英文复杂得多：

维度 中文 英文
分词 需要额外的分词步骤，无天然分隔符 天然以空格分隔，部分缩写需要特殊处理
字音转换 多音字问题突出，约10%的汉字有多个读音 拼写→发音规则较统一，不规则拼写（如colonel→kernel）较少
声调 四个声调+轻声，声调影响词义 无词汇声调，只有句子层面的语调
数字规范 数字读法变化多：电话号码逐位读，金额有"万""亿" 数字读法相对统一，用thousand/million/billion
缩写 缩写较少（如"北大"→"北京大学"） 缩写非常普遍（如"Dr."→"Doctor","St."→"Street"）
连读现象 变调（三声变二声）、儿化音 连读（want to→wanna）、失去爆破

值得特别一提的是中文的声调建模。在中文TTS中，连续的声调变化会使音高轮廓复杂化。比如两个第三声相连时，前一个要变为第二声（"你好"的"你"→nǐ变成ní）。经典的"一"和"不"的变调规则也非常复杂——"一个"中的"一"读二声，"第一"中的"一"读一声，"看一看"中的"一"读轻声。这些规则不仅要在前端处理中准确实现，还要和后端声学模型协同工作——如果前端标记了声调但声学模型没有正确建模声调轮廓的变化，输出就会听起来"怪怪的"。

前端处理能力的边界：即使是最先进的TTS前端系统，也无法完美处理所有情况。一个典型的工业级TTS前端通常包括：一个基于词典的基础G2P模块、一个基于规则的TN模块、一个基于BERT的多音字消歧模型、一个基于序列标注的韵律预测模型、以及几百条针对特定场景定制的硬编码规则。它不是一个"端到端"的模型，而是一个由多个子系统构成的"专家系统"——这在深度学习时代显得有些格格不入，但对于追求稳定性的TTS产品来说，这种"笨办法"反而最可靠。

前端文本处理是TTS系统中"最不性感但最不可或缺"的模块。它的好坏不体现在MOS分数上（因为前端处理的好坏不影响音质），而是体现在你日常使用中是否会碰到"把长深高速读成grow deep highway"这种问题。一个好的前端系统应该做到让人意识不到它的存在——它默默地处理了所有边界情况，让你听到的每个句子发音都准确无误。

前端处理的输出——音素和韵律标注——是后端声学模型的"输入原料"。前端的质量直接决定了后端能发挥的上限。从下一章开始，我们将进入TTS最激动人心的前沿话题：零样本语音克隆。仅需几秒钟的音频就能模仿任何人的声音——这一技术正在彻底改变我们对TTS能力的认知。
