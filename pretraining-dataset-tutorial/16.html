开篇：你的模型还没开始训练，内存先撑爆了

大家好，我是你们的讲师。上一章我们解决了"数据太多，单机装不下"的问题，用分布式架构把数据分散到多台机器上处理。但你可能马上会遇到一个新的、更"狡猾"的难题：数据加载环节把内存撑爆了。

让我讲一个真实的故事。2024 年我参与了一个 7B 模型的训练项目，团队花了两周做数据清洗，数据质量打分、去重、过滤样样不落，预处理结果完美无缺。结果在正式开始训练的第一天，训练脚本在 load_dataset() 这一步直接卡死了。什么提示都没有——就是 OOM（Out of Memory），被 Linux kernel OOM killer 直接 kill 了。团队成员面面相觑："我们的数据只有 2 TB 啊，服务器有 512 GB 内存，怎么不够？"

原因很简单：Hugging Face Datasets 的默认行为是把整个数据集加载到内存中。2 TB 的数据，经过 tokenization 之后会膨胀到 4-5 TB——远远超过 512 GB。这就是典型的 "数据加载方式决定训练成败" 的案例。本章我们来系统性地解决这个问题。

16.1 Hugging Face Datasets Streaming：懒加载的艺术

解决方案其实很简单：不要一次性加载全部数据，而是流式加载（streaming）。Hugging Face Datasets 从 2.0 版本开始支持流式模式，关键只是一个参数：

from datasets import load_dataset

# ❌ 错误做法：一次性加载全部数据到内存
dataset = load_dataset("json", data_files="s3://my-bucket/train/*.jsonl")
# 这里数据集所有文件都会被读到内存中，OOM 风险极高

# ✅ 正确做法：使用 streaming=True
dataset = load_dataset(
    "json",
    data_files="s3://my-bucket/train/*.jsonl",
    split="train",
    streaming=True  # 关键参数！
)

# 现在 dataset 是一个 generator，每次迭代只加载一个样本
for i, example in enumerate(dataset):
    if i >= 1000:
        break
    process(example)  # 每次只处理一个样本

当 streaming=True 时，load_dataset 不会读取任何文件内容，而是仅仅扫描文件索引（即"数据在哪里"）。真正的数据读取延迟到 for 循环迭代的瞬间。这样，无论数据集是 10 GB 还是 10 TB，Python 进程的常驻内存占用都只取决于单个样本的大小——通常只有几百 KB 到几 MB。

对比维度 非流式加载 流式加载
内存占用 整个数据集（TB 级） 当前 batch（MB 级）
启动时间 数分钟到数小时（全部读取完） 秒级（只扫描索引）
随机访问 支持（数据在内存中） 不支持（除非使用索引缓存）
shuffle 全局 shuffle 方便 需要 buffer shuffle
重复迭代 零额外开销（数据已加载） 每次重新读取文件
适合场景 探索性分析、小数据集 预训练、大规模流水线
小技巧：在流式加载下做 shuffle 可以用 dataset.shuffle(buffer_size=10000)。这会在内存中维护一个 10,000 条样本的缓冲区，每次随机从中取出一个样本并用新样本补充。这样做虽然不能保证完美均匀的 shuffle（因为缓冲区大小有限），但对于预训练来说已经足够。

16.2 WebDataset 格式：为流式而生

Hugging Face Datasets 的流式模式虽然好用，但它有一个根本性的限制：每次迭代都需要从存储系统发起文件打开操作。如果数据文件数量是百万级别（对于 TB 级数据集来说这很常见），文件系统压力会非常大。

WebDataset 应运而生。它的核心理念是：把大量小文件打包成若干个大的 tar 包（通常每个 1-5 GB），然后通过 tar 格式的流式读取能力来逐个解包其中的样本。

import webdataset as wds

# 创建 WebDataset 格式的数据集
urls = [
    "s3://my-bucket/shard-000000.tar",
    "s3://my-bucket/shard-000001.tar",
    # ... 1000 个 tar 包，每个包含 10000 条样本
]

dataset = (
    wds.WebDataset(urls, shardshuffle=True)
    .shuffle(10000)
    .decode("pil")
    .to_tuple("input_ids", "attention_mask")
)

# 使用 DataLoader
loader = wds.WebLoader(
    dataset,
    batch_size=32,
    num_workers=4,
)

WebDataset 有几个显著优势：

  • 高效随机访问：每个 tar 包内的样本是顺序存储的，但 tar 包的顺序可以 shuffle。通过维护一个大的缓存池，可以实现良好的随机性。
  • S3 友好：tar 包可以使用 S3 Range Read 来只读取需要的数据，无需下载整个 tar 包。
  • 与多进程 DataLoader 完美配合：每个 worker 线程可以独立访问不同的 tar 包，天然并行。
重要区别：Hugging Face Datasets streaming 是"按文件流式读取"，WebDataset 是"按 tar 包流式读取"。二者的性能差异在百万级文件的场景下极为显著。如果数据文件数量超过 10 万个，强烈推荐 WebDataset 格式。

16.3 内存预算规划

流式加载解决了"数据加载时 OOM"的问题，但训练过程中的内存管理仍然需要精心规划。我们来看看在典型的 LLM 训练中，内存是怎么被消耗掉的：

内存消耗计算公式（以 7B 模型为例）：

模型参数 (FP16):             7B × 2 bytes = 14 GB
梯度 (FP16):                 7B × 2 bytes = 14 GB
优化器状态 (Adam, FP32):     7B × 12 bytes = 84 GB  ← 大头！
激活值 (batch_size=4, seq=4096): ~15 GB
DataLoader 缓冲:             取决于 prefetch_factor
─────────────────────────────────────────
总计:                        ~127 GB + 数据缓冲

可以看到，光是模型训练本身就消耗了 127 GB 显存，这已经接近一张 A100-80G 的承载上限了（需要用 ZeRO-3 或张量并行来分摊）。数据加载能"分到"的内存非常有限——通常只有几个 GB。以下是一个综合性的内存预算表：

组件 建议内存预算 用途 可调优参数
DataLoader 缓冲区 2-4 GB 预处理后的 token 序列缓存 prefetch_factor
Streaming 缓冲区 1-2 GB 流式读取的原始数据缓存 buffer_size
Tokenization 缓存 1-2 GB 避免重复 tokenization 缓存策略
Shuffle 缓冲区 0.5-1 GB 流式 shuffle 的工作缓冲区 shuffle_buffer
进程间通信 0.5-1 GB multiprocessing 队列 shared_memory
GC 预留 1-2 GB Python 垃圾回收的临时开销 多次手动 gc.collect()

这里有一个容易被忽视的点：Python 的内存分配策略。Python 的 gc（垃圾回收）模块默认采用引用计数 + 分代回收策略，但回收时机不可预测。在大规模数据处理中，内存泄漏往往不是真的"泄漏"，而是 Python 还没来得及释放不再使用的对象。

import gc
import torch

def train_epoch(dataloader):
    for batch in dataloader:
        # 训练步骤
        loss = model(batch)
        loss.backward()
        optimizer.step()
        optimizer.zero_grad()

        # 定期手动触发垃圾回收
        if step % 100 == 0:
            gc.collect()        # 回收 CPU 内存
            torch.cuda.empty_cache()  # 回收 GPU 缓存

        # 监控内存使用
        if step % 1000 == 0:
            cpu_mem = psutil.Process().memory_info().rss / 1e9
            gpu_mem = torch.cuda.memory_allocated() / 1e9
            print(f"Step {step}: CPU={cpu_mem:.2f}GB, GPU={gpu_mem:.2f}GB")
踩坑实录：有一次我们的训练脚本在运行了大约 12 小时后突然 OOM 了。排查后发现是 DataLoader 的 collate_fn 中有一行代码 batch = list(batch) —— 这一行把生成器一次性展开成了列表保存了所有 batch！在流式模式下，这一行代码就等于把整个数据集的副本一次性塞进了内存。修复后内存占用从 200 GB 降到了 3 GB。有时候，OOM 的罪魁祸首只是一行代码。

16.4 Sequence Packing：如何高效拼接序列

预训练数据和推理数据有一个重要区别：预训练的输入不需要"完整"的语义单元——你不需要每一条文本都是一个完整的句子或段落。相反，你只需要把文本切分或拼接成固定长度（比如 4096 token）的序列。

这就是 Sequence Packing 的用武之地。它的核心思想是：把多条短文本首尾相连，用 <|endoftext|> 等特殊 token 隔开，拼成一条固定长度的序列。

def pack_sequences(
    documents: list[str],
    tokenizer,
    max_length: int = 4096,
    eos_token_id: int = 50256,
) -> list[dict]:
    """
    将多个文档拼接成固定长度的序列。
    每个文档之间用 EOS token 分隔。
    """
    sequences = []
    current_tokens = []
    current_length = 0

    for doc in documents:
        # Tokenize 当前文档
        tokens = tokenizer.encode(doc)
        doc_len = len(tokens) + 1  # +1 为 EOS token

        if current_length + doc_len > max_length:
            # 当前序列已满：补齐到 max_length 并保存
            pad_len = max_length - current_length
            current_tokens.extend([eos_token_id] * pad_len)
            sequences.append({
                "input_ids": current_tokens,
                "attention_mask": [1] * current_length + [0] * pad_len,
            })
            current_tokens = []
            current_length = 0

        # 添加当前文档（含 EOS）
        current_tokens.extend(tokens + [eos_token_id])
        current_length += doc_len

    # 处理最后一段不足 max_length 的序列
    if current_tokens:
        pad_len = max_length - current_length
        current_tokens.extend([eos_token_id] * pad_len)
        sequences.append({
            "input_ids": current_tokens,
            "attention_mask": [1] * current_length + [0] * pad_len,
        })

    return sequences

Sequence Packing 带来的好处是巨大的：它可以把数据利用率从 60-70%（用简单的 padding 方案）提升到接近 100%。对于 total_loss 的计算来说，你只需要在 loss 函数中忽略掉 padding 位置的贡献（通过 attention mask），拼接的痕迹就不会影响模型训练。

关键洞察：Sequence Packing 不仅仅是"节省 padding token"，它还有另一个隐含效果——让模型在因果语言建模的训练中看到更多样的上下文边界。模型需要学会识别 EOS token 并理解"前面的文档结束了，一个新的文档开始了"。这其实增强了模型的文档理解和切换能力。

16.5 Arrow 格式 vs Parquet：列式存储的对比

在数据存储格式的选择上，Apache Arrow 和 Apache Parquet 是最常被放在一起比较的两个列式存储格式。它们都来自 Apache 生态，但设计哲学和使用场景有显著差异。

对比维度 Apache Arrow Apache Parquet
设计目标 内存中的列式分析格式 磁盘上的列式存储格式
读取速度 零拷贝（memory-mapped） 需要反序列化
内存占用 高（数据常驻内存） 低（按需读取）
压缩率 低（无内置压缩） 高（Snappy, ZSTD 等）
随机访问 极快（指针计算） 快（Row Group 索引）
适合场景 训练数据加载、流式处理 数据存档、分析查询
Hugging Face 支持 原生支持（Datasets 默认格式） 支持（通过 parquet 参数）
内存映射 I/O 支持（mmap） 不支持

对于预训练数据加载，Arrow 格式有一个特别重要的特性——内存映射（memory-mapped I/O）。当 Arrow 文件被 mmap 到虚拟地址空间时，操作系统只在真正访问某个页面时才将其加载到物理内存，而不会一次性读取整个文件。这意味着你可以安全地"假装"整个数据集都在内存中——即使它实际上只有一小部分常驻物理内存。

import pyarrow as pa

# 使用 Arrow 格式的 memory-mapped I/O
mmap_stream = pa.memory_map("s3://my-bucket/data/train.arrow")

# 读取 schema（元数据）- 不加载数据
schema = pa.ipc.open_stream(mmap_stream).schema
print(f"Total rows: {schema.num_rows}")  # 秒级返回

# 只读取第 1000-2000 行
table = pa.ipc.open_stream(mmap_stream).read_all()
subset = table.slice(1000, 1000)  # 零拷贝切片
存储最佳实践：在数据清洗和预处理阶段使用 Parquet 格式存储中间结果（压缩率高，节省存储费用），在训练阶段转换为 Arrow 格式（内存映射读取快）。很多团队的实际做法是：S3 上存 Parquet，训练节点本地缓存 Arrow 格式的 mmap 文件。

16.6 多 Worker 数据加载与 Prefetch 调优

在 PyTorch 训练中，数据加载通常使用 DataLoader 的多 worker 模式来掩盖 I/O 延迟。但这里有一个微妙的平衡：

from torch.utils.data import DataLoader

# 基础配置
dataloader = DataLoader(
    dataset,
    batch_size=32,
    num_workers=8,          # 8 个 worker 并行加载
    prefetch_factor=2,      # 每个 worker 预取 2 个 batch
    pin_memory=True,        # 使用锁页内存加速 CPU→GPU 传输
    persistent_workers=True # 避免每个 epoch 重新创建 worker
)

prefetch_factor 是这里最关键的参数。它控制着每个 worker 在内存中预加载多少个 batch。值为 2 意味着：GPU 正在训练第 N 个 batch 时，数据加载 worker 已经在准备第 N+1 到 N+2*8 个 batch。这样 GPU 就不会因为等待数据而空闲。

但 prefetch_factor 不是越大越好。每个预取的 batch 都需要占用内存——对于 seq_length=4096, batch_size=32 的情况，每个 batch 大约需要 32 × 4096 × 2 ≈ 0.25 MB 的 token 数据（FP16）。prefetch_factor=2 时，内存占用 = 8 worker × 2 × 0.25 MB = 4 MB，可以忽略不计。但如果你的预处理包含图像解码或复杂的 tokenization，每个 batch 可能是几十 MB，这时候 8 worker × 2 batch 可能就变成几百 MB 了。

场景 建议 num_workers 建议 prefetch_factor 说明
纯文本 tokenization 4-8 2-4 I/O 轻量，模型训练是瓶颈
带图像解码的多模态 8-16 1-2 解码耗 CPU，需更多 worker
流式 S3 读取 2-4 4-8 网络延迟大，需更多 prefetch
本地 SSD 读取 2-4 1-2 本地读取快，不需要预取太多
一个常见的假象：很多人认为 num_workers 越大数据加载越快。实际上，当 worker 数量超过 CPU 核心数时，操作系统会因为频繁的 context switch 而降低整体吞吐量。在 CPU 利用率已经接近 100% 的情况下，增加 worker 只会让情况更糟。正确的调优方法是：用 nvidia-smi 观察 GPU 利用率，如果 GPU 利用率偏低（< 90%），且 CPU 没有跑满，才增加 worker 或 prefetch。

16.7 常见内存陷阱与诊断方法

最后，我们总结一下预训练数据加载中常见的"内存杀手"，以及如何诊断和修复它们。

内存陷阱自查清单：

[ ] 1. 调用了 dataset.to_iterable_dataset() 但没有用流式模式
    → 症状：load_dataset 耗时极长
    → 修复：添加 streaming=True

[ ] 2. map() 函数中返回了包含大对象的列表
    → 症状：map 操作后内存持续增长
    → 修复：用 yield 替代 return，使用 generator

[ ] 3. tokenizer 调用时未设置 return_tensors
    → 症状：tokenizer 输出 Python list，不是 tensor
    → 修复：设置 return_tensors="pt"

[ ] 4. 错误地把整个数据集保存到了变量中
    → 症状：dataset = list(dataset) 或 dataset = [x for x in dataset]
    → 修复：始终使用迭代器模式

[ ] 5. collate_fn 中创建了数据集规模的中间变量
    → 症状：OOM 发生在 collate 阶段
    → 修复：只在 collate_fn 中处理当前 batch

[ ] 6. 多个 DataLoader worker 共享了同一个数据集句柄
    → 症状：S3 限额被消耗殆尽
    → 修复：每个 worker 独立打开文件句柄

[ ] 7. 没有设置 persistent_workers=True
    → 症状：每个 epoch 结束时 OOM
    → 修复：persistent_workers=True 避免重复创建

[ ] 8. 使用了过多的 shuffle buffer
    → 症状：启动后内存急速上升
    → 修复：buffer_size 设为一个合理的值 (如 10000)

诊断内存问题时，我最常用的工具有三个：

  • psutil（Python 库）：在训练循环中打印进程的 RSS 内存使用，定位内存增长的"拐点"。
  • tracemalloc（Python 标准库）：追踪每一行 Python 代码分配了多少内存，精确定位泄漏源头。
  • /proc/[pid]/smaps（Linux）：查看进程的虚拟内存映射，分辨 RSS 和 VSZ 的区别，发现 mmap 是否正确。
调试要诀：遇到 OOM 时，先确认是"虚拟内存不足"还是"物理内存不足"。Linux 允许虚拟内存超出物理内存（通过 swap），但 LLM 训练绝不应该依赖 swap——一旦触发 swap，训练速度会跌到原始速度的 1/100 甚至更慢。如果发现 swap 使用量持续增长，请立即检查代码中是否存在数据集的"完整拷贝"操作。

本章小结

流式加载和内存管理是规模化预训练数据工程中"看不见的暗礁"。从 Hugging Face Datasets streaming 到 WebDataset、从 Arrow 的 memory-mapped I/O 到 Sequence Packing、从多 worker prefetch 调优到内存陷阱诊断——每一个细节都可能在千万亿次的训练迭代中被放大为灾难。核心原则只有一条：永远不要假设数据"已经装进内存了"，而是假设数据"还在硬盘上"，只在需要的那一刻才加载它。
