第5章 传感网协议栈

大家好，我是你们的讲师。前几章我们分别学习了传感器（感知层）、无线通信（物理层）和网络拓扑（网络组织方式）。现在是时候把所有东西串起来了——这就要靠协议栈（Protocol Stack）。

协议栈这个概念听起来有点"学院派"，但它的核心思想其实很朴素：把复杂的通信问题拆成几个层次，每个层次只解决一个特定问题。就像一家物流公司：快递员（物理层）只负责把包裹送到下一站；分拣中心（网络层）规划包裹走哪条线路；客服（应用层）处理客户的问题——每个岗位各司其职，通过标准接口协作。WSN的协议栈也是同样的逻辑。

但我必须先给你打个"预防针"：WSN的协议栈和你在计算机网络上学的TCP/IP协议栈有显著不同。TCP/IP是为"可靠的有线/高带宽网络"设计的，而WSN是"不可靠的无线/极低带宽网络"。能量是WSN的第一优先级，这在协议栈的每一层都有体现。这一章，我们不走"教科书罗列"的老路子，而是用一个实际的数据包旅行故事带你把五层协议栈完整走一遍——这样你对每一层的职责和相互配合就会有直观的理解。

本章路线： 协议栈的分层逻辑 → 五层概述 → 核心标准（IEEE 802.15.4、6LoWPAN、RPL）→ 跨层优化——为什么WSN需要"打破分层"。

为什么需要协议栈？

设想你设计了一个WSN节点，传感器采集了温度数据。这个数据需要：被正确编码→封装为无线帧→选择合适的信道发送→对端节点确认收到→如果出错要重传→转发给网关→最终被应用读取。如果所有逻辑都写在一起，代码会变得极度复杂且难以维护。

分层协议栈的核心优势：

  • 关注点分离（Separation of Concerns）：每一层只关心自己那一摊事。物理层不用操心"这个数据是温度还是振动"；应用层不用关心"数据是用DSSS还是O-QPSK调制的"。
  • 模块化开发：替换物理层模块（比如从2.4GHz换到868MHz）不影响上层的协议代码——只要接口保持不变。
  • 标准化互操作：不同厂商的设备只要遵循相同的协议栈标准，就可以互相通信。正是标准的协议栈促成了今天万亿级的IoT设备生态系统。

WSN五层协议栈

WSN的协议栈沿用了OSI七层模型的简化版本——去掉了表示层和会话层，聚焦于五层结构。下表展示了每一层的功能、关键约束和代表性协议：

层次 核心职责 WSN特殊约束 代表性协议/标准
应用层 数据聚合、查询接口、事件检测、安全认证 极简应用层协议，避免HTTP等重量级方案 CoAP, MQTT-SN, LWM2M, AMQP
传输层 端到端可靠性、拥塞控制 避免TCP的开销（三次握手、窗口管理），通常直接用UDP UDP, DTLS, STCP, PORT
网络层 路由发现与维护、数据转发、寻址 低开销路由协议，以数据为中心的寻址 RPL, LOADng, AODV, LEACH, GPSR
MAC层 信道访问控制、帧封装、错误检测、CSMA/CA 极低的占空比，避免空闲监听（主要耗电源） IEEE 802.15.4 MAC, TSCH, CSL
物理层 射频收发、调制解调、信号/能量检测 低功耗射频设计，多种调制方式可选 IEEE 802.15.4 PHY, LoRa PHY, NB-IoT

这张表重要的是"WSN特殊约束"那一列——它清楚地告诉你：WSN协议栈不是"瘦身版的TCP/IP"，而是为能量效率和资源约束从头设计的。比如传输层：TCP需要建立连接（SYN-SYN/ACK-ACK三次握手），仅握手包就要40-60字节；而一个WSN节点的RAM总共只有8KB——TCP的一个连接状态表可能就吃掉几百字节。所以WSN传输层几乎总是选择UDP，并在应用层实现精简的重传逻辑（或者根本不需要可靠传输——有些环境监测应用丢几个包也没关系）。

一个数据包的旅行

让我们跟随一个温度读数从传感器节点到网关的旅途，理解每一层对它做了什么：

// 协议栈逐层封装过程示意
// 传感器节点：发送"temp=25.3℃" 到网关

[应用层]    温度读数 25.3℃
            → 按CoAP格式封装：
              POST /sensor/temp 25.3
            ↓

[传输层]    使用UDP（无需连接）
            → 添加UDP头部：
              源端口=5683（CoAP默认端口）
              目的端口=5683
            ↓

[网络层]    RPL路由协议决定下一跳
            → 添加IPv6头部（经6LoWPAN压缩）：
              源地址=FE80::1234（链路本地地址）
              目的地址=FE80::5678（网关地址）
            → 添加RPL路由扩展头（可选）
            ↓

[MAC层]    IEEE 802.15.4 CSMA/CA信道接入
            → 添加MAC帧头：
              帧控制字段（帧类型=数据帧，ACK请求=1）
              序列号=42
              源PAN ID=0x0001
              目的PAN ID=0x0001
              源地址=0x1234（16位短地址）
              目的地址=0x5678
            → 添加FCS帧校验序列（CRC-16）
            ↓

[物理层]    发送前先做CCA（空闲信道评估）
            → 用O-QPSK调制成无线信号
            → 在2.405GHz（信道11）上发送
            → 发送完成后等待ACK
            → 若无ACK则在退避后重传
            ↓

[接收端]    逆向操作：物理层解调 → MAC验FCS →
            网络层检查RPL路由 → UDP解包 →
            CoAP解析 → 温度数据提取

在这个"旅行"中，特别值得注意的是6LoWPAN层的IPv6压缩——这是WSN协议栈的核心创新之一。标准的IPv6头部是40字节，而一个IEEE 802.15.4帧最大载荷才102字节（2.4GHz频段）——如果放一个完整的IPv6头部，上层数据就没剩多少空间了。6LoWPAN通过上下文感知压缩，将40字节的IPv6头压缩到2-12字节，效率提升了一个数量级。

补充知识： 6LoWPAN压缩靠什么节省空间？主要靠"假设缺失即默认"——链路本地地址的前缀（FE80::/10）被省略、接口ID从MAC地址推导、流标签和流量等级取默认值。收发双方通过上下文共享这些"背景知识"来压缩和解压IPv6头。如果你在学校学过"信息论"，6LoWPAN就是利用先验知识进行熵编码的一个绝佳工程实例。

IEEE 802.15.4标准族

IEEE 802.15.4是WSN领域最重要、引用最广泛的基础标准。它定义了低速无线个人区域网（LR-WPAN）的物理层和MAC层规范，是ZigBee、6LoWPAN、Thread、WirelessHART、ISA 100.11a等众多上层协议的基础。

802.15.4（2003/2006/2011）

最初的版本。定义了2.4GHz（全球，16个信道，250 kbps）、868MHz（欧洲，1个信道，20 kbps）和915MHz（北美，10个信道，40 kbps）三个频段的物理层。MAC层采用CSMA/CA信道接入，支持星型和点对点拓扑。这是"经典WSN"的起跑线。

802.15.4e（2012）— MAC层革命

这是WSN标准发展史上里程碑式的一步。802.15.4e对MAC层进行了彻底改造，引入了多种新的MAC行为模式（称为"MAC Behaviors"）。最重要的两个：

  • TSCH（Time-Slotted Channel Hopping）：时隙化+信道跳频。节点在固定时隙内通信，每个时隙跳到不同信道上。这解决了工业场景中最头疼的两个问题——确定性延迟和抗干扰。TSCH是WirelessHART和ISA 100.11a的核心机制，也是后来6TiSCH标准的基础。
  • DSME（Deterministic and Synchronous Multi-channel Extension）：在信标使能模式下支持多信道通信，提供保障时隙（GTS）和竞争接入两种模式的组合。

802.15.4g（2012）— 智能电网专用

为智慧电网（Smart Utility Networks, SUN）设计的物理层扩展。支持多种调制方式（MR-FSK、MR-OFDM、MR-O-QPSK），频段覆盖470MHz到2.4GHz。MR-FSK在Sub-1GHz频段可达到200 kbps的数据率，远高于经典802.15.4的20-40kbps。中国的Wi-SUN（基于802.15.4g）正在大规模应用于电力采集和水务计量。

IETF标准：6LoWPAN与RPL

IEEE 802.15.4定义了物理层和MAC层。但要让WSN真正与互联网互通，还需要网络层和传输层的标准化——这就是IETF（互联网工程任务组）的工作。

6LoWPAN（RFC 4944 / 6282）

全称"IPv6 over Low-Power Wireless Personal Area Networks"。核心思想：把IPv6数据包压缩后通过802.15.4网络传输。6LoWPAN不是一个"新的网络层协议"，而是位于IP层和MAC层之间的适配层。它处理IPv6包的分片与重组（当IPv6包超过802.15.4帧的最大载荷时）、头部压缩（如前所述）、地址映射（IPv6地址与16位短地址的双向映射）。有了6LoWPAN，任何支持IPv6的互联网应用都可以直接与WSN传感器通信——不需要协议转换网关。

RPL（RFC 6550）

全称"IPv6 Routing Protocol for Low-Power and Lossy Networks"。这是IETF为LLN（Low-power and Lossy Networks）设计的专用路由协议。RPL的核心是一个有向无环图（DODAG, Destination-Oriented DAG）——所有节点形成一棵以网关为根的路由树，数据沿树向上（汇聚到根）或向下（从根分发到终端）。RPL支持多路径、路由修复和负载均衡，是当前WSN网络层事实上的标准协议。

跨层优化——打破分层

我们前面花了大量篇幅强调"分层"的好处。但现在我要告诉你一个WSN中的"公开秘密"：严格的分层在WSN中往往不是最优的。

为什么？因为分层的一个核心假设是"层与层之间不需要互相了解内部状态"。但WSN的资源极度受限——每一层都需要了解其他层的状态才能做出最优决策。举两个例子：

例1：传输功率与路由选择。 网络层选择下一跳节点时，如果知道物理层可以调整发射功率，那么它可以做更精细的路由选择——对近的邻居用低功率、对远的邻居用高功率。但在严格分层中，网络层只知道"该节点可达"，不知道"到达它需要多少能量"。

例2：MAC层状态感知路由。 路由协议在做路由决策时，如果知道MAC层当前的信道质量（如LQI值），可以避开受干扰严重的链路。但严格分层禁止了这种"跨层信息共享"。

这就是跨层优化（Cross-Layer Optimization, CLO）的由来。CLO的核心思想是：打破严格的层间边界，允许相邻（甚至非相邻）层之间直接交换状态信息，以提升整体效率。

常见的跨层优化方案：

  • AMC（Adaptive Modulation and Coding）：物理层根据信道质量（SNR）动态调整调制编码方式——信道好时用高速率16-QAM，差时用稳健的BPSK。这需要物理层向MAC层报告信道质量，MAC层据此调整数据包大小和重传策略。
  • JAM（Joint MAC-Routing）：MAC层维护邻居链路质量信息，路由层直接使用这些信息来做路由决策，避免额外的"路由探测包"开销。
  • PLM（Power-Aware Routing）：物理层将发送单位比特的能量消耗共享给网络层，网络层选择能效最高的路径而非跳数最少的路径。
跨层优化的"度"在哪里？ 跨层优化虽然高效，但也带来了严重的工程问题：紧耦合导致代码不可移植。如果你做了一个跨层优化——例如MAC层和网络层混在一起写了——以后想换一种MAC协议（比如从CSMA/CA换成TSCH），整个网络层要一起重写。所以在工程实践中，建议的做法是：保持分层接口，但增加"信息通告"机制——各层通过信息通告接口共享关键状态，而不是把各层的代码混在一起。这样既获得了跨层优化的好处，又保留了模块化设计的灵活性。

协议栈是WSN的"软件骨架"——它把传感器的裸数据、无线通信的原始比特流转换成了有意义的应用信息服务。这一章我们沿着"数据包旅行"的路径走了一遍五层协议栈，然后深入讨论了IEEE 802.15.4标准族、6LoWPAN和RPL这些关键标准，最后探讨了跨层优化这个设计与工程之间的平衡问题。

从下一章开始，我们将进入一个更"动手"的阶段——嵌入式系统基础。要理解WSN，你不可能永远停留在"网络协议"的层次上，最终你要在真实的嵌入式硬件上运行代码。第6章将带你深入了解WSN节点的MCU架构、实时操作系统和低功耗设计——这是从"协议理解"到"系统开发"的关键一步。
