第23章 OpenSSL命令行实战

学了大半本密码学的理论，你可能一直在想一个问题：这些东西到底怎么在我的电脑上跑起来？RSA 的密钥长什么样？证书文件里到底存了什么？TLS 握手时客户端和服务器到底互相发了什么？这一章我们就来动手——打开终端，用 OpenSSL 把一个真实的密码学工具箱交到你手里。

OpenSSL 是地球上使用最广泛的密码学软件库。无论是 Nginx、Apache 还是 curl，几乎所有用 TLS 的软件底层都在调用它。但很多人只把它当作一个"装好就不用管"的依赖——直到某天需要生成一个自签名证书，或者排查一条 "certificate verify failed" 错误，才手忙脚乱地去百度命令。本章会让你从"会用"变成"懂用"。

23.1 OpenSSL 的前世今生

OpenSSL 诞生于 1998 年，源自对 Netscape 的 SSL 实现的一个开源分支。二十多年来，它逐渐演变为一个超过 50 万行 C 代码的庞然大物，支持几十种密码算法、协议和编码格式。但成也萧何败也萧何——代码库过于庞大，维护人手不足，埋下了不少安全隐患。

分支 开发者 核心特点 主要用户
OpenSSL 开源社区 最完整的功能，最广泛的使用 几乎所有 Linux 发行版
LibreSSL OpenBSD 团队 从 OpenSSL fork 出来，大幅精简代码 OpenBSD、macOS（曾短暂使用）
BoringSSL Google 为 Chrome/Android 定制，去掉不用的算法 Google 系产品（Chrome、Android）

2014 年 Heartbleed 漏洞爆发后，业界对 OpenSSL 的代码质量产生了严重质疑，于是催生了 LibreSSL 和 BoringSSL 这两个分支。不过 OpenSSL 仍然是服务器端的事实标准——大多数 Linux 发行版都预装了它。让我们开始实战。

23.2 生成密钥对

打开终端，先检查你的 OpenSSL 版本：

openssl version
# OpenSSL 3.0.12 24 Oct 2023 (Library: OpenSSL 3.0.12 24 Oct 2023)

生成 RSA 密钥对是最基础的操作。下面这条命令生成一个 2048 位的 RSA 私钥：

openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:2048 \
    -out private_key.pem

如果你希望私钥在磁盘上是加密存储的（推荐），加上 -aes-256-cbc 参数：

openssl genpkey -algorithm RSA -aes-256-cbc \
    -pkeyopt rsa_keygen_bits:2048 -out private_key_enc.pem

用 openssl pkey 查看私钥的详细信息：

openssl pkey -in private_key.pem -text -noout

你会看到私钥中包含了模数 n、公钥指数 e、私钥指数 d，以及 CRT 优化参数 p、q、dp、dq、qinv。一个 RSA 私钥文件实际上存储了完整的密钥对——公钥可以从私钥中提取出来：

openssl pkey -in private_key.pem -pubout -out public_key.pem

生成 ECC 密钥对则更简洁，因为椭圆曲线的参数是标准化的：

# 列出所有支持的曲线
openssl ecparam -list_curves

# 生成 prime256v1（即 secp256r1）密钥对
openssl ecparam -genkey -name prime256v1 -out ecc_private.pem

# 提取公钥
openssl ec -in ecc_private.pem -pubout -out ecc_public.pem
💡 实践建议：RSA 2048 位是目前兼容性最好的选择——几乎所有的客户端都支持。但如果你追求性能和安全性的平衡，ECC（特别是 prime256v1 或 X25519）是更好的选择。典型的做法是：证书用 RSA 2048 保证兼容性，TLS 密钥交换用 ECC 提供前向安全性。

23.3 自签名证书与 CSR

有了密钥对之后，下一步是生成证书。证书的本质就是把你的公钥和身份信息（域名、组织名等）绑定在一起，并由一个权威机构（CA）签名。但是在开发和内网场景中，我们可以自己给自己签名——这就是自签名证书。

# 一步生成自签名证书（包含私钥）
openssl req -x509 -newkey rsa:2048 -nodes \
    -keyout server_key.pem -out server_cert.pem \
    -days 365 -subj "/CN=localhost/O=MyOrg/C=CN"

如果你需要提交给真正的 CA 签名（比如 Let's Encrypt），则需要先生成证书签名请求（CSR）：

# 先生成私钥
openssl genpkey -algorithm RSA -out myapp.key

# 然后生成 CSR
openssl req -new -key myapp.key -out myapp.csr \
    -subj "/CN=myapp.example.com/O=MyCompany/C=CN"

# 查看 CSR 的内容
openssl req -in myapp.csr -text -noout

CSR 文件是一个 Base64 编码的 PKCS#10 格式对象，里面包含了你的公钥和身份信息，以及用你的私钥对这些信息的签名——CA 用 CSR 里的公钥验证签名，确认你确实拥有对应的私钥。

📌 重要：自签名证书在浏览器中会显示为"不安全"，因为浏览器不信任你的根证书。解决方法有两种：（1）在开发环境中将自签名证书导入为受信任的根证书；（2）使用 mkcert 这样的工具自动完成这个过程。千万不要在生产环境使用自签名证书。

23.4 搭建迷你 CA

理解了 CA 的工作原理之后，你可以用 OpenSSL 搭建一个自己的迷你 CA。这在企业内部网或开发测试环境中非常有用。

# 第一步：生成 CA 根密钥和根证书
openssl genpkey -algorithm RSA -out ca.key
openssl req -x509 -new -key ca.key -out ca.crt \
    -days 3650 -subj "/CN=MyMiniCA/O=DevRoot/C=CN"

然后，用这个根证书来签发自建网站的证书：

# 生成服务器私钥和 CSR
openssl genpkey -algorithm RSA -out server.key
openssl req -new -key server.key -out server.csr \
    -subj "/CN=dev.local/O=MyApp/C=CN"

# 用 CA 根证书签名
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key \
    -CAcreateserial -out server.crt -days 365

# 查看签名后的证书
openssl x509 -in server.crt -text -noout | grep "Subject\|Issuer\|Not Before"

签名后的证书中，Issuer（颁发者）是 "MyMiniCA"，Subject（主题）是 "dev.local"。浏览器收到这个证书后，会检查它是否由某个受信任的 CA 签发——如果你把 ca.crt 导入到浏览器的受信任根证书列表里，这个自建证书就不会再报安全警告了。

23.5 TLS 服务器配置

生成完证书后，我们来配置一个真正的 TLS 服务器。OpenSSL 内置了一个 s_server 命令，可以快速启动一个测试服务器：

# 启动 TLS 服务器（监听 4433 端口）
openssl s_server -accept 4433 -cert server.crt -key server.key \
    -www -tls1_3

# 另开一个终端，用 s_client 连接
openssl s_client -connect localhost:4433 -showcerts

s_client 是一个非常强大的调试工具。连接成功后，它会输出完整的握手信息：

  • 证书链（Certificate chain）——服务器发送的证书及其签发者信息
  • SSL 会话参数——协商使用的 TLS 版本、密码套件、会话 ID
  • 服务器证书详情——包括 Subject、Issuer、有效期等

常用的 s_client 调试参数：

# 检查远程服务器的证书信息
openssl s_client -connect example.com:443 -showcerts

# 只检查证书链是否有效
openssl s_client -connect example.com:443 -verify_return_error

# 指定密码套件进行连接
openssl s_client -connect example.com:443 \
    -ciphersuites "TLS_AES_256_GCM_SHA384"

# 检查是否支持 TLS 1.2
openssl s_client -connect example.com:443 -tls1_2
⚠️ 安全警告：在配置 TLS 服务器时，建议遵循以下原则：（1）禁用 SSLv3、TLS 1.0 和 TLS 1.1——它们都有已知的严重漏洞；（2）优先使用 TLS 1.3，它在握手性能和安全性上都有质的提升；（3）密码套件优先使用 AEAD 模式（GCM 或 ChaCha20-Poly1305）；（4）启用 HSTS 和证书透明度（Certificate Transparency）。

23.6 文件加密与解密

OpenSSL 除了处理证书和 TLS，还可以直接对文件进行加解密。一个典型的场景是用公钥加密文件，只有持有对应私钥的人才能解密：

# 第一步：生成对称密钥
openssl rand -out session_key.bin 32

# 第二步：用公钥加密对称密钥
openssl rsautl -encrypt -in session_key.bin \
    -pubin -inkey public_key.pem -out session_key.enc

# 第三步：用对称密钥加密大文件
openssl enc -aes-256-gcm -in secret.txt \
    -out secret.enc -pass file:session_key.bin

# 解密时反向操作
openssl rsautl -decrypt -in session_key.enc \
    -inkey private_key.pem -out session_key_dec.bin
openssl enc -d -aes-256-gcm -in secret.enc \
    -out secret_dec.txt -pass file:session_key_dec.bin

这种"混合加密"模式就是 TLS 握手的本质——用非对称密码交换对称密钥，然后用对称密码加密批量数据。RSA 无法加密超过密钥长度的数据（2048 位 RSA 最多加密 245 字节），所以绝不能直接用公钥加密大文件。

23.7 踩坑故事：Heartbleed 漏洞

2014 年 4 月 7 日，安全研究员 Neel Mehta 公开了 OpenSSL 中一个被命名为 Heartbleed（CVE-2014-0160）的漏洞。这个漏洞的影响之大，超过了当时几乎所有人的想象。

🩸 Heartbleed 漏洞原理：OpenSSL 实现了 TLS 的心跳扩展（Heartbeat），客户端发送一个心跳请求，服务器需要回复同样的数据来证明连接还活着。问题是——客户端在请求中声明了 "payload length"，但没有验证这个长度与实际发送的数据是否一致。攻击者可以声称 payload length 为 65535 字节，而实际只发送 1 个字节。服务器就会读取内存中紧跟在 1 字节后面的 65534 字节，并把它们原样返回给攻击者。

这个"读取内存越界"的问题意味着：攻击者可以通过反复发送恶意心跳包，逐步泄露服务器内存中的私钥、会话 Cookie、用户密码、甚至数据库查询内容。更可怕的是，这个过程不会在服务器日志中留下任何痕迹——心跳请求本身是完全合法的。

据估计，当时互联网上大约有 17% 的 HTTPS 服务器（约 50 万台）受到了影响。包括 Yahoo、Stack Overflow、GitHub 等巨头在内的网站都紧急更新了 OpenSSL 并更换了私钥。这个漏洞的发现直接催生了 Linux 基金会主导的 Core Infrastructure Initiative，为 OpenSSL 等关键基础设施项目提供资金支持——一个单字符级别的内存越界读取，让全世界重新审视了开源安全的经济学问题。

🔑 教训：Heartbleed 告诉我们三件事：（1）用 C 语言实现密码学协议极其危险，因为内存安全完全依赖程序员的谨慎；（2）代码审查和自动化测试对于安全关键项目不可或缺；（3）"广泛使用"不等于"安全审查充分"，OpenSSL 在 2014 年之前被数亿人使用，但核心代码的审查者寥寥无几。
