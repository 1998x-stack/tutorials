GPU硬件架构概览

大家好，我是你们的讲师。上一章我们搭好了开发环境，你现在应该在终端里敲 nvcc --version 能看到输出了。但在写第一行CUDA代码之前，我们需要先回答一个关键问题：GPU内部到底长什么样？

很多初学者犯的一个错误是：把GPU当作"更快一点的CPU"来编程。你把for循环拆成线程，以为线程越多就越快——结果发现性能不如预期，甚至更慢。原因很简单：GPU和CPU的架构设计哲学完全不同。不理解硬件就把代码往上堆，等于蒙着眼开车。这一章，我们打开引擎盖，看看GPU这颗"计算引擎"的内部构造。

SM（流式多处理器）内部结构

GPU的核心构成单元是SM（Streaming Multiprocessor，流式多处理器）。你可以把SM理解为一个"迷你计算单元"——每个SM内部有自己的CUDA核心、寄存器文件、共享内存、warp调度器，以及可选的Tensor Core。一张GPU芯片由几十到数百个这样的SM组成，它们并行工作，构成GPU强大的计算能力。

下图描述了一个典型SM的内部布局（以Ampere架构GA100为例），虽然没有真正的图片，但这个结构图能帮你在脑海中建立印象：

组件 作用 对开发者的意义
Warp调度器 管理warp的执行顺序，每个周期选择一个待执行的warp发射指令 SM内有多个调度器意味着可以隐藏延迟——一个warp在等数据时换另一个上
CUDA Core 执行标量浮点和整数运算的基本单元 我们写的kernel大部分指令都在这里执行

一个关键数字： 在A100上，每个SM有65536个寄存器。听起来很多，但如果每个线程用64个寄存器，一个SM上的2048个线程就需要131072个寄存器——超出部分只能"溢出"（spill）到L1缓存甚至全局内存，性能会下降一个数量级。所以，减少每个线程的寄存器使用量是优化的重要方向。

内存层次全景图

GPU的内存层次是理解"为什么GPU编程需要特殊技巧"的核心。从最快到最慢，GPU的内存体系如下：

内存类型 延迟 带宽 作用域 容量
寄存器 ~1 cycle 极高 单个线程 255个/线程（最大）
共享内存 ~5 cycles 极高 单个线程块 ~48-164 KB/SM
L1缓存 ~5-10 cycles ~TB/s级 单个SM ~128-256 KB/SM
L2缓存 ~200 cycles ~TB/s级 整颗GPU芯片 ~6-40 MB
全局内存 ~400-800 cycles ~2 TB/s (H100) 所有线程 + Host ~4-80 GB
寄存器是最快的，但每个线程只有255个——用完了就溢出到local memory。 而local memory本质上存在全局内存里，延迟是寄存器的400倍以上。所以当你写kernel时，如果在函数内声明了一个大数组，表面上它"在寄存器里"，实际上编译后很可能已经被搬到了local memory。使用 --ptxas-options=-v 编译可以看到寄存器溢出情况。

记住一个原则：让数据尽可能待在高速存储中。能放寄存器的放寄存器，需要在线程间共享的放共享内存，最后才是全局内存。这个原则贯穿了CUDA性能优化的方方面面。

计算能力（Compute Capability）详解

CUDA中有一个非常重要的概念叫计算能力（Compute Capability，简称CC），用 主版本.次版本 表示，比如 8.0、8.9。计算能力不是"算力"——它更像一个"Feature Flag集"，告诉你的代码：这块GPU支持哪些硬件特性。下面是最主要架构的CC映射：

架构 CC 代表设备 关键新特性
Tesla 1.0 G80 CUDA诞生
Fermi 2.0 GTX 480 / GTX 580 ECC支持、真正的C++类/继承
Kepler 3.0 / 3.5 GTX 680 / K20 Dynamic Parallelism、Hyper-Q
Maxwell 5.0 / 5.2 GTX 980 / M40 共享内存与L1分离
Pascal 6.0 / 6.1 GTX 1080 / P100 Unified Memory、NVLink
Volta 7.0 V100 Tensor Core 1代、独立线程调度
Turing 7.5 RTX 2080 / T4 RT Core、Turing Tensor Core
Ampere 8.0 / 8.6 / 8.9 A100 / RTX 3090 3代Tensor Core、MIG、BF16、TF32、稀疏化
Ada Lovelace 8.9 RTX 4090 4代Tensor Core、FP8、光流加速器
Hopper 9.0 H100 / H200 Transformer Engine、DPX指令、4代Tensor Core、FP8
Blackwell 10.0 B200 / B100 双芯片设计、FP4、5代NVLink

用 nvcc -arch=sm_XX 指定目标架构。比如要为A100（CC 8.0）编译，用 -arch=sm_80。如果你的代码要兼容多种GPU，可以用 -gencode 生成多个架构的cubin（二进制代码）和PTX（中间代码），让驱动在运行时选择最合适的版本。

向下兼容原则： CUDA有很强的向下兼容性。为CC 7.0编译的代码可以在CC 8.0、9.0的GPU上运行。反过来不行——为CC 9.0编译的二进制不能跑在CC 8.0的卡上。所以如果你要发布一个库给用户，通常选择较低的CC作为目标（比如7.0），确保兼容大多数GPU。

Warp与线程调度

现在我们来谈一个GPU编程中最核心也最容易被误解的概念：Warp。

一个warp由32个线程组成，它是GPU调度和执行的最小单位。你写kernel时启动的成千上万个线程，在硬件层面被组织成warp来管理。每32个连续线程（按 threadIdx 排列）形成一个warp。SM上的warp调度器每个周期选择一个就绪的warp，发射一条指令——这32个线程同时执行相同的指令，操作各自的数据。

这就是SIMT（Single Instruction, Multiple Threads）执行模型。和CPU的SIMD（如AVX-512）不同，SIMT允许每个线程有自己的程序计数器、寄存器和执行路径。但这有一个重要陷阱——warp divergence（warp分支发散）：

// warp divergence 示例：同一个warp内的线程走了不同的分支
__global__ void divergence_kernel(float *data, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        if (data[idx] > 0.0f) {
            // 路径A：正数处理
            data[idx] = sqrtf(data[idx]);
        } else {
            // 路径B：非正数处理
            data[idx] = 0.0f;
        }
    }
}

当同一个warp内的32个线程面对同一个 if 语句时，如果有的走 then 有的走 else，硬件会串行执行两个分支：先执行路径A的同时屏蔽掉走B的线程，再执行路径B的同时屏蔽掉走A的线程。这意味着warp的吞吐量降为原来的一半。

真实的故事：我参与过一个项目，团队花了两周优化一个kernel，各种内存访问都调好了，但性能始终上不去。最后用Nsight看warp分析发现，一个看似无害的 if-else 导致了严重的warp divergence——因为数据是按类别排序的，相邻线程的数据恰好属于不同类别。解决方案很简单：重新排列数据，让相同类别的数据连续存放，这样每个warp内的32个线程走同一个分支。改完性能提升了40%。

这不是说不要用if： 如果32个线程全部走同一分支，就不会有性能损失。问题只出现在同一个warp内的线程走了不同分支。所以一个重要的优化思路是：尽量保证同一个warp内线程的数据访问模式一致——这在CUDA社区被称为"warp-coherent"编程。

Tensor Core简介

最后，简要提一下Tensor Core——它可能是过去十年GPU架构中最重要的创新。Tensor Core是专门为矩阵乘累加（D = A × B + C）设计的硬件单元。在Volta架构（V100）中首次引入，后来每一代都大幅增强。

Tensor Core和CUDA Core的工作方式不同：CUDA Core每次执行一个标量操作（如两个float相加），而Tensor Core一次执行一个小矩阵的乘法——比如在Volta上是4×4×4的FP16矩阵乘法。这个"一次性搞定4×4矩阵乘法"的能力，让Tensor Core的吞吐量比CUDA Core高出8-16倍（取决于精度）。

Tensor Core并非自动使用——你需要在代码中显式调用。有两种方式：

  • 直接编程：通过CUDA C++的warp矩阵函数（nvcuda::wmma 命名空间）手动调用Tensor Core。适合需要精细控制的场景
  • 通过cuBLAS等库：大多数情况下，你不需要直接写Tensor Core代码——cuBLAS、cuDNN等库在底层已经为你做了这件事
Tensor Core的精度注意事项： Tensor Core支持多种精度模式——FP16、BF16、TF32（Ampere引入）、INT8、INT4、FP8（Hopper引入）。精度越低吞吐越高，但你可能需要关注数值精度对模型收敛的影响。其中TF32是个很好的"甜点"：在A100上，TF32 Tensor Core的吞吐量是FP32 CUDA Core的8倍，而精度几乎无损。

好，这一章我们了解了GPU的内部结构——SM、内存层次、计算能力、warp调度和Tensor Core。有了这些硬件知识，下一章我们终于可以写第一个真正的CUDA kernel了——一个简单的向量加法。不会很长，但它将是你亲手写出的第一行在GPU上并行执行的代码。
