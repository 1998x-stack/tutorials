第22章 Agent多步推理数据

大家好，我是你们的讲师。上一章我们学习了函数调用数据——让模型学会"在正确的时机调用正确的工具"。但如果把工具调用看作"说一句话"，那么 Agent 多步推理就是"写一篇论文"——它要求模型在多轮交互中，根据环境的反馈动态调整自己的行动策略，最终完成一个复杂的任务。

这一章要讨论的，就是这种更高级的推理模式——Agent 多步推理数据（Multi-Step Agent Data）。它是后训练数据体系中最复杂、最动态、也最具挑战性的数据类型。

Agent 数据的特殊性——不是 text pair，而是 environment episode

我们先想一个问题：到目前为止，我们讨论的所有数据——SFT 数据、偏好数据、CoT 数据、函数调用数据——它们的结构有什么共同点？

答案是：它们都是静态的、预先确定的。每一条数据在被模型看到之前就已经被标注好了。模型读入一条数据，学习其中的模式，然后进入下一条。数据之间没有依赖关系，也没有"反馈-调整"的动态过程。

Agent 数据完全颠覆了这个模式。Agent 数据不是 text pair，而是 environment episode（环境交互剧集）。一条 Agent 数据是一个完整的交互序列：

Episode = [
  (observation_1, action_1, reward_1),
  (observation_2, action_2, reward_2),
  (observation_3, action_3, reward_3),
  ...
  final_outcome
]

在这个序列中，每一个 action 都会改变环境（environment），而环境的变化又会影响下一步的 observation。Agent 的训练数据不是"一个问题对应一个答案"——它是"一个初始状态对应一条完整的行动轨迹"。这就好比训练一名棋手：你不能只给他看"这个棋局该怎么走"——你要让他真正去下棋，经历输赢，从胜负中学习。

这个区别意味着 Agent 数据的构建方式完全不同的：

  • 你需要一个环境模拟器（比如代码编译器、网页渲染器、游戏引擎）来执行 Agent 的动作并返回新的状态
  • 你需要轨迹采样策略来决定 Agent 如何探索环境——是完全随机探索，还是使用某种启发式搜索
  • 你需要过程奖励（intermediate rewards）来评估每一步行动的质量，而不仅仅是最终任务完成与否

SWE-bench — 2000+ 真实 GitHub Issue + 代码修复

在 Agent 评测和训练数据领域，SWE-bench 是一个绕不开的名字。SWE-bench 由普林斯顿大学的研究人员在 2024 年发布，它包含了 2,000+ 个来自真实 GitHub 仓库的 Issue（问题报告）和对应的代码修复（PR），覆盖 Python 生态中的 12 个流行仓库。

SWE-bench 的特殊之处在于：它不是一个人工构造的玩具数据集，而是来源于真实的软件开发项目。每个样本的结构如下：

  • Issue 描述：用户报告的一个 bug 或功能请求，包含错误信息和复现步骤
  • 代码仓库快照：Issue 提交时的代码状态（包含 bug 的原始代码）
  • 真实修复（Ground Truth）：开发者最终提交的 PR 中修改的代码 diff
  • 测试验证：验证修复是否正确的单元测试用例

对于 Agent 训练而言，SWE-bench 的价值在于它构建了一个完整的"任务定义 → 环境交互 → 方案评估"闭环：

环境初始化：创建一个 Docker 容器，包含指定仓库的指定版本代码

Agent 任务：阅读 Issue 描述，理解 bug 原因，修改代码使其通过所有测试

交互过程：
  Step 1:  Agent 浏览代码仓库（cat, grep, find, ls）
  Step 2:  Agent 定位 bug 位置并理解逻辑
  Step 3:  Agent 编辑文件修复 bug（sed, patch, 或直接写入）
  Step 4:  Agent 运行测试验证修复
  Step 5:  Agent 遇到测试失败 → 回到 Step 2 调整修复方案
  Step 6:  所有测试通过 → 任务完成

评估：Agent 的最终代码 diff 是否与真实修复匹配？
      或者至少是否能通过所有测试用例？

这个流程听起来简单，但实际执行中的 Agent 交互长度可能达到数十甚至上百步——每一步都需要模型做出正确的决策。SWE-bench 是目前衡量代码 Agent 能力的"黄金标准"。

一个关键洞察：在 2024 年初，最好的 AI 编程助手在 SWE-bench 上的准确率只有约 15%。到了 2025-2026 年，最先进的 Agent（如 Claude Code、Devin）已经将这一数字提升到 60% 以上。这个提升的背后，除了模型本身能力的增强之外，一个关键因素是 Agent 训练数据的丰富——越来越多的团队使用 SWE-bench 数据和类似的真实开发轨迹来训练模型，让模型学会了"如何像人类开发者一样阅读代码、定位 bug、修复代码"。

Agent 轨迹数据的结构

理解了 SWE-bench 的设定之后，让我们系统性地分析 Agent 轨迹数据的标准结构。一条完整的 Agent 轨迹包含以下要素：

{
  "task_id": "django__django-15202",
  "task_description": "When using MySQL backend...",
  "repository": "django/django",
  "base_commit": "a1b2c3d4e5f6...",

  "trajectory": [
    {
      "step": 1,
      "observation": {
        "type": "env_initialized",
        "working_dir": "/repo",
        "available_commands": ["cat", "grep", "find", "python", "sed"]
      },
      "action": {
        "type": "bash",
        "command": "grep -r 'MySQL' django/db/backends/mysql/ --include='*.py'"
      },
      "reward": 0.0,
      "feedback": "Found 12 files related to MySQL backend"
    },
    {
      "step": 2,
      "observation": {
        "type": "bash_output",
        "stdout": "django/db/backends/mysql/operations.py\n...",
        "exit_code": 0
      },
      "action": {
        "type": "read",
        "file": "django/db/backends/mysql/operations.py",
        "lines": "100-150"
      },
      "reward": 0.0,
      "feedback": "Reading lines 100-150 of operations.py"
    },
    // ... 更多步骤，包括定位 bug、编写修复、运行测试
    {
      "step": 25,
      "observation": { "type": "test_result", "passed": 12, "failed": 0 },
      "action": { "type": "finalize", "diff": "--- a/...\n+++ b/...\n@@ ..." },
      "reward": 1.0,
      "feedback": "All 12 tests passed. Task completed."
    }
  ],
  "final_outcome": "success",
  "total_steps": 25
}

这个结构揭示了 Agent 数据的几个关键特征：

  • 状态空间明确：每一步都有一个明确的 observation（环境状态），action 的选择依赖于这个 observation
  • 动作空间受限：Agent 不是"自由输出文本"，而是在一个受限的动作空间中选择（如 bash 命令、文件编辑、代码搜索）
  • 奖励是过程性的：大部分中间步骤的奖励为 0，只有到达关键里程碑或最终完成任务时才会获得正奖励——这就是"稀疏奖励"问题（sparse reward problem）
  • 轨迹长度可变：简单任务可能只需要 5 步，复杂任务可能需要 50 步甚至更多
理解稀疏奖励的挑战：在 Agent 训练中，"稀疏奖励"是一个核心难题。想象一个需要 30 步才能完成的任务——前 29 步的奖励全部是 0，只有第 30 步成功后才能获得奖励。模型怎么知道"第 5 步的那个搜索动作是正确的"？答案通常有两种：一是通过大量轨迹采样，让模型从"成功轨迹"和"失败轨迹"的对比中学习；二是引入过程奖励模型（Process Reward Model），对轨迹中的每个中间步骤进行自动打分——这一步距离目标更近了吗？更远了吗？有了过程奖励，训练信号就不再是稀疏的，而是密集的。

Agent 数据的奖励设计

Agent 数据的"奖励"比简单的"对/错"要复杂得多。一个好的 Agent 奖励函数通常考虑以下维度：

奖励维度 说明 示例
任务完成度 是否成功完成了预定义的任务目标 修复了 bug 且所有测试通过 → +1.0
工具调用合法性 调用的工具是否存在、参数是否正确 使用不存在的参数名 → -0.5
步骤效率 是否以最少的步骤完成任务 不必要的重复搜索 → -0.1/次
安全边界遵守 是否在权限范围内操作，没有越界 试图删除系统文件 → -2.0
结果准确性 最终产出是否精确地符合要求 代码 diff 恰好修复了问题 → +0.5

在实际系统中，这些维度的权重需要精心调校。一个常见的问题是：如果任务完成度权重过高，模型会"不择手段"地完成任务——即使这意味着生成大量无意义的中介步骤或牺牲代码质量。反之，如果效率权重过高，模型会倾向于走捷径，可能错过更稳健的解决方案。

Agent 数据与普通 SFT 数据的根本区别对比表

为了方便你回顾和总结，下面这张对比表清晰地区分了 Agent 数据与传统 SFT 数据的核心差异：

维度 普通 SFT 数据 Agent 多步数据
数据结构 静态的 instruction-response pair 动态的 episode（状态-动作-奖励序列）
数据属性 确定性——标注好了就不再变化 动态性——后续状态依赖于前面动作的选择
推理步数 单步——输入 → 输出 多步——每一步都可能是推理决策节点
反馈机制 无反馈——训练时看不到用户评价 反馈驱动——每一步都有环境反馈（工具输出/错误信息）
数据生成 标注完成即可，一次性使用 需要环境模拟器，可能需要在线采样
奖励信号 隐含于正确回答中（交叉熵） 显式奖励——需要精心设计奖励函数
泛化方式 模式匹配——学习常见任务的回答模式 策略学习——学习在未知环境中做决策的通用策略

环境交互数据生成策略

生成高质量的 Agent 训练数据，最关键的是避免过拟合到特定的交互模式。如果 Agent 只学会了一种交互模式（比如"先用 grep 搜索，再用 cat 查看，最后用 sed 编辑"），当遇到一个需要"先用 find 定位目录结构"的场景时，它就会手足无措。

为了解决这个问题，业界提出了多脚手架（Multi-Scaffold）策略：在生成 Agent 训练数据时，使用多种不同的"脚手架"（scaffold）来定义 Agent 的决策循环方式，确保模型的 Agent 能力是通用的：

  • ReAct 脚手架：Thought → Action → Observation 的循环。最经典的 Agent 范式，适用性最广。
  • Plan-then-Edit 脚手架：先制定完整计划，然后逐步执行编辑。适合复杂的代码修改任务，减少过程中的犹豫和回溯。
  • Test-Driven Loop 脚手架：先写测试，再写实现，然后验证测试通过。受 TDD 开发方法论启发，适合训练模型"以测试验证驱动开发"的习惯。

实际使用中，最佳实践是混合使用多种脚手架来生成训练数据，并在评测时用不同的脚手架来测试 Agent——如果模型只在 ReAct 模式下表现好，但在 Plan-then-Edit 模式下表现差，说明它的 Agent 能力还不够通用，需要补充更多样化的训练数据。

数据生产实战建议：构建 Agent 训练数据最有效的方式是"基于成功轨迹的离线学习 + 基于弱模型的探索数据"。具体来说：先用一个强大的模型（如 Claude 或者 GPT-4o）在环境中探索，只保留"成功完成任务"的轨迹作为正样本。然后，为了增加数据的多样性，用一些较弱的模型或在不同的脚手架配置下采样的数据来补充——这些数据中可以包含部分成功的轨迹或虽然失败但有局部正确决策的轨迹。这两种数据混合训练，既保证了正样本的质量，又提供了丰富的"失败案例"供模型学习。我们在第 28 章的综合实战中会详细演示这个过程。

这一章的内容应该说信息量很大。我们认识了 Agent 数据的"环境交互剧集"本质，深入研究了 SWE-bench 这个代码 Agent 的黄金基准，分析了 Agent 轨迹数据的标准结构和奖励设计原则，并通过对比表清晰地看到了 Agent 数据与普通 SFT 数据的根本区别。最后，我们还探讨了"多脚手架"数据生成策略，以避免 Agent 能力过拟合到特定交互模式。

完成了第 19-22 章的学习，你已经掌握了"后训练数据"体系中最为复杂和前沿的部分——从 CoT 推理数据到 GRPO 强化学习，从工具调用到 Agent 多步交互。这些能力让模型从一个"被动回答的问答系统"进化成了一个"主动思考、运用工具、多步推理的智能体"。

然而，高质量数据的构建远不止"设计"和"生成"这么简单。你还需要一整套工程化的框架来确保数据的可靠性、可复现性、和可追溯性。下一章，我们将进入第四部分的内容——数据质量管线搭建。
