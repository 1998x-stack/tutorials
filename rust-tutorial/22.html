开篇：安全世界的"核按钮"

本书的前21章都在"安全Rust"的框架内——编译器像一个经验丰富的审查员，逐行检查你的代码，确保没有悬空指针、没有数据竞争、没有use-after-free。但现实世界有安全Rust无法表达的需求：调用操作系统API需要C ABI的函数签名、与硬件寄存器交互需要特定的内存布局、实现高性能的集合需要自定义内存管理。这时，我们需要按下Rust的"核按钮"——unsafe关键字。

不过在开始之前，请先记住一个最重要的原则：unsafe并不意味着代码不安全。它的真正含义是：编译器放弃了对这块代码的安全检查，把责任转交给你——程序员。unsafe代码需要你自己保证内存安全、引用有效性、数据竞争等所有条件。标准库中大量的安全抽象（Vec、Box、Arc等）底层都有unsafe代码——但它们对外暴露的是安全的API。这就是Rust安全抽象的核心模式：用unsafe构建内部实现，用安全API包裹外部接口。

CRITICAL — 关于未定义行为（Undefined Behavior）：Rust标准中对unsafe代码的约束有一长串——解引用空指针、创建无效的枚举值、使用不对齐的引用等都会触发未定义行为。UB最可怕的地方在于：任何行为都是合法的——程序可能立刻崩溃，也可能看似正常运行，然后在你上线三个月后一个深夜突然崩溃，数据损坏。C/C++开发者对这些不陌生，但Rust的unsafe并未免除这些风险。一旦你在unsafe块中犯了错，编译器就不再是你的后盾了。

什么时候需要unsafe — 四种特权操作

unsafe关键字赋予你四种普通Rust中不允许的"超级权限"：

操作 描述 风险级别 典型场景
解引用裸指针 *const T 或 *mut T 的取值操作 ⭐⭐⭐ 与C代码交互、手动内存管理
调用unsafe函数 调用标记为unsafe的函数或方法 ⭐⭐ FFI绑定、标准库底层函数
访问/修改可变静态变量 对static mut的读写 ⭐⭐⭐ 全局状态、C兼容的全局变量
实现unsafe trait 如Send、Sync的unsafe实现 ⭐⭐ 自定义并发原语

请注意：unsafe 不关闭借用检查。在unsafe块中，所有权规则、生命周期检查、借用规则仍然生效。它只开放了上述四种操作。这是一个常见的误解——认为unsafe是完全的"法外之地"——其实不是。

裸指针 — *const T / *mut T

裸指针（raw pointer）是Rust中最接近C指针的类型。与引用相比，它没有编译期保障：可以是空指针、可以指向无效内存、不自动处理生命周期。

创建和解引用

fn main() {
    let mut x = 42;

    // 从引用创建裸指针——安全操作
    let r1 = &x as *const i32;
    let r2 = &mut x as *mut i32;

    // 解引用裸指针——需要在 unsafe 块中
    unsafe {
        println!("r1 指向的值: {}", *r1);
        *r2 = 100;
        println!("修改后: {}", x); // 100
    }
}

注意创建裸指针本身是安全的——"违规"发生在你解引用的时候。这就是为什么as *const T可以在安全代码中执行——真正危险的操作是*r1。

与引用对比

特性 &T / &mut T *const T / *mut T
可为空 否（永远是有效引用） 是（可以是null）
自动解引用 是 否（必须手动解引用）
生命周期检查 是 无
别名规则 严格（不可变/可变互斥） 无约束
发送到其他线程 受限（Sync trait） 无约束
安全解引用 是 仅在unsafe块中

安全保证的自我约束

在unsafe中使用裸指针时，你必须手动确保以下条件：

  • 有效性：指针指向的内存必须是已分配且对齐的，不能解引用悬垂指针
  • 不重叠：遵守Rust的别名规则——不要同时有可变和不可变裸指针指向同一内存
  • 初始化：不能读取未初始化的内存
  • 无数据竞争：跨线程的裸指针访问需要同步机制
// 典型的安全抽象模式：内部 unsafe，外部安全
use std::ops::{Deref, DerefMut};

/// 一个自定义的"安全裸指针包装器"
/// 保证：new时不为空，解引用时一定有效
struct SafePtr<T> {
    ptr: *mut T,
}

impl<T> SafePtr<T> {
    /// 创建一个新的安全指针——条件：传入的指针必须有效
    ///
    /// # Safety
    /// 调用者必须保证 ptr 指向有效的、已初始化的 T
    unsafe fn new(ptr: *mut T) -> Self {
        SafePtr { ptr }
    }
}

impl<T> Deref for SafePtr<T> {
    type Target = T;
    fn deref(&self) -> &T {
        // unsafe 块被隐藏在安全的 Deref 实现中
        unsafe { &*self.ptr }
    }
}

impl<T> DerefMut for SafePtr<T> {
    fn deref_mut(&mut self) -> &mut T {
        unsafe { &mut *self.ptr }
    }
}
实践法则：unsafe代码的最佳实践是"最小化原则"——每个unsafe块只做一件事，添加注释说明为什么这是安全的。// SAFETY:注释是Rust社区的标准做法，解释在什么前置条件下这个unsafe操作是安全的。标准库的每个unsafe操作都有详尽的Safety说明，这是你应该模仿的。

FFI与extern — C绑定

Rust程序很少完全自给自足。很多时候你需要调用现有的C库——比如系统调用、图形库、数字计算库等。这就是FFI（Foreign Function Interface）。

声明外部函数

// 在 Rust 中调用 C 标准库的 malloc 和 free
use std::ffi::c_void;

extern "C" {
    // 声明 C 函数
    fn malloc(size: usize) -> *mut c_void;
    fn free(ptr: *mut c_void);
}

fn main() {
    unsafe {
        let ptr = malloc(100);
        if ptr.is_null() {
            eprintln!("内存分配失败！");
            return;
        }
        // ... 使用这 100 字节 ...
        free(ptr);
    }
}

extern "C"块告诉Rust编译器这些函数使用C的ABI（调用约定）。然后你可以在unsafe块中调用它们——因为Rust无法验证外部函数的行为是否安全。

#[no_mangle] — 导出Rust函数给C调用

如果你需要写一个能从C调用的Rust库——比如为已有系统添加Rust模块——需要用#[no_mangle]防止Rust编译器修改函数名（Rust有名称修饰机制，类似C++的name mangling）：

// 这个函数可以从 C 代码中调用！
#[no_mangle]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
    a + b
}

// 暴露一个处理字符串的函数（复杂一点）
use std::ffi::CStr;
use std::os::raw::c_char;

/// 从C字符串计算长度
///
/// # Safety
/// s 必须指向一个有效的、以 null 结尾的 C 字符串
#[no_mangle]
pub unsafe extern "C" fn rust_strlen(s: *const c_char) -> usize {
    // 将 C 字符串转换为 Rust 的 &str——不安全操作
    let c_str = unsafe { CStr::from_ptr(s) };
    c_str.to_bytes().len()
}

对应的C端声明：

// C 代码中声明
extern int rust_add(int a, int b);
extern size_t rust_strlen(const char *s);

int main() {
    int sum = rust_add(1, 2);         // 3
    size_t len = rust_strlen("hello"); // 5
    return 0;
}
安全包装FFI：永远不要直接暴露unsafe的FFI函数给用户。标准做法是用安全函数包装它： 
/// 安全的包装器——隐藏了unsafe细节
pub fn safe_strlen(s: &str) -> usize {
    // 内部调用 unsafe FFI
    unsafe { rust_strlen(s.as_ptr() as *const c_char) }
}
这样用户的代码完全不需要接触unsafe——你替他们承担了正确调用FFI的责任。

可变静态变量

在Rust中，static变量默认是不可变的，并且具有'static生命周期。但有些场景（比如全局计数器、C兼容的全局状态）需要可变静态变量——这是unsafe的：

static mut COUNTER: u32 = 0;

fn add_to_counter(val: u32) {
    unsafe {
        COUNTER += val; // 读取和修改都需要 unsafe
    }
}

fn main() {
    add_to_counter(42);

    unsafe {
        println!("COUNTER: {}", COUNTER); // 42
    }

    // 多线程访问 static mut 会导致数据竞争！
    // 考虑用 AtomicU32 或 Mutex 作为安全替代
}

你立即注意到问题：多线程下对static mut的读写是数据竞争的！即使在单线程中，在unsafe块外部修改一个全局变量仍然可能导致不可预测的行为——因为编译器可能基于"这个值不会变"的假设做优化。这就是为什么Rust把static mut的访问也列为unsafe操作——它需要你确保没有数据竞争并且编译器优化不会破坏语义。

替代方案：大多数情况下你不需要static mut。用以下安全替代品：
  • std::sync::atomic::AtomicU32 —— 原子全局计数器
  • std::sync::OnceLock<T> —— 一次性初始化的全局状态
  • lazy_static! 或 std::sync::LazyLock —— 延迟初始化
  • thread_local! —— 线程本地存储
这些抽象内部使用unsafe，但对外提供了安全的API。

unsafe trait — 如Send/Sync的unsafe实现

一个trait被标记为unsafe意味着：实现这个trait需要调用者承担额外的安全义务。Rust标准库中最典型的例子是Send和Sync——它们是标记trait（没有方法），但要安全地实现它们需要确保类型确实可以跨线程：

// Send 和 Sync 的简化定义（标准库中）
// pub unsafe auto trait Send {}
// pub unsafe auto trait Sync {}

/// 一个自定义的、包装了裸指针的类型
struct MySendPtr<T>(*mut T);

// 编译器不会自动为 MySendPtr 实现 Send
// 因为 *mut T 不是 Send——裸指针可以被复制到其他线程

// 但我们作为类型的设计者知道：这个类型在特定条件下可以跨线程
unsafe impl<T: Send> Send for MySendPtr<T> {}

// 同样实现 Sync
unsafe impl<T: Sync> Sync for MySendPtr<T> {}

关于unsafe impl的关键规则：

  • 绝大多数情况下你不需要手动实现Send/Sync——编译器会自动推导
  • 当你引入裸指针或UnsafeCell时，自动推导会失效——因为编译器不再能确认安全性
  • 如果确定你的类型是线程安全的，用unsafe impl Send告诉编译器——但如果你错了，未定义行为
  • 永远优先考虑使用Arc、Mutex等现成的并发原语，而不是手动裸实现Send/Sync

在Rust生态中，很多底层库（如crossbeam、parking_lot）都使用了unsafe impl Send/Sync来提供并发原语。它们经过了极为严格的测试和形式化验证——这正是"unsafe并不意味着不安全"的最佳例证：通过严谨的unsafe代码构建出人人可用的安全抽象。

一个真实的故事：一行unsafe引发的血案

2019年，一个知名的Rust项目（我匿名化一下）发布了新版本。用户报告说在特定条件下，程序会以"看起来完全随机"的方式崩溃——有时在某行报segfault，有时在另一行报SIGILL。项目维护者花了三周时间逐步二分查找，最终定位到一行代码——一个unsafe块中对裸指针的写操作越界了4个字节。更糟的是，这行代码来自开发者在凌晨提交的一个"看起来很简单"的优化补丁。

4个字节的越界写入在堆上覆盖了相邻对象的内存——但这个错误不像空指针那样"立竿见影"地崩溃，而是表现为隐蔽的"数据损坏"。被损坏的数据可能几个小时都不被发现，然后某个完全不相干的函数读取了已损坏的数据——崩溃发生在离bug植入点几百万条指令之后。这就是内存安全漏洞最难调试的地方：原因和结果在时空上完全分离。

这个故事告诉我们两件事：第一，unsafe代码必须经过极端仔细的代码审查；第二，用MIRI（Rust的未定义行为检测工具）来测试unsafe代码——它能检测到大多数类型的UB。用以下方式运行测试：

rustup +nightly run miri test

MIRI是检测unsafe代码中未定义行为的最佳工具。如果你写了unsafe代码（或者用了第三方库中的unsafe代码），强烈建议你在CI中加入MIRI检查。

本章小结

Unsafe Rust是Rust语言最深邃的部分。我们学习了四种unsafe特权操作：裸指针解引用、unsafe函数调用、可变静态变量、unsafe trait实现。然后深入探讨了FFI绑定——如何调用C代码以及如何将Rust代码暴露给C。我们理解了unsafe是一个"契约"——编译器放弃检查，但你要自己保证安全。正确使用unsafe的核心模式是：最小化unsafe块，用安全抽象包裹它，加上详尽的Safety注释，并用MIRI验证。

unsafe是Rust作为"系统级语言"的能力证明——当你需要时，它给你C/C++级别的控制力。但请记住，即便是Rust标准库，也是用unsafe构建安全抽象的成功范例。你的目标不是"不用unsafe"，而是"在正确的地方用正确的unsafe，并且用严谨的工程实践确保它的安全性"。
