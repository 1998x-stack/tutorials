合成数据：新时代的数据引擎

大家好，我是你们的讲师。到今天为止，我们讨论的所有数据来源都是"自然采集"的——爬取、清洗、去重、按比例混合。但有没有想过一个问题：如果自然界中根本就不存在你需要的数据呢？

比如，你想让模型学会"用 JSON 格式输出用户意图分析结果"，但你的训练语料里可能只有 100 条这样的数据。或者你想让模型学会"在对话中主动追问以澄清歧义"，但你的语料里全是"一问一答"的模式。这就是合成数据（Synthetic Data）登场的场景。

合成数据——用模型生成训练数据——已经成为 2025-2026 年 AI 工程中最热门的实践之一。从 OpenAI 使用 GPT-4 生成数据训练 GPT-4o，到 DeepSeek 使用合成数据增强数学推理能力，合成数据不再是"有更好"的东西，而是"必须有"的核心竞争力。行业经验表明，合成数据的最佳比例大约在 30% 左右——低于这个比例你还没充分利用合成数据的潜力，高于这个比例你开始承受模型坍塌的风险。

核心洞察： "合成数据的本质不是制造新信息，而是重组和蒸馏已存在的信息。好的合成数据流水线，就像一位优秀的教师——自己深刻理解知识后，能针对学生的水平设计最适合的练习题和讲解方式。它没有创造新知识，但它大大提升了知识传递的效率。"

三大生成范式

合成数据生成并非一个方法通吃。根据数据形态和生成方式的不同，目前业界已经形成了三种主流范式。

范式一：Persona-Prompted（角色驱动）

这是最直接的方法：给定一个"角色"描述（比如"你是高中数学老师"）和一个"主题"提示（比如"生成 10 道关于三角函数的题目"），让 LLM 批量生成数据。这种方法的优势是简单高效，但缺点是覆盖的多样性受限于你设计的角色模板。

范式二：Taxonomy-Stratified（分层采样）

这个方法更系统化：先构建一个目标数据类型的分类体系（taxonomy），然后按类别分层采样。例如，要为数学推理生成数据，你的分类体系可能包括"代数"、"几何"、"概率"、"数论"四个大类，每个大类下再细分小类。然后在每个叶子节点上生成数据，保证所有类型的覆盖。

范式三：Evolution-Based（进化驱动）

这种方法最激进：从一个种子数据集出发，通过"进化"操作——深度进化（增加复杂性）、广度进化（增加多样性）、变异（改变表达方式）——不断衍生新的数据。WizardLM 就是这种范式的代表。

范式代表方法多样性控制质量稳定性扩展性适用场景
角色驱动Self-Instruct, Magpie中中（依赖种子数据）高通用指令数据、对话
分层采样Taxonomy-Instruction高（系统化）高中结构化能力、考试题
进化驱动WizardLM, Evol-Instruct高（持续演化）中低（可能退化）高复杂推理、创意写作

Self-Instruct 与 WizardLM

Self-Instruct：从零开始的指令生成

Self-Instruct (Wang et al., 2022) 是合成数据领域的奠基性工作。它的流程直观而优雅：

  1. 种子数据：准备 175 条手工编写的种子指令（涵盖不同任务类型）
  2. 生成指令：用 LLM 基于种子数据生成新的指令，配合"任务类型"提示来控制多样性
  3. 输入输出生成：对每个指令，让模型生成对应的输入-输出对
  4. 质量过滤：用规则过滤掉低质量或重复的样本
  5. 迭代：将新生成的数据加入种子池，重复以上过程

Self-Instruct 生成的 52k 指令数据（配合 GPT-3 的 API）可以显著提升模型的指令遵循能力。从这个意义上说，Self-Instruct 打开了"模型教模型"的大门。

Magpie：零样本生成

Magpie (Xu et al., 2024) 在 Self-Instruct 基础上更进一步：它不需要任何种子数据。Magpie 利用了一个观察——对齐后的 LLM 在生成回复时，其注意力分布天然存在一个"前缀模式"：模型在生成回复时隐式地假设了一个对话上下文。Magpie 从这种隐式上下文中反向提取指令-回复对，实现了从"无"到"有"的零样本合成数据生成。

WizardLM：进化指令

WizardLM (Xu et al., 2023) 的 Evol-Instruct 方法将合成数据带到了新的高度。它的核心是一个"进化循环"：

# Evol-Instruct 的进化操作（简化示意）

import json
import random
from typing import Dict, List

class EvolInstruct:
    def __init__(self, llm_generator):
        self.llm = llm_generator

    def deep_evolve(self, instruction: str) -> str:
        """深度进化：增加指令的复杂性"""
        prompt = f"""请将以下指令变得更复杂、更具挑战性。
可以添加约束条件、增加推理步骤、引入多跳推理等。

原始指令：{instruction}

更复杂的版本（直接输出指令，不要解释）："""
        return self.llm.generate(prompt)

    def breadth_evolve(self, instruction: str) -> str:
        """广度进化：生成与原始指令不同但相关的指令"""
        prompt = f"""基于以下指令，生成一个完全不同但相关的指令。
改变任务类型、更换场景或域。

原始指令：{instruction}

不同的版本（直接输出指令，不要解释）："""
        return self.llm.generate(prompt)

    def mutate(self, instruction: str) -> str:
        """变异：改变表达方式但保持核心任务"""
        prompt = f"""用完全不同的方式重新表述以下指令。
改变措辞风格、句式结构、细节程度。

原始指令：{instruction}

重新表述（直接输出指令，不要解释）："""
        return self.llm.generate(prompt)

    def quality_filter(self, source: str, evolved: str) -> bool:
        """质量过滤：确保进化后的指令没有丢失关键信息"""
        prompt = f"""判断进化后的指令是否保留了原始指令的完整意图。
只返回 YES 或 NO。

原始指令：{source}
进化后的指令：{evolved}"""
        result = self.llm.generate(prompt)
        return result.strip().upper() == "YES"

    def evolve_dataset(self, seeds: List[str],
                       num_rounds: int = 3) -> List[str]:
        """对种子数据集进行多轮进化"""
        evolved = list(seeds)
        for round_idx in range(num_rounds):
            new_instructions = []
            for inst in random.sample(evolved,
                                      min(1000, len(evolved))):
                op = random.choice([
                    self.deep_evolve,
                    self.breadth_evolve,
                    self.mutate
                ])
                evolved_inst = op(inst)
                if self.quality_filter(inst, evolved_inst):
                    new_instructions.append(evolved_inst)
            evolved.extend(new_instructions)
        return evolved

WizardLM 的多轮进化保证生成的数据集合持续提质增量。但注意，进化操作本身也需要质量筛选——不加控制的进化可能产生"为了复杂而复杂"的退化样本。

实践建议： 如果你资源有限，建议优先使用 Magpie（零样本）或小种子 Self-Instruct（175条种子），而不是直接上 Evol-Instruct。进化驱动的方法虽然上限高，但需要仔细调试质量过滤器和进化轮数，否则生成成本会失控。

知识蒸馏：大模型教小模型

合成数据的另一种重要用途是知识蒸馏——让一个大模型（教师）把知识传授给一个小模型（学生）。不同于合成数据"生成训练样本"的目标，知识蒸馏更关注"转移模型的推理能力"。

Distilling Step-by-Step

Distilling Step-by-Step (Hsieh et al., 2023) 提出了一个简洁而高效的蒸馏框架：教师模型不只是给出最终答案，还给出推理步骤。学生模型同时学习两个目标——预测正确答案 + 复现推理过程。实验表明，这种"过程蒸馏"相比"结果蒸馏"在样本效率上有 2-5 倍的提升。

Symbolic CoT：符号思维链

Symbolic CoT (Liu et al., 2024) 在 CoT 蒸馏的基础上更进一步：它将推理过程从自然语言转换为符号化的中间表示（比如逻辑表达式、数学公式），让学生模型学习这种抽象的符号推理能力。这种方法的好处是符号表示可以消除自然语言中的模糊性，使得推理更精确、更易于泛化。

LIMO：Less Is More for Reasoning

LIMO (Zhou et al., 2025) 挑战了"合成数据越多越好"的假设。LIMO 发现，对于推理能力的蒸馏，只需 817 个精心设计的"极限"样本就能达到超越数千倍数据量的蒸馏效果。关键在于样本的选择——不是任何高难度样本都行，而是那些"跨越知识边界"的边缘样本，即模型在正确和错误之间摇摆的样本。这些样本让教师模型的推理过程最有信息量。

关键发现： LIMO 证明蒸馏的核心不是数据量，而是数据的信息密度。817 个精心挑选的推理样本，每个都处于学生模型的"能力边界"上，其蒸馏效果超过了 10 万个随机采样的合成数据。这与我们第 1 章就谈到的"数据质量比数据量重要"的核心理念一脉相承。

合成数据的质量陷阱

合成数据并非灵丹妙药。使用不当，它会带来严重的质量问题。

Model Collapse：模型坍塌

模型坍塌（Model Collapse, Shumailov et al., 2024）是合成数据最著名的陷阱。当模型在自身生成的"合成数据"上反复训练时，模型的输出分布会逐步坍缩——多样性降低、长尾知识丢失、错误被放大和固化。论文的结论令人警醒：当训练数据中合成数据的比例超过 50% 时，模型坍塌现象开始出现；比例超过 60% 时，模型质量出现断崖式下跌。

这个发现直接解释了为什么行业共识将合成数据的"安全上限"设定在 30% 左右。但具体的安全比例取决于几个因素：

  • 教师模型的多样性：使用多个不同的教师模型生成合成数据，可以延缓坍塌
  • 混合策略：将合成数据与真实数据交错混合，而不是先合成后训练
  • 数据新鲜度：定期注入新的真实数据，稀释合成数据的负面影响

多样性度量与"敌意过滤"

合成数据另一个常见问题是"表面多样性但深层单一"。你可能生成了 10 万条指令，看起来每个都不一样，但仔细分析会发现它们都采用相同的句式和思维模式。这被称为"hidden homogeneity"（隐性同质化）。

对抗这个问题的方法是引入 "hostile filtering"（敌意过滤）——用一个专门寻找"相似性"的判別器来批处理合成数据，主动去除那些和已有数据过于相似的样本。Hostile filtering 的判別器被训练为"给定一个新样本，判断它是否包含已有数据集中没有的新模式"。这种"敌意"设置保证了合成的数据真正能为数据集带来多样性增量。

质量陷阱表现形式检测方法缓解策略
模型坍塌输出分布收敛到少数模式跟踪 n-gram 熵 + 多样性指标合成数据比例 < 30%，多教师混合
隐性同质化表面多样但深层结构雷同嵌入空间中的平均最近邻距离Hostile filtering，聚类后按簇采样
幻觉固化模型生成错误知识被其他模型复制与基准知识库交叉验证事实核查 + 不确定性采样
风格单一化所有输出呈现一致的"AI味"人类评估 + 风格分类器注入真实对话数据

实战案例：合成数据训练的模型只会说"作为一个AI助手"

踩坑实录： 某团队在一款客服场景的模型上大量使用了 GPT-4 合成的客服对话数据进行训练。初期效果极好——意图识别准确率提升了 15%。但上线后用户反馈如潮："为什么这个 AI 客服每句话都以'作为一个AI助手'开头？""回答像模板一样死板。"更严重的问题是，模型对非标准问法（比如"能帮我个忙吗？"这种隐含请求）完全无法处理——因为合成数据中的"用户提问"全部是标准化的"请帮我……"。最终团队不得不砍掉 60% 的合成数据，重新注入真实的客服对话记录，问题才得以解决。教训是：合成数据擅长复制模式，但很难复制真实交互中的多样性和意外性。

这个案例揭示了一个深层次的原理：合成数据的"便利性"也是它最大的"诅咒"——因为它太容易批量生成了，你很自然地倾向于生产大量模式一致的样本，而忽略了真实数据中弥足珍贵的"脏"和"意外"。好的合成数据策略，应该像好的教学一样——既要有标准教案，也要保留真实课堂的即兴互动。
