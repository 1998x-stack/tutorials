第22章 GPU体系结构

同学们好！上一章我们学习了SIMD——一条指令处理多份数据，这是并行计算的"轻量级方案"。今天我们要见识真正的"并行巨兽"——GPU（Graphics Processing Unit）。GPU可以同时运行数万个线程，是当代AI革命的算力基石。但GPU的设计哲学和CPU完全不同，理解它的内部结构是发挥其威力的前提。让我们一起揭开GPU的神秘面纱。

22.1 GPU vs CPU：两种哲学

CPU和GPU的设计目标截然不同，我们可以用一句话概括：CPU是延迟优化的，GPU是吞吐量优化的。

CPU为了降低单条指令的延迟，投入了大量芯片资源：深流水线、分支预测、乱序执行、大容量cache、寄存器重命名、前瞻执行……所有这些超标量技术都在追求"让一条指令跑得更快"。代价是——这些控制逻辑占到了CPU芯片面积的60-70%，真正做计算的ALU只占一小部分。

GPU的选择完全不同。它用数千个简单的核心来换取总吞吐量（Throughput）。GPU核心是顺序执行的（in-order），没有分支预测，没有乱序执行，cache较小。它隐藏延迟的方式不是通过复杂的预测，而是通过线程切换：当一组线程在等待内存访问完成时，调度器立即切换到另一组准备好执行的线程。只要有足够的线程，GPU的执行单元就能始终保持忙碌。

结果就是：一个现代CPU（如Intel Core i9-14900K）的单线程浮点性能约200 GFLOPS，而一块RTX 4090的理论浮点性能超过80 TFLOPS——相差400倍！当然，GPU只有在足够多的并行工作负载下才能发挥这个性能。

22.2 SIMT执行模型

NVIDIA引入了一种称为SIMT（Single Instruction, Multiple Threads，单指令多线程）的执行模型，这是理解GPU编程的关键。

在SIMT模型中，Warp（线程束）是最基本的执行单元——32个线程组成一个warp，它们在同一个CUDA核心上以锁步（lockstep）方式执行同一指令。每个线程有自己的寄存器状态和程序计数器，但warp中的所有线程在同一时刻执行相同的指令。

如果warp中所有线程都走同一个分支路径，硬件利用率是100%。但如果线程发生分支分歧（Branch Divergence），情况就不同了：

// CUDA 中的分支分歧示例
__global__ void divergent_kernel(float* data, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        if (data[idx] > 0.0f) {
            data[idx] = sqrt(data[idx]);  // 路径A：正数处理
        } else {
            data[idx] = 0.0f;             // 路径B：非正数清零
        }
    }
}

// 在同一个warp中（32个相邻thread，假设threadIdx.x 0-31）：
// 如果data[0] > 0, data[1] <= 0, data[2] > 0, ...
// 那么路径A和路径B的线程会分成两组
// 硬件先执行路径A（thread 0,2,3...），thread 1,4等被mask掉（不写入）
// 然后执行路径B（thread 1,4...），thread 0,2等被mask掉
// 两种分支→ 两倍的执行时间！

SIMT和SIMD的区别很微妙但重要：SIMD（如SSE/AVX）要求程序员显式地将数据打包到向量寄存器中，用向量指令操作；而SIMT让程序员为单个线程编写标量代码，硬件自动将线程组织成warp执行。程序员写的是"每个线程做什么"，硬件负责将线程分组并行的细节。这大大简化了编程模型。

22.3 NVIDIA GPU内部结构

以NVIDIA的架构为例，GPU的层次结构从大到小是：GPU芯片 → GPC（图形处理簇）→ SM（流多处理器）→ CUDA Core / Warp Scheduler。

SM（Streaming Multiprocessor，流多处理器）是GPU的核心计算单元。一个典型的SM包含：

  • CUDA Core：64-128个整数/浮点运算单元
  • Warp Scheduler：4个调度器，每个控制一组warp
  • Register File：65536 × 32-bit（256KB！这比CPU L2 cache还大）
  • Shared Memory + L1 Cache：可配置共享（0-100KB），用于线程块间通信
  • Tensor Core：混合精度矩阵乘法加速器
  • Special Function Units (SFU)：处理超越函数（sqrt, sin, cos）

以H100（Hopper架构）为例：每个SM有4个warp调度器，每个调度器可以跟踪16个warp（共64个warp/SM），每个warp 32个线程，所以每个SM可以跟踪64×32 = 2048个线程。H100共有132个SM，因此总线程容量达到132 × 2048 ≈ 27万个并发线程！这就是GPU隐藏延迟的核心能力——当某个warp在等待内存时，warp调度器可以立即切换到另一个warp，零开销上下文切换。

22.4 GPU存储层次

理解GPU的存储层次是优化GPU程序的核心。从快到慢：

存储层级 作用域 典型大小 延迟 用途
寄存器 每个线程私有 255个 32-bit/线程 ~1 cycle 局部变量、中间结果
Shared Memory 线程块内共享 0-228KB / SM（H100） ~5-20 cycles 线程块协作、数据复用
L1/纹理Cache SM内 可配置（与shared memory共享） ~20-30 cycles 通用数据缓存
L2 Cache 全局 60MB（H100） ~120-200 cycles 芯片级缓存
HBM3 全局 80GB（H100） ~300-600 cycles 主存，高带宽（3.35 TB/s）

最重要的优化之一是合并访问（Coalesced Memory Access）。当一个warp中的32个线程访问全局内存时，硬件会尝试将这些访问合并成尽可能少的内存事务（Memory Transactions）。如果32个线程访问的是连续对齐的128字节块，所有访问可合并为1个事务，带宽利用率100%。但如果每个线程访问的是间隔很大的地址（如列优先矩阵访问），可能产生32个独立事务，带宽利用率暴跌到3%以下。

// 合并访问 vs 非合并访问
#define N 4096
__global__ void coalesced(float* A, float* B, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        B[i] = A[i] * 2.0f;     // 连续访问，warp内线程访问A[0..31]
                                  // 完美合并：1个事务传输128字节
    }
}

__global__ void strided(float* A, float* B, int n, int stride) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        B[i * stride] = A[i * stride] * 2.0f;  // stride=32时，
        // warp内线程访问A[0], A[32], A[64], ...
        // 每次访问不同cache行，无法合并！
    }
}

// 矩阵乘法 — 使用shared memory优化合并访问
__global__ void matmul_shared(float* A, float* B, float* C, int M, int N, int K) {
    __shared__ float As[32][32];
    __shared__ float Bs[32][32];

    int row = blockIdx.y * 32 + threadIdx.y;
    int col = blockIdx.x * 32 + threadIdx.x;
    float sum = 0.0f;

    for (int t = 0; t < K; t += 32) {
        As[threadIdx.y][threadIdx.x] = A[row * K + t + threadIdx.x];
        Bs[threadIdx.y][threadIdx.x] = B[(t + threadIdx.y) * N + col];
        __syncthreads();

        for (int k = 0; k < 32; k++)
            sum += As[threadIdx.y][k] * Bs[k][threadIdx.x];
        __syncthreads();
    }
    C[row * N + col] = sum;
}

22.5 Tensor Core与AI时代的GPU

2017年，NVIDIA在Volta架构（V100）中首次引入了Tensor Core——一种专门为矩阵乘法加速的硬件单元。这是一个革命性的设计，因为它标志着GPU从"图形处理器"向"AI处理器"的转型。

Tensor Core的核心是一个混合精度矩阵乘加运算：D = A × B + C，其中A和B通常是FP16/BF16/FP8格式，C和D可以是更高精度。一次Tensor Core操作可以在一个时钟周期内完成多个矩阵运算——一个H100 Tensor Core每个周期完成1024次FP16运算！

历代GPU的规格对比：

架构 GPU 年份 SM数 CUDA Core Tensor Core 显存带宽 FP16 TFLOPS
Kepler K80 2014 13×2 2496×2 — 480 GB/s —
Pascal P100 2016 56 3584 — 732 GB/s —
Volta V100 2017 80 5120 640 900 GB/s 125
Turing T4 2018 40 2560 320 320 GB/s 65
Ampere A100 2020 108 6912 432 2.0 TB/s 312
Hopper H100 2022 132 16896 528 3.35 TB/s 990
Blackwell B200 2024 — — — 8 TB/s 2250

Blackwell（B200）的发布标志着GPU进入了一个新阶段——FP4和FP6精度支持、Transformer Engine第二代（动态调度模型各层的计算精度）、以及更大规模的多芯片封装（两个GPU die通过10TB/s的NVLink连接）。GPU已经不再是"图形卡"，它是AI工厂的核心引擎。

🔑 GPU不是更快的CPU：GPU不是"更快的CPU"——它是为不同目的设计的完全不同类型的处理器。理解它的SIMT模型和存储层次是利用其算力的关键。CPU擅长"一件事做得非常快"（低延迟），GPU擅长"一万件事做得差不多快"（高吞吐）。在AI时代，这两种处理器不是竞争关系，而是协作关系——CPU负责指挥调度和不可并行的逻辑，GPU负责大规模并行计算。
