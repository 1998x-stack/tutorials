第14章 哈希表扩容：渐进式 Rehash

大家好，我是你们的讲师。上一章我们探讨了哈希表的基本操作——查找、插入、删除。但哈希表面临一个根本性的问题：随着数据量增长，负载因子（item 数 / 桶数）不断升高，碰撞越来越频繁，查找效率从 O(1) 逐渐退化到 O(n)。

解决办法是扩容——增加桶的数量，将已有 item 重新分布到更多的桶中。但问题是，在服务百万 QPS 的线上环境，如果扩容需要一次性 rehash 所有的 key，这个过程可能需要几百毫秒甚至几秒钟——对于延迟敏感的业务来说，这是一场灾难。Memcached 采用了一种优雅的解决方案：渐进式 rehash（incremental rehash），将扩容的代价分摊到每一次查询中。本章我们一文打尽。

扩容触发条件

Memcached 在什么时候触发哈希表扩容？答案在 assoc.c 的 assoc_insert() 函数中：

int assoc_insert(item *it) {
    // ... 执行插入 ...

    // 插入后检查是否需要扩容
    if (hash_items > (hashsize(hashpower) * 1.5)) {
        // 负载因子超过 1.5，触发扩容
        assoc_expand();
    }

    return 1;
}

触发条件是：hash_items > hashsize(hashpower) * 1.5。

注意一个有趣的特性：每次扩容后，负载因子会回落到大约 0.75（扩容阈值 1.5 的一半）。这意味着扩容后哈希表的性能会"焕然一新"——平均每个桶只有 0.75 个 item，链表长度几乎等于 1。随着 item 继续增加，负载因子逐渐回升到 1.5，再次触发扩容。这是一个"锯齿形"的增长曲线。

初始 hashpower 配置：如果你的服务启动后很快就会存储大量 item（比如 100 万以上），可以通过 -o hashpower_init=19 将初始桶数从 65536 增加到 524288。这样可以减少扩容次数，因为每次扩容虽然渐进，但仍然有一定开销。不过这个参数需要在 Memcached 启动时设置，无法运行时修改。

assoc_expand()：创建双倍大小的新表

当触发扩容时，assoc_expand() 被调用。它的工作非常简单——只分配内存，不做数据迁移：

static void assoc_expand(void) {
    // 新表的大小是旧表的两倍
    old_hashtable = primary_hashtable;
    old_hashpower = hashpower;

    hashpower++;

    primary_hashtable = calloc(hashsize(hashpower), sizeof(void *));
    if (primary_hashtable == NULL) {
        // 内存不足，回退
        hashpower--;
        primary_hashtable = old_hashtable;
        return;
    }

    // 设置扩容进行中的标志
    expanding = true;
    expand_bucket = 0;
}

此时，系统处于一个"双表共存"的状态：

  • primary_hashtable 指向新表（更大）
  • old_hashtable 指向旧表（将被逐步迁移）
  • expanding = true，表示扩容正在进行
  • expand_bucket = 0，从第 0 个桶开始迁移

注意新表是用 calloc 分配的，所有桶指针初始化为 NULL。旧表的数据还全部在旧表中。接下来的迁移工作，由一个专门的后台线程完成。

assoc_maintenance_thread：逐桶迁移

assoc_maintenance_thread 是负责渐进式 rehash 的后台线程。它的循环逻辑是：

static void *assoc_maintenance_thread(void *arg) {
    while (1) {
        // 每迁移一批 bucket 就休眠一下
        pthread_mutex_lock(&cache_lock);

        if (!expanding) {
            // 没有扩容进行中，等待信号
            pthread_cond_wait(&maintenance_cond, &cache_lock);
        }

        // 迁移：每次处理 hash_bulk_move 个 bucket
        int moved = 0;
        while (expanding && moved < hash_bulk_move) {
            item *it, *next;

            // 遍历 expand_bucket 指向的旧表桶
            for (it = old_hashtable[expand_bucket]; it != NULL; it = next) {
                next = it->h_next;

                // 重新计算在新表中的位置
                uint32_t hv = hash(ITEM_key(it), it->nkey);
                unsigned int new_bucket = hv & hashmask(hashpower);

                // 头插法插入新表
                it->h_next = primary_hashtable[new_bucket];
                primary_hashtable[new_bucket] = it;
            }

            // 清空旧表当前桶
            old_hashtable[expand_bucket] = NULL;
            expand_bucket++;
            moved++;

            // 如果所有桶都迁移完了
            if (expand_bucket == hashsize(old_hashpower)) {
                // 扩容完成！
                expanding = false;
                free(old_hashtable);
                old_hashtable = NULL;
            }
        }

        pthread_mutex_unlock(&cache_lock);

        // 休眠一定时间后继续下一批
        usleep(1000);  // 1ms
    }
}

关键参数和变量：

变量/参数 含义 默认值
hash_bulk_move 每轮迁移的桶数 256（硬编码在 assoc.c 中）
expand_bucket 当前迁移到的桶索引 0（扩容开始时）
expanding 是否正在扩容 false
hashsize(n) 2^n 个桶 依赖于 hashpower

每轮迁移 256 个桶（约 256 次链表遍历和重插入操作），然后休眠 1ms，释放 cache_lock，让 worker 线程有机会处理请求。通过这种"短脉冲"的方式，扩容操作完全不会阻塞正常的服务。

双表期间的查找/插入/删除逻辑

在双表共存期间，每个操作都需要知道"数据在哪张表里"。核心判断依据是 expand_bucket：

// 查找时
uint32_t hv = hash(key, nkey);
unsigned int bucket = hv & hashmask(hashpower);  // 新表索引

item *it;
if (expanding && bucket < expand_bucket) {
    // 这个桶已经迁移到新表了，查新表
    it = primary_hashtable[bucket];
    // 注意：需要用新表的索引直接查
    // 已经迁移的桶在旧表中被清空，所以只查新表就够了
} else {
    // 尚未迁移，查旧表
    unsigned int old_bucket = hv & hashmask(old_hashpower);
    it = old_hashtable[old_bucket];
}

这个逻辑用一张图来总结：

                     expand_bucket
                           │
               ┌───────────┴───────────┐
               │       已迁移           │       未迁移
               │   (查新表)            │    (查旧表)
               ▼                       ▼
    新表:  [bucket 0..N]          [空]
    旧表:  [空]                   [bucket N+1..M-1]

    判断公式: 新表索引 < expand_bucket ? 新表 : 旧表

插入和删除操作的判断逻辑与查找类似——都是通过比较 expand_bucket 和计算出的新表桶索引来决定操作哪张表。

一个重要的限制：在扩容期间，不能再次触发扩容。所以当 expanding = true 时，即使 hash_items > hashsize(hashpower) * 1.5，也不会调用 assoc_expand()。扩容过程中的负载因子可以暂时超过 1.5，直到当前扩容完成。

对比 Redis 的 rehash 设计：Redis 的 rehash 采用了类似的双表机制（ht[0] → ht[1]），但有以下主要区别：
1) Redis 采用"条件触发"——在每次增删改查操作中顺带迁移 1-2 个 bucket，不做专门的维护线程
2) Redis 支持"rehash 暂停/恢复"（用于持久化 fork 场景）
3) Redis 有"rehash 不成功"的回滚机制（内存不足时保持双表）
两种设计各有利弊：Memcached 的维护线程方式更"积极"，确保扩容尽快完成；Redis 的"顺手牵羊"方式更"保守"，对正常请求的影响更小。

实际扩展示例：从 16 到 25 的演进

我们来看一个实际场景。假设你的 Memcached 实例从 0 开始接收数据：

  1. 初始状态：hashpower = 16，65536 个桶
  2. item 数达到 98,304：触发第一次扩容，hashpower → 17，桶数 → 131,072。后台线程开始逐桶迁移。
  3. item 数达到 196,608：触发第二次扩容，hashpower → 18。此时前一次扩容可能还没完成！但前文提到过"扩容期间不能再次触发扩容"——所以第二次扩容会等到第一次完成后才开始。在扩容期间，item 会继续增长，但负载因子可能暂时超过 1.5。
  4. 以此类推，每增加约 1.5 倍的 item 就会触发一次扩容。

对于一个存储 5000 万 item 的生产实例，hashpower 最终会增长到大约 25（2^25 = 33,554,432 个桶，负载因子 ~1.5）。新表的大小是 33,554,432 × 8 字节（指针大小）≈ 256 MB。这是哈希表本身的内存开销——对于 5000 万 item 的数据来说，这是可以接受的。

扩容的持续时间取决于 item 数量和系统负载。在正常负载下，从 hashpower=16 到 17 的迁移（~65536 个桶）大约需要 1-2 秒（每轮 256 个桶 × 1ms 休眠 ≈ 256ms 的纯迁移时间 + 锁竞争等待）。如果系统 QPS 很高，cache_lock 竞争激烈，迁移线程可能每次拿锁的时间更短，扩容完成时间会相应延长。

踩坑：扩容期间 CPU 飙升导致服务抖动

最后，分享一个和 rehash 相关的生产事故。有一次，我们的一个 Memcached 实例每隔几小时就会出现一次 p99 延迟抖动——从 1ms 跳到 100ms，持续大约 3-5 秒，然后自己恢复。

经过大量数据采集和关联分析，发现抖动时间点与 stats 中 hash_is_expanding 标志位为 true 的时间段高度重合。进一步分析火焰图，发现抖动期间 assoc_find() 中出现了大量的 cache miss——原因是双表查询的"额外间接跳转"。

但等等——正常情况下双表查询应该只是多一个条件判断，不至于导致 100 倍的延迟增加吧？

真正的问题出在：扩容期间，旧表中的 item 正在被逐个迁移到新表。这意味着在同一个 bucket 中，一部分 item 已经在旧表中、另一部分已经在新表中。查找时，如果 item 已经被迁移走了，在旧表中找不到，还要再到新表中再查一次——等等，不对，查找逻辑是先判断桶是否已迁移，然后只查一张表。但仔细想想，这里确实有一个微妙的问题。

实际上，真正的 bug 是这样的：在旧版本的 Memcached（1.4.x）中，双表查询是在旧表中遍历一遍，如果没找到再到新表中遍历一遍。这种"双倍查找"在迁移过程中会导致一些本来 O(1) 的查找退化为 O(1 + bucket 长度) ——也就是多了一次完整的链表遍历。对于某些 bucket 来说，这个链表可能有 10-20 个元素，所以查找时间会增加一倍。但还不至于到 100 倍。

真正的根因：经过更深入的排查，发现问题的关键是cache_lock 竞争。rehash 线程在迁移每个 bucket 时都会持有 cache_lock，虽然每次只持有很短的时间（256 个 bucket 的迁移），但在高并发下，这个锁的竞争会导致大量 worker 线程排队等待。尤其是当 item_size_max 设置较大时，每个 bucket 的链表中有大量 item，单次迁移时间更长，锁持有时间更长。最终，我们通过以下措施解决了问题：

1) 将 hash_bulk_move 从默认值调整为更小的值（128），减少每次持锁时间
2) 设置 -o hashpower_init=18，从一开始就使用更大的哈希表，减少扩容次数
3) 升级到 1.6.x 版本，该版本优化了 rehash 线程的锁竞争策略

另外，如果你的场景是"实例启动后 item 数快速从 0 增长到数千万"，你会经历连续多次的哈希表扩容。建议通过设置 hashpower_init 来"跳跃"到合适的初始大小，避免"扩容 → 完成 → 立刻又扩容"的频繁触发。

第 7-14 章总结

到这里，我们完成了整个"内存管理核心"部分的学习。这是全课程最长、也最重要的一半内容。我们来快速回顾一下这 8 章的内在逻辑：

  • 第 7-9 章 (Slab Allocator)：回答了"内存从哪里来"——slab allocator 从 OS 预申请大块内存，按固定大小切分成 chunk，通过 slots 链表做 O(1) 分配和回收，通过 slab rebalance 做跨 class 的平衡。
  • 第 10-11 章 (Item)：回答了"数据长什么样"——item 是一个紧凑的内存结构体，key/suffix/value 紧跟在头部之后。它的生命周期涉及分配、链接、访问、删除，通过 refcount 保证安全，通过 lazy expiration 淘汰过期数据。
  • 第 12 章 (LRU)：回答了"内存不够时淘汰谁"——三级 LRU 队列（HOT/WARM/COLD/TEMP）解决了扫描型流量污染的问题，lru_bump 实现队列间升降级，LRU crawler 负责后台清理和队列间迁移。
  • 第 13-14 章 (哈希表)：回答了"怎么找到数据"——链地址法哈希表，通过 h_next 指针串联冲突元素，通过渐进式 rehash 在不中断服务的情况下扩容。

这四个模块之间环环相扣：slab allocator 分配内存，item 使用内存，LRU 回收内存，哈希表 定位内存中的 item。少了任何一个，Memcached 的核心功能都无法工作。
