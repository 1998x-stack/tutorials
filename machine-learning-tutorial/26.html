26.1 LLM：2020年代最大的技术浪潮

2022年11月ChatGPT发布，两个月内用户突破1亿——成为历史上增长最快的消费产品。这不仅仅是又一个AI模型，而是一个时代的转折点。GPT-4、Claude、Gemini、LLaMA、Qwen——这些大语言模型（LLM）正在重塑软件开发、内容创作、客户服务和科学研究的每一个角落。

作为ML从业者，你不需要从零训练一个LLM（那需要数千万美元），但你必须理解LLM的工作原理、如何微调、如何用RAG构建应用——这些是2026年最炙手可热的工程技能。

26.2 LLM如何工作：从token到文本

LLM的核心是一个"下一个token预测器"。给定一段文本，它预测最可能的下一个词（准确说是token）。这个过程重复下去，就生成了连贯的文本。

Tokenization

文字不能直接喂给模型，需要先切成token。英文大约1个token ≈ 0.75个词，中文约1个token ≈ 0.5个字。GPT-4的上下文窗口128K tokens大约能容纳一本300页的书。

Scaling Laws

一个被反复验证的规律：模型越大 + 数据越多 = 效果越好，且这个趋势似乎没有触顶的迹象。这就是为什么各大公司都在疯狂扩大模型规模和训练数据量。

26.3 Prompt Engineering：撬动LLM的杠杆

好的prompt和差的prompt，效果可以差一个数量级。几个关键技巧：

  • Chain-of-Thought（CoT）：让模型"一步一步思考"再给答案。比如加上"Let's think step by step"，在推理类任务上准确率大幅提升
  • Few-shot：给几个示例（输入→输出），模型就能学会输出的格式和风格
  • System Prompt：设定角色和约束，如"你是一个有10年经验的Python工程师"
  • 结构化输出：明确要求JSON/Markdown格式，方便程序解析
💡 CoT的原理：大语言模型是"自回归"的——先输出的token会影响后续的token。当模型先输出推理步骤再输出答案时，它实际上在"给自己的思考分配更多计算"。这就像让一个学生在草稿纸上演算而不是心算——多写几步中间推导能显著提高最终正确率。没有CoT的模型相当于"直觉判断"，有CoT的模型相当于"推理判断"。

26.4 RAG：让LLM拥有"外部知识库"

LLM有两个致命弱点：(1) 知识截止于训练日期，不知道最新信息；(2) 对不熟悉的内容可能"幻想"（hallucination）出看起来合理但完全错误的答案。

RAG（Retrieval-Augmented Generation）是解决这两个问题的首选方案：

  1. 索引（Indexing）：把你的文档（产品手册、公司wiki、知识库）切成小段，用embedding模型转成向量，存入向量数据库
  2. 检索（Retrieval）：用户提问时，把问题也转成向量，在数据库中搜索最相关的文档片段
  3. 生成（Generation）：把检索到的相关文档片段和用户问题一起塞进prompt，让LLM基于这些资料回答
import chromadb
from sentence_transformers import SentenceTransformer

# 1. 初始化embedding模型和向量数据库
embedder = SentenceTransformer('BAAI/bge-large-zh-v1.5')
client = chromadb.Client()
collection = client.create_collection("company_docs")

# 2. 索引文档
documents = [
    "公司年假政策：入职满1年享有5天年假，满3年10天，满5年15天。",
    "加班政策：工作日加班1.5倍工资，周末2倍，法定假日3倍。",
    "远程办公政策：每周可申请2天远程办公，需主管审批。",
]
for i, doc in enumerate(documents):
    collection.add(
        embeddings=[embedder.encode(doc).tolist()],
        documents=[doc],
        ids=[f"doc_{i}"]
    )

# 3. 检索
query = "我工作3年了，一年有多少天假期？"
query_embedding = embedder.encode(query).tolist()
results = collection.query(query_embeddings=[query_embedding], n_results=2)

# 4. 构建prompt发给LLM
context = "\n".join(results['documents'][0])
prompt = f"""基于以下公司政策回答问题：

{context}

问题：{query}
回答："""
# 把这个prompt发给LLM...
🔑 RAG的最佳实践要点：(1) 文档分块（chunking）大小很关键——太小缺乏上下文，太大检索不准。常用500-1000字符一块，且块之间要有重叠。(2) 检索时不仅用向量相似度，还可以加入关键词匹配（BM25）做混合检索。(3) rerank——检索到20条候选，再用更精准的模型（如Cohere Rerank或BGE-Reranker）精排到5条。(4) 在prompt中明确要求"如果找不到相关信息，就说不知道，不要编造"——这是减少幻觉的有效手段。

26.5 Agent：让LLM使用工具

LLM不仅能回答问题，还能作为"大脑"调用外部工具：搜索引擎、代码解释器、数据库查询、API调用。这就是Agent架构——给LLM配上一个工具箱，让它自己决定什么时候用什么工具。

ReAct（Reasoning + Acting）模式是目前最流行的Agent范式：Thought（思考我需要做什么）→ Action（执行工具调用）→ Observation（观察结果）→ Thought（基于结果重新思考）→ ... 循环直到任务完成。这种"思考-行动-观察"的循环让LLM能够处理之前无法完成的多步复杂任务。

26.6 LoRA微调：用小成本定制大模型

全参数微调一个70B的模型需要几十张GPU。但LoRA（Low-Rank Adaptation）通过在预训练权重旁插入小的可训练矩阵，将可训练参数量减少99%以上，同时效果接近全参数微调。8B模型用LoRA微调甚至能在单张消费级GPU上完成。

💡 什么时候用RAG，什么时候用微调？需要动态更新的知识（如实时数据、最新政策）→ RAG。需要改变模型行为/风格/输出格式（如医疗报告风格、特定领域术语）→ 微调。需要两者兼顾 → RAG + 微调。现实中最强的企业应用往往两者都做：微调确保模型理解领域规范，RAG注入最新数据。
