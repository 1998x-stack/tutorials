线程层次结构

大家好，我是你们的讲师。上一章我们写了第一个CUDA程序——向量加法。你肯定注意到了那句代码：int idx = blockIdx.x * blockDim.x + threadIdx.x; 对初学者来说，这是全程序最令人困惑的一行。blockIdx是什么？threadIdx又是什么？为什么不用一个简单的循环变量？

答案是：因为CUDA的线程不是一维的，而是三维分层的。不理解这个层次结构，你写的索引就是错的——而索引错误在CUDA中不会触发编译错误，只会产出错误的结果。可以毫不夸张地说：搞对索引，就搞定了CUDA编程80%的细节。这一章，我们彻底吃透线程层次结构。

Grid/Block/Thread三层结构

CUDA的线程组织是一个三层嵌套结构：

Grid (网格)
  └── Block 0 (线程块)
  │     └── Thread 0
  │     └── Thread 1
  │     └── Thread 2
  │     └── ...
  └── Block 1
  │     └── Thread 0
  │     └── ...
  └── Block 2
  │     └── ...
  └── ...

具体来说：

  • Thread（线程）：最小的执行单元。每个线程执行kernel函数中的同一段代码，但操作不同的数据
  • Block（线程块）：一组线程的集合。同一个block内的线程可以协作——通过共享内存交换数据和调用 __syncthreads() 同步（第7章）
  • Grid（网格）：一组block的集合。一个kernel启动对应一个grid

每个维度都可以是1D、2D或3D。CUDA提供了若干个内置变量来让每个线程知道自己的位置：

内置变量 类型 含义 取值范围
threadIdx dim3 线程在其所属block内的索引 [0, blockDim-1]
blockIdx dim3 block在grid内的索引 [0, gridDim-1]
blockDim dim3 block的维度（每个维度的线程数） 由 <<<>>> 的第二个参数指定
gridDim dim3 grid的维度（每个维度的block数） 由 <<<>>> 的第一个参数指定

每个变量都有 .x、.y、.z 三个分量。即使你只用1D，.y 和 .z 的值也默认为1（不是0！）——这个细节有时候会坑到你。

1D索引计算

大部分CUDA初学者遇到的问题都是1D的——处理一个一维数组，启动一维的grid和block。这种情况下，全局索引的计算公式是：

// 1D全局索引：当前线程在整个数据中的位置
int idx = blockIdx.x * blockDim.x + threadIdx.x;

来拆解一下这个公式：

  • blockIdx.x：当前是第几个block（从0开始）
  • blockDim.x：每个block有多少个线程
  • blockIdx.x * blockDim.x：在当前block之前已经启动了多少个线程
  • + threadIdx.x：加上当前线程在block内的偏移，就得到了全局唯一索引

举个例子：如果 blockDim.x = 256，blockIdx.x = 3，threadIdx.x = 42，那么全局索引 idx = 3 × 256 + 42 = 810。也就是说，你启动的第810号线程。

// 1D向量加法的完整例子
__global__ void vecAdd(float *a, float *b, float *c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        c[idx] = a[idx] + b[idx];
    }
}

// 启动配置
int n = 1000000;
int blockSize = 256;
int gridSize = (n + blockSize - 1) / blockSize;  // 向上取整 = 3907
vecAdd<<<gridSize, blockSize>>>(d_a, d_b, d_c, n);

2D与3D索引

很多实际问题天然是二维或三维的——比如图像处理（二维像素）、物理模拟（三维网格）、矩阵运算（二维行列）。CUDA允许你用2D或3D的grid/block来匹配问题的维度，这样索引计算更加直观。

// 2D grid/block —— 适合矩阵操作
// 启动: dim3 blockDim(16, 16); dim3 gridDim(w/16, h/16);
// kernel: matrixAdd<<<gridDim, blockDim>>>(d_a, d_b, d_c, w, h);

__global__ void matrixAdd(float *a, float *b, float *c, int width, int height) {
    int col = blockIdx.x * blockDim.x + threadIdx.x;  // 列 (x方向)
    int row = blockIdx.y * blockDim.y + threadIdx.y;  // 行 (y方向)

    if (col < width && row < height) {
        int idx = row * width + col;   // 展平为1D索引
        c[idx] = a[idx] + b[idx];
    }
}

// 3D grid/block —— 适合体数据处理
// 启动: dim3 blockDim(8, 8, 8); dim3 gridDim(dx/8, dy/8, dz/8);
// kernel: volumeProcess<<<gridDim, blockDim>>>(d_data, dx, dy, dz);

__global__ void volumeProcess(float *data, int dx, int dy, int dz) {
    int ix = blockIdx.x * blockDim.x + threadIdx.x;
    int iy = blockIdx.y * blockDim.y + threadIdx.y;
    int iz = blockIdx.z * blockDim.z + threadIdx.z;

    if (ix < dx && iy < dy && iz < dz) {
        int idx = iz * dx * dy + iy * dx + ix;  // 展平
        // 处理 data[idx] ...
    }
}
什么时候用2D/3D？ 如果你的问题维度天然匹配（如图像、矩阵、体素网格），就用2D/3D。它的好处是索引计算更自然，更重要的是——它影响block在SM上的调度顺序，对缓存局部性有微妙的优化作用。但对于一个纯1D问题（如处理一维数组），强行用2D只会让代码更复杂，没有任何好处。

Block大小选择策略

block size（每个block的线程数）可能是CUDA中最常被问的问题。没有"唯一正确答案"，但有明确的指导原则：

Block Size 优点 缺点 适用场景
128 SM占用率高，灵活性好 单个block工作量较少 大多数通用场景的稳妥选择
256 计算密集与内存访问均衡 寄存器压力可能增加 最常用的默认值，通用场景
512 单个block内更多协作空间 可能限制每个SM上的活跃block数 共享内存使用较多的kernel
1024 最大化单个block的线程数 严重依赖高寄存器预算，容易spill shared memory密集且寄存器使用少的kernel

几个硬性约束：

  • 必须是32的倍数：block size必须能被warp size（32）整除。否则会有"浪费"的线程——硬件会启动整个warp，但部分线程什么都不做
  • 最大1024：所有计算能力 >= 2.0的设备，单个block最多1024个线程
  • 每个维度有上限：maxThreadsDim 定义了每个维度的大小限制（通常是 1024 × 1024 × 64）

另外，你需要考虑占用率（occupancy）——每个SM上活跃的warp数量与最大可能warp数量的比值。理论上，占用率越高，SM隐藏延迟的能力越强。但高占用率不一定意味着高性能——如果每个线程的计算量很大（compute-bound），低占用率也可能达到高吞吐。我们会在第17章（性能优化方法论）中深入讨论这个权衡。

经验法则： 对于大多数情况，从 blockSize = 256 开始。这是最平衡的选择——它既能提供足够的线程供SM调度，又不会因为block太大而限制每个SM上的block数量（A100每个SM最多同时驻留32个block）。然后微调时再尝试128、512、1024，看哪个在你的硬件和kernel上效果最好。

边界检查与网格跨度循环

很多时候，你的数据量不是block size的整数倍。比如 n = 1000000，blockSize = 256，gridSize = 3907。最后一个block只有 1000000 - 3906 × 256 = 1000000 - 999936 = 64 个有效元素，剩余的192个线程会访问越界内存。所以你必须做边界检查——这就是 if (idx < n) 的重要性。

但有时你想让每个线程处理多个元素，而不是"一个线程一个元素"。这就是网格跨度循环（grid-stride loop）模式：

// 网格跨度循环 —— 每个线程处理多个元素
__global__ void vecAdd_stride(float *a, float *b, float *c, int n) {
    // 用总线程数作为步长
    int stride = gridDim.x * blockDim.x;

    // 当前线程的起始索引
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    // 以步长循环，直到覆盖所有元素
    while (idx < n) {
        c[idx] = a[idx] + b[idx];
        idx += stride;
    }
}

// 启动：使用较少的block，每个block处理更多数据
// blockSize = 256, gridSize = 128（远小于需要的 3907）
// 但每个线程通过循环处理多个元素

网格跨度循环的好处：

  • 处理任意大小的数据：不用精确匹配grid和block配置到数据大小
  • 更好的负载均衡：每个线程处理的工作量大致相同
  • 便于性能调试：你可以用较少的线程（比如只启动1个block）来调试，然后扩大到完整规模——kernel逻辑不变
边界检查不是可选项，是必选项。 几年前，一个同事在写图像滤波kernel时忘掉了边界检查。大多数图片都是640×480——恰好被某个block配置整除。直到有一天他用了一张641×481的图片——kernel直接在数组边界外写数据，把后面Staging Buffer中的其他数据给覆盖了。结果是一张完全变色的图片，加上一个小时后才定位到的bug。从那以后，他养成了写kernel时先写边界检查的习惯。
真实踩坑故事： 我刚开始用CUDA时，有一次写了一个处理点云数据的kernel。点云大小不固定——有时几万个点，有时几百万个。我用了一个固定的 <<<256, 256>>> 配置但没有做边界检查。大部分时候点云大于65536个点，所以没问题。直到有一天，测试数据只有1000个点——kernel访问了 a[1000] 到 a[65535] 之间的所有内存，而这些内存还在"上一个"数据集的残留数据范围内。我没有得到段错误，但输出了随机值。最可怕的是：这个bug间歇性出现，让我查了整整两天。所以说：永远做边界检查。永远。
