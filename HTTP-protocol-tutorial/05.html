第五章：HTTP状态码

大家好，欢迎来到第五章。前四章咱们把HTTP的基本骨架搭起来了——协议历史、URL、报文结构、请求方法。今天要讲的是HTTP响应里最重要的信息之一：状态码（Status Code）。

可以说，状态码是服务器给你的"一句话反馈"——我的请求处理得怎么样了？是成功了、被拒绝了、还是出错了？服务器用三位数字告诉你。别小看这三个数字，我Debug的时候第一眼看的就是它。毫不夸张地说，读懂状态码是Web开发者的基本功，就跟医生读懂化验单一样。

5.1 状态码分类

HTTP状态码由三位数字组成，首位数字定义了响应的类别。总共五大类：

分类范围含义通俗理解
1xx100-199Informational（信息性）"收到请求，正在处理"
2xx200-299Success（成功）"没问题，搞定了"
3xx300-399Redirection（重定向）"你要的东西在别处"
4xx400-499Client Error（客户端错误）"你的请求有问题"
5xx500-599Server Error（服务端错误）"服务器出状况了"

记忆口诀很简单：1字头接着来，2字头万事OK，3字头换个地方，4字头你搞错了，5字头我崩了。你可能在实际中很少看到1xx和3xx的状态码出现在浏览器开发者工具里——1xx浏览器自动处理了，3xx浏览器帮你跳转完了。但它们确实在背后默默工作。

核心要点：状态码的设计是分层的。首位数字告诉你是哪一类问题，后两位告诉你具体细节。比如看到4xx，你就知道是客户端的问题——先检查自己的请求，别急着骂服务器。这个思维习惯我在排故时用了几百遍了。

5.2 常用2xx——一切正常

2xx系列是最让人放心的状态码。常见的就三个：

200 OK——最标准的结果。请求成功，响应体里就是你想要的数据。无论是HTML页面、JSON数据还是图片下载，成功返回就是200。我跟你打赌，你每天浏览器里看到的请求，70%以上都是200。

201 Created——创建成功。通常用在POST请求后，表示资源创建成功。比如你注册了一个账号、提交了一篇文章，服务器返回201，并在Location头里告诉你去哪里找到这个新资源。用201比200更精确，让你的API消费者明确知道"一个新的东西诞生了"。

204 No Content——请求成功了，但响应体为空。这个在DELETE操作或某些更新操作中很常见。比如删除一条记录，服务器成功删除了，返回204，告诉客户端"删完了，没啥可说的"。我见过有人不管三七二十一全都返回200，然后在body里塞个 {"success": true}，这当然也能跑，但你丢失了语义的精确性。

实战建议：开始设计RESTful API时，尽量使用精确的状态码。200用来表示"读取成功"，201表示"创建成功"，204表示"删除成功"。精准的状态码让你的API更加自描述，也方便监控和日志分析。你的同事会感谢你的。

5.3 3xx重定向——你找的东西在别处

重定向状态码初看有点复杂，核心就是两个维度：永久 vs 临时，是否允许改变请求方法。

状态码含义永久/临时方法是否改变典型场景
301Moved Permanently永久通常改为GET域名变更、HTTP→HTTPS
302Found临时通常改为GET临时跳转、登录后跳转
307Temporary Redirect临时不变临时跳转，要求保持POST
308Permanent Redirect永久不变永久跳转，要求保持POST

这里最坑的一点是301和302在浏览器实现中，几乎一定会把POST改为GET。早期HTTP/1.0只有301和302，规范说浏览器可以自主决定是否更改方法——结果所有浏览器都选了"改成GET"。这就导致如果你用301/302重定向一个POST请求，请求体数据就丢了。

307和308是HTTP/1.1后补充的，明确规定不能改变请求方法。所以如果你需要保持POST方法做重定向，用307（临时）或308（永久）。

踩坑故事：早年做支付系统时，我们用一个中间页面做表单提交，遇到会话过期就302跳转到登录页。结果支付POST请求被302重定向后变成GET，数据全丢了，用户看到的是空白页面。查了整整一个下午才发现是重定向把请求方法改了。后来全换成了307——血的教训。

5.4 4xx客户端错误——别怪服务器，看看你的请求

4xx是日常开发中最常打交道的一类。它们大多意味着"你的请求有问题"。

400 Bad Request——通用客户端错误。请求语法错误、参数校验失败、JSON格式不对，都能丢400。缺点是不够具体，所以很多API宁愿用422 Unprocessable Entity或者自定义响应体来说明具体问题。

401 Unauthorized vs 403 Forbidden——这俩的区分非常重要。401表示"你没登录"或者"身份认证失败"，服务器说"请先告诉我你是谁"。403表示"你登录了，但权限不够"，服务器知道你是谁但说"你不能看这个"。简单说：401要你先登录，403把你拒之门外。我见过大量系统把这俩混用，返回401表示"没权限"，这不准确。

404 Not Found——Web界最知名的状态码。资源不存在。用户输错了URL、链接已经失效、或者攻击者在扫描目录。404有专门的文化现象（404页面设计大赛），但它本质上就是个错误码。

405 Method Not Allowed——请求方法不对。比如API只允许POST，但你发了GET。服务器会在Allow头里列明支持的请求方法。

429 Too Many Requests——你请求得太快了，触发限流了。现在几乎所有公开API都有限流，比如GitHub API一小时5000次。服务器通常还会返回 Retry-After 头告诉你要等多久。

Debug技巧：遇到4xx状态码时，第一步永远检查请求本身——URL对不对？参数格式对不对？Header有没有带对？认证token是不是过期了？80%的情况下，是请求写错了，而不是服务端出了bug。

5.5 5xx服务端错误——服务器出状况了

5xx对用户来说是"网站打不开"，对开发者来说是"赶紧起来修Bug"。

500 Internal Server Error——通用服务端错误。后端代码抛异常了、数据库挂了、配置错了，全都可以返回500。在开发环境你会看到详细的错误堆栈，生产环境则是一个模糊的"服务器内部错误"。这是为了保护服务器信息不泄露。

502 Bad Gateway——网关/代理收到上游服务器的无效响应。典型的架构：浏览器 -> Nginx（反向代理）-> 应用服务器。如果应用服务器挂了，Nginx就会返回502。最常见的情况：应用服务器进程挂了、端口没起来、或者超时了没响应。

503 Service Unavailable——服务暂时不可用。通常是服务器过载了或者正在维护。与502的区别是：502是上游挂了，503是自己挂了。服务器通常返回 Retry-After 头建议客户端多久后再试。

504 Gateway Timeout——网关/代理等上游响应等超时了。跟502的区别：502是上游返回了非法响应，504是上游压根没在时间内返回。最经典的场景是：数据库查询太慢、下游API调用太久，Nginx等不住了（默认60s），于是返回504。

踩坑故事：刚工作时有个后台报表功能，查一个月的数据要跑几十秒。当时架构是Nginx反代Tomcat，Nginx默认60s超时，平时没事。赶上月底数据量大查询超过60s，Nginx就返回504，报表页面白屏了。用户投诉了一堆，我们才发现Nginx的 proxy_read_timeout 默认值没调。所以遇到504，先检查代理超时配置。

5.6 自定义状态码的最佳实践

HTTP协议允许非标准的状态码吗？技术上可以，但强烈不建议。

原因有三：第一，浏览器、代理、CDN等中间设备可能不认识你的自定义码，导致无法正确处理。第二，你的API客户端库可能只识别标准状态码，非标准码会被当作未知错误。第三，监控系统和日志分析工具对标准码有成熟的分类和告警规则。

正确的做法是：用标准状态码表示大类，用响应体给出具体错误详情。

比如用户提交的数据校验失败，你可以返回400 Bad Request，然后在JSON body里写清楚是哪个字段有问题：

HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": "validation_failed",
  "message": "参数校验未通过",
  "details": [
    { "field": "email", "message": "邮箱格式不正确" },
    { "field": "age", "message": "年龄必须在0-150之间" }
  ]
}

这种模式既保持了状态码的通用性，又提供了足够的调试信息。很多大厂（Google、GitHub、Stripe）的API都是这样设计的。
