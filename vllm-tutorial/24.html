第24章 调试与问题排查

跑 vLLM 总会遇到各种奇怪的问题。这一章是我这几年踩过的坑和对应的排查思路——从 CUDA OOM 到 NCCL 超时，从模型加载失败到性能突然退化。好记性不如烂笔头，记下来省得下次再踩。

24.1 常见错误速查表

错误信息可能原因解法
torch.cuda.OutOfMemoryErrorKV cache 或模型权重超出显存降低 gpu-memory-utilization / 启用 FP8 / 减少 max-model-len
NCCL WARN: Failed to initialize shmDocker 共享内存太小加 --ipc=host 或 --shm-size=8g
NCCL timeoutGPU 间通信异常检查 NCCL_DEBUG=INFO 日志，确认 NVLINK 状态
Connection reset by peer客户端在推理完成前断开增加 proxy_read_timeout
Model not foundHF Hub 下载失败/无网络手动下载到本地路径，或用 HF_ENDPOINT 镜像
启动卡在 "Loading model..."模型下载慢或权重转换中等（首次加载 2-10min）；检查磁盘和网络
输出乱码或重复采样参数不当降低 temperature，加 repetition_penalty

24.2 CUDA OOM 排查流程

# 1. 确认显存使用情况
nvidia-smi
# 看 "Memory-Usage" 列: 如果有进程已占大量显存，可能是上一个实例没释放

# 2. 精确显存计算
# 模型权重 + KV cache 预留 + CUDA context ≈ 总显存需求
# 模型权重: ~2 × 参数量 (FP16, 单位 bytes)
# KV cache: 2 × 层数 × KV头数 × 头维度 × max_model_len × max_num_seqs × dtype_bytes

# 3. 渐进式排查
vllm serve model \
    --gpu-memory-utilization 0.80 \   # 先降到 0.80
    --max-model-len 4096 \            # 先减少一半
    --max-num-seqs 64                 # 先减少一半
# 如果 OK → 逐步回增参数

# 4. 查看 vLLM 日志中的显存分配细节
# INFO: "GPU KV cache size: XXX tokens" → 这是 KV cache 占了多少
# 如果这个数很小但 OOM，说明模型权重本身太大

24.3 NCCL 调试

# 启用 NCCL 详细日志
export NCCL_DEBUG=INFO
export NCCL_DEBUG_SUBSYS=INIT,NET

# 关键检查:
# 1. NCCL 版本 vs CUDA 版本兼容吗？
# 2. GPU 之间有 NVLINK 吗？→ nvidia-smi topo -m
# 3. Docker 里 IPC_LOCK 和 SYS_ADMIN 权限加了吗？

# 验证多 GPU 通信
python -c "
import torch
import torch.distributed as dist
dist.init_process_group('nccl')
"

24.4 性能退化排查

症状: "之前 2000 tok/s，突然变成 800 tok/s"

排查清单:
☐ nvidia-smi → GPU 都在满频率跑吗？有没有降频(thermal throttling、P-state)？
☐ 模型变了吗？（新版本可能有更多的层/参数）
☐ 请求 pattern 变了吗？（更多短请求→更多调度开销；更多长请求→KV cache 压力）
☐ Prefix Cache 命中率下降？（检查 vllm:prefix_cache_hit_rate 指标）
☐ free_blocks 经常耗尽？（增加 max-num-seqs 或降低 max-model-len）
☐ CPU 占用异常高？（V0 的 prefix cache 有数据拷贝，V1 修复了）

# vLLM 性能分析
vllm serve model --disable-log-requests  # 关闭请求日志（减少 I/O）
export VLLM_LOGGING_LEVEL=WARNING        # 减少日志级别

真实案例：一个生产服务的吞吐突然从 2100 → 900 tok/s。排查发现上游改了一个参数——原来 system prompt 是固定的（所有请求相同），改成每个用户个性化的。prefix cache 命中率从 78% 跌到 3%，所有 prompt 要重新计算。恢复了 system prompt 的统一性就好转了。

24.5 模型加载问题

# 问题: 下载模型卡住不动
export HF_ENDPOINT=https://hf-mirror.com    # 用镜像
# 或者先手动下载
huggingface-cli download model-id --local-dir /models/
vllm serve /models/

# 问题: safetensors 不兼容
# 如果模型只有 .bin 格式（旧 PyTorch checkpoint）
vllm serve model --load-format pt

# 问题: 模型架构不支持
# 检查 vLLM 支持列表:
# https://docs.vllm.ai/en/latest/models/supported_models/
