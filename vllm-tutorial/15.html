第15章 生产级部署

前面学了不少理论，但只有把 vLLM 推到生产环境、真正接受用户流量的考验，才算毕业。这一章咱们从 Docker 部署、量化策略、高可用设计三个维度，拆解生产级部署的完整方案。

15.1 Docker 部署：一次性搞定的方案

Docker 是 vLLM 官方推荐的部署方式——环境一致、隔离性好、回滚方便。

# 拉取官方镜像
docker pull vllm/vllm-openai:latest

# 标准启动命令（单 GPU）
docker run --gpus all \
    -v ~/.cache/huggingface:/root/.cache/huggingface \
    -p 8000:8000 \
    --ipc=host \
    --name vllm-prod \
    --restart unless-stopped \
    vllm/vllm-openai:latest \
    --model meta-llama/Llama-3-8B \
    --host 0.0.0.0 \
    --port 8000 \
    --gpu-memory-utilization 0.95 \
    --max-model-len 8192 \
    --enable-prefix-caching \
    --kv-cache-dtype fp8
Docker 参数作用必须？
--gpus all暴露所有 GPU 给容器✅
-v ~/.cache/huggingface:...持久化模型缓存（否则每次重启重新下载）✅
--ipc=host共享主机的 IPC 命名空间（NCCL 需要 shm）✅
--restart unless-stopped进程崩溃或宿主机重启后自动恢复推荐
--shm-size=8g增大共享内存（多 GPU 通信需要）多 GPU 时推荐

15.2 GPU 显存规划

生产部署最核心的决策：给 KV cache 留多少显存？

以 A100-80GB + Llama-3-8B 为例：

总显存: 80 GB
模型权重: ~16 GB (占 20%)
CUDA context: ~1 GB (占 1.25%)
剩余给 KV cache: 80 - 16 - 1 = 63 GB

如果 --gpu-memory-utilization=0.95:
  KV Cache 可用 = 63 × 0.95 ≈ 60 GB

每个 token KV cache ≈ 0.125 MB (Llama-3-8B, GQA)
最大并发 8192-token 请求数 = 60 GB / (0.125 MB × 8192) ≈ 58 个并发

建议:
- 实际并发设 max-num-seqs=48（留 20% buffer 应对突发）
- 如果想支持更多并发 → 启用 FP8 KV cache (容量翻倍 → ~116 并发)

15.3 反向代理与负载均衡

生产环境不直接暴露 vLLM，前面加一层 Nginx 或 Envoy：

# Nginx 配置示例
upstream vllm_backend {
    least_conn;  # 最少连接策略
    server 10.0.1.1:8000 max_fails=3 fail_timeout=30s;
    server 10.0.1.2:8000 max_fails=3 fail_timeout=30s;
    server 10.0.1.3:8000 max_fails=3 fail_timeout=30s;
}

server {
    listen 443 ssl;
    location /v1/ {
        proxy_pass http://vllm_backend;
        proxy_read_timeout 300s;  # 长请求超时
        proxy_buffering off;      # SSE 流式需要关 buffering
        proxy_set_header X-Request-ID $request_id;
    }
}

生产提示：如果你启用了 Prefix Caching，尽量用"一致性哈希"或"前缀感知路由"——把同样 system prompt 的请求路由到同一个 vLLM 实例，最大化缓存命中率。简单轮询或最少连接会让缓存效果大打折扣。

15.4 健康检查与就绪探测

# vLLM 内置健康检查端点
curl http://localhost:8000/health     # 返回 200 if ready
curl http://localhost:8000/version    # 返回版本信息

# K8s 就绪探针
readinessProbe:
  httpGet:
    path: /health
    port: 8000
  initialDelaySeconds: 60    # 模型加载可能 30-120s
  periodSeconds: 10
