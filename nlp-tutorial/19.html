第十九章：迁移学习与微调

同学们好！如果你跟完前面十八章走到这里，恭喜你——你已经掌握了NLP的绝大多数理论知识。但有一个问题：你手里有一个7B的大模型，怎么让它做你的特定任务？答案是微调。这一章可能是整门课对你实际工作帮助最大的一章——因为现在业界90%的NLP项目，核心工作就是微调。

先理解一个大背景：预训练大模型就像是一个接受了通识教育的博士生——什么都懂一点，但暂时做不了专业工作。微调就像是送这个博士去参加一个为期两周的专业培训，让ta快速变成一个细分领域的专家。这就是所谓的"最后一公里"问题：通用的知识已经有了，怎么高效地把它适配到你的具体场景？

19.1 全参数微调：经典但昂贵

全参数微调（Full Fine-tuning）是最直接的方法——把预训练模型的所有参数都放开，在你自己的数据集上继续训练。BERT时代的经典论文都是这么做的：用预训练权重初始化，然后在GLUE、SQuAD等任务上更新所有参数。

什么时候该用全参数微调？
  • 你的下游任务和预训练任务差异很大（比如预训练是语言建模，下游是翻译）
  • 你有充足的计算资源（GPU内存够大、训练时间不是瓶颈）
  • 你有足够多的标注数据（几万条以上），不容易过拟合
  • 你需要达到这个任务的SOTA（state-of-the-art）精度
全参数微调的缺点
  • 内存开销大：需要保存所有参数的梯度、优化器状态。一个7B的模型用AdamW微调，仅优化器状态就需要约56GB显存（每个参数存adam的m和v两个动量，加梯度）。
  • 灾难性遗忘（Catastrophic Forgetting）：这是最大的坑——模型在微调数据上表现越来越好，但在通用能力上可能断崖式下降。比如你把Llama-2微调成客服机器人，它可能忘了怎么做数学题。
  • 存储成本高：每个下游任务保存一份完整模型权重。
# 全参数微调示例（使用HuggingFace Trainer）
from transformers import AutoModelForSequenceClassification, Trainer, TrainingArguments

model = AutoModelForSequenceClassification.from_pretrained(
    "bert-base-uncased",
    num_labels=2
)
# 默认所有参数都是可训练的——这就是全参数微调
print(f"可训练参数: {sum(p.numel() for p in model.parameters() if p.requires_grad):,}")
# => 109,483,778（约1.09亿，全量更新）

training_args = TrainingArguments(
    output_dir="./my-finetuned-bert",
    learning_rate=2e-5,
    per_device_train_batch_size=16,
    num_train_epochs=3,
    weight_decay=0.01,
)
trainer = Trainer(model=model, args=training_args, ...)
trainer.train()

19.2 Adapter层：给模型加点"插件"

Adapter的思路和全参数微调完全不同：冻结预训练模型的全部参数，只在Transformer层之间插入少量可训练的小模块。这些小模块就是Adapter层——通常是一个"降维→非线性激活→升维"的 bottleneck 结构。

# Adapter层的内部结构
# 假设hidden_size=768，bottleneck_size=64
#
# 输入 (768维)
#   ↓
# Linear(768 → 64)    ← 降维，参数很少
#   ↓
# ReLU / GELU         ← 非线性
#   ↓
# Linear(64 → 768)    ← 升维回原尺寸
#   ↓
# + 残差连接
#   ↓
# 输出 (768维)
#
# 两个Adapter矩阵总共 768×64 + 64×768 = 98,304 个参数
# 而原始FFN层有 768×3072 + 3072×768 ≈ 470万 个参数
# Adapter只占原始FFN的 ~2% ！

Adapter的优势很明显：每个任务只需要保存一组小Adapter权重（通常是原模型参数的1-5%），不同任务的Adapter可以灵活插拔。推理时稍微多一点点计算量（因为多了额外的前向传播），但训练时的显存压力远小于全参数微调。

缺点是：Adapter增加了推理的延迟（layer多了，计算图变深了），并且需要修改模型结构。这也引出了下一个更流行的方法——LoRA。

19.3 LoRA：低秩适配的数学魔法

LoRA（Low-Rank Adaptation，2021年由微软发表）是目前最流行的参数高效微调方法。它的核心洞察非常优雅：模型在微调过程中的权重更新量（Delta W = W_new - W_pretrained）是低秩的。也就是说，虽然原始权重矩阵可能是768×768，但微调带来的变化量可以用两个小矩阵的乘积来近似。

LoRA的数学直觉

假设一个线性层的权重矩阵 W ∈ R^{d×k}。微调不修改W本身，而是学习一个增量ΔW：

h = Wx + ΔWx = Wx + BAx

其中：
  W ∈ R^{d×k}     ← 预训练权重，冻结不变
  B ∈ R^{d×r}     ← 可训练
  A ∈ R^{r×k}     ← 可训练
  r << min(d, k)  ← r是秩（rank），通常取4-64

参数量对比：
  原始W的参数量：d × k
  LoRA的参数量：d × r + r × k

当 d=k=768, r=8 时：
  原始：768×768 = 589,824
  LoRA：768×8 + 8×768 = 12,288
  节省了 48 倍！

为什么这么小的秩（比如r=8）就够了？因为预训练模型的特征空间本身已经高度结构化。微调只需要在这个已经很好的特征空间里做"微调"，不是"重构"。实验表明，即使在r=1的极端情况下，LoRA的效果也远好于随机基线。这告诉我们：微调带来的变化本质上是低维流形上的偏移。

LoRA的另一个重要技巧是缩放因子：ΔW = (α / r) × BA。推理时，可以把BA合并到W中（W' = W + BA），这样推理延迟就为零了——这比Adapter优雅得多。

# 使用PEFT库实现LoRA微调
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model, TaskType
import torch

# 加载基础模型（以Llama-2 7B为例）
model_name = "meta-llama/Llama-2-7b-hf"
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

# 配置LoRA
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=8,                     # 秩
    lora_alpha=32,           # 缩放因子（一般设为r的2-4倍）
    lora_dropout=0.1,        # 防过拟合
    target_modules=[         # 对哪些模块应用LoRA
        "q_proj", "v_proj",  # 通常只对attention的Q和V投影矩阵做LoRA
        "k_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj",  # MLP层也做
    ],
)

# 用LoRA包装模型（冻结原始权重，只插入低秩矩阵）
peft_model = get_peft_model(model, lora_config)

# 查看参数量
trainable = sum(p.numel() for p in peft_model.parameters() if p.requires_grad)
total = sum(p.numel() for p in peft_model.parameters())
print(f"可训练参数: {trainable:,} / 总参数: {total:,}")
print(f"训练比例: {trainable / total * 100:.2f}%")
# 7B模型一般只训练 ~0.1-1% 的参数

# 训练完成后保存（只保存LoRA权重，几十MB）
peft_model.save_pretrained("./my-lora-adapter")

# 推理时合并权重（可选）
# from peft import PeftModel
# merged_model = peft_model.merge_and_unload()
# 合并后变成一个普通模型，推理没有额外开销

实践建议：LoRA的秩r和α是一对需要调参的好基友。经验法则：r从8开始，α设为r的2-4倍。r太小（≤2）有时会欠拟合，太大（≥64）会失去参数效率的优势。如果你需要更多模型容量，优先加数据质量而不是加r。

19.4 参数高效微调（PEFT）方法全景

LoRA不是唯一的PEFT方法。下面这个表帮你理清主流方法的区别：

方法核心思想可训练参数推理延迟代表作
Adapter 在Transformer层间插入bottleneck模块 ~1-5% 略有增加（额外层） Houlsby et al. (2019)
LoRA 用低秩矩阵逼近权重更新量ΔW ~0.1-1% 无（可合并） Hu et al. (2021)
Prefix Tuning 在输入序列前添加可学习的虚拟token ~0.01-0.1% 无 Li & Liang (2021)
Prompt Tuning 类似Prefix但只在Embedding层加虚拟token ~0.01% 无 Lester et al. (2021)
IA3 用向量对激活值做逐元素缩放 ~0.01-0.1% 无 Liu et al. (2022)
Q-LoRA LoRA + 4-bit量化 ~0.1-1% 无 Dettmers et al. (2023)
选型建议：
  • 如果你需要无额外推理开销：LoRA或Q-LoRA是最佳选择
  • 如果你在极低资源（单卡消费级GPU，如RTX 3090）：Q-LoRA能让65B模型微调成为可能
  • 如果你的任务是文本生成质量要求极高：全参数微调仍可能是上限最高的，但性价比存疑
  • 如果你需要多任务服务：LoRA/Adapter更灵活，每个任务一个小checkpoint，动态切换

19.5 学习率策略：不能一视同仁

微调的学习率设置比从零训练要敏感得多——这是初学者最容易翻车的地方。全参数微调用2e-5这种小学习率，但LoRA因为只训练很少参数，学习率可能需要更大。以下是几个关键策略：

Warmup + Cosine Decay

这是最主流的搭配。Warmup让学习率从0缓慢上升到目标值（通常占总步数的5-10%），避免模型在刚开始训练时被大梯度冲垮。Cosine Decay让学习率按余弦曲线平滑衰减到接近0，让训练后期精细收敛。

# Transformers中的warmup + cosine decay配置
from transformers import get_cosine_schedule_with_warmup

training_args = TrainingArguments(
    learning_rate=2e-5,            # 峰值学习率
    warmup_ratio=0.1,              # 前10%步数做warmup
    lr_scheduler_type="cosine",    # cosine decay
    # 或者用线性衰减
    # lr_scheduler_type="linear",
)
判别式学习率（Discriminative Learning Rates）

这是ULMFiT（2018年）提出的经典技巧：模型的不同层学习不同的事情，因此应该用不同的学习率。底层学到的是通用语言特征（语法、句法），应该用小学习率微调；顶层学到的是任务相关的语义，可以用大学习率。具体做法是给不同层分配不同的学习率，呈倍数关系。

# 判别式学习率示例（手动实现）
# 假设BERT有12层，学习率从底层到顶层递增

def get_layer_lrs(optimizer, base_lr=2e-5, layer_mult=1.5):
    """给BERT每层分配递增的学习率"""
    # 思路：embeddings用最小lr，顶层用最大lr
    param_groups = []
    # Embedding层: base_lr / 2
    # 第0-3层: base_lr
    # 第4-7层: base_lr * 1.5
    # 第8-11层: base_lr * 2.25
    # 分类头: base_lr * 3
    for i, (name, param) in enumerate(model.named_parameters()):
        if "embeddings" in name:
            lr = base_lr / 2
        elif "layer.0" in name or "layer.1" in name or "layer.2" in name or "layer.3" in name:
            lr = base_lr
        elif "layer.4" in name or "layer.5" in name or "layer.6" in name or "layer.7" in name:
            lr = base_lr * layer_mult
        elif "layer.8" in name or "layer.9" in name or "layer.10" in name or "layer.11" in name:
            lr = base_lr * (layer_mult ** 2)
        elif "classifier" in name or "score" in name:
            lr = base_lr * (layer_mult ** 3)
        else:
            lr = base_lr
        param_groups.append({"params": param, "lr": lr})
    return param_groups

# 实际项目中推荐使用HuggingFace的Trainer，
# 但判别式LR需要自定义Optimizer

实际项目里，判别式学习率的好处可能没有LoRA时代那么明显了——因为LoRA训练的参数少，且分散在各层。但如果你是做全参数微调，这仍然是一个非常有效的提分技巧。

学习率重启动（Cosine Annealing with Restarts）

一个有用的trick：让学习率周期性上升再下降，帮助模型跳出局部最优。在LLM微调中偶尔使用，但对于常规下游任务，标准的cosine decay已经够用。

踩坑故事：被学习率坑过的两个月

我刚入行时做过一个对话系统微调项目。用Llama-2 13B，LoRA r=16，默认lr=2e-4。训练了几天，loss降得很漂亮，但生成结果一直在重复同一句话。我换了数据集、调了prompt、甚至怀疑模型坏了。折腾了两周，直到一个同事问："你试过调低学习率吗？"我把lr降到2e-5，不到半天就正常了。教训：loss好看不等于模型好用。微调时loss在降但生成质量差，第一个怀疑的因素就是学习率太高——模型记住了训练数据的表面模式（重复），但没学会真正的对话逻辑。

后来另一位老哥分享了他的故事：他把BERT微调学习率设成1e-3（是从零训练的常用值），结果训练集loss迅速降到接近0，但测试集准确率比随机还低。这就是典型的灾难性遗忘——大学习率把预训练学到的通用特征全冲掉了。从那以后我的铁律：做微调，学习率从2e-5开始，只调低不调高。

19.6 实战微调工作流

最后，给你一个经过验证的微调工作流，不管是BERT还是GPT都可以套用：

第一阶段：准备
  1. 评估基座模型：先用few-shot prompt测试预训练模型在你的任务上的表现。如果few-shot已经很好了，你需要的可能不是微调而是更好的prompt。
  2. 清洗数据：微调的核心不是"更多数据"而是"更好的数据"。每条数据都要检查：标签是否正确？有没有模板泄漏（比如测试集知识出现在训练集里）？
  3. 确定方法：有足够算力、需要最高精度？→全参数微调。单卡、需要快速迭代？→LoRA。需要在有限显存里跑超大模型？→Q-LoRA。
第二阶段：训练
  1. 基线跑通：先用小数据（几百条）跑一个完整流程，确保数据管道、模型加载、评估代码都没问题。
  2. 学习率探索：从2e-5（全参数）或1e-4（LoRA，大模型需要更大LR）开始。跑一个epoch看loss曲线——平滑下降是好的，震荡或飙升说明LR太高了。
  3. 监控验证集：不要只看训练loss，要同步看验证集指标。如果训练loss降但验证集升，说明过拟合了——加dropout、减r（LoRA）、或者增加数据。
  4. 早期停止：验证集指标连续几个epoch不涨就停。大模型微调通常3-5个epoch就够了，不需要跑几十个epoch。
第三阶段：评估与部署
  1. 多维度评估：不要只看一个指标。准确率高不等于生成质量好。做文本生成的话，人工看50条输出比任何自动指标都靠谱。
  2. 灾难性遗忘检查：保留一批通用NLP评测样例（如常识推理、数学），看看微调后这些能力掉了多少。如果有明显下降，考虑混合一些通用数据一起训练。
  3. 合并与部署：LoRA权重合并到原始模型后再部署，避免额外推理依赖。
# 一个完整的LoRA微调脚本骨架
from transformers import (
    AutoModelForCausalLM, AutoTokenizer,
    TrainingArguments, Trainer, DataCollatorForSeq2Seq
)
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training
from datasets import load_dataset

# 1. 加载模型（可以选择量化）
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct",
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

# 2. LoRA配置
peft_config = LoraConfig(
    r=16, lora_alpha=64, lora_dropout=0.05,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    bias="none",
    task_type=TaskType.CAUSAL_LM,
)
model = get_peft_model(model, peft_config)

# 3. 加载数据
dataset = load_dataset("json", data_files="my_data.jsonl")
tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-7B-Instruct")

def format_func(example):
    # 把对话格式化为模型需要的输入格式
    text = f"用户: {example['question']}\n助手: {example['answer']}"
    return tokenizer(text, truncation=True, max_length=2048)

tokenized_dataset = dataset.map(format_func)

# 4. 训练
training_args = TrainingArguments(
    output_dir="./qwen-lora",
    learning_rate=2e-4,                   # LoRA用稍大的lr
    warmup_ratio=0.1,
    lr_scheduler_type="cosine",
    per_device_train_batch_size=4,
    gradient_accumulation_steps=8,        # 等效batch_size=32
    num_train_epochs=3,
    logging_steps=10,
    save_strategy="epoch",
    fp16=False,
    bf16=True,
)
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized_dataset["train"],
    data_collator=DataCollatorForSeq2Seq(tokenizer=tokenizer),
)
trainer.train()

# 5. 保存
model.save_pretrained("./qwen-lora-final")
tokenizer.save_pretrained("./qwen-lora-final")

核心哲学：微调的本质不是"训练"，而是"适配"。你不需要让模型重新学习语言——它已经会了。你要做的是告诉它：你的知识该用在这些地方。所以微调里数据质量远重要于数据量，学习率策略远重要于模型架构创新。少即是多——用最少的数据、最少的参数改动、最少的训练步数，达到最好的适配效果。

掌握了微调，你就拥有了让通用大模型为你定制服务的能力。但微调不是万能的——每次微调都需要数据、算力、时间。下一章我们聊聊另一种更轻量的"调教"方式：提示工程。有时候，一个好的prompt比一次微调更有效——但什么时候用prompt，什么时候该微调，正是你接下来要学的核心判断力。
