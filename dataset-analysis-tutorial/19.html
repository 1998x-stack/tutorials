第19章：NLP经典数据集

自然语言处理可能是AI所有分支中"数据集种类最丰富"的领域。不像计算机视觉有ImageNet一统江湖，NLP的任务类型千差万别——分类需要标签、生成需要参考答案、问答需要篇章、NER需要BIO标注。一个做文本分类的数据集和一个做摘要的数据集，它们的结构、规模、加载方式几乎没有任何共同点。

这种多样性既是NLP的魅力，也是它的陷阱。选错数据集比选错模型更致命——因为数据集从根本上决定了你的模型能学到什么、不能学到什么。本章将系统性介绍NLP领域最核心的几类数据集：从语言理解基准到问答、摘要、序列标注和情感分类，逐一讲解它们的结构、加载方式、评测指标和选型策略。

本章路线图：我们将按照"理解→检索→生成→标注→分类"的逻辑线，覆盖NLP五大数据品类。每个品类都会给出HF Hub上的加载标识符、典型样例和常见陷阱，让你在实战中能快速找到并正确使用最合适的数据集。

通用语言理解基准 — GLUE与SuperGLUE

如果你只想用一个系列的数据集来衡量模型的语言理解能力，那就是 GLUE（General Language Understanding Evaluation）。GLUE由纽约大学、华盛顿大学和DeepMind等机构联合推出，包含9个自然语言理解子任务，覆盖了句法、语义、推理、相似度等多个维度。2019年提出的 SuperGLUE 则是它的"进阶版"，去掉了已经饱和的任务，替换成更难的新任务。

GLUE的加载方式非常特别——同一个数据集入口通过 configuration 参数选择子任务：

from datasets import load_dataset

# 加载GLUE的MRPC子任务（释义检测）
mrpc = load_dataset("glue", "mrpc")
print(mrpc["train"][0])
# {'sentence1': '...', 'sentence2': '...', 'label': 0, 'idx': 0}

# 加载SST-2（情感分类）
sst2 = load_dataset("glue", "sst2")
print(sst2["train"][0])
# {'sentence': '...', 'label': 1, 'idx': 0}

下表是GLUE全部9个子任务的速览：

子任务 缩写 任务类型 评估指标 训练样本数
语料库语言可接受性 CoLA 二分类（语法判断） Matthews相关系数 8,551
斯坦福情感树库 SST-2 二分类（正面/负面） Accuracy 67,349
微软研究释义语料 MRPC 二分类（是否同义） F1 / Accuracy 3,668
Quora问题对 QQP 二分类（是否重复） F1 / Accuracy 363,846
语义文本相似度基准 STS-B 回归（相似度1-5分） Pearson/Spearman 5,749
多类型自然语言推理 MNLI 三分类（蕴涵/矛盾/中立） Accuracy（matched/mismatched） 392,702
问答自然语言推理 QNLI 二分类（是否包含答案） Accuracy 104,743
文本蕴涵识别 RTE 二分类（蕴涵/非蕴涵） Accuracy 2,490
Winograd自然语言推理 WNLI 二分类（指代消解） Accuracy 634

有趣的是，WNLI只有634条训练样本，是GLUE中最小的子任务——因为Winograd句子本身构造困难。而SuperGLUE更进一步，引入了BoolQ（布尔问答）、CB（蕴涵）、COPA（因果推理）等更硬核的任务。加载方式与GLUE相同：

# SuperGLUE的例子
boolq = load_dataset("super_glue", "boolq")
copa = load_dataset("super_glue", "copa")
选型建议：如果你的模型在GLUE上已经接近人类基线（例如RoBERTa级别），别再死磕了——转去SuperGLUE。GLUE的很多子任务已被刷到接近天花板，区分度越来越低。SuperGLUE的BoolQ和COPA对推理能力要求更高，更适合评估大模型。

问答与阅读理解

问答数据集大概是NLP领域"最容易出坑"的数据类型。因为问答任务的变体太多了：抽取式vs生成式、单篇章vs开放域、有答案vs无答案、单轮vs多轮。每一个设计选择都会改变你的模型行为。

SQuAD — 最经典的抽取式问答

SQuAD（Stanford Question Answering Dataset）是问答领域最著名的数据集，目前有两个版本。它们的核心区别在于：v1.1 假设所有问题在给定段落中都有答案，而 v2.0 引入了无答案问题（unanswerable questions）——这是模型从"搜索机器"变成"理解系统"的关键一步。

下面是两个版本的加载对比：

# SQuAD v1.1 — 所有问题都有答案
squad_v1 = load_dataset("squad", split="train")
print(squad_v1[0])
# {'id': '...', 'title': '...', 'context': '...',
#  'question': '...', 'answers': {'text': ['...'], 'answer_start': [123]}}

# SQuAD v2.0 — 可能无答案
squad_v2 = load_dataset("squad_v2", split="train")
print(squad_v2[0])
# 相比v1多了 'is_impossible': False 字段
# 当 is_impossible=True 时，answers 为空列表
踩坑实录：某智能客服团队用SQuAD v1.1训练了一个问答模型，内测时效果看起来不错。上线后却发现一个严重问题——当用户问"我不知道"或者"我不清楚"时，模型总是强行从知识库中找出一段最相关的文字作为"答案"，而不是说"我无法回答"。这是因为v1.1的训练数据里每一问必有答，模型根本没有学会"拒绝回答"这个行为。换成SQuAD v2.0重新训练后，模型学会了在无答案时输出空响应，用户体验明显提升。这个教训告诉我们：训练数据的设计直接影响模型的行为边界。

开放域问答 — Natural Questions

如果说SQuAD是"闭卷考"，Natural Questions（NQ）就是"开卷考中的开卷考"。它包含Google搜索的真实用户查询，每个问题对应一个维基百科页面作为上下文。NQ的答案是自由格式的，不像SQuAD那样限定为段落中的连续片段。这使得NQ更接近真实场景，但也更难评测。

from datasets import load_dataset

nq = load_dataset("natural_questions", split="train")
print(nq[0].keys())
# dict_keys(['id', 'question', 'document', 'annotations', ...])
# document 包含整个维基页面的HTML，需要解析

注意NQ的 document 字段是HTML格式的，不能直接当文本用——你得做HTML清洗。这是很多NLP数据集的一个隐藏门槛：你以为加载完就能用，其实还需要大幅预处理。

问答数据集选型口诀：SQuAD v1.1用于验证模型的精确提取能力，SQuAD v2.0用于训练模型的"拒绝回答"机制，Natural Questions用于检测开放域搜索+理解的综合能力。不要拿v1.1的模型直接上线——除非你能保证所有用户问题都有确定答案。

文本生成与摘要

文本生成数据集的特殊性在于：输出空间是无限的。不像分类任务只有几个标签，摘要任务需要与原文语义一致的高质量短文本。这对数据标注质量的要求极高。

CNN / Daily Mail — 新闻摘要的工业标准

CNN / Daily Mail 是文本摘要领域使用最广泛的数据集，包含约30万篇新闻文章及其对应的高亮摘要。文章的摘要由新闻网站编辑提供，质量比自动生成的基线高得多。加载方式很直接：

cnn_dailymail = load_dataset("cnn_dailymail", "3.0.0", split="train")
print(cnn_dailymail[0])
# {'article': '...全文...', 'highlights': '...摘要...', 'id': '...'}

# 注意版本参数 "3.0.0" 不可省略
# 不同版本的数据清洗程度和文章数量不同

XSum — 极致摘要

如果说CNN/Daily Mail是"压缩版"摘要，XSum就是"嗜血版"——它要求模型用一句话概括整篇BBC新闻。这种极端压缩使XSum成为评估模型"理解核心信息"能力的试金石。

xsum = load_dataset("xsum", split="train")
print(xsum[0])
# {'document': '...全文...', 'summary': '一句话摘要', 'id': '...'}

XSum的挑战在于：很多新闻的"核心信息"在文章不同位置的表述可能完全不同，模型必须判断什么是最重要的。CNN/Daily Mail的摘要则可以包含2-3个句子，压力小很多。

ROUGE与BLEU — 生成质量的两把尺

评测生成类任务离不开这两个指标：

指标 全称 核心思想 适用场景
ROUGE Recall-Oriented Understudy for Gisting Evaluation 计算参考摘要中n-gram被生成文本覆盖的比例 文本摘要
BLEU Bilingual Evaluation Understudy 计算生成文本中n-gram出现在参考文本中的精度 机器翻译、文本生成

简单说：ROUGE偏召回（"模型提到的重要信息够不够多"），BLEU偏精度（"模型生成的文本是不是够准确"）。在实际使用中，一般两者都会报告，同时还会加上人工评估——因为自动指标和人的感知差距不小。

使用建议：评估摘要模型时，不要只看ROUGE-1和ROUGE-2，ROUGE-L（基于最长公共子序列）更能反映句子级流畅度。另外注意，ROUGE对同义词敏感——把"汽车"写成"车辆"不算匹配，所以高分不一定代表高质量。

序列标注 — NER与更多

序列标注任务包括命名实体识别（NER）、词性标注（POS）、语义角色标注等。这类数据集的"数据格式"比文本分类复杂得多——每个token都需要一个标签。

CoNLL-2003 — NER的入门基准

CoNLL-2003 是最著名的英文NER数据集之一，标注了四种实体类型：PER（人名）、LOC（地点）、ORG（组织）和 MISC（其他专名）。它使用经典的 BIO标注格式——B表示实体的开始（Begin）、I表示实体的内部（Inside）、O表示非实体（Outside）。

conll = load_dataset("conll2003", split="train")
print(conll[0])
# {'tokens': ['EU', 'rejects', 'German', 'call', ...],
#  'pos_tags': [22, 42, 16, 21, ...],
#  'chunk_tags': [11, 21, 11, 12, ...],
#  'ner_tags': [3, 0, 7, 0, 0, ...]}

# ner_tags 是整数编码，需要映射回标签字符串
ner_labels = conll.features["ner_tags"].feature.names
print(ner_labels)
# ['O', 'B-PER', 'I-PER', 'B-ORG', 'I-ORG',
#  'B-LOC', 'I-LOC', 'B-MISC', 'I-MISC']

这里有一个容易忽略的细节：datasets库会自动将标签编码为整数。要拿到可读的标签名，一定要通过 .features["ner_tags"].feature.names 获取映射表。不少人在训练时直接用整数ID，等到分析预测结果时才傻眼——不知道每个数字代表什么。

OntoNotes 5.0 — 工业级多任务标注

如果说CoNLL-2003是入门练手，OntoNotes 5.0 就是工业界标准。它包含了更丰富的标注信息：除了NER（18种实体类型），还有句法树、语义角色、共指消解和词义消歧。一个样本同时包含多项标注，适合多任务学习。缺点是规模大（约300万词）且加载配置复杂：

# OntoNotes 需要指定语言和分片
ontonotes = load_dataset("onto_notes", "english_v12", split="train")
print(ontonotes[0].keys())
# dict_keys(['part_id', 'words', 'pos_tags', 'parse_tree',
#            'predicate_lemmas', ...])
重要提示：OntoNotes 5.0的许可证不是完全免费的——它属于LDC（Linguistic Data Consortium），学术研究可以免费使用（需通过HF Hub授权），但商业使用需要额外购买。在 HF Hub 上检查数据集的许可证字段，永远不要假设"能在HF上加载 = 可以随便用"。

情感分析与文本分类

文本分类是NLP最常见的入门任务，对应的数据集也是最容易加载的——基本就是"文本+标签"的简单结构。但"简单"不意味着"不重要"。情感分析数据集的标签分布、领域偏移和语言差异会深刻影响模型在实际场景中的表现。

IMDB Reviews — 情感二分类经典

IMDB Reviews 包含5万条电影评论，正负面各半，是情感分析的标准入门数据集。它的加载极其简单，适合作为任何NLP流程的第一个"呼吸测试"：

imdb = load_dataset("imdb", split="train")
print(imdb[0])
# {'text': 'I rented this movie...', 'label': 0, 'label': 0}
# label: 0=负面, 1=正面

# 检查标签平衡性
from collections import Counter
print(Counter(imdb["label"]))
# Counter({0: 12500, 1: 12500})  # 完美平衡！
实战提醒：IMDB是高度"干净"的数据集——没有缺失值、没有标签噪音、正负平衡。但在真实场景中，你遇到的数据集很少这么规整。如果你用IMDB训练的模型直接部署到真实评论系统上，准确率往往会掉10-15个百分点，因为真实评论的写作风格、长度分布和情感表达方式与IMDB差异很大。这就是分布偏移（distribution shift）的典型表现。

Amazon Reviews — 多语言多星级

Amazon Reviews 比 IMDB 更进一步：它包含多种语言和5星制评分（1-5星），而不是简单的正负二分类。这让你可以做更细粒度的情感分析。加载时可以根据需要选择子集：

# 加载Amazon Reviews 5星分类子集（英文）
amazon = load_dataset("amazon_reviews_multi", "en", split="train")
print(amazon[0])
# {'review_title': '...', 'review_body': '...', 'stars': 5.0,
#  'product_category': '...', 'language': 'en'}

# 转换为三分类用于更稳健的情感分析
def to_sentiment(example):
    if example["stars"] <= 2:
        example["sentiment"] = "negative"
    elif example["stars"] == 3:
        example["sentiment"] = "neutral"
    else:
        example["sentiment"] = "positive"
    return example

amazon = amazon.map(to_sentiment)

注意这里用 .map() 做标签转换——这是datasets库最常用的数据预处理模式。10万条评论，一行map搞定，Arrow后端自动处理并行化。

NLP数据集选型指南

面对琳琅满目的NLP数据集，新手最容易犯的错误是"什么热门用什么"。正确的做法是：根据任务类型、数据规模和领域匹配度来选择。下表是一个快速参考：

你的任务 首选数据集 加载标识 训练估算时间（BERT-base）
通用语义理解 GLUE / SuperGLUE load_dataset("glue", ...) 2-6小时
抽取式问答 SQuAD v2.0 load_dataset("squad_v2") 3-4小时
开放域问答 Natural Questions load_dataset("natural_questions") 8-12小时
文本摘要 CNN/Daily Mail load_dataset("cnn_dailymail", "3.0.0") 6-10小时
命名实体识别 CoNLL-2003 load_dataset("conll2003") 1-2小时
情感分析（入门） IMDB Reviews load_dataset("imdb") 1-2小时
情感分析（精细） Amazon Reviews load_dataset("amazon_reviews_multi", "en") 4-8小时
规模建议："训练时间"取决于你的硬件、批大小和训练轮数。上表的估算基于单卡V100和标准超参数配置。如果数据集特别大（比如Natural Questions），强烈建议先用 streaming=True 配合 .take(1000) 做一个快速基线测试，确认数据质量没问题再全量训练。

选好数据集只是第一步。第20章我们将转向计算机视觉领域——那里的数据形态完全不同：不再是文本和标签，而是像素矩阵、边界框和分割掩码。准备好切换大脑的"模态开关"了吗？
