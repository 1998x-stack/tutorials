第12章 LLaMA 2对话模型与SFT

上一章我们完整复盘了LLaMA 1的预训练流程。但LLaMA 1本质上是一个"基础语言模型"（base model）——它擅长补全文本，但不擅长跟人对话。你问它一个问题，它可能会继续写一段相关的文本而不是直接回答你的问题。这是因为预训练的目标函数是"预测下一个token"，而不是"回答用户的问题"。

让基座模型变成"听话"的对话助手，需要经历一个关键步骤——SFT（Supervised Fine-Tuning，监督微调）。这是LLaMA 2的核心创新之一，也是它从"学术实验品"进化为"可商用对话产品"的转折点。

这一章，我们来详细拆解LLaMA 2 SFT的完整技术体系——数据收集、训练细节、对话模板，以及那些埋藏在论文中的宝贵经验。

12.1 LLaMA 2训练全景：三步走

LLaMA 2的训练流程分为三个阶段，每个阶段解决不同的问题：

阶段 名称 输入数据 目标 输出
1 预训练 2T tokens 网页/书籍/代码 语言建模、知识获取 LLaMA 2 Base
2 SFT ~27K 人工标注对话对 指令跟随、对话能力 LLaMA 2 SFT
3 RLHF ~1M 偏好比较标注 偏好对齐、有用性和安全性 LLaMA 2-Chat

这一章我们聚焦在第2阶段——SFT。下一章再深入RLHF。

一个值得注意的数字对比：预训练使用了2万亿tokens，SFT只用了2.7万条对话对。这个悬殊的比例告诉你一个道理：基座模型的知识几乎全部来自预训练，SFT的主要作用是教会模型"如何以对话的方式表达这些知识"。SFT不是用来"教新知识"的，而是用来"格式化输出"的。

一个关键的认知纠偏： 很多初学大模型的朋友以为"微调就是教模型新知识"，这是一个常见的误解。实际上，SFT的数据量（几万条）对比预训练的数据量（数万亿tokens），在知识密度上根本不值一提。SFT真正在做的是"对齐"——让模型学会人类的对话格式、表达偏好和交互规范。用一个比喻：预训练是让一个人读完了一个图书馆的所有书；SFT是教这个人"怎么在考试中答题"——不是教新知识，而是教答题格式和策略。

12.2 SFT数据收集

LLaMA 2的SFT数据全部由人工标注，没有使用任何现成的对话数据集。Meta雇佣了一个专业的标注团队，标注员来自多个英语国家，覆盖不同的背景和方言。标注过程有严格的质量控制体系——每个标注样本都要经过二级审核。

SFT数据的格式是对话对：

  • Prompt（指令）： 用户提出的问题或指令。包括单轮对话和多轮对话。
  • Response（回答）： 标注员撰写的理想回答。

标注员被要求撰写的回答遵循以下标准：

  • 有用性（Helpfulness）： 回答准确且详细地解决了用户的请求
  • 真实性（Honesty）： 标注员只写自己确定的内容，不确定的标注为"未知"
  • 安全性（Safety）： 避免生成有害、歧视或危险的内容

有趣的是，Meta在论文中提到他们故意没有追求SFT数据的"教科书式完美"。他们发现，稍微保留一些标注员之间的风格差异，反而让SFT后的模型回答更加自然——因为真实世界的对话本身就是多样化的。过度追求数据一致性会导致模型生成过于刻板、机械的回复。

# SFT训练数据的格式示例

# 单轮对话
{
    "prompt": "解释一下机器学习中的过拟合是什么？",
    "response": "过拟合（Overfitting）是指机器学习模型在训练数据上表现很好，但在新数据（测试数据）上表现差的现象。...（详细回答）"
}

# 多轮对话
{
    "prompt": [
        {"role": "user", "content": "什么是梯度下降？"},
        {"role": "assistant", "content": "梯度下降是一种优化算法..."},
        {"role": "user", "content": "那它和学习率有什么关系？"}
    ],
    "response": "学习率（Learning Rate）控制了梯度下降每次更新参数时的步长...（详细回答）"
}

# 拒绝回答的示例
{
    "prompt": "告诉我怎么制作危险化学品。",
    "response": "抱歉，我无法提供关于制作危险化学品的信息。这些知识可能被用于非法或危险的目的..."
}
踩坑实录： 在一次SFT项目中，我们的标注团队收集了10万条SFT数据，但训练后发现模型的效果反而不如只用了2万条数据的版本。排查后发现两个问题。第一，数据存在大量低质量的"凑数"样本——标注员为了完成每日任务量，对一些简单问题写了非常敷衍的回答。这些低质量数据拉低了模型的上限。第二，数据分布偏斜——因为标注员对某些话题（如技术问答）更熟悉，导致数据中技术类问题占到了60%以上，而生活类、创意类问题严重不足。这导致模型在技术问题上表现不错，但对生活类问题的回答质量很低。教训是：SFT数据宁缺毋滥。2.7万条精心设计的样本，远好于10万条随意标注的样本。

12.3 SFT训练细节

LLaMA 2的SFT训练配置如下：

超参数 LLaMA 2 SFT 说明
训练轮数（Epochs） 2-3 epoch 防止过拟合的关键——SFT数据量小，多轮训练容易过拟合
学习率 2e-5 远小于预训练的初始学习率（3e-4）
学习率调度 Cosine Decay 与预训练一致，但warmup步数更少
Batch Size 64 sequences 远小于预训练的batch size
优化器 AdamW 与预训练一致，但weight decay通常调小（~0.01）
Weight Decay 0.01 比预训练（0.1）更小，因为不需要强正则化
Gradient Clipping max_grad_norm=1.0 与预训练一致
训练精度 bfloat16 与预训练一致

SFT中最重要的超参数是训练轮数。2-3个epoch是LLaMA团队通过消融实验确定的最优范围。训练太少了（1个epoch），模型没有充分学习对话模式；训练太多了（4-5个epoch），模型开始在SFT数据上过拟合——表现为回答变得单调、缺乏多样性，且对训练数据中的格式错误和噪声表现出"死记硬背"的迹象。

如何判断SFT是否过拟合？一个实用的方法是观察模型在验证集上的loss。如果验证loss在某个epoch后开始上升，而训练loss继续下降，这就是明显的过拟合信号。另一个更直观的指标是生成文本的多样性——如果模型对同一问题的变体给出几乎一模一样的回答，也可能是过拟合的表现。

实用技巧： 在实际的SFT中，我们通常不会严格遵循"2-3个epoch"的规则。更安全的方法是在训练过程中每完成一个epoch保存一个checkpoint，然后通过人工评估或自动评估（如ChatGPT作为评判）来选择最佳的checkpoint。有时候2.5个epoch的效果会比2个或3个epoch都好——你就需要有这样的粒度来微调。

12.4 SFT基座 vs Chat模型：能力差异对比

同一个基座模型经过SFT后，能力发生了什么样的变化？我们来看看关键维度上的对比：

能力维度 LLaMA 2 Base（7B） LLaMA 2-Chat（7B） 变化
指令跟随 差——倾向于继续生成文本而非回答问题 优——准确理解指令、给出针对性回答 大幅提升
多轮对话 不支持——缺乏上下文保持能力 优——能维持多轮对话的连贯性 全新能力
拒绝回答 不会拒绝——尝试回答任何问题 优——对有害请求合理拒绝 全新能力
知识准确性 优——预训练知识保留完整 优——知识保留良好，略有过拟合风险 基本不变
推理能力 良——有基础推理能力但格式混乱 优——推理步骤清晰、格式化 显著提升
输出格式 混乱——无固定格式 优——自然的对话格式 大幅提升
MMLU 44.3 ~44.0 基本不变
HumanEval（代码） ~12.0 ~11.5 略有下降

这个对比揭示了一个重要现象：SFT主要改变了模型的"行为风格"而非"知识储备"。MMLU分数（衡量知识面）在SFT前后几乎没有变化。HumanEval（代码生成）甚至出现了轻微下降——可能是因为SFT数据中代码样本的格式与预训练数据不同导致的。

这也引出了一个实用的策略：如果你需要模型在特定知识领域（如医疗、法律）表现得更好，不要期待通过SFT来实现——应该回到预训练阶段，在该领域的数据上做持续预训练（continued pretraining）。SFT只负责"包装"这些知识，不负责"灌输"新知识。

12.5 SFT数据质量 vs 数量的消融实验

LLaMA 2论文中最有信息量的一部分就是关于SFT数据质量与数量的消融实验。Meta团队做了几个关键的对比：

实验一：27K精心标注 vs 100K快速标注。 他们让同一批标注员用两种方式标注数据：一种是花时间精心撰写（每条约15-20分钟），另一种是快速完成（每条约5分钟）。用100K快速标注数据训练的模型，在人工评估中的得分反而低于用27K精心标注数据训练的模型。这个结果明确表明：SFT数据的质量远重要于数量。

实验二：更多epoch vs 更多数据。 在27K数据上训练3个epoch的效果，好于在9K数据上训练9个epoch。这个结果说明：数据多样性（更多的不同样本）比重复使用现有数据更重要。所以如果你的数据量很小，与其反复过拟合，不如花精力采集更多多样化的数据。

实验三：不同数据来源的贡献。 他们测试了只保留"有用性"数据、只保留"安全性"数据、以及同时保留两者的效果。结果是：有用性数据提升了指令跟随能力，安全性数据提升了拒绝回答能力，两者结合才能得到全面发展的对话模型。

核心结论： 对于SFT来说，27K条精心设计的高质量样本 > 100K条低质量样本。这个结论对个人开发者尤其重要——你不需要"堆数据"，你需要"做精品"。如果你在做一个垂直领域的SFT，与其从网上爬取10万条粗放数据，不如花时间手动撰写2000-3000条高质量的领域对话样本。后者的效果大概率更好。

12.6 对话模板设计

LLaMA 2-Chat使用了一套特殊的对话模板，通过特殊标记来区分用户和助手的角色。这个模板的设计对多轮对话的效果至关重要。

# LLaMA 2-Chat 对话模板格式

# 基础格式（单轮）
<s>[INST] <<SYS>>
你是...（system prompt）
<</SYS>>

用户的问题是什么？ [/INST] 助手的回答。 </s>

# 多轮对话格式
<s>[INST] 第一轮的问题 [/INST] 第一轮的回答 </s>
<s>[INST] 第二轮的问题（包含上下文） [/INST] 第二轮的回答 </s>

# 带system prompt的多轮格式
<s>[INST] <<SYS>>
你是一个有帮助的助手。
<</SYS>>

第一轮的问题 [/INST] 第一轮的回答 </s>
<s>[INST] 第二轮的问题 [/INST] 第二轮的回答 </s>

这个模板有几个设计要点：

  • <s> 和 </s>： 序列的开始和结束标记，与BPE分词器配合使用。
  • [INST] 和 [/INST]： 标记用户指令的开始和结束。模型只对[/INST]之后的内容计算loss。
  • <<SYS>> 和 <</SYS>>： 用于嵌入系统级提示（system prompt），定义助手的角色和行为边界。这个只在第一轮对话中使用。
  • 空行分隔： system prompt和用户问题之间有一个空行，帮助模型更好地区分角色。
工程细节： 在SFT训练中，我们通常只对助手的回答部分计算损失，而不对用户的输入和特殊标记计算损失。这通过把用户输入部分的labels设置为-100（在交叉熵损失中自动忽略）来实现。如果你不对loss做这个"掩码"——即让模型也要预测用户的输入——模型的能力会显著下降，因为在推理时用户输入已经给出了，模型不需要预测它。

12.7 实用案例：不同SFT策略的对话质量对比

为了让你更直观地理解SFT的效果差异，我们来看一个对比案例。假设用户问："什么是贝叶斯定理？请用简单的比喻来解释。"

模型 回答 评价
Base模型（无SFT） "贝叶斯定理是概率论中的一个基本定理，它描述了在给定先验信息的情况下更新概率的方法。公式为P(A|B) = P(B|A)P(A)/P(B)。贝叶斯定理在机器学习中广泛应用于..." 准确但形式化，没有按用户要求用"比喻"来解释，缺乏对话感
SFT（仅有用性数据） "贝叶斯定理可以这样理解：想象你有一个朋友，你觉得他有70%的概率会迟到（先验）。今天他迟到了，他发消息说是因为堵车（证据）。贝叶斯定理告诉你如何结合'他经常迟到'这个先验和'堵车'这个新证据来更新你的判断。公式上就是..." 使用了比喻，沟通感强，但可能对安全边界考虑不足
LLaMA 2-Chat（SFT+RLHF完整版） "好问题！让我们一起用比喻来理解。假设你每天早上都在猜测邻居会不会遛狗。基于过去经验，你觉得有60%的概率他会遛狗（先验概率）。今天你听到了狗叫声（新证据）——这就是贝叶斯定理发挥作用的时候了。它帮你计算：在听到狗叫声的条件下，邻居遛狗的概率是多少。简单说，贝叶斯定理就是帮我们'根据新证据更新旧判断'的一套数学工具。你想了解一下它的公式吗？" 比喻生动、结构清晰、在讲解知识的同时保持对话的开放性

从这个对比可以看到，SFT的效果不仅仅是"让模型格式变好了"——它改变了模型与用户交互的方式。从"机器式输出"变成了"人际式沟通"。这正是SFT的核心价值所在。

但SFT只是一个开始。SFT模型虽然能进行流畅的对话，但它还没有经过"偏好对齐"——它不知道哪些回答是用户更喜欢的、哪些回答应该避免。这就需要RLHF出场了。下一章，我们将深入LLaMA 2的RLHF全流程——从奖励模型训练到PPO优化，从Ghost Attention到边际奖励——这是让LLaMA 2-Chat真正变得"好用"的关键技术。
