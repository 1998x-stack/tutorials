"差不多"的重复也是重复

大家好，我是你们的讲师。上一章我们解决了"完全一样"的问题，但现实世界中更常见的是"差不多一样"——同一篇新闻被不同媒体改写，同一段维基百科解说被不同的博客以略有不同的措辞引用，同一个 GitHub README 被翻译成不同的语言版本。

我在 2024 年做数据质量分析时遇到过一个典型案例：一个 1TB 的中文语料，经过精确去重后，我用 MinHash 做了模糊去重，结果发现其中一篇关于"Transformer 注意力机制"的科普文章存在 37 个不同的"变体"——有的改了标题，有的加了开头段，有的修改了部分措辞。虽然它们都不完全一样，但信息含量几乎是相同的。如果把这些模糊副本全部保留，模型在注意力机制这个话题上的"学习信号"会被严重稀释。

这一章我们来深入讲解工业界最常用的模糊去重方法——MinHash + LSH（局部敏感哈希）。这个组合是许多大规模预训练数据集（包括 C4、FineWeb、RedPajama）去重管线的核心。

11.1 MinHash 的直觉

MinHash 的核心思想是：用一组紧凑的"签名"来估计两个文档之间的 Jaccard 相似度。

Jaccard 相似度定义为两个集合的交集大小除以并集大小。对于文档来说，我们可以把文档表示为它包含的 n-gram 集合（比如所有连续的 5 个 token）：

J(A, B) = |A ∩ B| / |A ∪ B|

举例：
  文档 A: {"我今天", "今天学", "天学习", "学习了"}
  文档 B: {"我今天", "今天学", "天学习", "学习Py", "Python"}
  J(A, B) = 3 / (4 + 2) = 3/6 = 0.5

MinHash 的巧妙之处在于：如果你对每个 n-gram 集合应用一个随机哈希函数，并取集合中最小的哈希值作为签名，那么两个文档的签名相等的概率恰好等于它们的 Jaccard 相似度。

使用多个独立的哈希函数（比如 256 个），你就得到了一组签名。两个文档在多个哈希函数上签名相同的比例，就是它们 Jaccard 相似度的无偏估计。

import hashlib
import random

def shingle(text: str, k: int = 5) -> set:
    """将文本分片为 k-shingles（字符级）"""
    return set(text[i:i+k] for i in range(len(text) - k + 1))

def minhash_signature(shingles: set, num_hashes: int = 256) -> list:
    """计算 MinHash 签名"""
    # 为每个哈希函数生成随机种子
    seeds = [hashlib.md5(str(i).encode()).hexdigest()[:8] for i in range(num_hashes)]
    signature = []

    for seed in seeds:
        # 对每个 shingle 计算 hash(seed + shingle)
        min_hash = min(
            int(hashlib.md5((seed + s).encode()).hexdigest()[:8], 16)
            for s in shingles
        )
        signature.append(min_hash)

    return signature

def estimate_jaccard(sig_a: list, sig_b: list) -> float:
    """通过签名估计 Jaccard 相似度"""
    matches = sum(1 for a, b in zip(sig_a, sig_b) if a == b)
    return matches / len(sig_a)

11.2 LSH（局部敏感哈希）：从 N² 到近似线性

MinHash 解决了"高效计算两个文档相似度"的问题，但在海量数据集中，你不知道哪些文档是相似的——如果对所有文档做两两比对，复杂度是 O(N²)，对于数亿个文档来说完全不可行。

LSH（局部敏感哈希）通过一种称为"banding"的技术解决了这个问题。它将 MinHash 签名分成 b 个 band，每个 band 包含 r 个签名值。对于每个 band，我们将 band 内的 r 个值合并成一个新的哈希值——称为"桶"。如果两个文档在任意一个 band 上落入同一个桶，它们就被认为是候选相似对。

关键的数学性质是：两个文档在至少一个 band 上碰撞的概率是：

P = 1 - (1 - J^r)^b

其中 J 是 Jaccard 相似度，b 是 band 数，r 是每个 band 的行数。

通过调节 b 和 r，你可以控制相似度阈值。例如，如果你希望找到 Jaccard > 0.8 的相似文档对，可以设置 b = 20, r = 5，这样相似度 < 0.5 的文档几乎不会被选为候选，而相似度 > 0.8 的文档有很大概率被检测到。

band 数 (b) 每 band 行数 (r) 50% 概率阈值 80% 概率阈值 假阳性风险
20 5 ~0.76 ~0.87 低
16 8 ~0.83 ~0.91 很低
10 4 ~0.67 ~0.81 中等
25 2 ~0.50 ~0.69 较高
参数选择的经验法则：在预训练数据去重中，通常将阈值设置在 0.7-0.85 之间。设得太低（比如 0.5）会导致大量"只是主题相关但内容不同"的文档被误判为重复；设得太高（比如 0.95）又会漏掉很多经过轻度重写的相似文档。FineWeb 使用的阈值为 0.87（b=14, r=13），Gopher 使用了 0.8（b=450, r=4，但总签名数更多）。

11.3 分布式 MinHash

当数据集达到数亿甚至数十亿文档时，单机 MinHash 已经无法胜任。分布式 MinHash 的经典实现是基于 MapReduce 的：

Mapper 阶段：将每个文档转换为它所属的所有 LSH 桶的键值对 (band_id, bucket_id) -> doc_id。

Shuffle 阶段：按 band_id + bucket_id 对所有键值对进行分组，确保落入同一个桶的所有文档聚集到同一个 Reducer。

Reducer 阶段：对每个桶内的文档做两两 Jaccard 比较，标记相似度超过阈值的文档对为重复，然后取其中一个保留。

# 分布式 MinHash 的 MapReduce 伪代码

def map_phase(doc_id, text):
    """Mapper: 为每个文档生成 LSH 桶键"""
    shingles = shingle(text, k=13)   # 13-gram
    signature = minhash_signature(shingles, num_hashes=182)

    # LSH：将 182 个签名分成 b=14 个 band，每 band r=13 行
    b, r = 14, 13
    for band_idx in range(b):
        start = band_idx * r
        bucket_items = signature[start:start + r]
        # 将 band 内的 r 个哈希值拼成一个桶键
        bucket_key = hash(tuple(bucket_items))
        # 输出: (band_idx, bucket_key) -> doc_id
        yield ((band_idx, bucket_key), doc_id)

def reduce_phase(bucket_key, doc_ids):
    """Reducer: 对每个桶内的文档做两两比较"""
    doc_ids = list(doc_ids)
    for i in range(len(doc_ids)):
        for j in range(i + 1, len(doc_ids)):
            jaccard = compute_jaccard(doc_ids[i], doc_ids[j])
            if jaccard > 0.87:
                mark_duplicate(doc_ids[j])  # 标记较短或质量较低的文档为重复

11.4 n-gram 的选择：字符级 vs Token 级

在构建 shingle（n-gram）集合时，一个关键决策是使用字符级 n-gram 还是 token 级 n-gram。这个选择对中文数据处理尤其重要。

  • 字符级 n-gram（如 k=13 character shingles）：对拼写错误不敏感，不需要分词，对 CJK 语言自然支持。缺点是字符级 n-gram 集合的大小非常大，而且对语序的敏感性较低。
  • Token 级 n-gram（如 k=5 word shingles）：通常需要先分词。对英文效果好，对中文需要额外的分词步骤（如 jieba），分词错误会影响去重质量。

在 C4 和 FineWeb 等大规模数据集中，广泛使用的是 字符级 13-gram。原因很简单：字符级 n-gram 不需要分词，语言无关，实现简单。对于中文，字符级 n-gram 实际上效果不错——因为汉字本身就是有意义的语义单元，13 个字符的滑动窗口已经能捕捉到足够的上下文信息。

中文场景的建议：对于中文预训练数据的 MinHash 去重，推荐使用字符级 8-13 gram。如果使用 token 级，需要先用 jieba 分词，然后将 token 拼接回字符串再用 MinHash——但要注意中文分词的不一致性会导致同一篇文章的不同分词结果产生不同的 shingle 集合，反而可能降低去重效果。

11.5 与其他模糊去重方法的对比

MinHash 不是唯一的模糊去重方法。在特定场景下，其他方法也可能更合适：

方法 原理 可扩展性 适用场景 局限
MinHash + LSH Jaccard 相似度近似的随机哈希方法 优秀（线性或近线性） 大规模文本去重（通用场景） 对短文本效果差
SimHash 基于随机超平面的哈希，产生位向量签名 优秀 网页去重、新闻聚类 对汉明距离阈值的敏感性高
嵌入向量 + 相似度 用 Sentence-BERT 等模型生成向量，然后计算余弦相似度 差（需要 GPU 和大量计算） 语义级去重（见第 12 章） 成本高，不适合全量数据
布隆过滤器 概率性成员查询数据结构 优秀 精确去重的内存优化 只能做精确匹配

11.6 使用 Datatrove 的 MinHashDeduplicator

Datatrove 框架内置了完整的 MinHash + LSH 去重实现，是 FineWeb 数据集的去重引擎。使用起来非常方便：

from datatrove.pipeline.dedup import MinHashDeduplicator
from datatrove.pipeline.readers import ParquetReader
from datatrove.pipeline.writers import ParquetWriter

# 配置 MinHash 去重器
dedup = MinHashDeduplicator(
    n_grams=13,           # 字符级 13-gram
    num_hashes=182,       # 182 个哈希函数
    num_buckets=14,       # 14 个 band
    num_rows_per_bucket=13,  # 每个 band 13 行
    threshold=0.87,       # Jaccard 相似度阈值
    storage="/output/minhash/"
)

# 在流水线中使用
pipeline = [
    ParquetReader("/input/deduped_exact/"),
    dedup,
    ParquetWriter("/output/deduped_fuzzy/"),
]

# 也可以计算 Jaccard 相似度的统计数据
for stats in dedup.stats():
    print(f"总文档数: {stats['total_docs']}")
    print(f"重复文档数: {stats['duplicate_docs']}")
    print(f"去重率: {stats['duplicate_docs'] / stats['total_docs']:.1%}")
踩坑故事：我们团队曾经在一个项目中忽略了 MinHash 的去重阈值调参，直接使用了默认的 0.5 阈值。结果去重完成后发现数据集缩减了 70%——大量"主题相似但内容并不重复"的文档被误删了。比如"两篇都讲注意力机制但内容不同的文章"因为共享了太多的"注意力"、"查询"、"键"等高频术语而被判为重复。恢复数据后改为 0.85 的阈值，去重率降到了健康的 22%。阈值不是越高越好，也不是越低越好，而是要根据你的数据分布通过实验找到"甜点"。

本章小结

MinHash 和 LSH 是工业界模糊去重的事实标准。MinHash 用一组紧凑签名估计文档相似度，LSH 通过 banding 将候选比较从 O(N²) 降到近线性。参数选择（n-gram 大小、哈希函数数、band/row 配比）直接影响去重效果和计算成本的平衡。

经过精确去重和 MinHash 模糊去重后，重复文档问题已基本解决。但重复不仅存在于文档级别——下一章我们将看到，即使文档本身不重复，其中的某些句子或段落可能与其他文档高度雷同。这正是 语义级与句子级去重 要解决的问题。
