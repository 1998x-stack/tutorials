第17章 代码执行与安全沙箱

在第 15 章和第 16 章中，我们已经看到 AutoGen 的一个核心能力——Agent 可以生成并执行代码。这赋予了 LLM Agent 远超单纯对话的能力：它能写 Python 脚本分析数据、生成图表、运行机器学习模型。但能力越大，责任越大——让 AI 生成的代码在真实系统上运行，安全风险不容忽视。

2025 年 3 月，我参与的一个项目就出了这样一件事：一个代码生成 Agent 在执行数据分析任务时，出于"好意"尝试修复它认为有问题的配置文件——它发现了一个 "timeout=30" 的设置，觉得太短，自动改成了 "timeout=300"。结果下游服务全部等待超时，导致了一次 15 分钟的线上事故。Agent 的"好心办坏事"提醒我们：代码执行能力必须配上严格的沙箱约束。

核心原则： 对于 Agent 代码执行，安全设计应该遵循"默认禁止，按需开放"的零信任原则。Agent 应该在一个受限的环境中运行，只拥有完成当前任务所必需的最小权限。

17.1 AutoGen 代码执行架构

AutoGen 的代码执行能力由 UserProxyAgent 的 code_execution_config 参数控制。当 AssistantAgent 在回复中生成代码块（以 ```python 或 ```bash 标记）时，UserProxyAgent 会执行这些代码，并将结果返回给对话。

代码执行的完整生命周期如下：

  1. 代码提取： UserProxyAgent 检测 AssistantAgent 回复中的代码块，提取代码内容和语言类型。
  2. 安全性检查： 检查代码是否包含被禁止的操作（如文件删除、网络请求等，取决于配置）。
  3. 执行调度： 根据配置选择执行环境（本地、Docker、远程），将代码发送到执行器。
  4. 结果收集： 收集执行的标准输出、标准错误、返回码、执行时间。
  5. 反馈注入： 将执行结果作为新的消息注入对话上下文，供 AssistantAgent 继续推理。
from autogen import AssistantAgent, UserProxyAgent

assistant = AssistantAgent(
    name="data_analyst",
    llm_config={"config_list": [{"model": "gpt-4o", "api_key": "..."}]},
    system_message="用 Python 分析数据。代码必须包含完整实现和注释。",
)

# 配置代码执行权限
user_proxy = UserProxyAgent(
    name="executor",
    human_input_mode="NEVER",
    code_execution_config={
        "work_dir": "sandbox_workspace",     # 工作目录，所有代码在此目录下运行
        "use_docker": "python:3.11-slim",    # Docker 镜像
        "timeout": 30,                        # 单次执行超时
        "last_n_messages": 3,                 # 只处理最近 3 条消息中的代码
    },
)

# 执行对话
user_proxy.initiate_chat(
    assistant,
    message="生成 100 个服从正态分布的随机数，计算均值、中位数和标准差，"
            "并绘制直方图保存为 chart.png"
)
执行原理： AutoGen 会在配置的 work_dir 下创建一个临时 Python 文件（如 tmp_code_xxxxx.py），然后通过子进程调用 Python 解释器运行。运行结束后，临时文件会被清理。如果开启了 Docker，整个过程会在容器内完成。

17.2 Docker 沙箱配置详解

Docker 沙箱是 AutoGen 提供的最安全的代码执行方式。它将 Agent 代码的执行环境与宿主操作系统完全隔离。即使 Agent 生成了恶意代码——比如尝试删除所有文件——破坏范围也被限制在容器内部。

Docker 配置选项

参数 类型 说明 推荐值
use_docker bool/str 是否使用 Docker；可指定镜像名称 "python:3.11-slim"
work_dir str 容器内的工作目录，挂载宿主某个目录 /tmp/agent_sandbox
timeout int 单次代码执行的最大秒数 30-120
memory_limit str 容器内存上限（AG2 支持） "512m"
network bool 是否允许容器访问网络 False（默认禁止）

生产级 Docker 配置示例

import tempfile

# 为每个任务创建独立的工作目录
sandbox_dir = tempfile.mkdtemp(prefix="agent_sandbox_")

# 生产级配置
user_proxy = UserProxyAgent(
    name="sandboxed_executor",
    human_input_mode="NEVER",
    code_execution_config={
        "work_dir": sandbox_dir,
        "use_docker": {
            "image": "python:3.11-slim",
            "container_name": f"agent-{task_id}",  # 容器命名便于追踪
            "volumes": {
                sandbox_dir: {"bind": "/workspace", "mode": "rw"}
            },
            "environment": {
                "PYTHONDONTWRITEBYTECODE": "1",
                "DEBIAN_FRONTEND": "noninteractive",
            },
        },
        "timeout": 60,
        "last_n_messages": 3,
    },
)
重要提醒： 默认情况下，Docker 容器内的代码执行没有网络访问权限。如果你的 Agent 需要安装 pip 包或下载数据，需要显式配置 network: True（在 AG2 中通过 docker_executor 配置）。但请谨慎：开放网络意味着逃逸风险增大。

17.3 本地执行 vs 远程执行

除了 Docker 沙箱，AutoGen 还支持其他代码执行模式。选择哪种模式取决于你的安全要求和场景需求：

执行模式 隔离级别 性能 适用阶段 安全风险
本地子进程 无隔离 原生速度 本地开发调试 高
Docker 容器 操作系统级隔离 接近原生 开发/生产 低
远程执行器 物理/虚拟化隔离 依赖网络 生产环境 极低
e2b / Flyte 云沙箱 中等 生产环境 极低

远程执行集成

如果你使用 AG2（AutoGen 社区分支），可以通过自定义命令执行器来集成远程沙箱服务：

# AG2 中使用远程执行器（概念示例）
from ag2 import UserProxyAgent, CommandExecutor

class E2BExecutor(CommandExecutor):
    """基于 e2b.dev 的云沙箱执行器"""
    def __init__(self):
        from e2b import Sandbox
        self.sandbox = Sandbox()

    def execute_code(self, code: str, **kwargs) -> dict:
        result = self.sandbox.run_code(code)
        return {
            "exit_code": result.exit_code,
            "output": result.stdout,
            "error": result.stderr,
        }

e2b_executor = E2BExecutor()

user_proxy = UserProxyAgent(
    name="cloud_executor",
    code_execution_config={
        "executor": e2b_executor,  # 自定义执行器
    },
)
选型建议： 本地执行仅适合在完全信任的环境下做快速验证。任何暴露给外部用户的 Agent 服务，都至少应该使用 Docker 沙箱。对安全要求极高的企业场景（比如金融、医疗），推荐使用云沙箱服务或专用的隔离集群。

17.4 代码生成与自修复循环

AutoGen 的代码执行能力配合 LLM 的迭代能力，可以形成一个强大的"自修复循环"——Agent 生成代码、执行、发现错误、修复代码、再次执行，直到成功或达到最大重试次数。这使得 Agent 可以自主修复编程错误，大幅提升了任务完成率。

# 自修复循环的典型流程
# 第1轮：Assistant 生成代码
# 第2轮：UserProxy 执行代码 -> 报错 -> 将错误信息发给 Assistant
# 第3轮：Assistant 分析错误并修复代码
# 第4轮：UserProxy 再次执行 -> 成功 -> 返回结果

# max_consecutive_auto_reply 控制自修复的最大次数
user_proxy = UserProxyAgent(
    name="auto_fix_executor",
    human_input_mode="NEVER",
    code_execution_config={
        "work_dir": "agent_workspace",
        "use_docker": True,
        "timeout": 30,
    },
    max_consecutive_auto_reply=5,  # 最多自动修复 5 次
)

assistant = AssistantAgent(
    name="self_healing_coder",
    llm_config={"config_list": [{"model": "gpt-4o", "api_key": "..."}]},
    system_message="你会 Python 编程。如果代码执行报错，"
                   "仔细分析错误信息，找出根本原因并修复。"
                   "确保修复后的代码正确处理边界情况。"
                   "修复后重新输出完整代码。",
)
效率数据： 根据我们的生产统计，带有自修复循环的 Agent 可以解决约 85% 的首次执行错误。最常见的错误类型是：缺少依赖库（35%）、API 使用错误（28%）、类型不匹配（18%）、文件路径错误（12%）。大多数错误在 1-2 轮修复内就能解决。

自修复循环虽然强大，但也需要设定合理的边界：

  • 最大重试次数： max_consecutive_auto_reply 建议设置为 3-5 次。超过后应停止并上报，避免无限循环。
  • 执行超时： 每个代码块设置合理的超时时间（通常 30-60 秒），防止死循环或死锁。
  • Token 预算： 每次修复都会消耗额外的 Token。建议为整个任务设置 Token 预算上限。
  • 错误降级： 如果经过多次修复仍然失败，Agent 应该生成一个详细的错误报告，而不是继续尝试或静默失败。

17.5 安全隔离最佳实践

经过大量生产实践，行业已经总结出了一套 Agent 代码执行的安全最佳实践。我将它们归纳为以下几个关键原则：

原则一：最小权限

Agent 容器应该只拥有完成当前任务所必需的权限。具体来说：

  • 文件系统：只挂载任务所需的数据目录，不要挂载整个宿主文件系统。
  • 网络访问：默认关闭，只有 Agent 确实需要下载数据时才开启。
  • 系统调用：使用 Docker 的 seccomp 策略限制危险系统调用。
  • 用户权限：容器内以非 root 用户运行（如 RUN useradd -m agent）。

原则二：资源限制

防止 Agent 代码消耗过多系统资源，影响其他服务：

# Docker 资源限制示例（在 docker run 参数中）
docker run \
    --memory="512m" \              # 内存上限 512MB
    --cpus="1.0" \                 # CPU 上限 1 核
    --pids-limit="100" \           # 进程数上限 100
    --read-only \                  # 文件系统只读
    --tmpfs="/tmp:size=100M" \     # 临时文件 100MB 上限
    --network="none" \             # 禁用网络
    python:3.11-slim

原则三：可审计

所有 Agent 代码执行应该留下完整的审计日志：

  • 代码快照： 记录每次执行的完整代码内容。
  • 执行记录： 记录代码的 stdin/stdout/stderr、执行时间、退出码。
  • 操作轨迹： 记录 Agent 在容器内的文件操作和网络请求。
  • 人工审查： 对高风险操作（如文件删除、数据写入）设置人工审批流程。

原则四：内容过滤

在代码进入执行器之前，应该经过安全检查：

# 代码安全过滤器示例
import re

BLOCKED_PATTERNS = [
    r"os\.remove\(.*\)",           # 删除文件
    r"os\.system\(.*",             # 执行系统命令
    r"subprocess\.",               # 子进程
    r"shutil\.rmtree",             # 递归删除目录
    r"(import|from)\s+ctypes",     # 底层 C 接口操作
    r"__import__\(",               # 动态导入
    r"eval\(.*\)",                 # 动态执行
    r"exec\(.*\)",                 # 动态执行
]

def security_scan(code: str) -> tuple[bool, str]:
    """扫描代码中是否包含危险操作"""
    for pattern in BLOCKED_PATTERNS:
        match = re.search(pattern, code)
        if match:
            return False, f"检测到禁止操作：{match.group()}"
    return True, "安全通过"

# 在 UserProxyAgent 中集成安全扫描
class SecureUserProxy(UserProxyAgent):
    def _execute_code(self, code, **kwargs):
        safe, msg = security_scan(code)
        if not safe:
            return {"exit_code": -1, "output": msg}
        return super()._execute_code(code, **kwargs)
风险提示： 正则表达式过滤不是万能的。攻击者可以通过编码、拼接、间接调用等方式绕过检测。内容过滤只能作为"第一道防线"，不能替代 Docker 隔离和权限限制。安全需要纵深防御——多层防线互相补充，任何一层被突破都有其他层兜底。

17.6 实战：数据分析 Agent

让我们将前面的所有知识整合到一个实战案例中——构建一个企业级数据分析 Agent，它既能灵活生成代码分析数据，又受到严格的安全沙箱保护。

import autogen
import tempfile
import os

# 创建隔离的工作目录
task_id = "analysis_task_001"
work_dir = f"/tmp/sandbox/{task_id}"
os.makedirs(work_dir, exist_ok=True)

# 数据文件（模拟挂载）
data_path = os.path.join(work_dir, "sales_data.csv")
with open(data_path, "w") as f:
    f.write("日期,产品,销售额,数量\n")
    f.write("2026-01,产品A,150000,300\n")
    f.write("2026-01,产品B,95000,180\n")
    f.write("2026-02,产品A,165000,330\n")
    f.write("2026-02,产品B,102000,195\n")
    f.write("2026-03,产品A,180000,360\n")
    f.write("2026-03,产品B,110000,210\n")

# 数据分析师 Agent
analyst = AssistantAgent(
    name="analyst",
    system_message="你是一位资深数据分析师。使用 Python 和 pandas "
                   "分析 CSV 数据，给出深入的商业洞察。"
                   "输出必须包含：数值分析、趋势判断、建议。",
    llm_config={"config_list": [{"model": "gpt-4o", "api_key": "..."}]},
)

# 安全执行者 Agent
executor = UserProxyAgent(
    name="secure_executor",
    human_input_mode="NEVER",
    code_execution_config={
        "work_dir": work_dir,
        "use_docker": {
            "image": "python:3.11-slim",
            "environment": {
                "PYTHONDONTWRITEBYTECODE": "1",
            },
        },
        "timeout": 30,
    },
    max_consecutive_auto_reply=5,
)

# 启动分析任务
executor.initiate_chat(
    analyst,
    message="分析 sales_data.csv 中的数据，计算每个产品的：\n"
            "1. 月均销售额和销量\n"
            "2. 月度环比增长率\n"
            "3. 绘制产品 A 和产品 B 的销售趋势对比图（折线图）\n"
            "4. 给出基于数据的商业建议"
)

这个数据分析 Agent 的工作流程：Secure Executor 接收用户请求 → 转发给 Analyst → Analyst 输出分析计划 → Secure Executor 在 Docker 沙箱内创建环境并安装 pandas/matplotlib → Analyst 生成 Python 分析代码 → Secure Executor 在 Docker 内执行代码 → 绘图结果保存到工作目录 → 分析结果返回给用户。整个过程在一个安全的、可审计的隔离环境中完成。

生产增强： 在实际部署中，你可以添加以下增强：将工作目录挂载到 S3/Blob 存储以持久化结果；集成日志服务记录每次代码执行的完整 Trace；在 Docker 镜像中预装常用的数据分析库（pandas、numpy、matplotlib、scikit-learn），避免每次安装依赖的延迟。

本章我们围绕"Agent 代码执行"这个能力展开，从 AutoGen 的代码执行架构到 Docker 沙箱的详细配置，从本地/远程执行模式的对比到自修复循环的设计，最后到安全隔离的最佳实践。代码执行是 Agent 从"对话助手"升级为"数字员工"的核心能力，但也是最需要谨慎对待的能力。

记住三个关键原则：默认隔离、最小权限、纵深防御。如果一个 Agent 系统因为代码执行安全性不足而导致事故，那说明问题不在于 Agent 的能力，而在于系统设计者的安全意识。
