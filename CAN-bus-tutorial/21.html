第21章 CANopen PDO通信：PDO映射、同步PDO与异步PDO、PDO触发方式

各位工程师朋友，咱们今天来聊聊CANopen里最核心、也最灵活的一个机制——PDO通信。说实话，我刚开始接触CANopen时，被PDO的映射和触发方式绕得有点晕。后来在项目里调了几个月，才真正摸透它的脾气。今天我把这些经验掰开揉碎讲给你听。

21.1 PDO到底是什么？

PDO，全称Process Data Object，过程数据对象。说白了，它就是用来传输实时数据的通道。比如电机转速、温度值、开关状态这些需要频繁更新的数据，都走PDO。

我个人习惯把PDO比作「快递专线」。SDO（服务数据对象）像普通快递，每次都要确认签收，速度慢但可靠。PDO像同城闪送，直接扔到门口就走，速度快但不保证你收到了。嗯，这个比喻你记住，后面会反复用到。

PDO的核心特点：

  • 一对一的通信模型（生产者→消费者）
  • 数据长度最多8字节
  • 无需应答，实时性高
  • 可配置映射关系

21.2 PDO映射——把数据装进盒子里

PDO映射，就是决定「哪些数据放进这8个字节里」。每个CANopen设备都有对象字典，里面存着各种参数。PDO映射就是把这些参数按顺序打包。

举个例子，我有一个伺服驱动器，需要实时上传当前转速（索引0x6041，子索引0x00）和当前位置（索引0x6064，子索引0x00）。这两个数据各占4字节，加起来正好8字节，可以塞进一个PDO里。

映射的配置方式：

// 假设我们要配置TPDO1的映射
// 对象字典中，TPDO1的映射参数在0x1A00

// 第一步：清空原有映射
写入对象 0x1A00, 子索引0x00, 数据0x00

// 第二步：设置第一个映射项
// 转速：索引0x6041，子索引0x00，长度16位
写入对象 0x1A00, 子索引0x01, 数据0x60410010

// 第三步：设置第二个映射项
// 位置：索引0x6064，子索引0x00，长度32位
写入对象 0x1A00, 子索引0x02, 数据0x60640020

// 第四步：设置映射数量
写入对象 0x1A00, 子索引0x00, 数据0x02

我的经验：映射项的数据长度必须严格匹配。我在项目中遇到过一个问题，明明映射了4字节的数据，但实际只发了2字节，结果接收方解析出来的数值完全不对。后来查了半天，发现是映射长度写错了。所以，每次配置完映射，一定要用CAN分析仪抓包确认一下。

21.3 同步PDO与异步PDO

这是PDO通信里最重要的分类。我刚开始做项目时，就因为选错了同步方式，导致整个系统时序乱套。

21.3.1 同步PDO

同步PDO依赖SYNC报文。主站周期性地发送SYNC，所有从站收到SYNC后，同时更新输出或采集输入。这种方式保证了整个网络的同步性。

同步PDO的工作流程：

  1. 主站发送SYNC报文（COB-ID：0x80）
  2. 从站收到SYNC后，立即发送TPDO或更新RPDO
  3. 所有从站在同一个时间基准上动作

我记得在做一个多轴同步控制项目时，4个伺服驱动器需要同时执行位置指令。如果不用同步PDO，每个驱动器收到指令的时间差可能达到几毫秒，轴之间的同步精度根本没法保证。用了同步PDO后，所有轴在同一个SYNC脉冲下动作，效果立竿见影。

21.3.2 异步PDO

异步PDO不依赖SYNC，由事件触发。比如温度超过阈值、按键按下、定时器超时等。异步PDO的实时性取决于事件发生的频率。

什么时候用异步PDO？

  • 数据变化不频繁（比如温度、湿度）
  • 需要快速响应外部事件（比如急停按钮）
  • 系统对同步性要求不高

避坑指南：我曾经在一个项目中，把急停信号配置成了同步PDO。结果急停按钮按下去后，要等到下一个SYNC周期（10ms后）才能触发。嗯，这显然不行。后来改成异步PDO，事件触发立即发送，响应时间从10ms降到了不到1ms。

21.4 PDO触发方式

触发方式决定了PDO什么时候发送。CANopen定义了三种触发方式，我分别说说。

21.4.1 事件触发

事件触发是最灵活的方式。当某个条件满足时，PDO自动发送。条件可以是：

  • 数据变化超过阈值
  • 定时器超时
  • 特定事件发生

配置事件触发时，需要设置事件定时器和事件计数器。比如，我希望温度每变化1度就发送一次，同时最多每100ms发送一次，防止总线被刷爆。

// 配置TPDO1的事件触发参数
// 对象字典：0x1800（TPDO1通信参数）

// 设置事件定时器为100ms
写入对象 0x1800, 子索引0x05, 数据0x0064

// 设置事件计数器为1（变化1次就发送）
写入对象 0x1800, 子索引0x06, 数据0x01

21.4.2 定时触发

定时触发最简单粗暴。不管数据变没变，到时间就发。适合周期性上报的数据，比如心跳包、状态字。

配置方式就是设置事件定时器，但把事件计数器设为0。这样PDO只按定时器周期发送，不受数据变化影响。

我的建议：定时触发虽然简单，但要注意总线负载。如果系统里有几十个节点，每个节点每10ms发一个PDO，总线很快就满了。我一般把定时周期设在50ms以上，除非有特殊要求。

21.4.3 远程帧触发

远程帧触发比较特殊。它由接收方主动请求，发送方被动响应。接收方发送一个远程帧，发送方收到后立即发送PDO。

这种方式适合「按需索取」的场景。比如主站想知道某个从站的当前状态，就发一个远程帧过去，从站收到后把数据发回来。这样避免了不必要的总线占用。

三种触发方式的对比：

触发方式 优点 缺点 适用场景
事件触发 灵活，节省带宽 配置复杂 数据变化不规律
定时触发 简单，可预测 可能浪费带宽 周期性数据上报
远程帧触发 按需索取，精准 需要额外请求 主从查询模式

21.5 实战中的PDO配置策略

说了这么多理论，咱们来点实际的。我在项目中总结了一套PDO配置的「三步法」：

  1. 先分类型：哪些数据需要同步？哪些可以异步？急停、报警这类必须异步，位置指令这类最好同步。
  2. 再定映射：把相关的数据打包在一起。比如转速和电流放在一个PDO里，位置和速度放在另一个PDO里。这样接收方解析起来方便。
  3. 最后选触发：根据数据变化频率和系统实时性要求，选择合适的触发方式。变化快的用事件触发，变化慢的用定时触发。

一个典型的配置示例：

  • TPDO1（同步触发）：转速+位置，用于运动控制
  • TPDO2（事件触发）：温度+电压，变化超过阈值才发
  • TPDO3（定时触发）：状态字+报警码，每100ms发一次
  • TPDO4（远程帧触发）：配置参数，主站需要时再请求

你想想看，如果所有数据都用同一种触发方式，要么总线被刷爆，要么实时性不够。所以，合理搭配才是王道。
