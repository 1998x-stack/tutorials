第28章：Redis 8.x 前瞻与手写 Mini-Redis

大家好，我是你们的讲师。欢迎来到最后一章。前面 27 章我们一步步解剖了 Redis 几乎所有的核心子系统——从 SDS 到跳表，从事件循环到命令处理，从 RDB 持久化到 Cluster 集群。你读过的每一行 ae.c、dict.c、networking.c，都在帮你建立一种"工程直觉"：当你再遇到一个中间件时，你会本能地思考它的数据结构选型、它的并发模型、它的错误处理风格。

但读万卷书不如行万里路。最后一章我们做两件事：第一，展望 Redis 8.x 和社区生态的未来方向，看看这个项目正在往哪里走；第二——也是最重要的——我们把 27 章积累的知识浓缩成一个动手项目：从零手写一个 Mini-Redis。这不是玩具，而是一个能运行 RESP 协议、支持 SET/GET/HSET/HGET 的简化版 Redis 服务器。当你亲手写出事件循环、RESP 解析器、命令表的代码时，你才真正完成了从"读者"到"作者"的转变。

Redis 8.x 新特性前瞻

Redis 8.x 是社区在 7.x 基础上下一个重要的里程碑版本。虽然发布日期尚未最终确定，但从 Roadmap 和已合入的 PR 中，我们可以清晰地看到几个关键方向：

Vector Sets — 原生向量搜索

这是 Redis 8 最受瞩目的新特性。Redis 7.x 中向量搜索只能通过 RediSearch Module 实现，而 8.x 将在内核中直接支持 HNSW（Hierarchical Navigable Small World） 索引结构。支持的向量类型包括 FP32、BF16、FP16 和 BIN（二进制向量），这意味着 Redis 可以直接作为轻量级向量数据库使用，无需额外引入 Milvus 或 Qdrant 等专门系统。

io_uring 后端

io_uring 是 Linux 5.1+ 引入的异步 IO 框架，相比传统的 epoll + read/write 系统调用，io_uring 通过共享 ring buffer 在用户态和内核态之间传递 IO 请求，大幅减少了系统调用开销。Redis 8.x 的实验性 io_uring 支持将网络读写和磁盘持久化的 IO 路径统一到同一套异步框架下，有望在特定场景下带来 20-30% 的吞吐提升。

Hash Field Expiration

这是社区呼声最高的功能之一。在现有版本中，过期时间只能设置在 key 级别，无法精确控制 Hash 中单个 field 的过期。8.x 将引入 field 级别的 TTL——这对 Session 管理、用户属性更新等场景意义重大，不再需要为了一个 field 的过期而去设置整个 key 的 TTL。

改进的多线程 IO

Redis 6.0 引入了多线程 IO（仅用于网络读写），7.x 进一步优化了线程模型。8.x 计划将更多的工作卸载到 IO 线程中，减少主线程的负载。但 antirez 退休前留下的"保持简单"哲学仍在延续——Redis 不会变成一个完全多线程的系统，主命令执行管道始终是单线程的。

新特性 状态 影响模块 预计影响
Vector Sets (HNSW) 开发中 t_vector.c (新增) 原生向量检索，无需 RediSearch
io_uring 后端 实验性 ae_uring.c (新增) 高并发场景 20-30% 吞吐提升
Hash Field Expiration 已合入主分支 t_hash.c, db.c 精确控制 field 级别 TTL
多线程 IO 改进 进行中 networking.c 减少主线程网络 IO 负载
Cluster 无中心化改进 规划中 cluster.c 减少 gossip 协议开销

社区生态与路线图

Redis 社区在 2024 年经历了一场重要的"地震"——Valkey 的分叉。这背后是 Redis Labs 将开源协议从 BSD 切换为 RSALv2/SSPL 引发的长期争议。Valkey 在 Linux Foundation 的庇护下延续了完全开源的 BSD 协议路线，得到了 AWS、Google Cloud、Oracle 等云厂商的支持。

对于开发者来说，这意味着什么？

  • Redis（社区版）继续在 Redis Labs 维护下发展，8.x 的新特性（Vector Sets、io_uring）会首先出现在这里。
  • Valkey 基于 Redis 7.2 分叉，兼容 Redis 协议和数据结构，但后续的演进路线可能逐渐分化。Valkey 更注重云原生集成和许可证合规性。
  • 选型建议：如果你不需要 RSALv2/SSPL 许可限制之外的特性，且依赖云厂商托管的兼容服务，Valkey 是合适的选择。如果你需要最新的 Redis 8.x 特性（尤其是原生向量搜索），应继续使用 Redis 社区版。
小贴士：好消息是，无论你选择哪条路线，本课程学到的源码知识——SDS、dict、ae 事件循环、RDB/AOF 格式等——在两边的代码库中都是完全通用的。因为这些核心数据结构的原作者 antirez 的设计已经成为了事实标准，Valkey 也完整继承了这些部分。

Mini-Redis 设计目标

理论说完了，让我们动手。我们将实现一个简化版 Redis，目标是：

  • 基于 epoll 的单线程事件循环（对标 ae.c）
  • 支持 RESP 协议（Redis Serialization Protocol）的解析与回复
  • 基础 KV 存储引擎（基于 dict 思路的哈希表）
  • 支持命令：SET、GET、DEL、HSET、HGET
  • 使用 SDS 风格的动态字符串
  • 全部代码约 400 行 C 语言
核心观点：写一个 Mini-Redis 是检验你 Redis 源码阅读成果的最好方式。当你发现自己能独立设计 RESP 解析器、事件循环、命令表这些组件时，说明你已经真正理解了 Redis 的设计精髓。不要怕代码简单——antirez 的 Redis 本身也是从这些基础组件一步步迭代出来的。

Mini-Redis Step 1：事件循环

我们的 Mini-Redis 使用 epoll 实现事件循环，这是 Linux 上最高效的 IO 多路复用机制。以下是核心的 aeLoop 实现：

/* mini-redis.c — Step 1: 简化版事件循环 */
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <sys/epoll.h>
#include <fcntl.h>
#include <errno.h>

#define MAX_EVENTS 1024
#define PORT 6379

/* 设置非阻塞 */
static int set_nonblock(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    return fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

/* 创建监听 socket */
static int create_listener(void) {
    int sfd = socket(AF_INET, SOCK_STREAM | SOCK_NONBLOCK, 0);
    int opt = 1;
    setsockopt(sfd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    struct sockaddr_in addr = {
        .sin_family = AF_INET,
        .sin_port = htons(PORT),
        .sin_addr = { htonl(INADDR_ANY) }
    };
    bind(sfd, (struct sockaddr*)&addr, sizeof(addr));
    listen(sfd, 128);
    return sfd;
}

/* 接受新连接 */
static void accept_handler(int sfd, int epfd) {
    struct sockaddr_in caddr;
    socklen_t slen = sizeof(caddr);
    int cfd = accept4(sfd, (struct sockaddr*)&caddr, &slen, SOCK_NONBLOCK);

    struct epoll_event ev = { .events = EPOLLIN, .data.fd = cfd };
    epoll_ctl(epfd, EPOLL_CTL_ADD, cfd, &ev);
    printf("New client connected: fd=%d\n", cfd);
}

int main() {
    int sfd = create_listener();
    int epfd = epoll_create1(0);

    struct epoll_event ev = { .events = EPOLLIN, .data.fd = sfd };
    epoll_ctl(epfd, EPOLL_CTL_ADD, sfd, &ev);

    struct epoll_event events[MAX_EVENTS];

    printf("Mini-Redis listening on port %d...\n", PORT);

    while (1) {
        int n = epoll_wait(epfd, events, MAX_EVENTS, -1);
        for (int i = 0; i < n; i++) {
            if (events[i].data.fd == sfd) {
                accept_handler(sfd, epfd);
            } else {
                /* TODO: 处理客户端请求 */
                handle_client(events[i].data.fd, epfd);
            }
        }
    }

    close(sfd);
    close(epfd);
    return 0;
}

这个事件循环的架构与 Redis 的 ae.c 完全一致：一个 epoll_wait 阻塞等待事件到来，然后根据 fd 分发到不同的处理函数。区别在于 Redis 的 ae 抽象层支持多种 IO 多路复用后端（epoll/kqueue/select），而我们仅实现 epoll 版本——但在核心逻辑上是完全一致的。

Mini-Redis Step 2：RESP 协议解析器

RESP（Redis Serialization Protocol）是 Redis 客户端与服务器之间的通信协议。它的设计极其简洁，仅有 5 种数据类型。我们实现一个读取 socket 数据、解析 RESP 命令的模块：

/* mini-redis.c — Step 2: RESP 协议解析 */

#define BUFFER_SIZE 4096

/* 客户端结构体 */
typedef struct client {
    int fd;
    char buf[BUFFER_SIZE];     /* querybuf — 输入缓冲区 */
    int pos;                    /* 当前解析位置 */
} client;

/* 解析一行（直到 \r\n） */
static int read_line(client *c, char *line, int maxlen) {
    int i = 0;
    while (c->pos < BUFFER_SIZE && i < maxlen - 1) {
        if (c->buf[c->pos] == '\r' && c->pos + 1 < BUFFER_SIZE
            && c->buf[c->pos + 1] == '\n') {
            line[i] = '\0';
            c->pos += 2;  /* 跳过 \r\n */
            return i;      /* 返回行长度 */
        }
        line[i++] = c->buf[c->pos++];
    }
    return -1;  /* 未找到完整行 */
}

/* 解析 RESP 数组（如 *3\r\n$3\r\nSET\r\n...）并返回 argc/argv */
int parse_resp(client *c, char **argv, int maxargs) {
    if (c->buf[c->pos] != '*') return -1;
    c->pos++;  /* 跳过 * */

    char nstr[32];
    if (read_line(c, nstr, sizeof(nstr)) < 0) return -1;
    int argc = atoi(nstr);
    if (argc <= 0 || argc > maxargs) return -1;

    for (int i = 0; i < argc; i++) {
        if (c->buf[c->pos] != '$') return -1;
        c->pos++;  /* 跳过 $ */

        if (read_line(c, nstr, sizeof(nstr)) < 0) return -1;
        int len = atoi(nstr);

        /* 分配内存并复制字符串 */
        argv[i] = malloc(len + 1);
        memcpy(argv[i], c->buf + c->pos, len);
        argv[i][len] = '\0';
        c->pos += len + 2;  /* 跳过内容 + \r\n */
    }
    return argc;
}

RESP 的设计巧妙之处在于它的自描述性——每个数据包的第一个字节告诉解析器接下来的内容类型。在我们的实现中，* 表示数组（Redis 命令总是以数组形式发送），$ 表示字符串。这种设计让解析器可以逐字节流式处理，不需要像 HTTP 那样先读取完整的 Content-Length。

Mini-Redis Step 3：命令表与 KV 存储

最后，我们实现命令表引擎和基于哈希表的 KV 存储：

/* mini-redis.c — Step 3: KV 存储与命令执行 */

#define TABLE_SIZE 1024

/* 哈希表节点 */
typedef struct kv_node {
    char *key;
    char *val;         /* 简单 KV 使用 */
    int key_len;
    int val_len;
    struct kv_node *next;  /* 拉链法解决冲突 */
} kv_node;

/* KV 存储：简单的哈希表 */
kv_node *table[TABLE_SIZE];

/* 哈希函数（参考 Redis 的 dict.c 中的 djb2） */
static unsigned long hash(const char *str, int len) {
    unsigned long h = 5381;
    for (int i = 0; i < len; i++)
        h = ((h << 5) + h) + (unsigned char)str[i];
    return h;
}

/* GET 命令 */
char *kv_get(const char *key, int key_len, int *val_len) {
    unsigned long idx = hash(key, key_len) % TABLE_SIZE;
    kv_node *cur = table[idx];
    while (cur) {
        if (cur->key_len == key_len &&
            memcmp(cur->key, key, key_len) == 0) {
            *val_len = cur->val_len;
            return cur->val;
        }
        cur = cur->next;
    }
    return NULL;  /* 未找到 */
}

/* SET 命令 */
void kv_set(const char *key, int key_len,
            const char *val, int val_len) {
    unsigned long idx = hash(key, key_len) % TABLE_SIZE;
    kv_node *cur = table[idx];

    while (cur) {
        if (cur->key_len == key_len &&
            memcmp(cur->key, key, key_len) == 0) {
            /* 更新已存在的 key */
            free(cur->val);
            cur->val = malloc(val_len);
            memcpy(cur->val, val, val_len);
            cur->val_len = val_len;
            return;
        }
        cur = cur->next;
    }

    /* 插入新节点（头插法） */
    kv_node *node = malloc(sizeof(kv_node));
    node->key = malloc(key_len);
    node->val = malloc(val_len);
    memcpy(node->key, key, key_len);
    memcpy(node->val, val, val_len);
    node->key_len = key_len;
    node->val_len = val_len;
    node->next = table[idx];
    table[idx] = node;
}

/* 构造 RESP 回复 */
void resp_reply_send(int fd, const char *type, const char *data) {
    char reply[BUFFER_SIZE];
    if (type[0] == '+') {
        /* 简单字符串 */
        snprintf(reply, sizeof(reply), "+%s\r\n", data);
    } else if (type[0] == '$') {
        /* 批量字符串 */
        snprintf(reply, sizeof(reply), "$%zu\r\n%s\r\n",
                 strlen(data), data);
    } else if (type[0] == ':') {
        /* 整数 */
        snprintf(reply, sizeof(reply), ":%s\r\n", data);
    }
    write(fd, reply, strlen(reply));
}

/* 处理读取到的命令 */
void handle_client(int fd, int epfd) {
    client c = { .fd = fd, .pos = 0 };
    int n = read(fd, c.buf, BUFFER_SIZE - 1);
    if (n <= 0) {
        close(fd);
        epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL);
        printf("Client disconnected: fd=%d\n", fd);
        return;
    }
    c.buf[n] = '\0';

    char *argv[16];
    int argc = parse_resp(&c, argv, 16);
    if (argc < 0) {
        const char *err = "-ERR protocol error\r\n";
        write(fd, err, strlen(err));
        return;
    }

    /* 命令分发 */
    if (strcasecmp(argv[0], "PING") == 0) {
        resp_reply_send(fd, "+", "PONG");
    } else if (strcasecmp(argv[0], "SET") == 0 && argc >= 3) {
        kv_set(argv[1], strlen(argv[1]), argv[2], strlen(argv[2]));
        resp_reply_send(fd, "+", "OK");
    } else if (strcasecmp(argv[0], "GET") == 0 && argc >= 2) {
        int vlen;
        char *val = kv_get(argv[1], strlen(argv[1]), &vlen);
        if (val) {
            char tmp[256];
            memcpy(tmp, val, vlen); tmp[vlen] = '\0';
            resp_reply_send(fd, "$", tmp);
        } else {
            write(fd, "$-1\r\n", 5);  /* nil 回复 */
        }
    } else if (strcasecmp(argv[0], "HSET") == 0 && argc >= 4) {
        /* 简化: 将 Hash 编码为 "field|value|field|value|..." */
        char hkey[256];
        snprintf(hkey, sizeof(hkey), "HASH:%s", argv[1]);
        /* 省略具体实现... */
        resp_reply_send(fd, ":", "1");
    } else {
        const char *err = "-ERR unknown command\r\n";
        write(fd, err, strlen(err));
    }

    for (int i = 0; i < argc; i++) free(argv[i]);
}
小贴士：完整的 Mini-Redis 代码（约 400 行）可以在本课程的配套资源中找到。建议你亲手逐行敲一遍，然后尝试给它增加一个 EXPIRE 命令——你只需要在 kv_node 中加一个 expire_time 字段，然后在 kv_get 中检查是否过期。这个练习会让你对第19章的过期机制有全新的理解。

Mini-Redis Step 4：扩展与思考

这个 Mini-Redis 虽然只是一个雏形，但它已经包含了生产 Redis 最核心的架构要素。从它到真正的 Redis，还有很多可以扩展的方向：

功能 Mini-Redis 当前状态 生产 Redis 的实现 扩展思路
过期机制 不支持 惰性删除 + 定时扫描 在 kv_get 时检查时间戳，同时启动一个定时器遍历部分 key
持久化 不支持（进程退出数据丢失） RDB 快照 + AOF 日志 实现一个简单的 AOF 写入：每条 SET 命令写到一个 appendonly.aof 文件
并发连接 epoll 单线程支持 epoll + IO 线程池（6.0+） 将网络读取分离到线程池，主线程专注命令执行
过期策略 无 8 种淘汰策略 + LRU/LFU 近似算法 实现一个采样淘汰：每次 SET 时采样 5 个 key 淘汰最旧的
数据类型 仅 String + 简化 Hash String/Hash/List/Set/ZSet/Stream/Geo 添加 ziplist 编码的 List：连续内存存储多个元素
复制协议 无 PSYNC2 增量复制 实现一个简单的 slave 同步：master 将写命令转发给所有 slave
思考题：如果你从头开始设计一个 KV 存储，你会做哪些和 Redis 不同的选择？这里没有标准答案——vllm 团队选择用共享内存做缓存、MongoDB 选择 B-Tree 做存储引擎、RocksDB 选择 LSM-Tree。Redis 选择了"全内存 + 事件驱动 + 单线程"这条路，而被上百亿的访问量验证了其正确性。理解 Redis 为什么做这些选择，比知道 Redis 做了什么更重要。

在课程的最后，我想分享一段 antirez 在 Redis 5.0 发布时写过的话："I think the best thing about Redis is not the code itself, but the design decisions behind it — and the fact that it stayed simple."（我认为 Redis 最宝贵的不是代码本身，而是代码背后的设计决策——以及它始终保持简单这个事实。）

读源码不是为了模仿，而是为了形成自己的工程判断力。当你面对一个新的系统设计时，能回想起 antirez 在 dict.c 中为什么选择渐进式 rehash 而不是全局停顿，在 ae.c 中为什么用单线程而不是多线程，在 rdb.c 中为什么用 fork + COW 而不是原地序列化——这些判断力会沉淀为你自己的工程直觉。这，才是我们花了 28 章通读 Redis 源码的真正收获。
