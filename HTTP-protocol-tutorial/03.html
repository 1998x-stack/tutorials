第三章：HTTP报文结构

大家好，我是你们的讲师。前面两章我们聊了HTTP的宏观概念和URL，今天终于进入核心——HTTP报文。可以把报文理解为HTTP的"语言"：浏览器说一句请求，服务器回一句响应，一问一答之间完成了数据的交换。如果你能读懂原始的HTTP报文，那调试网络问题的时候就相当于开了"透视眼"。

3.1 请求报文解剖

一个HTTP请求报文由三部分组成：请求行（Request Line）、头部（Headers）、消息体（Body）。请求行和头部之间用回车换行（CRLF，即 \r\n）分隔，头部和消息体之间有一个空行分隔。

来看一个真实的HTTP GET请求报文：

GET /api/users?page=1 HTTP/1.1
Host: api.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36
Accept: application/json
Accept-Language: zh-CN,zh;q=0.9
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Cache-Control: no-cache

我们来拆解第一行——请求行：

  • GET — 请求方法（Method），告诉服务器你想干什么
  • /api/users?page=1 — 请求URI（Request URI），不包含host信息
  • HTTP/1.1 — HTTP版本号

请求行的格式就是 Method SP Request-URI SP HTTP-Version CRLF，SP是空格，CRLF是换行。这个格式简单到几乎有些简陋，但这就是HTTP的设计哲学——让人类也能读懂。

接着是请求头部：每个头部一行，格式是 Header-Name: Value，大小写不敏感。Host头是HTTP/1.1中唯一必需的头部——它告诉服务器你要访问哪个域名，这样才能在一台服务器上部署多个网站（虚拟主机）。

最后留了一个空行。因为这是GET请求，没有消息体。如果有消息体（比如POST请求），空行之后就是body的内容。

3.2 响应报文解剖

服务器收到请求后，会返回一个响应报文。响应报文的结构和请求报文类似，区别在于第一行不是请求行，而是状态行（Status Line）：

HTTP/1.1 200 OK
Date: Mon, 10 Jun 2024 08:30:15 GMT
Content-Type: application/json; charset=utf-8
Content-Length: 156
Server: nginx/1.24.0
Access-Control-Allow-Origin: *
Cache-Control: no-store

{"users":[{"id":1,"name":"张三"},{"id":2,"name":"李四"}],"total":2,"page":1}

状态行格式为 HTTP-Version SP Status-Code SP Reason-Phrase CRLF：

  • HTTP/1.1 — 服务端使用的HTTP版本
  • 200 — 状态码，三位数字，200表示成功
  • OK — 原因短语，对人类友好的状态说明

响应头部和请求头部格式完全一致，也是 Name: Value 一行一个。这里 Content-Type 告诉客户端返回的是JSON数据，Content-Length 告诉body大小是156字节。空行之后就是body——实际的JSON数据。

重要概念：请求报文和响应报文的结构非常对称，就第一行不同。一个是"我想干嘛"，一个是"结果如何"。HTTP的这种对称设计使得客户端和服务器都能用同一套解析逻辑处理报文——这也是HTTP中间件（代理、网关）能透明工作的原因。

3.3 HTTP头部的格式

头部字段（Header）是HTTP报文中最有"信息量"的部分，格式规则如下：

  • 基本格式：字段名: 字段值，字段名大小写不敏感，字段值中空格和空行有特殊含义
  • 多行头部：字段值可以通过在下一行开头加空格或制表符续行，不过实践中很少用到
  • 重复头部：同名字段可以出现多次，等同于用逗号拼接（如 Accept: text/html, application/json）

常用的HTTP头部可以按功能分为以下几类：

分类示例头部说明
通用头Date、Connection、Cache-Control请求和响应都可以出现
请求头Host、User-Agent、Accept、Cookie客户端发送给服务器的信息
响应头Server、Set-Cookie、WWW-Authenticate服务器返回给客户端的信息
实体头Content-Type、Content-Length、Content-Encoding描述消息体属性的头部

这里有个细节：头部字段名虽然大小写不敏感（Content-Type 和 content-type 是一样的），但业界惯例是首字母大写的驼峰格式。我见过几次线上因为自定义头部的命名不一致导致代理层无法识别的案例——虽然HTTP协议说大小写不敏感，但你永远不知道中间件是怎么实现的。建议统一规范。

3.4 消息体（Body）

消息体是HTTP报文的"大货仓"。不是所有请求都有body——GET、DELETE、HEAD等方法通常没有body；POST、PUT、PATCH等方法通常在body中携带数据。

消息体的大小和格式由两个关键头部决定：

  • Content-Type：告诉接收方body的格式，比如 application/json、application/x-www-form-urlencoded、multipart/form-data、text/plain 等。
  • Content-Length：body的字节长度，接收方通过这个值知道要读多少数据。

但如果body太大，或者是流式生成的内容，无法提前知道长度怎么办？这就涉及到 Transfer-Encoding: chunked（分块传输编码）。服务器把数据分成一块一块的发，每块前面有一个十六进制的长度标记，最后用一块大小为0的块来结束：

HTTP/1.1 200 OK
Content-Type: text/plain
Transfer-Encoding: chunked

25
This is the first chunk of data.
1C
and this is the second one.
0

分块传输在实时数据流、大文件下载等场景非常实用。注意，Transfer-Encoding: chunked 和 Content-Length 是互斥的——有Transfer-Encoding就不能有Content-Length。

实战提醒：在调试POST请求的body时，我经常发现开发同学搞不清 application/x-www-form-urlencoded 和 multipart/form-data 的区别。简单来说：前者适合短文本（key=value&key2=value2），后者适合传文件或二进制数据。如果表单里有 <input type="file">，务必用 multipart/form-data。

3.5 用curl查看原始报文

理论讲完了，来看看怎么在现实中观察HTTP报文。我强烈推荐大家学会使用 curl 命令，它是网络调试的瑞士军刀。curl -v （verbose模式）会输出完整的请求和响应报文：

$ curl -v https://api.example.com/users/1

*   Trying 93.184.216.34:443...
* Connected to api.example.com (93.184.216.34) port 443
* TLSv1.3 handshake completed
> GET /users/1 HTTP/1.1
> Host: api.example.com
> User-Agent: curl/8.4.0
> Accept: */*
>
< HTTP/1.1 200 OK
< Content-Type: application/json; charset=utf-8
< Content-Length: 45
< Server: nginx/1.24.0
<
< {"id":1,"name":"张三","email":"zhangsan@example.com"}

注意 > 开头的行是发出的请求，< 开头的行是接收到的响应。TLS握手完成后，curl先发送请求行和头部，然后服务器返回响应。顶部的 * 行是curl的元信息（TCP连接、TLS握手等），不是HTTP报文的一部分。

其他有用的curl参数：

  • curl -v — 显示请求和响应的原始报文
  • curl -X POST — 指定请求方法
  • curl -H "Authorization: Bearer xxx" — 添加自定义头部
  • curl -d "key=value" — 发送POST body（默认Content-Type为 application/x-www-form-urlencoded）
  • curl -o output.json — 将响应body保存到文件

踩坑回顾：有次做线上故障排查，后端说"我返回了数据"，前端说"我收到了400"。两边各执一词。我让后端用curl重现请求，发现他测试时 curl http://localhost:8080/api 用的是HTTP，而前端请求的是HTTPS，Nginx层做了HTTPS卸载后转发到后端，但有个请求头 X-Forwarded-Proto 拼接逻辑有bug。最终靠 curl -v 对比了两次请求的原始头部差异才定位到问题。所以说，看原始报文是定位网络问题的基本功。
