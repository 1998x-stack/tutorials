CUDA库生态

大家好，我是你们的讲师。在 CUDA 编程中有一条非常重要的原则——"能用库就不要手写 kernel"。这句话不是我说的，是 NVIDIA 官方文档里写的。你可能会想："但是我已经学了这么多 kernel 编程技巧，不写岂不是白学了？"

我的回答是：学 kernel 编程是为了让你能理解 GPU 的工作原理，以及当现成的库无法满足你的需求时，有能力写出自己的高性能实现。但在绝大多数情况下，CUDA 生态中已经存在高度优化的库——它们由 NVIDIA 或社区的顶级工程师维护，针对每一代 GPU 架构都做了针对性优化。你自己花几周写出来的 kernel，很难与 cuBLAS 十年迭代的成果竞争。

这一章我们来看 CUDA 生态中最常用的几个库，了解它们能做什么、怎么做、以及如何决定"什么时候用库，什么时候自己写"。

先知道有什么，再决定做什么。 很多 CUDA 初学者写 kernel 只是因为"不知道可以用库"。读了这一章，你会系统地了解 CUDA 的库生态，以后遇到问题，第一反应是搜索"有没有现成的库"而不是"开始写 kernel"。

cuBLAS——GPU上的BLAS

cuBLAS（CUDA Basic Linear Algebra Subprograms）是 CUDA 生态中最基础也最常用的线性代数库。它实现了 CPU 上 BLAS 标准的 GPU 版本——包括向量-向量运算（Level 1）、矩阵-向量运算（Level 2）、矩阵-矩阵运算（Level 3）。

cuBLAS 的 GEMM（通用矩阵乘法）是其核心功能。它的性能经过了十多年的深度优化，在几乎所有 GPU 架构上都能达到接近理论极限的带宽利用率——手动写出比 cuBLAS 更快的 GEMM 是非常困难的事。下面是一个使用 cuBLAS 做 SGEMM（单精度矩阵乘法）的完整示例：

#include <cublas_v2.h>
#include <cuda_runtime.h>
#include <stdio.h>

int main() {
    int N = 1024;
    float *A, *B, *C;         // 设备端矩阵
    float *h_A, *h_B, *h_C;   // 主机端矩阵
    size_t size = N * N * sizeof(float);

    // 分配主机和设备内存
    h_A = (float*)malloc(size);
    h_B = (float*)malloc(size);
    h_C = (float*)malloc(size);
    cudaMalloc(&A, size);
    cudaMalloc(&B, size);
    cudaMalloc(&C, size);

    // 初始化数据并拷贝到GPU
    // ... (初始化代码略)
    cudaMemcpy(A, h_A, size, cudaMemcpyHostToDevice);
    cudaMemcpy(B, h_B, size, cudaMemcpyHostToDevice);

    // 创建cuBLAS句柄
    cublasHandle_t handle;
    cublasCreate(&handle);

    // 调用cuBLAS SGEMM: C = alpha * A * B + beta * C
    float alpha = 1.0f, beta = 0.0f;
    // 注意: cuBLAS默认使用列主序(column-major), 如果在行主序下使用需要转置
    cublasStatus_t status = cublasSgemm(
        handle,
        CUBLAS_OP_N, CUBLAS_OP_N,  // A和B不做转置
        N, N, N,                    // m, n, k
        &alpha,
        A, N,                       // A矩阵, leading dimension = N
        B, N,                       // B矩阵, leading dimension = N
        &beta,
        C, N                        // C矩阵, leading dimension = N
    );

    if (status != CUBLAS_STATUS_SUCCESS) {
        fprintf(stderr, "cuBLAS Sgemm failed!\n");
        return 1;
    }

    // 取回结果
    cudaMemcpy(h_C, C, size, cudaMemcpyDeviceToHost);

    // 清理
    cublasDestroy(handle);
    cudaFree(A); cudaFree(B); cudaFree(C);
    free(h_A); free(h_B); free(h_C);
    return 0;
}

需要注意的一点是：cuBLAS 默认使用列主序（column-major）存储格式，这是 Fortran 和传统 BLAS 的惯例。如果你使用行主序（row-major，C/C++ 的惯例），需要让 cuBLAS 认为矩阵是转置的，或者手动调换参数顺序。

建议： 不要跟 cuBLAS 的 GEMM 较劲。除非你有非常特殊的稀疏模式或融合需求，否则 cuBLAS 的 GEMM 就是你 GPU 上能跑出最快的矩阵乘法。把精力花在 cuBLAS 无法覆盖的地方。

cuFFT——GPU上的FFT

cuFFT（CUDA Fast Fourier Transform）库提供了 GPU 上的一维、二维、三维 FFT 实现。它在信号处理、图像处理、卷积神经网络（用 FFT 加速卷积）、以及科学计算（如求解偏微分方程）中有广泛的应用。

#include <cufft.h>
#include <cuda_runtime.h>

int main() {
    int N = 1024;  // FFT长度

    cufftComplex *d_data;  // 设备端复数数组
    cudaMalloc(&d_data, N * sizeof(cufftComplex));

    // 创建一维FFT计划
    cufftHandle plan;
    cufftPlan1d(&plan, N, CUFFT_C2C, 1);  // C2C = 复数到复数, 1 = 1个batch

    // 执行前向FFT
    cufftExecC2C(plan, d_data, d_data, CUFFT_FORWARD);

    // 执行逆FFT
    cufftExecC2C(plan, d_data, d_data, CUFFT_INVERSE);

    // 销毁计划
    cufftDestroy(plan);
    cudaFree(d_data);
    return 0;
}

// batched FFT —— 一次处理多个独立的FFT
void batched_fft_example() {
    int batch_size = 256;   // 256个独立的FFT
    int N = 1024;
    cufftHandle plan;
    // batch参数告诉cuFFT一次性处理多个FFT
    cufftPlan1d(&plan, N, CUFFT_C2C, batch_size);
    // ... 执行和清理
}

cuFFT 的一个重要特性是batched transform——如果你需要做大量独立的相同长度的 FFT，不要在一个 for 循环里重复调用 cuFFT。相反，在创建 plan 时传入 batch 参数，让 cuFFT 在一个 kernel launch 中完成所有变换。这能极大地减少启动开销并提升整体吞吐量。

cuFFT 还有一个扩展叫做 cuFFTMp（multi-process），它支持在多个 GPU 上并行执行大规模 FFT——适合需要处理超大规模数据的分布式计算场景。

Thrust——GPU上的C++模板库

Thrust 是一个 C++ 模板库，可以理解为"GPU 上的 STL"。它提供了大量通用的并行算法——排序、归约（reduce）、扫描（prefix sum）、变换（transform）等。如果你需要对这些操作手写 kernel，那就走错方向了——Thrust 在底层为你调用了高度优化的实现。

#include <thrust/device_vector.h>
#include <thrust/sort.h>
#include <thrust/reduce.h>
#include <thrust/transform.h>
#include <thrust/sequence.h>

int main() {
    int N = 1 << 20;  // 约100万元素

    // 在GPU上分配向量
    thrust::device_vector<float> d_vec(N);

    // 填充序列: 0, 1, 2, ..., N-1
    thrust::sequence(d_vec.begin(), d_vec.end());

    // 所有元素乘以2 (使用lambda表达式, 需要--expt-extended-lambda)
    thrust::transform(d_vec.begin(), d_vec.end(),
                      d_vec.begin(),
                      [] __host__ __device__ (float x) { return x * 2.0f; });

    // 排序
    thrust::sort(d_vec.begin(), d_vec.end());

    // 求和 (reduce)
    float sum = thrust::reduce(d_vec.begin(), d_vec.end(), 0.0f,
                               thrust::plus<float>());

    // 最大值
    float max_val = thrust::reduce(d_vec.begin(), d_vec.end(), 0.0f,
                                   thrust::maximum<float>());

    // 拷贝回主机
    thrust::host_vector<float> h_vec = d_vec;
    return 0;
}

Thrust 的代码看起来和 C++ STL 非常相似——这正是它的设计目标。你在 CPU 上怎么用 std::sort，在 GPU 上就怎么用 thrust::sort。接口一致，底层自动调用 GPU 加速。

Thrust 的性能也非常优秀。它的 thrust::sort 底层使用 cub::DeviceRadixSort 或 cub::DeviceMergeSort，在 A100 上可以对数十亿个 32 位整数在几毫秒内完成排序。thrust::reduce 底层使用高度优化的 warp-level 和 block-level reduction，比大多数手写实现要快得多。

一个常见的坑： Thrust 默认会为每个算法调用 cudaDeviceSynchronize()，这意味着连续调用多个 Thrust 算法时，GPU 和 CPU 之间会产生不必要的同步。如果你对性能有极高的要求，可以研究一下 thrust::cuda::par.on(stream) 的执行策略，将多个操作放进同一 stream 中避免同步开销。

CUTLASS——模板化的GEMM与Tensor Core支持

CUTLASS（CUDA Templates for Linear Algebra Subroutines）是一个比 cuBLAS 更底层的 C++ 模板库。它的定位是：cuBLAS 是给用户用的黑盒，CUTLASS 是给高级开发者用的白盒。

CUTLASS 允许你用模板参数来定制矩阵乘法的每一个细节——tile size、warp-level 的编排、数据加载方式、累加精度、是否使用 Tensor Cores 等。它的代码生成在编译期完成，因此没有运行时开销。

为什么需要 CUTLASS？典型的场景是：

  • cuBLAS 不支持你需要的操作——比如混合精度 GEMM 的某种特殊组合（例如 FP32 累加 + FP8 输入）
  • 你需要 fusing——在 GEMM 计算之后立即做 ReLU、bias add 等操作，不想把中间结果写回全局内存
  • 你的矩阵形状非常特殊——cuBLAS 对标准形状做得好，但非规则 shape 可能表现不佳

CUTLASS 3.x 引入了 Cute 框架（以 CuTe 为核心），提供了更高级的 tiling 抽象和 Tensor Core 编程模型。NVIDIA 的 cuBLAS 底层在 Hopper 和 Blackwell 架构上也大量使用 CUTLASS 的代码路径。

一句话总结： cuBLAS 是给你的应用程序用的，CUTLASS 是给你的库或框架用的。如果你是做上层开发（如训练模型、跑模拟），用 cuBLAS；如果你在开发一个深度学习框架或一个需要极致性能的算子库，才需要考虑 CUTLASS。

其他重要库速览

CUDA 生态远不止上面几个库。下面这张表列出了其他一些重要的 CUDA 库及其核心应用场景。你不需要记住所有细节，但应该知道它们的存在——这样遇到相应问题时，你能想起"啊，这个有专门的库"：

库名 核心功能 典型应用场景 学习优先级
cuSPARSE 稀疏矩阵运算（SpMV、稀疏三角求解、稀疏 QR 分解） 图计算、有限元分析、推荐系统 中（学科相关时才需要）
cuSOLVER 稠密和稀疏矩阵的分解与求解器（LU、Cholesky、SVD、QR） 数值线性代数、物理模拟、优化问题 中（学科相关时才需要）
cuRAND GPU 上的随机数生成器（支持多种分布） 蒙特卡罗模拟、随机算法、强化学习 高（经常被用到）
cuDNN 深度学习算子库（卷积、池化、归一化、激活函数、RNN） 深度学习训练与推理（PyTorch/TensorFlow 底层均使用它） 高（深度学习必知）
cuTensor / cuTensorNet 张量运算和张量网络收缩 量子计算模拟、张量网络方法、AI 科学计算 低（前沿领域使用）
NCCL 多 GPU 通信库（AllReduce、AllGather、Broadcast 等集合通信） 分布式深度学习训练、多 GPU 并行 高（多 GPU 场景必学）
CUB CUDA 内核级原语（block-level/warp-level 的 parallel primitives） 自定义 kernel 开发中调用高度优化的 reduce/scan/sort 中（写 kernel 时很有用）

值得一提的是 CUB——它不是一个"应用程序库"，而是一个"kernel 作者的工具箱"。当你在写自定义 kernel 并需要一个高效的 block-level reduction 或 warp-level scan 时，CUB 提供了可以直接调用的设备端原语。它的性能通常比你手写的要好，因为 CUB 的作者是 CUDA 编译器团队的工程师。

学习策略： 对于以上库，建议的掌握程度为：cuBLAS 和 Thrust 要熟练使用；cuFFT 和 cuDNN 要了解基本用法；cuSPARSE、cuSOLVER、cuRAND 需要时再查文档即可。CUB 值得深入了解，因为它在自定义 kernel 开发中非常实用。

先库后核原则

我们在这一章的开头就说了"能用库就不要手写 kernel"。但现实当然不是这么黑白分明的。这里给出一个更实用的决策框架，帮助你在"用库"和"自己写"之间做选择：

  • 情况一：库能直接满足需求 → 用库。比如你要做矩阵乘法 → cuBLAS；排序 → Thrust；FFT → cuFFT。不要犹豫，直接上库。库的实现经过了大量架构验证，而且随着 CUDA 版本升级会自动受益于新的优化。
  • 情况二：库能实现大部分功能，但需要少量自定义 → 看情况。如果可以在库的输出上做后处理（比如 relu after GEMM），可以先用库再做后续操作。但如果这个"后处理"本身需要大量带宽，你可能需要考虑自己写融合 kernel。
  • 情况三：库无法实现你的算法 → 动手写 kernel。但写之前先问自己两个问题：第一，有没有类似的库可以参考实现（比如 CUTLASS 的代码可以作为手写 GEMM 的参考）？第二，能不能用多个库的组合来达到目的？
  • 情况四：你需要极致性能且库的性能不够 → 这通常是框架开发者的场景。比如 cuBLAS 的 GEMM 已经很快了，但你想要 fuse 一个 bias add 和一个 ReLU 来节省一次全局内存的读写。这时 CUTLASS 是你的朋友——它让你在保持高性能的同时加入自定义逻辑。

最后一条建议：先测量，再决定。在决定"要不要写一个自定义 kernel"之前，先跑一下现有的库，看看它的性能是否满足你的需求。很多情况下，库已经够快了——你写一个更快的并不会给你的应用带来肉眼可见的提升。

一个让我铭记的教训

最后分享一个亲身经历的故事。几年前我需要在一个 GPU 加速的图算法中做一个并行归约（parallel reduction）——计算数十亿个浮点数的和。我花了两周时间，仔细阅读了 NVIDIA 的 reduction 优化博客（就是那个著名的"Optimizing Parallel Reduction in CUDA"），手写了一个逐层优化的归约 kernel——warp-level reduction、避免 bank conflict、循环展开、完全展开最后的 warp 归约。最终的 kernel 跑起来，我很得意。

然后我无意中看到了 CUB 库中的 cub::DeviceReduce::Sum()，就顺手测了一下。结果——CUB 比我手写的 kernel 快了 3 倍。3 倍。两星期的工作，被一个 5 行的库调用完胜。

更扎心的是，当我回去分析原因时发现，CUB 团队在 SM 级别的调度策略和寄存器分配上做了很多我根本不知道的优化——而这些优化只在最新的架构上有效，CUB 每次 CUDA 版本更新都会重新调整。我一个人永远追不上一个专门团队持续迭代的速度。

从那以后，我彻底接受了"先找库"的开发流程。不是不需要学习 kernel 编程——实际上我正是因为理解了 kernel 编程的原理，才能更好地评判一个库是否适合我的场景。但在绝大多数情况下，信任库比信任自己更明智。

在下一章，我们将进入 CUDA 的一个高级特性——CUDA Graph。它允许你把一系列 kernel launch 和内存操作预先定义成一个"图"，然后一次性提交给 GPU 执行。对于有大量重复性计算的应用（如深度学习训练中的每个 step），CUDA Graph 可以将 kernel launch 的开销降低到几乎为零。这是 CUDA 10.0 以来最重要的性能优化特性之一，值得你花一整章来学习。
