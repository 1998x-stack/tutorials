大模型做裁判

上一章我们讲到 FineWeb-Edu 用 Llama-3-70B 标注了 50 万条数据来训练教育质量分类器。这里有一个有趣的循环：我们正在用大模型来评判大模型的训练数据质量。

2024 到 2025 年间，行业经历了一个明显的范式转变。2023 年及以前，数据质量标注主要靠人工规则和启发式方法；2024 年开始，越来越多的团队选择用 LLM 来做"裁判"——让大模型对 Web 页面的教育价值、内容质量、信息密度进行打分。FineWeb 团队使用 Llama-3-70B 对数十万 Web 页面做教育评分；AI2 的 DCLM 项目则使用 StarCoder 对代码数据进行自动质量评估。

为什么用 LLM 做标注？答案很简单——效率。让人类标注 50 万条数据需要几十个人工作好几个月，成本数十万美元。而让 LLM 标注，只需要几天的 API 调用时间和几千美元的预算。更重要的是，LLM 的标注标准比人类更一致——同一个文档，不同的人可能打不同的分，但同一个 LLM 在相同的 prompt 下会给出可复现的结果。

Prompt 工程设计

用 LLM 做标注，核心在于 prompt 设计。一个好的评分 prompt 需要明确指出评估维度和评分标准，减少 LLM 的主观臆断。

下面是一个实际用于教育质量评分的 prompt 示例，中文版本：

你是一位教育内容质量评估专家。请对下面的文档进行评分。

评分维度：
1. **内容质量**（0-5）：信息是否准确、深度如何、是否有实质内容
2. **教育价值**（0-5）：是否有助于学习、是否结构清晰、是否有教学意义
3. **权威性**（0-5）：来源是否可靠、是否有引用、是否经过审核
4. **可读性**（0-5）：语言是否流畅、组织是否合理、难度是否适中

综合评分规则：
- 5分：高质量的教科书、学术论文级别
- 4分：优质的技术博客、Wikipedia条目
- 3分：一般的参考文章、中等的论坛回答
- 2分：浅显的入门内容、纯信息罗列
- 1分：SEO堆砌、机器翻译、无意义内容
- 0分：纯噪音、广告、非内容页面

文档内容：
---
{文档内容}
---

请先给出四个维度的单独评分（JSON格式），然后给出综合评分（0-5的整数），最后用一句话解释评分理由。
重要观察： 使用 LLM 标注意味着标注质量本身依赖 LLM 的能力，这是一个系统性偏差。如果用的标注模型在某个领域表现较弱（比如用英语为主的 Llama-3 标注中文内容），那标注结果可能对中文高质量内容有偏见。FineWeb-Edu 团队发现，同样的 prompt 用不同版本的 LLM 标注，结果一致性大约在 85-90%——不是 100%。

在 FineWeb-Edu 的实验中，他们对比了多种 prompt 变体对标注结果的影响。一个关键发现是：prompt 中的评分标准和示例对结果影响巨大。如果 prompt 中强调"学术风格"，LLM 会偏好正式、严谨的文本；如果强调"实用价值"，LLM 会更青睐教程和指南类内容。FineWeb-Edu 最终选择了偏向"教育价值"的 prompt 版本，因为他们的目标是优化模型在学术 benchmark 上的表现。

还有一个细节点：标注的粒度。是让 LLM 输出一个整数分数（如 4 分），还是输出一个浮点数（如 3.7 分）？FineWeb-Edu 的实验表明，整数评分更稳定——LLM 在整数评分上的跨模型一致性（Cohen's Kappa）比浮点数高约 10%。原因可能是浮点数评分让 LLM 有了更多的"纠结空间"，反而引入了不必要的随机性。

蒸馏：从 LLM 到轻量分类器

虽然 LLM 标注质量很高，但它太慢了。Llama-3-70B 在一张 A100 上每秒只能处理大约 2 篇文档，而 FineWeb 语料有 15 万亿 token 的文档需要评分——用 LLM 硬算需要几万 GPU 小时。

解决方案是蒸馏（distillation）：用 LLM 标注一小部分数据（比如 50 万条），然后用这些标注数据训练一个轻量级分类器（如 fastText 或 tiny Transformer），让分类器学会"复制"LLM 的判断模式。一旦训练完成，分类器就可以代替 LLM 去处理剩余的数十亿文档。

方法推理速度硬件需求每百万文档成本质量
Llama-3-70B 直接标注~2 docs/sec1× A100-80G~$800最佳（~92% 准确率）
GPT-4 API 标注~5 docs/sec（API 限制）无~$2,000-5,000类似 70B（~91%）
fastText 分类器（蒸馏后）~100K+ docs/sec1× CPU 核~$0.01良好（~85%）
Tiny Transformer（蒸馏后）~10K docs/sec1× T4 GPU~$0.10较好（~88%）

这不就是经典的"教师-学生"知识蒸馏框架吗？LLM 是那个"知识渊博但行动缓慢"的教师，fastText 是那个"学习能力强但不够聪明"的学生。教师把知识浓缩在标注数据中，学生通过学习来模仿教师的判断。

蒸馏流程中还有一个值得注意的工程问题：标注数据的"难度分布"。如果所有被 LLM 标注的文档都集中在"明显高质量"和"明显低质量"两端，那训练出来的分类器对"中间地带"的文档会判断不准。FineWeb-Edu 团队发现，主动在标注集中增加 中等质量文档的采样比例，让分类器在 2-4 分区域有更多的训练样例，可以显著提升分类器的整体判别能力。这个优化带来了约 1.5-2 个百分点的 F1 提升。

蒸馏的艺术： 在 FineWeb-Edu 的实践中，蒸馏的关键不是简单地用 LLM 标注 → 训练分类器。重要的是做"标签平滑"——不是用 LLM 的硬标签（如"4 分"），而是用 LLM 打分的概率分布（如 "4 分: 0.7, 3 分: 0.2, 5 分: 0.1"）。这样分类器学到的不仅是"什么分数是对的"，还学到了"什么样的文本介于两个分数之间"，效果提升约 2-3 个百分点。

LLM 标注的偏差和局限

LLM 做标注并非没有问题。实际上，它引入了一些非常微妙的偏差，需要认真对待。

自指风险： 用 LLM 标注数据 -> 用标注数据训练分类器 -> 用分类器过滤训练数据 -> 用过滤后的数据训练新的 LLM。这个循环存在一个隐患：如果用来标注的 LLM 本身有认知偏差（比如偏好某种写作风格、低估某些领域的价值），这些偏差会通过标注数据被放大。2026 年零熵公司（Zero Entropy）在一篇分析中指出：反复使用 LLM 标注和筛选会产生 "语料单一化（corpus monoculture）"——数据越来越像 LLM 的"理想"输出，多样性持续下降。

另外，FineWeb-Edu 的评分明显偏向学术风格。这意味着一些非正式的但极具价值的内容——比如一个程序员在论坛上写的精彩技术解答、一篇富有创意的小说、一段生动的对话数据——可能会被分类器低估。这些内容虽然在学术 benchmark 上"没用"，但对于训练一个全面、通用的 LLM 来说是宝贵的。

如何缓解这个偏差？一个做法是使用多个不同的 LLM 做标注，取它们的共识（如三个模型都同意才作为标签）。另一个方法是定期引入人工评估作为"校准点"，纠正 LLM 的系统性偏差。DCLM 团队采用了一种有趣的混合策略：用 StarCoder 做代码质量标注，同时用 GPT-4 做文本质量标注——让每个领域的"专家模型"做自己擅长的评审工作。

还有一个值得注意的实操细节：标注数据的分布控制。如果用 LLM 标注 50 万条数据，这 50 万条应该是怎样的分布？FineWeb-Edu 的做法是从全体 FineWeb 语料中随机抽样，力求抽样分布和整体分布一致。但 DCLM 实验表明，主动采样低质量区域的文档加入训练集（即确保分类器见过足够多的"反面案例"），可以让分类器的判别边界更加清晰。两种策略各有优劣，取决于你想要分类器在哪个区域更敏感。

好了，LLM 辅助标注的潜力巨大，但也有其风险。从数据清洗的角度看，我们一直聚焦于"选出好数据"和"去掉坏数据"。但还有一种特殊的数据过滤场景需要我们警惕——训练数据中意外包含的评测集内容。如果模型在训练阶段就"看过"考试题目，那它的考试成绩就失去了意义。下一章我们聊评测集去污染——维护模型评估可信度的最后一道防线。
