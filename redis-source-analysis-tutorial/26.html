第26章：Lua 脚本与事务

大家好，我是你们的讲师。前几章我们详细解剖了 Redis 的网络 IO 和命令处理流程，你已经知道了一条 SET 命令从网卡到内核再到 redisServer 的完整旅程。但 Redis 不仅仅是一个单命令执行器——它还提供了事务和Lua 脚本两种机制，让你能批量执行多条命令并保证原子性。很多同学把 Lua 脚本单纯当成"原子性批处理"，把事务当成"简化版脚本"，但如果你读过源码就会发现：这两套机制尽管目标相似，却走了完全不同的实现路径，各自的性能特性和适用场景差异巨大。

本章我们从源码层面深入 script.c / script_lua.c 和 multi.c，看看 Lua 引擎是怎么嵌入 Redis 的，MULTI/EXEC 事务又是如何用状态机和命令队列拼凑出"原子性"的。理解了这些，你才能真正用好 EVAL 和 MULTI，避免那些让人头皮发麻的生产事故。

Lua 脚本在 Redis 中的位置

Redis 从 2.6 版本开始嵌入了 Lua 5.1 解释器，这是整个 deps/lua/ 目录存在的意义。为什么是 Lua？因为 Lua 的解释器极其轻量，核心代码只有约 2 万行 C 代码，编译后体积仅 100KB 左右，且提供了沙箱化执行的能力。这些特性让 Lua 成为嵌入到 Redis 这样追求极致性能的系统中的理想选择。

Lua 引擎在 Redis 中的核心骨架由三个文件构成：

  • script.c：脚本管理的通用框架层。它定义了脚本执行的生命周期——加载、缓存、执行、超时中断、复制。它不直接与 Lua 交互，而是通过一个抽象的 scriptRun() 接口与后端对接，这个设计其实为后来引入的 Redis Function（7.0+）奠定了基础。
  • script_lua.c：Lua 引擎的具体实现。它创建 lua_State 虚拟机，注册 Redis 提供的 C 函数（如 redis.call、redis.pcall），处理错误和类型转换。
  • functions.c：7.0 引入的 Redis Function 子系统，可以理解为脚本的"一等公民"版——脚本被注册到 Redis 中并持久化到 RDB/AOF，不像 EVAL 那样依赖客户端重复发送。
小贴士：每个连接到 Redis 的客户端都有一个独立的 lua_State 吗？不是的。Redis 全局只维护一个 lua_State，所有脚本共享同一个 Lua 环境。这意味着一个脚本中定义的全局变量会被其他脚本看到——虽然实践中很少这样用，但了解这个细节有助于你理解 Lua 沙箱的安全边界。

EVAL / EVALSHA 源码解析

当我们执行 EVAL "return redis.call('GET', KEYS[1])" 1 mykey 时，Redis 内部发生了什么？让我们走进 evalCommand 的核心流程：

/* script_lua.c — evalCommand 简化流程 */
void evalCommand(client *c) {
    /* 1. 检查脚本是否为 SHA1 格式 */
    if (isSHA1(c->argv[1]->ptr)) {
        /* 尝试从缓存字典中查找 */
        sds sha = c->argv[1]->ptr;
        if (dictFind(server.lua_scripts, sha) == NULL) {
            addReplyError(c, "NOSCRIPT No matching script");
            return;
        }
        evalShaCommand(c);  /* 走 EVALSHA 路径 */
        return;
    }

    /* 2. 从参数中分离 KEYS[] 和 ARGV[] */
    int numkeys = atoi(c->argv[2]->ptr);
    /* 3. 创建 Lua 函数：将脚本包装为 f_<SHA1> 函数 */
    char funcname[256];
    int sha1_len = 40 + 2;  /* f_ + 40 位 hex */
    /* 计算脚本的 SHA1 摘要 */
    sha1hex(sha1, c->argv[1]->ptr, sdslen(c->argv[1]->ptr));
    /* 4. 调用 luaCallFunction 执行 */
    luaCallFunction(c, funcname, numkeys, 0);
}

这个流程中有几个关键设计值得一提：

  • SHA1 摘要：每个脚本在执行前都会计算 SHA1 哈希，这个哈希既用作缓存键（存入 server.lua_scripts 字典），也用作函数名。所以 EVALSHA 可以直接用 SHA1 找到已缓存的脚本，避免重复传输脚本体。
  • server.lua_scripts 字典：这是一个 dict *，以 SHA1 为 key，以脚本源码的 sds 为 value。所有通过 EVAL 或 SCRIPT LOAD 加载的脚本都会缓存到这里，直到 SCRIPT FLUSH 或服务器重启才会清空。
  • 脚本缓存不上限：这是有意为之——antirez 认为脚本的数量应该由开发者控制，Redis 不做强制限制。但如果滥用，会导致内存持续增长。

再看 SCRIPT LOAD 的实现——它只做了两件事：计算 SHA1、存入 server.lua_scripts，但不执行脚本。这比 EVAL 少了一个执行步骤，适合在初始化阶段预先加载脚本。

脚本执行机制

当 luaCallFunction 真正执行 Lua 脚本时，内部有一整套精心设计的参数传递和错误处理机制：

/* script_lua.c — luaCallFunction 核心逻辑 */
int luaCallFunction(client *c, char *funcname, int numkeys, int flags) {
    lua_State *lua = server.lua;  /* Redis 全局唯一的 Lua VM */

    /* 1. 将 KEYS[] 推入 Lua 栈 */
    lua_newtable(lua);
    for (int j = 0; j < numkeys; j++) {
        lua_pushstring(lua, c->argv[j + 3]->ptr);  /* argv[3] 之后是 key */
        lua_rawseti(lua, -2, j + 1);
    }

    /* 2. 将 ARGV[] 推入 Lua 栈 */
    int arg_idx = 3 + numkeys;
    lua_newtable(lua);
    for (int j = arg_idx; j < c->argc; j++) {
        lua_pushstring(lua, c->argv[j]->ptr);
        lua_rawseti(lua, -2, j - arg_idx + 1);
    }

    /* 3. 执行 Lua 函数 */
    if (lua_pcall(lua, 2, 1, 0) != LUA_OK) {
        /* 处理 Lua 运行时错误 */
        addReplyErrorFormat(c, "ERR Error running script: %s",
                            lua_tostring(lua, -1));
        lua_pop(lua, 1);
        return C_ERR;
    }

    /* 4. 将 Lua 返回值转换为 Redis 回复 */
    luaReplyToRedisReply(c, lua);
    return C_OK;
}

执行过程中的两个重要机制：

  • lua-replicate-commands：这个配置项（默认开启，7.0+ 成为唯一行为）控制脚本内的写命令如何复制到从库。开启时，脚本中的每个写命令会被单独复制（逐命令复制），而不是将整个脚本作为整体复制。这样做的好处是：主库和从库各自执行相同的命令序列，即使 Lua 本身有随机性（比如 math.random），也不会导致主从不一致。
  • 脚本超时中断：Redis 默认的 lua-time-limit 是 5000ms。当脚本运行超过这个阈值，Redis 不会立即杀死它（那样会破坏 Lua VM 的状态），而是给脚本发一个"你超时了，请自己结束"的信号。Redis 通过 lua_timedout 标志位标记超时状态，每次 redis.call() 返回时检查这个标志。这意味着：如果你的脚本不调用 redis.call()，只是纯 Lua 做死循环，Redis 是无法中途中断它的——这是 Lua 沙箱的一个硬伤。
踩坑实录：我曾经在一个电商大促项目中遇到过一次严重故障。运营同学写了一个 Lua 脚本，里面循环调用了 5 万次 redis.call('GET', key) 做复杂的价格校验，本意是想原子性保证库存扣减。但问题是，这个脚本执行耗时达到了 8 秒——直接导致整个 Redis 实例在这 8 秒内无法服务任何其他请求。所有依赖这个 Redis 实例的服务集体超时，雪崩效应导致线上瘫痪了 15 分钟。事后分析：首先，EVAL 每次都要传输完整的脚本体，网络开销大；其次，脚本执行期间单线程阻塞了所有其他命令。正确的做法是：用 SCRIPT LOAD 预先加载脚本得到 SHA1，然后在业务代码中用 EVALSHA 调用；更重要的是，脚本执行时间必须控制在 1ms 以下的原则绝对不能突破。

事务实现：MULTI / EXEC / WATCH

事务在 Redis 中的实现与 Lua 脚本有本质不同。如果说 Lua 脚本是"去 C 侧执行自定义逻辑"，那么事务就是"在 C 侧批量排队执行已有命令"。事务的核心文件是 multi.c，整个流程围绕一个简单的状态机展开。

/* multi.c — MULTI 命令设置事务标志 */
void multiCommand(client *c) {
    /* 设置 CLIENT_MULTI 标志位，标记客户端处于事务模式 */
    c->flags |= CLIENT_MULTI;
    addReply(c, shared.ok);
}

/* multi.c — 事务中的命令不立即执行，而是入队 */
int processCommand(client *c) {
    /* 如果客户端处于事务模式，将命令入队而非执行 */
    if (c->flags & CLIENT_MULTI) {
        /* 某些命令（如 WATCH、MULTI、EXEC、DISCARD）不受事务影响 */
        if (!(c->cmd->proc == execCommand ||
              c->cmd->proc == discardCommand ||
              c->cmd->proc == multiCommand ||
              c->cmd->proc == watchCommand)) {
            queueMultiCommand(c);  /* 入队到 c->mstate.commands */
            addReply(c, shared.queued);
            return C_OK;  /* 不入队时不执行 */
        }
    }
    /* 正常执行路径... */
}

/* multi.c — EXEC 批量执行所有已入队命令 */
void execCommand(client *c) {
    /* 1. 检查 WATCH 键是否被修改 */
    if (isWatchedKeyModified(c)) {
        c->flags |= CLIENT_DIRTY_CAS;
    }

    /* 2. 如果 WATCH 检测到修改，事务失败 */
    if (c->flags & CLIENT_DIRTY_CAS) {
        addReply(c, shared.nullarray[c->resp]);  /* 返回 nil */
        unwatchAllKeys(c);
        return;
    }

    /* 3. 逐条执行已入队的命令 */
    for (int j = 0; j < c->mstate.count; j++) {
        c->mstate.commands[j].argc = ...;
        c->mstate.commands[j].argv = ...;
        call(c, c->mstate.commands[j], CMD_CALL_FULL);
    }
}

关键数据结构 client.mstate（multiState 类型）记录了事务的状态：

/* server.h — multiState 结构体 */
typedef struct multiState {
    multiCmd *commands;    /* 已入队的命令数组 */
    int count;             /* 已入队的命令数量 */
    int alloc;             /* 已分配的容量 */
    int cmd_flags;         /* 累积的命令标志 */
    int cmd_inv_flags;     /* 反转的命令标志 */
} multiState;

typedef struct multiCmd {
    robj **argv;           /* 参数数组 */
    int argc;              /* 参数个数 */
    struct redisCommand *cmd;  /* 指向命令表条目 */
} multiCmd;

WATCH 乐观锁的实现也很精巧。当客户端执行 WATCH key 时，Redis 将这个客户端注册到 server.watched_keys 字典中。这个字典以 key 为键，以 watchedKey 链表为值，每个 watchedKey 记录了哪个客户端正在监视这个 key。当任何命令修改了这个 key（比如 SET、HSET、DEL 等），都会触发 touchWatchedKey() 函数，将监视该 key 的客户端标记为 CLIENT_DIRTY_CAS。之后这个客户端执行 EXEC 时，看到脏标记就会直接返回 nil——这就是"乐观锁"的全部秘密。

Redis 事务 vs ACID

很多从关系数据库转过来的同学会问：Redis 事务支持 ACID 吗？答案比你想象的要复杂一些：

ACID 属性 Redis 事务支持情况
原子性（Atomicity） 部分支持。EXEC 批量执行过程中不会被打断，但如果中间某条命令失败，后面的命令仍然继续执行，不会回滚
一致性（Consistency） 基本支持。数据类型约束在命令执行层面会被检查，不会出现类型混乱
隔离性（Isolation） 天然支持。Redis 是单线程执行模型，EXEC 执行期间不会有其他命令穿插
持久性（Durability） 取决于持久化配置。仅 AOF + always fsync 才能保证，但性能代价极高

Redis 事务不支持回滚是 antirez 有意为之的设计决策。他认为：事务中的命令失败只有两种可能，一是语法错误（命令本身写错了），这类错误应该在入队时就发现；二是运行时错误（比如对 string 类型执行了 LPUSH），这类错误属于编程错误，应该修正代码而不是依赖回滚。与其在 Redis 中实现复杂的回滚机制（这会影响性能和代码复杂度），不如让开发者在上层检测并处理错误。

小贴士：语法错误和运行时错误的区别很关键。语法错误（比如命令名写错）会在 queueMultiCommand 时被检测到，Redis 会直接拒绝整个 EXEC。但运行时错误（比如对 string 执行 LPUSH）只有在 EXEC 执行到那条命令时才会发现——此时前面的命令已经执行完毕，无法撤回。这也意味着在 EXEC 返回的数组结果中，可能既有成功回复也有错误回复，你需要自己检查每个结果。

脚本与事务的原子性保证

事务和 Lua 脚本都声称自己具有"原子性"，但此原子性非彼原子性。我们来做一个对比：

Redis 事务的原子性：MULTI 加 EXEC 保证了多条命令在执行批次上的原子性——EXEC 命令被处理时，Redis 会在这个事件循环迭代中连续执行所有入队命令，中间不处理其他客户端的请求。但如果执行到一半机器崩溃，已经执行的部分不会回滚。

Lua 脚本的原子性：整个脚本作为一个整体在事件循环中执行，期间 Redis 不会处理任何其他命令。而且 Lua 脚本还多了一层保证：如果脚本执行过程中 Redis 崩溃重启，脚本要么完全没执行，要么执行完毕——因为 AOF 和复制日志是以整个脚本为单位记录的。

在实际生产中，Lua 脚本通常比事务更受欢迎，原因很简单：Lua 脚本可以包含条件逻辑和循环，能够实现"读-改-写"这种 CAS 操作，而事务的每条命令都是预先排好的，没有条件判断能力。

特性 Lua 脚本 (EVAL) 事务 (MULTI/EXEC) Pipeline
原子性 整个脚本原子执行 批量命令原子执行 无保证
条件逻辑 支持（if/else/loop） 不支持（命令预编排） 不支持
网络往返 1 次（脚本体大时传输开销大） MULTI + N条QUEUED + EXEC = N+2 次 1 次
可调试性 差（黑盒执行） 中（每条命令独立回显） 好
缓存支持 EVALSHA + SCRIPT LOAD 不适用 不适用
适合场景 复杂原子操作、限流、计数器 批量无依赖操作 批量只读查询
经验总结：大多数需要"原子性"的场景，都应该优先考虑 Lua 脚本而不是事务。事务在 Redis 中的地位更像是一个"渐进增强"——当你已经有了一套命令序列，想确保它们不被穿插执行时，套上 MULTI/EXEC 是最简单的改造方式。而如果你需要根据某个 key 的当前值来决定下一步操作，那必须用 Lua 脚本。

Lua 脚本和事务都是 Redis 在"单线程事件循环"框架下实现原子性的不同思路，它们各自有清晰的适用边界。下一章我们要讨论的话题更加深入——如何通过 Module 系统完全跳出 Redis 内置命令的限制，用 C/C++ 扩展 Redis 的能力边界。从"怎么用"到"怎么扩展"，这正是源码阅读带来的进阶之路。
