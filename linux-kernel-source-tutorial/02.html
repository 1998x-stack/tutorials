大家好，我是你们的讲师。上一章我们了解了Linux的诞生故事和它为什么会成为我们今天学习的对象。从这一章开始，我们要做一件重要的事情——建立一幅完整的内核地图。

想象一下，你面前耸立着一座巨大的建筑。如果你直接走进去，在走廊里瞎转，很快就会迷失方向。你需要一张建筑蓝图——知道哪里是入口、哪里是核心机房、哪里是配电室、哪里是管道井。这样当你走进任何一扇门时，你都知道自己处在建筑的什么位置。

Linux 0.11的内核源码就是这样一座建筑。本章我们将从5000米的高空俯瞰整座建筑，理解它的设计哲学、目录结构、模块关系以及启动流程的宏观轮廓。这章没有逐行代码分析，而是为接下来27章的分模块逐行阅读做好导航。

引言——从整体俯瞰Linux 0.11的14个目录

Linux 0.11的源码压缩包解开后，会呈现出大约14个顶级目录以及顶层的Makefile和build.c文件。整个内核代码大约14,000行C代码和不到2,000行汇编——什么概念呢？一本《三体》大约有20万字，而Linux 0.11的所有代码打印出来，大概只有一本薄薄的200页小册子。

但这份"小册子"的内容密度极高。它的宏内核设计哲学可以概括为一句话：所有核心操作系统的服务——进程管理、内存管理、文件系统、设备驱动、网络协议——全部运行在内核态，共享同一个地址空间。没有消息传递的开销，没有用户态/内核态频繁切换的代价。简单、直接、高效——这就是Linus的设计风格。

宏内核的代价：代码的任何一个模块崩溃，整个内核就会崩溃。这也是Tanenbaum批评宏内核的核心论点——微内核中驱动崩溃只会影响该驱动，不会拖垮整个系统。但对于1991年的PC硬件来说，宏内核的性能优势才是王道。

源码目录结构

下面这张表列出了Linux 0.11的14个顶级目录和关键文件。我建议你打开源码目录对照着看，每一行都对应真实的代码。

目录/文件代码量作用
boot/~600行汇编引导启动：bootsect.S、setup.S、head.S，从BIOS手中接过控制权，进入保护模式
init/~200行C内核初始化入口：main.c，调用所有子系统初始化函数
kernel/~2,000行C+汇编内核核心：进程调度（sched.c）、信号（signal.c）、系统调用（sys_call.s）、中断（traps.c, asm.s）
kernel/blk_drv/~900行C块设备驱动：硬盘（hd.c）、软盘（floppy.c）、RAM盘（ramdisk.c）
kernel/chr_drv/~1,200行C字符设备驱动：串口（serial.c）、键盘（keyboard.S）、终端（tty_io.c, console.c）
kernel/math/~800行C+汇编数学协处理器仿真（80387 math emulation）
mm/~900行C内存管理：页面分配（memory.c）、page.s、段页式转换
fs/~3,000行CMINIX文件系统完整实现：open、read、write、exec、pipe、文件描述符管理
lib/~400行C内核库函数：字符串操作、malloc、标准I/O的底层支撑
include/~1,500行头文件主目录（各种.h文件）
include/asm/~200行汇编相关头文件：segment.h、system.h、io.h
include/linux/~1,000行内核数据结构头文件：fs.h、sched.h、mm.h、tty.h、head.h等
include/sys/~500行系统调用相关头文件：types.h、stat.h、fcntl.h等
tools/~150行C构建辅助工具：build.c，将编译产物打包成可启动内核镜像

注意看，Linux 0.11没有net/目录——网络协议栈是在v0.95之后才加入的。0.11也没有arch/目录——x86是当时唯一的支持架构。这种极简结构使得源码的导航变得异常容易。

如果按照功能耦合度来划分，这些模块的关系大致如下：

+------------------------------------------+
|           用户进程 (User Process)         |
+------------------------------------------+
        | 系统调用 (sys_call.s) |
        v
+------------------------------------------+
|  fs/        |  kernel/       |  mm/      |
| 文件系统    | 进程调度/信号  | 内存管理  |
|  open/read  |  sched/signal  |  memory   |
|  exec/pipe  |  sys_call      |  page.s   |
+-------------+---------------+-----------+
|  kernel/blk_drv/  |  kernel/chr_drv/   |
|  hd.c / floppy.c |  tty_io.c / keyboard.S |
+------------------------------------------+
|              init/ (main.c)               |
+------------------------------------------+
|              boot/ (启动引导)              |
+------------------------------------------+
|                硬件层 (x86)               |
+------------------------------------------+

这个层次结构一目了然：boot/在最底层直接与硬件对话，init/负责拉起所有子系统，kernel/、mm/、fs/构成内核的核心服务层，驱动层暴露设备接口，最上层是用进程。

宏内核 vs 微内核

第一章我们提到了1992年的Tanenbaum-Torvalds论战。现在是时候深入理解这场论战的技术本质了。

宏内核（Monolithic Kernel）和微内核（Microkernel）是两种截然不同的操作系统设计范式。它们的主要区别在于——内核态中应该包含多少功能。

宏内核把所有操作系统服务都放在内核态：进程管理、内存管理、文件系统、设备驱动、网络协议栈，全部运行在Ring 0，共享同一个地址空间。模块之间直接通过函数调用通信，性能极高。但任何一个模块的Bug都会导致整个内核崩溃。

微内核只把最基本的功能放在内核态：进程间通信（IPC）、基本的调度、最基本的内存管理。文件系统、设备驱动、网络协议等全部以用户态服务的形式运行。服务之间通过IPC通信，互相隔离，一个驱动崩溃不会影响内核。但每次IPC调用都有上下文切换的开销。

特性宏内核 (Linux)微内核 (MINIX)
内核态代码量大（所有服务在内核态）极小（仅IPC和基本调度）
性能高（直接函数调用）较低（IPC通信开销）
模块隔离性差（一个Bug拖垮全系统）好（每个服务独立进程）
开发难度中等高（复杂的IPC设计）
典型代表Linux, FreeBSD, WindowsMINIX, QNX, seL4, macOS (混合)
混合内核的真相：有趣的是，现代Linux内核虽然仍然是宏内核架构，但已经通过"内核模块"（Loadable Kernel Module, LKM）机制实现了部分微内核的优点——驱动可以在运行时加载和卸载。而macOS采用的XNU内核则是混合内核——内核中既有宏内核的部分也有微内核的部分。实际工程中，很少有"纯粹"的宏内核或微内核。

Linus的立场非常务实：对于1991年的个人电脑，微内核的IPC开销是不可接受的。他在那场论战中写道："在386上，一个进程切换需要微秒级别的时间。如果每次文件系统操作都要经过一次IPC（从用户态到文件服务器再到磁盘驱动），你会损失几十微秒。乘以每秒几百次的系统调用，这个损失就非常显著了。" 历史证明，Linus的实用主义选择是正确的。

内核编译产物

当我们执行make之后，Linux 0.11会生成一系列中间文件和最终产物。理解编译产物的结构，对理解内核的内存布局非常有帮助。

最终的产出是Image文件——这是一个完整的、可以写入软盘启动扇区的内核镜像。它是这样构建出来的：

# Linux 0.11 编译产物结构

编译过程:
  1. bootsect.s  →  bootsect   (引导扇区, 512字节)
  2. setup.s     →  setup      (设置代码, 4个扇区)
  3. 所有C文件 + head.s  →  system    (完整内核, ~130KB)

  4. build 工具将三者拼接:
     +--------+--------+------------------------+
     |bootsect|  setup |        system          |
     | 512B   |  ~2KB  |   head.s + 所有C代码    |
     +--------+--------+------------------------+
     |                  Image (~134KB)           |
     +------------------------------------------+

运行时内存布局:
  0x00000 - 0x8FFFF: system模块(包括head.s和所有内核代码)
  0x90000 - 0x901FF: bootsect (临时驻留,后被覆盖)
  0x90200 - 0x903FF: setup   (临时驻留,后被覆盖)
  0x100000 - 0x17FFFF: 内核代码的最终位置(通过分页映射到0x0)

build.c 是tools/目录下一个不起眼但非常关键的工具。它的任务是将bootsect、setup和system三个二进制文件拼接成一个完整的Image，并确保每个部分的大小正确。如果bootsect超过512字节（实际上必须恰好是512字节，最后两个字节是魔数0x55AA），build会报错终止。

注意一个重要的细节：bootsect和setup的代码是在实模式下执行的，而system中的head.s是保护模式的入口。这意味着build工具必须精确控制各段的起始位置——这个工作是在编译时通过链接脚本（ld script）来保证的。

关键概念：build.c其实是一个非常简单但极其重要的工具。它只有大约150行C代码。它的存在告诉我们一个道理——早期的Linux开发中，很多现在看来应该由复杂的构建系统做的事情，都是用最简单的C程序加shell脚本来完成的。不要被"构建工具"四个字吓到，它就是读几个文件、写一个文件、检查一下大小而已。

Makefile体系方面，Linux 0.11有一个顶层Makefile和每个子目录的Makefile。顶层的Makefile定义了编译规则、LDFLAGS（链接参数）、以及如何调用build工具。每个子目录的Makefile则负责将自己目录下的源文件编译成.o文件，然后通过make -C subdir被递归调用。

启动流程概览

有了目录结构和编译产物的基本认知，我们现在可以从宏观上俯瞰内核的完整启动流程了。从按下电源键到你在屏幕上看到#提示符，中间经历了这些步骤：

+-------------+     +--------------+     +------------+
|  按下电源键  | --> | BIOS自检     | --> | BIOS读取   |
|            |     | (POST)       |     | MBR到0x7C00 |
+-------------+     +--------------+     +------------+
                                                |
                                                v
+-------------+     +--------------+     +------------+
| bootsect.S  | --> | setup.S      | --> | head.S     |
| 搬移自身     |     | 获取内存信息  |     | 设置GDT/IDT |
| 加载setup   |     | 关中断       |     | 开启A20门   |
| 调用setup   |     | 进入保护模式  |     | 设置分页    |
+-------------+     +--------------+     +------------+
                                                |
                                                v
+-------------+     +--------------+     +------------+
| main.c      | --> | mem_init()   | --> | trap_init()|
| 各种初始化   |     | 内存管理     |     | 中断设置    |
+-------------+     +--------------+     +------------+
        |
        v
+-------------+     +--------------+     +------------+
| sched_init()| --> | move_to_user | --> | init()     |
| 任务0的TSS  |     | _mode()进    |     | fork(1)    |
|            |     | 入用户态     |     | exec sh    |
+-------------+     +--------------+     +------------+
                                                |
                                                v
                                        +----------------+
                                        | shell 等待输入  |
                                        | # _            |
                                        +----------------+

在这张图上，我标出了每个阶段的关键任务。注意，从bootsect到head.S都是用汇编写的，到了main.c才开始用C语言。这背后的原因是：在保护模式和分页机制正式启用之前，C语言的运行时环境还没有准备好——栈指针没有设置、GDT没有加载、和全局变量的位置还没有确定。在汇编阶段完成这些基础设置之后，内核才能安全地跳转到C语言世界。

读源码的方法论

在开始逐行分析之前，我想分享一套经过验证的"内核源码阅读方法论"。这套方法是我自己多年读内核源码的经验总结，也融合了Linus本人和很多内核开发者的学习心得。

第一，从main()开始。阅读任何大型软件系统时，都不要从头（文件列表的第一个文件）开始读。应该从入口函数开始。对Linux 0.11来说，入口就是init/main.c中的main()函数。main()的顺序就是系统的初始化顺序——这个顺序本身就是一份文档。从main()出发，每次看到一个函数调用（比如mem_init()），就跳转到这个函数的定义处阅读，形成调用链。

第二，跟踪调用链。我建议准备一个文本文件或笔记，记录每个函数的调用关系。比如：

main()
  +-- mem_init()         // 内存管理初始化
  |     +-- PAGE_ALIGN() // 计算对齐后的内存地址
  |     +-- memset()     // 初始化mem_map数组
  +-- trap_init()        // 中断向量表初始化
  |     +-- set_trap_gate()  // 设置中断门
  |     +-- set_system_gate() // 设置系统调用门
  +-- sched_init()       // 调度器初始化
  |     +-- set_tss()    // 设置任务0的TSS
  |     +-- set_ldt()    // 设置任务0的LDT
  +-- ...

第三，关注数据结构。内核源码的本质就是数据结构 + 对这些数据结构的操作。一个数据结构的定义（在include/linux/sched.h、include/linux/fs.h等头文件中）比代码实现更值得花时间精读。比如说，理解了task_struct（在0.11中其实叫struct task_struct，即进程控制块PCB）就理解了进程管理的80%。数据结构是内核设计的骨架，代码只是骨架上的血肉。

第四，画图辅助。不要只在脑子里想象，一定要动手画图。我是认真的——用纸笔、画图软件、甚至是ASCII图都行。当你在追踪一段复杂的内存映射或链表操作时，一张图比一千行注释都有用。Linus本人说他在写内核时经常在餐巾纸上画图——"在画出来之前，你并不真正理解它。"

具体操作建议：在您读代码的电脑上同时开两个窗口——一个放源码，一个放笔记（比如Markdown或者简单的txt文件）。在笔记里画ASCII架构图、记录调用链、列出关键数据结构字段。不要高估你的记忆力——内核源码里有太多相互关联的细节。

第五，不要试图一次理解所有细节。内核代码中有大量边界检查、错误处理、硬件兼容性处理。初次阅读时，先关注正常路径（happy path），忽略所有的边界情况和错误分支。比如在读sys_read()时，先关注"文件存在、权限正确、缓存命中"这条最顺畅的路径，以后再回来看那些if (error)分支。

第六，运行它。静态阅读代码只能理解50%。在Bochs或QEMU中运行内核，设置断点，观察变量的变化，才能真正理解代码的动态行为。下一章我们将搭建完整的实验环境。

以上就是整门课程的地图和学习方法。现在，我们的第一站是搭建开发环境——没有Bochs，你就看不到内核在"跑步"时的样子。准备好了吗？让我们动手装一套能运行Linux 0.11的开发环境。
