第9章 线程与协程

大家好，我是你们的讲师。上一章我们讲了进程——操作系统中资源分配的基本单位。但如果每次想并发执行多个任务都创建一个新进程，你会发现开销太大了：每个进程有自己的地址空间、文件描述符表、信号处理函数……光是复制这些数据结构就够CPU忙活一阵子。更关键的是，多个进程之间共享数据非常麻烦——必须用管道、消息队列、共享内存这些IPC机制，又慢又复杂。

那有没有一种更轻量的并发单位？有的，这就是线程（Thread）。线程和进程共享地址空间，但各自有独立的程序计数器、栈和寄存器上下文。切换线程比切换进程快得多。而协程（Coroutine）则更进一步，把调度权从内核态拿到了用户态，实现了微秒级别的切换开销。这章我们沿着"越来越轻"的路线，把这三者的区别和联系讲透。

先讲一个我遇到的性能案例。几年前我优化一个游戏服务器的网络模块，当时每个连接用一个pthread处理。服务器连接数从1000涨到5000时，延迟突然飙升了10倍。用perf分析后发现，超过一半的CPU时间花在了内核态的线程切换上——频繁的上下文切换导致cache疯狂miss。后来改成了基于epoll的事件驱动+有限线程池，同样处理5000连接，CPU占用反而下降了70%。这个教训让我深刻理解了"线程不是免费的"。

线程：CPU调度的最小单位

教科书上的经典定义：进程是资源分配的基本单位，线程是CPU调度的基本单位。这意味着什么？

一个进程可以包含多个线程，这些线程共享进程的地址空间（代码段、数据段、堆），但每个线程有自己的栈、寄存器和程序计数器。切换线程时，因为地址空间不变（页表不用换），TLB不用刷新，cache热数据也还在——所以线程切换的开销远小于进程切换。

对比维度进程线程
地址空间独立地址空间共享进程地址空间
资源开销大（独立PCB、页表、文件表）小（只分配栈和寄存器上下文）
切换开销高（TLB刷新、cache失效）低（无需刷新地址空间）
数据共享需要IPC（共享内存、管道等）直接读写全局变量
隔离性强（一个进程挂了不影响其他）弱（一个线程crash可能导致整个进程崩溃）
创建速度慢（需要复制/COW页表）快（共享页表）

看到"隔离性"这一行——线程因为共享地址空间，一个线程的非法内存访问（比如写野指针）可能直接破坏另一个线程的堆或栈。这也是为什么多线程编程远比多进程编程更需要小心。

关键认知：从内核的视角来看，它根本不知道"线程"这个概念。内核眼里只有"可调度的实体"——其实就是task_struct。但在Linux中，线程和进程的本质区别在于：多个线程之间共享了哪些CLONE_*标志。这一点我们马上细讲。

Linux中线程就是轻量级进程

在Linux的哲学里，没有"线程"和"进程"的严格区分——一切都是"任务"（task）。创建所谓的"线程"时，用的系统调用和创建进程一样都是 clone()，只不过传入了不同的标志位，决定了父子之间共享哪些资源。

这就引出了Linux线程模型最特殊的"艺术"——clone()的flags参数。

clone()的flags艺术

Linux内核中，创建进程和线程都通过 clone() 系统调用实现。区别只在于flags参数：

// 创建进程：fork() 实际上调用 clone(SIGCHLD, 0)
clone(SIGCHLD, 0);

// 创建线程：pthread_create() 内部调用
// clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND |
//       CLONE_THREAD | CLONE_SETTLS | CLONE_PARENT_SETTID |
//       CLONE_CHILD_CLEARTID | CLONE_SYSVSEM, ...)

关键的flags含义如下：

Flag含义共享了什么
CLONE_VM共享地址空间父子进程使用同一个mm_struct（页表）
CLONE_FS共享文件系统信息当前目录、umask等
CLONE_FILES共享文件描述符表打开的文件、socket等
CLONE_SIGHAND共享信号处理函数signal_struct
CLONE_THREAD成为同一线程组tgid相同（getpid()返回值相同）

看到没有？pthread_create()创建线程时传入了 CLONE_VM，表示"我们共享地址空间"——这就是线程和进程最根本的区别。如果你创建一个子进程时不传递 CLONE_VM，那么它就是普通进程——有自己的地址空间。

底层视角：在shell里执行 ps -eLf，你会看到每个线程都有一条记录，但同一进程的线程有相同的PID（tgid）和不同的LWP（Lightweight Process ID）。从内核角度看，每个LWP就是一个独立的task_struct——差别在于它们之间的task_struct共享了哪些资源指针。

1:1 vs M:N 线程模型

我们刚才说了，Linux内核里每个线程（轻量级进程）都是一个独立的调度实体（task_struct）。这意味着用户态创建的一个pthread，在内核中就对应一个可被独立调度的实体——这就是1:1线程模型。

但这不是唯一的实现方式。事实上，业界有三种主流的线程模型，各有取舍。

多对一（M:1）线程模型

多个用户态线程映射到一个内核态线程。线程的创建、切换全在用户态完成，不需要系统调用，速度快。但一个线程阻塞（如I/O）会导致整个进程阻塞，因为内核不知道用户态还有别的线程可以运行。这种模型在纯用户态线程库中比较常见（如早期的GNU Portable Threads），现在已经很少用了。

一对一（1:1）线程模型

每个用户线程对应一个内核线程。这是Linux上pthread的实现方式。优点是充分利用多核，一个线程阻塞不影响其他线程。缺点是每次线程操作（创建、切换）都需要系统调用，开销较大。Linux的NPTL（Native POSIX Threads Library）就是1:1模型的代表作。

多对多（M:N）线程模型

M个用户态线程映射到N个内核线程。结合了前两者的优点——用户态线程轻量切换、内核态线程充分利用多核。但实现复杂度极高，需要精巧的用户态调度器。既要在用户态管理M个线程的运行队列，又要处理内核线程的阻塞同步。

特性M:1（用户级）1:1（内核级）M:N（混合）
创建开销极低（用户态）高（系统调用）中等
切换开销低高（陷入内核）中低
多核利用不能能能
一个线程阻塞全进程阻塞仅该线程阻塞仅该内核线程阻塞
代表GNU Pth（已废弃）Linux NPTL (pthread)Go Goroutine

Go Goroutine的M:N调度

Go语言的Goroutine是M:N模型最成功的工业实践。Go运行时维护一个M（Machine，内核线程）对N（Goroutine，用户态协程）的映射，用G（Goroutine）、M（Machine）、P（Processor）三个结构体实现了高效的M:N调度。Go的调度器在程序主动让出（channel操作、time.Sleep、系统调用）时自动切换Goroutine，避免了内核态切换的开销。

性能对比：创建一个pthread大约需要微秒级别的时间和数KB的栈空间（可设置最小栈）。而创建一个Goroutine的开销仅为约4KB的栈和不到百纳秒的创建时间。这也是为什么Go可以轻松启动数十万个Goroutine，而同等数量的pthread会导致系统资源耗尽。

协程的本质

讲完了线程，我们来聊聊协程（Coroutine）。其实协程的思路比线程更古老——它最早可以追溯到1958年Melvin Conway的论文。协程的核心思想就是：让函数可以暂停执行，在某个时刻恢复执行。

线程是抢占式调度的——内核在任意时刻都可以打断一个线程的执行，切换到另一个线程。协程是协作式调度的——协程自己决定何时让出执行权（yield），操作系统没有能力在任意时刻打断一个协程。

有栈协程 vs 无栈协程

协程的两种实现方式，是很多初学者容易混淆的。我们直接对比：

维度有栈协程（Stackful）无栈协程（Stackless）
实现方式拥有独立的调用栈复用调用者的栈（状态用对象保存）
可嵌套调用可以（在子函数中yield）不可以（只能在顶层函数中yield）
内存开销较大（需要预分配栈空间）极小（几字节的状态对象）
语言代表Go Goroutine、C++ Boost.CoroutinePython async/await、Rust async、C++20 coroutine

Go的Goroutine是有栈协程。每个Goroutine有自己的栈（初始4KB，动态增长），可以在任意嵌套函数中调用 runtime.Gosched() 让出。Python的async/await是无栈协程——它把协程的"暂停点"用状态机表示，不维护独立的调用栈。

async/await的底层原理

无栈协程背后的实现机制其实是一个编译器驱动的状态机变换。以Python为例：

async def fetch_data():
    # 当执行到这里时，控制权返回到事件循环
    data = await http_request(url_1)
    # 当http_request完成时，从这里继续执行
    result = await http_request(url_2)
    return result

# 编译器实际上把上面的函数变换成类似这样的状态机：
class fetch_data__coroutine:
    def __init__(self):
        self.__state = 0

    def __next__(self):
        if self.__state == 0:
            self.__state = 1
            return http_request(url_1)   # await点1
        elif self.__state == 1:
            self.__state = 2
            self.__data = 
            return http_request(url_2)   # await点2
        elif self.__state == 2:
            self.__result = ...          # 最终结果
            raise StopIteration

每次await都是一个"暂停点"，协程把当前状态保存到一个状态对象中，返回控制权给事件循环。事件循环（如Python的asyncio或Node.js的libuv）负责监控所有等待的I/O事件，一旦某个事件就绪，就唤醒对应的协程从暂停点继续执行。整个过程完全在用户态完成，不需要内核参与。

本质洞察：协程说到底是把"控制流的切换"从内核态拿到了用户态。它利用编译器（状态机变换）或运行时（独立栈+寄存器保存）来实现函数执行的中断和恢复。对于I/O密集型的应用（比如Web服务器），协程可以轻松管理数十万并发连接——这是线程模型难以企及的。

回顾一下这条"轻量化"的演进路线：进程→线程→协程。进程提供最完善的隔离但切换最重，线程共享地址空间减轻了切换代价，协程则彻底把调度放到了用户态实现了微秒级切换。没有哪种方案是压倒性最优的——隔离、开销、复杂度之间的权衡，取决于你的具体应用场景。
