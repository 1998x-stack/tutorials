第21章：主从复制（下）—— 高级场景

大家好，我是你们的讲师。上一章我们吃透了全量同步和 PSYNC 部分同步这两个核心机制。但真正把主从复制放到生产环境中时，你会遇到更多"基础之外"的问题：全量同步时磁盘 IO 打满怎么办？几十个从库同时连接主库，带宽扛不住怎么办？从库重启后 PSYNC 还是失败了怎么办？

本章我们继续深入 replication.c 的剩余部分，覆盖无盘复制、级联复制拓扑、数据一致性保障、安全机制等高级话题。这些都是生产环境架构师必须掌握的硬知识。

无盘复制

上一章提到的全量同步中，主库需要执行一次 BGSAVE 将数据库写入 RDB 文件，再从磁盘读取这个文件发送给从库。这意味着一轮全量同步涉及写一次 + 读一次的磁盘 IO。如果从库数量多，磁盘就成了瓶颈。

无盘复制（Diskless Sync）的出现就是为了绕开这个问题。当 repl-diskless-sync yes 开启时，主库的 RDB 子进程直接将数据通过 socket 写入从库的 TCP 连接，不经过磁盘。我们来仔细看看这个流程：

/* replication.c - 无盘复制的启动逻辑 */
void startBgsaveForReplication(int mincapa) {
    int socket_target = server.repl_diskless_sync && (mincapa & SLAVE_CAPA_EOF);

    if (socket_target) {
        /* 无盘模式：不写文件，直接通过 socket 发送 RDB */
        serverLog(LL_NOTICE, "开始无盘复制 BGSAVE");
        if (rdbSaveToSlavesSockets(
                rdbflags, &slaves, &best_error) != C_OK)
        {
            /* 失败处理 */
        }
    } else {
        /* 磁盘模式：写入 RDB 文件 */
        if (rdbSaveBackground(server.rdb_filename, rdbflags) != C_OK) {
            /* 失败处理 */
        }
    }
}

无盘复制的数据传输过程在 rdbSaveToSlavesSockets() 函数中实现。它同样是 fork 子进程，但子进程直接将 RDB 格式的数据逐块（chunk）写入从库的 socket 描述符：

/* rdb.c - rdbSaveToSlavesSockets 核心流程（简化） */
int rdbSaveToSlavesSockets(int rdbflags, list *slaves, int *error) {
    /* 1. fork 子进程 */
    if ((childpid = fork()) == 0) {
        /* 子进程 */

        /* 2. 关闭不需要的 fd */
        closeListeningSockets(0);

        /* 3. 为每个从库创建专用的 socket 连接 */
        /* 子进程持有所有从库连接的 fd 副本 */

        /* 4. 遍历数据库，序列化写入 socket */
        if (rdbSaveRioWithEOFMark(&rdb, &slavecfds, numfds) == C_OK) {
            /* 数据发送完毕 */
            exit(0);
        }
    }

    /* 5. 父进程继续正常服务 */
    /* 子进程完成后父进程收到 SIGCHLD 信号 */
    /* 在 backgroundSaveDoneHandler() 中进行收尾处理 */
}

从库的 readSyncBulkPayload() 也需要相应支持 EOF marker 协议（无盘传输时 stream 结束的标记方式）。主库在数据末尾添加一个特定的 EOF 标记，从库读到标记就知道接收完毕了。

注意 repl-diskless-sync-delay 配置（默认 5 秒）——在无盘模式下，主库不会立即为第一个连接的从库启动 BGSAVE，而是等待这个延迟时间，让更多从库加入。这样一次 BGSAVE 就能服务多个从库，避免了多次 fork 子进程的开销。

选择建议：无盘复制适合磁盘 IO 是瓶颈的场景（比如 HDD 磁盘或 IOPS 受限的云盘）。但如果网络带宽才是瓶颈，无盘模式反而会加剧问题——因为它不再依赖磁盘的"缓冲"作用，数据直接从内存到网络，带宽消耗更集中。在 SSD 实例上，磁盘模式可能更稳妥。

级联复制拓扑

想象一个场景：你的主库有 30 个从库，每个从库全量同步时都需要主库 fork 子进程并发送数据。30 次全量同步的带宽消耗足以打垮任何网卡。这时候就需要级联复制（Cascading Replication / Chain Replication）。

级联复制的核心思路是：从库也可以有自己的从库。主库只服务有限的几个"一级从库"，这些一级从库再去服务更多的"二级从库"。这样就形成了复制链（replication chain）：

Master
 ├── Replica A  (replica-priority = 100)
 │    └── Replica A1 (通过 REPLICAOF Replica_A_IP Port 设置)
 │    └── Replica A2
 └── Replica B  (replica-priority = 200)
      └── Replica B1
      └── Replica B2

配置级联复制的方式很简单——让一个从库执行 REPLICAOF <上级从库IP> <端口> 即可。对 Redis 来说，这个"上级从库"同样扮演主库的角色，它的复制逻辑完全相同。

级联复制中有一个重要的控制参数——replica-priority（旧称 slave-priority）。它在 Sentinel 故障转移时用于决定谁应该成为新主库：优先级数字越小，权重越高（100 优于 200）。如果你的级联拓扑中某个一级从库性能更强、位置更优，就给它设更小的 priority。

拓扑设计：级联复制能减轻主库的负担，但也有代价——数据链路的每一级都会增加复制延迟。如果你的业务对一致性要求很高（比如要求读取自己写入的数据），建议复制链不超过 2 层（主 -> 一级从 -> 二级从）。对于读写分离架构，前端应用应该连接一级从库而非二级从库。

复制过程中的数据一致性

前面提到，全量同步期间主库不会停止写入。那么问题来了：从库在加载 RDB 文件期间，主库一直在生产新的写入命令，这部分数据怎么同步？

答案还是 replication backlog。全量同步期间，所有写入命令除了正常执行，还会被复制传播并写入 backlog：

/* replication.c - feedReplicationBuffer 命令传播写入 */
void feedReplicationBuffer(char *s, size_t len) {
    /* 写入 backlog 环形缓冲区 */
    while (len > 0) {
        size_t avail = server.repl_backlog_size - server.repl_backlog_histlen;
        size_t towrite = len > avail ? avail : len;

        /* 检查是否覆盖了未消费的数据 */
        if (avail < len) {
            /* 要覆盖了！将最早的数据丢弃 */
            server.repl_backlog_histlen -= ...;
        }

        /* 数据写入环形缓冲区 */
        memcpy(server.repl_backlog->buf + server.repl_backlog_idx, s, towrite);
        server.repl_backlog_idx = (server.repl_backlog_idx + towrite) % server.repl_backlog_size;
        server.repl_backlog_histlen += towrite;
        server.master_repl_offset += towrite;

        s += towrite;
        len -= towrite;
    }
}

从库加载完 RDB 后，会立即向主库发送 REPLCONF ACK <offset>，然后主库根据这个 offset 从 backlog 中取出从库尚未收到的命令，追加发送。

这个机制保证了从库最终能与主库保持一致——即使全量同步持续了 10 分钟，这 10 分钟内主库的所有写入也不会丢失。从库跟上主库之后，就会进入"稳态复制"模式，实时接收主库传播的命令。

复制安全机制

生产环境中，主从之间的网络不可能永远稳定。Redis 提供了多层安全机制来保障复制的可靠性：

心跳保活

主库会定期向从库发送 REPLCONF ACK 请求，这既是心跳检测，也是延迟测量。相关参数：

  • repl-ping-replica-period（默认 10 秒）：主库向从库发送 PING 的间隔。
  • repl-timeout（默认 60 秒）：如果在这段时间内没有收到从库的响应，主库判定从库已断开，清理连接资源。

写入安全阀

min-replicas-to-write 和 min-replicas-max-lag 是一对配合使用的参数，用于控制主库在从库不足时拒绝写入：

/* replication.c - 检查最小从库数量 */
int replicationCheckMinReplicas(void) {
    /* 如果配置了 min-replicas-to-write */
    if (server.repl_min_replicas_to_write &&
        server.repl_min_replicas_max_lag)
    {
        int replicas = 0;
        listIter li;
        listNode *ln;
        listRewind(server.slaves, &li);
        while ((ln = listNext(&li)) != NULL) {
            client *slave = ln->value;
            /* 计算从库的延迟是否在 max_lag 范围内 */
            if (slave->repl_ack_time + server.repl_min_replicas_max_lag >= server.unixtime)
                replicas++;
        }
        if (replicas < server.repl_min_replicas_to_write) {
            /* 可用从库不足，拒绝写入 */
            server.repl_min_replicas_avail = 0;
            return 0;
        }
    }
    server.repl_min_replicas_avail = 1;
    return 1;
}

当启用了 min-replicas-to-write，如果当前延迟合格的从库数量不足，主库会返回 "No writeable replicas" 错误给客户端。这是应对"主库网断了但从库还没感知"场景的一种保护——避免了主库收不到确认就盲目写入导致的数据差异。

踩坑实录：某公司运维工程师在维护 Redis 集群时，为了防止主库在从库全部掉线时继续写入导致数据丢失，设置了 min-replicas-to-write=3 和 min-replicas-max-lag=10。听起来很合理对吧？但问题是他的架构只有 3 个从库。某次短暂的网络抖动导致其中一个从库的延迟超标——从库数量降到 2，不满 3 个——主库立即拒绝所有写入请求，整个服务写入瘫痪了 30 秒，直到从库重新跟上。运维同学复盘时说："我以为我在保护数据安全，其实我在生产环境关上了写入的大门。"这个教训告诉我们：min-replicas-to-write 一定要预留余量。如果你有 3 个从库，建议设成 min-replicas-to-write=2，容忍 1 个从库故障。高可用性和数据安全性之间需要谨慎平衡。

从库只读与命令传播

从库默认是只读的，这个行为由 replica-read-only yes 控制。但有同学可能会好奇：如果我在从库上执行 SET key value 会怎样？

/* server.c - processCommand 中的权限检查 */
int processCommand(client *c) {
    /* ... */

    /* 如果是从库，且不是只读命令，且开启了只读模式，拒绝 */
    if (server.masterhost && server.repl_slave_ro
        && !(c->flags & CLIENT_MASTER)
        && c->cmd->flags & CMD_WRITE)
    {
        addReplyError(c, "READONLY You can't write against a read only replica.");
        return C_OK;
    }

    /* ... */
}

从库不接受写入不是"功能限制"而是"策略限制"——即使你在从库上成功写入了一份数据，主库下一次全量同步或这条数据所属 key 的过期删除也会把它抹掉。从库的设计初衷就是"主库数据的完整副本"。

命令传播的核心函数是 propagate() 和 feedReplicationBuffer()。前者负责将写命令同步写入 AOF 和从库缓冲区，后者负责将其实际写入 backlog 和每个从库的输出缓存：

/* replication.c - propagate 函数 */
void propagate(int dbid, robj **argv, int argc, int target) {
    /* target 可以是 PROPAGATE_AOF、PROPAGATE_REPL 或两者 */
    if (server.aof_state != AOF_OFF && target & PROPAGATE_AOF)
        feedAppendOnlyFile(dbid, argv, argc);   /* 写入 AOF 缓冲区 */
    if (target & PROPAGATE_REPL)
        replicationFeedSlaves(server.slaves, dbid, argv, argc);  /* 传播给从库 */
}

/* 进一步调用 feedReplicationBuffer 将命令写入 backlog */
void replicationFeedSlaves(list *slaves, int dictid, robj **argv, int argc) {
    /* 将命令序列化为 Redis 协议格式 */
    char buf[256];
    int buflen = redisFormatCommand(buf, sizeof(buf), argv, argc);

    /* 写入 backlog */
    feedReplicationBuffer(buf, buflen);

    /* 写入每个从库的输出缓冲区 */
    listRewind(slaves, &li);
    while ((ln = listNext(&li)) != NULL) {
        client *slave = ln->value;
        if (slave->replstate == SLAVE_STATE_ONLINE)
            addReply(slave, buf, buflen); /* 添加到从库的输出缓存 */
    }
}

注意 addReply() 只是把数据写入从库连接的输出缓冲区，真正的网络传输由事件循环中的 sendReplyToClient() 完成——这也是 Redis 单线程事件驱动模型的特点：所有 IO 都是非阻塞的。

主从复制解决了数据冗余，但故障切换还依赖人工操作吗？如果主库挂了，难道要运维人员半夜爬起来手动执行 REPLICAOF NO ONE？显然不能。下一章我们迎来 Redis 高可用的两座大山——Sentinel 哨兵和 Cluster 集群，它们将实现真正的自动故障转移。我们下章见。
