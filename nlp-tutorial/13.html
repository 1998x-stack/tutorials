第十三章：GPT系列

2020年5月，OpenAI完成了GPT-3的训练，但迟迟没有公开发布论文和模型。坊间流传的原因是——模型"太危险了，不能放出来"。当时我所在的几个技术群里炸开了锅：有人说这是OpenAI的营销噱头，有人说GPT-3生成的假新闻足以操纵舆论，还有人担心它会取代大量文字工作者的岗位。

最终，OpenAI确实以"担心恶意使用"为由，没有开源GPT-3的完整模型，只通过API提供有限访问。这件事在NLP社区引发了巨大争议——"不开放"还是不是"Open"AI？但无论争议如何，GPT-3的能力确实让整个AI界震惊：它不需要微调，给它几个例子它就能"学会"做新任务。这种能力被称为 In-Context Learning（上下文学习），它彻底改变了我们对"训练"的认知。

但GPT不是一天建成的。从2018年的GPT-1（117M参数）到2020年的GPT-3（175B参数），OpenAI花了三年时间，走了三步关键的棋。每一步都在告诉我们同一个道理：规模放大了一切。这一章，我们就来拆解GPT系列的三代进化。

13.1 自回归生成：从左到右的哲学

在深入GPT各版本之前，我们需要先理解GPT系列最核心的底层哲学——自回归生成（Autoregressive Generation）。

所谓自回归，就是模型在生成文本时，一个词一个词地从左往右生成。每一步只看到当前词左边的所有词，然后预测下一个词是什么。用公式表达就是：

# 自回归语言模型的目标：最大化整个序列的似然
# P(w_1, w_2, ..., w_n) = P(w_1) * P(w_2|w_1) * P(w_3|w_1,w_2) * ... * P(w_n|w_1,...,w_{n-1})

# 在Transformer中的实现：
# 使用causal mask（因果掩码 / 三角掩码）
# 每个位置只能attend到自己及左侧的位置
# 右侧的信息被"遮住"了

# 例如，"I love NLP" 生成过程：
# Step 1: [BOS]         → 预测 P(I|BOS)
# Step 2: [BOS, I]      → 预测 P(love|BOS,I)
# Step 3: [BOS, I, love]→ 预测 P(NLP|BOS,I,love)
# 每一步只能用已生成的词预测下一个词

这种"从左到右"的生成方式有个关键特性：模型天然适合生成任务。写作文、翻译、对话——这些都是从左到右逐词产生的过程。但代价是，模型看不到右边的上下文——它不能像BERT那样利用双向信息来做理解。

核心洞察：自回归生成 vs 双向编码，这不仅仅是技术路线的选择，更是"生成"和"理解"两种任务的本质分歧。GPT选择生成，BERT选择理解。后来我们发现，足够大的生成模型也能做好理解任务——这是GPT-3教会我们的最重要一课。

你可能会问：为什么不用双向信息？你看后面章节的BERT，它用MLM（掩码语言模型）同时看左右两侧的信息，理解能力确实更强。但BERT的架构天生不适合"生成"——当你需要从头写一段文字时，你不能先看完全文再写第一个字。这就是GPT坚持自回归的根本原因。

13.2 GPT-1：首次生成式预训练

2018年6月，OpenAI发表了GPT-1论文《Improving Language Understanding by Generative Pre-Training》。这篇论文只有12页，标题直白地说了一件事：用生成式的预训练来提升语言理解。这个名字本身就很反直觉——"生成式的预训练怎么反而用来做理解？"

GPT-1的架构是Transformer的解码器部分（12层，每层12个注意力头，隐藏维度768，共117M参数）。它做的事情其实很简单：在BooksCorpus（约7000本未出版书籍，800M词）上做标准的自回归语言模型预训练。

预训练之后，针对具体任务（如文本分类、自然语言推理等），只需要在模型上加一个线性分类层做微调。与当时流行的ELMo（需要训练整个任务网络）相比，GPT-1的迁移效率高得多——你只需要调最后几层。

模型参数量预训练数据架构关键发现
GPT-1117MBooksCorpus (~800M tokens)12层Transformer解码器生成式预训练也能提升理解任务；零样本迁移初见端倪
GPT-21.5BWebText (~8M网页, 40GB)48层Transformer解码器零样本能力涌现；长文本生成质量大幅提升
GPT-3175BCommon Crawl+WebText2+Books (~570GB)96层Transformer解码器In-Context Learning；Scaling Laws成立

GPT-1在当时的效果怎么样？在12个NLP任务中的9个达到了SOTA（包括文本分类、自然语言推理、问答等）。虽然很快被BERT反超，但GPT-1证明了一个极其重要的观点：生成式预训练学到的语言知识，完全可以迁移到理解任务上。这个"生成→理解"的桥梁，在GPT-1之前没有人成功搭建过。

13.3 GPT-2："太危险"和零样本涌现

2019年2月，OpenAI发布了GPT-2。这一次，他们做了三件大事：第一，把模型从117M放大到1.5B参数（13倍）；第二，从干净的书籍数据扩展到WebText——从Reddit上收集了高赞链接对应的800万网页；第三，他们没有完整发布。

GPT-2的发布经历堪称NLP界最戏剧性的事件之一。OpenAI先是发了一篇博客说"我们的模型太强了，担心被滥用，所以只发布小版本"。这种欲说还休的操作立刻引爆了媒体——"OpenAI发明了会写假新闻的AI不敢放出"、"AI可能被用于大规模虚假信息攻击"。大量公众号用"细思极恐"、"颠覆人类"等标题赚足眼球。

但冷静下来看，GPT-2真正重大的发现是 零样本（Zero-Shot）能力涌现——在没有经过任何微调的情况下，GPT-2在阅读理解、翻译、问答等任务上展现出了有意义的性能。论文中最令人印象深刻的例子是：给GPT-2一个英文句子"法国首都是——"，模型会自动补全"巴黎"的英文。它没有接受过任何翻译或问答训练，仅仅是"读"了800万网页就自然地学会了。

零样本能力的意义：GPT-2之前，大家都认为"预训练+微调"是标准范式——先在海量数据上预训练，再在少量标注数据上微调。GPT-2打破了这种认知：当模型足够大、数据足够多时，预训练本身就能让模型学会很多任务。参数规模从117M到1.5B，能力不是线性增长，而是发生了质变。

当然，GPT-2的零样本能力还不算可靠。翻译任务上它的BLEU分数远低于当时的SOTA，文本生成的连贯性也经常掉链子。但它确确实实展示了"规模"带来的惊喜——放大模型不只是让已有能力更好，而是催生了新能力。

13.4 GPT-3：上下文学习和规模定律

2020年5月的GPT-3，是GPT系列真正的"封神"之作。175B参数（GPT-2的117倍），训练成本据估计超过1200万美元。这已经不是"大模型"了——在那个年代，大多数研究者连单卡加载模型权重都做不到（175B参数在FP32下需要350GB显存）。

GPT-3最核心的贡献不是架构创新，而是发现了 In-Context Learning（上下文学习）。简单来说，你不需要给GPT-3做任何微调或参数更新。只需要在输入中给它几个例子，它就能"照猫画虎"：

# In-Context Learning 示例
# 用户输入（包含几个示例）：
# 中文: "今天天气真好" → 英文: "The weather is great today"
# 中文: "我在学习人工智能" → 英文: "I am studying artificial intelligence"
# 中文: "这本书很有趣" → 英文:

# 模型输出：
# "This book is very interesting"

# 没有梯度更新，没有微调，模型仅仅通过"看"前面的例子
# 就学会了"翻译"这个任务——这是GPT-3最令人震惊的能力

# Few-shot: 提供10-100个示例（不需要梯度更新）
# One-shot: 提供1个示例
# Zero-shot: 不提供任何示例，直接给指令

In-Context Learning意味着什么？意味着整个"预训练-微调"范式被颠覆了。你不再需要收集标注数据、不再需要做反向传播微调、不再需要保存多个任务模型。只需要一个巨大的通用模型，给它几个例子就能做任何任务。这就是"通用人工智能雏形"这个说法的来源。

配合GPT-3，DeepMind在同期发表了 Scaling Laws（规模定律） 论文。他们发现模型的性能与三个因素之间存在平滑的幂律关系：参数量、数据量和计算量。模型越大、数据越多、算力越强，性能越好——目前为止还没看到天花板。

⚠️ 踩坑实录：Scale的神话与误解

2020年GPT-3刚出的时候，我兴冲冲地写了个自动代码修复工具。心想GPT-3这么强大，把bug代码扔进去它就能修好。结果测试下来，它确实能"看懂"代码，但修好的代码经常引入新的bug。问题在于——In-Context Learning不等于真正的理解。GPT-3很擅长模拟"看起来正确"的输出，但它并不真正理解代码的语义。

后来复盘发现，我的踩坑经历是典型的"Scale崇拜"——以为模型够大就能解决一切。实际上，GPT-3在需要精确推理、多步规划的任务上表现很不稳定，而且它对Prompt的措辞极其敏感，稍微改几个字输出就完全不同。这告诉我们：Scale很重要，但不是万能药。即使到今天，大模型在数学推理、事实一致性等方面依然有根本性缺陷。

GPT-3还催生了 Prompt Engineering（提示工程） 这个全新的研究领域。研究者们发现，同样的任务换一个说法，模型的表现可能天差地别。比如"把这句中文翻译成英文"和"Translate this Chinese sentence to English"得到的翻译质量可能完全不同。甚至有人发现，在Prompt末尾加一句"让我们一步步思考"（Let's think step by step）能显著提升推理任务的准确率——这就是后来的Chain-of-Thought推理。

13.5 GPT vs BERT：分叉路的终点

2018-2020这两年间，NLP界最大的阵营就是GPT和BERT。它们共享同一个基石（Transformer），但走上了完全不同的道路。理解这种分叉，能帮你从根本上把握整个NLP的发展脉络。

对比维度GPT（Decoder-Only）BERT（Encoder-Only）
预训练目标自回归（预测下一个词）掩码语言模型（预测被遮住的词）
注意力范围单向（只能看左边）双向（可以看左右两边）
强项任务生成（写文章、翻译、对话）理解（分类、NER、推理）
微调方式Few-shot / In-Context Learning预训练 + 任务特化微调
规模趋势一路变大（175B → 万亿级）适中规模（一般不超过 340M）
推理效率逐词生成，较慢单次前向，较快
代表性模型GPT-1/2/3, GPT-4, LLaMABERT, RoBERTa, ALBERT, DistilBERT

从这张表可以看得很清楚：GPT和BERT不是谁好谁坏的关系，而是"擅长的事情不一样"。如果你要做文本分类、命名实体识别、情感分析——BERT类模型在小样本下往往表现更好、效率更高。但如果你要写文章、对话、翻译——GPT类模型是唯一的选择。

有趣的是，随着GPT模型越做越大，它作为Decoder竟然也盖过了Encoder的理解能力。这就是Scaling Laws的神奇之处——当参数量大到一定程度，模型不再受限于架构缺陷，而能通过海量知识来弥补。GPT-3在分类任务上已经逼近BERT的效果，而GPT-4更是全面超越。但这是靠"量变引起质变"，不是架构设计上的胜利。

换个角度说：中小规模（<1B参数）下，BERT是更理性的选择。直到今天，很多工业界的文本分类系统依然用BERT或它的变体——因为更小、更快、更容易部署。只有当你有充足的算力和海量数据时，Decoder-Only大模型才显示出它的优势。

13.6 动手实践：用HuggingFace跑GPT-2

理论聊了这么多，我们来上手写代码。用HuggingFace的Transformers库加载GPT-2做文本生成，代码非常简洁：

# 安装：pip install transformers torch

from transformers import GPT2LMHeadModel, GPT2Tokenizer

# 加载GPT-2模型和分词器
# 默认是GPT-2 Small（124M参数）
model_name = "gpt2"
tokenizer = GPT2Tokenizer.from_pretrained(model_name)
model = GPT2LMHeadModel.from_pretrained(model_name)

# 准备输入文本
input_text = "深度学习是人工智能的一个重要分支，它"
inputs = tokenizer(input_text, return_tensors="pt")

# 生成续写文本
# max_length: 生成的最大长度（含输入）
# temperature: 温度参数，越高生成越随机
# do_sample: 使用采样而非贪心解码
# top_k: 只从概率最高的K个词中采样
# top_p: 核采样，只从累积概率>=p的词中采样
# num_return_sequences: 生成几个候选序列
outputs = model.generate(
    inputs.input_ids,
    max_length=100,
    temperature=0.8,
    do_sample=True,
    top_k=50,
    top_p=0.95,
    pad_token_id=tokenizer.eos_token_id,
    num_return_sequences=3
)

# 解码输出
for i, output in enumerate(outputs):
    generated_text = tokenizer.decode(output, skip_special_tokens=True)
    print(f"=== 候选 {i+1} ===")
    print(generated_text)
    print()

这段代码做了三件事：加载GPT-2模型和分词器、把输入文本编码成token ID、然后用generate()方法做自回归生成。关键参数都加了注释，建议你动手调一调——把temperature调到0.1（保守生成）和1.5（疯狂生成）对比看效果差异。

关于温度（temperature）的解释：温度越低（→0），模型越倾向于选择概率最高的词，生成结果更保守更可预测；温度越高（→1.5），低概率词被采样的机会增大，生成结果更多样但也更容易脱轨。GPT-2默认的temperature=1.0是一个平衡点。

⚠️ 踩坑实录：HuggingFace GPT-2的常见陷阱

第一次用HuggingFace跑GPT-2时，我犯了两个低级错误。第一是忘了设 pad_token_id——GPT-2没有pad token，如果不显式设为eos_token_id，generate()会疯狂报警告。第二个错误是用了 max_length=100 然后以为只生成100个新词——实际上 max_length 包含输入长度。如果你输入50个字，实际只生成50个字。正确做法是用 max_new_tokens 指定生成的新词数量。

另外，不要在CPU上跑GPT-2 Large——1.5B参数的模型在CPU上生成一句话可能要几分钟。用 model.to("cuda") 移动到GPU。

现在我们来对比一下不同规模GPT-2的生成效果差异。GPT-2有四个版本，HuggingFace都提供了：gpt2（124M）、gpt2-medium（355M）、gpt2-large（774M）、gpt2-xl（1.5B）。只需改model_name即可切换。你会发现更大的模型不仅生成更流畅、内容更丰富，而且对Prompt的理解也更准确。

这种"规模越大效果越好"的趋势，正是GPT-3和Scaling Laws的理论基础。GPT-2只是放大了13倍（从117M到1.5B）就涌现了零样本能力，那么再放大117倍（从1.5B到175B）会怎么样？答案是——In-Context Learning的能力涌现了。

GPT系列的故事告诉我们一条简单而深刻的规律：当我们不知道怎么提升模型时，试着把它变大。听起来粗暴，但过去几年的发展反复验证了这一点。当然，变大不是无限的——它在算力、数据、能源等方面都有天花板。但至少在GPT-3时代，这个策略还远没有走到尽头。

GPT系列的发展还没结束。GPT-4（虽然OpenAI的文档从未明确说是"GPT-4"本身）加入了多模态能力，后续还有o1系列（推理模型）等重大迭代。但本章我们聚焦在GPT-1/2/3这三次关键跃迁——它们奠定了大语言模型的所有核心范式。

下一章，我们要进入实战环节——用前面学到的所有知识，做一个完整的文本分类项目。从数据准备、模型选型（BERT vs GPT）、训练到评估，咱们走一遍完整的流程。你会发现，把理论和实践结合起来，才是真正掌握NLP的唯一途径。
