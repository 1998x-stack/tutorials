第6章 数据生命周期管理

大家好，我是你们的讲师。前几章我们一直在讨论"怎么获得好数据"。但得到好数据之后呢？很多人会以为事情到此为止了——数据搞定了，丢进训练跑就完事了。

如果你这么想，那就忽略了一个关键的工程事实：数据不是一次性消费品。它是需要持续管理、迭代演进的资产。就像软件代码需要版本管理一样，后训练数据也需要自己的生命周期管理。

想象一下这个场景：你花了两周精心构建了一个 SFT 数据集，训练了模型，效果不错。但一个月后，你发现数据集中有 20% 的样本存在事实错误。你修改了这些样本，但：你还记得原来数据集包含哪些文件吗？你能精确找到哪些样本被修改了吗？你能确保这次修改不会破坏之前已经训练好的能力吗？如果这些问题让你觉得棘手，那就说明——你需要数据生命周期管理。

6.1 数据生命周期六阶段

一个完整的数据生命周期包含六个阶段，每个阶段都有明确的目标和交付物：

数据生命周期六阶段流程
    │
    1. 数据采集 (Collection)
    │   从开源、标注、合成等渠道获取原始数据
    │   交付物：原始数据存档 + 来源元数据
    │
    2. 数据清洗 (Cleaning)
    │   去除重复、过滤低质量、规范格式
    │   交付物：清洗规则日志 + 清洗后数据
    │
    3. 标注/生成 (Annotation / Generation)
    │   人工标注指令-回答对，或模型生成合成数据
    │   交付物：标注规范文档 + 训练样本
    │
    4. 质量验证 (Validation)
    │   自动化 + 人工抽检，确保数据符合质量门槛
    │   交付物：质量评估报告 + 评分卡
    │
    5. 版本管理 (Versioning)
    │   记录数据集的版本、变更历史、配比信息
    │   交付物：版本标签 + Data Card
    │
    6. 退役 (Retirement)
    │   识别低效或有害的数据，从训练 pipeline 中移除
    │   交付物：退役数据清单 + 移除影响评估

这六个阶段构成了一个闭环，但不是一条直线走到底——在实践中，团队会在阶段之间来回迭代。比如在质量验证阶段发现数据有问题，就退回清洗或标注阶段进行修复。

实践经验：每个阶段都应该有明确的"完成定义"（Definition of Done）。比如"清洗阶段完成"意味着所有规则都已执行、清洗日志已保存、数据量变化有记录。模糊的"差不多"在数据工程中是最大的敌人。

6.2 数据飞轮：让模型自我进化

数据生命周期管理的最高境界，是建立数据飞轮——让模型在使用的过程中不断产生新的高质量训练数据，形成一个自我强化的循环。

DeepSeek-R1 的冷启动 + RL + 拒绝采样循环

DeepSeek-R1 的成功标志着数据飞轮理念的成熟。它的训练流程大致如下：

  1. 冷启动——用少量人工标注的 CoT 数据微调基座模型，让模型学会基本的推理格式
  2. GRPO 强化学习——在数学和代码任务上运行大规模 GRPO 训练，模型自己生成海量推理轨迹
  3. 拒绝采样——用奖励模型从模型生成的轨迹中筛选出正确且清晰的高质量样本
  4. 蒸馏与合并——将筛选出的优质轨迹混合到下一轮训练数据中，反复迭代

关键洞察是：每一轮 GRPO 训练不仅提升了模型能力，还产生了更好的训练数据。这种"训练 → 生成数据 → 过滤 → 再训练"的飞轮机制，让 DeepSeek-R1 实现了自我进化。到了第 3 轮迭代之后，模型产出的推理轨迹质量已经超过了最初的冷启动人工标注数据。

UI-TARS-2 的轨迹飞轮

2025 年底发布的 UI-TARS-2 将数据飞轮的概念进一步延伸到了 GUI Agent 领域。它的轨迹飞轮机制是这样的：模型被部署到模拟环境中执行 GUI 操作任务，每次操作都会被记录为一条"轨迹"。如果任务成功完成，这条轨迹就被标记为正样本加入训练集；如果失败，则记录失败原因供后续策略改进。通过这种方式，UI-TARS-2 在数月内积累了数百万条高质量 GUI 操作轨迹，而人工参与成本几乎为零。

数据飞轮的核心前提是：你的模型已经"够好"到可以产生有用的数据。在模型能力较弱的前期，还是需要依赖人工标注或顶级模型生成来构建初始数据集。一旦飞轮启动，人工投入就可以逐渐减少。

6.3 版本控制最佳实践

数据版本管理的目标是：任何时候你都能精确复现某个模型训练所使用的数据集。以下是经过验证的最佳实践：

实践 推荐工具 说明
数据文件版本管理 DVC / Git LFS 数据集通常很大（GB ~ TB 级别），不适合直接存 Git。DVC 管理数据文件的哈希和版本，Git LFS 跟踪大文件
数据变更记录 Data Card + CHANGELOG 每个数据集版本都应该有 Data Card，记录数据来源、清洗规则、统计信息、变更说明
可复现性保障 配置文件 + Docker 数据管线的每个步骤（清洗、转换、过滤）都应该有配置文件记录参数，确保可以完全复现
实验数据溯源 实验管理平台（如 W&B、MLflow） 每次训练实验记录使用的数据集版本、数据配比、预处理参数

使用 DVC 进行数据版本管理的基本流程：

# 初始化 DVC
dvc init

# 添加数据目录到 DVC 跟踪
dvc add data/sft_v1/
git add data/sft_v1.dvc .gitignore
git commit -m "Add SFT dataset v1"

# 当数据更新时
dvc add data/sft_v2/          # 计算新版本的哈希
git add data/sft_v2.dvc
git commit -m "Update SFT dataset v2: fix factual errors in 2000 samples"

# 回溯到旧版本
git checkout <old-commit>     # 切换 Git 历史
dvc checkout                   # 同步数据到对应版本
为什么版本管理如此重要？想象一下，你发现当前模型在某些问题上表现奇怪，怀疑是训练数据有问题。如果没有版本管理，你甚至无法确认训练时用的是哪个版本的数据。有了 DVC 和 Data Card，你可以精确回退到那个版本的数据集，检查具体哪个样本可能导致了问题。这在模型迭代调试中是极其宝贵的能力。

6.4 数据血缘追踪

数据血缘（Data Lineage）追踪的是每条训练样本的"来龙去脉"：这条数据来自哪个原始来源？经过了什么清洗规则？属于哪次标注批次？被配比到了哪个训练集版本？

一个实际的数据血缘记录可能长这样：

{
  "sample_id": "sft_v3_001245",
  "origin": "source_aug_2025",
  "source_type": "gpt4_generated",
  "generation_prompt": "generate_creative_writing_v2",
  "quality_score": 4.2,
  "cleaning_rules_applied": ["dedup_md5", "length_filter", "safety_filter"],
  "human_reviewed": true,
  "reviewer_id": "annotator_087",
  "dataset_versions": ["sft_v1", "sft_v2", "sft_v3"],
  "training_runs": ["run_007", "run_012", "run_019"]
}

有了这样的血缘追踪，当你发现某个训练运行（run_019）的模型在安全评测上表现不佳时，可以快速追溯到共享该数据来源的所有训练运行，定位问题源头——也许是 source_aug_2025 批次的数据存在系统性质量缺陷。

常见误区：很多团队在初期觉得"数据血缘追踪太麻烦了，先不做"。等到数据量达到百万级别、模型迭代了十几个版本之后，再想追溯就为时已晚了——数据来源早已混杂不清，有问题根本不知道从何查起。建议从第一个数据集、第一次训练运行就开始记录数据血缘。一开始多花 10% 的时间建立追踪机制，未来能省 90% 的排查时间。

数据生命周期管理的核心，就是把数据从"消耗品"变成"资产"。数据飞轮让数据自我进化，版本控制保障可复现性，血缘追踪赋予可追溯性。这三者共同构成了数据工程的制度化基础。
