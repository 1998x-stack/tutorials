第18章 从 AutoGen 到 MAF

前 3 章我们深入学习了 AutoGen 的架构、GroupChat 和代码执行。如果你已经动手实践过，应该能感受到 AutoGen 在多 Agent 对话方面的设计确实精巧。但你可能也注意到了一件事——AutoGen 的更新速度在 2025 年下半年明显放缓了。

这不是你的错觉。2025 年 10 月，微软正式宣布将 AutoGen 置于维护模式，不再开发新功能。消息一出，社区一片哗然——毕竟 AutoGen 是 GitHub 上最受欢迎的多 Agent 框架之一，拥有超过 3.5 万 Star。但仔细想想，这个决定有其内在逻辑：微软的 Agent 战略正在全面转向 Microsoft Agent Framework (MAF)，一个与 Azure AI Foundry 深度融合的新一代框架。

战略背景： 微软的 AI 战略重心已经从"开源框架"转向"平台化服务"。MAF 不是简单的 AutoGen 升级版，而是微软将 Agent 能力整合进 Azure 云生态的战略支点——类似于 AWS 推出 Bedrock Agent 的逻辑。对开发者来说，这意味着选择 AutoGen 还是 MAF，本质是在选择"独立框架"还是"云平台集成"。

18.1 AutoGen 的终点：微软战略转向

要理解微软为什么放弃 AutoGen，需要先看清 2025-2026 年 Agent 框架市场的变化趋势。

三条战线

到 2025 年中，Agent 框架的竞争已经形成了三条清晰的战线：

战线 代表 策略
开源框架之争 LangChain/LangGraph vs AutoGen vs CrewAI 靠框架功能、易用性、生态吸引开发者
云平台绑定 AWS Bedrock Agent vs Azure MAF vs GCP Vertex AI Agent 靠云平台集成、托管服务、企业特性锁定客户
厂商原生 SDK OpenAI Agents SDK vs Anthropic MCP vs Google ADK 靠模型原生支持和最新特性保持竞争力

在开源框架这条线上，AutoGen 面临 LangChain 的激烈竞争。LangChain 凭借更早的起步时间（2023 年初）、更大的社区（10 万+ Star）、更丰富的工具生态，在通用框架领域占据明显优势。AutoGen 虽然在多 Agent 对话这个细分方向上有独特优势，但作为一个通用框架，它的用户规模、第三方集成数量都远不及 LangChain。

在这种情况下，微软做了一个务实的决策：与其在一个已经落后的开源战场上继续消耗资源，不如将重心转移到自己具有天然优势的云平台战场。于是，AutoGen 进入维护模式，MAF 作为其继任者登场。

历史参照： 这并不是微软第一次做这种调整。回顾历史，微软曾有过多个"开源→平台化"的先例：.NET Framework → .NET Core（后来再开源）、Bot Framework SDK → Power Virtual Agents。模式都是一样的：先用开源框架培育生态和创新，再将成熟能力整合进平台产品，走商业化变现路径。

18.2 AG2 社区分支：特性与兼容性

微软宣布 AutoGen 维护模式后，社区并没有就此放弃。一群核心贡献者 fork 了代码库，创建了 AG2（原 AutoGen 2 或 AutoGen AG2）——一个由社区驱动的 AutoGen 继承者。

AG2 的主要变化

维度 AutoGen (原始) AG2 (社区分支)
维护状态 维护模式，仅修 Bug 活跃开发，每月新版本
核心 API 0.2.x / 0.3.x 保持向后兼容，增加新 API
Python 版本 3.8+ 3.10+
模型支持 OpenAI + 部分兼容 OpenAI、Anthropic、Google、Ollama 等全系列
新特性 冻结 MCP 集成、Agent 模板、改进的 Docker 执行器
包名 pyautogen ag2

从 AutoGen 切换到 AG2

迁移过程非常简单——大部分代码只需要改 import 路径：

# 安装 AG2
# pip install ag2

# 第1步：修改 import 语句
# 原来（AutoGen）：
# from autogen import AssistantAgent, UserProxyAgent, GroupChat
# 改为（AG2）：
from ag2 import AssistantAgent, UserProxyAgent, GroupChat

# 第2步：配置无变化，API 保持兼容
assistant = AssistantAgent(
    name="assistant",
    llm_config={"config_list": [
        {"model": "gpt-4o", "api_key": "..."},
        {"model": "claude-sonnet-4-20250514", "api_key": "..."},  # AG2 直接支持
    ]},
)

# 第3步：AG2 新增特性：MCP 工具集成
from ag2 import MCPAgent

mcp_agent = MCPAgent(
    name="mcp_tool_user",
    mcp_server_config={
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-filesystem", "./data"],
    },
    llm_config={"config_list": [{"model": "gpt-4o", "api_key": "..."}]},
)
迁移建议： 如果你现有的 AutoGen 项目已经稳定运行，不需要立刻迁移。AutoGen 0.2.x 的 Bug 修复仍在继续，短期内没有安全隐患。但如果你要新建项目，强烈建议直接使用 AG2——它能获得最新的功能更新（如 MCP 集成、更多模型支持）和更活跃的社区支持。

18.3 Microsoft Agent Framework (MAF) 架构

如果说 AG2 是 AutoGen 的"社区继承者"，那么 Microsoft Agent Framework (MAF) 就是微软官方推出的"平台继承者"。MAF 不是一个简单的框架升级，而是整个 Agent 开发范式的重新设计。

MAF 架构分层

层次 组件 说明
应用层 Agent Application 业务逻辑定义，类似 AutoGen 的 AssistantAgent，但使用声明式配置
编排层 Orchestrator 多 Agent 任务调度、对话管理、状态持久化（替代 GroupChatManager）
工具层 Tool Registry / MCP Connector 工具注册中心，原生支持 MCP 协议，也兼容 OpenAPI 和 gRPC
基础设施层 Azure AI Foundry 模型部署、监控、安全策略、数据存储等云服务

MAF 核心特性

  • 声明式 Agent 定义： 使用 YAML 或 JSON 配置 Agent，而非纯代码。这使得非开发人员（如产品经理）也能参与 Agent 配置。
  • 云原生设计： 与 Azure AI Foundry 深度集成，提供一键部署、自动扩缩容、监控告警等企业级特性。
  • MCP 原生支持： MAF 的工具层直接实现了 MCP 协议，可以无缝接入任何 MCP Server。
  • 策略引擎： 内置安全策略、数据合规、内容过滤等企业治理能力，通过声明式策略配置实现。
  • 多模型支持： 不仅支持 Azure OpenAI，还支持通过 Azure AI Foundry 托管的第三方模型。
# MAF Agent 配置示例 (YAML)
# agents/sales_analyzer.yaml

name: sales_analyzer
description: 销售数据分析 Agent
model:
  provider: azure_openai
  deployment: gpt-4o
  temperature: 0.1

instructions: |
  你是一位资深销售分析师。你的职责是：
  1. 回答关于销售数据的各类问题
  2. 生成销售趋势分析报告
  3. 识别异常销售模式并提出预警

tools:
  - type: mcp
    server:
      command: npx
      args: ["@mcp/database-server", "--db", "sales_db"]
  - type: function
    name: send_email_alert
    description: 发送预警邮件
    parameters:
      type: object
      properties:
        recipient: { type: string }
        subject: { type: string }
        body: { type: string }

orchestration:
  max_turns: 20
  human_in_the_loop:
    actions: ["send_email_alert"]  # 发送邮件需要人工确认
  memory:
    type: semantic
    config:
      vector_store: azure_ai_search
      namespace: sales_analyst
设计理念： MAF 的声明式设计沿袭了 Kubernetes 的"声明式配置"哲学。Agent 的行为、工具、策略都被定义为"配置"而非"代码"。这带来的好处是：配置可以版本化管理、可以自动校验、可以图形化编辑。微软的目标是让 Agent 开发从"编程工作"变成"配置工作"。

18.4 从 AutoGen 迁移到 MAF

如果你已经在 AutoGen 上构建了 Agent 系统，迁移到 MAF 需要一定的工程投入。下面是迁移的关键路径和策略：

迁移步骤

  1. 概念映射： 将 AutoGen 的概念映射到 MAF 对应的概念上。AssistantAgent → MAF Agent Application；GroupChat → Orchestrator + 多 Agent 配置；Tool Registration → MCP Connector / Function Tool；code_execution_config → 内置 Code Interpreter 或 Docker Executor。
  2. 工具迁移： 将 Python 函数注册的工具转换为 MCP Server 或 MAF Function Tool 声明。如果工具较多，推荐先封装为 MCP Server，再接入 MAF。
  3. 对话逻辑重构： AutoGen 的对话逻辑是"代码驱动"的（通过 Python 控制对话流程），MAF 则是"配置驱动"的（通过 YAML/JSON 声明工作流）。需要将对话控制逻辑转换为声明式编排配置。
  4. 集成测试： 在迁移过程中逐步替换组件，确保每个步骤都有对应的测试验证业务逻辑不变。
# 迁移对比示例：工具注册
# AutoGen 方式（代码定义）
@user_proxy.register_for_llm(name="query_db", description="查询数据库")
def query_db(sql: str) -> list:
    return database.execute(sql)

# MAF 方式（声明式配置 - YAML）
# tools/query_db.yaml
# type: function
# name: query_db
# description: 查询数据库
# parameters:
#   type: object
#   properties:
#     sql:
#       type: string
#       description: SQL 查询语句
# implementation:
#   type: python
#   source: |
#     def query_db(sql: str) -> list:
#         return database.execute(sql)

渐进式迁移策略

我推荐采用"混合模式"进行渐进式迁移：

  • 第一阶段（并行运行）： 新增功能使用 MAF 开发，已有 AutoGen 功能保持不变。通过统一的 API 网关将两类 Agent 暴露给调用方。
  • 第二阶段（组件替换）： 逐个模块进行替换，从"与核心逻辑耦合度最低"的工具调用层开始，再到对话编排层。
  • 第三阶段（全面切换）： 当所有核心功能在 MAF 上验证通过后，关闭 AutoGen 部署，完成全面切换。
经验之谈： 不要试图"一步到位"地迁移。MAF 和 AutoGen 的编程模型差异较大，强行一次性全面迁移风险很高。我见过一个团队花了 3 个月做全面迁移，结果出现了大量兼容性问题，最后不得不回滚。相比之下，采用渐进式策略的团队通常在 2-3 个月内平稳完成迁移。

18.5 MAF + Azure AI Foundry 集成

MAF 的真正威力在于与 Azure AI Foundry 的深度集成。Azure AI Foundry 是微软的统一 AI 开发平台，提供了模型管理、数据集成、监控运维等全套服务。

集成能力矩阵

能力 AutoGen 方式 MAF + Azure AI Foundry 方式
模型管理 在代码中配置 API Key 和 Model 通过 AI Foundry Playground 选择和管理模型，自动生成配置
监控 自行集成 LangSmith 或自定义日志 内置 Trace、Token 消耗、延迟、错误率等指标面板
安全 自行实现内容过滤和权限控制 内置 Azure AI Content Safety，支持 PII 检测、毒害内容过滤
部署 自行部署 API 服务或使用云服务 一键部署到 Azure Container Apps，自动扩缩容
数据 自行管理向量数据库和文档存储 集成 Azure AI Search、Blob Storage、Cosmos DB
评测 自行编写测试脚本 内置 E2E 评测工作流，支持人工评估和自动评估 
# MAF + Azure AI Foundry 部署脚本示例
# deploy_agent.py

from azure.ai.resources import AIResource
from azure.ai.agent import AgentClient

# 连接到 Azure AI Foundry 项目
ai_resource = AIResource.from_connection_string(
    ""
)

# 在 Foundry 中创建和注册 Agent
agent_client = AgentClient(ai_resource)
agent = agent_client.create_agent(
    name="production_sales_analyzer",
    model_deployment="gpt-4o",
    instructions="你是一位销售分析师...",
    tools=[
        {"type": "code_interpreter"},               # 内置代码执行器
        {"type": "file_search",                      # 文件搜索
         "vector_store_ids": ["vs_sales_data"]},
        {"type": "mcp_server",                       # MCP 工具
         "server_url": "https://mcp.internal/sales"},
    ],
)

# 部署到 Azure Container Apps
agent.deploy(
    target="azure_container_apps",
    replicas=3,
    scaling_rule={"cpu": 70, "memory": "1Gi"},
)

print(f"Agent 已部署：{agent.endpoint_url}")
成本考量： 使用 MAF + Azure AI Foundry 意味着你将享受云平台的规模效应——不需要自己管理基础设施、不需要自己搭建监控系统、不需要自己处理扩缩容。但代价是厂商锁定和按量计费。评估时建议做一个 TCO（总拥有成本）对比，特别是你的 Agent 调用量很大时。

18.6 选型建议：什么时候用什么

经过前面的分析，我们已经清楚地看到了三条路径：AutoGen（维护模式）、AG2（社区驱动）、MAF（微软官方/云平台）。作为开发者，你应该如何选择？

决策树

你正在构建什么项目？

├── 现有 AutoGen 项目已稳定运行：
│   └── ➜ 保持 AutoGen，无需迁移
│
├── 新项目，需要多 Agent 对话能力：
│   ├── 项目是开源自托管，需要长期维护？
│   │   └── ➜ 使用 AG2（社区活跃，持续更新）
│   ├── 项目是 Azure 生态内的企业应用？
│   │   └── ➜ 使用 MAF（云集成最佳体验）
│   └── 项目需要快速原型验证？
│       └── ➜ 先用 AG2（上手快），必要时迁移到 MAF
│
├── 项目需要生产级企业特性：
│   ├── 监控/安全/合规是硬性要求？
│   │   └── ➜ MAF（内置企业特性）
│   └── 需要与 Azure AI Foundry 集成？
│       └── ➜ MAF（原生集成）
│
└── 学习目的，理解多 Agent 设计模式：
    └── ➜ 学习 AutoGen 设计思想（概念通用），
        再过渡到 AG2 或 MAF

场景推荐总结

场景 推荐 理由
个人开发者/学习 AG2 免费开源，社区活跃，文档完善
小型团队 SaaS AG2 轻量可控，无供应商锁定
Azure 客户 MAF 与 Azure 生态深度集成，运维省心
大型企业（合规敏感） MAF 内置企业治理、内容安全、审计日志
多 Agent 研究/实验 AG2 支持更多模型，社区迭代快
长期维护的老项目 AutoGen 0.2.x 稳定不折腾，Bug 修复仍在继续
我的建议： 如果你正在为团队做技术选型，不要把这个问题简单地看作"选 A 还是选 B"。更务实的思路是：核心逻辑与框架解耦。将 Agent 的业务逻辑抽象为清晰的接口层，使得底层框架更换时不需要重写业务代码。这样无论生态如何变化，你的核心资产都受到保护。我们会在第 28 章讨论框架选型的更多方法论。
