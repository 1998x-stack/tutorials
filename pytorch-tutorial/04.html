第4章 自动求导机制

大家好，我是你们的讲师。上一章我们学会了各种 Tensor 运算——你可以随意地加、乘、转置、拼接。但有一个问题我们还没回答：当我们把一堆运算串起来形成一个计算链条后，PyTorch 怎么知道每个中间结果关于输入的梯度？

在深度学习中，训练一个模型本质上就是"用梯度下降法优化参数"。而梯度的计算，如果每次都手写解析导数公式，工作量是不可想象的——尤其是当你有个 100 层的 Transformer 时，链式法则展开能写满几块白板。

PyTorch 的 Autograd（Automatic Differentiation，自动求导） 机制解决了这个问题。它会自动追踪你的所有计算操作，构建一张"计算图"，然后在你需要的时候自动算出梯度。这一章我们就来深入理解它的工作原理和正确用法。

什么是自动求导？

在自动求导出现之前，人们计算导数主要有两种方式：

方法 原理 优点 缺点
数值求导 f'(x) ≈ (f(x+h) - f(x-h)) / 2h 实现简单，不需解析公式 计算量大、有截断误差、不精确
符号求导 对表达式进行符号级求导得到闭式解 精确 表达式爆炸（一个 100 层网络的导数公式会指数级增长）
自动求导 将计算过程分解为基础运算，用链式法则逐步求导 精确且高效 需要构建计算图（内存开销）

自动求导的核心思想是：任何复杂的数学函数都可以分解为一系列基础运算（加减乘除、指数、对数、三角函数等）。你不需要对整个函数求导——只需要对每个基础运算求导，然后通过链式法则把结果串起来。

计算图的构建：前向传播

当你执行一个 Tensor 操作时，PyTorch 在幕后做两件事：

  1. 计算出结果 Tensor
  2. 在内存中记录这个操作，构建一个由 Function 节点组成的有向无环图（DAG）

这个图就是计算图——每个节点代表一个运算，每条边代表数据的流向。

import torch

# 设置 requires_grad=True 来追踪这个张量的操作
x = torch.tensor([2.0], requires_grad=True)
print(x.requires_grad)  # True

# 执行一些运算
y = x ** 2
z = y * 3 + 1

print(z)  # tensor([13.], grad_fn=<AddBackward0>)
print(y)  # tensor([4.], grad_fn=<PowBackward0>)

看到 grad_fn 了吗？这个属性告诉你这个 Tensor 是通过什么运算得到的。y 的 grad_fn 是 PowBackward0（幂运算），z 的 grad_fn 是 AddBackward0（加法）。如果 Tensor 是手动创建的（叶节点），grad_fn 就是 None。

计算图的结构（对应上面的代码）： 
x (leaf, grad_fn=None)
 └─ PowBackward0 → y = x²
     └─ MulBackward0 → y × 3
         └─ AddBackward0 → (y×3) + 1 → z
这是前向传播时自动构建的。PyTorch 记住的不仅仅是计算结果，还有计算的方式。

反向传播：backward() 的工作原理

当你调用 z.backward() 时，PyTorch 从 z 开始，沿着计算图反向遍历，使用链式法则自动计算梯度：

z.backward()  # 从 z 出发反向传播

# 查看梯度
print(x.grad)  # tensor([12.]) — dz/dx = 12

# 验证：z = 3x² + 1 → dz/dx = 6x → x=2 时 6*2 = 12 ✓

手动推导验证一下：z = 3x² + 1，对 x 求导得 dz/dx = 6x，当 x = 2 时，梯度为 12。完全正确。

计算图的反向传播逻辑可以形象地理解为：每个运算节点都知道如何计算它自己的输出的梯度（局部梯度），然后通过链式法则，从后往前把梯度逐层传递。

理解梯度累积：为什么需要 zero_grad

下面这个知识点特别重要——不理解的后果就是你的模型 loss 会莫名其妙变成 NaN。

PyTorch 中，每次调用 backward() 时，梯度不会自动归零，而是累加到 .grad 属性中。

这是 PyTorch 有意为之的设计——在某些场景下（比如 RNN 训练中的梯度累积），你确实希望累加梯度。但对于最常规的训练流程，你需要手动清零。

先讲一个真实的故事。2020 年，我带的实习生小李在训练一个图像分类模型时，遇到了一个诡异的 bug：前几个 batch 的 loss 下降完全正常，但跑到第 10 个 batch 左右，loss 突然变成了 NaN。他以为学习率太大，调小了还是 NaN；以为数据有问题，检查了数据也没问题。折腾了一整天，最后我在他代码里看到了这个：

for batch in dataloader:
    loss = model(batch)
    loss.backward()
    # 忘了 optimizer.zero_grad()！
    optimizer.step()

少了 zero_grad()，梯度在每个 batch 中不断累加，越来越大，最终数值溢出变成 NaN。这是 PyTorch 初学者最常犯的错误之一，而且它有一个很迷惑人的特点——前几个 batch 看着正常，让你误以为代码没问题。

记住这个铁三角：在每一个训练步骤中，这三个调用的顺序是固定的——
  1. optimizer.zero_grad() — 清空上一轮的梯度
  2. loss.backward() — 计算当前批次的梯度
  3. optimizer.step() — 用梯度更新参数
顺序写错（比如先 backward 再 zero_grad）等于白做！

梯度模式对比：requires_grad, no_grad, detach, clone

在编写训练代码时，你会遇到各种需要控制梯度计算的场景。让我们系统地梳理一下：

操作 影响 数据是否共享 是否保留计算图 典型使用场景
requires_grad=True 追踪该张量的所有操作 - 是 模型参数需要梯度来更新
requires_grad=False 停止追踪该张量的操作 - 否 输入数据、标签（不需要梯度）
torch.no_grad() 上下文内所有操作不追踪 - 否 推理、模型评估、特征提取
.detach() 从计算图中分离，返回新张量 是（共享内存） 否 取出数值用于可视化、日志记录
.clone() 复制数据并保持梯度计算 否（独立副本） 是 需要独立副本但保留梯度传播路径 
x = torch.tensor([2.0], requires_grad=True)

# 1. torch.no_grad() — 整个上下文不追踪
with torch.no_grad():
    y = x * 2
    print(y.requires_grad)  # False
    print(y.grad_fn)        # None — 没有记录操作

# 2. .detach() — 从当前计算图分离
z = (x ** 2).detach()
print(z.requires_grad)  # False
print(z.grad_fn)        # None

# 3. .clone() — 保持计算图但数据独立
w = x.clone()
print(w.requires_grad)  # True
print(w.grad_fn)        # None（是叶节点，但与原 x 共享梯度路径）

# 4. 不同模式对内存和速度的影响
import time

big_tensor = torch.randn(1000, 1000)

# 追踪梯度
t1 = time.time()
y1 = big_tensor @ big_tensor
t2 = time.time()
print(f"有 grad_fn: {t2-t1:.4f}s")  # 稍慢，需要保存中间结果

# 不追踪梯度
t1 = time.time()
with torch.no_grad():
    y2 = big_tensor @ big_tensor
t2 = time.time()
print(f"无 grad_fn: {t2-t1:.4f}s")  # 更快，而且省显存
最佳实践：
  • 推理时一定要用 torch.no_grad() —— 节省显存和计算时间
  • 要把 Tensor 转 NumPy 画图或打印时，用 .detach().cpu().numpy() 三连
  • 验证集评估时，也建议包在 torch.no_grad() 中

retain_graph 与高阶梯度

默认情况下，每次调用 backward() 后，PyTorch 会释放计算图以节省内存。这意味着你不能对同一个计算图调用两次 backward()——除非你指定 retain_graph=True。

x = torch.tensor([2.0], requires_grad=True)
y = x ** 2

# 第一次 backward
y.backward(retain_graph=True)  # 保留计算图
print(x.grad)  # tensor([4.])

x.grad.zero_()  # 记得清空梯度！

# 第二次 backward
y.backward()  # 这次不保留
print(x.grad)  # tensor([4.])

# 如果不加 retain_graph=True，第二次 backward 会报错
y = x ** 2
y.backward()  # 第一次
# y.backward()  # RuntimeError! 计算图已经被释放了

retain_graph 的使用场景：

  • GAN 训练中需要分别对生成器和判别器计算梯度
  • 某些元学习（meta-learning）算法需要二阶梯度
  • 分析梯度流时需要在同一个计算图上多次 backward
性能警告：retain_graph=True 会阻止计算图的释放，显著增加显存占用。只有在确实需要时才使用它。如果你发现显存随着训练步数持续增长（不释放），检查一下是不是某处用了 retain_graph=True 但没有及时释放。

反向传播的细节：grad 在哪个张量上？

一个常见的困惑是：backward() 之后，梯度到底保存在哪里？

答案是：梯度只保存在设置了 requires_grad=True 的叶节点（leaf tensor）上。非叶节点的梯度在反向传播完成后就被释放了——因为不需要再更新非叶节点的值。

x = torch.tensor([2.0], requires_grad=True)  # 叶节点
y = x ** 2                                     # 非叶节点
z = y * 3                                      # 非叶节点

z.backward()

print(x.grad)  # tensor([12.]) — 叶节点有 .grad
print(y.grad)  # None — 非叶节点的梯度默认不保存
print(z.grad)  # None

如果你想获取非叶节点的梯度，可以使用 .retain_grad()：

x = torch.tensor([2.0], requires_grad=True)
y = x ** 2
y.retain_grad()  # 告诉 PyTorch 保留 y 的梯度
z = y * 3

z.backward()

print(y.grad)  # tensor([3.]) — 现在可以看到了

在调试复杂的网络时，.retain_grad() 是一个有用的工具——比如你可以检查中间层的梯度是否为 NaN，来判断梯度爆炸的位置。

完整的训练步骤中的 Autograd 流程

让我们用一个完整的例子把学到的知识串起来：

import torch
import torch.nn as nn

# 1. 准备数据
x = torch.tensor([[1.0], [2.0], [3.0]])
y_true = torch.tensor([[2.0], [4.0], [6.0]])  # y = 2x

# 2. 定义模型（requires_grad=True 参数）
linear = nn.Linear(1, 1)

# 查看参数梯度初始状态
print(linear.weight.grad)  # None — 还没计算

# 3. 前向传播
y_pred = linear(x)
loss = ((y_pred - y_true) ** 2).mean()

print("计算图:", loss.grad_fn)
# 输出: MeanBackward0 → ... → AddmmBackward0 → ...

# 4. 反向传播
loss.backward()

# 5. 查看参数梯度（现在有了！）
print(linear.weight.grad)  # tensor([[...]]) — 梯度值
print(linear.bias.grad)    # tensor([[...]])

# 6. 用优化器更新参数（第6章详细讲）
# optimizer = torch.optim.SGD(linear.parameters(), lr=0.01)
# optimizer.step()

注意到 backward() 之前参数梯度是 None，之后就有了值——这就是 autograd 的魔力。

Autograd 的工作流（必须记住）：
  1. 前向传播：执行计算 → 自动构建计算图（记住每步运算）
  2. 计算 loss：loss 是计算图的终点
  3. backward()：从 loss 出发反向遍历 → 每个参数得到梯度
  4. zero_grad()：清空梯度（准备下一轮）
  5. 重复
步骤 1-3 构成了一个完整的"前向-后向"周期。

本章小结

这一章我们深入学习了 PyTorch 的自动求导机制。来回顾一下核心要点：

概念 说明 代码标识
计算图 前向传播时自动构建的 DAG，记录所有运算 grad_fn 属性
backward() 从 loss 出发反向传播，计算梯度 loss.backward()
梯度累积 梯度累加到 .grad，不会自动清零 每步需 zero_grad()
no_grad 推理时禁用梯度追踪，节省显存 torch.no_grad()
detach 从计算图中分离出无梯度的副本 tensor.detach()
retain_graph 保留计算图供多次 backward backward(retain_graph=True)

理解自动求导是理解 PyTorch 训练流程的钥匙。有了这个基础，下一章我们就可以开始构建第一个真正的神经网络了——不再是单独的 Tensor 运算，而是把层（Layer）堆叠成一个完整的模型。下一章见！
