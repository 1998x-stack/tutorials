10. CAN报文发送：发送邮箱机制、发送优先级、发送流程、发送完成中断

好，咱们今天聊聊CAN报文的发送。说实话，很多初学者觉得发送就是往寄存器里写数据，然后等着发出去就行了。嗯，要是真这么简单，那汽车上就不会有那么多通信故障了。

我个人习惯把CAN发送比作「快递发货」。你想想看，一个快递站有多个发货口，每个口一次只能发一个包裹，而且包裹还有轻重缓急之分。CAN的发送邮箱，说白了就是干这个的。

10.1 发送邮箱机制

CAN控制器内部一般有3个发送邮箱（有些芯片更多，比如STM32有3个，有些高端芯片有8个甚至16个）。每个邮箱就是一个独立的「发货窗口」。

邮箱的结构：每个邮箱包含标识符、数据长度、数据场、控制位这些信息。你往邮箱里填好数据，告诉控制器「发吧」，剩下的硬件自己搞定。

关键点：发送邮箱是硬件资源，不是软件队列。你往邮箱里写数据，硬件就自动开始发送流程。如果三个邮箱都满了，你再想发，就得等其中一个发完。

我在项目中遇到过一个问题：某个节点需要频繁发送诊断报文，结果因为邮箱被其他报文占满了，诊断响应超时了。后来怎么解决的？我把诊断报文单独用一个邮箱，并且优先级设得最高。

10.2 发送优先级

CAN总线的优先级，不是靠邮箱编号决定的，而是靠报文ID。ID越小，优先级越高。这个大家应该都知道。

但这里有个细节：当多个邮箱同时有数据要发送时，硬件会怎么选？

答案是：硬件会先看哪个邮箱的报文ID最小，就优先发哪个。如果ID相同，再看邮箱编号，编号小的优先。

邮箱编号 报文ID 发送优先级
邮箱0 0x100 最高（ID最小）
邮箱1 0x200 中等
邮箱2 0x300 最低

我曾经踩过一个坑：把紧急报文放在邮箱2里，结果邮箱0和邮箱1一直有数据在发，紧急报文死活发不出去。后来我把紧急报文放在邮箱0，问题就解决了。

我的建议：把最紧急、最关键的报文放在邮箱0，次紧急的放邮箱1，普通的放邮箱2。这样即使邮箱满了，关键报文也能优先发出。

10.3 发送流程

CAN报文的发送流程，其实就三步：填数据、请求发送、等待完成。

第一步：填数据

你要往邮箱的寄存器里写：

  • 标识符（ID）
  • 数据长度码（DLC）
  • 数据场（最多8字节）
  • 控制位（比如是否使用扩展帧）

第二步：请求发送

把邮箱的发送请求位置1。硬件检测到这个位被置1，就开始尝试发送。

第三步：等待完成

硬件发送完成后，会把发送完成位置1。你可以轮询这个位，也可以开中断。

下面是一个典型的发送代码示例：

// 假设使用邮箱0发送
void CAN_SendMessage(uint32_t id, uint8_t *data, uint8_t len)
{
    // 1. 等待邮箱0空闲
    while (CAN->TSR & CAN_TSR_TME0) {
        // 等待，或者超时处理
    }
    
    // 2. 填写标识符
    CAN->sTxMailBox[0].TIR = (id << 21) | CAN_TIxR_IDE;  // 标准帧
    
    // 3. 填写数据长度
    CAN->sTxMailBox[0].TDTR = len;
    
    // 4. 填写数据
    CAN->sTxMailBox[0].TDLR = data[0] | (data[1] << 8) | 
                                (data[2] << 16) | (data[3] << 24);
    CAN->sTxMailBox[0].TDHR = data[4] | (data[5] << 8) | 
                                (data[6] << 16) | (data[7] << 24);
    
    // 5. 请求发送
    CAN->sTxMailBox[0].TIR |= CAN_TIxR_TXRQ;
}

嗯，这里要注意：发送请求位一旦置1，就不能再修改邮箱里的数据了。否则会导致发送错误。我见过有人一边发一边改数据，结果报文内容乱套了。

10.4 发送完成中断

轮询发送完成位，其实挺浪费CPU的。你想想看，CPU一直在那傻等，啥也干不了。所以，更高效的做法是用中断。

发送完成中断，就是硬件发完一个报文后，自动触发一个中断。你在中断服务函数里处理后续逻辑，比如释放邮箱、更新状态、或者发送下一条报文。

中断的配置步骤：

  1. 使能发送完成中断（一般有个寄存器位控制）
  2. 在中断向量表中注册中断服务函数
  3. 在中断服务函数里读取状态，清除中断标志

代码示例：

// 发送完成中断服务函数
void CAN_TX_IRQHandler(void)
{
    // 检查是哪个邮箱完成了发送
    if (CAN->TSR & CAN_TSR_RQCP0) {
        // 邮箱0发送完成
        CAN->TSR |= CAN_TSR_RQCP0;  // 清除中断标志
        
        // 处理后续逻辑，比如发送下一条报文
        SendNextMessage();
    }
    
    if (CAN->TSR & CAN_TSR_RQCP1) {
        // 邮箱1发送完成
        CAN->TSR |= CAN_TSR_RQCP1;
        // ...
    }
    
    if (CAN->TSR & CAN_TSR_RQCP2) {
        // 邮箱2发送完成
        CAN->TSR |= CAN_TSR_RQCP2;
        // ...
    }
}

警告：中断服务函数里不要做太多事情！我曾经见过有人在中断里调用延时函数，结果整个系统卡死了。中断里只做最必要的事情，比如置个标志位，复杂逻辑放到主循环里处理。

还有一个坑：发送完成中断和发送错误中断是共用一个中断源的。你处理的时候，一定要区分是正常完成还是错误导致的完成。否则，你可能会把错误报文当成正常报文处理。

10.5 避坑指南

最后，我总结几个实际项目中容易踩的坑：

  • 邮箱溢出：三个邮箱都满了，你还往里写数据，数据就丢了。解决办法：要么增加邮箱数量（换芯片），要么用软件队列缓冲。
  • 优先级反转：低优先级报文占着邮箱，高优先级报文发不出去。解决办法：合理分配邮箱，紧急报文用独立邮箱。
  • 中断嵌套：发送完成中断里又触发了发送完成中断，导致递归。解决办法：中断里只清标志，不做发送操作。
  • 总线关闭：发送错误太多，节点进入总线关闭状态。解决办法：监控错误计数器，及时复位。

我曾经在一个项目中，因为发送完成中断里调用了发送函数，导致中断递归，堆栈溢出。查了两天才找到原因。从那以后，我定了个规矩：中断里只做标志位操作，不做任何函数调用。
